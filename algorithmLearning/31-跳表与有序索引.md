# 跳表与有序索引

红黑树很强,但实现起来复杂:插入删除要旋转、染色、修复各种情况。跳表(Skip List)走的是另一条路:不用旋转,给链表加多层"快速通道",靠随机层高把查询复杂度做到期望 O(log n)。Redis zset、LevelDB MemTable、etcd 内部都用过跳表或类似结构,原因很朴素:**好实现、好范围扫描、性能稳定够用**。

> 一句话先记住:**跳表是用多层链表模拟平衡树**。底层保存所有节点,上层保存部分节点作为索引;查找时从最高层一路往右、往下走。它用随机性换掉红黑树的复杂旋转。

---

## 一、普通链表为什么不够

有序链表支持范围扫描:

```text
1 -> 4 -> 7 -> 9 -> 13 -> 20
```

查找 `13` 只能从头扫:

```text
1 -> 4 -> 7 -> 9 -> 13
```

复杂度 O(n)。

如果给链表加一层索引:

```text
      1 ------> 7 ------> 13
      |         |          |
base: 1 -> 4 -> 7 -> 9 -> 13 -> 20
```

查找时先在上层跳,再下到底层。层数越多,越像二分。

跳表就是这个思想的工程化版本。

---

## 二、跳表的结构

一个节点不只有一个 `next`,而是一组 `forward` 指针:

```ts
class SkipNode<K, V> {
  key: K
  value: V
  forward: Array<SkipNode<K, V> | null>

  constructor(key: K, value: V, level: number) {
    this.key = key
    this.value = value
    this.forward = new Array(level).fill(null)
  }
}
```

如果一个节点高度是 4,它会出现在第 0、1、2、3 层。

```text
level 3:  head ----------------------> 20
level 2:  head ----------> 9 --------> 20
level 1:  head --> 4 ----> 9 --------> 20
level 0:  head --> 4 -> 7 -> 9 -> 13 -> 20
```

第 0 层保存所有节点。越高层节点越少。

---

## 三、查找:往右走不动就往下

查找 `target` 的过程:

1. 从最高层的 head 开始
2. 如果右边节点 key 小于 target,就往右
3. 如果右边节点 key 大于等于 target,就往下
4. 到第 0 层后检查下一个节点

```text
从高层快速跳过一大段,最后在底层精确定位。
```

代码骨架:

```ts
function findGreaterOrEqual<K, V>(
  head: SkipNode<K, V>,
  level: number,
  key: K,
  compare: (a: K, b: K) => number
): SkipNode<K, V> | null {
  let current = head

  for (let i = level - 1; i >= 0; i--) {
    while (current.forward[i] && compare(current.forward[i]!.key, key) < 0) {
      current = current.forward[i]!
    }
  }

  return current.forward[0]
}
```

## 四、随机层高

跳表不靠旋转维持平衡,靠随机层高。

常见策略:

```text
第 1 层概率: 100%
第 2 层概率: 50%
第 3 层概率: 25%
第 4 层概率: 12.5%
...
```

代码:

```ts
function randomLevel(maxLevel: number, p = 0.5, random = Math.random): number {
  let level = 1

  while (level < maxLevel && random() < p) {
    level++
  }

  return level
}
```

这会让高层节点越来越少,整体高度期望 O(log n)。

注意这是**期望平衡**,不是红黑树那种严格平衡。极端情况下跳表也可能退化,但概率很低。

## 五、完整简化实现

下面是一个支持 `get`、`put`、`range` 的简化跳表。

```ts
type Compare<K> = (a: K, b: K) => number

class SimpleSkipList<K, V> {
  private head: SkipNode<K, V>
  private level = 1
  private maxLevel: number
  private compare: Compare<K>

  constructor(compare: Compare<K>, maxLevel = 32) {
    this.compare = compare
    this.maxLevel = maxLevel
    this.head = new SkipNode<K, V>(null as K, null as V, maxLevel)
  }

  get(key: K): V | undefined {
    const node = this.findGreaterOrEqual(key)
    return node && this.compare(node.key, key) === 0 ? node.value : undefined
  }

  put(key: K, value: V): void {
    const update = new Array<SkipNode<K, V>>(this.maxLevel)
    let current = this.head

    for (let i = this.level - 1; i >= 0; i--) {
      while (current.forward[i] && this.compare(current.forward[i]!.key, key) < 0) {
        current = current.forward[i]!
      }
      update[i] = current
    }

    const existing = current.forward[0]
    if (existing && this.compare(existing.key, key) === 0) {
      existing.value = value
      return
    }

    const nodeLevel = randomLevel(this.maxLevel)

    if (nodeLevel > this.level) {
      for (let i = this.level; i < nodeLevel; i++) {
        update[i] = this.head
      }
      this.level = nodeLevel
    }

    const node = new SkipNode(key, value, nodeLevel)

    for (let i = 0; i < nodeLevel; i++) {
      node.forward[i] = update[i].forward[i]
      update[i].forward[i] = node
    }
  }

  delete(key: K): boolean {
    const update = new Array<SkipNode<K, V>>(this.maxLevel)
    let current = this.head

    for (let i = this.level - 1; i >= 0; i--) {
      while (current.forward[i] && this.compare(current.forward[i]!.key, key) < 0) {
        current = current.forward[i]!
      }
      update[i] = current
    }

    const target = current.forward[0]
    if (!target || this.compare(target.key, key) !== 0) {
      return false
    }

    for (let i = 0; i < this.level; i++) {
      if (update[i].forward[i] !== target) break
      update[i].forward[i] = target.forward[i]
    }

    while (this.level > 1 && this.head.forward[this.level - 1] === null) {
      this.level--
    }

    return true
  }

  range(start: K, end: K): Array<[K, V]> {
    const result: Array<[K, V]> = []
    let node = this.findGreaterOrEqual(start)

    while (node && this.compare(node.key, end) <= 0) {
      result.push([node.key, node.value])
      node = node.forward[0]
    }

    return result
  }

  private findGreaterOrEqual(key: K): SkipNode<K, V> | null {
    let current = this.head

    for (let i = this.level - 1; i >= 0; i--) {
      while (current.forward[i] && this.compare(current.forward[i]!.key, key) < 0) {
        current = current.forward[i]!
      }
    }

    return current.forward[0]
  }
}
```

复杂度:

| 操作 | 期望复杂度 |
| --- | --- |
| 查找 | O(log n) |
| 插入 | O(log n) |
| 删除 | O(log n) |
| 范围扫描 | O(log n + k) |
| 空间 | O(n) |

`k` 是范围内返回的元素数量。

## 六、跳表为什么适合范围扫描

有序索引最重要的能力之一是范围查询:

```text
查 key 在 [100, 200] 之间的所有记录
```

跳表的做法:

1. 用 O(log n) 定位到第一个 `>= 100` 的节点
2. 沿着第 0 层链表向后扫到 `> 200`

第 0 层本来就是完整有序链表,所以范围扫描非常自然。

这也是 Redis zset 喜欢跳表的原因之一:

- 按 score 排序
- 支持范围查询
- 插入删除频繁
- 实现比红黑树简单

---

## 七、Redis zset 里的跳表

Redis 的 zset 可以理解为两套结构配合:

```text
dict:    member -> score
skiplist: score + member -> ordered nodes
```

为什么要两套?

- 按 member 查 score:哈希表 O(1)
- 按 score 排序 / 范围查询:跳表 O(log n + k)

如果只用跳表,按 member 查会慢。

如果只用哈希表,按 score 范围查询要全表扫。

这正是算法系列反复讲的:**一个工程结构经常是多个基础结构拼出来的**。

---

## 八、跳表 vs 红黑树 vs B+ 树

| 结构 | 查找 | 插入删除 | 范围扫描 | 实现复杂度 | 典型场景 |
| --- | --- | --- | --- | --- | --- |
| 红黑树 | O(log n) 最坏 | O(log n) 最坏 | 好 | 高 | `TreeMap`,内存有序 Map |
| 跳表 | O(log n) 期望 | O(log n) 期望 | 很好 | 中 | Redis zset,MemTable |
| B+ 树 | O(log n) | O(log n) | 很好 | 高 | 数据库磁盘索引 |

选择信号:

- 内存结构、想要实现简单、范围扫描多 → 跳表
- 语言标准库 / 需要最坏复杂度保证 → 红黑树
- 磁盘 / SSD / 页式存储 → B+ 树

跳表不是红黑树的全面替代品。它只是工程上很舒服的折中。

## 九、工程注意事项

### 1. 比较器必须稳定

跳表依赖全序关系。比较器如果不稳定,比如同样两个 key 有时返回大、有时返回小,整个结构会坏。

### 2. 复合 key 要处理并列

Redis zset 的 score 可能相同,所以实际排序是:

```text
(score, member)
```

先比 score,score 相同再比 member,保证 key 唯一且顺序稳定。

### 3. 随机层高需要上限

`maxLevel` 常见取 16、32、64。不要无限增长。

### 4. 并发跳表不是简单加锁

并发跳表要处理多层指针更新的可见性和一致性。业务里优先用成熟库,不要临时手写无锁跳表。

### 5. 内存开销不可忽略

每个节点有多个 forward 指针。跳表换来了实现简单和范围扫描友好,代价是额外指针内存。

---

## 十、一句话总结

> **跳表是工程师版本的平衡树**。它不靠旋转维持严格平衡,而是靠随机层高构造多级索引,用更简单的实现换来期望 O(log n) 的查找、插入、删除和天然顺滑的范围扫描。看到"内存有序索引 + 范围查询 + 实现复杂度要低",就该想到跳表。
