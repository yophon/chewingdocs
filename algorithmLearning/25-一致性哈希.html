一致性哈希 ​

一致性哈希(Consistent Hashing)是分布式系统的"小聪明"——用数学构造让节点上下线只影响一小部分数据,避免了"扩容时所有数据全部搬家" 的灾难。Memcached 集群、Redis Cluster、CDN 的负载均衡、分库分表的路由全靠它。

一句话先记住:一致性哈希把节点和数据都哈希到一个"环"上,数据归属"顺时针遇到的第一个节点"。增删一个节点,只影响环上相邻的那段数据(平均 1/N),不像普通哈希那样全部重映射。

一、问题:普通哈希为什么不行 ​

把数据按 hash(key) % N 路由到 N 台服务器:

hash(key) = 12345
12345 % 3 = 0  → 服务器 0

问题:N 一变,几乎所有 key 的归属都变了。

N=3,key=12345  → 12345 % 3 = 0 → 服务器 0
N=4,key=12345  → 12345 % 4 = 1 → 服务器 1  (变了!)

加一台机器或挂掉一台,90% 以上的数据要重新搬运。缓存大面积失效 → 雪崩 → 数据库被打爆。

这是普通哈希的死穴,一致性哈希就是为了解决这个。

二、一致性哈希的核心思想 ​

把哈希值想象成一个圆环(0 到 2³² - 1,首尾相连):

       0/2³²
        |
   N1   |   N2
      \ | /
       (环)
      / | \
   N4   |   N3
        |

步骤:

  1. 每个节点经过哈希后落在环上某个位置
  2. 每个 key 也经过哈希落在环上某个位置
  3. 数据归属"顺时针走遇到的第一个节点"
key 哈希后落在 N1 和 N2 之间 → 属于 N2(顺时针先碰到)

三、增删节点只影响"那一段" ​

加一个节点 ​

原:N1 → [data] → N2
新:N1 → [部分 data] → N5 → [部分 data] → N2

只有从 N1 到 N5 这段的数据需要从 N2 搬到 N5,其他数据归属不变。

搬运比例 = 1/N(平均)——加第 N+1 台机器,只影响 1/(N+1) 的数据。

删一个节点 ​

原:N1 → [data] → N2 → [data] → N3
删 N2:N1 → [data + data] → N3

原本属于 N2 的数据,全部顺时针给 N3。其他不动。

四、朴素实现的两个问题 ​

问题 1:节点分布不均 ​

只有 3 个节点,哈希到环上位置随机——可能 3 个节点挤在一起,导致一个节点承担全部 70% 的数据。

问题 2:删节点后负载倾斜 ​

删 N2 后,N2 的数据全部给 N3,N3 负担瞬间翻倍。

这两个问题的解决方案是同一个:虚拟节点。

五、虚拟节点(Virtual Nodes) ​

核心思路:每个真实节点生成多个虚拟节点,把虚拟节点撒在环上,数据归属虚拟节点 → 真实节点。

真实节点:    N1, N2, N3
虚拟节点:    N1#0, N1#1, ..., N1#150,
            N2#0, N2#1, ..., N2#150,
            N3#0, N3#1, ..., N3#150
共 450 个虚拟节点撒在环上

效果:

  1. 分布更均匀(450 个点比 3 个点撒得均匀)
  2. 删节点时,N1 的 150 个虚拟节点的"下一个"会分散到 N2 和 N3 的虚拟节点上——负载平均分担,不会单点爆炸

经验值:每个真实节点 100~200 个虚拟节点够用。Redis Cluster 用 16384 个 slot 就是这个思想的极致版本。

六、完整实现 ​

typescript
import { createHash } from 'crypto'

class ConsistentHash {
  private ring: { hash: number, node: string }[] = []  // 排序后的虚拟节点
  private vNodesPerNode: number

  constructor(nodes: string[] = [], vNodes = 150) {
    this.vNodesPerNode = vNodes
    nodes.forEach(n => this.addNode(n))
  }

  private hash(key: string): number {
    const h = createHash('md5').update(key).digest('hex')
    return parseInt(h.slice(0, 8), 16)
  }

  addNode(node: string): void {
    for (let i = 0; i < this.vNodesPerNode; i++) {
      this.ring.push({ hash: this.hash(`${node}#${i}`), node })
    }
    this.ring.sort((a, b) => a.hash - b.hash)
  }

  removeNode(node: string): void {
    this.ring = this.ring.filter(e => e.node !== node)
  }

  getNode(key: string): string | null {
    if (this.ring.length === 0) return null
    const h = this.hash(key)
    // 二分查找第一个 ≥ h 的虚拟节点
    let lo = 0, hi = this.ring.length
    while (lo < hi) {
      const mid = (lo + hi) >> 1
      if (this.ring[mid].hash >= h) hi = mid
      else lo = mid + 1
    }
    if (lo === this.ring.length) lo = 0  // 环回到起点
    return this.ring[lo].node
  }
}

复杂度:

  • 增删节点:O(V·log V)(V = 总虚拟节点数)
  • 查询:O(log V)(二分)

七、工程优化 ​

1. 用更好的哈希函数 ​

MD5 / SHA-1 太慢且过度,MurmurHash3 / xxHash 是工业标准——快、散得好。

2. 跳表替代有序数组 ​

加节点时不用整体排序,跳表 O(log V) 插入。节点频繁变动时用跳表。

3. 节点权重(异构集群) ​

机器配置不同——大机器 200 个虚拟节点,小机器 50 个。虚拟节点数本身就是权重。

4. 备份机制 ​

每个 key 顺时针存到前 N 个节点(N=2 / 3),提供容灾。任一节点挂了,数据还在备份节点上。

八、Redis Cluster 的另一种"一致性哈希" ​

Redis Cluster 不用经典一致性哈希,而是用 16384 个固定哈希槽(slot):

key → CRC16(key) % 16384 → slot 编号
slot → 分配给某个节点

为什么不用一致性哈希环:

  • 集群规模通常不大(几十台),16384 个槽已经够细
  • slot 是离散且明确的——节点上线 / 下线时,用工具显式迁移指定的 slot,不像环上"那段数据"难精确定位
  • 运维更可控——能清楚知道哪些 slot 在哪个节点

本质上"slot + 节点"还是一致性哈希思想的离散化:加节点时只迁移部分 slot,不影响其他。

九、一致性哈希 vs 普通取模 ​

维度取模一致性哈希
实现复杂度极简 hash(k) % N中等(环 + 二分)
加节点影响几乎全部数据1/N 的数据
负载均衡完美均匀需要虚拟节点保证
适合场景N 永远不变节点动态变

实战:

  • N 完全不变(如分库,数据库数量极少改)→ 取模简单 + 直观
  • N 经常变(缓存节点上下线、集群动态扩容)→ 一致性哈希

十、其他用途 ​

1. 分布式缓存(memcached / Redis) ​

最经典应用。Twemproxy、Codis 等代理层用一致性哈希分发。

2. CDN ​

CDN 内部多个边缘节点,用一致性哈希决定回源到哪个节点的内存——保证同一 url 总是落在同一个节点,提高缓存命中率。

3. 分库分表 ​

分库 = hash(uid) → shard,扩容时一致性哈希让搬家成本可控。

4. P2P 网络(DHT) ​

Chord、Kademlia 这些 DHT 协议本质就是一致性哈希。BitTorrent、IPFS 用 DHT 找资源。

5. 负载均衡(Session 黏性) ​

按 user 哈希分发到固定后端,保证 session 一致——比纯轮询更适合有状态服务。

十一、一致性哈希解决不了的事 ​

它只解决"路由"问题,不解决"数据迁移"问题。

加节点时,虽然只有 1/N 的 key 归属变了,但这些 key 的真实数据还在旧节点上。得有人去搬数据。

搬运策略:

  • 预热:新节点上线后,先复制部分热数据,避免冷启动 cache miss
  • 流量切换:慢慢把读流量切到新节点,确认稳定再切写
  • 双写:迁移期内新旧节点都写,验证一致后切

生产中真正难的是迁移过程的一致性,不是路由本身。

十二、一句话总结 ​

一致性哈希用"环"把"加节点 = 全部重排"的痛点降到"加节点 = 1/N 重排"。配合虚拟节点解决分布不均,几乎是分布式系统的标配工具。记住核心:节点和 key 都哈希到环上,数据归"顺时针第一个节点"——简单到一句话,却撑起了整个分布式缓存的世界。
