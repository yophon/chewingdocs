# 时间与空间复杂度

大 O 记号是工程师的**第二母语**——你跟人讨论性能,不用"慢不慢",用 O(n) / O(log n) / O(n²)。但 95% 的人对大 O 的理解停留在"背出常见几个",遇到稍微拐弯的场景就懵:为什么 HashMap 说"平均 O(1) 最坏 O(n)"?为什么动态数组的 push 是"均摊 O(1)"?为什么有些算法"期望 O(n log n) 最坏 O(n²)"?这一篇讲透。

> 一句话先记住:**大 O 不是公式,是一个关于"数据量变大时,耗时涨得多快"的粗略描述**。常数、低阶项都被吃掉,只剩**增长趋势**。理解这一点,90% 的复杂度题都能口算。

---

## 一、大 O 到底说了什么

定义上,`f(n) = O(g(n))` 意思是:**存在某个 n₀,从 n₀ 开始,f(n) 永远不会超过 c·g(n)**(c 是某个常数)。翻译成人话:

- 只看**数据量变大时的增长率**
- 不看**常数倍数**(100n 和 n 都是 O(n))
- 不看**低阶项**(n² + n 就是 O(n²),n 那一项被吃掉)

> 为什么能忽略常数?因为**常数是跟硬件、实现有关的,跟算法本身无关**。一台机器跑 100n,另一台跑 10n,都是 O(n)——你换机器能解决的事,不是算法层面的问题。

**直觉锚**:

| n 增长 10 倍 | 耗时变化 |
| --- | --- |
| O(1) | 不变 |
| O(log n) | +1 左右(就多跑一两轮) |
| O(n) | ×10 |
| O(n log n) | ×约 11(log n 那里多加 1) |
| O(n²) | ×100 |
| O(2ⁿ) | ×10¹⁰⁰⁰(别玩) |

**工程直觉**:算法面前,n = 10⁶ 是"能接受"的分界线:
- O(n) / O(n log n):10⁶ 一秒内能跑完,**安全**
- O(n²):10⁶ 平方是 10¹²,**绝对跑不完**

所以看到 n = 10⁶ 的输入,你必须**O(n log n) 以内**,这是硬指标。

## 二、常见复杂度对照与"从哪来"

| 复杂度 | 典型来源 |
| --- | --- |
| O(1) | 哈希查、数组下标取、栈 push/pop |
| O(log n) | 二分、平衡树、堆操作 |
| O(n) | 一次遍历、双指针 |
| O(n log n) | 排序、分治、每个元素做 log 次操作 |
| O(n²) | 双层循环、冒泡 / 选择排序 |
| O(n · m) | 两个数组 / 字符串交叉遍历 |
| O(2ⁿ) | 子集枚举、回溯 |
| O(n!) | 全排列 |

**一个识别信号**:看到"排序后扫一遍",基本是 O(n log n);看到"嵌套循环",警惕 O(n²);看到"枚举子集",大概率 O(2ⁿ) 配合剪枝。

## 三、最坏 vs 平均 vs 均摊 vs 期望

这四个词经常被混用,**但它们说的不是一回事**。

### 最坏复杂度(Worst Case)

**所有输入里最慢的那次**。快排最坏 O(n²)(每次都挑到最大或最小作基准)。
工程上最常用,因为**生产环境上限决定了你敢不敢用**。

### 平均复杂度(Average Case)

**所有输入的数学期望**。需要假设输入分布(通常假设均匀随机)。
这个值有用但也容易骗人——**真实世界的输入从来不是均匀随机的**。

### 均摊复杂度(Amortized)

**一系列操作的总时间,除以操作次数**。单次可能很慢,但平均下来稳定。

经典例子是**动态数组 push**:
- 大多数 push 是 O(1)
- 偶尔触发扩容,那一次是 O(n)(拷贝整个数组)
- 但扩容不是每次都发生——容量翻倍后,下次扩容要等 n/2 次 push
- 总共 push n 次,累计耗时 ≈ 2n,均摊 **O(1)**

均摊和平均的区别:**均摊是对时间的平均,不管输入是什么**;平均是对输入的平均。均摊更"硬",因为它是**确定性的上界**。

### 期望复杂度(Expected)

**算法内部使用随机性时的期望值**。随机化快排期望 O(n log n),最坏还是 O(n²),但因为基准是随机选的,**构造出最坏情况的概率极低**,所以工程上当 O(n log n) 用。

> **为什么哈希表是"平均 O(1) 最坏 O(n)" 不矛盾**:哈希冲突严重时,链表退化,查找就变 O(n)。但在良好哈希函数 + 合理负载因子下,冲突概率极低,**期望**就是 O(1)。Java 8 的 HashMap 还做了一层保险:链表长度超过 8 自动转红黑树,**把最坏从 O(n) 拉回 O(log n)**。

## 四、空间复杂度经常被忽视

面试和业务里**空间复杂度经常被忽略**,但它能决定程序跑不跑得起来。

常见陷阱:
- **递归的隐形栈空间**:一个看起来只占 O(1) 额外空间的递归,其实栈深度是 O(n),空间是 O(n)
- **JS 里的 `arr.slice()`**:看起来写得优雅,**每次都拷贝一遍**,循环里滥用就 O(n²) 空间
- **不必要的中间数组**:`arr.map(...).filter(...).reduce(...)` 每步一个新数组,**流式处理可以避免**

**空间换时间 vs 时间换空间**:两种思路在算法里反复出现。
- 空间换时间:哈希表、DP 表、前缀和、缓存
- 时间换空间:流式处理、位运算、原地算法

工程决策标准:**内存贵的时候换时间,计算贵的时候换空间**。现代服务器内存便宜,大多数时候是**空间换时间**。

## 五、递归的复杂度怎么算

递归算复杂度要考虑两件事:**每层干多少活、总共多少层**。

```
T(n) = 子问题总时间 + 本层合并时间
```

典型模式:

| 递推式 | 复杂度 | 例子 |
| --- | --- | --- |
| T(n) = T(n/2) + O(1) | O(log n) | 二分 |
| T(n) = T(n-1) + O(1) | O(n) | 简单递归 |
| T(n) = 2·T(n/2) + O(n) | O(n log n) | 归并排序 |
| T(n) = 2·T(n/2) + O(1) | O(n) | 二叉树遍历 |
| T(n) = T(n-1) + O(n) | O(n²) | 选择排序式递归 |
| T(n) = 2·T(n-1) + O(1) | O(2ⁿ) | 不剪枝的枚举 |

**不用背主定理**,记住几个经典模式,剩下的靠类比就够用了。

## 六、工程视角的"真实复杂度"

大 O 只是**理论模型**,工程上还有几个"大 O 看不到"的因素:

### 1. 缓存命中率(常数差 10 倍)

连续访问数组 vs 链表,理论都是 O(n),**实际链表慢 5-10 倍**——因为数组是连续内存,对 CPU 缓存极其友好;链表每跳一次都是 cache miss。

### 2. 常数因子

两个 O(n log n) 算法,常数差 3 倍很常见。**归并排序稳定但常数大,快排常数小是业界默认**。

### 3. n 很小的时候

O(n²) 在 n < 50 可能比 O(n log n) 快,因为常数小、没递归调用开销。
**标准库的排序(如 JDK 的 Arrays.sort)在小数组上切换成插入排序**,就是这个原因。

### 4. 分支预测 / SIMD

现代 CPU 对**有规律的循环**做并行指令。算法写得对不对分支预测友好,能影响性能 2-3 倍。

> **结论**:大 O 定方向(选 O(n log n) 还是 O(n²)),**常数定细节**(同级别里选哪种实现)。工程师两个都要看。

## 七、快速估复杂度的套路

给你一段代码,30 秒估出复杂度,套路:

1. **看最深层循环**,最深嵌套 k 层就是 O(n^k) 起步
2. **看循环里有没有 log 操作**(二分 / 堆 / 平衡树),有就乘 log n
3. **看递归**,T(n) 形式套上面的表
4. **看数据结构操作**:HashMap 是 O(1)、TreeMap 是 O(log n)、排序是 O(n log n)

**反过来估**:你要做的事上限是 10⁶,那你必须控制在 O(n log n),有嵌套循环就危险;上限是 10⁴,O(n²) 还能活;上限是 20,O(2ⁿ) 都能玩。

```
n ≤ 10       →  O(n!) 可以
n ≤ 25       →  O(2ⁿ) 可以
n ≤ 100      →  O(n⁴) 可以
n ≤ 500      →  O(n³) 可以
n ≤ 5000     →  O(n²) 可以
n ≤ 10⁶      →  O(n log n) 必须
n ≤ 10⁸      →  O(n) 必须
n > 10⁸      →  O(log n) 或 O(1)
```

**把这张表印在脑子里**,面试看到数据范围就能倒推算法类别。

## 八、一句话总结

> **大 O 看增长趋势,均摊看操作序列,期望看随机性,最坏看你敢不敢上线**。这四个词分清,你已经比 80% 的工程师懂复杂度了。
