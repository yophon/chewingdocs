# 调度算法

调度算法是**藏在每个程序背后的"看不见的手"**——OS 怎么决定下一个跑哪个进程、定时任务框架怎么决定下一个该执行哪个任务、Goroutine 怎么被分配到 CPU 核——背后都是调度算法。这一篇讲三个**最实用的调度模型**:**时间轮**(Kafka / Netty 用)、**优先队列调度**(Linux CFS)、**工作窃取**(Go / Fork-Join 用)。

> 一句话先记住:**调度算法的核心问题是"在 N 个候选任务里,以最低的成本选出下一个执行的那个"**。不同场景的"成本最低"不一样——海量定时任务用时间轮 O(1) 推进、CPU 时间片用优先队列保公平、多核并行用工作窃取防空闲。

---

## 一、最朴素的调度:优先队列

**有 N 个任务,每个有"优先级"或"下次执行时间",每次取最该执行的那个**。

数据结构:**最小堆(按时间戳)**。

```typescript
class TimerScheduler {
  private heap = new MinHeap<{ time: number, task: () => void }>(
    (a, b) => a.time - b.time
  )

  schedule(delayMs: number, task: () => void): void {
    this.heap.push({ time: Date.now() + delayMs, task })
  }

  start(): void {
    setInterval(() => {
      const now = Date.now()
      while (this.heap.size() > 0 && this.heap.peek()!.time <= now) {
        this.heap.pop()!.task()
      }
    }, 1)  // 1ms tick
  }
}
```

**复杂度**:
- 添加:O(log N)
- 取下一个:O(log N)
- 总开销 O(N log N) 处理 N 个任务

**问题**:**任务太多时(百万级)** O(log N) 也吃不消,而且每个 tick 要扫整个堆顶。

**Java `ScheduledThreadPoolExecutor` 用的就是堆**。中等量任务足够。

## 二、时间轮(Hashed Wheel Timer):海量定时任务的杀手

**比喻**:一个钟表,**指针每秒跳一格**。每格挂着"该这一秒触发的任务列表"。

```
        0s
        |
   59s ─┼─ 1s
       /│\
      / │ \
   58s  ↓  2s
        ...
       30s
```

### 操作

- **添加任务**(delay = 5 秒):算出"5 格之后的位置",挂到那一格的任务列表。**O(1)**
- **指针跳一格**:执行那一格上的所有任务。**O(那格的任务数)**

**总平摊复杂度 O(1)** —— 添加和触发都是常数时间。

### 时间轮的局限

**最大延迟受轮的大小限制**——60 格只能存最多 60 秒的任务。

### 解决:层级时间轮(Hierarchical Timing Wheel)

像钟表的"秒-分-时"层级:

```
秒轮:60 格,每格 1 秒(60 秒覆盖)
分轮:60 格,每格 60 秒(1 小时覆盖)
时轮:24 格,每格 1 小时(1 天覆盖)
```

- **delay < 60s** → 放秒轮
- **60s ≤ delay < 1h** → 放分轮
- 分轮指针走一格时,**把这一格的所有任务"降级"放到秒轮对应位置**

**Kafka 的 `TimingWheel` 就是这个,Netty 的 `HashedWheelTimer` 也类似**。

### 实现要点

```typescript
class TimingWheel {
  private wheel: ((() => void)[])[]  // 每格一个任务列表
  private cursor = 0
  private tickMs: number

  constructor(slots: number, tickMs: number) {
    this.wheel = Array.from({ length: slots }, () => [])
    this.tickMs = tickMs
  }

  add(delayMs: number, task: () => void): void {
    const ticks = Math.floor(delayMs / this.tickMs)
    if (ticks >= this.wheel.length) throw new Error("超出时间轮范围,需要层级")
    const slot = (this.cursor + ticks) % this.wheel.length
    this.wheel[slot].push(task)
  }

  start(): void {
    setInterval(() => {
      const tasks = this.wheel[this.cursor]
      this.wheel[this.cursor] = []
      tasks.forEach(t => t())
      this.cursor = (this.cursor + 1) % this.wheel.length
    }, this.tickMs)
  }
}
```

**O(1) 添加,O(1) 触发(均摊)**。百万级定时任务无压力。

## 三、CFS(Linux 完全公平调度器)

Linux 内核 2.6.23 之后用 CFS。**用红黑树**按"虚拟运行时间"vruntime 排序,**总是调度 vruntime 最小的进程**。

```
红黑树:
              [vruntime=15]
              /            \
        [vruntime=10]   [vruntime=20]
        /        \
   [vruntime=5] [vruntime=12]
            ↑
          下一个调度的进程
```

**vruntime** = 进程实际运行时间 / 权重。优先级越高(权重大),vruntime 涨得越慢,**调度机会越多**。

**操作复杂度**:
- 找下一个调度:O(log n)
- 进程入队:O(log n)

**为什么红黑树而不是堆**:除了"取最小",还要支持"按 vruntime 找一个进程位置"和"删除任意进程"——堆做不到这些。

## 四、工作窃取(Work-Stealing):多核并行的灵魂

**问题**:N 个 CPU 核,M 个任务。最朴素的"全局队列"做法:所有核去抢同一个队列,**锁竞争剧烈**。

**解决**:**每个核一个本地队列**。本地队列空了就**从别的核"偷"任务**。

```
Core 0:  [任务1, 任务2, 任务3]
Core 1:  [任务4]
Core 2:  []
Core 3:  []  ← 空了,去 Core 0 偷
```

### 关键设计

1. **本地队列**:每核独占,**push / pop 自己的尾部**(后进先出,缓存友好)
2. **窃取**:其他核空闲时,**从别人的头部偷**(先进先出,跟自己 pop 错开,减少冲突)
3. **双端队列(Deque)**:本端 push/pop,远端 steal——**典型的并发数据结构**

### 应用

- **Go 调度器(GMP 模型)**:每个 P 一个本地任务队列,空了从别的 P 偷一半
- **Java ForkJoinPool**:`Stream.parallel()` / CompletableFuture 底层
- **Rayon (Rust)**:数据并行库
- **Cilk**:工作窃取的发明者

**优势**:**核多时几乎线性扩展**,负载自动均衡。**比集中式调度强一个量级**。

## 五、几种调度算法的对比

| 场景 | 算法 | 复杂度 |
| --- | --- | --- |
| 中等定时任务 | 堆 | O(log N) 添加 / 触发 |
| 海量定时任务 | 时间轮 | O(1) 添加 / 触发 |
| 进程时间片 | 红黑树 | O(log N) |
| 多核任务并行 | 工作窃取 | 接近 O(1),空闲核主动偷 |
| 任务带优先级 | 优先队列 / 多级反馈队列 | O(log N) |

## 六、调度的常见考量

### 1. 公平性 vs 吞吐

- **公平优先**:CFS 的 vruntime 设计,每个进程都能跑
- **吞吐优先**:批处理(Hadoop / Spark)按 throughput 优化

### 2. 实时性 vs 平均延迟

- **硬实时**:航天器、医疗设备——任务必须在 deadline 前完成。EDF(Earliest Deadline First)
- **软实时**:网络包处理——尽量快但不致命

### 3. 抢占 vs 协作

- **抢占式**:OS 可以打断进程换上别的(Linux / Windows)
- **协作式**:任务自己 yield(早期 macOS、JS 单线程的 Promise / async)

### 4. 反饥饿

低优先级任务**等太久应该被提优先级**,否则永远跑不到——MLFQ(多级反馈队列)的核心。

## 七、调度在工程里的真实身影

### 1. Kafka 的延迟队列

**TimingWheel**——百万 producer 的延迟消息处理。

### 2. Netty 的 HashedWheelTimer

网络框架里的超时管理:每个连接一个超时定时器,**百万连接全部用时间轮管**。

### 3. Quartz / Spring Scheduler

应用层定时任务调度。**通常用堆**(任务量小)。

### 4. Cron 表达式调度器

**Cron + 时间轮 / 堆 的组合**。Cron 解析下次触发时间,放进调度器。

### 5. 数据库连接池(HikariCP)

任务排队 + 工作窃取的混合,**连接被借出后归还时按等待时间唤醒等的最久的请求**。

### 6. K8s 调度器

调度 Pod 到 Node。算法不是简单优先队列,而是**两阶段:filter(筛掉不能放的) + score(打分排序)**,有点像加权调度。

## 八、设计调度系统的几个原则

### 1. 选对核心数据结构

- 任务多 + 时间敏感 → 时间轮
- 任务少 + 优先级动态 → 堆
- 多核并行 → 工作窃取
- 公平调度 + 大量进程 → 红黑树

### 2. 控制锁粒度

调度器是热点,锁粒度大就成瓶颈。**本地化(per-core)是关键**。

### 3. 避免饥饿

任何调度策略都要有"等久了被提权"的机制,否则会饿死低优先级任务。

### 4. 可观测

调度器是看不见的——一定要有监控:**队列长度、平均等待时间、p99 响应时间、各 worker 利用率**。

---

## 九、一句话总结

> **调度算法的世界里没有银弹**:海量定时任务用时间轮、CPU 时间片用红黑树、多核并行用工作窃取、应用层任务用堆。**核心问题永远是"在 N 个候选里以最低成本选出下一个"**——选对结构,O(N log N) 能压成 O(1)。
