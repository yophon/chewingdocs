    字符串

    字符串题是面试重灾区,真实工程里也无处不在——搜索、日志分析、编辑器、URL 路由、敏感词过滤全是字符串算法。难点在于:朴素解法几乎都 O(n·m),高级解法都需要几个非常精妙的洞察。本篇讲透三大经典:KMP(线性子串匹配)、Rabin-Karp(滚动哈希)、Trie(前缀树,预告下一篇),再加字符串题目通用模板。

    一句话先记住:字符串就是"字符数组 + 一堆性质"——子串、前缀、后缀、回文、周期、匹配。每个"性质"都对应一套算法。暴力 O(n·m) 是底线,高级 O(n+m) 或 O(n) 靠预处理或哈希。

    一、朴素匹配:为什么慢

    问题:在文本 T 里找模式 P 第一次出现的位置。

    暴力:

    function naiveMatch(T: string, P: string): number {
  for (let i = 0; i <= T.length - P.length; i++) {
    let j = 0
    while (j < P.length && T[i + j] === P[j]) j++
    if (j === P.length) return i
  }
  return -1
}

    最坏 O(n·m):比如 T = "aaaaa...aaab",P = "aaab",每次都比到最后一位才失败。

    慢的原因:失配后,T 的指针白白退回去重比——但前面比过的信息完全浪费了。

    二、KMP:不让指针后退

    KMP(Knuth-Morris-Pratt)的核心思想:失配时,用已比过的信息决定 P 该跳多远,T 指针永远不回退。

    next 数组(失配函数)

    对模式 P,预处理一个 next 数组:next[i] = "P[0..i] 这段前缀的最长公共真前后缀长度"。

    例:P = "ababc"

    i:      0  1  2  3  4
P[i]:   a  b  a  b  c
next:   0  0  1  2  0
    • next[2] = 1:"aba" 的前缀 "a" = 后缀 "a",长度 1
    • next[3] = 2:"abab" 的前缀 "ab" = 后缀 "ab",长度 2

    匹配过程

    function kmp(T: string, P: string): number {
  const next = buildNext(P)
  let i = 0, j = 0
  while (i < T.length) {
    if (j === P.length) return i - j       // 命中
    if (T[i] === P[j]) { i++; j++ }         // 匹配,双指针前进
    else if (j > 0) j = next[j - 1]          // 失配,P 跳,T 不动
    else i++                                  // P 第一个就不匹配,T 前进
  }
  return j === P.length ? i - j : -1
}

    为什么是 O(n + m)

    • T 指针 i 单调递增,最多 n 次
    • 每次失配 j 减少,匹配时 j 增加;j 总增减次数被 i 的增长限制 → 也是 O(n)
    • next 数组构建 O(m)
    • 总计 O(n + m)

    实战里怎么用

    老实说,工程代码里手写 KMP 的机会很少——标准库的 indexOf / str.find 用的是 Boyer-Moore / Two-Way 算法(常数更优)。KMP 的价值在: 1. 面试会考 2. 理解字符串自身性质(前缀函数 / 失配跳跃的思想在其他算法里反复出现) 3. 变体应用:周期串判定、字符串拼接里找循环节

    三、Rabin-Karp:滚动哈希

    另一种思路:把字符串转成数字,比较数字就行。

    核心:滑动窗口 + 滚动哈希。窗口向右移一位时,不用重算整个 hash,用数学公式 O(1) 更新。

    hash(T[i+1..i+m]) = (hash(T[i..i+m-1]) - T[i] * base^(m-1)) * base + T[i+m]

    代码

    function rabinKarp(T: string, P: string): number {
  const m = P.length, n = T.length
  if (m > n) return -1

  const base = 256n
  const mod = (1n << 61n) - 1n   // 大素数,减少碰撞

  let baseM = 1n
  for (let i = 0; i < m - 1; i++) baseM = (baseM * base) % mod

  let hashP = 0n, hashT = 0n
  for (let i = 0; i < m; i++) {
    hashP = (hashP * base + BigInt(P.charCodeAt(i))) % mod
    hashT = (hashT * base + BigInt(T.charCodeAt(i))) % mod
  }

  for (let i = 0; i <= n - m; i++) {
    if (hashP === hashT && T.substr(i, m) === P) return i
    if (i < n - m) {
      hashT = (hashT - BigInt(T.charCodeAt(i)) * baseM + mod) % mod
      hashT = (hashT * base + BigInt(T.charCodeAt(i + m))) % mod
    }
  }
  return -1
}

    复杂度与应用

    • 期望 O(n + m),最坏 O(n·m)(所有 hash 都碰撞)
    • 实战比 KMP 好用的场景:多模式匹配。同时查 k 个模式,Rabin-Karp 只要存 k 个 hash 到 Set,滑窗时 O(1) 查存在;KMP 要各建 next 数组。

    四、字符串题通用模板

    1. 双指针 / 反转

    function reverseString(s: string[]): void {
  let l = 0, r = s.length - 1
  while (l < r) { [s[l], s[r]] = [s[r], s[l]]; l++; r-- }
}

    2. 滑动窗口(无重复字符最长子串)

    function lengthOfLongestSubstring(s: string): number {
  const seen = new Map<string, number>()
  let left = 0, max = 0
  for (let right = 0; right < s.length; right++) {
    if (seen.has(s[right]) && seen.get(s[right])! >= left) {
      left = seen.get(s[right])! + 1
    }
    seen.set(s[right], right)
    max = Math.max(max, right - left + 1)
  }
  return max
}

    模板骨架:

    扩大窗口 → 不满足条件时缩小左端 → 记录当前最优 → 循环

    滑动窗口篇会专讲。

    3. 回文判定:中心扩展

    function longestPalindrome(s: string): string {
  let start = 0, maxLen = 0
  function expand(l: number, r: number): void {
    while (l >= 0 && r < s.length && s[l] === s[r]) { l--; r++ }
    const len = r - l - 1
    if (len > maxLen) { maxLen = len; start = l + 1 }
  }
  for (let i = 0; i < s.length; i++) {
    expand(i, i)     // 奇数长度(单中心)
    expand(i, i + 1) // 偶数长度(双中心)
  }
  return s.substring(start, start + maxLen)
}

    每个位置试两次(单中心和双中心),O(n²)。更快的 Manacher 算法 O(n) 很精巧,工程用不到,除非面官特别问。

    4. 字符计数:固定大小数组比 HashMap 快

    function isAnagram(s: string, t: string): boolean {
  if (s.length !== t.length) return false
  const count = new Array(26).fill(0)
  for (let i = 0; i < s.length; i++) {
    count[s.charCodeAt(i) - 97]++
    count[t.charCodeAt(i) - 97]--
  }
  return count.every(c => c === 0)
}

    ASCII 小写字母只有 26 个,用定长数组比哈希快 3-5 倍。

    5. 前缀 / 后缀预处理

    倒着扫一遍,存每个位置往后的某个信息(最近的字母位置、后缀和等),然后正着扫使用——这是很多字符串题的典型套路。

    五、字符串的编码陷阱

    工程里比算法更容易翻车的是编码。

    UTF-8 / UTF-16 / Unicode 代码点

    "a".length        // 1
"α".length        // 1
"😀".length       // 2  ← 什么鬼?
"a̐".length      // 2  ← 也是?

    JS 的字符串长度是 UTF-16 code unit 数,不是字符数。emoji 和组合字符会让你的算法 bug。

    正确统计字符数:

    [..."😀"].length  // 1

    for...of 按代码点迭代。但即使这样,"a̐" 还是 2 个代码点(基础字符 + 组合符号),真正按"看起来的字符数"要用 Intl.Segmenter。

    实战建议

    • 业务代码里做文本处理,慎用 charAt / [] / length
    • ASCII-only 场景(比如处理 URL、JSON key)随意
    • 面算法题,默认所有字符都在 ASCII 范围,除非题目明说 Unicode

    六、字符串 + DP = 一大类题

    • 最长公共子序列(LCS):两字符串 → 二维 DP
    • 编辑距离(Edit Distance):增删改三种操作 → 二维 DP
    • 最长回文子序列:区间 DP
    • 通配符匹配 / 正则匹配:状态机 + DP
    • 最长重复子串:二分 + 滚动哈希

    这类题DP 篇会专门讲。字符串 DP 的状态定义是难点:dp[i][j] 通常代表"两个字符串前缀 s[0..i] 和 t[0..j] 的某个性质"。

    七、Trie 预告

    Trie(字典树)是字符串高频场景的终极答案: - 前缀匹配(自动补全) - 多模式匹配(AC 自动机 = Trie + KMP) - 敏感词过滤 - IP 路由表(压缩 Trie)

    结构:

           root
      /  |  \
     a   b   c
    /|   |
   p b   e
   |
   p
   l
   e
   (end)  ← "apple"

    第 12 篇专讲。重点是它的"共享前缀"特性,省空间也省时间。

    八、一句话总结

    字符串本质是字符数组,难的是利用它的"结构"——前缀相等、回文对称、周期重复。KMP 用 next 数组避免回退,Rabin-Karp 用滚动哈希 O(1) 滑窗,Trie 用树结构共享前缀。每一个都是"用预处理换时间"的典范。

    下一篇进入树的世界,先从二叉树开始——遍历四法、递归和迭代的统一框架、Morris 遍历 O(1) 空间。
