# 数组与动态数组

数组是**最常用、最被低估**的数据结构。大多数人把它当"装东西的桶",其实它是 **CPU 和内存交响协作的产物**——连续内存 + O(1) 下标访问 + 缓存极度友好。理解数组到位,你对所有高级数据结构(哈希 / 堆 / 动态数组)的"底层为什么这么设计"都会豁然开朗。

> 一句话先记住:**数组的超能力是"O(1) 下标取",代价是"插删中间要 O(n) 搬东西"**。选数据结构时想一步:我查得多还是改得多?——答案决定了你选数组还是链表/哈希。

---

## 一、数组的物理模型:连续内存 + 固定步长

数组在内存里长这样:

```
地址   1000  1004  1008  1012  1016  1020
值     [10]  [20]  [30]  [40]  [50]  [60]
下标    0     1     2     3     4     5
```

**三个关键事实**:

1. **一块连续的内存**
2. **每个元素大小固定**(int 4 字节,对象引用 8 字节)
3. **下标访问靠数学**:`addr(arr[i]) = base + i × size`

这就是为什么 `arr[i]` 是 O(1)——不是"快",是**一次地址计算就到位,根本不需要遍历**。

---

## 二、数组的"CPU 缓存友好性":一个 10 倍性能差

这是数组相对链表**最大的实战优势**,也是算法书几乎不讲的。

现代 CPU 访问内存有多层缓存:

```
寄存器     →  1 cycle
L1 cache   →  ~4 cycles
L2 cache   →  ~12 cycles
L3 cache   →  ~40 cycles
主内存     →  ~300 cycles   (差 75 倍!)
```

CPU 每次从内存读数据,不是只读 4 字节,而是读**一整个 cache line**(通常 64 字节,能装 16 个 int)。

**结果**:
- 遍历数组时,读 `arr[0]`,顺带把 `arr[1]..arr[15]` 全加载到 cache,后面 15 次访问全是 L1 命中
- 遍历链表时,每跳一次都是随机地址,cache miss,每次 300 cycles

**实测:同样 100 万元素求和,数组比链表快 5-10 倍**。理论复杂度都是 O(n),但"常数"差一个数量级。

> 这也是为什么 Java 8 的 HashMap 冲突用红黑树而不是链表的原因之一——**红黑树的遍历虽然复杂度更高,但因为节点少、访问模式可预测,缓存表现比长链表更好**。

## 三、静态数组 vs 动态数组

**静态数组**:创建时就定长,满了就没办法(C 的数组、Java 的 `int[]`)。
**动态数组**:能自动扩容(JS 的 `Array`、Java 的 `ArrayList`、Python 的 `list`、C++ 的 `vector`)。

动态数组是**静态数组加一个"容量管理层"**。看实现:

```typescript
class DynamicArray<T> {
  private data: T[]
  private size: number       // 实际元素数
  private capacity: number   // 数组总容量

  constructor(initialCapacity = 8) {
    this.capacity = initialCapacity
    this.data = new Array(initialCapacity)
    this.size = 0
  }

  push(val: T): void {
    if (this.size === this.capacity) {
      this.grow()  // 满了就扩容
    }
    this.data[this.size++] = val
  }

  private grow(): void {
    this.capacity *= 2  // 关键:翻倍!
    const newData = new Array(this.capacity)
    for (let i = 0; i < this.size; i++) {
      newData[i] = this.data[i]
    }
    this.data = newData
  }

  get(i: number): T {
    if (i < 0 || i >= this.size) throw new RangeError()
    return this.data[i]
  }
}
```

**关键设计是扩容策略**:不是"满了加 1",而是**翻倍**。

## 四、为什么是"翻倍"而不是"加固定值"

这里是所有新手搞不懂、老手装懂的地方。**策略选错,push 从均摊 O(1) 变成 O(n)**。

### 方案 A:每次扩容 +10

- push 10 次:不扩容
- push 第 11 次:扩容到 20,拷贝 10 个
- push 第 21 次:扩容到 30,拷贝 20 个
- ...
- push n 次:扩容总次数 n/10,**累计拷贝 10 + 20 + 30 + ... + n ≈ n²/20**
- **均摊:O(n)**,不是 O(1)

### 方案 B:每次扩容 ×2

- 容量 8 → 16 → 32 → 64 → 128 → ...
- push n 次,扩容共 log n 次,**累计拷贝 1 + 2 + 4 + ... + n/2 + n < 2n**
- **均摊:2n / n = O(1)** ✓

**翻倍扩容的核心思想**:**拷贝的代价被后续"免费 push"平摊**。每次扩容后,后面的 n 次 push 都不会再扩容,摊下来每次只分到 2 次基本操作。

**Java 的 ArrayList 是 1.5 倍,Python 是约 1.125 倍 + 补偿**——都是翻倍量级,只是为了节省一点空间。**真正能"+10" 扩容的,是代码里的 bug**。

## 五、最常见的三个"O(n²) 性能坑"

### 坑一:循环里 `arr.unshift(x)`

```typescript
const arr: number[] = []
for (let i = 0; i < n; i++) {
  arr.unshift(i)  // 💥 每次都要把整个数组往后搬一位
}
// 总复杂度 O(n²)
```

**原因**:数组头部插入,后面所有元素都要向后挪一位。
**修复**:先 `push`,最后 `reverse`。或者用**双端队列**(Deque)。

### 坑二:循环里 `arr.splice(0, 1)`

```typescript
for (let i = 0; i < n; i++) {
  const head = arr.splice(0, 1)[0]
  process(head)
}
// 💥 O(n²)
```

**修复**:用下标推进,别真从数组里删。

```typescript
let head = 0
while (head < arr.length) {
  process(arr[head])
  head++
}
```

这就是队列"假删除"的套路——**只动指针,不动数据**。

### 坑三:循环里 `arr + otherArr`(拼接)

```typescript
let result: number[] = []
for (const batch of batches) {
  result = result.concat(batch)  // 💥 每次都新建数组
}
// O(n²) 空间 + 时间
```

**修复**:`result.push(...batch)` 或 `result = [...result, ...batch]` ❌(一样慢),或 `for` 循环 push。最优解:**一次 flat**。

```typescript
const result = batches.flat()
```

## 六、数组常用技巧速查

### 前缀和

```typescript
const prefix = [0]
for (let i = 0; i < nums.length; i++) {
  prefix.push(prefix[i] + nums[i])
}
// 区间和 [l, r] = prefix[r+1] - prefix[l]
```

**用途**:多次查询区间和,**预处理 O(n),查询 O(1)**。

### 差分数组

```typescript
const diff = new Array(n).fill(0)
// 对区间 [l, r] 统一加 v
diff[l] += v
if (r + 1 < n) diff[r + 1] -= v
// 最后做前缀和就是结果
```

**用途**:批量区间加减,**单次更新 O(1)**。

### 双指针

- **快慢指针**:查找中位、检测环
- **对撞指针**:有序数组找和、反转
- **滑动窗口**:子串 / 子数组问题

后面有独立一篇细讲。

### 原地操作

```typescript
// 原地反转
function reverse(arr: number[]) {
  let l = 0, r = arr.length - 1
  while (l < r) {
    [arr[l], arr[r]] = [arr[r], arr[l]]
    l++; r--
  }
}
```

**O(1) 空间**,比 `arr.slice().reverse()` 省一次拷贝。

## 七、数组的"隐藏坑"

### JS 的"稀疏数组"

```javascript
const arr = []
arr[1000] = 1  // 中间全是 "空洞"
console.log(arr.length)  // 1001
```

JS 允许不连续下标,引擎内部会**退化成哈希表模式**,访问慢几倍。**不要刻意跳下标赋值**。

### 负数下标

- Python: `arr[-1]` 是最后一个
- JS / Java: `arr[-1]` 是 `undefined` / 报错

写跨语言代码时注意。

### 浅拷贝

```javascript
const a = [[1, 2], [3, 4]]
const b = [...a]  // 浅拷贝!
b[0][0] = 999
console.log(a[0][0])  // 999 — a 也变了
```

**二维数组要用 `a.map(row => [...row])` 深拷贝一层**,或者结构化克隆 `structuredClone(a)`。

## 八、选型决策表

| 需求 | 选啥 | 理由 |
| --- | --- | --- |
| 随机访问多 | 数组 | O(1) 下标 |
| 尾部增删多 | 数组 | push/pop 都 O(1) |
| 头部 / 中间增删多 | 链表 / Deque | 避免 O(n) 搬运 |
| 频繁查找元素 | Set / Map | 避免 O(n) 扫描 |
| 范围查询多 | 前缀和 / 差分 / 线段树 | 预处理换查询速度 |
| 长度固定、极度追求性能 | 静态数组(C / TypedArray) | 避开动态扩容 |

**JS 特别提醒**:追求性能时用 `Int32Array` / `Float64Array` 等 TypedArray——**固定类型、连续内存,比普通 Array 快一倍以上**。

---

## 九、一句话总结

> **数组是线性结构里最快的,只要你别在头部动它**。扩容均摊、缓存友好、O(1) 下标——这三点决定了数组是默认选项,其他结构都是在解"数组某个短板"时诞生的。

下一篇讲**链表**——数组的对偶面:头插 O(1)、随机访问 O(n)、永远出现在面试题、真实工程里用得比想象中少。
