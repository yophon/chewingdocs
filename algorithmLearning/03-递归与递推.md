# 递归与递推

递归是算法里最容易**学会但用错**的工具。会写阶乘不算会递归;**能看出一道题该不该用递归、递归写完能不能翻译成迭代、递归会不会栈溢出**——这才是工程师要的。递推是递归的"反面"——同样的转移关系,从底往上推,不用栈、不会爆。这一篇把两者讲透。

> 一句话先记住:**递归就是把"自己"压到系统栈上的迭代**——你眼睛看到的是"函数调自己",CPU 看到的是"push 参数 → 跳转 → 返回 → pop"。理解这一点,递归就没任何神秘感,而且你知道**什么时候它会爆栈**。

---

## 一、递归的三个必要条件

能写递归,必须满足:

1. **能把原问题拆成更小的同类子问题**(否则递归回不来)
2. **有明确的终止条件**(否则死递归)
3. **子问题的解能组合出原问题的解**(否则拆了也白拆)

三条缺一不可。看一个反例——"计算 1 + 2 + 3 + ... + n":

```typescript
function sum(n: number): number {
  if (n === 0) return 0        // 终止条件 ✓
  return n + sum(n - 1)         // 拆成更小的同类问题 ✓ + 组合 ✓
}
```

子问题是 `sum(n-1)`,比原问题小 1,终止在 `n=0`。**三条都满足,递归成立**。

---

## 二、递归的执行模型:系统栈

这是理解递归的**唯一关键**。`sum(3)` 执行时,系统栈长这样:

```
调用 sum(3):
  栈:[sum(3)]
  → 需要 sum(2) 才能算

调用 sum(2):
  栈:[sum(3), sum(2)]
  → 需要 sum(1) 才能算

调用 sum(1):
  栈:[sum(3), sum(2), sum(1)]
  → 需要 sum(0) 才能算

调用 sum(0):
  栈:[sum(3), sum(2), sum(1), sum(0)]
  → 返回 0

回到 sum(1):1 + 0 = 1,弹栈
回到 sum(2):2 + 1 = 3,弹栈
回到 sum(3):3 + 3 = 6,弹栈
```

**每层调用都占一个栈帧**,存参数、局部变量、返回地址。这就是为什么:
- 递归深度 n 的空间复杂度是 O(n),不是 O(1)
- 深度过大会**栈溢出**(JS 引擎通常一万多层就炸)
- 递归慢是因为**函数调用本身有开销**,迭代没有

## 三、自顶向下 vs 自底向上

同一个问题,两种写法:

**自顶向下(递归)**:
```typescript
function fib(n: number): number {
  if (n <= 1) return n
  return fib(n - 1) + fib(n - 2)
}
```

**自底向上(递推)**:
```typescript
function fib(n: number): number {
  if (n <= 1) return n
  let a = 0, b = 1
  for (let i = 2; i <= n; i++) {
    [a, b] = [b, a + b]
  }
  return b
}
```

**两种哪种好?看场景**:

| 维度 | 自顶向下 | 自底向上 |
| --- | --- | --- |
| 代码可读性 | 更直观(和定义一样) | 需要反过来想 |
| 性能 | 慢(函数调用开销 + 可能重复计算) | 快 |
| 栈溢出 | n 大时会炸 | 不会 |
| 实现难度 | 简单 | 有时难想到遍历顺序 |

**工程上 80% 场景选自底向上**——DP 基本都这么写。递归写着爽,但生产代码不敢用,尤其 JS / Python 栈小。

## 四、朴素递归的两个致命问题

### 问题一:指数级重复计算

`fib(5)` 看着只调几次,实际:

```
                fib(5)
              /        \
          fib(4)       fib(3)
          /    \        /    \
       fib(3) fib(2) fib(2) fib(1)
       /   \
    fib(2) fib(1)
    ...
```

`fib(3)` 算了 2 次、`fib(2)` 算了 3 次、`fib(1)` 算了 5 次——**指数爆炸**,复杂度 O(2ⁿ)。

**解法**:**记忆化**(顶向下 DP):把算过的缓存起来。

```typescript
function fib(n: number): number {
  const memo = new Map<number, number>()
  function helper(n: number): number {
    if (n <= 1) return n
    if (memo.has(n)) return memo.get(n)!
    const result = helper(n - 1) + helper(n - 2)
    memo.set(n, result)
    return result
  }
  return helper(n)
}
```

复杂度从 O(2ⁿ) 降到 O(n)——**只是加了个哈希表**。这是 DP 的第一层:**记忆化搜索**。

### 问题二:栈溢出

```typescript
function sum(n: number): number {
  if (n === 0) return 0
  return n + sum(n - 1)
}
sum(100000)  // 💥 Maximum call stack size exceeded
```

JS 没有尾递归优化(大多数引擎没实现),Python 默认栈深 1000,Java 大概几万。**递归深度是硬上限**。

**解法三选一**:
1. **改成迭代**(首选)
2. **改成手动栈**(用数组模拟系统栈)
3. **增大栈**(Python 可以 `sys.setrecursionlimit`,JS 基本没办法)

## 五、递归 ↔ 迭代 的通用转换

任何递归都能改成迭代,**因为递归只是系统栈做的事,你可以自己维护一个显式栈**。

**二叉树前序遍历——递归版**:

```typescript
function preorder(root: TreeNode | null, res: number[]) {
  if (!root) return
  res.push(root.val)
  preorder(root.left, res)
  preorder(root.right, res)
}
```

**改成迭代版**(手动栈):

```typescript
function preorder(root: TreeNode | null): number[] {
  const res: number[] = []
  if (!root) return res
  const stack: TreeNode[] = [root]
  while (stack.length > 0) {
    const node = stack.pop()!
    res.push(node.val)
    if (node.right) stack.push(node.right)  // 先压右
    if (node.left) stack.push(node.left)    // 后压左(先出)
  }
  return res
}
```

**套路**:
- 系统栈 → 你自己维护的数组
- 函数调用 → `stack.push`
- 函数返回 → `stack.pop`
- 局部变量 → 放到栈帧对象里

**什么时候必须手动改**:深度可能超过 10⁴、或者需要在"递归中途"保存其他状态(比如中断 / 迭代器)。

## 六、尾递归优化(你可能用不上)

**尾递归**:递归调用是函数的**最后一个动作**,返回值不做任何额外操作。

```typescript
// 不是尾递归(返回时要 n + ...)
function sum(n: number): number {
  if (n === 0) return 0
  return n + sum(n - 1)
}

// 是尾递归(返回时直接返回 helper 的结果)
function sum(n: number, acc = 0): number {
  if (n === 0) return acc
  return sum(n - 1, acc + n)
}
```

尾递归**理论上**可以被编译器优化成迭代(复用栈帧),不会爆栈。

**现实**:
- **Scheme / OCaml / Scala**:支持
- **Java**:不支持(JVM 字节码层面有限制)
- **Python**:Guido 明确拒绝
- **JavaScript**:ES6 规范要求,**但 V8 / SpiderMonkey 都没实装**,Safari 曾经支持又撤了

**结论**:工程里**不要依赖尾递归优化**。深度大就手动改成迭代,别寄希望于编译器。

## 七、递推式怎么写:状态 + 转移

递推和递归是同一个东西的两面。**先想递归,再翻译成递推**:

1. **定义状态**:`f(i)` 代表什么?(爬到第 i 级台阶的方法数)
2. **写出转移**:`f(i)` 怎么由更小的状态算出?(`f(i) = f(i-1) + f(i-2)`)
3. **定 base case**:最小状态的值(`f(0) = 1, f(1) = 1`)
4. **定遍历顺序**:从小到大(自底向上)就是递推

这四步就是 DP 的核心,后面 DP 篇会反复用。**递归让你理解,递推让你能跑**。

---

## 八、什么时候该用递归

| 场景 | 推荐 |
| --- | --- |
| 树 / 图遍历(深度可控) | ✅ 用递归,可读性碾压 |
| 分治(归并 / 快排) | ✅ 递归最自然 |
| 回溯(全排列 / 组合) | ✅ 递归必选 |
| 链表操作(反转 / 合并) | 都行,迭代空间更省 |
| 简单线性递推(斐波那契) | ❌ 改迭代 |
| 深度可能很大(n > 10⁴) | ❌ 改迭代或手动栈 |
| DP | ❌ 自底向上递推更稳 |

**判断准则**:看深度。深度 < 10³ 随意,10³~10⁴ 小心,> 10⁴ 别用。

---

## 九、一句话总结

> **递归是把系统栈当免费工具用,递推是自己管着栈往上堆**。两者解的同一类问题,选哪种看深度、性能、可读性——**能迭代就别递归**,这是工程默认选项。
