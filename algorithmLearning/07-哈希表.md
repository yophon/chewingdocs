# 哈希表

哈希表是**算法界的万金油**——"不会就哈希一发"是真·金句。两数之和、去重、计数、缓存、查存在——一半业务代码底层都是它。会用哈希是 L1,懂**冲突怎么处理、负载因子为什么 0.75、为什么 Java 8 HashMap 会树化**是 L2;到 L3 你能手写一个能跟 jdk 比肩的哈希表,并解释为啥 Redis 的哈希扩容是"渐进式"的。

> 一句话先记住:**哈希表 = 数组 + 哈希函数 + 冲突处理**。数组给 O(1) 下标,哈希函数把 key 映射到下标,冲突处理负责"同一个下标被多个 key 占了"。**三件事任何一件设计得差,你的 O(1) 就会塌成 O(n)**。

---

## 一、哈希表的直觉模型

想把"名字 → 电话"存下来,最笨的办法是数组里两两存:[["张三", "139..."], ["李四", "158..."]]。查找 O(n)。

哈希表的思路:**把名字通过一个函数算成数字下标**,直接存到数组对应位置。

```
"张三" → hash("张三") = 7   →  arr[7] = "139..."
"李四" → hash("李四") = 2   →  arr[2] = "158..."
```

查找也用同一个函数算下标,**O(1) 搞定**。

**三个子问题**立刻浮现:
1. 哈希函数怎么设计,才能均匀分布?
2. 两个 key 算出同一个下标(冲突)怎么办?
3. 数组多大?太大浪费、太小冲突多。

这三个就是哈希表的全部设计空间。

---

## 二、哈希函数:要快、要散

好哈希函数的两个标准:

1. **均匀分布**:不同输入散布到数组不同位置,冲突概率低
2. **计算快**:否则 O(1) 的优势就打折

常见做法:

**整数 key**:`h(k) = k % capacity`。简单,但如果 capacity 有因数,容易 cluster(比如 capacity=10,所有 10 倍数都撞)。
**字符串 key**:多项式哈希。Java 的 `String.hashCode` 是:

```java
// h = 31*h + c[i]   for each char
```

`31` 不是魔数——它是奇素数,乘法可以被 JVM 优化成位移减法(`31*x == (x << 5) - x`),**快且散得开**。

**扰动函数(Java 8 HashMap)**:

```java
hash = key.hashCode() ^ (key.hashCode() >>> 16)
```

**原因**:即使 `hashCode` 分布不错,计算下标时通常用 `hash & (n-1)`(capacity 是 2 的幂),**只有低几位起作用**。把高 16 位 xor 到低 16 位,让高位也参与下标计算,大幅降低冲突。

---

## 三、冲突处理:两大流派

再好的哈希函数也会冲突(**鸽笼原理**:n+1 个鸽子放 n 个笼子必然撞)。怎么处理是哈希表设计的核心。

### 流派一:拉链法(Separate Chaining)

**每个桶是一个链表**,冲突的元素挂在同一个链表里。

```
arr[2] →  ["李四", "王五"]   (冲突了,挂链表)
arr[3] →  ["张三"]
arr[7] →  null
```

- 查找:算出下标 → 遍历那个链表 → 找到或没找到
- 插入:算出下标 → 链表头插 O(1)(允许重复 key 的话)
- 删除:遍历链表删除

**Java 8 的优化**:链表长度 > 8 时**转红黑树**,退化情况从 O(n) 拉回 O(log n)。降到 6 又转回链表(避免抖动)。

**优点**:实现简单、扩容容易、删除方便。
**缺点**:链表慢(缓存不友好)、额外指针开销。

**JDK `HashMap` / Python `dict`(旧版)/ Go `map` 都用拉链**(Go 略做优化)。

### 流派二:开放寻址(Open Addressing)

**冲突了就往后找空位**。不另外开链表。

探测方式:
- **线性探测**:`h, h+1, h+2, ...`(简单,容易 cluster)
- **二次探测**:`h, h+1², h+2², h+3², ...`(减少 cluster)
- **双重哈希**:`h + i * h2(k)`(最均匀)

```
插入 "王五",hash=2,arr[2] 被"李四"占了
→ 看 arr[3]:被"张三"占
→ 看 arr[4]:空!放进去
```

**查找**:算出起始下标 → 从那个位置开始挨个往后找 → 直到找到 key 或碰到空。

**删除的坑**:不能直接置空!否则破坏探测链。要标记"墓碑"(tombstone)。

**优点**:缓存友好(连续数组)、无额外指针。
**缺点**:删除麻烦、扩容开销大、负载因子要更低。

**Python dict(新版)、Rust HashMap、Redis dict 都用开放寻址**。

---

## 四、负载因子:性能与空间的平衡

**负载因子 = 元素数 / 桶数**,决定什么时候该扩容。

- **低(0.25)**:冲突少,查找快,但浪费空间
- **高(0.9)**:空间省,但冲突爆炸,退化成 O(n)
- **Java HashMap: 0.75**(经验最优平衡)
- **Python dict: 2/3**(开放寻址要更低)

**到达阈值就扩容**——通常**翻倍**,所有元素**重新 hash**放到新表。

**扩容是 O(n) 操作**,是哈希表性能最大的单次开销。

## 五、Redis 的"渐进式 rehash"

Redis 的哈希表(`dict` 结构)扩容用了个绝妙的办法——**不一次性搬完,分多次搬**。

```
struct dict {
    dictht ht[2];   // 两张表
    int rehashidx;  // 搬到哪了,-1 表示没在 rehash
}
```

**rehash 流程**:
1. 分配 `ht[1]`(新表,2 倍大)
2. `rehashidx` 从 0 开始
3. **每次操作 dict(增删改查)时,顺手搬 `ht[0][rehashidx]` 下一个桶到 `ht[1]`**,然后 `rehashidx++`
4. 查询时先查 `ht[0]`,没有再查 `ht[1]`
5. 所有桶搬完后,`ht[0] = ht[1]`, `rehashidx = -1`

**为什么这么做**:Redis 是**单线程**,一次 O(n) rehash 会阻塞所有请求。百万 key 的表一次搬,延迟爆表。**摊到每次操作里,每次只多做一点点,感知不到卡顿**。

这是**工程里"摊销"的最好案例之一**,跟动态数组翻倍是同一个思想。

## 六、哈希表的三大工程坑

### 坑一:用可变对象当 key

```java
Map<List<Integer>, String> m = new HashMap<>();
List<Integer> k = new ArrayList<>(Arrays.asList(1, 2));
m.put(k, "hello");
k.add(3);              // ⚠️ 改了 key
m.get(k);              // null!
```

**key 变了,hash 变了,在哈希表里"找不到原来那个桶"**。**永远用不可变对象当 key**:String、Integer、record、frozenset。

### 坑二:`equals` 和 `hashCode` 不匹配

Java 里自定义类当 key,**必须同时重写 `equals` 和 `hashCode`**,且保持"equals 相等 → hashCode 相等"。只重写一个会导致:存进去后查不出来(hash 不同进了不同桶)或查到错的(hash 相同但 equals 不同)。

### 坑三:哈希碰撞攻击

**恶意输入可以构造大量 hash 相同的 key**,把哈希表退化成链表,拒绝服务攻击(HashDoS)。

**防御**:
- Java 8 树化(退化到 log n)
- 哈希函数**加入随机 seed**(Go / Python3.3+ / Rust 都做了),每次程序启动 seed 不同,攻击者无法离线构造冲突

## 七、哈希表的"算法用法"

面试里哈希表的**招式库**:

### 1. 计数 / 频率统计

```typescript
const count = new Map<string, number>()
for (const s of arr) count.set(s, (count.get(s) ?? 0) + 1)
```

### 2. 去重

```typescript
const seen = new Set<number>()
for (const x of arr) {
  if (seen.has(x)) { /* 重复 */ }
  seen.add(x)
}
```

### 3. 两数之和(经典用哈希换遍历)

```typescript
function twoSum(nums: number[], target: number): number[] {
  const map = new Map<number, number>()
  for (let i = 0; i < nums.length; i++) {
    const want = target - nums[i]
    if (map.has(want)) return [map.get(want)!, i]
    map.set(nums[i], i)
  }
  return []
}
```

**O(n²) → O(n)**,就靠哈希查存在 O(1)。

### 4. 判断"见过"

有环链表、重复元素、循环、访问过的节点——全套路。

### 5. 分组 / groupBy

```typescript
function groupAnagrams(strs: string[]): string[][] {
  const map = new Map<string, string[]>()
  for (const s of strs) {
    const key = s.split('').sort().join('')
    if (!map.has(key)) map.set(key, [])
    map.get(key)!.push(s)
  }
  return [...map.values()]
}
```

**相似元素共享一个 key**。异位词、按某种特征分组,都走这个模板。

## 八、HashMap vs HashSet vs TreeMap

| 结构 | 底层 | 有序? | 复杂度 |
| --- | --- | --- | --- |
| HashMap / HashSet | 哈希表 | ❌ | O(1) 平均 |
| LinkedHashMap | 哈希 + 双链表 | 插入顺序 | O(1) 平均 |
| TreeMap / TreeSet | 红黑树 | 按 key 排序 | O(log n) |

**选型**:
- 只求快 → HashMap
- 要保留插入顺序(LRU 就是用 LinkedHashMap 改的) → LinkedHashMap
- 要按 key 排序 / 范围查 → TreeMap

---

## 九、一句话总结

> **哈希表是空间换时间的典范——用一个数组 + 一个函数,把 O(n) 查找变成 O(1)**。但 O(1) 是平均,**冲突处理、负载因子、哈希函数任何一个出问题,都会塌到 O(n)**。理解到这一层,你才敢说"会用哈希"。
