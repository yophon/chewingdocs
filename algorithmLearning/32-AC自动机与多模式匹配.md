# AC 自动机与多模式匹配

KMP 解决的是"一个模式串在一段文本里出现在哪里"。但工程里更常见的是另一类问题:有几万条敏感词、几千条日志规则、几百个安全特征,要在一段文本里一次性找出所有命中。对每个模式串跑一遍 KMP,文本会被扫几千几万次。AC 自动机(Aho-Corasick)解决的就是这个问题。

> 一句话先记住:**AC 自动机 = Trie + fail 指针**。Trie 负责共享前缀,fail 指针负责匹配失败时不回退文本指针。构建一次自动机后,文本只需要扫一遍,就能找出所有模式串。

---

## 一、多模式匹配为什么不能暴力

假设:

- 文本长度 `n = 1,000,000`
- 模式串数量 `k = 10,000`
- 平均模式串长度 `m = 10`

如果每个模式串都扫一遍文本:

```text
O(k * n)
```

就是 100 亿级别的字符比较。

Trie 能共享前缀,但单纯 Trie 还有问题:匹配失败时不知道跳到哪里,经常要从文本下一个位置重新开始。

AC 自动机的关键是给 Trie 加 fail 指针:

> 当前路径匹配失败时,跳到"当前后缀里最长的可匹配前缀"继续。

这和 KMP 的失配函数是同一类思想。

---

## 二、先建 Trie

模式串:

```text
he
she
his
hers
```

Trie 大概长这样:

```text
root
├── h
│   ├── e (he)
│   │   └── r
│   │       └── s (hers)
│   └── i
│       └── s (his)
└── s
    └── h
        └── e (she)
```

每个终止节点记录一个或多个模式串。

```ts
class ACNode {
  children = new Map<string, ACNode>()
  fail: ACNode | null = null
  outputs: string[] = []
}
```

插入模式串:

```ts
function insert(root: ACNode, pattern: string): void {
  let node = root

  for (const ch of Array.from(pattern)) {
    if (!node.children.has(ch)) {
      node.children.set(ch, new ACNode())
    }
    node = node.children.get(ch)!
  }

  node.outputs.push(pattern)
}
```

这里用 `Array.from` 是为了按 Unicode code point 迭代,比直接按 UTF-16 code unit 稍稳一点。真正要处理自然语言,还要做归一化、大小写、繁简、空白符等规范化。

---

## 三、fail 指针是什么

fail 指针表示:

> 如果当前节点继续匹配失败,应该跳到哪个节点。

例子:已经匹配到 `she`,下一字符失败。`she` 的后缀有:

```text
he
e
```

如果 Trie 里有 `he`,那就可以跳到 `he` 节点,不用把文本指针退回去。

构建 fail 指针用 BFS:

1. root 的子节点 fail 都指向 root
2. 按层遍历每个节点
3. 子节点的 fail = 父节点 fail 沿着同字符转移能到的节点
4. 子节点继承 fail 节点的 outputs

继承 outputs 很关键。因为匹配到 `she` 时,也应该同时命中后缀 `he`。

## 四、完整实现

```ts
class AhoCorasick {
  private root = new ACNode()

  constructor(patterns: string[]) {
    for (const pattern of patterns) {
      if (pattern.length > 0) insert(this.root, pattern)
    }
    this.buildFailureLinks()
  }

  search(text: string): Array<{ index: number; pattern: string }> {
    const result: Array<{ index: number; pattern: string }> = []
    const chars = Array.from(text)
    let node = this.root

    for (let i = 0; i < chars.length; i++) {
      const ch = chars[i]

      while (node !== this.root && !node.children.has(ch)) {
        node = node.fail!
      }

      if (node.children.has(ch)) {
        node = node.children.get(ch)!
      }

      for (const pattern of node.outputs) {
        result.push({
          index: i - Array.from(pattern).length + 1,
          pattern
        })
      }
    }

    return result
  }

  private buildFailureLinks(): void {
    const queue: ACNode[] = []
    this.root.fail = this.root

    for (const child of this.root.children.values()) {
      child.fail = this.root
      queue.push(child)
    }

    while (queue.length > 0) {
      const current = queue.shift()!

      for (const [ch, child] of current.children) {
        let fallback = current.fail!

        while (fallback !== this.root && !fallback.children.has(ch)) {
          fallback = fallback.fail!
        }

        if (fallback.children.has(ch)) {
          child.fail = fallback.children.get(ch)!
        } else {
          child.fail = this.root
        }

        child.outputs.push(...child.fail.outputs)
        queue.push(child)
      }
    }
  }
}
```

使用:

```ts
const ac = new AhoCorasick(["he", "she", "his", "hers"])
console.log(ac.search("ushers"))

// [
//   { index: 1, pattern: "she" },
//   { index: 2, pattern: "he" },
//   { index: 2, pattern: "hers" }
// ]
```

复杂度:

| 阶段 | 复杂度 |
| --- | --- |
| 建 Trie | O(模式串总长度) |
| 建 fail 指针 | O(状态数 * 字符转移成本) |
| 匹配文本 | O(文本长度 + 命中数量) |
| 空间 | O(模式串总长度) |

## 五、匹配策略:全部命中还是最长命中

AC 自动机默认会返回所有命中,包括重叠命中。

比如模式串:

```text
bad
badword
word
```

文本:

```text
badword
```

可能命中:

```text
bad      [0, 2]
badword  [0, 6]
word     [3, 6]
```

工程里通常要明确策略:

| 策略 | 含义 | 场景 |
| --- | --- | --- |
| 全部命中 | 返回所有模式 | 规则审计、日志分析 |
| 最长优先 | 同一起点只取最长 | 敏感词替换 |
| 左最短 | 找到就停 | 快速判断是否包含 |
| 不重叠最长 | 从左到右取最长非重叠片段 | 文本标注 |

算法只是找出候选,最终业务还要定义怎么解释候选。

## 六、工程应用

### 1. 敏感词过滤

把词库构造成 AC 自动机,每条文本扫一遍。命中后可以:

- 拒绝发布
- 替换为 `***`
- 标记风险等级
- 进入人工审核

重点不在算法本身,而在词库规范化:

- 大小写
- 全角半角
- 空白符插入
- 繁简转换
- 同音 / 变体字

这些会让问题从"字符串匹配"变成"文本规范化 + 规则系统"。

### 2. 日志规则匹配

例如在日志中匹配多个错误特征:

```text
OutOfMemoryError
connection reset
deadlock
timeout
```

AC 自动机适合固定规则集、高吞吐扫描。

### 3. WAF / 安全规则

很多安全规则是多模式匹配的一部分。但真实 WAF 不会只靠 AC 自动机,还会结合正则、解析器、上下文规则和行为特征。

### 4. DNA / 序列匹配

字符集小、模式多、文本长,非常适合 AC 自动机。

## 七、AC 自动机 vs 其他方案

| 方案 | 适合 | 不适合 |
| --- | --- | --- |
| KMP | 单个模式串 | 上万个模式串 |
| Rabin-Karp | 多模式长度相近、可接受哈希碰撞校验 | 复杂重叠匹配 |
| Trie | 前缀查询 | 文本中任意位置匹配 |
| AC 自动机 | 多模式精确匹配 | 模糊匹配、复杂语法规则 |
| 正则引擎 | 模式表达能力强 | 超大固定词库可能慢 |

**经验法则**:

- 词库型精确匹配 → AC 自动机
- 单模式搜索 → KMP 或语言内置查找
- 模糊规则 → 正则 / parser / 专用规则引擎

## 八、常见坑

**坑一:每次词库变化都重建自动机**

AC 自动机构建成本不低。词库高频变化时,可以批量更新、双缓冲切换,或者按版本构建后原子替换。

**坑二:忽略文本规范化**

用户不会老老实实输入标准文本。大小写、空格、符号、全角半角、组合字符都会绕过朴素匹配。

**坑三:输出数量过大**

有些文本会命中大量重叠模式。匹配复杂度里有 `+ 命中数量`,结果集可能很大。需要限制最大命中数或做策略过滤。

**坑四:用 AC 自动机替代所有规则系统**

AC 自动机只做精确多模式匹配。上下文相关规则、语义判断、正则分组、结构化协议解析,不是它的强项。

**坑五:Unicode 下标混乱**

`Array.from` 得到的是 code point 下标,不一定等于用户看到的字符位置。展示高亮时要特别小心。

---

## 九、一句话总结

> **AC 自动机把"成千上万个模式串匹配"压成"文本扫一遍"**。Trie 共享前缀,fail 指针复用后缀,outputs 记录命中。它不是万能规则引擎,但在词库型精确匹配里,是最值得掌握的工程算法之一。
