# 随机化与采样算法

随机不是"随便来一下"。工程里用随机,通常是为了三个目标:公平、抗偏、降成本。打乱列表要保证每个排列概率相同;从无限日志流里抽样要保证每条日志机会相同;A/B 实验要让同一个用户稳定落到同一组;快速选择用随机 pivot 是为了避免被构造最坏输入打穿。

> 一句话先记住:**随机算法最怕"看起来随机,实际有偏"**。洗牌用 Fisher-Yates,流式抽样用水库抽样,按权重选择用前缀和 + 二分或 Alias Method,A/B 分流用稳定哈希,安全场景不要用普通伪随机。

---

## 一、工程里为什么要随机

随机化常见在这些地方:

| 场景 | 目标 | 典型算法 |
| --- | --- | --- |
| 打乱播放列表 / 推荐列表 | 每个排列等概率 | Fisher-Yates |
| 从海量日志中抽样 1 万条 | 不把全量数据放内存 | 水库抽样 |
| 按概率抽奖 / 流量分配 | 权重越高越容易被选中 | 前缀和 + 二分 |
| 快排 / QuickSelect | 避免最坏输入 | 随机 pivot |
| A/B 实验 | 稳定、均匀分桶 | 哈希分桶 |
| 分布式采样 | 降低上报成本 | 哈希采样 / Reservoir |

随机化的核心不是"不确定",而是**可控的不确定**:

- 该等概率时必须等概率
- 该可复现时必须可复现
- 该不可预测时必须不可预测

这三件事经常互相冲突。

## 二、Fisher-Yates 洗牌:唯一该手写的洗牌

错误洗牌最常见写法:

```ts
arr.sort(() => Math.random() - 0.5)
```

这不是均匀洗牌。不同排序算法调用比较函数的次数和路径不同,会造成排列概率不相等。

正确写法是 Fisher-Yates:

```ts
function shuffle<T>(items: T[], random = Math.random): T[] {
  const arr = [...items]

  for (let i = arr.length - 1; i > 0; i--) {
    const j = Math.floor(random() * (i + 1))
    ;[arr[i], arr[j]] = [arr[j], arr[i]]
  }

  return arr
}
```

直觉:

- 第 `i` 轮,从 `[0, i]` 中随机挑一个元素放到位置 `i`
- 每个元素被放到当前位置的概率都是 `1 / (i + 1)`
- 从后往前固定位置,最终每个排列概率都是 `1 / n!`

### 可复现洗牌

测试、实验和游戏回放常常需要"随机但可复现"。这时要传入带 seed 的伪随机数生成器,不要直接用 `Math.random()`。

```ts
function mulberry32(seed: number): () => number {
  return () => {
    seed |= 0
    seed = (seed + 0x6d2b79f5) | 0
    let t = Math.imul(seed ^ (seed >>> 15), 1 | seed)
    t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296
  }
}

const random = mulberry32(20260604)
const result = shuffle([1, 2, 3, 4, 5], random)
```

这类 PRNG 适合测试和普通模拟,不适合密码学。

## 三、水库抽样:从未知长度流里等概率抽 k 个

问题:

> 数据流长度未知,只能扫一遍,内存只能放 k 个元素。怎样保证最后每个元素被选中的概率都相同?

这就是 Reservoir Sampling。

### k = 1

第 `i` 个元素到来时,以 `1 / i` 的概率替换当前答案。

```ts
function sampleOne<T>(stream: Iterable<T>, random = Math.random): T | undefined {
  let result: T | undefined
  let count = 0

  for (const item of stream) {
    count++
    if (random() < 1 / count) {
      result = item
    }
  }

  return result
}
```

为什么公平?

第 1 个元素最终留下的概率:

```text
1 * (1 - 1/2) * (1 - 1/3) * ... * (1 - 1/n) = 1/n
```

第 i 个元素先以 `1/i` 进入,后面每轮不被替换:

```text
1/i * i/(i+1) * (i+1)/(i+2) * ... * (n-1)/n = 1/n
```

所有元素一样。

### k 个样本

先把前 k 个放入水库。第 `i` 个元素到来时,以 `k / i` 的概率进入水库;如果进入,随机替换水库中的一个位置。

```ts
function reservoirSample<T>(stream: Iterable<T>, k: number, random = Math.random): T[] {
  const reservoir: T[] = []
  let count = 0

  for (const item of stream) {
    count++

    if (reservoir.length < k) {
      reservoir.push(item)
      continue
    }

    const j = Math.floor(random() * count)
    if (j < k) {
      reservoir[j] = item
    }
  }

  return reservoir
}
```

水库抽样适合:

- 从日志流中抽样排查
- 从数据库大表游标里抽样
- 在线统计时保留固定大小样本
- 不知道总量时做均匀抽样

不适合:

- 需要按权重抽样
- 需要精确 Top K
- 需要保留时间最近性

## 四、加权随机:按概率选择

问题:

> A 权重 5,B 权重 3,C 权重 2。每次随机选一个,概率分别是 50%、30%、20%。

最常用做法是前缀和 + 二分:

```text
weights:  [5, 3, 2]
prefix:   [5, 8, 10]
random x in [0, 10)

x < 5      -> A
5 <= x < 8 -> B
8 <= x     -> C
```

代码:

```ts
class WeightedPicker<T> {
  private items: T[]
  private prefix: number[]
  private total: number

  constructor(entries: Array<[T, number]>) {
    this.items = []
    this.prefix = []
    this.total = 0

    for (const [item, weight] of entries) {
      if (weight <= 0) continue
      this.items.push(item)
      this.total += weight
      this.prefix.push(this.total)
    }
  }

  pick(random = Math.random): T {
    if (this.items.length === 0) {
      throw new Error("no positive weights")
    }

    const target = random() * this.total
    let left = 0
    let right = this.prefix.length - 1

    while (left < right) {
      const mid = Math.floor((left + right) / 2)
      if (this.prefix[mid] > target) right = mid
      else left = mid + 1
    }

    return this.items[left]
  }
}
```

复杂度:

- 构建 O(n)
- 每次抽样 O(log n)

如果权重不变、抽样极其频繁,可以用 Alias Method 把单次抽样压到 O(1)。但 Alias 实现复杂,多数业务场景前缀和 + 二分已经够用。

## 五、随机化快排与 QuickSelect

快排最怕什么?

> pivot 选得太差,每次只切掉一个元素,从 O(n log n) 退化到 O(n²)。

随机 pivot 的意义是:让攻击者很难构造稳定最坏输入。

```ts
function quickSelect(nums: number[], k: number): number {
  let left = 0
  let right = nums.length - 1
  const target = nums.length - k

  while (left <= right) {
    const pivotIndex = left + Math.floor(Math.random() * (right - left + 1))
    const p = partition(nums, left, right, pivotIndex)

    if (p === target) return nums[p]
    if (p < target) left = p + 1
    else right = p - 1
  }

  throw new Error("unreachable")
}

function partition(nums: number[], left: number, right: number, pivotIndex: number): number {
  const pivot = nums[pivotIndex]
  ;[nums[pivotIndex], nums[right]] = [nums[right], nums[pivotIndex]]

  let store = left
  for (let i = left; i < right; i++) {
    if (nums[i] < pivot) {
      ;[nums[store], nums[i]] = [nums[i], nums[store]]
      store++
    }
  }

  ;[nums[store], nums[right]] = [nums[right], nums[store]]
  return store
}
```

QuickSelect 找第 K 大:

- 平均 O(n)
- 最坏 O(n²)
- 原地
- 适合内存数组

如果输入来自外部不可信用户,并且最坏复杂度不能接受,用堆或确定性选择算法更稳。

## 六、A/B 分流:不要每次真的 random

A/B 实验的随机不是每次请求都随机。否则同一个用户今天进 A 组,明天进 B 组,实验结果会乱。

正确做法是**稳定哈希分桶**:

```ts
function bucket(userId: string, experimentId: string, bucketCount = 100): number {
  const hash = fnv1a(`${experimentId}:${userId}`)
  return hash % bucketCount
}

function inExperiment(userId: string, experimentId: string, percent: number): boolean {
  return bucket(userId, experimentId) < percent
}

function fnv1a(input: string): number {
  let hash = 0x811c9dc5

  for (let i = 0; i < input.length; i++) {
    hash ^= input.charCodeAt(i)
    hash = Math.imul(hash, 0x01000193)
  }

  return hash >>> 0
}
```

关键点:

- `experimentId` 必须参与哈希,否则不同实验会互相绑定
- 分桶数量要足够细,比如 1000 或 10000
- 用户 ID 要稳定,不要用会变化的 session ID
- 扩流时只扩大阈值,不要重新打散所有用户

这类随机的目标不是不可预测,而是**均匀 + 稳定**。

## 七、随机数的三个层级

| 层级 | 场景 | 可用工具 |
| --- | --- | --- |
| 普通伪随机 | 洗牌、抽样、模拟、测试 | `Math.random()` / seed PRNG |
| 稳定哈希随机 | A/B 分流、采样开关、分桶 | MurmurHash / xxHash / FNV |
| 密码学随机 | token、验证码、密钥、nonce | Web Crypto / Node `crypto` |

**不要混用**。

尤其不要用 `Math.random()` 生成:

- 登录 token
- 重置密码链接
- API key
- 加密 nonce
- 抽奖系统里需要审计的随机结果

这类场景要用密码学安全随机数。

## 八、常见坑

**坑一:用排序函数洗牌**

`sort(() => Math.random() - 0.5)` 有偏,而且不同运行时行为不同。洗牌只用 Fisher-Yates。

**坑二:取模偏差**

如果随机整数范围不是目标范围的整数倍,直接 `% n` 会有偏。普通业务影响不大,安全和抽奖审计场景要用 rejection sampling。

**坑三:该稳定时不稳定**

A/B 分流、采样开关、灰度发布都需要稳定分桶。每次请求都 `random()` 是错的。

**坑四:该不可预测时可预测**

带 seed 的 PRNG 可复现,也就可预测。安全场景必须用 CSPRNG。

**坑五:抽样后忘了偏差**

采样日志只能代表采样规则覆盖的总体。如果只采错误日志、只采大客户、只采白天流量,结论不能外推到全量。

---

## 九、一句话总结

> **随机算法的核心是控制概率分布**。洗牌要每个排列等概率,水库抽样要每个元素等概率,加权随机要按权重成比例,A/B 分流要稳定均匀,安全随机要不可预测。看起来"差不多随机"的写法,往往正是事故源头。
