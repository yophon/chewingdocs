# 位图与基数估算

判断一个用户 ID 有没有出现过,最直接的办法是 HashSet。但如果 ID 范围很大、数据量上亿,HashSet 的对象和指针开销会很吓人。位图(Bitmap)把"是否存在"压到一个 bit。Roaring Bitmap 在稀疏和密集之间自动切换。HyperLogLog 更进一步:不保存元素本身,只估算"不同元素大概有多少个"。

> 一句话先记住:**Bitmap 适合精确表示整数集合,Roaring Bitmap 适合压缩大规模整数集合并做集合运算,HyperLogLog 适合估算去重数量**。它们都在用空间换可接受的能力边界。

---

## 一、为什么不用普通 Set

假设有 1 亿个用户 ID,要判断某个用户是否出现过。

`Set<number>` 的问题:

- 每个数字不只占 4 或 8 字节
- 哈希桶、对象头、指针都有额外开销
- 内存局部性差

Bitmap 的做法:

```text
第 x 个 bit 表示 x 是否存在
```

如果 ID 范围是 1 亿:

```text
100,000,000 bits ≈ 12.5 MB
```

这就是位图的杀伤力。

---

## 二、Bitmap:一个 bit 表示一个布尔值

基本操作:

- `add(x)`:把第 x 位设为 1
- `has(x)`:检查第 x 位是否为 1
- `remove(x)`:把第 x 位设为 0

```ts
class BitMap {
  private bytes: Uint8Array

  constructor(maxValue: number) {
    this.bytes = new Uint8Array(Math.ceil((maxValue + 1) / 8))
  }

  add(value: number): void {
    const byteIndex = Math.floor(value / 8)
    const bitIndex = value % 8
    this.bytes[byteIndex] |= 1 << bitIndex
  }

  remove(value: number): void {
    const byteIndex = Math.floor(value / 8)
    const bitIndex = value % 8
    this.bytes[byteIndex] &= ~(1 << bitIndex)
  }

  has(value: number): boolean {
    const byteIndex = Math.floor(value / 8)
    const bitIndex = value % 8
    return (this.bytes[byteIndex] & (1 << bitIndex)) !== 0
  }
}
```

Bitmap 的优势:

- 极省空间
- 查询 O(1)
- 插入 O(1)
- 集合运算快

集合运算本质就是位运算:

```text
A ∩ B = A & B
A ∪ B = A | B
A - B = A & ~B
```

这比逐个元素遍历 Set 快很多,而且 CPU 对连续字节数组很友好。

---

## 三、Bitmap 的限制

Bitmap 有几个硬边界:

### 1. 只能直接处理整数下标

字符串、对象、邮箱地址不能直接放 Bitmap。需要先映射成整数 ID。

### 2. 范围太大时会浪费

如果只出现 100 个 ID,但最大 ID 是 `10^12`,直接开 Bitmap 会浪费到不可接受。

```text
10^12 bits ≈ 125 GB
```

### 3. 稀疏集合不划算

Bitmap 对密集整数集合很强,对稀疏集合不一定强。

这就是 Roaring Bitmap 出场的原因。

---

## 四、Roaring Bitmap:稀疏和密集都照顾

Roaring Bitmap 的核心思路:

> 把 32 位整数按高 16 位分桶,每个桶里存低 16 位。桶内根据密度选择不同容器。

例如:

```text
value = high16 | low16
```

同一个 `high16` 的值放在同一个容器里。

容器常见三种:

| 容器 | 适合 | 表示 |
| --- | --- | --- |
| Array Container | 稀疏 | 排序数组存 low16 |
| Bitmap Container | 密集 | 65536 个 bit |
| Run Container | 连续区间多 | 存 run-length |

这样就兼顾了两种极端:

- 稀疏时,用小数组,不浪费 65536 个 bit
- 密集时,切成 bitmap,集合运算极快
- 连续区间多时,用 run 压缩

### 为什么 Roaring 工程上受欢迎

因为很多业务集合既不是完全稀疏,也不是完全密集。

比如:

- 某活动命中的用户 ID
- 某标签下的用户集合
- 某天访问过页面的用户集合
- 搜索引擎倒排列表
- ClickHouse / Druid 里的 bitmap 索引

Roaring Bitmap 能在这些场景里把存储、查询和集合运算都做得比较均衡。

---

## 五、Bitmap 在工程里的用法

### 1. 用户集合运算

```text
买过 A 的用户       -> bitmapA
买过 B 的用户       -> bitmapB
同时买过 A 和 B     -> bitmapA & bitmapB
买过 A 但没买过 B   -> bitmapA & ~bitmapB
买过 A 或 B         -> bitmapA | bitmapB
```

这就是很多画像、广告、人群圈选系统的底层味道。

### 2. 去重标记

如果 ID 范围可控,Bitmap 可以直接做精确去重。

```text
seen[id] = 1
```

比 HashSet 省很多内存。

### 3. 权限和特征开关

某个用户是否具备某个权限、某个实验是否开启,也可以用 bitset 表达。但如果维度很多,通常会按用户或按特征组织多个 bitmap。

### 4. 倒排索引

搜索系统里:

```text
term -> docId bitmap
```

查询 `termA AND termB` 就是两个 bitmap 取交集。

## 六、HyperLogLog:不保存元素,只估算基数

Bitmap 是精确集合。HyperLogLog(HLL)解决的是另一个问题:

> 我不关心有哪些用户,只关心不同用户大概有多少个。

比如统计 UV:

```text
今天访问首页的独立用户数是多少?
```

如果用 HashSet 保存所有 userId,内存会随用户数线性增长。

HLL 的思路是:

1. 对元素做哈希,得到均匀随机的 bit 串
2. 用一部分 bit 选择桶(register)
3. 用剩余 bit 统计前导零个数
4. 前导零越长,说明总体基数越可能大

直觉:

```text
看到 1 个前导零:不稀奇
看到 10 个前导零:大概要见过很多样本
看到 30 个前导零:总体规模很可能非常大
```

HLL 维护很多桶,最后用调和平均减少方差。

---

## 七、HLL 的能力边界

| 维度 | HLL |
| --- | --- |
| 能否判断某元素是否存在 | 不能 |
| 能否取出元素 | 不能 |
| 能否精确计数 | 不能 |
| 能否合并多个统计 | 可以 |
| 空间是否固定 | 基本固定 |
| 典型误差 | 约 `1.04 / sqrt(m)` |

`m` 是桶数量。桶越多,误差越小,空间越大。

HLL 最大的工程价值是**可合并**:

```text
HLL(A ∪ B) = merge(HLL(A), HLL(B))
```

合并时对每个 register 取最大值即可。

这意味着:

- 每台机器本地统计一个 HLL
- 最后汇总合并
- 不需要传全量 userId

这对分布式 UV 统计非常重要。

## 八、Redis 里的 HyperLogLog

Redis 提供:

```text
PFADD page:home:2026-06-04 user123
PFCOUNT page:home:2026-06-04
PFMERGE page:all page:home page:detail
```

它的用途是估算去重数量,不是保存集合。

常见误用:

- 想知道某个 user 是否访问过 → HLL 做不到
- 想列出访问用户列表 → HLL 做不到
- 想精确计费 → HLL 不适合

HLL 适合趋势统计、容量估算、监控指标,不适合强精确业务。

---

## 九、和 Bloom / Count-Min Sketch 的区别

概率结构很容易混:

| 结构 | 回答的问题 | 是否精确 | 是否能删除 | 典型用途 |
| --- | --- | --- | --- | --- |
| Bitmap | x 是否存在 | 精确 | 可以 | 整数集合 |
| Roaring Bitmap | 大整数集合和集合运算 | 精确 | 可以 | 用户集合、倒排索引 |
| Bloom Filter | x 可能存在吗 | 有假阳性 | 普通版不行 | 防缓存穿透、去重预判 |
| Count-Min Sketch | x 出现了多少次 | 只会高估 | 不自然 | 频率估计、Heavy Hitters |
| HyperLogLog | 不同 x 有多少个 | 近似 | 不关心删除 | UV / 基数估算 |

选型关键:

- 要精确集合 → Bitmap / Roaring
- 要省空间判断存在,允许误判 → Bloom
- 要估频率 → Count-Min Sketch
- 要估去重数量 → HyperLogLog

## 十、常见坑

**坑一:ID 范围太大还硬开 Bitmap**

先看最大 ID,不是只看元素数量。范围太大且稀疏,用 Roaring 或 HashSet。

**坑二:把 HLL 当 Set 用**

HLL 不能判断某个元素是否存在,也不能列出元素。它只估算基数。

**坑三:近似结构用于强精确业务**

UV 趋势可以近似,计费、扣款、权限判断通常不能近似。

**坑四:哈希质量太差**

Bloom、CMS、HLL 都依赖哈希分布。哈希不均匀,误差和碰撞会明显变差。

**坑五:忽略序列化成本**

Bitmap 很适合内存和批量运算,但跨网络传输、落盘、压缩格式也要算成本。Roaring 的价值之一就是在这些环节省空间。

---

## 十一、一句话总结

> **位图家族解决的是"用极少空间表达海量集合信息"**。Bitmap 精确但怕稀疏大范围,Roaring Bitmap 把稀疏和密集都处理得更好,HLL 则放弃元素本身,只保留去重数量的近似估计。工程里看到用户集合、倒排索引、UV 统计,就该想到这几种结构。
