# 限流算法

限流是后端工程师的**入职第三天就要懂、入职三年还在踩坑**的题目。看似简单(每秒不超过 N 次),实际四五种实现各有漏洞——固定窗口有"边界双倍"问题,漏桶不能突发,令牌桶要平衡参数……这一篇把四种经典限流算法的**实现 + 坑 + 选型**讲透。

> 一句话先记住:**限流算法的核心问题是"如何在时间维度上控制请求量"**。固定窗口最简单但有边界问题,滑动窗口修复了边界,漏桶强制匀速,令牌桶允许突发——**没有最优算法,只有最适合你业务的算法**。

---

## 一、为什么需要限流

**保护下游 / 自身 / 防恶意**:
- 下游服务能力有限,不限流就雪崩
- 数据库连接池有限,不限流就拖垮全站
- API 防爬 / 防刷 / 防 DDoS

**限流的位置**:网关层 → 服务层 → 数据库连接层。**越往外限越好**——把流量挡在系统外面比让它进来再拒绝更省。

---

## 二、固定窗口(Fixed Window)

**最简单的实现**:每秒计数,到达上限就拒绝。下一秒重置。

```typescript
class FixedWindowLimiter {
  private count = 0
  private windowStart = Date.now()

  constructor(private limit: number, private windowMs: number = 1000) {}

  allow(): boolean {
    const now = Date.now()
    if (now - this.windowStart >= this.windowMs) {
      this.count = 0
      this.windowStart = now
    }
    if (this.count >= this.limit) return false
    this.count++
    return true
  }
}
```

### 致命缺陷:边界双倍

假设限流 100/秒。在 0:00:00.999 来 100 个请求,在 0:00:01.001 又来 100 个——**两个 0.002 秒内 200 个请求都通过了**。

```
窗口边界:
  时间:[0.5, 1.0, 1.5]
  实际限速允许:
    [0.999s] 100 个 ← 窗口 1
    [1.001s] 100 个 ← 窗口 2
  → 0.002s 内 200 个 = 真实速率超 100 倍
```

**生产环境慎用**,只适合"粗粒度限流"。

## 三、滑动窗口(Sliding Window Log)

**精确版**:存所有请求时间戳,每次检查"过去 1 秒内有多少个请求"。

```typescript
class SlidingWindowLimiter {
  private timestamps: number[] = []

  constructor(private limit: number, private windowMs: number = 1000) {}

  allow(): boolean {
    const now = Date.now()
    // 清掉窗口外的
    while (this.timestamps.length > 0 && this.timestamps[0] <= now - this.windowMs) {
      this.timestamps.shift()
    }
    if (this.timestamps.length >= this.limit) return false
    this.timestamps.push(now)
    return true
  }
}
```

**优点**:严格 ≤ limit,无边界问题。
**缺点**:**内存 O(limit)**,每个请求要存。limit 大或并发大时内存爆炸。

**改进**:用循环数组,或更精妙的"分桶滑动窗口"。

---

## 四、滑动窗口(Sliding Window Counter)

**折中版**:把窗口分成 k 个小桶,**只统计每个桶的计数**(不存每个请求)。

```
窗口 1 秒分成 10 个 100ms 桶:
  桶 1: 5 个
  桶 2: 8 个
  ...
  桶 10: 3 个
查询时:把当前 1 秒覆盖的所有桶相加
```

```typescript
class SlidingWindowCounter {
  private buckets: number[]
  private bucketMs: number
  private windowStart: number

  constructor(
    private limit: number,
    private bucketCount: number = 10,
    private windowMs: number = 1000,
  ) {
    this.bucketMs = windowMs / bucketCount
    this.buckets = new Array(bucketCount).fill(0)
    this.windowStart = Date.now()
  }

  allow(): boolean {
    const now = Date.now()
    this.advance(now)
    const total = this.buckets.reduce((a, b) => a + b, 0)
    if (total >= this.limit) return false
    this.buckets[this.bucketCount - 1]++
    return true
  }

  private advance(now: number) {
    const elapsedBuckets = Math.floor((now - this.windowStart) / this.bucketMs)
    if (elapsedBuckets <= 0) return
    if (elapsedBuckets >= this.bucketCount) {
      this.buckets.fill(0)
    } else {
      this.buckets.splice(0, elapsedBuckets)
      while (this.buckets.length < this.bucketCount) this.buckets.push(0)
    }
    this.windowStart += elapsedBuckets * this.bucketMs
  }
}
```

**精度可调**:k 越大越接近精确,但内存和 CPU 也涨。**k = 10 是常见平衡**。

**Sentinel / Resilience4j 用这个**。

## 五、漏桶(Leaky Bucket)

**比喻**:水(请求)倒进桶,桶底有恒定速率漏水(处理)。**桶满了水溢出**(请求被拒)。

```
请求 →  [水位] → 漏水(恒定速率)
             ↓
          桶底处理
```

**特点**:
- **强制平滑输出速率**(无论输入多突发,输出永远恒定)
- **延迟换平滑**:请求会在桶里排队等待

实现:其实就是个**有限容量的队列 + 后台定时器**。

```typescript
class LeakyBucket {
  private queue: (() => void)[] = []
  constructor(
    private capacity: number,
    private leakRate: number,  // 每秒处理多少
  ) {
    setInterval(() => {
      if (this.queue.length > 0) this.queue.shift()!()
    }, 1000 / leakRate)
  }

  submit(task: () => void): boolean {
    if (this.queue.length >= this.capacity) return false
    this.queue.push(task)
    return true
  }
}
```

**适合**:对下游"绝对匀速"敏感的场景(老旧系统、第三方 API 限速)。
**不适合**:用户体验敏感 + 偶尔突发的场景(用户会觉得卡)。

## 六、令牌桶(Token Bucket):工业最常用

**比喻**:桶里以恒定速率添加令牌,**请求来了消耗一个令牌**;**没令牌就拒绝**(或等待)。桶满后不再添加(令牌不无限累积)。

**关键特性**:**允许突发** —— 桶满时,瞬间能放出 capacity 个请求,然后回到平均速率。

```typescript
class TokenBucket {
  private tokens: number
  private lastRefill: number

  constructor(
    private capacity: number,    // 桶容量
    private refillRate: number,  // 每秒加多少令牌
  ) {
    this.tokens = capacity
    this.lastRefill = Date.now()
  }

  allow(n = 1): boolean {
    this.refill()
    if (this.tokens < n) return false
    this.tokens -= n
    return true
  }

  private refill() {
    const now = Date.now()
    const elapsed = (now - this.lastRefill) / 1000
    this.tokens = Math.min(this.capacity, this.tokens + elapsed * this.refillRate)
    this.lastRefill = now
  }
}
```

**对比漏桶**:
| 维度 | 令牌桶 | 漏桶 |
| --- | --- | --- |
| 输出速率 | 平均恒定,瞬时可突发 | 严格匀速 |
| 用户体验 | 好(允许快) | 一般(被排队) |
| 适合 | 通用 API 限流 | 强制匀速场景 |

**Guava `RateLimiter`** 就是令牌桶。**Nginx 的 `limit_req` 也是令牌桶**(尽管文档叫 leaky bucket,实际行为是令牌桶语义)。

## 七、四种算法横向对比

| 算法 | 实现复杂度 | 平滑度 | 突发容忍 | 内存 | 边界问题 |
| --- | --- | --- | --- | --- | --- |
| 固定窗口 | 极简 | 差 | 不可控 | O(1) | **有** |
| 滑动窗口(日志) | 中 | 极佳 | 不允许 | O(N) | 无 |
| 滑动窗口(计数) | 中 | 好 | 不允许 | O(k) | 微弱 |
| 漏桶 | 中 | 完美 | **不允许** | O(队列) | 无 |
| 令牌桶 | 简单 | 好 | **允许** | O(1) | 无 |

**选型决策**:
- **简单粗暴 + 流量本来就稳** → 固定窗口
- **需要精确 + 内存吃得起** → 滑动窗口日志
- **需要精确 + 内存有限** → 滑动窗口计数
- **必须强制匀速** → 漏桶
- **通用 + 允许偶发突发** → **令牌桶**(80% 场景)

## 八、分布式限流

单机限流好做,**分布式系统的全局限流**才是难题。100 台机器各跑一个本地令牌桶,加起来限速远超期望。

### 方案 1:Redis + Lua

把令牌桶状态存 Redis,用 Lua 脚本原子更新:

```lua
local tokens = tonumber(redis.call("GET", KEYS[1]) or ARGV[1])
local last = tonumber(redis.call("GET", KEYS[2]) or 0)
local now = tonumber(ARGV[3])
local rate = tonumber(ARGV[2])
local cap = tonumber(ARGV[1])

local elapsed = (now - last) / 1000
tokens = math.min(cap, tokens + elapsed * rate)

if tokens < 1 then return 0 end
tokens = tokens - 1
redis.call("SET", KEYS[1], tokens)
redis.call("SET", KEYS[2], now)
return 1
```

**优点**:精确、原子。
**缺点**:每个请求一次 Redis,**Redis 成瓶颈**。

### 方案 2:本地限流 + 周期同步

每个节点本地令牌桶,**按总配额比例分配**(如 100 台机器每秒 10000,每台 100 / 秒)。
**坏处**:配额僵化。某个节点闲着,另一个节点流量大就被拒。

### 方案 3:本地预扣 + 集群对账

本地扣令牌,周期向中心节点上报实际消耗。中心节点动态调整每个节点的配额。**Sentinel 集群限流就是这种思路**。

### 方案 4:网关层做

K8s + Envoy / Nginx + lua 脚本,在网关侧统一限流。**最常见的生产方案**。

## 九、实战注意事项

### 1. 限流的颗粒度

- 全局:简单但可能伤好用户
- 按 IP:防爬虫,但要小心 NAT(几千用户共享一个 IP)
- 按用户 ID / API key:精细但要拿到身份
- 按接口 + 维度组合:`uid:apiName` 级,工程量大

### 2. 拒绝策略

- **直接 429**(最简单)
- **降级**(返回缓存 / 默认值)
- **排队**(漏桶天然支持)
- **熔断**(连续拒绝时停止接收一段时间)

### 3. 限流配置可热更新

写死限流值是反模式。要能**配置中心动态调整**——业务不同时段、不同活动需要不同限流。

### 4. 监控指标

- 当前 QPS / 限流上限 / 命中率
- 拒绝率 / 拒绝原因分布
- 限流被触发的接口 / 维度

**没有监控的限流就是黑盒**——出问题不知道是不是限流误伤。

---

## 十、一句话总结

> **限流四件套:固定窗口最简单但有边界、滑动窗口精确但费内存、漏桶强制匀速、令牌桶允许突发**。生产 80% 用令牌桶 + 网关层。**单机限流容易,分布式限流是工程难题——通常用 Redis + Lua 或本地预扣 + 集群对账**。

下一篇讲**一致性哈希**——分布式系统让节点上下线不引发"全表搬家"的关键算法,缓存集群 / 分库分表必备。
