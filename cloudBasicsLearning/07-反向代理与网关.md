# 反向代理与网关:Nginx、Cloudflare、API Gateway 的位置

> 一句话解释:**反向代理和网关都是站在用户请求前面的入口层,帮后面的服务接请求、转发请求、做统一控制**。

你可以把它理解成一家小公司的前台。用户不会直接冲进每个办公室找人,而是先到前台说明来意。前台根据路径、域名、登录状态、规则,把请求转给官网、API、后台管理、文件服务,或者直接拒绝。

反向代理更偏"把请求转到正确的后端";网关更偏"在进入后端前做统一治理"。实际产品里二者经常重叠,Nginx、Caddy、Traefik、Cloudflare、API Gateway、Ingress Controller 都可能承担其中一部分工作。

最容易混淆的是:**反向代理不是负载均衡的同义词,网关也不是后端服务本身**。负载均衡是把流量分给多台机器的一种能力;反向代理可以做负载均衡,也可以只转发到一台服务器。网关可以做鉴权、限流、路由、日志,但真正的业务逻辑仍然在后端服务里。

---

## 一、放在系统哪里

一个小团队常见链路可能长这样:

```text
用户浏览器 / App
  -> DNS
  -> CDN / WAF
  -> 反向代理或网关
  -> Web 服务 / API 服务 / 管理后台 / 文件服务
  -> 数据库 / 缓存 / 队列
```

反向代理和网关通常在服务入口处。它们的典型工作包括:

- 根据域名转发:`www.example.com` 到官网,`api.example.com` 到 API。
- 根据路径转发:`/api/*` 到后端,`/admin/*` 到后台,`/static/*` 到静态文件。
- 统一处理 HTTPS 证书。
- 记录访问日志。
- 做基础限流。
- 给后端隐藏真实 IP 和端口。
- 在多台后端之间做负载均衡。

一个真实场景:

```text
你有一台云服务器,上面跑了三个服务:
  - Next.js 前端:localhost:3000
  - Node API:localhost:4000
  - Admin 后台:localhost:5000

你不希望用户记端口:
  - https://example.com       -> localhost:3000
  - https://example.com/api   -> localhost:4000
  - https://admin.example.com -> localhost:5000

这时 Nginx / Caddy 就可以作为反向代理。
```

如果项目变成微服务,入口层就更重要:

```text
/api/users     -> 用户服务
/api/billing   -> 账单服务
/api/uploads   -> 上传服务
/api/internal  -> 只允许内网访问
```

这时 API Gateway 会把路由、鉴权、限流、日志、灰度发布等入口能力集中管理。否则每个服务都自己写一套入口逻辑,很快就会混乱。

---

## 二、常见套餐和使用限制

反向代理可以是你自己在服务器上装的 Nginx,也可以是云厂商托管的网关服务。两者限制很不一样。

自己部署 Nginx / Caddy 的限制通常是机器资源:

- 云服务器 CPU 和内存。
- 带宽上限。
- 文件描述符和连接数。
- TLS 握手带来的 CPU 消耗。
- 日志占用磁盘。
- 你自己负责升级、配置、证书、故障恢复。

托管 API Gateway / 边缘网关的限制通常是套餐维度:

- 请求数。
- 每秒请求数或突发流量。
- 路由数量。
- 自定义域名数量。
- 插件或策略数量。
- 鉴权、限流、转换规则是否收费。
- 日志保留时间。
- 跨区域或公网出站流量费用。

小团队最容易踩的坑有三个。

第一,把所有请求都穿过网关,但没看请求数计费。一个页面加载可能触发 10 个 API 请求,10000 次页面访问就可能变成 100000 次网关请求。

第二,在网关上做了复杂转换。比如请求体改写、响应体改写、JWT 校验、外部鉴权调用、动态路由匹配。每个动作都可能增加延迟,也可能进入更高套餐。

第三,入口层配置和应用配置不一致。比如网关允许上传 50 MB,后端只允许 10 MB;网关超时时间 30 秒,后端任务要跑 60 秒;网关转发了用户真实 IP,应用却没有正确读取,导致限流全部按网关 IP 计算。

## 三、小团队建议

如果你只有一个 Web 应用和一个 API,不要一开始就上复杂网关。一个托管平台自带的路由能力,或者一台服务器上的 Nginx / Caddy,通常已经够用。

小团队更应该优先把入口边界想清楚:

- 哪些域名对外开放。
- 哪些路径是静态资源。
- 哪些路径是 API。
- 哪些路径需要登录。
- 哪些路径只允许管理员访问。
- 哪些请求应该限流。
- 日志里要保留哪些字段,例如 path、status、latency、user-agent、真实 IP。

一个保守起点:

```text
example.com
  -> CDN
  -> Web 前端

api.example.com
  -> 网关 / 反向代理
  -> API 服务

admin.example.com
  -> 反向代理
  -> 管理后台
```

这样做的好处是边界清楚。前端静态资源走 CDN,API 单独观察,后台单独加访问控制。以后要加 WAF、限流、灰度发布,也比较容易分层处理。

什么时候该升级到更专业的网关?

- API 变多,路径和版本管理开始混乱。
- 有多个后端服务,需要统一鉴权和限流。
- 需要按客户、Token、IP、区域做不同策略。
- 需要完整访问日志和审计。
- 需要灰度发布、蓝绿发布、熔断、重试。
- 自己维护 Nginx 已经成为风险点。

最低限度的护栏是:

- 入口层配置要纳入版本管理,不要只靠控制台手点。
- 所有公网入口都要有 HTTPS。
- 管理后台不要和公开 API 混在同一个无保护路径下。
- 上传大小、超时时间、请求体限制要和后端一致。
- 保留基础访问日志,否则出问题时无法判断是用户、网关还是后端慢。

---

## 四、一句话总结

**反向代理和网关是系统入口的交通规则层;小团队不需要一开始就复杂化,但必须清楚请求从哪个入口进来、被转到哪里、在哪里做鉴权、限流和日志。**
