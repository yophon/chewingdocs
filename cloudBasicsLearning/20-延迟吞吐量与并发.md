# 延迟、吞吐量与并发:性能指标怎么读

> 一句话先记住:**延迟是一次请求等多久,吞吐量是一段时间能处理多少请求,并发是同一时刻有多少请求或任务正在处理**。

性能问题最容易被一句"网站有点慢"糊弄过去。但工程上必须拆开看:到底是单个请求慢,还是请求太多处理不过来,还是同时连接太多把数据库或函数额度打满?

延迟、吞吐量、并发不是三个孤立指标。它们常常一起出现,也常常互相影响。

---

## 一、一句话解释

### 1.1 延迟:用户等一次请求的时间

延迟通常指一次操作从发起到得到结果需要多久,例如:

```text
打开首页:
  800ms

调用登录接口:
  300ms

生成报表:
  20s

上传图片:
  5s
```

对用户来说,延迟就是"等多久"。对系统来说,延迟可能由很多部分组成:

- DNS 查询时间
- TLS 握手时间
- CDN 命中或回源时间
- 后端处理时间
- 数据库查询时间
- 第三方 API 响应时间
- 文件上传和下载时间

所以"接口慢"不一定是后端代码慢,也可能是数据库慢、网络远、缓存没命中、第三方服务慢。

### 1.2 吞吐量:单位时间能处理多少

吞吐量看的是系统处理能力,常见表达是:

- 每秒请求数,例如 100 RPS
- 每分钟任务数,例如每分钟处理 1000 条队列消息
- 每秒写入量,例如每秒写入 500 条日志
- 每秒传输数据量,例如 50 MB/s

吞吐量高,不代表每个用户都快。一个系统可以每秒处理很多请求,但单个请求仍然很慢。反过来,一个系统在低流量时响应很快,但流量一上来吞吐量不够,就开始排队和超时。

### 1.3 并发:同一时刻有多少事在进行

并发看的是"同时进行中的数量":

- 同时在线用户数
- 同时打开的 HTTP 连接数
- 同时执行的 Serverless 函数数
- 数据库同时连接数
- 队列中同时处理的任务数
- 浏览器同时下载的静态资源数

并发最容易撞到硬限制。比如数据库最大连接数只有 60,你的 API 每个请求都新建连接,那 100 个同时请求就可能让系统开始报错。

## 二、放在系统哪里

### 2.1 用户请求链路里的性能指标

一次用户请求可能穿过这些层:

```text
浏览器
  -> DNS
  -> CDN / WAF
  -> 负载均衡 / 网关
  -> 应用服务 / Serverless 函数
  -> 数据库 / 缓存 / 队列
  -> 第三方 API
```

每一层都可能影响延迟、吞吐量和并发:

```text
DNS:
  影响首次访问延迟

CDN:
  命中缓存时降低延迟和源站压力
  回源过多时可能把压力打到后端

网关:
  可能限制每秒请求数或连接数

应用服务:
  受 CPU、内存、线程池、函数执行时间影响

数据库:
  受连接数、慢查询、锁、IO、存储容量影响

队列:
  受消费速度、积压数量、重试策略影响

第三方 API:
  受对方限流和响应速度影响
```

### 2.2 三个指标之间的关系

一个简单理解:

```text
并发上升:
  同时处理的请求变多

系统资源接近上限:
  请求开始排队

排队变长:
  延迟上升

超时和错误变多:
  有效吞吐量下降
```

这就是为什么一个网站平时很快,一上热搜就慢到不可用。不是每个请求突然变复杂了,而是同时来的请求太多,队列、连接池、函数并发、数据库连接数被打满。

### 2.3 平均值经常骗人

性能指标里最危险的是只看平均延迟。

```text
100 个请求:
  95 个请求 100ms
  5 个请求 5s

平均值:
  看起来可能还可以

用户体验:
  那 5 个请求的用户非常痛苦
```

所以常见监控会看 P50、P95、P99:

```text
P50:
  一半请求比这个值快

P95:
  95% 请求比这个值快

P99:
  99% 请求比这个值快
```

小团队不一定要一开始追求复杂性能分析,但至少要知道:平均值不能代表尾部用户体验。

## 三、常见套餐和使用限制

### 3.1 云服务常限制的不是一个维度

免费套餐或低价套餐常见限制包括:

- 每秒请求数
- 每月请求数
- 最大并发函数数
- 单次函数执行时长
- CPU 和内存规格
- 数据库最大连接数
- 数据库读写次数或计算单元
- 队列消息吞吐量
- 对象存储上传下载速率
- 出站流量
- 构建分钟和部署次数

你只看"请求数免费 100 万"是不够的。真正出问题时,可能不是月请求数用完,而是某一小时并发太高,数据库连接数先爆了。

### 3.2 Serverless 最容易被忽略的限制

Serverless 平台常见性能边界:

```text
冷启动:
  长时间没人访问后,第一次请求更慢

最大执行时间:
  超过时间直接中断或返回超时

并发限制:
  同时执行函数数量有限

出站连接:
  每个函数都连数据库,可能快速打满连接数

区域距离:
  函数离用户或数据库太远,延迟增加
```

Serverless 很适合小团队起步,但不代表可以不理解性能限制。它只是把服务器管理隐藏了,没有把物理限制消除。

### 3.3 数据库连接数是小团队常见瓶颈

很多早期项目真正先坏的不是 CPU,而是数据库连接:

```text
错误做法:
  每个 API 请求新建一个数据库连接
  请求结束后没有及时释放
  Serverless 函数并发一高,连接数瞬间打满

结果:
  新请求排队
  登录超时
  保存失败
  API 返回 500 或 502
```

免费数据库的连接数通常有限。升级套餐可以缓解,但代码里的连接复用、连接池、读写模式同样重要。

### 3.4 限流不一定是坏事

限流看起来像限制,其实也是保护:

- 防止单个用户刷爆系统
- 防止爬虫消耗资源
- 防止下游数据库被打穿
- 防止第三方 API 额度被瞬间用完
- 防止账单失控

没有限流的小系统,在流量突增时可能不是慢一点,而是直接崩掉。

## 四、小团队建议

### 4.1 先监控核心路径的 P95 延迟

不要一开始就做复杂压测平台。先问:

```text
用户最常做的 3 个动作是什么?
  打开首页?
  登录?
  搜索?
  创建订单?
  上传文件?

这些动作的 P95 延迟是多少?
错误率是多少?
高峰时是否明显变慢?
```

如果核心路径 P95 延迟长期偏高,用户会觉得产品不稳定,即使服务没有彻底挂掉。

### 4.2 把"慢任务"移出请求链路

很多性能问题来自把重活放在同步请求里:

- 生成大报表
- 批量发送邮件
- 调用多个第三方 API
- 图片压缩和转码
- AI 推理或长文本处理
- 大文件导入导出

更稳的做法是:

```text
用户请求:
  快速接收任务
  返回任务已创建

后台队列:
  慢慢处理
  失败可重试

前端:
  轮询或通知用户结果
```

这样可以降低接口延迟,也能避免单个请求超时。

### 4.3 不要只靠升级套餐解决性能问题

升级套餐有用,但不是万能。先确认瓶颈在哪里:

```text
CPU 满:
  可能需要更大实例或优化计算

内存满:
  可能有缓存过大或内存泄漏

数据库慢:
  可能缺索引、查询太重、连接池不合理

CDN 回源多:
  可能缓存策略不对

第三方 API 慢:
  可能需要异步化、缓存或降级

并发打满:
  可能需要限流、队列、连接池或扩容
```

先定位,再花钱。否则你可能买了更贵的实例,瓶颈仍然在数据库或第三方服务。

### 4.4 给小项目的最低配置

一个准备上线的小项目,至少应该知道:

- 首页和核心 API 的 P95 延迟
- 高峰时每秒请求数大概是多少
- 数据库最大连接数和当前连接数
- Serverless 函数最大执行时间和并发限制
- 哪些接口可能调用第三方服务
- 哪些任务应该异步处理
- 是否有基本限流和超时设置

这些信息不需要你变成性能专家,但能让你在出问题时知道先看哪里。

---

## 五、一句话总结

**延迟回答"一次请求等多久",吞吐量回答"一段时间能处理多少",并发回答"同一时刻扛多少";小团队先看核心路径 P95、数据库连接数和高峰并发,再决定优化代码还是升级套餐。**
