# WAF 与 DDoS 防护:它们能挡什么,挡不了什么

> 一句话解释:**WAF 主要防应用层的恶意请求,DDoS 防护主要防大流量把你的网络和服务压垮**。

很多人把安全产品想成一个"开了就安全"的开关。现实不是这样。WAF 和 DDoS 防护挡的是不同问题,而且都只能降低风险,不能替你写安全的代码,也不能保证你的业务永远不被打断。

WAF,Web Application Firewall,更像一个站在 HTTP 入口前的规则检查员。它会看请求路径、参数、Header、Body、User-Agent、IP、国家地区、请求频率,判断这个请求像不像 SQL 注入、XSS、路径扫描、恶意爬虫、漏洞探测。

DDoS 防护更像网络层的抗压设施。攻击者不一定关心你的业务逻辑,只是用大量流量、大量连接或大量请求把你的带宽、连接数、CPU、上游线路压满,让正常用户进不来。

最容易混淆的是:**WAF 不等于 DDoS 防护,DDoS 防护也不等于业务安全**。有人疯狂撞你的登录接口,这可能需要 WAF、限流、验证码、账号风控一起做;有人用超大流量打你的 IP,这更接近 DDoS;有人拿到了你的管理员密码,这两个都救不了你。

---

## 一、放在系统哪里

常见链路是:

```text
用户 / 攻击者
  -> DNS
  -> CDN / DDoS 清洗 / WAF
  -> 反向代理或网关
  -> 应用服务
  -> 数据库
```

WAF 通常放在 CDN、负载均衡、API Gateway 或云厂商安全服务上。它越靠前,越能在请求到达源站前拦截。

DDoS 防护也越靠前越有效。原因很简单:如果攻击流量已经打到你的云服务器网卡或机房出口,你的应用再聪明也来不及了。真正的大流量攻击需要在云厂商、CDN、运营商或清洗中心这一层处理。

一个真实场景:

```text
你做了一个小型 SaaS:
  - 官网和文档走 CDN
  - API 在 api.example.com
  - 登录接口是 /api/login
  - 管理后台是 /admin

某天日志里出现:
  - 大量访问 /wp-admin.php
  - 大量访问 /.env
  - 请求参数里有 "' OR 1=1"
  - 同一个 IP 每秒打 /api/login 几十次

这些更像 WAF、Bot 防护、限流要处理的问题。

另一天出现:
  - 服务器带宽被打满
  - SSH 都连不上
  - 正常请求还没到应用日志就失败
  - 云厂商提示流量异常

这更像 DDoS 或网络层攻击。
```

判断问题在哪一层很重要。如果应用日志里能看到大量恶意请求,WAF 和应用限流通常还能发挥作用。如果应用日志里什么都没有,但用户访问超时、服务器带宽满了,问题可能发生在更前面的网络层。

## 二、常见套餐和使用限制

WAF 免费或入门能力通常包括一些托管规则,例如常见漏洞扫描拦截、恶意路径拦截、基础 Bot 识别、国家地区规则、简单速率限制。但限制经常出现在这些地方:

- 自定义规则数量。
- 速率限制规则数量。
- 托管规则集是否完整。
- Bot 管理是否可用。
- 日志可见性和保留时间。
- 是否支持按账号、Token、路径、Header 做精细策略。
- 误杀后的调试工具是否足够。

DDoS 防护的限制通常更硬:

- 免费防护只覆盖常见小规模攻击。
- 超过清洗能力可能被限速、黑洞或要求升级。
- 只保护经过 CDN 或高防入口的流量,不一定保护源站真实 IP。
- 某些协议、端口、区域可能不在默认保护范围内。
- 大流量攻击的日志和报表可能需要更高套餐。

小团队特别容易踩的坑是**源站 IP 暴露**。你把域名接到 CDN,以为所有流量都经过防护。但如果源站 IP 曾经被 DNS 解析过、写在历史记录里、出现在邮件 Header、GitHub 配置、错误页、第三方回调白名单里,攻击者可以绕过 CDN 直接打你的服务器。

另一个坑是**把 WAF 当成修漏洞工具**。例如你的接口没有权限校验,用户改一个 `user_id` 就能看别人订单。WAF 很难知道这是业务越权。再比如你把管理后台放在公开路径,密码又弱,WAF 也不能保证一定拦住撞库。

## 三、小团队建议

小团队不需要一开始购买昂贵的安全套件,但应该尽早建立入口防护的基本层次:

```text
域名
  -> CDN / WAF
  -> 反向代理 / 网关
  -> 应用限流
  -> 权限校验
  -> 日志和告警
```

最低限度建议:

- 所有公网服务都放在明确的入口后面,不要随便暴露测试端口。
- 源站只允许 CDN、负载均衡或网关访问,能用安全组限制就限制。
- 登录、注册、短信、邮件、支付、搜索接口必须有限流。
- 管理后台加额外保护,例如固定入口、MFA、IP allowlist、单独域名。
- 开启基础 WAF 托管规则,但先用观察模式或低强度策略看误杀情况。
- 保留访问日志,至少能看到 IP、路径、状态码、延迟、User-Agent。
- 设置流量、5xx、登录失败、验证码失败的告警。

什么时候该升级?

- 产品开始收费,攻击会直接影响收入。
- 有登录、支付、提现、邀请码、优惠券等容易被刷的功能。
- 曾经被扫、被撞库、被恶意注册或被 DDoS。
- 有企业客户,需要安全报表和审计。
- 源站不能轻易停机,需要更高等级的 DDoS 清洗和应急支持。

不要把防护只放在云厂商一层。一个更可靠的思路是:

- 网络层:不要暴露不必要端口。
- 入口层:CDN/WAF/网关做第一道拦截。
- 应用层:权限校验、参数校验、业务限流必须自己做。
- 账号层:MFA、强密码、登录异常检测。
- 运维层:日志、告警、应急预案。

如果只能先做三件事,优先做:

1. 隐藏和限制源站入口。
2. 给高风险接口加限流。
3. 打开日志和告警。

---

## 四、一句话总结

**WAF 负责尽量拦住可识别的恶意 HTTP 请求,DDoS 防护负责尽量吸收大流量冲击;它们是入口防线,不是业务安全、权限设计和成本告警的替代品。**
