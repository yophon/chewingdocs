# Cloudflare 免费套餐拆解:适合什么,不适合什么

## 一句话解释

Cloudflare 免费套餐的核心价值,是把 DNS、CDN、HTTPS、基础防护和部分边缘能力放到网站入口层,让小项目用很低成本获得一层全球入口。

它最容易被误解成"用了 Cloudflare 就万事大吉"。Cloudflare 很适合做入口和缓存,但它不是万能后端、不是数据库替代品,也不是所有场景都能无限免费承接流量。

看 Cloudflare 免费套餐时,不要只问"免费额度是多少",而要问:我的流量会不会命中缓存,请求会不会回源,有没有动态计算,有没有上传大文件,有没有安全和合规要求。

---

## 放在系统哪里

Cloudflare 通常站在用户和源站之间:

```text
用户浏览器
  -> Cloudflare DNS
  -> Cloudflare CDN / HTTPS / WAF / Rules / Workers
  -> 源站:VPS / 对象存储 / Pages / API / 数据库后端
```

它能做几类事:

- DNS 托管:把域名解析交给 Cloudflare
- HTTPS 入口:给用户到 Cloudflare 的连接提供证书
- CDN 缓存:缓存静态资源,减少源站压力
- 反向代理:隐藏源站 IP,统一入口规则
- 基础安全:拦截一部分恶意请求、爬虫、异常流量
- 边缘计算:用 Workers、Pages Functions 等在边缘执行轻量逻辑
- 对象存储和媒体能力:例如 R2、Images、Stream 等组合产品

这意味着 Cloudflare 更像"网站入口层和边缘层",不是传统意义上的一台服务器。它能帮你挡掉很多重复请求,但你的业务数据、登录状态、支付逻辑、后台任务,仍然要有明确的后端和数据库设计。

## 常见套餐/使用限制

### 查看方法

拆 Cloudflare 免费套餐时,建议按产品分别看,不要只看 Free 计划总页:

1. 看 DNS、CDN、SSL/TLS、Rules、WAF 的 Free plan 页面
2. 如果用了 Pages,单独看 Pages 的构建、部署、带宽、团队协作限制
3. 如果用了 Workers,单独看请求数、CPU 时间、运行时限制、绑定资源限制
4. 如果用了 R2,单独看存储量、Class A/B 操作、出站规则和兼容性
5. 如果用了 Images、Stream、Queues、KV、D1、Durable Objects,逐个看对应的 Free / Paid / Limits 页面
6. 重点查 Overage、Abuse、Fair use、Rate limiting、Terms 这些词

Cloudflare 的产品线很宽,同一个账号里可能同时用了 CDN、Pages、Workers、R2、KV、D1。它们的限制不是一个总额度,而是分散在不同产品里。

### 常见限制维度

Cloudflare 免费套餐常见限制维度包括:

- DNS 和域名管理:记录管理、代理状态、部分高级规则
- CDN 缓存:缓存规则数量、页面规则、缓存命中率、回源行为
- 安全能力:WAF 托管规则、高级 Bot 管理、速率限制、细粒度策略
- Workers:请求数、CPU 时间、执行时间、内存、并发和运行时 API
- Pages:构建次数、构建时长、预览部署、团队协作和项目限制
- R2:存储容量、读写操作、生命周期、兼容性、数据访问路径
- KV / D1 / Durable Objects:读写次数、存储容量、延迟、事务或一致性边界
- 日志与分析:日志保留时间、实时日志、高级分析能力
- 支持与 SLA:免费计划通常不能期待企业级支持和正式 SLA

这些维度里,最关键的是"缓存是否命中"。如果大部分请求都被 Cloudflare 缓存住,源站压力和成本会明显下降。如果都是动态请求、登录态请求、不可缓存 API,Cloudflare 只是入口代理,不能自动帮你省掉后端成本。

## 小团队建议

Cloudflare 适合:

- 个人网站、文档站、营销页、博客、静态站
- 需要便捷 DNS 托管和 HTTPS 的小项目
- 需要给 VPS 或对象存储加一层 CDN 缓存的网站
- 访问以静态资源为主、动态接口较少的产品
- 想用 Workers 做轻量 API、重定向、A/B 路由、边缘校验
- 想用 R2 放公开下载资源或网站静态资源

Cloudflare 不适合直接当成:

- 重计算后端
- 长时间运行任务平台
- 强一致数据库
- 复杂后台队列系统
- 高度定制的企业安全平台
- 所有动态 API 的免费无限代理

主要风险点:

- 缓存规则写错:用户看到旧页面,或者本该缓存的资源一直回源
- SSL 模式选错:浏览器到 Cloudflare 是 HTTPS,但 Cloudflare 到源站不安全或握手失败
- 源站 IP 暴露:攻击者绕过 Cloudflare 直接打源站
- 动态请求过多:CDN 没有命中,源站和后端账单仍然上涨
- Workers 滥用:把重业务逻辑放到边缘,碰到运行时和 CPU 限制
- 免费计划支持有限:遇到复杂故障时不能指望专属响应
- 平台规则变化:免费套餐、滥用判定、产品边界可能调整,需要定期复查官方限制页

小团队比较稳的用法是:

- DNS 放 Cloudflare
- 静态资源尽量缓存
- 源站只暴露给 Cloudflare 或加防火墙
- 动态 API 单独做限流和鉴权
- Workers 只做轻逻辑,不要承载核心复杂后端
- 上线后定期看缓存命中率、回源流量、错误率和安全事件

---

## 一句话总结

Cloudflare 免费套餐最适合做网站入口、DNS、HTTPS、CDN 缓存和轻量边缘逻辑,但不能把它误当成无限后端;真正要盯的是缓存命中、回源流量、边缘运行限制和安全配置。
