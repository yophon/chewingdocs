# 账号安全:Root 账号、MFA、子账号、最小权限

## 一句话解释

**账号安全是保护云资源入口的第一道门:Root 账号只用于最高权限管理,MFA 用来防止密码泄露后被直接接管,子账号和最小权限用来限制日常操作的破坏范围**。

很多人把账号安全理解成"密码复杂一点"。这只对了一小半。云账号和普通网站账号不一样,它后面连着服务器、数据库、域名、对象存储、账单、密钥和生产数据。一个云账号被接管,不是丢一个登录状态,而是可能被删除资源、公开数据、挖矿刷账单、创建后门用户。

Root 账号最容易被误用。Root 不是"我自己的账号",而是这个云账户的最高权限身份。它通常能改账单、关停服务、改安全设置、创建管理员、删除关键资源。日常部署、查看日志、改 DNS、上传文件,都不应该长期使用 Root。

## 放在系统哪里

账号安全不在用户访问链路里,而在管理链路里:

```text
团队成员
  -> 登录云平台 / 部署平台 / 域名平台
  -> 修改 DNS、服务器、数据库、存储、CI/CD、密钥
  -> 影响线上系统和账单
```

用户访问你的网站时,通常不会碰到 Root 账号。但你每一次进控制台、跑 CLI、配置 GitHub Actions、给同事开权限,都在使用管理身份。管理身份一旦出问题,前面的 CDN、WAF、HTTPS 都救不了你,因为攻击者已经站在控制台里面了。

一个典型事故是:

```text
开发者把云平台主账号密码存在浏览器里
  -> 电脑中木马或密码库泄露
  -> 攻击者登录控制台
  -> 没有 MFA,直接进入
  -> 创建新的访问密钥
  -> 删除日志、开高配实例、下载对象存储数据
```

如果 Root 账号启用了 MFA,并且日常只用低权限子账号,这个事故的破坏范围会小很多。密码泄露仍然严重,但不一定等于整个平台失守。

## 常见套餐和使用限制

账号安全功能在不同平台里的免费程度差异很大。MFA 通常免费,但更细的团队管理、审计日志、SSO、强制 MFA、细粒度权限策略,可能放在团队版、企业版或付费组织计划里。

小团队要特别看这些限制:

- 能不能给每个人单独创建账号,还是多人共用一个登录.
- 能不能强制所有成员开启 MFA.
- 能不能按项目、资源、环境分配权限.
- 能不能查看谁在什么时候改了配置、创建了密钥、删除了资源.
- 离职或外包结束后,能不能一键移除成员权限.
- 免费计划是否只支持一个管理员,不支持角色分工.

很多便宜平台不是没有安全能力,而是团队权限模型很弱。结果就是所有人共用一个账号,部署、账单、数据库、域名全在一起。这个模式在 demo 阶段可以凑合,一旦有真实用户和收入,就是明显风险。

还有一种隐形限制是恢复能力。账号被盗后,你能否证明资源是被谁改的?能否找回被删除的项目?能否冻结异常账单?这些能力不一定属于免费套餐。安全不是等出事后找客服,而是提前让事故不要扩大。

## 小团队建议

最低限度先做四件事:

- Root 账号开启强 MFA,优先使用认证器 App 或硬件安全密钥,不要只依赖短信.
- Root 账号不用于日常开发、部署和 API 调用,只在账单、安全、组织级配置时使用.
- 每个成员使用自己的账号,不要多人共用一个邮箱和密码.
- 按最小权限分配角色:能看日志的人不一定能删数据库,能部署的人不一定能改账单.

最小权限不是一开始就写几十条复杂策略,而是避免"所有人都是管理员"。独立开发者也可以把身份分开:

```text
Root / Owner
  -> 只做账单、安全设置、紧急恢复

日常管理员
  -> 管理项目配置、域名、部署

应用运行身份
  -> 只访问需要的数据库、对象存储、队列

只读身份
  -> 查看日志、监控、账单,不能改资源
```

MFA 也要有恢复方案。只开启 MFA,但恢复码只存在同一台手机里,手机丢了就会变成账号灾难。恢复码应该离线保存,并且至少让可信负责人知道恢复流程。小团队可以把恢复码放进团队密码库,限制少数人可见。

当团队超过 3-5 人,或开始让外包、运营、客服进入后台时,就应该认真整理权限。权限不是信任问题,而是降低误操作和账号泄露后的损失。一个实习生不应该能删除生产数据库,一个 CI 任务不应该能修改账单信息。

## 一句话总结

**云账号安全的底线是 Root 少用、MFA 必开、成员分账号、权限按需给;真正的安全不是相信每个人都不会出错,而是让一次密码泄露或误操作不要毁掉整个系统**。
