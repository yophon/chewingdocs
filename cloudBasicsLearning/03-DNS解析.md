# DNS 解析:记录类型,TTL,权威 DNS,递归 DNS

## 一句话解释

DNS 解析就是把域名翻译成网络能访问的地址,让浏览器知道 `example.com` 应该去找哪台服务器,哪个 CDN 或哪个邮件服务.

它最容易被误解成"我在控制台里填一条记录,全世界马上生效".真实情况是,DNS 是一个分层查询和缓存系统,你改的是权威 DNS 里的答案,但用户拿到的是递归 DNS 缓存后的答案.

最小心智模型是:

```text
权威 DNS = 真正保存你域名解析记录的地方
递归 DNS = 用户网络运营商,公共 DNS 或公司网络帮用户查询并缓存答案的地方
TTL = 这份答案可以被缓存多久
```

所以你改了 DNS 后,有些人马上看到新网站,有些人还访问旧网站,不一定是云厂商坏了,可能只是不同递归 DNS 的缓存还没过期.

## 放在系统哪里

DNS 发生在真正访问网站之前.

一次访问链路可以简化成这样:

```text
浏览器输入 https://www.example.com
  -> 查询 www.example.com 的 DNS 记录
  -> 得到 IP,CNAME 或 CDN 地址
  -> 建立 TCP / TLS 连接
  -> 发送 HTTP 请求
  -> 服务器返回页面
```

DNS 不负责返回网页内容.它只回答"这个名字应该去哪里".

常见记录可以按场景理解:

- A 记录:把域名指向一个 IPv4 地址,例如 `203.0.113.10`.
- AAAA 记录:把域名指向一个 IPv6 地址.
- CNAME 记录:把一个名字指向另一个名字,例如 `www.example.com -> your-app.vercel.app`.
- MX 记录:告诉别人这个域名的邮件应该投递到哪里.
- TXT 记录:常用于域名验证,邮件防伪,第三方服务校验.
- NS 记录:告诉上层 DNS,这个域名应该由哪些权威 DNS 服务器管理.

独立开发者最常接触的是 A,CNAME,TXT.比如:

```text
把根域名 example.com 指到一台云服务器:
  A example.com -> 203.0.113.10

把 www 指到前端托管平台:
  CNAME www.example.com -> project.pages.dev

给邮件服务做验证:
  TXT example.com -> 一段服务商要求你添加的字符串
```

如果你不理解这些记录的用途,就很容易在同一个主机名上乱加记录.比如 `www.example.com` 同时配置 A 和 CNAME,很多 DNS 服务会直接拒绝,因为 CNAME 的意思是"这个名字完全交给另一个名字",不能再同时塞别的答案.

## 常见套餐和使用限制

DNS 看起来很小,但它有几个非常实际的限制.

### 1. 记录数量和高级能力

免费 DNS 通常足够管理小网站,但可能限制记录数量,负载均衡,健康检查,地理线路,审计日志,API 调用频率.

个人项目一般用不到高级能力.真正需要关注的是:记录能不能稳定解析,控制台是否清晰,是否支持基本的 DNSSEC,是否有变更记录.

### 2. TTL 不是越低越好

TTL 表示递归 DNS 可以缓存多久.例如 TTL 是 300 秒,理论上最多 5 分钟后用户会重新查询.

上线迁移时,你可能想把 TTL 调低,这样切换 IP 更快.但 TTL 太低也会增加 DNS 查询量,在大流量场景下可能带来额外成本或压力.

小团队常见做法:

```text
平时 TTL 设置为 5-30 分钟
迁移前一天调低到 1-5 分钟
迁移完成且稳定后再调回正常值
```

注意:如果你迁移前才把 TTL 调低,旧的高 TTL 可能已经被缓存了.TTL 调整也需要提前生效.

### 3. 根域名和 CNAME 的限制

很多平台会让你把 `www.example.com` CNAME 到它们的域名,这很简单.但根域名 `example.com` 传统上不能直接配置 CNAME,因为根域名还需要承载 NS,SOA 等记录.

一些 DNS 服务提供 ALIAS,ANAME 或 CNAME flattening,可以在体验上解决这个问题.但这些不是 DNS 标准里最基础的记录类型,不同厂商叫法和行为不完全一样.

这就是为什么很多小团队会选择:

```text
example.com      -> 跳转到 www.example.com
www.example.com  -> CNAME 到托管平台
```

这样简单,稳定,排障也容易.

### 4. DNS 故障不是网站故障

有时候服务器是好的,但用户还是打不开网站.原因可能是:

- 权威 DNS 配错.
- NS 改到了另一个平台.
- 递归 DNS 缓存了旧答案.
- DNSSEC 配置错误导致验证失败.
- 某些地区的网络无法访问某个 DNS 服务.

排障时不要只看服务器日志.如果服务器没有任何请求进来,先查 DNS.

## 小团队建议

DNS 管理要追求"少,清楚,可恢复".

最低限度建议:

- 明确一个主 DNS 托管商,不要在多个平台同时改记录.
- 记录根域名,`www`,`api`,`admin`,邮件相关记录分别指向哪里.
- 对生产域名的 DNS 变更写简单变更备注.
- 迁移前提前降低 TTL.
- 不要随手删除 TXT,MX 记录,它们可能关系到邮件,证书或第三方验证.
- 为正式业务保留 `www` 和根域名的统一跳转策略.

真实场景里,新手常说"DNS 还没生效".这句话太粗.你应该继续问三个问题:

```text
1. 当前域名的 NS 指向谁?
2. 权威 DNS 里查到的答案是什么?
3. 用户本地递归 DNS 查到的答案是什么?
```

如果权威 DNS 已经是新答案,但用户本地还是旧答案,大概率是缓存.如果权威 DNS 里就是错的,那就是你配置错了.

什么时候基础 DNS 够用?

大多数独立项目,落地页,SaaS 早期,一个普通 DNS 托管商就够.你需要的是稳定解析和清晰控制台,不是复杂线路调度.

什么时候升级?

当你需要多地域容灾,健康检查自动切换,企业审计,细粒度权限,超高查询量或严格可用性承诺时,再考虑更专业的 DNS 方案.

## 一句话总结

DNS 解析不是"填一条记录就完事",而是权威 DNS 给答案,递归 DNS 缓存答案,TTL 控制缓存时间;小团队把 NS,记录类型和 TTL 搞清楚,就能避开大多数上线事故.
