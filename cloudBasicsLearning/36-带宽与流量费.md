# 带宽与流量费:为什么出口流量最容易超预算

## 一句话解释

**带宽描述单位时间内能传多少数据,流量描述一段时间内实际传了多少数据;云账单里最需要警惕的通常是出站流量**。

它最容易和"网速"混淆。带宽像水管粗细,流量像一段时间流过的水量。带宽高,说明同一时间能传得快;流量大,说明总共传了很多数据。云服务里常见的收费点不是你家宽带那种固定月费,而是用户从云平台下载了多少数据。

先记住三个方向:

```text
入站流量:用户上传到云服务,例如上传图片、提交表单
出站流量:云服务发给用户,例如下载页面、图片、视频、API 响应
内部流量:同一云厂商、同一区域内服务之间通信
```

很多云产品对入站流量比较宽松,对出站流量更敏感。原因很简单:云厂商把数据发到互联网,需要承担带宽和网络成本。对小团队来说,账单里最容易突然变大的,往往也是出站流量。

一句话先记住:**用户下载你的内容,才是流量费最常见的来源**。

## 放在系统哪里

带宽和流量贯穿整个用户访问链路:

```text
用户浏览器
  -> CDN 边缘节点
  -> 反向代理 / 网关
  -> 应用服务
  -> 对象存储 / 数据库
```

最常见的流量发生在这些位置:

| 位置 | 产生什么流量 | 典型风险 |
| --- | --- | --- |
| CDN 到用户 | HTML、JS、CSS、图片、下载文件 | 图片太大、热门传播、爬虫 |
| 源站到 CDN | CDN 回源拉取资源 | 缓存命中率低、缓存频繁失效 |
| 对象存储到用户 | 文件下载、图片访问 | 外链传播、热链、公开桶 |
| 应用到用户 | API 响应、SSR 页面 | 响应过大、分页不当 |
| 跨区域服务通信 | 数据库复制、备份、跨地域调用 | 误把内部调用放到公网或跨区 |
| 日志和监控上报 | 大量日志、指标、追踪数据 | debug 日志没关、异常循环 |

一个静态网站也会产生流量:

```text
用户访问首页
  -> 下载 index.html
  -> 下载 app.js
  -> 下载 style.css
  -> 下载 logo.png
  -> 下载多张文章图片
```

如果首页总资源是 5 MB,1 万次访问就是大约 50 GB 出站数据。这里不需要后端、不需要数据库,只要用户下载文件,就会产生出站流量。

再看一个对象存储例子:

```text
你把一个 100 MB 安装包放到对象存储
某个论坛分享了下载链接
一天下载 10000 次
实际出站流量约 1000 GB
```

文件存储本身可能不贵,真正放大的通常是下载流量。

## 常见套餐和使用限制

带宽和流量相关的限制,常见有几种写法。

第一种是按出站流量计费。文档里可能叫 bandwidth、data transfer、egress、traffic。你要确认它统计的是:

- CDN 到互联网的流量.
- 源站到互联网的流量.
- 对象存储到互联网的流量.
- 跨区域或跨可用区的流量.
- 是否区分国内、海外、不同区域.

第二种是固定带宽。云服务器常见公网带宽上限,例如你选择一个最大出口速率。固定带宽的好处是成本可控,坏处是高峰时可能慢。按量流量的好处是弹性大,坏处是账单更不确定。

第三种是套餐内包含一定流量,超额后计费或限速。看这种套餐时,不要只看包含多少,要看超额行为:

```text
超额后继续服务并额外计费
超额后限速
超额后暂停服务
超额后要求升级套餐
超额后需要人工开通
```

第四种是 CDN 回源和缓存规则。CDN 不是免费流量消除器,它只是把重复请求尽量挡在边缘节点。如果缓存配置不好,每次都回源,你可能同时付 CDN 出站流量和源站出站流量,源站还会承受压力。

容易踩坑的场景包括:

- 图片、视频、PDF、安装包直接放在公开地址,没有大小控制.
- 前端打包产物过大,每个用户都下载几 MB 甚至几十 MB JS.
- API 一次返回大量数据,没有分页、压缩和字段裁剪.
- SSR 页面每次都回源,CDN 缓存命中率很低.
- 对象存储桶公开读,文件链接被外部网站热链.
- 开启跨区域备份或复制,但没注意跨区域流量.
- debug 日志把大对象、请求体、响应体全部上报.
- 爬虫、扫描器、恶意请求持续下载大资源.

查看方法上,至少要看四个地方:

```text
1. 产品 pricing 页面:出站流量是否收费,套餐包含多少
2. limits / quotas:带宽峰值、单文件大小、请求频率限制
3. usage 面板:按服务、区域、域名、bucket 拆分的流量
4. CDN / 对象存储日志:哪些 URL 消耗最多流量
```

如果平台只给总流量,无法看 URL 或域名维度,排查成本会很高。生产项目最好保留能定位大流量来源的访问日志或分析工具。

## 小团队建议

小团队控制流量费,核心不是省每一个字节,而是防止少数大资源和异常访问把账单放大。

第一,上线前检查资源体积:

- 首页首屏不要依赖超大图片和视频.
- 图片使用合适尺寸,不要把原图直接给移动端.
- JS bundle 做基本拆分,后台页面不要塞进公开首页.
- API 响应启用压缩,列表接口做分页.
- 下载文件单独放域名或路径,方便统计和限流.

第二,把静态资源放到合适的位置。网站静态文件适合走 CDN 或静态托管,用户上传文件适合对象存储加 CDN,后端服务不要直接扛大量文件下载。云服务器适合跑逻辑,不适合当大文件分发中心。

第三,设置防热链和访问控制:

- 对私有文件使用签名 URL.
- 对大文件设置过期链接.
- 对公开图片开启防盗链或 referer 策略,但不要把它当成安全边界.
- 对下载接口加登录、频率限制或验证码.
- 对异常 IP、异常路径、异常 user agent 做基本拦截.

第四,盯住 CDN 缓存命中率。命中率低时,用户请求会频繁打到源站:

```text
缓存命中率高:
  用户 -> CDN,源站压力低

缓存命中率低:
  用户 -> CDN -> 源站,源站流量和计算都上升
```

静态资源应该带合理的 cache-control。带 hash 的 JS、CSS、图片可以缓存很久;HTML 可以短缓存或按需刷新;用户私有数据不要被公共 CDN 错误缓存。

第五,设置预算和用量告警:

- 月度出站流量达到某个比例时提醒.
- 某个域名、bucket、路径流量异常增长时提醒.
- CDN 回源量异常升高时提醒.
- 单日流量超过过去 7 天平均值很多时提醒.
- 大文件下载量突然上升时提醒.

第六,准备降级动作。比如临时关闭大文件下载、替换首页视频、降低图片质量、对公开 API 加更严格限流、把热门资源迁到更适合分发的平台。流量异常时不要只看账单,要能快速减少数据输出。

一个保守原则是:**凡是可能被用户、爬虫或外部网站重复下载的内容,都要提前知道它在哪里统计、在哪里限速、在哪里关停**。

## 一句话总结

**带宽决定传得有多快,流量决定一段时间传了多少;小团队最该防的是出站流量被大文件、缓存失效、热链和爬虫放大**。
