# 数据库服务:托管数据库、连接数、读写额度和备份常识

很多独立开发者刚开始会把数据库理解成"一个存数据的地方"。这个理解太粗了。真正上线以后,数据库不是一个普通组件,而是整个系统里最难随便替换、最怕误删、最容易成为瓶颈的组件。

> 一句话先记住:托管数据库是云厂商帮你运行、备份、监控、升级一部分数据库基础设施,但数据模型、索引、连接使用、权限控制、慢查询和恢复演练仍然是你的责任。

---

## 一、一句话解释

数据库服务通常指云厂商提供的托管数据库,例如:

- 托管 PostgreSQL
- 托管 MySQL
- 托管 MongoDB
- Serverless Postgres
- Firebase / Firestore
- Supabase Postgres
- PlanetScale 这类 MySQL 兼容服务
- 阿里云 RDS / 腾讯云数据库 / AWS RDS

它和"自己在服务器上装数据库"的区别是:

```text
自建数据库:
  你负责安装、升级、备份、监控、磁盘、主从、故障恢复

托管数据库:
  云厂商负责一部分基础设施
  你负责数据怎么用、怎么连、怎么查、怎么保护
```

托管数据库不是"不用懂数据库"。它只是把一部分运维工作交给平台。你依然要知道:

- 最大连接数是多少
- 存储空间是否会自动扩容
- 备份保留多久
- 是否支持时间点恢复
- 慢查询怎么看
- 索引是否合理
- 读写是否有额度或性能上限
- 升级套餐会不会停机
- 数据在哪个地域

最常见的误解是:

```text
用了 Supabase / Firebase / RDS = 数据库问题由平台负责
```

不对。平台负责数据库服务尽量可用,不负责帮你写正确的 SQL,不负责防止你的代码创建 10000 个连接,也不负责替你判断哪些数据该加密、哪些字段不能进日志。

---

## 二、放在系统哪里

数据库通常放在后端之后,不应该直接暴露给普通用户浏览器:

```text
用户浏览器
  -> 前端页面
  -> 后端 API / Serverless Function
  -> 数据库
```

如果有对象存储,常见关系是:

```text
数据库:保存用户、订单、文件记录、权限、状态
对象存储:保存头像、附件、导出文件、备份文件
```

如果有缓存,常见关系是:

```text
后端先查 Redis
  -> 命中:直接返回
  -> 未命中:查数据库
  -> 把结果写入 Redis
```

如果有队列,常见关系是:

```text
用户提交任务
  -> 后端写数据库记录
  -> 发送消息到队列
  -> Worker 异步处理
  -> 更新数据库状态
```

所以数据库在系统里是"事实来源"。缓存可以丢,队列可以重试,静态页面可以重新构建,对象存储可以从备份恢复,但数据库里的业务状态一旦乱了,修复成本最高。

一个简单 SaaS 的数据库可能保存:

```text
users
teams
projects
subscriptions
invoices
files
audit_logs
```

这些不是静态文件,也不是缓存。它们需要约束、事务、查询、备份和恢复。

## 三、常见套餐和使用限制

托管数据库的套餐限制通常比静态托管更致命,因为超额后不只是多花钱,还可能直接导致业务不可用。

| 限制项 | 说明 | 常见风险 |
| --- | --- | --- |
| 存储容量 | 数据和索引占用的空间 | 日志表、审计表、历史数据膨胀 |
| CPU / 内存 | 查询执行、排序、连接都消耗资源 | 慢查询拖垮整个实例 |
| 最大连接数 | 同时连接数据库的客户端数量 | Serverless 高并发时瞬间打满 |
| IOPS / 吞吐 | 磁盘读写能力 | 批量导入、报表查询影响线上请求 |
| 读写额度 | Serverless 数据库常按请求或计算量限制 | 免费额度被后台任务耗尽 |
| 备份保留 | 自动备份能保留多久 | 发现误删时备份已经过期 |
| 时间点恢复 | 能否恢复到某个具体时间 | 只能恢复整库,无法精准回滚 |
| 连接池 | 是否提供 Pooler / Proxy | 短连接太多导致连接耗尽 |
| 地域 | 数据所在区域 | 用户延迟、合规、跨境访问问题 |
| 只读副本 | 是否能分担读流量 | 读多写少业务可能需要 |

数据库常见的"免费套餐陷阱"不是存储太小,而是连接数和性能边界太低。

例如一个 Serverless API 写法:

```text
每次请求
  -> 创建一个新数据库连接
  -> 查询一次用户
  -> 查询一次项目
  -> 请求结束后连接没有及时释放
```

平时 20 个用户没问题。流量突然增加时,函数实例变多,每个实例都想连数据库。数据库最大连接数可能只有几十或几百,打满以后新请求就开始 timeout。

另一个常见坑是"备份以为有,恢复没练过"。

```text
自动备份存在
  不等于你知道怎么恢复
支持恢复
  不等于可以只恢复一张表
能恢复到昨晚
  不等于能恢复到误删前 3 分钟
```

数据库套餐页里看到"自动备份"时,一定继续看:

- 保留几天
- 是否按套餐不同缩短或延长
- 是否支持 PITR,也就是时间点恢复
- 恢复是覆盖原库,还是生成新实例
- 恢复期间业务怎么切换
- 备份是否跨地域
- 备份是否额外计费

## 四、小团队建议

小团队选数据库,不要先追求"最先进",先追求"你能理解、能备份、能恢复、能迁移"。

常见保守选择:

```text
关系型业务数据:PostgreSQL 或 MySQL
快速原型 + Auth + API:Supabase / Firebase 等一体化平台
全球边缘读多写少:再考虑专门的分布式数据库
简单个人项目:托管免费层可以起步,但要看连接数和备份
```

如果你不知道该选什么,多数小型 SaaS 从 PostgreSQL 起步是合理的。原因不是 PostgreSQL 永远最好,而是它通用、生态成熟、可迁移性强,能覆盖用户、订单、权限、内容、账单等大部分结构化业务。

最低限度的护栏:

- 数据库不要公网裸奔,尽量限制来源 IP 或走平台内网
- 前端不要直接持有数据库管理员密钥
- 生产库和开发库分开
- 为慢查询和错误率配置监控
- 为连接数设置告警
- 使用连接池,尤其是 Serverless 后端
- 重要表加索引前先理解查询方式
- 自动备份要打开,恢复流程要实际演练一次
- 删除数据前先考虑软删除、归档或导出
- 账号权限按最小权限分配

什么时候需要升级?

- 连接数经常接近上限
- 慢查询影响正常请求
- 存储增长速度超过预期
- 备份保留时间不满足业务要求
- 免费套餐没有足够监控和恢复能力
- 需要只读副本、连接池、私有网络、审计日志
- 数据库已经成为收入业务的核心依赖

小团队最容易犯的错误是"前端平台选得很现代,数据库用得很随意"。实际上,前端平台迁移可能只要几小时,数据库迁移可能要几天甚至几周。越早把数据库当成核心资产,后面越少还债。

---

## 五、一句话总结

托管数据库减少的是基础设施运维,不是数据库责任本身;看套餐时不要只看容量,更要看连接数、性能、备份保留、时间点恢复、监控、地域和权限边界。
