# IP,端口与 HTTP:用户访问网站时发生了什么

## 一句话解释

IP 是机器在网络上的地址,端口是这台机器上某个服务的入口,HTTP 是浏览器和网站服务器之间说话的规则.

如果用现实世界类比:

```text
IP = 大楼地址
端口 = 大楼里的某个窗口
HTTP = 你和窗口工作人员沟通的格式
```

用户输入 `https://example.com/articles`,浏览器最后不是"访问域名"这么简单.它会先通过 DNS 找到 IP,再连接某个端口,再用 HTTP 发送请求.

很多小团队部署失败,不是代码没跑,而是这三层没对上:

```text
DNS 指向了错误 IP
服务器防火墙没开放端口
应用只监听了 localhost
反向代理没有转发到正确端口
HTTP 和 HTTPS 配置混在一起
```

这些问题看起来像"网站打不开",实际原因完全不同.

## 放在系统哪里

从用户点击链接到看到页面,大概经过这些步骤:

```text
1. 浏览器解析 URL
2. DNS 把域名解析成 IP
3. 浏览器连接 IP:端口
4. 如果是 HTTPS,先做 TLS 握手
5. 浏览器发送 HTTP 请求
6. 服务器或代理把请求交给应用
7. 应用返回 HTTP 响应
8. 浏览器渲染页面
```

几个常见端口要先记住:

- 80:HTTP 默认端口.
- 443:HTTPS 默认端口.
- 3000 / 5173 / 8080:开发环境常用端口,不应该直接暴露为正式入口.
- 22:SSH 登录服务器常用端口.

浏览器访问 `https://example.com` 时,默认连的是 443 端口.访问 `http://example.com` 时,默认连的是 80 端口.你不写端口,不代表没有端口,只是浏览器帮你用了默认值.

在生产系统里,常见结构是:

```text
用户
  -> CDN 或负载均衡
  -> 反向代理(Nginx / Caddy / 云网关)
  -> 应用服务(例如 Node.js 监听 3000)
  -> 数据库 / 缓存 / 对象存储
```

对外暴露的通常是 80 和 443.应用内部可以监听 3000,8000,9000 等端口,但不一定直接暴露给公网.

## 常见套餐和使用限制

IP,端口和 HTTP 相关限制,经常藏在云服务器,PaaS,Serverless,CDN,负载均衡的套餐里.

### 1. 公网 IP 不一定免费或固定

云服务器可能自带公网 IP,也可能需要单独购买弹性公网 IP.某些平台的免费实例没有固定公网 IP,重启后地址可能变化.

如果你的 DNS A 记录直接指向一个会变化的 IP,重启后网站就可能打不开.小团队如果使用 PaaS 或前端托管平台,通常会用 CNAME 指向平台域名,避免自己管理 IP.

### 2. 端口开放受安全组和防火墙影响

应用日志里显示"server started on port 3000",不代表公网用户能访问.

你至少要确认:

```text
应用是否监听 0.0.0.0,而不是 127.0.0.1
云服务器安全组是否开放入口端口
操作系统防火墙是否允许访问
反向代理是否把请求转发到应用端口
```

很多入门者在服务器上启动了应用,然后访问域名失败.原因往往不是 Node.js,Go,Python 写错了,而是端口根本没对外通.

### 3. HTTP 请求有大小和时间限制

云平台经常限制:

- 单个请求最大 body 大小.
- 上传文件最大尺寸.
- 请求最长执行时间.
- 并发连接数.
- 每月请求数.
- 出站流量.

比如你做了一个图片上传工具,本地测试没问题,部署到 Serverless 后大文件上传失败.原因可能不是代码 bug,而是平台限制了请求体大小或函数执行时间.

再比如你做了一个 AI 总结接口,一次请求要跑 90 秒,免费 Serverless 平台可能 10 秒或 30 秒就终止.用户看到的是 504 timeout,但真正原因是运行时限制.

### 4. HTTP 状态码是排障入口

不要只说"打不开".先看状态码:

- 200:请求成功.
- 301 / 302:重定向.
- 400:请求格式不对.
- 401 / 403:没有登录或没有权限.
- 404:路径不存在.
- 429:请求太多,被限流.
- 500:应用内部错误.
- 502 / 504:代理,网关,上游服务或超时问题.

状态码不能告诉你全部答案,但能告诉你应该查哪一层.

## 小团队建议

小团队要把"入口层"和"应用层"分清楚.

最低限度建议:

- 正式网站只暴露 80 和 443,应用端口放在内网或由反向代理转发.
- 不要把开发端口当生产入口.
- DNS 指向稳定入口,例如 CDN,负载均衡或平台提供的域名.
- 服务器安全组只开放必要端口.
- 上传,长任务,AI 生成,视频处理不要默认放在普通 HTTP 请求里同步完成.
- 记录常见 4xx,5xx 日志,排障时先看状态码和上游错误.

对于独立开发者,最保守的早期架构通常是:

```text
前端静态站 -> 托管平台
API -> PaaS / 云服务器
数据库 -> 托管数据库
域名 -> DNS 指向托管平台或网关
HTTPS -> 平台自动证书
```

这样你少碰公网 IP 和端口细节,但仍然要理解它们.因为一旦出现 502,504,连接超时,端口不通,你需要知道问题在 DNS,网络,代理,应用还是平台限制.

什么时候该更深入管理 IP 和端口?

当你开始部署自己的云服务器,容器集群,负载均衡,WebSocket 服务,私有 API 或多服务后端时,就不能只依赖平台自动配置.你需要画清楚每个服务监听哪个端口,谁能访问它,是否经过 HTTPS,是否有超时限制.

## 一句话总结

域名让用户找到入口,IP 和端口让浏览器连到具体服务,HTTP 决定请求和响应怎么表达;网站打不开时,先分清是哪一层没接上.
