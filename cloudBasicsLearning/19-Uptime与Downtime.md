# Uptime 与 Downtime:99%、99.9%、99.99% 到底差多少

> 一句话先记住:**Uptime 是服务可用的时间比例,Downtime 是服务不可用的时间;多个 9 看起来只差一点点,换成故障时间差很多**。

很多套餐页会写 99%、99.9%、99.99%,初看都很高。但对线上产品来说,99% 和 99.9% 不是"差 0.9 个百分点"那么简单,它可能是每月 7 小时故障和每月 43 分钟故障的差别。

更重要的是,用户不关心你写了多少个 9。用户只关心打开页面时能不能用、付款时会不会失败、数据会不会丢、故障时有没有说明。

---

## 一、一句话解释

### 1.1 Uptime 是"能用",不是"机器开着"

Uptime 常被翻译成可用时间或在线时间。它真正应该关心的是用户视角的"能用":

```text
服务器进程还在:
  不一定代表可用

首页能打开:
  也不一定代表核心业务可用

用户能完成关键动作:
  更接近真实可用
```

例如你的服务器没有宕机,但数据库连接满了,用户无法登录。这时从机器角度看"还活着",从用户角度看就是不可用。

### 1.2 Downtime 是"不可用",但定义要讲清楚

Downtime 也不是一个绝对词。不同场景有不同定义:

- 全站打不开
- API 返回大量 5xx
- 登录服务不可用
- 支付回调失败
- 图片和文件无法加载
- 后台管理无法访问
- 某个地域用户访问异常

所以讨论可用性之前,先要问一句:我们说的不可用,到底是哪个功能、哪个用户群、哪个时间段不可用?

## 二、放在系统哪里

### 2.1 Uptime 应该从用户链路看

一个典型小网站的访问链路可能是:

```text
用户浏览器
  -> DNS
  -> CDN / WAF
  -> 前端托管
  -> API 服务
  -> 数据库
  -> 对象存储
  -> 第三方服务
```

任意一层出问题,都可能造成 Downtime。常见例子:

- DNS 配错,用户根本找不到网站
- CDN 回源失败,页面间歇性 502
- 前端部署坏了,首页白屏
- API 超时,页面能打开但无法保存
- 数据库维护,登录和下单失败
- 对象存储权限改错,图片全部 403
- 第三方支付挂了,订单无法完成

所以 Uptime 不是某一台机器的指标,而是整条业务链路的结果。

### 2.2 状态页的 Uptime 不等于你的 Uptime

云厂商状态页显示"全部正常",不代表你的应用正常。原因很简单:

```text
厂商状态页:
  统计的是厂商服务整体健康状况

你的应用:
  还包含你的配置、代码、账户额度、权限、部署流程、第三方依赖
```

例如对象存储服务整体正常,但你的 Bucket 权限被改错了,用户仍然看不到图片。这个 Downtime 属于你的系统问题,不是云厂商整体故障。

### 2.3 可用性要按核心路径分层

小团队不需要一开始把所有功能都做成同样可用。更合理的是分层:

```text
第一层:必须可用
  首页、登录、注册、付款、核心 API

第二层:尽量可用
  搜索、评论、通知、文件预览

第三层:可以延迟
  数据导出、报表生成、邮件批量发送、后台统计
```

这样故障发生时,你知道优先恢复什么,也知道哪些功能可以临时降级。

## 三、常见套餐和使用限制

### 3.1 几个 9 换成时间更容易理解

粗略按 30 天一个月计算:

```text
99%      每月大约 7.2 小时不可用
99.5%    每月大约 3.6 小时不可用
99.9%    每月大约 43 分钟不可用
99.95%   每月大约 22 分钟不可用
99.99%   每月大约 4 分钟不可用
99.999%  每月大约 26 秒不可用
```

按一年粗略理解:

```text
99%      每年大约 3.65 天不可用
99.9%    每年大约 8.8 小时不可用
99.99%   每年大约 53 分钟不可用
99.999%  每年大约 5 分钟不可用
```

这张表的目的不是让你精算,而是让你看到:从 99.9% 到 99.99%,不是文案上多一个 9,而是工程要求明显上升。

### 3.2 免费套餐常见边界

免费套餐或低价套餐通常可能有这些限制:

- 没有正式 SLA
- 只提供社区支持或邮件支持
- 故障时不保证响应时间
- 日志保留时间短
- 监控粒度粗
- 不支持多区域部署
- 不支持自动故障切换
- 超过额度后限流、暂停或计费

这些限制会直接影响 Downtime。比如没有足够日志,故障时你不知道哪里坏了;没有多区域,单区域故障时只能等恢复;没有预算告警,超额后服务可能被暂停。

### 3.3 计划维护不一定算 Downtime

很多服务条款会把计划维护排除在 SLA 之外。也就是说:

```text
厂商提前公告:
  某天凌晨维护数据库 30 分钟

你的用户体验:
  这 30 分钟可能确实不能用

SLA 统计:
  可能不算厂商违约
```

对用户来说,打不开就是打不开。对合同来说,可能不算赔偿范围。小团队需要理解这两个口径的差别。

### 3.4 局部故障也会影响真实体验

有些 Downtime 不是全站挂掉,而是局部异常:

- 某个国家或地区访问慢
- 某个 CDN 节点出错
- 某个 API 路由失败
- 某类文件无法下载
- 移动端正常,桌面端白屏
- 新用户不能注册,老用户还能访问

如果你只监控"首页返回 200",这些问题可能很晚才发现。

## 四、小团队建议

### 4.1 先定义"什么叫可用"

不要一上来就问"我们要几个 9"。先写清楚:

```text
对这个产品来说:
  用户能打开首页算不算可用?
  用户能登录但不能付款算不算可用?
  管理后台挂了算不算严重事故?
  报表晚 1 小时生成算不算不可用?
  某个地区访问慢是否算不可用?
```

定义清楚以后,再设置监控和目标。否则 Uptime 只是一个漂亮数字。

### 4.2 对早期产品,99.9% 通常比 99.99% 更现实

99.99% 往往意味着:

- 更可靠的基础设施
- 更好的监控和告警
- 更快的故障响应
- 更成熟的发布和回滚
- 更少的单点依赖
- 更高的人力和云资源成本

如果你的产品还在验证阶段,用户规模不大,没有强合同承诺,盲目追求 99.99% 可能不划算。更实际的目标是:核心路径足够稳定,故障能快速发现,恢复流程不靠临场发挥。

### 4.3 把故障时间分成三段看

小团队排查 Downtime 时,可以把故障拆成:

```text
发现时间:
  从故障发生到你知道它发生

定位时间:
  从知道故障到找到原因

恢复时间:
  从找到原因到用户恢复可用
```

很多团队真正的问题不是恢复很慢,而是发现太晚。用户已经抱怨半小时,团队才知道服务挂了。监控和告警能优先缩短这段时间。

### 4.4 最小可行做法

一个小产品至少应该有:

- 外部可用性探测,从用户视角访问首页和核心接口
- 错误率监控,知道 5xx 是否突然升高
- 延迟监控,知道服务是否变慢
- 数据库连接和容量监控
- 部署失败和构建失败通知
- 简单状态页或公告渠道
- 预算告警,避免欠费或超额导致停机

这些比一开始做复杂高可用更重要。

---

## 五、一句话总结

**Uptime 和 Downtime 要从用户能不能完成核心动作来理解;几个 9 是结果,不是护身符,小团队先定义可用性、监控核心路径、缩短发现和恢复时间。**
