# Serverless 是什么:函数、冷启动、触发器、限制

> 一句话先记住:Serverless 不是没有服务器,而是你不直接管理服务器,平台按请求、事件或运行时间帮你启动和调度计算资源。

Serverless 最吸引人的地方是简单:写一段函数,绑定一个 HTTP 路由、定时任务、队列消息或对象存储事件,平台就能运行它。你不需要先买服务器,不用配置进程守护,不用手动扩容。

但 Serverless 也最容易被神化。它适合事件驱动、短任务、低运维、弹性流量,不适合所有后端。它会把服务器管理问题换成平台限制、冷启动、超时、并发、连接数、可观测性和成本模型问题。

---

## 一、一句话解释

Serverless 是一种运行模型:你提交函数或轻量服务,平台负责调度运行环境,按调用、耗时、资源用量或事件次数计费。

常见形态包括:

- Function as a Service,比如 HTTP 函数、API Routes、边缘函数
- Serverless 容器,比如按请求启动的容器服务
- Serverless 数据库或队列,比如自动扩缩容的托管后端组件
- 静态托管里的构建函数、图片处理函数、边缘中间件

最容易误解的是"Serverless 等于不用管运维"。更准确的说法是:你不用管理机器,但仍然要管理代码、依赖、权限、数据库连接、超时、重试、日志、告警和账单。

可以粗略理解成:

```text
传统服务器:
  我先租一台机器,让程序一直运行,有没有请求都付钱

Serverless:
  我提交一段可被触发的代码,有事件时平台调度它运行,通常按使用量付钱
```

---

## 二、放在系统哪里

Serverless 位于计算层,但它通常和网关、事件源、托管存储紧密绑定。

HTTP 场景的链路可能是:

```text
用户浏览器
  -> DNS
  -> CDN / WAF
  -> API Gateway / 平台路由
  -> Serverless Function
  -> 数据库 / 缓存 / 对象存储 / 第三方 API
```

事件驱动场景的链路可能是:

```text
用户上传图片到对象存储
  -> 对象存储产生事件
  -> 触发 Serverless Function
  -> 函数压缩图片 / 写数据库 / 发消息
```

Serverless 很适合把"有事件才需要运行"的逻辑拆出来:

- 表单提交
- Webhook 回调
- 图片压缩
- 定时清理
- 订单状态同步
- 邮件发送
- 低频后台任务
- 轻量 API

但它不天然适合长连接和持续运行:

- WebSocket 长连接
- 长时间视频处理
- 大型爬虫常驻调度
- 高并发低延迟数据库代理
- 需要本地磁盘长期保存状态的服务

有些平台提供了边缘函数,运行在更靠近用户的节点上。它适合鉴权、重定向、A/B 测试、请求改写、轻量缓存判断等短逻辑。边缘函数通常运行限制更严格,不适合把完整后端都塞进去。

## 三、常见套餐和使用限制

Serverless 的套餐限制通常按多个维度同时计算:

- 调用次数
- 执行时间
- CPU / 内存
- 最大超时时间
- 并发数
- 请求体大小和响应大小
- 冷启动延迟
- 构建分钟
- 日志保留
- 出站流量
- 区域和边缘节点
- 环境变量数量和大小
- 定时任务数量

它的免费额度看起来很慷慨,但要注意每个维度是独立限制。一个项目可能调用次数没超,但执行时间超了;执行时间没超,但出站流量超了;流量没超,但日志保留或构建分钟开始收费。

Serverless 的几个典型坑:

**1. 冷启动**

函数一段时间没有请求后,平台可能回收运行环境。下一次请求需要重新初始化运行时、加载依赖、建立连接,延迟会变高。对后台任务影响不大,对登录、支付、首屏 API 这类路径可能影响体验。

**2. 超时**

Serverless 平台通常有最大执行时间。短函数没问题,但批量导入、视频处理、大模型调用、慢第三方 API,都可能被平台强制终止。

**3. 数据库连接**

传统服务器可以复用少量长连接。Serverless 高并发时可能瞬间启动很多函数实例,每个实例都连接数据库,很容易打满数据库连接数。很多"Serverless + Postgres"的问题,根源不是函数,而是连接池和并发控制。

**4. 重试和重复执行**

事件触发函数失败后,平台可能自动重试。你的代码必须考虑幂等性。比如扣款、发券、发邮件、写订单状态,不能因为重试执行两次就产生错误结果。

**5. 账单不可见**

Serverless 的成本和流量形态高度相关。平时几乎免费,突然被爬虫、Webhook 风暴、错误重试、缓存失效打穿时,调用次数和执行时间会一起上涨。

## 四、小团队建议

什么时候 Serverless 很适合?

- 流量不稳定,有明显波峰波谷
- 低频后台任务,没必要常驻一台服务器
- Webhook、定时任务、图片处理、轻量 API
- 前端项目需要少量后端能力
- 团队不想维护服务器
- 业务还在验证期,希望按使用量起步

什么时候要谨慎?

- 核心接口对延迟非常敏感
- 请求需要运行很久
- 需要大量数据库长连接
- 需要复杂本地状态
- 需要稳定低成本承载持续高流量
- 平台限制会影响业务逻辑
- 团队没有做预算告警和调用监控

Serverless 不一定比服务器便宜。低流量、波动流量、事件驱动场景,它通常很省。持续高流量、每个请求都要运行较久、出站流量大、日志多的场景,固定服务器或容器平台可能更可控。

一个保守架构是:

```text
前端:
  静态托管 / CDN

轻量 API:
  Serverless Function

持久数据:
  托管数据库 / 对象存储

异步任务:
  队列 + Serverless Function

高频核心后端:
  服务器 / 容器 / PaaS,视规模而定
```

最低限度的护栏:

- 设置预算告警和用量告警
- 给函数加超时和错误日志
- 对外部 API 调用设置超时
- 数据库使用连接池或平台推荐的连接方式
- 关键写操作保证幂等
- 对公开接口做限流和鉴权
- 不要把大文件处理和长任务硬塞进短超时函数
- 确认免费额度超出后是停用、限流还是自动计费

Serverless 最适合的是"把不值得常驻服务器的计算交给平台"。它不是后端架构的终点,也不是成本优化的保证。

---

## 五、一句话总结

Serverless 让你少管服务器,按事件和请求运行代码,非常适合短任务、轻量 API 和波动流量;但它有冷启动、超时、并发、连接数和计费边界,小团队应该把它当成一种部署选择,不是万能后端。
