# 静态托管与前端部署:Pages、Vercel、Netlify、OSS 静态站放在什么位置

很多小团队第一次上线网站时,会把"前端部署"想象成买一台服务器,装 Nginx,把 `dist` 目录上传上去。这个方式当然能用,但它不是今天独立开发者最常见的起点。更常见的起点是:代码推到 GitHub,平台自动构建,再把 HTML、CSS、JS、图片这些静态文件分发到 CDN 边缘节点。

> 一句话先记住:静态托管不是"没有服务器",而是你不直接管理服务器;平台帮你把构建产物放到离用户更近的位置,并处理 HTTPS、CDN、回滚和域名绑定。

---

## 一、一句话解释

静态托管,就是托管一批不会在每次请求时现场计算的文件:

```text
index.html
app.css
app.js
logo.png
robots.txt
```

用户访问时,平台直接把这些文件返回给浏览器。典型产品包括:

- Cloudflare Pages
- Vercel
- Netlify
- GitHub Pages
- 阿里云 OSS / 腾讯云 COS / AWS S3 的静态网站托管

它最容易和"后端服务"混淆。静态托管本身不负责数据库查询、登录鉴权、订单创建、后台任务这些动态逻辑。它可以配合 Serverless Function、API Gateway、传统后端一起工作,但静态文件这一层的核心任务是"把前端页面稳定、便宜、快速地发给用户"。

一个很常见的误解是:

```text
React / Vue / Svelte 项目 = 一定需要 Node.js 服务器
```

不一定。很多前端项目最后构建出来就是一堆静态文件。只有当你用了 SSR、API Routes、动态图片优化、服务端鉴权、实时渲染时,才会引入运行时计算。

---

## 二、放在系统哪里

静态托管通常放在用户访问链路的最前面,靠近 CDN:

```text
用户浏览器
  -> DNS
  -> CDN / 边缘节点
  -> 静态托管平台
  -> 静态文件
```

如果还有后端,链路会变成:

```text
用户浏览器
  -> 静态托管平台返回前端页面
  -> 浏览器里的 JS 调用 API
  -> API 网关 / 后端服务
  -> 数据库 / 对象存储 / 队列
```

这意味着前端部署和后端部署是两件事:

- 前端部署负责页面、资源、构建、CDN 分发
- 后端部署负责业务逻辑、鉴权、数据读写、异步任务
- 数据库和对象存储通常不应该被浏览器直接完全暴露访问

以一个独立开发的 SaaS 为例:

```text
官网首页:静态托管
登录页:静态托管 + 调用 Auth API
用户控制台:静态托管 + 调用后端 API
上传头像:前端拿临时上传凭证,文件进对象存储
账单页面:前端调用后端,后端查数据库
```

静态托管在这里不是整个系统,而是"用户打开应用时看到的壳"。它越静态,成本越低,抗流量突增能力越强。

## 三、常见套餐和使用限制

静态托管平台的免费套餐通常不会只限制一个维度。你需要分开看:

| 限制项 | 说明 | 常见风险 |
| --- | --- | --- |
| 带宽 / 出站流量 | 用户下载页面、JS、图片都会消耗 | 图片太大或被爬虫刷爆 |
| 构建分钟 | 每次 push 后自动构建会消耗 | 频繁提交、构建慢,额度很快用完 |
| 构建次数 | 有的平台按次数或并发限制 | 多分支预览会放大消耗 |
| 项目数量 | 免费账户可建项目有限 | 多个边项目混在一个账号里 |
| 团队成员 | 商业协作通常要付费 | 小团队一多人协作就触发升级 |
| Serverless 函数 | Vercel / Netlify 常带函数能力 | 页面以为是静态,实际每次都在跑函数 |
| 图片优化 | 自动压缩、转码、按尺寸生成 | 图片访问量高时可能单独计费 |
| 日志保留 | 免费套餐只保留短时间日志 | 出问题时查不到历史请求 |

Pages、Vercel、Netlify 的共同点是"Git 驱动部署":

```text
push 代码
  -> 平台拉代码
  -> 安装依赖
  -> 执行构建命令
  -> 产出静态文件
  -> 发布到边缘网络
```

这种模式很适合前端项目,但要注意两个边界。

第一,构建不是免费的魔法。依赖安装慢、图片处理多、静态页面生成数量大,都会消耗构建资源。一个博客只有 20 篇文章时很快,有 20000 篇文章时每次构建就可能变成瓶颈。

第二,现代前端框架会让"静态"和"动态"混在一起。比如 Next.js、Nuxt、SvelteKit 这类框架,同一个项目里可能同时有:

```text
静态页面:构建时生成,访问时直接返回
SSR 页面:每次请求到来时服务器渲染
ISR 页面:先缓存,过期后重新生成
API Routes:本质是后端函数
Image Optimization:本质是图片处理服务
```

你以为自己在托管静态站,账单上可能出现的是函数调用、边缘执行、图片处理、构建分钟和带宽。

OSS / S3 / COS 这类对象存储静态站更底层一些。它们通常负责"存文件和返回文件",你需要自己理解:

- 是否支持静态网站索引页,例如 `/` 返回 `index.html`
- 404 是否能回退到前端路由入口
- HTTPS 证书在哪里配置
- CDN 是否单独开通
- 跨域 CORS 是否要配置
- 公共读权限是否会暴露不该公开的文件

对象存储静态站便宜、稳定、通用,但体验不像 Vercel / Netlify 那样一站式。它更像一块可靠的文件底座,不是完整的前端开发平台。

## 四、小团队建议

如果你只是上线官网、文档、博客、产品介绍页,优先选择静态托管:

```text
官网:静态托管
文档:静态托管
博客:静态托管
营销落地页:静态托管
下载页:静态托管 + 对象存储
```

这类页面的特点是读多写少,内容变化不频繁,适合提前构建。流量来了以后,CDN 可以扛住大部分访问,源站压力很小。

如果你在做应用控制台或 SaaS,建议把前端和后端边界切清楚:

```text
前端:Pages / Vercel / Netlify
后端:Serverless / 云服务器 / 容器平台
数据库:托管数据库
文件:对象存储
缓存和异步任务:Redis / MQ
```

不要一开始就把所有业务都塞进前端部署平台的函数里。可以用,但要知道它们适合轻量 API、Webhook、表单处理、预览环境,不一定适合长时间任务、复杂后台作业、大量数据库连接。

最低限度的护栏:

- 给项目绑定正式域名,不要长期依赖平台赠送域名
- 设置预算告警或用量提醒
- 图片尽量压缩,大文件放对象存储或专门的下载服务
- 能静态生成的页面不要用 SSR
- 检查是否开启了不必要的图片优化、函数、日志增强功能
- 预览环境设置访问限制,避免测试链接被公开传播
- 生产环境的 API Key 不要写进前端代码

什么时候该升级?

- 团队成员增加,需要权限、审计、组织管理
- 商业项目需要更高 SLA 或支持响应
- 构建时间明显影响发布效率
- 访问量稳定超过免费额度
- 需要更多日志、回滚、边缘规则、缓存控制
- SSR / API Routes 成为主要流量入口

小团队最保守的做法是:先把能静态化的东西全部静态化,把动态能力留给真正需要动态的接口。这样不仅省钱,也更容易排障。

---

## 五、一句话总结

静态托管是小团队上线前端的默认起点:它适合发页面和静态资源,不适合承载所有后端逻辑;看套餐时不要只看"免费",要同时看带宽、构建、函数、图片优化、日志和团队协作限制。
