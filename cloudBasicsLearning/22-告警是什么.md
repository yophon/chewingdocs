# 告警是什么:阈值、误报、漏报、升级路径

## 一句话解释

**告警就是当系统指标、日志或外部探测结果超过某个风险边界时,主动把人叫醒的一套机制**。

它最容易和"监控"混淆。监控是收集和展示状态,告警是根据状态触发动作。你可以有监控面板但没有告警,也可以有告警但没有完整面板。对小团队来说,真正有价值的不是页面上有多少漂亮曲线,而是线上出问题时,谁会在几分钟内知道,知道后该做什么。

一个很常见的误解是:告警越多越安全。实际正好相反。告警太多会让团队麻木,最后真正严重的问题也被当成噪音忽略。告警太少则会漏掉用户已经大面积受影响的事故。好的告警不是"什么都报",而是只在需要人工判断或人工处理时打扰人。

## 放在系统哪里

告警通常放在三类位置:

```text
用户侧探测
  -> 网站打不开 / API 失败率升高 / 页面加载很慢

服务内部指标
  -> CPU / 内存 / 磁盘 / 数据库连接数 / 队列积压 / 5xx 比例

业务关键动作
  -> 支付失败 / 登录失败 / 订单创建失败 / 邮件发送失败
```

用户侧探测回答的是:"用户现在还能不能用?" 例如每 1 分钟从外部访问一次首页和健康检查接口。如果探测失败,说明不管你的服务器指标看起来多正常,用户体验已经出问题。

服务内部指标回答的是:"哪一层可能快到极限了?" 例如数据库连接数长期接近上限,磁盘使用率超过 85%,Serverless 函数错误率突然升高。这类告警不一定意味着用户已经不可用,但它能让你在事故扩大前介入。

业务关键动作回答的是:"技术系统还活着,但业务是否已经坏了?" 例如网站 200 正常,但所有注册验证码都发不出去;API 延迟正常,但支付回调连续失败。小团队最容易忽略这一层,因为它不在云厂商默认监控里,需要自己埋点或写日志。

## 常见套餐和使用限制

免费套餐或入门套餐里的告警,通常限制在这些地方:

- 告警规则数量有限,只能配置少量核心指标.
- 通知渠道有限,可能只有邮件,没有电话、短信、PagerDuty、飞书或 Slack 集成.
- 检查频率较低,例如 5 分钟探测一次,不是 10 秒或 30 秒.
- 数据保留时间较短,事后只能看到最近几小时或几天的曲线.
- 多人升级路径有限,很难配置"先通知值班人,10 分钟未确认再通知负责人".

这些限制不一定是坏事。对独立开发和小团队来说,一开始不需要企业级值班系统。但你必须知道限制在哪里:如果你的免费监控只保留 24 小时数据,那么周末发生的问题,周一再看可能已经没有足够上下文;如果告警只发邮件,而你晚上不开邮箱,它就不是能叫醒人的告警。

还有一种隐形限制是"告警噪音成本"。免费工具经常很容易配置阈值,但默认规则不一定适合你的系统。比如 CPU 超过 80% 就告警,对一个偶尔跑定时任务的小服务器可能会天天误报;反过来,数据库连接数 95% 才告警,对连接数很低的免费数据库可能已经太晚。

## 小团队建议

小团队先不要追求大而全的告警体系,先建立 5 类最低限度告警:

```text
1. 首页或核心 API 外部探测失败
2. 5xx 错误率明显升高
3. 数据库连接数 / 存储空间接近套餐上限
4. 队列或异步任务积压持续增长
5. 账单或资源用量接近免费额度 / 预算上限
```

阈值不要只看绝对值,要结合持续时间。一次请求失败不值得叫醒人,连续 5 分钟失败才值得。CPU 瞬间到 90% 不一定危险,但内存长期 90% 且不断上升就很危险。告警条件最好写成"超过阈值并持续 N 分钟",这样能减少误报。

通知渠道要按严重程度分层:

- 普通提醒:邮件或聊天工具即可.
- 用户可见故障:聊天工具加手机推送.
- 收入或数据安全相关故障:必须能在非工作时间触达负责人.

每条告警都应该能回答三个问题:它代表什么风险,先看哪个面板或日志,第一步处理动作是什么。如果一条告警只能告诉你"某个值异常",但没人知道下一步怎么做,它更像噪音。

最后要定期删除无用告警。过去 30 天里频繁误报、但没有导致任何行动的告警,要么调阈值,要么降级为普通通知,要么删除。告警系统的目标不是证明你很重视稳定性,而是在真正出事时保持注意力。

## 一句话总结

**告警不是把所有异常都推给人,而是把最可能影响用户、数据和账单的问题,在还能处理的时候送到对的人面前**。
