# 小团队云服务选型:从 0 到 1 的保守架构

## 一句话解释

小团队云服务选型,不是选"最强平台",而是在成本、复杂度、可靠性、迁移能力之间选一个当前阶段不会拖垮团队的组合。

最容易犯的错有两个:一是过度设计,一开始就上 Kubernetes、多地域、微服务和复杂云产品;二是过度依赖免费套餐,把所有能力堆在一个平台里,直到流量、账单、数据库或权限规则一起出问题。

从 0 到 1 的保守架构要做到三件事:能上线,能看懂,能停损。

---

## 放在系统哪里

一个小团队最小可用云架构通常可以这样看:

```text
用户
  -> 域名 / DNS
  -> CDN / HTTPS / WAF
  -> 前端托管
  -> API / Serverless / 后端服务
  -> 数据库 / 对象存储 / 队列
  -> 监控 / 日志 / 告警 / 备份
  -> 账单 / 预算告警
```

选型不是只选一个产品,而是给这些位置分别找合适的服务:

- 入口层:域名、DNS、HTTPS、CDN、基础防护
- 计算层:静态托管、Serverless、容器、VPS、托管平台
- 数据层:关系数据库、NoSQL、对象存储、缓存、队列
- 运维层:日志、监控、告警、备份、恢复
- 成本层:账单、预算、用量提醒、资源标签
- 权限层:账号、MFA、团队成员、API Key、Secret 管理

小团队的问题通常不是"没有云服务可选",而是每一层都选了一个看起来很方便的产品,最后没有人知道请求从哪里进来、数据存在哪里、账单从哪里增长。

## 常见套餐/使用限制

### 查看方法

做选型时,不要从厂商首页开始,要从自己的系统需求开始。

建议按这个顺序看:

1. 列出应用形态:静态站、内容站、SaaS、移动 App、API 服务、内部工具
2. 列出核心路径:访问首页、注册登录、提交数据、上传文件、支付、后台任务
3. 为每条路径标出会经过的云服务
4. 对每个云服务查 Pricing、Limits、Quota、Overage、SLA、Backup、Export
5. 判断限制是软限制还是硬限制
6. 估算正常流量、活动流量、异常流量下最先撞到的限制
7. 确认数据是否能导出,服务是否能迁移,域名是否能快速切换
8. 设置预算告警、用量告警、错误告警和备份策略

这个过程比直接比较"哪家免费额度更多"更可靠。因为真正决定能不能长期用的,不是某一项额度,而是你的系统最容易放大的那条路径。

### 常见限制维度

小团队选型时必须检查这些维度:

- 域名和 DNS:是否能快速切换,是否支持需要的记录类型,是否有清晰的托管边界
- CDN 和边缘:缓存规则、回源流量、WAF、速率限制、日志能力
- 前端托管:构建分钟、构建并发、预览环境、带宽、回滚能力
- Serverless:调用次数、运行时间、冷启动、并发、内存、区域、日志
- VPS / 容器:CPU、内存、磁盘、网络、系统维护、安全补丁
- 数据库:连接数、存储、读写性能、备份、恢复、迁移、区域
- 对象存储:容量、出站流量、读写操作、权限、生命周期、CDN 集成
- 认证:用户数量、登录方式、邮件发送、风控、账号恢复
- 日志监控:日志保留、指标数量、告警渠道、错误追踪
- 团队协作:成员数、权限、审计、生产环境保护
- 成本控制:预算告警、超额行为、支付方式、资源标签

对小团队来说,限制不是越高越好,而是越清楚越好。一个额度较小但行为明确的平台,通常比一个看起来免费但超额规则模糊的平台更适合生产使用。

## 小团队建议

### 1. 从保守默认架构开始

多数从 0 到 1 的 Web 产品,可以先用这类组合:

```text
Cloudflare: DNS / CDN / HTTPS / 基础防护
Vercel 或 Netlify 或 Cloudflare Pages: 前端托管
Supabase 或 Firebase: Auth / Database / Storage
Sentry 或平台内置日志: 错误追踪
云平台账单告警: 成本护栏
```

如果后端逻辑较重,可以把 API 单独放在一个更可控的位置:

```text
Cloudflare: 入口和缓存
前端托管平台: 静态页面和预览部署
VPS / PaaS / 容器平台: 后端 API
托管数据库: 生产数据
对象存储: 图片、附件、导出文件
```

这两种都不是唯一答案,但它们有一个共同点:层次清楚。入口、前端、后端、数据库、存储、监控、账单分别能说清楚。

### 2. 不要过早引入复杂平台

小团队早期通常不需要:

- Kubernetes 集群
- 多云主动容灾
- 自建高可用数据库
- 复杂微服务拆分
- 自研认证系统
- 自建 CDN
- 大量云厂商专有组件

这些东西不是没价值,而是会把团队的注意力从产品验证转移到平台维护。除非你已经有明确的规模、合规、性能或组织需求,否则先选能托管、能备份、能导出、能告警的服务。

### 3. 给每个服务写退出条件

小团队选型时,每个核心服务都要回答:

- 什么时候免费套餐够用
- 什么时候必须升级
- 什么时候必须迁移
- 数据怎么导出
- 域名怎么切换
- Secret 怎么轮换
- 备份在哪里恢复
- 如果平台宕机,用户看到什么

例如:

- 前端托管撞到构建或带宽限制:先静态化、减少图片优化、调整缓存,再升级
- 数据库撞到连接数:先连接池和慢查询优化,再升级规格
- 对象存储出站流量上涨:先接 CDN 和生命周期策略,再换存储方案
- Serverless 函数超时:先拆后台任务,再考虑独立后端或队列

### 4. 保持可迁移边界

小团队不可能完全避免厂商绑定,但可以减少锁死:

- 域名注册和 DNS 托管不要混乱,确保能独立切换
- 数据库优先考虑可导出、可备份、生态成熟的方案
- 文件放对象存储,路径和权限规则保持清楚
- 业务核心逻辑不要散落在太多平台规则里
- 环境变量和 Secret 统一记录来源和用途
- 不把生产数据只留在一个没有备份的免费项目里

---

## 一句话总结

小团队云服务选型的目标不是一步到位,而是用层次清楚、限制可见、成本可控、数据可迁移的保守架构,先把产品稳定地从 0 推到 1。
