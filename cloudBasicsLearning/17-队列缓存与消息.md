# 队列、缓存与消息:Redis、MQ、异步任务的入门常识

小项目一开始通常是"请求来了就现场处理"。用户点按钮,后端查数据库,生成结果,返回页面。这个模型简单直接,但当任务变慢、并发变高、外部服务不稳定时,你就会需要缓存、队列和消息系统。

> 一句话先记住:缓存是为了少做重复读取,队列是为了把慢任务放到后台排队做,消息是为了让系统之间可靠传递事件;它们都不是数据库替代品。

---

## 一、一句话解释

这三个词经常一起出现,但解决的问题不同。

缓存:

```text
把常用结果临时放起来,下次更快读取
```

典型产品是 Redis、Memcached、Cloudflare KV、Upstash Redis。常见用途:

- 登录 Session
- 热门配置
- API 查询结果
- 排行榜
- 限流计数器
- 短期验证码

队列:

```text
把任务先放进队伍,后台 Worker 一个个处理
```

典型产品是 SQS、RabbitMQ、Kafka、阿里云 MNS / RocketMQ、腾讯云 CMQ / TDMQ、Cloudflare Queues。常见用途:

- 发邮件
- 图片压缩
- 视频转码
- 生成报表
- Webhook 重试
- 批量导入

消息:

```text
一个系统发生了事件,通知其他系统
```

常见事件:

```text
user.created
order.paid
file.uploaded
invoice.failed
comment.created
```

缓存、队列、消息最容易和数据库混淆。数据库保存事实,缓存保存临时副本,队列保存待处理任务,消息保存事件通知。它们可以一起用,但职责不同。

---

## 二、放在系统哪里

缓存通常放在后端和数据库之间:

```text
用户请求
  -> 后端 API
  -> Redis 缓存
  -> 数据库
```

一次典型读取:

```text
后端查 Redis
  -> 命中:返回缓存结果
  -> 未命中:查数据库
  -> 写入 Redis,设置过期时间
  -> 返回结果
```

队列通常放在请求处理和后台 Worker 之间:

```text
用户提交导出任务
  -> 后端创建任务记录
  -> 消息进入队列
  -> Worker 异步生成文件
  -> 文件上传对象存储
  -> 数据库更新任务状态
  -> 用户稍后下载
```

消息系统通常放在多个服务之间:

```text
支付服务确认订单已支付
  -> 发布 order.paid 消息
  -> 邮件服务发送收据
  -> 权限服务开通会员
  -> 数据分析服务记录转化
```

对小团队来说,最常见的不是上来就设计复杂事件架构,而是先解决两个痛点:

```text
热点数据反复查数据库 -> 加缓存
慢任务拖住 HTTP 请求 -> 加队列
```

例如用户点击"导出全部订单"。如果你在 HTTP 请求里现场生成 CSV,请求可能跑 30 秒,中途超时,用户刷新又重新跑一次。更稳的方式是:

```text
1. 后端创建导出任务,状态为 pending
2. 把任务 ID 放进队列
3. 立即返回"任务已创建"
4. Worker 后台生成 CSV
5. 文件放对象存储
6. 数据库把任务状态改成 done
```

## 三、常见套餐和使用限制

缓存服务常见限制:

| 限制项 | 说明 | 常见风险 |
| --- | --- | --- |
| 内存容量 | Redis 主要吃内存 | key 太多或 value 太大导致淘汰 |
| 最大连接数 | 客户端连接数量 | Serverless 短连接打满连接数 |
| QPS / 吞吐 | 每秒读写能力 | 热点 key 或限流计数器压力大 |
| 数据持久化 | 是否落盘、是否可恢复 | 把缓存当数据库会丢数据 |
| 淘汰策略 | 内存满了删哪些 key | 重要临时数据被提前淘汰 |
| 网络位置 | 和后端是否同区域 | 跨区域访问增加延迟和费用 |

Redis 特别容易被滥用。它很快,但快不等于适合保存所有东西。你要默认缓存可能丢:

```text
Redis 重启
内存满了触发淘汰
过期时间到了
平台故障切换
开发者误删 key
```

所以可以丢的放缓存,不能丢的进数据库。

队列和 MQ 常见限制:

| 限制项 | 说明 | 常见风险 |
| --- | --- | --- |
| 消息数量 | 免费额度或套餐限制 | 突发任务堆积后超额 |
| 消息大小 | 单条消息不能太大 | 把文件内容塞进消息会失败 |
| 保留时间 | 消息最多保存多久 | Worker 挂太久,消息过期丢失 |
| 投递次数 | 失败后重试多少次 | 毒消息反复失败拖垮队列 |
| 并发消费者 | Worker 同时处理数量 | 太高会打爆数据库或第三方 API |
| 顺序保证 | 是否保证先进先出 | 订单状态类任务需要特别注意 |
| 至少一次投递 | 消息可能重复到达 | 处理逻辑必须幂等 |

队列最重要的一句话是:

```text
消息可能重复,可能延迟,可能失败,但不应该让系统整体卡死。
```

很多 MQ 默认追求"至少一次投递",意思是平台尽量保证消息被处理,代价是同一条消息可能被投递多次。所以 Worker 不能假设"这件事只会执行一次"。比如扣款、发券、创建订单这类操作必须有幂等设计:

```text
同一个 message_id 已处理过
  -> 直接跳过
否则
  -> 执行业务
  -> 记录已处理
```

## 四、小团队建议

不要为了"架构完整"上 Redis 和 MQ。先问两个问题:

```text
有没有重复读导致数据库压力?
有没有慢任务导致用户请求超时?
```

如果答案都是否,先不用。一个简单应用可以靠数据库和对象存储跑很久。

适合先加缓存的场景:

- 首页热门内容读取频繁
- 用户权限或配置每次请求都要查
- 第三方 API 有速率限制
- 需要做登录 Session
- 需要做 IP 或用户级限流

适合先加队列的场景:

- 发邮件、短信、Webhook
- 图片压缩、视频转码
- 导入导出大文件
- 调用不稳定的第三方服务
- 需要失败重试
- 任务执行时间超过 HTTP 请求可接受时间

最低限度的护栏:

- 缓存 key 要有命名规范和过期时间
- 不要把唯一事实只存在 Redis
- 缓存失效后要能从数据库重建
- 队列消息里放 ID,不要放大文件本体
- Worker 处理要幂等,能安全重试
- 设置死信队列或失败记录
- 控制 Worker 并发,避免后台任务打爆数据库
- 监控队列积压长度、失败次数、处理耗时
- Redis 和 MQ 密码不要暴露到前端
- 生产和开发环境分开

一个保守的小团队演进路径:

```text
阶段 1:
  后端 + 数据库 + 对象存储

阶段 2:
  加 Redis 做 Session、限流、热点缓存

阶段 3:
  加队列处理邮件、导出、图片压缩

阶段 4:
  多服务之后再考虑事件总线、Kafka、复杂 MQ
```

什么时候该升级?

- Redis 内存经常接近上限
- 连接数被 Serverless 后端打满
- 队列积压持续增长
- 失败任务没有可追踪记录
- 后台任务影响数据库正常请求
- 需要更长消息保留时间或更强顺序保证
- 消息已经承载核心交易链路

小团队常见的错误是"缓存没命中就全站变慢,队列失败就没人知道"。缓存和队列都是为了让系统更稳,但前提是你给它们设置了过期、重试、告警和降级。

---

## 五、一句话总结

Redis 解决重复读取和短期状态,MQ 解决异步任务和事件传递;小团队不要把它们当数据库,看套餐时重点看内存、连接数、消息大小、保留时间、重试语义、并发和失败处理能力。
