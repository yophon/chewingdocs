# 备份、快照与恢复:备份不是有文件就行

## 一句话解释

**备份是把重要数据复制到另一个可保存的位置,快照是某个时间点的状态记录,恢复是把这些数据真正还原成可用系统**。

这三个词经常被混在一起。很多人以为"我有备份"就等于"我能恢复",这是非常危险的误解。备份文件存在,不代表文件完整;文件完整,不代表版本正确;版本正确,不代表能在事故时按预期恢复;能恢复,也不代表恢复时间能被业务接受。

对小团队来说,备份的价值不是让控制台里出现一个"已开启备份"的开关,而是在误删数据、数据库损坏、账号被盗、程序 bug 批量写错数据时,你能把系统拉回一个可接受的状态。

## 放在系统哪里

备份和快照通常覆盖这些层:

```text
数据库
  -> 用户、订单、配置、业务核心数据

对象存储
  -> 用户上传文件、图片、附件、导出文件

云服务器磁盘
  -> 系统环境、上传目录、本地状态、临时脚本

代码和配置
  -> Git 仓库、环境变量、IaC 配置、部署参数
```

不同数据的恢复方式不一样。数据库备份通常需要恢复到某个时间点,因为你可能只想回到误删前 5 分钟。对象存储更像文件版本管理,你关心某个文件是否被覆盖或删除。云服务器快照可以快速复制整块磁盘,但它不一定适合数据库的一致性恢复。代码和配置如果只存在某个人电脑里,那也属于没有备份。

一个常见事故是:

```text
后台脚本写错 where 条件
  -> 把 3000 个用户的会员状态改成过期
  -> 数据库还有昨晚自动备份
  -> 但今天白天又有 200 个新订单
```

这时你不能简单把整个数据库恢复到昨晚,否则会丢掉今天的新订单。你需要知道是否支持按时间点恢复,是否能恢复到临时库里做数据对比,是否能只回填被改错的字段。这就是"有备份"和"可恢复"的区别。

## 常见套餐和使用限制

免费套餐和低价套餐的备份限制通常很明显:

- 自动备份可能不开启,或只保留 1-7 天.
- 按时间点恢复可能是付费功能.
- 快照数量有限,超出后需要手动删除或额外计费.
- 跨地域备份可能不包含在免费计划里.
- 对象存储版本控制、生命周期、删除保护可能需要单独配置.
- 恢复到新实例可能收费,恢复过程也可能占用额外存储和计算资源.

最需要警惕的是"备份存在,但恢复路径很慢或不可用"。例如某个数据库每天自动备份一次,但恢复只能整库恢复,不能恢复单表;某个对象存储开启了版本控制,但删除标记和生命周期规则配置错误,旧版本很快被清理;某个云服务器有磁盘快照,但应用真正的数据在外部数据库里,恢复服务器并不能恢复业务。

备份还有两个关键指标:

```text
RPO:最多能接受丢多少数据
RTO:最多能接受停多久
```

如果每天凌晨备份一次,那么最坏情况下会丢 24 小时数据,RPO 就接近 24 小时。如果恢复数据库、改配置、重新部署总共需要 4 小时,RTO 就是 4 小时。很多免费套餐默认给你的 RPO / RTO 并没有写在首页大字里,需要你自己推算。

## 小团队建议

小团队先把数据分级:

```text
必须恢复
  -> 用户账号、订单、支付记录、权限、核心业务数据

最好恢复
  -> 用户上传图片、附件、评论、内容草稿

可以重建
  -> 缓存、搜索索引、构建产物、临时文件
```

核心数据库至少要有自动备份,并确认保留时间。只要开始有真实用户和收入,就不要依赖"免费数据库默认会帮我处理"这种假设。你需要明确知道:备份保留几天,能不能按时间点恢复,恢复到原实例还是新实例,恢复要多久,是否需要人工工单。

对象存储里的用户上传文件,建议开启版本控制或至少做周期性复制。很多团队只备份数据库,忘了用户文件。结果数据库恢复了,订单记录还在,但发票、头像、合同附件全没了。

最重要的是做恢复演练。一个简单做法是每月或每次大版本前做一次:

```text
1. 从备份恢复到一个临时数据库
2. 启动应用连接临时数据库
3. 抽查登录、下单、后台查询等核心流程
4. 记录恢复耗时和需要手动操作的步骤
```

没有演练过的备份,只能算"可能有用的文件"。备份策略也要和删除保护配合:生产数据库不要让普通账号能一键删除;对象存储桶不要让应用运行时密钥拥有管理生命周期和删除桶的权限;关键备份最好和生产账号做权限隔离。

## 一句话总结

**备份的目标不是保存一份数据,而是在真实事故里按可接受的数据损失和停机时间恢复业务;没有验证过恢复的备份,不能算可靠备份**。
