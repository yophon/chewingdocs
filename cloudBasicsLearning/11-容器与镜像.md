# 容器与镜像:为什么部署不只是上传代码

> 一句话先记住:镜像是应用运行环境的打包结果,容器是镜像启动后的运行实例。

很多人第一次听到 Docker、容器、镜像,会以为它们只是"新的部署工具"。但容器真正解决的问题不是上传代码,而是"让代码带着它需要的运行环境一起走"。

同一份代码,在你的电脑能跑,在服务器不能跑,常见原因不是代码变了,而是 Node.js 版本、Python 依赖、系统库、环境变量、文件路径、时区、权限、端口都不一样。容器就是为了解决这种环境不一致。

---

## 一、一句话解释

镜像可以理解成一个只读模板,里面包含应用代码、运行时、依赖包、系统库、启动命令等内容。

容器可以理解成从这个模板启动出来的进程环境。一个镜像可以启动多个容器,就像一个安装包可以安装出多个运行中的程序。

它最容易和虚拟机混淆。虚拟机模拟的是一整台机器,里面有完整操作系统内核。容器共享宿主机内核,隔离的是进程、文件系统、网络和资源视图。容器通常更轻,启动更快,但隔离边界也不同于完整虚拟机。

可以粗略理解成:

```text
代码:我的应用逻辑
镜像:代码 + 运行环境 + 启动方式
容器:镜像真正跑起来之后的实例
仓库:存放镜像的地方
```

容器不是银弹。它解决的是环境一致和交付标准化,不是自动解决架构、性能、安全、备份、监控和成本。

---

## 二、放在系统哪里

容器通常位于计算层内部。

如果用户访问一个容器化部署的 Web 应用,请求链路可能是:

```text
用户浏览器
  -> DNS
  -> CDN / WAF
  -> 负载均衡 / 反向代理
  -> 宿主机上的容器
  -> 数据库 / 缓存 / 对象存储
```

容器不会替代 DNS、CDN、HTTPS、数据库。它只负责把你的应用进程以标准方式跑起来。

在一台云服务器上,你可以直接运行应用:

```text
服务器
  -> Node.js / Python / Java
  -> 应用进程
```

也可以用容器运行:

```text
服务器
  -> Docker / containerd
  -> 容器
  -> 应用进程
```

区别是后者把应用依赖写进镜像构建过程。新服务器只要能运行容器,就可以用同一个镜像启动同一个应用。团队协作时,这会明显减少"我本地是好的"这类问题。

容器也常出现在 PaaS、Serverless 容器、Kubernetes、CI/CD 里。很多平台虽然界面上只让你连接 Git 仓库,背后也是先构建镜像,再把镜像部署到某种运行环境。

## 三、常见套餐和使用限制

容器本身不一定收费,但运行容器的地方会收费。你可能在以下地方遇到限制:

- 云服务器上的 Docker
- 托管容器平台
- Kubernetes 集群
- Serverless 容器服务
- CI/CD 构建环境
- 镜像仓库

常见限制包括:

- CPU 和内存上限
- 容器最大运行时间
- 镜像大小
- 镜像拉取次数
- 构建分钟数
- 并发容器数
- 出站流量
- 日志保留时间
- 私有镜像仓库容量

小团队最容易忽略的是镜像大小和构建时间。一个随便写的 Dockerfile,可能把开发依赖、缓存、测试文件、node_modules、编译中间产物全塞进镜像。结果是:

```text
每次部署:
  -> 构建慢
  -> 上传慢
  -> 拉取慢
  -> 启动慢
  -> CI 构建分钟增加
  -> 镜像仓库存储增加
```

另一个常见坑是把容器当成数据存储。容器文件系统通常应该被视为临时的。容器重建、迁移、扩容、平台回收时,写在容器内部的文件可能消失。用户上传文件、数据库数据、业务持久化内容,应该放到对象存储、数据库、云盘或明确挂载的持久卷里。

还有一个坑是环境变量和密钥。容器镜像不应该内置生产数据库密码、API Key、私钥。镜像应该是通用交付物,不同环境通过环境变量、密钥管理或平台配置注入差异。

## 四、小团队建议

什么时候应该用容器?

- 项目依赖复杂,本地和服务器经常环境不一致
- 团队成员多,需要统一开发和部署环境
- 一个应用包含多个服务,比如 Web、Worker、定时任务
- 需要可重复部署和快速回滚
- 未来可能迁移到托管容器平台或 Kubernetes

什么时候可以先不用?

- 只是一个静态站
- 只是一个非常简单的脚本或单进程服务
- 团队还没有基本 Linux 和网络排障能力
- 你只是为了"看起来专业"而容器化

容器会增加一层抽象。如果连应用本身监听哪个端口、日志输出在哪里、数据库怎么连都没搞清楚,直接上容器只会让排障路径变长。

对小团队来说,比较务实的路线是:

```text
第一阶段:
  直接部署到 VPS / 云服务器,把端口、进程、日志、反向代理搞清楚

第二阶段:
  用 Dockerfile 固化运行环境,用 docker compose 管理本机多服务

第三阶段:
  用 CI 构建镜像,推送到镜像仓库,服务器拉取镜像部署

第四阶段:
  当服务数量、实例数量、发布频率变复杂后,再考虑托管容器平台或 K8s
```

最低限度的实践:

- Dockerfile 不要把密钥写进镜像
- 用 `.dockerignore` 排除无关文件
- 尽量让容器无状态
- 日志输出到 stdout / stderr,交给平台收集
- 配置健康检查
- 明确资源限制,不要让一个容器吃完整台机器
- 数据库和用户文件不要只放在容器内部

容器最适合做"交付标准化",不是用来掩盖应用设计问题。一个会泄露连接、没有超时、没有日志、没有备份的应用,放进容器后仍然是同一个有问题的应用。

---

## 五、一句话总结

镜像让应用和运行环境一起交付,容器让这个交付物以隔离进程的方式运行;小团队应该在环境不一致和部署重复性开始变成真实成本时使用容器,而不是为了追赶技术名词而提前复杂化。
