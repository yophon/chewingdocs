# 监控是什么:指标、日志、链路追踪的分工

> 一句话先记住:**监控不是摆几张漂亮图表,而是在系统出问题前后,让你知道哪里异常、影响多大、该先查什么**。

很多人把监控理解成 CPU、内存、磁盘三张图。它们当然有用,但对一个真实网站或 SaaS 来说远远不够。用户登录失败、支付回调丢失、CDN 回源异常、数据库连接打满、第三方接口变慢,这些都可能发生在机器指标看起来还正常的时候。

监控的目标不是"看起来专业",而是缩短三个时间:

```text
发现问题的时间
定位问题的时间
恢复服务的时间
```

---

## 一、一句话解释

### 1.1 监控到底监什么

监控要回答四类问题:

```text
可用性:
  用户能不能访问?

正确性:
  请求有没有成功?

性能:
  响应是不是变慢?

容量:
  资源是不是快用完?
```

如果只监控服务器是否在线,你只能知道"机器大概还活着"。但用户关心的是"我能不能完成操作"。

### 1.2 指标、日志、链路追踪分别做什么

这三个词经常一起出现,但分工不同:

```text
指标(Metrics):
  用数字看趋势
  例如错误率、延迟、CPU、内存、请求数、队列积压

日志(Logs):
  用事件看细节
  例如某次请求为什么失败、具体错误堆栈是什么

链路追踪(Tracing):
  用调用链看一次请求经过了哪些服务
  例如前端 -> API -> 数据库 -> 支付服务,每一步花了多久
```

一句简单区别:

```text
指标告诉你"哪里不对劲"
日志告诉你"发生了什么"
链路追踪告诉你"一次请求卡在哪一段"
```

### 1.3 监控和告警不是一回事

监控是收集和展示系统状态。告警是当状态异常时通知人。

```text
监控:
  5xx 错误率从 0.1% 升到 8%

告警:
  把这件事通过短信、电话、邮件、群消息告诉负责人
```

没有监控,告警没有依据。没有告警,监控可能只是事后翻图。

## 二、放在系统哪里

### 2.1 监控应该覆盖用户请求链路

一个小产品可以按链路看监控点:

```text
用户侧:
  页面是否能打开
  首屏加载是否过慢

入口层:
  DNS 是否正常
  CDN 命中率
  WAF 拦截量
  4xx / 5xx 比例

应用层:
  API 请求数
  错误率
  P95 / P99 延迟
  函数执行时间

数据层:
  数据库连接数
  慢查询
  存储容量
  缓存命中率

异步层:
  队列积压
  任务失败次数
  重试次数

外部依赖:
  支付、短信、邮件、地图、AI API 的成功率和延迟
```

监控不一定一开始全部做满,但你要知道每一层出问题时会影响什么。

### 2.2 外部探测很重要

很多小团队只看云控制台里的服务状态,忽略了外部探测。外部探测是从真实用户附近定时访问你的网站或接口:

```text
每 1 分钟:
  访问首页
  调用健康检查接口
  检查返回状态码和响应时间
```

它能发现一些内部监控看不到的问题:

- DNS 配置错误
- CDN 节点异常
- TLS 证书过期
- WAF 误拦截
- 首页部署白屏
- 某个地区访问失败

如果你只监控服务器内部,这些问题可能会漏掉。

### 2.3 业务监控比机器监控更接近用户

机器监控告诉你资源状态,业务监控告诉你产品是否正常运转:

```text
机器监控:
  CPU 80%
  内存 70%
  磁盘剩余 20GB

业务监控:
  最近 10 分钟注册成功率下降
  支付回调失败率上升
  文件上传失败次数增加
  队列积压超过 10000 条
```

对小团队来说,业务监控经常比机器监控更早暴露真实问题。

## 三、常见套餐和使用限制

### 3.1 免费监控常限制保留时间和粒度

免费套餐或低价套餐常见限制:

- 指标只保留 1 天、7 天或 30 天
- 日志保留时间短
- 日志量超过后丢弃或收费
- 监控粒度较粗,例如 1 分钟或 5 分钟一个点
- 自定义指标数量有限
- 告警规则数量有限
- 通知渠道有限
- 链路追踪采样率有限
- 多团队协作和权限功能受限

这些限制平时不明显,出事故时很要命。因为你最需要查历史的时候,可能发现日志已经过期;你最需要看细节的时候,发现免费套餐只保留聚合数据。

### 3.2 日志成本可能比想象中高

日志不是越多越好。大量 debug 日志、请求体日志、重复错误日志都会带来成本和风险:

- 存储费用增加
- 查询费用增加
- 日志平台超额
- 敏感信息泄露风险增加
- 排查时被噪音淹没

小团队常见错误是上线时忘了关 debug 日志,一次流量增长后日志量暴涨,账单也跟着涨。

### 3.3 链路追踪通常不是免费无限用

链路追踪很有价值,但也常受限制:

- 采样率限制
- trace 数量限制
- 保留时间限制
- 高级分析付费
- 跨服务接入需要额外配置

如果系统还很简单,可以先用日志里的 request_id 串起一次请求。等服务变多、调用链变长,再引入完整 tracing。

### 3.4 状态页不等于监控系统

状态页是给用户或客户看的,监控系统是给工程团队排障用的。

```text
状态页:
  "我们正在调查 API 异常"

监控系统:
  "从 10:02 开始,登录接口 5xx 升到 12%,数据库连接数达到上限,主要影响亚洲用户"
```

状态页可以提升沟通效率,但它不能替代监控。

## 四、小团队建议

### 4.1 先做最小监控面

一个小团队上线初期,至少要有:

- 首页外部可用性探测
- 核心 API 健康检查
- 5xx 错误率
- P95 延迟
- 数据库连接数和存储容量
- 队列积压和失败任务数
- 部署失败通知
- 证书过期提醒
- 账单和用量告警

这套监控不复杂,但能覆盖大多数早期事故。

### 4.2 每个告警都要能行动

不要为了显得专业设置几十个告警。告警太多会让人麻木。

好的告警应该满足:

```text
有人负责:
  谁收到?谁处理?

影响明确:
  影响用户还是只影响后台任务?

阈值合理:
  不因为短暂抖动频繁报警

有下一步:
  收到后先看哪张图?哪份日志?哪个服务?
```

如果一个告警响了以后没人知道该做什么,它就是噪音。

### 4.3 保留关键日志,少存无用日志

建议至少保留这些日志:

- 请求入口日志
- 错误日志和异常堆栈
- 登录、支付、权限变更等关键业务事件
- 第三方 API 调用失败记录
- 后台任务执行结果
- 管理员操作记录

同时避免记录:

- 明文密码
- 完整银行卡信息
- 长期有效的 API Key
- 敏感 Token
- 大量重复 debug 输出

日志既是排障工具,也是安全和合规风险点。

### 4.4 监控要和发布流程连起来

很多故障发生在发布后几分钟。小团队应该养成习惯:

```text
发布前:
  确认核心路径有监控

发布后:
  看错误率
  看 P95 延迟
  看登录、注册、支付等核心动作是否正常

出问题:
  先回滚或降级
  再慢慢分析根因
```

监控不是上线后才补的装饰,它应该是发布流程的一部分。

---

## 五、一句话总结

**监控的价值不是图表,而是让你更早发现故障、更快定位原因、更稳恢复服务;小团队先覆盖外部可用性、核心 API、错误率、延迟、数据库和账单用量。**
