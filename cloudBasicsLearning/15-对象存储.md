# 对象存储:Bucket、Object、S3、R2、OSS 到底解决什么问题

对象存储经常被初学者理解成"云上的硬盘"。这个理解能帮你入门,但很快会误导你。对象存储不是给程序像本地磁盘一样随便 `open`、`write`、`append` 的文件系统,它更像一个通过 API 读写的大型文件仓库。

> 一句话先记住:对象存储负责低成本、可扩展地保存文件,适合图片、视频、备份、日志、导出文件、静态站资源,不适合当数据库用,也不适合频繁修改小片段的文件。

---

## 一、一句话解释

对象存储里最核心的两个词是:

```text
Bucket:一个存储桶,类似一个文件仓库或命名空间
Object:一个对象,通常就是一个文件加上一些元数据
```

一个对象一般包含:

```text
key:路径式名称,例如 uploads/2026/05/avatar.png
body:文件内容
metadata:Content-Type、缓存时间、自定义标签
permission:公共读、私有、临时授权
version:有些平台支持版本控制
```

典型产品包括:

- AWS S3
- Cloudflare R2
- 阿里云 OSS
- 腾讯云 COS
- Google Cloud Storage
- MinIO 这类 S3 兼容自建方案

对象存储最容易和三类东西混淆。

第一,和服务器硬盘混淆。服务器硬盘适合程序本地读写,但容量、备份、迁移、扩容都要自己操心。对象存储通过 HTTP API 访问,容量可以很大,但不是本地文件系统。

第二,和网盘混淆。网盘面向人,强调同步、预览、分享。对象存储面向程序,强调 API、权限、生命周期、CDN、计费。

第三,和数据库混淆。数据库适合查询结构化数据,例如用户、订单、账单。对象存储适合保存文件本体,例如头像、合同 PDF、导出 CSV。你可以把文件地址存在数据库里,但不要把对象存储当成可以随意查询的表。

## 二、放在系统哪里

对象存储通常放在应用的文件层:

```text
用户浏览器
  -> 前端页面
  -> 后端 API
  -> 对象存储
```

常见模式有两种。

第一种,后端中转上传:

```text
浏览器上传文件
  -> 后端接收文件
  -> 后端上传到对象存储
  -> 后端把 object key 写入数据库
```

这个方式简单,权限容易控制,但大文件会占用后端带宽和执行时间。

第二种,浏览器直传对象存储:

```text
浏览器请求上传凭证
  -> 后端生成临时签名 URL
  -> 浏览器直接上传到对象存储
  -> 后端记录文件信息
```

这个方式更适合图片、视频、大文件。后端只负责授权和记录,不搬运文件本体。

对象存储常和 CDN 搭配:

```text
用户访问图片
  -> CDN
  -> 缓存命中:直接返回
  -> 缓存未命中:回源到对象存储
```

这样可以减少对象存储的读取压力和跨地域延迟。静态网站托管也常用这个模式:

```text
构建产物
  -> 上传到对象存储 Bucket
  -> CDN 加速
  -> 用户访问 HTML / JS / CSS
```

在系统设计里,对象存储通常保存"大块非结构化数据",数据库保存"可查询的结构化索引":

```text
数据库:
  file_id
  user_id
  object_key
  size
  content_type
  created_at

对象存储:
  uploads/user-123/avatar.png
```

## 三、常见套餐和使用限制

对象存储的计费和限制通常拆成好几个维度:

| 限制项 | 说明 | 常见风险 |
| --- | --- | --- |
| 存储容量 | 文件占用多少 GB / TB | 备份、日志、视频长期堆积 |
| 出站流量 | 文件被用户下载产生的流量 | 热门图片、安装包、视频最容易超 |
| 请求次数 | PUT、GET、LIST、DELETE 等操作 | 小文件多、列表频繁、爬虫访问会放大 |
| 跨地域流量 | 不同区域之间传输数据 | 应用和 Bucket 不在同一区域 |
| 生命周期规则 | 自动转低频、归档、删除 | 配错可能导致文件过早归档或删除 |
| 数据取回费 | 归档或低频存储取回可能收费 | 以为便宜,恢复时才发现贵 |
| 最小存储时间 | 某些低频层要求至少存一段时间 | 频繁删除反而不省钱 |
| 公网访问权限 | 公共读、私有读、签名 URL | 配错会泄露文件或导致 403 |

S3、R2、OSS 这类产品名字不同,但你看套餐时都应该问同一组问题:

```text
存进去多少钱?
取出来多少钱?
请求一次多少钱?
跨区域多少钱?
是否有免费额度?
是否有出口流量优惠?
是否和 CDN 绑定后价格不同?
删除、归档、恢复有没有隐藏成本?
```

Cloudflare R2 经常被拿来和 S3 对比,重点通常在出口流量模型和 S3 兼容 API。阿里云 OSS、腾讯云 COS 在国内业务里常见,优势是国内区域、备案和同厂商服务集成。AWS S3 是事实上的对象存储接口标准,很多工具都支持 S3 API。

但小团队不要只看"哪个最便宜"。对象存储一旦接入业务,迁移会涉及:

- 文件 URL 是否写死
- Bucket 名称和区域
- 签名算法和 SDK
- CDN 域名
- CORS 配置
- 生命周期策略
- 数据库里的 object key

所以更重要的是选一个你能长期维护、能看懂账单、能设置权限和告警的方案。

## 四、小团队建议

对象存储最适合放这些东西:

- 用户头像、封面图、附件
- 导出的 CSV、PDF、报表
- 静态网站构建产物
- 备份文件
- 日志归档
- 安装包、压缩包、公开下载文件
- 图片、音频、视频原文件

不建议放这些东西:

- 需要复杂查询的数据
- 高频更新的单个 JSON 文件
- 需要事务保证的业务数据
- 密钥、Token、数据库密码
- 没有访问控制的用户隐私文件

一个保守的小团队架构可以这样设计:

```text
前端:上传前向后端申请临时 URL
后端:校验用户身份、文件大小、文件类型
对象存储:保存文件本体
数据库:保存文件记录和 object key
CDN:加速公开文件读取
```

最低限度的护栏:

- 默认 Bucket 私有,只把确实公开的目录或文件公开
- 上传前限制文件大小和 MIME 类型
- 用户私有文件用短期签名 URL 访问
- 不要把永久 AK / SK 放进前端代码
- 给 Bucket 设置生命周期规则,清理临时文件和过期导出
- 给出站流量和请求数设置预算告警
- 对重要文件开启版本控制或定期备份
- 文件 URL 不要只存在对象存储里,数据库要有业务记录

一个典型坑是"用户上传了头像,但旧头像永远不删"。单个头像只有几百 KB,看起来不值一提;用户多了以后,历史头像、失败上传、临时裁剪文件、导出报表会一起堆积。对象存储便宜,但不是无限垃圾桶。

什么时候该升级或重新设计?

- 文件访问量已经成为主要流量成本
- 上传文件变大,后端中转开始超时
- 需要跨地域访问或合规数据位置
- 需要防盗链、图片处理、转码、水印
- 需要更细粒度的权限和审计
- 备份和恢复流程开始影响业务连续性

小团队最重要的不是一开始就买高级存储层,而是把"文件本体"和"业务记录"分清楚。文件放对象存储,索引和权限放数据库,访问控制由后端发放。

---

## 五、一句话总结

对象存储是云上保存文件的基础设施:S3、R2、OSS 这类产品解决的是低成本存取大规模文件的问题;看套餐时要同时看容量、出站流量、请求次数、生命周期、权限和跨地域成本。
