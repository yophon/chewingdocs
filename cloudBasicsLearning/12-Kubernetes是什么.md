# Kubernetes 是什么:适合谁,不适合谁

> 一句话先记住:Kubernetes 是一个管理大量容器的调度和编排系统,它帮你决定容器运行在哪里、挂了怎么重启、怎么扩容、怎么发布,但它不会替你把应用设计好。

Kubernetes 通常简称 K8s。它不是一种云服务器,不是一种容器,也不是一种部署按钮。它是在多台机器上管理容器集群的系统。

很多小团队对 K8s 有两种极端误解:一种觉得它是现代部署的必选项,不用就不专业;另一种觉得它只是大厂炫技,完全没价值。更准确的说法是:K8s 很强,但它强在"复杂度已经存在时统一管理复杂度",不是给简单项目凭空增加复杂度。

---

## 一、一句话解释

Kubernetes 是容器编排平台。

如果你只有一个容器,手动 `docker run` 就能跑。如果你有三五个服务,一台机器上的 docker compose 也能管理。但如果你有几十个服务、多个实例、多台机器、滚动发布、自动重启、服务发现、配置管理、横向扩容、节点故障迁移,K8s 就开始有意义。

它最容易和 Docker 混淆。Docker 更偏向构建和运行容器,K8s 更偏向在集群里管理容器。可以理解成:

```text
Docker / containerd:把一个容器跑起来
Kubernetes:在一组机器上持续管理很多容器
```

K8s 也容易和 PaaS 混淆。PaaS 通常隐藏底层复杂度,给你一个更简单的部署体验。K8s 则把底层能力暴露成一套通用 API。它更灵活,也更需要工程能力。

---

## 二、放在系统哪里

K8s 位于计算层和平台层之间。它下面是服务器、网络、磁盘和容器运行时;它上面是你的应用服务。

一个典型链路可能是:

```text
用户浏览器
  -> DNS
  -> CDN / WAF
  -> 云负载均衡
  -> Kubernetes Ingress / Gateway
  -> Service
  -> Pod
  -> 容器里的应用进程
  -> 数据库 / 缓存 / 对象存储
```

K8s 里面常见几个对象:

- Pod:最小调度单位,通常里面跑一个或几个容器
- Deployment:声明应用要跑几个副本,以及如何滚动更新
- Service:给一组 Pod 提供稳定访问入口
- Ingress / Gateway:把外部 HTTP 流量转进集群
- ConfigMap / Secret:管理配置和敏感信息
- PersistentVolume:给有状态服务提供持久化存储
- Namespace:做逻辑隔离

这些对象解决的都是"运行中的系统如何持续保持期望状态"。

例如你声明某个服务需要 3 个副本,K8s 会尽量保证集群里一直有 3 个可用副本。如果某个 Pod 崩溃,它会拉起新的。如果某台节点挂了,它会尝试把 Pod 调度到其他节点。如果发布新版本,它可以按策略逐步替换旧版本。

但注意,K8s 保证的是基础编排能力,不是业务正确性。应用启动成功不代表功能正确,Pod 可运行不代表数据库没问题,副本变多不代表瓶颈消失。

## 三、常见套餐和使用限制

K8s 有自建和托管两种常见形态。

自建 K8s 是自己维护控制平面、节点、网络插件、存储插件、证书、升级和安全补丁。它适合学习或强定制需求,但不适合没有平台经验的小团队承载生产系统。

托管 K8s 是云厂商帮你管理部分控制平面,你主要管理工作节点、集群插件和应用。它降低了运维门槛,但不代表没有运维。

常见限制和成本包括:

- 控制平面是否收费
- 工作节点按云服务器计费
- 负载均衡按实例和流量计费
- 云盘和快照计费
- 日志、监控、链路追踪计费
- 镜像仓库和镜像拉取限制
- 公网出口流量计费
- 集群节点数量和 Pod 数量上限
- 集群版本升级窗口和兼容性

小团队常犯的错误是只看"托管 K8s 免费控制平面",却忽略工作节点、负载均衡、日志、监控、云盘、NAT 网关和公网流量。最后账单不是来自 K8s 这个名字,而是来自它带起来的一整套云资源。

另一个限制是人力。K8s 的 YAML 能跑起来不难,难的是长期维护:

- 集群版本升级
- Ingress 控制器升级
- 证书续期
- 镜像漏洞修复
- 资源限制设置
- HPA 扩缩容策略
- 日志和监控保留
- 节点磁盘清理
- Secret 管理
- 网络策略和权限控制

如果团队里没有人愿意负责这些,K8s 会变成一个看起来标准、实际无人维护的平台。

## 四、小团队建议

什么时候 K8s 适合你?

- 服务数量已经明显增多
- 一个服务需要多个副本和滚动发布
- 你需要跨多台机器调度容器
- 团队已经用容器标准化交付
- 有专人或明确轮值负责平台维护
- 需要统一管理多环境、多团队、多服务
- 已经遇到 docker compose 或单机部署的真实边界

什么时候 K8s 不适合你?

- 只有一个 Web 应用和一个数据库
- 还没有稳定用户
- 主要问题是产品没验证,不是部署规模不够
- 团队连容器、反向代理、日志、备份都还不熟
- 只是因为招聘要求或文章推荐才想上
- 没有人能解释集群出问题时怎么排查

对独立开发和小团队来说,很多项目的合理路线是:

```text
静态站 / 前端应用
  -> 静态托管

单体后端 / 早期 SaaS
  -> VPS / 云服务器 / PaaS

几个服务 + 少量后台任务
  -> Docker + docker compose + 托管数据库

服务数量增加 + 多实例发布 + 团队协作复杂
  -> 托管容器平台或托管 K8s

复杂平台化需求
  -> K8s + 专门的平台维护能力
```

如果你只是想减少运维,直接上 K8s 通常不会减少运维。它减少的是"多服务、多节点、多环境在没有统一平台时的混乱",但会增加"集群本身的维护成本"。

最低限度的护栏:

- 优先用托管 K8s,不要轻易自建生产集群
- 给每个服务设置 CPU / 内存 requests 和 limits
- 配置健康检查和优雅关闭
- 日志、监控、告警先落地再谈自动扩容
- 数据库优先用托管数据库,不要一开始就把核心数据库塞进集群
- 不要把 Secret 明文提交到 Git
- 集群升级要有测试环境和回滚路径

---

## 五、一句话总结

Kubernetes 是管理大量容器和多节点部署的强大平台,适合已经有规模和平台维护能力的团队;对大多数早期小项目来说,它不是起点,而是单机、PaaS、容器化部署都碰到真实边界之后的下一步。
