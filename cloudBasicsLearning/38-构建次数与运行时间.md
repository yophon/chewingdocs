# 构建次数与运行时间:CI/CD、Pages、Serverless 的隐形限制

## 一句话解释

**构建次数和运行时间,是在数你的代码被平台构建了多少次、构建用了多久,以及运行时代码实际执行了多久**。

它最容易和"部署成功"混淆。很多平台把流程包装得很简单:

```text
git push
  -> 自动构建
  -> 自动部署
  -> 自动预览
```

但自动不代表没有成本。安装依赖、编译、测试、生成静态页面、优化图片、上传构建产物,都会消耗构建资源。Serverless 函数、边缘函数、后台任务、定时任务虽然不用你管理服务器,但每次执行也会消耗运行时间和调用额度。

一句话先记住:**CI/CD 消耗的是构建资源,Serverless 消耗的是运行资源,两者都可能成为免费套餐里的隐形上限**。

## 放在系统哪里

构建次数和运行时间分别发生在两个阶段。

第一阶段是交付阶段:

```text
开发者提交代码
  -> Git 平台触发 webhook
  -> CI/CD 拉取代码
  -> 安装依赖
  -> 运行测试
  -> 执行 build
  -> 上传产物
  -> 部署到 Pages / CDN / 容器 / 函数平台
```

这里消耗的是构建次数、构建分钟、构建并发、缓存容量、产物存储、预览部署数量。

第二阶段是运行阶段:

```text
用户请求
  -> CDN
  -> Serverless / Edge Function / API Route
  -> 数据库 / 对象存储 / 第三方 API
  -> 返回响应
```

这里消耗的是函数调用次数、执行时间、内存、CPU 时间、并发、超时时间、后台任务运行时间。

现代前端平台经常把这两个阶段放在一个产品里。例如 Pages、Vercel、Netlify、Cloudflare、Firebase、Supabase Edge Functions 这类平台,既可能帮你构建前端,也可能运行函数。于是小团队很容易以为"网站托管是免费的",却忽略了:

- 每次 push 都会触发构建.
- 每个 pull request 都可能生成预览环境.
- 每次构建都安装依赖和生成页面.
- SSR 页面每次访问都可能运行函数.
- 定时任务即使没人访问也会运行.
- 失败构建和失败任务同样消耗额度.

## 常见套餐和使用限制

构建相关限制通常包括:

| 限制项 | 含义 | 常见风险 |
| --- | --- | --- |
| 构建分钟 | 构建任务占用执行器的时间 | 依赖安装慢、静态页面多、测试慢 |
| 构建次数 | 每月或每日可触发多少次构建 | 频繁 push、多分支预览 |
| 构建并发 | 同时能跑几个构建 | 多人提交时排队 |
| 构建超时 | 单次构建最长能跑多久 | 大项目或卡住的脚本失败 |
| 缓存容量 | node_modules、构建缓存、测试缓存 | 缓存失效导致每次全量安装 |
| 预览部署 | PR 或分支预览环境数量 | 分支不清理,占用额度 |
| 产物大小 | build 输出文件大小 | 大文件、source map、图片混入产物 |

运行相关限制通常包括:

| 限制项 | 含义 | 常见风险 |
| --- | --- | --- |
| 调用次数 | 函数被触发多少次 | API 太碎、爬虫、轮询 |
| 执行时间 | 函数从开始到结束用了多久 | 慢查询、第三方 API 慢 |
| 最大时长 | 单次函数最多运行多久 | 文件处理、AI 调用、批任务超时 |
| 内存 | 函数可用内存 | 图片处理、PDF 生成、批量数据 |
| 并发 | 同时运行多少个实例 | 流量突增时排队或失败 |
| 冷启动 | 空闲后首次启动延迟 | 低频接口偶发变慢 |
| 定时任务次数 | cron 或 scheduled job 触发次数 | 任务频率过高 |

免费套餐里,这些限制经常不是单独出现,而是组合出现:

```text
构建分钟用完 -> 新版本无法自动发布
构建并发不足 -> 多人提交排队
函数调用超额 -> 开始计费或拒绝请求
函数执行超时 -> 用户看到 504 或任务失败
内存不足 -> 图片处理、PDF 生成直接崩溃
后台任务无限重试 -> 同时消耗调用次数和运行时间
```

看套餐时,要专门确认四件事。

第一,构建是否按分钟、次数还是并发限制。有的平台主要限制总分钟,有的平台更关注并发和队列,有的平台对团队协作和预览部署另算。

第二,失败是否计入用量。多数情况下,失败构建、失败测试、失败函数、超时任务也会消耗资源。一个错误配置的构建脚本如果反复触发,可能没有成功发布任何东西,却消耗了大量额度。

第三,预览环境是否消耗额度。PR 预览很好用,但每个分支都自动构建,会把个人项目变成持续消耗构建分钟的机器。多人小团队尤其要注意。

第四,运行时限制是否适合业务。短 API 可以放 Serverless,长时间导入、视频处理、批量邮件、爬虫任务、AI 批处理未必适合放在短超时函数里。强行放进去,不是省服务器,而是把失败和重试转移到账单里。

查看方法上,至少看:

```text
Deployments:每次构建耗时和失败原因
Build logs:时间花在依赖安装、测试还是生成页面
Usage:构建分钟、函数调用、运行时间
Functions:每个函数的调用量、耗时、错误率
Cron / Jobs:定时任务频率和执行结果
Billing details:超额后是计费、限速还是暂停
```

## 小团队建议

小团队要把构建和运行分开治理。

构建侧先做这些事:

- 不要每个无关分支都自动部署生产级预览.
- 合理使用构建缓存,避免每次全量安装依赖.
- 把慢测试分层,提交时跑关键测试,定时或合并前跑完整测试.
- 静态页面数量很大时,评估增量构建或按需生成.
- source map、大图片、临时文件不要无意进入产物.
- 删除长期不用的预览环境和旧部署.
- 构建失败要停止循环触发,不要让机器人反复提交或重跑.

运行侧重点是把短请求和长任务分开:

```text
适合 Serverless:
  - 登录回调
  - 表单提交
  - 短 API
  - webhook 处理
  - 轻量数据转换

不适合直接放短超时 Serverless:
  - 大文件转码
  - 长时间爬虫
  - 大批量数据导入
  - 长时间 AI 任务
  - 需要稳定连接的实时服务
```

不适合 Serverless 的任务,可以考虑队列加 worker、容器任务、定时批处理、专门的后台服务。判断标准不是"能不能跑起来",而是"失败后能不能恢复,重试会不会放大成本,最长时间是否超过平台限制"。

对预算护栏,建议至少设置:

- 构建分钟接近上限提醒.
- 单次构建耗时异常提醒.
- 函数调用次数异常增长提醒.
- 函数平均耗时和超时率提醒.
- 定时任务失败重试次数提醒.
- 月度账单和用量预算提醒.

对工程流程,建议有三个小规则:

```text
1. main 分支才自动生产部署
2. PR 预览按需开启或定期清理
3. 后台任务必须有最大运行时间、最大重试次数和幂等设计
```

另外,不要把构建平台当成通用计算平台。有些人会用 CI 去跑爬虫、批量转码、数据同步、定时脚本。短期看方便,长期看会挤占构建额度,也会让发布流程和后台任务互相影响。CI 的主要职责是交付代码,不是承载全部后台计算。

一个保守原则是:**构建资源保障发布,运行资源保障用户请求,后台长任务要有自己的队列和停止条件**。

## 一句话总结

**构建次数和运行时间是自动部署时代最容易被忽略的成本维度;push、预览、测试、SSR、函数和定时任务都会消耗额度,必须设置用量告警和停止条件**。
