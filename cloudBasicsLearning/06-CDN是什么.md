# CDN 是什么:缓存、边缘节点、回源、命中率

> 一句话解释:**CDN 是把用户常访问的静态内容提前放到离用户更近的节点上,让用户少跑远路,也让你的源站少扛压力**。

如果你的网站只有一台服务器在新加坡,北京用户每次打开图片、CSS、JS 都要跨境访问这台服务器。页面不是不能打开,而是慢,而且一旦某篇文章被很多人转发,这台服务器要同时处理大量图片和脚本下载,很容易带宽打满。

CDN 的做法是:用户先访问 CDN 的边缘节点。节点上有缓存,就直接返回;没有缓存,再去你的源站取一次,然后把结果存下来。下一位用户再访问同一个文件,就不用再打到源站。

最容易混淆的是:**CDN 不是你的服务器,也不是数据库,它通常不负责生成业务结果**。它擅长缓存图片、视频、CSS、JS、下载包、静态 HTML,也可以帮你做 HTTPS、压缩、简单访问控制。但用户登录、下单、写评论这类动态逻辑,最终还是要回到你的应用服务器或后端服务。

## 一、放在系统哪里

一个最常见的小网站访问链路大概是这样:

```text
用户浏览器
  -> DNS 解析
  -> CDN 边缘节点
  -> 源站服务器 / 静态托管 / 对象存储
  -> 数据库或其他后端服务
```

CDN 位于用户和源站之间。它像一个前置缓存层,负责拦住那些不需要每次都问源站的请求。

举个真实场景:

```text
你做了一个产品官网:
  - 首页 HTML:偶尔更新
  - logo.png:几乎不变
  - app.css:每次发版才变
  - bundle.js:每次发版才变
  - /api/signup:用户注册接口,每次请求都不一样

适合 CDN 缓存:
  - logo.png
  - app.css
  - bundle.js
  - 带版本号的静态图片

通常不该随便缓存:
  - /api/signup
  - /api/orders
  - /dashboard
  - 带用户 Cookie 的私人页面
```

CDN 的关键动作叫**回源**。当边缘节点没有缓存、缓存过期,或者规则要求不能缓存时,CDN 会把请求转发到源站。源站可以是云服务器、对象存储、静态托管平台,也可以是另一个反向代理。

所以排查 CDN 问题时,要分清两件事:

- CDN 返回慢:可能是节点、网络、缓存没命中。
- 源站返回慢:CDN 只是转发,真正慢的是你的服务器或后端。

一个很实用的指标是**缓存命中率**。命中率 90% 的意思是,100 次请求里大约 90 次由 CDN 直接返回,只有 10 次回源。命中率低不一定错,但如果一个图片站、下载站、文档站命中率长期很低,通常说明缓存规则、URL 版本、响应头或资源路径设计有问题。

## 二、常见套餐和使用限制

CDN 免费套餐或入门套餐常见限制不在"能不能开 CDN",而在这些地方:

1. **流量或带宽限制**

   CDN 帮你挡了源站压力,但流量仍然是资源。图片、视频、下载包特别容易把出站流量拉高。很多小项目账单不是被服务器 CPU 打爆,而是被图片和文件下载打爆。

2. **缓存规则数量**

   免费套餐可能只允许配置少量页面规则、缓存规则、重定向规则。刚开始够用,但当你想分别处理 `/api/*`、`/assets/*`、`/admin/*`、图片压缩、国家地区访问策略时,规则数量就会变成限制。

3. **缓存刷新能力**

   你发版后发现用户还看到旧 JS,需要清缓存。套餐可能限制手动刷新次数、刷新路径数量,或者全站刷新速度。小项目最常见的事故是:代码已经部署,但 CDN 还在返回旧文件。

4. **动态内容加速限制**

   静态文件缓存比较便宜,动态请求加速、边缘计算、图片处理、智能路由通常更贵,也更容易有请求数、CPU 时间、执行时间限制。

5. **日志和分析保留时间**

   免费套餐通常只能看很短时间的访问统计,或者只能看聚合数据。出了问题之后,你可能只知道"昨天流量暴涨",但看不到具体哪个 URL、哪个地区、哪个来源打上来。

超额后可能发生三种情况:

- 继续服务,但开始按量计费。
- 限速或降低功能。
- 直接拒绝部分请求或要求升级。

具体是哪一种,必须看套餐页和服务条款。不要只看"免费 CDN"四个字,要看流量、请求数、规则数、日志、刷新、图片处理是不是分开计。

## 三、小团队建议

如果你只是部署一个官网、文档站、博客、开源项目主页,CDN 很值得尽早使用。它能让静态资源更快,也能让你的源站少暴露在公网压力下。

但不要把 CDN 当成万能加速器。小团队最应该先做好几件朴素的事:

- 静态资源文件名带版本号,例如 `app.8f3a.js`,发版后让新旧文件自然区分。
- 不要随便缓存带登录态的页面,尤其是带 Cookie、Authorization、用户订单、用户资料的接口。
- 给 `/api/*` 和 `/assets/*` 分开设置缓存策略。
- 图片尽量压缩后再上传,不要指望 CDN 替你承担所有图片成本。
- 上线前确认源站只接受必要流量,避免用户绕过 CDN 直接打源站。
- 设置预算告警和流量告警,尤其是有大图、下载包、视频的项目。

一个常见的保守架构是:

```text
域名
  -> DNS 托管
  -> CDN
  -> 静态托管 / 对象存储
  -> API 服务
  -> 数据库
```

对独立开发者来说,前端静态资源走 CDN,API 先少缓存或不缓存,是比较稳的起点。等你知道哪些接口可以公开缓存,再逐步加规则。

什么时候该升级?

- 静态资源流量稳定超过免费额度。
- 需要更细的缓存规则、访问规则和日志。
- 有明显跨地域用户,源站单一区域响应慢。
- 需要图片处理、视频分发、下载加速。
- 遇到恶意流量,需要和 WAF、Bot 管理、DDoS 防护配合。

---

## 四、一句话总结

**CDN 的核心不是"让网站神奇变快",而是把可缓存的内容放到离用户更近的位置,减少回源次数;小团队用 CDN 的关键,是清楚哪些内容能缓存、缓存多久、超额后怎么收费。**
