# 防刷与 Bot 管理:验证码,限流,风控,误杀

## 一句话解释

**防刷是限制异常高频或低成本滥用行为;Bot 管理是识别请求背后是真人,正常爬虫,自动脚本还是恶意流量**.

小团队第一次真正需要防刷,通常不是被黑客"攻破",而是业务功能被低成本滥用:

```text
注册接口被批量创建账号
短信验证码被刷爆额度
登录接口被撞库
搜索接口被爬虫打满
优惠券被脚本批量领取
AI 接口被人套壳调用
表单被垃圾内容灌满
```

这些请求看起来都像正常 HTTP 请求.它们不一定触发 WAF 规则,也不一定是 DDoS.问题在于:单次请求合法,大量请求合起来就变成了成本,风险和用户体验问题.

最容易误解的是:**防刷不是"加一个验证码"就完事**.验证码会增加攻击成本,也会增加正常用户成本.限流能保护系统,也可能误伤公司网络,校园网,移动网络出口下的一批正常用户.Bot 管理能识别自动化流量,但识别永远有误判和漏判.

所以防刷的核心不是把所有可疑请求都拦掉,而是在安全,成本,转化率和误杀之间做分层控制.

## 放在系统哪里

防刷通常跨好几层:

```text
用户 / Bot
  -> DNS / CDN
  -> WAF / Bot 管理 / Rate Limit
  -> API Gateway / 反向代理
  -> 应用服务
  -> 账号系统 / 风控规则
  -> 数据库 / 队列 / 第三方服务
```

越靠前的防护越便宜,越靠后的判断越准确.

CDN 或 WAF 层适合做粗粒度拦截,例如同一 IP 每分钟请求过多,明显恶意 User-Agent,访问不存在路径,来自高风险地区的异常请求.它的优点是请求还没到源站就被挡住,能省服务器和数据库成本.

应用层适合做业务语义判断,例如同一个手机号 10 分钟内只能发 3 次验证码,同一用户每天只能领取一次优惠券,同一账号登录失败 5 次后需要二次验证.云厂商入口层通常不知道"这个手机号是不是刚刚领过券",这必须由应用自己判断.

一个真实链路:

```text
用户点击"发送短信验证码"
  -> CDN 限制同一 IP 的总体频率
  -> 网关限制 /send-code 路径的请求速率
  -> 应用检查手机号,设备,账号,历史发送次数
  -> 风险较低:直接发送
  -> 风险中等:要求验证码或等待
  -> 风险较高:拒绝并记录
  -> 短信服务商真正扣费
```

这条链路里,最重要的保护点通常在第三方扣费动作之前.短信,邮件,AI 推理,支付查询,地图 API,OCR 等接口都一样:只要每次调用会产生外部成本,就要先防刷,再调用.

## 常见套餐和使用限制

免费或入门套餐常见能力包括:

- 基础 WAF 规则.
- 简单 IP rate limit.
- 简单验证码或挑战页.
- 基础 Bot 分数或 Bot 标签.
- 按路径,国家地区,User-Agent 写少量规则.

限制通常出现在这些地方:

- 可创建的限流规则数量有限.
- 只能按 IP 限流,不能按账号,Token,手机号,设备指纹限流.
- Bot 管理高级识别需要付费套餐.
- 验证码调用量,短信量,邮件量,AI 调用量会单独计费.
- 日志和报表保留时间短,误杀后很难复盘.
- 规则只在经过 CDN 或网关的流量上生效,绕过入口就没有保护.

最常见的误杀来自 IP 维度太粗:

```text
一个公司办公室 200 人共用同一个出口 IP
一个学校宿舍楼共用几个出口 IP
移动网络用户频繁切换 NAT 出口
海外用户走代理或企业 VPN
```

如果你只按 IP 做严格限流,一个正常团队同时试用你的产品,可能被当成攻击流量.登录,注册,支付,提交表单这类关键路径一旦误杀,损失的不是一点流量,而是转化率和信任.

另一个常见问题是只看请求数量,不看业务成本.比如首页接口每秒 100 次可能问题不大,但 AI 生成接口每秒 2 次就可能烧钱.防刷阈值不能只按"服务撑不撑得住"设,还要按"业务亏不亏得起"设.

## 小团队建议

小团队先做高风险接口,不要一开始追求完整风控平台.优先级可以这样排:

```text
会产生直接成本的接口
  -> 短信,邮件,AI,OCR,第三方 API

会影响账号安全的接口
  -> 登录,注册,找回密码,修改邮箱,绑定手机号

会影响业务资产的接口
  -> 优惠券,邀请码,积分,试用额度,提现

会拖垮系统的接口
  -> 搜索,导出,上传,批量查询,复杂报表
```

最低限度建议:

- 对登录,注册,验证码,找回密码加单独限流.
- 对所有会产生外部成本的接口设置每日预算和调用上限.
- 限流维度不要只有 IP,尽量组合账号,手机号,邮箱,设备,Token,路径.
- 先用温和策略,例如延迟,排队,二次验证,不要动不动永久封禁.
- 给被拦截用户留出恢复路径,例如稍后重试,联系客服,换一种验证方式.
- 记录拦截原因和规则命中,否则误杀时无法解释.
- 对管理后台和内部 API 使用更严格的 allowlist,MFA 和访问控制.

什么时候该升级?

- 短信,邮件,AI 或第三方 API 成本开始明显受刷量影响.
- 出现撞库,批量注册,垃圾内容,爬虫采集.
- 产品有免费试用,优惠券,积分,邀请奖励.
- 企业客户要求可解释的安全策略和审计记录.
- 你已经无法靠几条固定限流规则区分正常高峰和恶意自动化.

防刷策略要可观察.至少要能回答:

- 哪些接口被限流最多.
- 哪些规则误杀投诉最多.
- 正常用户高峰时的请求频率大概是多少.
- 被拦截请求是否仍然造成了第三方成本.
- 攻击者是否绕过了 CDN 或网关直接打源站.

不要把正常用户当成攻击者.一个合理的策略通常是分级处理:

```text
正常请求 -> 放行
轻微异常 -> 降速或排队
中等风险 -> 验证码 / 邮箱验证 / MFA
高风险 -> 临时封禁并记录
确定恶意 -> 封禁,告警,拉黑关联标识
```

## 一句话总结

**防刷和 Bot 管理的目标不是拦掉所有机器人,而是用最小误杀保护账号安全,外部成本和核心接口;规则越靠前越省钱,判断越靠近业务越准确**.
