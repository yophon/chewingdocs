# 16 · 信息熵:压缩为什么有极限

`gzip -9` 跑了半天压不动了,再换 `xz -9e` 也只多挤出 2%——**不是工具不够强,是这个文件的"信息量"已经接近物理下限了**。这个下限有名字,叫**信息熵 H**。本篇把 H 讲明白:它从哪儿来、怎么算、为什么 GPT 的 Tokenizer 也在偷偷最小化它、什么时候你应该停止再调压缩参数。

> 一句话先记住:**熵 H 是"你这堆数据里平均每个符号最少要多少 bit 来编码"的下界**。Huffman / 算术编码 / zlib / zstd / BPE Tokenizer——所有"压缩"的终点都是 H,谁也越不过去。

---

## 一、为什么必须懂这个

工程里你会反复撞到 H,但课本不会告诉你"它在哪个旋钮上":

| 场景 | 不懂 H 时 | 懂 H 后 |
| --- | --- | --- |
| `gzip` 压完只剩 30%,还能再小吗 | 换工具乱试 | 算一下数据的 H,知道天花板在哪 |
| Tokenizer 词表设 32k 还是 128k | 拍脑袋 | 知道是在最小化 token 序列的 H |
| 训练数据噪声很大,模型学不动 | 怀疑模型架构 | 数据本身 H 高,信噪比差 |
| 决策树用什么准则分裂 | 抄默认 | 信息增益就是 H 的减少量 |
| RLHF 的 KL 罚项往哪罚 | 调 β 玄学 | 先得搞清 H 才能搞清 KL |

> **经验法则**:**任何"压缩 / 编码 / 选择 / 区分能力"的算法,底下大概率有 H 在做尺子**。看不到不代表没有,只代表这个旋钮你还不会调。

## 二、直觉:信息 = 惊讶度

Shannon 当年就一个问题:**怎么量化"惊讶"**?

```
    事件 A:太阳明天会升起   → 你不会惊讶 → 信息 ≈ 0
    事件 B:你抽到了大乐透   → 你很惊讶 → 信息很大
    事件 C:抛一次硬币是正面 → 中等惊讶 → 信息适中
```

Shannon 的设计是:**事件 x 的信息量 = -log p(x)**。p 越小,-log p 越大。底数取 2 就是 bit、取 e 就是 nat。

```
p = 0.5      → -log₂(0.5)   = 1 bit
p = 0.25     → -log₂(0.25)  = 2 bit
p = 0.001    → -log₂(0.001) ≈ 9.97 bit
```

> **核心直觉**:**罕见事件携带的信息多,常见事件携带的信息少**。"今天上班路上没出事"是 0 bit,"今天上班路上下了陨石"是几十 bit。

把所有事件的"惊讶度"按概率加权平均,就是这个分布的**信源熵**:

$$
H(X) = -\sum_x p(x) \log_2 p(x) \quad (\text{单位 bit})
$$

## 三、三个最值钱的小例子

```
例1:公平硬币(正反 0.5/0.5)
   H = -(0.5·log0.5 + 0.5·log0.5) = 1 bit
   含义:平均要 1 bit 才能告诉别人结果

例2:8 个等概率事件(骰子扩展)
   H = -8·(1/8·log(1/8)) = log₂8 = 3 bit
   含义:8 选 1,正好 3 bit,经典

例3:偏硬币(正 0.9 / 反 0.1)
   H = -(0.9·log0.9 + 0.1·log0.1) ≈ 0.469 bit
   含义:大部分时候是正面,你已经"猜到"一半了,
        每次告诉你结果只需要 0.469 bit

例4:全是正面(p=1.0)
   H = 0
   含义:没悬念,根本不需要 bit
```

ASCII 看一下熵随分布的变化(二项,横轴 p):

```
H(p)
 1.0 |          ___---___
     |       _-/         \-_
     |     -/               \-
 0.5 |    /                   \
     |   /                     \
     |  /                       \
 0.0 |/_________________________ \
     0.0    0.25   0.5   0.75   1.0  → p
```

> **经验法则**:**分布越均匀,H 越大;分布越集中(某个值概率高),H 越小**。这一条贯穿后面所有章节。

## 四、信源熵公式拆解

```
H(X) = - Σ p(x) · log p(x)
       │  │      │
       │  │      └── 单点信息 -log p(x)
       │  └────────── 用概率加权
       └───────────── 整体取负(因为 log p 是负数)
```

几个工程师必须记住的性质:

| 性质 | 数学 | 工程含义 |
| --- | --- | --- |
| 非负 | H(X) ≥ 0 | 没有"负信息" |
| 上界 | H(X) ≤ log\|X\| | 等概率分布达到上界 |
| 加常数 / 重命名不变 | H(X) = H(f(X)),f 双射 | 怎么给字符编号不影响 H |
| 独立时可加 | H(X,Y) = H(X) + H(Y) | 两个独立信源 = 两个 H 相加 |

> **避坑**:`p log p` 在 p=0 处是 `0 · (-∞)`,**约定 = 0**(用极限,洛必达就出来)。代码里要么用 `np.where(p>0, p*np.log(p), 0)`,要么用 `scipy.stats.entropy` 已经处理好。

## 五、联合熵、条件熵、链式法则

一个变量讲完,两个变量怎么办?加一维就行:

```
联合熵   H(X, Y) = -ΣΣ p(x,y) log p(x,y)
条件熵   H(Y|X) = -ΣΣ p(x,y) log p(y|x)
链式     H(X, Y) = H(X) + H(Y|X)
```

链式法则的直觉:**先告诉你 X 用 H(X) bit,再在 X 已知的前提下告诉你 Y 用 H(Y|X) bit**——总数等于 H(X,Y),正合理。

来个具体数字感受一下,假设天气和带伞的联合分布:

```
            带伞 y=1   不带 y=0
雨 x=1       0.3        0.05         (P(雨)=0.35)
晴 x=0       0.05       0.6          (P(晴)=0.65)
```

```python
import numpy as np
p = np.array([[0.3, 0.05],[0.05, 0.6]])

px = p.sum(axis=1)               # 边缘 P(X)
py = p.sum(axis=0)               # 边缘 P(Y)

H_X  = -(px * np.log2(px)).sum()                  # ≈ 0.934
H_Y  = -(py * np.log2(py)).sum()                  # ≈ 0.954
H_XY = -(p   * np.log2(p   + 1e-12)).sum()        # ≈ 1.466
H_Y_given_X = H_XY - H_X                          # ≈ 0.532

print(H_X, H_Y, H_XY, H_Y_given_X)
```

> **核心直觉**:**H(Y|X) 是"知道了 X 之后,Y 还剩多少不确定"**。下雨时几乎一定带伞,所以 H(Y|X) 远小于 H(Y)。这一差额就是 17 篇要讲的**互信息**。

## 六、Huffman 编码:贪心逼近 H

知道了"H 是下界",下一步问:**怎么真的编出接近 H 的码?**Huffman 是最朴素的答案。

### 6.1 构造一个最小例子

字符 A B C D 的概率:0.5 / 0.25 / 0.125 / 0.125

```
  H = -(0.5·log0.5 + 0.25·log0.25 + 2·0.125·log0.125)
    = 0.5 + 0.5 + 0.75 = 1.75 bit/字符
```

Huffman 贪心(每次合并两个最小概率):

```
步骤 1: C(0.125)+D(0.125) → CD(0.25)
步骤 2: B(0.25)+CD(0.25)  → BCD(0.5)
步骤 3: A(0.5)+BCD(0.5)   → 1.0(根)

最终编码(左 0 右 1):
   A → 0
   B → 10
   C → 110
   D → 111
```

平均码长 = 0.5·1 + 0.25·2 + 0.125·3 + 0.125·3 = **1.75 bit**——**正好等于 H**。

> **经验法则**:**当所有概率都是 2 的负幂时,Huffman 能精确达到 H**;否则有"取整"的浪费,平均码长 ∈ [H, H+1)。

### 6.2 一段能跑的实现

```python
import heapq
from collections import Counter

def huffman(text):
    freq = Counter(text)
    heap = [[w, [c, ""]] for c, w in freq.items()]
    heapq.heapify(heap)
    while len(heap) > 1:
        lo = heapq.heappop(heap); hi = heapq.heappop(heap)
        for pair in lo[1:]: pair[1] = '0' + pair[1]
        for pair in hi[1:]: pair[1] = '1' + pair[1]
        heapq.heappush(heap, [lo[0]+hi[0]] + lo[1:] + hi[1:])
    return dict((c, code) for c, code in heap[0][1:])

text = "AAAAAAAABBBBCCDD"  # 8A 4B 2C 2D
codes = huffman(text)
avg = sum(text.count(c)*len(code) for c, code in codes.items()) / len(text)
print(codes)   # {'A': '0', 'B': '10', 'C': '110', 'D': '111'} 或等价
print(avg)     # 1.75
```

> **避坑**:Huffman 是**前缀码**——任意一个码不是另一个码的前缀。这保证了解码时不需要分隔符。代码实现时如果搞错前缀关系,解码会乱套。

## 七、算术编码 / zlib / zstd:再榨一点

Huffman 的浪费来自"每个符号必须用整数 bit"。**算术编码**把整段消息编成 [0,1) 区间内的一个实数,**平均码长可以无限逼近 H**(代价是实现复杂、专利曾经困扰过)。

主流压缩库不直接讲熵,但底层都在做同一件事:

| 工具 | 核心思路 |
| --- | --- |
| `gzip` / `zlib` | LZ77(找重复子串)+ Huffman(编码偏移和字面量) |
| `bzip2` | BWT(块排序)+ MTF + Huffman |
| `zstd` | LZ77 + **FSE(有限状态熵编码,本质是算术编码的工程版)** |
| `xz` / LZMA | LZ77 变种 + Range Coder(算术编码) |

> **经验法则**:**LZ 系列负责"消除重复"、Huffman/算术编码负责"逼近 H"**。前者把熵降下来(把"abc abc abc"折成"abc ×3"),后者再贴近这个新的 H。两步都到极限了,你就压不动了。

## 八、Shannon 信源编码定理(口语版)

完整的定理只有一句话:**任何无损编码的平均码长 L ≥ H(X);且存在编码使 L 任意接近 H(X)**。

```
        L (平均码长)
         │
         │
         │   你能做到的最低
         │ ─────────  ←  H(X)
         │
         │  Huffman: H ≤ L < H+1
         │  算术编码: L → H
         │
         └──────────────────  超不过 H,这是物理事实
```

> **核心直觉**:**H 是数据的"信息密度",压缩工具只是搬运工,不能创造信息减少**。一个真正随机的 1GB 文件 H ≈ 8 bit/byte,**任何压缩工具都不可能压小**——这就是为什么 `/dev/urandom` 输出和已经压过的 zip 文件再压几乎没用。

## 九、工程对应与局限

### 9.1 压缩比的物理天花板

```
英文文本    H ≈ 1.3 bit/char (Shannon 1951 实验估计,基于人类预测)
            UTF-8 是 8 bit/char  → 理论压到 16% 左右
            实际 zstd -19 大约 25-30%(模型 + 编码 overhead)

随机数据    H = 8 bit/byte  → 压不动,有些工具甚至会变大(头部开销)

源代码      H ≈ 4-5 bit/char  → 比英文还能压(变量名、关键字大量重复)

JSON 日志   重复字段名极多   → zstd 字典模式能压到 5-10%
```

> **避坑**:**已经压过的文件不要再压**——`gzip` 套 `gzip` 通常变大几个字节(头部开销),熵已经接近 8 bit/byte。

### 9.2 Tokenizer 的 BPE 也在最小化 H

GPT-4 / Llama / Claude 用的 **BPE(Byte Pair Encoding)**,本质是**贪心找高频字节对合并,把 token 序列的熵尽量降低**:

```
原文 (UTF-8 字节序列):  H ≈ 8 bit/byte
切成 BPE token 后:      序列变短,但每个 token 的"概率分布"更均匀
                        → token 序列的 H 反而接近 token 表大小的 log
                        → 总信息量接近不变,但表示更紧凑
```

为什么词表大小常见 32k / 64k / 128k?**32k ≈ 2^15,所以每 token 平均 15 bit 编码空间**——和数据自然的信息量匹配,token 太少会过细,太多会浪费上下文窗口。

> **经验法则**:**BPE 词表大小是个 H 平衡题**。词表大→序列短→更省 context,但每 token 编码长;词表小则反过来。32k-128k 是工业上反复试出来的甜区。

### 9.3 数据集熵越低,模型越好学

```
高熵数据(熵接近上界)
  - 互联网爬虫原始数据:噪声多、风格杂、H 大
  - 模型学起来收敛慢、loss 高位震荡

低熵数据(熵远小于上界)
  - 严选过的 SFT 对话数据:格式统一、答案规整
  - 模型学起来 loss 快速下降、易过拟合
```

> **核心直觉**:**模型本质上是在拟合数据分布,数据 H 越大,模型需要的容量也越大**。这就是为什么"数据清洗"和"换更大的模型"经常等价——都是在让模型容量 ≥ 数据 H。

### 9.4 局限

- **H 是渐近极限**——单条短消息编不到 H,必须长序列上看
- **H 假设独立同分布**——真实数据有依赖,所以 LZ77 这种"找重复"才有效,这本质是降低**条件熵**
- **H 不告诉你内容质量**——一段全是 "asdfqwerty" 的随机字符 H 高得很,但毫无意义
- **真实压缩有 overhead**——存编码表、文件头、对齐,小文件压完可能更大

### 9.5 一行代码工具箱

```python
import scipy.stats as st
import numpy as np

# 离散分布的熵(scipy 已处理 0 概率)
p = np.array([0.5, 0.25, 0.125, 0.125])
H = st.entropy(p, base=2)        # → 1.75

# 任意数据序列的经验熵估计
from collections import Counter
def empirical_entropy(seq, base=2):
    counts = np.array(list(Counter(seq).values()), dtype=float)
    p = counts / counts.sum()
    return st.entropy(p, base=base)

empirical_entropy("AAAAAAAABBBBCCDD")   # → 1.75
empirical_entropy(open("big.json").read())   # 看你 JSON 有多重复
```
