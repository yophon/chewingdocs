    图论基础

    工程师每天都在处理图,只是没意识到。npm/cargo/apt 装包是 DAG 拓扑排序;微服务调用链是有向图;社交关注、商品共购、知识图谱、依赖检测、路由协议——全都是图。这些场景里你不需要"证明欧拉定理",你需要的是:给你一个图,你能立刻判断"用邻接矩阵还是邻接表"、"BFS 还是 DFS"、"Dijkstra 还是 Bellman-Ford"——选错了不是慢一点,是直接 OOM 或答案错。

    一句话先记住:图论的工程心法 = 先看图密集还是稀疏(决定表示法)→ 再看权重正负与全源单源(决定最短路算法)→ 再看是否需要回溯/剪枝(决定 BFS 还是 DFS)。学会这三步分类,80% 的图问题就能秒选算法。

    一、为什么要懂图论

    场景 不懂图论时 懂了之后
    包管理器报"依赖循环" 一头雾水,不知道怎么找环 知道是 DFS 找回边或拓扑排序失败
    路由表设计 抄默认就行 知道 OSPF 是 Dijkstra,RIP 是 Bellman-Ford
    社交"二度好友" 写多重 for 循环 一眼 BFS 第二层
    推荐系统找相似商品 搬模型 知道是协同过滤 / 二分图随机游走
    报表算公司架构层级 递归查 DB 知道是树/DAG 上的拓扑或最长路径
    Spark / Airflow DAG 调度 看不懂 stuck 在哪 知道是拓扑顺序 + 关键路径

    图论真正难的不在算法实现(NetworkX 一行就好),难在选对算法 + 设计图模型——把业务问题翻译成"什么是节点、什么是边、是否有向、是否带权"。

    二、图的表示:邻接矩阵 vs 邻接表

    2.1 一张图就明白

    考虑一个 5 节点、7 边的有向图:

       1  →  2
   ↓     ↓
   3  →  4
   ↓     ↗
   5  ──┘

边集:1→2, 1→3, 2→4, 3→4, 3→5, 5→4

    邻接矩阵 A[i][j] = 1 表示有边 i→j:

           1  2  3  4  5
    1[ 0  1  1  0  0 ]
    2[ 0  0  0  1  0 ]
A = 3[ 0  0  0  1  1 ]
    4[ 0  0  0  0  0 ]
    5[ 0  0  0  1  0 ]

    邻接表(每个节点 → 邻居列表):

    1 → [2, 3]
2 → [4]
3 → [4, 5]
4 → []
5 → [4]

    2.2 选哪个

    维度 邻接矩阵 邻接表
    空间 O(V²) O(V+E)
    查"u→v 是否有边" O(1) O(deg(u))
    遍历 u 的所有邻居 O(V) O(deg(u))
    添加 / 删除边 O(1) O(1) 或 O(deg)
    适合 稠密图(E ≈ V²) 稀疏图(E ≪ V²)
    工程典型 小规模 + 频繁查边(< 几千节点) 几乎所有大规模图

    经验法则:节点数 N > 1 万,直接用邻接表;N ≤ 1000 又频繁查边可以考虑邻接矩阵。中间档要看 E/V² 这个密度比。

    2.3 一个真实陷阱

    社交图 N=10 亿、平均每人 200 个关注:

    邻接矩阵:N²    = 10^18 个 bit ≈ 10^17 字节   → 不可能存
邻接表:N + E = 10^9 + 2·10^11 ≈ 2 × 10^11 项   → 几 TB 可处理

    避坑:永远先估一下 V² 是否爆炸。Python 用 numpy.zeros((N,N)) 想偷懒做邻接矩阵,N=5 万就要 2.5 GB(float32);N=10 万就 10 GB,直接 OOM。稀疏矩阵用 scipy.sparse.csr_matrix——本质就是邻接表的压缩存储。

    2.4 度 / 入度 / 出度 / 自环 / 重边

                    ┌──────┐
                │      │   ← 自环(self loop):u → u
                ▼      │
                u  ────┘

   v ─────────────► u
   v ─────────────► u    ← 重边(multi-edge):同一对节点多条边
    概念 定义 工程典型
    度 deg(u) u 邻居总数(无向) 微博粉丝数 = 入度
    入度 in-deg(u) 指向 u 的边数 PageRank 的"被多少人引用"
    出度 out-deg(u) u 指向别人的边数 "u 关注了多少人"
    自环 u→u 业务逻辑里的递归引用、SQL 自连接
    重边 同一对节点多条边 转账系统(同一对账户多笔交易)

    避坑:很多算法默认"简单图(无自环、无重边)"。NetworkX 的 Graph 不允许重边,要重边用 MultiGraph。处理交易/通信记录前先决定要不要保留重边——丢了就丢了关键信息。

    三、连通性:理解一个图的"大块"

    3.1 几个关键概念

    概念 定义
    路径 (path) 一系列节点 v₀ v₁ ... vₖ,相邻有边
    简单路径 不重复经过节点
    回路 / 环 (cycle) 起点等于终点的路径
    连通分量 (CC) 无向图里互相可达的极大节点集
    强连通分量 (SCC) 有向图里互相双向可达的极大节点集

    3.2 连通分量 vs 强连通

    无向图:
   A ─ B
        │
   C ─ D       两个 CC:{A,B,D},{C}? 不对,C 通过 D 跟 A,B 连
                       一个 CC:{A,B,C,D}

有向图:
   A → B
   ↑   │
   D ← C       一个 SCC(整个 A→B→C→D→A 是回路)


   A → B
       ↓
       C       三个 SCC:{A},{B,C},别的不双向

    3.3 Tarjan 算法核心思路(只点)

    核心直觉:DFS 走的时候,每个节点维护两个数:dfn[u] = 自己被访问的时间戳;low[u] = 在自己 DFS 子树里能回到的最早 dfn。如果 low[u] = dfn[u],u 就是一个 SCC 的"根"。

    不展开实现,记住一个事就够:Tarjan 一次 DFS 时间 O(V+E),是工程里唯一在用的 SCC 算法——Kosaraju(两次 DFS)和 Gabow 都没人用。

G = nx.DiGraph([("A","B"), ("B","C"), ("C","A"), ("C","D")])
list(nx.strongly_connected_components(G))
# [{'A', 'B', 'C'}, {'D'}]

    工程对应:

    • 微服务调用链找"循环依赖"
    • 编译器变量定义-使用找循环
    • 区块链账户图找洗钱回路

    四、BFS:层次遍历的瑞士军刀

    4.1 BFS 的形状

    起点 S,层层向外推:

         Layer 0:    [S]
                      │
         Layer 1:    [A B C]       ← S 的直接邻居
                     │  │  │
         Layer 2:   [D E F G H]   ← A、B、C 的邻居
                                    (去重)

    用一个 队列 (FIFO):出队一个节点,把它没访问过的邻居全入队。

    from collections import deque

def bfs(graph, start):
    visited = {start}
    queue = deque([(start, 0)])              # (节点, 层数)
    while queue:
        u, depth = queue.popleft()
        yield u, depth
        for v in graph[u]:
            if v not in visited:
                visited.add(v)
                queue.append((v, depth + 1))

    4.2 BFS 能干嘛

    用途 关键
    无权最短路 因为按层扩展,第一次访问 v 时的层数就是最短距离
    二分图判定 给 BFS 顺序染色 0/1/0/1,如果发现冲突说明有奇环
    最少步数 / 最少操作数 把"状态"当节点,BFS 第一次到目标即最优
    二度好友 / 三度好友 BFS 到 depth=2 / 3
    网格连通块 BFS 的简化版(grid 上四邻接)
    最小变换路径 LeetCode "单词接龙" 经典

    避坑:带权图不能用 BFS 做最短路——BFS 默认权重 = 1。哪怕权重都是正整数 2/3/5,也得用 Dijkstra。只有边权全等的图才能用 BFS 当最短路。

    4.3 二分图判定的 ASCII

    BFS 染色:S=红,邻居=蓝,邻居的邻居=红...

       S(红)
      / | \
   A  B  C    ← 全部 蓝
   |  |
   D  E       ← 全部 红

如果 BFS 中遇到"邻居跟我同色"的边 → 不是二分图

    工程应用:任务-资源分配、广告主-广告位匹配、左右合并/分裂检测。

    五、DFS:回溯与拓扑排序

    5.1 DFS 的形状

    DFS(S):
  访问 S → 任选未访问的邻居 v → DFS(v) → 回退 → 选下一个邻居

       S
       │
       A          ← 一条路走到底
      / \
     B   C
     │
     D            DFS 顺序: S A B D ... C ...

(BFS 是"层层扩散水波",DFS 是"一条路捅到底再回头")

    def dfs(graph, u, visited=None):
    if visited is None:
        visited = set()
    visited.add(u)
    yield u
    for v in graph[u]:
        if v not in visited:
            yield from dfs(graph, v, visited)

    5.2 拓扑排序(DAG 上)

    核心直觉:有向无环图(DAG)总能排成一列,使所有边都从左指向右。这就是拓扑序——npm/cargo/apt 找出"先装谁后装谁"的顺序。

    两种实现:

    算法 思路 实现
    Kahn 反复取入度 0 的节点 队列 + 入度数组
    DFS 后序倒序就是拓扑序 递归 + 栈

    工程意义:

    • npm 装包顺序:从无依赖的包开始装
    • Spark / Airflow DAG 调度:按拓扑顺序触发
    • Excel / 表格软件:列顺序更新公式
    • Make / Bazel 编译:目标文件先后顺序

    避坑:拓扑排序只对 DAG 有效。一旦有环,拓扑排序失败——这就是包管理报"circular dependency"的根本原因。检测环 = Kahn 算法剩下入度非 0 的节点 / DFS 找回边。

    import networkx as nx
G = nx.DiGraph([("a","b"), ("b","c"), ("a","c")])
list(nx.topological_sort(G))     # ['a', 'b', 'c']

G.add_edge("c","a")
list(nx.topological_sort(G))     # NetworkXUnfeasible: 有环

    5.3 桥与割点(网络韧性)

    概念 定义
    割点 (cut vertex) 删掉它图就不连通
    桥 (bridge) 删掉它图就不连通的边

    工程意义:

    • 网络拓扑找单点故障(数据中心交换机)
    • 知识图谱找"桥接"实体(连接两个领域的节点)
    • 社交网络的"信息桥"

    NetworkX 一行解决:nx.bridges(G) / nx.articulation_points(G),底层是 Tarjan 算法的 DFS 变种。

    5.4 BFS vs DFS 速查

    维度 BFS DFS
    数据结构 队列(FIFO) 栈(LIFO)/ 递归
    内存 O(最大层宽度) O(最深递归深度)
    找最短路(无权) 适合 不适合
    找所有路径 不适合 适合
    拓扑排序 Kahn 风格 后序倒序
    检测环 麻烦 找回边即可
    二分图判定 染色 也可以,但 BFS 更直观
    强连通分量 不适合 Tarjan / Kosaraju 都基于 DFS
    网格 / 树 一般 BFS 一般 DFS(简短)

    经验法则:问"最少步数 / 最短"用 BFS;问"找一个解 / 所有解 / 回溯 / 排序"用 DFS。

    六、最短路:四个算法各管一段

    算法 单源/全源 边权要求 复杂度 何时用
    BFS 单源 全等(=1) O(V+E) 无权图
    Dijkstra 单源 非负 O((V+E) log V) with heap 路由协议 OSPF、地图导航
    Bellman-Ford 单源 允许负权,能检测负环 O(VE) 路由协议 RIP、套利检测
    Floyd-Warshall 全源 任意(无负环) O(V³) 节点 < 几百的全源
    A* 单源 非负 + 启发式 取决于启发函数 游戏寻路、地图大图

    6.1 Dijkstra 的核心一步

    dist[起点] = 0;其他全 ∞
优先队列 (min-heap) 放 (dist, node)

每次:
  pop 当前 dist 最小的 u
  对 u 的每个邻居 v:
      if dist[u] + w(u,v) < dist[v]:
          dist[v] = dist[u] + w(u,v)
          push (dist[v], v)

    避坑:Dijkstra 不能处理负权——一旦有负权,"已经定下的最短距离"可能被后来发现的负边推翻,松弛过程会出错。负权用 Bellman-Ford。

    6.2 Bellman-Ford 与负环

    核心直觉:Bellman-Ford 做 V-1 轮"全图所有边松弛",如果第 V 轮还能松弛,说明有负环(无限刷负值)。

    工程对应:外汇套利检测——节点是货币,边权 = -log(汇率),负环就是"绕一圈能换出更多钱"的套利机会。

    6.3 Floyd-Warshall 的三层循环

    # dist[i][j] 初始化为边权,无边为 ∞
for k in range(V):                          # 中转点
    for i in range(V):                      # 起点
        for j in range(V):                  # 终点
            if dist[i][k] + dist[k][j] < dist[i][j]:
                dist[i][j] = dist[i][k] + dist[k][j]

    经验法则:节点 ≤ 500 用 Floyd-Warshall;节点更多用 V 次 Dijkstra(近似 V²·E·log V,稀疏图反而更快)。

    6.4 A* 启发式

    f(v) = g(v) + h(v)
       │       │
       │       └─ 启发函数:从 v 到终点的"估算代价"(必须 ≤ 真实代价)
       └─ 已走代价:从起点到 v 的真实距离

    核心直觉:Dijkstra 是 A* 在 h(v) ≡ 0 的特例——什么都不知道,只能"朝最近的方向均匀扩展"。给个像样的启发函数(地图寻路用直线距离),搜索空间能压缩好几个数量级。

    工程典型:

    • 游戏 NPC 寻路(曼哈顿距离 / 欧氏距离)
    • 谜题求解(15 数码、Rubik's Cube)
    • 大型地图导航(在 Dijkstra 上加距离启发)

    七、最小生成树(MST):花最少的钱连通所有点

    场景:N 个城市要修电网,给定 (城市i, 城市j, 修线成本),如何用最少总成本让所有城市连通。

    7.1 Kruskal vs Prim

    算法 思路 数据结构 复杂度 适合
    Kruskal 边按权排序,从小到大加,不形成环就加 并查集 (Union-Find) O(E log E) 稀疏图
    Prim 从一个点开始,贪心扩展最小边的邻居 优先队列 / 堆 O(E log V) 稠密图

    7.2 Kruskal 的并查集

    按边权排序:
  (1,2,1) (3,4,2) (1,3,3) (2,4,5) (1,4,7)

加 (1,2,1):  1-2          ✓
加 (3,4,2):  3-4          ✓
加 (1,3,3):  1-2-3-4 全连 ✓
加 (2,4,5):  会形成环     ✗ 跳过
加 (1,4,7):  会形成环     ✗ 跳过

MST = {(1,2), (3,4), (1,3)},总成本 = 6

    并查集做"u 和 v 是否已经连通"——find(u) == find(v) 就跳过。

    class DSU:
    def __init__(self, n):
        self.p = list(range(n))
    def find(self, x):
        while self.p[x] != x:
            self.p[x] = self.p[self.p[x]]   # 路径压缩
            x = self.p[x]
        return x
    def union(self, x, y):
        rx, ry = self.find(x), self.find(y)
        if rx == ry: return False
        self.p[rx] = ry
        return True

def kruskal(edges, n):
    edges.sort(key=lambda e: e[2])
    dsu = DSU(n)
    mst, total = [], 0
    for u, v, w in edges:
        if dsu.union(u, v):
            mst.append((u, v, w))
            total += w
    return mst, total

    • 网络拓扑设计(数据中心 spine-leaf 的最小成本)
    • 聚类(单链接层次聚类等价于 MST 切边)
    • Image Segmentation(基于图的分割算法)

    八、网络流(只点结论)

    8.1 最大流问题

    源 s → ... → 汇 t,每条边有"容量"
求:从 s 到 t 最多能"运"多少流量

    8.2 最大流 = 最小割

    核心定理(Max-Flow Min-Cut):任何网络的最大流值,等于把图分成两半(s 在一半、t 在另一半)的所有"割边"容量之和的最小值。

    这个定理是网络流的全部美感。工程意义:

    • 求最大流(物流、网络带宽)
    • 求"最薄弱环节"(最小割就是网络瓶颈所在)
    • 二分图最大匹配(转化成最大流)
    • 项目选择(收益最大化)

    8.3 Ford-Fulkerson 的 idea

    反复:
  在残差图里找一条 s → t 的"增广路径"
  用这条路径上最小容量增加流量
直到找不到增广路径为止

    不展开。工程上几乎不用手写,用 NetworkX maximum_flow 或 scipy.sparse.csgraph.maximum_flow——它们用的是 Edmonds-Karp / Dinic 等优化算法,对几万节点都够。

    import networkx as nx
G = nx.DiGraph()
G.add_edge("s","a", capacity=10)
G.add_edge("s","b", capacity=5)
G.add_edge("a","t", capacity=8)
G.add_edge("b","t", capacity=10)
flow_value, flow_dict = nx.maximum_flow(G, "s", "t")
print(flow_value)   # 13

    9.2 两种解法

    解法 复杂度 适用
    匈牙利算法(增广路) O(V·E) 经典,代码短
    转化为最大流 取决于流算法 二分图带权 / 多重 / 容量上限

    10.1 直接能用的库

    库 适合规模 强项
    NetworkX < 几十万节点 API 全、文档好,科研/原型首选

    10.2 选型决策树

    节点 < 1 万         → NetworkX(API 顺手)
节点 1 万~100 万    → igraph 或 networkit
节点 > 100 万 离线  → Spark GraphFrames
要图数据库 + 查询    → Neo4j
要 GNN              → PyG / DGL
和 numpy 强配合      → scipy.sparse.csgraph

    10.3 关键超参 / 设计选择

    决策 影响
    邻接矩阵 vs 邻接表 内存爆与不爆的分界
    稀疏矩阵格式 (CSR/CSC/COO) 行操作 vs 列操作 vs 增删
    Dijkstra 的堆 (binary / Fibonacci / pairing) 几十万节点起 Fibonacci 才有微弱优势,实战二叉堆够
    BFS 队列实现 (deque vs list) Python 用 collections.deque,list.pop(0) 是 O(n)
    并查集是否做路径压缩 + 按秩合并 不做就退化成 O(log n),做了就接近 O(α(n)) ≈ O(1)

    10.4 局限与现实

    • 图的"距离"语义复杂——加权 vs 跳数 vs 时延 vs 信任度,不是同一个最短路问题
    • 真实图都是动态的——边会增删,实时维护连通性 / 最短路 / 中心度都比静态版难得多(Dynamic Graph Algorithms 是单独一个领域)
    • 大图基本上跑不了精确算法——Twitter 的 PageRank、Google 的网页图都用近似/分布式版本
    • "六度分隔"不是数学定理是统计观察——Milgram 1967 实验、Facebook 2011 重做(平均距离 4.74),都是实证不是证明

    避坑:NetworkX 的算法对 100 万以上节点几乎都跑不动——不是因为算法慢,是 Python 对象开销 + 字典哈希成本。生产规模图分析必须切到 igraph / Spark / Neo4j,别在 NetworkX 上死磕优化。
