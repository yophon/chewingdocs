    近似、误差与渐进分析

    这一篇接 02 篇,继续在"数怎么算"这一层挖。02 讲的是"一个数怎么被存"——精度问题。这一篇讲"一段算法在多少步内、误差有多大"——近似问题。两者在工程里经常缠在一起。

    一句话先记住:工程里几乎所有数学都是近似的——大 O 隐藏常数、收敛速率、误差上界、概率保证。看不到近似的位置,你就在用"理论上 O(1)"的算法跑出 50ms 延迟,在用"理论上收敛"的优化器调出永不下降的 loss,在用"理论上误差很小"的近似算法做出错得离谱的报表。

    一、为什么必须先过这一篇

    1.1 工程里这层数学撞到哪儿

    场景 表面问题 底下的数学 本篇章节
    dict[k] 平均 O(1),偶尔 50ms 哈希分布不均 / rehash 摊还分析 / 期望复杂度 §3
    排序 1 亿个 float,std::sort vs np.sort 差 3 倍 缓存、SIMD、隐藏常数 大 O 的工程局限 §2
    训练 loss 后期不下降 学习率没降 / 二阶曲率 线性 vs 次线性收敛 §4
    牛顿法看似快,实际 LLM 训练全用 Adam 二阶 Hessian 太贵 收敛速率 × 单步代价 §4
    Bloom filter 设置错 误判率公式不会用 (1-1/m)^kn §6
    HyperLogLog 估计基数偏差 5% 期望误差 1.04/√m 概率上界 §6
    LSH 推荐没召回到目标 概率不命中 1-(1-p^k)^L §6
    算高维距离慢 不需要精确,需要 ε-近似 相对 vs 绝对误差 §7
    模型评测 metric 抖动 没区分相对 / 绝对误差 (1±ε)-近似 vs ε-近似 §7
    排序求和结果不稳定 浮点累加顺序敏感 数值稳定性 §5

    共同特征:这些 bug 都不是"算法写错了"——是"对算法的数学保证理解错了"。

    1.2 这一篇的目标

    读完你应该能回答:

    1. 看到 O(n log n) 算法,你知道隐藏常数大概会差多少倍
    2. 看到一个迭代算法,你知道它是线性还是二次收敛、对应代码该怎么停
    3. 看到一个"概率算法"(Bloom / HLL / LSH / CMS),你知道它的误差保证是什么形式
    4. 看到一个数值计算,你知道它有没有 catastrophic cancellation 风险

    二、大 O 重新审视:工程师眼里的渐进

    2.1 大 O 教科书定义和它的盲区

    f(n) = O(g(n))  ⇔  存在常数 C 和 n0,使得 n > n0 时 f(n) ≤ C·g(n)

    教科书定义里有三个工程灾难点:

    1. 常数 C 被隐藏:O(n) 的常数可能是 10000,O(n²) 的常数可能是 1。n < 100 时 O(n²) 完胜 O(n)。
    2. n0 被隐藏:有些算法 n > 10000 才显出优势,工程数据集 n=200 永远跑不到那个临界。
    3. 常数操作的代价不一:一次缓存命中 1ns,一次缓存未命中 100ns,一次分支预测失败 10ns——大 O 全当 1 个 unit。

    2.2 真实数据对比:O(n²) 不总是输

    下面这个对照表是写汇编 / SIMD / 算法库的人天天活在里面的常识——做应用层的人见过就好,不必内化:

    n 朴素 O(n²) 排序(cycles) 标准库 O(n log n)(cycles) 谁赢
    8 ~64 ~400 朴素胜
    32 ~1000 ~3000 朴素胜
    64 ~4000 ~7000 平
    128 ~16000 ~16000 平
    256 ~65000 ~36000 标准库胜
    4096 ~1.6e7 ~8e5 标准库胜 20×

    std::sort 实际是混合算法(introsort / pdqsort)——大块用 quicksort,小块切到插入排序就是这个原因。Python list.sort 用 Timsort,Java 也是,起步时对小数组用插入排序。

    经验法则:n < 50 时,选什么数据结构 / 算法基本无所谓——常数差异远大于渐进差异。优化精力放在 hot path 上、放在数据量大的地方。

    2.3 缓存与分支预测:大 O 看不到的两个维度

    读一个 L1 cache 数据:          ~1 ns
读一个 L2 cache 数据:          ~3 ns
读一个 L3 cache 数据:          ~12 ns
读一个主内存数据:              ~100 ns       ← 比 L1 慢 100 倍
分支预测正确:                  ~0.5 ns
分支预测错误:                  ~10-20 ns
SIMD 一条指令处理 8 个 float:  ~1 ns

    这意味着:

    • 顺序访问数组(缓存友好)比"理论上更优的"链表 / 树 / 哈希 快 5-50 倍——n=10000 时链表的 O(n) 比数组的 O(n) 慢 10 倍。
    • 分支可预测的代码比 branchless 还快;分支不可预测的代码再简单也慢(Mike Acton 的演讲讲过)。
    • SIMD 一次处理 8 个数,本质给你常数加速 8×——大 O 没变,但工程上是不是装了 -O3 / AVX 决定性能。

    避坑:别只看大 O 选数据结构。std::vector 几乎在所有场景都比 std::list 快——缓存就这么强势。Python 里同理 list 通常比 deque 快(除非两端都要 pop)。

    2.4 最坏 / 平均 / 期望:三个完全不同的承诺

    复杂度 含义 谁在用
    最坏(worst-case) 任何输入下都不超过 实时系统 / 安全(防 DoS)
    平均(average-case) 输入服从某个分布下的期望 通用算法分析
    期望(expected) 算法内部用了随机化,对所有输入都给概率保证 哈希 / 快排 / 跳表 / 蓄水池采样
    摊还(amortized) 一系列操作的总代价 / 操作数 动态数组 / 并查集 / 哈希表 rehash

    这四个区别,工程上最常被混的是平均 vs 期望。

    • 快速排序"平均 O(n log n)"——这是假设输入是随机的。如果输入是已排序的,标准快排退化到 O(n²)。
    • 随机化快排"期望 O(n log n)"——算法内部洗牌或随机选 pivot。对任何输入都期望 O(n log n),不依赖输入分布。

    经验法则:面对 untrusted 输入(网络、用户上传),用期望复杂度的算法。用户能控制输入,就能构造最坏输入打垮你——这是 HashDoS 攻击的本质(详见 25 篇)。

    2.5 摊还分析:动态数组与并查集

    动态数组的 append 大部分时候是 O(1),但满了要扩容、复制整个数组 O(n)。最坏情况 O(n),但 n 次操作总代价 O(n),摊到每次是 O(1) 摊还。

    容量 1 → 2 → 4 → 8 → 16 → ... → 2^k

往一个空数组 push n 个元素:
  扩容总代价 = 1 + 2 + 4 + ... + n/2 + n = 2n - 1
  push 本身代价 = n
  总代价 = 3n - 1
  平均每次 = O(1) ✓

    核心直觉:"指数增长容量"是为了把扩容代价摊到 O(1)。Python list、Java ArrayList、Go slice 都这么做——增长因子通常 1.5 或 2。写自己的 buffer 别用 +1 扩容,会变 O(n²)。

    三、期望复杂度的两个经典例子

    3.1 哈希表 O(1) 的精细前提

    d = {}
d[k] = v    # 平均 O(1)
v = d[k]    # 平均 O(1)

    O(1) 成立的前提:

    1. 哈希函数把 key 均匀分布到 bucket
    2. 装载因子(load factor)有界——通常 < 0.75
    3. bucket 内冲突 list 长度有界——理论上期望 O(1+α)

    这三个条件全失效时:

    # 攻击场景:HashDoS
# 攻击者构造 N 个 hash 碰撞的 key,全落到一个 bucket
# 一次查找退化到 O(N)

    Python / Java / .NET 都改过哈希实现(2003 年 Crosby & Wallach 攻击之后):hash 加入随机 seed,或链表退化到红黑树(Java 8 的 HashMap)。

    避坑:字典 / 集合在 untrusted 数据上,要么用 randomized hash(Python 3.3+ 默认开),要么用 BTreeMap / TreeSet 保证最坏 O(log n)。

    3.2 快速排序 O(n log n) 的精细前提

    朴素快排:pivot 选第一个元素 → 已排序输入退化到 O(n²)。

    随机化快排:pivot 随机选 → 期望 O(n log n),对任何输入。

    import random

def quicksort(a):
    if len(a) <= 1:
        return a
    pivot = random.choice(a)           # ← 随机化关键
    lo = [x for x in a if x < pivot]
    eq = [x for x in a if x == pivot]
    hi = [x for x in a if x > pivot]
    return quicksort(lo) + eq + quicksort(hi)

    期望分析直觉:每次划分,期望 pivot 落在中间 50% → 树深期望 O(log n) → 总代价 O(n log n)。最坏情况(每次 pivot 都是最小或最大)概率 (n!/n^n) 极小,n=20 已经几乎不可能撞到。

    标准库 std::sort 不止随机化——是 introsort:

    快排深度 > 2·log n 时切到堆排序,保证最坏 O(n log n)
切到 < 16 元素时切到插入排序(缓存友好)

    核心直觉:期望复杂度不是"平均",是"算法内部抛骰子保证的"。同一份输入跑两次,运行时长会略不同——这是期望的代价。

    四、收敛速率:迭代算法的"快慢"

    ML 训练、PageRank、Power Method、牛顿法解方程——全是迭代算法,它们的"快慢"用收敛速率(convergence rate)描述。

    4.1 三档收敛速率

    设 e_k = |x_k - x*| 是第 k 步误差。

    速率 公式 直觉 典型
    次线性(sublinear) e_k ≤ C / k^p "慢慢逼近,越接近越慢" SGD 非强凸目标
    线性(linear) e_(k+1) ≤ ρ · e_k,0 < ρ < 1 "每步固定缩小一个比例" GD 强凸目标
    二次(quadratic) e_(k+1) ≤ C · e_k² "已经接近时,精度每步翻倍位数" 牛顿法靠近最优时

    4.2 一张图理解三档差异

    下面这张表对照"要把误差从 0.1 降到 1e-10 需要多少步":

    目标:e_k < 1e-10,初始 e_0 = 0.1

次线性 (e_k = 1/k):       需要 1e10 步         (难以接受)
次线性 (e_k = 1/√k):     需要 1e20 步         (彻底没救)
线性 (ρ=0.9):              ln(1e9)/ln(0.9) ≈ 197 步
线性 (ρ=0.5):              ln(1e9)/ln(0.5) ≈ 30 步
二次:                      0.1 → 0.01 → 1e-4 → 1e-8 → 1e-16,4 步

    核心直觉:线性收敛已经够工程用了,二次收敛是"几乎免费"。但牛顿法每步代价 O(n³)(算 Hessian 加逆)——单步贵到只有小问题才值得。

    4.3 GD vs Newton 的工程权衡

    梯度下降 (GD):
  x_(k+1) = x_k - α · ∇f(x_k)
  单步代价: O(n)        ← n 是参数数
  收敛速率: 线性(强凸时)
  总代价(线性): O(n · log(1/ε))

牛顿法:
  x_(k+1) = x_k - H^(-1)(x_k) · ∇f(x_k)
  单步代价: O(n³)       ← 算 H 加解线性方程
  收敛速率: 二次(靠近最优时)
  总代价: O(n³ · log log(1/ε))

    LLM 训练 n = 7e9 参数:牛顿法的 O(n³) = 3.4e29——根本不可能跑。所以哪怕牛顿法"理论上更快收敛",实际工程从来不用。

    Adam 做的折中:用历史梯度的二阶矩(diagonal,不是完整 Hessian)做自适应学习率——比 GD 多一点二阶信息,但单步代价仍是 O(n)。

    优化器 用了几阶信息 单步代价 收敛速率 适合
    SGD 一阶 O(n) 次线性(非凸) 大模型
    Momentum 一阶+动量 O(n) 线性(凸) CV 大模型
    Adam / AdamW 一阶+对角二阶 O(n) 实际接近线性 NLP / LLM 主流
    L-BFGS 近似二阶 O(mn),m=10-20 超线性 小问题 / 经典优化
    Newton 完整二阶 O(n³) 二次 仅小问题

    经验法则:深度学习的"收敛速率"被三件事抢戏:非凸目标(线性收敛无法保证)、SGD 随机噪声(只能到 noise floor 附近)、调度策略(warmup + cosine 比纯收敛速率更影响实际曲线)。

    4.4 迭代算法的"什么时候停"

    每个迭代算法都需要停止判据:

    判据 公式 何时用
    绝对误差 \|x_k - x_(k-1)\| < ε x 量级稳定时
    相对误差 \|x_k - x_(k-1)\| / \|x_k\| < ε x 量级跨多个数量级
    梯度小 \|∇f(x_k)\| < ε 优化(凸时等价误差小)
    目标值变化 \|f(x_k) - f(x_(k-1))\| < ε 不容易算梯度
    最大迭代数 k > K_max 兜底,防 infinite loop 
    # 标准模板
for k in range(max_iter):
    x_new = update(x)
    if abs(x_new - x) < tol * max(abs(x_new), 1.0):   # 相对+绝对混合
        break
    x = x_new
else:
    print(f"warning: 未在 {max_iter} 步内收敛")

    避坑:纯相对误差在 x 接近 0 时除 0;纯绝对误差在 x 很大时永远满足不了。混合判据 |Δ| < tol·max(|x|, 1) 是工程默认。

    五、数值稳定性:catastrophic cancellation

    02 篇讲过浮点的"精度从哪来",这一节讲算法层面怎么把精度搞丢——同样的数学公式,换个写法误差差几个数量级。

    5.1 (a - b) 的灾难

    import numpy as np

# 解二次方程 ax² + bx + c = 0
# 标准公式:x = (-b ± √(b²-4ac)) / 2a
a, b, c = 1, 1e8, 1

# 算 x1 = (-b - √(b²-4ac)) / 2a:大数 - 大数,这一根 OK
# 算 x2 = (-b + √(b²-4ac)) / 2a:两个接近的大数相减,灾难

disc = np.sqrt(b**2 - 4*a*c)
x1 = (-b - disc) / (2*a)
x2 = (-b + disc) / (2*a)
print(x1, x2)   # -1e8, -7.45e-9    ← x2 实际应该是 -1e-8

# 数值稳定的写法:用 Vieta 公式 x1*x2 = c/a
x2_stable = c / (a * x1)
print(x2_stable)   # -1e-8 ✓

    根因:b² - 4ac 算出来 ≈ b²,开方 ≈ b,-b + b = 两个 1e8 量级数相减,15 位有效数字基本全抵消,只剩噪声。

    5.2 排序求和:顺序敏感

    import numpy as np

# 1 个 1e8 + 1 亿个 1e-8,数学上结果是 2
x = np.array([1e8] + [1e-8] * int(1e8))

# 朴素累加:从左往右
s = 0.0
for v in x:
    s += v
# s ≈ 1e8     ← 1e-8 加到 1e8 上完全消失

# Kahan 求和(补偿求和)
def kahan(x):
    s = 0.0
    c = 0.0   # compensation
    for v in x:
        y = v - c
        t = s + y
        c = (t - s) - y
        s = t
    return s
# kahan(x) ≈ 2.0  ✓

# 配对求和(NumPy 用的)
np.sum(x)   # 接近 2.0,因为 NumPy 用 pairwise summation

    三种求和的精度:

    算法 误差量级 何时用
    朴素 left-fold O(n·ε) Python 内置 sum
    配对求和 O(log n·ε) NumPy np.sum
    Kahan 求和 O(ε)(几乎与 n 无关) 需要高精度时

    核心直觉:朴素求和的误差随 n 线性涨。n = 1e8 时,FP32 的 ε ≈ 1e-7,累计误差就是 10——和真实值同量级了。NumPy 默认 pairwise 是好心,但你写循环时不要假设求和是精确的。

    5.3 方差的两种公式

    # 公式 A:E[X²] - E[X]²
def var_naive(x):
    return np.mean(x**2) - np.mean(x)**2

# 公式 B:E[(X - μ)²]
def var_stable(x):
    mu = x.mean()
    return np.mean((x - mu)**2)

# 公式 C:Welford 在线算法(单次扫描)
def var_welford(x):
    n, mean, M2 = 0, 0.0, 0.0
    for v in x:
        n += 1
        delta = v - mean
        mean += delta / n
        M2 += delta * (v - mean)
    return M2 / n

x = np.array([1e9, 1e9 + 1, 1e9 + 2])
var_naive(x)    # 负数(因为 catastrophic cancellation)
var_stable(x)   # ~0.667
var_welford(x)  # ~0.667 ✓

    避坑:任何"两个统计量相减"的写法都先想想稳定性。NumPy 的 np.var 内部用稳定算法,但你手写 np.mean(x**2) - np.mean(x)**2 就完蛋。

    5.4 一个稳定性自检表

    不稳定写法 稳定写法 风险等级
    log(sum(exp(x))) logsumexp(x) 高
    log(softmax(x)) log_softmax(x) 高
    (a - b) 两个接近的大数 重排公式 / 共轭乘法 高
    mean(x²) - mean(x)² mean((x - mean(x))²) 高
    sum(small_numbers) 朴素累加 Kahan / pairwise 中
    sqrt(a² + b²) np.hypot(a, b) 中
    1 - exp(x) 当 x ≈ 0 -np.expm1(x) 中
    log(1 + x) 当 x ≈ 0 np.log1p(x) 中

    六、近似算法:用数学保证换性能

    所有"O(1) 估算 1 亿规模"的算法都在卖一个产品:误差概率换性能。这一节是这一篇的"高密度"区——四个工业级近似数据结构的数学保证。

    6.1 Bloom Filter:存在性查询

    问题:判断一个元素是否在集合里,只需要返回 "可能在" / "一定不在"。允许误判,绝不漏判。

    算法:m 位 bit 数组,k 个独立 hash 函数。add(x) 把 k 个位置 1,query(x) 检查 k 个位置是不是全是 1。

    插入 n 个元素后,某一位仍是 0 的概率:
  p_0 = (1 - 1/m)^(kn) ≈ e^(-kn/m)

某一位是 1 的概率:
  p_1 = 1 - e^(-kn/m)

误判率(查不存在的元素时,k 个位置全是 1):
  P_fp ≈ (1 - e^(-kn/m))^k

    给定 n 和目标误判率 f,最优参数:

    m = -(n · ln f) / (ln 2)²    ← 位数
k = (m / n) · ln 2           ← hash 函数数
    n f m(bits) m(MB) k
    1M 1% 9.6 M 1.2 7
    1M 0.1% 14.4 M 1.8 10
    100M 1% 960 M 120 7
    1B 1% 9.6 G 1200 7

    经验法则:Bloom filter 是"用 9.6 bits/element 换 1% 误判率"。比起直接存 hash set(每元素 ~50 bytes),省 40 倍。

    工程对应:pybloom-live(Python)、Guava BloomFilter(Java)、bloomfilter crate(Rust)、Redis 的 BF.ADD 命令。

    6.2 HyperLogLog:基数估计

    问题:集合里有多少个不同元素(基数),给定大数据流,内存有限。

    算法:把 hash 值的二进制前缀里"前导零个数"记下来,用调和平均估计基数。关键公式:

    误差期望:1.04 / √m,其中 m 是寄存器个数

m = 2048(典型):误差 ~2.3%
m = 16384:      误差 ~0.8%

每个寄存器 5 bits 够用,总内存 m × 5 bits
    m 内存 期望误差 能估到多少基数
    1024 640 B 3.25% ~1e10
    16384 10 KB 0.81% ~1e10
    65536 40 KB 0.41% ~1e10

    核心直觉:HLL 用 10 KB 估计 100 亿规模基数,误差 < 1%。比起 set()(每元素 50 bytes,100 亿要 500 GB),省 5000 万倍。

    工程对应:Redis 的 PFCOUNT 用 12 KB 估到 0.81% 误差;Google BigQuery、ClickHouse、Spark 都内置。Counts Distinct 在大数据里基本都是 HLL。

    6.3 Count-Min Sketch:频率估计

    问题:估计每个元素在数据流里出现了多少次,只要近似。

    算法:d × w 的二维计数矩阵,d 个独立 hash 函数。add(x) 把 d 个位置 +1,query(x) 返回 d 个位置的最小值。

    保证:对任意元素,估计值 f̂ 满足
  P[f̂ > f + ε·N] ≤ δ
  其中 N = 总流量,ε = w 决定,δ = d 决定

参数选择:
  w = ⌈e / ε⌉        ← e 是欧拉数 2.718
  d = ⌈ln(1/δ)⌉      ← 多少行 hash
    想要 ε 想要 δ w d 内存
    0.01 0.01 272 5 5.3 KB
    0.001 0.001 2719 7 76 KB
    0.0001 0.0001 27183 10 1.1 MB

    避坑:CMS 永远高估,从不低估——所以叫"min" 取多个 hash 的最小值。做"top-K 热点"很合适(Heavy Hitters);做"准确计数"完全不行——这是设计就这样。

    工程对应:Apache Spark MLLib 的 CountMinSketch、Twitter Algebird、ClickHouse 的 topK()。做 trending topics、热门搜索、热点 IP 全是它。

    6.4 LSH(Locality-Sensitive Hashing):近似近邻

    问题:找一个查询向量的近邻,在数十亿规模向量里。精确最近邻是 O(N) 比对,LSH 是次线性。

    核心定义:对距离近的两个点,LSH 函数大概率把它们映射到同一个桶;对距离远的两个点,大概率映射到不同桶。

    对欧氏距离的 LSH(p-stable):
  h(x) = ⌊(a · x + b) / r⌋
       a 是随机高斯向量,b ∈ [0, r) 均匀,r 是桶宽

  P[h(x) = h(y)] = p(||x-y||)    ← 距离越远越小

构造 (L, K) LSH:
  - K 个 hash 函数串成一个 hash 签名 → 签名一致才算"碰撞"
  - L 个独立签名 → 任一签名碰撞就算候选

召回:1 - (1 - p^K)^L     ← 这是 LSH 调参的核心公式

    这条公式背后的工程含义:

    • K 大 → 单个签名精确,但召回率掉
    • L 大 → 召回率高,但内存和计算贵
    • 工程上通常先固定召回目标(比如 90%),反解 (K, L)

    经验法则:LSH 在 d > 50 维、N > 1M 时显著快于 KD-Tree。但 ANN 现在大多用 HNSW(图索引),LSH 在工程里份额下降——Faiss / Milvus / pgvector 都默认 HNSW。

    工程对应:datasketch.MinHashLSH(Python)、Faiss 的 IndexLSH、Annoy 的随机投影树(LSH 变体)、Milvus / Pinecone(向量数据库主要用 HNSW + IVF,LSH 是备选)。

    6.5 四个近似算法的对照

    算法 解决问题 误差保证 单点查询 工程主场
    Bloom Filter 存在性 误判率 f,无漏判 O(k) 缓存穿透 / 黑名单
    HyperLogLog 基数 期望误差 1.04/√m O(1) UV 统计
    Count-Min Sketch 频率 高估 ε·N 概率 δ O(d) 热点 / TopK
    LSH 近邻 召回率公式 次线性 向量检索(逐步被 HNSW 替代)

    核心直觉:这四个算法的数学共性,是"用概率换内存 / 时间"。它们都给你一个"误差上界 + 失败概率"的双参数承诺——而精确算法只承诺正确但要付全部代价。工程上 99% 场景"差 1% 不算事",这就是它们能取代精确算法的根本。

    七、相对误差 vs 绝对误差;ε-近似的工程语义

    最后这一节短但密度高——所有近似算法的论文摘要里都有它,看不懂就找不到能用的算法。

    7.1 两种误差的定义

    真实值 x,估计值 x̂

绝对误差:|x̂ - x|
相对误差:|x̂ - x| / |x|       ← 当 x ≠ 0

    什么时候用哪个:

    场景 用绝对 用相对
    x 量级稳定(比如概率 [0,1]) ✓ —
    x 跨数量级(从 1e-6 到 1e6) — ✓
    算到 0 附近(相对误差爆) ✓ —
    工程"对大值容错" — ✓(允许大值小数位不准)

    实际经验:几乎所有 ML metric 用绝对误差(loss、accuracy 都是 [0,1] 量级);几乎所有数值计算用相对误差(求根、解方程、积分)。

    7.2 ε-近似 vs (1±ε)-近似:论文里的固定术语

    这两个术语在不同论文里有时混用,但工程上严格区分:

    ε-近似(absolute):
  |x̂ - x| ≤ ε

(1±ε)-近似(relative,multiplicative):
  (1 - ε) · x ≤ x̂ ≤ (1 + ε) · x

ε-additive 近似:
  x̂ ≤ x + ε · N         ← Count-Min Sketch 用这种

ε-multiplicative 近似:
  x̂ ∈ [(1-ε)x, (1+ε)x]  ← HyperLogLog 用这种

    举例:

    • HyperLogLog 给的是 (1±ε) 近似:基数估计 1000 和 100000 都给 1% 相对误差——这才合理,1% 的 1000 是 10,1% 的 100000 是 1000。
    • Count-Min Sketch 给的是 ε-additive 近似:估计值高出真实值不超过 ε·N——所以对 hot key 准,对 cold key 噪声相对很大。

    避坑:看论文摘要时分清这两种。"ε-近似"在数据库领域通常指 additive,在 ML / 数值领域通常指 multiplicative——上下文决定。

    7.3 PAC 框架:Probably Approximately Correct

    很多近似算法的承诺写成 PAC 形式:

    P[ |x̂ - x| > ε ] < δ

中文:估计值偏离真实值超过 ε 的概率小于 δ

    两参数双独立:ε 控制误差大小(精度),δ 控制误差概率(可信度)。两个都需要给——只说"误差很小"是不够的,"99% 的情况下误差小于 1%"才是工程合同。

    典型工程参数:

    场景 ε(精度) δ(失败概率)
    用户行为统计 1-5% 1%
    实时风控指标 0.1-1% 0.1%
    ML 模型评测 0.1-0.5% 5%(置信区间)
    A/B 测试效应估计 由功效公式定 通常 5%
    加密签名验证(零误差) 不适用 PAC 必须 0

    7.4 工程对应:相对 / 绝对的 API

    import numpy as np
import torch
import math

# Python 内置
math.isclose(a, b, rel_tol=1e-9, abs_tol=0.0)   # 默认相对

# NumPy
np.isclose(a, b, rtol=1e-5, atol=1e-8)          # 混合:|a-b| ≤ atol + rtol·|b|
np.allclose(A, B, rtol=1e-5, atol=1e-8)         # 整个 tensor

# PyTorch
torch.isclose(a, b, rtol=1e-5, atol=1e-8)
torch.allclose(A, B, rtol=1e-5, atol=1e-8)

# 单元测试常用
np.testing.assert_allclose(actual, desired, rtol=1e-5)
torch.testing.assert_close(actual, desired, rtol=1e-5, atol=1e-8)

    默认 rtol / atol 选择:

    FP64 数学等价检测:   rtol=1e-13, atol=1e-15
FP32 数学等价检测:   rtol=1e-6,  atol=1e-7
FP16/BF16 检测:      rtol=1e-3,  atol=1e-3
模型推理两次一致:    rtol=1e-3,  atol=1e-5
不同硬件/框架对照:    rtol=1e-2,  atol=1e-3   ← 不同 GPU 的 GEMM 结果不完全一样

    经验法则:测试两个浮点 tensor 是否相等,永远用 assert_close + 显式 rtol / atol——== 几乎一定错,np.array_equal 也是错的(它要求二进制完全一致)。
