# 假设检验与 A/B 测试

「这个实验 p<0.05,显著!上线!」——这是工程团队最常见、也是最致命的统计错误现场。**p<0.05 不等于"B 比 A 好的概率 95%",不等于"上线一定有效",不等于"实验跑够了"**。这一篇专门拆开假设检验,讲清楚 p 值真正含义、α/β 取舍、为什么不能 Peek、多重比较怎么救、什么时候改用贝叶斯——**所有"上线决策"背后的统计语言,以及它们最常被误用的地方**。

> 一句话先记住:**假设检验是"反证法的统计版本"——假设没差异,看观测到的差异有多反常**。p 值回答的是"假设 H0 是真的,看到这么极端结果的概率",**不是 H0 是真的的概率**。把这句话刻进脑子,90% 的统计误用就能避开。

---

## 一、为什么必须懂这套东西

工程师不需要会推 t 检验的公式,但**几乎每个迭代都涉及"这次改动有没有效"的决策**:

| 场景 | 不懂时的错 | 懂了之后 |
| --- | --- | --- |
| 改了推荐算法,CTR +1% | "上线!" | 没算样本量,可能在噪声里 |
| 一个 A/B 跑两天,p=0.04 | "已经显著了,停!" | Peeking 把 α 实际推到 0.15+ |
| 同时实验 20 个变种 | "至少一个显著就上" | Bonferroni 没做,纯巧合 |
| 实验组 +10%,看起来牛 | "上线大改" | Twyman's law:太显著先查数据 |
| 想"小流量先试" | "5% 流量看几天" | 样本不够,p 值随机抖 |
| 经理问"B 比 A 好的概率多少" | "p<0.05 所以 95%" | 这是错的;贝叶斯 A/B 才能回答 |
| 老板说"必须 +5% 才上" | "我也想知道要跑多久" | 这是 MDE,决定样本量 |
| 实验失败的指标"和大盘对齐" | "看起来没差就行" | 没算 power,β 高,有效果也看不出来 |

> **核心痛点**:**假设检验是为"实验科学"设计的——一次实验、独立样本、事先决定的样本量、单一假设**。互联网公司的 A/B 测试**几乎条条违反**(连续看、多臂、用户重复、网络效应)。**直接套教科书公式必翻车**,要么白上线,要么错过有效改动。

## 二、假设检验心智:反证法的统计版本

### 2.1 套路

```
1. 零假设 H0    :B 和 A 没差(默认假设)
2. 备择假设 H1  :B 比 A 好(我想证明的)
3. 设计实验,收集数据
4. 计算"假如 H0 是真的,看到这么极端或更极端结果的概率"= p 值
5. p < α(预设阈值,常用 0.05)→ 拒绝 H0,接受 H1
   p ≥ α                     → 没足够证据拒绝 H0(不是"H0 真")
```

**反证法的统计版本**——你不能"直接证明 B 比 A 好",只能"先假设 B 和 A 一样(H0),如果数据让这个假设站不住,就接受 B 比 A 好"。

### 2.2 不要走反方向

| 数据结果 | 错误的话术 | 正确的话术 |
| --- | --- | --- |
| p < 0.05 | "证明了 B 比 A 好" | "有证据反驳'B 和 A 没差',支持 B 比 A 好" |
| p > 0.05 | "证明了 B 和 A 一样" | "没足够证据说 B 和 A 有差(可能样本不够)" |
| p = 0.04 | "B 比 A 好的概率 96%" | "假设没差,看到这种结果的概率 4%" |

**p > 0.05 不是"无差异",是"还不知道有没有差异"**——可能差异真的小,也可能样本量没到,也可能你的检验没 power。**接受 H0 ≠ 证明 H0**,这是统计入门第一坑。

> **核心直觉**:**假设检验只有两种结论——"拒绝 H0" 或 "暂时不拒绝 H0"**。它**永远不证明 H0**。想证明"B 等于 A",得用等价性检验(equivalence testing,TOST)。

## 三、p 值的真正含义(把这一节读三遍)

### 3.1 定义

```
p 值 = P(观测到这么极端或更极端结果 | H0 为真)
       └────────────────────────────┘   └──────┘
              事件                       条件
```

**注意条件是 H0**,不是别的。

### 3.2 五种常见误读 vs 正确解释

| 误读 | 真实含义 |
| --- | --- |
| p = 0.04 → H0 是假的概率 96% | 错。p 值假设 H0 是真,不算 H0 是否为真 |
| p = 0.04 → B 比 A 好的概率 96% | 错。要这个数得做贝叶斯,有先验 |
| p > 0.05 → H0 是真的 | 错。只是没足够证据反驳它 |
| 重复实验,有 95% 概率得到同样结论 | 错。这是 power,不是 p 值 |
| p 值越小,效果越大 | 错。p 小可能因为效果大,也可能因为样本大 |

**为什么"p<0.05 = 95% 概率 H1 真"是错的?**

这两个概率方向反着:

```
p 值算的:     P(数据 | H0)
你想要的:     P(H1 | 数据)

要从前者得到后者,贝叶斯公式:
P(H1 | data) = P(data | H1) · P(H1) / P(data)
                                ↑
                                先验! p 值里没有这玩意儿
```

**要"H1 是真的概率",必须**:
1. 算 P(data | H1)(不只是 H0)
2. 给 H1 一个先验概率
3. 用贝叶斯公式

这就是贝叶斯 A/B 测试干的事(下一节)。**只有 p 值,没法回答"B 比 A 好的概率多少"**。

### 3.3 p 值会被这些事玩坏

```
样本量越大 → p 越容易 < 0.05(即使效果极小也"显著")
数据相关  → p 值无效(独立性假设没满足)
多次比较  → 总有一个 p < 0.05(纯巧合)
看一眼就停 → 实际 α 比标称大几倍(Peeking)
事后选指标 → 在 N 个指标里挑显著的(p-hacking)
```

> **避坑**:**任何"显著"结果,先问四件事**:
> 1. 样本量是不是大到任何小差异都"显著"?(效应量也得看)
> 2. 是不是 peek 出来的?(序贯检验校正没?)
> 3. 同时测了多少个指标 / 多少个变种?(多重比较校正没?)
> 4. 这个差异在业务上有意义吗?(0.001% 提升再显著也不值得上)

## 四、第一类、第二类错误与检验功效

### 4.1 四象限

```
                      真实世界
                  H0 真         H0 假(H1 真)
              ┌────────────┬───────────────┐
   决策    H0 │  ✓ 正确    │  Type II (β)  │
        不拒绝│            │  漏报         │
              ├────────────┼───────────────┤
        拒绝 │ Type I (α) │   ✓ 正确      │
        H0   │ 误报       │   (功效 1-β)  │
              └────────────┴───────────────┘
```

| 概念 | 工程语言 | 默认阈值 | 谁来定 |
| --- | --- | --- | --- |
| α(Type I 错误率) | 假阳/误报率 | 0.05 | 实验科学家(怕白上线) |
| β(Type II 错误率) | 漏报率 | 0.20 | 业务方(怕错过好功能) |
| Power = 1 - β | 检测出真效果的概率 | 0.80 | 决定样本量 |

### 4.2 α 和 β 的取舍:工程上"宁可错杀还是宁可放过"

α 和 β **此消彼长**——把 α 调小(0.01),误报变少,但漏报变多;反过来一样。

```
α 严格(0.001) → 决策更保守                β 松(0.5) → 漏报很多
                                                   ↓
   ┌── 实验室科研:               ┌── 互联网产品迭代:
   │   假阳=发错论文,代价巨大     │   假阳=回滚就行
   │   宁可错杀                   │   假阴=错过好改动,损失更大
   │   α = 0.001 / 0.01           │   α = 0.05 / 0.1 可以接受
   │                              │   β 反而要严(Power ≥ 0.80)
   └──────────────────────────────└──────────────────────────────
```

> **经验法则**:**互联网 A/B 测试默认 α=0.05、Power=0.80**(对应 β=0.20)。**生死攸关场景(药品、安全)α 调到 0.001**,因为假阳代价无穷大。**Power 不是 1 而是 0.80,意思是"如果效果真的存在,80% 的概率检测出来"——剩下 20% 是漏报**。

### 4.3 α 与 β 的可视化

```
              H0 分布(B 和 A 没差)         H1 分布(B 真的好 d 个单位)
              ▁▂▄▆██▆▄▂▁                   ▁▂▄▆██▆▄▂▁
            ▁▃▆█    █▆▃▁                ▁▃▆█    █▆▃▁
         ─────────────┬──────────────────────────────────
         0                  ↑                  d
                       临界值 c
                       (拒绝 H0 的阈值)

         ◄──── 不拒绝 H0 ────►  ◄──── 拒绝 H0 ────►

         H0 真时落右边的概率 = α(误报)
         H1 真时落左边的概率 = β(漏报)
         H1 真时落右边的概率 = 1 - β(检测出来了,Power)
```

把 c 往右移 → α 变小、β 变大(更难拒绝)
把 c 往左移 → α 变大、β 变小(更容易拒绝)

**唯一能同时降低 α 和 β 的办法:增加样本量**——两个分布会变窄、重叠减小。

## 五、常见检验:各用在哪

| 检验 | 用在哪 | 数据假设 | scipy 函数 |
| --- | --- | --- | --- |
| **t 检验**(独立两样本) | 比较两组均值 | 近似高斯,等方差 | `ttest_ind` |
| **Welch t 检验** | 两组方差不等 | 近似高斯 | `ttest_ind(..., equal_var=False)` |
| **配对 t 检验** | 同一组前后对比 | 差值近似高斯 | `ttest_rel` |
| **Mann-Whitney U** | 两组中位数比较 | 任意分布 | `mannwhitneyu` |
| **Wilcoxon 符号秩** | 配对版本的非参数 | 任意分布 | `wilcoxon` |
| **卡方检验** | 分类频次差异 | 期望频数 ≥5 | `chi2_contingency` |
| **Fisher 精确检验** | 卡方的小样本版本 | 适用 2×2 表 | `fisher_exact` |
| **KS 检验** | 整个分布形状差异 | 任意 | `ks_2samp` |
| **置换检验** | 任何统计量 | 任意 | 自己写 / `scipy.stats.permutation_test` |
| **ANOVA** | 多组均值比较 | 高斯,等方差 | `f_oneway` |

**工程选用决策树**:

```
比较两组指标
├─ 指标是连续型(延迟、营收、CTR×千次)
│  ├─ 样本量大(N>30 per group)→ Welch t 检验(不假设等方差,稳)
│  ├─ 样本量小且分布偏斜       → Mann-Whitney U(非参数)
│  └─ 想看整个分布而不仅均值   → KS 检验 / 置换检验
├─ 指标是计数 / 频次(转化数、点击数)
│  ├─ 大样本                  → 卡方检验
│  └─ 小样本                  → Fisher 精确检验
└─ 比较多于两组               → ANOVA(后续要 post-hoc,见多重比较)
```

> **避坑**:**A/B 测试报告里 99% 的 t 检验默认 equal_var=True,这是错的**——线上指标几乎都不是等方差的(高方差组拖累低方差组)。**永远用 Welch t**(`equal_var=False`),除非你能证明等方差。

## 六、A/B 测试设计:样本量与 MDE

### 6.1 必须事先算样本量(否则白做)

样本量计算公式(t 检验近似版):

```
N per group ≈ 2 · σ² · (z_{α/2} + z_β)² / Δ²

σ      :指标的标准差(用历史数据估)
Δ      :最小可检测效应 MDE(业务上有意义的最小差距)
α      :显著性水平(常用 0.05,z_{α/2} = 1.96)
β      :漏报率(常用 0.20,z_β = 0.84)
        (z_{α/2}+z_β)² ≈ 7.84,所以 N ≈ 16 · σ² / Δ²(粗算口诀)
```

工程口诀:**N ≈ 16 · (σ/Δ)²** —— 也就是**信噪比的平方乘 16**。

### 6.2 MDE 怎么定

MDE = "**业务上,小于这个效果不值得上线**"。

```
推荐 CTR 当前 5%,想检测 +1%(相对提升 20%)→ Δ=0.01
延迟当前 200ms,想检测 +10ms                  → Δ=10
营收 ARPU 10 元,想检测 +0.5 元                → Δ=0.5
```

**Δ 越小,样本量越大**(N ∝ 1/Δ²)——**MDE 减半,样本量翻 4 倍**。

```
例子:CTR 5%,σ ≈ 0.22(伯努利方差 √(p(1-p)))
  想检测 +1%(MDE=0.01)→ N ≈ 16 · (0.22/0.01)² ≈ 77,440 per group
  想检测 +0.5%           → N ≈ 16 · (0.22/0.005)² ≈ 310,000 per group
  想检测 +0.1%           → N ≈ 16 · (0.22/0.001)² ≈ 7,750,000 per group !
```

> **核心结论**:**A/B 测试有效不是"流量够大",是"流量 ≥ 样本量计算结果"**。**0.1% 提升要 700 万样本/组,这就是为什么小公司根本检测不到微小提升**——不是没有,是看不见。

### 6.3 statsmodels 一行算样本量

```python
from statsmodels.stats.power import tt_ind_solve_power, NormalIndPower

# 连续型指标
n = tt_ind_solve_power(effect_size=0.1, alpha=0.05, power=0.8,
                       ratio=1, alternative='two-sided')
print(f"每组样本量 ≈ {int(n)}")    # effect_size = Δ/σ

# 比例(转化率)
from statsmodels.stats.power import NormalIndPower
analysis = NormalIndPower()
n = analysis.solve_power(effect_size=0.1, alpha=0.05, power=0.8)
```

## 七、多重比较陷阱:做 20 次实验,巧合就来

### 7.1 现象

```
α = 0.05 意味着 H0 真的时候,5% 概率误判为"显著"
做 20 个独立检验,全 H0 真:
  至少一个显著的概率 = 1 - 0.95^20 ≈ 0.64
  → 你以为是"发现一个有效改动",其实 64% 概率纯巧合
```

**工程里常见的多重比较**:

- 同时跑 N 个变种(C、D、E、F vs A)
- 在 N 个指标里报告
- 在 N 个用户细分里看
- 在 N 个时间窗口里看
- 在 N 个埋点里挖

每一种都在膨胀实际 α。

### 7.2 校正方法

| 方法 | 思路 | 严格度 | 适用场景 |
| --- | --- | --- | --- |
| **Bonferroni** | α 除以检验数 N,每个用 α/N | 极保守 | 检验数 ≤10,怕假阳 |
| **Holm-Bonferroni** | 排序后逐步用 α/N, α/(N-1), ...| 略松,严格 | Bonferroni 升级版 |
| **FDR (Benjamini-Hochberg)** | 控制"被报告为显著的里假阳的比例" | 适中 | 大规模检验(基因组、ML 调参) |
| **置换检验** | 重新随机分组,看原差异有多极端 | 灵活 | 不假设分布、能处理依赖 |

**Bonferroni 太严**:做 100 个检验,每个 α=0.0005,几乎啥都检测不出来。**FDR(BH 法)是工程上更合理的妥协**——不要求"一个假阳都没有",而是"显著结果里假阳比例 ≤5%"。

### 7.3 BH 法实战(scipy 一行)

```python
from statsmodels.stats.multitest import multipletests

# 假设你做了 20 个 A/B 测试,p 值如下
pvals = [0.001, 0.008, 0.039, 0.041, 0.042, 0.06, 0.074, 0.205,
         0.212, 0.216, 0.222, 0.251, 0.269, 0.275, 0.34, 0.456,
         0.648, 0.759, 0.901, 0.939]

# Bonferroni
reject, p_corrected, _, _ = multipletests(pvals, alpha=0.05, method='bonferroni')
print("Bonferroni 显著:", reject.sum())   # 通常只剩 1-2 个

# Benjamini-Hochberg(FDR)
reject, p_corrected, _, _ = multipletests(pvals, alpha=0.05, method='fdr_bh')
print("BH (FDR) 显著:", reject.sum())     # 通常会保留几个
```

> **经验法则**:**ML 论文 / 调参实验用 BH 法,A/B 实验平台用 Bonferroni**(决策方向重要,宁可错杀)。**两个都比"原始 p<0.05 就报"靠谱得多**。

## 八、Peeking 与序贯检验

### 8.1 Peeking:为什么不能"提前看结果就 stop"

A/B 测试跑了 3 天,p=0.04,看起来显著——直接 stop?

```
名义 α = 0.05
但你"每天看一次,p<0.05 就停":
  实际 α 不再是 0.05,可能涨到 0.15+
  → 假阳率翻 3 倍
```

**直观原因**:每次检查都给"运气好碰到 p<0.05"的机会。固定样本量 t 检验的 α=0.05 是"**只看一次**"的前提,你看 N 次,N 次里只要有一次"显著"就停,假阳率必然膨胀。

```
模拟:H0 是真的(两组毫无差异),每天检查 p<0.05 就停
不 peek(只 day 14 看):     假阳率 ≈ 5%(标称值)
每天 peek 14 次:           假阳率 ≈ 23%
每小时 peek(实时大屏):     假阳率 ≈ 50%+
```

> **核心避坑**:**A/B 实验大屏的"实时 p 值"是统计上最有毒的东西**。看着曲线一升一降,人类决策必然在"看起来显著的瞬间"按停止键——**这是 p-hacking 的工程化形态**。

### 8.2 序贯检验:可以 peek 的合法工具

业界拯救方案——**序贯检验(Sequential Testing)**,允许多次检查,但每次用更严格的阈值:

| 方法 | 思路 | 工程实现 |
| --- | --- | --- |
| **SPRT(Wald 序贯概率比检验)** | 每次算似然比,跨越上下界停 | 经典论文 |
| **Group Sequential**(O'Brien-Fleming / Pocock) | 预设几个检查点,每点用不同 α | 医药临床试验 |
| **Always-Valid p-values**(Howard et al. 2021) | 任何时点 stop 都保证 α | Optimizely、Statsig |
| **mSPRT**(混合 SPRT) | 不需要预先知道效应量 | Optimizely 论文 |

**主流 A/B 平台都已经用序贯检验**:Optimizely、Statsig、Eppo、Stripe 内部实验平台,基本都默认 always-valid。**你看到的 p 值是被校正过的,可以放心 peek**——但**这是平台帮你做的,自己写脚本算的 p 值不能 peek**。

> **经验法则**:**用统计平台的现成实验功能,别自己写 ttest_ind 脚本算 p 值后大屏 peek**。如果一定自己算,固定样本量 / 固定时间,过了再看,不许中途 stop。

## 九、贝叶斯 A/B 测试:不需要预先样本量

### 9.1 思路

频率派假设检验:**问"假设 B 和 A 没差,看到这种结果多反常"**。
贝叶斯 A/B:**直接问"B 比 A 好的概率多少"**——这就是经理一直想要的答案。

```
假设 B 的转化率 θ_B ~ Beta(α_B, β_B)
       A 的转化率 θ_A ~ Beta(α_A, β_A)

观测后,后验仍是 Beta(共轭先验)
   ↓
从两个后验各采样 10000 次,数有多少次 θ_B > θ_A
   ↓
"B 比 A 好的概率" = 那个比例
```

### 9.2 10 行代码

```python
import numpy as np
rng = np.random.default_rng(0)

# A 组:10000 曝光,500 转化
# B 组:10000 曝光,540 转化
a_alpha, a_beta = 1 + 500, 1 + 9500
b_alpha, b_beta = 1 + 540, 1 + 9460

# 从两个后验采样
n_samples = 100_000
a_samples = rng.beta(a_alpha, a_beta, n_samples)
b_samples = rng.beta(b_alpha, b_beta, n_samples)

p_b_better = (b_samples > a_samples).mean()
lift = ((b_samples - a_samples) / a_samples)
print(f"B 比 A 好的概率:{p_b_better:.1%}")
print(f"提升 95% CI:[{np.quantile(lift, 0.025):.1%}, {np.quantile(lift, 0.975):.1%}]")
```

经理拿到这个 "82% 概率 B 比 A 好" 才能做决策——**比"p=0.04"直接、合规、可解释**。

### 9.3 贝叶斯 A/B 的优势

| 维度 | 频率派 | 贝叶斯 |
| --- | --- | --- |
| 需要预先算样本量 | **是**(MDE/power) | **否**(后验随时可看) |
| 能不能 peek | **不能** | **能**(每次看到的都是合法后验) |
| 输出 | p 值 + CI | "B 好的概率" + 后验分布 |
| 业务沟通 | 难解释 | 直接给概率 |
| 多臂(>2 个变种) | 多重比较麻烦 | 直接采样,"P(B 最优)" |
| 损失评估 | 算不出 | "预期损失" = E[(θ_A - θ_B) · 1[B not best]] |

> **核心直觉**:**贝叶斯 A/B 测试不需要"预先算样本量",也不需要"等够样本",任何时点都可以看后验,得到"B 比 A 好的概率"——这玩意儿对工程团队太友好了**。代价是要算后验(共轭情况就是几行代码,复杂指标要 MCMC)。

工业落地的工具:VWO、Convert、Statsig 都有贝叶斯 A/B 模式;Google 的内部实验平台、Netflix 的 XP 都混合使用。

## 十、Twyman's law:看起来太显著的结果,先怀疑数据

```
Twyman's Law:
   "Any figure that looks interesting or different
    is usually wrong."
   任何看起来有意思 / 反常的数据,通常是错的。
```

A/B 测试报 +50% 提升——**不是激动,是先查埋点**。

**该查的事**:

| 现象 | 大概率原因 |
| --- | --- |
| 提升异常大(>20%) | 埋点错、流量分桶错、SRM(样本比失衡) |
| 提升异常一致(每天都 +5%) | 缓存 / CDN / 计算时机错位 |
| 凌晨 1 点开始变化 | 部署有问题、时区错配 |
| 某个细分组特别牛 | Simpson 悖论、用户重复曝光 |
| p 值刚刚卡在 0.049 | 是否 peek 出来的 |
| 转化数完全相同 | 系统 bug,实验没真正分流 |

**SRM 检查(Sample Ratio Mismatch)** 是 A/B 测试上线前的必查项:

```python
# 理论分流 50/50,实际是多少
from scipy.stats import chisquare

n_a, n_b = 49800, 50200
chi2, p = chisquare([n_a, n_b], f_exp=[(n_a+n_b)/2]*2)
if p < 0.001:
    print("⚠️ SRM!分流有问题,实验数据不能信")
```

**SRM 不通过,任何"显著"结果都作废**——因为分流本身有偏,A 组和 B 组人群可能根本不一样。

> **避坑**:**A/B 报告第一行不应该是 p 值,应该是 SRM 检查、样本量、MDE**——如果分流都不对,后面所有数字都是垃圾。Twyman 法则告诉你:**反常结果先质疑,再庆祝**。

## 十一、CUPED:方差缩减让样本量减半

### 11.1 思路

A/B 实验里指标方差 σ² 决定样本量。**如果能用"实验前的协变量"解释掉一部分方差,样本量就能减小**。CUPED(Controlled-experiment Using Pre-Experiment Data,Microsoft 2013)就是干这个的。

```
Y'_i = Y_i - θ · (X_i - E[X])
       ↑          ↑
       新指标     实验前的相关协变量(同用户上周的指标)

θ = Cov(Y, X) / Var(X)    # 最优系数,最小化 Var(Y')

→ Var(Y') = Var(Y) · (1 - ρ²)
            ↑
            如果 X 和 Y 的相关性 ρ=0.7,方差缩到 51%
            → 样本量需求减半
```

### 11.2 一段实现

```python
import numpy as np

# Y:实验期指标,X:实验前同用户的指标(预期相关)
def cuped_adjust(Y, X):
    theta = np.cov(Y, X)[0, 1] / np.var(X)
    return Y - theta * (X - X.mean())

# 然后对调整后的指标做标准 t 检验
Y_adj_treatment = cuped_adjust(Y_treatment, X_treatment)
Y_adj_control   = cuped_adjust(Y_control,   X_control)
# t_test(Y_adj_treatment, Y_adj_control)  ← 显著性 / 置信区间更紧
```

**工程价值**:**几乎免费降低样本量需求 40-50%**——Microsoft、LinkedIn、Netflix、ByteDance 的内部 A/B 平台都默认开启 CUPED 或其变种(双重稳健估计、回归调整等)。

### 12.3 何时实验本身不靠谱

**A/B 测试不是万能的**,以下场景实验结果可能误导:

| 场景 | 问题 | 救法 |
| --- | --- | --- |
| 网络效应(社交、推荐) | 一个用户的实验影响另一组 | 集群随机化(按城市/学校分桶) |
| 长期效应 vs 短期 | 短期 +5%,长期 -10%(疲劳、习得) | 长跑实验 + 留存指标 |
| 新奇效应 | 改动初期都涨,后期回落 | 至少观察 1-2 个 novelty cycle |
| 用户量很小(B2B) | 样本永远不够 | 准实验、合成对照、贝叶斯 + 强先验 |
| Holdout 污染 | 控制组也被改动间接影响 | 严格隔离 / 城市级隔离 |
| 重大业务事件 | 节假日、营销活动盖过实验信号 | 排除事件期、加协变量调整 |

> **核心避坑清单**:
> 1. p<0.05 ≠ "H1 真的概率 95%";只是"H0 真时数据这么极端的概率 5%"
> 2. A/B 不能 peek,除非用序贯检验或贝叶斯 A/B
> 3. 同时测多个指标 / 变种 → BH 法 或 Bonferroni 校正
> 4. 样本量事先算(口诀:N ≈ 16 σ²/Δ²),否则有效果也看不出来
> 5. Welch t 比 Student t 稳,默认 `equal_var=False`
> 6. SRM 检查必跑;不通过的实验数据全作废
> 7. Twyman:看起来太显著的提升先怀疑数据,不要急着上线
> 8. 贝叶斯 A/B 给"B 比 A 好的概率",对业务沟通友好;能 peek
> 9. CUPED 能让样本量减半,几乎免费的工程优化
> 10. 实验设计 ≠ 实验分析:埋点、分流、隔离做不对,任何统计都是装饰

这一篇是「概率与统计」层的收尾——前面 11-12 讲分布、13 讲均值方差、14 讲似然与先验,**这一篇把它们落到"做决策"这件事上**。下一层(信息论)开始,信息熵、KL 散度、互信息会再用一次同样的语言,但目标变成"度量信息和不确定性"。
