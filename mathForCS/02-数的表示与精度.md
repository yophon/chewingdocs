# 数的表示与精度

这一篇是这套系列里**最贴日常事故的一篇**——loss 突然 NaN、FP16 训练发散、量化后精度暴跌、softmax 上溢、log(0) 爆炸,**这些 bug 的根全在"数怎么在机器里被表示"**。

> 一句话先记住:**计算机里没有"实数",只有"以某种格式编码的有限位浮点"**。`0.1 + 0.2 != 0.3` 不是 bug,是设计。**FP16 训练比 FP32 差也不是玄学,是 5 位阶码的物理限制**。这一篇讲清这个,你才能看懂 BF16 / FP8 / INT4 量化的工程动机。

## 一、为什么必须先过这一篇

工程师对"数"的默认假设:`a + b` 就是 `a + b`,精度问题"理论上有,实际不会撞到"。**这个假设在 ML / 数值算法 / 加密 / 大数据统计里全部失效**——精度问题不是"理论上",是**每周都会撞到**。

## 二、整数表示:看似简单的坑

整数比浮点简单很多,但**Python 和 C / Java 的差别一旦理解错,就是隐 bug**。

### 2.1 补码:正数减负数的硬件实现

```
8 位补码,能表示 -128 到 +127:

  +1 = 0000 0001
  -1 = 1111 1111   ← 取反 +1
  -128= 1000 0000

加法不分正负,直接二进制加:
  5 + (-3):
    0000 0101
  + 1111 1101
  ─────────────
    0000 0010   ← 进位被丢掉,结果 = 2  ✓
```

**补码的好处**:**硬件加法器一套电路搞定正负数加减,不用"先看符号位再决定运算"**。代价是表示范围不对称:`-128` 有,`+128` 没有。

> **避坑**:`abs(INT_MIN)` 在 C 里是**未定义行为**——`-(-128) = +128` 溢出。Linus 2008 年在 Linux 内核里专门修过这种 bug。Python 里没事,因为 Python int 无限大。

### 2.2 溢出:无声的灾难

C / Java / Rust 的整数有固定位宽,**溢出会"环绕"或"截断"**,不报错:

```c
int32_t a = 2147483647;   // INT_MAX
int32_t b = a + 1;        // = -2147483648,环绕
```

```java
long t1 = System.currentTimeMillis();
int diff = (int)(t1 - t0);   // 大时间戳相减 → 静默溢出
```

**真实事故**:**Boeing 787 在连续运行 248 天后会掉电**——内部计数器 INT32 溢出。**这个 bug 是航空例子,但工程上同款 bug 工程师每年都写**。

### 2.3 Python vs C / Java 的整数

```
                  最大值                  溢出行为      性能
─────────────────────────────────────────────────────────────────
C int            2^31 - 1                 环绕         快
C long long      2^63 - 1                 环绕         快
Java int         2^31 - 1                 环绕         快
Java long        2^63 - 1                 环绕         快
Java BigInteger  任意                     不溢出       慢(对象)
Python int       任意                     不溢出       慢(动态扩展)
NumPy int32      2^31 - 1                 环绕         快
NumPy int64      2^63 - 1                 环绕         快
```

**坑点**:**Python 写惯了大数 `2**100`,转 NumPy 时一律截到 int64**——`np.array([2**100])` 直接报错。**做 ML 时数据用 NumPy / Torch,标签千万别超过 int64 范围**。

> **经验法则**:**Python 之外的语言,所有"大整数 / 大时间戳 / 大计数"先想清楚位宽**。密码学(RSA 用 2048+ 位)必须用大数库(Python int / Java BigInteger / C 的 GMP)。

### 2.4 一个具体的坑:`(a + b) / 2` 计算中位数

```c
int mid = (low + high) / 2;   // 错!low + high 可能溢出
int mid = low + (high - low) / 2;   // 对
```

**Joshua Bloch 2006 年在 Java 标准库的二分查找里改这个 bug**——这个 bug 在 `Arrays.binarySearch` 里活了 9 年。**底子干净的工程师都会撞**。

## 三、IEEE 754:浮点的真实结构

整数讲完了,真正的事故现场是浮点。**先把结构画出来——三个字段,搞不清这三个字段的相对宽度,后面所有精度问题都没法理解**。

### 3.1 浮点的三段式

一个浮点数 = **符号位(sign) + 阶码(exponent) + 尾数(mantissa)**。

```
                       sign  exponent           mantissa
                       │     │                  │
FP32:                  [ 1 | 8 bits | 23 bits             ]
                                                       │
FP16 (half):           [ 1 | 5 bits | 10 bits ]
BF16 (brain float):    [ 1 | 8 bits | 7 bits  ]
TF32 (NVIDIA):         [ 1 | 8 bits | 10 bits ]   ← 内部 19 bits,对外仍按 FP32 存
FP8 E4M3:              [ 1 | 4 bits | 3 bits ]
FP8 E5M2:              [ 1 | 5 bits | 2 bits ]
```

**对应的数学**(以 FP32 为例):

```
value = (-1)^sign × 1.mantissa × 2^(exponent - 127)
                    └─ 1.xxxxx 隐含的 1
                                 └─ bias = 127
```

**三个字段各干什么的**:

| 字段 | 干什么的 | 多了什么效果 |
| --- | --- | --- |
| sign | 1 位,正负 | — |
| exponent(阶码) | 决定**动态范围**(能表示多大、多小) | 多 1 位,范围翻平方 |
| mantissa(尾数) | 决定**精度**(每个数有多少有效位) | 多 1 位,精度翻倍 |

> **核心直觉**:**阶码管"范围",尾数管"精度"——两者抢同一个总位宽**。FP16 阶码 5 位、尾数 10 位:精度还行,但范围窄,梯度容易下溢。BF16 阶码 8 位、尾数 7 位:范围和 FP32 一样大,但精度差。**大模型训练选 BF16,是宁愿精度差也要范围够**。

### 3.2 为什么 0.1 + 0.2 ≠ 0.3

```python
>>> 0.1 + 0.2
0.30000000000000004
```

**根因**:**0.1 在二进制里是无限循环小数**:

```
0.1 (十进制) = 0.0001100110011001100... (二进制)   ← 循环节
```

**二进制只能精确表示分母是 2 的幂的小数**(0.5、0.25、0.125…),其他全是近似:

| 十进制 | 二进制 | 能精确? |
| --- | --- | --- |
| 0.5 | 0.1 | 是 |
| 0.25 | 0.01 | 是 |
| 0.75 | 0.11 | 是 |
| 0.1 | 0.0001100110011…(循环) | 否 |
| 0.2 | 0.001100110011…(循环) | 否 |
| 0.3 | 0.0100110011001…(循环) | 否 |

**FP64 把 0.1 截到 52 位尾数,得到一个略大于 0.1 的近似值**。两个略大的数加起来,还是略大于 0.3——肉眼看是 `0.30000000000000004`。

> **避坑**:**任何涉及钱的系统不要用浮点存金额**。要么乘以 100 用整数存"分",要么用 `decimal.Decimal`(Python)/ `BigDecimal`(Java)/ `Numeric`(PostgreSQL)。**用 float 存了 0.1 元,迟早对账对不上**。

### 3.3 浮点比较

**`==` 在浮点上几乎永远是错的**。两个数学上相等的浮点数,在内存里可能差一个 ULP(Unit in the Last Place)。

```python
import numpy as np

a = 0.1 + 0.2
b = 0.3
a == b               # False
np.isclose(a, b)     # True
abs(a - b) < 1e-9    # True
```

**怎么写**:

```python
def almost_equal(a, b, rel=1e-9, abs_=1e-12):
    return abs(a - b) <= max(rel * max(abs(a), abs(b)), abs_)
```

或者直接用 `math.isclose` / `np.isclose` / `torch.allclose`——**别自己造轮子,边界情况(NaN / Inf / 0)很容易写错**。

### 3.4 阶码与精度的 trade-off:一张关键的对比

下面这张表是这一篇最值得记的东西——**不背这张表,看 BF16 / FP8 的论文全是黑话**:

| 类型 | 总位 | 阶码 | 尾数 | 最大值 | 最小正规数 | 有效十进制位 | 典型场景 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| FP32 | 32 | 8 | 23 | 3.4e38 | 1.2e-38 | ~7 | 通用、CPU 默认 |
| FP64 | 64 | 11 | 52 | 1.8e308 | 2.2e-308 | ~15-17 | 科学计算、金融 |
| FP16 | 16 | 5 | 10 | 65504 | 6.1e-5 | ~3-4 | 推理、早期混精训练 |
| BF16 | 16 | 8 | 7 | 3.4e38 | 1.2e-38 | ~2-3 | 大模型训练主流 |
| TF32 | 19 | 8 | 10 | 3.4e38 | 1.2e-38 | ~3-4 | A100/H100 上 FP32 自动加速 |
| FP8 E4M3 | 8 | 4 | 3 | 448 | 1.95e-3 | ~1-2 | 前向激活、权重 |
| FP8 E5M2 | 8 | 5 | 2 | 57344 | 6.1e-5 | ~1 | 反向梯度(范围大) |
| INT8 | 8 | — | — | 127 | -128 | — | 量化推理 |
| INT4 | 4 | — | — | 7 | -8 | — | 大模型量化推理 |

**这张表上最重要的两行对比**:

```
FP16: 阶码 5 位 → 最大 65504,小于 6e-5 就下溢成 0
BF16: 阶码 8 位 → 最大 3.4e38,小于 1.2e-38 才下溢

训练时梯度量级常常在 1e-7 到 1e-3,FP16 直接吃掉一截。
```

> **核心直觉**:**FP16 精度高但范围窄,BF16 范围宽但精度低**。**深度学习更怕"梯度下溢"(整段消失),不怕"少几位精度"**——所以 BF16 赢了。

## 四、ML 精度族谱:为什么 BF16 成主流

### 4.1 训练精度的演化时间线

```
2010 之前    FP32 一统天下
2015 左右    NVIDIA 推 FP16(P100/V100),开始混合精度
2017-2019    Apex / amp:FP16 训练成熟,但 loss scaling 是个坑
2018         Google TPU 推 BF16
2020         A100 推 TF32,FP32 自动加速
2022         H100 推 FP8(E4M3 + E5M2)
2024-2025    大模型基本统一在 BF16 训练,FP8 在推理/激活流行
```

### 4.2 FP16 的两个坑

**坑 1:梯度下溢**

```python
import torch
g = torch.tensor([1e-7], dtype=torch.float16)   # 想存 1e-7
print(g)   # tensor([0.], dtype=torch.float16)  ← 下溢成 0
```

**FP16 最小正规数是 6.1e-5**,任何小于这个值的梯度全部下溢成 0,**那一层的权重就停止更新**——表面 loss 下降变慢、模型不收敛。

**坑 2:logits 上溢**

```python
import torch
x = torch.tensor([5e4, 5e4, 5e4], dtype=torch.float16)
torch.exp(x)   # tensor([inf, inf, inf], dtype=torch.float16)
```

**FP16 最大值 65504**,`exp(11.1) ≈ 65535` 就到边了。**softmax 直接出 NaN**。

### 4.3 BF16:同动态范围 + 低精度

**BF16 的设计哲学**:**保持 FP32 的阶码 8 位不变,牺牲尾数到 7 位**——动态范围一样,精度差一点,但梯度不下溢、激活不上溢。

```
FP32:  [1 | 8 | 23]   ┐
                       │  阶码完全一致
BF16:  [1 | 8 | 7 ]   ┘

BF16 ↔ FP32 转换:直接截断 / 补零,无需对齐阶码
```

**结果**:**BF16 训练几乎不需要 loss scaling 的复杂度**——直接换 dtype 就跑。**这是它取代 FP16 的工程理由**。

### 4.4 TF32:NVIDIA 的私货

**TF32 不是 IEEE 754 标准类型**——是 NVIDIA A100/H100 内部的一个"加速桥":

```
你写 FP32 矩阵乘法
       ↓
A100 内部自动截到 TF32(阶码 8 + 尾数 10)做 GEMM
       ↓
结果累加到 FP32 寄存器,对外仍然是 FP32
```

**代价是少了 13 位尾数精度**,**好处是 GEMM 吞吐量直接翻倍**。**你不用改代码就拿到加速**——但科学计算要求高精度的别开,默认 PyTorch 2.x 是开着的:

```python
torch.backends.cuda.matmul.allow_tf32 = False   # 关掉,要求严格 FP32
```

### 4.5 FP8:H100 / B100 时代的训练精度

**FP8 有两个变种**,对应不同位置:

| 类型 | 阶码 | 尾数 | 用途 |
| --- | --- | --- | --- |
| E4M3 | 4 | 3 | 前向激活、权重(精度更重要) |
| E5M2 | 5 | 2 | 反向梯度(范围更重要) |

**FP8 训练通常需要 per-tensor 或 per-channel 的动态 scaling**——把每个 tensor 在量化前先乘一个因子,把它的范围拉到 FP8 能表示的区间。**这是和 BF16 最大的区别——BF16 直接换 dtype,FP8 需要 framework 支持**(Transformer Engine、TorchAO、MS-AMP)。

### 4.6 dtype 选择速查表

| 场景 | 推荐 dtype | 理由 |
| --- | --- | --- |
| 大模型预训练 | BF16 | 范围够、不用 loss scaling |
| 大模型微调 | BF16 | 同上 |
| 小模型训练(MLP / CNN) | FP32 或混精 | 精度更重要,范围够用 |
| 大模型推理(高质量) | BF16 / FP16 | 显存省一半 |
| 大模型推理(极限省显存) | INT8 / INT4 | 见下节量化 |
| 科学计算、金融 | FP64 | 不能丢精度 |
| 嵌入式 / 边缘部署 | INT8 / FP16 | 算力 / 内存限制 |
| A100/H100 上加速 FP32 | TF32(默认开) | 透明加速 |

## 五、量化:把浮点压成整数

**量化的本质**:**把 FP32 / BF16 的连续值,映射到 INT8 / INT4 的有限格点**——精度下降换内存和算力。

### 5.1 对称量化 vs 非对称量化

```
对称量化(symmetric):
  原始范围 [-A, A] → INT8 范围 [-127, 127]
  公式:q = round(x / scale),scale = A / 127

  优点:零点对齐,反量化时不用加 zero_point
  缺点:如果原始分布偏一边(比如全正数),浪费一半精度

非对称量化(asymmetric):
  原始范围 [min, max] → INT8 范围 [0, 255]
  公式:q = round(x / scale + zero_point)
        scale = (max - min) / 255
        zero_point = -round(min / scale)

  优点:充分利用 INT8 全部 256 个格点
  缺点:运算时要带 zero_point,稍慢
```

**经验**:**权重通常对称量化**(分布近似零均值),**激活通常非对称量化**(ReLU 后全是正数)。

### 5.2 量化的精度损失从哪来

**两个来源**:**截断误差** + **异常值**。

```
正常值集中区:[-1.2, 1.2]
异常值:      [-1.2, ..., 5.0]   ← 一个异常值把整个 scale 拉爆

对称量化:scale = 5.0 / 127 ≈ 0.039
        正常值精度 = 0.039  ← 比原来差太多
```

**这就是 LLM 量化最大的难点**——LLM 的某些激活通道有**离群点**(outlier),会把 scale 拉爆,正常通道精度全毁。

### 5.3 主流 LLM 量化算法

| 算法 | 思路 | 适用 | 工具 |
| --- | --- | --- | --- |
| RTN(Round-To-Nearest) | 最朴素,直接量化 | 小模型、INT8 | 任意框架自带 |
| GPTQ | 基于 Hessian 信息的逐层校准 | 4 bit / 3 bit LLM | AutoGPTQ |
| AWQ | 识别"重要"通道,保持其精度 | 4 bit LLM | AutoAWQ |
| SmoothQuant | 把激活异常值"smooth" 到权重 | INT8 LLM 部署 | TensorRT-LLM |
| QLoRA | 4 bit 权重 + LoRA 微调 | 微调超大模型 | bitsandbytes |

**数学共同点**:**都在找"哪些数值对最终输出最重要,优先保它"**——这是个**重要性采样的子问题**(详见 11 篇)。

> **经验法则**:**INT8 PTQ(Post-Training Quantization)对小模型够用,但 LLM 一定要校准 / GPTQ / AWQ,直接 RTN 一般掉点严重**。

### 5.4 量化的工程对应

```python
# PyTorch 原生 INT8 量化(适合小模型)
import torch.quantization as tq
model_int8 = tq.quantize_dynamic(model, {torch.nn.Linear}, dtype=torch.qint8)

# bitsandbytes 4 bit(LLM 微调标配)
from transformers import AutoModelForCausalLM
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b",
    load_in_4bit=True,         # ← 一行换 4 bit
    bnb_4bit_quant_type="nf4", # ← NormalFloat 4-bit
)

# AutoGPTQ(更精细的 4 bit)
from auto_gptq import AutoGPTQForCausalLM
model = AutoGPTQForCausalLM.from_quantized(
    "TheBloke/Llama-2-7B-GPTQ",
    use_safetensors=True,
)
```

**量化收益对照**:

| 精度 | 7B 模型显存 | 推理速度 | 质量损失 |
| --- | --- | --- | --- |
| FP32 | 28 GB | 1× | 0 |
| BF16 | 14 GB | ~1.5-2× | < 0.5% |
| INT8 | 7 GB | ~2-3× | 0.5-2% |
| INT4 | 3.5 GB | ~3-4× | 1-5% |

> **避坑**:**INT4 不是免费午餐**。模型越小、任务越难,量化掉点越明显。**7B INT4 一般还行,1.5B INT4 经常崩**。

## 六、真实事故现场

### 6.1 NaN 训练:三种最常见的原因

```python
# 原因 1:log(0)
loss = -torch.log(prob)   # prob 里有 0 → -inf

# 原因 2:除零
x = a / b   # b 里有 0 → inf;0 / 0 → NaN

# 原因 3:exp 上溢
e = torch.exp(logits)   # logits > 88 (FP32) 或 > 11 (FP16) → inf
softmax = e / e.sum()   # inf / inf → NaN
```

**排查顺序**:

```python
# 一行检测 NaN 来自哪一层
torch.autograd.set_detect_anomaly(True)

# 或者手动检查
for name, p in model.named_parameters():
    if torch.isnan(p).any():
        print(f"NaN in {name}")
    if torch.isnan(p.grad).any():
        print(f"NaN grad in {name}")
```

### 6.2 softmax 溢出:经典案例

```python
import numpy as np

# 不稳定写法
def softmax_naive(x):
    e = np.exp(x)
    return e / e.sum()

x = np.array([1000., 1000., 1000.])
softmax_naive(x)
# RuntimeWarning: overflow encountered in exp
# array([nan, nan, nan])
```

**问题**:`exp(1000)` 远远超过 FP64 的 1.8e308,直接 inf。

**修复**:**减去最大值,softmax 输出不变(指数相减等于做了归一化)**:

```python
def softmax_stable(x):
    x = x - x.max()   # ← 关键一行
    e = np.exp(x)
    return e / e.sum()

softmax_stable(np.array([1000., 1000., 1000.]))
# array([0.333, 0.333, 0.333])  ✓
```

> **核心直觉**:`softmax(x) == softmax(x - c)` for any constant c——**任何 softmax 实现都该先减最大值**。PyTorch / NumPy / JAX 内置的 softmax 已经这么做了,**只有你自己手写时才会撞**。

### 6.3 log(0):交叉熵的隐藏地雷

```python
# 不稳定:softmax 输出可能是 0,log(0) = -inf
probs = softmax(logits)
loss = -torch.log(probs[target])

# 稳定:log_softmax 内部用 log-sum-exp,永远不算 log(0)
log_probs = F.log_softmax(logits, dim=-1)
loss = -log_probs[target]

# 最佳:cross_entropy 直接吃 logits
loss = F.cross_entropy(logits, target)
```

> **避坑**:**永远不要手写 `log(softmax(x))`**——用 `log_softmax` 或直接 `cross_entropy`。这两个 API 存在的唯一原因就是为了数值稳定。

### 6.4 (a - b) 灾难性抵消(catastrophic cancellation)

```python
a = 1.0000001
b = 1.0000000
a - b   # 1.0000000031710769e-07   ← 应该是 1e-7,但有噪声

# 更严重的例子:方差的两种算法
import numpy as np
x = np.array([1e9, 1e9 + 1, 1e9 + 2])

# 公式 1:E[X²] - E[X]²(数学等价但数值不稳)
var1 = np.mean(x**2) - np.mean(x)**2
# var1 = -171798691.84   ← 负数!应该是 0.667

# 公式 2:E[(X - μ)²](Welford 等算法用这个)
mu = x.mean()
var2 = np.mean((x - mu)**2)
# var2 = 0.667   ✓
```

**根因**:**两个接近的大数相减,前面的有效位全抵消掉,只剩下噪声**。FP64 有效位 ~15,1e9² 就吃掉 18 位,完全没法和均值平方比较。

> **经验法则**:**任何"两个接近的大数相减"的公式都要换写法**。统计里算方差用 Welford 算法,几何里算距离不要展开 `(a-b)²` 公式,数值微分用中心差分而非前向差分。

## 七、log-sum-exp:数值稳定的瑞士军刀

这个技巧本身简单,但**它是整个数值稳定话题里出现频率最高的一个**——softmax、cross entropy、贝叶斯模型、注意力都在用。值得专门一节。

### 7.1 问题:log(sum(exp(x))) 怎么算

很多 ML 公式长这样:

```
log Σ exp(x_i)
```

**直接算的两个坑**:

1. `exp(x_i)` 上溢:`x_i = 1000` 就 inf
2. `exp(x_i)` 下溢:`x_i = -1000` 全是 0,log(0) = -inf

### 7.2 解法:减去最大值

```
log Σ exp(x_i)
  = log Σ exp(x_i - M) · exp(M)         ← 凑因子
  = log [exp(M) · Σ exp(x_i - M)]
  = M + log Σ exp(x_i - M)              ← 关键变形
```

**其中 `M = max(x)`**——这样 `exp(x_i - M)` 最大值是 1,最小值是个很小的正数(不下溢成关键项)。

### 7.3 五行代码

```python
import numpy as np

def logsumexp_stable(x):
    M = np.max(x)
    return M + np.log(np.sum(np.exp(x - M)))

# 对比
x = np.array([1000., 1001., 1002.])

# 不稳定
np.log(np.sum(np.exp(x)))   # inf

# 稳定
logsumexp_stable(x)          # 1002.4076...

# scipy 内置
from scipy.special import logsumexp
logsumexp(x)                 # 1002.4076...
```

**`scipy.special.logsumexp` / `torch.logsumexp` / `tf.math.reduce_logsumexp` 全是这个实现**。

### 7.4 哪些 API 内部用了它

| API | 内部用 log-sum-exp | 用途 |
| --- | --- | --- |
| `F.softmax` | 是(减 max) | 不上溢 |
| `F.log_softmax` | 是 | softmax 后取 log 不 -inf |
| `F.cross_entropy` | 是 | 分类损失 |
| `F.nll_loss` | 间接(配 log_softmax) | 负对数似然 |
| `torch.distributions.Categorical.log_prob` | 是 | 采样器的对数概率 |
| Attention 里的 softmax | 是 | 防止 logits 大时溢出 |
| HMM 的前向 / 后向算法 | 是(连乘变连加) | 不下溢 |
| 贝叶斯模型计算 evidence | 是 | 多个概率相乘 |

> **经验法则**:**看到代码里有 `exp` + `sum` + `log` 任意两个组合,先问"用 logsumexp 吗"**。99% 情况答案是要用。

### 8.1 PyTorch / NumPy 速查

```python
# dtype 检查与转换
x.dtype                       # 看类型
x = x.to(torch.bfloat16)      # 转 BF16
x = x.float()                 # 转 FP32
x = x.half()                  # 转 FP16

# 混合精度训练(自动)
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()         # FP16 才需要;BF16 不需要
for inputs, labels in data:
    with autocast(dtype=torch.bfloat16):   # 推荐 BF16
        outputs = model(inputs)
        loss = loss_fn(outputs, labels)
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()

# 数值稳定的 API
F.log_softmax(x, dim=-1)      # 不要写 log(softmax(x))
F.cross_entropy(logits, y)    # 不要写 -log(softmax(logits))[y]
torch.logsumexp(x, dim=-1)    # 不要写 log(sum(exp(x)))
torch.logaddexp(a, b)         # 不要写 log(exp(a) + exp(b))

# 检查 NaN / Inf
torch.isnan(x).any()
torch.isinf(x).any()
torch.isfinite(x).all()       # 都是有限值
```

### 8.2 库与框架的精度选项

| 库 / 框架 | 混精训练 API | 量化 API |
| --- | --- | --- |
| PyTorch | `torch.cuda.amp` | `torch.quantization`(动态/静态) |
| NVIDIA Apex | `apex.amp`(已废弃,推 amp) | — |
| HuggingFace | `accelerate`(包 amp + DDP) | `bitsandbytes`(4/8 bit) |
| vLLM | 内置 BF16 / FP8 推理 | AWQ / GPTQ 加载 |
| TensorRT-LLM | FP8 / INT8 推理 | SmoothQuant |
| llama.cpp | — | GGUF 多档量化(Q2-Q8) |
| Apple MLX | 默认 FP32 / FP16 | 4/8 bit |

### 8.3 这一层的局限

**这一层数学解决的是"数值不爆炸"问题**——但不解决:

- **算法本身的稳定性**:Adam 比 SGD 数值稳,但**模型设计的问题**(梯度消失 / 爆炸)还得改架构(LayerNorm / 残差 / 初始化)
- **分布漂移**:训练 vs 推理 dtype 不一致带来的精度损失,需要 calibration
- **算子的实现差异**:不同 GPU / 不同框架的同一个算子(比如 softmax),最低一位可能不同——**这一层是 IEEE 754 没规定的实现细节**
