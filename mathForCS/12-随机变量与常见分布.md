# 随机变量与常见分布

上一篇把概率心智搭起来了——条件概率、贝叶斯、独立性。**这一篇把视角从「单个事件」拉到「分布」**:一个随机变量取值的整体分布长什么样、参数怎么影响形状、不同分布适合描述什么样的现实数据。

> 一句话先记住:**「分布选错」是工程里最贵的概率错误之一**。监控告警按高斯算 P99(实际是长尾)、风控按高斯算极端损失(实际是肥尾)、推荐冷启动按均匀算先验(实际是幂律)——**每个都是直接给你的系统埋上低估尾部风险的雷**。「the average is a lie」这句话之所以反复被说,根源就是大部分人没意识到自己默认的分布假设是错的。

## 一、为什么必须懂分布

### 1.1 三个真实场景

**场景 1**:你做 API 监控,延迟数据 mean=100ms、std=20ms。按高斯三 σ 法则,P99.7 ≈ 160ms。**实际 P99 是 800ms,P999 是 3 秒**——因为延迟是**重尾分布**,高斯严重低估了尾部。

**场景 2**:你估算公司收入分布,用均值看「平均客户贡献」做决策。**实际上头部 1% 客户贡献了 50% 收入,均值被几个大客户拉高了 5 倍**——「平均客户」根本不存在。

**场景 3**:你做请求到达率建模,假设是泊松,**计算出来「同时来 10 个请求」是百年一遇**。实际上你的请求**带突发性**(用户每天高峰期集中点击),百年一遇平均每周来一次。

| 场景 | 用错的分布 | 真实分布 | 后果 |
| --- | --- | --- | --- |
| 延迟监控 | 高斯 | 长尾(log-normal / Pareto) | P99 严重低估,SLA 报警失效 |
| 收入估算 | 高斯 | 幂律 / Pareto | 「平均客户」无意义 |
| 请求到达 | 平稳泊松 | 非平稳 / 突发 | 容量规划不足 |
| 文件大小 | 高斯 | log-normal | 存储估算偏差 |
| 网页访问量 | 高斯 | 幂律(Zipf) | 缓存命中率算错 |

> **核心直觉**:**默认假设「高斯」是工程师最常犯的错——高斯是个特殊分布,只在「大量独立小因素叠加」时才会出现,真实工程数据(延迟、收入、PV、错误率)大多不满足这个条件**。

## 二、离散 vs 连续 / PMF vs PDF / CDF / 分位数

### 2.1 两类随机变量

```
离散(discrete)
  取值是可数的(0, 1, 2, ... 或一组类别)
  描述工具:PMF(probability mass function,概率质量函数)
  例:扔骰子结果、点击与否、请求是不是错的

连续(continuous)
  取值是实数区间
  描述工具:PDF(probability density function,概率密度函数)
  例:身高、延迟时长、温度
```

### 2.2 PMF 与 PDF 的区别

**关键差异**:PMF 是「概率」(直接读)、PDF 是「密度」(要积分才是概率)。

```
PMF:                                   PDF:
P(X = k)                              P(X ∈ [a, b]) = ∫ pdf(x) dx
直接读值                                密度本身不是概率!
        ↑                                       ↑
      离散                                    连续
                                              
P(X = k=2) = 0.3                      pdf(2.5) = 0.4
←直接的概率值                          ←每单位长度的概率密度
                                       P(X = 任何单点) = 0
```

**新手陷阱**:连续分布里 `pdf(x)` 可以 **> 1**(比如均匀分布在 [0, 0.5] 上 pdf=2)。**密度不是概率,可以大于 1;只有它的积分才是概率,积分必须 ≤ 1**。

### 2.3 CDF:统一两类的工具

CDF(cumulative distribution function,累积分布函数):

```
F(x) = P(X ≤ x)

对离散:F(x) = Σ_{k≤x} P(X=k)
对连续:F(x) = ∫_{-∞}^{x} pdf(t) dt

性质:
  F 单调不减,从 0 到 1
  F(+∞) = 1
  P(a < X ≤ b) = F(b) - F(a)
```

```
   PDF                              CDF
    │   ╱╲                          │            ╭───── 1.0
    │  ╱  ╲                         │          ╱
    │ ╱    ╲      ↑积分             │        ╱
    │╱      ╲                       │      ╱
────┴────────╲──> x        ────────╱─────────> x
              ╲                  ╱
               ╲              ╱
                ─────         0.0
```

### 2.4 分位数:CDF 的反函数

「P90」「P99」「中位数」都是分位数(quantile):

```
分位数 q_p:满足 F(q_p) = p 的那个 x

中位数 = q_{0.5}
P90    = q_{0.9}     ← 90% 的样本 ≤ 这个值
P99    = q_{0.99}
P999   = q_{0.999}
```

**为什么分位数比均值鲁棒**:

```
样本: [10, 20, 30, 40, 50, 60, 70, 80, 90, 10000]

均值   = 1045          ← 被极端值 10000 拉高 20 倍
中位数 = 55             ← 几乎不动
P90    = 90             ← 不受极端值影响
```

> **经验法则**:**对长尾 / 重尾分布,用中位数和分位数,不用均值**。监控延迟、报告收入、衡量响应时间——**这些场景默认用 P50 / P90 / P95 / P99,不用 mean**。

## 三、高斯分布:为什么这么常见

### 3.1 形状与参数

```
PDF: f(x) = (1 / (σ√(2π))) · exp(-(x-μ)² / (2σ²))

       ╱─╲
      ╱   ╲           ← μ 控制位置(中心)
     ╱     ╲          ← σ 控制宽度(波动)
   ╱─────────╲
 ─────┬─┬─┬───── x
   μ-σ μ μ+σ

特征:
  对称(关于 μ)
  快速衰减(尾部按 exp(-x²) 趋零,极快)
  68-95-99.7 法则
```

### 3.2 三 σ 法则

```
P(|X - μ| < σ)   ≈ 68%       ← 1 个 σ 范围内
P(|X - μ| < 2σ)  ≈ 95%       ← 2 个 σ 范围内
P(|X - μ| < 3σ)  ≈ 99.7%     ← 3 个 σ 范围内,只有 0.3% 在外
```

**这就是「3σ 报警」「6σ 质量控制」的来源**——假设你的数据是高斯,3σ 外的是「极端事件」。**但只有真高斯才适用,长尾分布完全不是这样**。

### 3.3 z-score 标准化

```
z = (x - μ) / σ
```

意思是「x 距离 μ 有多少个 σ」。**所有高斯都能通过 z-score 化成标准正态 N(0, 1)**——这就是查表的依据。

工程上的对应:**深度学习里的 Batch Norm / Layer Norm 都在做 z-score**——把每个特征 / 每个 token 标准化成均值 0 方差 1,让训练更稳定。

```python
# LayerNorm 的核心数学
def layer_norm(x, eps=1e-5):
    mean = x.mean(dim=-1, keepdim=True)
    var = x.var(dim=-1, keepdim=True)
    return (x - mean) / (var + eps).sqrt()
```

### 3.4 为什么权重初始化常用高斯

**Xavier / He 初始化**都基于高斯:

```python
# Xavier(适合 tanh / sigmoid)
W ~ N(0, 1 / fan_in)

# He(适合 ReLU)
W ~ N(0, 2 / fan_in)
```

**核心思路**:让每一层输出的方差和输入大致一致——既不爆炸(梯度变 inf)也不消失(梯度变 0)。高斯的「均值 0、方差可控」特性恰好适合。

### 3.5 高斯为什么这么常见?CLT 一句话

**中心极限定理(Central Limit Theorem)**:大量独立同分布(或满足较弱条件)的随机变量之和,无论原始分布是什么,**和的分布趋向高斯**。

**工程含义**:**当你的观测值是「很多独立小因素叠加」的结果时,默认假设高斯是对的**。比如:
- 多次抛硬币计正面次数(二项 → 高斯)
- 一群学生的身高(基因 + 营养 + 运动 + ... 多因素叠加)
- 测量误差(每个误差源都很小,叠加起来约高斯)

**反过来**:**当观测值是「乘性叠加」或「极端值由少数大因素主导」时,不是高斯**。比如:
- 收入(社会因素 × 运气 × 教育 × ... 是乘性的)
- 文件大小(几个大文件主导)
- 延迟(慢请求被几个 outlier 主导)

> **避坑**:**「数据看着像钟形」不等于是高斯**——log-normal 也是单峰、对称(在 log 尺度上对称),容易看错。**用 QQ-plot 验证比肉眼看靠谱**。

## 四、伯努利、二项:二元结果

### 4.1 伯努利分布

**最简单的随机变量**:**0/1 两个值,P(X=1)=p、P(X=0)=1-p**。

```python
# scipy
from scipy.stats import bernoulli
bernoulli.pmf(1, p=0.3)    # 0.3
bernoulli.rvs(p=0.3, size=10)   # 抽 10 个样本

# numpy
np.random.binomial(1, 0.3, size=10)
```

**工程对应**:
- 点击与否(CTR)
- 转化与否(CVR)
- A/B 测试每个用户是否触发了目标行为
- 监控每次请求是否报错

### 4.2 二项分布

**n 次独立伯努利,求总成功数**:

```
P(X = k) = C(n, k) · p^k · (1-p)^(n-k)
              ↑
           组合数

均值 = n · p
方差 = n · p · (1-p)
```

**关键超参**:
- `n`:试验次数
- `p`:每次成功概率

**工程对应**:
- N 个用户来访,有多少人点击(N 是曝光,p 是 CTR)
- 一天有多少笔交易是欺诈
- 一批服务器,多少台会在某段时间内宕机

### 4.3 当 n 大、p 不接近 0 时,二项 ≈ 高斯

```
B(n, p) ≈ N(np, np(1-p))    当 n 大时
```

**所以**:**做 A/B 测试时,N 足够大(几千以上),CTR 这类二项就可以用高斯近似来算置信区间**——这是大部分实验平台用的 Wald 区间。

### 4.4 当 n 大、p 极小时,二项 ≈ 泊松

```
B(n, p) ≈ Poisson(λ=np)    当 n 大、p 小、np 适中
```

## 五、泊松:稀有事件计数

### 5.1 定义

**单位时间(或单位面积)内某事件发生次数,前提是事件独立 + 平稳到达**:

```
P(X = k) = e^(-λ) · λ^k / k!

唯一参数 λ:单位时间内事件的平均发生数
均值 = λ
方差 = λ
```

### 5.2 直觉

```
λ = 3 的泊松分布(平均 3 次/单位时间):

P(X=0)  ≈ 0.050   ← 0 次的概率
P(X=1)  ≈ 0.149
P(X=2)  ≈ 0.224
P(X=3)  ≈ 0.224   ← 峰值在 λ 附近
P(X=4)  ≈ 0.168
P(X=5)  ≈ 0.101
P(X=10) ≈ 0.0008  ← 10 次几乎不可能
```

**特点**:**均值和方差相等**——这是验证「数据是否符合泊松」的一个快速检验。**如果实际数据 var >> mean(over-dispersion),就不是泊松,得换 negative binomial 等**。

### 5.3 泊松假设什么时候不成立

```
泊松假设:
  1. 事件独立(一个事件不影响下一个)
  2. 平稳(任意单位时间事件率相同)
  3. 同时刻同时发生概率为 0
```

**真实工程数据违反这些的常见方式**:

| 场景 | 违反的假设 | 真实分布 |
| --- | --- | --- |
| 网站请求按小时统计 | 平稳性(白天忙、晚上闲) | 非齐次泊松 |
| 微博转发数 | 独立性(转发引发更多转发) | 自激励 Hawkes 过程 |
| 错误日志(连环报错) | 独立性 | 突发性,负二项分布 |
| 用户 session 内点击 | 独立性 | 用户行为强相关 |

### 5.4 工程对应

```python
from scipy.stats import poisson

# 假设网站平均每分钟 50 个请求
lambda_ = 50

# 下一分钟来 80 个请求的概率
poisson.pmf(80, mu=lambda_)         # ≈ 6e-5

# 下一分钟 ≥ 80 个的概率(尾概率)
1 - poisson.cdf(79, mu=lambda_)     # ≈ 1e-4

# 99% 分位:下一分钟最多多少请求
poisson.ppf(0.99, mu=lambda_)       # ≈ 67
```

**实际工程场景**:
- 容量规划:「P99 流量是多少」决定要部署多少台机器
- 反爬:用泊松估「正常用户单位时间访问次数」,超过 N σ 的判异常
- 队列建模:M/M/1 排队模型假设到达过程是泊松
- 错误率监控:每分钟错误数偏离泊松假设时报警

> **避坑**:**真实生产环境的请求几乎从不严格满足泊松**——突发流量、定时任务集中、用户行为模式都会让方差远大于均值。**做容量规划时,实际 P99 流量通常是泊松估计的 2-5 倍**。

## 六、指数分布:等待时间

### 6.1 定义与直觉

**对应「下一次事件来之前要等多久」**——和泊松成对出现:

```
泊松:单位时间内事件数        (离散,k = 0, 1, 2, ...)
指数:相邻两次事件的时间间隔  (连续,t ≥ 0)

PDF: f(t) = λ · e^(-λt)

均值 = 1/λ
方差 = 1/λ²

形状:从 t=0 处的高峰开始单调衰减
   │\
   │ \
   │  \
   │   \____
   └────────── t
```

### 6.2 无记忆性:反直觉但工程上常用

**指数分布的关键性质**:

```
P(X > s + t | X > s) = P(X > t)
```

**直觉**:「已经等了 s 秒还没来,接下来要等的时间分布,和一开始没等过时完全一样」。**过去的等待时间不影响未来**。

**反直觉例子**:假设公交车到达间隔服从指数分布,平均间隔 10 分钟。

```
你在车站等了 5 分钟,公交还没来。问平均还要等多久?
直觉:5 分钟(总共 10 分钟,已经等了 5,剩下 5)
正确:**还是 10 分钟**——因为无记忆性
```

**反直觉的原因**:指数分布偶尔会有非常长的间隔(尾部),抵消了「已经等过 5 分钟」带来的「应该快了」的预期。

### 6.3 工程对应

```python
from scipy.stats import expon

# 假设一个服务器平均每分钟 1 次 GC,GC 间隔服从指数(λ=1)
# 问下一次 GC 在 2 分钟内的概率
expon.cdf(2, scale=1)    # ≈ 0.865

# 50% 分位(中位间隔)
expon.ppf(0.5, scale=1)  # ≈ 0.693(注意不是均值 1!)
```

| 场景 | 含义 |
| --- | --- |
| 排队论 M/M/1 的服务时间 | 服务耗时假设指数 |
| 电子元器件寿命(早期) | 失效率恒定时是指数 |
| 函数调用间隔 | 假设无记忆时是指数 |
| 监控:服务无故障运行时间 | 假设故障泊松到达 → 间隔指数 |

> **避坑**:**真实系统的「故障间隔」基本不是指数**——硬件有老化(故障率上升)、新部署有 bug(故障率先高后低,bathtub curve)、不同时段流量不同。**用指数估「下一次故障」是过度简化,做演练 / 容量规划仅作参考**。

## 七、长尾 / 重尾分布:为什么真实数据这么多

### 7.1 什么叫长尾

**长尾(long tail)/ 重尾(heavy tail)**指的是**尾部衰减比指数慢的分布**——极端值出现概率比高斯 / 指数高得多。

```
高斯尾部:f(x) ~ exp(-x²)         极快衰减
指数尾部:f(x) ~ exp(-x)          快衰减
长尾:    f(x) ~ x^(-α)           按多项式衰减,慢
重尾:    所有 moment 可能不存在  比长尾更狠
```

**对比图(对数纵轴)**:

```
log f(x)
   │
   │\
   │ \         ← 高斯(exp 衰减,曲线)
   │  \
   │  ─\
   │   ─\      ← 指数(exp 衰减,直线在对数线性图上)
   │    ─\
   │      ──\
   │         ──\
   │            ────_____ ← 幂律(直线在 log-log 图上)
   └─────────────────────── log x
```

**两个关键含义**:
1. **极端值远比高斯预期的频繁**(P999 不再是 3σ,可能是 10σ、100σ)
2. **均值可能完全没意义**(被极端值主导,样本越多均值越漂)

### 7.2 常见的长尾家族

| 分布 | PDF 形状 | 典型工程场景 |
| --- | --- | --- |
| **幂律(Pareto)** | `f(x) ~ x^(-α-1)`, x > x_m | 收入、城市人口、网页 PV、词频(Zipf) |
| **log-normal** | log(X) 是高斯 | 文件大小、股价、生物群体大小、API 延迟 |
| **Weibull** | 形状参数控制尾巴 | 寿命数据、风速 |
| **Cauchy** | 重尾到没均值没方差 | 极端建模(物理 / 金融) |

### 7.3 80/20 法则与幂律

经典的「80/20 法则」(Pareto 原则)对应 α=1.16 的幂律:

```
20% 的客户贡献 80% 收入
20% 的城市住 80% 人口
20% 的代码出 80% bug
20% 的文件占 80% 磁盘
```

**幂律的特征**:**对数纵 + 对数横坐标下是直线**——这就是检测幂律的最简单方法:

```python
import numpy as np
import matplotlib.pyplot as plt

# 假设你有一组数据 x(已排序)
ranks = np.arange(1, len(x) + 1)
plt.loglog(ranks, sorted(x, reverse=True))
plt.xlabel('rank (log)')
plt.ylabel('value (log)')
# 如果是直线 → 幂律
```

### 7.4 log-normal:乘性叠加

如果一个变量是**很多正因素的乘积**(而非和),它倾向 log-normal:

```
X = A · B · C · ...

log(X) = log(A) + log(B) + log(C) + ...   ← CLT 让 log(X) 趋高斯
                                          ← 所以 X 趋 log-normal
```

**工程对应**:
- API 延迟:`总延迟 = DNS × 连接 × TLS × HTTP × 服务端 × 网络`(乘性)→ log-normal
- 文件大小:文件大小由多个因素相乘(块数 × 块大小 × 复制因子……)→ log-normal
- 财富:利率复利、多代积累 → log-normal

```python
from scipy.stats import lognorm

# 假设延迟是 log-normal,log 后均值 ln(100ms)、标准差 0.5
# 那么 mean=100ms 处大致是峰值
delay_dist = lognorm(s=0.5, scale=100)

# 关键统计量
print(delay_dist.mean())     # ≈ 113 ms
print(delay_dist.median())   # ≈ 100 ms  ← 比均值小
print(delay_dist.ppf(0.99))  # ≈ 360 ms  ← P99 是中位数的 3.6 倍
print(delay_dist.ppf(0.999)) # ≈ 555 ms  ← P999 是中位数的 5.5 倍
```

**对比高斯**(假设同样 mean、同样 std):

```python
from scipy.stats import norm

mean, std = delay_dist.mean(), delay_dist.std()
fake_normal = norm(loc=mean, scale=std)

print(fake_normal.ppf(0.99))   # ≈ 245 ms  ← 比 log-normal 真实的 360 低 32%
print(fake_normal.ppf(0.999))  # ≈ 304 ms  ← 比 log-normal 真实的 555 低 45%
```

**结果**:**用高斯估同样均值方差的延迟分布,P99 低估 32%、P999 低估 45%**——这就是「the average is a lie」最直接的工程证据。

## 八、分布选错的真实代价

### 8.1 案例:延迟监控

```
真实延迟数据(假设是 log-normal,μ_log=ln(100), σ_log=0.5)

   方法                             P99 估计     真实 P99     误差
   ─────────────────────────────────────────────────────────────
   高斯三 σ                         μ + 3σ        ─           ✗
   ←拿样本 mean 和 std 套高斯       ~ 245 ms     360 ms      -32%
   
   样本分位数(直接看数据 P99)
   ←10000 个样本                    ≈ 360 ms     360 ms       ≈0%
   
   Prometheus histogram_quantile
   ←按 bucket 插值                   ≈ 350 ms     360 ms      -3%
   
   t-digest / HDR histogram
   ←专门为分位数设计的数据结构      ≈ 360 ms     360 ms       ≈0%
```

> **核心直觉**:**直接从样本求分位数 / 用专门的分位数估计算法,远比「假设高斯算 3σ」靠谱**。这就是 Prometheus / Grafana 不用「mean ± std」而是用 `histogram_quantile` 的原因。

### 8.2 案例:风控的极端损失

如果资产收益服从高斯,VaR(在险价值)算出来很「温和」。**但 2008 年金融危机一来,真实数据是肥尾——「百年一遇」的事件十年里发生了三次**。

工程对应:**风控不能只算 P99**,要算「条件期望损失」(CVaR / Expected Shortfall):**「假设损失超过 P99,平均会损失多少」**。这个数对长尾敏感得多。

### 8.3 案例:推荐系统冷启动

**新物品的 CTR 估计**——直接用「曝光/点击比」是频率估计;**但 CTR 全局分布是长尾的**,新物品的「先验」不应该是均匀,而应该是 Beta(对应历史 CTR 分布)。

```python
from scipy.stats import beta

# 历史数据显示全局 CTR 均值 5%,std 2%
# 拟合一个 Beta 先验:matched μ and σ
mu, sigma = 0.05, 0.02
nu = mu * (1 - mu) / sigma**2 - 1
alpha = mu * nu
beta_ = (1 - mu) * nu

prior = beta(alpha, beta_)
# 新物品有 10 次曝光,1 次点击 → 后验
posterior = beta(alpha + 1, beta_ + 9)

print(prior.mean(), posterior.mean())
# 0.05  0.063     ← 没观测时 0.05,有 1/10 后微调到 0.063,而不是 0.1
```

## 九、重尾的统计量陷阱

### 9.1 均值不稳定

**重尾分布下,样本均值收敛极慢**——你需要远超直觉数量的样本才能估准。

```python
import numpy as np

# 用 Cauchy(极重尾)演示
samples = np.random.standard_cauchy(100000)

# 累积均值
cum_mean = np.cumsum(samples) / np.arange(1, len(samples) + 1)
# 永远不会收敛——Cauchy 没有均值

# log-normal(中等重尾)
ln_samples = np.random.lognormal(0, 2, 100000)
cum_mean_ln = np.cumsum(ln_samples) / np.arange(1, len(ln_samples) + 1)
# 收敛,但慢——10000 个样本时仍可能有 50% 偏差
```

### 9.2 方差爆炸

**重尾分布的方差极大、有时甚至发散**。**「3σ 报警」「6σ 质量控制」遇到重尾全失效**。

### 9.3 中位数 / 分位数才是王道

```
样本: [10, 20, 30, 40, 50, 60, 70, 80, 90, 10000]

均值 = 1045
中位数 = 55
P90 = 90
```

**加入一个极端值**:

```
样本: [10, 20, 30, 40, 50, 60, 70, 80, 90, 10000, 1000000]

均值 = 91955    ← 跳了 88 倍
中位数 = 60      ← 几乎不动
P90 = 10000     ← 跳到了原来的极端值
```

> **经验法则**:**重尾数据,优先用中位数 + 分位数描述,均值仅作辅助**。如果一定要算「期望」,用截尾均值(trimmed mean,去掉 top/bottom k%)或 winsorized mean。

## 十、代码:把 scipy.stats 用活

scipy 的所有分布都有统一的 API——记住这套接口,所有分布都能用:

```python
from scipy import stats

# 通用接口(任何分布都一样)
dist = stats.norm(loc=0, scale=1)    # 标准正态

dist.pdf(x)            # 概率密度
dist.cdf(x)            # 累积分布
dist.ppf(p)            # 分位数(CDF 的反函数)
dist.rvs(size=1000)    # 抽样

dist.mean()            # 均值
dist.median()          # 中位数
dist.var()             # 方差
dist.std()             # 标准差

# 经常用的几个分布
stats.norm(loc=μ, scale=σ)         # 高斯
stats.bernoulli(p)                 # 伯努利
stats.binom(n, p)                  # 二项
stats.poisson(mu=λ)                # 泊松
stats.expon(scale=1/λ)             # 指数(注意 scale = 1/λ)
stats.lognorm(s=σ_log, scale=exp(μ_log))   # log-normal
stats.pareto(b=α)                  # Pareto
stats.beta(a, b)                   # Beta(先验常用)
stats.gamma(a, scale=θ)            # Gamma
```

### 10.1 一个完整脚本:对比四种分布的尾部

```python
import numpy as np
from scipy import stats
import matplotlib.pyplot as plt

# 同样的 mean ≈ 100,看尾部有多不同
dists = {
    'Normal':       stats.norm(loc=100, scale=50),
    'Lognormal':    stats.lognorm(s=0.5, scale=100/np.exp(0.125)),  # 调整使 mean=100
    'Exponential':  stats.expon(scale=100),
    'Pareto(2.5)':  stats.pareto(b=2.5, scale=100*1.5/2.5),
}

# 各分布的尾部分位数
for name, d in dists.items():
    print(f"{name:14s}: mean={d.mean():.1f}, "
          f"P50={d.median():.1f}, "
          f"P90={d.ppf(0.9):.1f}, "
          f"P99={d.ppf(0.99):.1f}, "
          f"P999={d.ppf(0.999):.1f}")
```

输出(数字示意,具体值看参数):

```
Normal        : mean=100.0, P50=100.0, P90=164.1, P99=216.3, P999=254.5
Lognormal     : mean=100.0, P50=88.3,  P90=167.6, P99=315.6, P999=487.6
Exponential   : mean=100.0, P50=69.3,  P90=230.3, P99=460.5, P999=690.8
Pareto(2.5)   : mean=100.0, P50=78.6,  P90=139.0, P99=379.3, P999=1198.5
```

**同样的均值,P999 差 5 倍以上**——选错分布的代价在这一行体现得最清楚。

## 十一、工程对应

### 11.1 监控:Prometheus histogram_quantile

Prometheus 监控里测延迟,**不用 mean,而是把数据放进 histogram bucket**:

```yaml
# Prometheus 的 histogram 指标
http_request_duration_seconds_bucket{le="0.005"} 24054
http_request_duration_seconds_bucket{le="0.01"}  33444
http_request_duration_seconds_bucket{le="0.025"} 100392
http_request_duration_seconds_bucket{le="0.05"}  129389
http_request_duration_seconds_bucket{le="0.1"}   133988
...
http_request_duration_seconds_bucket{le="+Inf"}  144320
```

查 P99:

```promql
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))
```

**内部数学**:在 bucket 内做线性插值(假设 bucket 内均匀分布)。**bucket 划分决定了精度**——你的 P99 在哪个数量级,就在那里放细的 bucket。

> **避坑**:**Prometheus histogram 的 bucket 是预定义的,选不好会让 `histogram_quantile` 误差极大**。**别用 default bucket**,**按业务实际延迟范围设计**——SLO 是 200ms,bucket 就要在 100-500ms 之间有足够密度。

### 11.2 高精度分位数:t-digest / HDR histogram

Prometheus histogram 在尾部精度差(bucket 数有限)。**真要精确算 P999、P9999,用专门的数据结构**:

| 数据结构 | 特点 | 工程实现 |
| --- | --- | --- |
| **t-digest** | 自适应 bucket,中位附近粗、尾部细 | Cassandra、ClickHouse |
| **HDR histogram** | log-scale bucket,均匀覆盖几个数量级 | gRPC、HBase |
| **DDSketch** | 相对误差有保证 | Datadog |
| **reservoir sampling** | 简单可并行 | Dropwizard Metrics |

### 11.3 「Average response time is a lie」

[Gil Tene 的著名演讲](https://www.youtube.com/watch?v=lJ8ydIuPFeU)论证为什么报均值响应时间是错的:

```
1000 个请求:
  999 个 10ms
  1 个 10s

平均 = (999*10 + 10000) / 1000 = 20ms

→ 你看到「平均 20ms」,以为系统还行
→ 实际上 0.1% 的用户等了 10 秒,这是个事故级体验
```

**正确报告**:`P50=10ms / P95=10ms / P99=10ms / P99.9=10000ms`——一看就明白尾部出问题了。

### 11.4 推荐 / 广告里的长尾

**网页访问量、用户活跃度、视频播放量都是幂律**:

```
头部 1% 内容/用户 → 占 50%+ 流量
中部 10% → 占 30%
长尾 89% → 占 20%
```

**对系统的影响**:
- 缓存设计:头部要命中,长尾要兜底(LRU 对长尾不友好,LFU + 分层缓存更好)
- 召回排序:头部已经被推爆了,长尾才是差异化机会(YouTube 推荐 50% 流量来自长尾)
- 数据集采样:训练数据如果按曝光采样,长尾物品几乎学不到

| 动作 | 数学上发生了什么 |
