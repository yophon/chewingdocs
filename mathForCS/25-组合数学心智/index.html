    组合数学心智

    工程师的组合数学,基本不需要做"5 选 3 有几种"那种题。真正用到的场景是这些:估算哈希碰撞概率、给布隆过滤器选 m 和 k、设计一致性哈希要多少虚拟节点、为什么 SHA256 是 256 bit 但实际抗碰撞只有 128 bit、HyperLogLog 凭啥用几 KB 内存数十亿个 UV。这些全是组合数学/概率的直接应用,不会算就是抄默认值,出事不知道在哪。

    一句话先记住:组合数学在工程里就是"在你下手前先估一估能不能撞、撞的概率多大、要多大空间"。它不告诉你"答案是多少",它告诉你"这个量级靠不靠谱"。鸽笼说"装不下必然撞",生日攻击说"装得下也大概率撞",布隆过滤器告诉你"代价多大可以容忍多少误判"。

    一、排列 vs 组合:你到底在不在意顺序

    1.1 两个公式

    名称 公式 含义
    排列 P(n,r) n! / (n-r)! n 个里挑 r 个,讲顺序
    组合 C(n,r) n! / (r!(n-r)!) n 个里挑 r 个,不讲顺序

    中间差了一个 r!——因为同样 r 个东西,排成一行 有 r! 种顺序。

    n=4 (a,b,c,d), r=2

P(4,2) = 12:
  ab ba ac ca ad da
  bc cb bd db cd dc

C(4,2) = 6:
  ab ac ad bc bd cd      (ab 和 ba 算同一个)

    1.2 工程里怎么决定用哪个

    场景 用什么 为什么
    抽奖给前 1/2/3 名 排列 第一名和第二名不是一回事
    抽奖随机送 3 人小礼品 组合 三个人之间没有差别
    数据库索引能覆盖几种查询条件组合 组合 (a,b) 和 (b,a) 是同一组列
    比赛 8 进 4 的对阵表 看赛制 有种子位 = 排列;盲抽 = 组合
    API 路由 GET /users/:id/posts/:pid 的 (id,pid) 全组合 排列(笛卡尔积) id=1,pid=2 ≠ id=2,pid=1

    1.3 对称性 C(n,r) = C(n,n-r)

    核心直觉:从 n 个里挑 r 个,等价于从 n 个里把另外 (n-r) 个剔掉。一个意思,所以 C(n,r) 和 C(n,n-r) 必然相等。

    工程里这条经常被忽略:

    • 决策树枚举特征子集时,挑 3 个和不挑 3 个对称,只要枚举到 r ≤ n/2 即可
    • 测试时挑用例子集做覆盖,挑 r 个 fail 等价于挑 n-r 个 pass
    • 算 C(100, 98) 别真去算 100!/(98!·2!),用 C(100,2) = 4950 一行了事
    import math
math.comb(100, 98)   # 4950,Python 3.8+ 自动用对称优化
math.comb(100, 2)    # 4950

    二、鸽笼原理:为什么必然撞

    n+1 个东西放进 n 个盒子,至少一个盒子里有 2 个。这是数学里最朴素的"必然性"原理之一,工程里几乎天天遇到。

    2.1 一张图

    6 个鸽子放 5 个笼:

笼 1   笼 2   笼 3   笼 4   笼 5
[ A ]  [ B ]  [ C ]  [ D ]  [ E ]
                                 → 第 6 只 F 必然要进某个已有鸽子的笼

    2.2 工程对应

    场景 鸽笼应用
    HashMap 容量 = 16,放 17 个 key 必然有冲突链
    IP 地址池(IPv4 32 位)分配给 50 亿设备 必然不够,这就是 NAT/IPv6 出现的原因
    32 位时间戳 + 1 秒精度 必然无法区分同一秒的事件,要加序号(Snowflake)
    哈希分片 256 个,放 257 个 key 必然有分片至少 2 个 key

    避坑:鸽笼只告诉你"必然撞",没告诉你"要多大才大概率撞"。后者是生日攻击。两者搞混会过度配置内存,或者反过来低估冲突。

    2.3 加强版:Σ pigeons / boxes

    如果有 N 个东西放进 K 个盒子,至少有一个盒子里有 ⌈N/K⌉ 个。

    import math
math.ceil(1000 / 7)   # 至少有一天有 ≥ 143 个事件

    工程意义:最坏情况下的负载倾斜下界。一致性哈希设计虚拟节点数,本质就是在控制这个 ⌈N/K⌉ 不能太大。

    三、容斥原理:OR 不等于相加

    |A ∪ B| = |A| + |B| − |A ∩ B|
|A ∪ B ∪ C| = |A| + |B| + |C| − |A∩B| − |A∩C| − |B∩C| + |A∩B∩C|

    为什么减去 ∩?因为 |A| + |B| 把 A∩B 算了两遍,要扣掉一遍。

    3.1 一张图

           A                  B
   ┌─────────┐       ┌─────────┐
   │         │       │         │
   │    ┌────┼───────┼────┐    │
   │    │    │       │    │    │
   │    │    │  A∩B  │    │    │
   │    │    │       │    │    │
   │    └────┼───────┼────┘    │
   │         │       │         │
   └─────────┘       └─────────┘

  |A| 单独算了 A∩B,|B| 又算了一遍
  → 必须减去一次 |A∩B|

    3.2 工程里的容斥

    场景 1:概率求联合事件

    两件独立事件 A、B,至少一件发生的概率:

    P(A ∪ B) = P(A) + P(B) − P(A) · P(B)

    例:接口 A 失败率 1%,接口 B 失败率 2%,调用两个接口的页面失败率不是 3%:

    P(失败) = 0.01 + 0.02 − 0.01·0.02 ≈ 0.0298 ≈ 3%

    差不多 3%——但这是因为概率小;当概率大时差别巨大:

    A 失败率 50%,B 失败率 50%
P(失败) = 0.5 + 0.5 − 0.25 = 0.75   不是 1

    场景 2:SQL OR 查询估算行数

    SELECT COUNT(*) FROM users WHERE city='上海' OR age > 30

    如果直接 |上海用户| + |30 岁以上| 会重复计算 上海且 30 岁以上的人。优化器估算行数时必须扣掉交集,不然会错估全表扫描的代价。

    场景 3:埋点去重

    页面有 3 个按钮,统计"点过任一按钮"的 UV:

    UV(任一) = UV(A) + UV(B) + UV(C)
        − UV(A∩B) − UV(A∩C) − UV(B∩C)
        + UV(A∩B∩C)

    直接相加会严重高估。这就是为什么用 HyperLogLog 去重时,UV(A∪B) 不能简单加。

    四、Stirling 公式:n! 量级估算

    n! ≈ √(2πn) · (n/e)^n

    这个公式不用记,记结论:

    log(n!) ≈ n·log(n) − n + O(log n)

    4.1 什么时候用

    • 信息论:N 个 token 序列的可能数是 |V|^N,但 N 个不同 token 的全排列数是 N!,算香农下界时常常 log(N!) 出现
    • 算法分析:比较排序的下界 log₂(n!) ≈ n·log₂(n),解释了为什么基于比较的排序至少 O(n log n)
    • 统计力学/熵:玻尔兹曼公式 S = k log W,W 是组合数,直接用 Stirling 算

    4.2 一段代码体感量级

    import math

for n in [10, 100, 1000, 10000]:
    real = math.lgamma(n + 1)              # log(n!)
    approx = n * math.log(n) - n           # Stirling 近似
    print(f"n={n:5d}  log(n!)={real:12.2f}  Stirling≈{approx:12.2f}")

    n=   10  log(n!)=       15.10  Stirling≈       13.03
n=  100  log(n!)=      363.74  Stirling≈       360.52
n= 1000  log(n!)=      5912.13  Stirling≈      5907.76
n=10000  log(n!)=     82108.93  Stirling≈     82103.40

    经验法则:n ≥ 100 时 Stirling 已经够用;n ≥ 1000 时误差完全可以忽略。需要更精确的就用 math.lgamma(n+1),Python 自带,数值稳定。

    五、生日悖论:为什么 23 人就撞

    5.1 反直觉的事实

    人数 至少两人同生日的概率
    10 ~12%
    23 ~50%
    30 ~71%
    50 ~97%
    70 ~99.9%

    很多人第一反应是"不该 183 人(365/2)才 50% 吗?" 错。直觉错在"我和谁同"——实际算的是"任意两人同",而 23 个人有 C(23,2) = 253 对组合,每一对都有机会撞。

    5.2 公式与直觉

    n 个人都不同生日的概率:

    P(全不同) = 365/365 · 364/365 · 363/365 · ... · (366-n)/365
P(至少一对同) = 1 - P(全不同)

    核心是平方根因子:在 N 个槽里塞 √N 个东西,撞的概率就接近 50%。

    N = 365
√N ≈ 19.1   ←  实际 23 人 ≈ 50% 吻合（差异在常数因子上）

N = 2^32       √N ≈ 65,536
N = 2^64       √N ≈ 4.3 × 10^9
N = 2^128      √N ≈ 1.8 × 10^19

    5.3 一段代码

    import math

def birthday_collision_prob(n, N=365):
    """n 个生日落在 N 个槽,至少一对相同的概率"""
    # 用 log 防溢出
    log_p_unique = sum(math.log(N - i) - math.log(N) for i in range(n))
    return 1 - math.exp(log_p_unique)

for n in [10, 23, 30, 50, 70]:
    print(f"n={n:3d}  P(collision)={birthday_collision_prob(n):.4f}")

    核心直觉:生日攻击的破坏力 = 平方根缩水。N 位哈希的安全空间是 2^N,但找到一对碰撞只需要枚举 2^(N/2) 次。所有"防碰撞"的安全位数都要除以 2 看。

    六、哈希冲突:为什么 SHA256 实际是 128 bit 抗碰撞

    6.1 两种攻击模型

    攻击 难度 安全位
    第二原像:给定 m,找另一个 m' 使 H(m')=H(m) 2^N N
    碰撞:找任意一对 m, m' 使 H(m)=H(m') 2^(N/2) N/2

    后者就是生日攻击。碰撞难度 = 平方根。

    6.2 常见摘要的"实际安全位"

    摘要 输出位数 抗碰撞强度 现状
    MD5 128 64 已破
    SHA1 160 80 2017 已实证碰撞(SHAttered)
    SHA256 256 128 安全,主流
    SHA512 512 256 长期安全
    SHA3-256 256 128 安全,备用算法

    避坑:别看到 256 bit 就以为有 256 bit 安全。证书签名、Git commit hash、内容寻址(IPFS、Docker layer)都是抗碰撞场景,SHA256 真实强度是 128 bit。Git 用 SHA1 现在已经在迁移到 SHA256 就是因为 80 bit 不够看。

    6.3 UUIDv4 的碰撞概率

    UUIDv4 = 随机 122 bit(16 字节里有 6 bit 做版本/变体标记)。

    经验法则:每秒生成 10 亿个 UUIDv4,85 年内碰撞概率仍 < 50%。日常应用闭眼用,不用担心。

    但反例:如果你用 16 bit 的"短 ID",只能撑 256 个就开始撞。短 ID 设计前必须算一遍。

    def expected_until_collision(N):

    return math.sqrt(math.pi * N / 2)

    七、布隆过滤器:用概率换内存

    7.1 它在做什么

    "这个 key 一定不在集合里"——布隆过滤器只回答这一句,且偶尔会误报"在"(假阳),永远不会假阴。

    插入 key:用 k 个哈希函数算 k 个 bit 位置,全置 1
查询 key:用同样 k 个哈希函数算位置,只要有一个是 0 → 一定不在
                                         全是 1   → 大概率在(也可能假阳)

    7.2 假阳率公式

    设 m 是位数组大小,n 是已插入元素数,k 是哈希函数个数:

    P(假阳) ≈ (1 - e^(-kn/m))^k

    最优 k:

    k_opt = (m/n) · ln(2) ≈ 0.693 · m/n

    最优时假阳率简化为:

    P(假阳) ≈ 0.5^k = (0.6185)^(m/n)

    7.3 工程指南(直接拿去用)

    想要假阳率 每个 key 需要 bits (m/n) 哈希函数个数 k
    1% ~9.6 7
    0.1% ~14.4 10
    0.01% ~19.2 13
    0.001% ~24 17

    经验法则:每个 key 给 10 bit + 7 个哈希,假阳率约 1%。要更低就线性涨。1 亿条记录的布隆过滤器,1% 假阳率只需要约 120 MB 内存。

    import math

def bloom_params(n, p):
    """给定要存的元素数 n 和目标假阳率 p,返回 (m, k)"""
    m = -n * math.log(p) / (math.log(2) ** 2)
    k = (m / n) * math.log(2)
    return int(math.ceil(m)), int(round(k))

print(bloom_params(1_000_000, 0.01))   # (9_585_059, 7)
print(bloom_params(1_000_000, 0.001))  # (14_377_588, 10)

    7.4 何时不该用

    • 要假阴(漏判)零容忍:布隆没问题(它不会假阴)
    • 要假阳零容忍:别用,加二级精确判定(查 DB / Cuckoo Filter)
    • 需要删除元素:标准布隆不能删,要用 Counting Bloom 或 Cuckoo
    • 存的不是固定集合,要动态扩容:用 Scalable Bloom

    避坑:别用布隆过滤器做"判断 user 是否登录"这种语义——那是状态查询,误判一次用户就被踢下线。布隆只适合"先粗筛、再精确判"的两级架构(Bigtable / Cassandra 的 SSTable 查询、CDN 的恶意 URL 检查)。

    八、HyperLogLog:几 KB 数十亿基数

    8.1 核心直觉

    用一连串哈希值前导零的最大长度,反推流里大约有多少不同元素。

    观察 10 个均匀分布的随机数,最大前导零位数 ≈ 3.3
观察 1000 个                              ≈ 9.97
观察 100 万个                             ≈ 19.93

→ 最大前导零位数 ≈ log₂(N)
→ 反过来:N ≈ 2^(最大前导零位数)

    为什么这能行?对均匀的二进制串,前导有 k 个 0 的概率是 2^(-k)。你要"碰巧"看到 k 个 0,你大约要看 2^k 个独立样本。这就是反推的依据。

    8.2 一张图

    hash(item1) = 00010110...   ← 3 个前导零
hash(item2) = 10001010...   ← 0
hash(item3) = 00000101...   ← 5 个前导零
hash(item4) = 01100011...   ← 1 个前导零
hash(item5) = 00100100...   ← 2 个前导零
                           ──
最大前导零数:5  →  估算约 2^5 = 32 个不同元素

    单观察方差很大,真实 HLL 把 hash 前几位当桶号、剩下用来数前导零、做调和平均——把方差压下去。

    8.3 工程参数

    参数 含义 Redis 默认
    桶数 m = 2^p 内存 ≈ m·6 bit p=14 → 16384 桶 ≈ 12 KB
    标准误差 ≈ 1.04/√m p=14 → ≈ 0.81%

    经验法则:12 KB 的 HLL 数 10 亿 UV,误差 < 1%。这就是 Redis 的 PFADD/PFCOUNT 用的参数。代价就是给不出精确值——只给"约 9.94 亿"这种答案。

    九、一致性哈希:虚拟节点要多少才够

    9.1 不带虚拟节点的问题

    普通一致性哈希把节点映射到环上几个点,N 个节点最坏倾斜可能是 log N 倍——节点哈希值挨太近,夹在中间的节点几乎拿不到 key。

    9.2 虚拟节点的数学保证

    把每个物理节点映射成 v 个虚拟节点(v=128 / 256 / 1000 是常见取值)。

    经验法则:v 取 100~200,N 个节点的最大负载偏差大约在 (1 ± O(√(log N / v))) 倍内。N=100, v=100 → 偏差约 ±20%;v=200 → 约 ±15%;v=1000 → 约 ±7%。

    给虚拟节点数的工程建议:

    场景 v 取值 理由
    节点 < 10 256 节点少需要更多虚拟点摊匀
    节点 10 ~ 100 128 默认值
    节点 > 100 64 节点本身已经足够多,v 给少省内存
    Cassandra tokens=256 (默认) 等价 v=256
    Bigtable / DynamoDB 内部用按字典序的范围分片(本质等价) 不直接暴露 v 
    import hashlib

class ConsistentHash:
    def __init__(self, nodes, vnodes=128):
        self.ring = {}
        for n in nodes:
            for i in range(vnodes):
                h = int(hashlib.md5(f"{n}#{i}".encode()).hexdigest(), 16)
                self.ring[h] = n
        self.sorted_keys = sorted(self.ring)

    def get_node(self, key):
        h = int(hashlib.md5(key.encode()).hexdigest(), 16)
        # 二分找第一个 ≥ h 的虚拟节点
        import bisect
        idx = bisect.bisect_right(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

    避坑:虚拟节点数不是越多越好——内存开销 = N·v 个 (hash, node) 对,v=10000、N=1000 就 1000 万项,二分查找虽然 O(log) 但常数会爆。默认 128~256 已经够用,真要再均衡就上加权一致性哈希或 Rendezvous Hashing。

    10.2 何时该退一步

    • 数据规模小(< 万):别套布隆/HLL,直接 set/dict 跑得又快又精确
    • 要精确去重 + 内存够:用 RoaringBitmap 而不是 HLL
    • 节点数极少(< 5):虚拟节点的均衡收益不如直接做加权轮询
    • 碰撞代价是钱:别只看概率,算期望损失 = P(碰撞) · 损失金额

    10.3 局限

    • 公式都基于"哈希均匀"假设——实际 hash function 不均匀(MurmurHash3 还行,Python 默认 hash 在某些版本会被攻击),所有概率分析都要打折
    • 生日攻击是离线攻击的下界——如果有"在线"约束(每秒只能尝试 1 次),实际安全位会高很多
    • HLL 给不了精确值——给监控系统看可以,给账单/审计计数绝对不行
