# 矩阵乘法的代价

`Y = X @ W` 在 NumPy 里就是一行,在 PyTorch 里就是一个 `nn.Linear`。**写起来一行,跑起来吞掉你 95% 的 GPU 时间**。LLM 训练里,前向反向的浮点运算 90% 以上落在矩阵乘上;Attention、FFN、Embedding 投影、KV cache 写回——全是矩阵乘的不同壳子。**这一行代码,理解它的代价,你才能看懂 batch size、hidden_dim、Tensor Core、FlashAttention、Roofline 这些超参与系统设计在抠的到底是什么**。

> 一句话先记住:**矩阵乘法不是"算得快",是"喂得快"**。GPU 上一次 FP16 GEMM 的算力上限是几百 TFLOPS,但 HBM 带宽只能搬几 TB/s——**算力闲着,等数据从显存搬过来**才是日常瓶颈。**所有的 GEMM 优化,从分块、Tensor Core 到 FlashAttention,本质都在一件事:让每搬一次数据,做更多次计算**。这个比值就叫 arithmetic intensity,Roofline 模型上看一眼就知道你卡在哪。

---

## 一、为什么 GEMM 这一篇必须存在

**GEMM = General Matrix Multiplication**——BLAS 标准里的术语,指的是 `C = α·A·B + β·C`(α、β 是标量,A、B、C 是矩阵)。**别被 General 唬住,核心就是矩阵乘**。

为什么单独拎出来讲:

| 场景 | 不懂 GEMM 的人 | 懂 GEMM 的人 |
| --- | --- | --- |
| batch_size 从 32 调到 8,训练慢一倍 | "为什么变慢这么多?" | GEMM 算术密度掉了,GPU 在等数据 |
| hidden_dim 选 1000 还是 1024 | 随便选 | 选 1024,Tensor Core 能用上 |
| FlashAttention 论文 | "看不懂为什么快" | 把 attention 重写成分块 GEMM,少访 HBM |
| `nvidia-smi` 利用率只有 30% | "GPU 没满载" | Memory-bound,不是 compute-bound |
| Strassen 算法 O(n^2.81) | "BLAS 怎么不用?" | 常数大、数值差、不能并行,生产不用 |

**GEMM 是 ML 系统的瓶颈,也是 ML 硬件设计的指北针**——Tensor Core、TPU MXU、华为 NPU 的矩阵单元,全是为 GEMM 设计的。**看懂 GEMM,你才能看懂为什么硬件长成这样**。

## 二、为什么 95% 的 ML compute 都是 GEMM

数一下 Transformer 里的矩阵乘:

```
单层 Transformer 的矩阵乘清单(忽略 LayerNorm / 激活)

  1. Q = X @ W_q          (b, n, d) @ (d, d)  → (b, n, d)
  2. K = X @ W_k          同上
  3. V = X @ W_v          同上
  4. attn = Q @ K^T       (b, h, n, d/h) @ (b, h, d/h, n) → (b, h, n, n)
  5. out = attn @ V       (b, h, n, n) @ (b, h, n, d/h)  → (b, h, n, d/h)
  6. proj = out @ W_o     (b, n, d) @ (d, d)  → (b, n, d)
  7. ffn1 = X @ W_1       (b, n, d) @ (d, 4d) → (b, n, 4d)
  8. ffn2 = act @ W_2     (b, n, 4d) @ (4d, d) → (b, n, d)

非 GEMM 的部分:LayerNorm、softmax、残差加、激活函数
                ——加起来不到 5% 的 FLOPS
```

| 模型 | GEMM 占总 FLOPS 的比例 |
| --- | --- |
| ResNet-50 | ~95%(主要是卷积,卷积底层是 im2col + GEMM) |
| BERT / GPT-2 | ~90% |
| LLaMA / Mistral | ~95%(去掉了 LayerNorm 用 RMSNorm,占比更纯) |
| Diffusion U-Net | ~92%(也是卷积) |

> **核心直觉**:**ML 模型 = 大量 GEMM + 少量逐元素操作**。所以 GPU、TPU 这些"加速器"做的事情,本质是"把 GEMM 加速到极限,顺便支持一下其他"。CUDA 里写一个高效的 GEMM 比你想象的难得多——cuBLAS 团队做了 15 年。

## 三、朴素 GEMM:O(n^3) 的代价

最朴素的实现,大学课本第一周教的:

```python
def naive_matmul(A, B):
    m, k = A.shape
    k2, n = B.shape
    assert k == k2
    C = np.zeros((m, n))
    for i in range(m):
        for j in range(n):
            for l in range(k):
                C[i, j] += A[i, l] * B[l, j]
    return C
```

**复杂度 O(m·n·k)**,方阵就是 O(n^3)。n=1024 时,~10 亿次乘加;n=4096(LLM 常见)时,~700 亿次。

### 为什么 Python 三层循环慢得离谱

```
朴素 Python 三循环 (n=1024):    ~30 秒
NumPy a @ b (调 BLAS):          ~50 毫秒
PyTorch CUDA on A100:           ~2 毫秒
```

**差了 15000 倍**。原因不是算法变了——都是 O(n^3),都做了 n^3 次乘加——是**怎么"喂"数据**的差异:

| 实现 | 算法 | 主要瓶颈 |
| --- | --- | --- |
| Python 三循环 | O(n^3) | Python 解释器本身慢、不利用缓存、无 SIMD |
| NumPy `@` | O(n^3) | 调 OpenBLAS / MKL,SIMD + 多线程 + cache tiling |
| cuBLAS GEMM | O(n^3) | 几千个 CUDA core 并行 + Tensor Core + 共享内存 tiling |

> **避坑**:**永远不要在 Python 里手写矩阵乘**——哪怕只是为了"验证一下结果"。用 `@` / `np.matmul` / `torch.matmul`,差几个数量级。

### Strassen 与亚立方算法

理论上有更快的算法:

| 算法 | 复杂度 | 年份 | 实际用了吗? |
| --- | --- | --- | --- |
| Strassen | O(n^2.807) | 1969 | 几乎不用 |
| Coppersmith-Winograd | O(n^2.376) | 1990 | **永远不用**(galactic algorithm) |
| 当前最佳理论 | O(n^2.371) | 2024 | **永远不用** |

**为什么 BLAS 不用 Strassen?**

```
Strassen 的代价:

  ✗ 常数大:实际要 n > 1000+ 才比朴素 O(n^3) 快
  ✗ 数值稳定差:误差累积比朴素严重(FP32 训练里这是大问题)
  ✗ 不规则递归:不利于 SIMD / GPU 并行
  ✗ 内存访问模式乱:cache 命中率低
  ✗ 实现复杂:写出来的代码 BLAS 团队不愿维护
```

> **经验法则**:**算法复杂度低 ≠ 实际跑得快**。GEMM 是这条经验法则最经典的反例——O(n^3) 的实现,因为 cache 友好、SIMD 友好、并行友好,在所有实际规模上都吊打 O(n^2.81)。**这也是为什么"理论复杂度"在系统圈不是评判标准**。

## 四、缓存与分块:GEMM 真正的战场

### 4.1 内存层次:为什么 cache 决定一切

```
现代 CPU/GPU 的内存层次(数字是大致量级):

           容量      访问延迟       带宽
  寄存器    KB       ~0.3 ns     很快
  L1 cache  ~32 KB    ~1 ns       数 TB/s
  L2 cache  ~256 KB   ~3 ns       1-2 TB/s
  L3 cache  ~32 MB    ~10 ns      数百 GB/s
  DRAM      数十 GB    ~100 ns     ~50 GB/s
  
GPU 这边:
  寄存器    KB       ~1 周期      极快
  共享内存  ~100 KB   ~30 周期     ~10 TB/s
  L2 cache  ~40 MB    ~200 周期    ~5 TB/s
  HBM       ~80 GB    ~500 周期    ~3 TB/s  ← LLM 卡在这
```

**两层之间差一个数量级**。从 HBM 搬一个 byte 到寄存器,够你做几百次浮点运算了——**所以"算"不是瓶颈,"搬"才是**。

### 4.2 朴素 GEMM 的 cache 灾难

回看朴素三循环,假设 A、B 都是 4096×4096 的 FP32 矩阵(64 MB 一个),L1 只有 32 KB:

```
for i:                    ← 遍历 A 的行
  for j:                  ← 遍历 B 的列
    for l:                ← 内积
      C[i,j] += A[i,l] * B[l,j]

内层循环:
  A[i,l] 顺序访问(行优先)→ cache 友好 ✓
  B[l,j] 跨行跳转,每次跳 4 KB → cache 灾难 ✗

每次内积都要把 B 的一整列从 DRAM 搬一遍,B 太大塞不进 L1
→ DRAM 访问爆炸,带宽打满,算力闲置
```

### 4.3 分块(tiling):把大矩阵切成 cache 装得下的小矩阵

```
分块矩阵乘:把 A、B、C 切成 b×b 的小块

  C[I, J] += A[I, L] @ B[L, J]   ← 现在 I, J, L 是块索引,@ 是子块乘法

      B
    ┌───┬───┬───┐
    │   │   │   │
    │   │   │   │
    └───┴───┴───┘
A   ┌───┐ ┌─────────┐  C
    │   │ │   ▓     │
    │   │ │         │  ← 子块 A[I,L] 和 B[L,J] 都装进 L1
    │   │ │         │     算完 ▓ 再换下一对子块
    └───┘ └─────────┘
```

**块大小 b 的选法**:`3·b² · 4 bytes ≤ L1 cache`(三个块同时在 L1 里)。L1=32 KB → b ≈ 50。**所以你看 cuBLAS 内部 tile size 通常 32 / 64 / 128**,不是随便选的。

```python
def blocked_matmul(A, B, block=64):
    m, k = A.shape
    _, n = B.shape
    C = np.zeros((m, n))
    for i in range(0, m, block):
        for j in range(0, n, block):
            for l in range(0, k, block):
                # 子块乘:这一段在 cache 里完成
                C[i:i+block, j:j+block] += A[i:i+block, l:l+block] @ B[l:l+block, j:j+block]
    return C
```

> **核心直觉**:**分块的本质是"加载一次,使用多次"**——把一对子块从 DRAM 搬到 L1 后,反复用它做 b³ 次乘加再换。**这就是 arithmetic intensity 的物理意义**(下一节)。

### 4.4 Cache-oblivious 算法:递归分块

手动调 block size 麻烦——不同机器 L1/L2/L3 大小不一样。**Cache-oblivious GEMM 用递归二分,自动适配所有层级 cache**:

```
recursive_matmul(A, B):
  if A 足够小:
    朴素乘
  else:
    把 A、B 沿最大维度二分
    递归 4 次小问题
    结果加起来
```

递归会自然地把"中间规模的子块"留在中间层 cache(L2/L3),"最小规模的子块"留在 L1——**不需要知道具体 cache 大小,自动最优**。这是 1999 年 Frigo 等人提出的思想,**FFTW 库就靠这个跨平台跑得比手调的 BLAS 还快**。

> **经验法则**:**写底层数值库,你要么知道目标硬件参数手调,要么用 cache-oblivious 思想自适应**。NumPy 的 OpenBLAS 是手调的,JAX 的 XLA 编译器走 auto-tuning 路线,不同哲学。

## 五、BLAS Level 1/2/3:算术密度的鸿沟

BLAS 把线性代数运算分三档,**这三档的性能差异是数量级的**:

| Level | 运算 | 数据量 | 计算量 | 算术密度(FLOPS/Byte) |
| --- | --- | --- | --- | --- |
| Level 1 | 向量-向量(`y += αx`,dot) | O(n) | O(n) | O(1)——很差 |
| Level 2 | 矩阵-向量(`y = Ax`,GEMV) | O(n²) | O(n²) | O(1)——还是很差 |
| Level 3 | 矩阵-矩阵(`C = AB`,GEMM) | O(n²) | O(n³) | O(n)——很好 |

**关键差异在 Level 3**:数据量是 O(n²),计算量却是 O(n³)。**每搬一个 byte,你能做 n 次浮点运算**——n=1024 时,搬一次用一千次,带宽就足够喂饱算力。

### 5.1 GEMV 为什么慢

```
GEMV: y = A @ x
  A: (n, n)   x: (n,)   y: (n,)

  数据量:n² (A) + n (x) + n (y) ≈ n²
  计算量:n²
  算术密度:1 FLOP / Byte
```

**每搬一个 byte 只算一次**——和 DRAM 带宽一样的速度上限。A100 的 HBM 带宽 2 TB/s,FP32 即 500 GFLOP/s——而它的算力上限是 19.5 TFLOPS,**GEMV 只能用到 2.5%**。

> **避坑**:**LLM 推理里 batch=1 的场景,矩阵乘退化成 GEMV**——因为输入只有一行 token。这就是为什么"大模型推理 batch 越大越划算"——把多个用户的请求拼成 batch,GEMV 升级成 GEMM,算术密度起来了,GPU 才能用满。**vLLM 的 continuous batching 解决的就是这个**。

### 5.2 GEMM 为什么快

```
GEMM: C = A @ B
  A: (n, n)   B: (n, n)   C: (n, n)

  数据量:3 n²
  计算量:n³ × 2 = 2n³
  算术密度:2n / 3 FLOP / Byte    ← n 越大越爽

  n=1024:~680 FLOP/Byte
  n=4096:~2700 FLOP/Byte
```

**A100 的算力/带宽比是 ~10**(FP32 算力 19.5 TFLOPS,HBM 2 TB/s,FLOP/Byte 上限约 10),**只要算术密度 ≥ 10,GPU 就能跑满**。GEMM 在 n ≥ 32 时就够了——所以 GPU 几乎可以一直跑满算力。

## 六、Roofline 模型:你在哪条线下面?

**Roofline 是分析 GPU/CPU 性能瓶颈的最重要工具**——一张图告诉你"你这个 kernel 是 compute-bound 还是 memory-bound"。

```
   Performance (GFLOPS)
        ▲
        │              ┌────────────── Compute roof
        │             ╱                (峰值算力)
        │            ╱
        │           ╱
        │          ╱
        │         ╱       ← 你在斜线上 = memory-bound
        │        ╱
        │       ╱            ← 你在水平线上 = compute-bound
        │      ╱
        │     ╱
        │    ╱
        │   ╱  ← 斜率 = 内存带宽
        │  ╱
        │ ╱
        │╱
        └──────────────────────────────► Arithmetic Intensity (FLOP/Byte)
              低                高
              GEMV / 逐元素      GEMM / Conv
```

**斜线**:你的算力被内存带宽卡住,`Performance = Bandwidth × AI`(线性增长)。
**水平线**:你的算力被峰值卡住,达到硬件上限。
**两条线交点**对应的 AI 值,叫 ridge point——**低于 ridge point 你是 memory-bound,要优化数据搬运;高于 ridge point 你是 compute-bound,要优化算力利用率**。

### A100 的 Roofline 数字

| 指标 | 数值 |
| --- | --- |
| FP32 峰值算力 | 19.5 TFLOPS |
| FP16 / Tensor Core 峰值算力 | 312 TFLOPS |
| HBM 带宽 | 2 TB/s |
| FP32 ridge point | ~10 FLOP/Byte |
| FP16 Tensor Core ridge point | ~156 FLOP/Byte |

**关键点**:**用了 Tensor Core 后,ridge point 飙升到 ~156**——意味着 AI 必须 ≥ 156 才能用满 Tensor Core。**所以小 batch、小 hidden_dim 用 Tensor Core 反而吃亏**——算力上去了,但你喂不饱。

> **经验法则**:**调优一个 kernel 前,先算它的 arithmetic intensity**。算出来 < ridge point,你优化算力是白费——优化数据搬运(减少 HBM 访问、复用 cache、kernel fusion)才有用。**这就是 FlashAttention 和 LayerNorm fusion 这一类优化的理论依据**。

## 七、GPU 为什么适合 GEMM

GPU 的设计哲学:**几千个简单核心 + 巨大带宽 + 牺牲单核灵活性**。这个设计**只对一类 workload 友好**:

```
GPU 喜欢的 workload:
  ✓ 数据并行——同样的运算应用到大量数据
  ✓ 算术密度高——每搬一次数据多算几次
  ✓ 数据局部性高——访问模式可预测
  ✓ 分支少——SIMT 不喜欢 if-else

GPU 不喜欢的 workload:
  ✗ 序列依赖(RNN 每步等上一步)
  ✗ 稀疏 / 不规则访问(图算法)
  ✗ 大量分支(决策树)
```

**GEMM 完美命中所有"喜欢"的特性**——这就是为什么 GPU 是 ML 的天选硬件。**Transformer 之所以替代 RNN 占领 NLP,部分原因就是它把序列依赖变成了大矩阵乘**(self-attention 的 `Q @ K^T`),GPU 友好。

### Tensor Core:GPU 里的 GEMM 专用单元

NVIDIA 从 V100 开始引入 Tensor Core,**就是为了再榨一遍 GEMM 的性能**:

```
Tensor Core 一周期做的事:
  D = A @ B + C
  其中 A 是 16×16 的 FP16/BF16 矩阵
       B 是 16×16 的 FP16/BF16 矩阵
       C, D 是 16×16 的 FP32 矩阵

  一次完成 16×16×16 = 4096 次乘加
  vs 普通 CUDA core 一次只做 1 次 FMA
```

**4 个数量级的差距**——这就是为什么 A100 FP16 算力(312 TFLOPS)比 FP32(19.5 TFLOPS)高 16 倍。

| 数据类型 | A100 算力 | H100 算力 | 适合场景 |
| --- | --- | --- | --- |
| FP32 | 19.5 TFLOPS | 67 TFLOPS | 老科学计算 |
| TF32 | 156 TFLOPS | 989 TFLOPS | 训练默认 |
| FP16 / BF16 | 312 TFLOPS | 1979 TFLOPS | 训练 / 推理 |
| INT8 | 624 TOPS | 3958 TOPS | 量化推理 |
| FP8 (H100) | — | 3958 TFLOPS | 训练 / 推理(2024+) |

> **避坑**:**Tensor Core 要求 m, n, k 都是 16 的倍数**(FP16)/8 的倍数(TF32)。`hidden_dim=1000` 比 `hidden_dim=1024` 慢一截——前者凑不齐 Tensor Core 的块,fallback 到普通 core。**这就是"hidden_dim 选 64/128 倍数"这个经验法则的由来**。

## 八、代码:看一眼性能差距

```python
import torch
import time

device = "cuda"
n = 4096

# 准备矩阵
A = torch.randn(n, n, device=device, dtype=torch.float32)
B = torch.randn(n, n, device=device, dtype=torch.float32)

# warmup
for _ in range(3):
    _ = A @ B
torch.cuda.synchronize()

# FP32 GEMM
t = time.time()
for _ in range(10):
    C = A @ B
torch.cuda.synchronize()
fp32_time = (time.time() - t) / 10
fp32_flops = 2 * n**3 / fp32_time / 1e12
print(f"FP32: {fp32_time*1000:.2f} ms, {fp32_flops:.1f} TFLOPS")

# FP16 GEMM(吃 Tensor Core)
A_h = A.half()
B_h = B.half()
torch.cuda.synchronize()
t = time.time()
for _ in range(10):
    C = A_h @ B_h
torch.cuda.synchronize()
fp16_time = (time.time() - t) / 10
fp16_flops = 2 * n**3 / fp16_time / 1e12
print(f"FP16: {fp16_time*1000:.2f} ms, {fp16_flops:.1f} TFLOPS")

# 故意选个不对齐的 size,看 Tensor Core 失效
n2 = 4097
A2 = torch.randn(n2, n2, device=device, dtype=torch.float16)
B2 = torch.randn(n2, n2, device=device, dtype=torch.float16)
torch.cuda.synchronize()
t = time.time()
for _ in range(10):
    C = A2 @ B2
torch.cuda.synchronize()
print(f"FP16 (n=4097): {(time.time()-t)/10*1000:.2f} ms  ← 比 4096 慢")
```

A100 上的典型输出:

```
FP32: 11.3 ms, 12.1 TFLOPS    ← 接近 19.5 TFLOPS 上限的 60%
FP16: 0.95 ms, 144 TFLOPS     ← Tensor Core 在干活,接近 312 TFLOPS 上限的 46%
FP16 (n=4097): 1.7 ms         ← 不对齐,慢了 80%
```

**FP16 比 FP32 快 12 倍**——主要靠 Tensor Core。**4097 比 4096 慢 80%**——Tensor Core 用不上。

### 看 GEMM 是 memory-bound 还是 compute-bound

```python
# 小 batch 推理:GEMV
x = torch.randn(1, n, device=device, dtype=torch.float16)
W = torch.randn(n, n, device=device, dtype=torch.float16)
torch.cuda.synchronize()
t = time.time()
for _ in range(100):
    y = x @ W
torch.cuda.synchronize()
gemv_time = (time.time() - t) / 100
gemv_ai = 2 * n / 3  # 算术密度
print(f"GEMV: {gemv_time*1000:.3f} ms, AI={gemv_ai:.1f} FLOP/Byte  ← memory-bound")

# 大 batch 训练:GEMM
X = torch.randn(64, n, device=device, dtype=torch.float16)
torch.cuda.synchronize()
t = time.time()
for _ in range(100):
    Y = X @ W
torch.cuda.synchronize()
gemm_time = (time.time() - t) / 100
gemm_ai = 2 * 64 * n / (64*n + n*n + 64*n) / 2  # 简化版
print(f"GEMM (b=64): {gemm_time*1000:.3f} ms")
print(f"GEMM/GEMV 时间比: {gemm_time/gemv_time:.1f}x  ← 远小于 64x,说明 GEMV 浪费")
```

典型输出:

```
GEMV: 0.27 ms, AI=2731 FLOP/Byte
GEMM (b=64): 0.95 ms
GEMM/GEMV 时间比: 3.5x  ← 64x 算力换来 3.5x 时间,效率提升 18 倍
```

> **核心直觉**:**batch=64 的 GEMM 用了 64 倍的算力,只用了 3.5 倍的时间**——剩下的 60 倍来自"数据复用率提升,从 memory-bound 升级到 compute-bound"。**这就是为什么大模型推理一定要 batch**。

## 九、FlashAttention:把 Attention 重写成分块 GEMM

**Attention 是个特殊的 GEMM 链**:

```
S = Q @ K^T          # GEMM 1
P = softmax(S)       # 逐元素 + 行求和
O = P @ V            # GEMM 2

朴素实现:
  S 中间结果 (n, n) 写回 HBM
  读回来算 softmax,P 写回 HBM
  读回来算 O @ V

总 HBM 访问:O(n²) 次读写 S, P
当 n=8192 时,S 是 256 MB 的中间矩阵
```

**朴素实现的瓶颈不是算力,是 HBM 带宽**——softmax 这一步 AI 极低(逐元素操作),GPU 在等 S 从 HBM 搬过来。

### FlashAttention 的核心想法

**分块 + 把 softmax 融进 GEMM,不写中间矩阵 S 到 HBM**:

```
for 每个 Q 的 tile (装进 SRAM):
  for 每个 K, V 的 tile (装进 SRAM):
    在 SRAM 里算 Q_tile @ K_tile^T   ← 部分 attention scores
    用 online softmax 更新累加  ← 不需要看到完整一行 S
    累加到 O_tile
  写 O_tile 回 HBM
```

**关键技巧**:**online softmax**——softmax 需要看到一整行的最大值才能数值稳定,FlashAttention 用一个递推公式,边算边更新最大值,**不需要把整行 S 都存下来**。

| 指标 | 朴素 Attention | FlashAttention |
| --- | --- | --- |
| HBM 访问 | O(n²) | O(n) |
| 中间矩阵 S | 显式存储,n² | 不存,只在 SRAM 临时算 |
| 速度(n=8192,A100) | ~慢 | 快 2-4 倍 |
| 显存占用 | 高 | 大幅降低 |
| 数学结果 | 完全等价 | 完全等价 |

> **核心直觉**:**FlashAttention 的速度提升不是来自算法变好(FLOPS 一样多),而是来自访 HBM 的次数减少**。**这就是 Roofline 模型告诉你的事**——把 memory-bound 的 kernel,通过 fusion 提到 compute-bound 的范围。**LayerNorm fusion、SwiGLU fusion 都是同一思路**。

## 十、工程对应与调参直觉

### 10.1 库与硬件的对应关系

| 层 | 库/硬件 | 干的事 |
| --- | --- | --- |
| 框架 | PyTorch / JAX | 调度 op,做 autograd |
| 算子库 | cuBLAS / cuDNN / oneDNN | 提供高度优化的 GEMM / conv |
| 模板库 | CUTLASS | NVIDIA 开源 GEMM 模板,允许定制 fusion |
| 编译器 | XLA / TVM / Triton | 自动 kernel fusion + auto-tuning |
| 硬件 | Tensor Core / AMX / TPU MXU | GEMM 专用电路 |

**写训练代码:99% 时间用 `torch.matmul` / `nn.Linear`** ——底下走 cuBLAS。**写新 kernel(FlashAttention 风格):用 Triton 或 CUTLASS**。

### 10.2 训练调参的"GEMM 友好"清单

| 超参 | 友好选法 | 为什么 |
| --- | --- | --- |
| batch size | 大(显存允许下) | 算术密度上去,GPU 利用率高 |
| hidden_dim | 64 / 128 倍数 | Tensor Core 对齐 |
| seq_len | 64 / 128 倍数 | Attention 的 GEMM 对齐 |
| num_heads | 让 head_dim ≥ 64 | 单个 head 的 GEMM 大小够 Tensor Core |
| 数据类型 | BF16 训练,FP8 实验 | Tensor Core 算力翻倍 |
| 梯度累积 | 显存不够时用 | 等价大 batch,但每步算力利用率不变 |

### 10.3 排错:GPU 利用率低时怎么看

```
nvidia-smi 看 GPU-Util:
  90%+:GPU 在算,看 SM 占用率(nsys/ncu profile)
       └─ SM 占用低 → kernel 太小或并行度不足 → 加 batch
       └─ SM 占用高 → 真的 compute-bound
  50-90%:典型的 ML 训练,搬数据 + 算交替
  <50%:bottleneck 在外面
       └─ DataLoader 不够快(CPU 预处理慢)
       └─ 通信慢(多卡训练 NCCL bottleneck)
       └─ Python 层 overhead(small op 太多)
```

**实际工程里,大多数训练慢不是 GEMM 本身慢**——是 GEMM 之间的 overhead(launch、同步、内存分配)、DataLoader、通信。**用 `torch.profiler` 或 `nsys` 看一遍,十次有八次是这些**。

### 10.4 GEMM 的数值稳定

| 场景 | 风险 | 应对 |
| --- | --- | --- |
| FP16 训练 | 累加溢出 → loss=NaN | mixed precision:乘用 FP16,累加用 FP32 |
| 大矩阵 + Strassen | 误差累积 | 不用 |
| Attention 的 softmax | exp 溢出 | 减最大值 / log-sum-exp |
| 量化推理 | INT8 溢出 / 精度损失 | per-tensor / per-channel 量化校准 |

> **避坑**:**`torch.matmul(a, b)` 在 FP16 下,中间累加默认是 FP32**——不要自己写 `torch.float16` accumulator,会炸。`torch.cuda.amp.autocast` 帮你处理这一切。

### 10.5 局限:GEMM 不是万能的

| 场景 | GEMM 不擅长 | 替代 |
| --- | --- | --- |
| 稀疏矩阵(>90% 零) | 算空气 | sparse-dense matmul / SpMM |
| 极小矩阵(n<32) | Tensor Core 用不上 | batched-GEMM,把多个小 GEMM 拼大 |
| 卷积大 kernel | im2col 占内存 | Winograd / FFT 卷积 |
| 在线推理 batch=1 | GEMV 退化 | continuous batching / speculative decoding |
| 长序列 attention | n² 太大 | FlashAttention / Linear Attention / Mamba |

**Mamba(状态空间模型)和 Linear Attention 这一类工作,本质都在挑战"attention 必须是 n² GEMM" 这件事**——把它换成 O(n) 的 scan,牺牲一点表达力换算力。**这条路还在演化中,2026 年还没定论**。
