# 微服务与服务网格

单体应用写到几十万行后会撞上几堵墙：编译慢、发布风险大、不同模块耦合越来越紧、团队没法并行迭代。**微服务（Microservices）** 把一个大应用拆成一组各自部署、各自演进的小服务——但拆完之后服务之间的**注册、调用、熔断、配置、追踪**全要重做一遍。

这一章讲两套主流方案：**Spring Cloud**（应用框架层解决）和 **Service Mesh**（基础设施层解决）。

---

## 一、单体 vs 微服务

| 维度 | 单体 | 微服务 |
| --- | --- | --- |
| 部署 | 一个包 | N 个独立服务 |
| 数据 | 一个库 | 每服务独立库 |
| 通信 | 函数调用 | RPC / HTTP / MQ |
| 扩展 | 整体扩 | 按服务独立扩 |
| 故障域 | 一挂全挂 | 局部隔离 |
| 复杂度 | 业务复杂度 | 业务 + 分布式复杂度 |

> 经验法则：**人少于 20 别上微服务**。微服务的运维与排障成本是平方级增长的。

---

## 二、微服务必须解决的 8 个问题

```
1. 服务注册与发现   服务多了找不到对方
2. 负载均衡          一个服务多副本，怎么选
3. 配置中心          几十个服务的配置统一改
4. 熔断 / 限流       下游挂别拖死上游
5. 分布式事务        跨服务的"要么都成功要么都失败"
6. 链路追踪          一个请求经过 10 个服务，错在哪
7. 网关              统一鉴权、限流、路由
8. 日志聚合          一次调用的日志散在 10 台机器上
```

Spring Cloud 给的是"每个问题一个 starter"的方案，Service Mesh 给的是"基础设施层透明解决"的方案。

---

## 三、Spring Cloud 全景

| 能力 | 老选项 | 现代推荐 |
| --- | --- | --- |
| 注册中心 | Eureka | **Nacos / Consul** |
| 配置中心 | Spring Cloud Config | **Nacos Config / Apollo** |
| 客户端调用 | Feign | **OpenFeign** |
| 负载均衡 | Ribbon | **Spring Cloud LoadBalancer** |
| 熔断限流 | Hystrix（停更） | **Resilience4j / Sentinel** |
| 网关 | Zuul | **Spring Cloud Gateway** |
| 链路追踪 | Sleuth + Zipkin | **Micrometer Tracing + OTel** |

### Nacos 注册中心示例

```yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: nacos:8848
```

启动后 Nacos 控制台立刻能看到 `order-service` 注册进来。

### OpenFeign 跨服务调用

```java
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/users/{id}")
    User getById(@PathVariable("id") Long id);
}

@Service
class OrderService {
    @Autowired UserClient userClient;

    public OrderDetail buildDetail(Long orderId) {
        Order o = orderDao.find(orderId);
        User u = userClient.getById(o.getUserId());   // 像调本地方法
        return new OrderDetail(o, u);
    }
}
```

调用过程：Feign → LoadBalancer 从 Nacos 拿 user-service 实例列表 → 轮询/权重选一个 → HTTP 发出。

### Resilience4j 熔断 + 限流 + 重试

```java
@CircuitBreaker(name = "userSvc", fallbackMethod = "fallbackUser")
@RateLimiter(name = "userSvc")
@Retry(name = "userSvc")
public User getUser(Long id) {
    return userClient.getById(id);
}

private User fallbackUser(Long id, Throwable t) {
    return User.unknown(id);   // 降级返回兜底值
}
```

```yaml
resilience4j:
  circuitbreaker:
    instances:
      userSvc:
        failure-rate-threshold: 50      # 失败率超 50% 打开
        sliding-window-size: 20
        wait-duration-in-open-state: 30s
```

## 四、Service Mesh 是什么

Spring Cloud 把治理逻辑写进**应用代码**——多语言团队就要给每种语言实现一遍。

Service Mesh 的思路：**给每个 Pod 注入一个 sidecar 代理（Envoy）**，所有进出流量都被它劫持，治理能力做在代理里，应用代码什么都不用改。

```
┌─────────────────────────────────────┐
│ Pod A                               │
│ ┌──────────┐    ┌────────────┐     │
│ │ App      │───▶│ Envoy(sidecar) │──▶ network
│ └──────────┘    └────────────┘     │
└─────────────────────────────────────┘
                                       │
                                       ▼
┌─────────────────────────────────────┐
│ Pod B                               │
│ ┌────────────┐    ┌──────────┐     │
│ │ Envoy      │◀───│ App      │     │
│ └────────────┘    └──────────┘     │
└─────────────────────────────────────┘
```

主流实现：

| Mesh | 特点 |
| --- | --- |
| **Istio** | 功能最全，复杂度最高 |
| **Linkerd** | 轻量、Rust 写的 sidecar，性能好 |
| **Consul Connect** | HashiCorp 生态 |
| **Cilium Mesh** | eBPF，无 sidecar |

## 五、Istio 能力速览

```yaml
# 把 50% 流量打到 v2
apiVersion: networking.istio.io/v1
kind: VirtualService
metadata:
  name: user-svc
spec:
  hosts: [user-service]
  http:
    - route:
        - destination: { host: user-service, subset: v1 }
          weight: 50
        - destination: { host: user-service, subset: v2 }
          weight: 50
```

```yaml
# mTLS：服务间自动加密
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata: { name: default, namespace: prod }
spec:
  mtls: { mode: STRICT }
```

得到的能力：

- 灰度发布、金丝雀、A/B
- mTLS 服务间双向 TLS（零信任）
- 故障注入（人为延迟 / 错误，做演练）
- 全链路指标 + tracing 自动采集
- 细粒度授权策略

## 六、Spring Cloud vs Service Mesh

| 维度 | Spring Cloud | Service Mesh |
| --- | --- | --- |
| 侵入性 | 高（代码改造） | 零（sidecar 透明） |
| 多语言 | 难 | 天然支持 |
| 性能开销 | 低（同进程） | 多一跳代理（~1ms） |
| 升级 | 改代码 + 重发 | 改 CRD 配置 |
| 学习曲线 | Java 生态熟则低 | K8s + Envoy 概念多 |

> 经验法则：**单语言（Java）小团队用 Spring Cloud；多语言或想做零信任 + 灰度的中大型用 Mesh**。也可以混用——Spring Cloud 做业务编排，Mesh 做安全和观测。

## 七、分布式事务（最难的那块）

跨服务事务在工业界**没有银弹**，常见方案：

| 方案 | 思路 | 适用 |
| --- | --- | --- |
| 2PC / XA | 两阶段提交 | 强一致，性能差，少用 |
| TCC | Try-Confirm-Cancel | 业务侵入大但精细 |
| Saga | 长事务 + 补偿 | 流程长、最终一致 |
| **本地消息表** | 业务 + 消息同库一事务 | 实现简单，最常用 |
| **事件驱动 + 幂等** | MQ 兜底 | 配合 Saga 最常见 |

> 经验法则：**先用本地消息表 + MQ 实现最终一致**，需要强一致时再考虑 TCC，慎用 XA。

## 八、新手最容易踩的坑

1. **过早拆服务**：拆完才发现两个服务永远一起改——合并比拆开难十倍
2. **共用一个数据库**：拆了服务没拆库，等于没拆，反倒多了 RPC 开销
3. **没有链路追踪**：一个请求穿 10 个服务，没 traceId 就别想排错
4. **没设超时**：默认无穷大超时 + 重试 = 雪崩生成器
5. **配置散落各处**：每个服务自己改 yaml，配置中心不上等于没上
6. **直接上 Istio**：先用 Spring Cloud 把分布式概念跑通，再决定要不要 Mesh
