# 配置中心与注册中心

32 章微服务讲了"为什么要拆"和"拆完之后怎么协同",但落到工程上,**两个最先要建的中间件**没单独讲清:

- **配置中心**:让配置脱离代码,运行时动态变更
- **注册中心**:让服务找到彼此,不用写死 IP

这两个东西看起来基础,但搭不好,微服务体系上层全是空中楼阁。

---

## 一、为什么要它们

### 配置中心解决的问题

```
单体时代:                     微服务时代:
- 一份 application.yml         - 几十个服务、各几个环境
- 改完重启                     - 数据库密码、限流阈值、
                                 开关、灰度白名单到处都有
                               - 每个服务自己一套 yaml = 灾难
```

具体痛点:

1. **多环境差异**:dev / staging / prod 的配置不一样,塞进同一份代码靠 profile 凑活
2. **改配置要重启**:线上调整限流阈值还要发版,黄花菜都凉了
3. **敏感信息**:数据库密码、API key 不能进 Git
4. **配置审计**:谁、什么时候改了什么 → 出锅查不到

### 注册中心解决的问题

```
单体:           微服务:
- 直接调函数     - 服务 A 要调 B,B 实例随时增减
                - 写死 IP → 扩容/迁移就坏
                - DNS 也行,但更新慢、做不了健康检查
                - 需要"实时、可靠、带健康检查"的地址簿
```

## 二、主流配置中心对比

| 产品 | 出身 | 强项 | 弱项 |
| --- | --- | --- | --- |
| **Nacos** | 阿里开源 | 一体化(配置+注册)、中文友好、UI 完整 | 大版本兼容性偶尔有坑 |
| **Apollo** | 携程开源 | 灰度发布、权限、审计完整 | 部署相对重、UI 偏老 |
| **Consul** | HashiCorp | 多数据中心、一体化(KV+服务+健康) | 国内用得少、UI 不如 Nacos |
| **Spring Cloud Config** | Spring 官方 | 与 Spring 生态贴合 | 默认基于 Git,不适合大规模 |
| **etcd** | CoreOS | 强一致(Raft)、K8s 底座 | 偏裸 KV,需要上层封装 |
| **AWS Parameter Store / SSM** | AWS | 云原生集成 | 锁死 AWS |

> 经验法则:**国内首选 Nacos,中大型企业可选 Apollo,云原生 K8s 体系直接用 ConfigMap + External Secrets**。

## 三、Nacos 配置中心实战

### 配置层级

```
Namespace(环境)        dev / staging / prod
  └─ Group              业务线 / 团队 / DEFAULT_GROUP
      └─ Data ID        application.yml / db.yml / redis.yml
          └─ Content    具体配置内容
```

设计原则:**Namespace 隔环境,Group 隔业务,Data ID 隔模块**。

### Spring Boot 接入

```yaml
# bootstrap.yml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      config:
        server-addr: nacos:8848
        namespace: prod                 # 环境
        group: ORDER                    # 业务
        file-extension: yaml
        shared-configs:                 # 共享配置(数据库、Redis)
          - data-id: common-db.yaml
            refresh: true
```

```java
@RestController
@RefreshScope                            // 配置变更后,Bean 自动重建
public class FeatureController {
    @Value("${feature.payment-v2-enabled:false}")
    private boolean paymentV2;
}
```

**`@RefreshScope`** 是动态刷新的关键:Nacos 推送变更 → Spring 销毁并重建带这个注解的 Bean → 新值生效,**不用重启**。

## 四、配置变更的安全姿势

```
方案 A:全量推送(危险)
   改一个值 → 全部实例同时拿到 → 出错全挂

方案 B:灰度发布(推荐)
   改一个值 → 标记一批机器先生效 → 观察指标 → 全量
```

Apollo 内置灰度,Nacos 需要业务侧配合(比如配置 key 里带机器 IP / 容器 hostname 路由)。

> ⚠️ **配置变更 = 一次发布**。把它当代码变更对待:
> - 改配置走 PR 评审(配置仓库化)
> - 配置变更触发告警通知群
> - 关键配置(限流、熔断阈值)留快照,可一键回滚

## 五、敏感配置怎么放

**不要把数据库密码 / 三方 token 放 Nacos 明文**——任何能读配置中心的人都能看到。

主流方案:

| 方案 | 说明 |
| --- | --- |
| **Vault** | HashiCorp,业界标准,动态密钥 + 租约自动轮换 |
| **AWS Secrets Manager / 阿里 KMS** | 云上首选,集成 IAM |
| **Sealed Secrets / SOPS** | GitOps 场景,加密后入库 |
| **Nacos + 自加密** | 配置内容用 KMS 加密后再存 Nacos,启动时解密 |

> 经验法则:**配置中心管"普通配置",密钥管理系统管"机密"**——分层管理是底线。

## 六、注册中心:服务发现的两种姿势

```
客户端发现(client-side discovery):
  Service A → 先查注册中心 → 拿到 B 实例列表 → 自己选一个调
  代表:Eureka、Nacos、Consul + Ribbon/LoadBalancer

服务端发现(server-side discovery):
  Service A → 调一个固定 LB → LB 自己查注册中心 → 转发
  代表:K8s Service、Envoy/Istio、Spring Cloud Gateway
```

| 维度 | 客户端发现 | 服务端发现 |
| --- | --- | --- |
| 实现复杂度 | 客户端要带 SDK | LB 集中处理,客户端不感知 |
| 多语言友好 | 弱(每语言要 SDK) | 强(只要会 HTTP) |
| 性能 | 少一跳 | 多一跳 LB |
| K8s 时代 | 渐衰 | 主流 |

> 经验法则:**纯 K8s 体系直接用 Service + DNS,跨集群 / 混合云用 Nacos / Consul 做联邦**。

## 七、主流注册中心对比

| 产品 | 一致性 | 健康检查 | 适用规模 | 备注 |
| --- | --- | --- | --- | --- |
| **Nacos** | AP(默认)/CP 可切 | 心跳 + TCP/HTTP | 中大型 | 一体化配置+注册 |
| **Eureka** | AP | 心跳 | 中型 | Spring Cloud 老牌,已半弃用 |
| **Consul** | CP(Raft) | 多种探测 | 中大型 | 多数据中心强 |
| **etcd** | CP(Raft) | TTL 续约 | 大型 / 基础设施 | K8s 底层用的就是它 |
| **ZooKeeper** | CP(ZAB) | session/ephemeral 节点 | 中大型 | 老牌,Dubbo / Kafka 早期标配 |
| **K8s Service / CoreDNS** | 集群本地 | Probe | 任意 | K8s 原生,默认不出集群 |

### CAP 选择:AP 还是 CP

- **AP**(Availability + Partition tolerance):分区时仍可读写,可能短暂返回老数据。**注册中心通常选 AP**——注册的实例少几个无所谓,注册中心宕了,业务就全停了。
- **CP**(Consistency + Partition tolerance):分区时宁可不可用也要一致。**配置中心可选 CP**——配置不一致比配置不可用危险。

> Nacos 默认注册中心走 AP、配置中心走 CP,把两个场景区分开,这是它的设计巧思。

## 八、健康检查与下线

注册不是终点,**保持注册的"实时性"**才是关键:

```
正常:
   服务实例每 5s 心跳 → 注册中心更新 lastBeat

异常情况:
   - 心跳 15s 收不到 → 标记不健康(暂时摘流量)
   - 心跳 30s 收不到 → 从列表删除
   - 实例主动 deregister(优雅停机)→ 立刻删
```

优雅停机的关键步骤:

```java
@PreDestroy
public void shutdown() {
    namingService.deregisterInstance(serviceName, ip, port); // 1. 先反注册
    Thread.sleep(15_000);                                    // 2. 等流量切走
    server.shutdown();                                       // 3. 再关进程
}
```

**没第二步**,流量还在涌进来时进程就死了 → 用户看到 502。Spring Boot 的 `server.shutdown=graceful` + K8s 的 `preStop` hook 配合做这件事。

## 九、与 K8s 的关系

```
传统体系:                    K8s 体系:
- Nacos / Eureka 注册中心     - 内置 Service + Endpoints
- 业务自己心跳                - kubelet + readinessProbe
- DNS 配合                    - CoreDNS 自动解析
                              - kube-proxy iptables/IPVS 转发
```

**K8s 上还需要 Nacos 注册中心吗?**

- **纯 K8s 内、Java/Go 多语言**:不需要,Service+DNS 够用
- **跨集群 / 跨云 / 混合部署**:需要,Nacos 当全局服务地图
- **存量 Spring Cloud 系统迁移过渡**:需要,逐步收敛到 Service Mesh

## 十、配置 + 注册一体化的 Nacos 接入完整图

```
┌─────────────────┐
│   Nacos Cluster │  ◀──── 心跳/配置变更
└─────────────────┘
   ▲      ▲     ▲
   │心跳  │心跳 │心跳
   │      │     │
[svc-A][svc-B][svc-C]
   │
   │ 1. 启动:加载共享配置 + 自身配置
   │ 2. 启动:把自己注册进去
   │ 3. 调 svc-B:从 Nacos 拿 B 实例 → LoadBalancer 选一个
   │ 4. 配置变更:推送过来 → @RefreshScope 重建 Bean
   ▼
[ Spring Cloud LoadBalancer ]
```

Spring Cloud OpenFeign + Nacos 的实际调用代码:

```java
@FeignClient("payment-service")  // 服务名,不写 IP
public interface PaymentClient {
    @PostMapping("/pay")
    PayResp pay(@RequestBody PayReq req);
}
```

底层会自动走 Nacos 找实例 + LoadBalancer 选一个 + Resilience4j 容错。

## 十一、常见踩坑

1. **配置中心放数据库密码明文**:换密钥管理系统
2. **配置变更没有审计 / 回滚**:错改了一个限流值,服务雪崩,找不到是谁改的
3. **`@RefreshScope` 用在 Singleton 关键 Bean 上**:重建时短暂不可用,关键路径要慎用
4. **服务下线没优雅停机**:流量打到死实例 → 大量 502
5. **Nacos 单节点部署**:Nacos 挂 = 服务发现挂 = 所有调用挂,生产至少 3 节点
6. **Namespace 用错**:dev 服务连了 prod 配置,事故现场
7. **Group / Data ID 命名混乱**:几个月后没人知道哪个配置归谁
8. **配置改完没 reload**:写了 `@Value` 但没 `@RefreshScope`,值是启动时快照
9. **K8s 上又叠一套 Nacos**:K8s Service 已经够,叠了反而维护两套发现
10. **健康检查只看进程在不在**:进程没死但 DB 连不上 → 还在收流量。检查必须是端到端的 readiness
