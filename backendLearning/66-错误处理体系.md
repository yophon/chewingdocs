# 错误处理体系

50 章测试体系把"代码怎么验对"讲清楚了——但出错的时候怎么处理、怎么传递、怎么告诉前端 / 告诉用户、怎么让排查变快,**几乎所有项目都很草率**。

错误处理是**反映工程团队成熟度最直接的镜子**——错误体系混乱的项目,排查事故时所有人在群里发"这个错是什么意思?"。这章把这一摊事讲透。

---

## 一、错误的"四种受众"

```
同一个错误,要分别"翻译"给四种人:

1. 终端用户       → 友好提示,带建议下一步("请稍后重试")
2. 前端开发        → 结构化 errorCode + 字段定位
3. 后端 / 排查工程师 → 完整上下文 + 堆栈 + traceId
4. 监控告警系统    → 分级 + 聚合 + 噪音过滤
```

**核心矛盾**:同一个错误,这四个视角需要的信息完全不一样。**好的错误体系是分层的**——同一份原始 Exception,**对外包装,对内保留**。

> 经验法则:**错误信息不是给程序员"自己看"的,是给四种角色分别看的**。给用户看的别带 stack trace,给开发看的别只有"系统繁忙"。

## 二、异常分层

```
       ┌───────────────────────────────┐
       │    BaseException(根)          │
       └──────────────┬────────────────┘
                      │
   ┌──────────────────┼──────────────────┐
   ▼                  ▼                  ▼
[BizException]   [SystemException]  [ThirdPartyException]
业务规则不通过       系统/中间件失败       第三方调用失败
"库存不足"          "DB 连不上"         "支付网关超时"
不打 stack         打 warn             打 warn + 重试
HTTP 4xx / 422     HTTP 500            HTTP 502 / 503
```

| 类 | 含义 | 处理 | 日志级别 |
| --- | --- | --- | --- |
| **BizException** | 用户操作 / 业务规则不通过 | 返回友好提示 | INFO(不告警) |
| **SystemException** | 自家系统出错(NPE / DB / Redis 挂) | 兜底 500 + 告警 | ERROR |
| **ThirdPartyException** | 三方接口失败 | 重试 / 降级 | WARN(频繁则告警) |
| **AuthException** | 鉴权失败 | 401 / 403 | INFO |
| **RateLimitException** | 限流触发 | 429 | INFO + 监控 |
| **NotFoundException** | 资源不存在 | 404 | INFO |

```java
public abstract class BaseException extends RuntimeException {
    private final ErrorCode code;
    private final Map<String, Object> meta;

    protected BaseException(ErrorCode code, String message, Throwable cause, Map<String, Object> meta) {
        super(message, cause);
        this.code = code;
        this.meta = meta == null ? Map.of() : meta;
    }

    public ErrorCode code() { return code; }
    public Map<String, Object> meta() { return meta; }
}

public class BizException extends BaseException {
    public BizException(ErrorCode code, String message) { super(code, message, null, null); }
    public BizException(ErrorCode code, String message, Map<String, Object> meta) {
        super(code, message, null, meta);
    }
}
```

> 经验法则:**异常类不超过 5-7 个根类**——再细的分类靠 `errorCode` 字段表达。每加一个异常类,catch 链就要加一行,维护成本是非线性的。

## 三、错误码规范

错误码不是 HTTP 状态码,**是业务粒度的"标识符"**。

### 命名风格三选一

```
1. 数字编码:10001, 10002...   → 紧凑,但读不出含义
2. 域.动作.状态:order.create.stock_insufficient → 可读,推荐
3. SCREAMING_SNAKE:STOCK_INSUFFICIENT → 简洁,但全局命名易冲突
```

**推荐风格**:`<domain>.<resource>.<reason>`

```
order.create.stock_insufficient      库存不足
order.pay.amount_exceeded            金额超限
user.auth.password_expired           密码过期
file.upload.size_too_large           文件过大
sys.db.connection_timeout            DB 超时
sys.thirdparty.payment_unavailable   支付网关挂
```

### 错误码注册表

集中维护:

```java
public enum ErrorCode {
    ORDER_STOCK_INSUFFICIENT("order.create.stock_insufficient", 422, "库存不足"),
    ORDER_AMOUNT_EXCEEDED   ("order.pay.amount_exceeded",       422, "金额超限"),
    USER_PASSWORD_EXPIRED   ("user.auth.password_expired",      401, "密码已过期,请重置"),
    SYS_DB_TIMEOUT          ("sys.db.connection_timeout",       500, "服务异常,请重试"),
    SYS_THIRDPARTY_PAY_DOWN ("sys.thirdparty.payment_unavailable", 503, "支付服务暂不可用");

    private final String code;
    private final int    httpStatus;
    private final String defaultMessage;
}
```

> 经验法则:**错误码全公司维护一份注册表**——文档化、版本化、对外公开。前端 / 客户端 / 文档都消费这一份。

## 四、统一响应结构

```json
// 成功
{
  "data": { "id": 123, "name": "Alice" }
}

// 错误(基于 RFC 7807 扩展)
{
  "type": "https://api.example.com/errors/order/stock_insufficient",
  "title": "库存不足",
  "status": 422,
  "code": "order.create.stock_insufficient",
  "detail": "商品 SKU-A 仅剩 3 件,请求 5 件",
  "traceId": "abc123-def456",
  "timestamp": "2026-04-15T09:30:00Z",
  "errors": [
    { "field": "items[0].qty", "code": "STOCK_INSUFFICIENT", "available": 3 }
  ]
}
```

**两条原则**:

- **HTTP 状态码用对**(56 章讲过):200/201/204 / 400/401/403/404/409/422/429 / 500/502/503
- **业务错误码放 body**——细粒度,前端按 code 路由处理

> ⚠️ **绝对避免**:HTTP 200 + body { code: -1 }——网关、监控、客户端 SDK 都不认。

## 五、Spring 全局异常处理

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(BizException.class)
    public ResponseEntity<ApiError> handleBiz(BizException e, HttpServletRequest req) {
        log.info("biz error: {} {}", e.code(), e.getMessage());
        return ResponseEntity.status(e.code().httpStatus()).body(ApiError.from(e, req));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiError> handleValidation(MethodArgumentNotValidException e, HttpServletRequest req) {
        var errors = e.getBindingResult().getFieldErrors().stream()
            .map(f -> Map.of("field", f.getField(), "code", f.getCode(), "msg", f.getDefaultMessage()))
            .toList();
        return ResponseEntity.badRequest().body(ApiError.validation(errors, req));
    }

    @ExceptionHandler(ThirdPartyException.class)
    public ResponseEntity<ApiError> handle3rd(ThirdPartyException e, HttpServletRequest req) {
        log.warn("3rd-party error", e);
        return ResponseEntity.status(503).body(ApiError.unavailable(e, req));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiError> handleAll(Exception e, HttpServletRequest req) {
        log.error("unhandled", e);          // ★ 兜底必打 ERROR + 完整堆栈
        return ResponseEntity.internalServerError()
            .body(ApiError.internal(req));   // 不要把 e.getMessage() 暴露出去
    }
}
```

> 经验法则:**`Exception.class` 兜底必须存在**——任何未处理异常都不能直接 500 + 默认 Whitelabel 页或漏 stack trace 给用户。

## 七、TraceId / 上下文透传

每个请求一个 `traceId`,从入口贯穿到所有下游服务和日志:

```java
@Component
public class TraceIdFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse resp, FilterChain chain)
            throws ServletException, IOException {
        String tid = Optional.ofNullable(req.getHeader("X-Trace-Id")).orElse(UUID.randomUUID().toString());
        MDC.put("traceId", tid);
        resp.setHeader("X-Trace-Id", tid);
        try { chain.doFilter(req, resp); }
        finally { MDC.clear(); }
    }
}
```

```
日志格式:%X{traceId} %-5level %logger - %msg

abc123-def456  ERROR  OrderService - failed to charge: ...
```

```java
// 错误响应 body 也带上 traceId
{
  "code": "...",
  "traceId": "abc123-def456"
}
```

**用户截图 + traceId → 工程师秒级定位日志**。这是排障速度的关键。

> 经验法则:**所有错误响应都带 traceId**——客服收到投诉,把 traceId 给工程师,5 分钟出报告。否则一个个用户日志去翻。

## 八、错误重试策略

不是所有错误都该重试:

| 错误类型 | 重试策略 |
| --- | --- |
| **业务错误**(库存不足、参数非法) | ❌ 不重试 |
| **鉴权 / 权限**(401 / 403) | ❌ 不重试 |
| **限流**(429) | ✅ 退避后重试 |
| **资源冲突**(409 乐观锁) | ✅ 立刻重试一次 |
| **依赖暂时不可用**(502 / 503 / 504) | ✅ 指数退避 |
| **超时**(连接 / 读) | ⚠️ 看接口幂等性 |
| **网络异常** | ✅ 指数退避 |
| **5xx 自家服务** | ❌ 重试也是错(让监控告警) |

**Resilience4j 重试**:

```java
RetryConfig cfg = RetryConfig.custom()
    .maxAttempts(3)
    .intervalFunction(IntervalFunction.ofExponentialBackoff(100, 2.0))
    .retryExceptions(IOException.class, TimeoutException.class)
    .ignoreExceptions(BizException.class)
    .build();
Retry retry = Retry.of("payment", cfg);
Supplier<Result> sup = Retry.decorateSupplier(retry, () -> paymentClient.charge(...));
```

> 经验法则:**业务异常绝不重试**——让用户重试或换路径。**网络/超时类才重试,且必须配合幂等**(63 章)。

## 九、降级:错误的"备用方案"

依赖挂了不一定要把请求挂,可以走降级:

| 场景 | 降级方案 |
| --- | --- |
| 推荐服务挂 | 返"热门商品"列表 |
| 用户中心挂 | 返缓存里的用户基础信息 |
| 评论服务挂 | 隐藏评论模块,主流程不挂 |
| 风控挂 | 默认通过,事后补单审核 |

```java
@CircuitBreaker(name = "recommend", fallbackMethod = "fallback")
public List<Product> recommend(String userId) {
    return recommendClient.list(userId);
}
public List<Product> fallback(String userId, Throwable t) {
    return cache.getHotList();
}
```

> 经验法则:**核心链路 + 重要 RPC** 必须有 fallback——否则下游一挂,整条链死。

## 十、敏感信息泄露的反模式

错误信息别带:

```
❌ "数据库连接 jdbc:mysql://10.0.1.5:3306/prod 失败"
❌ "用户 admin@example.com 密码错误"  ← 泄露用户存在
❌ Stack trace 直接吐到前端 ← 泄露代码结构、依赖版本
❌ "INVALID SQL: SELECT * FROM users WHERE id=..." ← SQL 信息
```

**正确做法**:

- 对外:**只给 errorCode + 友好文案**
- 内部日志:完整堆栈、上下文、参数(脱敏后)

```java
// 给前端
"detail": "登录失败,请检查账号或密码"

// 内部日志
log.warn("login failed: userId={} reason={}", userId, "PASSWORD_MISMATCH");
```

> 经验法则:**Spring Boot 生产环境 `server.error.include-stacktrace=never`**——哪怕兜底也别吐。代码细节是攻击者的免费情报。

## 十一、Result 模式 vs 异常

某些团队推崇"用 Result<T,E> 替代异常"(Rust / Go 风格):

```java
// 异常
Order pay() { ...; throw new BizException(...); }

// Result
Result<Order, PayError> pay() { ...; return Result.error(PayError.STOCK_OUT); }
```

| 维度 | 异常 | Result |
| --- | --- | --- |
| 流程清晰 | 可能跨层抛 | 强制处理 |
| 性能 | 抛异常贵(stack 构造) | 几乎零开销 |
| 业务表达 | 自然 | 啰嗦 |
| 与生态融合 | Spring / JPA / Validation 都用异常 | 要适配 |

> 经验法则:**Java 生态默认用异常**(Spring 整套都基于异常)——但**循环里频繁的"业务结果"用 Result 类型更优**。混合用,**异常表达"异常情况",Result 表达"两种正常结果"**。

## 十二、错误监控与聚合

**Sentry / Bugsnag / Rollbar**:把异常自动聚合,**重复的合并,新出现的告警**。

```java
@ExceptionHandler(Exception.class)
public ResponseEntity<?> all(Exception e, HttpServletRequest req) {
    Sentry.captureException(e);     // 自动带 traceId / user / request
    return ...;
}
```

| 维度 | 监控 |
| --- | --- |
| 错误率 | 5xx / 总数,SLO 烧速 |
| 错误增量 | 突发新错误 → 立刻告警 |
| 影响用户数 | 同一错误影响 N 个 user |
| 指纹聚合 | 相同堆栈合并成一组 |
| 关联部署 | 错误激增是不是跟某次发布相关 |

> 经验法则:**错误监控用专业工具,别自己拼日志告警**——Sentry 这种免费层就能用,**把"看错误"做成日常习惯**比啥都强。

## 十三、错误日志的格式化

```
❌ log.error("出错了!", e);                             // 无上下文
❌ log.error("user=" + userId + " err=" + e);            // 字符串拼接
❌ log.info("error=" + e);                                // 没异常对象
❌ log.error("orderId=" + orderId, e);                    // 占位符没用上

✅ log.error("order.pay failed: orderId={} userId={}", orderId, userId, e);
   ↑ 占位符填业务上下文,异常对象作为最后参数
```

**结构化日志(JSON)**:

```json
{
  "ts": "2026-04-15T09:30:00Z",
  "level": "ERROR",
  "logger": "OrderService",
  "msg": "order.pay failed",
  "traceId": "abc123",
  "orderId": "ORD-001",
  "userId": "U-002",
  "errorCode": "sys.thirdparty.payment_unavailable",
  "stack": "..."
}
```

JSON 日志直接进 ELK / Loki,**不用正则解析**——查询和告警速度天差地别。

## 十四、用户友好的文案

| 场景 | 烂文案 | 好文案 |
| --- | --- | --- |
| DB 连不上 | "Connection refused" | "服务暂时不可用,请稍后重试" |
| 表单错 | "validation error" | "邮箱格式不正确,请重新输入" |
| 库存不足 | "stock < requested" | "您选的商品仅剩 3 件,请减少数量" |
| 鉴权 | "401 Unauthorized" | "登录已过期,请重新登录" |
| 第三方挂 | "Upstream timeout" | "支付服务繁忙,请稍后重试,款项不会被扣除" |

**文案三原则**:

1. **告诉用户发生了什么**(不暴露技术细节)
2. **告诉用户该怎么办**("请重试 / 联系客服 / 检查输入")
3. **必要时安抚**("款项不会被扣除"——付款失败的高焦虑场景)

> 经验法则:**错误文案找产品 / 客服一起定**——他们知道用户看到这句话会怎么反应。技术人写的"系统异常 errCode=500"是反人类的。

## 十五、常见踩坑

1. **HTTP 200 + body errCode**:网关 / 监控 / SDK 全废
2. **抛 RuntimeException 不分类**:全 500,无法精细处理
3. **catch (Exception e) 吞了不打日志**:出错查不到
4. **catch (Exception e) { throw new RuntimeException(e); }**:堆栈截断,只看到包装层
5. **e.printStackTrace()**:输出到 stdout 没接日志收集
6. **业务错误也打 ERROR**:告警噪音,真错误被淹没
7. **没 traceId**:用户投诉一个错,客服查半小时
8. **error message 给用户看堆栈**:泄露技术信息 + 不友好
9. **错误码每个团队各自发明**:全公司 200 个不重复的"401"
10. **用 magic number 当错误码(-1, -2)**:文档 / 代码对不起来
11. **重试业务异常**:库存不足重试 3 次,日志爆炸
12. **没 fallback,下游一挂全死**:小问题变大事故
13. **Sentry / Bugsnag 没接**:线上错误没人主动看
14. **日志 logger.error 没传异常对象**:`log.error("err=" + e)` 字符串拼接,没 stack
15. **拒绝错误数据 / 发短信失败 也吃了**:业务以为成功,数据漂移
16. **不区分客户端错(4xx)和服务端错(5xx)**:监控指标失真

## 小结

错误处理是**工程文化的"X 光机"**——看一个项目错误体系,几乎能直接判断这个团队的成熟度。

记住三件事:

1. **错误信息有四种受众**——一份原始 Exception,对外脱敏,对内详细,分级日志,聚合告警
2. **错误码是公共契约**——前端、客户端、文档、对外 API 共用一份,集中维护
3. **traceId + 结构化日志 + Sentry** 是排障三件套——少了任一,故障定位慢一个量级
