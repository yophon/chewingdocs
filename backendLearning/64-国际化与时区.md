# 国际化与时区

i18n / 时区 / 多语言看着是"改个文案"的小事——**实际上是后端最容易踩"几年都没发现的隐性 bug"** 的领域。

时区错一小时可能导致结算差几百万,语言文案硬编码可能导致出海版本永远过不了本地化审核。这章把出海项目的硬功夫讲清。

---

## 一、i18n / l10n / g11n 的关系

```
i18n (internationalization, 18 字母):工程层把"地区差异"抽出来
l10n (localization, 10 字母):为某个具体地区做适配(翻译、格式)
g11n (globalization, 11 字母):整体战略,涵盖产品 / 法律 / 商业
```

i18n 是**工程能力**,l10n 是**内容产出**,g11n 是**业务决策**。后端主要做 i18n。

---

## 二、Locale:语言 + 地区的组合

```
zh-CN  → 简体中文 / 中国
zh-TW  → 繁体中文 / 台湾
zh-HK  → 繁体中文 / 香港
en-US  → 英语 / 美国
en-GB  → 英语 / 英国(日期 / 货币不同)
es-MX  → 西语 / 墨西哥
es-ES  → 西语 / 西班牙
```

**Locale = 语言代码(ISO 639)+ 地区代码(ISO 3166)**。

```java
Locale loc = Locale.forLanguageTag("zh-CN");
// 或
Locale loc = new Locale("zh", "CN");

NumberFormat.getCurrencyInstance(Locale.forLanguageTag("en-US"))
    .format(1234.56);   // → "$1,234.56"

NumberFormat.getCurrencyInstance(Locale.forLanguageTag("de-DE"))
    .format(1234.56);   // → "1.234,56 €"
```

> 经验法则:**永远存 locale 而非"language"**——同样的中文,zh-CN / zh-TW / zh-HK 数字、日期、用语都不同。

## 三、识别用户语言

```
优先级(从高到低):
  1. 用户在 UI 里显式选择(saved in profile / cookie)
  2. URL 路径前缀:/zh-CN/products
  3. 子域名:zh.example.com
  4. 请求 Header:Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
  5. 服务器默认
```

```java
@Bean
public LocaleResolver localeResolver() {
    var r = new AcceptHeaderLocaleResolver();
    r.setDefaultLocale(Locale.US);
    r.setSupportedLocales(List.of(
        Locale.US, Locale.SIMPLIFIED_CHINESE, Locale.TRADITIONAL_CHINESE
    ));
    return r;
}
```

> 经验法则:**别只看 IP 猜语言**——VPN / 代理 / 出差用户全错。**`Accept-Language` 是浏览器告诉你的最准确信息**。

## 四、Spring 文案国际化

```
src/main/resources/
├── messages.properties           ← 默认(英文 / fallback)
├── messages_zh_CN.properties     ← 简中
├── messages_zh_TW.properties     ← 繁中
├── messages_ja_JP.properties     ← 日语
└── messages_es.properties        ← 西语(不分地区)
```

```properties
# messages_zh_CN.properties
greeting=你好,{0}!
order.created=订单 {0} 已创建,金额 {1}
error.amount.invalid=金额必须大于 0
```

```java
@Autowired private MessageSource messageSource;

public String greet(String name, Locale loc) {
    return messageSource.getMessage("greeting", new Object[]{name}, loc);
}
```

```yaml
spring:
  messages:
    basename: messages
    encoding: UTF-8
    fallback-to-system-locale: false   # 重要:fallback 到 default,不要系统 locale
    cache-duration: 3600
```

**Bean Validation 国际化**:

```properties
# messages.properties
NotBlank.user.email=Email is required
Email.user.email=Invalid email format
```

```java
public record CreateUserReq(
    @NotBlank @Email String email
) {}
```

## 五、复数 / 性别 / 上下文:别相信"简单替换"

```
1 message  → "1 message"
2 messages → "2 messages"
俄语:1, 2-4, 5-20 三套不同形式
波兰语:5 套形式
阿拉伯语:6 套形式
```

属性文件搞不定,**用 ICU MessageFormat**(国际标准):

```
{count, plural,
  =0 {No messages}
  one {# message}
  other {# messages}
}
```

```java
// 用 com.ibm.icu.text.MessageFormat
MessageFormat fmt = new MessageFormat(pattern, Locale.forLanguageTag("ru"));
fmt.format(Map.of("count", 5));   // 自动选俄语正确形式
```

> 经验法则:**面向 5+ 语言的产品必上 ICU**——属性文件 + `{0}` 占位符在俄语、阿语、希伯来语、印度语全废。

## 六、时区:最容易出生产事故的一块

### 1. 区分"瞬时时刻"和"日历时刻"

```
"2026-04-15 09:00 UTC"        ← 瞬时时刻(全球同一刻)
"2026-04-15 09:00 北京时间"   ← 也是瞬时(等于 01:00 UTC)
"2026-04-15"                   ← 日历日(无时区,因人而异)
"每周一 09:00"                ← 用户本地的概念
```

**存储原则**:

| 数据 | 存什么 |
| --- | --- |
| 事件发生时间(订单创建、登录) | **UTC 瞬时时刻**(`Instant` / `timestamp with time zone`) |
| 用户生日 | **日历日**(`LocalDate`,无时区) |
| 用户预约"每周一 9 点" | **本地时间 + 时区**(`LocalDateTime` + `ZoneId`) |
| 报表"3 月 15 日营业额" | 业务规定的"业务日"(可能 UTC、可能机构当地) |

### 2. 数据库类型

```sql
-- PostgreSQL
TIMESTAMP                         -- ❌ 不带时区,有歧义
TIMESTAMP WITH TIME ZONE          -- ✅ 存 UTC,带 tz 信息
DATE                              -- 日历日

-- MySQL
DATETIME                          -- ❌ 不带时区
TIMESTAMP                         -- ✅ MySQL 自动转 UTC 存储,但只到 2038(Y2K38)
```

> 经验法则:**新项目用 PostgreSQL `TIMESTAMP WITH TIME ZONE`**,MySQL 用 `DATETIME` 但**应用层强制 UTC 写入读取**。

### 3. JVM 默认时区是大坑

```
JVM 启动时拿系统时区 → 系统是"Asia/Shanghai" → 你以为存的是 UTC,其实是 +08:00
```

```bash
# 强制 JVM 用 UTC
-Duser.timezone=UTC
```

```yaml
spring:
  jackson:
    time-zone: UTC          # JSON 序列化时的时区
  jpa:
    properties:
      hibernate:
        jdbc:
          time_zone: UTC    # JDBC 写入时区
```

```java
// 数据库连接也要指定
spring.datasource.url=jdbc:mysql://...?serverTimezone=UTC
```

### 4. Java 8+ 时间 API:必学

```java
// 立即时刻(UTC)
Instant now = Instant.now();
// 带时区的日历时刻
ZonedDateTime zdt = now.atZone(ZoneId.of("Asia/Shanghai"));
// 用户本地无时区时间
LocalDateTime ldt = zdt.toLocalDateTime();
// 日历日(无时区)
LocalDate today = LocalDate.now(ZoneId.of("Asia/Shanghai"));

// 转换
Instant i = ldt.atZone(ZoneId.of("Asia/Shanghai")).toInstant();

// 永远不要用 java.util.Date / Calendar / SimpleDateFormat
```

> 经验法则:**全代码库禁用 `java.util.Date` / `Calendar` / `SimpleDateFormat`**——时区不安全、API 反直觉、SimpleDateFormat 还是非线程安全的。统一用 `java.time.*`。

## 七、夏令时(DST):一年发生两次的小灾难

```
美国 2026-03-08 02:00 → 跳到 03:00(春令时,失去一小时)
美国 2026-11-01 02:00 → 退回 01:00(冬令时,多一小时)

→ 02:30 这个时刻在春令时根本不存在
→ 01:30 在冬令时出现两次

任何"按当地时间运行"的定时任务都可能跑两次或漏跑
```

破法:

- 关键定时任务用 **UTC + cron**(不会 DST)
- 必须按当地时间的任务(用户本地"每周一早 9 点提醒"),用 ZonedDateTime 计算下一个有效时刻

```java
ZoneId tz = ZoneId.of("America/New_York");
ZonedDateTime next = LocalDate.now().atTime(2, 30).atZone(tz);
// 如果 2:30 不存在,JDK 会自动调整到 3:30
```

## 八、日期 / 数字 / 货币格式

```java
Locale us = Locale.US;
Locale de = Locale.GERMANY;
Locale ar = Locale.forLanguageTag("ar-SA");

DateTimeFormatter fmtUs = DateTimeFormatter.ofLocalizedDateTime(FormatStyle.MEDIUM).withLocale(us);
DateTimeFormatter fmtDe = DateTimeFormatter.ofLocalizedDateTime(FormatStyle.MEDIUM).withLocale(de);

ZonedDateTime now = ZonedDateTime.now();
fmtUs.format(now);    // "Apr 15, 2026, 9:00:00 AM"
fmtDe.format(now);    // "15.04.2026, 09:00:00"

NumberFormat.getInstance(de).format(1234567.89);   // "1.234.567,89"
NumberFormat.getInstance(us).format(1234567.89);   // "1,234,567.89"
NumberFormat.getInstance(ar).format(1234567.89);   // "١٬٢٣٤٬٥٦٧٫٨٩"
```

| 易错点 |
| --- |
| 美国:逗号千分位、点小数;月/日/年 |
| 欧洲多数:点千分位、逗号小数;日/月/年 |
| 中国:逗号千分位、点小数;年/月/日 |
| 日本:用"令和" / "平成"年号(政府文件) |
| 阿拉伯:东阿数字 + RTL 布局 |
| 印度:十万 / 千万计数法(Lakh / Crore),千分位是 12,34,567 |

> 经验法则:**不要自己拼日期 / 货币字符串**——一律用 JDK 的 `DateTimeFormatter` / `NumberFormat`,locale 一切。

## 九、字符串 / 排序 / 大小写的陷阱

```java
"İstanbul".toLowerCase();                                  // "i̇stanbul"(英语规则)
"İstanbul".toLowerCase(Locale.forLanguageTag("tr"));       // "istanbul"(土耳其规则)

"Apple".compareTo("apple");        // 65 vs 97,Z 以前的所有大写都"小于"小写
Collator c = Collator.getInstance(Locale.forLanguageTag("zh"));
c.compare("a", "A");               // 按本地化规则比较
```

**坑**:`String.toUpperCase()` / `toLowerCase()` 默认用系统 locale——**线上 locale 可能跟你测试时不同**。**永远显式传 Locale**。

```java
"USER@EXAMPLE.COM".toLowerCase(Locale.ROOT)       // 不依赖系统
```

## 十、数据库的字符集与排序

```sql
-- MySQL:必须 utf8mb4(3 字节 utf8 支持不了 emoji)
CREATE DATABASE app DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- 排序规则
utf8mb4_general_ci    -- 旧,不准确,建议不用
utf8mb4_unicode_ci    -- 较准
utf8mb4_0900_ai_ci    -- MySQL 8 默认,Unicode 9.0,准确
```

> 经验法则:**MySQL 永远 utf8mb4**——`utf8` 不是真 UTF-8(只支持 BMP 内 3 字节),emoji / 部分汉字会插入失败。

PostgreSQL 默认 UTF-8,**多语言友好**,排序规则可以指定字段:

```sql
SELECT name FROM users ORDER BY name COLLATE "zh-x-icu";
```

## 十一、LTR / RTL / 双向文本

```
LTR(Left-to-Right):中、英、日、韩
RTL(Right-to-Left):阿语、希伯来语、波斯语
```

后端层面:

- 字符串里**正确处理 Unicode 双向算法控制符**(LRM / RLM / FSI / PDI)
- 数据库存原文,**渲染交给前端**
- 输出邮件 / PDF / 短信时,**模板要支持 RTL 布局**(`dir="rtl"`)

> 经验法则:**做阿语市场前,UI 设计师 + 前端 + 后端模板都要懂 RTL**——这不是"翻译就完"的事,菜单方向、按钮位置、图标都要镜像。

## 十二、内容翻译流水线

技术 + 内容流程:

```
代码里只用 key:t("order.created", { id: 123 })
           │
           ▼
   翻译平台(Crowdin / Lokalise / Phrase / Localazy)
           │
           ▼
   生成各语言 messages_xx_XX.properties
           │
           ▼
   CI 拉下来打包
```

**翻译流程**:

- 每周自动从代码扫 key,推送到平台
- 翻译者在线编辑,带上下文截图
- AI 预翻译(GPT / DeepL)+ 人工 review
- 发布时自动拉下来更新

> 经验法则:**翻译用专业平台,别用 Excel 来回发邮件**——产品做大后翻译管理是真痛点,Crowdin 这种平台值这个钱。

## 十三、合规与隐私

不同地区有不同法律要求:

| 地区 | 法律 | 关键要求 |
| --- | --- | --- |
| 欧洲 | **GDPR** | 删除权、知情权、数据本地化 |
| 加州 | **CCPA / CPRA** | 类似 GDPR |
| 中国 | **个人信息保护法 / 数据安全法** | 数据出境申报 / 大客户安全评估 |
| 巴西 | LGPD | 类 GDPR |
| 韩国 | PIPA | 同意机制严格 |

**数据本地化**:欧洲 / 中国部分行业要求"数据不出境"——**架构上要支持按 region 部署独立栈**。

> 经验法则:**做欧洲市场前先和法务团队过一遍 GDPR**——技术能做的就 30%,业务流程、合同、Cookie banner 都要配合。**违反 GDPR 罚款上限 4% 全球营收**。

## 十四、电话 / 地址 / 姓名

| 字段 | 国际化考虑 |
| --- | --- |
| 电话 | E.164 格式 `+86 13800138000`;libphonenumber 校验 |
| 地址 | **千万别 hardcode 字段**(中国"省市区",日本"都道府县市町村",美国 ZIP)用结构化对象 |
| 姓名 | 西方 firstName / lastName 不通用——日韩中是"姓 + 名",阿拉伯有 "kunya / nasab / laqab" 多段 → 用单一 `fullName` 字段 |
| 邮编 | 不是所有国家都有(爱尔兰部分地区);格式各异 |

```java
// 用 Google libphonenumber
PhoneNumberUtil util = PhoneNumberUtil.getInstance();
PhoneNumber n = util.parse("13800138000", "CN");
util.isValidNumber(n);    // true
util.format(n, PhoneNumberFormat.E164);   // "+8613800138000"
```

> 经验法则:**地址 / 姓名 / 电话用社区库,别自己写校验**。`libphonenumber` / `libpostal` 已经踩过所有国家的坑。

## 十五、常见踩坑

1. **JVM 启动用系统时区**:测试是 +08,生产 UTC,行为不一致
2. **数据库 DATETIME 不带时区**:跨时区查询全错
3. **MySQL 用 utf8 而非 utf8mb4**:emoji 插不进去报错
4. **代码里硬编码 "中文"**:出海版改不动
5. **`SimpleDateFormat` 共享非线程安全**:并发下时间错乱
6. **toLowerCase() 不带 Locale**:土耳其语 i 字母事故
7. **DST 没考虑**:某天定时任务跑两次或不跑
8. **报表"今日"用 UTC midnight**:中国用户看不到当天数据
9. **BOM / nbsp / 全角空格**:用户输入复制粘贴带进来,equals 失败
10. **编码 latin1 → utf-8 转换错**:数据库出现 ä á 这种垃圾
11. **Bidi 文本截断**:阿语字符串中间截了导致显示乱
12. **货币用 double**:0.1 + 0.2 ≠ 0.3,**永远 BigDecimal**
13. **货币四舍五入随便选模式**:金融场景必须 HALF_EVEN(银行家舍入)
14. **localeResolver 没缓存**:每个请求重解析 Accept-Language
15. **静态资源(图片)硬编码语言**:`logo_zh.png` 写死,新增语言要改代码
16. **数据本地化没准备**:欧洲客户要求"数据留在欧洲",架构改不了

## 小结

国际化是**"不上的时候没事,上了之后处处是雷"** 的领域。

记住三件事:

1. **永远存 UTC + 本地化输出**——存储和展示分离
2. **不自己拼日期 / 数字 / 货币 / 电话 / 地址**——交给社区库
3. **JVM / 数据库 / Jackson 三处时区配置必须一致**——任一不一致就有偏差
