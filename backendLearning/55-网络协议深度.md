# 网络协议深度

2 章讲了 HTTP 与 RESTful 的"业务面",但 HTTP 之下是一整套协议栈:**TCP / TLS / HTTP/2 / HTTP/3 / QUIC**。

做纯业务可以略过这一章,但**做网关、做中间件、做性能调优、排"网络抖动"问题**,这一层就是绕不开的硬功夫。

---

## 一、协议栈全景

```
        ┌────────────────────┐
        │  HTTP/1.1, /2, /3 │  应用层
        ├────────────────────┤
        │  TLS 1.2 / 1.3    │  安全层
        ├────────────────────┤
        │  TCP    或    UDP │  传输层
        ├────────────────────┤
        │     IP            │  网络层
        ├────────────────────┤
        │  以太网/Wi-Fi      │  链路层
        └────────────────────┘
```

| 协议 | 跑在 | 关键贡献 |
| --- | --- | --- |
| HTTP/1.1 | TCP + TLS | 文本、Keep-Alive、Pipeline |
| HTTP/2 | TCP + TLS | 二进制、多路复用、Header 压缩 |
| HTTP/3 | **UDP** + QUIC | 无队头阻塞、0-RTT、连接迁移 |

> 经验法则:**新业务直接上 HTTP/2**,有移动端/海外用户的考虑 HTTP/3。HTTP/1.1 只在"老接口、必须裸 TCP debug"场景留着。

## 二、TCP:看似古老,坑无数

### 三次握手 / 四次挥手

```
客户端 ──SYN──▶ 服务端     建立:
        ◀─SYN+ACK──
        ──ACK──▶                 1 RTT 才能开始发数据

客户端 ──FIN──▶ 服务端     关闭:
        ◀─ACK──
        ◀─FIN──                  双向独立关闭
        ──ACK──▶
```

握手 1 RTT 是**HTTP/1.1 + TLS 在国际线路慢的根本原因**——握 TCP 1 次 + 握 TLS 1~2 次,光建连就 200~600ms。

### TIME_WAIT 怎么处理

主动关连接的一方会进 TIME_WAIT,**默认持续 2MSL ≈ 60 秒**。短连接 + 高 QPS 系统(比如压测机器、爬虫)会堆几十万 TIME_WAIT,**端口耗尽**。

```bash
# 看连接状态分布
ss -tan | awk '{print $1}' | sort | uniq -c

# 内核参数
net.ipv4.tcp_tw_reuse = 1               # 复用 TIME_WAIT 端口(客户端用,安全)
net.ipv4.ip_local_port_range = 10000 65535
```

> ⚠️ **不要开 `tcp_tw_recycle`**——这个参数 2018 年已从 Linux 内核移除,留在网上的"调优文档"全过时了,开了 NAT 后面用户会随机连接失败。

### 拥塞控制:BBR vs CUBIC

```
CUBIC(传统默认):基于丢包探测拥塞——丢包就降速
BBR(2016, Google):基于带宽 + 延迟建模——主动找带宽上限
```

**BBR 在跨地域、有丢包的场景下吞吐能比 CUBIC 高一个量级**。

```bash
# 启用 BBR
echo 'net.core.default_qdisc=fq' >> /etc/sysctl.conf
echo 'net.ipv4.tcp_congestion_control=bbr' >> /etc/sysctl.conf
sysctl -p
```

> 经验法则:**国内本地机房用 CUBIC 够,跨国 / CDN 节点 / 公网链路一律开 BBR**——成本几乎 0,跨地域吞吐立刻起飞。

## 三、TLS:从 1.2 到 1.3 的革命

### TLS 1.2 握手(2 RTT)

```
Client ──ClientHello(随机+支持的算法)──▶ Server
       ◀──ServerHello+Cert+ServerKeyExchange──
       ──ClientKeyExchange+ChangeCipherSpec──▶
       ◀──ChangeCipherSpec(完成)──
       ━━━━ 应用数据 ━━━━

光建立连接:TCP 1 RTT + TLS 2 RTT = 3 RTT,跨国 600ms 起步
```

### TLS 1.3 握手(1 RTT,部分场景 0 RTT)

```
Client ──ClientHello + KeyShare(预猜算法)──▶ Server
       ◀──ServerHello+Cert+Finished──
       ━━━━ 应用数据 ━━━━

仅 1 RTT,且去掉了一堆已被攻破的算法(RSA 密钥交换、CBC、SHA1...)
```

**0-RTT(Early Data)**:复用之前 session,客户端 ClientHello 里就带应用数据,**第一个包就能打业务**——但要注意"重放攻击"风险,只用于幂等请求(GET)。

> 经验法则:**生产环境强制 TLS 1.3,禁用 1.0/1.1**(已废)。开 OCSP Stapling、HSTS、ALPN——这些一行 Nginx 配置就能开,带来巨大安全/性能收益。

### 证书管理

```bash
# Let's Encrypt 自动签 + 续期
certbot certonly --dns-cloudflare ...

# 用 cert-manager 在 K8s 里全自动
apiVersion: cert-manager.io/v1
kind: Certificate
spec:
  dnsNames: ["api.example.com"]
  issuerRef: { name: letsencrypt-prod, kind: ClusterIssuer }
```

**证书过期是经典生产事故**——每个证书都该接监控,提前 30 天告警,提前 7 天强制续。

## 四、HTTP/1.1 的死穴:队头阻塞

```
HTTP/1.1 一个 TCP 连接同时只能发一个请求(管道化也几乎没人用)

Connection 1: ──Req1──▶ ◀─Res1─── ──Req2──▶ ◀─Res2───
                        慢的 Res1 把后面全卡住

浏览器开多个连接(默认 6 个/域名)缓解,但本质问题还在
```

**变通**:

- 域名分片:`static1.x.com`, `static2.x.com`
- 雪碧图、合并 JS/CSS
- CDN

这些都是 HTTP/1.1 时代被迫的优化,HTTP/2 之后**都过时了**。

## 五、HTTP/2:二进制 + 多路复用

```
1 个 TCP 连接,N 个 Stream 并行,不互相阻塞

   ──┬─ Stream 1: GET /a.js     ──▶
     ├─ Stream 3: GET /b.css    ──▶
     ├─ Stream 5: GET /api/user ──▶
     └─ Stream 7: GET /api/list ──▶
   ◀─┴─ 各自的响应帧交错回来 ─────
```

| 特性 | 价值 |
| --- | --- |
| **二进制分帧** | 高效解析、不再是文本 |
| **多路复用** | 一连接多请求,告别 6 连接限制 |
| **HPACK 头压缩** | 重复 header 只传一次 ID |
| **Server Push**(已废弃) | 服务端主动推资源,实际效果差,Chrome 已移除 |
| **Stream 优先级** | 浏览器告诉服务端"这个先返" |

> 但 HTTP/2 仍跑在 TCP 上 → **TCP 层丢包,所有 Stream 都阻塞**——这是它的最大遗憾,也是 HTTP/3 要解决的问题。

### Spring Boot 开 HTTP/2

```yaml
server:
  http2:
    enabled: true
  ssl:                # HTTP/2 必须 TLS(浏览器规定)
    enabled: true
    key-store: classpath:keystore.p12
    key-store-password: secret
```

### gRPC 默认 HTTP/2

37 章 gRPC 跑在 HTTP/2 上,**双向流式 = HTTP/2 多路复用 + Stream**。这就是为什么 gRPC 比 REST 高效。

## 六、HTTP/3 与 QUIC:把"传输层"换了

QUIC = **UDP 上自己实现的可靠传输 + TLS 1.3 + 多路复用**。HTTP/3 = HTTP 跑在 QUIC 上。

```
HTTP/3            HTTP/2
   │                 │
QUIC              TLS 1.3
   │                 │
TLS 1.3            TCP
   │                 │
  UDP                IP
   │
  IP
```

| 特性 | HTTP/3 / QUIC | HTTP/2 over TCP |
| --- | --- | --- |
| 队头阻塞 | **每 Stream 独立** | TCP 层阻塞,全卡 |
| 建立连接 | 1 RTT(TLS 内嵌)/ 0 RTT(复用) | 2~3 RTT |
| 连接迁移 | **IP 变了,连接还在**(WiFi 切 4G) | 必须重连 |
| 拥塞控制 | 用户态实现,可热升级 | 内核态,升级慢 |
| 丢包恢复 | 更智能 | TCP 标准 |

### 为什么 QUIC 用 UDP 不直接改 TCP

- TCP 在内核,改一次几年才能上所有机器
- 中间盒(防火墙、NAT)对 TCP 行为有死板假设,新特性走不通
- UDP 是"裸传输",QUIC 在用户态自由实现

> 经验法则:**移动端、跨国、网络差的场景,HTTP/3 比 HTTP/2 P99 延迟低 30%~50%**。但部署复杂(UDP 经常被运营商限速 / QoS,要做 fallback 到 HTTP/2)。

### 怎么让浏览器知道用 HTTP/3

服务端响应 HTTP/2 时带:

```
Alt-Svc: h3=":443"; ma=86400
```

浏览器下次访问就走 HTTP/3。

## 七、DNS:被忽视的延迟元凶

```
请求 → DNS 查询 → TCP 握手 → TLS 握手 → 发请求 → 收响应

DNS 一查可能 50~200ms,但很多人根本不监控
```

**优化点**:

| 优化 | 收益 |
| --- | --- |
| **DNS 缓存(本地 / Resolver)** | TTL 内零开销 |
| **DNS over HTTPS / TLS** | 防劫持(运营商劫持是国内常见问题) |
| **HTTPDNS** | App 直接用 HTTP 查 DNS,绕过运营商 LocalDNS |
| **EDNS Client Subnet** | CDN 智能调度需要 |
| **预解析(Prefetch)** | 浏览器 `<link rel="dns-prefetch">` |

**JVM 默认 DNS 缓存有坑**:

```
networkaddress.cache.ttl=-1       (默认永久缓存)
networkaddress.cache.negative.ttl=10
```

`-1` 意味着**JVM 启动后第一次解析的 IP 永远用**——目标服务换 IP 你不知道。

```java
java.security.Security.setProperty("networkaddress.cache.ttl", "30");
```

## 九、连接池:服务端 / 客户端两侧都要

### 客户端连接池

```java
// HTTP Client(Java 11+)
HttpClient client = HttpClient.newBuilder()
    .version(Version.HTTP_2)
    .connectTimeout(Duration.ofSeconds(2))
    .build();   // 默认有连接池

// OkHttp
OkHttpClient client = new OkHttpClient.Builder()
    .connectionPool(new ConnectionPool(50, 5, TimeUnit.MINUTES))
    .build();

// Apache HttpClient 5
PoolingHttpClientConnectionManager cm = PoolingHttpClientConnectionManagerBuilder.create()
    .setMaxConnTotal(200)
    .setMaxConnPerRoute(50)
    .build();
```

> 经验法则:**任何 HTTP 客户端必须配连接池**——每次新建连接 = 每次重新 TCP+TLS 握手 = 每次多几百 ms 延迟 + 把目标服务的端口耗尽。

### 服务端常见参数(Tomcat)

```yaml
server:
  tomcat:
    threads:
      max: 200
      min-spare: 20
    accept-count: 100         # 队列长度
    max-connections: 8192     # 同时存活连接数
    connection-timeout: 20s
    keep-alive-timeout: 60s
```

## 十、长连接 vs 短连接

| 场景 | 推荐 |
| --- | --- |
| 后端互调(微服务) | **长连接 + 连接池** |
| 数据库 / Redis | **长连接 + 连接池** |
| 浏览器 → 服务 | HTTP/1.1 Keep-Alive,HTTP/2 自动复用 |
| 移动端 → 服务 | 长连接(WebSocket / gRPC bidi) |
| 一次性脚本 | 短连接简单 |

## 十一、性能调优清单

```
路径上的每一跳都要查:

1. DNS:cache TTL、HTTPDNS
2. TCP:握手次数(连接复用)、BBR、TIME_WAIT
3. TLS:1.3、Session Resumption、OCSP Stapling
4. HTTP:HTTP/2、Keep-Alive、Header 压缩
5. 应用:连接池、线程池、Compress
6. CDN:边缘缓存、回源策略
7. 监控:每段延迟分别测
```

var resp = client.send(req, BodyHandlers.ofString());
log.info("total={}ms", (System.nanoTime() - start) / 1_000_000);

## 十二、常见踩坑

1. **不开 Keep-Alive**:每个请求都重新 TCP+TLS 握手,延迟翻三倍
2. **HTTP/2 但没 TLS**:浏览器拒绝,默认仍走 HTTP/1.1
3. **TLS 用 1.0/1.1**:扫描器报红、PCI DSS 不过、客户审计挂
4. **证书快过期才发现**:周末半夜全站 502
5. **JVM DNS TTL 默认 -1**:目标服务换 IP 不感知
6. **HttpClient 不复用 instance**:每次 new,连接池白配
7. **客户端连接池太小**:压测时一堆等连接,看似 server 慢实则客户端瓶颈
8. **没设 connectTimeout / readTimeout**:对端僵死时调用方线程被吃光
9. **TIME_WAIT 堆积**:压测机器端口耗尽
10. **开 `tcp_tw_recycle`**:NAT 后用户随机失败(且新内核已删)
11. **Server Push 寄希望太大**:Chrome 已废,加了反而拖慢
12. **不监控 DNS 解析时间**:DNS 抖动以为是后端慢
13. **公司内网都不开 TLS**:东西流量没加密,内网漫游就漏数据
14. **HTTP/3 全量上,UDP 被运营商限速**:必须做 fallback
15. **gzip 压缩开了但 br 没开**:Brotli 比 gzip 小 20% 还快

## 小结

网络协议是**所有服务的"地基的地基"**——业务再复杂,最后都得过 TCP/TLS/HTTP。

记住三件事:

1. **每一层 RTT 都是钱**——能省一次握手,延迟立省一档
2. **加密不慢,慢的是协商**——TLS 1.3 + Session Resumption 之后 TLS 几乎零成本
3. **HTTP/3 不是噱头**——移动端、跨国 P99 真的能砍一半
