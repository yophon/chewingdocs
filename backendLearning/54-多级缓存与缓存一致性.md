# 多级缓存与缓存一致性

18~20 章把 Redis 讲完了——但只覆盖到"缓存层在 Redis"。真实的高并发系统**缓存是多层的**:JVM 进程内本地缓存 → 分布式 Redis → CDN/边缘缓存,每一层解决不同问题。

更难的是**一致性**:数据库改了,缓存怎么办?Cache-Aside / Write-Through / Write-Behind 各有什么坑?这章把缓存这一栈的"软"知识补齐。

---

## 一、为什么要"多级"

```
请求路径(从快到慢):
   浏览器/CDN(0ms)
        ↓ 没命中
   边缘缓存(5~20ms,跨地域)
        ↓
   API 网关缓存(<1ms,同机房)
        ↓
   ★ 进程内本地缓存(纳秒级,同 JVM)
        ↓
   ★ Redis(分布式,1~5ms 内网)
        ↓
   ★ DB(磁盘 / SSD,10~100ms)
```

每加一层都拿"内存换性能",越靠近用户越便宜。

| 层 | 解决的问题 |
| --- | --- |
| **CDN / 边缘** | 静态资源、跨地域延迟 |
| **本地缓存(Caffeine)** | 减少网络往返,扛"穿透到 Redis"的流量 |
| **Redis 集群** | 跨实例共享、热数据池 |
| **DB** | 真相之源,缓存最终回源 |

> 经验法则:**单机 QPS > 1k 的服务都该用本地缓存**。Redis 再快,网络往返也比内存读慢 1000 倍——能在本地命中,延迟 P99 立刻降一档。

---

## 二、Caffeine:JVM 进程内缓存的事实标准

Guava Cache 的"现代版",**作者就是原 Guava Cache 作者**——更快、API 更新、内置自适应淘汰。

```java
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)                      // 容量上限
    .expireAfterWrite(Duration.ofMinutes(5))  // 写入后 5 分钟过期
    .refreshAfterWrite(Duration.ofMinutes(1)) // 1 分钟后异步刷新(读时返回旧值,后台拉新值)
    .recordStats()                            // 命中率统计
    .build(key -> userMapper.selectById(key));  // miss 时回源

User u = cache.get("u-1");
```

### Caffeine 的几个核心特性

| 特性 | 含义 |
| --- | --- |
| **W-TinyLFU** | 比 LRU 命中率高 30%~50%,扛得住"突发热点" |
| **expireAfterWrite vs Access** | 按写入 vs 按最后访问算过期 |
| **refreshAfterWrite** | 过期前异步刷新,**避免击穿** |
| **AsyncLoadingCache** | 异步加载,适配 CompletableFuture / 虚拟线程 |
| **Weight** | 按值大小算容量,大对象不要按数量算 |

```java
// 异步缓存 + 异步加载
AsyncLoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(5))
    .buildAsync((key, exec) -> CompletableFuture.supplyAsync(() -> load(key), exec));

cache.get("u-1").thenAccept(u -> ...);
```

> 经验法则:**Caffeine 的 `refreshAfterWrite` 是防击穿的银弹**——key 过期了不阻塞用户请求,后台拉新。Redis 配合上 Caffeine,99% 的"缓存击穿"问题就消失。

## 三、Spring Boot 一行接 Caffeine

```yaml
spring:
  cache:
    type: caffeine
    caffeine:
      spec: maximumSize=10000,expireAfterWrite=5m,recordStats
```

```java
@Service
public class UserService {

    @Cacheable(value = "users", key = "#id")
    public User findById(Long id) { ... }

    @CachePut(value = "users", key = "#user.id")
    public User update(User user) { ... }

    @CacheEvict(value = "users", key = "#id")
    public void delete(Long id) { ... }
}
```

**Spring Cache 抽象的好处**:换底层缓存(Caffeine ↔ Redis ↔ EhCache)只需改 yaml,业务代码不动。

## 四、双层缓存(L1 Caffeine + L2 Redis)

最常见的多级缓存形态:

```
请求 → Caffeine(L1, 本地) ── miss → Redis(L2, 集群) ── miss → DB
                                       │
                                       └─ 命中 → 回写 L1
```

```java
@Service
public class TwoLevelCache {
    private final Cache<String, Product> l1 = Caffeine.newBuilder()
        .maximumSize(10_000).expireAfterWrite(1, MINUTES).build();
    @Autowired private RedisTemplate<String, Product> redis;
    @Autowired private ProductMapper db;

    public Product get(Long id) {
        String key = "product:" + id;
        Product p = l1.getIfPresent(key);
        if (p != null) return p;

        p = redis.opsForValue().get(key);
        if (p != null) { l1.put(key, p); return p; }

        p = db.selectById(id);
        if (p != null) {
            redis.opsForValue().set(key, p, 10, MINUTES);
            l1.put(key, p);
        }
        return p;
    }
}
```

### L1 的麻烦:多实例不一致

```
[实例A: L1 缓存 product:1 = "旧"]
[实例B: L1 缓存 product:1 = "旧"]
   ↓ 用户在实例A 上更新
[实例A: 删了自己 L1, 改了 DB, 删了 Redis]
[实例B: L1 还是"旧"]   ←  ⚠️ 不一致
```

破法三选一:

| 方案 | 工作方式 | 一致性窗口 |
| --- | --- | --- |
| **TTL 短** | L1 设 30s 过期,接受短期不一致 | 30s |
| **Redis Pub/Sub 通知** | 改 DB 时广播,所有实例清自家 L1 | <100ms |
| **Caffeine + 消息队列(MQ)** | 用 Kafka topic 广播失效 | <500ms |

> 经验法则:**L1 缓存只放"短期一致即可"的数据**——商品基础信息、字典、配置。**用户余额、订单状态这种"必须最新"的不进 L1**。

## 五、缓存读写四种模式

### 1. Cache-Aside(旁路缓存,最常用)

```
读:先查缓存,miss 查 DB,回填缓存
写:先写 DB,再删缓存(注意:删而非更新)
```

```java
public Product get(Long id) {
    Product p = redis.get("product:" + id);
    if (p == null) {
        p = db.find(id);
        if (p != null) redis.set("product:" + id, p, 10, MINUTES);
    }
    return p;
}
public void update(Product p) {
    db.update(p);
    redis.delete("product:" + p.getId());
}
```

**为什么是"删"不是"更新"**:

- 减少不一致窗口(更新有"DB 改完缓存覆盖"的乱序风险)
- 懒加载——下一次读才回填,避免"写多读少"场景缓存被无意义刷新

### 2. Read-Through

应用只跟缓存说话,缓存自己负责回源:

```java
LoadingCache<Long, Product> cache = Caffeine.newBuilder()
    .build(id -> db.find(id));    // 缓存自动 miss 回源

Product p = cache.get(id);   // 不用关心 DB
```

简化业务代码,**Caffeine 天然支持**。

### 3. Write-Through(写穿透)

写操作同步穿到 DB:

```java
public void update(Product p) {
    cache.put(p.getId(), p);
    db.update(p);   // 同步到 DB
}
```

强一致,但写性能受 DB 牵制。

### 4. Write-Behind(写回 / Write-Back)

写只到缓存,**异步**批量到 DB:

```java
public void update(Product p) {
    cache.put(p.getId(), p);
    queue.add(p);   // 异步 worker 批量入库
}
```

写吞吐爆炸,**但故障时丢数据**——只用在能容忍丢的场景(浏览次数、点赞计数)。

> 经验法则:
>
> - 默认用 **Cache-Aside**(占 90% 场景)
> - 业务写入很重 + 一致性必须 → **Write-Through**
> - 计数器、统计类 → **Write-Behind**(可丢失)
> - 配合 Caffeine 用 **Read-Through** 简化代码

## 六、一致性的真相:三种"先后"组合

更新场景下,DB 和缓存先后顺序的组合(以下都假设 Cache-Aside + 删):

### 1. 先删缓存,再更新 DB

```
T1: 删缓存
T2(并发): 读,miss → 查 DB(还是旧)→ 回填缓存(旧)
T1: 更新 DB(新)
==> 缓存里是旧的,长期不一致 ❌
```

### 2. 先更新 DB,再删缓存(主流)

```
T1: 更新 DB
T1: 删缓存
T2(并发): 读,miss → 查 DB(新)→ 回填(新)
==> OK ✅
```

但仍有罕见竞态:

```
T2: 读,miss → 查 DB(旧)→ 还没回填
T1: 更新 DB → 删缓存(此时缓存空)
T2: 回填(旧)  ←  ⚠️ 缓存又变旧了
```

概率极低(查 DB 的耗时要长于完整更新流程),工程上一般忽略。

### 3. 延时双删(防御上面那个罕见竞态)

```
更新 DB → 删缓存 → 等 500ms → 再删一次
```

第二次删能清掉竞态期写进去的旧值。**简单粗暴,牺牲一点点性能**。

### 4. 订阅 binlog(终极方案)

用 46 章的 CDC(Debezium / Canal):

```
DB 写 → binlog → Canal/Debezium → MQ → 缓存失效消费者 → 删缓存
```

**业务代码完全不知道有缓存**,缓存自动跟 DB 同步。**缺点**:架构复杂度上一档,需要数据团队介入。

> 经验法则:
>
> - 99% 用 **Cache-Aside + 先 DB 后删**
> - 对一致性敏感再加 **延时双删**
> - 业务方零侵入要求 → **CDC 订阅 binlog**

## 七、缓存三大故障:穿透 / 击穿 / 雪崩

### 1. 穿透(Penetration):查不存在的 key

恶意请求 `id=-1, -2, -3...`,DB 里都没有,缓存层永远 miss,**每次都打 DB**。

**破法**:

| 方案 | 优劣 |
| --- | --- |
| **缓存空值** | `redis.set(key, "NULL", 60s)`——简单,但浪费内存 |
| **布隆过滤器** | 上游拦截:`bloom.contains(id)` 不在直接拒;**节省内存,有少量误判** |

```java
// Redisson 布隆过滤器
RBloomFilter<Long> bloom = redisson.getBloomFilter("product:exists");
bloom.tryInit(10_000_000L, 0.01);    // 1000 万容量,误判率 1%

if (!bloom.contains(id)) return null;     // 直接挡掉
return get(id);                           // 走正常缓存逻辑
```

### 2. 击穿(Breakdown):热点 key 过期瞬间

某个超热 key(首页推荐、爆款商品)正好过期,**几万请求同时打 DB**。

**破法**:

| 方案 | 工作方式 |
| --- | --- |
| **互斥锁** | miss 时 `SETNX` 占锁,只放一个去查 DB,其余等结果 |
| **Caffeine refreshAfterWrite** | 过期前后台异步刷新,用户始终拿到值 |
| **永不过期 + 后台更新** | 业务级"逻辑过期",不靠 TTL |
| **多级缓存** | L1 + L2,L2 过期时 L1 还能挡一会 |

```java
// 互斥锁防击穿
public Product getWithMutex(Long id) {
    String key = "product:" + id;
    Product p = redis.get(key);
    if (p != null) return p;

    String lockKey = "lock:" + key;
    if (redis.setIfAbsent(lockKey, "1", 5, SECONDS)) {
        try {
            p = db.find(id);
            redis.set(key, p, 10, MINUTES);
        } finally { redis.delete(lockKey); }
    } else {
        sleep(50);
        return getWithMutex(id);   // 等一会重试
    }
    return p;
}
```

### 3. 雪崩(Avalanche):大量 key 同时过期

定时任务在凌晨刷了一万个 key 都设 1 小时 TTL → 第二天凌晨 1 小时后**集体过期**,**DB 瞬间崩**。

**破法**:

- TTL 加随机抖动:`5min + random(0~60s)`,避免同时过期
- **过期时间错峰**:批量回填时随机化
- **多级缓存** + L1 抗一波

> 经验法则:**所有 TTL 都加 ±10% 随机**——一行代码避免雪崩。

## 八、热点 key 的特殊处理

某个 key 突然变得超热(秒杀商品),**单 Redis 实例被打爆**。

**破法**:

| 方案 | 适用 |
| --- | --- |
| **本地缓存(L1)** | 把热点放本地,Redis 只承担"miss" |
| **多副本拆 key** | `product:1` 拆成 `product:1:0`...`product:1:9`,客户端随机选一个,分摊到不同 slot |
| **客户端短缓存** | 网关/SDK 层 1 秒缓存 |
| **预热** | 大促前主动加载 |

热点检测可以用**Redis 4.0+ 的 hotkeys 命令** 或 Sentinel 自动识别。

## 九、缓存 Key 设计

```
推荐:bizDomain:type:id[:子维度]
例:user:profile:123
   product:detail:456:zh-CN
   order:list:user:123:page:1

避免:
  1.全用 user_123(没命名空间,清不了)
  2.太长(> 1KB,memcache 直接挂,Redis 浪费内存)
  3.带空格 / 中文(可读但易坑)
  4.把 SQL 当 key(几 KB,且改 SQL 全失效)
```

**版本化**:`product:v2:123`,Schema 升级时直接换 v3,旧版本自然过期。

## 十、缓存监控:命中率是核心 KPI

```java
CacheStats stats = caffeineCache.stats();
log.info("hit={}, miss={}, hitRate={}, evictions={}", 
    stats.hitCount(), stats.missCount(), stats.hitRate(), stats.evictionCount());
```

接到 Prometheus:

```java
@Bean MeterBinder caffeineMetrics(Cache<?,?> cache) {
    return new CaffeineCacheMetrics<>(cache, "user_cache", Tags.empty());
}
```

| 指标 | 健康范围 |
| --- | --- |
| **命中率** | > 80%(<60% 几乎白缓存) |
| **平均 load 时间** | <100ms |
| **驱逐速率(evictions)** | 突增 = 容量不够 |
| **size / maxSize** | 长期触顶 = 加容量 |

> 经验法则:**没监控的缓存等于没缓存**——命中率掉到 30% 都不知道,白浪费内存。

## 十一、缓存预热

冷启动 / 滚动发布时,缓存空,**第一波流量直接打 DB**。

```java
@PostConstruct
public void warmUp() {
    var hotIds = db.findTop1000HotProducts();
    hotIds.parallelStream().forEach(this::get);   // 加载到缓存
}
```

更工程化做法:

- 启动后**先关闭健康检查**,预热完再放流量
- K8s `readinessProbe` 等预热完成
- 蓝绿发布时新版本预热到一定命中率再切流量

## 十二、Redis 集群下的本地缓存广播

Caffeine 失效广播两种工程实现:

### 1. Redis Pub/Sub(轻量,够用)

```java
// 写入端
public void update(Product p) {
    db.update(p);
    redis.delete(key);
    redis.convertAndSend("cache.invalidate", key);  // 广播
}
// 读取端订阅
@Bean MessageListenerAdapter cacheInvalidator(LocalCache l1) {
    return new MessageListenerAdapter((MessageListener) (msg, p) -> {
        l1.invalidate(new String(msg.getBody()));
    });
}
```

### 2. Kafka 通知(可靠,可回放)

```
DB 写 → Kafka topic "cache.evict" → 所有实例消费 → 清自家 L1
```

适合**实例多、要保证不丢失**的场景。

## 十三、常见踩坑

1. **缓存大对象**:一个 200KB 的对象被频繁读,网卡先打满
2. **缓存空值不设 TTL**:占一辈子内存
3. **批量操作循环查 Redis**:每次 RTT 1ms × 1000 个 key = 1s,**用 mget / pipeline**
4. **Hot Key 没识别**:某个 key QPS 占了集群一半,别的 key 被排队
5. **缓存 KEY 模糊匹配 keys *** :Redis 单线程,几百万 key 时直接卡死,**用 SCAN**
6. **缓存击穿没 mutex**:热点 key 过期,DB CPU 100%
7. **缓存雪崩没加随机 TTL**:整点 DB 崩
8. **L1 缓存订单状态**:用户付款后还看到"待付款"
9. **更新 DB 后忘删缓存**:30 分钟脏数据
10. **缓存 = DB 的镜像**:全量缓存大表(几亿条),内存撑爆;只缓存"热数据"
11. **同步 DB binlog 到缓存,但消费有延迟**:延迟 30s,业务以为"实时"
12. **Cache-Aside 删失败没重试**:缓存就是旧的;删失败要进 MQ 重试 / 加报警
13. **本地缓存放业务对象,不是不可变的**:对象引用被外部修改,缓存内容跟着变
14. **不监控命中率**:发现是的时候已经性能事故了

## 十四、本章 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ 单机 QPS > 1k 上 Caffeine | 本地命中比 Redis 快 1000 倍 |
| ✅ Caffeine refreshAfterWrite 防击穿 | 异步刷新,用户始终有值 |
| ✅ Cache-Aside + 先 DB 后删 | 默认模式 |
| ✅ TTL 加 ±10% 随机 | 防雪崩 |
| ✅ 布隆过滤器 / 缓存空值 | 防穿透 |

| ✅ L1 不放强一致数据 | 余额、订单状态走 L2 |
| ✅ Pub/Sub / Kafka 广播 L1 失效 | 多实例一致性 |
| ✅ 命中率监控 + 报警 | <70% 立刻看 |
| ✅ 大 key / 热 key 检测 | 提前发现 |
| ✅ 启动预热,readiness 等就绪 | 冷启动 DB 不被打爆 |

## 小结

缓存的本质是**用空间换时间 + 用一致性换性能**。设计时永远问自己三件事:

1. **能容忍多大延迟的数据不一致?** 决定 TTL 和模式
2. **谁是热点 key?** 决定要不要本地缓存
3. **DB 在缓存挂的时候能不能扛?** 决定 fallback 策略

记住三句:

1. **缓存不是 DB 的镜像,是热点的副本**——别想着全量同步
2. **TTL 永远加抖动**,这是预防雪崩的最便宜手段
3. **没监控的缓存等于没缓存**,命中率必须看

下一章我们往下钻一层——**TCP / TLS / HTTP/2 / HTTP/3 / QUIC**。做网关、做中间件、做性能调优,这一层是必须的功底。
