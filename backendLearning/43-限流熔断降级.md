# 限流、熔断、降级

36 章压测告诉你"系统能扛多少",这一章解决另一个问题:**扛不住的时候怎么办?**

答案是高可用三板斧——**限流(挡)、熔断(断)、降级(让)**。三者各管一件事,组合起来才能让系统在过载时优雅降级,而不是雪崩。

---

## 一、三者的角色分工

| 板斧 | 解决什么 | 何时触发 | 类比 |
| --- | --- | --- | --- |
| **限流(Rate Limiting)** | 入口流量超过承载能力 | 请求 QPS 超阈值 | 商场门口排队限流 |
| **熔断(Circuit Breaker)** | 下游不可用,自己被拖死 | 依赖错误率 / 耗时超阈值 | 电路保险丝跳闸 |
| **降级(Fallback)** | 关键路径出问题,要保核心 | 上述两种触发 / 主动开关 | 飞机迫降:扔货保人 |

```
   流量来 ──▶ [ 限流 ] ──▶ [ 你的服务 ] ──▶ [ 熔断 ] ──▶ [ 下游服务 ]
                  │                              │
                  ▼                              ▼
              拒绝请求                      调用降级逻辑
                                                │
                                                ▼
                                            返回兜底数据 / 缓存
```

> 经验法则:**限流保入口、熔断保下游、降级保自己**。三者不能互相替代——只做一个等于没做。

---

## 二、限流:四种经典算法

### 1. 计数器(固定窗口)

```
每 1 秒重置计数,超过 1000 拒绝
```

**问题**:窗口边界容易被刷穿——0.99s 来 1000 请求 + 1.01s 来 1000 请求,实际 0.02 秒内 2000 请求。

### 2. 滑动窗口

把 1 秒切成 10 个 100ms 子窗口,统计最近 10 个子窗口总和。**比固定窗口平滑,实现稍复杂**。

### 3. 漏桶(Leaky Bucket)

```
请求 ──▶ [桶,容量 N] ──▶ [固定速率 R 流出]
桶满了 → 拒绝
```

**特点**:出流速率严格恒定,适合**整形**(把毛刺流量平滑掉)。

### 4. 令牌桶(Token Bucket)

```
[令牌生成器,速率 R] → [桶,容量 N]
                          │
                          ▼
请求来 → 拿令牌 → 没令牌就拒绝
```

**特点**:允许**突发**——桶里攒着的令牌可以瞬时消耗。多数场景的最佳选择。

| 算法 | 平滑度 | 允许突发 | 实现 |
| --- | --- | --- | --- |
| 固定窗口 | 差 | 是 | 最简单 |
| 滑动窗口 | 好 | 是 | 稍复杂 |
| 漏桶 | 极好 | 否 | 中等 |
| **令牌桶** | 好 | 是 | 中等 |

> 经验法则:**业务限流首选令牌桶**——既能限速,又能容忍合理的瞬时突发。

---

## 三、限流的"维度"比"算法"更重要

很多人卡在选算法上,实际上**选错维度才是大问题**:

| 维度 | 何时用 |
| --- | --- |
| **全局 QPS** | 保护后端总容量 |
| **单 IP 限流** | 防爬虫、防刷 |
| **单用户限流** | 防同一账号高频操作 |
| **接口粒度** | 不同接口阈值不同(支付严格、查询宽松) |
| **租户/商家粒度** | SaaS 多租户隔离 |
| **资源维度** | 数据库连接池、线程池本身就是限流 |

实战中往往是**多维组合**:

```
全局: 10000 QPS
  ├─ 单 IP: 100 QPS
  ├─ 单用户: 10 QPS  
  └─ 接口 /pay: 全局 1000 QPS
```

---

## 四、限流放在哪一层

```
[ CDN / WAF ]   ◀── 防 DDoS,粗粒度
     │
[ 网关层 ]      ◀── 全局 QPS、IP 限流(主战场)
     │
[ 应用层 ]      ◀── 业务限流(接口、用户、租户)
     │
[ 资源层 ]      ◀── DB 连接池、线程池(隐形限流)
```

> 经验法则:**越上层做得越粗、越早拦截越省钱**。能在网关挡的别让请求走到应用层。

---

## 五、熔断器状态机

熔断不是简单的开关,而是一个状态机:

```
       失败率 / 慢调用率达阈值
   ┌──────────────────────┐
   │                      ▼
[CLOSED]             [OPEN]
正常放行             直接拒绝(快速失败)
   ▲                      │
   │  半开期探测请求成功   │  等冷却时间(如 10s)
   │                      ▼
   └──────────────── [HALF-OPEN]
                     放少量请求试探
```

| 状态 | 行为 |
| --- | --- |
| **CLOSED** | 正常调用,统计错误率/耗时 |
| **OPEN** | 直接拒绝,不发出真实请求(给下游喘息) |
| **HALF-OPEN** | 放少量请求试探,成功转 CLOSED,失败转 OPEN |

触发条件常用两个:**错误率**(>50%)、**慢调用比例**(RT > 1s 占比 > 30%)。

---

## 六、为什么熔断比"重试"更重要

**反直觉的事实**:下游故障时,**重试是雪崩的加速器**。

```
正常:1 个请求 → 1 次调用
故障 + 重试 3 次:1 个请求 → 4 次调用
↓
本就承受不住的下游被打得更惨
↓
更多服务超时
↓
更多重试
↓
雪崩
```

熔断的核心价值就是 **"承认下游挂了,先别打它"**——给下游恢复机会,顺带保护自己的资源(线程、连接)不被慢调用占满。

## 七、降级:你必须想清楚的三个问题

降级 = **"返回不完美但可用的结果"**。设计降级前必须回答:

1. **哪些功能可以降级?**
   - 推荐、广告、个性化:可以(返回热门列表)
   - 支付、下单:不行(必须强一致)
2. **降级返回什么?**
   - 缓存的旧数据
   - 默认值 / 空列表
   - 前置在客户端的兜底
3. **降级是自动还是手动?**
   - 自动:熔断触发自动 fallback
   - 手动:运营开关(双 11 关掉评论列表保下单)

```java
@SentinelResource(value = "getUserProfile", fallback = "getProfileFallback")
public UserProfile getUserProfile(Long userId) {
    return userRpc.getUser(userId);  // 远程调用
}

public UserProfile getProfileFallback(Long userId, Throwable t) {
    return UserProfile.guest();        // 兜底返回访客信息
}
```

> 经验法则:**核心链路不许降级,辅助功能必须能一键关掉**。这两条画清楚,系统的"鲁棒性"就上来了。

## 八、主流框架对比

| 框架 | 出身 | 强项 | 现状 |
| --- | --- | --- | --- |
| **Hystrix** | Netflix | 老牌,把熔断带火 | 已停止维护,新项目别用 |
| **Resilience4j** | Java 8+ 重写 Hystrix | 函数式 API、模块化 | Spring 官方推荐 |
| **Sentinel** | 阿里 | 限流/熔断/系统保护一体、控制台强 | 国内主流 |
| **Envoy / Istio** | Service Mesh | 在 sidecar 层做,业务零侵入 | 云原生潮流方向 |

## 九、Resilience4j 实战

```yaml
resilience4j:
  circuitbreaker:
    instances:
      payment:
        sliding-window-size: 20
        minimum-number-of-calls: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 10s
        permitted-number-of-calls-in-half-open-state: 3
  ratelimiter:
    instances:
      payment:
        limit-for-period: 100
        limit-refresh-period: 1s
        timeout-duration: 0
  retry:
    instances:
      payment:
        max-attempts: 3
        wait-duration: 200ms
```

```java
@Service
class PaymentClient {
    @CircuitBreaker(name = "payment", fallbackMethod = "fallback")
    @RateLimiter(name = "payment")
    @Retry(name = "payment")
    public PayResp pay(PayReq req) {
        return restTemplate.postForObject("/pay", req, PayResp.class);
    }
    PayResp fallback(PayReq req, Throwable t) {
        return PayResp.pendingManualReview(req.getOrderId());
    }
}
```

注解的执行顺序:`Retry → CircuitBreaker → RateLimiter → Bulkhead → 业务`(从外到内)。

> ⚠️ **Retry + CircuitBreaker 一起用要小心**:重试在熔断之外 → 熔断后还重试是浪费;重试在熔断之内 → 熔断器统计错误率会被重试放大。Resilience4j 默认外层是 Retry,符合大多数预期。

## 十、Sentinel 实战

Sentinel 把限流、熔断、系统保护、热点参数限流都整合在一起,**控制台动态推规则**是它最爽的特性。

```java
@SentinelResource(value = "createOrder",
        blockHandler  = "blockHandler",   // 限流时走这里
        fallback      = "fallback")       // 业务异常时走这里
public OrderResp createOrder(OrderReq req) {
    return orderService.create(req);
}

public OrderResp blockHandler(OrderReq req, BlockException e) {
    return OrderResp.tooBusy();
}
public OrderResp fallback(OrderReq req, Throwable t) {
    return OrderResp.fail("系统繁忙");
}
```

控制台里可以动态加规则:

| 规则类型 | 用途 |
| --- | --- |
| 流控规则 | QPS 限流、并发线程数限流 |
| 熔断规则 | 慢调用比例、异常比例、异常数 |
| 热点规则 | 单 userId / 单商品 ID 维度 |
| 系统规则 | 全局 Load / CPU / 入口 QPS 自适应 |
| 授权规则 | 黑白名单 |

**热点参数限流**是 Sentinel 的杀手锏——比如同一商品 ID 在秒杀场景下要单独限流,直接配规则,不用写代码。

## 十一、网关侧 vs 应用侧

| 侧 | 优势 | 劣势 |
| --- | --- | --- |
| **网关侧**(Spring Cloud Gateway / Kong / Nginx + Lua) | 统一拦截,业务无侵入,粗粒度限流轻量 | 拿不到业务上下文(只能按 URL/IP) |
| **应用侧**(Sentinel / Resilience4j) | 能感知业务(按用户、按租户) | 每个服务都要配,治理成本高 |

> 经验法则:**网关做粗粒度,应用做细粒度,两层叠加**。网关挡住明显恶意流量,应用做业务级精细控制。

## 十二、Service Mesh 视角

Istio / Envoy 把限流熔断从应用代码移到 sidecar:

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata: { name: payment-cb }
spec:
  host: payment
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
    connectionPool:
      tcp: { maxConnections: 100 }
      http: { http1MaxPendingRequests: 50 }
```

**好处**:多语言统一治理,业务零代码。
**代价**:架构复杂度上升、调试难、性能轻微开销。

## 十三、过载保护:系统级最后一道防线

应用级限流再强,有时候 CPU/Load 还是顶不住——比如慢查询打满 IO、GC 长停顿、磁盘满。这时需要**自适应保护**:

```
基于系统指标自动降级:
  CPU > 80% → 拒绝低优先级请求
  Load > 8  → 触发限流
  RT P99 > 5s → 熔断
```

Sentinel 的"系统规则"、Linux 的 cgroups、JVM 的 GC throttle、内核的 OOM killer——一层一层都是过载保护。

## 十四、常见踩坑

1. **限流值拍脑袋定**:不压测就定阈值,设太小限误杀,设太大没用。**先压测再设阈值**
2. **限流维度太粗**:只做全局 QPS,几个大商家把容量吃光,小商家完全用不上。要分维度
3. **熔断只看错误率,不看 RT**:慢调用比错误更危险——错误快速返回,慢调用占线程
4. **降级返回 null 没文档**:调用方一脸懵——拿到 null 是没数据还是降级了?要有明确语义
5. **重试无限叠加**:网关重试 3 次 + 业务重试 3 次 + RPC 重试 3 次 = 27 次,雪崩加速
6. **Hystrix 还在用**:升级到 Resilience4j
7. **限流没有可观测性**:被限了多少、谁被限了——不知道,运营无法判断阈值合不合理
8. **降级路径没演练过**:真触发时降级逻辑也挂了
9. **熔断窗口太短**:刚 OPEN 又 HALF-OPEN,跟没熔断一样
10. **不区分核心 / 非核心接口**:把所有接口同等对待,关键时刻保不住核心

## 十五、本章 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ 三板斧都要有 | 限流、熔断、降级缺一不可 |
| ✅ 限流维度细化 | 全局 + IP + 用户 + 接口 |
| ✅ 限流阈值靠压测定 | 不要拍脑袋 |
| ✅ 熔断同时看错误率与 RT | 慢调用更致命 |
| ✅ 核心 / 非核心区分清楚 | 降级时知道保谁 |
| ✅ 降级路径要演练 | 真触发时别再挂 |
| ✅ 重试有退避 + 上限 | 不然雪崩加速 |
| ✅ 监控覆盖三板斧 | 限了多少、断了多少、降了多少 |
| ✅ 系统级自适应保护兜底 | 应用级限流之外 |
| ✅ 控制台可动态调阈值 | 不重启改规则 |

## 小结

限流、熔断、降级是构建高可用系统的"必修课"——它们不是可选项,是底线。

设计这一套时常常陷入"先做哪个"的困惑,记住这个顺序:

1. **先把核心 / 非核心分清**(决定降级策略)
2. **再做接口限流**(挡住瞬时流量)
3. **再加熔断**(隔离下游故障)
4. **最后做系统自适应保护**(兜底)
