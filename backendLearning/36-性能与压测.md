# 性能与压测（JMeter / wrk / k6）

写完业务、上完监控，下一个绕不开的问题是：**这套系统到底能扛多少？**

光看 CPU 没意义——单台 8C 16G 在某些业务能跑 5 万 QPS，在另一些跑不到 500。**必须压**才知道真实容量，才能做扩容、做预算、做容灾演练。

---

## 一、几个必须分清的指标

| 指标 | 含义 | 好不好 |
| --- | --- | --- |
| **QPS / TPS** | 每秒请求数 / 事务数 | 高更好 |
| **并发数** | 同时在飞的请求数 | 取决于场景 |
| **RT（Response Time）** | 单次响应耗时 | 低更好 |
| **吞吐量（Throughput）** | 每秒处理的数据量 | 高更好 |
| **延迟分布** | P50 / P95 / P99 / P999 | 看尾延迟 |
| **错误率** | 失败数/总数 | < 0.1% |

**Little's Law**：

```
平均并发 = 吞吐量 × 平均 RT
```

举例：QPS = 1000、平均 RT = 200ms，那系统里平均同时有 200 个请求在跑。这条公式能让你判断"压测客户端的并发数够不够把服务压满"。

---

## 二、为什么 P50 没用，要看 P99

```
请求耗时分布（QPS=1000）

    ┌────────────┐
P50 │ 50ms       │   一半请求快得很
    ├────────────┤
P95 │ 300ms      │
    ├────────────┤
P99 │ 1.2s       │   1% 用户在哭
    ├────────────┤
P999│ 5s         │   0.1% = 每天数千次
    └────────────┘
```

业务感知靠的是尾巴，**P99 才是用户体验**。优化方向永远是"压低 P99"，看 P50 永远岁月静好。

## 三、三大压测工具对比

| 工具 | 实现 | 强项 | 弱项 |
| --- | --- | --- | --- |
| **wrk** | C / Lua | 极简、单机能压几十万 QPS | 报表弱、场景复杂时难写 |
| **k6** | Go + JS 脚本 | 现代化、CI 友好、报表好 | JS 性能弱于 wrk |
| **JMeter** | Java + GUI | 协议全（HTTP/JDBC/MQ）、企业老用户多 | 重、单机压力上不去 |
| Vegeta | Go | 恒定速率压、命令行友好 | 场景不能太复杂 |
| Locust | Python | 写脚本灵活、分布式好 | 单机性能差 |
| Gatling | Scala | 报表精美、企业级 | 学习曲线陡 |

> 经验法则：**接口快测用 wrk，业务场景压测用 k6，企业全家桶要 JDBC/SOAP/JMS 用 JMeter**。

## 四、wrk 极简实战

```bash
# 4 线程、200 并发、压 30 秒
wrk -t4 -c200 -d30s http://api.example.com/users/42

# 输出
Running 30s test @ http://api.example.com/users/42
  4 threads and 200 connections
  Thread Stats   Avg      Stdev     Max   +/- Stdev
    Latency    50.2ms    20ms      0.9s   85%
    Req/Sec     1.0k    100        1.5k   75%
  Requests/sec:  4012.50
  Transfer/sec:  1.2MB
```

带 POST + JSON：

```lua
-- post.lua
wrk.method = "POST"
wrk.headers["Content-Type"] = "application/json"
wrk.body = '{"name":"tom"}'
```

```bash
wrk -t4 -c100 -d30s -s post.lua http://api.example.com/users
```

> ⚠️ wrk 默认只看 connection 上发起请求的耗时，**不包括连接建立**——压短连接接口可能数据偏好。

## 五、k6 实战

k6 用 JS（不是 Node，是 Go 内嵌的 JS VM）写脚本：

```js
// load.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 50  },   // 30s 内升到 50 VU
    { duration: '1m',  target: 200 },
    { duration: '1m',  target: 500 },
    { duration: '30s', target: 0   },
  ],
  thresholds: {
    http_req_duration: ['p(99)<500'],   // P99 < 500ms 才算通过
    http_req_failed:   ['rate<0.001'],
  },
};

export default function () {
  const res = http.post('http://api.example.com/orders',
    JSON.stringify({ productId: 42 }),
    { headers: { 'Content-Type': 'application/json',
                 'Authorization': `Bearer ${__ENV.TOKEN}` } });

  check(res, {
    'status 201': (r) => r.status === 201,
    'has orderId': (r) => r.json('id') !== undefined,
  });

  sleep(1);
}
```

```bash
k6 run --env TOKEN=xxx load.js
k6 run --out influxdb=http://influx:8086/k6 load.js   # 数据落库 Grafana 出图
```

输出自带 P50 / P95 / P99，threshold 不达成时**进程返回非 0**——直接接 CI/CD 做性能门禁。

## 六、JMeter 适用场景

JMeter 的不可替代之处：

- **协议丰富**：HTTP / JDBC / JMS / FTP / SMTP / WebSocket / gRPC
- **脚本可视化**：业务方也能改
- **多机分布式压测**：master + slave
- **录制脚本**：BadBoy / Blazemeter 录浏览器操作转 JMX

缺点是 GUI 重、单机并发上不去（每个线程一个 OS 线程）。压测只用 GUI 调，**真跑用 non-GUI 模式**：

```bash
jmeter -n -t plan.jmx -l result.jtl -e -o report/
```

## 七、压测流程标准动作

```
1. 明确目标
   - 业务峰值 QPS = ?
   - 可接受 P99 = ?
   - 错误率上限 = ?

2. 准备环境
   - 与生产同规格（CPU/内存/JDK/JVM 参数）
   - 数据量与生产同量级（别压空表）
   - 监控就位（Prometheus / 链路追踪 / 慢查询）

3. 阶梯压测
   QPS 从低到高，找拐点（QPS 不再上升、RT 暴涨）

4. 找瓶颈
   CPU / 内存 / IO / 网络 / DB 连接 / GC / 锁

5. 优化 → 复压

6. 容量评估
   单机 N QPS → 业务峰值需要几台
```

> 经验法则：**压测前先做 5 分钟暖机（warm up）**，JIT、连接池、缓存都要先稳态。

## 八、常见瓶颈与排查

| 现象 | 大概率原因 |
| --- | --- |
| QPS 低、CPU 不高 | 锁 / IO 等待 / 线程池打满 |
| QPS 高时 RT 暴涨 | DB 慢查询、连接池满 |
| QPS 不变但 RT 慢 | 下游慢、GC 停顿 |
| 错误率突然飙升 | 限流 / 熔断打开 / 资源耗尽 |
| 内存持续涨 | 内存泄漏（Heap dump 看） |
| 偶发 1 ~ 2 秒尖刺 | Full GC / 网络抖动 |

排查工具组合：

```
线程：    jstack / arthas thread / async-profiler
内存：    jmap / Eclipse MAT / VisualVM
GC：      -Xlog:gc* / GCViewer
火焰图：  async-profiler / perf
DB：      SHOW PROCESSLIST / 慢查询 / pt-query-digest
网络：    tcpdump / ss -s / netstat
```

## 九、容量规划公式

```
需要的服务器数 = 峰值 QPS / 单机最大稳态 QPS × (1 + 冗余系数)
```

冗余系数取经验值：

- 内部业务：30%
- 用户面业务：50%
- 大促 / 突发场景：100%（甚至更高）

> ⚠️ 单机最大稳态 ≠ 极限值。**留 30% 余量给 GC / 抖动 / 上线高峰**，把机器跑满 95% 再扩，永远晚一步。

## 十、新手最容易犯的错

1. **压自己电脑**：1Gbps 网卡 + 浏览器代理压 = 压客户端不是压服务
2. **压空数据库**：没数据查询飞快，上线后慢查询雪崩
3. **没暖机**：第一秒成绩最难看也最不真实
4. **只压单接口**：真实流量是混合场景，得按流量比例混压
5. **数据是同一条**：缓存命中率 100%，假象一片好
6. **不监控**：只看压测端报告，服务端瓶颈完全看不到
7. **直接压生产**：影响真实用户，必须有独立预发环境
