# 分布式事务

14 章讲了 MySQL 单机事务:四大特性 ACID、隔离级别、死锁。但只要你的业务跨了两个数据源——**两个数据库、一个 DB + 一个消息队列、一个 DB + 一个第三方接口**——单机事务就罩不住了。

这一章把分布式事务这个"业务做大了一定会撞上"的难题讲清楚。

---

## 一、问题从哪来

最经典的例子:**转账**

```
张三 → 李四,转 100 元
账户表 A 库:UPDATE accounts SET balance = balance - 100 WHERE id = '张三'
账户表 B 库:UPDATE accounts SET balance = balance + 100 WHERE id = '李四'
```

两条 UPDATE 在不同库,**没有数据库能给你保证两个一起成功或一起失败**。可能出现:

| 场景 | 后果 |
| --- | --- |
| A 成功 + B 失败 | 张三钱扣了,李四没收到——钱凭空消失 |
| A 失败 + B 成功 | 张三没扣,李四收到了——平台贴钱 |
| 两个库网络超时 | 不知道成没成,业务怎么处理? |

**这就是分布式事务要解决的核心问题:跨多个独立资源的"要么都成、要么都不成"。**

---

## 二、CAP 与 BASE

### CAP 定理(必须二选一)

| 字母 | 含义 |
| --- | --- |
| **C** | Consistency 强一致——任何时刻读到的数据都是最新的 |
| **A** | Availability 可用——请求一定有响应,不会超时挂死 |
| **P** | Partition Tolerance 分区容忍——网络分区时仍能工作 |

> P 在分布式系统里**默认必须有**(网络一定会分区),所以选择实际上是 **CP 还是 AP**。

### BASE(分布式实践派)

放弃强一致,改追求**最终一致**:

- **B**asically Available:基本可用
- **S**oft state:软状态(数据有中间态)
- **E**ventually consistent:最终一致

> 经验法则:**分布式事务的 99% 解决方案都是 BASE 派的**——付出"短暂不一致"的代价,换可用性和性能。强一致(2PC)的代价太大,只用在金融核心。

---

## 三、解决方案全景

```
强一致(同步阻塞)              最终一致(异步推进)
   │                                  │
   ├─ 2PC(XA)                         ├─ 本地消息表
   ├─ 3PC                             ├─ 事务消息(RocketMQ)
   └─ TCC(应用层 2PC)                ├─ Saga(长流程编排)
                                       └─ 对账兜底
```

按"侵入业务的程度"排序(从轻到重):
**对账 < 本地消息表 ≈ 事务消息 < Saga < TCC < 2PC**

---

## 四、2PC / XA:数据库级方案

**两阶段提交**:有一个全局协调者,先让所有参与者准备(prepare),都说 OK 再统一 commit。

```
   Coordinator
      │
   ┌──┴──┐
1. │prepare?
   ▼     ▼
   DB1   DB2
   "OK"  "OK"
   │     │
   ▼     ▼
2. │commit
   DB1   DB2
```

主流数据库都支持 XA 协议(MySQL 也支持)。

**优点**:语义最干净——真正的 ACID。
**缺点**:

1. **同步阻塞**:prepare 后参与者锁住资源,等协调者发 commit。协调者一挂 → 所有参与者锁死
2. **性能差**:多一轮网络往返,吞吐降一个数量级
3. **不容错**:协调者宕机就是单点

> 经验法则:**业务侧基本不用 2PC**。只在数据库内部、消息队列内部这种"基础设施级"场景出现(比如 Kafka 的 transactional producer 就是 2PC 思想)。

---

## 五、TCC:应用层 2PC

**Try - Confirm - Cancel**:把每个业务动作拆成三个接口。

```
Try:    预留资源(冻结张三 100 元)
Confirm: 真正提交(扣款)
Cancel: 释放资源(解冻 100 元)
```

转账例子:

```java
// 账户 A:扣款
class AccountService {
    @TwoPhaseBusinessAction(name = "deduct", commitMethod = "confirm", rollbackMethod = "cancel")
    boolean tryDeduct(String userId, BigDecimal amount) {
        // 检查余额 + 冻结金额(写一行 frozen_balance)
    }
    boolean confirm(BusinessActionContext ctx) {
        // 真正扣款,清掉冻结
    }
    boolean cancel(BusinessActionContext ctx) {
        // 解冻
    }
}
```

**优点**:不锁数据库行,性能好,业务可控。
**缺点**:

1. **业务侵入巨大**:每个业务动作要写 3 个接口
2. **要保证幂等 + 空回滚 + 悬挂控制**(很烧脑)
3. **维护成本高**

> 经验法则:**只在金融、库存这种"必须及时一致"且 Saga 不够用的场景上 TCC**。

### TCC 的三个魔咒

| 问题 | 怎么破 |
| --- | --- |
| **幂等**:Confirm/Cancel 可能被重试 | 操作前检查"是不是已经做过" |
| **空回滚**:Try 没到,Cancel 先到 | 记录"全局事务状态",Try 没记录就不执行 Cancel |
| **悬挂**:Cancel 已执行,Try 才到 | 同上,Try 前看到 Cancel 已执行就跳过 |

---

## 六、Saga:长流程编排

把一个长事务拆成多个**本地事务 + 对应的补偿动作**。失败时按反向执行补偿。

```
正向:
  T1: 扣款 ──▶ T2: 减库存 ──▶ T3: 创建发货单

如果 T3 失败:
  反向补偿:C2: 加回库存 ──▶ C1: 退款
```

两种实现方式:

| 方式 | 说明 |
| --- | --- |
| **编排式(Choreography)** | 各服务通过事件触发下一步,去中心化 |
| **协调式(Orchestration)** | 有一个 Saga 协调器统一推进,集中式 |

```java
// 协调式(Seata Saga 风格)
@SagaOrchEnabled
class OrderSaga {
    @SagaStep(compensate = "refund")
    void deduct() {...}
    @SagaStep(compensate = "addStock")
    void deductStock() {...}
    @SagaStep(compensate = "cancelShipment")
    void createShipment() {...}
}
```

**优点**:每一步是本地事务,无锁,适合长流程。
**缺点**:

1. **没有隔离性**:中间状态对外可见(其他人可能读到"已下单但未发货"的中间态)
2. **补偿逻辑要业务自己想**:有些动作天然不可补偿(比如已经发了短信)
3. **流程复杂度高**:十步的 Saga 调试起来要命

> 经验法则:**业务流程长(>3 步)、可补偿、能容忍中间不一致 → Saga 是最合适的**。订单履约、退款流程、跨服务工作流。

## 七、本地消息表:最朴素也最实用

把"跨库操作"改成"本地操作 + 异步通知":

```
1. 本地事务:UPDATE 业务数据 + INSERT 消息表(同一个 DB,本地事务保证)
2. 后台 worker 扫消息表 → 投递到 MQ
3. 消费者收到 → 处理 → ACK
4. 失败重试,直到成功
```

```sql
-- 消息表
CREATE TABLE outbox_message (
  id BIGINT PRIMARY KEY,
  topic VARCHAR(64),
  payload JSON,
  status TINYINT,        -- 0 待投递 / 1 已投递
  retry_count INT,
  created_at TIMESTAMP,
  next_try_at TIMESTAMP
);
```

```java
@Transactional
public void createOrder(OrderReq req) {
    orderMapper.insert(...);                       // 业务数据
    outboxMapper.insert(new Message("ORDER_CREATED", req));  // 消息
    // 同一个事务,要么都成功要么都失败
}

// 后台 worker
@Scheduled(fixedDelay = 1000)
public void publishPending() {
    List<Message> pending = outboxMapper.findPending();
    for (var m : pending) {
        kafka.send(m.getTopic(), m.getPayload());
        outboxMapper.markPublished(m.getId());
    }
}
```

**优点**:

1. **简单可靠**——只要数据库的本地事务对就行
2. **无需引入 TCC/Saga 框架**——一张表 + 一个定时任务搞定
3. **天然异步、解耦**

**缺点**:**最终一致**(消费者可能延迟收到消息)、消费侧必须幂等。

> 经验法则:**90% 的"业务事件 + 下游处理"场景,本地消息表都够用**。下单成功发短信、用户注册同步搜索引擎、库存变更通知缓存——能异步就别搞 TCC。

## 八、事务消息(RocketMQ 的杀器)

把"本地消息表 + worker 投递"这套封装到 MQ 里:

```
1. Producer 发 half message(MQ 收到但不投递)
2. 执行本地事务
3. 提交 commit / rollback
   - commit:MQ 把 half message 转为正式消息,投递
   - rollback:删除 half message
4. 如果 producer 中间挂了,MQ 主动回查"那个事务到底做没做"
```

```java
TransactionMQProducer producer = new TransactionMQProducer("group");
producer.setTransactionListener(new TransactionListener() {
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try { businessService.doIt(); return COMMIT_MESSAGE; }
        catch (Exception e) { return ROLLBACK_MESSAGE; }
    }
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // MQ 回查时,业务自己根据 ID 判断状态
        return businessService.isDone(msg) ? COMMIT_MESSAGE : ROLLBACK_MESSAGE;
    }
});
```

**优点**:免去自己维护消息表 + worker。
**缺点**:仅 RocketMQ 原生支持(Kafka 的 transactional producer 是另一个含义,做不了"业务事务+消息"的协调)。

## 九、Seata 全家桶

阿里开源的分布式事务框架,把上面几种模式都做了集成:

| 模式 | Seata 里叫什么 | 适用 |
| --- | --- | --- |
| **AT** | Auto Transaction | 自动改写 SQL 加 undo log,业务无侵入(默认推荐) |
| **TCC** | TCC | 自己写三接口,极致性能 |
| **Saga** | Saga | 长流程编排 |
| **XA** | XA | 数据库 XA 协议 |

### Seata AT 模式怎么做到无侵入

```
1. 业务执行 UPDATE(被代理) → 自动生成 undo log(记录变更前的镜像)
2. 全局事务注册到 TC(Transaction Coordinator)
3. 全局提交:删 undo log
4. 全局回滚:用 undo log 反向恢复
```

```java
@GlobalTransactional
public void transfer(String from, String to, BigDecimal amount) {
    accountAService.deduct(from, amount);   // A 库
    accountBService.add(to, amount);         // B 库
    // 任一失败,Seata 自动回滚两个库
}
```

**优点**:业务零侵入,加个注解就行。
**缺点**:

1. 需要每个数据库表都有 PK
2. 全局锁会影响并发性能
3. 不适合超高并发场景

## 十、各方案怎么选

```
┌─────────────────────────────────────────────────────────────┐
│ 业务流程长?有补偿可能?能容忍中间不一致?                       │
│                                                              │
│ 是 → Saga                                                    │
│                                                              │
│ 否 → 强实时一致 & 高并发 → TCC                                │
│      ──────────────────────────                              │
│      普通业务事件下游处理 → 本地消息表 / 事务消息              │
│      ──────────────────────────                              │
│      传统 Spring 项目少改代码 → Seata AT                      │
└─────────────────────────────────────────────────────────────┘
```

| 场景 | 推荐 |
| --- | --- |
| 转账、扣款 | TCC / Seata AT |
| 下单后扣库存 + 创建物流 | Saga |
| 订单创建后发短信、推搜索 | 本地消息表 / 事务消息 |
| 跨多个 RDS 实例的强一致 | Seata AT 起步,极端再上 TCC |
| 业务可对账兜底 | 别上分布式事务,做对账更省心 |

---

## 十一、对账:最被低估的"分布式事务"

很多团队上来就找 Seata,其实**多数业务不一致用对账就能解决**:

```
1. 跑批(每小时/每天):比对两边数据
2. 发现差异 → 自动修复 / 人工介入
3. 异常告警
```

适合的场景:

- 跨服务最终一致即可(几分钟差不要紧)
- 业务能容忍少量人工介入
- 有清晰的"对账维度"(订单号、流水号)

> 经验法则:**对账兜底是分布式系统的"安全气囊"**——不管你上面用 TCC 还是 Saga,**对账始终要做**,因为再严密的事务也防不住"代码 bug + 数据漂移"。

---

## 十二、幂等:分布式事务的隐形前提

所有分布式事务方案都隐含一个假设:**重试是安全的**。这要求接口必须幂等。

幂等实现方式:

| 方式 | 适用 |
| --- | --- |
| **唯一键约束** | INSERT 操作,DB 拒绝重复 |
| **状态机** | 业务有明确状态流转(PENDING→PAID),只允许特定状态变更 |
| **请求 ID + 去重表** | 客户端生成唯一 reqId,服务端记录已处理 |
| **乐观锁(version 字段)** | 更新时带 version,冲突重试 |
| **分布式锁** | 同一资源串行处理(性能差,慎用) |

```java
public void pay(String orderId, String reqId) {
    if (redisTemplate.opsForValue().setIfAbsent("idempotent:" + reqId, "1", 24, HOURS)) {
        // 第一次进来,正常处理
    } else {
        // 已处理过,直接返回上次结果
    }
}
```

> 经验法则:**接口幂等是分布式事务能跑起来的最低门槛**。先把所有写接口改成幂等的,再讨论用哪种事务方案。

## 十三、常见踩坑

1. **事务范围过大**:`@Transactional` 套了一个 RPC 调用,数据库连接卡几秒,池子炸了
2. **本地消息表的 worker 单点**:消息堆积没人投递
3. **TCC 没做幂等**:Confirm 重试 → 重复扣款
4. **Saga 补偿不可逆**:已经发了短信,补偿不了——设计时就该排除这种步骤
5. **依赖网络超时判失败**:超时 ≠ 失败,可能成功了你不知道。要用幂等 + 查询确认
6. **Seata 全局锁阻塞**:高并发下 TC 成瓶颈
7. **跨库 JOIN 拆分后没数据冗余**:只能业务层组装,效率差
8. **没有对账兜底**:某天发现某个边界 case 静默错了三个月
9. **MQ 消费者不幂等**:重试一次就脏数据
10. **2PC 上来就用**:用过的人都知道有多疼

## 十四、本章 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ 接口必须幂等 | 分布式事务的前提 |
| ✅ 优先选最终一致(BASE) | 强一致代价高 |
| ✅ 优先用本地消息表 | 简单、可靠、覆盖大多数场景 |
| ✅ 长流程上 Saga | 想清楚每步的补偿动作 |
| ✅ 高并发金融场景才考虑 TCC | 三个方法 + 三个魔咒 |
| ✅ Seata AT 是最低改造方案 | 老项目改造首选 |
| ✅ 必有对账兜底 | 哪怕你用了 Seata |
| ✅ 事务范围最小化 | 不要包 RPC / 远程调用 |
| ✅ 重试 + 幂等 + 死信队列 | 失败处理三件套 |
| ✅ 全局事务 ID 透传 | 可观测性的基础 |

---

## 小结

分布式事务的本质是 **"放弃 ACID 的某一部分换可用性"**——你必须想清楚业务能容忍哪种程度的"不一致"。

- 能容忍秒级不一致 → 本地消息表
- 能容忍流程式不一致 → Saga
- 必须及时一致 + 高性能 → TCC
- 老项目少改代码 → Seata AT
- 兜底永远做对账

下一章我们看分布式系统的另一面——**当流量来势汹汹时,怎么保护自己不被冲垮**:限流、熔断、降级。
