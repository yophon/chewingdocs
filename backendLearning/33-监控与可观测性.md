# 监控与可观测性

服务部署上线只是开始。**"没监控的系统等于没上线"** 这句行话不是夸张——你不知道它现在是不是健康的、慢在哪儿、错在哪儿，就只能等用户来告诉你。

可观测性（Observability）的目标：**从外部看不进去的系统状态，能被推断出来**。它的标准答案是三支柱。

---

## 一、可观测性三支柱

```
   Metrics                Logs                  Traces
┌────────────┐      ┌──────────────┐      ┌──────────────┐
│ 数值聚合   │      │ 离散事件     │      │ 请求级因果链 │
│ "QPS=1k"   │      │ "用户登录"   │      │ "A→B→C 50ms" │
│ Prometheus │      │ ELK / Loki   │      │ Jaeger/Tempo │
└────────────┘      └──────────────┘      └──────────────┘
        告警        发生了什么            是谁慢谁错
```

| 支柱 | 看什么 | 长处 | 短处 |
| --- | --- | --- | --- |
| Metrics | 趋势 / 比率 / 阈值 | 体积小、高基数代价高 | 看不到具体一次请求 |
| Logs | 事件细节、上下文 | 信息全 | 量大、检索贵 |
| Traces | 一次请求经过哪些服务 | 定位跨服务慢 / 错 | 采样率与存储平衡难 |

> 经验法则：**Metrics 告警 → Traces 定位服务 → Logs 看现场**，三者打通靠 traceId。

## 二、Prometheus 模型

Prometheus 是 **Pull 模型**：服务暴露 `/metrics` HTTP 端点，Prometheus 定时来抓。

```
                ┌──────────────┐
   /metrics ◀── │ Prometheus   │ ◀── PromQL 查询 ◀── Grafana
   /metrics ◀── │ (scrape)     │
   /metrics ◀── └──────────────┘
                       │
                       ▼
                  Alertmanager ──▶ 钉钉/企微/邮件/PagerDuty
```

四种基本指标：

| 类型 | 用途 | 例 |
| --- | --- | --- |
| **Counter** | 只增不减 | 请求总数、错误总数 |
| **Gauge** | 可增可减 | 当前在线人数、内存使用 |
| **Histogram** | 分布（带 bucket） | 请求耗时分布、P99 |
| **Summary** | 分布（客户端算分位） | 同上，但聚合不便 |

> 经验法则：**用 Histogram 不用 Summary**——Histogram 可以跨实例聚合算分位，Summary 不行。

## 三、Spring Boot 暴露指标

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, prometheus
  metrics:
    tags:
      application: ${spring.application.name}
```

启动后 `GET /actuator/prometheus` 就能看到一堆默认指标（HTTP、JVM、数据源池）。

### 自定义业务指标

```java
@Service
class OrderService {
    private final Counter created;
    private final Timer payTimer;

    public OrderService(MeterRegistry reg) {
        this.created = Counter.builder("order.created")
            .tag("env", "prod").register(reg);
        this.payTimer = Timer.builder("order.pay")
            .publishPercentiles(0.5, 0.95, 0.99).register(reg);
    }

    public void create(...) {
        // ... 业务
        created.increment();
    }

    public void pay(...) {
        payTimer.record(() -> doPay());
    }
}
```

## 四、PromQL 速查

```text
# QPS（每秒请求数）
rate(http_server_requests_seconds_count[1m])

# 错误率
sum(rate(http_server_requests_seconds_count{status=~"5.."}[1m]))
  / sum(rate(http_server_requests_seconds_count[1m]))

# P99 延迟
histogram_quantile(0.99,
  sum(rate(http_server_requests_seconds_bucket[5m])) by (le, uri))

# 内存使用（MB）
jvm_memory_used_bytes{area="heap"} / 1024 / 1024

# CPU 限流次数（K8s）
rate(container_cpu_cfs_throttled_periods_total[5m])
```

## 五、告警规则示例

```yaml
groups:
  - name: app
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m])) by (app)
            / sum(rate(http_server_requests_seconds_count[5m])) by (app) > 0.05
        for: 5m
        labels: { severity: critical }
        annotations:
          summary: "{{ $labels.app }} 5xx 错误率 > 5% 持续 5 分钟"

      - alert: P99HighLatency
        expr: histogram_quantile(0.99, sum(rate(http_server_requests_seconds_bucket[5m])) by (le, app)) > 1
        for: 10m
        labels: { severity: warning }
```

> 经验法则：**告警基于 SLO，不基于资源**。"CPU 90%" 不一定要告警；"用户请求 P99 > 1s" 一定要。

## 六、Grafana 仪表盘

Grafana 是可视化层，连 Prometheus / Loki / Tempo / MySQL 等多种数据源。

**RED 方法**（面向请求服务的标准三图）：

| 指标 | 含义 |
| --- | --- |
| **R**ate | 每秒请求数 |
| **E**rrors | 每秒错误数 / 错误率 |
| **D**uration | 延迟分布（P50 / P95 / P99） |

**USE 方法**（面向资源）：Utilization / Saturation / Errors。CPU、内存、磁盘、连接池都按这套画。

> 推荐导入官方仪表盘：JVM Micrometer (4701)、Spring Boot 2.x Statistics (12856)、Node Exporter Full (1860)。

## 七、OpenTelemetry：统一标准

OpenTelemetry（OTel）是 CNCF 项目，目标是**统一 Metrics / Traces / Logs 的采集协议**，取代各家私有 SDK。

```
应用 ── OTel SDK ── OTLP ──▶ OTel Collector ──▶ Prometheus / Tempo / Loki
                                              ──▶ Jaeger / Datadog / NewRelic
```

Spring Boot 接入：

```xml
<dependency>
    <groupId>io.opentelemetry.instrumentation</groupId>
    <artifactId>opentelemetry-spring-boot-starter</artifactId>
</dependency>
```

```yaml
otel:
  service.name: order-service
  exporter.otlp.endpoint: http://otel-collector:4317
  traces.sampler: parentbased_traceidratio
  traces.sampler.arg: 0.1     # 采样 10%
```

零代码修改即可拿到 HTTP / DB / Kafka 自动埋点。

### 手动埋点

```java
@WithSpan("createOrder")
public Order create(@SpanAttribute("userId") Long userId, OrderReq req) {
    // ...
}
```

## 八、Trace 怎么定位慢

```
trace_id=abc123  duration=820ms
├─ gateway          15ms
├─ order-service    780ms
│   ├─ db.select    10ms
│   ├─ user-svc     750ms ◀── 90%在这
│   └─ db.insert    15ms
└─ notify-svc       async
```

一眼看出 user-svc 是瓶颈，再点开它的 trace 进一步下钻。**没有 trace 时，这个问题要靠在 10 个服务里翻日志拼凑。**

## 九、生产监控 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ 业务 RED 三图 | QPS / 错误率 / 延迟分布 |
| ✅ 资源 USE 三图 | CPU / 内存 / 磁盘 / 网络 |
| ✅ 中间件指标 | DB 连接池、Redis、MQ lag |
| ✅ JVM 关键 | GC 时间、堆使用、线程数 |
| ✅ 慢查询监控 | DB 与外部 RPC |
| ✅ 业务北极星指标 | 订单数、支付成功率等 |
| ✅ 告警基于 SLO | 错误预算耗尽再吵醒人 |
| ✅ Trace + Log 互跳 | traceId 贯穿所有日志 |

## 十、新手踩坑

1. **指标基数爆炸**：把 userId 当 label，几百万 series 直接打挂 Prometheus
2. **告警风暴**：阈值定得太敏感，凌晨刷屏没人看
3. **没有 SLO 概念**：盯 CPU 不盯用户体验，治标不治本
4. **日志和 Trace 不通**：没把 traceId 写进日志，等于没上 Trace
5. **采样率 100%**：Trace 全量存 = 三支柱里最贵的那个，10% 起步
6. **只看 P50**：用户骂的是 P99，看 P50 永远没问题
