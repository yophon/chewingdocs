# Kubernetes 核心概念

Docker 解决了"单机跑容器"的问题。但生产环境有几十、几百个容器，需要**自动调度、故障自愈、滚动升级、弹性伸缩**——这就是 Kubernetes（K8s）的价值。

---

## 一、K8s 是什么

Kubernetes 是一个**容器编排平台**，核心能力：

| 能力   | 说明                         |
| ---- | -------------------------- |
| 自动调度 | 把容器放到合适的节点（CPU/内存/亲和性）     |
| 自愈   | 容器崩溃自动重启，节点宕机自动迁移          |
| 弹性伸缩 | 根据 CPU/内存/自定义指标自动扩缩容       |
| 滚动升级 | 零停机发布，支持回滚                 |
| 服务发现 | 内置 DNS，Pod 互相访问不需要 IP      |
| 配置管理 | ConfigMap / Secret 解耦配置与代码 |

## 二、集群架构

```
┌─────────────────────────────────────────────┐
│                Control Plane                 │
│  ┌──────────┐ ┌──────────┐ ┌─────────────┐  │
│  │ API      │ │ etcd     │ │ Scheduler   │  │
│  │ Server   │ │(配置存储) │ │(调度决策)   │  │
│  └──────────┘ └──────────┘ └─────────────┘  │
│  ┌──────────────────────────────────────┐    │
│  │ Controller Manager（控制循环）        │    │
│  └──────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
         │ kubectl / API
┌────────┴──────────────────────────────────┐
│                 Worker Nodes               │
│  ┌──────────────────┐ ┌─────────────────┐ │
│  │ Node 1           │ │ Node 2          │ │
│  │ kubelet  kube-   │ │ kubelet  kube-  │ │
│  │ proxy    [pods]  │ │ proxy    [pods] │ │
│  └──────────────────┘ └─────────────────┘ │
└───────────────────────────────────────────┘
```

### 控制平面组件

| 组件 | 职责 |
| --- | --- |
| **API Server** | 所有操作的唯一入口，REST API + 认证授权 |
| **etcd** | 分布式 KV 存储，保存集群全部状态 |
| **Scheduler** | 决定 Pod 跑在哪个节点 |
| **Controller Manager** | 运行各种控制循环（副本、节点、Job…） |

### 节点组件

| 组件 | 职责 |
| --- | --- |
| **kubelet** | 接收 Pod 规范，驱动容器运行时（containerd） |
| **kube-proxy** | 维护 iptables / IPVS 规则，实现 Service 转发 |
| **Container Runtime** | 真正跑容器（containerd / CRI-O） |

## 三、声明式 vs 命令式

K8s 的核心哲学：**告诉 K8s 你想要什么状态，K8s 负责让现实与目标一致**。

```
命令式：kubectl run / kubectl scale          ← 告诉 K8s 做什么
声明式：kubectl apply -f deployment.yaml    ← 告诉 K8s 期望什么
```

控制器不断做"调谐循环（Reconcile Loop）"：

```
while true:
    current = 读取当前状态
    desired = 读取目标状态
    if current != desired:
        执行操作使 current → desired
```

> 经验法则：**生产里所有资源都用 YAML 文件管理（GitOps）**，不要手敲命令改状态，改了你不知道，也没法回滚。

## 四、最重要的对象：Pod

**Pod 是 K8s 的最小调度单元**，一个 Pod 里可以跑一个或多个容器，共享网络命名空间（localhost 互通）和存储卷。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: demo-pod
  labels:
    app: demo
spec:
  containers:
    - name: app
      image: nginx:1.25
      ports:
        - containerPort: 80
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
```

### 为什么不直接用 Pod

Pod 是一次性的，节点宕机 Pod 就没了。生产里不直接用裸 Pod，而是用**工作负载控制器**（Deployment、StatefulSet…）来管理 Pod 的生命周期。

## 五、标签与选择器

K8s 通过**标签（Label）+ 选择器（Selector）**建立对象之间的关联，而不是硬引用名字。

```yaml
# Pod 上打标签
metadata:
  labels:
    app: backend
    env: prod
    version: v2

# Service 用选择器找 Pod
spec:
  selector:
    app: backend
    env: prod
```

这套机制让**服务发现、滚动升级、蓝绿部署**都能优雅实现。

## 六、命名空间 Namespace

Namespace 提供资源隔离，常见划分：

```
default        ← 不指定时的默认空间
kube-system    ← K8s 自身组件（不要动）
kube-public    ← 公开只读数据
monitoring     ← 监控栈（Prometheus / Grafana）
production     ← 生产业务
staging        ← 预发业务
```

```bash
kubectl get pods -n production
kubectl apply -f deploy.yaml -n production
```

## 七、kubectl 常用命令

```bash
# 查看
kubectl get nodes
kubectl get pods -A                      # 所有 namespace
kubectl get pods -n production -o wide   # 显示 IP / 节点
kubectl describe pod <name> -n <ns>      # 详细信息（排错必用）
kubectl logs <pod> -n <ns> -f            # 追踪日志
kubectl logs <pod> -c <container>        # 多容器时指定
kubectl exec -it <pod> -- /bin/sh        # 进入容器

# 应用
kubectl apply -f manifest.yaml
kubectl delete -f manifest.yaml
kubectl diff -f manifest.yaml            # 查看变更 diff

# 调试
kubectl get events -n <ns> --sort-by=.lastTimestamp
kubectl top pod -n <ns>                  # 需要 metrics-server
```

## 九、给新手的建议

1. **先搞懂声明式**：K8s 不是 Docker 的命令行扩展，是状态管理系统
2. **Label 是核心**：Service 怎么找 Pod，全靠 Label Selector
3. **resources 一定要填**：不填 requests/limits，调度器是瞎的，也容易 OOM kill
4. **不要直接改 Pod**：改 Deployment，让控制器滚动更新
