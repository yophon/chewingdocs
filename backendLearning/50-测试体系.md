# 测试体系

49 章里都没专门讲测试,这是最大的疏漏——**任何不写测试的项目都是定时炸弹**。

很多人觉得"测试浪费时间",真相是:**没测试的代码改 1 个 bug 引入 2 个新 bug**,迭代速度越来越慢直到所有人都不敢动代码。这一章从"测试金字塔"讲到工程实操。

---

## 一、测试金字塔(Test Pyramid)

```
              ▲
              │              [ E2E ]            少而慢,贵
              │           /          \          
              │       [ 集成测试 ]               中等
              │      /              \           
              │  [    单元测试    ]              多而快,便宜
              ▼
              数量             速度/成本
```

| 类型 | 范围 | 速度 | 成本 | 比例 |
| --- | --- | --- | --- | --- |
| **单元测试** | 单个类 / 函数 | 毫秒级 | 极低 | 70% |
| **集成测试** | 多模块协作 / DB | 秒级 | 中 | 20% |
| **E2E 测试** | 整个系统 / API / UI | 分钟级 | 高 | 10% |

> 经验法则:**反过来叫"冰激凌甜筒"**——上面 E2E 一堆,中间集成测试少,底下单元测试空——是最常见也最痛的反模式。跑得慢、修得贵、出问题难定位。

## 二、单元测试:JUnit 5 + AssertJ + Mockito

```java
class OrderServiceTest {
    @Mock private OrderRepository repo;
    @Mock private PaymentClient   payment;
    @InjectMocks private OrderService service;

    @BeforeEach
    void setup() { MockitoAnnotations.openMocks(this); }

    @Test
    void should_create_order_when_payment_succeeds() {
        // given
        var req = new CreateOrderReq("u1", BigDecimal.valueOf(100));
        when(payment.charge(any())).thenReturn(PayResult.ok("tx-1"));
        when(repo.save(any())).thenAnswer(a -> a.getArgument(0));

        // when
        var order = service.create(req);

        // then
        assertThat(order.status()).isEqualTo(OrderStatus.PAID);
        assertThat(order.paymentTxId()).isEqualTo("tx-1");
        verify(repo).save(argThat(o -> o.amount().equals(BigDecimal.valueOf(100))));
    }

    @Test
    void should_throw_when_payment_fails() {
        when(payment.charge(any())).thenThrow(new PaymentException("INSUFFICIENT_FUNDS"));

        assertThatThrownBy(() -> service.create(new CreateOrderReq("u1", BigDecimal.TEN)))
            .isInstanceOf(PaymentException.class)
            .hasMessageContaining("INSUFFICIENT_FUNDS");
    }
}
```

要点:

- **Given-When-Then** 三段式,用空行分隔——一眼看出意图
- **方法名是文档**:`should_X_when_Y` 比 `testCreate1` 强 100 倍
- **断言用 AssertJ**,不要 JUnit 自带的——AssertJ 链式 + 流畅,可读性碾压
- **每个测试只测一件事**——失败时一眼知道哪坏了

## 三、Mock 的边界:别什么都 mock

新手常犯的两个错:

1. **过度 mock**:连 `String`、`List` 都 mock,测试代码比业务代码还长
2. **mock 完了测试啥也没测**:把所有依赖 mock 成"返回成功",一路绿灯但实际跑不通

> 经验法则:**只 mock"外部依赖"**——数据库、HTTP 客户端、消息队列。**值对象、工具类、内部 service** 直接用真实实现。

```java
// ❌ 不要这样
@Mock private List<Order> orders;
@Mock private LocalDateTime now;

// ✅ 这样
@Mock private OrderRepository repo;
@Mock private PaymentClient   payment;
@Spy  private DiscountCalculator calc = new DiscountCalculator();   // 真实业务
```

## 四、集成测试:Testcontainers 是答案

集成测试最大的痛点曾经是:**怎么搞一个真数据库?**

- H2 内存库:跟 MySQL 行为不一样,测了等于没测
- 共享测试 DB:多人跑测试互相污染
- Docker Compose:启动慢、清理麻烦

**Testcontainers 解决一切**——测试启动时自动拉起 Docker 容器,跑完自动清理。

```java
@SpringBootTest
@Testcontainers
class OrderRepositoryIT {
    @Container
    static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16")
            .withDatabaseName("test")
            .withReuse(true);              // 多个测试复用同一个容器

    @DynamicPropertySource
    static void props(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url",      pg::getJdbcUrl);
        r.add("spring.datasource.username", pg::getUsername);
        r.add("spring.datasource.password", pg::getPassword);
    }

    @Autowired private OrderRepository repo;

    @Test
    void should_persist_and_query_by_user() {
        repo.save(new Order("u1", BigDecimal.TEN));
        repo.save(new Order("u1", BigDecimal.valueOf(20)));
        repo.save(new Order("u2", BigDecimal.valueOf(30)));

        var orders = repo.findByUserId("u1");

        assertThat(orders).hasSize(2)
            .extracting(Order::amount)
            .containsExactlyInAnyOrder(BigDecimal.TEN, BigDecimal.valueOf(20));
    }
}
```

Testcontainers 支持几乎所有中间件:**PostgreSQL / MySQL / Redis / Kafka / Elasticsearch / Localstack(模拟 AWS)/ MinIO**——你需要什么真实环境,它都拉得起来。

> 经验法则:**集成测试一律用 Testcontainers,告别 H2 + Mock 数据库时代**。

## 五、Spring Boot 切片测试

跑全 Spring Context 慢——**切片测试**只加载需要的层:

| 注解 | 加载范围 |
| --- | --- |
| `@WebMvcTest` | 只 Controller + MockMvc,不加载 service/repo |
| `@DataJpaTest` | 只 JPA Repository + 内嵌 / Testcontainers DB |
| `@JsonTest` | 只 Jackson 序列化 |
| `@RestClientTest` | 只 RestTemplate / WebClient + MockWebServer |
| `@SpringBootTest` | 全量(慢,但跨层场景必须) |

```java
@WebMvcTest(OrderController.class)
class OrderControllerTest {
    @Autowired private MockMvc mockMvc;
    @MockBean  private OrderService service;

    @Test
    void should_return_400_when_amount_negative() throws Exception {
        mockMvc.perform(post("/orders")
                .contentType(MediaType.APPLICATION_JSON)
                .content("""
                    {"userId":"u1","amount":-5}
                    """))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.errors[0].field").value("amount"));
    }
}
```

> 经验法则:**Controller 用 `@WebMvcTest`、Service 用纯单元测试、Repository 用 `@DataJpaTest` + Testcontainers**——分层精准,跑得也快。

## 六、E2E 测试:慎重投入

E2E 测试模拟真实用户从入口到出口的完整流程。**它的价值高,但维护成本同样高**。

| 工具 | 适用 |
| --- | --- |
| **REST Assured** | API E2E,Java 风格 |
| **Karate** | DSL 风格 API 测试,简洁 |
| **Postman / Newman** | 接口测试 + CI 集成 |
| **Playwright / Cypress** | 浏览器 E2E |
| **k6** | 压测 + E2E 混合 |

```java
@SpringBootTest(webEnvironment = RANDOM_PORT)
@Testcontainers
class CheckoutE2ETest {
    @LocalServerPort int port;

    @Test
    void user_can_checkout() {
        given()
            .baseUri("http://localhost:" + port)
            .header("Authorization", "Bearer " + login("u1"))
            .body(Map.of("items", List.of(Map.of("sku", "A", "qty", 2))))
            .contentType(JSON)
        .when()
            .post("/api/checkout")
        .then()
            .statusCode(200)
            .body("orderId", notNullValue())
            .body("totalAmount", equalTo(200));
    }
}
```

> 经验法则:**E2E 只覆盖核心流程**(下单/支付/退款/登录)——不超过 20 条。剩下场景交给单元 + 集成测试。E2E 写多了维护成本会压垮整个团队。

## 七、TDD:测试驱动开发的真相

TDD 的循环:**Red → Green → Refactor**

```
1. 写一条失败的测试(Red)
2. 写最少代码让它过(Green)
3. 重构(Refactor),保持绿
```

**TDD 的真实价值不是"测试覆盖率",而是"逼你想清楚需求和接口"**。

```java
// 第一步:Red
@Test
void total_should_apply_discount_for_vip() {
    var cart = new Cart(VIP);
    cart.add(item(100), 2);
    assertThat(cart.total()).isEqualByComparingTo("180");  // 9 折
}

// 第二步:Green —— 最简实现
class Cart {
    BigDecimal total() { return BigDecimal.valueOf(180); }   // 临时硬编码
}

// 第三步:再加一个测试,逼出真实逻辑
@Test
void total_should_no_discount_for_normal() {
    var cart = new Cart(NORMAL);
    cart.add(item(100), 2);
    assertThat(cart.total()).isEqualByComparingTo("200");
}
// → 现在不得不写真正的折扣逻辑
```

> 经验法则:**TDD 不是宗教,但"先写测试再写代码"在复杂业务逻辑里 90% 时候是值得的**——它强制你想清楚边界条件。

## 八、覆盖率:有用,但别迷信

覆盖率工具:**JaCoCo / Cobertura**,在 CI 里输出覆盖率报告。

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <executions>
        <execution>
            <goals><goal>prepare-agent</goal></goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals><goal>report</goal></goals>
        </execution>
        <execution>
            <id>check</id>
            <goals><goal>check</goal></goals>
            <configuration>
                <rules><rule>
                    <element>BUNDLE</element>
                    <limits><limit>
                        <counter>LINE</counter>
                        <minimum>0.70</minimum>
                    </limit></limits>
                </rule></rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

> ⚠️ **覆盖率高 ≠ 测试好**。一行不写 assert 的测试也能让覆盖率 100%。覆盖率只是"没测到"的负向指标——**没覆盖一定有问题,覆盖了不一定没问题**。

## 九、契约测试:微服务时代的关键

微服务里集成测试有个大问题:**所有服务都启动 = 跑测试要 30 分钟**。

**契约测试**(Consumer-Driven Contract):

```
1. 消费方写"期望对方怎么响应"
2. 生成契约文件
3. 提供方根据契约生成测试,自己跑
4. 双方都过 = 协作没问题,不用真启动联调
```

主流工具:**Pact**(语言无关) / **Spring Cloud Contract**(Spring 生态)。

```groovy
// Spring Cloud Contract DSL
Contract.make {
    request {
        method 'GET'
        url '/users/1'
    }
    response {
        status 200
        body([id: 1, name: 'Tom'])
        headers { contentType('application/json') }
    }
}
```

> 经验法则:**多服务团队跑联调慢、爱出锅 → 上契约测试**。它不是必须的,但能省掉大量"你改了我才知道"的事故。

## 十、性能测试与负载测试

36 章压测有覆盖,这里只补"测试体系视角":

| 类型 | 关心什么 |
| --- | --- |
| **基准测试**(JMH) | 单方法性能极限 |
| **负载测试** | 正常流量下的响应、吞吐 |
| **压力测试** | 加压到崩,看在哪崩 |
| **耐久测试** | 长时间跑,看内存泄漏 / GC 长尾 |
| **峰值测试** | 瞬时高峰是否扛得住(秒杀) |

性能测试**应该接进 CI**——每次发版自动跑基准,P99 退化超过 10% 就告警。

## 十一、变异测试:测试质量的"X 光机"

光看覆盖率不够,**变异测试(Mutation Testing)** 验证测试有没有真正捕获 bug:

```
1. 工具自动改你的代码(把 + 改成 -,把 > 改成 >=)
2. 跑你的测试套件
3. 如果测试还是 pass —— 你的测试漏了这个 bug,白写
```

工具:**PIT (Pitest)** —— Maven 插件即可。

```bash
mvn org.pitest:pitest-maven:mutationCoverage
```

> 经验法则:**核心业务模块至少跑一次变异测试**——你会发现自己的测试比想象的弱很多。

## 十二、测试数据管理

测试有个大坑:**数据怎么准备**。三种姿势:

### 1. 内联(简单场景)

```java
var user = new User("u1", "Tom");
```

### 2. 工厂方法 / Builder

```java
var user = aUser().withId("u1").withName("Tom").build();
```

### 3. 测试夹具(Fixture / Seed)

```java
@Sql("/fixtures/orders.sql")
@Test
void test() { ... }
```

复杂场景推荐 **Object Mother + Builder** 组合:

```java
class UserMother {
    public static User vip() { return aUser().withLevel(VIP).build(); }
    public static User newbie() { return aUser().withLevel(NORMAL).withCreatedAt(today()).build(); }
}
```

测试代码可读性立刻起飞。

## 十三、CI 中的测试策略

```
PR 触发:                                            主干合并:
  - 单元测试(必须 < 30s)                            - 全量集成测试
  - 集成测试(关键路径,< 5min)                      - E2E
  - Lint / 覆盖率检查                                 - 性能基准
                                                      - 安全扫描

夜间构建:
  - 全量 E2E
  - 变异测试
  - 长跑 / 耐久测试
```

> 经验法则:**PR 阶段的测试要"快狠"**——10 分钟以上没人愿意等。慢测试推到夜跑。

## 十四、常见踩坑

1. **测试依赖执行顺序**:用 `@Order` / 共享状态——一改顺序全挂
2. **共享测试数据库**:并行跑互相污染,改用 Testcontainers
3. **`Thread.sleep` 等异步结果**:用 Awaitility,防 flaky
4. **Mock 一切 + 没断言**:绿灯但实际没测到
5. **测试方法名 `test1`/`test2`**:出错时一脸懵
6. **覆盖率刷数**:写一堆 `assertThat(true).isTrue()` 凑指标
7. **私有方法测**:测公共行为,私有方法是实现细节
8. **测试 print 不断言**:跑过=过,实际错了不知道
9. **flaky 测试视而不见**:今天过明天挂,慢慢失去信任
10. **写一次再不维护**:业务改了测试不改,慢慢全注释掉
11. **集成测试连真生产 DB**:写脏数据
12. **不区分单元/集成**:全混在一起,跑一次半小时
13. **测试代码没人 review**:复制粘贴,质量比业务代码差远了
14. **没有"快速失败"**:第一个错误才该停,集成测试无脑全跑

## 小结

写测试的回报曲线是**先慢后爆发**——前两周觉得多写了 30% 代码,但三个月后改 bug 时间从 3 小时降到 30 分钟,半年后敢做大重构。

记住三件事:

1. **没测试的"快"是错觉**——你只是把代价推迟了
2. **测试是产品的一部分**——和业务代码同等重视、同等 review
3. **测试金字塔顺过来,反过来就是甜筒** —— 单元为主,E2E 点缀
