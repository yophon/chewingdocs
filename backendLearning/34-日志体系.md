# 日志体系（ELK / Loki）

监控告诉你**有没有问题**、链路追踪告诉你**问题在哪个服务**，而日志告诉你**到底发生了什么**。线上 80% 的问题都是"翻日志能解决但没人翻"——前提是日志写得能翻。

这一章讲三件事：**怎么写日志**、**怎么收集日志**、**怎么查日志**。

---

## 一、日志的核心原则

```
1. 日志只写到 stdout / stderr，不要写文件
2. 一条日志一行 JSON（结构化）
3. 必带 traceId、spanId、userId
4. 等级清晰，error 不要刷屏
5. 不写敏感信息（密码、token、身份证）
```

为什么不写文件？容器化时代，**容器是无状态的**——文件路径、磁盘清理、轮转都是负担。让日志走 stdout，由 Kubernetes / Docker 接管，再由采集器统一搬走。

---

## 二、日志等级别滥用

| 等级 | 用途 | 频率 |
| --- | --- | --- |
| **ERROR** | 系统出错，需要人排查 | 越少越好 |
| **WARN** | 异常但不致命 | 偶尔 |
| **INFO** | 关键业务流转 | 节制 |
| **DEBUG** | 细节排查 | 只在 dev / 排障开 |
| **TRACE** | 细到字段级 | 几乎不用 |

> ⚠️ 把异常 catch 掉只 `log.error("出错了")` 不带堆栈是新手最大罪行。**一定要 `log.error("xxx", e)`**——把 Throwable 作为最后一个参数传进去。

## 三、结构化日志

非结构化的日志：

```
2026-05-04 10:23:45 INFO 用户 42 登录成功 IP 1.2.3.4
```

结构化（JSON）：

```json
{"ts":"2026-05-04T10:23:45Z","level":"INFO","logger":"AuthService",
 "msg":"login success","userId":42,"ip":"1.2.3.4",
 "traceId":"abc123","spanId":"def456","app":"user-svc"}
```

后者好处：

- **能按字段过滤**：`level=ERROR AND userId=42`
- **能聚合**：每个用户错误数 top10
- **跨服务关联**：用 traceId 把多服务的日志拼起来

### Spring Boot + Logback JSON

```xml
<!-- pom.xml -->
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

```xml
<!-- logback-spring.xml -->
<configuration>
  <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
      <customFields>{"app":"${spring.application.name}"}</customFields>
      <includeMdcKeyName>traceId</includeMdcKeyName>
      <includeMdcKeyName>spanId</includeMdcKeyName>
      <includeMdcKeyName>userId</includeMdcKeyName>
    </encoder>
  </appender>
  <root level="INFO"><appender-ref ref="STDOUT"/></root>
</configuration>
```

业务代码用 SLF4J + MDC：

```java
MDC.put("userId", String.valueOf(user.getId()));
log.info("login success");   // userId 自动写入
MDC.clear();
```

## 四、采集与传输

```
应用 stdout
   │
   ▼ (Kubernetes 节点 /var/log/containers/*.log)
┌──────────────────────┐
│ 采集器（每节点一个） │  Filebeat / Fluent Bit / Vector / Promtail
└──────────────────────┘
   │
   ▼
┌──────────────────────┐
│ 缓冲 / 解析           │  Kafka / Logstash（可选）
└──────────────────────┘
   │
   ▼
┌──────────────────────┐
│ 存储 + 检索           │  Elasticsearch / Loki / ClickHouse
└──────────────────────┘
   │
   ▼
                       Kibana / Grafana
```

### 采集器选型

| 采集器 | 特点 | 资源占用 |
| --- | --- | --- |
| **Filebeat** | Elastic 官方 | 中 |
| **Fluent Bit** | C 写的，轻 | 低 |
| **Vector** | Rust，新生代之光 | 低 |
| **Promtail** | Loki 配套 | 低 |

> 经验法则：**ELK 用 Filebeat / Fluent Bit；Loki 用 Promtail / Vector**。Logstash 现在主要用作"重转换"中间层，能不用就不用。

---

## 五、ELK 栈

ELK = **E**lasticsearch + **L**ogstash + **K**ibana。现在通常是 EFK（用 Fluent Bit 替 Logstash）或 BELK（前面加 Kafka 缓冲）。

| 组件 | 角色 |
| --- | --- |
| Elasticsearch | 全文索引存储 |
| Logstash | 解析、转换（重） |
| Kibana | 可视化检索 |
| Beats / FluentBit | 采集 |

### 索引策略

```
日索引：logs-app-2026.05.04
ILM：热（7天SSD）→ 温（30天HDD）→ 冷（90天对象存储）→ 删除
```

> 经验法则：**日志按天建索引 + ILM 自动转冷**。一个超大索引查询慢、删数据也痛苦。

### Kibana 检索 DSL

```
level: ERROR AND app: order-service AND traceId: "abc123"
```

```
@timestamp: [now-1h TO now] AND status: [500 TO 599]
```

## 六、Loki：日志的 Prometheus

Loki 出自 Grafana Labs，定位是 **"低成本日志"**：

- 只索引 **labels**（app、level、namespace），不索引正文
- 正文压缩存对象存储（S3 / OSS / MinIO）
- 用 **LogQL** 查询，语法像 PromQL

```
存储成本：Loki 通常是 ES 的 1/10
查询能力：精确到具体一条慢，但 group/agg 弱
```

### LogQL 速查

```
{app="order-service"} |= "ERROR"

{app="order-service"} |~ "order_id=\\d+" | json | duration > 1s

# 错误率
sum(rate({app="order-service"} |= "ERROR"[5m]))
  / sum(rate({app="order-service"}[5m]))
```

### Promtail 配置（K8s）

```yaml
scrape_configs:
  - job_name: kubernetes-pods
    kubernetes_sd_configs: [{ role: pod }]
    relabel_configs:
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace
      - source_labels: [__meta_kubernetes_pod_label_app]
        target_label: app
    pipeline_stages:
      - json:
          expressions:
            level: level
            traceId: traceId
      - labels:
          level:
```

## 七、ELK vs Loki 怎么选

| 维度 | ELK | Loki |
| --- | --- | --- |
| 存储成本 | 高 | 低 |
| 全文搜索 | 强 | 弱（只能 grep 式） |
| 聚合分析 | 强 | 一般 |
| 与 Grafana 整合 | 一般 | 原生（同公司） |
| 运维复杂度 | 高 | 低 |
| 适合 | 业务日志分析、安全审计 | K8s 容器日志、调试 |

> 经验法则：**资源紧 / K8s 为主用 Loki，做日志分析 / BI 用 ELK**，大公司常常**两套都用**——Loki 给 SRE 排障，ELK 给业务做分析。

## 八、敏感信息脱敏

绝对不能进日志的字段：

```
password, token, refresh_token, id_card, bank_card,
phone（建议中间4位脱敏）, email（@前打码）, address
```

### 通用脱敏 Filter（Logback）

```java
public class MaskingConverter extends MessageConverter {
    private static final Pattern PHONE = Pattern.compile("(1[3-9]\\d)\\d{4}(\\d{4})");
    @Override
    public String convert(ILoggingEvent e) {
        String msg = e.getFormattedMessage();
        return PHONE.matcher(msg).replaceAll("$1****$2");
    }
}
```

> ⚠️ 业内出事最快的就是日志带身份证 / 手机号被合规扫到，**写日志前先想"如果这个 JSON 被 dump 出去会不会被罚"**。

## 九、日志保留与成本

| 数据类型 | 建议保留 |
| --- | --- |
| 业务日志 | 30 ~ 90 天热，1 年冷归档 |
| 接入日志 / Nginx | 7 ~ 30 天 |
| 安全审计 | 6 个月 ~ 1 年（合规要求） |
| Debug 日志 | 不进生产 |

成本控制三招：**采样、压缩、冷热分层**。一个 100 节点中型业务，日志月开销很容易上万——压缩比 8x、采样 INFO 50% 可以直接砍掉一半成本。
