# 对象存储

文件上传、图片、视频、附件、备份——几乎每个后端项目都要存"非结构化数据"。

很多团队第一反应是"放服务器本地",然后立刻撞上一堆问题:磁盘不够、迁移困难、CDN 加不上、备份难、多实例不一致。

这一章讲清现代后端的标准答案:**对象存储**。

---

## 一、为什么不能放本地文件系统

| 痛点 | 表现 |
| --- | --- |
| **容量受限** | 单机磁盘上限,扩容要停机 |
| **不能水平扩展** | 一台机器存的图,另一台读不到 |
| **没有冗余** | 磁盘坏一块,文件丢一片 |
| **CDN 接入麻烦** | 要再做一层 nginx 镜像 |
| **元数据差** | 没有 ACL、版本、生命周期 |
| **K8s 不友好** | 容器无状态,挂 PV 又重又脆 |

> 结论:**业务文件不放本地磁盘**——这是后端的"已知答案"。

---

## 二、对象存储是什么

把"文件"抽象成 **对象(Object)** + **键(Key)** + **元数据(Metadata)**:

```
Bucket(桶,顶层命名空间)
  └─ Object Key:    images/2026/05/avatar.png
     ├─ Content:    [二进制]
     ├─ Metadata:   Content-Type / Cache-Control / 自定义 KV
     └─ Versioning: v1, v2, v3...
```

特点:

- **扁平**——没有目录概念,key 里的 `/` 只是字符串(虽然 SDK / 控制台会"装作"是目录)
- **HTTP API**——上传 PUT、下载 GET、删除 DELETE,完全 RESTful
- **几乎无限容量**——按用量计费
- **高持久性**——通常 11 个 9(99.999999999%)
- **不能修改**——只能"覆盖"或"版本化",对象本身不可改

## 三、S3 协议:事实标准

AWS S3 不只是产品,**它的 API 是行业标准**。所有主流对象存储都兼容 S3 协议:

| 服务 | 类型 | 备注 |
| --- | --- | --- |
| **AWS S3** | 公有云 | 鼻祖,生态最全 |
| **阿里云 OSS** | 公有云 | 国内首选,自有 SDK 也兼容 S3 |
| **腾讯云 COS** | 公有云 | 国内,S3 兼容 |
| **GCP GCS / Azure Blob** | 公有云 | 各自有原生 API,S3 兼容 |
| **MinIO** | 自建 | 完全 S3 兼容、单文件部署、K8s 友好 |
| **Ceph RGW** | 自建 | 大型私有云首选,兼容 S3 |
| **SeaweedFS** | 自建 | 海量小文件友好 |
| **Cloudflare R2** | 公有云 | S3 兼容、出网零费用 |

> 经验法则:**写代码用 S3 SDK**——不论后端跑哪家,换提供商只改 endpoint 和 key,业务代码零改动。

## 四、上传的三种姿势

### 1. 服务端中转(最常见、也最差)

```
[ Client ]  ─── upload file ─── ▶ [ 后端 ]  ─── PUT object ─── ▶ [ OSS ]
```

**问题**:文件流过后端,带宽 / 内存全占。1 万人传 100MB 视频,后端先死。

### 2. 直传(推荐)

```
1. Client → 后端:"我要传文件 avatar.png"
2. 后端 → Client:返回预签名 URL(临时有效)
3. Client → 直接 PUT 到 OSS,无需经过后端
4. Client → 后端:"传完了,key 是 xxx"
```

```java
// 生成预签名 URL
GeneratePresignedUrlRequest req = new GeneratePresignedUrlRequest(
        "my-bucket", "images/" + UUID.randomUUID() + ".png",
        HttpMethod.PUT);
req.setExpiration(Date.from(Instant.now().plusSeconds(600)));   // 10 分钟有效
req.setContentType("image/png");
URL url = s3Client.generatePresignedUrl(req);
return url.toString();
```

**核心好处**:后端不经手大文件,只做鉴权和发 URL。

### 3. STS 临时凭证

```
后端 → 申请临时 AccessKey(权限受限) → 给 Client → Client 用 SDK 直传
```

适合**移动端 / Web 大量上传**——一次发凭证,后续多个文件直接用。

## 五、大文件分片上传

S3 协议规定单次上传上限 5GB(预签名 PUT 通常更小)。大文件用**分片上传**:

```
1. CreateMultipartUpload     → 拿到 uploadId
2. UploadPart x N            → 每片 5MB ~ 5GB
3. CompleteMultipartUpload   → S3 拼接所有 part
4. AbortMultipartUpload      → 取消(否则碎片永远存在 + 计费)
```

```java
String uploadId = s3.initiateMultipartUpload(
        new InitiateMultipartUploadRequest("bucket", key)).getUploadId();

List<PartETag> parts = new ArrayList<>();
for (int i = 0; i < chunks.size(); i++) {
    UploadPartResult r = s3.uploadPart(new UploadPartRequest()
            .withBucketName("bucket").withKey(key)
            .withUploadId(uploadId).withPartNumber(i + 1)
            .withInputStream(chunks.get(i))
            .withPartSize(chunks.get(i).length));
    parts.add(r.getPartETag());
}

s3.completeMultipartUpload(new CompleteMultipartUploadRequest(
        "bucket", key, uploadId, parts));
```

**好处**:断点续传(失败的 part 重传即可)、并行上传(几个 part 同时传,提速)。

> ⚠️ **未完成的分片不会自动清理**,在 OSS 里就是钱。一定要配置 **生命周期规则:N 天后自动 abort 未完成的 multipart upload**。

## 六、下载的几种方式

### 公开读

```
bucket 设为 public-read,直接 https://bucket.s3.amazonaws.com/key
```

**适合**:头像、商品图、营销图——这些就是要让所有人看的。

### 预签名下载 URL

```java
URL url = s3.generatePresignedUrl(
    new GeneratePresignedUrlRequest("bucket", key)
        .withExpiration(Date.from(Instant.now().plusSeconds(60))));
```

**适合**:私有文件、临时分享、合同/发票下载链接。

### 后端中转(私有 + 鉴权)

```
Client → 后端 → 鉴权检查 → 流式返回 OSS 内容
```

**适合**:必须严格鉴权 / 需要审计 / 二次处理(水印、转码)的场景。

## 七、CDN 配合:才是真正的"用得起"

直接走 OSS 拉取:跨地域慢、按量计费贵。**所有面向用户的对象存储,前面都该挂 CDN**。

```
用户 ─── ▶ [ CDN 边缘节点 ] ─── 缓存 miss ─── ▶ [ OSS 源站 ]
                  │
              缓存 hit(直接返回,不打 OSS)
```

收益:

1. **首字节快**——边缘节点比 OSS 源近
2. **省钱**——CDN 流量便宜,OSS 出网贵
3. **抗刷**——CDN 有 WAF / 限流

CDN + OSS 的几个关键配置:

| 配置 | 怎么做 |
| --- | --- |
| **Cache-Control** | 上传时带,告诉 CDN 缓存多久(`max-age=31536000` 长缓存) |
| **缓存 Key 规则** | 默认是完整 URL,带签名参数会让缓存失效——签名走 cookie / 单独域名 |
| **Range 请求支持** | 视频边下边播必须 |
| **HTTPS** | 必开,SEO + 安全 |
| **回源鉴权** | 防止有人绕开 CDN 直接刷 OSS |

## 八、私有读 + 防盗链

公开桶最常见的事故:**桶名被扫到 → 私密图被人随便看 / 当图床用 / 流量被刷光**。

正确做法:

1. **桶设为私有读**——默认任何人都不能访问
2. **正常用户走预签名 URL**——临时有效、可绑定 IP
3. **CDN 回源走签名认证**——只有 CDN 能去 OSS 拉
4. **Referer 防盗链**——CDN 侧做白名单
5. **Token 鉴权**——CDN URL 带过期 token,防止 URL 泄露

## 九、典型架构图

```
[ App Client ] ─────────┐
                        │ 1. 申请上传凭证 / 预签名 URL
                        ▼
                   [ 后端 API ]  ──── 写元数据 ────▶ [ MySQL ]
                        │ 2. 返回 URL
[ App Client ] ◀────────┘
       │
       │ 3. 直传文件
       ▼
   [ OSS / S3 ]  ─── 4. 触发事件 ───▶ [ 处理服务(转码/水印) ]
       │
       │ 5. 用户下载
       ▼
   [ CDN ] ◀─────── [ App Client ]
```

要点:

- **元数据进数据库**(filename, size, owner, key),OSS 里只存原始字节
- **OSS 事件触发后处理**:图片自动生成缩略图、视频转码、违规扫描
- **数据库的 file_id 是真相源**,OSS 的 key 是从属

## 十、生命周期管理

对象存储的存储类型 + 生命周期 = **降本利器**:

| 存储类型 | 单价 | 取数延迟 | 用途 |
| --- | --- | --- | --- |
| **Standard** | 高 | 毫秒 | 热数据 |
| **Infrequent Access (IA)** | 中 | 毫秒 | 偶尔访问 |
| **Archive** | 低 | 分钟 ~ 小时 | 备份、归档 |
| **Deep Archive** | 极低 | 小时 ~ 天 | 长期合规存储 |

生命周期规则示例:

```
30 天后转 IA      → 节省 50%
90 天后转 Archive → 再省 70%
365 天后删除      → 自动清理
```

> 经验法则:**所有对象存储都该配生命周期规则**。日志、备份、用户上传、未完成分片——按业务定不同的转冷/删除策略,账单能砍一半。

## 十一、Spring Boot + S3 SDK 实战

```xml
<dependency>
    <groupId>software.amazon.awssdk</groupId>
    <artifactId>s3</artifactId>
</dependency>
```

```java
@Configuration
class S3Config {
    @Bean
    S3Client s3Client(@Value("${oss.endpoint}") String endpoint,
                      @Value("${oss.region}") String region,
                      @Value("${oss.access-key}") String ak,
                      @Value("${oss.secret-key}") String sk) {
        return S3Client.builder()
                .endpointOverride(URI.create(endpoint))
                .region(Region.of(region))
                .credentialsProvider(StaticCredentialsProvider.create(
                        AwsBasicCredentials.create(ak, sk)))
                .build();
    }
}
```

```java
@Service
class FileService {
    private final S3Presigner presigner;

    public String genUploadUrl(String key) {
        return presigner.presignPutObject(req -> req
                .signatureDuration(Duration.ofMinutes(10))
                .putObjectRequest(p -> p.bucket("my-bucket").key(key))
        ).url().toString();
    }

    public String genDownloadUrl(String key) {
        return presigner.presignGetObject(req -> req
                .signatureDuration(Duration.ofMinutes(5))
                .getObjectRequest(g -> g.bucket("my-bucket").key(key))
        ).url().toString();
    }
}
```

## 十二、MinIO:自建对象存储的最佳选择

如果不能上公有云(数据合规、内网、私有化交付),MinIO 几乎是**唯一推荐**:

```bash
docker run -d --name minio \
    -p 9000:9000 -p 9001:9001 \
    -e MINIO_ROOT_USER=admin \
    -e MINIO_ROOT_PASSWORD=admin12345 \
    -v /data/minio:/data \
    minio/minio server /data --console-address ":9001"
```

特点:

- **完全 S3 兼容**——你的 SDK 不用改一行
- **单二进制**——部署最简单
- **K8s Operator**——分布式部署成熟
- **纠删码 / 副本**——数据可靠性内置

> 经验法则:**自建场景默认选 MinIO**。Ceph 适合超大规模(PB 起),小团队驾驭不了。

## 十三、关键安全点

1. **AccessKey 不进代码**:走环境变量 / 配置中心 / 临时 STS
2. **桶默认私有**:除非真的是公开素材
3. **Bucket Policy 最小权限**:谁能读、谁能写、IP 白名单
4. **HTTPS 强制**:Bucket Policy 拒绝 HTTP
5. **服务端加密(SSE)**:云盘默认加密,合规场景必开
6. **客户端加密**:超敏感场景在上传前加密(密钥不进 OSS)
7. **审计日志**:S3 Access Log / OSS 日志服务,事故排查必备
8. **病毒扫描**:用户上传必须过 ClamAV / 三方扫描服务
9. **MIME 校验**:别只看后缀,服务端二进制头校验
10. **大小限制**:网关 / 后端层都加,防止刷爆配额

## 十四、常见踩坑

1. **服务端中转大文件**:后端被打爆,改预签名直传
2. **没生命周期规则**:三个月后账单翻倍才发现
3. **未完成分片不清理**:静默累积,可能比正式数据还多
4. **桶设为公开 + 明文 key**:扫描工具一秒拿到所有内容
5. **CDN 没配 Cache-Control**:回源率 90%,白搭一层 CDN
6. **图片不做尺寸 / 数量限制**:用户传 100 张 50MB 高清图,空间瞬间没
7. **元数据不入库**:OSS 里几百万个 key,业务侧一查"我的文件"全靠 list
8. **直传成功就当结束**:实际可能传了一半,要业务侧确认 + 校验
9. **跨区域访问**:把 OSS 和应用部到不同地域,延迟 + 跨地域费用
10. **删除靠业务删**:用户删账号,OSS 里的资源还在——配生命周期 / 写删除清理逻辑
