# 流处理

39 章 ClickHouse、46 章 CDC 把"数据存与搬"的链路打通了——但生产里有一类需求是 **"数据流过来的时候就要立刻算出结果"**:

- 实时大屏:每秒 GMV、每分钟订单量
- 实时风控:同一账号 1 秒内 10 次登录失败 → 封号
- 实时推荐:用户刚点过的商品立即影响下次推荐
- 监控告警:CPU 持续 5 分钟 > 80% → 触发

这就是流处理(Stream Processing)。

---

## 一、批处理 vs 流处理

| 维度 | 批处理 | 流处理 |
| --- | --- | --- |
| 数据 | 有界(已经攒好) | 无界(源源不断) |
| 计算时机 | 数据齐了才算 | 一来就算 |
| 延迟 | 分钟 ~ 小时 | 毫秒 ~ 秒 |
| 状态 | 通常无状态 | 有状态(累加器、窗口) |
| 失败重跑 | 简单(从头跑) | 复杂(要 checkpoint) |
| 代表 | Spark、Hive、MapReduce | Flink、Kafka Streams、Spark Structured Streaming |

**Lambda 架构 vs Kappa 架构**:

```
Lambda:批 + 流双链路,合并出最终结果(老经典,维护两套)
Kappa: 只有流,把"批"也当成"重放历史流",一套搞定(现代主流)
```

> 经验法则:**新项目走 Kappa,把流处理当默认**。Lambda 在 2015 年是先进,现在维护双链路是负担。

## 二、流处理的三个硬核概念

### 1. 时间语义

```
Event Time:        事件实际发生的时间(用户点击发生于 14:00:00)
Processing Time:   系统处理这条事件的时间(可能是 14:00:05)
Ingestion Time:    数据进入流系统的时间
```

**绝大多数业务必须用 Event Time**——否则乱序、延迟数据会让结果错乱。

### 2. 窗口

无界数据要算"每分钟 GMV",必须切窗口:

| 窗口 | 含义 | 用途 |
| --- | --- | --- |
| **Tumbling**(滚动) | 不重叠,固定大小 | 整点统计:每分钟 GMV |
| **Sliding**(滑动) | 重叠,定长 + 滑步 | 移动平均:近 5 分钟 GMV,每 1 分钟一次 |
| **Session** | 按活跃间隔自动切 | 用户会话:30 分钟无操作算结束 |
| **Global** | 不分窗 | 配合自定义触发器 |

```
Tumbling 5min:    [00:00-05:00)  [05:00-10:00)  [10:00-15:00)
Sliding 5min/1min:[00:00-05:00)
                    [01:00-06:00)
                      [02:00-07:00)
Session(gap=10min):  事件1 间隔3min 事件2 间隔15min(切窗) 事件3
```

### 3. 状态(State)

聚合天然有状态——"当前累计 GMV""5 分钟内访问数"都要记。
状态后端:

| 类型 | 特点 |
| --- | --- |
| **内存** | 快,但重启丢 |
| **RocksDB** | Flink 默认,本地磁盘存,大状态 |
| **远端**(HBase/Redis) | 可共享,但延迟高 |

## 三、迟到数据怎么办

```
事件时间 14:00 的数据,因为网络延迟,14:05 才到——窗口已经关闭了,怎么办?
```

两个机制:

### 1. Watermark(水位线)

**给系统打"我认为不会再有比这更早的数据了"的标记**。Watermark = 当前最大 event time - 允许延迟 N 秒。

```
事件流:t=10s, t=11s, t=14s, t=12s(迟到)
Watermark(允许延迟 5s):
  收到 t=14s 时,Watermark = 14 - 5 = 9s
  → 表示"9s 之前的窗口可以关闭"
  → t=12s 仍在 Watermark 之上,正常处理
  → 如果再来一条 t=8s,迟到了,丢弃或走旁路
```

### 2. Allowed Lateness

窗口关闭后再等 N 时间,如果有迟到的数据继续更新窗口结果。

> 经验法则:**业务能容忍秒级延迟 → Watermark 设小(1~5s);要严格准确 → Watermark + Allowed Lateness 双层兜底**。

## 四、Exactly-Once 语义

流处理三种语义:

| 语义 | 含义 | 代价 |
| --- | --- | --- |
| **At-most-once** | 最多一次,可能丢 | 简单,可接受丢数据时用 |
| **At-least-once** | 至少一次,可能重 | 加幂等就够了,常见 |
| **Exactly-once** | 精确一次 | 需要 checkpoint + 事务性 sink |

Flink 的 Exactly-once = **Chandy-Lamport 分布式快照算法**:

1. 每隔 N 秒打一个 Checkpoint:把所有算子状态 + 输入位点写入持久化存储
2. 失败重启:从最近 checkpoint 恢复 + 重放该位点之后的输入
3. Sink 必须支持事务:要么两阶段提交,要么幂等覆盖

**真正端到端 Exactly-once = Source 可重放 + 算子有状态快照 + Sink 事务性写**。Kafka 源 + Flink 算子 + Kafka/JDBC 事务性 Sink 是经典组合。

## 五、主流框架对比

| 框架 | 出身 | 强项 | 弱项 |
| --- | --- | --- | --- |
| **Apache Flink** | 德国 dataArtisans | 事件时间、状态、Exactly-once、低延迟、SQL 完整 | 学习曲线、运维不轻 |
| **Kafka Streams** | Confluent | 嵌入应用,无独立集群,与 Kafka 一体 | 仅 Kafka 输入输出,SQL 弱 |
| **Spark Structured Streaming** | UC Berkeley | 批流统一,与 Spark 生态贴合 | 微批架构,延迟比 Flink 高 |
| **ksqlDB** | Confluent | SQL on Kafka,简单流处理 | 复杂场景能力有限 |
| **Pulsar Functions** | StreamNative | 嵌入 Pulsar,轻量 | 生态较小 |
| **Materialize / RisingWave** | 新一代 | 流式物化视图,SQL 即流处理 | 新,生产案例少 |

> 经验法则:**Java/中型以上业务选 Flink;只在 Kafka 内部流转选 Kafka Streams;数据团队批流统一选 Spark**。

## 六、Flink 入门:WordCount 风格

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

DataStream<String> stream = env.fromSource(
        KafkaSource.<String>builder()
            .setBootstrapServers("kafka:9092")
            .setTopics("clicks")
            .setStartingOffsets(OffsetsInitializer.earliest())
            .setValueOnlyDeserializer(new SimpleStringSchema())
            .build(),
        WatermarkStrategy.<String>forBoundedOutOfOrderness(Duration.ofSeconds(5)),
        "kafka-source");

stream
    .map(this::parse)                                    // String → ClickEvent
    .keyBy(ClickEvent::userId)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new CountAgg())
    .sinkTo(JdbcSink.sink(
        "INSERT INTO user_clicks_per_min VALUES (?, ?, ?)",
        (ps, r) -> { ps.setLong(1, r.userId); ps.setLong(2, r.minute); ps.setLong(3, r.cnt); },
        ...));

env.execute("clicks-per-min");
```

## 七、Flink SQL:让分析师也能写

Flink 的杀手锏不是 Java API,是 SQL——把流当表用:

```sql
-- 创建源表(Kafka)
CREATE TABLE clicks (
   user_id BIGINT, page STRING, ts TIMESTAMP(3),
   WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector' = 'kafka',
  'topic' = 'clicks',
  'properties.bootstrap.servers' = 'kafka:9092',
  'format' = 'json'
);

-- 创建汇表(MySQL)
CREATE TABLE result (
   minute TIMESTAMP(3), page STRING, pv BIGINT, uv BIGINT
) WITH (
  'connector' = 'jdbc',
  'url' = 'jdbc:mysql://...',
  'table-name' = 'page_metrics'
);

-- 一条 SQL 就是一个完整的流任务
INSERT INTO result
SELECT
    TUMBLE_END(ts, INTERVAL '1' MINUTE) AS minute,
    page,
    COUNT(*)               AS pv,
    COUNT(DISTINCT user_id) AS uv
FROM clicks
GROUP BY TUMBLE(ts, INTERVAL '1' MINUTE), page;
```

> 经验法则:**80% 的流处理需求 Flink SQL 能搞定**——别上来就写 DataStream API,先看 SQL 够不够。

## 八、Kafka Streams:嵌入应用的轻量选择

不需要独立集群,和 Spring Boot 一起跑:

```java
@Bean
public KStream<String, OrderEvent> stream(StreamsBuilder builder) {
    KStream<String, OrderEvent> orders = builder.stream("orders",
            Consumed.with(Serdes.String(), orderSerde));

    orders.filter((k, v) -> "PAID".equals(v.status))
          .groupBy((k, v) -> v.userId, Grouped.with(Serdes.Long(), orderSerde))
          .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
          .count()
          .toStream()
          .filter((win, cnt) -> cnt >= 3)
          .to("vip-candidates");

    return orders;
}
```

**优点**:无独立集群、和业务一起部署、调试方便。
**缺点**:输入输出只能是 Kafka,作业规模小。

## 十、典型场景:实时风控

```
登录事件流 ──▶ [ 同账号 1 分钟失败 5 次 ] ──▶ 封号事件
                    [ 同 IP 1 分钟登录 50 个账号 ] ──▶ IP 拉黑
                    [ 异地登录 + 新设备 ] ──▶ 强制验证
```

风控的特点:

- **多事件关联**:登录 + 下单 + 支付要联动判断,Flink 的 CEP(Complex Event Processing)就是干这个
- **状态多**:每个用户的近 N 分钟历史
- **延迟敏感**:慢一秒就被薅羊毛

```java
// Flink CEP 示例:5 秒内 3 次失败登录
Pattern<LoginEvent, ?> pattern = Pattern.<LoginEvent>begin("fails")
        .where(e -> !e.success)
        .timesOrMore(3)
        .within(Time.seconds(5));

CEP.pattern(stream.keyBy(e -> e.userId), pattern)
    .process(new PatternProcessFunction<>() { ... })
    .sinkTo(blockSink);
```

## 十一、典型场景:CDC + 流处理

把 46 章 CDC 链路接到 Flink:

```sql
-- 直接读 binlog
CREATE TABLE orders (
  id BIGINT, user_id BIGINT, amount DECIMAL(18,2), status STRING, ts TIMESTAMP_LTZ,
  PRIMARY KEY (id) NOT ENFORCED
) WITH ('connector' = 'mysql-cdc', ...);

CREATE TABLE users (
  id BIGINT, vip_level INT, region STRING,
  PRIMARY KEY (id) NOT ENFORCED
) WITH ('connector' = 'mysql-cdc', ...);

-- 流式 JOIN,实时拉宽
CREATE TABLE wide_orders (...) WITH ('connector' = 'jdbc', ...);

INSERT INTO wide_orders
SELECT o.id, o.amount, o.status, o.ts, u.vip_level, u.region
FROM orders o LEFT JOIN users FOR SYSTEM_TIME AS OF o.proc_time AS u
ON o.user_id = u.id;
```

这是 **流式数仓**(Streaming Data Warehouse)的核心做法——业务库一改,宽表立刻更新,下游可以实时分析。

## 十二、和定时任务的边界

什么时候用流处理,什么时候 44 章的定时任务就够?

| 维度 | 流处理 | 定时任务 |
| --- | --- | --- |
| 延迟要求 | 秒级 | 分钟 ~ 小时 |
| 数据量 | 大且持续 | 一次跑完 |
| 状态 | 复杂多状态 | 简单或无 |
| 维护成本 | 高(集群、监控) | 低 |
| 调试 | 难 | 容易 |

> 经验法则:**能用定时任务就别上流处理**。Flink 集群是个"大猛兽",没专人维护会反过来咬你。

## 十三、运维难点

流任务和无状态服务有本质不同:

| 痛点 | 怎么治 |
| --- | --- |
| **状态膨胀** | 状态过期(TTL)、增量 checkpoint、定期清理 |
| **作业升级** | Savepoint 停机 → 升级代码 → 从 savepoint 起 |
| **反压(backpressure)** | 下游慢导致上游堆积,看 Flink Web UI 追踪瓶颈算子 |
| **数据倾斜** | KeyBy 之后某个 key 数据量爆炸 → 加盐 / 二次 keyBy |
| **乱序严重** | Watermark 设大点,接受更高延迟换准确性 |
| **任务重启长** | 大状态恢复要几分钟,业务感知 |
| **资源估算难** | 加机器靠拍脑袋,要看 metrics 决定 |

---

## 十四、常见踩坑

1. **用 Processing Time**:对乱序敏感的业务永远算不准
2. **Watermark 太激进**:迟到数据全丢,业务怎么找回来?
3. **状态无 TTL**:跑半年状态膨胀到 100GB,checkpoint 巨慢
4. **Sink 不幂等**:Exactly-once 端到端断在最后一步
5. **窗口选错**:用 Sliding 算 Tumbling 该的指标,结果重复计数
6. **数据倾斜不处理**:某 key 占 80% 数据,那个 task 永远卡
7. **没 Savepoint 就改代码**:重启从 checkpoint 启不起来,数据从头算
8. **CEP 模式太复杂**:写出几十个 .followedBy,调试到怀疑人生
9. **流处理当批用**:每天攒一堆数据再跑,Flink 不是干这个的
10. **不监控反压**:慢慢堆,某天 Kafka 满磁盘才报警

## 十五、本章 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ Event Time + Watermark | 业务正确性的前提 |
| ✅ 状态有 TTL | 不然永远膨胀 |
| ✅ Sink 幂等或事务 | 端到端 Exactly-once |
| ✅ Savepoint + 版本管理 | 升级靠它 |
| ✅ 反压监控 | 第一时间发现瓶颈 |
| ✅ Schema 演进可控 | 字段改名 / 类型变都要兼容 |
| ✅ 优先 Flink SQL | 维护成本低 |
| ✅ 简单需求别上 Flink | 定时任务能办就办 |
| ✅ checkpoint 间隔合理 | 太短性能差,太长重启久 |
| ✅ 本地 IDE 可调试 | 别上来就压上集群 |

## 小结

流处理是后端从"批量驱动"走向"事件驱动"的分水岭——会了它,实时大屏、实时风控、实时推荐这些以前要堆很多机器的需求,变成 SQL 能解决的事。

但流处理的复杂度也是真实的:**时间、窗口、状态、Watermark、Exactly-once**——每个都是坑。
