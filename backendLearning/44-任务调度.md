# 任务调度

后端业务里有一类需求是"按时间自动执行"——对账、报表、清理过期数据、定时推送、缓存预热、数据同步……几乎每个项目都跑不掉。

这一章把任务调度的演进路径讲清:**从单机定时,到分布式调度,到工作流编排**。

---

## 一、最朴素的方式:Linux Crontab

```bash
# 每天凌晨 2 点跑对账脚本
0 2 * * * /opt/scripts/reconcile.sh >> /var/log/reconcile.log 2>&1
```

**优点**:零依赖、最稳。
**缺点**:

1. **没有可视化**——任务多了管理不动
2. **没有失败重试**——脚本挂了不知道
3. **没有任务编排**——A 跑完才能跑 B?自己写依赖
4. **多实例会重复执行**——同一个脚本部署到 3 台机器,跑 3 次

> 经验法则:**5 个以下脚本、单机部署、对失败容忍 → crontab 完全够用**。别上来就 XXL-Job。

## 二、Spring 的 @Scheduled

应用内嵌定时任务:

```java
@EnableScheduling
@Component
public class ReportTask {
    @Scheduled(cron = "0 0 2 * * ?")              // 每天 2:00
    public void dailyReport() { ... }

    @Scheduled(fixedDelay = 60_000, initialDelay = 5_000)
    public void healthCheck() { ... }
}
```

| 触发方式 | 含义 |
| --- | --- |
| `cron = "..."` | Cron 表达式 |
| `fixedRate = N` | 每 N ms 执行一次(不等上次跑完) |
| `fixedDelay = N` | 上次完成后再等 N ms |
| `initialDelay = N` | 启动后延迟 N ms 再开始 |

**Spring 单机定时的两个致命问题**:

1. **多实例重复执行**:部署 3 个 pod,凌晨 2 点跑 3 次对账
2. **重启可能漏跑**:2:00 重启了,这一次就丢了

## 三、单机问题 → 分布式问题

要解决"多实例重复 + 不漏跑",有几种粗暴方案:

### 方案 A:用分布式锁

```java
@Scheduled(cron = "0 0 2 * * ?")
public void dailyReport() {
    if (redisLock.tryLock("reportLock", 30, TimeUnit.MINUTES)) {
        try { doReport(); } finally { redisLock.unlock("reportLock"); }
    }
}
```

**优点**:实现简单。
**缺点**:看不见任务执行情况、不能动态调度、无分片。

### 方案 B:数据库乐观锁

```sql
UPDATE schedule_lock SET owner = ?, expires_at = ?
WHERE name = 'reportLock' AND (owner IS NULL OR expires_at < NOW())
```

抢到锁的那台机器执行,其他跳过。`Shedlock` 库就是干这个的,适合中小项目。

### 方案 C:专门的分布式调度系统

任务一多就上专业框架——下一节展开。

## 四、分布式调度框架对比

| 框架 | 出身 | 核心特性 | 适合 |
| --- | --- | --- | --- |
| **XXL-Job** | 大众点评 | 中文友好、UI 完整、轻量 | 国内中小型团队首选 |

| **ElasticJob** | 当当 → 阿帕奇 | 分片、弹性 | 数据量大需要分片场景 |
| **PowerJob** | 蚂蚁 | XXL-Job 进化版,支持 DAG | 任务有依赖关系 |
| **Airflow** | Airbnb | DAG、Python 生态、回填 | 数据 ETL、离线任务 |
| **DolphinScheduler** | 国产 | 可视化 DAG | 数据中台、ETL |
| **K8s CronJob** | K8s 原生 | 容器化、声明式 | 任务都容器化的环境 |

## 五、XXL-Job 架构与上手

```
┌──────────────┐
│  Admin 控制台 │   ◀── 调度中心:决定何时调谁
└──────────────┘
       │ 触发
       ▼
┌──────────────────────────┐
│  Executor 执行器集群       │   ◀── 业务应用,自动注册到 Admin
│  ┌─────┐ ┌─────┐ ┌─────┐  │
│  │svc-1│ │svc-2│ │svc-3│  │
│  └─────┘ └─────┘ └─────┘  │
└──────────────────────────┘
```

**Admin** 负责调度策略、任务分发、失败重试。**Executor** 是嵌入业务进程的 SDK,启动时把自己注册到 Admin,等被触发。

### Spring Boot 接入

```yaml
xxl:
  job:
    admin: { addresses: http://xxl-admin:8080/xxl-job-admin }
    executor:
      app-name: order-service
      port: 9999
```

```java
@Component
public class ReportHandler {
    @XxlJob("dailyReport")
    public void dailyReport() {
        XxlJobHelper.log("start...");
        // 业务逻辑
        XxlJobHelper.handleSuccess();
    }
}
```

控制台里配 cron + 路由策略即可执行。

### 路由策略(决定打到哪个 executor)

| 策略 | 含义 |
| --- | --- |
| FIRST / LAST | 永远打第一个 / 最后一个 |
| ROUND | 轮询 |
| RANDOM | 随机 |
| **CONSISTENT_HASH** | 一致性哈希(同一任务总打同一台,适合需要本地缓存) |
| **LEAST_FREQUENTLY_USED** | 最少使用 |
| **SHARDING_BROADCAST** | 广播分片(每台机器跑一片,关键特性) |

## 六、分片广播:大数据量的杀器

```
任务:处理 1000 万条订单,扫表慢得要命

错误做法:一台机器扫,跑 6 小时
正确做法:3 台机器各扫 1/3,2 小时跑完
```

XXL-Job / ElasticJob 都支持分片:

```java
@XxlJob("processOrders")
public void process() {
    int shard      = XxlJobHelper.getShardIndex();   // 当前机器的分片号
    int shardTotal = XxlJobHelper.getShardTotal();   // 总分片数

    // 按分片号过滤数据
    orderMapper.scan(orderId -> orderId % shardTotal == shard)
               .forEach(this::processOne);
}
```

> 经验法则:**任务跑超过 30 分钟、数据量大、可按 ID 分片 → 上分片广播**。能把跑批时间砍到 1/N。

## 七、失败重试与告警

调度框架的"工程化"主要体现在这里:

| 机制 | 作用 |
| --- | --- |
| **失败重试** | 失败后等 N 秒重试 K 次 |
| **失败告警** | 连续失败 / 超时 → 邮件 / 钉钉 / 企微通知 |
| **执行日志** | 任务输出可在 Admin 实时查看 |
| **超时熔断** | 跑超过预期时间自动 kill |
| **任务依赖** | A 完成后才触发 B(子任务) |
| **阻塞策略** | 上次还没跑完,这次怎么办——单机串行 / 丢弃 / 覆盖 |

阻塞策略最容易踩坑:

```
SERIAL_EXECUTION:  排队等上次完成(默认,最稳)
DISCARD_LATER:     新任务直接丢(适合心跳/上报型)
COVER_EARLY:       kill 旧任务,跑新的(危险,慎用)
```

## 八、任务编排:DAG 才是真本事

简单 cron 是"独立任务",但实际业务里任务之间有依赖:

```
拉数据 ──▶ 清洗 ──▶ ┬──▶ 汇总报表
                    └──▶ 推荐模型训练
                              ▼
                          推送结果
```

这就是 **DAG(有向无环图)**——Airflow / DolphinScheduler / PowerJob 的拿手好戏。

### Airflow 例子

```python
from airflow.decorators import dag, task
from datetime import datetime

@dag(start_date=datetime(2026,1,1), schedule="0 2 * * *", catchup=False)
def daily_pipeline():
    @task
    def fetch():   ...
    @task
    def clean(d): ...
    @task
    def report(d): ...
    @task
    def train(d):  ...

    cleaned = clean(fetch())
    [report(cleaned), train(cleaned)]

daily_pipeline()
```

Airflow 的优点是 **Python 即代码**——任务定义、依赖、参数都在 Python 里,可版本化、可复用。

> 经验法则:**纯定时任务 → XXL-Job / 类似;数据 ETL / 复杂依赖 → Airflow / DolphinScheduler**。两者用途不重叠。

## 九、K8s CronJob:容器化场景的轻量方案

如果你的任务已经容器化:

```yaml
apiVersion: batch/v1
kind: CronJob
metadata: { name: daily-report }
spec:
  schedule: "0 2 * * *"
  concurrencyPolicy: Forbid          # 上次没跑完不再起
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 5
  jobTemplate:
    spec:
      backoffLimit: 3                # 失败重试 3 次
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: report
            image: report-job:1.0
            command: ["/app/run.sh"]
```

**优点**:声明式、与应用统一管理、不需要单独部署调度系统。
**缺点**:没有可视化任务管理、跨节点编排弱、复杂依赖搞不定。

> 经验法则:**简单的、隔离的、容器化的批任务 → K8s CronJob**。复杂调度还是上 XXL-Job / Airflow。

## 十、调度系统的可观测性

调度任务最忌"跑完没人看"——出问题往往是**几天后才被发现**。最低限度三件事:

1. **每次执行记录入库**:任务名、开始/结束时间、状态、耗时、错误堆栈
2. **失败立即告警**:钉钉/企微机器人,带任务名 + 错误摘要 + 控制台链接
3. **关键任务有 SLA 监控**:对账任务每天 8 点前必须跑完,过期未完成 → 告警

```
告警分级:
  P0:核心对账失败 / 超时           → 电话叫醒
  P1:重要日报失败                  → 群里 @
  P2:辅助任务失败                  → 邮件,工作时间看
```

## 十一、跨业务的常见调度任务模板

| 类型 | 例子 | 注意 |
| --- | --- | --- |
| **对账** | 跨系统数据比对 | 必有兜底人工介入流程 |
| **报表** | 日报 / 周报 | 跑批失败要可手动补 |
| **清理** | 删过期数据 / 清缓存 | 加 LIMIT 分批,别一把删 |
| **推送** | 营销短信 / push | 限速,防被运营商封 |
| **预热** | 缓存预热 / 模型预加载 | 在低峰期跑 |
| **同步** | 全量同步 ES / ClickHouse | 用 CDC 替代,见 46 章 |
| **巡检** | 数据一致性自检 | 发现差异自动修复 |

## 十二、常见踩坑

1. **没有幂等性**:补跑一次结果翻倍——比如对账金额累加两次
2. **任务跑超时也不停**:阻塞下次执行,雪球越滚越大
3. **日志全打 stdout**:要查就要登机器,XXL-Job 日志组件不用白瞎
4. **任务里 catch 一切**:吞掉异常,Admin 看到全 success,实际数据没处理
5. **凌晨集中跑批**:00:00 一堆任务排队 → DB 被打爆,错峰
6. **忘了时区**:服务器 UTC 但 cron 按 +08 写 → 任务延后 8 小时
7. **大任务不分片**:单机跑 6 小时,有分片机制不用
8. **任务里直接调外部 HTTP**:不限速 / 不重试 / 没超时
9. **失败重试不是幂等的**:重试一次就脏数据
10. **没有"补跑"通道**:某天任务失败了,需要跑历史数据 → 要么改 cron 要么手工传参,事先就该规划

## 十三、本章 Checklist

| 项 | 说明 |
| --- | --- |
| ✅ 任务必须幂等 | 重跑/补跑前提 |
| ✅ 任务有超时熔断 | 不能无限跑下去 |
| ✅ 失败有告警 | 第一时间发现 |
| ✅ 大任务分片 | 单机跑 1 小时以上要考虑 |
| ✅ 错峰运行 | 别都堆在 00:00 |
| ✅ 关键任务 SLA 监控 | 必须几点前完成 |
| ✅ 日志可在线查 | 不用登机器 |
| ✅ 日志能保留 N 天 | 排查回溯 |
| ✅ 阻塞策略明确 | 串行 / 丢弃 / 覆盖 |
| ✅ 提供补跑入口 | 单次执行 + 可指定参数 |

## 小结

任务调度看起来简单,做"业务跑通"几行代码就行;做"运维放心"要解决重复执行、失败重试、超时熔断、分片、可视化、告警、补跑——这些工程能力。

选型快速参考:

- **<5 个简单任务、单机** → crontab
- **少量任务、嵌入应用** → Spring `@Scheduled` + Shedlock
- **十几到上百任务、需要可视化** → XXL-Job / PowerJob
- **数据 ETL、有依赖关系** → Airflow / DolphinScheduler
- **容器化、简单批任务** → K8s CronJob
