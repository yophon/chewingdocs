# K8s 工作负载与服务

Pod 是 K8s 最小单元，但实际生产**不直接管 Pod**——用"工作负载（Workload）"来描述"要跑几个、怎么跑、挂了怎么办"。本章覆盖 Deployment、StatefulSet、DaemonSet、Job，以及 Service/Ingress 的流量入口体系。

---

## 一、Deployment：无状态服务的标配

适合：Spring Boot、ElysiaJS、Web Server 等**无状态**应用。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多开 1 个新 Pod
      maxUnavailable: 0  # 滚动期间保证可用数不减少
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api
          image: myrepo/api-server:1.2.0
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:           # 就绪探针：未就绪不接流量
            httpGet:
              path: /actuator/health
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
          livenessProbe:            # 存活探针：失败自动重启
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
```

### 常用操作

```bash
kubectl apply -f deployment.yaml
kubectl rollout status deployment/api-server
kubectl rollout history deployment/api-server
kubectl rollout undo deployment/api-server          # 回滚上一版本
kubectl rollout undo deployment/api-server --to-revision=2
kubectl scale deployment/api-server --replicas=5
```

### HPA 自动伸缩

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
```

> 经验法则：`requests` 必须设，HPA 的 CPU 利用率是基于 requests 算的。没有 requests 等于 HPA 失效。

## 二、StatefulSet：有状态服务

适合：MySQL、Redis、Elasticsearch、Kafka——**需要稳定网络标识和持久化存储**的应用。

与 Deployment 的差异：

| 特性 | Deployment | StatefulSet |
| --- | --- | --- |
| Pod 名字 | 随机后缀 | 固定序号（`pod-0`、`pod-1`） |
| 启动顺序 | 并行 | 顺序（0→1→2） |
| 存储 | 共享 PVC 或无 | 每个 Pod 独享 PVC |
| DNS | 无固定 | `pod-0.svc-name.ns.svc.cluster.local` |

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql          # 必须配合 headless Service
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
        - name: mysql
          image: mysql:8.0
          env:
            - name: MYSQL_ROOT_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: mysql-secret
                  key: root-password
          volumeMounts:
            - name: data
              mountPath: /var/lib/mysql
  volumeClaimTemplates:        # 每个 Pod 自动创建一个 PVC
    - metadata:
        name: data
      spec:
        accessModes: [ReadWriteOnce]
        resources:
          requests:
            storage: 20Gi
```

## 三、DaemonSet：每个节点跑一个

适合：日志采集（Fluentd）、监控 Agent（Prometheus Node Exporter）、网络插件（Calico）。

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-exporter
spec:
  selector:
    matchLabels:
      app: node-exporter
  template:
    metadata:
      labels:
        app: node-exporter
    spec:
      hostNetwork: true        # 使用宿主机网络
      hostPID: true
      tolerations:             # 允许调度到 master 节点
        - operator: Exists
      containers:
        - name: node-exporter
          image: prom/node-exporter:latest
          ports:
            - containerPort: 9100
              hostPort: 9100
```

## 四、Job / CronJob：一次性 & 定时任务

### Job

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  backoffLimit: 3    # 失败重试 3 次
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: myrepo/migrate:latest
          command: ["./migrate", "up"]
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-secret
                  key: url
```

### CronJob

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: daily-report
spec:
  schedule: "0 2 * * *"     # 每天凌晨 2 点
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: report
              image: myrepo/report:latest
```

## 五、Service：集群内流量

Pod IP 是临时的，重启就变。**Service** 提供稳定的虚拟 IP 和 DNS，自动负载均衡到后端 Pod。

### 类型对比

| 类型 | 用途 | 能否集群外访问 |
| --- | --- | --- |
| `ClusterIP`（默认） | 集群内服务发现 | ❌ |
| `NodePort` | 开发测试，暴露宿主机端口 | ✅（`NodeIP:NodePort`） |
| `LoadBalancer` | 云平台（AWS/阿里云）自动创建 LB | ✅ |
| `ExternalName` | 把外部域名映射进集群 | — |

```yaml
apiVersion: v1
kind: Service
metadata:
  name: api-server-svc
spec:
  selector:
    app: api-server        # 匹配 Pod 标签
  ports:
    - port: 80             # Service 端口
      targetPort: 8080     # Pod 端口
  type: ClusterIP
```

### Headless Service（StatefulSet 配套）

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql
spec:
  clusterIP: None          # ← headless 关键
  selector:
    app: mysql
  ports:
    - port: 3306
```

Headless Service 不分配 VIP，DNS 直接解析到各 Pod IP——`mysql-0.mysql.default.svc.cluster.local`。

## 六、Ingress：集群外 HTTP 流量

Service `LoadBalancer` 每个服务都要一个 LB，成本高。**Ingress** 统一在一个 LB 上做 7 层路由。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
    - hosts: [api.example.com]
      secretName: api-tls
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /v1
            pathType: Prefix
            backend:
              service:
                name: api-server-svc
                port:
                  number: 80
          - path: /admin
            pathType: Prefix
            backend:
              service:
                name: admin-svc
                port:
                  number: 80
```

> 需要先部署 Ingress Controller（最常用：`ingress-nginx`）：
> ```bash
> kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/cloud/deploy.yaml
> ```

## 七、探针 cheatsheet

| 探针 | 失败后果 | 典型用途 |
| --- | --- | --- |
| `livenessProbe` | 重启 Pod | 检测死锁、OOM 后应用僵死 |
| `readinessProbe` | 从 Service 摘流量 | 应用启动中、数据库未连上 |
| `startupProbe` | 替换 liveness 在启动阶段 | 慢启动应用（JVM 预热） |

```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  failureThreshold: 30   # 最多等 30 * 10s = 300s
  periodSeconds: 10
```

## 八、小结

| 工作负载 | 适合 |
| --- | --- |
| Deployment | 无状态服务（最常用） |
| StatefulSet | 数据库、有序服务 |
| DaemonSet | 每节点 Agent |
| Job / CronJob | 一次性 / 定时任务 |

- Service `ClusterIP` 管集群内通信，Ingress 管集群外 HTTP 入口。
- **readinessProbe 一定要配**，不然 Pod 还没就绪就接到流量，导致 502。
