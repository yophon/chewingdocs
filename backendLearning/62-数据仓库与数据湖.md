# 数据仓库与数据湖

39 章讲了 ClickHouse(实时 OLAP),46 章讲了 CDC,47 章讲了流处理——但**数据团队真正吃饭的家伙**还没讲:**离线数仓 + 数据湖**。

业务库存"现在的状态",数据仓库存"历史 + 维度",数据湖存"全量原始 + 半结构化"。三者搭起来才是"现代数据栈"。这章面向后端工程师讲清楚这块的概念、技术和分工。

---

## 一、为什么不用业务库做分析

```
业务库 OLTP:
  - 行式存储,小事务多并发
  - 几十张表,关系范式
  - 主键查询快,聚合慢

分析需求:
  - "过去 3 年每月各省销售额按品类排" → 全表扫几亿条
  - 在 OLTP 上跑 → DBA 报警,业务库被打趴
  - → 必须分离
```

| 维度 | OLTP(业务库) | OLAP(分析库) |
| --- | --- | --- |
| 数据量 | TB 级 | PB 级 |
| 查询模式 | 点查 + 短事务 | 全表扫 + 大聚合 |
| 写模式 | 高频小写入 | 批量大写入 |
| 存储 | 行式(InnoDB) | 列式(Parquet / ORC) |
| 并发 | 几千 | 几十 |
| 延迟要求 | ms | s ~ min |
| 场景 | 日常业务 | BI / 报表 / AI 训练 |

## 二、数仓 vs 数据湖 vs 湖仓一体

```
数据仓库(Data Warehouse)
   - 结构化数据,有 schema
   - 经过清洗、规整
   - 用 SQL 查询(Hive / Snowflake / BigQuery)
   - 适合 BI 报表

数据湖(Data Lake)
   - 任何格式:CSV / JSON / Parquet / 图像 / 日志
   - 原始数据,先存后用
   - 一份原始数据多种使用方式
   - S3 / HDFS / OSS 上一堆文件

湖仓一体(Lakehouse) ← 当前主流
   - 数据湖的存储 + 数仓的事务/schema/性能
   - Iceberg / Delta Lake / Hudi 是核心
   - 一套数据,既能 BI 又能 ML
```

> 经验法则:**新项目直接上湖仓一体**(Iceberg / Delta Lake)。传统纯 Hive 数仓 + 单独 S3 数据湖的"双套架构"已经过时,运维成本高,数据搬来搬去。

## 三、数据分层(Lambda → Medallion)

```
Bronze(原始层 / ODS)
   ├─ 业务库 CDC、日志、第三方接口的原样数据
   └─ "保留原貌,事后追溯"

Silver(明细层 / DWD)
   ├─ 清洗、去重、补字段、关联维度
   └─ 可信、规范化、字段命名统一

Gold(汇总层 / ADS)
   ├─ 按主题聚合:用户日活、商品 GMV、地区销量
   └─ BI 报表 / 仪表板直接消费

External(应用层)
   └─ 推到 Redis / ES / ClickHouse 给业务查
```

每层职责清晰,**bronze 出问题不影响 gold 的稳定性**。

> 经验法则:**严格按层流动,不允许跨层引用**。Gold 不要直接读 Bronze——一旦上游 schema 改,Gold 也跟着炸。

## 四、Hive:老牌数仓

Hive 把"SQL 翻译成 MapReduce / Spark / Tez 任务",跑在 HDFS / S3 上。

```sql
CREATE TABLE order_fact (
  order_id   STRING,
  user_id    STRING,
  amount     DECIMAL(10,2),
  created_at TIMESTAMP
)
PARTITIONED BY (dt STRING)         -- 按日期分区
STORED AS ORC                      -- 列式格式
TBLPROPERTIES ('orc.compress' = 'SNAPPY');

-- 写入指定分区
INSERT INTO order_fact PARTITION (dt='2026-04-15')
SELECT order_id, user_id, amount, created_at FROM ods.order WHERE dt='2026-04-15';

-- 查询
SELECT user_id, SUM(amount) AS total
FROM order_fact
WHERE dt BETWEEN '2026-04-01' AND '2026-04-30'
GROUP BY user_id;
```

**Hive 还活着的理由**:

- 公司历史数据已经在 Hive 表里
- HiveQL 兼容性强,Spark / Presto / Trino 都能查 Hive 元数据
- Metastore(元数据仓库)是事实标准

**Hive 的坑**:

- MapReduce 引擎慢(Spark / Tez 引擎好得多)
- 不支持 ACID(只有 ACID Tables 实现勉强,生产慎用)
- 小文件问题严重

> 经验法则:**新项目别从零搭 Hive**——直接上 Iceberg + Spark/Trino,Metastore 还是能复用 HMS 协议。

## 五、Parquet / ORC:列存格式

数据湖的底层文件格式:

| 格式 | 出处 | 特点 |
| --- | --- | --- |
| **Parquet** | Twitter / Cloudera | 跨语言生态最广,列存事实标准 |
| **ORC** | Hortonworks / Hive | 压缩稍好,主要在 Hive 生态 |
| **Avro** | 行存,适合 Kafka 流式 | 写入快,查不优化 |
| **CSV / JSON** | 通用 | 不要在数据湖里用 |

**列存的两个杀手锏**:

```
1. 列式扫描:
   只 SELECT 3 列 → 只读这 3 列的数据 → IO 减少 90%+
   
2. 编码压缩:
   重复值多的列(地区、性别)用字典编码,体积小一个量级
```

```python
# Pandas 写 Parquet
df.to_parquet('orders.parquet', compression='snappy', index=False)

# Spark 读
df = spark.read.parquet('s3a://bucket/orders/')
df.filter("amount > 100").select("user_id", "amount").show()
```

> 经验法则:**数据湖里只用 Parquet/ORC 存,绝不用 CSV/JSON**。CSV 一份数据,Parquet 同等列少 80% IO,查询快几十倍。

## 六、Iceberg:湖仓一体的事实标准

Apache Iceberg 是 Netflix 开源的"表格式"——**在 Parquet 文件之上加了一层元数据,让数据湖有数仓的特性**:

| 能力 | 普通 Parquet | Iceberg |
| --- | --- | --- |
| ACID 事务 | ❌ | ✅ |
| Schema 演进 | 改字段会炸 | ✅ 加列 / 改类型自动兼容 |
| 分区演进 | 改分区策略要重写 | ✅ 不重写历史 |
| 时间旅行 | ❌ | ✅ 任意历史版本可查 |
| Compaction | 手工 | 内置 |
| 隐式分区 | 业务必须知道 dt 字段 | ✅ 自动 |

```sql
-- Iceberg 建表(Spark SQL)
CREATE TABLE catalog.db.orders (
  id BIGINT,
  user_id STRING,
  amount DECIMAL(10,2),
  created_at TIMESTAMP
)
USING iceberg
PARTITIONED BY (days(created_at))      -- 隐式按天分区,业务不用知道
TBLPROPERTIES ('format-version' = '2');

-- 写入,无需指定分区
INSERT INTO orders VALUES (1, 'u1', 99.0, current_timestamp());

-- 时间旅行
SELECT * FROM orders TIMESTAMP AS OF '2026-04-01 00:00:00';
SELECT * FROM orders VERSION AS OF 12345;

-- Schema 演进
ALTER TABLE orders ADD COLUMN country STRING;
```

**Iceberg 的核心**:**元数据快照**。每次写入产生一个 snapshot,读时锁定到某个 snapshot——这就是 ACID 和时间旅行的基础。

```
table/
├── data/
│   └── 2026-04-15/
│       ├── 00000-xxx.parquet
│       └── 00001-yyy.parquet
└── metadata/
    ├── v1.metadata.json     ← 当前表元数据
    ├── snap-12345.avro      ← 快照清单
    └── manifest-list-xxx    ← 文件清单
```

## 七、Delta Lake / Hudi:同类竞品

| 产品 | 出处 | 特点 |
| --- | --- | --- |
| **Iceberg** | Netflix → Apache | 开放标准,引擎中立(Spark / Flink / Trino 都强) |
| **Delta Lake** | Databricks → 开源 | Spark 生态最强,Databricks 主推 |
| **Hudi** | Uber → Apache | **更新 / 增量摄入强**,适合 CDC |

**怎么选**:

| 场景 | 推荐 |
| --- | --- |
| 多引擎查询(Spark + Trino + Flink) | **Iceberg** |
| 全 Databricks 栈 | **Delta Lake** |
| 频繁 update/delete + CDC | **Hudi** |
| 不知道选什么 | **Iceberg**(社区最活跃,中立) |

> 经验法则:**新项目大多数选 Iceberg**——**开放、中立、引擎适配最广**。Delta 和 Hudi 都在加大兼容,但 Iceberg 的中立位赢面最大。

## 八、查询引擎:Spark / Trino / Presto

数据存储和计算解耦,**存储在 S3,计算用谁随便挑**:

| 引擎 | 强在哪 | 适合 |
| --- | --- | --- |
| **Spark** | 复杂 ETL、ML、批处理 | 离线作业 |
| **Flink** | 流处理 + 批处理一体 | 实时数仓 |
| **Trino**(原 PrestoSQL) | 联邦查询(跨 Hive / Iceberg / MySQL / Kafka) | 即席查询 / BI |
| **Presto**(Meta) | 同上 | 同上 |
| **Doris / StarRocks** | 实时 OLAP | 报表系统 |
| **DuckDB** | 单机分析,Pandas 替代 | 数据探索 |
| **Athena** | AWS 托管 Presto | AWS 上零运维 |

```sql
-- Trino 跨源查询(联邦)
SELECT
  o.id, u.email, p.name
FROM iceberg.warehouse.orders o
JOIN mysql.crm.users u ON o.user_id = u.id          -- 业务库
JOIN hive.product.catalog p ON o.product_id = p.id   -- Hive 仓
WHERE o.dt = '2026-04-15';
```

**Trino 是数据团队和业务团队的"通用查询入口"**——不需要把数据全搬到一个库,SQL 跨源直接 JOIN。

## 九、ETL / ELT:数据搬运的灵魂

```
ETL(传统):Extract → Transform → Load
   数据从源头抽出 → 在内存/中间层转换 → 装进目的地

ELT(现代):Extract → Load → Transform
   数据原样装进数据湖/仓 → 在仓里 SQL 转换
```

**ELT 的优势**:存储便宜,Transform 都在仓库的 SQL 引擎里跑,可重跑、可审计。

| 工具 | 干什么 |
| --- | --- |
| **Airflow** | 调度 DAG,定时跑 ETL 作业 |
| **dbt** | "SQL 即转换"——SQL + Jinja 模板,数据团队的瑞士军刀 |
| **Dagster / Prefect** | Airflow 替代,体验更好 |
| **Spark / Flink** | 真正干活的引擎 |
| **Airbyte / Fivetran** | 数据集成,从各种 SaaS 抽数据 |

### dbt 实例

```sql
-- models/silver/dim_user.sql
WITH src AS (
    SELECT * FROM {{ ref('bronze_user') }}
)
SELECT
    id,
    LOWER(TRIM(email)) AS email,
    COALESCE(country, 'UNKNOWN') AS country,
    created_at
FROM src
WHERE deleted = FALSE;
```

```bash
dbt run                    # 跑所有模型
dbt test                   # 跑数据质量测试
dbt docs generate          # 自动出血缘图
```

> 经验法则:**dbt 是 2024 年代数据团队的 Spring Boot**——所有数据转换全用 SQL + dbt 写,血缘自动可视化,版本化进 Git。

## 十、数据建模:维度建模(Kimball)

最主流的数仓建模方法:**事实表 + 维度表 + 星型 / 雪花模型**。

```
事实表(Fact):
  - 业务事件:订单、支付、点击
  - 大表,几亿~几百亿行
  - 几个维度 ID + 度量(金额、数量)

维度表(Dimension):
  - 描述对象:用户、商品、时间、地区
  - 小表,几万~几百万行
  - 大量描述字段

星型模型:事实表中心,维度表围一圈
```

```
                ┌─────────────┐
                │ dim_user    │
                └──────┬──────┘
                       │ user_id
                       ▼
┌─────────┐    ┌──────────────┐    ┌──────────┐
│dim_time │───▶│  fact_order  │◀───│dim_product│
└─────────┘    └──────┬───────┘    └──────────┘
                       │ region_id
                       ▼
                ┌─────────────┐
                │ dim_region  │
                └─────────────┘
```

**SCD(渐变维度)**:用户名字改了,历史订单上的 user_name 要不要跟着改?

| 类型 | 含义 |
| --- | --- |
| SCD Type 1 | 直接覆盖(不留历史) |
| **SCD Type 2** | 插入新记录 + 标记有效期(留历史,主流) |
| SCD Type 3 | 加旧值列(只留一版历史) |

> 经验法则:**用户类维度一律 SCD2**——金融、医疗合规要求"可还原任意时刻的状态"。

## 十一、数据质量与数据治理

```
数据质量五维度:
  - 完整性:必填字段不缺
  - 一致性:跨表字段含义对齐
  - 准确性:与真实业务相符
  - 及时性:按时产出
  - 唯一性:主键不重复
```

| 工具 | 干什么 |
| --- | --- |
| **dbt tests** | 在 SQL 模型里直接定义校验 |
| **Great Expectations** | 数据契约 + 自动校验 |
| **Soda** | 数据质量监控 |
| **DataHub / OpenMetadata / Atlan** | 元数据管理 + 血缘 + 治理 |
| **Apache Atlas** | 老牌元数据 |

### dbt tests 实例

```yaml
# models/schema.yml
models:
  - name: dim_user
    columns:
      - name: id
        tests: [unique, not_null]
      - name: email
        tests:
          - not_null
          - dbt_utils.email_format
```

> 经验法则:**数据质量必须自动化**——靠人巡检数据,迟早出 BI 报表数字错了一个量级被业务发现。每天 dbt test 不通过就告警。

## 十二、实时数仓的演进

```
经典 Lambda(分两路):
  实时层:Flink → Kafka → Druid/CK
  离线层:Hive 每天 T+1 跑批
  → 两套代码维护,数据可能不一致

Kappa(只用一路):
  Flink 流处理一切,落到 Iceberg
  → 简单,但故障重放成本高

湖仓 + 流批一体(2024+):
  CDC → Flink → Iceberg(实时写)
  Iceberg 既是流处理 sink,也是离线表
  → 一份数据,流批共用 ★
```

> 经验法则:**新建数据栈直接上"湖仓 + 流批一体"**——Iceberg + Flink 已经成熟,告别 Lambda 那套双写。

## 十三、和后端工程师的接口

后端工程师不用直接写 dbt / Spark,但要懂这几件事:

1. **CDC 的设计配合**:让数据团队拿到 binlog,业务表加 `created_at / updated_at` / 软删标志
2. **Schema 变更通知**:加字段 / 删字段 → 必须提前通知数据团队,否则下游报表崩
3. **PII / 敏感字段**:数据团队要做脱敏,**业务侧标好哪些字段是敏感的**
4. **数据回填请求**:业务方要"3 年前的数据补一份"——能不能跑取决于业务库还存不存
5. **数据消费回路**:数据团队的标签 / 分群结果回写到业务库 / Redis,业务消费

## 十四、常见踩坑

1. **业务库直接跑分析 query**:大聚合把业务库打趴
2. **数据湖里堆 CSV / JSON**:查询慢百倍
3. **不分区**:全表扫几十亿条
4. **小文件灾难**:每分钟一个 Parquet,几百万小文件,NameNode/Metadata 撑爆
5. **不做 Compaction**:Iceberg 的 manifest 越来越长,query plan 越来越慢
6. **直接 Hive INSERT OVERWRITE**:写时全表锁,查询都阻塞
7. **没有数据分层**:gold 直接读 ods,上游一改全炸
8. **dbt 模型没 test**:数据质量出问题没人发现
9. **业务表没 updated_at**:CDC 增量同步不行
10. **PII 没脱敏直接进湖**:GDPR 投诉
11. **DAG 定 23:55 跑,业务库 T+1 还没出**:数据空跑
12. **报表 SQL 业务团队自己写**:不用维度建模,每个人重复算指标,口径乱
13. **Schema 演进破坏性 ALTER**:历史数据无法读
14. **数据没血缘**:出问题查不到上游
15. **湖仓选了 3 套(Iceberg + Delta + Hudi 都有)**:运维爆炸,数据搬来搬去

## 小结

数据仓库 / 数据湖是后端的"姐妹学科"——后端把数据写下来,数据团队把它变成"决策"。

记住三件事:

1. **现代数据栈 = 湖仓一体(Iceberg / Delta / Hudi)**——Lambda 双套架构是历史
2. **存储和计算分离**——数据在 S3,引擎按需选(Spark / Flink / Trino)
3. **后端服务数据团队的最大贡献是"业务表设计 CDC 友好 + Schema 变更提前通知"**

下一章我们换业务侧——**支付集成与三方对接**:对账、回调安全、幂等、异步通知。所有 toC / toB 业务都绕不开的"钱的事"。
