# 消息队列基础（Kafka / RabbitMQ / NATS）

到这里你已经能写出"请求来了→查库→返响应"的同步链路。但工业级系统会遇到这些问题：

- 注册要发邮件、推送、加积分——一个请求做 5 件事，谁慢全慢
- 秒杀峰值 10 倍流量，数据库直接被打挂
- A 服务想通知 B、C、D，但谁都不想互相耦合

这些都是 **消息队列（MQ）** 的主战场：**异步化、削峰、解耦**。

---

## 一、为什么需要 MQ

把同步调用换成"发消息、监听消息"，立刻得到三个收益：

| 收益 | 含义 |
| --- | --- |
| 异步 | 主流程不等下游，秒级动作变毫秒级响应 |
| 削峰 | 请求堆在队列里慢慢消费，DB 不被打爆 |
| 解耦 | 上游只管发，下游想加几个就加几个 |

但代价也很真实：

- **复杂度上升**：多一个中间件要部署、监控、扩容
- **顺序与重复**：网络重试导致同一条消息被消费多次
- **最终一致**：再也不能假设"调用完了就是改完了"

---

## 二、三大主流选型直觉

| 产品 | 模型 | 强项 | 弱项 |
| --- | --- | --- | --- |
| **Kafka** | 持久化日志 + 分区 | 海量、回溯、流式 | 单条延迟稍高、运维重 |
| **RabbitMQ** | 经典 AMQP，路由灵活 | 业务消息、复杂路由 | 单队列吞吐有上限 |
| **NATS / JetStream** | 轻量 Pub/Sub | 微秒延迟、易部署 | 生态小、企业特性少 |
| RocketMQ | 类 Kafka + 事务消息 | 国内电商场景成熟 | 海外社区弱 |
| Pulsar | 存算分离 | 多租户、超大规模 | 学习曲线陡 |

> 经验法则：**日志/事件流用 Kafka，业务消息用 RabbitMQ，超低延迟用 NATS**，国内电商可考虑 RocketMQ。

## 三、Kafka 核心概念

```
Producer ──▶ Topic ──▶ Partition 0 ──▶ Consumer Group A
                  ├──▶ Partition 1 ──▶ Consumer Group A
                  └──▶ Partition 2 ──▶ Consumer Group B
```

- **Topic**：逻辑通道
- **Partition**：分区，一个 Topic 拆 N 份，决定并行度上限
- **Offset**：每条消息在分区里的位置，消费者自己记
- **Consumer Group**：同组共享消费，不同组互不影响

关键认知：**Kafka 是日志，不是队列**。消息默认按时间保留（如 7 天），不会"消费就删"——这让回放、多消费者、流计算都成立。

### Spring Boot 极简示例

```java
// 生产者
@Service
class OrderEventPublisher {
    @Autowired
    private KafkaTemplate<String, String> kafka;

    public void send(Order order) {
        // 用 orderId 作为 key，保证同一订单的消息进同一分区
        kafka.send("order.created", order.getId(), toJson(order));
    }
}

// 消费者
@KafkaListener(topics = "order.created", groupId = "notify-svc")
public void onOrder(String payload) {
    log.info("got order: {}", payload);
}
```

```yaml
spring:
  kafka:
    bootstrap-servers: kafka:9092
    consumer:
      group-id: notify-svc
      auto-offset-reset: earliest
      enable-auto-commit: false   # 手动提交，避免丢消息
    producer:
      acks: all                    # 所有副本确认才算成功
```

## 四、RabbitMQ 核心概念

```
Producer ──▶ Exchange ──(binding key)──▶ Queue ──▶ Consumer
              │
              ├── direct      （精确匹配）
              ├── topic       （通配符 order.*.paid）
              ├── fanout      （广播）
              └── headers     （按头部匹配）
```

- 消息**经过 Exchange 路由到 Queue**，消费者监听 Queue
- 默认**消费成功才删除**（ack），失败可以 nack 重入
- 支持 **死信队列、TTL、优先级、延迟队列**

### 典型业务场景

| 场景 | Exchange 类型 |
| --- | --- |
| 订单发邮件 | direct |
| 一条消息多人订阅 | fanout |
| 订单状态机（待付/已付/已发） | topic |
| 延迟 30 分钟取消未支付 | delayed_message_exchange 插件 |

```java
@RabbitListener(queues = "order.email")
public void send(OrderMsg msg, Channel ch, @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
    try {
        emailService.send(msg);
        ch.basicAck(tag, false);
    } catch (RetryableException e) {
        ch.basicNack(tag, false, true);   // 重入队列
    } catch (FatalException e) {
        ch.basicNack(tag, false, false);  // 进死信
    }
}
```

## 五、NATS：轻量级选手

NATS 用一句话：**Redis Pub/Sub 的工业版**。单二进制启动，毫秒级延迟，配合 **JetStream** 才有持久化能力。

```bash
# 启动一个开了 JetStream 的 NATS
docker run -p 4222:4222 nats:latest -js
```

```ts
// Bun / Node
import { connect, StringCodec } from "nats";

const nc = await connect({ servers: "nats://localhost:4222" });
const sc = StringCodec();

nc.publish("greet", sc.encode("hello"));

const sub = nc.subscribe("greet");
for await (const m of sub) console.log(sc.decode(m.data));
```

适合 IoT、边缘、内部服务间通信，不适合做"事件中心 + 流计算"那种重场景。

## 六、消息可靠性的三个等级

| 等级 | 含义 | 代价 |
| --- | --- | --- |
| At most once | 最多一次，可能丢 | 性能最高 |
| At least once | 至少一次，可能重复 | 默认选择 |
| Exactly once | 恰好一次 | 难，依赖事务/幂等 |

工业实践：**业务侧做幂等**，不要寄希望于 broker 给你 exactly-once。

```sql
CREATE TABLE processed_msg (
  msg_id VARCHAR(64) PRIMARY KEY,
  processed_at DATETIME
);
```

```java
@Transactional
public void handle(OrderMsg msg) {
    if (msgDao.exists(msg.getId())) return;   // 已处理，幂等返回
    orderService.create(msg);
    msgDao.markProcessed(msg.getId());        // 同一事务标记
}
```

## 七、典型踩坑

1. **消费者慢、消息堆积**：监控 lag，扩 partition / consumer，别等堆爆才看
2. **顺序错乱**：同一业务实体的消息要走**同一个 Key**，落到同一 partition
3. **重启丢消息**：必须开 `acks=all`（Kafka）/ `publisher confirm`（RabbitMQ）
4. **没有死信**：处理失败的消息无限重试拖垮整组消费者，DLQ 必备
5. **Schema 漂移**：生产者升级字段，消费者炸——上 Schema Registry 或至少 JSON Schema 校验
6. **大消息**：单条 > 1MB 的 binary 别走 MQ，把对象存到 OSS/S3，MQ 只发 URL

---

## 八、何时还**不**该上 MQ

- QPS 不到几百、单库扛得住——上 MQ 反而增加排障面
- 强一致、不能丢、必须立刻反馈——同步 RPC 更合适
- 团队没人懂 MQ 运维——挂了比不挂更糟

> 经验法则：**先把单库 + 同步链路压到极限再考虑 MQ**。MQ 不是性能银弹，是架构选择。
