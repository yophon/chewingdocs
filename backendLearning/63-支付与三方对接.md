# 支付与三方对接

42 章讲了分布式事务,这一章是它的"现实战场":**钱的事最不能错**。

支付是最经典的"跨服务、跨公司、跨网络的最终一致性问题"。同时也是和"第三方"打交道的范本——回调、签名、幂等、对账、重试,所有三方对接套路都在这里。

---

## 一、支付的全景图

```
用户                你的服务            第三方支付         银行 / 卡组织
   │                    │                    │                    │
   │  ① 下单 → 选择支付  │                    │                    │
   │ ──────────────────▶│                    │                    │
   │                    │ ② 创建预订单         │                    │
   │                    │ ────────────────▶│                    │
   │                    │ ③ 返回支付参数        │                    │
   │                    │ ◀────────────────│                    │
   │ ④ 跳转支付/调起 SDK │                    │                    │
   │ ◀──────────────────│                    │                    │
   │ ⑤ 在第三方页/客户端付款                   │                    │
   │ ─────────────────────────────────────▶│                    │
   │                    │                    │ ⑥ 真扣款            │
   │                    │                    │ ─────────────────▶│
   │                    │                    │ ⑦ 成功             │
   │                    │                    │ ◀─────────────────│
   │                    │ ⑧ 异步回调通知       │                    │
   │                    │ ◀────────────────│                    │
   │                    │ ⑨ 验签 + 改订单状态  │                    │
   │ ⑩ 同步跳转回结果页  │                    │                    │
   │ ◀──────────────────│                    │                    │
   │                    │ ⑪ 查询/对账/退款      │                    │
   │                    │ ────────────────▶│                    │
```

**核心难点**:

- 步骤 ⑧(异步回调)是"真相之源"——同步返回的页面跳转**绝对不能信**(用户可能骗你说付了)
- 第三方网络抖动 / 重试 → **回调可能重复来 N 次**
- 第三方 / 银行可能"扣款成功但通知失败"——必须主动查询 + 对账兜底

## 三、签名与回调安全

**回调接口是公网暴露的 endpoint,必须验签**——否则任何人调你 `/notify` 就能伪造"支付成功"。

### 签名机制(微信 / 支付宝)

```
请求方:
  1. 把所有参数按 key 排序
  2. 拼成 a=1&b=2&c=3
  3. 加上密钥 → SHA256 / SHA1 / HMAC
  4. 把签名作为 sign 字段一起发

接收方(回调):
  1. 取出 sign
  2. 用同样规则计算预期签名
  3. 对比,不一致直接拒
```

```java
public boolean verify(Map<String, String> params, String secret) {
    String receivedSign = params.remove("sign");
    
    String toSign = params.entrySet().stream()
        .sorted(Map.Entry.comparingByKey())
        .filter(e -> !e.getValue().isEmpty())
        .map(e -> e.getKey() + "=" + e.getValue())
        .collect(Collectors.joining("&"));
    toSign += "&key=" + secret;
    
    String expected = sha256(toSign);
    return MessageDigest.isEqual(
        expected.getBytes(StandardCharsets.UTF_8),
        receivedSign.getBytes(StandardCharsets.UTF_8)        // 时间常数比较防时序攻击
    );
}
```

### 现代支付用 webhook 签名(Stripe 风格)

```
HTTP Header:
  Stripe-Signature: t=1700000000,v1=abc123...

服务端:
  1. 取 timestamp t 和 signature v1
  2. payload = t + "." + raw_request_body
  3. 期望签名 = HMAC-SHA256(payload, webhook_secret)
  4. 时间常数比较 + 检查 t 不能太老(防重放)
```

**关键点**:

- 签名要用 **原始请求体**,不能反序列化后再拼——JSON key 顺序变了签名就错
- HMAC-SHA256 而非纯 SHA(防长度扩展攻击)
- **时间窗口校验**(5 分钟):防止旧回调被重放

> 经验法则:**所有第三方回调都要 IP 白名单 + 签名 + 时间戳校验**——三层防护,任一漏洞都可能被伪造支付成功。

## 四、幂等:回调可能来 100 次

第三方为了"确保你收到",**回调会重复发**直到收到 200 OK。你必须**多次处理同一条等于处理一次**。

```java
@Transactional
public void handleNotify(Notify n) {
    // 1. 验签(略)
    
    // 2. 用第三方 transaction_id 唯一约束去重
    if (notifyLogMapper.existsByThirdPartyTxId(n.txId)) {
        return;        // 已处理过,直接返回 200
    }
    notifyLogMapper.insert(n);    // 唯一索引,重复则抛异常,事务回滚
    
    // 3. 找到本地订单,检查状态(防止重复扣分/重复发货)
    Order order = orderMapper.lockById(n.outTradeNo);
    if (order.status() == OrderStatus.PAID) return;
    if (order.status() != OrderStatus.PENDING) {
        log.warn("非法状态变更");
        return;
    }
    
    // 4. 真正业务:改状态、写流水、发消息
    orderMapper.updateStatus(order.id(), OrderStatus.PAID);
    walletService.add(order.userId(), order.cashback());
    eventPublisher.publish(new OrderPaid(order));
}
```

**幂等三件套**:

1. **唯一索引(notify_log.third_party_tx_id)**——数据库层兜底
2. **状态机校验**(只能 PENDING → PAID,反向 / 跳跃都拒)
3. **事务包整段**——任何一步失败,前面的回滚

## 五、查询接口:回调丢了也要扛得住

**永远不要"只依赖回调"**——网络抖、第三方 bug、自己机器宕机都会导致回调丢。**主动查询是兜底**。

```java
@Scheduled(fixedDelay = 60_000)         // 每分钟
public void pollPending() {
    List<Order> pending = orderMapper.findPendingPaidLongerThan(Duration.ofMinutes(2));
    for (Order o : pending) {
        QueryResult qr = wechatPayClient.queryByOutTradeNo(o.outTradeNo());
        if (qr.isPaid()) {
            handleNotify(qr.toNotify());     // 复用回调逻辑(幂等)
        } else if (qr.isExpired()) {
            orderMapper.markExpired(o.id());
        }
        // 其他状态保持 PENDING,下一轮再查
    }
}
```

> 经验法则:**支付链路 = 回调 + 主动查询 + 对账,三者缺一**。回调是默认通道,主动查询是兜底,对账是事故捕获。

## 六、对账:支付的"安全气囊"

每天凌晨,你和第三方各出一份"昨日交易明细":

```
你的对账文件:
  order_id, third_party_tx_id, amount, paid_at, status

第三方对账文件(从微信/支付宝下载):
  third_party_tx_id, your_order_id, amount, paid_at

比对:
  ① 我有它没有 → 我误标 PAID,可能少收钱
  ② 它有我没有 → 我状态没更新,可能用户重复支付
  ③ 金额不一致 → 严重事故,人工介入
```

```java
@Scheduled(cron = "0 30 1 * * ?")        // 每天 01:30
public void reconcile() {
    LocalDate yesterday = LocalDate.now().minusDays(1);
    File billFile = wechatPayClient.downloadBill(yesterday);
    
    Map<String, Bill> theirs = parse(billFile);
    Map<String, Order> ours = orderMapper.findPaidOn(yesterday).stream()
        .collect(Collectors.toMap(Order::thirdPartyTxId, identity()));
    
    Set<String> all = new HashSet<>();
    all.addAll(theirs.keySet()); all.addAll(ours.keySet());
    
    for (String tx : all) {
        Bill b = theirs.get(tx); Order o = ours.get(tx);
        if (b == null) reconcileQueue.add(tx, "我有他无");
        else if (o == null) reconcileQueue.add(tx, "他有我无");
        else if (!b.amount.equals(o.amount)) reconcileQueue.add(tx, "金额不一致");
    }
    
    if (!reconcileQueue.isEmpty()) alertService.send("对账异常 " + reconcileQueue.size() + " 条");
}
```

> 经验法则:**对账每天必跑 + 异常立刻告警 + 人工 review**——这是支付系统的最后防线,出大事故全靠它先发现。

## 七、退款:更隐蔽的坑

退款的回调和支付一样,**幂等 + 状态机 + 对账** 全套照搬。但有几个特殊点:

### 1. 部分退款

```
原订单 100 元
退 30 元 → 剩余可退 70
再退 50 元 → 剩余可退 20
再退 25 元 → 报错(超限)
```

```java
@Transactional
public Refund createRefund(Long orderId, BigDecimal amount, String reason) {
    Order o = orderMapper.lockById(orderId);
    BigDecimal refunded = refundMapper.sumRefundedByOrderId(orderId);
    if (o.amount().subtract(refunded).compareTo(amount) < 0) {
        throw new BizException("超过可退金额");
    }
    Refund r = new Refund(orderId, amount, RefundStatus.PROCESSING, reason);
    refundMapper.insert(r);
    
    // 调第三方
    var resp = wechatPayClient.refund(o, r);
    return r;
}
```

### 2. 退款异步化

退款大多是异步的——**业务接口立刻返回"退款申请已受理"**,真实退款几小时到几天才到账。回调到了再标 SUCCESS。

### 3. 退款金额回写钱包 / 优惠券

退完钱后,**优惠券要不要恢复? 积分要不要扣回?** 这些"逆向操作"业务复杂度比正向支付高。

> 经验法则:**退款链路当独立子系统设计**——别和支付混一个 service。退款的状态机、对账、幂等都比支付复杂。

## 八、异步通知 vs 主动询问的取舍

```
异步通知:
  ✅ 实时性好(毫秒级)
  ❌ 网络/对方 bug 时丢失
  ❌ 你的服务挂了就没了

主动询问(轮询):
  ✅ 自己控制,挂了不丢
  ❌ 实时性差(分钟级)
  ❌ 浪费 QPS

最优策略:两者结合
  ① 通知到 → 立刻处理
  ② 5 分钟后没通知 → 主动查
  ③ 24 小时没结果 → 自动取消订单
```

## 十一、PCI DSS 与卡数据

涉及"用户卡号"的业务,必须遵守 **PCI DSS**(支付卡行业数据安全标准)。

**核心红线**:

```
绝对不能:
  - 卡号、CVV、有效期 入你自己的数据库
  - 卡号过你自己的服务器(哪怕日志)
  - 截图 / 抓包里出现卡号

正确姿势:
  - 用第三方 Hosted Field / Stripe Elements / Adyen Drop-in
  - 卡号直接从浏览器到第三方,你只拿一个 token
  - 后续支付都用 token,你永远不见卡号
```

> 经验法则:**99% 的项目都该走 Hosted Field 模式**——除非你是金融牌照公司,**绝不要碰原始卡号**。一旦碰了,PCI DSS 审计上百万、合规人员拉一队。

## 十二、三方对接的通用套路

这一章总结的能力**不止用于支付**——所有三方对接都是这套:

| 模式 | 解决 |
| --- | --- |
| **签名(HMAC / 非对称)** | 防伪造 |
| **时间戳 + nonce** | 防重放 |
| **IP 白名单** | 防外部直调 |
| **幂等 + 唯一索引** | 防重复处理 |
| **回调 + 主动查询** | 防丢失 |
| **对账批处理** | 防数据漂移 |
| **重试 + 退避 + 死信队列** | 防瞬时故障 |
| **熔断 / 降级** | 防三方不可用拖死自己 |
| **沙箱环境** | 联调不动真钱 |
| **监控:回调延迟、对账差异** | 早发现 |

**短信、推送、风控、地图、天气、AI API** —— 都用这套。

## 十三、常见踩坑

1. **同步页面跳转就标已支付**:用户改 URL 直接 PAID,白送钱
2. **回调不验签**:任何人发请求都能骗你"成功"
3. **不做幂等**:回调重发导致重复发货
4. **状态机不严格**:CANCELLED 的订单被回调改成 PAID
5. **超时没处理**:用户付了款 30 分钟没收到 → 客服爆炸
6. **回调用同步处理**:第三方限制 5 秒,你处理 10 秒,被认为失败一直重发
7. **不做对账**:某天发现少收了 1 万块
8. **退款金额累加错**:超额退款给商户
9. **存原始卡号**:PCI 不合规,被处罚
10. **沙箱测过了直接上**:真实环境的边界 case 没覆盖
11. **错误码不映射**:第三方返 SUB_NO_AUTH,你直接抛给前端
12. **网络异常当失败处理**:实际付款成功,你 cancel 又导致用户重复付
13. **签名算法用 MD5**:弱碰撞,被攻
14. **HMAC 比较用 String.equals**:时序攻击
15. **没监控回调延迟**:某天回调全堆积没人发现

## 小结

支付是后端工程师**最容易出大事故、也最练人**的领域——一行代码错可能直接亏几十万到几百万。

记住三件事:

1. **异步通知不是真相,自己查 + 对账才是**——回调只是优化体验,不是依赖
2. **幂等 + 状态机 + 唯一索引** 三件套防重复,缺一就出生产事故
3. **不要碰原始卡号**——PCI 红线,Hosted Field 是必修
