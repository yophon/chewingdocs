# 日志系统选型:ELK vs Loki vs ClickHouse / 结构化日志 / 采样

讲日志系统的文章 90% 都从「ELK 三件套架构图」开始——这是错的。**日志系统选型不是"挑哪个工具",是"想清楚你的日志是写多还是查多、是要全文搜索还是只看 trace_id、月预算给得起多少"**。同一份业务日志,放 ELK 一年要 30 万,放 Loki 5 万,放 ClickHouse 8 万但能 SQL 分析——**差的不是技术,是你回答这三个问题的答案**。08 篇讲了指标的长存储,这一篇接着讲日志——**和指标比,日志的"写多查少 + 数据量大 100 倍"这个本质差异,决定了它的所有选型逻辑都得反过来**。

> 一句话先记住:**日志系统的核心矛盾是"写入即免费,查询即昂贵"——这句话反过来读才是真理**:**写入永远不免费,只是成本转嫁到了存储 / 索引 / 网络上**。ELK 把写入成本转成了"索引时算 + 存储多 3x";Loki 把成本转成了"查询时全文扫";ClickHouse 把成本转成了"提前结构化 + 列存压缩"。**你选哪个,本质上是在选"我愿意把成本付在哪个环节"**——没有"全都便宜"的方案,**所有声称自己"全都便宜"的工具都是营销话术**。

## 一、ELK 时代的五个痛点

ELK(Elasticsearch + Logstash + Kibana)从 2014 年开始几乎是日志系统的事实标准。**到 2020 年之后,中型团队普遍用得很难受**——不是 ELK 变烂了,是数据规模涨了 10 倍但预算没涨。**五个痛点几乎每个用过 ELK 的团队都中过**:

### 1.1 写放大:存 1GB 日志,占用 3-5GB 磁盘

Elasticsearch 是 Lucene 的封装,**对每一个字段都建倒排索引**——这是"全文搜索快"的代价。

```
   原始日志 1 行(JSON):200 字节
   { "ts": "...", "level": "INFO", "msg": "...", "user_id": 9527 }
                              ↓ Elasticsearch 处理
   ┌─────────────────────────────────────────────────────┐
   │ _source(原始 JSON)         200 字节                │
   │ _all 字段(默认,可关)      200 字节                │
   │ keyword 倒排索引            150 字节                │
   │ text 分词倒排索引           300 字节(可能更多)    │
   │ doc_values 列存             100 字节                │
   │ ──────────────────────                              │
   │ 总占用                      ≈ 800-950 字节          │
   │ 副本(默认 1 副本)         × 2 = 1600-1900 字节    │
   │ 放大倍率                    ≈ 4-5x                  │
   └─────────────────────────────────────────────────────┘
```

**结果**:1 TB 原始日志,ES 集群要 4-5 TB 存储。**这是 ELK 账单爆炸的根因**。

### 1.2 字段 mapping 冲突:一个错误日志能让整个索引写不进

ES 的 mapping 是"第一条文档定的类型,后续不能变"。**经典坑场景**:

```
   Day 1:第一条日志写 user_id 字段
   { "user_id": 9527 }  → ES 推断 user_id 是 long

   Day 2:某个服务 bug,user_id 写成了字符串
   { "user_id": "anonymous" }
   → ES:mapping conflict! 整个 index 当天的写入全卡住
```

**修复**:全部加 `dynamic: strict` 或者 schema 显式声明——**但中型团队有上百个服务,统一 schema 难于上青天**。

### 1.3 Index split brain:分片配置错了集群就崩

ES 把数据分成 shard,每个 shard 是独立 Lucene 索引。**分片数是建索引时定死的,不能改**。

```
   错误一:分片太少
   单 shard > 50GB → 查询慢、merge 慢、reindex 难
   
   错误二:分片太多  
   1000 个 index × 10 shard = 10000 shard
   → master 节点 cluster state 几百 MB,gossip 协议拖死整个集群
```

中型团队的常见姿势是**每天一个 index,每个 5 shard**——3 年下来 5000+ shard,**ES master 节点的 cluster state 同步成了瓶颈**。

### 1.4 查询慢:大时间范围 + 高基数 group by 就是定时炸弹

```
   GET /logs-*/_search
   {
     "query": { "range": { "@timestamp": { "gte": "now-7d" } } },
     "aggs": { "by_user": { "terms": { "field": "user_id", "size": 10000 } } }
   }
```

这个查询要做的事:**扫 7 天的所有索引 × 所有 shard,对 user_id 做高基数聚合**——10 个节点的 ES 集群跑 30 秒不算夸张,**严重时直接 OOM 全集群**。

### 1.5 维护重:ES 升级是一场战役

ES 大版本升级(5→6→7→8)每次都有 breaking change:**type 移除、_all 移除、协议变 HTTPS 强制、安全模型变化**。**3 年没升级的 ES 集群,升级要专人投入 1-2 个月**——这是「事实上的技术债」,只是没人愿意承认。

**ELK 不是技术不行,是**——「**用一套全文搜索引擎来当日志库,本来就是"工具能做但不适合"**」。**ES 是搜索引擎不是日志系统**,这一点 Loki 的作者(Grafana Labs)在 2018 年就看穿了。

## 二、Loki:把成本反过来分

Loki 是 Grafana Labs 2018 年开源,**它的设计哲学只有一句话**:

> **「日志不需要全文索引,只需要按 label 找到那一组日志,然后内容压缩到对象存储里。要全文搜索?查询时再扫」**

这句话是对 ELK 三层假设的**完全反转**:

```
   ELK 的假设:                       Loki 的假设:
   ──────────────────                 ────────────────────
   写入慢一点没关系                   写入要快,索引要小
   存储贵一点没关系                   存储用对象存储,极便宜
   查询任意字段都要快                  查询主要靠 label,内容扫描可以慢
```

**架构图**:

```
   日志生产者
   ┌──────────────────┐
   │ Promtail/Vector  │ ── 给每行加 label(app, env, pod)
   └────────┬─────────┘
            │ HTTP push
            ▼
   ┌──────────────────────────────────────┐
   │  Distributor:按 label 哈希到 Ingester │
   └────────┬─────────────────────────────┘
            ▼
   ┌──────────────────────────────────────┐
   │  Ingester:内存缓冲 + 压缩 chunk      │
   │  ─────────────────────────────       │
   │   label → 索引(几 KB)               │
   │   chunk → 对象存储(压缩后 1-2 字节/行) │
   └────────┬─────────────────────────────┘
            ▼
       S3 / OSS / GCS

   查询路径:
   ─────────────
   LogQL ──▶ Querier ──▶ 用 label 选 chunks ──▶ 流式扫描内容
   {app="api"} |= "error" → 先按 label 拿到 chunk → 再 grep "error"
```

**核心数据**:Loki 的索引只有 ELK 的 1/100,**存储成本(对象存储 + 压缩)是 ELK 的 1/10**。

### 2.1 Loki 的"成本反过来"具体省了什么

```
   1 TB 原始日志(7 天保留):
   ─────────────────────────────────────────
   ELK:    
     - 索引 + 副本 + 全文倒排    ≈ 4-5 TB SSD 
     - 计算节点(高内存高 CPU)   3-5 节点
     - 月成本                    ≈ 1.5-3 万元/TB
   
   Loki:
     - label 索引                ≈ 10-20 GB(本地 SSD)
     - 内容 chunk(对象存储)    ≈ 200-300 GB(已压缩)
     - 计算节点(轻量)          1-2 节点 + 几个 worker
     - 月成本                    ≈ 1500-3000 元/TB
   
   差距:10 倍
```

### 2.2 Loki 的代价:LogQL 不是 SQL

Loki 的查询语言 LogQL 像 PromQL 加了管道:

```
   {app="payment", env="prod"} |= "error" | json | line_format "{{.user_id}} {{.msg}}"
   ────────────────────────── ─────── ──── ────────────────────────────
        label 选 stream        过滤    解析       格式化
```

**做不了的事**:

- 没法做 `GROUP BY user_id` 的统计 ——LogQL 的聚合是基于时间序列的(`rate / sum_over_time`),**不是基于内容的 group by**
- 跨 label 的 join 几乎跑不动
- 全文搜索性能远不如 ES——Loki 的 `|= "error"` 是流式扫描,**7 天的高频日志查一次几十秒**

**结论**:**如果团队需要做"日志分析"(用户行为统计 / 异常聚合 / 自定义维度切分),Loki 不是好选择**——它是"日志归档 + trace_id 反查"工具。

### 2.3 Loki 该用 / 不该用的边界

```
   Loki 适合:
   ✓ 日志主要用途是"出问题时按 trace_id / pod / app 反查"
   ✓ 已经在用 Grafana,Loki 是数据源天然集成
   ✓ 团队不做复杂的日志分析
   ✓ 预算紧张,愿意接受查询慢一点
   ✓ 已经有 S3/OSS,对象存储是熟手

   Loki 不适合:
   ✗ 团队 BI/数据分析师天天 group by 业务字段
   ✗ 需要"全文搜索 < 1s"的体验
   ✗ 需要跨多日志源做关联查询
   ✗ 没有 Grafana,用 Kibana 类的可视化
```

## 三、ClickHouse 做日志:第三条路

最近 2-3 年,**国内中型团队上 ClickHouse 做日志的越来越多**。这不是赶时髦——**ClickHouse 在「既要日志又要分析」的场景下,性价比是三者最高的**。

### 3.1 为什么 CH 适合日志

ClickHouse 是列式数据库,**它的核心三个特性正好打在日志的痛点上**:

```
   特性 1:列存 + 高压缩比
   ────────────────────────────────────────────
   日志的字段大量重复(level, app, env 都是几种值)
   列存压缩比 5-20x,比 ES 的 4-5x 行存高一个量级
   
   特性 2:SQL 查询,真正的 GROUP BY
   ────────────────────────────────────────────
   SELECT user_id, count() 
   FROM logs WHERE level='ERROR' AND ts > now() - INTERVAL 1 DAY
   GROUP BY user_id 
   ORDER BY count() DESC LIMIT 100
   → ClickHouse 几秒搞定;ES 全文 + 高基数聚合可能要 30s+
   
   特性 3:向量化执行,扫表也快
   ────────────────────────────────────────────
   就算没用索引,扫 1TB 数据找 "error" 关键字,CH 比 ES 快 2-3x
   底层用 SIMD 加速字符串比对
```

### 3.2 CH 日志表的典型 schema

```sql
CREATE TABLE logs (
  ts          DateTime64(3) CODEC(Delta, ZSTD),
  level       LowCardinality(String),
  app         LowCardinality(String),
  env         LowCardinality(String),
  pod         LowCardinality(String),
  trace_id    String,
  user_id     UInt64,
  msg         String CODEC(ZSTD(3)),
  attrs       Map(LowCardinality(String), String),
  
  INDEX idx_trace trace_id TYPE bloom_filter GRANULARITY 1,
  INDEX idx_user  user_id  TYPE minmax       GRANULARITY 4
)
ENGINE = MergeTree
PARTITION BY toYYYYMMDD(ts)
ORDER BY (app, env, ts)
TTL ts + INTERVAL 30 DAY;
```

**5 个关键设计**:

1. **`LowCardinality(String)`** 给 level / app / env / pod——这些字段只有几种到几百种值,**LowCardinality 用字典编码,存储压到 1 字节**
2. **`Map(LowCardinality(String), String)` 存动态字段**——比每个字段一列灵活,代价是查询稍慢
3. **`ORDER BY (app, env, ts)`** 决定主键和数据排序——**最常查的字段放前面**,跳数索引才有效
4. **`bloom_filter` 给 trace_id** ——trace_id 是高基数 UUID,bloom 在「找具体一个 trace」时极快
5. **`TTL ... + INTERVAL 30 DAY`** ——自动过期,**不用写 cron job 删数据**

### 3.3 CH 做日志的现实代价

- **写入复杂**:CH 偏好"批量大写",**1 条 / 次写法 CH 会崩**——必须有缓冲层(Kafka / Vector aggregator)攒批
- **没有"全文搜索"语义**:`LIKE '%error%'` 慢,要换 `tokenbf_v1` 索引或 `hasToken()` 函数,**没有 ES 那种"语义分词"**
- **Schema 升级痛**:加列要 ALTER TABLE,大表 ALTER 不能阻塞写入,**有讲究**
- **单 partition 写入限制**:`PARTITION BY toYYYYMMDD` 后,**单天 partition 一旦超过 1 TB 数据,合并速度跟不上**

**结论**:**CH 做日志,适合"日志量 100-500 GB/天 + 真有日志分析需求"的团队**——以下小了不值得,以上需要专门容量规划。

## 四、三者选型矩阵

```
   ┌─────────────────────────────────────────────────────────────┐
   │  团队规模:< 10 服务,日志量 < 100 GB/天                    │
   │  → 别上日志聚合平台                                          │
   │  方案:K8s 直接 kubectl logs / docker logs                    │
   │       问题机器:journalctl + grep / awk                       │
   │  理由:日志聚合的运维成本 >>  解决的痛点                     │
   ├─────────────────────────────────────────────────────────────┤
   │  10-50 服务,100-500 GB/天,主要用来"反查"                    │
   │  → 选 Loki                                                  │
   │  方案:Promtail/Vector → Loki + S3 → Grafana                 │
   │  理由:成本低,trace_id 反查够用,Grafana 集成无缝            │
   ├─────────────────────────────────────────────────────────────┤
   │  50-200 服务,500 GB-2 TB/天,既要反查又要业务分析            │
   │  → 选 ClickHouse                                            │
   │  方案:Vector → Kafka → CH;Superset/Grafana 查询             │
   │  理由:列存压缩 + SQL 分析,长期最划算                       │
   ├─────────────────────────────────────────────────────────────┤
   │  团队已经深度依赖 ES / Kibana,有现成栈                       │
   │  → 留着 ELK,但加 hot/warm/cold 分层 + 强 TTL                 │
   │  理由:迁移成本巨大,优化现有比换新更值                       │
   ├─────────────────────────────────────────────────────────────┤
   │  > 200 服务,> 2 TB/天,SaaS 化场景                          │
   │  → 混合栈:Loki 反查 + CH 分析 + ES 给特定全文搜索             │
   │  理由:单一栈撑不住,每个工具做它擅长的                       │
   └─────────────────────────────────────────────────────────────┘
```

**90% 的中型团队最优解是 Loki**——这一点和指标长存储选 VM 是一样的逻辑:**先选最便宜 / 最简单的,真撞墙再换**。

### 4.1 决策树:三个问题决定选哪个

```
   Q1: 你团队日志量 / 天?
   ├── < 100 GB    → grep 就够,别折腾
   ├── 100-500 GB  → Loki(便宜 + 简单)
   └── > 500 GB    → 继续问 Q2
   
   Q2: 你的核心查询模式是什么?
   ├── 反查(trace_id / pod / app)  → Loki 仍然够
   ├── 业务分析(group by / 趋势) → ClickHouse
   └── 全文 + 复杂搜索              → 还是 ES(或 hybrid)
   
   Q3: 团队有没有专人维护这套?
   ├── 没有                  → Loki(运维最简单)
   ├── 有兼职 SRE            → CH(中等运维难度)
   └── 有专职数据 / 平台团队  → 三者皆可
```

## 五、结构化日志:换什么后端都救不了你的烂日志

**这一段比所有"选哪个后端"都重要**——**90% 团队的日志成本爆炸,根因不是后端选错,是日志本身写得烂**。

### 5.1 结构化日志的硬要求

**不结构化的日志(反例)**:

```
2024-01-15 14:32:01 INFO User 9527 logged in from 1.2.3.4 with device iPhone15 traceId=abc-123
```

**结构化的日志(正例)**:

```json
{
  "ts": "2024-01-15T14:32:01.123Z",
  "level": "INFO",
  "msg": "user logged in",
  "user_id": 9527,
  "ip": "1.2.3.4",
  "device": "iPhone15",
  "trace_id": "abc-123",
  "app": "auth-service",
  "pod": "auth-7f8d-x9k2"
}
```

**为什么必须 JSON**:

- 后端无需正则解析,**直接当结构化字段索引 / 查询**
- 字段查询语义清晰:`user_id=9527` 不会误命中其他字段
- 加字段不破坏旧日志的兼容性
- **采集端解析 CPU 成本接近零**(对比正则解析提升 5-10x)

### 5.2 必须有的 6 个字段

无论用哪个后端,**这 6 个字段每条日志都要有**:

| 字段 | 必要性 | 说明 |
| --- | --- | --- |
| `ts` | 必须 | ISO8601 带毫秒和时区,**别用本地时间** |
| `level` | 必须 | INFO / WARN / ERROR / DEBUG,**枚举** |
| `app` / `service` | 必须 | 哪个服务出的日志 |
| `trace_id` | 强烈推荐 | 链路追踪 id,**和 OTel 联动** |
| `msg` | 必须 | 人读的描述,**不放参数** |
| `pod` / `host` | 必须 | 哪台机器 / 哪个 pod |

**反对的写法**:

```python
# 烂日志(我天天看见)
logger.info(f"User {user_id} did action {action} at {time}, result: {json.dumps(result)}")

# 后端拿到的 msg 字段:
# "User 9527 did action checkout at 2024-01-15 14:32:01, result: {...}"
# → 想统计每个 action 的次数?要 regex 抓"did action ...",太脆弱

# 正确写法
logger.info("user action",
            user_id=user_id, action=action, result=result, trace_id=trace_id)
# msg = "user action",user_id / action / result 都是独立字段
```

### 5.3 不要塞 stack trace 进 msg

**最常见的踩坑写法**:

```python
try:
    do_something()
except Exception as e:
    logger.error(f"Failed: {e}\n{traceback.format_exc()}")
    # msg 字段变成几 KB 的 stack trace
```

**问题**:

- 单条日志膨胀 10-100 倍,**写入带宽和存储都爆炸**
- ES 的 text 字段长度有限,**超过会截断,丢上下文**
- Loki 多行处理麻烦,**stack trace 容易被切碎**

**正确姿势**:

```python
logger.error("operation failed",
             error=str(e),
             error_type=type(e).__name__,
             stack_trace=traceback.format_exc(),  # 独立字段
             trace_id=trace_id)
# msg 仍然是 "operation failed",stack_trace 是单独字段
# 查询时可以选择只看 msg 不看 stack
```

更好的做法:**把 stack trace 关联到 trace_id 上,日志只记 `error_type` + `trace_id`,完整 stack 到 trace 系统(Tempo / Jaeger)看**。

## 六、采样:把"无用日志"在源头就丢掉

**1000 行 INFO 日志里,99% 是"我处理了一个请求"这种垃圾信息**。**全量保留 = 给云厂商捐钱**。

### 6.1 三种采样策略

```
   ┌───────────────────────────────────────────────────────┐
   │  策略 1:头部采样(Head Sampling)                       │
   │  ─────────────────────────────────                      │
   │  入口决策:看不看这个请求,在最早处决定                  │
   │  实现:Trace ID 哈希 mod 100 < N → 留;否则丢            │
   │  优点:简单、CPU 低、上下游一致                          │
   │  缺点:好坏请求一样的命运,可能丢掉关键错误               │
   └───────────────────────────────────────────────────────┘
   
   ┌───────────────────────────────────────────────────────┐
   │  策略 2:尾部采样(Tail Sampling)                       │
   │  ─────────────────────────────────                      │
   │  请求结束后决策:错的全留,慢的全留,正常的低概率留        │
   │  实现:OTel Collector 的 tail_sampling processor          │
   │  优点:关键事件 100% 不丢                                │
   │  缺点:要短暂缓冲整个 trace,Collector 内存压力          │
   └───────────────────────────────────────────────────────┘
   
   ┌───────────────────────────────────────────────────────┐
   │  策略 3:分级采样(Tiered Sampling)                     │
   │  ─────────────────────────────────                      │
   │  按业务规则分等级,不同级别不同采样率                    │
   │  ERROR/WARN  → 100% 留                                  │
   │  支付 / 登录类 INFO → 100% 留                            │
   │  普通 200 OK INFO   → 1% 留                              │
   │  健康检查 / 静态资源 → 0.1% 或不留                       │
   └───────────────────────────────────────────────────────┘
```

### 6.2 推荐的"错误全采 + 正常长尾"模式

```python
# 应用层采样(伪代码)
def should_log(level, status_code, latency_ms):
    # ERROR / WARN 100% 留
    if level in ('ERROR', 'FATAL', 'WARN'):
        return True
    
    # 慢请求 100% 留(可能在排查问题)
    if latency_ms > 1000:
        return True
    
    # 5xx 100% 留
    if status_code >= 500:
        return True
    
    # 4xx 10% 留
    if status_code >= 400:
        return random.random() < 0.1
    
    # 健康检查 / 静态资源 不留
    if path.startswith('/healthz') or path.startswith('/static'):
        return False
    
    # 普通 200 OK,1% 留
    return random.random() < 0.01
```

**预期效果**(实测数据):

```
   原始日志量    1000 GB/天
   ───────────────────────────
   应用采样后    ≈ 50-150 GB/天
   存储 7 天     ≈ 350 GB-1 TB
   存储 30 天    ≈ 1.5-4.5 TB
```

**省了 80-90%,关键日志一条不丢**——这是采样的精髓。

### 6.3 采样的隐藏代价:统计偏差

**采样之后,直接 count 出来的数字不准了**——这一点很多团队没意识到。

```
   采样前:        100,000 次登录
   1% 采样后:    1,000 次登录(日志里看到的)
   
   如果按日志统计:登录次数 = 1,000(错!)
   正确算法:    1,000 / 0.01 = 100,000
```

**结论**:**日志用于精确计数,必须用全量日志 / metric**——采样日志只能看趋势 / 异常,**不能算账(订单数 / 转化率)**。**用 metric counter 算业务指标,用日志做事后排查**——这是正确的姿势。

## 七、一个真实的踩坑:Loki 的 cardinality 爆炸

讲一个我亲眼看见的事故——**某团队上 Loki 之后 2 周,集群崩了 3 次,根因是 label 配错**。

### 7.1 事故重现

团队在 Promtail 里这么配:

```yaml
scrape_configs:
- job_name: app-logs
  static_configs:
  - targets: [localhost]
    labels:
      job: app
      __path__: /var/log/app/*.log
  pipeline_stages:
  - json:
      expressions:
        tenant_id: tenant_id          # 业务字段提为 label!
        user_id: user_id              # 业务字段提为 label!
        request_id: request_id        # 业务字段提为 label!
  - labels:
      tenant_id:
      user_id:
      request_id:
```

**直觉**:把这些字段当 label,**查询时方便**。

**现实**:Loki 的索引粒度是**唯一 label 组合 = 一个 stream**。

```
   tenant_id: 100 个
   user_id:   100 万个
   request_id: 100 万 / 小时
   ────────────────────────────────
   唯一组合:100 × 1M × 1M = 10^14
   每个 stream 都要建索引、占内存!
   → Ingester 内存 1 小时涨到 50 GB,OOM
```

### 7.2 修复

**label 的硬规则**:**label 是用来"切分日志流"的,不是"存数据"的**。

```yaml
# 正确做法
pipeline_stages:
- json:
    expressions:
      tenant_id: tenant_id
      user_id: user_id
      request_id: request_id
# 不要把它们提到 label!
# 让它们留在日志 line 里,查询时再 grep / regex
```

**label 该用什么**:

| 字段 | 该不该当 label |
| --- | --- |
| `app` / `service` | **是**(几十种值) |
| `env` (prod/staging) | **是**(几种值) |
| `pod` | **是**(几百种值,但可控) |
| `level` (INFO/ERROR) | **是**(< 10 种值) |
| `tenant_id` (高基数) | **绝对不是** |
| `user_id` | **绝对不是** |
| `trace_id` | **绝对不是** |
| `path` (URL 路径) | **看情况**——如果是固定路由模板可以,如果是带 query 参数的就不行 |

**经验法则**:**一个 label 的唯一值不要超过 1000**——超过就是 cardinality 炸药。

### 7.3 Loki 团队官方的 "10 stream rule"

Grafana Labs 自己的建议:**单个应用的日志,理想 stream 数是 10-100,极限不能超过 1000**。

```
   计算公式:
   streams = 笛卡尔积(所有 label 的唯一值)
   = app_count × env_count × pod_count × ...
   
   100 服务 × 3 环境 × 10 pod = 3000 stream ← 临界
   再加 instance / version / region → 上万 ← 危险
```

## 八、结构化日志 + 三个后端的差异

同一段结构化日志,**在 ELK / Loki / CH 里查询长什么样**——这能让你直观感受三者的差异。

### 8.1 ELK 的 Kibana KQL

```
# 找某用户最近一小时的 ERROR
level: "ERROR" AND user_id: 9527 AND @timestamp > "now-1h"
```

**特点**:语法接近自然语言,**对每个字段都是索引查询,毫秒级返回**——这是 ELK 的强项。

### 8.2 Loki 的 LogQL

```
# 找某用户最近一小时的 ERROR
{app="auth-service", level="ERROR"} 
  | json 
  | user_id="9527"
```

**特点**:`app` / `level` 是 label(走索引,快);`user_id` 是 JSON 字段(查询时解析,慢)——**所以高基数字段不能当 label,只能查询时扫**。

### 8.3 ClickHouse 的 SQL

```sql
SELECT ts, msg, attrs
FROM logs
WHERE app = 'auth-service'
  AND level = 'ERROR'  
  AND user_id = 9527
  AND ts > now() - INTERVAL 1 HOUR
ORDER BY ts DESC
LIMIT 100;
```

**特点**:`user_id` 是 UInt64 一等公民,**列存 + minmax 索引秒级返回**。**还能 GROUP BY user_id 统计 / 关联其他表**——这是 ELK / Loki 都做不到的。

### 8.4 综合对比

| 场景 | ELK | Loki | ClickHouse |
| --- | --- | --- | --- |
| trace_id 反查 | 极快 | 快(label 过滤后扫) | 快(bloom filter) |
| 时间范围内 grep 关键字 | 快 | 慢(线性扫描) | 中(向量化扫描) |
| group by 业务字段 | 中(高基数慢) | **不支持** | **极快** |
| 跨日志源 join | 慢 | 不支持 | 快 |
| 全文模糊搜索 | **最强** | 弱 | 中 |
| 写入压力承受 | 中(限制大) | 极强 | 强(批量) |
| 月成本(1TB/天) | 3 万 | 3000 | 1 万 |

## 九、何时不该上日志聚合

**这部分是给"团队人少又被推着上日志平台"的兄弟看的**——以下情况,**别折腾**:

### 9.1 < 10 服务 + < 100 GB/天

**直接 `kubectl logs` + `journalctl` 够用了**。一个 1 TB 的 ELK 集群运维成本(机器 + 人力)一年至少 10 万,**你的整个团队产生的日志价值都不到这个数**。

**进阶**:用 `stern` 或 `kubectl-tail` 工具一次 follow 多个 pod 日志——**比堆架构便宜 100 倍**。

### 9.2 日志查询频率 < 每天一次

如果团队整周才查一次日志(就是出问题时),**就用对象存储 + grep 脚本**:

```
   每天把日志压缩传 S3:
   ───────────────────────
   gzip /var/log/app/*.log → s3://team-logs/2024/01/15/app.log.gz
   
   要查的时候:
   aws s3 cp s3://... - | gzip -d | grep "user_id=9527"
   
   月成本:1 TB × 0.12 元 = 120 元(对比 Loki 3000+ 元)
```

**何时升级**:**查询频率 > 每天 10 次 + 跨多个 app 关联**——这才到了搭聚合平台的临界点。

### 9.3 团队没人懂可观测性

**日志聚合平台不是"装上就完事"**——要有人懂 label 设计、采样策略、TTL 配置、告警联动。**没这个人的团队,装了 Loki / ELK 半年就变成"日志全保留但没人查"的状态**——**就是给云厂商捐钱**。

**例外**:Loki + Grafana 这套,如果团队已经在用 Grafana,**轻量上 Loki 是低成本切入**——不要因为这条放弃尝试,**但前提是真的用 Grafana**。

### 9.4 日志的真实价值低于"上聚合平台" 的人力成本

我见过一个团队,5 个开发 + 10 个微服务,**强行上了 ELK 三节点集群**。
- 月成本机器费 6000 元
- 半个 SRE 时间(月薪折合 1.5 万)
- 实际日志查询频率:**每周 1-2 次**

**复盘**:**这套日志系统平均每次查询的成本是 4000 元**——比每次让运维登服务器手工 grep 贵 100 倍。

**结论**:**日志聚合是规模 / 复杂度的函数,不是"先进性"的标志**。

## 十、回到一开始的问题

**日志系统选型不是技术选型,是经济选型**:

- **ELK 把成本付在写入和存储**(全文索引贵)
- **Loki 把成本付在查询**(用便宜对象存储换查询慢)
- **ClickHouse 把成本付在 schema 设计**(结构化前置换列存压缩)

**没有"全都好"的方案,只有"匹配你团队规模 + 查询模式 + 预算"的方案**。

**这一篇要给你留下的不是"哪个工具最好",而是 4 个判断维度**:

1. **量级**:< 100 GB/天 不上、100-500 GB Loki、500GB+ 看分析需求
2. **查询模式**:反查 vs 分析 vs 全文,完全不同的工具
3. **结构化**:**任何工具的前提是日志本身结构化**——烂日志放哪都贵
4. **采样**:**比换工具节省 10 倍成本的事**,但绝大多数团队没做

> 经验法则:**先把日志结构化 + 采样做好,再选后端**——这两件事不做,装最贵的 Splunk 也救不了你;这两件事做好,Loki 都能给你 10 倍的快感。
