可观测性是什么:监控 ≠ 可观测性 / 三件套 + Profile / Cardinality 陷阱 ​

讲可观测性最大的混乱来自:业界把"Observability"用成了"高级版的 Monitoring"——卖工具的厂商懒得讲两者的真正区别,买工具的客户也懒得弄清楚。结果就是大家以为"装了 Prometheus + Loki + Jaeger 三件套就是上了 Observability",实际上很多团队装了三件套,出事的时候还是只会"ssh 上去 tail -f log"。可观测性不是"装了什么工具",是"系统出新问题时,你能不能现场问出来"——这个能力跟工具数量没有直接关系,跟你怎么"埋点 + 关联 + 查询"才有关系。这一篇就讲清楚:Monitoring 和 Observability 到底差在哪、三件套各自的边界、为什么 user_id 当 label 让 Prometheus 挂掉、可观测性的 3 个心智。

一句话先记住:Monitoring 回答"已知的未知"(known unknowns)——我提前知道要问什么,把那个问题做成 dashboard 和告警;Observability 回答"未知的未知"(unknown unknowns)——系统出新问题时,我能现场问出来。前者是"探照灯",照亮你提前指好的地方;后者是"你能在黑暗里摸索的能力"。一个团队"装了多少工具"决定不了 Observability,"出事时能不能 5 分钟定位根因"才决定。这一篇是这个系列的第二层(05-12 共 8 篇)的"入门票"——看完才知道为什么后面要花 8 篇讲 Metrics / Logs / Traces / Profile。

一、问题场景:有"监控"没"可观测性"的团队长什么样 ​

1.1 一个真实的反面教材 ​

团队 12 人,微服务 80 个,Prometheus + Grafana + ELK 都装了
看着挺像"现代化"了,但实际:

凌晨 02:30,P0 告警:订单服务 P99 飙到 8 秒
值班工程师老李打开 Grafana:
   - 翻 30 个订单服务相关的 dashboard,只有 P99 是红的
   - 没有 latency 的"按下游服务分组"视图
   - 没有 trace 数据(链路追踪没接)
   - 看了 30 分钟,只能确认"是慢了,不知道为什么"

老李切到 Kibana:
   - 输入 "service:order AND status:slow"
   - 7,000 条日志,翻不完
   - 也不知道这 7000 条里哪一条是"慢"的代表
   - 想筛某个特定慢请求的 trace,但日志里没 trace_id

老李 ssh 到一台机器:
   - top:CPU 50%,看着不饱和
   - 但实际 GC 时间高 —— Prometheus 没采 JVM 指标
   - 实际下游 RDS 慢 —— 但没人监控 RDS 的 P99 查询时间

老李 03:30 之后开始猜:
   - 重启服务,P99 短暂下降,5 分钟后回升
   - 改并发参数,无效
   - 04:15 凭直觉判断"可能是某个下游 RPC 慢",降级了一个非核心调用
   - 偶然恢复,不知道为什么

04:30 事故"恢复",写复盘:
   "根因不明,推测是下游某服务异常,已降级"

这个团队装了 Prometheus / Grafana / ELK 三件套,看着挺现代化,但本质上还是"靠老李的直觉 + 运气"。这就是"有监控没可观测性"的真实长相——工具堆了,但出事时还是黑盒。

1.2 这个场景的根本问题 ​

不是 Prometheus 不够强
   → 它能采的指标都采了

不是 ELK 不够全
   → 日志全在里面,7000 条 / 分钟也接得住

不是工程师不够强
   → 老李在 SRE 行业 8 年了

是 "可观测性"的工程动作没做:

  问题 1:数据之间没有"关联键"
    - Metrics、Logs、Traces 各自存在
    - 看到 P99 飙高 → 不知道哪条日志是这次慢的
    - 看到日志慢 → 不知道这次请求经过了哪些下游
    → 三件套各自孤立,看不到"全链路视图"

  问题 2:维度不够细
    - dashboard 是预设的,只显示"整体 P99"
    - 不能现场"按下游分组"、"按 user_id 范围分组"
    - 现场需要的视角必须现场建,但建不出来
    → 这就是"未知的未知"问题

  问题 3:数据采集面不够广
    - JVM 指标没采
    - RDS 慢查询日志没接
    - K8s 节点的 IO Wait 没看
    → 真正能解释根因的数据不在系统里

  问题 4:查询能力弱
    - 7000 条日志没法"按 trace 聚合"
    - PromQL 不会写复杂关联
    → 数据在那里,但取不出来

这四个问题加起来,就是"装了三件套但没做可观测性"的核心症状。

1.3 中型团队踩这种坑的频率 ​

团队规模 / 服务数      可观测性问题严重程度

< 5 人 / < 10 服务      △ 还能靠"全在脑子里"
5-10 人 / 10-50 服务    ⚠ 临界点,装了工具但用不熟
10-30 人 / 50-200 服务  🔴 必崩,工具装了但不解决问题
> 30 人 / > 200 服务    🔥 全员痛苦,但已经投入巨大改造成本

5-10 人 / 50 服务这个阶段最危险——已经过了"靠脑子记"的体量,但还没认真做过可观测性工程,最容易出"装了工具但出事还是抓瞎"的尴尬场面。本系列服务的中型团队(10 人 / 100 微服务 / 5000 QPS)正好踩在这个坑上。

二、Monitoring vs Observability:本质差别 ​

这两个词看起来像"近义词",实际是两种不同的思维方式。

2.1 一个比喻:探照灯 vs 在黑暗里的眼睛 ​

Monitoring(监控)= 探照灯
                              ╲│╱
                            ── O ──   ← 探照灯
                              ╱│╲
                          ┌────┴────┐
                          │ ╱ ╲ ╱ ╲ │
                          │ 亮  亮  │
                          │/ ╱ ╲ ╱ ╲ \│
                          ───────────
                          ░░░░░░░░░░░ ← 暗处看不见

   提前指好探照灯的方向 → 那个方向看得很清楚
   暗处发生事情 → 你完全看不见
   出新问题 → 探照灯没指到 → 黑盒

Observability(可观测性)= 在黑暗里的眼睛 + 手电
                              ╲│╱
                              👁️    ← 你的"眼睛"(查询能力)
                              ╱ ╲
                         ┌────────────┐
                         │ 任何方向都  │
                         │ 能现场探索  │
                         │            │
                         └────────────┘

   不预先指方向,你拥有"现场探索任何地方"的能力
   出新问题 → 现场问出来 → 即使没预设也能查

这个比喻的精髓:Monitoring 是"我事先知道要问什么,把答案做成图";Observability 是"我有能力在出新问题时,现场问出问题的答案"。

2.2 known unknowns vs unknown unknowns ​

四种"知识"的分类(出自 Donald Rumsfeld):

         我知道我知道           我知道我不知道
       (known knowns)        (known unknowns)
            │                       │
            │                       │
  这部分被 Monitoring 覆盖     这部分被 Monitoring 覆盖
  → 用 dashboard 看现状         → 用告警提醒"该问什么"

       我不知道我知道          我不知道我不知道
     (unknown knowns)        (unknown unknowns)
            │                       │
            │                       │
   这部分要靠"现场探索"          这部分是 Observability
   → 系统里有数据但你没意识到   → 出新问题时能不能现场问出来

Monitoring 解决前两个象限,Observability 解决后两个象限——尤其右下角的"unknown unknowns"是 Observability 的核心价值。任何只覆盖前两个象限的系统都不能叫 Observability,只能叫 Monitoring + 告警。

2.3 具体差别长什么样 ​

┌──────────────────────┬──────────────────────────────────┐
│  Monitoring          │  Observability                   │
├──────────────────────┼──────────────────────────────────┤
│  预设 dashboard       │  ad-hoc 查询                     │
│  固定告警规则         │  动态切片                         │
│  阈值触发             │  关联分析                         │
│  按"指标"看           │  按"事件"看                       │
│  服务级聚合           │  请求级追踪                       │
│  事后看图             │  现场探索                         │
│  采集成本可控         │  采集成本可能爆炸                  │
│  适合"日常运营"       │  适合"出新问题时定位"              │
└──────────────────────┴──────────────────────────────────┘

实战意义:
   Monitoring 是"我每天看看系统怎么样" → 日常巡检
   Observability 是"凌晨告警来了我能查清楚" → 应急能力

两者不互斥,而是层层包含:
   Observability ⊃ Monitoring
   有 Observability 的系统自然能做 Monitoring
   只做 Monitoring 的系统达不到 Observability

2.4 一个判定题 ​

怎么判断你的系统有 Observability 还是只有 Monitoring?

判断题(必须 4 题都答 ✓ 才算有 Observability):

□ 出事时,你能 5 分钟内定位"是哪个服务的问题"?
□ 定位到服务后,你能 5 分钟内定位"是哪个具体请求的问题"?
□ 定位到请求后,你能 5 分钟内看到"这个请求经过的所有下游 + 各自耗时"?
□ 如果根因和"业务参数"有关(某个 user_id / 某个商品类目),
   你能现场切片找到那个具体特征?

4 个 ✓ = 有 Observability
3 个 ✓ = 接近 Observability,缺一块
2 个 ✓ = 只有 Monitoring 加上了一点高级功能
< 2 个 ✓ = 只是装了工具,本质是 Monitoring + 日志

国内中型团队 80% 在 2-3 个 ✓ 之间——装了三件套,但关联和查询能力不够。本系列第二层 8 篇要做的就是把这 4 个 ✓ 都拉满。

三、三件套:Metrics / Logs / Traces ​

可观测性的三件套是工业界事实标准——这三种数据类型各自擅长不同,组合起来才完整。

3.1 三件套的本质差别 ​

                  Metrics      Logs        Traces
                  ───────      ────        ──────

数据形态          数字+维度    文本事件    带因果关系的事件链
数据量            小 (聚合)    大 (高保真) 中 (跨服务采样)
查询粒度          聚合视图     单条事件    单次请求全链路
查询代价          低           中-高       中
存储代价          低           高          中
保留时长          14-90 天     7-30 天     1-7 天

擅长回答的问题:
  Metrics  ───→  "整体趋势是什么样?"
                  "有多少请求 / 错误率多高 / P99 多久?"

  Logs     ───→  "这条具体记录里写了什么?"
                  "出错的请求 / 业务事件 / 调试输出?"

  Traces   ───→  "这次请求经过了什么,各自耗时多少?"
                  "跨服务的因果关系是什么?"

3.2 三件套的覆盖区与重叠区 ​

                Metrics
              ┌──────────┐
              │  趋势     │
              │  聚合     │
              │  告警     │
              └────┬─────┘
                   │ Exemplar(指标里带 traceID)
                   │
              ┌────┴─────┐
              │          │
        Logs  │          │  Traces
        ┌─────┤          ├─────┐
        │ 事件 │ trace_id │ 链路 │
        │ 文本 │  关联    │ 因果 │
        │      │  字段    │      │
        └─────┴─────┬────┴─────┘
                    │
                    ▼
              重叠区(三件套关联键)
              这一区是 Observability
              真正发生的地方

三件套各自存在不叫可观测性,三件套通过关联键(trace_id)被串起来才叫可观测性:

关联三件套的核心键:trace_id / span_id

在 Metrics 里:用 Exemplar 携带 trace_id
   → 看 dashboard 上一个慢点,能跳到那次请求的 trace

在 Logs 里:每条日志带上 trace_id
   → 看 trace 里某一步慢,能查那一步的具体日志

在 Traces 里:每个 span 包含 service / endpoint / 耗时
   → 看错误 metric,跳到 trace,跳到 log,全链路一气呵成

这个"关联键贯穿"是可观测性工程最值钱的一件事——11 篇(OpenTelemetry)会展开怎么落地。

3.3 各自擅长 / 不擅长 ​

3.3.1 Metrics 擅长 / 不擅长 ​

Metrics 擅长:
   ✓ 回答"系统整体怎么样"(QPS / 错误率 / P99 延迟)
   ✓ 长期趋势(过去 30 天 / 90 天)
   ✓ 告警(数值越过阈值)
   ✓ 容量规划(看流量增长曲线)
   ✓ 成本极低(每个指标点几个字节)

Metrics 不擅长:
   ✗ 回答"哪一次请求出问题"
     → 因为是聚合,丢失了"个体"
   ✗ 携带任意字段(每个 label 都是 cardinality 代价,见后)
   ✗ 文本(描述 / 错误信息 / 上下文)
   ✗ 因果关系(哪个引起哪个)

适用场景:
   - 仪表盘
   - 告警
   - SLO 计算
   - 容量规划

3.3.2 Logs 擅长 / 不擅长 ​

Logs 擅长:
   ✓ 回答"具体发生了什么"(完整文本 / 上下文)
   ✓ 业务事件(用户操作 / 订单流转)
   ✓ 调试信息(printf 大法)
   ✓ 审计追溯

Logs 不擅长:
   ✗ 整体趋势(从 1 亿条日志里聚合 QPS 是噩梦)
   ✗ 实时告警(延迟和成本都高)
   ✗ 跨服务关联(除非加 trace_id)
   ✗ 持久存储成本(每条几百字节)

适用场景:
   - 调试 / 排查
   - 业务审计
   - 合规要求
   - 异常上下文

3.3.3 Traces 擅长 / 不擅长 ​

Traces 擅长:
   ✓ 回答"这次请求经过了什么"
   ✓ 跨服务因果关系
   ✓ 各步骤耗时分解
   ✓ 异常的"传染路径"

Traces 不擅长:
   ✗ 整体统计(因为是采样,统计不准)
   ✗ 长期保留(数据量大)
   ✗ 业务字段(主要记技术调用)
   ✗ 没有 instrumentation 的服务

适用场景:
   - 跨服务排障
   - 性能下钻
   - 异常根因定位
   - 服务依赖图

3.4 三件套的"优先建设"顺序 ​

中型团队从零开始建可观测性,建议顺序:

第 1 步:Metrics
   → 投入最低,价值最高
   → Prometheus + Grafana,一周能上
   → 先有"整体视图",再谈细节

第 2 步:Logs
   → 大部分团队已经有(打日志是默认的)
   → 加结构化 + 集中收集
   → Loki / ELK 任选

第 3 步:Traces
   → 投入相对大(代码侧需要 instrument)
   → 但 ROI 极高(跨服务排障)
   → OpenTelemetry + Tempo/Jaeger

第 4 步:三件套关联
   → 在 Logs 里加 trace_id
   → 在 Metrics 里加 Exemplar
   → 让三件套从 Grafana 里能一键跳转

第 5 步:Profile(下一节展开)
   → 不是必需,是"能让效率再上一个台阶"
   → 适合有 1000+ QPS / 10+ 服务的团队

不要一上来就 4 件套全上——Metrics 没用熟谈 Traces 是空中楼阁。90% 的事故 Metrics + 关联 Logs 已经能定位,Traces 是给剩下 10% 救命的。

四、第四件套:Continuous Profiling ​

近几年(2020-2025)兴起的"第四件套",很多人没听过,但已经在改变可观测性的边界。

4.1 为什么需要 Profile ​

现有三件套的盲区:

Metrics 告诉你:
   "服务 CPU 利用率 80%"
   ← 但不告诉你"哪个函数占了 80% 的 CPU"

Logs 告诉你:
   "这个请求慢了 5 秒"
   ← 但不告诉你"这 5 秒里 CPU / IO / 锁 各自占多少"

Traces 告诉你:
   "请求 → 服务 A → 服务 B → 数据库"
   ← 但不告诉你"在服务 A 内部哪个函数最耗时"

Profile 填补的是:"函数级别的耗时归因"
   - 哪个函数吃 CPU
   - 哪个函数分配内存
   - 哪个函数被锁阻塞

4.2 Continuous Profiling 是什么 ​

传统 Profiling:
   - 出问题 → 手动启动 profiler → 采 1 分钟样本 → 分析
   - 问题:每次出事都要重头采样,且采样时性能影响大

Continuous Profiling:
   - 始终采样(低频低开销,< 1% CPU)
   - 数据持续存储(7-30 天)
   - 任何时刻都能"回看历史"
   - eBPF / pprof 实现采样

4.3 工业界主流工具 ​

Pyroscope(Grafana 收购)
   - Go / Java / Python / Ruby 等多语言
   - 集成 Grafana
   - 开源 + SaaS 双形态

Parca(Polar Signals)
   - eBPF-based,可以采"任意语言"
   - 不需要语言层 SDK
   - 开源

Datadog Continuous Profiler
   - 商用,集成 Datadog
   - 不开源

Pixie(New Relic 收购)
   - eBPF-based,kubernetes 原生
   - 不只 profile,也做 metrics / traces

4.4 何时上 Profile ​

该上的信号:
   ✓ Metrics 显示 CPU / 内存高,但找不到具体哪个函数
   ✓ 团队 > 10 人,服务 > 50 个
   ✓ 性能优化已经是日常工作(不只是出事时)
   ✓ 你的服务有"长尾延迟"需要解释

不该上的信号:
   ✗ 5 人小团队,5 个服务
     → 出事直接 pprof on-demand 也够
   ✗ 性能不是瓶颈(没人在乎)
   ✗ 还没把 Metrics / Logs / Traces 用熟

12 篇会展开 Continuous Profiling 的工程化,这里只是"知道有这第四件套,以及它解决什么前三件套解决不了的问题"。

五、Cardinality 陷阱(这一篇的重点) ​

讲完三件套 + Profile 的概念,这一节讲一个非常具体的工程陷阱——它是中型团队上 Prometheus 第一年最容易踩的坑,而且踩了就是 P0。

5.1 什么是 Cardinality ​

Cardinality(基数)= 某个 label 的不同值的数量

例:
   指标 http_requests_total{service, endpoint, status}

   假设:
   - service 有 100 个不同值
   - endpoint 平均每个 service 有 20 个不同值
   - status 有 10 个不同值(200/201/4xx/5xx 等)

   这个指标的 cardinality:
   100 * 20 * 10 = 20,000 个 time series

   每个 time series 在 Prometheus 里:
   - 占用约 1-3 KB 内存
   - 写入 IOPS / 网络流量都是单独计算
   - 查询时需要扫描

   20,000 series → 约 50 MB 内存,可接受

5.2 Cardinality 爆炸的样子 ​

某天你心血来潮加了一个 label:
   http_requests_total{service, endpoint, status, user_id}

假设你有 1,000,000 个 user

新的 cardinality:
   100 * 20 * 10 * 1,000,000 = 20,000,000,000 个 time series
                              = 200 亿!

200 亿 series 在 Prometheus 里:
   - 约 30 TB 内存 ← 不可能存得下
   - 写入吞吐量爆炸
   - 查询完全卡死

实际发生的:
   - Prometheus 进程 OOM(从 100GB 内存撑到机器限制)
   - 重启后再 OOM
   - 查询全部超时
   - 整个监控系统挂掉

这就是 cardinality 爆炸——一个 label 改错,一夜之间监控系统瘫痪。

5.3 哪些 label 是高危的 ​

低危(cardinality 低,可以做 label):
   ✓ service / app(几十到几百)
   ✓ method / endpoint(几十)
   ✓ status code(几十)
   ✓ environment(几个)
   ✓ region / az(十几个)
   ✓ error_type 分类(几十种)

高危(cardinality 高,绝不能做 label):
   ✗ user_id(百万级)
   ✗ session_id(每次会话不同)
   ✗ request_id / trace_id(每次请求不同)
   ✗ ip(可能千万级)
   ✗ url(带 query string 的)
   ✗ exception_message(每次出错可能略不同)

边界情况(看具体场景):
   △ tenant_id(多租户系统,可能几十到几千)
     → < 100 可以,> 1000 危险
   △ container_name(K8s 里,pod 重启后变)
     → 用 deployment 名替代,别用 pod 名
   △ version(可能几十个)
     → 多数情况可以,小心"版本日积月累膨胀"

5.4 真实事故案例 ​

案例 A:trace_id 当 label

某团队上 OTel,工程师为了"在 Prometheus 里能跳 trace",
加了一个 label: trace_id

每次请求生成新 trace_id → 每条请求都是新 series
QPS 1000 → 一分钟 60,000 个新 series
一小时 360 万 series,Prometheus 内存爆炸

修复:回滚 label,改用 Exemplar(只在某些样本上带 trace_id)


案例 B:user_id 在 metric 上

某团队为了"看每个用户的 SLO",在 latency metric 上加 user_id

100 万用户上线第一天:
   Prometheus 内存从 10G 飙到 80G
   写入延迟从 < 1s 升到 30s
   告警延迟到没意义
   挂了

修复:user_id 不放 metric,放 trace / log 里
       Metric 只看"整体" + "按用户分桶(top 100 用户)"


案例 C:exception_message 当 label

某团队的 error metric 上加了 error.message label
不同请求的 error message 略有不同(带参数)

  "user 12345 not found"
  "user 12346 not found"
  ...

每个不同 user_id 出错都是新 series → cardinality 爆炸

修复:error.message → error.code(枚举值,有限种)
       原 message 保留在 log 里

5.5 Cardinality 预算估算公式 ​

Prometheus 单实例的 cardinality 预算估算:

  series 数量上限 ≈ 总内存(GB) * 100,000

  例:32GB 内存的 Prometheus
      可承载约 3,200,000 series
      留 50% 余量 → 1,600,000 series 是安全上限

  超过这个数量:
   - 写入开始慢
   - 查询超时
   - GC 占用 CPU
   - 最终 OOM

如何估算你的 cardinality:
  方法 1:Prometheus 自带查询
    prometheus_tsdb_head_series  ← 当前活跃 series 数

  方法 2:看每个指标的 cardinality
    count by (__name__) ({__name__=~".+"})

  方法 3:Grafana Mimir / Cortex 的 cardinality API
    /api/v1/cardinality/active_series

实操建议:

设 Cardinality 预算硬阈值:
   1M series   → 警告
   2M series   → 严重(必须人为介入)
   3M series   → 危险(开始拒新指标)
   4M series   → 拒服务(只保留核心指标)

监控这个数字本身的告警:
   - prometheus_tsdb_head_series > 2,000,000
   - 增长率 > 10% / 天 → 报警

5.6 怎么"在保留观测能力"的同时"控制 cardinality" ​

原则 1:Metric 不放高 cardinality 字段
   - 高 cardinality 字段放 Logs / Traces
   - Metric 只保留聚合后的"分桶"(top N 用户、按租户分组)

原则 2:用 Exemplar 桥接
   - Metric 不带 trace_id,但每个数据点可以附 1 个 trace_id
   - 看 Grafana panel 时能"跳" trace
   - Exemplar 不影响 cardinality

原则 3:Recording Rule 预聚合
   - 高 cardinality 的原始指标只短期保留
   - 用 Recording Rule 聚合成低 cardinality 的派生指标
   - 长期存储用聚合后的(06 / 07 篇会展开)

原则 4:多实例隔离
   - 一个 Prometheus 不应该跨大业务线
   - 每个团队 / 每个业务一个 Prometheus
   - 用 Federation 或 VM-cluster 做全局视图(08 篇)

六、可观测性的 3 个心智 ​

讲完工具和陷阱,最后用 3 个心智收尾——这是衡量"团队的可观测性成不成熟"的核心维度。

6.1 心智 1:能定位故障(Locate) ​

最低限度:出事时能 5 分钟定位"是哪个组件"

具体能力:
   - 看到告警 → 知道"哪个服务在烧"
   - 看到服务 → 知道"哪个 endpoint 在烧"
   - 看到 endpoint → 知道"是慢还是错"
   - 看到慢 → 知道"是哪个下游慢"
   - 看到错 → 知道"是哪类错"

工具组合:
   Metrics 黄金 4 信号 dashboard
   + 服务依赖图(Service Map)
   + 错误堆栈聚合(Top N 错误类型)

判断你是否做到:
   □ 出事时,5 分钟内能在 Grafana 上画出"故障传播路径"
   □ 知道"哪些服务正常,哪些服务异常"的边界
   □ 能给"故障范围"打一个准确的标签:核心 / 边缘 / 全局

这是可观测性的"低阶能力",没做到这个,后面两个心智都谈不上。

6.2 心智 2:能解释为什么(Explain) ​

进阶能力:出事后能 30 分钟讲清楚"为什么"

具体能力:
   - 定位到组件 → 能查到"那一次/那段时间发生了什么"
   - 能区分:发布引起的 / 流量引起的 / 依赖引起的 / 容量引起的
   - 能给出根因链(不是"猜",是"证据链")

工具组合:
   定位能力(心智 1)
   + Logs 的精确查询
   + Traces 的下钻
   + 变更系统(发布历史 / 配置变更)
   + Profile(必要时)

判断你是否做到:
   □ 复盘文档里"根因"能写到"哪行代码 / 哪个配置 / 哪个依赖"
   □ 不再用"应该是 XX,但没确凿证据"这种话
   □ 复盘和事故时间线能 100% 对得上

这是大多数中型团队卡住的台阶——出事能恢复但讲不清,导致同样的事故 6 周后再来。

6.3 心智 3:能预测下次(Predict) ​

高阶能力:从这次出事推测下次会出什么

具体能力:
   - 看到"P99 在悄悄涨" → 预测"两周后会撞 SLO"
   - 看到"内存增长曲线" → 预测"X 天后会 OOM"
   - 看到"某个依赖的错误率上升" → 预测"它会拖垮我们"
   - 看到"流量增长" → 预测"什么时候需要扩容"

工具组合:
   解释能力(心智 2)
   + 长期 Metrics(30-90 天)
   + 容量规划工具(30 篇会讲)
   + 趋势告警(不是阈值,是"变化速度"告警)

判断你是否做到:
   □ 团队定期(月度)开"健康趋势 review"
   □ 容量瓶颈是"预测"的,不是"撞上"的
   □ 容量扩容是"计划"的,不是"应急"的
   □ 有"早期预警告警"——在用户感知前就动作

这是 Google SRE 团队的水平,中型团队能做到 50% 已经很好。

6.4 三个心智的层级关系 ​

心智 1: Locate(能定位)
   ↑
   │ 必须先做到,才能往上走
   ↓
心智 2: Explain(能解释)
   ↑
   │ 在能定位的基础上,加上"证据链"
   ↓
心智 3: Predict(能预测)
   ↑
   │ 在能解释的基础上,看趋势 + 容量
   ↓
   (最高级)

中型团队的合理目标:
   - 6 个月做到 心智 1
   - 1 年做到 心智 2 一部分
   - 2 年做到 心智 3 雏形

不要直接追心智 3——基础没打好,预测都是猜测

八、怎么落地:从无到有的 3 步 ​

8.1 第 1 步:从 Prometheus + Grafana 开始 ​

最小可行的可观测性 V0:

部署:
   - 一个 Prometheus(< 1M series 不用 federation)
   - 一个 Grafana
   - 一个 Alertmanager

埋点:
   - 每个服务暴露 /metrics(client_golang / Micrometer / prom-client)
   - 至少有黄金 4 信号(02 篇讲的)

可观测层级:
   ✓ 能看整体趋势
   ✓ 能告警
   ✗ 不能定位单条请求
   ✗ 不能跨服务关联

这一步上线后,你已经超过 50% 的国内中型团队——很多团队连这一步都没做齐。

8.2 第 2 步:加结构化日志 + 集中收集 ​

最小可行的可观测性 V1:

加上:
   - 日志改成结构化(JSON,带 trace_id 字段)
   - Loki 或 ELK 集中收集
   - Grafana 接 Loki/ES 数据源

可观测层级:
   ✓ 整体趋势(V0)
   ✓ 告警(V0)
   ✓ 单条请求查询(从 trace_id 查)
   ✓ 错误堆栈聚合
   ✗ 还不能跨服务因果

8.3 第 3 步:加 OpenTelemetry + Traces ​

最小可行的可观测性 V2:

加上:
   - OpenTelemetry SDK 在所有服务接入
   - OTel Collector 集中接收 + 路由
   - Tempo 或 Jaeger 存 Traces
   - Grafana 接 Tempo

可观测层级:
   ✓ 整体 + 告警 + 单请求 + 跨服务因果
   ✓ "三件套关联"(metric Exemplar → trace → log)
   ✓ 能定位 + 能解释

做到这一步,你的团队就是国内可观测性的"前 20%"——再往下做 Profile / Predict 是锦上添花。

九、何时不该追求"全面可观测" ​

不是所有团队都该上完整的可观测性栈。

9.1 不需要全栈可观测的场景 ​

场景 1:5 人小团队 + 单服务
   → SSH + 文件日志 + 简单 Prometheus 已经够
   → 上 OTel 是过度工程

场景 2:实验项目 / 内部工具
   → 没人 7x24 看,装了 dashboard 也没人看
   → 用云厂商自带的(CloudWatch / 阿里云 ARMS)更省事

场景 3:单台 VM 时代
   → 一台机器,日志直接看就行
   → 没必要 Loki / ELK

场景 4:预算极度紧张
   → 全栈可观测每月成本 ¥5000-50000
   → 月入百万以下的小厂,先看 ROI

场景 5:团队人手不够维护
   → 装了 Prometheus 没人调 PromQL
   → 装了 ELK 没人调 ES 配置
   → 与其装而不会用,不如不装

9.2 何时该上的判断 ​

触发可观测性投资的信号:

✓ 单月发生 > 2 次"出事但说不清根因"
✓ 团队规模 > 8 人
✓ 微服务数量 > 20 个
✓ QPS > 1000
✓ 有 7x24 业务需求
✓ 客户开始问"SLA 怎么定义"
✓ 上次招到一个开发,他问"你们有 Grafana 吗"——这表示行业期望

满足 3 个以上 → 该上了
满足 5 个以上 → 你已经晚了

十、踩坑提醒 ​

  1. 以为"装了 Prometheus 就是可观测性"——只是 Monitoring 的一部分
  2. 三件套各自为政,没关联键——trace_id 没透传,三件套就是三个孤岛
  3. user_id / trace_id 当 Prometheus label——cardinality 爆炸,系统挂掉
  4. 没有 SLO 兜底的 dashboard——画了 100 个图没人看,没有"健康/不健康"判定线
  5. 只看 metric 不看 trace——对慢请求无能为力,只能整体看不能下钻
  6. 日志非结构化——查询难,没法和 trace 关联
  7. 采集的指标太少——核心指标缺(JVM / DB / 队列),根因无从查
  8. 采集的指标太多——cardinality 爆炸,Prometheus 拖垮
  9. 告警满天飞——做 Monitoring 不做告警工程(15 篇会讲)
  10. 没人会写 PromQL——会问问题但查不出答案
  11. dashboard 没人维护——上线时漂亮,半年后过期
  12. 不区分 Monitoring 和 Observability——以为装更多工具就是 Observability,实际是要"查询能力"

十一、本篇的硬指标 ​

看完这一篇,你应该能在白板前讲清楚:

□ Monitoring 和 Observability 的本质差别(known unknowns / unknown unknowns)
□ Metrics / Logs / Traces 各自擅长什么、不擅长什么
□ 三件套怎么通过 trace_id 关联,关联后能解决什么
□ Cardinality 爆炸是什么,user_id 当 label 为什么会让 Prometheus 挂
□ Cardinality 预算怎么估算(每 GB 内存约 10 万 series)
□ Continuous Profile 解决了三件套解决不了的什么问题
□ 可观测性的 3 个心智(Locate / Explain / Predict)
□ 中型团队的可观测性落地路线(V0/V1/V2 三步)

并且能给自己团队做出 2 件具体动作:

1. 评估当前可观测性在哪个心智层级
   - Locate / Explain / Predict
   - 列出"未达到"的能力缺口

2. 检查 Prometheus 的 cardinality 风险
   - 当前 series 数
   - 高风险 label 排查(user_id / trace_id / message)
   - 设 cardinality 预算硬告警
