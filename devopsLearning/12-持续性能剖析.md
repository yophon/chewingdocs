# 持续性能剖析:Continuous Profiling / Pyroscope / Parca / eBPF 视角

讲性能剖析的文章一般都从「go tool pprof」起手——这是错的视角。**pprof 是"出事后跑一下的工具",Continuous Profiling 是"7×24 持续采集 / 系统级常驻 / 像 Prometheus 一样查询的可观测性支柱"**。这俩不是同一个工具的两种用法,**是两种心智**。10 / 11 篇讲完日志和链路追踪,这一篇收尾可观测性的最后一支柱——**它不像 metrics / logs / traces 那么主流,但中型团队上了之后**,**生产里那些"CPU 莫名涨 30%、内存毛刺找不到原因、某次发布之后 GC 抖动"的诡异问题,定位时间从天降到小时**。

> 一句话先记住:**Profile 是可观测性的第四件套——metrics 告诉你"有问题"(P99 涨了),trace 告诉你"问题在哪一跳"(payment 调 risk 时慢),profile 告诉你"那一跳里哪行代码在烧 CPU"**。**前三件套各自只能下钻到一个层次,profile 是下钻到代码行**。这一篇默认你看过 osLearning/26-27 的 eBPF 基础——本系列不重复讲 eBPF 怎么工作,**只讲"持续性能剖析"这个工程化命题**。

## 一、为什么 Profile 是第四件套

可观测性三件套(metrics / logs / traces)广为人知,**为什么还要加上 profile**?——讲一个真实的故障对比。

### 1.1 一个三件套救不了的场景

**场景**:某天上午,**payment-service 的 CPU 从 30% 涨到 80%**——但请求量没变,P99 也正常。

```
   排查路径 1:看 metrics
   ──────────────────────────────────────────
   ✓ CPU 涨了 → 但 QPS、P99、GC time 都没变化
   ✗ metrics 看到现象,看不到原因
   
   排查路径 2:看 traces
   ──────────────────────────────────────────
   ✓ 找一个请求看,trace 显示总耗时还是 200ms
   ✓ 每一跳都和往常一样
   ✗ trace 是"请求级"的视图,看不到 CPU 在哪烧
   
   排查路径 3:看 logs
   ──────────────────────────────────────────
   ✓ 日志没有异常,没有 ERROR
   ✗ logs 是"事件级"的视图,记录不了 CPU 占用
   
   ──────────────────────────────────────────
   三件套都告诉你"有问题",但都说不出"问题在哪行代码"
   ──────────────────────────────────────────
```

**如果有 profile**:**打开 Pyroscope,选这个服务最近 1 小时的 CPU 火焰图,跟昨天的对比**——

```
   差分火焰图(今天 vs 昨天):
   ──────────────────────────────────────────
   昨天:json.Marshal 占 CPU 5%
   今天:json.Marshal 占 CPU 35%   ← 涨了 7 倍
   
   下钻:
   json.Marshal
     └── encoding/json.(*encodeState).marshal
         └── encoding/json.reflectValueEncoder
             └── reflect.Value.Interface       ← CPU 烧在这
   
   一看就明白:
   今天有人发了个版,新加了个接口,response body 是一个巨大的嵌套结构,
   每次 JSON 序列化都通过 reflect 反复算类型,根本没用 cached encoder
```

**5 分钟定位完**。**这就是 profile 的价值**——它是**唯一能"在代码行级别"定位资源消耗**的工具。

### 1.2 四件套各自能解决的问题

```
   ┌────────────────────────────────────────────────────────┐
   │  Metrics                                                │
   │  能回答:"P99 怎么样?CPU 多少?QPS 多少?"             │
   │  粒度:服务级 / 实例级                                 │
   │  价值:全局健康度 + 告警入口                            │
   ├────────────────────────────────────────────────────────┤
   │  Logs                                                   │
   │  能回答:"那一次错误是什么原因?"                       │
   │  粒度:事件级                                          │
   │  价值:事后取证 + 业务事件回放                          │
   ├────────────────────────────────────────────────────────┤
   │  Traces                                                 │
   │  能回答:"这个慢请求慢在哪一跳?"                       │
   │  粒度:请求级 / 跨服务                                 │
   │  价值:分布式系统的因果关系                            │
   ├────────────────────────────────────────────────────────┤
   │  Profile                                                │
   │  能回答:"那一跳里,CPU / 内存花在哪一行代码?"          │
   │  粒度:代码行级                                        │
   │  价值:资源消耗的根因定位                              │
   └────────────────────────────────────────────────────────┘
```

**关键事实**:**这四件套不是冗余,是从粗到细的下钻链路**——任何一层都解决不了另一层的问题。**100 个微服务 + 5000 QPS** 的中型团队,这四层缺一层,排查链路就断一截。

## 二、从 pprof 到 Continuous Profiling

**性能剖析这件事 20 年前就有**(Linux perf / Java jstack / Go pprof),**为什么 2022 年之后突然火起来变成 Continuous Profiling**?——本质上是**采集方式 + 存储方式 + 查询方式**都变了。

### 2.1 传统 pprof 的工作流

```
   传统姿势:出问题再跑 pprof
   ──────────────────────────────────────
   
   T+0      告警:CPU 异常涨
   T+5      工程师上线
   T+10     找 pod:kubectl exec -it ... 
   T+15     在 pod 里跑:go tool pprof http://localhost:6060/debug/pprof/profile
            等 30 秒采集
   T+45     pprof 交互界面分析
   T+60     web 命令打开浏览器看火焰图
   
   问题:
   - "T+5 到 T+60" 这一小时,问题可能已经消失
   - 短时尖峰(几秒钟)抓不到——上线时早过了
   - 历史问题完全不可回溯
   - "上次和这次的对比"做不了(没有历史 profile)
```

**传统 pprof 是个被动的、临时的、孤立的工具**——**只在出事后 + 在线时跑一次**。**这种模式在 5000 QPS / 100 微服务规模下完全不够**:

- **大部分故障不可重现**(发版后某个时间窗口的事,过了就没了)
- **想看"一周前那次诡异的 P99 抖动",pprof 给不了**
- **微服务多了,挨个跑 pprof 不现实**

### 2.2 Continuous Profiling 是什么

**Continuous Profiling = 把"出事跑 pprof"改成"7×24 后台采集 + 集中存储 + 按时间 / 服务 / 标签查询"**。

```
   持续剖析的工作流:
   ──────────────────────────────────────
   
   持续阶段(零干预):
   每隔 10s,Agent 自动采集每个服务的 CPU profile(10s 采样)
   每隔 10s,Agent 自动采集每个服务的 heap profile
   profile 数据带着 service / pod / version / env 标签
   推送到中心化存储(Pyroscope / Parca / Polar Signals)
   
   故障排查阶段:
   ─────────────────────
   T+0      告警:某服务 CPU 异常
   T+1      打开 Pyroscope dashboard
   T+2      选服务 + 时间范围(故障时刻)
   T+3      看火焰图,or 看差分火焰图(故障 vs 正常)
   T+5      定位完
```

**核心差异**:**pprof 是"我得在线"才能跑,Continuous Profiling 是"数据一直在,等你来查"**——和 metrics / logs / traces 的逻辑完全对齐。

### 2.3 为什么 2022 之后才普及

之前不行的两个限制,**2022 前后被打破**:

1. **采集开销**——传统 pprof 用 SIGPROF 信号 100Hz 采样,CPU 开销 1-3%——**全量持续采集就会涨到 5-10%,生产不能上**。**eBPF 把开销降到 < 0.5%**(后面展开)。

2. **存储成本**——pprof 文件每个几 MB,**全公司全服务 7×24 采集,一天产生 TB 级数据**。**Pyroscope / Parca 用专门的"火焰图压缩"算法,把存储压到 1/100**。

**这两个限制突破后**,Continuous Profiling 才从"理论好" 变成"中型团队真能用"。

## 三、Pyroscope / Parca / Polar Signals:三个工具的画像

当前 Continuous Profiling 生态有三个主流玩家,**画像差异很大**。

### 3.1 Pyroscope:Grafana 系亲儿子

Pyroscope 2020 年开源,**2023 年被 Grafana Labs 收购,集成进 Grafana 全家桶**。**和 Loki / Tempo 一个套路**——同一个 dashboard 能看 metrics / logs / traces / profiles 四件套。

```
   ┌──────────────────────────────────────────────────────┐
   │ Pyroscope                                             │
   │   采集方式:                                          │
   │     - SDK 模式(应用主动 push)                       │
   │     - Pull 模式(类似 Prometheus,从 /debug/pprof 拉) │
   │     - eBPF 模式(零代码改动)                          │
   │   存储:                                              │
   │     - 自研列式存储,本地盘或对象存储                  │
   │   查询:                                              │
   │     - PromQL-like 语法 + 火焰图 UI                    │
   │     - Grafana 集成                                    │
   └──────────────────────────────────────────────────────┘
```

**亮点**:

- **Grafana 集成最深**——dashboard 上点 trace_id 就跳 Pyroscope,**和 Tempo trace 关联**(从 trace 一键跳到那个时刻的 profile)
- **SDK 体验最好**——Go / Java / Python / Node.js / Ruby SDK 都有,**2 行代码接入**
- **支持多种 profile 类型**——CPU / heap / goroutine / mutex / block
- **对象存储后端**——和 Loki 一样可以接 S3,**长期保留成本低**

**短板**:

- **eBPF 支持是后加的**——不如 Parca 原生
- **多语言深度不一**——Go / Java 最好,PHP / Ruby 一般

**适合**:**已经在用 Grafana 全家桶的团队**——零思考切入。

### 3.2 Parca:eBPF 原生

Parca 是 Polar Signals 公司 2021 年开源的,**核心卖点是 eBPF 原生**——**不需要改代码 / 不需要 SDK / 不需要重启服务,装个 DaemonSet 就开始采集**。

```
   ┌──────────────────────────────────────────────────────┐
   │ Parca                                                 │
   │   采集方式:                                          │
   │     - parca-agent (DaemonSet) ← 主推                  │
   │     - 内核 eBPF 钩子 + perf_event 采样                │
   │     - 零代码改动                                      │
   │   存储:                                              │
   │     - 自研 FrostDB(列式 + Arrow)                    │
   │   查询:                                              │
   │     - PromQL-like + 火焰图 UI                         │
   └──────────────────────────────────────────────────────┘
```

**亮点**:

- **零侵入**——不动应用代码,**老系统也能采**
- **跨语言一致**——Go / Java / Python / Node / Rust / C++ 都从内核视角采,**采集姿势统一**
- **资源占用极低**——eBPF 内核态采样,用户态开销 < 0.5%

**短板**:

- **符号化是难点**——JIT 语言(Java / Node.js / Python)的栈展开需要专门的 symbolizer
- **需要较新内核**(5.4+)
- **社区比 Pyroscope 小**

**适合**:**多语言异构 + 不想改业务代码 + 内核够新的团队**。

### 3.3 Polar Signals:商业 SaaS

Polar Signals 是 Parca 背后的公司,**SaaS 是商业产品**——零运维,**Polar Signals Cloud 是托管 Parca**。

**适合**:**< 30 服务、不想自建、预算允许**——和 Datadog / New Relic 一个逻辑。**100+ 服务团队的成本会让你想自建**。

### 3.4 三者对比

| 维度 | Pyroscope | Parca | Polar Signals |
| --- | --- | --- | --- |
| 主流采集方式 | SDK / Pull / eBPF | **eBPF 原生** | eBPF + SDK |
| 多语言支持 | 强(SDK 多) | **强(eBPF 通吃)** | 同 Parca |
| 资源开销 | SDK 1-2% / eBPF 0.5% | **< 0.5%** | < 0.5% |
| Grafana 集成 | **深** | 中 | 中 |
| 长存对象存储 | **有(S3)** | 有 | 托管 |
| 学习曲线 | 低 | 中 | 极低(SaaS) |
| 适合规模 | 中小团队 | 中大团队 | 任意 |

**90% 中型团队的最优解是 Pyroscope**——和 Grafana / Loki / Tempo 一套栈,**新东西最少**。

## 四、eBPF 视角:为什么 Profile 进入"零侵入"时代

**eBPF(extended Berkeley Packet Filter)是 Linux 内核 4.4+ 引入的可编程内核 hook 机制**——**osLearning/26-27 详细讲过**,这里只讲它和持续剖析的关系。

### 4.1 传统采样 vs eBPF 采样

```
   传统采样(SIGPROF 信号):
   ──────────────────────────────────────
   1. 进程注册 SIGPROF 信号处理器
   2. setitimer 设置每 10ms 触发一次
   3. 信号触发时,内核给进程发信号
   4. 进程的信号处理器读栈、写文件
   
   问题:
   - 必须有信号处理器 → 改代码
   - 多线程时信号竞争 → 数据不准
   - 进程 hang 时采不到
   - GC 期间信号被屏蔽 → 漏数据
   
   ───────────────────────────────────
   
   eBPF 采样(内核态):
   ──────────────────────────────────────
   1. eBPF 程序挂到 perf_event(每 10ms 触发一次)
   2. 内核态采集所有 CPU 上当前正在执行的进程栈
   3. 栈数据写到 perf buffer
   4. 用户态 agent 异步读取
   
   优势:
   - 完全在内核态,不依赖进程合作
   - 任何进程(改不改代码)都能采
   - 进程 hang 也能采到(对排查死锁有用)
   - 跨进程统一视角
```

**eBPF 的核心价值**:**采集和被采集进程解耦**——agent 是一个独立的 DaemonSet,**业务进程不知道自己被采集了**。

### 4.2 JIT 语言的符号化挑战

**eBPF 采到的栈是"地址 + 指令指针"**,**要变成"函数名 + 行号"需要"符号化"**(symbolization)——这一步对编译型语言(Go / C++ / Rust)简单,**对 JIT 语言(Java / Node.js / Python / .NET)是个工程难点**。

```
   编译型语言:
   ──────────────────────────────────────
   二进制文件里有 DWARF debug info
   地址 0x40123A → main.processOrder (order.go:42)
   一次查表完事
   
   ───────────────────────────────────
   
   JIT 语言:
   ──────────────────────────────────────
   JVM / V8 在运行时把字节码编译成机器码
   同一个方法的机器码地址每次启动都不同
   还会动态优化(deoptimize)
   
   需要:
   - JVM 给 agent 提供 perf-map 文件
     (运行时持续更新地址 → 方法名映射)
   - eBPF agent 解析 perf-map,做符号化
```

**Pyroscope / Parca 都支持 JIT 语言,但配置比编译型语言复杂**:

```bash
# Java 应用启动时加 agent,生成 perf-map
java -agentpath:/lib/libperf-jvmti.so \
     -XX:+PreserveFramePointer \
     -jar app.jar

# Parca-agent 读取 /tmp/perf-{pid}.map
```

**经验**:**Go / Rust / C++ 是 eBPF profiling 的甜区**,**Java 能用但要配置**,**Node.js / Python 能用但精度打折**,**PHP / Ruby 目前生态最弱**(后面"何时不该用"会展开)。

## 五、火焰图怎么读

**火焰图是 profile 数据的可视化**——**Brendan Gregg 2011 年发明,是性能调优的标准语言**。**不会读火焰图 = 不会用 profile**。

### 5.1 火焰图的视觉规则

```
   ┌─ y 轴:调用栈(下→上,父→子) ────────────┐
   │                                                  │
   │   ┌─main─┐                                       │
   │   │      └─[一栋楼,横轴长度 = CPU 占比 %]      │
   │   │                                              │
   │   └──── x 轴:CPU 占比,从左到右排序 ────────┘
   
   读图四步:
   ─────────────────────────────────────────────
   1. 从下往上看(父→子)
   2. 横轴长度 = 函数占 CPU 比例
   3. 找"宽的平顶"——平顶 = 该函数自身耗 CPU(不是子函数)
   4. 比对历史 → 看哪个函数比以前宽了
```

### 5.2 看图实例

一个典型的 CPU 火焰图:

```
   ──────────────────────────────────────────────────────────────
   │ runtime.systemstack    │     │     │       │              │ 
   │                        │     │     │       │              │
   │ ┌─encoding/json─┐                │  ┌─database/sql─┐     │
   │ │ Marshal       │                │  │ Query          │     │
   │ │ ┌─reflect─┐   │                │  │ ┌─net.Read─┐  │     │
   │ │ │ Value   │   │                │  │ │ syscall  │  │     │
   │ │ └─────────┘   │                │  │ └──────────┘  │     │
   │ └───────────────┘                │  └────────────────┘    │
   │                                  │                          │
   │ main.handleOrder ──────────────────────────────────────── 80%
   │ net/http.ServerHandler.ServeHTTP ──────────────────────── 90%
   │ main.main ─────────────────────────────────────────────── 100%
   ──────────────────────────────────────────────────────────────

   解读:
   - handleOrder 占整个 CPU 的 80%(剩 20% 是其他 handler)
   - handleOrder 里:JSON 序列化 40%(reflect 占大头),DB Query 30%,
     其他 10%
   - reflect.Value 是"平顶"——说明大量 CPU 烧在反射本身
   → 优化方向:用 code-gen JSON 库(easyjson)避免 reflect
```

### 5.3 差分火焰图:对比是王道

**单个火焰图能看出"热点",但找"为什么这次比上次慢"靠对比**。**差分火焰图**(differential flame graph)把两个时刻的 profile 叠在一起:

```
   差分火焰图:故障时刻 vs 正常时刻
   ──────────────────────────────────────────
   红色 = 故障时刻比正常多耗的 CPU
   蓝色 = 故障时刻比正常少耗的 CPU
   灰色 = 没变化
   
   典型应用场景:
   - 发版前 vs 发版后(找回归)
   - 上一小时 vs 这一小时(找突发)
   - prod vs staging(找环境差异)
```

**Pyroscope / Parca 都内置差分火焰图**——**这是它们比传统 pprof 强大的核心**。

## 六、生产持续剖析的开销 / 隐私 / 安全

**Continuous Profiling 在生产 7×24 跑,有三个真实顾虑**——这一节讲清楚怎么应对。

### 6.1 开销:CPU 1% 是个值得的代价

```
   eBPF 持续采样:
   ─────────────────
   每秒采 100 次栈(100Hz)
   栈展开 + 写 perf buffer:每次 ~10us
   总开销:100 × 10us = 1ms/s = 0.1% CPU
   
   实际生产实测(全套 Pyroscope / Parca):
   ─────────────────────────────────────
   - CPU 开销:0.3-1%(eBPF) / 1-2%(SDK)
   - 内存开销:agent 50-150 MB
   - 网络开销:1-5 KB/s 上传(压缩后)
```

**1% CPU 的代价能换"代码行级别的可观测性"**——**这是高性价比的交易**。**例外**:**真极致 latency-sensitive 的场景**(交易 / 高频)——这种业务往往关掉 profile,只在出事时短期开。

### 6.2 隐私:profile 数据会不会包含 PII

**理论上 profile 是"函数名 + 调用栈"**——不直接包含数据。**但有两个间接风险**:

1. **函数参数可能进栈名**——某些 instrumentation(如 OpenTelemetry profiling experiment)会把参数值带上
2. **堆 profile 包含对象类型 / 数量**——理论上是 metadata,但如果有 `User_Phone_13812345678` 这种类名(罕见),就泄露了

**应对**:

- **采集端配置只采"函数级别",不采"参数级别"**——默认就是这样
- **PII 在源头不应该进函数名**(代码评审能拦)

**经验**:**profile 数据比 logs 安全多了**——但**审计时 profile 数据也要算敏感数据,访问要授权**。

### 6.3 安全:eBPF 不是任意运行的内核代码

**eBPF 程序运行在内核态,听起来很危险**——**实际上 Linux 内核有完整的 verifier**:

- **加载时静态分析**——确保程序在固定时间内结束(没死循环)
- **内存安全**——不能访问任意内核内存
- **能力限制**——需要 `CAP_BPF` 或 `CAP_SYS_ADMIN`

**生产部署的实际安全风险**:

- **agent pod 需要 hostPID / privileged 权限**——这是 K8s 安全敏感点
- **eBPF 代码本身**——用主流工具(Pyroscope / Parca)的预编译代码就行,**别自己写**

**应对**:**用社区主流 agent + K8s 默认 Pod Security Standards baseline 之外的特权 namespace 隔离**——**别 agent 跑在业务 namespace 里**。

## 七、最小 Pyroscope SDK 接入:Go 例子

挑 Go 因为最简单,**这段代码 2 行启动 + 1 个标签维度,生产可用**。

```go
package main

import (
    "os"
    "github.com/grafana/pyroscope-go"
)

func main() {
    // === 2 行启动 Pyroscope ===
    pyroscope.Start(pyroscope.Config{
        ApplicationName: "order-service",
        ServerAddress:   "http://pyroscope:4040",
        Logger:          pyroscope.StandardLogger,
        
        // 关键:加标签维度,Pyroscope 上能按标签切分
        Tags: map[string]string{
            "env":     os.Getenv("ENV"),         // prod / staging
            "version": os.Getenv("APP_VERSION"),  // 发版定位回归用
            "region":  os.Getenv("REGION"),       // 多机房切分
        },
        
        ProfileTypes: []pyroscope.ProfileType{
            pyroscope.ProfileCPU,
            pyroscope.ProfileAllocObjects,
            pyroscope.ProfileAllocSpace,
            pyroscope.ProfileInuseObjects,
            pyroscope.ProfileInuseSpace,
        },
    })

    // === 业务代码完全不变 ===
    runOrderService()
}

// 进阶:用 LabelsWrapper 给特定代码块加动态标签
func processOrder(ctx context.Context, orderID string) {
    pyroscope.TagWrapper(ctx, pyroscope.Labels(
        "endpoint", "/api/orders",
        "user_tier", "vip",
    ), func(ctx context.Context) {
        // 这段代码里的 CPU 都会带上面的标签
        // → Pyroscope 上能筛"vip 用户的 /api/orders 慢在哪"
        doExpensiveStuff(ctx)
    })
}
```

**这段代码的 5 个关键点**:

1. **`pyroscope.Start` 是异步的**——不阻塞应用,业务代码零改动
2. **`Tags` 是 Pyroscope 切分维度**——和 Prometheus label 一个逻辑,**dashboard 上能按 env / version / region 筛**
3. **`ProfileTypes` 同时采 5 类**——CPU + heap + alloc——**全套数据后期都能查**
4. **`TagWrapper` 给特定代码块动态加标签**——**vip 用户和普通用户的代码路径分开看**,这是 SDK 模式独有的能力
5. **`os.Getenv("APP_VERSION")`** 必加——**版本回归排查靠这个**(发版前 / 后差分火焰图)

## 八、最小 Parca DaemonSet yaml

如果团队选 Parca(零代码改动),**部署是个 DaemonSet**——下面这段 25 行能生产用:

```yaml
# parca-agent-daemonset.yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: parca-agent
  namespace: parca
spec:
  selector:
    matchLabels: { app: parca-agent }
  template:
    metadata:
      labels: { app: parca-agent }
    spec:
      hostPID: true                                # 关键:看到所有进程
      serviceAccountName: parca-agent
      containers:
      - name: parca-agent
        image: ghcr.io/parca-dev/parca-agent:v0.31.0
        args:
          - /bin/parca-agent
          - --node=$(NODE_NAME)
          - --remote-store-address=parca-server:7070
          - --remote-store-insecure
          - --profiling-cpu-sampling-frequency=19   # 19 Hz,避免和 100Hz 共振
        env:
        - name: NODE_NAME
          valueFrom: { fieldRef: { fieldPath: spec.nodeName } }
        securityContext:
          privileged: true                          # eBPF 需要
          readOnlyRootFilesystem: true
        resources:
          requests: { cpu: 200m, memory: 256Mi }
          limits:   { cpu: 1, memory: 1Gi }
        volumeMounts:
          - { name: tmp, mountPath: /tmp }
          - { name: run, mountPath: /run, readOnly: true }
          - { name: modules, mountPath: /lib/modules, readOnly: true }
      tolerations:
      - { operator: Exists }                         # 跑在所有节点
      volumes:
        - { name: tmp, emptyDir: {} }
        - { name: run, hostPath: { path: /run } }
        - { name: modules, hostPath: { path: /lib/modules } }
```

**配置 5 个关键决定**:

1. **`hostPID: true`** + **`privileged: true`**——eBPF 需要内核能力,**这两个权限要 K8s 集群管理员审批**
2. **`profiling-cpu-sampling-frequency=19`**——故意用 19 Hz(质数),**避免和应用其他 100 Hz 采样器共振导致采样偏差**
3. **`tolerations: Exists`**——所有节点都装,**包括 master / tainted 节点**(确保覆盖)
4. **`resources.limits`**——必加,**没限制的 agent 是定时炸弹**(10 篇讲过同样道理)
5. **`hostPath: /lib/modules`**——eBPF 需要读内核符号表

## 九、何时启用 Continuous Profiling

**这一节给可操作建议**——什么场景会让你后悔"早该上"。

### 9.1 场景一:CPU / 内存毛刺定位

**症状**:服务 CPU 周期性飙到 100% / 内存周期性下降 1 GB / GC time 偶发涨——**指标看到"有",但找不到"为什么"**。

**Profile 价值**:**任意时刻拉历史 profile,差分看变化**——**直接定位到 GC pause 是因为某个对象分配率突增**。

### 9.2 场景二:微服务延迟回归

**症状**:发版后某接口 P99 从 200ms 涨到 400ms,trace 看是 payment-service 慢了,**但不知道为什么慢**。

**Profile 价值**:**对比发版前 / 后的 payment-service profile**——**差分火焰图直接显示新加的某个函数耗 CPU**。**5 分钟定位完版本回归**。

### 9.3 场景三:GC / 内存泄漏

**症状**:某 Java 服务每 24 小时 OOM 一次,**heap dump 太大 / 分析慢**。

**Profile 价值**:**heap profile 持续采集,看哪类对象的分配 / 占用持续增长**——**比 jstack / jmap 高效 10 倍**。

### 9.4 场景四:容量规划

**症状**:**业务要扩容预算,但不知道 CPU 到底用在哪**——**是核心业务还是 framework 开销**?

**Profile 价值**:**长期 profile 数据告诉你 70% CPU 烧在 framework / 20% 业务 / 10% 日志**——**优化方向清晰**。

## 十、何时不该上

**这一节给小团队 / 限制场景**——**以下情况,先放放**。

### 10.1 < 10 微服务,先打基础

**还没把 metrics / logs 做好,直接上 profile 是本末倒置**——**05-12 篇的前 7 件事都没做完,profile 不是优先级**。

### 10.2 PHP / Ruby 主栈

**Pyroscope / Parca 对 PHP / Ruby 的支持目前最弱**——eBPF 符号化困难、SDK 不成熟、JIT 行为复杂。**真有需求的话商业 SaaS(Datadog APM)可能更靠谱**。

### 10.3 内核 < 5.4 / 老 K8s

**eBPF 主要特性要内核 5.4+**,**老内核版本能用但功能打折**。**升级内核比上 profile 优先级高**。

### 10.4 团队无人懂火焰图

**Profile 数据采集了没人看,等于零**——**先培训人,再上工具**。**至少有一两个工程师能熟练读 / 用差分火焰图**,**否则系统装上就吃灰**。

## 十二、回到一开始

**Continuous Profiling 是可观测性的第四件套,也是 2024 之后崛起最快的一块**。

这一篇要给你留下的不是"Pyroscope / Parca 怎么装",**是 5 件事**:

1. **profile 是代码行级的可观测性**——metrics / logs / traces 都到不了这个粒度
2. **持续剖析 > 临时 pprof**——pprof 在生产现场跑已经太晚
3. **eBPF 让 profile 进入零侵入时代**——多语言异构 + 不改代码,1% CPU 开销
4. **差分火焰图是核心工具**——发版回归 / 故障对比 / 容量演化的杀手锏
5. **何时上要看团队成熟度**——< 10 微服务先打基础,PHP / Ruby 主栈先观望

> 经验法则:**100 微服务 + 多语言异构 + 经常发版**的中型团队,**Pyroscope + Grafana** 这套组合接进现有可观测栈,**1 个工程师 1-2 周能落地,3 个月后会有人在群里发"这个版本 CPU 涨 30% 是因为 X 函数"**——**这就是 profile 在团队里产生价值的时刻**。
