# 配置管理:Ansible / 配置中心 / Secret 管理 / Vault / SOPS

讲配置管理最大的两个误区:**一个是"我们上 K8s 了不需要 Ansible"**——错,K8s 之外的 VM / 数据库 / 网络设备 / 跳板机都还在,这些用什么管?**另一个是"配置就是 ConfigMap,Secret 就是 base64"**——错,K8s 的 Secret 是 base64 编码不是加密,kubectl 一拉出来就明文,这玩意进 Git 跟密码贴墙上没区别。

backendLearning/58 浅讲过 Helm Chart 里 values.yaml 怎么写,**那一篇是入门**。**这一篇专讲配置和 Secret 的工程化**——Ansible 在 K8s 时代的真实定位、配置中心(Nacos / Apollo / Consul / etcd)各自解决什么、Secret 管理的三条红线、Vault / SOPS / Sealed-Secrets / External-Secrets 的选型逻辑。**Secret 管理这一节请认真读三遍——这是 IaC 这一层最容易出大事故的地方**。

> 一句话先记住:**配置 ≠ Secret**。配置是"程序运行参数"(超时时间、批量大小、feature flag),Secret 是"凭证"(密码、token、API key)。**把 Secret 当配置管的团队迟早进新闻**——半年前某大厂把数据库密码塞 ConfigMap,被员工 kubectl get cm 看到截图发群,事故就是这么来的。**这一篇的核心红线:Secret 不进 Git / 不进日志 / 不进监控,三条任何一条破就是事故**。

## 一、Ansible 在 K8s 时代还有用吗

直接给答案:**有,且不可替代**。但定位变了——不再是"自动化运维主力",而是"管 K8s 之外的世界"。

### 1.1 Ansible 还在管什么

```
K8s 集群内部:                    Helm / Kustomize / ArgoCD —— Ansible 不碰
                                ↑
                                K8s 控制平面
                                ↓
K8s 集群本身的部署 / 升级:        K8s 集群本身往往是 Terraform 起的(EKS / GKE / AKS)
                                自建 K8s 才用 Ansible + kubespray

K8s 集群外的服务器:
  - 跳板机 / Bastion              ← Ansible
  - 自建 MySQL / PG / Redis        ← Ansible
  - 自建 Kafka / RabbitMQ          ← Ansible
  - 监控 agent 部署                ← Ansible
  - 安全基线配置                   ← Ansible
  - 网络设备(Cisco / 华为)        ← Ansible(配 NETCONF)
  - 跑批任务的物理机                ← Ansible
```

**经验**:**中型团队 80% 的工作负载在 K8s,但剩下 20% 的"集群外"才是配置管理的真战场**。这 20% 包含数据库、缓存、消息队列、跳板机、安全 agent——**这些机器没有 K8s 的 reconciliation,需要 Ansible 这种"远程批量执行 + 幂等"工具兜底**。

### 1.2 Ansible 心智:幂等 + Playbook + Role + Inventory

```
Inventory(目标主机清单):    inventory.ini / inventory.yaml
                             "我要操作哪些机器"

Playbook(任务列表):          playbook.yaml
                             "对这些机器,要做什么"

Role(可复用的任务包):        roles/nginx/
                             "把 nginx 安装+配置打包成一个 Role,可在多个 Playbook 用"

Module(原子操作):           apt / copy / template / service / shell
                             "Playbook 里实际执行的动作"

幂等(idempotent):           跑一遍跟跑十遍效果一样
                             Ansible 大部分 Module 自带幂等
                             用 shell module 时要自己保证
```

### 1.3 一份最小 Ansible Playbook

```yaml
# inventory.yaml
all:
  hosts:
    bastion:
      ansible_host: 10.0.1.10
      ansible_user: ubuntu
  children:
    db_servers:
      hosts:
        db1: { ansible_host: 10.0.2.10 }
        db2: { ansible_host: 10.0.2.11 }
      vars:
        db_role: replica
    web_servers:
      hosts:
        web1: { ansible_host: 10.0.3.10 }
        web2: { ansible_host: 10.0.3.11 }
```

```yaml
# playbook.yaml
- name: 配置 Web 服务器
  hosts: web_servers
  become: yes                         # sudo
  vars:
    app_version: "1.4.2"
    log_dir: /var/log/myapp

  tasks:
    - name: 安装依赖
      apt:
        name:
          - nginx
          - python3-pip
          - logrotate
        state: present
        update_cache: yes

    - name: 创建应用目录
      file:
        path: "{{ item }}"
        state: directory
        owner: app
        group: app
        mode: "0755"
      loop:
        - /opt/myapp
        - "{{ log_dir }}"

    - name: 拉取应用二进制
      get_url:
        url: "https://artifacts.acme.io/myapp/{{ app_version }}/myapp"
        dest: /opt/myapp/myapp
        mode: "0755"
        checksum: "sha256:{{ lookup('file', 'checksums/' + app_version + '.sha256') }}"
      notify: restart myapp

    - name: 写 nginx 配置(用 template)
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/sites-available/myapp
        validate: nginx -t -c %s       # 写之前先校验语法
      notify: reload nginx

    - name: 启用 site
      file:
        src: /etc/nginx/sites-available/myapp
        dest: /etc/nginx/sites-enabled/myapp
        state: link

    - name: nginx / myapp 开机自启 + 运行
      systemd:
        name: "{{ item }}"
        enabled: yes
        state: started
      loop: [nginx, myapp]

  handlers:
    - name: restart myapp
      systemd: { name: myapp, state: restarted }
    - name: reload nginx
      systemd: { name: nginx, state: reloaded }
```

**关键取舍**:

1. **`become: yes`** —— 整个 play 用 sudo,**不要在每个 task 里写**。但敏感任务用 `become_user` 收敛(比如 db 操作用 `postgres` user)
2. **`notify` + `handlers`** —— 配置改了才重启;只改一次 service 不会被重启多遍
3. **`template` 的 `validate`** —— **写前先校验,避免"配错了重启,服务起不来"**
4. **`checksum`** —— 拉二进制必带校验,避免供应链投毒
5. **`loop` 而不是裸 task 重复** —— Ansible 跑得快,逻辑清晰
6. **不用 `shell` / `command` module 除非真没办法** —— 这两个不幂等

### 1.4 Ansible 的真实痛点

1. **慢**——SSH 一台一台连,100 台机器跑下来要十几分钟。**用 `pipelining: true` + Mitogen 能加速但不是数量级**
2. **agent-less 的代价**——没有 daemon 持续 reconcile,跑完就走。状态漂移要再跑一遍 Playbook 才发现
3. **YAML 的固有问题**——逻辑复杂时(嵌套 loop / when / block)写起来痛苦
4. **跨版本兼容性**——Ansible 2.10 → 2.18 之间核心 module 改过行为,升级要小心
5. **Inventory 维护**——动态 inventory(dynamic-inventory 脚本)对接云 API,但维护是另一项工作
6. **不适合"长期收敛"**——Ansible 是"推送式",跑一次完事。要持续治理用 Puppet / Chef(但这俩在云原生时代式微了)

### 1.5 Ansible vs Chef / Puppet / Salt

```
Ansible:   Push,agent-less,SSH 推送,YAML 写 Playbook
Chef:      Pull,需要 agent,持续 reconcile,Ruby DSL
Puppet:    Pull,需要 agent,持续 reconcile,Puppet DSL
SaltStack: 双模,需要 master + minion,YAML/Jinja
```

**现状**:Ansible 是事实标准,**90% 的中型团队用 Ansible**。Chef / Puppet 在 K8s 之前更主流,**云原生时代被 Ansible(短期 push)+ K8s(长期 reconcile)替代**。SaltStack 在某些专用场景(网络设备、超大规模)还有人用。

> 经验:**新团队上 Ansible 就够,别折腾 Chef / Puppet**。要"持续 reconcile",K8s 已经做了。

## 二、配置中心:Nacos / Apollo / Consul / etcd 怎么选

讲完"机器级配置",回到"应用级配置"。

### 2.1 配置中心要解决的三个问题

```
问题 1:动态生效
  传统:配置写在 application.yaml,改了要重启服务
  痛点:大批量重启慢(100 个微服务全重启 10 分钟),业务有损
  解决:配置中心推送变更,应用监听并热更新

问题 2:多环境分离
  传统:application-dev.yaml / application-prod.yaml 同进 Git
  痛点:dev 的配置混进 prod,prod 的密钥进 dev 库
  解决:配置中心按 environment / namespace / cluster 维度组织

问题 3:灰度发布
  传统:同一个配置所有实例一起改
  痛点:配置错了所有实例一起挂
  解决:配置中心支持"按 IP / 按集群 / 按比例"灰度推送
```

### 2.2 四种主流选型

```
Nacos (阿里):
  - 配置 + 服务发现一体
  - Spring Cloud 生态原生支持
  - 国内文档好
  - 集群部署 + 自带 Raft 选主
  - 适合:Spring Cloud / Java 微服务为主的国内团队

Apollo (携程):
  - 老牌国产配置中心
  - 多语言 SDK 较全(Java / .NET / Python / Go)
  - 强环境隔离 + 灰度发布 + 审批流
  - 部署稍复杂(adminservice / configservice / portal 三组件)
  - 适合:对配置变更审批要求严的团队

Consul (HashiCorp):
  - KV 存储 + 服务发现 + DNS + 健康检查
  - 跟 Vault / Terraform 同生态
  - 多数据中心原生支持
  - 配置功能相对简单,没有 GUI 的"按环境/集群"概念
  - 适合:多语言 / 多 DC / 已经在用 HashiCorp 全家桶

etcd (CNCF):
  - 纯 KV,K8s 底层用的就是这个
  - 极简,只解决"分布式一致的 KV"
  - 配置中心功能要自己包(没有 UI)
  - 适合:自建配置中心的底层 / 已经熟 etcd
```

### 2.3 决策矩阵

| 场景 | 推荐 |
| --- | --- |
| Spring Cloud 微服务为主 | **Nacos** |
| 多语言 + 严审批 + 配置变更频繁 | **Apollo** |
| 已有 HashiCorp 生态(Vault / Terraform) | **Consul** |
| 想自建轻量 + 已熟 etcd | **etcd** |
| 只是 K8s 内部应用 | **ConfigMap + Reloader / Operator**(下面讲) |
| 团队 < 5 人 / 服务 < 20 个 | **不要上配置中心**——直接 ConfigMap 重启 |

### 2.4 K8s 时代:ConfigMap + Reloader

如果你的应用全在 K8s 里,**很多时候不需要专门配置中心**——K8s 本身有 ConfigMap,加个 Reloader / Operator 就能"配置热更新":

```yaml
# Deployment 标注让 Reloader 监听 ConfigMap
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  annotations:
    reloader.stakater.com/auto: "true"          # ConfigMap 变了自动 rollout
spec:
  template:
    spec:
      containers:
        - name: app
          envFrom:
            - configMapRef: { name: myapp-config }
```

**Reloader / Stakater Reloader** 是 K8s 一个开源 controller,**ConfigMap / Secret 变更自动触发 Deployment rollout**。**这套机制对中型团队足够,不一定要专门配置中心**。

**何时需要超出 ConfigMap 的配置中心**:

- 配置需要"按用户 / 按地区 / 按时间"动态切换(feature flag 类)→ 上专业 Feature Flag 工具(Unleash / LaunchDarkly,第 22 篇讲过)
- 配置变更要审批流 / 灰度推送 / 回滚 → Apollo / Nacos
- 跨 K8s 集群共享配置 → 配置中心
- 非 K8s 应用 → 配置中心(K8s ConfigMap 进不去 VM)

### 2.5 配置管理的反模式

**反模式 1:配置在代码里**

```java
// 写死在代码里
public static final String DB_URL = "jdbc:postgres://prod-db:5432/orders";
public static final int BATCH_SIZE = 100;
```

**问题**:改配置要重新编译部署,违反"配置 vs 代码分离"。

**反模式 2:配置在环境变量(且包含密码)**

```bash
DB_PASSWORD=Pa55w0rd! java -jar myapp.jar
```

**问题**:`ps -ef` / `/proc/<pid>/environ` 能看到环境变量。**容器场景下 `docker inspect` 也能看到**——任何能读容器元数据的人都能拿到密码。

**反模式 3:配置写死在 ConfigMap(且包含密码)**

```yaml
apiVersion: v1
kind: ConfigMap
metadata: { name: myapp }
data:
  DB_PASSWORD: "Pa55w0rd!"           # 灾难
```

**问题**:**ConfigMap 不加密,任何能 `kubectl get cm` 的人都能看**。Secret 也只是 base64 编码,**`kubectl get secret -o yaml | base64 -d` 一秒钟明文**。

**正确做法**:

- 配置(非敏感)→ ConfigMap / 配置中心
- Secret(敏感)→ Vault / SOPS / Sealed-Secrets / External-Secrets

下面专讲 Secret。

## 三、Secret 管理的三条红线

**这是 IaC / 配置这一层最容易出大事故的地方**。我把红线先写在最前面,后面所有工具选型都围绕这三条:

```
红线 1:Secret 不能进 Git
  - 直接提交 .env / values.yaml 含密码 → Git 历史永久泄密
  - 即使 force push 删了 commit,公网 fork / clone 已经传开
  - GitHub / GitLab 的 Secret 扫描器盯着,几小时内就告警

红线 2:Secret 不能进日志
  - 应用 logger.info("connecting with: " + db_password) → 日志系统永久存
  - HTTP 请求 / 响应 / Authorization header 不脱敏 → 日志泄密
  - kubectl describe pod 显示环境变量 → 任何能 describe 的人都能看

红线 3:Secret 不能进监控
  - Prometheus metric 把密码当 label 上报 → Grafana 一搜全有
  - APM trace 抓 SQL 把密码捕获 → trace 平台永久存
  - 报错 alert 带上完整请求体 → alertmanager 告警群里直接发密码
```

**这三条红线被破时,事故就是"密码泄露 + 紧急轮换 + 安全审计 + 用户通告 + 上新闻"**。我亲历过两次,**都是因为日志没脱敏 + ELK 公网暴露**——前后处理两个多月,内部 retrospective 写了几十页。

### 3.1 K8s Secret 的两个常见误解

**误解 1:K8s Secret 是加密的**

```bash
$ kubectl get secret db-creds -o yaml
data:
  password: UGE1NXcwcmQh             # base64 编码

$ echo "UGE1NXcwcmQh" | base64 -d
Pa55w0rd!
```

**事实**:**K8s Secret 是 base64 编码,不是加密**。base64 是编码格式,**任何能 `kubectl get secret` 的人都能 1 秒解码**。**Secret 进 Git 跟 ConfigMap 进 Git 一样危险**。

**误解 2:开了 etcd encryption 就安全了**

K8s 1.13+ 支持 `EncryptionConfiguration`,**让 etcd 里的 Secret 加密存储**。但这个加密只防"etcd 被直接 dump"的场景,**对 `kubectl get secret` 完全没用**——API server 解密后返回明文。

**真正的 Secret 安全**:**Secret 不该出现在 K8s 集群里(或者只用一次,马上 rotate)**——下面四种工具都是为此设计的。

## 四、Vault:动态密钥 + 短期 token + 自动 rotation

HashiCorp Vault 是"重运维但功能全"的 Secret 管理方案。

### 4.1 Vault 的核心心智

```
传统 Secret 管理:
  生成密钥(一次)→ 存某处 → 应用永远用这个密钥 → 半年后没人轮换

Vault 模型:
  应用启动 → 向 Vault 认证(K8s ServiceAccount / approle / aws iam)
            → Vault 临时签发 / 动态生成 一个短期密钥(15 分钟)
            → 用完丢弃,过期前续期
            → 离职 / 异常?revoke token 立即作废
```

**关键概念**:

- **Secret Engine**:Vault 的"功能模块",每个 engine 管一类 Secret
  - `kv` engine:简单 KV 存
  - `database` engine:**动态生成数据库账号**(每个应用拿一个临时账号,15 分钟过期)
  - `aws` engine:动态生成 AWS IAM 临时凭证
  - `pki` engine:签发短期证书
  - `transit` engine:加解密服务(Vault 不存数据,只做加解密)
- **Auth Method**:应用怎么向 Vault 证明身份
  - `kubernetes` auth:用 K8s ServiceAccount token 换 Vault token
  - `approle`:role_id + secret_id(给非 K8s 应用)
  - `aws iam`:用 IAM 角色换 Vault token
- **Policy**:谁能访问哪些 path 的什么操作(类似 IAM)
- **Lease**:每个 secret / token 都有过期时间,过期自动失效

### 4.2 一份最小 Vault + K8s Auth + 应用读取的流程

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  ┌──────────────┐   1. 启动时 mount ServiceAccount token         │
│  │  Pod         │ ────────────────────────────────────►          │
│  │  app         │                                                │
│  │              │   2. 用 SA token 向 Vault 认证                  │
│  │              │ ────────────────────────────────────►          │
│  │              │              ┌────────────────────────┐         │
│  │              │   3. Vault   │ Vault                  │         │
│  │              │   返回 token  │  - 验证 SA token 有效   │         │
│  │              │ ◄──────────  │  - 查 Policy(允许什么) │         │
│  │              │              │  - 颁发短期 client token│         │
│  │              │   4. 用 token │                        │         │
│  │              │   请求 db    │  database engine:      │         │
│  │              │   credentials│  - 动态生成 DB 账号     │         │
│  │              │ ────────────►│    (15 分钟过期)        │         │
│  │              │   5. 拿到    │  - 写入 PG/MySQL       │         │
│  │              │   user/pass  │  - 返回临时凭证         │         │
│  │              │ ◄──────────  └────────────────────────┘         │
│  │              │                                                │
│  │              │   6. 连数据库,业务跑                            │
│  │              │ ─────────────► RDS                              │
│  │              │                                                │
│  │              │   7. token 快过期了,自动 renew                  │
│  │              │ ──────────────► Vault                           │
│  │              │                                                │
│  └──────────────┘                                                │
└─────────────────────────────────────────────────────────────────┘
```

**实战用法**——通过 **Vault Agent Sidecar Injector** 自动完成 1-7:

```yaml
# Pod 上加这些 annotation
apiVersion: apps/v1
kind: Deployment
metadata:
  name: orders
spec:
  template:
    metadata:
      annotations:
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "orders-app"
        vault.hashicorp.com/agent-inject-secret-db-creds: "database/creds/orders-readonly"
        vault.hashicorp.com/agent-inject-template-db-creds: |
          {{- with secret "database/creds/orders-readonly" }}
          export DB_USER="{{ .Data.username }}"
          export DB_PASS="{{ .Data.password }}"
          {{- end }}
    spec:
      serviceAccountName: orders
      containers:
        - name: app
          # 应用启动前 sidecar 已经把凭证写到 /vault/secrets/db-creds
          command: ["/bin/sh"]
          args: ["-c", "source /vault/secrets/db-creds && /opt/myapp"]
```

**Vault Agent Sidecar 干的事**:

1. Pod 启动时,sidecar 用 K8s SA token 向 Vault 认证
2. 拉取 `database/creds/orders-readonly` 这个 secret(Vault 动态生成 PG 账号)
3. 把账号密码渲染成 `/vault/secrets/db-creds` 文件
4. **持续轮转**:lease 快过期时,生成新凭证 + 重新写文件 + 通知应用(reload)

### 4.3 Vault 的取舍

**好处**:

- **动态凭证**——每个 Pod 拿不同账号,出问题 revoke 一个不影响别人
- **自动 rotation**——15 分钟过期,无需人工轮换
- **集中审计**——Vault 记录每次 secret 访问
- **多 Engine**——一个 Vault 管 DB / AWS / PKI / encryption

**坏处**:

- **运维重**——Vault 集群本身要 HA / 备份 / 升级,**Vault 挂了所有应用拿不到 Secret 启动不了**
- **Vault 的 Unseal 流程繁琐**——重启后要"开锁",有 Shamir's Secret Sharing 这种安全机制但操作复杂
- **学习曲线陡**——Policy / Engine / Auth Method 一套概念
- **企业版才有的功能**——多租户 / 复制 / DR 是付费版
- **应用改造成本**——所有应用都要对接 Vault SDK 或者用 Sidecar

### 4.4 何时上 Vault

```
适合:
  - 团队 ≥ 30 人,服务 ≥ 50 个,Secret 管理已经混乱
  - 合规要求严(SOC2 / PCI-DSS / 等保三级以上)
  - 需要"动态 DB 账号"或"短期 AWS 凭证"等高级特性
  - 有专人运维(SRE / 平台团队 ≥ 1 个 FTE 投入)

不适合:
  - 团队 < 10 人,Secret 数量 < 50 个 —— SOPS 更轻量
  - 没有专人运维 Vault —— 挂了拖死所有服务
  - 只需要"K8s 里 Secret 加密存" —— Sealed-Secrets 就够
```

> 经验:**中型团队上 Vault 是奢侈品**。10 人 / 100 服务的团队,Vault 的运维成本和应用改造成本通常超过收益。**先用 SOPS / External-Secrets,真撞到痛点再升级 Vault**。

## 五、SOPS:加密 Git 里的 YAML

SOPS(Secrets OPerationS,Mozilla 出的)解决一个简单问题:**让 Secret 能"安全地"进 Git**——加密存,需要时解密。

### 5.1 SOPS 的工作流

```
开发者写 secret.yaml:
  db_password: Pa55w0rd!
  api_key: sk_live_xxxx

用 SOPS 加密(用 KMS / GPG / age key):
  sops --encrypt --in-place secret.yaml

加密后的 secret.yaml(可以进 Git):
  db_password: ENC[AES256_GCM,data:abc...,iv:def...,tag:ghi...,type:str]
  api_key:     ENC[AES256_GCM,data:jkl...,iv:mno...,tag:pqr...,type:str]
  sops:
    kms:
      - arn: arn:aws:kms:us-east-1:123:key/abc
        ...

部署时解密:
  sops --decrypt secret.yaml | kubectl apply -f -

或者 K8s 用 Helm Secrets / ArgoCD KSOPS / FluxCD SOPS integration 自动解密
```

### 5.2 SOPS 的关键特性

1. **细粒度加密**——只加密 value,key 还能看(`grep db_password` 还能搜得到)
2. **多 KMS 支持**——AWS KMS / GCP KMS / Azure Key Vault / Vault Transit / age / GPG
3. **多人共享**——加密时指定多个 KMS key,任何一个能用 KMS 解就行
4. **Git diff 友好**——加密文件能 diff,但内容看不出
5. **跟 GitOps 集成好**——ArgoCD / Flux 都有 SOPS plugin

### 5.3 一份最小 SOPS 配置

```yaml
# .sops.yaml(项目根目录)
creation_rules:
  - path_regex: secrets/dev/.*\.yaml$
    kms: arn:aws:kms:us-east-1:111:alias/sops-dev
  - path_regex: secrets/prod/.*\.yaml$
    kms: arn:aws:kms:us-east-1:222:alias/sops-prod
  - path_regex: .*\.yaml$
    encrypted_regex: '^(data|stringData)$'    # 只加密 data 字段
    kms: arn:aws:kms:us-east-1:111:alias/sops-default
```

```yaml
# 加密前 secrets/prod/db-creds.yaml
apiVersion: v1
kind: Secret
metadata: { name: db-creds, namespace: orders }
stringData:
  username: orders_user
  password: SuperSecret123!
```

```bash
sops --encrypt --in-place secrets/prod/db-creds.yaml
# stringData 字段下的值被加密,文件可以提交 Git
```

```yaml
# helmfile / ArgoCD 配置(用 KSOPS plugin)
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata: { name: orders }
spec:
  source:
    plugin:
      name: argocd-vault-plugin-kustomize-helm-sops    # 一个集成 plugin
```

### 5.4 SOPS 的取舍

**好处**:

- **极简**——一个 CLI 工具 + KMS,没有 server
- **Git 友好**——加密文件是文本,能 diff 能 merge
- **多人协作**——授权 KMS key 就行,不用 share 密码
- **跟 GitOps 流程天然契合**

**坏处**:

- **解密时机**——CI 解密 / 集群内解密各有 trade-off。CI 解密会把明文 secret 经过 CI 内存
- **轮换密钥要重新加密**——KMS key 轮转后所有 SOPS 文件要重新加密
- **不能动态生成**——SOPS 是"加密静态文件",不能像 Vault 那样动态生成 DB 账号
- **`sops --decrypt` 后明文在内存**——还是要小心日志 / 进程 dump

### 5.5 何时上 SOPS

```
适合(强烈推荐):
  - 中型团队(10-50 人),GitOps 文化
  - K8s 重度用户
  - Secret 数量适中(< 几百个),变更频率低
  - 想要"代码评审 Secret 变更"(SOPS 文件提 PR,评审通过才 merge)

不适合:
  - 需要动态生成 Secret —— Vault
  - Secret 太多 / 变更太频繁 —— External-Secrets
  - 团队不用 Git 当唯一来源 —— SOPS 价值打折
```

> 经验:**SOPS 是中型团队 Secret 管理的甜蜜区**。它把"Secret 当代码"管,但用 KMS 加密 + Git 流程兜底。**我个人首推**。

## 六、Sealed-Secrets vs External-Secrets vs SOPS:选型

**这三个是 K8s 时代 Secret 管理的三剑客**。我把它们摆一张表:

| 维度 | Sealed-Secrets | External-Secrets | SOPS(+ KSOPS) |
| --- | --- | --- | --- |
| **Secret 在哪** | 加密后存 Git(SealedSecret CRD) | 外部(Vault / AWS SM / GCP SM / Azure KV) | 加密后存 Git |
| **解密在哪** | 集群内 controller | 集群内 controller | CI 或集群内 plugin |
| **加密用什么 key** | 集群内私钥(controller 生成) | 外部 KMS(由 secret store 管) | 外部 KMS(AWS/GCP/age) |
| **动态更新** | 不行(手动改) | 自动(轮询外部 store) | 不行 |
| **运维复杂度** | 低(就一个 controller) | 中(controller + 外部 store) | 低(CI 集成) |
| **多集群** | 难(每个集群一份私钥) | 容易(共享外部 store) | 容易(共享 KMS) |
| **可读性** | 加密后乱码,看不懂 | Secret 引用,可读 | 加密后乱码,但 key 可见 |
| **代码评审** | 加密内容没法 review | 引用本身可 review | 字段可 review |
| **适合** | K8s 原生 / 单集群 / 简单 | 多 Secret 源 / 多集群 / 动态轮换 | Git 中心化 / 中型团队 |

### 6.1 Sealed-Secrets 详解

**心智**:你写一个 SealedSecret CR,**用 controller 的公钥加密**,提交到 Git。Controller 在集群内用私钥解密成普通 K8s Secret。

```bash
# 装 controller
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/...

# 加密
echo -n 'Pa55w0rd!' | kubectl create secret generic db --dry-run=client --from-file=password=/dev/stdin -o yaml \
  | kubeseal --controller-namespace=kube-system -o yaml > db-sealed.yaml

# 提交 Git
git add db-sealed.yaml && git commit
```

**痛点**:

- **私钥跟集群绑定**——换集群要重新加密所有 secret,或备份/迁移私钥
- **加密 secret 不可读**——`db-sealed.yaml` 里只有一坨密文,review 很难判断改对没
- **轮换 controller key 麻烦**——所有 SealedSecret 要重新生成

**适合**:**单集群 K8s + 想要 K8s 原生 + 简单需求**。多集群 / 跨环境就有点扭曲。

### 6.2 External-Secrets 详解

**心智**:Secret 不存 Git,**存外部 secret store(Vault / AWS SM / GCP SM / Azure KV / 1Password / ...)**。集群里只放"引用",controller 把引用同步成真实 Secret。

```yaml
# SecretStore 定义外部源
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata: { name: aws-sm, namespace: orders }
spec:
  provider:
    aws:
      service: SecretsManager
      region: us-east-1
      auth:
        jwt:
          serviceAccountRef: { name: external-secrets-sa }   # IRSA

---
# ExternalSecret 引用
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata: { name: orders-db, namespace: orders }
spec:
  refreshInterval: 1h          # 每小时同步一次
  secretStoreRef:
    kind: SecretStore
    name: aws-sm
  target:
    name: orders-db            # 生成的 K8s Secret 名字
  data:
    - secretKey: username
      remoteRef: { key: prod/orders/db-creds, property: username }
    - secretKey: password
      remoteRef: { key: prod/orders/db-creds, property: password }
```

**优势**:

- **Git 里只有引用**——Secret 本身在外部 store,**Git 历史里永远没有密钥**
- **自动 rotation**——外部 store 改了,1 小时内集群同步
- **多 Secret 源**——一个 controller 对接 Vault / AWS SM / Azure KV
- **跟 IAM 集成好**——AWS 上用 IRSA(IAM Roles for Service Accounts)

**痛点**:

- **依赖外部 store**——AWS SM 限速、Vault 挂了,集群里 Secret 不更新
- **冷启动慢**——新建 ExternalSecret 后,要等 controller 同步一次才能用
- **审计窗口**——Secret 改了到集群同步有延迟(默认 1 小时)
- **External-Secrets 本身 controller 是高权限**——能读所有 store 的 Secret,这个 controller 的 RBAC 要严

**适合**:**多集群 / 多 Secret 源 / 需要外部 store 集中管理 / 跟云 KMS / Vault 已经投资过**。

### 6.3 选型决策

```
开始用 K8s,Secret 不多,单集群        → Sealed-Secrets
中型团队,Git 是源,审计严              → SOPS + KSOPS
多集群 / 已有 Vault / AWS Secrets Mgr  → External-Secrets
团队 ≥ 30 人,要动态 DB 账号 + rotation → Vault + External-Secrets 组合
```

**混用也常见**:**Vault 存"真正动态的 Secret"(DB 账号、AWS 凭证),SOPS 存"静态但敏感的配置"(API key、第三方 token)**——根据 Secret 性质分流。

## 七、6 个 Secret 管理坑

实战里几乎每个团队都会撞这些坑:

### 7.1 不区分 dev / prod

```
反模式:
  所有环境用同一个 Vault / 同一个 KMS key / 同一份 SOPS 文件
  dev 工程师能看 prod 密钥
  → 实习生本机泄露 prod 密钥
```

**修复**:**不同环境完全独立**——dev / staging / prod 各自的 KMS key、各自的 Vault namespace、各自的 IAM 权限。**SOPS `.sops.yaml` 按 path 路由到不同 KMS**(前面的示例那样)。

### 7.2 同一个 Vault token 用半年

```
反模式:
  应用配置写死一个 Vault token,半年没换
  这个 token 权限大(能读所有 Secret),被偷直接拿走全部
```

**修复**:

- **不用静态 token**,用 K8s SA 认证 / approle / aws iam 这种"绑身份"的方式
- 必须用 token 的场景(CI 等),**token TTL 短(< 1 小时)**,使用前现申请
- **定期 rotation**——所有静态 secret 至少每 90 天换一次

### 7.3 Secret 写进容器层(layer cache 泄露)

```dockerfile
# 反模式
FROM alpine
COPY .env /app/.env             # .env 含密码
ENV DB_PASSWORD=Pa55w0rd!       # ENV 也是层,镜像 dump 出来能看到
RUN curl -u admin:secret ...    # secret 进 layer history
```

**修复**:

- **build 阶段需要 secret(私有 registry / npm token)用 BuildKit `--mount=type=secret`**——不进 layer
- **运行时 secret 用 K8s Secret + envFrom / volumeMount 注入**,不进镜像
- **镜像扫描器**(trivy / grype / docker scout)能扫出"镜像层带密码"的问题

### 7.4 kubectl get secret 出来是 base64(不是加密)

前面说过——**base64 ≠ 加密**。常见误解的衍生:

- "Secret 进 etcd 是加密的"——除非主动开了 EncryptionConfiguration,默认 etcd 里就是 base64
- "RBAC 限制 `kubectl get secret`"——某个 namespace 内的所有 Pod **默认都能 mount 该 namespace 下的 Secret**,namespace 隔离比 RBAC 更关键

**修复**:

- 开启 `EncryptionConfiguration`,etcd 落盘加密
- **namespace 严格隔离**——不要把所有应用塞一个 namespace
- 用 External-Secrets / Vault 让真正的 Secret 不进集群

### 7.5 备份策略缺失

```
反模式:
  Vault 跑得好好的,从来没备份过
  某天 Vault Raft leader 挂了 + 备份不可用
  → 所有应用拿不到 Secret 启动不了
  → 全公司应用大规模故障
```

**修复**:

- **Vault**:每天 snapshot 到 S3,异地容灾,**演练过 restore**
- **AWS Secrets Manager**:开 cross-region 复制
- **SOPS**:Git 本身就是备份,但 **KMS key 也要备份**(KMS key 删了所有 SOPS 文件解不开)
- **K8s Secret**:Velero / k8up 备份 etcd

### 7.6 Rotation 不自动

```
反模式:
  公司密码策略:每 90 天必须 rotation
  实际:工程师手动改,改完通知所有应用重启
  半年后:大家都嫌烦,密码改了 1 年没动
```

**修复**:

- **Vault 动态 DB 账号**——15 分钟自动 rotation,工程师不用管
- **AWS Secrets Manager 自动 rotation**——配 Lambda,定期换 RDS 密码
- **External-Secrets + refreshInterval**——外部 store 改了,集群自动同步,无需手动通知
- **不能自动 rotation 的(第三方 API key 等)**——上 Vault 或者 AWS SM 至少集中管理,提醒到期

## 八、配置和 Secret 管理 checklist

把这份贴到团队 onboarding 文档:

### 配置

- [ ] **配置不在代码里**——所有可变参数走 ConfigMap / 配置中心
- [ ] **环境变量不存密码**——`ps`、`docker inspect`、`/proc` 都能看
- [ ] **ConfigMap 不存密码**——任何 `kubectl get cm` 的人都能看
- [ ] 配置中心选型基于团队规模和实际痛点,**不为了"高大上"上 Apollo / Nacos**
- [ ] **K8s 内应用优先 ConfigMap + Reloader**,够用就不上专门配置中心
- [ ] 配置变更走 PR / 审批(配置中心带审批流的话)

### Secret —— 红线

- [ ] **Secret 不进 Git**(明文)——任何形式
- [ ] **Secret 不进日志**——脱敏 `password / token / authorization` 字段
- [ ] **Secret 不进监控**——Prometheus label / APM trace / alert 内容都要查
- [ ] **Git push 前扫**(git-secrets / gitleaks / detect-secrets)
- [ ] **Secret 历史泄露立即 rotate**——不要"先压热度再说"

### Secret —— 工具

- [ ] **不用裸 K8s Secret**——base64 不是加密,选 Sealed / External / Vault 中一个
- [ ] **K8s 单集群简单需求 → Sealed-Secrets**
- [ ] **中型团队 GitOps → SOPS**
- [ ] **多集群 / 多 Secret 源 → External-Secrets**
- [ ] **动态 DB 账号 / 短期凭证 → Vault**
- [ ] dev / staging / prod **完全独立的 KMS / Vault namespace / Secret store**
- [ ] **静态 token 不超过 90 天**,优先用绑身份的认证

### 应用接入

- [ ] **应用读 Secret 用文件 / Vault Agent**,不用环境变量
- [ ] **Secret 变更不重启应用也能更新**(Vault Agent template / Reloader)
- [ ] **应用日志框架配置脱敏**(常见 password / token 字段)
- [ ] **HTTP middleware 脱敏**(Authorization / Cookie / Set-Cookie)
- [ ] **error / exception 不抛出 Secret**(SQL 报错带连接串 → 脱敏)

### 运维

- [ ] **Secret store 备份**——Vault snapshot / AWS SM 复制 / SOPS KMS key 备份
- [ ] **演练 restore**——至少每季度一次,只备份不演练等于没备份
- [ ] **审计日志保留**——谁在什么时候读了什么 Secret,保 1 年以上
- [ ] **Secret rotation 自动化**——能自动的别让人手动

## 九、踩坑提醒

1. **以为"K8s Secret 是加密的"**——base64 不是加密,kubectl get 一秒明文
2. **Secret 提交到 Git**(即使 force push 删了)——Git 历史已经 fork,GitHub 扫描器已经告警
3. **生产 Secret 在 ConfigMap**——比红线 1 更不可饶恕
4. **环境变量存密码**——`ps aux` / `/proc/<pid>/environ` 都能看
5. **Dockerfile 把 Secret 烤进 layer**——镜像 dump 出来历史层带明文
6. **dev / prod 用同一份 Secret**——dev 工程师本机泄露 = prod 泄露
7. **静态 Vault token / API key 用半年**——一次泄露全盘皆输
8. **不脱敏日志的 password / authorization**——日志系统永久留密
9. **报错把连接串(含密码)抛上去**——APM / Sentry / 监控都有了
10. **Sealed-Secrets 跨集群迁移没规划**——controller 私钥换了,所有 SealedSecret 重新加密
11. **SOPS KMS key 删了**——所有 SOPS 文件解不开,Secret 永久丢失
12. **External-Secrets 的 controller RBAC 太宽**——一个 controller 被攻破 = 所有 Secret 泄露
13. **Vault 没备份**——某天 Vault 集群挂了,所有应用拿不到 Secret 启动失败
14. **Ansible Playbook 把密码硬编码**——`ansible-vault` 都没用就提交了
15. **Ansible 的 `shell` module 滥用**——非幂等,跑两遍效果不同

## 十、小结:配置和 Secret 是两件事

回到开篇:**配置 ≠ Secret**。

**配置层**:

1. **机器级配置**:Ansible 管 K8s 之外的所有机器
2. **应用级配置**:ConfigMap + Reloader 起步,复杂需求上 Nacos / Apollo
3. **配置中心解决三个问题**:动态生效 / 多环境分离 / 灰度发布
4. **K8s 内应用不一定要专门配置中心**——ConfigMap 重启就够

**Secret 层**:

1. **三条红线**:不进 Git / 不进日志 / 不进监控,**这三条任何一条破就是事故**
2. **K8s Secret base64 不是加密**——必须配合 Sealed / External / Vault / SOPS
3. **工具选型**:
   - Sealed-Secrets:单集群 K8s 原生
   - SOPS:中型团队 Git 中心化
   - External-Secrets:多集群 / 多 Secret 源
   - Vault:动态密钥 / 短期凭证 / 严合规
4. **环境隔离**:dev / staging / prod 完全独立的 KMS / Vault / Store
5. **Rotation 自动化**:静态 token 超过 90 天就是定时炸弹

**这一层做不好,IaC 这层做得再漂亮也白搭**——基础设施代码版本化、自动化、Drift 治理全做了,**结果数据库密码塞 ConfigMap 提进 Git**——一次事故抹平半年的工程努力。

**配置和 Secret 管理是 SRE / 平台团队的基本功**。**新人入职第一天就该明白这三条红线**。
