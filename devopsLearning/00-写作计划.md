# DevOps / SRE 系列写作计划

> 来源:`未来系列规划.md` 第三类「纵深主题」第八项(H),⭐⭐⭐⭐。
> backendLearning 用 4 篇浅讲了「监控 / 日志 / CICD / SRE 实践」(33 / 34 / 40 / 59),networkLearning 给了「反代 / 排障方法论」,osLearning 给了「性能工具 / eBPF」——**这个系列把这些散点接成一张完整的"上线之后"地图**:让系统能跑、能看见、能修、能复盘。
> **看完这层,你不再只是"把代码合到 main 上"的工程师**,而是能定 SLO、能写 Runbook、能扛 On-call、能在凌晨三点指挥事故响应的「**对生产系统负责的人**」。

---

## 设计思路

六层结构,从「文化与心智」到「真刀真枪的生产实战」:

```
DevOps / SRE 心智(开发 vs 运维 vs SRE / Toil / 错误预算)
   → 可观测性三件套(Metrics / Logs / Traces / Profiles)
   → SLO 与告警工程(SLI 选择 / 告警分级 / 降噪)
   → CI/CD 与发布工程(GitOps / 渐进发布 / Feature Flag)
   → IaC 与配置管理(Terraform / Ansible / 配置漂移)
   → 生产实战(On-call / 混沌 / 容量 / 复盘 / FinOps)
```

**这个系列不是 Google SRE Book 翻译**——SRE Book 偏理念,**本系列偏"中型团队工程师视角"**:每篇都对应"你这周可能就要做的一件事"——配 Prometheus 抓 JVM、写一份 Runbook、给灰度发布加门禁、扛一次半夜告警、主持一次复盘。

不深入到「Google 全球流量调度」「字节级硬件运维」这种极端规模,**而是讲清楚"100-10000 QPS 的真实业务系统怎么稳"**——你写 Java / Go / Node,但代码合上去之后还有一半工作量没做。**这个系列就是那"另一半"。**

**写作主线**:每一篇都回答同一个问题——「这件事不做会怎样,做了能多扛多少事故」。不讲"业界最佳实践"这种空话,只讲"我们团队现在 10 个人 / 100 个微服务 / 5000 QPS,这件事该不该做、做到什么程度"。

**一条贯穿全系列的暗线**:三个数字——「**MTTR(故障恢复时间) / MTBF(故障间隔时间) / Change Failure Rate(发布失败率)**」。第二层让 MTTR 降下来,第三层让告警足够早地预警 MTBF 在变短,第四层让 Change Failure Rate 可控,第六层用复盘把这三个数字闭环。读者每读完一层都该回头看一眼:**我把这三个数字往哪个方向推了**?

## 写作模板

每篇遵守"问题场景 → 工具是什么 → 怎么落地 → 何时不该用"的四段式:

```
1. 问题场景
   - 没这套东西时,生产出过什么事故
   - 团队规模到多大、流量到多少会撞上

2. 工具 / 方法是什么
   - 核心概念 + 一张架构图
   - 别家的同类方案怎么比

3. 怎么落地
   - 一段最小可用的配置 / 代码
   - 接入到现有 K8s / 微服务的关键步骤
   - 避坑清单(5-10 条踩过的坑)

4. 何时不该用 / 边界
   - 小团队 / 低流量场景该不该上
   - 长期维护成本是什么
   - 不要把"工具"当成"文化"的替代品
```

不堆 yaml,**只贴关键的 20-50 行配置片段**,让读者「看完就能照着改自己的环境」就够了。

**举一个反例**:如果一篇文章打开是 200 行 `prometheus.yaml`、然后一句"按需修改",这就是写错了。读者要的是「这条 `relabel_configs` 为什么必须加」「这个 `scrape_interval` 设 30s 而不是 15s 的取舍」「为什么我们最终把 `alerting.alertmanagers` 单独抽出来」——是**取舍的理由**,不是配置的字面值。

## 节奏

```
01-04  心智:看完知道"SRE 不是高级运维,是用工程方法做可靠性"
05-12  可观测性:**最值钱的一层**,90% 的事故先在这里被发现
13-17  SLO:**最反直觉**,讲完你才知道告警为什么一直在吵 / 一直在漏
18-23  CICD:**最贴日常**,每周都在用,但 80% 团队的发布流程都不及格
24-27  IaC:**最容易半途而废**,讲清楚状态文件 / drift 才能真上手
28-34  生产实战:**面试和真实工作的硬通货**,On-call / 复盘 / 容量 / FinOps
```

**一个硬指标**:看完 05-12 + 13-17 + 28-34 这 20 篇,你应该能在白板前讲清楚:

- "为什么你 dashboard 上的 p99 是骗人的,histogram_quantile 该怎么算"
- "为什么 99.9% 一个月只有 43 分钟错误预算,这 43 分钟该花在哪"
- "为什么告警按静态阈值配一定会爆炸,多窗口多燃烧率是怎么救命的"
- "凌晨三点收到 P0,前 5 分钟你做什么、前 30 分钟你做什么"
- "事故复盘上有人说'这是 XX 的锅',你怎么把这场会救回来"
- "团队只有 5 个人,该不该上 ArgoCD / Terraform / ChaosMesh,边界在哪"
- "成本同比涨了 40%,FinOps 该从哪个指标先看"

能讲清楚一题这个系列就值了。

**第二个硬指标**:看完整套之后,你应该能独立写出团队的:

- 一份 SLO 文档(SLI 选择 + 错误预算 + 政策)
- 一份事故响应手册(角色 + 升级路径 + 沟通模板)
- 一份发布门禁清单(CI 必须过的检查 + 发布灰度策略)
- 一份 On-call 手册(轮值 + 告警分级 + 升级规则)

这四份文档不是模板填空——是你**对自己团队的工程承诺**。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「装 Prometheus 三步走」——不是工具教程,是工程视角
2. 「先讲个 Google 全球故障故事」——直接进重点
3. 罗列 yaml 配置——只贴关键 20-50 行,完整配置丢仓库
4. 「DevOps 是一种文化」式的口号——文化也要能落到流程和工具上
5. 抄 Google SRE Book / 《SRE 工作手册》——那两本书在那儿,这里只补它们对中型团队没讲透的工程细节
6. 不画图就讲 GitOps / 链路追踪 / 错误预算燃烧率——这些**不画图永远讲不清楚**
7. 「上 K8s 就解决一切」——K8s 只是底座,可观测性 / SLO / 发布 / On-call 一样都不能少
8. 鼓吹"全自动化"——Toil 要减,但盲目自动化只会造出新的 Toil
9. 「这是面试题」——读者来这里是为了真懂,不是背关键词
10. 把工具厂商的市场材料当结论——Datadog / New Relic 的白皮书是广告,不是工程经验
11. 一刀切建议——任何"必须 / 一定 / 永远不要",都要给出适用条件
12. 不画 ASCII 图就讲发布流程 / 告警链路 / 事故响应角色——这些**不画图永远讲不清楚**(再说一遍)

## 立场声明

本系列**只服务于"对生产系统负责"的工程师视角**——SRE 不是替开发背锅的运维,而是用软件工程方法把可靠性做成可度量、可管理、可改进的产品属性。

所有示例配置以 **K8s + Prometheus + Loki + OTel + ArgoCD + Terraform** 这套主流栈为默认假设,但每一篇都会标注:换成 VM 部署 / 别的监控栈 / 别的发布工具时,哪些结论依然成立、哪些需要调整。**工具会换,方法不会。**

最后:这个系列**不替代**你团队的真实事故、真实告警、真实压测。它只是给你一张地图,告诉你哪里有坑、哪里别人已经踩过。**真正的工程能力,只能在生产系统上长出来。**

读完整个系列之后,你应该有一种新的本能:看到任何一个新服务上线,会下意识地问——「它的 SLI 是什么 / 错误预算给多少 / 告警怎么配 / 灰度怎么发 / 出事谁 On-call / 复盘什么时候开」。**这套问题就是这个系列要教会你的。**
