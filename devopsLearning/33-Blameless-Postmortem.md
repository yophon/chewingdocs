# Blameless Postmortem:无指责复盘 / 5 Whys / 行动项跟踪 / 文化建设

事故在 32 篇里已经修完了。**真正的工程功夫,从这一篇开始**。

修事故是"让损失停下",**复盘是"让同一类事故再也不发生"**。前者考验团队当下的技术与组织协同,后者考验团队**有没有从事故里学到东西的能力**。一个团队能不能扛十年,99% 取决于复盘做没做好——技术债可以填,人员可以招,但"同一个事故第三次出现"的团队基本无救。

中型团队的复盘十有八九开成两种灾难:**批斗大会**(找一个替罪羊)或**走过场会**(总结几句"加强意识"散场)。两种结局的本质是同一个——**没有人愿意把真相摆上桌**。批斗会让大家学会隐瞒,走过场会让大家学会糊弄,**这两条路通向同一个地狱:下次事故时,你还是看不到真相**。

Blameless(无指责)复盘是 SRE 给出的工程答案——**不是道德口号,是一套让人敢说真话的机制**。这一篇讲怎么把这套机制在中型团队里落地,以及最重要的:**Blameless 不是"算了不追究",它有自己更硬的纪律**。

> 一句话先记住:**Blameless 复盘的真意不是"不追责",是"把'某个人犯了错'这种没法修的结论,翻译成'某个系统漏了一道防线'这种可以修的结论"**。一个工程师误删生产表,blameful 会问"为什么他乱删";blameless 会问"**为什么生产权限给了他 / 为什么删表没有二次确认 / 为什么没有回收站机制**"——同一个事故,前者只能开除一个人换一个人继续踩坑,后者修完之后**这类事故永远不会再发生**。

## 一、问题场景:复盘最容易开成什么样

### 1.1 批斗大会版

```
会议室,10 个人,出事的服务 owner 小张坐在最前排,脸色发白。

老板:这次 P0 事故,我们要彻底查清楚责任。小张你来说说怎么回事?

小张:那个......我在改配置的时候......

老板:你改了什么?为什么要改?为什么没有人 review?

DBA:对啊我也想问,我们流程里明明要 review 的。

小张:那天上线赶时间,组长说先发了再补流程......

组长:我什么时候说的?

小张:周二下午站会上你说的......

老板:行了行了,这种小问题别推卸责任。小张,你这个月的绩效要扣。
      大家以后都要"加强意识","严格遵守流程"。散会。

   ↓
30 天后
   ↓

小张离职。
新人小李接手该服务。
3 个月后,同一个配置坑,小李又踩了一次。
```

这是典型的**批斗会**。出现频率:**80% 的中国团队**。后果:

- 工程师学会**隐瞒**——下次小问题能压就压,不报告
- **离职率上升**——背锅没人愿意,核心成员先撤
- **流程文档膨胀**——所有人都开始 cover 自己的屁股,加各种没人执行的"双重 review"
- **事故循环**——根本问题(配置无 review 机制 / 上线赶时间的压力 / 流程靠口头)没动,3 个月后必然复发

### 1.2 走过场版

```
会议室,5 个人,大家用电脑做自己的事。

主持人:这次事故大家说一下感想。

A:我们以后要加强测试。
B:我们以后要加强监控。
C:我们以后要加强代码 review。
D:我们以后要加强意识。
主持人:好,这些都记下来,我整理一份 Action Items。散会。
```

**Action Items 在事后文档里 20 条**——半年后回顾,**0 条完成**。3 个月后同一事故再来。

### 1.3 为什么这两种都失败

```
批斗会:    找到一个"该负责的人" → 处理这个人 → 系统漏洞还在
走过场:    列了一堆"加强 X" → 没人执行 → 系统漏洞还在
Blameless: 找到漏掉的那道防线 → 派人修这道防线 → 系统漏洞被堵

   ↑                                                  ↑
   人是变量,事故必然复发                      系统是常量,事故被根治
```

**关键洞察**:**事故的根因永远是系统问题,不是人的问题**——因为人会犯错是一个客观存在的、不可消除的事实。**一个会因为"某个工程师手抖"而塌的系统,是这个系统设计错了,不是这个工程师错了**。

## 二、Blameless 的真意:不是不追责,是换一种追法

### 2.1 别误解 Blameless

很多人第一次听 Blameless 会以为是"出事不追责"——这是最大的误解。**Blameless 不弱化责任,它把责任的对象从"人的行为"换成"系统的漏洞"**:

| 维度 | Blameful 复盘 | Blameless 复盘 |
| --- | --- | --- |
| 主语 | 谁犯的错 | 哪道防线漏了 |
| 问题 | 为什么他这么做 | 为什么系统允许他这么做 |
| 结论 | 处罚他 / 训练他 | 改流程 / 加防线 / 改工具 |
| 行动 | 警告 / 扣绩效 | 工程 ticket |
| 效果 | 这个人下次小心,但系统还在 | 这类事故无论谁来值都不会再有 |
| 文化代价 | 隐瞒、互相甩锅 | 心理安全、敢报告问题 |

**注意右下角那一格"文化代价"**——这是 Blameless 最被低估的价值。**有指责文化下,工程师学会的不是"小心",是"低调"**:

- 小问题能压就压
- 出事先想怎么甩锅
- 别人的服务出事自己别评论
- 关键决策走口头不留邮件

**这种团队的事故响应速度永远是慢的**——因为问题一直在地下烂,**等到它浮出水面就已经是 P0**。

### 2.2 一个真实例子:工程师 drop table 生产库

```
2 月凌晨 1 点,工程师小张在排查问题时,误把生产库当成测试库,
执行了 DROP TABLE orders;
24 万条订单数据消失。

╔══════════════════════════════════════════════════╗
║ Blameful 的复盘 (错误示范)                       ║
╠══════════════════════════════════════════════════╣
║ 问题: 为什么小张乱删生产数据?                    ║
║ 结论: 小张操作失误,警告 + 扣绩效                ║
║ 改进: 大家以后操作生产库要"加强意识"             ║
║                                                  ║
║ 一个月后:小张离职,新人小王接班                  ║
║ 三个月后:小王在排查问题时连错环境,又 drop 一次  ║
╚══════════════════════════════════════════════════╝

╔══════════════════════════════════════════════════╗
║ Blameless 的复盘 (正确示范)                      ║
╠══════════════════════════════════════════════════╣
║ 问题 1: 为什么生产库的 DROP 权限给了一个普通研发? ║
║ 问题 2: 为什么生产和测试的连接配置长得这么像?     ║
║ 问题 3: 为什么 DROP 操作不需要二次确认?           ║
║ 问题 4: 为什么没有"五分钟回收站"?(soft delete)  ║
║ 问题 5: 为什么生产终端的提示符不显眼?             ║
║                                                  ║
║ 5 个工程 ticket,2 个月内全部修完                ║
║ 6 个月后:相似手抖事件 0 次                      ║
╚══════════════════════════════════════════════════╝
```

**两次复盘指向同一个事故,但下一次同类事故的概率完全不同**。

### 2.3 Blameless 的硬纪律

Blameless 不是软文化,它有几条**比 Blameful 更硬的纪律**:

1. **必须找根因**——不允许停在"加强意识"这种结论
2. **必须出 Action Items**——每条有 owner / 有 due / 可验证
3. **必须 follow up**——下次复盘要回顾上次 Action 的完成情况
4. **重复事故 = 复盘失败**——同类事故重复发生,不是责怪当事人,是责怪上次复盘的人

**最后一条最关键**:Blameless 不是"对人的失误网开一面",**是把追责的对象从"操作者"换成"上次复盘的责任人"**。如果你上次复盘草草收场,下次同类事故时**你才是要被问的人**。

## 三、5 Whys 实战:挖到哪一层才算到底

### 3.1 5 Whys 的基本方法

5 Whys 不是"问 5 个为什么",是**通过连续追问把表层症状一路挖到根本原因**:

```
表层症状 ── 为什么? ──→ 直接原因
              │
              ↓ 为什么?
          二层原因
              │
              ↓ 为什么?
          三层原因
              │
              ↓ 为什么? ......
              ↓
          根因(系统 / 流程 / 文化)
```

数字"5"是经验值,**不是教条**——可能 3 层就够,可能挖到第 7 层。**判断"挖够了"的标准**:再问一次"为什么",答案落到"系统设计 / 流程缺陷 / 文化问题"这一层,**而不是"某个人决定 X"**。

### 3.2 5 Whys 怎么挖才不跑偏

**反例**(挖偏了的方向):

```
事故:订单接口错误率飙升
W1: 为什么? → 数据库 CPU 跑满
W2: 为什么? → 一个慢查询全表扫
W3: 为什么? → 缺索引
W4: 为什么? → 上线时索引没加
W5: 为什么? → 工程师没注意
结论:训练工程师 / 加强 review

    ↑
    挖到 W4 已经到了"系统漏洞" (上线流程没拦索引)
    W5 又退回到"人的问题",前功尽弃
```

**正例**(挖对的方向):

```
事故:订单接口错误率飙升
W1: 为什么 → 数据库 CPU 跑满
W2: 为什么 → 一个慢查询全表扫
W3: 为什么 → 缺索引
W4: 为什么 → 上线时索引没加
W5: 为什么 → CI 没有"DDL 必须有索引验证"的检查
W6: 为什么 → 团队没有"线上 SQL 必须经过 explain 验证"的流程

根因:CI / CD 缺少 SQL 性能门禁
行动:
   1. 在 CI 加入 sql-lint 检查 (owner: 平台组, due: 2 周)
   2. 加 explain 自动跑 + p99 >100ms 阻断 (owner: DBA, due: 4 周)
   3. 上线前必须附 explain 结果到 PR description (owner: 平台组, due: 1 周)
```

**判断挖对没挖对的 3 条小测试**:

```
测试 1: 替换主语测试
   "把当事人换成另一个工程师,这事还会发生吗?"
   如果答案是"会"——挖对了(系统问题)
   如果答案是"不会"——挖偏了,要继续往下

测试 2: 时间测试
   "如果这个问题一年前问,答案一样吗?"
   "为什么这一年里这个漏洞一直存在,直到今天才暴露?"
   如果答案揭示了一个长期存在的盲点,挖对了

测试 3: 工程 ticket 测试
   "这个根因能不能写成一个具体的工程任务?"
   "改 X 工具,加 Y 流程,做 Z 校验"——可以 → 挖对
   "下次大家小心 / 加强意识"——不行 → 还要挖
```

### 3.3 不要变成"为什么这个工程师手抖"

5 Whys 最容易跑偏的地方:**第一层就指向人**。

```
错误的起手:
W1: 为什么数据库被 drop? → 因为小张手抖
W2: 为什么小张会手抖? → 因为他没看清楚环境
W3: 为什么他没看清楚? → 因为他疲劳
W4: 为什么他疲劳? → 因为他加班
W5: 为什么他加班? → 因为项目排期紧
结论:不要让工程师加班

   ↑
   一路都在挖"人为什么犯错",不是挖"系统为什么允许犯错"
```

**正确的起手**:

```
W1: 为什么数据库被 drop?
    把这个 Why 拆成两支:
    A: 为什么小张能执行 DROP TABLE? (权限问题)
    B: 为什么 DROP TABLE 没有保护机制? (流程 / 工具问题)
    C: 为什么 DROP 之后没法快速恢复? (备份 / 回滚问题)

每一支独立挖,得到 3 个根因,3 个工程 Action
```

**5 Whys 不是单线挖,是多线挖** —— 一个事故经常有多个独立漏洞,**每一个漏洞都该被挖到底,每一个漏洞都该有自己的修复行动**。

## 四、复盘报告模板:一份能直接抄的格式

### 4.1 标准模板

```
=============================================
事故复盘报告
=============================================

# 标题
[一句话描述 + 严重度]
例: 订单接口错误率飙升 (P0)

# Summary (摘要)
- 起止时间: YYYY-MM-DD HH:MM ~ HH:MM (持续 29 min)
- 严重度:   P0
- 影响:    估算 12,400 笔订单失败,5,200 用户感知
- 根因:    数据库迁移误删关键索引
- 修复:    回滚发布到 v2.4.6

# Timeline (时间线)
[摘自事故时间线,只记事实]
03:01 - 错误率告警触发
03:02 - 小王成为 IC
...

# Impact (影响)
- 业务影响: 失败订单 12,400 笔,GMV 损失 ~80 万
- SLO 影响: 错误预算消耗 85% (本月)
- 客户影响: 大客户 A / B 各有 1 次客诉
- 公关影响: 无 (Status Page 5 分钟内更新,无媒体)

# What Went Well (做对了什么)
- IC 制度运转,29 分钟内恢复
- Status Page 同步及时,客服话术统一
- 战时频道纪律好,无刷屏

# What Went Wrong (哪里出问题)
- migration 删除关键索引前未做性能评估
- CI 缺少 DDL 安全检查
- DBA 未参与该次发布的 review

# Root Cause Analysis (5 Whys 分析)
[多线挖,见上一节]

主线 A: 为什么删了索引能上线
   W1 → W2 → W3 → W4 → W5
   根因: CI 缺 SQL 安全检查

主线 B: 为什么没人 review
   W1 → W2 → W3
   根因: 流程里 SQL 变更没有 DBA 强制 review

主线 C: 为什么影响这么大才发现
   W1 → W2 → W3 → W4
   根因: 慢查询告警阈值过高

# Action Items
[见 5.x 节,每条有 owner / due / verifiable]

# Lessons Learned (经验教训)
- 数据库变更和代码变更必须分开发布
- 索引删除 = 高风险,必须 DBA 单独 review
- 复盘的下次跟踪在 1 个月后

# Appendix (附录)
- 完整时间线
- 监控截图
- 相关 PR / Commit 链接
- 战时频道归档
```

### 4.2 报告写作的 5 条硬要求

```
1. 文中不出现"由于 XX 同学的疏忽"这类表述
   → 改成"由于流程 / 工具 / 系统的某项缺失"

2. 文中不出现"加强 / 强化 / 严格 / 提高意识"这类口号
   → 改成具体的工程任务

3. 所有结论必须有数据 / 监控截图佐证
   → 不允许"感觉是 / 大概是 / 估计是"

4. 每个 Action Item 必须 SMART (Specific / Measurable /
   Achievable / Relevant / Time-bound)
   → "加强测试"不是 Action,"为 order-api 加合约测试,
     在 CI 中阻断" 才是

5. 报告必须公开 (公司内全公开)
   → 不能藏在某个团队 wiki,要让其他团队也能学
```

### 4.3 谁来写复盘报告

```
事故 owner (出事服务的负责人) → 起草
SRE / IC                       → 审阅
所有相关人                       → 补充
SRE Lead / CTO                 → 终审 + 公布

时间窗口: 事故后 5 个工作日内必须完成
   超时不交 = 这个 owner 的下一次值班顺延
   (不是惩罚,是把"复盘"刻进默认流程)
```

## 五、Action Items:复盘最容易死掉的地方

### 5.1 Action Items 的硬要求

每一条 Action Item 必须满足 5 条:

```
1. Owner — 具体到一个人(不是"平台组")
2. Due Date — 具体到一天(不是"尽快")
3. Verifiable — 完成时怎么验证(不是"加强 X")
4. Severity — 这条不做下次事故会有多严重
5. Tracking — 在哪个系统里跟(JIRA / GitHub Issue / Linear)
```

**一个反例**:

```
错的 Action Items:
- 加强 SQL 评审 (owner: 平台组, due: 后续)
- 改进监控 (owner: SRE, due: 长期)
- 团队培训 (owner: 组长, due: 持续)

   ↑ 三条都没法跟踪,半年后这三条还在那儿挂着
```

**对应的正例**:

```
对的 Action Items:
- 在 CI 加 sqlcheck 静态扫描,所有 DDL PR 必须通过
  Owner: 张三 / 平台组
  Due: 2026-05-25 (2 周)
  Verify: PR #1234 合并 + 监控显示 sqlcheck 已生效
  Severity: 高(同类事故下次概率 → 0)
  Tracking: PLATFORM-456

- 慢查询告警阈值从 5s 改为 500ms,加入主告警群
  Owner: 李四 / DBA
  Due: 2026-05-18 (1 周)
  Verify: Alertmanager 配置 PR 合并,本周新增告警数据
  Severity: 中(发现速度提升)
  Tracking: SRE-789

- 编写"高危 DDL 操作 Runbook",列出索引 / 表结构变更检查表
  Owner: 王五 / DBA
  Due: 2026-05-25
  Verify: Runbook 在 wiki 发布 + 团队 review
  Severity: 中(标准化降低人因风险)
  Tracking: DOC-101
```

### 5.2 Action Item 反模式

```
反模式 1: "加强 XX"
   症状: 一片"加强测试 / 加强 review / 加强意识"
   后果: 没法跟踪,半年后还挂着
   修复: 必须翻译成具体工程任务

反模式 2: Owner 是组而不是人
   症状: "Owner: 平台组" "Owner: 后端团队"
   后果: 责任弥散,没人真负责
   修复: 必须指定一个人,组长也行,但必须有名字

反模式 3: Due Date 是 "持续 / 长期 / Q3"
   症状: 没有具体日期
   后果: 永远 due 不到
   修复: 必须落到具体某一天

反模式 4: 一次列 20 条
   症状: 复盘热情高,Action 列了 20 条
   后果: 半年完成 3 条
   修复: 最多 7 条,按 Severity 排序,P0 一定要 4 周内完成

反模式 5: 列完不跟踪
   症状: 复盘结束 = 任务结束
   后果: Action 永远在文档里,不在系统里
   修复: 所有 Action 当天进 JIRA / GitHub Issue,
        每周一次扫描完成率
```

### 5.3 Postmortem of the Postmortem

**最关键的一条规则**:Action Items 完成率必须自己监控,**作为一个核心指标贴在团队墙上**。

```
团队复盘健康度看板(每月更新):

  本月新增 Action Items:       12
  本月完成 Action Items:        7
  本月超期未完成:               3
  本月完成率:                  58%       ← 健康线 80% 以上

  累计 Action Items 池:         48
  累计完成:                    35
  累计完成率:                  73%

  最老的未完成 Action:
    PLATFORM-321: 编写 K8s Runbook (Due: 2025-12-15, 超期 5 个月)
                  Owner: 张三
                  ↑ 这条要拎出来在团队会上单独讨论
```

**为什么这一条至关重要**:**"做了复盘"不等于"修了系统"——只有 Action 完成才等于修了**。一个团队的真实可靠性,**不看复盘开了多少次,看 Action 完成率有多少**。

### 5.4 复盘后跟踪的 cadence

```
事故 D 日:        事故发生
D + 2~5 天:      复盘会议,产出 Action Items
D + 7 天:        所有 Action 进入 ticket 系统,owner 确认
D + 14 天:       第一次进度检查(SRE Lead 主持)
D + 30 天:       中期检查,看高 Severity Action 是否完成
D + 60 天:       全部 Action 完成或重新评估
D + 90 天:       同类事故指标观察 — 同类事故有没有再来

每月:           复盘健康度报表上传
每季度:         所有未完成 Action 重新审议
```

## 六、复盘会议:谁主持、谁参加、什么时候开

### 6.1 时间窗口:不要太早也不要太晚

```
事故关闭 → 复盘
   太早 (< 2 天):  当事人还在情绪里,讲不清楚
                  时间线和监控数据还没整理好
   合适 (2-5 天):  事故的记忆还鲜活,情绪也冷静了
                  数据 / 监控 / 截图都拉出来了
   太晚 (> 1 周): 记忆开始失真,细节丢失
                  心理上"已经过去了"
```

**雷打不动:5 个工作日内开完**。超期不开 = SRE Lead 的责任。

### 6.2 谁主持:中立 + 有威信

```
最佳人选:
   - SRE Lead / 平台 Lead (中立)
   - 没出事的服务 Lead (中立)
   - 经验丰富的资深工程师

绝对不能:
   - 出事服务的 Lead (利益相关)
   - 出事的当事人 (心理负担)
   - 直接 manager (变批斗会)
   - 完全外行 (问不出关键 Why)
```

**主持人的核心职责不是"问",是"挡"**——挡掉所有"小张为什么这么蠢"式的发问,把会议拉回"系统漏洞"维度。

### 6.3 谁参加:相关人 + 一个不相关的人

```
必须到:
   - 事故 IC
   - Ops 当事人(动手的那些)
   - Scribe(记时间线的)
   - 服务 owner
   - 上下游服务相关人
   - SRE / DBA / 平台代表(看根因领域)
   - Comms 代表(看对外影响)

强烈建议:
   - 一个完全不相关的人(其他业务线的工程师)
     原因: 他不在情绪里,会问出"为什么 X 这么诡异?"
           这种内部人觉得理所当然的好问题

不建议:
   - 高管 (CTO / VP) — 让人不敢说真话,事后通报即可
   - 完全外行 — 浪费时间
   - 复盘人数 > 10 人 — 会变成走过场,不要超过 8 人
```

### 6.4 会议流程:一份能跑的 60 分钟 agenda

```
=================================================
复盘会议 Agenda (60 分钟)
=================================================

00:00 - 05:00  开场 + 重申 Blameless 原则
   主持人开场白:
   "今天我们是来找系统漏洞的,不是来找责任人的。
    任何'某人的错'式的发言,我会打断。"

05:00 - 15:00  Timeline 回顾 (Scribe 主讲)
   过一遍时间线
   有任何遗漏 / 不清楚的地方,在此补充

15:00 - 25:00  影响评估
   业务影响、SLO 影响、客户影响、公关影响
   数字必须有依据

25:00 - 45:00  Root Cause 挖掘 (5 Whys)
   主持人引导,所有人参与
   多线并行,挖到工程 ticket 层

45:00 - 55:00  Action Items 设计
   每条 SMART
   Owner 当场认领
   Due 当场定

55:00 - 60:00  收尾
   总结根因
   总结 Action
   下次同类事故的 follow-up 时间
   感谢所有参与者(包括当事人)

=================================================
```

**主持人最重要的一句话,放在会议开场**:

> "我们今天不是来追究'谁犯了错',是来找'我们的系统在哪里漏了'。**任何在这里说真话的人,都不会因为说出真相而被追责**——这是我们这个团队能学到东西的唯一前提。"

这句话不是套话,是**契约**——主持人说出来之后,后续如果出现"批斗"语气,他必须立刻打断。如果他自己破坏这条契约,**下次没人会在复盘上说真话**。

## 七、Severity Rating:分级复盘,小事故走简化流程

不是每个事故都需要 60 分钟会议 + 5 页报告。**滥用复盘会让团队疲劳,真出 P0 时反而草草收场**。

```
Severity   触发条件                  复盘形式
─────────  ──────────────────────  ─────────────────
P0         全站宕 / 数据丢          强制 blameless 完整流程
           核心交易停 ≥ 15 分钟      60 分钟会议 + 完整报告
                                    所有相关人 + 高管通报

P1         影响 > 1% 用户            建议完整复盘
           核心功能不可用 ≥ 30 分钟  30 分钟会议 + 简化报告
                                    相关人 + SRE Lead

P2         影响 < 1% 用户            轻量复盘
           降级可用 / 边缘功能       书面报告(不开会)
                                    Owner + SRE review

P3         自愈成功的告警            只做记录
           单实例异常                合并到月度复盘合集
```

**重点**:**P3 不开复盘会,但 P3 要月度合集**——一个月一次,把所有 P3 拉出来扫一遍,**找统计规律**(某个服务 P3 频发 = 这个服务有隐性问题)。

## 八、复盘文化建设:从 0 到 1 怎么落地

### 8.1 不要照抄 Google / Netflix

Google 的 Postmortem 文化是真的——他们有专门的"Postmortem Wiki",所有事故公开,新人入职就刷以前的 Postmortem 学习。Netflix 的"Chaos 文化"更夸张——主动制造故障,故障比例当 KPI 看。

**但小团队照抄会立刻死**:

```
Google 抄过来的失败:
   - 要求所有 P2 以上事故都写 5 页报告 → 工程师 burnout
   - 要求所有报告全公司公开 → 大家学会写水帐
   - 要求每周一次 Postmortem Review Meeting → 30 个人各自做自己的事

   后果: 半年后所有人都开始隐瞒 P2 事故,只汇报"修完了"
```

**中型团队的落地路径**:

```
0 → 1:
   开始要求所有 P0 写报告,模板尽量简单(2 页够了)
   不要求公开,只在团队内部存档
   每月统计 Action 完成率,作为团队 OKR 的一部分

1 → 2:
   要求 P1 也走简化复盘
   报告全公司开放,但不强制其他团队学
   开始做"经典事故合集",方便新人学

2 → 3:
   全公司公开所有 Postmortem
   开始引入"无关人提问"角色
   开始有专门的"复盘协调员"角色
```

每一步只在**上一步真正稳定后**才走 — 步子太大,文化崩。

### 8.2 文化建设的 4 个反模式

```
反模式 1: CEO / CTO 说"我们要 Blameless"但自己开会就批人
   后果: 没人信,团队学会"嘴上 Blameless,行动甩锅"
   修复: 高管不参加复盘会(看报告就够了)

反模式 2: 复盘走过场,Action 不跟踪
   后果: 半年下来,所有人觉得复盘是浪费时间
   修复: 把 Action 完成率作为团队的硬指标

反模式 3: 只有 P0 才复盘
   后果: P1 / P2 经验白白浪费
   修复: 分级复盘,P2 可以轻量但必须存档

反模式 4: 复盘报告写完锁起来
   后果: 知识在 owner 脑子里,新人继续踩同一个坑
   修复: 报告全公司可读,新人入职必读最近 10 个 P0
```

### 8.3 一个见效快的工程师本能训练

每次有同事讲事故,**注意自己第一反应**:

```
不健康的本能:
   "他怎么这么菜啊"
   "我才不会犯这种错"
   "幸好不是我值班"

健康的本能:
   "这事如果是我,我会不会也犯?"
   "我们系统是不是也有这个漏洞?"
   "下次类似情况,我有什么提前防的机制?"
```

**Blameless 不只是会议上的规则,是每一个工程师的本能**。这个本能不靠喊口号,靠每次反思:**"如果是我,我也会犯这个错吗?"** — 如果会,系统就有问题。

## 九、反例:批斗大会的话术,以及怎么救回来

下面是真实复盘里常听到的话术,主持人必须能识别并拦回去。

### 反例 1:"这个人不该手抖"

```
错的发言:
   "小张那天就不应该手抖删生产库,做技术的基本素质都没有。"

主持人应该说:
   "等一下,'手抖'不是我们能修的东西。
    我们今天要找的是:为什么系统允许一个手抖能 drop 生产表?
    咱们换个角度,为什么 DROP 操作没有二次确认?"
```

### 反例 2:"我之前就说这是定时炸弹"

```
错的发言:
   "这个服务的代码我之前 review 就觉得有问题,我说过这是定时炸弹。"

主持人应该说:
   "你当时怎么记录的?有 PR comment 吗?有 issue 吗?
    如果没有,我们今天要补一个 Action:
    重要风险点必须落到 issue,不能停在'我说过'。"
```

### 反例 3:"我们就不该信任他"

```
错的发言:
   "新人不应该有生产权限,我们就不该信任他。"

主持人应该说:
   "权限模型不应该建立在'信任谁'之上。
    我们今天要看:权限矩阵是什么时候定的?
    新人应该默认多少权限?生产权限是怎么提权的?
    把信任问题翻译成权限设计问题。"
```

### 反例 4:"以后大家都注意点"

```
错的发言:
   "总之以后大家做生产操作都注意点。"

主持人应该说:
   "'注意'不是一个 Action,'注意'不能 code review,
    '注意'不能 CI 检查,'注意'不能告警。
    我们要把这句话翻译成 3 条具体的工程动作。"
```

### 反例 5:沉默

```
症状:
   主持人问"为什么 X?",所有人低头不说话。
   尤其是当事人在场时。

主持人应该:
   不要点名当事人。
   先点一个**不相关的人**:"小李你的系统也有类似机制,
   你觉得这种情况会怎么发生?"
   破冰后再回到当事人。

   或者: 让当事人最后发言 — 别人都说完,
        他听到所有人都把矛头指向系统而不是他,
        他才敢说真话。
```

## 十、何时不该上完整复盘

跟事故响应一样,**复盘也不该滥用**:

```
不需要完整复盘的情况:

1. 自愈成功的告警 (P3)
   → 月度合集即可

2. 短时间内重复出现的同根因事故
   → 不重新挖根因,直接 follow up 上次的 Action 为什么没做

3. 演练 / 混沌产生的事故
   → 复盘演练本身(响应速度 / 流程是否顺畅)
     不复盘"系统漏洞"(那是设计目的)

4. 第三方完全不可控的故障(云厂商区域宕)
   → 复盘"我们的依赖韧性",不是云厂商的事故本身

5. 已知问题的再次触发(明知有 bug 没修)
   → 不开会,直接执行原计划修
     如果是因为"没排上期"导致没修,复盘排期机制
```

## 十一、踩坑提醒

```
1. Blameless 等于不追责
   → 错。Blameless 把追责从"人"转到"系统",纪律更硬

2. 复盘开成感想会
   → 没有 Action,等于没开

3. Action Items 不跟踪
   → 写出来不修等于没写。完成率是核心指标

4. 复盘报告锁起来
   → 知识没复用,新人继续踩坑

5. 只有 P0 才复盘
   → P1 / P2 的经验是最值钱的(频次高、代价小)

6. 复盘会让高管参加
   → 大家不敢说真话,看报告就够了

7. 一个事故只挖一条主线
   → 多数事故有多个独立漏洞,要多线挖

8. Action Items 列 20 条
   → 完成不了的 Action 是负债,7 条以内,排优先级

9. 复盘 owner 自己写报告又自己审
   → 必须中立的人审,否则会自动美化

10. 同类事故再来时只追当事人
    → 真正要追的是"上次复盘为什么没拦住"——上次的 Action owner
```

## 十二、写在最后

复盘这件事,**最难的不是开会,是文化**。一个团队能不能开出 Blameless 复盘,跟工具无关,跟流程关系也不大——**取决于团队里的资深人和管理者,能不能在压力下守住"对事不对人"**。

这件事的难度被严重低估:

```
理论上:    "我们要 Blameless"
实操中:    凌晨四点的 P0 修完了,大家眼睛通红,客户在催,
           CEO 在群里 @ 你,某个工程师明显手抖了。
           这一刻,你是要做 Blameless 的解释,
           还是要"今天就先这样,先把人骂了出出气"?
```

99% 的团队选了后者,然后他们的事故响应能力就停在那一刻不再进步了。

**所以这一篇的核心,不是教你写复盘模板,是请你做一个承诺:从下一个事故开始,你的复盘会议上不允许出现"是 XX 的错"。** 哪怕当事人真的犯了低级错误,你也要把这句话翻译成"我们的系统应该有 X 来防止这种错"。

工程师对系统的信任,不是靠"我相信我的同事都很厉害"建立的——是靠"**就算我犯了错,系统也能兜住,公司也不会让我背锅**"建立的。这种信任建立起来,团队才能在压力下还说真话;**说真话的团队,才能学到东西;能学到东西的团队,事故才会越来越少**。

**这是 Blameless 真正的意义——不是道德,是工程**。
