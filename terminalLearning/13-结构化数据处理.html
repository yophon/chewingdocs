结构化数据处理:jq 是 JSON 时代的 awk ​

打开任意一台 2026 年工程师的笔记本,看一下他最近一周敲过的命令。有 30% 是产生 JSON 的——kubectl get pods -o json、gh pr list --json number,title,author、aws ec2 describe-instances、docker inspect、cargo metadata --format-version=1、npm view react --json、gcloud compute instances list --format=json、tofu show -json、gh api repos/foo/bar/issues。这些命令的默认输出本来是表格,你主动加 -o json / --json / --format=json 是因为下游要 parse。

下游是谁?90% 是你自己——你 grep 那个 JSON 找一个字段、awk 切一刀拿子串、用 Python 写个 30 行脚本 parse、复制到浏览器的 JSON viewer 里点开看。这五种姿势全是错的——grep '"name":' 撞到嵌套对象就崩,awk 切引号字段必死,Python 30 行的活 jq 一行能干,浏览器看是给人看的不是给管道用的。2026 年还不会 jq,你的现代 CLI 体验大半没了——你拿到的是结构化数据,你处理它的方式还停留在「文本管道」时代。

文本三剑客(grep / sed / awk)统治的是 1980 年代的 stdout——行为基本单位,空格 / Tab 为字段分隔符,无嵌套结构。但 2010 年之后,REST API、Kubernetes YAML、容器 inspect、云厂 SDK 的输出全部是嵌套 JSON / YAML——一个 kubectl get pods -o json 出来的对象 8 层深、5000 行。这种数据用 awk 切就是行为艺术。jq 是 JSON 时代的 awk——它的关系于 JSON,等同于 awk 之于平面文本:一个把"过滤、提取、变换"做成可组合管道的小语言。

一句话先记住:jq 是 JSON 时代的 awk——学会它你的管道升一个量级,从"行 + 字符串"升级到"对象 + 数组"。yq 是它的 YAML 版,xsv / miller 是它的 CSV 版,dasel 是它的"一个 selector 吃所有格式"的尝试。不会 jq 的工程师,生产力上限被钉死在 2010 年。

这一篇拆开讲:jq 心智模型(filter 表达式,不是 grep)、jq 必会的 10 个语法、10 个真实场景(kubectl / gh / aws / docker / npm / git)、yq 两个项目怎么选、xsv / miller 处理 CSV/TSV、dasel 跨格式 selector、何时该换 DuckDB / pandas / Python(数据规模和算力的分水岭)、以及一组"反对的写法"——这些坑我每个团队都见过。

一、为什么需要 jq ​

1.1 没 jq 的世界 ​

bash
# 想从 kubectl 出来的 JSON 抽出所有 Running 状态的 Pod 名字
kubectl get pods -o json > pods.json

# 错法 1:grep
grep '"name":' pods.json
# 输出一堆 "name": "foo-abc",还要再 sed 抠引号
# 而且所有 nested object 的 name 字段都抓出来了
# 包括 container 的 name、port 的 name、env var 的 name

# 错法 2:python
python3 <<'EOF'
import json
with open('pods.json') as f:
    data = json.load(f)
for item in data['items']:
    if item['status']['phase'] == 'Running':
        print(item['metadata']['name'])
EOF
# 6 行 Python,还要切到独立文件,变量名也得想

# 错法 3:awk 切 JSON
awk -F'"' '/"name"/{print $4}' pods.json
# 撞到 "name": {"first": "..."} 就崩
# JSON 里有 \" 转义引号就完全错位

这三条全是反模式——grep 不懂结构、Python 重炮打蚊子、awk 切 JSON 是历史伤害。jq 一条:

bash
kubectl get pods -o json | jq -r '.items[] | select(.status.phase=="Running") | .metadata.name'

10 倍短、零 bug、纯管道。这就是这一节要讲的事——jq 不是"命令行工具",jq 是 JSON 上的 awk + sed + grep 三合一。

1.2 jq 不是什么 ​

是                                  不是
─────────────────────────────────────────────────────────────
JSON 的 filter 语言                 通用编程语言(50 行 jq 该重写成 Python)
管道里的小工具                      数据库引擎(JOIN / aggregate 性能差)
读 JSON 写 JSON / 文本              读 YAML(那是 yq)
                                    读 CSV(那是 xsv / mlr)
单行表达式为主                      多行脚本(虽然支持 `-f script.jq`,但用得少)
零依赖、跨平台                      可视化 GUI(JSON viewer 不是 jq)

最常见的误解:看到 jq '.foo' 以为 jq 是"JSON 的 grep"。不是——.foo 是一个完整的 filter 表达式,它的语义是"输入是一个对象,输出这个对象的 foo 字段"。jq 的所有操作都是 filter 表达式,它是函数式语言,不是匹配工具。

二、jq 心智模型 ​

这张图比下面的所有命令都重要:

┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  输入 JSON   │ ───▶ │  filter 表达式│ ───▶ │  输出 JSON   │
│ (从 stdin)   │      │ (你写的那串)  │      │ (到 stdout)  │
└──────────────┘      └──────────────┘      └──────────────┘
        │                    │                    │
   一个 / 多个值          组合管道             默认 JSON
                                              -r → raw 文本
                                              -c → 一行 compact

核心契约:

  1. 输入是 JSON 值——可以是对象、数组、字符串、数字、bool、null,也可以是 JSON 流(多个值,空格 / 换行分隔)
  2. filter 表达式——一个或多个 filter 用 | 组合,每个 filter 接收一个值,输出零或多个值
  3. 输出是 JSON 值——默认 pretty-print(带缩进),-c 压一行,-r 把字符串解掉外层引号变成原始文本

关键认知:jq 的 | 不是 shell 的 |——shell 的 | 接的是字节流,jq 的 | 接的是 JSON 值。语义不同,但写起来像。

2.1 三种最常见的输入形态 ​

bash
# 形态 1:一个 JSON 对象
echo '{"name":"alice","age":30}' | jq '.name'
# "alice"

# 形态 2:一个 JSON 数组
echo '[{"name":"alice"},{"name":"bob"}]' | jq '.[].name'
# "alice"
# "bob"

# 形态 3:JSON 流(多个 JSON,空格 / 换行分隔)
echo '{"a":1} {"a":2} {"a":3}' | jq '.a'
# 1
# 2
# 3

JSON 流形态是 kubectl get -o json | jq 和 gh api --paginate | jq 的真实形态——多个对象,不在数组里。jq 默认能处理。

2.2 -r 是 80% 场景的必备 ​

bash
# 不加 -r,字符串带引号
echo '{"name":"alice"}' | jq '.name'
# "alice"   ← 字符串带 "

# 加 -r,raw 输出
echo '{"name":"alice"}' | jq -r '.name'
# alice     ← 纯文本,可以喂 xargs / shell var

# 配合 shell 用:
NAME=$(echo '{"name":"alice"}' | jq -r '.name')
echo "$NAME"  # alice

经验法则:只要 jq 的输出要喂给下游 shell 命令,就加 -r——下游需要纯文本而不是带引号的 JSON 字符串。

三、jq 的 10 个必会语法 ​

这一节列学了就能 hold 住 80% 场景的语法。不堆全部 builtin,只挑高频。

3.1 .key —— 取对象字段 ​

bash
echo '{"name":"alice","age":30}' | jq '.name'
# "alice"

# 嵌套
echo '{"user":{"name":"alice"}}' | jq '.user.name'
# "alice"

# 字段名有特殊字符,用 ."xxx"
echo '{"my-name":"alice"}' | jq '."my-name"'
# "alice"

# 字段可能不存在,用 ? 不报错
echo '{"a":1}' | jq '.b?'
# null
echo '{"a":1}' | jq '.b'
# null  (jq 默认就是 null,但深层不存在时 ? 更安全)
echo '{"a":1}' | jq '.b.c?'
# null  (没有 ? 会报 Cannot index null)

3.2 .array[] —— 拆数组成流 ​

bash
# 把数组解开成多个值
echo '[1,2,3]' | jq '.[]'
# 1
# 2
# 3

# 取数组第 N 个
echo '[10,20,30]' | jq '.[1]'
# 20

# 切片
echo '[10,20,30,40]' | jq '.[1:3]'
# [20,30]

最高频用法:.items[] 把 kubectl / aws / gh 返回的 items 数组拆开,每个元素逐个处理。

3.3 select(condition) —— 过滤 ​

bash
# 数字过滤
echo '[1,2,3,4]' | jq '.[] | select(. > 2)'
# 3
# 4

# 字段过滤
echo '[{"phase":"Running"},{"phase":"Pending"}]' \
  | jq '.[] | select(.phase=="Running")'
# {"phase":"Running"}

# 多条件
echo '[{"a":1,"b":2},{"a":1,"b":3}]' \
  | jq '.[] | select(.a==1 and .b > 2)'
# {"a":1,"b":3}

# 取反
jq '.[] | select(.phase != "Running")'
# 等价于
jq '.[] | select(.phase == "Running" | not)'

select 是 jq 的 grep,但它认结构,不是认字符串。

3.4 map(...) —— 数组映射 ​

bash
# 每个元素加 1
echo '[1,2,3]' | jq 'map(. + 1)'
# [2,3,4]

# 取每个对象的字段
echo '[{"name":"a"},{"name":"b"}]' | jq 'map(.name)'
# ["a","b"]

# map(f) 等价于 [.[] | f] —— 拆开、应用、收集

3.5 | —— 管道(filter 串联) ​

bash
echo '{"users":[{"name":"a"},{"name":"b"}]}' \
  | jq '.users | .[] | .name'
# "a"
# "b"

# 等价写法
jq '.users[] | .name'
jq '.users[].name'

jq 的 | 和 shell 的 | 心智一样——左边输出喂给右边。但 jq 的 | 不创建进程,在同一个 jq 进程里。

3.6 {} 构造对象 ​

bash
# 把对象重组成新 schema
echo '{"metadata":{"name":"foo","namespace":"prod"},"status":{"phase":"Running"}}' \
  | jq '{name: .metadata.name, ns: .metadata.namespace, ok: (.status.phase=="Running")}'
# {
#   "name": "foo",
#   "ns": "prod",
#   "ok": true
# }

# 简写:字段名同变量名时
echo '{"foo":1,"bar":2,"baz":3}' | jq '{foo,bar}'
# {"foo":1,"bar":2}

重组对象是 jq 的核心威力之一——把云厂 SDK 那种 50 字段的对象抽成你要的 3 字段子集。

3.7 [] 构造数组 + , 并行 filter ​

bash
# 把多个 filter 的输出收成数组
echo '{"a":1,"b":2}' | jq '[.a, .b]'
# [1,2]

# , 让多个 filter 并行,各自输出一个值
echo '{"a":1,"b":2}' | jq '.a, .b'
# 1
# 2

, vs | 的区别:a, b 是 a 和 b 都从原输入算,各输出一个值;a | b 是 b 从 a 的输出算。

3.8 length —— 长度 / 大小 ​

bash
echo '[1,2,3]' | jq 'length'        # 3
echo '"hello"' | jq 'length'         # 5
echo '{"a":1,"b":2}' | jq 'length'   # 2(字段数)
echo 'null' | jq 'length'            # 0

3.9 group_by —— 按字段分组 ​

bash
echo '[{"team":"a","ok":true},{"team":"a","ok":false},{"team":"b","ok":true}]' \
  | jq 'group_by(.team)'
# [
#   [{"team":"a","ok":true},{"team":"a","ok":false}],
#   [{"team":"b","ok":true}]
# ]

# 配 map 算每组数量
jq 'group_by(.team) | map({team: .[0].team, count: length})'
# [{"team":"a","count":2},{"team":"b","count":1}]

3.10 --arg / --argjson —— 从 shell 传变量 ​

bash
# 字符串变量
NS="prod"
kubectl get pods -A -o json \
  | jq --arg ns "$NS" '.items[] | select(.metadata.namespace == $ns) | .metadata.name'

# 数字 / 布尔 / JSON 对象用 --argjson
THRESHOLD=80
df --output=pcent / | tail -1 | awk '{print $1+0}' \
  | jq --argjson t "$THRESHOLD" '. > $t'  # true / false

不要用字符串拼接传变量——"$NS" 拼到表达式里,引号、特殊字符全是 bug 源。--arg / --argjson 是正解。

四、10 个真实场景 ​

这一节是这一篇的核心——把 jq 套到日常 CLI 输出上。每条都是我或团队当周敲过的。

场景 1:列出所有 Pod 名字 ​

bash
kubectl get pods -o json | jq -r '.items[].metadata.name'
# my-app-abc123
# my-app-def456
# nginx-xyz789

对比 kubectl 自带的 -o name:

bash
kubectl get pods -o name
# pod/my-app-abc123
# pod/my-app-def456

注意有 pod/ 前缀。jq 这一条让你完全控制输出格式。

场景 2:过滤 status ​

bash
# 只看 Running 的 pod
kubectl get pods -o json \
  | jq -r '.items[] | select(.status.phase=="Running") | .metadata.name'

# 看不 Ready 的(任意 container Ready=false)
kubectl get pods -o json \
  | jq -r '.items[] 
         | select(.status.containerStatuses[]?.ready == false) 
         | .metadata.name'

# 看重启过的(restartCount > 0)
kubectl get pods -o json \
  | jq -r '.items[] 
         | select((.status.containerStatuses[]?.restartCount // 0) > 0) 
         | "\(.metadata.namespace)/\(.metadata.name) restarts=\(.status.containerStatuses[0].restartCount)"'

// 是 jq 的"默认值"运算符——(a // b) 是"a 为 null/false 时取 b"。这条很高频,避免 null 误判。

场景 3:重组对象提关键字段 ​

bash
# 把 5000 行的 pod JSON 抽成关键 4 字段
kubectl get pods -A -o json \
  | jq '.items[] | {
      name: .metadata.name,
      ns: .metadata.namespace,
      phase: .status.phase,
      node: .spec.nodeName,
      age_sec: (now - (.metadata.creationTimestamp | fromdate))
    }'
# {
#   "name": "my-app-abc",
#   "ns": "prod",
#   "phase": "Running",
#   "node": "node-1",
#   "age_sec": 3600
# }

这种"重组成自己的 schema"是 jq 最值钱的能力——你不再需要记原始 path,在管道头一次抽出来,后面所有处理都是平的。

场景 4:计数 / group_by ​

bash
# 按 namespace 统计 Pod 数
kubectl get pods -A -o json \
  | jq '[.items[] | .metadata.namespace] | group_by(.) | map({ns: .[0], count: length})'
# [
#   {"ns": "default", "count": 3},
#   {"ns": "kube-system", "count": 12},
#   {"ns": "prod", "count": 45}
# ]

# 按 phase 统计
kubectl get pods -A -o json \
  | jq '[.items[] | .status.phase] 
        | group_by(.) 
        | map({phase: .[0], count: length})'

# 集群 Pod 总数
kubectl get pods -A -o json | jq '.items | length'

对比写 Python:同样的事 Python 至少 10 行,jq 一行。

场景 5:从 PR list 提 number + author ​

bash
# 列出最近 50 个 open PR 的 number + author + title
gh pr list --json number,author,title --limit 50 \
  | jq -r '.[] | "\(.number)\t\(.author.login)\t\(.title)"'
# 1234    alice   feat: add billing
# 1235    bob     fix: race in worker
# 1236    alice   docs: update README

# 按 author 分组数 PR
gh pr list --json number,author --limit 100 \
  | jq 'group_by(.author.login) | map({author: .[0].author.login, count: length}) | sort_by(-.count)'

# 找 alice 的所有 PR 标题
gh pr list --json number,author,title --limit 200 \
  | jq -r '.[] | select(.author.login == "alice") | .title'

gh CLI 的 --json 模式 + jq 是 GitHub 数据分析的银弹——你不需要 GitHub GraphQL API,90% 报表 jq 一行解决。

场景 6:aws ec2 找特定 tag 的实例 ​

bash
# 找 Environment=prod 的所有实例 ID
aws ec2 describe-instances --output json \
  | jq -r '.Reservations[].Instances[] 
         | select(.Tags[]? | select(.Key=="Environment" and .Value=="prod")) 
         | .InstanceId'

# 提关键信息:ID + Name tag + Public IP
aws ec2 describe-instances --output json \
  | jq -r '.Reservations[].Instances[] 
         | "\(.InstanceId)\t\((.Tags[]? | select(.Key=="Name") | .Value) // "no-name")\t\(.PublicIpAddress // "no-ip")"'

# 找所有运行中实例并按 InstanceType 分组计数
aws ec2 describe-instances --output json \
  | jq '[.Reservations[].Instances[] | select(.State.Name == "running") | .InstanceType] 
        | group_by(.) | map({type: .[0], count: length})'

AWS 的 tag 结构是 [{Key: "Name", Value: "xxx"}, ...]——这种"键值对数组"用 jq 取一遍就是上面那种 .Tags[]? | select(.Key=="Name") | .Value。记住这个 pattern,AWS 数据立刻好处理 10 倍。

场景 7:Docker 容器 inspect 提关键字段 ​

bash
# 提所有运行中容器的 name + image + ports + mounts
docker ps -q | xargs docker inspect \
  | jq -r '.[] | {
      name: .Name | ltrimstr("/"),
      image: .Config.Image,
      state: .State.Status,
      ports: (.NetworkSettings.Ports | keys),
      mounts: [.Mounts[] | "\(.Source):\(.Destination)"]
    }'

# 一句话看 IP
docker inspect my-container | jq -r '.[0].NetworkSettings.IPAddress'

# 看挂载点
docker inspect my-container \
  | jq -r '.[0].Mounts[] | "\(.Source) -> \(.Destination)"'

场景 8:npm package.json 抽信息 ​

bash
# 列出所有 deps 和版本
jq -r '.dependencies | to_entries[] | "\(.key)@\(.value)"' package.json
# react@18.2.0
# axios@1.4.0

# 列所有 scripts
jq -r '.scripts | keys[]' package.json

# 配 fzf 选一个 script 跑(上一篇技巧的回响)
npm run $(jq -r '.scripts | keys[]' package.json | fzf)

# 检查有没有用某个 dep
jq -e '.dependencies.lodash' package.json && echo "yes" || echo "no"
# -e 让 jq 在结果为 null/false 时返回非零退出码,适合脚本

to_entries 是把 {"a":1,"b":2} 转成 [{"key":"a","value":1},{"key":"b","value":2}] 的 builtin——对象迭代必备。反向 builtin 是 from_entries。

场景 9:git log 转 JSON 后处理 ​

bash
# 用 --pretty=format 加 jq parse
git log --pretty=format:'{"hash":"%H","author":"%an","date":"%ad","msg":"%s"}' --date=iso-strict -100 \
  | jq -s 'group_by(.author) | map({author: .[0].author, count: length}) | sort_by(-.count)'

# 注意:%s 里有引号会破坏 JSON,真实项目用 git log --format=json(2.43+)
git log --format=json -100 | jq '.[] | select(.author.name == "alice") | .commit.subject'

# 或者用 -z(NUL 分隔)+ awk + jq(更老 git 也能跑)
git log -100 --pretty=format:'%H%x09%an%x09%s' \
  | jq -Rn '[inputs | split("\t") | {hash:.[0], author:.[1], msg:.[2]}]'

-R 把输入当 raw 字符串读,-n 不读默认输入,inputs 在 -n 下读所有行——这三个加一起是 jq 处理纯文本输入的标准 idiom。

场景 10:Terraform / Tofu plan 看变更 ​

bash
# tf plan 出 JSON
tofu show -json tfplan > plan.json

# 看所有 resource 变更类型
jq -r '.resource_changes[] | "\(.change.actions | join("+"))\t\(.address)"' plan.json
# create  aws_instance.web
# update  aws_lb.main
# delete  aws_security_group.old

# 只看要删除的资源(危险变更)
jq -r '.resource_changes[] | select(.change.actions[] == "delete") | .address' plan.json

# 计数变更
jq '[.resource_changes[] | .change.actions[0]] | group_by(.) | map({action: .[0], count: length})' plan.json

Terraform plan 的 JSON 是 IaC review 的核心数据源——评 PR 时一眼看哪些资源要删,比看 plan 文本快 10 倍。

五、一段完整管道:从 kubectl 到 Slack 报告 ​

把上面所有技巧串起来,写一个真实场景:每天早上跑一次,生成"昨晚 prod 命名空间有哪些 Pod 重启过"的报告。

bash
#!/usr/bin/env bash
# pod-restart-report.sh

set -euo pipefail

# 1. 从 kubectl 拿 prod 命名空间所有 Pod 的状态
kubectl get pods -n prod -o json \
  | jq -r '
      [.items[]
        | select(.status.containerStatuses != null)
        | {
            name: .metadata.name,
            restarts: ([.status.containerStatuses[].restartCount] | add),
            last_restart: .status.containerStatuses[0].lastState.terminated.finishedAt,
            reason: .status.containerStatuses[0].lastState.terminated.reason
          }
        | select(.restarts > 0)
      ]
      | sort_by(-.restarts)
      | .[]
      | "• \(.name)\t重启 \(.restarts) 次\t上次: \(.last_restart // "未知")\t原因: \(.reason // "未知")"
    ' \
  | tee /tmp/restart-report.txt

# 2. 没数据就别发了
if [ ! -s /tmp/restart-report.txt ]; then
  echo "无重启,跳过通知"
  exit 0
fi

# 3. 发 Slack
SLACK_WEBHOOK="${SLACK_WEBHOOK:?need slack webhook}"
TEXT=$(jq -Rs '{text: "prod 重启报告 \n\(.)"}' < /tmp/restart-report.txt)
curl -sf -X POST -H 'Content-Type: application/json' -d "$TEXT" "$SLACK_WEBHOOK"

逐行讲:[.items[]] 把 items 包成数组(sort_by 要数组);select(.status.containerStatuses != null) 跳过 Pending(还没起来的);{name, restarts, ...} 重组成自己的 schema;add 把所有 container 的 restartCount 累加;sort_by(-.restarts) 倒序;"• \(.name)\t..." 字符串插值输出。第二段 jq -Rs '{text: ...}' 把整个 stdin 当一个字符串包成 JSON——-R raw 输入,-s slurp,生成合法 JSON 喂给 Slack。

这一段脚本是 jq 的"标准生产姿势"——不是 oneliner,但仍然是几行 jq + 几行 shell,比写 50 行 Python 直观、好维护。

六、yq:JSON 的 YAML 表兄 ​

YAML 在 Kubernetes / Helm / GitHub Actions / Ansible / docker-compose 的统治地位决定了 yq 必备。但 yq 有个历史包袱——"yq"这个名字被两个项目用:

mikefarah/yq          Go 写,本系列推荐
  github.com/mikefarah/yq
  brew install yq
  - 语法非常像 jq(但有微妙差异)
  - 直接 parse YAML(不转 JSON)
  - 保留 YAML 注释和顺序
  - 能 inplace 修改 YAML 文件
  - 支持 multi-document YAML(---  分隔)
  - 现在主流,k8s 社区默认

kislyuk/yq            Python 写,jq 的 wrapper
  github.com/kislyuk/yq
  pip install yq
  - 实际是先把 YAML 转 JSON,再喂 jq
  - 语法 100% 是 jq
  - 不保留注释
  - 改 YAML 不友好(转回去会重排)
  - 历史悠久,但现在用得少

怎么选:新项目用 mikefarah——更主动维护、更现代、Go 二进制零依赖。pip install yq 历史项目还有,但 2026 年没新理由选它。

6.1 mikefarah/yq 基础 ​

bash
# 装
brew install yq                # macOS
# 验证
yq --version                   # mikefarah 版本号 v4.x

# 读字段
yq '.metadata.name' deploy.yaml
yq '.spec.replicas' deploy.yaml

# 改字段(inplace)
yq -i '.spec.replicas = 3' deploy.yaml

# 多文档(---  分隔)
yq '.metadata.name' multi-doc.yaml
# 每个 doc 各输出一次

# 选 kind == Deployment 的文档
yq 'select(.kind == "Deployment")' multi-doc.yaml

# 转 JSON
yq -o=json . deploy.yaml

# 从 JSON 转 YAML
echo '{"a":1,"b":[2,3]}' | yq -P

6.2 yq vs jq 的微妙差异 ​

bash
# jq:点开头才是 filter
echo '{"a":1}' | jq '.a'        # 1

# yq:同样
yq '.a' file.yaml               # 1

# 但 yq 的 select 语法稍有不同
yq '.items[] | select(.kind == "Pod")'      # 工作
jq '.items[] | select(.kind == "Pod")'      # 也工作

# yq 的字符串插值不一样
yq '.metadata.name + ":" + .metadata.namespace'   # yq 用 + 拼字符串
jq '"\(.metadata.name):\(.metadata.namespace)"'    # jq 用 \(...)

90% 的场景 yq 和 jq 表达式可互换——但字符串拼接、null 处理、复杂 builtin 时有细微差异。别死记——遇到差异查官方 cheatsheet 比记忆有效。

6.3 yq 改 K8s 部署的高频场景 ​

bash
# 改镜像版本
yq -i '.spec.template.spec.containers[0].image = "myapp:v2"' deployment.yaml

# 加环境变量
yq -i '.spec.template.spec.containers[0].env += [{"name":"DEBUG","value":"true"}]' deployment.yaml

# 改 resources
yq -i '.spec.template.spec.containers[0].resources.requests.cpu = "500m"' deployment.yaml

# 合并两个 YAML
yq eval-all '. as $item ireduce ({}; . * $item)' base.yaml override.yaml

# 从 Helm values 抽某个值
yq '.image.tag' values.yaml

这种"程序化改 YAML"比 sed 安全 100 倍——sed 切 YAML 是地狱(缩进、注释、多文档全是坑),yq 是结构化操作。

七、xsv:Rust 写的 CSV 处理 ​

CSV 是 2026 年依然活着的最古老格式之一——数据导出、报表、Excel 互通、SQL \copy、Pandas 兜底全在用。bash 处理 CSV 的姿势从 awk 跳到 xsv 是质的飞跃。

bash
# 装
brew install xsv

# 看前几行(自动对齐,比 head 漂亮 10 倍)
xsv headers data.csv
xsv slice -e 5 data.csv | xsv table
# 输出对齐的表格:
# id  name     age  email
# 1   alice    30   alice@x.com
# 2   bob      25   bob@x.com

# 按列选
xsv select name,age data.csv

# 按列过滤
xsv search -s status 'active' data.csv

# 频次统计(某列各值出现多少次)
xsv frequency -s country data.csv | xsv table

# 统计(min/max/mean/stddev/...)
xsv stats data.csv | xsv table

# JOIN 两个 CSV
xsv join id users.csv user_id orders.csv

# 抽样
xsv sample 100 data.csv > sample.csv

# 索引(大文件加速,只用一次)
xsv index large.csv     # 生成 large.csv.idx,后续操作快 10-100x

xsv 的卖点是性能——Rust 写、单文件 GB 级 CSV 几秒内 stats 完。对比 awk:awk 切 CSV 引号字段("a, b", c)必死,xsv 内置 CSV parser 不会撞。

7.1 xsv 的局限 ​

  • 不写 SQL——xsv join 是简单等值 JOIN,复杂的还得 DuckDB
  • 不画图——只产生表格 / CSV,可视化要外部工具
  • 流式不支持 GB 级——内存中处理,极大 CSV 要切片

经验:100MB 以内的 CSV xsv 一把过,GB 级用 DuckDB(后面讲)。

八、miller (mlr):多格式 + DSL ​

xsv 是"只吃 CSV 的小钢炮",miller 是"什么都吃的瑞士军刀"——CSV / TSV / JSON / JSONL / Parquet / pprint / DKVP 它都认,而且自带一门小语言(DSL)。

bash
# 装
brew install miller

# CSV → JSON
mlr --icsv --ojson cat data.csv

# JSON Lines → CSV
mlr --ijsonl --ocsv cat events.jsonl

# 看前几行(任意格式)
mlr --icsv --opprint head -n 5 data.csv

# 按列选 + 过滤 + 排序(像 SQL 但是 oneliner)
mlr --icsv --opprint \
  filter '$status == "active"' \
  then put '$score = $views * 0.1 + $likes' \
  then sort -nr score \
  data.csv

# 把 N 个 CSV 合并(自动对齐 header)
mlr --csv cat *.csv > all.csv

# 分组聚合(类似 SQL group by)
mlr --icsv --opprint \
  stats1 -a sum,mean,count -f revenue -g country \
  sales.csv
# country sum_revenue mean_revenue count
# US      12000       300          40
# UK      8000        266          30

miller 的杀手是 DSL——put '$new = $a + $b' 这种"加一列计算字段"在 awk 里写起来又长又错,miller 里一行。

8.1 miller vs xsv 怎么选 ​

xsv     轻、快、CSV 专精,语法是 subcommand(easy)
miller  万能、有 DSL,语法略陡(then ... then ... then 像 SQL pipeline)

90% CSV 处理 xsv 够用
20% 需要计算新列 / 分组聚合 / 跨格式 → miller

两者不冲突,可以并装。我团队的实际比例:xsv 占 70%(快速查看 / 切列 / 频次),miller 占 30%(转格式 / 算新列 / 报表)。

8.2 miller 实战:JSONL 日志分析 ​

bash
# Nginx access log JSON 格式,统计每个 status 出现多少次
mlr --jsonl stats1 -a count -f request_uri -g status access.log.jsonl

# 算 P99 latency 按 endpoint 分组
mlr --jsonl \
  stats1 -a p99,mean -f latency_ms -g endpoint \
  api.log.jsonl

JSONL(JSON Lines,一行一个 JSON 对象)是日志的事实标准——比 CSV 灵活、比纯 JSON 流式友好,miller 是处理 JSONL 的最佳工具之一。

九、dasel:跨格式 selector ​

dasel 想解决一个真问题——JSON / YAML / TOML / XML 的 selector 语法不一样,学一套要么 jq 要么 yq 要么 toml-cli。dasel 说:我让你一个语法读所有格式。

bash
# 装
brew install dasel

# JSON
dasel -f config.json '.server.port'
# 12345

# YAML(同样的 selector)
dasel -f config.yaml '.server.port'
# 12345

# TOML
dasel -f Cargo.toml '.package.version'
# "0.1.0"

# XML(虽然你 2026 还在用 XML 我也无话可说)
dasel -f pom.xml '.project.dependencies.dependency.[0].artifactId'

# 跨格式转换
dasel -r json -w yaml -f config.json
# (json → yaml)

9.1 dasel 该不该用 ​

支持:

  • 一个工具吃所有格式,学一套语法
  • 简单读 / 转格式很方便
  • Go 写,单二进制

反对:

  • 语法不是 jq——它选了第三条路,jq 用户进来要重学
  • 复杂查询比 jq / yq 弱——没有 group_by、没有强大的 builtin、没有完整的函数式表达力
  • 生态比 jq 小一个量级——遇到问题 StackOverflow 答案少
  • 三个工具 dasel / jq / yq 装齐反而比"只学 dasel"更通用

经验结论:dasel 适合「读简单字段 / 转格式」,jq / yq 仍然是复杂处理的主力。我团队的实际使用:jq(JSON 处理 80%)+ yq mikefarah(YAML 处理 18%)+ dasel(2%,只用 dasel -r json -w yaml 这一招)。

别因为 dasel 看起来"统一"就只学它——遇到第一个 group_by 需求你还得回 jq。学 jq + yq 的回报是终身的,学 dasel 只能解决"简单读取"。

十、何时该换到 DuckDB / pandas / Python ​

jq / yq / xsv / miller 是管道里的小工具——它们的设计前提是"数据流过、提取、变换"。当数据规模或操作复杂度上升,小工具的回报递减,该换更重的工具。

┌──────────────────────────────────────────────────────────┐
│  数据规模          推荐工具                                │
├──────────────────────────────────────────────────────────┤
│  < 1MB             jq / yq / xsv / miller                │
│  1-100MB           xsv / miller(内存里跑)               │
│  100MB-10GB        DuckDB(列式 + SQL,跑得飞快)         │
│  > 10GB            DuckDB / Spark / 数据仓库              │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│  操作类型          推荐工具                                │
├──────────────────────────────────────────────────────────┤
│  单字段提取         jq / yq                                │
│  过滤 + 重组       jq / yq(50 行内)                     │
│  分组聚合           miller / xsv stats / DuckDB            │
│  多表 JOIN         DuckDB(jq 写 JOIN 是反人类)           │
│  ML / 统计建模     pandas / Python                        │
│  画图               Python(matplotlib / seaborn)         │
│  Web 报表           Python / DuckDB + Streamlit            │
└──────────────────────────────────────────────────────────┘

10.1 DuckDB:CSV / Parquet / JSON 的 SQL 引擎 ​

bash
# 装
brew install duckdb

# 直接 SQL 查 CSV(不用导入)
duckdb -c "SELECT country, COUNT(*) FROM 'sales.csv' GROUP BY country ORDER BY 2 DESC"

# JOIN 两个 CSV
duckdb -c "
  SELECT u.name, COUNT(o.id) AS orders
  FROM 'users.csv' u
  LEFT JOIN 'orders.csv' o ON u.id = o.user_id
  GROUP BY u.name
"

# JSONL 直接查
duckdb -c "SELECT status, COUNT(*) FROM read_json_auto('logs.jsonl') GROUP BY 1"

# Parquet
duckdb -c "SELECT * FROM 'data.parquet' WHERE year = 2025"

DuckDB 的杀手:不用建库不用 import,SQL 直接查 CSV / Parquet / JSON。性能是 Pandas 的 5-10x(列式 + 向量化)。100MB 以上的 CSV 处理,直接上 DuckDB——比 xsv 强,比 pandas 快。

10.2 pandas:数据分析、不只是查询 ​

python
import pandas as pd

df = pd.read_csv('data.csv')
df.groupby('country')['revenue'].sum().sort_values(ascending=False)

# 画图
df.plot(kind='bar', x='country', y='revenue')

# 时序
df['date'] = pd.to_datetime(df['date'])
df.set_index('date').resample('M')['revenue'].sum().plot()

用 pandas 的时机:画图、ML 建模、时序分析、和 numpy / scipy 联动——这些 jq / DuckDB 做不到。不用 pandas 的时机:单纯的过滤 / 提取 / 聚合——DuckDB 一行 SQL 比 pandas 几行简洁,xsv 一行命令比 pandas 启动 Python 快。

10.3 一个判断分支 ​

你在终端里处理结构化数据,问自己 3 个问题:

  1. 数据 < 100MB?
       是 → 继续问 2
       否 → 直接 DuckDB

  2. 操作是「提取 / 过滤 / 简单聚合」?
       是 → jq / xsv / miller
       否 → 继续问 3

  3. 需要 JOIN / 画图 / ML?
       JOIN → DuckDB
       画图 / ML → Python (pandas + matplotlib)

这条决策树覆盖 95% 真实场景——剩下 5% 是流式日志(用 vector / Fluentbit)或实时数仓(用 ClickHouse),那是另一篇的事。

十一、反对的写法 ​

每个我亲眼见过的反模式,都来自团队代码 review 时的真实 PR。

11.1 用 grep 找 JSON 字段 ​

bash
# 错的
kubectl get pods -o json | grep '"name":'
# 输出 50 行,因为 Pod、container、port、env var 都有 name 字段
# 你拿到一堆字符串,不知道是哪一层的 name

# 错的升级版
kubectl get pods -o json | grep -A 0 '"name":' | head -5
# 仍然不分层,仍然撞嵌套

# 对的
kubectl get pods -o json | jq -r '.items[].metadata.name'

grep 不懂结构——它只匹配字符串。JSON 用 jq,YAML 用 yq,TOML 用 dasel,XML 用 xmlstarlet。

11.2 用 awk 切 CSV 带引号字段 ​

bash
# 错的
awk -F, '{print $2}' users.csv
# 撞到 `"Smith, John",30,alice@x.com` 这种字段,引号里的逗号让 awk 切错位

# 对的
xsv select 2 users.csv
# 或
mlr --icsv --opprint cut -f name users.csv

RFC 4180 的 CSV 允许字段里有引号包裹的逗号——awk 的 -F, 切不过。任何 CSV 处理,都该用真实现 CSV parser 的工具(xsv / miller / Python csv 模块)。

11.3 把 jq 当编程语言写 50 行 ​

bash
# 错的(模糊化的真实例子)
kubectl get pods -A -o json | jq '
  def is_unhealthy:
    if .status.phase != "Running" then true
    elif .status.containerStatuses == null then true
    elif any(.status.containerStatuses[]; .ready == false) then true
    elif any(.status.containerStatuses[]; .restartCount > 5) then true
    else false end;
  def severity:
    if .status.phase != "Running" then 3
    elif any(.status.containerStatuses[]?; .restartCount > 10) then 2
    else 1 end;
  .items[]
    | select(is_unhealthy)
    | {
        name: .metadata.name,
        ns: .metadata.namespace,
        sev: severity,
        ...
        # 后面还有 30 行
      }
    | ...
'

这种 50 行的 jq 表达式应该重写成 Python——jq 是 filter 语言不是通用编程语言。经验阈值:

  • jq 一行 → 完美
  • jq 5-10 行 → 还行,加几个 def 函数
  • jq 30 行+ → 该换 Python,未来三个月没人(包括你自己)能看懂

判断条件:如果你要写 def f1: ... def f2: ... def f3: ... 链式 jq 函数,换 Python。

11.4 把 jq 输出再喂 grep ​

bash
# 错的
kubectl get pods -o json | jq -r '.items[].metadata.name' | grep prod
# 把过滤推到 jq 里更对

# 对的
kubectl get pods -o json | jq -r '.items[] | select(.metadata.namespace == "prod") | .metadata.name'
# 或者直接 kubectl get pods -n prod

jq 内置 select 比 grep 后处理快、准——grep 是字符串匹配,select 是字段匹配,不会撞到 name 里恰好有 "prod" 子串的别的 namespace。

11.5 yq 给当 sed 用改 YAML ​

bash
# 错的
sed -i 's/replicas: 3/replicas: 5/' deployment.yaml
# 撞到 readinessProbe.successThreshold: 3 这种值也会被改

# 对的
yq -i '.spec.replicas = 5' deployment.yaml

sed 对 YAML 是字符串处理,yq 是结构化处理——sed 撞到同名字段就出 bug。任何"修改 YAML 字段"的需求,都该走 yq。

11.6 不用 -r 把字符串带引号喂下游 ​

bash
# 错的
FILE=$(jq '.path' config.json)
cat $FILE
# 错误:cat: '"./data/file.txt"': No such file or directory
# 因为 $FILE 是 "./data/file.txt"(带引号字符串)

# 对的
FILE=$(jq -r '.path' config.json)
cat "$FILE"

jq 输出喂给 shell 用,99% 该加 -r。

11.7 jq 处理 GB 级 JSON ​

bash
# 错的
jq '.items[]' huge-10gb.json
# jq 默认全文件加载到内存,10GB JSON 会 OOM

# 对的(jq streaming 模式)
jq -n --stream 'fromstream(1 | truncate_stream(inputs))' huge-10gb.json
# 但 streaming 语法极其反人类

# 更对(换工具)
duckdb -c "SELECT * FROM 'huge-10gb.json' LIMIT 1000"
# 或者 jq 之前用 split / head 切小

jq 不是流式优先——10GB 的 JSON 用 jq 死,用 DuckDB 几秒。

11.8 把 dasel 当 jq ​

bash
# 错的(以为 dasel 等价于 jq)
dasel -f data.json 'group_by ...'
# dasel 不支持 group_by,你以为它"统一"实际上它"缩水"

# 对的:复杂处理回 jq
jq 'group_by(...)' data.json

dasel 是「简单读」的工具,不是「复杂分析」的工具——别误用。

十二、性能感觉:小工具的速度阶梯 ​

每个工具的"轻量"程度不同,真实跑分大致是这样:

工具          1MB 处理      100MB 处理     1GB 处理       内存上限
─────────────────────────────────────────────────────────────────────
jq            ~50ms        ~1-2s          OOM 风险       全文件入内存
yq (mikefarah) ~100ms      ~3-5s          OOM 风险       全文件入内存
xsv           ~10ms        ~200ms         ~2s           流式 + 索引
miller        ~30ms        ~500ms         ~5s           流式
DuckDB        ~100ms       ~500ms         ~3-5s         流式 + 列式
pandas        ~500ms       ~5s            ~30s+        内存爆,需要 chunk

经验法则:

  1. < 1MB 任意工具都行——选最熟悉的
  2. 1-100MB 优先 xsv / miller / jq —— 启动快、过滤准
  3. 100MB-10GB 上 DuckDB —— SQL + 列式
  4. 10GB+ 上 Spark / 数据仓库 —— 不是 CLI 的事了

十三、组合姿势:jq + fzf + 业务命令 ​

把第 12 篇的 fzf 和这一篇的 jq 联动起来——这是 2026 终端工作流的标配:

bash
# 1. 用 jq 提候选,fzf 让人选,后续命令拿去用
kubectl get pods -o json \
  | jq -r '.items[] | "\(.metadata.namespace)/\(.metadata.name)"' \
  | fzf \
  | awk -F/ '{print "kubectl logs -n " $1 " " $2}' \
  | sh

# 2. gh PR 列表 + fzf 选一个 checkout
gh pr list --json number,title --limit 50 \
  | jq -r '.[] | "\(.number)\t\(.title)"' \
  | fzf \
  | awk '{print $1}' \
  | xargs gh pr checkout

# 3. aws 实例 + fzf 选一个 ssh
aws ec2 describe-instances --output json \
  | jq -r '.Reservations[].Instances[] 
         | select(.State.Name == "running") 
         | "\(.InstanceId)\t\(.PublicIpAddress)\t\((.Tags[]? | select(.Key=="Name") | .Value) // "")"' \
  | fzf \
  | awk '{print $2}' \
  | xargs -I{} ssh ec2-user@{}

这三个例子的模式:结构化数据(JSON)→ jq 提平铺候选 → fzf UI 选 → shell 命令消费。这是现代 CLI 工程师的标准流水线——4 个工具组合,替代 IDE 50% 的"列表 + 选择 + 执行"操作。

十四、jq 的 5 个进阶 flag ​

学完 builtin,还有 5 个 flag 是真正"脚本里离不开"的——记住名字,场景到了能想起:

bash
# -s / --slurp:把 JSON 流包成数组(适合 sort_by / group_by 这种要数组的)
echo '{"a":1} {"a":2}' | jq -s '.'
# [{"a":1},{"a":2}]

# -R / --raw-input + -s:读纯文本,适合 jq 不依赖 JSON 输入的场景
jq -Rs 'split("\n") | map(select(. != ""))' /etc/hosts

# -n / --null-input + inputs:不读默认输入,显式 inputs 收集所有
jq -Rn '[inputs | split(",")] | .[1:]' data.csv

# to_entries / from_entries / with_entries:对象当字典迭代
jq 'with_entries(select(.key != "secret"))' config.json    # 删 key
jq 'to_entries[] | "\(.key)=\(.value)"' env.json            # 转成 KEY=VALUE 文本

# -e / --exit-status:脚本里做条件判断(字段为 null/false → 退出码非 0)
if jq -e '.dependencies.lodash' package.json > /dev/null; then
  echo "lodash is a dep"
fi

walk(f) 是另一个"杀手 builtin" —— 递归遍历整个 JSON 树,任意深度都被访问。walk(if . == null then "" else . end) 一行把所有 null 替换成空字符串,深度无关。

十六、下一篇预告 ​

14-网络调试一线工具.md——curl / httpie / xh / dig / nc / mtr / tcpdump:应用工程师的网络第一现场。

服务挂了 5xx,90% 工程师第一反应是看代码 / 看日志,这是错的。正确的第一步是 curl + dig 验证——你不知道是 DNS 飘了、还是上游 5xx、还是 TLS 握手失败、还是 LB 把你路由错了。这些网络层的 print debug,curl / dig / nc / mtr 四件套是事实标准。下一篇讲清楚:

  • curl 的 5 个杀手 flag:-v / -w "%{time_total}" / --resolve / -H / --data-binary,90% 排查用得着
  • httpie / xh:JSON 时代 curl 的现代替代,http POST url field=value 比 curl -X POST -H 'Content-Type: ...' -d '{...}' 短 3 倍
  • dig 排 DNS:+short / @8.8.8.8 / +trace / MX / TXT —— 本地 DNS 污染、CDN 调度两个高频场景
  • nc / mtr 测连通:nc -zv host port 一秒判端口、mtr 实时看丢包
  • tcpdump 入门:不深入(那是 networkLearning 的事),但你得会 sudo tcpdump -i any -nn port 80 抓一下落 .pcap 给 Wireshark
  • 一段真实排查全流程:服务 5xx,从 dig 到 tcpdump 一步步定位

读完那一篇,你看到任何 HTTP 报错,第一反应不再是「查代码」,而是「curl -v 一下看看」——这是工程师和"会写代码的人"的分水岭。
