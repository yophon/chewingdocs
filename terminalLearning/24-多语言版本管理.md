# 多语言版本管理:mise 一统五件套

每个干这行三年以上的工程师,`~/.zshrc` 里都堆过这一坨东西:

```bash
# nvm —— Node 版本
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"

# pyenv —— Python 版本
export PYENV_ROOT="$HOME/.pyenv"
export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
eval "$(pyenv virtualenv-init -)"

# rbenv —— Ruby 版本
eval "$(rbenv init -)"

# jenv —— Java 版本
export PATH="$HOME/.jenv/bin:$PATH"
eval "$(jenv init -)"

# rustup —— Rust 版本(它自己装好的)
. "$HOME/.cargo/env"

# direnv —— 项目级 .env 注入
eval "$(direnv hook zsh)"
```

数完发现:**6 个 init,每个都在打开终端时跑一遍**。`nvm.sh` 单独就要 200-400ms,**pyenv 再 100-150ms,rbenv 再 50ms,jenv 再 30ms,direnv 再 20ms**——加起来你每开一个新终端要等 **800ms-1s 才看到提示符**。**为了让 `node` / `python` / `ruby` 这种命令能"按项目切到正确版本",你付出了"每次开终端都卡 1 秒"的代价**。

更破防的是:**这 5 个工具做的是同一件事**——「同机器多版本共存 + 按项目自动切换」。**任务一模一样,只是每种语言各做一遍**。**这就是反模式的标志**:**N 个工具做 1 件事,各管各的元数据,互相不知道对方存在**。

> 一句话先记住:**`mise`(Rust,原名 rtx)是 2026 年取代 nvm + pyenv + rbenv + jenv + direnv 五件套的统一答案——一个工具管所有语言、启动 < 30ms、`.tool-versions` / `.mise.toml` 进 git 让团队人手一致**。

mise 不是新概念——**它是 asdf 的下一代**。asdf(2014 出生,Erlang 圈做的)第一次把「language-agnostic version manager」做出来,**一个工具管所有语言**;但 asdf 是 Bash + Ruby 写的,启动也慢,shim 调用成本高。**mise(2023)是 Rust 重写的 asdf 兼容超集**——asdf 所有 plugin 都能用、启动快两个数量级、把 `env` / `tasks` / `secret` 一并塞进来,**直接吃掉 direnv 和半个任务运行器**。

这一篇拆开讲:**为什么 nvm / pyenv 这一代被吃掉**(shim 心智的根本代价)、**mise 心智速通 + 一份完整 `.mise.toml`**、**asdf vs mise**、**三个典型场景**(本地切版本 / CI 对齐 / env 注入)、**和 brew / system Python 的边界**、**从 nvm + pyenv 迁过来怎么做**、**常见坑**、最后一组**反对的写法**——这些坑我每个团队都见过。

## 一、为什么 nvm / pyenv 这一代必死

### 1.1 它们在做什么:shim 拦截心智

理解 nvm / pyenv 慢在哪,先理解它们的工作模式——**shim 拦截**:

```
你在终端里敲 `node script.js`:
        │
        ▼
  shell 在 $PATH 里找 `node`
        │
        ▼
  找到 $HOME/.nvm/versions/node/v20.10.0/bin/node   ← nvm 把这个塞进 PATH
        │                                              (init 时根据当前 .nvmrc 决定)
        ▼
  exec node script.js
```

pyenv / rbenv 的姿势稍有不同,**它们走 shim**:

```
你敲 `python script.py`:
        │
        ▼
  $PATH 里找到 $HOME/.pyenv/shims/python   ← pyenv 在你 $PATH 最前面塞了 shims 目录
        │
        ▼
  这个 python 是个 bash 脚本(shim):
        │   #!/bin/bash
        │   exec pyenv exec python "$@"
        │
        ▼
  pyenv exec 再去查 "当前应该用哪个 Python 版本":
        │   - 当前目录 .python-version?
        │   - 父目录 .python-version?
        │   - 全局 ~/.pyenv/version?
        │
        ▼
  exec $HOME/.pyenv/versions/3.12.5/bin/python script.py
```

**这个心智模型有 3 个死穴**:

```
死穴 1: 每次跑命令都走一次 shim 脚本
       ──> shim 是 bash 脚本,fork + 解析 + exec,慢 5-30ms
       ──> 你跑 100 次 python,纯 shim 开销 0.5-3 秒
       
死穴 2: 每个 init 都要在 shell 启动时做工作
       ──> pyenv init 要扫 PATH、初始化函数、setup completion
       ──> 启动延迟 100-150ms,五件套加起来 800ms+
       
死穴 3: 每个工具各管自家元数据
       ──> Node 用 .nvmrc
       ──> Python 用 .python-version
       ──> Ruby 用 .ruby-version
       ──> Java 用 .java-version
       ──> 项目根四个文件,没人统一管
```

### 1.2 启动慢有多致命

```
不开 mise 的 .zshrc(nvm + pyenv + rbenv + jenv + direnv):
   $ time zsh -i -c exit
   zsh -i -c exit  0.81s user 0.42s system 105% cpu 1.165 total

只用 mise 的 .zshrc:
   $ time zsh -i -c exit
   zsh -i -c exit  0.04s user 0.02s system 99% cpu 0.061 total
```

**一秒 vs 60ms,差 16 倍**。一秒看似不多,但你一天会:
- 开 30 个新 tmux pane(每开一个等 1 秒)
- 开 50 个新终端 tab(每开一个等 1 秒)
- 跑 100 次 `bash -c '...'`(脚本里每次 shell 启动都等)

**一天累计 3-5 分钟纯在等 shell 启动**——更可怕的是,**这 3 分钟切碎成几百个 100ms 卡顿,每次都打断你的思路**。**这种"切碎的延迟"对开发心流的伤害,远大于一次 30 秒的等待**。

### 1.3 元数据分散:团队协作的死结

新人 onboarding 你的项目:

```
你:这个项目要用 Node 20、Python 3.12、Ruby 3.3。
新人:.nvmrc 是空的呀?
你:看 README,我们要求 .nvmrc 写 20。
新人:那 Python 呢?
你:看 .python-version。
新人:Ruby?
你:.ruby-version。
新人:Java?
你:.java-version,但是 jenv 还得手动 enable 一下 versions plugin。
新人:rust 呢?
你:rust-toolchain.toml,这个是 rustup 自己的。
新人:那我 cd 进去这五个东西自动就切了吗?
你:Node 要装 .nvmrc 钩子,Python 走 pyenv-virtualenv,你得 ...

(新人崩溃,转身打开 Docker)
```

**这就是"五个工具各管各"的代价**——**没有任何一个文件能"一眼看出这个项目需要什么"**。新人要学 5 套工具、5 个文件、5 个钩子。**这是工程负债,不是工程债**。

## 二、mise:一个工具吃五个

### 2.1 mise 是什么

```
mise (原名 rtx,2023 改名避免 NVIDIA 商标冲突)
   ┌─────────────────────────────────────────────────────┐
   │  Rust 写的、asdf 兼容、language-agnostic version    │
   │  manager + env 注入 + 任务运行器 + secret 集成      │
   └─────────────────────────────────────────────────────┘
   
吃掉的东西:
   ┌──────────────┬──────────────────────────────────────┐
   │ nvm          │ Node 版本管理                        │
   │ pyenv        │ Python 版本管理                      │
   │ rbenv        │ Ruby 版本管理                        │
   │ jenv         │ Java 版本管理                        │
   │ goenv        │ Go 版本管理                          │
   │ rustup       │ Rust 版本管理(部分,见下)         │
   │ direnv       │ 项目级 env 注入                      │
   │ asdf         │ 完整继承(plugin 生态)             │
   │ 半个 make    │ tasks 段做轻量任务运行器             │
   └──────────────┴──────────────────────────────────────┘
```

**注意 rustup 的特殊性**:Rust 的版本管理通常推荐继续用 `rustup`(因为 rustup 还做 component 管理、target 管理、toolchain 切换等更复杂的事),mise 装的 rust 适合简单场景。其他语言 mise 是无脑替代。

### 2.2 mise 为什么快:不用 shim 用 PATH 注入

mise 的核心心智革新:**不用 shim,改用 PATH 注入**。

```
传统 pyenv shim 模式(每次执行命令都过一次 shim):
  $PATH = ~/.pyenv/shims:/usr/bin:...
   敲 python  ──>  ~/.pyenv/shims/python(bash 脚本)
                  ──>  fork + 解析 + exec
                  ──>  pyenv exec python(再 fork)
                  ──>  $HOME/.pyenv/versions/3.12.5/bin/python

mise 模式(cd 进项目时一次性算好 PATH):
  cd ~/my-project
   ↓ mise 检测到 .mise.toml,算出该用哪个 Python
   ↓ mise 把 $HOME/.local/share/mise/installs/python/3.12.5/bin 注入 $PATH 最前
  $PATH = $HOME/.local/share/mise/installs/python/3.12.5/bin:/usr/bin:...
   敲 python  ──>  直接执行真正的 Python,无中间层
```

**性能差异**:

```
                    pyenv shim       mise
命令启动开销         10-30ms          0ms(纯 PATH 解析)
shell init 开销     100-150ms         < 30ms
跑 1000 次 python    > 10 秒           < 1 秒
```

**为什么 mise 能这么做**:因为它在 shell 里挂一个 `chpwd` hook(zsh)或 `PROMPT_COMMAND`(bash),**进入目录时一次性算好 PATH**,跑命令时没有任何 mise 介入。**而 pyenv 因为要兼容老 bash 没有 hook,只能走 shim 这条路**。

### 2.3 安装 + 第一次上手

```bash
# 一行装
curl https://mise.run | sh

# 加进 ~/.zshrc 最后(顺序很重要,见后)
echo 'eval "$(mise activate zsh)"' >> ~/.zshrc

# bash:
echo 'eval "$(mise activate bash)"' >> ~/.bashrc

# fish:
echo 'mise activate fish | source' >> ~/.config/fish/config.fish

# 重启 shell 后,验证
mise --version    # 2025.x.x
mise doctor       # 检查环境
```

**装完第一件事:列工具**

```bash
# 看 mise 知道哪些语言 / 工具
mise plugins ls-remote | head -20
# node
# python
# ruby
# java
# go
# rust
# deno
# bun
# terraform
# kubectl
# helm
# ...(总共 500+ 个)

# 看某个语言的可用版本
mise ls-remote node | tail -10
# 22.6.0
# 22.7.0
# 22.8.0
# 22.9.0
# 22.10.0
```

**装语言**

```bash
# 装(下载预编译二进制 + 解压)
mise install node@22 python@3.12 go@1.23

# 设全局默认
mise use --global node@22 python@3.12

# 看当前活跃版本
mise current
# node    22.10.0
# python  3.12.7

# 看装了哪些版本
mise ls node
# 18.20.4
# 20.18.0
# 22.10.0  (active)
```

**项目级**

```bash
cd ~/my-project
mise use node@20 python@3.11
# 生成 .mise.toml(或 .tool-versions,看配置)
# 把 node@20 / python@3.11 写进去
# 立刻在当前 shell 切到这两个版本

cat .mise.toml
# [tools]
# node = "20"
# python = "3.11"
```

`cd` 进 `~/my-project` → Node 自动切到 20、Python 自动切到 3.11。`cd ..` 出来 → 切回全局的 22 / 3.12。**无 shim、无 fork、无延迟**。

## 三、配置文件:`.tool-versions` vs `.mise.toml`

mise 支持两种配置格式,**一定要选对**:

### 3.1 `.tool-versions`(asdf 兼容,简单)

```
# 项目根 .tool-versions
node 22.4.0
python 3.12.5
ruby 3.3.0
go 1.23.0
```

**特点**:

```
✓ 跟 asdf 完全兼容,asdf 用户切 mise 无感
✓ 行格式简单,一眼看懂
✗ 不能写 env / tasks
✗ 不能写注释以外的元信息
```

**什么时候用**:**项目同时被 asdf 和 mise 用户用**(混合团队),或者纯版本需求,不需要 env / tasks。

### 3.2 `.mise.toml`(现代,推荐)

```toml
# 项目根 .mise.toml
[tools]
node = "22"
python = "3.12"
go = "1.23"

# 锁定到具体版本(进 lockfile 用)
# node = "22.10.0"
# python = "3.12.7"

[env]
# 进入项目时自动 export 这些
DATABASE_URL = "postgres://localhost/myapp_dev"
NODE_ENV = "development"
PYTHONDONTWRITEBYTECODE = "1"

# 把项目本地 node_modules/.bin 加进 PATH
_.path = ["./node_modules/.bin", "./bin"]

# 加载额外的 .env 文件
_.file = [".env.local"]

[tasks.test]
description = "run tests"
run = "pytest -v"

[tasks.dev]
description = "start dev server"
run = "npm run dev"

[tasks.lint]
description = "lint code"
run = """
ruff check .
ruff format --check .
"""
```

**特点**:

```
✓ 一个文件搞定:版本 + env + 任务
✓ 替代 direnv(env 段)
✓ 替代轻量 Makefile(tasks 段)
✓ 支持模板:{{ exec("op read op://Personal/api/key") }} 取 1Password
✗ asdf 不认识(混合团队需要权衡)
```

**什么时候用**:**新项目、纯 mise 团队,直接 `.mise.toml`**——这是 2026 推荐姿势。

### 3.3 进 git 是关键

**这两个文件都必须 commit 进 git**:

```bash
git add .mise.toml
git commit -m "chore: pin tool versions"
```

**这是工程化的本质**——团队所有人 clone 下来 `mise install` 一行,**版本完全对齐**。新人入职:

```
新人:git clone git@xxx/xxx.git
新人:cd xxx
新人:mise install     # 自动读 .mise.toml,装所有缺的版本
新人:mise current     # node 22, python 3.12,跟你的一模一样
新人:开干

       ↑ 30 秒入坑,vs 之前 5 个工具学一周
```

### 3.4 全局配置:`~/.config/mise/config.toml`

```toml
# ~/.config/mise/config.toml(全局)
[tools]
node = "22"
python = "3.12"
go = "1.23"
# 全局默认版本,新机器装好后这些命令立刻能用

[settings]
# 自动信任父目录的 .mise.toml(避免每次 cd 都 mise trust)
trusted_config_paths = ["~/work"]

# 不让 mise 升级自己时改 ~/.zshrc
not_found_auto_install = false

# 一次升级所有 plugin
plugin_autoupdate_last_check_duration = "1 week"
```

## 四、mise 的「超能力」:不仅是版本管理

### 4.1 替代 direnv:`[env]` 段

**direnv 解决的问题**:`cd` 进项目时自动注入环境变量(比如 `DATABASE_URL`、`AWS_PROFILE`)。

**mise 的 `[env]` 段做同样的事**,而且统一在一个文件里:

```toml
# .mise.toml
[env]
DATABASE_URL = "postgres://localhost/myapp_dev"
AWS_PROFILE = "myapp-dev"
GOPROXY = "https://goproxy.cn,direct"

# 引用其他 env 变量
PATH_TO_BIN = "{{config_root}}/bin"

# 从 1Password 取(执行时才求值)
SECRET_KEY = "{{ exec(command='op read op://Personal/myapp/secret_key') }}"

# 从 .env 文件加载
_.file = [".env.local", ".env"]

# 加 PATH
_.path = ["./node_modules/.bin", "{{config_root}}/scripts"]
```

`cd` 进项目自动 export 这些变量,`cd ..` 自动 unset。**等同于 direnv,但不用装 direnv**。

**对比 direnv**:

```
                    direnv          mise [env] 段
配置文件           .envrc           .mise.toml
语法              bash             TOML
和版本管理同步     ✗(两个文件)     ✓(一个文件)
模板 / 函数        无               有(jinja 风格)
secret 集成        无               有(1Password / vault)
学习成本          已知 bash        要学 TOML 模板
```

**direnv 不死也不再增长**,大多数 mise 用户直接迁过来。

### 4.2 任务运行器:`[tasks]` 段

**轻量 Makefile 替代品**:

```toml
[tasks.test]
description = "run all tests"
run = "pytest -v"

[tasks.test-unit]
description = "run unit tests only"
run = "pytest -v tests/unit/"

[tasks.lint]
description = "lint and format check"
run = [
    "ruff check .",
    "ruff format --check .",
    "mypy src/",
]

[tasks.db-up]
description = "start local postgres"
run = "docker compose up -d postgres"

[tasks.dev]
description = "start full dev environment"
depends = ["db-up"]
run = "npm run dev"
```

**用法**:

```bash
mise run test          # 跑 test
mise run lint          # 跑 lint
mise run dev           # 跑 dev(会先跑 db-up)
mise tasks             # 列所有任务

# 简写
mise r test
```

**Make / Just 还有更复杂的场景(28 篇专讲任务运行器)**,但 mise tasks 适合"项目里 10-30 个轻量命令"的场景——**省一个 Makefile**。

### 4.3 secret 集成

**实际工程场景**:`.env` 文件里有 API key,但 `.env` 不能 commit 进 git。**怎么让团队所有人都能拿到?**

- **传统方案**:加密 `.env`(sops / git-crypt)+ 文档教大家怎么解密
- **mise 方案**:`.mise.toml` 里写"从 1Password / vault 取",commit 进 git,所有人自动拿:

```toml
[env]
# 从 1Password 取(每次 cd 进项目时执行 op read)
STRIPE_SECRET = "{{ exec(command='op read op://Engineering/stripe/secret') }}"
SLACK_TOKEN = "{{ exec(command='op read op://Engineering/slack/bot_token') }}"

# 从 HashiCorp Vault 取
DB_PASS = "{{ exec(command='vault kv get -field=password secret/myapp/db') }}"

# 从 aws-vault
AWS_SECRET = "{{ exec(command='aws-vault exec myapp -- env | grep AWS_SECRET') }}"
```

**这相当于把 secret 注入收编进版本管理工具**,从此 `.env` 里再没有明文 secret。**这是 mise 比 direnv 多出的杀手特性**。

### 4.4 plugin 生态:吃 asdf 整个生态

mise 兼容 asdf plugin。所以 asdf 用了 10 年积累的 500+ plugin,**mise 一行命令就能用**:

```bash
# 装 terraform
mise plugin add terraform
mise install terraform@1.9.0

# 装 kubectl
mise plugin add kubectl
mise install kubectl@1.31.0

# 装 helm
mise plugin add helm
mise install helm@3.16.0

# 一个项目同时管这些 ops 工具
cat .mise.toml
# [tools]
# terraform = "1.9.0"
# kubectl = "1.31.0"
# helm = "3.16.0"
```

**SRE / DevOps 工程师特别爱这个**:**所有 ops CLI 版本都 commit 进 git,团队对齐**。

## 五、asdf vs mise:全方位对比

```
┌─────────────────────┬──────────────────┬──────────────────┐
│                     │   asdf           │   mise           │
├─────────────────────┼──────────────────┼──────────────────┤
│ 出生年份             │   2014           │   2023           │
│ 实现语言             │   Bash + Ruby    │   Rust           │
│ 启动延迟             │   100-300ms      │   < 30ms         │
│ 命令执行延迟         │   慢(shim)     │   零(PATH)      │
│ shim 性能            │   每次 fork bash │   无 shim         │
│ 配置文件             │   .tool-versions │   .mise.toml     │
│                     │                  │   + 兼容前者       │
│ env 注入            │   无(要 direnv)│   内置 [env]     │
│ 任务运行器           │   无             │   内置 [tasks]   │
│ secret 集成          │   无             │   1Password/vault │
│ plugin 生态          │   完整(500+)   │   100% 兼容 asdf │
│ 跨平台               │   Linux/macOS    │   Linux/macOS/Win │
│ rustup 集成          │   弱             │   原生            │
│ 升级自己             │   asdf update    │   mise self-update│
│ 社区活跃度           │   稳定但慢       │   快              │
│ 大公司用             │   GitLab、Adobe  │   Cloudflare,    │
│                     │   (老用户)     │   越来越多        │
└─────────────────────┴──────────────────┴──────────────────┘
```

**结论**:**新项目无脑选 mise**。

**asdf 仍在的合理场景**:

```
1. 已有团队用了 asdf 5 年,迁移有政治成本
2. 不想换工具的工程师(.bashrc 里那点东西陪 10 年)
3. CI 用的 asdf-action 已经稳定,不愿意切
```

**迁 mise 的成本**:

```bash
# 1. 装 mise
curl https://mise.run | sh

# 2. 已有的 .tool-versions 直接读
mise install  # 读 .tool-versions,装齐所有版本

# 3. 卸 asdf
brew uninstall asdf  # 或对应包管理器
rm -rf ~/.asdf

# 4. 把 ~/.zshrc 里 asdf 行删掉,加 mise activate
```

**整个过程 10 分钟,无痛迁移**——这是 mise 设计的杀手优势,**asdf 兼容是它的护城河**。

### 6.3 场景 3:env 注入 + secret

**真实场景**:项目根有这些 env:

```bash
DATABASE_URL=postgres://localhost/myapp_dev
REDIS_URL=redis://localhost:6379
S3_BUCKET=myapp-dev
STRIPE_SECRET=sk_test_xxx
AWS_REGION=us-east-1
LOG_LEVEL=debug
```

前 4 个是非 secret,后两个是 secret。**该怎么管**?

**反对的写法**:

```bash
# 反对 1:全写进 .env(包括 STRIPE_SECRET),.gitignore 排除
# 问题:新人 clone 后没 .env,要找老员工要;离职员工电脑里还有 key

# 反对 2:写进 ~/.zshrc
# 问题:全局污染,不同项目同名 env 互相覆盖
```

**mise 写法**:

```toml
# .mise.toml(commit 进 git)
[env]
DATABASE_URL = "postgres://localhost/myapp_dev"
REDIS_URL = "redis://localhost:6379"
S3_BUCKET = "myapp-dev"
AWS_REGION = "us-east-1"
LOG_LEVEL = "debug"

# secret 从 1Password 取(commit 进 git 安全)
STRIPE_SECRET = "{{ exec(command='op read op://Engineering/stripe/test-secret') }}"

# 个人覆盖(不进 git)
_.file = [".env.local"]
```

新人 clone 后:

```bash
# 1. 装 1Password CLI
brew install 1password-cli
op signin

# 2. cd 进项目,mise 自动跑所有逻辑
cd ~/work/myapp
mise install     # 装 Node / Python 等
# (mise activate 已经把 [env] 注入到 shell)

# 3. 立刻能跑
echo $STRIPE_SECRET   # sk_test_xxx (从 1Password 取来的)
npm test              # 用对的 env 跑
```

**整个流程没有任何"先找 .env 文件"或"老员工发 key 给我"的步骤**——secret 由 1Password 集中管,mise 在执行时取。**这是 secret 管理的工程终点**。

## 七、和 brew / system Python 的边界

新手最容易混的问题:**`brew install python` 和 `mise install python@3.12` 有什么区别**?

```
                    brew install python    mise install python
装在哪              /opt/homebrew/bin       ~/.local/share/mise/installs/
                                            python/3.12.7/bin
是否进 PATH          是(系统级)            只有在项目用时才进
多版本共存          困难(python@3.11 link) 天然支持
按项目切             不支持                  支持
团队对齐             无                      .mise.toml 进 git
卸载干净            brew uninstall          mise uninstall
                                            ──> 删个目录
```

**推荐姿势**:

```
brew 装的:    永久存在、系统级、工具类
   - mise 自己(对,先用 brew 装 mise)
   - git
   - ssh, openssh
   - 终端模拟器(iTerm2, Ghostty)
   - 编辑器(neovim, helix)
   - 浏览器、Slack、IDE(cask)
   - 系统级 CLI(rg, fd, bat, fzf, jq)

mise 装的:    语言运行时、项目相关
   - node, python, ruby, go, rust, java
   - 项目依赖的 ops 工具(terraform, kubectl, helm)
   - 跟项目版本绑定的 CLI(pnpm, yarn, poetry, bundler)
```

**判断准则**:**"这个工具的版本和具体项目相关吗?"** → 是,mise;否,brew。

**反例**:用 brew 装 Node 是错的:

```bash
# 反对:brew install node
# 问题:brew 仓库只有一个"主流版本"(目前是 22)
#       想用 Node 18 / 20 同时存在?brew install node@18 + node@20,
#       手动 link / unlink 切换,反人类
#       团队对齐?brew bundle 只能锁主版本,patch 跟不上

# 推荐:mise install node@18 node@20 node@22
#      项目 .mise.toml 写死,跨平台对齐
```

**反例**:用 mise 装 ripgrep 是错的:

```bash
# 反对:mise install ripgrep
# 问题:ripgrep 不是语言运行时,跟项目版本无关
#       全局工具用 brew 一行装就好

# 推荐:brew install ripgrep  (或 apt / pacman / nix)
```

## 九、从 nvm + pyenv 迁过来:实操步骤

实际迁移路径,亲测,15 分钟搞定:

### 步骤 1:装 mise

```bash
# macOS
brew install mise

# 或一行装
curl https://mise.run | sh
```

### 步骤 2:扫描你 ~/.zshrc / .bashrc

找到所有跟版本管理相关的行:

```bash
# 在 ~/.zshrc 里搜
grep -E 'nvm|pyenv|rbenv|jenv|goenv|asdf|direnv' ~/.zshrc
```

### 步骤 3:把这些行替换成 mise

```bash
# ~/.zshrc 改前:
# nvm
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
# pyenv
export PYENV_ROOT="$HOME/.pyenv"
export PATH="$PYENV_ROOT/bin:$PATH"
eval "$(pyenv init -)"
# rbenv
eval "$(rbenv init -)"
# direnv
eval "$(direnv hook zsh)"

# ~/.zshrc 改后(就一行):
eval "$(mise activate zsh)"
```

### 步骤 4:迁移已装版本

```bash
# Node: 看 nvm 装了什么
nvm ls
# v18.20.4
# v20.18.0
# v22.10.0

# 在 mise 装同样版本
mise install node@18 node@20 node@22

# Python: 看 pyenv 装了什么
pyenv versions
# 3.10.14
# 3.11.10
# 3.12.7

# 在 mise 装
mise install python@3.10 python@3.11 python@3.12

# 设全局默认(对应 nvm alias default / pyenv global)
mise use --global node@22 python@3.12
```

### 步骤 5:迁移项目级配置

每个项目根:

```bash
cd ~/work/some-project

# 如果有 .nvmrc / .python-version,mise 自动读(asdf 兼容模式)
# 想升级到 .mise.toml:
mise use node@$(cat .nvmrc) python@$(cat .python-version)
# 这一行会生成 .mise.toml

# 删掉旧文件(可选,mise 仍兼容这些)
# rm .nvmrc .python-version .ruby-version
```

### 步骤 6:卸老的

确认 mise 跑起来一切正常后:

```bash
# 卸 nvm
rm -rf ~/.nvm

# 卸 pyenv
brew uninstall pyenv pyenv-virtualenv
rm -rf ~/.pyenv

# 卸 rbenv
brew uninstall rbenv ruby-build
rm -rf ~/.rbenv

# 卸 direnv(如果用 mise [env] 段替代)
brew uninstall direnv
```

### 步骤 7:验证

```bash
# 新开一个 terminal
time zsh -i -c exit
# zsh -i -c exit  0.04s user 0.02s system 99% cpu 0.061 total
# 60ms vs 之前的 1 秒,差 16 倍

# cd 进各项目验证版本切换
cd ~/work/api-server && node --version    # 应该是 18
cd ~/work/dashboard && node --version     # 应该是 22

# 全部 OK,迁移成功
```

## 十一、常见坑

### 11.1 `mise activate` 顺序错

**问题**:`mise activate` 必须放在 ~/.zshrc 里**所有 PATH 修改之后**,但**在 prompt 渲染之前**。

```bash
# 错的顺序(mise 在 starship 之前 activate,但又在 conda init 之后)
eval "$(conda init zsh)"             # 这一步把 conda 塞 PATH 最前
eval "$(mise activate zsh)"          # mise PATH 注入被 conda 抢走
eval "$(starship init zsh)"          # prompt

# 正确顺序
eval "$(starship init zsh)"          # 这个跟 mise 无关,放哪都行
# 不用 conda init,直接 mise install python(无 conda)

eval "$(mise activate zsh)"          # 放在所有 PATH 注入之后
```

**经验**:`mise activate` 通常放 ~/.zshrc 末尾就对。

### 11.2 direnv + mise 同时装

**问题**:`.envrc` 和 `.mise.toml [env]` 段同时存在,两边在打架。

**解决**:**二选一**。推荐迁全 mise(`[env]` 段已经够了),把 direnv 卸了。

如果必须保留 direnv(老项目),`.envrc` 写:

```bash
# .envrc
use mise          # direnv 用 mise 提供的逻辑
```

让 direnv 委托给 mise,而不是两边各干各的。

### 11.3 shim 残留(从 asdf 迁过来)

**问题**:asdf 在 `~/.asdf/shims/` 里塞了一堆 shim,`$PATH` 还有这些。

**解决**:

```bash
# 1. ~/.zshrc 里删掉所有 asdf 相关 export 和 init
# 2. mise reshim(让 mise 重建符号链接)
mise reshim

# 3. 清 asdf 残留
rm -rf ~/.asdf

# 4. 新开 terminal,echo $PATH 不应该再有 ~/.asdf/shims
echo $PATH | tr ':' '\n' | grep asdf   # 应该空
```

### 11.4 `mise trust` 安全提示

**问题**:第一次 `cd` 进一个有 `.mise.toml` 的项目,mise 提示:

```
mise WARN  ~/work/foo/.mise.toml is not trusted. Trust it with `mise trust`.
```

**为什么**:`.mise.toml` 里可能有 `exec(...)` 这种执行任意命令的语句(比如 secret 取),mise 不会自动信任陌生人的配置。

**解决**:

```bash
# 信任当前项目
mise trust

# 批量信任某目录下所有项目(全局配置)
# ~/.config/mise/config.toml
[settings]
trusted_config_paths = ["~/work", "~/personal"]
```

### 11.5 全局 vs 项目级冲突

**问题**:全局 `mise use --global node@22`,项目 `.mise.toml` 写 `node = "20"`。`cd` 进项目应该用 20。

**怎么验证**:

```bash
cd ~/project
mise current node
# node    20.18.0   <- 项目级生效
mise where node
# /Users/me/.local/share/mise/installs/node/20.18.0
```

如果发现没切对:

```bash
mise doctor          # 看有没有警告
mise trust           # 是否未信任
which node           # PATH 第一个是不是 mise 的
```

### 11.6 Apple Silicon Rosetta 问题

**问题**:M1/M2 Mac,装某个 Node 旧版(< 16)装失败。

**原因**:Node 16 之前没 ARM64 二进制,要 Rosetta 跑 x86_64。

**解决**:

```bash
# 让 mise 用 Rosetta 装
arch -x86_64 mise install node@14
# 或全局配置
mise settings set node.compile false   # 不本地编译,用预编译二进制
```

### 11.7 Python `python3` vs `python`

**问题**:`mise install python@3.12` 后,`python` 命令找不到,只能 `python3`。

**原因**:Python 官方二进制只提供 `python3` 链接。

**解决**:

```toml
# .mise.toml
[tools]
python = { version = "3.12", virtualenv = ".venv" }

# 或者在 alias 里
# ~/.zshrc
alias python=python3
```

或者用 `uv`(Astral 出的 Python 工具链),mise 也支持:

```bash
mise use uv@latest
```

## 十二、性能调优:让 mise 更快

```toml
# ~/.config/mise/config.toml

plugin_autoupdate_last_check_duration = "never"
```

```toml
[settings]
disable_default_shorthands = false
# mise 启动时不检查自身更新
```

### 12.3 用 systemd-style cache

```toml
[settings]
verbose = false
quiet = true            # 不输出进度条等
```

### 12.4 检查启动耗时

```bash
# 看 mise activate 那一段耗时
time eval "$(mise activate zsh)"

# 如果还是 > 50ms,跑 doctor
mise doctor
```

## 十四、反对的写法

```
6. 用 mise 装 ripgrep / fd / bat
   ──> 这些是全局工具,跟项目无关
   ──> 走 brew / apt / nix 就好,mise 浪费配置

7. shell rc 里手动 export NODE_VERSION
   ──> 这种全局 hack 在切项目时会污染
   ──> 用 mise [env] 段集中管

8. .env 文件里写 secret 然后 .gitignore
   ──> 老员工有 secret、新员工没有
   ──> 离职员工电脑里还有 secret
   ──> 用 mise + 1Password / vault

9. CI 里手动 setup-node / setup-python action
   ──> CI 版本和本地不一致,经常错

10. 从 asdf 迁过来不卸 asdf
    ──> shim 残留,PATH 混乱
    ──> mise reshim + rm -rf ~/.asdf

11. mise activate 放错位置
    ──> 被 conda init / 其他 PATH 修改抢
    ──> 放在 ~/.zshrc 最末尾稳

12. 期待 mise 处理 rustup 的所有能力
    ──> rustup 还做 toolchain / component / target
    ──> 简单场景 mise 够,复杂 Rust 项目仍用 rustup
```

- **决定哪些工程师该跟你升级到 Nix**(25 篇)

如果上面 8 条你都能做到,**这一篇就值了**。

## 十六、下一篇预告

**`25-Nix心智.md`**——这一篇讲了「**一个工具吃版本管理 + env + 任务**」,**下一篇讲「一个工具吃整个操作系统**」。

```
mise 解决"项目级"的可复现:
   - 哪些语言版本
   - 哪些 env
   - 哪些任务
   
但 mise 不解决:
   - 这台机器的系统包是什么版本?(brew install postgresql 16,半年后是 17)
   - 我的 dotfiles 应该是什么?(谁手动写过 ~/.gitconfig)
   - 我装了哪些 GUI 应用?(brew --cask 状态不可控)
   - 这台机器 5 年后能不能跑出今天一模一样的环境?
   - 团队跨 macOS / Linux,brew / apt 给出的版本不一致怎么办?

Nix 解决这一切:
   - 装的所有东西在 /nix/store/HASH-name-version/,内容寻址
   - 整台机器的状态用 flake.nix 描述
   - flake.lock 锁定所有依赖到 patch 级
   - 跨平台一致(Linux + macOS,同 flake 跑同结果)
   - 原子升级 / 回滚(generation 机制)
   
代价:
   - 学一门新语言(Nix DSL)
   - 学一套新心智(声明式 vs 命令式)
   - 文档碎、错误信息差
   - 80% 的人两周后放弃
```

下一篇讲清楚:**Nix 是什么、为什么这么难、谁该学、入坑路径**——读完你能判断"我团队 / 我自己该不该上 Nix"。**这是这个系列最分裂的一篇**——粉丝说"包管理的尽头",反对说"学曲线对得起十年工资",两边都有道理。
