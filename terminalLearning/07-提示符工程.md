# 提示符工程:prompt 是热路径不是审美

新人配 zsh 的标准流程:**装完 oh-my-zsh / zinit,下一步就是搜「好看的 prompt 主题」**——agnoster 装上、看一眼颜色变了、感觉"配置完成"。这就是 90% 工程师对 prompt 的理解:**审美问题**。换个主题就像换个手机壳,装完拍照晒群,继续干活。

**这是这一篇要彻底拆掉的认知**。prompt 不是手机壳,**它是你每次按下回车都会跑一次的代码**——一个普通工程师一天敲 500 次回车,你的 prompt 渲染逻辑就跑 500 次。一个 200ms 的 prompt = 一天浪费 100 秒纯等待,**比 06 篇讨论的"启动 shell 慢 200ms"严重 50 倍**——后者一天 5 次,前者一天 500 次。**prompt 是热路径(hot path),不是冷启动**。

更狠的是:**你按下回车到看见下一行 prompt 出现的延迟,是工程师对"系统响应速度"最敏感的体感指标**。这个延迟一旦超过 100ms,潜意识就开始"等",敲命令的节奏被打乱,**生产力悄无声息地往下掉**。

> 一句话先记住:**prompt 是热路径(hot path),不是配置——任何同步 IO(git status / kubectl / aws sts)放进 prompt 必然成为瓶颈**。要么用异步方案(Starship / P10k 的 instant prompt),要么粗暴只显示 cwd——**没有第三条路**。

这一篇拆开来讲:**prompt 渲染的延迟死亡螺旋怎么形成**、**为什么 Starship 必须用 Rust 写**、**P10k 的 instant prompt 凭什么把感知延迟降到 0**、**自己写 PROMPT 写到什么程度该停手**。**不教你"配一个炫酷 prompt 显示 12 种状态"**——那是给截图用的,不是给生产用的。

## 一、为什么 prompt 是工程问题不是审美问题

### 1.1 一次回车的代价

按下 Enter 实际发生的事:

```
按下 Enter
  ↓
shell 执行刚刚的命令(time = T_cmd,你关心)
  ↓
命令结束,shell 回到提示符状态
  ↓
shell 调用 precmd 钩子 / 渲染 PROMPT(time = T_prompt,你不关心)
  ↓
prompt 显示出来,你能敲下一条命令

真正的"响应延迟" = T_cmd + T_prompt
```

**T_cmd 是业务时间,你心理上接受**——`make build` 跑 30 秒,你认了。**T_prompt 是渲染时间,你心理上不接受**——`ls` 瞬间出结果,但 prompt 卡 300ms 你才能敲下一条,**这 300ms 就是纯工程缺陷**。新人感知不到"prompt 卡了 300ms"这件事,**只感觉"terminal 怪怪的"、"我电脑慢了"**——根因被归到 git / 电脑头上,**优化思路就错了**。

### 1.2 一天敲 500 次回车这个数字哪来的

```
一次开发循环:写代码 → 切终端 → 跑命令 → 看结果 → 改代码
   ↓ 每个循环 3-5 次 Enter
专注 4 小时 = 240 分钟,平均每分钟 2 次回车
   ↓
480 次 Enter / 天(就算砍半算 250 次)
```

200ms 的 prompt 延迟 = 一天 50 秒纯等待。**这不是连续的 50 秒**(那你可以喝咖啡),**是 250 次每次 200ms 的卡顿**——每次都把注意力拉回"为什么这么慢"。**注意力切换的代价远大于等待本身**。

### 1.3 跟"启动 shell 慢"的对比

| 维度 | shell 启动延迟(06 篇) | prompt 渲染延迟(本篇) |
| --- | --- | --- |
| 触发频率 | 一天 5-10 次 | 一天 200-500 次 |
| 单次延迟 | 50-300ms | 50-500ms |
| 累计每天 | 250ms - 3s | 25s - 4 分钟 |
| 心理感受 | "新开终端有点慢" | "我电脑变卡了" |
| 优化优先级 | 中 | **高** |
| 一周自愈率 | 你习惯了不再注意 | 永远不习惯 |

**结论**:**06 篇调优启动时间是大头,但 07 篇调优 prompt 是更大的头**——前者冷路径,后者热路径。大家都在卷启动时间,**没人卷热路径**——这是 zsh 工程化里最被低估的一块。

## 二、prompt 的演化简史

要理解 2026 的选型,得知道这条线怎么走过来。

### 2.1 时间线

```
1980s              1990s              2010s              2020+
  │                 │                  │                  │
  ▼                 ▼                  ▼                  ▼
单字符        bash 默认 PS1       同步主题时代        异步 prompt 时代
$ / >        user@host:cwd$       agnoster / pure     Starship / P10k
  │                 │                  │                  │
  └─ csh / sh       └─ bash / tcsh     └─ oh-my-zsh       └─ instant prompt
                                          时代              transient prompt
                                          首次卡顿          异步 worker
```

每一代解决上一代的什么痛点:

- **1980s** `$` `>`:终端就是 80x24,prompt 越短越好——**无信息**,你完全靠记忆维护上下文
- **1990s** `user@host:cwd$`:多用户系统、ssh 跳板出现,**user / host / cwd 三件套定下来**
- **2010s** agnoster / pure:Git 普及,prompt 变成"工程上下文展示面板"。**agnoster 把 git_prompt_info / ruby_prompt_info / virtualenv_prompt_info 全做成同步函数**——每次回车串行跑一遍,**慢仓库直接卡 200ms+**
- **2020+** Starship / P10k:**异步成为标配**。Starship 用 Rust 重写,每个 segment 独立超时;P10k 首创 instant prompt——**先显示缓存,后台再加载真实配置**

### 2.2 2026 主流 prompt 工具对比

```
┌─────────────────────────────────────────────────────────────┐
│ 工具          │ 语言    │ 跨 shell │ 异步策略         │ 状态  │
├───────────────┼─────────┼──────────┼──────────────────┼───────┤
│ Starship      │ Rust    │ 是       │ 子进程并发+超时  │ 活跃  │
│ Powerlevel10k │ zsh     │ 否(zsh)│ instant prompt   │ 半停滞│
│ Oh-My-Posh    │ .NET    │ 是       │ 子进程并发       │ 活跃  │
│ Tide          │ fish    │ 否(fish)│ fish 异步特性    │ 活跃  │
│ pure          │ zsh     │ 否(zsh)│ 异步 git         │ 维护  │
│ spaceship     │ zsh     │ 否(zsh)│ 同步             │ 慢   │
│ agnoster      │ zsh     │ 否(zsh)│ 同步             │ 不推荐│
└─────────────────────────────────────────────────────────────┘
```

**2026 视角**:**主推 Starship**(跨 shell、Rust 速度、配置简单)、**P10k 仅保留旧装机**(新装不推)、**Tide 仅限 fish-only 用户**、**agnoster / spaceship 该退休了**。

### 2.3 为什么 Starship 用 Rust 写

prompt 渲染的硬约束:

```
- 每次回车都跑一次
- 启动延迟必须 < 10ms(否则就掉到表里的"慢"档)
- 单二进制,易分发(跨 shell 跨 OS)
- 内存常驻不合理(每次回车都 fork)

候选语言启动延迟:
   Bash / Zsh:< 1ms,但表达力差、跨 shell 不通用
   Python:    50-100ms(import 慢),pass
   Node.js:   100-200ms,pass
   Go:        5-10ms,二进制 10MB,可以
   Rust:      1-3ms,二进制紧,极致
```

**Rust 是这道题的最优解**。**Oh-My-Posh 用 .NET 写的代价就是启动慢(冷启 30-50ms)**,实际体验有差距。**prompt 是「每次回车都重新 fork 一个进程跑一遍」的程序——只有 Rust / Go 这种"启动即响应"的语言才扛得住**。

## 三、prompt 的"延迟死亡螺旋"

prompt 慢起来不是单点慢,是**多个同步 IO 串行累加**。

### 3.1 一份典型同步 prompt 都在做什么

```
PROMPT = "$(cwd) $(git_branch) $(k8s_ctx) $(aws_profile) $(node_ver) %# "
              │           │           │            │             │
              │           │           │            │             ▼
              │           │           │            │       读 .nvmrc + nvm current
              │           │           │            │       ~ 50-200ms(shim 加载)
              │           │           │            ▼
              │           │           │     aws sts get-caller-identity
              │           │           │     ~ 100-500ms(网络调用!)
              │           │           ▼
              │           │    kubectl config current-context
              │           │    ~ 10-50ms(读 kubeconfig)
              │           ▼
              │    git rev-parse --abbrev-ref HEAD
              │    git status --porcelain
              │    ~ 5-200ms(看仓库大小 / 网络盘)
              ▼
           pwd / 简单
           ~ 1ms

────────────────────────────────────────────────
累计:166-951 ms 每次回车
```

**一个有 aws sts 的 prompt 在网络抖动时直接卡半秒以上**——真实数据,不是危言耸听。

### 3.2 真实测量

我在不同场景测过,**每次回车 + 第一行 prompt 出现的时间**(M2 / starship 默认配置 / 同步 git):

| 场景 | prompt 延迟 |
| --- | --- |
| 空目录(`/tmp`) | 8 ms |
| 普通 git 仓库(< 1k 文件) | 25 ms |
| 大 monorepo(> 50k 文件) | 180 ms |
| iCloud 同步盘里的 git 仓库 | 450 ms |
| 网络挂载盘 (NFS) 里的 git | 1200 ms |
| 同上 + 加 aws sts segment(VPN 在用) | 1800 ms |

**网络挂载盘里写代码、prompt 显示 git + aws——一次回车卡接近 2 秒**。这是真实工程师每天的体感。

### 3.3 怎么测自己的 prompt 延迟

**陷阱**:`time zsh -i -c exit` 只测启动,**不测稳态 prompt 渲染**。要测交互场景:

```bash
# 方法 1:cd 到目标目录,跑一个空命令,测整体回车响应
cd ~/big-monorepo
time true                   # 这个时间约等于一次"回车后下次 prompt 出现"的时间

# 方法 2:用 zsh 钩子直接测
typeset -F SECONDS
preexec() { _start=$SECONDS }
precmd() {
  local elapsed=$(( SECONDS - _start ))
  printf 'prompt render: %.3fs\n' $elapsed
}

# 方法 3:Starship 自带诊断(最强)
STARSHIP_LOG=debug starship explain
# 列出当前目录每个 segment 花了多少 ms
```

**方法 3 是最强的**——Starship 直接告诉你**哪个 segment 慢**。

**启动延迟 vs prompt 延迟**对比一下:

```
启动延迟:time zsh -i -c exit → 50ms,一天开 5 个 terminal,总成本 250ms
prompt 延迟:cd ~/repo && time true → 200ms,一天 500 次回车,总成本 100s
```

**两者差 400 倍**。即使 06 篇花了一整篇调启动,**本篇花一整篇调 prompt 也是必要的**。

## 四、同步 prompt 的本质问题

为什么"同步 prompt"必然是死胡同?讲机制。

### 4.1 shell 的 prompt 是个变量

不管 bash 还是 zsh,prompt 的本质是**一个字符串变量**,shell 在显示前会做变量展开:

```bash
# bash
PS1='\u@\h:\w\$ '            # 静态字符串,转义符自动展开

# zsh
PROMPT='%n@%m:%~ %# '         # 同上,只是转义符不同

# 想要动态怎么办?用函数 + setopt PROMPT_SUBST
function _git_branch() {
  git rev-parse --abbrev-ref HEAD 2>/dev/null
}
setopt PROMPT_SUBST           # 允许 PROMPT 里展开 ${...} 和 $(...)
PROMPT='%~ $(_git_branch) %# '
```

**关键的事**:**`$(_git_branch)` 在每次 prompt 显示前都会被 fork 一个子 shell 执行**——你 prompt 字符串里多少个 `$()`,就 fork 多少次。

### 4.2 为什么不能 cancel

```
你按 Ctrl-C 想中断 prompt 渲染?
   ↓
但 prompt 渲染发生在"shell 准备接收下一条命令前"
   ↓
Ctrl-C 是发给"当前前台进程组"的 SIGINT
   ↓
而 prompt 渲染时,前台进程组 = shell 自己
   ↓
shell 自己不会 SIGINT 自己
   ↓
你只能等 git status 跑完
```

**实战体感**:cd 到一个慢仓库,prompt 卡 5 秒,**这 5 秒里按 Ctrl-C 根本没用**——只能干等。

### 4.3 monorepo 杀手:git status

`git status` 是 prompt 同步 IO 里**最常见也最阴的杀手**:

```
小仓库:< 5ms        → 你以为没问题
中仓库:5-50ms       → 还行
大 monorepo:50-500ms → 你开始觉得 zsh 慢
有未追踪且很多:1-5 秒 → 你重启 terminal

更恶心的场景:
   - git 工作目录在网络盘(NFS / iCloud / Dropbox / OneDrive)
   - 仓库带大量大文件(node_modules / target/ / build/)
   - 没配 .gitignore 导致 git 要 stat 所有文件
   - git lfs 仓库,checking lfs status
```

**每一个都能让 git status 从 5ms 变成 5 秒**。

**自检**:`cd <你常待的项目> && time git status > /dev/null`——这就是你 prompt 的下限。

### 4.4 反模式集合

prompt 里出现就是错:

| 错误写法 | 原因 |
| --- | --- |
| `$(git status ...)` | git 在大仓库慢 |
| `$(kubectl config current-context)` | kubectl 启动 50ms+ |
| `$(aws sts get-caller-identity ...)` | 网络调用 |
| `$(curl ...)` | 网络调用,头号杀手 |
| `$(eval $(complex_command))` | 每次回车 fork 整套链 |
| `$(node --version)` | node 启动 30ms+ |
| `$(python -c '...')` | python 启动 50ms+ |

**这些放进 PROMPT 等于把每次回车都变成一次小型 RPC 调用**——必死。

## 五、异步 prompt 的三种实现

既然同步是死路,异步是唯一出路。

### 5.1 三种实现对比

```
┌───────────────────────────────────────────────────────────────┐
│  实现 1:transient prompt(P10k 首创,Starship 也支持)        │
├───────────────────────────────────────────────────────────────┤
│   T0:     用户按 Enter                                        │
│   T0+1ms: 显示"快速 prompt"(只有 cwd / %#)                  │
│   T0+200ms: 后台跑完 git status / k8s ctx                     │
│   T0+201ms: 重绘 prompt,显示完整状态                          │
│                                                               │
│   用户感知 = 1ms,实际渲染 = 201ms                             │
│   ✓ 用户感知"瞬间"                                            │
│   ✗ 200ms 内显示的 prompt 信息不全                            │
└───────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│  实现 2:async worker(zsh-async / zsh-defer)                  │
├───────────────────────────────────────────────────────────────┤
│   T0:       用户按 Enter                                      │
│   T0+1ms:   显示"快速部分"(cwd / 退出码)                     │
│   T0+1ms:   fork worker 子进程跑慢任务(git / k8s)            │
│   T0+200ms: worker 完成,通过 USR1 信号通知主 shell            │
│   T0+201ms: 主 shell 重绘 prompt,把异步结果填上               │
│                                                               │
│   ✓ 灵活,每个 segment 单独控制                                │
│   ✗ 实现复杂,zsh-only,IPC 用信号不优雅                       │
└───────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────┐
│  实现 3:跨进程 cache + 单二进制(Starship)                    │
├───────────────────────────────────────────────────────────────┤
│   T0:       用户按 Enter                                      │
│   T0+1ms:   shell 调用 `starship prompt`(Rust 进程)          │
│   T0+1ms:   starship 内部并发跑所有 segment                   │
│             每个 segment 有独立 command_timeout(默认 500ms)  │
│   T0+200ms: starship 输出 prompt 字符串                       │
│   T0+201ms: shell 显示                                        │
│                                                               │
│   ✓ 跨 shell 通用 / 单二进制 / 内部并发                       │
│   ✓ Rust + 超时控制,worst case 也就 500ms                     │
└───────────────────────────────────────────────────────────────┘
```

### 5.2 三种实现的边界

- **transient prompt**:用户体验最好,**但要求 shell 支持 prompt 重绘**——zsh 用 `zle reset-prompt` 实现,bash 不支持(bash readline 不让你重绘已显示的 prompt)。**所以 bash 用户基本只能用 Starship**。

- **async worker**:zsh 生态特有,**P10k 内部用的就是这套**——P10k 自己重写了一个 C++ 版本叫 gitstatusd,把 git status 做成常驻 daemon 子进程,prompt 渲染时只问 daemon 拿结果。**这是 P10k 快的核心秘密**——别的工具每次都 fork git,P10k 是直接和 daemon 对话。

- **跨进程 cache**:Starship 的思路——**信任 OS、信任 Rust 启动快、信任 segment 超时控制**。**对中型用户最易用**,worst case 仍是 segment 超时(500ms)。

### 5.3 选型表

| 你是 | 推荐实现 | 推荐工具 |
| --- | --- | --- |
| 多 shell 用户(bash + zsh + fish) | 跨进程 cache | **Starship** |
| zsh-only,要极致速度 | async worker(gitstatusd) | **Powerlevel10k** |
| fish-only,要漂亮 | 异步(fish 内建特性) | **Tide** |
| 不想装东西,40 行手写 | transient(zsh)/ 同步(bash) | **手写 PROMPT** |

## 六、Starship 深度

主推工具,**这一节给一份生产级配置**。

### 6.1 设计哲学

```
关键决定:
   1. Rust 写,单二进制(brew install starship 一行装完)
   2. 跨 shell:bash / zsh / fish / elvish / powershell / cmd
   3. 配置统一在 ~/.config/starship.toml(TOML)
   4. 每个 segment 独立,可单独开关
   5. 每个 segment 独立超时(command_timeout,默认 500ms)
   6. 不维护内部 daemon,每次回车都是新进程
   7. 启动延迟 < 5ms(Rust + 单二进制 + 无 JIT)
```

**启动 < 5ms 是关键**——所以即使每次回车都 fork,也感觉不到。

### 6.2 装 + 启用

```bash
# macOS
brew install starship

# Linux
curl -sS https://starship.rs/install.sh | sh

# .zshrc / .bashrc / config.fish 任选
eval "$(starship init zsh)"
eval "$(starship init bash)"
starship init fish | source
```

`eval "$(starship init zsh)"` 干的事:**注入一个 `precmd` 钩子**,每次回车调用 `starship prompt`,把输出赋给 `PROMPT`。

### 6.3 一份生产级 starship.toml

放 `~/.config/starship.toml`,**直接抄能用**:

```toml
# ~/.config/starship.toml
# 生产级配置:cwd + git + 语言版本 + k8s + aws + 退出码
# 目标:single-line 显示,延迟 < 50ms

# ─────── 全局 ───────
# 单次 prompt 渲染整体超时(超过这个全部 segment 都被砍)
command_timeout = 1000          # 1 秒,网络慢时给点缓冲;追求极速可改 300
scan_timeout = 30               # 扫描当前目录(检测语言项目)超时
add_newline = false             # 不在 prompt 前加空行(单行省屏幕)

# ─────── 格式 ───────
# 模块顺序:directory > git > 语言版本 > k8s > aws > 行尾退出码
# 注意每个 $module 不写就不显示
format = """
$directory\
$git_branch\
$git_status\
$python\
$nodejs\
$rust\
$golang\
$kubernetes\
$aws\
$cmd_duration\
$line_break\
$character"""

# ─────── 字符 ───────
[character]
success_symbol = "[➜](bold green)"     # 成功:绿色箭头
error_symbol = "[➜](bold red)"          # 失败:红色箭头(一眼能看到)
vimcmd_symbol = "[V](bold cyan)"        # vi 模式(若启用)

# ─────── 目录 ───────
[directory]
truncation_length = 3                   # 路径最多显示 3 级
truncate_to_repo = true                 # 在 git 仓库里截到 repo 根
style = "bold cyan"
read_only = " 󰌾"                        # 只读目录显示锁(nerd font)
read_only_style = "red"

# ─────── Git 分支 ───────
[git_branch]
symbol = " "                           # nerd font 的 git 图标
style = "bold purple"
truncation_length = 20                  # 分支名太长截断

# ─────── Git 状态 ───────
# 这个 segment 是最容易卡的,慎重
[git_status]
style = "yellow"
ahead = "↑${count}"
behind = "↓${count}"
diverged = "↕↑${ahead_count}↓${behind_count}"
modified = "!${count}"
staged = "+${count}"
untracked = "?${count}"
conflicted = "✗"
disabled = false

# ─────── 命令耗时 ───────
[cmd_duration]
min_time = 2_000                        # 跑超过 2 秒才显示耗时
format = "took [$duration](yellow) "
show_milliseconds = false

# ─────── Python ───────
[python]
symbol = " "
format = '[${symbol}${pyenv_prefix}(${version})( \($virtualenv\))]($style) '
style = "yellow bold"
pyenv_version_name = false              # 不用 pyenv 时不调它,省 50ms
detect_extensions = ["py"]              # 只在有 .py 文件的目录显示

# ─────── Node.js ───────
[nodejs]
symbol = " "
style = "bold green"
detect_files = ["package.json", ".nvmrc", "tsconfig.json"]
detect_folders = ["node_modules"]

# ─────── Rust / Go ───────
[rust]
symbol = " "
style = "bold red"
detect_files = ["Cargo.toml"]

[golang]
symbol = " "
style = "bold cyan"
detect_files = ["go.mod", "go.sum"]

# ─────── Kubernetes(SRE / k8s 项目才开) ───────
[kubernetes]
symbol = "☸ "
style = "bold blue"
disabled = false                        # 不需要的人改 true
format = '[$symbol$context( \($namespace\))]($style) '
detect_files = ["k8s.yaml", "kustomization.yaml", "Chart.yaml"]
detect_folders = ["k8s", ".kube"]

# ─────── AWS(默认关,有云项目再开) ───────
# 注意不显示账号 id,只显示 profile 名(避免 aws sts 卡)
[aws]
symbol = "  "
style = "bold yellow"
disabled = true
format = '[$symbol($profile )(\($region\) )]($style)'

# ─────── 显式关掉的 segment(噪声) ───────
[hostname]
disabled = true                         # 本地无意义,ssh 单独配
[username]
disabled = true
[battery]
disabled = true                         # 系统栏看就行
[time]
disabled = true                         # 终端模拟器有时间
[memory_usage]
disabled = true                         # fork shell 看内存浪费
[package]
disabled = true                         # 在 Cargo.toml / package.json 目录显示版本号,没用
```

**设计取舍**:

1. **single-line**(`add_newline = false`):节省屏幕
2. **command_timeout = 1000**:平衡"等慢调用"和"卡用户"
3. **kubernetes / aws 默认关**:这俩慢且不是所有人用
4. **language detection 用 detect_files**:只在有 `package.json` 的目录显示 Node 版本
5. **明确 disable 噪声 segment**:hostname / username / battery / time 全关

### 6.4 怎么禁掉不要的 segment

每个 segment 配 `disabled = true`。**或者直接从 `format` 里删掉**——那样 segment 根本不被调用,**比 disabled 更彻底**。`format` 不引用的 segment **不会被执行**,首选这种。

### 6.5 实测延迟

我在 M2 上跑的这版配置,**不同场景**:

| 场景 | starship explain 出来的延迟 |
| --- | --- |
| 空目录 | 3 ms |
| 小 git 仓库(<1k 文件) | 12 ms |
| node 项目(有 package.json) | 18 ms |
| 大 monorepo | 80 ms |
| 大 monorepo + 开 k8s segment | 110 ms |
| 同上 + 开 aws segment(有 VPN) | 350 ms |
| 同上 + aws + 网络抖动 | 1000 ms(被 command_timeout 砍) |

**只要不开 aws,延迟基本都在 100ms 以内**——这就是 prompt 工程的硬目标。

## 七、Powerlevel10k 深度

P10k 还在用的人离不开它,**主要因为一个东西:instant prompt**。

### 7.1 instant prompt 怎么做到 0 延迟

```
正常 zsh 启动:
   load .zshrc(全部插件 source) → 50-300ms
   渲染 prompt → 5-50ms
   你看到 prompt

P10k instant prompt:
   zsh 启动
   ↓
   P10k 先读 ~/.cache/p10k-instant-prompt-<user>.zsh(上次的 prompt 字符串)
   ↓ 2-5ms
   直接显示这个"上次的 prompt"  ← 用户看到!
   ↓
   后台继续 source .zshrc / 加载插件
   ↓ 200ms
   .zshrc 加载完,P10k 重绘真实 prompt
```

**用户感知**:**zsh 启动瞬间(< 10ms)就出 prompt**,可以马上敲命令。**唯一副作用**:你打第一条命令的前 200ms 里,**插件还没全部加载**(syntax highlighting 可能还没启)——对大多数人无所谓。

### 7.2 启用 instant prompt

```zsh
# 必须放在 .zshrc 的最最最前面(任何 source 之前)
if [[ -r "${XDG_CACHE_HOME:-$HOME/.cache}/p10k-instant-prompt-${(%):-%n}.zsh" ]]; then
  source "${XDG_CACHE_HOME:-$HOME/.cache}/p10k-instant-prompt-${(%):-%n}.zsh"
fi

# ... 你的其他 .zshrc 内容 ...

# .zshrc 最后:
source ~/.zinit/plugins/powerlevel10k/powerlevel10k.zsh-theme
[[ ! -f ~/.p10k.zsh ]] || source ~/.p10k.zsh
```

P10k 第一次跑会引导你做配置(`p10k configure`),生成 `~/.p10k.zsh`——**几百行**,你不需要懂每一行。

### 7.3 .p10k.zsh 关键片段

只看几个对工程意义大的:

```zsh
# 双行 + transient(信息密度高 + 提交后简化)
typeset -g POWERLEVEL9K_PROMPT_ON_NEWLINE=true
typeset -g POWERLEVEL9K_TRANSIENT_PROMPT=always

# 左侧 prompt 显示什么
typeset -g POWERLEVEL9K_LEFT_PROMPT_ELEMENTS=(
  dir vcs prompt_char
)

# 右侧 prompt
typeset -g POWERLEVEL9K_RIGHT_PROMPT_ELEMENTS=(
  status command_execution_time background_jobs
  context kubecontext aws time
)

# gitstatusd 配置(P10k 的 git 异步秘密)
# 这个值控制 gitstatusd 如何检查 "dirty" 状态
typeset -g POWERLEVEL9K_VCS_MAX_INDEX_SIZE_DIRTY=-1
#  -1 = 总是检查(慢)
#   0 = 从不检查
#   n = 文件数 < n 时检查(平衡)
```

**P10k 最强的就是 gitstatusd**——一个 zsh 启动时拉起的常驻 C++ 子进程,**专门跑 git status**,prompt 渲染时直接问它要结果。比 starship 每次 fork git 还快。

### 7.4 P10k 的问题

1. **绑死 zsh**:跨 shell 用不了
2. **配置文件 .p10k.zsh 几百行**,一旦想手改某个细节,要在 500 行里翻
3. **维护停滞**:原作者 romkatv 2022 末几乎不再活跃,仓库还在,但新 issue 响应慢
4. **instant prompt 偶尔出 bug**:.zshrc 里如果有 echo / print,会污染 instant prompt 缓存,需要 `typeset -g POWERLEVEL9K_INSTANT_PROMPT=quiet` 抑制

### 7.5 2026 该不该用 P10k

```
你已经在用 P10k 多年              → 继续用,没必要折腾
你是 zsh-only,追求极致速度        → 用 P10k(gitstatusd 还是最强)
你想跨 shell                      → 用 Starship
新装机器,从零开始                → 用 Starship(社区更活跃)
你在维护团队的 dotfiles           → 用 Starship(新人门槛低)
```

**简单说**:**已有用户保留、新装一律 Starship**。

## 九、手写 prompt(极简派)

不想装 starship / P10k——**也可以手写,40 行内**。

### 9.1 zsh 手写三档

```zsh
# ─────── 档位 1:最简 ───────
PROMPT='%~ %# '
# %~  = cwd,$HOME 显示成 ~
# %#  = root 时 #,普通用户 %

# ─────── 档位 2:带颜色 + git ───────
autoload -Uz vcs_info
precmd() { vcs_info }
zstyle ':vcs_info:git:*' formats '(%b)'        # %b = branch name
setopt PROMPT_SUBST                             # 允许 ${vcs_info_msg_0_} 展开
PROMPT='%F{blue}%~%f %F{purple}${vcs_info_msg_0_}%f %# '
# %F{color}  = 前景色开始
# %f         = 颜色重置
# vcs_info 是 zsh 自带的 git/svn 状态查询函数,同步,代码现成

# ─────── 档位 3:多行 + 颜色 + 退出码 + git ───────
autoload -Uz vcs_info
zstyle ':vcs_info:git:*' formats '(%b)'
zstyle ':vcs_info:git:*' actionformats '(%b|%a)'
setopt PROMPT_SUBST
precmd() { vcs_info }

# 第一行:cwd + git;第二行:箭头(退出码红/绿)
PROMPT='%F{cyan}%~%f %F{purple}${vcs_info_msg_0_}%f
%(?.%F{green}.%F{red})➜%f '
# %(?.A.B) = 三元:上次退出码 0 显示 A,否则 B
```

档位 3 实测延迟:**空目录 1ms / 中型 git 仓库 10-30ms / 大 monorepo 50-200ms(vcs_info 同步)**。

**手写边界**:**vcs_info 跑得动,40 行手写就够**——你不需要 starship / p10k。**vcs_info 跑不动的仓库(大 monorepo / 网络盘),才必须上异步方案**。

### 9.2 bash 手写 PS1

bash 比 zsh 难写——**没有 `%F{...}` 这种语法,只能用裸 ANSI 转义码**:

```bash
# 最简
PS1='\w \$ '
# \w = cwd(完整路径)/ \W = 只显示当前目录名
# \$ = root 时 #,普通用户 $

# 带颜色(已经开始难看)
PS1='\[\e[34m\]\w\[\e[0m\] \$ '
# \[...\]    告诉 bash 这里是不可打印的转义(不算长度,readline 才能算行宽)
# \e[34m     ANSI 蓝色
# \e[0m      重置
```

**bash 手写超过 5 行就该止损**——**复杂 PS1 不推荐手写,直接 `eval "$(starship init bash)"`**。

### 9.3 zsh 转义符速查

| 转义 | 含义 |
| --- | --- |
| `%~` | cwd,`$HOME` 显示为 `~` |
| `%/` | cwd,绝对路径 |
| `%c` 或 `%C` | cwd 的最后一段 |
| `%n` | 用户名 |
| `%m` / `%M` | 主机名(short / 完整) |
| `%#` | root 时 `#`,普通时 `%` |
| `%?` | 上一条命令退出码 |
| `%j` | 后台任务数 |
| `%D{...}` | 日期(strftime 格式) |
| `%F{color}` ... `%f` | 前景色开始 / 结束 |
| `%K{color}` ... `%k` | 背景色 |
| `%B` ... `%b` | 加粗开始 / 结束 |
| `%(?.A.B)` | 三元:`%?` 为 0 时 A,否则 B |

**记 10 个就够日常**——bash 的等效物完全没这么强。

## 十、prompt 该显示什么:信息密度 vs 噪声

**显示太多是仪表盘灾难,显示太少是失明**,这一节给取舍。

### 10.1 三档分类

```
必装(几乎所有场景):
   □ cwd(你在哪)— 没有这条等于失明
   □ exit code(上一条命令成功/失败)— 失败明显才能立刻发现
   □ prompt char($ / # / ➜)

看情况(开发场景):
   □ git 分支 — 在 git 仓库里才显示
   □ git 状态(脏/干净)— 写代码党
   □ 后台任务数 — 经常 `&` 的人

角色相关:
   □ 语言版本 — 装了 mise / asdf 多版本才需要
   □ k8s context — SRE / DevOps
   □ aws profile — 云开发
   □ Python venv — 数据 / ML
   □ command duration — 跑长命令的(测试 / 编译)

绝对别加:
   ✗ 时间 — 终端模拟器 / 系统栏已经有
   ✗ user@host — 本地无用,ssh 单独配
   ✗ 完整 IP / hostname — 视觉污染
   ✗ 电量 — 系统栏有
   ✗ 天气 / 一句箴言 — 你不是要写自传
   ✗ 当前 shell 名 — 你自己开的你不知道?
```

### 10.2 5 种工程师角色 → segment 推荐

```
┌──────────────────────────────────────────────────────────────────────┐
│  角色          │ 必开 segment                                          │
├────────────────┼───────────────────────────────────────────────────────┤
│ 前端           │ cwd / exit / git / node                              │
│                │ (可选:package.json 版本号 / pnpm/yarn 标识)          │
├────────────────┼───────────────────────────────────────────────────────┤
│ 后端(纯)     │ cwd / exit / git / 主语言(go/python/rust)            │
├────────────────┼───────────────────────────────────────────────────────┤
│ SRE / DevOps   │ cwd / exit / git / k8s context / aws profile         │
│                │ (强烈建议命令耗时 cmd_duration)                       │
├────────────────┼───────────────────────────────────────────────────────┤
│ 数据 / ML      │ cwd / exit / git / python venv / conda env           │
│                │ (cmd_duration 必装,实验经常跑几分钟)                 │
├────────────────┼───────────────────────────────────────────────────────┤
│ 全栈 / 跨语言  │ cwd / exit / git / 当前目录的语言(starship 自动检测) │
└──────────────────────────────────────────────────────────────────────┘
```

**关键**:**starship 的 detect_files 让"语言 segment 只在该目录显示"**——cd 进 Rust 项目,prompt 才有 Rust 版本;cd 出来,自动消失。

### 10.3 prompt 即仪表盘 = 反模式

```
错误的 prompt(我截图见过的真实案例):
┌───────────────────────────────────────────────────────────────┐
│ user@host:~/proj  branch:main (✓2 ↑1 +3 ?5)  py 3.12          │
│ (.venv)  node 20.10.1  go 1.22.1  rust 1.75  k8s:prod         │
│ aws:dev-profile  ☁ 23°C  ⚡ 78%  ⏱ 14:32:01                  │
│ ➜ █                                                            │
└───────────────────────────────────────────────────────────────┘

每次回车都跑 8 个 IO 调用,卡 600ms+
```

**经验**:**prompt 上每一个字符都要回答"为什么它在这里"**——回答不出就删。**3-5 个 segment 是甜点区**,**超过 7 个 segment 几乎一定有冗余**。

## 十一、多行 vs 单行 / transient

### 11.1 两种风格对比

```
单行(传统 bash 风格):
   ~/very/long/path/that/goes/on/proj $ npm test --grep "..."
                                       ↑ 命令被挤到屏幕右边,看不清前面 cwd

多行(2 行,info 在上、命令在下):
   ~/very/long/path/that/goes/on/proj (main)
   ➜ npm test --grep "..."
              ↑ 命令永远从行首开始,cwd 再长也不影响
```

**多行的优点**:cwd 长 / 信息多时不挤掉命令、视觉清晰。**多行的缺点**:多吃一行屏幕、复制日志时多了一行 prompt 装饰。

**推荐**:**默认多行**(P10k / Starship 都默认多行)。

### 11.2 transient prompt:鱼和熊掌兼得

```
当前行(正在输入):多行,显示完整状态
   ~/proj (main) (took 2s)
   ➜ █

回车后,这一行变简化版(只保留命令 + 退出码):
   ➜ npm test --grep "..."     ← 上一条命令,变成单行简洁版

新的当前行又是多行:
   ~/proj (main)
   ➜ █
```

**transient prompt** = **"打字时多行,提交后单行"**——视觉清爽 + 复制不脏 + 信息够。

**怎么开**:

```
Starship:zsh-transient-prompt 集成方案(github olets/zsh-transient-prompt)
P10k:typeset -g POWERLEVEL9K_TRANSIENT_PROMPT=always
```

**强烈推荐打开**——prompt 工程的"小众但高 ROI"配置。

## 十二、右侧 prompt(RPROMPT)

### 12.1 RPROMPT 是什么

```
zsh / fish 支持 RPROMPT(在 prompt 字符串后,把另一段字符串显示到屏幕右边)

   ~/proj (main)                                           took 2s | 14:32
   ➜ █
   ↑                                                       ↑
   PROMPT (左)                                            RPROMPT (右)

bash 不支持 RPROMPT
```

### 12.2 适合放什么

```
适合(右侧不抢屏):
   ✓ 时间戳(只在右侧不浪费 prompt 主体空间)
   ✓ 命令耗时(took 2s 之类)
   ✓ 当前后台任务数
   ✓ 当前 Python venv 名
   
不适合:
   ✗ 路径(已经在左侧)
   ✗ git 状态(关键信息别甩右边)
   ✗ 超过 30 字符的内容(小窗口被截断)
```

### 12.3 RPROMPT 的坑

复制粘贴可能误带 RPROMPT 内容——iTerm2 / Wezterm / Ghostty 处理得比较好,但老终端 / mosh / 部分 SSH 客户端可能复制空格。**解法**:**RPROMPT 不放任何"被搜索时需要精准的"内容**。

一个推荐配置(zsh):

```zsh
# 只在命令跑 ≥ 2 秒时显示耗时
function _cmd_time() {
  if [[ -n $_start ]]; then
    local elapsed=$(( SECONDS - _start ))
    (( elapsed >= 2 )) && echo "${elapsed}s"
  fi
}
preexec() { _start=$SECONDS }
RPROMPT='%F{yellow}$(_cmd_time)%f %F{cyan}%D{%H:%M}%f'
```

## 十四、反对的写法

实际项目里反复见到的 prompt 反模式。

### 14.1 同步 prompt + 大仓库

```zsh
PROMPT='%~ $(git status --porcelain | wc -l)-dirty %# '
```

**monorepo 里这一行卡 200-500ms**。**解法**:上 Starship 或 P10k 的异步 git。

### 14.2 显示完整 hostname / IP / 时间戳秒

```
errors-prod-us-east-1.k8s.cluster.internal.our.company.com:~/proj $
[2026-05-12 14:32:45.123] ~/proj $
```

视觉污染、注意力杀手。**解法**:`%m` 显示 short hostname、或干脆别显示;时间戳放 RPROMPT 且只显示时分(`%D{%H:%M}`)。**hostname 只在乱跳 ssh 时有用,本地永远没用**。

### 14.3 emoji 一大堆

```
🚀 ~/proj 🌿 main ✨ 🐍 py3.12 📦 node20 ⚡ k8s ☁ aws ➜
```

**问题**:终端字体不全时显示成方框、emoji 占两个字符宽 readline 算错光标错位、跨平台显示不一致、视觉噪声大于信息量。

**解法**:**用 nerd font 的图标**(单字符宽、设计统一),一个 prompt 上 1-2 个图标够了。

### 14.4 prompt 里 eval $(complex_command)

```zsh
PROMPT='$(eval $(_complex_logic_that_calls_5_tools)) %# '
```

**每次回车 fork 一次整条调用链**——必死。

**解法**:**precmd 钩子里算一次,存到变量,prompt 引用变量**:

```zsh
precmd() {
  _my_state=$(_complex_logic_that_calls_5_tools)
}
PROMPT='${_my_state} %# '
```

仍然慢(`_complex_logic` 还是同步跑),但**只算一次,不是 fork 一次**。最终解法还是异步。

### 14.5 24 位 RGB 颜色

```zsh
PROMPT='%F{#3a86ff}%~%f %# '
```

**24 位真彩色支持依赖**:终端 + tmux + ssh 三层都要支持。实际链路:本地 iTerm2 支持 truecolor → tmux 默认不传降级 256 色 → ssh 服务器 TERM=xterm 进一步降级 → 颜色变怪。

**解法**:**用 256 色名**(red / green / yellow / blue / magenta / cyan / white)——**所有地方显示一致**。

### 14.6 大量颜色拼贴 + 红色脱敏

```
%F{red}A%f%F{green}B%f%F{blue}C%f%F{yellow}D%f%F{magenta}E%f
```

每段都换色 = 视觉嘈杂,一眼扫过去找不到重点。

**解法**:**色彩留给"出错时"用**——正常状态单色 + 加粗对比、出错时变红。**日常滥用红色会脱敏**。

### 14.7 不测就说"我配置很快"

```
你同事:"我 .zshrc 50 行,prompt 飞快"
   ↓
你 cd 到他的 monorepo,prompt 卡 400ms
   ↓
他没测过
```

**解法**:**任何 prompt 配置都附带 starship explain / time true 数据**——没数据的"快"是吹牛。拿数据说话。
