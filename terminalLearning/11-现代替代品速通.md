# 现代替代品速通:rg / fd / bat / eza / dust / sd / hyperfine / delta

2015 年之后,Hacker News 上每隔一周就冒出一篇「我用 Rust 重写了 X」——X 可能是 cat、可能是 sed、可能是 top。**这种帖子 80% 是噪声**:有的作者只是练手 Rust、有的换了个名字加点彩色、有的解决的是作者自己想象出来的问题。**但剩下 20% 是真的好东西**——一年内进入主流工程师的 dotfiles、被 GitHub 几万星认可、真的替代了某个传统工具的位置。这一篇就是把这 20% 挑出来,讲清楚**它们好在哪、什么时候该换、什么时候不能换**。

这一波工具的共同特征:**Rust 写、SIMD/并行加速、默认尊重 .gitignore、彩色对齐、跨平台一份二进制**——拿出来跟传统 Unix 工具放在一起,人眼一看就知道哪个是 2026 的。**但「现代」不等于「替换」**:你的脚本永远只能依赖 POSIX,SSH 进陌生机器永远只有 grep / find / cat,容器里 Alpine 镜像永远不会预装 rg。**这一篇的核心立场是:学新的当 daily driver,留旧的当 fallback**——不是「全部 alias 成新工具」的盲目升级。

> 一句话先记住:**Rust CLI 工具普遍比传统 Unix 工具快 5-10×、UX 好一档、跨平台,但不是无脑替换——你的脚本仍然只能依赖 POSIX,SSH 进陌生机器仍然只有 ls / grep / find。学新的当 daily driver,留旧的当 fallback**。

这一篇拆开来讲:**这一波 Rust CLI 工具普遍快的根因**、**核心 10 个工具速对表**、**rg / fd / bat / eza / dust / sd / hyperfine / delta 的最小可用配置**、**怎么不让"新工具上瘾症"发作**。**不抄 README**——README 在那儿,这里只补 README 没讲透的「为什么」和「什么时候不要装」。

## 一、为什么这波工具普遍快、UX 好

把这波工具放一起看,**它们好的理由是同一套**——不是某个工具单独牛,是 2015 年之后 CLI 工具的工程范式整体升级。

### 1.1 Rust + 现代算法 = 默认快一档

```
传统工具:           大多是 1970s-1990s 的 C 代码
   - 单线程居多
   - 用 glibc / regex 库,通用但慢
   - 没有 SIMD 优化
   - 不读 .gitignore

Rust CLI 工具:      2015 年之后 Rust 生态成熟后的产物
   - 内建多线程(rayon 并行)
   - 自带向量化字符串搜索(SIMD)
   - 把"递归 + 过滤 + 高亮"做成默认行为
   - 默认尊重 .gitignore / .ignore
```

**rg 的核心快源于 regex crate**——这个 crate 是 BurntSushi(ripgrep 作者)同一个人写的,内部用 lazy DFA(惰性确定性有限自动机) + SIMD。**fd 的快源于 ignore crate + rayon**——它和 rg 共享 ignore 这个库,所以两者「忽略规则」完全一致。**这种"基础库被多个工具共享"是 Rust 生态的红利**——传统 C 工具各写各的,质量参差。

### 1.2 gitignore 默认尊重——开发者不用再解释

新人第一次用 grep 在大仓库里搜代码:

```bash
grep -r 'foo' .
# 等了 30 秒
# 输出里 90% 是 node_modules / dist / .git/ 里的垃圾
# 翻不到真正想要的那一行
```

然后新人去搜「grep 怎么忽略 node_modules」,**学到一堆 `--exclude-dir` 的写法**——但每次都要写一遍。`rg 'foo'` 默认就跳过所有 .gitignore 里列出的目录——**对开发者来说,这是"减少认知负担"的工程胜利**。

```
rg / fd / dust / tokei 都默认尊重:
   .gitignore        ← git 配的
   .ignore           ← 这些工具特有的"通用 ignore"
   .rgignore         ← rg 特有
   .fdignore         ← fd 特有
   全局 ~/.config/git/ignore
```

**这套"分层 ignore"是 BurntSushi 设计的事实标准**——后来的工具基本都跟。**对比传统 grep**:每次手写 `--exclude-dir={node_modules,dist,.git}`,**重复劳动**。

### 1.3 默认彩色 + 表格对齐——人眼友好

传统 ls 输出:

```
README.md  src  package.json  node_modules  tests
```

eza --git --icons:

```
 README.md           Mar 12 10:30    -M
 src/                Mar 12 11:00
 package.json        Mar 10 09:00    -M
 node_modules/       Mar 08 14:00
 tests/              Mar 11 08:00
```

**`-M` 是 git 状态(modified)**,**图标是 nerd fonts 提供的字形**。**信息密度高了一档,但视觉负担没增加**——颜色和图标做了视觉分组。

**对比传统**:ls -la 你看 `-rw-r--r--@ 1 qijixin staff 9806 May 12 09:29 00-写作计划.md` 这种 timestamp 横版输出,**信息全在但读起来累**。

### 1.4 聪明默认 = 减少 flag 心智

```
grep 'foo' .              ← 报错,grep 不递归默认
grep -r 'foo' .           ← 要加 -r
rg 'foo'                  ← 默认递归

find . -name '*.py'       ← 默认全名匹配,要写 glob
fd '.py$'                 ← 默认正则,但匹配 path 子串

find . -name 'foo' -type f -exec grep 'bar' {} \;
                          ← find + grep 的经典组合,语法折磨人
rg 'bar' -g '*foo*'       ← rg 直接搞定
```

**"减少 flag 心智"是这波工具的核心 UX 胜利**——你不用记 `-r` 还是 `-R`、`--exclude` 还是 `--exclude-dir`、`find -name` 还是 `find -path`,工具帮你选好了 95% 场景下的合理默认。

### 1.5 跨平台一份二进制

```
brew install ripgrep      ← macOS
apt install ripgrep       ← Debian / Ubuntu
pacman -S ripgrep         ← Arch
cargo install ripgrep     ← 任何有 Rust 的地方
choco install ripgrep     ← Windows
```

**Cargo 编译产物是单二进制**——不依赖 glibc 版本、不需要装 N 个 .so、Windows / macOS / Linux 三平台都跑。**这一点是 Rust 生态送给 CLI 工具的最大红利之一**——传统 C 工具的跨平台编译是工程噩梦,Rust 一行 `cargo install` 完事。

**注意例外**:Alpine 用 musl libc,某些 Rust 工具的预编译版要么没有要么有 bug,**容器场景需要明确测一下**。

## 二、核心 10 个工具速对表

把这一篇要讲的 10 个工具放一起看:

| 旧 | 新 | 作用 | 「好在哪」一句话 | 难度 |
| --- | --- | --- | --- | --- |
| `grep` | `rg`(ripgrep) | 递归搜索 | 快 5-10×、默认 gitignore、彩色 | 平 |
| `find` | `fd` | 找文件 | 简洁语法、并行、默认 ignore | 平 |
| `cat` | `bat` | 看文件 | 语法高亮 + 行号 + git diff | 平 |
| `ls` | `eza` / `lsd` | 列目录 | 颜色 + 图标 + git 状态 + tree | 平 |
| `du` | `dust` | 看大小 | 可视化树状、扫得快 | 平 |
| `sed` | `sd` | 替换 | 更安全的语法、默认非 regex | 平 |
| `ps` | `procs` | 看进程 | 彩色 + 树状 + 实时 | 平 |
| `time` | `hyperfine` | benchmark | 多次跑取统计 / warmup / 对比 | 中 |
| `diff` | `delta` | 看 diff | 语法高亮 + 行号 + side-by-side | 中(配 git) |
| `top` | `bottom` / `btop` | 实时监控 | 图表 + 鼠标 + 跨平台 | 平 |

**几条经验观察**:

```
1. 这 10 个工具,如果只让你装 3 个:rg / fd / bat
   ——这三个是「装上立刻不可逆」的,工作流彻底变了
   ——剩下 7 个是「锦上添花」,可装可不装

2. 但 delta 极特殊:你装上配进 git,看 diff 的体验一辈子改变
   ——只是它和 git 强耦合,需要一次 ~/.gitconfig 改动

3. 监控类(procs / bottom / btop):本地用很爽,
   ——远端 / 容器里仍然只有 top / htop / ps
   ——这类工具的「跨机器复用」最弱

4. 不要装 30 个 must-have CLI:
   ——多数是"作者写来给自己用"的,你的工作流不是作者的
   ——装一个,用两周,真的快了就留;没感觉就卸
```

下面挨个拆。

## 三、rg(ripgrep)详解——这一波最值钱的工具

如果 2026 你的 macOS 上只装一个新 CLI,**装 rg**。**理由**:它彻底改变了「在代码库里找东西」这件事的体感,**而这件事工程师每天做 50 次**。

### 3.1 安装

```bash
brew install ripgrep
# 或
cargo install ripgrep
# 或
apt install ripgrep                  # Ubuntu 20.04+
```

**包名是 ripgrep,二进制叫 rg**——`rg --version` 检查。

### 3.2 默认行为(对照 grep)

**rg 的默认行为本身就是它的卖点**——不加任何 flag,它已经做了 90% 你想要的:

```
rg 'pattern'
   ↓ 默认
   - 递归当前目录(grep 要 -r)
   - 尊重 .gitignore(grep 不知道这是什么)
   - 二进制文件跳过(grep 默认会搜然后印一行 "Binary file matches")
   - 隐藏文件跳过(grep 默认搜)
   - 彩色输出(grep 默认不上色)
   - UTF-8 默认(grep 看 locale)
   - 多线程并行(grep 单线程)
```

**对比表**:同一个搜索任务,**grep 要写多长 vs rg 要写多长**:

| 任务 | grep | rg |
| --- | --- | --- |
| 递归搜文本 | `grep -r 'foo' .` | `rg 'foo'` |
| 忽略 .git / node_modules | `grep -r --exclude-dir={.git,node_modules} 'foo' .` | `rg 'foo'`(默认就忽略) |
| 只搜 Python 文件 | `grep -r --include='*.py' 'foo' .` | `rg -t py 'foo'` |
| 大小写不敏感 | `grep -ri 'foo' .` | `rg -i 'foo'` |
| 上下文 3 行 | `grep -r -A 3 -B 3 'foo' .` | `rg -A 3 -B 3 'foo'`(flag 一样) |
| 只输出文件名 | `grep -r -l 'foo' .` | `rg -l 'foo'` |

**rg 的 flag 设计有意保留 grep 的命名**——`-i / -v / -l / -A / -B / -c` 这些 grep 用户的肌肉记忆都在,**迁移成本低**。

### 3.3 关键 flag(高频)

```bash
# ─── 基础 ───
rg 'pattern'                    # 默认递归当前目录
rg 'pattern' src/               # 指定目录
rg -i 'Error'                   # 大小写不敏感(--ignore-case)
rg -v 'TODO'                    # 反向匹配
rg -w 'foo'                     # 全词匹配(避免 foobar 被匹中)

# ─── 文件类型过滤 ───
rg -t py 'pattern'              # 只 Python 文件
rg -T py 'pattern'              # 排除 Python 文件
rg --type-list                  # 看所有支持的类型
rg -g '*.{js,ts}' 'pattern'     # glob 过滤
rg -g '!*.test.*' 'pattern'     # 排除测试文件(! 是反向)

# ─── 上下文 ───
rg -A 3 'ERROR'                 # 匹配后 3 行(After)
rg -B 3 'ERROR'                 # 匹配前 3 行(Before)
rg -C 3 'ERROR'                 # 前后各 3 行(Context)

# ─── 输出 ───
rg -l 'pattern'                 # 只输出文件名(给 xargs / fzf)
rg -c 'pattern'                 # 每文件匹配次数
rg --count-matches 'pattern'    # 总匹配次数
rg --json 'pattern' | jq        # JSON 输出(给脚本处理)
rg -o 'foo\w+'                  # 只输出匹配部分(--only-matching)

# ─── 突破默认忽略 ───
rg -u 'pattern'                 # 搜隐藏文件
rg -uu 'pattern'                # 搜隐藏 + .gitignore 里的
rg -uuu 'pattern'               # 全搜(包括二进制)

# ─── 字面 vs 正则 ───
rg -F 'literal.$$.string'       # 字面匹配,不当正则
rg 'PATTERN'                    # 默认是正则
rg -P '(?<=foo)bar'             # PCRE2(支持 lookahead 等高级特性)
```

**`-u / -uu / -uuu` 的含义记一下**——遇到「我明明知道有这个字符串但 rg 找不到」的情况,**90% 是被 .gitignore 忽略了**,加 `-uu` 试试。

### 3.4 三个真实场景

**场景 1:在大 monorepo 里找一个函数定义**

```bash
# 假设你要找 function foo 的定义,可能是 def foo / function foo / fn foo
rg '(def|function|fn)\s+foo\b' -t py -t ts -t rs

# 找类定义
rg '^class\s+UserService\b' -t py
```

**场景 2:批量改文件名前先看哪些被引用**

```bash
# 想把 old_module 改名,先看谁 import 了
rg 'old_module' -l                   # 输出文件名清单
rg 'old_module' -l | xargs sd 'old_module' 'new_module'
                                     # 把所有引用替换(sd 后面讲)
```

**场景 3:从日志找错误 + 上下文**

```bash
# 在一堆 .log 文件里找 ERROR,前 3 行后 3 行
rg -C 3 'ERROR' /var/log -g '*.log'

# 加 --color=always | less -R 让结果分页带颜色
rg -C 3 'ERROR' --color=always | less -R
```

### 3.5 配置:~/.config/ripgrep/config 或 $RIPGREP_CONFIG_PATH

rg 支持配置文件,常用的:

```
# ~/.config/ripgrep/config
--max-columns=150
--max-columns-preview
--smart-case
--colors=match:fg:yellow
--colors=path:fg:cyan
```

环境变量挂上去:

```bash
export RIPGREP_CONFIG_PATH=~/.config/ripgrep/config
```

**`--smart-case` 值得说一句**:**小写时不敏感、有大写时敏感**——`rg foo` 等于 `-i`,但 `rg Foo` 严格大小写。**这是 90% 场景下的合理默认**,推荐永久打开。

### 3.6 不会替代 grep 的情况

```
1. 写 POSIX shell 脚本
   #!/bin/sh
   grep -E 'foo' file.txt        # 这里要用 grep,你不能假设对方装了 rg

2. SSH 进陌生服务器
   ssh prod-db
   grep -i 'oom' /var/log/syslog # 服务器上多半没装 rg

3. Alpine / 极简容器
   docker exec -it pod grep ...  # 容器里默认只有 busybox grep

4. CI 脚本
   .github/workflows/test.yml
   - run: grep -q 'FAIL' ...     # CI runner 的预装清单不一定有 rg
```

**rg 的位置是「daily driver」**:你 80% 的搜索发生在本地、在你装好 rg 的机器上、在交互式 shell 里——**这 80% 用 rg**。**剩下 20% 是脚本/远端/CI**——**用 grep**。这一节最重要的话就在这里:**rg 不替代 grep,它替代「你在本地交互式用 grep 的那部分场景」**。

## 四、fd 详解——find 的现代脸

`find` 是 1970s 设计的工具,**它的语法是"动作语言"**:`find . -name '*.py' -type f -mtime -7 -exec grep 'TODO' {} \;`——你描述「在哪、什么名字、什么类型、什么时间、找到后干嘛」。**这套语法的好处是表达力强,坏处是写不快**。

`fd` 的设计哲学相反:**先猜你 80% 的场景是什么,猜对了就给你最短的写法**。

### 4.1 安装

```bash
brew install fd
# 或
cargo install fd-find
apt install fd-find                  # 但 Ubuntu 上叫 fdfind!
```

**Ubuntu 的坑**:`apt install fd-find` 装的二进制叫 `fdfind` 不是 `fd`(因为有另一个老包占了 fd 名字)。**解法**:`alias fd=fdfind`,或 `mkdir -p ~/.local/bin && ln -s $(which fdfind) ~/.local/bin/fd`。

### 4.2 默认行为(对照 find)

```
fd pattern
   ↓ 默认
   - 递归当前目录(find 默认也递归,但要 . 起头)
   - 模糊匹配(把 pattern 当 path 子串,find 要 -name '*pattern*')
   - 尊重 .gitignore(find 不知道)
   - 跳过隐藏文件(find 默认搜)
   - 彩色输出(find 没颜色)
   - 多线程(find 单线程)
```

**对比表**:

| 任务 | find | fd |
| --- | --- | --- |
| 找名字含 config 的 | `find . -name '*config*'` | `fd config` |
| 找 .py 文件 | `find . -name '*.py'` | `fd -e py`(或 `fd '\.py$'`) |
| 找文件(排除目录) | `find . -type f -name '*.log'` | `fd -t f -e log` |
| 包含隐藏 | `find .`(默认就包含) | `fd -H pattern` |
| 7 天内修改的 | `find . -mtime -7` | `fd --changed-within 1week` |
| 找到后执行 | `find . -name '*.py' -exec wc -l {} \;` | `fd -e py -x wc -l` |

### 4.3 关键 flag

```bash
# ─── 基础 ───
fd                              # 列出所有(除了 ignored 的)
fd pattern                      # 模糊匹配名字
fd '^test.*\.py$'               # 正则严格匹配
fd -F 'literal.string'          # 字面匹配,不当正则

# ─── 类型过滤 ───
fd -t f pattern                 # 只文件
fd -t d pattern                 # 只目录
fd -t l pattern                 # 只符号链接
fd -e py pattern                # 扩展名是 py(可叠加:-e py -e pyx)
fd -e py                        # 所有 .py 文件(pattern 可省)

# ─── 突破默认忽略 ───
fd -H pattern                   # 包含隐藏文件
fd -I pattern                   # 不忽略 .gitignore
fd -HI pattern                  # 两个一起,真的全搜

# ─── 时间 ───
fd --changed-within 1week
fd --changed-within 24h
fd --changed-before 2024-01-01

# ─── 大小 ───
fd --size +100M                 # 大于 100MB 的
fd --size -1k                   # 小于 1KB 的

# ─── 找到后干嘛(-x 是 xargs,-X 是批量) ───
fd -e log -x rm                 # 找到所有 .log 文件并删除
fd -e jpg -X convert            # 批量 convert(一次传所有)
fd 'pattern' -x cp {} /tmp/     # {} 是 placeholder
fd 'pattern' -x echo {.}        # {.} 是去扩展名的 basename
```

**`{}` / `{.}` / `{/}` 速查**:

```
{}      完整路径,e.g. ./src/foo.py
{.}     去掉扩展名:./src/foo
{/}     basename:foo.py
{//}    dirname:./src
{/.}    basename 去扩展名:foo
```

这些和 GNU parallel 的语法对齐,**记一套通用**。

### 4.4 两个真实场景

**场景 1:找出最近一周改过的所有 markdown 文件**

```bash
fd -e md --changed-within 1week
```

**对照 find**:

```bash
find . -name '*.md' -mtime -7         # GNU find,BSD find 语法略不同
```

fd 的优势:**跨 GNU / BSD find 不一致**(macOS 的 find 跟 Linux 的 find 不一样),**fd 一致**。

**场景 2:批量重命名一批旧后缀**

```bash
# 把所有 .yaml 改成 .yml(用 fd 找,sd 改后缀这种 fd 自己做不了,这里用 -x mv)
fd -e yaml -x bash -c 'mv "$0" "${0%.yaml}.yml"' {}
```

### 4.5 不会替代 find 的情况

```
1. 复杂逻辑组合
   find . \( -name '*.log' -o -name '*.tmp' \) -mtime +30 -delete
   ↑ AND / OR / NOT 嵌套 fd 表达不了,find 这种"动作语言"在这里更强

2. -newer 比文件新
   find . -newer reference.txt
   ↑ fd 没这个

3. 远端机器 / 服务器
   ssh server "find /var -type f -size +1G"
   ↑ 远端多半没 fd

4. -inum / -links 这种 inode 级别
   find / -inum 12345
   ↑ fd 不做这种系统级查询
```

**fd 的位置和 rg 一样**:**daily driver 用 fd,脚本和远端用 find**。

## 五、bat 详解——cat 的"看代码"模式

`cat` 是 1969 年设计的,**它只做一件事:把文件原样吐到 stdout**。**对脚本和管道,这个设计是完美的**——cat 不该有颜色、不该有行号、不该高亮——否则脚本就炸。

但「**我想看一眼这个 .py 文件**」这件事,cat 的输出体验极差:**几百行没颜色没行号的纯文本**。`bat` 就是干这件事的——**和 cat 共存,不是替换 cat**。

### 5.1 安装

```bash
brew install bat
# 或
cargo install bat
apt install bat                      # Ubuntu 上又叫 batcat!
```

**同样的 Ubuntu 坑**:`apt install bat` 装的叫 `batcat`,`alias bat=batcat` 一下。

### 5.2 默认行为

```bash
bat README.md
```

输出长这样:

```
───────┬─────────────────────────────────────────────────
       │ File: README.md
───────┼─────────────────────────────────────────────────
   1   │ # MyProject
   2   │
   3   │ A short description.
   4   │
   5   │ ## Install
   6   │
   7   │   ```bash
   8   │   brew install myproject
   9   │   ```
───────┴─────────────────────────────────────────────────
```

**默认行为**:语法高亮(syntect 库) + 行号 + 文件名 header + 分页(如果长)。**Markdown / Python / Rust / JSON / YAML 之类几百种语言开箱即用**。

### 5.3 关键 flag

```bash
# ─── 基础 ───
bat file.py                     # 默认带行号 / 高亮 / header
bat -p file.py                  # plain,无 header 无行号(适合管道)
bat -A file.txt                 # 显示空白字符(空格 / tab / CRLF)
bat -n file.py                  # 只行号,无 header

# ─── 管道 ───
echo '{"a":1}' | bat -l json    # 强制语言
cat /etc/os-release | bat -p    # 用 bat 看管道输入

# ─── 范围 ───
bat -r 50:100 file.py           # 只看 50-100 行
bat -r :50 file.py              # 前 50 行
bat -r 50: file.py              # 50 行往后

# ─── 多文件 ───
bat src/*.py                    # 多文件,每个带 header

# ─── 作为 manpager / git pager ───
export MANPAGER="sh -c 'col -bx | bat -l man -p'"
export PAGER="bat"              # 替代 less

# ─── 主题 ───
bat --list-themes               # 看所有主题
bat --theme="OneHalfDark" file.py
```

**自动检测管道**:`bat README.md | head` 时 bat 知道自己在管道里,**自动退化成 cat 的行为**(无 header / 无颜色 / 不分页)——**这一点设计极好**,意味着 bat 在管道里不会破坏脚本。

### 5.4 配置:~/.config/bat/config

```
# ~/.config/bat/config
--theme="OneHalfDark"
--style="numbers,header,grid"
--paging=auto
```

`--style` 可选值:`auto, full, plain, changes, header, header-filename, header-filesize, grid, rule, numbers, snip`,**逗号组合**。

### 5.5 bat + 其他工具的协同

**作为 git pager**(但 delta 更好,后面讲):

```bash
git config --global core.pager 'bat --paging=always'
```

**作为 man pager**:

```bash
# .zshrc / .bashrc
export MANPAGER="sh -c 'col -bx | bat -l man -p'"
```

之后 `man ls`,**所有 man 页带语法高亮**——`-l man` 让 bat 用 man 的语法表。

**配 fzf 当 preview**(12 篇专讲):

```bash
fzf --preview 'bat --color=always --style=numbers {}'
```

**这是 bat 最大杀手锏**:在 fzf 里选文件时,**右边预览框带高亮**。**bat 和 fzf 是绝配**——12 篇会详细讲。

### 5.6 不替代 cat 的情况

```
1. 脚本 / 管道
   cat file | grep foo            # 用 cat,不是 bat
   ↑ bat 在管道里行为正确,但脚本里你怎么知道对方有没有 bat?

2. > 重定向
   cat in.txt > out.txt           # 用 cat
   ↑ bat 也行,但语义模糊

3. 拼接文件
   cat header.txt body.txt > full.txt
   ↑ cat 的本职工作:concatenate,bat 就不是干这个的

4. 远端 / 容器
   ssh server "cat /etc/hosts"
   ↑ 服务器不会装 bat
```

**bat 是"看"工具,cat 是"读"工具**——**人看用 bat,机器读用 cat**。这一句话记住,你就知道什么时候用哪个。

## 六、eza / lsd——ls 的"门面"升级

**eza** 是 **exa 的 fork**(原项目 2021 停滞,社区接管成 eza)。**lsd** 是另一个类似产品,主打"nerd fonts 图标"。两个都比 ls 漂亮,功能基本对齐——选哪个看个人审美。

### 6.1 安装

```bash
brew install eza                     # 或 lsd
cargo install eza
apt install eza                      # Ubuntu 24.04+ 才有
```

### 6.2 默认行为对比

```
ls -la
   ↓
-rw-r--r--@ 1 qijixin staff 9806 May 12 09:29 00-写作计划.md
-rw-r--r--@ 1 qijixin staff ...
```

```
eza -la --git --icons
   ↓
.rw-r--r--@ qijixin  9.8k  May 12 09:29   00-写作计划.md  -M
.rw-r--r--@ qijixin  ...
```

eza 给你的:**人类可读大小**(9.8k 不是 9806)、**git 状态列**(-M 表示 modified)、**图标**(如果你装了 nerd font)、**默认时间格式**(不是 ls 那种"6 个月前用月日,6 个月内用时分"的混乱)。

### 6.3 关键 flag

```bash
eza                             # 列当前目录
eza -l                          # long format
eza -la                         # 含隐藏
eza -lah                        # 人类可读大小
eza --git                       # 加 git 状态列
eza --icons                     # 加图标(需要 nerd font)
eza --tree                      # 树状视图
eza --tree --level=2            # 限制深度
eza -lah --git --icons --group-directories-first    # 目录排前面
eza --sort=size --reverse       # 按大小排,大的在前
eza --sort=modified --reverse   # 按修改时间排,新的在前
```

### 6.4 推荐 alias

**不要 `alias ls=eza`**——**这会让脚本里的 ls 行为变**。**推荐**:

```bash
# 保留原 ls,新命令名走 eza
alias ll='eza -lah --git --icons --group-directories-first'
alias lt='eza --tree --level=2 --git --icons'
alias la='eza -a --icons'
```

**新别名 = 新命令名**,这样:

```
ls               ← 仍然是系统 ls(脚本里 / 远端走它)
ll               ← 你日常看目录用 eza
lt               ← 树状看
```

### 6.5 eza --tree 的坑

**eza --tree 不是 tree 命令的替代**——大目录上 eza --tree 会一次扫所有层级,**几万个文件的 node_modules 直接卡 5 秒**。

**真要看树**:

```bash
# 限制深度
eza --tree --level=2

# 或者用专门的 dust(后面讲)/ tre(更轻量的 tree)
dust ~                         # 看大小 + 树
```

### 6.6 eza vs lsd 选哪个

```
eza:    社区接管,维护积极
        更接近 exa 的传统设计,大小 / 时间默认更冷静
        nerd font 图标可选

lsd:    主打 "ls deluxe",图标更丰富
        默认配色更花哨
        Rust 写,启动快

推荐:   eza —— 默认显示更"工程",图标可选不强制
        花哨派可选 lsd
```

## 七、sd——sed 的「安全」替代品

**sed 是一切"流编辑器"的祖先**,但它的语法是 70 年代设计的——`s/pat/repl/g`、`-i` 行内编辑、`-E` 扩展正则,**默认正则、默认贪婪、转义规则复杂**。**新人写 sed 经常炸**——错一个 `/` 就替换整盘。

`sd` 的设计哲学是「**像现代语言里的 .replace() 函数**」——**默认字面替换、需要 regex 时显式开**。

### 7.1 安装

```bash
brew install sd
cargo install sd
```

### 7.2 基础语法对比

```bash
# sed:替换文件里所有的 foo 为 bar
sed -i 's/foo/bar/g' file.txt
# 或 macOS BSD sed:
sed -i '' 's/foo/bar/g' file.txt    # 注意 macOS 要 -i '' !!

# sd:同样的事
sd 'foo' 'bar' file.txt             # 默认就是全局替换
                                    # 默认是 regex,但 . / * 之类要转义
```

**sd 的语法**:`sd PATTERN REPLACEMENT [FILE...]`,**位置参数**,**无需 s/// 分隔符**。

### 7.3 关键场景

```bash
# ─── 简单替换 ───
sd 'foo' 'bar' file.txt              # 默认 regex,如果 pattern 里没特殊字符,跟字面一样
echo "foo bar" | sd 'foo' 'BAZ'      # 从 stdin 读,输出到 stdout

# ─── 字面替换(不当 regex)───
sd -F 'price: $9.99' 'price: $19.99' file.txt
                                     # -F / --fixed-strings 切到字面模式
                                     # 这下你不用转义 $ 了

# ─── 捕获组(regex 模式)───
sd '(\w+)@(\w+)' '$2/$1' file.txt    # 用 $1 / $2 引用捕获组
                                     # 注意:不是 \1 / \2(sed 那一套)

# ─── 多文件 ───
sd 'old_name' 'new_name' src/**/*.py
fd -e py -x sd 'old_name' 'new_name'  # 配 fd 用更稳

# ─── 预览(不写文件)───
sd 'foo' 'bar' file.txt              # 写文件
sd 'foo' 'bar' < file.txt            # 从 stdin 读,输出 stdout,文件不变
sd -p 'foo' 'bar' file.txt           # -p / --preview 显示 diff,文件不变
```

**`-p` 预览**是 sd 比 sed 安全的一个关键特性——**先看改了什么再决定写不写**。

### 7.4 sd 比 sed 安全在哪

```
1. 默认全文匹配,不分行
   sed 默认按行处理,跨行替换要 -z 或者复杂的 N 命令
   sd 默认整个文件当一个字符串看

2. Unicode 默认对
   sed 对 Unicode 的 \b / \w 在不同 locale 表现不同
   sd 默认 Unicode 类对齐 Rust regex,行为一致

3. 转义少,捕获组用 $1
   sed:  s/\(foo\)/\1-bar/    要转义 ( )
   sd:   '(foo)' '$1-bar'     不转义

4. 错误更清楚
   sed 错了经常静默,sd 错了直接报清楚

5. 文件读写更稳
   sed -i 在 macOS 要 '' 占位,Linux 不要,跨平台坑
   sd 一致行为
```

### 7.5 不替代 sed 的情况

**sd 是替换工具,不是流编辑器**——sed 能做的很多事 sd 不做:

```
1. 行号范围编辑
   sed '5,10d' file.txt              # 删第 5-10 行 —— sd 没这个
   sed '/^#/d' file.txt              # 删所有 # 开头的行 —— sd 没这个

2. 复杂地址匹配
   sed '/BEGIN/,/END/d'              # 删 BEGIN 到 END 之间 —— sd 没这个

3. 多命令组合
   sed -e 's/foo/bar/g' -e 's/baz/qux/g'  # 多条 s 命令 —— sd 没这个(要管道)

4. 行编辑 / 插入 / 追加
   sed '5a\hello'                    # 第 5 行后插 hello —— sd 没这个

5. 远端 / 脚本兼容
   ssh server "sed -i 's/foo/bar/g' /etc/conf"
                                     # 远端只有 sed
```

**总结**:**sd 替代「简单替换」,sed 仍然是「流编辑器」**。**简单 `s/foo/bar/` 类型的活,sd 安全得多;`5,10d` 这种 sed 是不可替代的**。

## 八、dust——du 的可视化版

**du 是 1969 年的工具**——它做对了一件事(算磁盘占用),但输出是纯文本数字,**人眼看大目录需要 `du -sh * | sort -h` 这种组合**。

`dust` 给的是**树状可视化 + 自动按大小排序**:

```bash
dust ~/projects
```

```
 4.9G   ┌── node_modules
 5.0G ┌─┴ frontend
 2.1G │   ┌── target
 2.5G │ ┌─┴ backend
 7.5G ┴─┴ projects
        ░░░░░░░░░░░░░░░░░░░░░░░ 100%
```

**自动找最大的目录、自动可视化条**——「看磁盘谁占了空间」这件事从 5 分钟变成 5 秒。

### 8.1 安装

```bash
brew install dust
cargo install du-dust                # 注意包名是 du-dust 不是 dust
```

### 8.2 关键 flag

```bash
dust                            # 看当前目录
dust ~                          # 看 home 目录
dust -d 3                       # 限制深度 3 层(--depth)
dust -n 30                      # 显示前 30 个(--number-of-lines)
dust -r                         # 反向(从小到大)
dust -s                         # 显示 stats(不画进度条)
dust -p                         # 用完整路径(不缩略)
dust -i                         # 忽略 .gitignore(默认尊重)
dust -X node_modules            # 排除某目录
```

### 8.3 真实场景

**找出 home 目录下谁占了最多空间**:

```bash
dust ~ -d 3 -n 30
```

**找 git repo 里能清的大文件**:

```bash
dust --filecount               # 按文件数排,不是按大小
dust -d 2                      # 看顶层 2 层哪些大
```

### 8.4 dust 不替代 du 的情况

```
1. 脚本里取数字
   USED=$(du -sb /var | awk '{print $1}')
   ↑ du 输出固定可解析,dust 是给人看的

2. NFS / 网络盘
   du / dust 在网络盘上都很慢,但 du 有 -x(--one-file-system)更稳

3. 远端
   ssh server "du -sh /var/log/*"
```

**dust 是「找谁占空间」的工具,du 是「精确测量」的工具**。

## 九、hyperfine——time 是单次,hyperfine 是统计

写完一个脚本想知道「快不快」,90% 工程师第一反应:`time python script.py`——**`time` 是 POSIX 内建,跑一次给一个数字**。

**但 `time` 是不可靠的**:

```bash
$ time python script.py
real    0m1.234s

$ time python script.py
real    0m0.876s

$ time python script.py
real    0m1.567s
```

**单次的 real time 抖动可以差 2 倍**——文件系统 cache、CPU 频率调节、其他后台进程都会影响。**用 time 做 benchmark 是不专业的**。

`hyperfine` 是为 benchmark 设计的:**多次运行 + warmup + 统计 + 对比**。

### 9.1 安装

```bash
brew install hyperfine
cargo install hyperfine
```

### 9.2 最小用法

```bash
hyperfine 'python script.py'
```

**输出**:

```
Benchmark 1: python script.py
  Time (mean ± σ):     876.5 ms ±  23.1 ms    [User: 654.3 ms, System: 89.2 ms]
  Range (min … max):   843.2 ms … 921.8 ms    10 runs
```

**默认跑 10 次**,**给均值 / 标准差 / 最小 / 最大**。**比 time 的「单次实数」专业一个量级**。

### 9.3 关键 flag

```bash
# ─── warmup(避免冷启动测量)───
hyperfine --warmup 3 'cmd'           # 先跑 3 次不计入,再正式 10 次
                                     # 文件系统 cache 之类的预热

# ─── 跑多少次 ───
hyperfine --runs 50 'cmd'            # 跑 50 次
hyperfine --min-runs 10 --max-runs 100 'cmd'    # 自适应

# ─── 对比两个命令 ───
hyperfine 'cmd_a' 'cmd_b'
                                     # 会直接说哪个快 X 倍

# ─── 准备 / 清理 ───
hyperfine --prepare 'rm -rf cache' 'cmd'
                                     # 每次跑前先清 cache(测冷启动)

hyperfine --setup 'init.sh' 'cmd'    # 一次性 setup
hyperfine --cleanup 'cleanup.sh' 'cmd'

# ─── 参数化(测不同输入)───
hyperfine -L size 100,1000,10000 'sort -n /tmp/{size}.txt'
                                     # 三个不同 size 跑同一个命令

hyperfine -P count 1 10 'wc -l file_{count}.txt'
                                     # 数字范围

# ─── 导出 ───
hyperfine 'cmd_a' 'cmd_b' --export-json results.json
hyperfine 'cmd_a' 'cmd_b' --export-markdown table.md

# ─── shell 设置 ───
hyperfine --shell=none 'cmd'         # 不通过 shell 跑(避免 shell 启动开销)
hyperfine --shell=bash 'cmd'         # 指定 shell
```

### 9.4 三个真实场景

**场景 1:对比两个实现**

```bash
hyperfine --warmup 3 \
    'python solve_v1.py' \
    'python solve_v2.py'
```

输出会自动给「v2 比 v1 快 X 倍 ± Y」,**比你自己测 10 次手算均值方差强多了**。

**场景 2:测冷启动 vs 热启动**

```bash
# 冷启动(每次清 cache)
hyperfine --prepare 'sync; echo 3 > /proc/sys/vm/drop_caches' 'cat /tmp/big.log'

# 热启动(默认就是)
hyperfine --warmup 3 'cat /tmp/big.log'
```

**场景 3:参数化测不同输入规模**

```bash
hyperfine -L size 1000,10000,100000,1000000 \
    'sort -n /tmp/numbers_{size}.txt' \
    --export-markdown results.md
```

输出一张 Markdown 表,**每个 size 一行,时间对比**——直接可贴进 PR 描述。

### 9.5 hyperfine 不替代 time 的情况

```
1. 一次性看耗时
   time make build               # 看一次构建多久,不是 benchmark

2. 真实长跑任务
   time ./train_model.py         # 训练 4 小时,不可能跑 10 次

3. 需要看 CPU/IO 细分
   /usr/bin/time -v cmd          # GNU time 给详细资源使用
```

**hyperfine 用于「statistical benchmark」,time 用于「快速看一次跑多久」**。

## 十、procs / bottom / btop——系统监控类

**top 和 ps 是 1970-80 年代的工具**——`ps aux` 列所有进程、`top` 看实时 CPU 内存。**它们工作,但 UI 是上世纪的**。

这一节挑 3 个**比传统 top/ps 漂亮、本地用值得装**的工具——**但注意监控类工具跨机器复用最弱**,远端服务器多半还得回到 top / htop。

### 10.1 procs(ps 的彩色版)

```bash
brew install procs
```

```bash
procs                       # 彩色表格列所有进程
procs --tree                # 树状,看进程父子关系
procs python                # 过滤名字含 python 的
procs --watch               # 持续刷新(类似 top)
procs --sortd cpu           # 按 CPU 倒序
```

**procs 比 ps 强在**:**默认彩色 + 表格对齐 + 容器 PID 关联**——你看 docker 进程时,procs 会显示对应的容器名。

### 10.2 bottom(btm)——top 的跨平台升级

```bash
brew install bottom
btm                         # 启动
```

**特点**:

```
- 跨平台:macOS / Linux / Windows 同一套 UI
- 图表化:CPU / 内存 / 网络 IO 实时折线图
- 鼠标可点:点哪一栏排序、可拖动调整面板
- 容器视图:btm 内建 Docker / Podman 显示
- 可截图:btm 自己可生成 SVG 输出
```

**vs htop**:**bottom 跨平台**(htop 在 macOS 上也能装,但 Windows 没有),**图表更直观**,**学习曲线更平**(htop 的快捷键多)。

**vs top**:bottom 完胜——top 几乎是"最后选择"。

### 10.3 btop——花哨派

```bash
brew install btop
btop
```

**btop 比 bottom 更花哨**:**CPU 每核柱状图 + GPU 监控 + 进程树**。**学习曲线略高于 bottom**,但视觉冲击力强——演讲 / 截图 / dotfiles 展示 用 btop 显得专业。

**坑**:btop 默认主题在某些终端字体下渲染会错位,要调主题或字体。

### 10.4 htop 仍然在的理由

```
1. Linux 服务器默认装(apt install htop 几乎所有发行版都有)
2. ssh 远端,bandwidth 低,htop 的纯文本 UI 比 btop 的图形 UI 流畅
3. 集群里所有人都用同一套,你装 btop 同事不认
4. 学习曲线最平,新人 30 秒上手
```

**实操推荐**:

```
本地 macOS:        btm 当 daily driver,看图清楚
远端 Linux 服务器: htop(默认有 / 别人也认)
真不得已:          top(永远在,无论多极简的容器)
```

## 十一、tokei——代码行数 50× 速度

**cloc** 是 Perl 写的代码行数统计工具,跑大仓库要十几秒。**tokei** 是 Rust 写的等价工具,**50× 速度**:

```bash
brew install tokei
tokei .
```

**输出**:

```
===============================================================================
 Language            Files        Lines         Code     Comments       Blanks
===============================================================================
 Markdown                4         2854         1903          134          817
 Python                 35         8421         6543          812         1066
 TypeScript             87        12453         9876         1432         1145
-------------------------------------------------------------------------------
 Total                 126        23728        18322         2378         3028
===============================================================================
```

### 11.1 常用 flag

```bash
tokei                       # 当前目录
tokei -t Python,Rust        # 只看特定语言
tokei -e tests/             # 排除目录
tokei -s code               # 按代码行排序
tokei --files               # 显示每个文件
tokei -o json               # JSON 输出
tokei -o yaml               # YAML 输出
```

### 11.2 场景

**项目 README 显示规模**:

```bash
tokei --output html > stats.html        # 或 markdown
```

**对比两个 commit 的代码量**:

```bash
git checkout main && tokei -o json > /tmp/main.json
git checkout feature && tokei -o json > /tmp/feature.json
diff /tmp/main.json /tmp/feature.json
```

**tokei 不替代 wc -l**:wc -l 数纯行,**包括空行 / 注释 / 二进制**;tokei 按语言识别注释 / 空行 / 代码,**给更工程的数字**。

## 十二、delta——git diff 的视觉飞跃

**git diff 默认输出**:

```diff
diff --git a/foo.py b/foo.py
index abc...def 100644
--- a/foo.py
+++ b/foo.py
@@ -10,5 +10,5 @@
-def old_func():
+def new_func():
     return 1
```

**delta 给的输出**(side-by-side + 语法高亮 + 行号):

```
foo.py
─────────────────────────────────────────────────────────────────────────
10 │ def old_func():               │ 10 │ def new_func():
11 │     return 1                  │ 11 │     return 1
─────────────────────────────────────────────────────────────────────────
```

**color + 行号 + 文件名居中标题 + 可选 side-by-side**——看 diff 这件事从"读 diff"变成"看代码"。

### 12.1 安装

```bash
brew install git-delta              # 包名是 git-delta!二进制叫 delta
cargo install git-delta
```

### 12.2 配置 git 走 delta

**~/.gitconfig**:

```ini
[core]
    pager = delta

[interactive]
    diffFilter = delta --color-only

[delta]
    navigate = true          # n/N 在 hunk 之间跳
    line-numbers = true      # 显示行号
    side-by-side = false     # 默认 unified,要 side-by-side 设 true
    syntax-theme = OneHalfDark
    file-style = cyan bold ul
    hunk-header-style = file line-number

[merge]
    conflictstyle = zdiff3   # 三路 merge 更清楚(delta 推荐)
```

**之后**:**git diff / git log / git show / git stash show 全部走 delta**,**自动语法高亮 + 行号**。

### 12.3 命令行直接用

```bash
diff -u a.txt b.txt | delta
git diff main feature | delta
```

**任何 unified diff 都可以 pipe 给 delta**——不限于 git。

### 12.4 关键 flag / 配置

```
# 在 ~/.gitconfig 的 [delta] 节
side-by-side = true              # 两列并排
line-numbers = true              # 行号
navigate = true                  # n/N 跳 hunk
hyperlinks = true                # 文件名变可点链接(终端支持时)
true-color = always              # 强制开 24-bit color
syntax-theme = OneHalfDark       # bat --list-themes 可选

# 行内 diff 高亮(改的字段单独标)
keep-plus-minus-markers = false  # 行首不要 + -

# 文件分隔的样式
file-style = bold yellow ul
hunk-header-style = file line-number syntax
```

### 12.5 GitHub style theme

```ini
[delta]
    features = github
[delta "github"]
    side-by-side = true
    line-numbers = true
    syntax-theme = GitHub
    plus-style = "syntax #003800"
    minus-style = "syntax #3f0001"
```

**features 是 delta 的"配置组"机制**——你可以定义多套 features 切换。

### 12.6 delta vs diff-so-fancy

**diff-so-fancy** 是 Perl 写的更老的同类工具,**作者本人推荐用户迁到 delta**——所以这里只讲 delta。

### 12.7 gitLearning 系列没讲,这里点一下

**gitLearning 教 git 工作流,但没讲 diff viewer 的选型**——**delta 是 2026 的事实标准**,**任何用 git 的工程师都该配上**,**一次配置一辈子受益**。

## 十三、怎么不让「新工具上瘾症」发作

讲了这么多新工具,**最大的陷阱是"全装"**——看完一篇博文就 `brew install` 30 个 CLI,实际只用 3 个,剩下 27 个的更新提示天天烦你。**这一节是反作用力**。

### 13.1 选型纪律:装一个,用 2 周

```
新工具流程:
   1. brew install foo
   2. 用 2 周(强制至少 14 天)
   3. 复盘:
      ✓ 真的快了 → 留
      ✓ 真的更准 → 留
      ✗ 没明显感觉 → brew uninstall
      ✗ 跟原工具一样,只是多个名字 → brew uninstall
```

**关键是"卸"的决心**——大多数工具装上你就懒得卸了,**导致 dotfiles 越来越大、新机器 setup 越来越慢、心智负担越来越重**。

### 13.2 不要把核心命令名 alias 成新工具

**反对**:

```bash
alias ls=eza                # 错!
alias cat=bat               # 错!
alias grep=rg               # 错!
alias find=fd               # 错!
```

**为什么错**:

```
1. 你的脚本 / Makefile / CI yaml 里有 ls / cat / grep / find
   alias 在交互 shell 生效,但脚本调用绕过 alias
   → 看起来"换了",实际只换了交互部分,容易引起认知混乱

2. 同事 / 新人来你机器,看到 ls 的输出不是标准 ls 行为,困惑

3. 远端机器 / 容器没装 eza,你的 muscle memory 跑 ls 还会落空

4. eza 的 flag 不完全等于 ls,某些 ls -X 的脚本在 alias 后炸
```

**正确**:**用新命令名,不覆盖旧命令**:

```bash
alias ll='eza -lah --git --icons'         # ll 是新名字,ls 仍然是 ls
alias lt='eza --tree --level=2'
# bat 直接当 bat 用,不 alias 成 cat
# rg 直接当 rg 用,不 alias 成 grep
# fd 直接当 fd 用,不 alias 成 find
```

**这是最重要的一条原则**——**新命令名 + 保留旧命令**,**让两套并存,而不是替换**。

### 13.3 例外:某些 alias 可以做

```bash
# 这些 alias 是安全的:
alias cat='bat -p --paging=never'    # 仍然行为接近 cat,但有高亮
                                     # ⚠ 但脚本里仍要用 /bin/cat
alias diff='delta'                   # diff 不太可能在脚本里被依赖

# 极强建议不要做:
alias ls=eza                         # 风险最高
alias grep=rg                        # rg 不支持所有 grep flag
alias find=fd                        # 语法完全不同
```

**一个判断标准**:**这个工具有没有可能被脚本依赖**?ls / grep / find / cat / sed 都被脚本大量依赖——**别 alias**。diff / time / du / ps 在脚本里少见——**alias 风险低**。

### 13.4 用 Brewfile 管理装了什么

**装的 CLI 工具应该被声明式管理**——这样换新机器一句话恢复:

```ruby
# ~/Brewfile
brew "ripgrep"
brew "fd"
brew "bat"
brew "eza"
brew "dust"
brew "sd"
brew "procs"
brew "bottom"
brew "btop"
brew "tokei"
brew "git-delta"
brew "hyperfine"
brew "fzf"
brew "jq"
brew "zoxide"
```

```bash
brew bundle install --file=~/Brewfile        # 装上所有
brew bundle cleanup --file=~/Brewfile        # 清掉 Brewfile 里没有的(谨慎)
```

**22-25 篇会讲 Brewfile / chezmoi / Nix 怎么管 dotfiles**,这里先点名:**你装的工具该有清单,不该靠记忆**。

### 13.5 装多了的症状自检

```
□ brew list 输出超过 100 行,你叫得出名字的不到 30 个
□ brew upgrade 跑 5 分钟,你不知道升级了什么
□ 装了的 CLI 工具 60% 这个月没用过
□ 新机器 setup 要装一晚上,你不记得到底装哪些
□ 你的 alias 列表超过 50 个

>=3 条命中 = 该 brew uninstall 一波了
```

**装多 = 配置债**——半年后变成"我也不知道为什么这个工具在这,但删了好像有点慌"的状态。

## 十四、跨机器问题:你的工具栈不能假设别人也装

新工具用得越爽,**离不开它的概率越高**——但 SSH 进同事的机器、运维让你登服务器排查问题、容器里要 grep 一行,**那些机器没装 rg / fd / bat**。

### 14.1 三种场景

```
场景 1:SSH 进同事 / 服务器
   你:   ssh prod-app
   prod: $ rg 'ERROR' /var/log/app.log
         zsh: command not found: rg
   
   ↑ 只有传统工具

场景 2:容器里调试
   docker exec -it pod sh
   /app # cat /etc/os-release
   /app # ls / grep / find         ← 仅这些(busybox 实现)
   /app # rg                       ← 没有,Alpine 不预装
   
   ↑ 极简环境

场景 3:CI runner
   .github/workflows/test.yml
   - run: rg 'PASS' test-results.log
                                   ← runner 可能没 rg
   
   ↑ 脚本必须用 grep
```

### 14.2 工程对策

```
1. 本地永远装新工具
   你 80% 时间在自己机器上,daily driver 用 rg / fd / bat

2. 永远会传统工具
   grep / sed / find / cat 的常用 flag 要熟,
   即使你不天天用,SSH 进去要能立刻干活

3. 脚本永远写 POSIX
   #!/bin/sh
   grep 'foo' file                ← 永远写 grep,不写 rg
   find . -name '*.log' -delete   ← 永远写 find

4. dotfiles 装新工具,但不依赖
   chezmoi / Brewfile 把 rg / fd / bat 自动装上,
   但脚本里不假设它们存在

5. 远端机器加 brew Linux / cargo install
   你的服务器如果归你管,装上 rg / fd 也行
   但别人的机器 / 短期排查机器,别折腾
```

**22-25 篇会讲 dotfiles + Brewfile + Nix 怎么自动化「新机器 30 分钟变成你家」**,这一篇只点这个边界。

### 14.3 一个真实的混合工作流

我自己的状态:

```
本地 macOS:
   - rg, fd, bat, eza, dust, sd, hyperfine, delta 全装
   - alias ll=eza, alias cat=bat -p (cat 这个 alias 我私下用,公开脚本里不依赖)
   - 配 delta 进 ~/.gitconfig

我管的服务器:
   - apt 装 ripgrep, fd-find, bat
   - alias rg / fd 不开
   - 但 git diff 配 delta(因为我自己看)

公司服务器 / 同事机器:
   - 不动,不装,用什么是什么
   - 上去就是 grep / find / cat 老规矩

CI / 容器镜像:
   - 镜像 Dockerfile 不加任何新工具
   - CI 脚本里只用 POSIX 工具
```

**这套混合工作流的核心是「本地放飞自我,生产环境严守 POSIX」**——你既享受了新工具的工程红利,**又不会因为依赖它们让远端 / 团队场景出问题**。

## 十五、反对的写法

这一节列我反复见过的反模式:

### 15.1 把所有传统命令 alias 成新工具

```bash
alias ls=eza
alias cat=bat
alias grep=rg
alias find=fd
alias sed=sd
alias ps=procs
```

**已经讲过了**——脚本 / 远端 / 容器全部翻车。**不要这么做**。

### 15.2 装了 30 个 must-have CLI,只用 3 个

新人看博客「2026 必装的 50 个 CLI」,一键全装,**实际 80% 这辈子用不上**。**brew list 越来越长,brew upgrade 越来越慢,新机器恢复要一晚上**。

**解法**:**先用,再装**——遇到具体痛点(grep 慢、ls 不好看)再去找工具,**不要预防性装**。

### 15.3 团队脚本里依赖 rg / fd

```bash
# scripts/lint.sh
#!/bin/bash
rg 'TODO' src/                          # 错!
```

**同事 clone 仓库跑这个脚本,他没装 rg → 炸**。

**解法**:

```bash
#!/bin/bash
grep -r 'TODO' src/                     # 用 grep
# 或
if command -v rg >/dev/null; then
    rg 'TODO' src/
else
    grep -r 'TODO' src/
fi
```

**团队脚本永远用 POSIX,个人 dotfiles 才能用新工具**。

### 15.4 抄网上「100 个 must-have CLI」

```
Hacker News / X 上经常出现:
"2026 必装的 50 个 Rust CLI"
"我每天都用的 100 个工具"

⚠ 那是别人的工作流,不是你的
```

**别人是 DevOps 工程师,你是前端;别人天天写 Rust,你写 Python;别人桌上 4K 显示器装 nerd font 加 ligature,你 13 寸笔记本看不清图标**。**抄别人的工具列表 = 装别人的牙刷**。

**解法**:**每个工具问一次「这解决我什么问题」**——答得上来再装。

### 15.5 用 eza --tree 看 node_modules

```bash
eza --tree node_modules
# 卡住,5 秒后输出 5000 行
```

**eza --tree 没限深度,大目录直接爆**。

**解法**:**eza --tree --level=2**,或者用 **dust**(自动按大小展示)/ 专门的 **tre**。

### 15.6 装 bat 但当 cat 用,管道里输出乱码

```bash
alias cat=bat
cat data.txt | jq '.field'
# 报错或 jq 拿到带颜色码的输入
```

**有时 alias 没正确退化**——尤其是某些版本的 bat 在管道里仍带颜色码。

**解法**:**alias cat='bat -p --paging=never'**,或者干脆**不要 alias cat=bat**。

### 15.7 delta 配了但不配 ~/.gitconfig

```bash
brew install git-delta
# 然后什么都不做
git diff                  # 还是默认 diff 样
```

**delta 装上必须配进 git**,否则白装。**12.2 节的配置必须做**。

### 15.8 在 Alpine 容器里 cargo install 重一遍

```dockerfile
FROM alpine
RUN apk add cargo
RUN cargo install ripgrep        # 编译 5 分钟,镜像增大 200MB
```

**Alpine 镜像大忌**——为了一个 rg 让镜像膨胀 5 倍。

**解法**:容器里就用 `apk add ripgrep`(如果有 musl 预编译版)或者**根本不在容器里装,容器里用 grep**。

## 十七、下一篇预告

下一篇:**`12-fzf心智.md`**——讲 **fzf 是这些工具的"瑞士军刀总集"**。

这一篇讲了 10 个独立工具——rg / fd / bat 各自是好工具,**但单独用都是命令行体验**。**fzf 把这一切粘起来**:

- **rg | fzf**:rg 搜出几百行,fzf 模糊筛选选一行
- **fd | fzf**:fd 列出几万文件,fzf 一选打开
- **fzf + bat --preview**:右侧预览框带高亮
- **git log | fzf**:在历史里 fuzzy 找一个 commit
- **cd $(fd -t d | fzf)**:目录跳转
- **kill $(ps aux | fzf | awk ...)**:选进程 kill

**fzf 是 CLI 的"模糊匹配 UI 模式"**——一旦内化,**你前面装的所有工具威力翻倍**。

12 篇会讲:

- **fzf 的心智模型**(stdin 进、stdout 出、TUI 在中间)
- **shell key bindings**(Ctrl+R 历史、Ctrl+T 文件、Alt+C 目录)
- **fzf + bat / rg / fd 的协同**(把这一篇所有工具连起来)
- **fzf-tab**(把 zsh 补全接到 fzf)
- **自定义 fuzzy finder**(给任意命令包一层 fzf)

读完 12 篇,你的终端工作流**就从「记 100 个命令」彻底转向「写 1 个模糊匹配 + 选一项」**——这是 2026 终端 UX 的最大跃迁。
