    终端的解剖:一次按键到屏幕显示之间发生了什么

    你用了 10 年终端,我问你一个问题:iTerm2 是 shell 吗?——大多数工程师顿住,然后含糊地说"差不多吧"。不是。iTerm2 是终端模拟器,bash 才是 shell,这两个东西甚至不在同一个进程里。再问:你按下 'l' 键,屏幕上为什么显示出 'l'?——大多数人觉得"键盘 → 终端 → 屏幕"三步搞定,真相是至少 6 个组件协作,中间还经过两次内核态切换。这一篇就是打开这个黑盒——把每天用的"终端"拆成四块:终端模拟器、pty、shell、你跑的程序,讲清楚每一块负责什么、它们怎么交互、Ctrl-C 到底是谁处理的、为什么 ls | less 不上色。

    一句话先记住:终端 = 终端模拟器(GUI 程序) + pty(内核对象) + shell(普通进程) + 你跑的程序。这四个东西经常被笼统地叫做"终端",但它们的职责完全不一样:终端模拟器画字符 + 处理键盘(用户态 GUI 程序),pty 是内核里的一根双向管子(连接 GUI 和命令行进程),shell 是跑在 pty 从端的一个普通进程(解析你输入的命令),你的程序是 shell fork 出来的子进程(继承同一个 pty)。理解这四块的分工,你才能解释「为什么 tmux attach 之后 echo $TERM 不一样」「为什么 ls | less 时 ls 不上色」「为什么 Ctrl-C 能中断卡死的进程」——这些日常困惑全在这张图里。

    一、为什么"键盘 → 终端 → 屏幕"是错的

    大多数工程师的心智模型大概长这样:键盘 → "终端" → shell 收到 'l' → 屏幕显示 'l'——三步搞定。这套模型有几个根本性错误:

    1. "终端"是谁?——iTerm2?bash?系统?这三个不是一回事
    2. shell 怎么收到 'l' 的?——从键盘?那 shell 跑在哪?
    3. 屏幕上的 'l' 是谁画的?——shell?那 shell 怎么画 GUI?
    4. Ctrl-C 是谁拦截的?——shell?那 vim 怎么也能用 Ctrl-C?

    回答不了任何一个问题。因为它把四个完全不同的东西揉成了一个叫"终端"的黑盒。真实链路是这样:

    键盘硬件 → OS 内核键盘驱动 → 窗口系统(macOS WindowServer / X11 / Wayland)
   → 终端模拟器进程(iTerm2)接到键盘事件
   → 终端模拟器把字符 'l' 写到 pty 主端
   → 内核 pty 子系统把字符传到 pty 从端
   → shell(zsh)从 stdin 读到 'l'
   → shell 把 'l' 回显(echo)到 pty 从端
   → 内核 pty 子系统把字符传回 pty 主端
   → 终端模拟器从 pty 主端读到字符
   → 终端模拟器在屏幕上画出 'l'

    "内核 pty 子系统"在这条链上出现两次——按键穿过它一次,回显再穿一次。屏幕上你看到的那个 'l',不是键盘直接画的,是 shell(或更早,pty 自己)主动回显的、经内核 pty 中转后被终端模拟器画到屏幕的。

    这个事实有几个反直觉推论:

    • shell 不回显,你打字就看不到字(stty -echo 之后试,屏幕黑)
    • 拔掉键盘换个程序往 pty 主端写,效果跟你打字一模一样(这是 expect / pty 自动化的原理)
    • 关掉终端模拟器,shell 没死,只是没人读 pty 主端了(nohup 和 disown 干的就是这个)
    • 本地 shell 和 ssh 进来的 shell 是不同的 pty(tty 命令能看出来)

    90% 的工程师讲不清这条链——不是因为复杂,是因为没人画给他们看。这一篇就是画给你看。

    二、四块拼图:历史回顾(短)

    直接讲解剖之前,用一张图讲清楚这四块东西怎么演化出来的——只回答一个问题:为什么会有 pty?

    1970s  物理终端(VT100 / VT220 / Wyse 50)
        硬件键盘 + 硬件屏幕,通过串口连主机
         ──→ Unix 内核里有 /dev/tty1, /dev/ttyS0 设备文件

1980s  X Window 兴起,xterm 把"终端"搬到 GUI
        屏幕变成 GUI 窗口,但 shell 还以为自己在跟物理终端说话
         ──→ pty(pseudo-terminal)诞生:软件模拟的 tty

2000s  GUI 终端模拟器爆发
        xterm → Konsole / GNOME Terminal / iTerm2 / Terminal.app
         ──→ 物理 tty 几乎只剩 Linux 虚拟控制台(Ctrl+Alt+F1)

2020s  GPU 加速 + AI 集成
        Alacritty / WezTerm / Kitty / Ghostty 用 GPU 渲染字符
        Warp 把命令补全 / 历史 / AI 助手做进终端
         ──→ 底层链路 30 年没变:还是 pty + shell

    为什么不直接让 xterm 跟 shell 用管道通信?——因为 shell 和所有 CLI 程序都是直接跟 tty 设备说话的,tty 有特殊行为(行编辑、Ctrl-C 生成 SIGINT、回显控制),管道没有。

    为了让现有的所有 CLI 程序在 GUI 环境下不用改一行代码就能跑,内核搞了"伪终端":对 shell 看,完全跟物理 tty 一样;对 GUI 程序看,是一对可读写的 fd。这就是 pty。pty 不是"假装是 tty",pty 在内核里真的就是 tty——只是没有真的物理硬件,主端是另一个用户态进程。

    三、必须画清楚的图:四块的关系

    把"终端"这个黑盒拆开,你会看到这样的结构:

       用户态(GUI 那一侧)
   ┌─────────────────────────────────┐
   │  终端模拟器进程(iTerm2)        │
   │   - 窗口、字体、颜色渲染          │
   │   - 键盘事件 → 写 pty 主端       │
   │   - 读 pty 主端 → 渲染到屏幕      │
   └────────────┬────────────────────┘
                │ 读/写 pty 主端 fd
   ═════════════│════════════════════
   内核态        │
                ▼
   ┌─────────────────────────────────┐
   │         pty 子系统               │
   │   ┌────────┐      ┌────────┐    │
   │   │ 主端    │←双向→ │ 从端    │    │
   │   │(master)│ buf  │(slave) │    │
   │   │ 普通 fd │      │ tty-like│   │
   │   └────────┘      └────┬───┘    │
   │                         │termios │
   │                         │(行/raw)│
   └─────────────────────────┼───────┘
                             │ stdin/stdout/stderr
   ══════════════════════════│═══════
   用户态(CLI 那一侧)        │
                             ▼
   ┌─────────────────────────────────┐
   │  shell 进程(zsh / bash)        │
   │   - fd 0/1/2 都指向 pty 从端     │
   │   - 是 pty 的"会话首领"          │
   │   - fork 出来的子进程继承同一 pty │
   └────────────┬────────────────────┘
                │ fork + exec
                ▼
   ┌─────────────────────────────────┐
   │  你跑的程序(ls / vim / python)│
   │   - fd 0/1/2 也是 pty 从端       │
   │   - 跟 shell 共享 pty 和前台进程组│
   └─────────────────────────────────┘

    这张图要看到几个东西:

    1. 终端模拟器和 shell 是两个完全独立的进程——一个 GUI,一个 CLI,中间靠 pty 通信
    2. pty 有两端——主端(master,终端模拟器用)、从端(slave,shell 用)
    3. shell 和你跑的程序共享同一个 pty 从端——所以你的程序也能读键盘、写屏幕
    4. 数据穿越内核两次——按键从主端到从端是一次,回显从从端到主端是一次
    5. termios 挂在 pty 从端上——所有"行编辑、信号生成"行为都是从端在做

    记住这张图。下面所有细节都是在解释这张图的某一部分。

    四、一次按键的全链路(重头戏)

    现在拿一个具体的按键来走一遍。你打开 iTerm2,光标在 zsh 提示符后面,按下 'l'。

    4.1 完整链路图

    你按下 'l' 键
   ↓
键盘硬件 → OS 内核键盘驱动 → 窗口系统(WindowServer / X11 / Wayland)
   ↓ 投递按键事件给前台窗口
终端模拟器(iTerm2)收到 keydown 事件
   ↓ keymap 转成字符 'l' (0x6c)
   ↓ write(pty_master_fd, "l", 1)
═════════════════════════════════════
内核 pty 子系统
   ↓ 数据进入"线规则"(line discipline)
   ↓ 当前是行模式 → 字符存入 input buffer,不发给 shell
   ↓ 但 termios 的 ECHO 标志位是 1
   ↓ → 内核 pty 把 'l' 同时回写到 pty 主端
═════════════════════════════════════
iTerm2 read(pty_master_fd) ← 'l'
   ↓ 把 'l' 字形(glyph)渲染到屏幕

   屏幕上出现了一个 'l'
       ↑
       └── 这个 'l' 是 ECHO 画的,不是 shell 画的
           shell 还没收到任何东西

    4.2 关键认知:屏幕上的字不是 shell 画的

    这是整篇文章最反直觉的一点:你打字时屏幕上出现的每个字符,默认情况下是内核 pty 子系统在"回显"(echo)给终端模拟器,shell 根本还没看到。

    证明:跑 stty -echo,此时打字屏幕上什么都不显示,但 shell 仍在收键盘——敲回车,命令照样执行。stty echo 恢复。关掉的是 pty 的回显行为,不是 shell 的输入。

    为什么这样设计?——历史包袱(物理 tty 时代回显由硬件做)+ 性能(不用 round-trip 到 shell)+ 兼容性(让所有 CLI 程序不改代码就能用 pty)。

    4.3 那 shell 什么时候收到键盘输入?

    继续上面的场景。你按完 'l' 之后,又按 's',然后回车:

       你按下 'l'  →  pty buffer: "l"     (回显 → 屏幕显示 l)
   你按下 's'  →  pty buffer: "ls"    (回显 → 屏幕显示 s)
   你按下 \n   →  pty buffer: "ls\n"  (行结束触发!)
                  ↓
                  内核 pty 把整行 "ls\n" 投递到 pty 从端
                  ↓
                  shell 的 read(stdin) 系统调用解除阻塞
                  ↓
                  shell 读到 "ls\n"
                  ↓
                  shell 解析、fork、exec /bin/ls

    关键点:行模式下,shell 一次性读到一整行,不是一个字符一个字符读。这就是为什么 ls、echo、grep 这些"老实"的命令行程序都不需要管按键——它们等回车,等到了再处理一整行。

    行模式下,你还能用 backspace、Ctrl-W(删一个词)、Ctrl-U(删整行)——这些编辑功能不是 shell 实现的,是 pty 的线规则(line discipline)实现的。证明:在 cat 命令里(cat 也是行模式)backspace 一样能用。

    4.4 ls 输出怎么显示到屏幕

    shell 接到 "ls\n",fork 一个子进程 exec ls。ls 写了一堆字符到 stdout:

    ls 进程
  │
  │ write(1, "file1.txt\nfile2.md\n...", 30)
  │ (fd 1 = stdout = pty 从端)
  ▼
═════════════════════════════════════
内核 pty 子系统(从→主方向)
═════════════════════════════════════
  │
  │ 这个方向没什么花活
  │ termios 的 OPOST 标志会做一些输出转换
  │ (比如 \n → \r\n,只在某些终端配置下)
  ▼
pty 主端
  │
  │ iTerm2 在 epoll/kqueue 等这个 fd
  ▼
iTerm2 read(pty_master_fd) → "file1.txt\nfile2.md\n..."
  │
  │ 解析 ANSI 转义序列(比如 \e[31m 是红色)
  │ 选字形、应用颜色
  ▼
渲染到屏幕

    注意"解析 ANSI 转义"这一步——颜色、光标移动、清屏全都是字节流里的特殊序列,不是 shell 处理的,是终端模拟器处理的。这个我们后面会专门讲。

    五、tty vs pty:真正的区别

    5.1 tty 是物理终端,pty 是软件模拟

    tty (TeleTYpewriter)
  历史包袱:1869 年的电传打字机 → 1970s 的物理终端 → /dev/tty
  含义:一个物理设备,内核有真实的设备驱动
  现状:几乎绝迹,只剩 Linux 虚拟控制台(Ctrl+Alt+F1-F6)
       /dev/tty1, /dev/ttyS0(串口),/dev/console

pty (Pseudo-TTY)
  发明:1980s,为了让 X Window 上的 xterm 跟传统 CLI 程序通信
  含义:内核里的一对 fd,主端和从端,主端有个用户态进程往里写
       从端表现得像 tty,有 termios、有信号、有行编辑
  现状:今天 99% 的"终端"会话都是 pty
       /dev/ptmx(主端打开点),/dev/pts/0, /dev/pts/1, ...
       macOS: /dev/ttys000, /dev/ttys001, ...

    5.2 实战:看自己挂在哪个 pty 上

    # 当前 shell 挂在哪个 pty
$ tty
/dev/ttys003           # macOS 的命名
# 或者
/dev/pts/0             # Linux 的命名

# 看系统上所有的 pty
$ ls -l /dev/pts/      # Linux
crw--w---- 1 user tty 136, 0 May 12 09:00 0
crw--w---- 1 user tty 136, 1 May 12 09:01 1
...

$ ls -l /dev/ttys*     # macOS
crw--w---- 1 user tty 16, 0 May 12 09:00 /dev/ttys000
crw--w---- 1 user tty 16, 1 May 12 09:01 /dev/ttys001
...

# 看自己所在进程组、会话、控制终端
$ ps -o pid,ppid,sid,tty,comm
  PID  PPID   SID  TTY  COMM
74321 74320 74321 ttys003 -zsh

    每开一个新终端窗口,系统就分配一个新 pty,从端命名递增。

    5.3 ssh 链路上的 pty

    本地 Mac                                远程 Linux
┌────────────────────┐                  ┌────────────────────┐
│ iTerm2             │                  │                    │
│  ↕ 本地 pty        │                  │  远程 pty          │
│  ↕                 │   TCP/22 加密    │  ↕                 │
│ ssh 客户端 ────────┼─────────────────►│  sshd ────► 远程 zsh│
│ (不跑 shell,只转发)│                  │                    │
└────────────────────┘                  └────────────────────┘

    你 ssh 进去之后,键盘到屏幕之间有两组 pty:本地一组(iTerm2 ↔ ssh 客户端)、远程一组(sshd ↔ 远程 shell)。中间 ssh 用加密 TCP 把字节流双向转发。

    证明:ssh 进远端后跑 tty,会显示一个远端的 pty 名字(/dev/pts/N),跟本地的 tty 输出不一样。ssh -t 强制分配 pty(默认开),ssh -T 禁用——禁用之后远端没 pty,适合纯管道场景(ssh server "cat file" > local_file)。

    六、shell 不是终端:澄清最大的混淆

    6.1 "终端"这个词的 4 种用法

    工程师讨论"终端"的时候经常在说不同的东西。下面这张表把它讲清楚:

    说"终端"时,可能指的是 实际是什么 进程身份
    GUI 那个黑窗口 终端模拟器(iTerm2 / Terminal.app / Ghostty) 用户态 GUI 程序
    里面那个命令提示符 shell(bash / zsh / fish) 用户态 CLI 进程
    /dev/pts/0 那个设备 pty(伪终端) 内核对象
    tty 命令返回的东西 当前进程的控制终端(pty 从端) 内核对象

    这四个东西职责完全不一样:

    终端模拟器(iTerm2)
   职责:GUI 渲染、字体、键盘事件、ANSI 解析、Tab 标签
   不做:执行命令、补全、解析管道

shell(zsh)
   职责:解析命令行、变量、管道、重定向、补全、历史
   不做:画字符、处理键盘事件、连网络

pty(/dev/pts/0)
   职责:在主从两端搬字节、生成信号、行编辑、回显
   不做:认识命令、有 UI

CLI 工具(ls / vim)
   职责:做自己的业务
   不做:它什么都不知道,只读 stdin、写 stdout

    6.2 "iTerm2 比 bash 快" 这种话本身就错位

    我见过工程师讨论"WezTerm 比 iTerm2 快 5 倍",另一个工程师跳出来说"那是因为 fish 比 zsh 慢"——两人都错了,因为他们没意识到这俩讨论的不是同一个东西:

    终端模拟器的快慢          shell 的快慢
  ↑                         ↑
  渲染性能                   解析、补全、启动
  - GPU vs CPU 渲染         - 多少插件、补全脚本多大
  - 滚动是否丢帧            - 启动后调多少个外部命令
  - ANSI 处理多优           - PROMPT 函数多复杂

    Alacritty 配 bash 一样快,iTerm2 配 fish 一样可能慢启动——这两个维度互相正交。

    6.3 shell 也可以脱离终端跑

    shell 不一定要跑在 pty 从端:

    $ bash myscript.sh         # bash 的 stdin 是脚本文件,不是 pty
$ echo "ls; pwd" | bash    # bash 的 stdin 是管道
$ nohup my_server &        # nohup 把进程的 stdin/out/err 重定向到 /dev/null,
                           # 进程跟控制终端脱钩,关掉终端也不会被 SIGHUP 干掉

    判断当前进程有没有连 pty:isatty() 系统调用,或者命令 tty(连 pty 输出设备名,没连返回 "not a tty")。

    七、TERM、ANSI 转义、terminfo

    7.1 TERM 环境变量是干什么的

    $ echo $TERM
xterm-256color

    这个变量不是终端模拟器自己用的,是 CLI 程序用的。CLI 程序读 TERM,然后查 terminfo 数据库,知道这个终端能做什么: - 支持多少颜色(8 色?256 色?TrueColor?) - 怎么清屏(发什么字节序列) - 怎么移动光标 - 怎么改背景色

    # terminfo 数据库在哪
$ infocmp xterm-256color | head
#       Reconstructed via infocmp from file: /usr/share/terminfo/...
xterm-256color|xterm with 256 colors,
        am, bce, ccc, km, mc5i, mir, msgr, npc, xenl,
        colors#0x100, cols#80, it#8, lines#24, pairs#0x10000,
        ...

# 查当前终端支持多少色
$ tput colors
256

    7.2 ANSI 转义码:终端模拟器认识的"指令"

    CLI 程序写到 stdout 的字节里,带特殊前缀(ESC = \x1b = \e)的字节序列被终端模拟器识别为指令,而不是字符:

    $ printf '\e[31mRED\e[0m\n'      # 红色文字
$ printf '\e[2J\e[H'             # 清屏 + 光标回左上
$ printf '\e[5;10H'              # 移动光标到 5 行 10 列

    常用 ANSI 序列(记几个常用的就行):

    \e[0m            重置所有属性
\e[1m            加粗     \e[4m  下划线
\e[31m           红色文字 \e[42m 绿色背景
\e[38;5;208m     256 色之一(208 = 橙)
\e[38;2;255;100;0m  TrueColor (R=255 G=100 B=0)
\e[2J            清屏      \e[H  光标回原点  \e[K  清到行尾

    这些指令是"终端协议"——终端模拟器认得,CLI 程序按规范发就行。不同终端模拟器实现的协议子集略有不同(比如老 xterm 没 TrueColor),所以才要 terminfo 数据库做能力声明。

    7.3 tmux 嵌套时 TERM 必须改

    这是一个所有用 tmux 的工程师迟早遇到的坑:

    # 在 iTerm2 里直接跑
$ echo $TERM
xterm-256color           # 一切正常

# 进 tmux
$ tmux
$ echo $TERM
screen                   # ← 坑!颜色少了,vim 颜色错

    原因:tmux 在 iTerm2 和你之间又插了一层,它对内层 shell 也是一个"终端",但 tmux 默认声明自己是 screen(8 色)。要让内层享受 256 色,你必须在 tmux 配置里告诉它:

    # ~/.tmux.conf
set -g default-terminal "tmux-256color"
set -ag terminal-overrides ",xterm-256color:RGB"

    第一行让内层 shell 看到的 TERM 是 tmux-256color(256 色),第二行让 tmux 把 TrueColor 透传到外层 iTerm2。

    为什么会这样:TERM 描述的是"我紧贴着的那一层终端的能力",不是物理屏幕的能力。层数变了,TERM 就要跟着变。tmux / screen / ssh 嵌套 N 层,TERM 就要在每层正确传递。

    7.4 一个最简单的"颜色检测"脚本

    $ tput colors           # 终端宣称的颜色数(8 / 16 / 256)
$ echo $COLORTERM       # truecolor / 24bit / 空
# 打印连续色带,平滑就是 TrueColor,有"梯"就是 256 色映射
for i in $(seq 0 255); do
    printf "\e[38;2;${i};100;200m█\e[0m"
done; echo

    八、行模式 vs 原始模式:两套人格

    8.1 行模式(canonical mode)

    默认状态:pty 等到回车再把整行发给 shell。这是 ls、bash、cat 默认的工作方式。

    特征:
  - 按键先在内核 pty buffer 里攒着
  - backspace、Ctrl-W、Ctrl-U 由内核处理(行编辑)
  - 回车之前 shell 看不到任何东西
  - Ctrl-C → 内核生成 SIGINT,发给前台进程组
  - Ctrl-D → 行尾发 EOF
  - Ctrl-Z → 内核生成 SIGTSTP,挂起前台进程
  - 默认 echo 开,你打字屏幕显示

适用:
  - shell 提示符
  - cat、grep、sed 等"一行一行读 stdin"的程序
  - read 命令

    8.2 原始模式(raw mode)

    vim / less / fzf / Claude Code 都干这个事:程序启动时切到原始模式,自己处理每个字节。

    特征:
  - 按一个键就投递一个字节,不等回车
  - 行编辑 / 信号生成全关掉,Ctrl-C 不再触发 SIGINT(是字节 0x03)
  - echo 关掉(程序自己决定要不要显示)
  - 程序读到的就是原始按键字节
  - 程序退出时必须自己恢复成行模式,否则 shell 错乱

适用:
  - 全屏交互程序:vim、less、tmux、htop、fzf、btop
  - REPL:python、node 的"按 ↑ 取历史"
  - TUI 框架:ncurses、textual
  - Claude Code 也是连 pty 的 raw 模式程序

    8.3 怎么进 raw 模式

    C 接口是 tcsetattr() + termios 结构体,但日常你只需要知道 stty:

    # 看当前 termios 设置
$ stty -a
speed 9600 baud; rows 50; columns 200; line = 0;
intr = ^C; quit = ^\; erase = ^?; kill = ^U; eof = ^D; eol = <undef>;
...
icanon iexten echo echoe echok ...    # 行模式标志位
                                      # 加 - 前缀就是关闭

# 进 raw 模式 (临时测试用)
$ stty -icanon -echo

# 恢复
$ stty sane     # 重置成"健全"状态
# 或
$ stty cooked   # 显式回到行模式

    如果 vim 崩溃没正常退出,你的终端就会卡在 raw 模式——打字不显示、回车没反应。救命命令是盲打 reset 或 stty sane:

    $ reset    # 盲打,看不见,敲完回车

    8.4 一个反直觉问题:为什么 cat 跑着 Ctrl-C 退出,vim 不退

    $ cat       # 行模式
你输入字符
^C          # 内核 pty 看到 Ctrl-C → 发 SIGINT → cat 死

$ vim       # raw 模式
你按 Ctrl-C
内核 pty 直接把 0x03 字节投递给 vim
vim 自己决定怎么处理 —— vim 当成"中断当前命令"
要退 vim 还得 :q

    Ctrl-C 不是 shell 处理的,不是程序处理的,默认情况下是 pty 内核(行模式时)处理的——它把按键解释成信号,发给前台进程组。raw 模式下 pty 不再做这个,程序自己处理那 1 个字节。

    九、行宽、颜色、Unicode、locale

    9.1 256 色 vs TrueColor

    8/16 色         \e[31m / \e[91m       1980s 标配
256 色          \e[38;5;Nm            2000s 主流
TrueColor(16M) \e[38;2;R;G;Bm        2014 后所有现代终端支持

    检测:tput colors(256 或更少)、echo $COLORTERM(truecolor / 24bit / 空)。

    9.2 Unicode 宽度问题(CJK 全角)

    终端是字符网格——每个"格子"是一个字符宽度。问题:

    ASCII 'a'       宽度 1 格
中文 '你'       宽度 2 格(全角)
emoji '🦀'      宽度 2 格,但有的终端给 1 格
肤色 emoji      ZWJ 组合,宽度 1 / 2 / 4 都见过,各终端实现不一致

    为什么 tmux 状态栏宽度对不齐 / 表格错位:终端模拟器和你的程序对 Unicode 宽度的判断不一致——程序按 wcwidth() 算宽度,终端按自己的字体度量画字符,光标位置就跟实际显示位置不同步。没有彻底解法,只能用现代终端 + 避免在状态栏里塞 emoji(尤其 ZWJ 组合)。

    9.3 locale 和编码乱码

    $ locale
LANG="en_US.UTF-8"
LC_CTYPE="en_US.UTF-8"
...

    locale 决定终端按什么编码解码字节:UTF-8 / GBK / Latin-1。中文显示成 ?? 的根因是 CLI 输出 UTF-8 字节、但 locale 是 C 或 POSIX(只认 ASCII),非 ASCII 字节就变 ?。

    修法:export LANG=en_US.UTF-8(或 zh_CN.UTF-8)、export LC_ALL=en_US.UTF-8。ssh 进远端机器经常乱码,因为远端的 locale 数据没装齐——Linux 上要先 locale-gen 生成对应 locale。

    十、管道与重定向时,终端在哪里

    10.1 isatty:程序怎么知道自己连着终端

    每个程序的 fd 0(stdin)、1(stdout)、2(stderr)都可以是不同的东西:

    直接跑 ls          ls | less              ls > file.txt

fd 1: pty 从端     fd 1: 管道写端          fd 1: 普通文件
isatty(1)=true     isatty(1)=false        isatty(1)=false

    isatty(fd) 是一个系统调用,返回这个 fd 是不是连着 tty/pty。CLI 工具用它判断"我现在是给人看,还是被管道喂下游"。

    10.2 实战:ls 为什么管道里不上色

    $ ls                   # 输出有颜色
$ ls | cat             # 输出没颜色

    ls --color=auto 的实现(伪代码):

    if os.isatty(sys.stdout.fileno()):
    use_color = True       # 终端,上色
else:
    use_color = False      # 管道或文件,不上色

    为什么默认这样设计:管道下游程序读到带 ANSI 转义的字节流会被它困住——比如 ls --color=always | grep file1,grep 找的字符串其实是 "\e[34mfile1\e[0m",可能 match 不上。

    现代 CLI 工具(ls、grep、rg、bat、git diff)都遵循同一个约定:isatty(stdout)=true → 上色,false → 纯文本。手动控制:--color=always 强制上色,--color=never 强制不上色。

    10.3 实战:在你自己代码里用 isatty

    import sys

if sys.stdout.isatty():
    print('\033[31mERROR\033[0m')   # 上色
else:
    print('ERROR')                  # 纯文本

    C / Go / Rust 都有对应的接口(isatty(fileno(stdout)) / term.IsTerminal() / IsTerminal::is_terminal())。给所有 CLI 程序加这个判断,是你工程素养的入门门槛——别再写出"管道下游一堆 ANSI 转义码"的工具了。

    10.4 重定向 stderr 时,colored stderr 怎么办

    $ my_tool 2>err.log              # stderr 写文件,isatty(stderr)=false,自动不上色
$ my_tool 2>&1 | less -R         # stderr 重定向到 stdout 再喂 less,less -R 解析 ANSI

    这就是为什么有 --color=always——某些场景下你故意要把颜色字节流送给下游(下游能 ANSI 解码,比如 less -R、bat)。

    十一、常见误解一锅端

    这一节是这一篇文章的精华——你身边 90% 的工程师对终端的认知卡在这些误解上:

    11.1 "iTerm2 比 bash 快"

    错位:iTerm2 是终端模拟器(渲染层),bash 是 shell(命令解析层)。两者维度不同,不能比。说"iTerm2 比 GNOME Terminal 快"或"zsh 比 fish 慢启动"都是合法陈述;混在一起就是搞不清栈分层。

    11.2 "把 bash 装上就有终端了"

    错:bash 是 shell,没有终端模拟器它没地方画字符。在一台只装了 bash 没装 GUI 的服务器上,你要么用物理串口接它,要么 ssh 进去——后者本质是用本地的 iTerm2 当远端 bash 的"终端模拟器"。bash 自己永远不画屏幕。

    11.3 "tmux 是 shell"

    错:tmux 是终端复用器(terminal multiplexer),它自己也是一个"终端模拟器 + 一堆 pty 管理器":tmux 在外层 pty 里跑,内层为每个 window/pane 创建一个新 pty,把外层和内层 pty 之间倒腾字节。所以 tmux 嵌套不会爆栈——只是多了几层 pty 而已。tmux 不是 shell,你在 tmux 里跑的还是 zsh/bash。这一段会在 16-17 篇深入讲。

    11.4 "ssh 进去用的还是本地的 shell"

    错:ssh 客户端不跑 shell,它只转发字节。你 ssh 进远端,远端 sshd 在远端 fork 一个 pty + 远端 shell,本地 ssh 把你键盘字节通过 TCP 加密送给远端 sshd,sshd 喂给远端 pty 主端。本地和远端是两套独立的 pty + shell。echo $SHELL 在 ssh 进去之后显示的是远端的 shell。

    11.5 "Ctrl-C 是 shell 处理的"

    错:Ctrl-C 在行模式下是 pty 内核子系统处理的——它把按键解释成 SIGINT,发给前台进程组(可能是 shell,可能是 shell fork 的子进程)。在 raw 模式下,Ctrl-C 就是一个 0x03 字节,投递给程序自己处理。shell 通常不参与 Ctrl-C 的解析。

    11.6 "ls 输出的颜色是 shell 加的"

    错:颜色是 ls 自己加的(在它写到 stdout 之前)。颜色用 ANSI 转义码表达,终端模拟器读到这些字节后渲染成颜色。shell 不参与上色。这就是为什么 cat file_with_ansi.txt 也能显示颜色——cat 没碰那些字节,直接喂给 pty,iTerm2 解析渲染。

    11.7 "Claude Code 在终端里跑,所以它就是 CLI"

    有点对,但不完整:Claude Code 是一个连 pty 的 raw 模式 TUI 程序——它跟 vim、fzf、htop 一类,接管整个屏幕、自己处理每个按键、画自己的界面。它不是"在 shell 里跑普通程序",它是"shell fork 出来,把 pty 接管,自己当主"。这就是为什么 Ctrl-C 在 Claude Code 里不一定中断它——raw 模式下的 0x03 字节,Claude Code 自己决定怎么处理。
