自动补全、别名与函数:配置债是怎么积出来的 ​

打开任何一个工程师的 .zshrc,有一半概率你能看到这样的段落:

zsh
alias gst='git status'
alias gco='git checkout'
alias gcm='git commit -m'
alias gca='git commit --amend'
alias gp='git push'
alias gpl='git pull'
alias gpr='git pull --rebase'
alias gl='git log --oneline --graph --decorate'
alias gla='git log --all --oneline --graph'
alias gd='git diff'
alias gdc='git diff --cached'
alias gst='git stash'
alias gsp='git stash pop'
# ...再来 50 行

这个段落看起来很专业——一个会用 git 的工程师才能写出这种缩写。但 80% 是抄来的:抄自 oh-my-zsh 的 git plugin,抄自某篇博客,抄自团队前同事的 dotfiles。别名表往往最终的命运是——半年后你只用其中 5 个,剩下 45 个挤在文件里占位、Ctrl+R 时干扰搜索、新人进项目 gst<TAB> 不出来还以为是 git 子命令。

更深的坑在另一个地方:这种别名表里 30% 其实是错的——

  • alias mkcd='mkdir -p && cd' —— 你以为 mkcd foo 会执行 mkdir -p foo && cd foo,其实是 mkdir -p && cd foo(参数追加在整个字符串末尾,不是插入)
  • alias gca='git commit -a --amend --no-edit' —— 在脚本里 bash deploy.sh 调用,别名根本不展开,报 gca: command not found
  • alias ll='ls -lah' —— 新同事 which ll 看不到,以为是命令,跑到 prod 机器上 ll 报错

这一篇拆开:alias、function、abbr、keybind、script 各自解决哪一层问题,什么时候用哪个,为什么大多数 alias 应该升级成 function,为什么 fish 的 abbr 是更好的设计,以及 zsh 补全系统(compsys)到底在做什么。

一句话先记住:alias 是「输入更短」的快捷,不是「行为更复杂」的封装。任何要传参、要分支、要管道的逻辑,用函数;任何要在脚本里复用的,写成可执行文件。把这条记住,你就能砍掉 .zshrc 里一半的债。

一、五个层级:从一键到一程序 ​

工程师在终端里"打字更省事"这件事,至少有五种实现方式——按"离命令源码远近"排序:

keybind   ── 把字符串注入到当前命令行                  (输入层)
abbr      ── 输入到空格/回车时展开为完整命令            (输入层)
alias     ── 命令前缀替换为另一字符串                  (解析层)
function  ── shell 函数,可传参、有逻辑、有局部变量    (执行层)
script    ── 独立可执行文件,$PATH 可达,跨 shell      (进程层)

每一层解决的问题不同。搞混层就是配置债的源头——你用 alias 干 function 的事、用 function 干 script 的事,最后一定有一边出问题。

1.1 五层对照表 ​

工具何时触发可参数化在脚本里有效跨 shell可发现性透明度
keybind按下绑定键✗✗✗★ 差(键位看不见)★ 高(插入字符明示)
abbr输入空格/回车✗✗fish 独有★★★ 好(展开当场看见)★★★ 完全透明
alias行首单词命中✗(只能末尾追加)✗(默认非交互不展开)bash/zsh 共享★★ 中(alias 命令可列)★ 不透明
function命令行查找命中✓ $@ / $1✓(同 shell 内)bash/zsh 共享、fish 不同★★ 中(type foo 可看)★ 不透明
script$PATH 找到✓ 标准 argv✓(完全独立进程)★★★ 全场景★★★ 好(which foo 可找)★★ 中(看脚本内容)

这张表是这一篇所有结论的根基——后面每一段都对应这张表里某一行的展开。

1.2 每一行一个真实例子 ​

zsh
# keybind:Ctrl+G 插入 "git status",回车自己按
bindkey -s '^g' 'git status^M'           # ^M 是 Enter

# abbr (fish):输入 gst 按空格,字面变成 git status
abbr -a gst 'git status'

# alias:行首 g 替换成 git
alias g='git'

# function:有参数有逻辑
mkcd() { mkdir -p -- "$1" && cd -- "$1"; }

# script:独立可执行文件,$HOME/.local/bin/git-clean
#!/usr/bin/env bash
set -euo pipefail
git branch --merged | grep -vE '^\*|main|master' | xargs -r git branch -d

这五段不是同一件事的五种写法,是五件不同的事。

二、alias 的真正限制 ​

alias 是 shell 里最被滥用的特性——因为它入门门槛低,新人一上手就能学会,但它的限制和坑在写过几年配置之前没人讲清楚。这一节挨条拆。

2.1 alias 不能"插入"参数,只能"追加" ​

最常见的误解:alias 像函数一样能在中间插参数。它不能。

zsh
alias mkcd='mkdir -p && cd'

# 你以为:mkcd foo  →  mkdir -p foo && cd foo
# 实际是:mkcd foo  →  mkdir -p && cd foo
#                       ↑ 这里没参数,mkdir 就建了一个空名目录(报错)
#                                      ↑ cd 用了 foo,目录没建出来更走不到

alias 的工作机制:纯文本替换——把行首匹配到的别名换成它的右值,剩下的字符照原样追加在后面。

原始: mkcd foo
替换: [mkdir -p && cd] foo
结果: mkdir -p && cd foo

foo 作为整行命令的"剩余参数"被附在末尾——末尾是 cd 的参数,不是 mkdir -p 的参数。alias 没法解析"我想在 mkdir -p 后面塞、再在 cd 后面也塞" —— 它根本不知道你心里的意图。

遇到这种场景,必须用函数:

zsh
mkcd() {
  mkdir -p -- "$1" && cd -- "$1"
}

函数有 $1,可以把参数插到任意位置——这是 alias 永远做不到的。

2.2 alias 在子 shell / 脚本里不展开 ​

bash
# .zshrc
alias gst='git status'

# 交互终端里:能用
gst                          # ✓ → git status

# 但在脚本里:
# deploy.sh
#!/bin/bash
gst                          # ✗ gst: command not found

根因:alias 默认只在交互 shell 展开——脚本(非交互 shell)读 .zshrc 也不展开 alias,因为 alias 是行编辑层的特性,不是解析层的特性。

bash 里可以强开,但要在脚本里 setopt:

bash
#!/bin/bash
shopt -s expand_aliases              # 强制展开 alias
source ~/.bashrc                     # 再加载有 alias 的文件
gst                                  # 这下能用了

但这是反模式——脚本依赖你的个人 alias,换台机器就崩。正确做法:脚本里直接用全名 git status,把 alias 当成只在你自己交互终端有效的东西。

zsh 同理:

bash
zsh -c 'gst'                         # ✗ 找不到 gst

-c 启动的是非交互 zsh,不读 .zshrc 也不展开 alias。

2.3 alias 等号两边不能有空格 ​

bash
alias gst = 'git status'             # ✗ 报错(shell 以为你在跑 alias 加三个参数)
alias gst='git status'               # ✓ 正确

shell 语法里赋值左右不能有空格——变量赋值、别名定义、export 都一样。

2.4 alias 覆盖标准命令的风险 ​

zsh
alias rm='rm -i'                     # "安全 rm",删之前问一下

听起来合理——直到你切到没这 alias 的机器(root 用户、prod 机器、SSH 容器),肌肉记忆基于错误的前提:

bash
rm large_file.bak                    # 你以为会问,直接删了
rm -rf node_modules                  # -rf 让 -i 失效,你以为有保护,其实没有

根本问题:让命令"看起来不像它实际是的样子"。alias vim='nvim' 同理——ssh 到只有 vim 的机器,肌肉记忆失效。

经验:改"显示"的 alias 可以(ls、grep --color),改"语义"的 alias 严重谨慎(rm、mv、cp)。真要"安全 rm" → 装 trash-cli 起个新名 rmm,不覆盖 rm。

2.5 绕过 alias + 调试 ​

bash
\rm file.bak                         # 反斜杠 = 不展开 alias(最常用)
command rm file.bak                  # command builtin 也跳过 alias
type gst                             # 查 gst 实际是什么(alias?function?command?)
alias                                # 列所有 alias

新人最常问"我打 gst 没反应",99% 是 alias 没生效,type gst 一查就知道。

关于递归:alias ls='ls --color=auto' 不会无限递归——shell 知道 alias 自展开后不再继续展开。但 alias foo=bar; alias bar=foo 这种互引用,shell 检测到会中断。

三、function 才是正确的 alias 替代 ​

function 才是 shell 里"封装命令"的正确工具——有参数、有逻辑、有局部变量、有 return code。前面 alias 不能做的事,函数都能做。

3.1 何时升级 alias 成 function ​

经验法则,满足任一条就升级:

□ alias 超过 20 个字符
□ 需要参数插入到中间(不是末尾追加)
□ 有管道 |
□ 有 if / for / case 分支
□ 有需要保护的引号
□ 需要返回非 0 退出码

举例:

zsh
# 别名版(超 20 字符 + 需要分支)
alias gpr='git pull --rebase origin $(git symbolic-ref --short HEAD)'
# ↑ 这条其实能跑,但读者要在脑子里展开,可维护性低

# 函数版(同等功能,清晰)
gpr() {
  local branch
  branch=$(git symbolic-ref --short HEAD 2>/dev/null) || {
    echo "Not on a branch" >&2
    return 1
  }
  git pull --rebase origin "$branch"
}

升级后的好处:

  • 失败时给提示(Not on a branch)
  • 返回非 0 退出码,可以接 && 链
  • 半年后回来看,知道这函数在做什么

3.2 函数语法 bash / zsh 通用 ​

bash
# bash / zsh 都支持的标准语法
mkcd() {
  mkdir -p -- "$1" && cd -- "$1"
}

# zsh 还支持 function 关键字(可选,基本一样)
function mkcd {
  mkdir -p -- "$1" && cd -- "$1"
}

记住一条:用 name() { ... } 这种语法,bash zsh 都吃,跨 shell 不出问题。

3.3 关键参数 + 局部变量 ​

bash
my_func() {
  local tmp                          # local 关键:防止污染调用方
  echo "全部参数: $@"                # "$@" 数组,各自独立(几乎总是用这个)
  echo "第一参数: ${1:-default}"     # 默认值
  echo "参数个数: $#"
  tmp=$(mktemp) && trap "rm -f $tmp" EXIT
}

$@ vs $*:几乎所有场景用 "$@"(带双引号),$* 把所有参数合成单字符串,通常不是你想要的。

没有 local 的后果:临时变量变成全局,污染调用方环境——大型脚本里这种 bug 极难追。函数里所有临时变量都用 local。

3.4 默认参数 vs alias 死字符串 ​

zsh
# alias 永远做不到:
alias gco='git checkout '            # 跑 gco 就报 "you must specify a branch"

# 函数轻松搞定:
gco() {
  git checkout "${1:-main}"          # 没传参就用 main
}

gco                                  # → git checkout main
gco feature/x                        # → git checkout feature/x

${var:-default} 是参数扩展(POSIX 标准),var 没设或为空就用 default。

3.5 zsh 的 fpath autoload 函数 ​

zsh 还有一种"自动加载函数"机制——懒加载、文件名 = 函数名、启动开销几乎为零:

zsh
# .zshrc
fpath=($HOME/.config/zsh/functions $fpath)
autoload -Uz mkcd gpull gco

文件 ~/.config/zsh/functions/mkcd(无后缀):

zsh
mkdir -p -- "$1" && cd -- "$1"

第一次调用 mkcd foo 时,zsh 才把这个文件读进来——启动时不读,节省冷启动时间。06 篇讲过这套机制,适合 10+ 个函数的项目。

3.6 fish 的函数语法不同 ​

fish
function mkcd
    mkdir -p -- $argv[1]
    and cd -- $argv[1]
end
# 保存为 ~/.config/fish/functions/mkcd.fish,fish 自动按文件名 load

fish 的设计哲学:函数是磁盘上的文件,一个文件一个函数,调用时按需 load。这是比 bash/zsh 更干净的设计——bash/zsh 的 fpath autoload 其实是模仿 fish 这套模型。

四、fish 的 abbr:被低估的好设计 ​

fish 独有一个叫 abbr(abbreviation,缩写) 的特性——输入到空格或回车时,缩写自动展开成完整命令,并显示在屏幕上。

fish
abbr -a gst 'git status'

# 你打字:
g s t<空格>                          # 屏幕上立刻变成 "git status "
                                     # 你看到的就是要执行的

4.1 abbr 和 alias 的关键差别 ​

alias gst='git status'
    输入 gst<回车> → 执行 git status
    屏幕/历史里都是 "gst"

abbr -a gst 'git status'
    输入 gst<空格> → 屏幕字面变成 "git status "
    历史里存的是 "git status"

abbr 的核心是「透明」:你看到的命令就是你跑的命令——不存在"alias 实际是什么"的隐藏映射。三个直接后果:

  1. 新人看你的终端,直接学到的是完整命令——记住的是 git status 本身,不是你的快捷映射
  2. Ctrl+R 搜的是展开后的完整命令——搜 git 就行,不用想"我当时打的 gst 还是 gs"
  3. 复制粘贴到聊天群,别人能直接跑——粘出去的是 git status

4.2 bash / zsh 模拟 abbr ​

zsh-abbr 是 zsh 的 abbr 实现,95% 体验对齐 fish abbr:

zsh
zinit light olets/zsh-abbr
abbr gst='git status'

bash 没有原生 abbr,可以用 bind + READLINE_LINE 自己写,不推荐,维护成本高。

经验:用 zsh 就装 zsh-abbr 替代大部分 alias——一个月后你的 alias 表能砍一半。

4.3 abbr / alias / function 怎么选 ​

要"输入更短",不改语义                 → abbr(zsh-abbr / fish 原生)
要"输入更短",兼容旧机器               → alias
要"传参",要"分支",要"组合"           → function
要"在脚本里复用"                       → script(独立可执行文件)

这就是核心结论——背下来。

五、何时用 keybind 注入字符串 ​

bindkey 是 zsh 的快捷键绑定——把"按某个键"和"执行某段 widget"绑起来。90% 用法是搬来插入字符串:

zsh
# Ctrl+G 在当前命令行插入 "git status",并按回车
bindkey -s '^g' 'git status^M'

# Ctrl+R(其实默认就是)反向搜索历史
bindkey '^r' history-incremental-search-backward

# Alt+. 插入上一条命令的最后一个参数(超有用)
bindkey '\e.' insert-last-word

-s 的意思是 send string —— 按下键就把字符串注入到当前命令行(^M 是回车)。

5.1 keybind 的优缺点 ​

优点:✓ 极快(比打字快 5 倍) ✓ 不留 alias/function 痕迹
缺点:✗ 发现性差 ✗ SSH 不稳 ✗ 跟 tmux 抢键位 ✗ 脚本里不能用

5.2 实际中 keybind 该用在哪 ​

不是用在"日常命令"——日常命令用 abbr / alias / function 更合适。keybind 适合:

  1. 整合 fzf(12 篇专讲):Ctrl+R 接管成 fzf 历史、Ctrl+T 文件选择、Alt+C 目录跳转
  2. 行编辑:Ctrl+A/E 行首尾、Alt+B/F 词间跳——emacs/vi 模式自带,不算自定义
  3. Alt+.:插入上一条命令的最后一个参数(超有用,默认就有)
zsh
# fzf 标准三件套(12 篇会展开)
source /opt/homebrew/opt/fzf/shell/key-bindings.zsh
source /opt/homebrew/opt/fzf/shell/completion.zsh

反模式:重绑 Ctrl+C(中断信号,工程灾难)、Ctrl+R 改成奇怪的东西(让 fzf-history 失效)。自己的快捷只绑 1-3 个真的高频的——多了你自己都记不住。

六、shell 补全是什么(以及它不是什么) ​

90% 的工程师把"补全"当 shell 自带功能——它不是。

6.1 补全的真相 ​

你打 `git che<TAB>`:
1. shell 补全引擎(zsh compsys / bash readline+bash-completion / fish 内建)接管 TAB
2. 引擎查"git 这个命令的 completion 函数"是哪个文件
3. 这个函数被调用,返回候选列表
4. shell 渲染候选(下拉、菜单、单匹配自动补)

补全不是 shell 自带的能力——补全是每个工具自己提供 completion 脚本,装到 shell 能找到的地方。git 的 completion 是 git 项目自己写的,kubectl 的是 kubectl 自己生成的,你打 git che<TAB> 能补出来是因为 brew 装 git 时把 completion 文件一起放到了 zsh 的 fpath 目录。

6.2 三大 shell 的补全位置 ​

shell补全文件位置文件名约定
bash/etc/bash_completion.d/、/usr/share/bash-completion/completions/命令名(git)
zsh$fpath 里的目录(/opt/homebrew/share/zsh/site-functions/)_命令名(_git)
fish~/.config/fish/completions/、/usr/share/fish/completions/命令名.fish
bash
echo $fpath                          # 查看 zsh 的 fpath
ls $(brew --prefix)/share/zsh/site-functions/ | grep _git

6.3 fish 的"开箱补全" ​

fish 比 bash / zsh 多两个能力,让 fish 用户几乎不用手工配 completion:

  1. 自动从 man page 解析 flag:fish 后台扫 /usr/share/man/man1/*.gz,解析每个命令的 --flag,自动生成 completion
  2. 从命令历史学习:你打过 npm run dev,下次 npm run <TAB> 把 dev 排前面

bash / zsh 想模仿"开箱补全"很难——zsh 的 _man 能补 man page 名,但不会自动给所有命令生成。

6.4 brew 自动装 completion ​

bash
brew install kubectl

# brew 自动把 completion 链接到 /opt/homebrew/share/zsh/site-functions/_kubectl
# 这个目录在 $fpath 里,下次开 zsh 就有 kubectl 补全

这是 macOS zsh 体验好的关键。Linux 上 apt / pacman / dnf 也类似——包内带 completion 文件,装到 /usr/share/zsh/vendor-completions/。

Linux 上用 apt / pacman / dnf 装,也类似——包内带了 completion 文件,装到 /usr/share/zsh/vendor-completions/ 之类目录。

七、zsh 补全系统(compsys)简介 ​

zsh 的补全系统叫 compsys(completion system),是 2026 业界最强大的补全引擎——能补 git 子命令、kubectl 子命令、ssh 主机名、tar 文件,配置极其丰富。

这一节不深教——这是工程师视角,知道有这套机制,需要时查文档。

7.1 三个核心 builtin ​

zsh
autoload -Uz compinit && compinit       # 加载补全;扫 $fpath 注册所有 _xxx 文件
compdef _git g                          # 告诉 zsh:别名 g 也用 git 的补全(alias 继承补全)

compinit 做什么:扫 $fpath 里所有 _xxx 文件,为每个注册补全,结果缓存到 ~/.zcompdump。第一次可能花 100-200ms——这是 06 篇讲过的 zsh 启动单点。

补全函数命名:_命令名(下划线开头),文件必须在 $fpath 里。

compdef 让 alias 继承补全:这是 zsh 比 bash 强的地方,bash 要写 complete -F _git g,语法不同。

假设你写了个工具 myapp,有子命令 start / stop / restart。文件 ~/.config/zsh/functions/_myapp:

zsh
#compdef myapp

_myapp() {
  local -a commands=(
    'start:启动 myapp 服务'
    'stop:停止 myapp 服务'
    'restart:重启 myapp 服务'
  )
  _arguments -C '1: :->command' '*::arg:->args'
  case $state in
    command) _describe 'command' commands ;;
    args)
      case ${words[1]} in
        start) _arguments '--port[端口]:port:' ;;
        stop)  _arguments '--force[强制]' ;;
      esac ;;
  esac
}
_myapp "$@"

关键行:#compdef myapp 标识、_arguments 主力函数、_describe 带描述候选、case $state 多层状态机。深入(动态候选、缓存、文件类型过滤)看 man zshcompsys——这一节的目的是让你知道这套机制存在。

八、三个常见 completion 装法 ​

你日常遇到的 90% completion,来自这三个来源:

8.1 brew / apt 自动装 ​

bash
brew install gh kubectl docker

# brew 把:
# /opt/homebrew/share/zsh/site-functions/_gh
# /opt/homebrew/share/zsh/site-functions/_kubectl
# /opt/homebrew/share/zsh/site-functions/_docker
# 全部装好

# 你只要 fpath 包含这个目录,compinit 跑过,就有补全

最省事——99% 的工具 brew 一装就有补全。新机器上 brew install 完直接开 zsh,大部分工具补全到位。

8.2 工具自己生成 ​

很多新一代 CLI 工具(尤其是 Go / Rust 写的)内建 completion 生成命令:

bash
# kubectl
kubectl completion zsh > ~/.config/zsh/functions/_kubectl

# gh(GitHub CLI)
gh completion -s zsh > ~/.config/zsh/functions/_gh

# cargo(Rust)
mkdir -p ~/.config/zsh/functions
rustup completions zsh cargo > ~/.config/zsh/functions/_cargo
rustup completions zsh rustup > ~/.config/zsh/functions/_rustup

# fnm
fnm completions --shell zsh > ~/.config/zsh/functions/_fnm

# uv(Python 包管理)
uv generate-shell-completion zsh > ~/.config/zsh/functions/_uv

# zoxide
zoxide init zsh                       # 不是文件,是 eval

模式:工具有 xxx completion zsh 子命令,输出补全脚本到 stdout,你重定向到 fpath 里的文件。

通常一次性——除非工具版本大改,你装完就再没动过。

8.3 第三方仓库 zsh-users/zsh-completions ​

社区维护一个额外的补全包,补充官方没出补全的工具:

zsh
# 用 zinit:
zinit ice blockf atpull'zinit creinstall -q .'
zinit light zsh-users/zsh-completions

# blockf:阻止插件改 fpath
# atpull'zinit creinstall -q .':更新后重新注册

装上后,你获得 200+ 工具的补全——brew、tmux、yarn、composer、各种小众工具。

但谨慎:zsh-completions 仓库的补全质量参差,有些和官方的冲突——你的 _brew 可能既来自 brew 自己又来自 zsh-completions,新的覆盖旧的(取决于 fpath 顺序)。实际用得起来,但偶尔出怪事。

8.4 fpath 顺序的优先级 ​

zsh
echo $fpath
# /opt/homebrew/share/zsh/site-functions          ← 1
# /Users/me/.zinit/completions                    ← 2
# /usr/share/zsh/5.9/functions                    ← 3

zsh 找 _git 时,从前往后找,第一个命中就用。所以:

  • brew 装的 (1) 会优先于 zsh-completions (2)
  • zsh-completions (2) 优先于系统自带 (3)

遇到补全异常,先 which _git 看实际加载的是哪个文件。

九、菜单 / 模糊补全:让 TAB 更聪明 ​

zsh 默认的 TAB 行为不算特别好用——单次 TAB 没匹配,第二次 TAB 才列出选项。配几条 zstyle,体验立刻起飞。

9.1 菜单 + 模糊匹配(最值钱的 4 条 zstyle) ​

zsh
# 1. 菜单选择:TAB 直接进菜单,方向键选
zstyle ':completion:*' menu select

# 2. 大小写不敏感 + 模糊匹配(允许中间字符)
zstyle ':completion:*' matcher-list \
  'm:{a-zA-Z}={A-Za-z}' \
  'l:|=* r:|=*'

menu select:打 git che<TAB>,第一次 TAB 就进入菜单,方向键选:

$ git che[TAB]
─────────────────────────────────────────
   checkout    cherry      cherry-pick
─────────────────────────────────────────

matcher-list 两条规则:

  • m:{a-zA-Z}={A-Za-z} —— 大小写互通(打小写匹大写)
  • l:|=* r:|=* —— 允许左右各匹配任意字符(打 chk 能匹 checkout)

9.2 启用补全描述 + 分组 ​

zsh
zstyle ':completion:*' verbose yes
zstyle ':completion:*' group-name ''
zstyle ':completion:*:descriptions' format '%F{yellow}-- %d --%f'

效果:

$ kubectl <TAB>
-- main commands --
get      -- 显示一个或多个资源
apply    -- 通过文件名或 stdin 应用配置
delete   -- 按文件名、stdin 等删除资源

带描述的补全 = 自带文档——新人不需要 kubectl --help,TAB 就教他了。

9.3 完整可用配置(8 行) ​

zsh
autoload -Uz compinit
if [[ -n ${ZDOTDIR:-$HOME}/.zcompdump(#qN.mh+24) ]]; then compinit; else compinit -C; fi

zstyle ':completion:*' menu select=2
zstyle ':completion:*' matcher-list 'm:{a-zA-Z}={A-Za-z}' 'l:|=* r:|=*'
zstyle ':completion:*' group-name ''
zstyle ':completion:*:descriptions' format '%F{yellow}-- %d --%f'
zstyle ':completion:*' verbose yes
setopt AUTO_PARAM_SLASH                    # 命中目录加 trailing /

8 行配置,每次 TAB 都比裸 zsh 体验高一个量级——而且没引入任何插件。

9.4 fzf-tab:用 fzf 接管 TAB ​

如果你已经在用 fzf(12 篇专讲),fzf-tab 是 zsh 补全的"现代终极态":

zsh
zinit light Aloxaf/fzf-tab
zstyle ':fzf-tab:*' switch-group ',' '.'

效果:TAB 不再是 zsh 自带的菜单,而是 fzf 风格的模糊搜索框——模糊 + 预览 + 颜色 + 多选。12 篇会专讲 fzf,这里只点名。

十、真实工程师的 alias / function 库 ​

讲了一堆原理,给一份实用范例。这是我维护多年的 aliases.zsh + functions.zsh——30-50 行内的"够用",不是 oh-my-zsh 那种 200 行。

10.1 短到值得 alias 的 ​

zsh
# ─────── 显示增强(改输出,不改语义) ───────
alias ls='ls --color=auto -F'           # 加颜色加文件类型符号
alias ll='ls -lah'                      # 长格式,人类可读大小
alias la='ls -A'                        # 显示隐藏文件
alias grep='grep --color=auto'
alias diff='diff --color=auto'

# ─────── 单字符高频缩写 ───────
alias g='git'                           # 但你应该用 abbr 替代
alias k='kubectl'
alias d='docker'
alias dc='docker compose'

# ─────── 路径快捷 ───────
alias ..='cd ..'
alias ...='cd ../..'
alias ....='cd ../../..'

# ─────── 让 alias 也走 alias 链 ───────
# zsh 的 `alias -g` 是全局别名,任意位置展开(慎用)
alias -g GP='| grep'                    # 用法:ls -la GP foo  →  ls -la | grep foo
alias -g GL='| less'                    # ls -la GL

全局 alias -g 是 zsh 特性 —— 不推荐多用,但 GP GL 这种管道前缀很方便。

10.2 应该是函数的 ​

zsh
# ─────── 通用工具 ───────
mkcd() { mkdir -p -- "$1" && cd -- "$1"; }

# 提取压缩包,自动识别类型
extract() {
  case "$1" in
    *.tar.gz|*.tgz)  tar xzf "$1" ;;
    *.tar.bz2|*.tbz) tar xjf "$1" ;;
    *.tar.xz)        tar xJf "$1" ;;
    *.tar)           tar xf  "$1" ;;
    *.zip)           unzip   "$1" ;;
    *.rar)           unrar x "$1" ;;
    *.7z)            7z x    "$1" ;;
    *) echo "Don't know how to extract: $1" >&2; return 1 ;;
  esac
}

# 跑命令并把退出码 / 耗时显示出来(脚本调试常用)
timed() {
  local start=$EPOCHREALTIME
  "$@"
  local rc=$?
  local elapsed=$(printf '%.2f' "$((EPOCHREALTIME - start))")
  echo "→ exit=$rc time=${elapsed}s" >&2
  return $rc
}

EPOCHREALTIME 是 zsh 自带的浮点时间戳——精度毫秒。bash 没这个,要用 date +%s.%N。

10.3 git 系列(几乎都要参数,函数更合适) ​

zsh
gcb() { git symbolic-ref --short HEAD 2>/dev/null; }      # 当前分支名

gco() { git checkout "${1:-main}"; }                       # checkout,默认 main
gcm() { git commit -m "$*"; }                              # commit 消息(支持空格)

gpr() {                                                    # pull rebase 当前分支
  local b; b=$(gcb) || { echo "Not on a branch" >&2; return 1; }
  git pull --rebase origin "$b"
}

gpc() {                                                    # push 当前分支(首次 -u)
  local b; b=$(gcb) || return 1
  git push -u origin "$b"
}

git-clean() {                                              # 删已合并到 main 的本地分支
  git branch --merged main | grep -vE '^\*|main$|master$' | xargs -r git branch -d
}

为什么是函数不是 alias:全部要参数或动态读 git 状态。

10.4 跨工具组合(函数的甜点区) ​

zsh
# 在所有包含某字符串的目录里挑一个 cd 过去
cdgrep() {
  local dir
  dir=$(rg -l "$1" 2>/dev/null | xargs -I{} dirname {} | sort -u | fzf) || return 1
  cd "$dir"
}

# 把当前目录压缩成 tar.gz
tarhere() {
  local name="${1:-$(basename "$PWD")}"
  tar -czf "../${name}.tar.gz" .
}

"两个工具 + 几行胶水"的场景,函数最合适——比写独立 script,改起来更快。

10.5 篇幅控制 ​

aliases.zsh 15-25 行,functions.zsh 30-50 行——超过就在攒债。半年清理一次,grep .zsh_history 找从未用过的名字删掉。

十一、跨 shell 的别名同步 ​

bash 和 zsh 可以共享 alias / function——fish 不行:

bash
# ~/.config/shell/aliases.sh —— bash/zsh 共用
alias g='git'
mkcd() { mkdir -p -- "$1" && cd -- "$1"; }

# 在 ~/.bashrc 和 ~/.zshrc 都加:
[ -f ~/.config/shell/aliases.sh ] && source ~/.config/shell/aliases.sh

只用 POSIX 子集才能两边都跑(echo -e、read -p、数组语法等都有微差)。复杂脚本各写各的。

fish 不是 POSIX shell——语法、变量、函数完全不同。别想"统一",bash/zsh 一份,fish 一份,这两套生态不同源,强行抽象会让两边都变扭。

dotfiles 管理(22 篇专讲):仓库分目录 bash/ zsh/ fish/ shared/,用 chezmoi / stow / yadm 落地。

十二、反对的写法 ​

这一节列我反复见过的反模式——你或多或少都犯过几条:

12.1 alias 多到 200 个 ​

抄 oh-my-zsh git plugin 把 80 个别名搬到自己 .zshrc,80% 你这辈子用不到。gcam 和 gca 你能不思考说出区别吗?

解法:只留你这个月真的用过 5 次以上的——剩下的删。真要用 git 的某个长命令,TAB 补全就够——git che<TAB> 已经够快。

12.2 alias 覆盖标准命令 ​

zsh
alias rm='rm -i'
alias cp='cp -i'
alias mv='mv -i'

前面讲过的——你的肌肉记忆基于"rm 会问我",到没这 alias 的机器上就翻车。

真要"安全 rm",装 trash-cli:

bash
brew install trash-cli
alias rmm='trash-put'                # 单独起个名,不覆盖 rm

rmm(删到回收站)和 rm(真删)语义分离——不混淆,不依赖肌肉记忆。

12.3 函数名跟标准命令重名 ​

zsh
vim() {
  command vim "$@" && echo "Exit code: $?"
}

这种"包装"的诱惑:加个统计、加个日志、加个 nvim 自动切换。但代价是:任何脚本里 vim file.txt 都先跑你的函数——调试时崩了不知道是 vim 本体的 bug 还是你的 wrapper 的 bug。

经验:除非有强需求,不要覆盖标准命令名——起个新名(vv / nv / myvim)。

12.4 用别名替代脚本 ​

zsh
alias deploy='cd ~/work/myapp && git pull && npm install && npm run build && pm2 reload all'

这条 alias 干了 5 件事——任何一步失败就全崩,还没 log。升级路径:超过两步逻辑 → 函数(有 && 短路 + 错误打印);需要日志、测试、跨机器复用 → 写成 ~/.local/bin/deploy 脚本(27 篇专讲)。

判断标准:重要、复杂、需要日志/测试 → 脚本。只你自己跑 → 函数。单纯字符串简化 → alias。

12.5 抄网上千行 dotfiles ​

GitHub 上某 5k star dotfiles 仓库 aliases.zsh 687 行 functions.zsh 1244 行——那是别人的肌肉记忆,不是你的,90% 你用不到。抄逻辑、不抄文件——从零写,只加你真用到的。

12.6 alias 套 alias 深嵌套 ​

zsh
alias g='git'
alias gs='g status'                    # 嵌套别名
alias gst='gs --short'                 # 嵌套两层

zsh 会展开,bash 不会。半年后没人读得懂 gst 跑的什么。展平:alias gst='git status --short',一眼看到本体。

12.7 fish 用户硬写 alias ​

fish 里 alias gst='git status' 其实是 function 语法糖,还得 funcsave。fish 用户应该用 abbr——abbr 的"展开后透明"是 fish 的设计哲学,别人 ssh 进来看到 gst 还以为找不到命令,fish abbr 帮你避开这个。

12.8 keybind 绑了 50 个 ​

zsh
bindkey -s '^[a' 'do_a^M'
bindkey -s '^[b' 'do_b^M'
# ... 47 more lines

你能记住吗?新人看见你两秒钟按了三个键跑了不知道什么命令,直接懵。keybind 限制在 3-5 个——主要是 fzf 三件套和上一参数(Alt+.)。

十三、看完这一篇,你应该能 ​

□ 看到 `alias mkcd='mkdir && cd'` 第一反应是「这不可能对」
  —— alias 不能"插入"参数只能"追加",必须用函数

□ 知道为什么 `bash -c 'gst'` 不识别你的 alias
  —— 非交互 shell 默认不展开 alias,这是设计不是 bug

□ 把超过 20 字符 / 需要传参 / 有管道的 alias 升级成函数
  —— 不再让 alias 干 function 的活

□ 用 fish abbr 或 zsh-abbr 替代 80% 的 alias
  —— 获得"展开后透明"的体验,新人能直接学到完整命令

□ 配出"模糊 + 菜单 + 分组"的 zsh 补全(8 行 zstyle)
  —— TAB 体验翻倍,而且不引入插件

□ 看见网上 1000 行 alias 表,第一反应是「这是别人的牙刷」
  —— 抄逻辑,不抄文件

13.1 自查清单 ​

□ 数你 .zshrc 里多少条 alias,删掉过去一个月没用过的
□ 把有空格 / 有参数插入 / 有管道的 alias 升级成函数
□ 装 zsh-abbr(或迁 fish),用 abbr 替代单纯输入快捷
□ 加 zstyle 菜单 + 模糊匹配 + 分组,提升 TAB 体验
□ 把所有覆盖标准命令的 alias(rm/cp/mv)改成新名
□ 检查常用工具有没有 completion(kubectl/gh/cargo),没有就装
□ `bindkey -L | wc -l` 超过默认 + 10 就清理一下

做完这 7 条,你的 alias / function / completion 这一块就过关了——不再是配置债,而是工程资产。
