# 终端工程系列写作计划

> 来源:`未来系列规划.md` 第三类「纵深主题」新增项,⭐⭐⭐⭐。
> claudeLearning 教了「**Claude Code 怎么用**」,devopsLearning 教了「**生产怎么跑**」,gitLearning 教了「**版本怎么管**」,但都没回答一个核心问题:**你每天在终端里的肌肉记忆,是怎么从一台机器复制到另一台、从去年用到明年、从本地推到远端的**。
> **看完这层,你能让一台新机器在 30 分钟内变成你「家」的样子**——tmux、Neovim、zsh、fzf、Nix、dotfiles、Claude Code,不再是一堆零散的工具名,而是知道每件解决什么问题、什么时候上、什么时候是过度配置。

---

## 设计思路

六层结构,从「心智」到「现代终端」:

```
心智 + 基础(tty / pty / 进程 / 信号 / Unix 文本流哲学)
   → Shell 选型与日常(zsh / fish / 提示符 / 补全 / 历史)
   → 核心 CLI 工具链(grep-sed-awk → rg / fd / bat → fzf → jq → ssh)
   → Multiplexer + 编辑器(tmux / Zellij + Vim / Neovim / Helix)
   → Dotfiles 与可复现环境(chezmoi / Nix / mise / devcontainer)
   → 工作流深入(脚本工程化 / 任务运行器 / Claude Code 联动 / 现代终端)
```

**这个系列不是 Linux 命令大全**——`man` 还在那儿,你不需要再背一遍 `tar` 的 30 个 flag;**默认你站在 2026 之后**:大家都在 macOS / Linux 上用 zsh / fish + tmux + Neovim,Claude Code 已经是日常工作流的一部分,Nix / chezmoi 不再小众,Warp / Ghostty 重新定义了「终端是什么」。

不深入到 ncurses 源码、tty 设备驱动、glibc readline 内部,**而是讲清楚"应用工程师 / AI 工程师 / 全栈工程师能装、能改、能复现、能传承的工作流"**——你每天在终端里跑测试、写代码、调 Claude、远端登录、批处理日志,这些都站在这套工具链上。

## 写作模板

每篇遵守"为什么这个工具 → 心智模型 → 最小配置 → 替代方案与局限"的四段式:

```
1. 为什么这个工具存在
   - 没它的世界长什么样(上一代是谁,死在哪)
   - 它解决的根本问题是什么

2. 心智模型(一张图 / 一段对照)
   - 概念图 / 数据流 / 状态机,任选其一
   - 关键概念用最少的名词讲清楚

3. 最小配置(能用、能改、能复现)
   - 贴 10-30 行真能跑的配置或脚本
   - 标注「这一行如果删掉会怎样」
   - 调优 / 排错的关键开关

4. 替代方案与局限
   - 同代位竞争者(tmux vs Zellij / Neovim vs Helix / Nix vs mise)
   - 什么场景不该用它
   - 2026 之后业界往哪个方向走
```

不抄官方 README——**只贴让读者「看到这一行配置改了什么」的最小片段**,让读者「拿走就能用」。

## 节奏

```
01-04  心智:看完知道"一次按键到屏幕显示之间发生了什么,信号到底是 SIGINT 还是 SIGTERM"
05-09  Shell:看完知道"为什么 zsh 启动慢、补全是怎么回事、历史命令为什么必须用 atuin"
10-15  CLI:看完知道"为什么 rg 比 grep 快 10 倍,fzf 嵌进所有命令是怎么做到的"
16-21  Multiplexer + 编辑器:**生产力的真正护城河**,tmux + Neovim 一旦内化就回不去了
22-26  环境管理:**新机器半小时复刻**,Nix / chezmoi / mise 决定你能不能"换机不换工作流"
27-30  工作流:**和 Claude Code / AI 工作流结合**,看完知道"终端的下一站长什么样"
```

**一个硬指标**:看完 16-26 这 11 篇,你应该能在一台干净的新 Mac / Linux 上,30 分钟内:

- 装好包管理器 + dotfiles 一键 sync
- tmux 配置 + 字体 + 主题 + 提示符全部到位
- Neovim / Helix 打开就能写代码 + 跑 LSP
- ssh config 就绪,远端机器一键 attach
- Claude Code 能在 tmux pane 里跑长任务,自己回头看输出

能做到这个,这个系列就值了。

再加上 27-30 这 4 篇,你应该能讲清楚:

- "为什么 just 比 Makefile 更适合 2026 的项目,什么时候还是该用 Make"
- "Warp / Ghostty / WezTerm 各自赌的是哪条路,选型该看什么"
- "Claude Code 跟终端工作流之间该怎么编排,才不会变成新型号的'写满 IDE 的 monorepo'"

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. **不写 Linux 命令大全 / man 复印件**——`man` 在那儿,本系列只在工具有反直觉之处插一句「这里坑在哪」
2. 「先讲个 Unix 起源故事 / Bell Labs / Multics」——直接进重点,历史只在脚注引用
3. 罗列工具的全部 flag——只贴「让读者用得起来」的最小子集
4. 「这个工具我用了 N 年」式个人安利——读者关心选型逻辑,不关心你的情怀
5. **不画图就讲 tty / pty / 进程组 / tmux 嵌套**——这些东西**不画图永远讲不清楚**
6. 抄 oh-my-zsh / LazyVim 的官方 README——官方文档在那儿,这里只补它没讲透的「为什么」和「什么时候不要用」
7. 「vim vs emacs 谁更牛」式的口水战——只讲各自擅长的场景,选型留给工程判断
8. 把「我的 dotfiles」当核心内容——本系列的目标是让读者建立自己的工作流,不是抄一份回去

## 立场声明

本系列**只服务于工程视角**——能装、能改、能复现、能传给团队新人。

所有示例配置限制在最小可跑片段,**不教你怎么从零写一个 1000 行的 .zshrc 或 init.lua**——那是个人审美,本系列的读者画像是「想把终端工作流系统化的工程师」,你需要懂这些工具是怎么编排的,但不需要花一个周末抄完别人的 dotfiles 再调一周。

**面向 2026:**

- macOS / Linux 双平台,不写 Windows(WSL 除外,放在 26 一句话带过)
- 默认 Apple Silicon / x86_64,不写远古架构
- 默认你已经会用 git,系列里不重复教
- 默认你已经在用 Claude Code,29 篇直接讲怎么和终端工作流接起来
