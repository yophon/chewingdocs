# 历史命令的工具化:从 .bash_history 到 atuin,把 shell 历史推到 2026

你按下 Ctrl+R,**想找"上周那条 kubectl rollout"**。打 `kub` —— 跳出一条无关的 `kubectl get pods`;打 `kubectl rollo` —— 跳出三条都是 `kubectl rollout status`,但都不是你要的;按 Ctrl+R 继续翻,翻了 30 屏全是字段类似但参数不对的命令,**最后你放弃了,打开 Slack 问"上周那条是不是 staging 那次"**。**这是 2026 年最荒诞的工程体验之一**——你的电脑保存了过去 5 年所有命令,**但你找不到任何一条**。

**真相**:bash / zsh 的 history 就是一个 1970 年代的文本文件 + 一个 1980 年代的线性反向搜索。**它不知道你这条命令在哪个目录跑、是否成功、跑了多久、是不是你常用的、是不是这台机器上跑的**。一个工程师 5 年的命令历史是几十万条潜在的肌肉记忆资产,**默认配置下 90% 都是查不出来的死数据**。

> 一句话先记住:**history 不是日志,是工程师的肌肉记忆数据库——它应该可搜索、可同步、有上下文、跨机器一致。这四点 bash/zsh 自带的 history 一项都没做到**。一个像样的现代 shell history 应该长得像 atuin:Rust + SQLite + 端到端加密同步 + 按 cwd / exit code / host 过滤的 fuzzy 搜索。**装一次,Ctrl+R 从此换一个体验**。

这一篇拆开来讲:**bash/zsh 默认的 history 哪里坏**、**三层升级路径**(shell 配置 / fzf-history / atuin)、**敏感命令怎么过滤**、**跨机器同步的工程考虑**、**history 数据该怎么再利用**。整套讲完,你会知道为什么我说"裸 .zsh_history 是工程债",以及怎么 30 分钟把这笔债还清。

## 一、bash / zsh 自带的 history 到底是什么

讲清楚现代化升级之前,**先得把"现状"的拆解做透**——不知道哪里坏,就不知道升级在升什么。

### 1.1 一句话物理模型

bash 和 zsh 的 history,**本质上就是一个纯文本文件**:

```
$ cat ~/.bash_history
ls -la
cd /tmp
git status
kubectl get pods
vim README.md
make test
...
```

每行一条命令,**就这么简单**。**这个设计来自 1980 年代的 ksh**,当时设计目标是"翻一翻最近几十条命令",**不是"找 3 年前那条"**。

zsh 的 `.zsh_history` 比 bash 多一点信息(开了 `EXTENDED_HISTORY` 之后):

```
$ cat ~/.zsh_history
: 1746518402:0;ls -la
: 1746518410:2;cd /tmp
: 1746518415:0;git status
: 1746518430:15;make test
```

格式:`: <timestamp>:<duration>;<command>`。**多了时间戳和耗时**,但**还是文本文件**,**还是一条命令一行**。

### 1.2 三件套:HISTFILE / HISTSIZE / SAVEHIST

无论 bash 还是 zsh,**history 系统由三个变量主导**:

| 变量 | 含义 | 默认值(zsh) | 推荐 |
| --- | --- | --- | --- |
| `HISTFILE` | history 写到哪个文件 | `~/.zsh_history` | 不动 |
| `HISTSIZE` | 内存里保留多少条 | 1000 | **100000** |
| `SAVEHIST` | 文件里保留多少条 | 0(!)/ 因发行版而异 | **100000** |

**反人类的默认**:zsh 默认 `SAVEHIST=0` —— 退出 shell **历史不落盘**。**90% 的人不知道这件事**,一直用着以为有 history,**直到换机器才发现啥都没了**。

bash 对应是 `HISTSIZE` / `HISTFILESIZE`,默认 500/500 —— **一周就满**。

### 1.3 bash 的默认致命缺陷:多终端会丢命令

bash 的默认 history 行为:**每个 shell 退出时**,**把内存里的 history 写到 HISTFILE,覆盖之前的**。

实测一下,**开两个 terminal**:

```bash
# 终端 A
$ echo "A1"
$ echo "A2"
$ echo "A3"

# 终端 B(同时打开)
$ echo "B1"
$ echo "B2"

# 终端 B 先 exit  → ~/.bash_history 写入: ..., B1, B2
# 终端 A 后 exit  → ~/.bash_history 写入: ..., A1, A2, A3
#                  → B1, B2 被覆盖,消失!
```

**这是 bash 默认行为下的真实数据丢失**——**你以为 history 记了所有命令,实际上后退出的 shell 覆盖了先退出的**。

zsh 默认行为略好(支持 `INC_APPEND_HISTORY` 之类的开关),**但你不主动开,就跟 bash 一个样**。

### 1.4 没记录的上下文

退一步看,**即便不丢命令,你拿到的也只是一串字符串**——`kubectl rollout status deployment/api-gateway`,**没有任何上下文**:

```
─ 这条命令在哪个目录跑的?              ✗ 没记录
─ 退出码是 0 还是非 0?                  ✗ 没记录(zsh 也只记 duration)
─ 跑了多久?                              ✗ bash 没记,zsh 开了 EXTENDED 才记
─ 你跑过几次?(频次)                   ✗ 没记
─ 这台机器叫什么?                       ✗ 没记
─ 在哪个 ssh session 里?                ✗ 没记
─ 上一条命令是什么?(命令上下文)       ✗ 没记
```

**你想要的"上周那条 kubectl"是个多维查询**——「**项目目录** = `~/work/payment` AND **关键词** = `rollout` AND **退出码** = 0 AND **大概时间** = 上周」——`.zsh_history` 那个文本文件**一个维度都没存**,**你只能靠关键词+体力 Ctrl+R 翻**。

### 1.5 Ctrl+R 反向搜索的真实体验

bash / zsh 的 Ctrl+R 是 **incremental reverse search** —— 增量反向搜索:

```
(reverse-i-search)`kub': kubectl get pods
```

按 Ctrl+R 再次搜下一条匹配。**核心问题**:

1. **精确子串匹配**——`kubectl rolllout`(拼错)什么都搜不到
2. **顺序敏感**——`rollout kubectl` 也搜不到,虽然你打的就是这两个词
3. **单行展示**——一次只看一条,**翻 30 条是 30 次 Ctrl+R**
4. **没有 fuzzy**——`krollst` 这种缩写记忆完全没用
5. **没有过滤**——不能"只显示在 ~/work 跑过的"

**1970 年代的 grep 模型**——精确字符串、线性搜索、单行回。

### 1.6 跨机器同步基本不可能

`.zsh_history` 是单机文件。**你换一台机器,这台机器上没有任何"你之前的命令"**。常见的"民间方案":

```bash
# 1. 用 git 同步 dotfiles 时把 .zsh_history 一起推
#    问题:多机器并发写会冲突,合并起来是噩梦

# 2. 用 syncthing / Dropbox 同步 .zsh_history
#    问题:同上 + 多机器同时打开 zsh,SHARE_HISTORY 边写边同步,数据混乱

# 3. 干脆放弃,每台机器 history 各自一份
#    问题:你最常用的"上次那条命令"经常在另一台机器上
```

**所有这些方案都是民间补丁**——核心矛盾是 `.zsh_history` 这个文本文件**没有任何 merge 语义**。**真正解决跨机器同步**,**必须用数据库 + 客户端去重**——这就是 atuin 解决的问题。

### 1.7 小结:为什么 1970 年代设计撑不到 2026

**bash/zsh 自带的 history 系统是 1970-1980 年代 ksh 思路的延续**——纯文本、行存储、线性搜索、单机。**符合那个时代每天敲几十条命令、用一台机器、不关心几年前命令的工程师的需求**。

到了 2026:**你每天敲几百到上千条命令、用 3 台以上机器(本地 macOS + 两台云 Linux)、想找 3 年前一条 `terraform import`**——**这套系统每一个维度都不够**:

| 维度 | bash/zsh 自带 | 你需要 |
| --- | --- | --- |
| 存储规模 | 几千条上限 | 几十万条 |
| 搜索 | 精确子串,反向 | fuzzy + 多关键词 |
| 上下文 | 命令文本 | cwd / exit / duration / host |
| 多终端 | bash 会丢,zsh 半凑合 | 实时共享 + 无冲突 |
| 跨机器 | 无 | 加密同步 |
| 重用 | Ctrl+R | 命令行查询 + 统计 |

下面三节,**就是把这张表的右列一项一项实现**。

## 二、第一层升级:把 shell 配置写对

**第一层免费、零依赖、立竿见影**——只是改 `.zshrc` 几行 setopt。**不做这一步,后面装 atuin 也半残**。

### 2.1 zsh 完整 history 配置

```zsh
# ~/.zshrc 的 history 段

HISTFILE=${ZDOTDIR:-$HOME}/.zsh_history
HISTSIZE=100000             # 内存里保留多少条
SAVEHIST=100000             # 文件里保留多少条

setopt EXTENDED_HISTORY        # 记录时间戳和耗时
setopt SHARE_HISTORY           # 所有终端实时共享
setopt HIST_EXPIRE_DUPS_FIRST  # 满了优先删重复
setopt HIST_IGNORE_DUPS        # 相邻重复不记
setopt HIST_IGNORE_ALL_DUPS    # 全局重复不记
setopt HIST_FIND_NO_DUPS       # 搜索时跳过重复
setopt HIST_IGNORE_SPACE       # 空格开头不记(临时敏感命令)
setopt HIST_REDUCE_BLANKS      # 去多余空白
setopt HIST_VERIFY             # !! 展开后先显示再回车
setopt INC_APPEND_HISTORY      # 命令结束就追加(SHARE_HISTORY 已含)
```

**每一行的意义**(删了会怎样):

| 选项 | 删了会怎样 |
| --- | --- |
| `HISTFILE` | history 不落盘,重启 shell 全部丢 |
| `HISTSIZE=100000` | 默认 1000,一周就用完;100000 占盘几 MB,几年都够 |
| `SAVEHIST=100000` | 默认 0(!),退出 shell 内存里那份不写文件 |
| `EXTENDED_HISTORY` | 不记时间戳和耗时——后面想分析"上个月那条"就没数据 |
| `SHARE_HISTORY` | 多终端互相看不到对方命令——这是 zsh 强于 bash 的最大开关 |
| `HIST_EXPIRE_DUPS_FIRST` | 满了删旧的,可能把你少用但珍贵的命令删了 |
| `HIST_IGNORE_DUPS` | 你连按 5 次 `ls` 进 5 条;影响阅读 |
| `HIST_IGNORE_ALL_DUPS` | 全局重复都记,history 90% 是 `git status` |
| `HIST_FIND_NO_DUPS` | Ctrl+R 翻历史时,翻 10 条都是同一条 `ls` |
| `HIST_IGNORE_SPACE` | **删了后你的 `export TOKEN=xxx` 会进 history,泄密** |
| `HIST_REDUCE_BLANKS` | `ls  -la`(多空格)和 `ls -la` 算两条 |
| `HIST_VERIFY` | `!!` 展开后直接执行——历史展开打错就翻车,这个开关让它先显示 |
| `INC_APPEND_HISTORY` | SHARE_HISTORY 已含,单独写也行(防 SHARE 被关) |

**这 11 行 setopt**,**是 zsh 用户的最小标配**。**没做这步,后面 atuin 也救不了你**——atuin 会从 zsh 的 history 导入数据,你 history 本身丢了一半,导出来也是缺的。

### 2.2 bash 对应版

bash 用户的对应配置:

```bash
# ~/.bashrc 的 history 段

HISTSIZE=100000
HISTFILESIZE=100000
HISTCONTROL=ignoreboth:erasedups    # ignoreboth = ignorespace + ignoredups
HISTTIMEFORMAT="%F %T "              # 给 history 命令加时间戳显示
shopt -s histappend                  # 退出时追加而不是覆盖(防多终端互相覆盖)

# 实时共享(代价:每条命令都有磁盘 IO)
PROMPT_COMMAND='history -a; history -n; '"${PROMPT_COMMAND}"
```

**`PROMPT_COMMAND` 这行是关键**:

- `history -a` 把当前 shell 的内存 history 追加到 HISTFILE
- `history -n` 从 HISTFILE 把别的 shell 写的新条目读进当前内存

**两条加在一起 = bash 的"伪 SHARE_HISTORY"**——bash 没有 zsh 那个真正的实时共享,这是民间最接近的方案。**代价**:每按一次回车就有两次磁盘 IO,**SSD 时代不是问题**。

### 2.3 实测:开关前后 Ctrl+R 体验对比

简单测一下,**这 11 行 setopt 加完后**,实际感受:

```
没改:                                改完后:
                                      
- HISTSIZE=1000(默认)                - HISTSIZE=100000
- SAVEHIST=0 → history 关 shell 丢   - SAVEHIST=100000 落盘
- 多终端互相覆盖                      - SHARE_HISTORY 实时共享
- ls 重复 30 条占着位置                - HIST_IGNORE_ALL_DUPS 去重
- 没时间戳                            - EXTENDED_HISTORY 有 epoch
- TOKEN=xxx 留在 history              - HIST_IGNORE_SPACE 不留痕(配合空格开头)

Ctrl+R 找命令:                       Ctrl+R 找命令:
"昨天那条 git rebase 在不在"          "昨天那条 git rebase"
70% 概率找不到(被覆盖 / 满了)        100% 在,不在就是 ALL_DUPS 删了
找到 1 条要翻 5-10 次 Ctrl+R          找到 1 条要翻 3-5 次(去重)
```

**还是不够好**(单行回、精确子串),但**地基已经稳了**。下面装 fzf 接管 Ctrl+R。

### 2.4 一个常见陷阱:HIST_EXPIRE_DUPS_FIRST 的副作用

`HIST_EXPIRE_DUPS_FIRST` 字面意思"history 满了时,优先删重复的"。**听起来很好**,**但有个隐患**:

```
HISTSIZE=10(假设极小),history 当前:
  1. ls
  2. cd ~/work
  3. git status
  4. ls
  5. make test
  6. ls
  7. git diff
  8. cd ~
  9. ls
  10. echo hi

← 现在你又敲了一条 vim README.md,需要挤出去一条

不开 HIST_EXPIRE_DUPS_FIRST:挤掉第 1 条 ls
开了:                         挤掉第 1, 4, 6, 9 中的任意一条(都是重复 ls)
```

**有了 HIST_IGNORE_ALL_DUPS 之后,HIST_EXPIRE_DUPS_FIRST 其实多余**——后者负责进入时去重,前者负责淘汰时去重,**两者只需要一个**。我推荐用 `HIST_IGNORE_ALL_DUPS`(更彻底)。

## 三、第二层升级:fzf 接管 Ctrl+R

**第一层是把数据存对**,**第二层是把搜索做模糊**——核心一句话:**让 Ctrl+R 不再是 1980 年代的精确反向搜索,而是 fzf 的 fuzzy multiline 选择器**。

### 3.1 fzf 是什么(只一行)

fzf = **command-line fuzzy finder**,Go 写的,**它的核心能力是"给一个候选列表,我做 fuzzy 匹配 + UI"**。fzf 完整介绍在 12 篇,**这一节只用它的 Ctrl+R 集成**——其他场景(选文件、选 git branch、选 process)那篇再说。

### 3.2 安装

```bash
# macOS
brew install fzf
$(brew --prefix)/opt/fzf/install     # 这一步会问几个问题,全选 y

# Ubuntu
sudo apt install fzf

# 通过 git(任意平台)
git clone --depth 1 https://github.com/junegunn/fzf.git ~/.fzf
~/.fzf/install
```

`install` 脚本会往你的 `.zshrc` 加一行:

```zsh
[ -f ~/.fzf.zsh ] && source ~/.fzf.zsh
```

`~/.fzf.zsh` 里做三件事:

1. 把 fzf 二进制加到 PATH
2. 注册 fzf 的 zsh **key-bindings**(Ctrl+R / Ctrl+T / Alt+C)
3. 注册 fzf 的 zsh **completion**

**Ctrl+R 就被接管了**。

### 3.3 接管后的体验

```
不装 fzf 的 Ctrl+R:
   (reverse-i-search)`kub': kubectl get pods
   ↑ 一行一次,按 Ctrl+R 翻下一条匹配,精确子串

装了 fzf 的 Ctrl+R:
   ╭─ fzf 弹出全屏 ──────────────────────────────────────╮
   │ > kub                                              │
   │   8765/8765 (5)                                    │
   │                                                    │
   │ > kubectl get pods                                 │
   │   kubectl rollout status deployment/api-gateway    │
   │   kubectl logs -f api-gateway-7d8f9c-x2k4m         │
   │   kubectl describe pod api-gateway-7d8f9c-x2k4m    │
   │   kubectl exec -it api-gateway-7d8f9c -- sh        │
   │   ...                                              │
   ╰────────────────────────────────────────────────────╯
   
   - 输入 kub → fuzzy 匹配,实时缩小候选
   - 上下方向键选,Enter 选中插入到命令行
   - 输入 "kub rol" → 两个词都包含的命令冒出来
   - Tab 多选(不太用在 history)
   - Esc 取消
```

**这个 UI 直接把"找命令"从 30 秒压到 3 秒**——你不需要记住完整命令,**只要记"大概有 kub 和 rol 两个词"**。

### 3.4 默认绑定的三个键

fzf 默认装好后,zsh 多了三个绑定:

| 键 | 功能 |
| --- | --- |
| **Ctrl+R** | history 模糊搜索(本节重点) |
| Ctrl+T | 当前目录 fuzzy 选文件,插入到命令行 |
| Alt+C | 子目录 fuzzy 选,选了之后 cd 过去 |

**Ctrl+T 和 Alt+C 是 12 篇的内容,这里只用 Ctrl+R**。

### 3.5 fzf-history 的几个有用环境变量

```zsh
# ~/.zshrc 里设(在 source ~/.fzf.zsh 之前)

# Ctrl+R 的高度:占当前终端 40%
export FZF_CTRL_R_OPTS="
  --height 40%
  --layout=reverse
  --preview 'echo {}'
  --preview-window down:3:wrap
  --bind 'ctrl-/:toggle-preview'
"

# 默认搜索算法:'v2' 比 'v1' 慢但更准,history 这种少量数据不用调
# 大小写:smart-case(全小写时忽略大小写,有大写时区分)
export FZF_DEFAULT_OPTS="
  --smart-case
  --tiebreak=index
"
```

**`--tiebreak=index` 是个细节**——多个匹配同等优先级时,**用 history 的原始顺序排**(最近的在上),**而不是 fzf 默认的字符串排序**。

### 3.6 fzf-history 的天花板

fzf 接管 Ctrl+R 之后,**95% 的"找命令"场景一秒搞定**。**但仍有几个天花板没破**:

1. **数据源还是 `.zsh_history` 那个文本文件**——没存的命令(被多终端覆盖丢的)还是找不到
2. **没有 cwd 过滤**——你想"只在 `~/work/payment` 里搜",做不到
3. **没有 exit code 过滤**——你想"只搜跑成功过的",做不到
4. **不能跨机器**——别的机器跑过的命令,这台机器搜不到
5. **没有统计**——你想知道"过去 30 天 git 命令占多少",做不到

**这五个天花板就是 atuin 要破的**。

## 四、第三层升级:atuin(主推)

**这一节是本篇的核心**。atuin 不是 fzf 的替代品,**它是 history 系统的替代品**——它接管 history 的存储、搜索、同步、统计,**把 1970 年代设计推到 2026**。

### 4.1 atuin 是什么

```
出身:     2021 年开始,作者 Ellie Huxtable(英国)
语言:     Rust
存储:     SQLite(本地数据库)
同步:     可选自建 server,客户端端到端加密
搜索:     fuzzy(默认)/ prefix / fulltext(可切)
集成:     zsh / bash / fish / nushell / xonsh 都支持
许可:     MIT
社区:     主流,2024 起加速,2026 是事实标准
```

**关键设计选择**:

- **本地 SQLite** —— 100 万条命令秒回查询
- **结构化字段** —— 命令文本不是唯一字段,还有 7 个维度
- **端到端加密同步** —— 客户端用 secret key 加密,server 看不到明文
- **替换 Ctrl+R** —— 装完后 Ctrl+R 直接走 atuin

### 4.2 atuin 记录的 7 个维度

**这是 atuin 区别于 fzf-history 的根本**——每条命令记的不只是字符串:

| 字段 | 含义 | 用例 |
| --- | --- | --- |
| `command` | 命令文本 | 搜关键词 |
| `cwd` | 跑命令时的工作目录 | 按项目过滤 |
| `exit` | 退出码 | 只看成功的 |
| `duration` | 跑了多久(纳秒) | 找慢命令 |
| `session` | shell session ID | 同次 ssh 里的命令 |
| `hostname` | 主机名 | 多机器场景 |
| `timestamp` | 跑的时间(UTC) | 时间范围过滤 |

**对比 zsh 的 EXTENDED_HISTORY**:zsh 只多了 timestamp 和 duration,**还少 cwd / exit / session / hostname 四个核心维度**。

### 4.3 安装与初始化

```bash
# 一键脚本(macOS / Linux)
curl --proto '=https' --tlsv1.2 -LsSf https://setup.atuin.sh | sh

# 或 brew
brew install atuin

# 或 cargo
cargo install atuin

# 注册账号(用于跨机器同步,可跳过)
atuin register -u myname -e me@example.com

# 不想跨机器同步就跳过 register,直接用本地数据库

# 把 zsh 的 .zsh_history 导入 atuin
atuin import auto      # 自动识别当前 shell 类型并导入

# 在 zsh 里启用 atuin(替换 Ctrl+R)
echo 'eval "$(atuin init zsh)"' >> ~/.zshrc

# bash 用户
echo 'eval "$(atuin init bash)"' >> ~/.bashrc

# fish 用户
echo 'atuin init fish | source' >> ~/.config/fish/config.fish

# reload shell
exec zsh
```

**5 分钟内**,**你的 Ctrl+R 已经是 atuin 了**——全屏 UI、fuzzy 搜索、上下文显示。

**对比 fzf-history**,多出来三个杀手锏:

1. **时间戳** —— 一眼看到"哪天跑的"
2. **目录** —— 一眼看到"在哪个项目里跑的"
3. **耗时** —— 一眼看到"这命令跑了 5 秒,不是马上回的"

**这三个上下文是从字符串变成"工程档案"的关键**——`kubectl rollout` 这种命令,**5 个项目都跑过,光看命令文本你区分不出来,看目录就秒分**。

### 4.5 atuin 默认替换 Ctrl+R 之后,Ctrl+R / 上箭头各做什么

```
默认绑定:
  Ctrl+R   →  atuin 弹全屏 UI,fuzzy search
  ↑(上箭头)→ 默认仍是 zsh 的 history-search-backward(按前缀过滤上一条)

你可以让上箭头也走 atuin:
  eval "$(atuin init zsh)"        ← 默认绑 Ctrl+R + Up
  eval "$(atuin init zsh --disable-up-arrow)"   ← 只绑 Ctrl+R
  eval "$(atuin init zsh --disable-ctrl-r)"     ← 只绑 Up,不接管 Ctrl+R
```

**我推荐 `--disable-up-arrow`** —— 上箭头保留 zsh 原本"按前缀过滤"的行为(打 `git` 然后↑就只翻 git 命令),atuin 全屏 UI 留给 Ctrl+R。**这样两套工作流互补,不冲突**。

### 4.6 atuin 的搜索模式

atuin 支持 3 种搜索模式(`~/.config/atuin/config.toml` 配):

```toml
# fuzzy(默认)—— 字符全部出现就匹配,顺序无所谓
search_mode = "fuzzy"

# prefix —— 必须从开头匹配
# search_mode = "prefix"

# fulltext —— 命令文本里包含输入(子串)
# search_mode = "fulltext"

# skim —— 比 fuzzy 更激进的模糊算法
# search_mode = "skim"
```

**90% 的人用 `fuzzy` 就够**。**`prefix` 模式适合"我只想找 git 开头的"**——但其实 fuzzy + 输入 `^git`(fzf 的精确前缀语法)效果一样。

## 五、atuin 命令行进阶用法

**Ctrl+R 是 atuin 的入口,但远不是全部**——atuin 自己是一个命令行工具,**可以从命令行直接查询历史**。这一节是"把 history 当数据库用"的实战。

### 5.1 atuin search:命令行查询


**这是 atuin 最杀的体验**——**"我的肌肉记忆跟着我走"**,不再依赖某一台机器在不在。

## 七、过滤敏感命令:不让 history 变成泄密源

**history 最大的安全风险**:你打了 `export AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE`,**这条命令现在永远留在 `.zsh_history`,在 atuin SQLite 里,并且(如果开了同步)在云端**。**一台机器丢了 / dotfiles 仓库公开 / atuin server 被攻破 = AWS 凭证泄露**。

**这一节是工程兜底**——不学这一段,前面装的所有现代化工具反而是新型泄密源。

### 7.1 第一道防线:空格开头不进 history

```zsh
setopt HIST_IGNORE_SPACE        # zsh
# bash 对应:HISTCONTROL=ignorespace 或 ignoreboth
```

**用法**:命令前加一个空格:

```
$ export TOKEN=secret123        ← 进 history,危险!
$  export TOKEN=secret123       ← 前面有空格,不进 history,安全
```

**注意**:空格开头**只防 history 文件**——**atuin 默认也尊重这个开关**(不记录空格开头的命令),**但你要确认 atuin 的配置**:

```toml
# ~/.config/atuin/config.toml
# atuin 默认开启,但不保证
filter_mode_shell_up_key_binding = "session"   # 跟此话题无关,但相关
```

**心智**:**敏感命令永远先打空格,不要先打命令再"哦糟糕"**——一旦敲完回车,数据已经写进 history 了,后悔来不及。

### 7.2 第二道防线:atuin 的 history_filter

`~/.config/atuin/config.toml` 里:

```toml
# 这些正则匹配的命令永远不存进 atuin 数据库
history_filter = [
    "^secret.*",
    "^pass.*",
    "AWS_ACCESS_KEY",
    "AWS_SECRET_ACCESS_KEY",
    "OPENAI_API_KEY",
    "ANTHROPIC_API_KEY",
    "GITHUB_TOKEN",
    "PRIVATE_KEY",
    "BEGIN.*PRIVATE",
]
```

**这是 atuin 独有的能力**——zsh 的 `HIST_IGNORE_SPACE` 只能"主动加空格",atuin 的 `history_filter` **基于正则被动拦截**,**不依赖你记得加空格**。

**两道防线一起用**:

- 空格开头:**意识到敏感时**主动用
- history_filter:**忘了用空格**时兜底

### 7.3 atuin 的 cwd_filter 和 secrets_filter

新版 atuin 还支持按目录过滤:

```toml
# 这些目录里跑的命令一律不记
cwd_filter = [
    "^/Users/me/.password-store",
    "^/etc/secrets",
]

# 自动检测明显的 secret 格式(2024+ 版本)
secrets_filter = true
```

`secrets_filter = true` 让 atuin 用内置正则识别**类似 secret 的字符串**(高熵字符串、known token 前缀如 `sk-`、`AKIA`、`ghp_` 等),**自动不记录**。

**强烈推荐打开**——这是兜底的兜底,**比正则更可靠**。

### 7.4 已经泄露的命令怎么清理

如果你发现 history 里已经有敏感命令了:

```bash
# 看一眼受影响范围
atuin search "AWS_ACCESS_KEY"

# 全部删除
atuin search --delete "AWS_ACCESS_KEY"
atuin search --delete "AWS_SECRET"
atuin search --delete "TOKEN="

# 同步到云(把删除同步出去,所有 atuin 机器一起删)
atuin sync

# 别忘了同步删 .zsh_history!
# atuin 删的是它自己的 DB,zsh 的文本文件还在
sed -i.bak '/AWS_ACCESS_KEY/d' ~/.zsh_history
```

**最后一步是常被忘的**——atuin 数据库干净了,**但旧的 `.zsh_history` 文件还存着原文**。**git 里的 dotfiles 仓库**,**iCloud / Dropbox 备份**,**这些地方的 .zsh_history 都要检查**。

**最重要的事**:**凭证一旦进过 history,即使删了,也要假设它泄露了**——立刻去 AWS / GitHub / OpenAI 后台 rotate 那个 key。**删 history 是降低后续泄露概率,不是回滚已发生的泄露**。

## 八、跨机器同步:工程考虑

atuin 的跨机器同步是它对 fzf-history / mcfly 的最大差异化优势,**但工程上有几个决策点**。

### 8.1 同步架构

```
┌─────────────────┐        ┌─────────────────┐        ┌─────────────────┐
│  Laptop         │        │  Server-Dev     │        │  Server-Prod    │
│  atuin CLI      │        │  atuin CLI      │        │  atuin CLI      │
│  ~/.local/share/│        │  ~/.local/share/│        │  ~/.local/share/│
│   atuin/        │        │   atuin/        │        │   atuin/        │
│   history.db    │        │   history.db    │        │   history.db    │
└────────┬────────┘        └────────┬────────┘        └────────┬────────┘
         │                          │                          │
         │  HTTPS                   │  HTTPS                   │  HTTPS
         │  (加密的 history blob)   │  (加密的 history blob)   │
         │                          │                          │
         └──────────────────────────┴──────────────────────────┘
                                    │
                                    ▼
                          ┌──────────────────────┐
                          │  atuin sync server   │
                          │  (公共 / 自建)        │
                          │                      │
                          │  储存:加密的 blob    │
                          │  不能解密:没 key    │
                          └──────────────────────┘
```

**关键设计**:

- 每条 history 在**客户端用 secret key 加密**,server 收到的是密文 blob
- secret key 在每个客户端本地保存,**注册时生成**,**靠你 `atuin key` 命令导出/导入**
- server 完全可以是公共服务,**因为它解密不了**
- 客户端去重 / 排序 / 搜索 **完全在本地**,server 只是同步介质

### 8.2 自建 vs 公共 server

**官方公共服务** `https://api.atuin.sh` 免费,**但有几个工程考虑**:

| 维度 | 公共 server | 自建 |
| --- | --- | --- |
| 启动成本 | 0(register 就用) | docker 起一个,5 分钟 |
| 加密保护 | 客户端加密,理论安全 | 同样客户端加密 |
| 法务 | 数据存在英国 / 第三方 | 自家服务器 / 内网 |
| 可用性 | 依赖官方 | 自己保证 |
| 限额 | 免费版有 | 看你硬件 |
| 多人共享 | ✗ 单人单账号 | ✓ 你可以给团队成员开账号 |

**自建一句话**:

```bash
docker run -d \
  --name atuin-server \
  -p 8888:8888 \
  -v ./atuin-data:/data \
  -e ATUIN_DB_URI=sqlite:///data/atuin.db \
  ghcr.io/atuinsh/atuin:latest \
  server start

# 或者用 official docker-compose
# https://docs.atuin.sh/self-hosting/
```

**客户端切到自建**:

```toml
# ~/.config/atuin/config.toml
sync_address = "https://atuin.mycompany.com"
```

**自建场景**:**你介意 history 元数据(不是明文)走第三方,或者你想给小团队共享 server**——比如公司给所有 SRE 共用一个 atuin server,**每个人独立账号**,但运维侧统一备份。

### 8.3 加密的实际意义

很多人问:**"既然客户端加密,server 看不到明文,为什么我还要在意 server 是谁的?"**

回答:**加密保护的是「命令内容」,不是「元数据」**。server 仍然能看到:

- 你这个用户**什么时候**有活动
- **多少条** history 在同步
- **从哪个 IP** 同步(不开 VPN 时)
- 加密 blob 的**大小**(可推测命令长短)

**这些元数据对 99% 工程师无所谓,对极小数高敏感场景(政府 / 军工)有所谓**。**普通工程师用公共 atuin server 完全合理**——但你的"secret key 不要泄露"这条是底线。

### 8.4 secret key 的管理

```bash
# 注册时 atuin 生成一个 secret key,保存在 ~/.local/share/atuin/key
atuin key                          # 打印当前 key

# 备份(必须做):
atuin key > ~/Dropbox/atuin.key    ← 但这又泄露给 Dropbox 了
# 更好的做法:1Password / Bitwarden 存 atuin key

# 新机器恢复:
atuin login -u myname              ← 问邮箱密码 + secret key
# 把 1Password 里的 key 粘进去,搞定
```

**几个常见坑**:

- **secret key 丢了 = 历史永久解不开**——server 上存的是密文,key 没了密文就是垃圾
- **千万别把 key 推到 dotfiles 仓库**——atuin key 类似 GPG 密钥,**当成最高密级保管**
- **多机器之间共用同一个 key**——新机器 `atuin login` 时输的就是同一个 key,**这样所有机器解密同一份云端数据**

- `ls` 12% —— ls 高频本身正常,**但你应该确认 ls 是带 colorize / -F 的**
- `cat` 占 6% —— **应该全部换成 `bat`**(11 篇)
- `find` 1.5% —— **应该换成 `fd`**(11 篇)
- `kubectl` 5% —— **k8s 占了你 5% 时间,有没有 k9s / kubectx 之类的工具**

**这张表是个人工作流的"体检报告"**——你以为你在写代码,**实际上 50% 时间在敲 7 个高频命令**。

### 9.2 从 history 生成 alias / function 建议

写一个脚本扫 history,找"高频但还没做成快捷的命令":

```bash
# ~/bin/suggest-aliases
#!/usr/bin/env bash
# 找出过去 30 天跑 50+ 次的完整命令
atuin search --before "now" --after "30 days ago" --format '{command}' \
  | sort | uniq -c | sort -rn \
  | awk '$1 >= 50 {$1=""; print}' \
  | head -20
```

输出(示意):

```
git status
git diff --staged
git log --oneline -10
kubectl get pods -n payment
docker compose ps
ls -lah
```

**`git status` 65 次?` g s` 一个 alias 走起**。**`kubectl get pods -n payment` 75 次?写个 `kgpp` 函数**。

**这种自动建议比"我抄一份别人的 alias 列表"靠谱多了**——你抄来的 alias 90% 不是你的工作流,**自己 history 出来的 100% 是**。

### 9.3 翻 history 写 runbook

**真实场景**:你昨天临时排查了一个 staging 故障,**今天上午要写一份 runbook 给团队**。你记得"我大概敲了 7-8 条命令"。**怎么把它们捞出来**?

```bash
# 看过去 24 小时内,在 ~/work/payment 目录跑过的所有 ssh 相关命令
atuin search --cwd ~/work/payment --after "24 hours ago" --format '{command}' \
  | grep -E '(ssh|kubectl|journalctl)' \
  > /tmp/runbook-raw.txt

# 或者直接拉那个 ssh session 的全部命令
# 找出 session id(从 atuin search 看)
atuin history list --session
```

**5 秒**就能拿到一份原始命令序列,**剩下的工作是删冗余 + 加注释**——比从记忆里"我先干了啥来着"靠谱 10 倍。

### 9.4 复盘:上周登录过哪些机器

```bash
$ atuin search --after "1 week ago" "^ssh "
2026-05-10 09:32  ~/work          ssh prod-db-1
2026-05-10 14:21  ~/work          ssh staging-payment-api
2026-05-09 11:00  ~/work          ssh bastion-prod
2026-05-09 16:45  ~/work          ssh prod-redis
...
```

**审计自己的 ssh 行为**——**这一周你登过 8 台机器,3 个 prod 机**,**有没有一台是"我也不记得为啥登它"的**?如果有,**安全侧应该有警觉**。

### 9.5 不要做的:把 history dump 给 LLM 训练

近一两年有些"用我的 history 训练个性化 LLM"的项目,**强烈反对**:

```
风险:
  1. history 里 100% 包含敏感信息(即使你过滤,边角料也漏)
  2. 命令背后的"项目目录"暴露公司架构
  3. ssh 目标暴露内网拓扑
  4. timestamps 暴露你的工作节奏(也属于隐私)
  5. 第三方训练:你的命令永久留在外部模型权重里
```

**自己机器上本地跑、本地推理、本地存的随便**——但**把 history dump 上传到任何第三方"AI 服务"**,**都是一次性签字放弃所有运维隐私**。

## 十、fish 用户简短一段

fish 用户来这一节看,**前面九节大部分是 zsh / bash 的话**。

### 10.1 fish history 的设计

fish 的 history 是 **universal variables**(`set -U`)——**fish session 之间自动共享,单机原生**。**不需要 `SHARE_HISTORY` 这种 setopt**,**fish 默认就给你 zsh + SHARE_HISTORY 那种体验**。

存储位置:`~/.local/share/fish/fish_history`(YAML-like 格式,有 timestamp)。

### 10.2 fish 的"上箭头按前缀过滤"是默认

```fish
# 打 git,然后按上箭头
$ git
# 只翻 git 开头的历史(类似 zsh 的 history-search-backward 但开箱)
```

**fish 这一点比 zsh / bash 都贴心**——**默认行为就是按前缀过滤**,新人不用配置。

### 10.3 fish 用户装 atuin

atuin 完全支持 fish:

```bash
brew install atuin
atuin import auto         # 自动从 fish_history 导入
echo 'atuin init fish | source' >> ~/.config/fish/config.fish
```

之后 Ctrl+R 接管,**跟 zsh 一样**。

### 10.4 fzf 在 fish 里

fzf 默认装好后,fish 集成也通过 `fish_user_key_bindings`,**Ctrl+R / Ctrl+T / Alt+C 一样能用**:

```fish
# ~/.config/fish/config.fish
fzf --fish | source       # 较新版 fzf 支持(4.1+)
```

**fish 用户的体验**:**默认 history 已经不错,加 atuin 直接到位**——比 zsh / bash 路径短一截。

## 十一、真实场景训练

讲了这么多工具,**用 4 个真实场景把它们串起来**。

### 11.1 场景 1:上周 staging 那条 kubectl rollout

**情境**:你记得上周三大概下午,在 `~/work/payment` 项目里,跑过一条 `kubectl rollout undo`——但你忘了完整命令是什么。

**不装 atuin 的解法**:

```
按 Ctrl+R,打 kubectl
→ 跳出 50 条 kubectl 开头的,全是 get / describe / logs,翻半天
→ 加 rollout 关键词
→ 跳出 8 条 rollout,看不出哪条是 staging
→ 翻文件,grep .zsh_history
→ 仍然只看到命令文本,没目录、没时间
→ 10 分钟,找不到
```

**装了 atuin**:

```bash
atuin search --cwd ~/work/payment --after "10 days ago" --before "3 days ago" "rollout"
```

**1 秒命中,3 条候选,看时间一眼就知道是哪条**。

### 11.2 场景 2:换新机器,history 没了

**情境**:你买了新 Mac,装好 brew / zsh,**但 `.zsh_history` 是空的**——5 年的肌肉记忆好像全没了。

**不装 atuin 的解法**:

```
从旧机器 scp 一份 ~/.zsh_history 过来
→ 但旧机器是 zsh 5.5,新机器 zsh 5.9,EXTENDED_HISTORY 格式微调
→ 部分行解析失败
→ 而且只能拿到一台机器的 history,
  服务器上跑过的命令完全没有
→ 凑合用,但很多"我记得我在 prod 跑过那条" 找不到
```

**装了 atuin**:

```bash
brew install atuin
atuin login -u myname     # 输 secret key
atuin sync                # 30 秒,所有机器的命令全部同步下来

# 立刻能搜:
atuin search --host laptop-old "deploy"
atuin search --host server-prod "systemctl"
```

**30 秒新机器变成"我的电脑"**。

### 11.3 场景 3:误输入 token,担心明文留 history

**情境**:你刚不小心打了 `export OPENAI_API_KEY=sk-abc...xyz`(忘了空格开头),回车了。

**不装 atuin 的解法**:

```bash
# 1. 立刻清当前 zsh 内存
history -d -1           # zsh / bash 都有的"删最后一条"

# 2. 改 .zsh_history 文件
sed -i.bak '/sk-abc/d' ~/.zsh_history

# 3. 但如果开了 dotfiles git 同步,git 里有
cd ~/.dotfiles && git log -p .zsh_history | grep sk-

# 4. 还得去 OpenAI 后台 revoke 那个 key
```

**装了 atuin**:

```bash
# 同上,先清 zsh 内存
history -d -1

# 清 atuin 数据库 + 同步删除
atuin search --delete "sk-abc"
atuin sync              # 把删除同步到云,所有机器一起删

# .zsh_history 单独清(atuin 不管这个文件)
sed -i.bak '/sk-abc/d' ~/.zsh_history

# 仍然要去 OpenAI 后台 revoke!
# 删 history 不等于"key 没泄露",只是"history 不再是泄露源"
```

**关键认知**:**一旦凭证进了任何持久化存储**(history 文件 / git / 同步服务 / atuin server),**都假设它泄露,立刻 rotate**。删 history 是降低后续扩散概率,不是回滚已发生事件。

### 11.4 场景 4:写 runbook,翻 history 抄上次的步骤

**情境**:你两周前给一个服务做了一次 hotfix,**走的步骤大概是 10 条命令**。今天 PM 要 runbook。

**不装 atuin 的解法**:

```
凭记忆敲一遍,大概 70% 对
→ runbook 给同事跑,跑到第 4 步发现"等等,这条参数不对"
→ runbook 进入修订循环
→ 一周内还出过 1 次小事故,因为"runbook 跟真实步骤差了一步"
```

**装了 atuin**:

```bash
# 找两周前在 hotfix 那个目录跑过的全部命令
atuin search --cwd ~/work/service-x --after "15 days ago" --before "12 days ago" \
  --format '{time} {command}' > /tmp/raw-commands.txt

# 人工筛选,删冗余,加注释,生成 runbook.md
```

**5 分钟出真实命令清单**——**比凭记忆抄准 10 倍**,**且时间戳让你能复原"我先干了啥"**。

## 十二、反对的写法

这一节列**反复见过的 history 反模式**,**或多或少都犯过**:

### 12.1 HISTSIZE=1000

```bash
# 不少人 .bashrc 默认就是
HISTSIZE=1000
HISTFILESIZE=2000
```

**1000 条一周就满**——一个略勤奋的工程师,一天敲 200 条命令很常见,**5 天 history 就开始覆盖**。

**解法**:**改 100000**。磁盘成本几 MB,内存成本可忽略。**没有任何理由不改**。

### 12.2 unset HISTFILE

```bash
# 有人觉得 history 容易泄密,索性
unset HISTFILE
```

**短期看似省心,长期肌肉记忆数据全丢**——你一年后还在凭脑子记"那条 awk 怎么写"。**不可持续**。

**解法**:**保留 HISTFILE,用 HIST_IGNORE_SPACE + atuin history_filter 做敏感过滤**。隐私和可用性不是非此即彼。

### 12.3 抄敏感命令进 history(没用空格开头)

```bash
$ export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
# 没用空格开头,进 history 了
```

**这是最常见的泄密源**——开发机 / 跳板机里 80% 的"我以为安全"的 token 都在 history 里。

**解法**:
1. **打 export 之前,先按一下空格**——养成肌肉记忆
2. **atuin config 里加 history_filter**,正则兜底
3. **开启 atuin secrets_filter = true**,自动识别 sk-/AKIA/ghp_ 等格式

### 12.4 把 .zsh_history 推到公共 git 仓库

```bash
# .dotfiles/.zsh_history → 推到 github.com/me/dotfiles(public)
```

**这是直接的安全事故**——任何浏览你 dotfiles 的人能看到你过去几年所有命令。

**解法**:
1. **.gitignore 加 `.zsh_history` 和 `.bash_history`**
2. **history 同步用 atuin**(加密),**别用 git**
3. **检查现有 dotfiles 仓库**,如果以前推过,**git filter-branch 或 BFG 清理历史**,**rotate 所有可能泄露的凭证**

### 12.5 公共 atuin server 同步含 prod 密令的 history

```bash
# 你在生产机上跑了
atuin sync     # 配的是公共 api.atuin.sh

# 但你 prod 的 history 里有公司架构的细节
# ssh prod-db-internal-aws-region-ap-east-1
# kubectl --context payment-prod-cluster-1 get pods
# 这些命令文本本身泄露架构
```

**虽然加密**,**但你公司合规可能不允许"任何形式的内部架构信息走第三方"**。

**解法**:
1. **自建 atuin server**,部署在公司内网
2. **或者关闭同步**,prod 机器和 laptop 各自一份(损失体验)
3. **或者 history_filter 把架构敏感字段都过滤掉**(太重)

**推荐 1**——5 分钟 docker 起一个,长期收益大。

### 12.6 mcfly + atuin 双开

```zsh
# 错的:
eval "$(mcfly init zsh)"
eval "$(atuin init zsh)"
```

**两者都接管 Ctrl+R**,**装两个互相覆盖**,**通常是后 eval 的赢**。

**解法**:**二选一,新人无脑 atuin**。

### 12.7 装了 fzf 然后又开 zsh 的 history-incremental-search-backward

```zsh
eval "$(/usr/local/opt/fzf/install)"
# 然后又:
bindkey '^R' history-incremental-search-backward
```

**第二行把 fzf 接管 Ctrl+R 给覆盖了**——你以为装了 fzf,**实际上每次 Ctrl+R 还是 zsh 原生的反向搜**,**奇怪为什么没有 UI**。

**解法**:**check `bindkey '^R'` 当前绑定的是什么**:

```bash
$ bindkey '^R'
"^R" fzf-history-widget                # 装了 fzf 之后应该是这个
```

不是这个就说明被覆盖了,**移除冲突绑定**。

### 12.8 一开始就装全套,结果不知道哪个工具在干嘛

```bash
brew install atuin mcfly fzf
# 各装一份配置,Ctrl+R 谁接管的不清楚
# 然后开始抱怨"Ctrl+R 怎么变了又变"
```

**新人最常见的反模式**——**装一个用一个**,**确定它解决了什么问题再加下一层**。

**解法**:**三层升级按顺序加,装一层用 2 周**:

```
Week 1-2:  把 zsh setopt 11 行加上,体感 Ctrl+R 多了一倍准确率
Week 3-4:  装 fzf 接管 Ctrl+R,体感 Ctrl+R 多了 5 倍效率
Week 5+:   装 atuin,体感同步 + 多机协作 + cwd 过滤的爆炸式收益
```

**每加一层,你能讲清楚它解决了什么、上一层没解决什么**。**装全套但讲不清的人,半年后会因为某个工具升级翻车,且不知道哪个 hook 出了问题**。

## 十四、下一篇预告

下一篇:**`10-文本三剑客.md`**——这一篇是 shell 选型(05)/ 配置(06-09)的收尾,**从下一篇开始进入 CLI 工具链层**。

**grep / sed / awk 是 1970 年代的 Unix 文本三剑客**——**到了 2026 还要不要学**?这是个真问题。一方面 `rg` 比 grep 快 10 倍、`sd` 比 sed 易用 10 倍、`miller / awk-go` 给结构化数据更好的工具;**另一方面这三个工具是 Unix 哲学最纯粹的体现**,**SRE / DBA / 系统管理员的世界还在用它们**——你 ssh 到一台没装现代工具的生产机,**只剩 grep / sed / awk 三个能用**。

下一篇拆开来讲:

- **三剑客各自的心智模型**:grep 是过滤 / sed 是流编辑 / awk 是迷你 DSL
- **2026 还该不该学**:学心智、不背 flag——日常用现代替代品,但读得懂祖传脚本
- **何时该用现代替代品**:rg 替 grep / sd 替 sed / miller 替 awk(结构化) / `awk` 还是 awk(单行 DSL)
- **三剑客的死角**:不能处理多行 JSON / 编码问题 / Unicode 边界 / 二进制——这些是现代替代品的入口
- **真实场景 head-to-head 对比**:同一道题用 grep vs rg、用 sed vs sd、用 awk vs miller 各跑一遍

**05-09 五篇讲的是"在 shell 里工作"**,**10-15 六篇讲的是"shell 调用什么工具"**——**两层完全独立**,**但只有把第二层也立起来,你的终端才真的是"工程师的第二大脑"**。
