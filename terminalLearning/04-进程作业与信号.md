# 进程、作业与信号:Ctrl-C 到底发给了谁

你写了个死循环,按下 Ctrl-C,程序退出了。你以为是「Ctrl-C = 让程序退出」,这是最危险的肌肉记忆——**因为有一天你会按下 Ctrl-C 而程序不响应**,而你完全不知道是哪一层出了问题。真相是:你按下 Ctrl 和 C 这两个键,**到程序真的退出之间至少隔了 5 层**——终端模拟器读到按键 → pty 终端驱动检测到「intr 字符」→ 内核产生 SIGINT 信号 → 把信号投递到「前台进程组」里的每个进程 → 每个进程的信号 handler 决定是不是真的退出。这 5 层任何一层处理不一样,Ctrl-C 的结果就不一样:vim 不退、less 不退、Python REPL 抛 KeyboardInterrupt、shell 脚本只断当前语句、ssh 把信号转发到远端、tmux 又转一道——**Ctrl-C 不是一个动作,是一条链路**。这一篇就是把这条链路彻底拆开。

> 一句话先记住:**一个进程能不能被你 Ctrl-C / Ctrl-Z / 关掉终端窗口杀死,取决于它的「进程组 + 会话 + 控制终端 + 信号处理」四件套——不是你的「意图」,是这四件套的状态**。同一条命令在前台和后台、在 tmux 里和裸 shell 里、在 ssh 里和本机里、在 docker 里和宿主机里,Ctrl-C 的结果可以完全不同——因为这四件套被改了。

## 一、进程的家族关系:五个名词必须先分清

讲信号之前必须把「进程的家族关系」讲清——**否则 SIGHUP 怎么传播、Ctrl-C 发给谁、nohup 挡的是哪一刀,全是糊的**。Unix 进程模型有五层身份,从小到大:

```
                      ┌────────────────────────────────────────┐
                      │              Session (SID)             │
                      │   会话 = 一组进程组的集合              │
                      │   会话有一个会话首进程(SID = PID)    │
                      │   会话可以有一个「控制终端」(/dev/pts/X)│
                      │                                        │
                      │   ┌──────────────────┐  ┌───────────┐ │
                      │   │ Foreground PGID  │  │ Bg PGID 1 │ │
                      │   │ (前台进程组)     │  │           │ │
                      │   │ ┌─────┐ ┌─────┐  │  │ ┌─────┐  │ │
                      │   │ │ PID │ │ PID │  │  │ │ PID │  │ │
                      │   │ └─────┘ └─────┘  │  │ └─────┘  │ │
                      │   └──────────────────┘  └───────────┘ │
                      │                                        │
                      │   ┌───────────┐                       │
                      │   │ Bg PGID 2 │                       │
                      │   │ ┌─────┐   │                       │
                      │   │ │ PID │   │                       │
                      │   │ └─────┘   │                       │
                      │   └───────────┘                       │
                      └────────────────────────────────────────┘

   每个控制终端只有一个「前台进程组」,其他都是「后台进程组」
   Ctrl-C 的信号只发给「前台进程组里的所有进程」
   SIGHUP 在控制终端消失时发给「会话首进程」(通常就是 shell)
```

五层名词逐个讲:

1. **进程 (process)**——一个 PID,内核里一个 `task_struct`。最熟悉的概念。
2. **进程组 (process group)**——一组相关进程的集合,有一个 **PGID**(进程组 ID)。PGID 等于组长(group leader)的 PID。`fork` 出来的子进程默认继承父进程的 PGID,**但 shell 启动一个新作业时会主动把它放进一个新进程组**(下一节细讲)。进程组的存在是为了让你「**给一组进程同时发信号**」——`kill -SIGTERM -PGID` 给整个组发,Ctrl-C 给前台组的每个进程发。
3. **会话 (session)**——一组进程组的集合,有一个 **SID**(会话 ID,等于会话首进程的 PID)。一个 shell 进程通常是一个会话首进程。
4. **控制终端 (controlling tty)**——会话**可以有一个**控制终端(也可以没有,比如 daemon)。控制终端绑定到一个 pty(伪终端设备,比如 `/dev/pts/3`)。控制终端是 SIGHUP / Ctrl-C / Ctrl-Z 信号的「发源地」——按键先到 pty,内核再判断该发给哪个会话的前台进程组。
5. **前台 vs 后台进程组**——一个控制终端,**任意时刻只有一个前台进程组**(`tcgetpgrp(fd)` 拿到的就是它),其他全是后台。**只有前台进程组能读 tty 输入**,后台进程读 tty 会被发 SIGTTIN 直接停掉。

现在用 `ps` 看看你机器上的真实情况:

```bash
$ ps -j -o pid,pgid,sid,tty,stat,command
  PID  PGID   SID TT       STAT COMMAND
12345 12345 12345 pts/3    Ss   -zsh                # 会话首 + 进程组长
12400 12400 12345 pts/3    S+   sleep 100           # 新进程组,在前台("+"标志)
12410 12410 12345 pts/3    S    sleep 200           # 新进程组,在后台(没有 "+")

# 解读:
# - 三个进程都属于 SID = 12345 的会话(都是 zsh 的子孙)
# - zsh 自己 PID = PGID = SID,会话首进程
# - 两个 sleep 各自独立成了进程组(PGID = PID),sleep 100 是前台
# - STAT 列的 "+" 表示前台,"s" 表示会话首,"S" 表示可中断睡眠
```

**核心结论**:**进程组和会话不是「自动形成」的,是 shell 主动 `setpgid` / `setsid` 调出来的**——这两个系统调用是这一篇的隐形主角。`nohup` `disown` `setsid` 的差别,本质上就是它们对这两个系统调用动手的位置不同。

## 二、shell 启动一个命令时发生了什么

你敲 `sleep 100` 回车,shell 不是「直接执行」——它做了一套精心编排的动作:

```
shell (PID 12345, PGID 12345, SID 12345, 前台 = 12345)
   │
   │  1. fork()   ─────────────────▶  child (PID 12400)
   │                                    继承 PGID = 12345 (此刻是 shell 的组)
   │
   │  2. 父子两边都调用 setpgid(child_pid, child_pid)
   │     把 child 放进一个新进程组,PGID = 12400
   │     (父子都调一遍是为了避免竞态,谁先生效都行)
   │
   │  3. 如果是前台命令:
   │     父调 tcsetpgrp(tty_fd, 12400)
   │     把控制终端的前台进程组设为 child 的组
   │
   │  4. child exec("sleep", "100")
   │     PID 不变,代码段被替换成 sleep
   │
   │  5. 父调 waitpid(12400, ...)
   │     等待子进程结束;在等的时候,前台是 child 的组
   │     —— 这就是为什么 sleep 跑的时候你不能在 shell 里继续敲命令
   │
   │  6. child 结束,父被唤醒
   │     父调 tcsetpgrp(tty_fd, shell_pgid)
   │     把前台重新设回自己,继续读输入
```

如果你加了 `&`(后台),改的是第 3 步——**父不把前台让给子**,自己继续读输入;但第 2 步仍然把子放进新进程组(后台进程组)。这就是为什么 `sleep 100 &` 之后你立刻能继续敲命令。

实战看一眼:

```bash
$ sleep 100 &
[1] 12400
$ sleep 200 &
[1]  12410
$ jobs
[1]  - running    sleep 100
[2]  + running    sleep 200
$ ps -j
  PID  PGID   SID TT       STAT COMMAND
12345 12345 12345 pts/3    Ss   -zsh
12400 12400 12345 pts/3    S    sleep 100
12410 12410 12345 pts/3    S    sleep 200

# 看到了吗?
# - 两个 sleep 的 PGID = 自己的 PID,各自独立进程组
# - 都属于 zsh 的会话(SID = 12345)
# - 都没有 "+" 标志,因为没有任何一个在前台
# - 此时前台是谁?是 zsh 自己,它在等你下一条命令
$ ps -o tpgid -p $$       # tpgid = 控制终端的前台进程组
TPGID
12345                     # 确认前台是 zsh 自己
```

**为什么要把每个作业放进独立进程组?** 因为这样 shell 能用「整组发信号」精确控制单个作业——`kill %1` 实际是 `kill -SIGTERM -12400`,给 PGID = 12400 的整组发,把 sleep 和它可能的所有子孙都收一遍。如果不分组,你 `kill %1` 就只能杀单个 PID,sleep 的子孙就漏了。

## 三、信号到底是什么

**信号 = 内核给进程的「软件中断」**。本质上就是「内核往进程的 `task_struct` 里塞一个标记,等这个进程下次回到用户态时强制跳到信号处理函数」。

对工程师来说,你只需要记住:

- 信号是**异步**的——你不知道它什么时候到,但它一来程序就会暂停手头的事先去处理它
- 大部分信号你可以**捕获**(`signal` / `sigaction`)、**忽略**(`SIG_IGN`)或**用默认行为**(`SIG_DFL`)
- 有两个信号**不能拦截**——SIGKILL(9)和 SIGSTOP(19)。这是内核留的最后底牌
- 信号没有「数据」(标准信号),只有信号编号本身;`SIGUSR1` 来了就来了,不带任何额外信息(实时信号 SIGRTMIN-SIGRTMAX 才有 siginfo,日常用不到)

`kill -l` 列出当前系统的所有信号:

```bash
$ kill -l
 1) SIGHUP      2) SIGINT      3) SIGQUIT     4) SIGILL      5) SIGTRAP
 6) SIGABRT     7) SIGBUS      8) SIGFPE      9) SIGKILL    10) SIGUSR1
11) SIGSEGV    12) SIGUSR2    13) SIGPIPE    14) SIGALRM    15) SIGTERM
16) SIGSTKFLT  17) SIGCHLD    18) SIGCONT    19) SIGSTOP    20) SIGTSTP
21) SIGTTIN    22) SIGTTOU    23) SIGURG     24) SIGXCPU    25) SIGXFSZ
26) SIGVTALRM  27) SIGPROF    28) SIGWINCH   29) SIGIO      30) SIGPWR
31) SIGSYS     ...
```

**工程师每天会碰到的就 10 来个**,死磕这张表:

| 信号 | 编号 | 默认行为 | 谁产生 | 用途 / 真实场景 |
| --- | --- | --- | --- | --- |
| **SIGHUP** | 1 | 终止 | 控制终端消失时内核发给会话首 | 「房子塌了你跑不跑」——后面整节讲。也常被守护进程当「重载配置」信号(nginx -s reload) |
| **SIGINT** | 2 | 终止 | 你按 Ctrl-C,pty 驱动产生 | 礼貌「请退出」。可拦截,程序可以 trap 后只做一次清理(比如 Python 的 KeyboardInterrupt) |
| **SIGQUIT** | 3 | 终止 + core dump | 你按 Ctrl-\\,pty 驱动产生 | 升级版退出,带 core dump。Java 进程收到会打全线程 stack(jstack 替身) |
| **SIGABRT** | 6 | 终止 + core | `abort()` / `assert()` | C/C++ 程序的「我自杀」。glibc 检测到内存破坏也会发这个 |
| **SIGKILL** | 9 | 终止(不可拦截) | 你 `kill -9` | **核武器**。进程一定死,但来不及清理(临时文件不删、buffer 不 flush) |
| **SIGUSR1** | 10 | 终止 | 自定义 | 服务常用作「重载配置」「dump 状态」。nginx / golang pprof |
| **SIGSEGV** | 11 | 终止 + core | 内核(非法内存访问) | C/C++ 的段错误。Go 也会触发但被 runtime 接走变 panic |
| **SIGUSR2** | 12 | 终止 | 自定义 | 另一个自定义信号,常用来「切换 log 文件」「触发 GC」 |
| **SIGPIPE** | 13 | 终止 | 内核(写已关闭的管道) | `yes \| head` 优雅退出靠它——head 读够 10 行关掉读端,yes 下次 write 收到 SIGPIPE 退出 |
| **SIGTERM** | 15 | 终止 | `kill <pid>` 默认 | 礼貌「请退出」。生产环境**首选**,给程序留清理时间。`docker stop` / k8s 优雅下线发的就是它 |
| **SIGCHLD** | 17 | 忽略 | 子进程退出时内核发给父进程 | 父进程靠它知道「该 wait 子进程了」。shell 的 jobs 状态变更靠它 |
| **SIGCONT** | 18 | 继续 | `fg` / `bg` / `kill -CONT` | 让 stopped 进程恢复执行 |
| **SIGSTOP** | 19 | 暂停(不可拦截) | `kill -STOP` | 强制暂停。注意这不是 Ctrl-Z(那个是 SIGTSTP) |
| **SIGTSTP** | 20 | 暂停 | 你按 Ctrl-Z,pty 驱动产生 | **可拦截的暂停**。vim 收到会自己存盘+退到 shell |
| **SIGTTIN** | 21 | 暂停 | 后台进程读 tty | 后台进程不准抢输入。`cat &` 然后回车给它就触发 |
| **SIGTTOU** | 22 | 暂停 | 后台进程写 tty(若 tostop 开启) | 后台进程不准抢输出,但默认 tostop 是关的所以一般不触发 |
| **SIGWINCH** | 28 | 忽略 | 窗口大小变化 | 终端窗口拉大,内核通知给前台进程;vim / tmux 靠它重排版面 |

记不住没关系——你只需要记住下面这条**最重要的语义分组**:

```
礼貌退出系列(都可以被 trap,程序可以清理后再退):
   SIGINT  (2)  — Ctrl-C
   SIGTERM (15) — kill 默认
   SIGHUP  (1)  — 终端断了

强制系列(不可拦截):
   SIGKILL (9)  — 死
   SIGSTOP (19) — 停

作业控制系列:
   SIGTSTP (20) — Ctrl-Z 暂停(可拦)
   SIGCONT (18) — 继续
   SIGTTIN/TTOU — 后台进程瞎抢 tty

自定义信号:
   SIGUSR1 / SIGUSR2 — 你自己定义,常用作「重载配置」
```

`kill <pid>` 不加参数发的是 **SIGTERM**——这就是你日常 95% 的场景。**SIGKILL(`kill -9`)是兜底,不是默认**——见后面「常见误解」。

## 四、Ctrl-C / Ctrl-Z / Ctrl-\\ 真相

很多人以为「Ctrl-C 是 shell 接到的快捷键」——**错**。shell 完全不参与。

完整链路(必画图):

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│   你按 Ctrl 和 C  ─────▶  终端模拟器(iTerm2 / Alacritty / kitty)   │
│                                  │                                  │
│                                  │  把 0x03 字节写入 pty master 端   │
│                                  ▼                                  │
│                          内核 pty 子系统                            │
│                          (线路规程 line discipline)                 │
│                                  │                                  │
│                                  │  isig 开启?ICANON 模式?         │
│                                  │  检测到 0x03 = intr 字符           │
│                                  ▼                                  │
│                          内核产生 SIGINT,投递给前台进程组          │
│                                  │                                  │
│                                  │  tcgetpgrp(tty) → 拿到前台 PGID   │
│                                  ▼                                  │
│                          组里每个进程的 task_struct 加上 SIGINT 标记 │
│                                  │                                  │
│                                  ▼                                  │
│                          进程下次进入用户态时检查信号                │
│                          → 跳到 handler(默认 = exit)               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**5 个关键事实**:

1. **shell 不接 Ctrl-C**。是 pty 终端驱动检测到 0x03 字节产生 SIGINT
2. **发给的不是 shell,是「前台进程组」**。shell 此刻在 waitpid,不是前台
3. **发给的是「组里每个进程」**——比如 `find / | grep foo`,find 和 grep 都收到 SIGINT
4. **三个 Ctrl 组合键对应三个不同信号**:
   - Ctrl-C → SIGINT(默认退出,可拦截)
   - Ctrl-Z → SIGTSTP(默认暂停,可拦截)
   - Ctrl-\\ → SIGQUIT(默认退出 + core dump,可拦截但很少拦)
5. **「intr 字符」是可配置的**——`stty -a` 看,`stty intr ^X` 改成 Ctrl-X(没人改,但可以)

实战看一眼 stty:

```bash
$ stty -a
speed 38400 baud; rows 48; columns 180; line = 0;
intr = ^C; quit = ^\; erase = ^?; kill = ^U; eof = ^D; eol = <undef>;
eol2 = <undef>; swtch = <undef>; start = ^Q; stop = ^S; susp = ^Z; rprnt = ^R;
werase = ^W; lnext = ^V; discard = ^O; min = 1; time = 0;
-parenb -parodd -cmspar cs8 -hupcl -cstopb cread -clocal -crtscts
-ignbrk -brkint -ignpar -parmrk -inpck -istrip -inlcr -igncr icrnl ixon -ixoff
-iuclc -ixany -imaxbel iutf8
opost -olcuc -ocrnl onlcr -onocr -onlret -ofill -ofdel nl0 cr0 tab0 bs0 vt0 ff0
isig icanon iexten echo echoe echok -echonl -noflsh -xcase -tostop -echoprt
echoctl echoke -flusho -extproc

# 关键字段:
# intr = ^C   → Ctrl-C 触发 SIGINT
# susp = ^Z   → Ctrl-Z 触发 SIGTSTP
# quit = ^\   → Ctrl-\ 触发 SIGQUIT
# isig        → 是否开启「按键转信号」机制(开)
# icanon      → 行缓冲模式(开)
# echo        → 输入回显(开)
```

### 4.1 为什么 vim / less / fzf 不被 Ctrl-C 杀

简单:**它们在 `tcsetattr` 把 isig 关了**——告诉内核「这些按键你别帮我转信号了,原始字节给我自己看」。这叫 **raw 模式**(或更准确叫 cbreak / non-canonical 模式)。

```c
// vim 启动时大致干的事
struct termios raw;
tcgetattr(STDIN_FILENO, &raw);
raw.c_lflag &= ~(ICANON | ECHO | ISIG);   // ← 关掉 ISIG
tcsetattr(STDIN_FILENO, TCSANOW, &raw);
```

所以 vim 收到 Ctrl-C 时**只是收到三个字节**(实际就一个 0x03),它的输入处理代码看到这个字节,完全可以解释成「中断当前命令」或者「干脆忽略」——内核不再插手。

退出 vim 时它会把 termios 改回去。**如果 vim 崩溃来不及恢复**——你就会遇到那个经典惨案:终端坏了,看不见输入,什么都不响应。救命招:`reset` 命令一把 termios 拉回 sane 状态。

### 4.2 Ctrl-C 杀不掉怎么办

升级链路:

```
Ctrl-C  → SIGINT  (可拦截,程序可能忽略)
   ↓ 失败
Ctrl-\  → SIGQUIT (可拦截,但大部分程序不拦,而且带 core dump)
   ↓ 失败
kill <pid>      → SIGTERM (可拦截)
   ↓ 失败
kill -9 <pid>   → SIGKILL (不可拦截,核武器)
```

**Ctrl-\\ 是被严重低估的快捷键**。它发的是 SIGQUIT,默认行为是退出 + core dump,**而且大部分程序不会拦截 SIGQUIT**(拦 SIGINT 是为了 Ctrl-C 优雅退出,拦 SIGQUIT 没有动机)。所以 Ctrl-C 杀不掉的程序,Ctrl-\\ 90% 能解决,而且不需要你查 PID。

但 Ctrl-\\ 也救不了:程序进了 `D` 状态(uninterruptible sleep,通常是卡在 IO 系统调用里)。这时连 SIGKILL 都没用——**`D` 状态进程要等内核 IO 路径返回**。诊断要用 `cat /proc/<pid>/stack` 看内核栈,然后修底层问题(磁盘卡死、NFS 不响应、bug driver 等)。

## 五、作业控制 (job control) 速查

**作业 (job) 是 shell 的概念,不是内核的概念**——内核只知道进程组,shell 在自己的本地数据结构里维护「我启动的子进程组列表」,给你编号 %1 %2 用。

最小指令集:

```bash
$ sleep 100 &            # 启动后台作业,立刻返回
[1] 12400                #   [1] 是作业号,12400 是 PID(= PGID)

$ vim file.txt           # 启动前台作业
# 在 vim 里按 Ctrl-Z
[2]+ Stopped     vim file.txt

$ jobs                   # 列出本 shell 的作业
[1]-  Running    sleep 100 &
[2]+  Stopped    vim file.txt

$ fg %1                  # 把 1 号作业拉回前台
sleep 100
# 这时你按 Ctrl-C 才能杀它,因为它现在是前台

$ bg %2                  # 把 2 号作业丢后台继续跑(发 SIGCONT)
[2]+ vim file.txt &
# 但 vim 在后台读 tty 会被 SIGTTIN 立刻停掉,所以这例子不好,换个 sleep

$ kill %1                # 杀作业(发 SIGTERM)
[1]+ Terminated    sleep 100
```

### 5.1 作业状态机(必画图)

```
              fg / wait
   ┌────────────────────────────────┐
   │                                ▼
┌────────┐  bg / 启动加 &  ┌────────┐
│Running │ ───────────────▶│Running │
│前台    │                  │后台    │
└────────┘ ◀───────────────┘
   │                                │
   │ Ctrl-Z                         │ kill -STOP
   │ → SIGTSTP                      │ → SIGSTOP
   ▼                                ▼
┌────────┐                  ┌────────┐
│Stopped │ ──── fg ────────▶│Stopped │
│        │                  │        │
└────────┘ ◀── bg ─────────┘
   │                                │
   │  SIGCONT → 回到 Running         │
   │                                │
   ▼                                ▼
┌──────────────────────────────────────┐
│            Done / Killed             │
│ 退出(正常 0 / 非零)/ 被信号杀掉    │
└──────────────────────────────────────┘
```

四种状态:**Running 前台 / Running 后台 / Stopped / Done(Terminated)**。fg / bg / Ctrl-Z / kill 在这四个状态间切。

### 5.2 `jobs` 的边界

**这是 shell 里被严重误解的命令**:

- `jobs` **只看本 shell 启动的作业**——不是「本机所有进程」
- 不看子孙的孙——比如你启动一个脚本,脚本里再启动后台任务,父 shell 的 `jobs` 看不到孙作业
- 不看其他终端、其他 shell 启动的作业——这些是别的 shell 的作业表
- shell 退出后作业表就没了(虽然进程可能还活着,但你「记账」的本子丢了)

要看全机器的进程,用 `ps -ef` / `ps aux` / `pgrep` / `htop`,**不用 `jobs`**。

### 5.3 真实演示:从前台到后台到杀掉

```bash
# 一段你可以复制粘贴的演示
$ sleep 1000
^Z
[1]+  Stopped                 sleep 1000
$ bg %1
[1]+ sleep 1000 &
$ jobs
[1]+  Running                 sleep 1000 &
$ ps -j
  PID  PGID   SID TT       STAT COMMAND
12345 12345 12345 pts/3    Ss   -zsh
12500 12500 12345 pts/3    S    sleep 1000   ← 后台,独立进程组
$ kill %1
$ jobs
[1]+  Terminated              sleep 1000
$ jobs    # 再列一次就空了
$
```

注意 `kill %1` 第一次执行后,作业还显示在 `jobs` 里(状态 Terminated)——这是 shell 给你看一眼;下一次 `jobs` 就清了。bash 和 zsh 行为略不同,但意思一致。

## 六、关掉终端窗口,谁会死,谁不会

这是这一篇最实用的一节。

### 6.1 关闭窗口时到底发生了什么

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   你点终端模拟器的 "x" 按钮                                │
│            │                                               │
│            ▼                                               │
│   终端模拟器关闭 pty master 端(它持有 master fd)         │
│            │                                               │
│            ▼                                               │
│   内核检测到 master 端被关:slave 端读到 HUP 状态          │
│   → 内核给「会话首进程」发 SIGHUP                          │
│            │                                               │
│            ▼                                               │
│   会话首进程通常是 shell(zsh / bash)                    │
│   shell 收到 SIGHUP,默认行为:                            │
│     ① 给所有作业再发一次 SIGHUP(bash / zsh 默认都这样)   │
│     ② 然后自己退出                                         │
│            │                                               │
│            ▼                                               │
│   每个作业的进程组里的所有进程都收到 SIGHUP                │
│   → 默认行为 = 退出                                        │
│            │                                               │
│            ▼                                               │
│   所有后台作业死亡,你 ssh 上跑的 train.py 也死了          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**核心问题**:shell 转发 SIGHUP 给所有作业,是默认行为。**你必须让你的进程「逃出 shell 的作业表」或「忽略 SIGHUP」,才能在终端关闭后继续活**。

### 6.2 救命的三种方式

```bash
# 方式 1:nohup —— 启动时让进程忽略 SIGHUP
$ nohup python train.py &
nohup: ignoring input and appending output to 'nohup.out'
[1] 13000
# nohup 做了三件事:
#   ① signal(SIGHUP, SIG_IGN) —— 让子进程忽略 SIGHUP
#   ② 重定向 stdin 到 /dev/null —— 因为关了终端 stdin 也废了
#   ③ 把 stdout/stderr 重定向到 nohup.out(若是 tty)
# 但!nohup 不改变进程组、不脱离会话,所以 jobs 里它还在

# 方式 2:disown —— 启动后从 shell 的作业表里删除
$ python train.py &
[1] 13001
$ disown %1
$ jobs       # 空了
# disown 做的事:把作业从 shell 的内部作业表里 remove
# 副作用:shell 退出时不再给它发 SIGHUP
# 但!如果终端模拟器关闭→pty slave 端 EOF,程序 read stdin 还是会读到 EOF

# 方式 3:setsid —— 让进程自己开新会话
$ setsid python train.py &
$ ps -j
  PID  PGID   SID TT       STAT COMMAND
13002 13002 13002 ?        Ss   python train.py
# setsid 调用 setsid(2) 系统调用,让 python 自己成为新会话首
# SID = 自己的 PID,完全脱离原来的会话
# 控制终端也丢了(TT 列是 "?"),所以这进程之后跟 tty 无关
```

### 6.3 四种启动方式对比表

|  | 进程组 | 会话 | 控制终端 | SIGHUP 处理 | 关掉终端后果 |
| --- | --- | --- | --- | --- | --- |
| `cmd` (前台) | 新 PGID | 沿用 shell 的 | 有 | 默认(退出) | 死 |
| `cmd &` | 新 PGID | 沿用 shell 的 | 有 | 默认(退出) | shell 转发 SIGHUP,**死** |
| `nohup cmd &` | 新 PGID | 沿用 shell 的 | 有 | **被设为 IGN(忽略)** | shell 还会发,但被忽略,**活** |
| `cmd & disown` | 新 PGID | 沿用 shell 的 | 有 | 默认 | shell 不发(作业表里没了),**活** |
| `setsid cmd` | **新 PGID + 新 SID** | **新会话** | **无** | 默认 | 跟原会话彻底无关,**活** |
| `tmux new -d 'cmd'` | tmux server 的子进程 | tmux server 的会话 | tmux 的 pty | 默认 | tmux server 是 daemon,不死,**活** |

**实践推荐**:

- 临时跑个长任务怕断网:`nohup cmd &` 是最快的招
- 已经启动了想保住:`disown %1` 救命
- 写 daemon 程序:`setsid` + 双 fork(经典 daemonize)
- 真的认真长跑:用 **tmux** 或 **screen**(16 篇细讲),比上面三个都靠谱——因为 tmux server 本身就是 daemon,你的进程跑在 tmux 的 pty 里,断网只是断了你 attach 的那条 ssh,程序在 tmux server 里继续活

### 6.4 nohup 不是万能挡箭牌

误解:「我用 nohup 了所以怎么都死不了」——错。**nohup 只忽略 SIGHUP**。

- Ctrl-C 杀的是 SIGINT,nohup 不挡 → 死
- `kill <pid>` 发的是 SIGTERM,nohup 不挡 → 死
- `kill -9` 发的是 SIGKILL,nohup 不挡 → 死
- 服务器 OOM Killer 给你发 SIGKILL → 死
- 进程自己 SEGV → 死

nohup 唯一的承诺:**控制终端消失时给你发的 SIGHUP,我帮你接住**。**就这一句**。

## 七、真实场景训练

### 7.1 场景 1:ssh 进服务器跑训练,网络断了,训练挂了

**现象**:你 ssh 进 GPU 机器,跑了 `python train.py`,十几分钟后家里网断了,重连上 `ps` 看,python 进程没了。

**链路**:

```
ssh 客户端断 → ssh server 端的连接进程死
                 → ssh server 持有的 pty master 端被关
                 → pty slave 端 HUP → 内核给会话首发 SIGHUP
                 → 会话首是远端的 login shell(bash / zsh)
                 → shell 转发 SIGHUP 给所有作业
                 → 你的 python train.py 收到 SIGHUP → 默认行为 = 死
```

**救法**(按推荐度排序):

1. **下次用 tmux**:`ssh server`,然后 `tmux new -s train`,在 tmux 里跑训练。断网只是断 ssh,tmux server 继续活,你下次 `ssh server` + `tmux attach -t train` 就回到原样。**这是工程师标准答案,16 篇会展开**
2. **已经在跑了但还没断网,救一下**:Ctrl-Z 暂停 → `bg` → `disown %1` 三连——把它从 shell 作业表里删,断网时 shell 不会转发 SIGHUP
3. **下次启动时用 `nohup ... &` 或 `setsid`**——能挡 SIGHUP,但 stdout 输出要重定向到文件,否则丢了
4. **再下次,认真长跑用 systemd 或 nomad**——把训练当 service 跑,跟 ssh 完全解耦(33 / 34 篇)

### 7.2 场景 2:Ctrl-C 杀不掉一个进程

**现象**:你跑了个脚本,按 Ctrl-C 屏幕没反应,进程还在跑。

**可能性**(从高到低):

1. **程序 trap 了 SIGINT 但没正确退出**——比如 Python 脚本里 `try / except KeyboardInterrupt: pass`,你按 Ctrl-C 它接住后什么都不做
2. **程序在 raw 模式(isig 关了)**——vim / less / fzf / docker exec -it 进容器后,Ctrl-C 不再转 SIGINT,而是变成原始字节给程序自己解释
3. **当前前台进程组不是你以为的那个**——`ps -j` 看 STAT 里的 "+" 是谁,Ctrl-C 只发给那个组
4. **程序在 D 状态(uninterruptible sleep)**——通常卡 IO,SIGKILL 都没用,只能等内核

**升级链**:

```bash
^C            # SIGINT,可能被 trap
^\            # SIGQUIT,默认带 core dump,大部分程序不拦
# 不行就开新窗口:
kill <pid>    # SIGTERM,可能被 trap
kill -9 <pid> # SIGKILL,核武器,99% 解决
# 如果 SIGKILL 都没用 → D 状态:
cat /proc/<pid>/stack    # 看内核栈,定位卡在哪
# 通常要重启或修底层(磁盘 / NFS / driver)
```

**有用的诊断技巧**:

```bash
# 看进程的信号配置(哪些被 block / ignore / catch):
$ cat /proc/<pid>/status | grep -E 'Sig(Cgt|Ign|Blk|Pnd)'
SigPnd: 0000000000000000   # 待处理
SigBlk: 0000000000000000   # 被屏蔽
SigIgn: 0000000000001000   # 被忽略(bit 13 = SIGPIPE)
SigCgt: 0000000180000002   # 被捕获(bit 2 = SIGINT 被 trap)

# 16 进制看哪些 bit set,对照信号编号(bit n = 信号 n+1)
# 0x180000002 → bit 1, 32, 33 → 程序 trap 了 SIGINT 和两个实时信号
```

### 7.3 场景 3:管道里中间死了一个,其他怎么办

`cmd1 | cmd2 | cmd3`,假设 cmd2 死了——cmd1 和 cmd3 怎么办?

```
cmd1 → write → pipe1 → read → cmd2 → write → pipe2 → read → cmd3

cmd2 死了:
   ① pipe1 的读端被关 → cmd1 下次 write 收到 SIGPIPE → 默认退出
   ② pipe2 的写端被关 → cmd3 下次 read 返回 EOF → 自然结束
```

**两个方向不一样**:

- 写端(上游)收到 **SIGPIPE**——这是个**致命信号**(默认行为退出),但程序可以 trap / ignore
- 读端(下游)读到 **EOF**——这不是信号,是 read 返回 0,程序自己看

这就是为什么 `yes | head -n 10` 优雅退出——head 读 10 行 close 读端,yes 下次 write 收到 SIGPIPE,死。

**反面教材**:Python 默认不处理 SIGPIPE,改成 IGN——所以 `python -c 'while True: print("a")' | head` 会刷一堆 `BrokenPipeError`。生产 Python 脚本要在开头加:

```python
import signal
signal.signal(signal.SIGPIPE, signal.SIG_DFL)
```

让 SIGPIPE 走默认行为(退出),管道才优雅。

### 7.4 场景 4:docker run -it,Ctrl-C 是退出容器还是杀容器

**取决于三件事**:`-it` 在不在,容器里 PID 1 是谁,容器有没有 `--init`。

```bash
# A. docker run -it ubuntu bash
# Ctrl-C 发给容器内前台进程组,通常就是 bash
# bash 收到 SIGINT 默认行为是「中断当前命令」,不退出
# 结论:Ctrl-C 没用,要 exit 或 Ctrl-D 才退出

# B. docker run -it ubuntu sleep 1000
# 容器 PID 1 = sleep,前台
# Ctrl-C → SIGINT → sleep 退出 → 容器退出

# C. docker run ubuntu sleep 1000(没 -it)
# 没附加 tty,Ctrl-C 在你这个 docker 客户端,不传到容器
# 要 docker kill <container> 才能停

# D. docker run --init -it ubuntu sleep 1000
# --init 让容器 PID 1 是 tini(一个轻量 init)
# tini 接信号,转发给 sleep,然后 reap 僵尸进程
# 适合「容器里跑的不是真正的 PID 1 设计」的程序
```

**关键陷阱**:很多程序当 PID 1 跑时,**默认不接收任何信号**——因为 Linux 内核对 PID 1 有保护(防止 init 被自己 kill 把系统弄崩)。除非程序显式 `signal(SIGTERM, handler)`,否则 `docker stop` 发的 SIGTERM 它收不到,等 10 秒超时被 SIGKILL。

**实用建议**:

- 容器里跑的二进制,如果是 shell 启动多进程的 wrapper,**用 `--init` 或 `tini` / `dumb-init`** 当 PID 1
- 容器里跑 Java / Python 等运行时,确认它们对 SIGTERM 的处理(JVM 默认会,Python 也会)
- `docker stop -t 30` 让 SIGTERM 后等 30 秒再 SIGKILL,给程序留清理时间

## 八、trap:在 shell 脚本里捕获信号

shell 脚本是「中长寿命」的进程,**写 shell 脚本不 trap 信号 = 不写清理逻辑**。

### 8.1 最小例子

```bash
#!/usr/bin/env bash
set -euo pipefail

trap 'echo "脚本退出,清理 tmp"; rm -f /tmp/myapp.lock' EXIT
trap 'echo "被中断"; exit 130' INT
trap 'echo "被终止"; exit 143' TERM

echo "$$ > /tmp/myapp.lock"
# 干活...
sleep 3600
```

**重点**:

- `trap '...' EXIT` 是**最有用的一个**——脚本以任何方式退出(正常 / Ctrl-C / kill / set -e 触发)都会执行,适合做清理
- `trap '...' INT` 单独处理 Ctrl-C
- `trap '...' TERM` 单独处理 kill 默认信号
- `trap - INT` 清除 INT 的 handler(回到默认)
- `trap '' INT` 让脚本**忽略** Ctrl-C(让你的脚本无法被 Ctrl-C 中断,**谨慎用**)

### 8.2 生产级用法:启动子进程 + 优雅清理

```bash
#!/usr/bin/env bash
set -euo pipefail

# 启动一个子进程
worker_pid=

cleanup() {
  echo "[$(date +%H:%M:%S)] 清理中..."
  if [[ -n "$worker_pid" ]] && kill -0 "$worker_pid" 2>/dev/null; then
    echo "停子进程 $worker_pid"
    kill -TERM "$worker_pid"           # 先礼后兵:SIGTERM
    sleep 2
    if kill -0 "$worker_pid" 2>/dev/null; then
      echo "子进程没退,SIGKILL"
      kill -KILL "$worker_pid"
    fi
  fi
  rm -f /tmp/myapp.lock
}

trap cleanup EXIT INT TERM

# 写 PID 文件,启动 worker
echo $$ > /tmp/myapp.lock
python worker.py &
worker_pid=$!

# 等子进程退出,或者收到信号被打断
wait "$worker_pid"
```

这是一段**真能用在生产 wrapper 脚本**的代码:

- `cleanup` 函数被三个时机触发:EXIT(脚本任何方式退出)/ INT(Ctrl-C)/ TERM(kill)
- `kill -0 $pid` 是经典「检查进程是否存活」的招——`-0` 不发信号,只检查权限和存在性
- 先 SIGTERM 给 2 秒,不退再 SIGKILL——**这就是 k8s / docker 优雅下线的做法**(`terminationGracePeriodSeconds`)
- `wait $pid` 让脚本「等子进程」期间能被信号打断,不然 trap 的清理函数要等子进程结束才能跑

### 8.3 trap 的陷阱

- **trap 不能在 wait 完全执行——shell 在 wait 时收到信号会先跑 handler,然后 wait 返回非 0**——这是设计如此,但很多人没意识到
- **trap 在子 shell 不继承**——`(sleep 100)` 起子 shell,父 shell 的 trap 在子 shell 里失效
- **EXIT 触发顺序**:`set -e` 失败 → trap EXIT 跑 → 脚本退出。错误码在 trap 里通过 `$?` 拿(在 trap 函数第一行 `local rc=$?` 保存,后面的命令会覆盖 `$?`)

## 九、退出码与 `$?`

每个进程退出时给父进程一个 **8 位整数**(0-255)。约定:

```
0            成功
1-125        程序自定义的失败码(具体含义看文档)
126          权限不够、不能执行
127          命令不存在
128 + N      被信号 N 杀死(SIGINT = 2 → 退出码 130
                              SIGTERM = 15 → 143
                              SIGKILL = 9 → 137)
```

实用查看:

```bash
$ ls /not-exist
ls: cannot access '/not-exist': No such file or directory
$ echo $?
2

$ sleep 100
^C
$ echo $?
130                       # 130 = 128 + 2,被 SIGINT 杀

$ false
$ echo $?
1
$ true
$ echo $?
0
```

### 9.1 短路操作:&& 和 ||

```bash
cmd1 && cmd2     # cmd1 成功才跑 cmd2
cmd1 || cmd2     # cmd1 失败才跑 cmd2
cmd1 && cmd2 || cmd3   # 注意!坑:cmd1 成功且 cmd2 失败也会跑 cmd3
```

**典型用法**:

```bash
# 启动前确认目录存在
mkdir -p /var/log/myapp && start_service

# 失败回退
deploy_new || deploy_old

# 测试 + 输出
[[ -f config.yml ]] || { echo "config missing"; exit 1; }
```

### 9.2 管道的退出码

```bash
$ false | true
$ echo $?
0
```

**默认 `$?` 只看管道最后一个命令**——这是个非常容易踩的坑,典型场景:

```bash
$ cat data.csv | head -n 1000 | grep ERROR | wc -l
0     # 但 cat 失败了你不知道(比如文件不存在)
```

修法:`set -o pipefail`(bash / zsh 都支持):

```bash
$ set -o pipefail
$ false | true
$ echo $?
1                  # 现在能拿到第一个失败的退出码
```

**生产 shell 脚本三件套**(27 篇细讲):

```bash
set -e             # 遇错退出
set -u             # 用未定义变量报错
set -o pipefail    # 管道任何一段失败都算失败
# 简写:set -euo pipefail
```

如果你写的脚本不带这三行,**它出错时大概率不会停而是继续往下跑**——这是 shell 脚本灾难的最大来源。

## 十、常见误解一锅端

| 误解 | 真相 |
| --- | --- |
| 「`kill -9` 是首选」 | 错。SIGKILL 不可拦截 = 程序来不及清理。**默认 `kill` 发 SIGTERM,给程序清理机会**,卡死了再 `-9` |
| 「nohup 就万无一失」 | 错。**nohup 只挡 SIGHUP**,Ctrl-C(SIGINT)/ `kill`(SIGTERM)/ `kill -9`(SIGKILL)照样能杀 |
| 「tmux 里跑的程序不怕断网」 | 对,但**仅限网络断**——tmux server 是 daemon 不死。如果你 `tmux kill-server` 或机器重启,里面的进程一样死(16 篇展开) |
| 「`&` 就是后台」 | 严格说是「**丢到后台进程组**」——shell 不等它就继续读输入,但它仍在原会话里。会话死了它也跟着死 |
| 「Ctrl-C 是 shell 处理的」 | 错。**pty 终端驱动检测组合键,内核产生信号,shell 完全不参与**。shell 只是看到子进程因 SIGINT 退出而已 |
| 「Ctrl-Z 是暂停,Ctrl-C 是退出」 | 部分对。Ctrl-Z 发 SIGTSTP(可拦截),Ctrl-C 发 SIGINT(可拦截),都是「礼貌请求」,程序可以拒绝 |
| 「`disown` 就让进程脱离 shell」 | 不完全。disown 只是**把作业从 shell 的作业表里删**,进程的进程组、会话、控制终端都没变——它仍在原 pty 上,只是 shell 退出时不再发 SIGHUP |
| 「kill PID 杀进程」 | 进程组场景下,**`kill -- -PGID` 给整组发信号**(注意是负号开头的 PGID),`kill PID` 只发给单个 PID。子孙进程要用进程组才杀得干净 |
| 「守护进程要 fork 两次,玄学」 | 不玄学。第一次 fork → setsid → 第二次 fork → 子进程不是会话首,以后再 `open(tty)` 也不会变成控制终端。**只是个防呆设计**,现代 systemd 服务不需要,容器里更不需要 |
| 「Ctrl-\\ 没人用」 | 严重低估。**Ctrl-\\ = SIGQUIT,90% 程序不拦它**,Ctrl-C 杀不掉时它经常救命 |
| 「程序卡死 SIGKILL 一定杀得掉」 | 错。**D 状态(uninterruptible sleep)连 SIGKILL 都没用**——进程卡在内核态 IO,要等内核返回才行。诊断要看 `/proc/<pid>/stack` |
| 「pipefail 默认开」 | 错。`set -o pipefail` 默认**关**,几乎所有 shell 都这样。这是 shell 脚本最常见的「静默失败」来源 |
| 「Python `KeyboardInterrupt` 抓住就完了」 | 不对。如果你 `except KeyboardInterrupt: pass` 不退出,**程序会拒绝 Ctrl-C**。生产代码里至少要 `sys.exit(130)` |
| 「容器里的程序信号都正常」 | 错。**容器里 PID 1 默认不接信号**(除非显式注册 handler),`docker stop` 给它发 SIGTERM 它收不到。要用 `--init` 或代码里手动注册 |

## 十一、看完这一篇你应该能

- **在白板上画出 Ctrl-C 的完整链路**:你按键 → 终端模拟器 → pty → 内核 isig 检查 → 产生 SIGINT → 投递给前台进程组 → 进程 handler → 退出
- **讲清楚 SIGHUP 的传播路径**:终端关 → pty HUP → 内核发给会话首 → shell 转发给所有作业 → 作业默认退出。**这是 ssh 断网训练挂的原因**
- **区分 `nohup` / `disown` / `setsid` / `tmux` 各自挡的是哪一刀**:nohup 让进程忽略 SIGHUP / disown 从 shell 作业表删 / setsid 开新会话 / tmux 让进程跑在另一个 daemon 持有的 pty 里
- **遇到 Ctrl-C 杀不掉时升级**:Ctrl-\\(SIGQUIT)→ `kill`(SIGTERM)→ `kill -9`(SIGKILL)→ 看 `/proc/pid/stack`(D 状态)
- **写 shell 脚本时**:开头 `set -euo pipefail`,清理逻辑用 `trap '...' EXIT`,启动子进程后用 `wait`
- **不再把信号当快捷键**:每个信号有「谁产生 / 谁接收 / 默认行为 / 可不可拦截」四要素,看到代码里 `signal(SIGTERM, ...)` 知道作者在防什么
