    文本三剑客 grep / sed / awk:三种范式、三套小语言,2026 还要学到什么程度

    每个工程师都用过 grep,但很少有人系统学过 sed 和 awk——它们看起来像一行行晦涩的咒语,在 Stack Overflow 上被复制、粘贴、再也不去理解。真相是:这三个工具是 1970s 三种不同的「文本处理范式」被精炼成的三门小语言(domain-specific language),每一个解决一类问题,各自的内部都有一致的设计美学。学清楚之后,80% 的简单需求一行解决,剩下 20% 你立刻知道应该换更现代的工具。

    这一篇不是 man 速查,是「为什么三剑客是这样设计的」——为什么 sed 一行一行处理、为什么 awk 有 BEGIN/END、为什么 grep 有三种正则方言,以及 2026 年这三个工具各自的合理生命周期还剩多少。

    一句话先记住:grep 找、sed 改、awk 算——三件事在 Unix 哲学下被拆成三个小语言,这是它们 50 年不死的原因。但写复杂逻辑时,任何一个都会变成「为了不写 Python 而硬写的怪物」。简单场景一行写完是优雅,超过 10 行就是技术债。

    一、三剑客的分工(先看清地图)

    很多人脑子里 grep / sed / awk 是一堆同类工具——其实它们解决三个不相交的问题:

                        grep            sed              awk
─────────────────────────────────────────────────────────────────
定位                找匹配           流编辑           列处理 + 小语言
单位                行              行(可多行)       字段(record + field)
核心操作            print 匹配      s/x/y(替换)     BEGIN / pattern / END
状态                无              有(hold space)  有(变量、数组、函数)
适合                查日志/找代码    批量替换         列聚合 / 报表 / 算
不适合              复杂提取         多行复杂逻辑     超过 30 行的脚本
代码量上限          1 行            1-3 行           1-10 行
超过就该换          rg              sd / 编辑器宏     Python / DuckDB

    读懂这张表,你已经知道 60% 的「该用哪个」:查日志找关键词 → grep;批量改一堆文件里的字符串 → sed;算 nginx 日志 top 10 IP / 求和 / 分组 → awk。

    把这三件事混着写就会写出怪物级 sed 脚本——状态机用 sed 的 hold space 模拟、嵌套条件用花括号——那是上世纪的 PTSD,不要参与。

    二、grep 心智:正则匹配 + 行过滤

    2.1 grep 是什么

    grep 名字来自 ed 编辑器的 g/re/p 命令:global regular-expression print——「全局执行这个正则,把匹配的行打印出来」。1973 年由 Ken Thompson 写出来,到 2026 年改变最少的工具之一。

    输入流 ──┬─→ 第 1 行 ──正则匹配?──→ 打印 / 不打印
        ├─→ 第 2 行 ──正则匹配?──→ 打印 / 不打印
        └─→ ...

    核心心智:一行一行扫,对每行套同一个正则,符合的打印。没有状态、没有多行、没有变量。这是 grep 的「窄设计」——窄到所有竞争者都干不掉它。

    2.2 三种正则方言

    grep 有三种正则的支持,这是 80% 的人不知道的:

    模式 flag 正则风格 何时用
    BRE(Basic) 默认 . * ^ $ [],但 + ? \| () 要加反斜杠 几乎不要用,设计反人类
    ERE(Extended) -E 或 egrep + ? \| () 直接用 日常默认就这个
    PCRE(Perl) -P 完整 Perl 正则,有 \b \d \s (?:...) (?=...) 复杂场景,需要回溯引用 / lookahead 
    # BRE(反人类,但默认):
grep 'foo\|bar' file               # 必须给 | 加反斜杠
grep '\(foo\)\(bar\)' file         # 分组要反斜杠

# ERE(推荐默认):
grep -E 'foo|bar' file
grep -E '(foo)(bar)' file

# PCRE(只在需要时):
grep -P '\bword\b' file            # 单词边界
grep -P '(?<=foo)bar' file         # 后向断言

    建议:别名给自己设 alias grep='grep -E',90% 场景直接用 ERE,正则不用满地撒反斜杠。

    egrep 和 fgrep 在 GNU grep 里是别名,POSIX 标准只有一个 grep——egrep ≡ grep -E,fgrep ≡ grep -F(固定字符串,不当正则)。

    2.3 必须会的 flag(只挑高频)

    -i        忽略大小写
-v        反向(不匹配的)
-r / -R   递归整个目录
-l        只列文件名(不打印匹配内容)
-c        只数匹配的行数
-n        显示行号
-w        匹配整个单词(不匹配 substring)
-A N      匹配行 + 后 N 行       ★ 排查神器
-B N      匹配行 + 前 N 行       ★ 排查神器
-C N      匹配行 + 前后各 N 行   ★ 排查神器
--include='*.py'     只在指定后缀里搜
--exclude-dir=node_modules   排除目录
-F        固定字符串(不当正则,提速 + 安全)
-E        ERE
-P        PCRE

    80% 的工程师只会 -i -v -r——剩下的 -A -B -C 是真正改变排查效率的:

    # 排查神器:找 ERROR 后续 5 行(看 stack trace)
grep -A 5 'ERROR' app.log

# 找 ERROR 前 3 行(看错误前的上下文)
grep -B 3 'ERROR' app.log

# 前后各 5 行(看完整上下文)
grep -C 5 'connection reset' /var/log/syslog

    没学过这三个 flag 的工程师,平均每次排错多花 5 分钟在 less +/ERROR 里翻。

    2.4 真实场景 5 例

    # 1. 找 import 行,排除 test 文件
grep -rn 'import requests' --include='*.py' . | grep -v '/tests/'

# 2. 反向找(不含 200 OK 的访问日志)
grep -v ' 200 ' access.log | head -20

# 3. 只列出包含 TODO 的文件名
grep -rl 'TODO' src/

# 4. 数当前 src 下有多少处用了某个 deprecated API
grep -rc 'OldClient(' src/ | grep -v ':0$'

# 5. 找两个关键词都出现的行(AND 语义,grep 没原生 AND)
grep 'error' app.log | grep 'timeout'

    第 5 例值得展开——grep 没有 AND,要表达「同时包含 A 和 B」就是两次 grep 管道。如果要 ERE 的 OR,直接 grep -E 'A|B'。这是 grep 的设计选择:复杂逻辑不归我管,组合管道。

    2.5 grep 的边界

    这些场景 grep 已经不够用了:

    # 1. 跨多行匹配(stack trace 跨行)
# grep 一次只看一行,跨行匹配要 -P + -z(危险)或换工具
# 正确做法:grep -A N 配合 ERROR 关键词,或用 ripgrep / Python

# 2. 大仓库递归搜索
grep -rn 'foo' .          # 1 GB 代码库慢得能去喝杯咖啡
rg 'foo'                  # ripgrep 快 5-10 倍,默认忽略 .gitignore

# 3. JSON 字段搜索:别用 grep,用 jq
# 4. 代码语义级搜索:别用 grep,用 ast-grep / LSP

    第 11 篇会讲 ripgrep(rg),几乎所有交互式查找都该换 rg——grep 留给学心智、留给 POSIX 严格的环境(如 Alpine 的 BusyBox grep)、留给 shell 脚本的健壮兼容。

    三、正则速记:够用的子集

    工程师的 80% 日常,这 10 个字符够了:

    .        任意单字符(不含换行)
*        前一个 0+ 次
+        前一个 1+ 次       (ERE / PCRE)
?        前一个 0/1 次      (ERE / PCRE)
^        行首
$        行尾
[]       字符类   [a-z]  [^0-9](取反)  [a-zA-Z0-9_]
()       分组              (ERE / PCRE)
|        或                (ERE / PCRE)
\        转义 . * + ? ^ $ [] () |
\b       单词边界          (PCRE,GNU grep 也支持)
\d \s    数字 / 空白       (PCRE 才有)

    真实例子:

    # IP 地址(简化版)
grep -E '[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+' access.log

# 行末以 .py 结尾的(grep 输出过滤)
ls | grep -E '\.py$'

# 单词边界(避免 ERROR_HANDLER 也被匹配)
grep -P '\bERROR\b' app.log
grep -wE 'ERROR' app.log            # -w 等价于加单词边界

# 邮箱(简化,真实邮箱正则是噩梦,不在这里追求严谨)
grep -E '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}' contacts.txt

# HTTP 5xx
grep -E ' 5[0-9]{2} ' access.log

# IPv4 但严格点(每段 0-255)→ 已经在 grep 边界了,换 Python

    对工程师:别学 lookahead / lookbehind / atomic group / 命名捕获——那是写 parser 的人的事。正则是 grep / sed / awk 共用的「输入语言」,不是 grep 独有。

    四、sed 心智:流编辑器

    4.1 sed 是什么

    sed = stream editor,1973 年从 ed(行编辑器)演化而来。核心心智一句话:

    sed 一行一行读输入,对每行套规则(命令),把结果打到 stdout

    输入 ──┬─→ 第 1 行 ──读入 pattern space ──执行命令 ──输出
      ├─→ 第 2 行 ──读入 pattern space ──执行命令 ──输出
      └─→ ...

    关键概念: - pattern space:当前正在处理的那一行(默认) - hold space:辅助暂存空间(高级用法,工程日常 95% 用不到) - 命令(command):对 pattern space 干什么——替换、删除、打印、追加、插入

    4.2 sed 的命令集(只挑日常)

    s/x/y/[flags]    替换:x → y;flags 有 g(全部)/ i(忽略大小写)/ N(第 N 个)
d                删除当前行(不打印)
p                打印当前行(常配 -n 用)
a\ text          在当前行后追加 text
i\ text          在当前行前插入 text
c\ text          替换整行为 text
y/abc/ABC/       字符级翻译(类似 tr)
=                打印行号
n                读入下一行
q                退出

    地址(address)前缀:命令前可以带地址,指定对哪些行执行:

    5d                第 5 行删除
5,10d             第 5-10 行删除
1,$ s/x/y/g       第 1 到末尾(全部),替换 x 为 y
/ERROR/d          匹配 /ERROR/ 的行删除
/start/,/end/d    从 /start/ 到 /end/ 之间的行删除(包含两端)
/ERROR/!d         不匹配 ERROR 的行删除(等价于「只保留 ERROR」)

    这就是 sed 的全部基础——三个东西:地址、命令、flag。和 grep 一样窄,但有「改」的能力。

    4.3 必须会的 5 个真实场景

    # 1. 全文替换 x 为 y
sed 's/foo/bar/g' file.txt

# 2. 删除空行
sed '/^$/d' file.txt

# 3. 删除某个块(从 START 标记到 END 标记)
sed '/# START/,/# END/d' config.txt

# 4. 把所有 IP 替换为 [REDACTED](脱敏)
sed -E 's/[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+/[REDACTED]/g' access.log

# 5. 只打印 100-200 行
sed -n '100,200p' large.log

    第 5 个例子用了 -n——sed 默认每行都打印一次,加 -n 是「闭嘴模式」,只在显式 p 时打印。这是 sed 的 idiom,等价于 awk 'NR>=100 && NR<=200'。

    4.4 -i 原地修改:macOS vs Linux 行为不同

    这是 sed 的史诗级坑:

    # GNU sed(Linux):
sed -i 's/foo/bar/g' file.txt              # 直接原地改

# BSD sed(macOS 默认):
sed -i '' 's/foo/bar/g' file.txt           # -i 必须带后缀参数,空串表示不备份

# 跨平台兼容写法(都带 .bak 备份):
sed -i.bak 's/foo/bar/g' file.txt          # macOS、GNU 都接受

    这一条坑导致无数 mac 用户从 Stack Overflow 抄 sed -i 's/...' 直接报错。macOS 用户两种解法: 1. brew install gnu-sed,装 gsed,用 gsed -i ... 2. 永远写 sed -i.bak ...(GNU 和 BSD 都兼容,生成 .bak 文件可以再 rm *.bak)

    4.5 sed 的「分隔符」可以换

    s/x/y/ 默认用 /,但 / 在文件路径里太常见,看着难受:

    # 用 / 当分隔符,路径里的 / 要转义,丑死:
sed 's/\/usr\/local\/bin/\/opt\/bin/g' config

# sed 允许换分隔符,用任何字符,比如 |
sed 's|/usr/local/bin|/opt/bin|g' config

# 或 #
sed 's#/usr/local/bin#/opt/bin#g' config

    实战 idiom:替换路径用 |,替换字符串用 /,替换 SQL 里的引号用 #。

    4.6 真实场景 5 例(扩展版)

    # 1. 把所有 console.log(...) 注释掉(JS / TS)
sed -i.bak 's|^\(.*console\.log.*\)$|// \1|' *.js
# \1 是「回溯引用」,引用第 1 个 \(...\) 捕获的内容

# 2. 把 yaml 里某个 key 的值改了(简单场景)
sed -i.bak 's/^replicas: [0-9]\+$/replicas: 5/' deployment.yaml

# 3. 删除 git diff 输出里的 + 号(把 diff 改回正常代码)
sed 's/^+//' patch.diff

# 4. 把 Markdown 里的 H1 全转成 H2
sed -E 's/^# /## /' README.md

# 5. 给每一行加行号(简陋版,不如 awk 'NR=NR; ...')
sed = file.txt | sed 'N;s/\n/\t/'
# 这个就开始变怪物了——见 4.7

    4.7 何时该换工具

    第 5 个例子已经在告诉你 sed 的边界:只要涉及「多行联动」「状态机」「计数」「数学运算」,就是 sed 的死区。

    sed 的死区:

    ✗ 跨行匹配(stack trace 处理)        → Python re.MULTILINE
✗ 计数 / 求和                          → awk
✗ 嵌套替换(JSON 字段)                → jq
✗ 多模式条件(if A 则 X,else 则 Y)    → awk / Python
✗ Unicode 多字节字符复杂处理            → Python(BSD sed 经常炸 UTF-8)

    现代替代: - 简单替换 + 安全(sd 工具,Rust 写的,默认正则就是 PCRE,不用反斜杠地狱) - 复杂多文件编辑 → 直接编辑器宏(Vim :bufdo、VSCode 多光标 + ripgrep) - 跨语言 / 跨平台脚本 → Python(re.sub + pathlib)

    第 11 篇会专讲 sd,这里只点名。

    五、awk 心智:列处理小语言

    这一节给最多篇幅,因为 awk 是被低估最严重的工具——90% 的工程师只会 awk '{print $1}',从不知道 awk 是一门完整的小编程语言:有变量、数组、函数、控制流、数学、字符串操作。

    5.1 awk 是什么

    awk 名字是三位作者的姓首字母:Aho、Weinberger、Kernighan(1977 年 Bell Labs)。Kernighan 就是 K&R C 那本书的 K,这三位都是 Unix 黄金时代的核心人物——awk 是他们设计的「让普通用户写报表的小语言」,目标读者是 1970s 的会计员、统计员,不是程序员。

    核心心智:

    awk 'BEGIN  { 初始化 }
     /pattern/ { 每条匹配 record 跑一次 }
     END    { 收尾 }' input.txt

执行流程:
  1. 跑一次 BEGIN(在读输入前)
  2. 对每条 record(默认是行)
       a. 把 record 按字段切开($1, $2, ...)
       b. 看是否匹配 pattern,匹配就跑对应 action
  3. 输入读完后跑 END

    两个关键抽象:record(记录,默认一行)和 field(字段,默认按空白切)。这是 awk 的「世界观」——所有输入都是「一条条记录,每条记录有几个字段」,这套抽象对结构化文本(日志、CSV、表格)简直完美贴合。

    5.2 awk 的内置变量

    $0      整条 record(整行)
$1      第 1 个字段
$2      第 2 个字段
$NF     最后一个字段(NF 是字段数)
$(NF-1) 倒数第 2 个字段

NR      当前 record 编号(从 1 开始,累加全局)
NF      当前 record 字段数
FS      输入字段分隔符(默认空白,可改)
OFS     输出字段分隔符(默认空格)
RS      输入 record 分隔符(默认 \n)
ORS     输出 record 分隔符(默认 \n)
FILENAME 当前文件名(多文件时有用)

    最常用的就是 $1 ... $NF + NR + NF,记住这 4 个就能搞定 80% 任务。

    5.3 必须会的 5 个真实场景

    # 1. 打印第 1 和第 3 列
awk '{print $1, $3}' file.txt

# 2. 求和(第 2 列加起来)
awk '{sum += $2} END {print sum}' sales.txt

# 3. 平均值(最后一列)
awk '{sum += $NF; n++} END {print sum/n}' timings.log

# 4. 按 IP 分组计数(类似 SQL GROUP BY)
awk '{ips[$1]++} END {for (ip in ips) print ips[ip], ip}' access.log | sort -rn | head -10

# 5. 条件过滤 + 列输出(类似 SQL WHERE)
awk '$3 > 100 {print $1, $3}' sales.txt

    第 4 个例子是 awk 的核心威力——关联数组(associative array),把 IP 当 key 存计数,在 END 时遍历输出。这一行 awk ≈ Python 写 10 行 dict + Counter。

    5.4 改字段分隔符:CSV / TSV / 自定义

    默认 FS 是「连续空白」(空格 / Tab 都行),处理 CSV 要改:

    # CSV(逗号分隔)
awk -F, '{print $1, $5}' data.csv
awk 'BEGIN{FS=","} {print $1, $5}' data.csv          # 等价

# TSV(Tab 分隔)
awk -F'\t' '{print $1}' data.tsv
awk 'BEGIN{FS="\t"; OFS="|"} {print $1, $2, $3}' data.tsv   # 输出用 | 分隔

# 多字符分隔符(只在 GNU awk / gawk 支持)
awk -F'::' '{print $1, $3}' /etc/services

    CSV 的真实痛:awk 不懂引号包裹的字段("name with, comma")——遇到这种 CSV,别硬用 awk,换 xsv / mlr / csvkit / Python pandas。这是 awk 的硬边界。

    5.5 pattern + action 的全部组合

    awk 的语法骨架是 pattern { action },两边可以缺一个:

    # 只有 pattern,action 默认 print
awk '/ERROR/' app.log              # 等价 grep ERROR

# 只有 action,对所有行执行
awk '{print NR, $0}'               # 给每行加行号

# pattern 可以是表达式(不只是正则)
awk '$3 > 100' sales.txt           # 第 3 列大于 100 的行
awk 'NR > 1' file.txt              # 跳过第 1 行
awk 'NF == 0' file.txt             # 空行
awk 'NR % 2 == 0' file.txt         # 偶数行

# 多 pattern + 多 action
awk '/ERROR/ {errors++} /WARN/ {warns++} END {print errors, warns}' app.log

    多 pattern { action } 块用空白分隔,每行 record 都会跑过所有块,匹配哪个就执行哪个 action——这是 awk 像 mini Python 而不像 sed 的根本原因。

    5.6 awk 的「编程语言」面

    很多人忘了 awk 是一门完整语言:

    # 变量(无需声明)
sum = 0
name = "alice"

# 算术
sum += $1
avg = sum / NR

# 字符串
greeting = "hello " name
len = length(name)
upper = toupper(name)
sub_str = substr($0, 1, 10)        # 截前 10 字符

# 控制流(if / for / while)
if ($3 > 100) print "high"
else if ($3 > 50) print "mid"
else print "low"

for (i = 1; i <= NF; i++) print $i
for (key in arr) print key, arr[key]

while (getline line < "other.txt") print line

# 函数(自定义)
function abs(x) { return x < 0 ? -x : x }
{ print abs($1) }

    这一组合下,awk 能解决「读一遍输入,做统计 + 简单变换」的几乎所有需求——只要逻辑不超过 20 行,awk 比 Python 短、比 sed 优雅、比 grep 强大。

    5.7 6 个进阶真实场景

    场景 1:nginx access log top 10 IP

    awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10

# 用 awk 自己也能搞,不用 sort/uniq:
awk '{ips[$1]++} END {for (ip in ips) print ips[ip], ip}' access.log | sort -rn | head -10

    两种写法的取舍:第一种(awk + sort + uniq)是 Unix 哲学的「小工具组合」,但 sort 是 O(n log n)、读两遍内存;第二种(纯 awk)只一遍输入,内存里维护 hash 表,对大文件更快。

    场景 2:HTTP 状态码分布

    awk '{codes[$9]++} END {for (c in codes) print c, codes[c]}' access.log | sort
# 输出:
# 200 84320
# 301 1502
# 404 829
# 500 12

    场景 3:错误响应的平均响应时间

    # 假设 $9 是 status,$NF 是 response_time
awk '$9 >= 500 {sum += $NF; n++} END {if (n > 0) print sum/n; else print "no errors"}' access.log

    场景 4:K8s pod 内存占用前 5

    kubectl top pod | awk 'NR > 1 {print $1, $3}' | sort -k2 -h | tail -5
# NR > 1 是跳过表头
# sort -h 是「human-readable」排序,懂 100Mi / 1Gi

    场景 5:多列条件过滤(类 SQL WHERE)

    # 第 3 列 > 100 且第 5 列等于 "premium"
awk '$3 > 100 && $5 == "premium" {print $1, $3}' users.txt

# 第 1 列匹配正则
awk '$1 ~ /^admin/' users.txt          # ~ 是 awk 的正则匹配
awk '$1 !~ /^admin/' users.txt         # !~ 是不匹配

    场景 6:多文件汇总,带文件名

    # 每个文件的行数(BEGIN/END 不能区分文件,用 FILENAME)
awk 'END {print NR, "lines total"}' *.log

# 按文件统计(GNU awk 才有 ENDFILE)
awk 'ENDFILE {print FILENAME, FNR}' *.log    # FNR 是当前文件内的行号

    5.8 BEGIN / END 的高频用法

    # BEGIN 初始化变量、设置 FS
awk 'BEGIN {FS=","; print "Name,Total"} {sum[$1]+=$3} END {for (n in sum) print n","sum[n]}' data.csv

# END 输出汇总
awk '{sum+=$1; n++} END {print "avg:", sum/n, "count:", n}' nums.txt

# 没有 input 也能跑(只用 BEGIN)——把 awk 当计算器
awk 'BEGIN {print 3.14 * 2}'                  # 6.28
awk 'BEGIN {for (i=1; i<=10; i++) print i*i}' # 平方表

    最后一招很有用——awk 是个一行计算器,比 python -c '...' 启动快得多。

    5.9 awk 的边界

    awk 的死区:

    ✗ 嵌套结构(JSON / XML)             → jq / yq
✗ 引号包裹的 CSV                       → xsv / mlr / pandas
✗ Unicode 复杂处理                    → Python(awk 默认按字节算 length)
✗ 大数据(GB+ 计算)                  → DuckDB / Polars(列存,向量化)
✗ 状态机 / 复杂逻辑                   → Python
✗ 多步骤复杂数据流                    → 写 .py

    何时该换 Python:awk 脚本超过 30 行,或开始用 getline、hold space 这类高级特性,立刻停——下面是临界点信号:

    ✗ awk 脚本里出现了 5+ 个全局变量
✗ awk 里在处理 JSON / 多层嵌套
✗ awk 里写嵌套 if / for 超过 3 层
✗ 你开始想要「单元测试这段 awk」

    任何一个信号亮起,立刻打开 .py 文件——awk 不是用来跟 Python 抢命的。

    5.10 awk vs gawk vs mawk vs nawk

    这些是 awk 的不同实现,行为有细微差别:

    实现 来源 性能 扩展
    awk POSIX 标准 - 最简
    mawk Mike Brennan 的 C 实现 最快 接近 POSIX
    gawk GNU awk 快 最多扩展

    macOS 默认 BSD awk(慢),Debian/Ubuntu 默认 mawk(快),CentOS / Fedora 默认 gawk(扩展多)。处理大文件时 macOS 建议 brew install gawk 后 alias awk=gawk。

    跨平台脚本中,只用 POSIX 子集,不要依赖 gawk 扩展(比如 gensub、patsplit)。

    六、三剑客的常见组合

    工程师日常的 80% 是「grep 找 + sed 改 + awk 算」的链式管道。几个经典模板:

    6.1 找 + 算

    # nginx 5xx 错误的 top 10 URL
grep ' 5[0-9][0-9] ' access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -10

# 找到匹配的行,统计某字段的总和
grep 'order_completed' events.log | awk -F, '{sum+=$3} END {print sum}'

    6.2 找 + 改

    # 找配置项的当前值
grep '^port' config.ini | sed 's/^port *= *//'

# 找出文件里所有 IP 并去重
grep -oE '([0-9]+\.){3}[0-9]+' access.log | sort -u

    6.3 算 + 排

    # K8s pod 内存占用前 5(去掉单位)
kubectl top pod | awk 'NR>1 {print $1, $3}' | sort -k2 -h | tail -5

# git 提交者 top 10
git log --format='%an' | sort | uniq -c | sort -rn | head -10
# 也可纯 awk:
git log --format='%an' | awk '{c[$0]++} END {for (n in c) print c[n], n}' | sort -rn | head -10

    6.4 三件套全用

    # Apache 错误日志:统计每天的 ERROR 数量
grep ERROR error.log \
  | awk '{print $1}' | sed 's/\[//' \
  | sort | uniq -c

    这种链式管道有时很优雅,但超过 4 步就该想想「是不是该写个 .py」——管道越长,可读性越差、错误越难定位。

    要学,但学到刚好够用——这是 2026 年的合理立场。

    7.1 grep:必学,几乎日终生有效

    ✓ 学 grep 心智(正则 + 行匹配)
✓ 学 -A/-B/-C(排查神器)

✗ 不学:复杂正则、PCRE 高级特性(那是 Python 的事)

    grep 心智 = 正则心智,正则是所有文本工具的基础语言,这个心智无论 2026 还是 2036 都有效。

    7.2 sed:基础够用,复杂场景早换

    ✓ 学 s/x/y/g(替换)
✓ 学 -i 原地修改(及 macOS/Linux 区别)
✓ 学地址范围(行号、正则、范围)
✓ 学常用命令 d / p / a / i
✗ 不学:hold space、多行模式、高级标签跳转

    sed 学到「能改一堆文件」就停——再深就是上世纪 PTSD,2026 这个深度没价值。

    7.3 awk:基础真值钱,深入会上瘾

    ✓ 学 $1 / $N / NR / NF
✓ 学 BEGIN / END
✓ 学关联数组(分组计数)
✓ 学 FS / OFS(CSV / TSV)
✓ 学条件 + 多 action 块
✗ 不学:getline 玩成 mini Python、用 awk 写 50 行脚本

    三行 awk 经常能省 30 行 Python——这就是它的核心价值。但 awk 的「上瘾」是真陷阱:你写完 30 行 awk 觉得自己是巫师,但 6 个月后回头看自己写的 awk 没人能维护(包括你)。

    7.4 一个工程立场

    2026 的合理边界:
  awk 简单聚合 / 分组 / 求和                 → 写 awk(3 行内)
  sed 简单替换 / 删除                       → 写 sed(1 行)
  grep 找匹配                               → 写 grep

  awk 算复杂逻辑                            → Python / DuckDB
  sed 跨行复杂替换                          → Python re.MULTILINE 或编辑器宏
  grep 大仓库递归搜索                       → rg(下一篇)
  grep 找 JSON 字段                         → jq(13 篇)
  grep 找代码符号                           → LSP / ast-grep

    九、UUOC:Useless Use of Cat

    「UUOC」是一种 Unix 文化梗——Useless Use of Cat,「无谓的 cat」。

    # 反面:
cat file | grep x                     # cat 是多余的!
cat file | wc -l                      # cat 是多余的!
cat file | awk '{print $1}'           # cat 是多余的!

# 正面:
grep x file
wc -l file        # 或 wc -l < file
awk '{print $1}' file

    为什么是「无谓」:grep / wc / awk 本来就能从文件读、能从 stdin 读,加一个 cat 多 fork 一个进程,多一次内存拷贝,多一段时间,毫无收益。

    唯一合理的 cat:多文件拼接,因为 grep 等工具的多文件输入有时不便:

    # 合理(虽然也能 grep x a.txt b.txt):
cat a.txt b.txt c.txt | grep x

# 更合理(进程替换):
grep x <(cat a.txt b.txt c.txt)

    但 90% 的「cat | xxx」都是多余的——写 cmd file 而不是 cat file | cmd,这是个免费效率提升。

    工程意义不只是「省一个进程」:cat | grep 暴露你不知道 grep 能直接读文件——这是个「我对工具不熟」的信号。这一条改掉后,你写的管道平均会短一段、读起来也更清晰。

    十、反对的写法

    这一篇结束前,明确反对几种常见的烂写法,见到了就改掉:

    10.1 awk 写 50 行脚本

    # 反例:
awk '
BEGIN { ... 复杂初始化 }
/pattern1/ { 状态机 ... }
/pattern2/ { 嵌套 if/for/while ... }
/pattern3/ { getline 读其他文件 ... }
END { 复杂报表 ... }
' input.txt

    到 30 行立刻停——打开 .py 文件,用 Python 重写。awk 不是用来跟 Python 抢命的,它的甜区是「3-10 行内的列处理」。

    10.2 sed 跨行复杂替换

    # 反例:
sed 'N;N;N;s/foo\nbar\nbaz/REPLACED/'      # 把连续 3 行替换

    这种「跨行模式」用 sed hold space + N 命令是噩梦——直接 Python re.sub + re.MULTILINE:可读、可测、可维护。

    10.3 grep 找 JSON 字段

    # 反例:
cat events.json | grep '"user_id":"12345"'    # 脆,引号、空格、换行都炸

# 正例:
jq 'select(.user_id == "12345")' events.json

    JSON 是结构化数据,用结构化工具(jq,13 篇会专讲)。

    10.4 grep 找代码符号

    # 反例:
grep -rn 'function foo' src/                # 字符串里的 "function foo" 也会被找到

# 正例:
ast-grep 'function foo($_) { $$ }'          # 语义级
# 或用 LSP 的 「Find references」

    代码是语法树,用语法树工具。

    10.5 UUOC 链

    # 反例(每个团队都有):
cat config.yaml | grep host | awk '{print $2}' | sed 's/"//g'

# 正例:
awk '/host/ {gsub(/"/, "", $2); print $2}' config.yaml
# 或更干脆:
yq '.host' config.yaml

    链越长越脆——4 段管道还能读,8 段管道就成了黑魔法。每多一段管道,出错可能 ×2。

    十一、看完这一篇你应该能

    1. 解释 grep / sed / awk 的分工
   答:grep 找匹配、sed 改字符、awk 算列
   答:对应「无状态过滤」「单行编辑」「字段处理小语言」

2. 区分 grep 的 BRE / ERE / PCRE
   答:BRE 默认但难用(+ ? | () 要转义)、ERE 用 -E、PCRE 用 -P
   答:日常默认 grep -E

3. 用 grep 的 -A / -B / -C 排查日志上下文
   答:-A 3 看错误后续、-B 3 看错误之前、-C 5 看前后

4. 在 macOS 和 Linux 上跨平台用 sed -i
   答:写 sed -i.bak 's/.../.../' 两边都兼容

5. 默写 awk 的 BEGIN / pattern { action } / END 心智
   答:BEGIN 初始化、每行匹配 pattern 跑 action、END 收尾
   答:$1...$NF 是字段,NR 是行号,NF 是字段数

6. 用 awk 关联数组做分组计数
   答:awk '{cnt[$1]++} END {for (k in cnt) print cnt[k], k}' file

7. 知道何时该换工具
   答:awk 超 30 行 → Python
   答:sed 跨行 → Python re.MULTILINE / 编辑器宏
   答:grep 大仓库 → rg
   答:grep 找 JSON → jq

8. 识别并改掉 UUOC
   答:cat file | grep x  →  grep x file

    达到这 8 条,你在三剑客上就达到了「够用,且知道边界」的合理水平——再深入是浪费时间,该把时间花在 rg、jq、Python 上。
