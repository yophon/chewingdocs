包管理器对比:brew / apt / pacman / Nix / 三层架构与选型 ​

工程师每天和包管理器打交道,但很少有人能讲清楚自己每天用的是哪几个、它们各管什么、哪些功能在打架。装个 git 用 brew、装个 Node.js 也用 brew、装个 Python 用 pip、装个 numpy 也用 pip、装个特定版本 Node 又用 nvm、装个 zsh 插件靠 zinit——一台普通工作机上同时跑着 5-10 个不同的包管理器,互相不知道对方装了什么,冲突、漂移、版本错乱是常态。这种混乱不是工程师不努力,而是没人在初学阶段把"包管理器分层"这件事讲清楚——大家以为"包管理器"是一回事,实际是三件事(三层),混用不同层的工具是 99% 包管理混乱的根源。

一句话先记住:包管理器要分三层看待——系统级(brew / apt / pacman / Nix)只装"基础设施"(git / ssh / curl / 二进制工具),语言级(cargo / npm / pip / gem)装"语言生态",版本管理器(mise / asdf)装"多版本工具" 。三层分清,装错层是混乱起源。最常见的灾难就是用 brew 装 Python 或 Node——表面能跑,长期就是炸弹,因为你需要的是"项目 X 用 Python 3.11、项目 Y 用 Python 3.13",而 brew 假设"系统只有一个 Python"——这是它压根没考虑的场景。

这一篇拆开来讲:三层架构怎么分、brew / apt / pacman / Nix 各自适用什么、Brewfile 工作流怎么落、跨平台一份 install.sh、什么时候选 Nix。不写"brew 命令大全"——brew help 在那儿,本系列只讲"装错层的代价"和"选型逻辑"。

一、为什么要分层 ​

1.1 一台典型工作机上的包管理器 ​

打开任意一台工程师的 Mac,数一下同时运行的包管理器:

brew                  →  装系统级 CLI(git / ripgrep / fzf)
brew cask             →  装 GUI 应用(Chrome / 1Password / Ghostty)
mise (或 asdf)         →  管 Node / Python / Ruby / Go 多版本
npm / pnpm / yarn     →  装 Node 包(全局或项目级)
pip / poetry / uv     →  装 Python 包
cargo                 →  装 Rust 包
gem                   →  装 Ruby 包(macOS 自带 Ruby 还在)
go install            →  装 Go 工具
zinit / antidote       →  装 zsh 插件
nvim 的 lazy.nvim       →  装 Neovim 插件
docker pull            →  装容器镜像

总计:11+ 个包管理器同时在跑

任何"我搞不清装到哪了"的事故,都出在这堆里:

# 经典场景:python 命令到底走哪个?
$ which python
/Users/me/.local/share/mise/installs/python/3.13.0/bin/python    # mise 管的

# 但你又装过 brew
$ brew list | grep python
python@3.11
python@3.12

# 那 pip install 装到哪?
$ pip install requests
# 装到 mise 管的 Python 里(因为 mise 的 shim 在 PATH 前)
# 但你以为装到了 brew 的 Python 里
# 一个项目用 pip,另一个用 pip3,两个走不同 Python——爆炸

1.2 三层架构 ​

┌──────────────────────────────────────────────────────────┐
│  应用工具(mise / asdf / nvm / pyenv / rbenv)             │
│  - 多版本:Node 18 + Node 20、Python 3.11 + 3.13         │
│  - 项目级 .tool-versions / .nvmrc                        │
│  - 跟着 cwd 自动切版本                                    │
│  - 24 篇专讲                                              │
├──────────────────────────────────────────────────────────┤
│  语言级(cargo / npm / pnpm / pip / poetry / uv / gem)    │
│  - 各语言原生的包                                          │
│  - 装到当前激活的语言版本里                                 │
│  - 项目级依赖在 package.json / pyproject.toml / Cargo.toml│
│  - 本篇不重点(各语言系列展开)                              │
├──────────────────────────────────────────────────────────┤
│  系统级(brew / apt / pacman / dnf / Nix)                  │
│  - git / ssh / curl / 二进制工具                          │
│  - 不绑定语言的 CLI                                        │
│  - 这一层是"基础设施",装好就忘                            │
│  - 本篇核心                                               │
└──────────────────────────────────────────────────────────┘

规则:
  ✓ 系统级只装"和语言无关的基础工具"
  ✓ 语言生态包永远不要走系统级(brew install python 是反模式)
  ✓ 多版本场景永远走应用工具层(mise / asdf)

这三层的关键判断:

"这东西是 git / ssh / curl 这种通用工具吗?" → 系统级(brew / apt)
"这东西是 Node / Python / Ruby / Go 本身吗?" → 应用工具(mise / asdf)
"这东西是 lodash / requests / clap 这种语言包吗?" → 语言级(npm / pip / cargo)

反过来,装错层会出什么事:

错 1:用 brew 装 Python
  → 系统升级时 brew 升 Python 到 3.13
  → 你的项目锁在 Python 3.11
  → 项目 venv 突然找不到 3.11
  → 重装 venv,又一堆 dependency 不兼容
  → 浪费一天

错 2:用 npm install -g 装 prettier
  → 你升级 Node(brew upgrade node 或 mise install)
  → 全局 npm 包都丢了
  → 重装

错 3:把 vim plugin 装到 brew
  → 不存在这种事,但有人想这么干
  → Vim/Nvim 用插件管理器(lazy.nvim / packer),不走 brew

这一篇集中讲第一层(系统级)——第二层下一篇(24 篇)讲,第三层各语言系列展开。

二、brew(macOS 默认) ​

Homebrew 在 macOS 上是事实标准——2009 年 Max Howell 起手,2026 年还是大多数 Mac 工程师装第一个工具。

2.1 装哪儿 ​

macOS Apple Silicon (M1/M2/M3/M4):
   /opt/homebrew/                 ← brew 主目录
   /opt/homebrew/bin/             ← 可执行文件
   /opt/homebrew/Cellar/          ← 实际包内容
   /opt/homebrew/Caskroom/        ← Cask(GUI 应用)
   /opt/homebrew/etc/             ← 配置文件
   /opt/homebrew/var/             ← 数据(数据库等)

macOS Intel:
   /usr/local/                    ← brew 主目录(同上目录结构)

Linux(Linuxbrew):
   /home/linuxbrew/.linuxbrew/    ← brew 主目录(同上)

关键差异:Apple Silicon 的 brew 装在 /opt/homebrew,Intel 装在 /usr/local——很多老脚本假设 /usr/local,在 M 系列 Mac 上跑会失败。

激活 brew 的 PATH:

zsh
# ~/.zshrc 或 ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"      # M 系列
# 或
eval "$(/usr/local/bin/brew shellenv)"         # Intel

这一行把 brew 相关的 PATH、MANPATH、INFOPATH 都设好。

2.2 Formula 和 Cask ​

Formula:      命令行工具 / 库 / 服务
  brew install git
  brew install ripgrep
  brew install postgresql@16
  → 装到 /opt/homebrew/Cellar/<pkg>/<version>/,symlink 到 /opt/homebrew/bin

Cask:         GUI 应用 / 字体 / 二进制 .pkg
  brew install --cask ghostty
  brew install --cask 1password
  brew install --cask font-jetbrains-mono-nerd-font
  → 装到 /Applications/(应用)或 /opt/homebrew/Caskroom/(字体等)

Cask 的存在是 brew 区别于 apt 的关键——apt 没有"GUI 应用"概念,在 Linux 上 GUI 是 flatpak / snap / 包管理器自带,但没有 brew cask 这种"一行命令装 Chrome"的统一体验。

2.3 日常命令 ​

bash
# 装 / 升级 / 卸载
brew install <pkg>                  # 装一个
brew install --cask <pkg>           # 装 cask
brew upgrade                        # 升级所有(谨慎)
brew upgrade <pkg>                  # 只升一个
brew uninstall <pkg>                # 卸载

# 搜索 / 看信息
brew search <keyword>               # 模糊搜
brew info <pkg>                     # 看版本、依赖、安装路径
brew list                           # 列已装的 Formula
brew list --cask                    # 列已装的 Cask
brew outdated                       # 列待升级的

# 自检
brew doctor                         # 检查 brew 自身健康
brew cleanup                        # 删旧版本

# 索引
brew tap <user/repo>                # 加第三方 tap
brew untap <user/repo>              # 删 tap

2.4 Brewfile(强 ROI) ​

Brewfile 是 brew 最值钱的功能——把"我装了哪些包"声明在一个文本文件里,一键复刻。

Brewfile 例子(生产级):

ruby
# Brewfile
# 我的工作机配置

tap "homebrew/bundle"
tap "homebrew/cask-fonts"

# ─── CLI 工具 ───
brew "git"
brew "git-delta"            # git diff 增强
brew "gh"                   # GitHub CLI
brew "lazygit"              # git TUI
brew "neovim"
brew "tmux"

# 现代 CLI 替代品
brew "ripgrep"              # 替代 grep
brew "fd"                   # 替代 find
brew "bat"                  # 替代 cat
brew "eza"                  # 替代 ls
brew "dust"                 # 替代 du
brew "duf"                  # 替代 df
brew "procs"                # 替代 ps
brew "hyperfine"            # benchmark

# fuzzy 搜索 / 历史 / 提示符
brew "fzf"
brew "atuin"                # 替代 zsh 历史
brew "starship"             # 提示符
brew "zoxide"               # 替代 cd / autojump

# 多版本 / dotfiles
brew "mise"                 # 替代 nvm/pyenv/rbenv
brew "chezmoi"              # dotfiles 管理
brew "direnv"               # 项目级 env

# 网络 / SSH
brew "curl"
brew "wget"
brew "httpie"
brew "mosh"                 # 替代 ssh,弱网友好

# 结构化数据
brew "jq"
brew "yq"

# 容器 / K8s(按需)
brew "docker"
brew "kubectl"
brew "k9s"

# 语言运行时(注意:这里 NOT install Python/Node,走 mise!)
brew "go"                   # Go 通常一个版本够,可以 brew 装
brew "rust-analyzer"

# ─── Cask(GUI)───
cask "ghostty"              # 终端模拟器
cask "1password"
cask "1password-cli"
cask "raycast"
cask "obsidian"
cask "rectangle"            # 窗口管理

# 字体
cask "font-jetbrains-mono-nerd-font"
cask "font-fira-code-nerd-font"

# ─── App Store(需要 mas 工具)───
# brew "mas"
# mas "Things 3", id: 904280696

用法:

bash
# 导出当前安装的所有包到 Brewfile
brew bundle dump --file=~/Brewfile

# 在新机器上一键装
brew bundle --file=~/Brewfile

# 检查 Brewfile 和当前安装的一致性
brew bundle check --file=~/Brewfile

# 卸载没在 Brewfile 里的包(危险但好用)
brew bundle cleanup --file=~/Brewfile --force

配合 chezmoi(22 篇讲过),把 Brewfile 放在 dotfiles 仓库:

~/.local/share/chezmoi/
├── Brewfile                            # 跨机器一致的包
├── Brewfile.work                       # 工作机额外的
└── run_onchange_brew-bundle.sh.tmpl    # chezmoi 钩子

run_onchange_brew-bundle.sh.tmpl(chezmoi 钩子):

bash
#!/usr/bin/env bash
# 当 Brewfile 内容变化时,自动重跑 brew bundle

{{ if eq .chezmoi.os "darwin" }}
brew bundle --file={{ .chezmoi.sourceDir }}/Brewfile
{{ if eq .class "work" }}
brew bundle --file={{ .chezmoi.sourceDir }}/Brewfile.work
{{ end }}
{{ end }}

# chezmoi 计算这个文件的 hash,变化时重跑;不变就 skip

这一套组合(chezmoi + Brewfile) 是 2026 macOS 工程化的标准:新机器一行 chezmoi init --apply,配置 + 所有 brew 包都到位。

2.5 brew 在 Linux(Linuxbrew) ​

能跑但不推荐——Linux 上 brew 装在 /home/linuxbrew/.linuxbrew/,会和系统 apt 包冲突(都装 git 就有两份),且 Linux 包管理器(apt / pacman)是发行版深度集成的,用 brew 等于在 Linux 上模拟 macOS——多此一举。

例外:你在 Linux 上特定包必须比 apt 新——比如 Ubuntu 22.04 的 git 是 2.34,你要 2.45,这时候可以 brew install git 装新版。但这是补丁,不是日常做法。

三、apt(Debian / Ubuntu) ​

apt 是 Debian 系的默认包管理器——Ubuntu / Debian / Mint / Pop!_OS / Kali 都用。

3.1 日常命令 ​

bash
# 更新包索引(必做!不更新就装会装到老的)
sudo apt update

# 升级所有已装包
sudo apt upgrade
sudo apt full-upgrade           # 比 upgrade 激进,允许卸载

# 装 / 卸载
sudo apt install <pkg>
sudo apt install <pkg>=<version> # 装特定版本
sudo apt remove <pkg>            # 卸载,留配置
sudo apt purge <pkg>             # 卸载,删配置
sudo apt autoremove              # 删孤儿依赖

# 搜索 / 信息
apt search <keyword>             # 不需要 sudo
apt show <pkg>
apt list --installed             # 已装
apt list --upgradable            # 待升

# 修复
sudo apt --fix-broken install
sudo dpkg --configure -a         # dpkg 是 apt 的底层

3.2 apt 的特点 ​

+ 包的稳定性高:Debian Stable 是出了名的"老但稳"
+ 跟发行版深度集成:升级路径清晰
+ 包数量大:Ubuntu 主仓 60000+ 包

- 包旧:Ubuntu 22.04 的 vim 是 8.2(社区在 9.1)
- 不滚动:大版本升级是个事件(do-release-upgrade)
- GUI 应用走 snap / flatpak / 第三方 deb,不统一

3.3 PPA(第三方源) ​

某些工具(比如 neovim 的 nightly)apt 主仓没有,用 PPA 加第三方源:

bash
sudo add-apt-repository ppa:neovim-ppa/unstable
sudo apt update
sudo apt install neovim

坑:PPA 不被官方维护,升级到下一个 Ubuntu 大版本时可能失效——do-release-upgrade 会禁用所有 PPA,升级后要手动重新启用并 update。

3.4 dpkg(底层) ​

apt 是 dpkg 的高级封装。dpkg 是真正干活的:

bash
# 装一个本地 .deb 文件
sudo dpkg -i ./some-package.deb

# 列已装包
dpkg -l

# 看一个文件属于哪个包
dpkg -S /usr/bin/vim
# vim-tiny: /usr/bin/vim
# vim-runtime: /usr/share/vim/...

# 看一个包装了哪些文件
dpkg -L vim

何时用 dpkg:装单个 .deb 文件(比如下载的 Chrome / VS Code .deb),其他场景用 apt 就够。

3.5 apt vs apt-get ​

apt 是新接口(更友好),apt-get 是老接口(脚本里推荐):

bash
# 交互式 / 日常:用 apt
sudo apt install neovim

# 脚本里:推荐 apt-get(输出更稳定)
sudo apt-get install -y neovim

apt 在脚本里会警告:WARNING: apt does not have a stable CLI interface. Use with caution in scripts.——所以脚本里用 apt-get。

四、pacman(Arch / Manjaro) ​

pacman 是 Arch Linux 的包管理器——滚动更新、包最新。

4.1 日常命令 ​

bash
# 同步包数据库(类似 apt update)
sudo pacman -Sy                   # sync,不升级
sudo pacman -Syu                  # sync + upgrade,推荐

# 装 / 卸载
sudo pacman -S <pkg>              # install
sudo pacman -R <pkg>              # remove,留依赖
sudo pacman -Rs <pkg>             # remove + 删孤儿依赖
sudo pacman -Rns <pkg>            # remove + 配置 + 孤儿

# 搜索 / 信息
pacman -Ss <keyword>              # search 远程
pacman -Qs <keyword>              # 搜本地已装
pacman -Si <pkg>                  # show info(远程)
pacman -Qi <pkg>                  # show info(本地)
pacman -Q                         # 列已装
pacman -Qe                        # 列显式装的(不含依赖)

# 清理
sudo pacman -Sc                   # 清缓存(只删非最新)
sudo pacman -Scc                  # 清所有缓存(谨慎)

pacman 的 flag 体系是字母组合——这是它最反直觉的地方:-S(sync)、-R(remove)、-Q(query)、-U(upgrade local),初学者会被 -Syu 这种组合吓到,熟了之后比 apt 简洁。

4.2 AUR(用户仓库) ​

AUR 是 Arch 的杀招——任何人可以提交"装某个包"的脚本到 AUR,社区维护,包数量远超 apt。

bash
# pacman 本身不直接装 AUR,需要 AUR helper:
# 选 yay 或 paru(Rust 写的,推荐)

# 装 paru
sudo pacman -S --needed base-devel git
git clone https://aur.archlinux.org/paru.git
cd paru && makepkg -si

# 用 paru 装 AUR 包
paru -S google-chrome
paru -S visual-studio-code-bin

paru 的命令和 pacman 几乎一样——-S / -Syu 等都通用,只是会同时检查 AUR。

注意:AUR 的包是"用户提交的脚本",官方不背书——装的时候 paru 会让你看 PKGBUILD,默认是要审的(虽然大多数人都按 Enter 跳过)。生产机不推荐 AUR,个人机可以。

4.3 Arch 的优势 / 劣势 ​

+ 包最新:rolling release,几小时内就跟上 upstream
+ AUR 万能:几乎任何东西都能找到
+ pacman 快:数据库 + 解析比 apt 快
+ 学习深度:Arch wiki 是 Linux 文档质量天花板

- 滚动有风险:某次 -Syu 可能跑出问题(虽然不常见)
- 不适合生产:Arch 在公司服务器上几乎没人用
- 新手友好度差:装个 Arch 本身要折腾(虽然有 archinstall 了)

适合人群:个人台式机 / 笔记本、想用最新工具、能容忍偶尔小问题。不适合:服务器、不想折腾的人。

五、dnf(Fedora / RHEL) ​

dnf 是 RPM 系的现代包管理器——Fedora / RHEL / CentOS Stream / Rocky / Alma 都用。apt 的 Red Hat 版,命令风格类似。

bash
sudo dnf install <pkg>             # 装
sudo dnf remove <pkg>              # 卸
sudo dnf upgrade                   # 升级
sudo dnf search <keyword>          # 搜
sudo dnf info <pkg>                # 信息
sudo dnf list installed            # 已装
sudo dnf autoremove                # 删孤儿

特点:

  • 包稳定,但比 Ubuntu 略新(尤其 Fedora 是激进的)
  • 商业生态强:RHEL 在企业服务器上是事实标准
  • dnf 替代了老的 yum(2015 年起)

本系列不重点讲 dnf——命令和 apt 几乎对应,会 apt 的人切到 dnf 半天就熟。

六、Nix(跨平台) ​

Nix 是另一个范式——前面 4 个(brew / apt / pacman / dnf)都是"命令式 + 共享 /usr/bin",Nix 是"函数式 + 每个包独立 store + 不可变"。

6.1 心智差异 ​

传统包管理器(brew / apt):
   全局 /usr/bin 共享一个目录
   你装 python3.11,/usr/bin/python3 指向它
   你升 python3.13,/usr/bin/python3 现在指 3.13
   如果有项目锁了 3.11,坏了

Nix:
   /nix/store/<hash>-python-3.11.0/
   /nix/store/<hash>-python-3.13.0/
   两个版本同时存在,完全隔离
   你的项目说"我要 python 3.11",通过 hash 直接指向
   永远不变,永远不冲突

6.2 一行命令试用 Nix ​

bash
# 装 Nix(macOS / Linux 都支持)
sh <(curl -L https://nixos.org/nix/install)

# 临时进入一个有 ripgrep + fd 的 shell(不污染系统)
nix-shell -p ripgrep fd

# 在这个 shell 里:
which rg
# /nix/store/xxxx-ripgrep-13.0.0/bin/rg

# 退出 shell,这两个工具就"消失"了
# 系统 PATH 没变,你的 /usr/local 没新文件

这种"临时进入一个有 X / Y 的 shell" 是 Nix 最优雅的功能——项目级临时环境,完全可复现。

6.3 Nix 的复杂面 ​

Nix 全家桶:
   nix                  → 核心包管理器
   nixpkgs              → 巨大的包仓库(20 万+ 包)
   nix flakes           → 现代项目级配置
   home-manager         → 用户配置 + 包(22 篇讲过)
   NixOS                → 一个完整的 Linux 发行版,系统配置也声明式
   nix-darwin           → macOS 系统配置(类似 NixOS 的子集)

学习曲线:Nix 语言、derivation、flake、overlay、channels——每一个都是新概念,第一个月很痛。

25 篇专讲:本系列 25 篇会拆开 Nix 心智、flake 工作流、home-manager 实战——这一篇只点名:Nix 是"系统级包管理器的最终答案",但学习成本是 brew 的 10 倍。

6.4 现在要不要学 Nix ​

判断点:

你现在的痛是"每台机器配置不一致"           → chezmoi 解决 80%,Nix 解决 100%
你现在的痛是"开发环境 vs 生产不一致"        → devcontainer / Nix shell 都行
你现在的痛是"包升级偶尔坏掉,想原子化"      → Nix
你现在没痛,只是听说 Nix 很酷                → 别学,先把 chezmoi + brew 用好

ROI 角度:
  brew + chezmoi 学习成本:1 周,覆盖 80% 痛点
  Nix 学习成本:1-2 月,覆盖 100% 痛点
  → 大多数人,brew + chezmoi 够;硬核或团队规模化时,上 Nix

七、跨平台抽象:一份 install.sh 跑 macOS + Linux ​

实际项目里,你的 dotfiles / 团队脚本经常要同时在 macOS 和 Linux 上跑——用 uname 检测 OS,分支调用对应包管理器。

最小可用版:

bash
#!/usr/bin/env bash
# install-essentials.sh - 跨平台装基础工具

set -euo pipefail

OS="$(uname -s)"

install_macos() {
  # 装 brew
  if ! command -v brew &>/dev/null; then
    /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
    eval "$(/opt/homebrew/bin/brew shellenv)"
  fi

  brew install \
    git \
    neovim \
    tmux \
    ripgrep \
    fd \
    fzf \
    bat \
    eza \
    starship \
    mise \
    chezmoi

  # GUI / 字体
  brew install --cask \
    ghostty \
    font-jetbrains-mono-nerd-font
}

install_debian() {
  sudo apt update
  sudo apt install -y \
    git \
    tmux \
    ripgrep \
    fd-find \
    fzf \
    bat \
    curl \
    build-essential

  # 部分包名不同 / 没有,用其他方式
  # neovim 走 PPA 装新版
  if ! command -v nvim &>/dev/null; then
    sudo add-apt-repository -y ppa:neovim-ppa/unstable
    sudo apt update
    sudo apt install -y neovim
  fi

  # eza / starship / mise / chezmoi:apt 没有,走官方 install script
  curl -sS https://starship.rs/install.sh | sh -s -- -y
  curl https://mise.run | sh
  sh -c "$(curl -fsLS get.chezmoi.io)" -- -b "$HOME/.local/bin"

  # eza 走 cargo 或下载二进制
  curl -sSL https://github.com/eza-community/eza/releases/latest/download/eza_x86_64-unknown-linux-gnu.tar.gz \
    | sudo tar -xz -C /usr/local/bin
}

install_arch() {
  sudo pacman -Syu --noconfirm \
    git \
    neovim \
    tmux \
    ripgrep \
    fd \
    fzf \
    bat \
    eza \
    starship \
    mise \
    chezmoi
}

install_fedora() {
  sudo dnf install -y \
    git \
    neovim \
    tmux \
    ripgrep \
    fd-find \
    fzf \
    bat \
    eza \
    starship
  # mise / chezmoi 走官方
  curl https://mise.run | sh
  sh -c "$(curl -fsLS get.chezmoi.io)" -- -b "$HOME/.local/bin"
}

case "$OS" in
  Darwin)
    install_macos
    ;;
  Linux)
    if command -v apt &>/dev/null; then
      install_debian
    elif command -v pacman &>/dev/null; then
      install_arch
    elif command -v dnf &>/dev/null; then
      install_fedora
    else
      echo "Unsupported Linux distribution" >&2
      exit 1
    fi
    ;;
  *)
    echo "Unsupported OS: $OS" >&2
    exit 1
    ;;
esac

echo ""
echo "Essentials installed. Now run:"
echo "  chezmoi init --apply https://github.com/yourname/dotfiles"

几个工程要点:

  1. set -euo pipefail 是脚本工程化的开场咒(27 篇详讲)
  2. command -v xxx &>/dev/null 检测命令存在,比 which 跨 shell 兼容
  3. 包名差异要处理:Debian 上 fd 叫 fd-find、Fedora 上也叫 fd-find、Arch 上叫 fd
  4. 不能 brew bundle 的工具(eza / mise / starship 在 apt 上不存在),走官方 install 脚本
  5. PPA / 第三方源在脚本里能用,但注意维护——升级 OS 后要重新加

八、本地编译工具链(被忽视的前置) ​

很多包管理器装东西需要本地编译工具链——你装 brew / apt 都会拉一堆开发头文件,但这一步本身需要先有编译器。

8.1 macOS:Xcode CLT ​

bash
xcode-select --install

这一条命令装上:gcc / clang / make / git(老版本)/ headers 等。brew 在新机器上装的第一件事就是触发这个——但你脚本里可以提前触发,避免中途 GUI 弹窗。

自动化引导脚本里:

bash
# 检查 CLT 是否安装
if ! xcode-select -p &>/dev/null; then
  xcode-select --install
  # CLT 是 GUI 安装,要等用户点完
  echo "Please complete Xcode Command Line Tools installation, then re-run this script."
  exit 0
fi

8.2 Linux:build-essential / base-devel ​

bash
# Debian / Ubuntu
sudo apt install -y build-essential

# Arch
sudo pacman -S --needed base-devel

# Fedora
sudo dnf groupinstall "Development Tools"

这一组包含:gcc / g++ / make / autoconf / pkg-config / 等。任何要 make install 的工具都依赖这个。

8.3 容器 / 极简镜像 ​

Alpine / distroless 镜像默认没有编译器——你在 Docker 里装东西要注意:

dockerfile
# Alpine 装编译工具
RUN apk add --no-cache build-base

# Ubuntu 极简镜像
RUN apt-get update && apt-get install -y --no-install-recommends build-essential

九、GUI 应用怎么管 ​

macOS:
   brew install --cask <app>            ← 标准
   App Store(用 mas-cli 自动化)
   官网下载 .dmg / .pkg                  ← 兜底

Linux:
   apt install <app>                     ← 主仓有 GUI(Firefox / GIMP)
   flatpak install flathub <app>         ← 跨发行版的现代 GUI 分发
   snap install <app>                    ← Ubuntu 默认推,但社区有争议
   AppImage                              ← 单文件,双击运行
   官网 .deb / .rpm                      ← 兜底

Windows:(本系列不重点)
   winget                                 ← 微软官方
   scoop                                  ← 类似 brew
   chocolatey                             ← 老牌

macOS 的 brew cask 是 GUI 管理的标杆——brew install --cask 1password 一行搞定下载、解压、放进 /Applications、自动更新。Linux 的 flatpak 是最接近的对应,但没有任何工具像 brew 那样把 CLI 和 GUI 用一套包管理器统一——这是 macOS 工程环境的一大优势。

flatpak 例子:

bash
# 装 flatpak(Ubuntu 24.04+)
sudo apt install flatpak gnome-software-plugin-flatpak

# 加 Flathub 源
flatpak remote-add --if-not-exists flathub https://flathub.org/repo/flathub.flatpakrepo

# 装应用
flatpak install flathub org.mozilla.firefox
flatpak install flathub com.github.ghostty.Ghostty

flatpak 的优势:沙盒化、跨发行版;劣势:镜像大(每个 app 自带运行时)、启动比原生 deb 慢。

十、更新策略 ​

包升级是双刃剑——升了出风险,不升攒债。

10.1 macOS / brew ​

bash
# 频率建议
brew update                  # 每天 / 每周一次(更新索引,不升包)
brew outdated                # 看哪些待升
brew upgrade                 # 每周一次,选定时间升

# 不要盲升的
brew outdated --cask         # cask 大版本看 release notes
brew upgrade --cask <app>    # 选定升,不要一次性

典型坑:周五下午 brew upgrade 一把梭,周一发现 PostgreSQL 升了大版本,本地数据库挂了——别在下班前升。

10.2 Linux / apt ​

bash
sudo apt update              # 每天 / 每周一次
sudo apt list --upgradable   # 看哪些待升
sudo apt upgrade             # 每周一次

# 大版本升级(Ubuntu 22 → 24)
sudo do-release-upgrade      # 谨慎!备份再升

10.3 自动升级? ​

生产服务器:不要自动升级(意外重启 / 服务中断)。 个人开发机:可以自动 update 索引,但不要自动 upgrade 包。

macOS / brew 可以装 terminal-notifier + cron:

bash
# crontab -e
0 9 * * 1  brew update && brew outdated | terminal-notifier -title "Brew upgrades"

每周一上午 9 点提醒,不自动升。

十一、包冲突排查 ​

症状:你装了同名工具但走的不是你以为的那个。

11.1 经典:macOS 自带 vs brew 装的 ​

bash
# macOS 自带 git(老版本)
$ /usr/bin/git --version
git version 2.39.5 (Apple Git-154)

# brew 装的(新)
$ /opt/homebrew/bin/git --version
git version 2.45.2

# 你 shell 里 git 走哪个?看 PATH
$ which git
/opt/homebrew/bin/git   ← brew 在前

# 但有时 PATH 顺序错了:
$ echo $PATH
/usr/bin:/opt/homebrew/bin:...  ← /usr/bin 在前,走系统 git!

解法:~/.zshrc 或 ~/.zprofile 里把 brew 路径放前面:

zsh
eval "$(/opt/homebrew/bin/brew shellenv)"    # 这个会把 /opt/homebrew/bin 放最前

11.2 brew 同名包多 tap ​

bash
$ brew install postgresql
# 装了哪个版本?
$ brew info postgresql
# postgresql@16(默认 alias)

如果你同时装了 postgresql@15 和 postgresql@16,PATH 里只能有一个的 psql——用 brew unlink / brew link:

bash
brew unlink postgresql@16
brew link postgresql@15

或者用 mise 管多版本(24 篇)。

11.3 brew 链接错乱 ​

bash
brew doctor
# 会告诉你哪些文件不该在 /opt/homebrew 下、哪些 symlink 断了

brew doctor 是常规体检——每月跑一次,不要等出问题才查。

十二、反对的写法 ​

12.1 用 brew 装 Python / Node ​

bash
brew install python@3.13     # ✗ 不要
brew install node            # ✗ 不要
brew install ruby            # ✗ 不要

brew 把语言当"普通工具"管——但你需要多版本(项目 A 用 3.11、项目 B 用 3.13),brew 假设系统一个版本。结果:

  • brew 升级 Python,你的旧 venv 找不到 interpreter
  • npm install -g 装的全局工具,brew 升 node 时丢光
  • pip / gem 装的包不知道在哪,因为 brew 默认装的 Python 用 site-packages 在 brew 目录

解法:所有语言运行时走 mise(24 篇专讲):

bash
mise install python@3.11.7
mise install python@3.13.0
mise install node@20
mise install node@22
# 项目级:.tool-versions / .mise.toml 指定该项目用哪个

例外:Go 通常一个版本够,可以 brew install go——但严格起见,Go 也应该走 mise。

12.2 没有 Brewfile,手动装一堆 ​

bash
brew install git
brew install neovim
brew install tmux
# ... 装了 30 个,没记录

新机器:重头来,忘了哪个是哪个。

解法:装包后立即 brew bundle dump --force,或者反过来,所有装包都先改 Brewfile 再 brew bundle。

12.3 root 装东西(macOS / brew) ​

bash
sudo brew install x          # ✗ brew 拒绝

brew 设计就是无 sudo——/opt/homebrew 是当前用户拥有的目录,用 sudo 反而是反模式(brew 检测到会报错)。

12.4 root 装东西(Linux 个人开发机) ​

bash
sudo make install            # 装到 /usr/local
sudo curl ... | sudo bash    # 危险

问题:

  • 装到 /usr/local 后,你不知道装了哪些文件,卸载难
  • curl | sudo bash 把外部脚本以 root 权限跑,任何 typo / 恶意都灾难

解法:优先用包管理器(apt / pacman / brew);实在要源码装,装到 ~/.local/bin 或专用 /opt/myapp/(便于卸载)。

12.5 用 brew 装 docker / kubectl 但 docker 还要 Desktop ​

bash
brew install docker          # 只装了 docker CLI,没 daemon
# 你还得另装 Docker Desktop 才能跑容器

brew 的 docker formula 是 CLI only——容器运行时单独。macOS 上一般装 Docker Desktop / OrbStack / Colima。

bash
brew install --cask docker            # Docker Desktop 一站式
# 或
brew install orbstack                 # OrbStack(更轻的替代)
brew install colima                   # Colima(开源)

12.6 把 npm install -g 当包管理器用 ​

bash
npm install -g prettier
npm install -g typescript
npm install -g eslint

问题:

  • 全局包跟着 Node 版本走,升 Node 就丢
  • 项目级工具应该在 package.json,不是全局
  • 全局工具占 PATH,但版本难管

解法:项目工具放 devDependencies,真要全局(比如 gh-cli、跨项目的 cli)用 brew / mise。

12.7 brew upgrade 一年一次 ​

半年没升级,某天 brew upgrade
500 个包同时升级,各种工具行为变了
你也不知道哪个先坏的

解法:每周 / 每两周一次,少量增量升级风险小。

12.8 全部走 brew(Linux 上不用 apt) ​

bash
# Linux 服务器
sh <(curl ... linuxbrew install)
brew install git neovim tmux
# 全部走 brew,不用 apt

问题:Linuxbrew 在 Linux 上和系统级工具脱节,很多包要重编译(慢)、glibc 兼容偶尔出问题。

解法:Linux 上用发行版自带 + 必要时走 cargo / 官方安装脚本,不要全套迁 brew。

十三、Brewfile 工作流总结 ​

把 brew + Brewfile + chezmoi 串起来,这是 2026 macOS 工程化的标准姿势:

新机器引导 (bootstrap.sh 一行)
        │
        ├─ 装 Xcode CLT
        ├─ 装 brew
        ├─ 装 chezmoi
        └─ chezmoi init --apply yourname/dotfiles
                │
                ├─ 渲染配置文件到 $HOME
                ├─ 跑 run_once_install-packages.sh.tmpl
                │       └─ brew bundle --file=$HOME/.local/share/chezmoi/Brewfile
                └─ 完成

总时间:20-30 分钟,全自动

Brewfile 在 dotfiles 仓库里随版本演进:

2025-Q4 commit:  加了 starship、ripgrep
2026-Q1 commit:  替换 fzf-history 到 atuin
2026-Q2 commit:  加了 mise(从 nvm 迁移)

任何时候新机器都能复刻最新状态——这就是包管理工程化的复利。

十四、看完这一篇,你应该能 ​

  • 讲清楚三层包管理器架构——系统级 / 应用工具 / 语言级,不会再用 brew 装 Python
  • 在 macOS 上用 brew + Brewfile——把你的包清单 commit 进 dotfiles,新机器一键还原
  • 在 Linux 上选对包管理器——apt(Debian/Ubuntu)/ pacman(Arch)/ dnf(Fedora),知道哪个适合你的发行版
  • 写一份跨平台 install.sh——用 uname -s + command -v 检测,分支调用对应包管理器
  • 理解 Nix 是什么 / 什么时候上——不是替代 brew,而是"完全可复现"的最终答案
  • 会处理 GUI 应用——macOS 走 brew install --cask,Linux 走 flatpak / 包管理器
  • 会排查包冲突——PATH 顺序、brew doctor、which xxx
  • 看到一个"用 brew 装 Python"的教程,第一反应是"这是反模式"

14.1 自查清单 ​

读完这一篇,做一遍:

□ 跑 `brew list | grep -E "python|node|ruby"`,如果有,准备迁到 mise(24 篇)
□ 跑 `brew bundle dump --force --file=~/Brewfile`,导出当前包清单
□ 看一遍 Brewfile,删掉你这半年没用过的(/Applications 里看哪个 cask 一次没点开)
□ 把 Brewfile commit 到 dotfiles 仓库(chezmoi cd 之后 git add)
□ 找一台备用机 / VM,跑 `brew bundle --file=~/Brewfile`,验证能复刻
□ 跑 `brew doctor`,处理所有 warning
□ 看一眼 `echo $PATH`,确认 /opt/homebrew/bin 在 /usr/bin 之前
□ 如果在 Linux 服务器上,确认你只用 apt / pacman,没有装 linuxbrew

做完这 8 条,你的包管理工程化就过线了——剩下的是 24 篇的多版本管理。

包管理器装"基础设施",但Node / Python / Ruby / Go 这些语言运行时是另一个工程——下一篇拆开来讲:
