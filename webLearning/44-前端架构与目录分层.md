# 前端架构与目录分层

项目变乱通常不是因为 React / Vue 不行,而是因为**代码没有边界**。

一句话先记住:**前端架构 = 把页面、业务、数据、状态、组件和基础设施分开放,让改一个需求时只动该动的地方。**

---

## 一、最常见的坏结构

```text
src/
  components/
    UserTable.tsx
    OrderModal.tsx
    ProductForm.tsx
  pages/
    UserPage.tsx
    OrderPage.tsx
  api.ts
  store.ts
  utils.ts
```

看起来简单,但项目一大就出问题:

- `components/` 变成垃圾桶,里面既有按钮,又有业务表格
- `api.ts` 越写越长,所有接口混在一起
- `store.ts` 什么都管,登录、筛选、弹窗、缓存全塞进去
- 页面直接拼接口字段,接口一改全站炸
- 复用靠复制粘贴,不是靠边界

**文件夹不是架构。边界才是架构。**

---

## 二、推荐结构

中大型业务前端推荐按 feature 分:

```text
src/
  app/
    router.tsx
    providers.tsx
    layout.tsx

  shared/
    ui/
      button.tsx
      dialog.tsx
      table.tsx
    lib/
      date.ts
      money.ts
      request.ts
    config/
      env.ts

  entities/
    user/
      model.ts
      api.ts
      format.ts
    order/
      model.ts
      api.ts
      status.ts

  features/
    auth/
      login-form.tsx
      use-login.ts
    user-search/
      user-filter.tsx
      user-query.ts
    order-approval/
      approval-dialog.tsx
      use-approval.ts

  pages/
    users/
      page.tsx
    orders/
      page.tsx
```

这不是要你照抄目录名,而是记住四层:

```text
shared    通用工具和基础 UI,不懂业务
entities  业务实体:User / Order / Product
features  一个可执行业务动作:登录、搜索、审批、上传
pages     路由页面,负责组装
```

---

## 三、依赖方向

必须单向:

```text
pages -> features -> entities -> shared
```

允许:

```ts
// pages/users/page.tsx
import { UserFilter } from '@/features/user-search/user-filter';
import { UserTable } from '@/entities/user/user-table';
```

禁止:

```ts
// shared/ui/button.tsx
import { useUserStore } from '@/features/user-search/store'; // 错
```

越底层越不能知道上层。`shared` 不能知道 `user`, `entities` 不能知道某个页面。

---

## 四、页面怎么写

页面只做编排:

```tsx
export function UsersPage() {
  return (
    <PageShell title="Users">
      <UserFilter />
      <UserTable />
    </PageShell>
  );
}
```

不要在页面里塞一堆请求、表单、弹窗、权限判断:

```tsx
// 坏味道:页面变成业务垃圾场
export function UsersPage() {
  const [keyword, setKeyword] = useState('');
  const [selected, setSelected] = useState<string[]>([]);
  const { data } = useQuery(...);
  const mutation = useMutation(...);
  const canDelete = currentUser.roles.includes('admin');

  return <div>{/* 500 行 */}</div>;
}
```

页面超过 200 行,通常说明 feature 没拆出来。

## 五、实体层怎么写

实体层放稳定业务概念。

```ts
// entities/user/model.ts
export type User = {
  id: string;
  name: string;
  email: string;
  role: UserRole;
  status: UserStatus;
  createdAt: string;
};

export type UserRole = 'owner' | 'admin' | 'member';
export type UserStatus = 'active' | 'disabled';
```

```ts
// entities/user/format.ts
export function formatUserRole(role: UserRole) {
  const map: Record<UserRole, string> = {
    owner: 'Owner',
    admin: 'Admin',
    member: 'Member',
  };
  return map[role];
}
```

实体层不要放页面状态:

```ts
// 不该放 entities/user
type UserTableSort = 'created-desc' | 'created-asc';
type UserFilterForm = { keyword: string };
```

这些属于 feature 或 page。

## 六、Feature 怎么拆

feature 是一个用户动作:

```text
login
create-user
user-search
order-approval
file-upload
```

一个 feature 里可以有 UI、hook、schema、状态:

```text
features/user-search/
  user-filter.tsx
  user-query.ts
  schema.ts
  url-state.ts
```

示例:

```ts
// features/user-search/schema.ts
import { z } from 'zod';

export const userSearchSchema = z.object({
  keyword: z.string().trim().optional(),
  role: z.enum(['owner', 'admin', 'member']).optional(),
  page: z.coerce.number().int().positive().default(1),
});

export type UserSearchInput = z.infer<typeof userSearchSchema>;
```

```ts
// features/user-search/user-query.ts
import { useQuery } from '@tanstack/react-query';
import { listUsers } from '@/entities/user/api';
import type { UserSearchInput } from './schema';

export function useUserSearch(input: UserSearchInput) {
  return useQuery({
    queryKey: ['users', input],
    queryFn: () => listUsers(input),
  });
}
```

## 七、Shared 层放什么

`shared` 必须业务无关:

```text
shared/ui       Button / Dialog / Table / Input
shared/lib      request / date / money / storage
shared/hooks    useDebounce / useMediaQuery
shared/config   env / constants
```

判断标准:

```text
换一个项目还能用 -> shared
只在当前业务有意义 -> entities / features
```

`MoneyText` 可以进 shared:

```tsx
export function MoneyText({ cents }: { cents: number }) {
  return <span>{(cents / 100).toFixed(2)}</span>;
}
```

`OrderMoneyText` 不该进 shared:

```tsx
// 订单优惠、退款、币种规则是业务规则
export function OrderMoneyText({ order }: { order: Order }) {
  return <span>{calculatePayable(order)}</span>;
}
```

## 八、状态放哪里

先分类:

| 状态 | 放哪里 |
| --- | --- |
| 服务端数据 | TanStack Query / SWR |
| URL 查询参数 | Router / search params |
| 表单输入 | React Hook Form / Vue form |
| 单页面 UI 状态 | 组件内部 `useState` / `ref` |
| 跨页面客户端状态 | Zustand / Pinia |
| 权限、用户会话 | auth feature + query/cache |

最常见错误:

```ts
// 错:把 API 数据塞全局 store
const useUserStore = create((set) => ({
  users: [],
  fetchUsers: async () => set({ users: await api.listUsers() }),
}));
```

正确:

```ts
const users = useQuery({
  queryKey: ['users', params],
  queryFn: () => listUsers(params),
});
```

API 数据有缓存、过期、重试、失效、乐观更新。全局 store 不该手写这些。

## 九、路径别名

配置别名,不要写相对路径地狱:

```ts
// vite.config.ts
import path from 'node:path';

export default defineConfig({
  resolve: {
    alias: {
      '@': path.resolve(__dirname, 'src'),
    },
  },
});
```

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}
```

然后:

```ts
import { Button } from '@/shared/ui/button';
import { User } from '@/entities/user/model';
```

## 十、什么时候不要过度分层

小项目不要一开始就建 40 个目录。

```text
1-3 个页面:
  src/pages + src/components + src/lib 就够

10+ 个页面 / 3+ 人协作:
  开始按 feature 拆

多个业务域 / 多团队:
  entities + features + shared 值得上
```

架构是为变化服务的。没有变化点,不要提前抽象。

## 十一、检查清单

- [ ] 页面文件是否只做组装
- [ ] API 调用是否集中在 entity / feature
- [ ] shared 是否业务无关
- [ ] 服务端数据是否交给 TanStack Query / SWR
- [ ] URL 状态是否真的放进 URL
- [ ] feature 是否代表一个用户动作
- [ ] 是否有从底层 import 上层的反向依赖
- [ ] 目录名是否表达业务,不是表达技术癖好

---

## 十二、心智模型

```text
pages     路由入口,负责组装
features  用户动作,负责业务流程
entities  业务名词,负责稳定规则
shared    通用基础设施,不懂业务
```

项目变大以后,真正救命的不是多会几个 hook,而是**每段代码知道自己属于哪里**。
