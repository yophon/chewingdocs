# 认证与安全:JWT / Session / OAuth、XSS / CSRF / CSP

Web 安全两大主题:

```
1. 认证(Authentication):你是谁?     → Session / JWT / OAuth
2. 授权(Authorization):你能做什么?   → RBAC / Scopes
3. 防御:防止别人冒充 / 攻击你的网站   → XSS / CSRF / CSP / SQLi
```

这一篇按"认证机制 → 攻击与防御"两条线,把生产实战要懂的部分讲清楚。

---

## 一、认证机制三大类

```
1. Session-based      传统 Web,服务端存状态
2. JWT (Token-based)  现代 SPA / API,客户端拿 token
3. OAuth 2.0          第三方登录(GitHub / Google)
```

### 1. Session(传统)

流程:

```
1. 用户登录 → 服务端校验 → 创建 session,存 sessionId 在数据库 / Redis
2. 服务端把 sessionId 写到 Cookie 返给浏览器
3. 浏览器之后每次请求自动带 Cookie
4. 服务端用 sessionId 查 session 数据 → 知道你是谁
```

```js
// 登录响应
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=86400
```

**优点**:
- 服务端可控(踢人 / 改权限立即生效)
- 安全(HttpOnly cookie,JS 偷不到)

**缺点**:
- 服务端要存 session(Redis / DB)
- 横向扩展时所有节点要共享 session 存储
- 跨域复杂

**适合**:传统服务端渲染应用、企业内部系统、需要"立即踢人"的场景。

### 2. JWT(JSON Web Token)

格式:

```
xxxxx.yyyyy.zzzzz
↓     ↓     ↓
header.payload.signature

base64(header) . base64(payload) . HMAC(header+payload, secret)
```

解码后:

```json
// header
{ "alg": "HS256", "typ": "JWT" }

// payload(自己定义)
{ "sub": "user123", "name": "Alice", "exp": 1735000000 }
```

**关键性质**:
- **不加密,只签名**。Payload 可以被任何人 base64 解码看到!
- 服务端只需保存 secret,**不需要存 session**
- 自包含:token 本身就携带用户信息
- 一旦签发,服务端**无法主动失效**(到期前都有效)

**适合**:
- 无状态 API
- 微服务之间通信
- 移动 App
- 跨域 / 多端共用

**不适合**:
- 需要立即踢人
- 高安全场景
- payload 大(每次请求带回去,流量贵)

#### JWT 实战

```js
// 服务端签发
import jwt from 'jsonwebtoken';

const token = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '7d' }
);

// 客户端存(下面有讨论)
localStorage.setItem('token', token);

// 每次请求带上
fetch('/api/me', {
  headers: { Authorization: `Bearer ${token}` }
});

// 服务端验证
const payload = jwt.verify(token, process.env.JWT_SECRET);
// payload.sub, payload.role
```

#### Access Token + Refresh Token(行业标准)

单纯用 JWT 有矛盾:
- 时间长 → 不安全(泄露后能用很久)
- 时间短 → 用户体验差(频繁登录)

解决:**双 token 机制**

```
Access Token   : 15 分钟,JWT,带在每次请求里
Refresh Token  : 30 天,存数据库,只用于换新 access token
```

流程:
```
1. 登录 → 服务端返回 (access, refresh)
2. 客户端用 access 调 API
3. access 过期 → 用 refresh 调 /refresh → 拿新 access
4. refresh 过期 / 被吊销 → 重新登录
5. 用户登出 → 服务端把 refresh 列入黑名单
```

#### JWT 存哪?(经典争议)

```
localStorage    : XSS 一中招就被偷,但实现简单
HttpOnly Cookie : XSS 偷不到,但要防 CSRF
内存(变量)    : 最安全,但刷新就没了
```

**生产推荐**:
- Access token → **内存或 HttpOnly Cookie**
- Refresh token → **HttpOnly + Secure + SameSite Cookie**

不要 localStorage 存敏感 token,这是 2024+ 共识。

### 3. OAuth 2.0(第三方登录)

不是"认证协议",是**授权协议**(让 App A 代表用户访问 App B 的资源)。但被广泛误用为登录,因此有了 **OpenID Connect**(基于 OAuth 的标准认证协议)。

#### Authorization Code Flow(最常见)

```
1. 用户点击"用 GitHub 登录"
2. 你的网站 redirect 到 https://github.com/login/oauth/authorize?client_id=xxx&redirect_uri=https://your.app/callback&scope=user:email
3. 用户在 GitHub 登录 + 授权
4. GitHub redirect 回 https://your.app/callback?code=AUTHCODE
5. 你的服务端用 code 调 https://github.com/login/oauth/access_token 换 access_token
6. 用 access_token 调 https://api.github.com/user 拿用户信息
7. 在你自己的系统里创建 session / JWT
```

**关键安全点**:
- `redirect_uri` 必须**预先注册**(防钓鱼)
- `state` 参数防 CSRF
- `code` 一次性,只能换一次 token
- 换 token 要 client_secret(在服务端,不在前端)

#### PKCE(SPA 必须)

SPA 没有"安全地存 client_secret"的地方。**PKCE**(Proof Key for Code Exchange)解决:

```
1. 客户端生成 code_verifier(随机字符串)
2. 算 code_challenge = SHA256(code_verifier)
3. 跳转时带 code_challenge
4. 换 token 时带 code_verifier
5. 服务端 SHA256(code_verifier) == code_challenge?  → 验证
```

不需要 client_secret,但仍然安全。**SPA / 移动 App 的 OAuth 必须 PKCE**。

#### 实战:用现成的库

不要自己实现 OAuth。用:
- **Auth.js**(原 NextAuth.js):React / Next.js / SvelteKit
- **Clerk**(SaaS):一行接入,有 UI 组件,适合不想自己搞的
- **Lucia Auth**:轻量,自建后端用
- **Supabase Auth** / **Firebase Auth**:接整套

```ts
// Auth.js 示例
import NextAuth from 'next-auth';
import GitHub from 'next-auth/providers/github';

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [GitHub],
});
```

**几行代码搞定 GitHub 登录**。

## 二、Cookie 安全属性(必须懂)

```
Set-Cookie: token=xxx; HttpOnly; Secure; SameSite=Lax; Path=/; Max-Age=86400
```

| 属性 | 作用 |
| --- | --- |
| **HttpOnly** | JS 无法读取(`document.cookie` 看不到)→ 防 XSS 偷 cookie |
| **Secure** | 只在 HTTPS 发送 |
| **SameSite=Strict** | 跨站点请求**完全不带** cookie → 强防 CSRF |
| **SameSite=Lax** | 顶级导航(GET)带,其他不带 → 默认值,平衡 |
| **SameSite=None** | 跨站都带,**必须配 Secure** → 跨域 SSO 用 |
| **Path** | 只发给特定路径 |
| **Domain** | 共享给子域名 |
| **Max-Age / Expires** | 过期时间 |

**生产 cookie 默认必加**:`HttpOnly; Secure; SameSite=Lax`

## 三、Web 安全攻击 Top 3

### 1. XSS(Cross-Site Scripting)

**原理**:别人把 JS 代码注入到你的页面,然后这段 JS 在用户浏览器里跑。

```html
<!-- 用户输入了:<script>alert(document.cookie)</script> -->
<div>{userInput}</div>
```

如果直接渲染,JS 会执行 → 偷 cookie / 改 DOM / 钓鱼。

#### 三种 XSS

```
反射型(Reflected) : 攻击代码在 URL,通过链接传播
存储型(Stored)    : 攻击代码存数据库,所有看到的人中招(最严重)
DOM 型             : 完全在前端,如 location.hash 直接 innerHTML
```

#### 防御

```jsx
// React 默认转义,安全
<div>{userInput}</div>      // ✅ '<script>' 被转义成 '&lt;script&gt;'

// 危险:直接插 HTML
<div dangerouslySetInnerHTML={{ __html: userInput }} />   // ❌
```

如果必须渲染 HTML(富文本编辑器):**先用 DOMPurify 清洗**:

```js
import DOMPurify from 'dompurify';
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(userInput) }} />
```

服务端也要清洗。**不信任任何用户输入**,无论前端还是后端。

#### 关键 cookie 加 HttpOnly

```
Set-Cookie: sessionId=xxx; HttpOnly
```

即使有 XSS,JS 也偷不到这个 cookie。**这是 XSS 后的最后防线**。

---

### 2. CSRF(Cross-Site Request Forgery)

**原理**:用户在 A 网站登录了,B 网站偷偷代表用户向 A 发请求。

```html
<!-- 用户在 evil.com,evil.com 上有: -->
<img src="https://your-bank.com/transfer?to=hacker&amount=10000" />
<!-- 浏览器自动带上 your-bank.com 的 cookie 发请求 -->
<!-- 你银行账户的钱被转走 -->
```

**前提**:
- 用户在 your-bank.com 已登录(cookie 在)
- 浏览器跨站发请求自动带 cookie

#### 防御:四道防线

##### A. SameSite Cookie(2020+ 默认)

```
Set-Cookie: sessionId=xxx; SameSite=Lax
```

跨站发请求**不会带这个 cookie**。**这是现代浏览器的默认防御**。

##### B. CSRF Token

```html
<form>
  <input type="hidden" name="csrf_token" value="<%= randomToken %>">
  ...
</form>
```

服务端给每个 session 一个随机 token,提交时校验。攻击者拿不到 token 就发不了请求。

##### C. 校验 Origin / Referer 头

```js
if (req.headers.origin !== 'https://your-bank.com') {
  return res.status(403).end();
}
```

##### D. 关键操作再认证

转账、改密码这种,**让用户重新输密码**。多一道防线。

#### JWT(在 Authorization 头)天然防 CSRF

```js
fetch('/api', {
  headers: { Authorization: `Bearer ${token}` }
});
```

token 不是 cookie,跨站发请求**不会被自动带上**(攻击者拿不到 token)。**这是 JWT 的一个安全优势**。

---

### 3. SQL 注入(后端)

```sql
-- 用户输入:' OR 1=1 --
SELECT * FROM users WHERE name = '<input>'
-- 拼出来:SELECT * FROM users WHERE name = '' OR 1=1 --'
-- 返回所有用户
```

#### 防御

**永远用参数化查询**,不要拼字符串:

```js
// ❌ 拼接(危险)
db.query(`SELECT * FROM users WHERE name = '${name}'`);

// ✅ 参数化(安全)
db.query('SELECT * FROM users WHERE name = ?', [name]);

// ORM 默认参数化
prisma.user.findMany({ where: { name } });
```

ORM(Prisma / Drizzle)默认安全,**不要绕过去手写 raw SQL**。

## 四、CSP(Content Security Policy)

CSP 是浏览器的"白名单":告诉浏览器**只信任来自这些源的资源**。

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.jsdelivr.net; style-src 'self' 'unsafe-inline';
```

```
default-src 'self'              默认只加载本域资源
script-src 'self' cdn.xxx       JS 只允许本域 + 指定 CDN
style-src 'self' 'unsafe-inline' CSS 允许本域 + 内联
img-src *                       图片任意源
```

**好处**:即使有 XSS,攻击者注入的 `<script src="evil.com/x.js">` 也会被 CSP 拦截,**不能加载外部脚本**。

**坑**:`'unsafe-inline'` 'unsafe-eval' 几乎让 CSP 失效,要尽量去掉。React/Vue 一般不需要 inline script,Vite 构建后默认是外部 JS。

## 五、HTTPS 与 HSTS

```
HTTP   : 明文传输,中间人能看 / 篡改
HTTPS  : TLS 加密,看不到内容,但能被降级(中间人让你走 HTTP)
HSTS   : 浏览器记住"这站只走 HTTPS,以后强制",防降级
```

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

**生产网站必开 HTTPS + HSTS**。Cloudflare / Vercel 默认开。

## 六、密码存储(后端必懂)

### 永远不要明文存密码

```js
// ❌ 万分之一最常见的事故
db.insert({ password: req.body.password });
```

### Hash 也不够

```js
// ❌ 普通 hash,彩虹表攻击
db.insert({ password: md5(password) });
db.insert({ password: sha256(password) });
```

### 用 bcrypt / argon2

```js
import bcrypt from 'bcrypt';

const hash = await bcrypt.hash(password, 12);   // 12 = cost factor
db.insert({ password: hash });

// 验证
const ok = await bcrypt.compare(input, hash);
```

bcrypt / argon2 内置 salt + 慢哈希,**专门为密码设计**。argon2 是 2024+ 推荐(更抗 GPU 攻击)。

## 七、HTTP 安全头(开箱即用)

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY                              ← 防点击劫持
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), camera=()      ← 限制 API 权限
```

Express + Helmet 一行搞定:

```js
import helmet from 'helmet';
app.use(helmet());
```

Next.js 在 `next.config.js` 配 `headers()`,Cloudflare / Vercel 也能在控制台配。

## 八、CORS(跨域)

CORS **不是安全机制**,是浏览器**保护用户的同源策略**。后端正确配置就能跨域。

```http
Access-Control-Allow-Origin: https://your-app.com
Access-Control-Allow-Credentials: true
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
```

```js
// Express
import cors from 'cors';
app.use(cors({
  origin: 'https://your-app.com',
  credentials: true,
}));
```

```js
// Hono
import { cors } from 'hono/cors';
app.use('/*', cors({ origin: 'https://your-app.com', credentials: true }));
```

**不要 `Access-Control-Allow-Origin: *` + `credentials: true`**,浏览器会拒绝,**也是安全漏洞**(任意网站都能带 cookie 调你 API)。

## 九、限流(Rate Limiting)

```
登录接口:每 IP 5 次 / 分钟       防爆破
注册:每 IP 3 次 / 小时           防垃圾账号
API:每用户 100 次 / 分钟          防滥用
```

```js
// Express
import rateLimit from 'express-rate-limit';

app.post('/login',
  rateLimit({ windowMs: 60_000, max: 5 }),
  loginHandler
);
```

**生产服务必装**。Cloudflare / Vercel 也有 WAF 一层。

## 十、常见 Trap

### Trap 1:JWT 算法 `none`

旧版库默认接受 `alg: none`,攻击者可以**自己签发任意 token**。修复:

```js
// 显式指定算法
jwt.verify(token, secret, { algorithms: ['HS256'] });
```

### Trap 2:OAuth state 漏写

不写 `state` 参数 → CSRF 攻击者可以让你"登录"成他们的账号(把别人的活动绑你账户)。**永远要 state**。

### Trap 3:JWT secret 太弱

```js
const secret = 'secret123';                          // ❌
const secret = process.env.JWT_SECRET;               // ✅(64 位随机)
```

```bash
# 生成强 secret
openssl rand -base64 64
```

### Trap 4:暴露详细错误信息

```js
// ❌
catch (e) {
  res.status(500).json({ error: e.stack });    // 暴露代码结构 / 数据库表名
}

// ✅
catch (e) {
  log.error(e);
  res.status(500).json({ error: 'Internal error' });
}
```

### Trap 5:依赖漏洞

```bash
pnpm audit
# 看看依赖里有没有已知漏洞
```

CI 里加 `pnpm audit --audit-level=high`,有高危漏洞拒绝合并。

### Trap 6:把秘钥写进前端代码

```js
const API_KEY = 'sk-xxxxxx';        // ❌ 用户能 view source 看到
```

任何 `VITE_` / `NEXT_PUBLIC_` 前缀的环境变量都会**暴露给浏览器**。秘钥永远在服务端,前端只调你自己的 API。

## 十一、生产 checklist(收藏)

### 认证

- [ ] 密码用 bcrypt / argon2,cost ≥ 12
- [ ] Session / JWT 有合理过期时间
- [ ] Access + Refresh token 分离
- [ ] Refresh token 在 HttpOnly Cookie 里
- [ ] 第三方登录走 OAuth + PKCE
- [ ] 登出时清服务端 session / refresh token

### Cookie

- [ ] 关键 cookie 加 `HttpOnly; Secure; SameSite=Lax`
- [ ] 跨站登录用 `SameSite=None; Secure`

### XSS / CSRF

- [ ] 用户输入永远转义(React/Vue 默认)
- [ ] 富文本用 DOMPurify
- [ ] 配 CSP,逐步去掉 `unsafe-inline`
- [ ] 状态变更接口校验 CSRF token 或用 JWT 头

### HTTP

- [ ] 全站 HTTPS + HSTS
- [ ] 加 helmet / 安全头
- [ ] CORS 白名单具体源,不用 *
- [ ] 关键接口限流

### 后端

- [ ] 用 ORM 或参数化查询(防 SQLi)
- [ ] 不暴露详细错误
- [ ] 日志不打密码 / token
- [ ] 依赖每周 audit

## 十二、心智模型

```
认证三选一:
  服务端有状态 → Session
  无状态 / 多端 → JWT(access + refresh + HttpOnly cookie)
  第三方登录 → OAuth 2.0 + PKCE

防御三大攻击:
  XSS  → 转义输入 + CSP + HttpOnly cookie
  CSRF → SameSite cookie + CSRF token / JWT 头
  SQLi → 参数化查询 / ORM

底线:
  HTTPS / HSTS / 安全头 / 限流 / 密码 hash
  不存敏感数据在 localStorage
  不暴露错误堆栈
  依赖定期 audit
```
