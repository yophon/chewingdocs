# 微前端架构

微前端 = **把一个大应用拆成多个独立子应用,每个子应用可独立开发 / 部署 / 技术栈**。

```
单体前端(传统)              微前端
所有团队改一个仓库          每个团队独立仓库 / 部署
统一技术栈                  React / Vue / Angular 共存
一处出错全站挂              子应用挂只挂一块
发布 = 全量发布              每个子应用独立发布
```

跟微服务的关系:**微前端是微服务的前端版本**,本质是"分而治之 + 团队自治"。

---

## 一、什么时候需要微前端

```
✅ 需要
  - 多个独立团队,各自负责不同业务模块
  - 老应用迁移(不能一次重写,要逐步替换)
  - 收购了别家产品要集成进去
  - 不同模块技术栈不同(老 Angular + 新 React)
  - 模块独立部署需求强(订单 / 商品 / 用户各自上线)

❌ 不需要
  - 中小项目,一个团队
  - 技术栈统一,模块界限不清
  - 没有独立部署需求
```

**警告**:微前端是"组织架构问题"的解决方案,**不要为了用而用**。多数项目用 monorepo(33 篇)就够了,不需要微前端。

---

## 二、微前端的几种实现思路

```
1. iframe         最老,最稳定,样式隔离最好
2. 路由分发        Nginx 按 URL 转发不同 SPA
3. Web Components  44 篇讲过,跨框架天然
4. Module Federation Webpack 5 / Rspack 方案,动态共享模块
5. qiankun        蚂蚁开源,基于 single-spa,生命周期托管
6. Wujie          腾讯开源,iframe + WebComponent 混合
7. Native Federation Angular / Vite 的 Federation 实现
```

### iframe(最简但有限)

```html
<iframe src="https://app1.example.com" width="100%"></iframe>
<iframe src="https://app2.example.com" width="100%"></iframe>
```

```
优点:完美隔离(JS / CSS 互不干扰),零配置
缺点:
  - 通信麻烦(postMessage)
  - URL 不统一(刷新丢状态)
  - 性能差(每个 iframe 独立 runtime)
  - 兼容老登录 / cookie 同源问题
```

**简单场景 iframe 够用**,别一上来就上重型框架。

### 路由分发(Nginx)

```nginx
location /app1/ { proxy_pass https://app1.com/; }
location /app2/ { proxy_pass https://app2.com/; }
```

不同 URL 是完全独立的 SPA。**没有真正的"集成"**,但简单。

## 三、Module Federation(主流方案)

Webpack 5 / Rspack / Vite 都支持。**核心:子应用把模块"暴露",主应用动态加载**。

### 1. 概念

```
Host(主应用)
  ↓ runtime 请求
Remote(子应用)的模块
  ↓ 浏览器加载子应用 JS
  ↓ 注册到主应用
  ↓ 渲染
```

子应用 export 一些组件 / 函数,主应用直接 `import` 用。**底层是 SystemJS / 动态 ESM**。

### 2. 子应用配置(暴露)

```ts
// remote-app/vite.config.ts
import federation from '@originjs/vite-plugin-federation';

export default {
  plugins: [
    federation({
      name: 'remote_app',
      filename: 'remoteEntry.js',
      exposes: {
        './Button': './src/Button.tsx',
        './App': './src/App.tsx',
      },
      shared: ['react', 'react-dom'],
    }),
  ],
  build: { target: 'esnext', minify: false, cssCodeSplit: false },
};
```

build 后产生 `dist/assets/remoteEntry.js`,部署到 CDN。

### 3. 主应用配置(消费)

```ts
// host-app/vite.config.ts
import federation from '@originjs/vite-plugin-federation';

export default {
  plugins: [
    federation({
      name: 'host_app',
      remotes: {
        remote_app: 'https://remote.example.com/assets/remoteEntry.js',
      },
      shared: ['react', 'react-dom'],
    }),
  ],
};
```

```tsx
// host-app/src/App.tsx
import React, { lazy, Suspense } from 'react';

const RemoteButton = lazy(() => import('remote_app/Button'));

function App() {
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <RemoteButton />
    </Suspense>
  );
}
```

**主应用动态从远程拉子应用的 Button 渲染**。子应用更新部署,主应用刷新就拿到新版。

### 4. shared 是关键

```ts
shared: ['react', 'react-dom']
```

主应用和子应用都用 React,**只加载一份**(否则两份 React 会出 hooks 错误)。Federation 会智能选最高版本兼容的那份。

### 5. 适用

- 多团队,**同技术栈**(都是 React 或都是 Vue)
- 需要运行时共享代码(共享组件库 / utility)
- 子应用独立部署,主应用不用重 build

## 四、qiankun(国内主流)

蚂蚁开源,基于 [single-spa](https://single-spa.js.org)。**生命周期托管 + 沙箱隔离**。

### 1. 概念

```
主应用(基座)
  ↓ 注册子应用
qiankun  → 监听路由变化
  ↓ 匹配到某子应用
qiankun  → 加载子应用 HTML/JS/CSS
  ↓ 调用子应用的 mount
子应用渲染到指定容器
```

### 2. 主应用

```ts
import { registerMicroApps, start } from 'qiankun';

registerMicroApps([
  {
    name: 'app1',
    entry: '//localhost:7100',
    container: '#sub-app',
    activeRule: '/app1',
  },
  {
    name: 'app2',
    entry: '//localhost:7200',
    container: '#sub-app',
    activeRule: '/app2',
  },
]);

start();
```

```html
<a href="/app1">App 1</a>
<a href="/app2">App 2</a>
<div id="sub-app"></div>
```

URL 切换 → qiankun 卸载旧子应用 + 加载新子应用。

### 3. 子应用

```ts
// 子应用入口
let app: any;

export async function bootstrap() {
  console.log('app bootstrap');
}

export async function mount(props: any) {
  app = createApp(App).mount(props.container);
}

export async function unmount() {
  app.unmount();
}
```

子应用要提供 `bootstrap / mount / unmount` 生命周期。Vite / Webpack 配 library mode 输出 UMD。

### 4. 沙箱

qiankun 有 **JS 沙箱**(代理 window,记录子应用对全局的修改,卸载时恢复)+ **CSS 沙箱**(scoped 或 shadow DOM)。

### 5. 通信

```ts
// 主应用初始化
import { initGlobalState } from 'qiankun';

const actions = initGlobalState({ user: 'Alice' });

// 子应用
props.onGlobalStateChange((state) => console.log(state));
props.setGlobalState({ user: 'Bob' });
```

**全局状态 + 事件总线**模式,跨技术栈通用。

### 6. 适用

- 多技术栈混合(老 Angular + 新 React + Vue)
- 需要严格沙箱
- 国内项目,中文文档丰富

## 五、Wujie(无界,腾讯)

```
qiankun:JS 模拟沙箱(可能漏)
Wujie:iframe + Web Components(原生隔离 + 通信)
```

```ts
import WujieReact from 'wujie-react';

<WujieReact
  width="100%"
  height="100%"
  name="app1"
  url="https://app1.example.com"
  sync={true}
  props={{ user }}
/>
```

特点:
- 利用 iframe 的隔离 + Shadow DOM 让样式不冲突
- 子应用切换不重载(预激活)
- 性能比 qiankun 好

新项目国内可以考虑 Wujie。

## 六、single-spa

最早的微前端框架(2017),**qiankun 基于它**。直接用比较底层,**国内一般用 qiankun 封装版**。

```ts
import { registerApplication, start } from 'single-spa';

registerApplication({
  name: 'app1',
  app: () => import('./app1'),
  activeWhen: '/app1',
});

start();
```

## 七、Native Federation / Vite Plugin Federation

Vite / esbuild 时代,Webpack 的 Module Federation 在迁移到原生 ESM:

- [`@originjs/vite-plugin-federation`](https://github.com/originjs/vite-plugin-federation):Vite 的 Federation
- [Native Federation](https://github.com/angular-architects/module-federation-plugin):Angular / 框架无关
- [Module Federation 2.0](https://module-federation.io/):2024 重构版,支持 Rspack / Webpack / Vite

**2025 推荐方案**:**Module Federation 2.0**(跨构建工具)。

## 八、跨子应用通信

```
方案
1. URL / 路由           最简单,可分享
2. localStorage / sessionStorage  适合状态共享(不实时)
3. Custom Event         同源可用
4. postMessage          跨 iframe / 跨域
5. 全局状态库            主应用提供 store,子应用订阅
6. 事件总线 / Pub-Sub    qiankun / Wujie 内置
```

**优先级**:
1. 能用 URL 就 URL(可分享 / 可后退)
2. 否则用主应用注入的 store / API

```ts
// 主应用提供 API
window.__shared_api__ = {
  getUser: () => state.user,
  navigate: (url) => router.push(url),
};

// 子应用消费
const user = window.__shared_api__.getUser();
```

## 九、样式隔离

### 方案对比

```
1. CSS Modules / Scoped CSS    各子应用各自处理,默认隔离
2. CSS-in-JS                    天然隔离
3. BEM 命名前缀                 .app1-btn / .app2-btn,人工保证
4. Shadow DOM                   终极隔离(浏览器原生)
5. Iframe                       最强隔离
```

实战:每个子应用 CSS 加唯一前缀(qiankun 自动加 `data-qiankun="appName"` 属性,选择器自动 scope)。

## 十、版本依赖管理

最大的坑之一:**主子应用版本不一致**。

```
子应用 A:React 17
子应用 B:React 18
主应用:React 18

→ A 加载会冲突 / hooks 报错
```

解法:
- 强制版本统一(monorepo 或 changeset 协调)
- Module Federation 的 `singleton: true` + 版本范围
- 实在不行各自打包(放弃 share,牺牲体积换稳定)

```ts
shared: {
  react: { singleton: true, requiredVersion: '^18.0.0' },
}
```

## 十一、部署

### 独立部署

```
子应用 A:https://app-a.example.com (CDN)
子应用 B:https://app-b.example.com
主应用:  https://main.example.com
        引用 A / B 的 remoteEntry.js
```

A 改了重新部署,主应用**不用重发布**。

### 版本管理

```
remoteEntry.js 永远是最新版?
  → 子应用上线立即影响所有用户(可能出 bug)

或带版本号:
  https://app-a.com/v1.2.3/remoteEntry.js
  → 主应用引用具体版本,可以灰度 / 回滚
```

实战推荐**带版本号 + 配置中心切换版本**。

### 灰度

主应用的配置中心说"10% 用户用 v1.3,90% 用 v1.2"。Federation 的 remote URL 在运行时决定。

## 十二、常见 Trap

### Trap 1:React 多份导致 hooks 报错

```
Error: Invalid hook call. Hooks can only be called inside the body of a function component.
```

99% 是主应用和子应用各打包了 React。**Federation 配 shared singleton**。

### Trap 2:CSS 互相覆盖

子应用 CSS 全局选择器(`button { ... }`)污染主应用。修复:
- 子应用所有样式加前缀
- 用 Shadow DOM
- 用 CSS-in-JS

### Trap 3:子应用单独跑没问题,集成进去 404

主应用拼的 URL 路径不对。Vite 配:

```ts
build: {
  rollupOptions: {
    output: { entryFileNames: 'assets/[name].js' }
  }
}
```

确保子应用资源路径稳定,配 `base` 选项。

### Trap 4:路由冲突

子应用 A 用 `/users`,子应用 B 也用 `/users`。
解法:**主应用规划路由前缀**,A 在 `/app-a/users`,B 在 `/app-b/users`。

### Trap 5:加载慢

每个子应用都是独立 SPA,**首次加载多份框架**。优化:
- 共享公共依赖(React / lodash / antd 通过 Federation share)
- 主应用预加载子应用(prefetch)
- 子应用按路由懒加载

### Trap 6:登录态共享

**所有子应用同源** → cookie 自动共享,简单
**不同域** → SSO + token 通过 postMessage / iframe 共享,复杂

最佳实践:微前端**强烈建议同域**(主应用 main.com,子应用都是 main.com 的子路径或子域)。

### Trap 7:开发体验差

每次改子应用要重启 / 主应用刷新,反馈慢。
**子应用支持独立开发模式**:不挂主应用就能跑,但接入主应用时切换到嵌入模式。

## 十三、心智模型

```
微前端的核心问题:
  "多个独立团队 / 应用如何在一个 URL 下协同"

实现层级(选最简够用的):
  1. URL 分发(Nginx)→ 简单,无集成
  2. iframe → 强隔离,通信麻烦
  3. Web Components(44 篇)→ 跨框架轻量
  4. Module Federation → 同栈最佳,运行时共享
  5. qiankun / Wujie → 多栈混合,生命周期托管

生产关键:
  - 同源 / 同子域(简化登录 / cookie)
  - 共享框架版本(否则崩)
  - 子应用独立部署 + 版本号
  - 样式隔离(scoped / shadow / 前缀)
  - URL 规划路由前缀
  - 性能:share + prefetch
```

## 十四、选型决策树

```
项目规模 / 团队
│
├─ 小项目 / 一团队
│  → 不用微前端,用 monorepo
│
├─ 中型,统一栈
│  → Module Federation(Vite/Webpack)
│
├─ 大型,多栈混合 / 中文场景
│  → qiankun 或 Wujie
│
├─ 老应用渐进重构
│  → iframe(不想折腾)或 qiankun(想保留体验)
│
└─ 跨公司 / 跨产品集成(SaaS 嵌入)
   → iframe + postMessage(最稳)
```

## 十五、推荐学习路径

如果你接到微前端任务:

1. 这一篇过一遍
2. **跑通官方 Demo**:Module Federation 或 qiankun(各 1 小时)
3. 写一个 toy 项目:主 + 2 个子应用,各种通信尝试一遍
4. 看现有大公司分享:阿里 / 腾讯 / Adobe 的微前端实践博客

如果只是了解:这一篇 + 知道这几种方案的存在就够了。**多数前端工程师一辈子用不上微前端**。
