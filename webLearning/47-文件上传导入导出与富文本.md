# 文件上传、导入导出与富文本

文件、导入导出、富文本是业务系统里最容易被低估的三类功能。

一句话先记住:**这三类功能都不是简单表单字段,它们有体积、格式、安全、进度、失败重试和服务端处理。**

---

## 一、文件上传的基本流程

```text
选择文件
  -> 前端校验类型 / 大小
  -> 获取上传凭证
  -> 直传对象存储
  -> 回写业务记录
  -> 展示上传结果
```

不要让大文件穿过业务后端:

```text
浏览器 -> 业务后端 -> 对象存储  // 后端带宽被打爆
```

推荐:

```text
浏览器 -> 对象存储
浏览器 -> 业务后端保存 fileId
```

## 三、前端校验

```ts
const maxSize = 20 * 1024 * 1024;
const allowedTypes = ['image/png', 'image/jpeg', 'application/pdf'];

export function validateFile(file: File) {
  if (!allowedTypes.includes(file.type)) {
    return 'Unsupported file type';
  }

  if (file.size > maxSize) {
    return 'File is too large';
  }

  return null;
}
```

前端校验只负责体验。服务端必须再校验一次。

## 四、带进度上传

`fetch` 原生上传进度支持不好,需要进度时用 `XMLHttpRequest` 或上传 SDK。

```ts
export function uploadWithProgress(file: File, url: string, onProgress: (n: number) => void) {
  return new Promise<void>((resolve, reject) => {
    const xhr = new XMLHttpRequest();

    xhr.upload.onprogress = (event) => {
      if (event.lengthComputable) {
        onProgress(Math.round((event.loaded / event.total) * 100));
      }
    };

    xhr.onload = () => {
      if (xhr.status >= 200 && xhr.status < 300) resolve();
      else reject(new Error(`Upload failed:${xhr.status}`));
    };

    xhr.onerror = () => reject(new Error('Network error'));
    xhr.open('PUT', url);
    xhr.send(file);
  });
}
```

上传大文件要支持取消:

```ts
const xhr = new XMLHttpRequest();
xhr.abort();
```

## 五、分片上传

大文件要分片:

```text
file
  -> slice chunk 1
  -> slice chunk 2
  -> slice chunk 3
  -> upload chunks concurrently
  -> complete multipart upload
```

前端分片:

```ts
function createChunks(file: File, chunkSize = 5 * 1024 * 1024) {
  const chunks: Blob[] = [];
  let start = 0;

  while (start < file.size) {
    chunks.push(file.slice(start, start + chunkSize));
    start += chunkSize;
  }

  return chunks;
}
```

分片上传必须有服务端协议:

```text
POST /uploads/init       -> uploadId + part urls
PUT  part url            -> upload part
POST /uploads/complete   -> merge parts
POST /uploads/abort      -> cleanup
```

不要自己发明对象存储协议,优先用 S3 / R2 / OSS / COS 的 multipart upload。

## 六、导入

导入不是"上传 Excel"这么简单。

标准流程:

```text
上传文件
  -> 服务端解析
  -> 字段映射
  -> 数据校验
  -> 预览错误
  -> 用户确认
  -> 后台任务导入
  -> 导入结果报告
```

前端页面至少有三步:

```text
1. Upload
2. Preview
3. Result
```

错误展示要到行级:

```ts
type ImportError = {
  row: number;
  field: string;
  message: string;
};
```

```text
第 12 行 email:格式错误
第 31 行 role:不支持的角色
第 54 行 name:不能为空
```

只提示"导入失败"等于没提示。

## 七、导出

小数据量可以前端导:

```ts
const blob = new Blob([csvText], { type: 'text/csv;charset=utf-8' });
const url = URL.createObjectURL(blob);

const a = document.createElement('a');
a.href = url;
a.download = 'users.csv';
a.click();

URL.revokeObjectURL(url);
```

大数据量必须服务端导:

```text
POST /exports
  -> exportJobId

GET /exports/:id
  -> pending / running / success / failed

GET /exports/:id/download
  -> file
```

不要让浏览器一次加载 100 万行。

## 八、富文本的核心风险

富文本最大的风险是 XSS。

用户输入:

```html
<img src=x onerror="fetch('/api/token').then(...)">
```

如果直接渲染:

```tsx
<div dangerouslySetInnerHTML={{ __html: content }} />
```

就可能出事。

必须清洗:

```ts
import DOMPurify from 'dompurify';

const safeHtml = DOMPurify.sanitize(content);
```

服务端也要清洗。前端清洗只保护当前页面,不保护其他消费端。

---

## 九、富文本存什么

常见格式:

| 格式 | 特点 |
| --- | --- |
| HTML | 展示简单,安全清洗复杂 |
| Markdown | 写作友好,能力有限 |
| JSON AST | 编辑器友好,迁移可控 |

现代编辑器通常存 JSON AST:

```json
{
  "type": "doc",
  "content": [
    {
      "type": "paragraph",
      "content": [{ "type": "text", "text": "Hello" }]
    }
  ]
}
```

展示时再转 HTML。

## 十、编辑器选型

```text
TipTap        ProseMirror 封装,现代富文本首选
Lexical       Meta 出品,React 友好
Slate         灵活,但维护成本高
Quill         老牌,简单场景够用
Monaco        代码编辑器,不是富文本
CodeMirror    代码 / Markdown 编辑器
```

选型规则:

```text
普通内容编辑       -> TipTap / Lexical
Markdown 写作      -> MDX / CodeMirror
代码编辑           -> Monaco / CodeMirror
简单公告编辑       -> Quill 也够
协同编辑           -> TipTap + Yjs / Lexical + 协同方案
```

## 十一、文件安全清单

- [ ] 前端校验类型和大小
- [ ] 服务端再次校验
- [ ] 上传到对象存储,不要压业务后端带宽
- [ ] 私有文件使用签名 URL
- [ ] 图片做压缩、缩略图、EXIF 清理
- [ ] PDF / Office 文件不要直接信任
- [ ] 富文本 HTML 必须清洗
- [ ] 下载文件名要转义
- [ ] 导入要有行级错误报告
- [ ] 大导出走后台任务

## 十二、心智模型

```text
上传 = 文件传输 + 进度 + 对象存储 + 业务回写
导入 = 文件解析 + 校验 + 预览 + 后台任务
导出 = 查询快照 + 文件生成 + 下载权限
富文本 = 编辑体验 + 数据格式 + XSS 防护
```

这些功能不是 UI 小活。把协议和状态设计好,后面才不会补洞。
