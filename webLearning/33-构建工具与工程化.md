# 构建工具与工程化:Vite 深入、Monorepo

前端"工程化"无非两件事:
1. **构建工具**:把你的源码(TS/JSX/CSS/Vue/...)变成浏览器能跑的东西
2. **项目组织**:多个项目共享代码,依赖关系怎么管

主角:**Vite + pnpm + Turborepo**(2025 标配)。

---

## 一、构建工具简史

```
Webpack(2014~)        : 配置爆炸,启动慢,但生态最大
Rollup(2015~)         : 库构建首选,体积小,Tree shaking 强
Parcel(2017~)         : 零配置,但慢慢被 Vite 替代
Vite(2020,尤雨溪)    : 开发用 ESM 不打包,生产用 Rollup,飞快
esbuild(2020,Go)     : 极快,Vite 内部用它做 TS 转译
swc(2020,Rust)       : 比 Babel 快 20 倍,Next.js 用它
Turbopack(2022,Vercel): Webpack 作者写的,Next.js 主推
Rolldown(2024,Rust)   : Vite 在做的 Rollup-Rust 替代版
Rsbuild / Rspack(字节) : Webpack 兼容的 Rust 版
```

**2025 选型**:**Vite 是新项目默认**,Webpack 老项目维护,Next.js 用 Turbopack。

## 二、Vite 为什么快

### 1. 开发模式:利用浏览器原生 ESM

```
传统 Webpack:启动 → 打包整个项目 → 启服务器
            慢(项目大要 30 秒+)

Vite:启动 → 启服务器 → 浏览器请求哪个文件 → 现编译那个
      快(冷启动 < 1 秒,大型项目也是)
```

工作原理:

```html
<script type="module" src="/src/main.ts"></script>
```

浏览器看到这个会发一系列请求:
```
GET /src/main.ts
  → main.ts 里 import './foo.ts' → GET /src/foo.ts
  → foo.ts 里 import './bar.css' → GET /src/bar.css
  ...
```

**Vite 是个智能的 dev server**,接到请求时:
- `.ts` → esbuild 实时转 JS 返回
- `.vue/.tsx` → 拆 template/script/style 返回
- `.css` → 注入 `<style>` 实现 HMR

按需编译,没访问到的文件根本不处理。**代码量再大,启动时间不变**。

### 2. 依赖预构建

```
node_modules 里的 lodash 等是 CJS 或多文件分散
→ 启动时 esbuild 打包成 ESM 单文件
→ 浏览器请求这一个就够,不发几百个请求
```

存在 `node_modules/.vite/deps/`,改了依赖才重新构建。

### 3. 生产模式:Rollup

`pnpm build` 时切到 Rollup,做完整的 Tree shaking + 代码分割 + 压缩。

```
开发:esbuild + 原生 ESM,极快但不优化
生产:Rollup,慢但产物最小
```

未来 Vite 用 Rolldown(Rust 写的 Rollup),开发和生产统一。

## 三、Vite 实战配置

### 1. 基础

```bash
pnpm create vite my-app --template react-ts
cd my-app
pnpm install
pnpm dev
```

模板:`react-ts` `vue-ts` `svelte-ts` `solid-ts` `vanilla-ts`...

### 2. `vite.config.ts`

```ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, 'src'),       // import '@/foo'
    },
  },
  server: {
    port: 3000,
    proxy: {
      '/api': 'http://localhost:8080',           // 解决跨域
    },
  },
  build: {
    target: 'esnext',
    sourcemap: true,
    rollupOptions: {
      output: {
        manualChunks: {
          react: ['react', 'react-dom'],         // 拆包
          ui: ['@radix-ui/react-dialog'],
        },
      },
    },
  },
});
```

### 3. 环境变量

```bash
# .env
VITE_API_URL=http://api.example.com
```

```ts
// 只有 VITE_ 前缀的变量才会被注入到客户端代码(防泄露)
const url = import.meta.env.VITE_API_URL;
```

不同环境用不同文件:`.env.development`、`.env.production`、`.env.local`。

### 4. 静态资源

```ts
import logoUrl from './logo.png';        // URL string
import logoRaw from './logo.svg?raw';     // 文本内容
import logoUrl2 from './logo.svg?url';     // 强制 URL
import Worker from './worker?worker';     // Web Worker
```

### 5. CSS 模块 / PostCSS / Sass

```ts
// 自动识别 *.module.css
import s from './a.module.css';

// 直接 sass(装 sass)
import './a.scss';

// PostCSS:放 postcss.config.js 自动加载
```

### 6. HMR(Hot Module Replacement)

Vite 默认开 HMR,**修改组件时只更新这个组件,不刷新页面、不丢 state**。React Fast Refresh、Vue HMR 都内置。

需要手动控制:

```ts
if (import.meta.hot) {
  import.meta.hot.accept('./foo', (newFoo) => {
    // 自定义更新逻辑
  });
}
```

99% 不用自己写。

### 7. 库模式

写库不是写应用:

```ts
export default defineConfig({
  build: {
    lib: {
      entry: 'src/index.ts',
      name: 'MyLib',
      formats: ['es', 'cjs'],
      fileName: 'index',
    },
    rollupOptions: {
      external: ['react', 'react-dom'],     // 不打包 peerDependencies
    },
  },
});
```

发包到 npm 用这种模式。

## 五、Vite 之外:其他构建工具

### Webpack(老项目)

```js
// webpack.config.js
module.exports = {
  entry: './src/index.ts',
  module: {
    rules: [
      { test: /\.tsx?$/, use: 'ts-loader' },
      { test: /\.css$/, use: ['style-loader', 'css-loader'] },
    ],
  },
  resolve: { extensions: ['.ts', '.tsx', '.js'] },
};
```

特点:
- 配置项多到爆炸,Loader / Plugin 概念分离
- 启动慢,大项目几十秒
- HMR 比 Vite 慢不少
- 但生态最大,所有 loader 都有

**新项目不要选 Webpack**。

### esbuild(底层工具)

```bash
esbuild app.ts --bundle --outfile=out.js
```

- Go 写的,**比 Webpack 快 100 倍**
- API 简单
- Tree shaking 不如 Rollup 完善
- 缺少 HMR 等高级特性
- **Vite 内部就用它做 TS 转译**

直接用 esbuild 适合工具脚本、CLI、小库。应用还是用 Vite。

### Turbopack(Next.js 主推)

- Webpack 作者 Tobias 写的,Rust
- Next.js 15+ 默认开发模式
- 生产模式还在用 Webpack(2025 春)

未来会替代 Webpack 在 Next.js 里的位置。

### Rspack / Rsbuild(字节)

- Rust 写的 Webpack 兼容版
- 老 Webpack 项目无痛迁移,**速度提升 10x**
- 字节内部大规模用
- 适合"已经一堆 Webpack 配置不想重写"的团队

## 六、包管理器:npm / yarn / pnpm

### 三个的区别

```
npm     : 最老,扁平 node_modules,有"幽灵依赖"问题
yarn    : 速度比 npm 快,classic 用 node_modules,berry 改用 PnP
pnpm    : 硬链接 + 符号链接,最省磁盘 + 最快 + 严格依赖
```

### pnpm 的 node_modules 结构

```
node_modules/
├── .pnpm/
│   ├── react@18.2.0/
│   │   └── node_modules/
│   │       └── react/  (硬链接到全局 store)
│   └── ...
└── react -> .pnpm/react@18.2.0/node_modules/react
```

**只有你声明的依赖才能 import**(不像 npm 把所有依赖都摊平)。这能避免:

```js
// npm/yarn classic 里能 import,但你 package.json 没写过它
// 升级时它消失,你的代码炸了
import _ from 'lodash';
```

pnpm 直接禁止。**严格 = 长期维护更稳**。

### 必装命令

```bash
pnpm add react              # 装依赖
pnpm add -D vite            # devDependencies
pnpm add -g pnpm            # 全局
pnpm install                # 装所有依赖
pnpm update                 # 升级
pnpm remove react           # 卸载
pnpm dlx create-vite        # 一次性运行包(类 npx)
```

### `package.json` 字段

```json
{
  "name": "my-app",
  "version": "1.0.0",
  "type": "module",          // ESM(2025 默认)
  "main": "dist/index.js",
  "module": "dist/index.mjs",
  "types": "dist/index.d.ts",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  },
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "test": "vitest"
  },
  "dependencies": {},
  "devDependencies": {},
  "peerDependencies": {}
}
```

## 七、Monorepo:多项目共享代码

### 1. 什么是 Monorepo

```
单仓库
├── apps/
│   ├── web/          (Next.js 网站)
│   ├── mobile/       (React Native)
│   └── admin/        (Vite 管理后台)
└── packages/
    ├── ui/           (共享组件)
    ├── utils/        (共享工具)
    └── config/       (共享 ESLint/TS 配置)
```

**优点**:
- 共享代码无需发包,改了 ui,所有 app 立即拿到
- 所有项目同步升级依赖版本
- PR 一次跨多个项目改

**适用**:
- 多端应用(Web + Mobile + Desktop)
- UI 组件库 + 文档站
- 中型以上前端团队

### 2. pnpm workspace(最简方案)

```yaml
# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
```

```json
// apps/web/package.json
{
  "dependencies": {
    "@repo/ui": "workspace:*"
  }
}
```

`workspace:*` 表示"用本仓库里的"。pnpm 会创建符号链接。

```bash
pnpm --filter web dev          # 只跑 web 的 dev
pnpm --filter "@repo/*" build  # 跑所有 @repo/ 开头的
pnpm -r build                  # 所有子包都跑 build
```

### 3. Turborepo(任务编排)

光 pnpm 不够,因为:
- `pnpm -r build` 是串行,慢
- 没缓存,改一行重跑全部
- 不知道哪个先跑(ui 必须先编译,web 才能用)

Turborepo 解决这些:

```bash
pnpm add -Dw turbo
```

```json
// turbo.json
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],         // 先构建依赖的子包
      "outputs": ["dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "test": {
      "dependsOn": ["build"]
    }
  }
}
```

```bash
turbo run build       # 智能并发 + 缓存
turbo run build --filter=web    # 只构建 web 及其依赖链
```

**缓存效果**:
- 第一次 build 全跑,假设 30s
- 改了 web 的代码再 build:只跑 web,2s
- 改了 ui:重跑 ui + web,15s
- 没改任何代码再跑:**全部命中缓存,0.5s**

可以**接 Vercel Remote Cache**,团队共享缓存,**新人 clone 后 build 直接用别人的缓存**,几秒搞定。

### 4. Nx(更重的方案)

- Nrwl 出品,Angular 生态主推
- 比 Turborepo 功能多(代码生成器、可视化依赖图)
- 学习成本更高
- 大型团队 / 复杂依赖适合

中小项目 **Turborepo 够用**。

## 八、Monorepo 实战:典型布局

```
my-monorepo/
├── apps/
│   ├── web/                    # 主站(Next.js)
│   │   ├── package.json
│   │   └── ...
│   └── docs/                   # 文档(VitePress)
├── packages/
│   ├── ui/                     # 共享组件
│   │   ├── src/
│   │   │   └── Button.tsx
│   │   └── package.json
│   ├── utils/                  # 工具函数
│   ├── config-eslint/          # 共享 ESLint
│   │   └── index.js
│   └── config-tsconfig/        # 共享 tsconfig
│       └── base.json
├── pnpm-workspace.yaml
├── turbo.json
├── package.json                # 根 package.json
└── .gitignore
```

### 共享 TS 配置

```json
// packages/config-tsconfig/base.json
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "module": "ESNext",
    ...
  }
}
```

```json
// apps/web/tsconfig.json
{
  "extends": "@repo/config-tsconfig/base.json",
  "compilerOptions": { ... }
}
```

### 共享 ESLint 配置

```js
// packages/config-eslint/index.js
module.exports = {
  extends: ['eslint:recommended', '@typescript-eslint/recommended'],
  rules: { ... }
};
```

```json
// apps/web/.eslintrc.json
{ "extends": ["@repo/config-eslint"] }
```

## 九、CI/CD 中的工程化

### 1. Lint + Format(必装)

```bash
pnpm add -Dw eslint prettier @typescript-eslint/parser @typescript-eslint/eslint-plugin
pnpm add -Dw lint-staged simple-git-hooks
```

`package.json`:

```json
{
  "simple-git-hooks": {
    "pre-commit": "pnpm lint-staged"
  },
  "lint-staged": {
    "*.{ts,tsx}": ["eslint --fix", "prettier --write"]
  }
}
```

提交前自动 lint + 格式化。

### 2. Commit Lint(可选)

```
feat: 新功能
fix: 修 bug
refactor: 重构
docs: 文档
chore: 杂项
```

用 [`commitlint`](https://github.com/conventional-changelog/commitlint) 强制规范,后面用 `changesets` 自动生成 changelog。

### 3. Changesets(monorepo 发包)

```bash
pnpm add -Dw @changesets/cli
pnpm changeset init
```

```bash
pnpm changeset           # 描述这次改了啥
pnpm changeset version   # 自动改 package.json 版本号
pnpm changeset publish   # 发到 npm
```

每个 changeset 一个 markdown 文件,自动合成 changelog。

## 十、常见 Trap

### Trap 1:依赖装错位置

```bash
# ❌ 装到了根目录,子包用不到
pnpm add react

# ✅ 指定子包
pnpm --filter web add react

# 或在子包目录里
cd apps/web && pnpm add react
```

### Trap 2:重复依赖版本不一致

`apps/web` 用 React 18,`apps/admin` 用 React 17,`packages/ui` 同时被两个用 → 编译过,运行可能 hooks 报错。

解决:**强制统一版本**:

```yaml
# pnpm-workspace.yaml
overrides:
  react: "^18.0.0"
  react-dom: "^18.0.0"
```

### Trap 3:`workspace:*` 发布前没替换

发到 npm 时 `"@repo/ui": "workspace:*"` 必须替换成实际版本号。`pnpm publish` 自动做,`changesets` 也自动做。

### Trap 4:Vite 默认不处理 SSR

服务端渲染要用 `vite-node` 或 SolidStart/SvelteKit 等框架。Vite 自身只是构建工具。

### Trap 5:动态 import 路径

```js
// ❌ Vite 静态分析不到
const m = await import(`./pages/${name}.tsx`);

// ✅ 用 import.meta.glob
const pages = import.meta.glob('./pages/*.tsx');
```

## 十一、心智模型

```
工具链分层:

代码 → 转译(esbuild/swc/babel) → 打包(Rollup/Webpack) → 输出

Vite 的本质:
  开发:用浏览器原生 ESM,按需现编译,启动飞快
  生产:还是 Rollup,产物最小

Monorepo 的本质:
  pnpm workspace = 多个项目共用一个 node_modules + 互相引用
  Turborepo = 跨项目跑命令时智能并发 + 缓存

2025 推荐栈:
  pnpm + Vite + TypeScript + Turborepo + Vercel
  这是 95% 项目的最佳组合
```
