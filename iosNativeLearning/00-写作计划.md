# 苹果原生开发系列写作计划

> 来源:`未来系列规划.md` 第六类「平台原生开发」之 P 项 iosNative,⭐⭐⭐。
> 本系列基线版本为 **iOS 18 / Swift 6 / Xcode 16 / SwiftUI**(截稿前以 **2026-05 时点的最新 minor / patch 版本**为参照;iOS 19 已正式发布,会在涉及新 API 的篇目里单独标注「iOS 19+」,主线仍以 iOS 18 为最低部署目标,避免文章漂移)。
> 本系列不做"Objective-C + Storyboard 老 iOS 复刻",也不做"SwiftUI 控件展览",而是围绕一个可演进的 iOS 应用原型,逐层拆解 Swift 心智、声明式 UI、Apple 平台能力、性能工程与 App Store 发布通路。

---

## 核心定位

这不是"iOS 全家桶百科",也不是"Swift 语法手册",更不是"Flutter 转 iOS 速成"。本系列只解决一个明确目标:

> 让一个熟悉编程但不熟悉 Apple 平台的人,能用 Swift 6 + SwiftUI + iOS 18 做出一款结构清楚、数据可持久化、能调用系统能力、能通过审核上架 TestFlight / App Store 的原生 iOS 应用。

因此本系列默认选择:

| 维度 | 选择 | 原因 |
| --- | --- | --- |
| 系统基线 | iOS 18 (最低部署目标),兼顾 iOS 19+ 新 API | 覆盖 2026 主流装机量,避免追新尾巴上的 beta API |
| 工具链 | Xcode 16 / Swift 6 toolchain | Swift 6 严格并发与 macro 已稳定,Xcode 16 是首批默认 |
| 主语言 | **Swift 6**,严格并发模式 (`SwiftSettings.swiftLanguageMode(.v6)`) | 编译期消灭数据竞争,贴近 Apple 官方未来方向 |
| UI 主线 | **SwiftUI** 声明式 UI + `@Observable` 状态 | Apple 全平台默认 UI 框架,不再讲 Storyboard / XIB / 传统 ViewController |
| 并发模型 | Swift Concurrency(`async/await` / `actor` / `Sendable` / `Task`) | GCD / RunLoop 仅作对比心智,不写老式教程 |
| 数据层 | **SwiftData** 优先,Core Data 仅在必要场景作为后备 | iOS 17+ 推荐路径,基于 Core Data 但 API 现代化 |
| 跨端 | **不覆盖**,由 `flutterLearning` 负责跨端表现层 | 本系列只讲 Apple 平台原生能力 |
| 兄弟平台 | macOS / iPadOS / visionOS / watchOS 仅在尾段简提 | 主线聚焦 iPhone,避免目标分散 |
| 文章目标 | 机制 + 工程取舍 + 可运行示例 | 不只贴 API,要讲为什么 Apple 这样设计、何时不要这样写 |

读者画像:

- 已经写过其他语言/框架的应用(Flutter / React Native / Android / Web),想真正用「Apple 原生方式」做 iOS。
- 看得懂英文 API 文档,但被 Swift Concurrency、`@Observable`、SwiftData、严格并发的取舍劝退过。
- 想做的是一款能上 App Store 的真应用,而不是 Playground 玩具或某个 SDK 的 demo。

## Swift 6 / iOS 18 / SwiftUI 关键新特性与本系列定位

Swift 6.0 于 2024-09 随 Xcode 16 发布;到 2026-05,Swift 6.x 已经经过若干 minor 版本迭代,iOS 19 也已发布。本系列把下列变更显式纳入相关篇目,避免读者把 Swift 5 / iOS 16 / 旧 SwiftUI 心智带进 2026 的项目模板:

| 变更 | 影响范围 | 本系列在哪篇展开 |
| --- | --- | --- |
| **Swift 6 严格并发(strict concurrency by default)** | 数据竞争编译期报错;`Sendable` / actor 隔离传染整张依赖图 | 第 04 / 05 篇心智段,第 26 / 27 篇排障 |
| **`@Observable` 宏 + Observation 框架** | 替代 `ObservableObject` + `@Published`;视图自动追踪字段而非整对象 | 第 07 篇主篇,第 08 / 13 / 26 篇 |
| **`@Entry` 宏(iOS 18 新)** | `EnvironmentValues` 自定义键从 30 行样板减为 1 行 | 第 08 篇环境值章节 |
| **SwiftData 稳定化 + CloudKit 自动同步** | `@Model` / `ModelContainer` / `@Query` 取代手写 Core Data;`@Model` 可直接对接 iCloud | 第 14 篇主线 |
| **NavigationStack + 类型化路由(iOS 16+,18 完善)** | `NavigationLink(value:)` + `.navigationDestination(for:)` 取代 `NavigationView` | 第 13 篇 |
| **SwiftUI 动画 / 转场 / `matchedGeometryEffect` / `TimelineView`** | 显式动画与隐式动画分离,共享元素过渡有原生 API | 第 11 篇 |
| **SwiftUI Gesture 组合(`SimultaneousGesture` / `ExclusiveGesture` / `HighPriorityGesture`)** | 手势冲突与优先级在声明式 UI 下重新组织 | 第 12 篇 |
| **WidgetKit / ActivityKit / Dynamic Island(iOS 16.1+ / 17 稳定)** | Widget / Live Activities / Dynamic Island 同一套 SwiftUI 子集 | 第 22 篇 |
| **StoreKit 2(iOS 15+,Swift 原生)** | `Product` / `Transaction.updates` 取代旧 `SKPaymentQueue`;JWS 收据本地校验 | 第 23 篇 |
| **Localizable.xcstrings 字符串目录(Xcode 15+)** | 取代旧 `Localizable.strings`,翻译状态可视化 | 第 24 篇 |
| **Core ML / Vision / Apple Intelligence on-device** | 端侧推理 + Vision Framework 视觉任务统一入口;A17/M 系列上有 ANE 加速 | 第 25 篇 |
| **Swift Testing 框架(`import Testing`)** | `@Test` / `#expect` / `#require` 取代 XCTest 大部分场景,但 UI Testing 仍走 XCTest | 第 29 篇 |
| **Privacy Manifests (`PrivacyInfo.xcprivacy`) 上架强制要求** | 第三方 SDK 与自家代码都需要声明 API reason / 数据使用 | 第 20 / 30 篇 |
| **SwiftUI 性能调优 API**(`_printChanges`、`@Observable` diff 行为) | 重渲染追踪心智不同于 React;Combine 仅在桥接处保留 | 第 26 篇 |
| **背景任务 BGTaskScheduler + BGProcessingTask** | 取代旧的后台 fetch 心智 | 第 21 篇 |
| **App Intents(iOS 16+)** | 与系统集成的统一入口(Shortcuts / Spotlight / Widgets / Siri) | 第 21 / 22 篇 |
| **`URLSession` 全面 async / `bytes(for:)` 流式 API** | 不再写 `dataTask` 回调链;Combine 仅在桥接处保留 | 第 15 篇 |
| **MetricKit `MXMetricPayload` + `OSLogStore` 检索** | 线上 Crash 与性能指标统一上报 | 第 27 篇 |

> 注:iOS 19 引入的 Liquid Glass 等视觉系统变化、新交互组件仅在涉及具体 API 时单独标注「iOS 19+」并给降级方案;本系列正篇仍以 iOS 18 为最低部署目标,与 App Store 主流装机量对齐。

## 七层路线

```
Swift 心智与项目骨架
   → SwiftUI 声明式 UI 与状态模型
   → 动画、手势与导航
   → 数据、网络与本地存储闭环
   → 系统能力、UIKit 互操作与权限
   → Widget / 商业化 / 智能化 / 辅助功能
   → 性能、测试、发布与 Apple 生态
```

核心节奏是:**前 15 篇先把"心智 + UI + 闭环"跑通,后 15 篇补系统能力、Widget / Live Activities、StoreKit 2、Core ML、性能与发布通路。**

## 篇目设计 (30 篇)

### 第一层:Swift 心智与项目骨架 (01-05)

- **01 Swift 6 / iOS 18 / SwiftUI 心智总览** (为什么选原生 / Apple SDK 分层 / `App` 协议入口 / 本系列最终会做出什么 / iOS 18 与 iOS 19 部署目标取舍)
- **02 Xcode 16 项目结构、Bundle、Info.plist 与 `xcconfig`** (`Project` vs `Workspace` / Scheme 与 Build Configuration / `Info.plist` 现代化 generated 模式 / `xcconfig` 分环境配置 / Asset Catalog 与 SF Symbols)
- **03 Swift 6 类型系统:值类型 / 引用类型 / 协议 / 泛型** (`struct` 与 `class` 内存语义 / `protocol` + `associatedtype` / opaque return `some View` / `any` existential / 与 Dart/Kotlin 类型系统对照)
- **04 Swift Concurrency 心智:`async/await` / `Task` / 结构化并发** (`Task` / `TaskGroup` / `async let` / 取消传播 / `MainActor` / GCD 仅作对比,不再用 `DispatchQueue.async` 教学)
- **05 `actor` / `Sendable` / 严格并发:Swift 6 数据竞争编译期消除** (`actor` 隔离 / `nonisolated` / `Sendable` 协议 / `@unchecked Sendable` 红线 / `@MainActor` 类型与方法标注 / 与 Rust `Send/Sync` 对照)

### 第二层:SwiftUI 声明式 UI 与状态模型 (06-10)

- **06 SwiftUI `View` 协议、声明式渲染与 `@State`** (`body: some View` / 视图是值类型 / SwiftUI 重计算 vs 重渲染 / `@State` 仅用于视图私有可变状态 / 与 React useState 对照心智)
- **07 `@Observable` / Observation 框架与状态共享** (`@Observable` 宏展开 / 字段级追踪 / `@Bindable` / 取代 `ObservableObject` + `@Published` / 与 MVVM 心智落地)
- **08 `@Binding` / `@Environment` / `@Entry` 与数据流向** (`@Binding` 双向绑定 / `EnvironmentValues` / **iOS 18 `@Entry` 宏简化自定义环境键** / 何时用 environment 而不是 prop drilling)
- **09 SwiftUI 布局系统:`HStack` / `VStack` / `ZStack` / `Grid` / `Layout` 协议** (布局协议 / proposal-response 心智 / `LazyVStack` 与可见性 / `Layout` 自定义协议入门 / 与 Flutter Flex / Web Flex 对照)
- **10 Modifier 链、`ViewModifier` 自定义与样式系统(`ButtonStyle` / `LabelStyle`)** (Modifier 顺序敏感 / 自定义 `ViewModifier` / `ButtonStyle` / `PrimitiveButtonStyle` / 样式与逻辑解耦)

### 第三层:动画、手势与导航 (11-13)

- **11 SwiftUI 动画与转场:`withAnimation` / `matchedGeometryEffect` / `TimelineView` / `PhaseAnimator`** (隐式动画 vs 显式动画 / `Animation.spring/easeInOut/interpolatingSpring` / `transaction` 控制 / `matchedGeometryEffect` 共享元素 / `TimelineView` 时间驱动动画 / `PhaseAnimator` (iOS 17+) / 与 UIKit `UIView.animate` 心智对照 / 动画与重渲染成本边界)
- **12 SwiftUI 手势系统:`Gesture` / 组合 / 优先级与冲突** (`TapGesture` / `LongPressGesture` / `DragGesture` / `MagnifyGesture` / `RotateGesture` / `SimultaneousGesture` / `ExclusiveGesture` / `HighPriorityGesture` / 手势状态 `@GestureState` / 与 `ScrollView` / `NavigationStack` 滑动返回的冲突 / `onScrollGeometryChange` (iOS 18+))
- **13 `NavigationStack` 与类型化路由** (`NavigationStack(path:)` / `NavigationLink(value:)` / `.navigationDestination(for:)` / `NavigationSplitView` for iPad / 弃用的 `NavigationView` / 路由状态可恢复 / Deep Link 与 `onOpenURL` 桥接)

### 第四层:数据、网络与本地存储闭环 (14-16)

> 本层每篇做"最小可行",目标是让原型能"列表 → 详情 → 编辑 → 持久化 → 同步"跑通;深度优化与系统能力留到后面层。

## 贯穿项目

全系列使用同一个逐步演进的 iOS 应用原型,工程目录约定放在 `iosNativeLearning/demo/`,主题是「**一款本地优先 + iCloud 同步 + 可推送 + 能录入图片与音频笔记的个人记录类 App**」(中文叫「写日记」一类形态,英文 working title `NotesIsland`)。

**例外**:第 17 篇(UIKit / Objective-C 互操作)允许独立最小工程演示老 SDK 桥接;第 28 篇(SPM 模块化)允许引用拆分后的多模块工程;第 30 篇延伸的 visionOS / watchOS 部分允许引用另一个最小 Multiplatform 工程。

| 阶段 | 原型状态 | 验收标准 |
| --- | --- | --- |
| 01-05 | 空工程骨架 | Xcode 16 项目可启动;基础目录、`xcconfig`、Swift 6 模式都已就位 |
| 06-10 | 可见的本地 UI | 有列表、有详情、有表单输入;`@Observable` 状态闭环跑通 |
| 11-13 | 动画与交互完整 | 隐式 / 显式动画 / 共享元素过渡完整;手势组合无冲突;NavigationStack 可深链 |
| 14-16 | 数据闭环 | SwiftData 持久化 + 网络拉数据 + Keychain 凭据安全;数据丢失/迁移可控 |
| 17-21 | 系统能力 | UIKit 互操作 / 推送 / 拍照与选图 / 权限合规 / BGTaskScheduler + App Intents |
| 22-25 | Widget / 商业化 / 智能化 / 辅助功能 | Widget + Live Activities 上线;StoreKit 2 内购 + Sign in with Apple 跑通;VoiceOver / 多语言验收;Vision / Core ML 端侧推理示例 |
| 26-30 | 可发布 | 性能可量化 + Instruments / MetricKit 上报 + SPM 模块化 + Swift Testing 覆盖 + TestFlight 内测 + Privacy Manifest 通过 + 商店审核检查表过关 |

## 单篇写作模板

每篇遵循固定结构,避免变成 API 散点:

```text
1. 机制定位
   - 这个机制在 iOS 应用里解决什么核心问题
   - 用 UIKit / 老 SwiftUI / 跨端框架的同类做法会遇到什么坑

2. Apple 平台心智
   - 对应核心类 / 协议 / 宏(给出类名与 framework,如 `URLSession` in Foundation,`@Model` in SwiftData)
   - 生命周期、隔离域(MainActor / actor / nonisolated)、数据流向

3. 工程实现
   - 30-150 行可运行 Swift 6 代码,标注 `// MARK:` 段落
   - 严格并发模式下能通过编译,无 `@unchecked Sendable`
   - 指明文件归属(`Features/Notes/NoteListView.swift` 等)

4. 调参与验收
   - 哪些参数影响性能 / 用户体验 / 审核通过率
   - 真机 vs 模拟器差异
   - 如何验证这一篇完成了(单步操作清单 + 期望结果)

5. 踩坑
   - Swift 6 / iOS 18 下的常见坑
   - 与 Swift 5 / iOS 16 旧教程、Stack Overflow 老答案的差异
   - 跨 iOS 版本兼容(`if #available`)的最低成本写法
```

## 交付标准

不是每篇都重建工程,而是维护一个贯穿式示例项目,并在关键机制处提供最小复现 Snippet。

| 交付物 | 要求 |
| --- | --- |
| Markdown 正文 | 讲清机制、Apple 平台取舍、实现步骤、踩坑边界,字数控制在 5k-9k 中文 |
| 示例项目 | `iosNativeLearning/demo/` 中维护同一个可运行 Xcode 16 工程,Swift 6 严格并发模式 |
| 代码风格 | 显式类型 / 明确并发隔离 / 优先值类型 / 不使用 `@unchecked Sendable` / 不写 `force unwrap` 兜底 |
| 图解 | 每层至少 1 张机制图;关键篇补 SwiftUI 数据流、并发隔离域、SwiftData schema、签名链与发布管线图 |
| 验收 | 每篇末尾给出手动验证步骤;关键篇补 Swift Testing 或 UI Testing 脚本化检查 |
| 官方指针 | 标注核心类名与所属 framework,如 `NavigationStack` (SwiftUI)、`ModelContainer` (SwiftData)、`UNUserNotificationCenter` (UserNotifications)、`BGTaskScheduler` (BackgroundTasks)、`AVCaptureSession` (AVFoundation) |
| 版本标注 | 涉及 iOS 19+ 新 API 单独标注「iOS 19+」并给 iOS 18 降级方案;涉及 Swift 6.x minor 差异时标注最低 toolchain |
