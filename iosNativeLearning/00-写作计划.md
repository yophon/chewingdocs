# 苹果原生开发系列写作计划

> 基线版本:**iOS 18 (API 18.0) / Swift 6 / Xcode 16 / SwiftUI**(2026/05 时点;iOS 19 新 API 涉及到时单独标注)。
> 不做"SwiftUI 控件展览",不做"Objective-C 复刻",不做"跨端选型对比"。

---

## 一句话定位

**让一个会写代码、但没怎么写过 iOS 的人,用 Swift 6 + SwiftUI 把一款能上 App Store 的现代 iOS 应用做出来。**

读者画像:已经熟悉一门主流语言(Go / Kotlin / TS / Python 都行),写过其他平台或跨端 App,但被 Swift Concurrency、`@Observable`、SwiftData、严格并发的选择劝退过。

---

## 统一心智(贯穿 27 章)

> **现代 iOS 是一棵被 Observation 追踪的 SwiftUI 视图树,挂在 `App` 协议入口上,被 Swift Concurrency 隔离并发,被 SwiftData 落盘,通过 Apple 平台框架接入系统能力。剩下的一切——Widget、推送、StoreKit、Core ML——都是这棵树的扩展或副作用边界。**

这句话压住全场,后面 27 章都是它的推论:

- **"一棵 SwiftUI 视图树"** → 05-10 篇,View / 状态 / 布局 / Modifier / 动画 / 手势
- **"被 Observation 追踪"** → 06 篇 `@Observable` 字段级 diff
- **"挂在 `App` 协议入口"** → 01 / 02 篇,`@main App` 取代 `AppDelegate`
- **"被 Swift Concurrency 隔离并发"** → 04 篇 `async/actor/Sendable`
- **"被 SwiftData 落盘"** → 12 篇 `@Model` + `ModelContainer` + `@Query`
- **"系统能力是树的扩展或副作用边界"** → 15-22 篇

如果哪一篇没法挂在这棵树上,这一篇就不该出现。

## 默认技术选择

| 维度 | 选择 | 一句话理由 |
| --- | --- | --- |
| 最低部署目标 | iOS 18 | 2026/05 装机量甜点位,iOS 19+ API 单独标注 |
| 主语言 | Swift 6,strict concurrency | 数据竞争编译期消灭,贴近 Apple 未来方向 |
| UI 框架 | SwiftUI + `@Observable` | 不写 Storyboard / XIB / `ObservableObject` |
| 并发 | `async/await` + `actor` + `Sendable` | GCD 仅作对比心智 |
| 数据持久化 | SwiftData,Core Data 仅作 fallback | iOS 17+ 推荐路径,`@Model` 几行替代几十行 |
| 云同步 | CloudKit 直挂 `@Model` | 不接 Firebase / 自建后端 |
| 网络 | `URLSession.data(for:)` async + `Codable` | 不装 Alamofire,不写 Combine |
| 三方依赖 | 默认零,SPM 按需加 | 空工程最长寿 |
| 测试 | Swift Testing + XCTest UI | `@Test` / `#expect` 取代 `XCTAssert*` |
| 发布 | App Store + TestFlight | 不做企业签名 / AltStore |

**不覆盖**:跨端见 [[flutterLearning]];协议栈见 [[networkLearning]];AI 模型本身见 [[aiLearning]];OWASP / 越狱见 [[securityLearning]]。

## 持续演进的样例应用:NotesIsland

整个系列共享一个样例应用 **NotesIsland**(本地优先 + iCloud 同步的笔记 App,带图片、录音、Widget、Live Activities、订阅),从第 05 篇起逐章长大。

这套功能恰好覆盖一个真实 iOS 应用的所有关键面:本地持久化 + 云同步、SwiftUI UI、录音 / 选图、推送、Widget / Live Activities、StoreKit 订阅、端侧 AI。比 To-Do 更接近真实,比社交 App 又不被业务细节带偏。

**但 NotesIsland 不是每篇都重提**——它是一根挂代码的线,不是主角。心智才是主角。多数代码示例是独立片段,跑得通就够;不强求拼成同一个 Xcode 工程。

## 写作风格(强约束)

1. 每篇开篇一句话压上核心 claim,不超 40 字。
2. 章节标题用「一、二、三」,**不用**「机制定位 / Apple 平台心智 / 调参与验收」这种学究腔。
3. 段落不超过 150 字,长段落必拆。
4. 强观点直接说。读者要的是判断,不是百科。
5. 表格只用于真正需要对比的地方,不堆砌。
6. 每篇结尾留一段"踩坑",对应一个真实事故场景,不写"建议注意"这种废话。
7. 不写多余的"小结"和"过渡段"。代码片段 30-80 行,不再塞 90+ 行的"最小骨架"。

## 与旧版本的差异

旧版 30 篇 / 1.6 万行,问题:标题就是 API 罗列(`@Binding 与 @Environment 与 @Entry 与数据流向`)、机械五段式模板、单篇均长 530 行(对照 git 系列 300 行)、第一篇先做销售文案、代码块过长。
新版 27 篇,把 03/04/05(类型 + 并发 + actor)合到 03/04 两篇,把 07/08(`@Observable` + `@Binding/@Environment`)合到 06 一篇,把 28/29(SPM + 测试)合到 26 一篇。Accessibility 单独成 23,因为是上架硬要求;SwiftUI 性能(24)与 Instruments(25)分开,因为前者是渲染心智,后者是工具链,合篇容易乱。
