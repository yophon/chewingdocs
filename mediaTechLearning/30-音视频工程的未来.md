# 音视频工程的未来

音视频不会被一个新协议彻底替代。未来几年更像是多条链路并存:点播继续追压缩和成本,直播追低延迟和规模,会议追质量和 AI 能力。

> 一句话先记住:**未来的音视频工程不是单点技术升级,而是编码、传输、边缘、AI 处理和质量观测一起演进**。

---

## 一、编码继续变强,但兼容永远慢

AV1、下一代视频编码会继续提高压缩效率。但生产采用不会只看效率:

- 编码速度
- 解码硬件支持
- 浏览器 / 设备兼容
- 专利和授权
- 转码成本

H.264 会继续活很久。不是因为它最好,而是因为它最稳。

---

## 二、低延迟会分层

不会所有直播都变 WebRTC。

```text
普通观看 -> HLS / DASH / CDN
强互动   -> WebRTC
专业回传 -> SRT / RIST
低延迟大规模 -> LL-HLS / 边缘 WebRTC / 混合架构
```

业务要先定延迟目标,再选技术。不要先选协议。

---

## 三、SFU 会更像媒体平台

未来 SFU 不只是转发 RTP:

- 录制
- 转推
- 自动降级
- 服务端插帧 / 合流
- AI 字幕
- 说话人检测
- 质量分析
- 边缘调度

会议系统会越来越像实时媒体平台。

---

## 四、AI 会进入媒体链路

已经很明确的方向:

- 实时字幕
- 噪声抑制
- 回声消除增强
- 超分辨率
- 背景替换
- 内容审核
- 智能剪辑
- 会议摘要

注意:AI 能力会吃 GPU / NPU / CPU,还会增加延迟。实时链路里不能无脑加模型。

---

## 五、质量观测会变成标配

未来音视频产品如果没有质量平台,基本不可运维:

```text
用户维度: 首帧 / 卡顿 / 丢包 / RTT / 设备
房间维度: 上下行码率 / SFU 转发 / 订阅关系
区域维度: ISP / CDN / TURN / 边缘节点
成本维度: 流量 / 转码 / 存储 / GPU
```

音视频不是"出了问题看日志"的系统。日志只是一部分,媒体指标才是主体。

---

## 六、你该掌握什么

读完这个系列,真正要留下的是这些能力:

1. 看懂容器和编码,知道为什么播不了。
2. 会用 FFmpeg 做基础处理,知道什么时候不能乱转码。
3. 理解 HLS / DASH 点播和分片。
4. 理解 WebRTC 建连链路,知道 STUN / TURN / ICE / SDP / RTP 各自干什么。
5. 能解释 Mesh / SFU / MCU 的取舍。
6. 能按指标排查卡顿、黑屏、断音、延迟。
7. 能算带宽、CDN、存储、转码和 SFU 成本。

---

## 七、下一步项目

最合适的闭环项目:

```text
小型视频会议 + 录制回放系统

1. WebRTC 一对一通话
2. TURN 兜底
3. 多人 SFU 房间
4. 屏幕共享
5. 服务端录制
6. FFmpeg 转 MP4 / HLS
7. 回放播放器
8. getStats 质量面板
```

这个项目能把本系列 80% 的知识串起来。只看文章不做这个闭环,很容易停在"概念都听过"。

---

## 八、收尾提醒

1. **协议不是目标,体验才是目标**。
2. **低延迟、高清、低成本不可能同时拉满**。
3. **先保音频,再保视频**。
4. **先打指标,再谈优化**。
5. **先跑通标准链路,再做花活**。

到这里,音视频工程的主线完成。后面遇到具体问题,回到对应层排查:文件问题看容器/编码/FFmpeg,播放问题看分片/缓冲/播放器,通话问题看 SDP/ICE/RTP/SFU,线上问题看指标和成本。
