# 播放器原理与缓冲策略

播放器不是简单 `<video src>`。它要拉数据、解封装、解码、同步音画、管理缓冲,还要在弱网下做取舍。

> 一句话先记住:**播放器的核心工作是让解码后的音视频帧按时间稳定输出**。缓冲太少会卡,缓冲太多会慢。

---

## 一、播放器管线

```text
网络拉取
  -> demux 解封装
  -> decode 解码
  -> audio/video queue
  -> clock 同步
  -> render 输出
```

浏览器里很多细节由内核和硬件解码器处理,但业务仍要关注:

- 首帧时间
- 缓冲长度
- 卡顿次数
- 清晰度切换
- 错误码

---

## 二、缓冲是什么

缓冲就是提前准备的可播放数据:

```text
当前播放点: 10s
已缓冲到:   25s
缓冲长度:   15s
```

HTMLVideoElement 可以看:

```js
const video = document.querySelector("video")
for (let i = 0; i < video.buffered.length; i++) {
  console.log(video.buffered.start(i), video.buffered.end(i))
}
```

---

## 三、首帧、卡顿、延迟是三角

| 目标 | 策略 | 代价 |
| --- | --- | --- |
| 首帧快 | 少缓冲就播 | 弱网容易卡 |
| 少卡顿 | 多缓冲 | 首帧慢 / 直播延迟高 |
| 低延迟 | 追直播边缘 | 更容易卡 |

点播可以多缓冲。会议不能多缓冲。直播在中间取舍。

---

## 四、ABR 切换

自适应码率播放器会根据:

- 估算带宽
- 当前缓冲
- 下载速度
- 设备解码能力
- 历史卡顿

选择下一片清晰度。切换不是立刻换当前画面,通常从下一片开始。

> ABR 不是选最高码率,而是在不卡顿的前提下尽量高清。

---

## 五、MSE 播放器

浏览器原生 `<video>` 不一定直接支持 FLV / DASH / 自定义切片。MSE(Media Source Extensions)允许 JS 往播放器喂媒体片段:

```text
fetch segment
  -> appendBuffer
  -> SourceBuffer
  -> video element
```

hls.js、dash.js、flv.js 这类播放器都建立在 MSE 能力上。

---

## 六、关键指标

| 指标 | 意义 |
| --- | --- |
| startup time | 点击播放到首帧 |
| rebuffer count | 卡顿次数 |
| rebuffer duration | 卡顿总时长 |
| bitrate | 当前码率 |
| dropped frames | 丢帧 |
| live latency | 直播落后真实时间 |

没有这些指标,播放器线上问题只能靠用户截图。

---

## 七、踩坑提醒

1. **只看能不能播**——生产要看首帧和卡顿。
2. **直播缓冲越大越稳**——稳是稳,但延迟会失控。
3. **播放器错误不打点**——线上排障没证据。
4. **切清晰度不对齐关键帧**——容易花屏或卡。
5. **忽略硬件解码能力**——低端机播 4K 会掉帧。
