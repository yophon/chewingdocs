# 编码格式:H.264 / H.265 / VP9 / AV1 / AAC / Opus

编码格式决定数据怎么压缩。选错编码,不是"清晰度差一点",而是播放不了、CPU 爆、延迟高、账单贵。

> 一句话先记住:**H.264 是兼容性基线,H.265 / VP9 / AV1 是压缩效率换复杂度,AAC 是通用音频,Opus 是实时语音王者**。

---

## 一、视频编码解决什么

视频编码器靠三类信息压缩:

| 类型 | 思路 |
| --- | --- |
| 空间冗余 | 一帧内部相邻像素相似 |
| 时间冗余 | 相邻帧大部分区域没变 |
| 感知冗余 | 人眼对某些细节不敏感 |

所以编码器不是简单压缩,它会运动估计、预测、变换、量化、熵编码。代价是 CPU / GPU 计算。

---

## 二、H.264:默认安全选项

优点:

- 浏览器、手机、电视、硬件解码支持最广
- 编码器成熟
- 工具链稳定

常用:

```bash
ffmpeg -i input.mov -c:v libx264 -crf 23 -preset medium -c:a aac output.mp4
```

生产里,如果你不知道目标端是什么,先用 H.264 + AAC。

---

## 三、H.265:更省码率,但兼容要查

H.265 / HEVC 通常比 H.264 更省码率,尤其 4K 场景明显。但问题是:

- 浏览器支持不统一
- 专利授权复杂
- 老设备硬解不一定有

适合:

- App 内播放且设备可控
- 4K 点播
- 私有分发

不适合:

- 面向所有浏览器的默认格式

---

## 四、VP9 / AV1:开放生态和更高压缩

VP9 常见于 WebM,AV1 是更新一代。AV1 压缩效率好,但编码慢,硬件支持还要看设备代际。

适合:

- 大规模点播平台
- 有转码集群
- 目标设备支持明确

不适合:

- 小团队一上来全量转 AV1
- 实时低延迟场景里用超慢参数

---

## 五、AAC:通用音频基线

AAC 是 MP4 / HLS / 移动端里最常见的音频选择:

```bash
ffmpeg -i input.wav -c:a aac -b:a 128k output.m4a
```

音乐、点播、普通视频都稳。WebRTC 不靠 AAC 做实时语音主力。

---

## 六、Opus:实时语音优先

Opus 是 WebRTC 音频核心。强项:

- 低延迟
- 低码率语音质量好
- 支持可变码率
- 弱网抗性好

语音会议优先保 Opus 音频。视频可以降分辨率、降帧率,音频断了会议直接不可用。

---

## 七、硬编码和软编码

| 类型 | 优点 | 缺点 |
| --- | --- | --- |
| 软编码 | 参数灵活,质量可控 | 吃 CPU |
| 硬编码 | 快,省电 | 质量和参数受硬件限制 |

移动端、直播推流、实时会议常依赖硬编码。点播离线转码更常用软编码或专用转码集群。

---

## 八、踩坑提醒

1. **只看压缩率不看兼容性**——AV1 再省,播不了就是 0。
2. **H.265 不等于 Web 默认可用**。
3. **实时场景别用离线慢参数**。
4. **会议先保 Opus 音频**。
5. **硬编码输出质量要实测**——不同设备差距很大。
