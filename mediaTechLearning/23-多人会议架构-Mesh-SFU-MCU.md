# 多人会议架构:Mesh / SFU / MCU

一对一通话扩到多人,最大问题不是代码复杂,是带宽和编码压力指数上升。

> 一句话先记住:**Mesh 适合小房间,SFU 是现代会议主流,MCU 用服务器混流换客户端简单**。

---

## 一、Mesh:每个人连每个人

```text
A <-> B
A <-> C
B <-> C
```

N 人时,每人上行要发 N-1 路。

| 人数 | 每人上行 |
| --- | --- |
| 2 | 1 路 |
| 4 | 3 路 |
| 8 | 7 路 |

优点:

- 架构简单
- 不需要媒体服务器
- 端到端路径短

缺点:

- 上行爆炸
- CPU 编码压力大
- 房间稍大就不可用

适合 2-4 人小房间 demo。

---

## 二、SFU:服务器转发媒体

```text
A -> SFU -> B/C/D
B -> SFU -> A/C/D
```

每个客户端通常只上传一路,下载多路。服务器不解码混画面,主要转发 RTP 包。

优点:

- 客户端上行压力低
- 延迟低
- 可做订阅控制
- 现代会议主流

缺点:

- 服务端带宽成本高
- 要处理转发、层级编码、质量控制
- 录制和混流要额外做

---

## 三、MCU:服务器混流

```text
A/B/C -> MCU 解码混合 -> 每人收到一路合成流
```

优点:

- 客户端最简单
- 每人只收一路
- 录制容易

缺点:

- 服务端要解码 + 合成 + 编码,CPU/GPU 贵
- 延迟更高
- 画面布局不灵活

适合传统会议硬件、弱设备、固定布局录制。

---

## 四、Simulcast 和 SVC

SFU 要解决不同用户带宽不同的问题。

Simulcast:

```text
客户端同时发 180p / 360p / 720p 多路
SFU 给不同观众转不同层
```

SVC:

```text
一个编码流里包含多个空间 / 时间层
SFU 按层转发
```

目的都是让差网络用户不拖垮好网络用户。

---

## 五、架构选择

| 场景 | 推荐 |
| --- | --- |
| 2 人通话 | P2P |
| 3-4 人小 demo | Mesh 可接受 |
| 正式多人会议 | SFU |
| 固定合成画面 / 传统会议 | MCU 或 SFU + 录制合成 |
| 大规模直播 | CDN / 直播系统,不是会议架构 |

---

## 六、踩坑提醒

1. **Mesh 做 10 人会议**——上行直接炸。
2. **以为 SFU 不花钱**——带宽成本很高。
3. **SFU 录制想当然**——转发不等于自动有合成回放。
4. **不做订阅控制**——没人看的视频还在下发。
5. **会议和直播混用架构**——目标不同。
