# 采样率、帧率、码率、分辨率

音视频参数看起来多,其实都在回答一个问题:**每秒要处理多少数据**。

> 一句话先记住:**分辨率和帧率决定原始数据量,码率决定压缩后每秒给多少预算,采样率决定音频时间精度**。清晰度、流畅度、延迟、成本都从这几个数长出来。

---

## 一、分辨率:一帧有多少像素

常见分辨率:

| 名称 | 尺寸 |
| --- | --- |
| 720p | 1280 x 720 |
| 1080p | 1920 x 1080 |
| 2K | 2560 x 1440 |
| 4K | 3840 x 2160 |

分辨率翻倍不是成本翻倍。1080p 到 4K:

```text
1920 x 1080 = 207 万像素
3840 x 2160 = 829 万像素
```

像素数是 4 倍,编码、解码、带宽、显存压力都会上去。

---

## 二、帧率:每秒多少张图

| 帧率 | 场景 |
| --- | --- |
| 24fps | 电影感 |
| 30fps | 普通视频 / 会议 |
| 60fps | 游戏 / 体育 / 高运动画面 |

帧率越高,运动越顺,但编码器每秒要处理更多帧。会议里 30fps 通常够用,低带宽下 15fps 也能接受。屏幕共享文档时,5-15fps 可能就够。

> 别无脑 60fps。对视频会议来说,稳定 24/30fps 比不稳定 60fps 好。

---

## 三、采样率:音频每秒多少点

常见采样率:

| 采样率 | 场景 |
| --- | --- |
| 8000 Hz | 电话窄带语音 |
| 16000 Hz | 语音识别常见 |
| 44100 Hz | CD / 音乐 |
| 48000 Hz | 视频 / WebRTC 常见 |

采样率不是越高越好。语音通话用 48kHz 是工程统一,不代表人声需要这么多信息。真正影响语音体验的通常是回声消除、降噪、丢包恢复和码率。

---

## 四、码率:压缩后每秒给多少比特

码率是编码器预算:

```text
码率高 -> 细节多,文件大,带宽贵
码率低 -> 文件小,容易糊,块状噪声明显
```

粗略经验:

| 内容 | 常见码率 |
| --- | --- |
| 语音 Opus | 16-64 kbps |
| 720p H.264 | 1-3 Mbps |
| 1080p H.264 | 3-6 Mbps |
| 4K H.265 / AV1 | 12-30 Mbps |

这些不是定律。动画、PPT、电影、游戏画面需要的码率差别很大。

---

## 五、CBR、VBR、ABR

| 模式 | 含义 | 适合 |
| --- | --- | --- |
| CBR | 尽量固定码率 | 直播上行、带宽固定场景 |
| VBR | 复杂画面多给,简单画面少给 | 点播文件 |
| ABR | 目标平均码率 | 点播 / 转码常用 |

点播优先 VBR / CRF,直播更常用 CBR 或受控 ABR。因为直播怕瞬时码率把上行或 CDN 边缘打爆。

---

## 六、CRF:给质量,不是给码率

x264 / x265 常用 CRF:

```bash
ffmpeg -i input.mp4 -c:v libx264 -crf 23 -preset medium output.mp4
```

CRF 越小质量越高、文件越大。常用范围:

```text
18  质量高
23  默认附近
28  更小但更糊
```

CRF 适合点播,不适合强约束直播。直播需要明确控制输出码率。

---

## 七、踩坑提醒

1. **只看分辨率不看码率**——1080p 低码率一样糊。
2. **只看平均码率**——峰值码率过高也会卡。
3. **会议无脑高帧率**——低带宽下先保音频,再保分辨率。
4. **把 CRF 用在直播上**——输出码率不可控,容易爆上行。
5. **忽略内容类型**——PPT、电影、游戏、体育的码率模型不同。
