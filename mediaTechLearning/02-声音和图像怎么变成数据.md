# 声音和图像怎么变成数据

音视频工程第一步不是 FFmpeg,是把"连续世界"变成"离散数字"。声音是空气压力随时间变化,图像是光在二维平面上的分布。计算机不能存连续变化,只能按固定间隔采样。

> 一句话先记住:**音频是按时间采样的一串数字,视频是按时间排列的一串图片**。后面所有编码、传输、播放,都在处理这两种序列。

---

## 一、音频:空气压力变成 PCM

麦克风采到的是电信号,声卡把它变成数字:

```text
空气振动 -> 麦克风电压 -> ADC -> PCM 样本
```

PCM 可以理解成最朴素的音频数据:

```text
第 0 个采样点:  120
第 1 个采样点:  180
第 2 个采样点:  160
第 3 个采样点: -40
...
```

常见参数:

| 参数 | 例子 | 意思 |
| --- | --- | --- |
| 采样率 | 48000 Hz | 每秒采多少个点 |
| 位深 | 16-bit | 每个点用多少位表示 |
| 声道 | 1 / 2 | 单声道 / 双声道 |
| 编码 | PCM / AAC / Opus | 原始样本或压缩码流 |

48kHz、16-bit、双声道 PCM 的码率:

```text
48000 x 16 x 2 = 1,536,000 bit/s = 1.536 Mbps
```

这就是为什么语音通话不会直接传 PCM。Opus 可以把语音压到几十 kbps。

---

## 二、视频:一秒几十张图

视频就是一串按时间排列的帧:

```text
Frame 0  ->  0ms
Frame 1  ->  33ms
Frame 2  ->  66ms
...
```

30fps 表示每秒 30 帧。每一帧本质是一张图片。最直观的 RGB 1080p 一帧:

```text
1920 x 1080 x 3 bytes = 6.22 MB
```

30fps 就是 186MB/s。原始视频不压缩没法传。

---

## 三、RGB 不适合直接压视频

屏幕显示常用 RGB,但视频编码更常用 YUV:

| 分量 | 含义 |
| --- | --- |
| Y | 亮度 |
| U / Cb | 蓝色色度差 |
| V / Cr | 红色色度差 |

人眼对亮度敏感,对颜色细节不那么敏感。所以可以保留完整 Y,降低 U/V 分辨率。最常见是 YUV 4:2:0:

```text
4 个像素共享一组 U/V 色度信息
亮度全保留,颜色降采样
```

这不是玄学,是视频压缩的底层收益之一。

---

## 四、时间戳比帧号重要

音视频不是只靠"第几帧"播放,而是靠时间戳:

| 字段 | 作用 |
| --- | --- |
| PTS | 什么时候显示 / 播放 |
| DTS | 什么时候送给解码器 |

有 B 帧时,解码顺序和显示顺序可能不同。播放器要按 PTS 播,解码器要按 DTS 吃数据。

> 很多音画不同步、HLS 分片异常、播放器拖动出问题,根因不是"文件坏了",而是时间戳乱了。

---

## 五、最小排查命令

先别急着转码,先看流:

```bash
ffprobe -hide_banner input.mp4
ffprobe -show_streams -show_format input.mp4
```

重点看:

```text
codec_name
codec_type
width / height
pix_fmt
sample_rate
channels
duration
bit_rate
time_base
```

看不懂 `ffprobe`,就等于闭眼修音视频。

---

## 六、踩坑提醒

1. **把音频当成小事**——会议里音频比视频重要。
2. **把 RGB 当视频内部格式**——大多数视频编码链路会转到 YUV。
3. **忽略采样率转换**——44.1kHz 和 48kHz 混用会引出同步问题。
4. **不知道 PTS / DTS**——遇到 B 帧、切片、拖动时一定会踩坑。
5. **直接传原始帧**——除非在本机内存里,否则一定会爆带宽。
