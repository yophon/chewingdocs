# 音视频工程系列写作计划

> 这个系列不是"FFmpeg 命令大全",也不是"WebRTC API 速查"。目标是把音视频工程里最容易混的几层拆开:采集、编码、封装、传输、播放、实时通话、直播、录制、转码、排障。
> 
> 一句话:**音视频工程 = 把连续的声音和画面压成可传输的数据,再在另一端按时间重新还原出来**。难点不在"能播",难点在延迟、卡顿、同步、兼容、带宽、成本和故障定位。

---

## 设计思路

四层结构,从本地文件一路走到实时通信:

```text
基础模型(声音 / 图像 / 时间轴 / 码率)
   -> 文件与点播(容器 / 编码 / FFmpeg / HLS / 播放器)
   -> 实时通信(WebRTC / SDP / ICE / RTP / SFU)
   -> 生产工程(直播 / 录制 / 质量评估 / CDN / 成本 / 排障)
```

不重复 networkLearning 已经讲过的网络基础,也不重复前端系列里浏览器 API 的泛讲。这里只关心音视频链路自己的问题:为什么同一个 MP4 有的浏览器不能播、为什么 WebRTC 本地通了线上不通、为什么直播延迟从 3 秒变成 30 秒、为什么转码机器 CPU 打满、为什么声音和画面对不上。

## 节奏

```text
01-06 是"看懂任何音视频问题"的地基
07-15 是"能处理文件和点播"的核心
16-24 是"能做实时通话和会议"的核心
25-30 是"上线后不被卡顿、延迟、账单打爆"的工程化部分
```

**一个硬指标**:看完 24 篇,你应该能解释清楚一对一 WebRTC 通话从 `getUserMedia` 到远端出画面中间发生了什么,并且知道 TURN、SDP、ICE、RTP、SFU 各自解决哪一段问题。

## 完结(2026/06/08)

30 篇主线完成。下一步不要继续堆概念,应该做一个项目闭环:

```text
小型视频会议 + 录制回放系统
  WebRTC 一对一
  -> TURN 兜底
  -> SFU 多人房间
  -> 屏幕共享
  -> 服务端录制
  -> FFmpeg 转码
  -> HLS 回放
  -> 质量指标面板
```
