# 屏幕共享、录制与回放

屏幕共享和录制看起来只是多一个 Track,实际会牵扯权限、帧率、分辨率、编码、存储和回放链路。

> 一句话先记住:**屏幕共享是采集问题,录制是媒体归档问题,回放是点播问题**。三件事不要混在一个模块里。

---

## 一、屏幕共享

```js
const displayStream = await navigator.mediaDevices.getDisplayMedia({
  video: { frameRate: 15 },
  audio: true
})
```

屏幕共享特点:

- 文字清晰比帧率重要
- 码率峰值可能很高
- 用户可随时停止
- 系统音频支持不统一

监听结束:

```js
displayStream.getVideoTracks()[0].addEventListener("ended", () => {
  stopScreenShare()
})
```

---

## 二、切换到屏幕共享

```js
const sender = pc.getSenders().find((s) => s.track?.kind === "video")
await sender.replaceTrack(displayStream.getVideoTracks()[0])
```

结束后换回摄像头:

```js
await sender.replaceTrack(cameraStream.getVideoTracks()[0])
```

不要为了切屏幕共享重建整个 PeerConnection。

---

## 三、录制三种方案

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 客户端 MediaRecorder | 简单 | 不可信,格式不统一 |
| 服务端分轨录制 | 质量好,可后处理 | 工程复杂 |
| Headless 浏览器录屏 | 接近用户视角 | 资源重,可控性一般 |

会议产品通常要服务端录制。客户端录制适合个人工具或辅助能力。

---

## 四、分轨录制

```text
audio_user_a.opus
video_user_a.ivf
audio_user_b.opus
video_user_b.ivf
```

后处理:

```text
同步时间线 -> 混音 -> 合成布局 -> 编码 MP4 -> 切 HLS
```

难点是时间同步和缺失片段处理。

---

## 五、回放链路

```text
录制原始文件
  -> 转码 MP4
  -> 生成封面
  -> 切 HLS
  -> 上传对象存储
  -> CDN 分发
  -> 播放器播放
```

回放就是点播系统。不要拿会议实时链路直接当回放链路。

---

## 六、踩坑提醒

1. **录制最后才做**——会议架构要提前考虑录制。
2. **屏幕共享用 30/60fps**——文档共享浪费带宽。
3. **客户端录制当权威记录**——用户关页面就没了。
4. **分轨不同步**——后期合成音画错位。
5. **回放不转 HLS**——大文件播放体验差。
