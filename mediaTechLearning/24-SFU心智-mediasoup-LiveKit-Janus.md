# SFU 心智:mediasoup / LiveKit / Janus

SFU 的核心不是"转发视频",而是管理一堆 RTP 流、订阅关系和质量反馈。

> 一句话先记住:**SFU 不混画面,它决定谁的哪些 RTP 层转给谁**。会议体验好不好,很大程度取决于 SFU 的订阅、带宽估计和降级策略。

---

## 一、SFU 基本对象

不同项目命名不同,但抽象类似:

```text
Room
  Peer / Participant
    Producer   上传的音视频
    Consumer   订阅的远端音视频
Transport      WebRTC 传输通道
```

客户端上传 Track,服务端创建 Producer。其他人想看,就创建 Consumer。

---

## 二、SFU 转发什么

SFU 主要处理:

- RTP 包转发
- RTCP 反馈
- 带宽估计
- Simulcast / SVC 层选择
- 关键帧请求
- 订阅关系

它通常不做:

- 解码画面
- 合成布局
- 美颜滤镜

这些要么在客户端做,要么另接录制 / 转码服务。

---

## 三、mediasoup

特点:

- Node.js 控制面 + C++ worker
- 抽象底层,灵活
- 适合自研会议系统

代价:

- 你要自己写房间、权限、信令、状态同步
- 学习曲线高

适合需要深度定制的团队。

---

## 四、LiveKit

特点:

- 产品化程度高
- 房间、权限、SDK、录制、云部署生态完整
- 上手快

代价:

- 深度定制要跟框架走
- 部署和成本模型要理解

适合快速做会议、语音房、互动直播。

---

## 五、Janus

特点:

- C 写的 WebRTC server
- 插件化
- 历史久,适合网关类场景

常见用途:

- WebRTC gateway
- SIP / RTP / Streaming 适配
- 老系统接入

---

## 六、录制怎么做

SFU 转发不等于录制。录制常见方案:

```text
1. 服务端订阅所有流 -> 分轨录制 -> 后处理合成
2. 启动 headless browser -> 像用户一样进房间 -> 录屏
3. SFU 提供 egress 服务 -> 输出 MP4 / HLS
```

分轨灵活,后处理复杂。浏览器录屏简单,资源重。

---

## 七、踩坑提醒

1. **以为 SFU 会自动混流**——不会。
2. **不做活跃说话人和订阅控制**——下行浪费。
3. **录制方案最后才想**——架构会返工。
4. **忽略 TURN 和边缘部署**——跨地域延迟高。
5. **没有 getStats + 服务端 RTP 指标**——质量问题查不动。
