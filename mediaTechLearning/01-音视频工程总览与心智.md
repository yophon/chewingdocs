# 音视频工程总览与心智

学音视频最大的障碍不是 API 多,是**层次混在一起**。新手一上来搜 "WebRTC 教程",结果同时撞上摄像头权限、SDP、ICE、STUN、TURN、RTP、编解码、NAT、SFU、HLS、FFmpeg。每个词都能查到文章,但串不成链路。这一篇不教命令,只讲一件事:**音视频工程到底在解决什么问题**。

> 一句话先记住:**音视频工程 = 按时间采集连续信号,压缩成码流,装进容器或包,通过网络或文件传过去,再按时间解码和播放**。所有问题都绕这条链路转:清不清楚、卡不卡、延迟高不高、声音画面对不对、带宽够不够、机器扛不扛得住。

---

## 一、音视频不是"一个文件",是一条时间线

普通文本文件没有时间。你打开一个 Markdown,第 1 行和第 100 行谁先显示无所谓。音视频不一样,它的核心是**时间**。

```text
音频: 0ms  20ms  40ms  60ms  80ms ...
视频: 0ms  33ms  66ms  99ms 132ms ...  (30fps)
```

播放器不是"把文件读出来显示",而是不断做三件事:

1. 读一点数据
2. 解码出音频样本和视频帧
3. 在正确时间把它们送到声卡和屏幕

所以音视频问题第一类就是**时间线问题**:

- 播放卡顿:数据来晚了,解码来不及,或者缓冲不够
- 音画不同步:音频时钟和视频时钟对不上
- 直播延迟高:缓冲越堆越多,播放器落后真实时间
- 会议回声:麦克风又采到了扬声器出来的声音

> 先把"音视频是时间线"刻进脑子。后面讲帧率、采样率、码率、GOP、RTP timestamp、jitter buffer,全是这件事的推论。

---

## 二、完整链路:从摄像头到屏幕

一条最完整的链路长这样:

```text
采集
  摄像头 / 麦克风
  得到原始画面帧和 PCM 音频
    ↓
编码
  H.264 / H.265 / VP9 / AV1 / AAC / Opus
  把巨大原始数据压成码流
    ↓
封装或打包
  文件:MP4 / WebM / FLV
  实时:RTP packet
  点播:HLS / DASH 分片
    ↓
传输
  本地文件 / HTTP / CDN / RTMP / SRT / WebRTC
    ↓
解封装 / 解包
  拿回音频包、视频包、时间戳
    ↓
解码
  码流 -> PCM / YUV / RGB
    ↓
播放
  声卡按采样点播放
  屏幕按帧刷新
```

所有工具都只是在这条链路上占一段:

| 工具 / 协议 | 主要位置 |
| --- | --- |
| FFmpeg | 采集、编码、封装、转码、切片、推流 |
| HLS / DASH | 点播和直播分片分发 |
| WebRTC | 低延迟实时采集、传输、播放 |
| RTMP / SRT | 推流和直播上行 |
| SFU | 多人会议里的媒体转发 |
| CDN | 分发 HLS / DASH / FLV 等内容 |

> 别把 FFmpeg、WebRTC、HLS 当成同一层的竞争关系。它们经常在同一个系统里同时出现:WebRTC 做低延迟通话,FFmpeg 做录制转码,HLS 做回放和大规模分发。

## 三、原始数据大到离谱,所以必须编码

先算一笔账。1080p、30fps、RGB 原始视频:

```text
1920 x 1080 x 3 bytes x 30 fps
= 186 MB/s
≈ 1.49 Gbps
```

这还不算音频。一个普通 1080p 视频如果不压缩,家用网络根本传不动,硬盘也很快写爆。所以音视频工程的第一条铁律是:

```text
原始音视频不能直接存和传,必须编码压缩。
```

编码器做的事不是"压缩 zip",而是利用人眼和人耳的弱点:

- 相邻视频帧大多相似,只存变化
- 人眼对颜色细节不如亮度敏感,可以做色度降采样
- 人耳听不到的频段可以丢
- 运动区域和静态区域用不同策略压

这就是为什么会有 H.264、H.265、VP9、AV1、AAC、Opus。它们不是文件格式,它们是**编码格式**。

---

## 四、容器和编码必须分开想

新手最常见错误:**把 MP4 当成 H.264**。

这两个不是一回事:

| 概念 | 例子 | 解决什么 |
| --- | --- | --- |
| 编码格式 | H.264、H.265、VP9、AV1、AAC、Opus | 数据怎么压缩 |
| 容器格式 | MP4、MKV、FLV、WebM、MOV | 音频、视频、字幕、时间戳怎么装在一起 |

一个 MP4 文件里可以装 H.264 + AAC,也可以装 H.265 + AAC。一个 WebM 通常装 VP9 / AV1 + Opus。播放器能不能播,取决于两件事:

```text
播放器支持这个容器吗?
播放器支持容器里的编码吗?
```

这解释了很多怪问题:

- 文件后缀是 `.mp4`,但浏览器播不了:可能里面是 H.265
- VLC 能播,Chrome 不能播:VLC 自带解码器,浏览器受平台和专利限制
- 转封装很快:只是换容器,不重新编码
- 转码很慢:要先解码再编码,CPU / GPU 真干活

> 后面 04-05 篇会专门拆容器和编码。先记住:**MP4 不是编码,H.264 不是文件格式**。

## 五、点播、直播、通话是三种不同问题

音视频系统大致分三类,不要混着设计。

### 5.1 点播:允许慢一点,追求稳定和成本

典型场景:视频网站、课程回放、录播。

```text
上传 -> 转码 -> 切 HLS / DASH -> CDN -> 播放器
```

点播不怕 10 秒准备时间,怕的是:

- 首屏慢
- 中途卡
- 清晰度切换不稳
- 存储和 CDN 费用高
- 多端兼容差

点播的关键是**预处理**:提前转好多档清晰度,切好分片,让播放器按网络情况选。

### 5.2 直播:允许几秒延迟,追求规模和稳定

典型场景:秀场直播、赛事直播、带货。

```text
主播推流 -> 转码 / 转封装 -> CDN 分发 -> 观众播放
```

直播的问题是:

- 上行不稳
- 观众规模大
- 延迟要可控
- 断流要恢复
- 热门房间 CDN 带宽贵

直播通常不追求 200ms 延迟。很多业务 3-10 秒可以接受,换来更好的 CDN 分发和播放稳定性。

### 5.3 通话:延迟是第一指标

典型场景:视频会议、语音通话、在线教室。

```text
采集 -> 编码 -> WebRTC -> jitter buffer -> 解码 -> 播放
```

通话的问题是:

- 200ms 以上就明显难受
- NAT 穿透失败
- 弱网丢包
- 回声、噪声、音画同步
- 多人会议带宽爆炸

通话不能靠"多缓冲几秒"解决问题,因为缓冲本身就是延迟。它要靠 WebRTC 的拥塞控制、丢包恢复、jitter buffer、回声消除、SFU 转发。

## 六、WebRTC 最难的不是 API,是网络现实

一个最小 WebRTC demo 看起来很简单:

```js
const stream = await navigator.mediaDevices.getUserMedia({ video: true, audio: true })
const pc = new RTCPeerConnection()
stream.getTracks().forEach((track) => pc.addTrack(track, stream))
```

但真上线会立刻遇到:

- 两个用户都在 NAT 后面,直连不通
- 公司网络屏蔽 UDP
- SDP 交换顺序错了,双方协商失败
- 本地能通,跨城市不通
- 不上 TURN 时一部分用户永远黑屏
- 多人会议 Mesh 带宽炸掉

WebRTC 的核心链路是:

```text
Signaling 交换 SDP
  -> ICE 收集候选地址
  -> STUN 尝试发现公网映射
  -> TURN 在直连失败时中继
  -> DTLS 建安全通道
  -> SRTP 传音视频包
  -> RTCP 回传质量反馈
```

> WebRTC API 只是入口。真正的工程难点是 ICE、TURN、RTP、拥塞控制和 SFU。只会 `getUserMedia` 不算会 WebRTC。

## 七、FFmpeg 是瑞士军刀,但别把它当黑盒

FFmpeg 能做很多事:

```bash
ffmpeg -i input.mov output.mp4
ffmpeg -i input.mp4 -c copy output.mkv
ffmpeg -i input.mp4 -vf scale=1280:-2 -c:v libx264 output.mp4
ffprobe -show_streams input.mp4
```

但你必须知道每条命令在链路上动了哪一层:

| 操作 | 是否重新编码 | 代价 |
| --- | --- | --- |
| 转封装 | 否 | 快,主要是 IO |
| 转码 | 是 | 慢,吃 CPU / GPU |
| 切 HLS | 可能 | 取决于是否复制码流 |
| 抽帧 | 需要解码 | 中等 |
| 加水印 | 需要解码 + 编码 | 慢 |

FFmpeg 排错的第一步永远是 `ffprobe`。先看容器、编码、时长、码率、帧率、时间戳,再决定要不要转码。

## 九、踩坑提醒(总览版)

1. **把 MP4 当 H.264**——容器和编码不是一层。
2. **一上来就写 WebRTC 业务代码**——先搞清 SDP / ICE / TURN,否则本地 demo 会骗你。
3. **不知道转封装和转码的区别**——`-c copy` 是救命参数,乱转码会把机器打爆。
4. **直播用会议思路做**——大规模直播靠 CDN,不是每个观众都进 WebRTC 房间。
5. **会议用直播思路做**——通话不能堆 5 秒缓冲,延迟会毁掉体验。
6. **不看时间戳**——音画不同步、分片异常、播放器卡顿,很多根因都在 PTS / DTS。
7. **不上 TURN**——WebRTC 没有 TURN 就不是生产系统,只是运气好的 demo。
8. **忽略音频**——会议体验里音频比视频更重要,视频糊还能忍,声音断没人能忍。
9. **只看平均码率**——峰值码率、GOP、分片大小才决定很多卡顿。
10. **没有质量指标**——不上报 RTT、丢包、jitter、卡顿、首帧时间,线上问题只能靠猜。
