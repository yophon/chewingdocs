# 直播系统:RTMP / SRT / WebRTC / HLS

直播不是视频会议放大版。直播的主目标是规模、稳定和成本,低延迟只是其中一个取舍项。

> 一句话先记住:**直播链路 = 推流 -> 转码/转封装 -> 分发 -> 播放**。RTMP/SRT/WebRTC 常在上行,HLS/FLV/WebRTC 常在下行。

---

## 一、标准直播链路

```text
主播端 OBS / App
  -> 推流网关
  -> 转码 / 截帧 / 审核
  -> CDN 分发
  -> 播放器
```

大直播系统还会加:

- 鉴权
- 断流重连
- 多码率转码
- 录制回放
- 实时截图审核
- 质量监控

---

## 二、RTMP

RTMP 是直播上行老标准:

```text
OBS -> rtmp://server/live/streamKey
```

优点:

- 推流工具支持广
- 生态成熟
- 延迟可控

缺点:

- 浏览器原生不播 RTMP
- 现代低延迟和弱网能力不如新协议

现在常见做法是 RTMP 推流,服务端转 HLS / FLV / WebRTC 给观众。

---

## 三、SRT

SRT 面向不稳定公网传输,比 RTMP 更重视抗丢包和远距离传输。

适合:

- 远程制作
- 跨地域上行
- 专业直播源回传

不适合:

- 普通浏览器直接播放

---

## 四、WebRTC 直播

WebRTC 下行可做到亚秒级延迟。适合:

- 连麦
- 拍卖
- 在线课堂互动
- 低延迟赛事互动

代价:

- CDN 生态不如 HLS 成熟
- SFU/边缘节点成本高
- 大规模分发复杂

不要为了"低延迟"把所有直播都改 WebRTC。先问业务是否真的需要。

---

## 五、HLS 直播

HLS 普通直播延迟通常几秒到十几秒,但:

- CDN 友好
- 兼容性好
- 成本可控
- 大规模分发稳

秀场、课程、带货很多场景几秒延迟可接受。稳定比极低延迟更值钱。

---

## 六、踩坑提醒

1. **用会议架构做万人直播**——成本和带宽会炸。
2. **所有场景追亚秒延迟**——业务未必需要。
3. **只做单码率直播**——弱网用户体验差。
4. **不录制**——直播结束没有回放资产。
5. **推流鉴权弱**——别人能推垃圾流进你的房间。
