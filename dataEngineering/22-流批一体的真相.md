# 流批一体的真相:Flink、Spark Structured Streaming、工程选型

「一份代码,既能跑实时也能跑批回算」——这是 Flink 和 Spark 都在喊了 5 年的口号。**真相是:理论上对了一半,实操上要看你怎么选**。Flink 走 Kappa 思路(批是流的特例),Spark 走批主导思路(流是微批)。这一篇拆清楚两家的实现差异、流批一体的"理想 vs 现实"、以及工程选型时怎么避开口水战做选择。

> 一句话先记住:**「流批一体」是「一份代码 + 一份语义」的双重承诺**。Flink 把它推到极致(DataStream 流批共用、Table API SQL 流批共用),Spark Structured Streaming 把流当微批简化。**「一份代码」做到了,「完全一份语义」仍有缝**——批模式没有 Watermark、迟到处理、retract 流。**实操选型按团队栈、延迟要求、流批比例选,不要为了"一体"而一体**。

---

## 一、流批一体在解决什么问题

### 1.1 没流批一体的世界(Lambda 时代,04 篇)

```
同一段业务逻辑(每天 GMV)写两遍:

批层:
  Spark SQL 跑 T+1 离线
  逻辑:SELECT date, SUM(amount) FROM orders WHERE date=...
  
速度层:
  Storm / Spark Streaming
  逻辑:Java 代码:Map → reduceByKey,5 分钟窗口
  
问题:同一个业务,两份代码,两个团队,结果漂移
```

### 1.2 流批一体的承诺

```
一份代码 + 一份语义:
  同一个 SQL / 同一个 DataStream
  跑批模式 → 离线回算历史
  跑流模式 → 实时持续更新
  
理想:代码改一处,两侧同步
现实:90% 一致,10% 有缝(下面展开)
```

## 二、Flink 流批一体:Kappa 思路

### 2.1 哲学:批是流的特例

```
有界数据 = 有界流(读完就停)
无界数据 = 无界流(永远不停)
```

Flink 用同一套 API(DataStream / Table)跑两种,通过 `execution.runtime-mode` 切换:

```python
StreamExecutionEnvironment.create()
    .setRuntimeMode(RuntimeExecutionMode.BATCH)  # 批
    # 或
    .setRuntimeMode(RuntimeExecutionMode.STREAMING)  # 流
```

### 2.2 流批两种模式的差异

**Flink 内部针对批模式做的优化**:
- 关 Checkpoint(批没必要,失败重跑)
- Sort-Shuffle 替代 Pipeline-Shuffle(写盘换并行度)
- BlockingResultPartition(算子之间也可阻塞,跑完一个再跑下一个,降低内存压力)
- Optimizer 知道是有界数据,可重排执行顺序

### 2.3 Flink SQL 流批一体

```sql
-- 同一段 SQL
CREATE TABLE source_orders (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2),
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '1' MINUTE
) WITH ('connector' = '...');

SELECT
    TUMBLE_START(event_time, INTERVAL '1' DAY) AS day,
    province,
    SUM(amount) AS gmv
FROM source_orders
GROUP BY 
    TUMBLE(event_time, INTERVAL '1' DAY),
    province;

-- 流模式:对接 Kafka,Watermark 触发窗口,持续输出
-- 批模式:对接 Iceberg / Parquet,跑完一遍输出
```

### 2.4 流批模式的语义差(关键!)

虽然「**一份代码**」做到了,但**语义不完全一致**:

| 行为 | 流模式 | 批模式 |
| --- | --- | --- |
| Watermark | 必须配,驱动窗口 | 无 Watermark 概念(数据全到了) |
| 迟到事件 | 有 Allowed Lateness | 不存在(批模式假定全到) |
| 早期触发 | 可配 EarlyTrigger | 无(等全部聚完) |
| Retract 流 | 可发更新(`+I/-U/+U/-D`) | 直接 append |
| 状态算子 | 状态后端持久化 | 内存 / 临时盘,跑完释放 |

**举例**:一个 `ROW_NUMBER() OVER (...)`,流模式下随着新数据到来不断更新 row_number;批模式下一次性算完。结果可能一致(整体看),但**中间状态完全不同**。

### 2.5 Flink 流批一体能力的现状

```
DataStream API:       几乎完全统一(2021+)
Table API / SQL:       SQL 流批一体最完整
ML / 复杂 CEP:         仍以流为主
状态算子:              批模式优化中
```

## 三、Spark Structured Streaming:微批主导

### 3.1 哲学:流是批的特例

```
微批(Micro-Batch):  把流切成"每 5 秒一批",每批跑批处理
连续模式(Continuous):真流(实验性,2.3+)
```

Spark 的视角是反过来:**先有批,流是批的小化**。

### 3.2 工作流

```
Kafka 流 →─┬─5 秒攒一批─┬─批跑 SQL─→ Iceberg
           │             │
           └─下一批─────  └─下次跑
```

每 5 秒一个微批,**每个微批本质就是一段 Spark 批 job**。

### 3.3 Spark Structured Streaming SQL

```python
df = (spark
    .readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "kafka:9092")
    .option("subscribe", "orders")
    .load())

result = (df
    .selectExpr("CAST(value AS STRING) as json")
    .selectExpr("json_tuple(json, 'id', 'amount', 'event_time') as (id, amount, event_time)")
    .withWatermark("event_time", "1 minute")
    .groupBy(F.window("event_time", "5 minutes"), "province")
    .agg(F.sum("amount").alias("gmv")))

(result
    .writeStream
    .outputMode("append")
    .trigger(processingTime="5 seconds")  # 微批间隔
    .format("iceberg")
    .option("checkpointLocation", "s3://.../ckpt")
    .toTable("shop.gold.daily_gmv")
    .awaitTermination())
```

### 3.4 同一段代码切批

```python
# 几乎一样的代码,改 readStream → read,writeStream → write
df = spark.read.format("kafka").load()  # 或 .parquet("...")
result = df.transform(...).agg(...)
result.write.format("iceberg").mode("overwrite").save()
```

**Spark 的承诺**:几乎一份代码,但 `readStream/writeStream` 和 `read/write` 是不同 API。

### 3.5 Continuous Mode(真流)

Spark 2.3+ 的 Continuous Mode 是真流,**亚毫秒延迟**:

```python
.trigger(continuous="1 second")
```

但**功能受限**:不支持 stateful aggregation、windowing 等流处理核心算子。**几乎没人在生产用**——大部分场景微批足够。

### 3.6 微批的优劣

| 优 | 劣 |
| --- | --- |
| 简单(直接套 Spark 批引擎) | 延迟秒级(微批间隔 + 批本身) |
| 跟离线 Spark 共享所有优化 | 不适合亚秒级需求 |
| 跟 Iceberg / Delta 集成深 | 状态管理不如 Flink |
| 适合 Databricks 生态 | 复杂 CEP / Pattern 不强 |

## 四、Flink vs Spark Structured Streaming:四维对比

### 4.1 延迟

| | Flink | Spark Structured Streaming |
| --- | --- | --- |
| 默认 | 真流,毫秒 | 微批,秒到分钟 |
| 极限 | 亚毫秒 | 亚秒(Continuous,但受限) |

### 4.2 状态管理

| | Flink | Spark Structured Streaming |
| --- | --- | --- |
| 状态后端 | RocksDB 强(21 篇) | 简化版 RocksDB(3.x+) |
| 大状态 | 几 TB 没问题 | 几百 GB 起来困难 |
| 增量 Checkpoint | 支持 | 支持(3.x+) |
| 状态 TTL | 灵活 | 有但简化 |

### 4.3 复杂处理

| | Flink | Spark Structured Streaming |
| --- | --- | --- |
| CEP / 模式匹配 | 原生 | 弱 |
| 复杂窗口 | 支持 Session、自定义 | 有限 |
| 时间语义 | 完整(Event/Processing/Ingestion) | 完整 |
| 自定义 Function | 灵活 | 灵活 |

### 4.4 生态集成

| | Flink | Spark Structured Streaming |
| --- | --- | --- |
| Kafka | 一等公民 | 一等公民 |
| Iceberg/Delta/Hudi | 完整 | 完整(尤其 Delta) |
| dbt | dbt-flink(社区) | dbt-spark / dbt-databricks(主流) |
| ML | 弱(不是核心) | 跟 Spark MLlib / PyTorch 集成深 |
| 商业支持 | Ververica、Aliyun、AWS MSK | Databricks |

## 五、选型决策:不是「谁更牛」,是「哪个适合」

### 5.1 选 Flink

- **延迟要求极低**(< 100ms)
- **复杂状态 + Watermark + Retract 流**
- **CEP / 风控规则匹配**
- **流是核心业务,批是次要**
- **团队愿意学 Flink(学习曲线陡)**

典型场景:实时风控、推荐近线召回、广告点击实时归因、IoT 实时监控、跨境支付。

### 5.2 选 Spark Structured Streaming

- **微批可接受**(秒级到分钟级)
- **离线 ETL 已经在 Spark,流式只是补充**
- **跟 Iceberg / Delta 集成是核心**
- **团队 Spark 熟练**
- **Databricks 生态**

典型场景:IoT 数据微批落湖、近实时数仓(分钟级延迟)、Iceberg 实时表、用 Delta 做 CDC 落地。

### 5.3 共存(常见!)

```
公司里既有:
  Flink 关键链路(风控、实时大屏、广告归因)   秒以下延迟
  Spark Structured Streaming(数仓 ODS 实时落湖) 5 分钟延迟可接受
  Spark 离线(DWD / DWS 凌晨跑)
  
都跑在同一份 Iceberg 表上(04 篇混合架构)
```

## 六、流批一体的「真相」:一份代码 ≠ 一份语义

### 6.1 现实里的差异

```
Flink:
  流模式 DDL:WATERMARK FOR event_time AS ...
  批模式 DDL:不需要 Watermark(数据全在)
  → 同一张表 DDL 可能要不同写法
  
  流模式:有 retract(更新行)
  批模式:只有 append
  → 下游 sink 处理逻辑可能不一样

Spark:
  流模式:withWatermark + outputMode
  批模式:不需要
  → API 接口差异
```

### 6.2 工程上的妥协

```
理想:写一段代码 → 跑流模式 / 跑批模式
现实:
  1. 写一段 SQL(高度共用)
  2. 但 Watermark / Output Mode 等需要根据模式配
  3. Sink 端可能不同(流要事务,批可直接覆盖)
  4. 测试要双重(流 + 批)
```

**真正完全一份的代码很少**,**90% 共用 + 10% 模式相关** 更准确。

### 6.3 Apache Beam:理论正确但生态薄

Apache Beam 是 Google 推的「**抽象 + Runner 模型**」:

```
代码用 Beam API 写 → 选 Runner:
  Direct Runner(本地)
  Dataflow Runner(GCP)
  Flink Runner
  Spark Runner
  ...
```

- 理论上跨引擎一份代码
- 实际:Google Cloud 之外用得少,各 Runner 的功能支持有差异
- 国内基本不用

**Beam 的承诺正确,生态拼不过 Flink / Spark 各自的深度**。

## 七、工程落地:典型混合架构

### 7.1 一条数据三种处理

```
Binlog ──► Debezium ──► Kafka topic: shop.orders
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
        Flink              Spark              批处理
        实时大屏           Structured          (T+1)
                          Streaming
        WindowAgg          落 Iceberg          dbt + Spark
        → ClickHouse       ODS                建 DWD/DWS/ADS
        延迟 1s            延迟 5min          延迟 1day
                              │
                              ▼
                          Iceberg orders 表(事实源)
                              │
                              ▼
                          所有下游消费
```

**关键**:Flink 处理「秒级实时」,Spark Structured Streaming 处理「分钟级近实时」,**离线 dbt 跑「日级建模」**——三种节奏,根据业务延迟需求选,**不强求统一**。

### 7.2 「真正一份代码」的少数场景

某些场景可以做到比较干净的一份代码:

```sql
-- Flink SQL,流批共用
INSERT INTO daily_summary
SELECT
    DATE_TRUNC('day', event_time) AS day,
    province,
    SUM(amount) AS gmv
FROM orders
GROUP BY 
    DATE_TRUNC('day', event_time),
    province;
```

流模式:source 是 Kafka,持续更新 daily_summary(用 retract 模式)
批模式:source 是 Iceberg,一次性回算

**适用场景**:简单 GROUP BY 聚合、无复杂状态算子、不依赖 Watermark 的窗口。

## 八、当下趋势:声明式实时表

### 8.1 趋势

```
Flink Materialized Table(2024+):
  CREATE MATERIALIZED TABLE daily_summary AS SELECT ... FROM orders
  Flink 自动维护表的物化,流式实时更新
  
Snowflake Dynamic Table(2023):
  类似概念,Snowflake 内部维护
  
Iceberg Streaming View(规划中):
  Iceberg 表声明为流式视图,引擎自动维护

Databricks DLT (Delta Live Tables):
  声明式管道,流批共用 SQL
```

### 8.2 价值

```
用户写:  CREATE MATERIALIZED TABLE X AS SELECT ...

引擎决定:
  - 哪些数据可批,哪些必须流
  - 增量 vs 全量
  - 触发频率
  - 物化策略
```

**「流批一体」的下一个阶段:用户不再操心」**——这是 2025+ 的方向。

## 九、选型 Checklist

```
Q1: 业务能接受多大延迟?
    < 100ms             → Flink
    1-10s               → Flink 或 Spark Continuous(很少)
    几十秒到几分钟       → Spark Structured Streaming 或 Flink 微批
    > 5 分钟             → 批,不需要流处理
    
Q2: 团队主栈?
    Spark / Databricks   → Spark Structured Streaming
    Flink 已有积累       → Flink
    全新项目             → 看 Q1
    
Q3: 状态规模?
    < 100GB              → 都行
    100GB - 1TB          → Flink 体验更好
    > 1TB                → Flink RocksDB 增量 Checkpoint
    
Q4: 复杂度?
    SQL 聚合 + 简单窗口    → 都行
    CEP / Pattern         → Flink
    流式 join 多表         → Flink
    
Q5: 跟离线管道关系?
    流和批语义统一是核心    → Flink SQL
    流是补充,核心仍批      → Spark Structured Streaming(批侧共享 Spark)
```

## 十、几个误区

### 10.1 「流批一体 = 一份代码」

**对一半**。Flink SQL 做得最好,但仍有少数差异。
**Spark Structured Streaming 是「读写 API 不同,中间逻辑共用」**。

### 10.2 「Flink 流批一体 → 取代 Spark」

不会。批场景下 Spark 的优化器(Catalyst)、ML 库、生态(dbt / 数据科学家用)仍然强。**Flink 的批模式在「跟流共用代码」这一点上爽,但纯批场景仍输 Spark**。

### 10.3 「Spark Structured Streaming 取代 Flink」

不会。**Spark 微批的延迟 + 状态管理 + CEP**,真实时场景上不来。

### 10.4 「Apache Beam 一份代码到处跑」

理论上对,实际生态弱、各 Runner 支持有差异、Google 之外用得少。
