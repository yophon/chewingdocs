列存原理:为什么 Parquet 在分析查询上比行存快 10 倍 ​

同样一张 10 亿行的订单表,你跑 SELECT SUM(amount) FROM orders WHERE date='2025-05-01' —— 在 MySQL 上 40 分钟、在 ClickHouse / Spark 读 Parquet 上 3 秒。性能差两个数量级,不是引擎差异,是存储布局差异:行存把一行的字段连续放,列存把同列的所有值连续放。这一个变化连带出一整套优化:压缩比、向量化、谓词下推、统计裁剪——列存不是单一技术,是为「扫描型查询」量身定制的存储 + 执行范式。这一篇画清楚布局图、Parquet 文件结构、三大编码、压缩、谓词下推,把「为什么列存快」拆到骨头。

一句话先记住:行存优化「点查一行的所有字段」,列存优化「全表扫一个字段」。列存赢在四件事:少读 IO(只取你要的列)+ 高压缩比(同类型相邻)+ 向量化执行(SIMD)+ 统计跳过(谓词下推到 row group 元数据)。代价是单行写慢、点查慢、不适合事务——所以 OLTP 仍然用行存,OLAP 才用列存。

一、行存 vs 列存:存储布局对比 ​

1.1 同一张表两种摆法 ​

假设一张订单表四列:id, user_id, amount, status,4 行数据:

逻辑表:
┌─────┬──────────┬────────┬────────┐
│ id  │ user_id  │ amount │ status │
├─────┼──────────┼────────┼────────┤
│ 1   │ 12345    │ 99.50  │ paid   │
│ 2   │ 67890    │ 199.00 │ paid   │
│ 3   │ 12345    │ 88.00  │ created│
│ 4   │ 11111    │ 50.00  │ paid   │
└─────┴──────────┴────────┴────────┘

行存(MySQL / PG)磁盘布局:

[1, 12345, 99.50,  paid    ]  ← 第 1 行
[2, 67890, 199.00, paid    ]  ← 第 2 行
[3, 12345, 88.00,  created ]  ← 第 3 行
[4, 11111, 50.00,  paid    ]  ← 第 4 行

一行紧挨着一行,读 user_id=12345 的整行只要一次 IO;但读所有行的 amount 要把所有行都加载,99% 数据浪费。

列存(Parquet / ClickHouse)磁盘布局:

id     列: [1, 2, 3, 4]
user_id列: [12345, 67890, 12345, 11111]
amount 列: [99.50, 199.00, 88.00, 50.00]
status 列: [paid, paid, created, paid]

读 SUM(amount) 只读 amount 列,IO 量直接降 75%(4 列只读 1 列);但读 user_id=12345 的整行,要分别访问 4 个列文件按行号拼接,单行查询慢。

1.2 性能差的根本来源 ​

列存对分析查询快 10 倍,不是一个技术加成,是四件事叠加:

加成原理倍数(典型)
列裁剪只读 query 需要的列表 50 列只用 5 列 → 10 倍
高压缩比同列同类型,字典 / RLE / Delta 压缩5-10 倍
向量化执行一次处理一批同类型值,SIMD 友好3-5 倍
谓词下推 + 统计跳过看 row group 的 min/max 跳过整块极端情况 100 倍

四个加成不是简单乘法,但叠加效果在生产分析负载上经常达到 10-100 倍。

2.2 四层各自的角色 ​

File          一个 .parquet 文件
                ↓
Row Group     水平切片,~128MB 一个,
              是「并行单位」和「下推单位」
                ↓
Column Chunk  垂直切片,每个 Row Group 里每列一个 Chunk,
              是「IO 单位」(读哪几列就读哪几个 Chunk)
                ↓
Page          Column Chunk 内部进一步切,~1MB 一个,
              是「编码 / 压缩单位」和「读时缓存单位」

2.3 Footer 在文件末尾的好玩之处 ​

Parquet 把 metadata 放在文件末尾,而不是开头:

  • 写时:数据顺序追加,footer 最后写——写入流式友好
  • 读时:先读末尾 4 字节(footer length),seek 回去读 footer,再按 offset 取需要的列——读时只 IO 实际需要的部分

这跟 Avro 的「schema 写在最前」是相反思路——列存的读模式假设你不会读整个文件,所以 metadata 放末尾,跳着读。

三、三大编码:在落盘前压缩 ​

Parquet 在压缩之前先做 encoding(编码),编码后的数据再用 Snappy / Zstd 等通用压缩算法压一次。两层叠加才有 5-10 倍压缩比。

3.1 Dictionary Encoding(字典编码) ​

原始: ["paid", "paid", "created", "paid", "paid", "shipped", "paid"]
       (7 个字符串,假设每个平均 8 字节 = 56 字节)

字典: {0: "paid", 1: "created", 2: "shipped"}
编码: [0, 0, 1, 0, 0, 2, 0]
       (7 个整数,1 字节足够 = 7 字节,加字典 ~24 字节 = 31 字节)

压缩比 ~ 1.8 倍(数据越多优势越大,因为字典是常数开销)

适用:取值数量少(<= 几万种)的列。典型场景:status(订单状态)、category(品类)、country、boolean 列。

Parquet 默认对每列都尝试 dictionary 编码,字典超过阈值(默认 1MB)就 fallback 到 plain。

3.2 RLE(Run-Length Encoding,游程编码) ​

原始: [paid, paid, paid, paid, paid, created, paid, paid, paid]
       (9 个值)

RLE:  [(paid, 5), (created, 1), (paid, 3)]
       (3 个元组 = ~12 字节)

压缩比 ~ 数据越重复越高

适用:大量连续重复的列。Parquet 中常和 dictionary 组合:dictionary 把字符串变小整数,再 RLE 压重复整数。

status (dict encoded): [0, 0, 0, 0, 0, 1, 0, 0, 0]
RLE:                   [(0,5), (1,1), (0,3)]

3.3 Delta Encoding(差分编码) ​

原始(单调递增的 timestamp 或 id):
       [1715000000, 1715000001, 1715000002, 1715000005, 1715000007]

Delta: [1715000000, +1, +1, +3, +2]
        (除第一个外都是小差值,bit-packing 后空间极小)

压缩比 ~ 高(差值小、bit-packing 友好)

适用:id(自增主键)、event_time、created_at、版本号等。

3.4 编码选择是自动的 ​

Parquet writer 默认按 hint 选择:

  • 字符串 / 整数 → 优先 Dictionary
  • 重复值多 → 加 RLE
  • 单调递增 → Delta + Bit-Packing
  • 编码失败 / 不划算 → 回落 Plain(直接存原值)

写代码时一般不需要手动配,但读 Parquet metadata 时可以看到每个 Column 用了哪种编码。

四、压缩:编码之上再压一次 ​

编码后是字节流,再过通用压缩:

算法速度压缩比现状
Snappy极快中等Parquet 历史默认
Zstd快高2024 后 Parquet 推荐默认
Gzip慢高兼容用,新写少用
LZ4极快低偶尔用
Brotli慢极高静态归档用
None//测试时用
python
# PyArrow 写时指定
import pyarrow.parquet as pq
pq.write_table(
    table, 
    "orders.parquet",
    compression="zstd",        # 默认 snappy,推荐 zstd
    compression_level=3,        # zstd 1-22,3 是甜点
    row_group_size=128 * 1024 * 1024,  # 128MB
    use_dictionary=True,
)

Zstd vs Snappy 实测:

  • 解压速度差距小(Zstd 慢 10-20%)
  • 压缩比 Zstd 高 30-40%
  • IO bound 的查询用 Zstd 整体快(读得少)
  • CPU bound 的查询用 Snappy 略快

2024 后业界趋势是默认 Zstd(Apache Parquet 1.13+ 推荐)。

五、谓词下推:列存的杀手锏 ​

5.1 没下推的世界 ​

sql
SELECT SUM(amount) FROM orders WHERE date = '2025-05-01' AND status = 'paid';

如果不用统计裁剪,引擎必须读完所有 Row Group 的 date、status、amount 三列,过滤后求和。

5.2 有下推的世界 ​

Parquet footer 每个 Row Group 记录每列的 min/max/null_count:

Row Group 1:  date  min=2025-01-01  max=2025-01-31
Row Group 2:  date  min=2025-02-01  max=2025-02-28
Row Group 3:  date  min=2025-03-01  max=2025-03-31
Row Group 4:  date  min=2025-04-01  max=2025-04-30
Row Group 5:  date  min=2025-05-01  max=2025-05-15
Row Group 6:  date  min=2025-05-16  max=2025-05-31

Query: WHERE date = '2025-05-01'
       ↓
       只 Row Group 5 可能满足 → 跳过其他 5 个 RG
       IO 量减少 5/6 ≈ 83%

5.3 下推到 Page 级 ​

更细粒度:每个 Page 也有 min/max,过滤可以下推到 Page。实际生产中开启 PageIndex(Parquet 2.9+):

Row Group 5
├─ Page 1: date min=2025-05-01 max=2025-05-03   ← 命中
├─ Page 2: date min=2025-05-04 max=2025-05-07   ← 跳
├─ Page 3: date min=2025-05-08 max=2025-05-11   ← 跳
└─ Page 4: date min=2025-05-12 max=2025-05-15   ← 跳

5.4 BloomFilter:补统计的不足 ​

statistics 对范围查询(>, <, = 时连续值)效果好,但对高基数等值查询(WHERE user_id = 12345)效果差——min/max 范围太宽,几乎不能跳。

Parquet 1.13+ 支持 Bloom Filter:每个 Row Group 可建一个 Bloom,查询 user_id = 12345 时先问 Bloom「这个 RG 可能有吗?」,Bloom 说没有 → 跳过整个 RG。

python
pq.write_table(table, "orders.parquet",
    bloom_filter_columns=["user_id"],
)

5.5 必要条件:数据按下推字段排序 ​

谓词下推的效果取决于数据在文件里的物理排序:

排好序的 date 列:
  RG1: 2025-01 ~ 2025-01
  RG2: 2025-02 ~ 2025-02
  ... (查 2025-05 → 跳 5 个 RG)

未排序的 date 列:
  RG1: 2025-01 ~ 2025-05  (min/max 范围大)
  RG2: 2025-02 ~ 2025-05
  ... (查 2025-05 → 全部命中,等于没下推)

所以工程上写 Parquet 前要按高频过滤字段排序,常见做法:

  • Spark: df.sortWithinPartitions("date")
  • Iceberg: WRITE ORDERED BY (date)
  • Delta: OPTIMIZE ... ZORDER BY (date, user_id)(多维排序)

六、向量化执行:CPU 视角 ​

6.1 行执行 vs 列执行 ​

c
// 行执行(MySQL 风格)
for each row in table:
    if row.date == '2025-05-01' && row.status == 'paid':
        sum += row.amount
// 每行一个 if 分支,CPU 分支预测失败、寄存器频繁切换

// 列执行(向量化)
date_col = read_column("date")        // 1024 个值一批
status_col = read_column("status")    // 1024 个值一批
amount_col = read_column("amount")    // 1024 个值一批

mask = (date_col == '2025-05-01') & (status_col == 'paid')
//      ↑ SIMD 指令一次处理 8/16 个 int 比较
sum = amount_col[mask].sum()
//                    ↑ SIMD 加法

SIMD(Single Instruction Multiple Data):Intel AVX2 一次处理 256 位(8 个 int32 或 4 个 int64),AVX-512 一次 512 位。循环开销摊薄、分支消失、缓存友好。

6.2 现代引擎都在向量化 ​

  • Apache Arrow:内存里的列式格式,跨语言、零拷贝
  • DuckDB / ClickHouse / Doris:从头到尾向量化
  • Photon(Databricks):C++ 重写 Spark 物理算子,向量化
  • Gluten / Velox:把 Spark 物理算子映射到 Velox 向量化引擎(开源 Photon 替代)
  • Polars / Daft:Python 端的向量化 DataFrame

Parquet + Arrow + 向量化引擎 是 2025 数据分析栈的标准三件套。

七、工程落地:写读 Parquet 的最小代码 ​

7.1 PyArrow ​

python
import pyarrow as pa
import pyarrow.parquet as pq

# 写
table = pa.table({
    "id": [1, 2, 3, 4],
    "user_id": [12345, 67890, 12345, 11111],
    "amount": [99.5, 199.0, 88.0, 50.0],
    "date": ["2025-05-01"] * 4,
})
pq.write_table(
    table, "orders.parquet",
    compression="zstd",
    row_group_size=128 * 1024 * 1024,
    bloom_filter_columns=["user_id"],
)

# 读(列裁剪 + 谓词下推)
table = pq.read_table(
    "orders.parquet",
    columns=["amount", "date"],         # 列裁剪
    filters=[("date", "=", "2025-05-01")]  # 谓词下推
)
print(table.to_pandas())

# 看 metadata
meta = pq.read_metadata("orders.parquet")
print(meta)
print(meta.row_group(0).column(2).statistics)  # amount 列的 min/max

八、ORC vs Parquet ​

ORCParquet
出身Hive(2013,Hortonworks)Twitter + Cloudera(2013)
主要生态Hive、Presto、TrinoSpark、Impala、Trino、Iceberg、Delta、Hudi
索引内置 Index + Bloom(强)Statistics + Page Index + Bloom(后来跟上)
ACID 支持Hive ACID 原生依赖外层(Iceberg/Delta)
嵌套结构支持支持(Dremel 编码,更原生)
列内压缩类似类似

功能上 ORC 在 Hive 强但生态弱;Parquet 在 Iceberg / Delta / Hudi 所有现代表格式里是默认。新项目无脑选 Parquet。

九、列存的代价 ​

9.1 单行查询慢 ​

WHERE id = 12345 在 OLTP 行存里是 B+ 树点查,毫秒级;在列存里要扫所有 Row Group(没索引)或扫 row group statistics(粗粒度)。

列存几乎不做点查,要点查就上 OLTP 或者搜索引擎(ES)或 KV 库(Redis)。

9.2 小批量更新难 ​

Parquet 是 immutable 文件,改一行 = 重写整个文件。

解决:

  • Iceberg / Delta / Hudi 的 ACID 层(08 / 09 篇)
  • 写时合并(CoW)或读时合并(MoR)
  • 增量 delete file(equality delete / position delete)

代价是引入元数据层 + 维护成本。

9.3 小文件问题 ​

Parquet 文件太小(< 几 MB)→ 每个文件 footer 开销 + 元数据开销 + 跨文件不能跨 Row Group 共享统计。

1000 个 1MB 的 parquet 文件   →  慢(footer overhead 占比高)
1   个 1GB 的 parquet 文件   →  快(一次 footer,统计有效)

工程上要定期 compaction(Iceberg / Delta 都有 OPTIMIZE 命令)。

9.4 Row Group 大小拐点 ​

< 8 MB        statistics 不准、Page Index 失效、整体慢
~ 128 MB      默认值,大多数场景甜点
~ 1 GB        单 task 读内存压力大,但全量扫吞吐高
> 1 GB        谨慎,内存炸风险

row_group_size 的选择跟下游引擎并行度、内存、过滤选择度都相关——没有银弹,生产上需要 profile。

十、现代列存的演进方向 ​

10.1 Apache Arrow ​

Arrow 是内存里的列式格式 + 零拷贝传输标准:

  • Parquet 是落盘格式,Arrow 是内存格式
  • 各引擎(DuckDB、Polars、Pandas、Spark)可以共享 Arrow 内存,避免序列化
  • Flight 协议直接用 gRPC 传 Arrow,跨进程零拷贝

关系:Parquet → Arrow → 计算引擎,中间不再多次拷贝。

10.2 新一代列存格式实验 ​

  • Lance(LanceDB):AI 时代列存,Parquet + 向量索引 + 零拷贝随机访问,比 Parquet 随机读快 100x
  • Vortex:Databricks/Spiral 提出,保持 Parquet 兼容 + 提升压缩 / 编码能力
  • Nimble(Meta):内部使用,优化点查 + 嵌套数据

Parquet 12 年没变,这些新格式都在尝试把它推到下一代。但Parquet 仍然是 2025 工程默认。

十一、看完这一篇,你应该能 ​

  • 在白板上对照画行存 / 列存的磁盘布局
  • 解释列存快 10 倍来自四个加成:列裁剪 / 高压缩 / 向量化 / 谓词下推
  • 默写 Parquet 的四层结构:File → Row Group → Column Chunk → Page
  • 选 dictionary / RLE / delta 三种编码的适用场景
  • 写 Parquet 时知道按高频过滤字段排序、选 Zstd、注意 row group 大小
  • 区分 ORC 和 Parquet 的生态归属,知道默认选 Parquet
