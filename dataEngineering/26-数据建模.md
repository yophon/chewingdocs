# 数据建模:维度建模、星型雪花、SCD、Data Vault

数据进了仓 / 湖,**表的"形状"决定了下游能跑多顺**:同样的订单数据,你建成宽表分析师秒查、建成 3NF 反复 join 报表慢得吓人;同样的用户表,你建成 SCD1 历史地址全丢、建成 SCD2 关联到下单当时的地址。**数据建模不是过时的话题**,即使在湖仓 + dbt 时代,Kimball 的维度建模仍然是 80% 数仓的基础。这一篇拆维度建模、星型 vs 雪花、SCD1/2/3、Data Vault、One Big Table 的取舍。

> 一句话先记住:**维度建模 = 事实表(度量)+ 维度表(描述)**——星型默认,雪花在大维度时用,SCD2 跟踪历史,Data Vault 适合多源 + 强治理大企业,One Big Table 适合「列存 + dbt + 存储不贵」的现代场景。**没有银弹,看你公司规模、查询模式、治理需求选**。

---

## 一、为什么仍然要建模

### 1.1 不建模的世界

```
原始 ODS 表:               下游分析:
orders                     "上周各品类各城市 GMV"
users                          ↓
products                   写 SQL 时 join 三张大表
regions                    每个分析师重复写
events                     口径不一致(漏掉退款单 vs 包含退款单)
inventories                结果 3 个团队 3 个数
...
```

### 1.2 建好模型之后

```
建好的 mart 层:
fct_orders_daily   按天 × 城市 × 品类的 GMV 已聚合
dim_users           用户维度宽表
dim_products        商品维度宽表
                       ↓
分析师查 fct_orders_daily 一句 SELECT 完事
口径统一(都在 fct 表的口径里定义清楚)
数仓团队改一次 = 全公司同步
```

**建模 = 把"复杂业务问题"提前固化成"简单查询"**。

## 二、维度建模(Kimball,1996)

### 2.1 两类表

```
事实表 (Fact)              维度表 (Dimension)
─────────────              ─────────────
业务事件 / 度量            描述属性
狭长(列少,行多)            宽矮(列多,行少)
几亿到几十亿行             几千到几百万行
typically 整数 / 数字       typically 字符串
                          
订单(amount, qty, ...)     用户(name, age, segment, ...)
点击(count, ...)           商品(category, brand, price, ...)
浏览时长(duration, ...)    地理(country, province, city, ...)
                          时间(date, weekday, is_holiday, ...)
```

### 2.2 必画图:星型模式

```
                    dim_date
                       │
                       │
   dim_users  ─── fct_orders ─── dim_products
                       │
                       │
                    dim_region
```

事实表在中间,维度表围绕,**事实表只跟维度表关联,维度表之间不直接关联** = 星型。

### 2.3 雪花模式

```
                    dim_date
                       │
                       │
   dim_users ─── fct_orders ─── dim_products
                       │              │
                       │              ▼
                    dim_region    dim_category
                       │              │
                       ▼              ▼
                  dim_country     dim_brand
```

维度被规范化(`dim_products → dim_category → dim_brand`)= 雪花。

### 2.4 星型 vs 雪花

| | 星型 | 雪花 |
| --- | --- | --- |
| Join 复杂度 | 少(事实 → 维度直接) | 多(链式) |
| 存储 | 略多(维度反范式重复) | 少(规范化) |
| 查询性能 | 快(join 少) | 慢(join 多) |
| 维护 | 维度改要全表更新 | 改子维度即可 |
| 适合 | OLAP 主流 | 强治理 / 极大维度 |

**2025 主流选择**:**星型默认**,雪花只在维度极大或强合规需求时。**列存时代,反范式不浪费存储,查询性能更重要**。

### 2.5 一致性维度(Conformed Dimension)

```
跨多个事实表共用的维度:

fct_orders ──┐
fct_returns ──┼──→ dim_users(同一张表,同一条记录)
fct_clicks  ──┘

→ 「订单的用户」「退款的用户」「点击的用户」对照同一条记录
  → 跨业务主题分析时口径一致
```

**没有一致性维度**:每个事实表自己 join 一份用户,字段不一致、分析无法跨表。

## 三、缓慢变化维(SCD)

### 3.1 问题

用户去年地址是北京,今年 5 月搬到上海。订单:
- 1 月下了一单(实际地址北京)
- 8 月下了一单(实际地址上海)

**dim_users 怎么存?**

### 3.2 三种 SCD

```
SCD1 (覆盖)
─────────────
user_id  name   province
1        Alice  Shanghai  ← 直接覆盖,北京历史丢

适合:不关心历史,只要当前
缺点:无法回溯"下单时的地址"


SCD2 (新行,带时间区间)
────────────────────────
user_id  name   province   valid_from   valid_to     is_current
1        Alice  Beijing    2024-01-01   2025-05-01   false
1        Alice  Shanghai   2025-05-01   NULL          true

适合:跟踪历史 + 关联到事件时刻
代价:维表变大(每次改加一行)
最常用


SCD3 (加列,前后两版)
─────────────────────
user_id  name   current_province  previous_province  changed_at
1        Alice  Shanghai          Beijing            2025-05-01

适合:只关心"上一个版本"
偶尔用
```

### 3.3 SCD2 的真实用法

```sql
-- 订单关联用户(取下单时刻的地址):
SELECT 
    o.id, o.event_time, o.amount,
    u.name, u.province                     -- 下单时的地址
FROM fct_orders o
LEFT JOIN dim_users u
    ON o.user_id = u.user_id
    AND o.event_time >= u.valid_from
    AND (o.event_time < u.valid_to OR u.valid_to IS NULL);

-- 不用 SCD2:
SELECT o.*, u.province
FROM fct_orders o
LEFT JOIN dim_users u USING (user_id);
-- 拿到的是"现在的地址",历史订单关联错(8 月之前订单全显示上海)
```

### 3.4 dbt 的 snapshot 自动维护 SCD2

```sql
{% snapshot dim_users_snapshot %}
{{ config(
    target_schema='snapshots',
    unique_key='user_id',
    strategy='timestamp',
    updated_at='updated_at',
) }}
SELECT * FROM {{ source('raw', 'users') }}
{% endsnapshot %}
```

`dbt snapshot` 自动加 `valid_from / valid_to / is_current` 列(参考 25 篇)。

## 四、Data Vault(2.0):大企业 + 强治理

### 4.1 出身

2000 年 Dan Linstedt 提出,2008 年 Linstedt + Inmon 推广 2.0。目标:**多源 + 强治理 + 强血缘审计**。

### 4.2 三种表

```
Hub        业务键 + 来源 + 加载时间
           (用户的统一主键,不带任何属性)
           
Link       Hub 之间的关系 + 来源 + 时间
           (订单连接用户和商品)
           
Satellite  描述属性 + 来源 + 时间(自带历史)
           (用户的 name / address / segment ...)
```

### 4.3 例子

```sql
-- Hub:用户身份
hub_user
  user_hk (hash key)
  user_id (business key)
  load_date
  source_system

-- Hub:产品
hub_product
  product_hk
  product_id
  load_date
  source

-- Link:订单(用户 + 产品)
link_order
  order_hk
  user_hk        -- FK
  product_hk     -- FK
  order_id (business key)
  load_date
  source

-- Satellite:用户属性(来自 CRM 系统)
sat_user_crm
  user_hk
  load_date
  name
  email
  segment
  
-- Satellite:用户属性(来自 ERP 系统)
sat_user_erp
  user_hk
  load_date
  customer_tier
  credit_limit
```

### 4.4 价值

- **多源融合**:同一个 Hub 可挂多个 Satellite(不同源)
- **强血缘审计**:每行带来源 + 时间
- **新增源不破坏老结构**:加新 Satellite 即可
- **法规友好**:GDPR / SOX 审计有数据

### 4.5 代价

- 表数量爆炸(简单业务也几十张表)
- 写 SQL 要 join 一堆,**Vault 不是直接给分析师查的**
- 需要在 Vault 上再建一层 Star Schema(Information Mart)

**Data Vault 适合大公司多源 + 强治理**,小公司过度设计。

## 五、One Big Table(OBT):反潮流

### 5.1 出发点

列存 + dbt + 存储便宜的时代,**何必拆事实 + 维度?直接宽表存所有列**:

```
fct_orders_obt (一张宽表):
order_id, event_time, amount, qty, ...,
user_id, user_name, user_province, user_segment, ...,
product_id, product_name, product_category, ...,
region_id, region_name, ...

3 张表 join → 1 张宽表,反范式
```

### 5.2 优势

- **查询简单**:`SELECT * FROM fct_orders_obt WHERE ...`,不用 join
- **列存压缩好**:字段重复多但列存压缩比高,存储成本可接受
- **缓存友好**:一次扫一张表
- **dbt 维护简单**:`{{ ref('dim_users') }}` join 一次写好

### 5.3 代价

- **更新维度成本爆**:用户地址改了 → fct_orders_obt 里**该用户的所有历史订单都要改**
- **存储确实多**:维度信息冗余存(几亿次)
- **失去维度修改的"一次性"**(维表改一行 vs 事实表改一亿行)

### 5.4 用 SCD2 的 OBT 折中

```sql
-- 在 dbt model 里 join SCD2 一次,落 OBT
SELECT 
    o.*,
    u.name, u.province AS user_province_at_order_time,
    p.name AS product_name, p.category
FROM {{ ref('fct_orders') }} o
LEFT JOIN {{ ref('dim_users_scd2') }} u
    ON o.user_id = u.user_id
    AND o.event_time BETWEEN u.valid_from AND COALESCE(u.valid_to, '9999-12-31')
LEFT JOIN {{ ref('dim_products') }} p USING (product_id)
```

**事实落 OBT,维度变化通过 SCD2 join 保留** —— 既有 OBT 的查询便利,又有 SCD2 的历史准确性。

### 5.5 谁在用 OBT

- **Stripe**:大部分分析表用 OBT
- **Airbnb**:报表层 OBT
- **Snowflake / BigQuery 用户**:列存 + 存储便宜,OBT 性价比高

## 六、数据建模的几种主流姿态

```
1. Kimball Star Schema           80% 公司默认
                                  事实 + 维度,星型为主
                                  
2. OBT(One Big Table)            列存 + dbt 时代越来越多
                                  分析师爽,工程师维护成本可控
                                  
3. Inmon 3NF Enterprise DW       传统大企业(银行 / 电信)
                                  规范化严格,但查询复杂
                                  
4. Data Vault                     多源 + 强治理大企业
                                  Vault + 表层 Star Schema 组合
                                  
5. Activity Schema                Stripe / Modern Data Stack 推
                                  「事件流 + 全实体」表
                                  适合产品分析

6. AOT(Anchor Modeling)           更极致的规范化
                                  极少用
```

**没有银弹**,根据公司规模、治理需求、查询模式选。

## 七、选型决策树

```
小公司、起步阶段:
  PostgreSQL + dbt
  Star Schema(简化版),OBT 也行
  
中型公司、多业务线:
  Snowflake / BigQuery / Iceberg + dbt
  Star Schema,部分 OBT
  SCD2 关键维度
  
大型公司、多源 + 强治理:
  Lakehouse + dbt + 自建 Data Vault 或简化版
  Data Vault 层 + 上面 Star Schema 给分析师
  
产品分析为主:
  Activity Schema(事件流)
  + 简化 Star
```

## 八、几个工程模式

### 8.1 退化维度

某些维度字段如果**只跟某个事实关联**(如订单号、发票号),可以直接放事实表,不建 dim 表:

```sql
fct_orders
  order_id     -- 退化维度,本身就是"维度",但没单独 dim 表
  order_number -- 同
  user_id      -- FK 到 dim_users
  amount
```

### 8.2 角色扮演维度

同一张维度被多次扮演:

```sql
fct_flights
  flight_id
  depart_airport_id  -- 都连到 dim_airports
  arrive_airport_id  -- 同
  depart_time
  arrive_time

-- 查询时用别名
SELECT f.*, da.name AS depart_airport, aa.name AS arrive_airport
FROM fct_flights f
JOIN dim_airports da ON f.depart_airport_id = da.id
JOIN dim_airports aa ON f.arrive_airport_id = aa.id;
```

### 8.3 桥接表(Bridge Table)

多对多关系:**一个订单多个商品**:

```sql
fct_orders
  order_id, user_id, total_amount

bridge_order_product
  order_id, product_id, quantity, price

dim_products
  product_id, ...
```

### 8.4 时间维度

预生成几十年的日期表:

```sql
dim_date
  date, year, month, day, weekday, week_of_year,
  is_weekend, is_holiday, quarter, fiscal_year, ...

SELECT d.year, d.month, SUM(o.amount)
FROM fct_orders o
JOIN dim_date d ON DATE(o.event_time) = d.date
WHERE d.is_holiday = true
GROUP BY 1, 2;
```

**好处**:可以查「**周末** vs 工作日」「**节假日** vs 平日」,这些信息不存在订单表里。

```
事实表        fct_  或  fact_

桥接表        bridge_  或  link_

```

## 十、几个反模式

### 10.1 全部 ODS

```
分析师直接查 ODS / source 表
→ 字段一变下游全炸
→ 每个分析师重复 join
→ 口径漂移
```

**正解**:至少有 stg + mart 两层,分析师不查 ODS。

### 10.2 没维度建模就上 BI

直接把业务库导一份给 Tableau / Looker → BI 模型在工具里建 → **维度散在 BI 工具,数仓没建模,跨 BI 工具不一致**。

**正解**:数仓建好模型(dbt + Star Schema),BI 工具只连建好的 mart。

### 10.3 用宽表代替建模

把所有数据塞一张宽表(几百列)→ 表巨大,维度变化无法跟踪,口径无法变。

**正解**:OBT 也要有事实表心智,维度变化通过 SCD2 处理。

### 10.4 维度表无 SCD

用户地址变了 → 历史订单的地址被改掉 → 财务报表口径漂移。

**正解**:对**核心维度**(用户、商品、组织架构)上 SCD2。

### 10.5 用 ID 代替业务键

```sql
-- ❌ user_pk (自增 ID) → join 用 PK
-- 但业务库重新导入 → user_pk 全变 → 所有事实失效

-- ✓ 用 business key(user_id from business system),保证跨重建稳定
```
