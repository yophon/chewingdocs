# 工作流编排:Airflow 心智

数据管道写完了,**怎么让它每天凌晨跑一遍?跑挂了怎么自动重试?跑慢了怎么报警?今天的任务依赖昨天的结果怎么排队?**——这些问题不是「写一段代码」能解决的,是工作流编排(orchestration)要回答的。**Airflow(2014 Airbnb 开源)是这个领域的事实标准,十年没死,但缺陷也越来越明显**。这一篇拉清楚 Airflow 的心智:DAG、Operator、Sensor、Scheduler、Executor,以及为什么下一代(24 篇 Dagster / Prefect)在抢它的位置。

> 一句话先记住:**Airflow = DAG-as-Python-code + 周期调度 + 依赖管理 + 重试 + 监控**。它的核心问题不是技术,是**「DAG-centric 而非 data-centric」**:Airflow 关心「task 跑了没」,但数据团队关心「数据对不对」。这就是 Dagster / Prefect 在抢的市场。

## 一、没编排的世界什么样

### 1.1 cron + shell 时代

```bash
# /etc/crontab
0 2 * * * /opt/etl/run_orders.sh
0 3 * * * /opt/etl/run_users.sh   # 必须等 orders 跑完
0 4 * * * /opt/etl/run_summary.sh # 必须等前两个
0 5 * * * /opt/etl/run_report.sh
```

问题:
- **依赖隐式**:run_users 真的等 run_orders 跑完了吗?写注释 + 祈祷
- **失败处理无**:run_orders 挂了,后面还会跑(产出脏数据)
- **重试无**:挂就挂了,人肉看
- **监控无**:每天有没有跑成功?谁知道
- **回算难**:今天发现昨天数据错了,要补跑哪些步骤?手动算
- **环境管理乱**:每个 shell 脚本自己 source 环境变量

### 1.2 工作流编排要解决的事

```
1. 显式依赖图(DAG):谁等谁,可视化
2. 定时调度:cron 表达式 / 间隔
3. 任务失败重试:配重试次数、间隔
4. 任务超时 / 报警
5. 依赖触发:上游成功才启动下游
6. 回算 / 补数据(Backfill)
7. 环境管理:Docker / 虚拟环境
8. UI 看进度、日志、指标
9. 资源池:控制并发(别把 Spark 集群打爆)
```

## 二、Airflow:开源工作流编排的事实标准

### 2.1 出身

```
2014  Airbnb 内部出 Airflow,Maxime Beauchemin 主导
2015  开源
2016  捐 Apache,变成顶级项目
2020  Airflow 2.0 大改:Scheduler 性能 N 倍、Task SDK 现代化
2022  TaskFlow API 让代码更 Pythonic
2024  Airflow 3.0 计划:Decoupled Scheduler / DAG Versioning
```

### 2.2 核心抽象

```
DAG          一个工作流的定义,有向无环图
Task         DAG 里的一个节点,一个工作单元
Operator     Task 的"模板"(BashOperator, PythonOperator, ...)
Sensor       特殊 Operator,阻塞等待某个条件(等文件、等 partition、等时间)
DagRun       一次 DAG 的执行实例
TaskInstance 一次 Task 的执行实例
Scheduler    后台进程,扫描 DAG,触发就绪 Task
Executor     执行 Task 的方式(Local/Celery/Kubernetes)
Worker       实际跑 Task 的进程
```

### 2.3 必画图:Airflow 部署架构

```
┌──────────────────────────────────────┐
│  Web UI (用户看 / 触发 / 重跑)        │
└──────────────────────────────────────┘
            │
            ▼
┌──────────────────────────────────────┐
│  Webserver  (Flask/FastAPI 进程)      │
└──────────────────────────────────────┘
            │
            ├─── Metadata DB (PostgreSQL) ←─┐
            │                                │
            ▼                                │
┌──────────────────────────────────────┐    │
│  Scheduler  (周期扫描 DAG)            │────┘
│   - 解析 DAG (.py 文件)              │
│   - 检查依赖、调度时间               │
│   - 推 ready 的 Task 到 Executor    │
└──────────────────────────────────────┘
            │
            ▼
┌──────────────────────────────────────┐
│  Executor  (LocalExecutor /          │
│             CeleryExecutor /         │
│             KubernetesExecutor)      │
└──────────────────────────────────────┘
            │
            ├──→ Worker 1 (跑 task A)
            ├──→ Worker 2 (跑 task B)
            └──→ Worker N
```

### 2.4 三种 Executor

```
LocalExecutor       Scheduler 同机起子进程跑 task
                    适合:小规模、单机
                    
CeleryExecutor      用 Celery + Redis/RabbitMQ 分发任务到多 worker
                    适合:中大规模、Worker 扩展
                    部署:Worker 节点常驻
                    
KubernetesExecutor  每个 task 启动一个 K8s Pod 执行
                    适合:云原生、按需弹性
                    部署:无常驻 Worker(每个 task 一个 Pod)
                    
KubernetesExecutor 是 2020+ 主流
```

## 三、写一个最小 Airflow DAG

### 3.1 基础版

```python
# dags/daily_orders_etl.py
from datetime import datetime, timedelta
from airflow import DAG
from airflow.operators.bash import BashOperator
from airflow.operators.python import PythonOperator
from airflow.sensors.external_task import ExternalTaskSensor

default_args = {
    'owner': 'data-team',
    'depends_on_past': False,
    'email_on_failure': True,
    'retries': 2,
    'retry_delay': timedelta(minutes=5),
}

with DAG(
    'daily_orders_etl',
    default_args=default_args,
    description='每天凌晨跑订单 ETL',
    schedule='@daily',  # = 0 0 * * *
    start_date=datetime(2025, 1, 1),
    catchup=False,
    tags=['orders', 'etl'],
) as dag:

    wait_partition = ExternalTaskSensor(
        task_id='wait_for_orders_partition',
        external_dag_id='upstream_orders_ingestion',
        external_task_id='write_partition',
        poke_interval=60,  # 每 60s 检查
        timeout=3600,
    )

    transform = BashOperator(
        task_id='dbt_run_orders',
        bash_command='cd /opt/dbt && dbt run --select tag:orders --target prod',
    )

    test = BashOperator(
        task_id='dbt_test_orders',
        bash_command='cd /opt/dbt && dbt test --select tag:orders --target prod',
    )

    def notify_slack(**context):
        # 发 slack 消息
        pass
    notify = PythonOperator(
        task_id='notify_completion',
        python_callable=notify_slack,
    )

    wait_partition >> transform >> test >> notify
```

### 3.2 TaskFlow API(更 Pythonic,Airflow 2.0+)

```python
from airflow.decorators import dag, task
from datetime import datetime

@dag(schedule='@daily', start_date=datetime(2025, 1, 1), catchup=False)
def daily_orders_etl():
    
    @task
    def extract():
        return read_from_kafka()
    
    @task
    def transform(raw):
        return clean(raw)
    
    @task
    def load(clean):
        write_to_iceberg(clean)
        return "done"
    
    raw = extract()
    cleaned = transform(raw)
    load(cleaned)

dag_instance = daily_orders_etl()
```

**TaskFlow API**:
- Task 之间数据通过 XCom 传递(自动序列化)
- 看起来像普通 Python 函数,易写易读
- 适合纯 Python 数据处理

## 四、Airflow 的核心难点

### 4.1 Idempotency(幂等性)

**Airflow 假设你的 task 是幂等的**:同一个 logical_date / execution_date 跑多次结果一样。

```python
# ❌ 不幂等
@task
def daily_summary():
    today = datetime.now().date()  # 重跑时是真今天,不是逻辑日期
    spark_sql(f"INSERT INTO summary SELECT ... WHERE date='{today}'")

# ✓ 幂等
@task
def daily_summary(execution_date=None):
    target_date = execution_date.date()
    spark_sql(f"DELETE FROM summary WHERE date='{target_date}'")  # 先删
    spark_sql(f"INSERT INTO summary SELECT ... WHERE date='{target_date}'")
```

**关键**:用 `{{ ds }}` / `{{ data_interval_start }}` 等模板变量,**不要用 `now()`**——回算时会跑成今天数据。

### 4.2 Backfill(回算)

```bash
# 重跑 2025-04-01 到 2025-04-30 这一个月
airflow dags backfill -s 2025-04-01 -e 2025-04-30 daily_orders_etl
```

Airflow 按 logical_date 一次跑全部,**前提是 task 幂等**(否则回算 = 灾难)。

### 4.3 logical_date 的迷惑

```
执行时间 vs 逻辑日期(关键!容易搞混):

DAG 配置 schedule='@daily',start_date=2025-01-01

logical_date     execution_date     真实跑的时间
2025-01-01       2025-01-01         2025-01-02 00:00  ← 跑数据是 1 月 1 日的
2025-01-02       2025-01-02         2025-01-03 00:00
...

logical_date 是数据的"逻辑日期",真实跑通常是它之后
```

**所以**:
- Task 里用 `{{ ds }}` 是 logical_date(2025-01-01)
- 不是真实跑的时间(2025-01-02)
- 这是为了符合「跑 1 月 1 日的数据,实际是 1 月 1 日数据收齐了之后再跑」

### 4.4 Sensor 的成本

```python
# ❌ 每个 Sensor 占一个 Task slot,十几个 Sensor 等数据 → Worker 全堵
ExternalTaskSensor(...)
S3KeySensor(...)
HivePartitionSensor(...)

# ✓ 用 deferrable/异步 Sensor(Airflow 2.2+)
S3KeySensor(deferrable=True, ...)
# Sensor 在等待时不占 slot,Triggerer 异步轮询
```

### 4.5 Worker 资源管理

```
不同 task 资源需求不同:
  dbt run    需要 1 CPU, 2GB
  Spark submit 需要触发 Spark 集群,本身轻
  Pandas ETL 需要 8 CPU, 32GB

Airflow Pool:控制特定资源池的并发
Pool('spark_pool', slots=10)
@task(pool='spark_pool', priority_weight=1)
def spark_etl(): ...
```

## 五、Airflow 的常见反模式

### 5.1 在 DAG 里直接跑大数据

```python
# ❌ Airflow Worker 上跑 Spark
@task
def heavy_spark_job():
    spark = SparkSession.builder.getOrCreate()
    spark.read.parquet(...).agg(...).write.parquet(...)

# Airflow Worker 不是计算节点!应该:
# ✓ 提交到外部 Spark 集群
@task
def submit_spark():
    subprocess.run(["spark-submit", "--master", "k8s://...", "/jobs/etl.py"])
# 或 KubernetesPodOperator,起独立 pod 跑
```

### 5.2 Task 之间传大数据

```python
# ❌ XCom 传 100MB 数据
@task
def t1():
    return huge_dataframe   # XCom 存 metadata DB,塞不下大数据

# ✓ 传引用
@task
def t1():
    df.write.parquet("s3://...")
    return "s3://..."
@task
def t2(path):
    df = spark.read.parquet(path)
    ...
```

### 5.3 用 PythonOperator 做事务

```python
# ❌ Python operator 调外部 API,失败了部分数据已写
@task
def call_apis():
    for record in records:
        api.send(record)  # 跑到一半挂了,前面已经发出去

# ✓ 上下游事务感知,失败可恢复
@task
def call_apis():
    for record in records:
        if already_sent(record): continue
        api.send(record)
        mark_sent(record)
```

### 5.4 把 Airflow 当业务调度

```
❌ Airflow 跑订单超时关闭、用户营销发券等业务逻辑
   Airflow Scheduler 不是为毫秒级触发设计
   
✓ 业务调度用业务系统(Kafka + Worker、专门的调度服务)
   Airflow 只跑数据 ETL / 周期分析
```

## 六、Airflow 的死结(为什么有下一代)

### 6.1 DAG-centric 而非 data-centric

```
Airflow 关心:Task A 跑了 → Task B 可以跑了
              (流程)
              
数据团队关心:订单表更新到了 2025-05-01 → 推荐特征可以重算
              (数据资产的状态)
              
Airflow UI 显示的是 task 状态,不是数据资产状态
```

### 6.2 数据血缘要靠插件

OpenLineage / Marquez 等插件可以做血缘,但不是 Airflow 一等公民——**改字段时,Airflow 看不到影响哪些下游 task**。

### 6.3 测试困难

```python
# Airflow Operator 跟 Airflow 实例耦合
@task
def my_task():
    df = pd.read_csv(...)  # 怎么单元测试这个 task?
    ...

# 24 篇 Dagster 通过 I/O Manager 解耦,
# my_task 接收输入(测试时给假数据)、返回输出(测试时检查)
```

### 6.4 配置和代码混合

```python
# DAG 文件混杂:配置(schedule, retries)+ 业务逻辑
# 难以做参数化、模板化
```

### 6.5 启动延迟

```
DAG 解析(Scheduler 跑 .py 文件)
  → 一次几秒
  → 几千个 DAG 时,Scheduler 压力大

Task 触发延迟(几秒到几十秒)
  → 不适合实时触发
```

### 6.6 Schema 演进

```
DAG 改了 → 历史 DagRun 还是老 DAG 结构
要看版本对比?要回滚?
Airflow 3.0 计划做 DAG Versioning(2025+)
```

## 七、Airflow 在 2025 仍是主流的原因

### 7.1 生态护城河

- 1000+ Provider(Operator)覆盖几乎所有数据源 / 服务
- Cloud Composer (GCP) / MWAA (AWS) / Astronomer (商业) 提供托管
- 大量公司有 5 年+ Airflow 投资,不是说迁就迁

### 7.2 简单 + 灵活

- DAG 是 Python 代码,什么都能写
- 不强迫你"资产化"或者"声明式",自由度高
- 团队入门门槛低(SQL + Python 即可)

### 7.3 监控 / 报警 / 重试 / 调度核心能力够用

虽然 Dagster / Prefect 更现代,但 Airflow 的核心能力解决了 80% 的痛点。**完美让位给"够用"**。

### 7.4 大公司格局

```
新建数据团队 (2024+):       倾向 Dagster
小团队 / 起步:               Airflow 仍多
已有 Airflow,5 年投资:      继续 Airflow,部分迁 Dagster
极简场景:                    Prefect / GitHub Actions / cron
```

## 八、Airflow 工程实践

### 8.1 DAG 组织

```
dags/
├── domain/
│   ├── orders/
│   │   ├── daily_orders_etl.py
│   │   ├── orders_quality_check.py
│   │   └── orders_backfill.py
│   ├── users/
│   └── inventory/
├── shared/
│   ├── operators/
│   │   └── custom_operator.py
│   └── utils/
└── dag_factory/
    └── generate.py    # 模板生成 DAG
```

按业务域组织,而不是按工具(避免 50 个 SparkOperator DAG 一锅炖)。

### 8.2 配置管理

```python
# Variables(Airflow 内置)
from airflow.models import Variable
spark_master = Variable.get("spark_master_url")

# Connections
from airflow.hooks.base import BaseHook
mysql_conn = BaseHook.get_connection("mysql_shop")

# Secret backend(推荐):
# Vault / AWS Secrets Manager / GCP Secret Manager
```

### 8.3 资源池

```python
# UI 配置:Pool 名称 + 槽位数
# DAG 引用
spark_etl = SparkSubmitOperator(
    task_id='etl',
    pool='spark_cluster_pool',
    pool_slots=2,  # 一个 task 占 2 个 slot
    ...
)
```

### 8.4 监控

```
关键指标:
- DAG / Task 失败率
- DAG 平均运行时长(SLA)
- Scheduler 心跳
- Worker 利用率
- Queue 积压
```

Datadog / Prometheus / OpenTelemetry 都有 Airflow exporter。

### 8.5 SLA / 报警

```python

```

## 九、典型 ETL DAG 模式

```python
@dag(schedule='@daily')
def daily_orders_pipeline():
    # 1. 等上游分区
    sense = ExternalTaskSensor(task_id='wait', ...)
    
    # 2. ELT 三步
    @task
    def extract():
        # CDC 已经在跑,这里检查上游 watermark
        return verify_data_freshness()
    
    dbt_run = BashOperator(
        task_id='dbt_run',
        bash_command='dbt run --select tag:daily',
    )
    
    dbt_test = BashOperator(
        task_id='dbt_test',
        bash_command='dbt test --select tag:daily',
    )
    
    # 3. 数据质量额外检查
    @task
    def quality_gate():
        return run_great_expectations()
    
    # 4. 通知
    @task
    def notify():
        send_slack("ETL done")
    
    sense >> extract() >> dbt_run >> dbt_test >> quality_gate() >> notify()
```
