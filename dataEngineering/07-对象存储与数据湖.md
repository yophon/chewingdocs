# 对象存储与数据湖:S3 凭什么干掉了 HDFS

2010 年大数据集群房间长这样:**几十个 4U 服务器、每台插 12 块 4TB 机械盘、组成 HDFS,NameNode 一台主一台备**。十五年后,这间机房空了——数据全在 S3 上,公司只为「用到的那一刻」付钱。这不是一次技术升级,是**数据底层的经济学反转**:存储和计算解绑,引擎随便换,存储几乎免费。但故事的另一面:**一堆 Parquet 扔进 S3 桶里不叫数据库,叫数据沼泽**——没 ACID、没 Schema 强约束、没原子提交、目录都是模拟的。这一篇讲清楚对象存储凭什么赢,以及为什么「文件 → 表」这一步需要 Iceberg / Delta / Hudi 的元数据层。

> 一句话先记住:**对象存储是「无限大、按用量付费、跨 AZ 11 个 9 持久」的 key-value 存储**——不是文件系统。它没有目录、没有 rename 原子性、list 操作贵且慢、最终一致(2020 后 S3 强一致)。**「Hive 风格的分区路径」是约定,不是协议**。把这套基础设施变成「能扛 ACID 分析的表」,需要 08 篇会讲的开放表格式。

## 一、HDFS 怎么死的

### 1.1 HDFS 设计假设(2006)

Hadoop Distributed File System 是为「自建机房 + 机械硬盘 + 万兆网」设计的:
- 数据节点(DataNode):一堆插满硬盘的机器,数据多副本(默认 3)
- NameNode:中心节点,记元数据(文件 → block → DN)
- 块(Block):默认 128MB,顺序读优化
- 计算就近(Data Locality):MapReduce / Spark 把任务调到数据所在的 DN

### 1.2 死法 1:NameNode 单点

NameNode 内存里维护所有文件 metadata:
- 1 亿个文件 ≈ 60GB 内存
- 重启时要从 fsimage + editlog 恢复,慢则小时级
- HA 方案(QJM / ZK)复杂,故障切换有数据不一致风险
- **小文件灾难**:10 亿个 1KB 小文件 = NameNode 直接 OOM

### 1.3 死法 2:存算耦合扩容贵

```
要扩存储 → 加机器 → 机器还配了 CPU/内存(浪费)
要扩计算 → 加机器 → 机器还配了硬盘(浪费)
存储和计算的需求增长速率不同,但 HDFS 强绑定
```

公有云出现后,这套硬件成本对比就崩了:**S3 每月 0.023 $/GB,自建 HDFS 算上机房、电力、运维 > 0.05 $/GB,还要承担故障**。

### 1.4 死法 3:运维复杂

- Balancer 平衡 DN 数据,占用网络
- 升级要滚动重启,影响业务
- Kerberos 安全配置门槛高
- 跨机房灾备难

### 1.5 退场进度

- 2015 前:HDFS 是大数据标配
- 2018 后:云上新项目几乎不再用 HDFS,直接 S3
- 2020 后:大公司开始把内部 HDFS 迁到对象存储(JuiceFS / OSS / 自建 Ceph 等)
- 2025:**新建数据平台用 HDFS 是反模式**,本地化场景用 MinIO / Ceph 替代

HDFS 没有"死亡"事件,它就是被慢慢替换出去的。

## 二、对象存储的核心特性

### 2.1 心智模型:超大的 KV 存储

```
对象存储 ≠ 文件系统
对象存储 = 超大、远程、最终(后来强)一致的 KV

  Key:   "bucket/path/to/file.parquet"  (一个字符串)
  Value: 二进制数据  (从几 KB 到几 TB)
  Metadata: content-type, etag, size, custom tags

不存在「目录」概念,只有「key 前缀」。
"bucket/year=2025/month=05/" 是一个前缀,不是目录。
```

### 2.2 关键特性

| 特性 | 解释 |
| --- | --- |
| **持久性** | S3 标准类 11 个 9(99.999999999%),数据自动跨 3+ AZ 副本 |
| **可用性** | S3 标准类 99.99%(每年宕机约 50 分钟) |
| **吞吐** | 单 prefix 几千 op/s,跨 prefix 可线性扩 |
| **延迟** | GET 首字节 30-100ms(冷),几 ms(预热 / CDN) |
| **一致性** | 2020 年后 S3 strong read-after-write(写完立刻能读到) |
| **写语义** | 单次 PUT 上传到 5GB,多段上传 multipart 到 5TB |
| **计费** | 存储(GB-月)+ API 请求(PUT/GET/LIST)+ 流量(出网)|

### 2.3 三家主流

| | AWS S3 | 阿里 OSS | Google GCS |
| --- | --- | --- | --- |
| API | S3(事实标准) | S3 兼容 | S3 兼容 + 原生 |
| 一致性 | Strong | Strong | Strong |
| 多区/单区 | Multi-AZ 默认 | 多种 | Multi/Single Region |
| 冷归档 | Glacier / Deep Archive | 归档 / 冷归档 | Coldline / Archive |
| 价格(标准类) | $0.023/GB-月 | ¥0.12/GB-月 | $0.020/GB-月 |

**S3 API 已经成为对象存储的事实标准**,自建系统(MinIO / Ceph / JuiceFS)都兼容 S3 API。

### 2.4 自建场景:MinIO / Ceph / JuiceFS

不上公有云,自己搭对象存储:
- **MinIO**:Go 写的,API 兼容 S3,轻量,小到中型公司常用
- **Ceph + RGW**:全功能,大规模可扩到 EB
- **JuiceFS**:不是对象存储本身,是「在对象存储上挂 POSIX 文件系统」,过渡场景用
- **HDFS**:仍可选,但只推荐已经在用且无云预算的场景

## 三、对象存储不是文件系统:踩坑大全

### 3.1 没有目录,只有前缀

```bash
aws s3 ls s3://bucket/path/

# 实际不是「列目录」,而是「按 prefix 'path/' 列 keys,分隔符 '/'」
# 操作复杂度 O(N),N 是该前缀下的总 key 数
# 千万级 key 的 list 操作几秒到几十秒
```

**坑**:Hive 启动一个 query 前要 list 所有分区目录,千万分区 = 启动几分钟。Iceberg 的 metadata 就是为了避免这种 list 而设计的(参考 08)。

### 3.2 没有 rename,只有 copy + delete

```bash
# 你以为 mv 是原子的(POSIX 文件系统是)
aws s3 mv s3://bucket/old/file s3://bucket/new/file

# 实际是:
#   1. COPY s3://bucket/old/file → s3://bucket/new/file
#   2. DELETE s3://bucket/old/file
# 中间任何时刻都可能两个 key 都存在或都不存在
```

**坑**:Hive 写表的临时目录 → 最终目录,依赖 rename 原子性,在 S3 上有竞态条件。Iceberg 改用「写新 metadata.json + 原子 swap pointer」,避开 rename。

### 3.3 列存友好但小文件爆炸

```
理想:                          反模式:
  1000 个 128MB Parquet         1000000 个 1KB 小文件
  list 1000 个 keys 快           list 100 万 keys 几分钟
  Footer 元数据开销小            Footer 元数据 > 数据
  跨 Row Group 统计有效          统计基本失效
```

对象存储**惩罚高频小 IO**——一次 GET 几十 ms latency,小文件场景大部分时间在 latency 上等。

### 3.4 一致性历史

- 2006-2020:S3 read-after-write 只在「create 新 key」时强一致,update / delete 是最终一致
- 2020 后:全部 strong consistency(写完立刻能读到)

但跨 region 仍然异步复制(Replication),跨 region 不保证强一致。

### 3.5 计费陷阱

```
存储费    便宜      0.023 $/GB-月,1TB ≈ 23 美元/月
PUT/POST  贵        每 1000 次 0.005 美元 → 1 亿次 = 500 美元
GET       便宜      每 1000 次 0.0004 美元
LIST      贵        每 1000 次 0.005 美元(跟 PUT 同价!)
出网流量  非常贵    0.09 美元/GB,1TB 出网 = 92 美元
跨 region 流量 非常贵
Glacier 取回 贵且慢
```

**实战中最容易爆账单的两件事**:
- **跨 region 流量**:不小心把 EMR 起在 us-west,数据在 us-east-1 → 每次读全跨 region
- **小文件 + 频繁 LIST**:小公司刚上数据湖,一周 list 几亿次直接 1000 美元起

工程对策:
- 数据和计算同 region
- 小文件 compaction
- 生命周期策略:冷数据迁 IA / Glacier
- 用 Iceberg / Delta 减少 list

## 四、Hive 风格分区:约定但不是协议

### 4.1 长这样

```
s3://bucket/orders/
    year=2025/
        month=01/
            day=01/
                file-001.parquet
                file-002.parquet
            day=02/
                ...
        month=02/
            ...
```

「分区路径里带 `key=value`」这套是 Hive 引入的约定。引擎通过解析路径推导出每个文件属于哪个分区,执行 `WHERE year=2025 AND month=05` 时只 list 那部分前缀。

### 4.2 它解决的问题

- 不需要单独的元数据存分区
- 引擎能下推过滤(分区裁剪)
- 人能直接 ls 看到数据组织

### 4.3 它没解决的问题

- **list 仍然慢**:几万个分区下 list,几分钟起
- **分区演进不灵活**:换分区粒度(天 → 小时)要重写所有数据
- **元数据散在路径里**:跨文件统计、版本管理、ACID 都做不到
- **小文件累积**:每个分区一堆小文件没人管

**Iceberg / Delta / Hudi 的全部价值,就是在 Hive 风格分区之上加一个元数据层**,把这些问题一次性解决。08 篇会展开。

## 五、文件 vs 表:数据湖与 Lakehouse 的关键区别

### 5.1 数据湖(Data Lake)

```
s3://bucket/orders/year=2025/month=05/day=01/*.parquet
s3://bucket/orders/year=2025/month=05/day=02/*.parquet
...

→ 一堆文件,自行约定路径
→ 谁来读谁来解析,无 ACID
→ 改一行 = 重写文件
→ 多人同时写,后写覆盖前写,无原子性
```

**这就是「数据湖」状态**——便宜、灵活、但没有任何"表"该有的保证。

### 5.2 加一层元数据 → 表

```
s3://bucket/orders/
    data/
        year=2025/month=05/day=01/file-001.parquet
        year=2025/month=05/day=01/file-002.parquet
    metadata/
        v1.metadata.json    ← 表的当前状态
        v2.metadata.json
        snap-1.avro         ← snapshot 描述
        snap-2.avro
        manifest-001.avro   ← manifest 文件
```

**关键差别**:metadata 文件记录「现在的表是哪些 data 文件组成的」+ 「每个文件覆盖什么数据」+ 「每次写产生新 snapshot」。

引擎读表时不再 list 所有 data 文件,而是:
1. 读 metadata.json 拿到当前 snapshot
2. 读 snapshot 拿到 manifest list
3. 读 manifest 拿到 data files + 每个文件的 partition 信息和统计
4. 按 query 谓词裁剪 → 只读需要的 data 文件

这就是 **Iceberg / Delta / Hudi 的核心思想**。

### 5.3 「文件 → 表」带来的能力

- **ACID**:metadata 原子切换,多写者有冲突解决
- **Time Travel**:`SELECT * FROM t VERSION AS OF 5` 看历史快照
- **Schema Evolution**:加列改名零数据重写
- **Partition Evolution**:换分区粒度,旧数据保留
- **Hidden Partitioning**:写 `WHERE ts > x`,引擎自动用 partition

代价是 metadata 维护(快照过期、manifest 压缩)。

## 六、工程落地:Spark / DuckDB / Trino 读 S3

### 6.1 Spark 读 S3 Parquet

```python
# Spark 配置
spark = SparkSession.builder \
    .config("spark.hadoop.fs.s3a.access.key", AWS_KEY) \
    .config("spark.hadoop.fs.s3a.secret.key", AWS_SECRET) \
    .config("spark.hadoop.fs.s3a.endpoint", "s3.amazonaws.com") \
    .config("spark.hadoop.fs.s3a.impl", "org.apache.hadoop.fs.s3a.S3AFileSystem") \
    .getOrCreate()

df = spark.read.parquet("s3a://bucket/orders/year=2025/month=05/")
df.filter("status = 'paid'") \
  .groupBy("province") \
  .agg(F.sum("amount").alias("gmv")) \
  .show()
```

EMR / Databricks 上 IAM Role 自动认证,不需要硬编码 key。

### 6.2 DuckDB 直接查 S3

```sql
INSTALL httpfs;
LOAD httpfs;
SET s3_region='us-east-1';
SET s3_access_key_id='...';
SET s3_secret_access_key='...';

SELECT province, SUM(amount) AS gmv
FROM read_parquet('s3://bucket/orders/year=2025/month=05/**/*.parquet')
WHERE status = 'paid'
GROUP BY province;
```

DuckDB 嵌入式查 S3,**笔记本电脑可以**——下游不需要 Spark 集群。

### 6.3 Trino 配 S3

```ini
# catalog/hive.properties
connector.name=hive
hive.metastore=glue
hive.s3.region=us-east-1
hive.s3.path-style-access=false
hive.recursive-directories=true
hive.parquet.use-column-names=true
```

```sql
SELECT province, SUM(amount) FROM hive.orders.orders
WHERE year = 2025 AND month = 5
GROUP BY province;
```

### 6.4 CLI 操作

```bash
# 同步本地到 S3(开发常用)
aws s3 sync ./local-dir/ s3://bucket/path/

# 跨账号传输(IAM Role 跨账号信任)
aws s3 cp s3://src/file s3://dst/file --source-region us-east-1

# 生命周期策略(冷数据自动迁 Glacier)
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-bucket \
  --lifecycle-configuration file://lifecycle.json
```

```json
// lifecycle.json:30 天后转 Glacier
{
  "Rules": [{
    "Status": "Enabled",
    "Filter": {"Prefix": "archive/"},
    "Transitions": [{"Days": 30, "StorageClass": "GLACIER"}]
  }]
}
```

## 七、几个常见误区

### 7.1 「我用 S3 就是数据湖了」

S3 桶 + 一堆文件 = 数据湖**雏形**,但还没到「能用」的程度:
- 没 ACID → 多写者乱;一写一读时读到半截写入的视图
- 没 schema → 字段一变下游全炸
- 没分区元数据 → 大表 list 慢到死
- 没快照 → 误删数据没法恢复

工程上的「数据湖」必须加上 Iceberg / Delta / Hudi 之一,才叫数据湖,否则只是「数据沼泽」。

### 7.2 「对象存储一致性弱,不能用于关键场景」

2020 年后 S3 已经 strong read-after-write,事务性问题剩下的是:
- **跨 key 的原子性**:没有(对象存储不提供事务)
- **跨 region 复制**:异步,不强一致

但**Iceberg / Delta 在单个 metadata.json 切换这件事上做到原子**(用 Compare-And-Swap 接口或 conditional write),整张表的 ACID 在表格式层保证,**不依赖对象存储本身的事务**。

### 7.3 「HDFS 比 S3 快」

部分场景成立:
- 数据本地性(Data Locality):HDFS 任务调度可以读本地盘,延迟低
- 局域网带宽:HDFS 用满 10G/40G,S3 受出口限制

但:
- S3 跨 region 提供按需弹性,HDFS 不行
- S3 维护成本远低于 HDFS
- 现代查询引擎(Spark / Trino / Iceberg)的 IO 模式已优化为对象存储友好
- 大部分公司「快一点点」不如「便宜得多 + 弹性扩」

**2025 年只有特殊场景才选 HDFS**(已有大量数据资产、对延迟极敏感、不接受云出口流量费)。

### 7.4 「我直接读所有 .parquet 文件就行」

```python
df = spark.read.parquet("s3://bucket/orders/")
# 这个调用要 list 所有 .parquet,几万个分区下慢且贵
```

工程上要么按分区路径精确指定(`s3://bucket/orders/year=2025/month=05/`),要么用 Iceberg / Delta(metadata 直接给文件列表,不 list)。

## 八、对象存储 + 计算的现代分工

### 8.1 计算存储分离的力学

```
              [对象存储]  S3 / OSS / GCS
              + 表格式    Iceberg / Delta / Hudi
              + Catalog   Glue / Nessie / Unity
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
   Spark(批)   Flink(流)   Trino(交互)
   DuckDB(单机)Snowflake    Athena
   按需启停     无状态可扩    弹性弹起
```

**核心红利**:同一份数据,多个引擎按需读。批处理用 Spark 跑离线,交互 BI 用 Trino,本地探索用 DuckDB,机器学习用 PyTorch + Arrow,**都是同一份 Parquet/Iceberg 数据**。

### 8.2 Tiered Storage:热温冷三层

```
热数据   (S3 Standard / 内存缓存)        过去 7 天,查询高频
温数据   (S3 Standard-IA)                  过去 90 天,偶尔查
冷数据   (S3 Glacier / Deep Archive)       归档,合规保留 7 年
```

工程实践:
- Iceberg 表配 lifecycle 策略
- 旧 snapshot expire,旧 manifest rewrite
- Kafka Tiered Storage(KIP-405)把 Kafka 旧数据下沉到 S3
