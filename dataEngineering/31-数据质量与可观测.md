# 数据质量与可观测:dbt test、Great Expectations、Soda、Monte Carlo

「**昨天的报表怎么数字翻了一倍?**」周一早上,运营在群里炸了。data team 三个工程师查了一下午,发现上游 ETL 某个 filter 条件被人改了——但**没人监控这件事**。这种事故每个数据团队都经历过,**「数据没监控」就像应用没 APM**。这一篇拆数据质量(规则断言)和数据可观测(像应用监控一样监控数据)、五维监控、工具选型(dbt test / Great Expectations / Soda / Monte Carlo / DataHub)、数据 SLA。

> 一句话先记住:**数据质量 = 规则断言**(amount > 0 / unique pk),**数据可观测 = 监控五维**(新鲜度 / 容量 / Schema / 分布 / 血缘)。**dbt test 在管道内、Great Expectations 在管道外、Monte Carlo 自动学基线、DataHub 是元数据治理**——四件事互补,不互相替代。

---

## 一、为什么数据质量是个问题

### 1.1 数据 bug vs 应用 bug

```
应用 bug:        500 / 错误率 / OOM
                 立刻报警,几分钟定位
                 
数据 bug:        昨天数据少了 10%、字段语义变了、口径漂移
                 没人发现,直到 BI 报表数字异常
                 几天甚至几周后才发现,影响面已经大了
```

### 1.2 数据 bug 的几种典型

```
1. 字段缺失:    上游加了字段没通知下游
2. 字段语义变:   "status=2" 以前是"已支付",现在是"已发货"
3. 行数突变:    某天的订单数据掉了 50%(上游 ETL 挂)
4. 重复:        同一订单出现两次(上游去重失效)
5. 类型变:      amount 从 decimal(10,2) 变成 int
6. 分布漂移:    用户年龄从均值 30 飘到 45(上游过滤逻辑变)
7. 新鲜度:      表上次更新是 3 天前(应该每小时)
```

### 1.3 没监控的代价

```
- 业务发现 → 业务怀疑所有数据
- 修复时间长(几天到几周)
- 信任崩塌:"数仓的数据不能信"
- ML 模型悄悄变差(特征漂移,AUC 慢慢降)
```

## 二、三个层次:质量、可观测、SLA

### 2.1 数据质量(Data Quality)

```
"这条数据 / 这张表对不对"
通过规则断言验证

例:    
  pk unique
  amount > 0
  status IN ('paid', 'cancelled')
  user_id FK 到 users.id
```

### 2.2 数据可观测(Data Observability)

```
"这张表的健康状态怎么样"
像应用 APM 一样持续监控

例:
  行数比昨天少 30% → 警报
  字段空值比例突然涨 → 警报
  Schema 字段被删 → 警报
  分布偏离基线 → 警报
  上游表没按时更新 → 警报
```

### 2.3 数据 SLA / SLO

```
"这张表对外承诺什么"
- 新鲜度:每天 UTC+8 6:00 前完成
- 完整性:>= 99.9% 的订单出现
- 准确性:跟业务库对账误差 < 0.1%
- 可用性:99.9% 的查询能在 5 秒内返回

对外提供:数据合约(05 篇)
违约:通知下游 + 自动降级
```

## 三、数据质量工具:dbt test / GE / Soda

### 3.1 dbt test(管道内嵌,首选)

```yaml
# models/marts/schema.yml
models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: user_id
        tests:
          - not_null
          - relationships:
              to: ref('dim_users')
              field: user_id
      - name: amount
        tests:
          - not_null
          - dbt_utils.accepted_range:
              min_value: 0
              max_value: 100000
      - name: status
        tests:
          - accepted_values:
              values: ['created', 'paid', 'shipped', 'cancelled']
```

```sql
-- tests/assert_no_negative_revenue.sql
SELECT * FROM {{ ref('daily_revenue') }} WHERE revenue < 0
-- 返回任何行 = 失败
```

```bash
dbt test --select fct_orders
```

**优点**:
- 集成 dbt,跟 model 一起 review
- CI 自然跑
- 失败可阻断 pipeline

**缺点**:
- 只能在 dbt model 上跑
- 测试是规则式,不能学基线

**适合**:**dbt 项目内的数据质量,首选**。

### 3.2 Great Expectations(通用 Python)

```python
import great_expectations as gx

context = gx.get_context()

batch = context.get_batch({
    "datasource_name": "my_postgres",
    "table_name": "fct_orders"
})

# 期望套件
batch.expect_column_to_exist("order_id")
batch.expect_column_values_to_be_unique("order_id")
batch.expect_column_values_to_not_be_null("user_id")
batch.expect_column_values_to_be_between("amount", min_value=0, max_value=100000)
batch.expect_column_values_to_be_in_set("status", ['created', 'paid', 'shipped', 'cancelled'])

# 跑验证
result = batch.validate()
```

**优点**:
- Python 框架,灵活
- 内置 100+ expectations
- 数据 source 多(SQL / Spark / Pandas / S3)

**缺点**:
- 配置复杂(套件 / Datasource / Checkpoint)
- 学习曲线
- 跟 dbt 重叠

**适合**:**非 dbt 栈、跨数据源验证、Python 团队**。

### 3.3 Soda(YAML-first + SaaS)

```yaml
# checks.yml
checks for fct_orders:
  - row_count >= 100
  - missing_count(user_id) = 0
  - duplicate_count(order_id) = 0
  - invalid_count(status) = 0:
      valid values: [created, paid, shipped, cancelled]
  - schema:
      warn: when forbidden column present [internal_test_field]
  - freshness(updated_at) < 1d
```

```bash
soda scan -d warehouse -c soda.yml checks.yml
```

**优点**:
- YAML 简洁
- 监控 + 告警一体(Soda Cloud)
- 集成多 source

**适合**:**云原生 + YAML 风格 + 想要告警一体化**。

## 四、数据可观测平台:Monte Carlo / Bigeye

### 4.1 vs 数据质量工具

```
数据质量(GE / Soda):     你写规则,断言
                          → 你必须先知道有什么问题
                          
数据可观测(Monte Carlo): 自动学基线,异常检测
                          → 系统发现你没想到的问题
```

### 4.2 Monte Carlo

```
自动监控五维:
1. Freshness:    表上次更新 vs 平时多久更新一次
2. Volume:       行数 vs 7 天均值
3. Schema:       字段加 / 删 / 改类型
4. Distribution: NULL 比例、唯一值数、值分布
5. Lineage:      血缘 + 影响面
```

工作流:
- 连接你的仓 / 湖(只读)
- 自动扫所有表的统计
- 学每张表的"正常基线"(几天到几周)
- 异常时告警(Slack / PagerDuty)

**优点**:**零配置覆盖大量表**,异常检测自动。

**缺点**:贵(企业 SaaS,按表 / 用户计费),误报需要调阈值。

### 4.3 Bigeye / Lightup / Datafold

同代竞品,各有侧重(Bigeye 强机器学习驱动,Lightup 偏自定义,Datafold 强 diff)。

### 4.4 开源方向

- **OpenMetadata**:开源数据平台 + 部分可观测
- **Apache Griffin**(LinkedIn):老牌但维护偏冷
- **Elementary**(基于 dbt):dbt 的可观测增强

## 五、元数据 + 血缘:DataHub / Amundsen

### 5.1 这是另一个层次

```
数据质量:        断言一张表数据对不对
数据可观测:      监控一张表健康度
元数据治理:      "这张表是什么?谁拥有?上下游?业务文档?"
                关于"数据"的"数据"
```

### 5.2 DataHub(LinkedIn 开源)

```
功能:
  - 自动从 Airflow / dbt / Snowflake / Spark / Kafka 抽元数据
  - 表 / 字段级别血缘自动生成
  - 业务术语表(Business Glossary)
  - 数据合约 / Ownership
  - 搜索 + 文档
  - 跟数据质量工具集成(显示每张表的最新质量分)
```

### 5.3 Amundsen(Lyft 开源)

```
偏「数据资产搜索」
轻量,适合小到中团队
```

### 5.4 Apache Atlas / Collibra / Alation

```
Atlas:    Hadoop 系老牌,维护偏冷
Collibra: 企业级商业,治理 + 合规
Alation:  企业级商业,数据目录
```

### 5.5 跟编排器集成

```
Dagster:        资产化原生,自带血缘
Airflow:        通过 OpenLineage 插件上报到 DataHub / Marquez
dbt:             docs generate + DataHub Ingestion
Snowflake:      Information Schema 抽元数据
Spark:          Lineage 通过 Spline / OpenLineage 抽
```

## 六、五维监控

### 6.1 Freshness(新鲜度)

```
表上次更新:     2025-05-11 02:00
平时更新频率:    每天 02:00 ± 15 min

异常:
  上次更新是 3 天前 → 严重(管道挂了)
  上次更新是今天上午 11:00 → 警告(延迟 9 小时)
```

dbt 用 source freshness:
```yaml
sources:
  - name: raw_shop
    tables:
      - name: orders
        loaded_at_field: ingested_at
        freshness:
          warn_after: {count: 1, period: hour}
          error_after: {count: 3, period: hour}
```

### 6.2 Volume(容量)

```
当日行数 vs 7 天均值:
  10000 行 vs 平均 10000 ± 5% → 正常
  10000 行 vs 平均 50000 → 警报(掉 80%)
  
Monte Carlo 自动算
GE / Soda 写规则:row_count > threshold
```

### 6.3 Schema(结构)

```
字段加:        通常 OK(向后兼容,05 篇)
字段删:        危险(下游可能依赖)
字段类型变:    很危险
字段语义变:    最危险(规则检测不到)
```

### 6.4 Distribution(分布)

```
amount 列:
  历史均值:50,标准差:30
  今天均值:200,标准差:100
  → 警报(可能是退款单变多 / 单价大涨,业务侧应该感知)

NULL 比例:
  历史 0.1%
  今天 30%
  → 警报(上游字段断了)

cardinality:
  历史 user_id 5000 个唯一值
  今天 5 个唯一值
  → 警报(测试数据混入 / 上游过滤错)
```

### 6.5 Lineage(血缘)

```
不是监控指标,是"影响面分析":
  fct_orders 数据异常 → 影响下游哪些表 → 影响哪些 dashboard
  改了 dim_users.address → 哪些 model 重算
```

## 七、数据 SLA / SLO

### 7.1 定义

```
fct_orders 的 SLA:
  Freshness:        每天 UTC+8 06:00 前完成
  Completeness:     >= 99.9% 的订单出现
  Accuracy:         跟业务库对账误差 < 0.1%
  Schema 稳定性:    Breaking changes 提前 30 天通知

下游 BI 报表的 SLA:
  每天 08:00 前可用
  历史数据可查询 3 年
```

### 7.2 实现

```yaml
# 数据合约(05 篇)
contract:
  name: fct_orders
  owner:
    team: data-platform
    slack: "#data-platform"
    on_call: pagerduty:data-team
  
  schema:
    fields: [...]
  
  sla:
    freshness: 6h
    completeness: 99.9%
    
  policies:
    schema_evolution: BACKWARD
    breaking_changes: require_review_and_30day_notice
```

监控这个合约的工具:**Soda Contracts**、**Monte Carlo Contracts**、**dbt contract**(部分)。

## 八、工程落地:一套完整的数据质量栈

### 8.1 分层

```
管道内嵌:
  dbt test           每次 dbt run 跑,失败阻断 pipeline
  
管道外监控:
  Monte Carlo / Soda 自动监控五维,异常告警
  
元数据治理:
  DataHub            血缘 + 文档 + 合约 + 搜索
  
数据合约:
  YAML / dbt contract 上游签 SLA,下游受益
  
通知 / 告警:
  Slack / PagerDuty  失败时通知 owner
```

### 8.2 一个最小配置

```python
# Dagster Pipeline
@asset(deps=[raw_orders])
def fct_orders():
    return dbt.run("fct_orders")

@asset_check(asset=fct_orders, blocking=True)
def check_fct_orders_volume():
    """容量监控"""
    today_count = query("SELECT COUNT(*) FROM fct_orders WHERE event_date = CURRENT_DATE")
    avg = query("SELECT AVG(cnt) FROM (SELECT COUNT(*) cnt FROM fct_orders ... GROUP BY event_date)")
    
    return AssetCheckResult(
        passed=today_count > avg * 0.5,
        metadata={"today": today_count, "avg": avg}
    )

@asset_check(asset=fct_orders)
def check_fct_orders_unique():
    """唯一性"""
    dups = query("SELECT order_id, COUNT(*) FROM fct_orders GROUP BY 1 HAVING COUNT(*) > 1")
    return AssetCheckResult(passed=len(dups) == 0)
```

## 九、几个工程真实坑

### 9.1 测试过多 → 跑半小时 → 大家关掉

```
500 个测试,每个跑 5 秒 → 全跑完 40 分钟
团队选择 dbt test --select tag:critical
其他测试形同虚设
```

**正解**:**测试分级**(critical / warn / info),CI 跑 critical,定期跑全集。

### 9.2 误报疲劳

```
Monte Carlo 每天报警 50 次,80% 是误报
团队不看告警了,真异常被埋
```

**正解**:**调阈值 + 分级 + 静默**(known-issue 临时静默)。

### 9.3 失败处理:停 pipeline 还是 warn

```
dbt test 默认 severity=error → 失败阻断
  问题:有些测试是"软规则",失败时下游可继续

severity=warn → 告警但继续
  问题:可能被忽略

正解:按 test 类型配
  unique / not_null / FK 完整性 → error
  分布 / 阈值类 → warn
```

### 9.4 数据 owner 不清

```
表叫 user_summary,不知道谁维护
出问题没人修
```

**正解**:**所有表必须有 owner**(团队 + 个人 + Slack),元数据治理强制。

### 9.5 上下游对账缺失

```
ODS 1 亿行,DWD 0.97 亿行(3% 损失)
没人对账 → 损失合理还是 ETL 错没人知道
```

**正解**:**ODS → DWD → DWS** 每一步留对账(总数 / 总金额 / 关键维度),失败告警。
