# 批处理替代品:Trino、Presto、DuckDB、ClickHouse、Polars

Spark 不是唯一答案。**亚秒级交互查询** Spark 干不了(启动太慢)、**笔记本上扫 100GB Parquet** 起 Spark 集群是浪费、**实时大屏 + 高并发分析** Spark 撑不住——这些场景每一种都有更合适的工具。这一篇拉一张选型地图:Trino / Presto 干联邦查询 + 交互 BI、DuckDB / Polars 干单机分析、ClickHouse / Doris / StarRocks 干实时 OLAP。看完知道每种场景该掏哪把刀。

> 一句话先记住:**Spark 强在「大、批、复杂、ML」**,但**「交互、单机、实时高并发」每一种都有更强的替代品**。2025 工程标配是「**Spark 跑离线 ETL + Trino 跑 Ad-hoc / BI + DuckDB 跑本地探索 + ClickHouse 跑实时聚合**」组合拳。

## 一、为什么需要 Spark 之外的东西

### 1.1 Spark 的短板

```
启动延迟      Driver + Executor 起来 5-30 秒
              交互式查询体验差
              
并发能力      单 Application 容纳少量并发查询
              几千 QPS 的 Web 查询不是 Spark
              
单机不友好    Local 模式跑得动但调度开销大
              100GB 以内的数据用 Spark 是浪费
              
实时延迟      Structured Streaming 是微批,秒级
              真流(亚秒)要 Flink
              
高并发聚合    PB 级数据高并发实时分析撑不住
              ClickHouse / Doris 更专
```

### 1.2 数据工程的负载光谱

```
负载                           推荐引擎
─────────────────────         ────────────────────
PB 级离线批 ETL                Spark
复杂多表 join + 转换           Spark
ML 训练 / 大批量特征工程       Spark + PyTorch / MLlib
微批流处理                     Spark Structured Streaming

交互式 SQL / BI / Ad-hoc       Trino (Presto)
联邦查询(多源)               Trino
低延迟分析查询                  Trino / Snowflake / BigQuery

单机数据探索 / Notebook         DuckDB / Polars
中等数据(< 100GB)             DuckDB

实时大屏 + 高并发              ClickHouse / Doris / StarRocks
用户行为分析                   ClickHouse
监控 / 日志分析                 ClickHouse / Druid

真流处理                       Flink

OLTP                          MySQL / PostgreSQL
```

## 二、Trino / Presto:交互查询 + 联邦

### 2.1 出身和分裂

```
2012  Facebook 内部出 Presto
2013  Facebook 开源
2018  原创团队离开 Facebook,fork PrestoDB → 改名 PrestoSQL
2020  PrestoSQL → 改名 Trino(避免商标问题)
      留下的 Facebook 分支仍叫 PrestoDB

2025  Trino 是事实主流(开源活跃、企业用户多)
      PrestoDB 在 Meta / Uber 仍在用,社区较冷
```

### 2.2 心智:MPP 内存查询引擎

```
              Coordinator
              ─────────────
              解析 SQL、规划查询、调度
                     │
        ┌────────────┼────────────┐
        ▼            ▼            ▼
    Worker 1     Worker 2     Worker 3
    ─────────    ─────────    ─────────
    本地内存计算  本地内存计算  本地内存计算
    跨 Worker     跨 Worker
    Exchange      Exchange  (网络 Shuffle,但都在内存)
        │            │            │
        └────────────┴────────────┘
                     │
              数据源(Iceberg、Hive、MySQL、Kafka、ES、...)
              通过 Connector 抽象
```

**关键特征**:
- **纯内存计算**:中间结果不落盘(Spark 落 shuffle 盘)
- **MPP 架构**:Coordinator 拆任务,Worker 并行
- **Connector 模型**:60+ 连接器,统一 SQL 查不同数据源
- **亚秒级到秒级响应**:适合交互

### 2.3 联邦查询

```sql
-- 一条 SQL 跨 MySQL + S3 + Kafka
SELECT
    m.user_id,
    m.email,
    s.total_amount,
    k.last_login
FROM mysql.shop.users m
JOIN iceberg.shop.orders_summary s ON m.user_id = s.user_id
JOIN kafka.live.user_events k ON m.user_id = k.user_id
WHERE m.created_at > DATE '2025-05-01';
```

**Trino 的杀手锏**:**「一条 SQL,跨任意数据源」**——分析师不用导数据,直接查。

### 2.4 优势

- 启动快(查询亚秒到几秒)
- SQL 标准化好(ANSI SQL)
- 资源管理细(每查询限内存 / cpu)
- 用户体验好(类似 Snowflake 直连体验)
- 联邦查询无可替代

### 2.5 劣势

- **中途失败要全部重跑**:无 checkpoint
- **超大 ETL 不适合**:跑几小时挂了一次重来要命
- **内存压力**:超大 sort / agg 容易 OOM
- **写入能力弱**:主要是查询引擎,不擅长复杂 ETL 写入(Iceberg 写入支持有,但 Spark 主导)

### 2.6 工程落地

```ini
# /etc/trino/catalog/iceberg.properties
connector.name=iceberg
iceberg.catalog.type=rest
iceberg.rest-catalog.uri=https://catalog/api

# /etc/trino/catalog/mysql.properties
connector.name=mysql
connection-url=jdbc:mysql://...
```

```sql
-- CLI:trino --server https://trino-coord:8080
SHOW CATALOGS;
SELECT * FROM iceberg.shop.orders LIMIT 10;

-- 跨 catalog
SELECT u.name, SUM(o.amount)
FROM mysql.shop.users u
JOIN iceberg.shop.orders o ON u.id = o.user_id
GROUP BY u.name;
```

### 2.7 典型场景

- 公司内 BI / 分析师 Ad-hoc
- 多源数据联邦
- Iceberg 表的交互式查询
- 替代 Hive on Tez/MR 的查询入口

### 2.8 主要发行版

- **OSS Trino**(开源)
- **Starburst**:Trino 主要商业公司,企业级功能
- **AWS Athena**:Trino 托管(serverless,按扫描量计费)
- **GCP BigQuery Omni** / **Azure Synapse**:类似托管

## 三、DuckDB:单机分析的王者

### 3.1 定位:OLAP 界的 SQLite

```
SQLite                         DuckDB
──────                         ──────
单文件 OLTP DB                 进程内嵌 OLAP 引擎
事务、行存                     列存、向量化
点查 / 单行更新                 全表扫描 / 聚合
                               
日常 OLTP(配置、应用本地)     日常 OLAP(分析、Notebook、CLI)
```

### 3.2 心智

```
你的 Python 进程
  │
  ├──→ DuckDB(进程内嵌,无 server)
       │
       ├─→ 读 S3 / 本地 / HTTP 上的 Parquet / CSV / JSON
       ├─→ 读 Iceberg / Delta
       ├─→ 读 PostgreSQL / MySQL(扩展)
       ├─→ 向量化 + 列存执行
       └─→ 返回结果给 Python
```

**完全嵌入式**,没有 server,没有调度,没有集群。

### 3.3 杀手特性

- **零部署**:`pip install duckdb`,一行
- **跑得快**:笔记本上扫 100GB Parquet 几秒到几十秒
- **SQL 完备**:窗口函数 / CTE / 集合操作 / JSON / 地理(geo)
- **集成强**:Python / R / Java / Node 都有
- **直接读云存储**:`SELECT * FROM 's3://bucket/file.parquet'`

### 3.4 工程落地

```python
import duckdb

# 直接查
result = duckdb.sql("""
SELECT province, SUM(amount) AS gmv
FROM 's3://bucket/orders/year=2025/month=05/*.parquet'
WHERE status = 'paid'
GROUP BY province
ORDER BY gmv DESC
LIMIT 10
""").df()
```

```python
# 跟 Pandas / Polars / PyArrow 互操作
import pandas as pd
df = pd.read_csv("orders.csv")
result = duckdb.sql("SELECT province, SUM(amount) FROM df GROUP BY province").df()
# 直接查内存里的 DataFrame!
```

```bash
# CLI(类似 sqlite3)
duckdb my.db
duckdb> SELECT * FROM 's3://bucket/file.parquet' LIMIT 10;
```

### 3.5 什么时候用

- 单机分析(< 100GB,甚至到 1TB 也能跑)
- Jupyter / Notebook 分析
- 本地 ETL 原型
- CI 测试(替代起 Spark)
- 静态网站 / 个人分析

### 3.6 MotherDuck

DuckDB 的云端协同版本(2023+),Snowflake / BigQuery 杀手定位。**「Local-first,Cloud-syncable」**——本地写代码,可同步到云端协作。

### 3.7 谁在用

- 数据分析师日常工具
- 数据工程师本地开发
- AI 应用嵌入式查询(代替 SQLite for OLAP)
- 中小公司全量分析栈(直接 DuckDB + dbt-duckdb 起步)

## 四、Polars:Rust 写的现代 DataFrame

### 4.1 定位:Pandas 杀手 + DataFrame 路线

Pandas 是 Python 数据分析事实标准,但:
- 单线程
- 内存模型不友好(SettingWithCopyWarning 各种)
- 大数据不行(超内存就崩)

**Polars(2020+ Ritchie Vink 出)**:Rust 写,**向量化 + 多线程 + 内存高效 + Lazy execution**。

### 4.2 心智

```python
import polars as pl

# 跟 Pandas 几乎一样的 API,但更快
df = pl.read_parquet("orders.parquet")

result = (df
    .filter(pl.col("status") == "paid")
    .group_by("province")
    .agg(pl.col("amount").sum().alias("gmv"))
    .sort("gmv", descending=True))

print(result)
```

### 4.3 优势

- **快**:常见聚合比 Pandas 快 5-50x
- **内存友好**:Arrow 后端,零拷贝
- **Lazy execution**:`pl.scan_parquet()` 推迟执行 → 全图优化
- **流式处理**:超内存也能跑(`collect(streaming=True)`)
- **API 现代**:method chaining,跟 Spark DataFrame 接近

### 4.4 跟 DuckDB 对比

| | DuckDB | Polars |
| --- | --- | --- |
| 接口 | SQL 优先 | DataFrame 优先 |
| 嵌入式 | 是 | 是 |
| Lazy execution | 部分 | 强 |
| 流式 | 弱 | 强 |
| SQL 语法 | 全 ANSI SQL | 有 SQL 但弱 |
| 生态 | 通用分析 | Python 偏好 |

**互补,不冲突**——很多人两个都用,Polars 写脚本,DuckDB 做查询。

### 4.5 工程落地

```python
# Lazy execution(类似 Spark)
lf = pl.scan_parquet("s3://bucket/orders/*.parquet")  # 不读
result = (lf
    .filter(pl.col("event_time") >= pl.lit("2025-05-01"))
    .group_by("province")
    .agg(pl.col("amount").sum())
    .collect())  # 这里才执行

# 跟 Pandas 互转
import pandas as pd
pdf = result.to_pandas()
pl_df = pl.from_pandas(pdf)
```

## 五、ClickHouse / Doris / StarRocks:实时 OLAP

### 5.1 定位:实时聚合 + 高并发分析

```
负载                              引擎
────────                          ────────
实时大屏(秒级 GMV 累加)          ClickHouse / Doris
用户行为漏斗 / 留存分析             ClickHouse
运营看板(几千并发)                Doris / StarRocks
日志分析 / APM                     ClickHouse
监控时序                           ClickHouse / TimescaleDB
```

### 5.2 ClickHouse

**Yandex(俄罗斯搜索)2016 开源**,C++ 写,极致聚合性能。

**优势**:
- 列存 + 向量化 + SIMD 极致
- 单机吞吐惊人:一台机器查 100 亿行常见
- 实时写入(每秒百万级)
- 不绑死云

**劣势**:
- JOIN 弱(最近 24.x 改善,但仍不如 Snowflake)
- 强一致性弱(ReplicatedMergeTree 异步)
- 部署 / 运维有学习曲线
- update / delete 弱(走 ALTER 异步 mutation)

```sql
CREATE TABLE orders (
    id UInt64,
    user_id UInt64,
    amount Decimal(10,2),
    province LowCardinality(String),
    event_time DateTime
) ENGINE = MergeTree
ORDER BY (province, event_time, id)
PARTITION BY toYYYYMM(event_time);

-- 实时聚合(秒级)
SELECT province, sumMerge(amount) FROM orders
WHERE event_time >= now() - INTERVAL 1 HOUR
GROUP BY province;
```

### 5.3 Apache Doris(原百度 Palo)

**百度 2018 捐 Apache**,后来 StarRocks 从 Doris fork。

**优势**:
- JOIN 比 ClickHouse 强(物化视图 + Colocate Join)
- 协议:MySQL 兼容(BI 工具直接接)
- 实时写入 + 物化视图自动更新
- 中文社区强(国内大量公司在用)

### 5.4 StarRocks

**Doris 商业 fork(2020+)**,核心团队从 Doris 出来。

**优势**:
- 联邦查询(类似 Trino)+ 本地 OLAP
- 性能比 Doris 提升明显
- 商业化支持强

### 5.5 选哪个

```
极致聚合 + 单表场景             ClickHouse
复杂 JOIN + BI 工具友好          Doris / StarRocks
国内 + 强社区支持               Doris / StarRocks
全球生态 + 极致性能              ClickHouse
```

### 5.6 跟 Lakehouse 的关系

```
Iceberg + Spark + Trino:         vs       ClickHouse / Doris:
─────────────────────────                  ─────────────────────────
PB 级原始数据                              TB 级聚合后数据
开放、跨引擎                                专用、绑死引擎
批 + 流 + ML + SQL                          实时聚合 + 高并发
延迟秒到分钟                               延迟毫秒到秒
```

**两者互补**:Iceberg 存事实,ClickHouse 存「打平后的高频查询表」。**典型架构**:
```
原始数据 → Iceberg(事实)→ Spark/dbt 转换 → 推到 ClickHouse(服务) → BI / 实时大屏
```

## 九、几个误区

### 9.1 「Spark 已死」

**不会**。Spark 仍是 PB 级离线批 + ML 训练的第一选择。它会被新 Native 引擎包装(Photon / Gluten),但核心 API 和生态稳定。

### 9.2 「DuckDB 取代 Snowflake」

**部分场景成立**。但 Snowflake 的「**托管 + 高并发 + 团队协作 + 治理**」DuckDB 替代不了。**DuckDB 取代的是分析师本地的 Pandas / Excel**。

### 9.3 「ClickHouse 取代 Iceberg」

**完全不冲突**。ClickHouse 是「专用聚合 OLAP 引擎」,Iceberg 是「开放表格式」。两个层次的东西。

### 9.4 「Trino 取代 Spark」

**部分场景成立**(交互查询、轻 ETL),**重 ETL 不行**(无 checkpoint、超大查询易挂)。
