# 流处理心智:时间、窗口、状态、Exactly-Once

批处理的世界很安静:数据是「过去」、是「全部」、跑完就结束。流处理的世界很乱:数据是「正在到达的」、是「永远没完」、还可能**乱序到达**(手机离线 3 小时后一起上传)。所以流处理几乎重新定义了「时间」「窗口」「状态」「正确性」这四件事——批的直觉过来全是坑。这一篇拉清楚流处理的四件底子,**理解了这一篇,19/20/21 篇的 Watermark / Flink / 状态后端 才有意义**。

> 一句话先记住:**流处理 = 时间 + 窗口 + 状态 + 容错语义**。事件时间 vs 处理时间是流处理的第一道坎,窗口决定「什么时候出结果」,状态决定「记住什么」,Exactly-Once 不是「数据只处理一次」,是**「结果对外可见只发生一次」**(端到端事务)。

---

## 一、批 vs 流的根本差异

### 1.1 数据形状

```
批处理 (Batch)              流处理 (Stream)
────────────                ──────────────
有界数据集(已经存在)        无界数据流(持续到达)
跑完结束                    永远不结束
全量可见再算                 边到达边算
                            
重新跑结果一样              输出会随时间演进
顺序无关                     顺序很重要(但常乱序)
                            
适合 ETL / 离线建模          适合实时大屏 / 实时风控 / 监控
```

### 1.2 「同一段逻辑」的不同含义

```sql
-- 同一段 SQL:
SELECT date, SUM(amount) FROM orders WHERE event_time >= '2025-05-01' GROUP BY date;

-- 批语义:
-- 一次性扫整张表,过滤,聚合,输出最终结果
-- 输入完整 → 输出完整 → 结束

-- 流语义:
-- 数据持续到达,过滤,持续累加 sum
-- 每多一行,sum 多一点
-- 什么时候"输出"?(下一节讲窗口)
-- 永远不结束
```

## 二、时间:Event Time vs Processing Time

### 2.1 两种时间的根本差异

```
Event Time:    数据本身记录的时间(事件实际发生)
                例:用户 12:00 点了"加购物车",这个 12:00 就是 event time
                
Processing Time:数据被处理系统接收的时间
                例:用户 12:00 点了,但手机离线,2:30 一起上传 → processing time = 2:30
                
两者之间永远有延迟,且可能极大(几小时、几天)
```

### 2.2 必须画图:时间轴

```
              Event Time (业务时间) →
               
  事件 A: 12:00 ───────────────────────────────────────
                      ↓
                      在网络 + 服务器 + 消息队列里传输
                      ↓
  事件 A 到达流处理系统: 12:02         ← Processing Time
  
  事件 B: 12:30 ─────┬─────────────────────────────────
                    ↓
                    手机离线 3 小时
                    ↓
  事件 B 到达系统: 15:35              ← Processing Time(晚到)
  
  事件 C: 13:00 ───────────────────────────────────────
  事件 C 到达系统: 13:01
  
  事件 D: 14:30 ───────────────────────────────────────
  事件 D 到达系统: 14:32
  
                                       Processing Time →
  到达系统的顺序:    A → C → D → B
  Event Time 顺序:   A → B → C → D
                     ↑          ↑
                     B 乱序到达,比 C/D 都晚
```

**乱序是必然的**——网络、缓存、离线设备、分布式延迟、消息队列重排,都会让 event time 顺序和到达顺序不一致。

### 2.3 用哪个时间

```
分析 "2025-05-01 当天 GMV"
       ↑
       事件时间(business 关心当天发生的)
       
监控 "过去 5 分钟系统报错数"
       ↑
       处理时间(关心现在的故障)
       
计费 "用户实际使用时间"
       ↑
       事件时间(签合同时按 event time 算)
```

**绝大部分业务场景要 event time**——这是流处理跟批处理对齐的关键。

## 三、窗口:什么时候出结果

### 3.1 流处理需要窗口的原因

流是无界的,聚合(SUM / COUNT / AVG)没有「全部」概念。所以必须切窗口:**"每 5 分钟,把这 5 分钟的数据聚合成一个输出"**。

### 3.2 三种主流窗口

```
1. 固定窗口 (Tumbling Window)
   |---5 min---|---5 min---|---5 min---|
   不重叠,每条事件属于一个窗口
   
   例:每 5 分钟统计 GMV
   12:00-12:05 → 30万
   12:05-12:10 → 28万
   ...

2. 滑动窗口 (Sliding Window)
   |---5 min---|
        |---5 min---|
             |---5 min---|
   窗口大小 5 min,每 1 min 滑动
   每条事件可能属于多个窗口
   
   例:最近 5 分钟 GMV,每 1 分钟更新

3. 会话窗口 (Session Window)
   |--user A 活跃--|        |--user A 又活跃--|
                  ↑
              5 min 不活跃 → 切窗口
   按用户活动间隔切
   
   例:用户连续浏览作为一个 session,gap 5 min 切
```

### 3.3 触发(Trigger)

「这个窗口什么时候算完,出结果?」

```
Event Time Trigger:  Watermark 越过 window end 时(19 篇展开)
Processing Time Trigger: 按物理时钟到点
Count Trigger:        累计 N 个事件后
Composite Trigger:    组合(早出近似 + 最终精确)
```

### 3.4 工程例子(Flink SQL)

```sql
-- 固定窗口
SELECT
    TUMBLE_START(event_time, INTERVAL '5' MINUTE) AS win_start,
    SUM(amount) AS gmv
FROM orders
GROUP BY TUMBLE(event_time, INTERVAL '5' MINUTE);

-- 滑动窗口
SELECT
    HOP_START(event_time, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE) AS win_start,
    SUM(amount) AS gmv
FROM orders
GROUP BY HOP(event_time, INTERVAL '1' MINUTE, INTERVAL '5' MINUTE);

-- 会话窗口
SELECT
    SESSION_START(event_time, INTERVAL '5' MINUTE) AS win_start,
    user_id,
    COUNT(*) AS clicks
FROM clicks
GROUP BY SESSION(event_time, INTERVAL '5' MINUTE), user_id;
```

## 四、状态:流处理 = 状态管理

### 4.1 为什么流处理本质是状态管理

```
SUM(amount):           累加器 = 状态
COUNT DISTINCT user:    存所有看过的 user id = 状态
JOIN stream A and B:    缓存 A / B 一侧等另一侧 = 状态
模式匹配 CEP:           保留事件序列 = 状态
最近 5 分钟统计:        窗口内的所有事件 = 状态
```

**没有状态 = 只能做无状态的 map/filter**(变换 / 过滤)——但那不是真正的流处理。

### 4.2 状态的种类

```
键控状态 (Keyed State)        非键控状态 (Operator State)
───────────────                ──────────────────────────
按 partition key 隔离          整个算子共享
最常用                          少用
                               
ValueState                      ListState (Kafka Source 存 offset)
ListState                       BroadcastState (规则广播)
MapState
ReducingState
AggregatingState
```

详见 21 篇(Flink 状态后端)。

### 4.3 状态从哪来,放哪儿

```
状态生命:
  创建 → 写入 → 读取 → (TTL 过期) → 清理

状态存储:
  内存(快但易丢)
  RocksDB(堆外 + 磁盘,大状态)
  外部(Redis、HBase,几乎没人这么用)

状态恢复:
  Checkpoint(Flink)→ 定期 snapshot 到 S3
  失败时从 checkpoint 恢复
```

详见 21 篇。

## 五、交付语义:At Most / At Least / Exactly Once

### 5.1 三种语义

```
At Most Once     最多一次
                 可能丢消息,不重复
                 没人在生产用(除非可丢数据的监控)

At Least Once    至少一次
                 不丢消息,可能重复
                 默认,业务侧靠幂等处理重复
                 
Exactly Once     精确一次(语义层面)
                 不丢,不重复(在输出可见层面)
                 Flink / Spark Streaming + 事务 Sink 都能做到
```

### 5.2 Exactly Once 的真实含义(关键!)

```
误解:每条数据在流处理引擎内部只处理一次  ❌
真相:对外可见的最终结果,每条数据只贡献一次 ✓
       (引擎内部可能 reprocess,但输出端做了去重 / 事务)
```

举例:Flink 处理 100 条订单,聚合 SUM 后写到 ClickHouse:
- Flink 内部:可能因为 checkpoint 失败重跑某些数据
- 写 ClickHouse:**通过两阶段提交 / 事务写入**,保证每条订单只贡献一次到 SUM 结果

### 5.3 Exactly Once 的实现要素

```
① Source 端可重放:Kafka offset 可 seek
② 引擎内部可恢复状态:Checkpoint 一致性快照
③ Sink 端事务/幂等:
    Kafka 事务、Iceberg snapshot 原子 commit、JDBC 幂等 UPSERT、
    ClickHouse Replicated 表 + insert_deduplicate
```

**三者缺一不可**。

```
Source: Kafka (可 seek offset)
       ↓
Engine: Flink (Checkpoint + Aligned Barrier)
       ↓
Sink:   Iceberg (snapshot 原子) / Kafka transactions / 幂等 sink
       
= Exactly Once
```

### 5.4 真实工程中怎么用

```python
# Flink Kafka Source + Iceberg Sink,Exactly Once 配置
env.enableCheckpointing(60000)  # 60s checkpoint
env.getCheckpointConfig().setCheckpointingMode(EXACTLY_ONCE)
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30000)

# Kafka source - 默认 Exactly Once 兼容
# Iceberg sink - 默认 Exactly Once(snapshot 原子)
```

## 六、Checkpoint:容错的核心

### 6.1 必画图

```
时间轴 →
          
Source ─── data ─── Op1 ─── data ─── Op2 ─── data ─── Sink
   │                  │                 │
   │                  │                 │
   └──────────────────┴─────────────────┘
                  Checkpoint Barrier
                  从 Source 注入,流过算子
                  
每个算子收到 barrier 后:
  1. 暂停处理新数据(or 缓存)
  2. 把当前状态 snapshot 到外部存储(S3 / HDFS)
  3. 转发 barrier 给下游
  4. 继续处理

所有算子都完成 → checkpoint N 成功

失败时:从 checkpoint N 恢复
  - Source 回到 offset (N 时刻的 offset)
  - 状态恢复到 N 时刻
  - 重新跑 N 之后的数据
```

### 6.2 Barrier Alignment

多输入算子(join、union)收到不同上游的 barrier 时:
- **Aligned Checkpoint**(默认):等所有上游的 barrier 都到,才 snapshot。**精确,但反压时阻塞**
- **Unaligned Checkpoint**(1.11+):不等,直接 snapshot,buffer 数据也一起存。**快,但 checkpoint 大**

详见 20 篇。

### 6.3 Savepoint vs Checkpoint

| | Checkpoint | Savepoint |
| --- | --- | --- |
| 触发 | 周期性自动 | 手动 |
| 目的 | 故障恢复 | 版本升级、迁移、A/B |
| 格式稳定性 | 跨版本可能不兼容 | 设计为跨版本兼容 |
| 性能优先 | 是(增量) | 否(完整) |
| 自动清理 | 是 | 否 |

## 七、流处理一定要面对的几个工程问题

### 7.1 乱序事件

```
Watermark + Allowed Lateness(19 篇)
```

### 7.2 状态膨胀

```
- 用户 ID 累加表,跑半年没 TTL → 状态 1TB → 内存炸 / Checkpoint 慢
- 解法:状态 TTL、RocksDB、增量 Checkpoint(21 篇)
```

### 7.3 反压

```
下游慢 → 网络 buffer 满 → 上游阻塞 → 整个 pipeline 慢
监控:Flink UI 标红的算子
解法:加资源、查 GC、改算子、加 sink 并发
```

### 7.4 时区 / 时间精度

```
event_time 是 UTC 还是本地?毫秒还是秒?
Watermark 单位错了 = 窗口永远不触发或者瞬间触发
```

### 7.5 Schema 演进

```
流式 source schema 变了,下游算子状态结构怎么改?
Flink Savepoint 升级时的 schema migration(21 篇)
```

## 八、流引擎选型(22 篇细讲)

### 8.1 Flink

- 真流处理(亚秒级)
- 强大状态管理 + Checkpoint
- SQL / DataStream API
- 复杂 CEP / Pattern matching
- **生产实时业务首选**

### 8.2 Spark Structured Streaming

- 微批(默认秒级,Continuous 实验性)
- 跟 Spark 生态共享(批 SQL 完全兼容)
- 简单易用
- **跟离线 Spark 共用代码场景首选**

### 8.3 Kafka Streams

- 应用嵌入式(JVM 库)
- 不需要集群,直接跑在你的服务里
- 状态本地 RocksDB
- **规模有限,适合微服务内嵌流处理**

### 8.4 其他

- ksqlDB(Confluent),SQL 写 Kafka 流
- Apache Beam(抽象 + Runner),Google 推
- Pathway / RisingWave(新晋,2023+),物化视图风格

## 九、工程落地:一个最小流处理 job

需求:实时统计「每 5 分钟、每个省份的 GMV」,允许 1 分钟乱序。

```sql
-- Flink SQL
CREATE TABLE kafka_orders (
    id BIGINT,
    user_id BIGINT,
    province STRING,
    amount DECIMAL(10,2),
    event_time TIMESTAMP(3),
    WATERMARK FOR event_time AS event_time - INTERVAL '1' MINUTE
) WITH (
    'connector' = 'kafka',
    'topic' = 'orders',
    'properties.bootstrap.servers' = 'kafka:9092',
    'format' = 'avro-confluent',
    'avro-confluent.url' = 'http://schema-registry:8081'
);

CREATE TABLE clickhouse_gmv (
    win_start TIMESTAMP(3),
    province STRING,
    gmv DECIMAL(20,2),
    PRIMARY KEY (win_start, province) NOT ENFORCED
) WITH ('connector' = 'clickhouse', ...);

INSERT INTO clickhouse_gmv
SELECT
    TUMBLE_START(event_time, INTERVAL '5' MINUTE) AS win_start,
    province,
    SUM(amount) AS gmv
FROM kafka_orders
GROUP BY 
    TUMBLE(event_time, INTERVAL '5' MINUTE),
    province;
```

这段代码包含:
- Source(Kafka,Avro,Schema Registry)
- Watermark(event time - 1 min)
- 窗口(5 分钟固定)
- 状态(每窗口每省的累加,Flink 内部 ValueState)
- Sink(ClickHouse,upsert 写)
- 实际 Exactly Once:Flink Checkpoint + ClickHouse insert_deduplicate

## 十、流处理的局限:不是万能药

### 10.1 状态维护成本

每个有状态算子都需要 checkpoint、TTL、compaction、监控。**不要为了"实时"而上流处理**——业务接受 10 分钟延迟,就走批 / 微批。

### 10.2 调试难

- 数据在飞,出错不容易复现
- Watermark 卡住、状态 corrupt、反压、checkpoint 失败,排查都不简单

### 10.3 写流式 SQL 是另一门技能

- 流的 JOIN / 窗口语义跟批不一样
- 数据迟到、窗口提前触发、retract 流的处理
- 团队需要训练

### 10.4 成本

Flink 集群常驻,RocksDB 状态 + S3 Checkpoint 写入持续费用。**不要把所有批任务都改成流**。
