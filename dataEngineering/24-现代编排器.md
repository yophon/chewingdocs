# 现代编排器:Dagster、Prefect、资产化、声明式管道

Airflow 设计的世界观是「**任务流(Task Flow)**」——关心 task 跑没跑、跑了多久、谁先谁后。但数据团队的实际心智是「**数据资产(Data Asset)**」——关心订单表是否更新到了今天、推荐特征是否就绪、模型训练数据是否完整。这两套世界观的错位,催生了 Dagster(2018)和 Prefect(2018)。这一篇拆「资产化」「声明式管道」这套新范式,以及现代编排器在工程实践里跟 Airflow 的核心差异。

> 一句话先记住:**Airflow 关心「task 跑了没」,Dagster 关心「数据资产是否就绪」**——一个是"流程视角",一个是"资产视角"。资产视角让数据团队能直接回答业务问题(「这张表上次更新在什么时候」「改了字段哪些下游受影响」),这是 Airflow 长不出来的。Prefect 走另一条路,把 Airflow 现代化(纯 Python、原生异步、动态 DAG)。

---

## 一、为什么 Airflow 之后还需要新东西

### 1.1 Airflow 的根本错位

```
Airflow 的视角:
  「Task A 跑了 → Task B 可以跑了」
  关注流程

数据团队的视角:
  「订单表更新到了 2025-05-01」
  「推荐特征表已就绪」
  「模型训练数据集 Frozen」
  关注数据资产
```

Airflow UI 给你看「DAG 跑了 100 次,89 次成功 11 次失败」——但回答不了「订单数据现在新到哪一天」。

### 1.2 工程层面的痛

```
Airflow 痛点                            现代编排器解法
────────────────────                    ──────────────────────
Task 难单元测试                          I/O Manager(Dagster)
代码 / 配置混杂                          Resource + Config(Dagster)
血缘要靠插件                              资产间依赖自动推
回算逻辑要手写                            按资产 materialization
数据质量散落                              内置 Asset Check
动态 DAG 受限                            原生支持(Prefect)
```

## 二、Dagster:资产中心的编排器

### 2.1 出身

```
2018  Nick Schrock 创立 Elementl(后改名 Dagster Labs)
      Schrock 是 Facebook GraphQL 主创之一
2019  Dagster 0.x 开源
2022  Software-defined Asset (SDA) 概念落地,资产化转向
2024  Dagster+ 商业云推出,生态成熟
```

### 2.2 核心抽象:Software-defined Asset

```python
from dagster import asset

@asset
def raw_orders():
    """从 Kafka 拉订单数据"""
    return read_from_kafka("orders")

@asset
def cleaned_orders(raw_orders):
    """清洗 + 标准化"""
    return clean(raw_orders)

@asset(deps=[cleaned_orders, "dim_users"])
def daily_summary():
    """按天聚合"""
    spark_sql("""
        SELECT date, SUM(amount) FROM cleaned_orders
        JOIN dim_users USING (user_id)
        GROUP BY date
    """)
```

Dagster 自动**从参数依赖推 DAG**——`cleaned_orders` 接收 `raw_orders` → 隐式声明依赖。

### 2.3 资产视角 vs 任务视角

```
Airflow:                           Dagster:
                                  
@task                              @asset
def extract(): ...                 def raw_orders(): ...
                                   (返回的就是这个资产的物化结果)
@task
def transform(data): ...           @asset
                                   def cleaned_orders(raw_orders): ...
                                   (清洗后的资产)
extract() >> transform()           # 自动推导依赖

UI:                                UI:
  task 状态                          资产是否物化、上次时间、上下游
  跑了多少次                         可点击「物化 cleaned_orders」
                                    
                                    SQL 改了 → 自动重新物化
```

### 2.4 I/O Manager:把 IO 跟逻辑解耦

```python
from dagster import io_manager, IOManager

class S3ParquetIOManager(IOManager):
    def handle_output(self, context, obj):
        # obj 是返回的 DataFrame
        obj.write.parquet(f"s3://bucket/{context.asset_key}/")
    
    def load_input(self, context):
        return spark.read.parquet(f"s3://bucket/{context.upstream_output.asset_key}/")

@io_manager
def s3_parquet_io_manager():
    return S3ParquetIOManager()

# 资产代码本身只关心业务逻辑
@asset(io_manager_key="s3")
def cleaned_orders(raw_orders):
    return clean(raw_orders)  # 不需要写 IO 代码
```

**好处**:
- 业务逻辑 = 接收数据 + 返回数据,纯函数式
- **测试时换成 InMemoryIOManager**,本地直接跑
- 生产 / 测试 / dev 用不同 IOManager,逻辑代码不变

### 2.5 Resource:依赖注入

```python
from dagster import resource

@resource
def spark_session(context):
    return SparkSession.builder.appName("...").getOrCreate()

@asset(required_resource_keys={"spark"})
def cleaned_orders(context, raw_orders):
    spark = context.resources.spark
    return spark.createDataFrame(raw_orders).transform(...)

# 测试时注入 mock
test_resources = {"spark": local_spark_session}
```

### 2.6 Asset Checks(内置数据质量)

```python
from dagster import asset_check, AssetCheckResult

@asset_check(asset=cleaned_orders)
def check_no_null_user_id(cleaned_orders):
    null_count = cleaned_orders.filter("user_id IS NULL").count()
    return AssetCheckResult(
        passed=null_count == 0,
        metadata={"null_count": null_count}
    )
```

**数据质量直接挂在资产上**——不需要单独 Great Expectations 套件(虽然 GE 也能集成)。

### 2.7 Schedule / Sensor / Job

```python
from dagster import schedule, sensor, define_asset_job

# 定义 Job(一组资产)
daily_orders_job = define_asset_job(
    "daily_orders_job",
    selection=["cleaned_orders", "daily_summary"],
)

@schedule(cron_schedule="0 2 * * *", job=daily_orders_job)
def daily_schedule(context):
    return {"ops": {...}}

@sensor(job=daily_orders_job)
def new_data_sensor(context):
    if check_s3_new_data():
        yield RunRequest(...)
```

### 2.8 Dagster 跟 dbt 深度集成

```python
from dagster_dbt import dbt_assets

@dbt_assets(manifest=...)
def shop_dbt_assets(context):
    yield from dbt.cli(["run"], context=context).stream()
```

dbt model 自动变成 Dagster Asset,跟其他 Asset 的依赖也自动连——**dbt + Dagster 是当下数据团队的"黄金组合"**。

## 三、Prefect:Airflow 的现代化重写

### 3.1 出身

```
2018  Jeremiah Lowin (Airflow 核心贡献者) 创立 Prefect
2019  Prefect 1.0 开源
2022  Prefect 2.0 重写,API 大改
2024  Prefect 3.0,生态扩展
```

**核心理念**:**「Airflow 的痛点我都懂,我重新写一遍」**。

### 3.2 核心抽象:Flow + Task

```python
from prefect import flow, task

@task
def extract():
    return read_from_kafka()

@task
def transform(data):
    return clean(data)

@task
def load(data):
    write_to_iceberg(data)

@flow
def daily_orders_etl():
    raw = extract()
    cleaned = transform(raw)
    load(cleaned)

if __name__ == "__main__":
    daily_orders_etl()
```

### 3.3 跟 Airflow 比的几个改进

| | Airflow | Prefect |
| --- | --- | --- |
| 代码风格 | Operator 模板 | Python decorator |
| 异步 / 并发 | Worker 限制 | 原生 asyncio |
| 动态 DAG | 受限(DAG 必须静态) | 运行时决定 task 数量 |
| 本地运行 | 需要 Airflow 实例 | `python my_flow.py` 直接跑 |
| 状态管理 | metadata DB | Prefect Cloud 或自建 |

### 3.4 动态 DAG 例子

```python
@flow
def process_files():
    files = list_s3_files()  # 运行时知道有多少文件
    for f in files:
        process_file.submit(f)  # 动态产生 N 个 task
```

Airflow 静态 DAG 做不到,Prefect 一行搞定。

### 3.5 Deployment

```python
from prefect import flow

@flow
def my_flow(): ...

if __name__ == "__main__":
    my_flow.serve(
        name="daily-etl",
        cron="0 2 * * *",
        tags=["daily"]
    )
```

`flow.serve()` 起一个 Worker 进程,常驻等 Schedule。也可 Deploy 到 Prefect Cloud。

### 3.6 Prefect Cloud

Prefect 主推 SaaS 模式:
- **控制面在云端**(免运维 Webserver / Scheduler / DB)
- **Worker 在你本地**(执行环境隔离,数据不出网)
- 类似 Dagster Cloud / Airflow Astronomer

## 四、Dagster vs Prefect vs Airflow:四维对比

### 4.1 心智模型

| | Airflow | Dagster | Prefect |
| --- | --- | --- | --- |
| 核心 | DAG + Task | Asset + Op | Flow + Task |
| 视角 | 任务流 | 数据资产 | 任务流(现代化) |
| 数据感知 | 弱 | 强 | 中 |
| Python 风格 | Operator 模板 | Asset/Op decorator | Flow/Task decorator |

### 4.2 工程体验

| | Airflow | Dagster | Prefect |
| --- | --- | --- | --- |
| 本地开发 | 需要起 Airflow | 单进程跑测试 | `python flow.py` |
| 单元测试 | 痛 | 内置好 | 容易 |
| 配置管理 | Variables / Connections | Resource + Config | Block(类似) |
| 血缘 | 插件(OpenLineage) | 原生 + Asset 视图 | 通过 Block + Asset(2.0+) |
| 数据质量 | 外部(GE / Soda) | 内置 Asset Check | 集成 GE / Soda |

### 4.3 生态

| | Airflow | Dagster | Prefect |
| --- | --- | --- | --- |
| Operator / Integration | 1000+ | 100+(但 dbt / Spark / Snowflake 强) | 100+ |
| dbt 集成 | 通过 BashOperator 调用 | **一等公民,Asset 自动同步** | 集成有,不如 Dagster |
| 商业支持 | Astronomer / MWAA / Composer | Dagster+ | Prefect Cloud |

### 4.4 适合规模

| | Airflow | Dagster | Prefect |
| --- | --- | --- | --- |
| 小团队 (< 5 人) | 可以但有点重 | 友好 | 极友好 |
| 中团队 (10-50 人) | 主流 | 主流 | 中 |
| 大团队 (50+ 人) | 老牌 | 渐成主流 | 偏少 |

## 五、声明式管道:工程范式的转向

### 5.1 命令式 vs 声明式

```
命令式(Airflow Task):
  「跑 dbt run --select daily_orders」
  → 用户告诉系统怎么做
  
声明式(Dagster Asset):
  「我要 daily_orders 这个资产存在,且依赖 cleaned_orders」
  → 用户告诉系统要什么,系统决定怎么实现
```

### 5.2 声明式的好处

```
1. 依赖关系自动推导
   asset 函数接收什么参数 = 它依赖什么
   
2. 自动算回算范围
   "重算 daily_summary" → 系统自动找到所有上游资产 + 重新物化
   
3. 数据资产视图直接可用
   UI 显示每张表的状态 / 上次更新 / 上游链路
   
4. 增量 / 全量 / 物化策略由系统选
   用户只声明"我要这张表",怎么物化系统配
```

### 5.3 dbt 的声明式精神

dbt 本来就是声明式:你声明 model 间的 `ref()`,dbt 自动算依赖。**Dagster 把这套精神扩展到 dbt model 之外的所有数据资产**(Python 转换、Spark 任务、ML 模型等)。

### 5.4 Delta Live Tables (Databricks)

Databricks 的 DLT 是声明式管道的另一个实现:

```sql
CREATE STREAMING LIVE TABLE bronze_orders
COMMENT "原始订单数据"
AS SELECT * FROM cloud_files("s3://...", "json");

CREATE STREAMING LIVE TABLE silver_orders
COMMENT "清洗"
AS SELECT 
    id, user_id, amount
FROM STREAM(LIVE.bronze_orders)
WHERE amount > 0;

CREATE LIVE TABLE gold_daily_summary
AS SELECT DATE(event_time), SUM(amount)
FROM LIVE.silver_orders
GROUP BY 1;
```

DLT 自动:
- 推导依赖
- 处理增量 / 全量
- 数据质量(`EXPECT amount > 0`)
- 监控
- 重试

## 七、典型迁移路径:Airflow → Dagster

### 7.1 起步:并行运行

```
现状:
  100 个 Airflow DAG 跑生产
  
第一步:
  新项目 / 新数据资产 → Dagster
  Airflow 继续跑老资产
  
两套并存几个月到一年
```

### 7.2 中期:dbt + Spark 资产迁

```
最容易迁:dbt 任务
  Airflow 里就是 `dbt run` 包装
  → Dagster 用 dagster-dbt,自动变 Asset
  
Spark 任务次之:
  Airflow BashOperator → Dagster @asset + Resource
```

### 7.3 长期:核心迁完

```
Airflow 只剩"非数据"任务(发邮件、第三方 API)
或完全淘汰
```

### 7.4 选择不迁

```
已有 Airflow 跑得稳 + 团队熟 + 没明显痛
→ 不迁,资源用别处
```

**迁工具是手段不是目标**。

## 八、工程落地:同一个 ETL 三种写法

### 8.1 业务

读 Kafka 订单 → 清洗 + 加维度 → 按天聚合 → 落 Iceberg。

### 8.2 Airflow

```python
@dag(schedule='@daily')
def daily_pipeline():
    extract = BashOperator(
        task_id='extract',
        bash_command='kafka-pull --topic orders --out /tmp/raw.json'
    )
    transform = BashOperator(
        task_id='transform',
        bash_command='dbt run --select silver_orders+'
    )
    test = BashOperator(
        task_id='test',
        bash_command='dbt test --select silver_orders+'
    )
    extract >> transform >> test
```

### 8.3 Dagster

```python
@asset(io_manager_key="kafka")
def raw_orders():
    return pull_from_kafka("orders")

@asset(io_manager_key="iceberg")
def silver_orders(raw_orders, dim_users):
    return clean_and_join(raw_orders, dim_users)

@asset(io_manager_key="iceberg")
def daily_summary(silver_orders):
    return aggregate(silver_orders)

@asset_check(asset=silver_orders)
def check_silver_orders(silver_orders):
    return AssetCheckResult(passed=silver_orders.count() > 0)

# Schedule
daily_job = define_asset_job("daily", selection="daily_summary")
daily_schedule = ScheduleDefinition(job=daily_job, cron_schedule="0 2 * * *")
```

### 8.4 Prefect

```python
@task
def extract():
    return pull_from_kafka("orders")

@task
def transform(raw, users):
    return clean_and_join(raw, users)

@task
def aggregate(silver):
    return summarize(silver)

@flow(name="daily_pipeline")
def daily_pipeline():
    raw = extract()
    users = load_dim_users()
    silver = transform(raw, users)
    daily = aggregate(silver)
    write_to_iceberg(daily)

if __name__ == "__main__":
    daily_pipeline.serve(name="daily-etl", cron="0 2 * * *")
```

## 十、几个误区

### 10.1 「Dagster 完全取代 Airflow」

不会(至少短期不会)。Airflow 的:
- 1000+ Operator 生态
- 大量公司投资
- 简单灵活的"什么都能跑"

仍然有价值。Dagster 在「**数据资产化**」场景胜出,但「**通用任务调度**」Airflow 仍然强。

### 10.2 「资产化必须用 Dagster」

不一定。dbt 已经是资产化(model = 资产),很多团队 dbt + Airflow 组合也能拿到 70% 资产化好处。Dagster 把这套扩展到所有数据资产。

### 10.3 「Prefect 是新 Airflow」

部分对。Prefect 的「现代 Python + 动态 DAG + 原生异步」是真的现代化,但**资产化视角它没 Dagster 强**。

### 10.4 「我们小,不需要编排器」

小项目几个 cron 也能跑,但数据团队一旦 > 3 人,**依赖管理 + 重试 + 监控** 用编排器收益就大。不一定上 Airflow,Prefect 或 dbt + cron 都行。
