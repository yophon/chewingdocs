# 模式演进与数据契约:上游加一个字段,为什么下游会一周内陆续炸

订单团队在 `orders` 表上加了个 `coupon_id` 字段,Slack 里没人通知,产品上线两周后陆续炸:**ETL 任务因为 schema 不匹配挂掉、BI 报表缺字段算错、推荐特征流水线产出脏数据、风控规则把新字段当 NULL 处理误判**——一个加字段的"小动作",在数据管道里能炸出一连串故事。这一篇讲清楚 Schema Evolution 的三种兼容性、Avro / Protobuf / JSON Schema 三选一、Schema Registry 怎么救命,以及 Data Contract 这套「数据有 API 合约」的新范式。

> 一句话先记住:**生产系统里,schema 不是设计文档,是部署中的运行时合约**——上游随便改,下游随便挂。**Schema Evolution 是工程纪律不是工具特性**:挑兼容方向(向后 / 向前 / 完全)、挑序列化(Avro / Proto)、挂 Schema Registry 强约束、再加 Data Contract 把责任明确到团队。这一套不上,你的数据管道就是定时炸弹。

## 一、没 schema 的世界什么样

### 1.1 一个真实事故的剧本

```
T+0    订单团队加字段 coupon_id (int, nullable)
       MySQL DDL 已上线,Binlog 已带上新字段
       Slack 工程频道有通知,但无下游消费者订阅

T+1    数据团队的 Flink CDC 任务挂了
       Avro schema mismatch: unknown field "coupon_id"
       夜班 oncall 重启加字段,问题暂时绕过

T+3    BI 报表"优惠总额"开始数字异常
       老逻辑用 amount 字段,新订单的优惠在 coupon_id
       BI 团队周一才发现,补数据 backfill

T+5    推荐特征流水线产出"用户优惠敏感度"特征
       coupon_id 在 ODS 里是 NULL(老解析逻辑没读)
       推荐模型把所有用户都当"不爱用券",离线 AUC 看不出来,上线 GMV 跌 1%

T+7    风控规则"短时间多次使用同一优惠"
       规则引擎里没读 coupon_id,误判率上升
       风控团队凌晨拉群定位
```

**一个加字段、五个团队连续踩坑、一个礼拜才完全稳定**。这就是没有 schema 治理的常态。

### 1.2 痛在哪

- **隐式契约**:上游写啥,下游就消费啥,字段含义靠口口相传
- **演进无规则**:加字段、删字段、改类型、改语义,谁都能改,谁也不知道影响面
- **失败延迟**:数据问题不像服务挂了立刻报警,要等 BI 报表 / 模型指标变化才发现,经常隔天甚至隔周
- **跨团队沟通成本高**:数据生产者(业务团队)和消费者(数据 / 分析 / 算法)经常不在一个组

## 二、Schema Evolution 的三种兼容性

### 2.1 三种兼容方向(必须记)

```
向后兼容 (BACKWARD)    新 schema 能读老数据
                       【典型】生产者升级新 schema,消费者不升
                       Kafka 默认选 BACKWARD

向前兼容 (FORWARD)     老 schema 能读新数据
                       【典型】消费者升级新 schema,生产者不升

完全兼容 (FULL)        双向都 OK
                       上下游各自独立升级,互不依赖

无兼容 (NONE)          不强制,完全自由
                       【典型】只用于初始建表
```

### 2.2 哪些操作走哪种兼容

```
                    BACKWARD  FORWARD  FULL
加字段(带 default)    ✓         ✓       ✓
加字段(无 default)    ✗         ✓       ✗
删字段(有 default)    ✓         ✓       ✓
删字段(无 default)    ✓         ✗       ✗
改字段类型(可转)      ✓         ✗       ✗
改字段名               ✗         ✗       ✗
改字段语义             ❌(永远不该做,所有规则都救不了)
```

**关键洞察**:**几乎所有"安全"的演进都需要带 default 值**——这是 Avro / Protobuf 反复强调的纪律。

### 2.3 实例:加 coupon_id

```json
// 老 schema (v1)
{
  "type": "record",
  "name": "Order",
  "fields": [
    {"name": "id", "type": "long"},
    {"name": "user_id", "type": "long"},
    {"name": "amount", "type": "double"}
  ]
}

// 新 schema (v2,BACKWARD 兼容)
{
  "type": "record",
  "name": "Order",
  "fields": [
    {"name": "id", "type": "long"},
    {"name": "user_id", "type": "long"},
    {"name": "amount", "type": "double"},
    {"name": "coupon_id", "type": ["null", "long"], "default": null}
    //                              ^ 必须 nullable + default
  ]
}
```

**default null + union with null** 是 Avro 的标准做法。生产者按 v2 写,消费者按 v1 读时,自动忽略未知字段;消费者按 v2 读时,老数据的 coupon_id 自动是 null。

## 三、三选一:Avro / Protobuf / JSON Schema

### 3.1 三家对照

| 维度 | Avro | Protobuf | JSON Schema |
| --- | --- | --- | --- |
| 出身 | Hadoop 生态(2009) | Google 内部(2008 开源) | JSON 标准(2009) |
| 序列化形态 | 二进制 + schema 单独 | 二进制 + tag 号 | 文本 JSON + 校验 |
| Schema 与数据 | **数据不带 schema**(单独传) | tag 号绑定字段 | 数据自描述但要校验 |
| 生态深度 | Kafka / Confluent / Hadoop / Spark | gRPC / 微服务 / Google Cloud | Web API / OpenAPI / 配置 |
| 演进规则 | 字段名 + default 救场 | tag 号是契约,字段名可改 | reserved keywords + version |
| 可读性 | 二进制(差) | 二进制(差) | 文本(好,但浪费体积) |
| 性能 | 中等(快) | 最快(纯二进制 + tag) | 最慢(文本 + 校验) |
| 体积 | 小 | 最小 | 大 |
| 学习曲线 | 中(默认值规则坑) | 中(tag 号规则坑) | 低(就是 JSON) |

### 3.2 选哪个

```
数据管道(Kafka + 数据仓库)        → Avro(Confluent Schema Registry 默认)
微服务接口(gRPC)                  → Protobuf
Web API / 公开接口 / 配置文件       → JSON Schema
事件溯源 / CQRS                     → Avro 或 Protobuf
跨语言通信(强类型)                 → Protobuf
```

**数据工程世界事实标准**:Avro。原因:
- Kafka + Confluent Schema Registry 深度集成
- Hive / Spark / Flink 原生支持
- Iceberg / Hudi / Parquet 都能存 Avro schema
- Schema 单独传 → 数据体积小,传输高效

### 3.3 Protobuf 的 tag 号陷阱

```protobuf
// v1
message Order {
  int64 id = 1;
  int64 user_id = 2;
  double amount = 3;
}

// v2:加 coupon_id,新 tag 4
message Order {
  int64 id = 1;
  int64 user_id = 2;
  double amount = 3;
  int64 coupon_id = 4;
}

// ❌ 危险:删 user_id 后又复用 tag 2 给新字段
message Order {
  int64 id = 1;
  string region = 2;  // 复用 tag 2!历史数据的 user_id 会被解析成 region 的乱码
  double amount = 3;
  int64 coupon_id = 4;
}

// ✓ 正确:删了 tag 必须 reserved
message Order {
  reserved 2;
  reserved "user_id";
  int64 id = 1;
  string region = 5;  // 用新 tag
  double amount = 3;
  int64 coupon_id = 4;
}
```

**Protobuf 的 tag 号是合同**,违约后果是**历史数据被错误解析,且没人能发现**——这是 Proto 比 Avro 更危险的地方,但也是它在性能和稳定性上更强的代价。

## 四、Schema Registry:把规则强制落地

### 4.1 没 Registry 之前

```
生产者                      消费者
  │                            │
  │  写 Kafka topic            │
  ├──► [bytes...] ────────────►│
  │     ↑                       │
  │   只有生产者知道            └─► 怎么解码???
  │   是按什么 schema 写的        猜?协调?写死?
```

### 4.2 引入 Confluent Schema Registry

```
生产者                Schema Registry            消费者
  │                       │                        │
  │ 1. 注册 schema v2     │                        │
  ├──────────────────────►│                        │
  │ 2. 拿到 ID=42         │                        │
  │◄──────────────────────┤                        │
  │ 3. 写 Kafka:           │                        │
  │   [magic][ID=42][bytes...]                     │
  ├─────────────────────────────────────────────► │
  │                       │                        │
  │                       │ 4. 拿 ID 反查 schema   │
  │                       │◄───────────────────────┤
  │                       │ 5. 返回 schema v2      │
  │                       ├───────────────────────►│
  │                       │       6. 反序列化      │
```

**两个核心**:
- **每条 Kafka 消息前 5 字节是 schema ID**(不是完整 schema)
- **Registry 维护版本演进规则**,违规注册会被拒绝(`{"compatibility": "BACKWARD"}` 配置)

### 4.3 兼容性检查在哪

```
生产者注册新 schema → Registry 拿到旧 schema 比对
                  ├─ BACKWARD:新 schema 能读旧数据? 校验
                  ├─ FORWARD:旧 schema 能读新数据? 校验
                  └─ FULL:双向?校验
                                ↓
                        通过 → 分配新 ID
                        失败 → 注册被拒,生产者报错
```

**关键**:**违规演进不需要等运行时炸,在注册环节就被挡住**。这是 Schema Registry 最大的工程价值。

## 五、工程落地:Kafka + Avro + Schema Registry 最小骨架

### 5.1 Schema 文件

```json
// orders-v1.avsc
{
  "type": "record",
  "namespace": "com.shop.events",
  "name": "Order",
  "fields": [
    {"name": "id", "type": "long"},
    {"name": "user_id", "type": "long"},
    {"name": "amount", "type": "double"},
    {"name": "created_at", "type": "long", "logicalType": "timestamp-millis"}
  ]
}
```

### 5.2 生产者(Python)

```python
from confluent_kafka import SerializingProducer
from confluent_kafka.serialization import StringSerializer
from confluent_kafka.schema_registry import SchemaRegistryClient
from confluent_kafka.schema_registry.avro import AvroSerializer

schema_registry = SchemaRegistryClient({"url": "http://schema-registry:8081"})

with open("orders-v1.avsc") as f:
    schema_str = f.read()

avro_ser = AvroSerializer(schema_registry, schema_str)
producer = SerializingProducer({
    "bootstrap.servers": "kafka:9092",
    "key.serializer": StringSerializer("utf_8"),
    "value.serializer": avro_ser,
})

producer.produce(
    topic="orders",
    key=str(order_id),
    value={
        "id": 123,
        "user_id": 456,
        "amount": 99.5,
        "created_at": 1715000000000,
    },
)
producer.flush()
```

### 5.3 消费者

```python
from confluent_kafka import DeserializingConsumer
from confluent_kafka.schema_registry.avro import AvroDeserializer

avro_de = AvroDeserializer(schema_registry)
consumer = DeserializingConsumer({
    "bootstrap.servers": "kafka:9092",
    "group.id": "orders-consumer",
    "value.deserializer": avro_de,
    "auto.offset.reset": "earliest",
})
consumer.subscribe(["orders"])

while True:
    msg = consumer.poll(1.0)
    if msg is None: continue
    order = msg.value()  # 自动按 schema ID 反序列化
    process(order)
```

**消费者代码完全不知道自己在读 v1 还是 v2**——AvroDeserializer 自动处理。这就是 Schema Registry 的价值。

### 5.4 演进 schema(加 coupon_id)

```json
// orders-v2.avsc
{
  ... // 同 v1
  "fields": [
    ... // 同 v1
    {"name": "coupon_id", "type": ["null", "long"], "default": null}
  ]
}
```

```bash
# CLI 注册并校验兼容性
curl -X POST -H "Content-Type: application/vnd.schemaregistry.v1+json" \
  --data @orders-v2.avsc \
  http://schema-registry:8081/subjects/orders-value/versions

# 如果 BACKWARD 兼容失败,会返回 409 Conflict 错误
# 部署 v2 生产者:消费者按 v1 仍能读新数据(忽略 coupon_id)
# 升级消费者按 v2:老数据的 coupon_id 自动是 null
```

## 六、Iceberg / 表格式的 Schema Evolution

数据湖侧不需要 Schema Registry(因为表本身就有 schema),Iceberg 直接在表层支持演进:

```sql
-- Iceberg 加列(任意时刻)
ALTER TABLE orders ADD COLUMN coupon_id BIGINT;

-- 删列(改 metadata,不重写数据)
ALTER TABLE orders DROP COLUMN region;

-- 改类型(部分支持:int → long、float → double 等向上扩展)
ALTER TABLE orders ALTER COLUMN amount TYPE DOUBLE;

-- 改字段名(改 metadata)
ALTER TABLE orders RENAME COLUMN coupon_id TO discount_coupon_id;

-- 重排字段顺序
ALTER TABLE orders ALTER COLUMN coupon_id AFTER amount;
```

**Iceberg 的演进核心**:每个字段有 `field-id`(整数,内部不变),metadata 维护 `field-id → 当前名/类型`,数据文件按 `field-id` 写入。**字段名变了不影响读老数据**——这跟 Protobuf 的 tag 号思想一致。

## 七、Data Contract:把责任写明

### 7.1 Schema 之外还缺什么

Schema 只解决「字段长什么样」,**不解决**:
- 字段语义(`status=2` 是已支付还是已取消?)
- 业务规则(`amount` 不能为负)
- 新鲜度(数据多久更新一次)
- 完整性(所有订单都该出现,不能漏)
- 所有权(出问题找谁)

这些是 **Data Contract** 要补的。

### 7.2 Data Contract 长什么样

```yaml
# orders-contract.yaml
contract:
  name: orders
  owner:
    team: order-platform
    slack: "#order-platform"
    on_call: pagerduty:order-platform
  
  schema:
    fields:
      - name: id
        type: long
        constraints: [primary_key, not_null]
      - name: user_id
        type: long
        constraints: [not_null, foreign_key:users.id]
      - name: amount
        type: double
        constraints: [not_null, "amount >= 0"]
      - name: status
        type: int
        constraints: [not_null, "status IN (1,2,3,4,5)"]
        semantics:
          1: created
          2: paid
          3: shipped
          4: completed
          5: cancelled
  
  sla:
    freshness: 5m       # 数据延迟不超过 5 分钟
    completeness: 99.9% # 99.9% 的订单必须出现
    
  policies:
    schema_evolution: BACKWARD
    breaking_changes: require_review
    deletion: snapshot_for_30days_then_delete
```

### 7.3 dbt 的 contract: true

dbt 1.5+ 内置 contract 概念:

```yaml
# models/marts/dim_users.yml
models:
  - name: dim_users
    config:
      contract:
        enforced: true   # 改 schema 时强制类型校验
    columns:
      - name: user_id
        data_type: bigint
        constraints:
          - type: not_null
          - type: primary_key
      - name: email
        data_type: varchar
        constraints:
          - type: not_null
```

部署时 dbt 跑 `dbt run`,如果 model 输出的 schema 跟 contract 不一致 → **直接报错,不让上线**。这就是合约的「执行力」。

### 7.4 工具生态

- **dbt contracts**:模型层合约
- **Datacontract.com / opendatacontract**:开源标准 + 工具
- **Soda Contracts**:数据质量 + 合约
- **Monte Carlo / Bigeye**:监控合约违约
- **DataHub**:合约登记 + 血缘 + 责任追溯

**合约不是文档,是 CI 里的 enforce**——这是和 README 的根本区别。

## 八、几个真实工程坑

### 8.1 无 default 的字段加上去

```json
// ❌ 错误演进
{"name": "coupon_id", "type": "long"}    // 必填!

// 老消费者读新数据 → 反序列化失败
// 新消费者读老数据 → 反序列化失败(老数据无此字段)
// 双向破坏
```

正确:`{"type": ["null", "long"], "default": null}`。

### 8.2 改字段语义不改名

```
v1: status (1=已下单, 2=已支付)
v2: status (1=待支付, 2=已支付, 3=已发货)  ← 同字段同名,语义变了
```

**Schema Registry 检测不到**(类型一样),**Data Contract 也检测不到**(没改 fields)——只有业务侧规则告警能抓到,但通常已经晚了。

正确做法:**新增字段** `status_v2`,老字段 deprecated,过渡期后删除。

### 8.3 Schema Registry 单点

Confluent Registry 单 Pod 跑 → 挂了所有生产/消费都 stuck。
- 生产环境必须 HA(3 节点 + Kafka 后端存储)
- 消费者本地缓存 schema,降级时仍能继续工作

### 8.4 跨团队部署顺序

升级 schema 要按顺序:
- BACKWARD:**先升消费者**(让新 schema 能读老数据),再升生产者
- FORWARD:**先升生产者**(让新 schema 能产新数据,老消费者也能读),再升消费者

搞反了就炸。**这个流程必须写进运维 SOP**。

### 8.5 Iceberg 字段重命名后老消费者

Iceberg `RENAME COLUMN` 是 metadata 操作,但是:
- **Spark / Trino / Flink 通过 Iceberg API 读** → 自动按 field-id,无感知
- **直接读 Parquet 文件**(绕过 Iceberg metadata) → 用旧字段名找不到
- **某些下游 dbt model** 可能有硬编码字段名

正解:重命名后,广播 + 留旧名 view 一段时间。

## 九、替代方案与立场

### 9.1 你完全可以不要 Schema Registry

**起步小项目**:
- 直接 JSON over Kafka,字段加加减减,team 内沟通
- PostgreSQL 里 jsonb 字段存原始事件
- 数据量不大、消费者就两个、改动不频繁 → 不上 Registry 没事

**风险来自规模**:消费者上 5 个、跨团队、跨服务、24x7 SLA 时,**没 Registry 几乎一定踩坑**。

### 9.2 Protobuf vs Avro 选型

```
数据管道(Kafka 中转)         → Avro(默认)
跨服务 RPC                    → Protobuf
混合(Kafka + gRPC 混用)      → Protobuf 一把梭(buf.build 工具链全套)
强 Web 友好                   → JSON Schema
```

字节、阿里这些大厂在 Kafka 里也用 Protobuf,Confluent Registry 同样支持 — 选型是工程偏好,不是技术对错。

### 9.3 Iceberg / Lakehouse 的 schema 演进取代 Schema Registry?

部分取代:
- 数据落湖之后,Iceberg 自管 schema
- Schema Registry 仍管「Kafka 流入这一段」

未来可能:
- Iceberg 表元数据 + Kafka schema 统一(Confluent 在做 Tableflow)
- Catalog 层(11 篇)做 schema 中心化登记
