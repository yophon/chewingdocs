# 数据工程系列写作计划

> 来源:`未来系列规划.md` 第三类「纵深主题」第 J 项,⭐⭐⭐。
> backendLearning 教了「**怎么用数据库 / 缓存 / 搜索引擎**」,aiLearning 零散提过 RAG 的数据预处理,但都没回答一个核心问题:**数据从生产到消费的整条管道,工程师怎么搭**。
> **看完这层,你能在白板前画出一家公司从埋点 / Binlog 到 BI 报表 + 推荐召回 + RAG 召回的完整数据链路**——Spark / Flink / Airflow / dbt / Iceberg 不再是"听过的名词",而是知道每个环节解决什么问题、什么时候上、什么时候过度设计。

---

## 设计思路

六层结构,从「心智」到「AI 时代」:

```
心智 + 范式(OLTP vs OLAP / ETL vs ELT / Lambda vs Kappa / 模式演进)
   → 存储与表格式(列存 / Parquet / Iceberg / Hudi / Delta / Lakehouse)
   → 批处理(Spark 三套 API / Shuffle / AQE / 调优)
   → 流处理(Flink / Watermark / Exactly-once / 状态后端 / Kafka)
   → 编排与建模(Airflow / Dagster / dbt / 维度建模 / Data Vault)
   → AI 时代数据工程(向量库 / 特征平台 / RAG 管道 / 数据质量)
```

**这个系列不是 Hadoop 教程**——Hadoop 那套以 MapReduce 为中心的世界已经过去,**本系列默认你站在 2025 之后**:对象存储 + 开放表格式 + 计算引擎解耦,Spark / Flink 是主力,Iceberg / dbt 是新约,向量库是 AI 时代的新增成员。

不深入到引擎源码,**而是讲清楚"数据工程师 / AI 工程师 / 后端工程师能看懂、要会用的图景"**——你写埋点、写离线任务、写实时大屏、写 RAG 检索,这些都站在数据管道上。

## 写作模板

每篇遵守"为什么这个工具 → 心智模型 → 工程落地 → 替代方案与局限"的四段式:

```
1. 为什么这个工具存在
   - 没它的世界长什么样(上一代是谁,死在哪)
   - 它解决的根本问题是什么

2. 心智模型(一张图)
   - 数据流图 / 时间线图 / 分层图,任选其一
   - 关键概念用最少的名词讲清楚
   - 不堆术语,术语第一次出现时一句话定义

3. 工程落地
   - 最小可跑的一段代码或配置(SQL / Python / YAML)
   - 真实场景:这家公司为什么这么用,踩过什么坑
   - 调优 / 监控 / 故障排查的关键指标

4. 替代方案与局限
   - 同代位竞争者(Spark vs Trino / Iceberg vs Hudi / Airflow vs Dagster)
   - 什么场景不该用它
   - 现在 2025 之后业界往哪个方向走
```

不堆官方文档配置项——**只贴关键的 10-30 行代码或 SQL**,让读者「看到一条数据从输入到输出的形状」就够了。

## 节奏

```
01-05  心智:看完知道"数据栈每一层在解决什么问题,不会再把数据湖和数据仓库混着说"
06-11  存储:看完知道"为什么开放表格式赢了,Iceberg 凭什么是赢家之一"
12-16  批处理:**离线场景的硬通货**,Spark 调优是数据工程师面试必答题
17-22  流处理:**实时业务的核心**,Flink + Kafka 是新一代数仓的发动机
23-27  编排与建模:**软件工程方法进数据**,dbt + Airflow 决定团队上限
28-32  AI 时代:**当下最热**,RAG 管道 / 特征平台 / 向量库是 AI 工程化的入口
```

**一个硬指标**:看完 06-11 + 23-27 这 11 篇,你应该能在白板前讲清楚:

- "为什么 Parquet 在分析查询能比行存快 10 倍,代价是什么"
- "为什么 Iceberg 在大公司能取代 Hive,小公司直接用 DuckDB 也行"
- "为什么 dbt 让 SQL 终于可以测试 / Code Review / CI,以前为什么不能"
- "为什么 Airflow 写 DAG 还能写出'数据回溯'的 bug,Dagster 怎么解"

再加上 28-32 这 5 篇,你应该能讲清楚:

- "RAG 的数据管道和传统离线特征管道有什么区别,增量更新怎么做"
- "为什么向量库不会替代搜索引擎,二者怎么混合"
- "特征平台为什么必须管离线和在线两边,训练-服务偏差怎么消"

能讲清楚两题这个系列就值了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. **不讲 Hadoop / MapReduce 历史**——MapReduce 时代过去了,不讲 HDFS NameNode 高可用,不讲 YARN 调度细节,不讲 Hive on MR;它们只在"上一代死在哪"作为背景出现,一句话带过
2. 「先讲个大数据起源故事 / Google 三大论文逐一拆解」——直接进重点,论文只在脚注引用
3. 罗列工具命令 / 配置参数大全——只贴让读者"看懂这一层"的最小配置
4. 「这是面试常考」——读者来这里是为了真懂数据怎么流,不是背名词
5. 不画图就讲 Shuffle / Watermark / Checkpoint / 列存——这些东西**不画图永远讲不清楚**
6. 抄官方 Quick Start——官方文档在那儿,这里只补它没讲透的「为什么」和「什么时候不要用」
7. 「Spark vs Flink 谁更牛」式的口水战——只讲各自擅长的场景,选型留给工程判断
8. 不区分 OLTP 和 OLAP 就开始讲——所有讨论的前提都是 OLAP / 分析型负载,事务型场景明确划走给 backendLearning

## 立场声明

本系列**只服务于工程视角**——能落地、能排错、能选型、能给团队建议。

所有示例代码限制在最小可跑片段,**不教你怎么从零搭一套 EMR / 自建 K8s Spark 集群**——那是平台团队的事,本系列的读者画像是「写数据管道的应用工程师 / AI 工程师」,你需要懂这些工具,但不需要从内核源码读起。
