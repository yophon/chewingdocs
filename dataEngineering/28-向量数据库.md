# 向量数据库:pgvector、Milvus、Weaviate、Qdrant、Chroma、LanceDB

2022 年 ChatGPT 之前,**向量数据库**是推荐系统圈的事(Faiss、Annoy 已经用了十年);2023 年 RAG 火了,**这套技术突然变成所有应用工程师都要懂的基础设施**。「向量数据库」不是单一概念,是一个生态:**pgvector 在 PG 上扩展、Milvus 是专用、Qdrant 是 Rust 后起之秀、Chroma 嵌入式、LanceDB 云原生**。这一篇拆向量索引算法(HNSW / IVF / PQ)、6 家选手对比、混合搜索、生产 RAG 怎么选。

> 一句话先记住:**向量库 = 列存(嵌入向量)+ ANN 索引(快速相似度搜索)+ 元数据过滤**。HNSW 是当下主流索引(内存大但召回好),IVF 内存友好但召回差,PQ 压缩省内存但召回更低。**生产首选 pgvector(数据库已经在)/ Qdrant(性能 + 过滤强)/ Milvus(超大规模)**。混合搜索(向量 + BM25)是 RAG 标配。

## 一、向量库在解决什么问题

### 1.1 关键词搜索的局限

```
用户查询:    「我要一个能跑步的运动鞋」
传统搜索:    倒排索引匹配:跑步、运动、鞋
            没匹配的:慢跑、健身、户外
            
向量搜索:    把 query embedding 成向量,搜最相似的商品向量
            "跑步" 跟 "慢跑/有氧/健身" 在语义空间靠近
            → 召回质量提升
```

### 1.2 语义检索的需求场景

```
RAG:           query → 找相关文档 → 喂 LLM 生成
推荐:         用户特征 → 找相似商品/内容
图片搜索:     图像 embedding → 找相似图像
代码搜索:     代码语义搜索
异常检测:     找跟"正常样本"距离远的向量
```

### 1.3 为什么需要专门的库

```
1. 向量长(几百到几千维),内存 / 磁盘要优化
2. 相似度计算贵(余弦 / 欧式距离,几百维一次几十 ns)
3. 精确 KNN 是 O(N),亿级数据搜不动
   → 需要近似算法(ANN)
4. 通常配合元数据过滤(WHERE region='CN' AND user_age > 18)
5. 大规模、可扩展、HA
```

## 二、相似度 + ANN 心智

### 2.1 三种距离

```
余弦相似度(Cosine):     1 - (A·B) / (|A||B|)
                       归一化后,关心方向,不关心模长
                       NLP embedding 最常用
                       
欧氏距离(Euclidean / L2): √Σ(Ai - Bi)²
                       几何意义,关心绝对位置
                       图像 embedding 常用
                       
点积(Dot Product):      Σ Ai × Bi
                       带模长信息
                       一些推荐场景
```

OpenAI / 大部分文本 embedding 模型 → 用 Cosine。

### 2.2 精确 KNN 的痛

```
找最相似的 K 个:
  for each 向量 in N 个数据:
    算距离
  排序选 top K
  
复杂度:O(N × D)
亿级数据 × 1000 维 → 单次查询几秒到分钟
```

### 2.3 ANN(近似最近邻)

牺牲一点召回率,换 O(log N) 或 O(N^0.5) 复杂度:

```
精确 KNN:    100% 召回,慢
ANN:        90%+ 召回,快 100-1000 倍
              
精度 / 速度的可调 trade-off
```

## 三、三大 ANN 算法

### 3.1 HNSW(Hierarchical Navigable Small World)

**当下主流**,几乎所有现代向量库默认。

```
                上层(稀疏)
                  ●
                  │
              ●───┼───●            (远程跳)
                  │
        ●─────────┼─────────●     
        │         │         │
   下层(密集)            
   每个点连周围近邻
   
查询:从顶层 entry point 开始,贪心走向更近的邻居
     逐层下降,每层网络更密
     最底层找到 K 个近邻
```

特点:
- **高召回**(95%+)
- **快查询**(O(log N))
- **内存大**(每个向量带几十个邻居 link,内存膨胀几倍)
- **写慢**(插入新向量要更新邻居)

适合:**内存够 + 召回优先 + 数据量 < 1 亿**。

### 3.2 IVF(Inverted File)

**聚类思想**:

```
1. 离线:k-means 聚类成 K 个 cell(几千个)
2. 每个向量分到最近的 cell
3. 查询:
   - 计算 query 到 K 个 cell 中心的距离
   - 选最近 N 个 cell(nprobe)
   - 在这 N 个 cell 内做精确搜索
```

特点:
- **内存友好**(没 HNSW 的 link 开销)
- **召回低**(cell 边界附近召回差)
- **可与 PQ 组合**(IVF-PQ)

适合:**内存紧、数据量极大、能接受较低召回**。

### 3.3 PQ(Product Quantization)

**向量切段独立量化压缩**:

```
原向量:[0.1, 0.5, 0.3, 0.8, 0.2, 0.7, 0.4, 0.9]   (8 维)

切成 2 段:
  [0.1, 0.5, 0.3, 0.8]  → 量化到 codebook 256 个代表点 → 1 字节
  [0.2, 0.7, 0.4, 0.9]  → 量化到 codebook 256 个代表点 → 1 字节

→ 8 个 float (32 字节) 压缩到 2 字节(16 倍)
```

特点:
- **内存极省**(几十倍压缩)
- **召回低**(有损压缩)
- **常配 IVF**:IVF-PQ 是 Faiss 经典组合

适合:**百亿级数据 + 极致内存约束**。

### 3.4 组合方案

```
小数据 + 高召回:HNSW
大数据 + 内存友好:IVF
极大数据 + 内存极紧:IVF-PQ
新数据 + 不在乎规模:Flat(暴力)
```

## 四、6 家向量库主要选手

### 4.1 pgvector(2021)

**PostgreSQL 扩展**,把向量能力嵌进熟悉的 PG。

```sql
CREATE EXTENSION vector;

CREATE TABLE documents (
    id BIGSERIAL PRIMARY KEY,
    content TEXT,
    embedding VECTOR(1536),
    region TEXT,
    created_at TIMESTAMP
);

-- HNSW 索引(0.5.0+)
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- 查询(SQL 友好)
SELECT id, content, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
WHERE region = 'CN'
  AND created_at > '2025-01-01'
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

**优点**:
- **SQL 友好**(join、过滤、事务都跟普通 PG 一样)
- **数据库已经在**(不用新部署)
- **生态完整**(权限、备份、HA 都用 PG 的)
- 百万到千万级 OK

**缺点**:
- 亿级以上吃力
- 性能不如专用库(HNSW 实现没那么极致)

**适合**:中小项目、已经在用 PG、想要 SQL 体验。**2025 RAG 生产首选**。

### 4.2 Milvus(2019,Zilliz 商业化)

**专用向量库**,主推大规模 + 分布式。

```python
from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

client.create_collection(
    collection_name="documents",
    dimension=1536,
    metric_type="COSINE",
    index_type="HNSW",
    index_params={"M": 16, "efConstruction": 64}
)

client.insert(collection_name="documents", data=[
    {"id": 1, "vector": [0.1, ...], "region": "CN", "text": "..."},
    ...
])

result = client.search(
    collection_name="documents",
    data=[query_vector],
    filter='region == "CN"',
    limit=10,
    output_fields=["text"]
)
```

**优点**:
- 十亿级
- 分布式 + K8s 原生
- 多索引类型(HNSW / IVF_FLAT / IVF_PQ / DiskANN / ...)
- 商业支持(Zilliz Cloud)

**缺点**:
- 部署复杂(很多组件:proxy / query node / data node / etcd / Pulsar / MinIO)
- 学习曲线
- 国内云生态友好

**适合**:十亿级以上、专用向量栈、Zilliz Cloud 用户。

### 4.3 Weaviate(2019)

**模块化**:自带 embedding 模型(可接 OpenAI / Cohere / 本地模型),GraphQL API。

```python
import weaviate

client = weaviate.connect_to_local()

client.collections.create(
    name="Document",
    vectorizer_config=Configure.Vectorizer.text2vec_openai(),
    properties=[Property(name="content", data_type=DataType.TEXT)]
)

docs = client.collections.get("Document")
docs.data.insert({"content": "...", "region": "CN"})

result = docs.query.near_text(
    query="跑步运动",
    limit=10,
    filters=Filter.by_property("region").equal("CN")
)
```

**优点**:
- **自动 embedding**(写文本,Weaviate 调模型生成向量)
- 适合从 0 起新项目
- GraphQL API 灵活

**缺点**:
- 生态不如 Milvus / pgvector
- 性能不是最强

**适合**:RAG 项目从 0 起步 + 不想自己管 embedding。

### 4.4 Qdrant(2021,Rust)

**Rust 写,性能强 + 过滤友好**。

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct, Filter

client = QdrantClient("http://localhost:6333")

client.create_collection(
    "documents",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)

client.upsert(
    "documents",
    points=[
        PointStruct(id=1, vector=[0.1, ...], payload={"region": "CN", "text": "..."}),
        ...
    ]
)

results = client.search(
    "documents",
    query_vector=query_embed,
    query_filter=Filter(
        must=[FieldCondition(key="region", match=MatchValue(value="CN"))]
    ),
    limit=10
)
```

**优点**:
- **性能强**(Rust + 优化)
- **元数据过滤强**(payload 索引,先 filter 再向量)
- 部署简单(单二进制)

**缺点**:
- 生态比 Milvus 小

**适合**:**性能 + 过滤是核心**的 RAG / 推荐场景。**2024+ 增长最快**。

### 4.5 Chroma(2022)

**Python 优先 + 本地嵌入式**。

```python
import chromadb

client = chromadb.PersistentClient(path="./chroma")

collection = client.create_collection("documents")

collection.add(
    documents=["跑步鞋适合户外"],
    metadatas=[{"region": "CN"}],
    ids=["1"]
)

results = collection.query(
    query_texts=["有氧运动装备"],
    n_results=10,
    where={"region": "CN"}
)
```

**优点**:
- **极简**:`pip install chromadb`
- **嵌入式**(数据放本地 / 持久化目录)
- 适合原型 + 小项目

**缺点**:
- 规模有限(几百万级)
- 性能不极致

**适合**:**RAG 原型、本地探索、教学**。LangChain / LlamaIndex 默认 demo 用它。

### 4.6 LanceDB(2023)

**基于 Lance 列存格式**(参考 06 篇),云原生。

```python
import lancedb

db = lancedb.connect("s3://bucket/lancedb")

tbl = db.create_table("documents", data=[
    {"vector": [0.1, ...], "text": "...", "region": "CN"},
    ...
])

results = (tbl
    .search(query_vector)
    .where("region = 'CN'")
    .limit(10)
    .to_pandas())
```

**优点**:
- 数据在 S3,零拷贝
- 跟 Lakehouse 协同(可在 Iceberg-like 存储上做向量搜索)
- 多模态友好(向量 + 图片 + 文本同一列存)

**缺点**:
- 新,生态在涨

**适合**:**AI Lakehouse**、多模态 RAG、S3-first 团队。

## 五、混合搜索:向量 + BM25

### 5.1 为什么混合

```
向量搜索:    语义,但精确字符不强
              查 "iPhone 15 Pro" → 召回 "iPhone 14 Pro Max"(语义近)
              
关键词搜索(BM25 / 倒排):
              精确,但语义不强
              查 "跑步鞋" → 没召回"运动鞋"(关键词不重合)
              
混合:        两者召回 + 排序融合 = 更好的召回质量
```

### 5.2 RRF(Reciprocal Rank Fusion)

```
向量搜索 top 10:    A B C D E F G H I J
BM25 搜索 top 10:    F K L A M N O B P Q

RRF score(每个 doc):
  Σ 1 / (rank_in_each_list + k)
  
合并后排序:
  A: 1/(1+60) + 1/(4+60) = ...
  F: 1/(6+60) + 1/(1+60) = ...
  ...

top 10 = 综合两边好的
```

### 5.3 实现

```sql
-- pgvector + tsvector (PG 全文检索)
WITH vector_results AS (
    SELECT id, RANK() OVER (ORDER BY embedding <=> $1::vector) AS vec_rank
    FROM documents LIMIT 100
),
keyword_results AS (
    SELECT id, RANK() OVER (ORDER BY ts_rank(tsv, query) DESC) AS kw_rank
    FROM documents, plainto_tsquery('chinese', $2) query
    WHERE tsv @@ query LIMIT 100
)
SELECT id, 
    COALESCE(1.0/(vec_rank + 60), 0) + COALESCE(1.0/(kw_rank + 60), 0) AS rrf_score
FROM vector_results FULL OUTER JOIN keyword_results USING (id)
ORDER BY rrf_score DESC LIMIT 10;
```

### 5.4 Reranker

混合召回 50 个 → 用 Cross-encoder 模型重排 top 10:

```python
from sentence_transformers import CrossEncoder
reranker = CrossEncoder("BAAI/bge-reranker-large")

candidates = hybrid_search(query, top_k=50)
scores = reranker.predict([(query, c.text) for c in candidates])
final = sorted(zip(candidates, scores), key=lambda x: -x[1])[:10]
```

Cross-encoder 比向量召回慢但准——**生产 RAG 标配**。

## 六、工程落地:RAG 的最小向量库使用

```python
from openai import OpenAI
import psycopg2

oa = OpenAI()
conn = psycopg2.connect("dbname=mydb")

# 1. 写入(把文档 embedding 后存)
def index_doc(doc_id, text, region):
    emb = oa.embeddings.create(model="text-embedding-3-small", input=text).data[0].embedding
    with conn.cursor() as cur:
        cur.execute(
            "INSERT INTO documents (id, content, embedding, region) VALUES (%s, %s, %s, %s)",
            (doc_id, text, emb, region)
        )
    conn.commit()

# 2. 查询
def retrieve(query, region, k=10):
    q_emb = oa.embeddings.create(model="text-embedding-3-small", input=query).data[0].embedding
    with conn.cursor() as cur:
        cur.execute("""
            SELECT id, content, 1 - (embedding <=> %s::vector) AS sim
            FROM documents
            WHERE region = %s
            ORDER BY embedding <=> %s::vector
            LIMIT %s
        """, (q_emb, region, q_emb, k))
        return cur.fetchall()

# 3. RAG 拼到 LLM
def rag_answer(question, region):
    docs = retrieve(question, region)
    context = "\n\n".join(d[1] for d in docs)
    response = oa.chat.completions.create(
        model="gpt-4",
        messages=[
            {"role": "system", "content": f"基于以下内容回答:{context}"},
            {"role": "user", "content": question}
        ]
    )
    return response.choices[0].message.content
```

## 七、几个常见误区

### 7.1 「向量库取代搜索引擎」

不会。**Elasticsearch / OpenSearch 在精确关键词、全文匹配、聚合统计上仍然强**。**混合搜索**(BM25 + Vector + Rerank)才是 RAG 的正解。

### 7.2 「ES / OpenSearch 也能做向量」

对(8.0+ 都有向量索引),如果已经在用 ES,不需要再加一个向量库——**ES 向量搜索 + BM25 一站式**。但 ES 性能不如专用向量库,大规模场景仍要 Milvus / Qdrant。

### 7.3 「向量库取代关系库」

不会。**事务、JOIN、强一致性,关系库强**。向量库是补充。

### 7.4 「embedding 越大越好」

不一定。OpenAI text-embedding-3-large(3072 维)比 small(1536 维)精度高,但:
- 内存 2 倍
- 查询慢 2 倍
- 大部分场景 small 够用

**生产经验**:**先用 small,RAG 召回质量不够再升 large**。

### 7.5 「向量足够 → 不需要 schema」

错。**向量 + 元数据混合查询**是生产标配:
- 时间过滤(最近 30 天的文档)
- 权限过滤(用户只能看自己的)
- 类别过滤(只查产品文档)

没元数据过滤的向量库,RAG 出错时根本回不去定位。
