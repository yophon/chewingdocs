部署与推理优化:vLLM、TensorRT-LLM、量化、KV Cache 一锅端 ​

LLM 训练靠科研论文,但部署和推理才是工程师的主战场。一个 70B 模型在 A100 上跑得慢且贵,但同一个模型用 vLLM + 连续批处理 + KV Cache 优化,QPS 能翻 10 倍。同样的对话场景,API 调用一年烧 50 万,自部署能压到 8 万。

这一篇讲清楚:推理为什么慢、贵从哪儿来、有哪些优化手段、什么时候自部署、什么时候用 API。

一句话先记住:自部署不是省钱,是换控制权和延迟。真便宜的临界点在 QPS 高到一定程度,大多数项目根本到不了。

一、推理成本来源 ​

要省钱、要提速,先得知道钱和延迟花在哪。

推理成本 = 参数量 × 精度 × token 数 × 调用次数
                ↓
       ┌────────┴────────┐
       │                 │
   显存(GPU)         算力(FLOPs)
因素影响你能动的
模型参数量(7B / 70B / 405B)决定显存和算力门槛选模型
精度(FP16 / INT8 / INT4)显存与速度,对质量有影响量化
Context 长度KV cache 显存,平方级影响延迟截断、压缩、滑窗
Batch size吞吐 vs 延迟的权衡调度策略
KV Cache 命中重复 prefix 能省 60-80%prefix caching
Prefill vs Decode 比例长输入贵 prefill,长输出贵 decode看场景

为什么 LLM 推理和训练性质不同 ​

训练:大 batch、并行度高、瓶颈在算力(FLOPs) 推理(decode 阶段):batch 小、串行、瓶颈在显存带宽

Decode 一个 token 需要:把整个模型权重从显存读一遍,做一次矩阵乘
       ↓
显存带宽:2-3 TB/s(A100)
       ↓
70B 模型 FP16 = 140GB → 单 token 理论极限 20 tokens/s

这就是为什么单条对话的 decode 速度有"硬上限":带宽决定下限,batch 决定上限。

训练时 GPU 利用率 60-90% 是常态,推理时(decode)15-30% 算正常。这不是配置问题,是架构本质决定的。

二、量化:FP16 → INT8 → INT4 → GPTQ → AWQ → FP4 ​

把权重从高精度压到低精度,显存和带宽都减少,速度自然就快。

精度每参数字节70B 显存占用质量损失
FP324280 GB基线
FP16 / BF162140 GB几乎无
INT8170 GB可忽略
INT40.535 GB1-3% 任务下降
FP4(2025+)0.535 GB比 INT4 略好

主流量化方案 ​

方案思路优点缺点
GPTQ分组量化 + 海森矩阵反向修正离线,质量好量化耗时长
AWQ重要权重保留高精度推理快、易部署需要校准集
GGUF(llama.cpp)k-quants 等多种粒度CPU/Apple Silicon 友好GPU 没有那么快
BitsAndBytes在线量化(load_in_4bit)HuggingFace 一行代码速度不如离线方案
FP4 / NF4Nvidia 新一代硬件支持H100/B200 加速明显需新硬件
bash
# 用 llama.cpp 量化一个模型,一行搞定
./llama-quantize ./models/llama-70b-f16.gguf ./models/llama-70b-q4_K_M.gguf q4_K_M

选哪个量化 ​

你的场景推荐
服务器 GPU、追求吞吐AWQ + vLLM
服务器 GPU、追求精度GPTQ 或 FP8
Mac / 边缘 / CPUGGUF(llama.cpp)
拿到模型先跑通看看BitsAndBytes 4bit
有 H100/H200/B200FP8 / FP4,硬件加速

选量化前先看任务对精度的敏感度。代码生成、数学推理对量化更敏感,日常对话 INT4 几乎察觉不到。关键任务必须量化前后跑 eval 对比。

三、KV Cache 与 PagedAttention ​

这是 vLLM 之所以是 vLLM 的根本原因。

KV Cache 是什么 ​

Transformer 自回归生成时,之前 token 的 Key 和 Value 不变,缓存起来不重算。

不用 KV Cache:每生成一个 token,所有历史重算一遍 → O(n²)
用 KV Cache:每个 token 只算自己,历史从缓存读 → O(n)

但 KV Cache 本身很吃显存:

70B 模型,每层 KV cache 大小:
  2(K+V)× hidden_size × seq_len × num_heads
  ≈ 几十 KB / token / layer
70B 有 80 层 → 单 token KV 缓存 ≈ 4MB
context 32k 单条 → 130GB,比模型还大

传统实现的浪费 ​

请求 A 申请 32k context,实际只用了 1k:剩下 31k 显存空着但不能给别人。这就是显存碎片。

PagedAttention(vLLM 的核心创新) ​

把 KV Cache 分页管理(像虚拟内存),按需分配 4KB 的块,多个请求共享显存池,显存利用率从 30-40% 拉到 90%+。

不用 PagedAttention用 PagedAttention(vLLM)
单请求预留全部 context 显存按需分配
显存利用率 30-40%80-95%
并发数受限并发能上几十倍
Prefix caching 难做天然支持(共享 page)

vLLM 不是"快了 10%",是架构上突破了瓶颈。这就是为什么它一出来就成了开源 serving 事实标准。

四、vLLM:吞吐之王 ​

2023 年 UC Berkeley 出来的项目,到 2026 年已经是开源 LLM serving 的默认选项。

bash
# vLLM 起服务,一行命令
pip install vllm

# OpenAI-compatible API 模式
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3-70B-Instruct \
  --tensor-parallel-size 4 \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.92

# 调用方式和 OpenAI SDK 完全一致
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "meta-llama/Llama-3-70B-Instruct",
    "messages": [{"role": "user", "content": "你好"}]
  }'

它把哪些事做对了 ​

  1. PagedAttention:解决显存碎片
  2. Continuous Batching:批次动态拼装,吞吐拉满
  3. Tensor Parallel:跨多卡跑大模型
  4. Prefix Caching:相同前缀(system prompt、few-shot)只 prefill 一次
  5. OpenAI API 兼容:客户端代码不用改
  6. 支持的模型多:几乎所有开源模型一发出来 vLLM 当周就支持

它的局限 ​

  • 主要面向 NVIDIA GPU,AMD / 国产卡支持在追赶
  • 单条请求延迟不一定最低(有 batch 调度开销)
  • 多模态、特殊架构(MoE 早期)支持有时滞

vLLM 调优参数(实战) ​

参数作用推荐
--tensor-parallel-size跨多卡并行= GPU 数量
--gpu-memory-utilization显存使用比例0.9-0.95
--max-model-len最大上下文长度按业务,别贪
--max-num-seqs最大并发序列看显存余量
--enable-prefix-caching开启 prefix 缓存system prompt 长就开
--quantization量化方式(awq / gptq)视模型

一上来 gpu-memory-utilization 别开 1.0,留 5-8% 余量给 cuda 工作内存,否则 OOM 概率大。

五、TensorRT-LLM:英伟达栈的速度之王 ​

NVIDIA 官方推理框架,单条请求延迟和吞吐都能做到极致,代价是编译流程麻烦、模型支持慢。

bash
# TensorRT-LLM 流程比 vLLM 复杂一个数量级
# 1. 转换权重格式
python convert_checkpoint.py --model_dir ./llama-70b \
  --output_dir ./trtllm-ckpt --dtype float16

# 2. 编译为 TRT engine(可能跑半小时)
trtllm-build --checkpoint_dir ./trtllm-ckpt \
  --output_dir ./trtllm-engines \
  --gemm_plugin float16 \
  --max_batch_size 64 \
  --max_input_len 4096 \
  --max_output_len 1024

# 3. 起服务(用 Triton Inference Server)
tritonserver --model-repository=./model_repo

它能比 vLLM 快多少 ​

场景vLLMTensorRT-LLM备注
7B 单卡吞吐基线+20-40%视模型
70B TP=4基线+30-60%TRT 优势更明显
首 token 延迟中更低算子融合做得激进
上线时间1 小时1-2 周编译 + 调试

什么时候选 TensorRT-LLM ​

情况选谁
中小规模、快速上线、模型经常换vLLM
大规模(几十张卡以上)、模型固定TensorRT-LLM
极致延迟敏感(实时语音、金融)TensorRT-LLM
AMD / 国产卡不是 TRT,看对应栈
团队没 CUDA 工程师别上 TRT,运维成本压不住

TensorRT-LLM 是有 CUDA 工程师才玩得起的优化。没人 own 它的话,模型升级时编译失败、算子不支持,排查代价远超 vLLM。

六、Continuous Batching:连续批处理 ​

传统 batching:等凑齐一批,一起跑,跑完一起出。问题:短请求陪长请求一起等。

传统 static batching:
  请求 1(10 token):████ 跑完了等其他
  请求 2(50 token):████████████████████ 还在跑
  请求 3(20 token):████████ 跑完了等其他
                    ↑ GPU 空转 ↑

continuous batching:
  请求 1 跑完 → 立刻插入请求 4
  请求 3 跑完 → 立刻插入请求 5
  GPU 始终有活干

vLLM、TensorRT-LLM 都默认开了。这是 LLM serving 比传统 ML 服务"难"的核心点之一:输入输出长度差异巨大,静态批处理浪费严重。

调度策略吞吐延迟稳定性复杂度
Static batching低高(可预测)低
Dynamic batching中中中
Continuous batching高略波动高
优先级调度 + Continuous极高可控极高

你不用自己实现 continuous batching,选 vLLM / TGI / TensorRT-LLM 就自动有了。记住一点:不要自己写 LLM serving,这块水太深。

七、流式输出:SSE、token-by-token ​

用户体验上,流式比一次性返回慢的"显得快"几倍。原因:首 token 延迟感受拉低。

python
# 服务端流式输出(FastAPI + SSE)
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat")
async def chat(req: dict):
    async def generate():
        async for chunk in llm.stream(req["messages"]):
            yield f"data: {chunk}\n\n"
        yield "data: [DONE]\n\n"
    return StreamingResponse(generate(), media_type="text/event-stream")
javascript
// 客户端消费 SSE
const res = await fetch("/chat", { method: "POST", body: JSON.stringify(msg) });
const reader = res.body.getReader();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  console.log(new TextDecoder().decode(value));
}

工程注意点 ​

  1. 首 token 延迟(TTFT)是关键指标:用户看到第一个字的时间,应该 < 1s
  2. 流式中间不要做大量后处理:任何 buffering 都会拖延用户感受
  3. 错误也要流式返回:别让连接挂着,客户端不知道
  4. SSE 比 WebSocket 简单,大多数场景够用;真要双向通信再考虑 WS
  5. 代理层小心 buffering:Nginx 默认会缓冲,要 proxy_buffering off 才透传流式

流式 vs 非流式 ​

场景选哪种
对话、聊天、长输出必须流式
短抽取、分类、JSON 结构化非流式更简单
工具调用,需要完整 JSON 解析非流式或等 stream 完整再 parse
长文档总结流式,但前端要拼接段落

十、推理时计算:o1 / Claude thinking 的成本特点 ​

2024 年下半年起,推理时计算(test-time compute) 成为新趋势。OpenAI o 系列、Claude 的 extended thinking、DeepSeek R 系列,都在"花更多 token 思考"。

传统模型:输入 → 模型 → 输出
            ↓
            1 倍 token 成本

thinking 模型:输入 → 模型(内部生成大量思考 token)→ 输出
                ↓
                5-50 倍 token 成本(取决于难度)

它带来的成本特点 ​

维度传统模型Thinking 模型
输出 token / 请求几百几千到几万
延迟秒级几十秒到分钟
单次成本一倍5-50 倍
适合任务对话、抽取、翻译数学、代码、复杂推理

用 thinking 的工程取舍 ​

  1. 不要默认开 thinking。多数任务不需要,白烧 token
  2. 设置 thinking budget:Anthropic API 可以指定最大 thinking token,控制成本上限
  3. 缓存 thinking 结果:同样输入的 thinking 可以复用,前提是你有缓存层
  4. 流式回 thinking 让用户安心:thinking 几十秒一片黑屏是糟糕体验,显示 "正在思考……" 或部分思考内容
  5. 路由策略:简单问题走快模型,复杂问题再走 thinking 模型
python
# Claude extended thinking 的用法
response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=8192,
    thinking={
        "type": "enabled",
        "budget_tokens": 4096,  # 控制 thinking 预算
    },
    messages=[{"role": "user", "content": "证明费马小定理"}],
)

Thinking 模型让"贵"和"慢"产生了新维度。算成本时不要只算入参 token,也要算 thinking 消耗。

十一、踩坑:显存爆、首 token 延迟、批量大小调优 ​

踩坑 1:显存计算靠脑补 ​

70B 模型 FP16 = 140GB,以为 H100(80GB)能塞下
实际:权重 + 优化器状态(训练)/ KV cache(推理)+ 激活
推理:140GB + KV cache(几十 GB)→ 单卡 H100 装不下,要 TP=2

算公式记住:

推理显存 ≈ 权重 × 量化系数 + KV_cache_per_token × max_seq_len × max_batch
权重 = 参数量 × 2(FP16)/ 0.5(INT4)
KV cache per token ≈ 4 × hidden_size × num_layers × 2 字节(FP16)

踩坑 2:首 token 延迟比想象慢 ​

prefill(处理输入)是计算密集,长输入(几千 token)单条 prefill 就要 1-2 秒。

优化效果
Prefix caching重复 system prompt 命中可省 80%+
Chunked prefill长 prefill 切块,不阻塞短请求
Speculative decoding小模型猜 → 大模型验,加速 1.5-3x
减少 system prompt 长度直接缩水,最朴素也最有效

踩坑 3:max_batch_size 调瞎了 ​

太小:GPU 利用率低,吞吐拉不满 太大:显存爆 OOM,或 batch 内首 token 延迟飙升

调优思路:

1. 从 max_batch_size = 8 开始
2. 压测看 P50/P95 延迟和吞吐
3. 慢慢加,直到 P95 延迟超阈值或 OOM
4. 退一档作为生产配置

踩坑 4:Tensor Parallel 配错卡间互联 ​

70B 模型 TP=4 跑 4 张 A100,如果是 PCIe 互联(没 NVLink),通信开销可能让"4 卡 ≈ 1.5 卡"。多卡 inference 必须 NVLink 或 NVSwitch。

踩坑 5:量化后没跑 eval 就上 ​

INT4 量化在通用对话上几乎察觉不到,但在数学推理、多步代码生成、长上下文任务上,质量退化可能高达 5-10%。

铁律:量化前后必须用同一个 eval set 对比,数字差超 3% 就要重新评估量化方案。

踩坑 6:OpenAI 兼容 API ≠ 行为兼容 ​

vLLM / Ollama 等都提供 OpenAI 兼容 API,但特殊参数(logit_bias、response_format、function calling 细节)经常不一致。

迁移时要盯紧:

  • tool / function calling 格式
  • streaming 的 chunk 边界
  • token 计数(同模型不同 tokenizer 算出来不一样)
  • 温度、top_p 默认值

踩坑 7:模型加载慢拖死部署 ​

70B 模型加载到显存要 2-5 分钟,k8s 滚动更新可能整体不可用 10+ 分钟。

对策:

  • 模型权重放本地 SSD,别每次拉对象存储
  • 用 readiness probe 区分"已启动"和"已就绪"
  • 多实例滚动(blue-green 或金丝雀),避免全部一起重启
  • 预拉模型镜像,启动再拉就晚了

踩坑与建议 ​

  1. 先想清楚要不要自部署。算清楚月 token 量,API 在 < 500M token / 月这个量级几乎都更划算。
  2. 量化必跑 eval。INT4 在对话场景几乎无感,但代码 / 数学 / 长上下文可能掉 5-10%。
  3. vLLM 是 serving 默认选项。除非你有 CUDA 团队 + 模型固定不换,否则别上 TensorRT-LLM。
  4. Mac 用户用 MLX。比 llama.cpp 还快 20-40%,M 系列 Mac 推理体验最好的栈。
  5. 流式输出几乎是必须。首 token 延迟感受比总延迟重要 10 倍。
  6. GPU 显存利用率别开满。0.9-0.95 留余量,顶到 1.0 必然 OOM。
  7. Prefix caching 一定要开。system prompt 长 + 高并发场景能省 60-80% prefill 成本。
  8. 多卡互联走 NVLink。PCIe 互联跑 TP,4 卡可能只有 1.5 卡的效果。
  9. Thinking 模型不要默认开。设 budget,只在复杂任务上启用,简单任务用快模型。
  10. OpenAI 兼容 ≠ 行为兼容。tool calling、stream 格式、token 计数,迁移前必测。
  11. 模型加载慢要预热。k8s 滚动更新前考虑模型本地化、readiness probe、多实例。
  12. 不要自己写 LLM serving。这块水太深(continuous batching、paged attention、prefix sharing),用 vLLM / TGI / Ollama 就行。
