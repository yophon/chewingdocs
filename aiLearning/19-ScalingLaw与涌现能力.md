# Scaling Law 与涌现能力:大力为什么真的能出奇迹

2020 年前的深度学习,有句话叫"模型够用就行,堆大没用"。Scaling Law 出来之后,这句话被彻底打脸:**模型越大、数据越多、算力越多,loss 真的会按一条数学曲线下降**,而且能下降到能"涌现"出新能力。这一篇讲清楚这条曲线和它带来的工程影响。

> 一句话先记住:**Scaling Law 告诉你"投多少能换多少进步",涌现能力告诉你"投到某个点会突然解锁新技能"。**

---

## 一、Scaling Law:loss 是 N、D、C 的幂律

OpenAI 2020 年的论文《Scaling Laws for Neural Language Models》,核心结论:

```
模型 loss L 是三个量的幂律函数:
  N: 模型参数量
  D: 训练数据 token 数
  C: 计算量(FLOPs)

L(N) ≈ (N_c / N) ^ α_N
L(D) ≈ (D_c / D) ^ α_D
L(C) ≈ (C_c / C) ^ α_C

其中 α 是经验拟合的小数(都在 0.05~0.1 之间)
```

直觉(不要怕公式):

```
loss ↓
  │
  │ \
  │  \
  │   \─────
  │         \─────────_____
  └─────────────────────────→  log(参数量) 或 log(数据) 或 log(算力)
```

**对数尺度下,loss 几乎是直线下降**。这意味着:

- 参数量翻 10 倍,loss 下降一个固定的 delta
- 这种关系在 7 个数量级上都成立(从百万参数到千亿参数)

### 几个反直觉的结论

| 观察 | 含义 |
| --- | --- |
| 不是每加一层都有用 | **结构在某个范围内不重要**,scale 才是 |
| 大模型更"sample efficient" | 同样的训练数据,大模型学得多 |
| loss 没饱和的迹象 | 在我们能观测的范围内,继续 scale 就继续下降 |

> 工程含义:**别在小模型上"调结构"调到地老天荒。把同样的精力花在做大、做更多数据、做更长训练上,产出更确定**。

---

## 二、Chinchilla:参数 vs 数据的最优配比(20 token/参数)

OpenAI 的 scaling law 暗示:**给定算力,先把模型做大,数据少点没关系**。GPT-3(175B 参数,300B token)就是这套思路的产物。

DeepMind 2022 年的 Chinchilla 论文重新算了一遍,得出**完全相反**的结论:

```
给定 FLOPs,最优配比是:
  N (参数量) 和 D (数据 token) 应该等比例增长

经验法则:D ≈ 20 × N
```

也就是说,7B 模型最好用 140B token 训练,70B 模型最好用 1.4T token 训练。

### Chinchilla 撕掉的"皇帝新衣"

| 模型 | 参数 | 训练 token | token/param | 是否"训过" |
| --- | --- | --- | --- | --- |
| GPT-3 (2020) | 175B | 300B | 1.7 | **严重欠训** |
| Gopher (2021) | 280B | 300B | 1.1 | **严重欠训** |
| Chinchilla (2022) | 70B | 1.4T | 20 | 训得正合适 |
| LLaMA-1 (2023) | 65B | 1.4T | 22 | 接近最优 |
| LLaMA-3 (2024) | 70B | 15T | 214 | **过训**(故意的) |

> Chinchilla 70B 在很多 benchmark 上吊打 Gopher 280B,**直接证明"参数量不是越大越好"**,数据匹配上才是关键。整个 LLM 圈子从此反过来:**先准备数据,再决定模型大小**。

---

## 三、训练最优 vs 推理最优(为什么现在的小模型也很强)

Chinchilla 的"最优"是**训练阶段算力最优**。但工程上你部署模型,推理算力远比训练算力贵——一个模型训一次,但推理几亿次。

```
训练算力:  C_train ∝ 6 × N × D
推理算力:  C_infer ∝ 2 × N × tokens_per_request × num_requests
```

如果一个模型要服务 10 亿次请求,推理总算力可能是训练的 100 倍。

**结论**:从"产品总成本"看,**故意训得"过度"小模型(用远超 Chinchilla 比例的数据)更划算**。

```
70B Chinchilla 最优:  D = 1.4T,推理慢
8B 过度训练:          D = 15T,效果接近 70B,推理快 9 倍
```

LLaMA-3-8B 用 15T token 训(token/param = 1875,远超 Chinchilla 的 20),效果直接逼近 LLaMA-1-65B。这就是 **"小但训得久"路线**。

| 路线 | 代表 | 哲学 |
| --- | --- | --- |
| 训练最优 | Chinchilla 70B | 算力有限,先把模型训好 |
| 推理最优 | Llama-3-8B、Qwen2.5-7B | 模型小、训得猛,部署便宜 |
| 混合(MoE) | DeepSeek-V3、Mixtral | 参数量大但激活少,见第五节 |

> 这就是为什么 2024 之后"7B 小模型"突然变得很强——**不是模型架构变了,是训练 token 数从 1T 变到 15T+**。同样的小模型,多喂三倍数据再来一遍,体验天差地别。

---

## 四、涌现能力:小模型不会、大模型突变

Scaling Law 说 loss 平滑下降。但人们发现一个奇怪现象:**很多任务不是平滑提升,而是"突然学会"**。

```
准确率
 1.0 │              ┌────
     │              │
     │              │
     │              │
 0.0 │ ─────────────┘
     └─────────────────────→  模型规模
                  ↑
              "涌现点"
```

这就是 Google 2022 年论文《Emergent Abilities of Large Language Models》提出的**涌现能力(Emergent Abilities)**。

### 哪些能力是"涌现"的

| 能力 | 涌现于 |
| --- | --- |
| 多步算术 | ~10B+ |
| 逻辑推理(简单) | ~30B+ |
| In-context learning(few-shot 学新任务) | ~10B+ |
| Chain-of-Thought 起作用 | ~60B+ |
| 复杂指令遵循 | ~30B+ |
| 工具调用决策 | ~70B+(早期模型) |
| 多步规划 | ~100B+(还在改善) |

### 涌现的"科学解释"之争

**一派**:涌现是真的,模型到了某个 scale 真的解锁了新能力。
**另一派**(Anthropic 2023 论文):涌现是评测指标的伪影。**用"准确率"这种 0/1 指标看着像突变,用 log-likelihood 这种连续指标看就是平滑提升**。

```
任务:把 5 个数字相加

小模型:答 "12345 + 6789 = 19133"     # 错
中模型:答 "12345 + 6789 = 19134"     # 错(差 1)
大模型:答 "12345 + 6789 = 19135"     # 对

如果指标是"完全正确":小=0,中=0,大=1   → 看起来是突变
如果指标是"对了几位":  小=2,中=4,大=5  → 看起来是平滑
```

但即便是平滑提升,**对应用层而言"差一位 = 错"还是"差一位 = 部分对"是有本质区别的**。所以涌现作为"工程现象"仍然成立。

> 工程含义:**有些任务在小模型上"努力也学不会",换大模型可能直接解决**。**别死磕优化小模型上的任务**,有时候换大模型是最便宜的方案。

---

## 五、Mixture of Experts (MoE):稀疏激活的 scaling 路径

直接堆大模型的瓶颈是**推理太贵**:每个 token 都要跑完整个模型。MoE 是个聪明的妥协:

```
传统 dense 模型:
  每个 token 走完整个模型(N 个参数全用上)

MoE 模型:
  每层有多个"专家"(experts),每个 token 只走 top-2 个
  总参数 N 大,但每次只用 ~2N/k(k 是专家数)
```

### 结构

```
输入 token x
    │
    ├─→ Router(一个小网络)给每个专家打分
    │
    ├─→ 选 top-2 专家
    │
    ├─→ Expert 1 (FFN)  →┐
    │                     ├─→ 加权求和 →  输出
    └─→ Expert 5 (FFN)  →┘
```

只激活部分参数。

### 数字感受

| 模型 | 总参数 | 激活参数 | 推理算力 |
| --- | --- | --- | --- |
| Llama-3-70B (dense) | 70B | 70B | 100% |
| Mixtral 8x7B | 47B | 13B | 18% |
| DeepSeek-V3 | 671B | 37B | 5.5% |

DeepSeek-V3 是个典型例子:**参数量看起来吓人,但每次推理只激活 37B**,推理速度和 30B 级别 dense 模型差不多,效果接近顶级闭源模型。

### 优缺点

| 维度 | dense | MoE |
| --- | --- | --- |
| 训练复杂度 | 简单 | 复杂(router 训练、负载均衡) |
| 推理速度 | 与参数成正比 | 远低于总参数 |
| 显存 | 与参数成正比 | **要全部加载**,激活只是计算上的事 |
| 知识容量 | 受参数限制 | 高(总参数大) |
| 部署难度 | 普通 | 难(显存大、router 调度) |

> MoE 不是"小模型的胜利",**它是"大参数 + 小激活"的折中**。你部署 DeepSeek-V3 还是要存 671B 的权重,只是算得快一点。

## 六、Inference-time compute:o1/o3/Claude thinking,推理时再 scale

2024-2026 最重要的 scaling 新方向:**不光训练时 scale,推理时也 scale**。

### 思路

传统 LLM 是"看到题目立即回答":

```
Question  →  [一次 forward pass]  →  Answer
```

新一代"推理模型"(o1、o3、Claude Opus 4 thinking)在回答前先"想":

```
Question
    ↓
[内部 think tokens,几千~几万个]   ← 这部分用户看不到,但消耗算力
    ↓
最终回答
```

模型在内部生成大量推理过程(self-verify、回溯、尝试不同路径),最后给出答案。**给的"思考时间"越多,答案越好**。

### Scaling Law 又来了一次

OpenAI 2024 年发现:**推理时计算量(thinking token 数)和回答正确率,也满足幂律**。

```
正确率 ↑
       │           ┌──────
       │       ┌───┘
       │   ┌───┘
       │   │
       └───┴────────────→  log(thinking tokens)
```

这意味着对一个固定模型,**多花算力推理可以换更高的智能**。比起再训一个更大的模型,推理时 scale 是个新维度。

### 工程层面的影响

| 模式 | 延迟 | 成本 | 适用 |
| --- | --- | --- | --- |
| 普通对话 | <1s | $ | 闲聊、抽取、简单问答 |
| 推理模式(low) | 几秒 | $$ | 中等复杂任务 |
| 推理模式(high) | 几十秒 | $$$ | 数学竞赛、代码 debug、复杂规划 |

API 层面:

```python
import anthropic
client = anthropic.Anthropic()

resp = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=8096,
    thinking={"type": "enabled", "budget_tokens": 5000},   # 给 5000 token 思考预算
    messages=[{"role": "user", "content": "帮我解这道数学题:..."}]
)
# resp.content 里会同时有 thinking 块和最终回答
```

```python
from openai import OpenAI
client = OpenAI()

resp = client.chat.completions.create(
    model="o3",
    reasoning_effort="high",       # low / medium / high
    messages=[...]
)
```

> Inference-time compute 是 2025-2026 LLM 进步的主线。**单纯堆参数遇到瓶颈了,但让模型"思考更久"还在持续涨分**。

## 七、对应用层的启示(不要早早微调小模型、模型选型策略)

### 7.1 不要早早微调小模型

Scaling 视角下的常见误区:

```
错误路径:
  发现 7B 模型某个任务做不好 → 微调它 → 还是不好 → 再大数据微调 → 还是不好

正确路径:
  发现 7B 模型某个任务做不好 → 试试 70B 模型 → 行不行
                                  ├── 行 → 评估"用大模型 vs 微调小模型"成本
                                  └── 不行 → 这是模型能力极限,微调救不了
```

关键问题:**任务在大模型上是不是涌现的?**

- 是涌现的:小模型再调也是平。直接上大模型。
- 不是涌现的:小模型 + 微调可能划算。

### 7.2 模型选型决策表

| 任务复杂度 | 数量级 | 推荐 |
| --- | --- | --- |
| 简单分类、抽取 | 高 QPS | 7B 微调 / Haiku 等小模型 |
| 通用对话、问答 | 中 QPS | Sonnet / GPT mini 中档 |
| 复杂推理、代码、数学 | 低 QPS,高质量 | Opus / o3 / 推理模型 |
| 极简任务(语言识别等) | 极高 QPS | 1B 量级专门小模型 |

### 7.3 推理时计算的工程模式

```python
# 模式一:用便宜模型筛,贵模型精修
def hybrid_pipeline(query):
    is_complex = haiku_classify(query)
    if is_complex:
        return opus_with_thinking(query)
    else:
        return haiku_answer(query)

# 模式二:self-consistency,采样多次取一致答案
def self_consistency(query, n=5):
    answers = [model.generate(query, temperature=0.7) for _ in range(n)]
    return most_common(answers)

# 模式三:推理预算可调
def adaptive_thinking(query, max_budget=10000):
    # 先低预算试,不行再加
    for budget in [1000, 3000, 10000]:
        resp = model.generate(query, thinking_budget=budget)
        if confident(resp):
            return resp
    return resp
```

### 7.4 别迷信"最强模型"

**不同任务最划算的模型不同**:

```
做客服 FAQ:  Haiku 4 + RAG 即可,Opus 浪费
做代码 review:  Sonnet 4.5 性价比最高
做数学竞赛:  上 o3 high 或 Opus thinking
做高 QPS 抽取:  微调 7B 比调用 API 划算
```

> 模型选型本质是 **质量/成本/延迟** 三角的权衡。Scaling Law 帮你判断"再大有没有用",但具体选谁,要算总账。

## 八、未来还能 scale 多远(数据墙、算力、合成数据)

三个限制因素:

### 8.1 数据墙

互联网上高质量人写文本估计在 50T~100T token,**已经被现有模型基本啃完**。

| 来源 | 估计 token 量 |
| --- | --- |
| 开放网页(Common Crawl 过滤后) | ~30T |
| GitHub 代码 | ~5T |
| 学术(arxiv、PubMed 等) | ~2T |
| 书籍 | ~10T |
| 视频转录(YouTube 等) | ~50T |
| **合计高质量** | ~50~100T |

**数据增长率比模型规模需求增长慢得多**。再往上 scale,纯人写数据要不够了。

### 8.2 合成数据

解决方案之一:**用强模型造数据训弱模型**。

```
GPT-4 生成 / 改写 / 标注 → 训练 LLaMA / Qwen / DeepSeek
```

效果比想象中好,Phi 系列、Qwen 系列、DeepSeek-V3 大量用合成数据。但有风险:

- **模型崩塌**:用模型自己的输出循环训自己,多代后会越来越糊
- **多样性下降**:合成数据风格趋同
- **错误累积**:生成数据里的小错误被放大

### 8.3 算力墙

```
GPT-4:    ~2.5 × 10^25 FLOPs
GPT-5:    估计 10^26 量级
GPT-6:    需要 10^27 量级 → 单次训练成本 >$10B
```

地球上的电力、芯片产能、数据中心建设速度都跟不上无限 scale。

### 8.4 几条可能的出路

| 方向 | 思路 |
| --- | --- |
| 推理时 scale | 见第六节,模型不变,推理多算 |
| 多模态数据 | 视频/图片有大量信息,token 化后可用 |
| Agentic 数据 | LLM 在交互/工具调用中产生新数据,再回炉 |
| 算法效率 | 同样数据榨出更多信息(Mamba、Mixture-of-Depths 等) |
| 物理世界数据 | 机器人、自动驾驶产生的高维信号 |

> 2026 年现在的状况:**纯文本预训练 scale 触到瓶颈,但推理时 scale + 合成数据 + 多模态正在接力**。"AI 进步停滞"的论调,基本来自只看预训练这一条路的人。

## 给新手的建议

1. **理解 scaling law 的形状,不必背公式**。"loss 是参数/数据/算力的幂律,在 log 尺度下接近线性"——记住这一句就够你做大部分工程判断。
2. **遇到效果上限,先想 scale,再想结构**。一个任务在小模型上做不好,优先试更大模型而不是优化结构/调参/微调。**业界几乎所有"突破"都是 scale 带来的**。
3. **20 token/参数是个粗略锚点**。看到一个开源模型,看 token/param 比例,你大致能判断它"是不是训够了"。Llama-3 / Qwen2.5 都是远超 20 的"过训"模型,意味着你能拿来用,效果不差。
4. **涌现意识到了就行,别"等涌现"**。新任务先在最小可行模型上跑,跑不动直接试大模型。**别花两周在 7B 上调参,跑去 70B 上 5 分钟就解决了**。
5. **小心微调"治表不治本"**。如果任务考验的是涌现能力,微调小模型治不了。微调有效的领域:风格、格式、特定领域语料适应。
6. **关注"推理时 scale"这条线**。会用 thinking budget、会做 self-consistency、会做 hybrid pipeline,这些技巧带来的提升比换模型还显著。
7. **不要预测"AI 上限"**。每隔半年都有人说"scale 见顶了",每次都被新方法打脸。**作为应用层工程师,假设 18 个月后有个比现在强 10 倍的模型,你的产品架构能不能用上?能就赢,不能就要重做**。
8. **节制使用大模型**。Opus / o3 / GPT-5 不是"什么都用上去最好",是"难任务上去"。简单任务用小模型,把贵模型留给非它不可的场景。
