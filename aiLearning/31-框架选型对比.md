# 框架选型对比:LangChain、LlamaIndex、Claude Agent SDK……到底用哪个

到了 2026 年,AI 应用层的"框架"已经多到劝退新人。LangChain、LlamaIndex、LangGraph、Claude Agent SDK、AutoGen、CrewAI、DSPy……每一个都说自己"开箱即用",但你真上线时会发现:**绝大多数项目,框架带来的麻烦比解决的问题多**。

这一篇把市面上主流框架掰开揉碎对比一遍,告诉你**什么时候该上、什么时候不该上、上哪一个**。

> 一句话先记住:框架的本质是"prompt + HTTP 请求 + 一点点状态管理"。理解了原理,框架只是一种风味,不是必需品。

---

## 一、AI 应用层"框架"到底是什么

剥开包装,这些框架做的事其实就三件:

```
1. 把 LLM API 调用包一层(retry、超时、流式、多模型切换)
2. 把常见模式做成现成组件(RAG、Agent、Tool Use、Memory)
3. 把流程编排起来(链式调用、状态机、多 Agent 协作)
```

它们不是 PyTorch 那种"没了它就没法干活"的基础设施,而更像 Web 开发里的 Express、Spring Boot —— **能加速,也会锁定**。

| 维度 | 框架做了什么 | 你自己写需要多久 |
| --- | --- | --- |
| API 调用封装 | retry / 超时 / 流式 | 半天 |
| Prompt 模板 | f-string + 变量替换 | 一小时 |
| RAG 流水线 | embedding + 检索 + 拼 prompt | 一天 |
| Agent 循环 | tool call + while 循环 + 状态 | 两天 |
| 多 Agent 协作 | 消息总线 + 角色编排 | 一周 |
| Eval / Trace | 复杂,自己造轮子不划算 | 两周以上 |

> 看清楚:**前四件事自己写都很快**。真正能让框架"赢"的是后两件 —— Eval 与 Trace。这也解释了为什么 LangSmith、Langfuse 这种"框架配套"才是真生意。

---

## 二、LangChain 全家桶

应用层最早火的框架,2022 年起家,到 2026 年生态最庞大:`langchain`、`langchain-core`、`langchain-community`、`langgraph`、`langsmith`、几百个 integrations。

```python
# LangChain 的"全家桶"写法,看着优雅,坑也最深
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个翻译助手"),
    ("human", "把这句翻成英文:{text}"),
])
chain = prompt | ChatAnthropic(model="claude-opus-4-5") | StrOutputParser()
result = chain.invoke({"text": "今天天气真好"})
```

### 优点

- 生态最广:任何模型、任何向量库、任何工具,基本都有现成 integration
- LCEL(LangChain Expression Language)用 `|` 把组件串起来,串行流式很顺
- LangSmith 的 trace / eval 工具是真好用,几乎是行业事实标准

### 缺点

1. **抽象层堆得太高**:`Runnable`、`RunnableLambda`、`RunnableParallel`、`RunnablePassthrough`,你写两个月都未必背得全
2. **版本兼容性灾难**:从 `langchain` 0.0.x → 0.1 → 0.2 → 0.3,每次大版本砍一刀,老代码经常跑不动
3. **看似简单实则深坑**:一个 `chain.invoke` 失败,traceback 能给你打 80 行,定位 bug 心累
4. **官方文档"全而不深"**:每个组件都列了出来,但没人告诉你"这个组件什么时候不该用"

### 适用场景

| 你是…… | 用 LangChain 合适吗 |
| --- | --- |
| 在做 demo / 黑客松 / 内部工具 | 合适,堆速度快 |
| 在做严肃的生产系统 | **慎重**,自己写薄一层更稳 |
| 在搭 RAG 流水线 | 不如 LlamaIndex 直接 |
| 在做复杂工作流 | 直接上 LangGraph,别用老的 `Chain` API |

> 圈内有个说法:"LangChain 是用来快速搭 demo 给老板看的,demo 通过了就重写"。这话糙但理不糙。

## 三、LlamaIndex

最初叫 GPT Index,做 RAG 起家。它和 LangChain 同期,但**专注度更高** —— 文档/数据接入、索引构建、检索策略,这块比 LangChain 干净。

```python
# LlamaIndex 的 RAG 写法,几行就跑通
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

docs = SimpleDirectoryReader("./data").load_data()
index = VectorStoreIndex.from_documents(docs)
query_engine = index.as_query_engine()
print(query_engine.query("公司的报销政策是什么?"))
```

### 它擅长什么

- **数据加载器**:200+ 种数据源,PDF、Notion、Confluence、Jira、Slack 都有现成的
- **索引策略**:VectorIndex、TreeIndex、KnowledgeGraphIndex,语义检索之外还有更花的玩法
- **路由与子查询**:Router Query Engine、Sub Question Query Engine,复杂检索场景比 LangChain 顺手

### 缺点

1. Agent 这块比 LangGraph / Claude Agent SDK 弱
2. 抽象也不少,`Retriever`、`NodePostProcessor`、`ResponseSynthesizer` 都得学
3. 国内文档资料比 LangChain 少

### LangChain vs LlamaIndex

| 维度 | LangChain | LlamaIndex |
| --- | --- | --- |
| 定位 | 通用 LLM 应用框架 | 专做数据接入与检索 |
| RAG 组件成熟度 | 够用 | **更强** |
| Agent 能力 | LangGraph 加持后强 | 一般 |
| 学习曲线 | 陡且杂 | 陡但聚焦 |
| 适合场景 | 端到端应用 | RAG 系统、知识库、企业搜索 |

> 选型建议:**RAG 为主用 LlamaIndex,Agent 为主考虑 LangGraph 或 Claude Agent SDK**,不要既要又要。

## 四、Claude Agent SDK

Anthropic 官方在 2024 年下半年推出、2025 年走向成熟的 Agent SDK。它最大的卖点不是"功能多",而是**和模型本身深度配合**:Claude 的 tool use、thinking、computer use、memory tool 这些原生能力,SDK 用起来最丝滑。

```python
# Claude Agent SDK,官方思路:少封装、贴 API
from anthropic import Anthropic
from anthropic.lib.tools import tool

client = Anthropic()

@tool
def get_weather(city: str) -> str:
    """查询某城市天气"""
    return f"{city} 今天 22°C 晴"

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    tools=[get_weather],
    messages=[{"role": "user", "content": "上海今天天气怎么样?"}],
)
```

### 优点

- **官方维护、跟模型同步**:Claude 一发新能力,SDK 就支持
- **抽象薄**:不像 LangChain 那样层层包,你能看到底层 API
- **稳定**:不会今天一个版本明天一个版本

### 缺点

- 只支持 Claude(虽然 Anthropic 认为这是优点)
- 多 Agent 协作、复杂工作流没有现成方案,要自己拼
- 国内访问 Claude 略麻烦,部分团队不便采用

### 什么时候上

| 场景 | 推荐度 |
| --- | --- |
| 已经决定用 Claude 系列 | 强烈推荐 |
| 想用 Anthropic 最新能力(thinking、computer use) | 必上 |
| 项目需要多模型切换 | 不合适,选 LiteLLM + 自写 |
| 团队不熟 LLM,想要"教科书式"的简洁实现 | 推荐,代码可读性最好 |

> 我的偏好:**生产环境如果固定用 Claude,直接 SDK,不要再套一层 LangChain**。多套一层就多一处可能崩的地方。

## 五、AutoGen / CrewAI / MetaGPT:多 Agent 流派

这三家都在做"多个 Agent 协作完成任务"。但走的路线不太一样。

| 框架 | 来源 | 核心思路 | 风格 |
| --- | --- | --- | --- |
| AutoGen | 微软 Research | 对话式多 Agent,Agent 之间发消息 | 学术派、灵活 |
| CrewAI | 独立项目 | 角色 + 任务 + 流程,模仿"团队协作" | 工程派、上手快 |
| MetaGPT | 国内开源 | "软件公司"隐喻:产品、架构、开发分工 | 演示派、有想象力 |

```python
# CrewAI 的 hello world,角色 + 任务很直观
from crewai import Agent, Task, Crew

researcher = Agent(role="研究员", goal="收集 AI 框架资料", backstory="...")
writer = Agent(role="撰稿人", goal="把资料写成报告", backstory="...")

t1 = Task(description="收集 2026 年主流 Agent 框架", agent=researcher)
t2 = Task(description="基于资料写一篇对比报告", agent=writer)

crew = Crew(agents=[researcher, writer], tasks=[t1, t2])
result = crew.kickoff()
```

### 多 Agent 真的有用吗

**真实情况**:多 Agent 在 demo 里非常炫,在生产里**九成是过度设计**。

原因有三:

1. 多 Agent = 多次 LLM 调用,token 成本翻几倍,延迟也翻几倍
2. Agent 之间互相"商量"经常陷入死循环或互相奉承,质量未必提高
3. 一个 Agent + 好的 prompt + tool use,在 80% 场景已经够了

> 不要早早上多 Agent。先把单 Agent 做扎实,真遇到"任务复杂到一个 Agent 装不下"再考虑拆。

### 三家选哪个

| 你想…… | 选 |
| --- | --- |
| 看一份高度成熟的多 Agent 学术参考实现 | AutoGen |
| 快速搭出一个"角色协作"的产品 demo | CrewAI |
| 看"AI 公司"演示、找灵感、不打算上线 | MetaGPT |
| 实际生产 | **重新评估,多半不需要多 Agent** |

## 六、LangGraph:工作流编排

LangChain 团队的"二代产品",承认了第一代 `Chain` 设计的问题,改用**显式状态机**。

```python
# LangGraph 的核心:节点 + 边 + 状态
from langgraph.graph import StateGraph, END
from typing import TypedDict

class State(TypedDict):
    query: str
    docs: list
    answer: str

def retrieve(state):
    return {"docs": vector_search(state["query"])}

def generate(state):
    prompt = f"基于:{state['docs']}\n回答:{state['query']}"
    return {"answer": llm.invoke(prompt)}

g = StateGraph(State)
g.add_node("retrieve", retrieve)
g.add_node("generate", generate)
g.add_edge("retrieve", "generate")
g.add_edge("generate", END)
g.set_entry_point("retrieve")
app = g.compile()
```

### 它强在哪

- **显式状态**:每一步在改什么、读什么,看 graph 一目了然
- **天然支持循环、条件分支、人机协作中断**
- **断点恢复**:state 持久化后,流程可中断重启,长任务很有用
- **trace 漂亮**:配合 LangSmith 看流程图非常直观

### 它适合什么

| 场景 | 用 LangGraph 合适吗 |
| --- | --- |
| 简单的 RAG | 杀鸡用牛刀 |
| 多步审批 / 复杂业务流 | **非常合适** |
| 长时运行任务、需要 human-in-the-loop | 几乎是首选 |
| 多 Agent 协作 | 也行,但比专门的多 Agent 框架更可控 |

> 我的判断:**LangGraph 是 LangChain 阵营里目前最值得学的一块**。它把"链式"的隐式抽象换成了"图 + 状态"的显式抽象,出错时能调试。

## 七、DSPy:声明式 Prompt 编程

斯坦福的 OmarKhattab 团队搞的,思路很激进:**别再手写 prompt,声明你要的输入输出,让框架去优化 prompt**。

```python
# DSPy 的核心思路:Signature + Module + Optimizer
import dspy

class QA(dspy.Signature):
    """根据上下文回答问题。"""
    context: str = dspy.InputField()
    question: str = dspy.InputField()
    answer: str = dspy.OutputField()

class RAG(dspy.Module):
    def __init__(self):
        self.retrieve = dspy.Retrieve(k=3)
        self.generate = dspy.ChainOfThought(QA)

    def forward(self, question):
        ctx = self.retrieve(question).passages
        return self.generate(context=ctx, question=question)

# 关键是这里:用训练集自动优化 prompt
optimizer = dspy.BootstrapFewShot(metric=my_metric)
compiled = optimizer.compile(RAG(), trainset=examples)
```

### 它牛在哪

- 你不写 prompt,你声明"输入是什么、输出是什么",DSPy 帮你试出 few-shot 示例和指令
- 跟你后续学的 Eval 天然合一:**有 metric 才能优化**
- 模型换了,prompt 自动重新优化,迁移成本低

### 它的代价

1. **学习曲线陡**:你得先理解"程序即可微"的范式
2. **生态小**:tool use、Agent 这类组件远不如 LangChain
3. **必须有训练集**:没几十条标注数据,优化器跑不动

| DSPy vs 手写 Prompt | 手写 | DSPy |
| --- | --- | --- |
| 启动成本 | 低 | 高 |
| 模型迁移成本 | 高(每次重新调) | 低(重跑 optimizer) |
| 可重复性 | 差,玄学 | 好,有 metric |
| 适合谁 | 90% 项目 | 有 eval、追求极致质量的团队 |

> DSPy 是个"未来感"很强的方向。如果你团队有 eval 文化、有数据,可以试。**没有 eval 别碰 DSPy**,会更乱。

## 八、轻量方案:直接 SDK + 一点代码

很多场景里,**最好的"框架"是没有框架**:官方 SDK + 几个工具函数,200 行代码搞定一个 Agent。

```python
# "无框架"风格,200 行能搞一个能干活的 RAG Agent
from anthropic import Anthropic
import chromadb

client = Anthropic()
db = chromadb.PersistentClient("./db").get_or_create_collection("docs")

def retrieve(query, k=3):
    return db.query(query_texts=[query], n_results=k)["documents"][0]

def chat(messages, tools=None):
    return client.messages.create(
        model="claude-opus-4-7",
        max_tokens=2048,
        tools=tools or [],
        messages=messages,
    )

def rag_answer(question):
    docs = retrieve(question)
    msg = [{
        "role": "user",
        "content": f"参考资料:\n{docs}\n\n问题:{question}"
    }]
    return chat(msg).content[0].text
```

这种写法的好处:

1. **代码全在你手里**,bug 在哪一目了然
2. **没有版本兼容焦虑**,SDK 升级一般兼容
3. **新人 1 小时就能读懂全部**,不像 LangChain 要先读三本文档
4. **想加任何中间步骤随便加**,不用学框架的"扩展点"

### 什么时候真的需要框架

| 你需要…… | 自己写够了 | 上框架 |
| --- | --- | --- |
| 单轮 LLM 调用 | 够 | 不需要 |
| 简单 RAG | 够 | 不需要 |
| 工具调用 + 几轮循环 | 够 | 不需要 |
| 复杂状态机、断点续跑 | 痛苦 | LangGraph |
| 多种数据源接入 | 痛苦 | LlamaIndex |
| 端到端 trace / eval | 几乎不可能 | LangSmith / Langfuse |
| 自动 prompt 优化 | 不可能 | DSPy |

## 九、选型矩阵

把上面所有维度合成一张表。横轴是项目阶段,纵轴是项目复杂度。

| | 简单(单功能) | 中等(几个模块) | 复杂(多业务线) |
| --- | --- | --- | --- |
| **POC / Demo** | 直接 SDK | LangChain / LlamaIndex | LangChain + LangGraph |
| **MVP 上线** | SDK + 薄封装 | LlamaIndex(RAG)/ Claude SDK(Agent) | LangGraph |
| **生产稳定** | SDK + 自封装 | LlamaIndex / Claude SDK | LangGraph + 自定义层 |
| **持续迭代** | 同上 | 引入 Eval(promptfoo / Langfuse) | LangGraph + DSPy + Eval |

### 团队规模角度

| 团队 | 推荐 |
| --- | --- |
| 1-2 人,快速验证 | 直接 SDK,200 行就能跑 |
| 3-10 人,有产品方向 | LlamaIndex(RAG)+ Claude SDK(Agent) |
| 10+ 人,多业务线 | LangGraph + 自建框架层,把规范沉淀进去 |
| 大厂中台 | 几乎都自研框架,不会直接吃开源 |

> 大公司不直接用 LangChain 的根本原因不是技术,是**控制权**。框架升级你压不住,关键路径不能押在外部项目上。

## 十、给新手的建议:第一个项目别上框架

这是我最想说的一条。

如果你刚开始做 AI 应用,**第一个项目用官方 SDK 直接写**。原因:

1. **你需要先理解 LLM 调用的"原貌"**:messages、tools、stream、token 消耗。框架会把这些藏起来,你以为自己懂了,其实没懂
2. **你需要知道"框架替你做了什么"**:不知道,你就分不清是 LLM 不行还是框架配错了
3. **你的需求会变**:第一版用框架,第二版业务复杂了,你发现框架的抽象不匹配,迁移代价比重写还大

学习路径建议:

```
第 1 个项目:Anthropic / OpenAI SDK,纯手撸
            ↓
第 2 个项目:RAG → 加 LlamaIndex 或 Chroma + 几行代码
            ↓
第 3 个项目:复杂工作流 → 引入 LangGraph
            ↓
第 4 个项目:开始关心 eval → 加 promptfoo / Langfuse
            ↓
有需要再考虑 DSPy
```

> 别反过来。先框架后原理,你会一直在"为什么 LangChain 报这个错"里打转,而不是在做产品。

## 十一、迁移成本:框架锁定的代价

最后讲一个被严重低估的点:**框架不仅是依赖,是一种"思维方式锁定"**。

| 锁定层级 | 例子 | 迁移代价 |
| --- | --- | --- |
| API 层锁定 | 用了 `ChatOpenAI` 而不是 OpenAI SDK | 低(改导入即可) |
| 抽象锁定 | 用了 `Runnable / LCEL` | 中(逻辑要重写) |
| 数据格式锁定 | 用 LlamaIndex 的 `Node` / `Document` 全套 | 中高 |
| 状态机锁定 | LangGraph 的 `StateGraph`,所有逻辑围着它转 | 高 |
| 工具链锁定 | LangSmith trace、CrewAI 角色定义 | 高 |

### 怎么降低锁定

1. **核心业务逻辑用纯 Python 写**,框架只负责"周边"(retry、并发、trace)
2. **不要让框架的类型穿透到业务边界**:对外接口收原始 dict / Pydantic,内部再 mapping 到框架对象
3. **写薄薄一层适配器**,把 `llm.invoke` 之类的调用收口,以后换框架只改这一层
4. **关键 prompt 不要硬编码在框架的 `PromptTemplate` 里**,放配置文件或 Git 仓库

> 写代码的第一天就要问自己一个问题:**这段代码,半年后我换框架,要改多少行?** 答案越大,你被锁得越死。

---

## 踩坑与建议

1. **第一个项目用 SDK 撸通,不要上框架**。先理解原貌,再选合适的封装层。
2. **不要既要又要**。RAG 为主就 LlamaIndex,Agent 为主就 Claude SDK / LangGraph,别把两套都引入。
3. **不要早早上多 Agent**。单 Agent 做扎实,真不够用再拆。多 Agent 在 demo 里好看,生产里多半是浪费 token。
4. **LangChain 的 LCEL / Runnable 别炫技**。生产代码用最朴素的写法,出 bug 时谁都能看懂。
5. **业务逻辑不要绑死框架的类型**。框架对象只在"边界层"出现,核心代码用 dict / Pydantic。
6. **没有 Eval 不要碰 DSPy**。它的价值前提是有 metric,没 metric 它就是个更复杂的 prompt 模板器。
7. **框架升级前先看 changelog**。LangChain 大版本之间砍 API 是常事,生产项目锁住小版本号。
8. **重要 prompt 不要藏在框架配置里**。当成代码资产管,版本管理、code review 都要走。
9. **Trace 工具(LangSmith / Langfuse)优先级高于框架本身**。能看到每次 LLM 怎么被调用的,比"链子写得优雅"重要 10 倍。
10. **国内访问问题**:LangChain 完全可用,Claude SDK 视环境而定,DSPy 默认用 OpenAI,选型时把"能不能稳定调到 API"列入第一优先级。
