# LLM Reasoning:o1、DeepSeek R1 与"推理模型"范式

2024 年 9 月 OpenAI 发布 o1,"在回答前先想很久"的模型第一次走到台前。2025 年 1 月 DeepSeek 开源 R1,把训练配方公开,整个开源社区一夜复刻。这不是普通的版本升级——**它代表着 LLM 从"语言模型"向"推理模型"的范式迁移**。

> 一句话先记住:**预训练是把知识压进模型,SFT/RLHF 是教它说人话,推理模型是再加一步——教它"在出答案前先长长地想一遍"。这一步用 RL 训出来,且推理时也要花钱花算力**。

---

## 一、什么叫"推理模型"

普通 LLM 拿到题就开始往外吐 token,thinking 是隐式的、薄薄一层。推理模型不一样:

```
普通 LLM:
  输入 → [短输出]
  
推理模型:
  输入 → [可能几千到几万 token 的"思考过程"] → [最终答案]
                  ↑
       o1 把思考藏起来,只展示摘要
       R1 / Qwen QwQ 把思考完整暴露
```

思考过程里典型动作:**自我提问、列举假设、自我反驳、试错、回溯、估算、代入验算**。

### 1.1 它解决什么问题

| 任务 | 普通 LLM | 推理模型 |
| --- | --- | --- |
| 数学竞赛(AIME / MATH) | 50% 左右 | 80-95% |
| 代码竞赛(Codeforces) | 1100 分 | 2000+ 分 |
| 物理 / 化学 PhD 级题(GPQA) | ~40% | ~75% |
| 长链推理 / 多步规划 | 容易跳步、漏前提 | 显著好 |
| 闲聊、改写、总结 | **更好(快、便宜)** | 杀鸡用牛刀 |

**不是越聪明越好——简单任务用推理模型只是慢和贵**。

### 1.2 关键观察:test-time compute 是新维度

过去 7 年,LLM 进步靠的都是 **训练时算力**(参数量 + 训练 token 数,见 19 篇 Scaling Law)。o1 引入第二条曲线:

```
训练算力 ↑ → 模型更聪明(老路)
推理算力 ↑ → 同一个模型答得更对(新路)
```

OpenAI 的图(o1 报告):AIME 准确率随**思考 token 数**的对数线性上涨。这个性质对工程师极其重要——**你能用钱换准确率**,而以前不能。

## 二、推理是怎么训出来的:三种思路

### 2.1 思路 A:Process Reward Model + 搜索(2023-2024 主流)

每一步给 reward:

```
模型生成多条思路 → PRM 给每步打分 → 沿高分路径继续 / 回溯(beam / MCTS)
```

代表:DeepMind 的 AlphaProof、各种 "process supervision" 论文。
缺点:**PRM 本身要标注百万级"步骤对错"数据**,人力极贵;搜索时计算翻数倍。

### 2.2 思路 B:RL on Outcome(o1 / R1 的路线)

只用最终答案对错当 reward。让模型在尝试中**自己摸索"长 CoT、自我反思"是有用的**。

```
模型生成 N 个完整 reasoning trace → 答对 +1, 答错 -1
       ↓
RL(PPO / GRPO)更新策略
       ↓
模型学到:多写"等等,我再检查一下"反而更容易答对
```

DeepSeek R1 的开创点:**长 CoT 行为是 RL 自发涌现的,不是 SFT 教的**。R1-Zero 完全不用 SFT,纯 RL 也学出了反思能力(虽然语言混乱、可读性差)。

### 2.3 思路 C:从强模型蒸馏 long CoT

最便宜的复刻路线。拿 R1 / o1 / QwQ 生成几十万条带思考过程的样本,SFT 一个小模型。

```
DeepSeek 自己:R1 → 蒸馏出 R1-Distill-Qwen-32B / 7B / 1.5B
社区做法:R1 / QwQ → 蒸馏出自己的 14B / 7B 思考模型
```

7B 蒸馏版能在 AIME 接近 60%,**比同尺寸普通模型翻倍**,这是开源生态最出活的路径。

## 三、DeepSeek R1 的训练管线(展开看)

R1 论文公开了完整四阶段配方,信息密度极高:

```
DeepSeek-V3-Base
   │
   ├─ 阶段 1:R1-Zero
   │    纯 RL(GRPO)+ 规则 reward(答案对错 + 格式)
   │    → 推理强,但语言混乱、有时跳中英文
   │
   ├─ 阶段 2:Cold Start SFT
   │    人工筛选 / 改写几千条干净 long CoT 数据 → SFT V3-Base
   │    → 修语言、修格式
   │
   ├─ 阶段 3:Reasoning RL
   │    继续在数学 / 代码题上做 GRPO + 语言一致性 reward
   │    → 数学/代码冲到 SOTA
   │
   ├─ 阶段 4:Rejection Sampling + 通用 SFT/RL
   │    扩展到通用任务(写作、闲聊、代码助手)
   │    避免"只会做奥数题"
   │
   └─ R1(最终模型)
```

### 3.1 GRPO:为什么不用 PPO

传统 PPO 需要一个 **critic / value model**(和 actor 一样大),内存翻倍。GRPO(Group Relative Policy Optimization)的妙招:

```
对每个 prompt,采样 G 条回答(典型 G = 8 或 16)
       ↓
组内归一化:r_i_norm = (r_i - mean) / std
       ↓
直接拿这个相对优势做策略梯度,省掉 critic
```

省一半显存、训练更稳。这是 R1 能用合理算力跑出来的关键工程取舍。

### 3.2 Reward 设计:简单到惊人

只有两类规则 reward:

| Reward | 怎么算 |
| --- | --- |
| **Accuracy** | 数学题:把答案抽出来和 ground truth 字符串比对;代码题:跑测例 |
| **Format** | 必须用 `<think>...</think>` 标签包思考过程 |

**没有 PRM,没有人评分员**。这反过来说明:**只要终点信号干净,长链推理是可学的涌现行为**。

---

## 四、OpenAI o1 / o3 的不同

OpenAI 没公开细节,但根据已有信号:

- 同样靠 RL on outcome,但 reward 来源更多(可能含模型评判 / 工具反馈)
- **隐藏思考**:用户只看摘要,完整 CoT 不返回——既防被蒸馏,也避免"误导"用户
- **思考预算可控**:`reasoning_effort: low/medium/high` 控制思考 token 数
- **o3** 加入了"工具使用"作为思考的一部分(读文件、执行代码)——把 ReAct 内化进训练

> 现在所有"前沿大模型"基本都在做这条:**Claude 4 / Gemini 2.5 Pro 也在 inference 时拉长 thinking token**。差别只是开源闭源、显式隐式。

## 五、推理模型的工程现实

### 5.1 在 API 里用

**Anthropic 的 extended thinking**(展开思考)是最干净的接口:

```python
import anthropic

client = anthropic.Anthropic()
resp = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=16000,
    thinking={"type": "enabled", "budget_tokens": 8000},  # 给思考留 8K
    messages=[{"role": "user", "content": "证明:对所有正整数 n,n^2 + n 是偶数。"}],
)

for block in resp.content:
    if block.type == "thinking":
        print("[思考]", block.thinking[:500], "...")
    elif block.type == "text":
        print("[最终答案]", block.text)
```

DeepSeek R1 / QwQ 也类似,只不过思考过程在 `<think>...</think>` 标签里,你自己 split 即可。

### 5.2 何时用 / 何时别用

| 场景 | 用推理模型? |
| --- | --- |
| 数学、算法、定理证明 | ✅ 必须 |
| 代码:复杂调试、架构设计 | ✅ 收益明显 |
| 代码:补全、改个变量名 | ❌ 浪费 |
| 长链规划(法律推断、多步医诊断) | ✅ |
| 闲聊、客服 FAQ、改写 | ❌(贵 5-10 倍,用户也等不及) |
| RAG 总结一段文档 | ❌ 大概率不需要 |
| Agent 决定下一步用什么工具 | ⚠️ 取决于任务复杂度,简单情况轻量模型够 |

### 5.3 价格曲线

```
推理模型的输出 token 价 ≈ 普通模型的 3-5 倍
但思考也要计费,思考 token 经常是答案 token 的 5-20 倍
合计单次回答花钱 ≈ 普通模型 15-100 倍
```

所以工程上常见的混合架构:

```
廉价模型:意图识别、路由、格式化
   │
   ├─ 简单 → 廉价模型直接答
   │
   └─ 复杂 → 转交推理模型 → 答完之后廉价模型做格式化/翻译
```

**只在"真要思考的那一段"花钱,前后都用便宜模型**。

## 六、推理模型 + 工具(o3 / R1 工具版)

最新趋势:让模型**在思考过程中调用工具**。

```
<think>
用户问 100 万美元 1995 年至今的购买力换算。
我需要 CPI 数据。调用 web 搜索:
</think>
[tool_use: web_search("US CPI 1995 to 2025")]
[tool_result: ...]
<think>
1995 年 CPI ≈ 152.4,2025 年 ≈ 320.5,通胀率 ≈ 110%。
所以 100 万 1995 美元 ≈ 210 万 2025 美元。
</think>
答案:约 210 万美元。
```

这一步把 21 篇(Tool Use)和 22 篇(RAG)和推理融到一起。**Claude Code 的 plan mode、o3 deep research、Gemini 的 Deep Research 都是这个范式**。

## 七、推理模型的局限与误区

1. **不是越想越对**。某些任务(感知、共情、风格化写作)思考反而拖累。
2. **会"装思考"**。如果训练里思考过程被绕过(short-circuit),模型会装模作样写 5000 字思考但答案早就锁死。
3. **思考过程不可信**。o1 / Claude 都警告过:**"思考"不是模型真实计算的忠实记录,只是另一段输出**。别把它当 audit log。
4. **延迟体验差**。一个简单数学题等 30 秒,用户体验受冲击。要么 streaming 思考过程,要么提示"模型正在思考...预计 X 秒"。
5. **小模型蒸馏≠真推理**。R1-Distill-7B 在数学题上看着分高,但**out-of-distribution 任务远不如 R1-671B**。"会思考"是大模型的奢侈品。
6. **prompt 越简洁越好**。推理模型不需要你 few-shot——它自己会想。一堆 CoT 模板反而干扰它。

## 八、给应用工程师的实操建议

1. **先用普通模型 + 好 prompt**。不行再升级到推理模型。别一上来就 o1。
2. **设思考预算**(budget_tokens / reasoning_effort)。**思考可控才能上线**,不然延迟和成本爆。
3. **缓存 prompt**(prompt caching)。推理模型 system prompt 通常很长,缓存能砍 80% 成本。
4. **结果二次检查**。推理模型仍然会编数字、拼写错代码。**生产环境永远要 verifier**(测试 / 计算器 / DB 验证)。
5. **eval 单独建**。原来的 eval 集容易"普通模型答错、推理模型秒杀",起不到区分作用——要做更难的题集。
6. **混合路由**。前端用便宜模型分流,把"真的需要推理"的请求才送进昂贵管线。

## 九、再往前一年看会是什么样

- **训练时 + 推理时双轴 scaling**:大公司用更长 CoT、更多 RL 步数继续推
- **工具内化**:模型自己决定何时搜、何时算、何时写代码,不靠外层 orchestration
- **过程透明 vs 商业保密**:开源继续暴露 CoT、闭源继续藏
- **小模型推理化**:14B / 32B 蒸馏版逐步生产可用,本地推理可行
- **推理 + Agent 融合**:o3 deep research / Claude Computer Use 已在路上

> 推理模型不是终点,它是**给 LLM 加了一根"内部计算尺"**。底层 Transformer 没变,真正变的是训练目标和 inference 形态。下一波的范式爆炸,大概会从这条线长出来。
