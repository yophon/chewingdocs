# Flutter 支付集成

支付是 App **最敏感、最容易踩政策坑**的领域。两条主线:

1. **虚拟商品 / 订阅(数字内容)** → **必须**走 Apple IAP / Google Play Billing
2. **实物 / 服务交易** → 可以走支付宝、微信、Apple Pay、Google Pay、Stripe

走错平台会被下架,**这是上架最大的雷**。

---

## 一、政策红线(必读)

### Apple

- App 里**售卖虚拟物品**(VIP、币、订阅、会员)→ **必须用 IAP**,Apple 抽 30%(小开发者 15%)
- 不能引导用户去网页、小程序购买(`不能放外部链接 / 二维码`)
- 实物 / 实体服务可以走第三方:打车、外卖、酒店、电商商品

### Google Play

类似 Apple,虚拟商品必须 Google Play Billing。物理商品可以用第三方。

### 国内市场(华为、小米、OPPO、VIVO)

各家也要求虚拟商品走它的内购。多渠道分发要适配各厂商。

### 误区

> "我做的是知识付费 App,内容算不算虚拟?"

→ **算**。课程、文章、咨询时长都算 IAP 范围。除非课程跨设备消费(如 web 也能看),Apple 仍可能要求 IAP。

## 二、虚拟商品:in_app_purchase(官方)

```yaml
dependencies:
  in_app_purchase: ^3.2.0
```

iOS 在 App Store Connect、Android 在 Google Play Console 创建商品(consumable / non-consumable / subscription)。

### 基本流程

```dart
final iap = InAppPurchase.instance;

// 1. 检查可用
final available = await iap.isAvailable();

// 2. 查询商品
final response = await iap.queryProductDetails({'gold_100', 'vip_monthly'});
final products = response.productDetails;

// 3. 监听购买回调(放 main 入口)
iap.purchaseStream.listen((purchases) async {
  for (final p in purchases) {
    if (p.status == PurchaseStatus.purchased || p.status == PurchaseStatus.restored) {
      // 校验收据(强烈建议在服务端做)
      final ok = await myBackend.verifyReceipt(p.verificationData);
      if (ok) {
        await deliverContent(p.productID);
        if (p.pendingCompletePurchase) {
          await iap.completePurchase(p);   // ⚠️ 必须 complete 否则苹果会重发
        }
      }
    } else if (p.status == PurchaseStatus.error) {
      print('购买失败:${p.error}');
    }
  }
});

// 4. 发起购买
final param = PurchaseParam(productDetails: products.first);
await iap.buyConsumable(purchaseParam: param);     // 一次性消耗品
// 或
await iap.buyNonConsumable(purchaseParam: param);  // 永久 / 订阅
```

### 商品类型

| 类型 | 例子 |
| --- | --- |
| Consumable(消耗) | 游戏币、礼物、能量 |
| Non-consumable(非消耗) | 永久解锁、去广告 |
| Subscription(订阅) | 月会员、年会员 |

类型选错了:消耗品被判定永久,用户买一次再也付不了钱。

### 服务端校验(必做)

```
iOS: 把 verificationData.serverVerificationData 发给后端
后端: 调 https://buy.itunes.apple.com/verifyReceipt 验证(沙箱用 sandbox.itunes.apple.com)
返回 status == 0 才发货

Android: 后端调 Google Play Developer API 校验 purchaseToken
```

**永远不要只信客户端**。客户端校验可被破解,后端校验是收入安全的最后一道。

### Restore(恢复购买)

非消耗 / 订阅商品换设备 / 重装 App 后要能恢复:

```dart
await iap.restorePurchases();
```

回调流跟新购买一样,但 `status == PurchaseStatus.restored`。

### 沙箱测试

- iOS:App Store Connect → 用户和访问 → 沙箱测试者 → 用沙箱账号登录设备
- Android:在 Play Console 测试者列表加邮箱,装 internal testing 版

⚠️ **沙箱不收真钱,但会有真实流程**。

## 三、实物 / 服务:Apple Pay / Google Pay

```yaml
dependencies:
  pay: ^2.0.0
```

把信用卡 / Apple Pay / Google Pay 集成到一起。

### Apple Pay

```dart
ApplePayButton(
  paymentConfigurationAsset: 'apple_pay_config.json',
  paymentItems: const [
    PaymentItem(label: '商品', amount: '12.34', status: PaymentItemStatus.final_price),
  ],
  style: ApplePayButtonStyle.black,
  width: 250,
  height: 50,
  onPaymentResult: (result) {
    // 拿到 paymentToken,发给支付网关(Stripe / Adyen / 自建)
  },
)
```

需在开发者账号配置 Merchant ID + 关联 App。

### Google Pay

```dart
GooglePayButton(
  paymentConfigurationAsset: 'google_pay_config.json',
  paymentItems: const [...],
  type: GooglePayButtonType.pay,
  onPaymentResult: ...,
)
```

注意:`pay` 包返回的是**支付 token**,本身不"扣款"。要把 token 发给真正的支付网关(Stripe、Braintree、Adyen 等)由它扣款。

## 四、Stripe(海外标杆)

```yaml
dependencies:
  flutter_stripe: ^11.1.0
```

```dart
Stripe.publishableKey = 'pk_live_xxx';
await Stripe.instance.applySettings();

// 创建 PaymentIntent(后端做)
final clientSecret = await api.createPaymentIntent(amount: 1234);

// 唤起支付表
await Stripe.instance.initPaymentSheet(
  paymentSheetParameters: SetupPaymentSheetParameters(
    paymentIntentClientSecret: clientSecret,
    merchantDisplayName: 'My Store',
  ),
);
await Stripe.instance.presentPaymentSheet();
```

Stripe 自带 Apple Pay / Google Pay / 信用卡 / Klarna / 各种本地支付(德国直接转账、荷兰 iDEAL),**一次集成多种方式**。海外项目首选。

## 五、支付宝:tobias

```yaml
dependencies:
  tobias: ^3.4.0
```

### 申请

支付宝开放平台 → 创建应用 → 提交"App 支付"产品 → 上传公钥 → 拿到 appid + 私钥

### 流程

```
App: 把订单信息(金额、商品名)发给后端
后端: 用支付宝私钥签名,生成 orderInfo 字符串,返回 App
App: tobias.pay(orderInfo)  → 唤起支付宝
支付宝回调 → 后端通过 notify_url 收到结果(可信)
App 也会收到回调(不可信,只用于 UI 跳转)
```

```dart
final result = await Tobias.pay(orderInfo, evn: AliPayEvn.ONLINE);

switch (result['resultStatus']) {
  case '9000': print('支付成功'); break;
  case '6001': print('用户取消'); break;
  case '4000': print('订单错误'); break;
  default: print('其他');
}
```

⚠️ **支付结果以服务端 notify 为准**,App 上的 9000 也可能是假的(中间人攻击)。

## 六、微信支付:fluwx

```yaml
dependencies:
  fluwx: ^4.5.6
```

### 申请

微信开放平台 → 创建移动应用 → 提交审核(7 天)→ 关联微信支付商户号

```dart
// 注册
await registerWxApi(
  appId: 'wx_app_id',
  doOnAndroid: true,
  doOnIOS: true,
  universalLink: 'https://yourdomain.com/app/',  // iOS 必须
);

// 发起
await payWithWeChat(
  appId: 'wx_app_id',
  partnerId: 'mch_id',
  prepayId: 'prepay_id',          // 后端预下单返回
  packageValue: 'Sign=WXPay',
  nonceStr: 'random',
  timeStamp: 1234567890,
  sign: 'signature',
);

// 回调
weChatResponseEventHandler.listen((res) {
  if (res is WeChatPaymentResponse) {
    if (res.errCode == 0) print('支付成功');
  }
});
```

⚠️ **iOS 必须配 Universal Link**(否则跳回 App 失败)。`Apple Developer → Identifiers → 应用 → Associated Domains`,服务端配 `apple-app-site-association` 文件。

## 九、订阅(Subscription)

最复杂的支付类型,要处理:

- 续费成功
- 续费失败(信用卡过期)
- 用户主动取消(到期才生效)
- 退款 / 申诉
- 升级 / 降级方案
- 优惠期 / 试用期

### iOS / Android 标准做法

后端做"订阅状态机":

```
incoming webhook from Apple/Google
  ↓
识别用户订阅状态
  ├─ active            : 当前付费
  ├─ in_grace_period   : 续费失败但宽限期
  ├─ expired           : 已过期
  ├─ in_billing_retry  : 重试中
  └─ refunded          : 已退款
```

App 启动时调后端**确认当前订阅状态**,不要只信客户端的"已购买"。

### RevenueCat(强烈推荐)

`purchases_flutter`,把 Apple / Google / Stripe 的订阅复杂度全屏蔽,提供统一 API:

```dart
await Purchases.configure(PurchasesConfiguration('public_key'));
final offerings = await Purchases.getOfferings();
final result = await Purchases.purchasePackage(offerings.current!.monthly!);
final isPremium = result.customerInfo.entitlements.active.containsKey('premium');
```

后端不用管收据校验、Webhook、退款,RevenueCat 都帮你做。
**收费**:< $2.5K/月免费,超过 1% 抽成。订阅业务非常值。

## 十、典型订单架构

不要让支付逻辑在 UI 里:

```
UI (CheckoutPage)
  ↓ 点击付款
PaymentService(抽象)
  ↓
具体网关(IAP / Alipay / Stripe)
  ↓ 唤起支付
平台支付
  ↓ 回调结果(不可信)
App 通知 UI
  ↓ 等待
UI 调后端 /orders/{id}/status,直到 paid 或 timeout
```

### 关键点

1. **创建订单在后端**(防伪造金额)
2. **App 拿到的 prepay 信息发起平台支付**
3. **支付结果以服务端 notify 为准**(平台异步推后端)
4. **App 用轮询 / Push 拿最终状态**
5. **失败处理**:用户离开后台、网络断了——所有这些情况后端都能正确收到 notify 并发货

## 十一、对账与防作弊

### 对账

每天后端跟支付平台拉账单文件,跟自己的订单库 diff:

- App 记成功但平台没记 → 异常,需手动处理
- 平台记成功但 App 没发货 → 漏发,补单
- 金额不一致 → 报警

### 防重复发货

`order_id` 唯一索引 + 状态机(unpaid → paid → delivered)。同一订单不能扣两次款,也不能发两次货。

### 防伪造收据

iOS:base64 解 receipt → 调 Apple verifyReceipt
Android:用 Google Play Developer API
**绝对不要在客户端校验签名**,客户端代码可被改。

---

## 十二、退款

iOS 不像传统支付能"立即退款",流程是:
- 用户向 Apple 申请退款
- Apple 审批
- 推送 Webhook 给你的服务端 (`REFUND` event)
- 你扣回他的虚拟物品 / 取消订阅

**App 内不能做"申请退款"按钮**(Apple 政策)。可以放"如何申请退款"教程链接到 Apple 官方。

## 十三、常见坑

### 1. iOS 沙箱账号登录主账号

→ 你的真实 Apple ID 被绑定到沙箱,以后 App Store 各种诡异错。**沙箱账号只在"设置 → App Store"专门那栏登录**,绝不要登 iCloud。

### 2. 测试时没在 ` ` 里完成支付被覆盖

→ 沙箱状态被卡住,新购买报"已购买过"。重启设备 + 重装 App 解决。

### 3. 微信回调进不到 App

→ Universal Link 配错了,或公司域名没 SSL。

### 4. 服务端没收到 notify

- 防火墙拦了支付平台 IP
- HTTPS 证书过期 / 自签
- notify_url 写错

支付平台 backoffice 都有"重试通知"按钮,先用它复现。

### 5. 多次发起支付

用户狂点付款按钮 → 创建多个订单。**按钮防抖 + Loading 状态 + 服务端幂等**。

### 6. 信用卡数据传到自己服务器

→ 违反 PCI DSS,法律风险高。**永远不要让卡号经过你的服务器**,用 Stripe / Adyen 这类合规网关。

### 7. 跨平台账号系统

iOS 买的会员,Android 登录后没了 → 用户骂街。**自己后端的"会员状态"独立于平台,购买后写入,登录后查询**,跟登录的是哪台设备无关。

## 十四、上架前 Checklist

- [ ] 商品已在 Apple / Google 后台审核通过
- [ ] 沙箱测试全流程(购买 / 取消 / 重购 / 恢复)
- [ ] 服务端收据校验
- [ ] notify_url 公网可达 + HTTPS
- [ ] 失败 / 退款的处理逻辑
- [ ] App 内 "恢复购买" 按钮(iOS 强制要求 Non-consumable / 订阅)
- [ ] 隐私政策提到支付收集的信息
- [ ] 不要在 App 里放跳转网页购买的引导(IAP 类)

## 十五、和已学知识的串联

- 支付 SDK 都是 PlatformChannel(16),底层是原生 SDK
- 订单状态用 Bloc / Riverpod 管理(09 / 10)
- 创建订单 → 调后端 Dio(13)→ 收到 prepay → 唤起支付 SDK
- 长流程要管 Loading / Error 状态(`AsyncValue`)
- 加密 token、商户密钥用 flutter_secure_storage(14)
- 接 RevenueCat 时它的事件流可以包成 StreamProvider(09)

## 十六、心智模型

```
"卖什么决定走哪个支付"

虚拟商品(IAP 强制)
  ├─ iOS  → in_app_purchase + 服务端校验 + RevenueCat(订阅)
  └─ Android → 同上

实物 / 服务
  ├─ 海外  → Stripe(包了 Apple Pay / Google Pay / 卡)
  └─ 国内  → 支付宝(tobias) + 微信(fluwx)

订单流程铁律:
  1. 后端创建订单
  2. App 唤起平台支付
  3. 平台 notify 后端
  4. 后端发货
  App 永远不发货,只展示状态
```

支付集成的复杂度不在 SDK 本身,**而在政策、合规、对账、防作弊**。把这些做好,SDK 调用本身只是 100 行代码。
