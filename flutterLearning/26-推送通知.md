# Flutter 推送通知

推送分两类:

| 类型 | 来源 | 触发 |
| --- | --- | --- |
| **远程推送(Remote / Push)** | 服务端发起,经 APNs / FCM / 厂商通道 | 后端推 |
| **本地通知(Local Notification)** | App 自己定时 / 立即触发 | 客户端 |

国内多渠道更复杂——**苹果走 APNs,Google 服务在国内被墙,Android 厂商各推自家通道**。

---

## 一、推送链路

```
你的服务端
   ↓
Apple APNs           → iOS 设备
Google FCM           → 海外 Android
华为 / 小米 / OPPO 等 → 国内 Android 各厂商通道
```

国内 Android 不接厂商通道:**App 杀掉就收不到**。
接厂商通道:**App 杀掉也能收**(系统级长连接)。

---

## 二、Firebase Cloud Messaging(FCM):海外标准

```yaml
dependencies:
  firebase_core: ^3.6.0
  firebase_messaging: ^15.1.3
```

### 配置

1. Firebase Console 创建项目
2. iOS:下载 `GoogleService-Info.plist` 放 `ios/Runner/`,Xcode 里启用 Push Notifications + Background Modes(Remote notifications)
3. Android:下载 `google-services.json` 放 `android/app/`,gradle 配 `com.google.gms.google-services`
4. APNs Key:Apple Developer → Keys → 创建带 APNs 的 .p8 → 上传到 Firebase

### 用法

```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp();

  // 后台 / 杀进程时收到推送的 handler(必须 top-level 函数)
  FirebaseMessaging.onBackgroundMessage(_bgHandler);

  runApp(const MyApp());
}

@pragma('vm:entry-point')
Future<void> _bgHandler(RemoteMessage msg) async {
  print('后台收到:${msg.notification?.title}');
}

// 申请权限
final settings = await FirebaseMessaging.instance.requestPermission();
print(settings.authorizationStatus);

// 拿到 token(发给后端)
final token = await FirebaseMessaging.instance.getToken();

// 监听 token 刷新
FirebaseMessaging.instance.onTokenRefresh.listen(_uploadToken);

// 前台收到推送
FirebaseMessaging.onMessage.listen((msg) {
  print('前台收到:${msg.data}');
  // 自己显示通知(前台 FCM 默认不显示)
  showLocalNotification(msg);
});

// 用户点击通知打开 App
FirebaseMessaging.onMessageOpenedApp.listen((msg) {
  // 跳转到对应页面
  context.go('/order/${msg.data['orderId']}');
});

// App 完全杀进程时被通知点开
final initial = await FirebaseMessaging.instance.getInitialMessage();
if (initial != null) {
  // 类似处理
}
```

### Topic / 用户分组

```dart
await FirebaseMessaging.instance.subscribeToTopic('promo');
await FirebaseMessaging.instance.unsubscribeFromTopic('promo');
```

后端发推可以指定 topic,所有订阅者都收。

### 后端发送

```python
# Python 示例
import firebase_admin
from firebase_admin import messaging

message = messaging.Message(
  notification=messaging.Notification(title='Hello', body='World'),
  data={'orderId': '42'},
  token='device-token',
)
messaging.send(message)
```

## 三、APNs(iOS 特有)

iOS 推送本质是 APNs。FCM 内部转发给 APNs,你也可以**直接用 APNs**(不依赖 Firebase):

```yaml
dependencies:
  flutter_apns: ^1.6.1
```

但绝大多数项目走 FCM 更省心,**APNs 直连只用在不能用 Google 服务的特殊场景**。

### iOS 配置要点

`Info.plist`:

```xml
<key>UIBackgroundModes</key>
<array>
  <string>remote-notification</string>
</array>
```

`Capabilities`:开启 `Push Notifications` 和 `Background Modes`。

### 苹果通知权限请求时机

```dart
// 不要 main 一进来就请求
// 应该在用户做某个动作后,告诉他们"开通后能...",再请求
```

iOS 14+ 拒绝过一次后,系统弹窗就再也不出了,只能引导用户去设置开。

## 四、国内厂商推送

iOS 走 APNs 没问题,国内 Android 必须接厂商通道才能"App 杀了也能收"。

主流方案:

| SDK | 特点 |
| --- | --- |
| **极光推送 (jpush_flutter)** | 国内最早最成熟,文档好,免费版 |
| **个推** | 国内大厂用得多 |
| **友盟推送** | 阿里系,跟阿里云生态绑定 |
| **小米 / 华为 / OPPO / VIVO 自有** | 自己接,最稳但麻烦 |

### 极光推送

```yaml
dependencies:
  jpush_flutter: ^4.1.0
```

```dart
final jpush = JPush();
jpush.applyPushAuthority(
  const NotificationSettingsIOS(sound: true, alert: true, badge: true),
);
jpush.setup(
  appKey: 'your-app-key',
  channel: 'developer-default',
  production: false,
);

// 拿到 registration id(类似 token)
final rid = await jpush.getRegistrationID();

// 监听通知
jpush.addEventHandler(
  onReceiveNotification: (msg) async => print('收到:$msg'),
  onOpenNotification: (msg) async {
    // 用户点开
  },
);
```

极光内部根据厂商自动选通道(华为走华为推送、小米走小米推送…),**不用你各家适配**。

### 自己接厂商(适合大公司)

把每家 SDK 都接了,根据 `Build.MANUFACTURER` 路由:

```kotlin
when (Build.MANUFACTURER.lowercase()) {
  "huawei" -> HMSPush.register()
  "xiaomi" -> MiPush.register()
  "oppo" -> HeytapPush.register()
  // ...
}
```

复杂、出问题难定位,小团队用第三方聚合服务(极光 / 个推)更划算。

## 五、本地通知:flutter_local_notifications

`闹钟、定时提醒、日程通知` 这种**不需要服务端**的场景。

```yaml
dependencies:
  flutter_local_notifications: ^17.2.4
  timezone: ^0.9.4
```

### 初始化

```dart
final plugin = FlutterLocalNotificationsPlugin();

await plugin.initialize(
  const InitializationSettings(
    android: AndroidInitializationSettings('@mipmap/ic_launcher'),
    iOS: DarwinInitializationSettings(),
  ),
  onDidReceiveNotificationResponse: (resp) {
    // 用户点开通知
    print('payload: ${resp.payload}');
  },
);

// iOS 13+ 申请权限
await plugin.resolvePlatformSpecificImplementation<IOSFlutterLocalNotificationsPlugin>()
    ?.requestPermissions(alert: true, badge: true, sound: true);

// Android 13+ 申请权限
await plugin.resolvePlatformSpecificImplementation<AndroidFlutterLocalNotificationsPlugin>()
    ?.requestNotificationsPermission();
```

### 立即显示

```dart
await plugin.show(
  0,                        // id
  '标题',
  '内容',
  const NotificationDetails(
    android: AndroidNotificationDetails('default', '默认渠道'),
    iOS: DarwinNotificationDetails(),
  ),
  payload: 'order_42',      // 点击时拿到
);
```

### 定时

```dart
import 'package:timezone/data/latest.dart' as tz;
import 'package:timezone/timezone.dart' as tz;

tz.initializeTimeZones();

await plugin.zonedSchedule(
  1,
  '提醒',
  '该喝水了',
  tz.TZDateTime.now(tz.local).add(const Duration(seconds: 10)),
  const NotificationDetails(...),
  androidScheduleMode: AndroidScheduleMode.exactAllowWhileIdle,
  uiLocalNotificationDateInterpretation: UILocalNotificationDateInterpretation.absoluteTime,
);

// 取消
await plugin.cancel(1);
await plugin.cancelAll();
```

### 大文本 / 图片 / 进度条 / 分组

```dart
const AndroidNotificationDetails(
  'default', '默认',
  styleInformation: BigTextStyleInformation('很长的内容...'),
)

// 进度条(下载用)
const AndroidNotificationDetails(
  'download', '下载',
  showProgress: true,
  maxProgress: 100,
  progress: 50,
)
```

## 六、点击通知跳转

通知点击 → 打开 App / 跳具体页面。三种状态:

```dart
// 1. App 在前台:onMessage(FCM) / onDidReceiveNotificationResponse(local)
// 2. App 在后台:onMessageOpenedApp(FCM)
// 3. App 已被杀:getInitialMessage(FCM) 在 main 启动时拿
```

### 通用跳转封装

```dart
Future<void> handleNotificationTap(Map<String, dynamic> data) async {
  final type = data['type'];
  switch (type) {
    case 'order':
      router.go('/order/${data['orderId']}');
      break;
    case 'message':
      router.go('/chat/${data['userId']}');
      break;
  }
}
```

main 里:

```dart
// 启动时
final initial = await FirebaseMessaging.instance.getInitialMessage();
if (initial != null) handleNotificationTap(initial.data);

// 后台跳前台
FirebaseMessaging.onMessageOpenedApp.listen((m) => handleNotificationTap(m.data));
```

## 七、Notification Channel(Android 8+)

Android 必须给通知分组(渠道),用户能各自开关:

```dart
const channel = AndroidNotificationChannel(
  'order_channel',
  '订单通知',
  description: '订单状态变化',
  importance: Importance.high,
);

await plugin.resolvePlatformSpecificImplementation<AndroidFlutterLocalNotificationsPlugin>()
    ?.createNotificationChannel(channel);
```

发通知时指定 channelId:

```dart
NotificationDetails(
  android: AndroidNotificationDetails('order_channel', '订单通知'),
)
```

不分渠道 → Android 8+ 通知不显示。

## 八、Notification Service Extension(iOS 富内容)

iOS 收到推送时让 App 处理(下图、解密、改文案),用 **Notification Service Extension**:

1. Xcode → File → New → Target → Notification Service Extension
2. 写 Swift 代码处理 `RemoteNotification`
3. 服务端发推时 `mutable-content: 1`

可以在用户看到通知前**改 title/body、下图片当大图通知**。Flutter 这一块不能直接搞,要写原生(Swift / Kotlin)。

## 九、Silent Push(静默推送)

不显示通知,只是唤醒 App 做事(同步数据、上报位置)。

iOS:`content-available: 1`,不带 alert
Android:FCM 不带 `notification`,只有 `data`

```dart
FirebaseMessaging.onBackgroundMessage(_bgHandler);

@pragma('vm:entry-point')
Future<void> _bgHandler(RemoteMessage msg) async {
  // 后台 isolate 跑,可以用 sqflite、http
  await syncData();
}
```

⚠️ iOS 限制:每小时几次,系统会节流。
⚠️ 国内厂商通道大多不支持静默推送。

## 十、推送送达率与优化

### iOS 几乎不丢

APNs 是 Apple 长连接,只要设备在线就送达。

### Android 国内丢得严重

不接厂商通道、电池优化设置 → 杀进程后收不到。

**优化思路**:
- 接 IM SDK(腾讯云、网易云信、融云)走自己长连接,不依赖系统通知
- 接极光 / 个推,内部多通道
- 引导用户给 App "电池白名单"
- 用 Foreground Service 保持心跳(微信 / QQ 在用)

### 测试丢失原因

- 推送 token 没刷新(用户重装 App,token 变了,后端还存的旧的)
- 推送平台限流(免费版 FCM 没限,但极光免费版有量级限制)
- 系统省电 / 数据省流量
- 用户禁了通知

每次发推,后端要记**真正发出去 / 推送平台返回的状态**,定期对账。

## 十一、推送的合规

- iOS 14+ 必须**先弹权限**,用户拒绝就没了
- iOS 16+ 推送数据要**加密**(Encrypted Payload)推荐
- Android 13+ POST_NOTIFICATIONS 也要请求
- 国内合规:推送 SDK 需要在隐私协议里说明收集的数据

## 十二、调试技巧

### Token 没变

发现推送收不到,先看 token 是否一致:
- 客户端 `getToken()` 打印
- 服务端数据库存的
- 推送平台后台显示的

三者一致才是对的。

### 直接平台后台测试

FCM Console、极光 Console 都能输入 token 直接推一条,绕过你的服务端。这能验证 token 链路。

### 自家服务端发的能否到客户端

```
client A → 你后端 → push 平台 → 设备 B
```

链路任何一段出错都收不到。**逐段排查**。

### 真机调试推送

iOS 模拟器**不支持**推送(自 Xcode 11.4 起部分支持,但限制多)。**用真机**。

## 十三、常见坑

### 1. iOS 沙箱 / 生产 token 不一样

debug 模式的 token 走沙箱 APNs,release 走生产 APNs,**两边 token 不通用**。
对应推送平台也要分环境。

### 2. AndroidManifest 没加 service

某些第三方 SDK 要在 Manifest 注册 Service,漏了就收不到。

### 3. Notification icon 显示纯白方块

Android 通知 icon 必须是**透明背景的白色 PNG**,带颜色会被系统覆盖。

### 4. 多账号切换没注销 token

A 账号绑了 token,A 退出,B 登录用同一 token → A 收到 B 的消息。
**登出时调推送平台 API 解绑**。

### 5. 国内 App 用了 Google FCM

Google 服务在国内被墙,FCM 收不到。**必须用极光 / 个推 / 厂商通道**。

### 6. 锁屏不显示

Android 通知 importance 没设 `high` 或没建 channel。

## 十四、典型集成方案

```
出海 App
  ├─ FCM(Firebase)
  ├─ flutter_local_notifications(本地通知)
  └─ 服务端用 firebase-admin

国内纯安卓
  ├─ 极光 / 个推(集成厂商通道)
  ├─ flutter_local_notifications
  └─ 后端调极光 REST

国内全平台
  ├─ iOS:APNs(直连或经 FCM 转发)
  └─ Android:极光(自动多厂商)
```

## 十五、和已学知识的串联

- 拿到 token 用 Dio(13)上报后端
- 通知点击跳转用 go_router(12)
- 通知数据可能加密,在 isolate 里解密(18)
- 在通知 Service Extension 里改文案 → 跟 Flutter 通信用 PlatformChannel(16)
- 用户登录登出时同步注册 / 解绑 token,放 AuthCubit(10)里做
- IM 自己长连接通常用 WebSocket → StreamProvider(09)
- 推送权限申请用 permission_handler(配合 17、22)

## 十六、心智模型

```
推送类型
  ├─ 远程推送 → 后端发,经云通道
  │    ├─ 海外:FCM(覆盖 Apple + Google + Web)
  │    └─ 国内:APNs + 极光/个推/厂商
  └─ 本地通知 → 客户端定时 / 立即,不联网

iOS  : APNs 必须真机测,debug/release token 分离
Android: 必须分 channel,国内必须接厂商通道
跳转 : 三种状态(前台/后台/杀掉)各自处理
合规 : 权限弹窗时机 + 解绑 token
```

**推送是看似简单实际坑很多的领域**。建议第一次集成跟着官方 sample 走完整流程,再看进阶(silent push / 富文本 / 多端)。

很多初创团队一开始接 FCM / 极光,接好够用一年,后期再考虑各厂商单独适配。
