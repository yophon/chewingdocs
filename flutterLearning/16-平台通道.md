# Flutter 平台通道:跟原生 iOS / Android 通信

Flutter 自己用 Dart 跑在自己的引擎上,**当你需要调用平台特有 API**(蓝牙、健康、传感器、原生 UI、第三方 SDK 等),就要走"平台通道"。

三种通道:

| 类型 | 用途 |
| --- | --- |
| `MethodChannel` | 调一次方法,等返回值(最常用) |
| `EventChannel` | 持续接收事件流(传感器、电量) |
| `BasicMessageChannel` | 双向发消息,任意类型 |

外加现代化方案:

| 方案 | 特点 |
| --- | --- |
| **Pigeon**(官方推荐) | 代码生成,类型安全 |
| **FFI**(`dart:ffi`) | 直接调 C/C++/Rust 库,不经原生层 |

---

## 一、什么时候需要原生通信?

✅ 必要场景:
- 调用平台特有 API(健康数据、Apple Pay、Android Intent)
- 集成第三方 SDK(支付宝、微信、地图)
- 嵌入原生 View(WebView、Camera、地图视图)
- 性能敏感的 C/C++ 库

❌ 不需要场景:
- 已有 Flutter pub 包(99% 的常见需求,先搜 pub.dev)

## 二、MethodChannel 基础

### Dart 端

```dart
import 'package:flutter/services.dart';

const channel = MethodChannel('com.example.app/battery');

Future<int> getBatteryLevel() async {
  final level = await channel.invokeMethod<int>('getBatteryLevel');
  return level ?? 0;
}
```

### Android 端(Kotlin)

`MainActivity.kt`:

```kotlin
class MainActivity: FlutterActivity() {
  private val CHANNEL = "com.example.app/battery"

  override fun configureFlutterEngine(flutterEngine: FlutterEngine) {
    super.configureFlutterEngine(flutterEngine)
    MethodChannel(flutterEngine.dartExecutor.binaryMessenger, CHANNEL)
      .setMethodCallHandler { call, result ->
        when (call.method) {
          "getBatteryLevel" -> {
            val level = getBatteryLevel()
            if (level != -1) result.success(level)
            else result.error("UNAVAILABLE", "电量不可用", null)
          }
          else -> result.notImplemented()
        }
      }
  }

  private fun getBatteryLevel(): Int {
    val bm = getSystemService(BATTERY_SERVICE) as BatteryManager
    return bm.getIntProperty(BatteryManager.BATTERY_PROPERTY_CAPACITY)
  }
}
```

### iOS 端(Swift)

`AppDelegate.swift`:

```swift
@UIApplicationMain
@objc class AppDelegate: FlutterAppDelegate {
  override func application(
    _ application: UIApplication,
    didFinishLaunchingWithOptions launchOptions: ...
  ) -> Bool {
    let controller = window?.rootViewController as! FlutterViewController
    let channel = FlutterMethodChannel(
      name: "com.example.app/battery",
      binaryMessenger: controller.binaryMessenger
    )

    channel.setMethodCallHandler { call, result in
      if call.method == "getBatteryLevel" {
        UIDevice.current.isBatteryMonitoringEnabled = true
        let level = Int(UIDevice.current.batteryLevel * 100)
        result(level)
      } else {
        result(FlutterMethodNotImplemented)
      }
    }

    GeneratedPluginRegistrant.register(with: self)
    return super.application(application, didFinishLaunchingWithOptions: launchOptions)
  }
}
```

### 数据类型映射

Flutter ↔ 原生的标准类型转换:

| Dart | Android (Java/Kotlin) | iOS (Swift) |
| --- | --- | --- |
| null | null | nil |
| bool | Boolean | NSNumber(bool:) |
| int | Long / Integer | NSNumber |
| double | Double | NSNumber |
| String | String | String |
| Uint8List | byte[] | FlutterStandardTypedData |
| List | List | Array |
| Map | HashMap | Dictionary |

**自定义对象需要先转 Map**(或用 Pigeon)。

## 三、EventChannel:持续事件流

电量、传感器、网络状态变化,这种"原生持续推数据"的场景。

### Dart

```dart
const eventChannel = EventChannel('com.example.app/battery_stream');

Stream<int> get batteryStream =>
    eventChannel.receiveBroadcastStream().map((e) => e as int);

// 使用
final sub = batteryStream.listen((level) {
  print('电量:$level');
});

// 离开页面时
sub.cancel();
```

### Android

```kotlin
EventChannel(flutterEngine.dartExecutor.binaryMessenger, "com.example.app/battery_stream")
  .setStreamHandler(object : EventChannel.StreamHandler {
    private var receiver: BroadcastReceiver? = null

    override fun onListen(args: Any?, events: EventChannel.EventSink) {
      receiver = object : BroadcastReceiver() {
        override fun onReceive(c: Context?, i: Intent?) {
          val level = i?.getIntExtra(BatteryManager.EXTRA_LEVEL, -1)
          if (level != null) events.success(level)
        }
      }
      registerReceiver(receiver, IntentFilter(Intent.ACTION_BATTERY_CHANGED))
    }

    override fun onCancel(args: Any?) {
      unregisterReceiver(receiver)
      receiver = null
    }
  })
```

### iOS

```swift
class BatteryStreamHandler: NSObject, FlutterStreamHandler {
  private var eventSink: FlutterEventSink?
  private var observer: NSObjectProtocol?

  func onListen(withArguments _: Any?, eventSink events: @escaping FlutterEventSink) -> FlutterError? {
    eventSink = events
    UIDevice.current.isBatteryMonitoringEnabled = true
    observer = NotificationCenter.default.addObserver(
      forName: UIDevice.batteryLevelDidChangeNotification,
      object: nil, queue: nil
    ) { _ in
      events(Int(UIDevice.current.batteryLevel * 100))
    }
    return nil
  }

  func onCancel(withArguments _: Any?) -> FlutterError? {
    if let o = observer { NotificationCenter.default.removeObserver(o) }
    return nil
  }
}
```

## 四、Pigeon:类型安全的方法通道(强烈推荐)

手写 MethodChannel 的两个痛点:**字符串方法名容易写错、参数没类型检查**。Pigeon 用代码生成解决。

### 安装

```yaml
dev_dependencies:
  pigeon: ^22.4.2
```

### 定义接口(Dart 文件,只是描述,不会运行)

`pigeons/battery.dart`:

```dart
import 'package:pigeon/pigeon.dart';

@HostApi()             // Flutter → Native(同步调原生)
abstract class BatteryApi {
  int getBatteryLevel();
  String getDeviceName();
}

@FlutterApi()          // Native → Flutter
abstract class BatteryEvents {
  void onBatteryChanged(int level);
}

class DeviceInfo {
  String? brand;
  String? model;
  int? sdk;
}
```

### 配置生成

`pigeons/battery.dart` 顶部加:

```dart
@ConfigurePigeon(PigeonOptions(
  dartOut: 'lib/src/messages.g.dart',
  kotlinOut: 'android/app/src/main/kotlin/com/example/app/Messages.g.kt',
  kotlinOptions: KotlinOptions(package: 'com.example.app'),
  swiftOut: 'ios/Runner/Messages.g.swift',
))
```

跑生成:

```
dart run pigeon --input pigeons/battery.dart
```

### 使用

Dart:

```dart
final api = BatteryApi();
final level = await api.getBatteryLevel();   // 强类型!IDE 补全!
```

Android:

```kotlin
class BatteryHandler : BatteryApi {
  override fun getBatteryLevel(): Long { ... }
  override fun getDeviceName(): String { ... }
}

// MainActivity
BatteryApi.setUp(flutterEngine.dartExecutor.binaryMessenger, BatteryHandler())
```

iOS:

```swift
class BatteryHandler: BatteryApi {
  func getBatteryLevel() throws -> Int64 { ... }
  func getDeviceName() throws -> String { ... }
}

BatteryApiSetup.setUp(binaryMessenger: ..., api: BatteryHandler())
```

**好处**:接口定义、类型、自定义对象、错误处理都一站式生成,**Flutter 团队自己的 plugins 全用 Pigeon**。

## 五、Plugin 项目:把通道封装成独立包

通用功能(蓝牙、相机)做成 plugin 发布到 pub.dev,用法:

```bash
flutter create --template=plugin --platforms=android,ios my_plugin
```

生成的项目结构:

```
my_plugin/
├── lib/                    Dart 端 API
├── android/                Android 端实现
├── ios/                    iOS 端实现
└── example/                示例 App
```

`lib/my_plugin.dart`:

```dart
class MyPlugin {
  static const _channel = MethodChannel('my_plugin');

  static Future<String?> getPlatformVersion() async {
    return await _channel.invokeMethod<String>('getPlatformVersion');
  }
}
```

发布:`flutter pub publish`。

## 六、Platform View:嵌入原生视图

需要在 Flutter UI 里嵌一个原生 View(地图、相机预览、WebView):

```dart
// Android
return AndroidView(
  viewType: 'my-native-view',
  creationParams: {'url': 'https://...'},
  creationParamsCodec: const StandardMessageCodec(),
);

// iOS
return UiKitView(
  viewType: 'my-native-view',
  creationParams: {'url': 'https://...'},
  creationParamsCodec: const StandardMessageCodec(),
);
```

需要在原生端注册 ViewFactory。性能比纯 Flutter 渲染慢一些(尤其 iOS),能用 Flutter 实现就别用 PlatformView。

## 七、FFI:直接调 C / C++ / Rust(不经原生层)

`dart:ffi` 让 Dart 直接调用动态库,**不走平台通道**,性能极高。

```dart
import 'dart:ffi';
import 'dart:io';
import 'package:ffi/ffi.dart';

// 加载 .so / .dylib / .dll
final dylib = Platform.isAndroid
    ? DynamicLibrary.open('libnative.so')
    : DynamicLibrary.process();

// 绑定 C 函数:int add(int a, int b);
typedef NativeAdd = Int32 Function(Int32 a, Int32 b);
typedef DartAdd = int Function(int a, int b);

final add = dylib.lookupFunction<NativeAdd, DartAdd>('add');

print(add(1, 2));    // 3
```

适合:**计算密集型**(图像处理、加密、ML 推理),配合 Rust 写的 native 库尤其香。

`ffigen` 工具能从 C 头文件自动生成绑定代码。

## 八、错误处理

### Dart 侧

```dart
try {
  await channel.invokeMethod('foo');
} on PlatformException catch (e) {
  print('错误码:${e.code},消息:${e.message}');
} on MissingPluginException {
  print('原生端没实现这个方法');
}
```

### 原生侧

Android:`result.error("CODE", "message", details)`
iOS:`result(FlutterError(code: "CODE", message: "msg", details: nil))`

## 九、线程模型

⚠️ 重要:

- **Method/EventChannel 默认在主线程**(原生 UI 线程)调用
- 重活(网络、磁盘 IO)必须在原生端开子线程,否则卡 UI

Android:

```kotlin
Thread {
  val data = heavyWork()
  runOnUiThread { result.success(data) }    // 回主线程返回
}.start()
```

iOS:

```swift
DispatchQueue.global().async {
  let data = heavyWork()
  DispatchQueue.main.async { result(data) }
}
```

## 十、常见坑

### 1. channel name 不一致

```
com.example.app/battery     ← Dart 写的
com.example.app/Battery     ← Android 写的(大写 B)
```

→ 调用直接静默失败。**复制粘贴别手输**。

### 2. 平台只实现了一边

iOS 实现了,Android 没实现 → Android 上抛 `MissingPluginException`。
**两边都要实现,或在 Dart 用 `Platform.isAndroid` 判断**。

### 3. 自定义对象传不过去

```dart
class User { ... }
channel.invokeMethod('setUser', user);   // ❌ User 不是标准类型
```

→ 转成 `Map<String, dynamic>`,或上 Pigeon。

### 4. EventChannel 忘记 cancel

订阅了不取消 → 原生端 broadcast 一直在跑,内存/电量泄漏。
**StatefulWidget 在 dispose 里 cancel**(回顾 06)。

### 5. PlatformView 嵌套 ListView 卡顿

iOS 上 PlatformView 跟 Flutter 渲染合成有性能成本,放在大列表里会卡。优先用纯 Flutter 实现。

### 6. 不知道 Plugin 还是 Channel

- 一次性自用功能 → 直接 MethodChannel
- 复用 / 发布 → 做成 Plugin 项目

## 十二、推荐选择

```
有现成 pub 包?         → 直接用,别造轮子
一两个简单方法?         → 手写 MethodChannel
多个方法 / 复杂数据?    → Pigeon
持续事件流?            → EventChannel(或 Pigeon FlutterApi)
要发布给别人用?         → Plugin 项目
要嵌原生 UI?           → PlatformView(慎用)
高性能 C/C++/Rust?     → dart:ffi
```

## 十三、和已学知识的串联

- 平台通道返回的数据 → 喂给 Cubit/Riverpod → emit → UI 重建(09/10)
- Plugin 接口注册到 get_it / Riverpod(11),业务层不接触 Channel 细节
- 异步调用要 mounted 守卫(04 / 06):`if (!mounted) return; setState(...)`
- EventChannel 的 Stream 可以包成 `StreamProvider`(09 Riverpod)

---

## 十四、心智模型

```
MethodChannel  = 函数调用(我问你答)
EventChannel   = 事件流(你不停推送给我)
Pigeon         = MethodChannel + 类型生成(像 gRPC)
PlatformView   = 把原生 UI 嵌进 Flutter UI
FFI            = 直接调 C 库,不走原生层
```

**首选 Pigeon**,字符串错、类型错都不会再发生;**复杂功能做成 Plugin**,长期收益最大。
