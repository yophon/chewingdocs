# Flutter 打包发布

从代码到上架,完整流程。Android、iOS、Web 三端都讲,加 CI/CD。

---

## 一、构建模式回顾

| 模式 | 命令 | 用途 |
| --- | --- | --- |
| Debug | `flutter run` | 开发,JIT,带调试符号 |
| Profile | `flutter run --profile` | 性能测试,接近 release |
| Release | `flutter build xxx --release` | 发布,AOT,体积最小 |

发布永远用 release,**不要把 debug 包给用户**(慢、大、有调试 hook)。

---

## 二、版本号管理

`pubspec.yaml`:

```yaml
version: 1.2.3+45
```

- `1.2.3` = versionName(给用户看)
- `45` = versionCode(必须递增,每次发布 +1)

iOS / Android 自动从这里读。或构建时覆盖:

```bash
flutter build apk --build-name=1.2.3 --build-number=45
```

## 三、Android 打包

### 1. 配置签名

生成 keystore:

```bash
keytool -genkey -v -keystore ~/upload-keystore.jks \
  -keyalg RSA -keysize 2048 -validity 10000 -alias upload
```

⚠️ **keystore 妥善保管**,丢了就再也无法更新已上架的 App!备份到 1Password / 公司密钥管理。

`android/key.properties`(**不要进 git**):

```properties
storePassword=你的密码
keyPassword=你的密码
keyAlias=upload
storeFile=/Users/xxx/upload-keystore.jks
```

`.gitignore` 加上:

```
android/key.properties
*.jks
*.keystore
```

### 2. 修改 `android/app/build.gradle`

```text
def keystoreProperties = new Properties()
def keystorePropertiesFile = rootProject.file('key.properties')
if (keystorePropertiesFile.exists()) {
    keystoreProperties.load(new FileInputStream(keystorePropertiesFile))
}

android {
    signingConfigs {
        release {
            keyAlias keystoreProperties['keyAlias']
            keyPassword keystoreProperties['keyPassword']
            storeFile keystoreProperties['storeFile'] ? file(keystoreProperties['storeFile']) : null
            storePassword keystoreProperties['storePassword']
        }
    }
    buildTypes {
        release {
            signingConfig signingConfigs.release
            minifyEnabled true
            shrinkResources true
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
        }
    }
}
```

### 3. 应用 ID / 包名

```text
android {
    namespace "com.example.myapp"
    defaultConfig {
        applicationId "com.example.myapp"
    }
}
```

应用 ID 一旦上架就**不能改**。

### 4. 应用图标 / 启动图

用 `flutter_launcher_icons` + `flutter_native_splash`:

```yaml
dev_dependencies:
  flutter_launcher_icons: ^0.13.1
  flutter_native_splash: ^2.4.1

flutter_launcher_icons:
  android: true
  ios: true
  image_path: "assets/icon.png"     # 1024x1024
  adaptive_icon_background: "#FFFFFF"
  adaptive_icon_foreground: "assets/icon_fg.png"

flutter_native_splash:
  color: "#FFFFFF"
  image: assets/splash.png
  android_12:
    image: assets/splash_android12.png
    color: "#FFFFFF"
```

```bash
dart run flutter_launcher_icons
dart run flutter_native_splash:create
```

### 5. 构建

```bash
# APK(给测试 / 内部分发)
flutter build apk --release

# App Bundle(上架 Google Play 必须)
flutter build appbundle --release
```

输出:
- APK:`build/app/outputs/flutter-apk/app-release.apk`
- AAB:`build/app/outputs/bundle/release/app-release.aab`

### 6. 多渠道 / 分架构

```bash
# 分 ABI 减小体积
flutter build apk --split-per-abi --release

# 多 flavor(开发 / 预发布 / 生产)
flutter build apk --flavor prod -t lib/main_prod.dart
```

`build.gradle` 配置 flavor:

```text
flavorDimensions "default"
productFlavors {
    dev { applicationIdSuffix ".dev" }
    prod { }
}
```

## 四、iOS 打包

### 1. Apple Developer 账号

- $99/年(个人 / 公司)
- App Store Connect 创建 App
- 配置 Bundle ID(类似 Android applicationId)

### 2. Xcode 配置

`open ios/Runner.xcworkspace`(注意是 .xcworkspace,不是 .xcodeproj):

- **General** → Bundle Identifier、Version、Build
- **Signing & Capabilities** → Team、Provisioning(选 Automatic)

### 3. 构建 ipa

```bash
flutter build ipa --release
```

输出:`build/ios/ipa/Runner.ipa`

### 4. 上传 App Store

方式 1:Xcode

```
Xcode → Product → Archive → Distribute App → App Store Connect
```

方式 2:`altool` 命令行

```bash
xcrun altool --upload-app --type ios -f Runner.ipa \
  --apiKey XXX --apiIssuer YYY
```

方式 3:**Transporter**(Mac App,最稳定)

打开 Transporter,把 ipa 拖进去 → Upload。

### 5. TestFlight 内测

App Store Connect → My Apps → TestFlight → 添加测试员邮箱。
用户装 TestFlight App,接受邀请就能装测试版。

### 6. 上架审核

- 准备元数据:截图、描述、关键词、隐私政策 URL
- 各分辨率截图(6.7"、6.5"、5.5"、iPad)
- 提交审核,通常 1-3 天
- 拒绝原因常见:权限说明不清(`NSCameraUsageDescription` 等)、登录账号不能给审核员用

### 7. 权限说明(Info.plist)

```xml
<key>NSCameraUsageDescription</key>
<string>用于拍摄商品照片</string>

<key>NSPhotoLibraryUsageDescription</key>
<string>用于选择头像</string>

<key>NSLocationWhenInUseUsageDescription</key>
<string>用于显示附近的店铺</string>
```

iOS 审核必查这个。**说明要写清楚用户能理解的目的**。

## 五、Web 打包

```bash
flutter build web --release
```

输出:`build/web/` 整个目录,部署到任意静态服务器(Nginx / Cloudflare Pages / Vercel / GitHub Pages)。

### 优化

```bash
# 默认 canvaskit 渲染(质量好)
flutter build web --release

# html 渲染(体积小但效果差)
flutter build web --web-renderer html

# Tree-shake icons
flutter build web --release  # 默认就开了

# Path-based URL(去掉 /#/)
# 在 index.html 修改 base href + 服务端 fallback 到 index.html
```

### Nginx 部署示例

```nginx
server {
  listen 80;
  root /var/www/myapp;
  index index.html;

  # SPA fallback
  location / {
    try_files $uri $uri/ /index.html;
  }

  # Flutter 资源大缓存
  location ~ \.(js|css|png|jpg|webp|svg|woff2)$ {
    expires 30d;
  }
}
```

### CORS / iframe 注意

Flutter Web 跑 canvaskit 默认禁 iframe 嵌入。要嵌的话指定 renderer 或修改 index.html。

## 七、混淆(Obfuscation)

```bash
flutter build apk --release --obfuscate --split-debug-info=build/symbols
```

- `--obfuscate` 把 Dart 代码混淆
- `--split-debug-info` 把调试符号单独输出(崩溃日志解析需要)

⚠️ **symbols 目录要保管好**,出了崩溃需要它来解析堆栈。

## 八、CI/CD:GitHub Actions

`.github/workflows/build.yml`:

```yaml
name: Build

on:
  push:
    tags:
      - 'v*'

jobs:
  android:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.24.0'

      - name: Setup keystore
        env:
          KEYSTORE: ${{ secrets.ANDROID_KEYSTORE_BASE64 }}
          KEY_PROPS: ${{ secrets.KEY_PROPERTIES }}
        run: |
          echo $KEYSTORE | base64 -d > android/upload-keystore.jks
          echo "$KEY_PROPS" > android/key.properties

      - run: flutter pub get
      - run: flutter test
      - run: flutter build appbundle --release

      - uses: actions/upload-artifact@v4
        with:
          name: aab
          path: build/app/outputs/bundle/release/app-release.aab

  ios:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.24.0'

      - name: Setup signing
        env:
          P12: ${{ secrets.IOS_P12_BASE64 }}
          P12_PWD: ${{ secrets.IOS_P12_PASSWORD }}
          PROVISION: ${{ secrets.IOS_PROVISION_BASE64 }}
        run: |
          # 创建临时 keychain,导入证书,安装 provisioning profile
          # ...(略,可用 fastlane match 简化)

      - run: flutter build ipa --release --export-options-plist=ios/ExportOptions.plist

      - uses: actions/upload-artifact@v4
        with:
          name: ipa
          path: build/ios/ipa/*.ipa
```

签名密钥放 GitHub Secrets:**ANDROID_KEYSTORE_BASE64**、**KEY_PROPERTIES**、**IOS_P12_BASE64** 等。

## 十、Fastlane(自动上架)

把"build → 上传 → 提交审核"全自动化。

`ios/fastlane/Fastfile`:

```ruby
default_platform(:ios)

platform :ios do
  desc "上传到 TestFlight"
  lane :beta do
    build_app(scheme: "Runner")
    upload_to_testflight
  end

  desc "上架 App Store"
  lane :release do
    build_app(scheme: "Runner")
    upload_to_app_store(submit_for_review: true)
  end
end
```

```bash
cd ios && fastlane beta
```

Android 同理:

```ruby
platform :android do
  lane :deploy do
    upload_to_play_store(
      track: 'production',
      aab: '../build/app/outputs/bundle/release/app-release.aab'
    )
  end
end
```

## 十一、热更新?

**Flutter 没有官方热更新**(类似 React Native 的 CodePush)。

原因:Apple 政策禁止下发可执行代码。

社区方案:
- `shorebird`(同名公司,Flutter 团队前成员做的)— 半商业化,iOS 受限
- 自己用 `flutter_dynamic` 之类(法律风险)

**实战建议**:
- iOS:不做热更新,走 TestFlight / App Store
- Android:可以用 shorebird,或纯下发 JSON 配置 + 服务端控制

## 十二、上架前 Checklist

### 通用
- [ ] release 模式包测一遍真机
- [ ] 应用图标、启动图清晰
- [ ] 应用名、版本号、build 号正确
- [ ] 网络请求都走 HTTPS
- [ ] 没有 print / debug log
- [ ] 没硬编码 API key 或密钥
- [ ] 隐私政策页(链接 / 内嵌)

### Android
- [ ] keystore 备份了
- [ ] AAB 文件,不是 APK
- [ ] 权限只声明用得到的
- [ ] target SDK 最新(Play Console 强制)
- [ ] 64-bit 支持(`flutter build appbundle` 默认开)

### iOS
- [ ] Bundle ID、Team 正确
- [ ] 隐私权限说明文案清楚
- [ ] App Store 截图、描述、关键词
- [ ] 测试账号给审核员
- [ ] 没有引用 IDFA(用了得在审核问卷勾选)
- [ ] App Tracking Transparency(iOS 14+)

### Web
- [ ] 路由 fallback 到 index.html
- [ ] 静态资源 CDN / 长缓存
- [ ] base href 正确
- [ ] favicon、manifest.json 配好(PWA)

## 十三、崩溃监控

### Firebase Crashlytics(免费,推荐)

```yaml
dependencies:
  firebase_core: ^3.6.0
  firebase_crashlytics: ^4.1.3
```

```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp();

  FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterFatalError;

  PlatformDispatcher.instance.onError = (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack, fatal: true);
    return true;
  };

  runApp(MyApp());
}
```

### Sentry(更全功能)

```yaml
dependencies:
  sentry_flutter: ^8.9.0
```

```dart
await SentryFlutter.init(
  (options) => options.dsn = '...',
  appRunner: () => runApp(MyApp()),
);
```

记得**上传 split-debug-info 的 symbols** 才能解析堆栈。

## 十四、灰度 / 分阶段发布

### Google Play

Play Console → 内测 → 公测 → 分阶段发布(1% → 10% → 50% → 100%)。
出问题随时暂停。

### App Store

阶段化发布:7 天内自动覆盖 1% → 100% 用户。出问题可暂停。

### 服务端 Feature Flag

不依赖应用更新就能开关功能:`FirebaseRemoteConfig` / `LaunchDarkly` / 自建。

## 十五、热点问题

### 1. iOS 模拟器能跑,真机崩

→ 模拟器是 x86_64,真机 ARM。检查 native 库支持,清缓存重 pod install。

### 2. Android 上某些设备字体显示方块

→ fallback 没配中文字体。回顾 17。

### 3. 包体积过大

```bash
flutter build apk --release --analyze-size
```

DevTools 会展示哪些资源最占体积。常见原因:
- 重复字体
- 未压缩图片
- 多架构未拆分

### 4. iOS 审核被拒:权限不清

权限说明 plist 里写人话,别写 "for app function"。

### 5. 上架后版本号不能回退

versionCode 只能升不能降。**永远递增**。

## 十六、和已学知识的串联

- DevTools 测性能(18) → release 包测才有意义
- 国际化(17):上架文案要支持目标地区语言
- 网络配置(13):release 包记得切到生产环境的 baseUrl
- 主题(17):iOS / Android 的 splash 用 `flutter_native_splash` 一次配两边
- 崩溃监控 + DevTools 性能数据 → 上线后真实反馈

## 十七、心智模型

```
开发  → debug
测试  → profile + release(真机)
发布  → release + 混淆 + symbols 备份
       + 签名 + 上架
       + 崩溃监控 + 灰度
       + 分阶段更新
```

**第一次上架最重要的两件事**:
1. **签名密钥备份好**(丢了就废)
2. **审核拒绝时认真读理由**,Apple/Google 大多数拒绝有合理理由,不是故意为难

后面每次发版 = 改版本号 → CI 自动跑 → 内测验证 → 灰度 → 全量。一旦流程跑通,发版变成"按一个按钮的事"。
