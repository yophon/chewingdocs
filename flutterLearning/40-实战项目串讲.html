Flutter 实战项目串讲:电商 / 社交 / IM ​

前 39 篇是"零件",这一篇把零件装成"整车"。三类典型 App,每个讲架构、模块、技术选型、关键难点。

一、共通基础架构 ​

任何中大型 Flutter 项目的骨架基本一致:

lib/
├── main.dart                    # 入口,初始化
├── bootstrap.dart               # 全局初始化(Sentry/DI/i18n)
├── app.dart                     # MaterialApp + Router + Theme
├── core/                        # 业务无关
│   ├── network/                 # Dio interceptor / cache
│   ├── storage/                 # Drift / Hive 封装
│   ├── di/                      # GetIt / Riverpod providers
│   ├── error/                   # Failure 体系
│   ├── analytics/               # 埋点封装
│   └── utils/
├── features/                    # 按业务垂直拆分
│   ├── auth/
│   │   ├── data/                # repository + datasource
│   │   ├── domain/              # entity + usecase
│   │   └── presentation/        # page + state
│   ├── home/
│   ├── cart/
│   └── ...
├── shared/                      # 跨 feature 复用
│   ├── widgets/
│   └── models/
└── l10n/

回顾 21(架构模式)、11(依赖注入)。

4. 商品列表无限滚动 + SWR ​

dart
class ProductsNotifier extends AsyncNotifier<PagedList<Product>> {
  @override
  Future<PagedList<Product>> build() => _load(page: 1);

  Future<void> loadMore() async {
    final cur = state.value!;
    final next = await _load(page: cur.page + 1);
    state = AsyncData(cur.copyWith(
      page: next.page,
      items: [...cur.items, ...next.items],
      hasMore: next.hasMore,
    ));
  }
}

6. 下单流程 ​

购物车选品 → 创建预订单(后端返回:商品快照、可用券、支付方式)
        → 用户确认地址、券、备注
        → 提交订单(后端创建订单 + 锁库存)
        → 跳支付(SDK)
        → 支付回调 / 轮询订单状态
        → 跳订单详情

支付绝不能信任客户端结果:回调以服务端通知 + 订单状态查询为准。

7. 性能要点 ​

  • 首页 + 详情页:首屏图片 LQIP / blurhash 占位
  • 列表项 widget 严格 const + key
  • 商品详情图懒加载,详情大图用 photo_view 单独路由
  • 网络列表:dio cache 中间件 + 本地 SWR
  • 包体积:cached_network_image + flutter_image_compress

8. 推送(回顾 26) ​

  • 订单状态变化 → 服务端推
  • 心愿单降价 → 定时任务 + 推送
  • 个性化运营 → 标签推送

class VideoControllerPool {
  final _cache = LinkedHashMap<String, VideoPlayerController>(maximumSize: 5);

  VideoPlayerController get(String url) {
    if (_cache.containsKey(url)) return _cache[url]!;
    if (_cache.length >= 5) _cache.remove(_cache.keys.first)!.dispose();
    final c = VideoPlayerController.networkUrl(Uri.parse(url));
    _cache[url] = c;
    return c;
  }
}

3. 消息存储模型 ​

sql
CREATE TABLE messages (
  id TEXT PRIMARY KEY,
  conv_id TEXT NOT NULL,
  sender_id TEXT NOT NULL,
  type INTEGER,                -- 1=text 2=image 3=audio ...
  content TEXT,
  created_at INTEGER,
  status INTEGER,              -- 0=sending 1=sent 2=delivered 3=read 4=fail
  local_id TEXT                -- 客户端临时 id,服务端确认后替换
);

CREATE INDEX idx_conv_time ON messages(conv_id, created_at DESC);

4. 发送消息状态机 ​

[本地写入 sending]
    ↓ socket.send
[发出 → 等 ack]
    ↓ ack
[sent]
    ↓ 对方设备拉取
[delivered]
    ↓ 对方读
[read]

若超时无 ack → status = fail,UI 显示重发按钮
dart
Future<void> sendText(String convId, String text) async {
  final msg = Message.local(convId: convId, type: 1, content: text);
  await db.insertMessage(msg);                                 // 1
  ref.read(convProvider(convId).notifier).appendOptimistic(msg);

  try {
    final ack = await socket.send(msg).timeout(15.seconds);
    await db.updateStatus(msg.localId, status: 1, serverId: ack.id);
  } catch (e) {
    await db.updateStatus(msg.localId, status: 4);
  }
}

5. 长连接 ​

App 在前台              → 长连
App 后台 / 锁屏(iOS)  → 走 APNs(系统级)
App 杀掉(Android)     → 走厂商通道
重连策略             → 指数退避 + jitter,网络变化重连
心跳                 → 30s 一次,3 次失败重连

回顾 42(WebRTC 与 IM)有更详细的 socket 客户端实现。

6. 离线消息 ​

登录后调 syncMessages(lastSeq)
  ↓
服务端返 lastSeq 之后的消息(带 seq 单调递增)
  ↓
本地按 seq 写入,跳过重复
  ↓
更新 lastSeq

永远以 server seq 为准,客户端时间不可靠。

7. 会话列表 ​

dart
SELECT c.id, c.title,
       (SELECT m.content FROM messages m
        WHERE m.conv_id = c.id ORDER BY created_at DESC LIMIT 1) AS lastMsg,
       (SELECT COUNT(*) FROM messages m
        WHERE m.conv_id = c.id AND status < 3 AND m.sender_id != ?) AS unread
FROM conversations c
ORDER BY (SELECT MAX(created_at) FROM messages WHERE conv_id = c.id) DESC;

或维护 conversations 表反范式 (lastMsg, unread 字段),消息变化时同步更新。消息几十万条后实时聚合查询会慢,反范式必须做。

9. 群聊 ​

群消息 = 单聊 + groupId。性能关键:

  • 服务端写一份(群消息),客户端按 group + seq 拉
  • "已读"不要每个成员单独存,只存"我读到的 seq"
  • 群头像九宫格用九张小图合成或服务端预生成

10. 端到端加密(可选高级) ​

Signal Protocol(双 Ratchet):每条消息一把 key,即便服务器被入侵也读不到。 Flutter 端有 libsignal_protocol_dart,但工程量大。普通商业 IM 通常不需要 E2EE,做 TLS + 服务端加密落库即可。

七、模块抽象示例(电商版) ​

dart
// domain
abstract class CartRepository {
  Stream<Cart> watch();
  Future<void> add(Sku sku, int qty);
  Future<void> remove(String skuId);
  Future<void> clear();
}

// data
class CartRepositoryImpl implements CartRepository {
  CartRepositoryImpl(this._api, this._db);
  final CartApi _api;
  final CartDao _db;

  @override
  Stream<Cart> watch() => _db.watch();

  @override
  Future<void> add(Sku sku, int qty) async {
    await _db.upsert(sku.id, qty);            // 本地立即
    try { await _api.add(sku.id, qty); }      // 异步同步
    catch (e) { /* 排队重试 */ }
  }
}

// presentation
final cartProvider = StreamProvider<Cart>((ref) {
  return ref.read(cartRepoProvider).watch();
});

各 feature 都按这模式拆,新人能 5 分钟懂。

八、心智模型 ​

任何商业 App = 数据 + 状态 + UI + 网络 + 持久化 + 监控

按业务划分 feature        →  按技术划分核心(core)
保持 feature 之间无引用   →  共享走 core 或 shared
统一异常处理 / 网络拦截   →  避免每页重复 try/catch

电商:从"商品到支付"的漏斗优化
社交:从"内容到互动"的留存优化
IM:从"长连到离线"的可靠性建设
