# Flutter 性能优化

性能问题的本质是**两个数字**:

```
帧率:60fps(每帧 16.67ms),120fps 的设备每帧 8.33ms
内存:不要让 native heap 一直涨
```

只要每帧能在预算内画完、内存不爆,App 就流畅。下面是怎么做。

---

## 一、第一原则:在 profile 模式下测

```bash
flutter run --profile
```

**永远不要在 debug 下评估性能**,因为:
- debug 包跑 JIT、断言、调试 hooks
- 一些动画在 debug 下故意放慢

profile 模式接近 release,但保留 DevTools 能用。

release 包不能调试,只用来发布。

---

## 二、Flutter DevTools:必学利器

启动 App 后命令行会打印 DevTools URL,或在 IDE 里直接打开。

### Performance(看帧率)

- **Timeline**:每一帧的耗时
- **红色帧 = 卡顿**(超过 16ms)
- **CPU 火焰图**:某个函数耗时多少

操作:复现卡顿 → Record → Stop → 看哪一帧红

### CPU Profiler

哪个方法消耗最多 CPU 时间。

### Memory

- **当前堆大小**
- **GC 频率**
- 抓 snapshot 看哪些对象没释放

### Inspector(看 Widget 树)

- 选 Widget,高亮对应位置
- **Highlight Repaints**:粉色边框出现 = 这一帧重绘了
- **Highlight Oversized Images**:大图被缩小显示会高亮(浪费内存)

### Network

看所有 Dio / http 请求(配合 logger / DevTools 网络层)。

## 三、Widget 重建:90% 性能问题在这

回顾 05:Widget 重建本身廉价,**昂贵的是子树里 Element/RenderObject 重建**。

### 1. 用 const 构造

```dart
// ❌
return Padding(padding: EdgeInsets.all(8), child: Text('hi'));

// ✅
return const Padding(padding: EdgeInsets.all(8), child: Text('hi'));
```

`const` 让 Flutter 跳过整个比较和 update 流程。**能加就加**。

### 2. 把不变的子树用 const / 抽出去

```dart
// ❌ 每次 build 都重建整个 Column
return Column(children: [
  Text('$count'),
  ExpensiveWidget(),     // 跟 count 没关系也重建
]);

// ✅
return Column(children: const [
  Text('$count'),
  ExpensiveWidget(),     // 不会重建
]);
```

### 3. AnimatedBuilder / Consumer 的 child 参数

```dart
// ✅ child 只构建一次
AnimatedBuilder(
  animation: _ctrl,
  builder: (_, child) => Transform.rotate(angle: _ctrl.value, child: child),
  child: const ExpensiveWidget(),
)
```

回顾 15:这是动画里最常被遗忘的优化。

### 4. Selector / context.select 精细订阅

回顾 08 / 09:**只订阅你真正需要的字段**,不要订阅整个 Model。

```dart
// ❌ 整个 user 变就重建
final user = context.watch<UserModel>();

// ✅ 只在 name 变化时重建
final name = context.select<UserModel, String>((u) => u.name);
```

### 5. setState 范围最小化

```dart
// ❌ 整个页面是 StatefulWidget,setState 重建整页
class _PageState extends State<Page> {
  int _count = 0;
  void _inc() => setState(() => _count++);
  ...
}

// ✅ 抽小组件,setState 只重建那一小块
class CounterButton extends StatefulWidget { ... }
```

或者直接用 `ValueNotifier + ValueListenableBuilder`(02)。

## 四、列表性能

### 1. 用 ListView.builder,不用 ListView(children: ...)

```dart
// ❌ 一次性构建所有
ListView(children: items.map((i) => ItemTile(i)).toList())

// ✅ 滚到了再构建
ListView.builder(
  itemCount: items.length,
  itemBuilder: (_, i) => ItemTile(items[i]),
)
```

10000 项的差异是天壤之别。

### 2. 给 ListView item 加 Key + 用 const

回顾 07:数据有 id 就 `ValueKey(item.id)`。

### 3. 复杂 item 用 RepaintBoundary

```dart
ListView.builder(
  itemBuilder: (_, i) => RepaintBoundary(
    child: ComplexItemTile(items[i]),
  ),
)
```

把 item 隔成独立绘制层,滚动时 Flutter 复用之前的图层,不重绘。

### 4. cacheExtent

```dart
ListView.builder(
  cacheExtent: 1000,    // 屏幕外 1000 像素都预先构建
  itemBuilder: ...
)
```

提高滚动流畅度,代价是内存。

### 5. 别在 itemBuilder 里做重活

```dart
// ❌ 每次构建都解析一遍
itemBuilder: (_, i) => Text(jsonDecode(items[i].raw)['name'])

// ✅ 提前解析好
itemBuilder: (_, i) => Text(items[i].name)
```

## 五、图片优化(超关键)

图片是最容易吃内存的资源。

### 1. cacheWidth / cacheHeight:解码到合适尺寸

```dart
// ❌ 4000x3000 的图原样解码,占 48MB 内存
Image.network(url)

// ✅ 解码到显示尺寸,只占几 MB
Image.network(
  url,
  cacheWidth: (200 * MediaQuery.devicePixelRatioOf(context)).toInt(),
)
```

DevTools Inspector → Highlight Oversized Images,凡是被高亮的都改一下。

### 2. 用 cached_network_image

```dart
CachedNetworkImage(
  imageUrl: url,
  memCacheWidth: 400,
  placeholder: (_, __) => const ColoredBox(color: Colors.grey),
)
```

回顾 14:磁盘 + 内存双重缓存,自动管理。

### 3. WebP / AVIF

PNG / JPG 用 WebP 替换,体积小一半,Flutter 原生支持。

### 4. 图标用 SVG / Icon font

```dart
import 'package:flutter_svg/flutter_svg.dart';

SvgPicture.asset('assets/icons/star.svg')
```

矢量,任意尺寸不糊,体积小。

### 5. 占位符避免布局抖动

```dart
SizedBox(
  width: 100, height: 100,
  child: CachedNetworkImage(...),
)
```

## 六、避免大 Widget rebuild

### 1. RepaintBoundary 隔离重绘

```dart
RepaintBoundary(
  child: AnimatedSpinner(),
)
```

这一块独立成 layer,绘制变化不影响外面。

### 2. ListView 大列表 sliver 化

复杂滚动用 `CustomScrollView + Slivers`,各 sliver 独立计算。

### 3. const 静态背景

```dart
return Stack(
  children: [
    const _StaticBackground(),     // 不动
    AnimatedForeground(),          // 只动这个
  ],
);
```

## 七、build 方法保持纯净

```dart
@override
Widget build(BuildContext context) {
  // ❌ 这些都不要在这里做
  fetchData();                                        // 网络请求
  Future.delayed(...);                                // 定时器
  expensiveCalc();                                    // 重计算
  controller.forward();                               // 启动动画
  Provider.of<X>(context, listen: false).doSomething(); // 改状态

  return ...;
}
```

`build` **只描述当前应该长什么样**。需要执行操作的事:
- 一次性 → `initState` / `didChangeDependencies`
- 响应事件 → onTap / onPressed
- 副作用 → `addPostFrameCallback`、`ref.listen`、`BlocListener`

## 八、避免布局抖动

### 1. 占位用固定尺寸

```dart
// ❌ 加载完才有尺寸,引发 layout shift
if (loading) CircularProgressIndicator() else List(...)

// ✅
SizedBox(
  height: 600,
  child: loading ? Center(child: ...) : List(...),
)
```

### 2. 异步数据加占位

`shimmer`、骨架屏让用户感觉不到等待。

### 3. 字体加载延迟

`google_fonts` 第一次加载需要网络,先用本地 fallback 字体避免 FOIT。

## 九、内存优化

### 1. 释放 Controller / Subscription

回顾 06:

```dart
@override
void dispose() {
  _controller.dispose();
  _subscription.cancel();
  _timer?.cancel();
  super.dispose();
}
```

### 2. autoDispose Provider

回顾 09:Riverpod 默认 autoDispose,搜索结果、详情页数据不久留。

### 3. 大对象主动释放

图片缓存、视频缓冲、文件流,用完调 dispose / close。

### 4. 监听 onMemoryWarning

```dart
SystemChannels.system.setMethodCallHandler((call) async {
  if (call.method == 'memoryPressure') {
    PaintingBinding.instance.imageCache.clear();
  }
});
```

## 十、启动速度

### 1. main 函数保持轻量

```dart
// ❌ main 里跑一堆初始化,白屏几秒
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await initFirebase();
  await initDb();
  await loadConfig();
  await fetchUser();
  runApp(MyApp());
}

// ✅ 必要的同步初始化 + 异步在 splash 阶段做
void main() {
  WidgetsFlutterBinding.ensureInitialized();
  runApp(const SplashApp());     // 立刻显示 splash
}

// SplashApp 里再异步初始化
```

### 2. 延迟初始化重资源

数据库、视频播放器、地图——**到对应页面再 init**,不要 main 里就建。

### 3. 减少 main bundle 大小

- 字体子集化(只保留用到的字)
- 图片用 WebP
- `flutter build apk --analyze-size` 分析

### 4. 预编译 AOT(release)

```bash
flutter build apk --release        # Android
flutter build ipa --release        # iOS
flutter build web --release        # Web
```

release 用 AOT,启动速度比 debug 快几倍。

## 十三、性能分析 checklist

发现卡顿后,按这个顺序排查:

```
1. profile 模式跑一遍,确认真的卡
2. DevTools Performance 看哪一帧超时
3. Timeline / CPU 火焰图找耗时函数
4. Inspector → Highlight Repaints → 看是不是大区域重绘
5. Inspector → Highlight Oversized Images → 看图片是否过大
6. 检查 ListView 是否用了 builder
7. 检查 build 方法里有没有副作用 / 重计算
8. 给嫌疑节点加 RepaintBoundary
9. 缩小 setState / Provider watch 范围
10. 还慢就上 ffi / Isolate
```

## 十四、Isolate:CPU 密集任务别堵 UI 线程

Dart 是单线程,大计算会卡住 UI:

```dart
// ❌ 解析 10MB JSON,UI 卡 1 秒
final data = jsonDecode(text);

// ✅ 在 isolate 里跑
final data = await compute(jsonDecode, text);
```

`compute` 是简化版 Isolate,适合一次性重活。
更复杂场景用 `Isolate.run`(Dart 2.19+):

```dart
final result = await Isolate.run(() => heavyCalc(input));
```

注意:Isolate 间不共享内存,数据要拷贝(SendPort 序列化)。

## 十五、常见性能坑

### 1. 不必要的 StatefulWidget

简单展示组件用 StatelessWidget 更轻。

### 2. 在 ListView 外包 Column / Padding

```dart
// 大量嵌套布局会增加 layout 时间
Column(children: [
  Padding(child: Container(child: Padding(child: ...))),
])
```

能合并就合并(`Container` 自己带 padding / decoration)。

### 3. setState 循环触发

异步回调里 setState → 又触发新请求 → 又 setState → ...

### 4. ChangeNotifier 没 dispose

→ Listener 一直挂着,引用链不释放。

### 5. ListView 高度无限

`ListView` 在 `Column` 里没设高度 → 报错或异常 layout。**用 `Expanded` 包,或 `shrinkWrap: true`**。

`shrinkWrap: true` 性能差,只用于内容很少的场景。

### 6. opacity 动画用 AnimatedOpacity / FadeTransition

```dart
// ❌ 在 build 里改 opacity 触发整层重绘
Opacity(opacity: _v, child: ...)

// ✅
FadeTransition(opacity: _anim, child: ...)
```

`FadeTransition` 走 GPU 合成层,更快。

## 十八、心智模型

```
性能 = 帧时间 + 内存

帧时间问题
  ├─ build 太慢 → 减少 rebuild 范围 / const / Selector
  ├─ layout 太慢 → 减少嵌套 / RepaintBoundary
  ├─ paint 太慢 → 图片优化 / 减少 effect / RepaintBoundary
  └─ raster 太慢 → GPU 合成层(FadeTransition 替代 Opacity)

内存问题
  ├─ 图片过大 → cacheWidth / WebP
  ├─ Controller 没释放 → dispose
  └─ Provider 常驻 → autoDispose
```

**性能优化的第一原则:不要过早优化**。先写正确的代码,profile 模式真出现卡顿再针对性改。**用 DevTools 数据驱动决策,不要拍脑袋**。
