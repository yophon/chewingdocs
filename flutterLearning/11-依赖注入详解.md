# Flutter 依赖注入(DI)详解

依赖注入(Dependency Injection)听起来高大上,**核心思想就一句话**:**一个类需要的依赖,不要自己 new,从外面传进来**。

为什么?为了**可测试**和**可替换**。

---

## 一、问题先行:不用 DI 会怎样?

```dart
class UserService {
  final api = HttpApi();              // ❌ 写死了
  Future<User> fetchMe() => api.get('/me');
}
```

问题:
- 测试时没法换成 MockApi
- 切换实现(比如换 Dio)要改源码
- API 配置硬编码,环境切换麻烦

DI 后:

```dart
class UserService {
  final Api api;
  UserService(this.api);              // ✅ 外面传进来
  Future<User> fetchMe() => api.get('/me');
}

// 生产:
final service = UserService(HttpApi());
// 测试:
final service = UserService(MockApi());
```

**DI 的本质就是"控制反转":让"创建依赖"和"使用依赖"分离**。

---

## 二、DI 的两种风格

### 1. 构造函数注入(Constructor Injection)

```dart
class A {
  final B b;
  A(this.b);
}
```

最朴素、最推荐的方式。Dart 没有任何特殊支持,纯靠语言本身。

### 2. Service Locator(服务定位器)

把所有依赖放到一个"全局容器",需要时从里面拿:

```dart
final api = locator<Api>();         // 从全局容器拿
```

代表:`get_it`、GetX 的 `Get.find`、Provider 的 `context.read`(在 Widget 里)。

---

## 三、Flutter 生态里的 DI 选择

| 方案 | 类型 | 推荐场景 |
| --- | --- | --- |
| **手动构造注入** | 构造注入 | 简单项目、库 |
| **get_it** | Service Locator | 中大型项目,标准选择 |
| **injectable**(get_it + 注解) | Service Locator + 代码生成 | 大型项目 |
| **Provider** | InheritedWidget 注入 | 数据偏向 UI 层 |
| **Riverpod** | 全局 Provider | 状态 + DI 一站式 |
| **GetX** | Service Locator | GetX 项目 |

---

## 四、get_it:最常用的 DI 容器

```yaml
dependencies:
  get_it: ^7.6.7
```

### 注册依赖

```dart
final getIt = GetIt.instance;

void setupLocator() {
  // 单例:整个 App 一个实例
  getIt.registerSingleton<Api>(HttpApi());

  // 懒加载单例:用到才创建
  getIt.registerLazySingleton<Db>(() => SqliteDb());

  // 工厂:每次拿都新建
  getIt.registerFactory<UserService>(() => UserService(getIt<Api>()));
}

// main 里调一次
void main() {
  setupLocator();
  runApp(const MyApp());
}
```

### 使用

```dart
final api = getIt<Api>();
final service = getIt<UserService>();

// 任何地方都能拿,不需要 context
```

### 三种生命周期

| 方法 | 生命周期 | 何时创建 |
| --- | --- | --- |
| `registerSingleton<T>(T instance)` | 整个 App | 立刻 |
| `registerLazySingleton<T>(() => T)` | 整个 App | 第一次 get 时 |
| `registerFactory<T>(() => T)` | 每次都新建 | 每次 get |

### 异步初始化(数据库、SharedPreferences 等)

```dart
getIt.registerSingletonAsync<Db>(() async => await Db.init());

// main 里
await getIt.allReady();
runApp(MyApp());
```

### 作用域(Scope):页面级 DI

```dart
// 进入页面时
getIt.pushNewScope(scopeName: 'detail');
getIt.registerSingleton<DetailController>(DetailController());

// 离开时
getIt.popScope();   // 自动释放该作用域内所有依赖
```

适合配合页面生命周期管理重资源。

## 五、injectable:get_it 的注解版

不想手写 `setupLocator`,用注解自动生成:

```yaml
dependencies:
  get_it: ^7.6.7
  injectable: ^2.4.1
dev_dependencies:
  injectable_generator: ^2.6.1
  build_runner: ^2.4.9
```

```dart
@injectable
class UserService {
  final Api api;
  UserService(this.api);              // 构造参数自动注入
}

@singleton
class HttpApi implements Api { ... }   // 注册成 singleton
```

跑 `dart run build_runner build`,自动生成 `injection.config.dart`:

```dart
@InjectableInit()
void configureDependencies() => getIt.init();

// main
void main() {
  configureDependencies();
  runApp(const MyApp());
}
```

适合大项目,**避免手写注册代码**。

## 六、Riverpod 作为 DI 容器

Riverpod 本身就是 Provider + DI 一体的方案。任何对象都能用 Provider 注入:

```dart
final apiProvider = Provider<Api>((ref) => HttpApi());

final userServiceProvider = Provider<UserService>((ref) {
  return UserService(ref.read(apiProvider));    // 自动从其他 Provider 拿依赖
});

// 使用
ref.read(userServiceProvider).fetchMe();
```

测试:

```dart
ProviderContainer(
  overrides: [
    apiProvider.overrideWithValue(MockApi()),    // 一行 mock
  ],
)
```

**好处**:状态管理 + DI 用同一套工具,不重复。
**缺点**:任何依赖都要写 Provider,纯后台 Service 也要包一层。

---

## 七、Provider 作为 DI

```dart
MultiProvider(
  providers: [
    Provider<Api>(create: (_) => HttpApi()),
    ProxyProvider<Api, UserService>(
      update: (_, api, __) => UserService(api),
    ),
  ],
  child: MyApp(),
)
```

缺点:依赖必须在 Widget 树内,纯逻辑代码不好用。

---

## 八、GetX 作为 DI

GetX 自带 DI:

```dart
Get.put<Api>(HttpApi());                          // 单例
Get.lazyPut<Db>(() => SqliteDb());               // 懒加载
Get.create<UserService>(() => UserService());     // 每次新建

// 使用
final api = Get.find<Api>();
```

**简单但全局**,跟 GetX 状态管理一样,大项目谨慎。

---

## 九、构造函数注入(最朴素)

不用任何库:

```dart
class App {
  final Api api;
  final Db db;

  App({required this.api, required this.db});
}

// main
void main() {
  final api = HttpApi();
  final db = SqliteDb();
  final userService = UserService(api, db);

  runApp(MyApp(userService: userService));
}
```

**适合**:小项目、库代码、需要极致控制依赖图。
**缺点**:依赖多时 main 函数膨胀。

---

## 十、推荐组合

### 小项目(< 5 个 Service)
- **构造函数注入** + Provider/Riverpod 处理状态

### 中型项目
- **Riverpod** 一站式(状态 + DI)
- 或 **get_it**(纯 DI)+ Riverpod/Bloc(状态)

### 大型项目
- **injectable + get_it**(代码生成,工程化)
- 配合 Bloc 做状态管理

## 十一、典型分层 + DI 示例

```dart
// 1. 接口(domain 层)
abstract class UserRepo {
  Future<User> getMe();
}

// 2. 实现(data 层)
@LazySingleton(as: UserRepo)             // injectable 注解
class UserRepoImpl implements UserRepo {
  final Api api;
  UserRepoImpl(this.api);

  @override
  Future<User> getMe() => api.get('/me').then(User.fromJson);
}

// 3. UseCase / Service
@injectable
class GetMeUseCase {
  final UserRepo repo;
  GetMeUseCase(this.repo);

  Future<User> call() => repo.getMe();
}

// 4. Cubit 用 UseCase
@injectable
class ProfileCubit extends Cubit<ProfileState> {
  final GetMeUseCase getMe;
  ProfileCubit(this.getMe) : super(ProfileState.initial());

  Future<void> load() async {
    emit(state.copyWith(loading: true));
    final user = await getMe();
    emit(state.copyWith(loading: false, user: user));
  }
}

// 5. UI 拿 Cubit
BlocProvider(
  create: (_) => getIt<ProfileCubit>()..load(),
  child: ProfilePage(),
)
```

每一层只依赖**抽象**,实现可替换、可测试,这就是 DI 的终极价值。

## 十二、测试时的 DI 替换

### get_it 方式

```dart
setUp(() {
  getIt.reset();
  getIt.registerSingleton<Api>(MockApi());
});
```

### Riverpod 方式

```dart
ProviderContainer(
  overrides: [
    apiProvider.overrideWithValue(MockApi()),
  ],
)
```

### 构造注入方式

```dart
final service = UserService(MockApi());   // 直接传
```

后两种最干净。

## 十三、常见坑

### 1. get_it 重复注册

```dart
getIt.registerSingleton<Api>(HttpApi());
getIt.registerSingleton<Api>(HttpApi());   // ❌ 报错:already registered
```

热重载时易踩,可以加 `if (!getIt.isRegistered<Api>())`,或用 `getIt.reset()` 清空。

### 2. 循环依赖

A 依赖 B,B 依赖 A → 死锁。
**解法**:抽出公共部分,或用 setter 注入(把依赖改成可后设置)。

### 3. 在 main 之前用依赖

```dart
final api = getIt<Api>();    // ❌ main 没跑完,容器还没初始化
```

任何全局变量都不能在 `setupLocator()` 之前求值。

### 4. Service 依赖 Widget(BuildContext)

```dart
class MyService {
  void show() => ScaffoldMessenger.of(context).showSnackBar(...);  // ❌
}
```

Service 应该跟 UI 解耦。如果非要,通过参数传 context,或者发出事件让 UI 自己响应。

## 十四、心智模型

```
DI 的目标:把"创建"和"使用"分离。

构造注入 = 父对象传依赖给子对象(显式,推荐)
Service Locator = 全局容器,任何地方都能拿(隐式,但方便)
```

**两种没有谁绝对好**,核心问题是你要回答:**这个依赖从哪儿来?能不能在测试里换掉?**只要能,就是好的 DI。

---

## 十五、一句话总结

> 依赖注入不是一个工具,而是一种**思维方式**——让每个类对它的依赖保持"愚蠢但诚实":我需要什么,你告诉我;别人怎么造,我不关心。
