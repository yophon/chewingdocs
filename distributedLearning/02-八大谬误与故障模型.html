八大谬误与故障模型 ​

1994 年,Sun 公司的 Peter Deutsch 列了一份「分布式计算八大谬误」(Fallacies of Distributed Computing),写到墙上贴给工程师看。30 年过去了,这份单子上的每一条都还在每年制造真实事故——AWS S3 大宕机、GitHub 24 小时数据库切换、Cloudflare BGP 黑洞,根上都是某条谬误被忽略。这一篇先把八大谬误配上真实事故讲完,然后引出更深一层的「故障模型」和「时间模型」——这两组词汇是后面 09 篇 FLP 不可能定理、14 篇 ZAB / EPaxos、15 篇拜占庭容错的前置弹药,搞不清楚故障模型,你看 Raft 论文都是云里雾里。

一句话先记住:八大谬误总结起来就是一句"网络不会按你期望工作"——网络会丢包、会延迟、会断开、会被攻击、会被运维改坏。故障模型回答的是"节点会以什么方式坏"——崩了就死(Crash-stop)、崩了又活过来(Crash-recovery)、收到消息不回(Omission)、还是直接撒谎(Byzantine)。时间模型回答的是"消息和处理有没有时间上限"——同步(有上限)、异步(没上限)、部分同步(大部分时间有但偶尔没有)。FLP 告诉你纯异步系统连共识都不可能——所以工程上都假设"部分同步"。

一、Peter Deutsch 八大谬误 ​

1. The network is reliable             网络是可靠的
2. Latency is zero                     延迟为零
3. Bandwidth is infinite               带宽是无限的
4. The network is secure               网络是安全的
5. Topology doesn't change             拓扑结构不会变
6. There is one administrator          只有一个管理员
7. Transport cost is zero              传输成本为零
8. The network is homogeneous          网络是同构的

这八条都是"反向的"——意思是"如果你假设这些为真,你就会出事"。每一条都对应一类真实事故。下面一条一条拆开,配真实 post-mortem 看。

1.1 谬误一:网络是可靠的 ​

真相:网络会丢包、断开、抖动——AWS / Azure / Google Cloud 每天都有几次"网络异常",生产系统必须假设网络永远会断。

TCP 给你字节流可靠
   ↓ 但
TCP 之下是 IP,IP 会丢包
   ↓ 而且
TCP 重传会让"慢" → "超时" → "连接断"
   ↓ 业务层看到的是
"对方挂了" or "网络断了"——你区分不出

真实事故 1:2017 年 AWS S3 us-east-1 大宕机(2 月 28 日)。运维输入命令打错一个参数,把比预期多得多的服务器从子系统中下线。S3 索引子系统大半节点离线,整个 us-east-1 区域的 S3 不可用 4 小时——而很多依赖 S3 的服务(Slack、Trello、Quora)也跟着挂。根因不是网络物理故障,而是"网络配置错误"导致节点被切断——但表现和"网络断了"完全一样。

真实事故 2:2019 年 Cloudflare BGP 路由抖动。某个边缘节点 BGP 配置错误,把全球流量的一部分错误地宣告到一个节点上,该节点过载而其他节点空闲——大量请求超时。整个 Cloudflare 全球网络出现 27 分钟异常。

工程对策:

  • 超时 + 重试(但要小心重试风暴——所有客户端同时重试压垮服务端,详见 systemDesign 限流篇)
  • 幂等设计——重试的请求不会副作用执行两次
  • 熔断降级——上游异常时主动放弃,不要把级联失败传播下去
  • 多可用区部署——单 AZ 网络故障不影响业务

1.2 谬误二:延迟为零 ​

真相:任何网络调用都有不可忽视的延迟——同机房 ~500 μs,跨机房 ~10 ms,跨地域 ~100 ms,这是物理常数。

单机内存访问 ~100 ns
RPC 同机房   ~500 μs   (5000 倍)
RPC 跨机房   ~10 ms    (10 万倍)
RPC 跨地域   ~100 ms   (100 万倍)

典型反模式:N+1 查询——本来一次能批量取的数据,改成循环 N 次调用,单机看不出问题,分布式直接超时:

python
# 反例:N 次 RPC
for user_id in user_ids:
    user = rpc_call(get_user, user_id)   # 100 个用户 → 100 次 RPC

# 正例:1 次批量
users = rpc_call(get_users_batch, user_ids)

真实事故:2016 年某电商促销——单接口本来 50 ms,促销当天链路上下游各加一次跨机房调用,链路从 1 次跨机房变 3 次,总延迟从 100 ms 涨到 300 ms,前端超时阈值 200 ms,整条链路雪崩。

工程对策:

  • 批量化:N 次调用合并成 1 次
  • 本地缓存:把"频繁但低频变化"的数据缓存在本地
  • 异步化:不阻塞主链路的逻辑改异步
  • 降级:超时时返回兜底数据,不无限重试

1.3 谬误三:带宽是无限的 ​

真相:带宽永远有限,而且单机网卡上限 ~25 Gbps(普通) / ~400 Gbps(高端),跨机房专线带宽更贵。

典型反模式:循环传大数据——同事写一个微服务,把整张 1 GB 的用户表 RPC 传过来,本地过滤。单机看不出来,联调时网卡打满,把同机其他服务也拖死。

真实事故:2018 年某直播平台——某条数据被错误地序列化成 50 MB(本应该 50 KB),每秒推送 1 万次 → 500 GB/s 带宽需求,机房交换机直接饱和,所有服务都断网。根因:序列化 bug,但通过"带宽假设无限"的网络架构放大成全机房故障。

工程对策:

  • 数据压缩:gzip / snappy / zstd(尤其是日志和大对象)
  • 协议优化:Protobuf / Thrift / FlatBuffers 比 JSON 小 5-10 倍
  • 流式处理:大数据用流式 API,不一次 dump
  • 流量整形:对单连接设流量上限,防止单租户打爆

1.4 谬误四:网络是安全的 ​

真相:任何网络都可能被攻击——内网不是 VPN,VPN 不等于零信任,默认所有网络都不可信。

传统模型:               零信任模型:
  内网 = 可信               每个请求都验证
  外网 = 不可信             不区分内外网
  防火墙隔开               服务间 mTLS

真实事故 1:2016 年 Mirai 僵尸网络——黑客通过默认密码的 IoT 设备组建 100 万节点僵尸网络,对 Dyn(DNS 服务)发起 DDoS,导致 Twitter、Spotify、Netflix 等大半个美国互联网不可用。

真实事故 2:2017 年 Equifax 数据泄漏——黑客利用 Apache Struts 漏洞渗透内网,因为内部服务之间没有认证,横向移动拿到 1.4 亿用户数据。

工程对策:

  • 零信任架构:服务间 mTLS,不依赖"内网信任"
  • 最小权限:每个服务只能访问它必须访问的资源
  • 审计日志:所有跨服务调用记录,出事能追溯
  • 密钥管理:Vault / KMS / Secrets Manager,不要把密钥写代码里

1.5 谬误五:拓扑结构不会变 ​

真相:节点会上下线、IP 会变、机房会切换、网段会调整——拓扑变化是常态,写死 IP 的系统迟早崩。

典型反模式:配置文件里写死 mysql.host=10.0.0.5——某天运维换机器,IP 变了,应用启动找不到 MySQL,全挂。

真实事故:2012 年 Knight Capital 4.5 亿美元损失——升级交易系统,8 台服务器中有 1 台没升级,跑着旧版本的死代码,被新版本配置激活,该服务器在 45 分钟内疯狂下单,直接造成公司破产。根因:部署时没考虑"拓扑不一致"的情况。

工程对策:

  • 服务发现:ZK / etcd / Consul / Nacos,配置中心动态推送
  • DNS / 负载均衡:用域名 + LB 替代写死 IP
  • K8s Service:服务名解析到当前 Pod 集合
  • 健康检查:节点下线自动从路由表摘除

1.6 谬误六:只有一个管理员 ​

真相:真实系统有 N 个团队、N 个运维——每个组件都有不同的人管,配置可能互相冲突。

真实事故 1:2018 年 GitHub 24 小时降级(10 月 21 日)。运维做日常网络维护,美东和美西的 MySQL 主从断开 43 秒,等恢复时两边都自己提了升主操作,出现"双主写入"。GitHub 团队选择回滚到东岸数据,期间 24 小时业务降级——根因不是单一故障,而是多团队协作下的复杂故障路径。

真实事故 2:2021 年 Facebook 6 小时大宕机。BGP 路由更新错误,导致 Facebook 全球网络从互联网消失;更尴尬的是内部 DNS 也用同样的网络,工程师 SSH 登录修复的渠道也断了,只能派人去机房物理操作。根因:多团队的依赖耦合 + 没有"管理通道独立"。

工程对策:

  • 配置统一收口:配置中心 + 评审机制
  • 变更审批 + 灰度:任何变更先小范围,再扩大
  • 变更回滚预案:每次变更都要有 rollback 脚本
  • 混沌演练:故意制造"两个团队同时变更"的场景

1.7 谬误七:传输成本为零 ​

真相:每次跨进程调用都有 CPU、内存、网卡的成本——序列化、反序列化、TLS 握手、连接维护,累加起来不便宜。

一次 RPC 的开销:
  序列化:1-100 μs (取决于数据量)
  网络发送:RTT ~ 0.5 ms 起步
  反序列化:1-100 μs
  对象 GC:不定 (Java 大对象触发 Full GC 可达秒级)

典型反模式:微服务粒度太细——一个业务请求拆成 50 个微服务,每个微服务再调 5 个下游,链路总长度 250+ 次 RPC,P99 延迟和故障率指数级放大。

真实事故:2019 年某电商大促降级方案失效——本来某接口直接查 DB,改造成"先查缓存服务 → 缓存服务调 Redis → 失败查 DB 服务 → DB 服务读 MySQL",4 次跨进程调用。日常流量没问题,大促时每环节都增加 5 ms 延迟,P99 从 50 ms 涨到 800 ms 直接超时。

工程对策:

  • 合理的微服务粒度:一个业务请求理想链路 ≤ 5 跳
  • 同进程优先:能在同进程做的逻辑别拆服务
  • 连接复用:HTTP/2、gRPC 多路复用,避免重连开销
  • 二进制协议:Protobuf / Thrift,比 JSON 序列化快几倍

1.8 谬误八:网络是同构的 ​

真相:真实网络由不同硬件、不同协议、不同 MTU 组成——客户端 4G、办公网千兆、机房万兆、跨地域专线,性能特征完全不同。

典型反模式:开发用本机测,所有请求都是 localhost(< 100 μs),上生产跨机房一跳 10 ms,核心链路超时。

真实事故:2017 年某 IM 应用海外推送失败——国内推送链路 100 ms 内完成,海外用户经跨地域专线 + 当地运营商,P99 延迟 5 秒,推送服务的硬编码超时 1 秒,所有海外推送全失败。

工程对策:

  • 配置可调超时:不同区域 / 不同链路设不同超时
  • 本地化部署:边缘节点、CDN、就近接入
  • 协议适配:弱网下用更耐丢包的协议(QUIC、HTTP/3)
  • 真实链路压测:不能只测同机房,必须测跨地域 + 弱网

二、八大谬误之上的"九大谬误" ​

行业里有人加了一条"第九谬误":Everything will be fine if I follow best practices——"我按最佳实践做就不会出事"。这是更隐蔽的谬误,因为它让人放松警惕。

真相是:最佳实践只是降低事故概率,不消除事故——你的系统永远会以你想不到的方式出事。心态比知识更重要——假设"我的系统一定会以某种方式坏",然后设计容错,而不是"我设计得很好所以不会坏"。

这种心态是后续 30 篇所有内容的底色——CAP / FLP / Linearizability 这些"不可能定理"反复在提醒你:有些事就是不可能,工程上必须妥协。

三、故障模型(Failure Models) ​

讲完八大谬误,我们换一个视角:节点会以什么方式坏?这就是故障模型——分布式算法的设计必须先假设一个故障模型,不同模型对应不同的算法和成本。

故障模型(从弱到强):

  Crash-stop      节点崩了就死,不再回来
       ↓
  Crash-recovery  节点崩了之后可能恢复(本地状态可能丢)
       ↓
  Omission        节点会"遗漏"——收到消息不回 / 发的消息丢
       ↓
  Byzantine       节点会"撒谎"——可以发任意错误消息,甚至有恶意

模型越强,算法越复杂、成本越高。95% 的分布式中间件假设 Crash-recovery,只有区块链和金融国防场景才假设 Byzantine。

3.1 Crash-stop(崩溃即死) ​

最简单的故障模型——节点要么活,要么彻底死,死了不复活。

节点 A 状态:
  正常 → 崩溃(永久离线)

不会发生:崩溃 → 恢复

特点:好分析、好实现,但不真实——真实世界节点重启后还会回来。

适用场景:理论研究的简化模型。FLP 不可能定理就是在 Crash-stop 假设下证明的——连最简单的故障模型都能让纯异步系统共识不可能。

3.2 Crash-recovery(崩溃可恢复) ​

最常见的工程模型——节点可能崩,但还能起来。

节点 A 状态:
  正常 → 崩溃 → 恢复(可能丢失部分内存状态)

恢复时:
  从持久化日志 / 快照恢复
  重新加入集群
  追上 Leader 的最新日志

关键问题:恢复后的"我"是不是原来的"我"?——这个问题不简单:

  • 持久化的状态可能不完整(crash 时正在写盘)
  • 网络上其他节点可能已经认为"它死了",而它复活后还以为自己是 Leader → 脑裂(详见 30 篇)

典型例子:etcd / ZooKeeper / Kafka / 任何用 Raft / Paxos 的系统——都假设 Crash-recovery 模型,节点重启后从持久化日志恢复。

3.3 Omission(遗漏故障) ​

节点不崩,但消息会丢——可能是接收方收到了不回,可能是发送方发不出去,也可能网络中间丢了。

两种子类型:
  Send omission:消息发不出(本地网卡故障 / 队列满)
  Receive omission:消息收到了不处理(协议栈丢 / 应用 bug)

特点:比 Crash 模型难处理——你不知道对方是真挂了还是临时不回。

典型场景:网络分区 + 节点活着——节点 A 和节点 B 都活着,但 A 和 B 之间通信不上,对 A 来说 B "好像挂了",对 B 来说 A "好像挂了"——双方都可能选举自己当 Leader,造成脑裂。

工程对策:Quorum + 租约——多数派同意 + 时间窗口防止双主(详见 27 篇选主与租约)。

3.4 Byzantine(拜占庭故障) ​

最强模型——节点可以撒谎——发送错误消息、伪造身份、串通其他节点、有恶意。

Crash-stop:节点只会"不响应"
Omission:节点可能"丢消息"
Byzantine:节点可能"发错的消息" / "发不一致的消息给不同节点"

得名于"拜占庭将军问题"——几个将军围攻一座城,通过信使商量进攻还是撤退,但有的将军是叛徒,会发不同的消息给不同的人。如何让忠诚的将军们达成一致行动?

关键数学结论:容忍 f 个拜占庭节点,需要至少 3f+1 个节点(Lamport / Shostak / Pease 1982)。容忍 1 个拜占庭节点需要 4 个节点,容忍 2 个需要 7 个——比 Crash 模型(2f+1)代价大得多。

典型场景:

  • 区块链:Bitcoin / Ethereum / 联盟链——节点之间不信任,可能有恶意
  • 金融跨机构清算:不同银行之间互不信任
  • 军事 / 航天:容错要求极高,硬件位翻转、电路故障都被视为"撒谎"

典型协议:PBFT(Practical Byzantine Fault Tolerance)、Tendermint、HotStuff——详见 15 篇。

3.5 几种模型的工程映射 ​

系统故障模型理由
KafkaCrash-recovery内部节点是自己人,不假设恶意
etcd / ZooKeeperCrash-recoveryRaft / ZAB 协议都假设非拜占庭
MySQL / PostgreSQLCrash-recovery单一集群内部信任
Redis ClusterCrash-recovery同上
Bitcoin / EthereumByzantine全球节点互不信任,必须容忍恶意
Tendermint / HyperledgerByzantine联盟链场景,部分节点可能作恶
Spanner / TiDBCrash-recoveryGoogle / PingCAP 内部信任

看一个分布式系统的文档,先找它假设什么故障模型——这决定了它能给你什么保证。假设 Crash 模型的系统不能放到不信任环境(联盟链不能用 etcd 做共识)、假设 Byzantine 模型的系统延迟和吞吐都差几个量级(区块链 TPS 几百 vs Raft 几万)。

四、时间模型(Timing Models) ​

故障模型回答"怎么坏",时间模型回答"消息延迟和处理时间有没有上限"。这一组词汇是 FLP 不可能定理(09 篇)的关键前置。

时间模型(从强到弱):

  Synchronous           消息延迟和处理时间都有已知上限
      ↓
  Partial Synchronous   存在上限,但你不知道具体值 / 偶尔会破
      ↓
  Asynchronous          没有任何时间假设

4.1 Synchronous(同步系统) ​

有严格的时间上限——消息一定在时间 T 内送达,节点处理一定在时间 P 内完成。

好处:可以用超时来"准确判断"节点是否挂了——超过 T+P 没回就是挂了,不是慢。

坏处:真实世界不存在严格同步的分布式系统——网络永远可能抖动,你给的上限永远可能被打破。

适用:专用网络环境——电信网络的硬实时控制、卫星通信、某些工业控制系统。

4.2 Asynchronous(异步系统) ​

没有时间上限——消息可以任意延迟,处理可以任意慢,你永远不能用超时判断"对方挂了"。

好处:最贴近真实互联网网络——没有任何"网络一定 X ms 内到"的保证。

坏处:FLP 定理证明,在纯异步系统 + 一个节点可能崩溃的前提下,共识不可能(09 篇详解)。

适用:严格的理论分析模型——证明"什么不可能",而不是工程实现。

4.3 Partial Synchronous(部分同步) ​

Dwork / Lynch / Stockmeyer 1988 年提出——存在时间上限 T,但你不知道具体值;或者只是"大部分时候"满足上限,偶尔会破。

实际含义:网络通常正常,偶尔抖动——这就是真实互联网的样子。

关键:几乎所有工程上能用的共识算法(Paxos / Raft / ZAB),都假设部分同步——理论上 FLP 说异步不可能,工程上靠"偶尔抖动结束后系统能收敛"绕过去。

       共识可达条件:

  Synchronous     → 共识总是可能 (但模型不真实)
  Asynchronous    → 共识不可能 (FLP)
  Partial Sync    → 共识"最终"可能 (Paxos / Raft 都靠这个)
                    抖动期间不可能 collapse 终止性
                    抖动结束后保证收敛

4.4 时间模型 + 故障模型 = 一个分布式问题的完整规约 ​

                    Crash-stop  Crash-recovery  Omission  Byzantine
Synchronous       │   简单      │   常见       │   可行   │  可行(开销大)
Partial Sync      │   FLP绕过   │ Paxos/Raft   │  可行    │  PBFT
Asynchronous      │   FLP 阻断  │   FLP 阻断   │  阻断    │  阻断

这张表告诉你:

  • 共识协议(Paxos / Raft)的精确假设是"Partial Synchronous + Crash-recovery"
  • 区块链协议(PBFT / Tendermint)的精确假设是"Partial Synchronous + Byzantine"
  • 任何号称"全异步 + 容错"的方案,要么是营销话术,要么是限定了 FLP 不影响的子问题(比如最终一致性)

六、故障检测的根本困难 ​

讲完模型,有一个深层问题必须挑明:在异步 / 部分同步系统里,你"永远"无法 100% 确定一个节点是真的挂了还是只是慢了。

节点 A 给节点 B 发请求,等了 10 秒没回:

  情况一:B 崩溃了
  情况二:B 还活着,但处理慢
  情况三:B 还活着,但 A 到 B 的网络断了
  情况四:B 还活着,B 的回复在路上(还没到 A)

  → A 无法区分这四种情况

这就是**"故障检测的不完美性"**——任何故障检测器都有 false positive(误报)和 false negative(漏报)。

Perfect Failure Detector (P):不误报、不漏报——异步系统中不可实现。 Eventually Perfect (◇P):最终不误报、不漏报(分区结束后收敛)——部分同步系统中可实现。几乎所有工程检测器都是 ◇P。

工程妥协:多次心跳超时(降低 FP)、Phi-accrual 输出"可信度"而非 0/1、Quorum 防脑裂(多数派同意才算挂)。这就是为什么 26 / 27 / 30 篇要花那么多笔墨——"如何在不完美的故障检测下安全做出破坏性决策",是分布式工程师每天面对的问题。

七、强观点:很多生产事故的根本原因 ​

95% 的"诡异"分布式 bug,都能映射到八大谬误的某一条 + 故障模型的某个边界条件:

现象根本原因对应谬误 / 模型
双主 / 脑裂网络分区 + Quorum 配置错谬误 1+5,Omission
数据丢失异步复制 + 主挂谬误 1,Crash-recovery
超时雪崩重试风暴 + 链路变长谬误 2+7
跨地域延迟部署假设单地域谬误 2+8

八、读完这一篇后该有的认知 ​

  1. 网络永远不可靠——八大谬误的核心,你的系统必须显式处理网络抖动、延迟、分区、安全问题
  2. 故障模型不是教科书概念,是协议的前置假设——读论文 / 文档时先确定它假设什么故障模型,才能判断它在你场景能不能用
  3. Crash-recovery 是工程主流,Byzantine 是金融区块链特例——选错模型,要么浪费成本(用 PBFT 做内部协调),要么暴露安全风险(用 Raft 做跨机构清算)
  4. 时间模型决定了能不能做共识——FLP 告诉你纯异步不行,工程都假设部分同步
  5. 不完美的故障检测是常态——所有"挂了"的判断都是概率,设计时必须假设有误判

九、踩坑提醒 ​

  1. 以为内网就安全——零信任时代,所有调用都要认证,无差别 mTLS
  2. 以为 TCP 可靠网络就可靠——TCP 给字节流可靠,连接断了重连后业务可能不可靠
  3. 本地开发没考虑跨机房 / 跨地域延迟——开发本地都 < 1 ms,上线发现跨机房 10 ms,链路超时雪崩
  4. 重试不加退避——所有客户端同时重试 → 服务端被 DDoS,重试必须 exponential backoff + jitter
  5. 超时设得太长或太短——太长 → 雪崩,太短 → 误判,不同链路不同超时
  6. 故障判定靠"一次心跳超时"——P99 网络可能 100 ms,正常心跳间隔太短就误判,用多次连续超时 + Phi-accrual
  7. 以为 Quorum 写就一定安全——少数派副本可能是旧的,读也要 Quorum 才能保证看到最新
  8. 用 Crash 模型协议跑在 Byzantine 环境——内部联盟链用 Raft,单个节点作恶就能改账本
  9. 以为"网络分区很罕见"——AWS / Azure 每月都有,生产代码必须容忍分区
  10. 以为最佳实践就万事大吉——所有大公司都按最佳实践做,事故照样发生——心态比知识重要

下一篇:03-一致性、共识、一致性模型.md——这三个词被混用了几十年,绝大多数面试 / 文档 / 文章都说错。我们把它们彻底分开:Consistency(一致性)是数据状态的相同程度,Consensus(共识)是节点达成一致的算法,Consistency Model(一致性模型)是程序员看到的契约。"Kafka 是强一致"为什么错、为什么 Linearizability 和 Sequential Consistency 完全不同、为什么单机内存模型(x86 TSO)和分布式一致性模型是同一套理论——读完再去看 Raft / Spanner / Riak 的论文,术语就不会再迷糊。
