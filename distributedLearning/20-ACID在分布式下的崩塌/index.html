    ACID 在分布式下的崩塌

    19 篇结尾说:强一致 → 共识,最终一致 → CRDT,中间地带 → 因果一致。这一篇换个视角:从「事务」这个所有后端工程师吃了十年的概念入手,看看单机引以为豪的 ACID,跨机后是怎么一个字母一个字母被打破的。这一篇是第五层的"开场白"——它解释了为什么后面 21-24 篇要讲那一堆方案:2PC / Saga / TCC / Percolator / Spanner 全是在补 ACID 的破口,每个方案补哪个字母、付什么代价,看完这篇你就有了底图。

    一句话先记住:ACID 在单机靠"一把锁 + 一个 WAL + 一次 fsync"撑住,跨机后 A 要靠 2PC、C 要靠全局约束(几乎做不到)、I 要靠串行化(代价巨大)、D 要靠多数派持久(单 fsync 不够)。要 ACID,就得 2PC + 共识,慢且脆;要可用,就得削弱某字母,降级成 BASE / Saga / 最终一致。Spanner 是当代最接近"分布式 ACID"的怪物——靠原子钟硬把外部一致性买回来,可大多数公司没这条件,只能在 ACID 和可用性之间反复横跳。

    一、单机 ACID:你以为是天经地义,其实是一台精密机器

    先回顾单机 ACID 怎么获得——只有理解了单机怎么搞,才能理解分布式怎么破。

    1.1 A(Atomicity,原子性)

    全做或全不做,不能做一半。

    实现:WAL(Write-Ahead Log,预写日志)+ Undo / Redo。

    执行事务 T:
  1. 写 WAL:T-BEGIN
  2. 写 WAL:T 修改了页 P,旧值 V1,新值 V2
  3. 在内存 buffer 中改页
  4. 写 WAL:T-COMMIT
  5. (异步)刷脏页到磁盘

崩溃恢复:扫 WAL
  - 已 COMMIT 的:redo
  - 未 COMMIT 的:undo

    A 的关键是 WAL 在 commit 前刷到磁盘——只要 WAL 上有 COMMIT 记录,事务就是 commit 了;否则就是没 commit,undo 掉。

    1.2 C(Consistency,一致性)

    数据库永远满足业务约束(主键、外键、CHECK、唯一索引)。

    实现:约束检查 + A I D 三个保证下自然导出。

    C 是"数据库视角的一致性",和 CAP 的 C(线性一致)是两个东西——这是分布式领域最容易混淆的两个概念,后面会专门强调。

    1.3 I(Isolation,隔离性)

    并发事务互不干扰。

    实现:锁 + MVCC。

    • 串行化(Serializable):两阶段锁(2PL)
    • 可重复读(RR):MVCC + 行锁
    • 读已提交(RC):MVCC + 短锁
    • 读未提交(RU):几乎不锁,脏读

    隔离级别越强,并发度越低——这是单机内部就有的取舍,分布式只是放大了它。

    1.4 D(Durability,持久性)

    commit 后即使断电也不丢。

    实现:fsync(WAL 写到物理磁盘 / SSD 的非易失介质)。

    # 简化的 commit
write_wal_record("T-COMMIT")
fsync(wal_fd)   # ← 这一行是 D 的命根子
return success

    没有 fsync,所谓 commit 只是 page cache 里的字节——掉电就没。

    1.5 单机 ACID 的物理基础

    ┌─────────────────────────────────────┐
│         单机 ACID 全靠              │
├─────────────────────────────────────┤
│  1. 一把 mutex(进程内全序保证)     │
│  2. 一个 WAL(顺序持久化)           │
│  3. 一次 fsync(物理介质持久化)     │
│  4. 一个内存页缓存(buffer pool)   │
└─────────────────────────────────────┘

    核心假设:有一个"共享内存 + 共享磁盘"的物理边界——所有线程在一个进程里,所有 IO 落到同一块磁盘。这个假设一跨机就死。

    二、跨机后,A 怎么破

    事务跨多台机器执行:转账,A 库扣 100,B 库加 100。

    T:
  begin
  A 库:UPDATE accounts SET balance -= 100 WHERE id = 'alice'
  B 库:UPDATE accounts SET balance += 100 WHERE id = 'bob'
  commit

    问题:commit 怎么"两边一起 commit"?

    • A 库 commit 成功,B 库 commit 失败 → 钱凭空消失
    • A 库 commit 失败,B 库 commit 成功 → 钱凭空多出
    • A 库 commit 成功,B 库 网络超时 → 不知道 B 库 commit 没有

    单机不会有这问题——单机的 commit 是一次 WAL 刷盘,要么成要么不成,没有"一半"状态。分布式有,因为两台机器各自有自己的 WAL,没有"一次刷"的物理动作。

    A 破的根源:单机靠"一次原子刷盘"得到 A,分布式没有这个原子动作——必须发明一个跨机的"伪原子动作"。这就是 2PC(21 篇)的来源。

    但 2PC 自身有命门:

    • 协调者挂了,参与者卡死(blocking forever)
    • 同步阻塞,延迟 = max(所有参与者)
    • 失败恢复极痛苦

    所以工程上经常放弃强 A,改用 Saga / TCC(22 篇)——业务级补偿:第二步失败就回滚第一步。这不是真 A,是"业务层 A"——中间状态业务可见,但最终结果"看起来像原子"。

    三、跨机后,C 怎么破

    全局约束跨节点几乎做不到。

    3.1 全局唯一

    分库分表:用户表按 user_id 分 16 片
约束:邮箱必须全局唯一

T1 在分片 3 插入:email = 'a@x.com'
T2 在分片 7 同时插入:email = 'a@x.com'
两个分片各自看本地,都允许 → 全局重复

    单机的 UNIQUE 索引救不了你——它只保证单库唯一,跨库要外加协调。

    怎么救:

    1. 路由层把同一约束的数据路由到同一节点——邮箱按 hash 分片,但这样查询不灵活
    2. 全局唯一服务(Redis SETNX、ZooKeeper 创建节点)——加一跳网络 + 强一致服务
    3. 乐观:先写后检测,冲突时回滚——业务复杂

    3.2 外键约束

    跨库外键 MySQL 直接不支持。业务层自己保证——但业务层写错了,数据库不会救你。

    3.3 业务级聚合约束

    "用户总订单数 ≤ 100"
订单分散在多个分片,无法在单事务里查完所有分片再判断

    DDD 的"聚合根"概念就是为了这个——把一致性边界限制在一个聚合内,跨聚合走最终一致。

    C 破的根源:全局约束需要"全局视图",但分布式系统连"全局视图"都难得到(看 18 篇因果一致那里)。工程妥协:把强约束限制在一个 shard / 聚合根内,跨边界走最终一致 + 对账兜底。

    3.4 注意:ACID 的 C ≠ CAP 的 C

    ACID 的 C CAP 的 C
    含义 业务约束不被破坏(外键 / 唯一 / CHECK) 多副本读写看到同一份数据(线性一致)
    范围 单事务内 全集群所有节点
    保证者 数据库引擎 + 应用 共识算法(Raft / Paxos)

    两个 C 完全不同——分布式领域的术语污染让无数人混淆。

    四、跨机后,I 怎么破

    串行化级别(SERIALIZABLE)代价巨大。

    4.1 单机串行化怎么做

    最朴素:全局表锁,事务排队。

    实用做法:2PL(两阶段锁)+ 死锁检测 或 SSI(Serializable Snapshot Isolation)+ 冲突检测。

    无论哪种,所有锁信息都在一个进程内——锁表查询 O(1),死锁检测扫一个图。

    4.2 跨机串行化的痛

    事务 T1 在节点 A 操作行 r1,节点 B 操作行 r2
事务 T2 在节点 A 操作行 r2,节点 B 操作行 r1

死锁!但锁信息分散在两个节点,谁来检测?

    工程方案:

    1. 全局锁服务(分布式 lock manager)——加一跳,延迟翻倍
    2. 2PC + 严格 2PL——commit 慢,死锁仍要检测
    3. OCC + 全局时间戳排序(Percolator / Spanner 路线)——23/24 篇详细讲

    4.3 实际上大多数分布式数据库的"最高"是 SI

    • MySQL 单机:最高 SERIALIZABLE
    • TiDB / CockroachDB:默认 SI(快照隔离),可选 SSI
    • Spanner:外部一致性 = 线性化 + SS2PL(代价是 TrueTime + 等待区间)
    • MongoDB / Cassandra:连 SI 都不到,应用层管隔离

    为什么不普及 Serializable?——分布式 SS2PL 的延迟和死锁代价,绝大多数业务接受不了。SI 已经能挡掉 95% 的并发问题,剩下 5% 业务自己加 SELECT FOR UPDATE 或乐观锁。

    I 破的根源:跨节点的锁管理 / 冲突检测,代价远大于单机。工程妥协:用 SI 代替 SERIALIZABLE,把"幻读"等小漏洞交给应用。

    五、跨机后,D 怎么破

    D 看起来最简单(刷盘就行),其实最容易踩雷。

    5.1 单 fsync 不够

    单机:write_wal + fsync → 数据持久

分布式:在 leader 上 fsync 了 → 数据是不是持久?

NO!
1. 这台 leader 接着挂了,磁盘坏了 → 数据没了
2. 这台 leader 接着挂了,SSD 还在 → 但被踢出集群,选了新 leader → 新 leader 没这个事务 → 实际上"丢"了

    真正的分布式 D 要求:事务已被多数派持久化(majority durable),才能算 commit。

    5.2 各种半同步 / 同步复制的真相

    模式 写入路径 何时返回 commit D 保证
    MySQL 异步 主写 WAL + fsync → 返回 主 fsync 完 主挂可能丢
    MySQL 半同步 主写 + 至少 1 个从收到 binlog → 返回 主 fsync + 1 个从内存收到 从挂可能丢(从没 fsync)
    MySQL Loseless 主写 + 至少 1 个从 fsync 主 + 1 从都 fsync 主和这从同时挂才丢
    PG 同步复制 主写 + N 个从 fsync(可配) N 从都 fsync 多数派持久
    Raft / Paxos 多数派持久后 leader 返回 majority fsync 多数派持久

    "我 commit 返回了"≠"数据安全"——必须问"几台机器持久了"。没有多数派持久化的 commit,都是单点 D——单点挂就完。

    5.3 真实事故:MySQL "异步复制 + 自动 failover"

    经典翻车场景:

    1. 主库写入 commit 返回成功
2. binlog 还没传给从库
3. 主库挂了 → 自动 failover,从库提升为主
4. 客户端读不到刚才那笔 → 用户傻眼

    这不是 MySQL bug,这是异步复制的本质 ——D 只在主库的磁盘上,没多数派持久。金融场景必须半同步 + 多数派,或者用 TiDB / CockroachDB / Spanner 这种原生多数派持久的系统。

    D 破的根源:单机 D = 单磁盘 fsync,分布式 D = 多数派持久——两者完全不同,不能混用。

    六、根本矛盾:ACID vs 可用性

    回到 04 篇 CAP:强一致(ACID)和可用性是天然冲突的。

    ACID 要求事务跨节点协调 → 任一节点不可达,事务失败 → 可用性下降
可用性要求任何分区下都能服务 → 不等其他节点 → 不一致

要 ACID  → 2PC + 共识 → 慢、且脆(协调者挂、参与者挂、网络抖)
要可用  → 削弱某字母 → BASE / Saga / 最终一致

    6.1 两条工程路线

    路线 1:强 ACID(NewSQL)
└─ Spanner / TiDB / CockroachDB / OceanBase
   - 2PC + Paxos/Raft + 全局时钟
   - 牺牲少量可用性(CP)、运维复杂、延迟高
   - 适合金融账户 / 强一致核心库

路线 2:BASE(互联网架构主流)
└─ MySQL 分库 + 应用层补偿 + 消息事务表
   - Saga(长事务) / TCC(预占 + 确认 / 取消)
   - 业务级 A、最终一致
   - 适合电商订单 / 社交 / 内容

    95% 的国内互联网公司用路线 2——理由是路线 1 运维成本高、改造历史系统贵。但金融核心、计费、对账,该上 ACID 还是要上。

    6.2 ACID vs BASE vs NewSQL 对比

    维度 单机 ACID BASE(应用补偿) NewSQL
    一致性 强 最终 强(SI / 线性)
    可用性 低(单点) 高 中(多数派可用)
    延迟 低 低(本地操作) 中(跨节点 RTT)
    复杂度 低 高(写补偿、对账) 中(数据库自己处理)
    运维 低 中 高
    适合 单库小业务 互联网通用 金融 / 强一致核心

    七、Spanner / TiDB:"分布式 ACID"是怎么硬买回来的

    Spanner 2012 论文核心一句话:"我们提供外部一致性,代价是 GPS + 原子钟"。

    7.1 Spanner 怎么做到 ACID

    - A:2PC(跨 Paxos group)
- C:数据库约束 + 串行化
- I:SS2PL(严格两阶段锁)+ MVCC,**TrueTime 提供线性化时间戳**
- D:每个 Paxos group 多数派持久

    关键创新是 TrueTime API:

    TT.now() → TTinterval { earliest, latest }   // 注意是区间不是点
TT.after(t) → bool                            // t 之前是否一定过去了
TT.before(t) → bool                           // t 之后是否一定还没到

    事务 commit 时等待"绝对最近的时间戳"过去(commit-wait),保证所有副本看到的时间戳全序。

    代价:

    • 每个数据中心装原子钟 + GPS(物理硬件)
    • 每个 commit 多等 7ms(commit-wait)
    • 5 个 9 可用性,但仍然是 CP——分区时少数派不可用

    只有 Google 这种基建公司玩得起。

    7.2 TiDB / CockroachDB 怎么"穷人版"做到

    没有原子钟,用 HLC + 中心化 TSO(Timestamp Oracle)代替 TrueTime:

    • TiDB:PD 集群单点发号(逻辑上集中,实际 Raft 高可用)
    • CockroachDB:HLC + uncertainty window,自动回滚冲突事务

    事务模型:Percolator(23 篇)——快照隔离 + 主键锁。

    代价:

    • 每个事务跨网络至少 2 个 RTT(取 TS + commit)
    • TSO 单点瓶颈(虽然有缓解)
    • 大事务慢

    TiDB / CockroachDB 是"准 ACID"——单数据中心强一致,跨地域降级。Spanner 是"全球 ACID 但有 commit-wait"。这两个挡掉了 99% 的工业需求,但代价仍然是单机 ACID 的几十倍延迟。

    八、MySQL 主从 ≠ 分布式 ACID:一个常见的幻觉

    初级工程师最大的误区:以为"我的 MySQL 是主从 + 半同步,所以是分布式 ACID"。

    不是。让我们逐条破:

    8.1 A 的幻觉

    跨库事务?XA(MySQL 内置)——21 篇会讲,是标准 2PC,生产几乎没人用(性能差、容易卡死)。大多数 MySQL 主从根本不做跨库事务——A 是单库 A,跨库 A 靠应用层补偿。

    8.2 C 的幻觉

    跨库唯一约束?做不到——分库的 UNIQUE 只保证本分片唯一。全局唯一靠 Redis SETNX 或专门发号器。

    8.3 I 的幻觉

    从库读到的数据滞后——即使你在主库 commit 了,从库可能还没拿到 binlog。从库读 + 主库写 = 已经破坏了线性一致。

    8.4 D 的幻觉

    半同步只保证 binlog 传到 至少一个从库,不等从库 fsync。从库收到但没 fsync 时挂了 → 数据丢。

    真正的多数派持久,要么 Loseless 半同步(MySQL 5.7+),要么 Group Replication / InnoDB Cluster。

    这是为什么互联网公司不真的把 MySQL 主从当强一致用——主从只解决"读扩容 + 一定的高可用",不解决"ACID"。强 ACID 需求要么单库吃,要么上 NewSQL,要么应用层 Saga。

    九、工程选型:什么场景该怎么选

    按业务场景给出选型矩阵:

    场景 强 A 需求 跨库否 推荐
    余额 / 账户 / 计费 高 否(单库) MySQL / PG 单库事务
    余额 + 跨地域 高 是 Spanner / OceanBase / TiDB
    订单 + 库存(跨服务) 中 是 TCC(预占 + 确认)
    订单流程(创建 → 支付 → 发货) 中 是 Saga
    用户注册(跨用户中心 + 积分) 中 是 消息事务表 + 异步补偿
    点赞 / 关注 低 是 CRDT / 最终一致
    数据同步到 ES / 缓存 极低 是 binlog 监听 + 异步

    判断口诀:

    1. 数据 "错一笔会上新闻" 吗? → 强 ACID,上 NewSQL 或单库
    2. 数据 "中间不一致几秒能接受" 吗? → BASE,Saga / 消息
    3. 数据 "并发更新天然可合并" 吗? → CRDT
    4. 数据 "几乎不并发" 吗? → 单库够了,别复杂化

    9.1 DDD 视角:聚合根 = 一致性边界

    DDD 的"聚合根"是给分布式事务的最大馈赠——它定义了"哪个范围内强一致,哪个范围外最终一致"。

    聚合内(单数据库):ACID
聚合间(跨服务、跨库):最终一致 + Saga / 事件

如:
- 订单聚合(订单 + 订单项):一个事务
- 库存聚合(库存项):独立事务
- 跨这两个:Saga / TCC / 消息

    不分清"哪些数据要在一个事务里",一上来就 2PC 全跨,系统必然慢且脆。先用 DDD 把一致性边界画好,然后边界内 ACID,边界外最终一致——这是工程上最实用的方法论。

    十、踩坑提醒

    1. 以为 MySQL 主从 = 分布式 ACID——主从只解决读扩容 + HA,跨库事务靠 XA(几乎没人用)或应用补偿
    2. 混淆 ACID 的 C 和 CAP 的 C——一个是业务约束,一个是多副本读写一致,完全不同
    3. 跨服务事务上 2PC——99% 场景该用 Saga / TCC,2PC 性能差且脆
    4. 以为半同步 = 强 D——半同步只保 binlog 到达,不等从库 fsync,要用 Loseless 模式
    5. 强一致全部走 SERIALIZABLE——大多数业务 SI(快照隔离)就够了,真要 Serializable 性能踩烂
    6. 没有对账兜底——最终一致系统必须有对账作业,任何"事件驱动"系统不对账都是定时炸弹
    7. 业务约束写到应用层不写数据库——能在数据库做的 UNIQUE / FK / CHECK 一定在数据库做,应用层守门必有漏
    8. 跨聚合事务不写补偿——Saga 的核心就是补偿,没补偿就是裸事务,出错只能人工救火
    9. 忽略幂等——补偿事务 / 重试都要幂等,靠业务唯一键 + 状态机
    10. 以为 NewSQL 等于 ACID 免费——TiDB / CockroachDB 跨节点事务比单库慢 5-10 倍,热点行高并发仍然踩坑(用乐观锁需要处理大量重试)

    收束

    ACID 是单机时代的胜利,它把"原子操作"从硬件抬到了应用——开发者不必关心物理介质和并发原语,只要 BEGIN / COMMIT。

    分布式让这个胜利破了功。A 破,C 破,I 破,D 破——每个字母都要重新发明一遍:

    • A 的重发明 → 2PC / 3PC(21 篇) / Saga / TCC(22 篇)
    • C 的重发明 → 全局唯一服务 + DDD 聚合边界
    • I 的重发明 → SI / SSI / Percolator(23 篇)
    • D 的重发明 → 多数派持久 → Paxos / Raft(11-13 篇)

    集大成是 Spanner(24 篇)——用 TrueTime 和 Paxos 硬把 ACID 拼回来。但 Spanner 之外的世界,大家都在和 ACID 妥协,选哪个字母让一步、让多少、用什么补,这就是分布式事务工程的全部艺术。
