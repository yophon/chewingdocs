    服务发现与配置中心

    微服务架构最先冒出来的问题不是"怎么拆",而是"拆完之后,我怎么知道你在哪"。订单服务有 30 个实例,某个实例刚崩了,支付服务怎么不把请求打到那个死实例上?生产环境每天有几百次实例上下线、扩缩容、滚动发布,消费者必须实时知道"谁还活着、在哪个 IP"——这就是服务发现的使命。这一篇把 ZooKeeper / etcd / Consul / Eureka / Nacos 五大方案掰开,讲清楚 CP vs AP 的选型分水岭、为什么 K8s 选 etcd、为什么 Spring Cloud 选 Eureka、Netflix 那个臭名昭著的"自我保护机制"到底救人还是坑人。

    一句话先记住:服务发现解决「消费者怎么找到生产者」——核心是「注册中心」存着「服务名 → 实例列表」的映射,实例上下线时实时更新。最大的选型分歧是 CP 还是 AP——金融 / K8s 控制面要 CP(宁可暂时找不到也别给错地址),互联网业务多数要 AP(注册中心挂了不能拖累全局)。配置中心和服务发现常合并(Consul / Nacos / Apollo)——因为底层都是"watch + 推送"的同一套。

    一、不用服务发现的世界长什么样

    最原始的方案:配置文件硬编码 IP。

    # order-service.yml
payment_service:
  hosts:
    - 10.1.2.3:8080
    - 10.1.2.4:8080
    - 10.1.2.5:8080

    问题清单:

    1. 某台机器挂了,要改配置 + 重启所有调用方
    2. 扩容了一台新机器,要改配置 + 重启
    3. 跨环境部署(dev/test/prod IP 不同),要维护多份配置
    4. K8s 调度的 Pod IP 每次都变,配置文件追不上

    结论:只要业务超过 5 个服务、超过 10 台机器,硬编码就死路一条。

    1.1 加个 LB 也不够

    order-service → 调用 nginx:8080 → 转发到后端实例

    Nginx 解决了"客户端不感知 IP"的问题,但 Nginx 自己怎么知道后端实例列表?——还是要有人把"实例上下线"告诉 Nginx,这件事就是服务注册中心干的。

    Nginx / Envoy / 网关都是"服务发现的客户端",真正的核心是注册中心。

    二、客户端发现 vs 服务端发现

    两种工作模式,先决定走哪条:

    2.1 客户端发现(Client-Side Discovery)

    注册中心(Eureka / Nacos)
        ▲
        │ 拉取实例列表
        │
  ┌─────┴─────┐
  │ 客户端 SDK │ → 自己挑一个实例 → 直接调
  └───────────┘

    代表:Netflix Eureka + Ribbon、Nacos + Spring Cloud Alibaba。

    优点:

    • 无中间层,网络一跳到位
    • 客户端可以自定义负载均衡策略(权重、地域亲和、灰度规则)

    缺点:

    • 客户端 SDK 要更新——多语言成本高
    • 客户端逻辑变重(健康检查、负载均衡、重试都自己写)

    2.2 服务端发现(Server-Side Discovery)

                                                    ┌── 实例 1
客户端 → 调用 service-name → 网关/LB / Service ─┼── 实例 2
                                  ↑              └── 实例 3
                                  │
                            注册中心(etcd / Consul)

    代表:K8s Service(kube-proxy + iptables/IPVS)、Nginx + Consul Template、Envoy + xDS、Istio。

    优点:

    • 客户端无感——只调用一个固定地址(service-name.namespace.svc.cluster.local)
    • 多语言天然支持(客户端不需要 SDK)
    • 路由策略集中管理(在网关 / sidecar 层)

    缺点:

    • 多一跳(虽然 K8s iptables/IPVS 是 kernel 级,几乎无延迟)
    • 网关 / sidecar 自身的可用性是新单点

    K8s 选了服务端发现(Service + Endpoints + kube-proxy),Spring Cloud 早期选了客户端发现(Eureka + Ribbon),Service Mesh 时代 sidecar(Envoy)是"客户端发现的服务端化"——逻辑下沉到 sidecar,业务代码看起来还是客户端发现。

    三、注册中心的核心功能

    不管哪种模式,注册中心都要做这三件事:

    1. 注册   :服务启动时把自己("我是 order-service,IP=10.1.2.3:8080")写入注册中心
2. 健康检查:周期性确认实例是否还活着(心跳 / TTL / 主动 probe)
3. 通知   :实例变化时通知消费者(watch / 长轮询 / 推送)

    3.1 健康检查的三种实现

    方式 工作机制 失效感知速度 代表系统
    临时节点 + Session 实例与 ZK 维持 session,session 断了节点自动删 秒级(session timeout) ZooKeeper
    TTL Lease 实例周期续约,过期自动失效 秒级(lease TTL) etcd
    心跳上报 实例定期 PUT /heartbeat 30-90 秒(默认) Eureka
    主动 Probe 注册中心主动 ping 实例 自定义周期 Consul Health Check

    临时节点和 TTL Lease 本质相同——都是"靠会话保活,断线就清除",前者是 ZK 的术语,后者是 etcd 的术语。Eureka 心跳上报最宽松,所以失效感知最慢。

    3.2 变更通知的两种实现

    方式 工作机制 代表
    Watch / 长连接 客户端订阅,服务端有变更立刻推 ZK / etcd / Nacos
    Long Polling 客户端发请求挂着,服务端有变更立刻返回 Consul / Apollo
    客户端拉取 客户端定时来问 Eureka(默认 30 秒)

    Watch 最快但服务端压力大(每个客户端一个长连接),Long Polling 折中,定时拉取最简单但变更感知慢——Eureka 30 秒间隔意味着"实例挂了之后最多 30 秒消费者还在用老地址"。

    四、五大主流方案对比

    系统 一致性 协议 主要用途 健康检查 变更通知 代表使用方
    ZooKeeper CP ZAB 强一致协调 / 配置 / 选主 临时节点 + Session Watch Kafka / HBase / Dubbo(旧版)
    etcd CP Raft K8s 控制面 / 配置 / 服务发现 TTL Lease Watch K8s / CoreDNS / Rook
    Consul CP(Raft) + AP(Gossip) Raft + Memberlist 服务发现 + 健康检查 + KV Gossip + Health Check Long Polling HashiCorp 生态
    Eureka AP 自研(P2P 复制) Spring Cloud 服务发现 心跳上报 客户端拉取 Netflix(已停止维护)
    Nacos AP(默认)/CP(可切换) Distro(AP) + Raft(CP) 阿里系微服务发现 + 配置 心跳 / 主动 Probe 推送(UDP/长连接) 阿里 / 国内中小公司

    4.1 ZooKeeper:老牌强一致协调器

    • 共识协议:ZAB(类 Paxos,见 14 篇),写要多数派
    • 数据模型:树形 znode,临时节点表示活实例
    • 强项:强一致 + Watch 机制可靠,适合配置 / 选主
    • 弱点:
    • 部署运维复杂(JVM,需要单独的 ZK 集群)
    • 写性能差(每次写要多数派确认),不适合频繁注册场景
    • CP 系统,分区时少数派不可用——业务感知:跨机房分区的少数派那侧服务发现挂掉
    • 典型用法:Kafka 用它存元数据、Dubbo 早期版本用它做注册中心、HBase 用它做 master election

    4.2 etcd:K8s 时代的赢家

    • 共识协议:Raft(13 篇),Go 写的,接口 gRPC
    • 数据模型:KV(扁平 key,可加前缀过滤)
    • 强项:简洁 + Go 生态 + Raft 易理解,Watch 接口设计干净
    • 弱点:
    • 仍然是 CP,跨机房分区有少数派不可用问题
    • 大集群下 Raft 写吞吐有限(< 10K QPS)
    • 典型用法:K8s 控制面唯一持久化存储——所有 Pod / Service / Deployment 都进 etcd

    为什么 K8s 选 etcd 不选 ZK:Google 内部 Borg 用 Chubby(类 ZK),K8s 团队最初也想用 ZK,但 ZK 的 Java 依赖 + 复杂 API + 弱社区文档让 K8s 团队转向 etcd——Go 写的、API 简洁、Raft 比 Paxos 易懂、CoreOS 团队主导(后被 RedHat 收购)。这一选改变了云原生十年的技术栈。

    4.3 Consul:服务发现的全栈方案

    HashiCorp 出品,思想是分层:

    控制面: Raft(CP)→ 服务注册数据、KV 配置
数据面: Gossip / SWIM(AP)→ 成员管理、故障检测
    • 强项:
    • 服务发现 + 健康检查 + KV 配置 + DNS 接口(直接 dig 查实例)一体
    • 跨数据中心原生支持(WAN Gossip)
    • 健康检查比对手强大(HTTP / TCP / Script / TTL)
    • 弱点:
    • 部署稍重(每节点一个 agent + 中心 server)
    • 国内用得不多(社区生态偏英文世界)

    典型用法:HashiCorp 生态(Nomad / Vault / Terraform)的协调中枢,中型互联网公司选它做注册 + 配置一站式。

    4.4 Eureka:Spring Cloud 的伴生品

    Netflix OSS 出品,Spring Cloud Netflix 时代的事实标准:

    • AP 设计:节点之间 P2P 复制,任何一个节点都能读写,牺牲一致性换可用性
    • 强项:
    • 简单(Java 写,Spring Boot 应用)
    • 客户端 SDK 成熟(Ribbon + Hystrix 组合拳)
    • AP 设计在大集群下不会因为少数派挂掉影响业务
    • 弱点:
    • 臭名昭著的"自我保护机制"(下面单独讲)
    • 数据不强一致——同一时刻不同 Eureka 节点看到的实例列表可能不同
    • 客户端默认 30 秒拉一次,实例下线后最多 30 秒还在用老地址
    • Netflix 已经停止 Eureka 2.0 开发,1.x 进入维护模式

    4.5 Nacos:阿里出品,国内主流

    • 特性:支持 AP 和 CP 两种模式可切换(默认 AP)
    • 强项:
    • 服务发现 + 配置中心一体
    • 推送机制(UDP + 长连接),变更感知快
    • 支持权重 / 灰度 / 多环境
    • 中文文档 + 阿里背书,国内中小公司大量采用
    • 弱点:
    • 早期版本 bug 多(2.x 起稳定)
    • AP/CP 切换语义复杂,不少团队踩坑

    典型用法:国内 Spring Cloud Alibaba 体系的注册 + 配置中心(替代 Eureka + Apollo)。

    五、CP vs AP:服务发现的核心选型分水岭

    这是整个服务发现领域最核心的选型决策:

    5.1 CP 派的逻辑

    "宁可暂时找不到服务,也别让消费者拿到错误地址"。

    • 注册数据强一致——所有客户端任意时刻看到的实例列表完全相同
    • 分区时少数派不可用——少数派 Eureka 客户端读不到注册中心,拒绝服务而不是返回旧数据

    适合:

    • K8s 控制面(Pod 调度不能容忍多源不一致)
    • 配置数据(一份配置变更要立刻全网生效)
    • 金融 / 银行系统(实例列表错就可能调错账户)

    5.2 AP 派的逻辑

    "注册中心挂了不能拖累全局,业务自己用旧地址试一下,挂了重试"。

    • 注册数据最终一致——不同节点看到的实例列表可能短暂不同
    • 分区时两边都能读写——少数派也能返回(可能过期的)实例列表

    适合:

    • 互联网业务(注册中心 1 秒不可用就影响几百万 QPS,完全不能接受)
    • 服务实例多(几千个),频繁上下线
    • 业务侧已有重试 + 熔断兜底(详见 systemDesign/11/12 篇)

    5.3 主流选择

    场景 主流选 理由
    互联网 To C 业务 AP(Eureka / Nacos AP) 注册中心不能拖累业务,旧地址试一下大不了重试
    K8s 控制面 CP(etcd) 调度数据必须强一致
    阿里 / 腾讯内部 AP(Nacos / 自研) 大规模,容忍最终一致
    银行 / 支付核心 CP(ZK / etcd) 宁可降级,不要错调
    中间件元数据(Kafka 早期) CP(ZK) broker 拓扑必须一致

    Patrick Eugster 一句话:"Distributed systems are hard. Choose your impossible carefully."——CP 还是 AP,不是技术问题,是业务可以容忍什么"不可能"的问题。

    六、Eureka 的自我保护机制:救人还是坑人?

    Eureka 有个出了名的争议特性:自我保护机制(Self-Preservation Mode)。

    6.1 它做什么

    正常情况:
  实例失去心跳 → Eureka 在 90 秒后剔除它

自我保护触发:
  Eureka Server 看到"15 分钟内心跳数掉了 85% 以上"
  → 进入自我保护模式
  → 不再剔除任何实例(即使它们已经死了)
  → 直到心跳数恢复

    6.2 它的初衷

    Eureka 不信任自己——心跳大量缺失可能是 Eureka Server 网络故障(不是实例真死了)。这种情况下剔除实例会导致大量误踢,业务受影响。所以宁可保留死实例(消费者重试就行),也别误踢健康的。

    6.3 它的坑

    真的有大批实例死了(机房断电、Pod 批量崩溃),Eureka 也以为是自己网络问题,死实例留在列表里几小时,消费者持续往死实例发请求,熔断器没配好就雪崩。

    Netflix 自己的事故:某次大批实例真死了,Eureka 自我保护拒绝剔除,导致下游持续报错,最终运维手动关闭自我保护才恢复。

    教训:"宁可保留过期数据"在大多数情况下是对的,但在大规模批量故障时就是错的。AP 的代价在异常场景下会被放大,业务侧必须有客户端健康探测 + 熔断作为兜底——别只信注册中心。

    七、配置中心:服务发现的"近亲"

    配置中心解决的是"动态配置怎么不重启就推到所有实例":

    开发改了限流阈值 100 → 200
  → 推到配置中心
  → 所有实例 watch 到变更
  → 立刻应用新值,不重启

    7.1 为什么和服务发现常合并

    底层都是"watch + 推送"的同一套机制:

    • 服务发现:watch "service-name 的实例列表"
    • 配置中心:watch "config-key 的值"

    底层都是 KV 存储 + 长连接通知——一套基础设施同时干两件事。

    典型合并方案:

    • Consul:KV 存配置 + 服务发现一体
    • Nacos:配置 + 注册一体
    • etcd:K8s ConfigMap + Service 都在 etcd 里
    • Apollo:纯配置中心,不做服务发现(携程开源)

    7.2 配置中心的额外特性

    普通 KV 不够,配置中心通常还要:

    • 灰度发布:配置先推 1% 实例,观察一会儿再推全量
    • 多环境隔离:dev / test / prod 各自一份
    • 回滚:版本历史,一键回到上一版
    • 审计:谁改的、什么时候、为什么(合规要求)
    • 权限:某些配置只能特定人改

    Apollo / Nacos / etcd + 工具链都能做,直接用 etcd 裸 KV 不够——这就是为什么 K8s 还需要 ConfigMap / Operator 之类的上层封装。

    7.3 配置中心架构(以 Apollo 为例)

        管理后台(配置编辑)
         │
         ▼
    ┌──────────┐  ←─── 异步通知
    │ ConfigDB │       ConfigService
    └──────────┘
         │
         │  watch / long polling
         │
   ┌─────┴─────┐
   │  客户端 SDK │ → 缓存本地配置 → 应用代码读
   └───────────┘

    Apollo 的设计点:

    • 客户端本地缓存配置——配置中心挂掉,业务用本地缓存继续跑
    • 长轮询而非长连接——简化部署
    • 配置变更秒级到客户端

    八、为什么 K8s 选 etcd 而不是 ZK / Consul

    K8s 早期讨论过用 ZK,最终选 etcd,核心原因:

    8.1 技术原因

    • Go 同源:K8s 是 Go 写的,etcd 也是 Go 写的,SDK 无缝集成
    • Raft 比 ZAB / Paxos 易懂:K8s 团队需要 hack 共识层(成员变更、租约、压缩等),Raft 文档+实现成本低
    • API 简洁:etcd 是 gRPC + KV,比 ZK 的 znode tree 简单
    • Lease + Watch 模型适合 K8s:每个 Pod 一个 lease,挂了自动清除,Endpoints 控制器 watch 到变更

    8.2 工程原因

    • CoreOS 团队主导(后被 RedHat 收购),配合 K8s 路线图深度优化
    • 社区文化:CoreOS / RedHat 生态偏云原生,Apache(ZK)偏传统 Java
    • 没有 JVM 依赖:JVM 启动慢、内存大,对控制面这种"必须稳定"的组件 Go 单二进制更友好

    8.3 反面教材:K8s etcd 挂了的事故

    etcd 是 K8s 的单点——etcd 全挂等于整个控制面失控:

    • 不能调度新 Pod
    • 现有 Pod 还能跑(kubelet 有本地缓存)
    • 不能创建/删除任何资源

    多次真实事故:

    • etcd 磁盘满:某公司日志写到 etcd 的 PV,disk 100%,etcd 不可写,K8s 失控
    • etcd 网络分区:跨机房 etcd 集群分区,少数派那侧 API Server 拒绝写入
    • etcd 内存爆:大集群(>5000 节点)Watch 客户端太多,etcd 内存吃满

    解决思路:

    • etcd 独立物理资源(别和业务混部)
    • 多实例 + 跨 AZ
    • 监控 etcd 的 leader 切换、写延迟、磁盘使用
    • 大集群拆分——超过 5000 节点考虑拆多套 K8s 集群,而不是一套硬撑

    九、生产真实事故案例

    9.1 Netflix Eureka 自我保护留死实例

    某次 Netflix 实例批量崩溃,Eureka 自我保护拒绝剔除,下游服务持续把请求打到死实例,雪崩 30 分钟。Netflix 内部反思:自我保护是好心办坏事的典型,要配合客户端主动健康探测——别只依赖注册中心剔除。

    9.2 K8s etcd 挂导致整集群失控

    某互联网公司,etcd 三节点其中两台机器同一时间硬盘故障(同一批次磁盘,寿命到期),etcd 失去多数派,API Server 全部 503,几小时内不能扩缩容、不能发布、不能调度新 Pod。根因是 etcd 多节点放在同一批硬件——后来改为跨 AZ 部署。

    9.3 ZK 跨机房分区导致 Dubbo 全网瘫痪

    某公司 Dubbo + ZK 注册中心,ZK 集群跨两个机房(2+3 节点)。机房间网络断了,少数派机房的 Dubbo 客户端(2 节点 ZK 那边)读不到注册中心,全部服务调用失败。根因:ZK 是 CP,分区时少数派强一致地拒绝读——业务应该用 AP 或者跨机房 ZK 部署应避免 2+3 这种"小机房会失联"的拓扑。

    9.4 Nacos AP/CP 切换导致数据丢失

    某团队把 Nacos 从 AP 切换到 CP,切换过程中正在写的注册数据丢了——AP 模式的 Distro 协议和 CP 模式的 Raft 协议存储不互通,切换不是热切换。根因:对 Nacos 双模式理解不够深,生产环境别频繁切。

    十、踩坑提醒

    1. 混淆 CP/AP 选型语义——业务能接受"实例列表偶尔不一致"就用 AP(Eureka/Nacos),不能接受就用 CP(etcd/ZK)。互联网业务 90% 该选 AP。
    2. K8s 集群外部还往 etcd 塞数据——etcd 是 K8s 的私有存储,业务往里写 KV 是大忌,容易挤爆 etcd 影响控制面。业务配置走单独的 Consul / Nacos / Apollo。
    3. Eureka 不关自我保护就上生产——批量故障时它会拒绝剔除死实例,配合熔断器 + 客户端主动健康探测才稳。
    4. 配置中心当 KV 数据库用——配置中心是给"低频改、高频读"的配置用的,频繁写会把 etcd / ZK 拖死。业务运行数据走 Redis / MySQL。
    5. 不缓存注册数据——客户端不缓存,注册中心一抖就全网调用失败。SDK 必须本地缓存 + 异步刷新,注册中心挂了还能用最后一份。
    6. 健康检查只检查端口存活——TCP 端口活着但应用线程死了(死锁、OOM 边缘),必须 HTTP /health 检查应用真实状态。
    7. 跨机房 ZK 部署成 2+3——少数派机房网络分区时直接全部不可用。3 机房 1+1+1 或 5 节点跨 3 机房 2+2+1才稳。
    8. Eureka 心跳超时设得太长——默认 90 秒,死实例可能 1.5 分钟才被剔除,业务感知差。改 30 秒或 45 秒。
    9. 配置变更不灰度直接全网推——一个错误配置(限流阈值、超时时间)推到所有实例,全部业务在 10 秒内崩。配置中心必须支持灰度推送,先 1% 实例观察。
    10. 服务实例下线不优雅——直接 kill -9 实例,注册中心几秒内才感知,期间客户端继续打过来。优雅下线:实例先调 deregister API → 等 N 秒(让消费者刷新缓存)→ 再退出。

    下一篇 30-故障检测与脑裂:服务发现解决了"谁活着,在哪"的元数据问题,但"谁真的活着"是个更深的问题——网络抖动让 A 看 B 死、B 看 A 死,怎么不误判?φ-accrual 故障检测 + Quorum 防脑裂 + Fencing Token + STONITH,收尾篇把整套"分布式判生死"的工具讲完,然后告诉读者 30 篇分布式理论的底层图景已经完整。
