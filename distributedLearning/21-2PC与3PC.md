# 2PC 与 3PC:分布式事务的鼻祖与它的两个致命问题

20 篇说**ACID 跨机后每个字母都破,A 破要靠 2PC 补**。这一篇就把 2PC 拆到底:**它怎么工作、为什么有用、为什么生产上几乎没人用**。**2PC 是分布式事务的基础范式**——后面 Saga / TCC / Percolator / Spanner 看似五花八门,**本质都是"2PC 减配"或"2PC 变种"**——所以即使你这辈子不会上线一个 XA,也必须把 2PC 啃透,**否则后面那些方案你只能背名字、看不到它们在补哪个洞**。

> 一句话先记住:**2PC = 协调者发起的 vote-then-commit 投票,目标是"全 commit 或全 abort"——但有两个永远绕不开的命门:同步阻塞和协调者单点不确定状态**。**3PC 加了一轮 PreCommit 试图绕过协调者单点,代价是多一轮 RTT,且分区下仍不安全**——所以**生产上 2PC 已经稀有,3PC 几乎绝迹**,真正活下来的是 2PC 的变种(Percolator / Spanner)和它的替代(Saga / TCC)。

---

## 一、为什么需要 2PC:跨机原子提交问题

回到 20 篇的痛点:转账分两边,**A 库扣 100 + B 库加 100,要么都成功,要么都失败**。

朴素方案 1:**先扣 A,再加 B**——A 成功后 B 失败,钱消失。
朴素方案 2:**先加 B,再扣 A**——B 成功后 A 失败,凭空多钱。
朴素方案 3:**两边同时发 commit**——任一边失败都不可救。

**核心矛盾**:**单机的 commit 是"一次原子刷盘",跨机没这个动作**。两台机器各自的 commit 是独立的物理事件,**网络不允许"同时发生"**。

### 1.1 2PC 的核心思想

**把"commit 决策"和"commit 执行"分开**:

- **阶段 1(Voting / Prepare)**:协调者问每个参与者"**你能不能 commit?**"——参与者本地把事务做到"差最后一步刷 commit",回复 YES / NO
- **阶段 2(Commit / Abort)**:协调者收齐回复,**只要有一个 NO 就全 abort,全 YES 才全 commit**——把决策广播给所有参与者

**关键不变量**:**只有所有参与者都 vote YES,协调者才会发 COMMIT;参与者一旦 vote YES,就必须能 commit**(已经把事务做到只差最后一步)。

### 1.2 角色

```
┌──────────────────┐
│   Coordinator    │  ←── 协调者(发起事务的节点,或专门的 TM)
└────────┬─────────┘
         │
   ┌─────┼─────┐
   ↓     ↓     ↓
┌─────┐┌─────┐┌─────┐
│ P1  ││ P2  ││ P3  │  ←── 参与者(实际执行的节点 / 数据库)
└─────┘└─────┘└─────┘
```

实际工程里:

- 协调者 = 业务服务、专门的事务管理器(TM,如 Atomikos / Seata TC)
- 参与者 = MySQL / Oracle / 消息队列(每个支持 XA)

## 二、2PC 完整时序图(成功路径)

```
Coordinator                P1                P2                P3
    │                       │                 │                 │
    │── PREPARE ──────────→│                 │                 │
    │── PREPARE ───────────────────────────→│                 │
    │── PREPARE ─────────────────────────────────────────────→│
    │                       │                 │                 │
    │                  [本地准备,写 prepare 日志,加锁]
    │                       │                 │                 │
    │←──── YES ─────────────│                 │                 │
    │←──── YES ──────────────────────────────│                 │
    │←──── YES ────────────────────────────────────────────────│
    │                       │                 │                 │
    │ [写"决定 commit"到协调者日志]   ← 决策点!
    │                       │                 │                 │
    │── COMMIT ────────────→│                 │                 │
    │── COMMIT ────────────────────────────→│                 │
    │── COMMIT ──────────────────────────────────────────────→│
    │                       │                 │                 │
    │                  [本地 commit,写 commit 日志,释放锁]
    │                       │                 │                 │
    │←──── ACK ─────────────│                 │                 │
    │←──── ACK ──────────────────────────────│                 │
    │←──── ACK ────────────────────────────────────────────────│
    │                       │                 │                 │
    │ [写"事务完成"到协调者日志,清理]
```

### 2.1 阶段 1:Prepare(投票阶段)

协调者发 PREPARE 给所有参与者,**参与者**:

1. 执行事务到 commit 前一刻(锁、约束检查、写 undo log 全部做完)
2. 写"PREPARED"到本地 WAL(**fsync**)
3. 回 YES

**关键**:参与者一旦回 YES,**就必须保留状态(锁不释放、prepare log 不删)** 直到收到协调者决议。**这是 2PC 同步阻塞的根源**——锁要握到第二阶段结束。

如果参与者本地约束失败 / 死锁 / 资源不足 → 回 NO,自己 abort 释放资源。

### 2.2 阶段 2:Commit(执行阶段)

协调者收齐所有 YES:

1. 写"DECIDED COMMIT"到本地 WAL(**fsync——这是不变量的命脉**)
2. 广播 COMMIT 给所有参与者
3. 收到所有 ACK,清理事务

参与者收到 COMMIT:

1. 本地 commit(原本只差这一步)
2. 释放锁
3. 写 commit 日志
4. 回 ACK

任意一个参与者 NO,协调者:

1. 写"DECIDED ABORT"到本地 WAL
2. 广播 ABORT
3. 参与者收到 ABORT → 本地 abort、释放锁

### 2.3 不变量

整个协议保证两条铁律:

```
不变量 1(Agreement):所有参与者要么都 commit,要么都 abort,不可能一部分 commit 一部分 abort
不变量 2(Validity):commit 只在所有参与者投 YES 时发生
```

这两条不变量是 2PC 的全部价值——**只要协议正确执行,跨机的"原子提交"就成立**。

## 三、2PC 状态机(伪代码)

### 3.1 协调者状态机

```python
class Coordinator:
    state = "INIT"

    def execute(self, participants, tx):
        # 阶段 1
        self.state = "PREPARING"
        responses = []
        for p in participants:
            r = p.prepare(tx)               # 网络调用
            responses.append(r)
            if r == "NO":
                break

        # 决策点
        if all(r == "YES" for r in responses):
            decision = "COMMIT"
        else:
            decision = "ABORT"

        # **关键:决策写入持久化日志,fsync**
        self.wal.write(f"DECIDED {decision} for {tx}")
        self.wal.fsync()
        self.state = decision

        # 阶段 2
        for p in participants:
            while True:
                try:
                    p.deliver(decision)     # 必须重试到成功
                    break
                except Timeout:
                    pass                    # 重试

        self.state = "DONE"
        self.wal.write(f"COMPLETED {tx}")
```

### 3.2 参与者状态机

```python
class Participant:
    def prepare(self, tx):
        try:
            self.do_work(tx)                # 锁、约束、undo log
            self.wal.write(f"PREPARED {tx}")
            self.wal.fsync()                # **关键**
            self.state = "PREPARED"
            return "YES"
        except:
            self.state = "ABORTED"
            return "NO"

    def deliver(self, decision):
        if self.state != "PREPARED":
            return                          # 重复消息或 abort 早完成
        if decision == "COMMIT":
            self.do_commit()
            self.wal.write(f"COMMITTED {tx}")
        else:
            self.do_abort()
            self.wal.write(f"ABORTED {tx}")
        self.state = "DONE"

    def recover_from_crash(self):
        # 启动时扫 WAL,找到所有 PREPARED 但未 COMMITTED/ABORTED 的事务
        for tx in self.wal.prepared_pending():
            # **必须问协调者要决议**——自己不能决定
            decision = self.coordinator.query_decision(tx)
            if decision is None:
                # 协调者也不知道(自己也挂过)→ 阻塞,等
                continue
            self.deliver(decision)
```

**这段伪代码里所有 `fsync()` 都不能省**——少 fsync 一次,2PC 就有可能丢一致性。这就是为什么 2PC 慢——**每个事务多 2-3 次 fsync 加上跨节点 RTT**。

## 四、2PC 的两个致命问题

2PC 数学上是对的(只要协议正确执行,Agreement 永远成立),**但工程上有两个永远绕不开的命门**。

### 4.1 命门一:同步阻塞

**参与者一旦 prepare,就要持锁等待协调者决议**。期间:

- 行被锁,其他事务读不到 / 改不了(取决于隔离级别)
- 任何参与者慢、网络抖,**全局事务都被它拖**
- 整体延迟 = max(所有参与者) + 2 RTT + 多次 fsync

```
正常事务延迟 = 1 RTT + 1 fsync ≈ 5-10ms
2PC 事务延迟 = 2 RTT + 3 fsync(协调者决策 + 双方 prepare) ≈ 30-50ms
```

**这是 5 倍的延迟**——对高 QPS 业务直接致命。

### 4.2 命门二:协调者单点不确定

**最致命的失败**:协调者写完 "DECIDED COMMIT" 但还没来得及广播就挂了。

```
Coordinator               P1                P2
    │                      │                 │
    │── PREPARE ─────────→│                 │
    │── PREPARE ──────────────────────────→│
    │←─ YES ───────────────│                 │
    │←─ YES ────────────────────────────────│
    │ [写 DECIDED COMMIT,fsync]
    │ ☠ 挂了!!尚未广播      │                 │
    
    P1 和 P2 状态 = PREPARED,持锁等待
    超时也不能擅自 commit 或 abort——
    协调者可能已经决定 commit(WAL 里写了),
    自己擅自 abort 就破坏 Agreement
    
    → 阻塞 forever,等协调者恢复
```

**P1 / P2 不敢擅自 abort,因为协调者磁盘上可能有"DECIDED COMMIT",将来恢复会广播 COMMIT,自己已经 abort 了 → 破坏 Agreement**。

**只能等**——等协调者重启读 WAL,把决议补播。如果协调者磁盘也挂了 → **永久阻塞**,只能人工救火。

> **这就是 2PC 的"协调者单点"问题**——不是"协调者挂了不可用",而是"协调者挂了之后参与者持锁等着,谁都不敢动"。这一锁一持就是几秒到几小时,**业务直接卡死**。

### 4.3 各种失败场景汇总

| 失败时机 | 谁可恢复 | 影响 |
| --- | --- | --- |
| 阶段 1 协调者发送 PREPARE 前挂 | 协调者重启重新发,或 abort | 无影响 |
| 阶段 1 参与者收到 PREPARE 前挂 | 协调者重发(参与者重启后等待) | 短暂阻塞 |
| 阶段 1 参与者 PREPARED 后挂 | 参与者重启从 WAL 恢复,问协调者要决议 | 短暂阻塞 |
| 阶段 2 协调者**决策后未广播**挂 | **参与者只能等协调者恢复** | **可能 forever blocking** |
| 阶段 2 协调者广播过程中挂(部分参与者收到) | 协调者重启读 WAL,补播给未送达的 | 短暂阻塞 |
| 阶段 2 参与者收到 COMMIT 但 ACK 前挂 | 参与者重启,看 WAL 有 COMMITTED → 完成 | 协调者重发 COMMIT(幂等) |

**这表里第 4 行的"决策后未广播挂"是 2PC 真正的命门**——其他情况都能恢复,这一种**没法靠协议本身解决,只能靠协调者高可用**(主从切换 / Raft 选主)。

## 五、3PC:加一轮 PreCommit 试图绕过单点

3PC(Three-Phase Commit,Skeen 1981)的核心思路:**把决策点拆成两步,让参与者在协调者挂时也能"自己推进"**。

### 5.1 三个阶段

```
Phase 1: CanCommit?     → 参与者答 YES / NO(本地校验,**不加锁**)
Phase 2: PreCommit      → 协调者收齐 YES,广播 PreCommit;参与者 prepare(加锁)+ ACK
Phase 3: DoCommit       → 协调者收齐 ACK,广播 DoCommit;参与者 commit + 释放锁
```

### 5.2 时序图

```
Coordinator                P1                P2
    │                       │                 │
    │── CanCommit? ────────→│                 │
    │── CanCommit? ──────────────────────────→│
    │←──── YES ─────────────│                 │
    │←──── YES ────────────────────────────────│
    │                       │                 │
    │── PreCommit ─────────→│                 │
    │── PreCommit ────────────────────────────→│
    │                  [本地 prepare,写 WAL]
    │←──── ACK ─────────────│                 │
    │←──── ACK ─────────────────────────────────│
    │                       │                 │
    │── DoCommit ──────────→│                 │
    │── DoCommit ─────────────────────────────→│
    │                  [本地 commit]
    │←──── ACK ─────────────│                 │
    │←──── ACK ─────────────────────────────────│
```

### 5.3 3PC 怎么"自救"

**核心引入 timeout**:

- 参与者收到 PreCommit 后,**如果协调者一直不发 DoCommit,等到超时,自己 commit**
- 参与者**没**收到 PreCommit 就超时 → 自己 abort

**逻辑**:既然到了 PreCommit,**说明所有参与者都答了 YES,协调者一定已经决定 commit**——所以即使协调者挂了,参与者超时后**自动 commit 也安全**。

### 5.4 3PC 的代价和命门

**代价**:**多一轮 RTT**,延迟比 2PC 还高。

**命门**(为什么生产几乎没人用 3PC):

**分区下仍然不安全**。考虑这种场景:

```
1. 协调者发 PreCommit 给 P1,P2 → P1 收到,P2 因网络分区没收到
2. 协调者挂了
3. P1 超时 → 自己 commit
4. P2 超时(没收到 PreCommit)→ 自己 abort

→ Agreement 破了!
```

3PC 假设"网络是同步的(消息要么及时到达要么及时检测到失败)",**但真实网络不是**——网络分区 + 节点崩溃同时发生,3PC 仍然出错。

> FLP 不可能定理(09 篇)其实已经预言了这一点:**异步网络 + 即使一个节点崩溃,共识协议必然有"不可终止"的执行**。3PC 想绕过它,只在同步网络假设下成立——而**真实网络是异步的**,所以 3PC 不安全。

## 六、2PC vs 3PC 对比

| 维度 | 2PC | 3PC |
| --- | --- | --- |
| 阶段数 | 2 | 3 |
| RTT | 2 | 3 |
| Fsync 次数 | 2-3 | 3-4 |
| 协调者挂后参与者 | 阻塞 | 超时自决 |
| 同步阻塞 | 强(prepare 后持锁) | 强(PreCommit 后持锁) |
| 分区下安全性 | 安全(可能阻塞) | **不安全**(可能不一致) |
| 生产采用 | 少(XA) | **几乎无** |

**结论**:**3PC 用一次多余的网络往返换"不阻塞",但买不到正确性**——所以**没人用 3PC**,即使要 2PC 替代,大家也用**带 Paxos / Raft 协调的 2PC**(协调者本身高可用)而不是 3PC。

## 七、XA:工业界的 2PC 标准

XA 是 X/Open 制定的分布式事务规范,**JDBC / JTA / Java EE / Spring 的分布式事务都基于 XA**。

### 7.1 XA 的角色

```
TM(Transaction Manager,事务管理器)   ← 协调者
    ├── Atomikos / Bitronix / Narayana
    ├── Spring 的 JtaTransactionManager
    └── Seata TC(简化版 2PC)

RM(Resource Manager,资源管理器)      ← 参与者
    ├── MySQL XA / PostgreSQL Prepared
    ├── Oracle XA
    └── 消息队列 XA(IBM MQ / ActiveMQ)
```

### 7.2 MySQL XA 命令

```sql
-- 阶段 1
XA START 'tx_id';
UPDATE account SET balance -= 100 WHERE id = 'alice';
XA END 'tx_id';
XA PREPARE 'tx_id';        -- ← prepare,WAL 写 prepared

-- 阶段 2(由 TM 发起)
XA COMMIT 'tx_id';         -- 或 XA ROLLBACK 'tx_id'
```

**生产真用 MySQL XA 的不多**——理由:

1. **性能**:5.6 之前有严重 bug(主从复制 XA 状态丢),5.7+ 修了但性能仍差
2. **运维**:卡住的 XA 事务要手工 XA RECOVER + XA COMMIT/ROLLBACK 清理
3. **工具链**:XA 锁不被慢查询统计、监控、慢日志友好支持

### 7.3 PostgreSQL Prepared Transactions

PG 的 PREPARE TRANSACTION 是同样的协议(2PC 协议中的 prepare 阶段),功能相同。

```sql
BEGIN;
UPDATE account SET balance -= 100 WHERE id = 'alice';
PREPARE TRANSACTION 'tx_id';
-- 此后任意时间(数秒到数天):
COMMIT PREPARED 'tx_id';     -- 或 ROLLBACK PREPARED 'tx_id'
```

PG 默认 `max_prepared_transactions = 0`(禁用),要用 2PC 必须显式开启——**官方都觉得这是少数派需求**。

### 7.4 Seata XA 模式

Seata 是阿里开源的分布式事务框架,**XA 模式底层就是 2PC**:

- Seata-TC = 协调者
- 各服务的数据库 = 参与者(走 MySQL XA 或 PG Prepared)
- 应用代码 `@GlobalTransactional` 包一层

**实际生产 Seata 大家更多用 AT 模式或 TCC 模式,XA 模式用得少**——同样因为 XA 的性能和运维痛点。

## 八、为什么 2PC 生产很少用

总结 2PC 不普及的工程理由:

### 8.1 性能差

- 5-10 倍正常事务延迟
- 锁握时间长,高并发时锁等待暴涨
- fsync 次数多,SSD 寿命也受影响

### 8.2 失败恢复痛

- 协调者挂了,参与者卡死,**得人介入**
- 残留的 prepared 事务要手工清理(XA RECOVER)
- 监控难——卡住的事务不在常规慢日志里

### 8.3 跨异构系统更痛

- MySQL + 消息队列 + Redis 跨三种 → 都得支持 XA,Redis 根本不支持
- 不同厂商的 XA 实现兼容性差(经典:MySQL XA 跟 Oracle XA 一起用容易出问题)

### 8.4 替代方案够用

- 业务层 Saga(22 篇):**长事务 + 业务级补偿**,容易理解、易于运维
- TCC(22 篇):**Try / Confirm / Cancel**,业务级"软事务",**互联网公司主流**
- Percolator(23 篇):**Google 的低延迟版 2PC**,TiDB 在用
- Spanner(24 篇):**TrueTime + Paxos 包了 2PC**,Google 内部用

> **2PC 不是错,是"用错了场景"**——它适合**少数高一致性需求的金融核心场景**,且**协调者要做 Raft 高可用**。互联网通用业务用它=自虐。

## 九、活下来的 2PC 变种

虽然原汁原味的 2PC 用得少,**它的思想活在很多地方**。理解 2PC,就理解了下面这些系统的底:

### 9.1 Percolator(23 篇详讲)

Google 2010 论文。**2PC 减配版**:

- 协调者 = 任一参与者(没有专门 TM)
- prepare = "primary key 加锁,其他 key 写临时记录"
- commit = "primary key 标记 commit,其他 key 异步清理"

**TiDB / TiKV 用 Percolator 实现分布式事务**——单数据中心强一致,跨地域降级。

### 9.2 Spanner(24 篇详讲)

Google 2012 论文。**2PC + Paxos + TrueTime**:

- 每个 shard 是一个 Paxos group(参与者高可用)
- 协调者也走 Paxos(协调者高可用)
- TrueTime 提供全序时间戳,**消除单点不确定状态**

**协调者单点问题被 Paxos 化的协调者解决**——这是工业界最优雅的 2PC 工程化。

### 9.3 Saga / TCC(22 篇详讲)

**严格说不是 2PC,但思想同源**:

- **Saga**:把长事务拆成多个本地事务,每个有补偿;**没有 prepare,有失败补偿**
- **TCC**:Try(预占)/ Confirm(确认)/ Cancel(取消)——**Try ≈ prepare,Confirm ≈ commit,但没有协调者持久化 WAL,靠业务补偿兜底**

**业务层把 2PC 的"原子性"重新发明,代价是中间状态对业务可见(业务要处理"半成品状态")**——这就是 BASE / 最终一致的工程哲学。

### 9.4 Kafka Transactional Producer

Kafka 的事务消息也是 2PC 思想:

- producer 写消息时分 prepare → commit 两阶段
- Kafka broker 内部有 transaction coordinator
- consumer 通过 read_committed 隔离级别只读 committed 消息

**这是把 2PC 嵌入到消息队列内部**——业务层透明,但仍有 2PC 的性能代价。

## 十、踩坑提醒

1. **以为 2PC 就是 "两步 commit"那么简单**——核心是**协调者把决策写 WAL + fsync**,**少这一步 2PC 就不正确**
2. **协调者不做 HA**——单协调者挂了所有参与者卡死,**至少 Raft 化或主备**
3. **以为 3PC 比 2PC 好**——3PC 在分区下不安全,**没人在生产用 3PC**
4. **跨服务事务上 XA**——99% 场景该用 Saga / TCC,**XA 性能差 + 运维痛**
5. **MySQL XA 跨 5.6 / 5.7 混用**——5.6 有 bug,要全部升 5.7 以上
6. **prepared 事务残留不清理**——`XA RECOVER` / `pg_prepared_xacts` 定期巡检,长时间残留的 prepared 事务持锁
7. **以为参与者超时自己 abort 安全**——2PC 里参与者**绝不能**擅自 abort prepared 状态,**必须等协调者**,擅自 abort 破坏 Agreement
8. **不打监控**——XA / Prepared 状态要单独监控,挂掉的协调者要告警
9. **业务流程长(秒级以上)还用 2PC**——锁持太久,**该上 Saga 让业务可见中间状态 + 补偿**
10. **不读 Gray 1978 原始论文**——分布式事务设计每隔几年就有人重新发明 2PC 然后发现命门,**站在巨人肩膀上比自己造轮子省 5 年**

## 收束:为什么 2PC 仍然重要

2PC 在生产上用得少,**但它是分布式事务的底图**。后面 22-24 篇看似花样翻新:

- **Saga / TCC**:用业务补偿替代 prepare,放弃 prepare 阶段的"持锁等待",**换可用性**
- **Percolator**:把协调者职责分散到参与者,**用 primary key 当锚点**,简化协调者 HA
- **Spanner**:用 Paxos 化协调者 + TrueTime 全序时间戳,**把 2PC 的命门各个击破**
- **Calvin / FaunaDB**:走"先全局排序,后并行执行"的另一条路,**完全绕开 2PC**

**每一种都是在 2PC 的基础上"减一刀"或"补一刀"**。不懂 2PC,你看不到他们减的是哪一刀、补的是哪一刀,**就只能记名词,记完忘**。

而**真要理解 2PC,看这一张图就够了**:

```
不变量:全 commit 或全 abort
        │
        ↓
    决策点(协调者写 WAL)─────┐
        │                     │ 决策后未广播挂
        ↓                     ↓
    第二阶段广播          ★ 命门 ★
        │              参与者持锁等待
        ↓              协调者不恢复就阻塞 forever
    成功
```

**这一张图记住,2PC 就懂了**。后面的方案都在画这张图的变体——你看一眼新方案的时序图,就能立刻定位"哦,它在这里改了一刀"。
