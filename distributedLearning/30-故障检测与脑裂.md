# 故障检测与脑裂

整个分布式系列绕了 29 篇,最后还是要回到那个最朴素的问题:**你怎么知道对面的机器死了**?这件事看似简单,**理论上却是无解的**——Chandra-Toueg 1996 证明了完美故障检测器不存在,**只要网络可能延迟,你就永远分不清"死了"和"暂时联系不上"**。这种不确定性的终极后果就是**脑裂(Split-Brain)**——两个子网都认为对方死了,**各自选主、各自写数据,恢复时两份不一致的数据撞在一起**。GitHub 2018 年那场 24 小时 MySQL 大事故、各家 Redis 主从切换丢数据的惨剧,全都是脑裂的工程后果。这一篇——也是 30 篇分布式系列的**收尾篇**——把故障检测的分级理论、φ-accrual 自适应检测、脑裂的五大防御方案讲透。

> 一句话先记住:**完美故障检测器不存在(Chandra-Toueg)**——只能近似,工程上 Cassandra/Akka 用 **φ-accrual** 给出"连续怀疑度"让业务自己决策阈值。**脑裂是故障检测不准的必然后果**——防御只能靠**Quorum / Fencing Token / Witness / STONITH** 四件套。**永远假设分区会发生,不要假设它"应该不会"**——GitHub 24 小时事故、Redis 双主丢数据全是这条假设错的代价。

## 一、故障检测的根本难题:Chandra-Toueg 1996

Chandra 和 Toueg 1996 年那篇 *The Weakest Failure Detector for Solving Consensus* 给故障检测下了精确定义,并证明:**完美的故障检测器不存在**。

### 1.1 故障检测器要满足两个性质

```
Completeness(完备性):
  所有真死的节点,最终都会被怀疑

Accuracy(准确性):
  没死的节点不应该被怀疑
```

**完美故障检测器**=Completeness + 严格 Accuracy:**死了一定怀疑、没死永远不怀疑**。

**为什么不存在**:**异步网络下,消息延迟无上限**——你 ping 一个节点 5 秒没回,它可能是真死了,也可能是网络拥塞、GC 暂停、虚拟机调度延迟。你**永远无法从"没收到回复"推导出"它死了"**——FLP 不可能定理(09 篇)的根。

### 1.2 故障检测器分级

Chandra-Toueg 给出四种弱化的检测器,工程上能实现的都是弱版本:

| 检测器 | Completeness | Accuracy | 工程能实现? |
| --- | --- | --- | --- |
| Perfect(P) | 强 | 强 | **不可能** |
| Strong(S) | 强 | 弱(某些时段可能误判) | 不可能 |
| Eventually Perfect(◇P) | 强 | 最终强(终有一刻不再误判) | 接近 |
| **Eventually Strong(◇S)** | 强 | 最终弱(终有一时段不误判) | **工程都是这个** |

**所有真实系统**(Cassandra、ZK、Consul、K8s)的故障检测都是 ◇S——**接受短暂误判,只要"最终能正确"**。

> 这是分布式系统最诚实的承认:**我们做不到完美判生死,只能"近似 + 业务容错"**。**任何告诉你"100% 准确判活死"的系统都在撒谎**。

## 二、朴素的超时检测:Heartbeat + Timeout

工程上最常见的实现:**周期心跳 + 超时阈值**。

```python
class SimpleFailureDetector:
    def __init__(self, timeout=5.0):
        self.last_heartbeat = {}
        self.timeout = timeout

    def receive_heartbeat(self, node):
        self.last_heartbeat[node] = time.now()

    def is_alive(self, node):
        return (time.now() - self.last_heartbeat[node]) < self.timeout
```

**问题**:**timeout 怎么定**?

- **太长**(60 秒):真死了也要等 60 秒才发现,故障切换慢
- **太短**(1 秒):网络抖动一下就误判,频繁切换造成系统抖动

> **真实生产里,不同环境的网络延迟差异巨大**——同机房 1ms、跨机房 30ms、跨大洲 200ms。**一个固定 timeout 不可能适用所有场景**。

## 三、φ-accrual 故障检测器(Hayashibara 2004)

Hayashibara 等人 2004 年那篇 *The φ Accrual Failure Detector* 提出了**自适应连续怀疑度**的想法,**至今是工业界标准**:

### 3.1 核心思想

**不返回 0/1(alive/dead),返回一个连续值 φ**——表示「**当前怀疑这个节点的程度**」:

```
φ 越大,越确信节点已死
φ → 0:刚收到心跳,完全相信它活
φ → ∞:很久没心跳,几乎确定死了
```

业务根据自己的"容忍度"挑阈值:

- 关键场景(主从切换):φ > 8 才动手(很谨慎)
- 非关键场景(路由表更新):φ > 3 就标记 suspect

### 3.2 算法核心

```
1. 维护历史心跳到达间隔的统计(滑动窗口,最近 1000 次)
2. 计算到达间隔的均值 μ 和标准差 σ
3. 当前距上次心跳已过 t_now 秒
4. 假设到达间隔服从正态分布,计算"距上次心跳超过 t_now 秒"的概率 P
5. φ = -log10(P)
```

**直觉**:

- 平时心跳间隔 1 秒,标准差 0.1 秒——**当前已经 5 秒没心跳了,概率极低 → φ 很大**
- 平时心跳间隔 1 秒,标准差 5 秒(网络很抖)——**当前 5 秒没心跳,概率不低 → φ 较小**

**自适应**:**网络越抖,容忍度越大**——这正是工程需要的。

### 3.3 φ 曲线示意

```
       φ
       ▲
   10  │                                ●
       │                            ●
    8  │ ──── 严格阈值 ────────── ●
       │                       ●
    6  │                    ●
       │                  ●
    4  │ ──── 宽松阈值 ─ ●
       │                ●
    2  │              ●
       │           ●
    0  │ ●●●●● 
       └──────────────────────────────────► 距上次心跳时间
       0        1秒    2秒    3秒    5秒
       
       心跳间隔均值 1 秒,标准差 0.1 秒
       业务可在不同阈值做不同决策
```

### 3.4 用了 φ-accrual 的真实系统

- **Cassandra**:节点状态判定全靠 φ,默认阈值 8
- **Akka**:Akka Cluster 默认 φ-accrual,可调阈值
- **Hazelcast**:可选 φ-accrual
- **Riak / ScyllaDB**:同样用

### 3.5 φ 调优经验

```
阈值高(φ=12):
  + 几乎不误判
  - 真死了要等很久才发现

阈值低(φ=3):
  + 反应快
  - 网络抖一下就误判,主从频繁切换

中庸(φ=8):
  Cassandra 默认值,平衡点
```

**生产经验**:**先在正常时期观察 φ 的分布**——99 分位 φ 大概是多少,把阈值定在 99.9 分位往上一点,**误判率自然就低**。

## 四、脑裂(Split-Brain):故障检测不准的终极后果

### 4.1 什么是脑裂

**网络分区让集群分成两个互不可见的子集**,**每个子集都认为对方死了**,**各自选主、各自接受写入**。**当分区恢复时,两份不一致的数据撞在一起,没法自动合并**。

```
正常状态:
   ┌──── 主 M ────┐
   │   ↓          │
   │  从 S1, S2   │
   └──────────────┘

网络分区:
   ┌─ M, S1 ─┐     ┌─ S2 ─┐
   │  ↓      │  ⫻⫻  │      │
   │ 老主继续 │ 分区 │ 选新主 │
   │  写     │     │  也写  │
   └─────────┘     └───────┘

   M 上接受了写 W1
   S2 上接受了写 W2

分区恢复:
   两份不一致的数据怎么 merge?
   W1 和 W2 是否冲突?用谁的?
```

### 4.2 脑裂的可怕之处

- **数据丢失**:合并时必然有一份写被丢弃
- **数据撕裂**:同一个账户,M 看余额 100,S2 看余额 80,**取哪个**?
- **双花 / 重复扣款**:转账期间脑裂,两侧各扣一次

> **脑裂不是"如果发生",是"什么时候发生"**——交换机故障、跨机房光纤抖动、防火墙规则错配,**所有运维过分布式系统的人都见过分区**。**没设计脑裂防护的系统,就是在等出大事**。

## 五、防御方案 1:Quorum 多数派(Raft / Paxos)

**最经典的反脑裂手段**:**任何决策都必须多数派同意**。

```
5 节点集群分区成 3 + 2:
   3 节点这边能凑齐多数派(3 > 5/2)→ 继续工作
   2 节点这边凑不齐(2 < 5/2)    → 停止接受写

恢复时:少数派 2 节点重新加入,落后的数据从多数派同步
```

**关键不变量**:**多数派和多数派必然相交**(鸽笼原理)——**任何两个多数派至少有一个共同节点**,**这个节点保证看过的写不会丢**。

```
3 + 2 分区时:
   左侧多数派:{A, B, C}
   右侧无多数派:{D, E}(只有 2 个)
   不可能两侧都凑齐 3 个 → 不可能两侧都写 → 不会脑裂
```

**用了 Quorum 的系统**:

- **Raft**:13 篇,etcd / Consul / Nomad / TiKV
- **Paxos / Multi-Paxos**:11/12 篇,Chubby / Spanner
- **ZAB**:14 篇,ZooKeeper

### 5.1 Quorum 的代价

- **必须奇数节点**:偶数节点(4 个)分区成 2+2 时**两侧都凑不齐多数派,全停**
- **少数派完全不可用**:CP 系统,可用性换一致性
- **写吞吐受限**:每次写要多数派确认,**leader 是瓶颈**

```python
def can_serve_writes(alive_nodes, total_nodes):
    return alive_nodes > total_nodes // 2
```

## 六、防御方案 2:Witness / Tiebreaker(轻量见证)

**问题**:**两机房部署,各放 2 节点,凑不出奇数**——分区时谁也凑不齐多数派。

**解法**:**在第三个地方放一个轻量见证节点(Witness)**:

```
机房 A: 2 节点(主)
机房 B: 2 节点(从)
机房 C: 1 节点(witness,只投票不存数据)

分区场景:
  A 和 B 断了,但 A、B 都还能看到 C
  哪一侧能拉到 C 的票,哪一侧凑齐多数派(2+1=3 > 5/2)
  那一侧继续服务,另一侧降级
```

**Witness 节点设计原则**:

- 不存数据(只参与投票)
- 在第三地(不能和任何主节点一起挂)
- 轻量(单台小机器够)

**用了 Witness 的系统**:

- **SQL Server AlwaysOn**:支持 Witness Server
- **MongoDB**:有 Arbiter 角色(只投票)
- **Galera Cluster**:Garbd(仲裁守护进程)

## 七、防御方案 3:Fencing Token(围栏令牌)

**Quorum 防住了"两边都选主",但防不住"老主继续干活"**——老主在分区那侧还以为自己是主,**对下游系统(数据库 / 存储)继续写**。

```
T0: 老主 M1 是主,有租约
T1: 网络分区,M1 那侧没多数派
T2: 多数派那侧选了新主 M2
T3: M1 不知道自己被废了,继续往后端写
T4: M2 也在往后端写
T5: 后端被两个主写脏了
```

**解法**:**Fencing Token**——每次选主都给一个**单调递增的编号(epoch / term)**,所有下游写都带上这个编号,**下游拒绝低于已知最大编号的写**。

```
T0: M1 是主,token = 5
T1: 分区,M2 当选,token = 6
T2: M1 还在写下游:"write X, token=5"
T3: 下游存储记得"最大见过的 token = 6",
    收到 token=5 的请求,拒绝
T4: M2 写下游:"write Y, token=6",
    下游接受,token 仍是 6
```

```
                M1(老主,token=5)
                     │
                     │ "写 X, token=5"
                     ▼
                ┌─────────┐
                │  存储   │ 已见过 token=6
                │         │ → 拒绝 ✗
                └─────────┘
                     ▲
                     │ "写 Y, token=6"
                     │
                M2(新主,token=6)
```

**用了 Fencing Token 的系统**:

- **Chubby Lease**:每次 lease 续约带 epoch
- **Kafka ISR**:每个 leader 有 epoch,follower 拒绝旧 epoch 的写
- **HDFS NameNode HA**:Standby 转 Active 时 fencing(原 Active 强制隔离)
- **GFS / Bigtable**:租约 + epoch
- **etcd / ZK**:lease + version

> **Fencing 是 26 篇分布式锁讨论 Redlock 时的核心争议**——Martin Kleppmann 喷 Redlock 的核心就是"没有 fencing token"。**没 fencing,锁就只是个建议,不是强制**。

## 八、防御方案 4:STONITH(枪毙对方)

**Shoot The Other Node In The Head**——**直接物理隔离老主**:断电、断网、kill 进程。

**实现手段**:远程电源开关(IPMI / iLO)断电、交换机 API 禁端口、存储 fence 让对方无法访问共享存储、Cloud API force terminate VM。

**用了 STONITH 的系统**:Linux-HA / Pacemaker、VMware HA、Oracle RAC(节点驱逐 node eviction)。

**代价**:需要远程设备 / 云 API,误判时把好节点也枪毙,不适合大规模集群。

> **STONITH 适合小规模 HA 集群**(2-5 节点的数据库主备),**大规模分布式系统用 Quorum + Fencing Token 更合适**。

---

## 九、防御方案 5:Quorum 磁盘 / 共享存储

**老派 HA 集群**(Oracle RAC、Veritas Cluster)用过的方案:**所有节点共享一块磁盘**,**抢到磁盘锁的那个才是主**。分区时只有一侧能抢到。

**问题**:共享存储自己是单点、现代云原生架构基本不用 SAN、复杂、贵。**新世代基本被 Quorum + Witness 取代**,只在传统数据库 HA 里能见到。

## 十、五大防脑裂方案对照表

| 方案 | 适用场景 | 代价 | 代表系统 |
| --- | --- | --- | --- |
| **Quorum 多数派** | 共识协议 / 强一致存储 | 必须奇数节点,少数派不可用 | Raft / Paxos / ZAB / etcd / ZK / TiKV |
| **Witness / Tiebreaker** | 双机房部署补奇数 | 第三地节点维护成本 | MongoDB Arbiter / SQL Server / Galera Garbd |
| **Fencing Token** | 防老主写下游 | 下游存储需支持 token 检查 | Chubby / Kafka ISR / HDFS HA / Spanner |
| **STONITH** | 小规模 HA 集群 | 需要物理 / 云 API,误判代价大 | Pacemaker / Linux-HA / Oracle RAC |
| **Quorum 磁盘** | 传统数据库 HA | 共享存储单点 | Oracle RAC(旧) / Veritas |

> **实际工程通常组合**:**Raft 给共识(Quorum)+ Fencing Token 给下游 + Witness 节点补奇数**——三件套配齐才稳。

## 十一、真实事故案例:GitHub 2018 MySQL 24 小时故障

**2018 年 10 月**,GitHub 经历了**24 小时部分服务降级**,根因就是脑裂。

### 11.1 事件经过

```
T0: 美国东岸主机房 vs 西岸主机房之间 43 秒网络分区
T1: Orchestrator(MySQL 自动故障切换工具)误判东岸主死了
T2: Orchestrator 把西岸的某台从库提升为主
T3: 网络恢复,东岸原主继续在接受写入
T4: 现在出现两个主:东岸老主(收旧 SQL),西岸新主(收新 SQL)
T5: 两个主都接受 binlog 写入,数据分叉
T6: 后续修复花了 24 小时(手动 reconcile 数据)
```

### 11.2 根因

- **跨机房分区**:东西岸 43 秒短暂分区
- **Orchestrator 故障切换太激进**:没等够、没要求 Quorum
- **MySQL 没有 Fencing**:老主继续写,没机制拒绝
- **业务流量没有引导到单一主**:DNS / 路由层没切

### 11.3 教训

- **跨地域 HA 切换要非常谨慎**——宁可手动切也别自动切
- **必须有 Fencing**:老主切换后**立刻**被 fence(改 read-only、blocked write)
- **业务路由要和故障切换联动**:DNS / 服务发现要立刻指向新主

> GitHub 那篇 post-mortem(*October 21 post-incident analysis*)是分布式工程师必读教材,**比任何教科书都讲得透"脑裂多痛"**。

## 十二、其他真实事故

- **Redis 主从切换丢数据**:Redis Cluster / Sentinel 主从异步复制 + 异步切换,主刚收到写还没复制就挂了,新主上线时不知道这些写,**直接丢**。Antirez 自己也承认:**Redis 主从切换不是为数据安全设计的,要数据安全请用 RDBMS**。
- **Elasticsearch 脑裂**:ES 6 之前没有正经 Quorum 选主,`discovery.zen.minimum_master_nodes` 配错就脑裂,两个集群各自演化、索引数据撕裂。**ES 7 引入 Coordination layer(类 Raft)才彻底治这病**。
- **MongoDB Replica Set 脑裂**:老版本在某些极端时序下出现过双主,**Jepsen 测试报告**里有具体场景。后续版本通过严格 Quorum + Election Term 修复。

## 十三、网络分区的纪律:别假设它不会发生

整个分布式工程的核心心法:

- **假设分区会发生,不是"如果"是"何时"**——同机房每月几次、跨机房每月十几次、跨地域每周。Aphyr 那篇 *Network is Reliable* 把全行业的网络故障案例整理出来——**没有"网络很可靠"这个事**。
- **明确分区时哪侧停服**:部署设计阶段就要回答"哪侧继续写、哪侧降级、业务路由如何感知"。
- **监控分区本身**:跨机房 ping(延迟暴增 = 分区前兆)、TCP 重传率、集群成员视图、Quorum 视图。
- **演练分区场景**:Chaos Engineering 故意制造分区,验证切换 + 降级 + 恢复全流程。

## 十五、踩坑提醒

1. **以为完美故障检测器存在**——Chandra-Toueg 已经证明不可能。**任何检测都是概率近似**,业务必须容错(重试、熔断、降级)。
2. **timeout 设得太短**——网络抖、GC 暂停、虚拟机调度都会让心跳延迟几秒,1 秒 timeout 会狂误判。**Cassandra 默认 φ=8 对应几秒到十几秒**,K8s 默认 40 秒。**别太激进**。
3. **timeout 设得太长**——5 分钟才发现死节点,业务感知极差。**用 φ-accrual 让阈值自适应**,而不是死死的固定值。
4. **偶数节点部署**——4 节点分区成 2+2 时全停,**永远奇数节点**或加 Witness。
5. **2 机房各 2 节点不加 Witness**——这就是 GitHub 那种灾难拓扑,**第三个机房放个 Witness 就能救命**。
6. **以为 Quorum 就够了**——Quorum 防"两边都选主",**防不住"老主继续写下游"**。**必须配 Fencing Token**。
7. **下游存储不支持 Fencing**——Redlock 经典问题(26 篇),**下游必须有"拒绝低 epoch 写"的能力**才完整。
8. **STONITH 用在大规模集群**——几千节点用 STONITH 不现实,**STONITH 只适合 2-5 节点的 HA 数据库**。大规模用 Quorum + Fencing。
9. **不做分区演练**——平时跑得好好的,分区那一刻发现切换流程没人会操作。**Chaos Engineering 定期演练分区**。
10. **跨地域全自动故障切换**——GitHub 教训,**跨地域延迟大、分区频繁,自动切换误判代价巨大**。重要场景宁可半自动(报警人工确认)。
