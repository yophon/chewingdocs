# 分布式系统理论系列写作计划

> 来源:`未来系列规划.md` 第三类「纵深主题」第七项(G. distributed),⭐⭐⭐⭐。
> systemDesign 教了「**怎么搭一个能抗住流量的系统**」(缓存 / 限流 / 多机房 / 容灾),本系列教了「**这些系统底下凭什么能跑对**」——为什么三台机器写同一个值这么难、为什么 2PC 会卡死、为什么 Raft 比 Paxos 火、为什么 Spanner 要装原子钟、为什么 Redlock 被 Martin Kleppmann 喷。
> **看完这层,你看任何分布式中间件都能直接看到底**——ZooKeeper / etcd / Kafka / TiDB / Cassandra / Spanner,全是这层往上长出来的。

---

## 设计思路

六层结构,从「机器视角的不可能」到「工程上的妥协与组合」:

```
心智 + 基础(为什么分布式难 / 八大谬误 / CAP / 一致性 vs 共识)
   → 时间与因果(物理时钟 / Lamport / 向量时钟 / HLC / TrueTime)
   → 共识与复制(FLP / Paxos / Raft / ZAB / 拜占庭)
   → 一致性模型(Linearizability / Sequential / Causal / Eventual / CRDT)
   → 分布式事务(2PC / 3PC / Saga / TCC / Percolator / Spanner)
   → 工程基石(分布式锁 / 选主 / Gossip / 服务发现 / 脑裂)
```

**这个系列不是 DDIA 重写**——DDIA 体量大、面向"全栈数据系统",**本系列偏"理论纵深 + 工程映射"**:每个抽象都讲清楚「为什么需要 → 数学/算法是什么 → 落到哪个工程系统上 → 局限与代价」,看完能直接对着 etcd / TiDB / Kafka 的设计文档"看懂他们在抄哪一篇论文"。

不展开到证明每一步推导,**而是讲清楚"应用工程师 / 中间件作者需要懂的理论图景"**——你不一定要会重写 Paxos,但选型、排错、容灾设计、面试白板都要懂这些。**95% 的"诡异分布式 bug"最终都落到这一层的某个不变量被破坏。**

## 读者画像

本系列假设读者已具备:

- 写过至少一年后端代码、用过至少一个分布式中间件(Redis / Kafka / MySQL 主从 / ZK / etcd 任一)
- 看过 systemDesign 第 04 篇(CAP 与一致性谱)或等价材料,知道一致性"有强弱之分"但说不清楚
- 不要求会读 TLA+,但要求"看到伪代码 + 状态机能跟下来"

如果你完全没碰过 Redis 主从延迟、MySQL 半同步、Kafka ISR 这些工程现象,建议先把 systemDesign / backendLearning 过一遍,再来这层补"为什么"。

## 写作模板

每篇遵守"为什么 → 是什么 → 工程对应 → 局限"的四段式:

```
1. 为什么需要这个东西
   - 不要它的世界长什么样(具体故障 / 不变量被破坏的例子)
   - 加了它解决了什么、付出了什么

2. 它到底是什么
   - 关键不变量(invariant)
   - 算法关键步骤 / 状态机
   - 一张时序图或状态图

3. 工程上谁在用
   - 对应到一个真实系统(etcd / TiDB / Cassandra / Spanner / Kafka …)
   - 该系统在原理基础上做了哪些工程妥协
   - 配置参数怎么对应到理论变量

4. 局限与代价
   - 这个方案不能做什么
   - 真实事故案例(Jepsen 报告 / 公司 post-mortem)
   - 何时该换方案
```

不堆论文证明——**只贴关键不变量、伪代码和时序图**,让读者「看到这一层在保护什么、被什么打破」就够了。

每篇遇到协议、状态机、消息时序,**一定画图**——Paxos / Raft / 2PC / Gossip 没图等于没写。

## 节奏

```
01-04  心智:看完知道"分布式贵在哪 / 为什么很多需求不该上分布式"
05-08  时间:看完不会再写"用本地时间戳做全局顺序"
09-15  共识:**最值钱的一层**,Paxos / Raft / ZAB 是所有协调服务的底
16-19  一致性模型:**最容易被忽略**,大多数"诡异 bug"是模型选错
20-24  事务:**面试和真实工作的硬通货**,从 2PC 一路到 Spanner
25-30  工程基石:每个中间件都在用的零件,选型必懂
```

**一个硬指标**:看完 09-15 + 16-19 + 20-24 这 16 篇,你应该能在白板前讲清楚:

- "为什么 Raft 比 Paxos 火,但 Paxos 不会被淘汰"
- "为什么 Redlock 不安全,Martin Kleppmann 的那篇喷文到底在喷什么"
- "Spanner 为什么要装原子钟,装不起原子钟的人怎么用 HLC 模仿"
- "为什么 Saga 不是 2PC 的替代品,它换走了什么"
- "为什么 ZK 是 CP 但 Eureka 选 AP,业务场景该怎么对应"

能讲清楚一题这个系列就值了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「Paxos 论文逐句翻译 / Raft 论文翻译」——不是论文导读,是工程师视角
2. 「先讲个 Google / Amazon 大故事」——直接进重点,案例只在「工程对应」一段引用
3. 抄 DDIA / 抄分布式课件——那些资料都在,这里只补它们没串起来的"为什么 + 工程映射"
4. 「这是面试常考」——读者来这里是为了真懂,不是背 CAP 三选二
5. 不画图就讲 Paxos / Raft / 2PC / Gossip / 向量时钟——这些**不画图永远讲不清楚**
6. 写业务系统设计(IM / 订单)——那是 systemDesign 的活
7. 深入到存储引擎 LSM / B+ Tree / WAL 细节——留给未来的 dbLearning,本系列只到"复制 / 事务 / 共识"
8. 区块链经济学 / 代币 / Layer2 ——拜占庭只到协议层(PBFT / Tendermint),不碰公链生态

## 立场声明

本系列**只服务于"理解 + 选型 + 排错"**——不是教你重写一个 Raft 库,而是教你看懂你正在用的那个 Raft 库为什么这么写、什么时候会坏、坏了往哪里查。

所有算法描述限制在伪代码与时序图层面,**不提供生产级实现**;所有工程映射都指向公开论文、开源系统或公司公开 post-mortem,不引用任何未公开的内部资料。
