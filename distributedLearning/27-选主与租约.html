选主与租约 ​

「多副本只能一个干活」是分布式系统里最常见的需求——K8s controller manager / scheduler 副本不能同时调度同一个 Pod、MySQL 主从只能一个主接写、GFS 上同一个 chunk 只能一个 primary 协调写顺序、Kafka 一个 partition 只能一个 leader。这就是"选主"。选主底下的实现机制是"租约"(Lease)——给主一段时限的承诺、期内独占、过期续约。租约不是心跳的简单升级——心跳是"被动确认对方还活着",租约是"主动拿到一段时限的权力授权";这一对的差别决定了能不能"避免双主"。这一篇把选主、租约、双主预防 三件事讲透——没看清这一篇,你设计的高可用系统大概率会双主导致数据写花。

一句话先记住:选主 = 共识的特例(对"谁是主"达成共识)。租约 = 带时限的授权——授权方在 lease 期内必须遵守"不撤销、不另发",被授权方在 lease 期外必须假设"权力已失"。双主预防的四件套:租约期 > 心跳超时、新主等老租约过期、Fencing Token 兜底、Quorum 选主单分区下不可能双主。租约长度的权衡是分布式系统的永恒主题——短了误判多,长了切换慢,Raft election 150-300ms / 锁 lease 5-10s 是工程经验值。

一、为什么需要选主——三个真实场景 ​

1.1 K8s Controller Manager:多副本只能一个干活 ​

kube-controller-manager 部署 3 副本(高可用)
3 个都连着 etcd
但调度 Pod 只能一个干 —— 如果 3 个都给同一个 Pod 调度,会调度到 3 个不同 Node

解法:3 副本选主,只有"主"实际干活,其他 2 个待机。主挂了,另一个秒级接管。

1.2 MySQL 主从:写只能去主 ​

MySQL 一主两从
所有写都打主 → binlog 复制到从
主挂了 → 选一个从升主
        如果同时两个从都升主 → 两个主同时接写 → 数据写花

解法:主挂时只能有一个新主——MHA / Orchestrator / Group Replication 都在解决这个。

1.3 GFS chunk 写顺序:同一个 chunk 一个 primary ​

GFS 一个 chunk 有 3 个 replica
客户端写 chunk → 必须按"所有 replica 看到同样的写顺序"
否则 replica 之间数据散

解法:Master 给某个 replica 颁发"chunk lease"成为 primary——所有写经过 primary 协调,primary 决定顺序。lease 期间 primary 独占,过期重新颁发。

三个场景的共同需求:一段时间内,只有一个"主"在工作;主挂了,能在合理时间内换新主;切换期间绝对不能"双主"。

二、选主的实现方式(四类) ​

2.1 基于共识算法(Paxos / Raft) ​

Raft 内置选主(13 篇详讲):Leader Election 是 Raft 协议的一部分——节点随机超时后变 Candidate,RequestVote RPC 拿 majority → 当选。

状态机:
  Follower → 选举超时 → Candidate → 拿到 majority → Leader
                                  ↓ 拿不到
                                  回到 Follower

Raft term 单调递增——每个新主带新 term,旧主收到更高 term 自动下台。

典型系统:etcd、TiKV、CockroachDB、Consul、所有 Raft-based 系统。

2.2 基于 ZK / etcd "抢锁" ​

不自己写 Raft,把选主活外包给 ZK / etcd:

所有候选实例都尝试在 ZK 创建临时节点 /leader
  - 只有一个成功 → 它是 leader
  - 其他失败 → 它们 watch /leader,等它消失再抢

优势:应用层不用懂共识——拿来主义,几行代码搞定选主。

典型系统:Kafka 早期 controller、HBase Master、K8s leader-election(用 etcd lease)。

2.3 Bully 算法(粗暴) ​

1982 年提出,直接按 ID 大者赢:

1. 节点 X 怀疑主挂了
2. X 向所有 ID > 自己 的节点发"选主"
3. 没人回应 → X 当主,广播
4. 有人回应 → 那人接管选主

问题:消息量 O(N²),无法防"网络抖动导致频繁重选"。老古董,几乎没现代系统用。

2.4 Ring 算法 ​

节点组成环,沿环传"选主"消息,O(N²) 消息量,节点变动适应差。理论价值大,工程很少用。

2.5 四类对比 ​

方式消息复杂度实现复杂度一致性典型
Paxos / RaftO(N)高强(majority)etcd / TiKV / CockroachDB
ZK / etcd 抢锁O(N)低强(依赖底层)K8s / Kafka controller
BullyO(N²)中弱(网络分区出错)老课本
RingO(N²)中弱老课本

工程界 99% 的选主:要么自己用 Raft(像 etcd)、要么抢 ZK/etcd 的锁(像 K8s)——Bully / Ring 在生产几乎绝迹。

三、Lease(租约):选主的"权力凭证" ​

3.1 定义 ​

Lease = 授权方给被授权方一段时限的承诺:

"在 T_grant 到 T_grant + duration 这段时间内,
 我承诺把【X 权力】只给你一个;
 你也必须在 T_grant + duration 之后假设这个权力已经失效,
 想继续用就来续约。"

两侧的不变量:

  • 授权方(给 lease 的):lease 期内不撤销、不另发(就算被授权方失联也要等过期才能给别人)
  • 被授权方(拿 lease 的):lease 期外不假设自己还有权力(就算心跳还在通也不行)

3.2 Lease vs 心跳:权力方向的差别 ​

心跳:

A → B:"你还活着吗?"
B → A:"活着"
→ A 知道 B 活着,但 A 没"授权"B 做任何事

Lease:

A → B:"申请 leader 权 10 秒"
A 内部记录:B 在 T+10 之前是 leader
B 拿到 lease → 在 T+10 之前确信自己是 leader
T+10 之后 B 必须停止"以 leader 身份行事",除非续约成功

关键差别:心跳是被动确认对方状态,lease 是主动获得权力授权——lease 自带"权力开始/结束时间"的承诺,心跳没有。

3.3 Lease 续约时序图 ​

T=0   B 申请 lease(duration=10s)
T=0.1 A 授权,记录"B is leader until T=10.1"
T=5   B 续约请求
T=5.1 A 续约,记录"B is leader until T=15.1"
T=10  B 又续约
T=10.1 A 续约,记录"B is leader until T=20.1"

如果 B 在 T=12 挂了:
T=12  B 不再发续约
T=15.1 A 等到原 lease 过期(没续约)
T=15.2 A 才能把 lease 给别人

→ 这就是关键的"等老 lease 过期"环节,防双主

3.4 Lease 续约的伪代码 ​

python
# 被授权方(client)
def lease_holder_loop(client, lease_duration=10):
    lease_id = grant_lease(lease_duration)
    last_renewal = now()
    while True:
        if now() - last_renewal > lease_duration / 3:
            # 在 lease 的 1/3 时间点就续约
            try:
                renew_lease(lease_id, lease_duration)
                last_renewal = now()
            except RenewFailed:
                # 续约失败 → 我必须放弃 leader 身份
                step_down()
                return
        if now() - last_renewal > lease_duration:
            # 续约迟到了 → lease 过期了 → 主动放弃
            step_down()
            return
        do_leader_work()


# 授权方(server,简化版)
def grant_lease(duration):
    lease = Lease(id=uuid(), expire=now() + duration)
    store(lease)
    return lease.id

def renew_lease(lease_id, duration):
    lease = load(lease_id)
    if lease.expire < now():
        raise LeaseExpired()  # 不允许续过期的 lease
    lease.expire = now() + duration
    save(lease)

关键细节:

  • 被授权方在 lease 的 1/3 时间点就续约——留缓冲应付网络往返
  • 续约失败立刻 step_down——别等 lease 自然过期
  • 授权方不允许续过期的 lease——必须重新申请

四、Lease 长度的取舍 ​

4.1 短 Lease(1-2s) ​

优势:主挂了切换快,主故障到新主上任 < 2s 劣势:网络抖动 / GC 暂停极容易误判——网络卡 1.5s 就被认为挂了,频繁切换

4.2 长 Lease(30s+) ​

优势:网络抖动不会误判,主挂了切换也是 30s劣势:切换慢,主真的崩了要等半分钟才有新主

4.3 工程经验值 ​

系统选主超时 / Lease 长度备注
Raft 选举150-300ms(随机)论文推荐,工程上 etcd 默认 1s
etcd lease默认 60s(可配,典型 5-30s)K8s leader election 用 15s
Chubby lease12s(Google 论文)长一点稳一点
GFS chunk lease60schunk 写不频繁,长一点
K8s leader electionlease=15s, renew=10s, retry=2s默认参数
MySQL Group Replication5s detection短到能用
Kafka Controller6s sessionZK session timeout

4.4 一个公式 ​

Lease duration ≈ N × (心跳间隔)
其中 N 通常取 3-5

例:
  心跳每秒一次 → lease 3-5 秒
  心跳每 5 秒一次 → lease 15-25 秒

直觉:让被授权方有 3-5 次"续约机会",任意一次成功就续上;全失败才算真的挂了。

五、GFS Master 的 Lease:经典工程案例 ​

5.1 背景 ​

GFS 一个 chunk 有 3 个 replica,客户端写 chunk 要保证 3 个 replica 看到同样的写顺序——这是"原子写"的要求。

5.2 解决方案:chunk lease ​

Master 给某个 replica 颁发 60s 的 chunk lease,称为 primary:

1. 客户端要写 chunk_X
2. 客户端问 Master:谁是 chunk_X 的 primary?
3. Master:replica_A 在 T+60 之前是 primary
4. 客户端推送数据到所有 replica(任何顺序)
5. 客户端发"提交"到 primary(replica_A)
6. primary 决定写顺序,广播给其他 replica
7. 所有 replica 按 primary 决定的顺序写
8. primary 回复客户端 "成功"

5.3 为什么用 lease 而不是用 Master 协调每次写 ​

Master 是 GFS 的单点协调者——每次写都问 Master 决定顺序,Master 撑不住。

Lease 把"决定顺序"的权力下放给 primary 60 秒——这 60 秒内 Master 不参与每次写,只在 lease 过期时介入一次——Master 流量降 100 倍以上。

5.4 Lease 过期 / primary 挂的处理 ​

情况 A:primary 正常,lease 快到期
       primary 主动向 Master 续约 → 续 60s

情况 B:primary 挂了,lease 还没到期
       Master 必须等 lease 过期才能给别的 replica 发新 lease
       为什么?——如果 primary 没真挂(只是网络分区),Master 又发给别的 replica,会双 primary

情况 C:primary 挂 + lease 过期
       Master 给另一个 replica 发新 lease
       新 primary 写的数据带新 version
       老 primary 即使醒来想写也会被 chunk server 拒(version 不对)
       —— 这就是 Fencing Token 的活

5.5 这套架构的精妙 ​

  • 数据流和控制流分离——大数据通过 replica 直接传(避免 Master 瓶颈),控制信号(顺序、提交)经过 primary
  • lease 让 Master 短期内"撒手不管"——降低 Master 流量
  • version 做 fencing——老 primary 即使活过来也写不进去

GFS 论文 (Ghemawat, Gobioff, Leung, 2003) 的这套 lease 设计影响了所有后续分布式存储——Bigtable / Spanner / HBase / Ozone 全在用类似机制。

六、避免双主——四件套 ​

双主(split-brain)是高可用系统最大的事故源——两个 primary 同时接写,数据写花后几乎无法恢复。避免双主的工程纪律有四条,缺一不可:

6.1 第一件套:Lease 期 > 心跳超时 ​

反例(双主路径):

心跳超时 5s,lease 5s

T=0   主 A 心跳正常,有 lease
T=4.5 网络抖动,A 的心跳没到 → 监控认为 A 挂了
T=5   监控发起选举,选出新主 B,给 B 发 lease
T=5   A 的旧 lease 还没过期(到 T=5)且 A 自己以为还在
T=5.1 A 和 B 都以为自己是主 → 双主!

纪律:Lease 必须明显大于心跳超时——让"被认定挂掉"先于"lease 过期"足够长时间,有时间让旧主自己 step_down。

工程标准:lease ≥ 2 × 心跳超时。

6.2 第二件套:新主必须等老租约过期 ​

纪律:新主上任前要等"老主的 lease 一定已过期"——即使老主不知道自己挂了,只要 lease 过期,他也不应再做主的事。

T=0    监控发现老主 A 挂(心跳超时)
T=lease_expire  A 的 lease 一定已过期
T=lease_expire + 安全冗余  B 上任为新主

→ 中间这段空窗期(可能几秒到十几秒)系统没主,业务卡
→ 但绝不双主

典型实现:Raft 用 election timeout 保证——新 leader 必须在比老 leader lease 更长的时间后才能发 AppendEntries(防 split brain);ZK 的 ephemeral 节点在 session 过期后才删,新主再创建。

6.3 第三件套:Fencing Token 兜底 ​

就算前两件套都做对,老主在 GC 暂停 / 时钟错乱下可能仍认为自己是主(Kleppmann 反例):

T=0   老主 A 拿 lease,token=33
T=2   A JVM 长 GC 暂停 20s
T=12  A 的 lease 过期
T=12  新主 B 上任,token=34
T=15  B 写存储 (token=34)
T=22  A GC 结束,以为自己还是主,写存储 (token=33)
T=22  存储检查:33 < 34,拒绝 A 的写

→ 即使 A 误以为是主,写不进去

这是 26 篇分布式锁那一章讲的 Fencing Token,在选主场景同样必不可少。

实现:

  • etcd:用 mod_revision 作 token
  • ZK:用 zxid 或 sequence number
  • GFS:每次发新 lease 时 chunk version +1,chunk server 拒旧 version
  • Spanner:用 Paxos epoch number 做 token

6.4 第四件套:Quorum 选主保证"单分区下不可能双主" ​

Raft / Paxos 的关键性质:新主必须拿 majority 投票才能当选。

网络分区时:

5 节点集群分裂为 [2 节点] 和 [3 节点] 两边

[2 节点] 想选主 → 凑不到 3 票 majority → 无法选主 → 那边不可用
[3 节点] 想选主 → 拿到 3 票 → 可以选主 → 那边能服务

→ 永远只有一边有主,另一边停服
→ 单分区下"绝对不可能双主"

这是 CAP 里"分区时选 C"的代价——少数派分区不可用,但不会双主。

对比:Redis Sentinel(早期版本)、MySQL MHA——没严格 quorum,网络分区下可能脑裂。

6.5 双主预防图 ​

                    保命的"四件套"
        ┌─────────────────────────────────────────┐
        │                                         │
        │ 1. lease 期 > 心跳超时                    │
        │    (老主 lease 过期前不会有新主)        │
        │                                         │
        │ 2. 新主等老 lease 过期                    │
        │    (空窗期内绝对不双主)                  │

七、Chubby:Google 选主与租约的工业典范 ​

7.1 Chubby 是什么 ​

Google 内部的分布式锁服务(Burrows 2006 论文),所有 Google 内部服务的"主选举"都靠它:

  • BigTable 的 master 选举
  • GFS 的 master(也用 Chubby)
  • MapReduce 的协调器选主

7.2 设计哲学:粗粒度锁 ​

Chubby 不是"高频锁",是"主选举锁"——锁的持有期长(小时到月),获取频率低。

单个 Chubby cell(集群)5 个副本,Paxos 共识,每秒几千次操作——够慢但够稳。

7.3 Lease 模型 ​

  • 客户端连 Chubby:12s session lease
  • session lease 内可以做任何操作(读、写、持锁)
  • session lease 过期前要"jeopardy"(危险期)
  • jeopardy 内还能续 → 续上;45s 还没续上 → session 死,所有锁释放

7.4 客户端看 lease 过期 ​

关键设计:Chubby 把 lease 过期时间也告诉客户端——客户端用"客户端本地时间 - 通信往返"来安全估算"我的 lease 还有多久"。

客户端发请求 → 服务端在 T_s 回 "lease 有效到 T_s + 12s"
客户端本地时间 T_c 收到 → 客户端保守估计 "我的 lease 安全到 T_c + (12s - 2 × RTT)"
→ 减掉两倍 RTT 作为安全冗余

这是 lease 协议的"通用工程模式"——永远从最保守的视角算 lease 还剩多久。

7.5 Chubby 影响了什么 ​

  • Apache ZooKeeper 直接抄 Chubby(Yahoo 2008)——API、session、ephemeral node 全是 Chubby 的思路
  • etcd 是 ZK 的现代化重写(Go + Raft)
  • K8s leader-election 跑在 etcd 上,思路与 Chubby 一脉相承

八、K8s leader-election:现代标准案例 ​

8.1 场景 ​

kube-controller-manager / kube-scheduler 多副本部署,只能一个干活。

8.2 实现 ​

用 etcd 的 coordination.k8s.io/Lease 资源:

yaml
apiVersion: coordination.k8s.io/v1
kind: Lease
metadata:
  name: kube-controller-manager
spec:
  holderIdentity: "kube-controller-manager-pod-1"
  leaseDurationSeconds: 15
  renewTime: "2026-05-11T15:00:00Z"
  acquireTime: "2026-05-11T14:50:00Z"
  leaseTransitions: 3

字段含义:

  • holderIdentity:当前持有者
  • leaseDurationSeconds:lease 长度(K8s 默认 15s)
  • renewTime:上次续约时间
  • leaseTransitions:历史上换过多少次主

8.3 选主代码框架(Go client-go) ​

go
leaderelection.RunOrDie(ctx, leaderelection.LeaderElectionConfig{
    Lock: &resourcelock.LeaseLock{
        LeaseMeta: metav1.ObjectMeta{
            Name:      "kube-controller-manager",
            Namespace: "kube-system",
        },
        Client:     kubeClient.CoordinationV1(),
        LockConfig: resourcelock.ResourceLockConfig{Identity: id},
    },
    LeaseDuration: 15 * time.Second,
    RenewDeadline: 10 * time.Second,
    RetryPeriod:   2 * time.Second,
    Callbacks: leaderelection.LeaderCallbacks{
        OnStartedLeading: func(ctx context.Context) {
            // 我是主了,开始干活
        },
        OnStoppedLeading: func() {
            // 我不是主了,停止干活
        },
    },
})

8.4 三个时间参数 ​

参数默认值含义
LeaseDuration15slease 长度——超过这个时间不续约就被认为挂
RenewDeadline10s我必须在这个时间内续约成功——否则主动 step_down
RetryPeriod2s续约请求的重试间隔

关系:RenewDeadline < LeaseDuration ——留缓冲应付续约失败。

8.5 K8s 这套的"工程美" ​

  • 不写一行 Raft 代码——把选主活全甩给 etcd
  • lease 资源是 K8s 原生 API——用 kubectl 就能查谁是 leader
  • 多副本应用通用模式——所有 Operator、Controller 都按这套写

十、踩坑提醒 ​

  1. 以为选主就是"谁先注册谁是主"——没有 quorum / lease 的选主,网络抖动就会双主;正确选主必须 quorum 投票 + lease 时限。
  2. Lease 期 < 心跳超时——老主 lease 过期前就被换掉,双主必然发生;lease ≥ 2 × 心跳超时。
  3. 新主上任不等老 lease 过期——老主还以为是主,新主已开始写,两个主同时干;Raft / ZK 的 majority + ephemeral 本质都在做这个。
  4. 选主用了但下游不验 Fencing Token——Kleppmann 反例在选主场景一样发生(老主 GC 醒来继续写);重要存储必须验 token / revision / epoch。
  5. Lease 长度全设一样——心跳频率、选主频率、业务耗时对 lease 的需求不同;Raft 内部 election 100ms 量级,业务 leader lease 5-30s 量级,不同场景分开调。
  6. 客户端不在 lease 1/3 时间点提前续约——到 lease 9 秒才续,网络抖动一下续不上,就被换主了;留至少 1/3 的余量。
  7. 客户端续约失败不主动 step_down——等着 lease 自然过期才下台,这段时间业务还以为有主;续约失败立刻停止 leader 行为。
  8. MySQL MHA / Sentinel 当 CP 用——这两套都没严格 quorum,生产网络分区下双主事故有公开 post-mortem;正确性场景用 Group Replication / Spanner-like 系统。
  9. K8s leader-election 用了但 controller 实现不幂等——主切换时,新老主都做过同样的 reconcile,需要 controller 本身幂等;lease 不能完全防止"重复执行"。
  10. 以为"Quorum 防脑裂"就万事大吉——Quorum 防的是"同时双主",不防"GC + 时钟"造成的"老主自以为是主";还要 Fencing Token + 业务幂等。

收束:工程基石的核心 ​

这是工程基石三件套的最后一篇。25(一致性哈希)/ 26(分布式锁)/ 27(选主与租约)三件,撑起了几乎所有分布式中间件的内部"协调层":

  • 一致性哈希告诉你"数据放哪台"
  • 分布式锁告诉你"互斥怎么做"
  • 选主与租约告诉你"哪台机器是当前的话事人"

这三件东西用对了,中间件能跑稳;用错任意一件,数据写花、双主、超卖——这些都是真实生产事故里反复出现的剧本。

下一篇 28-Gossip 协议:前面几篇都假设"有一个集中协调者(Master / Leader / Lock Service)",但很多场景不需要中心,节点间互相'八卦'就能达成最终一致——Cassandra / Consul / Serf 的成员管理 / Bitcoin 的区块传播都靠这个。Gossip 是分布式系统里少有的"指数级收敛"的算法——logN 步内覆盖全网;下一篇拆 SWIM、反熵、Phi-Accrual 故障检测,讲清楚"为什么 Cassandra 几千节点能不需要中心协调"。
