# 一致性模型谱

前面 09-15 篇讲了**共识**——多个节点怎么对一个值达成一致;但应用层真正要面对的不是「值」,是**「并发操作的历史」**——「我刚写完,另一个客户端读到的是新值还是旧值?」「A、B 两个写并发,后来者会不会看到一个我自己都没经历过的中间态?」**这一整套关于"合法历史"的契约,叫一致性模型(consistency model)**——它跟 ACID 里的 C 没半毛钱关系,跟"数据正确"也不是一回事,**它是数据库 / 存储系统跟你签的合同**:「你能看到的并发执行结果只能长这样,不能长那样」。

> 一句话先记住:**一致性模型不是"数据对不对",是"程序员看到的并发历史合不合法"——它是一份契约,规定了哪些读结果可以合理解释,哪些是 bug**。**模型从强到弱是一条谱**——Linearizable ⊃ Sequential ⊃ Causal ⊃ Read-Your-Writes ⊃ Eventual,**越强越像单机,越弱越能扛延迟和分区**。**真实工程不是"选一档"**——同一个系统不同操作走不同档(余额线性、点赞最终),**全选最强系统跑不动,全选最弱用户骂街**。

---

## 一、先把术语清干净:三个 C 不要再混

写之前先把"消歧"做掉,这是这一层最容易踩的雷:

| 缩写 | 全名 | 在哪一层 | 在保护什么 |
| --- | --- | --- | --- |
| **C(ACID)** | Consistency | 数据库事务 | 事务后约束(外键、唯一键、check)不被破坏 |
| **C(CAP)** | Consistency | 分布式系统 | 实质等同 Linearizability(单对象/单操作级别) |
| **一致性模型** | Consistency Model | 并发与分布式 | 多客户端对同一系统看到的"操作历史"合不合法 |

**这一篇讲的是第三个**——不是事务约束,也不只是 CAP 里那一档,**是一整张"从最严到最松"的谱**,CAP 的 C 只是这条谱最顶端那一档。

> 第 03 篇讲过「一致性、共识、一致性模型」三件套,**共识是机器达成一致的算法,一致性模型是程序员能依赖的契约**。共识是手段,模型是目的。

---

## 二、为什么需要一个"谱"——直接全用最强不行吗?

朴素想法:都用 Linearizability(看起来像单机)不就行了?

**问题**:Linearizability 在分布式下要付出三个代价:

1. **每次读写都要走多数派共识**——10ms+ 起步,跨地域 50-100ms
2. **分区时少数派必须拒服务**——CAP 严格证明(下一篇详细推)
3. **吞吐天花板低**——所有写串行化进一个全局序

**真实业务的"可容忍延迟 / 错误率"差异巨大**:

```
┌─────────────────────┬───────────────────────────────────────┐
│ 业务                │ 必须的一致性档位                       │
├─────────────────────┼───────────────────────────────────────┤
│ 银行转账            │ Linearizable(余额不能错)              │
│ 抢购扣库存          │ Linearizable(不能超卖)                │
│ 评论列表            │ Causal(评论不能在被回复者之前出现)    │
│ 朋友圈刷新          │ Monotonic Read(刷新不能"丢条")        │
│ 点赞数              │ Eventual(几秒内对齐就行)              │
│ CDN 内容            │ Eventual(分钟级一致都没事)            │
└─────────────────────┴───────────────────────────────────────┘
```

如果你把朋友圈的点赞数也跑 Raft,**每秒抗不住几千 QPS**;反过来,如果你把余额跑 eventual,**用户能在 ATM 取出比账上多 100 块的钱**。

> **一致性是一组档位,不是一个开关**。**应用层的工作 = 给每个数据点选档位**。

## 三、谱系全景图(强 → 弱)

先一张全景:

```
                  强 ─────────────────────────────────── 弱
                  ↓                                       ↓

   Strict ⊋ Linearizability ⊋ Sequential ⊋ Causal ⊋ PRAM ⊋ Eventual
                  │                │           │       │
                  │                │           │       └─ 收敛即可,中间任意
                  │                │           │
                  │                │           └─ 因果相关有序,并发随意
                  │                │
                  │                └─ 所有节点同序,但和真实时间无关
                  │
                  └─ 每个 op 看起来在 [invoke,resp] 区间内某瞬时原子完成
                     真实时间偏序被保留

   并列的"客户端会话保证"(session guarantees,Terry 1994):
     Read-Your-Writes  / Monotonic Reads / Monotonic Writes / Writes-Follow-Reads
   ── 这四条是"客户端体验"维度,不在主谱上,但常被叠加在弱模型之上
```

### 3.1 包含关系(数学)

```
Strict           ⊋  Linearizability:多了一个"瞬时同步"
                                    (上帝视角的绝对时间)
Linearizability  ⊋  Sequential:多了"保留真实时间偏序"
Sequential       ⊋  Causal:多了"全局顺序"
Causal           ⊋  Eventual:多了"因果保持"
```

任何 Linearizable 历史都是 Sequential 历史;任何 Sequential 都是 Causal;**反之不成立**。所以"满足强档的系统自动满足弱档",但反过来不行——这是后面选型的根基。

### 3.2 各档子要记住一句话

| 模型 | 一句话 |
| --- | --- |
| Strict | 全网瞬时同步(理论玩具,工程上做不到) |
| Linearizability | 每个操作看起来"在某一瞬间原子完成",真实时间偏序保留 |
| Sequential | 所有客户端看到同一个全局顺序,但顺序可以和真实时间无关 |
| Causal | 因果相关的操作有序,并发的可以乱序 |
| Read-Your-Writes | 你自己刚写的,自己后续一定读得到 |
| Monotonic Reads | 一旦读到版本 v,后面不会读到比 v 更旧的版本 |
| Monotonic Writes | 同一客户端的写按发起顺序生效 |
| Writes-Follow-Reads | 你读到 v 后再写的 v',v' 一定排在 v 之后 |
| Eventual | 给足时间最终收敛(中间随便) |

## 四、Linearizability(线性一致性)

最严的工程档。**直觉**:外部观察者看不出系统是分布式的——所有操作就像在一台原子机上排好队跑完。**下一篇 17 整篇都在讲它**,这里只列要点:

- 每个操作占一个 `[invocation, response]` 区间
- 选区间内一个点,作为该操作"原子完成的瞬间"
- **不变量**:真实时间偏序保留——op1 在 op2 开始前完成,则线性化序里 op1 必须在 op2 之前
- **工程**:Raft / Paxos 单 key 写 + leader read(或 read index / quorum read)
- **代价**:跨地域 50-100ms 延迟,分区时不可用

**实例**:etcd / ZooKeeper 默认单 key 写、Spanner 的 External Consistency(比 Linearizability 更严,包含真实时钟顺序)。

---

## 五、Sequential Consistency(顺序一致性)

Lamport 1979。**比 Linearizable 弱在哪**:**不要求保留真实时间**。

```
真实时间轴:

  Client A:  [-- write x=1 --]
  Client B:                       [-- read x ---]  返回 0

线性一致:不合法(B 的读完全晚于 A 的写完成,必须看到 1)
顺序一致:合法(全局顺序可以是 read→write,只要所有客户端都这么看)
```

**经典例子**(单核 CPU 视角):

```
进程 P1: x = 1; print(y)   →  打印 0
进程 P2: y = 1; print(x)   →  打印 0

可能吗?
顺序一致:不可能(任何全局顺序都至少有一个赋值在另一个 print 前)
但弱一致(如 PRAM):可能(每个进程看自己写,别人未必看到)
```

**工程上**:**顺序一致很少作为主推档**——它比 Causal 严,但比 Linearizable 弱,大多数系统要么直接给 Linearizable,要么退到 Causal。**多核 CPU 内存模型才是 Sequential Consistency 的真正舞台**(Java volatile 不是 Sequential,要 happens-before 才到 Sequential 级别)。

## 六、Causal Consistency(因果一致性)

**因果相关有序,并发的随意**。第 18 篇整篇讲它,这里先给直觉:

```
case 1(因果相关):
  Alice: 发帖 P
  Bob:   看到 P,回复 R (R 因果依赖 P)
  → 任何客户端必须先看到 P 再看到 R
     不能"看到回复,看不到原帖"

case 2(并发):
  Alice: 发帖 P1
  Bob:   发帖 P2(同时进行,互不知道)
  → 任何客户端可以先看到 P1 也可以先看到 P2,顺序不固定
```

- **判定因果**:happens-before(第 06 篇),靠向量时钟 / 版本向量(第 07 篇)
- **代价**:跟踪依赖,**元数据膨胀**(每条消息要带依赖列表)
- **好处**:**分区下仍可用**——Mahajan 2011 证明因果可以和 P 并存,**绕开了 CAP 的死结**

**工程**:COPS / Bolt-on Causal / Yjs / Google Docs / Figma 协同编辑。

## 七、客户端会话保证(Session Guarantees)

Terry et al. 1994,在 **Bayou** 系统提出。**不是描述系统级历史合法性,而是单个客户端的"观感保障"**——叠加在弱一致系统上的常用补丁。

### 7.1 Read-Your-Writes(读到自己写)

**契约**:同一客户端,write(x=v) 后,后续 read(x) 一定返回 v 或更新值。

**反例**:

```
你刚改了头像 → 主库
刷新页面 → 从库(异步复制还没同步)
看到旧头像 → 用户:"我改了没生效啊?!"
```

**实现方式**:

- **粘性会话**:同一用户的读写都打到主库 / 同一副本
- **读写时间戳**:客户端记下"我写完后的版本号 V",读时要求服务端返回 ≥ V 的版本
- **写后强制主读**:写完后 N 秒内,该用户的读强制走主库

### 7.2 Monotonic Reads(单调读)

**契约**:一旦读到版本 v,后续读不会返回比 v 更旧的版本(可能比 v 更新)。

**反例**:

```
9:00:00  刷新朋友圈,看到 100 条
9:00:05  又刷一次,只看到 95 条 ←  退化!
       (原因:第二次刷打到一个复制更慢的从库)

用户感知:"我朋友圈丢条了!"
```

**实现**:同一客户端的读固定路由到同一副本,或带版本号要求服务端"≥ 已读版本"才返回。

### 7.3 Monotonic Writes(单调写)

**契约**:同一客户端的写按发起顺序对外可见。

**反例**:

```
A 客户端:write(x=1); write(x=2)
B 客户端读到的顺序:x=2, x=1  ←  违反

用户感知:"我先改了名字 'A',又改成 'B',别人看到的却变成 'A'。"
```

### 7.4 Writes-Follow-Reads

**契约**:客户端读到 v 之后再写的 v',v' 在全局顺序里排在 v 之后。

**用于**:评论引用的语义——你读了原帖才写回复,回复要在原帖之后。

> **这四条非常常用**,**90% 的"用户感知 bug"都来源于会话保证缺失,而不是底层模型不对**——主库强一致没用,从库没读到自己写,用户就觉得"系统坏了"。

## 八、各主流系统的默认档位

横向对比一眼能看完(**默认值**,大多数都可调):

| 系统 | 默认一致性 | 可调到 | 备注 |
| --- | --- | --- | --- |
| **etcd / ZooKeeper** | 单 key Linearizable(写),Sequential(读) | etcd 支持线性化读(--consistency=l) | Raft / ZAB |
| **Spanner** | External Consistency(强于 Linearizable) | — | TrueTime,跨 key 全局序 |
| **CockroachDB / TiDB** | Linearizable / Serializable | — | HLC / TSO + Raft |
| **DynamoDB** | Eventual(默认) | Strong consistent read 可选(贵且慢) | NWR,可调 |
| **Cassandra** | Tunable(默认 ONE = Eventual) | LOCAL_QUORUM / ALL = 接近 Linearizable | NWR 三档 |
| **MongoDB** | readConcern = "local"(可能读到未提交主库数据) | majority / linearizable | readConcern + writeConcern 笛卡尔积 |
| **Riak** | Eventual + 向量时钟 / Causal+ | bucket type 可选 strong | 主推 Causal |
| **Redis 主从** | Eventual(主从异步) | WAIT 命令模拟同步(不保证) | 主挂会丢数据 |
| **Cosmos DB** | Session(读自己写)默认 | Strong / Bounded staleness / Consistent prefix / Eventual 五档可选 | **五档明示** |
| **Kafka(单分区)** | 单分区内 FIFO + acks=all → 接近 Linearizable | — | 跨分区无序 |
| **S3** | Read-after-write strong consistency(2020 改) | — | 历史上是 eventual |

**重点观察**:

1. **NoSQL ≠ AP / 弱一致**——MongoDB / HBase 默认偏 CP,Riak 主推 Causal,**别用标签代替思考**
2. **Cosmos DB 是教科书级"五档明示"**——它把上面这条谱直接做成配置项,**学习一致性模型最好的工程参考**
3. **大多数系统是"可调一致性"**——读写时附加参数选档位(`readConcern`、`consistency level`、`ReadOptions{LinearizableRead: true}`)

## 九、选型矩阵:不同业务该走哪一档

这一节就一张表,直接照着对:

| 业务类型 | 数据例子 | 选哪档 | 工程方案 |
| --- | --- | --- | --- |
| 金融账户 | 余额、可用额度 | **Linearizable** | Spanner / TiDB / 单库 + 同步复制 |
| 电商扣库存 | 商品库存、秒杀名额 | **Linearizable**(或预占 + 异步对账) | Redis Lua + 持久化补偿 / DB select for update |
| 订单状态 | "已支付"、"已发货" | **Linearizable**(主键级) | Saga / TCC + 状态机 |
| 用户自己的资料 | 头像、昵称 | Eventual + **Read-Your-Writes** | 主从异步 + 粘性会话 / 版本号 |
| 社交时间线 | feed、评论 | **Causal** | 向量时钟 / 因果存储 / 写时打依赖 |
| 协同编辑文档 | Google Docs / Figma | **Causal + CRDT** | 操作变换(OT)或 CRDT(第 19 篇) |
| 点赞 / 关注计数 | 微博点赞数 | **Eventual + CRDT(G-Counter)** | 异步聚合 |
| 缓存查询 | 商品详情、推荐 | **Eventual** | 缓存 + TTL |
| 配置 / 服务发现 | 注册表、灰度配置 | **Linearizable(写)** + Eventual(读) | etcd / ZooKeeper |
| 锁与选主 | 分布式锁、leader 标识 | **Linearizable + Fencing**(26 篇) | etcd / ZK,**禁用 Redis** |

**反面教材**:

- 用 Cassandra 存余额(默认 eventual)——**双花**
- 用 ZK 存订单详情(写吞吐低)——**写抖动业务挂**
- 用 Redis 主从存订单状态(主挂丢数据)——**订单丢失**
- 用 DynamoDB 默认配置做计数器(eventual)——**计数不准但你以为准**

> **绝大多数线上事故 = 业务需要的档位 > 系统提供的档位**。**选型前先问业务要哪档,再问候选系统能提供哪档**。

## 十、伪代码:三档历史的合法性判定

用 Jepsen 风格的并发历史展示三档差异:

```
clients:  C1, C2     register:  x (initial 0)

history:
  C1: invoke write(x, 1) at  t=1
  C1: response          at  t=3
  C2: invoke read(x)    at  t=2
  C2: response 0        at  t=4
  C2: invoke read(x)    at  t=5
  C2: response 1        at  t=6

时间轴:
  C1: [ write(x,1) ────────────]
  C2:        [read=0]    [read=1]
       1  2  3  4  5  6  时间
```

**逐档判定**:

- **Linearizability**:write 选择"原子点" t∈[1,3],read1 选 t'∈[2,4]。如果 t < t',read1 必须看到 1;但 read1 返回 0,所以要 t > t'。可行(write 原子点选 t=3,read1 t'=2)。**合法 ✓**

- **Sequential**:全局序里同一客户端顺序保留即可。一个合法序:`read1=0, write=1, read2=1`(C2 的两次读分别在 write 前后)。**合法 ✓**

- **Causal**:read1 和 write 无因果关系(read1 没读到 write 的结果),可并发。read2 读到 write 后的值,符合因果。**合法 ✓**

**反例**(违反线性一致):

```
history:
  C1: write(x, 1) [t=1,3]
  C2: read(x)     [t=4,5]  returns 0     ← 违反:read 完全在 write 之后,必须看到 1
```

**写工程系统时**用这种小历史去试,**比读教科书快十倍**——拿 Knossos / Porcupine 跑(下一篇详讲)。

## 十一、一致性 × 性能 × 可用性的工程权衡

**铁律**(PACELC 升级版):

```
正常时:    一致性档位 ↑    →    延迟 ↑     吞吐 ↓
分区时:    一致性档位 ↑    →    可用性 ↓
```

```
                  ┌──────────────────────────────────────┐
                  │  跨地域 Linearizable 一次写:50-150ms │
                  │  跨地域 Causal 一次写:        10-30ms│
                  │  跨地域 Eventual 一次写:       <5ms  │
                  └──────────────────────────────────────┘
```

**优化档位的两条工程招式**:

### 11.1 操作级降档

整库不必全用 Linearizable,**按操作降档**:

```
# 余额变更:Linearizable
db.transaction(isolation=SERIALIZABLE) {
  account.balance -= 100
}

# 余额查询展示:Eventual + 缓存
balance = redis.get("user:1:balance") or db.read(consistency=ONE)
```

### 11.2 数据级分档

**热数据强一致(放主库 / Raft 组),冷数据弱一致(放副本 / 缓存)**:

```
余额、订单状态、扣库存       →  TiDB(Linearizable)
用户档案、商品详情、评论     →  MySQL 异步从库(Eventual)
点赞计数、推荐结果           →  Redis(Eventual)
```

> **没有"系统级一致性",只有"数据级 / 操作级一致性"**——这是 09 年后所有 NewSQL / 互联网架构的共识。

## 十二、踩坑提醒

1. **混淆 ACID 的 C 和一致性模型的 C**——前者是事务约束,后者是并发历史合法性,**根本不是一回事**
2. **以为 CAP 的 C 就是"一致性"全部**——它只是 Linearizability 这一档,**下面还有 6-7 档**
3. **整库选一档**——浪费性能或埋雷,正确做法是按操作 / 数据细分
4. **看到"最终一致"就以为不安全**——大量场景(点赞、推荐、CDN)用最终一致是**最优解**
5. **写完立刻读用从库**——典型违反 Read-Your-Writes,用户感知最差
6. **不做粘性会话**——同一用户每次打到不同副本,Monotonic Read 必然破
7. **以为 Spanner 是 Linearizable**——它实际是 **External Consistency**(更强,包含真实时钟序)
8. **MongoDB 默认 readConcern=local 用了不知道**——读到主库未提交数据,主切换后回滚 = 用户看到的数据消失
9. **Cassandra 用 QUORUM 就当 Linearizable**——QUORUM 写 + QUORUM 读保证 **read-after-write**,**但不等于 Linearizable**(中间窗口仍能违反)
10. **跨表 / 跨 key Linearizable**——单 key 强一致 ≠ 跨 key 强一致,**跨 key 要事务或 TrueTime / TSO**
