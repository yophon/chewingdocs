FLP 不可能定理 ​

1985 年 Fischer、Lynch、Paterson 写了一篇只有 9 页的论文,给"分布式共识"判了死刑——"Impossibility of Distributed Consensus with One Faulty Process"。结论是一句话:在纯异步网络里,只要有 1 个节点会 crash,就不存在一个总能在有限时间内达成共识的确定性算法。这听起来像「共识不可能」,但你天天用的 etcd、ZK、Spanner 全都在做共识——它们怎么活下来的?这一篇拆开 FLP,解释「FLP 不是"共识不可能",是"在某个理论模型下共识不可能"」,Paxos / Raft 都是通过"偷换前提"绕开 FLP——这种"偷换"才是工程的精髓。

一句话先记住:FLP = 异步 + 一个 crash + 确定性 ⇒ 没有总能 terminate 的共识算法。它没说共识不可能,只说在这四个前提下不可能——拿掉其中一个就有出路。工程上三条路:Dwork-Lynch-Stockmeyer 1988 的"部分同步"(Paxos/Raft 走这条);Ben-Or 1983 的"随机化"(BFT 共识用);Chandra-Toueg 1996 的"失败检测器"。Raft 的 election timeout 看起来朴素,本质上就是在偷换 FLP 的"异步"假设——它把异步偷换成"部分同步",从而绕过定理。

一、FLP 在说什么:三句话讲清 ​

1.1 共识问题的最小定义 ​

N 个节点,每个有一个初始值(0 或 1),要达成"共识"——意思是所有非故障节点最后都决定(decide)同一个值,且这个值是某个节点提议过的。

正确的共识算法需要三条性质:

Agreement (一致性):所有正常节点 decide 同一个值
Validity   (有效性):decide 的值必须是某个节点提议过的(不能凭空捏造 42)
Termination(终止性):每个正常节点最终都会 decide(不能永远卡着)

1.2 FLP 的故障模型 ​

"异步系统 + 至多 1 个 crash 故障":

  • 异步:消息延迟没有上界(可以是 1ms,也可以是 1 小时,但终究会到)
  • crash 故障:节点突然停止响应,不会发恶意消息(非拜占庭)
  • 确定性算法:同样输入同样输出,不依赖随机数

1.3 FLP 的结论 ​

在异步系统 + 至多 1 个 crash + 确定性算法的前提下,
不存在一个共识协议能同时满足 Agreement + Validity + Termination。

通俗版:只要网络是纯异步的(没有超时机制),哪怕只有 1 台机器可能挂,你写不出一个共识算法保证总能在有限时间内拍板。

二、为什么这么反直觉:核心难点的直觉 ​

2.1 异步网络里你区分不了"挂了" vs "很慢" ​

节点 A 给节点 B 发消息,等回复。
等了 5 秒还没回。B 是挂了还是网络拥塞?

在同步网络:B 5 秒没回 = B 挂了(因为有"消息一定 1 秒内到"的保证)
在异步网络:B 5 秒没回 = 不知道
            消息可能还在路上,可能 10 秒后才到
            B 可能正在 GC,1 分钟后会回过神

如果 A 决定"B 挂了"就开始新一轮,但 B 其实活着——B 的旧消息后到,可能让协议状态不一致。

如果 A 决定"再等等",但 B 真的挂了——A 永远等下去。

这就是 FLP 的核心:异步网络里"挂了"和"很慢"在算法层面没有区别——任何确定性算法都可能被"无限延迟"卡死。

2.2 Bivalent 状态的构造(证明草图) ​

FLP 论文用了一个精巧的归纳构造,核心概念是 bivalent 配置(双价配置):

  • 0-valent 配置:从这个全局状态出发,无论后续怎么走,最终一定 decide 0
  • 1-valent 配置:从这个全局状态出发,无论后续怎么走,最终一定 decide 1
  • bivalent 配置(双价):既可能走向 0,也可能走向 1(还没"定型")

FLP 证明三步:

1. 初始配置中存在 bivalent 配置
   (如果某些节点初始值不同,系统启动时既可能 decide 0 也可能 decide 1)

2. 任何 bivalent 配置都存在一条"特定的消息延迟序列",
   能让系统永远保持 bivalent —— 永远没"定型"
   (用 1 个 crash 故障 + 调度器无限拖延某条关键消息来构造)

3. 因此存在一条无限执行路径,系统永远不 decide
   → 违反 Termination

通俗说:对抗性的"网络调度者"总能挑选一条消息延迟序列,让系统在"该 0 还是该 1"之间反复横跳——只要还有 1 个节点可能 crash,调度者就有足够的"延迟自由度"维持这种 ambivalence。

这不是一个具体算法的失败,是对所有可能算法的失败——任何你能写出来的确定性共识协议,都能被这个对抗调度构造击败。

三、FLP 三角:四要素必须舍一 ​

把 FLP 重新表述成"四选三":

            异步(Async)
              ╱  ╲
             ╱    ╲
            ╱      ╲
   故障(Fault)──共识(Safety+Liveness)
            ╲      ╱
             ╲    ╱
              ╲  ╱
          确定性(Deterministic)

要 Agreement(Safety) + Termination(Liveness),
+ Async + 1 crash + Deterministic = 不可能(FLP)

舍一才能解:
舍 Async        → 部分同步(Paxos / Raft)
舍 Fault        → 假设 0 个 crash(没意义)
舍 Deterministic → 随机化(Ben-Or)
舍 Liveness 偏弱  → 接受"可能永远不 terminate,但 safety 永远不破"

3.1 "Safety 永远 > Liveness" 是工程信条 ​

Raft 论文里有句名言:

"It is OK to be unavailable, but never inconsistent."

对应到 FLP——所有真实共识算法都保证 Safety(Agreement + Validity)永远不破,但允许 Liveness 在极端情况下暂时挂掉。

Raft 在 split-brain 间隙没法选出 Leader → 短暂不可用,但绝不会两个 Leader 同时提交
Paxos 在两个 Proposer "互相抢" 时可能反复打架 → 短暂不前进,但绝不会两个值都被 chosen

FLP 没法让你"既快又对"——工程接受"偶尔慢一会,但永远对"。

四、出路一:部分同步(Dwork-Lynch-Stockmeyer 1988) ​

Paxos / Raft / ZAB 全都靠这条出路。

4.1 部分同步模型 ​

DLS 1988 论文提出三种网络模型:

Synchronous      消息延迟有已知上界 Δ —— 不现实(网络总会抖)
Asynchronous     延迟无上界 —— FLP 死刑
Partially Sync   延迟最终有上界 Δ,但 Δ 未知,或者只在某个"未知时刻 GST"之后才生效
                 GST = Global Stabilization Time(全局稳定时间)

部分同步的本质:

网络可以任意抖,但总会有"安静下来"的时段——只要在那个时段里能选出 Leader、完成一轮共识,Liveness 就保住了。

4.2 Paxos / Raft 怎么用部分同步 ​

Raft 的 election timeout(默认 150-300ms 随机)就是部分同步假设的实现:

1. Follower 在 election timeout 内没收到 Leader 心跳 → "假设 Leader 挂了"
2. 自荐 Candidate,发 RequestVote
3. 收到多数派票 → 成为 Leader
4. 如果没收到多数派(同时有多个 Candidate)→ 等下一轮随机 timeout 重试

关键玩法:随机化 timeout——让两个 Candidate "撞车"的概率小,加上一旦撞车下一轮 timeout 不同,最终一定有人单独 timeout 先。

这能绕过 FLP 吗?

FLP 说:对手能构造一条延迟序列让算法永不 terminate
Raft 说:对手可以 — 但只要在某个 GST 之后网络稳定 Δ ms,
        某个节点的 election timeout 一定会触发,
        而且其他节点的票一定能在 Δ 内到达,
        所以选主一定能完成。

Raft 偷换的就是"异步"假设——它假设"虽然不知道 Δ 多大,但 Δ 总是会有"。这个假设在真实数据中心 99.99% 时间成立。

4.3 不成立时会发生什么? ​

极端 case:某个数据中心持续 30 分钟全程网络抖动,每 100ms 抖一下
→ Raft 选主反复失败(每个 Candidate 都在 timeout 前被新心跳打断)
→ Liveness 挂了,集群无 Leader,所有写阻塞
→ 但 Safety 保留(没有任何写被错误 commit)

真实事故:etcd 在某些云上抖动严重的场景出现过持续几分钟无 Leader——这是 FLP 在工程里"露脸"的时刻。

五、出路二:随机化(Ben-Or 1983) ​

第二条路:放弃确定性。

5.1 Ben-Or 算法直觉 ​

每轮:
  1. 节点发自己当前值给所有人
  2. 收 N-f 个回复(f 是允许故障数)
  3. 如果多数派一致 → decide
  4. 否则 → 随机抛硬币选 0 或 1,作下一轮初值
  5. goto 1

为什么能绕过 FLP:

FLP 假设算法是确定性的 — 对手知道算法,能精确构造延迟攻击
Ben-Or 引入随机 — 对手不知道下一轮硬币结果
        → 不能精准让系统永远 bivalent
        → 系统以"接近 1 的概率"在有限轮收敛(期望终止 O(2^N) 轮)

注意:Ben-Or 只保证 probabilistic termination(概率上终止)——可能永远不终止但概率为 0。

5.2 工程影响 ​

Ben-Or 在拜占庭共识(15 篇)里有变种:HoneyBadgerBFT、Algorand 都用了"密码学随机数 + Ben-Or 思路"实现拜占庭+异步共识——这是 FLP 在拜占庭场景的标准绕过法。

Paxos / Raft 不用随机化绕 FLP(它们走部分同步路线),但 Raft 的随机 election timeout 是借鉴随机化思想缩短选主延迟——本质不是绕过 FLP,只是缓解 split vote。

六、出路三:失败检测器(Chandra-Toueg 1996) ​

第三条路:不强求异步,引入一个"oracle"告诉你谁挂了。

6.1 失败检测器是什么 ​

抽象一个外部模块 FD,负责回答"节点 X 是否故障?"——有不同强度:

级别完整性准确性实现方式
P (Perfect)所有故障都被检测不误判活节点需要同步网络,工程难做
S (Strong)所有故障都被最终检测最终不误判部分同步可实现
◇S (Eventually Strong)最终所有故障被检测最终不误判工程的标配

6.2 Chandra-Toueg 的关键结论 ​

有 ◇S 失败检测器 + 多数派正常 ⇒ 可以做共识(异步网络)

这等价于偷换了"异步"——你说网络异步,我说我有外部 oracle 能"告诉我谁挂了",于是 FLP 不再适用。

工程上 ◇S 的实现就是"心跳 + timeout"——和 Raft 的 leader heartbeat 几乎一样。所以 Chandra-Toueg 和 DLS 1988 在工程上是同一条路的两种说法。

6.3 φ-accrual 失败检测器(Cassandra / Akka) ​

真实工程里"心跳 + 阈值 timeout" 太硬,Cassandra/Akka 用 φ-accrual:

  • 不给"挂没挂"的二值答案,给一个数 φ
  • φ 越大,"挂了"的可能性越高
  • 阈值由应用层选(φ > 8 ≈ 1/10000 概率误判)

这是失败检测器的工程精细化,详见 30 篇故障检测。

七、Paxos / Raft 是怎么"踩着 FLP" 工作的 ​

回到工程,Paxos 和 Raft 的设计可以这么理解:

Paxos / Raft 的 Safety:任何情况下都不会出现"两个不同值都被 commit"
Paxos / Raft 的 Liveness:在"网络最终稳定 + 多数派活着" 的条件下,共识在有限时间内达成

——这就是 FLP 三选三的妥协:
  ✓ Agreement
  ✓ Validity
  ◇ Termination(只在部分同步时刻,不是无条件)

7.1 Raft 几条"FLP 反例" ​

反例 1:无限选主循环
  五个节点同时 election timeout → 互相投自己 → 没人拿多数票 → 下一轮
  理论上 FLP 允许这个循环永远不停
  Raft 的对策:随机 timeout → "撞车" 概率小,期望几轮就有一个先出来

反例 2:网络持续抖动
  Leader 心跳总是 timeout 边缘到达 → Follower 反复重选
  Raft 不能完全解决 → 这是"部分同步" 假设不成立的代价
  工程对策:调大 election timeout(从 150ms 调到 1s),牺牲故障切换速度换稳定

反例 3:对称故障
  五节点,网络分成 2+2+1 三组,任何一组都没多数 → 任何一边都选不出 Leader
  Raft 直接 Liveness 挂 → 等网络合并
  Safety 保留 → 不会写出冲突数据

7.2 Multi-Paxos 的类似妥协 ​

Multi-Paxos 用"distinguished proposer"(就是 Leader)避免反复打架,但选 Leader 的过程本身仍可能在异步抖动下卡死——和 Raft 一回事。

结论:Paxos / Raft 都"基本接受了 FLP 的存在,但用工程化的部分同步把它压到 99.99% 不发生"——这就是工程的胜利。

八、FLP vs CAP:同一种"不可能"的两种表达 ​

FLP:异步 + 1 crash + 确定性 ⇒ 不存在总能 terminate 的共识
CAP:网络分区时,C 和 A 二选一

关系:CAP 是 FLP 在"分区故障"场景下的特例
  分区 = 一组节点间消息延迟变无穷大 = 异步极端化
  在分区下要做 C,等价于要做共识 → FLP 适用 → 不可能同时保 A(Liveness)

更精细的对应:

FLPCAP
异步(消息延迟无上界)分区(子集间消息丢失)
TerminationAvailability
AgreementConsistency
部分同步出路分区恢复后重新可用

所以 CAP 不是独立的定理,是 FLP 的工程化叙事——Brewer 2000 提的时候没引用 FLP,但本质上同一个东西。

8.1 FLP 比 CAP "贵" ​

FLP 的可怕之处:不需要分区就生效——只要 1 个节点 crash + 异步网络,共识就可能卡死。CAP 至少要等到分区发生才"二选一",FLP 在网络稍微一抖就开始要命。

九、FLP 的工程实战意义 ​

9.1 设计共识协议时 ​

1. 先证明 Safety(invariant 永远成立)
2. 再说明 Liveness 在哪些假设下成立
3. 明确写出"Liveness 不保证"的边界条件(网络抖动模式、故障数)

Raft 论文 §5.4 专门论证 Safety——这是 FLP 教给我们的纪律。

9.2 配置共识系统时 ​

election_timeout:太小 → 误判 Leader 死(违反"部分同步"假设)
                  太大 → 真挂了切主慢,影响 Liveness
                  推荐 ≥ 10 × 网络平均 RTT,留足抖动空间

heartbeat_interval:< election_timeout / 3
                   保证心跳 miss 一两个不会误触发选主

这些参数的本质是在调"部分同步"假设的强度——你越保守,Safety 越稳但故障切换越慢。

9.3 排查"Liveness 挂了" ​

生产事故的诊断顺序:

1. 是不是网络持续抖动?(部分同步假设不成立)
2. 是不是少数派活着?(多数派全挂,共识算法不可能继续)
3. 是不是时钟偏差超 max_offset?(HLC/Raft 的隐式假设)
4. 是不是 GC 暂停 > election_timeout?(误判节点挂)

95% 的 Raft 集群"卡死"事故都能归到这四类——本质都是工程假设不成立。

十、踩坑提醒 ​

  1. 以为"FLP 说共识不可能"——它说的是"在异步 + 1 crash + 确定性下不可能",改任何一个前提就有出路。别用这个唬人,你天天用的 etcd 就在做共识。
  2. 以为"加超时就能 100% 解决 FLP"——超时只是"部分同步"的工程实现,网络持续抖动时 Raft 也会 Liveness 挂。FLP 在某些极端工况下仍会露脸。
  3. 追求"Liveness 永远不挂"——办不到。接受"宁可暂时不可用,绝不破坏一致性"——这是 Raft 论文的工程哲学。
  4. election_timeout 配太小——150ms 在跨机房抖动场景下会反复选主,Liveness 严重退化。跨机房至少 1s。
  5. 以为随机化能"绕过"FLP——Ben-Or 只是 probabilistic termination,期望终止时间 O(2^N) 节点多了根本不能用。生产共识不靠纯随机化。
  6. 拿 FLP 否定 Paxos / Raft——FLP 的"不可能"和工程的"几乎总能" 不矛盾。FLP 是理论 lower bound,Paxos / Raft 是在 99.99% 工况下的工程 upper bound。
  7. 混淆 Safety 和 Liveness 故障——Safety 故障(双 Leader / 双 commit)是数据损坏,必须修代码;Liveness 故障(无 Leader / 卡选主)是可用性问题,调参数 / 加节点。两者根本不是一回事。
  8. 以为"网络分区 = FLP 场景"——分区只是 FLP 异步的极端情况,FLP 在"网络正常但抖动"时就生效了——所以共识系统在小抖动下也会偶发 Liveness 失败。
  9. 拿 FLP 反对"全异步设计"——异步通信和异步系统模型不是一回事,RabbitMQ 异步消息不等于 FLP 的"异步网络";FLP 针对的是"延迟无上界",生产网络抖动有上界(虽然抖)。
  10. 不理解"FLP 不可能" ≠ "做不出来"——做出来的关键是加假设。Raft 加"部分同步";Spanner 加"原子钟"(等于加"时间近似同步");PBFT 加"3f+1 多数派"(等于强化故障假设)——每个真实系统都在偷换前提。学共识就是学"每个算法到底偷换了哪个 FLP 前提"。
