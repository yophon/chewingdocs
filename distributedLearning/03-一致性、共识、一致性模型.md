# 一致性、共识、一致性模型

英文里 **Consistency**、**Consensus**、**Consistency Model** 是三个完全不同的概念,中文翻译都带"一致"两个字,**于是从十多年前开始全行业混着用,新人入门时被术语硬生生绕晕**。"Kafka 是强一致"、"Raft 提供线性一致性"、"我们用 Paxos 实现一致性"——这些句子里"一致"指的是同一件事吗?**不是。混着用,选型和排错都会跑偏**。这一篇专门把三个词分清,**这一层的术语干净了,后续 16-19 篇一致性模型、09-15 篇共识算法,你才能直接读懂论文**。

> 一句话先记住:**Consensus(共识)是手段、Consistency(一致性)是结果、Consistency Model(一致性模型)是契约**。共识是"N 个节点对一个值达成一致"的算法(Paxos / Raft);一致性是"多副本数据状态相同程度"的工程描述(强 / 弱 / 最终);一致性模型是"程序员视角的内存可见性契约"(Linearizability / Sequential / Causal)。**"Kafka 是强一致"是错的——它的 ISR 同步复制只能给你 monotonic reads,达不到 Linearizability**——这个 confusion 后面会反复打你。

---

## 一、为什么必须把三个词分开

先用一个生活类比理解为什么这三个词必须分开:

```
共识 (Consensus) ── 像三个人投票决定"今晚吃什么"
                    ├─ 这是个"算法/过程"
                    └─ 解决"怎么从分歧到一致"

一致性 (Consistency) ── 像投票完成后"三个人都知道吃火锅"
                       ├─ 这是个"结果/状态"
                       └─ 描述"多副本数据相同程度"

一致性模型 (Consistency Model) ── 像规定"投票时必须实时通知所有人"
                                  ├─ 这是个"契约/规约"
                                  └─ 描述"程序员看到什么样的内存视图"
```

**三个词回答的是三个不同问题**:

| 问题 | 答案 | 落到工程 |
| --- | --- | --- |
| 节点之间怎么达成同一个决定? | Consensus(共识算法) | Paxos / Raft / ZAB / PBFT |
| 多个副本的数据有多接近? | Consistency(一致性程度) | 强一致 / 弱一致 / 最终一致 |
| 并发读写时程序员能看到什么? | Consistency Model(一致性模型) | Linearizability / Causal / Eventual |

这三个**不在同一层**——共识算法是"机制",它产出"一致性这个结果",而程序员对外承诺"一致性模型这个契约"。**混着用,就像把"投票"、"投票结果"、"民主制度"混着说一样**。

## 二、Consensus(共识):节点对一个值达成一致

**定义**:**N 个节点,通过一个算法,对单一个"值"达成一致**——这个"值"可以是一个数字、一条日志条目、一个状态变更。

### 2.1 共识问题的标准形式

```
N 个节点,每个节点可能 propose 一个值 v_i:
  N1 propose v_1
  N2 propose v_2
  N3 propose v_3

共识算法必须保证:
  1. Termination(终止性):所有非故障节点最终决定一个值
  2. Validity(有效性):决定的值必须是某个节点提议过的
  3. Agreement(一致性):所有非故障节点决定相同的值
```

**注意**:**"所有非故障节点决定相同的值"——这里的"一致"是 Agreement,不是"一致性"那个词**。Agreement 是共识算法的目标属性,**和 Consistency Model 里的"一致"是两层概念**。

### 2.2 共识的经典算法

| 算法 | 故障模型 | 节点数 | 典型实现 |
| --- | --- | --- | --- |
| Basic Paxos | Crash-recovery | 2f+1 | 几乎没有直接生产使用 |
| Multi-Paxos | Crash-recovery | 2f+1 | Google Chubby、Spanner |
| Raft | Crash-recovery | 2f+1 | etcd、Consul、TiKV、CockroachDB |
| ZAB | Crash-recovery | 2f+1 | ZooKeeper |
| EPaxos | Crash-recovery | 2f+1 | 实验性,无 Leader Paxos |
| PBFT | Byzantine | 3f+1 | Hyperledger Fabric、联盟链 |
| HotStuff | Byzantine | 3f+1 | Diem(原 Libra)、Aptos |
| PoW | Byzantine + 匿名身份 | 算力多数 | Bitcoin |

**共识算法做的事就一件**:**让 N 个节点在面对故障、网络分区、并发提议的情况下,对某个"值"达成一致**。**它不直接提供一致性模型**——一致性模型是建在共识之上的应用层契约。

### 2.3 共识 ≠ 复制

很多人把"复制"和"共识"画等号——**不对**。

```
复制 (Replication):一份数据在多台机器上有副本
                  └─ 复制可以异步、可以丢、可以延迟
                  └─ MySQL 主从就是复制,没有共识

共识 (Consensus):多个节点对"值"达成一致(强一致的决定)
                └─ 共识协议保证 Agreement,即所有正确节点决定同一个值
                └─ Raft / Paxos 是复制 + 共识
```

> **共识是"强复制"——它保证多副本看到的写顺序完全一致**。一般的异步复制可以"先写完主再传给从",从机滞后几秒;**共识协议要求"多数派同步确认才算写完"**,这是它的代价(延迟高),也是它的价值(强一致)。

### 2.4 共识协议解决的不变量

```
共识协议给你的不变量:

  1. 决定唯一:对任何提议序列,所有正确节点决定同样的值
  2. 决定不变:一旦决定就不会改(safety)
  3. 决定可达:多数派活着就一定能决定(liveness,在 partial sync 下)
```

**这三个不变量是建一切分布式系统的基石**——分布式锁(必须达成"谁持有锁"的共识)、选主(必须达成"谁是 Leader"的共识)、状态机复制(必须达成"日志顺序"的共识)。**所有 etcd / ZK / Consul / TiKV 提供的服务,底层都是共识协议的应用**。

## 三、Consistency(一致性):多副本的数据相同程度

**定义**:**多副本数据在某一时刻"有多接近"**——通俗讲就是"主写完了,从什么时候能看到"。

```
强一致 (Strong) ────────────────────────── 弱一致 (Weak)
   │                                          │
   主写完瞬间从就能看到                    主写完后过一会从才能看到
   读永远拿到最新值                         读可能拿到旧值
   实现成本高                              实现成本低
```

### 3.1 一致性的程度

| 级别 | 描述 | 典型实现 |
| --- | --- | --- |
| Strong | 写完瞬间所有副本可见 | 同步复制 + 共识协议 |
| Eventual | 给足够时间,最终所有副本一致 | 异步复制(MySQL 异步、Cassandra) |
| Weak | 没有强保证,中间状态可见 | 缓存层、CRDT 的中间态 |

**注意**:**这里说的"一致性"是工程描述,不是严格的形式化定义**——后面讲的"一致性模型"才是形式化的契约。**这是行业常用语,但理论上 Consistency 的含义只有 ACID 里的 C(数据库约束完整性)和 Consistency Model 两种严格定义**——下面用"一致性程度"避免混淆。

### 3.2 一致性程度 vs 共识

**一致性程度是结果,共识是手段**——但不是所有强一致都靠共识。

```
强一致达成的几种方式:

  方式 1:单机(没有多副本,自然强一致)
  方式 2:同步复制(主写完后等所有从确认再返回)
  方式 3:共识协议(Paxos / Raft,多数派同意)
  方式 4:分布式事务(2PC,所有参与者都提交)
```

**MySQL 半同步复制就是"同步复制"达成的强一致**——它没用共识协议。**Raft 是"共识协议"达成的强一致**——它用了 majority quorum。**两者的故障表现不同**:

- MySQL 半同步:主挂了 → 必须人为切换 → 期间不可用
- Raft:Leader 挂了 → 集群自动选举新 Leader → 几秒钟恢复

> **共识协议 = 同步复制 + 自动选主 + 故障容错**——这是它比"纯同步复制"贵但更工程化的原因。

### 3.3 ACID 里的 C 是另一回事

**数据库 ACID 里的 C(Consistency)指的是"事务前后,数据满足应用层约束"**——比如"账户余额不能负"、"外键约束"、"唯一索引"。**这和分布式的"多副本一致性"是完全不同的概念**,只是英文都叫 Consistency。

```
ACID 的 C:事务执行后,数据库依然满足业务约束
              └─ 应用层定义的不变量
              └─ 比如转账后 A.balance + B.balance 不变

分布式的 C:多副本数据相同程度
              └─ 副本之间的对齐程度
              └─ 比如主写了 x=1,从能不能立刻看到 x=1
```

**两者的混淆**:很多教科书把 CAP 的 C 解释成 ACID 的 C——**错**。**CAP 的 C 是 Linearizability**(详见下一节),是分布式一致性模型的最强档,**完全不是 ACID 的 C**。

## 四、Consistency Model(一致性模型):程序员看到的契约

**这是最严格、最理论的概念**——**形式化定义"并发读写下,程序员能看到什么样的内存视图"**。

```
一致性模型 (Consistency Model)
   │
   ├─ Linearizability   (线性一致性,最强)
   ├─ Sequential        (顺序一致性)
   ├─ Causal           (因果一致性)
   ├─ Read-Your-Writes (读自己写)
   ├─ Monotonic Reads  (单调读)
   ├─ Monotonic Writes (单调写)
   └─ Eventual         (最终一致,最弱)
```

**一致性模型回答的问题是**:**"程序员在多线程 / 多客户端并发访问时,会看到什么样的执行历史?"**

### 4.1 Linearizability(线性一致性):看起来像单机

最强的一致性模型——**所有操作看起来像在单机上原子地、按某个全局顺序执行**。

```
程序员视角:
  3 个客户端并发操作 → 看起来像
  ↓
  把所有操作排成一个全局顺序,然后单机原子执行
```

**关键性质**:

- **每个操作有一个"线性化点"**——在请求开始和返回之间
- **如果操作 A 在操作 B 开始前完成,那么在全局顺序里 A 必须排在 B 前**(real-time order)
- **每个读必须返回最近一次写的值**(在全局顺序里最近的)

**典型实现**:Raft / Paxos / 2PC——**所有共识协议都给 Linearizability**(配合正确的客户端逻辑)。

**代价**:**延迟高**——每次写都要走多数派,跨地域延迟成百毫秒。

### 4.2 Sequential Consistency(顺序一致性):像单机但不要求实时

**比 Linearizability 弱一档**——同样要求全局顺序,**但不要求和真实时间一致**。

```
Linearizability:
  - 全局顺序 ✓
  - 必须符合"操作完成时间"的偏序(real-time)

Sequential Consistency:
  - 全局顺序 ✓
  - 不要求和真实时间一致

允许场景:操作 A(10:00 完成) → 操作 B(10:01 完成)
        但全局顺序里可以是 B → A,只要每个客户端"内部顺序"对就行
```

**关键性质**:**每个客户端看到的操作序列必须和它发出的顺序一致(program order),但客户端之间的相对顺序可以任意**。

**典型实现**:**ZooKeeper 的写**(ZAB 协议给 Sequential 但不是 Linearizability,因为它的快速读不走 Leader,可能读到旧 Leader 的数据)。

**重要细节**:**单 CPU 内核的指令执行就是 Sequential Consistency**——这是计算机体系结构里的原始概念,**1979 年 Lamport 提出**,后来才被分布式系统借用。

### 4.3 Linearizability vs Sequential 的关键区别

**这两个模型最容易被混淆**。看这个例子:

```
真实时间轴 (wall clock):

  Time 1:  Client A: write(x, 1) [完成]
  Time 2:  Client B: read(x) → ???
  Time 3:  Client C: write(x, 2) [完成]
  Time 4:  Client D: read(x) → ???

Linearizability:
  Time 2 的读必须返回 1 或 2(不能是 0)
  因为 A 的写在 Time 2 之前已完成

Sequential Consistency:
  Time 2 的读可以返回 0 / 1 / 2
  因为不强制 real-time order

       Linearizability ⊊ Sequential ⊊ Causal ⊊ Eventual
       (前者比后者更强)
```

> **Linearizability 给你"分布式系统看起来像一台单机"的幻觉**——这是它最强的卖点,也是 etcd / ZK / TiKV 选它的原因(应用程序员可以"忘记"分布式)。**代价是每次写都要 quorum + 同步**——延迟成本不可忽视。

### 4.4 Causal Consistency(因果一致性)

**因果有关的操作必须按因果顺序看到**,**无关的可以乱序**。

```
"因果关系"的定义(Lamport happens-before):
  1. 同一客户端的两个操作:先发的有因果于后发的
  2. 一个客户端的写,另一个客户端读到了:写有因果于读
  3. 因果是传递的:A → B 且 B → C,则 A → C

因果一致性保证:
  如果操作 X 因果先于操作 Y,那么所有节点必须先看到 X 再看到 Y
  没有因果关系的操作,顺序无所谓
```

**典型用例**:**评论系统**——评论必须在发帖之后被看到(因果),**但不同帖子的评论顺序可以乱序**(无关)。

**实现**:**向量时钟 / 版本向量**(详见 07 篇)——Dynamo、Riak、COPS 用这个模型。

**为什么重要**:**因果一致性是 CAP 之外的甜点**——可以同时满足 Available + Causal,**不需要付出 Linearizability 的延迟代价**(详见 18 篇)。

### 4.5 RYW / Monotonic Reads / Eventual(弱档三件套)

| 模型 | 保证 | 典型场景 | 实现 |
| --- | --- | --- | --- |
| Read-Your-Writes | 你自己的写,自己立刻能读到 | 改头像后立刻刷新看到 | Session 粘滞 / 写后读主 |
| Monotonic Reads | 看到某版本后,不会读到更旧的 | 刷朋友圈不能"刷新少几条" | 客户端带最高版本号 |
| Eventual | 足够时间后所有副本一致 | 点赞数 / 关注数 | 异步复制 / Gossip / CRDT |

```
Alice: write(x, 1) → read(x) → 必须返回 1 (RYW)
Bob:                  read(x) → 可能返回 0 (RYW 不要求)
```

最终一致是最弱的——**中间状态可能任意旧、任意乱**,只承诺"停止写入后,所有副本最终收敛"。**典型实现**:异步复制(Dynamo / Cassandra)、Gossip、CRDT。

### 4.6 一致性模型层级图

```
                Linearizability
                       │
                       ▼
                Sequential Consistency
                       │
                       ▼
                Causal Consistency
                       │
                  ┌────┼────┐
                  ▼    ▼    ▼
              RYW   MR   MW    (Read-Your-Writes / Monotonic Reads / Monotonic Writes)
                       │
                       ▼
                Eventual Consistency
```

**向上越强、越贵;向下越弱、越快**。**真实工程是给每类操作选不同档**——余额选 Linearizability,头像选 RYW,点赞数选 Eventual。

## 六、工程混淆案例:为什么"Kafka 是强一致"是错的

这是面试和文档里最常见的错误说法。**Kafka 提供"分区内顺序 + ISR 同步复制"——但这不等于 Linearizability**。

### 6.1 Kafka 的承诺

```
acks=all + min.insync.replicas=2 + replication.factor=3:

  生产者发送消息 → Leader 收到 → 复制到所有 ISR 副本 → 全部确认 → 返回 ack

  保证 1: 同一分区内消息顺序一致(分区内顺序)
  保证 2: ack 返回后,消息至少在 ISR 中 2 个副本上
  保证 3: 主挂了,从 ISR 中选举新主,数据不丢
```

**Kafka 的这套机制可以叫"持久化保证"或"durability + ordering",但不是 Linearizability**。

### 6.2 为什么不是 Linearizability?

**关键反例**:**消费者看到的顺序可能和生产者真实写入的实时顺序不一致**。

```
Producer A → Partition 0: msg1
Producer B → Partition 1: msg2 (真实时间晚于 msg1)

Consumer 同时订阅两个分区:可能先消费到 msg2 再消费到 msg1
→ 违反 Linearizability 的 real-time order
→ Kafka 仅保证"分区内顺序"——比 Sequential 还弱
```

### 6.3 其他常见混淆

| 系统 | 错误说法 | 正确说法 |
| --- | --- | --- |
| Kafka | 强一致 | 分区内有序 + ISR 持久化(非 Linearizability) |
| Redis 主从 | 强一致 | 异步复制,Eventual |
| MySQL 半同步 | Linearizability | 同步复制达到 RYW + Monotonic |
| ZooKeeper | Linearizability | **写 Linearizability,默认读 Sequential**(读不走 Leader) |
| etcd | 强一致 | Linearizability(读默认走 Leader 或 ReadIndex) |
| Cassandra | 最终一致 | **可调,默认 Eventual,Quorum 可达 RYW** |

> **看一个系统的文档,要看它对每类操作分别承诺什么模型**——一刀切地说"强一致"或"最终一致",**99% 的概率是说错了**。

## 七、单机内存模型:一致性模型的"原型"

很多人不知道的事:**Linearizability、Sequential Consistency 这些词最早是单机多核 CPU 的内存模型**——**1979 年 Lamport 在 *How to Make a Multiprocessor Computer...* 提出 Sequential Consistency,后来才被分布式系统借用**。

### 7.1 多核 CPU = 微缩分布式系统

```
                    内存 (x=0)
                   /         \
              CPU 1 缓存    CPU 2 缓存
              (x=0)         (x=0)

CPU 1: x = 1 (写自己缓存) → CPU 2 还看到 x = 0
→ 多核 CPU 也有"分布式数据一致性"问题
→ 解决:MESI 协议 + Memory Barrier
```

多核 CPU 的"多副本(L1/L2/L3 缓存)、消息传递(总线协议)、并发访问"——**就是一个微缩的分布式系统**。所有分布式一致性模型的概念,在单机内存层都已经被反复推过一遍。

### 7.2 x86 TSO vs ARM Relaxed

**x86 TSO(Total Store Order)**——比 Sequential 弱一点,**允许 Store→Load 重排**(Store Buffer 吸收 Store,Load 直接读 Cache):

```
程序顺序:store A=1; load B
可能执行:load B; store A=1   ← x86 允许
→ Peterson 算法在 x86 上需要 mfence 指令
```

**ARM Relaxed**——比 x86 更弱,**几乎所有顺序都可以重排**,除非显式 `dmb` / `ldar` / `stlr` 屏障。**这就是用 C++ 写跨平台多线程要用 `std::atomic` + memory order 的原因**。

### 7.3 单机模型和分布式模型的对应

| 单机内存模型 | 分布式一致性模型 |
| --- | --- |
| Sequential Consistency | Sequential Consistency |
| x86 TSO | 接近 Linearizability(差 store-load reorder) |
| ARM Relaxed | Causal 甚至更弱 |
| Memory Barrier | Synchronous Replication / Read Barrier |
| MESI 缓存协议 | 主从复制 / Quorum 复制 |
| CPU Cache / Cache Line | 副本 / 数据条目 |

> **你写 Java `volatile`、C++ `std::atomic` 的内存序参数,本质就是在选择"一致性模型"**——只是范围在单机。**理解了单机内存模型,分布式一致性模型就只是"把单机概念放大到集群"**。Herlihy & Wing 1990 的 Linearizability 论文里**完全不区分"单机多核"和"分布式"**——同一套理论。

---

## 八、为什么这套术语混乱这么久

1. **中文翻译都叫"一致"**——Consistency / Consensus / Consistency Model 都翻"一致性"或"共识/一致性",**中文读者入门就分不清**
2. **早期论文不严格**——1970-1980 年代概念在演化,**很多文章用语不严谨**
3. **工程文档为了简单**——"我们提供强一致"听起来比"我们提供 Linearizability 写 + Sequential 读"清楚,**但牺牲了精确性**
4. **CAP 定理的传播**——Brewer 的 CAP 里 C 是 Linearizability,**但后来被讲成 ACID 的 C,再被讲成"多副本一致"**,**层层走样**
5. **面试题文化**——简化成"三选二"、"BASE",**牺牲精确换记忆性**

**这个系列后续 30 篇,术语会严格区分**:**Consensus 永远指算法,Consistency Model 永远指契约**,**避免笼统说"一致性"**。

---

## 九、读完这一篇后该有的认知

1. **共识、一致性、一致性模型是三个完全不同的概念**——前者是算法,中者是工程结果,后者是程序员契约
2. **共识算法(Paxos / Raft)产出一致性结果,对外承诺一致性模型**——这条因果链是分布式系统的骨架
3. **"强一致"是不严谨的工程口语,真正要看的是 Linearizability、Sequential、Causal、Eventual 这些精确模型**
4. **Linearizability ≠ Sequential ≠ Causal**——三者之间差着具体的、可形式化的不变量,**选错档位系统就崩**
5. **Kafka / ZK / etcd 提供的一致性模型不同**——别凭印象选,看官方文档对每个操作的精确承诺
6. **单机内存模型和分布式一致性模型是同一套理论**——`std::atomic`、`volatile` 就是单机的"一致性模型选择"

---

## 十、踩坑提醒

1. **混着用三个术语**——文档 / 简历 / 面试**严格区分**,"我们用 Paxos 实现共识,提供 Linearizability 模型"才是精确表达
2. **以为 Kafka 是强一致**——它只是分区内有序 + 持久化,**不是 Linearizability**,**跨分区无序**
3. **以为 ZooKeeper 全 Linearizability**——**写是 Linearizability,默认读是 Sequential**(不走 Leader,可能读旧值),要 Linearizability 读必须 sync() 后再读
4. **以为 Redis 主从是强一致**——异步复制,**主挂数据可能丢**,Eventual 都谈不上
5. **以为最终一致 = 不一致**——最终一致有收敛窗口,**通常 100 ms - 几秒,99% 用户感知不到**
6. **以为 Sequential 比 Linearizability 慢"一点"**——不,**两者实现差异巨大**:Linearizability 需要严格全局顺序,Sequential 允许 reorder
7. **把 ACID 的 C 等同于 CAP 的 C**——**完全不同**!ACID 的 C 是业务约束,CAP 的 C 是 Linearizability
8. **写业务代码不思考一致性模型**——所有"分布式 bug"几乎都源自"程序员假设的模型 > 系统实际提供的模型"
9. **不区分共识和复制**——MySQL 主从是复制不是共识,**主挂了不能自动选主,数据可能丢**
10. **用"我们用 Raft 所以一定强一致"作为终点**——Raft 给写 Linearizability,**默认读可能拿旧值**,要严格 Linearizability 读必须 ReadIndex / Lease Read
