# 向量时钟与版本向量

上一篇 Lamport 逻辑时钟解决了「**给所有事件打全序**」,但它**只给了一半**——你能根据时间戳判断「A 一定在 B 之前」,**却没法反过来说"A 和 B 是并发的"**。在 Dynamo / Riak / Git / 协同编辑这些场景里,**识别"并发"比排序更重要**——只有先知道"两个写是并发的",你才知道"该合并还是该让用户挑"。这一篇讲清楚向量时钟(Vector Clock)和版本向量(Version Vector)——**因果追踪的标准答案**,以及它在 Dynamo 系一票工程系统里付出的代价。

> 一句话先记住:**Lamport 时钟单向——A→B 推出 LC(A) < LC(B),反过来不成立**;**向量时钟双向——能完整识别"因果先后"和"并发"**。**代价是每个节点都要带一个长度为 N 的向量**,N 一大就吃不消。**Dynamo / Riak 在生产证明了它能用,但 sibling 暴增能让你 OOM**——CRDT(19 篇)就是为了治这个病而生。

---

## 一、Lamport 时钟的死结:不识别并发

回顾上一篇:Lamport 逻辑时钟给每个事件一个标量 LC,规则是「本地事件 +1、收消息取 max+1」,**性质**:

```
A happens-before B  ⇒  LC(A) < LC(B)   ✓ (单向)
LC(A) < LC(B)       ⇒  A happens-before B   ✗ (反过来不成立)
```

举个反例:

```
节点 1: ●─────●        (e1 LC=1)  (e3 LC=2)
                 ╲
                  ╲
节点 2: ●─────●    ╲   (e2 LC=1)
              ╲    ╲
               ╲    ╲
节点 3:         ●────●  (e4 LC=2, 收 e3)

LC(e3) = 2, LC(e2) = 1
LC(e2) < LC(e3),但 e2 和 e3 之间没有任何消息往来 —— 它们是**并发的**
```

**Lamport 时钟看 LC(e2)=1 < LC(e3)=2,会错以为 e2 在 e3 之前**——实际上它们并发。

### 1.1 为什么这个死结要命

在 Dynamo / Riak 这种**多主无主复制**的系统里,**并发写是常态**:

- 用户 U 在客户端 A 写 `cart = [book]`
- 同时(几乎)在客户端 B 写 `cart = [milk]`
- 两个写都成功,系统里出现两个版本

**这时系统必须知道**:这两个写**是并发的**(不是 A 写完 B 才写),从而**不能直接覆盖**——要么合并(`cart = [book, milk]`),要么把两个版本都返回给应用层让它解决。

**用 Lamport 时钟做不到**——它只能给出一个数,**无法回答"这两个写有没有因果关系"**。

### 1.2 我们到底想要什么

我们想要一个时间戳函数 `VC(e)`,使得:

```
A happens-before B   ⇔   VC(A) < VC(B)   (严格小于,双向)
A 和 B 并发         ⇔   VC(A) 与 VC(B) 既不 ≤ 也不 ≥
```

**双向**——这是关键。向量时钟就是为此而生。

## 二、向量时钟的定义(Mattern 1989 / Fidge 1988)

### 2.1 数据结构

**每个节点 i 维护一个长度为 N 的向量** `VC_i = [c_1, c_2, ..., c_N]`,其中 `c_j` 表示「节点 i 所知的节点 j 已经发生过多少事件」。

```
N = 3 个节点的系统:
    节点 1: [0, 0, 0]
    节点 2: [0, 0, 0]
    节点 3: [0, 0, 0]
```

### 2.2 三条规则

**规则 1(本地事件)**:节点 i 发生本地事件,自己那一位 +1

```
节点 1 写了一次 → VC_1 = [1, 0, 0]
```

**规则 2(发送消息)**:节点 i 发消息,先自己 +1,然后把向量随消息发出去

```
节点 1: 本地事件 [1,0,0] → 发消息时再 +1 → [2,0,0] → 消息携带 [2,0,0]
```

**规则 3(接收消息)**:节点 j 收到消息向量 `m`,**逐分量取 max**,然后自己那一位 +1

```
节点 2 当前 VC_2 = [0, 1, 0]
收到消息向量 m = [2, 0, 0]
逐分量 max: [max(0,2), max(1,0), max(0,0)] = [2, 1, 0]
节点 2 自己那一位 +1: [2, 2, 0]
```

### 2.3 一张图看懂

```
节点 1   [1,0,0]●─────────────●[2,0,0]
                 ╲                ╲
                  ╲                ╲ 发消息 m=[2,0,0]
                   ╲                ╲
节点 2         [0,1,0]●─────────●─────────●[2,3,0]
                       ╲              ╲收到 m, max+1 → [2,2,0]
                        ╲ 本地事件      └─ 本地 → [2,3,0]
                         ╲
节点 3                    ●[0,0,1]──────●[0,0,2]
                          本地           本地
```

**每一根向量都是一个"我所看到的世界状态"**——节点 2 看到 `[2,3,0]` 意味着:"我知道节点 1 发生过 2 个事件,我自己发生过 3 个,节点 3 我还没听说"。

## 三、关系判定:三条比较规则

给定两个向量时钟 `VC(A) = [a_1,...,a_n]` 和 `VC(B) = [b_1,...,b_n]`,有三种关系:

```
1. VC(A) < VC(B)
   ⇔  对所有 i,a_i ≤ b_i,且至少一个 j 有 a_j < b_j
   ⇔  A happens-before B(因果先于)

2. VC(A) > VC(B)
   ⇔  对所有 i,a_i ≥ b_i,且至少一个 j 有 a_j > b_j
   ⇔  B happens-before A

3. 既不 ≤ 也不 ≥
   ⇔  存在 i 使 a_i > b_i,且存在 j 使 a_j < b_j
   ⇔  A 和 B 并发(concurrent)
```

### 3.1 例子

```
VC(A) = [2, 1, 0]
VC(B) = [3, 2, 0]
A 每一位 ≤ B,且第 1、2 位严格小于 → A → B (A 先于 B)

VC(A) = [2, 1, 0]
VC(B) = [1, 3, 0]
A 第 1 位 > B,A 第 2 位 < B → A 和 B 并发(系统检测到冲突)

VC(A) = [2, 1, 0]
VC(B) = [2, 1, 0]
完全相等 → 同一个事件
```

> **这一刀是工程的关键**——只有「严格不可比」才叫并发,**Dynamo / Riak 一看见并发就触发 sibling(冲突版本并存)**。

## 四、向量时钟 vs 版本向量:工程上的"亲兄弟"

学术上两个概念稍有区别,**工程上经常混用**:

| 维度 | 向量时钟(Vector Clock) | 版本向量(Version Vector) |
| --- | --- | --- |
| 计什么 | **所有事件**(本地、收发消息都 +1) | 只计**写操作**(读不 +1) |
| 用途 | 因果追踪(任何事件之间) | 副本数据的版本追踪 |
| 谁提出 | Mattern 1989 / Fidge 1988 | Parker et al. 1983(LOCUS) |
| Dynamo 用的是 | 名字叫 vector clock,**行为更像 version vector**(只在 put 时 +1) | — |

**结论**:**Dynamo 论文里写"vector clock"但严格说是 version vector**。**工程文档中你看到这两个词,基本可以当同一个东西**——都是「逐分量计数 + 逐分量取 max」的因果追踪机制。

## 五、工程映射 1:Dynamo / Riak / Voldemort 的冲突检测

### 5.1 Dynamo 的写流程

```
1. 客户端 PUT(key, value, context)
   - context 里带上一次读到的 vector clock(版本号)
2. 协调者节点(coordinator)处理写:
   - 取自己 node_id 上的计数 +1
   - 把新的 vector clock 一起写入,复制到 N 个副本
3. 客户端 GET(key)
   - 返回 (value, vector_clock) 或多个 sibling 版本
```

**例**:三个节点 A、B、C,key=`cart`

```
T0: 初始 cart="[]", VC = [A:0, B:0, C:0]
T1: 客户端 1 在 A 节点 PUT cart="[book]"
    新 VC = [A:1, B:0, C:0]
T2: 网络分区 A | B
    客户端 2 在 B 节点 PUT cart="[milk]"(基于 T0 的版本读到的)
    新 VC = [A:0, B:1, C:0]
T3: 分区恢复,A 收到 B 的版本
    比较 [A:1,B:0,C:0] vs [A:0,B:1,C:0]
    → A 第 1 位 > B,A 第 2 位 < B → 并发
    → 触发 sibling: 两个值都保留
T4: 客户端 GET cart
    系统返回:[("[book]", VC1), ("[milk]", VC2)]
    应用层负责合并(通常合并为 "[book, milk]")
```

### 5.2 Riak 的 sibling 资源管理

Riak 把 Dynamo 思路落到生产,**冲突版本叫 sibling**,关键参数:

- `allow_mult = true`:允许保留并发版本(Riak 2.x 默认开)
- `last_write_wins = true`:遇并发用墙钟时间决胜(简单但不安全,可能丢数据)

**应用层负责合并逻辑**——比如购物车合并(集合并集)、计数器(求和,但要小心)、地址簿(用户挑)。

### 5.3 Voldemort(LinkedIn)

跟 Dynamo 几乎一模一样的玩法,LinkedIn 内部用,**主要存社交关系数据**——合并语义天然友好(关系列表合并)。

## 六、工程映射 2:Git 的 DAG 是隐式向量时钟

Git 没有显式的"向量时钟",但 **commit DAG 本质上就是一个因果图**:

```
            C1 ─── C2 ─── C3 (main)
              ╲
               ╲
                C4 ─── C5 (feature)

C3 和 C5 并发(没有任何祖先关系直接相连)→ 合并时可能冲突
```

**Git 用"父 commit hash"代替向量时钟**——每个 commit 都指向它的父 commit,**祖先关系**就是因果关系。**Merge 时 Git 找 LCA(最近公共祖先)**,这个过程**等价于「检测两个 commit 的向量时钟是否并发」**。

**Mercurial 用的更直接**——内部就叫 "changeset DAG",每个节点有 parent revisions 列表,**这就是向量时钟的退化版**。

## 七、工程映射 3:Yjs / Automerge / 协同编辑的因果基础

CRDT 协同编辑(详见 19 篇)对每个编辑操作打**因果时间戳**,确保:

- 删除一个字符 → 必须知道这个字符是哪个版本插入的
- 撤销一个操作 → 撤销的"对象"必须有清晰因果链

**Yjs / Automerge 内部都维护版本向量**:

```
Yjs 的 state vector:
    { client_1: 42, client_2: 17, client_3: 31 }

意思:我已经看到了 client_1 的第 42 个操作、client_2 的第 17 个、client_3 的第 31 个

同步时只传"我没看到的"——A 给 B 发 sync 时,B 只回 B 知道但 A 不知道的操作
```

**这本质上是向量时钟在 CRDT 上的工程化**——把版本向量当**协议层的同步原语**用。

```
A 的 SV: { c1:42, c2:17, c3:31 }
B 的 SV: { c1:45, c2:17, c3:31 }
B 看到 A 缺 c1 的 43-45 → 把这三个操作发给 A
```

**没有这一层,协同编辑无法"按因果顺序应用操作"**——最后文档会乱。

## 八、向量时钟的代价

### 8.1 向量长度 = 节点数 = O(N)

这是最大的痛——**100 个节点,每条数据带 100 个数字的元数据**。

```
key="cart"
value="[book, milk]"
vector_clock = {
  "node_1": 234,  "node_2": 567,  "node_3": 12, ...
  "node_100": 89  # 总共 100 个
}
```

**实际开销**:每个数字 8 字节,100 节点 → 800 字节 / 条数据。**比 value 本身还大**。Dynamo 论文也承认这是个问题。

### 8.2 客户端必须带版本号

GET 之后客户端拿到 `(value, vector_clock)`,**PUT 必须把 vector_clock 一起回传**——否则系统不知道你"基于哪个版本写的",会和"全新的写"混淆。

```python
# 错误写法(忽略 vector_clock)
value, _ = client.get("cart")
value.append("milk")
client.put("cart", value)   # ← 系统不知道这是基于哪个版本

# 正确写法(带回 context)
value, vc = client.get("cart")
value.append("milk")
client.put("cart", value, context=vc)  # ← Dynamo 才能正确判因果
```

**应用层很容易忘**——Riak / Dynamo SDK 都用 `context` 对象隐藏这一层,但概念上必须懂。

### 8.3 GC 问题:向量永远不会缩

节点 1 早就退役了,但所有数据上还带着 `node_1: xxx` 这一位——**向量永远不会变短**,只会越来越长。

**对策**:

- **Dotted Version Vectors**(Almeida et al. 2014):用 (node, counter) 对的集合,只保留必要的因果信息,可以缩
- **TTL 老数据**:周期性扫库,把陈旧的向量条目清理掉
- **Pruning 阈值**:超过 K 个节点的位就合并(代价:丢失部分并发检测精度)

### 8.4 sibling 爆炸(真实事故)

**Jepsen 测试 Riak 时发现的灾难场景**:

```
1. 应用层不合并 sibling,只是把它们当成"多版本读"
2. 每次写都基于"最老的版本",新写就和老版本并发
3. sibling 数量 N → N+1 → N+2 → ... 单 key 数百个 sibling
4. 单条记录元数据膨胀到 MB 级
5. 内存爆,Riak OOM
```

**Jepsen 报告**:Riak 在某些工作负载下**单 key 可以累积上万个 sibling**——这是 vector clock 工程的最暗黑面。

**对策**:**应用层必须实现 merge**——读到多个版本时立即合并,写回单一版本。

## 九、向量时钟 vs Lamport vs HLC 一张表

| 维度 | Lamport 标量时钟 | 向量时钟 | HLC(下篇) |
| --- | --- | --- | --- |
| 数据结构 | 单个数 | 长度 N 的向量 | (物理时间, 逻辑位) |
| 因果判定 | 单向(只能排除非因果) | **双向**(能识别并发) | 双向但近似 |
| 元数据开销 | O(1) | O(N) | O(1) |
| 用途 | 全序排序 | 并发检测 | 给事务打外部一致 timestamp |
| 典型系统 | 早期 Lamport 论文 | Dynamo / Riak / Git | CockroachDB / YugabyteDB |

**记忆口诀**:**Lamport 一根数,向量是数组,HLC 物理+逻辑两个数**——一根数省空间但漏并发,数组完整但贵,HLC 折中。

## 十、踩坑提醒

1. **以为 Lamport 时钟就够了**——它不识别并发,**Dynamo 系统必须用向量时钟**。把 Lamport 套到多主复制上,会把"并发写"误判成"顺序写",**直接丢数据**。
2. **客户端 PUT 不带 context**——Dynamo / Riak 必须回传上次读的 vector clock,**否则每次写都被当"全新写"**,sibling 满天飞。
3. **不实现 sibling merge**——Jepsen 验证过的 Riak OOM 路径,**业务必须有 merge 函数**(集合并集、计数求和、用户挑)。
4. **用 last-write-wins 决胜**——`last_write_wins = true` 看起来省心,但**墙钟不可信**(05 篇),会丢数据。**金融、订单这种宁可应用层合并**。
5. **节点 ID 用 IP 或主机名**——重启换 IP 就变新节点,向量永远膨胀。**用稳定 UUID + 持久化**。
6. **向量不做 GC**——退役节点的位永久占着位置,**几年后单条记录元数据几 KB**。要做 Pruning 或换 Dotted Version Vector。
7. **跨数据中心节点都进同一个向量**——N 跨到几千节点,向量就废了。**分层(per-DC vector + global merge)**才能撑住。
8. **以为 Git merge 是"按时间合并"**——Git 是按 DAG 合并,**冲突判定就是"向量时钟并发"**;搞错这点写出来的同步工具会丢提交。
9. **CRDT 不带向量时钟**——CRDT 的 join 满足交换/幂等,**但你怎么知道"对端已经看到哪些操作了"?**那就是版本向量的活,缺了它同步效率为 0。
10. **指望向量时钟解决"语义冲突"**——它只解决"是否并发"。**两个并发写到底该 union 还是 last-write-wins 还是让用户挑,业务自己定**。向量时钟只是给你「冲突检测」的工具,不是「冲突解决」的工具。
