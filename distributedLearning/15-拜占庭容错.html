拜占庭容错 ​

到这里为止,前 14 篇讲的所有共识协议——Paxos、Raft、ZAB、EPaxos——都假设节点是"诚实"的。节点要么正常工作,要么直接挂掉(crash-stop),它不会撒谎、不会伪造消息、不会被攻击者控制。这个假设在内网集群里基本成立,所以 Raft / Paxos 才能用 N ≥ 2f+1 的多数派搞定。但金融监管系统、联盟链、跨组织协作、公链——一旦节点不再都属于"自己",有节点可能被攻击、被收买、硬件出错返回脏数据,Raft 就完全失效——一个撒谎的 leader 能让整个集群相信"账户余额是 0"。这就是拜占庭故障(Byzantine fault),也是公链 / 联盟链 / 容灾级金融为什么要用一套完全不同的协议——PBFT / HotStuff / Tendermint。

一句话先记住:拜占庭故障 = 节点可任意行为(撒谎 / 伪造 / 攻击 / 硬件错),共识需要 N ≥ 3f+1(诚实节点超过 2/3)。PBFT(Castro-Liskov 1999)三阶段 Pre-prepare / Prepare / Commit,O(N²) 消息复杂度;HotStuff(Yin 2019)链式 BFT,O(N) 改进;Tendermint / Cosmos = PBFT + 轮换 leader + PoS,工业代表。公链 PoW/PoS 不是严格 BFT——概率最终性,性能远不如 PBFT。谁需要 BFT:联盟链 / 公链 / 极个别金融容灾;99% 的业务不需要——内网集群 Paxos/Raft 足够,BFT 是 3-5 倍开销的奢侈品。

一、拜占庭故障是什么 ​

1.1 故障模型谱 ​

第 2 篇讲过故障模型(回忆下):

故障模型从宽到严:
    Crash-stop      节点要么正常,要么挂掉,挂了不恢复
    Crash-recovery  挂了能恢复,日志还在(Raft / Paxos 假设)
    Omission        消息可能丢失(网络分区,前 14 篇都假设)
    Byzantine       节点可以任意行为(包括撒谎、伪造、攻击)

拜占庭是最严格的故障模型——它假设节点是敌对的,可以发任意消息给任意人。

1.2 名字的由来 ​

Lamport / Shostak / Pease 1982 论文 "The Byzantine Generals Problem":

故事:几个拜占庭将军围攻一座城,他们必须就"进攻 or 撤退"达成一致
      但他们只能通过信使通信
      信使可能叛变(消息篡改/伪造)
      将军本身也可能叛变(发不同消息给不同将军)
      问题:在 f 个叛变将军的情况下,忠诚将军如何达成一致?

证明:N ≥ 3f+1 是充要条件(其中 f 是叛变将军数)

Lamport 取拜占庭这个名字,因为它够"陌生"——他想避免任何政治敏感(Albanian Generals 之类),拜占庭帝国是中世纪东罗马,跟现实政治无瓜葛。

1.3 真实世界的拜占庭故障来源 ​

不只是恶意攻击,包括:

来源例子
硬件错误内存位翻转、磁盘静默损坏、CPU 计算错(intel Pentium FDIV bug)
网络中间人TLS 不强制 / 中间人篡改 / 路由器劫持
代码 bug不同版本节点行为不一致(升级期间)
配置错误节点配置不同,行为不可预测
恶意攻击节点被入侵,有意发错消息
多方共建不同组织部署的节点,互不信任

联盟链场景:几家银行共建一个清算网络,每家都可能"为了自己利益"作弊。这不是技术,是商业现实——没有任何一家银行愿意把账本完全托管给另一家。BFT 的本质是"在互不信任的多方间达成一致"。

二、N ≥ 3f+1 的证明直觉 ​

为什么是 3f+1,不是 2f+1?Lamport 论文给了严格证明,这里给个直观解释。

2.1 Crash 模型:为什么 2f+1 够 ​

N = 2f+1 个节点,最多 f 个 crash

任何决策需多数派(f+1 个) ack
即使 f 个挂了,还有 f+1 个能 ack
两个不同多数派必有交集,且交集节点诚实(它"不挂时"答案一致)
→ 一致性保证

关键:crash 节点要么正常工作,要么不工作——它不会"看起来工作但给错答案"。

2.2 Byzantine 模型:为什么需要 3f+1 ​

N = 2f+1 不够,因为 f 个拜占庭节点可以"假装"任意答案

例:3 节点,1 拜占庭
    诚实节点 A:看到 x = 5,告诉 B 和 C 
    诚实节点 C:看到 x = 5,告诉 A 和 B
    拜占庭 B:告诉 A "x = 5",告诉 C "x = 7"
    
    A 收到 (A: 5, B: 5, C: 5) → 一致,认为 x = 5
    C 收到 (A: 5, B: 7, C: 5) → 不一致,认为 x = 7?
    
    一个节点说不同的话,诚实节点看到的"多数派"不同

N ≥ 3f+1 的必要性:

N 节点,f 拜占庭
诚实节点 = N - f
任何决策需要从 N - f 个 ack 中确定(因为拜占庭可以不回复 / 回错)
两个不同决策的诚实 ack 集合必有交集(才能保证一致)
    → (N - f) + (N - f) > N - f
    → N > 3f - f = 2f...

但还要"交集中至少有一个诚实节点":
    (N - f) + (N - f) - (N - f) > f
    → N - f > f
    → N > 2f
    
再算上"诚实节点占主导":
    N - f > f + f
    → N > 3f
    → N ≥ 3f + 1

直觉:诚实节点必须占严格多数(> 2/3),才能"投出来的票"经得起拜占庭节点的搅局。

2.3 典型配置 ​

f(可容忍拜占庭)N(总节点)应用
f = 1N = 4最小 BFT(实际很少,容错太弱)
f = 2N = 7小型联盟链
f = 3N = 10中型联盟链
f = 10N = 31大型联盟链(Cosmos validator 数量级)
f = 100+N = 300+公链(以太坊 PoS、大型 Cosmos)

节点数越多,通信复杂度越高——PBFT 是 O(N²),N = 100 时一次共识要发 10000 条消息,带宽爆炸。这是为什么公链都在改进消息复杂度(HotStuff、Aleph、Narwhal-Bullshark 等)。

三、PBFT:1999 年的里程碑 ​

3.1 论文背景 ​

Practical Byzantine Fault Tolerance(Castro-Liskov, OSDI 1999)是 BFT 工程化的开端。1999 年之前的 BFT 协议都假设同步网络(消息延迟有上界),根本不实用。PBFT 的贡献:第一个在异步网络下实用的 BFT 协议。

当时性能:

  • 100ms 级延迟(局域网)
  • 千级 TPS(对 1999 年硬件来说很猛了)

3.2 PBFT 三阶段 ​

PBFT 类似 Raft,有一个 primary(leader),用三阶段完成一次共识:

              Pre-prepare      Prepare        Commit
Client          │                │               │
  │ Request     │                │               │
  ├────────────▶│                │               │
  │             │ Pre-prepare    │               │
  │             ├──────[broadcast]──────────────▶│         所有节点
  │             │                │               │
  │             │                │ Prepare       │
  │             │                ├─[broadcast]──▶│         所有节点
  │             │                │ 收到 2f+1 个   │
  │             │                │ Prepare       │
  │             │                │               │ Commit
  │             │                │               ├─[broadcast]─▶
  │             │                │               │ 收到 2f+1 个
  │             │                │               │ Commit
  │             │                │               │ → execute
  │◀────── Reply ──────────────────────────────────────────────

三阶段角色:

Pre-prepare(预准备) ​

  • Primary 收到 client request,分配序号 n,发 <PRE-PREPARE, view, n, request> 给所有副本
  • 副本收到 → 验证(签名、view、序号没冲突)→ 如 OK,进 Prepare

作用:由 primary 强加一个顺序——但 primary 可能撒谎(发不同消息给不同副本),所以需要下一阶段对齐。

Prepare(准备) ​

  • 每个副本收到 Pre-prepare 后 → 向所有其他副本广播 Prepare
  • 每个副本收集 ≥ 2f 个其他副本的 Prepare(共 2f+1 含自己) → 进 prepared 状态

作用:确保多数派(2f+1)都同意"primary 给的序号是这个 request 的"——如果 primary 撒谎,不同副本看到的 Prepare 集合不同,prepared 状态不一致。但只要 2f+1 个一致,就说明诚实节点已经达成共识。

Commit(提交) ​

  • 副本进入 prepared 后 → 广播 Commit
  • 每个副本收集 ≥ 2f+1 个 Commit → 进 committed 状态
  • 委托状态机执行 → 回 client

作用:确保即使 primary 故障切换,新 view 也能恢复 prepared 状态对应的决议——这是 Commit 阶段的核心,它保证了跨 view 的一致性。

3.3 为什么是三阶段不是两阶段 ​

直觉:

  • Pre-prepare:primary 提议顺序(可能撒谎)
  • Prepare:副本互相对齐"我们都看到 primary 说了什么"(消除撒谎)
  • Commit:副本互相对齐"我们都已 prepared"(保证 view 切换后仍可恢复)

少一个阶段会怎样:

只两阶段(只 Pre-prepare + Prepare):
    某副本 prepared 后立刻 execute
    但其他副本可能还没 prepared
    primary 故障 → 新 view 选举 → 新 view 不知道这个 prepared 状态 → 可能选出冲突 request
    → 违反一致性

所以 Commit 阶段是为了"view change 后能恢复 prepared 状态"——这是 PBFT 跟 Raft 最大的区别(Raft 是两阶段)。

3.4 view change(主切换) ​

如果 primary 挂了 / 撒谎 / 太慢,副本要切 view:

  1. 副本本地超时(类 Raft election timeout)
  2. 副本发 <VIEW-CHANGE, new_view, prepared_set>(带上自己已 prepared 的 request)
  3. 新 view 的 primary 收集 2f+1 个 view-change 消息 → 发 NEW-VIEW
  4. 新 primary 用 prepared_set 重做未完成的 request

复杂度:view change 是 PBFT 最复杂的部分——要确保所有已 commit 的 request 在新 view 中保留,所有未 prepared 的 request 可以重新提议。

3.5 PBFT 的 O(N²) 消息复杂度 ​

Pre-prepare:N - 1 条 (primary → 各副本)
Prepare:每个副本广播给其他 N - 1 个 → N × (N-1) 条
Commit:同理 → N × (N-1) 条

总计:O(N²)

N = 4 时 24 条消息,N = 100 时 19800 条——这是 PBFT 在大规模场景(公链)的硬伤。

四、HotStuff:O(N) 的现代 BFT ​

HotStuff(Yin et al., 2019,Facebook Libra 用)是 PBFT 二十年后的重要改进。核心思想:用线性消息复杂度 + 链式 commit + threshold signature。

4.1 关键优化:链式 BFT(Chained HotStuff) ​

PBFT:三阶段对每条 request 独立完成
HotStuff:把多条 request 串成链,新 request 的 Prepare 阶段顺带完成上一条的 Commit
Block N:    Prepare ─▶ Pre-commit ─▶ Commit ─▶ Decide
Block N+1:           Prepare ─▶ Pre-commit ─▶ Commit ─▶ Decide
Block N+2:                    Prepare ─▶ Pre-commit ─▶ Commit ─▶ Decide
                              
                              ▲
                              4 个连续 block 流水线化推进
                              每个 block 在前面那个的"下一阶段"时进入

好处:摊销开销,实际单 block 的延迟降到 4 个轮次,但消息可以批量。

4.2 关键优化:Threshold Signature(门限签名) ​

PBFT 痛点:副本之间互相广播 Prepare / Commit,每条消息要带签名 → N² 个签名验证。

HotStuff 用 threshold signature:2f+1 个副本对一条消息签名,可以聚合成一个签名——leader 只需收集 2f+1 个签名,聚合后广播给所有副本,所有副本只需验证一个聚合签名。

传统:每副本验证 2f+1 个签名,O(N²) 验签
HotStuff:每副本验证 1 个聚合签名,O(N) 通信

代价:门限签名需要预先做 DKG(Distributed Key Generation),节点加入退出复杂。

4.3 HotStuff 流程(简化) ​

View N:
    Leader 收 commands → 提议 block
    阶段 1(prepare):leader 收 2f+1 个签名 → 形成 QC(Quorum Certificate)
    阶段 2(pre-commit):leader 用上一阶段 QC 提议 → 收 2f+1 → 新 QC
    阶段 3(commit):同上
    阶段 4(decide):同上 → 执行

四个阶段都是"leader 广播 → 副本签名 → leader 聚合"的模式
通信复杂度:O(N) 每副本(leader 收 N,副本只发 1 个给 leader)

HotStuff 是 Libra(后改名 Diem)的共识基础,后被多个项目采用(Aptos / Sui / Flow 的早期都基于 HotStuff)。它在 PBFT 的安全性基础上,把消息复杂度从 O(N²) 降到 O(N),让 100+ 节点的 BFT 变得可行。

五、Tendermint / Cosmos:工业级的 BFT + PoS ​

Tendermint 是 Jae Kwon 2014 年提出的 BFT 共识,PBFT 改进 + leader 轮换,后来发展成 Cosmos 区块链生态的核心。

5.1 设计要点 ​

特征Tendermint 做法
算法基础PBFT + 改进(2 阶段 prepare + pre-commit)
Leader每轮轮换(round-robin / weighted by stake)
故障切换内置(轮次超时自动换 leader)
节点身份PoS(Proof of Stake) —— 抵押币当 validator
拜占庭容错严格 N ≥ 3f+1
最终性即时最终性(deterministic finality)—— 区块 commit 后不会回滚

5.2 Tendermint 三阶段(类 PBFT) ​

Round R:
    Propose:本轮 leader 提议 block
    Prevote:validator 广播 prevote
    Precommit:收到 2f+1 prevote → 广播 precommit
    Commit:收到 2f+1 precommit → 区块 commit,进入下一 height

如果本轮 leader 挂或超时:
    自动进入下一轮(R+1),换 leader 重试

跟 PBFT 比的区别:

  • PBFT 是有故障才换 leader(view change 是事件触发)
  • Tendermint 是每轮都换 leader(round-robin)——降低单 leader 性能压力,均衡 validator

5.3 BFT + PoS 的妙处 ​

PoS 解决了"谁是 validator"问题:

传统 PBFT:N 个 validator 是固定配置,管理员决定
        → 适合联盟链(已知参与方)
        → 不适合公链(谁都能加入)

PoS:任何持币者都可以抵押币成为 validator
    抵押越多投票权越大(stake-weighted)
    作恶被罚没抵押币(slashing)
    → 适合公链,有经济激励对抗作恶

Cosmos / Tendermint = 公链版 PBFT:

  • BFT 保证强一致(N ≥ 3f+1 的 voting power)
  • PoS 保证 validator 集合开放且有反作恶激励
  • 即时最终性:区块 commit 后绝对不会回滚(对比比特币的"概率最终性")

5.4 工业应用 ​

  • Cosmos Hub(ATOM)、Binance Chain、Terra(经典)、Crypto.com Chain
  • Hyperledger Burrow 早期用 Tendermint
  • 大量 Cosmos SDK 链:几十个独立链都用同一套共识

六、PBFT 不是公链共识 ​

很多人混淆:PoW / PoS 公链是"拜占庭容错的",但PoW / PoS 不是 PBFT 家族。

6.1 PoW / PoS 的本质 ​

PoW(Bitcoin):
    所有节点竞争算 hash,赢者打包区块
    最长链 = 真链(probabilistic finality)
    6 个块确认后,逆转概率 < 10^-9
    
PoS(Ethereum 2.0):
    抵押币的节点按权重轮流出块
    新块被多数 validator 投票后视为 finalized
    finalized 后逆转需要超过 1/3 stake 同时作恶(惩罚极重)

两者都不是严格 BFT——它们是经济学 + 概率论 + 部分共识的组合。

6.2 PoW/PoS vs BFT 对比 ​

维度PBFT/HotStuff/TendermintBitcoin PoWEthereum PoS
最终性即时确定概率性(6 块 ≈ 1 小时)混合(Casper finality, ≈ 12 分钟)
TPS1k - 10k+≈ 7≈ 15-30(L1)
节点数上限数十到数百(消息复杂度限制)数万数十万(validator)
节点身份已知(联盟)或 stake-weighted匿名,任何人挖抵押 32 ETH
能耗极低(普通服务器)巨大(挖矿)低
抵御 51% 攻击f < N/3 拜占庭即可需 51% 算力需 1/3 stake

公链 PoW/PoS 是为"无许可、大规模、匿名节点"设计的,性能远不如 PBFT(因为节点身份未知,信任成本高)。联盟链 / 容灾级金融用 PBFT/Tendermint,公链生态用 PoW/PoS——这是商业模型决定的,不是技术好坏。

七、几个 BFT 协议对比 ​

协议年份阶段数消息复杂度Leader最终性工业代表
拜占庭将军问题1982理论O(N^(f+1))——学术
PBFT19993O(N²)静态即时Hyperledger Fabric(Kafka 模式之外)

HotStuff20194(链式)O(N)轮换即时Diem / Aptos / Sui

趋势:从 PBFT 的 O(N²) 走向 HotStuff 的 O(N),从静态 leader 走向轮换/无 leader,从批量打包走向 DAG 并行。核心目标:让 BFT 在 100+ 节点规模下仍能跑出几千 TPS。

八、PBFT 在哪里被工业用了 ​

系统用 PBFT 家族备注
Hyperledger Fabric可选 PBFT(早期);新版 Raft 模式(crash-only)联盟链;Raft 模式不是 BFT,只防 crash
Hyperledger SawtoothPoET / Raft / 可插拔联盟链
R3 CordaNotary 可选 BFT(Raft 也行)金融联盟

注意:Hyperledger Fabric 早期支持 PBFT,但社区主推的是 Raft(crash-only 排序服务)——因为联盟链的 validator 节点通常由参与方运维,虽然互不信任,但拜占庭故障的实际概率很低,Raft 性能更好。

这反映了一个工业现实:真正用 BFT 的场景比想象中少。即使是号称"防拜占庭"的联盟链,很多落地实现也降级用 Raft + 业务层签名 + 审计——纯算法层面的 BFT 用得不多。

九、什么时候真的需要 BFT ​

9.1 真的需要 BFT 的场景 ​

✅ 公链(无许可,匿名节点)
    Bitcoin / Ethereum / Cosmos / Solana ...
    
✅ 联盟链(多方互不信任,有明确法律合规)
    跨银行清算、供应链溯源、数字身份联盟
    
✅ 高安全要求的多方协作
    电子选举、隐私计算的协调层

9.2 不需要 BFT 的场景(99% 的业务) ​

❌ 公司内部集群(节点同属一家公司)
    用 Raft / Paxos / ZAB,信任内部运维

❌ 大多数互联网服务
    高可用足够,拜占庭故障极少发生

❌ 一般金融系统(银行内部 / 支付公司内部)
    强一致 + 审计 + 对账,不需要 BFT 性能开销

❌ 中间件 / 协调服务
    etcd / Consul / ZK 都不是 BFT

9.3 BFT 的代价 ​

维度Raft / PaxosPBFT
节点数(容忍 1 节点故障)34
消息复杂度O(N)O(N²)
典型延迟1-10ms30-100ms+
典型 TPS万级-十万级千级-万级
代码复杂度中(几千行 Raft 库)高(view change + 签名 + 拜占庭防御)
运维复杂度中高(签名密钥管理、PoS slashing 逻辑)

粗略估计:BFT 比 Raft 的工程开销大 3-5 倍——节点数、消息量、延迟、代码复杂度都翻倍以上。这是为什么 99% 的业务不用 BFT:多花 3 倍成本,换的是"对抗你内部从不存在的攻击者"。

十、面试 / 思考时的常见错误 ​

10.1 「区块链 = 拜占庭容错」 ​

部分对。公链是 BFT 的实际应用,但它们用的算法(PoW/PoS)不是经典 BFT,而是经济学 + 概率论的混合。说"区块链用 PBFT"是错的(只有联盟链和部分公链如 Cosmos 用 PBFT 家族)。

10.2 「Raft 也能拜占庭」 ​

错。Raft 假设 crash-only,一个撒谎的 leader 能让所有 follower 接受错误日志——Raft 不验证消息内容,也没有签名机制。要让 Raft 抗拜占庭,要叠消息签名 + multi-party 验证,但这就不是 Raft 了。

10.3 「N ≥ 3f+1 是因为多数派要 > 2/3」 ​

部分对。更精确的说法是:多数派之间必有"诚实节点交集",且交集必须够大支撑下一轮决策——3f+1 是数学下界。

10.4 「BFT 性能差 → 不用 BFT」 ​

错位。BFT 性能差是为 BFT 场景准备的代价。在不需要拜占庭抗性的场景,用 Raft 性能远好于 BFT,这是正确选型,不是 BFT 失败。

10.5 「PoW 是 BFT」 ​

错。PoW 没有 finality guarantee(只是概率最终性),理论上任何时候都可能被更长链回滚。经典 BFT 定义要求 finality + safety after commit——PoW 不满足。所以严格地说,Bitcoin 不是 BFT 系统,而是"实践中安全的概率共识"。

十一、真实事件:BFT 没救的攻击场景 ​

即使用了 BFT,也不能防一切:

11.1 51% 攻击(算力 / stake 集中) ​

BFT 假设 f < N/3 的拜占庭节点——如果攻击者控制超过 1/3,可以阻止共识(liveness 攻击);超过 2/3,可以做任意事(safety 攻击)。

  • Ethereum Classic 多次 51% 攻击(PoW)
  • 小型 Cosmos 链如果 validator 集中,也面临 stake 集中风险

11.2 长程攻击(Long-Range Attack) ​

PoS 系统特有的攻击——老的 validator 私钥泄露后,可以从早期分叉伪造一条更长的链。对策:weak subjectivity checkpoint(节点定期从可信源获取近期区块作锚点)。

11.3 跨链桥被黑 ​

跨链桥本质是多方签名(BFT)+ 中继——Ronin Bridge / Wormhole / Nomad 都被黑过,损失数亿美元。根本原因不是 BFT 算法漏洞,而是 multi-sig 私钥管理 / 签名验证逻辑 bug——BFT 协议正确,代码实现可能错。

11.4 客户端漏洞 ​

协议 BFT 不代表实现 BFT——节点软件本身的 bug(如 DoS、内存安全)可以让 f 个节点同时挂,等效于拜占庭故障。多客户端实现(Geth / Erigon / Nethermind 等)是公链对抗这一点的方式。

十二、踩坑提醒 ​

  1. 以为自己需要 BFT —— 99% 的内部业务用 Raft 就够,别因为"高大上"上 BFT,工程复杂度爆炸
  2. 联盟链选错共识 —— Hyperledger Fabric 的 Raft 排序服务 ≠ BFT,真要拜占庭抗性,要选 PBFT/SmartBFT/etc.
  3. N ≥ 3f+1 配少了 —— 4 个节点(f=1)看起来够,但只容忍 1 节点故障 + 0 拜占庭,实际几乎没用;生产至少 7 节点(f=2)
  4. 消息复杂度爆炸 —— PBFT 在 N > 30 时 O(N²) 不可接受,大规模上 HotStuff / Tendermint
  5. 不验证消息签名 —— BFT 协议的核心是验证每条消息的签名,忘了验证 = 退化成 Raft 假设(假设节点诚实)
  6. 门限签名密钥管理混乱 —— HotStuff / Cosmos 用门限签名,密钥泄露会破坏 BFT 假设,密钥管理是工程重点
  7. 混淆 finality —— PoW 是概率最终性,PBFT 是即时最终性,业务侧"区块确认后是否能回滚"完全不同
  8. PoS slashing 逻辑没设计好 —— 抵押币没收逻辑写错,作恶节点不受惩罚,等于没有经济安全
  9. 以为公链 BFT 抗一切 —— 51% / 长程 / 桥攻击都能击穿,BFT 是算法层,不是系统层
  10. 拜占庭跟"故障 = 攻击"画等号 —— 硬件错误也算拜占庭(磁盘静默损坏、内存翻转),金融级系统即使无攻击也可能想要 BFT 处理硬件错
  11. BFT 系统的客户端不验证回复 —— PBFT client 应该等 f+1 个相同回复才信,只信 1 个回复 = 完全没用

十三、收束:这一层的全景 ​

到这里,共识协议这一层(09-15 篇)讲完了——

        FLP 不可能定理(异步 + 一个故障 = 共识不可能)
            │
            ▼
        加上"部分同步"假设(实际网络通常是 partial sync)
            │
        ┌───┴───┬─────┬───────┐
        ▼       ▼     ▼       ▼
    Crash 模型 → Paxos → Raft → ZAB(为 ZK 定制)
                       │       
                       ├── multi-Raft(规模化)
                       │       
                       └── EPaxos(无 leader,跨地域优化)
                       
                       ▼
                       
        Byzantine 模型 → PBFT(1999)
                       → HotStuff(O(N) 改进)
                       → Tendermint(BFT + PoS,Cosmos)
                       → PoW/PoS(概率最终性,公链)

共识协议的工程图景:

  • Raft 是当下绝对主流——95% 的新系统选它
  • Paxos 留在老牌系统——Google 内部、Cassandra LWT、SQL Server
  • ZAB 留在 ZK——Hadoop 时代的协调王,新时代被 Raft 替代
  • EPaxos 是理论王者——工业落地几乎为 0
  • PBFT/HotStuff/Tendermint 在联盟链 + 公链——99% 的业务用不上

这一层的核心教训:

  1. 没有银弹——每个协议都是在「安全性 / 可用性 / 性能 / 复杂度」上做权衡
  2. 简单胜过完美——Raft 的胜利证明,易实现易理解的协议比理论最优的协议更重要
  3. 场景决定选型——拜占庭场景上 BFT,crash 场景上 Raft,互不替代,各司其职
  4. 理论与工程之间巨大鸿沟——Paxos 论文 1989,工业广泛使用要等 Raft 2014;PBFT 论文 1999,工业大规模应用要等区块链 2014+
  5. 再好的算法实现得错也白搭——Jepsen 报告里几乎每个共识库都被找到过 bug

下一篇我们离开共识层,进入"一致性模型"这一层(16-19 篇)——共识协议保证了"日志一致",但应用程序看到的"数据一致"还有 5-6 个档位:Linearizability、Sequential、Causal、Eventual...... 共识只是手段,一致性模型是用户看到的承诺。99% 的"诡异分布式 bug"是一致性模型选错了——这一层会让你看清"为什么 Redis Cluster 不是真线性一致"、"为什么 DynamoDB 的强一致读不是真线性一致"。

看完这五篇共识(11-15),你应该能在白板上画出 Paxos / Multi-Paxos / Raft / ZAB / PBFT 的状态机和流程,能回答"何时该选哪个"。做到这点,你就读懂了所有现代协调服务、分布式数据库、区块链系统的底层秘密。

第三层小结(09-15 共识层) ​

这一层是分布式系统理论的最值钱七篇,因为:

  • 几乎所有分布式中间件的底都在这一层——你天天用的 etcd / ZK / Kafka / TiDB / CockroachDB / MongoDB / Consul,底下要么 Raft 要么 Paxos 要么 ZAB
  • 故障排查最常落到这一层——脑裂、leader 切换抖动、commit 卡住、follower 落后,全是共识协议的某条不变量被工程实现破坏
  • 面试的"硬核区" —— 高级架构师 / 中间件岗位的核心面试题就在这层
