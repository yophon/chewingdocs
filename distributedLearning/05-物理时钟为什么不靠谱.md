# 物理时钟为什么不靠谱

写分布式系统的人,**第一个要被打掉的直觉就是"我可以用本地时间戳排序"**。你以为 `System.currentTimeMillis()` 拿到的是"现在",**它其实是一台跑着 NTP 的机器对"现在" 的近似估计,误差可以从几毫秒到几小时不等**。更糟的是这个值**会倒退**——NTP 会把时钟拨回去,系统重启会跳跃,闰秒会让一秒变两秒。**95% 的"诡异分布式 bug"最终都能追到"有人在跨机器场景下信了物理时钟"这一条**。

> 一句话先记住:**物理时钟是一个"对真实时间的本地估计",不是真实时间**——它会漂、会被纠正、会倒退、会卡住、会在闰秒时变怪。**跨机器排序、跨机器超时、跨机器证据,统统不能用挂钟(wall clock)**——要么用单调时钟做本地间隔测量,要么用逻辑时钟(Lamport / VC / HLC)做跨机因果排序,要么上 GPS / 原子钟(Spanner TrueTime)把误差控制到已知边界。**任何"时间戳大就是后写"的代码,都是潜在事故**。

## 一、晶振漂移:你的服务器时钟到底有多准

服务器主板上的实时时钟(RTC)是一片**石英晶振**——通电后以固定频率震荡(典型 32768 Hz),每一次震荡计一个 tick。**问题是这个频率不准**:

- 标称频率:32768 Hz
- 实际频率:**因温度、电压、老化、批次而变**

晶振的偏差用 **ppm(parts per million)** 描述:

| 等级 | 偏差 | 一天累计 |
| --- | --- | --- |
| 普通服务器 RTC | ±20-50 ppm | **±1.7 ~ 4.3 秒/天** |
| 高精度商用 RTC(TCXO) | ±2 ppm | ±170 ms/天 |
| OCXO(恒温晶振) | ±0.01 ppm | ±0.86 ms/天 |
| 原子钟(铯钟) | ±10⁻¹³ | 约 30 万年差 1 秒 |

```
                  晶振一天能漂多少?
       ┌─────────────────────────────────┐
       │ 一天 = 86400 秒                  │
       │ 50 ppm = 50/1,000,000           │
       │ 漂移 = 86400 × 50 / 1M = 4.32 秒 │
       │                                  │
       │ → 一台不联网的服务器,**两周能漂出一分钟**│
       └─────────────────────────────────┘
```

### 1.1 温度的影响

晶振对温度极敏感。**数据中心一台服务器从冷机房挪到机柜深处,温度上升 10°C,频率漂移可以变 30-50 ppm**。这就是为什么:

- 同一批服务器,**不同机柜位置的时钟漂移速率不一样**
- 高负载机器(CPU 满载发热)和闲置机器**漂得不一样快**
- 空调坏掉一晚,**整个机柜的时钟一致地漂歪**

### 1.2 老化效应

晶振用得越久频率越漂。**典型石英晶振每年漂移可以累计 1-5 ppm**。一台用了 5 年的服务器和新服务器**对"频率"的看法是不同的**。

> **结论:不联网的服务器,时钟在几小时到几天的尺度上就开始不可靠**。**联网用 NTP 校时,但 NTP 也有自己的问题**——下面继续。

## 二、NTP:工程上的"软校时",但有它自己的坑

Network Time Protocol(NTP)是 1985 年的协议,几乎所有服务器都跑 ntpd / chronyd / systemd-timesyncd。**工作原理是分层级的(Stratum)**:

```
Stratum 0:  GPS / 原子钟 / 无线电时间源(物理基准)
              │
              ▼
Stratum 1:  直接接 Stratum 0 的服务器(公共 NTP 服务,如 NIST)
              │
              ▼
Stratum 2:  从 Stratum 1 同步,机房级 NTP
              │
              ▼
Stratum 3:  从 Stratum 2 同步,业务服务器
              │
              ▼
   你的应用服务器(通常 Stratum 3 或 4)
```

每一层都有累积误差,**生产环境普通服务器和"真实时间"的偏差通常在 1-100ms**。

### 2.1 NTP 的两种纠错方式:slew vs step

```
slew:   慢慢调,通过微调时钟频率让时间漂回去
        优点:时钟单调递增,不跳跃
        缺点:慢——纠正 1 秒偏差大约需要 2000 秒(0.5 ms/s 的纠错率)
        
step:   直接跳,把时钟一次性拨到正确值
        优点:立刻同步
        缺点:**时间可能往后跳(变大)或往前跳(变小,时钟倒流!)**
```

**ntpd 默认行为**:

- 偏差 < 128ms:slew(慢慢调)
- 偏差 > 128ms:step(直接跳)——**这就是时间倒流的来源**
- 偏差 > 1000s:拒绝同步(认为时间源有问题)——**这条曾让运维直接懵了:为什么我的服务器一直不校时?**

```
真实场景:
  你的服务器开机时 BIOS 时钟错了 30 分钟
  ntpd 启动 → 偏差 1800s → 拒绝同步
  → 时钟一直错 30 分钟
  → 业务报"日志时间不对" → 运维查不出来
```

### 2.2 chronyd:现代替代,但还是会跳

systemd-timesyncd 和 chronyd 是现代主流,**chronyd 比 ntpd 收敛快、对网络抖动适应好**——但它仍然在大偏差时 step,只是阈值可配。

### 2.3 NTP 的精度边界

| 部署 | 典型误差 |
| --- | --- |
| 公网 NTP(NIST、pool.ntp.org) | **10-100ms** |
| 同 AZ 机房内部 NTP | **1-10ms** |
| 同机架(局域网) | **< 1ms** |
| PTP(IEEE 1588,需硬件支持) | **< 100μs** |

> **不要相信"我们用了 NTP,所以时钟很准"**——同机房的两台服务器,**时钟差 5-50ms 是常态**。**跨机房差 100-500ms 是常态**。如果你的逻辑依赖"两台机器时钟差 < 1ms",**那 bug 不是会不会出,而是什么时候出**。

## 三、闰秒:工程灾难的发源地

地球自转在变慢(月球潮汐摩擦),所以**国际原子时(TAI)和世界协调时(UTC)会越拉越远**。为了让 UTC 跟着地球走,**每隔几年加一个"闰秒"**——UTC 在某天的 23:59:59 之后**插入一个 23:59:60**,然后才是次日 00:00:00。

```
平时:    23:59:58 → 23:59:59 → 00:00:00
闰秒:    23:59:58 → 23:59:59 → 23:59:60 → 00:00:00
                                  ^^^^^^^^
                                  这一秒怎么办?
```

操作系统接到闰秒的反应:**有的实现把 23:59:59 重复两次,有的实现卡在 23:59:60,有的实现直接跳过**——**全世界没有统一答案**。

### 3.1 2012 年 6 月 30 日:Linux 内核 hang

闰秒插入,**Linux 内核 hrtimer 子系统的一处 bug 被触发**——很多 Java 服务、MySQL、Cassandra 进程进入 100% CPU 死循环:

- Reddit、Mozilla、LinkedIn 都出现大面积服务异常
- 部分服务器只能重启才能恢复
- **触发原因**:闰秒导致的时钟跳跃让内核计时器逻辑死循环

### 3.2 2017 年 1 月 1 日:Cloudflare DNS bug

Cloudflare 自家代码在闰秒插入时出错:

- Cloudflare 的 RRDNS 内部计算时间间隔时,假设两次调用 `time.Now()` 总是返回**递增**的时间
- 闰秒插入时,**第二次调用返回的时间比第一次小** → 计算出**负数的时间间隔**
- 这个负数被传到一段 `time.Duration` 的代码 → **崩溃**
- 全球 DNS 解析受影响约 1 小时

> Cloudflare 的 post-mortem 标题就叫 *How and why the leap second affected Cloudflare DNS*——**这是教科书级别的"假设挂钟单调递增"的反面教材**。

### 3.3 Google 的 "Smear" 方案

Google 在 2008 闰秒前发明了一个工程招数:**Leap Smear**——把闰秒的 1 秒"抹"在 24 小时内,每秒钟慢一点点(放慢 ~11.6 μs/s),24 小时后总共多了 1 秒。

```
传统方案:           23:59:59 → 23:59:60 → 00:00:00
Google Smear:      白天每秒慢 11.6 μs,24h 累计 1 秒
                   闰秒到来时,系统已经平滑过渡完成
```

**优点**:没有跳跃,所有应用代码透明。
**缺点**:**Smear 期间,你的时钟和 UTC 差 0-1 秒**——和外部系统通讯时不一致。

Amazon 后来也采用了类似方案,**但 NTP 公共服务器没有统一**——你的服务器订哪个 NTP 源,决定你看到的是"跳"还是"Smear"。

### 3.4 IERS 的最新决定

2022 年国际计量大会决定**到 2035 年取消闰秒**——大家都觉得闰秒带来的灾难比解决的对齐问题大。**未来 10 年里你还要处理闰秒,但 2035 之后不需要了**。

## 四、真实事故:物理时钟踩坑大全

### 4.1 Cassandra LWW 数据丢失

Cassandra 默认冲突解决策略是 **LWW(Last Write Wins)**——同一个 key 多次写入,**保留 timestamp 最大的**。timestamp 怎么生成?**默认用客户端的本地时钟**。

事故场景:

```
T0:  客户端 A(时钟正常):写 K=1, ts=100
T1:  客户端 B(时钟快了 1 小时):写 K=2, ts=3700
T2:  客户端 A 想纠正:写 K=3, ts=101
     → Cassandra 比较 ts:101 < 3700,丢弃 K=3
     → 用户看到的是 K=2(过时数据),无论 A 怎么写都不会生效
```

**直到 B 的时钟被纠回来 + 1 小时后,A 的写才能"覆盖"那条旧写**。生产事故里这一类问题**最难排查**——所有写日志都正常,但数据"消失"了。

### 4.2 MongoDB 选主异常

MongoDB 副本集选主时,**比较各节点的最新写入时间(optime)**。如果某节点的时钟快了 10 分钟,**它会以为自己有"最新的写入"**——而实际上它只是时钟错了。

```
节点 A:时钟正常,最新写 optime=1000s
节点 B:时钟快 10 分钟,最新写 optime=1600s(其实是 1000s + 时钟偏差)
节点 C:时钟正常,最新写 optime=1000s

选主时:B 谎报"自己有最新的写"
       → B 当选 → 实际上数据回退 10 分钟
```

历史上 MongoDB Jepsen 测试就因为时钟假设暴露过多个一致性问题。

### 4.3 跨机房交易乱序

跨机房写订单,每个机房用本地时钟打 timestamp:

```
T 真实:北京 13:00:00.000 下单(北京机器时钟准)
T 真实:上海 13:00:00.001 下单(上海机器时钟慢 50ms,记为 12:59:59.951)

后台同步到中心库,按 timestamp 排序:
  → 上海的订单 (12:59:59.951) 排在北京 (13:00:00.000) 之前
  → 但物理上上海是后下的!
```

如果业务依赖"按时间排序判断秒杀谁先到",**这种乱序直接让结果错乱**。

### 4.4 Kafka 消息时间戳错乱

Kafka 0.10+ 支持 `LogAppendTime` 和 `CreateTime` 两种消息 timestamp。**如果你选 CreateTime(默认)且 producer 时钟错了**,**消息在 Kafka 里的"事件时间"就是错的**——Stream 处理的窗口聚合全错。

### 4.5 分布式锁因时钟跳跃失效

Redlock 类锁(下面 26 篇详谈)用 TTL + 本地时钟判断"锁是否过期"。**如果持锁机器的时钟被 NTP 突然往前跳 30 秒,锁瞬间"过期"**——但持锁线程不知道,继续以为自己持有锁。这就是 Martin Kleppmann 喷 Redlock 的核心论点之一。

## 五、CLOCK_REALTIME vs CLOCK_MONOTONIC:工程上必须分清

Linux / POSIX 提供两个时钟接口:

```c
// 挂钟时间(wall clock)
clock_gettime(CLOCK_REALTIME, &ts);

// 单调时钟(monotonic clock)
clock_gettime(CLOCK_MONOTONIC, &ts);
```

| 属性 | CLOCK_REALTIME | CLOCK_MONOTONIC |
| --- | --- | --- |
| 含义 | 系统对"现在 UTC"的估计 | 自某个未指定起点的单调递增计数 |
| 单调性 | **不保证**(NTP 会调) | **严格单调递增** |
| 跨机器一致 | 大致可比(NTP 校时) | **完全不可比**(每台机器起点不同) |
| 是否会跳跃 | **会**(step、闰秒、人为改) | **不会**(slew 影响频率,但不跳) |
| 适用 | 日志、显示时间、跨机粗略比对 | **测量时间间隔、超时、性能计时** |

```
正确用法:
                                            
  测量"A 到 B 用了多久":CLOCK_MONOTONIC
  打日志显示"现在几点":CLOCK_REALTIME
  设置 30 秒超时:CLOCK_MONOTONIC(start + 30s)
  存数据库的 createdAt:CLOCK_REALTIME(显示用)
  
  错误用法(必踩坑):
  
  用 CLOCK_REALTIME 算超时
    → NTP 把时钟拨回 10 秒
    → 超时永远不到 / 立刻超时
    
  用 CLOCK_MONOTONIC 排序事件
    → 不同机器的 MONOTONIC 没有可比性
    → 排序毫无意义
```

### 5.1 Java 里对应的 API

```java
// 挂钟(对应 REALTIME),会跳
long wall = System.currentTimeMillis();

// 单调(对应 MONOTONIC),不会跳
long mono = System.nanoTime();
```

**Java 老代码里满是 `System.currentTimeMillis() - lastTime > timeout` 这种代码,几乎每一个都是潜在 bug**。正确写法是用 `System.nanoTime()` 测量间隔。

### 5.2 Go 的处理

Go 的 `time.Now()` 返回的 `Time` 结构里**同时记录了 wall 和 monotonic 两个时钟**——计算时间间隔时自动用 monotonic:

```go
start := time.Now()
// ... do work ...
elapsed := time.Since(start)  // 自动用 monotonic 部分,不会被时钟跳影响
```

**这是 Go 标准库做得很对的一件事**——大多数语言要程序员自己注意。

## 六、工程对策:对物理时钟的最低期望

把物理时钟当成一个"会撒谎的传感器",建立四条工程纪律:

### 6.1 跨机不要用物理时钟排序

**因果排序用逻辑时钟(下一篇 Lamport,07 篇向量时钟)**;有 GPS / 原子钟时用 TrueTime / HLC(08 篇)。

### 6.2 超时 / 间隔用单调时钟

任何"等 N 秒后超时" / "测量耗时" / "限流时间窗口"的代码,**用 MONOTONIC,不要用 REALTIME**。

### 6.3 不假设跨机时钟差小于 N ms

NTP 校过的同机房机器**仍可能差几十毫秒**。如果你的逻辑依赖"跨机时钟差 < 1ms",**那不是工程,那是赌博**。

如果非要假设,**也要给一个明显大的 bound**(如 ±500ms)+ 在 bound 之外**主动检测并报警**:

```python
# 简化伪代码:某些选主算法会做类似检查
def check_clock_skew():
    my_time = local_clock()
    peer_time = rpc_call_to_peer_for_time()
    skew = abs(my_time - peer_time)
    if skew > 500_ms:
        alert(f"Clock skew too large: {skew}ms")
        refuse_to_serve()  # 防止时钟错乱导致脑裂、错主、错主键
```

### 6.4 用 HLC / TrueTime 把"未知边界"变成"已知边界"

Spanner 的 TrueTime 接口很关键——**不返回一个时间值,而是一个区间 [earliest, latest]**:

```
TrueTime.now() → TT.now()
返回:[earliest, latest]
保证:真实物理时间一定在这个区间里
```

工程上你不能消除时钟误差,但**你可以让误差边界已知**——这就是 Spanner 用 GPS + 原子钟 + TrueTime API 做的事(详见 08 篇)。**误差未知 = 不可用,误差已知 = 可用**。

## 七、为什么逻辑时钟才是分布式的"时间"

物理时钟有三个根本问题,**这三个问题决定了它不适合做分布式的"时间"**:

| 问题 | 后果 |
| --- | --- |
| 漂移 + 跳跃 | 时间戳大不代表"后发生" |
| 没有真实因果信息 | "A 早 1ms" 不代表"A 引起 B" |
| 跨机不可对齐 | 不同机器的时间戳**根本不在同一个坐标系** |

**分布式系统真正需要的"时间"是因果——A 在 B 之前,因为 A 影响了 B**。这不是物理时钟能表达的:

```
反例:
  机器 A:13:00:00.000 写 K=1
  机器 B:13:00:00.005 读到 K=1,基于它写 K=2

  这里 A → B(因果关系成立)
  时间戳 A 早 5ms,看起来对
  
  但如果机器 B 的时钟慢了 10ms:
  机器 B 记录的写时间是 12:59:59.995,比 A 还早
  → 物理时间戳错误暗示 "B → A",但因果是 A → B
```

**物理时钟无法表达"因果",这是逻辑时钟存在的根本理由**——下一篇 Lamport 逻辑时钟,就是要把"因果"建模成一个不依赖物理时间的数学结构。

## 八、把这一篇总结成一张表

| 时钟类型 | 来源 | 单调性 | 跨机可比 | 因果信息 | 用途 |
| --- | --- | --- | --- | --- | --- |
| 本地 RTC(原始硬件时钟) | 晶振 | 否(可被改) | 否(漂移各异) | 无 | BIOS 启动时间 |
| CLOCK_REALTIME | OS + NTP | **否**(会跳) | 大致可比(±NTP 误差) | 无 | 日志显示、粗对齐 |
| CLOCK_MONOTONIC | OS | **是** | **否** | 无 | 间隔测量、超时 |
| NTP | 外部时间源 | — | 误差 1-100ms | 无 | 校 REALTIME |
| PTP(IEEE 1588) | 外部 + 硬件 | — | 误差 < 1ms | 无 | 高精度场景 |
| GPS / 原子钟 + TrueTime | 物理基准 | — | **误差已知** | 无 | Spanner 等强一致系统 |
| Lamport Clock | 算法 | **是**(算法保证) | **是**(规则统一) | **部分**(同进程 / 因果) | 全序、SMR |
| Vector Clock | 算法 | — | **是** | **完整因果** | Dynamo / Riak 冲突检测 |
| HLC(Hybrid) | 算法 + 物理 | **是** | **是** | 因果 + 物理时间近似 | CockroachDB / YugabyteDB |

> **能用逻辑时钟绝不用物理时钟**。**必须用物理时钟时,把误差边界变已知**(TrueTime / NTP 误差监控)。**用 wall clock 排序事件是分布式系统的第一大原罪**。

## 九、做一遍这个小实验

不需要分布式环境,在你自己机器上跑一段代码就能感受到问题:

```python
import time

# 实验 1:挂钟会倒退
for _ in range(10):
    t1 = time.time()
    time.sleep(0.001)
    t2 = time.time()
    if t2 < t1:
        print(f"时间倒流了!t1={t1}, t2={t2}")

# 实验 2:挂钟和单调钟的差异
# 把系统时间往后调 5 分钟(需要 root)
# 然后观察 time.time() 和 time.monotonic() 的变化
```

**真实环境里,把日志里加一行 `monotonic_diff = monotonic_clock_now - prev_monotonic_clock_now`,你能直接看到 NTP 介入时,wall clock 跳了但 monotonic 没跳**。

## 十、踩坑提醒

1. **用本地时间戳排序跨机事件**——分布式系统第一大原罪,**95% 的诡异 bug 来源**
2. **用 wall clock 算超时**——NTP 跳一次时钟,超时永远不到 / 立刻超时
3. **用 wall clock 算限流时间窗**——闰秒 / 时钟跳能让限流瞬间失效或瞬间触发
4. **假设跨机时钟差 < 1ms**——同机房常态 5-50ms,跨机房 100-500ms
5. **假设挂钟严格单调递增**——它会跳、会倒流,**Cloudflare 2017 就栽在这上**
6. **不区分 REALTIME 和 MONOTONIC**——Java `currentTimeMillis()` 是 REALTIME,**测耗时用 `nanoTime()`**
7. **依赖 LWW 用客户端时钟**——客户端时钟乱了,数据"消失"无法排查
8. **闰秒不演练**——闰秒前一周关注 IERS 公告,自己跑一次"时钟跳 1 秒"的混沌测试
9. **混用 NTP 源**——一部分机器订 Smear NTP(Google / AWS),另一部分订标准 NTP,**闰秒时这两批机器看到的"现在" 差 1 秒**
10. **NTP 偏差监控没建**——`ntpq -p`、`chronyc tracking` 要进监控,**偏差 > 100ms 必须告警**

---

**这一篇收束**:物理时钟在分布式系统里**不是用来排序的,是用来显示的**。一旦你写下"`if (a.ts > b.ts) { a happened after b }`",**你已经在埋雷**。**真正能用来"排序" 的是因果——下一篇 Lamport 逻辑时钟,把"因果" 做成一个不依赖任何物理设备的数学对象**,让你跨机比较"谁先谁后"时,不再依赖那块石英晶振。
