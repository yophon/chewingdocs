    一致性哈希再深入

    systemDesign/13 已经把 Karger 经典环 + 虚拟节点讲透了——「环上找顺时针最近的节点」「虚拟节点解决倾斜」这一层不重复。这一篇要回答的是下一层的问题:Karger 环本身有什么不够?Google 为什么造了 Jump Hash 和 Maglev?Rendezvous Hashing 比环更早(1997)、更简单、影响 key 更少,为什么反而没火?——这是工程选型层的事:不同算法换的不是"思想",换的是"时间-空间-变动影响" 三角的不同权衡。看完这一篇,你应该能解释 Discord 为什么用 Rendezvous、Google Maglev 为什么造 65537 的表、Vimeo 为什么从 Karger 换到 HRW。

    一句话先记住:Karger 环 = O(log N) 时间 + O(K·V) 空间 + 1/N 影响,被工程界当"通用最优";Jump Hash = O(log N) 时间 + O(1) 空间,但只能在尾部加减节点;Maglev = O(1) 时间 + O(M) 空间(M=65537),构表 O(M·N) 是它唯一的痛;Rendezvous = O(N) 时间 + O(1) 空间 + 影响 key 理论最优 1/N。没有"通用最好"——节点 N 多大、变动多频、要不要任意删,定了答案。

    一、为什么 Karger 环不是终点

    Karger 1997 年的环算法是奠基性的,但它有三个真实工程痛点:

    1.1 内存:虚拟节点撑爆路由表

    环算法要求每个客户端在内存里维护「所有虚拟节点 → 真实节点」的映射:

    1000 个真实节点 × 200 虚拟节点 = 200000 个条目
每个条目 ≈ 32 字节(虚拟 hash 4 字节 + 真实节点 ID 字符串)
路由表内存 ≈ 6.4 MB

    对几千实例的微服务,客户端启动时这张表光加载就费几百 ms;路由表 gossip 同步占的带宽也不可忽视(Cassandra 早期苦于此)。

    1.2 查询:O(log N) 的二分查找

    idx = bisect.bisect(self.sorted_keys, hash(key))

    20 万个虚拟节点的二分查找 ≈ log2(200000) ≈ 18 次比较——每次路由都做 18 次比较。Google Maglev 每秒要处理几百万包,18 次内存访问已经不能忍(L1 cache miss 累加起来比转发本身还慢)。

    1.3 均匀性:虚拟节点是补丁不是治本

    虚拟节点把均匀性从「节点数 N 上的均匀」变成「N×V 上的均匀」,但这是概率均匀——理论上 σ/μ ≈ 1/√(N·V)。要把负载方差压到 1% 以下,V 要开到几百,内存进一步炸。

    三个痛点合起来:Karger 环适合"节点几十到几百"的中等规模——比如 Memcached 一个集群、Cassandra 一个 ring;一旦节点上千、或者每秒查询上百万,Karger 就不够了——这就是 Google 在 2014 / 2016 连续造 Jump / Maglev 的原因。

    二、Jump Hash:Google 2014 的"无环"奇迹

    论文:A Fast, Minimal Memory, Consistent Hash Algorithm(Lamping & Veach, Google, 2014)。

    核心问题:Karger 是"先把节点放环上再找",Jump Hash 直接问"key 在 N 个桶里属于哪个桶"——不存任何状态,纯计算。

    2.1 算法直觉

    想象桶数从 1 涨到 N,问 key 每次"换桶"的时刻:

    N = 1 时:所有 key 都在桶 0
N = 2 时:每个 key 有 1/2 概率"跳"到桶 1
N = 3 时:剩下的 key 有 1/3 概率"跳"到桶 2
N = k+1 时:还在原桶的 key 有 1/(k+1) 概率跳到桶 k

    最终位置 = key 最后一次跳到的桶号。

    关键发现:这个过程可以用伪随机数模拟,不需要真的扫 N 步——用对数级别的步数就能跳到答案。

    2.2 Google 论文里的 7 行代码

    int32_t JumpConsistentHash(uint64_t key, int32_t num_buckets) {
    int64_t b = -1, j = 0;
    while (j < num_buckets) {
        b = j;
        key = key * 2862933555777941757ULL + 1;  // LCG 伪随机
        j = (int64_t)((b + 1) * (double)(1LL << 31) / ((key >> 33) + 1));
    }
    return (int32_t)b;
}

    就这 7 行——Google 把它放到论文正文,没有任何辅助数据结构。

    Python 版:

    def jump_hash(key, num_buckets):
    b, j = -1, 0
    while j < num_buckets:
        b = j
        key = (key * 2862933555777941757 + 1) & 0xFFFFFFFFFFFFFFFF
        j = int((b + 1) * (1 << 31) / ((key >> 33) + 1))
    return b

    2.3 性能与影响

    • 时间复杂度:O(log N)——平均跳 ≈ ln(N) 步
    • 空间复杂度:O(1)(不存节点表)
    • 均匀性:几乎完美(LCG 伪随机数性质)
    • 影响 key 比例:N → N+1 时,正好 1/(N+1) 的 key 重新分配(理论最优)

    2.4 致命缺陷:只能在尾部加减

    桶号必须是 0, 1, 2, ..., N-1
加节点 → 桶数从 N → N+1 → key 可能跳到新桶 N
删节点 → 只能删最后一个 → 桶数从 N → N-1

    不能任意删某个节点——比如要删桶 3,剩下的桶号变成 {0,1,2,4,5},Jump Hash 算不了"4 和 5"应该接收哪些 key(算法假设桶号连续)。

    工程绕路(代价大):

    1. 删桶 3 时,把它的数据迁移到桶 N-1,然后整体桶数 -1
    2. 维护一张"逻辑桶号 → 物理节点"映射,删节点改映射不动桶数

    Jump Hash 适合的场景:节点数缓慢增长、按下标管理、不会任意下线——比如分片表数(逐步翻倍)、CDN 后端按预编号、流量调度池(整体扩缩)。不适合:云上随时挂掉的实例集群。

    2.5 谁在用 Jump Hash

    • Google 内部:大规模分片表(论文里说应用在 Google 多个 production 系统)
    • YouTube:存储分片
    • Vimeo:早期用 Karger,后来一部分场景换 Jump Hash
    • 众多 Sharding 库(Vitess 内部分片有用)

    三、Maglev Hashing:Google 2016 NSDI 的"查找表"

    论文:Maglev: A Fast and Reliable Software Network Load Balancer(Eisenbud et al., Google, NSDI 2016)。

    背景:Google 的软件负载均衡器(LB),每秒几百万包级别。Karger 太慢、Jump 不能任意删——Maglev 提出"O(1) 查表"的新路子。

    3.1 算法核心:一张大表

    M = 一个大质数(论文用 65537,实际部署 65537 或 655373)
查找表 lookup: [N] × M(长度 M 的数组,每个槽存节点 ID)

查询:bucket = lookup[hash(key) % M]

    就这一步——一次内存访问,O(1)。

    3.2 构表(关键算法)

    目标:让每个节点占的槽数尽量接近 M/N(均匀),且节点变化时表的"重排"尽量小。

    做法:每个节点维护一个长度 M 的"优先列表" permutation,按列表顺序填表。

    节点 i 的 permutation:[p_i(0), p_i(1), ..., p_i(M-1)]
  其中 p_i(j) = (offset_i + j × skip_i) mod M
  offset_i = hash1(node_i) mod M
  skip_i = (hash2(node_i) mod (M-1)) + 1

    填表流程(伪代码):

    def build_maglev_table(nodes, M):
    N = len(nodes)
    # 1. 每个节点准备 permutation
    permutations = []
    for node in nodes:
        offset = hash1(node) % M
        skip = (hash2(node) % (M - 1)) + 1
        perm = [(offset + j * skip) % M for j in range(M)]
        permutations.append(perm)

    # 2. 轮流填表:每个节点按 permutation 抢自己的"心仪槽位"
    table = [-1] * M
    next_idx = [0] * N  # 每个节点扫到了 permutation 的第几位
    filled = 0
    while filled < M:
        for i in range(N):
            # 节点 i 找它 permutation 的下一个空槽
            while True:
                slot = permutations[i][next_idx[i]]
                next_idx[i] += 1
                if table[slot] == -1:
                    table[slot] = i
                    filled += 1
                    break
            if filled == M:
                break
    return table

    直觉:每个节点都按自己的"心仪顺序"轮流抢槽——抢到的先到先得,直到表填满。结果是每个节点恰好占 ⌈M/N⌉ 或 ⌊M/N⌋ 个槽,且槽位分布近似均匀。

    3.3 构表流程图

             ┌─────────────────────────────────────┐
         │   M = 65537 个槽位的查找表           │
         │  [_,_,_,_,_, ... ,_]                │
         └─────────────────────────────────────┘

节点 A 的 permutation: [3, 19, 41, 7, ...]
节点 B 的 permutation: [11, 27, 5, 33, ...]
节点 C 的 permutation: [17, 2, 49, 21, ...]

轮 1:A 抢 3 → B 抢 11 → C 抢 17
轮 2:A 抢 19 → B 抢 27 → C 抢 2
轮 3:A 抢 41 → B 抢 5  → C 抢 49
...
直到 65537 个槽全部填满

    3.4 节点变化时:重建整张表

    节点加 / 删时,Maglev 重新执行上面的构表——M 不变,只是有些节点没了 / 多了。

    关键性质(论文证明):节点 N → N+1 时,表里大约只有 1/N 的槽变了节点——和 Karger 同一量级,但实现是"重建表"而非"环上插入"。

    3.5 性能与对比

    指标 数值
    查询时间 O(1)(一次数组访问)
    查找表内存 M × sizeof(node_id) ≈ 65537 × 4 = 256 KB
    构表时间 O(M · log N)(论文优化版),实际 65537 × 100 节点 ≈ 几 ms
    均匀性 槽位差异 < 1%(M 是质数 + permutation 设计)
    节点变化影响 < 1% 的 key 变更归属(M 远大于 N)

    3.6 谁在用 Maglev

    • Google 全公司软件 LB(GFE 网关后面那一层)
    • GCP Cloud Load Balancing:对外服务
    • Envoy Proxy:支持 Maglev 算法(maglev 负载均衡策略)
    • Katran(Facebook):L4 LB,用了类似 Maglev 的查找表思路
    • Cilium(eBPF 网络栈):L4 LB 用 Maglev

    Maglev 是"超大流量、超快查询、容忍构表慢"场景的标配——L4/L7 LB、API Gateway。不适合:节点 < 50 的小集群(开 256KB 表浪费),或节点频繁变(每次重建表有开销)。

    四、Rendezvous Hashing(HRW):比环更老更优

    论文:A Name-Based Mapping Scheme for Rendezvous(Thaler & Ravishankar, 1996),比 Karger 1997 还早一年。又叫 Highest Random Weight (HRW)。

    4.1 算法直白到离谱

    def rendezvous(key, nodes):
    return max(nodes, key=lambda n: hash(f"{key}:{n}"))

    对 key 和每个节点都算一次 hash,取最大值对应的节点。完。

    4.2 为什么它"理论最优"

    • 均匀性:直接的——hash 函数足够好,每个节点被选中的概率严格 1/N
    • 不需要虚拟节点:天然均匀,M = 1 就够
    • 节点变化影响:N → N+1 时,新节点对每个 key 都重新算 hash,只有 hash 比所有老节点都大的那 1/(N+1) 比例的 key 会迁移——这是数学上的最优

    4.3 时间复杂度的痛

    N 个节点的查询:O(N)(每个节点都要 hash 一次)

    N 上百时,每次路由要算 100+ 次 hash——比 Karger 的 O(log N) 慢一个数量级。

    为什么以前没火:90 年代到 2010 年代初,大家觉得「N=200 时算 200 次 hash 不能忍」——所以选了 Karger 的环。

    为什么近年又火:

    • xxHash / CityHash 等现代 hash 函数:每次 hash 几纳秒,N=200 总共 < 1 微秒
    • SIMD / 向量化:并行算多个节点的 hash
    • N 通常 < 100 的场景(微服务、shard pool):O(N) 完全可接受

    4.4 算法变种

    Weighted Rendezvous(支持节点权重)

    def rendezvous_weighted(key, nodes):
    def weighted_score(n):
        h = hash(f"{key}:{n.id}") / 0xFFFFFFFFFFFFFFFF  # 0~1
        return -n.weight / math.log(h)  # 论文里的 trick
    return max(nodes, key=weighted_score)

    节点权重不同(大机器权重大),还能保持"理论最优影响 key 比例"——Karger 环加权要靠"虚拟节点数 ∝ 权重",远不如这干净。

    Skeleton-based HRW

    对 N 上千的极端场景,把节点组织成树,O(log N) 查询——少有人用,通常 N < 200 时 O(N) 就够。

    4.5 谁在用 Rendezvous

    • Discord:消息分片(N = ~150 服务实例)——他们的 blog 明说从 ring 换到 HRW,理由是没虚拟节点更省内存、加权更直接
    • Vimeo:视频转码任务路由——节点少,O(N) 不是问题
    • Cassandra 4.x 候选:某些场景考虑用 HRW 替代 vnode(还没合入主干)
    • Apache Ignite:用 HRW 做数据分区
    • Cloudflare:早期 CDN 内部某些路由层用 HRW(他们 blog 提过)

    HRW 是"节点 < 200、要 weighted、要影响 key 最少"的"小众但最优"选择——简单到一行代码,均匀性免费,加权友好。贵在 N 大的时候,但现代 hash 函数让"贵"的阈值往后挪了。

    五、四算法横向对比

    算法 时间 空间 节点变化影响 key 任意删除 加权 适用 N 典型系统
    Karger 环 O(log N·V) O(N·V) 1/N 支持 虚拟节点数加权 几十~几千 Cassandra / Memcached / DynamoDB
    Jump Hash O(log N) O(1) 1/N(理论最优) 只能删尾 不直接支持 任意 YouTube / Vitess
    Maglev O(1) O(M) ≈ 256 KB < 1% 支持(重建表) 间接(改 permutation) 几十~几千 Google LB / Envoy / Cilium
    Rendezvous O(N) O(1) 1/N(理论最优) 支持 原生支持 < 200 Discord / Vimeo

    5.1 一句话选型

    节点变动频繁、规模中等(几百以内)、要支持任意删 → Karger 环
节点按下标管理、缓慢增长、不会任意下线        → Jump Hash
LB 场景、超低查询延迟、节点几百到几千           → Maglev
节点少(< 200)、要加权、要影响 key 最少        → Rendezvous

    六、工程映射:四算法谁落在哪儿

    系统 算法 备注
    Memcached(libmemcached) Ketama(Karger 变种) 200 vnode/节点,事实标准
    Redis Cluster 固定 16384 slot(算 Karger 简化) slot 是虚拟节点的极端形态
    Cassandra Karger + vnode + Murmur3 hash 默认 256 vnode/节点
    DynamoDB(原始 Dynamo 论文) Karger 环 MD5 hash + N=1 副本因子
    ScyllaDB Karger + vnode 完全对齐 Cassandra
    Google Maglev / GFE Maglev M = 65537
    Envoy Proxy Maglev(maglev LB 策略) M 可配
    Cilium / Katran(FB) Maglev / 类 Maglev eBPF L4 LB
    YouTube 分片 Jump Hash Google 论文明说
    Vitess Jump Hash 分片表选址
    Discord Rendezvous (HRW) 消息分片,blog 明说
    Vimeo 转码 Rendezvous 任务路由

    没有"通用最好"——Memcached 选 Ketama 因为 2000 年那时只有 Karger;Google 自己造 Maglev/Jump 是因为 Karger 不够快;Discord 选 HRW 是因为 N 小+要加权。算法选型 = 看场景的三个参数:N 量级、查询频率、节点变动频率。

    =========== Karger 环 ===========
add_node(n):
  for i in 0..V:
    h = hash(n + ":" + i)
    ring.insert(h, n)
sorted_keys = sorted(ring.keys)

get_node(key):
  h = hash(key)
  idx = binary_search(sorted_keys, h)
  return ring[sorted_keys[idx % len]]

=========== Jump Hash ===========
jump_hash(key, num_buckets):
  b, j = -1, 0
  while j < num_buckets:
    b = j
    key = key * 2862933555777941757 + 1
    j = (b + 1) * 2^31 / ((key >> 33) + 1)
  return b

=========== Maglev ===========
build_table(nodes, M = 65537):
  for each node:
    offset = hash1(node) % M
    skip = (hash2(node) % (M-1)) + 1
    permutation[node] = [(offset + j*skip) % M for j in 0..M]
  table = [-1] * M
  while not_full(table):
    for each node:  # 轮流抢
      slot = next_open_slot(permutation[node])
      table[slot] = node

get_node(key):
  return table[hash(key) % M]

=========== Rendezvous (HRW) ===========
get_node(key, nodes):
  return max(nodes, key=lambda n: hash(key + ":" + n))

    八、超出"哈希"的分片策略

    一致性哈希解决的是「无序 key 怎么分到节点」,但真实数据不一定要 hash——有时「按顺序分」反而更好。简短列三种正交策略:

    8.1 Range Sharding(范围分片)

    节点 1: 用户 ID  [0, 10000)
节点 2: 用户 ID  [10000, 20000)
节点 3: 用户 ID  [20000, +∞)

    优势:范围查询天然友好(SELECT * WHERE id BETWEEN 5000 AND 7000 落在一个节点) 劣势:热点风险——最新 ID 段(自增主键)总打到最后一个节点

    典型系统:HBase(RegionServer)、Spanner / TiDB(自动分裂的 range tablet)

    8.2 Hash + Range(混合)

    分两层:
  外层 hash 分桶:user_id % 16 → 16 个 partition
  内层 partition 内部按 timestamp 排序

查询用户 X 最近 100 条消息:
  → 先 hash 找到 partition
  → 再 partition 内按时间范围扫

    Cassandra 的 partition key + clustering key 就是这个模式——((user_id), timestamp DESC)。

    8.3 Directory-based(目录分片)

    一张"shard 表":key → 节点(任意映射)
所有路由先查这张表

    优势:最灵活——可以手动调度热点 key 到大机器、可以按业务调优 劣势:目录服务本身是瓶颈和单点——通常用 ZK / etcd 存目录,加缓存兜底

    典型系统:Vitess(VTGate 查 shard 表)、MongoDB(config server)、早期 Foursquare(惨痛事故:目录服务挂导致全站不可用)

    8.4 一张表选策略

    数据访问模式 推荐分片
    纯 KV、点查多、扩缩频 Karger / Maglev / HRW(取决于 N)
    范围扫描多 Range
    时序数据(用户 × 时间) Hash + Range(Cassandra 模式)
    需要手动调度热点 Directory
    节点编号连续、缓慢扩容 Jump Hash

    九、一致性哈希的"近年新趋势"

    9.1 Consistent Hashing with Bounded Loads(Google 2016)

    经典 Karger 环 + 虚拟节点,仍然可能出现"某节点暂时拿到 1.5× 平均负载"。Google 2016 论文:给每个节点设上限,超了就溢出到下一个节点。

    Vimeo 在 nginx-upstream 用过、Haproxy 有相关补丁。

    9.2 AnchorHash(2018)

    对 Jump Hash 的扩展:支持任意删除(用一个"已删除节点"的 bitmap 配合)。学术上有意思,工业上还没大规模铺。

    9.3 Multi-probe Consistent Hashing(Appleton & O'Reilly 2015)

    比 Karger 内存少:不存大量虚拟节点,改用「k 次探测」——查询时算 k 次 hash 取最低负载。Twitter 的 Cache 路由层试过。

    过去 30 年:1997 Karger → 1996 HRW → 2014 Jump → 2016 Maglev → 2016 Bounded Loads → 2018 AnchorHash——一致性哈希这个看似"简单"的题目,学术工业一直在卷。

    十、踩坑提醒

    1. 以为"一致性哈希"就是 Karger 环——还有 Jump、Maglev、HRW 三个工程主流,选型看场景的三个参数(N、查询频率、变动频率),别一刀切。
    2. 用 Jump Hash 还期望任意删节点——Jump 假设桶号连续,删中间桶会让后面所有 key 错位。只能删尾,不行就别用 Jump。
    3. Maglev 表太小——M 至少 100 × N,M = 100 时 N = 5 就严重不均匀;Google 用 65537 是有道理的。
    4. HRW 在 N=1000 还硬上——O(N) 查询每次算 1000 次 hash,比 Karger 慢一个数量级。HRW 的甜区是 N < 200。
    5. 环上的 hash 函数选 Java hashCode / Python hash——碰撞概率高、分布不均;用 Murmur3 / xxHash / CityHash。
    6. 节点 ID 用 IP——重启换 IP 就变新节点,整片 key 重新分配;用稳定 UUID + 持久化(Cassandra 学过这个教训)。
    7. 客户端不同步路由表——某个客户端的环停在旧状态,大量请求路由到已下线节点;用 gossip 或控制面统一同步。
    8. Maglev 节点变动时不重建表——某节点下线了表里还指着它,请求 fail;节点状态变化必须触发表重建,且原子切表。
    9. 以为 Redis Cluster 用了"一致性哈希"——它用的是固定 16384 slot 的"目录分片"(配 Gossip 同步),节点变动是 slot 迁移,严格说不算 Karger 环。
    10. 范围分片场景硬上哈希——按时间分的日志数据上 Karger,范围查询变成扇出到 N 个节点;有顺序的数据老老实实用 Range / Hash+Range。
