分布式锁 ​

分布式锁是最容易"看起来 work、实际全错"的分布式原语。SETNX 漏 EXPIRE 一脱机锁永久占用、Redis 主从切换两个客户端同时拿到锁、Redlock 被 Martin Kleppmann 痛批"在 GC 暂停下不安全"——这些都是真实生产事故的种子。这一篇要做两件事:一是把"锁的两个用途"分清楚(效率锁 vs 正确性锁)——选错用途,选什么实现都是错的;二是把 Redlock 论战(2016)讲清楚——Kleppmann 是对的还是 antirez 是对的,看场景。看完这一篇,你应该能在白板前讲清楚:"什么锁可以用来去重,什么锁绝对不能用来扣库存"。

一句话先记住:锁的核心不变量是"互斥"——同一资源同一时刻只能有一个持有者;但分布式锁多了两个隐性不变量:"锁过期但持有者仍以为自己持有"绝不允许、"锁过期后被另一客户端拿到,旧客户端的写不能落地"必须保证。Redis 单实例锁能保证互斥但保证不了后两个;Redlock 解决了部分但被 GC 暂停 + 时钟跳变击穿;etcd / ZK + Fencing Token 才是"正确性锁"的标准答案。金融转账用 Redlock 是事故等级,用 Redis 单锁是灾难等级。

一、锁的"两个用途"——这是选型的根 ​

读完这一节,你就不会再问"分布式锁该用 Redis 还是 ZK"——你会先问"我这个锁是哪一种"。

1.1 效率锁(Efficiency Lock) ​

目的:避免重复劳动——同一份工作不要多个 worker 重复做。

场景:定时任务调度
  - 100 个实例都跑了同一段代码
  - 用锁保证"每分钟只有一个实例真正跑任务"
  - 如果"偶尔"两个实例同时跑了一次 → 浪费点 CPU,任务多跑一次,不影响正确性

特征:失败的代价是浪费,不是错乱。

适合的实现:Redis 单实例锁(够快、便宜、偶尔双跑可接受)。

1.2 正确性锁(Correctness Lock) ​

目的:保护不变量——同一资源同一时刻被多个客户端写,数据会破。

场景:扣库存
  - 库存 = 1,客户 A 和 B 同时下单
  - 锁保证"只有一个客户能扣成功"
  - 如果两个都扣成功 → 超卖,经济损失、客户投诉

特征:失败的代价是数据错乱、不可恢复。

适合的实现:etcd / ZK + Fencing Token + 业务幂等兜底。

绝对不适合:Redis 单实例锁、Redlock——理由后面 Kleppmann 论战会讲透。

1.3 第三类:计算资源调度 ​

场景:K8s Controller Manager 选主
  - 多个副本只能有一个真正干活
  - 其他副本待机,主挂时立刻接管
  - 这是"选主"——下一篇 27 讲透,但底层就是锁

特征:接近正确性锁,但持有期长(分钟到小时)——通常用 lease 而非锁。

1.4 选型决策的第一问 ​

写代码前先问自己:

"如果这个锁失效了、两个客户端同时进入临界区,我会丢钱吗?会出错乱吗?会有不变量被破坏吗?"

  • 会 → 正确性锁 → etcd / ZK + Fencing + 业务幂等
  • 不会,只是浪费 → 效率锁 → Redis 够用

90% 的"分布式锁出事故"都是把正确性锁的活给了效率锁的实现。

二、Redis 单实例锁:演进与每一代的坑 ​

2.1 第一代:SETNX + EXPIRE 两条命令 ​

bash
SETNX my_lock "client_1"      # 设置成功 = 拿到锁
EXPIRE my_lock 10              # 10 秒过期

致命缺陷:两条命令不是原子的——SETNX 成功后客户端崩溃,EXPIRE 没执行,锁永远占用(没 TTL)。

凡是看到生产代码里有这两条命令分开写的,认定是错的。

2.2 第二代:Lua 脚本组合 ​

lua
if redis.call('SETNX', KEYS[1], ARGV[1]) == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[2])
    return 1
else
    return 0
end

有效但啰嗦——直到 Redis 2.6.12 加入原子的 SET ... NX PX。

2.3 第三代:SET key value NX PX(2013 至今) ​

bash
SET my_lock "client_1_uuid" NX PX 10000
# NX:不存在才设置
# PX:毫秒级过期
# value 必须是唯一 ID(下面解锁时要用)

一行搞定 "互斥 + 过期"——Redis 锁的现代标准写法。

2.4 解锁:必须验"锁主"——Lua 脚本 ​

新人最容易写错的代码:

python
# 错误写法
if redis.get("my_lock") == my_uuid:
    redis.delete("my_lock")

问题:get 和 delete 之间锁过期了、被其他客户端拿走了,这里 delete 会删别人的锁。

正确写法(Lua 原子):

lua
-- 解锁脚本
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end

调用:

python
LUA_UNLOCK = """
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
"""
redis.eval(LUA_UNLOCK, 1, "my_lock", my_uuid)

关键纪律:解锁必须验锁主——判断"我还是不是当前锁的持有者"和"删除"必须原子。

2.5 锁状态机 ​

   ┌──────────┐
   │ FREE     │ 没人持有
   └────┬─────┘
        │ SET NX PX 成功
        ▼
   ┌──────────┐
   │ LOCKED   │ 当前持有者: client_X (uuid)
   │ TTL=10s  │
   └────┬─────┘
        │
        ├─ Lua 解锁(验 uuid + del):FREE
        ├─ TTL 到期:FREE(但 client_X 不知道!⚠)
        └─ Redis 重启 / 主挂从升:状态丢失 ⚠

两个"⚠" 就是 Redis 锁的真正风险——下面 Kleppmann 喷的就是这两个。

2.6 Redis 单实例锁的"主挂丢锁"问题 ​

T0: 客户端 1 在主节点 set my_lock NX PX,成功
T1: 主节点把写复制到从节点(异步,可能还没到)
T2: 主节点挂了
T3: Sentinel 把从升主
T4: 客户端 2 在新主上 set my_lock NX PX,成功(从节点没收到 T1 的写)

→ 两个客户端同时持有同一把锁 ⚠

Redis 主从是异步复制(05 篇讲过),主挂时未同步的写丢失——锁也是写。

antirez 提的 Redlock 就是为了治这个。

三、Redlock:antirez 2014 的"跨多实例 majority"方案 ​

3.1 思想 ​

不依赖单个 Redis 实例——部署 5 个独立 Redis 实例(没有主从关系,各自独立),只要在 majority(3 个)上拿到锁,就算拿到。

3.2 算法步骤 ​

客户端 acquire(key, ttl):
1. 记录当前墙钟时间 T_start
2. 对所有 5 个实例并发执行:
     SET key uuid NX PX ttl
   设置一个较短的网络超时(比如 50ms),失败立刻继续下一个
3. 统计成功数 N_ok
4. 计算耗时 T_elapsed = now() - T_start
5. 如果 N_ok >= 3(majority)且 T_elapsed < ttl:
     → 拿到锁,实际有效期 = ttl - T_elapsed
   否则:
     → 失败,**异步对所有 5 个实例发解锁**(包括"成功"的)

解锁 release(key, uuid):
   对所有 5 个实例发 Lua 解锁脚本(验 uuid + del)

3.3 关键直觉 ​

  • 大多数 Redis 实例不会同时挂(独立部署、独立电源、不同机架)
  • 3/5 majority 拿到锁,任意 2 个挂掉,剩下 3 个仍能解锁或拒绝其他客户端
  • 客户端必须扣掉网络耗时——拿到锁的"有效时间"比 TTL 短

3.4 为什么 antirez 觉得这是"安全的" ​

对比 Redis 主从锁的失效路径:主挂时数据没同步到从 → 新主上别人也能拿锁。

Redlock 没这个问题——5 个独立实例,任意一个挂了,其他 4 个数据还在;majority 仍然能否决新的 acquire。

antirez 在 redis.io 上挂出 Redlock 算法,推荐给生产——这时 Martin Kleppmann 出手了。

四、Kleppmann vs antirez(2016):分布式锁史上最知名的论战 ​

4.1 Kleppmann 的喷文(2016-02-08) ​

标题:How to do distributed locking

核心论点:Redlock 不安全,因为它依赖"墙钟时间"和"假设没有长 GC 暂停"——这两个假设在真实分布式系统里都不成立。

4.2 喷点 1:GC / 进程暂停可让"过期锁"被同时持有 ​

经典时序图(Kleppmann 文章里的核心反例):

T=0   客户端 1: 拿到锁,TTL=10s
T=1   客户端 1: 准备写数据库...
       (此时 JVM 突然 STW GC 暂停了 15s,客户端 1 卡住)
T=10  锁过期,Redis 自动释放
T=11  客户端 2: 拿到同一把锁,TTL=10s
T=12  客户端 2: 写数据库(库存 -1)
T=16  客户端 1: GC 结束,以为"自己还持有锁",写数据库(库存 又 -1)

→ 两个客户端"同时"写了数据库,锁完全失效

关键:Redis 不能"通知客户端 1 你的锁已经过期了"——客户端 1 是按自己的本地时钟以为"还在 TTL 内"。

这不是 Redis 的 bug,是"基于 TTL + 客户端自检"的锁的根本性缺陷——ZK / etcd 也有这个问题,单纯依赖锁本身保证不了。

4.3 喷点 2:墙钟跳变击穿算法 ​

Redlock 算法显式依赖墙钟时间:T_elapsed = now() - T_start。

如果在这两次 now() 之间,系统墙钟被 NTP 跳了 +30 秒:

  • 客户端算出 T_elapsed = 30s,以为锁早就用完,丢弃这把锁
  • 但实际上 Redis 端的 TTL 还在,锁还在被"占着"
  • 别的客户端也无法 acquire(因为 Redis 上锁还没过期)

反过来,墙钟回拨:

  • 客户端算出 T_elapsed = -5s(负的)
  • 以为锁还有效,继续用一个早就被 Redis 自动释放、别人也拿到的锁

05 篇的"墙钟不可信"在这里发挥得淋漓尽致——Lamport 早在 1978 就警告过,Redlock 仍然踩了这个坑。

4.4 antirez 的反驳 ​

antirez 当天就在 blog 写了回应:Is Redlock safe?

几个核心论点:

  1. GC 暂停问题是所有锁都有的——不是 Redlock 独有。ZK / etcd 也一样,客户端 GC 15 秒、ZK 临时节点过期、客户端醒来继续写,同样错乱。
  2. 生产环境的 NTP 不会跳那么大——他主张「合理配置 NTP + 监控时钟漂移」就够了。
  3. 真实使用 Redlock 的生产案例没有大规模翻车——他认为 Kleppmann 是"理论纯洁主义"。

4.5 谁对谁错——分两个维度看 ​

维度Kleppmannantirez
正确性论证对的——GC + 时钟跳变能击穿 Redlock反驳没有真正回应"互斥"被打破的风险
工程实用性偏严苛——99% 业务场景下 Redlock 够用对的——大部分场景生产可用

Kleppmann 的真正结论(常被误读):

不是"Redlock 永远不能用",而是"如果你需要 correctness(正确性、防破不变量),不能只靠 Redlock,必须配合 Fencing Token"。

4.6 关键修正:Fencing Token ​

Kleppmann 给出的修复:

锁服务发"单调递增的 token"
T=0   客户端 1 acquire → token = 33
T=11  客户端 2 acquire → token = 34(原锁过期)
T=16  客户端 1 GC 结束,带 token=33 去写存储
T=12  客户端 2 已经带 token=34 写过
T=16  存储检查:33 < 34,拒绝客户端 1 的写

→ 客户端 1 虽然以为持锁,但写不进去

核心:锁本身不保护"过期但客户端不知道"的情况——下游存储用 fencing token 兜底。

Redlock 给不了 fencing token(Redis 的 SET NX 返回值是 OK/null,不是单调号);ZK 用 zxid、etcd 用 mod_revision——这就是为什么 Kleppmann 推 ZK / etcd。

4.7 论战的真实教训 ​

不要在金融、库存、转账这种"正确性绝对不能破"的场景用 Redlock(也不要用 Redis 单实例锁)。用 etcd / ZK + Fencing Token + 业务幂等三层兜底。

去重、定时任务、API 限频、缓存重建这种"偶尔失效不致命"的场景,Redis 单实例锁就够了——不用上 Redlock,因为 Redlock 的成本(5 个实例)+ 复杂度(GC、网络) 划不来。

五、Fencing Token 详解 ​

5.1 流程图 ​

       ┌──────────────┐
       │ 锁服务         │ 发单调递增 token
       │ (ZK/etcd)    │
       └──┬─────────┬──┘
          │ token=33│ token=34
          ▼         ▼
     ┌────────┐  ┌────────┐
     │client 1│  │client 2│
     └───┬────┘  └───┬────┘
         │           │
         │ write(data, token=33)
         │           │ write(data, token=34)
         ▼           ▼
       ┌─────────────────┐
       │  下游存储          │
       │  记录:last_token │ 单调检查:写入 token 必须 > last_token
       └─────────────────┘

5.2 锁服务发 token 的实现 ​

  • ZK:每次创建顺序节点 /lock/0000000033,zxid 单调递增;也可以用节点名后缀的 sequence number(更直接)
  • etcd:每次 Put 操作返回 mod_revision,全集群单调递增;Lease 加锁时拿到的 revision 就是 token
  • Redis:没有原生支持——antirez 提过用 INCR 维护一个全局计数器作为 token,但 Redis 主挂会导致计数器倒退,自己又踩进同一个坑

5.3 下游存储的"单调拒旧" ​

幂等性写入的标准 pattern:

python
def write_with_fencing(key, value, token):
    last_token = storage.get_last_token(key)
    if token <= last_token:
        return ERROR("stale token: %s < %s" % (token, last_token))
    storage.set_last_token(key, token)
    storage.write(key, value)

真实工程:

  • HBase / TiKV:内部的写入流程都有 epoch / revision 单调检查
  • MySQL:用版本号字段 + UPDATE WHERE version = X AND ...
  • GFS chunk server:每个 chunk lease 带 version,老 primary 续约失败后即使写也被 chunk server 拒(下一篇 GFS Master Lease 详讲)

Fencing Token 是分布式锁"正确性"的最后一道闸——没有它,任何锁都可能在 GC / 时钟下被击穿。

六、ZooKeeper 锁:临时顺序节点 ​

6.1 数据结构 ​

ZK 路径:
  /my_lock/
    ├── lock_00000001   (临时顺序节点, 客户端 A 创建)
    ├── lock_00000002   (客户端 B 创建)
    └── lock_00000003   (客户端 C 创建)

关键性质:

  • 临时节点(ephemeral):客户端 session 断开,节点自动删除——锁自动释放
  • 顺序节点(sequential):ZK 自动给节点名加单调递增后缀

6.2 加锁流程(伪代码) ​

python
def acquire_zk_lock(zk, lock_path):
    # 1. 在 /my_lock 下创建 ephemeral + sequential 节点
    my_node = zk.create(
        f"{lock_path}/lock_",
        flags=EPHEMERAL | SEQUENTIAL
    )
    # my_node 形如 "/my_lock/lock_00000007"

    while True:
        # 2. 列出所有子节点,排序
        children = sorted(zk.get_children(lock_path))

        # 3. 我是不是最小的?
        my_name = my_node.split("/")[-1]
        if children[0] == my_name:
            return my_node   # 拿到锁!

        # 4. 不是最小,监听"前一个"节点的删除
        my_idx = children.index(my_name)
        prev_node = f"{lock_path}/{children[my_idx - 1]}"

        # 5. 设 watcher,前面节点删除时被唤醒
        if zk.exists(prev_node, watch=True):
            wait_for_watch_event()
        # 唤醒后重新检查(可能前面又有插队)

def release_zk_lock(zk, my_node):
    zk.delete(my_node)

6.3 为什么这套设计很"工程级" ​

临时节点 = 自动解锁 ​

客户端崩溃 / 网络断开,ZK 在 session timeout 后自动删除该节点——无需 TTL,无需手动续约(其实 ZK session 内部就在续约,但用户无感)。

顺序节点 = 防惊群 ​

朴素做法:所有客户端都监听 /my_lock 的子节点变化——任何一个客户端释放锁,所有等待者都被唤醒,然后都去争锁(惊群)。

ZK 的优化:每个等待者只监听"前一个"节点的删除——N 个等待者,释放锁只唤醒 1 个(下一个最小的)。

00001 (持锁)
00002 (监听 00001)
00003 (监听 00002)
00004 (监听 00003)

00001 释放 → 只唤醒 00002 → 00002 拿到锁
00002 释放 → 只唤醒 00003 → ...

zxid / sequence = 天然 Fencing Token ​

节点名里的 sequence(00000007)就是 Kleppmann 喊的 Fencing Token——全局单调,下游存储拿来做单调检查就行。

6.4 ZK 锁的局限 ​

  • session timeout 是双刃剑:太短,网络抖一下客户端误以为还持锁实际已被释放(GC + 长 session timeout 也救不了 Kleppmann 的反例,但能缓解);太长,客户端真的崩了,锁要等好久才释放
  • 写性能不高:ZK 写要 Quorum 共识(ZAB),每秒几千写,做锁服务够用,做高频锁不行
  • 复杂:API、Watcher、Recipe 都是坑——Curator 库帮你封装了,但还是要懂

6.5 ZK 锁状态机 ​

   ┌──────────────┐
   │ NOT_OWN      │ 没创建过节点
   └──────┬───────┘
          │ create ephemeral_sequential
          ▼
   ┌──────────────┐
   │ WAITING      │ 创建了,但前面还有节点
   │ watch(prev)  │
   └──────┬───────┘
          │ prev_node 被删 + 我是最小
          ▼
   ┌──────────────┐
   │ OWN          │ 我是最小节点,持锁
   │              │
   └──────┬───────┘
          │
          ├─ 主动 delete: NOT_OWN
          └─ session 失效: 节点自动删除 → NOT_OWN

七、etcd 锁:Raft + Lease ​

7.1 etcd 锁的设计 ​

etcd 也有 ephemeral 概念,但实现是Lease(租约,下一篇详讲)。

1. 客户端向 etcd 申请一个 Lease(比如 10s)
2. 用这个 Lease 写一个 key /my_lock = client_uuid
   写入时附带 lease_id —— key 跟 lease 绑定,lease 过期 key 自动删
3. 客户端持续给 lease "keep alive"(续约),只要客户端活着,lease 不过期
4. 客户端崩了 → keep alive 停 → 10s 后 lease 过期 → key 自动删 → 锁释放

**对应 ZK 的临时节点,etcd 用 Lease 实现。

7.2 etcd 用 mod_revision 做 Fencing Token ​

go
// etcd v3 锁(伪 Go 代码)
resp, _ := client.Grant(ctx, 10)         // 10s lease
lease_id := resp.ID
client.KeepAlive(ctx, lease_id)          // 后台续约

// 用 transaction 实现 SETNX 语义
txn := client.Txn(ctx).
    If(clientv3.Compare(clientv3.CreateRevision("/my_lock"), "=", 0)).
    Then(clientv3.OpPut("/my_lock", uuid, clientv3.WithLease(lease_id))).
    Else(clientv3.OpGet("/my_lock"))
result, _ := txn.Commit()

// 拿到的 result.Header.Revision 就是 Fencing Token
fencing_token := result.Header.Revision

etcd 官方提供 concurrency.NewMutex 封装——内部就是上面的逻辑 + ZK 那套"顺序 + 监听前一个"的防惊群优化。

7.3 K8s 用 etcd 做什么锁 ​

  • kube-controller-manager / kube-scheduler 的 leader election:多副本只能一个干活,用 etcd lease 实现(下一篇详讲)
  • CRD Operator 的 leader 选举:Kubernetes 标准库 leader-election,全是 etcd 锁
  • Pod 调度的资源占用记录:某种意义上也是锁

九、决策矩阵:你应该用哪种锁? ​

你的锁失效,会丢钱 / 错乱 / 破坏不变量吗?
├── 是 → 正确性锁
│       ├── 已经用 K8s / Go 生态 → etcd Lease + mod_revision fencing
│       ├── 已经用 ZK / Java 生态 → ZK 临时顺序节点 + sequence fencing
│       └── 没有 ZK/etcd,但需求强       → 上 etcd(轻量级、易部署)
│
└── 否,只是浪费 / 重复劳动 → 效率锁
        ├── 单机房 / 简单场景 → Redis 单实例 SET NX PX + Lua 解锁
        ├── 主挂可接受偶尔丢锁 → Redis 单实例足够
        └── 跨机房 / 不能忍受丢锁 → ZK / etcd(别用 Redlock,复杂度不划算)

还要追加一条铁律:

正确性场景的最终防线永远是"业务幂等 + 对账"——任何锁都可能因为意外失效(GC、时钟、网络),业务层必须设计成"重复执行也不出错"(扣库存用 UPDATE ... WHERE stock > 0、转账用幂等号 + 流水对账)。锁是优化,不是兜底。

十、踩坑提醒 ​

  1. SETNX 不带 EXPIRE 就当锁用——客户端崩了锁永远占着,直接生产事故;用 SET NX PX 一行命令。
  2. 解锁不验"锁主"——直接 DEL 会删掉别人的锁(本地 GC / 网络延迟导致锁已过期被他人拿走);用 Lua 脚本 GET + 比较 + DEL 原子。
  3. Redis 单实例锁做转账 / 扣库存——主挂从升的瞬间两个客户端同时拿锁,超卖、错账;用 etcd / ZK + Fencing。
  4. 用 Redlock 做金融 ——Kleppmann 论战的核心反例,GC 暂停能击穿;金融场景必须 Fencing Token + 业务幂等。
  5. 以为 Redlock 比 Redis 安全很多——主从异步丢锁的问题它解决了,但 GC + 时钟的问题没解决;没拿 Fencing 之前,Redlock 不算"正确性锁"。
  6. ZK / etcd 锁不用 Fencing Token——光靠"我持锁"还不够,下游存储不验单调号,Kleppmann 反例同样发生在 ZK 上。
  7. 锁的 TTL 设得太短——业务逻辑超时(比如调外部 API 卡 30s),锁过期,临界区被别人也进入;TTL > 业务最大延迟 × 安全倍数,或用续约机制。
  8. 锁的 TTL 设得太长——客户端崩了锁要等很久才释放,整段业务卡住;配合 watchdog 续约(Redisson 标准做法)。
  9. 不区分效率锁和正确性锁——所有锁都上 ZK(浪费、慢)或所有锁都上 Redis(正确性破)——最常见的工程错误;按场景分。
  10. 指望分布式锁取代业务幂等——任何锁都会有失效的瞬间,业务层用幂等 ID / 唯一约束 / CAS 做最后一道闸;没有幂等的扣减用任何锁都不安全。

下一篇 27-选主与租约:锁的近亲就是"选主"——多副本只能一个干活,这本质上是"长期持锁"。Lease(租约)是"锁的延伸版本"——给一段时限的承诺、期内独占、过期续约。下一篇拆开 GFS Master 怎么用 lease 协调 chunk 写顺序、Chubby 怎么用 lease 做选主、K8s 怎么用 etcd lease 让 controller-manager 单实例运行;核心是"如何避免双主"——租约 > 心跳超时、新主等老租约过期、Fencing 兜底、Quorum 单分区下不可能双主。没看清这一篇,你设计的高可用系统大概率会双主导致数据写花。
