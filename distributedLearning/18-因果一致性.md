# 因果一致性

上一篇讲了 Linearizability——最严的一档,但代价是**分区时必须放弃可用性(CAP 定理的严格结论)**。**那么有没有一种一致性,既保留"合理的因果顺序",又允许分区时所有副本都继续接受写?**——有,**因果一致性(Causal Consistency)就是 CAP 之外的甜点**。**Mahajan & Alvisi 2011 论文 *Consistency, Availability, and Convergence* 严格证明了:Causal Consistency 是分区下可同时保证可用性的最强一致性档**。也就是说,**因果一致是"绕过 CAP 的极限"**——任何比它强的(Linearizable / Sequential)在分区下都得放弃可用性,**它正好压线**。**协同编辑(Google Docs / Yjs / Figma)、社交时间线、跨地域多写、评论引用前一条**——这些场景都靠它撑着。

> 一句话先记住:**因果一致 = "因果相关的操作有序,并发的随意"——评论必须在被回复者之后,但两条互不相关的发帖可以乱序**。**它绕开了 CAP**——Mahajan 2011 证明:**Causal+ 是分区下兼容可用性的最强一致性**,所有跨地域多写、协同编辑的系统都在这一档上。**代价是元数据膨胀**——每条消息要带依赖元信息(版本向量),**GC 是真正的难题**。

## 一、为什么需要因果一致——Linearizable 太贵,Eventual 太松

回到 03 / 16 篇的谱:

```
Linearizable  ───  Sequential  ───  Causal  ───  Eventual
强,贵                                              弱,任意

跨地域:
  Linearizable:每次写 50-150ms(等多数派,可能跨洲)
  Eventual:    每次写 <5ms(本地副本接,异步同步)

跨地域多写场景(美西、欧、亚 都能写):
  Linearizable:几乎做不到("跨洲共识"性能崩)
  Eventual:    数据冲突没语义,合并靠"最后写赢"=丢数据
  Causal:      本地写立返,因果依赖跨地域传播 ── 甜点
```

**典型不能要 Linearizable 但又不能用纯 Eventual 的场景**:

```
社交平台:Alice 发帖 "我刚买了房" P
        Bob   看到 P 后,回复 "恭喜!"   R   (R 因果依赖 P)
        Carol 来浏览

合法历史:
  Carol 先看到 P,再看到 R  ✓
  Carol 没看到 R,只看到 P  ✓(R 还没传到 Carol 的副本)
  Carol 看到 R,没看到 P    ✗  ←  "看到回复,没看到原帖" = bug
```

**Eventual 不阻止第三种**——副本传播没顺序,可能 R 先到。**Causal 强制阻止**——Carol 副本看到 R 时必须先看到 P 才能让 R 可见。

这就是因果一致最实用的"保护"——**不要求全局有序,但要求"因果链不断"**。

## 二、happens-before 与因果的精确定义

复习 06 / 07 篇:Lamport 的 **happens-before**(`→`):

```
1. 同一进程内:a → b 当且仅当 a 在 b 之前
2. 跨进程:    send(m) → recv(m)
3. 传递:      a → b 且 b → c,则 a → c
4. 否则:      并发(concurrent),记作 a ‖ b
```

**因果一致性的定义**:

> 任何客户端看到的操作历史,**都和某个保持 happens-before 偏序的串行化等价**。**对并发操作,不同客户端看到的顺序可以不同**。

注意几个关键:

- **happens-before 不是真实时间**——是逻辑因果
- **并发操作无所谓顺序**——但**同一客户端必须看到统一的顺序**(否则 Monotonic Read 都破了)

**判定因果**:**向量时钟 / 版本向量**(07 篇)。每次写带上"导致这次写的依赖集",副本拿到写后,要等所有依赖到达才能可见。

## 三、因果依赖图(一张图把概念吃掉)

直接画:

```
                  Alice 客户端                Bob 客户端           Carol 客户端

  t=1   写 P1: "我买房了"  ──→
                          │
                          ▼ 副本传播
  t=2                                  读到 P1
                                         │
                                         ▼
  t=3                                  写 R1: "恭喜!"           ──→
                                         depends_on={P1}            │
                                                                    ▼ 副本传播
                                                                  
  t=4                                                              读 ←  R1 来了,但 P1 还没到
                                                                       因果一致:R1 必须等 P1 才可见
                                                                       因此 Carol 阻塞 R1 直到 P1 到

依赖图(DAG):
                  P1
                  │
                  ▼
                  R1

  R1 的依赖集 deps(R1) = {P1}
  任何副本在让 R1 可见前,必须先有 P1
```

**并发例子**(两人同时发帖):

```
  t=1   Alice 写 P1: "天气真好"
        Bob   写 P2: "我刚跑了 10km"     ←   两个写没因果关系(互不知道)

  P1 ‖ P2(并发)

  其他客户端可以先看到 P1 再看到 P2,也可以反过来 ── 都合法
```

## 四、Mahajan 2011:CAP 的"擦边球"

**论文** *Consistency, Availability, and Convergence*(Mahajan, Alvisi, Dahlin,2011)。**核心结论**:

> **在异步网络下,Causal Consistency + Convergence(收敛性)是兼容总可用性的最强一致性**。

也就是说,所有比它强的(Sequential / Linearizable)**必须**在分区时牺牲可用性;而 Causal **不必**——分区时各副本仍可继续接受写,网络恢复后通过传播因果依赖,**最终所有副本收敛到合法的因果序**。

**这条结论改变了分布式存储设计的一个范式**——**跨地域多活的"上限"就是 Causal+**,**再强就过不了 CAP 这一关**。

```
                  CAP 边界
                  ↓
   ─────┬─────────┴──────────┬─────────
        Linearizable      Causal+
        分区时不可用       分区时可用!
```

> **"Causal Consistency is the strongest consistency model that satisfies Availability"** —— Mahajan et al.,这一句话**值整个第四层**。

## 五、COPS:第一个工业级因果一致 KV 存储

**论文** *Don't Settle for Eventual: Scalable Causal Consistency for Wide-Area Storage with COPS*(Lloyd et al., SOSP 2011)。

### 5.1 设计目标

- 跨地域多副本,**本地写立返**(不跨洲共识)
- **保证 Causal+**(Causal 一致 + 收敛冲突处理)
- 在 Facebook 类社交场景下吞吐高

### 5.2 数据模型

**KV 存储**,每个写带上"依赖集"——你这次 put 之前**看到过**的所有 key 的版本:

```
put(key, value, deps={ (k1, v1), (k2, v2), ... })
get(key) → (value, version)
```

### 5.3 客户端跟踪依赖

每个客户端维护一个 `context`(读过的所有 key/version),每次 put 时把 context 作为 deps:

```python
class CopsClient:
    def __init__(self):
        self.context = {}    # key → version,客户端读到过的最新版本

    def get(self, key):
        value, version = server.get(key)
        self.context[key] = version
        return value

    def put(self, key, value):
        server.put(key, value, deps=self.context.copy())
        self.context[key] = new_version   # 自己写的,自己已知
```

### 5.4 副本侧:依赖检查

副本收到一个写(value=v, deps=D)时,**必须等 D 中所有 (k,v) 都已可见,才让 v 可见**:

```python
def on_replicate(key, value, deps):
    for dep_key, dep_version in deps:
        while not is_visible(dep_key, dep_version):
            wait_for(dep_key, dep_version)
    make_visible(key, value)
```

### 5.5 Causal+:收敛冲突处理

**纯 Causal 允许并发写有不同顺序,会"分歧"**——副本 A 选 v1 → v2,副本 B 选 v2 → v1。**Causal+ 加一条**:**所有副本最终收敛到同一个值**(用 last-writer-wins 或应用定义的 merge 函数)。

```
COPS 的 conflict handler:
  - 默认:LWW(last writer wins,用 Lamport 时间戳)
  - 应用可注册自定义 merge(如计数器 = 和)
```

### 5.6 优化:依赖压缩

朴素的依赖集会**爆炸**(读 1000 个 key,写一次带 1000 条依赖)。COPS 做了**最近依赖压缩**:**只带"最近的因果前驱"**,旧的因为传递性可以推出来,不用带。

## 六、Bolt-on Causal:在最终一致存储上加一层因果

**论文** *Bolt-on Causal Consistency*(Bailis et al., SIGMOD 2013)。

**思想**:**底层用现成的最终一致存储(Cassandra / DynamoDB / Riak),上层加一个"因果层" shim**。客户端通过 shim 读写,**shim 负责依赖跟踪和可见性控制**:

```
应用 ────► Bolt-on shim ────► 底层 eventual store(Cassandra)
              │
              ├─ 维护客户端 context
              ├─ 检查依赖
              └─ 提供 Causal Get / Put 接口
```

**这就是工程上"先用大厂存储,再补一致性"的范式**——不用重写存储引擎,只加 client + 一层 proxy。**Apache 的 Geode、Microsoft 的 Cosmos DB(Consistent Prefix / Session 档)都借鉴了这套思路**。

## 七、工程实践:协同编辑 / 社交 / 跨地域多写

### 7.1 协同编辑:Google Docs / Yjs / Figma

**协同编辑是因果一致 + CRDT(下一篇 19 详讲)的杀手场景**:

- 多人同时编辑一份文档,**本地写立刻显示**(不能等跨洲共识)
- **每次编辑带因果依赖**:"我这次插入是基于看到 v 那个版本"
- 副本同步时,**因果依赖保证编辑顺序合理**——你不会"看到 B 写的回复,却没看到 A 写的原句"

**Yjs**(用得最多的开源 CRDT 库,Figma / Notion 协同的底层之一):

```
每个操作携带:
  - origin_id      (操作 id)
  - origin_clock   (Lamport / 版本向量)
  - left_origin    (插入位置的左邻居 id)   ←  这就是因果依赖

副本合并时:
  - 按 left_origin 排序,确保插入顺序
  - 并发插入用 tiebreaker(client_id)
  ── 收敛到所有副本一样的文档
```

### 7.2 社交时间线 / 评论引用

**Facebook / Twitter 早期就有"看到回复看不到原帖"的 bug**——内部用 TAO 改进时引入了 causal 跟踪。

**简单实现**:**评论带原帖 id + 原帖 version**,前端 / 后端在展示评论前**检查原帖是否已加载**——这就是应用层做的 Causal+ 检查。

### 7.3 跨地域多写

**经典场景**:全球用户在不同地区写同一份共享文档 / 工单 / 协作看板。

```
美西副本:接受美西用户的写,本地立返
欧洲副本:接受欧洲用户的写,本地立返
亚洲副本:接受亚洲用户的写,本地立返

副本间异步传播因果依赖
  ── 因果保持
  ── 并发写用 CRDT 合并(下一篇 19)
```

**这就是 Riak / Cosmos DB / Couchbase XDCR 的设计基础**。**对比 Spanner 的 External Consistency**:

| | Spanner | Causal+(Riak / Cosmos) |
| --- | --- | --- |
| 一致性 | External Consistency(更强) | Causal+(更弱) |
| 跨地域写延迟 | 50-150ms(等多数派) | <10ms(本地副本接) |
| 分区可用 | 否 | 是 |
| 适合 | 银行级跨地域事务 | 协同 / 社交 / 多活 |

## 八、代价:元数据膨胀和 GC

因果一致**不是免费的**。三大代价:

### 8.1 元数据膨胀

每次写都要带**依赖集**(版本向量 / Lamport 时间戳)。**朴素实现**:依赖集大小 = O(N)(N = 副本数)。

```
1000 个 client,每个版本向量 1000 项,每项 8B
  → 一次写带 8KB 元数据
```

**优化**:

- 版本向量压缩(只带活跃 client)
- 依赖压缩(只带最近因果前驱)
- 用 HLC(08 篇)代替向量时钟,把 O(N) 压成 O(1)

### 8.2 跨地域元数据传输

**因果依赖跟着数据走**,跨地域复制时**元数据也要传**。带宽成本不容忽视——大文档协同时,操作元数据可能比内容还大。

### 8.3 GC 问题(最难)

**因果依赖链会无限增长**——10 年前的操作仍可能被新操作依赖到。**怎么 GC**?

- **稳定性条件**:某个操作"所有副本都已确认收到",可以从依赖图里删
- **检测稳定性**:**需要副本间持续通信**(类似快照 / 反熵)
- **副本下线 / 长期不可达**:**会拖住整个 GC**——这是 Riak / Yjs 的真实问题

**Yjs 的做法**:**doc 关闭时把整个 CRDT 状态"压缩"成一个新的 base state,丢弃旧操作历史**。

> **GC 是因果一致系统最容易栽的暗坑**——Aphyr 报告过多个系统因为 GC 不及时导致元数据爆炸 + 性能崩。

## 九、什么时候该选 Causal,什么时候别选

### 9.1 该选

- **不需要全局严格顺序,但需要"自己看到的世界因果合理"**
- **跨地域多写**(本地写立返,异步同步)
- **协同 / 评论 / 社交流 / IM 离线消息**(自然有因果链)
- **应用语义能容忍并发冲突合并**(LWW、CRDT)

### 9.2 别选(必须强一致的场景)

- **金融账户余额**——并发取款不能"两副本都同意各取一次"
- **库存扣减**——并发下单不能"两副本都同意扣到 1 件"
- **分布式锁 / 选主**——必须 Linearizable,**别用 Causal 实现锁**
- **唯一性约束**——username / 订单号唯一性,Causal 撑不住

### 9.3 选了 Causal 但又踩雷的常见原因

- **以为 Causal 就够了,实际业务还要 Read-Your-Writes / Monotonic Reads**——叠加客户端会话保证
- **忘了 Causal+ 中的 "+"——并发冲突必须有 merge 策略**,纯 Causal 不收敛
- **依赖 LWW 但时钟不可靠**——NTP 漂移 / 闰秒,**用 HLC 或 vector clock**
- **没做元数据 GC**——一年后版本向量上千项,延迟飙升

## 十、踩坑提醒

1. **以为 Causal = Eventual 加点料**——不,**Causal 强制保留 happens-before 偏序**,Eventual 完全不保留
2. **以为有版本号就是 Causal**——版本号只能比较先后,**因果检查要的是 happens-before 偏序判定**,得用向量时钟 / HLC
3. **客户端不维护 context**——一切因果跟踪基础失效,**Bolt-on / COPS 都依赖客户端记 context**
4. **副本不做依赖检查就让写可见**——因果链断,可能"看到回复看不到原帖"
5. **依赖集不压缩**——元数据爆炸,带宽 / 延迟双崩
6. **不做 GC**——长期跑下来元数据线性增长,**重启变慢、内存被吃光**
7. **认为 Causal 能取代分布式锁 / CAS**——**绝对不能**,锁和 CAS 必须 Linearizable
8. **并发冲突没 merge 策略**——纯 Causal 不收敛,**必须配 Causal+(LWW / CRDT / 应用 merge)**
9. **用 wall clock 做 timestamp 排序**——时钟回拨破因果,**用 HLC**
10. **以为 Google Docs 是 OT 不是 Causal**——OT 是表象,**本质还是因果一致 + 操作变换 / CRDT**
