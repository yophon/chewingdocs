    CAP 与 PACELC 与 BASE

    CAP 是分布式系统课的第一道开胃菜,也是被讲坏得最厉害的一道。面试时背 "三选二" 能拿三分,真正写系统时背 "三选二" 能让你出大事故。这一篇不重复 systemDesign 那篇的工程口径,而是补三件事:Brewer 2000 PODC 演讲到底说了什么、Gilbert-Lynch 2002 论文怎么把它形式化(以及证明的边界在哪)、为什么 2012 年之后业界用 PACELC 取代 CAP、BASE 怎么在工程上真正落地。看完这一篇,你应该能在白板上对架构师讲清楚"我们这个系统在 PACELC 上的位置在哪、为什么这样选"——而不是"我们是 CP 公司"这种没意义的标签。

    一句话先记住:CAP 是一句关于"网络分区下的不变量"的断言,不是分布式系统的世界观。P 不是给你选的——它是异步网络的物理事实,真正的取舍发生在两个时刻:分区时(C vs A)和正常运行时(C vs L)。PACELC 把后半句补上,BASE 把工程妥协具象化。"CA 系统" 是伪概念,"AP / CP 标签" 是粗暴标记——真实系统是"按数据 / 按操作 / 按时刻"精细化选择的。

    一、CAP 的原始版本到底说了什么

    2000 年 7 月,Eric Brewer 在 PODC(Principles of Distributed Computing)做了一个 keynote,题目叫 Towards Robust Distributed Systems。他没有写论文,只放了几页 slides——CAP 这个词第一次出现就是在这个 keynote 里,而且只是一个口头观察(conjecture),不是定理。

    Brewer 当年讲的原文大意是:

    在共享数据的分布式系统里,一致性(Consistency)、可用性(Availability)、分区容忍性(Partition tolerance)三个属性,你最多只能同时拿到两个。

    注意三个细节:

    1. 这是"猜想",不是"定理"——Brewer 自己说这是经验观察,不是数学结果
    2. 没有给出三个属性的精确定义——Consistency 到底多强、Availability 到底多高、Partition 到底是什么样的分区,全部模糊
    3. 没有数学证明——只是从多年做 Inktomi 搜索引擎、做 BASE 系统的经验里总结出来的直觉

    所以最初的 CAP 是一句"工程师的格言",不是教科书拿来发挥成"三选二定理"的那个东西。把它讲成"三选二的铁律"是后人加戏。

    二、Gilbert-Lynch 2002:把 CAP 形式化(以及边界在哪)

    两年后,MIT 的 Seth Gilbert 和 Nancy Lynch 写了一篇 Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services,把 CAP 在一个特定模型下证明了。这才是后来教科书里那个"CAP 定理"。

    但这个证明的前提非常具体,不读论文你不知道它在证什么:

    元素 论文里的具体定义
    Consistency 线性一致性(linearizability)——读返回最近一次写,所有节点视图等价于单机
    Availability 每个非故障节点收到的请求,必须最终返回非错响应——不限时长,但要求有响应
    Partition tolerance 网络可以丢任意多条消息(模型化为消息可任意延迟或丢失)
    环境 异步模型——节点之间没有共享时钟,消息延迟无上限

    核心证明思路(白板上能讲清楚):

    假设系统在分区时既能保 C 又能保 A。
设两个节点 N1、N2,初始数据相同(都是 v0)。
网络分区,N1 和 N2 互相收不到消息。

  客户端 A 写 N1:v0 → v1   (N1 必须接受,A 要保)
  客户端 B 读 N2          (N2 必须返回,A 要保)
                          (N2 必须返回 v1,C 要保)

但 N2 收不到 N1 的消息,根本不知道 v1 存在。
矛盾。证毕。

    短短几行就能把这条结论推出来。但这个证明的"边界"经常被忽略:

    1. 只在异步模型下成立。论文也证了"部分同步模型(节点知道一个延迟上限)下,可以在分区结束后恢复 C+A"——也就是说现实工程中"分区恢复后回到强一致" 是允许的,CAP 不否认这个
    2. Consistency 限定为线性一致性。因果一致、最终一致这些较弱的模型,不在 CAP 的禁令里——这是为什么 CRDT / Bolt-on Causal 那一派能"绕过 CAP"
    3. Availability 是"总有响应",不是"低延迟响应"——CAP 不管你响应要等多久,这是 PACELC 后来补的洞

    CAP 是一个具体模型下的具体不可能结果,它不是一个能拿来贴在任何系统上的标签。把它误读成"普遍真理"是教科书的锅,不是 Gilbert-Lynch 的锅。

    三、为什么 "CA 系统" 是个伪概念

    很多人讲 CAP 时画一个三角形:CA / CP / AP 三个角,告诉你 MySQL 是 CA、ZK 是 CP、Cassandra 是 AP。这个三角形是错的。

    P 不是选项。在任何跨网络的分布式系统里,网络分区是物理事实——你不能"选择不容忍分区",分区会自己来:

    • 机房交换机重启 30 秒
    • 跨城光纤被挖掉
    • 防火墙规则配错
    • 节点 GC stop-the-world 看起来像分区
    • 节点慢得跟分区一样(灰故障)

    如果你"不容忍分区",那分区发生时你的系统直接挂——这等于你"用不可用来换取假装一致性",但实际上一致性也保不住(因为节点根本不响应,你怎么知道它一致?)。

    理论上的 CAP 三角(错误教学版):
                    C
                   ╱ ╲
              CA ╱   ╲ CP
                ╱     ╲
              ╱        ╲
            A ───────── P
                  AP

实际上的样子(正确版):
              C ───────── A
                  ╲     ╱
                   ╲   ╱
                    ╲ ╱
                  分区时
                  二选一

              (P 永远在那里,不在你的选项里)

    3.1 "MySQL 是 CA" 的常见误读

    有人说 MySQL 单机是 CA、不考虑分区,所以是 CA。严格说也对——但 MySQL 单机不是分布式系统,套 CAP 没意义。

    一旦你说"MySQL 主从",它就是分布式系统了:

    • 主从异步复制 + 主挂自动切从:分区时主从可能脑裂(都以为自己是主)→ 不是 C,是 AP
    • 主从异步复制 + 主挂不切:分区时主不可用 → 不是 A,是 CP(还是个弱化的 CP,因为复制本来就是弱一致)
    • 半同步 + 主挂自动切:分区时如果多数从断连,主写不下去 → 偏 CP

    没有"CA 数据库"。任何一个真实分布式系统都在 CP / AP 的连续谱上。

    四、PACELC:把 CAP 漏掉的那一半补回来

    CAP 只回答了"分区时怎么办"。但真实系统99.9% 的时间都没分区——这 99.9% 的时间里,系统在做什么取舍?CAP 不回答。

    2010 年,耶鲁大学的 Daniel Abadi 提出 PACELC:

    PACELC 念法:Pack-Elk

if (P) then choose between A and C
else      choose between L and C

P  = Partition
A  = Availability
C  = Consistency
E  = Else (no partition)
L  = Latency

    中文表述:分区时(P),选 A 还是 C;否则(E),选 L 还是 C。

            ┌──────────────────────────────────┐
        │       PACELC 决策树              │
        │                                  │
        │   网络分区?                      │
        │     │                            │
        │     ├─ 是 → PA  or  PC           │
        │     │                            │
        │     └─ 否 → EL  or  EC           │
        │                                  │
        └──────────────────────────────────┘

四个组合,贴到真实系统:

  PA / EL : 分区时选 A,平时选低延迟(放弃强一致)
            → DynamoDB, Cassandra, Riak

  PC / EC : 分区时选 C,平时也选 C(接受高延迟)
            → BigTable, HBase, Spanner, MongoDB(majority)

  PA / EC : 分区时选 A,平时选 C
            → 比较罕见,理论上存在

  PC / EL : 分区时选 C,平时选 L
            → MySQL 异步复制(分区时主不可用,平时低延迟但弱一致)

    4.1 用 PACELC 重新标主流系统

    系统 PACELC 解读
    DynamoDB(默认最终一致读) PA/EL 分区时继续服务,正常时低延迟优先
    DynamoDB(强一致读模式) PC/EC 强一致读必须等多数派,延迟高
    Cassandra(默认 ONE) PA/EL 任意一个副本响应就返回,最快但可能读到旧数据
    Cassandra(QUORUM 写 + QUORUM 读) PC/EC 多数派确认,慢但强一致(注意 R+W>N 才严格成立)
    Spanner PC/EC TrueTime + Paxos,任何时候都强一致,代价是 TrueTime 等待
    MongoDB(WriteConcern=majority) PC/EC 多数派确认,选主分区时少数派不可写
    BigTable / HBase PC/EC 单 Tablet 单主,分区时少数派不可用
    ZooKeeper / etcd PC/EC 共识协议要求多数派,分区时少数派 read 也可能拒(zk linearizable read)
    MySQL 异步主从 PA/EL 主挂可丢已确认事务,平时低延迟
    Redis 主从 PA/EL 默认异步复制,主挂可能丢数据
    Kafka(acks=all + min.insync.replicas=majority) PC/EC 少数派 broker 拒写

    PACELC 比 CAP 实用 10 倍——它把"平时的延迟代价"也放上桌面。强一致从来不是免费的:即使没分区,跨地域的强一致也意味着一次写要跨太平洋一个 RTT——150ms 起步。这就是为什么 Spanner 装原子钟也无法把延迟降到 0。

    五、BASE:CAP 工程化的现实派

    CAP 是学术派,BASE 是工程派。BASE 来自 eBay 工程师 Dan Pritchett 在 2008 年 ACM Queue 上写的一篇文章,把当年互联网行业绕开 ACID 的实战经验总结成了一句口号:

    B - Basically Available  基本可用
A - Soft state           软状态
E - Eventual consistency 最终一致

    systemDesign 那篇已经讲过 BASE 怎么对应到电商业务,这里补"BASE 工程化怎么真的落地"——三件事缺一不可。

    5.1 Basically Available:降级而不是全挂

    "基本可用" 不是"挂着但凑合用",而是主动设计降级路径。要在系统里写代码:

    def get_recommendations(user_id):
    try:
        return recommendation_service.get(user_id, timeout=50ms)
    except (Timeout, ServiceDown):
        return fallback_popular_items()  # 兜底
    • 熔断器(Hystrix / Sentinel / Resilience4j):超过阈值的失败率就直接走兜底,不再打下游
    • 舱壁隔离(Bulkhead):线程池 / 信号量隔离,避免一个慢服务拖垮全栈
    • 业务降级开关:大促前关闭推荐、评价、积分写入

    "全有或全无" 是错误的可用性思维。真实系统永远在做"哪些功能可以临时降级"的取舍——做不到这一点的系统,SLA 永远到不了 99.95%。

    5.2 Soft State:接受中间态,但要可观测

    "软状态" 是承认 数据在副本间、服务间、缓存与 DB 之间的不一致是常态——但不能不可见。工程上要做的是:

    • 每个不一致的状态有 TTL —— 缓存有过期、订单 pending 有超时关单
    • 每个跨服务状态变更有事件 / 消息(订单状态变更触发库存、积分、消息推送)
    • 有"对账"机制——离线 Job 每天检查关键数据(订单 vs 支付 vs 库存)
    软状态的危险:不一致 + 不可观测 + 不收敛 = 灾难

正确做法:
  数据有 TTL → 自动收敛(过期重读)
  状态有事件 → 触发下游(消息驱动)
  系统有对账 → 兜底修复(每天一次)

    5.3 Eventual Consistency:最终一致,但要给"最终"加一个数

    最容易踩的坑——"最终一致" 听起来像"反正会一致",但你不能不知道是多久。

    每个 eventual consistency 的链路都要回答两个问题:

    1. 收敛时间窗口是多少? 100ms / 1s / 10s?决定上游能否同步读
    2. 不收敛时怎么知道? 监控、对账、人工告警

    工程上要做:

    • 测量复制延迟(MySQL Seconds_Behind_Master、Kafka consumer lag、Redis 主从延迟)
    • 设置告警阈值(主从延迟 > 5s 报警)
    • 对账兜底(每日 / 每小时定期对账)
    • 不允许"读自己写" 的链路依赖最终一致(用户改完头像必须立刻看到,不能等异步复制)

    互联网公司的 99% 都跑在 BASE 上。强一致(ACID + 2PC)是金融业的根基,但电商、社交、内容、广告、推荐都用 BASE 撑住了 100 倍以上的流量。学会 BASE 思维 = 学会做互联网架构。

    六、反 CAP 误读手册:这些坑你迟早会踩

    6.1 "三选二" 是错的

    正确说法是 "分区时二选一"。平时(没分区)CAP 不限制你,你 C / A 可以都拿。

    6.2 "我们是 CP / AP 公司" 是错的

    CAP 不是公司标签。同一家公司、同一个系统、同一份代码,不同接口可以选不同侧:

    接口 倾向 理由
    POST /transfer(转账) CP 不能多扣
    GET /products(商品列表) AP 看到旧数据没事,系统挂了流失大
    POST /like(点赞) AP(异步) 几秒不一致用户感知不到
    POST /login(登录) CP(对黑名单) 黑名单同步要快

    6.3 "ACID = CP、BASE = AP" 是过简

    ACID 描述的是事务的本地属性(原子、一致、隔离、持久),和 CAP 不是同一个 axis:

    • 单机 ACID:不涉及 CAP
    • 分布式 ACID(2PC):协调者挂了会卡死,这是另一种"不可用",不能简单说是 CP
    • BASE 不是 "AP",BASE 是"工程取舍",它可以叠在 AP 上(主流),也可以叠在 CP 上(强一致系统也需要降级 / 兜底)

    6.4 "NoSQL = AP" 是错的

    数据库 默认倾向
    MongoDB(WriteConcern=majority) CP
    HBase CP
    Cassandra(默认) AP
    Cassandra(QUORUM) CP
    DynamoDB(默认) AP
    DynamoDB(强一致读) CP

    SQL / NoSQL 的标签和 CAP 没有必然联系。SQL 数据库(CockroachDB / Spanner / TiDB)是分布式 CP,NoSQL 也分 CP / AP。

    6.5 "强一致 = 慢" 是过简

    强一致一定有共识开销,但不一定慢得不能用:

    • 同机房 Raft:写 1-3ms
    • 同地域 Raft:写 5-10ms
    • 跨地域 Raft:写 50-100ms(注定的物理延迟)
    • Spanner 跨洲:写 100-200ms

    慢不慢取决于"你能不能接受这个延迟",而不是"强一致必然不能用"。

    6.6 "分区很少发生" 是错的

    Google SRE 在 The Tail at Scale 里讲过:生产环境每月都会有显著的分区事件——交换机故障、配置错误、防火墙规则、跨机房抖动、灰故障(节点慢得跟分区一样)。没设计分区容错的系统迟早出大事。

    七、Brewer 2012:CAP 十二年后的反思

    2012 年,Brewer 在 IEEE Computer 写了一篇 CAP Twelve Years Later: How the "Rules" Have Changed——自我修正了当年那个 keynote 留下的误读。三个关键观点:

    7.1 "三选二" 是误导

    Brewer 自己承认:当年那个口号过于简化。真实系统在分区时也不是"完全放弃 C 或 A",而是部分放弃:

    • 分区时:对关键操作(转账)选 C(拒绝写),对非关键操作(浏览)继续选 A
    • 分区结束后:主动修复期——通过对账、合并、版本协调把分区期间的不一致补回来

    7.2 高可用 CP 不等于破 CAP

    Spanner 是 CP 系统(PC/EC),但 Google 公布的可用性是 5 个 9 (99.999%)——一年宕机不到 5 分钟。怎么做到的?

    高可用 ≠ 不需要 P
高可用 = 通过工程手段把"分区导致的不可用时间"压到极短

工程武器:
  1. 多副本(5 副本 Paxos)→ 单点分区不影响多数派
  2. 多机房 + 多区域 → 整个机房分区也能多数派活
  3. 快速选主(秒级) → 分区导致 Leader 切换时间短
  4. 客户端重试 → 短暂不可用对用户不可见
  5. 跨地域 Quorum → 1 个地域分区不影响全局可用

    "CP 系统不可用率高" 是过时的认知。用足够多副本 + 足够多地域,CP 系统的可用率可以比 AP 系统还高——因为 AP 系统的"假可用"(返回脏数据导致业务出错)不算真可用。

    7.3 分区不是一个开关,是一个连续谱

    真实的"分区" 不是网络断了 / 没断这么二元,而是延迟 / 丢包率连续变化:

    • 完全正常:延迟 1ms,丢包 0
    • 轻度抖动:延迟 100ms,丢包 0.1%
    • 明显抖动:延迟 1s,丢包 5%
    • 灰故障:90% 正常 + 10% 超时
    • 半分区:A 能到 B,B 不能到 A
    • 完全分区:互不可达

    CAP 把分区当成离散事件,但生产环境最多的是"灰分区"——比硬分区更难处理。这是为什么 Jepsen 测试要专门测网络抖动、丢包、半分区,而不只是测断网。

    九、面试 / 白板高频问题

    9.1 "为什么 ZK 是 CP,Eureka 是 AP"

    • ZK:写要 majority Quorum,分区时少数派不可写——配置中心场景宁可拒写也不要错配
    • Eureka:任意节点都能写,分区时各自接受注册——服务发现场景宁可路由到旧实例也不要服务列表为空

    同一个"注册中心" 类别下,选 CP 还是 AP 完全取决于业务——配置不能错 → CP;服务发现宁可旧 → AP。

    9.2 "为什么金融业还在用 2PC,而不是 Saga"

    • 2PC:CP,慢但强一致。协调者挂会卡死,但金融业宁可卡死也不要钱算错
    • Saga:AP / 最终一致,快但每一步都要写"补偿逻辑"

    金融核心账户 → 2PC;支付的周边业务(通知、记录、营销)→ Saga / 最终一致。别一刀切。

    9.3 "Spanner 是 CP,怎么做到 5 个 9 可用性"

    • 多副本(5 副本 Paxos 默认)
    • 多机房(同地域 3 zone)
    • 多区域(跨地域)
    • TrueTime 把"等待"控制在 7ms 内
    • 客户端重试 + 快速选主

    CP ≠ 低可用。工程做得足够好,CP 系统的可用性比 AP 的"假可用"高得多。

    9.4 "我们用最终一致,延迟无所谓吧"

    错。最终一致也有 SLA——通常承诺 P99 收敛时间(如 99% 数据 1s 内一致)。

    • 主从延迟超阈值要告警
    • 读自己写的链路不能走最终一致
    • 必须有对账兜底

    十、踩坑提醒

    1. 把 CAP 当世界观——它只是一个特定模型下的不可能结果,不是"分布式系统的全部"
    2. 画 CA / CP / AP 三角形——P 不在你的选项里,只有 CP 和 AP 的连续谱
    3. 整系统打 CP / AP 标签——CAP 是"按操作 / 按数据"细分的,不是公司战略
    4. 以为 ACID = CP、BASE = AP——这是两个 axis,不能简单等号
    5. 以为 NoSQL = AP——MongoDB / HBase 默认 CP,Cassandra 可调 CP
    6. 以为强一致就一定慢得不能用——同机房 Raft 写 1-3ms,跨地域 50ms,慢不慢看你能否接受
    7. 忽略 PACELC 的 ELC 部分——平时的延迟代价比分区时的可用代价发生得更频繁
    8. BASE 不配对账机制——异步复制可能丢消息,没有对账兜底的 eventual consistency 是定时炸弹
    9. 以为"我们没分区"——生产环境每月都有,灰分区比硬分区更常见也更难处理
    10. 把 CAP 决策只交给技术——CAP 是产品 + 业务决策,技术不知道业务真正在乎什么,拍出来的决策几乎都错

    这一篇收束:CAP 不是用来背的,是用来"指向"的——指向"分区时你要在 C 和 A 之间选"这件事,然后用 PACELC 把"平时延迟" 也指出来,最后用 BASE 把工程落地的姿势讲清楚。面试时背"三选二"会扣分,真懂的人讲"按操作精细化选择 + 用 PACELC 看正常态 + 用 BASE 落地工程"。
