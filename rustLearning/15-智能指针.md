# 15-智能指针

> 一句话导读：智能指针是在普通引用之外管理所有权、共享和可变性的工具，常用于递归结构、共享数据和运行时借用检查。

普通引用 `&T` 只是借用，不拥有数据。智能指针通常拥有或管理它指向的数据，并在离开作用域时执行额外逻辑。Rust 里常见的智能指针包括：

- `Box<T>`：把值放到堆上，仍然只有一个所有者。
- `Rc<T>`：单线程引用计数，允许多个所有者共享只读数据。
- `RefCell<T>`：把借用检查从编译期推迟到运行期，实现内部可变性。
- `Arc<T>`：线程安全引用计数，多线程共享时使用。

这一篇重点讲 `Box`、`Rc`、`RefCell`，最后说明它们在真实工程里的边界。

## 一、机制心智：智能指针靠 Deref 和 Drop 工作

智能指针之所以像引用，是因为它们实现了 `Deref`。离开作用域时，它们通过 `Drop` 释放或更新资源。

```rust
use std::ops::Deref;

struct MyBox<T>(T);

impl<T> MyBox<T> {
    fn new(value: T) -> Self {
        MyBox(value)
    }
}

impl<T> Deref for MyBox<T> {
    type Target = T;

    fn deref(&self) -> &Self::Target {
        &self.0
    }
}

fn main() {
    let x = MyBox::new(String::from("hello"));
    println!("{}", x.len());
}
```

`x.len()` 能工作，是因为 Rust 可以通过 `Deref` 自动把 `MyBox<String>` 当成 `&String` 使用。

你不需要经常自己实现智能指针，但理解 `Deref` 和 `Drop` 能帮你看懂标准库类型的行为。

## 二、Box<T>：堆分配和递归类型

`Box<T>` 是最简单的智能指针：把一个值放到堆上，栈上保存指向它的指针。它仍然遵守单一所有权。

```rust
fn main() {
    let value = Box::new(42);
    println!("{}", value);
}
```

单独为了“放到堆上”使用 `Box` 并不常见。它最典型的用途是让递归类型拥有确定大小。

```rust
#[derive(Debug)]
enum List {
    Cons(i32, Box<List>),
    Nil,
}

fn main() {
    use List::{Cons, Nil};

    let list = Cons(1, Box::new(Cons(2, Box::new(Cons(3, Box::new(Nil))))));
    println!("{list:?}");
}
```

如果写成 `Cons(i32, List)`，编译器无法确定 `List` 的大小，因为它会无限嵌套。`Box<List>` 是一个指针，大小固定，因此可以放进枚举里。

真实场景：

- 链表、树、表达式 AST 等递归结构。
- 大对象移动成本高，希望移动指针而不是移动整个值。
- Trait 对象：`Box<dyn Trait>`。

```rust
trait Command {
    fn run(&self);
}

struct PrintCommand {
    message: String,
}

impl Command for PrintCommand {
    fn run(&self) {
        println!("{}", self.message);
    }
}

fn main() {
    let command: Box<dyn Command> = Box::new(PrintCommand {
        message: String::from("deploy"),
    });

    command.run();
}
```

## 三、Rc<T>：单线程共享所有权

有些数据天然会被多个地方共享：配置、树节点的子结构、UI 组件模型、图结构中的公共节点。`Rc<T>` 通过引用计数允许多个所有者共享同一份数据。

```rust
use std::rc::Rc;

fn main() {
    let config = Rc::new(String::from("debug=true"));

    let a = Rc::clone(&config);
    let b = Rc::clone(&config);

    println!("config={}", a);
    println!("strong_count={}", Rc::strong_count(&config));

    drop(b);
    println!("strong_count={}", Rc::strong_count(&config));
}
```

`Rc::clone` 只增加计数，不深拷贝内部数据。为了表达这个语义，工程里通常写 `Rc::clone(&value)`，而不是 `value.clone()`。

`Rc<T>` 只提供共享不可变访问。如果你需要修改内部数据，要结合 `RefCell<T>`，或者重新设计所有权。

注意：`Rc<T>` 不是线程安全的。多线程用 `Arc<T>`。

## 四、RefCell<T>：内部可变性

`RefCell<T>` 允许你在外部只有不可变引用时，修改内部值。它把借用规则从编译期移动到运行期检查。

```rust
use std::cell::RefCell;

fn main() {
    let counter = RefCell::new(0);

    *counter.borrow_mut() += 1;
    *counter.borrow_mut() += 1;

    println!("counter={}", counter.borrow());
}
```

如果运行时违反“多个不可变借用或一个可变借用”的规则，程序会 `panic`。

```rust
use std::cell::RefCell;

fn main() {
    let value = RefCell::new(10);

    let first = value.borrow();
    println!("first={first}");

    drop(first);

    let mut second = value.borrow_mut();
    *second += 5;

    println!("second={second}");
}
```

如果不 `drop(first)`，在不可变借用仍存在时调用 `borrow_mut()` 会运行时崩溃。

真实场景：

- 测试 double/mock 需要在 `&self` 方法里记录调用次数。
- 单线程 UI 状态由多个组件共享修改。
- 图或树节点需要从多个边访问并修改。

## 五、Rc<RefCell<T>>：单线程共享可变状态

`Rc<T>` 解决多个所有者，`RefCell<T>` 解决内部可变性。组合起来就是单线程共享可变状态。

```rust
use std::cell::RefCell;
use std::rc::Rc;

#[derive(Debug)]
struct Node {
    name: String,
    children: Vec<Rc<RefCell<Node>>>,
}

impl Node {
    fn new(name: &str) -> Rc<RefCell<Node>> {
        Rc::new(RefCell::new(Node {
            name: name.to_string(),
            children: Vec::new(),
        }))
    }
}

fn main() {
    let root = Node::new("root");
    let child = Node::new("child");

    root.borrow_mut().children.push(Rc::clone(&child));
    child.borrow_mut().name.push_str("-renamed");

    println!("{root:#?}");
}
```

这个模式在 DOM 树、解释器对象图、GUI 状态树里很常见。但它也会降低代码的静态可验证性，因为借用错误变成运行时错误。

## 六、Weak<T>：避免引用循环

`Rc` 的计数归零才会释放内存。如果两个节点互相持有 `Rc`，计数永远不会归零，造成内存泄漏。解决办法是把非拥有关系写成 `Weak<T>`。

```rust
use std::cell::RefCell;
use std::rc::{Rc, Weak};

#[derive(Debug)]
struct Node {
    name: String,
    parent: RefCell<Weak<Node>>,
    children: RefCell<Vec<Rc<Node>>>,
}

fn main() {
    let leaf = Rc::new(Node {
        name: String::from("leaf"),
        parent: RefCell::new(Weak::new()),
        children: RefCell::new(vec![]),
    });

    let branch = Rc::new(Node {
        name: String::from("branch"),
        parent: RefCell::new(Weak::new()),
        children: RefCell::new(vec![Rc::clone(&leaf)]),
    });

    *leaf.parent.borrow_mut() = Rc::downgrade(&branch);

    let parent = leaf.parent.borrow().upgrade();
    println!("leaf parent exists: {}", parent.is_some());
    println!("branch children: {}", branch.children.borrow().len());
    println!("leaf name: {}", leaf.name);
}
```

父节点拥有子节点，子节点只弱引用父节点。这才符合所有权关系：父节点决定子节点生命周期，子节点不应该让父节点活着。

## 七、Arc<T>：多线程共享

多线程共享所有权时使用 `Arc<T>`，它的引用计数操作是原子的。

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let config = Arc::new(String::from("production"));
    let mut handles = Vec::new();

    for id in 0..3 {
        let config = Arc::clone(&config);
        handles.push(thread::spawn(move || {
            println!("worker {id}: {config}");
        }));
    }

    for handle in handles {
        handle.join().expect("worker should finish");
    }
}
```

如果还要跨线程修改数据，常见组合是 `Arc<Mutex<T>>`，而不是 `Rc<RefCell<T>>`。

## 八、常见错误与修正

### 1. 用 Rc 跨线程

```rust
use std::sync::Arc;
use std::thread;

fn main() {
    let value = Arc::new(String::from("shared"));
    let cloned = Arc::clone(&value);

    let handle = thread::spawn(move || {
        println!("{cloned}");
    });

    handle.join().expect("thread should finish");
}
```

`Rc<T>` 不实现 `Send`，不能在线程间移动。跨线程用 `Arc<T>`。

### 2. RefCell 运行时重复可变借用

```rust
use std::cell::RefCell;

fn main() {
    let value = RefCell::new(vec![1, 2, 3]);

    {
        let mut data = value.borrow_mut();
        data.push(4);
    }

    {
        let mut data = value.borrow_mut();
        data.push(5);
    }

    println!("{:?}", value.borrow());
}
```

把 `borrow_mut()` 的作用域压短，避免同时存在多个可变借用。

### 3. 为了绕开借用检查滥用 Rc<RefCell<T>>

```rust
#[derive(Debug)]
struct Counter {
    value: usize,
}

impl Counter {
    fn inc(&mut self) {
        self.value += 1;
    }
}

fn main() {
    let mut counter = Counter { value: 0 };
    counter.inc();
    counter.inc();
    println!("{counter:?}");
}
```

如果普通 `&mut self` 就能解决，不要引入 `RefCell`。智能指针应该表达真实所有权需求，而不是绕过设计问题。

## 九、工程使用边界

`Box<T>` 适合：

- 递归类型。
- trait 对象。
- 大对象移动时只移动指针。
- 明确单一所有权但需要堆分配。

`Rc<T>` 适合：

- 单线程多所有者共享只读数据。
- 图、树、UI 模型中的共享节点。
- 避免深拷贝大型不可变数据。

`RefCell<T>` 适合：

- 编译器无法证明安全，但你能保证运行时借用不冲突。
- `&self` 方法内部需要记录状态，比如 mock、缓存、计数。
- 单线程共享状态的小范围封装。

`Rc<RefCell<T>>` 适合：

- 单线程复杂对象图。
- GUI、解释器、DOM 这类共享可变节点。
- 原型阶段快速表达关系。

谨慎使用：

- `Rc<RefCell<T>>` 会让错误从编译期转到运行期。
- 引用循环需要 `Weak` 处理，否则会泄漏。
- 多线程不要用 `Rc`/`RefCell`，对应选择通常是 `Arc`/`Mutex`/`RwLock`。
- 能用普通所有权和借用表达的，就不要上智能指针。

## 十、结尾总结

智能指针不是“高级语法”，而是所有权模型的补充工具。`Box` 解决堆分配和递归大小问题，`Rc` 解决单线程共享所有权，`RefCell` 解决运行时内部可变性，`Weak` 解决引用循环，`Arc` 解决跨线程共享。真实工程里，先用普通所有权建模；只有当数据结构确实需要共享、递归或内部可变时，再引入对应智能指针。
