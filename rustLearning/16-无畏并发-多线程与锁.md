# 16-无畏并发：多线程与锁

> 一句话导读：Rust 不是让并发变得没有复杂度，而是把“数据竞争、悬垂引用、错误共享所有权”这些高危错误提前拦在编译期。

多线程编程最麻烦的地方从来不是“怎么开线程”，而是“多个线程同时访问同一份数据时，谁能改、谁该等、谁负责释放”。在 C/C++ 里，这些约束多数靠纪律；在 Java、Go 里，运行时提供了工具，但仍然允许你写出数据竞争。Rust 的特别之处在于，它把所有权、借用、生命周期和 `Send` / `Sync` trait 组合起来，让很多并发错误无法通过编译。

这就是 Rust 社区常说的 **Fearless Concurrency**。它不等于不用思考锁、不可能死锁，也不等于所有并发代码都自动高性能。它的意思是：只要代码通过类型系统，最危险的内存安全问题基本已经被排除。

## 一、并发心智模型：线程、所有权与共享状态

可以把 Rust 多线程想成三层：

```text
业务任务
  |
  | thread::spawn 创建 OS 线程
  v
所有权边界
  |
  | move 把数据交给线程，Arc 让多个线程共享所有权
  v
同步原语
  |
  | Mutex/RwLock/Condvar/Atomic 控制读写时机
  v
真实内存
```

关键规则有三条：

1. 线程可能比创建它的作用域活得更久，所以传入线程的数据通常要满足 `'static`。
2. 一个值要能跨线程移动，类型必须实现 `Send`。
3. 一个值要能被多个线程共享引用，类型必须实现 `Sync`。

你平时不需要手写这两个 trait，但一定要理解它们的含义。`String`、`Vec<T>`、`Mutex<T>` 通常可以跨线程使用；`Rc<T>`、`RefCell<T>` 不能直接跨线程使用，因为它们内部计数或借用检查不是线程安全的。

## 二、创建线程与 `move` 闭包

`std::thread::spawn` 会创建一个操作系统线程。返回的 `JoinHandle` 可以用来等待线程结束，并接收线程返回值。

```rust
use std::thread;
use std::time::Duration;

fn main() {
    let handle = thread::spawn(|| {
        for i in 1..=3 {
            println!("spawned thread: {i}");
            thread::sleep(Duration::from_millis(10));
        }
        42
    });

    for i in 1..=2 {
        println!("main thread: {i}");
        thread::sleep(Duration::from_millis(10));
    }

    let result = handle.join().expect("child thread panicked");
    println!("child returned {result}");
}
```

如果子线程要使用当前作用域里的变量，通常要加 `move`：

```rust
use std::thread;

fn main() {
    let names = vec!["alice".to_string(), "bob".to_string()];

    let handle = thread::spawn(move || {
        for name in names {
            println!("hello {name}");
        }
    });

    handle.join().unwrap();
}
```

`move` 的意思不是“复制数据”，而是把闭包捕获到的值按所有权规则移动进去。上面的 `names` 移入子线程后，主线程不能再使用它。这样编译器可以保证：子线程读写它时，不会有另一个线程同时拿着同一个 `Vec` 做可变操作。

## 三、`thread::scope`：借用局部变量的安全方式

有时你不想把数据移动进线程，只想让几个线程临时借用当前栈上的数据。普通 `thread::spawn` 做不到，因为子线程可能逃出当前作用域。

标准库提供了 `thread::scope`，它保证作用域结束前所有线程都已经结束，因此允许安全借用局部变量。

```rust
use std::thread;

fn main() {
    let numbers = vec![10, 20, 30, 40];

    thread::scope(|scope| {
        scope.spawn(|| {
            println!("left = {}", numbers[0]);
        });

        scope.spawn(|| {
            println!("right = {}", numbers[3]);
        });
    });

    println!("numbers is still usable: {:?}", numbers);
}
```

这在批量读取大数组、并行计算分片时很实用。注意：共享不可变借用没问题；如果多个线程要改同一份数据，仍然需要锁、原子类型，或者把数据切成互不重叠的可变分片。

## 四、共享可变状态：`Arc<Mutex<T>>`

`Mutex<T>` 负责“同一时间只允许一个线程访问内部数据”。`Arc<T>` 负责“多个线程共同拥有同一份值”。多线程共享可变状态的经典组合就是：

```text
Arc<Mutex<T>>
```

完整可编译示例：

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0_u64));
    let mut handles = Vec::new();

    for _ in 0..10 {
        let counter = Arc::clone(&counter);

        let handle = thread::spawn(move || {
            for _ in 0..1000 {
                let mut guard = counter.lock().expect("mutex poisoned");
                *guard += 1;
            }
        });

        handles.push(handle);
    }

    for handle in handles {
        handle.join().expect("worker panicked");
    }

    let result = *counter.lock().expect("mutex poisoned");
    println!("result = {result}");
}
```

这里有几个关键点：

- `Arc::clone(&counter)` 只增加引用计数，不复制 `Mutex` 里的数据。
- `lock()` 返回的是 `MutexGuard`，它实现了 `DerefMut`，所以可以像 `&mut T` 一样修改内部值。
- `MutexGuard` 离开作用域会自动释放锁，不需要手动 `unlock`。
- `join()` 一定要处理，否则主线程结束时，子线程可能还没干完。

## 五、锁作用域要尽量短

锁不是拿到就一直持有。持锁时间越长，其他线程等待越久，吞吐越差，死锁风险也越高。

下面这个写法会在打印时也持有锁：

```rust
use std::sync::{Arc, Mutex};

fn slow_log(counter: Arc<Mutex<u64>>) {
    let mut guard = counter.lock().unwrap();
    *guard += 1;
    println!("counter = {}", *guard);
}
```

更好的写法是先把需要的数据拷出来，让锁尽快释放：

```rust
use std::sync::{Arc, Mutex};

fn fast_log(counter: Arc<Mutex<u64>>) {
    let current = {
        let mut guard = counter.lock().unwrap();
        *guard += 1;
        *guard
    };

    println!("counter = {current}");
}
```

生产代码里尤其要避免“持锁做 I/O”：打印日志、写文件、访问网络、数据库查询都可能慢到不可控。

## 六、`RwLock` 与原子类型

如果读多写少，可以用 `RwLock<T>`。它允许多个读者同时进入，但写者需要独占。

```rust
use std::sync::{Arc, RwLock};
use std::thread;

fn main() {
    let config = Arc::new(RwLock::new(String::from("v1")));

    let reader = {
        let config = Arc::clone(&config);
        thread::spawn(move || {
            let value = config.read().unwrap();
            println!("read config: {value}");
        })
    };

    {
        let mut value = config.write().unwrap();
        *value = String::from("v2");
    }

    reader.join().unwrap();
}
```

如果只是计数器、标志位、简单状态，可以考虑 `AtomicUsize`、`AtomicBool` 等原子类型，它们比锁更轻。

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
use std::sync::Arc;
use std::thread;

fn main() {
    let counter = Arc::new(AtomicUsize::new(0));
    let mut handles = Vec::new();

    for _ in 0..4 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            for _ in 0..1000 {
                counter.fetch_add(1, Ordering::Relaxed);
            }
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("{}", counter.load(Ordering::Relaxed));
}
```

`Ordering::Relaxed` 只保证这个原子变量自身的读写是原子的，不保证和其他内存操作的顺序关系。做计数器通常够用；做跨线程协议、发布配置、无锁结构时必须认真理解 `Acquire` / `Release` / `SeqCst`。

## 七、常见错误与修正

### 错误 1：在线程里借用局部变量

```rust
use std::thread;

fn main() {
    let text = String::from("hello");

    // 编译失败：闭包可能比 text 活得更久
    // thread::spawn(|| println!("{text}"));
}
```

修正方式：把所有权移入线程，或者使用 `thread::scope`。

```rust
use std::thread;

fn main() {
    let text = String::from("hello");
    thread::spawn(move || println!("{text}")).join().unwrap();
}
```

### 错误 2：用 `Rc<RefCell<T>>` 跨线程

```rust
use std::cell::RefCell;
use std::rc::Rc;
use std::thread;

fn main() {
    let value = Rc::new(RefCell::new(0));

    // 编译失败：Rc 和 RefCell 都不是线程安全的
    // thread::spawn(move || *value.borrow_mut() += 1);
}
```

修正方式：跨线程用 `Arc<Mutex<T>>`。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let value = Arc::new(Mutex::new(0));
    let cloned = Arc::clone(&value);

    thread::spawn(move || {
        *cloned.lock().unwrap() += 1;
    })
    .join()
    .unwrap();

    println!("{}", *value.lock().unwrap());
}
```

### 错误 3：忘记处理锁中毒

如果线程持有 `Mutex` 时 panic，锁会进入 poisoned 状态。之后 `lock()` 返回 `Err`，提醒你内部数据可能处于不一致状态。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));
    let cloned = Arc::clone(&data);

    let _ = thread::spawn(move || {
        let mut guard = cloned.lock().unwrap();
        guard.push(4);
        panic!("boom");
    })
    .join();

    match data.lock() {
        Ok(guard) => println!("data = {:?}", *guard),
        Err(poisoned) => {
            let guard = poisoned.into_inner();
            println!("recover after poison: {:?}", *guard);
        }
    }
}
```

业务系统里不要一律 `unwrap()`。对计数器这类可恢复状态，可以记录日志并 `into_inner()`；对复杂事务状态，应该让上层重建对象或直接失败。

### 错误 4：双锁顺序不一致导致死锁

Rust 防数据竞争，但不自动防死锁。

```rust
// 线程 A: 先锁 user，再锁 order
// 线程 B: 先锁 order，再锁 user
// 两边都可能互相等待。
```

修正方式：

- 给锁定义全局顺序，所有代码都按同一顺序获取。
- 缩短锁作用域，不要嵌套锁。
- 用 channel 把“共享状态”改成“单线程拥有状态，其他线程发消息”。
- 复杂场景用 `parking_lot`、`dashmap`、任务队列等更合适的工具。

## 八、工程取舍：什么时候用线程和锁

适合用线程和锁的场景：

- CPU 密集型任务，需要真实多核并行。
- 需要调用同步阻塞库，而且无法改成 async。
- 状态较小、访问模式简单，比如计数器、缓存索引、配置快照。
- 程序是 CLI、批处理工具，线程数量可控。

不适合直接堆线程和锁的场景：

- 高并发网络服务，每个请求一个线程会带来巨大内存和调度成本。
- 业务状态关系复杂，大量锁嵌套会增加死锁风险。
- 任务之间更像流水线或事件流，此时 channel 更清晰。
- 持锁期间需要 `.await`。异步代码里要非常谨慎，通常使用 `tokio::sync` 或重新设计状态所有权。

生产经验是：先让数据所有权清晰，再决定同步方式。不要先写一堆 `Arc<Mutex<_>>`，再试图从混乱的共享状态里恢复结构。

## 九、小结

Rust 的多线程模型围绕所有权展开：能移动到线程里的值必须 `Send`，能被共享引用的值必须 `Sync`，需要共享可变状态时用 `Arc<Mutex<T>>`、`RwLock<T>` 或原子类型表达同步规则。

真正写生产代码时，要记住三件事：锁作用域尽量短；锁中毒和线程 panic 要有处理策略；Rust 保证内存安全，但不会替你保证业务不会死锁、不会饥饿、不会过度争用。
