# 13-生命周期（Lifetimes）

> "不用怕，所有的引用都有生命周期。编译器只是在自己算不清楚引用的作用域时，才强制要求你像标记路标一样手工标注一下。"

生命周期（Lifetime）是 Rust 最劝退新手的特性。但只要你记住它的**唯一目的**，你就能战胜它：
**避免悬垂引用（Dangling References）。** 即：确保任何引用指向的数据在其生命周期内都是有效的。

## 一、借用检查器 (Borrow Checker) 

看看这段**报错**的代码：
```rust
{
    let r;                // ---------+-- 'a (r 的生命周期)
    {                     //          |
        let x = 5;        // -+-- 'b  |
        r = &x;           //  |       |
    }                     // -+       | (x 在这里被销毁了！)
                          //          |
    // println!("r: {}", r); //       | (但这里居然还想用 r 指向的 x)
}                         // ---------+
```
Rust 编译器内部有个“借用检查器”，它通过对比生命周期 `'a` 和 `'b` 的长短，发现被引用的变量 `x` 的寿命比引用 `r` 短。所以编译报错，保护了你。

## 二、函数签名中的生命周期标注

大多数时候，编译器能自动推断出引用的生命周期。但如果一个函数接收两个引用，返回一个引用，编译器就懵了：返回的引用，它的存活时间到底取决于第一个参数，还是第二个参数？

```rust
// ❌ 报错：expected named lifetime parameter
// fn longest(x: &str, y: &str) -> &str {
//     if x.len() > y.len() { x } else { y }
// }
```
我们必须手动给引用加上**生命周期标记**。生命周期的语法用撇号开头，比如 `'a`。

```rust
// ✅ 正确写法
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```
**这个标记不是用来改变变量存活时间的！** 它的意思是：这个函数返回的引用的生命周期，**与参数 `x` 和 `y` 中生命周期较短的那一个**保持一致。
这样借用检查器就知道，你在外面接收返回值的那个变量，它的使用范围不能超过传入的 `x` 和 `y` 的共同存活范围。

## 三、结构体中的生命周期

只要结构体里面存放了**引用**，而不是有所有权的数据类型（如 `String`, `i32`），就必须为结构体加上生命周期标注。

```rust
// 意思是：这个结构体实例的存活时间，绝对不能超过它内部引用的那个字符串切片的存活时间。
struct ImportantExcerpt<'a> {
    part: &'a str, 
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().expect("Could not find a '.'");
    let i = ImportantExcerpt {
        part: first_sentence,
    };
}
```

## 四、静态生命周期 `'static`

`'static` 是一个特殊的生命周期，表示该引用能够存活于整个程序的运行期间。
所有的字符串字面量都拥有 `'static` 生命周期。

```rust
let s: &'static str = "I have a static lifetime.";
```
不要遇到编译不通过的问题就疯狂加上 `'static`。这通常说明你的架构设计有问题（比如把本该在堆上管理所有权的数据，强行写成了全局引用）。
