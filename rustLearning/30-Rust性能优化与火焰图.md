# 30-Rust 性能优化与火焰图

> "写出来的 Rust 代码已经够快了，但如果不小心用了太多的 `.clone()` 或者乱用锁，还是会拖慢程序。我们来看看如何科学地优化。"

不要盲目优化，优化前第一件事是：**性能剖析（Profiling）**。

## 一、发布模式编译 (Release Profile)

在你开始抱怨 Rust 程序慢之前，请确认你是否加了 `--release`！
```bash
cargo run --release
```
Debug 模式包含大量调试符号、未优化的计算，还会检查溢出。**Release 模式下，速度通常有 10 到 100 倍的提升！**

你还可以通过配置 `Cargo.toml` 进一步压榨性能：
```toml
[profile.release]
opt-level = 3        # 最大优化（默认就是 3）
lto = true           # 开启 Link Time Optimization (链接时优化)，会显著变快但编译更慢
codegen-units = 1    # 牺牲并行编译时间，让 LLVM 更好地分析整个代码库并优化
```

## 二、定位瓶颈：生成火焰图 (Flamegraph)

火焰图能直观地看出 CPU 的时间都花在了哪个函数上。

安装工具：
```bash
cargo install flamegraph
```

运行项目并生成图表：
```bash
cargo flamegraph
```
跑完后，当前目录下会生成一个 `flamegraph.svg`，用浏览器打开。
- **宽度**：代表占用 CPU 的时间长短。
- **高度**：代表函数调用栈。
**找瓶颈的方法：找最顶层、最长的那块“平顶山”！那就是性能卡点。**

## 三、常见性能陷阱与优化套路

通过火焰图定位到问题函数后，常见的优化手段有：

1. **滥用 `.clone()`**：
   - 症状：火焰图里出现大量的 `memcpy` 或内存分配。
   - 解法：考虑改用引用（借用），或者如果字符串只读，改用 `Rc<str>` 或 `Arc<str>`。
2. **频繁的小块内存分配**：
   - 症状：火焰图里全是 `malloc` 和 `free`。比如在一个循环里不断地 `String::new()` 或 `Vec::new()`。
   - 解法：在循环外使用 `String::with_capacity(size)` 预分配内存，循环内用 `.clear()` 重复利用。
3. **锁的争用 (Lock Contention)**：
   - 症状：多线程程序中，CPU 利用率上不去。
   - 解法：缩小锁的临界区（只把必要的操作放进 `lock()` 里面，拿到数据赶紧 `drop` 锁）；或者改用 `RwLock`（读写锁）；实在不行改用无锁数据结构（如 `crossbeam` 的队列）或消息传递。
4. **哈希碰撞和计算开销**：
   - Rust 默认的 `HashMap` 为了防范 DOS 攻击，用的是安全的 SipHash，速度相对较慢。
   - 解法：如果你完全信任你的数据源，可以换成 `rustc-hash` 或 `ahash` 这种极致追求速度的哈希算法。
