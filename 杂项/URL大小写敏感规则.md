# URL 大小写敏感规则

> 一份「随时回看」的速查:URL 不是整体统一一套规则,而是**逐段**各有各的脾气。
> 核心心智:**域名以左不区分,路径以右大多区分——分界点在第一个 `/`。**

---

## 一、一张图看清分界

```text
 https :// Example.COM : 443 / Path/To/Page ? Name=Tom # Section
 └─┬─┘    └────┬─────┘ └┬┘ └─────┬───────┘ └───┬────┘ └──┬───┘
 scheme      host     port    path          query     fragment
不区分      不区分    —      区分           区分        区分
└──────── 不区分大小写 ────────┘└──────────── 区分大小写 ────────────┘
                              ↑
                        分界:第一个 "/"
```

**记忆口诀:`://` 到第一个 `/` 之间(协议+域名)不区分;之后(路径+参数+锚点)都区分。**

---

## 二、逐段规则

| 部分 | 示例 | 区分大小写 | 原因 |
|------|------|:---:|------|
| scheme 协议 | `HTTP://` = `http://` | ❌ 不区分 | RFC 3986 规定按小写归一 |
| host 域名 | `Example.COM` = `example.com` | ❌ 不区分 | DNS 解析不区分大小写 |
| port 端口 | `:443` | — | 纯数字,无大小写 |
| path 路径 | `/Page` ≠ `/page` | ✅ 区分 | 由服务器/文件系统决定 |
| query 查询串 | `?Name=Tom` ≠ `?name=tom` | ✅ 区分 | 参数名与值都由后端代码解析 |
| fragment 锚点 | `#Top` ≠ `#top` | ✅ 区分 | 由前端/浏览器按字符匹配 |
| userinfo 用户名密码 | — | ✅ 区分 | — |
| 百分号编码 | `%2F` = `%2f` | ❌ 不区分 | 十六进制位不区分大小写 |

---

## 三、实际会踩的坑

| 场景 | 表现 |
|------|------|
| Linux 服务器 | 文件系统**大小写敏感**,`/Img/a.png` 与 `/img/a.png` 是两个文件 |
| Windows / macOS 服务器 | 文件系统默认**不敏感**,本地测试「看起来不区分」,上线 Linux 就 404 |
| 后端路由框架 | Spring / Express / Nginx `location` 默认区分,可配置忽略 |
| CDN / 缓存 | 按 URL 当 key,大小写不同 → 命中两条独立缓存,浪费 + 不一致 |
| 短链 / 二维码 | 用户手输、OCR、口头传播极易大小写错乱 → 打不开 |

---

## 四、实践建议

1. **路径一律小写**:统一用 `kebab-case`(短横线),不用驼峰,杜绝大小写歧义。
2. **生成型 URL(短链 / 到店码 / 分享链接)只用小写**,必要时只用大小写不混淆的字符集(去掉 `O/0`、`l/1`、`I` 等)。
3. **服务端做大小写归一**:对域名、协议强制小写;路径若想容错,统一在网关层 301 重定向到小写版本,而不是两份都返回 200(避免重复内容、缓存分裂)。
4. **查询参数名约定固定大小写**,前后端对齐一套命名规范,别依赖框架的「容错」。

---

> 一句话总结:**域名随便大小写都行,`/` 之后的部分请当作严格区分——能小写就全小写。**
