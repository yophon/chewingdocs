# 文件与音视频分发原理

> 直播是「实时生产、实时消费」,拼的是延迟;文件 / 音视频点播是「内容已存在」,拼的是**缓存命中率、带宽成本、首屏与拖拽体验**。本篇覆盖:静态文件分发、点播(VOD)、大文件分发、防盗链/DRM、CDN 成本调优,以及生产项目的资产分发选型。
>
> 配套阅读:`杂项/CDN直播原理与低延迟协议全景.md`(直播侧)。

---

## 一、静态文件分发(图片 / JS / 安装包 / 文档)

CDN 最原始、最擅长的活——内容是**不变的文件**,天然适配「缓存收敛」。

### 核心机制:缓存命中

```text
用户请求 -> 边缘节点
   ├─ 命中(HIT)   -> 直接返回,极快、不耗源站
   └─ 未命中(MISS)-> 回源拉取 -> 缓存到边缘 -> 返回
```

衡量 CDN 好坏第一指标就是**命中率**:越高 → 源站压力越小、用户越快、带宽成本越低。

### 决定缓存行为的 HTTP 头

| 头 | 作用 |
|----|------|
| `Cache-Control: max-age=` | 缓存多久,最核心 |
| `ETag` / `Last-Modified` | 内容指纹,用于「缓存还新鲜吗」的协商(304 Not Modified) |
| `Vary` | 按某请求头分别缓存(如按 `Accept-Encoding` 区分 gzip/br) |

### 关键难题:缓存更新(失效)

文件缓存到全球节点后,改了源站怎么让全网更新?两种思路:

1. **主动刷新 / 预热**:发版后调 CDN 的 `purge`(刷新)让旧缓存失效,或 `preheat`(预热)提前把新文件推到边缘。
2. **改 URL(更推荐)**:文件名带内容哈希,如 `app.3f8a2c.js`。内容一变 URL 就变,旧的永久缓存、新的自然 MISS 回源。前端构建产物普遍用这招——既能设超长 `max-age`,又永不担心更新。

> 一句话:**静态文件分发 = 把命中率做高 + 把缓存更新管对。**

## 二、音视频点播(VOD)

点播 = 内容预先存好(转码、切片),用户随时点开、可拖拽、可选清晰度。建立在静态文件分发之上,多了「媒体特有处理」。

**与直播的根本区别**:直播流式实时产生;点播**文件已存在**,可充分预处理、充分缓存。

### 1. 预处理:转码 + 切片

```text
原始 mp4 ──转码──> 1080p / 720p / 480p ──切片──> 一堆 .ts/.m4s 文件 + .m3u8 索引
```

和直播 HLS 产物一样,但点播是**离线提前切好全部切片**(直播是边播边切)。

### 2. 分发协议(与直播拉流段高度重合)

| 协议 | 点播特点 |
|------|---------|
| **HLS / DASH** | 主流。切片是文件,CDN 当普通静态文件缓存,完美适配;支持 ABR 多清晰度、随意拖拽 |
| **MP4(渐进式 / Range)** | 短视频、小文件常用;靠 HTTP Range 请求实现拖拽和边下边播 |

### 3. 点播特有关键点

- **HTTP Range 请求(拖拽核心)**:拖进度条时播放器发 `Range: bytes=5000000-`,服务器返回 `206 Partial Content`;CDN 可对大文件**分片缓存**,不必整文件命中才能服务。
- **首屏秒开**:把 `moov`(索引)放文件头部(`faststart`),否则要下到文件尾才能开播;热门内容前几个切片预热。
- **ABR 自适应**:按网速在多档清晰度切换;点播可比直播更激进地预缓冲(内容已存在)。
- **防盗链 / 鉴权**:见第四节。

## 三、大文件分发(安装包 / 游戏更新 / 镜像)

大文件(几百 MB ~ 几十 GB):单文件就能打满带宽、命中率波动大、网络一断前功尽弃。四个核心技术。

### 1. 分片(Chunking)
把大文件切成固定块(如 4MB),分发/缓存/校验都以块为单位。
- **CDN 分片缓存**:不必整文件命中,这块 HIT、那块 MISS 分别处理。
- **并行下载**:客户端多连接同时拉不同块,打满带宽。
- **按块校验**:每块带哈希,坏了只重传那一块。

### 2. 断点续传(Range 请求)
和点播拖拽同一套 `Range: bytes=` 机制:

```text
下载到 60% 断网 -> 记录已下字节 -> 重连 Range: bytes=60000000- -> 接着下
```

### 3. 增量更新(差量 / Delta)
游戏/APP 更新关键省流手段:只下「变了的部分」。
- **二进制 diff(bsdiff 等)**:老版本 + 补丁 = 新版本,补丁常只有整包几十分之一。
- **块级去重**:新老版本按块比对,只下不同块(rsync 思路)。
- 代价:服务端为「每对版本」生成补丁或维护块索引,**计算/存储换带宽**。

### 4. P2P-CDN(削峰杀手锏)
大型更新(海量用户同时下)纯靠 CDN 带宽成本爆炸,让用户互传:

```text
CDN 提供「种子 + 调度」,用户A下好的块直接传给用户B
   -> CDN 只承担一部分,大头由用户上行带宽分摊
```

- 典型:游戏更新、大型直播削峰。
- 代价:NAT 穿透、调度复杂、上行参差、隐私顾虑。通常 **CDN 兜底 + P2P 削峰**混合。

> **大文件心法**:分片是地基,Range 管断点,Delta 省更新流量,P2P 削峰值带宽。

## 四、防盗链 / DRM(分层递进)

内容是有价值的存量资产,要防三件事:**盗用带宽、盗看内容、盗录内容**。越往后越强也越贵。

### 第 0 层:Referer 防盗链(最弱,挡君子)
CDN 校验 `Referer` 是否在白名单域名。零成本,但 Referer 可伪造/可为空,只挡「别的网站直接盗链」。

### 第 1 层:URL 签名鉴权(主力,必上)
给资源 URL 加时效签名,点播标配:

```text
https://cdn.example.com/video/123.m3u8?t=1718500000&sign=a3f9c2...
                                        ↑过期时间      ↑ HMAC(path+t+密钥)
```

- 边缘节点用密钥重算签名,对不上或 `t` 过期 → 拒绝。
- 密钥只在后端和 CDN 之间,客户端拿不到,无法自造合法 URL。
- 进阶:绑定用户 IP、限制 UA。
- **HLS 细节**:m3u8 引用的每个 `.ts` 切片 URL 也要各自带签名(或统一鉴权回源),通常对**整个路径前缀签名**。

### 第 2 层:鉴权回源 / Token(动态校验)
边缘收到请求时回调业务后端问「该用户有无权限」,后端查订单/会员后放行。
- 比静态签名灵活:能实时吊销、按权限动态控制。
- 代价:多一次回源鉴权,有延迟和后端压力,通常缓存鉴权结果。

### 第 3 层:DRM(版权加密,付费内容终极防线)
前面几层只防「链接被盗」,内容本身仍是明文(抓包可得)。DRM 把**内容本身加密**,只有授权设备能解密:

```text
内容用密钥加密分发 -> 播放器向 License Server 申请密钥
   -> 校验授权 + 设备安全等级 -> 下发密钥 -> 在安全环境(TEE)内解密播放
```

| DRM | 平台 |
|-----|------|
| **Widevine** | Android / Chrome |
| **FairPlay** | iOS / Safari |
| **PlayReady** | Windows / Edge |

- 配合 **CENC(通用加密)**:一份加密内容三种 DRM 共用,不用加密三遍。
- 配合硬件层(HDCP / 安全输出)可禁截屏录屏。
- 代价:接入复杂、有授权成本、需 License 服务。**只有付费版权内容才值得上。**

### 落地分层建议

```text
普通图片/短视频      -> Referer 防盗链 + URL 签名
有价值的自有视频内容  -> URL 签名 + 鉴权回源
付费课程 / 版权影视   -> 上述 + DRM(CENC + Widevine/FairPlay)
```

> 一句话:**签名 URL 防蹭带宽,鉴权回源防越权白嫖,DRM 防内容本身被拿走。** 按内容价值逐层加。

## 五、CDN 命中率 / 带宽成本优化(实战)

CDN 计费 = 流量/带宽,**命中率直接决定成本**:命中率从 90% 提到 95%,回源流量直接砍一半。

### 1. 提高命中率

- **缓存键(Cache Key)治理(最常见的坑)**:默认缓存键含完整 URL。若 URL 带无关参数(`?from=wechat&utm=xxx`),CDN 会当**不同资源**分别缓存 → 命中率暴跌。解法:配置 CDN **忽略无关 query**,只留真正影响内容的参数。
- **合理 `Cache-Control`**:带哈希文件名的静态资源 → `max-age=31536000, immutable`;会变的资源 → 适中 max-age + `ETag` 协商(304 省流);别给能缓存的东西发 `no-cache`/`private`(常见误配,直接废掉 CDN)。
- **回源合并(请求折叠)**:热点突发时大量边缘同时 MISS 回源会冲垮源站 → 开 collapsed forwarding,同一资源并发回源合并成一次。
- **预热(Preheat)**:大促/新版本/热门内容上线**前**主动推到边缘。
- **分层缓存**:边缘 MISS 先找中心节点(L2),L2 再回源,源站只面对少量 L2 回源。

### 2. 降低带宽成本

- **压缩**:文本类 gzip/br(省 60-80%);图片用 **WebP/AVIF**(比 JPG 省 30-50%)。
- **图片/视频按需处理**:CDN 边缘**实时裁剪/压缩/转格式**,同一原图按设备场景下发不同尺寸。
- **冷热分层 + 长尾治理**:冷门大文件命中率低、是成本黑洞,单独低频存储或不放 CDN。
- **多 CDN 调度**:大流量接多家,按价格 + 质量 + 区域动态调度,压价又容灾。
- **看懂账单**:区分回源流量(贵,靠命中率压)和下行流量(走量)。

### 3. 监控指标
- 命中率(看**字节**命中率,不是请求命中率)
- 回源率 / 回源带宽
- 首字节时间(TTFB)、下载速度
- 5xx / 回源失败率

> 心法:**先治缓存键和 Cache-Control(命中率),再上压缩和图片处理(体积),最后多 CDN 调度(单价)。** 顺序别反——命中率没治好,后面都是给漏桶补水。

## 六、生产项目资产分发选型

把上面的原理收口到一个真实生产项目的常见四类资产。

### 1. 图片(量最大、最高频)
- 协议:普通 HTTP + CDN 缓存,文件天然适配,不用改造。
- **上「图片处理 CDN」**:存一张原图,URL 带参数实时出不同尺寸/格式/质量(`pic.jpg?w=400&format=webp`),列表缩略图、详情大图、不同 DPR 屏各取所需,省带宽 + 提首屏。
- 缓存:文件名带版本/哈希,`max-age` 拉满,改图换 URL。
- 缓存键:**忽略 `utm` 等营销参数**,只留 `w/h/format` 这类真正影响产物的参数。
- 格式:优先 WebP/AVIF,按 `Accept` 头自动降级 JPG。

### 2. 视频
- **点播 HLS/DASH**:转码多档码率 + 切片,ABR 自适应,文件型 CDN 直接分发。
- 防护按价值分层:免费/营销视频 → URL 签名;**付费内容** → URL 签名 + 鉴权回源(校验是否已购);版权敏感、防盗录要求高再评估 DRM(先用签名+鉴权回源跑起来,有盗版实锤再上 DRM)。
- 首屏:`faststart`(moov 前置)+ 热门切片预热。

### 3. 安装包 / 热更新包
- 安装包:大文件那套——分片 + Range 断点续传,CDN 分片缓存。
- **优先增量(Delta)更新**,只下变化部分。
- 新版本发布**预热**到边缘,避免发版瞬间海量回源打爆源站。
- 量级真的很大再考虑 P2P-CDN,初期不必。

### 4. 前端静态资源(JS/CSS)
- 构建产物**文件名带内容哈希**,`max-age` 拉满 + `immutable`。
- HTML 入口 `no-cache` 或短 max-age(它引用带哈希的资源,保证更新即时生效)。

### 选型小结

| 资产 | 协议/方式 | 缓存策略 | 防护 | 重点 |
|------|----------|---------|------|------|
| 图片 | HTTP + 图片处理 CDN | 哈希URL + 长max-age | Referer 防盗链 | 实时压缩/转格式、缓存键治理 |
| 视频 | 点播 HLS + ABR | 切片长缓存 | 签名URL + 鉴权回源(付费),DRM 待定 | 防白嫖、首屏秒开 |
| 安装包/更新 | 大文件 分片+Range | 长缓存 + 预热 | 签名URL | 增量更新省流 |
| 前端 JS/CSS | HTTP | 哈希URL immutable | — | 入口短缓存,资源永久缓存 |

> **优先级**:① 先做图片的「图片处理 CDN + 缓存键治理」(收益最大、最快见效);② 视频用「HLS + 签名URL + 鉴权回源」起步,DRM 留后手;③ 前端用哈希文件名标准方案;④ 安装包做增量更新。**别一上来就上 DRM、P2P 这些重武器,按内容价值和规模渐进加。**

## 七、与直播对照(把整个体系连起来)

| 维度 | 直播 | 点播 / 文件 |
|------|------|-----------|
| 内容产生 | 实时流式 | 预先存好的文件 |
| 核心诉求 | **低延迟** | **高命中率 + 低成本 + 拖拽体验** |
| 切片 | 边播边切 | 离线提前全部切好 |
| 缓存 | 切片生命周期极短(秒级) | 长期缓存,命中率可做到很高 |
| 协议 | RTMP/SRT 推 + HLS/FLV/WebRTC 拉 | HLS/DASH/MP4(纯拉) |
| CDN 改造 | WebRTC 等流型要重改造 | 几乎不用改,文件天然适配 |
| 关键指标 | 延迟、卡顿率 | 命中率、首屏、带宽成本 |
| 鉴权 | 推流鉴权为主 | URL 签名 / DRM 等内容保护更重 |

> **总结**:直播是「时间换不回来,所以拼延迟」;点播/文件是「内容不会变,所以拼缓存、拼成本、拼内容保护」。**HLS 切片这套技术两边通用**——很多平台直播结束后能无缝转点播回看,就是因为切片文件留下来即点播。
