# 撤销操作三剑客

git 里"撤销"一件事可以用三个不同命令:`reset` / `revert` / `restore`(和老命令 `checkout`),它们干的不是同一件事——**撤销什么、撤销到哪、保留不保留历史,各不相同**。这一篇把这三个梳清楚:**reset 改指针(可能动历史)、revert 加新 commit(不动历史)、restore 只刷文件(不动 commit)**。学会区分之后,撤销操作再也不是"试一下不行就再试"。

> 一句话先记住:**reset 是"我从未做过这件事"——动指针、可能改历史**;**revert 是"做错了,补一刀抵消"——加新 commit、保留历史**;**restore 是"工作区改回去"——只动文件,不动 commit**。已 push 的用 revert,本地没 push 的随便 reset,只想撤工作区改动用 restore。

---

## 一、`git reset`:改指针

`reset` 把 HEAD(以及当前分支指针)**移到另一个 commit**,根据模式不同,可能也改 staging / working dir。

```bash
git reset --soft   <commit>   # 只动 HEAD 和分支指针
git reset --mixed  <commit>   # 动 HEAD + staging(默认)
git reset --hard   <commit>   # 动 HEAD + staging + working dir ⚠️
```

**最常用例子**:撤销最后一次 commit。

```
A → B → C → D     ← main(HEAD)

git reset --soft HEAD^

A → B → C         ← main(HEAD)
            ↑
          D 的改动还在 staging

git reset HEAD^   # 默认 mixed

A → B → C         ← main(HEAD)
            ↑
          D 的改动还在 working,staging 也清了

git reset --hard HEAD^

A → B → C         ← main(HEAD)
            
          D 的改动彻底没(reflog 还能找)
```

| 模式 | HEAD | staging | working dir |
| --- | --- | --- | --- |
| `--soft` | 改 | 不动 | 不动 |
| `--mixed`(默认) | 改 | 清成 HEAD | 不动 |
| `--hard` | 改 | 清成 HEAD | 清成 HEAD ⚠️ |

> `--hard` 是 git 里**最危险的操作之一**——working dir 没保存的改动直接没。reflog 救得回 commit,但**救不回从未 commit 过的工作区改动**。每次 `--hard` 之前先深呼吸。

---

## 二、`git revert`:加反向 commit

`revert` **不改历史**——它新建一个 commit,内容是"目标 commit 的反向 diff"。

```
A → B → C → D     ← main

git revert C

A → B → C → D → C'    ← main
                ↑
            "撤销 C" 的新 commit
```

C' 的 diff 跟 C 完全相反。最终效果:**C 的改动被抵消**,但 C 这个 commit **依然在历史里**。

```bash
git revert <commit>                # 单个 commit
git revert <commit1> <commit2>     # 多个,顺序生成 revert commit
git revert -n <commit>             # 应用 revert 但不立刻 commit(可以累计后再 commit)
git revert --abort                 # revert 出冲突时放弃
```

**典型场景**:**已经 push 出去的 commit 出 bug**——不能 reset(会改远端历史),只能 revert。

---

## 三、`git restore`:只动文件

git 2.23 之后从 `checkout` 拆出来的命令,**只动文件,不动任何 commit / 分支**。

```bash
git restore <file>                       # 工作区改动还原成 staging 里的版本
git restore --staged <file>              # staging 区还原成 HEAD 里的版本(等于撤销 add)
git restore --source=HEAD <file>         # 工作区还原成 HEAD 版本
git restore --source=<commit> <file>     # 工作区还原成某 commit 的版本
git restore --staged --worktree <file>   # 同时还原 staging 和 working
git restore .                            # 还原所有(慎用)
```

| 想干什么 | 命令 |
| --- | --- |
| 撤销工作区改动 | `git restore <file>` |
| 撤销 add | `git restore --staged <file>` |
| 用某 commit 的内容覆盖工作区 | `git restore --source=<commit> <file>` |
| 撤销所有未 commit 改动 | `git restore .`(注意会丢工作区改动) |

老命令对照:

| 老命令 | 新命令 |
| --- | --- |
| `git checkout -- <file>` | `git restore <file>` |
| `git checkout HEAD -- <file>` | `git restore --source=HEAD <file>` |
| `git reset HEAD <file>` | `git restore --staged <file>` |

> `restore` 是 git 2.23 的设计改善——把"撤销文件"和"切分支"两件根本无关的事拆开了。**新代码里别再用 `git checkout -- file` 这种老语法**,容易跟切分支混。

## 四、`reset` vs `revert` vs `restore`:决策树

```
我要撤销什么?
│
├─ 工作区某个文件的改动                    → restore <file>
├─ staging 里的某个文件                    → restore --staged <file>
├─ 最后一次 commit,改动留下继续改         → reset --soft HEAD^
├─ 最后一次 commit,完全扔掉(本地)       → reset --hard HEAD^
├─ 已经 push 出去的某个 commit             → revert <commit>
├─ 已经 push 的多个 commit                 → revert 多个 / revert merge commit
├─ 切到某个老 commit 看看                  → switch --detach <commit> 或 checkout <commit>
└─ 把整个分支挪到某 commit 重新开始        → reset --hard <commit>(只对没 push 的)
```

## 五、撤销 push 出去的 commit:**只能 revert**

```bash
# 错误做法 ❌
git reset --hard HEAD^
git push --force            # 改写远端历史,队友哀嚎

# 正确做法 ✅
git revert HEAD
git push                    # 加新 commit,远端历史完整
```

**为什么不能 reset?** 同 08 篇 rebase 的 golden rule——改写已 push 历史会让队友本地哈希错位、CI 缓存失效、tag 指错。

> 这条规则没有例外。**已 push 的 commit 出错 = revert,不是 reset**。即使你是仓库的唯一开发者,养成习惯也好——某天加同事的时候不踩坑。

## 六、revert merge commit

merge commit 有两个 parent,revert 时需要指定保留哪一边:

```bash
git revert -m 1 <merge-commit>
```

`-m 1` 意思是:**保留 merge commit 的第 1 个 parent 那条线,撤销其他线带来的改动**。

```
       M ← merge commit, parents: P1, P2
      / \
     P1  P2
     │
   main 这边
```

`-m 1` = 当作"M 没 merge 进来",保留 main 这边、撤销 P2(被合进来)那边带的所有改动。

**典型场景**:整个 feature PR merge 进 main 后发现有问题,要整 PR 回滚:

```bash
git revert -m 1 <PR 的 merge commit>
```

一行搞定。**这是"原子 PR + 留 merge commit" 工作流的核心好处**(详见 07 / 19 篇)。

## 七、revert 之后想再合那个 feature 怎么办

revert 了 merge commit M 之后,该 feature 的所有改动被抵消。后续如果想再合 feature 进来:

**问题**:git 看到"feature 的 commit 已经在 main 历史里"(M 那次合过),所以 `merge feature` 不会带任何改动。

**解决**:**先 revert 那个 revert**(再次抵消):

```bash
git revert <revert-commit>    # 把"撤销 M 的 commit"再撤销一次,等于又把 feature 的改动加回来
```

或者,**rebase feature 到最新 main 后再合**:

```bash
git switch feature
git rebase main           # 重写成新 commit,不再被认为"已合过"
git switch main
git merge feature
```

> 这个坑很经典,Linus Torvalds 在 Linux 内核邮件列表上专门写过一篇详解。**记住一句:revert merge commit 之后,要再合那个 feature 必须用上面两种姿势之一**。

## 八、典型撤销场景速查

### 8.1 改坏了一个文件,想撤回

```bash
git restore src/a.ts
```

### 8.2 add 错了一个文件,想撤回 staging

```bash
git restore --staged src/a.ts
```

### 8.3 commit 错了 message,想改

```bash
# 还没 push:
git commit --amend -m "正确的 message"

# 已经 push:别改,让它留着;实在要改用 force-with-lease(只对私人分支)
```

### 8.4 commit 漏了一个文件

```bash
# 还没 push:
git add 漏的文件
git commit --amend --no-edit

# 已经 push:加个新 commit,不要 amend
git add 漏的文件
git commit -m "fix: add missing file"
```

### 8.5 commit 之后改主意,想撤回所有改动

```bash
git reset --hard HEAD^      # ⚠️ 工作区改动也没,reflog 救得回
```

### 8.6 想把分支退回到几小时前的状态

```bash
git reflog                  # 看 HEAD 移动史,找到对的 commit hash
git reset --hard <hash>
```

### 8.7 误删了文件

```bash
# 还没 commit:
git restore <file>

# 已经 commit 删除:
git checkout HEAD~1 -- <file>     # 用上一个 commit 的版本恢复
```

### 8.8 误删了分支

```bash
git reflog | grep <branch-name>   # 找到那分支最后指的 commit
git switch -c <branch-name> <hash>
```

## 九、`git clean`:清掉 untracked 文件

```bash
git clean -n             # dry-run,只列出会删什么(必跑)
git clean -f             # force,真删 untracked 文件
git clean -fd            # 同时删 untracked 目录
git clean -fdx           # 同时删 ignored 文件 ⚠️ 大杀器
```

**别一上来就 `-fdx`**——会把 `node_modules` / `.env` / 编辑器配置全删。**先 `-n` 看一眼再 `-f`**。

> `git clean` 不属于"撤销操作"严格意义,但每次 `reset --hard` 之后想彻底干净都需要它。

## 十、踩坑提醒

1. **`reset --hard` 之前不看清现状**——working dir 改动直接没
2. **想撤已 push commit 用 reset + force push**——破坏远端历史,队友受灾
3. **revert merge commit 不指定 `-m`**——git 报错"is a merge but no -m"
4. **revert 之后忘了"revert 那个 revert"想再合 feature**——以为 feature 不工作,实际是被 git 当成已合过
5. **`git checkout -- file` 老语法**——和切分支命令混淆,**新版用 restore**
6. **`reset HEAD^` 之后又 `reset --hard`**——以为这两步等价,实际第二步又把工作区刷了
7. **不知道 reflog**——撤错了找不回(reflog 90 天保留一切)
8. **`git clean -fdx` 一时冲动**——`.env` 和 IDE 配置一起没
