# 工作流模型

到这一篇为止你掌握了 git 的全部"机械操作"——但**怎么和团队配合**是另一个层面的问题。"我们用什么工作流?"——这是任何团队前 3 天必决的事。这一篇讲清楚四种主流工作流(GitFlow / GitHub Flow / Trunk-Based / Release Branch)、每种的适用规模、和**2026 年的工业界趋势:trunk-based 越来越主流,GitFlow 渐被淘汰**。

> 一句话先记住:**工作流的本质是"分支怎么定义角色"和"什么时候合,合到哪"**。GitFlow 严谨但繁琐;GitHub Flow 简单但少 release 概念;Trunk-Based 极简但要求强大的 CI / 特性开关;Release Branch 适合多版本并行维护的产品。**没有"最好的"工作流——只有"匹配团队规模和产品节奏的"**。

---

## 一、GitFlow(经典 / 老派)

2010 年提出,长期是行业标杆。**5 种分支**:

```
main         ──○───○───○──         ← 永远的"已发布版本"
                  │
develop      ──○─┴─○───○──         ← 集成分支,日常开发
                       │
feature/*    ──○───○───┘            ← 新功能,从 develop 分,合回 develop
                  │
release/v1   ───○─┴─○──             ← 发布候选,稳定后合 main + develop
                  │
hotfix/*     ──○──┴──               ← 线上 bug,从 main 分,合 main + develop
```

### 1.1 分支角色

| 分支 | 用途 | 来自 | 合到 |
| --- | --- | --- | --- |
| `main` | 生产代码 | release / hotfix | (终点) |
| `develop` | 集成开发 | feature / release / hotfix | (终点,持续) |
| `feature/xxx` | 新功能 | `develop` | `develop` |
| `release/x.y` | 发布准备 | `develop` | `main` + `develop` |
| `hotfix/xxx` | 线上紧急修复 | `main` | `main` + `develop` |

### 1.2 优缺点

**优点**:

- 严格区分"开发中"和"已发布"
- 多版本并行维护好
- 分支语义清晰

**缺点**:

- 5 种分支学习成本高
- 大量 merge commit 污染历史
- feature 分支寿命长,合并冲突多
- **跟现代 CI / CD 节奏不符**——CI 强调快速合主、持续交付,GitFlow 强调"先 develop 攒一批,再 release"

### 1.3 适合谁

- **2010-2018 主流**
- 季度 / 半年发布的产品
- 桌面软件 / 嵌入式 / SDK——发布周期长
- 多版本并行维护(v1.0、v2.0 都要修 bug)

> **2026 大多数 web / SaaS 团队不再用 GitFlow**——不发版而是持续部署,develop / release 这套抽象已经过度设计。

## 二、GitHub Flow(简单)

GitHub 内部用,极简:

```
main      ──○───○───○───○───○──     ← 永远可发布
            │       │
feature  ───○───○───┘                ← 任何工作都开分支,PR 合回 main
```

### 2.1 规则

1. **`main` 永远可发布**——任何时候 deploy main 都不会挂
2. **任何工作开分支**——名字描述功能(`fix-login-bug`、`refactor-auth`)
3. **PR 是合并的唯一入口**——直接 push main 禁止
4. **PR review + CI 通过**才能合
5. **合完立刻 deploy** 或允许 deploy

### 2.2 优缺点

**优点**:

- 简单到极致
- 适合持续部署
- 学习成本低

**缺点**:

- 没有 release 概念——不发版本号的产品(SaaS)适合,需要发版本号的产品需要补 tag
- 没有 hotfix 概念——所有都从 main 开,但**一旦有"上一个版本要修 bug"就没招**

### 2.3 适合谁

- **SaaS / Web 应用 / 内部工具**
- 发布频率高(每周 / 每天)
- 单版本(只维护最新版)
- 小到中等团队(2-50 人)

> GitHub 自己用、Heroku 用、Shopify 用——**只要你的产品"只有一个版本"且持续部署,GitHub Flow 通常够用**。

## 三、Trunk-Based Development(主干开发)

2020 年后大厂推崇的现代工作流。**核心理念**:

```
main      ──○─○─○─○─○─○─○─○─○──     ← 所有 commit 直接合主干
              ↑           ↑
            短分支         短分支
        (1-2 天就合)   (1-2 天就合)
```

### 3.1 规则

1. **所有人在 main 或短命分支(< 2 天)上工作**
2. **没有长寿命的 feature 分支**
3. **大功能用 feature flag 控制可见性**——代码合进去了但用户看不到,直到 flag 打开
4. **强 CI/CD 保障 main 永远绿**
5. **每天合多次**——比 GitHub Flow 更激进

### 3.2 关键技术:特性开关(feature flag)

```typescript
if (featureFlags.NEW_AUTH_FLOW) {
  return newAuthLogic();
} else {
  return oldAuthLogic();
}
```

代码合进 main 了,但 flag off → 用户看不到。等 QA 完毕 / 灰度通过 → flag on → 用户看到。

工具:LaunchDarkly、Statsig、Unleash、自家系统都行。

### 3.3 优缺点

**优点**:

- 集成最快,合并冲突最少(没长分支堆积)
- 持续交付天然支持
- 历史最干净

**缺点**:

- 要求强 CI(任何 push 都跑全套测试)
- 要求 feature flag 系统
- 团队纪律要求高——大改不能"先慢慢做完再合",必须拆小
- 中等以下团队上 feature flag 系统是负担

### 3.4 适合谁

- **Google / Meta / Netflix / Spotify 等大厂用**
- 高频部署的产品(每天 N 次)
- 有 feature flag 基建
- 团队文化能拆小任务

> 2020 后**新建团队 / 重构工作流的项目首选 trunk-based**。但需要前期投入(CI、flag 系统)——**小团队没必要硬上**,先用 GitHub Flow,长大了再迁。

## 四、Release Branch(发布分支)

适合**多版本并行维护**:

```
main          ──○───○───○───○───○──   ← 持续开发
                  │       │
release/1.0  ────○─┐──○─┐               ← v1.0 系列
                    │     │
                    ↓     ↓ 
                  v1.0.0 v1.0.1
                  
release/2.0  ─────────○───○───┐
                              ↓
                            v2.0.0
```

### 4.1 规则

- main 持续开发(类似 trunk-based 或 GitHub Flow)
- 发版时从 main 切一个 `release/x.y` 分支,只接收 bugfix
- v1.0、v2.0、v3.0 同时维护——分别是各自的 release 分支
- bug 在 main 修了,**cherry-pick 反向传播**到各 release 分支(详见 12 篇)

### 4.2 适合谁

- SDK / 库 / 框架(用户锁定多个版本)
- 桌面 / 移动 app(老版本用户多)
- 操作系统 / 数据库 / 编译器
- 任何"老版本要长期支持"的产品

> Linux 内核、Postgres、Node.js 都是这种模式——main 一直跑,但同时维护 v18 / v20 / v22 等多个 LTS 分支。

## 五、四种工作流对比

| 维度 | GitFlow | GitHub Flow | Trunk-Based | Release Branch |
| --- | --- | --- | --- | --- |
| 分支数 | 5+ | 2(main + features) | 1 + 短分支 | 2-N(main + releases) |
| 学习曲线 | 陡 | 平 | 中(需要 flag) | 中 |
| 发布频率 | 季度 / 月 | 周 / 天 | 天 / 多次 | 月 / 季度 |
| 多版本支持 | ✅ | ❌ | ❌ | ✅ |
| CI 要求 | 中 | 中 | 高 | 中 |
| 团队规模 | 不限 | 小到中 | 中到大 | 任意 |
| 现代主流度 | ↓ | ↑ | ↑↑ | 平 |

## 六、选择决策树

```
你的产品是?
│
├─ 持续部署 SaaS / Web 应用
│  ├─ 团队 < 20:GitHub Flow
│  └─ 团队 > 20 且有 flag 基建:Trunk-Based
│
├─ 多版本支持的库 / 框架 / SDK
│  → Release Branch
│
├─ 桌面 / 移动 / 嵌入式,季度发布
│  ├─ 想严谨完整:GitFlow
│  └─ 想现代轻巧:Release Branch + 现代 CI
│
└─ 不确定 / 新团队
   → 从 GitHub Flow 开始,长大再说
```

## 七、常见混合模式

### 7.1 GitHub Flow + Release Tag

不开 release 分支,直接在 main 上打 tag:

```bash
git tag -a v1.2.0 -m "..."
git push --tags
```

**适合 SaaS + 偶尔需要版本号**(给客户报告用、给依赖方做对接)。

### 7.2 Trunk-Based + Release Branch

main 是 trunk,release 分支只在发版时短期存在:

```
main         ─○─○─○─○─○─○─○─○─
                  │
release/1.0  ───○ │
                  ↓
                v1.0.0(打完 tag,release 分支基本不用了)
```

**适合中等成熟度团队**——主体 trunk-based,但发版时有个"冻结点"做最后 QA。

### 7.3 GitFlow Lite

GitFlow 的简化版:去掉 develop,只留 main + feature + release + hotfix:

```
main         ──○────○────○──
                │        │
feature/*    ──○─┘  ┌────┘
                    │
hotfix/*     ───────○────
```

**适合**对 GitFlow 熟悉但团队嫌 develop 累赘的过渡阶段。

## 八、protected branch:工作流的硬执行

不管你选什么工作流,**用 GitHub / GitLab branch protection 强制执行**:

- 禁止直接 push main / develop / release/*
- 必须 PR 合并
- PR 必须有 N 人 review approval
- PR 必须 CI 通过
- 禁止 force push
- 禁止删除分支
- 强制签名 commit(高级)

> 没 branch protection 的工作流是**靠自觉**——长期靠不住。**第一天就配上**,而不是出事故后再补。

---

## 九、CODEOWNERS:让 review 自动到位

GitHub / GitLab 支持 `.github/CODEOWNERS`:

```
# 各目录的负责人
/src/auth/        @alice @bob
/src/payment/     @carol
/docs/            @dave
*                 @team-leads
```

PR 改了 `src/auth/` → 自动 request alice 和 bob review。**比"PR 出来再 @人"靠谱**。

## 十、迁移工作流的代价

不要随便迁。每次迁工作流:

- 文档要改
- CI 要改
- 团队习惯要变
- 半年内会反复"我应该往哪合"的混乱

**工作流应该跟产品阶段匹配**:

- 0-50 人:GitHub Flow,够了
- 50-300 人 + 高频部署:迁 trunk-based,值得
- 多版本产品:Release Branch
- 稳定 GitFlow 团队:**没必要为了"现代化"而迁**——能跑就行

---

## 十一、踩坑提醒

1. **小团队上 GitFlow**——5 种分支让 3 人团队天天迷路
2. **大团队 GitHub Flow 没 feature flag**——main 经常爆,大家不敢部署
3. **trunk-based 没 CI 基建**——直接合主干 = 直接出事故
4. **Release Branch 不知道 cherry-pick**——hotfix 反向传播全靠人手 patch
5. **没 branch protection**——工作流纸面规则,实际全靠自觉
6. **混合工作流没文档化**——新人来了一脸懵
7. **频繁切工作流**——每次切都是认知成本,半年内事故频发
8. **以为工作流=用什么命令**——其实是"什么时候做什么决定"
