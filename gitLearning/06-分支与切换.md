# 分支与切换

回到 01 篇那句话——**分支只是指向某个 commit 的指针**。这一篇把这件事彻底用透:为什么新建分支秒级、为什么删分支不丢 commit、为什么 detached HEAD 不可怕、为什么 git 2.23 之后要把 `checkout` 拆成 `switch` 和 `restore`。**理解了分支的指针本质,git 协作里 80% 的"这是怎么了"都能秒解**。

> 一句话先记住:**分支是 `.git/refs/heads/` 下的一个文本文件,内容就一行——某个 commit 的 hash**。`git branch new` = 写一个新文件;`git checkout` = 改 HEAD + 刷工作区;`git branch -d` = 删那个文件,commit 一个不少留着。**分支就是这么轻**。

---

## 一、分支到底是什么

```bash
ls .git/refs/heads/
# main  feature  hotfix

cat .git/refs/heads/main
# a3f5b2c4d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3
```

41 字节(40 hash + 换行)。这就是 `main` 分支的全部物理存在。

**HEAD** 也是个文件:

```bash
cat .git/HEAD
# ref: refs/heads/main
```

意思是"我现在在 main 这个分支上"。也可以是 detached HEAD:

```bash
git checkout a3f5b2c
cat .git/HEAD
# a3f5b2c4d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3
```

直接是个 hash——**不在任何分支上,直接指向某个 commit**(下面第六节细讲)。

---

## 二、分支的基本操作

```bash
# 查看
git branch                  # 列出本地分支(* 标当前)
git branch -a               # 包括远程跟踪分支
git branch -r               # 只看远程
git branch -v               # 带最后一次 commit message
git branch --merged main    # 已合并到 main 的(可以安全删)
git branch --no-merged main # 还没合到 main 的

# 新建
git branch feature          # 基于 HEAD 建,但不切过去
git branch feature a3f5b2c  # 基于某个 commit 建
git checkout -b feature     # 建并切过去(老命令)
git switch -c feature       # 建并切过去(2.23+ 新命令)

# 切换
git checkout main           # 切到 main(老命令)
git switch main             # 切到 main(2.23+ 新命令)
git switch -                # 切回上一个分支(类似 cd -)

# 删除
git branch -d feature       # 安全删:已 merge 才删,否则报错
git branch -D feature       # 强删:不管 merge 没,直接删 ⚠️

# 改名
git branch -m newname              # 改当前分支名
git branch -m oldname newname      # 改指定分支
```

> 强删 `-D` 不会真丢 commit——commit 还在 reflog 里 90 天,可以救回(详见 14 篇)。但**习惯上不要随便强删**,留一道 merge 检查能避免 80% 的"哎呀我刚才那个分支白做了"。

## 三、`switch` / `restore` vs `checkout`

git 2.23(2019)把 `checkout` 拆成两个命令,因为 `checkout` 干了两件**完全不同的事**,初学者老搞混:

| 老命令 | 干什么 | 新命令 |
| --- | --- | --- |
| `git checkout main` | 切分支 | `git switch main` |
| `git checkout -b new` | 新建并切 | `git switch -c new` |
| `git checkout <commit>` | detached HEAD | `git switch --detach <commit>` |
| `git checkout -- file` | 撤销工作区改动 | `git restore file` |
| `git checkout HEAD -- file` | 撤销并刷成 HEAD 版本 | `git restore --source=HEAD file` |
| `git checkout commit -- file` | 用某 commit 的版本覆盖工作区 | `git restore --source=commit file` |

**强烈推荐用新命令**——语义清晰、不容易误操作。`checkout` 留作老脚本兼容。

> `git checkout <branch>` 和 `git checkout -- file` 名字一样、行为天差地别——很多人不小心打了 `git checkout file`(没 `--`),如果有个分支恰好叫 `file` 就切过去了,工作区被刷一遍。**`switch` / `restore` 一拆完全没这个歧义**。

## 四、跟踪分支(upstream / tracking)

每个本地分支可以"跟踪"一个远程分支——告诉 git "这个本地分支跟远程哪个分支对位"。

```bash
git branch -vv                          # 看每个本地分支跟踪谁
# main    a3f5b2c [origin/main] fix: ...
# feature d8e9f0a [origin/feature: ahead 2] feat: ...

git branch --set-upstream-to=origin/main main    # 设跟踪
git push -u origin feature              # 第一次 push,顺便设跟踪
```

跟踪关系给你这些便利:

```bash
git status              # 显示"领先远程 N 个 / 落后 N 个"
git pull                # 不用打 origin main,自动拉对应远程
git push                # 不用打 origin main,自动推
git log @{u}..HEAD      # @{u} = upstream
```

> 没设 upstream 的分支,`git pull` / `git push` 都要带完整参数,`status` 也不会告诉你跟远程的差距。**第一次 push 用 `-u` 是肌肉记忆**。

## 五、分支的"分叉"与 fast-forward

合并两个分支时(详见 07 篇),git 有两种情形:

**情形 A:fast-forward(可快进)**

```
A ─→ B ─→ C ─→ D
              ↑
            main
              ↑
                 ↑
              feature
```

main 和 feature 在同一条线上,feature 比 main 多 1 个 commit。这时候 `git merge feature` 不需要新建 commit,**直接把 main 指针挪到 D**——这叫 fast-forward。

**情形 B:三方合并(true merge)**

```
A ─→ B ─→ C ─→ D    ← main
        ↘
          E ─→ F    ← feature
```

main 和 feature 各自有独立的 commit。`git merge feature` 必须**新建一个 merge commit M**,M 有两个 parent(D 和 F)。

```
A ─→ B ─→ C ─→ D ─→ M    ← main
        ↘         ↗
          E ─→ F        
```

> 分支这件事的"形状"决定 merge 的行为。fast-forward 留下的是**线性历史**;三方合并留下的是**真分叉历史**——后者保留了"feature 当时是独立做的"信息。各有偏好(详见 07 / 08 篇)。

## 六、Detached HEAD 不可怕

新人见到 `You are in 'detached HEAD' state` 就慌。实际上**它就是字面意思:HEAD 不在任何分支上,直接指向一个 commit**。

```bash
git checkout a3f5b2c
# Note: switching to 'a3f5b2c'.
# You are in 'detached HEAD' state. ...
```

什么时候会进入 detached HEAD:

- `git checkout <commit-hash>` 直接看历史
- `git checkout v1.2.3` 看某个 tag
- rebase 中间(rebase 是一系列 cherry-pick,过程中处在 detached 状态)
- bisect 过程中

**为什么"危险"**:在 detached HEAD 上 commit 的话,新 commit 没有任何分支指着它——一旦 `git switch main` 切走,新 commit 就**漂浮**了(只有 reflog 找得到,90 天后被 GC)。

**怎么不丢**:在 detached HEAD 想保留改动?**先建分支**:

```bash
git switch -c rescue-branch    # 当前位置建分支,改动有家了
```

> Detached HEAD 不是 bug 也不是 git 抽风——是一种**有意设计的"看历史"模式**。理解之后,你会发现它是看 tag 内容、跑某个老版本的最快办法。**只是改动想保留要先建分支**——记住这一条就行。

## 七、分支命名约定

技术上 git 不限分支名(只要文件系统支持),但团队习惯一组前缀:

| 前缀 | 用途 |
| --- | --- |
| `main` / `master` | 主分支 |
| `develop` | 集成分支(GitFlow,详见 19 篇) |
| `feature/xxx` | 功能开发 |
| `fix/xxx` / `bugfix/xxx` | bug 修复 |
| `hotfix/xxx` | 线上紧急修复 |
| `release/v1.2.0` | 发布候选 |
| `chore/xxx` | 杂活 |
| `wip/xxx` | 还没成型的 |

**禁忌**:

- 别用空格(`feature x` 实际上是两个 token)
- 别用大写(macOS / Windows 大小写不敏感,会撞)
- 别太长(超过 30 字符 push / log 都难看)
- 别用 `/` 开头 / 结尾(git 会报错)

> `feature/`、`fix/` 这种**斜杠分组**有个意外好处:很多 git GUI 工具会按斜杠树状显示——一两年的项目里有 50+ feature 分支时,树状视图比扁平列表清爽 100 倍。

## 八、清理已合并的分支

时间长了本地积累一堆已经合掉的分支。一行清掉:

```bash
git branch --merged main | grep -v "^\*\|main$\|develop$" | xargs git branch -d
```

逐段意思:

- `--merged main`:列出已合到 main 的分支
- `grep -v "^\*"`:排除当前分支(标 `*`)
- `\|main$\|develop$`:排除 main 和 develop
- `xargs git branch -d`:对每个名字 `git branch -d`

更安全做法(只列出来,不真删):

```bash
git branch --merged main | grep -v "^\*\|main$\|develop$"
```

> 远程分支也同步剪掉:`git fetch --prune`(或者设 `fetch.prune=true` 永久生效)。**没装 prune 的本地 `git branch -r` 会显示一堆早就被远程删掉的"鬼分支"**。

## 九、查看分支历史

```bash
git log --oneline --graph --all --decorate         # 看整体拓扑
git log --oneline main..feature                    # feature 比 main 多哪些
git log --oneline --left-right main...feature      # 双方各多哪些
git show-branch                                    # 比较多分支的相对位置(老式)
```

**最实用的"我这条分支是从哪建出来的"**:

```bash
git merge-base main feature        # 两条分支的最近共同祖先 commit
git log --oneline $(git merge-base main feature)..feature   # feature 独有的 commit
```

## 十、踩坑提醒

1. **`git checkout file` 忘了 `--`**——如果恰好有个分支叫 `file` 就切过去了,工作区被刷;**新版用 `git restore file` 没这问题**
2. **Detached HEAD 上 commit 没建分支**——切走后 commit 漂浮,只能 reflog 找回
3. **本地分支不设 upstream**——每次 push / pull 都要带完整参数,`status` 看不出领先 / 落后
4. **`git branch -D` 强删后才发现还没 merge**——reflog 90 天能救,但没 reflog 习惯的人会以为代码丢了
5. **大写 / 空格分支名**——macOS / Windows 大小写不敏感,大写名字撞另一个仅大小写不同的分支
6. **`git branch -d` 用在 detached HEAD 上**——错位,要先 switch 到任意分支
7. **本地分支太多不清理**——半年后 50+ 已 merge 分支,`git branch` 出来一屏屏
