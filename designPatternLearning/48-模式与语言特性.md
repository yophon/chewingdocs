# 模式与语言特性:Java、Go、Rust、TypeScript 写法为什么不同

同一个设计模式在不同语言里长得不一样.很多人误以为"不像 Java 经典写法就不是模式",这会导致两种错误:在现代语言里硬套类层级,或者在静态语言里忽略类型边界.

设计模式不是固定代码模板,而是**某种变化管理结构在语言里的表达**.

---

## 一、问题:把 Java 写法当成唯一答案

例如策略模式,经典 Java 写法是接口加实现类:

```java
interface PricePolicy {
    BigDecimal apply(BigDecimal amount);
}

class VipPricePolicy implements PricePolicy {
    public BigDecimal apply(BigDecimal amount) {
        return amount.multiply(new BigDecimal("0.8"));
    }
}
```

这在 Java 里合理,因为接口、类、依赖注入和测试替身都很成熟.

但在 TypeScript 里,简单策略可能只是函数:

```ts
type PricePolicy = (amount: number) => number;

const vipPolicy: PricePolicy = (amount) => amount * 0.8;
```

它们解决的是同一个变化点:价格算法可替换.表达方式不同,不是模式消失了.

---

## 二、Java:类、接口和框架容器很强

Java 适合用显式接口表达稳定边界:

```java
interface PaymentGateway {
    PayResult pay(PayCommand command);
}

class StripePaymentGateway implements PaymentGateway {
    public PayResult pay(PayCommand command) {
        return new PayResult("ok");
    }
}
```

常见形态:

- 工厂和建造者常用于复杂对象创建.
- 策略、模板方法、责任链常和 Spring Bean 结合.
- 代理模式常通过 AOP、动态代理实现.
- Repository、Service、Controller 分层明显.

代价是类和接口容易膨胀.Java 项目里尤其要警惕"每个类都有接口",但接口没有真实替换价值.

---

## 三、Go:小接口、组合和显式依赖

Go 没有传统继承,接口是隐式实现.它更鼓励在消费方定义小接口:

```go
type PaymentGateway interface {
    Pay(ctx context.Context, cmd PayCommand) (PayResult, error)
}

type OrderService struct {
    payment PaymentGateway
}
```

Go 里的模式常常更扁:

- 适配器是普通 struct 包一层第三方 SDK.
- 装饰器是函数或 struct 包装接口.
- 责任链常见于 HTTP middleware.
- 工厂常是 `NewXxx(...) (*Xxx, error)`.

示例:

```go
func Logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        log.Println(r.URL.Path)
        next.ServeHTTP(w, r)
    })
}
```

这就是装饰器 / 责任链思想,但没有复杂类图.

---

## 四、Rust:所有权、trait 和类型状态

Rust 的设计会把很多运行时问题提前到编译期.

策略可以用 trait:

```rust
trait PricePolicy {
    fn apply(&self, amount: i64) -> i64;
}

struct VipPolicy;

impl PricePolicy for VipPolicy {
    fn apply(&self, amount: i64) -> i64 {
        amount * 80 / 100
    }
}
```

状态模式还可以用类型表达:

```rust
struct DraftOrder;
struct PaidOrder;

impl DraftOrder {
    fn pay(self) -> PaidOrder {
        PaidOrder
    }
}
```

这里 `DraftOrder` 被消费掉,支付后只能得到 `PaidOrder`.这比运行时检查 `status` 更强,但也更重,适合状态转移稳定且安全要求高的核心逻辑.

Rust 的代价是抽象边界需要更早想清楚,否则生命周期和泛型会让代码变复杂.

---

## 五、TypeScript:结构类型、函数和框架运行时

TypeScript 的接口是结构类型,很多模式可以轻量表达:

```ts
type PaymentGateway = {
  pay(command: PayCommand): Promise<PayResult>;
};

function createOrderService(payment: PaymentGateway) {
  return {
    async payOrder(orderId: string) {
      return payment.pay({ orderId });
    },
  };
}
```

常见形态:

- 策略是函数、对象字典或 class.
- 适配器是把接口响应转成前端模型.
- 观察者是事件 emitter、Observable、Signal.
- MVVM 常表现为 Hook / Composable / store.
- 装饰器可能是高阶函数或框架 decorator.

TypeScript 的风险是类型看起来安全,运行时并不一定安全.外部接口数据仍然需要 schema 校验.

---

## 六、落地判断:看语言给了什么便宜能力

选写法时先问:

- 这个语言表达函数是否很轻?
- 接口是显式实现还是隐式实现?
- 泛型和 trait 是否会增加理解成本?
- 框架是否已经提供了容器、代理、生命周期?
- 团队更熟悉类层级还是函数组合?

同一个模式的最低成本表达可能不同:

| 模式意图 | Java | Go | Rust | TypeScript |
| --- | --- | --- | --- | --- |
| 策略 | interface + class | interface + struct | trait | function / object |
| 装饰器 | wrapper class / AOP | middleware | trait wrapper | HOC / function |
| 工厂 | factory bean | NewXxx | associated fn | factory function |
| 状态 | state class | explicit switch / map | type state | union type |

不要为了"像模式"而放弃语言优势.

---

## 七、一句话总结

> **模式是意图,语言是表达介质**.真正要识别的是变化点和稳定边界,不是代码长得像不像某本书里的 Java 类图.
