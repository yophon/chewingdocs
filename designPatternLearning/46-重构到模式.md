# 重构到模式:从坏味道开始,不要从模式开始

很多人学完设计模式后,第一反应是到处找地方套模式.看到 if-else 就想策略模式,看到 new 就想工厂,看到一个类大一点就想拆.这样重构很危险,因为你不是在解决问题,而是在寻找模式存在感.

真正可靠的顺序是:**从坏味道出发,识别变化点,再重构到合适的模式**.

---

## 一、问题:从模式开始会制造过度设计

假设有一段折扣逻辑:

```ts
function discount(type: string, amount: number) {
  if (type === "vip") return amount * 0.8;
  if (type === "coupon") return amount - 20;
  return amount;
}
```

有人会立刻说:"这是策略模式."然后创建:

```text
DiscountStrategy
VipDiscountStrategy
CouponDiscountStrategy
DefaultDiscountStrategy
DiscountStrategyFactory
DiscountContext
```

如果业务只有三个规则,一个月改一次,这就是过度设计.模式没有消灭复杂度,只是把一个函数拆成了六个文件.

但如果规则开始变成这样:

- 不同国家有不同折扣.
- 折扣要灰度发布.
- 运营配置规则.
- 规则要单测和审计.
- 新规则每周上线.

这时策略、规则表、责任链甚至规则引擎才有讨论价值.

---

## 二、判断框架:坏味道 -> 变化点 -> 模式

重构到模式可以按这个链路:

```text
坏味道
  -> 哪个变化点导致它反复修改
  -> 哪些稳定点不能被破坏
  -> 哪种模式能隔离这个变化
  -> 引入后测试和理解成本是否下降
```

常见映射:

| 坏味道 | 真实问题 | 可能模式 |
| --- | --- | --- |
| if-else 反复加分支 | 算法族变化 | 策略 / 规则表 |
| 构造参数越来越多 | 对象创建复杂 | 建造者 |
| 到处调用第三方 SDK | 外部接口不稳定 | 适配器 / 外观 |
| 状态判断散落各处 | 状态迁移复杂 | 状态模式 / 状态机 |
| 请求处理链不断插逻辑 | 步骤可组合 | 责任链 / 中间件 |
| 多处重复权限校验 | 横切逻辑 | 代理 / 装饰器 / AOP |

注意:"可能模式"不是答案,只是候选.

---

## 三、示例:从 if-else 到策略,但只走必要步

第一步不是创建一堆类,而是先把变化点命名:

```ts
type DiscountRule = (amount: number) => number;

const rules: Record<string, DiscountRule> = {
  vip: (amount) => amount * 0.8,
  coupon: (amount) => amount - 20,
  none: (amount) => amount,
};

function discount(type: string, amount: number) {
  const rule = rules[type] ?? rules.none;
  return rule(amount);
}
```

如果规则仍然简单,到这里就够了.你已经把分支变成了可替换规则.

当规则需要依赖外部服务、配置和测试替身时,再升级接口:

```ts
interface DiscountPolicy {
  supports(type: string): boolean;
  apply(amount: number): Promise<number>;
}

class DiscountService {
  constructor(private policies: DiscountPolicy[]) {}

  async discount(type: string, amount: number) {
    const policy = this.policies.find((p) => p.supports(type));
    return policy ? policy.apply(amount) : amount;
  }
}
```

这就是重构到模式:先有痛点,再逐步引入结构.

## 四、落地判断:每一步都要可回滚

好的重构不是"一口气换架构",而是连续的小步:

1. 给现有行为补测试或快照.
2. 提取重复表达式,不改结构.
3. 给变化点命名,例如 `DiscountRule`.
4. 把分支搬进表、函数或对象.
5. 只有当依赖和生命周期复杂时,再引入接口和类.
6. 删除旧路径,避免新旧结构并存太久.

每一步都应该能回答:

- 行为有没有变?
- 复杂度有没有搬到更合适的位置?
- 新结构是否让下一次改动更小?
- 测试是否更容易写?

如果回答不了,暂停.

## 五、代价与误用

重构到模式的代价是抽象成本.抽象会增加命名、文件、间接调用和团队共识成本.

常见误用:

**误用一:为了消灭 if-else 而消灭 if-else**.有些分支就是业务规则的直接表达,表驱动或 switch 反而更清楚.

**误用二:一上来就抽接口**.如果只有一个实现,且没有测试替身、插件化或未来明确变化,接口可能只是噪音.

**误用三:重构范围过大**.一次 PR 同时改命名、改目录、改模式、改业务,评审很难判断风险.

**误用四:新旧抽象并存**.半套策略加半套旧 if-else,比原来更难懂.

---

## 六、一句话总结

> **不要从"我要用哪个模式"开始,要从"这段代码被哪种变化反复折磨"开始**.模式是重构后的形状,不是重构的理由.
