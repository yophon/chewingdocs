# 依赖倒置与依赖注入

很多代码难测试,不是因为业务复杂,而是因为对象在内部到处 `new` 具体依赖。你想测试订单逻辑,它顺手连数据库,调短信,发 HTTP 请求。最后单元测试变成集成测试,本地跑一次像启动一个小系统。

依赖倒置和依赖注入解决的就是这个问题:核心业务不要被具体技术细节绑死。

> 一句话先记住:依赖倒置是设计原则,依赖注入是实现手段,IoC 容器只是工具。

---

## 一、先看问题

```ts
class OrderService {
  async submit(orderId: string) {
    const repo = new MysqlOrderRepository();
    const sms = new AliyunSmsClient();

    const order = await repo.find(orderId);
    await repo.markSubmitted(orderId);
    await sms.send(order.phone, "submitted");
  }
}
```

这段代码的问题:

- `OrderService` 被 MySQL 绑死
- `OrderService` 被阿里云短信绑死
- 测试必须准备数据库和短信配置
- 想切换实现,要改业务服务

真正稳定的是提交流程。变化的是订单存储和通知渠道。

---

## 二、依赖倒置:让核心依赖抽象

先把业务真正需要的能力说出来:

```ts
interface OrderRepository {
  find(id: string): Promise<Order>;
  markSubmitted(id: string): Promise<void>;
}

interface Notifier {
  send(phone: string, message: string): Promise<void>;
}
```

业务服务依赖这些抽象:

```ts
class OrderService {
  constructor(
    private orders: OrderRepository,
    private notifier: Notifier,
  ) {}

  async submit(orderId: string) {
    const order = await this.orders.find(orderId);
    await this.orders.markSubmitted(orderId);
    await this.notifier.send(order.phone, "submitted");
  }
}
```

具体实现放在外面:

```ts
class MysqlOrderRepository implements OrderRepository {
  async find(id: string) {
    return db.orders.find(id);
  }

  async markSubmitted(id: string) {
    await db.orders.update(id, { status: "submitted" });
  }
}
```

依赖方向变成:

```text
OrderService -> OrderRepository(interface)
MysqlOrderRepository -> OrderRepository(interface)
```

高层业务和低层技术都依赖抽象,这就是依赖倒置。

---

## 三、依赖注入:把依赖从外面传进来

依赖注入最简单的形式就是构造函数参数:

```ts
const service = new OrderService(
  new MysqlOrderRepository(),
  new AliyunNotifier(),
);
```

测试时换成 fake:

```ts
class FakeOrders implements OrderRepository {
  async find(id: string) {
    return { id, phone: "10086" } as Order;
  }

  async markSubmitted(id: string) {}
}

class FakeNotifier implements Notifier {
  messages: string[] = [];

  async send(phone: string, message: string) {
    this.messages.push(message);
  }
}
```

测试业务逻辑时不再需要真实 MySQL 和短信服务。

---

## 四、三种常见注入方式

### 1. 构造函数注入

最常用,依赖明确,对象创建后不容易变:

```ts
class UserService {
  constructor(private users: UserRepository) {}
}
```

适合必需依赖。

### 2. 方法参数注入

依赖只在某个方法里临时使用:

```ts
function exportReport(report: Report, renderer: Renderer) {
  return renderer.render(report);
}
```

适合函数式写法和一次性策略。

### 3. 属性注入

框架里常见,但依赖不够显式:

```java
class UserService {
  @Autowired
  private UserRepository users;
}
```

它方便,但对象在构造完成时可能还不可用,测试时也不如构造函数清晰。

## 五、IoC 容器做了什么

容器本质上是一个对象装配器:

```text
读取配置 / 注解
  -> 创建对象
  -> 解析依赖
  -> 管理生命周期
  -> 注入到需要的地方
```

Spring, NestJS, Angular 都有类似机制。容器适合依赖很多,生命周期复杂的大项目。小项目手动装配就够了。

手动装配并不低级:

```ts
export function buildOrderService() {
  const orders = new MysqlOrderRepository(db);
  const notifier = new AliyunNotifier(config.sms);
  return new OrderService(orders, notifier);
}
```

这段代码清楚,可控,也很容易测试。

## 六、落地判断

应该引入依赖倒置的信号:

- 业务类里直接 new 数据库,缓存,HTTP 客户端,SDK
- 单元测试必须连接真实外部系统
- 同一业务能力有多个实现,比如本地 / 云端 / fake
- 技术实现可能切换,比如 MySQL 到 PostgreSQL,短信供应商切换
- 业务核心需要和框架解耦

可以暂时不抽象的信号:

- 只是纯计算对象,没有外部副作用
- 依赖稳定且没有替换需求
- 抽象出来的接口只是机械复制具体类方法
- 团队还没有清楚的模块边界

---

## 七、代价与误用

**误用一:所有类都配接口**

`UserService` 和 `UserServiceImpl` 一一对应,接口没有第二个实现,也没有隔离作用,通常只是噪音。

**误用二:接口定义在错误的层**

接口应该由使用方需求驱动。业务只需要 `send(message)`,不要暴露短信 SDK 的 20 个参数。

**误用三:容器隐藏依赖**

到处自动注入会让依赖关系不透明。构造函数参数过多时,不要只怪容器,那通常是类职责过大。

**误用四:把依赖注入当成全局变量**

容器不是服务定位器。业务代码到处 `container.get()` 会重新制造隐式依赖。

---

## 八、一句话总结

> 依赖倒置让核心业务依赖稳定抽象,依赖注入把具体实现从外面传进来。目标不是优雅,是可替换,可测试,可演进。
