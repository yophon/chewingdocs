# 六边形架构:端口、适配器与业务核心保护

很多系统的业务代码不是被业务本身污染的,而是被外部世界污染的。支付 SDK 的异常类型直接出现在订单逻辑里,ORM Entity 直接成为领域对象,消息队列的重试语义写进用例流程,第三方接口字段名到处传播。等到要换支付渠道、换数据库、补单、写单元测试时,你会发现"业务核心"其实已经不存在了,只剩一团框架和 SDK 的胶水。

六边形架构要解决的问题是:**让业务核心不依赖外部技术细节,外部世界通过端口接入核心**。

---

## 一、问题:外部依赖倒灌进业务核心

先看一个常见写法:

```ts
async function payOrder(orderId: string) {
  const order = await prisma.order.findUnique({ where: { id: orderId } });
  if (!order) throw new Error("not found");
  if (order.status !== "PENDING") throw new Error("cannot pay");

  const result = await stripe.paymentIntents.create({
    amount: order.amount,
    currency: "usd",
  });

  await prisma.order.update({
    where: { id: orderId },
    data: { status: "PAID", paymentId: result.id },
  });
}
```

这段代码有三个变化点:

- 数据库访问方式可能变:Prisma、MyBatis、SQL、内存仓储.
- 支付渠道可能变:Stripe、支付宝、微信、内部账务.
- 业务规则可能变:订单状态、金额冻结、幂等、补偿.

稳定点是"支付订单"这个用例:加载订单 -> 判断能否支付 -> 请求支付能力 -> 保存结果。六边形架构会把稳定的业务核心放中间,把数据库、HTTP、MQ、第三方 SDK 都放外面。

---

## 二、模式:端口定义需要什么,适配器实现怎么做

六边形架构也叫 Ports and Adapters:

```text
             HTTP Adapter
                  |
                  v
        +----------------------+
CLI --> |   Application Core   | <-- MQ Consumer
        |  Use Case + Domain   |
        +----------------------+
             ^            ^
             |            |
      OrderPort      PaymentPort
             |            |
       DB Adapter    Stripe Adapter
```

端口是核心定义的接口,表达"业务需要什么能力"。适配器是外部实现,负责把具体技术接到端口上。

```ts
interface OrderRepository {
  findById(id: string): Promise<Order | null>;
  save(order: Order): Promise<void>;
}

interface PaymentGateway {
  charge(input: { orderId: string; amount: number }): Promise<{ paymentId: string }>;
}

class PayOrderUseCase {
  constructor(
    private orders: OrderRepository,
    private payments: PaymentGateway,
  ) {}

  async execute(orderId: string) {
    const order = await this.orders.findById(orderId);
    if (!order) throw new Error("not found");

    order.ensurePayable();
    const paid = await this.payments.charge({ orderId, amount: order.amount });
    order.markPaid(paid.paymentId);

    await this.orders.save(order);
  }
}
```

核心只知道 `PaymentGateway`,不知道 Stripe。只知道 `OrderRepository`,不知道 Prisma。这就是依赖方向的变化。

---

## 三、落地判断:它和分层架构差在哪

分层架构通常是:

```text
Controller -> Service -> Repository -> DB
```

六边形架构强调的是:

```text
Adapter -> UseCase -> Port <- Adapter
```

区别不在目录形状,而在**接口归谁定义**。普通分层里,Service 往往直接依赖具体 Repository。六边形里,核心定义端口,数据库适配器来实现端口。这样核心可以在没有数据库、没有 Web 框架的情况下测试。

适合使用六边形架构的信号:

- 一个用例需要接多个外部系统,比如支付、库存、物流、风控.
- 核心业务规则重要,不能被框架生命周期牵着走.
- 外部供应商不稳定,经常替换 SDK 或协议.
- 你希望大量单元测试不启动数据库、MQ、HTTP 服务.
- 系统有多个入口,比如 HTTP、定时任务、消息消费、CLI 都会触发同一个用例.

在 Java 里,端口通常是核心模块中的 interface,适配器在 infrastructure 模块实现。在 Go 里,接口可以定义在 usecase 包附近,实现放到 mysql、redis、payment 包里。在 TypeScript 里,可以用构造函数注入和测试替身实现。

## 四、代价与误用

六边形架构的代价是抽象数量增加。每个外部能力都可能多一个 Port、多一个 Adapter、多一个测试替身。如果系统只是简单后台 CRUD,这些接口很容易变成"为了架构而架构"。

最常见误用是把所有 Repository 都机械地抽接口,但接口只有一个实现,测试也不用替身。这样的端口没有保护变化点,只是增加跳转成本。

第二个误用是把端口设计成外部 SDK 的翻版。比如:

```ts
interface StripePort {
  createPaymentIntent(args: StripeCreatePaymentIntentArgs): Promise<StripeResult>;
}
```

这不叫保护核心,只是给 Stripe 包了一层皮。更好的端口应该表达业务需要:

```ts
interface PaymentGateway {
  charge(input: ChargeRequest): Promise<ChargeResult>;
}
```

第三个误用是以为六边形架构等于微服务。它是代码架构模式,单体也可以用;微服务是部署和组织边界模式,两者不是同一个层面的东西。

---

## 五、应用架构和前端模式的边界

六边形架构主要讨论业务核心和外部系统的依赖方向。前端也可以借鉴"端口和适配器"思想,比如把浏览器 storage、HTTP client、埋点 SDK 包起来,但组件模式、单向数据流、响应式模式解决的是 UI 组合和状态传播问题,不要把它们强行画成一个后端六边形。

判断边界时问一句:**我是在保护业务规则免受外部技术变化影响,还是在组织界面状态和渲染?**前者更像六边形,后者属于前端模式。

---

## 一句话总结

六边形架构用端口把业务核心和外部技术隔开,适合保护重要且多变的业务能力,但如果没有真实的外部变化点,它只会制造一堆空接口。
