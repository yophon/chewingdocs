# 分层架构:Controller / Service / Repository 的边界

很多业务系统一开始不是被复杂算法拖垮的,而是被"哪里都能写业务逻辑"拖垮的。Controller 里查库、拼 DTO、扣库存、发消息;Repository 里顺手判断订单状态;Service 越写越像一个杂物间。短期看功能上线很快,长期看每改一个需求都要全局搜索,测试也只能从接口一路打到数据库。

分层架构要解决的不是"把代码分成几个文件夹",而是回答一个更具体的问题:**一段逻辑应该放在哪一层,它依赖谁,它不应该知道谁**。

---

## 一、问题:代码没有边界时会怎么坏

最常见的坏结构是这样的:

```ts
app.post("/orders", async (req, res) => {
  const user = await db.user.find(req.body.userId);
  const product = await db.product.find(req.body.productId);

  if (!user.active) throw new Error("user disabled");
  if (product.stock <= 0) throw new Error("out of stock");

  await db.product.update(product.id, { stock: product.stock - 1 });
  const order = await db.order.insert({ userId: user.id, productId: product.id });
  await mq.publish("order.created", order);

  res.json(order);
});
```

这段代码的问题不是"不够优雅",而是变化点混在一起:

- HTTP 协议会变:REST 可能换成 GraphQL,也可能加 RPC.
- 入参格式会变:字段名、校验规则、DTO 都会变.
- 业务规则会变:库存、会员、优惠、风控都会变.
- 数据存储会变:ORM、SQL、缓存、分库策略都会变.
- 副作用会变:消息、邮件、日志、审计都会变.

稳定点是什么?下单这个业务用例的主流程相对稳定:校验用户 -> 校验库存 -> 创建订单 -> 触发后续动作。分层架构就是把这些变化点按方向隔开。

---

## 二、模式:三层不是三堆文件

典型 Web 后端会拆成:

```text
Client
  |
  v
Controller  接收请求,参数转换,返回响应
  |
  v
Service     编排业务用例,事务边界,调用领域逻辑
  |
  v
Repository  数据访问,查询封装,持久化细节
  |
  v
DB / Cache / MQ
```

关键规则只有一条:**依赖从外层流向内层的稳定抽象,不要让底层反过来知道上层**。在普通分层架构里,Controller 可以依赖 Service,Service 可以依赖 Repository,Repository 不应该依赖 Controller。

一个最小的 TypeScript 结构:

```ts
class OrderController {
  constructor(private orders: OrderService) {}

  async create(req: Request) {
    return this.orders.createOrder({
      userId: req.body.userId,
      productId: req.body.productId,
    });
  }
}

class OrderService {
  constructor(
    private products: ProductRepository,
    private orders: OrderRepository,
  ) {}

  async createOrder(cmd: CreateOrderCommand) {
    const product = await this.products.findById(cmd.productId);
    if (!product || product.stock <= 0) throw new Error("out of stock");

    await this.products.decreaseStock(product.id);
    return this.orders.save({ userId: cmd.userId, productId: product.id });
  }
}

class OrderRepository {
  async save(order: NewOrder) {
    return db.order.insert(order);
  }
}
```

Controller 不判断库存,Repository 不决定能不能下单,Service 不关心 HTTP status code。这就是边界。

## 三、落地判断:每层到底放什么

判断逻辑归属时,不要看文件名,看它为什么变化:

| 逻辑 | 应该放哪 | 原因 |
| --- | --- | --- |
| 参数解析、登录态读取、HTTP 状态码 | Controller | 跟入口协议绑定 |
| 用例编排、事务、权限粗校验 | Service | 跟应用流程绑定 |
| 订单是否可支付、库存是否可扣 | Domain / Service | 跟业务规则绑定 |
| SQL、ORM、分页查询、索引友好查询 | Repository | 跟存储实现绑定 |
| DTO 和 ViewModel 转换 | Controller 或 Mapper | 跟展示/API 契约绑定 |

在 Spring 里,常见形态是 `Controller -> ApplicationService -> Repository`。在 Go 里,常见形态是 `handler -> service -> store`。在 NestJS 里,则经常是 `controller -> provider -> repository`。

分层架构适合:

- CRUD 为主,但业务规则已经开始变复杂.
- 团队多人协作,需要稳定的代码放置规则.
- 需要测试 Service,但暂时不想引入六边形或 Clean Architecture 的完整抽象.
- 技术栈以 Web + ORM + 数据库为主,框架本身已经鼓励分层.

不适合把所有项目都分成同样厚的三层。后台管理页面的简单查询可以直接 `Controller -> Repository`;复杂订单流程可以有更厚的应用服务和领域对象。

## 四、代价与误用

分层架构的代价是"层与层之间要传递模型"。DTO、Entity、PO、VO 如果没有边界意识,很容易变成重复劳动。过度分层会让一个新增字段穿过五个对象,最后大家开始讨厌架构。

常见误用有三类。

第一,把 Service 写成上帝类。所有业务都塞进 `OrderService`,几千行代码里既有下单,也有退款、发票、物流和报表。这不是分层,只是把 Controller 的混乱搬到了 Service。

第二,把 Repository 写成业务层。比如 `findPayableOrders` 可以接受,因为它表达查询意图;但 `payOrderIfVipUser` 就不应该在 Repository,因为它决定业务行为。

第三,把应用架构和前端模式混为一谈。分层架构主要解决后端或业务应用的模块边界;前端组件树、单向数据流、响应式依赖是 UI 状态组织问题。它们可以同时存在,但不是同一类模式。

替代方案也要看复杂度:

- 简单脚本:函数拆分就够了.
- 普通 CRUD:薄 Service + Repository 就够了.
- 业务核心复杂:考虑 DDD 战术模式.
- 外部系统很多:考虑六边形架构保护核心.

---

## 一句话总结

分层架构的价值不是多建几个目录,而是让入口协议、业务流程和存储细节按变化原因分开,但一旦层数超过问题本身,它就会从秩序变成负担。
