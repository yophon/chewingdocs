# 设计模式系列写作计划

> 这个系列面向已经能写业务代码,但开始频繁遇到"代码越改越乱"的工程师。
> 它不把设计模式写成 23 个名词背诵表,而是回答一个更实际的问题:当需求变化、团队协作、框架约束、测试维护一起压上来时,代码结构应该怎么长。
> MVC / MVP / MVVM 也会讲,但它们属于应用架构模式,不是 GoF 设计模式。这个边界必须先讲清楚。

---

## 设计思路

五层结构,从"为什么需要模式"到"真实项目怎么落地":

```text
模式心智与原则
   -> GoF 设计模式(创建型 / 结构型 / 行为型)
   -> 应用架构模式(MVC / MVP / MVVM / 分层 / 六边形 / DDD)
   -> 前端与客户端模式(组件 / 状态管理 / 单向数据流 / 微前端)
   -> 工程判断(重构 / 反模式 / 选型 / 模式滥用)
```

本系列的重点不是"这个模式的 UML 图长什么样",而是:

- 它解决哪一种变化
- 它把复杂度从哪里搬到哪里
- 它适合什么语言和框架
- 它什么时候会变成过度设计
- 它在 Spring / React / Vue / Flutter / Go / Rust 等真实生态里对应什么写法

每篇尽量用同一个套路:

1. 先给一个没有模式时会坏掉的代码结构
2. 再抽出变化点和稳定点
3. 然后引入模式
4. 最后讲代价、误用和替代方案

## 篇目规划

### 第一层:心智与基础(01-05)

- [ ] 01 设计模式总览:模式不是背诵题,是变化管理工具
- [ ] 02 UML 与代码结构图速通:类图、依赖、组合、聚合、时序图
- [ ] 03 SOLID 原则:哪些有用,哪些容易被误解
- [ ] 04 组合优于继承:复用、扩展点与抽象层的真实代价
- [ ] 05 依赖倒置与依赖注入:从 new 到 IoC 容器

### 第二层:创建型模式(06-10)

- [ ] 06 单例模式:全局唯一、生命周期与测试噩梦

- [ ] 09 原型模式:复制对象、快照与模板化创建
- [ ] 10 对象池与资源池:连接池、线程池与复用边界

### 第三层:结构型模式(11-17)

- [ ] 11 适配器模式:旧接口、新接口与第三方 SDK 包装

- [ ] 13 代理模式:远程代理、缓存代理、权限代理与 AOP

## 写作模板

每篇优先按四段式写:

```text
1. 问题
   - 没有这个模式时,代码怎么坏
   - 变化点是什么,稳定点是什么

2. 模式
   - 角色分工
   - 简化 UML / 结构图
   - 最小代码示例

3. 落地
   - 在主流框架或语言里的真实形态
   - 测试、扩展、性能、并发上的影响

4. 代价
   - 什么时候不该用
   - 常见误用
   - 替代方案
```

代码示例要短,优先 TypeScript / Java / Go 三种表达力强、读者容易迁移的语言。能用伪代码讲清楚时,不要贴一整套框架项目。

## 反对的写法

这个系列不写这些东西:

1. 把 23 个 GoF 模式逐个翻译定义,没有真实变化场景
2. 为了用模式而用模式,把简单代码拆成十几个类
3. 只贴 UML,不解释为什么这样切边界
4. 把 MVC / MVVM 和单例 / 工厂混在一个分类里
5. 只讲 Java 经典写法,忽略现代语言的函数、闭包、trait、泛型、ADT
6. 把设计模式写成面试八股,不讲代码评审和维护成本

## 一个硬指标

看完这个系列后,你应该能在代码评审里讲清楚:

- 这个抽象是在保护哪个变化点
- 这个模式让测试变容易还是变困难
- 现在引入它是不是过早
- 用语言特性或框架机制能不能更简单地解决
- MVC / MVP / MVVM / 分层 / 六边形之间到底差在哪

能做到这些,设计模式就从"背过"变成了"会用"。
