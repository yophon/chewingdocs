# 解释器模式:DSL、小语言与规则引擎的边界

很多系统都会长出"规则".比如优惠规则、告警规则、权限表达式、搜索过滤、工作流条件.最开始你可能写几个 `if`:

```ts
if (user.level === "vip" && order.amount > 100) {
  discount = 20
}
```

后来运营想配置规则,产品想组合条件,客户想自定义表达式.代码里的 `if` 不够用了.

从变化点看,解释器模式处理的是:规则表达会变化,甚至要交给非研发配置.稳定点是:系统需要把规则解析、校验并执行.

---

## 一、问题场景

假设一个告警系统最初只有固定规则:

```ts
function shouldAlert(metric: Metric) {
  return metric.cpu > 80 && metric.memory > 70
}
```

后来客户提出这些需求:

- A 服务用 `cpu > 80 and memory > 70`
- B 服务用 `error_rate > 0.05 or latency > 1000`
- 不同环境阈值不同
- 规则要能保存、展示、复制、禁用
- 规则命中时要告诉用户哪一段条件成立

继续把规则写在代码里,每改一次都要发版.直接开放 JavaScript `eval`,又会带来安全和可控性问题.

这时可以考虑设计一门很小的表达式语言,再用解释器执行它:

```text
cpu > 80 and memory > 70
```

注意这里的"语言"不一定是编程语言,也可以是搜索语法、权限表达式、筛选条件或告警规则.重点是它有一套稳定语法,系统能把它转成结构化表达式并求值.

---

## 二、模式结构

解释器模式的核心是:把一门小语言的语法表示成对象结构,再用解释器求值.

结构可以简化成:

```text
Expression
  -> interpret(context)

TerminalExpression
  -> Field
  -> Literal

NonTerminalExpression
  -> And
  -> Or
  -> GreaterThan
```

几个角色:

- `AbstractExpression`:表达式接口
- `TerminalExpression`:终结符表达式,比如字段、常量
- `NonTerminalExpression`:非终结符表达式,比如 `And`、`Or`、`GreaterThan`
- `Context`:解释时需要的数据
- `Parser`:如果规则来自字符串,还需要解析器把字符串变成表达式树

从变化点看,表达式树承接"规则如何组合".业务系统只提供上下文和执行入口.

---

## 三、TypeScript 示例

下面先不写完整 parser,直接手工构造表达式树.这能更清楚地看到解释器模式的结构.

```ts
type Context = Record<string, number | string | boolean>

interface Expr {
  eval(ctx: Context): boolean
  explain(ctx: Context): string
}

class GreaterThan implements Expr {
  constructor(private field: string, private value: number) {}

  eval(ctx: Context) {
    return Number(ctx[this.field]) > this.value
  }

  explain(ctx: Context) {
    return `${this.field}(${ctx[this.field]}) > ${this.value}`
  }
}

class Equals implements Expr {
  constructor(private field: string, private value: string | boolean) {}

  eval(ctx: Context) {
    return ctx[this.field] === this.value
  }

  explain(ctx: Context) {
    return `${this.field}(${ctx[this.field]}) == ${this.value}`
  }
}
```

组合表达式:

```ts
class And implements Expr {
  constructor(private left: Expr, private right: Expr) {}

  eval(ctx: Context) {
    return this.left.eval(ctx) && this.right.eval(ctx)
  }

  explain(ctx: Context) {
    return `(${this.left.explain(ctx)} AND ${this.right.explain(ctx)})`
  }
}

class Or implements Expr {
  constructor(private left: Expr, private right: Expr) {}

  eval(ctx: Context) {
    return this.left.eval(ctx) || this.right.eval(ctx)
  }

  explain(ctx: Context) {
    return `(${this.left.explain(ctx)} OR ${this.right.explain(ctx)})`
  }
}
```

使用时构造一棵规则树:

```ts
const rule = new And(
  new GreaterThan("cpu", 80),
  new Or(
    new GreaterThan("memory", 70),
    new Equals("env", "prod"),
  ),
)

const context = {
  cpu: 91,
  memory: 65,
  env: "prod",
}

console.log(rule.eval(context)) // true
console.log(rule.explain(context)) // (cpu(91) > 80 AND (memory(65) > 70 OR env(prod) == prod))
```

真实系统如果允许用户输入表达式,还需要词法分析、语法分析、错误提示、类型检查和安全控制.比如把:

```text
cpu > 80 and (memory > 70 or env == "prod")
```

解析成上面的表达式树.

解释器模式不要求你一次写出完整语言.很多系统第一步是只支持结构化 JSON:

```json
{
  "and": [
    { "gt": ["cpu", 80] },
    { "eq": ["env", "prod"] }
  ]
}
```

这牺牲了一点可读性,但能显著降低 parser 复杂度,也更容易做校验和可视化编辑.

## 四、适用边界

适合解释器模式的信号:

- 规则可以用小语法表达
- 语法规模可控,不会无限膨胀
- 规则需要被保存、展示、校验和执行
- 业务希望组合条件,而不是写死分支
- 性能要求可通过缓存 AST 或预编译解决

常见落地包括:

- 权限表达式
- 搜索过滤语法
- 告警规则
- 优惠条件
- 工作流流转条件
- 表单显隐规则

不适合的信号:

- 规则本质是复杂流程,不是表达式
- 语法每天变,还没有稳定边界
- 非研发配置错误会造成高风险事故
- 你需要的是成熟规则平台,不是自己写解释器
- 规则需要复杂推理、冲突解决、优先级和审计

落地时最重要的是限制能力.一个小 DSL 应该只表达必要能力,不要悄悄长成一门通用编程语言.

## 五、和策略、规格、规则引擎的边界

解释器模式容易被滥用,所以边界要清楚.

策略模式适合在几种已知算法里选择.比如不同优惠算法由研发实现,运行时选择一个.它强调"算法可替换".

规格模式适合组合业务条件.比如 `VipUserSpec.and(HighAmountSpec)`.它不一定有完整语言语法,也不一定需要 parser.

解释器模式适合你真的有一套表达式语言,需要解析、组合、求值.它强调"语法结构可解释".

规则引擎比解释器更重.它通常包含规则管理、冲突解决、优先级、事实库、推理、版本、灰度、审计.不要把一个 `eval` 方法叫成规则引擎.

一个实用判断是:如果规则只能由研发通过代码发布,策略或规格通常够用;如果规则要被保存、编辑、展示、校验和解释,解释器才开始有价值;如果规则之间还有复杂调度和治理,才考虑规则引擎.

---

## 六、误用

解释器模式的代价很高.你要处理语法、错误信息、类型、性能、安全、版本兼容和调试体验.

常见误用有五种.

第一,用字符串拼接表达式再 `eval`.这会带来注入和安全问题,也很难做静态检查.

第二,语法没有边界.今天加函数,明天加循环,后天加变量赋值,最后你维护了一门没人愿意学的语言.

第三,忽略可观测性.规则为什么命中、哪一段条件为 false、哪个版本生效,都要能解释.

第四,把复杂业务决策外包给配置.规则可配置不代表责任消失,高风险规则仍然需要测试、审批和回滚.

第五,不做版本兼容.旧规则保存后,新版本解释器仍然要能执行或给出清晰迁移方案.

---

## 七、测试与重构建议

解释器模式的测试要分层.

第一层测试表达式节点:

```ts
it("evaluates greater-than expression", () => {
  const expr = new GreaterThan("cpu", 80)

  expect(expr.eval({ cpu: 90 })).toBe(true)
  expect(expr.eval({ cpu: 70 })).toBe(false)
})
```

第二层测试组合规则:

```ts
it("evaluates composed alert rule", () => {
  const rule = new And(
    new GreaterThan("cpu", 80),
    new Equals("env", "prod"),
  )

  expect(rule.eval({ cpu: 90, env: "prod" })).toBe(true)
  expect(rule.eval({ cpu: 90, env: "test" })).toBe(false)
})
```

第三层测试 parser 和错误信息.如果用户输入 `cpu > and memory`,系统应该指出语法错误位置,而不是抛出一段内部异常.

第四层测试安全和兼容.包括未知字段、类型不匹配、空值、旧版本规则、新增操作符、超长表达式、递归过深等.

从一堆 `if` 重构到解释器时,建议按这个顺序:

1. 收集现有规则,确认它们是否真能归纳成小语法
2. 先设计 AST 或 JSON 结构,暂时不要急着做字符串 parser
3. 实现最少的表达式节点,比如 `and`、`or`、`eq`、`gt`
4. 给每个节点补测试和解释信息
5. 加规则版本、字段白名单和操作符白名单
6. 最后再考虑可视化编辑、字符串语法或预编译优化

不要从"我要做一个规则引擎"开始.更好的起点是:"当前变化点能不能被一组有限表达式稳定表示".

---

## 八、一句话总结

> 解释器模式适合实现边界清晰的小语言,它能把规则从代码里抽出来,但一旦语法失控,你维护的就不是模式,而是一套脆弱的平台.
