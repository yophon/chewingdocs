# 解释器模式:DSL、小语言与规则引擎的边界

很多系统都会长出"规则"。比如优惠规则、告警规则、权限表达式、搜索过滤、工作流条件。最开始你可能写几个 if:

```ts
if (user.level === "vip" && order.amount > 100) {
  discount = 20;
}
```

后来运营想配置规则,产品想组合条件,客户想自定义表达式。代码里的 if 不够用了。变化点是"规则表达会变化,甚至要交给非研发配置"。稳定点是"系统需要把规则解析并执行"。

## 一、问题

解释器模式把一门小语言的语法表示成对象结构,再用解释器求值。

```text
Expression -> interpret(context)
```

它适合非常小、边界清晰、语法稳定的语言。注意,这里的"语言"不一定是编程语言,也可以是:

- `age > 18 and city == "shanghai"`
- `role:admin or owner:true`
- `amount > 1000 && risk == low`

## 二、模式

角色:

- AbstractExpression:表达式接口
- TerminalExpression:终结符表达式,比如字段、常量
- NonTerminalExpression:非终结符表达式,比如 And、Or、GreaterThan
- Context:解释时需要的数据

TypeScript 示例:

```ts
type Context = Record<string, number>;

interface Expr {
  eval(ctx: Context): boolean;
}

class GreaterThan implements Expr {
  constructor(private field: string, private value: number) {}

  eval(ctx: Context) {
    return ctx[this.field] > this.value;
  }
}

class And implements Expr {
  constructor(private left: Expr, private right: Expr) {}

  eval(ctx: Context) {
    return this.left.eval(ctx) && this.right.eval(ctx);
  }
}

const rule = new And(
  new GreaterThan("amount", 100),
  new GreaterThan("score", 80),
);

console.log(rule.eval({ amount: 120, score: 90 }));
```

这个例子没有写 parser,而是直接手工构造 AST。真实系统如果允许用户输入表达式,还需要词法分析、语法分析、错误提示和安全控制。

## 三、和策略、规格、规则引擎的边界

解释器模式容易被滥用,所以边界要清楚。

策略模式适合在几种已知算法里选择。比如不同优惠算法由研发实现,运行时选择一个。

规格模式适合组合业务条件。比如 `VipUserSpec.and(HighAmountSpec)`。它不一定有完整语言语法。

解释器模式适合你真的有一套表达式语言,需要解析、组合、求值。

规则引擎比解释器更重。它通常包含规则管理、冲突解决、优先级、事实库、推理、版本、灰度、审计。不要把一个 `eval` 方法叫成规则引擎。

## 四、落地判断

适合解释器模式的信号:

- 规则可以用小语法表达
- 语法规模可控,不会无限膨胀
- 规则需要被保存、展示、校验和执行
- 性能要求可通过缓存 AST 或预编译解决

不适合的信号:

- 规则本质是复杂流程,不是表达式
- 语法每天变,还没有稳定边界
- 非研发配置错误会造成高风险事故
- 你需要的是成熟规则平台,不是自己写解释器

落地时最重要的是限制能力。一个小 DSL 应该只表达必要能力,不要悄悄长成一门通用编程语言。

## 五、代价与误用

解释器模式的代价很高。你要处理语法、错误信息、类型、性能、安全、版本兼容和调试体验。

常见误用:

第一,用字符串拼接表达式再 `eval`。这会带来注入和安全问题,也很难做静态检查。

第二,语法没有边界。今天加函数,明天加循环,后天加变量赋值,最后你维护了一门没人愿意学的语言。

第三,忽略可观测性。规则为什么命中、哪一段条件为 false、哪个版本生效,都要能解释。

第四,把复杂业务决策外包给配置。规则可配置不代表责任消失,高风险规则仍然需要测试、审批和回滚。

## 六、一句话总结

解释器模式适合实现边界清晰的小语言。它能把规则从代码里抽出来,但一旦语法失控,你维护的就不是模式,而是一套脆弱的平台。
