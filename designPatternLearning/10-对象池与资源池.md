# 对象池与资源池

对象池经常被误解成"创建对象很慢,所以都放池子里复用"。这在现代语言里大多数时候是错的。普通小对象创建和垃圾回收已经很快,强行池化反而会增加状态污染和并发 bug。

真正值得池化的通常不是普通对象,而是昂贵资源:数据库连接,线程,Socket,大 buffer,GPU 资源。

> 一句话先记住:池化不是优化对象创建,而是控制昂贵资源的数量、复用和生命周期。

---

## 一、先看问题

如果每次请求都新建数据库连接:

```ts
async function query(sql: string) {
  const conn = await createConnection();
  try {
    return await conn.query(sql);
  } finally {
    await conn.close();
  }
}
```

问题很明显:

- 建连成本高
- 数据库承受大量连接抖动
- 并发高时连接数失控
- 连接生命周期难管理

连接池的思路是提前维护一组可复用连接:

```text
Request -> Pool.borrow()
             |
             v
          Connection
             |
Request -> Pool.release(connection)
```

---

## 二、最小对象池结构

下面是简化版,真实连接池会复杂得多:

```ts
class Pool<T> {
  private idle: T[] = [];

  constructor(
    private create: () => Promise<T>,
    private reset: (item: T) => Promise<void>,
    private max: number,
  ) {}

  async borrow(): Promise<T> {
    const item = this.idle.pop();
    if (item) return item;
    return this.create();
  }

  async release(item: T) {
    await this.reset(item);
    if (this.idle.length < this.max) {
      this.idle.push(item);
    }
  }
}
```

这个例子只展示核心关系:

- `borrow`: 获取资源
- `release`: 归还资源
- `reset`: 清理状态
- `max`: 控制池大小

真实生产池还要处理等待队列,超时,健康检查,关闭,泄漏检测等。

---

## 三、资源池保护的稳定点

池化通常在保护这些稳定点:

- 资源数量不能无限增长
- 创建和销毁成本高
- 资源需要统一初始化和清理
- 调用方不该管理底层生命周期
- 高并发时需要背压

连接池,线程池,协程池,Buffer 池,对象存储客户端池,浏览器实例池,都属于这个思路。

注意:池本身通常是进程级共享资源,但不等于业务单例。你仍然应该把池作为依赖传入需要它的模块。

---

## 四、池大小不是越大越好

池化的本质是限流。池太小会排队,池太大会把下游打爆。

数据库连接池尤其典型:

```text
应用实例数 * 每实例最大连接数 <= 数据库能承受的连接数
```

如果有 20 个应用实例,每个实例连接池最大 50,数据库理论上会看到 1000 个连接。很多事故不是代码查询慢,而是池大小随便配。

池大小要结合:

- 下游最大连接数
- 应用实例数量
- 平均请求耗时
- 峰值并发
- 超时和重试策略

---

## 五、状态清理是核心

复用资源最怕脏状态。比如连接借出去以后改了事务隔离级别,设置了 session 变量,打开了事务却没提交,下一个请求直接继承这些状态。

归还前必须清理:

```ts
async function resetConnection(conn: DbConnection) {
  await conn.rollbackIfNeeded();
  await conn.clearSessionVariables();
  await conn.setAutoCommit(true);
}
```

池化的难点不在数组 push/pop,而在资源是否能被安全地重新交给下一个调用方。

---

## 六、对象池和缓存的区别

| 机制 | 目的 | 归还? | 例子 |
| --- | --- | --- | --- |
| 对象池 | 复用资源,控制并发 | 需要 release | 数据库连接池 |
| 缓存 | 避免重复计算或读取 | 不需要 release | Redis 缓存,本地 LRU |
| 单例 | 保证进程内一个实例 | 不存在借还 | Logger,Config |

不要把三者混用。缓存关心命中率和过期,对象池关心借出归还和资源上限。

---

## 七、落地判断

适合使用池的信号:

- 创建成本明显高,比如网络连接,线程,浏览器实例
- 下游资源数量有限
- 并发需要排队或限流
- 资源可以被清理后安全复用
- 已有成熟库提供池化能力

不适合使用池的信号:

- 普通业务对象,创建很便宜
- 对象内部状态复杂,难以重置
- 池化只是为了减少 GC,但没有性能证据
- 语言运行时已经有更好的分配优化
- 忘记归还会造成更严重的问题

优先使用成熟连接池和线程池。自己写池之前,先确认库没有现成方案。

---

## 八、代价与误用

**误用一:池化普通对象**

在 JavaScript,Go,Java 这类现代运行时里,小对象分配通常很快。手写对象池可能比直接创建更慢,还更难维护。

**误用二:忘记 release**

资源借出后不归还,池会被耗尽。真实代码要用 `try/finally`:

```ts
const conn = await pool.borrow();
try {
  return await conn.query(sql);
} finally {
  await pool.release(conn);
}
```

**误用三:池大小拍脑袋**

池大小不是性能开关。调大可能只是把压力转嫁给数据库,消息队列或远程 API。

**误用四:没有健康检查**

池里的连接可能已经失效。借出前或归还后需要检查,否则会把坏资源反复交给调用方。

---

## 九、一句话总结

> 对象池真正管理的是昂贵资源的生命周期和上限。只有当创建成本、并发控制和状态清理都说得清时,池化才值得引入。
