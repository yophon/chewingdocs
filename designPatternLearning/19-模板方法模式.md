# 模板方法模式:框架钩子,生命周期与反向控制

有些流程的整体步骤很稳定,但其中某些细节会变化.

比如导入文件通常是:读取文件 -> 解析数据 -> 校验数据 -> 保存数据 -> 记录结果.不同文件格式的解析方式不同,校验规则也可能不同,但主流程不应该每个实现都复制一遍.

从变化点看,模板方法处理的是:流程中的局部步骤会变化.稳定点是:流程骨架、执行顺序和资源边界.

---

## 一、问题场景

假设系统要支持 CSV、JSON、Excel 三种数据导入.最直觉的写法是每个导入器都写完整流程:

```ts
class CsvImporter {
  async import(file: string) {
    const raw = await read(file)
    const rows = parseCsv(raw)
    validate(rows)
    await save(rows)
    await audit("csv import success")
  }
}

class JsonImporter {
  async import(file: string) {
    const raw = await read(file)
    const rows = JSON.parse(raw)
    validate(rows)
    await save(rows)
    await audit("json import success")
  }
}
```

这段代码的问题不是重复的 `read` 和 `save` 几行,而是流程控制权被复制了.

以后要加这些能力:

- 导入前检查文件大小
- 导入过程加事务
- 失败时记录审计日志
- 保存后刷新缓存
- 统一统计耗时
- 导入结束释放临时文件

如果每个导入器都掌握完整流程,这些稳定的流程要求就要在多个类里同步修改.变化点和稳定点混在一起,越改越容易漏.

---

## 二、模式结构

模板方法模式的核心是:父类定义算法骨架,把可变步骤留给子类实现.

结构可以简化成:

```text
AbstractClass.templateMethod()
  -> fixedStep()
  -> abstract variableStep()
  -> hook()
  -> fixedStep()

ConcreteClass
  -> implements variableStep()
  -> optionally overrides hook()
```

几个角色:

- `templateMethod`:稳定流程,通常不希望子类改写
- `primitive operation`:必须由子类实现的变化步骤
- `hook`:可选扩展点,子类需要时覆写
- `ConcreteClass`:填充具体变化逻辑

模板方法本质上体现了一种反向控制.不是业务代码随便安排流程,而是框架或父类控制生命周期,业务实现被调用的钩子.

## 三、TypeScript 示例

下面用导入流程做例子:

```ts
type ImportResult = {
  total: number
  success: number
}

abstract class Importer<T> {
  async import(file: string): Promise<ImportResult> {
    const startedAt = Date.now()

    try {
      await this.beforeImport(file)

      const raw = await this.read(file)
      const records = this.parse(raw)

      this.validate(records)
      const result = await this.save(records)

      await this.afterImport(file, result, Date.now() - startedAt)
      return result
    } catch (error) {
      await this.onError(file, error)
      throw error
    } finally {
      await this.cleanup(file)
    }
  }

  protected async read(file: string): Promise<string> {
    return `content of ${file}`
  }

  protected abstract parse(raw: string): T[]

  protected validate(records: T[]) {
    if (records.length === 0) {
      throw new Error("empty import file")
    }
  }

  protected async save(records: T[]): Promise<ImportResult> {
    return { total: records.length, success: records.length }
  }

  protected async beforeImport(_file: string) {}

  protected async afterImport(
    _file: string,
    _result: ImportResult,
    _costMs: number,
  ) {}

  protected async onError(file: string, error: unknown) {
    console.error("import failed", file, error)
  }

  protected async cleanup(_file: string) {}
}
```

具体导入器只实现变化部分:

```ts
type UserRow = {
  name: string
  email: string
}

class JsonUserImporter extends Importer<UserRow> {
  protected parse(raw: string): UserRow[] {
    const data = JSON.parse(raw)
    if (!Array.isArray(data)) {
      throw new Error("json import expects array")
    }
    return data
  }

  protected validate(records: UserRow[]) {
    super.validate(records)

    for (const record of records) {
      if (!record.email.includes("@")) {
        throw new Error(`invalid email: ${record.email}`)
      }
    }
  }
}
```

`import` 是模板方法.它控制读取、解析、校验、保存、审计、异常和清理的顺序.子类不能随便重排流程,只能实现 `parse`、`validate` 这类变化点.

如果不想使用继承,也可以用高阶函数表达同样思想:

```ts
async function importFile<T>(
  file: string,
  parse: (raw: string) => T[],
  validate: (records: T[]) => void,
) {
  const raw = await read(file)
  const records = parse(raw)
  validate(records)
  return save(records)
}
```

模式的核心不是"一定要继承",而是把稳定流程收口,只开放明确的变化点.

## 四、适用边界

模板方法适合这些场景:

- 流程步骤固定,局部步骤可变
- 需要统一事务、日志、错误处理、资源释放
- 框架希望控制生命周期,业务只实现钩子
- 多个实现复制了同一套流程骨架
- 子类之间的差异能被少量钩子表达

典型例子包括:

- Web 框架生命周期钩子
- 测试框架的 `setUp` / `tearDown`
- 数据导入导出流程
- 游戏或 UI 框架的初始化、更新、销毁流程
- 批处理任务的读取、处理、提交、回滚流程

不适合的信号:

- 不同实现连步骤顺序都不同
- 子类需要经常跳过父类流程
- 钩子数量越来越多,且组合关系复杂
- 业务更需要运行时替换算法,而不是固定生命周期

如果变化集中在"某一步算法",策略模式更合适.如果流程是多个处理器串起来,且每个处理器可以决定是否继续,责任链更合适.如果流程需要可视化编排和状态迁移,模板方法通常太轻.

## 五、误用

模板方法的主要代价来自继承.继承会把父类和子类绑得很紧,子类必须理解父类什么时候调用哪个钩子.

常见误用有四种.

第一种是父类钩子太多.一个模板方法里开放十几个钩子,子类要靠猜才能知道该覆写哪个.这通常说明流程边界没有设计清楚.

第二种是子类破坏流程假设.父类以为 `parse` 只解析数据,子类却在里面写数据库、发消息、改全局状态,流程边界就乱了.

第三种是父类过度保护.所有方法都 `private`,子类没法扩展;或者所有方法都 `protected`,子类可以乱改.模板方法要明确哪些是稳定步骤,哪些是扩展点.

第四种是用模板方法处理变化很大的流程.如果不同实现有完全不同的生命周期,强行抽父类会得到一个满是空钩子的抽象类.

## 六、测试与重构建议

测试模板方法时,不要只测子类某个小方法,还要测完整流程顺序.可以用一个测试子类记录调用顺序:

```ts
class TestImporter extends Importer<string> {
  calls: string[] = []

  protected parse(raw: string) {
    this.calls.push("parse")
    return [raw]
  }

  protected validate(records: string[]) {
    this.calls.push("validate")
    super.validate(records)
  }

  protected async afterImport() {
    this.calls.push("afterImport")
  }

  protected async cleanup() {
    this.calls.push("cleanup")
  }
}
```

然后断言模板方法确实保证了顺序:

```ts
it("runs import lifecycle in order", async () => {
  const importer = new TestImporter()

  await importer.import("users.json")

  expect(importer.calls).toEqual(["parse", "validate", "afterImport", "cleanup"])
})
```

从重复代码重构到模板方法时,建议按这个顺序:

1. 找出多个类里完全一致的流程骨架
2. 先抽出一个普通函数或基类方法,不要急着开放钩子
3. 只把确实变化的步骤提成抽象方法或回调
4. 给每个钩子写清楚职责和禁止事项
5. 用测试锁住流程顺序、异常处理和资源释放

如果你发现重构后父类越来越胖,子类越来越难懂,就要停下来重新判断:这到底是稳定生命周期,还是一组可以组合的独立步骤.

---

## 七、一句话总结

> 模板方法模式把稳定流程放在父类或框架里,把变化步骤留给子类或回调,适合生命周期明确的场景,但要警惕继承带来的隐式约束.
