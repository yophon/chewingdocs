# 模板方法模式:框架钩子,生命周期与反向控制

有些流程的整体步骤很稳定,但其中某些细节会变化.

比如导入文件通常是:读取文件 -> 解析数据 -> 校验数据 -> 保存数据 -> 记录结果.不同文件格式的解析方式不同,校验规则也可能不同,但主流程不应该每个实现都复制一遍.

变化点是:流程中的某些步骤.稳定点是:流程骨架和执行顺序.

---

## 一、问题

如果每个导入器都自己写完整流程,代码会重复:

```ts
class CsvImporter {
  async import(file: string) {
    const rows = await read(file)
    const data = parseCsv(rows)
    validate(data)
    await save(data)
  }
}

class JsonImporter {
  async import(file: string) {
    const rows = await read(file)
    const data = parseJson(rows)
    validate(data)
    await save(data)
  }
}
```

重复的不是几行代码,而是流程控制权.以后要加审计日志、失败清理、事务边界,每个导入器都要改.

---

## 二、模式

模板方法模式的核心是:父类定义算法骨架,把可变步骤留给子类实现.

结构可以简化成:

```text
AbstractClass.templateMethod()
  -> step1()
  -> hook()
  -> abstract step2()
  -> step3()
```

模板方法通常体现一种反向控制:调用方调用父类提供的稳定流程,子类只填入变化部分.

---

## 三、示例

```ts
abstract class Importer<T> {
  async import(file: string) {
    const raw = await this.read(file)
    const data = this.parse(raw)
    this.validate(data)
    await this.save(data)
  }

  protected async read(file: string) {
    return "file content"
  }

  protected abstract parse(raw: string): T

  protected validate(data: T) {
    if (!data) throw new Error("invalid data")
  }

  protected async save(data: T) {
    console.log("save", data)
  }
}

class JsonImporter extends Importer<unknown> {
  protected parse(raw: string) {
    return JSON.parse(raw)
  }
}
```

`import` 是模板方法.子类不能随便改流程顺序,只能实现 `parse` 这样的变化点.

---

## 四、落地判断

模板方法适合这些场景:

- 流程步骤固定,局部步骤可变
- 需要统一事务、日志、错误处理、资源释放
- 框架希望控制生命周期,业务只实现钩子
- 多个实现复制了同一套流程骨架

典型例子包括:

- Web 框架生命周期钩子
- 测试框架的 `setUp` / `tearDown`
- 数据导入导出流程
- 游戏或 UI 框架的初始化、更新、销毁流程

如果语言支持高阶函数,也可以不用继承:

```ts
async function importFile<T>(file: string, parse: (raw: string) => T) {
  const raw = await read(file)
  const data = parse(raw)
  await save(data)
}
```

这保留了模板方法的思想,但用函数参数表达变化点.

---

## 五、代价与误用

模板方法的主要代价来自继承.继承会把父类和子类绑得很紧,子类必须理解父类什么时候调用哪个钩子.

常见误用有三种.

第一种是父类钩子太多.一个模板方法里开放十几个钩子,子类要靠猜才能知道该覆写哪个.

第二种是子类破坏流程假设.父类以为 `parse` 只解析数据,子类却在里面写数据库,流程边界就乱了.

第三种是用模板方法处理变化很大的流程.如果不同实现连步骤顺序都不同,策略、管道或责任链可能更合适.

---

## 六、一句话总结

> 模板方法模式把稳定流程放在父类或框架里,把变化步骤留给子类或回调,适合生命周期明确的场景,但要警惕继承带来的隐式约束.
