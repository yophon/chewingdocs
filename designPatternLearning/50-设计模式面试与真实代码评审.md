# 设计模式面试与真实代码评审:怎么讲清取舍,不背模板

设计模式面试最差的回答是背定义:"策略模式定义了一系列算法,把它们封装起来..." 这类话不一定错,但没有工程判断.真实代码评审里也没人关心你能不能背出 UML,大家关心的是这段抽象能不能扛住需求变化.

这一篇讲两个场景:面试怎么讲,评审怎么说.

---

## 一、问题:背模板无法证明你会用

面试官问:"策略模式有什么用?"

弱回答:

```text
策略模式可以消除 if-else,符合开闭原则.
```

这句话太空.因为:

- if-else 不一定坏.
- 开闭原则不是所有地方都值得追求.
- 没讲代价.
- 没讲适用条件.

更好的回答结构:

```text
先讲场景 -> 再讲变化点 -> 再讲结构 -> 再讲代价 -> 最后讲替代方案
```

例如:

```text
如果不同会员类型有不同计价规则,而且规则经常新增,我会把计价算法抽成策略.
稳定点是订单计价流程,变化点是具体规则.
简单时可以用函数表,复杂时再用接口和实现类.
代价是文件和间接调用变多,所以规则少且低频变化时不用急着抽.
```

这比定义更像工程师.

---

## 二、面试回答框架

回答任何模式,都可以按六句讲:

1. 它解决哪类变化.
2. 没有它时代码会怎么坏.
3. 它把角色怎么拆.
4. 它让测试或扩展哪里变容易.
5. 它引入什么代价.
6. 什么时候不用它.

以适配器模式为例:

```text
适配器解决的是外部接口和内部模型不一致的问题.
如果业务代码直接调用第三方 SDK,以后 SDK 字段、错误码、鉴权方式一变,业务层会到处改.
所以我会定义内部需要的接口,用 adapter 包住 SDK.
测试时可以替换 adapter,业务层也不用知道 SDK 细节.
代价是多一层映射,字段很多时要维护转换逻辑.
如果只是一次性脚本或非常薄的调用,不一定需要单独适配器.
```

这套回答可以迁移到工厂、策略、代理、观察者、状态模式.

---

## 三、代码评审:不要说"这里应该用模式"

评审里最没用的评论是:

```text
这里建议用策略模式.
```

它没有指出问题,也没有解释收益.更好的评论是:

```text
这个 switch 现在按 channel 分了 8 个分支,而且每次新增渠道都要改同一个函数.
这里的稳定点是 send(message),变化点是渠道实现.
建议先提一个 ChannelSender 映射表,让新增渠道只新增实现和注册项.
如果后续 sender 需要依赖不同配置,再升级成接口实现类.
```

好的评审评论要包含:

- 具体坏味道.
- 真实变化点.
- 最小修改建议.
- 风险或代价.
- 可以接受的替代方案.

## 四、评审示例:分支逻辑

原代码:

```ts
function send(channel: string, message: Message) {
  if (channel === "email") return sendEmail(message);
  if (channel === "sms") return sendSms(message);
  if (channel === "push") return sendPush(message);
  throw new Error("unsupported channel");
}
```

评审不是立刻要求 class:

```ts
type Sender = (message: Message) => Promise<void>;

const senders: Record<string, Sender> = {
  email: sendEmail,
  sms: sendSms,
  push: sendPush,
};

function send(channel: string, message: Message) {
  const sender = senders[channel];
  if (!sender) throw new Error("unsupported channel");
  return sender(message);
}
```

评审意见可以这样写:

```text
这里 channel 分支后续大概率继续增加,可以先改成 sender 表.
这样新增渠道不需要改 send 主流程,测试也能单独覆盖每个 sender.
暂时不需要引入 Sender class,因为当前 sender 没有独立生命周期和依赖注入需求.
```

这就是工程判断:解决问题,但不过度.

## 五、评审示例:全局状态

原代码:

```ts
appStore.setState({
  keyword,
  currentDialogOpen: true,
  productList: data,
});
```

评审意见:

```text
这三个状态生命周期不同.
keyword 影响页面刷新和分享,更适合放 URL.
currentDialogOpen 只属于当前页面交互,放本地 state 就够.
productList 来自接口,建议交给 query cache 管理失效和刷新.
全部放 appStore 会让远处组件也能读写这些临时状态,后续调试成本会变高.
```

这里没有说"必须用某某状态管理模式",但模式判断已经在里面.

## 六、面试和评审都要讲代价

一个成熟回答一定包含代价:

- 策略模式:分支扩展容易,但实现类可能膨胀.
- 工厂模式:创建逻辑集中,但隐藏依赖会让调试变难.
- 单例模式:访问方便,但测试隔离和生命周期管理困难.
- 观察者模式:解耦发送方和接收方,但调用链不直观,可能泄漏.
- 微前端:发布边界清楚,但运行时复杂度上升.

不讲代价的模式回答,听起来像背书;讲清代价和不用场景,才像真正用过.

---

## 七、落地判断:把模式语言翻译成评审语言

代码评审里少用口号,多用具体影响:

| 模式语言 | 评审语言 |
| --- | --- |
| 违反开闭原则 | 新增渠道要改旧主流程,回归范围变大 |
| 应该依赖倒置 | 业务层直接依赖 SDK,测试和替换供应商困难 |
| 可以用外观模式 | 调用方需要知道 5 个子系统顺序,建议收敛成一个入口 |
| 这里是观察者 | 发送方不需要等待接收方结果,可以用事件解耦 |
| 不要过度设计 | 目前只有一个实现,接口没有替换点,先保持直接调用 |

这样说,对方能知道为什么改、怎么改、改到什么程度.

---

## 八、一句话总结

> **面试讲模式,要讲变化点和取舍;评审讲模式,要讲具体风险和最小改法**.能把模式名字翻译成维护成本,才是真的掌握设计模式.
