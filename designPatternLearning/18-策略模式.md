# 策略模式:消灭 if-else,但别制造类爆炸

业务代码里最常见的坏味道之一,是不断增长的 `if-else`.

```ts
if (type === "vip") {
  // ...
} else if (type === "new_user") {
  // ...
} else if (type === "festival") {
  // ...
}
```

一开始只有两种规则,写分支没有问题.后来规则越来越多,每加一种都要改同一个函数,测试也要覆盖一大坨分支.更糟的是,不同规则的参数、边界和依赖不一样,最终一个函数会变成规则垃圾桶.

从变化点看,策略模式处理的是:算法或业务规则会替换、扩展和组合.稳定点是:调用方只需要以同一种方式得到结果.

---

## 一、问题场景

以订单折扣为例.最初只有会员折扣:

```ts
function discount(type: string, amount: number) {
  if (type === "vip") return amount * 0.2
  return 0
}
```

后来新增新人券、节日活动、满减、企业客户价、指定品类折扣.函数会变成这样:

```ts
function discount(type: string, amount: number, user: User, order: Order) {
  if (type === "vip") {
    if (user.level < 3) return 0
    return amount * 0.2
  }

  if (type === "new_user") {
    if (!user.isNew) return 0
    return Math.min(amount * 0.1, 50)
  }

  if (type === "festival") {
    if (!order.tags.includes("festival")) return 0
    return amount >= 300 ? 60 : 0
  }

  throw new Error("unknown discount")
}
```

真正的问题不是分支本身,而是这个函数承担了太多变化:

- 规则增加时要改它
- 规则参数变化时要改它
- 规则依赖外部服务时也要改它
- 某个规则出 bug,容易影响整个结账逻辑
- 测试必须围绕一个越来越大的函数展开

如果促销规则是高频变化点,就应该把它从稳定结账流程里拆出去.

---

## 二、模式结构

策略模式的核心是:把一组可替换的算法封装成同一接口,调用方在运行时选择一个策略执行.

结构可以简化成:

```text
Client
  -> Context
      -> Strategy
          -> ConcreteStrategyA
          -> ConcreteStrategyB
          -> ConcreteStrategyC
```

几个角色:

- `Strategy`:统一策略接口,定义调用方需要的能力
- `ConcreteStrategy`:具体算法或业务规则
- `Context`:持有或接收策略,负责在稳定流程里调用策略
- `Client`:根据配置、请求参数或业务状态选择策略

从变化点看,策略对象承接"怎么计算".调用方保留"什么时候计算"和"计算结果怎么使用".

## 三、TypeScript 示例

下面用折扣计算做一个较完整的例子.先定义上下文和策略接口:

```ts
type User = {
  id: string
  level: number
  isNew: boolean
}

type Order = {
  id: string
  amount: number
  tags: string[]
}

type DiscountContext = {
  user: User
  order: Order
}

interface DiscountStrategy {
  code: string
  calculate(ctx: DiscountContext): number
}
```

每个策略只关心自己的规则:

```ts
class VipDiscount implements DiscountStrategy {
  code = "vip"

  calculate(ctx: DiscountContext) {
    if (ctx.user.level < 3) return 0
    return ctx.order.amount * 0.2
  }
}

class NewUserDiscount implements DiscountStrategy {
  code = "new_user"

  calculate(ctx: DiscountContext) {
    if (!ctx.user.isNew) return 0
    return Math.min(ctx.order.amount * 0.1, 50)
  }
}

class FestivalDiscount implements DiscountStrategy {
  code = "festival"

  calculate(ctx: DiscountContext) {
    if (!ctx.order.tags.includes("festival")) return 0
    return ctx.order.amount >= 300 ? 60 : 0
  }
}
```

结账服务不再关心具体规则细节,只负责查找并执行策略:

```ts
class CheckoutService {
  private strategies: Map<string, DiscountStrategy>

  constructor(strategies: DiscountStrategy[]) {
    this.strategies = new Map(strategies.map(s => [s.code, s]))
  }

  finalPrice(strategyCode: string, ctx: DiscountContext) {
    const strategy = this.strategies.get(strategyCode)
    if (!strategy) throw new Error(`unknown discount strategy: ${strategyCode}`)

    const discount = strategy.calculate(ctx)
    return Math.max(ctx.order.amount - discount, 0)
  }
}

const checkout = new CheckoutService([
  new VipDiscount(),
  new NewUserDiscount(),
  new FestivalDiscount(),
])

const price = checkout.finalPrice("vip", {
  user: { id: "u1", level: 4, isNew: false },
  order: { id: "o1", amount: 500, tags: [] },
})

console.log(price) // 400
```

新增一种折扣时,新增一个策略并注册即可.核心结账流程不需要继续膨胀.

在 TypeScript 里,策略不一定非要是类.如果规则很轻,函数也可以是策略:

```ts
type Discount = (ctx: DiscountContext) => number

const discounts: Record<string, Discount> = {
  vip: ctx => (ctx.user.level >= 3 ? ctx.order.amount * 0.2 : 0),
  none: () => 0,
}
```

语言特性越强,模式长得越轻.关键不是"有没有类",而是"变化规则是否被隔离到了统一协议后面".

## 四、适用边界

策略模式适合这些信号:

- 一个函数里有多种算法分支
- 分支会频繁新增,且每个分支逻辑相对独立
- 调用方只关心统一结果
- 希望单独测试每种规则
- 规则可以被配置、注册或按运行时条件选择

常见落地包括:

- 支付渠道选择
- 价格、优惠、税费计算
- 排序规则和匹配规则
- 文件导出格式
- 风控规则
- 推荐召回或打分算法

不适合的场景也要说清楚:

- 分支很少,且多年不变
- 分支之间共享大量中间状态,很难拆开
- 不同分支的输入输出完全不同
- 你真正需要的是规则编排、优先级和冲突解决,而不是简单策略选择

策略模式解决"同一位置有多种可替换算法"的问题.如果规则之间还要按链路传递、短路、组合,责任链或规格模式可能更合适.如果规则需要由非研发配置成表达式,解释器或规则引擎可能更接近.

## 五、误用

策略模式的代价是对象数量和跳转关系增加.一个简单三行分支,拆成三个类,读代码反而更慢.

常见误用有四种.

第一种是为了消灭所有 `if`.分支本身不是坏事.稳定、短小、不会扩展的分支直接写更清楚.

第二种是策略类爆炸.每个微小差异都建一个类,最后项目里充满 `VipStrategyV2ForApp` 这种名字.很多差异可以用参数、配置或组合策略解决.

第三种是选择逻辑仍然混乱.策略解决的是"规则怎么执行",不自动解决"选哪个规则".选择策略的注册表、配置中心或工厂也要设计清楚.

第四种是策略里偷偷写流程控制.比如折扣策略里直接发券、写订单、调用支付,就会让策略变成新的大杂烩.策略应该尽量保持输入明确、输出明确.

## 六、测试与重构建议

从旧代码重构到策略模式,不要一上来就拆很多类.更稳妥的做法是:

1. 先给原来的大分支补上回归测试,覆盖每种分支的关键边界
2. 提取一个统一接口,让原函数继续作为临时 `Context`
3. 每次只迁移一个分支到独立策略
4. 为每个策略建立独立单元测试
5. 最后再清理原来的分支和重复参数

测试策略时,重点不是测试"类存在",而是测试规则边界:

```ts
describe("VipDiscount", () => {
  it("gives 20 percent discount to level 3 users", () => {
    const strategy = new VipDiscount()

    const discount = strategy.calculate({
      user: { id: "u1", level: 3, isNew: false },
      order: { id: "o1", amount: 100, tags: [] },
    })

    expect(discount).toBe(20)
  })

  it("does not apply to low level users", () => {
    const strategy = new VipDiscount()

    const discount = strategy.calculate({
      user: { id: "u1", level: 2, isNew: false },
      order: { id: "o1", amount: 100, tags: [] },
    })

    expect(discount).toBe(0)
  })
})
```

如果策略依赖外部服务,优先把外部依赖封装到策略构造参数里,测试时注入 fake 或 mock.不要让策略在内部随手创建数据库连接、HTTP 客户端或全局单例,否则策略虽然拆出来了,测试还是很难.

---

## 七、一句话总结

> 策略模式把可替换规则从大分支里拆出来,适合规则频繁变化的场景,但目标是管理变化,不是机械消灭所有 if-else.
