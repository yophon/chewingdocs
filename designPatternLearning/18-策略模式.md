# 策略模式:消灭 if-else,但别制造类爆炸

业务代码里最常见的坏味道之一是不断增长的 `if-else`.

```ts
if (type === "vip") {
  ...
} else if (type === "new_user") {
  ...
} else if (type === "festival") {
  ...
}
```

一开始只有两种规则,写分支没问题.后来规则越来越多,每加一种都要改同一个函数,测试也要覆盖一大坨分支.更糟的是,不同规则的参数、边界和依赖不一样,最终一个函数会变成规则垃圾桶.

变化点是:算法或业务规则会替换、扩展和组合.稳定点是:调用方只需要得到统一结果.

---

## 一、问题

以折扣计算为例:

```ts
function discount(type: string, amount: number) {
  if (type === "vip") return amount * 0.8
  if (type === "new_user") return Math.min(amount, 50)
  if (type === "none") return 0
  throw new Error("unknown discount")
}
```

这个函数的问题不是分支本身,而是它会成为所有促销规则的共同修改点.促销规则通常变化很快,共同修改点越大,冲突和回归风险越高.

---

## 二、模式

策略模式的核心是:把一组可替换的算法封装成同一接口,调用方在运行时选择一个策略执行.

结构可以简化成:

```text
Context -> Strategy
             -> ConcreteStrategyA
             -> ConcreteStrategyB
```

`Context` 负责使用策略,具体规则由不同策略实现.

---

## 三、示例

```ts
interface DiscountStrategy {
  calculate(amount: number): number
}

class VipDiscount implements DiscountStrategy {
  calculate(amount: number) {
    return amount * 0.2
  }
}

class NewUserDiscount implements DiscountStrategy {
  calculate(amount: number) {
    return Math.min(amount, 50)
  }
}

class CheckoutService {
  constructor(private strategies: Record<string, DiscountStrategy>) {}

  finalPrice(type: string, amount: number) {
    const strategy = this.strategies[type]
    if (!strategy) throw new Error("unknown discount")
    return amount - strategy.calculate(amount)
  }
}
```

新增一种折扣时,新增策略并注册即可.核心结账流程不需要继续膨胀.

---

## 四、落地判断

策略模式适合这些信号:

- 一个函数里有多种算法分支
- 分支会频繁新增,且每个分支逻辑独立
- 调用方只关心统一结果
- 希望单独测试每种规则

常见落地包括:

- 支付渠道选择
- 价格、优惠、税费计算
- 排序规则和匹配规则
- 文件导出格式
- 风控规则

在 TypeScript 里,策略不一定非要是类.函数也可以是策略:

```ts
type Discount = (amount: number) => number

const discounts: Record<string, Discount> = {
  vip: amount => amount * 0.2,
  none: () => 0
}
```

语言特性越强,模式长得越轻.

---

## 五、代价与误用

策略模式的代价是对象数量和跳转关系增加.一个简单三行分支,拆成三个类,读代码反而更慢.

常见误用有三种.

第一种是为了消灭所有 `if`.`if-else` 本身不是坏事,稳定且简单的分支直接写更清楚.

第二种是策略类爆炸.每个微小差异都建一个类,最后项目里充满 `VipStrategyV2ForApp` 这种东西.很多差异可以用参数、配置或组合策略解决.

第三种是选择逻辑仍然混乱.策略解决的是"规则怎么执行",不自动解决"选哪个规则".选择策略的注册表、配置中心或工厂也要设计清楚.

---

## 六、一句话总结

> 策略模式把可替换规则从大分支里拆出来,适合规则频繁变化的场景,但目标是管理变化,不是机械消灭所有 if-else.
