# 单例模式

单例模式看起来很简单:全局只有一个实例。真正麻烦的是,一旦一个对象变成全局唯一,它的生命周期,状态,并发,测试隔离都会变成系统问题。很多人学设计模式第一个会写单例,也是第一个把测试写崩的模式。

> 一句话先记住:单例解决的是全局唯一和统一访问,但代价是全局状态和隐藏依赖。

---

## 一、先看问题

配置中心,日志器,连接池这类对象通常不希望到处创建:

```ts
class Config {
  constructor() {
    // read env and config files
  }
}

const a = new Config();
const b = new Config();
```

如果每个模块都 new 一份,可能带来:

- 重复读取配置
- 状态不一致
- 资源浪费
- 生命周期不可控

于是你会想:能不能全局只保留一个?

---

## 二、最小单例写法

TypeScript 里可以这样写:

```ts
class Config {
  private static instance: Config | undefined;

  private constructor(private values: Record<string, string>) {}

  static getInstance() {
    if (!this.instance) {
      this.instance = new Config(process.env as Record<string, string>);
    }
    return this.instance;
  }

  get(key: string) {
    return this.values[key];
  }
}

const config = Config.getInstance();
```

结构很简单:

```text
Config
  - static instance
  - private constructor
  + static getInstance()
```

它把创建入口收起来,所有调用方只能通过 `getInstance` 拿同一个对象。

---

## 三、单例真正保护的稳定点

单例适合保护这些稳定点:

- 系统里确实只能有一个逻辑实例
- 初始化成本高,不想重复创建
- 需要统一管理生命周期
- 多处调用需要共享同一个资源入口

典型例子:

- 日志门面
- 配置读取器
- 进程内指标收集器
- 数据库连接池管理器
- 应用级事件总线

注意这里说的是"进程内"。分布式系统里,单例不等于全局唯一。多进程,多 Pod,多机器都会各有一份。

---

## 四、并发和懒加载

Java 里单例经常要考虑线程安全。推荐写法之一是静态内部类:

```java
public class Config {
  private Config() {}

  private static class Holder {
    private static final Config INSTANCE = new Config();
  }

  public static Config getInstance() {
    return Holder.INSTANCE;
  }
}
```

这种写法利用类加载保证懒加载和线程安全。

Go 里常用 `sync.Once`:

```go
var (
	instance *Config
	once     sync.Once
)

func GetConfig() *Config {
	once.Do(func() {
		instance = &Config{}
	})
	return instance
}
```

单线程语言或模块系统里,单例可能更简单。比如 Node.js 的模块缓存天然就能提供进程内单例:

```ts
export const config = new Config(process.env);
```

这比手写 `getInstance` 更直白。

---

## 五、单例和依赖注入的冲突

最危险的写法是业务代码直接调用单例:

```ts
class OrderService {
  submit(order: Order) {
    Logger.getInstance().info("submit");
  }
}
```

这会制造隐藏依赖。测试时你很难替换 Logger,也很难清理它的状态。

更好的做法是:单例可以存在,但通过依赖注入传入业务对象:

```ts
class OrderService {
  constructor(private logger: Logger) {}

  submit(order: Order) {
    this.logger.info("submit");
  }
}

const service = new OrderService(Logger.getInstance());
```

这样全局唯一的生命周期在外层管理,业务逻辑仍然可测试。

## 六、落地判断

适合使用单例的信号:

- 对象代表进程级资源,比如配置,日志,指标
- 多实例会导致资源浪费或语义错误
- 初始化需要集中控制
- 对象本身最好无业务可变状态

不适合使用单例的信号:

- 只是为了少传参数
- 对象里有用户状态,请求状态,租户状态
- 测试需要频繁替换它
- 将来可能按环境,租户,数据源创建多个实例
- 分布式全局唯一需求,却只做了进程内单例

如果你想写单例只是因为"到处都能访问很方便",大概率应该停一下。

---

## 七、代价与误用

**误用一:把单例当全局变量**

全局可访问加可变状态,会让调用顺序影响结果。测试之间互相污染,线上问题也难复现。

**误用二:把业务服务做成单例**

业务服务可以由容器按单例生命周期管理,但它不应该自己掌控全局访问入口。否则依赖关系会被隐藏。

**误用三:忽略多进程环境**

Kubernetes 里 10 个 Pod 就有 10 个单例。需要全局锁,全局计数,全局唯一 ID 时,要用 Redis,数据库,协调服务等外部机制。

**误用四:让单例负责太多事**

`AppContext.getEverything()` 这种对象会迅速变成服务定位器,所有模块都偷偷依赖它。

---

## 八、一句话总结

> 单例可以管理进程内唯一资源,但不要让它变成业务代码的隐式依赖。生命周期可以全局,依赖关系最好显式。
