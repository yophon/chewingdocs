# 装饰器模式:不改原类也能叠能力

很多功能一开始都很简单:保存订单、发送请求、读取文件.后来需求开始叠加:要打日志、要缓存、要限流、要重试、要统计耗时、要做权限校验.

最直接的做法是在原类里不断加代码.问题是,这些能力往往不是核心业务,而且组合方式很多.某个调用需要缓存和日志,另一个调用只需要日志,第三个调用需要重试和指标.如果全塞进原类,类会越来越胖.

变化点是:附加能力会增减和组合.稳定点是:核心对象的主职责不应该变.

---

## 一、问题

假设有一个读取配置的接口:

```ts
interface ConfigSource {
  get(key: string): Promise<string | undefined>
}
```

最初只有文件实现:

```ts
class FileConfigSource implements ConfigSource {
  async get(key: string) {
    return process.env[key]
  }
}
```

后来你想加缓存.如果直接改 `FileConfigSource`,它就不再只是文件配置源,还负责缓存策略.再后来要加日志和统计,这个类会继续膨胀.

---

## 二、模式

装饰器模式的核心是:装饰器和被装饰对象实现同一个接口,装饰器内部持有一个原对象,调用前后增加额外行为.

结构可以简化成:

```text
Client -> ConfigSource
              ^
              |
       CacheDecorator -> FileConfigSource
       LogDecorator   -> ConfigSource
```

因为装饰器也实现 `ConfigSource`,所以它们可以一层套一层.

---

## 三、示例

```ts
interface ConfigSource {
  get(key: string): Promise<string | undefined>
}

class EnvConfigSource implements ConfigSource {
  async get(key: string) {
    return process.env[key]
  }
}

class CachedConfigSource implements ConfigSource {
  private cache = new Map<string, string | undefined>()

  constructor(private inner: ConfigSource) {}

  async get(key: string) {
    if (this.cache.has(key)) return this.cache.get(key)
    const value = await this.inner.get(key)
    this.cache.set(key, value)
    return value
  }
}

class LoggedConfigSource implements ConfigSource {
  constructor(private inner: ConfigSource) {}

  async get(key: string) {
    console.log("read config", key)
    return this.inner.get(key)
  }
}

const source: ConfigSource = new LoggedConfigSource(
  new CachedConfigSource(new EnvConfigSource())
)
```

这里缓存和日志都没有侵入 `EnvConfigSource`.组合顺序也很清楚:先日志再缓存,或者先缓存再日志,效果可以按需要调整.

---

## 四、落地判断

装饰器适合处理横向附加能力:

- HTTP client 加重试、超时、日志、指标
- Repository 加缓存
- Reader / Writer 加压缩、加密、缓冲
- UI 组件外层加边框、权限、埋点
- Service 外层加事务、限流、审计

判断是否该用装饰器,看这个能力是否满足三个条件:

1. 它不是核心职责
2. 它需要按场景组合
3. 它可以通过同一个接口包住原对象

如果只是固定的一步业务流程,直接写在方法里更清楚.

从变化点看,装饰器关注的是"能力组合会变化".核心对象解决主问题,装饰器解决附加问题.如果这些附加能力会随着环境、租户、入口、配置而变化,装饰器比继承和一堆布尔参数更稳.

比如 HTTP client 经常会出现这样的需求:

```text
本地开发:    base client
测试环境:    log(base client)
生产环境:    metrics(retry(timeout(base client)))
后台任务:    retry(base client)
```

如果把这些能力都塞进一个 `HttpClient` 并通过参数控制,很快就会得到 `enableRetry`, `enableLog`, `cacheTtl`, `timeoutMs`, `metricName` 等配置泥潭.装饰器把每个能力拆成一个对象,组合关系放在组装处,读起来更接近真实运行链路.

---

## 五、测试与重构建议

装饰器的测试要抓住两个点:它是否保留原接口语义,以及它是否只增加自己负责的行为.

以缓存装饰器为例,可以这样测:

```ts
class MemoryConfigSource implements ConfigSource {
  calls = 0

  constructor(private values: Record<string, string>) {}

  async get(key: string) {
    this.calls++
    return this.values[key]
  }
}

async function demo() {
  const inner = new MemoryConfigSource({ theme: "dark" })
  const cached = new CachedConfigSource(inner)

  await cached.get("theme")
  await cached.get("theme")

  console.log(inner.calls) // 1
}
```

日志装饰器就不该测试缓存行为;缓存装饰器也不该关心日志格式.每个装饰器的测试越小,组合时越容易定位问题.

重构时可以从最痛的横切逻辑开始抽.例如一个方法里同时有读取配置、打日志、计时、缓存,先抽出稳定接口 `ConfigSource`,再把原实现改名成 `EnvConfigSource`,最后把缓存、日志分别包成装饰器.不要一次性把所有横切逻辑都重写,否则很容易把行为顺序改坏.

组装代码最好集中放在工厂、依赖注入容器或模块入口:

```ts
function createConfigSource(): ConfigSource {
  return new LoggedConfigSource(
    new CachedConfigSource(new EnvConfigSource())
  )
}
```

调用方看到的是 `ConfigSource`,但系统入口清楚地说明了包装顺序.

## 六、适用边界

装饰器的主要代价是调用链变长.调试时你看到的是一串包装对象,需要知道顺序才能判断行为.

装饰器适合附加能力彼此独立、可按需组合的场景.如果一个能力必须修改核心算法的每个步骤,它可能不是装饰器,而是策略、模板方法或直接重构核心流程.如果附加能力需要控制对象是否可访问,比如权限、远程访问、懒加载,代理这个名字通常更贴切.

装饰器也不适合隐藏强业务语义变化.例如 `ConfigSource.get` 明确承诺返回最新配置,缓存装饰器却返回过期值,那就不是透明增强,而是改变契约.这时要么改接口名字,要么把缓存策略暴露为明确配置.

---

## 七、代价与误用

常见误用有三种.

第一种是装饰器顺序含义不清.比如缓存应该包在日志里面还是日志包在缓存里面?这会影响是否记录缓存命中.需要在组装处写得明确.

第二种是装饰器修改接口语义.比如一个 `ConfigSource.get` 本来返回实时值,缓存装饰器却不设置过期策略,调用方可能以为拿到的是最新配置.

第三种是用继承模拟装饰器.`CachedLoggedRetryFileConfigSource` 这种类名一出现,基本就是组合维度爆炸了.

---

## 八、一句话总结

> 装饰器模式适合把缓存、日志、重试这类可叠加能力放到对象外层,让核心类保持单一职责,但要控制包装顺序和语义变化.
