# 享元模式:大量小对象的内存优化

享元模式解决的不是代码结构优雅问题,而是资源问题.

当系统里有大量相似小对象时,如果每个对象都保存完整数据,内存会被重复内容吃掉.比如文本编辑器里有百万个字符,地图里有几十万个树木图标,游戏里有大量相同贴图的子弹和粒子.

变化点是:对象数量会非常大.稳定点是:其中一部分状态是可以共享的.

---

## 一、问题

假设地图上有很多树:

```ts
type Tree = {
  x: number
  y: number
  name: string
  color: string
  texture: Image
}
```

`x`, `y` 每棵树都不同,但 `name`, `color`, `texture` 对同一种树是一样的.如果有 100000 棵松树,每个对象都保存同一份贴图引用和元数据,会产生大量重复.

享元模式会把状态拆成两类:

- 内部状态:可共享,比如树的品种、颜色、贴图
- 外部状态:不可共享,比如坐标、当前缩放、运行时位置

---

## 二、模式

享元模式的核心是:把可共享状态抽出来复用,对象实例只保存必要的外部状态或在调用时传入外部状态.

结构可以简化成:

```text
FlyweightFactory -> Flyweight(shared state)
Client supplies external state
```

享元工厂负责复用已有对象,避免重复创建.

---

## 三、示例

```ts
class TreeType {
  constructor(
    public name: string,
    public color: string,
    public texture: string
  ) {}

  draw(x: number, y: number) {
    console.log("draw", this.name, this.texture, x, y)
  }
}

class TreeTypeFactory {
  private types = new Map<string, TreeType>()

  get(name: string, color: string, texture: string) {
    const key = `${name}:${color}:${texture}`
    let type = this.types.get(key)
    if (!type) {
      type = new TreeType(name, color, texture)
      this.types.set(key, type)
    }
    return type
  }
}

class Tree {
  constructor(
    private x: number,
    private y: number,
    private type: TreeType
  ) {}

  draw() {
    this.type.draw(this.x, this.y)
  }
}
```

`TreeType` 是共享对象,`Tree` 保存每棵树自己的坐标.

## 四、落地判断

享元适合这些场景:

- 大量对象数量级很高,比如十万、百万
- 对象中有明显重复的大块状态
- 共享状态可以设计成不可变
- 性能瓶颈确实来自内存或对象创建

常见落地包括:

- 字符串常量池
- 数据库连接池里复用连接对象的思想,但连接池更接近对象池
- UI 虚拟列表复用行渲染结构
- 游戏和地图中的贴图、材质、模型复用

注意,享元不是缓存一切.它强调共享对象内部状态,并由外部传入变化状态.

---

## 五、代价与误用

享元的代价是状态边界变复杂.你必须非常清楚哪些状态可以共享,哪些状态必须外部传入.

常见误用有三种.

第一种是共享可变状态.多个对象共用同一个享元,如果享元内部可变,一个调用方修改后会影响其他调用方.

第二种是没有性能压力却提前优化.享元会增加工厂、键生成、状态拆分的复杂度.如果对象数量不大,普通对象更清楚.

第三种是 key 设计不稳定.享元工厂依赖 key 查找共享对象,key 如果遗漏字段,不同对象会错误复用;key 如果包含外部状态,又会失去共享效果.

---

## 六、一句话总结

> 享元模式通过共享大量对象的内部状态来节省内存,适合明确的高数量级场景,但前提是共享状态必须稳定且最好不可变.
