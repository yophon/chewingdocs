# 享元模式:大量小对象的内存优化

享元模式解决的不是代码结构优雅问题,而是资源问题.

当系统里有大量相似小对象时,如果每个对象都保存完整数据,内存会被重复内容吃掉.比如文本编辑器里有百万个字符,地图里有几十万个树木图标,游戏里有大量相同贴图的子弹和粒子.

变化点是:对象数量会非常大.稳定点是:其中一部分状态是可以共享的.

---

## 一、问题

假设地图上有很多树:

```ts
type Tree = {
  x: number
  y: number
  name: string
  color: string
  texture: Image
}
```

`x`, `y` 每棵树都不同,但 `name`, `color`, `texture` 对同一种树是一样的.如果有 100000 棵松树,每个对象都保存同一份贴图引用和元数据,会产生大量重复.

享元模式会把状态拆成两类:

- 内部状态:可共享,比如树的品种、颜色、贴图
- 外部状态:不可共享,比如坐标、当前缩放、运行时位置

---

## 二、模式

享元模式的核心是:把可共享状态抽出来复用,对象实例只保存必要的外部状态或在调用时传入外部状态.

结构可以简化成:

```text
FlyweightFactory -> Flyweight(shared state)
Client supplies external state
```

享元工厂负责复用已有对象,避免重复创建.

---

## 三、示例

```ts
class TreeType {
  constructor(
    public name: string,
    public color: string,
    public texture: string
  ) {}

  draw(x: number, y: number) {
    console.log("draw", this.name, this.texture, x, y)
  }
}

class TreeTypeFactory {
  private types = new Map<string, TreeType>()

  get(name: string, color: string, texture: string) {
    const key = `${name}:${color}:${texture}`
    let type = this.types.get(key)
    if (!type) {
      type = new TreeType(name, color, texture)
      this.types.set(key, type)
    }
    return type
  }
}

class Tree {
  constructor(
    private x: number,
    private y: number,
    private type: TreeType
  ) {}

  draw() {
    this.type.draw(this.x, this.y)
  }
}
```

`TreeType` 是共享对象,`Tree` 保存每棵树自己的坐标.

使用方式可以是:

```ts
const factory = new TreeTypeFactory()

const pine = factory.get("pine", "green", "pine.png")
const oak = factory.get("oak", "dark-green", "oak.png")

const trees = [
  new Tree(10, 20, pine),
  new Tree(30, 40, pine),
  new Tree(50, 60, oak)
]

trees.forEach(tree => tree.draw())
```

两棵松树共享同一个 `TreeType`,但坐标仍然分别保存在各自的 `Tree` 上.共享的是内部状态,不是整棵树.

## 四、落地判断

享元适合这些场景:

- 大量对象数量级很高,比如十万、百万
- 对象中有明显重复的大块状态
- 共享状态可以设计成不可变
- 性能瓶颈确实来自内存或对象创建

常见落地包括:

- 字符串常量池
- 数据库连接池里复用连接对象的思想,但连接池更接近对象池
- UI 虚拟列表复用行渲染结构
- 游戏和地图中的贴图、材质、模型复用

注意,享元不是缓存一切.它强调共享对象内部状态,并由外部传入变化状态.

从变化点看,享元保护的是"数量增长".当对象从几百增长到几十万时,原来不明显的重复字段会变成真实成本.享元把重复部分变成少量共享对象,把变化部分留给调用方或轻量实例.

享元和对象池也容易混淆:

```text
享元:共享不可变内部状态,减少重复数据
对象池:复用可变对象实例,减少创建销毁成本
缓存:保存计算结果或查询结果,避免重复计算/IO
```

它们都和性能有关,但解决的问题不一样.把连接池叫享元不算大错,但设计时要知道自己到底是在共享状态,还是在复用生命周期昂贵的对象.

## 五、测试与重构建议

享元的测试要覆盖两件事:相同内部状态是否复用,不同内部状态是否隔离.

```ts
const factory = new TreeTypeFactory()

const a = factory.get("pine", "green", "pine.png")
const b = factory.get("pine", "green", "pine.png")
const c = factory.get("oak", "green", "oak.png")

console.log(a === b) // true
console.log(a === c) // false
```

还要测试外部状态没有被错误放进 key.如果把坐标也放进 `TreeTypeFactory` 的 key,每棵树都会得到不同的享元,共享就失效了.反过来,如果 key 漏掉 `texture`,不同贴图的树可能错误共享同一个类型.

重构时不要凭感觉上享元.先用内存分析或对象数量统计证明重复状态确实是问题.然后按这个顺序拆:

1. 列出对象字段,区分可共享内部状态和不可共享外部状态.
2. 把内部状态放进不可变对象,比如 `TreeType`.
3. 用工厂按完整 key 复用内部状态.
4. 让原对象只保存外部状态和享元引用,或在调用时传入外部状态.

如果语言支持不可变类型或只读字段,尽量把享元做成不可变.共享对象一旦可变,问题会非常隐蔽.

---

## 六、适用边界

享元的代价是状态边界变复杂.你必须非常清楚哪些状态可以共享,哪些状态必须外部传入.

享元适合数量级足够大、重复状态足够明显、共享状态足够稳定的场景.如果对象只有几十个,享元工厂和 key 管理的复杂度可能比节省的内存更贵.如果内部状态经常变化,共享会带来一致性问题,不如保持普通对象.

享元也不应该让业务代码到处手动管理共享对象.通常由工厂、资源管理器、渲染引擎或容器统一负责创建和复用.调用方只表达"我要一棵 pine",不应该关心 Map key 怎么拼.

---

## 七、代价与误用

常见误用有三种.

第一种是共享可变状态.多个对象共用同一个享元,如果享元内部可变,一个调用方修改后会影响其他调用方.

第二种是没有性能压力却提前优化.享元会增加工厂、键生成、状态拆分的复杂度.如果对象数量不大,普通对象更清楚.

第三种是 key 设计不稳定.享元工厂依赖 key 查找共享对象,key 如果遗漏字段,不同对象会错误复用;key 如果包含外部状态,又会失去共享效果.

---

## 八、一句话总结

> 享元模式通过共享大量对象的内部状态来节省内存,适合明确的高数量级场景,但前提是共享状态必须稳定且最好不可变.
