# 组合优于继承

继承很诱人:父类写一次,子类复用,看起来干净。问题是项目一变复杂,继承常常把"复用代码"和"表达分类"绑死在一起。你本来只是想复用一段逻辑,最后却继承了一整套生命周期,状态,限制和历史包袱。

> 一句话先记住:继承表达 is-a,组合表达 has-a。大多数业务扩展需要的是组合,不是继承。

---

## 一、先看继承怎么失控

假设有一个导出器:

```ts
class Exporter {
  export(data: Data) {
    this.validate(data);
    const file = this.render(data);
    this.upload(file);
  }

  validate(data: Data) {}
  render(data: Data): Buffer {
    return Buffer.from("");
  }
  upload(file: Buffer) {}
}

class PdfExporter extends Exporter {
  render(data: Data) {
    return Buffer.from("pdf");
  }
}
```

一开始没问题。后来需求来了:

- PDF / Excel / CSV 三种格式
- 本地保存 / S3 上传 / 邮件发送三种输出
- 普通校验 / 严格校验两种校验

如果继续继承,类名会开始膨胀:

```text
StrictPdfS3Exporter
StrictPdfEmailExporter
NormalExcelS3Exporter
NormalCsvLocalExporter
...
```

这就是继承最典型的问题:多个变化维度被压进同一棵继承树。

---

## 二、组合的思路

把变化维度拆成独立能力:

```text
ExportService
  has Validator
  has Renderer
  has Output
```

代码可以这样写:

```ts
interface Validator {
  validate(data: Data): void;
}

interface Renderer {
  render(data: Data): Buffer;
}

interface Output {
  write(file: Buffer): Promise<void>;
}

class ExportService {
  constructor(
    private validator: Validator,
    private renderer: Renderer,
    private output: Output,
  ) {}

  async export(data: Data) {
    this.validator.validate(data);
    const file = this.renderer.render(data);
    await this.output.write(file);
  }
}
```

现在新增一种格式,只新增一个 `Renderer`。新增一种输出方式,只新增一个 `Output`。这些变化不再互相乘法爆炸。

---

## 三、稳定点和变化点

这个例子里:

- 稳定点:导出流程 = 校验 -> 渲染 -> 输出
- 变化点:校验规则,渲染格式,输出渠道

继承把变化点塞进同一棵树:

```text
Exporter
  +-- PdfExporter
  +-- ExcelExporter
  +-- StrictPdfExporter
  +-- S3StrictPdfExporter
```

组合把变化点拆成横向能力:

```text
ExportService
  -> Validator: Normal / Strict
  -> Renderer: Pdf / Excel / Csv
  -> Output: Local / S3 / Email
```

这就是"组合优于继承"真正想表达的东西:不是禁用继承,而是不要用一条继承链承载多个变化方向。

---

## 四、什么时候继承是合适的

继承不是原罪。下面这些场景可以用:

- 框架已经定义了生命周期钩子,比如 `BaseController`, `React.Component`
- 类型之间确实满足稳定的 is-a 关系
- 父类只提供少量稳定模板方法,子类只补变化点
- 继承层级浅,通常 1 到 2 层

例如模板方法模式:

```java
abstract class Job {
  public final void run() {
    load();
    execute();
    save();
  }

  protected abstract void execute();

  private void load() {}
  private void save() {}
}

class ImportJob extends Job {
  protected void execute() {
    // import data
  }
}
```

这里父类控制稳定流程,子类只填一个明确的变化点。这个继承关系还算克制。

## 五、组合在不同语言里的样子

TypeScript 里常用接口和对象组合:

```ts
const service = new ExportService(
  new StrictValidator(),
  new PdfRenderer(),
  new S3Output(),
);
```

Go 里常用小接口:

```go
type Renderer interface {
	Render(Data) []byte
}

type ExportService struct {
	Renderer Renderer
}
```

React / Vue 里,组合更多体现为组件组合和 hooks / composable:

```ts
function useExport(renderer: Renderer, output: Output) {
  return async (data: Data) => {
    await output.write(renderer.render(data));
  };
}
```

同一个思想,不必长成同一种类图。

## 六、落地判断

优先考虑组合的信号:

- 有两个以上独立变化维度
- 子类名字开始表达多个条件,比如 `CachedRemoteAdminService`
- 子类重写父类方法时经常需要知道父类内部状态
- 父类改一点,一批子类跟着坏
- 测试某个子类必须准备父类的一堆环境

继承还能接受的信号:

- 变化点单一,层级浅
- 父类提供稳定流程,子类只实现少量钩子
- 调用方确实可以把子类当父类使用
- 框架约束要求继承

---

## 七、代价与误用

**误用一:把组合写成参数泥潭**

如果一个对象构造函数塞了 12 个依赖,组合也会变难用。此时要重新审视职责边界,或者引入工厂 / 建造者集中装配。

**误用二:过度小接口**

每个一行方法都抽接口,会让代码跳转成本变高。接口应该围绕真实替换点,不是围绕"可能有一天"。

**误用三:组合后缺少装配规则**

组合把能力拆开了,但谁来决定 Pdf + S3 + Strict 这种组合?如果规则复杂,需要工厂或配置层,不能散落在业务代码里。

**误用四:完全否定继承**

很多框架就是靠继承或抽象类定义生命周期。关键是控制继承深度,不要让它承载多个变化维度。

---

## 八、一句话总结

> 继承适合表达稳定分类,组合适合管理变化能力。业务代码里变化维度越多,越应该优先组合。
