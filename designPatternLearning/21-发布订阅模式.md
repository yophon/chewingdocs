# 发布订阅模式:EventBus、消息队列与观察者的边界

很多系统一开始只是"下单后发短信"。后来又要加积分、优惠券、风控、数据埋点、搜索索引刷新。最直接的写法是在 `createOrder` 里不断加调用:

```ts
createOrder(input);
sendSms();
addPoints();
syncSearchIndex();
trackEvent();
```

短期看很顺手,长期会变成一个问题:订单模块知道了太多下游模块。变化点是"订单创建后谁要响应",稳定点是"订单创建这个事实已经发生"。发布订阅模式就是把这两者拆开。

## 一、问题

观察者模式里,被观察者通常直接持有观察者列表。发布订阅模式多了一层中介:发布者不认识订阅者,订阅者也不认识发布者,它们只通过主题、事件名或消息通道协作。

这层中介可以很轻,比如前端 EventBus;也可以很重,比如 Kafka、RabbitMQ、Redis Stream。关键不是工具大小,而是依赖方向变了:

```text
Publisher -> Broker/EventBus -> Subscriber
```

发布者只关心"发生了什么",不关心"谁来处理"。订阅者只关心"我对什么事件感兴趣",不关心"是谁发出来的"。

## 二、模式

发布订阅模式的角色很少:

- Publisher:发布事件
- Subscriber:订阅事件并处理
- Broker/EventBus:保存订阅关系,转发事件
- Event:描述已经发生的事实

一个最小 TypeScript 版本如下:

```ts
type Handler<T> = (event: T) => void;

class EventBus {
  private handlers = new Map<string, Handler<any>[]>();

  on<T>(topic: string, handler: Handler<T>) {
    const list = this.handlers.get(topic) ?? [];
    list.push(handler);
    this.handlers.set(topic, list);
  }

  emit<T>(topic: string, event: T) {
    for (const handler of this.handlers.get(topic) ?? []) {
      handler(event);
    }
  }
}

const bus = new EventBus();

bus.on("order.created", e => console.log("send sms", e.orderId));
bus.on("order.created", e => console.log("add points", e.userId));

bus.emit("order.created", { orderId: "O1001", userId: "U7" });
```

这个例子里,订单模块只发布 `order.created`。短信和积分逻辑是否存在、执行顺序如何、失败怎么处理,都不再由订单模块直接承担。

## 三、发布订阅和观察者的边界

这两个模式很像,但边界要讲清楚。

观察者模式更偏对象内协作:一个对象状态变化,通知它持有的一组观察者。典型例子是 UI 控件监听、领域对象监听、响应式依赖收集。

发布订阅模式更偏事件通道协作:发布者和订阅者不直接认识,中间有 broker。典型例子是 EventBus、浏览器事件、Node.js EventEmitter、消息队列、领域事件总线。

如果代码里是 `subject.addObserver(observer)`,更像观察者。如果代码里是 `bus.publish("topic", event)`,更像发布订阅。

GoF 原书重点讲的是观察者模式。发布订阅常被当成观察者的工程化变体,尤其在跨模块、跨进程、跨服务场景里更常见。

## 四、落地判断

适合使用发布订阅的信号:

- 一个事实发生后,后续动作经常增加或减少
- 发布方不应该依赖所有消费方
- 某些处理可以异步、延迟、重试
- 事件本身有审计、回放、集成价值

不一定需要发布订阅的信号:

- 只有一个调用方和一个被调用方
- 调用结果必须立刻参与主流程判断
- 处理顺序强依赖业务语义
- 失败必须立即回滚当前事务

落地时要先区分两类事件:

领域事件表示业务事实,例如 `OrderPaid`。它属于业务语言,应该稳定。

技术事件表示实现细节,例如 `CacheInvalidated`。它可以存在,但不要伪装成业务事件。

## 五、代价与误用

发布订阅最大的代价是可见性下降。直接调用时,IDE 一跳就知道谁被调用。事件发布后,你需要靠事件名、文档、日志、链路追踪才能知道后面发生了什么。

常见误用有四种:

第一,把所有函数调用都改成事件。结果是简单流程被拆散,调试困难,错误传播不清楚。

第二,事件命名太技术化,比如 `AfterSaveHook`。这让订阅者绑定到了实现细节,不是业务事实。

第三,默认异步。异步会引入顺序、重试、幂等、重复消费、最终一致性问题。不是所有事件都值得上消息队列。

第四,没有治理事件契约。事件字段随便改,消费者就会在运行时炸掉。跨服务事件尤其需要版本、兼容和监控。

## 六、一句话总结

发布订阅模式不是为了显得"解耦",而是把"一个事实发生"和"有多少响应者"分开管理。响应者经常变化时它很有用,主流程强一致时要谨慎。
