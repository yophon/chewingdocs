# 常见反模式:上帝对象、贫血模型、过度抽象、回调地狱

反模式不是"写得丑"这么简单.它指的是一种看起来能解决问题,短期也确实能跑,但长期会系统性制造维护成本的结构.很多项目不是缺设计模式,而是被反模式拖住了.

这一篇不背定义,只看代码评审里最常见的几类坏结构.

---

## 一、上帝对象:什么都知道,什么都能改

典型代码:

```ts
class OrderManager {
  createOrder() {}
  cancelOrder() {}
  payOrder() {}
  refundOrder() {}
  sendSms() {}
  updateInventory() {}
  writeAuditLog() {}
  calculatePromotion() {}
}
```

上帝对象的问题不是方法多,而是变化原因太多.支付规则变、库存规则变、短信供应商变、审计格式变,都要改它.

评审时可以问:

- 这个类有几个独立变化原因?
- 它是否直接依赖太多外部系统?
- 测试它是否必须准备半个世界?

重构方向不是机械拆小,而是按职责切:

```text
OrderService        -> 订单用例编排
PaymentGateway      -> 支付外部端口
InventoryService    -> 库存能力
PromotionPolicy     -> 优惠规则
AuditLogger         -> 审计记录
```

注意:编排类本身可以存在,但它应该协调依赖,不应该吞掉所有规则.

## 二、贫血模型:对象只有数据,规则散落外面

贫血模型常见于所有业务对象都只是 DTO:

```java
class Order {
    public String status;
    public BigDecimal amount;
}

class OrderService {
    void pay(Order order) {
        if (!order.status.equals("CREATED")) {
            throw new IllegalStateException();
        }
        order.status = "PAID";
    }
}
```

如果系统只是 CRUD,这没问题.但当规则变复杂,状态和行为分离会导致同一条规则散落在多个 service 里.

可以把不变量放回领域对象:

```java
class Order {
    private OrderStatus status;
    private BigDecimal amount;

    void pay() {
        if (status != OrderStatus.CREATED) {
            throw new IllegalStateException("only created order can be paid");
        }
        status = OrderStatus.PAID;
    }
}
```

但也别把所有东西塞进实体.调用支付网关、发消息、写数据库这些外部动作仍然属于应用服务或基础设施.

判断点是:**规则是否围绕对象自身不变量**.是,放对象里更稳;不是,放用例或服务里.

## 三、过度抽象:为不存在的变化付费

典型结构:

```text
UserRepository
UserRepositoryImpl
UserRepositoryFactory
AbstractUserRepositoryFactory
UserRepositoryProvider
```

但实际只有一个 MySQL 实现,没有测试替身,没有多数据源,也没有替换计划.

过度抽象的坏处:

- 新人要跳很多文件才能看懂一条调用链.
- 改一个字段要改接口、实现、工厂、配置.
- 抽象名字很大,业务含义很小.
- 真实变化来时,旧抽象不一定对得上.

评审时不要只问"以后会不会变",而要问:

- 这个变化是否已经发生过?
- 发生频率有多高?
- 不抽象时改动成本有多大?
- 现在抽象会不会锁死未来设计?

有时最好的抽象只是一个函数:

```ts
function normalizeUser(raw: ApiUser): User {
  return {
    id: raw.id,
    name: raw.display_name,
  };
}
```

## 四、回调地狱:控制流被切碎

回调地狱不只是缩进深,而是错误处理、取消、超时和顺序关系不清楚:

```ts
login(user, () => {
  loadProfile(() => {
    loadPermissions(() => {
      loadDashboard(() => {
        render();
      });
    });
  });
});
```

可以先用 async/await 拉平:

```ts
async function bootstrap() {
  await login();
  const profile = await loadProfile();
  const permissions = await loadPermissions(profile.id);
  const dashboard = await loadDashboard(permissions);
  render(dashboard);
}
```

如果流程有分支、重试、补偿和等待事件,普通 async 也不够,可以考虑状态机或工作流:

```text
idle -> loggingIn -> loadingProfile -> ready
                    -> failed
```

关键不是语法更优雅,而是让控制流重新可见.

## 五、落地判断:先识别损害

反模式的判断不靠个人审美,靠维护损害:

- 改一个需求要碰很多不相关文件.
- 单元测试准备成本很高.
- 一个类或函数有多个变化原因.
- 新人无法从命名理解职责.
- 错误处理路径不可见.
- 抽象层很多,但没有隔离真实变化.

修复也不要一次性推倒:

1. 先补行为测试.
2. 找出最频繁变化的区域.
3. 按变化原因拆边界.
4. 删除无意义转发层.
5. 保留能解释业务的命名.

---

## 六、一句话总结

> **反模式的共同点是短期省事,长期让变化变贵**.代码评审时不要只说"不优雅",要指出它让哪类未来改动更难、更慢或更危险.
