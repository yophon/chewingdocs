# 迭代器模式:遍历协议、惰性序列与流式处理

遍历数组很简单,一个 `for` 就够了.但真实项目里的集合不总是数组:树、分页接口、数据库游标、文件流、消息流、惰性计算序列,都可能需要"一个一个拿元素".如果调用方必须知道内部结构,遍历逻辑就会和数据结构绑死.

从变化点看,迭代器模式处理的是:集合内部结构和取数方式会变化.稳定点是:调用方只想按顺序访问元素.

---

## 一、问题场景

假设一个组织系统要遍历部门下所有员工.第一版数据结构是树:

```ts
type Employee = { id: string; name: string }

type Department = {
  name: string
  employees: Employee[]
  children: Department[]
}
```

调用方直接递归:

```ts
function printEmployees(department: Department) {
  for (const employee of department.employees) {
    console.log(employee.name)
  }

  for (const child of department.children) {
    printEmployees(child)
  }
}
```

后来数据来源变了:

- 子部门需要分页加载
- 员工来自远程接口
- 有些调用方要深度优先,有些要广度优先
- 数据量太大,不能一次性放进内存
- 遍历过程中要支持中断

如果遍历逻辑散落在调用方,每次数据结构或取数方式变化,所有调用方都要改.

迭代器模式就是把"怎么遍历"从"集合怎么存"里拆出来,给调用方一个稳定的访问协议.

---

## 二、模式结构

经典结构可以简化成:

```text
Aggregate / Iterable
  -> createIterator()

Iterator
  -> hasNext()
  -> next()

ConcreteIterator
  -> hides traversal details
```

几个角色:

- `Aggregate`:可遍历集合
- `Iterator`:迭代器接口
- `ConcreteIterator`:具体遍历逻辑
- `Client`:只依赖遍历协议,不依赖内部结构

现代语言通常内置迭代协议.在 TypeScript/JavaScript 里,同步迭代实现 `Symbol.iterator`,异步迭代实现 `Symbol.asyncIterator`.

从变化点看,迭代器承接"元素如何被取出".调用方只保留"拿到元素后做什么".

---

## 三、TypeScript 示例

先看一个同步迭代器.这个范围对象不是数组,但可以被 `for...of` 遍历:

```ts
class NumberRange implements Iterable<number> {
  constructor(private start: number, private end: number) {}

  *[Symbol.iterator]() {
    for (let i = this.start; i <= this.end; i++) {
      yield i
    }
  }
}

for (const n of new NumberRange(1, 3)) {
  console.log(n)
}
```

`yield` 让序列可以惰性产生.调用方不需要提前拿到完整数组.

再看部门树遍历.我们把深度优先遍历封装到集合内部:

```ts
type Employee = {
  id: string
  name: string
}

class Department implements Iterable<Employee> {
  constructor(
    public readonly name: string,
    private readonly employees: Employee[],
    private readonly children: Department[] = [],
  ) {}

  *[Symbol.iterator](): Iterator<Employee> {
    for (const employee of this.employees) {
      yield employee
    }

    for (const child of this.children) {
      yield* child
    }
  }
}

const root = new Department("Engineering", [
  { id: "e1", name: "Ada" },
], [
  new Department("Platform", [{ id: "e2", name: "Linus" }]),
])

for (const employee of root) {
  console.log(employee.name)
}
```

调用方看起来只是在遍历员工,不需要知道部门树怎么递归.

异步取数也可以用迭代器表达.比如分页接口:

```ts
type User = { id: string; name: string }

async function fetchUsers(page: number): Promise<User[]> {
  console.log("fetch page", page)
  return page <= 2 ? [{ id: `u${page}`, name: `User ${page}` }] : []
}

async function* listUsers(): AsyncIterable<User> {
  let page = 1

  while (true) {
    const users = await fetchUsers(page)
    if (users.length === 0) return

    yield* users
    page += 1
  }
}

for await (const user of listUsers()) {
  console.log(user.id)
}
```

调用方不关心翻页细节.它只消费一个异步序列.

## 四、适用边界

迭代器适合这些信号:

- 调用方需要遍历,但不应该知道内部结构
- 数据量大,不能一次性加载到内存
- 遍历策略可能有多种,比如 DFS / BFS
- 取数过程可能是惰性的、远程的、流式的
- 希望调用方可以自然地中断遍历

常见落地包括:

- 数据库 `ResultSet` 或 `Cursor`
- 分页 API 自动翻页
- 文件按行读取
- 日志流逐条消费
- AST 的深度优先或广度优先遍历
- 惰性序列、Stream、Generator
- 组合模式里的树结构遍历

不适合的信号:

- 数据就是一个小数组,没有隐藏复杂度
- 调用方需要频繁随机访问、按索引跳转
- 遍历过程强依赖业务上下文,很难抽成通用协议
- 取数失败、重试、事务语义比遍历本身更重要

迭代器尤其适合和组合模式、访问者模式一起出现.组合模式提供树结构,迭代器提供遍历方式,访问者提供对节点的操作.

## 五、误用

迭代器的代价是控制流变得间接.尤其是惰性迭代,代码看起来没执行,实际要等消费时才执行.

常见误用有五种.

第一,在迭代过程中修改底层集合.很多语言会 fail-fast,有些语言不会,但都容易产生边界 bug.

第二,把迭代器做成只能用一次,却没有说明.流式迭代器通常消费后不可重放,调用方需要知道.

第三,隐藏了昂贵 IO.`for...of` 看起来很轻,但每次 `next` 可能都在查数据库或发网络请求,必须有监控、批量策略和取消机制.

第四,过度封装简单数组.没有变化点时,直接用数组方法更清楚.

第五,把业务过滤、权限判断、状态修改都塞进迭代器.迭代器应该主要负责取元素,复杂业务处理可以放在调用方、管道或访问者里.

## 六、测试与重构建议

测试迭代器时,要关注遍历结果、顺序、惰性和边界.

同步迭代器可以直接展开成数组:

```ts
it("iterates department employees depth first", () => {
  const root = new Department("root", [
    { id: "e1", name: "A" },
  ], [
    new Department("child", [{ id: "e2", name: "B" }]),
  ])

  expect([...root].map(employee => employee.id)).toEqual(["e1", "e2"])
})
```

异步迭代器可以写一个收集工具:

```ts
async function collect<T>(iterable: AsyncIterable<T>): Promise<T[]> {
  const result: T[] = []

  for await (const item of iterable) {
    result.push(item)
  }

  return result
}
```

重构建议:

1. 先找出散落在调用方的重复遍历逻辑
2. 明确调用方真正需要的是顺序访问、过滤、映射还是随机访问
3. 先实现最小遍历协议,不要马上加入复杂查询能力
4. 对树结构补顺序测试,对分页结构补停止条件测试
5. 对流式迭代器写清楚是否可重放、是否会触发 IO、如何取消

如果迭代器背后有数据库或远程 API,测试里要避免真的打外部服务.用 fake 数据源模拟分页、空页、错误页和中断消费,比只测正常路径更有价值.

---

## 七、一句话总结

> 迭代器模式把"怎么遍历"和"集合怎么存"分开,在树、流、分页、惰性序列里很有用,但不要为了包装一个普通数组而制造抽象.
