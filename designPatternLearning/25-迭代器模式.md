# 迭代器模式:遍历协议、惰性序列与流式处理

遍历数组很简单,一个 `for` 就够了。但真实项目里的集合不总是数组:树、分页接口、数据库游标、文件流、消息流、惰性计算序列,都可能需要"一个一个拿元素"。如果调用方必须知道内部结构,遍历逻辑就会和数据结构绑死。

变化点是"集合内部结构和取数方式会变化"。稳定点是"调用方只想按顺序访问元素"。迭代器模式就是把遍历方式从集合内部拆出来。

## 一、问题

假设一个部门树要遍历员工。如果调用方直接递归树结构,以后树改成分页加载、缓存加载、远程加载,所有遍历代码都要改。

更好的方式是让集合提供统一遍历协议:

```text
Collection -> Iterator -> next()
```

调用方只依赖 `next`,不依赖底层是数组、树还是流。

## 二、模式

角色:

- Aggregate/Iterable:可遍历集合
- Iterator:迭代器接口
- ConcreteIterator:具体遍历逻辑

TypeScript 里已经内置迭代协议,实现 `Symbol.iterator` 就可以被 `for...of` 使用:

```ts
class NumberRange implements Iterable<number> {
  constructor(private start: number, private end: number) {}

  *[Symbol.iterator]() {
    for (let i = this.start; i <= this.end; i++) {
      yield i;
    }
  }
}

for (const n of new NumberRange(1, 3)) {
  console.log(n);
}
```

这里的 `yield` 让序列可以惰性产生。调用方不需要提前拿到完整数组。

Java 里也有同样思想:

```java
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    System.out.println(it.next());
}
```

语言内置迭代协议以后,经典 GoF 迭代器常常变成语言特性,但思想没有消失。

## 三、落地形态

迭代器不只是遍历集合。它还经常承担"隐藏取数复杂度"的责任:

- 数据库 ResultSet 或 Cursor
- 分页 API 自动翻页
- 文件按行读取
- 日志流逐条消费
- AST 的深度优先或广度优先遍历
- 惰性序列、Stream、Generator

例如分页接口可以包装成异步迭代器:

```ts
async function* listUsers() {
  let page = 1;
  while (true) {
    const users = await fetchUsers(page);
    if (users.length === 0) return;
    yield* users;
    page++;
  }
}

for await (const user of listUsers()) {
  console.log(user.id);
}
```

调用方看起来是在遍历用户,不需要关心翻页细节。

## 四、落地判断

适合使用迭代器的信号:

- 调用方需要遍历,但不应该知道内部结构
- 数据量大,不能一次性加载到内存
- 遍历策略可能有多种,比如 DFS / BFS
- 取数过程可能是惰性的、远程的、流式的

不适合的信号:

- 数据就是一个小数组,没有隐藏复杂度
- 调用方需要随机访问、按索引跳转
- 遍历过程强依赖业务上下文,很难抽成通用协议

迭代器尤其适合和组合模式、访问者模式一起出现。组合模式提供树结构,迭代器提供遍历方式,访问者提供对节点的操作。

## 五、代价与误用

迭代器的代价是控制流变得间接。尤其是惰性迭代,代码看起来没执行,实际要等消费时才执行。

常见误用:

第一,在迭代过程中修改底层集合。很多语言会 fail-fast,有些语言不会,但都容易产生边界 bug。

第二,把迭代器做成只能用一次,却没有说明。流式迭代器通常消费后不可重放,调用方需要知道。

第三,隐藏了昂贵 IO。`for...of` 看起来很轻,但每次 `next` 可能都在查数据库或发网络请求,必须有监控和批量策略。

第四,过度封装简单数组。没有变化点时,直接用数组方法更清楚。

## 六、一句话总结

迭代器模式把"怎么遍历"和"集合怎么存"分开。它在树、流、分页、惰性序列里很有用,但不要为了包装一个普通数组而制造抽象。
