# 响应式模式:Observable、Signals、Computed 与副作用

界面和客户端程序里,最烦人的代码往往不是计算本身,而是"数据变了以后谁要跟着更新"。用户改了数量,总价要变;总价变了,优惠提示要变;筛选条件变了,列表要刷新;权限变了,按钮要禁用。如果每个地方都手动调用更新函数,很快就会漏掉某条路径。

响应式模式要解决的问题是:**声明数据之间的依赖关系,让依赖在数据变化时自动传播**。

---

## 一、问题:手动同步状态容易漏

没有响应式时,你可能会写:

```ts
let price = 100;
let quantity = 1;
let total = price * quantity;

function setQuantity(next: number) {
  quantity = next;
  total = price * quantity;
  updateTotalText(total);
  updateDiscountText(total);
  updateSubmitButton(total);
}
```

这段代码的问题是更新路径靠人记。以后新增一个 shipping fee、coupon、tax,每个 setter 都要补一遍。变化点是依赖关系不断增加,稳定点是某些值可以由其他值推导出来。

响应式模式会让你声明:

```text
quantity ----\
              -> total -> discountText
price --------/
```

当 `quantity` 或 `price` 变化时,`total` 自动重新计算,依赖 `total` 的地方再更新。

---

## 二、模式:Observable、Computed、Effect

不同框架名字不同,但核心角色接近:

```text
Signal / Observable
  可被观察的源状态

Computed / Derived
  从源状态推导出来的值

Effect / Reaction
  状态变化后执行副作用
```

一个简化 TypeScript 伪代码:

```ts
const price = signal(100);
const quantity = signal(1);

const total = computed(() => price.get() * quantity.get());

effect(() => {
  console.log("total changed", total.get());
});

quantity.set(2); // total 变成 200,effect 自动执行
```

RxJS Observable 风格更偏事件流:

```ts
const quantity$ = new BehaviorSubject(1);
const price$ = new BehaviorSubject(100);

const total$ = combineLatest([price$, quantity$]).pipe(
  map(([price, quantity]) => price * quantity),
);

const sub = total$.subscribe(total => {
  console.log(total);
});
```

Signals 更像细粒度状态依赖;Observable 更像随时间发生的值流。两者都属于响应式思想,但适用场景不同。

---

## 三、落地判断:Signals、Observable、Computed 怎么选

适合 Computed / Signals 的场景:

- UI 状态之间有同步推导关系.
- 派生值可以由当前状态直接计算.
- 希望细粒度更新,避免整棵组件树重算.
- 典型例子:表单校验、价格计算、筛选结果、权限派生.

适合 Observable / Stream 的场景:

- 数据随时间连续到达.
- 需要组合、节流、防抖、取消、重试.
- 典型例子:搜索输入流、WebSocket、拖拽、实时数据、事件管道.

不一定需要响应式的场景:

- 一次性计算,没有持续依赖.
- 状态变化路径很少,普通函数更清楚.
- 业务流程需要明确步骤和事务边界.

响应式不是业务架构的替代品。它擅长处理"值变了谁更新",不擅长表达"下单必须先校验库存再创建支付单"这种用例流程。

---

## 四、副作用:响应式里最容易失控的部分

Computed 应该保持纯粹:

```ts
const total = computed(() => price.get() * quantity.get());
```

副作用应该放在 effect 或订阅里:

```ts
effect(() => {
  analytics.track("total_changed", { total: total.get() });
});
```

但 effect 不能滥用。太多 effect 会形成隐式流程:

```text
effect A 写入 state B
effect B 监听 state B 再写入 state C
effect C 调接口后写入 state A
```

这会变成响应式版的回调地狱。复杂业务流程更适合显式 use case、action 或 command,而不是靠 effect 串起来。

---

## 五、代价与误用

响应式模式的代价是依赖变隐式。代码看起来少了,但运行时谁依赖谁需要工具或经验才能看清。尤其是自动依赖收集的系统,一个 computed 里多读了一个 signal,就多了一条更新路径。

常见误用之一是把所有值都做成响应式。常量、临时变量、函数局部计算不需要 signal。响应式状态越多,依赖图越复杂。

常见误用之二是在 computed 里做副作用。computed 应该是可缓存、可重复执行、无副作用的推导。如果里面发请求、写状态、打日志,调试会很痛。

常见误用之三是忘记取消订阅。Observable 如果没有清理,组件销毁后仍然响应事件,可能造成内存泄漏和重复请求。

```ts
const sub = total$.subscribe(render);

onUnmount(() => {
  sub.unsubscribe();
});
```

---

## 六、与前端和应用架构的边界

响应式模式主要是前端和客户端状态传播模式,也会出现在后端流处理里。但它不是分层架构、Clean Architecture 或 DDD 的替代品。后者关心业务边界和依赖方向,响应式关心数据依赖和更新传播。

它和单向数据流也不是对立关系。你可以用 action 统一修改入口,再用 computed/signals 派生视图状态。关键是分清:

- 单向数据流管"谁能改状态".
- 响应式管"状态变了谁自动更新".
- 组件模式管"界面如何拆分组合".

---

## 一句话总结

响应式模式用 Observable、Signals、Computed 把数据依赖声明出来,适合 UI 派生状态和事件流,但副作用和隐式依赖一多,它也会从自动更新变成难追踪的隐形流程。
