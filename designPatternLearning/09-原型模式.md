# 原型模式

有些对象创建不难,但初始化很贵。有些对象大部分字段相同,只改少数几个值。有些业务需要从模板快速生成实例。每次从零构造都能做,但会重复大量初始化逻辑。

原型模式解决的是:通过复制已有对象来创建新对象。

> 一句话先记住:原型模式适合"从一个已配置好的样板复制出新对象",重点是复制边界和可变状态。

---

## 一、先看问题

假设有一个报表配置:

```ts
const report = new ReportConfig();
report.setTheme("dark");
report.setColumns(["date", "amount", "status"]);
report.setFilters(defaultFilters());
report.setExporter(new PdfExporter());
```

现在不同用户只改几个字段:

```ts
const userReport = new ReportConfig();
userReport.setTheme("dark");
userReport.setColumns(["date", "amount", "status"]);
userReport.setFilters(defaultFilters());
userReport.setExporter(new PdfExporter());
userReport.setOwner(userId);
```

重复创建的问题:

- 初始化步骤复制
- 默认值散落
- 改模板时容易漏改
- 创建成本可能很高

这时可以把已配置对象当作原型。

---

## 二、最小原型写法

```ts
class ReportConfig {
  constructor(
    public theme: string,
    public columns: string[],
    public filters: Filter[],
    public owner?: string,
  ) {}

  clone() {
    return new ReportConfig(
      this.theme,
      [...this.columns],
      this.filters.map(filter => ({ ...filter })),
      this.owner,
    );
  }
}

const defaultReport = new ReportConfig("dark", ["date", "amount"], []);

const userReport = defaultReport.clone();
userReport.owner = "user-1";
```

结构很简单:

```text
Prototype object -> clone() -> New object
```

调用方不需要知道默认配置怎么一步步构造,只要复制并修改差异。

## 三、浅拷贝和深拷贝

原型模式最大的坑是复制边界。

浅拷贝:

```ts
const copy = { ...template };
```

它只复制第一层。如果里面有数组或对象,仍然共享引用:

```ts
const a = { columns: ["date"] };
const b = { ...a };
b.columns.push("amount");

console.log(a.columns); // ["date", "amount"]
```

深拷贝会复制嵌套对象,但成本更高,也不一定能正确复制函数,类实例,连接句柄等资源。

原型模式落地时必须明确:

- 哪些字段共享是安全的
- 哪些字段必须复制
- 哪些字段不能复制,必须重新创建
- 复制后对象是否允许修改

## 四、原型注册表

如果系统里有多种模板,可以用注册表管理:

```ts
class ReportPrototypeRegistry {
  private prototypes = new Map<string, ReportConfig>();

  register(name: string, prototype: ReportConfig) {
    this.prototypes.set(name, prototype);
  }

  create(name: string) {
    const prototype = this.prototypes.get(name);
    if (!prototype) throw new Error("unknown prototype");
    return prototype.clone();
  }
}
```

调用:

```ts
registry.register("finance", financeReport);
registry.register("sales", salesReport);

const report = registry.create("finance");
report.owner = "user-1";
```

这和工厂有点像。区别是:工厂通常通过构造逻辑创建对象,原型注册表通过复制样板对象创建对象。

## 五、真实场景

原型模式在业务里常见但不一定叫这个名字:

- 表单模板复制成用户表单
- 游戏里复制怪物配置
- 工作流从流程模板创建流程实例
- 文档从模板创建草稿
- 测试里复制默认 fixture,只改少数字段
- 前端状态里基于当前配置创建新版本

测试数据里尤其常见:

```ts
const defaultUser = {
  name: "Alice",
  role: "member",
  active: true,
};

const admin = {
  ...defaultUser,
  role: "admin",
};
```

这就是轻量版原型。

## 六、落地判断

适合原型的信号:

- 对象初始化成本高
- 多个对象大部分字段相同
- 系统有明确模板概念
- 创建逻辑复杂但复制后只改少数字段
- 需要保留某一时刻的快照再派生

不适合原型的信号:

- 对象持有数据库连接,文件句柄,锁等不可复制资源
- 对象内部可变引用很多,复制边界难讲清
- 创建逻辑本身很简单
- 复制后需要大幅修改,不如重新构造

---

## 七、代价与误用

**误用一:以为 clone 一定安全**

默认浅拷贝很容易共享内部数组和对象。线上出现"改了 A 却影响 B",常常就是复制边界没处理好。

**误用二:复制资源对象**

连接池,线程池,文件句柄这类资源不应该 clone。新对象应该重新获取资源引用,或者只复制配置。

**误用三:模板对象被意外修改**

原型最好是不可变的,或者放在注册表里不暴露可变引用。否则模板被改,所有后续创建都会变。

**误用四:用深拷贝逃避建模**

`JSON.parse(JSON.stringify(obj))` 看起来简单,但会丢失 Date,Map,undefined,方法和类实例。复杂对象要显式定义复制规则。

---

## 八、一句话总结

> 原型模式的价值在于复用已配置好的样板对象,但成败取决于你是否清楚哪些状态该复制、共享或重新创建。
