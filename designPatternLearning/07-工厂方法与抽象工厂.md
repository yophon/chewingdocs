# 工厂方法与抽象工厂

对象创建一开始通常很简单:直接 `new`。但当创建逻辑依赖配置,环境,租户,渠道,平台版本时,`new` 就不再只是语法,而是一段业务规则。如果这些规则散落在各个服务里,修改一次创建逻辑就会牵动一片。

工厂模式解决的就是:把创建逻辑从业务流程里拔出来。

> 一句话先记住:工厂不是为了少写 new,而是为了集中管理"创建哪一个"和"怎么创建"。

---

## 一、先看问题

```ts
class PaymentService {
  async pay(order: Order, channel: string) {
    let provider: PaymentProvider;

    if (channel === "alipay") {
      provider = new AlipayProvider(config.alipayKey);
    } else if (channel === "wechat") {
      provider = new WechatProvider(config.appId, config.secret);
    } else {
      throw new Error("unsupported channel");
    }

    return provider.charge(order);
  }
}
```

问题不在 if-else 只有两行,而在创建细节进入了业务流程:

- 支付服务知道每个渠道的构造参数
- 新增渠道要改支付服务
- 测试支付流程时必须关心具体 provider 怎么建
- 创建规则无法复用

---

## 二、简单工厂:先集中创建逻辑

```ts
class PaymentProviderFactory {
  create(channel: string): PaymentProvider {
    if (channel === "alipay") {
      return new AlipayProvider(config.alipayKey);
    }
    if (channel === "wechat") {
      return new WechatProvider(config.appId, config.secret);
    }
    throw new Error("unsupported channel");
  }
}

class PaymentService {
  constructor(private factory: PaymentProviderFactory) {}

  async pay(order: Order, channel: string) {
    const provider = this.factory.create(channel);
    return provider.charge(order);
  }
}
```

这通常已经够用。严格 GoF 里"简单工厂"不算一个正式模式,但工程里非常常见。

结构是:

```text
PaymentService -> PaymentProviderFactory -> PaymentProvider
                                      +-- AlipayProvider
                                      +-- WechatProvider
```

---

## 三、工厂方法:把创建延迟到子类

当一套流程稳定,但某个产品创建由子类决定时,可以用工厂方法。

```java
abstract class ReportJob {
  public final void run() {
    Renderer renderer = createRenderer();
    byte[] file = renderer.render();
    upload(file);
  }

  protected abstract Renderer createRenderer();

  private void upload(byte[] file) {}
}

class PdfReportJob extends ReportJob {
  protected Renderer createRenderer() {
    return new PdfRenderer();
  }
}

class ExcelReportJob extends ReportJob {
  protected Renderer createRenderer() {
    return new ExcelRenderer();
  }
}
```

稳定点是 `run` 流程。变化点是创建哪种 `Renderer`。父类不直接 new,而是调用工厂方法。

现代业务代码里,工厂方法不一定非要用继承,也可以用函数参数:

```ts
class ReportJob {
  constructor(private createRenderer: () => Renderer) {}

  run() {
    const renderer = this.createRenderer();
    return renderer.render();
  }
}
```

## 四、抽象工厂:创建一组相关对象

如果变化的不是一个对象,而是一整套对象族,就会用到抽象工厂。

比如不同 UI 平台需要一组匹配组件:

```ts
interface UIFactory {
  createButton(): Button;
  createDialog(): Dialog;
}

class WebUIFactory implements UIFactory {
  createButton() {
    return new WebButton();
  }
  createDialog() {
    return new WebDialog();
  }
}

class MobileUIFactory implements UIFactory {
  createButton() {
    return new MobileButton();
  }
  createDialog() {
    return new MobileDialog();
  }
}
```

调用方只依赖一组抽象产品:

```ts
class SettingsPage {
  constructor(private ui: UIFactory) {}

  render() {
    const dialog = this.ui.createDialog();
    const button = this.ui.createButton();
    return dialog.with(button);
  }
}
```

抽象工厂保护的是"产品族一致性":Web 的 Button 应该配 Web 的 Dialog,Mobile 的 Button 应该配 Mobile 的 Dialog。

## 五、工厂方法和抽象工厂怎么选

| 场景 | 更适合 |
| --- | --- |
| 根据一个条件创建一个对象 | 简单工厂 |
| 稳定流程里留一个创建钩子 | 工厂方法 |
| 需要创建一组相关对象 | 抽象工厂 |
| 创建参数很多且要分步设置 | 建造者 |
| 只是替换依赖方便测试 | 依赖注入 |

很多项目里,简单工厂加依赖注入就够了。不要为了贴近 GoF 定义强行引入抽象工厂。

---

## 六、落地判断

应该引入工厂的信号:

- `new` 前面有大量条件判断
- 创建对象需要读取配置,环境,租户信息
- 同一种创建规则在多个地方复制
- 创建出来的对象必须满足一组配套关系
- 业务流程不该知道具体实现类

可以直接 new 的信号:

- 对象是简单值对象,没有复杂依赖
- 创建逻辑没有变化
- 只有一个调用点
- 抽工厂后只是把一行 `new` 搬到另一行

---

## 七、代价与误用

**误用一:所有 new 都进工厂**

`new Money(100, "CNY")` 这种值对象直接创建就很好。工厂应该服务复杂创建规则,不是消灭 new。

**误用二:工厂变成上帝对象**

一个 `AppFactory` 创建所有东西,会把系统依赖集中成一团。工厂也应该按业务边界拆分。

**误用三:抽象工厂过早**

如果现在只有 Web 一套产品族,未来是否有 Mobile 还不确定,抽象工厂大概率过度设计。

**误用四:把选择逻辑散在工厂外**

如果调用方先判断类型,再调用不同工厂方法,说明创建规则并没有真正集中。

---

## 八、一句话总结

> 工厂模式的重点不是创建对象,而是把创建规则集中到稳定位置,让业务流程不被具体实现和构造细节污染。
