# 状态模式:把状态机从 if-else 里救出来

订单、工单、审批、支付、直播间、连接会话,都逃不开状态。刚开始状态只有两个:`pending` 和 `done`,一个 `if` 就够了。后来加上 `paid`、`shipped`、`cancelled`、`refunding`、`refunded`,每个操作都要判断当前状态,代码很快变成这样:

```ts
if (order.status === "created") { ... }
else if (order.status === "paid") { ... }
else if (order.status === "cancelled") { ... }
```

真正的问题不是 if 多,而是状态规则散落在各个方法里。变化点是"不同状态下行为不同,状态转换规则会变化"。稳定点是"对象对外暴露的操作基本稳定"。

## 一、问题

以订单为例,外部可能一直调用 `pay`、`cancel`、`ship`。但是这些操作在不同状态下含义不同:

- created 可以 pay,也可以 cancel
- paid 可以 ship,通常不能直接 cancel
- shipped 不能 pay,也不能普通 cancel

如果每个方法都自己判断状态,新增一个状态就要全局搜索。状态模式把每个状态下的行为收拢到对应的状态对象里。

## 二、模式

状态模式的角色:

- Context:持有当前状态,对外暴露业务操作
- State:状态接口
- ConcreteState:具体状态,决定当前状态下如何响应操作,以及是否切换状态

TypeScript 示例:

```ts
interface OrderState {
  pay(order: Order): void;
  cancel(order: Order): void;
}

class Order {
  constructor(public state: OrderState) {}

  pay() {
    this.state.pay(this);
  }

  cancel() {
    this.state.cancel(this);
  }
}

class CreatedState implements OrderState {
  pay(order: Order) {
    console.log("paid");
    order.state = new PaidState();
  }

  cancel(order: Order) {
    console.log("cancelled");
    order.state = new CancelledState();
  }
}

class PaidState implements OrderState {
  pay() {
    throw new Error("order already paid");
  }

  cancel() {
    throw new Error("paid order needs refund flow");
  }
}

class CancelledState implements OrderState {
  pay() {
    throw new Error("cancelled order cannot be paid");
  }

  cancel() {
    throw new Error("order already cancelled");
  }
}
```

状态对象让规则集中起来。看到 `PaidState`,就能知道已支付状态下允许什么、不允许什么。

## 三、状态模式和策略模式的边界

状态模式和策略模式结构相似,但意图不同。

策略模式是外部选择算法,对象通常不知道策略为什么变化。比如选择满减、折扣、会员价。

状态模式是对象内部状态变化导致行为变化。状态对象通常会推动下一个状态。比如订单从 created 变成 paid。

如果"切换"来自用户配置,更像策略。如果"切换"来自生命周期推进,更像状态。

状态模式也不是状态机的全部。复杂业务通常还需要显式状态表、转移表、守卫条件、动作和审计记录。状态模式适合把状态相关行为组织起来,但不自动解决流程治理。

## 四、落地判断

适合使用状态模式的信号:

- 一个对象有明确生命周期
- 同一个操作在不同状态下行为不同
- 状态判断散落在多个方法中
- 新增状态经常导致多处修改

不一定需要的信号:

- 状态只有两个,规则稳定
- 状态只用于展示文案
- 状态转换完全由数据库配置或流程引擎管理
- 业务更需要可视化流程图而不是代码内对象

在后端系统里,状态模式常和状态字段、事务、事件一起出现。状态对象负责规则,数据库负责事实,事件负责对外通知。不要只改内存状态而忘了持久化一致性。

## 五、代价与误用

状态模式的代价是对象数量增加,并且状态切换路径可能不如一张表直观。状态很多时,只靠类继承会让人迷路。

常见误用:

第一,状态不稳定时过早抽象。需求还在探索阶段,状态随时重命名、合并、删除,先用清晰的转移表可能更好。

第二,把跨状态公共逻辑复制到每个状态类。公共校验、审计、锁、事务应该放在 Context 或应用服务层。

第三,遗漏非法转移。状态模式不能只写 happy path,每个状态下不允许的操作也要明确处理。

第四,把状态模式当流程引擎。审批流、工作流、长事务补偿、人工节点,通常需要更完整的流程模型。

## 六、一句话总结

状态模式适合生命周期明确、状态行为差异明显的对象。它不是为了消灭所有 if,而是把状态规则从散乱判断收拢到可维护的位置。
