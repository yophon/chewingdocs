# 代理模式:远程代理,缓存代理,权限代理与 AOP

有些对象不能直接访问,或者直接访问会带来额外问题.

远程服务要处理网络失败,大文件要延迟加载,敏感接口要先做权限判断,昂贵查询要缓存结果.你当然可以在每个调用点都写这些逻辑,但调用点越多,重复越多,遗漏也越多.

变化点是:访问对象之前和之后的控制逻辑会变化.稳定点是:调用方希望像使用真实对象一样使用它.

---

## 一、问题

假设订单服务里有一个查询接口:

```ts
interface OrderService {
  getOrder(id: string): Promise<Order>
}
```

业务代码只想查订单.但真实环境里,这个查询可能需要:

- 判断当前用户有没有权限
- 走远程 HTTP 调用
- 缓存热点订单
- 记录访问日志

如果这些逻辑散在每个调用点,权限漏一次就是线上事故.

---

## 二、模式

代理模式的核心是:代理对象和真实对象实现同一个接口,调用方访问代理,代理再决定是否访问真实对象,以及如何访问.

结构可以简化成:

```text
Client -> Proxy -> RealSubject
```

代理和装饰器很像,区别在意图上:

- 装饰器强调叠加能力
- 代理强调控制访问

实际项目里二者会重叠,不必为了名字争太久,关键是看它保护的变化点是什么.

---

## 三、示例

```ts
type Order = { id: string; ownerId: string }

interface OrderService {
  getOrder(id: string): Promise<Order>
}

class RemoteOrderService implements OrderService {
  async getOrder(id: string): Promise<Order> {
    return { id, ownerId: "u1" }
  }
}

class PermissionOrderProxy implements OrderService {
  constructor(
    private real: OrderService,
    private currentUserId: string
  ) {}

  async getOrder(id: string): Promise<Order> {
    const order = await this.real.getOrder(id)
    if (order.ownerId !== this.currentUserId) {
      throw new Error("forbidden")
    }
    return order
  }
}
```

调用方依赖的仍然是 `OrderService`,但访问已经被代理控制了.

## 四、落地判断

代理常见形态很多:

- 远程代理:本地对象代表远程服务,比如 RPC stub
- 虚拟代理:真正需要时才加载大对象,比如图片懒加载
- 保护代理:访问前做权限校验
- 缓存代理:重复请求直接返回缓存
- AOP 代理:在方法调用前后织入事务、日志、指标

Spring 里的事务代理就是典型例子.业务方法看起来只是普通方法,但调用进入代理后,框架会开启事务、提交或回滚.

判断是否要用代理,看你是否需要统一控制"访问过程",而不是只是给对象加一个小能力.如果是缓存、日志、重试这种可组合能力,装饰器也能解决.如果是权限、远程访问、生命周期控制,代理这个名字更贴切.

从变化点看,代理保护的是访问路径.真实对象可以保持纯粹,调用方也保持简单,变化集中在代理层.比如权限规则从"只能看自己的订单"变成"管理员可以看所有订单",改的是保护代理;远程服务从 HTTP 切到 RPC,改的是远程代理;热点数据需要缓存,改的是缓存代理.

代理也常用于延迟创建昂贵对象:

```ts
interface Report {
  render(): Promise<string>
}

class HeavyReport implements Report {
  constructor(private id: string) {
    console.log("load report template and data", id)
  }

  async render() {
    return `<html>report ${this.id}</html>`
  }
}

class LazyReportProxy implements Report {
  private real?: HeavyReport

  constructor(private id: string) {}

  async render() {
    if (!this.real) {
      this.real = new HeavyReport(this.id)
    }
    return this.real.render()
  }
}
```

调用方拿到的仍是 `Report`,但真正昂贵的加载被推迟到 `render` 时.这就是虚拟代理的价值.

## 五、测试与重构建议

代理测试要明确"访问控制"是否生效.不要只测正常路径,还要测代理拒绝访问、缓存命中、远程失败、懒加载只发生一次等边界.

权限代理可以这样测:

```ts
class FakeOrderService implements OrderService {
  constructor(private order: Order) {}

  async getOrder(id: string) {
    return this.order
  }
}

async function demo() {
  const real = new FakeOrderService({ id: "o1", ownerId: "u2" })
  const proxy = new PermissionOrderProxy(real, "u1")

  await proxy.getOrder("o1") // should throw forbidden
}
```

重构时,先找到重复出现的访问前后逻辑.如果多个调用点都在做同样的权限判断、懒加载、远程错误翻译,就可以抽出同接口代理.抽完后要保证调用方依赖的是接口,而不是具体实现,否则代理很难插进去.

在 AOP 或框架代理里,还要专门加集成测试验证代理确实生效.例如事务代理常见问题是类内部自调用绕过代理,单元测试可能看不出来,需要从容器里拿到对象后调用公开方法来验证.

## 六、适用边界

代理适合"访问真实对象之前必须做决定"的场景:能不能访问、何时访问、访问哪一个远程实例、是否复用已有结果.如果你只是想给返回值做格式转换,更像适配器.如果你只是给调用链叠加日志和指标,装饰器也足够.

代理还要控制透明性的边界.远程代理虽然像本地对象,但网络失败、超时、重试、幂等都不是本地调用的语义.好的代理会隐藏协议细节,但不应该隐藏分布式系统事实.比如接口返回 `Promise`、暴露超时配置、错误类型可识别,都比假装一切只是普通函数调用更诚实.

---

## 七、代价与误用

代理的代价是行为不再完全透明.代码上看是普通方法调用,实际可能发生网络请求、事务提交、权限判断、缓存命中.这会增加调试成本.

常见误用有三种.

第一种是隐藏太多副作用.一个看起来很轻的方法,代理里却发了多个远程请求,调用方很难估算性能.

第二种是代理绕过问题.在 AOP 框架里,同一个类内部自调用可能绕过代理,导致事务或权限不生效.这不是模式问题,是代理机制的边界.

第三种是把业务规则放进通用代理.权限代理可以判断是否允许访问,但复杂业务审批规则最好仍放在领域服务或应用服务里.

---

## 八、一句话总结

> 代理模式用一个同接口对象站在真实对象前面,把权限、远程调用、缓存、事务这类访问控制集中起来,代价是调用行为会变得不那么直观.
