# 观察者模式:事件,订阅,响应式与内存泄漏

很多对象之间不是"你调用我,我立刻返回"的关系,而是"某件事发生后,很多地方都要响应".

用户注册成功后,可能要发欢迎邮件、送优惠券、写审计日志、通知 CRM.最差的写法是把这些动作都塞进注册函数里:

```ts
await userRepo.create(user)
await mail.sendWelcome(user)
await coupon.send(user)
await audit.log(user)
await crm.sync(user)
```

这样注册流程会越来越重,每新增一个响应动作都要改核心注册代码.

变化点是:事件发生后的响应方会变化.稳定点是:事件本身和发布时机相对稳定.

---

## 一、问题

直接调用的问题有两个.

第一,发布者知道太多订阅者.注册服务本来只负责创建用户,现在还知道邮件、优惠券、CRM.

第二,响应动作之间相互影响.邮件失败是否应该导致注册失败?CRM 慢是否应该拖慢接口?这些问题如果都塞在主流程里,边界会很难讲清楚.

观察者模式让发布者只发布事件,订阅者自己决定如何响应.

---

## 二、模式

观察者模式的核心是:被观察者维护一组观察者,状态变化或事件发生时通知它们.

结构可以简化成:

```text
Subject -> Observer[]
Subject.notify(event)
```

现代工程里,它常以事件监听器、回调、响应式流、状态订阅的形式出现.

---

## 三、示例

```ts
type UserCreated = { userId: string; email: string }
type Handler<T> = (event: T) => Promise<void> | void

class EventBus {
  private handlers: Handler<any>[] = []

  subscribe<T>(handler: Handler<T>) {
    this.handlers.push(handler as Handler<any>)
    return () => {
      this.handlers = this.handlers.filter(h => h !== handler)
    }
  }

  async publish<T>(event: T) {
    for (const handler of this.handlers) {
      await handler(event)
    }
  }
}

const bus = new EventBus()

bus.subscribe<UserCreated>(async event => {
  console.log("send welcome mail", event.email)
})

await bus.publish<UserCreated>({ userId: "u1", email: "a@example.com" })
```

`subscribe` 返回取消订阅函数,这在前端组件和长生命周期服务里很重要.

---

## 四、落地判断

观察者适合这些场景:

- 一个事件发生后有多个响应方
- 响应方经常增减
- 发布者不应该依赖具体响应方
- 事件可以作为系统扩展点

常见落地包括:

- DOM 事件监听
- Node.js EventEmitter
- React/Vue 状态订阅
- 领域事件
- 文件变化监听
- WebSocket 消息分发

观察者和发布订阅很接近.简单区分是:观察者通常发布者知道观察者列表;发布订阅通常通过中间 broker 解耦双方.下一篇发布订阅会更强调 EventBus、消息队列和跨进程边界.

---

## 五、代价与误用

观察者的代价是调用关系变隐式.代码里看不到谁会响应这个事件,排查问题要去查订阅关系.

常见误用有四种.

第一种是内存泄漏.订阅了但没有取消,组件销毁后回调还在,既占内存又可能访问无效状态.

第二种是事件粒度混乱.`UserChanged` 太粗,订阅者不知道到底变了什么;`UserNameInputKeyUp` 又太细,会把 UI 细节泄漏给业务.

第三种是错误处理不清.一个观察者失败,是否影响其他观察者?是否影响发布者?同步观察者和异步观察者的语义必须明确.

第四种是用事件掩盖业务流程.核心强一致流程不要轻易拆成一堆事件,否则顺序、事务和回滚会变得很难维护.

---

## 六、一句话总结

> 观察者模式适合把事件发布者和多个响应方解耦,但要管理好取消订阅、事件粒度、错误传播和隐式调用链.
