# 观察者模式:事件,订阅,响应式与内存泄漏

很多对象之间不是"你调用我,我立刻返回"的关系,而是"某件事发生后,很多地方都要响应".

用户注册成功后,可能要发欢迎邮件、送优惠券、写审计日志、通知 CRM.最差的写法是把这些动作都塞进注册函数里:

```ts
await userRepo.create(user)
await mail.sendWelcome(user)
await coupon.send(user)
await audit.log(user)
await crm.sync(user)
```

这样注册流程会越来越重,每新增一个响应动作都要改核心注册代码.

从变化点看,观察者模式处理的是:事件发生后的响应方会变化.稳定点是:事件本身和发布时机相对稳定.

---

## 一、问题场景

假设注册用户的核心动作是创建账号:

```ts
class UserService {
  async register(email: string) {
    const user = await this.userRepo.create({ email })

    await this.mail.sendWelcome(user)
    await this.coupon.sendNewUserCoupon(user)
    await this.audit.log("user_created", user.id)
    await this.crm.sync(user)

    return user
  }
}
```

这个写法有几个问题.

第一,发布者知道太多订阅者.注册服务本来只负责创建用户,现在还知道邮件、优惠券、CRM、审计.

第二,响应动作之间相互影响.邮件失败是否应该导致注册失败?CRM 慢是否应该拖慢接口?审计失败要不要重试?这些问题如果都塞在主流程里,边界会很难讲清楚.

第三,扩展点不稳定.新增一个"注册后推送站内信",也要改注册服务.这说明变化点没有被隔离.

观察者模式让发布者只发布事件,订阅者自己决定如何响应.

---

## 二、模式结构

观察者模式的核心是:被观察者维护一组观察者,状态变化或事件发生时通知它们.

结构可以简化成:

```text
Subject
  -> attach(observer)
  -> detach(observer)
  -> notify(event)

Observer
  -> update(event)
```

几个角色:

- `Subject`:事件源或被观察者
- `Observer`:观察者,对事件作出响应
- `Event`:描述发生了什么
- `Subscription`:订阅关系,通常要支持取消订阅

现代工程里,它常以事件监听器、回调、响应式流、状态订阅的形式出现.前端里的 DOM 事件、Node.js 的 `EventEmitter`、React/Vue 状态订阅、领域事件,都能看到它的影子.

## 三、TypeScript 示例

下面实现一个按事件类型订阅的轻量事件总线:

```ts
type UserCreated = {
  type: "user.created"
  userId: string
  email: string
}

type CouponGranted = {
  type: "coupon.granted"
  userId: string
  couponId: string
}

type DomainEvent = UserCreated | CouponGranted
type EventType = DomainEvent["type"]
type Handler<T extends DomainEvent> = (event: T) => Promise<void> | void

class EventBus {
  private handlers = new Map<EventType, Set<Handler<any>>>()

  subscribe<T extends DomainEvent>(
    type: T["type"],
    handler: Handler<T>,
  ): () => void {
    const handlers = this.handlers.get(type) ?? new Set()
    handlers.add(handler)
    this.handlers.set(type, handlers)

    return () => {
      handlers.delete(handler)
      if (handlers.size === 0) {
        this.handlers.delete(type)
      }
    }
  }

  async publish<T extends DomainEvent>(event: T) {
    const handlers = this.handlers.get(event.type) ?? new Set()

    for (const handler of handlers) {
      await handler(event)
    }
  }
}
```

注册服务只发布事件:

```ts
class UserService {
  constructor(
    private userRepo: { create(input: { email: string }): Promise<{ id: string; email: string }> },
    private events: EventBus,
  ) {}

  async register(email: string) {
    const user = await this.userRepo.create({ email })

    await this.events.publish({
      type: "user.created",
      userId: user.id,
      email: user.email,
    })

    return user
  }
}
```

响应方独立订阅:

```ts
const bus = new EventBus()

const unsubscribeMail = bus.subscribe<UserCreated>("user.created", async event => {
  console.log("send welcome mail", event.email)
})

bus.subscribe<UserCreated>("user.created", async event => {
  console.log("write audit log", event.userId)
})

unsubscribeMail()
```

`subscribe` 返回取消订阅函数,这在前端组件和长生命周期服务里很重要.没有取消订阅,观察者模式很容易变成内存泄漏入口.

## 四、适用边界

观察者适合这些场景:

- 一个事件发生后有多个响应方
- 响应方经常增减
- 发布者不应该依赖具体响应方
- 事件可以作为系统扩展点
- 响应动作不是核心事务的一部分

常见落地包括:

- DOM 事件监听
- Node.js EventEmitter
- React/Vue 状态订阅
- 领域事件
- 文件变化监听
- WebSocket 消息分发
- 本地插件系统

观察者和发布订阅很接近.简单区分是:观察者通常发布者或本地事件源知道观察者列表;发布订阅通常通过中间 broker 解耦双方,更强调跨模块、跨进程或跨服务的消息分发.

不适合的信号:

- 响应动作必须和主流程强一致提交
- 响应顺序是核心业务语义
- 调用方必须立即拿到每个响应方的结果
- 事件会掩盖本来清晰的业务流程

比如"扣库存、创建订单、扣余额"这种强一致链路,不应该为了看起来解耦就拆成一堆本地事件.这些动作的顺序、事务和补偿都属于核心业务.

---

## 五、误用

观察者的代价是调用关系变隐式.代码里看不到谁会响应这个事件,排查问题要去查订阅关系.

常见误用有五种.

第一种是内存泄漏.订阅了但没有取消,组件销毁后回调还在,既占内存又可能访问无效状态.

第二种是事件粒度混乱.`UserChanged` 太粗,订阅者不知道到底变了什么;`UserNameInputKeyUp` 又太细,会把 UI 细节泄漏给业务.

第三种是错误处理不清.一个观察者失败,是否影响其他观察者?是否影响发布者?同步观察者和异步观察者的语义必须明确.

第四种是用事件掩盖业务流程.核心强一致流程不要轻易拆成一堆事件,否则顺序、事务和回滚会变得很难维护.

第五种是事件命名随意.如果同一个事件有 `UserCreated`、`user.created`、`createdUser` 三种叫法,订阅关系会越来越难搜.

## 六、测试与重构建议

观察者模式至少要测三类东西.

第一,发布者是否在正确时机发布正确事件:

```ts
it("publishes user.created after registration", async () => {
  const events: DomainEvent[] = []
  const bus = new EventBus()

  bus.subscribe<UserCreated>("user.created", event => {
    events.push(event)
  })

  const service = new UserService(
    { create: async input => ({ id: "u1", email: input.email }) },
    bus,
  )

  await service.register("a@example.com")

  expect(events).toEqual([
    { type: "user.created", userId: "u1", email: "a@example.com" },
  ])
})
```

第二,订阅者是否正确处理事件.订阅者应该可以脱离发布者单独测试,输入一个事件,断言它调用了邮件、审计或缓存依赖.

第三,订阅关系是否能取消.前端组件、长连接、插件系统尤其要测:

```ts
it("unsubscribes handler", async () => {
  const bus = new EventBus()
  let count = 0

  const unsubscribe = bus.subscribe<UserCreated>("user.created", () => {
    count += 1
  })

  unsubscribe()
  await bus.publish({ type: "user.created", userId: "u1", email: "a@example.com" })

  expect(count).toBe(0)
})
```

从直接调用重构到观察者时,建议先从"非核心、可异步、可失败隔离"的响应动作开始.不要一次性把所有后置逻辑都事件化.每抽一个事件,都要写清楚:

- 事件名和字段含义
- 发布时机,是在事务内还是事务后
- 处理失败是否影响发布者
- 是否允许多个订阅者并行
- 订阅者是否需要幂等

事件是系统扩展点,也是新的契约.契约不清楚,解耦会变成混乱.

---

## 七、一句话总结

> 观察者模式适合把事件发布者和多个响应方解耦,但要管理好取消订阅、事件粒度、错误传播和隐式调用链.
