# 微服务模式:服务拆分、Saga、API Gateway 与反腐层

很多团队上微服务不是因为单体真的扛不住,而是因为"微服务看起来更先进"。结果一个原本能在 IDE 里跳转、一个事务里提交的系统,变成十几个仓库、几十个接口、到处超时重试、联调靠吼、排错靠猜。微服务没有消灭复杂度,只是把进程内复杂度换成了分布式复杂度。

微服务模式要解决的问题是:**当业务边界、团队边界、部署节奏已经不同步时,用服务边界隔离变化和交付**。

---

## 一、问题:单体什么时候真的开始痛

单体不是坏词。一个结构清晰的模块化单体,通常比一组混乱的微服务更容易维护。需要考虑拆分的信号包括:

- 不同业务模块发布节奏冲突,一个小改动要等整个系统窗口.
- 团队之间互相改同一块代码,冲突和回归越来越多.
- 某个模块的扩展性需求明显不同,比如搜索、推荐、消息.
- 某些数据或能力需要独立权限、审计、合规边界.
- 单体内部依赖已经按业务边界稳定切开,拆出来不会牵一发动全身.

如果连模块边界都说不清,直接拆服务只会把混乱网络化。

---

## 二、模式:微服务不是只拆接口

一个典型微服务系统会包含:

```text
Client
  |
  v
API Gateway
  |
  +--> Order Service  ---> Order DB
  |
  +--> Payment Service -> Payment DB
  |
  +--> Inventory Service -> Inventory DB
  |
  +--> User Service ----> User DB
```

核心原则:

- 每个服务围绕业务能力拆,不是围绕技术层拆.
- 服务拥有自己的数据,不要多个服务共享同一组表.
- 跨服务调用要显式处理失败、超时、重试、幂等.
- 跨服务一致性通常靠 Saga 或事件,不是本地数据库事务.

一个服务边界可以这样理解:

```text
Order Service
  负责:订单生命周期,订单状态,订单查询
  不负责:实际扣款,库存明细,用户资料主数据

Payment Service
  负责:支付单,渠道交互,退款,对账
  不负责:订单是否能发货
```

## 三、关键模式:Gateway、Saga、反腐层

### 3.1 API Gateway:统一入口,不要变成上帝服务

API Gateway 负责路由、认证、限流、协议转换、聚合少量面向端的接口。

```text
Mobile / Web
    |
    v
API Gateway
    |---- /orders   -> Order Service
    |---- /payments -> Payment Service
    |---- /users    -> User Service
```

误用是把业务逻辑写进 Gateway。Gateway 一旦开始判断订单状态、计算价格、编排复杂流程,它就会变成新的单体。

### 3.2 Saga:跨服务长事务

下单可能涉及订单、支付、库存。不能用一个本地事务包住所有服务,常见做法是 Saga:

```text
Create Order
  -> Reserve Stock
  -> Create Payment
  -> Confirm Order

失败补偿:
  Payment failed -> Release Stock -> Cancel Order
```

编排式 Saga:

```ts
class PlaceOrderSaga {
  async execute(cmd: PlaceOrder) {
    const orderId = await orders.create(cmd);
    try {
      await inventory.reserve(orderId, cmd.items);
      await payments.create(orderId, cmd.amount);
      await orders.confirm(orderId);
    } catch (e) {
      await inventory.release(orderId);
      await orders.cancel(orderId);
      throw e;
    }
  }
}
```

协同式 Saga 则通过事件让各服务自己响应,耦合低,但链路更难追踪。

### 3.3 反腐层:不要让外部模型污染内部

当一个服务依赖另一个遗留系统或第三方系统时,用反腐层隔离对方模型:

```ts
class LegacyUserClient {
  async getCustomer(id: string): Promise<CustomerProfile> {
    const raw = await legacyApi.fetchUser(id);
    return {
      id: raw.USER_ID,
      displayName: raw.NICK || raw.REAL_NAME,
      level: mapLevel(raw.VIP_TYPE),
    };
  }
}
```

反腐层不是简单 wrapper,而是模型翻译边界。它保护内部语言不被外部系统带偏。

## 四、落地判断:先模块化,再服务化

微服务适合:

- 团队规模已经大到需要独立交付.
- 业务边界稳定,能按领域能力划分.
- 有成熟 DevOps:CI/CD、监控、日志、追踪、告警.
- 能接受分布式一致性的复杂度.
- 服务之间调用频率可控,不会每个请求都跨五六个服务.

不适合:

- 早期产品,需求每天大改.
- 团队很小,部署和运维能力有限.
- 数据边界不清,多个模块必须频繁 join 同一批表.
- 性能瓶颈主要在 SQL 或缓存,不是交付边界.

推荐演进路径:

```text
混乱单体
  -> 分层单体
  -> 模块化单体
  -> 按业务能力拆出少量服务
  -> 只有必要时继续拆细
```

## 五、代价与误用

微服务的代价很硬:

- 网络调用会失败,本地方法调用不会有这些失败模式.
- 数据一致性从本地事务变成最终一致和补偿.
- 测试从单进程变成契约测试、集成环境和端到端链路.
- 排错需要 trace id、集中日志、指标和告警.
- 版本兼容成为日常工作.

常见误用之一是按数据库表拆服务。`UserService`、`OrderService` 不等于每张表一个服务。服务应该围绕业务能力和一致性边界。

常见误用之二是共享数据库。多个服务直接读写同一张表,表面上拆了服务,实际上仍然是一个耦合数据库单体。

常见误用之三是同步调用链过长。一次请求从 A 调 B 调 C 调 D,任何一个服务慢都会拖垮全链路。

---

## 六、与应用架构和前端模式的边界

微服务是部署和组织边界模式,分层、六边形、Clean Architecture 是单个服务内部的代码架构模式。一个微服务内部仍然可以是分层架构或 Clean Architecture。

前端的微前端和这里的微服务也不是同一个问题。微服务拆的是后端业务能力和数据所有权;微前端拆的是前端团队交付、运行时集成和页面边界。两者可以配合,但不要用微前端术语解释后端数据一致性。

---

## 一句话总结

微服务的价值是按业务和团队边界独立交付,但它会引入网络、数据一致性和运维复杂度;如果单体还没有清晰模块边界,先别急着拆服务。
