# 外观模式:给复杂子系统一个稳定入口

系统越做越大,一个业务动作背后往往要调用很多子系统.比如下单可能要校验库存、计算价格、锁库存、创建订单、发消息、写审计日志.

如果每个调用方都直接编排这些子系统,重复逻辑会越来越多.更麻烦的是,子系统内部一改,所有调用方都要跟着改.

变化点是:子系统内部结构和调用顺序会变.稳定点是:外部只需要一个更粗粒度的业务入口.

---

## 一、问题

没有外观时,Controller 可能长这样:

```ts
await inventory.check(skuId, count)
const price = await pricing.calculate(skuId, count)
await inventory.lock(skuId, count)
const order = await orders.create(userId, skuId, count, price)
await events.publish("OrderCreated", order.id)
```

这段代码可能出现在 Web Controller、后台任务、测试工具、开放 API 里.以后锁库存顺序变了,或者要增加风控校验,每个地方都得改.

---

## 二、模式

外观模式的核心是:为复杂子系统提供一个简单、稳定、面向使用者的入口.

结构可以简化成:

```text
Client -> Facade -> Subsystem A
                  -> Subsystem B
                  -> Subsystem C
```

Facade 不一定隐藏所有能力.它只是把常用流程收束成清晰入口,让调用方不必理解子系统内部细节.

---

## 三、示例

```ts
class CheckoutFacade {
  constructor(
    private inventory: InventoryService,
    private pricing: PricingService,
    private orders: OrderRepository,
    private events: EventBus
  ) {}

  async checkout(userId: string, skuId: string, count: number) {
    await this.inventory.check(skuId, count)
    const price = await this.pricing.calculate(skuId, count)
    await this.inventory.lock(skuId, count)
    const order = await this.orders.create({ userId, skuId, count, price })
    await this.events.publish("OrderCreated", order.id)
    return order
  }
}
```

Controller 只需要:

```ts
return checkoutFacade.checkout(userId, skuId, count)
```

这样调用方依赖的是稳定的结账入口,而不是一堆子系统的细节.

---

## 四、落地判断

外观模式适合这些场景:

- Controller 里开始堆很多服务编排
- 一个业务流程被多个入口重复调用
- 子系统接口太细,调用方需要理解太多顺序
- 想给 SDK 或内部平台提供更友好的入口

在后端项目里,应用服务层经常承担外观角色.比如 `OrderApplicationService.placeOrder` 对外是一句下单,内部编排库存、价格、订单和消息.

在前端项目里,API client 也常做外观.页面不直接拼多个 HTTP 请求,而是调用 `checkoutApi.submitOrder`.

---

## 五、代价与误用

外观的代价是它可能变成新的上帝对象.所有流程都塞进一个 `SystemFacade`,最后它知道所有子系统,也承担所有业务判断.

常见误用有三种.

第一种是外观太薄.只是把 `a.doA()` 包成 `facade.doA()`,没有降低调用复杂度,只是在转发.

第二种是外观太厚.外观不仅编排流程,还实现库存算法、价格规则、权限策略,导致子系统被架空.

第三种是用外观掩盖糟糕边界.如果子系统本身职责混乱,外观只能暂时降低调用方痛苦,不能真正修复内部设计.

---

## 六、一句话总结

> 外观模式把复杂子系统收束成一个稳定入口,适合降低调用方复杂度,但不能把它写成吞掉所有职责的巨型服务.
