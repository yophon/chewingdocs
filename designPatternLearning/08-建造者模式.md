# 建造者模式

构造对象最怕两种情况:参数太多,或者创建过程有顺序和约束。参数一多,调用方就开始传一串看不懂的布尔值和 null。过几个月再看,没人知道第 5 个参数代表什么。

建造者模式解决的是复杂对象创建的可读性和一致性。

> 一句话先记住:建造者适合创建复杂对象,尤其是参数多、可选项多、需要校验或分步组装的时候。

---

## 一、先看问题

```ts
const client = new HttpClient(
  "https://api.example.com",
  3000,
  true,
  3,
  null,
  { Authorization: "token" },
);
```

这段代码能跑,但阅读体验很差:

- `3000` 是连接超时还是读取超时?
- `true` 是开启重试还是开启日志?
- `null` 是代理还是证书?
- 参数顺序一变,调用方全要改

当构造函数开始变成"参数隧道",就该考虑建造者或配置对象。

---

## 二、先用配置对象解决一半问题

很多场景不需要完整建造者,一个配置对象就够了:

```ts
const client = new HttpClient({
  baseUrl: "https://api.example.com",
  timeoutMs: 3000,
  retry: 3,
  headers: { Authorization: "token" },
});
```

它已经解决了参数可读性问题。

如果还需要链式设置,默认值,校验,不可变对象,再引入建造者。

## 三、建造者的基本写法

```ts
class HttpClientBuilder {
  private options: HttpClientOptions = {
    baseUrl: "",
    timeoutMs: 1000,
    retry: 0,
    headers: {},
  };

  baseUrl(value: string) {
    this.options.baseUrl = value;
    return this;
  }

  timeoutMs(value: number) {
    this.options.timeoutMs = value;
    return this;
  }

  retry(value: number) {
    this.options.retry = value;
    return this;
  }

  header(key: string, value: string) {
    this.options.headers[key] = value;
    return this;
  }

  build() {
    if (!this.options.baseUrl) {
      throw new Error("baseUrl is required");
    }
    return new HttpClient({ ...this.options });
  }
}

const client = new HttpClientBuilder()
  .baseUrl("https://api.example.com")
  .timeoutMs(3000)
  .retry(3)
  .header("Authorization", "token")
  .build();
```

结构是:

```text
Client code -> Builder -> Product
```

调用方负责描述要什么,Builder 负责怎么组装和校验。

## 四、建造者保护的变化点

建造者不是为了链式调用好看,而是为了管理创建复杂度:

- 默认值可能变化
- 参数组合需要校验
- 创建步骤有顺序
- 同一对象有多种创建场景
- 最终对象希望不可变

比如不可变配置:

```ts
class HttpClient {
  constructor(private readonly options: Readonly<HttpClientOptions>) {}
}
```

Builder 可以是可变的,但 build 出来的产品应该尽量稳定。

## 五、Java 里的典型写法

Java 没有 TypeScript 那么方便的对象字面量,建造者更常见:

```java
public class User {
  private final String name;
  private final String email;
  private final int age;

  private User(Builder builder) {
    this.name = builder.name;
    this.email = builder.email;
    this.age = builder.age;
  }

  public static class Builder {
    private String name;
    private String email;
    private int age;

    public Builder name(String name) {
      this.name = name;
      return this;
    }

    public Builder email(String email) {
      this.email = email;
      return this;
    }

    public Builder age(int age) {
      this.age = age;
      return this;
    }

    public User build() {
      return new User(this);
    }
  }
}
```

调用:

```java
User user = new User.Builder()
  .name("Alice")
  .email("a@example.com")
  .age(20)
  .build();
```

Java 里建造者常和不可变对象一起出现。

## 六、和工厂的区别

| 问题 | 工厂 | 建造者 |
| --- | --- | --- |
| 重点 | 选择创建哪个类 | 分步骤创建一个复杂对象 |
| 变化点 | 产品类型变化 | 参数和组装过程变化 |
| 常见输入 | type, config, env | 多个字段,多个可选项 |
| 结果 | 返回某个实现 | 返回组装完成的对象 |

如果你的核心问题是 "alipay 还是 wechat",用工厂。如果核心问题是 "这个 client 有 20 个配置项怎么安全创建",用建造者。

## 七、落地判断

适合建造者的信号:

- 构造函数参数超过 4 到 5 个,且有多个同类型参数
- 可选参数很多,默认值很多
- 创建前需要校验参数组合
- 创建步骤有顺序,但调用方不该记细节
- 产品对象希望不可变

不适合建造者的信号:

- 只是两个参数的简单对象
- 配置对象已经足够清晰
- Builder 只是机械转发 setter
- 对象创建没有约束,没有默认值,没有复杂步骤

---

## 八、代价与误用

**误用一:把所有 DTO 都配 Builder**

普通请求对象,响应对象,数据库记录不一定需要建造者。大量 Builder 会让代码膨胀。

**误用二:Builder 不做校验**

如果 Builder 只是链式 setter,没有默认值,没有校验,没有隐藏组装细节,它的价值很低。

**误用三:建造过程泄漏到业务里**

如果每个业务服务都要知道 10 个构建步骤,Builder 没有真正封装复杂度。可以提供预设方法或工厂方法。

**误用四:可变 Builder 被复用**

Builder 通常不应该跨请求共享。复用可变 Builder 很容易把上一次的参数带到下一次。

---

## 九、一句话总结

> 建造者模式不是为了链式 API,而是为了让复杂对象的创建过程可读、可校验、可稳定演进。
