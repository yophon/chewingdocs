# 状态管理模式:本地状态、服务端状态、全局状态怎么分

很多前端项目不是一开始就乱,而是在第三次加需求时乱:一个弹窗要读列表筛选条件,一个按钮要刷新用户信息,一个页面跳转回来要保留表单,最后所有东西都进了全局 store.表面上是"状态管理",本质上是**没有先区分状态的生命周期和来源**.

一句话先记住:**状态管理不是把数据集中起来,而是把状态放在它最自然的生命周期里**.

---

## 一、问题:所有状态都全局化会坏在哪里

常见坏味道是这样:

```ts
type AppState = {
  user: User | null;
  products: Product[];
  productLoading: boolean;
  searchKeyword: string;
  isFilterPanelOpen: boolean;
  selectedTab: string;
  draftComment: string;
};
```

这段代码的问题不是字段多,而是这些字段的变化原因完全不同:

- `user` 来自登录态,跨页面共享
- `products` 来自服务端,需要缓存和重新拉取
- `searchKeyword` 可能只属于当前页面
- `isFilterPanelOpen` 只是一个 UI 展开状态
- `draftComment` 是临时输入,离开页面可能就该丢掉

如果它们都放进一个全局 store,你会得到三个后果:

1. 调试困难:任何组件都可能改任何字段.
2. 缓存困难:服务端数据需要失效、重试、刷新,本地 UI 状态不需要.
3. 复用困难:组件被迫依赖全局上下文,搬到别的页面就坏.

稳定点是"页面需要从状态渲染 UI",变化点是"状态的来源、生命周期、共享范围和一致性要求".

---

## 二、判断框架:先给状态分类

状态管理可以先问四个问题:

| 问题 | 典型答案 | 放哪里 |
| --- | --- | --- |
| 谁产生它 | 用户输入 / 服务端 / 路由 / 设备 | 决定数据源 |
| 谁消费它 | 单组件 / 页面 / 多页面 / 全应用 | 决定作用域 |
| 活多久 | 一次交互 / 当前页面 / 会话 / 长期缓存 | 决定生命周期 |
| 谁是事实来源 | 前端内存 / URL / 服务端 / 本地存储 | 决定同步方式 |

常见可以分成四类:

```text
本地 UI 状态      -> useState / ref / local component state
页面状态          -> page store / route state / context
服务端状态        -> query cache / loader / repository
全局客户端状态    -> auth / theme / feature flags / app session
```

关键是别把"跨组件使用"直接等同于"全局状态".有时 props、组合组件、URL 参数就够了.

## 三、示例:把状态拆到合适的位置

不要这样:

```ts
// 所有页面共享一个巨大 store
const useAppStore = create<AppState>((set) => ({
  products: [],
  keyword: "",
  filterOpen: false,
  setKeyword: (keyword) => set({ keyword }),
}));
```

更好的切法是:

```ts
function ProductPage() {
  const [keyword, setKeyword] = useSearchParam("q");
  const [filterOpen, setFilterOpen] = useState(false);

  const products = useProductsQuery({ keyword });

  return (
    <ProductView
      products={products.data ?? []}
      loading={products.isLoading}
      keyword={keyword}
      onKeywordChange={setKeyword}
      filterOpen={filterOpen}
      onFilterOpenChange={setFilterOpen}
    />
  );
}
```

这里的分工是:

- `keyword` 放 URL,因为它影响页面可分享、可回退、可刷新.
- `filterOpen` 放组件本地,因为它只是视觉状态.
- `products` 交给 query cache,因为它来自服务端,需要缓存、重试、失效.
- `ProductView` 只关心渲染,不关心数据从哪里来.

服务端状态尤其不要手写成普通全局状态:

```ts
function useProductsQuery(params: { keyword: string }) {
  return useQuery({
    queryKey: ["products", params],
    queryFn: () => api.getProducts(params),
    staleTime: 30_000,
  });
}
```

这类库解决的是"服务端数据在前端的缓存副本",不是普通变量.

## 四、落地判断

可以用一个简单顺序:

1. 能放组件内部,先放组件内部.
2. 需要父子共享,先考虑 props、插槽、组合组件.
3. 需要同一路页面多处共享,考虑页面级 context 或 page store.
4. 需要刷新、回退、分享,考虑 URL.
5. 来自服务端,优先 query cache 或路由 loader.
6. 真正跨应用共享,再进全局 store.

全局 store 适合放这些东西:

- 登录用户和权限摘要.
- 主题、语言、实验开关.
- WebSocket 连接状态.
- 多页面都要参与的编辑会话.

不适合放这些东西:

- 一个按钮是否 hover.
- 一个弹窗是否打开,且只在当前组件使用.
- 某个列表接口返回值,但已经有 query cache.
- 表单每个输入框的临时值,除非确实要跨页面恢复.

## 五、代价与误用

状态管理的代价不在 API,在认知负担.状态一旦全局化,就等于承诺它可以被远处代码读写,测试也要准备全局上下文.

常见误用有三种:

**误用一:用全局状态替代组件设计**.组件之间传值麻烦,就塞 store.结果组件边界消失,复用性下降.

**误用二:用客户端状态复制服务端状态**.接口数据手动塞 Redux,再手动处理 loading、error、refetch、cache invalidation.除非业务需要精细控制,否则大多不值得.

**误用三:把状态管理库当架构**.Redux、Pinia、Zustand、MobX 只是工具.真正的架构问题是状态归属、生命周期和事实来源.

替代方案也要记住:

- URL 可以管理页面筛选、分页、排序.
- 路由 loader 可以管理页面首屏数据.
- 表单库可以管理复杂表单状态.
- query cache 可以管理服务端缓存.
- 组合组件可以避免无意义的全局共享.

---

## 六、一句话总结

> **状态管理的第一原则不是集中,而是归位**.先判断状态的来源、作用域、生命周期和事实来源,再决定它应该留在组件、页面、URL、服务端缓存还是全局 store.
