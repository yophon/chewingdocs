# 路由与页面模式:嵌套路由、布局、加载器与守卫

很多项目把路由当成 URL 到组件的映射表.小项目没问题,但一旦出现登录态、权限、布局、面包屑、首屏数据、错误页、缓存恢复,路由就会变成一堆散落在页面里的 if-else.

路由与页面模式要解决的问题是:**页面不是孤立组件,页面是 URL、布局、数据、权限和导航状态的组合边界**.

---

## 一、问题:页面逻辑散落在组件里

常见写法:

```ts
function OrderDetailPage() {
  const { user } = useAuth();
  const { orderId } = useParams();
  const navigate = useNavigate();
  const order = useOrderQuery(orderId!);

  useEffect(() => {
    if (!user) navigate("/login");
    if (user && !user.permissions.includes("order:read")) navigate("/403");
  }, [user]);

  if (order.isLoading) return <Spinner />;
  if (order.isError) return <ErrorView />;

  return <OrderDetail order={order.data} />;
}
```

这段代码的问题不是不能跑,而是页面组件同时承担:

- 参数解析.
- 登录和权限判断.
- 数据加载.
- 错误处理.
- UI 渲染.

这些能力本来属于路由层或页面边界,不应该每个页面复制.

---

## 二、模式:把路由当作页面协议

一个成熟的路由配置不只是 path:

```text
Route
  -> path: URL 结构
  -> layout: 页面外壳
  -> loader: 首屏数据
  -> guard: 访问规则
  -> error boundary: 错误展示
  -> children: 嵌套路由
```

示例:

```ts
const routes = [
  {
    path: "/app",
    element: <AppLayout />,
    guard: requireLogin,
    children: [
      {
        path: "orders/:orderId",
        loader: loadOrderDetail,
        element: <OrderDetailPage />,
        errorElement: <OrderErrorPage />,
      },
    ],
  },
];
```

页面组件就可以更专注:

```ts
function OrderDetailPage() {
  const order = useLoaderData() as Order;
  return <OrderDetail order={order} />;
}
```

稳定点是 URL 对应一个页面意图,变化点是页面需要的布局、数据和访问规则.

---

## 三、嵌套路由:页面结构和 URL 结构一致

嵌套路由适合表达"共享布局 + 局部页面变化":

```text
/settings/profile   -> SettingsLayout + ProfilePage
/settings/security  -> SettingsLayout + SecurityPage
/settings/billing   -> SettingsLayout + BillingPage
```

对应结构:

```ts
{
  path: "/settings",
  element: <SettingsLayout />,
  children: [
    { path: "profile", element: <ProfilePage /> },
    { path: "security", element: <SecurityPage /> },
    { path: "billing", element: <BillingPage /> },
  ],
}
```

不要为了复用布局在每个页面手写:

```ts
function ProfilePage() {
  return <SettingsLayout><ProfileForm /></SettingsLayout>;
}
```

这样布局、导航选中态、面包屑和权限规则会到处复制.嵌套路由的价值是让页面层级成为显式结构.

---

## 四、加载器与守卫:把前置条件放到页面入口

加载器适合处理"进入页面前必须有的数据":

```ts
async function loadOrderDetail({ params }: LoaderArgs) {
  const orderId = params.orderId;
  if (!orderId) throw new Response("Not Found", { status: 404 });
  return api.getOrder(orderId);
}
```

守卫适合处理"进入页面前必须满足的条件":

```ts
function requirePermission(permission: string) {
  return (ctx: RouteContext) => {
    if (!ctx.user) return redirect("/login");
    if (!ctx.user.permissions.includes(permission)) return redirect("/403");
    return null;
  };
}
```

守卫不要滥用成业务规则引擎.它只回答"能不能进入这个页面",不要处理"这个订单能不能退款".后者应该在业务层或页面模型里判断.

---

## 五、落地判断

路由层适合放:

- URL 参数解析和校验.
- 页面布局嵌套.
- 登录态和粗粒度权限.
- 首屏关键数据加载.
- 页面级错误边界.
- 跳转、重定向、404.

页面组件适合放:

- 页面内交互状态.
- 数据到视图模型的整理.
- 局部弹窗、表单、选择.
- 细粒度业务动作.

业务层适合放:

- 订单能否支付、退款、取消.
- 用户角色和资源关系的复杂判断.
- 跨端复用的规则.

---

## 六、代价与误用

代价是路由配置会变厚,团队需要统一约定.如果没有约定,路由文件会变成另一个上帝对象.

常见误用:

**误用一:所有数据都放 loader**.有些数据只在用户点击 tab 后才需要,不必阻塞首屏.

**误用二:路由守卫写业务细节**.守卫越复杂,越难测试,也越容易和后端权限不一致.

**误用三:URL 不表达状态**.筛选、分页、详情 id 都藏在内存里,用户刷新或分享就丢.

**误用四:页面目录和路由结构完全脱节**.开发者找一个页面要在多个目录里跳来跳去.

---

## 七、一句话总结

> **路由不是组件映射表,而是页面边界协议**.把布局、参数、首屏数据、访问规则和错误边界放在入口,页面组件才有机会保持清晰.
