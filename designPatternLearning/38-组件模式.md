# 组件模式:Props、Slots、Composition 与复用边界

前端代码最容易从"页面"变成"一大坨模板"。一个订单详情页里同时有数据请求、权限判断、表格渲染、弹窗状态、按钮逻辑、样式细节。复制到退款页时改一点,复制到后台页再改一点,很快同一个按钮、同一个卡片、同一段校验散落在十几个地方。

组件模式要解决的问题是:**把界面拆成可组合、可替换、可测试的 UI 单元,同时控制复用带来的耦合**。

---

## 一、问题:页面级代码无法承受界面变化

没有组件边界时,变化点会混在一起:

- 展示变化:布局、颜色、文案、空状态.
- 数据变化:字段来源、加载状态、错误状态.
- 交互变化:点击、选择、展开、拖拽.
- 复用变化:同一个 UI 要出现在多个页面.

稳定点通常是某个界面概念:按钮、表单项、列表、弹窗、订单摘要、用户头像。组件模式就是围绕这些稳定概念拆分。

---

## 二、模式:Props 输入,事件输出,组合扩展

一个组件最基本的边界:

```text
Parent
  |
  | props
  v
Component
  |
  | events / callbacks
  v
Parent
```

React 示例:

```tsx
type OrderSummaryProps = {
  orderNo: string;
  amount: number;
  status: "pending" | "paid" | "canceled";
  onPay?: () => void;
};

function OrderSummary(props: OrderSummaryProps) {
  return (
    <section>
      <div>{props.orderNo}</div>
      <div>{props.amount}</div>
      <button disabled={props.status !== "pending"} onClick={props.onPay}>
        Pay
      </button>
    </section>
  );
}
```

组件不自己决定订单从哪里来,也不直接调用支付 API。它接收数据,渲染界面,把用户意图通过回调交出去。

更灵活的组合可以用 slots 或 children:

```tsx
function Panel(props: { title: string; actions?: React.ReactNode; children: React.ReactNode }) {
  return (
    <section>
      <header>
        <h2>{props.title}</h2>
        {props.actions}
      </header>
      <div>{props.children}</div>
    </section>
  );
}
```

`Panel` 稳定的是布局框架,变化的是内容和操作区。

## 三、组件边界:展示组件、业务组件、页面组件

实际项目里可以粗分三类:

```text
UI 基础组件
  Button, Input, Modal, Table
  关注交互和视觉一致性

业务组件
  OrderSummary, UserBadge, PriceTag
  关注业务展示概念

页面组件
  OrderDetailPage, RefundPage
  关注路由、数据加载、用例编排
```

不要让基础组件知道订单状态,也不要让页面组件重复实现按钮和弹窗细节。

一个常见结构:

```text
pages/OrderDetailPage.tsx
features/order/OrderSummary.tsx
features/order/PayOrderDialog.tsx
components/Button.tsx
components/Modal.tsx
```

组件拆分的实用判断:

- 这段 UI 是否在多个地方重复?
- 它是否有独立的状态和交互?
- 它的输入输出能否说清楚?
- 拆出来后调用方是否更清晰?
- 未来变化是内部变化,还是每个使用方都要跟着改?

---

## 四、落地判断:复用不是唯一目标

组件不只是为了复用,还为了隔离复杂度。一个只用一次的复杂上传组件,也值得拆出来;一个只渲染两行文字的组件,即使用了三次也未必值得抽。

好的组件通常有几个特征:

- Props 是组件需要的最小数据,不是整坨后端对象.
- 事件表达用户意图,不是暴露内部 DOM 细节.
- 视觉状态完整:loading、empty、error、disabled 都考虑.
- 组合优先于配置爆炸.
- 业务组件不假装成通用组件.

配置爆炸是组件复用最常见的坏味道:

```tsx
<UserCard
  showAvatar
  showLevel
  showActions
  compact
  bordered
  fromAdmin
  fromOrderPage
/>
```

这种组件已经承担了太多场景。更好的方式是拆分稳定骨架,把变化区域交给组合:

```tsx
<UserCard user={user} actions={<BanUserButton userId={user.id} />} />
```

## 五、代价与误用

组件模式的代价是边界设计。拆得太少,页面臃肿;拆得太碎,跳转困难,Props 层层透传。

常见误用之一是过早抽通用组件。两个页面长得像,不代表变化方向一样。如果一个组件需要十几个布尔参数才能复用,它可能不该被复用。

常见误用之二是把业务请求塞进底层 UI 组件。`Button` 不应该知道如何提交订单,`Modal` 不应该知道退款流程。底层组件越通用,越不应该依赖业务。

常见误用之三是把组件模式和后端分层混为一谈。组件模式解决 UI 组合和渲染边界;分层、六边形、Clean Architecture 解决业务代码和外部依赖边界。前端应用可以同时有组件层和应用服务层,但它们回答的问题不同。

---

## 六、前端模式边界

组件模式回答"界面怎么拆"。单向数据流回答"状态怎么改"。响应式模式回答"依赖变化后谁自动更新"。组件可以使用单向数据流,也可以使用响应式系统,但组件本身不是状态架构的全部。

当你在讨论 Props、Slots、children、composition、渲染边界时,你在讲组件模式。当你在讨论 action、store、reducer、computed、effect 时,已经进入状态模式或响应式模式。

---

## 一句话总结

组件模式用 Props、事件和组合把 UI 拆成稳定单元,价值在隔离界面复杂度而不只是复用,但过早通用化会把组件变成难维护的参数机器。
