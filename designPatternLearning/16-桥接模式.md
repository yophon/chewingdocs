# 桥接模式:两个维度独立变化时怎么拆

有一种代码膨胀特别隐蔽:两个维度同时变化.

比如消息通知有不同类型:告警、账单、营销.发送渠道也有不同类型:短信、邮件、站内信.如果用继承硬写,很快会出现:

```text
SmsAlert
EmailAlert
InAppAlert
SmsBill
EmailBill
InAppBill
```

每新增一种消息类型或渠道,类数量都会成倍增加.

变化点是:抽象维度和实现维度都可能变化.稳定点是:它们之间只是协作关系,不应该被继承绑死.

---

## 一、问题

继承适合表达稳定的 "is-a" 关系,但不适合同时承载多个变化轴.

以报表为例:

- 报表类型:日报、周报、月报
- 输出格式:HTML、PDF、CSV

如果用继承组合,就会得到 `DailyHtmlReport`, `DailyPdfReport`, `WeeklyCsvReport`.以后新增 Excel 格式,所有报表类型都要加一遍类.

这不是复用,是组合爆炸.

---

## 二、模式

桥接模式的核心是:把抽象部分和实现部分拆开,让它们可以独立变化.

结构可以简化成:

```text
Abstraction -> Implementor
     |              |
 Refined        Concrete
```

更直白地说:一个维度用业务抽象表达,另一个维度用接口注入进来.

---

## 三、示例

```ts
interface Sender {
  send(to: string, content: string): Promise<void>
}

class EmailSender implements Sender {
  async send(to: string, content: string) {
    console.log("email", to, content)
  }
}

class SmsSender implements Sender {
  async send(to: string, content: string) {
    console.log("sms", to, content)
  }
}

abstract class Notification {
  constructor(protected sender: Sender) {}
  abstract notify(to: string): Promise<void>
}

class AlertNotification extends Notification {
  async notify(to: string) {
    await this.sender.send(to, "system alert")
  }
}

class BillNotification extends Notification {
  async notify(to: string) {
    await this.sender.send(to, "your bill is ready")
  }
}
```

通知类型和发送渠道被拆成两个维度.新增 `PushSender` 不需要改 `AlertNotification`;新增 `MarketingNotification` 也不需要改发送渠道.

---

## 四、落地判断

桥接模式适合这些信号:

- 类名开始出现两个维度,比如 `PdfMonthlyReport`
- 新增一个维度值时,要批量新增很多类
- 继承层级越来越深,但本质只是组合变化
- 你希望运行时切换实现,比如同一份报表输出不同格式

在现代代码里,桥接常常不像 GoF 书里那么正式.依赖接口、构造函数注入、组合优于继承,很多时候就是桥接思想的落地.

Go 里也常见这种写法:

```go
type Sender interface {
    Send(to string, content string) error
}

type Notification struct {
    Sender Sender
}
```

语言不同,名字可以不同,关键是把两个变化轴拆开.

---

## 五、代价与误用

桥接的代价是抽象层增加.原来一个类能看完的逻辑,现在要同时看抽象和实现接口.

常见误用有三种.

第一种是变化轴并不存在.只有一个报表、一个输出格式时,提前桥接只是过度设计.

第二种是拆错维度.比如把"普通用户通知"和"VIP 用户通知"拆成抽象维度,但实际差异只是模板文案,那用配置或策略可能更简单.

第三种是接口太泛.`Sender.send(anything)` 看似通用,实际让每个实现自己解析参数,类型安全和可读性都会下降.

---

## 六、一句话总结

> 桥接模式用组合拆开两个独立变化的维度,避免继承组合爆炸,但只有当两个维度真的会独立变化时才值得引入.
