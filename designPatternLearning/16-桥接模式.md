# 桥接模式:两个维度独立变化时怎么拆

有一种代码膨胀特别隐蔽:两个维度同时变化.

比如消息通知有不同类型:告警、账单、营销.发送渠道也有不同类型:短信、邮件、站内信.如果用继承硬写,很快会出现:

```text
SmsAlert
EmailAlert
InAppAlert
SmsBill
EmailBill
InAppBill
```

每新增一种消息类型或渠道,类数量都会成倍增加.

变化点是:抽象维度和实现维度都可能变化.稳定点是:它们之间只是协作关系,不应该被继承绑死.

---

## 一、问题

继承适合表达稳定的 "is-a" 关系,但不适合同时承载多个变化轴.

以报表为例:

- 报表类型:日报、周报、月报
- 输出格式:HTML、PDF、CSV

如果用继承组合,就会得到 `DailyHtmlReport`, `DailyPdfReport`, `WeeklyCsvReport`.以后新增 Excel 格式,所有报表类型都要加一遍类.

这不是复用,是组合爆炸.

---

## 二、模式

桥接模式的核心是:把抽象部分和实现部分拆开,让它们可以独立变化.

结构可以简化成:

```text
Abstraction -> Implementor
     |              |
 Refined        Concrete
```

更直白地说:一个维度用业务抽象表达,另一个维度用接口注入进来.

---

## 三、示例

```ts
interface Sender {
  send(to: string, content: string): Promise<void>
}

class EmailSender implements Sender {
  async send(to: string, content: string) {
    console.log("email", to, content)
  }
}

class SmsSender implements Sender {
  async send(to: string, content: string) {
    console.log("sms", to, content)
  }
}

abstract class Notification {
  constructor(protected sender: Sender) {}
  abstract notify(to: string): Promise<void>
}

class AlertNotification extends Notification {
  async notify(to: string) {
    await this.sender.send(to, "system alert")
  }
}

class BillNotification extends Notification {
  async notify(to: string) {
    await this.sender.send(to, "your bill is ready")
  }
}
```

通知类型和发送渠道被拆成两个维度.新增 `PushSender` 不需要改 `AlertNotification`;新增 `MarketingNotification` 也不需要改发送渠道.

---

## 四、落地判断

桥接模式适合这些信号:

- 类名开始出现两个维度,比如 `PdfMonthlyReport`
- 新增一个维度值时,要批量新增很多类
- 继承层级越来越深,但本质只是组合变化
- 你希望运行时切换实现,比如同一份报表输出不同格式

在现代代码里,桥接常常不像 GoF 书里那么正式.依赖接口、构造函数注入、组合优于继承,很多时候就是桥接思想的落地.

Go 里也常见这种写法:

```go
type Sender interface {
    Send(to string, content string) error
}

type Notification struct {
    Sender Sender
}
```

语言不同,名字可以不同,关键是把两个变化轴拆开.

从变化点看,桥接保护的是"两个方向各自变化".抽象维度表达业务含义,实现维度表达技术落地.它们通过接口协作,但不通过继承绑定.

报表场景也可以这样写:

```ts
interface ReportExporter {
  export(title: string, rows: string[][]): string
}

class CsvExporter implements ReportExporter {
  export(title: string, rows: string[][]) {
    return [title, ...rows.map(row => row.join(","))].join("\n")
  }
}

class HtmlExporter implements ReportExporter {
  export(title: string, rows: string[][]) {
    const body = rows.map(row => `<tr>${row.map(cell => `<td>${cell}</td>`).join("")}</tr>`).join("")
    return `<h1>${title}</h1><table>${body}</table>`
  }
}

abstract class Report {
  constructor(protected exporter: ReportExporter) {}
  abstract build(): string
}

class DailyReport extends Report {
  build() {
    return this.exporter.export("Daily Report", [["orders", "42"]])
  }
}
```

`DailyReport` 负责报表内容,`ReportExporter` 负责输出格式.新增 `WeeklyReport` 和新增 `PdfExporter` 互不影响.

## 五、测试与重构建议

桥接的测试可以按两个维度拆开:

- 抽象侧测试:日报是否生成正确标题和数据行,可以使用假的 exporter 捕获输入.
- 实现侧测试:CSV exporter 是否正确转义逗号,HTML exporter 是否正确生成表格.
- 组合测试:少量验证 `DailyReport + CsvExporter` 这样的真实组合能跑通.

假的实现很适合测试抽象侧:

```ts
class SpyExporter implements ReportExporter {
  last?: { title: string; rows: string[][] }

  export(title: string, rows: string[][]) {
    this.last = { title, rows }
    return "ok"
  }
}
```

重构时,先从类名里找变化轴.如果你已经有 `SmsAlert`, `EmailAlert`, `SmsBill`, `EmailBill`,可以先抽出 `Sender`,再把重复发送逻辑搬到实现维度.然后把 `SmsAlert` 改成 `new AlertNotification(new SmsSender())`,逐步删除组合爆炸类.

桥接重构最容易出错的是拆错责任.抽象侧应该保留业务差异,实现侧应该保留技术差异.如果 `EmailSender` 开始判断"账单通知要加某段文案",说明业务逻辑漏到了实现侧.

---

## 六、适用边界

桥接的代价是抽象层增加.原来一个类能看完的逻辑,现在要同时看抽象和实现接口.

桥接适合两个维度都有独立扩展需求的场景.如果只有一个维度变化,策略模式或简单接口注入就够了.如果只是想把第三方 SDK 包起来,那更像适配器.如果只是根据类型创建不同组合,可能还需要工厂来负责组装,但工厂不是桥接本身.

桥接也不要求所有组合都合法.比如某些通知类型不能走短信,某些报表不能导出 CSV,可以在组装层限制组合,或者在抽象侧暴露能力约束.不要为了"任意组合"牺牲业务规则.

---

## 七、代价与误用

常见误用有三种.

第一种是变化轴并不存在.只有一个报表、一个输出格式时,提前桥接只是过度设计.

第二种是拆错维度.比如把"普通用户通知"和"VIP 用户通知"拆成抽象维度,但实际差异只是模板文案,那用配置或策略可能更简单.

第三种是接口太泛.`Sender.send(anything)` 看似通用,实际让每个实现自己解析参数,类型安全和可读性都会下降.

---

## 八、一句话总结

> 桥接模式用组合拆开两个独立变化的维度,避免继承组合爆炸,但只有当两个维度真的会独立变化时才值得引入.
