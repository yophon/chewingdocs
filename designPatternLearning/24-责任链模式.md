# 责任链模式:中间件、过滤器、审批流与错误传播

一个请求进系统后,常常要经过鉴权、参数校验、限流、日志、缓存、业务处理、异常转换。最直观的写法是一个大方法从头写到尾:

```ts
checkAuth(req);
validate(req);
rateLimit(req);
handle(req);
```

这在步骤固定时没问题。但如果不同接口要不同组合,不同环境要插拔步骤,某些步骤要提前终止,大方法会越来越硬。变化点是"处理步骤和顺序会变化",稳定点是"请求要沿着一条处理路径前进"。

## 一、问题

责任链模式把多个处理者串起来。每个处理者只关心自己能不能处理,以及是否继续往后传。

```text
Request -> Handler A -> Handler B -> Handler C
```

它的核心不是"链表",而是控制权可以在链上流动。某个节点可以继续、拦截、短路、抛错或改写上下文。

## 二、模式

角色:

- Handler:处理接口
- ConcreteHandler:具体处理节点
- Context/Request:沿链传递的数据
- Chain/Next:下一个处理者

Go 风格的中间件写法很典型:

```go
package main

import "fmt"

type Context struct {
	User string
}

type Handler func(*Context) error
type Middleware func(Handler) Handler

func Auth(next Handler) Handler {
	return func(ctx *Context) error {
		if ctx.User == "" {
			return fmt.Errorf("unauthorized")
		}
		return next(ctx)
	}
}

func Log(next Handler) Handler {
	return func(ctx *Context) error {
		fmt.Println("before")
		err := next(ctx)
		fmt.Println("after")
		return err
	}
}

func Business(ctx *Context) error {
	fmt.Println("hello", ctx.User)
	return nil
}
```

这里 `Auth` 和 `Log` 都不关心最终业务是什么。它们只包装 `next`,形成一条可组合的链。

## 三、落地形态

责任链在真实项目里非常常见:

- Web 中间件:鉴权、CORS、日志、异常处理
- Servlet Filter / Spring Interceptor
- Netty Pipeline
- 前端请求拦截器
- 编译器或构建工具的 plugin pipeline
- 审批流中的逐级审批

但这些场景并不完全一样。Web 中间件通常是线性链,每个节点决定是否 `next`。审批流可能是动态链,下一个节点由金额、部门、角色决定。错误处理链可能只找第一个能处理异常的 handler。

## 四、和装饰器、管道的边界

责任链和装饰器都可能长得像一层套一层。

装饰器强调"给对象叠加能力",通常每层都会调用被装饰对象。责任链强调"请求在多个处理者之间传递",某个节点可以不继续。

管道强调数据经过多个阶段转换,每阶段通常都会执行。责任链强调处理者有选择权,可以处理、跳过或终止。

所以判断关键是:你在组织能力叠加,还是在组织请求路由和处理控制权。

## 五、落地判断

适合责任链的信号:

- 多个处理步骤有统一输入输出
- 步骤顺序有意义,但组合会变化
- 某些步骤可能拦截请求
- 横切逻辑不该散落在业务方法中

不适合的信号:

- 步骤之间强依赖复杂中间结果
- 每一步都必须执行且只是数据转换
- 业务流程需要分支、并行、回滚、人工节点
- 调试链路比复用收益更重要

落地时要明确三件事:顺序怎么定义,错误怎么传播,上下文能不能被修改。很多链式代码难维护,不是模式错,而是这三件事没有约束。

## 六、代价与误用

责任链的代价是调用路径不再直观。一个请求为什么被拒绝,可能要看注册顺序和多个 handler 的逻辑。

常见误用:

第一,链上节点偷偷依赖前面节点的副作用,但没有类型或文档约束。后面换顺序就出 bug。

第二,把复杂业务流程硬塞进责任链。责任链适合线性传递,不适合复杂工作流。

第三,错误处理不统一。有的节点返回错误,有的节点抛异常,有的节点写 response 后继续 next,最后行为不可预测。

第四,顺序不可见。中间件注册散落在多个文件,真实顺序没人说得清。

## 七、一句话总结

责任链模式适合把一组可插拔、可短路的处理步骤串起来。它的关键不是链式写法,而是清楚定义顺序、终止条件和错误传播。
