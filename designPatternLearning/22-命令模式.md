# 命令模式:撤销、重做、队列化与操作日志

很多业务代码会把"判断能不能做"、"真正执行"、"记录日志"、"失败重试"写在同一个方法里。比如编辑器点击加粗、订单后台点击退款、运维平台点击重启机器,表面上都是一次调用,但后面很快会出现新要求:支持撤销、支持排队、支持审批、支持重放、支持操作日志。

变化点是"操作会越来越多,而且操作的执行方式会变化"。稳定点是"系统需要把一次操作当成一个对象来管理"。命令模式解决的就是这个问题。

## 一、问题

直接调用的问题在于,调用者只知道函数结果,不知道这次操作本身。比如:

```ts
editor.bold();
editor.insertText("hello");
```

如果要撤销,你得额外记很多上下文。如果要重做,你要知道上一次调用了什么。如果要把操作放进队列,函数调用本身也不好序列化。

命令模式把"请求"封装成对象:

```text
Invoker -> Command -> Receiver
```

调用者触发 Command,Command 再调用真正的业务对象 Receiver。

## 二、模式

命令模式的角色:

- Command:统一的命令接口,通常有 `execute`
- ConcreteCommand:具体命令,保存执行需要的数据
- Receiver:真正做事的对象
- Invoker:触发命令的人,比如按钮、队列、调度器

TypeScript 示例:

```ts
interface Command {
  execute(): void;
  undo(): void;
}

class Editor {
  text = "";

  insert(value: string) {
    this.text += value;
  }

  deleteLast(n: number) {
    this.text = this.text.slice(0, -n);
  }
}

class InsertTextCommand implements Command {
  constructor(private editor: Editor, private value: string) {}

  execute() {
    this.editor.insert(this.value);
  }

  undo() {
    this.editor.deleteLast(this.value.length);
  }
}

const editor = new Editor();
const history: Command[] = [];

const command = new InsertTextCommand(editor, "hello");
command.execute();
history.push(command);

history.pop()?.undo();
```

这里的重点不是多写一个类,而是"操作"有了身份。它可以被保存、排队、撤销、重试、审计。

## 三、落地判断

适合命令模式的场景:

- UI 操作需要撤销和重做
- 后台任务要排队执行
- 操作要写入日志,以后可以重放
- 请求要经过审批、限流、权限、重试
- 同一套触发入口要支持不同操作

典型落地形态包括:

- 编辑器里的 undo / redo stack
- 游戏里的输入指令
- CQRS 里的 Command
- 后台 Job / Task
- HTTP 请求进入应用层后的用例对象

命令模式和策略模式很像,都能把行为抽出去。区别是:策略强调"多种算法可替换",命令强调"一次请求可被对象化管理"。如果你关心选择哪种算法,更像策略。如果你关心排队、撤销、日志、重放,更像命令。

## 四、代价与误用

命令模式的代价是类或对象会变多。对于只有一两个简单按钮的页面,直接函数调用就够了。

常见误用:

第一,把每个 service 方法都包装成 Command。没有撤销、队列、审计、重试需求时,这通常只是样板代码。

第二,命令对象塞进太多业务逻辑。Command 应该表达一次操作并协调 Receiver,不应该变成新的上帝对象。

第三,撤销逻辑想得太简单。真实业务的 undo 往往不是把状态改回去,而是发起一条反向业务操作。比如退款不能简单 undo 成"未退款",可能要走补款、冲正或人工流程。

第四,把命令日志当成事件日志。命令表示"想做什么",事件表示"已经发生什么"。命令可能失败,事件不应该代表未发生的事实。

## 五、落地建议

如果只是前端本地交互,命令对象可以很轻,甚至是闭包:

```ts
type SimpleCommand = { do: () => void; undo: () => void };
```

如果是后端业务命令,要更重视输入校验、幂等 key、权限上下文、执行结果和审计字段。命令一旦进队列,就不能假设它只执行一次。

## 六、一句话总结

命令模式的价值不是"把方法换成类",而是让操作可以被保存、调度、撤销、重试和审计。只有当你需要管理操作本身时,它才值得引入。
