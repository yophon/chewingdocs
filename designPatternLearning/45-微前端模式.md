# 微前端模式:模块联邦、沙箱、路由隔离与组织边界

微前端最容易被误解成技术时髦词.很多团队一听"前端太大",第一反应就是拆微前端.结果原来只是一个仓库难维护,拆完变成十个应用一起难维护,还多了构建、发布、样式冲突、依赖版本和运行时通信问题.

微前端首先不是技术模式,而是**组织边界在前端运行时的投影**.

---

## 一、问题:大前端应用为什么会撑爆

单体前端会在这些情况下变痛:

- 多个业务团队共用一个仓库,发布互相等待.
- 一个团队改营销页,另一个团队改交易页,却要一起回归.
- 技术栈升级困难,任何依赖变化都影响全站.
- 权限、菜单、路由越来越多,应用启动变慢.
- 不同业务域的代码互相 import,边界消失.

稳定点是"用户看到的是一个产品",变化点是"不同业务团队、发布节奏、技术栈和生命周期不同".

微前端试图把变化点隔离:

```text
Shell / Host
  -> 登录态、全局导航、布局、公共能力

Remote / Micro App
  -> 独立业务域
  -> 独立构建
  -> 独立发布
  -> 受控接入 Shell
```

---

## 二、常见模式

**模式一:构建时集成**

多个包在构建时组合成一个应用,常见于 monorepo:

```text
packages/
  shell/
  order/
  user/
  shared-ui/
```

优点是运行时简单,类型和依赖容易检查.缺点是发布仍然可能绑定.

**模式二:运行时集成**

Shell 在运行时加载远程应用:

```ts
const OrderApp = lazy(() => import("order/App"));

function AppRoutes() {
  return <Route path="/orders/*" element={<OrderApp />} />;
}
```

模块联邦就是这类方案的典型实现.优点是独立部署能力强,缺点是运行时失败模式更多.

**模式三:iframe 隔离**

iframe 隔离最强,样式、脚本、全局变量天然分开.缺点是体验和通信成本高,路由、弹窗、埋点、登录态都要额外处理.

---

## 三、关键边界:路由、沙箱、依赖和通信

微前端落地要先定四个边界.

**路由边界**

```text
/orders/*  -> Order App
/users/*   -> User App
/billing/* -> Billing App
```

每个子应用只管理自己的子路由,不要抢全局路由.

**沙箱边界**

要防止子应用污染全局:

- CSS 命名冲突.
- window 全局变量.
- 事件监听不卸载.
- 定时器不清理.

**依赖边界**

共享依赖不是越多越好.React、路由库、设计系统可以共享,业务工具不要随便共享,否则又会形成隐形单体.

**通信边界**

优先用明确协议:

```ts
type ShellBridge = {
  getAccessToken(): Promise<string>;
  navigate(path: string): void;
  emit(event: "cart:changed", payload: { count: number }): void;
};
```

不要让子应用直接 import Shell 内部模块.

## 四、落地判断

适合微前端的信号:

- 至少两个以上独立团队长期并行开发.
- 业务域边界清楚,例如订单、会员、财务.
- 发布节奏确实不同,且互相等待成本高.
- 已经有稳定的设计系统、登录态、监控、发布平台.
- 团队能承担运行时治理成本.

不适合的信号:

- 只是代码目录乱.
- 只是构建慢,但团队和发布边界没有分裂.
- 只是想让不同页面用不同框架.
- 公共能力还没稳定,Shell 自己都经常变.
- 没有自动化测试和灰度发布能力.

很多时候,先做 monorepo、模块边界、路由懒加载、包拆分,就能解决 70% 的问题.

## 五、代价与误用

微前端引入的是分布式系统问题:

- 远程资源加载失败怎么办.
- Host 和 Remote 版本不兼容怎么办.
- 共享依赖重复加载怎么办.
- 子应用样式污染怎么办.
- 跨应用跳转和埋点怎么统一.
- 本地开发怎么同时启动多个应用.

常见误用:

**误用一:用微前端替代模块化**.模块边界都没想清楚,运行时拆分只会放大混乱.

**误用二:共享包变成新的单体核心**.所有业务都依赖 `shared-business`,最后改共享包比改单体还难.

**误用三:通信变成全局 EventBus**.事件名没人管,数据结构没人管,调用链不可追踪.

**误用四:忽略用户体验**.子应用加载慢、样式跳动、错误页不统一,用户不会关心你是不是微前端.

---

## 六、一句话总结

> **微前端解决的是组织和发布边界,不是目录整理问题**.只有当团队、业务域和发布节奏真的需要隔离时,模块联邦、沙箱和路由隔离才值得付出成本.
