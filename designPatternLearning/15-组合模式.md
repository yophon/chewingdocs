# 组合模式:树形结构,菜单,文件系统与 UI 组件

很多业务对象天然是树形的:部门和员工、菜单和菜单项、文件夹和文件、评论和回复、页面组件和子组件.

如果你总是区分"这是单个节点"还是"这是容器节点",调用方会写很多分支:

```ts
if (node.type === "file") {
  size += node.size
} else {
  for (const child of node.children) {
    size += ...
  }
}
```

节点类型越多,分支越多.变化点是:树上的节点种类和层级会变化.稳定点是:调用方通常只想对树执行统一操作.

---

## 一、问题

以文件系统为例,文件有大小,文件夹的大小是所有子项大小之和.

如果调用方需要知道每种节点:

```ts
function totalSize(node: FileNode | FolderNode): number {
  if (node.kind === "file") return node.size
  return node.children.reduce((sum, child) => sum + totalSize(child), 0)
}
```

这段代码能跑,但问题是操作一多,重复递归就会散落到各处.比如渲染、搜索、权限检查、导出结构,每个地方都要理解树的内部形态.

---

## 二、模式

组合模式的核心是:让叶子节点和容器节点实现同一个接口,调用方用统一方式处理单个对象和对象组合.

结构可以简化成:

```text
Component
  -> Leaf
  -> Composite -> Component[]
```

`Leaf` 是叶子.`Composite` 内部包含一组 `Component`,所以可以递归组成树.

---

## 三、示例

```ts
interface Entry {
  name: string
  size(): number
}

class FileEntry implements Entry {
  constructor(public name: string, private bytes: number) {}

  size() {
    return this.bytes
  }
}

class FolderEntry implements Entry {
  private children: Entry[] = []

  constructor(public name: string) {}

  add(entry: Entry) {
    this.children.push(entry)
  }

  size() {
    return this.children.reduce((sum, child) => sum + child.size(), 0)
  }
}

const root = new FolderEntry("root")
root.add(new FileEntry("a.txt", 100))

const src = new FolderEntry("src")
src.add(new FileEntry("index.ts", 200))
root.add(src)

console.log(root.size())
```

调用方只看 `Entry`,不需要关心它到底是文件还是文件夹.

如果要渲染树,也可以继续让公共操作停留在接口上:

```ts
interface RenderableEntry extends Entry {
  render(indent?: number): string
}

class RenderableFile extends FileEntry implements RenderableEntry {
  render(indent = 0) {
    return `${" ".repeat(indent)}- ${this.name}`
  }
}
```

重点不是把所有操作都塞进 `Entry`,而是把"调用方希望统一处理的操作"抽出来.不同操作可以有不同接口,不要为了一个统一接口牺牲类型表达.

---

## 四、落地判断

组合模式适合稳定的树形结构:

- 文件系统:文件和文件夹
- 菜单系统:菜单组和菜单项
- 组织架构:部门和员工
- UI 组件:容器组件和叶子组件
- AST:表达式节点和组合表达式

判断是否该用组合模式,看两个问题:

1. 调用方是否经常需要递归处理树
2. 叶子和容器是否可以抽出共同操作

如果共同操作很少,或者容器和叶子差异巨大,强行统一接口会让抽象变别扭.

从变化点看,组合模式保护的是"树的形状会变".调用方不应该随着层级增加而增加分支.今天文件夹下面有文件,明天文件夹下面还有快捷方式、挂载目录、压缩包视图,只要它们能响应同一个操作,调用方就可以继续递归处理 `Entry`.

组合模式也常和访问者模式配合.当树结构稳定、操作经常新增时,访问者更适合;当操作稳定、节点类型经常新增时,组合本身就够用.不要把所有树形问题都压到一个模式里.

---

## 五、测试与重构建议

组合模式的测试要覆盖叶子、容器和多层嵌套三类情况.

```ts
const root = new FolderEntry("root")
root.add(new FileEntry("a.txt", 100))

const src = new FolderEntry("src")
src.add(new FileEntry("index.ts", 200))
src.add(new FileEntry("app.ts", 300))
root.add(src)

console.log(root.size()) // 600
```

这类测试很简单,但很有价值:它能防止你在新增节点类型时破坏递归语义.

重构时,先找重复递归.如果多个地方都在 `if file else folder` 地遍历,说明共同操作可以向节点对象移动.步骤通常是:

1. 抽出最小公共接口,比如 `size()` 或 `render()`.
2. 让叶子节点实现这个接口.
3. 让容器节点保存 `Component[]`,并在自己的方法里递归调用子节点.
4. 把调用方从类型判断改成调用公共方法.

不要急着把 `add`, `remove`, `getChildren` 放进公共接口.这些是容器管理能力,不是叶子和容器的共同能力.如果调用方确实需要遍历孩子,可以给容器单独提供只读方法,或者提供一个专门的遍历器.

---

## 六、适用边界

组合模式的代价是接口容易被拉宽.为了让文件和文件夹实现同一个接口,有人会把 `add`, `remove`, `children` 都放到 `Entry` 上.结果文件也有 `add` 方法,调用后只能抛异常.

组合模式适合整体和部分在某些操作上可以等价看待的场景.如果叶子和容器只有外形像树,但业务动作完全不同,就不该强行组合.例如电商分类树中的"类目"和"商品"可能都挂在树上,但类目用于导航,商品用于交易,它们未必应该实现同一个大接口.

性能边界也要提前想清楚.递归计算 `size()` 在小树上很自然,在百万节点树上可能要缓存、增量更新或异步计算.组合模式让调用方简单,但不自动解决大树性能问题.

---

## 七、代价与误用

更稳妥的做法是:公共接口只放真正共同的操作,比如 `size`, `render`, `validate`.只有容器需要的管理方法,放在容器类自己身上.

常见误用还有两种.

第一种是把所有层级差异都藏起来.调用方有时确实需要知道节点类型,比如只有文件能下载.此时保留类型判断并不丢人.

第二种是树太大时递归没有边界.深树可能导致栈溢出,大树可能导致性能问题,需要迭代遍历、缓存或分页加载.

---

## 八、一句话总结

> 组合模式让调用方用同一种方式处理叶子和容器,很适合树形结构,但公共接口必须克制,不要为了统一而制造无意义方法.
