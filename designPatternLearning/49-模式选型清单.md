# 模式选型清单:需求变化点、团队能力、测试成本与性能约束

设计模式选型最怕一句话:"这里是不是该用某某模式?" 更好的问法是:"这段代码未来最可能怎么变,现在的结构能不能承受?" 模式不是答案库,而是权衡工具.

这一篇给一份工程判断清单,用于写方案、做重构、过代码评审.

---

## 一、问题:只按模式名字选型会失真

看到很多 if-else,不一定要策略模式.看到复杂对象,不一定要建造者.看到跨模块调用,不一定要事件驱动.

例如:

```ts
if (country === "US") return usTax(order);
if (country === "JP") return jpTax(order);
if (country === "CN") return cnTax(order);
```

如果国家规则一年改一次,三行分支很清楚.如果每周增加地区、规则由税务团队维护、需要灰度和审计,策略或规则引擎才有价值.

模式选型要先看变化压力,不是看代码形状.

---

## 二、清单一:变化点是否真实

先问:

- 这个变化已经发生过几次?
- 下次变化会改哪里?
- 变化是增加新类型,还是修改旧流程?
- 变化来自业务、技术栈、性能,还是组织协作?
- 不抽象时,下一次改动成本是多少?

判断:

```text
变化低频 + 逻辑简单 -> 保持直接写法
变化高频 + 分支扩展 -> 策略 / 规则表
变化高频 + 流程扩展 -> 模板方法 / 责任链
变化高频 + 外部接口变 -> 适配器 / 外观
变化高频 + 状态迁移变 -> 状态模式 / 状态机
```

如果你只能说"以后可能会变",通常还不够.

---

## 三、清单二:稳定点是否清楚

抽象需要稳定点.没有稳定点,抽象只是猜测.

例如支付:

```text
稳定点:
  - 发起支付需要 PayCommand
  - 返回 PayResult
  - 失败要有错误码

变化点:
  - Stripe / PayPal / 微信支付
  - 签名算法
  - 回调验签
```

这时 `PaymentGateway` 是合理边界:

```ts
type PaymentGateway = {
  pay(command: PayCommand): Promise<PayResult>;
};
```

如果连 `PayCommand` 和 `PayResult` 都不稳定,先不要急着抽接口,应该先整理业务语言.

---

## 四、清单三:团队是否能维护这个模式

一个模式是否合适,不只看技术,还看团队.

要问:

- 团队是否熟悉这种抽象?
- 新人能否从命名看懂职责?
- 代码生成、依赖注入、框架约定是否统一?
- 评审者能否判断新增实现是否符合边界?
- 文档和示例是否足够?

有些模式在高手手里是清晰边界,在团队不熟时是迷宫.比如 DDD、CQRS、事件驱动、微前端,都需要治理能力.

局部策略、适配器、外观、建造者通常更容易落地.全局架构模式要谨慎.

---

## 五、清单四:测试成本有没有下降

好的模式应该让关键逻辑更容易测.

例如把第三方 SDK 包起来:

```ts
class OrderService {
  constructor(private payment: PaymentGateway) {}

  async pay(order: Order) {
    const result = await this.payment.pay({ orderId: order.id });
    order.markPaid(result.transactionId);
  }
}
```

测试时可以替换 `PaymentGateway`:

```ts
const fakePayment: PaymentGateway = {
  pay: async () => ({ transactionId: "tx_1" }),
};
```

如果引入模式后:

- 测试要 mock 更多东西.
- 调用链更长.
- 行为分散到多个生命周期钩子.
- 很难定位哪一层错.

那这个模式至少没有在测试维度上创造价值.

---

## 六、清单五:性能和运行时约束

模式会改变性能形态:

- 代理可能增加调用开销.
- 装饰器链可能增加栈深和延迟.
- 观察者可能导致重复通知.
- 状态管理可能导致过度渲染.
- 微前端可能增加资源加载和初始化成本.
- 享元会节省内存,但增加共享对象管理复杂度.

大多数业务系统不该为了微小性能损耗放弃清晰结构,但性能敏感路径要单独评估.

评估方式:

```text
是否在热路径?
数据规模多大?
是否会被频繁调用?
是否影响首屏或交互延迟?
是否可以通过缓存、懒加载、批处理补偿?
```

## 七、选型输出模板

做方案时可以这样写:

```text
问题:
  订单状态判断散落在 5 个 service 中,新增状态需要改多处.

变化点:
  状态迁移规则和可执行动作会继续增加.

稳定点:
  订单只能按有限状态流转,每次流转都要记录审计.

候选:
  继续 if-else / 状态模式 / 状态机库.

选择:
  先使用显式状态机配置,暂不引入外部状态机库.

原因:
  状态数量有限,团队不熟悉库,但需要集中校验和测试.

代价:
  增加一份状态表,新增动作必须同步测试.
```

这个模板比"我们使用状态模式"更有工程价值.

---

## 八、一句话总结

> **模式选型不是找最高级的模式,而是找最便宜地保护真实变化点的结构**.变化、稳定、团队、测试、性能这五项过不了,模式名字再正确也没用.
