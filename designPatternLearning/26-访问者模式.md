# 访问者模式:AST、编译器 Pass 与表达式问题

有些系统的对象结构很稳定,但对这些对象的操作经常增加。典型例子是 AST:节点类型可能长期稳定,比如数字、加法、变量、函数调用;但你会不断增加操作,比如 pretty print、类型检查、求值、代码生成、静态分析。

如果把所有操作都写进节点类,节点会越来越胖。变化点是"对结构的操作会增加",稳定点是"结构里的元素类型相对稳定"。访问者模式解决的就是这类问题。

## 一、问题

假设表达式节点这样写:

```ts
class Add {
  eval() {}
  print() {}
  typeCheck() {}
  generateCode() {}
}
```

每增加一种操作,所有节点类都要加方法。节点知道了太多外部任务。访问者模式反过来:节点只负责接待 visitor,具体操作放到 visitor 里。

```text
Element.accept(visitor)
Visitor.visitElement(element)
```

## 二、模式

角色:

- Element:被访问的元素,提供 `accept`
- ConcreteElement:具体元素
- Visitor:访问者接口,为每种元素提供 visit 方法
- ConcreteVisitor:具体操作

TypeScript 示例:

```ts
interface Expr {
  accept<T>(visitor: ExprVisitor<T>): T;
}

interface ExprVisitor<T> {
  visitNumber(expr: Num): T;
  visitAdd(expr: Add): T;
}

class Num implements Expr {
  constructor(public value: number) {}
  accept<T>(visitor: ExprVisitor<T>): T {
    return visitor.visitNumber(this);
  }
}

class Add implements Expr {
  constructor(public left: Expr, public right: Expr) {}
  accept<T>(visitor: ExprVisitor<T>): T {
    return visitor.visitAdd(this);
  }
}

class EvalVisitor implements ExprVisitor<number> {
  visitNumber(expr: Num) {
    return expr.value;
  }

  visitAdd(expr: Add) {
    return expr.left.accept(this) + expr.right.accept(this);
  }
}

const expr = new Add(new Num(1), new Num(2));
console.log(expr.accept(new EvalVisitor()));
```

新增 `PrintVisitor` 时,不需要改 `Num` 和 `Add` 的内部逻辑。

## 三、表达式问题

访问者模式的本质取舍可以用"表达式问题"理解:

- 容易新增操作,例如新增 Eval、Print、TypeCheck
- 不容易新增元素类型,例如新增 Multiply 要改 Visitor 接口和所有实现

如果你的系统经常新增节点类型,访问者会很痛苦。如果你的节点类型稳定,操作经常增加,访问者很合适。

这也是为什么访问者常见于编译器、解释器、静态分析器:语言语法节点相对稳定,对 AST 的 pass 会很多。

## 四、落地判断

适合访问者模式的信号:

- 对象结构稳定,元素类型有限
- 对同一结构有很多互不相同的操作
- 不想把分析、导出、校验等逻辑塞进元素类
- 需要对树结构做多轮 pass

不适合的信号:

- 元素类型频繁变化
- 操作很少,放在对象方法里就够
- 语言本身有强大的模式匹配和代数数据类型
- 团队不熟悉 double dispatch,维护成本高

现代 TypeScript 有时会用判别联合替代经典访问者:

```ts
type Expr =
  | { kind: "num"; value: number }
  | { kind: "add"; left: Expr; right: Expr };

function evalExpr(expr: Expr): number {
  switch (expr.kind) {
    case "num": return expr.value;
    case "add": return evalExpr(expr.left) + evalExpr(expr.right);
  }
}
```

这种写法更轻,但新增操作时仍然要写新的 switch。选择哪种,取决于语言能力和团队习惯。

## 五、代价与误用

访问者模式的代价很明显:样板代码多,调用路径绕,新增元素类型很麻烦。

常见误用:

第一,对象结构还没稳定就上访问者。结果每加一个节点,所有 visitor 都要补方法。

第二,为了访问私有数据破坏封装。visitor 需要足够数据,但不应该逼元素暴露所有内部细节。

第三,把业务对象做成访问者结构。普通订单、用户、商品不是 AST,不要轻易套访问者。

第四,忽略语言特性。Rust/Scala/Kotlin/TypeScript 在某些场景下用 enum、sealed class、pattern matching 会更直接。

## 六、一句话总结

访问者模式适合"结构稳定,操作多变"的对象体系。它是编译器和 AST 场景的利器,但在普通业务系统里很容易变成难懂的样板代码。
