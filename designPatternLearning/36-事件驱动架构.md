# 事件驱动架构:领域事件、集成事件与最终一致性

很多业务流程一开始都是同步调用串起来的:下单成功后扣库存、发优惠券、发短信、写积分、通知仓库。第一版很好理解,但后面每加一个后续动作,下单接口就更慢、更脆弱。短信服务挂了,下单也失败;积分服务超时,订单事务被拖住;新增一个营销动作,还要改订单核心代码。

事件驱动架构要解决的问题是:**把已经发生的事实发布出去,让后续反应从主流程中解耦**。

---

## 一、问题:同步流程把无关变化绑在一起

常见代码:

```ts
async function createOrder(cmd: CreateOrderCommand) {
  const order = await orderService.create(cmd);
  await inventoryService.deduct(order.items);
  await couponService.markUsed(cmd.couponId);
  await smsService.sendOrderCreated(order.userId);
  await pointsService.add(order.userId, order.amount);
  return order;
}
```

这里有两类逻辑:

- 主流程:创建订单,扣库存,保证下单成立.
- 后续反应:短信、积分、通知、统计、营销.

如果全部同步串行,任何一个后续反应都会影响主流程稳定性。变化点是订阅方和处理方式,稳定点是"订单已创建"这个事实。

---

## 二、模式:用事件表达已经发生的事实

事件驱动的核心结构:

```text
Producer
  |
  | publish OrderCreated
  v
Event Bus / Message Broker
  |          |           |
  v          v           v
Inventory  Notification  Analytics
Consumer   Consumer      Consumer
```

事件应该用过去式表达事实:

```ts
type OrderCreated = {
  eventId: string;
  occurredAt: string;
  orderId: string;
  userId: string;
  amount: number;
};
```

生产者:

```ts
class CreateOrderUseCase {
  constructor(
    private orders: OrderRepository,
    private events: EventPublisher,
  ) {}

  async execute(cmd: CreateOrderCommand) {
    const order = Order.create(cmd);
    await this.orders.save(order);

    await this.events.publish({
      eventId: crypto.randomUUID(),
      occurredAt: new Date().toISOString(),
      orderId: order.id,
      userId: order.userId,
      amount: order.amount,
    });

    return order.id;
  }
}
```

消费者:

```ts
class SendOrderSms {
  async handle(event: OrderCreated) {
    await sms.send(event.userId, `order ${event.orderId} created`);
  }
}
```

## 三、领域事件和集成事件的边界

事件至少要分清两类。

领域事件:

```text
发生在领域模型内部
表达业务事实
主要用于同一应用内部解耦
例:OrderPaid, ContractApproved
```

集成事件:

```text
跨服务或跨系统发布
是对外契约
需要版本、兼容性、幂等
例:order.paid.v1, payment.completed.v2
```

领域事件可以比较贴近业务对象,集成事件必须更谨慎,因为一旦发布就有外部消费者依赖。不要把数据库表结构直接当事件 schema 发出去,否则表结构变化会变成跨系统破坏。

常见落地结构:

```text
Domain Model -> Domain Events
Use Case     -> collect events
Outbox       -> persist events with transaction
Publisher    -> publish integration events
Consumers    -> idempotent handling
```

Outbox 模式很重要:业务数据和待发布事件先在同一个数据库事务里落库,再由后台任务可靠发布,避免"订单保存成功但事件发布失败"。

## 四、落地判断:什么时候用事件

适合事件驱动的场景:

- 一个业务事实会触发多个后续动作.
- 后续动作不应该阻塞主流程.
- 订阅方经常变化,生产方不应该频繁修改.
- 系统可以接受最终一致.
- 需要审计、异步扩展、削峰或跨服务解耦.

不适合事件驱动的场景:

- 调用方必须马上知道结果.
- 强一致事务内必须完成所有步骤.
- 事件顺序和幂等无法保证,但业务又无法容错.
- 团队没有消息堆积、重试、死信、追踪和告警能力.

一个实用原则是:**主流程内的强一致动作谨慎异步,主流程外的副作用优先事件化**。比如支付扣款通常不能随便异步;支付成功后的短信、积分、报表可以异步。

## 五、代价与误用

事件驱动架构最大的代价是从"调用栈可见"变成"链路分散"。以前一个接口超时能看堆栈,现在要查消息、消费位点、重试记录、死信队列和 trace。

常见误用之一是事件命名成命令。`SendSms`、`DeductStock` 是命令,不是事件;`OrderCreated`、`StockReserved` 才是事实。命令要求某个接收者去做事,事件只是宣布发生了什么。

常见误用之二是把事件当 RPC。生产者发布事件后不应该等待每个消费者都完成,否则又退回同步耦合。

常见误用之三是忽略幂等。消息至少一次投递很常见,消费者必须能处理重复事件:

```ts
async function handle(event: OrderPaid) {
  if (await processed.exists(event.eventId)) return;
  await points.add(event.userId, event.amount);
  await processed.save(event.eventId);
}
```

## 六、与前端模式的边界

事件驱动架构是应用和系统层面的模式,重点是跨模块、跨服务、跨进程的事实传播和最终一致。前端里的 DOM event、组件 emit、RxJS stream 也叫事件,但它们主要服务于 UI 交互和状态传播。名字相同,问题层级不同。

判断方式:如果事件要进入消息队列、影响业务一致性和跨服务契约,这是事件驱动架构;如果事件只在组件树或浏览器运行时传播,它属于前端交互或响应式模式。

---

## 一句话总结

事件驱动架构用事实事件解耦主流程和后续反应,适合异步扩展和最终一致场景,但它会把复杂度转移到幂等、顺序、追踪和补偿上。
