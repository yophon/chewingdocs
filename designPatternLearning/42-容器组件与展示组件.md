# 容器组件与展示组件:什么时候有用,什么时候过时

组件写乱的典型过程是:一开始只是渲染一个列表,后来要调接口、读路由、接埋点、弹 toast、判断权限、处理 loading,最后一个 `UserList` 既像页面,又像 service,还像 UI 组件.改样式怕影响数据,改数据怕影响交互.

容器组件与展示组件的价值,就是把"怎么拿数据"和"怎么显示数据"分开.但它不是永远都要套一层,也不是现代 Hooks 出现后就完全过时.

---

## 一、问题:一个组件承担太多变化

坏味道通常长这样:

```ts
function UserList() {
  const { role } = useAuth();
  const params = useSearchParams();
  const [selectedId, setSelectedId] = useState<string | null>(null);
  const users = useUsersQuery({ q: params.get("q") ?? "" });

  useEffect(() => {
    analytics.track("user_list_view");
  }, []);

  if (users.isLoading) return <Spinner />;

  return (
    <div>
      {users.data.map((user) => (
        <button
          disabled={role !== "admin"}
          onClick={() => setSelectedId(user.id)}
        >
          {user.name}
        </button>
      ))}
    </div>
  );
}
```

这段代码的问题是变化点混在一起:

- 数据来源会变:REST、GraphQL、本地 mock、缓存.
- UI 表现会变:列表、表格、卡片、空状态.
- 权限规则会变.
- 埋点和副作用会变.

稳定点是"用户列表需要被展示",变化点是"数据怎么来、交互怎么处理、外部副作用怎么接入".

---

## 二、模式:容器负责连接世界,展示负责表达 UI

经典分工:

```text
Container Component
  -> 读路由、调接口、读 store、处理副作用
  -> 组装 props
  -> 不关心具体视觉细节

Presentational Component
  -> 接收 props
  -> 渲染 UI
  -> 通过 callback 抛出事件
  -> 不直接依赖路由、接口、全局 store
```

重构后:

```ts
type UserListViewProps = {
  users: User[];
  canSelect: boolean;
  selectedId: string | null;
  onSelect: (id: string) => void;
};

function UserListView(props: UserListViewProps) {
  return (
    <div>
      {props.users.map((user) => (
        <button
          key={user.id}
          disabled={!props.canSelect}
          data-active={props.selectedId === user.id}
          onClick={() => props.onSelect(user.id)}
        >
          {user.name}
        </button>
      ))}
    </div>
  );
}
```

容器组件:

```ts
function UserListContainer() {
  const { role } = useAuth();
  const [selectedId, setSelectedId] = useState<string | null>(null);
  const users = useUsersQuery();

  if (users.isLoading) return <Spinner />;

  return (
    <UserListView
      users={users.data ?? []}
      canSelect={role === "admin"}
      selectedId={selectedId}
      onSelect={setSelectedId}
    />
  );
}
```

现在 UI 可以单测、Storybook 展示、复用到别的页面.容器负责接入真实世界.

## 三、什么时候仍然有用

这个模式在三类场景很有用:

**第一,复杂页面的核心视图**.比如订单详情、权限配置、数据看板.展示组件可以稳定表达 UI,容器组件可以频繁替换数据来源.

**第二,跨端或跨场景复用**.同一个 `UserListView` 可以在管理后台、选择弹窗、只读详情中复用.

**第三,需要独立测试视觉状态**.展示组件只靠 props,可以枚举 loading、empty、error、disabled、selected 等状态.

但小组件不必强行拆:

```ts
function LikeButton({ liked, onToggle }: Props) {
  return <button onClick={onToggle}>{liked ? "Liked" : "Like"}</button>;
}
```

这种已经足够清楚,再拆容器只会增加跳转成本.

---

## 四、为什么有人说它过时了

在 class component 时代,容器组件常常是逻辑复用的主要办法.Hooks / Composable 出现后,很多数据逻辑可以抽成函数:

```ts
function useUserList() {
  const { role } = useAuth();
  const users = useUsersQuery();

  return {
    users: users.data ?? [],
    loading: users.isLoading,
    canSelect: role === "admin",
  };
}
```

然后页面可以直接组合:

```ts
function UserListPage() {
  const model = useUserList();
  if (model.loading) return <Spinner />;
  return <UserListView {...model} onSelect={() => {}} />;
}
```

所以过时的不是"分离数据与展示",而是"为了分离而制造很多薄容器文件".现代写法更倾向于:

- 逻辑复用交给 Hooks / Composable.
- 页面编排留在 Page.
- 复杂 UI 仍然抽展示组件.

## 五、落地判断

可以用三个问题判断要不要拆:

1. 这个组件是否直接依赖接口、路由、store、权限、埋点?
2. 这个组件是否有多种视觉状态需要独立验证?
3. 这个 UI 是否可能被不同数据来源复用?

如果答案有两个是"是",拆成容器和展示通常值得.

拆的时候不要机械命名成 `XxxContainer` 和 `XxxPresenter`.更自然的命名是:

- `UserListPage`: 页面容器.
- `UserListSection`: 页面中的业务区块.
- `UserListView`: 纯展示视图.
- `useUserList`: 数据与交互模型.

## 六、代价与误用

代价是文件数量增加,调用链变长,简单需求会显得绕.

常见误用:

**误用一:展示组件也偷偷读全局状态**.一旦展示组件内部调用 `useAuth` 或 `useStore`,它就不再纯展示.

**误用二:容器只是转发 props**.如果容器没有隔离外部依赖,只是 `<View {...props} />`,多半没价值.

**误用三:把所有组件都二分**.按钮、输入框、简单卡片不需要这种拆法.

更务实的原则是:**组件越靠近页面,越可能需要容器思维;组件越靠近基础 UI,越应该保持纯展示**.

---

## 七、一句话总结

> **容器组件与展示组件不是命名套路,而是边界套路**.当数据来源、副作用、权限和 UI 表达开始互相拖累时,把连接世界的代码和表达界面的代码分开.
