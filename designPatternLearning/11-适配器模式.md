# 适配器模式:旧接口,新接口与第三方 SDK 包装

很多代码变乱,不是因为业务本身复杂,而是因为系统边界一直在变.

一开始你直接调用短信平台:

```ts
smsClient.send(phone, text)
```

后来供应商换了,新 SDK 叫 `sendMessage`,参数变成对象,返回值也不一样.最差的写法是全项目搜索 `smsClient.send`,然后在每个调用点写兼容逻辑.这样做短期能跑,长期会把第三方 SDK 的细节扩散到业务代码里.

这里的变化点是:外部系统的接口形态会变.稳定点是:业务只需要表达"发送通知"这个动作.

---

## 一、问题

没有适配器时,业务代码会被外部接口牵着走:

```ts
async function register(phone: string) {
  await oldSms.send(phone, "welcome")
}
```

如果换成新 SDK:

```ts
await newSms.sendMessage({ to: phone, body: "welcome" })
```

修改点就会散落在注册、登录、找回密码、营销活动等地方.真正危险的不是改一次,而是以后每换一次渠道、每新增一次降级逻辑,业务代码都要被翻出来改.

---

## 二、模式

适配器模式的核心是:让业务依赖一个稳定的目标接口,把旧接口、新接口、第三方接口包在适配器里.

结构可以简化成:

```text
业务代码 -> Target 接口 -> Adapter -> 第三方 SDK / 旧系统
```

`Target` 是你希望业务看到的接口.`Adapter` 负责把这个接口翻译成外部系统能理解的调用.

---

## 三、示例

```ts
interface Notifier {
  send(to: string, message: string): Promise<void>
}

class SmsSdk {
  async sendMessage(req: { mobile: string; content: string }) {
    console.log("send sms", req.mobile, req.content)
  }
}

class SmsAdapter implements Notifier {
  constructor(private sdk: SmsSdk) {}

  async send(to: string, message: string): Promise<void> {
    await this.sdk.sendMessage({ mobile: to, content: message })
  }
}

async function register(notifier: Notifier, phone: string) {
  await notifier.send(phone, "welcome")
}
```

业务函数 `register` 不知道 SDK 的参数名,也不关心短信平台怎么鉴权、怎么返回错误.它只依赖 `Notifier`.

---

## 四、落地判断

适配器适合出现在系统边界:

- 调第三方 SDK,比如支付、短信、地图、对象存储
- 对接旧系统,比如老接口返回 XML,新系统想用 JSON 模型
- 迁移框架,比如从旧 HTTP client 切到新 HTTP client
- 多渠道统一接口,比如短信、邮件、站内信都包装成 `Notifier`

判断是否应该写适配器,看两个问题:

1. 外部接口是否由别人控制
2. 业务里是否已经出现了外部接口的参数名、异常类型、返回结构

如果答案都是 yes,适配器通常值得加.

测试上,适配器也能让业务更容易测.业务测 `Notifier` 的假实现,适配器单独测和真实 SDK 的映射关系.

还有一个实用判断:当你发现业务代码里开始出现 `mobile`, `accessKey`, `vendorCode`, `rawResponse`, `errCode` 这类外部供应商语言时,边界已经泄漏了.适配器不是为了让代码更"面向对象",而是为了把语言翻译回来:业务讲业务语言,外部系统讲外部语言,翻译层放在边界.

适配器也可以分两层:

```text
业务接口 Notifier
  -> SmsNotifierAdapter
      -> TencentSmsClient / AliyunSmsClient
```

业务层只依赖 `Notifier`.适配器内部再依赖具体 SDK.以后供应商变化,最多新增一个适配器,调用方不需要感知.

---

## 五、测试与重构建议

适配器的测试重点不是"短信真的发出去了",而是映射关系是否正确.可以把测试拆成三类:

1. 业务测试:给 `register` 传入一个假的 `Notifier`,断言它会发送正确的欢迎语.
2. 适配器单元测试:用假的 SDK 或 mock SDK,断言 `send(to, message)` 会被转换成 SDK 要求的参数.
3. 少量集成测试:在测试环境调用真实供应商沙箱,验证鉴权、错误码和限流行为.

重构时可以按这个顺序做:

```ts
interface Notifier {
  send(to: string, message: string): Promise<void>
}

class FakeNotifier implements Notifier {
  sent: Array<{ to: string; message: string }> = []

  async send(to: string, message: string) {
    this.sent.push({ to, message })
  }
}
```

先抽出业务真正需要的目标接口,再给旧 SDK 写一个适配器,最后逐步把调用方从 `oldSms` 改成 `Notifier`.不要一开始就设计一个覆盖所有供应商能力的大接口.目标接口越接近业务用例,适配器越稳.

如果迁移压力很大,可以让适配器先包住旧 SDK,保持行为不变;等调用方都切到目标接口后,再把内部实现换成新 SDK.这样风险比"全项目搜索替换"低得多.

---

## 六、适用边界

适配器的代价是多一层间接调用.对大多数业务系统来说,性能损耗可以忽略,真正的代价是维护额外接口.

适配器适合外部接口不稳定、不可控或与业务语言不一致的地方.如果某个 SDK 只在一个脚本里调用一次,而且没有复用和迁移压力,直接调用通常更清楚.如果差异不是接口形态,而是算法策略不同,比如同一个价格要按多种规则计算,更像策略模式.如果你要在调用前后加缓存、日志、重试,装饰器或代理可能更合适.

适配器还要避免过度承诺.比如短信供应商有模板短信、营销短信、国际短信、签名管理、回执查询等能力,业务当前只需要发送验证码,那 `Notifier` 就不该提前把所有能力都塞进去.稳定接口应该来自当前业务边界,不是来自供应商功能清单.

---

## 七、代价与误用

常见误用有三种.

第一种是为每个小函数都写适配器.如果外部调用只出现一次,也没有变化压力,直接调用更简单.

第二种是适配器泄漏第三方概念.比如 `Notifier.send` 返回 `TencentSmsResponse`,这等于把 SDK 细节又暴露回业务层.

第三种是把适配器写成万能胶水层,里面混入鉴权、重试、限流、日志、业务规则.适配器应该主要负责接口转换,复杂横切逻辑可以交给装饰器、代理或中间件.

---

## 八、一句话总结

> 适配器模式不是为了多写一层,而是把不稳定的外部接口挡在系统边界外,让业务代码只依赖自己认可的稳定接口.
