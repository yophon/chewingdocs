# 单向数据流:Flux / Redux / Elm 架构的核心约束

前端状态最容易坏在"谁都能改"。组件 A 改全局变量,组件 B 在 watch 里补一刀,组件 C 收到事件后再改回来。页面最后显示错了,你很难回答:这个状态到底是谁改的,为什么改,改之前是什么。

单向数据流要解决的问题是:**让状态变化有唯一入口和可追踪路径,用约束换可预测性**。

---

## 一、问题:双向和隐式更新让状态不可追踪

小页面里,父子组件互相传值、事件总线、双向绑定都很方便。问题出现在状态变多后:

```text
SearchBox  -> 改 query
FilterBar  -> 改 filters
Table      -> 改 selectedRows
Modal      -> 改 currentOrder
Watcher    -> query 变化时重置 page
EventBus   -> order.updated 后刷新列表
```

任何一个状态变化都可能触发其他变化。变化点是用户操作和异步返回,稳定点是"状态应该如何从旧值变成新值"。单向数据流把这个过程固定下来。

---

## 二、模式:Action -> Update -> State -> View

Flux / Redux / Elm 的共同核心:

```text
View
  |
  | dispatch action
  v
Update / Reducer
  |
  | produce new state
  v
State / Store
  |
  | render
  v
View
```

Redux 风格 TypeScript 示例:

```ts
type State = {
  count: number;
};

type Action =
  | { type: "increment" }
  | { type: "add"; value: number };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case "increment":
      return { count: state.count + 1 };
    case "add":
      return { count: state.count + action.value };
  }
}
```

View 不能随便改 state,只能 dispatch action。reducer 根据旧 state 和 action 计算新 state。这样每一次变化都有记录。

Elm 架构更明确:

```text
Model  当前状态
Msg    发生的消息
Update 如何从 Model + Msg 得到新 Model
View   如何把 Model 渲染成界面
```

## 三、落地判断:什么时候需要这种约束

单向数据流适合:

- 多个组件共享同一份状态.
- 状态变化来自多个入口:用户操作、WebSocket、定时刷新、路由.
- 需要撤销/重做、时间旅行、操作日志或可回放.
- 页面状态复杂,调试"谁改了它"很痛.
- 团队协作时需要统一状态修改规则.

不一定需要:

- 表单很小,状态只在一个组件里使用.
- 一次性页面,没有复杂共享状态.
- 服务端状态已经由 React Query / SWR / Apollo 管理.
- 框架自带的局部响应式状态已经足够.

一个实用分层:

```text
Local UI State
  输入框展开、tab、hover、临时弹窗

Server State
  列表数据、详情数据、缓存、刷新、过期

Client App State
  当前用户选择、复杂编辑器状态、跨页面状态
```

单向数据流最适合第三类,不应该接管所有状态。

## 四、异步副作用放在哪里

纯 reducer 不应该直接请求接口:

```ts
function reducer(state: State, action: Action): State {
  // 不要在这里 fetch
  return state;
}
```

异步副作用通常放在 action creator、middleware、effect 或 command 层:

```ts
async function submitOrder(dispatch: Dispatch, form: OrderForm) {
  dispatch({ type: "order/submitting" });
  try {
    const order = await api.createOrder(form);
    dispatch({ type: "order/submitted", order });
  } catch (error) {
    dispatch({ type: "order/failed", message: String(error) });
  }
}
```

这样状态变化仍然通过 action 进入 store,异步只负责产生 action。

## 五、代价与误用

单向数据流的代价是样板代码和间接性。一个按钮点击本来可以 `setOpen(true)`,如果也要 action、reducer、selector、middleware,就过度了。

常见误用之一是把所有状态都塞全局 store。输入框临时内容、弹窗开关、hover 状态通常放本地组件就够了。全局化会让状态生命周期变长,清理更难。

常见误用之二是 action 粒度混乱。`setData` 太技术化,调试时看不出用户意图;`userClickedSaveOrder` 又可能太 UI 化。好的 action 应该表达应用层意图,比如 `orderDraftSubmitted`。

常见误用之三是把单向数据流当成后端 CQRS。两者都区分"动作"和"状态",但 CQRS 分的是业务读写模型,单向数据流分的是客户端状态更新路径。层级不同,不要混用概念。

---

## 六、与组件和响应式的边界

组件模式回答 UI 怎么拆;单向数据流回答共享状态怎么改;响应式模式回答依赖如何自动更新。React + Redux 是组件 + 单向数据流;Vue/Pinia 也可以用单向数据流约束 mutation;Elm 则把这种约束变成语言和架构的一部分。

响应式系统不等于没有数据流。你仍然需要决定状态修改入口在哪里。Signals 或 Observable 能让依赖更新更自动,但如果任何地方都能写,复杂应用依然难追踪。

---

## 一句话总结

单向数据流用 Action -> State -> View 的固定路径换来可预测和可调试,适合复杂共享状态,但把所有局部状态都纳入它会让简单交互变重。
