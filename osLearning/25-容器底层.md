# 容器底层

Docker 看起来像虚拟机,但本质不是。容器共享宿主机内核,只是用 namespace 隔离"你能看到什么",用 cgroup 限制"你能用多少",用 OverlayFS 叠出镜像文件系统,再由 runc 按 OCI 规范把进程拉起来。这一篇讲清楚:**namespace、cgroup、OverlayFS、镜像层、runc、容器为什么不是安全边界、K8s 资源限制为什么会影响 OOM 和 CPU 抖动**。

> 一句话先记住:**容器 = namespace 隔离 + cgroup 限制 + rootfs 组合 + 一个普通 Linux 进程**。它不是小虚拟机,没有自己的内核。`ps` 里看不到宿主机进程,是 PID namespace;只能用 1 核,是 cgroup;文件系统像独立系统,是 OverlayFS。**容器问题最终仍然是 Linux 问题**。

---

## 一、容器和虚拟机的区别

虚拟机:

```
硬件
  → Hypervisor
  → Guest OS kernel
  → 进程
```

容器:

```
硬件
  → 宿主机 Linux kernel
  → namespace/cgroup 隔离的进程
```

容器启动快,因为不是启动一个内核,只是启动一个进程。

代价:共享内核,隔离强度弱于 VM。

---

## 二、namespace:隔离视图

常见 namespace:

| namespace | 隔离什么 |
| --- | --- |
| PID | 进程号视图 |
| NET | 网卡、路由表、端口 |
| MNT | 挂载点 |
| UTS | hostname/domain |
| IPC | System V IPC/POSIX MQ |
| USER | 用户/组 ID 映射 |
| CGROUP | cgroup 视图 |
| TIME | 时间命名空间 |

### 2.1 PID namespace

容器内:

```bash
ps aux
```

看到自己的进程 1。

宿主机上同一个进程有另一个 PID。

### 2.2 NET namespace

容器有自己的:

- veth 网卡
- IP
- 路由表
- iptables 视图
- 端口空间

Docker bridge 模式大致:

```
container eth0 ← veth pair → docker0 bridge → host network
```

## 三、cgroup:限制资源

cgroup 控制资源:

- CPU
- memory
- pids
- blkio/io
- cpuset
- hugetlb

### 3.1 CPU quota

K8s CPU limit 会变成 CFS quota:

```
cpu.max = quota period
```

例如 500m:

```
每 100ms 只能跑 50ms CPU
跑完就 throttle 到下个周期
```

这会造成 p99 抖动:

```
CPU 瞬间打满 → quota 用完 → throttle → 请求排队
```

查:

```bash
cat /sys/fs/cgroup/cpu.stat
```

### 3.2 memory limit

容器内存超过 limit:

- cgroup OOM
- 杀容器内进程
- 不是一定触发宿主机 OOM

查:

```bash
cat /sys/fs/cgroup/memory.current
cat /sys/fs/cgroup/memory.events
```

## 四、OverlayFS:镜像层

镜像是多层只读 layer:

```
lowerdir: image layer1 + layer2 + layer3
upperdir: container writable layer
merged:   容器看到的 rootfs
```

写文件时 copy-on-write:

```
读: 从 lower 读
写: 先 copy 到 upper,再修改
```

影响:

- 第一次写大文件可能慢
- 删除 lower 文件只是创建 whiteout
- 容器可写层不适合放重要持久数据

持久化应该用 volume/bind mount。

---

## 五、runc 与 OCI

Docker/containerd 最终会调用 runc:

```
docker
  → containerd
  → runc
  → clone/unshare/setns/cgroup/mount/seccomp
  → exec 你的进程
```

OCI 规范定义:

- runtime spec:怎么运行容器
- image spec:镜像格式

容器底层不是黑盒,就是 Linux syscall 组合。

---

## 六、容器安全

容器不是强安全边界。常见加固:

- USER 非 root
- user namespace
- capabilities 最小化
- seccomp syscall 白/黑名单
- AppArmor/SELinux
- read-only rootfs
- 禁止 privileged
- 不挂 Docker socket

危险配置:

```bash
--privileged
-v /:/host
-v /var/run/docker.sock:/var/run/docker.sock
--pid=host
--net=host
```

这些会显著削弱隔离。

## 七、容器里的常见误判

### 7.1 看到的 CPU 不等于可用 CPU

`nproc` 可能看到宿主机核数,但 cgroup quota 只给 1 核。

现代运行时需要识别 cgroup,否则线程池开太大。

### 7.2 free 看到宿主机内存

某些环境下 `free` 显示宿主机内存,但容器实际受 memory limit 限制。

以 cgroup 文件为准。

### 7.3 PID 1 问题

容器主进程是 PID 1:

- 默认信号处理语义特殊
- 要回收僵尸子进程

需要 tini/dumb-init 或应用自己正确处理。

## 八、排查命令

```bash
docker inspect <id>
nsenter -t <pid> -n ip addr
lsns
cat /proc/<pid>/cgroup
cat /proc/<pid>/status
mount | grep overlay
```

cgroup v2:

```bash
cat /sys/fs/cgroup/cpu.max
cat /sys/fs/cgroup/cpu.stat
cat /sys/fs/cgroup/memory.current
cat /sys/fs/cgroup/memory.events
cat /sys/fs/cgroup/pids.current
```

> **结论**:容器把 Linux 的隔离和限制包装成好用的交付单元。排障时不要停在 Docker/K8s 名词上,往下看 namespace、cgroup、OverlayFS,答案通常在那里。
