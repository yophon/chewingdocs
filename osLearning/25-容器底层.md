# 容器底层

"Docker 就是轻量级虚拟机"——这句话大概是容器领域流传最广的错误认知,而且它真的会坑人:容器里 OOM 了,你去查宿主机内存,发现还空着 20G,百思不得其解;给 Pod 设了 `cpu: 500m`,服务 p99 周期性飞天,你查代码查 DB 查了一星期;`nproc` 显示 64 核,线程池开了 64 个线程,结果跑得比 4 线程还慢。这些问题的共同根源是:**你以为容器是个"小虚拟机",但它根本没有自己的内核——它就是宿主机上一个被"戴了眼罩、套了缰绳"的普通 Linux 进程**。这一篇把眼罩和缰绳拆开看。

> 一句话先记住:**容器 = namespace(隔离"你能看到什么")+ cgroup(限制"你能用多少")+ OverlayFS(叠出"你以为的文件系统")+ 一个普通 Linux 进程**。`ps` 里看不到宿主机进程是 PID namespace,只能用半个核是 cgroup,文件系统像独立系统是 OverlayFS。**容器问题最终全是 Linux 问题**。

---

## 一、容器 vs 虚拟机:差的是一整个内核

把两套架构并排画出来,差别一目了然:

```
   虚拟机                          容器
─────────────────             ─────────────────
   App                            App
   Guest OS 内核   ← 每台一个       (没有这一层!)
   Hypervisor                    namespace + cgroup
   宿主机内核                      宿主机内核        ← 所有容器共享这一个
   硬件                           硬件
```

虚拟机启动要走完一整个 OS 的启动流程(内核初始化、驱动加载、init 系统……),秒级到分钟级;**容器启动只是 fork + exec 一个进程**,毫秒级。虚拟机一台吃掉几百 MB 内存打底(Guest 内核自己就要占),容器的"额外开销"约等于零。

代价同样直白:**所有容器共享宿主机内核**。一个内核漏洞被容器内进程利用,整台宿主机连同所有邻居容器全部沦陷——这是第六节"容器不是安全边界"的根源。

| 维度 | 虚拟机 | 容器 |
| --- | --- | --- |
| 隔离单位 | 完整虚拟硬件 + 独立内核 | 内核数据结构的"视图" |
| 启动 | 秒~分钟 | **毫秒** |
| 密度 | 一台宿主机几十个 | 一台宿主机几百上千个 |
| 隔离强度 | 强(逃逸需穿透 Hypervisor) | **弱(共享内核)** |
| 本质 | 模拟一台机器 | **约束一个进程** |

## 二、namespace:隔离的是"视图",不是资源

namespace 干的事一句话能说完:**让一组进程看到一个属于自己的、和别人不一样的系统视图**。它不限制你用多少资源(那是 cgroup 的活),只决定"你睁眼能看见什么"。

Linux 目前有 8 种 namespace,每种遮住一类东西:

| namespace | 隔离什么 | 容器里的体感 |
| --- | --- | --- |
| PID | 进程号视图 | 容器里 `ps` 只看到自己,主进程是 PID 1 |
| NET | 网卡 / 路由表 / 端口 / iptables | 容器有自己的 eth0 和独立端口空间 |
| MNT | 挂载点 | 容器看到自己的 `/`,不是宿主机的 |
| UTS | hostname / domain | 每个容器有自己的主机名 |
| IPC | System V IPC / POSIX 消息队列 | 共享内存不会跨容器串味 |
| USER | 用户 / 组 ID 映射 | 容器里的 root 可以映射成宿主机的普通用户 |
| CGROUP | cgroup 视图 | 容器看不到宿主机的 cgroup 树全貌 |
| TIME | 系统时钟偏移 | 容器可以有不同的 uptime |

### 2.1 PID namespace:一个进程,两个 PID

最能体现"视图"二字的例子。容器里跑 `ps aux`,你的服务是 PID 1;但在宿主机上 `ps` 一看,**同一个进程是 PID 23847**。它就是一个进程,只是在两个 namespace 里有两个号码——像一个人在公司有工号、在家叫小名。

```
宿主机 PID namespace:    1(systemd) ... 23847(你的服务) ...
                                          ║ 同一个进程
容器 PID namespace:                       1(你的服务)
```

副作用就是上一篇(24 篇)讲过的 **PID 1 问题**:容器主进程是它那个 namespace 里的 1 号,要承担 init 的职责——收养孤儿、回收僵尸,而且**未注册 handler 的信号对 PID 1 不执行默认动作**(SIGTERM 发了等于没发)。所以要么程序自己处理好信号和僵尸回收,要么用 tini / dumb-init 垫在前面。

### 2.2 NET namespace:每个容器一套独立网络栈

容器有自己的网卡、IP、路由表、iptables 规则、端口空间——所以两个容器可以同时监听"自己的"80 端口,互不冲突。容器和外界靠 veth pair(虚拟网线,一头插容器、一头插宿主机网桥)连通:

```
容器 A: eth0 ──veth──┐
                      ├── docker0 网桥 ── 宿主机协议栈 ── 物理网卡
容器 B: eth0 ──veth──┘
```

这就是 Docker 默认 bridge 模式。`-p 8080:80` 端口映射,底下不过是宿主机上一条 iptables DNAT 规则。

### 2.3 排查武器

```bash
lsns                              # 列出系统上所有 namespace
ls -l /proc/<pid>/ns/             # 看某个进程属于哪些 namespace
nsenter -t <pid> -n ip addr       # "钻进"目标进程的网络 namespace 执行命令
```

`nsenter` 是排查容器网络的神器——**容器里没装 ip/ss/tcpdump?不用进容器,从宿主机钻进它的 namespace 用宿主机的工具查**。

## 三、cgroup:限制的是"用量"

namespace 管"看见",cgroup(control group)管"用量":CPU、内存、进程数(pids)、块设备 IO(io)、绑核(cpuset)……它是一棵树,资源限制沿树往下分:

```
/sys/fs/cgroup/                     ← cgroup v2 统一树根
├── system.slice/                   ← systemd 服务们
└── kubepods.slice/
    ├── pod-aaa/
    │   ├── container-1/   cpu.max = "50000 100000"   ← 你的 CPU limit 落在这
    │   └── container-2/   memory.max = "536870912"   ← 你的 memory limit 落在这
    └── pod-bbb/ ...
```

**K8s 的 `resources.limits` 不是什么编排黑魔法,就是往这些文件里写数字**。下面两个小节,是容器性能问题的两大名场面。

### 3.1 CPU limit = CFS quota:p99 飞天事件

K8s 里 `limits.cpu: 500m` 会被翻译成 CFS 带宽控制(16 篇调度器讲过 CFS):

```
cpu.max = 50000 100000
          ↑quota  ↑period
意思:每 100ms 的周期内,这组进程总共只能跑 50ms CPU
```

注意是"总共"——名场面来了:你的服务有 8 个工作线程,流量尖峰时 8 个线程同时开跑,**50ms 的配额 6ms 就烧完了**(8 线程 × 6ms ≈ 50ms),接下来这个周期剩下的 94ms 里,整个容器被内核摁住一动不能动(throttle),所有请求干等:

```
period(100ms):  |■■■■■■ 烧完quota(6ms) | ░░░░░░░░ 全员冻结 94ms ░░░░░░░░ |
                                          ↑ 此刻到达的请求,延迟 +94ms 起步
p50 看起来很正常,p99 周期性地多出几十上百毫秒 → "p99 飞天,但 CPU 利用率才 50%"
```

**CPU 平均利用率完全正常、监控曲线岁月静好,但 p99 周期性抖动**——这是容器时代最经典的性能谜案。确诊只需要一条命令:

```bash
cat /sys/fs/cgroup/cpu.stat
# nr_throttled    被限流的周期数 ← 这个在涨,就是它
# throttled_usec  总共被冻结了多久
```

解法:调大 limit、把线程数压到和 quota 匹配、或对延迟敏感的服务干脆不设 CPU limit(只设 request)。

### 3.2 memory limit = cgroup OOM:容器死了,宿主机没事

另一个名场面:**容器被 OOM kill 了,你登上宿主机一看,`free -h` 显示还有 20G 空闲**。

这不矛盾。容器的 memory limit 是 cgroup 的私有红线:**容器内进程的内存总和摸到 `memory.max`,cgroup 的 OOM killer 就在这个容器内部杀进程**——跟宿主机整体内存余量没有任何关系。宿主机的 OOM killer 是另一套独立机制,只在整机内存耗尽时出手。

```bash
cat /sys/fs/cgroup/memory.current    # 当前用量(包含 page cache!)
cat /sys/fs/cgroup/memory.events    # oom_kill 计数在涨 = 发生过 cgroup OOM
```

两个工程要点:

- **memory.current 包含这个 cgroup 产生的 page cache**——读写大文件会把它顶上去,但 cache 是可回收的,逼近 limit 时内核会先回收 cache 再 OOM。看到用量贴着 limit 别先慌,查 `memory.stat` 分清 RSS 和 cache。
- **runtime 必须感知容器限制**。JVM 老版本按宿主机内存算默认堆大小,在 64G 宿主机上的 512M 容器里直接起飞就 OOM(新版 JVM 默认 `UseContainerSupport`);Go 用 `GOMEMLIMIT`,线程池按 cgroup quota 而不是 `nproc` 配——详见第七节"三大错觉"。

## 四、OverlayFS:镜像是怎么叠出来的

容器的文件系统看起来是个完整的 Linux 根目录,其实是 OverlayFS 把多层目录"叠"出来的幻象:

```
merged(容器看到的 /)        ← 你以为的文件系统
─────────────────────────
upperdir(容器可写层)        ← 你写的所有东西都落在这,容器删了就没
─────────────────────────
lowerdir: 镜像 layer 3       ← COPY app /app
          镜像 layer 2       ← RUN apt install ...
          镜像 layer 1       ← FROM ubuntu 基础层
                              (全部只读,多个容器共享同一份!)
```

Dockerfile 里每条产生文件变更的指令就是一层。**只读层全机共享**——100 个容器用同一个 ubuntu 基础镜像,磁盘上那 80MB 只存一份,这是容器密度高的另一半秘密。

读写规则是经典的写时复制(copy-on-write,06 篇 fork 的 COW 是同一个思想):

- **读**:从上往下找,谁先有就读谁的
- **写**:文件在只读层?先把**整个文件**拷到 upperdir 再改(copy-up)
- **删**:只读层的文件删不掉,在 upperdir 放一个 whiteout 标记把它"遮住"

三个工程后果:

1. **第一次写大文件可能卡一下**——改 1GB 文件的 1 个字节,要先 copy-up 整个 1GB。
2. **镜像瘦不下来**:`RUN rm -rf /big-file` 只是加了层 whiteout,下面那层的文件还在镜像里占体积。要在**同一条 RUN** 里下载并删除。
3. **可写层不是放数据的地方**——容器删除即蒸发,且 OverlayFS 的写路径性能不如原生文件系统。**数据库、日志、一切要活过容器生命周期的数据,用 volume / bind mount 直通宿主机文件系统**(文件系统写入路径详见 19 篇)。

## 五、runc 与 OCI:容器是怎么被"拉起来"的

把 namespace、cgroup、OverlayFS 三块拼图组装成一个跑起来的容器,这活由 runc 干。整条链路:

```
docker run nginx
  → dockerd(API、镜像管理)
  → containerd(容器生命周期管理)
  → runc(真正干活的,一个小 CLI)
       → clone(CLONE_NEWPID | CLONE_NEWNET | ...)   创建各 namespace
       → 写 /sys/fs/cgroup/...                       套上 cgroup 限制
       → pivot_root 切到 OverlayFS 的 merged 目录     换上容器的 rootfs
       → 装 seccomp 过滤器、丢 capabilities           收走危险权限
       → execve("/usr/sbin/nginx")                   变身成你的进程
```

看清楚最后一步——**runc 自己 exec 成了你的容器进程**。没有常驻的"容器引擎"夹在中间,容器跑起来之后就是一个普通进程,Docker 退场。

OCI(Open Container Initiative)规范定义了两件事:**runtime spec**(一个 JSON 描述怎么运行容器)和 **image spec**(镜像的层和元数据怎么打包)。所以 Docker、containerd、Podman、K8s 的 CRI 实现可以互换底层 runtime——**容器底层不是黑盒,就是一串 Linux syscall 的标准化组合**(进程创建那一段,11 篇 fork/exec 全讲过)。

## 六、容器不是安全边界

再强调一次:**所有容器共享宿主机内核**。VM 逃逸要穿透 Hypervisor,难如登天;容器"逃逸"很多时候只需要一个内核漏洞,或者——更常见的——**一个作死的配置**:

```bash
--privileged                                # 全部 capabilities + 全部设备,等于宿主机 root
-v /:/host                                  # 宿主机根目录直接挂进来,还隔离什么
-v /var/run/docker.sock:/var/run/docker.sock # 能跟 dockerd 说话 = 能起 privileged 容器 = root
--pid=host / --net=host                     # 直接放弃 PID / 网络隔离
```

这几条里任何一条,都等于把宿主机钥匙交给了容器。加固清单(每条对应一类攻击面):

- **容器内用非 root 用户跑**(Dockerfile 写 `USER`)——逃逸出去也只是普通用户
- **user namespace 重映射**——容器里的 root 在宿主机上是个 nobody
- **capabilities 最小化**——默认就别给 `SYS_ADMIN` 这种万能钥匙
- **seccomp**——按白名单过滤 syscall,砍掉攻击面(Docker 默认 profile 禁了 40+ 个)
- **AppArmor / SELinux**——强制访问控制兜底
- **read-only rootfs**——写不进去,植入后门都难

真需要强隔离(多租户跑不可信代码),答案是 gVisor / Kata / Firecracker microVM——**给容器找回一个属于自己的内核**,28 篇会再提到。

## 七、容器里的三大错觉

容器内的程序经常"看错"自己的处境,三个最常见的:

**错觉一:`nproc` 说我有 64 核。** namespace 不隔离 CPU 视图,容器内看到的是宿主机全部核数;但 cgroup quota 可能只给你 1 核。无脑按 `nproc` 开线程池,64 个线程抢 1 核的 quota——回看 3.1 节,throttle 抖动直接拉满。**线程数要按 cgroup quota 配**(现代 JVM / Go 已自动识别,老版本和自己写的代码要手动)。

**错觉二:`free` 说我有 64G 内存。** 同理,`/proc/meminfo` 没有被 namespace 化,`free` 看到的是宿主机内存。容器真正的红线在 `/sys/fs/cgroup/memory.max`。**以 cgroup 文件为准,永远**。

**错觉三:我的进程不用管 init 的事。** 你是 PID 1,你就是 init:不收养孤儿、不回收僵尸、不转发信号,这些锅都是你的(24 篇讲透了)。一行 `ENTRYPOINT ["tini", "--", "your-app"]` 能省很多事。

## 八、排查工具箱:什么症状查哪里

| 症状 | 查什么 |
| --- | --- |
| p99 周期性抖动 | `cat /sys/fs/cgroup/cpu.stat` 看 `nr_throttled` |
| 容器莫名重启 / 进程被杀 | `cat /sys/fs/cgroup/memory.events` 看 `oom_kill`;`dmesg -T \| grep -i oom` |
| 内存用量比预期高 | `memory.current` vs `memory.stat` 里的 RSS / cache 拆解 |
| 容器网络不通 | `nsenter -t <pid> -n ip addr / ip route / ss -lntp` |
| 想知道某进程在哪个容器 | `cat /proc/<pid>/cgroup`(路径里有容器 ID) |
| 进程数涨到上限 fork 失败 | `cat /sys/fs/cgroup/pids.current` vs `pids.max` |
| 容器文件系统行为怪 | `mount \| grep overlay` 看 lower/upper 的真实目录 |
| 看容器的全部隔离配置 | `docker inspect <id>`、`lsns` |

记住这张表的用法:**别停在 Docker / K8s 的名词层面排障**。"Pod 被驱逐""容器 CrashLoopBackOff"都是表象,往下挖一层,答案几乎总在 namespace、cgroup、OverlayFS 这三件套里。

## 踩坑提醒

1. **把容器当轻量级虚拟机**——它没有自己的内核,隔离强度、安全边界、资源视图全都和 VM 不是一回事,这个心智错误是后面所有坑的总根源。
2. **按 `nproc` / `free` 配置线程池和堆大小**——容器里这俩看到的是宿主机,真实限制在 `/sys/fs/cgroup` 下,老版本 JVM 因此直接起飞 OOM。
3. **设了 CPU limit 却想不通 p99 抖动**——CFS quota 烧完就冻结到周期结束,`nr_throttled` 一查便知,CPU 平均利用率不高不代表没被限流。
4. **容器 OOM 跑去查宿主机内存**——cgroup OOM 是容器自己的红线,和宿主机剩多少内存无关。
5. **重要数据写进容器可写层**——容器一删数据蒸发,OverlayFS 写性能也差,持久数据必须走 volume。
6. **Dockerfile 里先 COPY 大文件再 rm**——删除只是 whiteout 遮住,镜像体积一点没小,下载和删除要在同一层。
7. **`--privileged` / 挂载 docker.sock 图省事**——等于把宿主机 root 拱手送人,生产环境一票否决。
8. **裸跑应用当 PID 1**——信号收不到(每次停机等 30s 强杀)、僵尸没人收,tini 两行配置的事。
9. **排障停在 K8s 名词层**——"Pod 异常"不是根因,`kubectl describe` 之后要敢往 cgroup 文件和 `dmesg` 里钻。
