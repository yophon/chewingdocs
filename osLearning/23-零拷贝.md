# 零拷贝

零拷贝不是"一次拷贝都没有",而是减少 CPU 参与的数据搬运。普通 `read + write` 发送文件要把数据从磁盘搬到内核,再拷到用户态,再拷回内核 socket buffer,最后 DMA 到网卡。零拷贝用 `sendfile`、`splice`、`mmap`、DMA、页引用传递等手段,减少用户态/内核态拷贝和上下文切换。这一篇讲清楚:**哪些拷贝能省、sendfile 为什么快、mmap 适合什么、零拷贝的边界在哪里**。

> 一句话先记住:**零拷贝的核心是少让 CPU 搬大块数据**。它通常省的是"内核 ↔ 用户态"拷贝,磁盘到内存、内存到网卡仍靠 DMA。**静态文件发送、代理转发、日志传输适合零拷贝;需要加密、压缩、改内容时往往绕不开用户态处理**。

---

## 一、普通 read + write 有几次拷贝

```c
read(file_fd, buf, len);
write(sock_fd, buf, len);
```

发送文件路径:

```
磁盘 → DMA → page cache          1 次 DMA
page cache → 用户 buffer         1 次 CPU copy
用户 buffer → socket buffer      1 次 CPU copy
socket buffer → 网卡             1 次 DMA
```

还有两次 syscall:`read` 和 `write`。

大文件传输时,CPU copy 很贵,而且污染 cache。

---

## 二、sendfile

```c
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
```

典型:

```c
sendfile(sock_fd, file_fd, &off, len);
```

路径:

```
磁盘 → DMA → page cache
page cache → socket/网卡描述符引用
网卡 DMA 读 page cache 数据发送
```

减少:

- 用户态 buffer
- 内核到用户态拷贝
- 用户态到内核拷贝
- 一次 syscall

Nginx 静态文件发送大量使用 sendfile。

### 2.1 限制

sendfile 适合"文件原样发到 socket"。

不适合:

- 需要压缩内容
- 需要应用层加密处理
- 需要改包体
- 源/目标不是支持的 fd 类型

TLS 是典型限制:用户态 TLS 要读出明文加密后再发。内核 TLS(kTLS)可以缓解部分场景。

## 三、mmap

```c
void *p = mmap(NULL, len, PROT_READ, MAP_PRIVATE, fd, 0);
write(sock_fd, p, len);
```

`mmap` 把文件映射进进程虚拟地址:

```
访问 p[i]
  → 缺页中断
  → 文件页读入 page cache
  → 页表映射到进程
```

省掉 `read` 的用户态 buffer 拷贝,但 `write` 到 socket 仍可能拷贝到 socket buffer。

适合:

- 随机读文件
- 多进程共享只读数据
- 大文件局部访问
- 数据库/搜索引擎某些索引结构

不适合:

- 顺序流式读写简单场景未必比 read 快
- 访问越界 SIGBUS
- page fault 延迟不可控
- 写 mmap 文件的一致性要小心

---

## 四、splice / tee / vmsplice

Linux pipe 可以作为内核中的数据管道。

```c
splice(file_fd, NULL, pipefd[1], NULL, len, 0);
splice(pipefd[0], NULL, sock_fd, NULL, len, 0);
```

数据在内核对象之间移动,尽量不经过用户态。

用途:

- 文件到 socket
- socket 到文件
- pipe 间复制
- 代理转发

`tee` 可以在 pipe 之间复制引用,`vmsplice` 可以把用户页映射到 pipe。

工程上比 sendfile 更灵活,也更难用。

---

## 五、DMA 与 scatter-gather

DMA(Direct Memory Access)让设备直接读写内存,CPU 不逐字节搬。

Scatter-gather IO 让设备一次处理多个内存片段:

```c
writev(fd, iov, iovcnt);
sendmsg(fd, &msg, 0);
```

优点:

- 避免把多个 buffer 拼成一个大 buffer
- 减少内存拷贝
- 网络协议栈常用

---

## 六、零拷贝和页缓存

sendfile 通常仍走 page cache:

```
磁盘数据进 page cache
网卡从这些页发出去
```

这对热点静态文件很好:第二次发送直接内存命中。

但对超大冷文件可能污染 page cache。可以考虑:

```c
posix_fadvise(fd, 0, 0, POSIX_FADV_SEQUENTIAL);
posix_fadvise(fd, 0, 0, POSIX_FADV_DONTNEED);
```

或用 `O_DIRECT` 让应用自己管理缓存。

---

## 七、什么时候零拷贝收益大

收益大:

- 大文件静态下载
- 视频点播
- 反向代理转发
- 日志/备份传输
- CPU 被 copy 打满

收益小:

- 小响应,业务计算占主导
- 数据需要修改
- TLS/压缩在用户态
- 瓶颈在磁盘或网络带宽

---

## 八、常见坑

1. **以为零拷贝就是 0 copy**:DMA 和必要的内核内部引用仍存在。
2. **忽略短发送**:`sendfile` 也可能只发一部分。
3. **TLS 破坏零拷贝路径**:加密要处理明文。
4. **mmap 后访问触发 SIGBUS**:文件被截断或映射越界。
5. **page cache 污染**:大冷文件挤掉热数据。
6. **跨文件系统/特殊 fd 不支持**:要处理 fallback。

---

## 九、排查与验证

```bash
strace -e read,write,sendfile,splice -p <pid>
perf top
pidstat -d 1
sar -n DEV 1
```

如果 `perf top` 里大量 `copy_user_enhanced_fast_string` / `memcpy`,说明 CPU 在搬数据,零拷贝可能有价值。

> **结论**:零拷贝是大吞吐场景的利器,但它优化的是数据搬运路径。先确认 CPU copy 是瓶颈,再选择 sendfile/splice/mmap/writev,不要为了术语复杂化普通业务代码。
