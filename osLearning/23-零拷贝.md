# 零拷贝

前两篇优化的是"怎么等 IO"(epoll)和"怎么发起 IO"(io_uring),这一篇盯住第三个成本:**数据本身的搬运**。很多人第一次听"零拷贝"(zero-copy)都会望文生义,以为是"一次拷贝都没有"——不是。数据从磁盘到网卡,物理上总得移动;零拷贝省的是**让 CPU 亲自搬大块数据的那几次**。别小看这件事:Kafka 能用很普通的机器扛住每秒几十万条消息,Nginx 发静态文件快得不讲道理,核心红利都来自同一个 syscall——`sendfile`。这一篇讲清楚:一次普通的文件发送到底拷了几次、哪几次能省、sendfile/mmap/splice 各自的玩法,以及零拷贝在什么时候帮不上忙。

> 一句话先记住:**零拷贝的核心是少让 CPU 当搬运工**。省的主要是"内核 ↔ 用户态"那两次 CPU 拷贝和伴随的上下文切换;磁盘到内存、内存到网卡的搬运一直由 DMA 干,本来就不花 CPU。**文件原样发出去(静态文件、代理转发、日志传输)是零拷贝的主场;数据要加密、压缩、改内容时,往往就得回到用户态老老实实处理**。

## 一、先数清楚:read + write 发个文件,数据走了几趟

最朴素的"发文件"代码,所有人都写过:

```c
read(file_fd, buf, len);     // 文件读进用户态 buffer
write(sock_fd, buf, len);    // 再写进 socket
```

看着两行,数据在底下走了四趟,外加两次 syscall、四次用户态/内核态切换:

```
              用户态                    内核态                     硬件
                                                                    
                                  ┌────────────┐    ① DMA      ┌──────┐
                                  │ page cache │ ←──────────── │ 磁盘 │
                                  └─────┬──────┘               └──────┘
              ┌──────────┐  ② CPU copy  │
   read() →   │ 用户 buf │ ←────────────┘
              └─────┬────┘
                    │       ③ CPU copy  ┌───────────────┐
   write() →        └─────────────────→ │ socket buffer │
                                        └───────┬───────┘
                                                │  ④ DMA       ┌──────┐
                                                └────────────→ │ 网卡 │
                                                               └──────┘
```

| 趟次 | 谁干的 | 冤不冤 |
| --- | --- | --- |
| ① 磁盘 → page cache | DMA | 不冤,硬件干活,CPU 不管 |
| ② page cache → 用户 buf | **CPU** | **冤**——你拿到数据后看都不看一眼 |
| ③ 用户 buf → socket buffer | **CPU** | **冤**——原样又搬回内核 |
| ④ socket buffer → 网卡 | DMA | 不冤 |

最要命的是 ② 和 ③:**数据进了用户态又原封不动地回去了,CPU 白搬两趟**。发一个 1GB 的视频文件,CPU 要亲手摸 2GB 数据——不光烧 CPU 周期,还顺手把 L1/L2/L3 缓存里的热数据全冲了(缓存污染,10 篇讲过这有多伤)。文件服务器吞吐一上来,CPU 全耗在 `memcpy` 上,这就是零拷贝要救的场。

## 二、sendfile:数据全程不出内核

既然用户态对数据"过手不过问",那干脆别让它过手:

```c
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);

// 典型用法:把文件 fd 的内容直接发到 socket fd
sendfile(sock_fd, file_fd, &off, len);
```

数据路径变成:

```
   ┌──────┐  DMA   ┌────────────┐   传页引用(几乎不拷)   ┌────────┐
   │ 磁盘 │ ─────→ │ page cache │ ───────────────────→  │ 网卡队列│
   └──────┘        └────────────┘                       └───┬────┘
                                                            │ DMA(scatter-gather,
                                                            ↓  直接从 page cache 的页读数据)
                                                         ┌──────┐
                                                         │ 网卡 │
                                                         └──────┘
```

在支持 scatter-gather 的网卡上,内核连"page cache → socket buffer"那一下都省了——**socket 这边只记一个"数据在那些页上"的描述符(页引用),网卡 DMA 直接从 page cache 的页里取数发送**。盘点战果:

- 用户态 buffer:没了
- ② ③ 两次 CPU 拷贝:没了
- 两次 syscall 变一次,上下文切换减半
- CPU 缓存:不再被大文件冲刷

**名场面一:Nginx。** 配置里一行 `sendfile on;`,静态文件场景吞吐和 CPU 占用立竿见影,这是它发静态资源快得不讲道理的底牌之一。

**名场面二:Kafka。** Kafka 的消费路径本质就是"把日志文件的一段原样发给消费者"——文件原样进 socket,sendfile 的教科书场景。再叠加顺序写盘 + page cache 当缓存,**消息从磁盘到网卡全程不进用户态、不烧 CPU 拷贝**,这就是 Kafka 用平凡硬件跑出变态吞吐的两大支柱之一(另一个是顺序 IO)。

### 2.1 sendfile 的边界

它的能力描述只有一句:**把 fd 的内容原样发到另一个 fd**。"原样"两个字就是全部限制——以下场景统统出局:

- 要 gzip 压缩响应——数据必须进用户态过压缩器
- 要应用层加密/改包体——同理
- 源/目标 fd 类型不支持(早期要求目标必须是 socket)

最典型的拦路虎是 **TLS**:HTTPS 下数据发出去之前要加密,明文必须被读到用户态、加密、再写回——零拷贝路径整个碎掉。缓解方案是 **kTLS(内核 TLS)**:握手仍在用户态完成,之后把会话密钥交给内核,加密在内核(甚至网卡硬件)做,sendfile 路径重新接通。Netflix 在 FreeBSD 上用同思路的 sendfile+kTLS 扛视频分发,Linux 4.13+ 也支持 kTLS。

## 三、mmap:把文件"贴"进地址空间

另一条路:不消灭拷贝,而是**让用户态直接看见 page cache**:

```c
void *p = mmap(NULL, len, PROT_READ, MAP_PRIVATE, fd, 0);
write(sock_fd, p, len);
```

mmap 把文件映射进进程虚拟地址空间(机制 06 篇讲过):

```
访问 p[i]
  → 缺页中断
  → 内核把对应文件页读入 page cache
  → 页表直接映射到这块 page cache 页
  → 之后访问 p[i] 就是访问 page cache 本身,没有第二份副本
```

对比 read:**省掉了"page cache → 用户 buf"那次 CPU 拷贝**(②),因为你的"用户 buf"就是 page cache 本身。但注意,后面 `write` 到 socket 时,数据仍可能被拷进 socket buffer(③ 还在)——所以**纯发文件,mmap 不如 sendfile 省**;mmap 的真正主场是"**既要省拷贝,又要亲自读写数据内容**":

适合:

- 随机访问大文件的某些片段(数据库、搜索引擎的索引文件)
- 多进程共享同一份只读数据(映射同一文件,物理内存只有一份)
- 大文件局部访问,不想整个读进来

不适合:

- 顺序流式读一遍就扔——未必比 read 快,缺页中断一页一次,而 read 可以预读大块
- 延迟敏感路径——page fault 什么时候来、要不要等磁盘,不可控
- 文件可能被别人截断——访问已截断区域直接 **SIGBUS**,进程没了(防御要 `sigsetjmp` 或锁文件)
- 写 mmap 文件——脏页什么时候落盘由内核定,一致性要 `msync` 自己管,坑很深

## 四、splice / tee / vmsplice:内核里的数据管道

sendfile 是"文件 → socket"的专用通道,splice 是泛化版:**借助 pipe 作为内核里的中转站,让数据在任意两个 fd 之间移动而不经过用户态**。

```c
int pipefd[2];
pipe(pipefd);

splice(file_fd, NULL, pipefd[1], NULL, len, 0);   // 文件 → pipe(传页引用)
splice(pipefd[0], NULL, sock_fd, NULL, len, 0);   // pipe → socket
```

精髓在于 pipe 在内核里本来就是一组页的环形缓冲,splice 在多数路径下**移动的是页的引用而不是页的内容**。能玩的方向:

- 文件 → socket(sendfile 同款)
- socket → 文件(下载落盘)
- **socket → socket**(代理转发——sendfile 干不了,代理服务器的刚需)
- pipe 之间复制

家族另外两个成员:`tee` 在两个 pipe 之间复制页引用(数据"分流"一份,比如边转发边落日志);`vmsplice` 把用户态内存页"嫁接"进 pipe。工程上 splice 比 sendfile 更灵活,也明显更难用——offset 管理、部分成功、fd 类型限制,都要自己兜着。

## 五、DMA 与 scatter-gather:被忽略的幕后功臣

上面反复出现的 DMA(Direct Memory Access)值得正名:**设备控制器直接读写内存,不需要 CPU 逐字节经手**。没有 DMA 的远古时代,磁盘每个字节都要 CPU 亲自从设备寄存器搬进内存——所谓零拷贝的"地板",就是 DMA 铺的。

它的进阶形态 scatter-gather(分散/聚集)解决另一个隐形拷贝:数据散在多块内存里时,设备能按一张"地址清单"逐段取走,**不需要先拼成一块连续大 buffer**。用户态对应的接口就是向量 IO:

```c
writev(fd, iov, iovcnt);     // 多个 buffer 一次写出
sendmsg(fd, &msg, 0);
```

典型场景:HTTP 响应头在 buffer A,响应体在 buffer B——`writev` 一次发出,**省掉"拼接大 buffer"的那次 memcpy 和一次 syscall**。协议栈内部、sendfile 的网卡直读 page cache,靠的都是同一个硬件能力。

## 六、零拷贝与 page cache:蜜糖和砒霜

注意 sendfile 的路径仍然过 page cache,这件事两面看。

**蜜糖**:热点文件第二次发送时磁盘那趟 DMA 都省了,直接从内存命中——Nginx 发热门静态资源、Kafka 消费最近的消息(还热乎,在 cache 里),都是"零拷贝 + 缓存命中"双重红利。

**砒霜**:发一个 200GB 的冷备份文件,这些"读一遍就再也不碰"的页会把 page cache 里真正的热数据全挤出去——**别的服务突然变慢,锅却很难找到你头上**。对策:

```c
posix_fadvise(fd, 0, 0, POSIX_FADV_SEQUENTIAL);  // 提示顺序读,加大预读
posix_fadvise(fd, 0, 0, POSIX_FADV_DONTNEED);    // 用完的区间,踢出 cache
```

或者干脆 `O_DIRECT` 绕开 page cache、应用自己管缓存——数据库(如 InnoDB 的 buffer pool)就是这么选的,代价是对齐要求和实现复杂度全归你。

## 七、什么时候零拷贝真有收益

| 收益大 | 收益小 |
| --- | --- |
| 大文件静态下载、视频点播 | 几 KB 的小响应,业务计算占大头 |
| 反向代理原样转发 | 数据要改(模板渲染、压缩、转码) |
| 日志/备份/消息传输 | TLS/压缩必须在用户态做 |
| `perf` 显示 CPU 被 copy 打满 | 瓶颈本来就在磁盘或网络带宽 |

判断标准一句话:**先确认 CPU 在搬数据,再谈零拷贝**。瓶颈在带宽时,把 CPU 拷贝优化到零,吞吐一点不涨——水管粗细没变,你只是换了个不累的搬水姿势。

验证方法:

```bash
strace -e read,write,sendfile,splice -p <pid>   # 确认实际走的 syscall
perf top                                         # 看 CPU 热点
pidstat -d 1                                     # 进程级磁盘 IO
sar -n DEV 1                                     # 网卡带宽是否已打满
```

如果 `perf top` 里 `copy_user_enhanced_fast_string` / `memcpy` 高居榜首——CPU 正在大量搬数据,零拷贝大概率有价值;反之它根本排不上号,就别折腾了。

## 踩坑提醒

1. **以为零拷贝 = 0 次拷贝**——DMA 两趟一直都在,内核内部的页引用传递也不是零成本;零的是"CPU 在用户态/内核态之间的大块搬运"
2. **忽略短发送**——`sendfile`/`splice` 和 write 一样可能只发出一部分就返回,不循环续传就丢数据
3. **HTTPS 下期待 sendfile 生效**——用户态 TLS 必须摸明文,零拷贝路径直接断;要么接受,要么上 kTLS
4. **mmap 后访问触发 SIGBUS**——文件被别的进程截断、或映射越界,一访问进程就没;mmap 共享文件要有防御意识
5. **拿 mmap 当万能加速器**——顺序读一遍的场景未必比 read 快,page fault 的延迟还不可控;它的主场是随机访问和多进程共享
6. **大冷文件污染 page cache**——传完一个备份,全机热数据被挤光;`fadvise DONTNEED` 或 `O_DIRECT` 伺候
7. **不查 fd 类型支持就上 splice/sendfile**——跨文件系统、特殊 fd 可能不支持,生产代码要留 read+write 的 fallback 路径
8. **没确认瓶颈就动手**——CPU 没在搬数据、或瓶颈在带宽,零拷贝改造是纯负收益:代码更复杂,吞吐不变
