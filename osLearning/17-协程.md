# 协程

协程不是魔法,它只是把"等待 IO 时让出执行权"这件事从内核线程切换,搬到了用户态调度。Go 的 goroutine、Python asyncio、JavaScript Promise、Rust async、Kotlin coroutine,表面语法差很多,底层都在回答同一个问题:**如何用少量线程承载大量并发任务**。这一篇讲清楚:**协程为什么轻、用户态调度怎么做、栈有栈/无栈差别、阻塞 syscall 为什么会毁掉协程、什么时候线程反而更好**。

> 一句话先记住:**协程 = 用户态可挂起/恢复的执行单元**。它轻是因为不需要每个任务一个内核线程,切换时不用进内核,栈也可以很小或按需增长。**协程适合 IO 并发,不自动提升 CPU 算力**——CPU 密集任务跑一万个协程,最后还是那几个核在算。协程系统的生命线是:**所有阻塞点都必须可被调度器接管**。

---

## 一、为什么需要协程

传统一请求一线程:

```
请求 A: 线程 1 → 查 DB 等 50ms
请求 B: 线程 2 → 查 Redis 等 5ms
请求 C: 线程 3 → 调 RPC 等 100ms
```

线程大部分时间在等 IO,但每个线程仍占:

- 内核调度实体
- 默认几百 KB 到几 MB 栈
- 上下文切换成本
- 线程池管理成本

如果 10 万连接都用线程扛,内存和调度器先崩。

协程的思路:

```
一个线程跑很多协程
协程遇到 IO → 挂起
线程继续跑别的协程
IO 就绪 → 恢复原协程
```

---

## 二、协程和线程的区别

| | 线程 | 协程 |
| --- | --- | --- |
| 调度者 | OS 内核 | 运行时/事件循环 |
| 切换成本 | μs 级,进内核 | ns-百 ns 级,用户态 |
| 栈 | 通常 MB 级 | KB 级/按需/无栈 |
| 阻塞 syscall | 只阻塞当前线程 | 可能阻塞整个调度线程 |
| 并行能力 | 多线程可多核并行 | 取决于底层线程数 |
| 适用 | CPU + IO 通用 | 高并发 IO |

协程不是线程替代品,而是**线程上面的并发抽象**。

## 三、两类协程:有栈 vs 无栈

### 3.1 有栈协程

每个协程有自己的调用栈,可以在深层函数里挂起。

Go goroutine:

```go
func handler() {
    a()
}

func a() {
    b()
}

func b() {
    resp, _ := http.Get(url) // 这里可以挂起 goroutine
    use(resp)
}
```

优点:

- 写法像同步代码
- 任意调用深度可挂起
- 运行时负责栈增长/收缩

缺点:

- runtime 更复杂
- 栈扫描和 GC 要配合

### 3.2 无栈协程

协程状态由编译器改写成状态机,没有独立调用栈。

Rust async / JS async:

```rust
async fn handler() {
    let resp = fetch().await;
    use_resp(resp).await;
}
```

编译器大致改成:

```
state = Start
poll():
  if Start: 发起 fetch, state = WaitingFetch, return Pending
  if WaitingFetch: fetch 完成,继续,发起 use_resp, state = WaitingUse, return Pending
  if WaitingUse: return Ready
```

优点:

- 内存极小
- 不需要保存完整栈
- 零成本抽象潜力大

缺点:

- 函数颜色问题:async 调 async 要一路 async
- 不能随便在普通函数深处 await
- borrow/lifetime 更复杂

## 四、事件循环与 IO 多路复用

协程运行时通常长这样:

```
任务队列 ready queue
      ↓
执行协程
      ↓ 遇到 IO
注册到 epoll/kqueue/io_uring
      ↓
事件循环等待就绪
      ↓
把协程放回 ready queue
```

伪代码:

```c
while (true) {
    while (ready_queue not empty) {
        coro = pop_ready();
        resume(coro);
    }

    events = epoll_wait(epfd, timeout);
    for (ev in events) {
        coro = ev.data.ptr;
        push_ready(coro);
    }
}
```

**协程不是替代 epoll,而是包装 epoll**。epoll 负责"谁就绪",协程负责"从上次 await 的地方继续跑"。

## 五、Go 的 G-M-P 模型

Go runtime:

```
G = goroutine      用户态任务
M = machine        OS 线程
P = processor      调度上下文/本地队列
```

```
P0 runq: G1 G2 G3  → M0 执行
P1 runq: G4 G5     → M1 执行
```

特点:

- goroutine 初始栈很小(几 KB),可增长
- 阻塞网络 IO 由 netpoller 接管
- 阻塞 syscall 会让 runtime 可能新建/唤醒 M
- work stealing 平衡各 P 队列

Go 的优势是把这些复杂度藏起来,你写起来像同步代码。

### 5.1 Go 什么时候会被坑

```go
for {
    // 长时间纯 CPU 循环,不调用函数/不阻塞
}
```

老版本 Go 这类代码可能影响抢占。现代 Go 已有异步抢占,但 CPU 密集 goroutine 仍然会抢占其他 goroutine 的 CPU。

**CPU 密集型任务要控制并行度**,不要无限开 goroutine。

## 六、Python asyncio

Python asyncio 是单线程事件循环 + 无栈协程:

```python
async def handler():
    resp = await client.get(url)
    return await resp.text()
```

关键点:

- 只有遇到 `await` 才让出
- 普通阻塞函数会卡住整个事件循环
- CPU 密集任务会卡住所有 coroutine

错误:

```python
async def bad():
    time.sleep(10)   # 阻塞线程,事件循环停 10 秒
```

正确:

```python
async def ok():
    await asyncio.sleep(10)
```

阻塞库要放线程池:

```python
await asyncio.to_thread(blocking_func)
```

## 七、Rust async

Rust async 的核心是 `Future`:

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

`await` 本质是反复 poll:

- `Poll::Pending`:没好,把 waker 注册起来
- `Poll::Ready(v)`:完成

Rust async 的优势:

- 无运行时绑定,Tokio/async-std/glommio 可选
- 无 GC
- 内存占用小

代价:

- 学习曲线高
- `Send` / `Sync` / lifetime 容易卡人
- 不能在 async 任务里随便调用阻塞函数

## 八、阻塞 syscall 是协程杀手

协程调度器只能管理它知道的等待。如果你在调度线程里直接调用阻塞 syscall:

```c
read(blocking_fd, buf, n);  // 没数据,整个线程睡死
```

这个线程上的所有协程都停了。

解决方式:

1. fd 设非阻塞,交给 epoll/io_uring
2. 阻塞调用丢到专门线程池
3. 使用 runtime 提供的 async 版本
4. 对 CPU 密集任务用 worker pool

这就是为什么 async 生态强调:"不要在事件循环里做阻塞操作"。

## 九、协程的性能边界

### 9.1 适合

- 网关/API server
- 长连接/WebSocket
- 爬虫
- RPC fan-out
- 高并发但大部分时间等 IO
- 状态机型业务流程

### 9.2 不适合单独解决

- 大量 CPU 计算
- 大对象内存拷贝
- 锁竞争严重
- 下游已经限流
- 单连接带宽瓶颈

协程能提高**并发承载**,不等于提高**单任务执行速度**。

---

## 十、常见错误

1. **把协程当线程无限开**:协程轻,但不是免费;每个任务仍有栈/状态/调度成本。
2. **在事件循环里调用阻塞库**:一个慢 DNS/文件 IO 卡住全部请求。
3. **CPU 密集任务不限制并行度**:协程排队,延迟爆炸。
4. **忘记背压**:上游开 10 万协程打下游,把 DB 打死。
5. **混用锁不当**:async 里拿普通 mutex 后 await,容易死锁或阻塞调度线程。

## 十一、工程判断

选模型时看问题本质:

| 场景 | 推荐 |
| --- | --- |
| CPU 密集 | 线程池/进程池,数量接近核数 |
| 高并发网络 IO | 协程/事件循环 |
| 少量并发 + 简单业务 | 线程模型足够 |
| 需要充分利用多核 + IO | 多线程 runtime 上跑协程(Go/Tokio) |
| 需要隔离崩溃 | 多进程 |

> **结论**:协程把"等待"变便宜,不是把"计算"变便宜。理解这一点,就不会用一万个协程去解决一个 CPU 已经打满的问题。
