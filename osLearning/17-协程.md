# 协程

"Go 能轻松开 100 万个 goroutine"——这句话忽悠了不少人,以为协程是某种"更快的线程",于是把 CPU 密集的图像处理切成一万个协程,结果比老老实实开 8 个线程还慢。协程不是魔法,它只干了一件事:**把"等待 IO 时让出执行权"这个动作,从内核的线程切换搬到了用户态**。Go 的 goroutine、Python asyncio、JavaScript Promise、Rust async、Kotlin coroutine,语法天差地别,底层都在回答同一个问题:**怎么用少量线程承载海量并发任务**。这一篇讲清楚:协程到底轻在哪、用户态调度怎么转、有栈和无栈是两条什么路、阻塞 syscall 为什么是协程系统的天敌——以及那道白板名题:**为什么 Go 协程这么轻量,但密集 CPU 任务用线程更快**。

> 一句话先记住:**协程 = 用户态可挂起/恢复的执行单元**。它轻,是因为不用每个任务配一个内核线程:切换不进内核(ns 级 vs 线程的 μs 级),栈小到 KB 级或干脆没有。但**协程只把"等待"变便宜,没把"计算"变便宜**——CPU 密集任务开一万个协程,最后还是那 8 个核在算。协程系统的生命线只有一条:**所有阻塞点都必须被调度器接管**,漏一个,整条线程上的协程全员陪睡。

## 一、没有协程的世界:10 万连接压垮线程模型

上一篇(16 篇)结尾留了钩子:IO 密集场景按公式算线程数,等待/计算比 100:1 时要开上万线程——为什么不行?把账算细:

```
一请求一线程的世界:
请求 A: 线程 1 → 查 DB,睡 50ms
请求 B: 线程 2 → 查 Redis,睡 5ms
请求 C: 线程 3 → 调下游 RPC,睡 100ms
```

这些线程 95% 的时间在睡,**但睡觉不免费**,每个线程持续占着:

- 一个内核调度实体(run queue、负载均衡都带着它记账);
- **默认几百 KB 到几 MB 的栈**;
- 每次睡醒一轮 μs 级上下文切换 + cache 重热(16 篇刚算过这笔税);
- 线程池本身的管理成本。

10 万长连接 × 哪怕 1MB 实际栈 = 100GB 内存,还没干活先没了;就算内存扛住,几万个线程睡睡醒醒,调度器和 cache 也被切换打成筛子。**这就是经典的 C10K 问题:线程模型的成本结构,撑不起"海量且大多在等"的并发**。

协程的思路是釜底抽薪——既然任务大部分时间在等,**那就别让"等"占用线程**:

```
一个线程跑很多协程:
  协程 X 发起 IO → 不睡线程,把自己挂起,登记"IO 好了叫我"
  线程立刻去跑协程 Y、Z……
  X 的 IO 就绪 → X 被放回就绪队列,排队恢复执行
```

线程从"每人陪一个任务睡觉"变成"流水线工人,只服务醒着的任务"。等待被压缩成一条登记记录,成本从 MB 级的线程降到 KB 级的协程状态。

## 二、协程和线程:一张表看清差别

| | 线程 | 协程 |
| --- | --- | --- |
| 谁调度 | OS 内核 | 语言运行时 / 事件循环 |
| 切换成本 | μs 级,要进内核 | ns ~ 百 ns 级,纯用户态 |
| 栈 | 通常 MB 级 | KB 级 / 按需增长 / 无栈 |
| 数量级 | 几千就吃力 | 百万不稀奇 |
| 阻塞 syscall | 只睡自己 | **可能睡死整条调度线程**(连坐) |
| 多核并行 | 天然可以 | 取决于底下铺了几个线程 |
| 适用 | CPU + IO 通吃 | 高并发 IO |

最该记住的是"阻塞 syscall"那行:线程世界里阻塞是局部事故,协程世界里是连坐事故——这是轻量换来的代价,第八节专门算这笔账。还有个定位别搞错:**协程不是线程的替代品,是架在线程上面的并发抽象**——协程最终要骑在线程上才碰得到 CPU,正确的问法不是"协程 vs 线程",是"任务调度放内核还是放用户态"。

## 三、有栈 vs 无栈:两条技术路线

"挂起一个协程"意味着把它的执行状态(跑到哪了、局部变量是什么)存起来。存在哪,分出两派。

### 3.1 有栈协程(stackful):每人一个真栈

Go 的 goroutine 是代表:每个协程有独立调用栈(初始几 KB,按需增长),挂起 = 保存寄存器和栈指针,跟线程切换同构,只是不进内核。

```go
func handler() { a() }
func a()       { b() }
func b() {
    resp, _ := http.Get(url)  // 调用栈三层深的地方,照样能挂起
    use(resp)
}
```

优点:**写起来和同步代码一模一样**,任意调用深度都能挂起。代价:runtime 必须很重——管栈的分配、增长、收缩,GC 还得会扫描这些栈。

### 3.2 无栈协程(stackless):编译器把你重写成状态机

Rust async / JS async / Python async 是这一派:协程**没有自己的栈**,编译器把 async 函数拆成状态机,跨越 `await` 存活的变量收编成状态机字段。

```rust
async fn handler() {
    let resp = fetch().await;
    use_resp(resp).await;
}
```

编译器大致改写成:

```
state = Start
poll():
  Start:        发起 fetch,state = WaitingFetch,return Pending
  WaitingFetch: fetch 完成,发起 use_resp,state = WaitingUse,return Pending
  WaitingUse:   return Ready
```

优点:内存极小(一个结构体)、不用保存完整栈、零成本抽象潜力大。代价是**函数染色(function coloring)问题**:async 函数只能被 async 函数 `await`,颜色沿调用链一路传染——你不能在普通函数深处随手挂起,而这正是有栈协程免费拥有的能力。Rust 还要叠加 borrow/lifetime 跨 await 的复杂度。

```
有栈:  自由挂起,运行时贵  → Go 选它,拿 runtime 复杂度换用户体验
无栈:  内存极省,写法受限  → Rust/JS 选它,拿表达力换零开销
```

## 四、事件循环:协程是 epoll 的人性化包装

用户态调度器长什么样?几乎所有协程运行时都是同一个骨架——**就绪队列 + IO 多路复用**:

```c
while (true) {
    while (ready_queue 不空) {
        coro = pop_ready();
        resume(coro);              // 跑到它下一次挂起为止
    }
    events = epoll_wait(epfd, timeout);   // 没活了,问内核谁的 IO 好了
    for (ev in events)
        push_ready(ev.data.ptr);          // 把等到结果的协程放回队列
}
```

看清这个关系:**协程不是替代 epoll,而是包装 epoll**(epoll 详见 20/21 篇)。epoll 负责"哪些 fd 就绪了",协程负责"从上次挂起的那行继续往下跑"。裸写 epoll 的痛苦在于业务逻辑被回调切得粉碎,协程把碎片重新缝成一条直线——**它卖的不是性能,是"用同步的写法享受异步的成本结构"**。

## 五、Go 的 G-M-P:工业级 M:N 调度长这样

Go 把这套做到极致,runtime 里三个角色:

```
G = goroutine   用户态任务(百万级)
M = machine     OS 线程(几十个)
P = processor   调度上下文,持有本地就绪队列(默认 = 核数)

G1 G2 G3 G4 G5 G6 G7 G8 ……(百万个 G)
 \  |  /     \  |  /
 ┌─────┐     ┌─────┐     ┌─────┐
 │ P0  │     │ P1  │     │ P2  │   ← P 的本地 runq
 │runq │     │runq │     │runq │
 └──┬──┘     └──┬──┘     └──┬──┘
    │           │           │       P 闲了去别的 P 偷活
 ┌──┴──┐     ┌──┴──┐     ┌──┴──┐    (work stealing)
 │ M0  │     │ M1  │     │ M2  │   ← OS 线程,真正上 CPU 的
 └─────┘     └─────┘     └─────┘
    ↑ 内核调度器(16 篇)只看得见 M,看不见 G
```

几个关键机制,每个都在堵一个具体的坑:

- **goroutine 初始栈只有几 KB**,不够了 runtime 自动扩容——百万并发的内存账就这么算平的;
- **网络 IO 走 netpoller**:G 等网络时只是登记到 epoll,M 完全不睡,转身跑下一个 G;
- **阻塞 syscall(如文件 IO)躲不掉时**:M 只能陪着陷进内核,但 runtime 会把 P 摘下来交给(或新建)另一个 M 继续跑队列里的 G——**牺牲一个线程,保全整个调度**;
- **work stealing**:哪个 P 队列空了,去别人队尾偷一半,负载自动摊平。

写的人完全无感——你写同步风格的代码,这套机器在底下转。这是 Go 最值钱的工程成就:**把 M:N 调度的全部复杂度埋进 runtime**。代价也有:早年 goroutine 只在函数调用点才能被抢占,一段不调函数的纯 CPU 紧循环能卡住整个 P(Go 1.14 起加了基于信号的异步抢占,缓解了);但**CPU 密集的 goroutine 该挤占别人还是挤占**——并行度要自己用 worker 池控制,不是无限 `go` 出去完事。

## 六、Python asyncio:单线程的纪律考验

asyncio 是更"赤裸"的形态:**单线程事件循环 + 无栈协程**,没有 Go 那层多线程兜底,纪律全靠自己守。

```python
async def handler():
    resp = await client.get(url)
    return await resp.text()
```

规则只有一条但极严格:**只有 `await` 处才让出**。于是一行手滑就是全局事故:

```python
async def bad():
    time.sleep(10)        # 阻塞的是线程!事件循环停摆 10 秒,
                          # 所有协程、所有请求全部冻住

async def ok():
    await asyncio.sleep(10)   # 挂起的是协程,事件循环照常服务别人
```

同理,任何阻塞库(老 DB 驱动、requests、重 CPU 计算)直接在协程里调,都等于掐住单线程的心脏。出路:

```python
await asyncio.to_thread(blocking_func)   # 扔进线程池,别堵事件循环
```

asyncio 是观察"协程本质"的最佳标本:**它一点并行能力都没有(单线程),却能扛上万并发连接**——再次证明协程卖的是"等待的廉价化",不是算力。

## 七、Rust async:把运行时也交给你选

Rust 走得最极端:语言只定义一个接口,运行时(Tokio / async-std / glommio)是普通的第三方库。

```rust
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}
```

`await` 的本质是 executor 反复 `poll` 这个状态机:返回 `Poll::Pending` 就把 waker 登记到 IO 事件上,事件来了 waker 叫醒 executor 再 poll;返回 `Poll::Ready(v)` 即完成。没有 GC、不绑定运行时、内存占用是几种方案里最小的——**学费也最贵**:函数染色、`Send`/`Sync` 约束、lifetime 跨 await,以及和所有协程系统相同的铁律——不能在 async 任务里随手调阻塞函数。

## 八、阻塞 syscall:协程系统的天敌

三种语言反复强调的是同一件事,单独钉在墙上:**协程调度器只能接管"它知道的等待"**。`await`、channel、netpoller 注册的 IO 它都看得见;但一条裸的阻塞 syscall,内核可不管你用户态有什么调度计划:

```c
read(blocking_fd, buf, n);   // 没数据 → 内核直接把整条线程睡了
```

线程一睡,**它身上挂的几百个协程全部连坐**——明明 ready,却没有线程来跑。在 asyncio 里这表现为事件循环冻结,在 Go 里 runtime 用"换 M 保 P"抢救,但你不该指望抢救。四条正路:

1. fd 设非阻塞,统一交给 epoll / io_uring(运行时的标准做法);
2. 躲不掉的阻塞调用(某些文件 IO、阻塞的 C 库)扔进专门线程池;
3. 优先用 runtime 给的 async 版本(`asyncio.sleep`、Tokio 的 `tokio::fs`);
4. CPU 密集任务进 worker 池限并行度——长计算不挂起,本质上也是一种"阻塞"。

"不要在事件循环里做阻塞操作"不是风格建议,**是协程系统的生死线**。

## 九、白板时刻:为什么 Go 协程这么轻,CPU 密集却该用线程

把全篇拼起来,这道题就是送分题。

**"轻"轻在哪**——三笔账:

```
内存:  线程栈 MB 级   vs  goroutine 初始 2KB 起步、按需长
切换:  线程 μs 级(进内核 + cache 税) vs 协程 ns~百 ns(用户态换寄存器)
数量:  线程几千封顶   vs  协程百万随便开
```

**但这三笔省的全是"等待"的钱**。任务等 IO 时,协程把"占一个线程睡觉"优化成"留一条登记记录"——省内存、省切换,数量上限直接拔掉。

**CPU 密集时,这些优势一项都兑现不了**:

- 任务不等待,从头算到尾 → "挂起点便宜"毫无用武之地,反而**多了一层用户态调度开销和抢占的不确定性**;
- 算力天花板是物理核数 → 8 核开 1 万个协程,还是 8 个核在算,多出的 9992 个只是用户态 run queue 里的排队者(16 篇的排队论原样适用,只是队从内核搬到了用户态);
- 真正的并行靠底下的 M(线程)→ 既然如此,**直接开"核数"个线程,让内核调度器全权处理:路径最短、抢占最公平、cache 亲和性内核也帮你管**。

一句话收束:**协程优化的是并发(concurrency,同时持有很多任务),线程解决的是并行(parallelism,同时使用很多核);IO 密集要前者,CPU 密集要后者**。

落到工程选型:

| 场景 | 推荐 |
| --- | --- |
| CPU 密集(编码、压缩、推理) | 线程池/进程池,数量 ≈ 核数 |
| 高并发网络 IO(网关、长连接、爬虫、RPC fan-out) | 协程/事件循环 |
| 并发量小、业务简单 | 线程模型足够,别为用协程而用协程 |
| 既要吃满多核又是 IO 密集 | 多线程 runtime 上跑协程(Go / Tokio) |
| 需要故障隔离(崩了别带走全家) | 多进程(详见 11 篇) |

还有几个协程**解决不了**的问题,别指望它:大对象内存拷贝(带宽瓶颈)、锁竞争严重(14 篇,协程抢锁照样排队)、下游已限流(并发开再大也只是把队伍排到自己这边)、单连接带宽瓶颈。**协程提高的是并发承载量,不是单任务速度**。

## 踩坑提醒

1. **把协程当"更快的线程"**——它只把等待变便宜,CPU 密集任务切一万个协程,比 8 个线程还慢
2. **协程无限开不设上限**——每个协程仍有栈/状态/调度成本,百万协程把内存和调度队列一起拖垮
3. **在事件循环里调阻塞库**——一次慢 DNS、一行 `time.sleep`,整条线程上的协程全员冻结
4. **CPU 密集任务混进协程池不限并行度**——长计算不让出,等于阻塞,其他协程延迟爆炸
5. **忘了背压(backpressure)**——上游轻松开 10 万协程 fan-out,把 DB/下游打死,并发能力越强事故越大
6. **async 代码里持普通 mutex 跨 await**——挂起时锁不放,别的协程在同一线程上等锁,死锁或卡死调度线程
7. **以为协程能绕过锁**——共享数据该加锁还得加(14 篇),协程只是换了调度层,没取消数据竞争
8. **指望 Go runtime 兜底一切阻塞**——换 M 保 P 是急救不是日常,大量阻塞 syscall 让线程数悄悄膨胀
9. **拿单线程 asyncio 扛 CPU**——它没有任何并行能力,重计算请 to_thread 或上进程池
10. **选型只看语法甜不甜**——有栈/无栈、运行时轻重、染色问题,都是架构决策,不是语法糖偏好
