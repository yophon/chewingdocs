# 进程

进程是**操作系统最重要的抽象**——你 `./hello` 就创建了一个进程,Chrome 标签页是几十个进程,Docker 容器本质也是进程。但「进程」这个词背后藏着的细节让大多数程序员困惑:**fork 怎么"复制"出新进程而不慢死、exec 是怎么"换皮"的、为什么 ps 里有 Z 状态的"僵尸"、Linux 上进程和线程为什么本质是同一种东西**。这一篇把 Linux 进程的内核数据结构、状态机、生命周期讲透。

> 一句话先记住:**Linux 进程 = task_struct(内核数据结构)+ 资源(虚拟内存 / 文件描述符 / 信号 / 凭证)**。**fork = "复制 task_struct + 用 COW 共享物理页"**(几乎免费)、**exec = "保留 task_struct,但换掉所有资源"**。**Linux 没有"线程"** —— 所谓线程就是"和别的进程共享了一些资源的进程",这就是 clone() 系统调用的本质。

## 一、进程的本质:task_struct

每个进程在内核里就是一个 `task_struct` 结构体(几 KB):

```c
struct task_struct {
    pid_t pid;                  // 进程 ID
    pid_t tgid;                 // 线程组 ID(主线程 = pid)
    
    long state;                 // 状态:R / S / D / Z / T
    void *stack;                // 内核栈指针
    
    struct mm_struct *mm;       // 内存映射
    struct files_struct *files; // 文件描述符表
    struct fs_struct *fs;       // 当前目录、根目录
    struct signal_struct *signal; // 信号处理
    struct cred *cred;          // 用户/组 ID
    
    struct task_struct *parent; // 父进程
    struct list_head children;  // 子进程列表
    
    // ... 还有几百个字段
};
```

**一个进程 = 内核里的一个 task_struct**。

```bash
ls /proc/
# 1   2   100  1234  ...
# 每个数字目录就是一个进程,对应一个 task_struct
```

## 二、进程状态机

```
       fork()
         ↓
  ┌─── R (Running) ─────┐
  │       ↑              │ wait IO / mutex
  │       │              ↓
  │   被调度选中     S (Sleeping)  ← signal 唤醒
  │       │              ↑
  │       │              │
  │       └─── 时间片结束 ┘
  │
  ↓ exit()
  Z (Zombie) ← 等父进程 wait
  ↓ 父进程 wait
  消失
```

### 2.1 五个状态

```
R  Running       正在跑或可运行(在 runqueue 里)
S  Sleeping      可中断睡眠(等 IO / 信号 / mutex)
D  Disk Sleep    不可中断睡眠(等磁盘 IO,kill 都杀不掉)
Z  Zombie        僵尸,等父进程回收
T  Stopped       被信号(SIGSTOP / SIGTSTP)停了
```

```bash
ps aux
# USER  PID  STAT  ...
# root    1   Ss   /sbin/init
# user 1234   R    ./your_program
# user 5678   D    ./io_intensive
```

### 2.2 D 状态:不可中断的睡眠

```bash
ps aux | awk '$8 ~ /D/'
```

D 状态的进程**连 SIGKILL 都不响应**。常见原因:

- 等磁盘 IO(NFS 卡了)
- 等内核锁
- 等驱动响应

**长期 D 状态 = 内核 / 硬件出问题** —— 唯一办法是**重启**(kill 不了)。

### 2.3 Z 状态:僵尸进程

```c
// 父进程 fork 子进程,子进程 exit 后,父进程没 wait
fork(); 
// 子进程跑完 exit(0)
// 父进程不调 wait()
// → 子进程变 Z 状态,task_struct 不释放
```

**为什么不释放**:**保留退出状态供父进程查询**(`waitpid` 读到的退出码)。

**僵尸进程的危害**:

- 占用 PID(系统 PID 是有上限的)
- task_struct 占内核内存

**修复**:

- 父进程主动 wait()
- 父进程忽略 SIGCHLD(`signal(SIGCHLD, SIG_IGN)`)→ 内核自动收割
- **`init` 进程兜底**(下文)

### 2.4 孤儿进程:init 收养

```
父进程死了,子进程还活着
→ 子进程的 parent 变成 init (PID 1)
→ 子进程 exit 时由 init 调 wait,不会成僵尸
```

**这就是 PID 1 的特殊职责** —— 兜底回收所有"孤儿"。

**容器的坑**:容器里 PID 1 通常是你的应用(不是 init),**它没有 wait 孤儿的责任** → 大量僵尸。**用 tini / dumb-init 做 PID 1**。

## 三、fork:Linux 创建进程的看家本领

### 3.1 fork 的语义

```c
pid_t pid = fork();
if (pid == 0) {
    // 子进程
} else if (pid > 0) {
    // 父进程,pid 是子进程的 PID
} else {
    // 错误
}
```

**一次调用,两次返回**——这是新手最迷的点。**实际上 fork 创建了一个几乎相同的副本**,父子各执行一次 fork 后的代码,返回值不同。

### 3.2 fork 的内核动作

```
1. 复制 task_struct(浅拷贝大部分字段)
2. 复制 mm_struct(虚拟内存),但所有物理页标 read-only(COW)
3. 复制 files_struct(文件描述符表),fd 引用计数 +1
4. 复制 fs_struct(当前目录等)
5. 复制 signal handler
6. 子进程加入父进程的进程组 / 会话
7. 把子进程加入 runqueue
```

**详见 06 篇 COW 部分**。

### 3.3 fork 的性能

```
复制 task_struct:           几 μs
复制页表(指向 COW 页):     1-100 μs(取决于进程内存大小)
真正复制物理页:              0(COW,用时再复制)

总耗时:几 μs 到几 ms
```

**fork 一个 100GB 内存的进程**:**只要几十 ms**(而不是复制 100GB 数据需要的几十秒)。

### 3.4 fork 的常见用途

```c
// 1. 创建 worker 进程
for (int i = 0; i < N; i++) {
    if (fork() == 0) {
        worker();
        exit(0);
    }
}

// 2. fork + exec(运行别的程序)
if (fork() == 0) {
    execve("/bin/ls", ...);
}

// 3. 守护进程化(double fork)
if (fork() == 0) {
    setsid();
    if (fork() == 0) {
        // 真正的守护进程
    }
    exit(0);
}
```

## 四、vfork 和 clone:fork 的兄弟

### 4.1 vfork

`fork` 的轻量版:

```c
pid_t pid = vfork();
// 子进程必须立刻 exec 或 _exit,否则死锁
```

**vfork 不复制页表** —— 父子共享同一个地址空间,父进程被阻塞,直到子进程 exec / exit。

**用途**:fork + exec 的优化(省去 COW 开销)。**但现代 Linux fork 已经够快,vfork 用得少了**。

### 4.2 clone:fork 的"超能力版"

```c
clone(child_func, stack, flags, arg);
```

`flags` 决定**复制什么 / 共享什么**:

```
CLONE_VM         共享虚拟内存(创建线程时用)
CLONE_FS         共享文件系统信息
CLONE_FILES      共享文件描述符
CLONE_SIGHAND    共享信号 handler
CLONE_THREAD     和父进程同一个 TGID(就是"线程"!)
CLONE_NEWPID     新 PID namespace(容器用)
CLONE_NEWNET     新网络 namespace
...
```

**fork 等价于**:

```c
clone(SIGCHLD, ...);   // 不共享任何东西
```

**pthread_create 等价于**:

```c
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, ...);
// 共享几乎一切,这就是"线程"
```

**Docker 容器创建等价于**:

```c
clone(CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | ...);
// 隔离 namespace,这就是"容器"
```

> **所以 Linux 没有"线程"也没有"容器"** —— 它们都是带不同 flag 的 clone() 创建的"特殊进程"。

## 五、exec:进程的"换皮"

### 5.1 exec 系列

```c
int execve(const char *path, char *const argv[], char *const envp[]);
// 还有 execv / execl / execvp / execlp ... 这些都是 glibc 包装,最后都调 execve
```

### 5.2 exec 的内核动作

```
1. 加载新 ELF 文件到内存(替换原来的代码段、数据段)
2. 重置堆、栈
3. 重置信号 handler 为默认
4. 关闭 close-on-exec 的文件描述符
5. 跳到新程序的入口

没有改变的:
  - PID(还是同一个进程!)
  - PPID(父进程关系)
  - 打开的文件描述符(除了 close-on-exec)
  - 用户 / 组 ID
  - 当前目录
  - 进程组 / 会话
```

**fork + exec 是 Linux 创建新程序的标准模式**:

```c
pid_t pid = fork();
if (pid == 0) {
    // 子进程
    execve("/bin/ls", argv, envp);
    // 如果 execve 成功,后面的代码不执行
    // 失败才到这里
    perror("execve");
    exit(1);
}
// 父进程
waitpid(pid, &status, 0);
```

### 5.3 close-on-exec:防泄漏

```c
int fd = open("/etc/secret", O_RDONLY);
// 默认:fork + exec 后子进程也能访问 fd
// 安全风险:子进程能读父进程打开的文件

// 修复:open 时加 O_CLOEXEC
int fd = open("/etc/secret", O_RDONLY | O_CLOEXEC);
// exec 时 fd 自动关闭
```

**所有现代代码都该用 O_CLOEXEC** —— 默认行为是不安全的历史遗留。

## 六、进程间的关系

### 6.1 PID / PPID / PGID / SID

```
PID  进程 ID
PPID 父进程 ID
PGID 进程组 ID(一组相关进程,通常同一个命令)
SID  会话 ID(通常对应一个终端)
```

```bash
ps -o pid,ppid,pgid,sid,cmd
#  PID PPID PGID  SID CMD
# 1234  100 1234 1234 -bash
# 5678 1234 5678 1234 ls -la | grep foo
# 5679 1234 5678 1234 ...        ← 同 PGID = 同一条管道
```

### 6.2 进程组的用途

```bash
kill -SIGTERM -1234     # 给 PGID 1234 的所有进程发信号(注意 - 号)
```

终端 Ctrl+C 时,**信号发给前台进程组的所有进程**。

### 6.3 守护进程(Daemon)

无控制终端的后台进程:

```
1. fork(父进程退出,子进程脱离终端)
2. setsid(创建新会话,自己当 leader)
3. fork(再 fork 一次,确保不能再获取控制终端)
4. chdir("/")(避免占用任何挂载点)
5. umask(0)
6. close 0/1/2,重定向到 /dev/null
```

**现代写法**:**让 systemd 帮你做这些**——你的程序前台跑就行,systemd 用 cgroup 管理。

## 七、Linux 进程 vs 线程:同一回事

```
"进程" = 用 fork() / clone() 创建的、独立资源
"线程" = 用 clone(CLONE_VM | CLONE_THREAD ...) 创建的、共享资源

内核对它们一视同仁:都是 task_struct
区别仅在于"共享了什么"
```

**ps 和 top 默认只显示进程**(线程组的主线程)—— 因为所有线程共享一个 PID(TGID)。

**显示所有线程**:

```bash
ps -eLf       # L 显示线程
top -H        # 看每个线程
ls /proc/<pid>/task/    # 每个目录是一个线程
```

详见 12 篇线程模型。

## 八、wait 系列

### 8.1 wait / waitpid / waitid

```c
int status;
pid_t pid = wait(&status);              // 等任意子进程
pid_t pid = waitpid(child_pid, &status, 0);  // 等指定子进程
waitpid(-1, &status, WNOHANG);          // 非阻塞,没退出就立刻返回
```

### 8.2 解析 status

```c
if (WIFEXITED(status))
    printf("exit code: %d\n", WEXITSTATUS(status));
else if (WIFSIGNALED(status))
    printf("killed by signal: %d\n", WTERMSIG(status));
```

### 8.3 SIGCHLD

子进程退出时,内核给父进程发 SIGCHLD 信号 —— **父进程可以注册 handler 在信号里调 wait**:

```c
void sigchld_handler(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0)
        ;       // 收割所有已退出的子进程
}
signal(SIGCHLD, sigchld_handler);
```

**或者直接忽略 SIGCHLD,让内核自动收割**:

```c
signal(SIGCHLD, SIG_IGN);  // 子进程退出后内核自动清理,不变僵尸
```

## 九、进程优先级:nice 和实时

### 9.1 nice 值

```
范围:-20(最高优先级)~ 19(最低优先级)
默认:0
```

```bash
nice -n 10 ./background_task     # 启动时设
renice 10 -p 1234                # 改运行中的
```

**普通用户只能降优先级(nice 变大)**;root 才能升。

### 9.2 实时优先级

```bash
chrt -f 50 ./latency_critical   # SCHED_FIFO,优先级 50
chrt -r 50 ./...                 # SCHED_RR
```

实时调度类的进程**永远先于普通进程**,**不让出 CPU 就一直跑**——危险,小心搞挂系统。

详见 16 篇调度器。

## 十、cgroup:进程的资源限制

cgroup(control group)= 限制一组进程的 CPU / 内存 / IO / 网络:

```bash
mkdir /sys/fs/cgroup/memory/mygroup
echo 1G > /sys/fs/cgroup/memory/mygroup/memory.limit_in_bytes
echo $$ > /sys/fs/cgroup/memory/mygroup/cgroup.procs
# 当前 shell 及其子进程被限制 1GB 内存
```

**Docker / K8s 容器的资源限制本质就是 cgroup**——详见 25 篇容器底层。

## 十一、看进程的工具

```bash
ps -ef                  # 所有进程
ps aux                  # BSD 风格,带 CPU/内存
top                     # 实时
htop                    # 更友好
pstree                  # 进程树

cat /proc/<pid>/status  # 详细状态
cat /proc/<pid>/cmdline # 启动命令
cat /proc/<pid>/environ # 环境变量
ls /proc/<pid>/fd       # 打开的文件
cat /proc/<pid>/maps    # 内存布局(06 篇讲过)
cat /proc/<pid>/stat    # 一行的内核状态
```

`/proc/<pid>/stat` 是 ps / top 的数据来源——一行包含进程的所有内核可见状态。

## 十二、一些有趣的 PID

```
PID 0:    swapper / idle 进程(每核一个,CPU 空闲时跑)
PID 1:    init / systemd
PID 2:    kthreadd(所有内核线程的父进程)
PID > 2:  用户进程
```

`ps -ef | head` 里看到的 `[kworker]` `[ksoftirqd]` 等带方括号的是**内核线程**,不是用户进程。

## 十三、踩坑提醒

1. **fork 不 wait** —— 僵尸进程堆积
2. **容器 PID 1 不处理 SIGCHLD** —— 用 tini / dumb-init
3. **fork 大进程** —— 内存可能爆(COW 失效)
4. **D 状态进程 kill 不掉** —— 是内核 / 硬件问题,只能重启
5. **fork 后 stdout 没 flush** —— 父子进程都打印"重复"输出(buffer 被复制)
6. **不用 O_CLOEXEC** —— exec 子进程继承敏感 fd
7. **守护进程不 chdir** —— 占用挂载点,umount 失败
8. **signal handler 调 fork** —— 死锁
9. **以为 PID 永不重复** —— PID 是有限的,会回绕
10. **以为 fork 跨平台** —— Windows 没有 fork
11. **多线程进程 fork** —— 子进程只有一个线程(其他不复制),pthread_atfork 救
12. **ps 看不到线程** —— 用 ps -eLf
