# 进程

你每天都在创建进程——`./hello` 是一个,Chrome 开个标签页是好几个,`docker run` 起来的容器本质也是一个。但教科书那句"进程是程序的一次执行"等于没说,它回答不了你真正会撞上的问题:**fork 为什么"一次调用,两次返回"?ps 里那个连 `kill -9` 都杀不死的 D 状态是什么鬼?容器里怎么会堆出一山僵尸?进程和线程凭什么说是同一种东西?** 这一篇把进程的内核数据结构、状态机、生死流程一次讲透。模型建对了,线程(12)、IPC(13)、调度(16)全是顺水推舟。

> 一句话先记住:**Linux 进程 = 内核里的一个 task_struct + 它挂着的一串资源(虚拟内存、文件描述符、信号、凭证)**。fork = 复制 task_struct + 用 COW 共享物理页(几乎免费);exec = 保留 task_struct,换掉里面所有资源;**线程和容器都只是带不同共享 flag 的 clone()**——内核眼里根本没有"线程"和"容器"这两个词。

## 一、进程在内核眼里是什么:一张户口登记表

为什么要有"进程"这个抽象?因为 OS 要同时跑几百个程序,必须给每个程序记账:占了哪些内存、打开了哪些文件、信号来了怎么办、爸爸是谁。没有这张账本的世界就是 DOS——一次跑一个程序,跑飞了整机重启。这张账本就是 `task_struct`:

```c
struct task_struct {
    pid_t pid;                    // 进程 ID
    pid_t tgid;                   // 线程组 ID(主线程的 tgid = pid)

    long state;                   // 状态:R / S / D / Z / T
    void *stack;                  // 内核栈指针

    struct mm_struct *mm;         // 虚拟内存映射
    struct files_struct *files;   // 文件描述符表
    struct fs_struct *fs;         // 当前目录、根目录
    struct signal_struct *signal; // 信号处理
    struct cred *cred;            // 用户 / 组 ID(凭证)

    struct task_struct *parent;   // 父进程
    struct list_head children;    // 子进程列表
    // ... 还有几百个字段
};
```

注意这个形状:**本体很小(几 KB),资源全是指针挂出去的**。这是后面一切的伏笔——"复制进程"可以只复制本体、共享资源;"共享多少"可以按需调节。线程和容器的秘密全藏在这几个指针里。顺便,`/proc` 就是这张账本的对外窗口:`ls /proc/` 里每个数字目录就是一个进程的 task_struct 视图。

## 二、进程的一生:状态机

```
        fork()
          ↓
   ┌──→ R (Running) ←──────────┐
   │      │                     │
   │      │ 等 IO / 锁 / 事件   │ 事件到达 / 信号唤醒
   │      ↓                     │
   │   S (Sleeping) ────────────┘
   │   D (Disk Sleep:不可中断版的 S)
   │
   │ exit()
   ↓
   Z (Zombie:等父进程 wait 收尸) ──父进程 wait()──→ 彻底消失
```

```
R  Running     正在跑,或在 runqueue 里排队等 CPU
S  Sleeping    可中断睡眠(等 IO / 信号 / 锁),多数进程平时都在这
D  Disk Sleep  不可中断睡眠(等磁盘 IO / 内核锁),kill 都杀不掉
Z  Zombie      已退出,等父进程回收
T  Stopped     被 SIGSTOP / SIGTSTP 暂停(Ctrl+Z 就是它)
```

`ps aux` 的 STAT 列看的就是这个。

### 2.1 D 状态:连 kill -9 都不理你

D 和 S 都在睡,区别是 D **连 SIGKILL 都不响应**——它正卡在内核里干一件不能中断的事(典型是磁盘 IO 做一半,打断会留下不一致)。常见原因:NFS 卡死、等内核锁、等驱动。听起来吓人,其实短暂的 D 很正常;**要命的是长期 D——那是内核或硬件出了问题,kill 不掉,唯一解药是重启**。排查:

```bash
ps aux | awk '$8 ~ /D/'
```

### 2.2 Z 状态:僵尸进程名场面

事故现场:凌晨三点告警,fork 新 worker 全报 `EAGAIN`,`ps` 一看**几万个 Z 进程把 PID 池吃光了**。罪魁祸首往往就是:父进程 fork 了子进程,子进程 exit 之后,**父进程从没调过 wait()**。

为什么内核不直接清掉?**退出码还没人领**。子进程死时,内核要保留"它怎么死的"(退出码 / 被什么信号杀的)等父进程 `waitpid` 来查——僵尸就是这张"死亡证明"。它不占用户内存,但占 PID 和一小块内核内存,堆多了就 fork 不出新进程。修复三选一:父进程老实 `wait()`;`signal(SIGCHLD, SIG_IGN)` 让内核自动收尸;父进程先死,让 init 兜底(下面讲)。

### 2.3 孤儿进程:init 收养

反过来,**父进程先死、子进程还活着**叫孤儿。内核会把孤儿的 parent 改成 PID 1(init / systemd),它将来退出时由 init 负责 wait——**孤儿不会变僵尸,这是 PID 1 的法定职责**。

容器的坑就在这:**容器里的 PID 1 通常是你的应用**,它根本没写"收养孤儿、回收僵尸"的逻辑,于是只要有进程 fork 过孙子,僵尸就开始堆。解法:用 `tini` / `dumb-init` 当 PID 1。

## 三、fork:一次调用,两次返回

```c
pid_t pid = fork();
if (pid == 0) {
    // 我是子进程
} else if (pid > 0) {
    // 我是父进程,pid 是子进程的 PID
} else {
    // fork 失败
}
```

"一次调用,两次返回"没有任何魔法:**fork 把当前进程复制成两个几乎一样的副本**,两边都从 fork 返回处继续跑,只是返回值不同——子进程拿 0,父进程拿子进程 PID。一份代码,两个进程。

### 3.1 为什么复制 100GB 内存只要几十 ms

fork 直觉上该巨慢,实际内核干的是:

```
1. 复制 task_struct(浅拷贝大部分字段)
2. 复制 mm_struct 和页表,所有物理页标 read-only(COW)
3. 复制 files_struct,fd 引用计数 +1
4. 复制 fs_struct、signal handler
5. 子进程加入父进程的进程组 / 会话
6. 扔进 runqueue
```

关键在第 2 步:**物理内存一个字节都不复制**,父子共享物理页;谁先写哪页,内核才在缺页中断里复制那一页(Copy-On-Write,详见 06 篇)。账单:

```
复制 task_struct:        几 μs
复制页表(指向 COW 页):  1-100 μs(取决于内存大小)
真正复制物理页:          0(写到哪页才复制哪页)
总耗时:几 μs 到几 ms;fork 100GB 的进程只要几十 ms
```

Redis 的 BGSAVE 就靠这招:fork 出子进程对着"冻结"的内存快照慢慢写盘,父进程继续服务。但 COW 也有反面:**大进程 fork 后双方狂写,COW 不断触发实际复制,内存占用可能瞬间翻倍**——"fork 大进程内存爆掉"的事故来源。

### 3.2 三个经典用法

```c
// 1. 多进程 worker(Nginx / PHP-FPM 的骨架)
for (int i = 0; i < N; i++) {
    if (fork() == 0) { worker(); exit(0); }
}

// 2. fork + exec:运行别的程序(shell 跑命令的本质)
if (fork() == 0) { execve("/bin/ls", argv, envp); }

// 3. 守护进程化(double fork,第七节细讲)
if (fork() == 0) {
    setsid();
    if (fork() == 0) { /* 真正的守护进程 */ }
    exit(0);
}
```

## 四、exec:换皮不换人

fork 只能复制自己,shell 怎么跑 `ls`?靠 exec——**保留进程这个"壳",把里面的程序整个换掉**。

```c
int execve(const char *path, char *const argv[], char *const envp[]);
// execv / execl / execvp / execlp ... 全是 glibc 包装,最后都调 execve
```

```
换掉的:                              没换的:
  加载新 ELF,替换代码段、数据段        PID / PPID(还是同一个进程)
  重置堆、栈                           打开的 fd(除 close-on-exec)
  信号 handler 重置为默认              用户 / 组 ID、当前目录
  关闭 close-on-exec 的 fd             进程组 / 会话
  跳到新程序入口
```

**fork + exec 是 Unix 创建新程序的标准模式**,你敲的每条 shell 命令都是它:

```c
pid_t pid = fork();
if (pid == 0) {
    execve("/bin/ls", argv, envp);
    perror("execve");   // exec 成功就"再也回不来",落到这里 = 失败
    exit(1);            // 千万别忘
}
waitpid(pid, &status, 0);
```

注意名场面:**exec 成功是不返回的**(老代码已被换掉)。如果失败后忘了 `exit(1)`,子进程会继续执行父进程的代码——两个进程跑同一段主逻辑,数据写双份,排查半天发现是 PATH 配错导致 exec 失败。

还有个安全暗坑:**子进程默认继承父进程所有 fd**,exec 出来的第三方程序也能读你打开的 `/etc/secret`。修复一个 flag:

```c
int fd = open("/etc/secret", O_RDONLY | O_CLOEXEC);   // exec 时自动关闭
```

**现代代码应该默认加 O_CLOEXEC**——"继承一切 fd"是 1970 年代的遗留,放今天就是漏洞。

## 五、clone:fork、线程、容器其实是一家

先一句话送走 vfork:它连页表都不复制,父子直接共享地址空间,父进程挂起直到子进程 exec / _exit——是"fork 完马上 exec,连 COW 都想省"的历史优化。现代 fork 已够快,**vfork 基本退役**。

真正的主角是 clone,它把"资源是复制还是共享"做成了一组 flag:

```c
clone(child_func, stack, flags, arg);

CLONE_VM         共享虚拟内存        CLONE_THREAD   同一个 TGID(线程组)
CLONE_FS         共享 cwd / root     CLONE_NEWPID   新开 PID namespace
CLONE_FILES      共享 fd 表          CLONE_NEWNET   新开网络 namespace
CLONE_SIGHAND    共享信号 handler    ...
```

本篇最值钱的一张图——三个等价关系:

```
fork()           ≈ clone(SIGCHLD)
                   什么都不共享 → "进程"

pthread_create() ≈ clone(CLONE_VM | CLONE_FS | CLONE_FILES |
                         CLONE_SIGHAND | CLONE_THREAD)
                   几乎全共享 → "线程"

docker run       ≈ clone(CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | ...)
                   不共享 + 另开一套 namespace → "容器"
```

> 所以那句反直觉的话现在很自然:**Linux 没有"线程"也没有"容器"——都是带不同 flag 的 clone() 创建的"特殊进程"**。共享旋钮拧到最大是线程,拧到最小再加隔离是容器。

这也解释了 ps 的行为:线程组共享 TGID(对外显示为 PID),所以 `ps` / `top` 默认只显示进程(主线程)。看线程:

```bash
ps -eLf                 # L 列出所有线程
top -H                  # 按线程显示
ls /proc/<pid>/task/    # 每个子目录是一个线程
```

线程的完整故事详见 12 篇。

## 六、wait:给子进程收尸

第二节说了僵尸的成因,这里给齐工具:

```c
int status;
pid_t pid = wait(&status);                    // 等任意一个子进程
pid_t pid = waitpid(child_pid, &status, 0);   // 等指定子进程
waitpid(-1, &status, WNOHANG);                // 非阻塞:没人退出立刻返回 0
```

退出状态要用宏解析:

```c
if (WIFEXITED(status))
    printf("exit code: %d\n", WEXITSTATUS(status));
else if (WIFSIGNALED(status))
    printf("killed by signal: %d\n", WTERMSIG(status));
```

父进程不能一直阻塞在 wait 上,所以子进程退出时内核会发 **SIGCHLD**。标准写法是在 handler 里用 WNOHANG 循环收割:

```c
void sigchld_handler(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0)
        ;   // 一次信号可能对应多个退出的子进程,要循环收完
}
signal(SIGCHLD, sigchld_handler);
```

压根不关心子进程死活?更省事:

```c
signal(SIGCHLD, SIG_IGN);   // 明确忽略 → 内核自动收尸,不产生僵尸
```

## 七、进程的家谱:PGID、SID 与守护进程

进程除了父子关系,还有两层"行政编制":

```
PID  进程 ID            PGID  进程组 ID(通常 = 一条命令 / 管道)
PPID 父进程 ID          SID   会话 ID(通常对应一个终端)
```

```bash
ps -o pid,ppid,pgid,sid,cmd
# 5678 1234 5678 1234 ls -la | grep foo
# 5679 1234 5678 1234 ...        ← 同 PGID:管道两端是一个进程组
```

进程组存在的理由就一个:**批量发信号**。Ctrl+C 时终端把 SIGINT 发给整个前台进程组——所以 `ls | grep foo` 两个进程一起死,不留半截。手动发要在 PID 前加负号:`kill -SIGTERM -1234`(给 PGID 1234 的所有进程)。

守护进程(Daemon)= 没有控制终端的后台进程,教科书六步:

```
1. fork,父进程退出              4. chdir("/")(不占挂载点)
2. setsid(开新会话脱离终端)     5. umask(0)
3. 再 fork(确保拿不回控制终端)  6. 关闭 0/1/2,重定向 /dev/null
```

这六步值得理解,但 **2026 年的正确写法是:别自己做,交给 systemd**——程序老实前台跑,日志打 stdout,systemd 用 cgroup 管生死,比手写 double fork 可靠得多。

## 八、优先级与资源笼子:nice、实时调度、cgroup

**nice 值**:-20(最高优先级)到 19(最低),默认 0。普通用户只能调低自己(nice 变大),升优先级要 root。

```bash
nice -n 10 ./background_task    # 启动时设
renice 10 -p 1234               # 改运行中的
```

**实时调度**:实时类进程**永远优先于普通进程,不主动让 CPU 就一直跑**——一个死循环的实时进程能卡死整台机器,慎用。详见 16 篇调度器。

```bash
chrt -f 50 ./latency_critical   # SCHED_FIFO
chrt -r 50 ./...                # SCHED_RR
```

**cgroup**(control group):不调先后,直接画资源上限:

```bash
mkdir /sys/fs/cgroup/memory/mygroup
echo 1G > /sys/fs/cgroup/memory/mygroup/memory.limit_in_bytes
echo $$ > /sys/fs/cgroup/memory/mygroup/cgroup.procs
# 当前 shell 及其所有子进程被框死在 1GB 内存里
```

**Docker / K8s 的资源限制底下就是 cgroup**——详见 25 篇容器底层。

## 九、排查工具箱

```bash
ps -ef / ps aux         # 所有进程(aux 带 CPU / 内存)
top / htop              # 实时
pstree                  # 进程树(看家谱最直观)

cat /proc/<pid>/status  # 详细状态
cat /proc/<pid>/cmdline # 启动命令
cat /proc/<pid>/environ # 环境变量
ls  /proc/<pid>/fd      # 打开的文件
cat /proc/<pid>/maps    # 内存布局(06 篇讲过)
cat /proc/<pid>/stat    # 一行压缩的内核状态(ps / top 的数据源)
```

几个特殊 PID,看到别慌:

```
PID 0  swapper / idle(每核一个,CPU 没活干时跑它)
PID 1  init / systemd(所有孤儿的养父)
PID 2  kthreadd(所有内核线程的父进程)
```

`ps -ef` 里名字带方括号的(`[kworker]` `[ksoftirqd]`)是**内核线程**,没有用户态地址空间。

## 踩坑提醒

1. **fork 了不 wait**——僵尸堆积,PID 耗尽,fork 报 EAGAIN
2. **容器里让业务进程当 PID 1**——没人收养孤儿,用 tini / dumb-init
3. **fork 超大内存进程后双方狂写**——COW 不断实际复制,内存可能瞬间翻倍
4. **试图 kill 掉 D 状态进程**——D 不响应任何信号,那是内核 / 硬件问题,只能重启
5. **fork 前 stdout 没 flush**——用户态缓冲被复制,父子各打一遍,输出"重复"
6. **不加 O_CLOEXEC**——exec 出来的程序继承敏感 fd,安全漏洞
7. **exec 失败后不 exit**——子进程继续跑父进程的代码,主逻辑被执行两遍
8. **守护进程不 chdir("/")**——占着挂载点,umount 永远失败
9. **在 signal handler 里调 fork / malloc**——非异步信号安全,可能死锁(详见 24 篇)
10. **以为 PID 永不重复**——PID 有限,会回绕复用,拿 PID 当唯一键迟早出事
11. **以为 fork 跨平台**——Windows 没有 fork,只有 CreateProcess(≈ fork+exec 合体)
12. **多线程进程里 fork**——子进程只剩调用 fork 的那个线程,其他线程持有的锁永远锁死;pthread_atfork 只是缓解(12 篇细讲)
