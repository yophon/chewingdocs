# 文件抽象

Unix 最强的抽象之一是"一切皆文件":普通文件、目录、管道、socket、设备、epoll 实例,都可以用 fd 表示。这个抽象让程序员用 `open/read/write/close` 操作复杂世界,也让很多性能问题变得隐蔽:同样一个 `read`,背后可能是页缓存命中、磁盘 IO、网络等待、管道阻塞或设备驱动。这一篇讲清楚:**VFS、inode、dentry、file descriptor、open file description、页缓存和权限检查**。

> 一句话先记住:**fd 是进程里的整数句柄,inode 是文件系统里的对象,VFS 是内核统一文件接口**。`read(fd)` 不是"读文件"这么简单,而是从进程 fd 表找到 `struct file`,再走 VFS 到具体文件系统/设备/socket。**文件抽象的威力是统一,代价是你必须知道 fd 背后到底是哪种对象**。

---

## 一、为什么要有文件抽象

没有统一抽象时,每类设备都要一套 API:

```
read_disk()
read_keyboard()
read_network()
read_pipe()
read_tty()
```

Unix 选择统一成:

```c
int fd = open(path, flags);
read(fd, buf, n);
write(fd, buf, n);
close(fd);
```

好处:

- shell 管道天然成立
- 重定向简单
- 设备可像文件一样访问
- 网络 socket 也能放进 epoll
- 权限模型统一

---

## 二、fd 到底是什么

```c
int fd = open("/tmp/a.txt", O_RDONLY);
```

`fd` 只是进程 fd 表里的下标:

```
进程 fd table
0 → stdin
1 → stdout
2 → stderr
3 → struct file("/tmp/a.txt", offset=0, flags=RDONLY)
```

内核对象关系:

```
fd(int)
  ↓ 当前进程 fd table
struct file                ← open file description(包含 offset/flags)
  ↓
dentry                     ← 目录项缓存,名字到 inode
  ↓
inode                      ← 文件元数据和数据块位置
  ↓
具体文件系统(ext4/XFS/tmpfs/...)
```

### 2.1 fd vs open file description

```c
int fd1 = open("a.txt", O_RDONLY);
int fd2 = dup(fd1);
```

`fd1` 和 `fd2` 是两个 fd,但指向同一个 `struct file`,所以共享文件偏移:

```
read(fd1, 10)  offset 0 → 10
read(fd2, 10)  offset 10 → 20
```

而两次 `open` 不共享 offset:

```c
int fd1 = open("a.txt", O_RDONLY);
int fd2 = open("a.txt", O_RDONLY);
```

### 2.2 fork 后 fd

`fork()` 后父子进程 fd 表复制,但 `struct file` 共享:

```
父 fd 3 ─┐
         ├→ struct file(offset 共享)
子 fd 3 ─┘
```

这就是为什么多进程同时写日志要用 `O_APPEND`,不能靠共享 offset 自己算。

## 三、VFS:虚拟文件系统

VFS 提供统一接口:

```c
struct file_operations {
    ssize_t (*read)(struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write)(struct file *, const char __user *, size_t, loff_t *);
    int (*open)(struct inode *, struct file *);
    int (*release)(struct inode *, struct file *);
    __poll_t (*poll)(struct file *, poll_table *);
};
```

不同对象填不同实现:

| fd 背后 | read/write 去哪 |
| --- | --- |
| ext4 普通文件 | ext4 + page cache |
| tmpfs | 内存文件系统 |
| socket | 网络协议栈 |
| pipe | 管道 ring buffer |
| character device | 驱动 |
| eventfd/timerfd | 内核事件对象 |

所以 `read(fd)` 的成本从纳秒到秒级都有可能。

## 四、inode 和 dentry

### 4.1 inode

inode 保存文件元数据:

- 文件类型
- 权限/owner/group
- size
- 时间戳
- link count
- 数据块位置

文件名不在 inode 里。文件名在目录项里。

### 4.2 dentry

dentry = directory entry,缓存路径解析结果:

```
/var/log/app.log
 / → var → log → app.log
```

每一段路径都会查 dentry cache。热路径解析几乎都在内存完成。

### 4.3 硬链接

多个文件名可以指向同一个 inode:

```bash
ln a.txt b.txt
```

```
a.txt ─┐
       ├→ inode 123
b.txt ─┘
```

删除文件名只是减少 link count。只有 link count = 0 且没有进程打开它,数据才真正释放。

这解释了经典问题:

```
rm big.log 后 df 仍不降
```

因为进程还打开着这个文件。查:

```bash
lsof | grep deleted
```

## 五、页缓存:文件 IO 的第一站

普通文件读写通常先走 page cache:

```
read:
  page cache 命中 → 直接拷贝到用户 buffer
  miss → 发起磁盘 IO → 填 page cache → 拷给用户

write:
  写 page cache,标 dirty
  后台 writeback 刷盘
```

所以 `write()` 返回成功,不等于数据已经落盘。

要强制落盘:

```c
fsync(fd);      // 文件数据 + 必要元数据
fdatasync(fd); // 主要数据
```

### 5.1 O_DIRECT

`O_DIRECT` 尽量绕过 page cache:

```c
open(path, O_RDONLY | O_DIRECT);
```

适合数据库这类自己管理缓存的软件。

不适合普通应用:

- 对齐要求严格
- 小 IO 性能差
- 失去 page cache 合并/预读

## 六、文件偏移与原子性

```c
write(fd, buf, n);
```

普通 write 会使用并更新 `struct file` 里的 offset。

多线程共享 fd 时:

- 单次 `write` 对 offset 更新是内核保护的
- 但多次 `lseek + write` 组合不是原子的

需要指定偏移时用:

```c
pread(fd, buf, n, offset);
pwrite(fd, buf, n, offset);
```

追加日志用:

```c
open("app.log", O_WRONLY | O_APPEND);
```

`O_APPEND` 保证每次写之前内核把 offset 移到文件末尾,避免多个进程覆盖。

## 七、特殊 fd

### 7.1 pipe

```c
int p[2];
pipe(p);
```

- `p[0]` 读端
- `p[1]` 写端
- 内核里是一个有限大小 ring buffer

### 7.2 socket

socket 也是 fd:

```c
int s = socket(AF_INET, SOCK_STREAM, 0);
read(s, buf, n);
write(s, buf, n);
```

但背后不是文件系统,而是网络协议栈。

### 7.3 eventfd/timerfd/signalfd

Linux 把很多事件也做成 fd:

- `eventfd`:用户态/内核事件计数器
- `timerfd`:定时器 fd
- `signalfd`:把信号变成可读事件

好处:都能塞进 epoll。

## 八、常见坑

1. **fd 泄漏**:忘记 close,最终 `EMFILE: Too many open files`。
2. **fork/exec 泄漏 fd**:没设 `FD_CLOEXEC`,子进程继承敏感 fd。
3. **rm 后磁盘不释放**:文件被进程打开。
4. **write 成功误以为落盘**:没有 fsync,掉电可能丢。
5. **共享 fd 偏移混乱**:dup/fork 后共享 open file description。
6. **把 socket 当普通文件**:短写、半关闭、EAGAIN 都要处理。

## 九、排查命令

```bash
ls -l /proc/<pid>/fd
lsof -p <pid>
cat /proc/sys/fs/file-nr
ulimit -n
strace -e trace=openat,read,write,close -p <pid>
```

看文件缓存:

```bash
free -h
vmtouch file
mincore()
```

> **结论**:文件抽象把世界统一成 fd,但性能分析时必须反向拆开:这个 fd 背后到底是磁盘、内存、网络、管道还是驱动。抽象让代码简单,排障时要把抽象剥开。
