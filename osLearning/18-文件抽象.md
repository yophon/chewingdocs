# 文件抽象

你每天都在 `open / read / write / close`,但大概率说不清这几件事:为什么 `rm` 掉一个 20GB 的日志,`df` 显示磁盘一点没释放?为什么服务跑了三天突然报 `Too many open files`,重启就好,过三天又来?为什么 `write` 返回成功了,机器一掉电数据就没了?这些事故的根源是同一个:**你把 fd 当成了"文件本身",而它其实只是一根三层间接指针的最外层把手**。这一篇把 fd → file → inode 这条链拆开讲透,后面文件系统(19 篇)和 IO 模型(20 篇)都建在这条链上。

> 一句话先记住:**fd 是进程私有的整数下标,struct file 是内核里"一次打开"的状态,inode 才是文件本体;VFS(Virtual File System)把磁盘、管道、socket、设备全部统一在这条链后面**。`read(fd)` 不是"读文件",是顺着这条链走下去——走到哪、花多少时间,取决于 fd 背后到底是什么。

---

## 一、为什么"一切皆文件"

先看没有这个抽象的世界。每类设备一套 API:

```
read_disk(disk_id, sector, buf)
read_keyboard(kbd_id, buf)
read_network(conn_id, buf)
read_pipe(pipe_id, buf)
read_serial(tty_id, buf)
```

每来一种新设备,所有程序都要学一套新接口;想把"键盘输入"接到"网络输出",得写专门的胶水代码。Windows 早期就是这个路子,API 数量爆炸。

Unix 的选择激进得多:**全部统一成四个调用**。

```c
int fd = open(path, flags);
read(fd, buf, n);
write(fd, buf, n);
close(fd);
```

普通文件、目录、管道(pipe)、socket、终端、磁盘设备、甚至 epoll 实例,全是 fd。这个统一的回报大到超出设计者预期:

- **shell 管道天然成立**——`cat a.log | grep error | wc -l`,三个进程互相不知道对方是谁,反正都是读写 fd
- **重定向是免费的**——`./app > out.txt 2>&1`,程序自己完全无感
- **网络和定时器能塞进同一个 epoll**——因为 socket 和 timerfd 都是 fd(详见 21 篇)
- **权限模型只需要一套**——rwx 那一套管所有东西

但抽象是有账单的:**同样一个 `read`,背后可能是页缓存命中(百纳秒级)、磁盘 IO(SSD 几十微秒,机械盘几毫秒)、网络等待(不确定,可能永远)、管道阻塞(看对端心情)**。写代码时抽象帮你省事,排性能问题时你必须把抽象剥开,先回答"这个 fd 到底是什么"。

## 二、fd → file → inode:三层结构

这是全篇最重要的一张图,后面所有"怪事"都从这张图上读出来:

```c
int fd = open("/tmp/a.txt", O_RDONLY);   // 返回 3
```

```
进程内                          内核全局
┌─────────────┐
│ fd table     │
│ 0 → stdin    │
│ 1 → stdout   │
│ 2 → stderr   │      ┌──────────────────┐
│ 3 ───────────┼────→ │ struct file       │  ← open file description
└─────────────┘      │  offset = 0       │    (一次 open 一个)
   每个进程一张        │  flags = O_RDONLY │
                      └────────┬─────────┘
                               ↓ 经 dentry(路径名缓存)
                      ┌──────────────────┐
                      │ inode             │  ← 文件本体
                      │  权限/大小/时间戳  │    (一个文件一个)
                      │  数据块位置        │
                      └────────┬─────────┘
                               ↓
                       具体文件系统(ext4 / XFS / tmpfs / ...)
```

三层各管各的事:

| 层 | 归属 | 存什么 | 生命周期 |
| --- | --- | --- | --- |
| fd | 进程私有 | 一个整数下标 | close 或进程退出 |
| struct file | 内核,可被多 fd 共享 | **offset、打开 flags** | 引用计数归零 |
| inode | 文件系统 | 元数据 + 数据位置 | link count 为 0 且无人打开 |

**为什么要中间这层 struct file?** 因为"文件读到哪了"(offset)既不属于进程(两个 fd 可以指向同一次打开),也不属于文件(两次打开各读各的),它属于"**这一次打开**"——所以需要一个独立对象。

### 2.1 dup:两个 fd,一个 offset

```c
int fd1 = open("a.txt", O_RDONLY);
int fd2 = dup(fd1);          // fd2 指向同一个 struct file
```

```
fd1 ─┐
     ├→ struct file(offset 共享)→ inode
fd2 ─┘

read(fd1, buf, 10);   // offset: 0 → 10
read(fd2, buf, 10);   // offset: 10 → 20,接着读!
```

而两次 `open` 是两个 struct file,offset 各自从 0 开始:

```c
int fd1 = open("a.txt", O_RDONLY);   // struct file A,offset 0
int fd2 = open("a.txt", O_RDONLY);   // struct file B,offset 0
```

shell 的重定向就是靠 dup 家族(`dup2`)实现的:把 fd 1 指到一个文件的 struct file 上,程序往 stdout 写,实际进了文件。

### 2.2 fork:父子共享 offset 的"名场面"

`fork()` 会复制 fd table,但**指向的 struct file 是同一个**:

```
父进程 fd 3 ─┐
             ├→ struct file(offset 共享!)
子进程 fd 3 ─┘
```

这就是多进程写同一个日志文件的经典事故来源:两个进程都"先 lseek 到文件尾再 write",中间窗口里对方也动了 offset,结果互相覆盖,日志出现诡异的半行。**正确做法是 `O_APPEND`**:

```c
open("app.log", O_WRONLY | O_APPEND);
```

`O_APPEND` 保证**每次 write 之前,内核原子地把 offset 移到文件末尾**——定位和写入是一个不可分割的动作,多进程怎么并发都不会互相覆盖。

顺带补全偏移这件事:多线程共享 fd 时,单次 `write` 对 offset 的更新由内核保护,但 `lseek + write` 两步组合不是原子的。**需要在指定位置读写时,用一个调用搞定**:

```c
pread(fd, buf, n, offset);    // 读指定位置,不动 offset
pwrite(fd, buf, n, offset);   // 写指定位置,不动 offset
```

## 三、VFS:一个 fd,五种命运

三层链的最底下,VFS 用一张函数指针表把"read 到底干什么"留给具体实现:

```c
struct file_operations {
    ssize_t (*read)(struct file *, char __user *, size_t, loff_t *);
    ssize_t (*write)(struct file *, const char __user *, size_t, loff_t *);
    int (*open)(struct inode *, struct file *);
    int (*release)(struct inode *, struct file *);
    __poll_t (*poll)(struct file *, poll_table *);
};
```

每种 fd 背后填的是完全不同的实现:

| fd 背后是 | read/write 实际去哪 | 典型耗时 |
| --- | --- | --- |
| ext4 普通文件 | 页缓存,miss 才下盘 | 百 ns ~ 几 ms |
| tmpfs 文件 | 纯内存 | 百 ns |
| socket | 网络协议栈 | 不确定,可能永远等 |
| pipe | 内核 ring buffer | 看对端 |
| 字符设备 | 驱动代码 | 看设备 |
| eventfd / timerfd | 内核事件对象 | 微秒级 |

**这张表就是"抽象的账单"本身**:接口完全一样,成本差六个数量级。排查"为什么这个 read 慢"的第一步永远是 `ls -l /proc/<pid>/fd` 看清 fd 的真身,而不是去怀疑代码。

## 四、inode 与 dentry:文件名不是文件

### 4.1 inode 里没有文件名

inode 保存一个文件的全部元数据:

- 文件类型(普通文件 / 目录 / 设备 / ...)
- 权限、owner、group
- 大小、三个时间戳
- **link count(有几个名字指着我)**
- 数据块在磁盘上的位置

唯独**没有文件名**。文件名存在目录里——目录本质上就是一张"名字 → inode 号"的表。这个设计直接推出硬链接(hard link):

```bash
ln a.txt b.txt
```

```
目录项 a.txt ─┐
              ├→ inode 123(link count = 2)
目录项 b.txt ─┘
```

两个名字,同一个文件,改谁都是改它。`rm a.txt` 只是删掉一个目录项、link count 减一,数据毫发无伤。

### 4.2 名场面:rm 了 20GB 日志,磁盘没释放

数据真正释放的条件是两个同时满足:**link count = 0,且没有任何进程打开着它**。于是有了这个运维经典:

```
磁盘 95%,发现一个 20GB 的 app.log
rm app.log
df -h —— 还是 95%???
```

因为写日志的进程还开着这个文件——struct file 还引用着 inode,内核不敢释放数据块。文件从目录里消失了,但它以"deleted 但仍被打开"的幽灵状态继续占着盘。排查:

```bash
lsof | grep deleted     # 找到还开着已删除文件的进程
ls -l /proc/<pid>/fd    # 能看到 → /path/app.log (deleted)
```

解法不是 `rm`,而是**清空它**(`truncate -s 0` 或 `> /proc/<pid>/fd/N`)或重启进程。这个坑的反面用法相当优雅:程序创建临时文件后立刻 unlink,文件名消失但 fd 还能读写,**进程一退出(哪怕 crash)空间自动回收**——很多临时文件就是这么做的。

### 4.3 dentry:路径解析的缓存

解析 `/var/log/app.log` 要逐级查目录:`/` → `var` → `log` → `app.log`,每一级都是一次"查目录表 + 拿 inode"。如果每次都下盘,open 会慢得没法用。所以内核维护 dentry(directory entry)cache,把"路径段 → inode"的结果缓存在内存里——**热路径的解析几乎全在内存完成**。这也是为什么"一个目录里堆几百万个文件"会出性能问题:dentry / inode cache 被打爆(详见 19 篇小文件部分)。

## 五、页缓存:write 成功 ≠ 数据落盘

普通文件的读写默认都经过 page cache:

```
read:
  页缓存命中 → 直接拷到用户 buffer(快)
  miss      → 发磁盘 IO → 填页缓存 → 再拷给你(慢)

write:
  拷进页缓存,标记 dirty → 立刻返回"成功"
  后台 writeback 线程稍后刷盘
```

**最要命的就是 write 这条**:返回成功只代表"数据进了内核内存",离持久化还差一个 writeback 周期(默认可达 30 秒)。这期间机器掉电,数据就没了——"我明明写成功了啊"是存储类事故的高频开场白。要真正落盘必须显式要求:

```c
fsync(fd);       // 数据 + 元数据都推到持久层
fdatasync(fd);   // 主要推数据,能省一次元数据写
```

fsync 为什么慢、journal 在中间干了什么、怎么用 rename 做原子更新,是 19 篇的主线,这里先立住一个结论:**可靠性是要显式买的,write 默认不含**。

### O_DIRECT:绕开页缓存

数据库这类自己管理缓存的软件,不想让内核再缓存一遍(双重缓存浪费内存,还干扰自己的淘汰策略),可以:

```c
open(path, O_RDONLY | O_DIRECT);
```

但普通应用别碰它:对齐要求严格(buffer 地址、偏移、长度都要对齐块大小)、小 IO 性能反而差、失去页缓存的合并与预读。**O_DIRECT 是"我比内核更懂我的访问模式"的宣言,99% 的程序没资格说这句话**。

## 六、特殊 fd 家族:Linux 把一切事件也做成了 fd

文件抽象最妙的一步棋,是 Linux 后来把"事件"也统一了进来:

**pipe**——`pipe(p)` 返回一对 fd,`p[0]` 读端、`p[1]` 写端,内核里是一个有限大小的 ring buffer。写满了 write 阻塞,读空了 read 阻塞,这就是 shell 管道的全部底层(进程间通信的细节详见 13 篇)。

**socket**——`socket()` 返回的也是 fd,read/write 照用,但背后是网络协议栈,不是任何文件系统。socket 有自己的脾气:短写(一次 write 只写出去一部分)、半关闭、`EAGAIN`,把它当普通文件用必出事(详见 20 篇)。

**eventfd / timerfd / signalfd**——计数器、定时器、信号,全部包装成"可读的 fd":

```
eventfd   →  用户态/内核事件计数器,可读 = 有事件
timerfd   →  定时器到期 = fd 可读
signalfd  →  信号到达 = fd 可读
```

为什么费这个劲?**因为只要是 fd,就能塞进 epoll**。一个事件循环统一等"网络数据、定时器、信号、跨线程唤醒",不需要四套通知机制——这是 Nginx / Node / Go runtime 事件循环的结构基础(详见 21 篇)。

## 七、名场面:fd 泄漏打满 ulimit

fd 是有限资源。每进程默认上限通常是 1024(`ulimit -n`),超了之后所有 `open / socket / accept` 统统报错:

```
EMFILE: Too many open files
```

典型剧本:服务上线没事,跑三天开始疯狂报错,重启立刻恢复,三天后复发——**教科书级的 fd 泄漏**。常见漏点:异常路径上忘了 close、HTTP 客户端没复用连接、每次查询新建数据库连接不关。排查三板斧:

```bash
ls -l /proc/<pid>/fd | wc -l     # 这个进程开了多少 fd
lsof -p <pid>                     # 都是些什么(socket?文件?pipe?)
cat /proc/sys/fs/file-nr          # 系统全局开了多少
strace -e trace=openat,close -p <pid>   # 实时看谁在 open 不 close
```

如果 fd 数随时间线性上涨,看 lsof 里堆积的是哪类 fd,基本就能定位到代码。

还有一个隐蔽变种:**fork/exec 泄漏**。fd 默认会被子进程继承,如果你 fork 出一个长命子进程,父进程的数据库连接、监听端口全被它攥着——父进程重启时端口 bind 失败,就是这么来的。解法是打开时就声明"exec 后自动关闭":

```c
open(path, O_RDONLY | O_CLOEXEC);
```

顺手补几个看页缓存的工具:`free -h` 看整体 cache 量,`vmtouch file` 看某个文件有多少在缓存里,程序里用 `mincore()` 精确查询。

## 踩坑提醒

1. **"fd 就是文件"**——fd 只是进程私有的下标,中间还隔着 struct file 和 inode 两层;dup / fork 的共享行为、rm 不释放空间,全靠这三层才解释得通。
2. **"write 返回成功 = 数据安全了"**——只是进了页缓存,掉电就丢;要持久必须 fsync,这是显式付费项。
3. **"rm 了文件空间就回来了"**——还有进程开着就不释放,`lsof | grep deleted` 找凶手,truncate 或重启进程才放空间。
4. **忘记 close,fd 慢性泄漏**——三天一爆的 `EMFILE` 几乎都是它;监控 `/proc/<pid>/fd` 数量,上涨曲线比日志诚实。
5. **fork/exec 不设 CLOEXEC**——子进程继承敏感 fd,轻则端口占用,重则安全问题;open 时带 `O_CLOEXEC` 成为肌肉记忆。
6. **多进程写日志靠 lseek 自己算偏移**——fork 共享 offset,必然互相覆盖;追加写永远用 `O_APPEND`,定点读写用 pread/pwrite。
7. **把 socket 当普通文件读写**——短写、半关闭、EAGAIN 不处理,数据悄悄丢;fd 接口统一,语义不统一。
8. **普通应用乱开 O_DIRECT**——对齐麻烦、小 IO 变慢、丢预读;那是数据库的玩具。
