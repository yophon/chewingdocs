# 内存屏障与重排序

你写的代码顺序,不等于 CPU 真正执行的顺序。编译器会重排,CPU 会乱序执行,store buffer 会延迟写入,多核之间还能看到不同的内存顺序。单线程里这些优化通常"看不出来",一到并发就变成诡异 bug:明明先写 `data` 再写 `ready`,另一个线程却看到 `ready=true` 但 `data` 还是旧值。这一篇讲清楚:**为什么会重排序、内存屏障挡住什么、acquire/release 是什么、volatile 为什么不是锁、Java/Go/C++ 的 happens-before 到底在保证什么**。

> 一句话先记住:**重排序是性能优化,内存屏障是给优化划边界**。CPU 和编译器可以在不改变单线程语义的前提下乱序,但多线程看到的是共享内存,所以必须用 `mutex` / `atomic` / `volatile` / channel 这类同步原语建立 **happens-before**。**不要靠"我代码先写这一行"推断多线程可见性**——没有同步关系,就是数据竞争。

## 一、为什么会有重排序

### 1.1 单线程语义不是执行顺序

```c
a = 1;
b = 2;
c = a + b;
```

编译器可能把 `a` 和 `b` 的写入顺序交换,CPU 也可能先算后写。只要当前线程最后看到的结果一样,这就是合法优化。

**问题在多线程**:

```c
// 线程 1
data = 42;
ready = 1;

// 线程 2
if (ready)
    printf("%d\n", data);
```

你以为线程 2 一定打印 42。现实是:可能打印 0。

原因可能有三个:

- 编译器把 `ready = 1` 提前
- CPU 乱序执行 store
- `data` 还在当前核 store buffer 里,别的核看不到

### 1.2 CPU 为什么要乱序

CPU 比内存快太多。为了不空等,现代 CPU 会:

- **乱序执行**:后面的指令如果不依赖前面,先跑
- **store buffer**:写入先放 buffer,慢慢刷到 cache
- **load buffer**:读内存时并行发起多个请求
- **分支预测**:先猜一条路执行,猜错再回滚

这些优化让单核性能提高数倍,代价是**多核可见顺序变复杂**。

## 二、内存模型:先别背术语,先抓本质

### 2.1 数据竞争

两个线程同时访问同一块内存,至少一个是写,且没有同步关系,就是数据竞争。

```c
int x = 0;

// T1
x++;

// T2
x++;
```

这不仅是"结果可能不对",在 C/C++ 里还是 **undefined behavior**。编译器可以假设没有数据竞争,然后做出你完全想不到的优化。

### 2.2 happens-before

**happens-before = 语言/运行时承诺的可见性顺序**。

如果 A happens-before B,那么:

- A 之前的写入,对 B 可见
- A 和 B 不会被重排到破坏这个关系

常见建立方式:

| 场景 | happens-before |
| --- | --- |
| mutex | unlock happens-before 后续 lock |
| atomic release/acquire | release store happens-before acquire load |
| Java volatile | volatile write happens-before volatile read |
| Go channel | send happens-before receive |
| thread start/join | start 前的写对新线程可见,线程结束前的写对 join 后可见 |

**没有 happens-before,不要谈"应该能看到"。**

## 三、内存屏障是什么

内存屏障(memory barrier / fence)是一条特殊指令,告诉 CPU 和编译器:

```
这条线前后的内存操作,不能随便跨过去
```

### 3.1 四类重排

```
LoadLoad:   读 A / 读 B 交换
LoadStore:  读 A / 写 B 交换
StoreStore: 写 A / 写 B 交换
StoreLoad:  写 A / 读 B 交换
```

最贵的是 **StoreLoad**。它往往要求当前核把 store buffer 刷出去,再允许后续 load。

### 3.2 acquire / release

工程里最常用的是 acquire/release,不用直接写全屏障。

```c++
// 生产者
data = 42;
ready.store(true, std::memory_order_release);

// 消费者
if (ready.load(std::memory_order_acquire)) {
    assert(data == 42);
}
```

含义:

- **release store**:它之前的写,不能跑到 release 后面
- **acquire load**:它之后的读写,不能跑到 acquire 前面
- 如果 acquire 读到了 release 写入的值,两者建立 happens-before

这就是"发布数据 + 读取标志"的正确写法。

### 3.3 seq_cst

`memory_order_seq_cst` 是最强顺序:所有线程看到一个全局一致的 atomic 顺序。

优点:好理解。
缺点:可能更慢。

经验法则:**先用 mutex 或 seq_cst 写对,确认性能瓶颈后再降到 acquire/release**。

## 四、volatile 到底能干什么

### 4.1 C/C++ volatile

C/C++ 的 `volatile` 只保证:

- 每次都真的从内存读/写
- 编译器别把它优化掉

它不保证:

- 原子性
- 跨线程可见性
- CPU 内存屏障

```c
volatile int ready = 0;  // 不是线程同步原语
```

它适合 MMIO(读写硬件寄存器)、signal handler 某些标志位,**不适合普通多线程同步**。

### 4.2 Java volatile

Java 的 `volatile` 是同步原语:

- volatile write 有 release 语义
- volatile read 有 acquire 语义
- `long/double` 读写原子性也被保证

```java
volatile boolean ready;
int data;

// writer
data = 42;
ready = true;

// reader
if (ready) {
    System.out.println(data); // 一定能看到 42
}
```

但 Java volatile 仍然不是锁:不能保护 `count++` 这种复合操作。

## 五、经典 bug:双重检查锁

错误写法:

```java
class Singleton {
    static Singleton instance;

    static Singleton get() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null)
                    instance = new Singleton();
            }
        }
        return instance;
    }
}
```

`new Singleton()` 不是一步:

```
1. 分配内存
2. 调用构造函数
3. 把地址赋给 instance
```

如果 2 和 3 重排,另一个线程可能看到非 null 的 `instance`,但对象还没初始化完。

正确写法:

```java
static volatile Singleton instance;
```

或者更简单:用静态内部类 / enum / 依赖注入容器。

## 六、语言视角

### 6.1 C++ atomic

```c++
std::atomic<int> x{0};

x.store(1, std::memory_order_release);
int v = x.load(std::memory_order_acquire);
```

常见选择:

| 顺序 | 用途 |
| --- | --- |
| relaxed | 只要原子性,不要顺序,如统计计数 |
| acquire/release | 发布-订阅、队列指针 |
| acq_rel | read-modify-write |
| seq_cst | 默认,先写对 |

### 6.2 Java

Java 内存模型的核心:

- `synchronized` unlock happens-before 后续 lock
- `volatile` write happens-before read
- `Thread.start()` 前的写对新线程可见
- 线程结束前的写对 `join()` 后可见
- final 字段有特殊初始化安全保证

### 6.3 Go

Go 不鼓励手写内存屏障。常用同步方式:

- `sync.Mutex`
- `sync/atomic`
- channel
- `sync.WaitGroup`

Go 的原则很直接:**不要通过共享内存通信,要通过通信共享内存**。当然底层仍靠内存屏障实现。

## 七、工程判断

### 7.1 什么时候需要关心内存序

- 写无锁队列 / ring buffer
- 写高性能计数器、引用计数
- 写 runtime / JVM / GC / 调度器
- 排查偶发并发 bug
- 做跨语言 FFI 共享内存

普通业务代码,优先用锁。锁的成本通常比一次 RPC、一次数据库查询低几个数量级。

### 7.2 常见错误

1. **把 volatile 当锁**:只能保证可见性,不能保证复合操作原子性。
2. **用 relaxed 发布对象**:计数可以 relaxed,发布指针通常不行。
3. **自己发明双检锁**:语言提供的 lazy 初始化更可靠。
4. **以为 x86 强内存序就没事**:编译器仍会重排,ARM 更弱,容器可能跑在不同架构。
5. **过早写无锁结构**:ABA、内存回收、缓存抖动会吞掉收益。

## 八、排查清单

并发可见性问题通常表现为"复现不了"。按这个顺序查:

1. 共享变量是否所有访问都被同一把锁保护
2. 如果用 atomic,是否存在明确的 release/acquire 配对
3. 是否把普通变量和 atomic 混用
4. 是否依赖 sleep 让另一个线程"应该已经写完"
5. 是否跨语言共享内存但只在一边加了同步
6. 是否在 ARM / Apple Silicon / 云上 Graviton 才出问题

> **结论**:内存屏障不是给业务代码天天手写的,而是让你理解"同步原语为什么有效"。真正的工程建议很朴素:**共享数据先用锁,性能不够再用 atomic,最后才考虑无锁结构**。
