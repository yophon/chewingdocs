# 内存屏障与重排序

并发 bug 里最折磨人的一类长这样:明明先写 `data` 再写 `ready`,另一个线程却看到 `ready = true` 而 `data` 还是旧值;测试环境一万次跑不出来,生产的 ARM 服务器上一天出三次;加一行打印,bug 消失了。遇到这种事的人第一反应是"内存坏了吧"——不是,是你脑子里那个"代码按我写的顺序执行"的模型从来就不是真的。**编译器会重排你的代码,CPU 会乱序执行,写入还会在 store buffer 里趴一会儿才被别的核看见**。单线程里这些优化天衣无缝,一到多线程就露馅。这一篇讲清楚:重排序从哪来、内存屏障(memory barrier)挡住什么、happens-before 到底承诺了什么、双重检查锁为什么坑了一代 Java 程序员。

> 一句话先记住:**重排序是性能优化,内存屏障是给优化划边界**。编译器和 CPU 的契约只有一条——"不改变单线程语义",多线程可见性从来不在契约里。所以跨线程共享数据,必须用 mutex / atomic / volatile / channel 建立 **happens-before**。**没有 happens-before,就没有"应该能看到"——只有数据竞争**。

## 一、你的代码顺序,是个善意的谎言

### 1.1 契约只保单线程

```c
a = 1;
b = 2;
c = a + b;
```

编译器完全可以先写 `b` 再写 `a`,CPU 也可以把两个 store 乱序发射——只要**当前线程**最后算出 `c == 3`,就是合法优化。这条契约被严格遵守了几十年,单线程程序员岁月静好。问题出在多线程。看最经典的"发布"模式:

```c
// 线程 1:准备好数据,举旗          // 线程 2:看到旗,用数据
data = 42;                          if (ready)
ready = 1;                              printf("%d\n", data);
```

你的直觉:要么不打印,要么打印 42。现实:**可能打印 0**。把时间线画出来,凶手至少有三个:

```
凶手一:编译器重排(编译期就改了顺序)
  线程 1 实际指令:  store ready=1   store data=42
  线程 2:           读到 ready=1,此刻 data 还没写 → 打印 0

凶手二:CPU 乱序执行(指令顺序对,执行顺序不对)
  data=42 和 ready=1 没有数据依赖
  → CPU 为流水线效率,先把 ready=1 发出去了

凶手三:store buffer(执行顺序对,可见顺序不对)
  时间 →
  线程1(核0): data=42 →[进 store buffer,还没刷到 cache]
               ready=1 →[碰巧先刷出去了]
  线程2(核1):          读 ready → 1 ✓
                         读 data  → 0 ✗(核0 buffer 里那个 42,我看不见)
```

三层都可能作案,**哪层作案取决于编译器版本、CPU 架构、当时负载**——这就是这类 bug "复现不了"的根本原因。

### 1.2 CPU 为什么非要乱序

因为 CPU 比内存快太多(详见 04 篇:访存 ~100ns,CPU 一个周期 ~0.3ns)。干等内存等于把法拉利开成拖拉机,所以现代 CPU 全员"不守序":

- **乱序执行**:后面的指令不依赖前面的,先跑;
- **store buffer**:写先扔进核心私有小缓冲,攒着慢慢刷到 cache——**刷出去之前别的核看不到**;
- **load 提前**:并行发起多个读请求,谁先回来用谁;
- **分支预测**:先猜一条路执行,猜错回滚。

这些优化让单核性能翻几倍,账记在一处:**多个核看到的内存操作顺序可能彼此不一致**。x86 算管得严的(只允许 StoreLoad 一种乱序),ARM / RISC-V 弱得多——**同一份有竞争的代码,x86 上"碰巧"对,搬到 ARM(手机、Apple Silicon、云上 Graviton)立刻翻车**,这种迁移事故每天都在发生。

## 二、数据竞争:不只是"结果不对",是未定义行为

把术语钉死:**两个线程访问同一块内存,至少一个是写,且没有同步关系——就是数据竞争(data race)**。

```c
int x = 0;
// T1            // T2
x++;             x++;
```

你以为最坏结果是"x 等于 1 而不是 2"?在 C/C++ 里更糟:**数据竞争是 undefined behavior**。编译器被标准授权**假设你的程序没有数据竞争**,然后放手优化。比如它看到循环里反复读 `flag` 而循环体没写它:

```c
while (!stop) { work(); }
// 编译器:循环里没人改 stop(它假设没有别的线程!)→ 优化成:
if (!stop) while (1) { work(); }   // 死循环,另一个线程改 stop 也救不了
```

程序"卡死",而且开 `-O0` 就不卡——"加优化就坏"的 bug,九成是数据竞争。**结论:跨线程共享可变数据,裸读裸写一律不合法,没有商量**。

## 三、happens-before:语言给你的唯一承诺

怎样才算"有同步关系"?各语言的内存模型用同一个概念回答:**如果 A happens-before B,语言保证 A 以及 A 之前的所有写对 B 可见,且编译器和 CPU 不会重排到破坏这个关系**。

注意它不是物理时间的"先发生"——它是**语言/运行时白纸黑字承诺的可见性顺序**。建立方式各语言都备好了:

| 同步原语 | happens-before 规则 |
| --- | --- |
| mutex | unlock happens-before 之后的 lock |
| atomic release/acquire | release store happens-before 读到该值的 acquire load |
| Java volatile | volatile write happens-before 之后的 volatile read |
| Go channel | send happens-before 对应的 receive |
| 线程 start / join | start 前的写对新线程可见;线程结束前的写对 join 后可见 |

回头看就明白了:**为什么"加锁就安全"?不只因为互斥,还因为 unlock→lock 这条边把临界区里的写"押送"给了下一个持锁者**。锁是互斥 + 可见性的双保险——这就是上一篇(14 篇)结尾埋的那句话。

> 铁律:**没有 happens-before,不要谈"应该能看到"**。"我先写的呀"在多线程世界里不构成任何证据。

## 四、内存屏障:给乱序划红线

同步原语底下靠的就是内存屏障(memory barrier / fence)——一条特殊指令:**这条线前后的内存操作,不许随便跨过去**。

### 4.1 四类重排,一类最贵

```
LoadLoad:    读A / 读B 交换
LoadStore:   读A / 写B 交换
StoreStore:  写A / 写B 交换
StoreLoad:   写A / 读B 交换   ← 最贵
```

StoreLoad 贵有物理原因:挡住它意味着**当前核必须把 store buffer 整个刷出去,才能执行后面的 load**——等于暂停 CPU 最赚钱的优化。x86 上 `mfence` 或 LOCK 前缀指令干这个,几十个周期起步。

### 4.2 acquire / release:工程上够用的一对

全屏障太贵也太粗,工程上最常用的是一对方向性屏障:

```c++
// 生产者
data = 42;                                      // ① 普通写
ready.store(true, std::memory_order_release);   // ② release 写

// 消费者
if (ready.load(std::memory_order_acquire)) {    // ③ acquire 读
    assert(data == 42);                         // ④ 保证成立
}
```

```
release store:  ┌─ 我之前的写(①)不许跑到我(②)后面 ──┐
                │   "把行李打包好,再举旗"                │
acquire load:   └─ 我之后的读写(④)不许跑到我(③)前面 ─┘
                    "看到旗,再开行李"

③ 读到了 ② 写的值 ⇒ 建立 happens-before ⇒ ④ 必能看到 ①
```

记这个比喻:**release 是"打包发货",acquire 是"签收拆箱"**——货(data)绝不会比签收单(ready)晚到。这就是"发布数据 + 检查标志"模式的唯一正确写法。

### 4.3 seq_cst:先写对,再谈快

`memory_order_seq_cst`(顺序一致)是最强档:所有线程看到的所有 seq_cst 操作构成**一个全局一致的顺序**,跟直觉里的"程序就该这么跑"完全一致。它是 C++ atomic 的默认值,也是 Java volatile 的实际强度。代价是更多屏障(尤其 StoreLoad),可能更慢。

经验法则:**先用 mutex 或 seq_cst 写对,profile 证明这里真是瓶颈,再小心降到 acquire/release**。从 relaxed 往上加是地狱难度,从 seq_cst 往下减是普通难度。

## 五、volatile:同一个词,两个世界

这个关键字坑人无数,因为它在 C/C++ 和 Java 里**根本是两个东西**:

| | C/C++ volatile | Java volatile |
| --- | --- | --- |
| 禁止编译器优化掉读写 | ✓ | ✓ |
| CPU 内存屏障 | ✗ | ✓(write 带 release,read 带 acquire) |
| 跨线程可见性 | ✗ | ✓ |
| 原子性 | ✗ | ✓(含 long/double 的读写) |
| 是线程同步原语吗 | **不是** | **是** |

C/C++ 的 `volatile` 只对编译器说"每次老老实实去内存读写,别优化"——CPU 该乱序照样乱序。它的正经用途是 **MMIO(读写硬件寄存器)和 signal handler 里的标志位**,拿它做线程同步是经典错误。

Java 的 `volatile` 是 JMM 钦定的同步原语:

```java
volatile boolean ready;
int data;

// writer                    // reader
data = 42;                   if (ready) {
ready = true;                    System.out.println(data);  // 保证打印 42
                             }
```

但注意,**Java volatile 也不是锁**:它保证可见性和顺序,不保证复合操作原子性——`count++` 还是三步,得用 `AtomicInteger` 或锁。

## 六、名场面:双重检查锁,坑了一代 Java 程序员

懒加载单例,有人嫌每次进同步块太亏,发明了"双重检查锁"(Double-Checked Locking):

```java
class Singleton {
    static Singleton instance;          // 注意:没有 volatile

    static Singleton get() {
        if (instance == null) {                    // 第一次检查:不加锁,快
            synchronized (Singleton.class) {
                if (instance == null)              // 第二次检查:防重复创建
                    instance = new Singleton();
            }
        }
        return instance;
    }
}
```

逻辑严丝合缝,对吧?**错,而且错得极隐蔽**。`new Singleton()` 不是一步,是三步:

```
1. 分配内存
2. 调用构造函数(初始化字段)
3. 把地址赋给 instance
```

2 和 3 之间**没有数据依赖**,JIT 或 CPU 完全可以重排成 1→3→2。于是:

```
线程 A:分配内存 → instance = 地址(还没构造!)→ 此刻被线程 B 插入
线程 B:第一次检查 instance != null → 不进同步块,直接 return
        拿到一个字段全是默认值的"半成品对象" → 后续使用静默出错
```

最毒的是:线程 B 走的是无锁快路径,**synchronized 的 happens-before 根本没轮到它**。这个 bug 在 Java 1.5 修订 JMM 之前连 volatile 都治不了,《The "Double-Checked Locking is Broken" Declaration》是并发领域最有名的檄文之一。1.5 之后的修法:

```java
static volatile Singleton instance;   // volatile write 带 release 语义
                                      // 构造(步骤2)不许重排到赋值(步骤3)后面
```

或者干脆别玩花活——**静态内部类持有、enum 单例、依赖注入容器**,让类加载机制替你保证安全。这个名场面的教训值得加粗:**"逻辑上推不出错"不等于"内存模型下没有错"——并发正确性必须沿着 happens-before 推,不能沿着代码行号推**。

## 七、三种语言的姿势

### 7.1 C++:手动挡

```c++
std::atomic<int> x{0};
x.store(1, std::memory_order_release);
int v = x.load(std::memory_order_acquire);
```

| 内存序 | 什么时候用 |
| --- | --- |
| relaxed | 只要原子性不要顺序:统计计数、监控指标 |
| acquire / release | 发布-订阅、无锁队列的头尾指针 |
| acq_rel | read-modify-write(fetch_add 同时要两边语义) |
| seq_cst | 默认,先写对 |

注意 relaxed 的边界:**计数可以 relaxed,发布指针不行**——relaxed 不携带"它之前的写"的可见性,用它发布对象等于把双检锁 bug 再发明一遍。

### 7.2 Java:JMM 的硬承诺

- `synchronized` 的 unlock happens-before 之后的 lock;
- `volatile` write happens-before 之后的 read;
- `Thread.start()` 前的写对新线程可见;线程的所有写对 `join()` 之后可见;
- **final 字段有特殊的初始化安全保证**:构造函数正常结束后,别的线程不加同步也能看到 final 字段的正确值——这是不可变对象天生线程安全的法理依据。

### 7.3 Go:自动挡

Go 故意不暴露内存屏障,官方姿势就四样:`sync.Mutex`、`sync/atomic`、channel、`sync.WaitGroup`。设计哲学一句话:**不要通过共享内存来通信,要通过通信来共享内存**——channel 的 send happens-before receive,把可见性问题整个藏进语言里。底层当然还是屏障,但那是 runtime 的事,不是你的事。

## 八、工程判断:什么时候需要想这一层

老实说,**普通业务代码不需要手写内存序**——一次 RPC 几毫秒,一次锁几十纳秒,差五个数量级,优化锁是在给大象减一根头发的肥。真正需要关心的场景:

- 写无锁队列 / ring buffer / 高性能计数器、引用计数;
- 写 runtime、JVM、GC、调度器这类基础设施;
- 跨语言 FFI 共享内存(两边内存模型可能对不齐);
- **排查偶发的、"复现不了"的并发 bug**——大多数人第一次被迫学这层的原因。

排查清单,按命中率排序:

1. 共享变量是不是**所有**访问路径都被同一把锁保护?(漏一条读路径就全完)
2. 用 atomic 的话,release 和 acquire 是否**配对**?(只有一边等于没有)
3. 是否把普通变量和 atomic 标志混用,指望标志"顺便"保护数据?
4. 是否靠 `sleep` 等"对方应该写完了"?(sleep 不建立任何 happens-before)
5. 是否只在 x86 上测过?**x86 内存序强会替你掩盖 bug,ARM / Apple Silicon / Graviton 一换就炸**;
6. 上探测器:C/C++ 用 ThreadSanitizer(`-fsanitize=thread`),Go 用 `-race`——比人眼靠谱一百倍。

## 踩坑提醒

1. **把 C/C++ volatile 当线程同步用**——只管编译器不管 CPU,既无屏障也无原子性,只配伺候硬件寄存器
2. **把 Java volatile 当锁用**——可见性有了,`count++` 这种复合操作照样竞争
3. **裸写共享变量,觉得"就一个 bool 没事"**——数据竞争在 C/C++ 是未定义行为,编译器能把你的循环优化成死循环
4. **靠代码行号推可见性**——"我先写的"不构成证据,只有 happens-before 构成证据
5. **自己发明双重检查锁**——`new` 的三步会重排,要么 volatile 要么静态内部类/enum
6. **用 relaxed 发布对象指针**——relaxed 不携带之前写的可见性,等于重演 DCL 事故
7. **只在 x86 上测并发**——x86 只许 StoreLoad 乱序,ARM 四类全开,上线 Graviton 才炸就晚了
8. **用 sleep "等对方写完"**——不建立 happens-before,只是把 bug 概率调低到测试测不出来
9. **一上来就抠内存序**——先 mutex/seq_cst 写对,profile 证明是瓶颈再降级
10. **不知道有竞争探测器**——TSan / Go race detector 一跑一个准,比盯代码三天强
