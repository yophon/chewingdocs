# 锁与原子操作

写并发代码的人容易掉进两个极端:一个是**所有共享数据加同一把大锁**——程序对了,但 32 核跑出单核性能;另一个是**追求"无锁"堆一堆 atomic**——性能没上去,还藏着 ABA 这种只在生产环境凌晨三点出现的 bug。两个极端的根源是同一个:**不知道锁底下到底发生了什么**。这一篇把锁拆开给你看:自旋和睡眠的本质区别、futex 怎么让"无竞争的锁"几乎免费、CAS 为什么是无锁世界的地基又为什么靠不住、以及 1997 年火星探路者号怎么被一把锁搞到反复重启。

> 一句话先记住:**锁的本质是"我占着,别人等"——所有锁的差别,只是"等的方式"不同**。自旋锁(spinlock)原地空转,适合临界区短于 1μs;互斥锁(mutex)睡眠等唤醒,适合更长的临界区。futex 让 mutex 在无竞争时只是一条用户态 CAS(~25ns),根本不进内核。atomic 也不免费——LOCK 指令加 cache 一致性流量,比普通操作慢 10-20 倍。

## 一、没有锁的世界:一行 `counter++` 怎么丢数据

```c
int counter = 0;
// 线程 1 和线程 2 同时执行:
counter++;
```

你眼里这是一行,CPU 眼里是三条指令:**load → add → store**。两个线程一交错:

```
T1: load 0    add → 1            store 1
T2:      load 0      add → 1            store 1
最终 counter = 1   ← 应该是 2,T1 的更新被覆盖
```

这叫**丢失更新(lost update)**,最要命的是它**不必现**——可能跑一万次出一次,测试全绿,上线每天丢几条数据。锁干的事就一件:**让"读 - 改 - 写"整段原子化**——我做时你不许碰。那"你不许碰"期间,你在干什么?这个问题的不同答案,就是不同的锁。

---

## 二、两种等法:自旋 vs 睡眠

抢不到锁的线程有两个选择:**自旋**(站门口原地转圈,几纳秒看一眼)或**睡眠**(留个电话回家睡,门开了让内核叫醒)。

| 维度 | 自旋锁 | 互斥锁(mutex) |
| --- | --- | --- |
| 等待时干嘛 | 空转烧 CPU | 睡眠,让出 CPU |
| 等待成本 | 每等 1μs 烧 1μs CPU | 睡眠 + 唤醒 ~1-5μs(进内核) |
| 适合临界区 | 极短(< 1μs) | 长(> 1μs) |
| 能用在中断里吗 | 能(不睡眠) | 不能 |

判断标准就一条:**等的预期时间 vs 一次"睡眠 + 唤醒"的成本**。临界区只有几十纳秒(改计数器、插链表节点),为它睡一觉等于为过马路叫出租车;临界区几十微秒,自旋就是纯烧电。

自旋锁长这样:

```c
while (atomic_test_and_set(&lock));  // 反复 CAS 直到成功
// 临界区
lock = 0;
```

适合三种场景:**临界区极短**、**多核并行**(持锁者正在另一个核上跑,马上放)、**不能睡眠的环境**(中断处理函数、内核某些路径)。反过来三种场景是灾难:**单核**(你转圈占着 CPU,持锁者跑不了,活活锁死)、**临界区长**(top 里 CPU 100%,工作量为零)、**竞争激烈**(几十个核对同一变量狂 CAS,cache line 被反复打失效,MESI 流量打爆总线,详见 10 篇)。

用户态有 `pthread_spin_lock`,但基本别碰:**用户态线程随时可能被调度器切走**(详见 16 篇),你持自旋锁时被抢占,其他线程就在空转等一个"正在睡觉的人"。内核能用 spinlock 是因为它可以关抢占,用户态没这特权。**用户态 99% 用 mutex,内核里 spinlock 才是大宗**。

## 三、mutex 与 futex:无竞争的锁几乎免费

mutex 的行为:抢不到就睡,持有者 unlock 时唤醒等待队列里的一个。听起来每次加锁都要进内核排队?早期(Linux 2.4 之前)确实——每次 lock/unlock 都是一次 syscall,哪怕没人抢。这很冤:**真实程序里绝大多数加锁是无竞争的**,为没人抢的锁付几百 ns 的 syscall(详见 02 篇)纯属浪费。

### 3.1 futex:Linux 2.5 的关键发明

**futex = Fast Userspace muTEX**,思想画成流程图:

```
            lock()
              │
        CAS(state, 0→1)
              │
      ┌──成功─┴─失败──────────────┐
      │                          │
  直接进临界区              标记"有人在等"(state→2)
  ~25ns,纯用户态,              │
  一次 syscall 都没有      futex_wait(&state, 2) ← 这才进内核睡眠
                                 │
                          被唤醒,重新 CAS 抢锁

            unlock()
              │
        atomic 减 state
              │
      ┌─原值是 1┴─原值是 2─────────┐
      │                          │
  没人在等,直接返回        futex_wake(&state, 1) ← 有人等才进内核
  (还是不进内核)
```

简化伪代码(glibc 真实实现的骨架):

```c
atomic_int state;   // 0 = 没锁, 1 = 锁着, 2 = 锁着且有人在等

void lock() {
    if (CAS(&state, 0, 1)) return;        // 快路径:无竞争,~25ns
    do {
        if (state == 2 || CAS(&state, 1, 2))
            futex_wait(&state, 2);        // 慢路径:syscall 睡眠
    } while (!CAS(&state, 0, 2));
}

void unlock() {
    if (atomic_dec(&state) != 1) {        // 从 2 减下来 → 有人在等
        state = 0;
        futex_wake(&state, 1);
    }
}
```

性能量级背下来:

```
无竞争拿锁:  ~25 ns(纯用户态 CAS)
有竞争睡眠:  ~1 μs(syscall)
唤醒:        ~1-5 μs
```

**关键洞察:锁本身不贵,竞争才贵**。"程序锁太多所以慢"通常是错误诊断,对的诊断是"在某把锁上**竞争**太凶"。

glibc 的 pthread_mutex 就是 futex 实现的,无竞争场景比 Linux 2.4 时代快 10-100 倍。而且**几乎所有现代锁原语都建在 futex 上**:mutex、条件变量、读写锁、信号量、Java 的 synchronized / ReentrantLock。

## 四、读写锁:看起来美,适用窗口窄

场景:一张配置表,1000 个线程读,偶尔一个线程写。mutex 让读者互相排队——但**读和读根本不冲突**,排队纯属冤枉。

```c
pthread_rwlock_rdlock(&lock);   // 读锁:多个读者并发
pthread_rwlock_wrlock(&lock);   // 写锁:独占
pthread_rwlock_unlock(&lock);
```

但适用窗口比想象中窄,**两个条件要同时满足**:**读远多于写**(如 95% 读 / 5% 写,写一多照样排队)、**读临界区够长**(几十 μs 以上——读写锁自身记账比 mutex 重,临界区短时省下的并发不够付开销,**反而比 mutex 慢**)。

读优先策略下还有经典死法——**写者饥饿(writer starvation)**:

```
读者 A 持有读锁 → 写者 B 等 → 读者 C、D、E…… 源源不断拿到读锁
写者 B:只要读者流不断,我永远等不到
```

对策是写优先(`pthread_rwlockattr_setkind_np` 设 `PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP`)——但这又可能饿读者。**两难,慎用**。

内核里有更狠的方案 **RCU(Read-Copy-Update)**:读者不加任何锁直接读(几乎零开销);写者复制一份改副本,atomic 换指针,等所有还在读旧版本的读者走完再释放。Linux 内核大量用(路由表、文件系统),用户态不常见,只适合"指针式数据结构"。思想值得记:**读多写少的极端场景,正确答案往往不是更好的锁,而是让读者根本不需要锁**。

## 五、条件变量:等"事情发生",不是等锁

锁解决互斥,但还有一类等待是"等队列里有东西、等任务完成"。死循环轮询太蠢,这是条件变量(condition variable)的活:

```c
// 等待方
pthread_mutex_lock(&mu);
while (!ready)                       // 注意:是 while,不是 if!
    pthread_cond_wait(&cv, &mu);     // 原子地"放锁+睡眠";醒来自动重新拿锁
pthread_mutex_unlock(&mu);

// 通知方
pthread_mutex_lock(&mu);
ready = 1;
pthread_cond_signal(&cv);            // 唤醒一个;broadcast 唤醒所有
pthread_mutex_unlock(&mu);
```

**为什么必须 while 不能 if**:`pthread_cond_wait` 存在**虚假唤醒(spurious wakeup)**——没人 signal 它也可能无故醒来(内核实现允许,消除它代价太大)。用 if,虚假醒来后条件没成立就往下跑,bug 埋下了。**while 重查条件,发现是假的接着睡**——这是铁律。Java 的 `wait` / `notify` / `notifyAll` 是同一模式,连"必须在循环里 wait"的规矩都一样。

## 六、原子操作:不用锁的最小武器

### 6.1 C11 atomic 与内存序

```c
#include <stdatomic.h>

atomic_int counter;
atomic_fetch_add(&counter, 1);                                  // 原子加
atomic_compare_exchange_strong(&counter, &expected, new_val);   // CAS
atomic_load(&counter);
atomic_store(&counter, 0);
```

原子计数器不需要锁——硬件层面(x86 的 `LOCK XADD`)保证读改写不被打断。每个 atomic 操作还带一个隐藏参数——内存序(memory order),共 6 种:

```
memory_order_relaxed    最弱:只保证这一个操作原子,不约束前后顺序
memory_order_consume    依赖排序(C++ 实践中几乎不用)
memory_order_acquire    读屏障:我之后的读写不许跑到我前面
memory_order_release    写屏障:我之前的写对拿到这个值的人可见
memory_order_acq_rel    acquire + release
memory_order_seq_cst    最强:全局顺序一致(默认)
```

现在不用全懂——**99% 的场景用默认 seq_cst 就对了**。什么时候降级、acquire/release 到底挡什么,是下一篇(15 篇)的主菜。

### 6.2 CAS 与 ABA

```c
atomic_compare_exchange_strong(&p, &expected, new_val);
// 原子地执行:
//   if (p == expected) { p = new_val; return true; }
//   else               { expected = p; return false; }
```

CAS(Compare-And-Swap)的套路:**读旧值 → 算新值 → CAS,失败说明有人抢先,重读重试**。futex 快路径、所有无锁数据结构,地基都是它。

但 CAS 只比较"值变没变",不知道"中间发生过什么"。看无锁栈的事故:

```
T1: 读到栈顶 head = A,准备 CAS(head, A, A->next)
T1 被调度走……
T2: pop A,pop B,把 A 的内存复用,push 回一个"新 A"
T1 醒来: CAS(head, A, ...) → 值还是 A,成功!
         但 A->next 早是悬空指针 → 崩溃或静默数据损坏
```

值是 A → 还是 A,**但语义已经换了人间**——这就是 ABA 问题。对策:**版本号**(指针旁带代数,double-width CAS 一起比)、**危险指针(hazard pointer)**(读者声明"我正在用",回收前先查)、**RCU 风格延迟回收**。每个都不简单。

### 6.3 atomic 不是免费的

```
普通 add:              ~1 ns
atomic add:            ~10-20 ns(LOCK 指令 + cache line 独占)
mutex 锁改解(无竞争): ~30 ns
```

atomic 比普通操作慢 10-20 倍,而且这是单线程数字。**多核狂改同一个 atomic 变量时,大头是 cache 一致性流量**——那条 cache line 在核间被反复抢(详见 10 篇 MESI),8 核同时 fetch_add 一个计数器可能比单核还慢。所以高性能计数器都按 per-CPU / per-thread 分片,最后汇总。

## 七、无锁数据结构:博士论文常客,工业代码稀客

把 CAS 玩到极致就是无锁(lock-free)数据结构:Michael-Scott 无锁队列、无锁栈、无锁哈希表。卖点诱人——没有锁所以没有死锁,理论上总有线程在前进,论文也好发。但工业上有四宗罪:

1. **ABA**:上面讲了,必须配版本号或 hazard pointer,复杂度翻倍;
2. **内存回收**:你想 free 一个节点,怎么确定没有别的线程**正拿着指针在读**?锁的世界里持锁即独占,无锁世界里这是最难的问题;
3. **正确性**:证明是论文级工作,普通工程师手写**几乎肯定有 bug**,而且只在特定交错下出现;
4. **性能边界**:写竞争激烈时一堆线程 CAS 失败重试,cache line 抢成一团,**不一定比一把 mutex 快**。

顺带分清术语:**lock-free** 指系统整体总有线程在前进(个别线程可能一直被饿);**wait-free** 指每个线程都能在有限步内完成(强得多,实现极难)。工程上说"无锁"99% 指前者。

> **99% 的业务代码不该自己写无锁数据结构**——要用就用现成的(Boost.Lockfree / Intel TBB / 内核 RCU)。你的手写版本大概率既不对也不快。

## 八、锁的三大事故:死锁、活锁、优先级反转

### 8.1 死锁

```
T1: 持有 lock_A ──等──→ lock_B(被 T2 持有)
T2: 持有 lock_B ──等──→ lock_A(被 T1 持有)
                ↑ 环形等待,永远僵住
```

预防手段朴素但有效:**全局规定加锁顺序,所有线程都先 A 后 B**——顺序一致,环就形不成。排查:

```bash
gdb -p <pid>
(gdb) info threads
(gdb) thread apply all bt    # 看所有线程的栈,找互相等锁的环
```

Java 更省事:`jstack <pid>` **自动检测死锁**并打印 "Found one Java-level deadlock"。

### 8.2 活锁

```
T1 看到资源被占 → 礼貌让出;T2 同样让出
两人同时重试 → 又撞上 → 又同时让 → 无限循环
```

像走廊里两人互相让路结果一直挡着对方:都在动,都没进展。对策:**随机退避(指数退避)**,让的时间加随机量,总有一次错开。

### 8.3 优先级反转:1997 年,火星

```
低优先级 L 拿着锁
高优先级 H 要这把锁 → 等 L
中优先级 M(跟锁无关)→ 优先级比 L 高,抢占 L 的 CPU
结果:M 一直跑 → L 放不了锁 → H 跟着跑不了
高优先级的 H,被不相干的中优先级 M 间接按在地上
```

这不是教科书虚构——**NASA 火星探路者号(Mars Pathfinder,1997)在火星上真实触发了它**:气象线程(低)持有总线锁,通信线程(中)长时间运行,总线管理(高)等锁超时,看门狗以为系统挂了,整机反复重启。JPL 工程师远程打补丁,打开的正是:

```c
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
pthread_mutex_init(&lock, &attr);
```

**优先级继承(priority inheritance)**:H 等 L 的锁时,L 临时升到 H 的优先级,跑完放锁打回原形。另一方案是**优先级天花板(priority ceiling)**:锁定一个固定优先级,谁持锁谁升到那级。写实时系统(配合 16 篇)不开优先级继承,就是在重演火星事故。

## 九、锁的粒度与实战手感

### 9.1 大锁 vs 细粒度

```c
// 大锁:一把罩住所有数据
pthread_mutex_lock(&big_lock);
/* 改 N 个数据结构 */
pthread_mutex_unlock(&big_lock);

// 细粒度:哈希表每桶一把锁
struct hashtable {
    pthread_mutex_t bucket_lock[1024];
    bucket_t        buckets[1024];
};
```

大锁简单、绝不死锁,但全部串行;细粒度并发高,但锁多了顺序难管,死锁风险陡增。**正确路线:先用大锁写对,profile 找到真实竞争点,再只对热点细化**。过早细化是双输——复杂度先付了,收益还不一定在你细化的地方。这条路 Linux 内核自己走了十几年:早期一把 **BKL(Big Kernel Lock,大内核锁)** 罩住整个内核,到 2.6 才基本拆完——**世界上最好的一批系统程序员,拆一把大锁拆了 10+ 年**。你先从大锁开始不丢人。

### 9.2 三条手感

**锁住数据,不锁住代码**:

```c
// 错:不碰共享数据的 do_local_calc 也被锁着
void update() {
    pthread_mutex_lock(&lock);
    do_local_calc();
    update_shared_data();
    pthread_mutex_unlock(&lock);
}

// 对:只锁碰共享数据的那几行
void update() {
    do_local_calc();
    pthread_mutex_lock(&lock);
    update_shared_data();
    pthread_mutex_unlock(&lock);
}
```

**用 RAII / defer 防忘记 unlock**(尤其中途 return / 抛异常的路径):

```cpp
std::lock_guard<std::mutex> guard(mu);   // C++:析构自动 unlock
```

```go
mu.Lock()
defer mu.Unlock()                        // Go
```

**降低竞争的常用招**,从简单到复杂:**分桶**(每桶一锁)、**per-thread 数据**(TLS / percpu,各写各的最后汇总)、**批量操作**(攒一批再加一次锁)、**乐观锁**(版本号 + 重试)、**RCU / COW**。

### 9.3 怎么看锁竞争

```bash
perf record -g ./your_program && perf report   # 热点里出现 futex/spin 就是锁竞争
jstack <pid>                                   # Java:找 BLOCKED 状态线程
bpftrace -e 'tracepoint:syscalls:sys_enter_futex { @[comm] = count(); }'
                                               # futex 调用排行:谁在疯狂进内核等锁
perf c2c record / report                       # cache line 级竞争(含 false sharing)
```

诊断链:**futex syscall 多 = 锁竞争凶 = 要么减小临界区,要么拆锁,要么换数据结构**。

## 踩坑提醒

1. **大锁包整个函数**——不碰共享数据的代码也被串行化,多核性能归零
2. **多把锁不按统一顺序加**——死锁的标准配方,定死全局加锁顺序就能根除
3. **持锁时调阻塞 IO**——一次 50ms 的 DB 查询让所有等锁线程陪等 50ms
4. **自旋锁用在长临界区或用户态**——CPU 空转烧电,被抢占还可能锁死
5. **读写锁场景错配**——读写比例接近或临界区短时,比 mutex 还慢
6. **cond_wait 用 if 不用 while**——虚假唤醒一来,条件没成立就往下跑
7. **把 atomic 当免费午餐**——LOCK 指令加 cache 一致性流量,热点计数器要分片
8. **自己手写无锁数据结构**——ABA、内存回收、正确性三座大山,用现成库
9. **实时系统不开优先级继承**——重演火星探路者,高优先级被中优先级活活按住
10. **死锁了只会重启**——`gdb thread apply all bt` / `jstack` 两分钟定位环形等待
11. **不 profile 就拆锁**——细化了一把根本不热的锁,白付复杂度
12. **以为"加了锁就万事大吉"**——锁的可见性保证靠底下的内存屏障,裸写共享变量什么保证都没有(下一篇就讲)
