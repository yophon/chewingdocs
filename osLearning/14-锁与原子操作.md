# 锁与原子操作

锁是并发编程的"红绿灯"——没有它多线程必崩,但用错了又会让性能崩 10 倍 / 死锁 / 优先级反转 / cache 一致性灾难。新手最常见的错是**所有共享数据都加同一把大锁**(性能极差),或**追求"无锁"用一堆 atomic**(写出来的代码自己都看不懂)。这一篇讲清楚:**自旋锁 vs 互斥锁的本质区别、读写锁何时真有用、CAS 和 ABA、futex 让"无竞争锁"几乎免费**,以及"无锁数据结构"为什么是**博士论文常见、工业代码罕见**的东西。

> 一句话先记住:**锁的本质是"我占着,别人等"**。等的方式不同 → 不同种类的锁:**自旋锁原地空转**(适合临界区 < μs),**互斥锁睡眠 + 唤醒**(适合临界区 > μs)。**futex 让 pthread_mutex 在"无竞争"场景几乎免费**(纯用户态 atomic CAS,不进内核)。**atomic 不是免费的** —— LOCK 指令 + cache 一致性开销;**无锁数据结构看起来很美,但 ABA 问题 + 内存回收 + 性能边界让它在工业上很危险**。

## 一、为什么需要锁

```c
int counter = 0;

// 线程 1 和线程 2 同时:
counter++;
```

`counter++` 实际是三步:`load → add → store`。两个线程同时做:

```
T1 load 0           T2 load 0
T1 add 1 → 1        T2 add 1 → 1
T1 store 1          T2 store 1
最终 counter = 1   ← 应该是 2,丢了一次更新
```

**锁的作用**:让"读 - 改 - 写"成为原子操作。

## 二、自旋锁(Spinlock):空转等待

### 2.1 原理

```c
while (atomic_test_and_set(&lock));  // 反复 CAS 直到成功
// 临界区
lock = 0;                             // 释放
```

**等的时候 CPU 一直转,不让出**。

### 2.2 适用场景

- **临界区极短**(μs 级):睡眠唤醒比空转贵
- **多核并行**:有别的核在跑,自己等几个 cycle 就行
- **不能睡眠的环境**:中断处理函数 / 内核

### 2.3 不适用场景

- **临界区长**:浪费 CPU
- **单核**:你转着等,别人没机会跑(死锁)
- **锁竞争激烈**:几百个线程同时空转 → 缓存一致性灾难

### 2.4 用户态有自旋锁吗

```c
pthread_spin_lock(&lock);
```

**用户态自旋锁很少用**——用户态没有"绝对不能睡"的场景,而且单核或抢占式调度容易死锁。**99% 用 mutex**。

**内核里的 spinlock 才是大宗**——保护短临界区(链表插入、计数器)。

## 三、互斥锁(Mutex):睡眠等待

### 3.1 原理

```c
pthread_mutex_lock(&lock);
// 临界区
pthread_mutex_unlock(&lock);
```

**抢不到锁就睡 → 等持有者释放时被唤醒**。

```
T1 lock → 进入临界区
T2 lock → 没抢到 → 睡眠(放进等待队列)
T1 unlock → 唤醒等待队列里的一个
T2 醒过来 → 拿到锁 → 进入临界区
```

### 3.2 适用场景

- 临界区长(> μs)
- 大多数用户态场景
- 数量不限

### 3.3 性能

```
无竞争(直接拿到):    ~25 ns(纯用户态 atomic CAS)
有竞争(要进内核睡眠): ~1 μs(syscall)
唤醒:                  ~1-5 μs
```

**关键洞察**:**无竞争的 mutex 几乎免费**——这就是 futex 的魔力。

## 四、futex:用户态锁的核心机制

### 4.1 什么是 futex

**futex = Fast Userspace muTEX**,Linux 2.5 引入。

**思想**:

```
无竞争:        atomic CAS 改一个用户态变量,不进内核
有竞争:        才调 futex_wait syscall 睡眠
释放时:        atomic 改变量,有人在等才调 futex_wake
```

### 4.2 简化伪代码

```c
// 锁结构(用户态)
atomic_int state;   // 0 = unlocked, 1 = locked, 2 = locked + waiters

void lock() {
    if (CAS(&state, 0, 1)) return;       // 无竞争:25ns
    
    // 有竞争
    do {
        if (state == 2 || CAS(&state, 1, 2)) {
            futex_wait(&state, 2);        // syscall 睡眠
        }
    } while (!CAS(&state, 0, 2));
}

void unlock() {
    if (atomic_dec(&state) != 1) {       // state 从 2 → 1,有人在等
        state = 0;
        futex_wake(&state, 1);            // syscall 唤醒一个
    }
    // state 从 1 → 0,无人等,直接返回
}
```

### 4.3 futex 的胜利

**glibc 的 pthread_mutex 用 futex 实现**——所以**无竞争场景比早期(Linux 2.4 之前的)mutex 快 10-100 倍**。

**几乎所有现代锁原语都基于 futex**:Mutex、Condition Variable、Read-Write Lock、Semaphore、Java 的 synchronized / ReentrantLock。

## 五、读写锁(Reader-Writer Lock)

```c
pthread_rwlock_rdlock(&lock);   // 读锁,可以多个并发
// 读临界区
pthread_rwlock_unlock(&lock);

pthread_rwlock_wrlock(&lock);   // 写锁,独占
// 写临界区
pthread_rwlock_unlock(&lock);
```

### 5.1 适用场景

- **读远多于写**(95% 读 / 5% 写)
- **读临界区长**(几十 μs+)

### 5.2 不适用场景

- 读写比例接近 → 读写锁开销超过 mutex
- 临界区短 → 读写锁额外开销大

### 5.3 写者饥饿问题

```
读者 A 持有读锁
写者 B 等
读者 C 来 → 拿到读锁(读优先策略)
读者 D 来 → 拿到读锁
...
B 永远等不到
```

**对策**:**写优先策略**(`pthread_rwlockattr_setkind_np(PTHREAD_RWLOCK_PREFER_WRITER_NONRECURSIVE_NP)`)——但读者也可能饿。**两难,慎用**。

### 5.4 RCU:读写锁的"零开销"替代

**RCU(Read-Copy-Update)**:

```
读者:无锁,直接读(几乎 0 开销)
写者:复制一份,改完后用 atomic 替换指针,等所有读者完成再释放旧版本
```

**用途**:Linux 内核大量使用(路由表、文件系统),**用户态不太常见**。

**优点**:读极快,**完全无锁**。
**缺点**:复杂,只适合"指针式数据结构"。

## 六、条件变量(Condition Variable)

```c
pthread_mutex_t mu;
pthread_cond_t cv;
int ready = 0;

// 等待方
pthread_mutex_lock(&mu);
while (!ready)                       // 注意是 while 不是 if!
    pthread_cond_wait(&cv, &mu);     // 释放锁,睡眠;醒来重新拿锁
// 临界区
pthread_mutex_unlock(&mu);

// 通知方
pthread_mutex_lock(&mu);
ready = 1;
pthread_cond_signal(&cv);            // 唤醒一个
pthread_mutex_unlock(&mu);
```

### 6.1 为什么是 while 不是 if

**虚假唤醒(spurious wakeup)**——`pthread_cond_wait` 偶尔会"无故醒"。**必须用 while 循环检查条件**。

### 6.2 broadcast vs signal

```
pthread_cond_signal(&cv);      唤醒一个
pthread_cond_broadcast(&cv);   唤醒所有
```

**Java 的 wait/notify/notifyAll 是同样模式**。

## 七、原子操作(Atomic)

### 7.1 C11 atomic

```c
#include <stdatomic.h>

atomic_int counter;
atomic_fetch_add(&counter, 1);
atomic_compare_exchange_strong(&counter, &expected, new_val);
atomic_load(&counter);
atomic_store(&counter, 0);
```

### 7.2 内存序(memory order)

```c
atomic_fetch_add_explicit(&counter, 1, memory_order_relaxed);
//                                       ↑
//                                       内存序选择
```

**6 种内存序**:

```
memory_order_relaxed    最弱,只保证原子,不保证顺序
memory_order_consume    依赖排序(C++ 几乎不用)
memory_order_acquire    读屏障(看后面的写都能看到)
memory_order_release    写屏障(前面的写都对其他人可见)
memory_order_acq_rel    acquire + release
memory_order_seq_cst    最强,顺序一致(默认)
```

**99% 用 default(seq_cst)够了**——除非性能极致优化。

### 7.3 CAS:无锁的基础

```c
atomic_compare_exchange_strong(&p, &expected, new_val);
// 等价于:
//   if (p == expected) { p = new_val; return true; }
//   else { expected = p; return false; }
// (整个操作原子)
```

### 7.4 ABA 问题

```
T1 读 head = A
T2 pop A,push B,push C,然后 push 回 A
T1 CAS(head, A, ...)  → 成功!但实际链表已经变了
```

**虽然值还是 A,但语义已经不同** —— CAS 检测不出来。

**对策**:

- **版本号**(double-CAS,带 generation)
- **危险指针**(hazard pointer)
- **RCU 风格**

### 7.5 atomic 不是免费的

```
普通 add:        ~1 ns
atomic add:      ~10-20 ns(LOCK 指令 + cache invalidate)
mutex add(无竞争): ~30 ns
```

**atomic 比普通操作慢 10-20 倍**——多线程改同一个变量时,**cache 一致性开销才是大头**。

## 八、死锁与活锁

### 8.1 死锁

```
T1 持有 lock_A,等 lock_B
T2 持有 lock_B,等 lock_A
→ 互相等,死锁
```

**预防**:**所有线程按同一顺序加锁**:

```c
// 必须先 A 后 B
pthread_mutex_lock(&lock_A);
pthread_mutex_lock(&lock_B);
```

### 8.2 活锁(Livelock)

```
T1 看到资源被占,让出
T2 看到资源被占,让出
两人都让 → 又同时尝试 → 又都让 → 永远做不下去
```

**对策**:加随机退避(指数退避)。

### 8.3 排查死锁

```bash
gdb -p <pid>
(gdb) info threads
(gdb) thread apply all bt    # 看所有线程栈
# 找两个线程互相等对方持有的锁
```

**Java 用 jstack** —— 自动检测死锁并打印。

## 九、优先级反转

```
低优先级线程 L 持有锁
高优先级线程 H 等这个锁
中优先级线程 M(不需要锁)抢占 L
→ M 一直跑,L 跑不了 → H 跑不了
→ 高优先级的 H 被中优先级的 M 间接 block
```

**真实事故**:NASA 火星探路者(1997 年)就是这个 bug,系统反复重启。

**对策**:

- **优先级继承(Priority Inheritance)**:H 等 L 时,L 临时升级为 H 的优先级
- **优先级天花板(Priority Ceiling)**:锁有个固定优先级,持锁者升到那个

```c
pthread_mutexattr_setprotocol(&attr, PTHREAD_PRIO_INHERIT);
pthread_mutex_init(&lock, &attr);
```

## 十、锁的粒度:大锁 vs 细粒度

### 10.1 大锁(粗粒度)

```c
pthread_mutex_t big_lock;

pthread_mutex_lock(&big_lock);
// 改 N 个数据结构
pthread_mutex_unlock(&big_lock);
```

**优点**:简单,没有死锁风险。
**缺点**:**全部串行化,没有并发**。

### 10.2 细粒度

```c
struct hashtable {
    pthread_mutex_t bucket_lock[1024];   // 每个桶一把锁
    bucket_t buckets[1024];
};
```

**优点**:并发度高。
**缺点**:复杂,容易死锁。

### 10.3 经验

- **先用大锁写正确**,再考虑性能
- **profile 出竞争点再细化**(过早优化是万恶之源)
- **能用 atomic 代替锁就用**

**Linux 内核走过的路**:1.x 是"BKL(大内核锁)",2.6 才彻底拆掉,**花了 10+ 年**。

## 十一、无锁数据结构

### 11.1 思想

```
不用锁,用 CAS / atomic 实现并发安全
读写都不阻塞,理论上"永远 progress"
```

**例子**:无锁队列(Michael-Scott queue)、无锁栈、无锁哈希表。

### 11.2 看起来很美

- 无锁,所以"无死锁"
- 性能好(理论上)
- 学术论文高产

### 11.3 工业上很危险

- **ABA 问题**:必须额外处理(版本号 / hazard pointer)
- **内存回收**:释放节点时怎么知道没人在读?
- **正确性证明**:论文水平,普通工程师写不对
- **性能边界**:**写竞争激烈时不一定比锁快**(cache 一致性开销大)

> **99% 业务别自己写无锁数据结构** —— 用现成的(Boost / TBB / 内核 RCU)。

### 11.4 折中:lock-free vs wait-free

```
lock-free:  整个系统总有线程在前进(可能某个线程一直被 starve)
wait-free:  每个线程都能在有限步内完成(更强,实现极难)
```

**99% 工程上"无锁"指的是 lock-free**。

## 十二、锁的实战经验

### 12.1 锁住数据,不锁住代码

```c
// 错:锁的范围是函数
void update() {
    pthread_mutex_lock(&lock);
    do_something_complex();   // 整个函数都锁着,慢
    pthread_mutex_unlock(&lock);
}

// 对:锁的范围是关键数据
void update() {
    do_local_calc();          // 不需要锁
    pthread_mutex_lock(&lock);
    update_shared_data();     // 只锁这一段
    pthread_mutex_unlock(&lock);
}
```

### 12.2 RAII / defer 防忘记 unlock

```cpp
// C++
std::lock_guard<std::mutex> guard(mu);   // 析构自动 unlock
```

```go
// Go
mu.Lock()
defer mu.Unlock()
```

### 12.3 减少锁竞争的招

- **分桶**(每桶一锁)
- **per-thread 数据**(TLS / percpu)
- **批量操作**(攒一批再加锁)
- **乐观锁**(版本号 + retry)
- **RCU / COW**

## 十四、踩坑提醒

1. **大锁包整个函数** —— 性能崩
2. **多锁不按统一顺序** —— 死锁
3. **持锁时调阻塞 IO** —— 其他线程全等
4. **自旋锁用在长临界区** —— 浪费 CPU
5. **读写锁场景错** —— 读写比例接近时反而比 mutex 慢
6. **cond_wait 不用 while** —— 虚假唤醒导致 bug
7. **atomic 当免费用** —— 有 LOCK + cache 开销
8. **写无锁数据结构** —— 自己写几乎肯定有 bug
9. **不防优先级反转** —— 实时系统出大事
10. **死锁不会查** —— `gdb info threads` + bt
11. **不 profile 就优化** —— 优化到根本不是热点的地方
12. **以为"加锁就安全"** —— 锁不解决可见性,要内存屏障(15 篇)
