# 系统调用机制

你写了一行 `read(fd, buf, 1024)`,内核到底做了什么?——这是理解 OS 边界的关键一问。新手最大的误解是**把 syscall 当成普通函数调用**:都是"调一下、拿个返回值",能有什么区别?区别大了——函数调用是 CPU 跳个地址,syscall 是**一次硬件级的特权切换**,CPU 要换身份、换栈、换世界,代价是普通函数的 100-1000 倍。**系统调用是用户态和内核态之间唯一的合法通道**,理解了它,你就同时理解了"为什么 IO 慢"、"为什么 mmap 有时比 read 快"、"为什么 io_uring 是革命"——这三个问题其实是同一个问题。

> 一句话先记住:**syscall = 用户态向内核态发起的"中断式"请求**——CPU 切到 ring 0、保存所有寄存器、执行内核代码、再切回来。**普通函数调用 ~1ns,syscall ~100-1000ns,慢的是"切换",不是"执行"**。于是高性能编程的第一原则就一句话:**减少 syscall 数量**——缓冲、mmap、vDSO、io_uring,全是这个思想的四种武器。

---

## 一、为什么必须有 syscall

先想象没有这条边界的世界:任何程序都能直接操作硬件——

```
随便一个程序 malloc 整 16GB → 系统挂
随便一个程序读 /etc/shadow → 密码全泄露
随便一个程序循环关中断 → 整机假死
```

所以 OS 规定:**任何"碰硬件"或"碰内核数据结构"的操作,用户态一律不能直接做**:

```
直接读磁盘?         不能(要走驱动)
直接发网络包?       不能(要走协议栈)
分配物理内存?       不能(页表是内核管的)
创建进程?           不能(进程控制块是内核数据结构)
睡眠 1 秒?          不能(定时器在内核)
```

想做?可以,**提交申请**——这个申请就是 syscall,内核统一做权限检查和资源仲裁。**保护 + 仲裁**,这就是 syscall 存在的两个理由。听起来像官僚主义,但正是这层官僚主义让你的笔记本能同时跑 IDE 和 50 个浏览器 tab 而不互相搞死。

## 二、syscall 的硬件机制:CPU 怎么"换身份"

### 2.1 特权级:ring 0 和 ring 3

x86 有 4 个特权级(ring 0-3),Linux 只用两个:

```
Ring 0:内核态
  能执行所有指令(包括改页表、关中断)
  能访问所有内存

Ring 3:用户态
  禁止执行特权指令
  只能访问本进程的虚拟地址空间
  访问内核地址直接 segfault
```

特权级是**CPU 硬件强制的**,不是内核"自觉遵守"的——用户态执行特权指令,CPU 直接拒绝并抛异常。

### 2.2 syscall 指令:陷入的那一行

x86_64 用 `syscall` 指令发起申请。约定很简单:调用号放 rax,参数放 rdi / rsi / rdx...:

```text
; 用户态汇编
mov rax, 1              ; 系统调用号:1 = sys_write
mov rdi, 1              ; 参数 1:fd = stdout
mov rsi, msg            ; 参数 2:buffer 地址
mov rdx, 13             ; 参数 3:长度
syscall                 ; ←— 这一行触发陷入
; 返回后,rax = 写入字节数
```

`syscall` 这条指令落地的瞬间,CPU 做了一整套动作:

```
1. 从 ring 3 切到 ring 0
2. 保存当前 RIP / RSP 等寄存器
3. 加载内核栈指针(从 MSR 寄存器)
4. 跳转到内核 syscall 入口(entry_SYSCALL_64)
5. 内核根据 rax 找到对应的处理函数
6. 执行(sys_write)
7. 用 sysret 切回用户态
```

### 2.3 三代陷入方式

```
int 0x80     传统中断方式,慢       Linux 2.4 之前
sysenter     Pentium II+,Intel    Linux 2.5 引入
syscall      AMD K6+              Linux 2.6 引入,x86_64 标配
```

现代 64 位 Linux 全用 `syscall` 指令——比 `int 0x80` 快 3-5 倍,因为省掉了中断描述符表的查找。你偶尔还会在老教程里看到 `int 0x80`,知道它是历史就行。

## 三、glibc:你以为是内核,其实是用户态

新手最大的混淆来了:**你代码里写的 `read()` 根本不是 syscall,它是 glibc 里的一个普通函数**——它的内部才发起真正的 syscall:

```c
// 你写的:
read(fd, buf, 1024);

// glibc 实际做的(简化):
ssize_t read(int fd, void *buf, size_t count) {
    return syscall(SYS_read, fd, buf, count);
}
```

为什么要垫这一层皮?四个理由,每个都很实在:

- **跨架构**:同一个 `read()`,x86 和 ARM 底下用不同的陷入指令,glibc 帮你屏蔽
- **错误处理**:syscall 返回负数表示错误,glibc 把它转成你熟悉的 `errno`
- **缓冲**:`fread` / `printf` 在用户态攒数据,**减少 syscall 数量**(下面细讲)
- **兼容**:新内核加了系统调用,glibc 提供向后兼容的包装

那能不能绕过 glibc 直接调?能,但没必要:

```c
write(1, "hi", 2);                 // 用 glibc(推荐)
syscall(SYS_write, 1, "hi", 2);    // 直接调(性能差不多,可读性差)
```

**99% 的情况用 glibc;只有极致优化或绕 glibc bug 时才裸调 syscall**。

最要命的认知误区是把 glibc 当成 Linux 的一部分——**它只是个用户态库,可以整个换掉**:

- **musl**(Alpine Linux 用)——更小更快
- **uClibc**(嵌入式)
- **bionic**(Android)

**Go 静态编译甚至完全不依赖 libc**——自己直接发 syscall。这就是为什么 Go 二进制扔到任何 Linux 上都能跑,也是为什么 Alpine 镜像里某些 C 程序会莫名 segfault(musl 和 glibc 行为有差异)。

## 四、syscall 到底有多贵

### 4.1 数字摆出来

| 操作 | 耗时 |
| --- | --- |
| 普通函数调用 | ~1 ns |
| 间接函数调用(通过指针) | ~2 ns |
| **syscall(空调用)** | **~100 ns** |
| read 1 字节 | ~500 ns |
| 缺页中断 | ~1-10 μs |
| 进程上下文切换 | ~1-5 μs |

### 4.2 慢在哪:不是执行,是切换

一次空 syscall 什么正事都没干,为什么要 100ns?账单如下:

```
1. CPU mode 切换:几十个周期
2. 保存 / 恢复寄存器:十几个寄存器
3. TLB 部分刷新:虚拟地址翻译缓存失效
4. 缓存污染:内核代码挤掉用户代码的 L1/L2 缓存
5. 分支预测失败:CPU 流水线被打断
```

注意第 4、5 条——**syscall 的伤害不止发生在调用那一刻,返回之后你的代码还要继续付"缓存被污染"的账**。这就是为什么 benchmark 测出来 100ns,真实程序里感受到的损失往往更大。

### 4.3 一次磁盘 read 的开销分解

```c
read(fd, buf, 4096);
```

```
syscall 切换:        100 ns
内核 sys_read:       100 ns
查页缓存(命中):     200 ns
拷贝到用户 buf:      400 ns
─────────────────────
总计(命中缓存):     ~1 μs

如果未命中页缓存:
+ 磁盘 IO:10-100 μs(NVMe SSD)或 5-10 ms(HDD)
```

看出门道了吗:**数据在哪决定了谁是大头**。命中页缓存时,syscall 开销占一半;打到磁盘时,syscall 开销可以忽略。所以"减少 syscall"这个优化,**对内存型负载收益巨大,对磁盘型负载收益有限**——动手前先想清楚瓶颈在哪。

### 4.4 经典翻车现场:1 字节读

```c
// 程序 A:每次读 1 字节
char c;
while (read(fd, &c, 1) == 1) {
    process(c);
}
// 100MB 文件 → 1 亿次 syscall → 几十秒

// 程序 B:每次读 4096 字节
char buf[4096];
ssize_t n;
while ((n = read(fd, buf, 4096)) > 0) {
    for (int i = 0; i < n; i++) process(buf[i]);
}
// 100MB 文件 → 25600 次 syscall → 几百毫秒
```

**性能差 100 倍——不是磁盘慢,是 syscall 次数差了 4096 倍**。经验值:buffer 至少 4KB(一页),理想 16KB-64KB(平衡延迟和吞吐)。

## 五、减少 syscall 的四种武器

知道了贵,接下来就是省。整个 Linux 性能优化史里有四件武器,一件比一件激进。

### 5.1 缓冲 IO:攒一批再交

`fwrite` / Java 的 `BufferedWriter` / Go 的 `bufio.Writer`,全是同一个思路:

```c
// 不缓冲:1024 次 syscall
for (i = 0; i < 1024; i++)
    write(fd, &c, 1);

// 缓冲:1 次 syscall
fwrite(buf, 1, 1024, fp);   // glibc 攒满 buffer 才 flush
```

这是最便宜、最普适的武器,**所有语言的标准库都帮你内置了**——前提是你别绕过它直接裸 write。

### 5.2 mmap:把文件变成内存,read 直接消失

```c
char *p = mmap(NULL, size, PROT_READ, MAP_PRIVATE, fd, 0);
// 之后访问 p[i] 就像读数组,一次 syscall 都不需要
```

第一次访问某页触发缺页中断把数据加载进来,**后续访问纯内存读,零 syscall**。代价是缺页中断本身也不便宜——所以小文件、一次性顺序大读,mmap 不一定赢。

### 5.3 vDSO:内核开的"性能后门"

`gettimeofday()` / `clock_gettime()` 这种调用很特殊:**只是读个时间,根本不需要内核做任何特权操作**。于是内核搞了 vDSO(virtual Dynamic Shared Object)——把一小段内核代码和时间数据**直接映射进每个进程的地址空间**:

```
gettimeofday()
↓
glibc 发现这是 vDSO 支持的调用
↓
直接读共享内存,~10 ns(普通 syscall 要 ~100 ns)
```

看起来是 syscall,实际上**根本没进内核**。这件事的设计极其优雅:高频、只读、无副作用的调用,凭什么要付特权切换的钱?

### 5.4 io_uring:N 次 IO,1 次 syscall

Linux 5.1+ 的革命——内核和用户态**共享一对环形队列**:

```
用户态:把 N 个 IO 请求写入 SQ(Submit Queue)
↓ 一次 syscall(io_uring_enter)
内核:批量处理 N 个请求
↓ 完成后写入 CQ(Complete Queue)
用户态:轮询 CQ 取结果
```

**N 个 IO 操作 = 1 次 syscall**,极端模式下(SQPOLL)甚至 0 次。这是从"省着用"到"改规则"的跃迁。详见 22 篇 io_uring。

## 六、syscall 全家福:400 个里你只需要认识几十个

Linux 大约 400 个 syscall,按用途分组,常用的就这些:

```
文件 / IO:     open / close / read / write / lseek
               stat / fstat / access
               mmap / munmap / fsync / fdatasync

进程管理:      fork / vfork / clone
               execve / exit / wait / waitpid
               getpid / getppid / kill / signal

内存管理:      brk / sbrk(扩展堆)
               mmap / munmap / mprotect
               madvise(内存使用建议)

IO 多路复用:   select / poll / epoll
               io_uring(新)

网络:          socket / bind / listen / accept / connect
               send / recv / sendto / recvfrom / shutdown

时间 / 同步:   gettimeofday / clock_gettime / nanosleep
               futex(用户态锁的内核支持)
```

所有 syscall 都在 man 手册第 2 章——`man 2 read` 就是权威文档。

## 七、strace:看你的程序到底在调什么

排查行为 / 性能问题的第一招——把程序的 syscall 全录下来:

```bash
strace -c ls

% time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ----------------
 28.6    0.000164          82         2           getdents64
 14.0    0.000080           5        16           write
 ...
```

`strace -e trace=read,write` 可以只盯特定 syscall。典型用途:

- 程序卡住了 → 看它阻塞在哪个 syscall 上
- 性能差 → 看哪个 syscall 调用次数爆炸
- 行为诡异 → 看它实际打开了什么文件、连了什么地址

**一个警告:别在生产环境随手 strace**——它基于 ptrace,会让目标进程慢 10 倍。生产环境用 perf / eBPF(详见 26 篇性能工具)。

## 八、终极形态:连内核都绕过去

当 syscall 优化到头还不够快,还有最后一招——**完全不用内核**。

**DPDK(网络)**:把网卡驱动搬到用户态,绕过整个 Linux 网络栈:

```
传统:  网卡 → 中断 → 内核网络栈 → socket buffer → 拷贝到用户态 → 应用
DPDK:  网卡 → 用户态驱动(轮询)→ 应用

代价:CPU 核永远 100% 在轮询,功耗拉满
收益:延迟从 50μs 降到 5μs
```

用在高频交易、5G 基站、超大流量网关。**SPDK(存储)** 是同样思路的 NVMe 版,绕过内核 IO 栈。

> 这是"反直觉的优化":当官僚机构(内核)成为瓶颈,就绕开官僚机构。**99% 的业务用不到,但你要知道天花板之上还有天花板**。

## 九、Go 的 syscall 处理:一个聪明的设计

Go 不依赖 glibc,自己直接发 syscall:

```go
// Go 运行时(简化)
func read(fd int, p []byte) (n int, err error) {
    n = syscall.Syscall(SYS_read, uintptr(fd), ...)
}
```

有意思的在后面:syscall 可能阻塞(比如读一个没数据的 socket),阻塞的是**整条 OS 线程**。Go 的解法:调度器(GMP)发现 goroutine 进入阻塞 syscall 时,**把这条 OS 线程从 P(处理器)上解绑**——P 转身带别的线程跑其他 goroutine。**这就是 Go "百万协程"扛得住的关键之一:syscall 阻塞不会卡死整个调度器**。详见 17 篇协程。

## 十、踩坑提醒

1. **以为 syscall 是普通函数**——它是硬件级特权切换,慢 1000 倍
2. **每次 read 1 字节**——syscall 次数爆炸,性能差 100 倍,buffer 至少 4KB
3. **热路径上频繁 gettimeofday**——有 vDSO 也不是零开销,能缓存就缓存
4. **不用 buffered IO**——`fprintf` 是缓冲的,裸 `write` 不是,别混着用
5. **以为 mmap 永远比 read 快**——小文件 / 顺序大读不一定,mmap 有缺页中断成本
6. **生产环境跑 strace**——目标进程慢 10 倍,生产用 perf / eBPF
7. **read 返回 -1 不看 errno**——syscall 的错误码全在 errno 里,不看永远排不出错
8. **以为 Go / Java 不走 syscall**——所有 IO 最后都是 syscall,语言只是包了几层
9. **不知道 glibc / musl 的差异**——Alpine 镜像里 segfault 时一头雾水
10. **以为 DPDK 谁都该用**——99% 的业务用不到,引入的复杂度远大于收益
