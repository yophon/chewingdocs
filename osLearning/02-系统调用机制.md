# 系统调用机制

「我写了 `read(fd, buf, 1024)` 一行代码,内核到底做了什么」——这是理解 OS 边界的关键问题。**系统调用是用户态和内核态之间唯一的合法通道**,理解它你就理解了"为什么 IO 慢"、"为什么 mmap 比 read 快"、"为什么 io_uring 是革命"。新手最大的误解是**把 syscall 当成普通函数**——实际上它是**特殊的 CPU 指令(int 0x80 / sysenter / syscall)**,会触发硬件级的特权切换,代价是普通函数调用的 1000 倍。

> 一句话先记住:**syscall = 用户态向内核态发起的"中断式"请求**——CPU 切到 ring 0,保存所有寄存器,切换页表(部分),执行内核代码,完了再切回来。**普通函数调用 ~1ns,syscall ~100-1000ns**——慢的是"切换",不是"执行"。**减少 syscall 数量 = 高性能编程的第一原则**:批量、零拷贝、io_uring 全是这个思想。

---

## 一、为什么需要 syscall

### 1.1 用户态不能直接做的事

```
直接读磁盘?         不能 (要走驱动)
直接发网络包?       不能 (要走协议栈)
分配内存?           不能 (页表是内核管的)
创建进程?           不能 (PCB 是内核数据结构)
睡眠 1 秒?          不能 (定时器在内核)
```

**任何"碰硬件"或"碰内核数据结构"的操作,都必须通过 syscall**。

### 1.2 为什么这么严格

```
没有这个限制:
  随便一个程序 malloc 整 16GB → 系统挂
  随便一个程序读 /etc/shadow → 安全崩
  随便一个程序循环关中断 → 整机假死

有了 syscall + 权限检查:
  内核统一仲裁,谁能干啥,谁不能
```

**保护 + 仲裁** —— 这就是 syscall 存在的两个理由。

## 二、syscall 的硬件机制

### 2.1 CPU 的特权级别

x86 有 4 个特权级(ring 0-3),Linux 只用两个:

```
Ring 0:内核态
  能执行所有指令(包括改页表、关中断)
  能访问所有内存
  
Ring 3:用户态
  禁止执行特权指令
  只能访问本进程的虚拟地址空间
  访问内核地址直接 segfault
```

### 2.2 syscall 指令

x86_64 用 `syscall` 指令(老的用 `int 0x80`,慢):

```text
; 用户态汇编
mov rax, 1              ; 系统调用号:1 = sys_write
mov rdi, 1              ; 参数 1:fd = stdout
mov rsi, msg            ; 参数 2:buffer 地址
mov rdx, 13             ; 参数 3:长度
syscall                 ; ←— 这一行触发陷入
; 返回后,rax = 写入字节数
```

**`syscall` 指令做了什么**:

```
1. CPU 从 ring 3 切到 ring 0
2. 保存当前 RIP / RSP 等寄存器
3. 加载内核栈指针 (从 MSR)
4. 跳转到内核 syscall 入口 (entry_SYSCALL_64)
5. 内核根据 rax 找到对应的处理函数
6. 执行 (sys_write)
7. 用 sysret 切回用户态
```

### 2.3 syscall vs int 0x80 vs sysenter

```
int 0x80     传统中断方式,慢       Linux 2.4 之前
sysenter     Pentium II+,Intel    Linux 2.5 引入
syscall      AMD K6+,AMD          Linux 2.6 引入
             x86_64 标配
```

> 现代 64 位 Linux 全用 `syscall` 指令——比 `int 0x80` 快 3-5 倍(少了中断描述符表查找)。

## 三、glibc:你以为是内核,其实是用户态

新手最大的混淆:**`read()` 不是 syscall,是 glibc 的函数**——它内部才调 syscall。

```c
// 你写的:
read(fd, buf, 1024);

// glibc 实际做的(简化):
ssize_t read(int fd, void *buf, size_t count) {
    return syscall(SYS_read, fd, buf, count);
}
```

### 3.1 为什么有这一层包装

- **跨架构**:同一个 `read()` 函数,x86 和 ARM 用不同的 syscall 指令
- **错误处理**:syscall 返回负数表示错误,glibc 把它转成 `errno`
- **缓冲**:`fread` / `printf` 在用户态做缓冲,**减少 syscall 数量**
- **跨内核版本**:新内核加了系统调用,glibc 提供向后兼容

### 3.2 直接调 syscall vs glibc

```c
// 用 glibc(推荐)
write(1, "hi", 2);

// 直接调 syscall(没必要)
syscall(SYS_write, 1, "hi", 2);

// 性能差不多,但 glibc 更可读、可移植
```

> **99% 情况用 glibc**;**只有性能极致优化或绕过 glibc bug 时才直接 syscall**。

### 3.3 glibc 不是 OS

很多人把 glibc 当成 Linux 的一部分——**它只是个用户态库**。可以替换成:

- **musl**(Alpine Linux 用)——更小更快
- **uClibc**(嵌入式)
- **bionic**(Android)

**Go 静态编译甚至不依赖 libc**——直接走 syscall。

## 四、syscall 的真实开销

### 4.1 数据

| 操作 | 耗时 |
| --- | --- |
| 普通函数调用 | ~1 ns |
| 间接函数调用(通过指针) | ~2 ns |
| **syscall(空)** | **~100 ns** |
| read 1 字节 | ~500 ns |
| 缺页中断 | ~1-10 μs |
| 进程上下文切换 | ~1-5 μs |

**syscall 比函数调用慢 100-1000 倍**——这是用户态最大的性能瓶颈之一。

### 4.2 慢在哪

```
1. CPU mode 切换:几十个周期
2. 保存 / 恢复寄存器:十几个寄存器
3. TLB 部分刷新:虚拟地址翻译缓存失效
4. 缓存污染:内核代码挤出用户代码的 L1/L2 缓存
5. 分支预测失败:CPU 流水线打断
```

**核心观察**:**syscall 本身代码不慢,慢的是切换**。

### 4.3 一次磁盘 read 的开销分解

```c
read(fd, buf, 4096);
```

```
syscall 切换:        100 ns
内核 sys_read:       100 ns
查页缓存(命中):     200 ns
拷贝到用户 buf:      400 ns
─────────────────────
总计(命中缓存):     ~1 μs

如果未命中页缓存:
+ 触发磁盘 IO:      10-100 μs (NVMe SSD) 或 5-10 ms (HDD)

如果是 SSD,IO 时间  ≫ syscall 时间
如果是内存命中,syscall 时间占大头
```

> **小请求多 syscall 是性能杀手** —— `for (i=0; i<1024; i++) write(fd, &buf[i], 1)` 比 `write(fd, buf, 1024)` 慢 1000 倍。

## 五、如何减少 syscall:四种武器

### 5.1 缓冲 IO

`fwrite` / `BufferedWriter` / `bufio.Writer` 全是这个思路:

```c
// 不缓冲:1024 次 syscall
for (i = 0; i < 1024; i++)
    write(fd, &c, 1);

// 缓冲:1 次 syscall
fwrite(buf, 1, 1024, fp);  // glibc 攒满 buffer 才 flush
```

### 5.2 mmap:把文件映射成内存

```c
char *p = mmap(NULL, size, PROT_READ, MAP_PRIVATE, fd, 0);
// 之后访问 p[i] 就像读内存,不再 syscall
```

**第一次访问触发缺页中断 → 加载页**;**后续访问纯内存读,无 syscall**。

### 5.3 vDSO:把 syscall 偷偷搬到用户态

`gettimeofday()` / `clock_gettime()` 这种"只读时间"的调用,**根本不需要进内核**——内核维护的时间数据在固定共享内存里,用户态直接读就行。

**vDSO**(virtual Dynamic Shared Object)= 内核映射到每个进程地址空间的一段代码。看起来是 syscall,实际上**没有切换内核**:

```
gettimeofday()
↓
glibc 看到是 vDSO 支持的调用
↓
直接读内存,~10 ns(普通 syscall ~100 ns)
```

> 这是 Linux 内核的"性能后门"——为高频但只读的 syscall 开了快速通道。

### 5.4 io_uring:批量提交 syscall

Linux 5.1+ 的革命——**内核和用户态共享一对环形队列**:

```
用户态:把 N 个 IO 请求写入 SQ (Submit Queue)
↓ 一次 syscall(io_uring_enter)
内核:批量处理 N 个请求
↓ 完成后写入 CQ (Complete Queue)
用户态:轮询 CQ 取结果
```

**N 个 IO 操作 = 1 次 syscall**——彻底打破 syscall 数量瓶颈。详见 22 篇 io_uring。

## 七、查看你的程序在调什么 syscall:strace

调试性能 / 行为问题的第一招:

```bash
strace -c ls

% time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ----------------
 28.6    0.000164          82         2           getdents64
 14.0    0.000080           5        16           write
 ...
```

`strace -e trace=read,write` 只看特定 syscall。

**典型用途**:

- 程序卡住了,看它在哪个 syscall 阻塞
- 性能差,看哪个 syscall 调用次数最多
- 程序行为奇怪,看它实际在干嘛(打开了什么文件)

> 详见 26 篇性能工具。

## 八、syscall 的"昂贵"案例:文件读慢的真凶

```c
// 程序 A:每次读 1 字节
char c;
while (read(fd, &c, 1) == 1) {
    process(c);
}
// 100MB 文件 → 1 亿次 syscall → 几十秒

// 程序 B:每次读 4096 字节
char buf[4096];
ssize_t n;
while ((n = read(fd, buf, 4096)) > 0) {
    for (int i = 0; i < n; i++) process(buf[i]);
}
// 100MB 文件 → 25600 次 syscall → 几百毫秒
```

**性能差 100 倍**——不是磁盘慢,是 syscall 调用次数差 4096 倍。

**经验值**:**buffer 至少 4KB**(一页),理想 16KB-64KB(平衡延迟和吞吐)。

## 九、用户态绕过内核:DPDK / SPDK

终极性能优化——**完全不用 syscall**:

### 9.1 DPDK(网络)

把网卡驱动搬到用户态,**绕过整个 Linux 网络栈**:

```
传统:  网卡 → 中断 → 内核网络栈 → socket buffer → user copy → 应用
DPDK:  网卡 → 用户态驱动(轮询)→ 应用

代价:CPU 永远在轮询,功耗高,但延迟从 50μs 降到 5μs
```

**用在**:高频交易、5G 基站、超大流量网关。

### 9.2 SPDK(存储)

NVMe 设备的用户态驱动,**绕过内核 IO 栈**——同样思路。

> 这是"反直觉的优化":**当 syscall 成为瓶颈,就把内核绕过去**。**99% 业务用不到,但要知道存在**。

## 十、Go 的 syscall 处理:有意思的设计

Go 不依赖 glibc,**自己直接调 syscall**:

```go
// Go 源码
func read(fd int, p []byte) (n int, err error) {
    n = syscall.Syscall(SYS_read, uintptr(fd), ...)
}
```

**特殊之处**:Go 用户态调度器(GMP)发现 goroutine 调用阻塞 syscall 时,**会把对应的 OS 线程从 P 上解绑**——这样其他 goroutine 仍能在其他 P 上跑。

**这就是 Go "百万协程"扛得住的关键**——syscall 不会卡死整个调度器。详见 17 篇协程。

## 十一、踩坑提醒

1. **以为 syscall 是普通函数**——慢 1000 倍
2. **每次 read 1 字节**——syscall 数量爆炸,性能崩
3. **频繁 gettimeofday**——别在热路径调,用 vDSO 也是有开销的
4. **不开 buffered IO**——`fprintf` 是缓冲的,直接 `write` 不是
5. **以为 mmap 永远比 read 快**——小文件 / 顺序大读不一定,mmap 有缺页中断成本
6. **strace 在生产环境跑**——会让目标进程慢 10 倍
7. **不知道 errno 是 syscall 错误码**——`read` 返回 -1 不看 errno,排不出错
8. **以为 Go / Java 不用 syscall**——所有 IO 最后都是 syscall(除非用 io_uring)
9. **glibc 和 musl 行为差异不知道**——Alpine 镜像 segfault 时一头雾水
10. **以为 DPDK 谁都该用**——99% 业务用不到,引入复杂度大
