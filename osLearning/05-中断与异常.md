# 中断与异常

01 篇说过一个反直觉的事实:**OS 绝大多数时候在睡觉,CPU 在跑你的程序**。那网卡来了包、定时器到点了、你的程序踩了野指针,OS 是怎么知道的?它总不能开个死循环不停问"网卡有数据没"——那叫轮询,等于雇人 24 小时盯着门铃看它响没响,纯烧 CPU。真正的答案是**中断:让事件主动来敲 CPU 的门**。中断是 OS 进入工作的**唯一入口**——没有中断,内核就是硬盘上一堆躺平的代码。理解了中断,你就同时理解了:OS 怎么和硬件打交道、高并发网络服务为什么必须把中断摊到多核、`top` 里那个神秘的 `%si` 高了意味着什么。

> 一句话先记住:**中断 = 硬件 / 软件主动叫醒 CPU,让它放下手头的事先处理我**。来源就三种:**外部硬件(IRQ)、CPU 异常(除零 / 缺页 / segfault)、软件主动(syscall)**,统称陷入(trap)。中断处理的铁律是**必须快**——上半部 / 下半部(top half / bottom half)就是"立即处理"和"延后处理"的分工。**网卡 PPS 一高,单核中断扛不住,必须 RSS / RPS 分散到多核**。

---

## 一、CPU 从不主动"分心"

先建一个干净的模型。CPU 是一台极其单纯的机器:沿着 PC 指针一条一条往下执行,**它自己永远不会东张西望**,能让它跳走的只有外部事件:

```
进程的代码(用户态 / 内核态)
   ↓ 中断来了
中断处理程序
   ↓ 处理完
回到原来的代码继续跑
```

所以"OS 在后台守护着系统"这个画面是错的——**OS 只在中断来的那一瞬间存在**,处理完就把 CPU 还给你的程序。

---

## 二、三种陷入:同一扇门,三种敲法

```
1. 中断(Interrupt)
   外部硬件主动通知:网卡 / 磁盘 / 键盘 / 定时器
   异步——和当前正在跑的指令毫无关系,随时砸过来

2. 异常(Exception)
   CPU 执行指令时出错或需要特殊处理:
   除零 / 非法指令 / 缺页 / 访问越界
   同步——由当前这条指令直接引起

3. 系统调用(Syscall)
   用户程序主动陷入内核(int 0x80 / sysenter / syscall 指令)
   同步——你自己叫的
```

区分的关键就一个词:**异步还是同步**。硬件中断是天外飞仙,跟你的代码无关;异常和 syscall 都是当前指令"自己"触发的。但三者的硬件机制几乎一样——切到内核态、查表找处理函数、执行、切回来。**Linux 内部统一叫 trap:一套门禁系统,三种访客**。syscall 就是三兄弟里"主动上门"的那个,`int 0x80` / `sysenter` / `syscall` 三代指令的演化 02 篇讲过,不重复。

## 三、硬件中断:从门铃响到开门

### 3.1 IRQ:每个设备一根铃线

每个外设接到中断控制器(APIC)上,用 IRQ 号区分谁在按铃:

```
定时器:      IRQ 0(每秒几百次,后面讲)
键盘:        IRQ 1
鼠标:        IRQ 12
网卡 0:      IRQ 32
网卡 1:      IRQ 33
```

想看你的机器上谁在按铃、按了多少次:

```bash
cat /proc/interrupts
#            CPU0       CPU1       CPU2       CPU3
#  0:        20         0          0          0   IO-APIC   2-edge      timer
#  1:        9          0          0          0   IO-APIC   1-edge      i8042
# 32:        12345      6789       1234       567 IO-APIC  32-fasteoi  eth0
```

每行一个 IRQ,**每列是这个核处理了多少次**——一眼看出哪个核被中断打爆了。这张表排查网络性能时还会回来看。

### 3.2 铃响之后,CPU 的标准动作

```
1. CPU 检测到 IRQ 信号
2. 执行完当前这条指令(CPU 在指令与指令之间检查中断)
3. 保存当前状态(寄存器、PC)
4. 切到内核态
5. 查中断描述符表(IDT),找到这个 IRQ 的处理函数
6. 跳转执行
7. 处理完,iret 指令恢复状态,回到原代码
```

整个流程几百纳秒——对硬件是闪电,对你的应用代码已经不便宜了。

### 3.3 对你的程序完全透明

```
用户程序:    a = b + c;   // 这条执行完
                ↓ 中断插进来
内核:        从网卡读包、放进队列
                ↓ iret
用户程序:    d = a + 1;   // 接着跑,毫无察觉
```

你的程序**永远不知道自己被打断过**——寄存器原样恢复,接着跑。但"无感"不等于"免费":被打断期间时间片在流逝、cache 被内核代码污染。**p99 延迟里那些解释不了的小毛刺,有一部分就是这么来的**。

## 四、上半部 vs 下半部:中断处理的分工哲学

### 4.1 一个两难

中断处理有条铁律:**必须快(微秒级)**——处理期间同类中断被屏蔽 / 排队,慢了就丢事件;这个核也干不了任何别的事,高频中断(网卡每秒几十万次)能直接把核吃光。

可是有些活就是重——一个网络包进来,要走完 TCP 协议栈、找到对应 socket、唤醒等待的进程,怎么可能微秒内做完?Linux 的答案是经典的分工:**门口只签收,包裹回屋慢慢拆**。

```
上半部(top half / hardirq):
  中断真正触发时立即执行
  只做"必须马上做"的最小动作——比如把网卡数据搬进内核 buffer
  不能睡眠、不能阻塞、越快越好

下半部(bottom half / softirq / tasklet / workqueue):
  上半部登记一下就走,稍后在合适的时机执行
  干重活:解析包、走 TCP 协议栈、唤醒 socket 上的等待者
```

### 4.2 下半部的三种实现

**softirq(软中断)**——最快的一档。类型是内核编译时写死的,固定几种,不能动态添加:

```
NET_TX / NET_RX:     网络收发(最大头)
BLOCK:               块设备 IO
TIMER / HRTIMER:     定时器 / 高精度定时器
TASKLET:             tasklet 调度
SCHED:               调度
RCU:                 RCU 回调
```

`top` 里的 `%si`(software interrupts)统计的就是它——**`%si` 高 = 网络或块 IO 的下半部在吃 CPU**。

**tasklet**——基于 softirq 实现,但允许驱动动态创建;同一个 tasklet 不会并发跑在多个 CPU 上(简化了驱动作者的并发心智)。

**workqueue(工作队列)**——直接扔给内核线程(就是 `ps` 里那些 `kworker`)异步执行。跑在线程上下文里,所以**可以睡眠**——三种机制里唯一能睡的,适合最重最慢的活。

| 机制 | 速度 | 能睡眠吗 | 适合 |
| --- | --- | --- | --- |
| softirq | 最快 | 不能 | 网络 / 块 IO 等高频核心路径 |
| tasklet | 快 | 不能 | 普通驱动的延后处理 |
| workqueue | 慢 | **能** | 重活、要拿锁要等 IO 的活 |

## 五、定时器中断:OS 的心跳

有一类中断值得单拎出来:**定时器**。它是 OS 的心跳——没有它,内核连"时间流逝"都感知不到,更别提"这个进程跑满时间片了,该换人了"。

### 5.1 HZ:每秒心跳几次

```
HZ = 100    每秒 100 次     早期 / 低功耗
HZ = 250    每秒 250 次     桌面默认
HZ = 1000   每秒 1000 次    服务器,调度响应快但中断开销大
```

```bash
grep ^CONFIG_HZ /boot/config-$(uname -r)
# CONFIG_HZ=250
```

每次心跳都进内核走一趟调度器,决定要不要切换进程——**时间片轮转的物质基础就是这个中断**,详见 16 篇调度器。

### 5.2 tickless:不用的时候别跳

每秒固定中断几百次,CPU 空闲时也跳?纯浪费电。现代内核(3.10+)支持动态 tick——**空闲时停掉定时器中断**,有事再唤醒。更极端的 `CONFIG_NO_HZ_FULL=y` 连用户态运行期间都不发 tick,留给 HPC / 实时场景:整颗核让给计算,一次都别打断。

## 六、网卡中断与 PPS 瓶颈:中断知识的实战主场

这一节是本篇最值钱的部分——前面所有概念在这里汇成一个真实的生产问题。

### 6.1 事故现场:单核被中断打爆

```
网卡进来 100 万 PPS(Packet Per Second)
→ 朴素实现:100 万次中断 / 秒,默认全打到 CPU0
→ CPU0 100% 耗在中断处理上,业务一行代码都跑不动
→ 其他 15 个核在围观
```

症状非常有迷惑性:**整机 CPU 利用率不高,但丢包、p99 飞天**。不看 `/proc/interrupts`,你永远想不到是一个核被中断单点打爆了。

### 6.2 RSS:网卡硬件分流

**RSS(Receive Side Scaling)**= 现代网卡的硬件能力:按包的哈希(源/目的 IP + Port)分进多个队列,**每个队列一个独立 IRQ,绑到不同核**:

```bash
ethtool -L eth0 combined 8     # 开 8 个队列
cat /proc/interrupts | grep eth0
# 32: ... → CPU0    队列 0
# 33: ... → CPU1    队列 1
```

100 万 PPS 摊到 8 个核,每核 12.5 万——扛得住了。注意哈希按流(同一连接走同一队列),**单条大流量连接还是只能压一个核**,这是 RSS 的固有边界。

### 6.3 RPS:没硬件队列就软件分

虚拟机 / 老网卡只有一个队列怎么办?**RPS(Receive Packet Steering)**在内核里做软件版的同样事情——上半部还在一个核,但把包转给其他核的 softirq 处理:

```bash
echo ff > /sys/class/net/eth0/queues/rx-0/rps_cpus
# bitmap,ff = 8 个核都参与
```

### 6.4 NAPI:高负载时干脆别用中断

中断适合"事件稀疏"的场景;包多到一定程度,每包一次中断就是中断风暴。NAPI 的策略是**两头通吃**:

```
低 PPS:纯中断模式(省 CPU,延迟低)
高 PPS:第一个中断后关掉中断改轮询,一批一批收,收完再开中断
```

这是现代网卡驱动的标配——**低负载靠中断的省,高负载靠轮询的稳**。还记得 02 篇的 DPDK 吗?它就是把"高负载轮询"做到极致:永远轮询,彻底告别中断。

## 七、异常:CPU 自己执行出的"事件"

硬件中断来自外部,异常来自**当前指令本身**。按后果分三类:

**Fault(故障)——可恢复,处理完重新执行那条指令**

```
缺页(Page Fault):访问的页不在物理内存
→ 内核把页准备好 → 回去重新执行同一条指令 → 这次成功
```

**Trap(陷阱)——故意的,处理完执行下一条**

```
断点(int 3):debugger 的实现基础
syscall 指令:主动陷入(对,syscall 在分类上就是个 trap)
```

**Abort(中止)——没救了**

```
机器检查异常(MCE):硬件故障
双重故障(Double Fault):处理异常时又异常,通常是内核 bug
→ 一般直接 kernel panic
```

### 7.1 缺页:最高频、也最被误解的异常

新手听到"缺页中断"以为是出错了。**错——缺页是机制,不是事故**:

```c
char *p = mmap(...);    // mmap 只是登记了虚拟地址,一个物理页都没给
p[0] = 1;               // 第一次访问 → MMU 查不到映射 → 缺页异常
                        // 内核:分配物理页、建立映射 → 重新执行这条指令
```

Linux 的内存分配整体是"先记账、用时再给"的懒人哲学,**缺页就是"用时再给"的触发器**——它每秒发生成千上万次,完全正常。什么时候它变成性能问题(主缺页打到磁盘)、怎么区分类型,详见 06 篇虚拟内存。

### 7.2 段错误:缺页处理的"否决分支"

```c
int *p = NULL;
*p = 1;     // 访问地址 0 → 同样触发缺页异常
            // 但内核一查:这个地址根本没有合法映射
            // → 不是"用时再给",是"你越界了" → 发 SIGSEGV
            // → 进程没装 handler → 默认 dump core 退出
```

有意思的地方:**SIGSEGV 和正常缺页走的是同一个异常入口**,区别只在内核查映射的那一步——查到了就补页,查不到就毙了你。信号机制详见 24 篇。

## 八、CPU 亲和性:把进程和中断钉在核上

懂了中断会抢核、换核会丢 cache(04 篇),自然推出一个工程手段:**绑核(CPU Affinity)**。

```bash
# 绑进程
taskset -c 0,1 ./my_program          # 只许在 CPU 0 / 1 上跑

# 绑中断
echo 4 > /proc/irq/32/smp_affinity   # bitmap,4 = CPU 2
```

```c
// 代码里绑
cpu_set_t set;
CPU_ZERO(&set);
CPU_SET(0, &set);
sched_setaffinity(0, sizeof(set), &set);
```

为什么要绑?两笔账:进程换核 → L1/L2 cache 全部白给;中断飘来飘去 → 包的处理路径在核间乱跳。于是高性能服务的常见架构是**划地盘**:

```
CPU 0-3:   专门处理网卡中断 + 软中断
CPU 4-15:  业务进程绑死在这里,不被任何中断打扰
```

DPDK、Nginx、Redis 的调优手册里全有这一招。

## 九、关中断:内核的"勿扰模式"及其边界

内核代码有时需要一小段绝对不被打断的临界区(比如改一个中断处理函数也会碰的数据结构):

```c
local_irq_disable();
// 临界区,本核中断被屏蔽
local_irq_enable();
```

自旋锁的中断安全变体内部就是这么干的。但这是双刃剑——**关中断期间整个核对世界失聪**,关久了就是丢事件、系统卡顿。铁律:临界区越短越好。

延伸到实时系统:实时(real-time)的核心诉求是**响应时间可预期**,而"不知道哪段关中断会突然挡路"正是可预期性的天敌。Linux 主线不是实时内核,**PREEMPT_RT 补丁**的大量工作就是把"不可打断的区间"切碎,让它逼近实时。

## 十、排查工具:把中断看见

```bash
cat /proc/interrupts        # 各核处理的硬中断次数(按 IRQ 分)
cat /proc/softirqs          # 软中断统计(按类型分)
mpstat -P ALL 1             # 每核使用率,含 %irq / %soft
watch -n 1 'cat /proc/interrupts | grep eth0'   # 盯着网卡中断的分布
```

`top` 里那一行 CPU 指标,现在你全能读懂了:

```
%us  用户态          %sy  内核态
%si  软中断  ← 高 = 网络 / 块 IO 下半部在吃 CPU
%hi  硬中断  ← 高 = 中断太频繁
%id  空闲            %wa  IO 等待
%st  被虚拟化宿主偷走的时间
```

**"CPU 很忙但业务没在跑"的排查钥匙,就是 %si / %hi 这两个数**。

## 十一、踩坑提醒

1. **中断处理函数里随手 printk**——慢,高频时还会丢日志
2. **中断上下文里睡眠**——直接 kernel panic,这是铁律不是建议
3. **高 PPS 服务不开 RSS / RPS**——单核中断打爆,整机看着空闲却疯狂丢包
4. **HZ 乱设**——服务器 1000 合理,嵌入式 100 够用,设高了纯费电
5. **关中断忘了开 / 关太久**——系统级卡顿,极难排查
6. **以为缺页异常是出错了**——它是内存管理的核心机制,每秒成千上万次很正常
7. **以为段错误全是野指针**——也可能是栈溢出、写只读段、SIGBUS(对齐 / mmap 文件被截断)
8. **高性能服务不绑 affinity**——换核 + cache 失效,白丢两位数性能
9. **把软中断和信号混为一谈**——softirq 是内核内部机制,signal 是发给用户进程的,两码事
10. **看不懂 %si / %hi**——"CPU 高但业务没跑"的答案就在这两个数里
