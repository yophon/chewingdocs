# 中断与异常

CPU 平时按部就班执行你的程序,但**总有事情打断它**——网卡来包了、定时器到了、你访问了非法地址、你调了 syscall。这些"打断"统称**陷入(trap)**,是 OS 进入工作的唯一入口。**没有中断,OS 根本不知道发生了什么**——它不会主动循环询问"网卡有数据没"(那叫轮询,极浪费 CPU)。理解中断,你就理解了"OS 怎么和硬件打交道"、"为什么高并发网络服务必须调度多核"、"为什么实时系统要禁中断"。

> 一句话先记住:**中断 = "硬件 / 软件主动叫醒 CPU,让它去干别的事"**。三种来源:**外部硬件(IRQ)、CPU 异常(除零、缺页、segfault)、软件主动(syscall / int 80)**。**中断处理必须快**——上下半部分(top half / bottom half)就是"立即处理"和"延后处理"的分工。**网卡 PPS 高时单核扛不住,必须 RSS / RPS 把中断分散到多核**。

---

## 一、CPU 平时在干嘛

CPU 永远在跑两种东西:

```
进程的代码(用户态 / 内核态)
   ↓ 中断来了
中断处理程序
   ↓ 处理完
回到原来的代码继续跑
```

**关键:CPU 自己不会"分心"** —— 它一直按 PC 指针往下跑,**只有外部事件能让它跳走**。

---

## 二、三种陷入(Trap)

```
1. 中断 (Interrupt)
   外部硬件主动通知:网卡 / 磁盘 / 键盘 / 定时器
   异步,与当前程序无关
   
2. 异常 (Exception)
   CPU 执行指令时出错或需要特殊处理
   除零 / 非法指令 / 缺页 / 访问越界
   同步,由当前指令引起
   
3. 系统调用 (Syscall)
   用户程序主动陷入内核
   用 int 0x80 / syscall / sysenter 指令
   同步,主动触发
```

**三者的硬件机制非常类似**——都是切换到内核态,执行处理函数,完了切回用户态。**Linux 内部统一叫 "trap"**。

## 三、硬件中断:CPU 怎么知道有事

### 3.1 IRQ:中断请求线

每个外设连接到中断控制器(APIC),通过 IRQ 号通知 CPU:

```
键盘中断:    IRQ 1
鼠标:        IRQ 12
网卡 0:      IRQ 32
网卡 1:      IRQ 33
定时器:      IRQ 0(每秒几百次,后面讲)
```

```bash
cat /proc/interrupts
#            CPU0       CPU1       CPU2       CPU3
#  0:        20         0          0          0   IO-APIC   2-edge      timer
#  1:        9          0          0          0   IO-APIC   1-edge      i8042
# 32:        12345      6789       1234       567 IO-APIC  32-fasteoi  eth0
```

每行是一个 IRQ,**列出每个 CPU 处理了多少次**——能看出哪个核被中断打爆了。

### 3.2 中断来了 CPU 在做什么

```
1. CPU 检测到 IRQ 信号
2. 完成当前指令(CPU 在每条指令之间检查中断)
3. 保存当前状态(寄存器、PC)
4. 切到内核态
5. 查中断描述符表(IDT),找到处理函数
6. 跳转执行处理函数
7. 处理完,iret 指令恢复状态,继续原代码
```

**整个流程几百纳秒** —— 相比应用代码已经很慢了。

### 3.3 中断"打断"用户程序

```
用户程序:    a = b + c;   // 这一行执行完
                ↓ 中断来
内核中断处理:网卡读包、放队列
                ↓ iret
用户程序:    d = a + 1;   // 继续下一行
```

用户程序**完全不知道被打断过** —— 中断对程序透明。

## 四、上半部 vs 下半部:中断处理的分工

### 4.1 为什么要分

中断处理函数有个铁律:**必须快**(几微秒内),否则:

- 期间其他中断被屏蔽 / 排队
- 当前 CPU 不能处理其他任务
- 高频中断挤爆 CPU

但有些任务真的复杂(协议栈解析、写文件)—— 怎么办?

**答案:分两半**:

```
上半部(top half / hardirq):
  - 中断真正触发时执行
  - 必须极快(只做"必须立即做"的事)
  - 不能睡眠,不能阻塞
  - 例子:从网卡读包到内核 buffer
  
下半部(bottom half / softirq / tasklet / workqueue):
  - 由上半部触发,延后执行
  - 可以慢一点,可以做复杂逻辑
  - 例子:解析包、走 TCP 协议栈、唤醒 socket 接收者
```

### 4.2 三种下半部机制

**softirq(软中断)**

最快,固定 10 种类型(网络、定时器、块 IO、调度等),内核启动时定义,**不能动态添加**。

```
NET_TX / NET_RX:    网络收发
BLOCK / BLOCK_IOPOLL: 块设备 IO
TIMER:               定时器
TASKLET:             tasklet 调度
SCHED:               调度
RCU:                 RCU 回调
HRTIMER:             高精度定时器
```

`top` 里的 `%si`(software interrupts)就是这一类。

**tasklet**

基于 softirq 实现,但允许动态创建。**单个 tasklet 不能并发跑在多个 CPU**(简化设计)。

**workqueue(工作队列)**

放到内核线程(`kworker`)异步执行 —— **可以睡眠**,适合最慢、最复杂的任务。

## 五、定时器中断:OS 的"心跳"

### 5.1 HZ:每秒中断次数

```
HZ = 100   每秒 100 次中断   早期低功耗
HZ = 250   每秒 250 次       桌面默认
HZ = 1000  每秒 1000 次      服务器,响应快但开销大
```

```bash
grep ^CONFIG_HZ /boot/config-$(uname -r)
# CONFIG_HZ=250
```

### 5.2 时间片调度依赖定时器

```
定时器触发 → 进入内核 → 调度器决定要不要切换进程
```

详见 16 篇调度器。

### 5.3 tickless kernel(动态 tick)

现代内核(Linux 3.10+)在 CPU 空闲时**不发定时器中断** —— 省电。需要时才唤醒。

```bash
CONFIG_NO_HZ_FULL=y    # 用户态运行时也不发 tick(适合 HPC / 实时)
```

## 六、网卡中断与 PPS 瓶颈

### 6.1 单核中断瓶颈

```
网卡 100 万 PPS (Packet Per Second):
  → 100 万次中断 / 秒
  → 单 CPU 核 100% 都在处理中断,业务跑不动
```

这就是**网络 PPS 瓶颈**。

### 6.2 RSS:网卡硬件分流

**RSS(Receive Side Scaling)** = 网卡硬件按包的哈希(IP/Port)分到不同队列,**每个队列对应一个中断,绑到不同 CPU**。

```bash
ethtool -L eth0 combined 8     # 8 个队列
ethtool -X eth0 hkey ...        # 配哈希
cat /proc/interrupts | grep eth0
# 32: ... CPU0 ...               队列 0
# 33: ... CPU1 ...               队列 1
# ...
```

**100 万 PPS 分摊到 8 核 → 每核 12.5 万**,扛得住。

### 6.3 RPS:软件再次分流

如果网卡只有一个队列(老网卡 / 虚拟机),**用 RPS 在内核里再分**:

```bash
echo ff > /sys/class/net/eth0/queues/rx-0/rps_cpus
# bitmap,ff = 8 核都参与
```

### 6.4 NAPI:中断 + 轮询混合

Linux 网络栈的优化:

```
低 PPS:   纯中断模式
高 PPS:   第一个中断后,关中断改轮询,处理完一批再开中断
```

**避免高 PPS 时中断风暴** —— 是现代网卡驱动的标配。

## 七、异常:CPU 自己出错

### 7.1 三类异常

**Fault(故障)**:可恢复

```
缺页(Page Fault):访问的页不在物理内存 → 内核换入 → 重新执行该指令
```

**Trap(陷阱)**:故意的

```
断点(int 3):    debugger 用
syscall 指令:    主动陷入
```

**Abort(中止)**:不可恢复

```
机器检查异常(MCE):硬件故障
双重故障(Double Fault):内核 bug
→ 通常导致内核 panic
```

### 7.2 缺页中断(Page Fault):最频繁的异常

```c
char *p = mmap(...);    // mmap 只是分配虚拟地址,没真分物理页
p[0] = 1;               // 访问触发缺页
                        // 内核分配物理页,建立映射,继续执行
```

**详见 06 篇虚拟内存** —— 缺页是 OS 内存管理的核心机制。

### 7.3 段错误(SIGSEGV)

非法内存访问:

```c
int *p = NULL;
*p = 1;                 // 访问地址 0,触发缺页
                        // 内核检查:这个地址没合法映射 → 发 SIGSEGV 给进程
                        // 进程没有 handler → 默认 dump core
```

**信号机制**详见 24 篇。

## 八、CPU 亲和性(CPU Affinity)

把进程 / 中断**绑到特定 CPU 核**:

### 8.1 绑进程

```bash
taskset -c 0,1 ./my_program     # 只在 CPU 0 / 1 上跑
```

```c
cpu_set_t set;
CPU_ZERO(&set);
CPU_SET(0, &set);
sched_setaffinity(0, sizeof(set), &set);
```

### 8.2 绑中断

```bash
echo 4 > /proc/irq/32/smp_affinity   # bitmap,4 = CPU 2
```

### 8.3 为什么要绑

**减少上下文切换 + 提高 cache 命中率**:

- 进程换核 → L1/L2 cache 全失效
- 中断在不同核处理 → 网络包到处乱跑

**高性能场景常见架构**:

```
CPU 0-3:处理网卡中断 + 软中断
CPU 4-15:跑业务进程,绑死,不被中断打扰
```

DPDK / Nginx 调优都依赖 affinity。

## 九、关中断 / 开中断

### 9.1 内核里禁中断

```c
local_irq_disable();
// 临界区,中断被屏蔽
local_irq_enable();
```

**用途**:保护与中断处理共享的数据(自旋锁的实现里就有)。

**禁忌**:**长时间禁中断 = 错过中断 = 系统卡顿**。

### 9.2 实时系统的需求

实时(real-time)系统要求**响应时间确定性** —— 关键操作期间不能被中断打断。

Linux 主线不是实时内核,但有 **PREEMPT_RT 补丁** 让它接近实时。

## 十、syscall 也是一种"中断"

第 02 篇讲过,**syscall 在硬件实现上也是 trap**:

```
int 0x80    早期 Intel
sysenter    Pentium II+
syscall     AMD,x86_64 标配
```

**和硬件中断的区别**:由用户主动触发,**同步**(不像硬件中断那样异步打断)。

## 十一、看中断的工具

```bash
cat /proc/interrupts        # 各 CPU 处理的中断次数
cat /proc/softirqs          # 软中断统计
mpstat -P ALL 1             # 每核 CPU 使用率(含 %irq / %soft)
top                         # 看 %si / %hi 占比

# 哪个 CPU 中断多?
watch -n 1 'cat /proc/interrupts | grep eth0'
```

`top` 里:

```
%us  用户态
%sy  内核态
%si  软中断 ← 高说明网络 / 块 IO 处理多
%hi  硬中断 ← 高说明中断频繁
%id  空闲
%wa  IO 等待
%st  虚拟化偷走的时间
```

## 十二、踩坑提醒

1. **中断处理函数里 `printk`(等价 printf)** —— 慢,有时会丢
2. **中断处理函数睡眠** —— 直接 panic
3. **不开 RSS / RPS** —— 单核网卡中断打爆,p99 抖动
4. **HZ 设太高** —— 服务器 1000 没问题,嵌入式 100 够,设错费电
5. **关中断不及时开** —— 系统卡顿
6. **以为缺页异常很罕见** —— mmap / lazy allocation 都靠它,频繁
7. **以为段错误是程序员错** —— 一半是 bug,一半是 SIGBUS / 写只读内存
8. **不绑 affinity 跑高性能服务** —— 上下文切换 + cache miss 性能损失大
9. **混淆软中断和信号** —— softirq 是内核内部机制,signal 是给用户进程的
10. **看 top 看不懂 %si / %hi** —— 排查"CPU 高但业务没在跑"的关键
