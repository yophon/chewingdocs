# eBPF

想象一个排障的死局:线上某个服务偶发慢几百毫秒,日志里什么都没有(出问题的代码路径没打日志),strace 不敢挂(上一篇说过,高 QPS 服务挂 ptrace 等于自造故障),改代码加日志要发版、要重启、要等下一次复现——而问题在内核里:可能是磁盘 IO 长尾,可能是调度排队,可能是 TCP 重传。**应用层工具到这里全部失明**。eBPF 解的就是这个局:**不改内核、不写内核模块、不重启任何东西,把一段自己写的小程序挂到内核事件点上,事件发生时就地执行、就地统计**。听起来像"往内核里注入代码"一样危险,但它恰恰建立在一个极其严格的安全机制上——这是本篇真正的主角。

> 一句话先记住:**eBPF = 经过验证器(verifier)数学级审查的小程序,挂在内核事件点上运行,用 map 和用户态交换数据**。verifier 保证它崩不了内核,hook 决定它挂在哪,map 负责把结果带出来。**可观测性、网络、安全是它的三大主战场**——它把 Linux 内核从黑盒变成了可编程平台。

---

## 一、为什么需要 eBPF:四条老路都不通

"看内核正在发生什么"这个需求一直存在,eBPF 之前有四条路,各有致命伤:

| 老办法 | 致命伤 |
| --- | --- |
| 改代码加日志 | 要发版重启;而且**内核里的事(调度/IO/网络栈)应用代码根本看不见** |
| strace | ptrace 开销巨大,高 QPS 服务挂上去性能掉几倍到几十倍 |
| 写内核模块 | 跑在内核态毫无防护,一个空指针 = 整机 panic,没人敢往生产装 |
| tcpdump | 只能看包,看不到"哪个进程、哪个函数、哪把锁"和包的关联 |

eBPF 的工作模式把这四个痛点全部绕开:

```
写一个小程序(几十行)
  → 加载到内核的某个 hook 点
  → 对应事件每次发生,程序就地执行(纳秒级)
  → 统计结果写进 map(内核里的共享数据结构)
  → 用户态随时读 map 拿结果
```

不改业务代码、不重启服务、开销低到敢常驻生产——**"低成本看到内核真相"是 eBPF 的全部卖点**。

## 二、凭什么敢往内核里塞代码:verifier

这是 eBPF 设计里最精彩的部分。内核模块的教训摆在那:内核态代码出错就是整机崩溃。eBPF 敢开放"任何人写代码进内核",靠的是加载时的强制安检——**verifier 会模拟执行你程序的所有可能路径,任何一条路径有出格的可能,直接拒绝加载**:

```
你写 C 程序 / bpftrace 脚本
      │ clang -target bpf 编译
      ▼
  BPF 字节码
      │ bpf() syscall 申请加载
      ▼
┌── Verifier 静态审查(模拟执行所有分支)─────────────┐
│  ✓ 程序必然终止?(不允许无界循环,不允许死循环)      │
│  ✓ 所有内存访问都在边界内?                          │
│  ✓ 内核指针解引用前都判过空、过了 helper 校验?        │
│  ✓ 栈用量 ≤ 512 字节?                              │
│  ✓ 只调用了白名单里的 helper 函数?                   │
│  任何一条不满足 ──────────────→ 拒绝加载,完事        │
└─────────────────────────────────────────────────┘
      │ 全部通过
      ▼
  JIT 编译成本机指令(跑起来几乎是原生速度)
      │
      ▼
  attach 到 hook 点,开始工作:
      内核事件发生 → 执行 BPF 程序 → 读写 map
                                       ↑
      用户态进程 ──── 读 map / ring buffer 取结果
```

关键的认知转换:**eBPF 程序不是"被信任才能跑",是"被证明无害才能跑"**。verifier 不看你是谁,只看程序在数学上是否不可能越界、不可能死循环、不可能解引用野指针。代价是表达能力受限——你写不了任意复杂的逻辑(循环必须有界、栈只有 512 字节),但换来的是**内核敢让你上车**。这笔交易就是 eBPF 整个生态的地基。

map 是另一半拼图:BPF 程序跑在内核、生命短暂(一个事件一次执行),**状态和结果都存在 map 里**——哈希表、数组、直方图、ring buffer,内核态写、用户态读,两个世界靠它对话。

## 三、hook 点地图:挂在哪,决定你能看见什么

eBPF 程序本身只是"一段会被执行的逻辑",真正决定它用途的是 attach 到哪个 hook:

| Hook | 挂在哪 | 典型用途 |
| --- | --- | --- |
| tracepoint | 内核预埋的稳定跟踪点 | 看 syscall、调度、IO 事件(**首选**) |
| kprobe / kretprobe | 几乎任意内核函数的入口/返回 | tracepoint 没覆盖时的万能后门 |
| uprobe / uretprobe | **用户态**程序的函数 | 看 SSL 加密前的明文、malloc 调用 |
| perf event | CPU 性能计数器 / 定时采样 | CPU profiling(26 篇 perf 的底层) |
| XDP | 网卡驱动收包的最早一刻 | DDoS 防御、负载均衡(第六节) |
| TC | 流量控制层(协议栈内) | 容器网络策略、限速 |
| cgroup hooks | 按 cgroup 挂网络/syscall 钩子 | 容器粒度的过滤和观测 |
| LSM | 安全模块挂点 | 运行时安全策略(BPF 版 SELinux) |

tracepoint 和 kprobe 的取舍要单独说,因为天天遇到:

| | tracepoint | kprobe |
| --- | --- | --- |
| 稳定性 | **内核 ABI 承诺,跨版本稳定** | 挂的是内核函数,改名/改参数就失效 |
| 覆盖面 | 只有预埋的点 | 几乎任意内核函数 |
| 字段 | 明确、文档化 | 要自己看内核源码对参数 |

原则:**tracepoint 够用就用 tracepoint,不够再 kprobe**——前者是官方承诺的接口,后者是"在别人家房梁上钉钉子",内核一升级你的脚本可能就哑了。

## 四、bpftrace:十分钟上手的入口

eBPF 原生开发(写 C、管加载、读 map)门槛不低,bpftrace 把它压缩成 awk 风格的单行脚本——**排障现场的第一武器**:

```bash
# 这台机器上谁在疯狂调 syscall?(全机 syscall 排行)
bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[comm] = count(); }'

# 哪个进程在反复打开什么文件?(配置被谁动了的破案神器)
bpftrace -e 'tracepoint:syscalls:sys_enter_openat
             { printf("%s %s\n", comm, str(args->filename)); }'

# vfs_read 延迟直方图:慢是整体慢,还是长尾慢?
bpftrace -e '
kprobe:vfs_read { @start[tid] = nsecs; }
kretprobe:vfs_read /@start[tid]/ {
  @lat = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'
```

第三个脚本的模式(入口记时间戳、返回算差值、聚成直方图)是 eBPF 观测的万能模板,**任何"XX 耗时分布"问题都套它**。输出的直方图能直接回答 26 篇剧本里那个关键问题:p99 抖动到底是普遍变慢,还是 1% 的长尾——两者的根因方向完全不同。

```bash
bpftrace -l 'tracepoint:syscalls:*'    # 列出可用 hook 点,现场翻字典
```

bpftrace 的定位:**快速排查、一次性脚本、验证猜想**。要写常驻的观测程序,看下一节。

## 五、BCC 与 libbpf + CO-RE:从脚本到产品

| | BCC | libbpf + CO-RE |
| --- | --- | --- |
| 形态 | Python 前端,内嵌 C 写 BPF | 纯编译型,C/Rust/Go |
| 编译时机 | **运行时现场编译**(每次启动) | 编译一次,到处运行 |
| 依赖 | 拖着整个 clang/LLVM,很重 | 一个小二进制 |
| 适合 | 学习、用现成工具 | **生产级常驻程序的标准答案** |

CO-RE(Compile Once, Run Everywhere)解决的是 eBPF 落地最大的痛点:不同内核版本的内部结构体布局不一样,以前的 BPF 程序要在目标机器上现场编译才能对上偏移。CO-RE 借助内核自带的 BTF(类型信息),**在加载时自动重定位字段偏移**——编译一次的二进制,扔到不同版本的内核上都能跑。现代生产级 eBPF 项目(Cilium、Pixie……)全是这条路线。

BCC 生态里那批现成工具不用写一行代码,直接对症下药:

| 症状 | 工具 | 回答什么 |
| --- | --- | --- |
| 机器上有奇怪的短命进程 | `execsnoop` | 谁在 exec 什么命令(抓挖矿脚本名场面) |
| 配置文件被谁改了/读了 | `opensnoop` | 全机每一次 open |
| 谁在对外建连 | `tcpconnect` / `tcptop` | 每条出站连接 + 流量排行 |
| 磁盘延迟可疑 | `biolatency` | 块设备 IO 延迟直方图 |
| CPU 没满但就是慢 | `runqlat` | **就绪后等 CPU 等了多久**(调度延迟) |
| 想要全机火焰图 | `profile` | 内核 + 用户态混合采样 |

`runqlat` 值得划重点:它量化的"任务就绪了却排队等 CPU"的时间,正是 26 篇说的传统工具盲区——CPU 利用率没满、应用没报错,延迟却凭空多出来,answer 往往在这。

## 六、网络主战场:XDP 和 TC

观测之外,eBPF 的第二战场是直接**处理**网络包。XDP(eXpress Data Path)挂在网卡驱动收包的最早一刻——**比内核协议栈还早,sk_buff 都还没分配**:

```
网卡收到包
  → XDP 程序先看一眼,裁决:
      DROP      → 当场丢弃(一个 sk_buff 都不用建,丢包成本趋近于零)
      PASS      → 放行,走正常协议栈
      TX        → 原网卡弹回去(改包转发)
      REDIRECT  → 转给其它网卡 / CPU / AF_XDP socket
  → 内核协议栈(只有 PASS 的包才走到这)
```

为什么 DDoS 防御非它不可:攻击流量打过来,iptables 丢包要等包走完协议栈前半程,**XDP 在门口就把包拍死**,单机抗千万 pps 级的攻击流量。负载均衡同理——Facebook 的 Katran、Cloudflare 的边缘清洗,都是 XDP 写的。

TC(traffic control)层的 hook 在协议栈更深处,慢一点但**能拿到完整的 skb 上下文**(知道包属于哪个 socket、哪个 cgroup),适合做策略:容器间网络规则、限速、服务网格数据面。

最大的现实案例是 **Cilium**:K8s 网络插件,用 eBPF 全面替换 kube-proxy 的 iptables 方案。iptables 是链式规则,service 一多就是 O(n) 线性匹配,几千个 service 时每个包都要过长队;eBPF 用 map 做 O(1) 查找,顺手把网络策略、负载均衡、可观测性全包了——**这是"eBPF 重写 Linux 网络基础设施"的代表作**(K8s 网络详见 backendLearning 系列)。

## 七、可观测性的几个名场面

把工具对回真实问题,这几个场景值得记:

- **"CPU 不忙但延迟高"** → `runqlat`:调度延迟直方图,确认是不是就绪队列在排队(容器里叠加 25 篇的 throttle 检查)。
- **"磁盘平均指标正常,应用偶发慢"** → `biolatency`:平均值看不出的 IO 长尾,直方图一目了然——99% 的 IO 在 1ms 内,1% 在 100ms,这 1% 就是你的 p99。
- **"网络偶发超时"** → 跟踪 `tcp_retransmit_skb`(现成工具 `tcpretrans`):每次重传打出四元组,瞬间知道是哪条链路在丢包。
- **"锁竞争,但 perf 看不到"** → CPU 采样只能看到"在烧 CPU 的代码",**等锁的线程不烧 CPU**,在 perf 火焰图上隐形。这时看 off-CPU 时间(线程不在 CPU 上的时间花在等什么)+ futex 跟踪,正好补上 on-CPU 分析的盲区。

规律:**perf 负责"CPU 在干什么",eBPF 负责"时间去哪了"**——后者覆盖等待、排队、长尾这些采样看不见的暗物质。

## 八、边界与风险:它不是免费的

eBPF 强,但每一条边界都要心里有数:

1. **内核版本差异**——能力随内核演进(4.x 基础、5.x 才有 ring buffer / 有界循环 / CO-RE 成熟),老内核很多玩法直接没有。
2. **verifier 限制**——栈 512 字节、循环有界、指令数受限,复杂逻辑要拆,有时被 verifier 拒得莫名其妙(报错信息堪称天书)。
3. **观测本身有开销**——hook 一个每秒百万次的事件、还在里面 printf,系统直接被你拖垮。**高频事件只做内核内聚合(count/hist),永远不要逐条输出**。
4. **kprobe 字段不稳定**——内核升级,函数改名、参数变化,脚本静默失效。生产长期跑的程序尽量 tracepoint + CO-RE。
5. **权限要求高**——通常要 root 或 CAP_BPF / CAP_SYS_ADMIN,这本身就是一个安全敏感面(能挂 uprobe 就能读 TLS 明文)。
6. **容器/云环境受限**——托管 K8s、安全策略可能禁用部分 BPF 能力,本地能跑的脚本线上不一定能。

使用原则浓缩成一张 checklist:**优先 tracepoint 再 kprobe;高频事件只聚合;先在测试环境跑;加 pid/comm/cgroup 过滤;观测窗口短而明确;长期程序用 libbpf + CO-RE**。

最后摆正位置:eBPF **不替代**应用日志、指标系统、分布式 tracing、语言级 profiler,更不替代容量规划——那些回答"应用在干什么",eBPF 回答的是**"应用看不见的内核层发生了什么"**。它是补盲区的,不是推倒重来的。

## 踩坑提醒

1. **把 eBPF 当"高级 strace"随便挂**——挂错 hook 一样能拖垮系统:高频 kprobe + 逐条 printf 的组合,开销不比 strace 小。聚合,永远先聚合。
2. **新手直挂 kprobe**——内核一升级函数没了,脚本静默失效还以为"没问题"。有 tracepoint 用 tracepoint。
3. **在生产首跑没验证过的脚本**——verifier 保证不崩内核,**不保证不拖慢内核**。先测试环境,先短窗口。
4. **忘了加过滤条件**——不带 pid/cgroup 过滤的全机跟踪,事件量可能比你预想的大几个数量级。
5. **用 perf 火焰图找等锁问题**——等待不烧 CPU,on-CPU 采样看不见,要 off-CPU 分析 / futex 跟踪。
6. **以为本地能跑 = 线上能跑**——内核版本、BTF 有无、容器安全策略,三道坎每道都可能拦住你。
7. **拿 eBPF 替代日志和指标**——它是内核层的补盲工具,应用层的可观测性基建一样都不能少。
8. **写复杂业务逻辑硬怼 verifier**——512 字节栈和有界循环是硬墙,复杂逻辑放用户态,内核里只做采集和聚合。
