# 进程的内存布局

每个 Linux 进程都有一张「内存地图」——代码、数据、堆、栈、共享库、内核区域各占一段。**理解这张图,你才能解释「Java 进程真实内存远不止 -Xmx」「为什么栈溢出叫 stack overflow 而不是别的」「为什么 malloc 大块用 mmap 而不是 brk」**。这一篇把进程的内存地图剖到字节级:**每个段是什么、谁负责、怎么扩张、怎么排查**。

> 一句话先记住:**进程的虚拟地址空间 = 文本段 + 数据段 + 堆(向上长)+ 共享库 + 栈(向下长)+ 内核空间**。**堆和栈中间留巨大空隙**(几 TB),是给 mmap 和共享库用的。**Java -Xmx 只限制堆,但实际进程内存还有线程栈 / Native 内存 / Metaspace / GC 元数据 / JIT 代码 / 共享库**——这就是 Java OOM Killer 杀进程时管理员一脸懵的原因。

---

## 一、整体地图(x86_64,典型布局)

```
0xFFFF_FFFF_FFFF_FFFF  ┌──────────────────────┐
                       │     内核空间(128 TB) │
                       │   所有进程共享        │
0xFFFF_8000_0000_0000  ├──────────────────────┤
                       │                      │
                       │   非规范地址空洞      │
                       │   (硬件不允许使用)   │
                       │                      │
0x0000_7FFF_FFFF_FFFF  ├──────────────────────┤
                       │     栈(向下增长)     │
                       │     Stack            │
                       ├──────────────────────┤
                       │                      │
                       │     mmap 区域         │
                       │     共享库 / 大块内存 │
                       │     (向下增长)        │
                       ├──────────────────────┤
                       │                      │
                       │     堆(向上增长)     │
                       │     Heap (brk)       │
                       ├──────────────────────┤
                       │     BSS 段            │ 未初始化全局变量
                       │     Data 段           │ 已初始化全局变量
                       │     Text(.text)段    │ 代码
0x0000_0000_0040_0000  ├──────────────────────┤
                       │     0 ~ 0x40_0000    │ 不映射(让 NULL 解引用 segfault)
0x0000_0000_0000_0000  └──────────────────────┘
```

**关键点**:

- **用户空间**:128 TB
- **内核空间**:128 TB(所有进程共享同一份内核映射)
- **中间巨大空洞**:硬件限制(64 位 CPU 实际只用 48 位地址)

## 二、文本段(.text):你的代码

存放**可执行的机器码**。

**特征**:

- **只读 + 可执行**(R-X)—— 防止程序意外覆盖自己的代码
- 多个进程共享(同一个二进制跑两次,只占一份物理内存)
- 通常从 `0x400000` 开始(non-PIE)或随机化(PIE,现代默认)

```bash
size /bin/ls
#    text       data        bss        dec        hex    filename
#  120741       4520        788     126049      1ec61    /bin/ls
```

**写文本段会怎样**:

```c
char *p = main;
*p = 0;        // 段错误,因为 .text 是只读的
```

## 三、数据段:全局 / 静态变量

### 3.1 .data:已初始化

```c
int g_counter = 100;        // 在 .data
char g_msg[] = "hello";     // 在 .data
```

**特征**:

- 可读可写
- 占文件空间(初始值要保存)
- 程序启动时从 ELF 文件加载到内存

### 3.2 .bss:未初始化

```c
int g_arr[1000000];         // 在 .bss
static int g_x;             // 在 .bss
```

**特征**:

- 可读可写
- **不占文件空间**(ELF 文件只记"我要 4MB 全为 0",不存内容)
- 程序启动时,内核分配清零页(零页共享,COW)

> **`int arr[100_000_000]`(全局)** 让你的二进制只大几 KB,但运行时占 400 MB —— 因为 .bss。

### 3.3 .rodata:只读数据

```c
const char *msg = "Hello";       // "Hello" 在 .rodata
const int max = 100;             // max 在 .rodata
```

**特征**:

- 只读(写会 segfault)
- 字符串字面量、const 全局变量都在这

```c
char *p = "hello";
p[0] = 'H';        // 段错误! "hello" 在 .rodata
```

## 四、堆(Heap):动态分配

`malloc` / `new` 来的内存。

### 4.1 堆的范围

```
brk 指针:堆的"顶端"
向上扩展:brk() syscall 改 brk 指针
```

```bash
cat /proc/$$/maps | grep heap
# 555a3d76d000-555a3d791000 rw-p 00000000 00:00 0   [heap]
```

### 4.2 brk vs mmap:malloc 的两种实现

glibc malloc 内部:

```
小对象(< 128 KB,M_MMAP_THRESHOLD):
  从堆里切一块(用 brk 扩堆)
  
大对象(≥ 128 KB):
  直接 mmap 一段匿名内存
  free 时直接 munmap 还给内核
```

**为什么这么分**:

- 小对象频繁分配释放 → 用 brk 池化,避免反复 syscall
- 大对象长期占用 → 用 mmap,free 时立即归还内核

**坑**:**堆的内存不会真还给 OS** —— 你 free 了,glibc 只是把它放回自己的池子,brk 指针不会缩。**长期跑的进程"内存只涨不跌"是常态**(并不一定泄漏)。

### 4.3 看堆使用

```bash
pmap -x <pid>           # 看 [heap] 段大小
malloc_stats()          # glibc 的内部统计
mallinfo2()             # API 形式
```

## 五、栈(Stack):函数调用与局部变量

### 5.1 栈的特征

- **向下增长**(从高地址往低地址)
- **每个线程有独立的栈**(主线程的栈在最高处,新线程的栈通常 mmap 在别处)
- **大小受限**(默认 8 MB,可调)

```bash
ulimit -s              # 当前栈大小限制(KB)
# 8192               = 8 MB
ulimit -s unlimited    # 取消限制
```

### 5.2 栈帧

每次函数调用,**压入一个栈帧**:

```
高地址
┌─────────────┐  ← 主函数的栈帧
│ 局部变量     │
│ 返回地址     │
├─────────────┤  ← 被调用函数的栈帧 (rsp)
│ 局部变量     │
│ 参数         │
│ 返回地址     │
└─────────────┘
低地址(继续向下)
```

**rsp 寄存器**指向栈顶。函数 prologue / epilogue 自动调整 rsp。

### 5.3 栈溢出(Stack Overflow)

```c
void recurse() {
    int big_array[100000];   // 每次递归占 400KB
    recurse();               // 永远递归
}
```

**栈不停增长 → 撞到栈区下界 → segfault**(**stack overflow** 这个词的来源)。

**Linux 的栈"自动扩展"**:

```
栈区一开始只分了几页
访问栈底之外的地址 → 缺页
内核检查:这是栈,允许扩展 → 分新页
直到达到 ulimit -s 上限 → 真正 segfault
```

### 5.4 多线程的栈

每个线程独立的栈,**通常 mmap 在堆和共享库之间**:

```
新线程默认栈 8MB(pthread_create 时配)
1000 个线程 = 8 GB 虚拟内存(实际物理通常很小,COW)
```

**坑**:**线程多了虚拟内存爆炸**——32 位系统跑不动 1 万线程(虚拟地址空间不够)。

## 六、共享库(Shared Libraries)

`.so` 文件被 mmap 到 mmap 区域:

```bash
cat /proc/$$/maps | grep \\.so
# 7f4a... r--p ... /lib/x86_64-linux-gnu/libc.so.6      .text 段(只读可执行)
# 7f4a... r-xp ...
# 7f4a... r--p ...
# 7f4a... rw-p ...                                      .data 段(私有可写,COW)
```

**每个 .so 通常映射 4-5 段**(.text / .rodata / .data / .bss + GOT/PLT)。

**节省内存**:N 个进程都用 libc,**.text 段只占一份物理内存**(共享只读)。

## 七、内核空间

**所有进程共享同一份内核空间映射**:

```
0xFFFF_8000_0000_0000 - 0xFFFF_FFFF_FFFF_FFFF   内核
```

用户态进程**不能直接访问**——硬件检查页表权限,违规直接 segfault。

只有 syscall 进入内核态才能访问。

### 7.1 KPTI 之后

Meltdown 漏洞后,KPTI(Kernel Page Table Isolation):

```
用户态运行时:页表里只映射极少量内核入口代码
进入内核时:切换页表,加载完整内核映射
```

**代价:syscall 慢 5-30%**,因为多了页表切换 + TLB 刷新。

## 八、Java 进程的真实内存

**Java -Xmx 4G**,但 `top` 看 RES 6 GB——为什么?

```
JVM 进程的内存构成:
┌──────────────────────────────┐
│ 堆 (Heap)         -Xmx 限   │ ← 4 GB
├──────────────────────────────┤
│ Metaspace                    │ ← 类元数据,默认无上限
├──────────────────────────────┤
│ Code Cache (JIT 编译产物)    │ ← 几百 MB
├──────────────────────────────┤
│ Direct Buffer (NIO)          │ ← 容易爆,Netty 大头
├──────────────────────────────┤
│ 线程栈 (Thread Stack)         │ ← 每线程默认 1 MB,1000 线程 = 1 GB
├──────────────────────────────┤
│ JNI 内存 (Native libraries)  │ ← 自定义 native 代码 / 数据库驱动
├──────────────────────────────┤
│ GC 元数据 (Card table 等)    │
├──────────────────────────────┤
│ 共享库 (.so)                  │
└──────────────────────────────┘
```

**总和远超 -Xmx**,这是 Java 容器内 OOMKilled 的常见原因。

**对策**:

```bash
-Xmx4g                                           # 堆
-XX:MaxMetaspaceSize=256m                        # Metaspace
-XX:ReservedCodeCacheSize=256m                   # Code Cache
-XX:MaxDirectMemorySize=1g                       # Direct Buffer
-Xss256k                                         # 单线程栈
```

**容器内推荐**:`MaxRAMPercentage=70.0` —— 让 JVM 自适应容器内存,**总和留 30% 给非堆**。

### 9.2 /proc/<pid>/smaps

```bash

#   PID  VSZ   RSS  CMD
#  1234 8GB   2GB   java ...
# VSZ:虚拟地址空间总大小

```

## 十、OOM Killer 选谁:oom_score

```bash
cat /proc/<pid>/oom_score
# 数值越高越优先被杀
```

计算公式(简化):

```
score = 进程 RSS / 系统总内存 + adj
adj 由 oom_score_adj 调节,范围 -1000 ~ 1000
```

**保护关键进程**:

```bash
echo -1000 > /proc/<pid>/oom_score_adj    # 永不被 OOM Killer 杀
```

K8s 里:`Pod.spec.containers.resources` 决定 oom_score_adj。**Burstable / BestEffort 优先被杀,Guaranteed 最稳**。

## 十一、stack guard:防止栈跨段写入

每个线程栈底有一段**"哨兵页"(guard page)**——未映射的虚拟页。

```
栈正常区域
↓
栈区底部
↓
guard page(未映射)
↓
其他映射(共享库?其他线程栈?)
```

**栈溢出 → 写到 guard page → segfault**,**而不是悄悄覆盖隔壁内存**(那才是真正的安全灾难)。

## 十二、Address Sanitizer(ASan):查内存 bug 神器

GCC / Clang 的运行时检测:

```bash
gcc -fsanitize=address -g program.c -o program
./program
```

发现:

- 栈溢出
- 堆缓冲区越界
- 释放后使用(use after free)
- double free
- 内存泄漏

**代价**:程序慢 2-3 倍,内存 2-3 倍。**测试环境必开,生产关掉**。

## 十三、地址布局总结

```
低地址            高地址
[ NULL 区 (0~4MB,故意不映射) ]
[ .text     代码,只读可执行,共享 ]
[ .rodata   只读数据,字面量 ]
[ .data     全局变量初值,可写 ]
[ .bss      全局变量零初值,可写 ]
[ heap      brk 向上扩展 ]
... 大空洞 ...
[ mmap 区   共享库、大块匿名内存,向下扩 ]
... 大空洞 ...
[ stack     向下扩展,带 guard page ]
[ vvar/vdso 内核共享的快速 syscall 区 ]
[ 内核空间 ]
```

## 十四、踩坑提醒

1. **写字符串字面量** —— `char *p = "x"; p[0] = 'y';` 段错误
2. **以为 free 把内存还给系统** —— glibc 只放回池子,RSS 不降
3. **大量小线程** —— 每个 1MB 栈,1000 线程 1 GB 虚拟内存
4. **递归没出口** —— 栈溢出 segfault
5. **栈上分配大数组** —— `char buf[10*1024*1024]` 直接爆栈
6. **以为 -Xmx 是 Java 进程总内存** —— 实际是 1.5-3 倍
7. **JVM 在容器里没设容器感知** —— -Xmx 比容器 limit 还大,被 OOMKilled
8. **不调 oom_score_adj** —— 关键服务被随机杀
9. **不看 PSS 看 RSS** —— 多进程共享库重复算
10. **Native 内存泄漏** —— Java 堆没漏但 Native 漏(JNI / Direct Buffer 不释放)
11. **mmap 大文件不 munmap** —— 虚拟地址耗尽
12. **以为 ASLR 关了能调试** —— 现代调试器自动处理,不用关
