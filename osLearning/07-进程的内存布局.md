# 进程的内存布局

运维拿着 `top` 来质问:"你这 Java 服务 -Xmx 才 4G,凭什么 RES 飙到 6G?"——如果你答不上来,说明你脑子里没有那张图:**每个 Linux 进程的虚拟地址空间是一张固定格式的地图**,代码、数据、堆、栈、共享库各占一段,各有各的脾气。堆只是其中一段,-Xmx 只管堆;栈溢出(stack overflow)这个词为什么叫"溢出"、`free` 了内存为什么不还给系统、字符串字面量为什么改一下就 segfault——答案全画在这张图上。这一篇把它逐段剖开。

> 一句话先记住:**进程地址空间 = 代码段 + 数据段 + 堆(向上长)+ mmap 区(共享库和大块内存)+ 栈(向下长)+ 内核空间**,堆和栈中间隔着几 TB 的空洞。**-Xmx 只限制 Java 堆,而进程真实内存还有线程栈、Metaspace、Direct Buffer、JIT 代码、Native 库**——这就是容器里 Java 被 OOMKilled 时大家一脸懵的原因。

---

## 一、先看全图

x86_64 上一个典型进程的地址空间(上一篇讲过,用户空间和内核空间各 128TB,中间是硬件造成的非规范地址空洞):

```
0xFFFF_FFFF_FFFF_FFFF ┌──────────────────────┐
                      │  内核空间(128TB)      │ 所有进程共享同一份
0xFFFF_8000_0000_0000 ├──────────────────────┤
                      │  非规范地址空洞        │ 硬件只用 48 位地址
0x0000_7FFF_FFFF_FFFF ├──────────────────────┤
                      │  栈 Stack    ↓ 向下长 │ 局部变量 / 函数调用
                      ├──────────────────────┤
                      │  mmap 区域   ↓        │ 共享库 / 大块 malloc / 线程栈
                      ├── (几 TB 的空洞) ─────┤
                      │  堆 Heap     ↑ 向上长 │ malloc / new
                      ├──────────────────────┤
                      │  .bss   未初始化全局   │
                      │  .data  已初始化全局   │
                      │  .rodata 只读数据      │
                      │  .text  代码          │
0x0000_0000_0040_0000 ├──────────────────────┤
                      │  0 附近故意不映射      │ 让空指针解引用必死
0x0000_0000_0000_0000 └──────────────────────┘
```

最底下那段"故意不映射"很妙:**NULL 解引用之所以稳定地 segfault 而不是悄悄读到垃圾,是因为内核刻意把低地址留空**——这是用地址空间换 debug 体验。

下面从低往高,一段一段讲。

---

## 二、代码段(.text):只读、可执行、大家共享

存机器码的地方。权限是 **R-X(只读 + 可执行)**——只读是为了防止程序意外(或恶意)改写自己的代码:

```c
char *p = (char *)main;
*p = 0;        // segfault,.text 不许写
```

两个工程意义上的好处:

- **同一个二进制跑 100 个进程,.text 只占一份物理内存**(只读所以能安全共享)
- 现代默认编译成 PIE(位置无关),配合 ASLR 每次加载地址随机化,防攻击;老式 non-PIE 固定从 `0x400000` 开始

```bash
size /bin/ls
#    text    data    bss     dec      hex   filename
#  120741    4520    788   126049   1ec61   /bin/ls
```

---

## 三、数据段三兄弟:.data / .bss / .rodata

全局变量和静态变量住在这里,但按"有没有初值、能不能改"分了三个小区:

| 段 | 住户 | 权限 | 占不占 ELF 文件体积 |
| --- | --- | --- | --- |
| `.data` | `int g = 100;` 有初值的全局/静态 | 可读写 | 占(初值得存着) |
| `.bss` | `int g_arr[1000000];` 零初值的 | 可读写 | **不占**(文件里只记"我要 4MB 全零") |
| `.rodata` | 字符串字面量、`const` 全局 | **只读** | 占 |

`.bss` 的设计很省:**`int arr[100000000]` 这种全局大数组只让二进制大几个字节,运行时却占 400MB**——内核启动时给它映射共享的零页(COW,写到哪页才真分哪页,和上一篇讲的缺页机制完美衔接)。

`.rodata` 则是一个新手必踩的坑的来源:

```c
char *p = "hello";
p[0] = 'H';        // segfault!"hello" 在 .rodata,只读
```

**字符串字面量是只读的**。想改就用 `char p[] = "hello";`(数组形式会把内容拷到栈或 .data)。

---

## 四、堆(Heap):malloc 的地盘,只涨不缩

`malloc` / `new` 的内存来自这里。堆有一个叫 **brk** 的指针标记堆顶,扩堆就是用 `brk()` 系统调用把它往上挪:

```bash
cat /proc/$$/maps | grep heap
# 555a3d76d000-555a3d791000 rw-p ... [heap]
```

但 glibc 的 malloc 并不是所有请求都走堆——它按大小分流:

```
小对象(< 128KB,M_MMAP_THRESHOLD):从堆里切一块(brk 扩堆)
大对象(≥ 128KB):           直接 mmap 一段独立的匿名内存
```

为什么这么分?**小对象分配释放极其频繁,用 brk 池化避免反复 syscall;大对象生命周期长,用 mmap 的好处是 free 时 munmap 直接还给内核**。

这里埋着本系列最重要的认知之一:**free 不会把堆内存还给操作系统**。glibc 只是把它放回自己的空闲池等下次复用,brk 指针不会缩。所以**长期运行的进程"内存只涨不跌"是常态,不一定是泄漏**——分清"碎片 / 池子"和"真泄漏"是 08 篇的主题。

观测工具:`pmap -x <pid>` 看 [heap] 段大小,代码里可以调 `malloc_stats()` / `mallinfo2()` 看 glibc 内部账目。

---

## 五、栈(Stack):向下生长,有限额,会溢出

栈管两件事:函数调用链和局部变量。三个关键性格:

- **向下生长**(从高地址往低地址)
- **每个线程一条独立的栈**
- **有限额**,默认 8MB:

```bash
ulimit -s        # 8192(KB)= 8MB
```

每次函数调用压入一个**栈帧(stack frame)**,`rsp` 寄存器指着栈顶:

```
高地址
┌──────────────┐ ← main 的栈帧
│ 局部变量      │
│ 返回地址      │
├──────────────┤ ← 被调函数的栈帧
│ 参数 / 局部变量│
│ 返回地址      │
└──────────────┘ ← rsp(继续向下)
低地址
```

### 栈的"自动扩展"和溢出

栈区一开始只映射几页,不是一上来就给 8MB:

```
访问超出当前栈底的地址 → 缺页中断
→ 内核一看:这地址紧挨着栈,是正常生长 → 默默分新页
→ 长到 ulimit -s 上限还想长 → 这次真的 segfault
```

这就是 **stack overflow** 这个词的字面来源——栈"溢"出了它的限额:

```c
void recurse() {
    int big[100000];   // 每层 400KB
    recurse();         // 没有出口
}                      // 20 层就撞 8MB 上限,segfault
```

两种最常见的爆法:**无限递归**,和**在栈上开大数组**(`char buf[10*1024*1024]` 一行就爆,大缓冲区请放堆上)。

### guard page:溢出时为什么是干净地死

每条线程栈的底部都有一段**哨兵页(guard page)**——故意不映射的虚拟页。栈长过头时会先撞上它,**立刻 segfault,而不是悄悄写穿到隔壁的映射(别的线程栈、共享库数据)**。悄悄写穿才是真正的安全灾难——guard page 让程序"死得明白"。

### 多线程的栈

主线程的栈在地址空间顶部,**其他线程的栈是 pthread 用 mmap 在 mmap 区分配的**,默认每条 8MB。算笔账:**1000 个线程 = 8GB 虚拟内存**(物理上因为按需调页通常没那么多)。32 位系统总共才 4GB 地址空间,根本跑不动一万个线程——这是"线程不能无限开"的地址空间维度的原因。

## 六、mmap 区:共享库和大块内存的杂居地

堆和栈之间那片巨大的区域归 mmap 管:共享库、大块 malloc、线程栈、文件映射都在这。看一眼共享库的映射:

```bash
cat /proc/$$/maps | grep '\.so'
# 7f4a... r--p ... libc.so.6     ← .rodata
# 7f4a... r-xp ... libc.so.6     ← .text(只读可执行)
# 7f4a... r--p ... libc.so.6
# 7f4a... rw-p ... libc.so.6     ← .data/.bss(私有,COW)
```

**每个 .so 通常映射 4-5 段**(.text / .rodata / .data / .bss / GOT 等)。重点还是那条:N 个进程都加载 libc,**只读的 .text 段物理内存只占一份**——这是动态链接库省内存的根本机制,也是 RSS 重复计算、要看 PSS 的原因(06 篇讲过)。

地图顶部还有几个内核放的小东西:`[vdso]` / `[vvar]` 是内核映射进每个进程的"快速系统调用页"(`gettimeofday` 这类高频调用不用真陷入内核)。再往上就是内核空间——用户态碰一下就 segfault,Meltdown 之后 KPTI 还把它在用户态页表里基本清空了,代价是 syscall 慢 5-30%(06 篇讲过)。

## 七、名场面:Java 进程的真实内存

现在可以回答开头运维的质问了。`-Xmx 4g` 的 Java 进程 RES 6GB,**因为 -Xmx 只管那一段 mmap 出来的 Java 堆,而进程地图上还有一堆别的住户**:

```
JVM 进程的内存构成:
┌────────────────────────────┐
│ Java 堆(-Xmx 管的就这个)   │ ← 4GB
├────────────────────────────┤
│ Metaspace(类元数据)        │ ← 默认无上限!
│ Code Cache(JIT 编译产物)   │ ← 几百 MB
│ Direct Buffer(NIO 堆外)    │ ← Netty 大户,最容易爆
│ 线程栈(默认 1MB × 线程数)   │ ← 1000 线程 = 1GB
│ JNI / Native 库内存         │
│ GC 元数据(card table 等)   │
│ 共享库(.so)                │
└────────────────────────────┘
```

总和到 -Xmx 的 1.5-3 倍很正常。容器里这就是命案现场:**limit 设成 -Xmx 一样大 → 非堆部分一长 → cgroup OOMKilled → 管理员看着"堆明明没满"一脸懵**。

对策是给每一块都上限,并给非堆留余量:

```bash
-Xmx4g
-XX:MaxMetaspaceSize=256m
-XX:ReservedCodeCacheSize=256m
-XX:MaxDirectMemorySize=1g
-Xss256k                          # 线程多的服务调小单栈
```

容器里更推荐 `-XX:MaxRAMPercentage=70.0`——让 JVM 按容器内存自适应,**总和给非堆留 30%**。(老版 JVM 还看不到 cgroup 限制,会按宿主机内存算堆,Java 10+ 的 `UseContainerSupport` 修了这个,详见 09 篇。)

## 八、排查工具箱

| 工具 | 看什么 |
| --- | --- |
| `pmap -x <pid>` | 每段映射的大小 / RSS / Dirty,最直观的"哪段占多少" |
| `cat /proc/<pid>/smaps` | 每段的 PSS / Swap / Anonymous 等明细 |
| `ps -o pid,vsz,rss,cmd` | VSZ(虚拟总量)和 RSS(物理占用)速查 |
| `jcmd <pid> VM.native_memory summary` | Java 各内存区明细(启动需加 `-XX:NativeMemoryTracking=detail`) |

再加两个相关机制:

**OOM Killer 选谁杀**:`cat /proc/<pid>/oom_score`,分数 ≈ RSS 占系统内存比例 + `oom_score_adj`(-1000 永不杀 ~ 1000 优先杀)。K8s 里这个值由 Pod QoS 决定——**BestEffort / Burstable 优先被杀,Guaranteed 最稳**。

**ASan(AddressSanitizer)**:查内存 bug 的神器,`gcc -fsanitize=address -g` 重编译后,栈溢出、堆越界、use-after-free、double free、泄漏全能抓现行。代价是慢 2-3 倍、内存翻 2-3 倍——**测试环境必开,生产必关**。

## 踩坑提醒

1. **改字符串字面量**——`char *p = "x"; p[0] = 'y';` 必 segfault,字面量在只读的 .rodata
2. **以为 free 把内存还给系统**——glibc 只放回自己的池子,RSS 不降是常态,不等于泄漏
3. **线程随便开**——每条默认 8MB 栈(JVM 默认 1MB),千级线程虚拟内存就是 GB 级
4. **递归不设出口 / 栈上开大数组**——8MB 限额一撞就是 stack overflow,大缓冲区放堆上
5. **把 -Xmx 当 Java 进程总内存**——真实占用是它的 1.5-3 倍,Metaspace / Direct Buffer / 线程栈都在堆外
6. **容器 limit 设得和 -Xmx 一样大**——非堆没活路,必被 OOMKilled;用 MaxRAMPercentage=70 留余量
7. **堆外(Native)泄漏没人查**——Java 堆 dump 干干净净,但 JNI / Direct Buffer 在悄悄漏,要靠 NativeMemoryTracking
8. **不调 oom_score_adj**——OOM Killer 专杀内存大户,而内存最大的往往是你最关键的服务
9. **看 RSS 不看 PSS**——共享库被每个进程重复计算
10. **mmap 大文件用完不 munmap**——虚拟地址空间也是会耗尽的资源
11. **生产环境开 ASan**——慢 2-3 倍,它是测试工具不是监控工具
12. **以为调试要关 ASLR**——现代调试器自动处理地址随机化,不用关
