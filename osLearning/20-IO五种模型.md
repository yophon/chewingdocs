# IO 五种模型

IO 模型回答一个问题:当数据还没准备好时,线程怎么办。阻塞 IO 让线程睡,非阻塞 IO 立刻返回,多路复用让一个线程等多个 fd,信号驱动 IO 用信号通知,异步 IO 则让内核完成后告诉你结果。很多人把"同步/异步"和"阻塞/非阻塞"混着用,导致理解 epoll、io_uring、Node、Nginx、Go netpoller 时全乱。这一篇把五种模型一次讲清楚。

> 一句话先记住:**阻塞/非阻塞说的是调用会不会卡住线程,同步/异步说的是数据拷贝由谁完成**。epoll 是同步 IO 多路复用:它只告诉你 fd 就绪,真正 `read/write` 还得你自己做。io_uring 更接近异步 IO:你提交操作,完成后拿结果。**高性能网络的主流不是"一个连接一个线程",而是非阻塞 fd + IO 多路复用 + 事件循环**。

---

## 一、一次 IO 的两个阶段

以 socket read 为例:

```
阶段 1: 等数据到达内核缓冲区
阶段 2: 从内核缓冲区拷贝到用户 buffer
```

不同 IO 模型,差别就在这两阶段谁等、怎么等。

---

## 二、阻塞 IO

```c
n = read(fd, buf, 4096);
```

没数据时线程睡眠:

```
用户线程 read
  → 内核:没数据
  → 线程睡眠
  → 数据到达,中断唤醒
  → 内核拷贝数据
  → read 返回
```

优点:

- 编程简单
- 适合少量连接

缺点:

- 一个连接占一个线程/进程
- 高并发下内存和调度成本高

---

## 三、非阻塞 IO

```c
fcntl(fd, F_SETFL, O_NONBLOCK);
n = read(fd, buf, 4096);
if (n < 0 && errno == EAGAIN) {
    // 没数据,下次再试
}
```

没数据立即返回 `EAGAIN`。

优点:

- 线程不被卡死
- 是事件循环的基础

缺点:

- 如果自己 while 轮询,CPU 空转

错误写法:

```c
while (read(fd, buf, n) == -1 && errno == EAGAIN) {
    // 空转烧 CPU
}
```

所以非阻塞通常要配合 epoll/kqueue。

---

## 四、IO 多路复用

一个线程等多个 fd:

```c
epoll_wait(epfd, events, 64, -1);
```

流程:

```
注册多个 fd
epoll_wait 睡眠
任意 fd 就绪 → 唤醒
用户态 read/write
```

select/poll/epoll 都属于这个模型。

| API | 问题 |
| --- | --- |
| select | fd 上限 1024,每次全量拷贝扫描 |
| poll | 无 1024 限制,但仍全量扫描 |
| epoll | 内核维护集合,只返回就绪 fd |

**注意**:epoll 不是异步 IO。它只通知"可以读/写",数据拷贝仍发生在你调用 `read/write` 时。

---

## 五、信号驱动 IO

```c
fcntl(fd, F_SETOWN, getpid());
fcntl(fd, F_SETFL, O_ASYNC | O_NONBLOCK);
```

数据就绪时内核发 `SIGIO`。

优点:

- 线程不用阻塞等

缺点:

- 信号语义复杂
- 可重入和异步信号安全限制多
- 工程上很少用于通用网络服务器

现代 Linux 更常用 epoll/signalfd,很少直接用 SIGIO 做网络主模型。

---

## 六、异步 IO

异步 IO 的目标:

```
用户提交 read 请求
内核等待数据 + 拷贝数据
完成后通知用户
用户拿到结果
```

Linux 历史上的 POSIX AIO 对普通文件支持有限,网络也不理想。真正改变局面的是 io_uring:

```c
io_uring_prep_read(sqe, fd, buf, len, offset);
io_uring_submit(&ring);
io_uring_wait_cqe(&ring, &cqe);
```

io_uring 把文件、socket、timeout、accept、send 等都统一成提交队列/完成队列。

---

## 七、同步/异步 vs 阻塞/非阻塞

| 模型 | 等数据 | 拷贝数据 | 典型 API |
| --- | --- | --- | --- |
| 阻塞 IO | 内核里睡 | 调用线程做 | blocking read |
| 非阻塞 IO | 立即返回 | 调用线程做 | O_NONBLOCK read |
| 多路复用 | epoll_wait 等 | 调用线程做 | epoll + read |
| 信号驱动 | 信号通知 | 调用线程做 | SIGIO |
| 异步 IO | 内核处理 | 内核完成 | io_uring/AIO |

重点:

- epoll 是同步非阻塞体系的一部分
- async/await 是语言抽象,底层可能是 epoll,不一定是内核异步 IO
- Go 网络 IO 看起来阻塞,底层是 nonblocking fd + netpoller

---

## 八、Reactor 与 Proactor

### 8.1 Reactor

```
事件循环等"就绪"
就绪后用户态处理 read/write
```

epoll/Nginx/Redis/Node 大多是 Reactor。

### 8.2 Proactor

```
提交操作
内核/运行时完成操作
回调处理结果
```

Windows IOCP、io_uring 更接近 Proactor。

---

## 九、模型选择

| 场景 | 推荐 |
| --- | --- |
| 简单脚本/少连接 | 阻塞 IO |
| 高并发网络 | 非阻塞 + epoll/kqueue |
| 文件 + 网络混合高性能 | io_uring |
| 语言 runtime | 封装 epoll/io_uring,暴露 async/协程 |
| 信号通知 | signalfd 或事件循环集成 |

---

## 十、常见坑

1. **非阻塞 fd 忘处理 EAGAIN**:把正常状态当错误。
2. **epoll ET 不读到 EAGAIN**:数据卡住。
3. **一直监听 EPOLLOUT**:CPU 空转。
4. **async 函数里调用阻塞 IO**:事件循环被卡死。
5. **以为 epoll 会帮你读数据**:它只告诉你就绪。
6. **文件 fd 用 epoll**:普通文件通常总是 ready,不适合 epoll 等待。

> **结论**:IO 模型的核心不是术语,而是"等待发生在哪里"。高性能服务的基本盘是:fd 非阻塞、等待交给 epoll/io_uring、业务层必须处理短读短写和背压。
