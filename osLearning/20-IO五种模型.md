# IO 五种模型

"同步、异步、阻塞、非阻塞"这四个词,大概是后端圈被用得最乱的四个词。有人说"epoll 是异步 IO"(不是),有人说"async/await 所以底层是异步 IO"(不一定),有人说"非阻塞就是异步"(完全两码事)。词一乱,你读 Nginx、Node、Redis、Go netpoller 的任何资料都像在看加密通信。其实这五种模型回答的是同一个朴素问题:**`read` 的时候数据还没到,线程怎么办?**——把这一个问题想清楚,四个词自动归位。

> 一句话先记住:**阻塞/非阻塞说的是"调用会不会卡住线程",同步/异步说的是"数据拷贝由谁完成"**。epoll 是同步 IO 多路复用——它只告诉你 fd 就绪了,真正的 read/write 还得你自己做;io_uring 才接近真异步——你提交操作,内核连等带拷全包,完成后给你结果。高性能网络的主流答案是:**非阻塞 fd + IO 多路复用 + 事件循环**,不是"一个连接一个线程"。

---

## 一、一次 IO 的两个阶段

所有模型的差别,都藏在这张图里。以 socket read 为例:

```
            阶段 1                          阶段 2
   ┌──────────────────────┐    ┌──────────────────────────┐
   │ 等数据到达内核缓冲区     │    │ 把数据从内核缓冲区          │
   │ (网络包还在路上,        │ →  │ 拷贝到你的用户态 buffer     │
   │  谁知道要等多久)        │    │ (这一步是确定的 CPU 工作)  │
   └──────────────────────┘    └──────────────────────────┘
```

阶段 1 是不确定的等待——对端可能一秒后发数据,可能永远不发;阶段 2 是确定的搬运工作。**五种 IO 模型,本质就是对"这两个阶段分别由谁负责、用什么方式等"的五种排列组合**。带着这个框架往下看,每种模型一句话就能说清。

---

## 二、阻塞 IO:最老实的等法

```c
n = read(fd, buf, 4096);   // 没数据?那我睡了
```

```
你的线程: read ──→ [睡眠 zzz...........] ──→ 拷贝数据 ──→ 返回
                   ↑                    ↑
                   内核:没数据,挂起你    数据到了,中断唤醒你
```

线程在阶段 1 直接睡掉,数据到了被内核唤醒,自己做阶段 2。注意一个常见误解:**阻塞期间线程不占 CPU**——它在睡眠队列里,调度器根本不看它。阻塞的代价不是烧 CPU,是**这个线程在等待期间什么别的都干不了**。

所以它的优缺点都很直白:

- 优点:**编程模型最简单**,代码就是顺序逻辑,适合脚本和少量连接
- 缺点:一个连接独占一个线程/进程;一万个连接就是一万个线程,每个线程几 MB 栈内存,加上上下文切换开销,**这就是经典的 C10K 问题**——机器没干活,光养线程就趴了

---

## 三、非阻塞 IO:我不等,我再来

```c
fcntl(fd, F_SETFL, O_NONBLOCK);

n = read(fd, buf, 4096);
if (n < 0 && errno == EAGAIN) {
    // 没数据——这不是错误!是"暂时没有,回头再来"
}
```

设了 `O_NONBLOCK` 之后,阶段 1 不等了:没数据立刻返回 `EAGAIN`。第一个要扭过来的认知是:**EAGAIN 不是 error,是一种正常状态**,新手把它当错误处理、直接断开连接,是非阻塞编程的第一跤。

但光有非阻塞还不够。如果你自己轮询:

```c
while (read(fd, buf, n) == -1 && errno == EAGAIN) {
    // 空转,CPU 100%,比阻塞还糟
}
```

线程是不卡了,CPU 被烧光了——**非阻塞解决了"卡住",制造了"该什么时候再来问"**。这个问题它自己答不了,要交给下一个模型。所以非阻塞 IO 几乎从不单独使用,它的真实身份是**事件循环的地基**:fd 先全部设成非阻塞,"什么时候该读"则交给多路复用器。

---

## 四、IO 多路复用:一个线程替一万个 fd 等

多路复用(multiplexing)的思路:既然每个 fd 单独等太浪费,**那就把一万个 fd 的"等"合并成一次系统调用**,谁有动静叫醒我,我再去处理谁。

```c
int n = epoll_wait(epfd, events, 64, -1);   // 一觉睡到任意 fd 就绪
for (int i = 0; i < n; i++) {
    // 就绪的 fd 自己去 read/write(阶段 2 还是你做)
}
```

```
       fd1  fd2  fd3 ... fd10000
        │    │    │         │
        └────┴────┴────┬────┘
                       ▼
                 epoll_wait(睡)
                       │  ← 任意 fd 就绪,唤醒
                       ▼
              对就绪 fd 逐个 read/write
```

select / poll / epoll 是同一模型的三代实现,差别在"内核怎么知道你关心哪些 fd":

| API | 机制 | 问题 |
| --- | --- | --- |
| select | 每次调用传全量 fd 集合,内核线性扫描 | **fd 上限 1024**,每次全量拷贝 + 两边各扫一遍 |
| poll | 同 select,换了数据结构 | 没有 1024 限制了,但还是全量拷贝全量扫 |
| epoll | **内核常驻保存关注集合,只返回就绪的 fd** | 基本没有,所以它赢了 |

一万连接里只有 1% 活跃时,select/poll 每次要扫一万个,epoll 只返回那 100 个——**这就是 epoll 快的全部原因:省掉了无效扫描,不是 read 本身变快了**(内部机制详见 21 篇)。

**划重点:epoll 不是异步 IO**。它把阶段 1 的"等"做到了极致高效,但只通知你"可以读了",阶段 2 的拷贝仍然发生在你调用 read 的那一刻、由你的线程完成。按定义,这是**同步 IO**。Nginx、Redis、Node 的底座全是它。

---

## 五、信号驱动 IO:理论很美,工程弃疗

```c
fcntl(fd, F_SETOWN, getpid());
fcntl(fd, F_SETFL, O_ASYNC | O_NONBLOCK);
// 数据就绪时,内核给你发 SIGIO 信号
```

思路是把阶段 1 的通知改成信号:线程该干嘛干嘛,数据到了内核发 `SIGIO`,你在信号处理函数里去读。听起来比 epoll 还潇洒,实际工程上几乎没人用,原因全在信号本身:

- 信号处理函数里能安全调用的函数极少(异步信号安全问题,详见 24 篇)
- 多个 fd 就绪时,SIGIO 不告诉你是哪个 fd,你还得自己查
- 信号可能合并、可能丢,语义又复杂又脆

现代 Linux 的态度很明确:**真想把信号纳入事件体系,用 signalfd 把信号变成 fd 塞进 epoll**(18 篇讲过)——让信号迁就事件循环,而不是让网络模型迁就信号。这个模型你需要认识它(五种模型凑数少不了它),但不需要用它。

---

## 六、异步 IO:连拷贝都不用你管

前四种模型,阶段 2 全是调用线程自己做。真正的异步 IO(asynchronous IO)把两个阶段全部外包:

```
你:提交一个 read 请求(带上我的 buffer)
内核:等数据 + 拷贝到你的 buffer,全程不打扰你
内核:完成了,通知你
你:直接拿结果用,数据已经躺在 buffer 里了
```

注意差别:epoll 通知你"**可以读了**"(你还要自己读),异步 IO 通知你"**已经读完了**"(结果就在那)。

Linux 在这条路上憋屈了很多年:POSIX AIO 实际是 glibc 用线程池模拟的,普通文件支持有限,socket 基本没法用,被嫌弃了二十年。真正翻盘的是 Linux 5.1 引入的 **io_uring**:

```c
io_uring_prep_read(sqe, fd, buf, len, offset);  // 把请求写进提交队列
io_uring_submit(&ring);                          // 提交(可以攒一批一起交)
io_uring_wait_cqe(&ring, &cqe);                  // 从完成队列收结果
```

io_uring 用两个用户态与内核共享的环形队列(提交队列 SQ / 完成队列 CQ)传递请求和结果,把 read、write、accept、send、timeout、甚至文件 IO **全部统一成"提交 → 完成"模型**,批量提交还能省系统调用。它是 Linux 第一个"文件和网络都好使"的异步 IO,细节留给 22 篇。

## 七、五种模型同框:一张时序图收尾

```
                 阶段1(等数据)              阶段2(拷贝)
阻塞 IO        ████████ 线程睡在 read 里 ████ │ ██ 线程拷贝 │
非阻塞 IO      ↻ 反复 read 得 EAGAIN(烧CPU)  │ ██ 线程拷贝 │
IO 多路复用    ████ 线程睡在 epoll_wait ████   │ ██ 线程拷贝 │ ← 但它替一万个 fd 一起等
信号驱动       线程干别的…(SIGIO!)            │ ██ 线程拷贝 │
异步 IO        线程干别的………………………………(完成通知!结果已就位)
                                              ↑ 拷贝是内核做的
```

| 模型 | 阶段 1 谁等 | 阶段 2 谁拷 | 分类 | 典型 API |
| --- | --- | --- | --- | --- |
| 阻塞 IO | 线程睡在调用里 | 调用线程 | 同步阻塞 | 默认 read |
| 非阻塞 IO | 线程反复来问 | 调用线程 | 同步非阻塞 | O_NONBLOCK + read |
| IO 多路复用 | 线程睡在 epoll_wait | 调用线程 | **同步**非阻塞 | epoll + read |
| 信号驱动 | 内核发信号通知 | 调用线程 | 同步 | SIGIO |
| 异步 IO | 内核 | **内核** | 异步 | io_uring / AIO |

看清表里那一列:**前四种的拷贝都是你做的,所以都是同步 IO;只有最后一种是异步**。"同步/异步"的分界线在阶段 2,"阻塞/非阻塞"的分界线在阶段 1——两组词描述两个不同的阶段,根本不该混用。

再澄清两个高频迷惑:

- **async/await 不等于异步 IO**。它是语言层的写法糖,底下大概率是 epoll(Node、Python asyncio 都是)——语法看着异步,IO 模型是同步多路复用。
- **Go 的网络读写"看着阻塞",其实不阻塞线程**。goroutine 里 `conn.Read()` 像阻塞调用,但 runtime 把 fd 设成了非阻塞,数据没到就把 goroutine 挂起、线程去跑别的 goroutine,netpoller(底层 epoll)就绪后再把它调回来。**阻塞的是协程,不是线程**——这是协程模型(17 篇)和 IO 模型在这里会师。

## 八、Reactor 与 Proactor:模型之上的架构名字

工程里你更常听到的是这两个词,它们是事件驱动架构的两种形态,和上面的模型一一对应:

**Reactor**(基于"就绪"通知):

```
事件循环 epoll_wait 等"谁就绪了"
就绪 → 回调里自己 read/write
```

epoll 体系的标准架构,Nginx、Redis、Node、Netty 都是 Reactor。

**Proactor**(基于"完成"通知):

```
提交操作 → 内核/运行时把活干完 → 回调里直接拿结果
```

Windows 的 IOCP 是老牌 Proactor,io_uring 让 Linux 也有了像样的 Proactor 底座。

记法:**Reactor 收到的是"可以干活了",Proactor 收到的是"活已经干完了"**。

## 九、选型:场景对答案

| 场景 | 推荐 | 理由 |
| --- | --- | --- |
| 脚本、工具、几个连接 | 阻塞 IO | 简单压倒一切,别过度设计 |
| 高并发网络服务 | 非阻塞 + epoll(Linux)/ kqueue(BSD/macOS) | 久经考验的主流 |
| 文件 + 网络混合的高性能场景 | io_uring | 唯一两边都管的方案 |
| 写语言 runtime / 框架 | 封装 epoll/io_uring,对外暴露 async 或协程 | 把复杂度藏在 runtime 里 |
| 要等信号 | signalfd 进事件循环 | 别碰裸 SIGIO |

还有一条容易踩的边界:**普通磁盘文件的 fd 不适合放进 epoll**——普通文件在 epoll 看来"永远就绪"(数据就在那,只是读起来慢),epoll 没法替你等磁盘 IO 完成。所以很多 runtime 是双轨制:网络走 epoll,文件 IO 丢线程池(或者上 io_uring 统一)。

## 踩坑提醒

1. **把 EAGAIN 当错误处理**——它是非阻塞 IO 的正常状态,意思是"暂时没有",直接报错断连是新手第一跤。
2. **"epoll 是异步 IO"**——它是同步 IO 多路复用,只通知就绪,数据还得你自己 read;异步的定义是拷贝由内核完成。
3. **以为 epoll 会帮你把数据读好**——epoll_wait 返回只代表"可以读",不调 read 数据永远在内核缓冲区躺着。
4. **ET 模式不读到 EAGAIN 就收手**——边缘触发只通知一次,剩下的数据没有下次通知,连接就此"卡死"(细节见 21 篇)。
5. **常驻监听 EPOLLOUT**——socket 大部分时间都可写,事件循环会被写事件打满,CPU 100%;只在写不完时才注册它。
6. **在 async 函数 / 事件循环里调阻塞 IO**——一个同步的数据库查询能把整个事件循环卡死,所有连接一起陪葬,这是 Node / asyncio 服务最经典的事故。
7. **拿普通文件 fd 喂 epoll**——普通文件总是"就绪",epoll 等不了磁盘;文件要么线程池,要么 io_uring。
8. **以为 async/await 在哪个语言里都是真异步 IO**——多数是 epoll 套皮;判断标准只有一个:数据拷贝是谁做的。
