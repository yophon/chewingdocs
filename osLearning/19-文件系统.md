# 文件系统

上一篇留了个钩子:write 返回成功,数据其实还在内存里飘着。这一篇就把"从 write 到磁盘"这条路走完。你会发现存储领域那些"玄学":为什么数据库压测 TPS 忽高忽低、为什么有人断电后配置文件变成 0 字节、为什么 Kafka 顺序写能跑满磁盘带宽而你的服务写几个小文件就 IO 打满——**全都不玄,全是文件系统在性能、空间和崩溃一致性(crash consistency)三者之间做的取舍**。看不见取舍的人觉得玄,看见了就只是工程。

> 一句话先记住:**文件系统 = 命名空间 + 元数据 + 数据块分配 + 崩溃一致性,前三个解决"怎么存",最后一个解决"断电不坏"**。普通 write 只进页缓存,fsync 才推到持久层;journal 不是为了快,是为了断电之后文件系统结构不散架。

---

## 一、块设备只会一件事:按号读写块

先看文件系统的"原材料"有多原始。磁盘 / SSD 作为块设备,对上只提供两个操作:

```
read  block 123
write block 456
```

没有文件名,没有目录,没有权限,没有"追加",没有大小的概念——就是一个巨大的、按块编号的字节数组。而你想要的是:

```
/var/log/app.log        ← 人类可读的路径
append 1KB              ← 往后加,别覆盖别人
rename conf.tmp → conf  ← 原子换名
chmod 0644              ← 权限
ls /var/log             ← 列目录
```

从"块数组"到"文件树"之间的全部落差,就是文件系统的工作清单:

- **路径 → inode 的映射**(目录就是一张名字表,18 篇讲过)
- **inode → 数据块的映射**(这个文件的第 N 个字节在哪个块上)
- **空闲空间管理**(哪些块没人用,分配时尽量连续)
- **权限与时间戳**
- **崩溃恢复**(写到一半断电,怎么不变成废墟)
- **缓存与预读**(和页缓存配合,让热数据不碰盘)

这份清单里最难、也最容易被程序员忽略的,是倒数第二项。先把读写路径走通,再回来收拾它。

## 二、一次读写的完整路径

### 2.1 读:热的是内存,冷的才是磁盘

```
read(fd)
  → VFS
  → 查页缓存
      命中:拷贝到用户态,结束(百 ns 级)
      未命中:
        文件系统把"文件内第 N 页"翻译成"设备上第几块"
        → 块层(合并、排序、调度)
        → 设备执行 IO(SSD 几十 µs,机械盘几 ms)
        → 数据填进页缓存
        → 拷贝到用户态
```

同一行 `read` 代码,命中和不命中差三四个数量级。**"这个接口平时 1ms,偶尔 200ms"的抖动,很多就是冷数据落到了真实磁盘上**。

### 2.2 写:全篇最重要的一张图

```
write(fd, buf, n)
   │
   ▼
┌─────────────────────┐
│ 页缓存(标记 dirty)   │ ← write 到这里就返回"成功"!
└─────────┬───────────┘
          │ 后台 writeback(秒级延迟,批量)
          ▼
┌─────────────────────┐
│ 文件系统(journal/分配)│ ← 决定块怎么写、先写什么
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│ 块层 + 设备缓存       │ ← 设备自己还有一层易失缓存
└─────────┬───────────┘
          ▼
┌─────────────────────┐
│ 持久介质              │ ← 到这里,断电才真的不丢
└─────────────────────┘
```

`write` 在第一层就返回了,所以它快;但**从第一层到最后一层之间的任何时刻断电,这笔数据都可能没了**。要把数据一路压到底,必须:

```c
write(fd, buf, n);
fsync(fd);          // 等数据穿过所有层、设备确认持久化,才返回
```

fsync 慢,慢得理直气壮——它等的是真实世界的物理写入。系统的 dirty 水位由 `vm.dirty_ratio` / `vm.dirty_background_ratio` 控制,排查 writeback 积压看:

```bash
cat /proc/meminfo | grep -E 'Dirty|Writeback'
sysctl vm.dirty_ratio vm.dirty_background_ratio
```

## 三、断电名场面与 journal

现在看最难的部分。"创建一个文件"在你眼里是一个动作,在文件系统眼里是**至少五次相互关联的写**:

```
1. 分配一个 inode
2. 分配数据块
3. 往父目录写目录项(名字 → inode)
4. 更新 inode 元数据(大小、块指针)
5. 更新空闲块位图
```

这五步不可能在一次磁盘写里完成。**如果写到第三步断电**,重启后的文件系统可能是:

- inode 分配了,但没有任何目录指向它(空间凭空消失)
- 目录项指向一个不存在的 inode(打开就报错)
- 位图说这块空闲,实际有文件在用(下次分配直接写穿别人的数据)

最后一种最恐怖——**不是丢一个文件,是文件系统结构性损坏,会持续扩散**。早年的解法是开机跑 `fsck` 全盘扫描修复,TB 级磁盘要扫几个小时。

journal(日志)的思路简单到漂亮,就是数据库 WAL 的同款:

```
1. 先把"我打算做哪些修改"完整写进一块专用日志区
2. 等日志落盘(这是一次顺序写,快)
3. 再去真正修改文件系统的各处结构
4. 崩溃重启后:日志完整 → 重放(replay);日志不完整 → 整条丢弃
```

要么全做,要么全不做——**单次操作的原子性,用"先声明后动手"换来了**。代价是元数据要写两遍(一遍日志一遍原地),所以说 journal 不是性能优化,是花性能买安全。

### ext4 的三档安全等级

journal 保护到什么程度,ext4 给你选:

| 模式 | 保护范围 | 速度 | 一句话 |
| --- | --- | --- | --- |
| `data=journal` | 数据 + 元数据都进日志 | 最慢 | 一切写两遍,最稳 |
| `data=ordered`(默认) | 元数据进日志,**数据先于元数据落盘** | 折中 | 结构不坏,内容自理 |
| `data=writeback` | 只有元数据进日志,数据顺序不保证 | 最快 | 崩溃后文件可能含旧垃圾数据 |

默认的 ordered 是个很工程化的折中:**文件系统自身结构保证不坏,但你的数据内容要不要持久,仍然归你的 fsync 管**。换句话说,journal 保护的是文件系统,不是你——这个分工必须想清楚。

## 四、fsync 为什么能慢成性能瓶颈

知道了下盘路径,fsync 的慢就能拆出来了:

- **设备延迟**——它必须等物理介质确认,机械盘一次寻道几 ms
- **journal commit**——你的 fsync 可能要先等一笔 journal 事务提交
- **写放大**——改 1 字节,可能要写整页、写日志、写元数据
- **flush 竞争**——多个文件同时 fsync,在设备队列里排队
- **云盘抖动**——云厂商的块存储后端延迟方差很大,p99 难看是常态
- **RAID / 设备缓存策略**——带电池缓存的 RAID 卡 fsync 飞快,没有的老老实实等盘

名场面:某团队自研存储,"每写一条记录 fsync 一次保证不丢",压测 TPS 只有 200——因为机械盘一秒就只能做这么多次持久化确认。**数据库性能工程的相当一部分,就是在和 fsync 搏斗**,套路也早已成熟:

```
每事务一次 fsync 太慢
  → group commit:攒一批事务,一次 fsync 摊薄成本
随机写数据页太慢
  → WAL 顺序写 + 落盘,数据页留在内存后台慢慢刷
```

MySQL 的 redo log、PostgreSQL 的 WAL、Kafka 的顺序追加,全是这一个思路的变奏:**把"必须持久"的部分变成顺序写,把随机写推到后台**。

## 五、原子更新配置:write + fsync + rename

另一个名场面:程序更新配置文件,直接打开原文件覆盖写,写到一半进程被 kill / 断电——重启后配置文件半新半旧,或者干脆 0 字节,服务起不来。

标准答案是个五步套路,值得背下来:

```c
fd = open("conf.tmp", O_CREAT|O_WRONLY|O_TRUNC, 0644);
write(fd, data, len);
fsync(fd);                    // 1. 新内容先持久化
close(fd);
rename("conf.tmp", "conf");   // 2. 原子换名
fsync(dirfd);                 // 3. 目录项更新也要落盘
```

它能成立靠的是一条文件系统承诺:**rename 在同一文件系统内是原子的**——任何时刻去读,要么拿到完整的旧文件,要么拿到完整的新文件,绝不会看到写了一半的中间态。

两个容易漏的细节:rename 之前必须 fsync 文件本身,否则可能出现"名字换过去了,内容还没落盘",断电后得到一个空的新文件;rename 之后还要 fsync 目录,因为"名字 → inode"这条目录项本身也是数据,也可能丢。**漏掉目录 fsync 的代码,平时一点问题没有,断电那天专门坑你**。

## 六、ext4 / XFS / btrfs:怎么选

| 维度 | ext4 | XFS | btrfs |
| --- | --- | --- | --- |
| 定位 | 通用默认 | 大文件大并发 | 功能丰富的 CoW |
| journal | 成熟稳定 | 元数据日志,扩展性强 | CoW 天然免 journal |
| 扩容 | 在线扩容可以,缩容支持有限 | 在线扩容成熟,**不能缩容** | 灵活 |
| 特色 | 均衡、省心 | 大目录、高并发元数据操作强 | 快照、校验和、子卷、压缩 |
| 默认采用 | Debian/Ubuntu 等 | RHEL/CentOS | openSUSE、Fedora(桌面) |

人话版:

- **ext4**:不知道选什么就选它。小文件、普通服务器表现均衡,坑都被踩平了。
- **XFS**:日志、对象存储、数据库数据盘、大容量磁盘——元数据并发和大文件是它的主场。记住它不能缩容,分区前想清楚。
- **btrfs**:CoW(copy-on-write,改数据不覆盖原地,写新块再切指针)带来快照、校验、子卷这些 ext4 给不了的能力,代价是运维复杂度和写放大。**数据库放 btrfs 要谨慎压测**——CoW 和数据库自己的 WAL 叠起来,写放大可能很难看。

## 七、小文件地狱

还有一种事故和"写得快不快"无关,和"文件有多少个"有关。某服务把用户上传按"一个对象一个文件"存,两年攒了八千万个 4KB 文件,然后:

```bash
df -h    # 磁盘容量还剩 40%
df -i    # inode 使用率 100%!
```

**空间没满,inode 先耗尽了**——新文件一个也建不了,报错还是误导性的 "No space left on device"。这是小文件海量堆积的众多症状之一:

- inode 数量逼近文件系统上限(格式化时就定死了)
- dentry / inode cache 内存压力大(用 `slabtop` 能看到这两项暴涨)
- 元数据 IO 远多于数据 IO,磁盘忙的全是"管理费"
- 每次创建文件都要 fsync 目录,成本线性上涨
- 备份、遍历、rsync 慢到怀疑人生

排查三件套:

```bash
df -i                          # inode 用量
find /path -type f | wc -l    # 数文件
slabtop                        # 看 dentry / inode_cache 内存
```

工业界的答案高度一致:**别让文件系统替你管海量小对象**。对象存储和日志系统都会把小文件合并进大文件(segment / SSTable),自己在应用层维护索引——LSM 树系的存储引擎本质上就是对"文件系统不擅长小文件"这个事实的绕行。

## 八、排查工具箱

```bash
df -h / df -i          # 空间 / inode
mount | column -t      # 挂载参数(journal 模式、noatime 等都在这)
findmnt                # 更清晰的挂载树
iostat -x 1            # 设备级:util%、await、吞吐
pidstat -d 1           # 进程级:谁在读写
filefrag -v file       # 文件碎片程度
stat file              # inode 号、链接数、时间戳一次看全
```

定位思路:`iostat` 看设备是不是真忙 → `pidstat -d` 找到是谁 → `strace` 看它在做什么样的 IO(大量小 write?疯狂 fsync?)→ 回到代码改写法。

## 踩坑提醒

1. **"write 成功 = 数据安全"**——进程崩溃数据不丢(已在内核),但机器断电就丢;持久化必须 fsync,没有例外。
2. **fsync 了文件,没 fsync 目录**——create / rename 产生的目录项更新也是数据;断电后文件内容在、名字没了,等于没在。
3. **直接覆盖写配置文件**——写一半挂掉就是半截文件;永远走 write temp + fsync + rename + fsync 目录的套路。
4. **以为 journal 保护你的数据**——默认 `data=ordered` 只保文件系统结构不坏,你的数据内容仍归 fsync 管,分工别搞混。
5. **海量小文件不当回事**——inode 耗尽时 `df -h` 还有空间,报错却说没空间;`df -i` 要进巡检清单。
6. **把 tmpfs 当持久化存储**——它是内存,重启即蒸发;只适合缓存和临时文件。
7. **NFS 当本地盘用**——延迟、缓存一致性、锁语义全都不同,本地盘上正确的代码到 NFS 上未必正确。
8. **数据库随手放上 CoW 文件系统**——CoW 写放大叠加数据库自己的 WAL,性能可能断崖;上线前必须压测。
