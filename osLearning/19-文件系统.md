# 文件系统

文件系统解决的是一个朴素但很难的问题:把一串字节可靠地放到块设备上,断电不乱,并发不坏,查找要快,空间别浪费。你看到的是 `/data/a.log`,内核看到的是 inode、目录项、页缓存、块分配器、journal、writeback、IO 调度。这一篇讲清楚:**ext4/XFS/btrfs 的取舍、一次写入如何落盘、journal 为什么存在、fsync 为什么慢、rename 为什么常用于原子更新**。

> 一句话先记住:**文件系统 = 命名空间 + 元数据 + 数据块分配 + 崩溃一致性**。普通 `write` 先写页缓存,后台再刷盘;`fsync` 才把数据推到持久层。**journal 不是为了更快,是为了断电后文件系统结构不坏**。数据库、日志、对象存储的很多"玄学性能",本质都是文件系统写入路径和 fsync 策略。

---

## 一、文件系统要解决什么

块设备只提供:

```
read block 123
write block 456
```

文件系统提供:

```
/var/log/app.log
append 1KB
rename temp → final
chmod 0644
list directory
```

它要维护:

- 路径到 inode 的映射
- inode 到数据块的映射
- 空闲空间管理
- 权限与时间戳
- 崩溃恢复
- 缓存与预读

---

## 二、一次读写路径

### 2.1 读

```
read(fd)
  → VFS
  → page cache 查页
      命中:拷贝到用户态
      未命中:文件系统算出块号
             块层发 IO
             磁盘/SSD 完成
             填 page cache
             拷贝到用户态
```

热文件读的是内存,冷文件读的是设备。

### 2.2 写

```
write(fd)
  → 拷贝到 page cache
  → 标记 dirty page
  → 返回
  → 后台 writeback 刷盘
```

所以 `write` 快,但不代表安全落盘。

```c
write(fd, buf, n);
fsync(fd);
```

`fsync` 慢,因为它要等设备确认数据持久化。

## 三、崩溃一致性与 journal

创建文件不是一次写:

```
1. 分配 inode
2. 分配数据块
3. 写目录项
4. 更新 inode 元数据
5. 更新空闲块位图
```

如果写到一半断电,文件系统可能出现:

- inode 已分配但目录没指向
- 目录指向不存在 inode
- 空闲块位图错误
- 文件大小和数据块不一致

journal 的思路:

```
先把"我要做哪些元数据修改"写进日志
日志落盘
再真正修改文件系统结构
崩溃后 replay journal
```

### 3.1 ext4 三种模式

| 模式 | 说明 |
| --- | --- |
| `data=journal` | 数据和元数据都 journal,最安全最慢 |
| `data=ordered` | 默认,数据先落盘,元数据 journal |
| `data=writeback` | 元数据 journal,数据顺序不保证 |

默认 ordered 是工程折中:文件系统结构安全,数据内容仍需应用自己 fsync。

## 四、ext4 / XFS / btrfs

### 4.1 ext4

特点:

- 通用、稳定、默认选择
- journal 成熟
- 小文件/普通服务器表现均衡
- 在线扩容可以,缩容支持有限

适合:大多数 Linux 业务机。

### 4.2 XFS

特点:

- 擅长大文件、大目录、高并发 IO
- 元数据扩展性强
- 在线扩容成熟,不能缩容
- RHEL/CentOS 常用默认

适合:日志、对象存储、数据库数据盘、大容量磁盘。

### 4.3 btrfs

特点:

- CoW(copy-on-write)
- 快照、校验、子卷、压缩
- 功能强,运维复杂度更高

适合:需要快照/校验/子卷的场景。生产数据库是否使用要谨慎压测。

## 五、原子更新:write temp + fsync + rename

配置文件更新常用套路:

```c
fd = open("conf.tmp", O_CREAT|O_WRONLY|O_TRUNC, 0644);
write(fd, data, len);
fsync(fd);
close(fd);
rename("conf.tmp", "conf");
fsync(dirfd);
```

为什么:

- `rename` 在同一文件系统内是原子的
- 读者要么看到旧文件,要么看到新文件
- 不会看到写一半的文件

注意最后 `fsync` 目录,保证目录项更新落盘。

## 六、fsync 为什么慢

`fsync` 要把 dirty page、相关元数据、journal 提交都推到持久层。

慢的原因:

- 设备延迟
- 写放大
- journal commit 等待
- 多个文件竞争 flush
- 云盘后端抖动
- RAID/缓存策略

数据库性能很大一部分就是在优化:

```
每事务 fsync 太慢 → group commit
日志顺序写 → 数据页后台刷
```

## 七、目录和小文件问题

小文件多的问题:

- inode 多
- dentry/inode cache 压力大
- 元数据 IO 多
- fsync 目录成本高
- 备份/遍历慢

排查:

```bash
df -i
find /path -type f | wc -l
slabtop   # 看 dentry/inode_cache
```

对象存储、日志系统通常会合并小文件,或改用 LSM/数据库格式。

---

## 八、常见坑

1. **只 write 不 fsync**:进程崩溃不丢,机器断电可能丢。
2. **fsync 文件不 fsync 目录**:rename/create 后目录项可能丢。
3. **小文件海量堆积**:inode 和 dentry cache 爆。
4. **把 tmpfs 当持久化**:重启就没。
5. **NFS 当本地盘用**:语义和延迟都不同。
6. **数据库上乱开 CoW**:可能写放大严重。

## 九、排查命令

```bash
df -h
df -i
mount | column -t
findmnt
iostat -x 1
pidstat -d 1
filefrag -v file
stat file
```

看 writeback:

```bash
cat /proc/meminfo | grep -E 'Dirty|Writeback'
sysctl vm.dirty_ratio vm.dirty_background_ratio
```

> **结论**:文件系统的核心不是"能存文件",而是"在性能、空间、崩溃一致性之间做取舍"。写高可靠存储代码时,`write`、`fsync`、`rename`、目录 `fsync` 的边界必须非常清楚。
