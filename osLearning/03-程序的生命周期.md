# 程序的生命周期

`gcc hello.c -o hello && ./hello`——看起来一行命令的事,背后是 OS 最长的一条流水线:预处理、编译、汇编、链接、装载、动态链接、初始化,最后才轮到你的 main()。大多数人对它的认知是"反正它能跑",直到某天翻车:链接器报 undefined reference、程序 scp 到另一台机器就缺 .so、GDB 里断点地址每次都不一样。这些问题全出在"main 之前"的世界里,**而 99% 的程序员从来没看过这个世界**。这一篇带你走完全程——理解了它,后面的内存布局、共享库、容器镜像、core dump 分析,全是这一篇的延伸。

> 一句话先记住:**程序的一生 = 编译期(源码 → ELF 文件)+ 运行期(execve → 装载 → 动态链接 → 初始化 → main)**。**ELF 是 Linux 世界"程序长什么样"的统一标准**——可执行文件、共享库、core dump 全是 ELF。搞懂 ELF 一种格式,三类文件全通。

---

## 一、全景:八步走

先把地图铺开,后面逐段放大:

```
源代码 (hello.c)
   ↓ 预处理 (cpp)
预处理后的源代码 (hello.i)
   ↓ 编译 (cc1)
汇编代码 (hello.s)
   ↓ 汇编 (as)
目标文件 (hello.o)
   ↓ 链接 (ld)
可执行文件 (hello, ELF 格式)
   ↓ execve syscall
内核装载到内存
   ↓ 转交动态链接器
动态库映射 + 符号解析
   ↓ 初始化(.init / 全局对象构造)
main() 被调用
   ↓
程序运行
   ↓ exit / _exit
进程结束
```

**前 4 步是编译期**(产物是硬盘上的一个 ELF 文件),**后 4 步是运行期**(把 ELF 变成内存里的活进程)。两个世界的交接点就是 `execve`——后面会细讲。

## 二、ELF:Linux 的"程序标准格式"

ELF(Executable and Linkable Format)是 Linux / BSD / Solaris 共用的标准。注意名字里那两个词——**Executable(可执行)和 Linkable(可链接)**,一种格式服务两个完全不同的阶段,这是 ELF 设计的精髓。三种文件都是 ELF:

```
.o    目标文件(可链接)
.so   共享库(可链接 + 可执行)
exec  可执行文件(可装载运行)
```

### 2.1 物理结构

```
┌──────────────────┐
│   ELF Header     │ ← 16 字节魔数 + 元信息
├──────────────────┤
│  Program Header  │ ← 装载视图(execve 看的)
│      Table       │
├──────────────────┤
│      .text       │ ← 代码段(机器码)
│      .rodata     │ ← 只读数据(字符串字面量)
│      .data       │ ← 已初始化的全局变量
│      .bss        │ ← 未初始化的全局变量(占位,文件里 0 字节)
│      .symtab     │ ← 符号表
│      .rel.text   │ ← 重定位信息
│      .debug_info │ ← 调试信息(-g 时才有)
├──────────────────┤
│  Section Header  │ ← 链接视图(ld 看的)
│      Table       │
└──────────────────┘
```

### 2.2 一份内容,两种视图:Section vs Segment

ELF 最有意思的设计是**同一份内容,给两个读者准备了两张目录**:

- **Section(节)= 链接视图**——给链接器 ld 看的,粒度细:`.text`、`.data`、`.bss`、`.symtab`……链接器要精确知道每一块是什么才能合并、重定位
- **Segment(段)= 装载视图**——给内核 / 装载器看的,粒度粗:几个 section 按权限分组打包

典型的 segment 划分:

```
LOAD (R-X):  .text + .rodata     ← 只读可执行
LOAD (RW-):  .data + .bss        ← 可读可写
DYNAMIC:     .dynamic             ← 动态链接信息
INTERP:      .interp              ← 动态链接器路径
```

为什么这么分?因为两个读者关心的事完全不同:**链接时关心细节,装载时只关心权限**——内核只需要知道"这一块映射成 RX、那一块映射成 RW",里面是代码还是字符串,它不在乎。

### 2.3 解剖 ELF 的工具箱

```bash
file hello                # 文件类型
readelf -h hello          # ELF Header
readelf -l hello          # Program Headers(segments)
readelf -S hello          # Section Headers
readelf -d hello          # 动态依赖
nm hello                  # 符号表
objdump -d hello          # 反汇编
strings hello             # 字符串
ldd hello                 # 动态库依赖树
```

这几个命令值得现在就敲一遍——**第一次亲眼看到自己程序的内脏,是建立心智模型最快的方式**。

## 三、链接:把碎片拼成完整程序

### 3.1 链接器在干什么

编译是按文件独立进行的——`main.o` 里调用了 `printf`,但 printf 的实现不在这个文件里,编译器只能留个"欠条"(未定义符号)。链接器的工作就是**收欠条、还欠条**:

```
1. 符号解析:把"引用 printf"对应到"libc 里的 printf 实现"
2. 重定位:计算每个函数 / 全局变量的最终地址,改写所有引用
3. 合并 section:多个 .o 的 .text 合并成一个 .text
4. 生成 ELF
```

链接错误就两种,全是欠条出了问题:**undefined reference**(欠条没人认——少链了库)、**multiple definition**(两个人都认领同一笔债——符号重复定义)。

### 3.2 静态链接:全打包带走

把所有依赖的 `.o` 和静态库 `.a` 全塞进一个可执行文件:

```bash
gcc -static hello.c -o hello
ls -lh hello
# -rwxr-xr-x 1 user user 1.2M hello   ← 1MB+,因为塞进了整个 libc
```

**Go 默认就是静态链接——这就是为什么 Go 二进制 10MB 起步,但部署是"扔过去就能跑"的极致简单**。

### 3.3 动态链接:运行时才借

`.so` 共享库不打进二进制,运行时才加载:

```bash
gcc hello.c -o hello
ls -lh hello
# -rwxr-xr-x 1 user user  16K hello    ← 只有 16KB

ldd hello
# linux-vdso.so.1
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
# /lib64/ld-linux-x86-64.so.2
```

两条路线的取舍:

| 维度 | 静态链接 | 动态链接 |
| --- | --- | --- |
| 文件大小 | 大(MB 级) | 小(KB 级) |
| 部署 | 扔过去就跑 | 目标机要有对应版本 .so,否则"DLL 地狱" |
| 内存 | 每个进程一份 libc | 多进程共享一份 |
| 库升级 | 重新链接 | 换 .so 即可(ABI 兼容前提下) |
| 启动 | 快 | 慢一点(动态链接耗时) |
| 代表 | Go | C/C++ 默认 |

## 四、动态链接的运行时细节

### 4.1 ld-linux.so:最神秘的程序

你以为 `./hello` 第一个跑的是 hello?错——**内核装载完 hello 后,跳转的是动态链接器 ld-linux.so 的入口**,由它加载所有 .so、解析完符号,才轮到你的程序:

```bash
./hello
# 等价于
/lib64/ld-linux-x86-64.so.2 ./hello
```

ld-linux.so 本身的身份很拧巴——**它是个动态库,但能直接执行**。它的职责:解析 `.dynamic` 段看需要哪些 .so → mmap 加载进进程地址空间 → 解析符号(printf 在 libc.so 里的实际地址)→ 重定位 GOT / PLT 表 → 跳转到程序入口。

### 4.2 GOT 和 PLT:动态链接的"间接表"

动态链接有个根本难题:**编译时不知道 printf 的最终地址**——每次运行地址都不同(ASLR,下面讲)。怎么 call 一个不知道地址的函数?答案是经典的间接层:**调用不走直达,走跳板**。

- **GOT(Global Offset Table)**:全局符号的真实地址表
- **PLT(Procedure Linkage Table)**:函数跳转的"跳板"代码

```
你的代码:call printf@plt
   ↓
PLT 跳板:跳到 GOT 里 printf 对应的格子
   ↓ 第一次调用:
       格子里还没有真地址 → 触发动态链接器解析 → 把真地址填回 GOT
   ↓ 后续调用:
       格子里有真地址 → 直接跳,几乎零开销
```

这个"第一次才解析"的机制叫**懒绑定(lazy binding)**——程序引用了 1000 个库函数,启动时一个都不解析,谁被调用了才解析谁。**启动速度和按需开销的权衡,被一张表优雅地解决了**。

### 4.3 LD_PRELOAD:运行时偷梁换柱

既然符号是运行时才解析的,就有操作空间——环境变量 `LD_PRELOAD` 让你的 .so **优先于系统库**被搜索:

```bash
# 我写了个 my_malloc.so,里面定义了 malloc()
LD_PRELOAD=./my_malloc.so ./hello
# hello 调用的 malloc 实际跑的是我的版本
```

正经用途:内存分析(jemalloc / tcmalloc 替换默认分配器)、性能埋点、安全审计。不正经用途也很多——**排查诡异行为时,记得看一眼环境变量里有没有 LD_PRELOAD**。

## 五、装载:execve 把 ELF 变成进程

### 5.1 execve 在干什么

```c
execve("/bin/hello", argv, envp);
```

内核做的事:

```
1. 读 ELF header,校验魔数
2. 清空当前进程的内存映射(原来的程序彻底没了)
3. 按 Program Header 用 mmap 把各 segment 映射进虚拟地址空间
4. 设置栈:argv / envp / auxv 推入栈
5. 找到 .interp 指定的动态链接器(/lib64/ld-linux-x86-64.so.2)
6. 跳转到动态链接器入口(注意:不是 main!)
```

注意第 3 步用的是 mmap——**装载不是把文件整个读进内存,而是建立映射**,真正的代码页是执行到才缺页加载的。这就是为什么巨大的程序也能"秒启动"。

### 5.2 ASLR:每次运行,地址全洗牌

每次运行,栈、堆、共享库的地址都不一样——这是 ASLR(地址空间随机化),目的是让攻击者无法预测地址来构造 buffer overflow 攻击:

```bash
./hello & cat /proc/$!/maps
# 7f8a3c000000-7f8a3c023000 r--p ...  /libc.so.6
# (再跑一次,地址全变)
```

ASLR 对调试不太友好——断点地址每次都漂。好消息是 **GDB 默认自动关掉 ASLR**(`set disable-randomization on` 是默认值),所以你在 GDB 里看到的地址是稳定的——也因此,**GDB 里能复现的地址,脱离 GDB 就不一定了**。

### 5.3 PIE:可执行文件自己也要能漂

ASLR 要随机化可执行文件本身的地址,前提是**这个文件能在任意地址运行**——这就是 PIE(Position Independent Executable)。现代 GCC 默认开启:

```bash
file hello
# ... ELF 64-bit LSB pie executable ...
```

老的 non-PIE 程序装载地址固定在 0x400000——攻击者闭着眼都知道你的代码在哪,不安全。

## 六、main 之前:你以为的入口不是入口

`main` 不是程序的真正入口——**`_start` 才是**(在 crt0.o 里,链接器自动塞进来的)。从动态链接完成到 main 之间,还有一整段路:

```
ld-linux.so 完成动态链接
   ↓ 跳到 _start
_start:
   设置好 argc, argv, envp
   调用 __libc_start_main
__libc_start_main:
   1. 调用 .preinit_array 里的函数
   2. 调用 .init_array 里的函数(C++ 全局对象构造、glibc 初始化)
   3. 调用 main(argc, argv, envp)
   4. main 返回后,调用 exit()
   5. exit 调用 .fini_array 里的函数(析构)
   6. 调用 _exit syscall 真正退出
```

### 6.1 C++ 全局对象:main 之前就跑了你的代码

```cpp
class Foo { Foo() { printf("foo!\n"); } };
Foo g_foo;       // 在 main 之前构造

int main() { return 0; }
```

输出 `foo!`——**g_foo 的构造函数在 main 之前已经执行**(就在上面的 .init_array 那一步)。

这里埋着 C++ 最著名的坑之一:**多个翻译单元(.o 文件)之间,全局对象的构造顺序是未定义的**。A.o 的全局对象构造时引用了 B.o 的全局对象,而后者可能还没构造——这就是「**static initialization order fiasco**」,出问题时症状是"程序还没进 main 就崩了",新手根本想不到去 main 之前找。

### 6.2 atexit:退出时的回调

```c
atexit(cleanup);  // main 返回 / 调 exit() 时,执行 cleanup
```

一个容易翻车的细节:**`exit()` 会触发 atexit 回调和析构,`_exit()` 不会**——后者直接 syscall 走人。日志没刷盘、文件没关干净,经常是有人调了 `_exit`(或者进程被信号杀了,同样不走 atexit)。

## 七、Hello World 到底加载了多少东西

```bash
ldd /bin/ls
# linux-vdso.so.1
# libselinux.so.1
# libc.so.6
# libpcre2-8.so.0
# /lib64/ld-linux-x86-64.so.2

cat /proc/$$/maps | wc -l
# 几十行 mmap 区域
```

最简单的程序也至少要加载:libc.so.6(C 标准库)、ld-linux.so(动态链接器)、vDSO(内核提供的快速 syscall,02 篇讲过)。**复杂程序(Java / Python)一启动加载 200+ 个 .so——这就是 Java 冷启动慢的一大原因**。

顺带一个进阶问题:两个 .so 都定义了 `init()` 怎么办?默认规则是**按加载顺序,第一个定义的赢**——大型程序里的定时炸弹。排查靠 `nm` 看符号、`ldd` 看加载顺序;根治靠符号版本控制(`.symver`)或加载隔离——**Python 就是用 `dlopen(..., RTLD_LOCAL)` 隔离每个 C 扩展模块的符号**,多个版本才能共存。

## 八、core dump:死后的"全身快照"

程序段错误时,内核可以把进程的内存 + 寄存器完整 dump 成一个文件:

```bash
ulimit -c unlimited       # 允许生成 core(很多发行版默认是 0!)
./crashy_program
# Segmentation fault (core dumped)

gdb crashy_program core
(gdb) bt                  # 看崩溃瞬间的调用栈
```

漂亮的闭环:**core dump 也是 ELF 格式**——程序生是 ELF,死也是 ELF,同一套工具(readelf / gdb)从头管到尾。

容器里的坑:core dump 的去向由**宿主机**的 `/proc/sys/kernel/core_pattern` 决定,常被 systemd-coredump 收走——容器里翻遍了也找不到 core,要去宿主机找。

## 九、容器镜像:程序生命周期的"打包发行版"

理解了这一篇,Docker 镜像就没有神秘感了——它本质上就是**把程序运行所需的一切打了个包**:程序本身(ELF)+ 依赖的 .so + 配置 / 数据文件 + 一个最小的 rootfs。

为什么 Alpine 镜像能做到几 MB?三招,全是本篇讲过的知识:musl 替代 glibc(小 5 倍)、静态链接(干脆去掉 .so 依赖)、删掉文档和 locale。**容器不是虚拟机,里面的程序还是 ELF 直接跑在宿主内核上**——详见 25 篇容器底层。

## 十、踩坑提醒

1. **以为 main 是入口**——_start 才是,main 之前已经跑了一堆初始化代码
2. **以为静态链接没缺点**——没法热升级库、二进制大、内存不共享
3. **部署失败不查 ldd**——程序到目标机跑不起来,八成是缺 .so
4. **glibc 版本不兼容**——新机器编译的二进制扔老机器,libc 版本太低直接挂
5. **假设 C++ 全局对象的构造顺序**——跨 .o 顺序未定义,fiasco 级别的 bug
6. **不知道 ASLR**——疑惑为什么地址每次都变,GDB 里稳定线上却不稳定
7. **动态链接器找不到符号**——启动报错先查 GOT / PLT / LD_LIBRARY_PATH
8. **LD_PRELOAD 滥用**——线上偷偷 hook,出问题排查极难
9. **不开 core dump**——线上崩了没现场,`ulimit -c unlimited` 提前配好
10. **以为 Hello World 简单**——背后几十个 syscall、几十个 .so
11. **发布前 strip 掉调试信息又不留符号文件**——出事时栈都看不懂
12. **以为容器是虚拟机**——容器只是 namespace,程序还是 ELF 直接跑
