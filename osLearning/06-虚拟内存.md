# 虚拟内存

你一定遇到过这些怪事:`malloc` 了 1GB,`top` 里 RES 却只涨了几 MB;Redis 做个 BGSAVE,内存占用突然翻倍把机器打挂;所有进程的内存加起来比物理内存还大,系统居然没崩。这些现象的根源是同一个东西——**你的程序从来没有摸过真实的物理内存,它操作的全是操作系统造出来的"假地址"**。不理解这层"假",你看 `top` 永远是雾里看花,排查 OOM 永远靠猜。这一篇就讲透这层最重要的幻觉:虚拟内存(Virtual Memory)。

> 一句话先记住:**虚拟内存 = 给每个进程发一本"假地址的账本"(页表),CPU 里的 MMU 负责实时把假地址翻译成真地址**。三个推论:**malloc 只记账不给钱**(物理页是访问时才分的)、**fork 只复制账本不复制钱**(写时复制 COW)、**翻译本身要靠 TLB 缓存加速**(TLB miss 是隐形性能杀手)。

---

## 一、没有虚拟内存的世界长什么样

回到 DOS 或早期嵌入式的年代,程序直接操作物理地址:

```
程序 A:往物理地址 0x1000 写自己的变量
程序 B:也往物理地址 0x1000 写
→ 互相覆盖,系统崩
```

这个世界有四个绝症:**没有隔离**(任何程序都能改别人的内存,一个 bug 全机陪葬)、**没法多任务**(两个程序的地址会撞)、**碎片致死**(想要一段大的连续物理内存可能永远凑不出来)、**天花板就是 RAM**(8MB 的机器跑不了 10MB 的程序,一个字节都不能多)。

虚拟内存把它们一次解决:**每个进程都看到一个独立的、连续的、巨大的(64 位下用户空间约 128TB)地址空间**,进程 A 的 `0x1234` 和进程 B 的 `0x1234` 是两个完全无关的地址,由 OS 映射到不同的物理页。

```
进程 A 的虚拟地址 0x1234 ──┐
                          MMU 查页表
进程 B 的虚拟地址 0x1234 ──┤
                          ↓
            物理内存:A → 0x9876_5000,B → 0x1111_2000
```

由此白送五个能力:**隔离**(写不到别人的内存)、**抽象**(程序眼里地址永远连续,物理上爱多碎多碎)、**超分配 overcommit**(虚拟地址可以远超物理内存,swap 兜底)、**共享**(多个进程映射到同一物理页,共享库只占一份)、**保护**(只读段、可执行段、内核区,权限页表说了算)。

## 二、地址翻译:MMU、页表和"为什么按页"

翻译这件事由 **MMU(Memory Management Unit)** 干——CPU 里的一个硬件单元,每次内存访问都过它的手。它查的那张映射表叫**页表(Page Table)**,由内核维护。

第一个设计决定:**按页(通常 4KB)映射,不按字节**——每个字节记一条映射的话,256TB 地址空间要 256 万亿条记录,光账本就把内存吃光了。

但 256TB / 4KB 仍是 640 亿个页,平铺的大表照样存不下。所以第二个决定:**多级页表——只为真正用到的地址范围建表**。x86_64 用 4 级,48 位虚拟地址这样拆:

```
[ 9 bits | 9 bits | 9 bits | 9 bits | 12 bits ]
  PML4     PDPT      PD       PT     页内偏移

翻译流程(MMU 硬件自动走):
1. 拿 PML4 的 9 位索引,查第一级表 → 得到 PDPT 表的物理地址
2. 拿 PDPT 索引查第二级 → PD 表地址
3. 拿 PD 索引查第三级 → PT 表地址
4. 拿 PT 索引查第四级 → 物理页框号
5. 拼上 12 位页内偏移 → 最终物理地址
```

注意到问题没有——**你访问一次内存,MMU 要先额外访问 4 次内存来查表**。这慢得没法接受,所以必须有缓存。

---

## 三、TLB:你看不见,但它决定你的程序快不快

**TLB(Translation Lookaside Buffer)** 是 MMU 内置的一个小硬件缓存,存最近用过的"虚拟页 → 物理页"映射:

```
TLB 命中:1 个 cycle 完成翻译
TLB miss:老老实实走 4 级页表,4-100 个 cycle
```

**每一次内存访问都要先过 TLB**——它的命中率直接决定程序的内存访问速度。而它小得可怜:

```
L1 TLB:64-128 条(每核独立)
L2 TLB:1500-2000 条
```

按 4KB 页算,**L1 TLB 只能覆盖 256-512KB 内存**。你的程序工作集随便就是几个 GB,在大堆上随机跳着访问,TLB miss 哗哗的——这就是"内存访问忽快忽慢"的第一个隐形来源。

### 解法:大页(Huge Pages)

既然条目数是硬件定死的,那就让**每条覆盖更多内存**——把页从 4KB 换成 2MB 甚至 1GB:

```
4KB 页:   64 条 × 4KB  = 覆盖 256KB
2MB 大页:  64 条 × 2MB  = 覆盖 128MB
1GB 大页:  16 条 × 1GB  = 覆盖 16GB
```

数据库(Oracle / Postgres)缓冲池、JVM 大堆(`-XX:+UseLargePages`)、Redis 大数据集、DPDK / SPDK 这类场景都靠它:

```bash
echo 1024 > /proc/sys/vm/nr_hugepages   # 预留 1024 个 2MB 大页 = 2GB

cat /sys/kernel/mm/transparent_hugepage/enabled
# always [madvise] never                ← THP:内核自动合并大页
```

听起来全是好处,其实 THP(Transparent Huge Pages)有著名反例——**MongoDB / Redis 官方都建议关掉它**,因为内核后台合并大页时会有停顿,延迟敏感的服务反而抖。**生产开大页之前必须压测**。

### 进程切换 = TLB 大失血

还有一个隐藏成本:进程切换时,虚拟地址还是那些虚拟地址,但映射全变了——**TLB 里的旧条目全部作废,几乎要清空重来**。这是上下文切换昂贵的主要原因之一(光换寄存器其实很快)。现代 CPU(Skylake+)用 **PCID(Process Context ID)** 缓解:给 TLB 条目打上进程 ID 标签,切换时不清空,只换当前生效的 ID。

## 四、缺页中断:不是错误,是机制

最颠覆直觉的一件事:**你 malloc / mmap 到的内存,大概率还不存在**。

```c
char *p = mmap(NULL, 1024*1024, PROT_READ|PROT_WRITE,
               MAP_PRIVATE|MAP_ANON, -1, 0);
// 此刻:只在页表里"预定"了 1MB 虚拟地址,物理内存一个字节没给

p[0] = 1;
// 此刻:MMU 查页表 → 这页没有物理映射 → 触发缺页中断(page fault)
// 内核:分配一个物理页、填好页表、回到这条指令重新执行
```

这就是开头那个困惑的答案——**`malloc(1GB)` 之后 RES 不涨,因为物理页是"用到哪页分哪页"**(demand paging,按需调页)。这不是偷懒,是刻意设计:绝大多数程序申请的内存远多于实际触碰的内存。

缺页分三种,严重程度天差地别:

| 类型 | 发生原因 | 代价 |
| --- | --- | --- |
| **次缺页**(Minor Fault) | 页在物理内存里,只是当前进程还没建映射(首次访问 / COW / 共享页) | 几微秒,纯 CPU |
| **主缺页**(Major Fault) | 页不在物理内存,要从磁盘读(swap 换回 / mmap 文件读入) | 微秒到毫秒,触发 IO |
| **段错误**(SIGSEGV) | 这个地址压根没有合法映射(空指针解引用之类) | 进程默认 dump core 退出 |

排查时直接看计数:

```bash
ps -o pid,majflt,minflt,cmd -p $$
#  PID MAJFLT MINFLT CMD
# 1234     12  56789 -bash
```

**majflt 高 = 进程在被 swap 折磨或频繁读 mmap 文件**——这是"程序莫名变慢"的经典信号。

## 五、写时复制(COW):fork 的免费午餐和它的账单

`fork()` 要复制父进程的整个地址空间——如果父进程占 10GB,难道要拷 10GB?当然不:

```c
pid_t p = fork();
// 看起来:复制了整个地址空间
// 实际上:只复制页表,父子共享全部物理页,且全部标成只读
// → 几毫秒搞定
```

这就是**写时复制(Copy-On-Write,COW)**。真正的复制被推迟到"有人写"的那一刻:

```
父或子写某一页 → 页是只读的,触发缺页中断
→ 内核发现这是 COW 页 → 复制一份新物理页给写入者(可写)
→ 回去继续执行,对程序完全透明
```

**只有真正被改的页才会复制**,99% 的页继续父子共享。fork、Redis BGSAVE(fork 子进程做 RDB 快照)、`mmap MAP_PRIVATE`(私有文件映射,改了不落盘)、glibc 的零页共享,全靠这一招。

但天下没有免费午餐,**COW 的账单是延迟支付的**。名场面——Redis BGSAVE 炸内存:

```
Redis 实例占 100GB,fork 子进程做 RDB
fork 本身几毫秒,看起来岁月静好
但主进程还在持续接收写入 → 每写一页就 COW 复制一页
→ 写入越猛,父子共享的页拆得越多
→ 内存峰值理论上能冲到 200GB → OOM Killer 出手,Redis 暴毙
```

对策:把 `vm.overcommit_memory` 设成 1 让内核乐观放行(Redis 官方要求),并预留足够内存余量或 swap。另外注意:**fork 大进程后如果要大量写,COW 缺页的累计开销可能比一次性复制还贵**——COW 优化的是"fork 后基本不写"的场景。

## 六、超分配、OOM Killer 和 swap:三位一体的兜底

既然虚拟地址只是记账,内核就面临一个哲学问题:**进程申请的虚拟内存总量超过物理内存时,批不批?**这由 `vm.overcommit_memory` 决定:

```
0  启发式(默认):看着办,明显过分的拒绝
1  从不拒绝:全批,出事再说(Redis 推荐)
2  严格模式:虚拟内存总量 ≤ swap + RAM × overcommit_ratio
```

模式 1 的"出事再说"就是 **OOM Killer**:物理内存真用光时,内核挑一个"罪过最大"的进程杀掉。评分看内存占用、子进程数,以及一个你能调的旋钮 `oom_score_adj`(-1000 到 1000):

```bash
echo -1000 > /proc/<pid>/oom_score_adj   # -1000 = 永不被杀
```

**生产必做**:数据库、网关这类关键进程调到 -1000,让 OOM Killer 去杀次要进程——否则它经常精准杀掉你最重要的那个(因为它内存最大)。

第三件套是 **swap**:内存不够时把冷页换到磁盘,腾出物理页。16GB 内存 + 32GB swap,进程理论上能用 48GB——代价是换出去的页再访问就是主缺页,毫秒级。**生产经验**:数据库服务器把 `vm.swappiness=1` 调到几乎不 swap(宁可干脆 OOM 也不接受 p99 飞天的慢性死亡);K8s 默认直接禁 swap。

## 七、top 里那四个数字到底什么意思

理解了上面这些,`top` / `ps` 里的数字终于能看懂了:

| 指标 | 含义 | 坑 |
| --- | --- | --- |
| **VSZ**(Virtual Size) | 虚拟地址空间总大小 | malloc 100GB 不访问,VSZ 涨 RSS 不涨——**VSZ 大不代表占内存** |
| **RSS**(Resident Set Size) | 实际占用的物理内存 | 共享页**每个进程都全额记一遍**,所有进程 RSS 加起来 > 物理内存 |
| **SHR**(Shared) | 其中和别人共享的部分(共享库 / 共享内存) | — |
| **PSS**(Proportional Set Size) | 共享页按比例分摊:N 个进程共享就每人记 1/N | **这才是公平的"我占多少"**,所有进程 PSS 之和 = 真实物理占用 |

```bash
cat /proc/<pid>/smaps | grep -i pss     # 看 PSS
```

记住这条排查铁律:**问"这个进程占多少内存",答案是 PSS,不是 RSS**。RSS 把 libc 这种几十个进程共享的库重复算了几十遍。

## 八、地址空间的另一半:内核

x86_64 的虚拟地址空间是这样劈开的:

```
0xFFFF_FFFF_FFFF_FFFF ┐
                      │ 内核空间(128TB)── 所有进程共享同一份映射
0xFFFF_8000_0000_0000 ┤
                      │ 非规范地址空洞(硬件不允许使用)
0x0000_7FFF_FFFF_FFFF ┤
                      │ 用户空间(128TB)── 每个进程独立
0x0000_0000_0000_0000 ┘
```

两个细节:64 位 CPU 实际只用 48 位地址,所以中间有巨大空洞,**可用的不是 16EB 而是 256TB**;**内核空间在所有进程的页表里都映射着同一份**——这就是 syscall 进内核不用切页表的原因,天生就快。

直到 2018 年 **Meltdown** 漏洞炸了:攻击者在用户态利用 CPU 乱序执行,能偷读映射在页表里的内核内存。补丁 **KPTI**(Kernel Page Table Isolation)在用户态运行时把内核映射基本清空,进内核再切完整页表——**代价是 syscall 慢 5-30%**,那年云上业务普遍莫名变慢就是它。配套的 **KASLR** 把内核加载地址随机化,加大攻击难度。

## 九、mmap 的三副面孔

页表映射机制还顺手送了三个大杀器:

```c
// 1. 文件映射:把文件直接"贴"进地址空间
int fd = open("data.bin", O_RDWR);
char *p = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
p[100] = 'X';     // 改内存 = 改文件(内核异步刷盘),省掉 read/write 拷贝

// 2. 匿名内存:不关联文件,纯内存
char *q = mmap(NULL, len, PROT_READ|PROT_WRITE,
               MAP_PRIVATE|MAP_ANON, -1, 0);
// glibc malloc 大块分配(>128KB)走的就是它,详见 08 篇

// 3. 进程间共享内存
char *shared = mmap(NULL, len, PROT_READ|PROT_WRITE,
                    MAP_SHARED|MAP_ANON, -1, 0);
fork();           // 父子读写 shared 看到同一份数据
// 无亲缘关系的进程之间:shm_open + mmap
```

大文件随机访问、共享内存 IPC、零拷贝,底层全是这套页表把戏。

## 十、排查第一站:/proc/\<pid\>/maps

进程的整张内存地图就在这个文件里:

```bash
cat /proc/$$/maps
```

```
55a3d5d3a000-55a3d5d44000 r--p ... /usr/bin/cat      ← 只读数据
55a3d5d44000-55a3d5d50000 r-xp ... /usr/bin/cat      ← 代码(只读可执行)
55a3d5d56000-55a3d5d57000 rw-p ... /usr/bin/cat      ← 可写数据
55a3d76d0000-55a3d76f1000 rw-p ... [heap]            ← 堆
7f9c4a800000-7f9c4a823000 r--p ... libc.so.6         ← 共享库
7ffe1a8e0000-7ffe1a901000 rw-p ... [stack]           ← 栈
7ffe1a984000-7ffe1a986000 r-xp ... [vdso]            ← 内核快速调用区
```

每行是一段映射:起止地址 + 权限 + 文件。**segfault 的地址落在哪个区、堆涨到多大、加载了哪些 .so,这里一目了然**——排查内存问题永远从这里开始。下一篇会把这张地图逐段讲透。

## 踩坑提醒

1. **以为 malloc 立刻分配物理内存**——只分虚拟地址,访问到哪页才分哪页,所以 RES 不涨是正常的
2. **把 RSS 当真实占用**——共享页重复计算,看 PSS 才公平
3. **不知道 TLB 的存在**——大工作集随机访问时 TLB miss 才是真凶,perf 都不一定第一眼看出来
4. **大内存应用不开大页**——数据库 / 大堆 JVM 白白损失性能;但 THP 要先压测,Redis / MongoDB 反而建议关
5. **fork 大进程不防 COW**——Redis BGSAVE 内存峰值翻倍触发 OOM,经典名场面
6. **swap 开着不管**——一旦开始 swap,主缺页打到磁盘,p99 直接飞天;数据库宁可 swappiness=1
7. **以为页永远是 4KB**——大页 / THP 会让你看到的行为完全不同
8. **以为 segfault 全是野指针**——也可能是栈溢出、写只读段(详见 07 篇)
9. **OOM 后不调 oom_score_adj**——关键服务被 OOM Killer 优先杀(它就爱杀内存大户)
10. **不看 /proc/\<pid\>/maps**——这是内存问题排查的第一站,跳过它全靠猜
11. **以为 64 位 = 16EB 可用**——实际只用 48 位地址,用户空间 128TB
12. **mmap 大文件用完不 munmap**——虚拟地址被占着,某些场景还会拖慢 fork
