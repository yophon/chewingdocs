# 虚拟内存

虚拟内存是**操作系统最伟大的发明之一**——没有它,你不可能有"我的 Java 进程独占 8GB 堆"这种幻觉,也不可能有"fork 一个进程"这种几乎免费的操作,更不可能有"32 位程序在 64 位系统上跑"这种向后兼容。**所有"内存怎么管"的问题最后都到虚拟内存这一层**——OOM、内存泄漏、性能抖动、`top` 里 RES 和 VIRT 的区别,根源全是它。这一篇讲清楚:**为什么要虚拟内存**、**页表怎么翻译地址**、**TLB 这个看不见的关键加速器**、**缺页中断的三种类型**,以及**为什么 fork 几乎不复制内存**(写时复制,COW)。

> 一句话先记住:**虚拟内存 = 给每个进程造一个"假的 8TB 地址空间",通过页表映射到物理内存(可能根本不存在)**。**进程操作的是虚拟地址,CPU 通过 MMU 实时翻译成物理地址**——这个翻译过程**靠 TLB 加速,TLB miss 是性能杀手**。**缺页中断不是错误,是机制**——80% 的内存分配都是"先给虚拟地址、用时再分物理页"。

## 一、为什么需要虚拟内存

### 1.1 没虚拟内存的世界(像 DOS / 早期嵌入式)

```
程序 A 直接访问物理地址 0x1000
程序 B 也访问物理地址 0x1000
→ 两个程序互相覆盖,系统崩
```

**问题**:

- **没有进程隔离**:一个程序能改另一个程序的内存
- **不能多任务**:无法同时跑多个程序
- **物理内存碎片**:大块连续内存不够时分配失败
- **内存大小受限于物理 RAM**

### 1.2 加上虚拟内存

每个进程都看到**自己独立的、连续的、巨大的(64 位下 256 TB)地址空间**:

```
进程 A 看到:
  0x0000_0000 - 0x7FFF_FFFF_FFFF (用户空间, ~128 TB)
  
进程 B 看到:
  0x0000_0000 - 0x7FFF_FFFF_FFFF (跟 A 完全独立)
  
两个进程的"同一个虚拟地址"对应不同的物理地址 —— 互不干扰
```

**核心价值**:

- **隔离**:进程 A 写 0x1234 不会影响进程 B 的 0x1234
- **抽象**:程序不用关心物理布局,逻辑上是连续的
- **超分配(overcommit)**:虚拟地址可以远超物理内存(swap 兜底)
- **共享**:多个进程能共享同一段物理内存(共享库 / 共享内存)
- **保护**:只读段、可执行段、用户 / 内核区,权限分明

## 二、地址翻译:虚拟 → 物理

### 2.1 MMU 和页表

**MMU(Memory Management Unit)** = CPU 里的硬件单元,负责把虚拟地址翻译成物理地址。

**页表(Page Table)** = OS 维护的"映射表",MMU 查它来翻译。

```
虚拟地址 0x4000_0000
   ↓ MMU 查页表
物理地址 0x9876_5000
```

### 2.2 为什么按"页"管理(不是按字节)

如果每个字节都映射,**256 TB / 1 字节 = 256 万亿个映射条目**——存映射表就把内存爆了。

**按页(通常 4 KB)管理**:

```
虚拟地址空间分成 4 KB 的"虚拟页"
物理内存分成 4 KB 的"物理页框"
映射的单位是页,不是字节

256 TB / 4 KB = 640 亿个页 —— 仍然太多,所以用多级页表
```

### 2.3 多级页表(x86_64 用 4 级)

直接放一张大表 → 内存爆。**多级页表只为"用到的"虚拟地址保存映射**:

```
虚拟地址(48 位)的拆分:
[ 9 bits | 9 bits | 9 bits | 9 bits | 12 bits ]
   PML4    PDPT     PD       PT       页内偏移

CPU 翻译流程:
1. 用 PML4 索引(9 bits)查第一级表 → PDPT 表的物理地址
2. 用 PDPT 索引(9 bits)查第二级表 → PD 表的物理地址
3. 用 PD 索引(9 bits)查第三级表 → PT 表的物理地址
4. 用 PT 索引(9 bits)查第四级表 → 物理页框号
5. 加上 12 bits 页内偏移 → 最终物理地址
```

**4 次内存访问翻译一次** —— 慢得令人发指。**所以需要 TLB 缓存**。

## 三、TLB:看不见的性能关键

### 3.1 TLB 是什么

**TLB(Translation Lookaside Buffer)** = MMU 内置的一个**小硬件缓存**,缓存最近用过的"虚拟页 → 物理页"映射。

```
TLB 命中:1 cycle 完成翻译
TLB miss:走完整 4 级页表,4-100 cycle
```

**所有内存访问都要先查 TLB** —— TLB 命中率决定性能。

### 3.2 TLB 大小

```
L1 TLB:64-128 个条目(每核独立)
L2 TLB:1500-2000 个条目
```

按 4KB 页算,**L1 TLB 能覆盖的内存只有 256-512 KB**。访问范围大于这个 → 大量 TLB miss。

### 3.3 大页(Huge Pages)

把页大小从 4 KB 改成 **2 MB / 1 GB**,**让一个 TLB 条目覆盖更多内存**。

```
4 KB 页:        TLB 64 entries × 4 KB = 256 KB
2 MB 大页:      TLB 64 entries × 2 MB = 128 MB
1 GB 大页:      TLB 16 entries × 1 GB = 16 GB
```

**用途**:

- 数据库(Oracle / Postgres)缓冲区
- JVM 大堆(`-XX:+UseLargePages`)
- Redis 大数据集
- DPDK / SPDK

```bash
# 配 2MB 大页
echo 1024 > /proc/sys/vm/nr_hugepages   # 分配 1024 个 2MB 大页 = 2 GB

# Transparent Huge Pages (THP) 自动用
cat /sys/kernel/mm/transparent_hugepage/enabled
# always [madvise] never
```

> THP 也有副作用——某些场景(MongoDB / Redis)反而变慢,因为合并大页有 STW。**生产前压测**。

### 3.4 进程切换 → TLB 失效

进程切换时,**整个 TLB 几乎要清空**(因为虚拟地址同样,但物理映射变了):

```
切换前:进程 A 的虚拟 0x1000 → 物理 0xABCD
切换后:进程 B 的虚拟 0x1000 → 物理 0xWXYZ
TLB 不能用 A 的旧条目了 → 失效
```

**TLB 失效是上下文切换慢的主要原因之一**。

**优化:PCID(Process Context ID)** 给每个进程一个 ID,TLB 条目带 ID,**切换时不全清,只换 ID** —— 现代 CPU(Skylake+)支持。

## 四、缺页中断:虚拟内存的核心机制

### 4.1 缺页是怎么发生的

```c
char *p = mmap(NULL, 1024*1024, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANON, -1, 0);
// mmap 只是"预定"了 1MB 虚拟地址,没真分物理内存

p[0] = 1;
// 访问 p[0] → MMU 查页表 → 这一页没映射物理页 → 触发缺页中断
// 内核处理:分配一个物理页,建立映射,继续执行
```

**新手常犯的错**:`malloc(1GB)` 后立刻 `top` 看,**RES 还是几 MB** —— 因为还没真分物理页。**只有访问到才分配**。

### 4.2 缺页的三种类型

#### A. 主缺页(Major Page Fault)

页不在物理内存,**要从磁盘读**:

```
访问 swap 出去的页:    要从 swap 分区读回
访问 mmap 文件的页:    要从文件读
```

**慢**(微秒到毫秒级,触发 IO)。

#### B. 次缺页(Minor Page Fault)

页在物理内存,**只是当前进程没建立映射**:

```
COW(写时复制)触发的 fault
共享页第一次访问
```

**快**(几微秒,纯 CPU 操作)。

#### C. 段错误(SIGSEGV)

访问的地址**根本没合法映射**:

```c
int *p = NULL;
*p = 1;     // 段错误
```

内核给进程发 SIGSEGV,默认行为是 dump core 退出。

### 4.3 看缺页:`top` / `ps`

```bash
ps -o pid,majflt,minflt,cmd -p $$
#  PID MAJFLT MINFLT CMD
# 1234     12  56789 -bash
```

- `majflt`:主缺页次数(累计)
- `minflt`:次缺页次数(累计)

**majflt 高 = 在 swap 或频繁读文件** —— 性能问题信号。

## 五、写时复制(Copy-On-Write,COW)

### 5.1 fork 的秘密

```c
pid_t p = fork();
// 看起来:复制父进程整个地址空间(可能几 GB)
// 实际:只复制页表,所有物理页共享(标记为 read-only)
// → fork 几毫秒完成
```

**COW 触发**:

```
父进程或子进程写某一页 → 触发缺页(因为只读)→
内核检查这是 COW 页 → 复制一份新物理页 →
让写入者用新页(可写)→ 继续执行
```

**只有真正修改的页才被复制** —— 99% 的页被父子共享,内存极省。

### 5.2 COW 的应用场景

- **fork**:Linux 的看家本领
- **Redis BGSAVE**:fork 子进程做 RDB,主进程继续写
- **mmap MAP_PRIVATE**:对文件的修改不影响原文件
- **glibc malloc**:小对象 / 共享对象用 COW

### 5.3 COW 的代价

```
fork 一个 10GB 内存的进程:
  fork 本身:几毫秒(只复制页表)
  
但如果父进程之后大量写入:
  每次写都触发 COW → 缺页中断 → 复制页
  累积起来可能比直接复制还慢
  
而且这一刻可能 OOM —— 父子原本共享的页都要变独立的
```

**Redis BGSAVE 的著名坑**:

```
Redis 实例 100GB 内存
fork 出子进程做 RDB
父进程继续接受写入 → 大量 COW → 内存峰值可能涨到 200GB
→ 触发 OOM Killer 杀 Redis
```

**对策**:配置 `vm.overcommit_memory = 1` 让内核乐观分配,或预留足够 swap。

## 六、超分配(Overcommit)

### 6.1 vm.overcommit_memory

```
0  启发式(默认):看着办,可能拒绝大分配
1  从不拒绝:全部允许(Redis 推荐)
2  严格:总虚拟内存 <= swap + RAM × overcommit_ratio
```

### 6.2 OOM Killer

物理内存真用光了 → 内核选择"罪过最大"的进程杀掉:

```
评分依据:
  - 内存占用大小
  - 子进程数
  - oom_score_adj(可调,-1000 = 永不被杀,1000 = 优先杀)

调整:
  echo -1000 > /proc/<pid>/oom_score_adj
```

**生产环境调优**:数据库 / 网关 等关键进程**调到 -1000**,让 OOM Killer 优先杀次要进程。

### 6.3 swap

物理内存不够时,**把不常用的页换到磁盘**(swap 分区):

```
内存 16GB,swap 32GB → 进程能用 48GB
代价:被换出的页再访问时触发主缺页 → 慢
```

**生产建议**:

- 数据库服务器:**关 swap 或调到极小**(`vm.swappiness=1`),宁可 OOM 也不接受 swap 抖动
- 容器:K8s 默认禁 swap

## 七、内存的"四个数字":VSZ / RSS / SHR / PSS

`top` / `ps` 看到的几个数字,新手最容易迷糊:

```
VSZ (Virtual Size):
  虚拟地址空间大小
  malloc 100GB 不访问 → VSZ 涨,RSS 不涨
  
RSS (Resident Set Size):
  实际占用的物理内存
  COW 共享的页算每个进程都占
  → 多个进程的 RSS 加起来可能 > 物理内存(重复算)
  
SHR (Shared):
  和其他进程共享的物理内存
  共享库 / 共享内存
  
PSS (Proportional Set Size):
  按"共享比例"分摊
  共享页被 N 个进程共享 → 每个进程算 1/N
  → 所有进程 PSS 加起来 = 实际物理内存使用
```

### 7.1 为什么要看 PSS

```bash
cat /proc/<pid>/smaps | grep -i pss
```

**真正的"我占多少内存" = PSS**,不是 RSS。RSS 会重复计算共享页。

## 八、用户空间 vs 内核空间

x86_64 的虚拟地址布局:

```
0x0000_0000_0000_0000 - 0x0000_7FFF_FFFF_FFFF  用户空间(128 TB)
0xFFFF_8000_0000_0000 - 0xFFFF_FFFF_FFFF_FFFF  内核空间(128 TB)

中间巨大的"非规范地址"区域不可用 —— 这是 x86_64 的硬件限制
```

**每个进程的用户空间独立,但内核空间所有进程共享**——这就是为什么 syscall 进入内核不需要切页表(整个虚拟地址映射就一份内核空间)。

### 8.1 KASLR 和 Meltdown

- **KASLR**(Kernel ASLR):内核空间地址也随机化,防 ROP 攻击
- **Meltdown 漏洞**:用户态利用乱序执行能读内核内存——所有 OS 紧急打补丁(KPTI:把内核映射在用户态页表里完全清空)

KPTI 的副作用:**syscall 慢了 5-30%** —— 这就是 2018 年云上业务普遍变慢的原因。

## 九、文件映射和共享内存

### 9.1 mmap 文件

```c
int fd = open("data.bin", O_RDWR);
char *p = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
p[100] = 'X';        // 改内存 = 改文件(异步刷盘)
munmap(p, len);
```

**优势**:

- 不用 read/write,直接当数组用
- 多进程 mmap 同一文件 → 共享内存(MAP_SHARED)
- 大文件随机访问比 read 快(避免 buffer 拷贝)

### 9.2 匿名内存(MAP_ANON)

```c
char *p = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANON, -1, 0);
// 不关联文件,纯虚拟内存
// glibc 大内存分配(>128KB)用这个,不用 brk
```

### 9.3 进程间共享内存

```c
// 父进程
char *shared = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_SHARED|MAP_ANON, -1, 0);
fork();
// 父子都能读写 shared,看到同一份数据

// 不相关进程之间共享:用 shm_open / mmap 文件
```

## 十一、踩坑提醒

1. **以为 malloc 立刻分物理内存** —— 不是,只分虚拟地址,访问时才分物理页
2. **看 RSS 当真实内存** —— 共享页重复算,看 PSS 才准
3. **TLB 失效不知道** —— 大堆 / 频繁切换进程时性能崩
4. **不开大页** —— 数据库 / 大内存 Java 应用性能损失
5. **fork 大进程不防 COW 内存炸** —— Redis BGSAVE OOM 经典坑
6. **swap 开着不调小** —— 一旦 swap 起来 p99 飞天
7. **以为 4KB 页是定的** —— 大页 / 透明大页让你看到的世界不一样
8. **以为 segfault 全是程序员错** —— 也可能是栈溢出 / 写只读段
9. **mmap 大文件不释放** —— 占用虚拟地址,某些场景影响 fork
10. **不看 /proc/maps** —— 排查内存问题的第一站
11. **OOM 不调 oom_score_adj** —— 关键服务被随便杀
12. **以为 64 位系统就有 16 EB 可用** —— 实际上只用 48 位虚拟地址(256 TB)
