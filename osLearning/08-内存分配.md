# 内存分配

`malloc(8)` 看起来就 8 个字节,**实际上 glibc 干了一系列复杂的事**——查 thread cache、查 fastbin、查 smallbin、不行就加锁去 main arena 切一块、再不行就 brk 扩堆。**理解 malloc 内部,你才能解释为什么 free 后 RSS 不降、为什么多线程程序内存爆涨、为什么换 jemalloc 性能能涨 30%、为什么 Go 不用 malloc 自己写 allocator**。这一篇拆开 malloc 的"内脏"。

> 一句话先记住:**malloc = "用户态的小池子 + 内核的大块来源"** —— 用户态 allocator(glibc / jemalloc / tcmalloc / Go runtime)负责"切碎、复用、缓存",**内核负责"批发"原料**(brk / mmap)。**多线程性能瓶颈 95% 在 allocator 锁竞争**——这就是 jemalloc / tcmalloc 比 glibc 快的原因(per-thread cache)。**碎片化是终极敌人**——长期跑的程序"内存只涨不跌"几乎都源于碎片。

---

## 一、malloc 在哪一层

```
你的代码:    malloc(8)
              ↓
glibc:        ptmalloc(默认 allocator)
              内部用 brk / mmap 拿大块,切小块给你
              ↓
内核:         brk / mmap syscall,真正分物理页
              ↓
硬件:         物理 RAM
```

**malloc 不是 syscall**——它是**用户态库函数**。99% 情况下不进内核(从池子里切一块给你),只有池子不够才调 brk / mmap。

**这就是 malloc 比 syscall 快的原因**——大部分调用没出用户态。

## 二、glibc ptmalloc 的核心数据结构

### 2.1 arena:多线程的"分配舞台"

```
main arena:   主线程用,通过 brk 扩展
thread arena: 每个线程一个(或共享几个),通过 mmap 创建
```

**为什么要多 arena**:多线程分配时,**arena 是加锁单位**——单 arena 就是单锁,所有线程抢一把锁,**性能爆炸**。

```
默认 thread arena 数量 = CPU 核数 × 8(64 位)
```

### 2.2 chunk:malloc 返回的"块"

每次 malloc 返回的内存,**实际上前面有 16 字节的 header**:

```
┌────────────────────┐
│ prev_size  (8 字节) │ ← 前一个 free chunk 的大小(如果当前 free)
│ size       (8 字节) │ ← 当前 chunk 大小 + 标志位
├────────────────────┤
│ user data           │ ← malloc 返回的指针指这里
│ ...                 │
└────────────────────┘
```

`size` 的低 3 位是标志位:

- `PREV_INUSE`(P):前一个 chunk 是否在使用
- `IS_MMAPPED`(M):是否 mmap 出来的
- `NON_MAIN_ARENA`(A):是否非主 arena

**这就是为什么 malloc 实际开销 ≥ 16 字节** —— 你 `malloc(1)` 至少占 32 字节(header + 对齐)。

### 2.3 bins:回收的"分类盒"

free 的 chunk 不直接还给 OS,**放进 bins 里复用**:

```
fastbin:    极小(16-160 字节),按大小分 10 个桶,LIFO,无锁
smallbin:   小(<512 字节),按大小分 62 个桶,FIFO
largebin:   大(>=512 字节),按大小范围分桶,有序
unsorted bin: 临时缓冲,刚 free 的先扔这,下次 malloc 优先看
top chunk:  arena 顶端的大空闲块,用 brk 扩展时增长
```

**malloc 流程(简化)**:

```
1. 找最近 free 的 chunk(unsorted bin)
2. 没找到 → 找对应大小的 fastbin / smallbin
3. 都没 → 切一块 top chunk
4. top chunk 不够 → brk 扩堆(或 mmap 大块)
```

## 三、brk vs mmap:小块和大块的分界

### 3.1 brk:扩展堆顶

```c
// glibc 内部:
sbrk(0x100000);     // 把 brk 指针往上挪 1MB
```

**特征**:

- 只能扩,不能缩(理论可以,实际 glibc 不收缩)
- 只对当前进程的连续堆生效
- **快**(就改一个指针)

**用途**:小对象的池子,大量小 malloc。

### 3.2 mmap:独立的匿名映射

```c
mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANON, -1, 0);
```

**特征**:

- 任意大小,任意位置
- free 时(munmap)直接还 OS
- 不受堆"只涨不缩"的限制
- **慢一点**(每次 syscall)

### 3.3 临界点:M_MMAP_THRESHOLD

```
默认:128 KB
< 128 KB:走 brk(切 chunk)
≥ 128 KB:走 mmap(独立映射)

可调:mallopt(M_MMAP_THRESHOLD, 64*1024);
```

**为什么有这个临界**:

- 大块 free 后 brk 不能缩 → 浪费虚拟地址
- 大块用 mmap → free 时直接还 OS

## 四、多线程的痛:arena 锁竞争

### 4.1 单 arena 的灾难

```
线程 A:malloc → 等锁
线程 B:malloc → 等锁  
线程 C:malloc → 等锁
线程 D:malloc → 在 arena 里切块
...
```

100 线程同时 malloc → 99 个在等。**这是 glibc 在多线程场景慢的根因**。

### 4.2 多 arena 缓解

每个线程绑定一个 arena,**减少锁冲突**:

```
线程 1, 9, 17 → arena A
线程 2, 10, 18 → arena B
...
```

**仍有冲突**——同 arena 多线程时;**且每个 arena 独立池子,内存重复占用**。

### 4.3 jemalloc / tcmalloc 的胜利:per-thread cache

```
每个线程有自己的小 cache(几 MB)
小对象分配:从 cache 切,完全无锁
cache 不够才去公共池
```

**性能**:多线程小对象分配比 glibc 快 2-5 倍。

**用法**:

```bash
# Linux 替换 glibc malloc
LD_PRELOAD=/usr/lib/libjemalloc.so.2 ./your_program

# 或编译时链接
gcc -ljemalloc your.c
```

**用了 jemalloc 的明星项目**:Redis、Firefox、FreeBSD libc。
**用了 tcmalloc 的**:Google 内部、Envoy。

## 五、对齐:malloc 的隐性规则

### 5.1 默认对齐

```
malloc 返回的指针保证 16 字节对齐(64 位)
```

**为什么 16**:能放下任何标量(包括 long double / SIMD)。

### 5.2 大对齐的 API

```c
void *p = aligned_alloc(64, 1024);   // 64 字节对齐
posix_memalign(&p, 4096, 8192);      // 4KB 对齐(页对齐)
```

**用途**:

- SIMD 要求(SSE 16 字节 / AVX 32 字节 / AVX-512 64 字节)
- DMA / O_DIRECT 要 512 字节或页对齐
- 自己实现 cache line 对齐(防 false sharing)

### 5.3 对齐的代价

```c
malloc(1) 实际占 32 字节(header 16 + 数据 16,对齐到 16 字节)
```

**大量小对象内存浪费严重** —— 这是为什么池化(对象池 / arena)在性能场景常见。

## 六、free 不还内存:碎片化

### 6.1 现象

```c
for (int i = 0; i < 1000000; i++)
    p[i] = malloc(8);
for (int i = 0; i < 1000000; i++)
    free(p[i]);
// RSS 没怎么降
```

### 6.2 原因

```
小 chunk 留在 fastbin / smallbin 里,等下次 malloc 复用
brk 不会缩(top chunk 释放才有可能)
mmap 块倒是还了
```

**长期跑的进程内存"只涨不跌"是正常的** —— 不一定是泄漏。

### 6.3 真正的内存泄漏

**有内存被分配了但没指针 = 泄漏**。检测工具:

- **Valgrind**(慢 10 倍,但全)
- **AddressSanitizer**(快,需要重编)
- **jemalloc 的 prof**(生产可用)
- **gperftools heap profiler**

### 6.4 主动归还内存:malloc_trim

```c
malloc_trim(0);    // 试图把 top chunk 还给 OS
```

**有用但有限** —— 只对 main arena 有效,且 top chunk 必须连续空闲。

## 七、内存碎片化的实际危害

```
长期跑 24 小时:
  虚拟内存 8 GB
  RSS 6 GB
  实际有用对象只有 2 GB
  碎片占了 4 GB
```

**对策**:

- **重启**(最简单粗暴,K8s 滚动重启)
- **池化**(对象池 / arena allocator)
- **换 jemalloc**(碎片管理更好)
- **GC**(Go / Java 自动 compact)

**Redis 的特殊解法**:`activedefrag yes` —— 后台主动整理碎片(jemalloc 提供 API)。

## 八、Java / Go 的内存分配

### 8.1 Java:JVM 堆里自管

```
JVM 启动时 mmap 一大块(-Xmx 大小)
对象在堆里 bump pointer 分配(只增加 top 指针,极快)
GC 负责回收 + compact(消除碎片)
```

**每次 new Object() < 100 ns** —— 比 malloc 快得多(无 lookup,无 lock)。

### 8.2 Go:用自家的 mcache / mcentral / mheap

```
mcache:    每个 P(逻辑处理器)一个,小对象分配无锁
mcentral:  按 size class 分类的中心池
mheap:     全局 heap,大对象在这,从 OS mmap
```

**结构上和 tcmalloc 几乎一样**(Go 的 allocator 借鉴自 tcmalloc)。

### 8.3 为什么不用 glibc malloc

- GC 需要知道所有对象,malloc 是黑盒
- 无锁分配比 malloc 快 10 倍
- 自家 allocator 能控制布局,做压缩、移动

**这就是 Java / Go / .NET 等"托管语言"性能优势的来源之一**。

## 九、内存分配的常见反模式

### 9.1 频繁小对象 malloc

```c
for (int i = 0; i < 1000000; i++) {
    Node *n = malloc(sizeof(Node));   // 每次 malloc syscall? 不,但很慢
    list_add(&l, n);
}
```

**优化**:

- 预分配大数组
- 对象池(slab)
- arena allocator

### 9.2 短期大对象 + 内存爆涨

```c
char *buf = malloc(10*1024*1024);   // 10 MB
process(buf);
free(buf);                           // 还了,但 RSS 不降
```

**优化**:大块用 mmap(自动归还):

```c
char *buf = mmap(NULL, 10*1024*1024, ...);
munmap(buf, ...);
```

### 9.3 内存泄漏

```c
for (;;) {
    char *p = malloc(1024);
    // 忘了 free
}
```

**用 ASan / Valgrind 查**。

### 9.4 use-after-free

```c
char *p = malloc(8);
free(p);
p[0] = 1;        // 可能 OK,可能崩,看你运气
```

**最难查的 bug** —— 行为不确定,**ASan 是必备工具**。

## 十、glibc 的 tunables(调参)

```bash
# 调小 mmap 临界点(让更多分配走 mmap,free 立即归还)
export MALLOC_MMAP_THRESHOLD_=65536
export MALLOC_TRIM_THRESHOLD_=65536

# 限制 thread arena 数量(减少内存碎片)
export MALLOC_ARENA_MAX=2
```

**用途**:

- 长期跑的服务想"内存能还回去" → 调小 mmap threshold
- 多线程程序内存太散 → 限制 arena 数量

> **不要瞎调** —— 默认值是 glibc 团队几年迭代来的,99% 场景够用。**调之前先压测验证**。

## 十一、查 malloc 行为的工具

### 11.1 ltrace

```bash
ltrace -e malloc,free,realloc -c ./your_program
```

看 malloc / free 的调用次数和大小分布。

### 11.2 mtrace

```c
#include <mcheck.h>
int main() {
    mtrace();
    // ... your code
}
```

```bash
MALLOC_TRACE=/tmp/mtrace.log ./your_program
mtrace your_program /tmp/mtrace.log
```

### 11.3 jemalloc heap profiler

```bash
MALLOC_CONF="prof:true,prof_active:true,lg_prof_sample:19" \
LD_PRELOAD=/usr/lib/libjemalloc.so.2 \
./your_program
```

生成 heap dump,**生产可用**(开销小)。

## 十二、踩坑提醒

1. **以为 free 还内存** —— glibc 只放池子,RSS 不降
2. **多线程用默认 glibc malloc** —— 锁竞争,换 jemalloc / tcmalloc
3. **大量小 malloc 不池化** —— overhead 大,性能差
4. **不知道 chunk header 16 字节** —— `malloc(1)` 占 32 字节
5. **栈和堆混淆** —— 大数组放栈上爆栈
6. **Native 内存泄漏不查** —— Java 堆 OK 但 Native 漏没人懂
7. **以为 valgrind 没事就没事** —— Valgrind 运行慢、不全(线程 / 异常路径覆盖差)
8. **生产开 ASan** —— 慢 2-3 倍,只在测试用
9. **mmap 大量小区域** —— 每个 mmap 至少占一页(4KB),浪费
10. **不看碎片** —— 24 小时后 RSS 6GB 但有效数据 2GB
11. **以为 jemalloc 是银弹** —— 单线程场景 jemalloc 不一定快
12. **改 MALLOC_ARENA_MAX 没压测** —— 调坏了反而 OOM
