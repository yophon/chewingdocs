# 内存分配

`malloc(8)` 看起来天真无邪——要 8 个字节而已。但这一行背后,glibc 要查线程缓存、翻好几层空闲链表、可能还要抢一把锁,实在不行才去找内核批发内存。**不拆开这台机器,你解释不了一堆日常怪象**:为什么 free 了一堆内存 RSS 纹丝不动、为什么多线程程序的 malloc 慢得离谱、为什么 Redis 换上 jemalloc 就快了、为什么 Java / Go 干脆抛弃 malloc 自己造轮子。这一篇把分配器(allocator)的内脏摊开看。

> 一句话先记住:**malloc 是"用户态的零售商":从内核批发大块内存(brk / mmap),切碎了零售给你,你 free 的货它收回货架而不是退给内核**。三个推论:**RSS 不降是货架在囤货**;**多线程慢是大家挤同一个收银台(arena 锁)**,jemalloc / tcmalloc 快在给每个线程开了自助通道;**碎片是零售业的终极顽疾**,长期跑的服务"内存只涨不跌"多半是它。

---

## 一、malloc 根本不是系统调用

先纠正一个普遍误解:malloc 不是 syscall,**它是 glibc 里的一个用户态库函数**(默认实现叫 ptmalloc)。

```
你的代码:  malloc(8)
            ↓
glibc:     ptmalloc —— 从自己的池子里切一块给你(99% 的情况到此为止)
            ↓ 池子不够时才往下走
内核:      brk / mmap syscall,批发虚拟内存
            ↓ 访问时缺页中断才真正分物理页(06 篇)
硬件:      物理 RAM
```

**99% 的 malloc 调用根本不进内核**——这是它比 syscall 快的原因,也是理解后面所有行为的前提:你和内核之间隔着一个有自己库存策略的"中间商"。

---

## 二、中间商的账本:arena、chunk、bins

### 2.1 arena:分配的"舞台",也是锁的单位

```
main arena:    主线程用,靠 brk 扩展堆
thread arena:  其他线程用(通过 mmap 创建),默认上限 = CPU 核数 × 8(64 位)
```

为什么要多个 arena?因为 **arena 是加锁单位**——所有线程挤一个 arena 就是挤一把全局锁,多线程性能直接爆炸。多 arena 让线程分摊到不同的"舞台"上,减少踩踏。

### 2.2 chunk:每块内存都带 16 字节"价签"

malloc 返回给你的指针,前面藏着一个 header:

```
┌─────────────────────┐
│ prev_size(8 字节)   │ ← 前一个空闲 chunk 的大小
│ size    (8 字节)   │ ← 本 chunk 大小 + 3 个标志位
├─────────────────────┤
│ user data           │ ← malloc 返回的指针指向这里
└─────────────────────┘
```

size 的低 3 位塞了标志:`PREV_INUSE`(前一块是否在用)、`IS_MMAPPED`(是否 mmap 来的)、`NON_MAIN_ARENA`(是否非主 arena)。

直接推论:**`malloc(1)` 实际占约 32 字节**(16 字节 header + 数据区对齐到 16 字节)。海量小对象场景,一半以上的内存可能花在"价签"上——这是对象池 / arena allocator 在高性能代码里流行的根本原因。

### 2.3 bins:free 回来的货,分类上架

你 free 的 chunk 不还内核,按大小进不同的"分类货架"等复用:

```
fastbin:      极小(16-160 字节),10 个桶,LIFO,操作近乎无锁——最热的快速通道
smallbin:     < 512 字节,62 个按大小的桶,FIFO
largebin:     ≥ 512 字节,按大小范围分桶,桶内有序
unsorted bin: "刚退货的临时筐"——刚 free 的先扔这,下次 malloc 优先翻
top chunk:    arena 顶端的大空闲块,都不够时从它切,brk 扩堆就是扩它
```

malloc 一次的寻货流程(简化):

```
1. 翻 unsorted bin(刚 free 的最可能合适)
2. 没有 → 按大小查 fastbin / smallbin
3. 还没有 → 从 top chunk 上切一块
4. top chunk 也不够 → brk 扩堆,或大块直接 mmap
```

## 三、brk vs mmap:128KB 是条分界线

中间商找内核批发,有两条渠道,glibc 按大小自动选:

| 渠道 | brk(扩堆顶) | mmap(独立匿名映射) |
| --- | --- | --- |
| 适用 | 小对象(< 128KB) | 大对象(≥ 128KB) |
| 速度 | 快,挪个指针 | 慢一点,每次都是 syscall |
| 位置 | 堆的连续延伸 | mmap 区任意位置 |
| **free 之后** | **不还内核**(brk 实际不收缩) | **munmap 立即归还** |

分界线叫 `M_MMAP_THRESHOLD`,默认 128KB,可调:`mallopt(M_MMAP_THRESHOLD, 64*1024)`。

为什么这样分?小对象生灭频繁,池化复用避免反复 syscall;大对象如果也混进堆里,free 后会被"只涨不缩"的 brk 困住,白白占着——走 mmap 就能干净归还。**记住这条:大块内存走 mmap 的,free 是真还;走堆的,free 只是上架**。

## 四、多线程的痛:为什么 jemalloc / tcmalloc 能赢

### 4.1 灾难现场

单 arena 时代,100 个线程同时 malloc:

```
线程 1:malloc → 拿到锁,在切块
线程 2-100:malloc → 排队等锁
```

99 个线程在干等。**多线程服务 malloc 慢,95% 慢在这把锁上**,不在分配算法本身。

### 4.2 glibc 的缓解和它的极限

ptmalloc 用多 arena 把线程分摊开,确实好了很多,但:同 arena 的线程照样冲突;**每个 arena 各囤一份库存,内存占用成倍上去**——多线程 glibc 程序"内存虚胖"的来源之一。

### 4.3 per-thread cache:jemalloc / tcmalloc 的杀招

```
每个线程自带一个小缓存(几 MB)
小对象分配:从自己缓存里切,完全无锁,一条指令级别的快
缓存空了才去公共池补货(这时才碰锁)
```

效果:**多线程小对象分配比 glibc 快 2-5 倍**。用法零成本,不用改代码:

```bash
LD_PRELOAD=/usr/lib/libjemalloc.so.2 ./your_program
# 或编译时:gcc -ljemalloc your.c
```

名单很能说明问题:**jemalloc 的用户有 Redis、Firefox、FreeBSD(直接做了 libc 默认);tcmalloc 的用户有 Google 全家和 Envoy**。注意它不是银弹——单线程场景 jemalloc 未必比 glibc 快,赢的是多线程和碎片管理。

glibc 自己也有补救旋钮(环境变量,改前必须压测):

```bash
export MALLOC_ARENA_MAX=2             # 限制 arena 数,治"内存虚胖"
export MALLOC_MMAP_THRESHOLD_=65536   # 调小 mmap 分界,让更多内存能真还
export MALLOC_TRIM_THRESHOLD_=65536
```

## 五、对齐:看不见的隐性规则

malloc 返回的指针保证 **16 字节对齐**(64 位)——够放任何标量,包括 long double 和基础 SIMD。需要更大对齐时:

```c
void *p = aligned_alloc(64, 1024);    // 64 字节对齐(cache line)
posix_memalign(&p, 4096, 8192);       // 页对齐
```

什么时候需要:SIMD 指令集的硬要求(SSE 16 / AVX 32 / AVX-512 64 字节)、DMA 和 `O_DIRECT` 要求 512 字节或页对齐、以及**手动把热点变量对齐到 cache line 防 false sharing**(10 篇的主角)。

代价就是前面说的:对齐 + header 让小对象的实际占用远大于请求量。

## 六、"内存只涨不跌":碎片,不一定是泄漏

### 6.1 现象

```c
for (int i = 0; i < 1000000; i++) p[i] = malloc(8);
for (int i = 0; i < 1000000; i++) free(p[i]);
// 全 free 了,RSS 几乎没降
```

### 6.2 为什么

小 chunk 全躺在 fastbin / smallbin 货架上等复用;brk 不收缩(只有堆顶的 top chunk 连续空闲时才有机会还);只有 mmap 的大块是真还了。再加上时间一长,存活对象和空洞犬牙交错——**空闲内存总量很大,但每块都很碎,既还不了内核,也满足不了大分配**。典型的 24 小时后:

```
RSS 6GB,真正存活的对象 2GB,碎片和货架库存占了 4GB
```

### 6.3 怎么治

- **重启**:最粗暴有效,K8s 滚动重启本质上就是定期清碎片
- **池化**:对象池 / arena allocator,同生共死的对象一起分配一起释放
- **换 jemalloc**:它的碎片管理显著好于 glibc;Redis 更进一步,`activedefrag yes` 用 jemalloc 的 API 在后台主动搬迁数据整理碎片
- **`malloc_trim(0)`**:手动尝试把 top chunk 还给内核——有用但有限,只对 main arena 且要求堆顶连续空闲
- **GC 语言天生免疫**:Java / Go 的 GC 会 compact,把存活对象挪到一起,碎片直接归零

### 6.4 和真泄漏划清界限

**泄漏的定义:分配了、但已经没有任何指针指着它**。碎片是"还找得到但没在用",泄漏是"永远找不回来了"。查泄漏的工具梯队:

- **Valgrind**:最全,但慢 10 倍,只适合测试环境小流量
- **ASan**(`-fsanitize=address`):快很多,需要重编译,测试必备(07 篇讲过)
- **jemalloc prof / gperftools heap profiler**:采样式,开销小到**生产可用**:

```bash
MALLOC_CONF="prof:true,prof_active:true,lg_prof_sample:19" \
LD_PRELOAD=/usr/lib/libjemalloc.so.2 ./your_program
```

轻量观测还有 `ltrace -e malloc,free -c ./prog`(统计调用次数和大小分布)和 glibc 自带的 `mtrace()`。

## 七、Java / Go 为什么不用 malloc

托管语言看了一眼 malloc 的这堆麻烦,选择自己造:

**Java:堆内 bump pointer**。JVM 启动时一次性 mmap 下整个 -Xmx,之后 `new Object()` 只是把一个 top 指针往前推——**不查 bins、不抢锁(每线程有 TLAB 缓冲),一次分配 < 100ns,比 malloc 快一个量级**。碎片?GC compact 时顺手清零。

**Go:mcache / mcentral / mheap 三层**:

```
mcache:   每个 P(逻辑处理器)一份,小对象分配完全无锁
mcentral: 按 size class 分类的中心池,mcache 没货来这补
mheap:    全局堆,大对象在这,从 OS mmap 进货
```

眼熟吗?**这就是 tcmalloc 的结构**——Go 的分配器就是从 tcmalloc 演化来的。

为什么必须自己写,而不是包一层 malloc:

1. **GC 要掌握所有对象的位置和布局**,malloc 是个黑盒
2. 无锁 bump pointer 比 malloc 快 10 倍
3. 自家分配器才能支持**移动对象、压缩堆**这些 GC 的高级动作

**"托管语言分配快"不是玄学,是放弃通用性换来的**——malloc 必须伺候所有 C 程序,JVM 只伺候 Java 对象。

## 八、四个经典反模式

**1. 海量小对象逐个 malloc**

```c
for (int i = 0; i < 1000000; i++) {
    Node *n = malloc(sizeof(Node));   // 不是 syscall,但 header + bins 开销逃不掉
    list_add(&l, n);
}
```

治法:预分配大数组 / 对象池 / arena allocator——一次批发,自己零切。

**2. 短命大对象走堆**

```c
char *buf = malloc(10*1024*1024);  process(buf);  free(buf);
// 如果它没过 mmap 阈值或阈值被动态调高了,这 10MB 就上货架了
```

确定性要归还的大块,直接 `mmap` + `munmap`。

**3. 泄漏**:循环里 malloc 忘 free——ASan / Valgrind 抓。

**4. use-after-free**

```c
char *p = malloc(8);
free(p);
p[0] = 1;     // 可能没事、可能崩、可能改了别人刚分到的同一块——看运气
```

**最难查的一类 bug,因为行为不确定**(那块内存马上会被复用)。ASan 是唯一靠谱的网。

## 踩坑提醒

1. **以为 free 把内存还给了 OS**——只是上了 glibc 的货架,RSS 不降是常态
2. **多线程服务用默认 glibc malloc 还嫌慢**——锁竞争是根因,LD_PRELOAD 换 jemalloc / tcmalloc 是性价比最高的一招
3. **海量小对象不池化**——每个对象 16 字节 header + 对齐,内存和速度双输
4. **不知道 `malloc(1)` 占 32 字节**——估算内存时差出一倍都不奇怪
5. **大数组放栈上**——那是 8MB 限额的栈(07 篇),爆栈秒杀,大缓冲区永远放堆
6. **只查 Java 堆不查 Native**——堆 dump 干净不代表没漏,JNI / Direct Buffer 的泄漏要靠别的工具
7. **Valgrind 跑过就放心**——它慢 10 倍,线程和异常路径常常根本没覆盖到
8. **生产开 ASan**——慢 2-3 倍内存翻倍,它属于测试环境
9. **mmap 大量小块**——每个映射最少一页(4KB),还污染页表
10. **不监控碎片**——RSS 6GB 里可能 4GB 是空气,jemalloc 的 stats 能看出来
11. **把 jemalloc 当银弹**——单线程未必快,换之前压测
12. **瞎调 MALLOC_ARENA_MAX / 阈值**——默认值是 glibc 多年迭代的结果,没压测别动
