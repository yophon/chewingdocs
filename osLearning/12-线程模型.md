# 线程模型

线程是程序员天天在用、却很少有人真正看透的东西。`pthread_create` 一行代码,看起来轻飘飘——直到某天你的服务连接数过万,CPU 全在做上下文切换,业务一行没跑;或者被问"为什么 Go 能开百万个 goroutine,Java 开十万个线程就跪了",你才发现自己从没想过**线程到底是谁在调度、到底贵在哪**。这一篇回答三个问题:线程在 Linux 内核里是什么、一共有哪三种线程模型、线程具体值多少钱——搞清楚这三个,Go / Java / Python 的并发差异一眼看穿。

> 一句话先记住:**线程模型 = "调度由谁做" + "共享什么资源"这两个问题的答案**。1:1(Linux pthread):每个用户线程对应一个内核线程,内核调度,简单但贵;N:1(早期 green thread / Python GIL 实质):全部用户线程挤一个内核线程,切换极快但用不了多核;N:M(Go / Erlang):M 个内核线程驮 N 个协程,两全其美但实现极复杂。**Linux 选 1:1 是工程取舍——把"百万并发"这道题留给了用户态的协程去解**。

## 一、Linux 的线程:共享了资源的进程

11 篇埋的伏笔现在揭开。`pthread_create` 底下就是一次 clone(),flag 拧到"几乎全共享":

```c
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, ...);
```

```
共享的(线程组一份):                各自一份的:
  CLONE_VM       虚拟内存              task_struct(内核里每线程一个)
  CLONE_FS       cwd / root            内核栈   16 KB / 线程
  CLONE_FILES    文件描述符表          用户栈   默认 8 MB / 线程
  CLONE_SIGHAND  信号 handler          寄存器上下文(切换时保存恢复)
  CLONE_THREAD   同一个 TGID           TLS(线程本地存储)、TID
```

**"线程共享内存"不是什么神奇机制,是它们压根就在同一个地址空间里**——这既是线程通信免费的原因,也是数据竞争的原因(14 / 15 篇的主题)。

PID 和 TID 别搞混:

```
进程 = 线程组(TGID);主线程的 TID = TGID = 你平时说的"进程 PID"

ps -eLf
# UID  PID  PPID  LWP  ...     ← LWP 列就是 TID
# user 1234    1  1234         ← 主线程
# user 1234    1  1235         ← 子线程
```

```c
#include <sys/syscall.h>
pid_t tid = syscall(SYS_gettid);   // 当前线程的 TID
pid_t pid = getpid();              // TGID(进程 PID)
```

## 二、三种线程模型:调度权在谁手里

所谓"线程模型",就是**用户眼里的线程和内核眼里的执行单元怎么对应**,一共三种画法。

### 2.1 1:1 —— 内核全包(Linux pthread)

```
用户态     线程A      线程B      线程C
            │          │          │
内核态  task_struct task_struct task_struct
            │          │          │
CPU      ←──────内核调度器分派──────→
```

每个用户线程背后站一个内核 task_struct,**调度完全交给内核**。

- 优点:多核真并行;一个线程阻塞在 syscall 上,别的照跑;实现简单
- 缺点:创建 / 切换都要进内核,贵;数量受内核内存限制(几万到几十万封顶)

**Linux pthread 就是 1:1**,当今的事实标准。

### 2.2 N:1 —— 用户态全包(早期 green thread)

```
用户态     线程A   线程B   线程C   ← 用户态调度器自己切
            └───────┬───────┘
内核态        一个 task_struct
                    │
CPU              一个核
```

- 优点:切换就是个函数调用,不进内核,极快;创建几乎免费;百万级随便开
- 缺点:**永远只能用一个核**;任何一个线程阻塞 syscall,全员卡死

早期 Java green thread 是它;**Python 多线程在 GIL 之下实质也是它**——所以 Python 多线程跑 CPU 密集任务零加速,只有 IO 密集有意义。

### 2.3 N:M —— 两层调度(Go / Erlang)

```
用户态   G G G G G G G G ... (10 万个 goroutine)
              │ 用户态调度器(runtime)
内核态     M M M M M M M M  (≈ CPU 核数个内核线程)
              │ 内核调度器
CPU        8 个物理核
```

M 个内核线程负责真正占住 CPU,用户态调度器把 N 个协程派到这 M 个线程上跑。Go 的 GMP 就是它的工业级实现:

```
G  Goroutine(协程,百万级)
M  Machine(内核线程,大致 = 核数)
P  Processor(逻辑处理器,M 执行 G 时必须持有的"工位")
```

- 优点:百万协程 + 多核并行 + 用户态快速切换,全占了
- 缺点:复杂到离谱——Go 团队打磨调度器花了好几年;阻塞 syscall 要特殊处理(M 陷进内核时让出 P,另起一个 M 接着跑别的 G)

用 N:M 的:Go、Erlang、早期 Solaris,以及 JVM Loom 虚拟线程(第九节)。协程的完整故事详见 17 篇。

## 三、Linux 为什么选了"最笨"的 1:1

1990 年代末 Linux 认真试过 N:M(IBM 主导的 NGPT 项目),结果 2003 年定稿的 NPTL(现在的 pthread 实现)选了 1:1,理由很工程师:

```
1. 简单:内核统一调度,没有两层调度器互相猜
2. 可靠:行为可预测,信号 / 调试 / 性能工具全都好做
3. N:M 的复杂度是长期维护的债
4. "百万并发"这个需求,留给用户态解更合适
```

事后看相当聪明:**内核提供简单可靠的 1:1,谁要百万并发,谁就在用户态搭协程**——Go runtime、asyncio、Loom 全是这么长出来的。代价就是裸线程贵,具体多贵,下一节算账。

## 四、线程到底有多贵:一笔明细账

### 4.1 内存

```
每线程固定开销:
  内核栈        16 KB(实打实的内核内存)
  用户栈        8 MB(虚拟地址,按需缺页分配,实际常用几十 KB)
  task_struct   几 KB
  TLS           ~4 KB
```

1 万个线程,光用户栈就是 **80 GB 虚拟地址空间**——物理内存没那么多(栈是惰性分配的),但虚拟地址、内核栈、task_struct 都是真消耗。栈可以调小:

```c
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setstacksize(&attr, 256 * 1024);   // 256 KB
pthread_create(&tid, &attr, worker, arg);
```

Java 对应 `-Xss256k`。

### 4.2 创建 / 销毁

```
pthread_create:  ~10-50 μs(syscall + 分配栈 + 初始化)
pthread_join:    ~10 μs
```

比 fork 快(地址空间共享,不用复制页表),但**比一次函数调用慢上万倍**——"每请求现场建一个线程"在高 QPS 下,光创建开销就能吃掉 CPU,这是线程池存在的第一理由。

### 4.3 上下文切换

```
线程切换(同进程内):  1-5 μs(不用切页表)
进程切换:             ~5 μs 起(还要切页表、TLB 部分失效)
```

单次很便宜,**要命的是次数**:每秒几千次无感,每秒几十万次时 CPU 大头在保存恢复寄存器、污染 cache,业务反而没时间跑。监控:

```bash
vmstat 1
# ... -system-- ------cpu-----
#       cs        us sy id wa
#     50000       30 20 50  0   ← cs = context switch / 秒
```

经验线:**cs 持续超过 5 万/秒,基本是线程开太多或锁竞争太凶**。

### 4.4 调度延迟

线程数远超核数时,就绪线程要排队等 CPU——**p99 唤醒延迟从几 μs 飙到几 ms**。你的服务 p99 抖动,有时不是 GC 不是慢 SQL,就是线程太多在排队。

## 五、"每连接一线程"的崩塌曲线

把账套到经典架构上——Apache prefork 时代的写法:

```c
while (1) {
    int conn = accept(listen_fd);
    pthread_create(&tid, NULL, handle_request, &conn);
}
```

随连接数增长,它是这样塌的:

```
< 1000 连接        没问题,而且简单可读,别瞎优化
1000 - 5000        开始抖:栈占几十 GB 虚拟空间,切换变多
5000 - 10000       切换吃掉可观 CPU,p99 飙升
> 10000            CPU 全在切换和调度,业务卡死 —— 经典 C10K 问题
```

**这条曲线就是 epoll、异步 IO、协程被发明出来的全部原因**:一个内核线程驮一个连接驮不动,要么让少量线程用事件驱动驮几万连接(20 / 21 / 22 篇),要么用 N:M 把"线程"做便宜(17 篇)。

## 六、线程池:把"贵"摊销掉

还在用线程的场景里,标准解法是线程池:

```
预创建 N 个线程 → 共享任务队列 → worker 取任务执行 → 干完不退出,取下一个
```

一次解决两个问题:**摊销创建销毁开销** + **给并发数设上限**(无上限的线程创建,一次流量尖峰或恶意攻击就能打爆机器)。Java 的 ExecutorService、Python 的 ThreadPoolExecutor、Go 手写 worker pool,全是这个模式。开多大?经验公式:

```
CPU 密集型:  线程数 ≈ CPU 核数(再多是纯切换损耗)
IO 密集型:   线程数 ≈ 核数 × (1 + 等待时间/计算时间),通常 50-200
混合型:      别算了,压测出来
```

## 七、TLS:每个线程一份的"私有抽屉"

线程共享一切内存,但有时你恰恰要"每线程自己一份"——这就是 TLS(Thread Local Storage,线程本地存储):

```c
__thread int my_local;       // GCC 扩展
thread_local int my_local;   // C11 / C++11 标准写法
// 每线程独立副本,读写不需要锁
```

三个你天天在用却没意识到的 TLS:**errno**(就是 TLS,全局一份的话线程 A 的错误码会被 B 冲掉)、**内存分配器的线程缓存**(jemalloc / tcmalloc 高性能的核心,malloc 多数时候不碰全局锁)、**请求上下文透传**(trace id,Java 的 ThreadLocal 同理)。

开销:比普通变量略慢——TLS 经 `fs`/`gs` 段寄存器定位,多一次间接寻址。体积也要注意:每线程 4 KB,1000 个线程就是 4 MB 起步。

## 八、信号在多线程里怎么路由

多线程 + 信号是出了名的雷区,规则其实就两条:

```
kill <pid>            → 发给"进程",内核挑一个没屏蔽它的线程处理
pthread_kill(tid, n)  → 精确发给指定线程

同步信号(SIGSEGV / SIGFPE / SIGPIPE):谁触发,发给谁
异步信号(SIGTERM / SIGINT / SIGUSR1):发给进程,内核随机挑线程
```

"随机挑线程"意味着 handler 可能在任何线程上执行,而 handler 里几乎什么都不能干(不能 malloc、不能加锁,大多数函数不是异步信号安全的)。**最佳实践是 sigwait 模式:所有线程屏蔽信号,专门一个线程同步地等**:

```c
sigset_t set;
sigemptyset(&set);
sigaddset(&set, SIGTERM);
sigaddset(&set, SIGINT);
pthread_sigmask(SIG_BLOCK, &set, NULL);   // 创建任何线程之前先 block

pthread_create(&t, NULL, signal_thread, &set);

void *signal_thread(void *arg) {
    int sig;
    while (1) {
        sigwait((sigset_t *)arg, &sig);
        // 这里是普通线程上下文,不是 handler——加锁、写日志、优雅退出随便干
    }
}
```

本质是**把"异步打断"翻译成"同步事件"**,信号的所有限制就都不存在了。详见 24 篇信号。

## 九、线程怎么死:退出、取消、回收

**正常退出**:worker 函数自然 `return`,或 `pthread_exit(NULL)`。

**pthread_cancel:看着很美,几乎别用**。取消发生在不确定的 cancellation point 上:线程持着锁被取消 → 锁永远不释放;文件写一半被取消 → 数据不一致。工业代码的共识是**协作式取消**——标志位,让线程自己挑体面的退出时机:

```c
volatile int stop = 0;

void *worker(void *arg) {
    while (!stop) { /* 干活,每圈检查一次 */ }
    return NULL;
}
// 主线程:
stop = 1;
pthread_join(tid, NULL);
```

**joinable vs detached**:

```c
pthread_join(tid, NULL);   // 默认 joinable:必须有人 join,资源才回收
pthread_detach(tid);       // detached:退出即自动回收,但再也不能 join
```

joinable 的线程没人 join 就是线程版"僵尸",栈和 task_struct 一直占着。**要结果 / 要协调退出就 join;纯后台 fire-and-forget 就 detach**——二选一,别都不做。

**顺带:JVM Loom 虚拟线程**。Java 19+ 把第二节的 N:M 模型搬进了 JVM:

```java
Thread.startVirtualThread(() -> { /* 跑在少量 carrier 平台线程上 */ });
// 开 100 万个也没问题
```

意义在于:**"每请求一线程"这个最好写的编程模型,在 N:M 加持下复活了**——不用再写回调地狱。和 goroutine 殊途同归。

## 十、同步与排查:工具箱

同步原语只报名字,14 / 15 篇逐个拆:

```
mutex / rwlock / spinlock / condition variable /
semaphore / barrier / atomic / futex
```

先记一条:**99% 的场景 mutex + condition variable 够用**;atomic / lock-free 是确认了瓶颈再上的武器。

```bash
ps -eLf                  # 所有线程(LWP 列是 TID)
top -H -p <pid>          # 进程内各线程 CPU
htop --tree              # 树状视图
ls /proc/<pid>/task      # 每线程一个目录
cat /proc/<pid>/status   # 看 Threads: N 字段

pidstat -t -p <pid> 1    # 每秒每线程 CPU
perf top -t <tid>        # 单线程热点

gdb -p <pid>             # attach
(gdb) info threads       # 所有线程
(gdb) thread N           # 切到线程 N
(gdb) bt                 # 看它卡在哪
```

死锁排查套路:gdb attach → `info threads` → 挨个 `bt` → 看哪几个线程互相停在 `pthread_mutex_lock` 里等对方手里的锁。

## 踩坑提醒

1. **每连接一线程扛高并发**——1 万连接以上必崩(C10K),换 epoll / 协程
2. **不给线程数设上限**——一次流量尖峰 / 恶意请求就能让你创建线程到机器爆掉
3. **用 pthread_cancel 杀线程**——锁泄漏、数据不一致,改协作式标志位
4. **多线程进程里 fork**——子进程只剩一个线程,别的线程持有的锁永远锁死(pthread_atfork 只是缓解)
5. **在信号 handler 里干正事**——用 sigwait 专职线程,handler 里几乎什么都不安全
6. **默认 8MB 栈开一万个线程**——80 GB 虚拟地址空间,32 位下直接开不出来
7. **TLS 变量又大又多**——每线程一份,数量一乘很可观
8. **指望 Python 多线程加速 CPU 密集任务**——GIL 之下实质是 N:1,只有 IO 密集有收益
9. **以为线程切换免费**——cs 超过 5 万/秒时,CPU 在切换不在干活
10. **不用线程池,任务来了现场建线程**——10-50 μs 的创建开销在高 QPS 下就是瓶颈
11. **线程池复用线程时不清 TLS**——上个请求的 ThreadLocal 残留,串号事故的经典来源
12. **detach 之后还想 join**——detached 是 fire-and-forget,要结果就别 detach
