# 线程模型

线程是**程序员每天用、但很少有人真正搞懂**的东西——`pthread_create` 一行代码,背后是内核 / 用户态调度、内存共享、栈分配、信号路由的一系列复杂决策。Linux 上线程和进程的本质相同(都是 task_struct),但**用户感知 / 编程模型完全不同**。这一篇讲清楚:**三种线程模型(1:1 / N:1 / N:M)、为什么 Linux 是 1:1、为什么 Go 是 N:M、线程的真实开销**,以及**为什么"每连接一线程"在 1 万连接以下能用,超过就要换 epoll / 协程**。

> 一句话先记住:**线程模型决定了"调度由谁做" + "共享什么资源"**。**1:1(Linux pthread)**:每个用户线程对应一个内核线程,内核调度,**简单但贵**;**N:1(早期 Java green thread)**:多个用户线程跑在一个内核线程上,**完全用户态调度,快但不能利用多核**;**N:M(Go / Erlang)**:M 个内核线程承载 N 个用户线程(协程),**两全其美但复杂**。**Linux 选 1:1 是工程取舍**——简单可靠,代价是线程开销大。

## 一、Linux 上线程 = 共享资源的进程

第 11 篇讲过:`pthread_create` 内部就是 `clone()` 加一堆共享 flag:

```c
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, ...);
```

```
共享了:
  CLONE_VM       虚拟内存(同一个地址空间)
  CLONE_FS       文件系统信息(cwd / root)
  CLONE_FILES    文件描述符表
  CLONE_SIGHAND  信号 handler
  CLONE_THREAD   同一个 TGID(线程组)

各自有:
  task_struct    每个线程一个
  内核栈         16KB / 线程
  用户栈         默认 8MB / 线程
  寄存器         上下文切换时保存
  TLS(Thread Local Storage)
  线程 ID(TID,内核里其实是 PID,只是同 TGID)
```

### 1.1 PID vs TID

```
进程 = 线程组(TGID)
主线程的 TID = TGID = "进程的 PID"
其他线程的 TID 不同,但 TGID 相同

ps -eLf
# UID  PID  PPID  LWP  ...   ← LWP 是 TID
# user 1234 1     1234       ← 主线程
# user 1234 1     1235       ← 子线程
# user 1234 1     1236       ← 子线程
```

```c
#include <sys/syscall.h>
pid_t tid = syscall(SYS_gettid);   // 拿当前线程的 TID
pid_t pid = getpid();               // TGID
```

## 二、三种线程模型

### 2.1 1:1(内核线程模型)

```
用户线程  →  内核线程
线程 A    →  内核 task_struct 1
线程 B    →  内核 task_struct 2
线程 C    →  内核 task_struct 3
```

**每个用户线程一个内核 task_struct**,**内核调度**。

**优点**:

- 多核能并行(每个线程独立调度)
- 阻塞 syscall 不影响其他线程
- 实现简单

**缺点**:

- 创建 / 切换贵(syscall + 内核数据结构)
- 数量受限(几万 ~ 几十万,内核内存有限)

**Linux pthread 用这个**——这是事实标准。

### 2.2 N:1(纯用户线程)

```
用户线程  →  一个内核线程
线程 A,B,C 都在 1 个内核线程上跑
用户态自己实现调度
```

**优点**:

- 切换极快(纯函数调用,不进内核)
- 创建几乎免费
- 可创建百万级

**缺点**:

- **不能利用多核**(只有一个内核线程)
- 一个线程阻塞 syscall → 全部阻塞

**早期 Java green thread / Python(GIL)实质就是这个模型**——所以 Python 多线程 CPU 密集场景**没有加速**。

### 2.3 N:M(混合模型)

```
N 个用户线程(协程)→  M 个内核线程
M 个内核线程负责真正的并行
用户态调度器把 N 个协程派到 M 个内核线程上
```

**Go 的 GMP 模型**:

```
G  Goroutine(协程)
M  Machine(内核线程,通常 = CPU 核数)
P  Processor(逻辑处理器,M 跑 G 时绑定的资源)

10 万个 G 在 8 个 M 上调度,8 个 M 真正在 8 个 CPU 核上跑
```

**优点**:

- 可以百万协程
- 多核并行
- 协程切换快(用户态)

**缺点**:

- 复杂(Go 团队花了几年优化调度器)
- syscall 阻塞要特殊处理(M 让出 P,新建 M 跑其他 G)

**用 N:M 的语言**:Go、Erlang、早期 Solaris、(部分)JVM Loom(虚拟线程)。

详见 17 篇协程。

## 三、Linux 为什么选 1:1

```
1990s 早期:Linux 也尝试过 N:M(NGPT 项目)
最终选了 1:1(NPTL,2003 年)

原因:
  1. 简单,内核统一调度
  2. 可靠,行为可预测
  3. 多核时代不再追求"百万线程"(协程是更好的解)
  4. N:M 实现复杂,维护成本高
```

**结果**:Linux pthread 简单稳定,但**线程开销大** —— 用户要协程时自己上 Go / 异步 IO。

## 四、线程的真实开销

### 4.1 内存

```
每线程默认占用:
  内核栈:     16 KB(固定,内核里)
  用户栈:     8 MB(虚拟地址,COW)
  task_struct: 几 KB
  TLS:         ~4 KB
  
理论上 1 万线程占 80 GB 虚拟内存(实际物理小很多,因为栈 COW 大部分没用)
```

**调小栈大小**:

```c
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setstacksize(&attr, 256 * 1024);   // 256 KB
pthread_create(&tid, &attr, ...);
```

**Java 调线程栈**:`-Xss256k`。

### 4.2 创建 / 销毁

```
pthread_create:  ~10-50 μs(syscall + 分配栈 + 初始化)
pthread_join:    ~10 μs
```

**比 fork 快**(共享内存不用复制),但**比函数调用慢 1 万倍**。

**这就是为什么"每请求一线程"在高并发场景不行** —— 创建开销吃掉所有 CPU。

### 4.3 上下文切换

```
线程切换:        1-5 μs(同进程内)
进程切换:        ~5 μs(还要切页表、刷 TLB)
```

**几千次/秒切换还行,几十万次/秒就 CPU 全在切换,业务跑不动**。

```bash
vmstat 1
# procs ----memory---- ---swap-- -----io---- -system-- ------cpu-----
#  r  b  swpd ...                                 cs   us sy id wa
#  2  0  0   ...                              50000   30 20 50 0
#                                              ↑ context switch / sec
```

`cs > 5 万/秒` 通常说明线程太多 / 锁竞争激烈。

### 4.4 调度延迟

```
线程数远超 CPU 核数 → 调度抖动大
p99 唤醒延迟从 几 μs 飙到 几 ms
```

## 五、"每连接一线程"模型的瓶颈

经典的 Apache prefork 模式:

```c
while (1) {
    int conn = accept(listen_fd);
    pthread_create(&tid, NULL, handle_request, &conn);
}
```

**性能边界**:

```
1000 连接以下:        OK,简单可读
1000-5000 连接:       开始抖,栈内存几十 GB
5000-10000 连接:      上下文切换多,p99 飙
> 10000 连接:         CPU 全在切换,业务卡死(C10K 问题)
```

**这就是 epoll / 异步 IO / 协程**出现的原因——突破"每连接一线程"的瓶颈。

详见 20 / 21 / 22 篇。

## 六、线程池:线程开销的常见解法

```
预创建 N 个线程
任务队列 + N 个 worker 取任务
任务完成后线程不死,继续取下一个
```

**优势**:

- 避免反复创建 / 销毁
- 控制并发数(防止线程爆炸)

**Java 的 ExecutorService、Python 的 ThreadPoolExecutor、Go 的 worker pool**——都是这个模式。

**线程池大小经验值**:

```
CPU 密集型:    线程数 = CPU 核数
IO 密集型:    线程数 = CPU 核数 × (1 + 等待时间/计算时间)
                通常 50-200
混合型:        测出来
```

## 七、TLS:线程本地存储

```c
__thread int my_local;     // GCC 扩展
thread_local int my_local; // C11 / C++11

// 每个线程有独立的 my_local 副本,无锁访问
```

**用途**:

- 线程独立的 cache(jemalloc / tcmalloc 的核心优化)
- 错误码(errno 就是 TLS,不然多线程会乱)
- 当前用户上下文(请求 ID 透传)

**开销**:**比普通变量稍慢**(TLS 用一个特殊的段寄存器 `fs`/`gs` 找)。

## 八、线程的信号路由

### 8.1 信号给"进程"还是给"线程"

```
kill <pid> SIGTERM
→ 信号送给进程,内核选一个"愿意接收"的线程处理

pthread_kill(tid, SIGTERM)
→ 直接送给指定线程
```

### 8.2 同步信号 vs 异步信号

```
同步信号(SIGSEGV / SIGFPE / SIGPIPE):
  哪个线程触发,送给那个线程
  
异步信号(SIGINT / SIGTERM / SIGUSR1):
  送给进程,内核选线程
  最佳实践:专门一个线程 sigwait,其他线程 mask 掉
```

### 8.3 sigwait 模式

```c
sigset_t set;
sigemptyset(&set);
sigaddset(&set, SIGTERM);
sigaddset(&set, SIGINT);
pthread_sigmask(SIG_BLOCK, &set, NULL);   // 主线程 block

// 专门一个 signal handler 线程
pthread_create(&t, NULL, signal_handler, &set);

void *signal_handler(void *arg) {
    sigset_t *set = arg;
    int sig;
    while (1) {
        sigwait(set, &sig);
        // 处理信号(可以做复杂的事,因为不在 signal handler 里)
    }
}
```

**为什么这样**:**signal handler 严格限制能做的事**(不能 malloc / 不能加锁 / 不能调多数 syscall)—— 用 sigwait 把信号转成同步事件,处理逻辑就自由了。

## 九、线程同步原语速览

详见 14 / 15 篇,这里只列名字:

```
mutex            互斥锁
rwlock           读写锁
spinlock         自旋锁(用户态很少用,内核常见)
condition var    条件变量
semaphore        信号量
barrier          栅栏(等所有线程到齐)

```

## 十、线程的取消和终止

### 10.1 pthread_exit / 自然返回

```c
void *worker(void *arg) {
    // ...
    pthread_exit(NULL);   // 线程结束
    // 或自然 return
}
```

### 10.2 pthread_cancel:取消另一个线程

```c
pthread_cancel(tid);
```

**坑**:

- 取消是**异步的**,可能在任何 cancellation point 触发
- 持有锁的线程被取消 → 锁泄漏
- 写过的资源 / 文件可能不一致

**实践**:**几乎不用 pthread_cancel** —— 用标志位 + 协作式取消。

```c
volatile int stop = 0;

void *worker(void *arg) {
    while (!stop) {
        // ...
    }
}

// 主线程
stop = 1;
pthread_join(tid, NULL);
```

### 10.3 detached vs joinable

```c
pthread_create(&tid, NULL, worker, arg);
pthread_join(tid, NULL);    // 默认 joinable,主动 join 才能回收资源

pthread_detach(tid);         // detached,自动回收,不能 join
```

**detached 适合**:不关心结果的后台线程。
**joinable 适合**:要拿结果或协调退出的线程。

## 十一、虚拟线程(JVM Loom)

Java 19+ 引入的**协程式线程**:

```java
Thread.startVirtualThread(() -> {
    // 这是虚拟线程,跑在 carrier(平台线程)上
});

// 创建 100 万个虚拟线程也没问题
```

**底层是 N:M 模型**——和 Go 的 goroutine 类似。

**意义**:Java 终于有了协程,**"每请求一线程"模型重生**——不用再写复杂的异步回调。

## 十二、线程相关的工具

```bash
ps -eLf                # 看所有线程
top -H -p <pid>        # 进程内线程
htop --tree            # 线程树
ls /proc/<pid>/task    # 线程目录
cat /proc/<pid>/status # 看 Threads: N 字段

pidstat -t -p <pid> 1  # 每秒每线程的 CPU
perf top -t <tid>      # 单线程的 perf

gdb -p <pid>           # attach 到进程
(gdb) info threads     # 看所有线程
(gdb) thread N         # 切到线程 N
(gdb) bt               # 看栈
```

**死锁排查**:`gdb` attach + `info threads` + 各 thread 的 `bt`,看哪几个线程在等彼此的锁。

## 十三、踩坑提醒

1. **每请求一线程** —— 1 万连接以上必崩
2. **不限制线程数** —— 攻击者一个 DDoS 让你创建无数线程
3. **pthread_cancel** —— 几乎总是错的,用协作式取消
4. **fork 多线程进程** —— 子进程只剩一个线程,其他线程持有的锁永远锁着
5. **信号给某个线程** —— 异步信号要 sigwait,别在 handler 里干活
6. **线程栈太大** —— 1 万线程占 80GB 虚拟内存
7. **thread-local 变量大量分配** —— 一个 4KB,1000 线程就 4MB
8. **以为 Python 多线程能加速 CPU 密集** —— GIL,只有 IO 密集有效
9. **以为线程切换免费** —— 上下文切换太多 cs 高
10. **不用线程池** —— 反复创建销毁线程性能差
11. **TLS 用错** —— ThreadPoolExecutor 复用线程时,TLS 残留
12. **以为 detached 线程能 join** —— 错,detached 只能 fire and forget
