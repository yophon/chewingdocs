# 调度器

有个真实到不能再真实的事故剧本:服务压测吞吐上不去,有人拍板"线程池从 200 加到 400",重新压——**吞吐反而降 15%,p99 翻倍**,会议室一片安静。根源是一个普遍误解:**以为线程是"雇一个人干一份活",人越多活干得越快**。实际上 CPU 核数是死的,线程只是"想上 CPU 的排队者",超过核数的部分全在队里站着——而排队、换人、搬家,每一步都有调度器在背后记账收费。这一篇讲清楚:CFS 怎么决定下一个谁跑、nice 值到底改了什么、上下文切换的钱花在哪、实时调度为什么能把整台机器"饿死"、线程数到底怎么定。

> 一句话先记住:**调度器是 CPU 时间的仲裁者,Linux CFS 的仲裁原则是"谁拿到的 CPU 最少,谁下一个跑"**。可运行线程数一旦超过核数,多出来的全在 run queue 排队,延迟直接涨。**CPU 密集型线程数 ≈ 核数,IO 密集型按等待比例放大**。实时调度能插队,但用错一行代码就能让 ssh 都连不上。

## 一、调度器在仲裁什么

一台 8 核机器上,随手 `ps -eLf | wc -l` 能数出上千个线程:

```
CPU:   8 个硬件执行位
线程:  1000+ 个"想跑"的执行流
问题:  谁先跑?跑多久?被谁打断?在哪个核上跑?
```

调度器要同时伺候五个互相打架的目标:

- **公平**:不能让任何进程永远轮不上;
- **吞吐**:CPU 别空转,切换别太频繁(切换本身不产出);
- **延迟**:你敲键盘的进程要立刻响应,哪怕后台在跑编译;
- **亲和性**:线程尽量留在原来的核上——换核等于 L1/L2 cache 全部白热(详见 10 篇);
- **优先级**:重要的多吃,不重要的让着点。

注意吞吐和延迟天然矛盾:**切换越少吞吐越高,切换越勤响应越快**。调度器的所有设计,都是在这根钢丝上找平衡。

## 二、谁有资格排队:任务状态与 run queue

调度器不是在"所有线程"里挑——绝大多数线程根本不想跑,它们在睡觉等事件。Linux 的任务状态(详见 11 篇):

```
TASK_RUNNING          可运行/正在运行     ← 只有它在抢 CPU
TASK_INTERRUPTIBLE    可中断睡眠(等 IO、等锁,ps 里的 S)
TASK_UNINTERRUPTIBLE  不可中断睡眠(ps 里的 D,通常卡在磁盘/NFS)
TASK_STOPPED          被暂停(Ctrl-Z / SIGSTOP)
TASK_ZOMBIE           僵尸,等父进程收尸
```

**真正参与排队的只有 TASK_RUNNING**。"机器上有 1000 个线程"不可怕,可怕的是"100 个 runnable 挤 8 个核"——`uptime` 的 load average 量的就是这个(加上 D 状态)。

每个 CPU 有**自己私有的 run queue**,不是全局一条队(全局队会让所有核抢同一把锁,详见 14 篇):

```
CPU0 rq: T1 T3 T8     ← 各排各的队
CPU1 rq: T2 T4
CPU2 rq: T5
CPU3 rq: (空闲)
```

队伍长短不齐时,**负载均衡器**周期性把任务从忙核搬到闲核。但搬家不免费:任务的热数据还留在原来核的 cache 里,搬过去等于带着空 cache 重新开张。**调度器在"均衡"和"亲和"之间永远只能取折中**。

## 三、CFS:用一棵红黑树实现"完全公平"

Linux 默认调度器叫 **CFS(Completely Fair Scheduler,完全公平调度器)**,设计思想出奇纯粹:不发固定时间片,只维护一个数字——

```
vruntime(虚拟运行时间)= 任务实际跑掉的 CPU 时间 × 权重修正
```

**仲裁规则就一条:谁的 vruntime 最小,说明谁"吃得最少、亏得最大",下一个就让谁跑**。跑的时候 vruntime 持续增长,涨到不再是最小,就换下一个。所有任务的 vruntime 像被一根橡皮筋拴着,谁也甩不开谁——这就是"完全公平"。

### 3.1 红黑树:O(log n) 找到最亏的人

可运行任务按 vruntime 组织成一棵红黑树:

```
                    ( vruntime=205 )
                   /               \
          ( 102 )                   ( 310 )
         /       \                 /
   ( 87 )       ( 150 )      ( 260 )
   /
( 41 ) ← 最左节点 = vruntime 最小 = 下一个上 CPU 的
─────────────────────────────────────────────
  挑任务:    摘最左节点(有缓存指针,实际 O(1))
  任务下场:  按新 vruntime 插回树里,O(log n)
  睡醒的任务: vruntime 补到接近当前最小值再入树
             (不让它"睡一觉血亏",也不让它"睡出特权")
```

优雅之处在于:**没有时间片轮转表,公平性是数据结构自动涌现的**——交互式任务大部分时间在睡(vruntime 涨得慢),一醒来就是树上最左,天然低延迟;批处理任务一直跑(vruntime 涨得快),自动往树右边沉,天然让路。

### 3.2 nice:改的是涨价速度,不是配额

`nice` 值范围 -20 到 19,越小越"凶":

```bash
nice -n 10 ./batch-job        # 启动时降权
renice -n -5 -p 12345         # 给运行中进程提权(负值需要 root)
```

nice 通过**权重**起作用:nice 低 → 权重大 → 同样跑 10ms,**vruntime 涨得慢** → 更久霸住"树最左" → 实际分到更多 CPU。相邻 nice 值权重差约 1.25 倍。

最要紧的纠偏:**nice 不是"固定分配百分之多少 CPU"**,它只改**有竞争时**的相对份额——机器空闲时,nice 19 的任务照样吃满整个核。

## 四、上下文切换:每一次都在交"cache 税"

上下文切换(context switch)的账单分两部分。

**直接成本**(切换动作本身):保存当前任务的寄存器、栈指针、程序计数器;切换内核栈;进程间切换还要换地址空间(页表基址),**TLB 大面积失效**(详见 06 篇);再恢复下一个任务的上下文。

**间接成本**(往往更贵):新任务把 L1/L2 cache 一点点填成自己的数据,**前一个任务切回来时热数据全凉了**,每次访存重新走慢路径。

```
直接成本:    一次切换 几百 ns ~ 几 μs
间接成本:    cache 重新热起来,可能拖慢之后几十 μs 的执行
线程 vs 进程: 同进程线程切换便宜些(不换地址空间、TLB 保得住),
              但 cache 污染一分不少
```

每秒几万次切换时,这笔税从"噪声"变成"主要支出"——**这正是开头"线程翻倍吞吐反降"的账本**:200 线程时排队不长切换不频;400 线程后 runnable 远超核数,CPU 时间的相当一部分花在换人和重热 cache 上,真正干活的比例反而降了。

### 4.1 怎么观察

```bash
vmstat 1            # cs 列:全机每秒上下文切换次数
pidstat -w 1        # 每进程:cswch(主动让出)/ nvcswch(被抢占)
perf sched record
perf sched latency  # 每个任务等了多久才上 CPU ← 排查 p99 的利器
```

`pidstat -w` 两列分开读:**cswch 高 = 任务自己频繁阻塞**(锁、小 IO、sleep);**nvcswch 高 = 想跑总被抢**(runnable 太多)。cs 高得离谱的常见嫌疑人:线程太多、锁竞争激烈(详见 14 篇,等锁=睡眠=切换)、小 IO 太碎、频繁 sleep/wakeup、线程池队列设计差。

## 五、抢占:你随时可能被请下场

调度不只在任务"自愿让位"时发生。完整的调度时机:

- 任务主动阻塞:读 IO、等锁、sleep;
- 时间片用完(CFS 里是"你的 vruntime 不再最小");
- 更高优先级 / 更小 vruntime 的任务被唤醒;
- 中断返回用户态前,检查 `need_resched` 标志;
- 系统调用返回前,同样检查。

这就是**抢占式调度(preemptive scheduling)**:好处是死循环程序霸占不了机器;代价是**你的代码可能在任何两条指令之间被切走**。所以"这两行挨得很近,不会被打断吧"是并发编程里最危险的侥幸——不会被打断的代码不存在,只有被锁保护的代码(14 篇)。

## 六、实时调度:能插队的贵宾,也能饿死全场

CFS 只是"普通舱"。完整菜单:

| 策略 | 一句话 |
| --- | --- |
| `SCHED_OTHER` | 普通 CFS,默认 |
| `SCHED_BATCH` | 批处理:"我不在乎延迟",少打扰交互任务 |
| `SCHED_IDLE` | 极低优先级,所有人都不跑了才轮到 |
| `SCHED_FIFO` | 实时:先到先跑,**不阻塞、不被更高优先级抢,就一直跑** |
| `SCHED_RR` | 实时:同优先级轮转,其余同 FIFO |
| `SCHED_DEADLINE` | 按"运行时长/周期/截止期"调度,最硬核 |

关键规则:**任何实时任务的优先级都高于全部普通任务**。CFS 的 vruntime 公平在实时任务面前一文不值。于是有了名场面——某团队给"延迟敏感"线程上了 `SCHED_FIFO`,代码里恰好有段忙等:

```
实时线程: while (!flag) {}     ← 忙等,永不阻塞,FIFO 永不下场
后果:     这个核上的所有普通任务 0 CPU
          核上跑着 sshd     → ssh 连不上
          跑着监控 agent    → 报警系统先瞎
          跑着 kworker      → 内核自己的活都干不了
现象:     机器"卡死",但 ping 得通 ← 中断还活着,只是没人调度用户进程
```

听起来吓人,其实内核留了最后一道保险:默认 `sched_rt_runtime_us / sched_rt_period_us = 950000/1000000`,**实时任务每秒最多吃 95%,强制留 5% 给普通任务喘气**——事故现场你还能登上去,多半靠的就是这 5%。但别赌它:**除非是音视频、机器人、交易撮合、工业控制这类真硬实时场景,且每条实时路径保证会阻塞,否则别碰 SCHED_FIFO/RR**。实时 + 锁还必须开优先级继承,不然就是 14 篇的火星探路者事故。

## 七、绑核与 NUMA:位置也是性能

### 7.1 CPU 亲和性

```bash
taskset -c 0-3 ./server       # 启动时绑定到核 0-3
taskset -p -c 2 12345         # 把运行中的进程绑到核 2
```

绑核(CPU affinity)的正当理由都围绕一件事——**保住 cache 热度和确定性**:延迟敏感线程独占核、网卡中断固定到专门的核、NUMA 场景把计算钉在内存旁边。不正当理由也常见:不了解负载就乱绑,几个核过载其余闲死——**负载均衡是你拿不准时的更优解,绑核是你非常确定时的精调**。容器里还要叠一层小心:cgroup 的 cpuset/quota 已经在限制 CPU,再手工绑核,两层规则打架,行为更难预测。

### 7.2 NUMA:内存有远近

多路服务器上"内存"不是均匀的池子:

```
┌─ Socket 0 ──────┐          ┌─ Socket 1 ──────┐
│ CPU 0-15        │  互联总线 │ CPU 16-31       │
│   │ 本地访问,快 │◄────────►│   │ 本地访问,快 │
│ 内存节点 0      │   跨节点  │ 内存节点 1      │
└─────────────────┘ 慢1.3-2倍 └─────────────────┘
```

调度器和内存分配器会尽量维持"线程在哪个 node 跑,内存就从哪个 node 分"。但线程被搬几次家后,容易落到"CPU 在 node0、内存在 node1"的别扭姿势——每次访存付 1.3-2 倍的跨节点税。排查与固定:

```bash
numactl --hardware                            # 看拓扑
numastat -p <pid>                             # 这个进程的内存落在哪个 node
numactl --cpunodebind=0 --membind=0 ./server  # CPU 和内存一起钉死在 node0
```

## 八、线程数到底怎么定

回到开头的事故。线程数的算法取决于线程在干嘛:

**CPU 密集型**(编码、压缩、计算):

```
线程数 ≈ CPU 核数
```

每个线程都能吃满一个核,多开的部分纯粹在排队 + 制造切换,**只亏不赚**。

**IO 密集型**(等 DB、等下游):线程大部分时间在睡,睡着的不占 CPU,可以多开:

```
线程数 ≈ 核数 × (1 + 等待时间 / 计算时间)

例:一次请求计算 5ms + 等 DB 45ms
    → 核数 × (1 + 45/5) = 核数 × 10
    8 核 ≈ 80 线程,刚好让 CPU 在大家轮流睡觉时不空转
```

但这个公式算的是**上限**,不是答案。真实世界还有四道闸门:DB 连接池(80 个线程抢 20 个连接,多出的 60 个在等池子不是等 CPU)、下游限流、每线程的栈内存、锁竞争(线程越多,14 篇的 futex 排队越长)。**正确做法:公式定起点,压测看 p99 和 cs,逐步逼近**——而不是"不够就翻倍"。

顺着公式也能看到天花板:等待/计算比 100:1 的场景(海量长连接),线程数要上万,内存和切换成本先崩——**这正是协程登场的理由,下一篇的主题**。

## 九、三个高频故障现场

### 9.1 CPU 100%,吞吐却不涨

```bash
uptime              # load 远超核数 → runnable 排长队
top -H -p <pid>     # 按线程看:谁在烧 CPU
pidstat -u -w 1     # 切换次数 + CPU 占用一起看
perf top            # 烧在哪个函数:业务?自旋锁?GC?memcpy?
```

常见病根:runnable 太多(回看第八节)、锁竞争退化成自旋空转、GC 线程抢 CPU、切换吃掉真实算力、**容器 CPU quota 被打满**(看下面)。

### 9.2 p99 偶发抖动

平均延迟很好,p99 偶尔飙——典型"调度税"症状:线程 ready 了却在 run queue 等几毫秒、跨核迁移后 cache 全凉、CPU 动态降频、网卡中断打在业务核上、**cgroup CPU throttle**(容器配 0.5 核,burst 一下就被掐到下个周期)。

```bash
cat /sys/fs/cgroup/cpu.stat    # nr_throttled / throttled_usec:被掐了多少次
perf sched latency             # 调度等待,直接量化"排队税"
cat /proc/interrupts           # 中断都打在哪个核
```

容器里 `nr_throttled` 持续涨,就别再看代码了——**是配额问题,不是程序问题**。

### 9.3 一堆 D 状态

`D`(不可中断睡眠)= 卡在磁盘/NFS/块设备 IO,**kill -9 都杀不动**:

```bash
ps -eo pid,stat,wchan,cmd | awk '$2 ~ /D/'   # wchan 列:卡在内核哪个函数
```

大量 D 状态时调度器是被冤枉的——它也想调度这些任务,但它们在等存储。根因去查磁盘、NFS、块设备队列,而不是调 nice。

## 踩坑提醒

1. **以为线程越多越快**——runnable 超过核数就是纯排队,翻倍线程换来吞吐下降是经典名场面
2. **以为 nice 是"分配百分比"**——它只改竞争时的相对权重,机器空闲时 nice 19 照样吃满核
3. **给忙等代码上 SCHED_FIFO**——实时任务不下场,普通任务全饿死,只剩内核留的 5% 保命
4. **实时线程用普通 mutex 不开优先级继承**——优先级反转,重演火星探路者(详见 14 篇)
5. **假设"代码很短不会被切走"**——抢占发生在任何两条指令之间,只有锁能保护你
6. **不了解负载就乱绑核**——几个核过载其余闲死,负载均衡本来做得比你好
7. **容器里 CPU 慢先怪代码**——先看 cpu.stat 的 throttle 计数,经常是 quota 配小了
8. **把 load average 当 CPU 使用率**——load 算 runnable + D 状态,大量 D 时 load 高但 CPU 很闲,根因在存储
9. **cs 高只会加机器**——先分清主动切换(锁/IO 碎)还是被动切换(线程太多),加机器治不了配置错
10. **跨 NUMA 不闻不问**——CPU 和内存分居两个 node,每次访存交 1.3-2 倍的税,numastat 一看便知
