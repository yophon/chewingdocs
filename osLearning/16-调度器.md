# 调度器

你的线程不是"一直在 CPU 上跑",而是被 OS 切成一小段一小段执行。每次 p99 抖动、CPU 打满但吞吐不涨、线程数越加越慢,背后都可能是调度器在做资源仲裁。这一篇讲清楚:**CFS 怎么决定谁跑、nice 值到底影响什么、上下文切换贵在哪、实时调度为什么危险、CPU 亲和性和 NUMA 为什么会影响延迟**。

> 一句话先记住:**调度器 = CPU 时间的仲裁者**。Linux CFS 追求"公平":谁拿的 CPU 少,谁优先跑。**线程不是越多越好**——可运行线程数超过 CPU 核数后,剩下的都在 run queue 里排队,延迟直接涨。**CPU 密集型线程数接近核数,IO 密集型线程数按等待比例放大**。实时调度能插队,但用错会把机器饿死。

---

## 一、调度器解决什么问题

一台机器有 8 个核,却可能有几千个线程:

```
CPU:      8 个硬件执行位置
线程:     1000 个想跑的执行流
问题:     谁先跑?跑多久?被谁打断?
```

调度器要同时兼顾:

- **公平**:不能让某个进程永远没机会
- **吞吐**:CPU 不要空着
- **延迟**:交互任务要快响应
- **亲和性**:尽量别把线程搬来搬去,否则 cache 全丢
- **优先级**:重要任务多拿一点 CPU

## 二、进程状态与 run queue

Linux 调度器主要在可运行任务里选:

```
TASK_RUNNING       可运行/正在运行
TASK_INTERRUPTIBLE 可被信号打断的睡眠,如等 IO
TASK_UNINTERRUPTIBLE 不可中断睡眠,常见 D 状态
TASK_STOPPED       暂停
TASK_ZOMBIE        僵尸
```

真正抢 CPU 的只有 `TASK_RUNNING`。

每个 CPU 有自己的 run queue:

```
CPU0 rq: T1 T3 T8
CPU1 rq: T2 T4
CPU2 rq: T5
CPU3 rq: 空
```

负载均衡器会周期性把任务从忙 CPU 迁到闲 CPU,但迁移有代价:cache 亲和性变差。

## 三、CFS:完全公平调度器

CFS(Completely Fair Scheduler)不是给每个线程固定时间片,而是维护一个核心指标:**vruntime**。

```
vruntime = 任务实际运行时间 × 权重修正
```

谁的 `vruntime` 最小,说明谁"亏了",下一次优先跑。

### 3.1 红黑树

CFS 把可运行任务按 `vruntime` 放进红黑树:

```
最左边 = vruntime 最小 = 下一个该跑
```

选下一个任务就是取最左节点,复杂度 O(log n)。

### 3.2 nice 值

`nice` 范围 -20 到 19:

- `nice -20`:优先级最高,权重大
- `nice 0`:默认
- `nice 19`:最谦让

权重大意味着同样跑 10ms,`vruntime` 增长更慢,所以下次更容易继续被选中。

```bash
nice -n 10 ./batch-job
renice -n -5 -p 12345
```

**nice 不是"固定分配 CPU 百分比"**,它只改变竞争时的相对权重。

## 四、上下文切换贵在哪

上下文切换 = CPU 从一个任务切到另一个任务。

要做的事:

1. 保存当前寄存器、栈指针、程序计数器
2. 切换内核栈
3. 切换地址空间(进程切换时)
4. 刷新/影响 TLB
5. cache 里的热数据变冷
6. 恢复下一个任务上下文

线程切换通常比进程切换便宜,因为同进程线程共享地址空间。但它仍然会破坏 cache。

经验量级:

```
一次上下文切换: 几百 ns 到几 μs
频繁切换:       p99 延迟杀手
```

### 4.1 怎么观察

```bash
vmstat 1          # cs: context switch/s
pidstat -w 1      # 每进程上下文切换
perf sched record
perf sched latency
```

如果 `cs` 高得离谱,常见原因:

- 线程太多
- 锁竞争激烈
- 小 IO 太多
- 频繁 sleep/wakeup
- 线程池队列设计差

## 五、抢占与时间片

调度发生在这些时机:

- 线程主动阻塞:读 IO、等锁、sleep
- 时间片耗尽
- 更高优先级任务唤醒
- 中断返回用户态前检查 `need_resched`
- 系统调用返回前检查

**抢占式调度**的好处:一个死循环用户程序不能独占 CPU。

坏处:你可能在任何地方被切走。所以多线程必须用锁,不能假设"这几行很短不会被打断"。

## 六、实时调度

Linux 常见调度策略:

| 策略 | 说明 |
| --- | --- |
| `SCHED_OTHER` | 普通 CFS |
| `SCHED_BATCH` | 批处理,少打扰交互任务 |
| `SCHED_IDLE` | 极低优先级 |
| `SCHED_FIFO` | 实时,先到先跑,不主动让出就一直跑 |
| `SCHED_RR` | 实时,轮转 |
| `SCHED_DEADLINE` | 按运行时间/周期/截止期调度 |

实时任务优先级高于普通任务。用错会很危险:

```
实时线程 while(true) {}
普通 ssh / systemd / 监控全饿死
机器看起来"卡死"
```

除非是音视频、机器人、交易撮合、工业控制,否则不要轻易上实时调度。

## 七、CPU 亲和性

```bash
taskset -c 0-3 ./server
taskset -p -c 2 12345
```

CPU 亲和性让进程/线程只在指定 CPU 上跑。

适用:

- 减少线程迁移
- 固定中断处理核
- 给延迟敏感线程独占核
- NUMA 场景绑定 CPU 和内存

不适用:

- 不了解负载就乱绑,可能让某几个核过载
- 容器里 CPU quota/cpuset 已经限制,再绑会更复杂

## 八、NUMA 与调度

多路服务器不是"所有内存一样近":

```
CPU Socket 0 ─ 本地内存 0
CPU Socket 1 ─ 本地内存 1
```

CPU0 访问本地内存快,访问另一个 socket 的内存慢 1.3-2 倍。

调度器和内存分配器会尽量保持:

```
线程在哪个 NUMA node 跑
内存就在哪个 node 分配
```

但线程频繁迁移后,可能出现"CPU 在 node0,内存在 node1"。

排查:

```bash
numactl --hardware
numastat -p <pid>
numactl --cpunodebind=0 --membind=0 ./server
```

## 九、工程经验:线程数怎么设

### 9.1 CPU 密集型

```
线程数 ≈ CPU 核数
```

加多了只会争抢 CPU,增加上下文切换。

### 9.2 IO 密集型

```
线程数 = CPU 核数 × (1 + 等待时间 / 计算时间)
```

例如一次请求:

- 计算 5ms
- 等 DB 45ms

线程数 ≈ 核数 × (1 + 45/5) = 核数 × 10

但这是上限估算,真实还要看:

- DB 连接池
- 下游限流
- 内存
- p99 延迟
- 锁竞争

## 十、常见故障

### 10.1 CPU 100%,吞吐不涨

可能原因:

- run queue 太长
- 锁竞争导致自旋
- GC 线程抢 CPU
- 上下文切换过多
- 容器 CPU quota 被打满

看:

```bash
uptime          # load average
top -H -p PID   # 看线程
pidstat -u -w 1
perf top
```

### 10.2 p99 偶发抖动

可能原因:

- 线程被调度出去
- 跨核迁移导致 cache miss
- CPU 频率缩放
- 中断打到业务核
- cgroup throttle

看:

```bash
cat /sys/fs/cgroup/cpu.stat
perf sched latency
cat /proc/interrupts
```

### 10.3 D 状态很多

`D` = 不可中断睡眠,常见于磁盘/NFS/块设备 IO。

```bash
ps -eo pid,stat,wchan,cmd | grep ' D'
```

如果大量 D 状态,调度器不是根因,根因通常是存储卡住。

## 十一、总结

调度器不是抽象概念,它直接决定服务延迟:

- runnable 线程太多 → 排队
- 切换太频繁 → cache 失效
- 迁移太多 → NUMA 远程访问
- 实时策略用错 → 普通任务饿死
- 容器 CPU quota 太低 → throttle

> **调度优化的第一原则**:先减少不必要的可运行线程,再谈优先级和绑核。线程创建容易,让它们高效地共享 CPU 才难。
