# 信号

信号是 Unix 留给进程的一套"异步打断机制":段错误发 SIGSEGV,Ctrl+C 发 SIGINT,父进程等子进程靠 SIGCHLD,优雅停机靠 SIGTERM。它很强,也很危险,因为信号处理函数可能在程序任意位置插进来执行。这一篇讲清楚:**常见信号、默认动作、signal vs sigaction、异步信号安全、SIGCHLD 回收、优雅退出、signalfd 为什么更适合事件循环**。

> 一句话先记住:**信号 = 内核/进程给进程的异步通知**。它不是普通函数调用,可能打断任何代码,所以 handler 里只能做极少数 async-signal-safe 的事。**生产程序处理信号的正确姿势通常是:handler 只设置标志或写 eventfd/pipe,真正逻辑放主循环执行**。

---

## 一、信号从哪来

常见来源:

- 键盘:Ctrl+C → SIGINT,Ctrl+\ → SIGQUIT
- 终端关闭:SIGHUP
- kill 命令:`kill -TERM pid`
- 子进程退出:SIGCHLD
- 非法内存访问:SIGSEGV
- 除零:SIGFPE
- 写已关闭 pipe/socket:SIGPIPE
- 定时器:SIGALRM

查看:

```bash
kill -l
```

---

## 二、常见信号

| 信号 | 默认动作 | 用途 |
| --- | --- | --- |
| SIGTERM | 终止 | 优雅退出首选 |
| SIGKILL | 终止 | 强杀,不可捕获 |
| SIGINT | 终止 | Ctrl+C |
| SIGHUP | 终止 | 终端断开/重载配置 |
| SIGCHLD | 忽略 | 子进程状态变化 |
| SIGSEGV | core | 段错误 |
| SIGPIPE | 终止 | 写已关闭 pipe/socket |
| SIGSTOP | 停止 | 不可捕获 |
| SIGCONT | 继续 | 恢复执行 |
| SIGUSR1/2 | 终止 | 应用自定义 |

`SIGKILL` 和 `SIGSTOP` 不能捕获、不能忽略、不能阻塞。

## 三、sigaction

不要再用老的 `signal()`。用 `sigaction()`:

```c
static volatile sig_atomic_t stopping = 0;

static void on_term(int signo) {
    stopping = 1;
}

int main() {
    struct sigaction sa = {0};
    sa.sa_handler = on_term;
    sigemptyset(&sa.sa_mask);
    sigaction(SIGTERM, &sa, NULL);
    sigaction(SIGINT, &sa, NULL);

    while (!stopping) {
        work_once();
    }
    cleanup();
}
```

`sig_atomic_t` 保证 handler 和主程序之间读写这个标志是安全的。

## 四、异步信号安全

信号 handler 可能在 `malloc`、`printf`、持锁区中间打断程序。

handler 里不能随便做:

```c
printf("signal\n");  // 不安全
malloc(10);          // 不安全
pthread_mutex_lock   // 不安全
```

安全做法:

- 设置 `volatile sig_atomic_t` 标志
- `write` 一个字节到 pipe/eventfd
- 尽快返回

```c
static int pipefd[2];

static void on_signal(int signo) {
    char c = (char)signo;
    write(pipefd[1], &c, 1); // write 是 async-signal-safe
}
```

主循环读 pipe,再做真正逻辑。

## 五、SIGCHLD 与僵尸进程

子进程退出后,内核保留退出状态,等父进程 `wait`。父进程不 wait,子进程变 zombie。

正确回收:

```c
static void on_chld(int signo) {
    int saved = errno;
    while (waitpid(-1, NULL, WNOHANG) > 0) {}
    errno = saved;
}
```

为什么 while:

- 一个 SIGCHLD 可能对应多个子进程退出
- 信号不排队(普通信号会合并)

如果不需要子进程状态,也可以设置:

```c
signal(SIGCHLD, SIG_IGN);
```

但不同 Unix 语义细节有差异,服务端更推荐显式 wait。

## 六、SIGPIPE

写一个对端已关闭的 socket/pipe,可能收到 SIGPIPE,默认动作是终止进程。

网络服务通常忽略:

```c
signal(SIGPIPE, SIG_IGN);
```

或发送时:

```c
send(fd, buf, len, MSG_NOSIGNAL);
```

否则一个客户端断开就可能把服务打死。

---

## 七、优雅退出

容器/K8s 停服务通常发 SIGTERM:

```
SIGTERM
  → 停止接新请求
  → 关闭 listen socket
  → 等 in-flight 请求完成
  → flush 日志/指标
  → 退出
```

如果超时,K8s 再发 SIGKILL。

所以服务必须处理 SIGTERM,不能只处理 Ctrl+C(SIGINT)。

---

## 八、signalfd

Linux 可以把信号变成 fd:

```c
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigprocmask(SIG_BLOCK, &mask, NULL);

int sfd = signalfd(-1, &mask, SFD_CLOEXEC | SFD_NONBLOCK);
```

然后把 `sfd` 放进 epoll。

优点:

- 信号处理回到普通事件循环
- 不用写复杂 handler
- 更适合网络服务

## 九、core dump

SIGSEGV/SIGABRT 等会产生 core dump(如果开启):

```bash
ulimit -c unlimited
cat /proc/sys/kernel/core_pattern
gdb ./app core
```

生产环境常由 systemd-coredump 或 crash 收集。

## 十、常见坑

1. **handler 里 printf/malloc/加锁**:可能死锁或破坏状态。
2. **忘记 wait 子进程**:僵尸堆积。
3. **不处理 SIGTERM**:容器停机只能强杀。
4. **被 SIGPIPE 杀死**:网络服务常见低级错误。
5. **以为信号可靠排队**:普通信号会合并,实时信号才排队。
6. **多线程信号投递混乱**:信号可能投到任意未屏蔽该信号的线程。

> **结论**:信号适合做通知,不适合做业务逻辑。handler 越短越好,复杂逻辑交给主循环或专门线程处理。
