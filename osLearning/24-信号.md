# 信号

你其实每天都在用信号(signal),只是没意识到:Ctrl+C 是给进程发 SIGINT,`kill <pid>` 默认发 SIGTERM,段错误的本质是内核给你的进程发了一个 SIGSEGV,`kill -9` 就是大名鼎鼎的 SIGKILL。但大多数人对信号的理解就停在"kill -9 万能",直到某天踩到真坑:服务被强杀,半截数据写进了文件;handler 里随手加了一行日志,程序开始偶发死锁;fork 出来的子进程退完了没人收尸,`ps` 里僵尸排成一排。这一篇要回答的问题是:**信号到底是什么时候、以什么方式插进你的程序的,以及为什么 handler 里几乎什么都不能干**。

> 一句话先记住:**信号是内核给进程的"异步打断",它可以在你任意两条指令之间插进来执行**。正因为"任意",handler 里只能做极少数 async-signal-safe 的事。**生产程序处理信号的正确姿势是:handler 只设一个标志、或往 pipe/eventfd 写一个字节,真正的逻辑回主循环做**。

---

## 一、为什么要有信号:进程也需要"中断"

第 05 篇讲过,硬件靠中断打断 CPU——网卡来包了不用 CPU 轮询,中断告诉它。进程层面有一模一样的需求:**内核或别的进程想"叫住"一个正在跑的进程**——你按了 Ctrl+C、子进程退出了、它访问了非法内存、运维要它下班。

没有信号的世界长什么样?进程只能自己不停轮询:"有人要杀我吗?有子进程退出吗?"——每个程序的主循环里都得塞一堆检查代码,而且检查间隔再短也有延迟。信号把这件事反过来:**进程正常跑自己的,事件来了内核负责打断它**。

所以信号的准确定位是:**用户态的"软件中断"**——硬件中断打断 CPU,信号打断进程。理解了这个对应关系,信号所有的怪脾气(异步、不可预测插入点、handler 限制多)就都有了出处:**中断处理函数不能睡眠、要快进快出,信号 handler 同理**。

## 二、信号的一生:产生 → 挂起 → 投递

信号不是"发出去就立刻执行",它有三个阶段,**理解投递时机是理解信号一切行为的钥匙**:

```
产生(generate)            挂起(pending)               投递(deliver)
─────────────────         ──────────────────          ─────────────────────────
kill() / 键盘 / 硬件异常 →  记在 task_struct 的         进程从内核态返回用户态的瞬间,
子进程退出 / 定时器到期      pending 位图里               内核检查 pending & ~blocked:
                           (每种信号只占 1 个 bit!)      有未屏蔽的挂起信号
                                                        → 修改用户态栈,先跳去执行 handler
                                                        → handler 返回后再回到被打断的指令
```

三个关键推论:

1. **投递时机是"从内核态返回用户态时"**——syscall 返回、时间片中断返回、缺页处理返回,都是检查点。一个进程每秒要进出内核成千上万次,所以信号"感觉上"是立刻送达的,但**严格说有延迟**。一个纯计算、不进内核的死循环,也会被时钟中断拽进内核,所以照样能收到信号。
2. **普通信号是位图,不排队**。pending 位图里每种信号只有 1 个 bit——连发 5 个 SIGCHLD,进程可能只看到 1 次。**这就是后面 SIGCHLD 必须用 while 循环收尸的根本原因**。实时信号(SIGRTMIN ~ SIGRTMAX)才是真正排队的,还能携带数据。
3. **屏蔽(block)不是丢弃**。`sigprocmask` 屏蔽的信号会一直挂起,解除屏蔽的瞬间投递。这是后面 signalfd 的玩法基础。

```bash
kill -l        # 列出所有信号(1-31 标准信号,34+ 实时信号)
```

## 三、常见信号一张表:认识你的对手

| 信号 | 默认动作 | 触发场景 |
| --- | --- | --- |
| SIGTERM | 终止 | `kill` 默认信号,**优雅退出首选** |
| SIGKILL | 终止 | `kill -9`,**不可捕获、不可忽略、不可屏蔽** |
| SIGINT | 终止 | Ctrl+C |
| SIGQUIT | 终止 + core | Ctrl+\ |
| SIGHUP | 终止 | 终端断开;daemon 界约定俗成用它"重载配置" |
| SIGCHLD | 忽略 | 子进程退出/暂停/恢复 |
| SIGSEGV | 终止 + core | 非法内存访问(详见 06 篇缺页中断) |
| SIGFPE | 终止 + core | 除零等算术异常 |
| SIGPIPE | 终止 | 写一个对端已关闭的 pipe/socket |
| SIGALRM | 终止 | `alarm()` / `setitimer()` 定时器到期 |
| SIGSTOP / SIGCONT | 停止 / 继续 | Ctrl+Z 发的是 SIGTSTP(可捕获),SIGSTOP 不可捕获 |
| SIGUSR1 / SIGUSR2 | 终止 | 留给应用自定义(nginx 用 USR1 重开日志) |

默认动作就五类:**终止、终止 + core dump、忽略、停止、继续**。

**SIGKILL 和 SIGSTOP 是仅有的两个"三不能"信号**——不能捕获、不能忽略、不能屏蔽。这是内核留给管理员的最后手段:无论进程多顽固,`kill -9` 必死。

但"必死"是有代价的,这是个名场面:**`kill -9` 杀掉的进程没有任何机会清理现场**——用户态缓冲区里还没 flush 的日志直接蒸发,写到一半的文件留下半截脏数据,持有的锁文件没人删,数据库连接没有挥手就消失。多少"重启后数据损坏"的事故,源头就是运维习惯性 `kill -9`。**正确的肌肉记忆是:先 `kill`(SIGTERM)给进程机会善后,等几秒不退再 `kill -9`**——systemd 和 K8s 都是这么干的(见第八节)。

## 四、装 handler:别再用 signal(),用 sigaction()

历史上的 `signal()` 函数在不同 Unix 上语义不一致(有的执行一次后自动复位、有的不会自动重启被打断的 syscall),**新代码一律用 `sigaction()`**:

```c
static volatile sig_atomic_t stopping = 0;

static void on_term(int signo) {
    stopping = 1;                  // 只干这一件事
}

int main(void) {
    struct sigaction sa = {0};
    sa.sa_handler = on_term;
    sigemptyset(&sa.sa_mask);      // handler 执行期间额外屏蔽哪些信号
    sa.sa_flags = SA_RESTART;      // 被打断的慢 syscall 自动重启
    sigaction(SIGTERM, &sa, NULL);
    sigaction(SIGINT, &sa, NULL);

    while (!stopping) {
        work_once();
    }
    cleanup();                     // 真正的清理逻辑在主循环之后
}
```

两个细节值得停一下:

- **`volatile sig_atomic_t`** 是 C 标准里唯一保证"handler 写、主程序读"安全的类型。`volatile` 防编译器把它优化进寄存器,`sig_atomic_t` 保证读写是原子的——不会读到写了一半的值。
- **`SA_RESTART`**:信号会打断阻塞中的慢 syscall(read / accept / wait……),不加这个 flag 的话,syscall 会返回 -1 且 `errno = EINTR`,你的代码若没处理 EINTR 就会误判成真错误。这是无数"偶现 bug"的来源——**要么加 SA_RESTART,要么所有阻塞调用都包一层 EINTR 重试**。

## 五、async-signal-safe:handler 里的天条

为什么 handler 里几乎什么都不能干?来推演一个真实死锁:

```
主程序:  malloc(100)
             → glibc 拿起 malloc 内部的锁
             → 正在整理空闲链表……
                  ↑ 此刻信号插进来!
handler:  printf("got signal\n")
             → printf 内部要分配缓冲区 → 调 malloc
             → malloc 去拿那把锁——锁在"自己"手里
             → 死锁。进程僵住,永远等不到那把永远不会释放的锁
```

**信号 handler 和被它打断的代码在同一个线程里**,普通的互斥锁救不了——同一个线程对同一把非递归锁加锁两次就是死锁。`malloc`、`printf`、`pthread_mutex_lock`,以及一切内部可能用锁或全局状态的函数,在 handler 里都是地雷:

```c
printf("signal\n");    // ❌ stdio 内部有锁和缓冲区
malloc(10);            // ❌ 分配器内部有锁
pthread_mutex_lock(&m);// ❌ 可能是自己持有的锁
syslog(...);           // ❌ 同理
```

POSIX 列了一份 async-signal-safe(异步信号安全)函数白名单(`man 7 signal-safety`),核心是 `write`、`_exit`、`kill`、`sigaction` 这类不碰全局状态的直系 syscall 包装。所以生产级写法只有两种:

**写法一:设标志**(上一节的代码)——handler 一行,主循环检查。

**写法二:self-pipe trick**——handler 往 pipe 写一个字节,主循环(往往是 epoll)读到再处理:

```c
static int pipefd[2];

static void on_signal(int signo) {
    char c = (char)signo;
    write(pipefd[1], &c, 1);       // write 在白名单上,安全
}
```

第二种的妙处是**把"异步信号"转成了"fd 可读事件"**,和网络事件统一进了一个事件循环——这正是第 13 篇说的"一切事件 fd 化"哲学。Linux 上它的官方升级版就是第十节的 signalfd。

> 记一条铁律:**handler 越短越好,理想长度是一行**。任何"在 handler 里写业务逻辑"的设计都是定时炸弹——炸的时机取决于信号恰好打断在哪一行,所以是最难复现的那类 bug。

## 六、SIGCHLD 与僵尸:父进程必须收尸

第 11 篇讲过:子进程退出后不会立刻消失,内核要保留它的退出状态(task_struct 残骸)等父进程 `wait`——这就是僵尸(zombie)进程。父进程不收尸,僵尸就一直占着 PID。**单个僵尸无害,但僵尸堆到几万个会耗尽 PID,系统再也 fork 不出新进程**。

内核的通知方式就是 SIGCHLD:子进程退出 → 父进程收到 SIGCHLD → 父进程 wait 收尸。标准写法:

```c
static void on_chld(int signo) {
    int saved = errno;                              // waitpid 会改 errno!
    while (waitpid(-1, NULL, WNOHANG) > 0) {}       // 注意是 while
    errno = saved;
}
```

三个为什么:

- **为什么 while**:第二节说过,普通信号不排队。5 个子进程几乎同时退出,SIGCHLD 位图只置 1 次,handler 只跑一次——**不循环 wait 就漏掉 4 个,留 4 个僵尸**。
- **为什么 WNOHANG**:没有可收的子进程就立刻返回,handler 不能阻塞。
- **为什么保存 errno**:handler 可能打断主程序的任意位置,主程序可能刚执行完一个 syscall、正准备检查 errno——handler 里的 waitpid 把 errno 改了,主程序就读到错的值。**handler 改了 errno 必须恢复,这是最隐蔽的一类信号 bug**。

如果你完全不关心子进程退出状态,还有个偷懒写法:

```c
signal(SIGCHLD, SIG_IGN);   // 告诉内核:子进程退了直接回收,别留僵尸
```

POSIX 保证这个语义,但历史上各 Unix 行为有差异,**服务端代码更推荐显式 wait**——你迟早会想知道子进程是正常退出还是被信号杀的。

## 七、SIGPIPE:网络服务的隐形杀手

名场面:一个写得好好的 HTTP 服务,某天突然整个进程消失,日志干干净净,什么 panic 都没有。排查半天,原因是**一个客户端在服务端发送响应的途中断开了连接**。

机制:往一个对端已经关闭的 socket/pipe 写数据,内核会给你发 SIGPIPE,而它的默认动作是——**终止进程**。也就是说,**任何一个客户端的不告而别,默认配置下都能杀死你的整个服务**。这个默认动作是给 shell 管道设计的(`cat bigfile | head` 里 head 退出后 cat 该跟着死),对网络服务纯属误伤。

两种解法,任选其一,但必须有一个:

```c
signal(SIGPIPE, SIG_IGN);              // 进程级:忽略,write 返回 -1 / EPIPE
send(fd, buf, len, MSG_NOSIGNAL);      // 调用级:这次 send 不触发 SIGPIPE
```

忽略之后,写失败会以 `EPIPE` 错误码的形式正常返回,你按错误处理就行。Go / Java / Node 的 runtime 都替你做了这件事,**但写 C/C++ 网络程序,这是 main 函数前三行的标配**。

## 八、优雅退出:SIGTERM 是云原生时代的下班铃

容器时代,信号最高频的用途是**优雅停机(graceful shutdown)**。K8s 删 Pod 的标准流程:

```
kubectl delete pod
   → 容器主进程收到 SIGTERM
   → 倒计时开始(terminationGracePeriodSeconds,默认 30s)
   → 时间内自己退出 → 善终
   → 超时还没退   → SIGKILL,强杀,没有任何商量
```

所以你的服务收到 SIGTERM 后应该走完这套动作:

```
SIGTERM
  → 停止接收新请求(关闭 listen socket / 从注册中心摘除)
  → 等 in-flight 请求处理完(设个上限,别死等)
  → flush 日志、指标、缓冲区
  → 关闭数据库连接
  → 退出
```

两个高频翻车点:

1. **只处理了 SIGINT 没处理 SIGTERM**——本地 Ctrl+C 测得好好的,一上 K8s 就被强杀。Ctrl+C 是 SIGINT,K8s 发的是 SIGTERM,**两个都得注册**。
2. **容器里进程是 PID 1 收不到默认信号处理**——PID 1 在 Linux 上是特殊的:没有显式注册 handler 的信号,**内核不会对 PID 1 执行默认动作**(防止 init 被随手杀掉)。所以一个没写信号处理的程序直接做容器主进程,SIGTERM 对它无效,每次停机都要等满 30 秒被 SIGKILL。解法:程序自己注册 SIGTERM handler,或者用 tini / dumb-init 当 PID 1 转发信号(详见 25 篇容器底层)。

另外提一句 SIGHUP 的现代用法:终端断开的本义早就淡了,daemon 界约定用它做**不停机重载配置**——`nginx -s reload` 底下就是给 master 进程发 SIGHUP。

## 九、多线程下的信号:到底投给谁

多线程程序的信号是个高频懵点。规则其实就两条:

- **面向线程的信号**(SIGSEGV 这类由某线程自己的动作触发的,以及 `pthread_kill` 指名道姓发的)→ 投给那个线程。
- **面向进程的信号**(`kill` 发的 SIGTERM / SIGINT / SIGCHLD……)→ 投给**进程里任意一个没有屏蔽该信号的线程**,内核随便挑。

"随便挑"意味着你的 SIGTERM 可能落在任何一个 worker 线程上,handler 和那个线程的局部状态搅在一起,行为不可预测。生产级多线程程序的标准解法是**收编**:

```
主线程启动时:所有线程屏蔽全部要管的信号(pthread_sigmask 继承给子线程)
   → 专门起一个"信号线程",用 sigwait() 同步地等信号
   → 信号来了,sigwait 返回,在普通线程上下文里从容处理
   → 没有 handler,没有 async-signal-safe 限制,想调什么调什么
```

**把异步信号变成同步事件,所有天条全部解除**——这和 self-pipe / signalfd 是同一个思想的三种实现。线程模型详见 12 篇。

## 十、signalfd:把信号变成 fd,回到事件循环

Linux 给"信号同步化"提供了官方武器——signalfd,把信号变成一个可以 read 的文件描述符:

```c
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGINT);
sigprocmask(SIG_BLOCK, &mask, NULL);       // 关键:先屏蔽,不让默认投递路径碰它

int sfd = signalfd(-1, &mask, SFD_CLOEXEC | SFD_NONBLOCK);
// 把 sfd 丢进 epoll,信号 = 一个普通的可读事件
```

注意第一步**必须先 sigprocmask 屏蔽**——否则信号还是会走老的异步投递路径,signalfd 就读不到了。

对网络服务来说这是最优雅的形态:**信号、定时器(timerfd)、线程通知(eventfd)、网络 IO,全部统一成 epoll 里的 fd 事件**,一个主循环按顺序处理,没有任何代码会被"异步插入"。epoll 详见 21 篇。

## 十一、core dump:SIGSEGV 的遗书

SIGSEGV / SIGABRT / SIGQUIT 这类信号的默认动作是"终止 + core dump"——把进程崩溃瞬间的内存快照写进文件,这是事后调试(post-mortem debugging)的核心物料:

```bash
ulimit -c unlimited                      # 允许生成 core(很多发行版默认是 0!)
cat /proc/sys/kernel/core_pattern       # core 文件写到哪、叫什么
gdb ./app core                           # 加载尸检:bt 看崩溃时的调用栈
```

生产环境一般不靠裸文件,而是 systemd-coredump 统一收集(`coredumpctl list` / `coredumpctl gdb`),或者容器环境把 core_pattern 指到收集管道。**线上偶现的段错误,没有 core dump 几乎不可能定位**——这一行 ulimit 配置的价值,等出事那天你会懂。

## 踩坑提醒

1. **`kill -9` 当默认操作**——SIGKILL 不给进程任何清理机会,缓冲区数据、锁文件、半截写入全部留在现场。先 SIGTERM,等不动再 -9。
2. **handler 里 printf / malloc / 加锁**——和被打断的代码抢同一把锁,死锁的复现条件是"信号恰好打断在那一行",基本无法调试。handler 只设标志或 write 一个字节。
3. **以为信号会排队**——普通信号是位图,连发只算一次。SIGCHLD 不用 while 循环 waitpid,必漏僵尸。
4. **handler 里改了 errno 不恢复**——主程序读到被污染的 errno,产生薛定谔的错误处理。
5. **不处理 EINTR 也不加 SA_RESTART**——信号打断慢 syscall 后返回 -1/EINTR,被当成真错误,偶现 bug 之王。
6. **网络服务不管 SIGPIPE**——一个客户端断开就能杀掉整个服务,C/C++ 网络程序 main 前三行必须 `SIG_IGN` 或处处 `MSG_NOSIGNAL`。
7. **只处理 Ctrl+C 不处理 SIGTERM**——本地测试通过,K8s 上每次停机都超时被强杀。
8. **容器 PID 1 不注册 handler**——PID 1 对未注册 handler 的信号免疫默认动作,SIGTERM 直接无效,要么自己注册要么上 tini。
9. **多线程程序到处装 handler**——进程级信号投给任意未屏蔽线程,行为随机。正解是全员屏蔽 + sigwait 专职线程,或 signalfd。
10. **不开 core dump**——`ulimit -c 0` 是很多环境的默认值,线上段错误连尸体都不留,神仙难救。
