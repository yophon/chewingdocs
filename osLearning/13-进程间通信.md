# 进程间通信

进程间通信(IPC,Inter-Process Communication)是**操作系统最古老也最丰富的话题**——POSIX / SysV 各有一套,加上 Linux 自家的 namespace、eBPF map、io_uring,种类繁多让人眼花。但**真实工程里 99% 用得到的就 5 种**:管道、Unix Socket、共享内存 + 信号量、消息队列、信号。这一篇按"性能 / 适用场景"把它们排个序,讲清楚什么时候该用什么——以及**为什么 Nginx Worker 之间用共享内存,Docker / Kubernetes 用 Unix Socket,而消息队列 SysV / POSIX 在生产几乎不用了**(用 Kafka / Redis 替代)。

> 一句话先记住:**IPC 的两个核心维度是"传输方式"和"同步方式"**。**共享内存最快**(零拷贝)但**要自己配同步**;**管道 / Unix Socket 内核帮做缓冲 + 同步**,简单但有拷贝;**消息队列 / 信号**用得越来越少,被现代 MQ 替代。**Unix Socket 比 TCP localhost 快 30-50%**——这就是 Docker / Nginx / Postgres 都首选 Unix Socket 的原因。

---

## 一、IPC 的全景

```
传输 + 同步:
  管道(Pipe / FIFO)        字节流,内核缓冲
  消息队列(SysV / POSIX)   离散消息,内核缓冲
  Unix Socket               字节流 / 数据报,本机网络抽象
  TCP / UDP                 通用网络
  
零拷贝(共享数据):
  共享内存(SysV / POSIX shm / mmap)
  
纯同步:
  信号量(Semaphore)
  信号(Signal)
  futex(用户态锁的内核支持)
  
其他:
  eventfd / signalfd / timerfd  把事件转成 fd
  io_uring                      Linux 5.x 的批量 IO
```

---

## 二、管道(Pipe):最古老的 IPC

### 2.1 匿名管道

```c
int fds[2];
pipe(fds);          // fds[0] 读端,fds[1] 写端
if (fork() == 0) {
    close(fds[0]);
    write(fds[1], "hello", 5);
} else {
    close(fds[1]);
    char buf[6];
    read(fds[0], buf, 5);
}
```

**特点**:

- **半双工**(单向)
- **只能在有亲缘关系的进程间用**(fork 共享 fd)
- 内核维护一个 64 KB(默认)的 ring buffer
- 写端关闭 → 读端 EOF;读端关闭 → 写端 SIGPIPE

### 2.2 命名管道(FIFO)

```c
mkfifo("/tmp/myfifo", 0644);
// 任何进程都能 open 这个 FIFO
int fd = open("/tmp/myfifo", O_RDONLY);
```

**特点**:

- 无亲缘关系也能用
- 文件系统中有名字(看着像文件)
- 仍是字节流、半双工

### 2.3 shell 里的 |

```bash
ls | grep foo
```

**等价于**:

```c
pipe(fds);
fork();    // 子进程跑 ls,stdout 重定向到 fds[1]
fork();    // 子进程跑 grep,stdin 重定向到 fds[0]
```

**这就是 Unix 哲学**:小工具用管道串起来。

### 2.4 性能

```
吞吐:  1-3 GB/s(主要是 cache 内拷贝)
延迟:  几 μs / 次 read-write
适合:  父子进程间 / shell 管道
```

## 三、Unix Domain Socket:本机网络的最佳 IPC

### 3.1 用法

```c
// 服务端
int sock = socket(AF_UNIX, SOCK_STREAM, 0);
struct sockaddr_un addr = { .sun_family = AF_UNIX, .sun_path = "/tmp/sock" };
bind(sock, ..., ...);
listen(sock, 128);
int conn = accept(sock, ...);

// 客户端
int sock = socket(AF_UNIX, SOCK_STREAM, 0);
connect(sock, &addr, ...);
```

**接口和 TCP 完全一样**,只是 family 是 `AF_UNIX`。

### 3.2 为什么比 TCP localhost 快

```
TCP localhost:
  数据走完整 TCP/IP 栈
  → 三次握手
  → 拥塞控制 / 流控
  → 校验和 / 分片
  → 进 socket buffer

Unix Socket:
  绕过 TCP/IP
  内核直接在 fd 之间拷贝数据
  → 比 TCP 快 30-50%
  → 延迟低(几 μs vs 几十 μs)
```

### 3.3 真实使用场景

- **Docker daemon socket**:`/var/run/docker.sock`
- **K8s kubelet 与 CRI runtime**:Unix Socket
- **Nginx 与 PHP-FPM**:fastcgi over unix socket
- **Postgres 客户端**:默认走 unix socket(`/var/run/postgresql/.s.PGSQL.5432`)
- **systemd**:控制接口
- **Wayland**:GUI 协议

**任何"本机两个进程通信" 优先考虑 Unix Socket**,而不是 TCP localhost。

### 3.4 SOCK_DGRAM(数据报)

```c
int sock = socket(AF_UNIX, SOCK_DGRAM, 0);
```

**类似 UDP**,但本机内可靠(不会丢包)。**适合发送固定大小的消息**。

### 3.5 传 fd:Unix Socket 的隐藏特性

**Unix Socket 能在进程间传递 fd**(其他 IPC 不行):

```c
sendmsg(sock, &msg, 0);   // msg.msg_control 含 SCM_RIGHTS
```

**用途**:

- Nginx master 把 listen fd 传给 worker
- Docker 把容器 stdout 的 fd 传给 docker daemon
- 多进程共享同一个 socket 做负载均衡

## 四、共享内存:零拷贝的 IPC

### 4.1 SysV shm(老 API)

```c
int shmid = shmget(IPC_PRIVATE, 1024, 0666);
char *p = shmat(shmid, NULL, 0);
// p 指向共享内存,任意进程 attach 后都能读写
```

**老 API,接口繁琐,逐渐被 mmap 替代**。

### 4.2 POSIX shm(更现代)

```c
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 1024);
char *p = mmap(NULL, 1024, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
```

**`/dev/shm/` 下能看到**——实际上是 tmpfs。

### 4.3 mmap 文件

```c
int fd = open("/tmp/shm_file", O_RDWR | O_CREAT, 0666);
ftruncate(fd, 1024);
char *p = mmap(NULL, 1024, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
// 不同进程 mmap 同一文件 → 共享
```

**和 shm 的区别**:落到磁盘文件(可以 fsync 持久化)。

### 4.4 共享内存的同步

**共享内存只是"内存被多个进程看见"**,**写入 / 读取的协调要自己做**——用信号量 / mutex / atomic。

```c
// 数据结构:头部 + 数据
struct shared {
    pthread_mutex_t lock;       // 用 PTHREAD_PROCESS_SHARED 属性
    int counter;
};

pthread_mutex_lock(&shared->lock);
shared->counter++;
pthread_mutex_unlock(&shared->lock);
```

**没有同步的共享内存 = 数据竞争 = 数据腐烂**。

### 4.5 性能

```
共享内存读写:    内存速度,纳秒级
没有拷贝
比管道 / socket 快 10-100 倍
```

**用途**:

- 高频数据交换(行情数据、传感器数据)
- 大块数据共享(图像、视频帧)
- 实现自己的 IPC 协议

**典型工程**:

- **PostgreSQL**:多进程共享 buffer pool 用 shm
- **Nginx**:worker 共享统计数据
- **Redis**:fork 子进程做 BGSAVE 用 COW

## 五、信号量(Semaphore):同步原语

### 5.1 SysV 信号量

```c
int semid = semget(IPC_PRIVATE, 1, 0666);
semctl(semid, 0, SETVAL, 1);   // 初始化为 1

struct sembuf op = { 0, -1, 0 };   // P 操作
semop(semid, &op, 1);
// 临界区
op.sem_op = 1;                      // V 操作
semop(semid, &op, 1);
```

**老 API,复杂**。

### 5.2 POSIX 信号量

```c
sem_t *sem = sem_open("/mysem", O_CREAT, 0666, 1);
sem_wait(sem);    // P
// 临界区
sem_post(sem);    // V
sem_close(sem);
```

**更易用,推荐**。

### 5.3 用途

- 限制并发数(连接池 / 资源池)
- 生产者 / 消费者协调
- 共享内存的同步

## 六、消息队列(Message Queue):内核维护的离散消息

### 6.1 SysV / POSIX MQ

```c
mqd_t mq = mq_open("/myq", O_CREAT | O_WRONLY, 0666, NULL);
mq_send(mq, "hello", 5, 0);
mq_close(mq);
```

**特点**:

- 内核维护消息队列,**离散消息**(不是字节流)
- 有优先级
- 持久化到内核启动期(重启丢)

### 6.2 为什么生产几乎不用

**SysV / POSIX MQ 的局限**:

- 单机内,跨机不行
- 容量小(默认 16K 消息)
- 没有持久化(重启丢)
- 没有重试 / 死信
- 没有 fanout
- 调优困难

**现代替代**:

- 单机 IPC → Unix Socket / 共享内存
- 跨机 → Kafka / RabbitMQ / Redis Stream

> **2026 年看到 SysV / POSIX MQ 几乎都是历史代码**——新项目不用了。

## 七、信号(Signal):异步通知

```c
kill(pid, SIGUSR1);    // 给进程发信号

// 接收方
signal(SIGUSR1, handler);
void handler(int sig) {
    // 注意:严格限制能做的事
}
```

**特点**:

- 极轻量
- 异步(打断接收方)
- **载荷极小**(就一个数字)

**用途**:

- 通知(reload 配置 / 优雅退出)
- 跨进程"中断"

详见 24 篇信号。

## 八、eventfd / signalfd / timerfd:把事件转成 fd

Linux 特色——**让 epoll 能统一处理一切**:

### 8.1 eventfd

```c
int efd = eventfd(0, 0);
write(efd, &val, 8);    // 通知
read(efd, &val, 8);     // 接收
```

**用途**:线程间 / 进程间的"轻量唤醒",**比信号靠谱**。

### 8.2 signalfd

```c
sigset_t mask;
sigaddset(&mask, SIGTERM);
int sfd = signalfd(-1, &mask, 0);
read(sfd, &si, sizeof(si));   // 把信号当文件读
```

**用途**:**把信号转成 fd 事件**,放进 epoll,**统一处理**。

### 8.3 timerfd

```c
int tfd = timerfd_create(CLOCK_MONOTONIC, 0);
timerfd_settime(tfd, 0, &spec, NULL);
read(tfd, &expirations, 8);
```

**用途**:**把定时器转成 fd**,放进 epoll。

> **现代事件循环都用 \*fd 把所有事件统一成 fd**——这就是 epoll 大一统的设计哲学。详见 21 篇 epoll。

## 九、futex:用户态锁的内核支持

futex(Fast User-space muTEX)= **用户态加锁,只在竞争时才进内核**。

```
无竞争:        atomic CAS,纯用户态,~10 ns
有竞争:        futex_wait syscall,等待
```

**pthread_mutex 内部就是 futex**——这就是为什么"无竞争的锁很便宜"。

详见 14 篇锁。

## 十、IPC 选型决策树

```
问题:                          选择:
─────────────────────────────────────────
本机 + 父子进程 + 字节流  →    管道
本机 + 任意进程 + 字节流  →    Unix Socket(SOCK_STREAM)
本机 + 任意进程 + 数据报  →    Unix Socket(SOCK_DGRAM)
本机 + 高频大数据         →    共享内存 + 信号量
本机 + 简单事件通知       →    eventfd / signal
跨机器                    →    TCP / Kafka / RPC
跨语言                    →    gRPC / REST
微服务                    →    详见 backendLearning/32
```

## 十一、性能对比(粗略)

```
共享内存(纯读写):     1-10 ns
eventfd 通知:          ~500 ns
管道 read/write:       2-5 μs
Unix Socket:           5-10 μs
TCP localhost:         15-30 μs
TCP 跨机房:            1-100 ms
```

**性能差距 4 个数量级** —— 选错 IPC 性能崩。

## 十二、看 IPC 资源

```bash
ipcs                    # 看 SysV 共享内存 / 信号量 / 消息队列
ipcs -m                 # 共享内存
ipcs -s                 # 信号量
ipcs -q                 # 消息队列

ls /dev/shm             # POSIX 共享内存
ls /var/run/            # Unix Socket(常见路径)

lsof -U                 # 看所有 Unix Socket 连接
ss -x                   # 看 Unix Socket 状态
```

**坑**:**SysV 资源不会自动清理** —— 进程崩了,共享内存还在。**用 `ipcrm` 手动清**。

## 十三、容器内的 IPC

容器**默认 IPC namespace 隔离**——容器内的 SysV shm / 信号量看不到宿主或其他容器的。

```bash
docker run --ipc=host ...      # 共享宿主 IPC namespace
docker run --ipc=container:other ...  # 共享其他容器的
```

**用途**:

- 容器内多进程共享 shm(默认就行,容器内进程同 namespace)
- 跨容器共享(慎用,破坏隔离)

---

## 十四、踩坑提醒

1. **TCP localhost 当本机 IPC** —— 慢 30-50%,改 Unix Socket
2. **共享内存不配同步** —— 数据竞争,腐烂
3. **SysV shm 进程崩了不清理** —— `ipcs -m` 看到一堆"Owner: nobody"
4. **管道写满阻塞** —— 写端没消费,write 会卡死
5. **管道读端关了写 → SIGPIPE** —— 没 handle 直接退出
6. **不知道 Unix Socket 能传 fd** —— 多进程共享 socket 做 LB 的关键
7. **mmap 共享文件不 sync** —— 进程崩数据可能丢
8. **以为信号能传数据** —— 信号只能传一个 int,载荷用别的 IPC
9. **POSIX MQ 当生产 MQ 用** —— 没持久化没扩展性
10. **不用 eventfd / signalfd / timerfd** —— 写复杂事件循环时血亏
11. **容器忘了 IPC namespace** —— 跨容器 shm 失败
12. **futex 直接调** —— 90% 应该用 pthread_mutex 包装好的
