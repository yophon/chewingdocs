# 进程间通信

前两篇说清楚了一件事:线程在同一个地址空间里,通信免费;进程被虚拟内存彻底隔开(06 篇),**拿着指针也读不到对方一个字节**。这个隔离是 OS 的最大善政——一个进程崩了不带走全家。但进程总得协作:shell 要把 `ls` 的输出喂给 `grep`,Nginx worker 要共享统计,Docker CLI 要跟 daemon 说话。于是内核开了一批"官方通道",统称 IPC(Inter-Process Communication,进程间通信)。这个领域最劝退的不是难,是**乱**——POSIX 一套、SysV 一套、Linux 还加私货。好消息是:**真实工程里 99% 用得到的就 5 种**:管道、Unix Socket、共享内存 + 信号量、消息队列、信号。这一篇按性能和场景把它们排好队,讲清楚什么时候用哪个、各自的名场面事故长什么样。

> 一句话先记住:**IPC 只有两个核心维度——"数据怎么传"和"双方怎么同步"**。共享内存最快(零拷贝、纳秒级)但同步全靠自己;管道 / Unix Socket 由内核包办缓冲和同步,简单但每次都有拷贝;消息队列和信号正在退出历史舞台。**Unix Socket 比 TCP localhost 快 30-50%**——这就是 Docker、Nginx、Postgres 本机通信全选它的原因。

---

## 一、先看全景:两个维度,一张地图

```
传输 + 同步(内核帮你缓冲、帮你排队):
  管道(pipe / FIFO)        字节流,内核 ring buffer
  Unix Socket               字节流 / 数据报,本机版的 socket
  消息队列(SysV / POSIX)   离散消息,带优先级
  TCP / UDP                 通用网络(跨机器才用)

零拷贝(数据共享,同步自理):
  共享内存(SysV shm / POSIX shm / mmap 文件)

纯同步 / 纯通知(不传数据,只传"事件"):
  信号量(semaphore)
  信号(signal)
  futex(用户态锁的内核支撑,详见 14 篇)
  eventfd / signalfd / timerfd(把事件变成 fd)
```

读这张图的方法:**先问"我要传多少数据、多频繁",再问"谁等谁"**。下面从最古老的讲起。

## 二、管道(Pipe):Unix 的第一件传家宝

### 2.1 匿名管道

管道是内核里一个 ring buffer(默认 64 KB),一头写一头读:

```c
int fds[2];
pipe(fds);                  // fds[0] 读端,fds[1] 写端
if (fork() == 0) {
    close(fds[0]);
    write(fds[1], "hello", 5);
} else {
    close(fds[1]);
    char buf[6];
    read(fds[0], buf, 5);
}
```

它的脾气要记牢:

- **半双工**(单向),要双向就开两根
- **只能在有亲缘关系的进程间用**——fd 是靠 fork 继承过去的,陌生进程拿不到
- 写端全关 → 读端 read 返回 0(EOF);**读端全关 → 写端收 SIGPIPE,默认直接死**
- buffer 满了,write 阻塞;buffer 空了,read 阻塞——内核替你做了流控

### 2.2 名场面:管道写满死锁

这套流控有个经典翻车姿势:用子进程跑命令,stdout 接到管道,打算等子进程退出后一次性读输出——

```
父进程:waitpid(child)  ← 等子进程退出
子进程:要输出 200 KB,写到 64 KB 管道满,write 阻塞
            ↓
子进程卡在 write 等人读 ── 父进程卡在 wait 等它死 → 互等,死锁
```

各语言的 subprocess 文档都警告"别先 wait 再读",根源就是这 64 KB。**正确姿势是边读边等**(或用 `communicate()` 这类封装)。

### 2.3 命名管道(FIFO)与 shell 的 |

亲缘限制可以用 FIFO 绕开——给管道在文件系统里起个名字,谁都能 open:

```c
mkfifo("/tmp/myfifo", 0644);
int fd = open("/tmp/myfifo", O_RDONLY);   // 任意进程都行
```

而你每天敲的 `ls | grep foo`,本质就是:

```c
pipe(fds);
fork();   // 子进程跑 ls,stdout 重定向到 fds[1]
fork();   // 子进程跑 grep,stdin 重定向到 fds[0]
```

**这就是 Unix 哲学的物理载体**:小工具 + 管道 = 任意组合。

### 2.4 性能

```
吞吐:1-3 GB/s(瓶颈是内核和用户态之间的拷贝)
延迟:几 μs / 一次 read-write
适合:父子进程、shell 流水线
```

## 三、Unix Domain Socket:本机通信的默认答案

管道是字节流但只能单向、只认亲戚;TCP 谁都能连但走全套协议栈。Unix Socket 取了中间值:**接口和 TCP 一模一样,数据却抄近道**。

```c
// 服务端
int sock = socket(AF_UNIX, SOCK_STREAM, 0);
struct sockaddr_un addr = { .sun_family = AF_UNIX, .sun_path = "/tmp/sock" };
bind(sock, (struct sockaddr *)&addr, sizeof(addr));
listen(sock, 128);
int conn = accept(sock, NULL, NULL);

// 客户端
int sock = socket(AF_UNIX, SOCK_STREAM, 0);
connect(sock, (struct sockaddr *)&addr, sizeof(addr));
```

唯一区别是 family 写 `AF_UNIX`、地址是个文件路径。**所有为 TCP 写的代码逻辑(accept / epoll / 框架)原样能用**。

### 3.1 为什么比 TCP localhost 快

```
TCP localhost:                     Unix Socket:
  走完整 TCP/IP 协议栈                绕过整个协议栈
  三次握手                            内核直接在两个 fd 的
  拥塞控制 / 流控                     buffer 之间搬数据
  校验和 / 分片
  ↓                                  ↓
  延迟几十 μs                         延迟几 μs,快 30-50%
```

数据明明不出这台机器,TCP 却照样做握手、校验、拥塞控制——全是白干,Unix Socket 把这些全省了。

### 3.2 谁在用它

- **Docker**:`/var/run/docker.sock`,CLI 跟 daemon 的全部通信
- **K8s**:kubelet 与 CRI runtime 之间
- **Nginx ↔ PHP-FPM**:fastcgi over unix socket
- **Postgres**:本机客户端默认走 `/var/run/postgresql/.s.PGSQL.5432`
- **systemd** 的控制接口、**Wayland** 的 GUI 协议

结论很硬:**本机进程通信,默认选 Unix Socket,别用 TCP localhost**。

### 3.3 SOCK_DGRAM:本机版"可靠 UDP"

```c
int sock = socket(AF_UNIX, SOCK_DGRAM, 0);
```

像 UDP 一样按"整条消息"收发,但因为不出内核,**不丢包、不乱序**。适合固定大小的小消息,省去自己在字节流上切包。

### 3.4 隐藏大招:传 fd

Unix Socket 有个其他 IPC 都做不到的能力——**把文件描述符本身送给另一个进程**:

```c
sendmsg(sock, &msg, 0);   // msg.msg_control 里塞 SCM_RIGHTS + fd
```

对方收到的是一个在它的 fd 表里**真正可用**的描述符。这件事的副作用极其优雅:Nginx master 把 listen fd 发给各 worker,多进程共享同一个监听端口;Docker 把容器的 stdout fd 递给 daemon 收日志;多进程负载均衡靠 socket 在进程间流转,谁闲谁接。

## 四、共享内存:最快的 IPC,也是最危险的

前面所有方案都有共同成本:数据要经内核拷贝(用户态 → 内核 buffer → 用户态),数据量大、频率高时拷贝就是瓶颈。共享内存釜底抽薪:**让两个进程的页表指向同一块物理页**——之后读写就是普通内存访问,内核完全不在场,零拷贝。三种开法,效果相同:

```c
// 1. SysV shm(老 API,接口繁琐,逐渐淘汰)
int shmid = shmget(IPC_PRIVATE, 1024, 0666);
char *p = shmat(shmid, NULL, 0);

// 2. POSIX shm(现代推荐,/dev/shm 下可见,本质是 tmpfs)
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, 1024);
char *p = mmap(NULL, 1024, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);

// 3. mmap 普通文件(区别:落盘,可以 fsync 持久化)
int fd = open("/tmp/shm_file", O_RDWR | O_CREAT, 0666);
ftruncate(fd, 1024);
char *p = mmap(NULL, 1024, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
```

### 4.1 代价:同步全靠自己

最要命的是:**共享内存只解决"看得见",不解决"看得对"**。两个进程同时写同一个 counter,数据立刻腐烂(为什么会烂,14 篇拆给你看)。所以共享内存永远要配同步原语——跨进程版的 mutex 要加 `PTHREAD_PROCESS_SHARED` 属性:

```c
struct shared {
    pthread_mutex_t lock;   // mutexattr 设 PTHREAD_PROCESS_SHARED
    int counter;
};

pthread_mutex_lock(&shared->lock);
shared->counter++;
pthread_mutex_unlock(&shared->lock);
```

### 4.2 性能与用武之地

```
共享内存读写:纳秒级(就是内存速度),零拷贝
比管道 / socket 快 10-100 倍
```

适合**高频大块数据**:行情推送、传感器数据、图像 / 视频帧。真实工程里:

- **PostgreSQL**:多进程共享 buffer pool,整个数据库的心脏在 shm 里
- **Nginx**:worker 之间共享限流计数、统计数据
- **Redis BGSAVE**:fork + COW,本质是"内核免费送的一次性只读共享内存"(11 篇讲过)

## 五、信号量(Semaphore):配共享内存的"红绿灯"

信号量是个带原子加减的计数器:P 操作减 1(减到负就睡),V 操作加 1(唤醒等待者)。它不传数据,专管"谁先谁后、最多几个"。

```c
// SysV 版(老,繁琐,见到能认识就行)
int semid = semget(IPC_PRIVATE, 1, 0666);
semctl(semid, 0, SETVAL, 1);
struct sembuf op = { 0, -1, 0 };   // P
semop(semid, &op, 1);
/* 临界区 */
op.sem_op = 1;                     // V
semop(semid, &op, 1);

// POSIX 版(推荐)
sem_t *sem = sem_open("/mysem", O_CREAT, 0666, 1);
sem_wait(sem);    // P
/* 临界区 */
sem_post(sem);    // V
sem_close(sem);
```

三个典型用途:**保护共享内存**(初值 1 当互斥锁)、**限制并发数**(初值 N 当资源池)、**生产者消费者协调**(计数表示库存)。

## 六、消息队列:正在退役的老兵

内核维护的离散消息队列,带优先级:

```c
mqd_t mq = mq_open("/myq", O_CREAT | O_WRONLY, 0666, NULL);
mq_send(mq, "hello", 5, 0);
mq_close(mq);
```

听起来很美——离散消息(不用自己切包)、有优先级。但生产环境几乎见不到新代码用它,天花板太低:

```
只能单机,跨机免谈
容量小(默认 16K 条消息)
重启即丢,没有持久化
没有重试、死信、fanout,调优手段几乎为零
```

现代替代很明确:**单机要传消息 → Unix Socket / 共享内存;跨机 → Kafka / RabbitMQ / Redis Stream**。2026 年你在代码库里见到 SysV / POSIX MQ,十有八九是历史遗产,别学它。

## 七、信号(Signal):只敲门,不送货

```c
kill(pid, SIGUSR1);            // 发送方

signal(SIGUSR1, handler);      // 接收方
void handler(int sig) { /* 这里能做的事极其有限 */ }
```

信号是最轻量的 IPC,但**载荷只有一个信号编号**——它能告诉你"有事了",说不了"什么事"。所以它的正确定位是纯通知:`kill -HUP` 让 Nginx reload 配置、`SIGTERM` 请求优雅退出。想传数据?配别的通道。handler 里的种种禁忌(不能 malloc、不能加锁)详见 24 篇。

## 八、eventfd / signalfd / timerfd:把一切变成 fd

Linux 的事件循环(epoll)只认 fd,但通知、信号、定时器天生不是 fd——于是 Linux 干脆造了三个转换器:

```c
// eventfd:一个 8 字节计数器,write 即通知,read 即接收
int efd = eventfd(0, 0);
write(efd, &val, 8);
read(efd, &val, 8);

// signalfd:把信号变成可 read 的 fd,从此信号也能进 epoll
sigset_t mask;
sigaddset(&mask, SIGTERM);
int sfd = signalfd(-1, &mask, 0);
read(sfd, &si, sizeof(si));

// timerfd:把定时器变成 fd,到点了 fd 可读
int tfd = timerfd_create(CLOCK_MONOTONIC, 0);
timerfd_settime(tfd, 0, &spec, NULL);
read(tfd, &expirations, 8);
```

价值在于统一:**网络事件、跨进程通知、信号、定时器,全部塞进同一个 epoll 循环**,不用为每类事件单开线程。eventfd 还是比信号靠谱得多的"轻量唤醒"(不打断你、不限制 handler)。详见 21 篇 epoll。

## 九、futex:顺便认识一下锁的地基

上面反复出现"mutex / 信号量",它们在 Linux 上的共同地基是 futex(Fast Userspace muTEX):

```
无竞争:用户态 atomic CAS 搞定,~10 ns,根本不进内核
有竞争:futex_wait 进内核睡觉,等人 futex_wake
```

**pthread_mutex 内部就是 futex**——这就是"无竞争的锁几乎免费"这句话的出处。展开讲是下一篇(14 篇)的主线,这里挂个钩子。

## 十、选型决策树 + 性能账单

```
你的场景                          选择
─────────────────────────────────────────────
本机 + 父子进程 + 字节流     →    管道
本机 + 任意进程 + 字节流     →    Unix Socket(SOCK_STREAM)
本机 + 任意进程 + 离散消息   →    Unix Socket(SOCK_DGRAM)
本机 + 高频大块数据          →    共享内存 + 信号量
本机 + 纯事件通知            →    eventfd / 信号
跨机器                       →    TCP / Kafka / RPC
跨语言 / 微服务              →    gRPC / REST(详见 backendLearning/32)
```

性能量级(粗略,但数量级可靠):

```
共享内存读写:     1-10 ns
eventfd 通知:     ~500 ns
管道 read/write:  2-5 μs
Unix Socket:      5-10 μs
TCP localhost:    15-30 μs
TCP 跨机房:       1-100 ms
```

从头到尾**差 4 个以上数量级**。IPC 选错,后面代码写得再好也救不回来。

## 十一、运维视角:看见和清理 IPC 资源

```bash
ipcs                # SysV 三件套总览
ipcs -m / -s / -q   # 分别看共享内存 / 信号量 / 消息队列

ls /dev/shm         # POSIX 共享内存(tmpfs)
ls /var/run/        # Unix Socket 的常见聚集地
lsof -U             # 所有 Unix Socket 连接
ss -x               # Unix Socket 状态
```

一个埋人无数的事实:**SysV 资源不跟随进程生命周期**——进程崩了,它建的 shm / 信号量还赖在内核里。`ipcs -m` 里一排 "nobody" 持有的段就是这么来的,要用 `ipcrm` 手动清(POSIX shm 同理,/dev/shm 下的文件不删就一直在)。

容器还要多想一层:**容器默认有独立的 IPC namespace**,容器内的 shm / 信号量和宿主机、其他容器互不可见。容器内部多进程共享 shm 不受影响;要跨容器:

```bash
docker run --ipc=host ...              # 共享宿主机 IPC namespace
docker run --ipc=container:other ...   # 共享指定容器的
```

慎用,这是在主动凿穿隔离墙。

## 踩坑提醒

1. **本机通信用 TCP localhost**——白白慢 30-50%,改 Unix Socket
2. **共享内存不配同步**——数据竞争,腐烂得无声无息
3. **先 wait 子进程再读管道**——输出超 64 KB 就互等死锁,要边读边等
4. **管道读端关了还在写**——SIGPIPE 默认杀进程,没 handle 就是"莫名其妙退出"
5. **SysV 资源不清理**——进程崩了 shm 还在,`ipcs` 定期看、`ipcrm` 清
6. **不知道 Unix Socket 能传 fd**——多进程共享 listen socket 做负载均衡全靠 SCM_RIGHTS
7. **mmap 共享文件不 sync**——进程崩溃时脏页可能没落盘,数据丢
8. **想用信号传数据**——信号只有一个编号的载荷,数据走别的通道
9. **拿 POSIX MQ 当生产消息队列**——没持久化没扩展性,上 Kafka / Redis Stream
10. **手写事件循环不用 eventfd / signalfd / timerfd**——为通知和定时器单开线程,纯属血亏
11. **跨容器共享 shm 忘了 IPC namespace**——默认隔离,连不上不是你代码的错
12. **直接调 futex**——90% 的场景该用 pthread_mutex / 信号量这些包装好的
