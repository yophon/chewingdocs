# OS 在云原生时代

学完 CPU、内存、进程、线程、IO、文件、容器和 eBPF,你会发现云原生并没有绕开 OS,反而把 OS 问题放大了:容器 CPU quota 导致 p99 抖动,cgroup OOM 杀进程,OverlayFS 影响写放大,Service Mesh 增加 socket 和上下文切换,eBPF 接管网络和观测。最后一篇把这条线收住:**现代后端工程师为什么仍然必须懂 OS,以及未来 Linux 内核演进会把性能问题带向哪里**。

> 一句话先记住:**云原生不是 OS 的替代品,而是 OS 抽象的大规模编排**。K8s 调 Pod,本质还是调 Linux 进程;Service Mesh 转流量,本质还是 socket 和 iptables/eBPF;Serverless 冷启动,本质还是进程/容器/文件系统加载。**越往云上走,越需要用 OS 视角解释性能和稳定性**。

---

## 一、这 28 篇的主线

```
CPU:      程序如何真正执行
内存:     虚拟地址、页表、分配、缓存
并发:     进程、线程、锁、调度、协程
IO:       fd、文件系统、epoll、io_uring、零拷贝
工程:     信号、容器、性能工具、eBPF
```

这不是孤立知识点,而是一条性能路径:

```
代码慢
  → CPU cache miss?
  → syscall 太多?
  → 锁竞争?
  → 调度延迟?
  → page fault?
  → IO wait?
  → cgroup throttle?
  → 网络栈/文件系统?
```

---

## 二、云原生把 OS 问题重新包装

### 2.1 Pod 是进程组

K8s Pod 看起来是调度单元,落到节点上是:

- namespace
- cgroup
- rootfs
- pause container
- 一组进程

Pod Pending 是调度问题;Pod Running 后的性能,大多仍是 Linux 问题。

### 2.2 CPU limit 是调度问题

`resources.limits.cpu: 500m` 不是"半个 CPU 慢慢跑",而是 CFS quota:

```
一个周期内跑够 quota → throttle
```

这会把突发 CPU 变成延迟尖刺。

### 2.3 memory limit 是 OOM 边界

容器内超过 memory limit,触发 cgroup OOM。JVM/Go/Node 必须感知容器限制,否则会按宿主机资源估算堆和线程。

### 2.4 OverlayFS 是文件系统问题

容器可写层适合临时文件,不适合高频持久写。数据库、日志、对象数据应放 volume。

## 三、Service Mesh 和网络栈

Sidecar 模式:

```
应用 → localhost → sidecar → 网络 → sidecar → 应用
```

代价:

- 连接数增加
- socket buffer 增加
- syscall 增加
- TLS 加解密增加
- 上下文切换增加
- 延迟链路变长

Ambient mesh/eBPF 数据面尝试减少 sidecar 成本,但本质仍在 Linux 网络路径上做取舍。

---

## 四、Serverless 和冷启动

冷启动涉及:

- 拉镜像
- 解压/挂载 rootfs
- 创建 namespace/cgroup
- 启动进程
- 加载动态库
- JIT/运行时初始化
- 预热连接

优化手段:

- 更小镜像
- lazy pull
- snapshot
- Firecracker microVM
- 预热池
- 减少动态初始化

这些都离不开 OS 的进程、文件系统、虚拟化和调度。

## 五、现代 Linux 的方向

### 5.1 io_uring

把同步 syscall IO 推向异步队列模型。未来高性能存储、代理、数据库会更多使用。

### 5.2 eBPF

把观测、网络、安全逻辑从"改内核/sidecar"转向"安全内核程序"。

### 5.3 cgroup v2

统一资源控制层次,更适合容器和 systemd 管理。

### 5.4 Rust for Linux

降低内核内存安全风险,但不会一夜之间改写整个内核。

### 5.5 硬件趋势

- 多核更多,NUMA 更常见
- SSD/NVMe 延迟更低,syscall 更显贵
- SmartNIC/DPU 把网络处理下沉
- ARM 服务器普及,弱内存序更重要

## 六、后端工程师该形成的 OS 直觉

### 6.1 看到延迟,先问等待在哪里

```
CPU run queue?
锁?
IO?
网络?
GC?
cgroup throttle?
下游?
```

### 6.2 看到内存,先分清虚拟和物理

```
VIRT 不等于 RSS
RSS 不等于堆
page cache 不等于泄漏
container limit 不等于宿主机内存
```

### 6.3 看到并发,先问调度和共享

```
线程数是否超过核数太多?
锁是否竞争?
cache line 是否抖动?
协程是否被阻塞 syscall 卡住?
```

### 6.4 看到 IO,先问路径

```
page cache 命中?
fsync?
小 IO 还是大 IO?
epoll 还是线程池?
copy 是否打满 CPU?
```

## 七、学习边界

这套系列到这里已经足够支撑工程排障和性能调优。再往下有三个方向:

1. **内核源码**:调度器、内存管理、网络栈、文件系统源码。
2. **系统编程**:C/Rust 写网络库、存储引擎、runtime。
3. **性能工程**:benchmark、profiling、容量规划、SLO。

不要为了"懂内核"而逐行读源码。带问题读:

- 为什么 futex 卡住?
- 为什么 page cache 不回收?
- 为什么 TCP 重传?
- 为什么 fsync 抖动?

## 八、最后的排障模板

线上问题先收证据:

```bash
date
uptime
top -b -n1 | head
vmstat 1 5
free -h
df -h
iostat -x 1 5
pidstat -urdw -p <pid> 1 5
ss -s
dmesg -T | tail -100
```

再下钻:

- CPU → perf / flamegraph
- syscall → strace / bpftrace
- IO → iostat / biolatency
- 调度 → perf sched / runqlat
- 网络 → ss / tcpdump / tcpretrans
- 内存 → smaps / heap profile / OOM log

## 九、总结

OS 系列真正要建立的是一套心智:

```
应用现象
  ↓
运行时行为
  ↓
系统调用
  ↓
内核机制
  ↓
硬件约束
```

当你能把一个 p99 抖动问题从 HTTP handler 一路追到 cgroup throttle、futex、page fault、TCP retrans 或 fsync,你就不再是在"猜性能",而是在做工程。

> **结尾一句话**:现代工程不是离 OS 越来越远,而是抽象越来越厚。抽象可以帮你交付,但排障时必须有能力把它一层层拆回 Linux。
