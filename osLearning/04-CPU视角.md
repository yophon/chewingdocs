# CPU 视角

你写下 `a + b`,脑子里的画面是"CPU 做了一次加法"。真实的画面是:现代 CPU 正同时执行十几条指令、预测你接下来走哪个分支、把猜测的路径提前算掉、在几层缓存之间搬运数据——**你以为的"一行一行执行",是 CPU 精心维护的假象**。这个假象平时无害,但它解释了一个让无数人困惑的现象:**同样的算法、同样的数据量、同样的机器,仅仅因为内存访问模式不同,性能能差 10 倍**。这一篇讲程序员该懂的 CPU:多核、寄存器、缓存层级、流水线、乱序执行、SIMD——以及为什么所有性能优化的尽头都是四个字:**cache 友好**。

> 一句话先记住:**现代 CPU = 预测式 + 流水线式 + 多核的执行机器**,而它有一个致命短板——**内存墙:CPU 比内存快 100 倍**。CPU 等一次内存的时间能执行 100 条指令,所以 **cache 命中率决定一切,程序优化的终极目标不是少写代码,是让数据待在 L1**。

---

## 一、先看地形:CPU 的物理结构与内存墙

```
CPU Package(一颗 CPU)
├── Core 0
│   ├── L1d (32 KB, 1 cycle)
│   ├── L1i (32 KB)
│   ├── L2  (256 KB, 10 cycles)
│   └── 寄存器
├── Core 1
│   ├── L1d / L1i / L2
│   └── 寄存器
├── ...
├── L3  (32 MB, 40 cycles, 所有核共享)
└── Memory Controller → DRAM (200+ cycles)
```

每个核有自己的 L1 / L2,所有核共享 L3,L3 之外才是内存。把延迟摆成一排:

```
寄存器:        ~0.3 ns(1 cycle @ 3GHz)
L1 cache:      ~1 ns
L2 cache:      ~3 ns
L3 cache:      ~12 ns
DRAM:          ~100 ns
SSD (NVMe):    ~10 μs
HDD:           ~10 ms
```

寄存器到 DRAM,**300 倍的落差**。一次 cache miss 打到内存,CPU 干等的时间够跑 100 条指令——**这就是"内存墙"**。本篇后面的一切技术(缓存、流水线、乱序、预取)本质上都是 CPU 厂商对抗内存墙的手段,理解了这一点,后面全部顺理成章。

---

## 二、寄存器:最快,但只有一把

寄存器是 CPU 内部直连 ALU 的存储,x86_64 一共就这些:

```
通用寄存器:   rax / rbx / rcx / rdx / rsi / rdi / rsp / rbp / r8-r15(16 个,各 64 bit)
浮点 / SIMD:  xmm0-15 (128 bit) / ymm0-15 (256 bit) / zmm0-31 (512 bit)
段寄存器:     cs / ds / ss / es / fs / gs
程序计数器:   rip(指向下一条指令)
标志寄存器:   rflags(零标志、进位、负数等)
```

编译器最核心的工作之一就是**把你的变量分配到寄存器上**:

```c
int sum(int *arr, int n) {
    int s = 0;                    // s → rax
    for (int i = 0; i < n; i++)   // i → rcx
        s += arr[i];
    return s;
}
```

热点循环里的变量全在寄存器里转,一次内存都不碰——这是快的根源。但寄存器总共就 16 个,**活跃变量太多就装不下,多出来的被"溢出(spill)"到栈上**,访问从 ~0.3ns 变成走内存,性能掉一截。

顺带处决一个古董:`register int i;` 关键字——**现代编译器直接无视它**,它自己分配得比你好。

## 三、流水线:CPU 的装配线

CPU 执行一条指令要经过取指、解码、执行、访存、写回五个阶段。如果"执行完一条再开始下一条",每个阶段的硬件 4/5 时间在闲着。CPU 的解法和福特造汽车一样——**装配线**:

```
          时刻 1   2   3   4   5
指令 1:   F   D   X   M   W
指令 2:       F   D   X   M   W
指令 3:           F   D   X   M   W
指令 4:               F   D   X   M   W

F = Fetch(取指)  D = Decode(解码)  X = Execute(执行)
M = Memory(访存)  W = Write back(写回)
```

5 级流水线 = 5 条指令同时在不同工位上加工。现代 CPU 有 14-19 级流水线,吞吐极高——**前提是装配线别停**。让装配线停摆的有三大杀手:

### 杀手 A:分支预测失败

```c
if (x > 0) { ... } else { ... }
```

分支还没算出来,流水线后面的工位吃什么?CPU 的做法很大胆:**猜**——预测会走哪个分支,提前把那条路的指令塞进流水线。猜对了白赚;**猜错了整条流水线作废清空,重新装填,罚 10-20 个 cycle**。

如果你比 CPU 更了解你的分支,可以提示编译器:

```c
if (__builtin_expect(x > 0, 1)) { ... }   // 告诉它:这个条件几乎总是真
```

### 杀手 B:数据依赖

```c
a = b + c;
d = a + 1;   // 必须等 a 算完
```

第二条指令死等第一条的结果,流水线卡住。CPU 用**乱序执行 + 寄存器重命名**缓解(下一节),但依赖链太长的代码(比如链表遍历)神仙难救。

### 杀手 C:缓存未命中

访存指令突然要等 100+ cycle,整条流水线冻结。**三大杀手里它最凶,也是程序员唯一能大幅控制的——这就是性能优化的主战场**(第五节)。

---

## 四、乱序执行:CPU 偷偷换了你的指令顺序

```c
a = load(x);    // cache miss,要等 100 cycle
b = 1 + 2;      // 不依赖 a
c = b * 3;      // 依赖 b
d = a + 1;      // 依赖 a
```

CPU 看到这段代码的反应不是傻等,而是:load 发出去等着,**趁等待的功夫把 b、c 先算了**,等 a 的数据回来再算 d。这就是乱序执行(Out-of-Order Execution)——只要结果对,过程随便排。

**你看到的"顺序执行"是 CPU 在退休(retire)阶段重新整理出来的假象**。单线程下这个假象天衣无缝;但多线程下,**一个核的乱序写入可能被另一个核以错误的顺序观察到**——这就是"内存可见性问题"的硬件根源,详见 15 篇内存屏障。

### 推测执行,以及它捅出的大娄子

CPU 不止乱序,还**赌未来**:分支预测之后,它会把"猜测要走的路径"提前真的执行掉(推测执行,Speculative Execution),猜对了直接用,猜错了回滚。

听起来很美,直到 2018 年——**Spectre / Meltdown 漏洞**:回滚能撤销寄存器的修改,**却撤销不了缓存里留下的痕迹**。攻击者构造"必然猜错的分支",让 CPU 推测执行一段访问敏感数据的代码,再测量缓存命中时间,**把无权读的数据一个比特一个比特地侧信道泄露出来**。全球云厂商紧急打补丁,代价是 syscall 性能普遍下降——一个硬件优化技巧引发的全行业地震。

## 五、缓存:程序员能控制的最关键变量

流水线和乱序是 CPU 自己的事,你插不上手;**cache 是你唯一能大幅影响的**——而它恰恰最值钱。

### 5.1 cache line:CPU 的最小进货单位是 64 字节

CPU 从不按字节读内存,**最小单位是 64 字节一行(cache line)**:

```c
char arr[1024];
char x = arr[0];   // 实际把 arr[0..63] 整行搬进了 cache
```

这意味着:你读了 arr[0],arr[1..63] 已经免费躺在 cache 里了。**顺序访问几乎免费,跳着访问每跳一次付一次全价**——这一条推出了下面所有优化。

### 5.2 顺序 vs 随机:5-10 倍的差距

```c
// 顺序:cache 友好
for (int i = 0; i < N; i++)
    sum += arr[i];

// 随机:cache 杀手
for (int i = 0; i < N; i++)
    sum += arr[hash(i)];
```

同样的加法次数,**顺序比随机快 5-10 倍**——不是 CPU 算得慢,是随机访问每次都 cache miss。这也是"数组普遍吊打链表"的真正原因:不是 O 记号,是 cache。

### 5.3 二维数组:行还是列,差 10 倍

```c
int matrix[1024][1024];

// 行优先(C 的内存布局就是按行存的):快
for (int i = 0; i < 1024; i++)
    for (int j = 0; j < 1024; j++)
        sum += matrix[i][j];

// 列优先:慢 5-10 倍
for (int j = 0; j < 1024; j++)
    for (int i = 0; i < 1024; i++)
        sum += matrix[i][j];
```

两段代码加法次数一模一样,差别只在内层循环方向:按行扫是顺序访问,cache 一路命中;按列扫每次跳 4KB,**每次访问都是一次 cache miss**。

### 5.4 数据结构层面:AoS vs SoA

```c
// 结构体数组(AoS):只想要 age,却把 name 也搬进了 cache
struct Person { char name[32]; int age; };
Person people[1000];
for (i = 0; i < 1000; i++) sum += people[i].age;

// 数组结构体(SoA):cache 里全是有用的数据
int ages[1000];
char names[1000][32];
for (i = 0; i < 1000; i++) sum += ages[i];
```

AoS 模式下,每 36 字节里只有 4 字节是你要的,**cache 利用率只有 11%**;SoA 是 100%。**SoA 比 AoS 快 2-5 倍**——游戏引擎(ECS 架构)、ML 框架(列式存储)全是这个思路。

### 5.5 prefetch:提前告诉 CPU 要什么

```c
for (int i = 0; i < N; i++) {
    __builtin_prefetch(&arr[i + 8]);  // 提前 8 个迭代发预取
    sum += arr[i];
}
```

不规则访问(哈希表、图遍历)场景下,手动 prefetch 能快 20-50%。但**顺序访问 CPU 的硬件预取器自己就会做,你再加反而碍事变慢**——先 perf 确认有 miss,再上 prefetch。

## 六、多核:缓存一致性与 false sharing

多核带来一个新问题:Core 0 把 x 改成 1(在它自己的 L1 里),Core 1 读 x,会读到旧值 0 吗?

**不会——MESI 协议**在硬件层保证各核的 cache 看到一致的值:一个核要写某个 cache line,得先让其他核手里的副本全部失效。一致性有了,但代价埋下了:**"让别人失效"这个动作很贵**——而最冤的付费方式,叫伪共享。

### false sharing:没共享数据,却付了共享的钱

```c
struct {
    int a;  // 线程 0 疯狂改这个
    int b;  // 线程 1 疯狂改这个
} shared;
```

a 和 b 毫无关系,两个线程各改各的,理论上零冲突。但**它们挤在同一个 64 字节 cache line 里**——Core 0 改 a,MESI 把 Core 1 的整行打失效;Core 1 改 b,又把 Core 0 的打失效。cache line 在核间疯狂弹跳,**性能比直接加锁还差**。这就是 01 篇"8 线程比 4 线程还慢"事故的真凶。

修复简单粗暴——padding 把它们隔开:

```c
struct {
    int a;
    char padding[60];   // 把 a 撑满一整行
    int b;
} shared;
```

**这是高性能并发代码的必修课**——Java 的 `@Contended`、各语言并发库里的 CachePadded 类型,全是干这个的。详见 10 / 15 篇。

## 七、SIMD:一条指令算一排数

SIMD(Single Instruction Multiple Data)= 一条指令同时操作多个数据:

```
普通加法:   a + b → c                           (1 个 int)
SSE2:       [a0,a1,a2,a3] + [b0,b1,b2,b3]       (4 个 int 同时加)
AVX-512:    16 个 int / 8 个 double 同时加
```

用法两条路:

```c
// 路 1:编译器自动向量化(最常见)
gcc -O3 -mavx2 ...

// 路 2:手写 intrinsics(热点函数)
#include <immintrin.h>
__m256i a = _mm256_load_si256(...);
__m256i b = _mm256_load_si256(...);
__m256i c = _mm256_add_epi32(a, b);
```

应用场景:数值计算 / 矩阵运算、图像视频处理、字符串搜索解析、哈希 / CRC、ML 推理。**你大概率不用手写,但你用的好库全在用**——BLAS / NumPy / OpenCV / FFmpeg 的快,一半是 SIMD 的快。

一个反直觉的坑:**AVX-512 指令功耗极高,会触发 CPU 整体降频**——某些负载用了它反而全局变慢。"更宽的指令"不等于"更快的程序"。

---

## 八、超线程:16 核里有 8 个是"影分身"

超线程(SMT / Hyper-Threading)= 一个物理核模拟成 2 个逻辑核——共享执行单元和 cache,各自有独立的寄存器集合:

```
Linux 显示 16 核,实际是 8 物理核 × 2
实际计算能力 ≈ 单核的 1.2-1.4 倍,不是 2 倍
```

原理:线程 A 等内存(等就是闲)的时候,让线程 B 占用空着的执行单元——**填的是内存墙造成的空泡**。所以 IO 密集 / 访存密集负载收益明显;而计算密集负载(数据库 / HPC)两个逻辑核互抢 cache 和执行单元,**有些场景关掉超线程反而更快**。

## 九、perf stat:三个数字看穿性能

光讲理论没用,得能测。`perf stat` 是看 CPU 真实行为的第一工具:

```bash
perf stat ./your_program

#  10,234,567,890 cycles
#  20,123,456,789 instructions              # 2.0 IPC
#   1,234,567,890 cache-references
#     123,456,789 cache-misses              # 10% miss rate
#      12,345,678 branch-misses             # 1% misprediction
```

三个关键指标,对应本篇的三大杀手:

- **IPC(每周期指令数)**:理想 4+;**< 1 说明 CPU 大部分时间在等**(内存或分支)
- **cache miss rate**:< 5% 还行;> 10% 该改数据结构 / 访问模式了
- **branch misprediction**:< 1% 正常;> 5% 该优化分支了

**这三个数足够定位 80% 的 CPU 性能问题**。详见 26 篇性能工具。

## 十、频率、功耗,以及多核时代为什么到来

两个工程常识收尾。

**频率不是恒定的**。现代 CPU 动态调频:空闲降到 800MHz 省电、高负载 Boost 到 4-5GHz、多核齐上又降回防过热。**性能测试必须锁频**(`cpupower frequency-set -g performance`),否则数字随室温飘。

**为什么 CPU 频率从 2005 年起十年卡在 3-4GHz?** 因为**功耗 ∝ 频率³**——频率翻倍,功耗 8 倍,散热做不到。撞上功耗墙后,厂商集体换路线:频率不涨,核数狂涨。**这就是"并行编程"成为这二十年主旋律的原因**——免费的单核性能午餐结束了,性能要靠你自己并行出来。

### 顺带:ARM vs x86

2025 年的格局:

```
x86_64(Intel / AMD):    服务器主流,生态最全,功耗高
ARM64(Apple / 高通 / 鲲鹏):移动端统治,服务器迅速崛起(AWS Graviton / Apple Silicon)
RISC-V:                  开源新势力,嵌入式起步
```

写跨平台代码要注意的差异:**ARM 内存模型更弱**(并发代码的坑,详见 15 篇)、SIMD 指令集不同(NEON / SVE,不是 AVX)、部分 syscall 编号不同、`gcc -march` 参数不同。好消息:Linux 抽象掉了 95% 的差异。

## 十一、踩坑提醒

1. **以为代码按顺序执行**——CPU 乱序 + 推测执行,"顺序"是退休阶段整理出的假象
2. **不考虑 cache 写代码**——同样的算法,访问模式不同性能差 10 倍
3. **结构体字段乱排不对齐**——浪费内存还污染 cache
4. **多线程写同一 cache line(false sharing)**——没共享数据也互相拆台,比加锁还慢
5. **热路径塞满难预测的分支**——预测失败一次罚 10-20 cycle
6. **二维数组按列访问**——每次访问跳 4KB,cache miss 爆表
7. **不开 `-O2 / -O3` 就谈性能**——编译器优化能差 5 倍,白测
8. **看不懂 perf stat**——IPC、cache miss、branch miss 三个数就能定位 80% 的问题
9. **以为超线程总是赚**——计算密集场景关掉反而快
10. **不锁频就跑 benchmark**——动态调频让结果飘到没法比
11. **滥用 AVX-512**——高功耗触发全核降频,局部加速换来全局变慢
12. **以为 ARM 和 x86 一样**——内存模型更弱,并发代码跨平台要格外小心
