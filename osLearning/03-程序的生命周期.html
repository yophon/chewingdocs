程序的生命周期 ​

gcc hello.c -o hello && ./hello 看起来三行命令,背后是 OS 最复杂的一段流程——从源代码到内存中跑起来的进程,要经过预处理、编译、汇编、链接、装载、动态链接、初始化、main 调用八步。每一步都有"翻车点":链接器找不到符号、装载时缺动态库、ASLR 让你 debug 时地址永远不一样、Hello World 实际上加载了 30+ 个 .so 文件。理解这一段,你才能理解后面所有内存布局 / 共享库 / 容器镜像 / GDB 调试问题的根因。

一句话先记住:程序的生命 = 编译期(源码 → ELF 文件)+ 运行期(execve → 装载 → 动态链接 → 初始化 → main)。ELF 是 Linux 的"程序长什么样"的标准格式,搞懂 ELF = 搞懂可执行文件、共享库、core dump 的统一结构。99% 的程序员从来没真看过 ELF,但核心库 / 调优 / 安全分析全靠它。

一、八步走 ​

源代码 (hello.c)
   ↓ 预处理 (cpp)
预处理后的源代码 (hello.i)
   ↓ 编译 (cc1)
汇编代码 (hello.s)
   ↓ 汇编 (as)
目标文件 (hello.o)
   ↓ 链接 (ld)
可执行文件 (hello, ELF 格式)
   ↓ execve syscall
内核装载到内存
   ↓ 转交动态链接器
动态库映射 + 符号解析
   ↓ 初始化(.init / 全局对象构造)
main() 被调用
   ↓
程序运行
   ↓ exit / _exit
进程结束

前 4 步是编译期,后 4 步是运行期。

二、ELF:Linux 的可执行文件格式 ​

ELF(Executable and Linkable Format)是 Linux / BSD / Solaris 共用的标准。三种文件都是 ELF:

.o   目标文件(可链接)
.so  共享库(动态链接 + 可执行)
exec 可执行文件(可装载运行)

2.1 ELF 的物理结构 ​

┌──────────────────┐
│   ELF Header     │ ← 16 字节魔数 + 元信息
├──────────────────┤
│  Program Header  │ ← 装载视图(execve 看的)
│      Table       │
├──────────────────┤
│      .text       │ ← 代码段(机器码)
│      .rodata     │ ← 只读数据(字符串字面量)
│      .data       │ ← 已初始化的全局变量
│      .bss        │ ← 未初始化的全局变量(占位,文件里 0 字节)
│      .symtab     │ ← 符号表
│      .rel.text   │ ← 重定位信息
│      .debug_info │ ← 调试信息(-g 时有)
├──────────────────┤
│  Section Header  │ ← 链接视图(ld 看的)
│      Table       │
└──────────────────┘

2.2 两种"视图":Section vs Segment ​

ELF 有意思的设计——同一份内容,从两个角度看:

Section(节) = 链接视图——给链接器 ld 用的,粒度细(.text / .data / .bss ...)

Segment(段) = 装载视图——给装载器 / 内核用的,粒度粗(几个 section 合并,按可读 / 可写 / 可执行的权限分组)

典型 segment 划分:

LOAD (R-X):  .text + .rodata     ← 只读可执行
LOAD (RW-):  .data + .bss        ← 可读可写
DYNAMIC:     .dynamic             ← 动态链接信息
INTERP:      .interp              ← 动态链接器路径

为什么这么分:链接时关心细节,装载时关心权限——内核只需要"这一块映射成 RX、那一块映射成 RW",不关心里面具体是什么 section。

2.3 看 ELF 的工具 ​

bash
file hello                # 文件类型
readelf -h hello          # ELF Header
readelf -l hello          # Program Headers (segments)
readelf -S hello          # Section Headers
readelf -d hello          # 动态依赖
nm hello                  # 符号表
objdump -d hello          # 反汇编
strings hello             # 字符串
ldd hello                 # 动态库依赖树

三、链接:从 .o 到可执行 ​

3.1 静态链接 ​

把所有依赖的 .o 和静态库 .a 全部打到一个可执行文件:

bash
gcc -static hello.c -o hello
ls -lh hello
# -rwxr-xr-x 1 user user 1.2M hello   ← 1MB+,因为塞进了 libc

优点:

  • 不依赖运行环境的库(可以 scp 到任何机器跑)
  • 启动快(不用动态加载 .so)
  • Go 默认就是静态链接,这就是为什么 Go 二进制 10MB 大但部署简单

缺点:

  • 文件大
  • 库升级要重链
  • N 个程序各塞一份 libc,内存浪费

3.2 动态链接(主流) ​

.so 共享库不打进二进制,运行时才加载:

bash
gcc hello.c -o hello
ls -lh hello
# -rwxr-xr-x 1 user user  16K hello    ← 只有 16KB

ldd hello
# linux-vdso.so.1
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
# /lib64/ld-linux-x86-64.so.2

优点:

  • 文件小
  • 多进程共享同一份 .so 内存
  • 库升级不用重链(只要 ABI 兼容)

缺点:

  • "DLL 地狱"——版本不匹配崩
  • 启动慢一点(动态链接耗时)
  • 部署复杂(目标机要装对应库)

3.3 链接器在做什么 ​

两个 .o 文件:
  main.o:    引用 printf(没实现,符号未定义)
  其他.o:    定义了一些函数

链接器的任务:
  1. 符号解析:把"引用 printf"对应到"libc 里的 printf 实现"
  2. 重定位:计算每个函数 / 全局变量在最终文件里的地址,改写所有引用
  3. 合并 section:多个 .o 的 .text 合并成一个 .text
  4. 生成 ELF

链接错误的根本原因:符号未定义(undefined reference)、符号重定义(multiple definition)。

四、动态链接的细节 ​

4.1 动态链接器:ld-linux.so ​

最神秘的程序——它本身是个动态库,但能直接执行:

bash
# 你以为 ./hello 是先跑 hello
# 实际上内核装载完 hello 后,跳转到 ld-linux.so 入口
# 由它负责加载所有 .so,然后才跳转到 hello 的 main

./hello
# 等价于
/lib64/ld-linux-x86-64.so.2 ./hello

职责:

  • 解析 .dynamic 段,看需要哪些 .so
  • mmap 加载这些 .so 到进程地址空间
  • 解析符号(printf 在 libc.so 的什么地址)
  • 重定位 GOT / PLT 表
  • 跳转到 main

4.2 GOT 和 PLT:动态链接的"间接表" ​

动态链接的难点:编译时不知道 printf 的最终地址(每次运行可能不同,因为 ASLR)。

解决方案:留一个间接表——

GOT(Global Offset Table):全局符号的真实地址表 PLT(Procedure Linkage Table):函数跳转的"跳板"

你的代码:call printf@plt
   ↓
PLT 跳板:跳到 GOT 里 printf 的地址
   ↓ 第一次调用时:
              GOT 里还没有真地址 → 触发动态链接器解析 → 填回 GOT
   ↓ 后续调用:
              GOT 里有真地址 → 直接跳转

懒绑定(lazy binding):第一次用才解析,避免启动时解析所有符号。

4.3 LD_PRELOAD:运行时插桩 ​

环境变量 LD_PRELOAD 让你用自己的 .so 覆盖系统库的符号:

bash
# 我自己写了个 my_malloc.so,定义了 malloc()
LD_PRELOAD=./my_malloc.so ./hello
# hello 调用的 malloc 实际上跑了我的版本

用途:

  • 内存泄漏检测(jemalloc / tcmalloc)
  • 性能分析(代替 syscall 加埋点)
  • 安全审计

五、装载:execve 在干什么 ​

5.1 execve syscall ​

c
execve("/bin/hello", argv, envp);

内核做的事:

1. 读 ELF header,校验
2. 清空当前进程的内存映射(原来的程序没了)
3. 按 Program Header 用 mmap 把 segments 映射进虚拟地址空间
4. 设置 stack:argv / envp / auxv 推入栈
5. 找到 .interp 指定的动态链接器(通常 /lib64/ld-linux-x86-64.so.2)
6. 跳转到动态链接器入口(不是 main!)

5.2 ASLR(地址空间随机化) ​

每次运行,栈、堆、共享库的地址都不同——防止攻击者预测地址做 buffer overflow:

bash
./hello & cat /proc/$!/maps
# 7f8a3c000000-7f8a3c023000 r--p ...  /lib/x86_64-linux-gnu/libc.so.6
# (再跑一次地址会变)

ASLR 让 GDB 调试痛苦 —— 每次断点位置都不一样;但用 GDB 时它会自动关 ASLR(set disable-randomization on,默认就开的)。

5.3 PIE(位置无关可执行文件) ​

ASLR 要工作,可执行文件本身也要能在任意地址运行——这就是 PIE。

现代 GCC 默认开 PIE:

bash
file hello
# ... ELF 64-bit LSB pie executable ...

老程序(non-PIE)装载地址固定为 0x400000——不安全。

六、初始化:main 之前发生了什么 ​

main 不是程序的真正入口——_start 才是。

ld-linux.so 完成动态链接
   ↓ 跳到 _start (在 crt0.o,链接器自动加进来的)
_start:
   设置好 argc, argv, envp
   调用 __libc_start_main
__libc_start_main:
   1. 调用 .preinit_array 里的函数
   2. 调用 .init_array 里的函数(C++ 全局对象构造、glibc 初始化)
   3. 调用 main(argc, argv, envp)
   4. main 返回后,调用 exit()
   5. exit 调用 .fini_array 里的函数(析构)
   6. 调用 _exit syscall 真正退出

6.1 C++ 全局对象的"构造时机" ​

cpp
class Foo { Foo() { printf("foo!\n"); } };
Foo g_foo;       // 在 main 之前构造

int main() { return 0; }

输出:foo!——g_foo 在 main 之前已经构造。

坑:多个翻译单元的全局对象构造顺序未定义——这就是著名的「static initialization order fiasco」。

6.2 atexit / __cxa_atexit ​

注册退出回调:

c
atexit(cleanup);  // main 返回 / exit() 时调 cleanup

注意:_exit 不会触发 atexit;exit 才会。

七、Hello World 实际加载了多少东西 ​

bash
ldd /bin/ls
# linux-vdso.so.1
# libselinux.so.1
# libc.so.6
# libpcre2-8.so.0
# /lib64/ld-linux-x86-64.so.2

cat /proc/$$/maps | wc -l
# 几十行 mmap 区域

最简单的程序也要加载:

  • libc.so.6(C 标准库)
  • ld-linux.so(动态链接器)
  • vDSO(内核提供的快速 syscall)

复杂程序(Java / Python) 一启动加载 200+ 个 .so —— 这就是为什么 Java 进程冷启动慢。

八、共享库的"符号冲突" ​

两个 .so 都定义了 init() 怎么办?

默认:符号优先级按加载顺序 —— 第一个定义的赢
工具:nm 看符号、ldd 看加载顺序
解决:版本控制(.symver)、命名空间隔离

Python 用 dlopen(..., RTLD_LOCAL) 隔离每个扩展模块 —— 才能让多个版本共存。

九、core dump:崩了之后的"快照" ​

程序段错误时,内核把进程的内存 + 寄存器 dump 成一个 ELF 文件:

bash
ulimit -c unlimited       # 允许生成 core
./crashy_program
# Segmentation fault (core dumped)

gdb crashy_program core
(gdb) bt                  # 看崩溃时的调用栈

core dump 也是 ELF 格式 —— 同一套工具能分析。

容器内的 core dump:由宿主 /proc/sys/kernel/core_pattern 决定,常常被 systemd-coredump 收走。

十、容器镜像就是"程序生命周期的打包" ​

Docker 镜像本质上是把程序运行所需的一切打包:

程序本身(ELF)
+ 依赖的 .so 文件
+ 配置文件
+ 数据文件
+ 一个最小的 rootfs

为什么 Alpine 镜像这么小:

  • musl 替代 glibc(小 5 倍)
  • 静态链接(去掉 .so 依赖)
  • 删除文档 / locale 文件

详见 25 篇容器底层。

十一、踩坑提醒 ​

  1. 以为 main 是入口——_start 才是,前面跑了一堆初始化
  2. 以为静态链接没缺点——没法热升级库,二进制变大
  3. 不查 ldd——程序部署到目标机失败,本质是缺 .so
  4. glibc 版本不兼容——CentOS 7 编译的二进制跑 CentOS 6,libc 版本太低 segfault
  5. C++ 全局对象顺序假设——不同 .o 顺序未定义,bug 找不到
  6. 不知道 ASLR——GDB 调试地址变来变去
  7. PLT / GOT 写错——动态链接器找不到符号,启动失败
  8. LD_PRELOAD 滥用——出问题排查极难
  9. core dump 不开——线上崩了没现场
  10. 以为 Hello World 简单——背后几十个 syscall 几十个 .so
  11. strip 掉调试信息发布——出事时栈反汇编看不懂
  12. 以为容器是虚拟机——容器只是 namespace,程序还是 ELF 直接跑
