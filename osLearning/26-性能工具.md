# 性能工具

性能排查最大的敌人不是问题难,是**"凭感觉优化"**:看到服务慢,凭直觉"应该是 DB 慢",加缓存;还慢,"应该是代码烂",重构三天;还慢,最后发现是云盘 await 飙到 200ms——前面一周全是无用功。性能工具的价值不在于命令本身(命令背不完,也不用背完),在于一套**缩小范围的方法**:先用全局指标把问题钉到 CPU、内存、磁盘、网络、调度、syscall 中的某一类,再用专项工具拿到证据,**有证据了才动代码**。这一篇给你这套方法,顺带把每个症状对应的工具配齐。

> 一句话先记住:**先分层定方向,再下钻拿证据,最后才改代码**。下钻前先问六个问题:CPU 忙不忙、运行队列长不长、内存有没有在回收、磁盘 await 高不高、网络有没有重传丢包、syscall 卡在哪里。**没有证据的优化都是赌博**。

---

## 一、排查的总路线图

把这张图刻在脑子里,所有性能问题都按它走:

```
症状(p99 高 / CPU 高 / 内存涨 / 吞吐掉)
   │
   ▼ 第一分钟:全局体检 ──────── uptime / top / vmstat / free / dmesg
   │   (问题在哪一类资源?)
   ▼ 定方向
   ├─ CPU      → top -H → perf top → 火焰图
   ├─ 内存     → free → smaps_rollup → pmap → heap profile
   ├─ 磁盘 IO  → iostat -x → pidstat -d → biolatency
   ├─ 网络     → ss → sar -n → tcpdump
   ├─ 调度     → pidstat -w → perf sched → cgroup cpu.stat
   └─ syscall  → strace -c → bpftrace
   │
   ▼ 拿到证据(火焰图的宽条 / 直方图的长尾 / 涨的计数器)
   ▼ 这时候,才改代码或参数
```

**最要命的反模式是跳过前两步直接改代码**——你优化的可能根本不是瓶颈。瓶颈在磁盘时,把 CPU 算法优化 10 倍,用户端延迟一毫秒都不会少。

## 二、第一分钟:60 秒全局体检

登上一台"出问题"的机器,前 60 秒固定跑这一套:

```bash
uptime              # load average:1/5/15 分钟,看趋势(在恶化还是恢复)
dmesg -T | tail     # 内核有没有喊救命:OOM、磁盘错误、网卡报错
vmstat 1            # 每秒一行的系统总览(下面细讲)
free -h             # 内存大盘
top                 # 谁在吃 CPU / 内存
df -h               # 磁盘有没有写满(写满的症状千奇百怪)
```

`vmstat 1` 是 60 秒里信息密度最高的一行,每列都在回答一个问题:

```
procs ─────memory──── ──swap── ───io──── ─system── ─────cpu─────
 r  b   free   cache   si  so   bi  bo    in  cs   us sy id wa st
 ↑  ↑                  ↑   ↑                  ↑    ↑  ↑     ↑  ↑
 │  │                  │   └ swap 换出>0 = 内存真不够了,性能悬崖
 │  │                  └ 换入
 │  └ 不可中断睡眠(D 状态)的进程数,多 = 卡在磁盘 IO
 └ 等 CPU 的进程数,持续 > 核数 = CPU 排队了
                                              cs 上下文切换暴涨 = 锁竞争/线程过多
                                       us 用户态  sy 内核态(高 = syscall/内核忙)
                                                  wa IO 等待  st 被虚拟化邻居偷走
```

一个常年被误读的指标要单独说:**load average 不是 CPU 利用率**。Linux 的 load 统计的是"R 状态(等 CPU)+ D 状态(等 IO)的进程数"——**load 高得吓人但 CPU 很闲,大概率是一堆进程卡在磁盘上**。看到高 load,先用 `vmstat` 的 r / b 两列拆开:r 高怪 CPU,b 高怪 IO。

## 三、CPU:从 top 到火焰图

### 3.1 top:谁在烧,烧在哪个态

```bash
top -H -p <pid>     # -H 展开到线程级,定位具体哪个线程在烧 CPU
```

CPU 的四个去向,每个高了含义不同:

| 指标 | 高了说明什么 | 下一步 |
| --- | --- | --- |
| `%us` | 用户态代码在烧(业务计算 / GC / 序列化) | perf 看热点函数 |
| `%sy` | 内核态在烧(syscall 频繁 / 网络栈 / 锁) | strace -c / perf |
| `%wa` | CPU 闲着等磁盘 | iostat -x |
| `%st` | **虚拟机邻居把物理 CPU 偷走了** | 找云厂商,代码无罪 |

`%st`(steal)值得单独点名:云主机上它高,意味着同一台物理机上的其它虚拟机在抢 CPU——**你的代码什么都没做错,但就是慢**。这是"代码层面无解"的少数情况之一,证据拿给云厂商。

### 3.2 perf:CPU 显微镜,先懂采样原理

perf 怎么知道 CPU 在干什么?**采样(sampling)**,原理一张图:

```
 ┌─ 每秒 99 次定时打断(-F 99)──────────────────────────┐
 │                                                       │
 ▼            ▼            ▼            ▼                ▼
─┬────────────┬────────────┬────────────┬────────────────┬──→ 时间
 │打断瞬间:   │            │            │
 │CPU 在跑哪个函数?调用栈是什么?记下来   │
 └──────────────────────────────────────┘
 30 秒 ≈ 3000 个样本 → 统计:某函数出现在 40% 的样本里
                       → 它就占了约 40% 的 CPU 时间
```

像随机抽查 3000 次"CPU 此刻在干嘛"——抽中次数多的就是热点。**开销极小**(每秒才打断 99 次,选 99 而不是 100 是为了避开和程序自身周期性活动共振),所以 perf 敢直接上生产。

```bash
perf top -p <pid>      # 实时排行榜:CPU 正在烧在哪些函数上
```

`perf top` 里看到的符号,本身就是诊断结论:

| 看到的符号 | 在说什么 |
| --- | --- |
| `copy_user_enhanced_fast_string` | 内核在用户态/内核态之间搬数据(考虑零拷贝,23 篇) |
| `futex_wait` / `futex_wake` | **锁竞争**(14 篇) |
| `tcp_sendmsg` / `tcp_recvmsg` | 网络收发占大头 |
| `ext4_*` / `xfs_*` | 文件系统路径在烧 |
| 你的业务函数名 | 老老实实优化算法去 |

### 3.3 火焰图:三分钟定位热点的名场面

`perf top` 只有平面排行,看不出调用关系。火焰图(flame graph)把整棵调用栈可视化:

```bash
perf record -F 99 -g -p <pid> -- sleep 30    # 采样 30 秒,-g 抓调用栈
perf script > out.perf
# 用 FlameGraph 工具:stackcollapse-perf.pl out.perf | flamegraph.pl > out.svg
```

读图只有一条规则:**看宽度,不看高度**。宽度 = 该函数(含其子调用)占的 CPU 时间比例;高度只是调用栈深度,**栈深不等于慢**。从底往上找"最宽的平顶"——宽且没有子调用的那一块,就是 CPU 真正烧掉的地方。

名场面:某服务 CPU 莫名 80%,代码 review 两天没结论。火焰图一拉——`json.Marshal` 一族占了整图 40% 宽度,根因是某个内部接口每次请求把一个大对象序列化了三遍。**从"完全没头绪"到"定位到行",三分钟**。这就是为什么火焰图是性能排查的第一主力武器。

## 四、内存:RSS、page cache 和"假泄漏"

内存排查第一课:**先分清楚几个"内存"**,它们经常被混为一谈:

| 指标 | 是什么 | 常见误读 |
| --- | --- | --- |
| VSZ / VIRT | 虚拟地址空间总量 | **不等于占用**!mmap 了没碰的都算它(06 篇) |
| RSS | 真正驻留物理内存的部分 | 这才是"进程用了多少" |
| page cache | 内核用空闲内存缓存的文件数据 | **不是泄漏,是福利**,要用时秒级回收 |
| Dirty / Writeback | 改过还没刷盘的页 | 持续很高 = 写入压力大 |
| slab | 内核自己的对象缓存(dentry/inode 等,09 篇) | 偶尔也能涨成"谁吃了我的内存"悬案 |

最经典的乌龙:`free` 显示"内存快用完了"——**Linux 的哲学就是空闲内存可耻,能拿来做 cache 绝不闲着**。看 `available` 列(真正可用的估计值),别看 `free` 列。

排查动线:

```bash
free -h                              # 大盘:available 才是真可用
cat /proc/<pid>/smaps_rollup         # 进程 RSS 的权威拆解(堆/栈/共享库/匿名映射)
pmap -x <pid>                        # 逐段看哪块映射在涨
slabtop                              # 内核 slab 排行(dentry 暴涨是经典案例)
vmstat 1                             # si/so 列:swap 一旦活跃,性能已经在悬崖边
dmesg -T | grep -i oom               # 有没有 OOM killer 出手过的尸检报告
```

RSS 确认在涨,且 `smaps_rollup` 显示是堆(heap)在涨,接下来就该交给语言层工具了——Java 的 heap dump、Go 的 pprof。**OS 层工具的职责是把范围缩小到"确实是堆泄漏",而不是替你找出哪个对象泄漏**。容器里另有一坑:cgroup 的 `memory.current` 包含 page cache,25 篇讲过,别看见贴近 limit 就慌。

## 五、磁盘 IO:await 是金指标

```bash
iostat -x 1          # 设备级:每块盘的负载全景
pidstat -d 1         # 进程级:谁在读写
iotop                # 交互式版本的"IO top"
```

`iostat -x` 关键列:`r/s` `w/s`(IOPS)、`rkB/s` `wkB/s`(吞吐)、**`await`(一次 IO 从排队到完成的平均毫秒数,延迟敏感服务的命门)**、`aqu-sz`(队列深度)、`%util`(设备忙碌比)。

判断矩阵,两个指标交叉就能定性:

| | `%util` 高 | `%util` 不高 |
| --- | --- | --- |
| **`await` 高** | 盘真忙:IOPS/带宽到顶了 | **可疑!** 云盘限流 / 阵列故障 / 调度问题 |
| **`await` 低** | 盘忙但健康(顺序大 IO 常见) | 一切正常 |

右上角那格是云时代的高发区:**盘看着不忙,延迟却高**——大概率是云盘的 IOPS 配额被打满(云厂商在你看不见的层面限流),证据就是这个组合。另外记住:**大量小随机 IO 死在 IOPS 上,大顺序 IO 死在带宽上**,优化方向完全不同(攒批 vs 换更快的盘)。

## 六、网络:ss 一家全包

```bash
ss -s                      # 连接状态总览:多少 ESTAB、多少 TIME_WAIT
ss -antp                   # 全部 TCP 连接 + 归属进程
ss -lntp                   # 谁在监听哪个端口
ss -ti dst <ip>            # 单条连接的 TCP 内幕:rtt / cwnd / retrans
sar -n DEV,TCP,ETCP 1      # 网卡吞吐 + TCP 重传率(retrans/s)
ip -s link                 # 网卡统计:drops / errors
```

什么症状看什么:

- **延迟抖动、吞吐上不去** → `sar -n ETCP` 看重传(retrans):重传 = 丢包,TCP 要等超时或快速重传,延迟直接上台阶
- **连接建不上、偶发超时** → listen 队列溢出,`ss -lnt` 看 `Recv-Q` 是否顶到 `Send-Q`(全连接队列满了内核默默丢握手包)
- **端口耗尽、连不出去** → `ss -s` 看 TIME_WAIT 数量,几万个 = 短连接风暴,该上连接池了
- **丢包** → `ip -s link` 的 drops:RX drop 常见于流量突发打爆 ring buffer

## 七、strace:syscall 级显微镜(小心用)

程序"卡住了"但 CPU 是 0%?它多半阻塞在某个 syscall 里。strace 能看到进程和内核的每一次对话(syscall 机制详见 02 篇):

```bash
strace -c -p <pid>          # 先看统计:哪个 syscall 被调最多、累计耗时最长
strace -tt -T -p <pid>      # 再看实况:每条 syscall 带时间戳和耗时
```

实况输出会直接告诉你卡在哪:

```
futex(0x7f..., FUTEX_WAIT, ...) = 0 <0.500123>
       ↑ 在等锁/条件变量,等了 500ms ↑
```

`futex` 等半天是锁竞争,`read` 等半天看 fd 是磁盘还是 socket(`ls -l /proc/<pid>/fd/<n>`),`connect` 卡住是下游不通——**每种"卡"都有 syscall 指纹**。

但 strace 有一个必须知道的代价:**它基于 ptrace,每个 syscall 要让目标进程停两次、和 strace 进程来回切换**。对一个每秒几万 syscall 的高 QPS 服务,挂上 strace 可能让它慢几倍甚至几十倍——**线上等于自带一场新故障**。守则:先 `-c` 看统计、限定 `-e trace=` 只抓关心的、挂的时间以秒计。要长时间低开销地看 syscall,正确工具是下一节的 bpftrace。

## 八、调度与容器:看不见的等待

有一类延迟在所有上面的工具里都"看不到":**进程明明就绪了,却在排队等 CPU**。

```bash
pidstat -w 1                 # 每秒上下文切换:cswch(主动)/ nvcswch(被动)
perf sched record -- sleep 10
perf sched latency           # 任务从"就绪"到"真正上 CPU"等了多久
```

`pidstat -w` 两列的含义完全不同:**voluntary(主动切换)高 = 进程老在等 IO / 锁,自己睡过去的;nonvoluntary(被动切换)高 = 时间片被抢走,CPU 上人太多了**。前者查 IO 和锁,后者减线程或加核。

容器里多一个隐藏选项,25 篇的名场面:

```bash
cat /sys/fs/cgroup/cpu.stat
# nr_throttled / throttled_usec 在涨 = CPU quota 限流,p99 抖动元凶
```

**宿主机指标全绿、p99 照样飞天**,十有八九是它——这是容器时代排查清单上必须新增的一行。

## 九、bpftrace:没有现成工具时的万能钩子

前面的工具各管一段,总有问题落在缝隙里:"哪个进程在反复打开这个配置文件?""read 延迟的分布长什么样,是整体慢还是长尾慢?"——这时用 bpftrace 现场写一行探针,直接在内核里做统计:

```bash
# 全机 syscall 排行:谁在疯狂调什么
bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[probe] = count(); }'

# read 延迟直方图:在内核里聚合,只把直方图带回用户态,开销极小
bpftrace -e '
tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }
tracepoint:syscalls:sys_exit_read /@start[tid]/ {
  @us = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'
```

它和 strace 的本质区别:**strace 把每个事件搬到用户态处理,bpftrace 把统计逻辑送进内核、事件就地消化**——同样看 syscall,开销差几个数量级,这才是能挂在生产环境上的形态。背后的 eBPF 机制和更多武器(runqlat / biolatency / tcpretrans),下一篇整篇细讲。

## 十、三个排查剧本

把方法串成肌肉记忆,三个最高频的症状各给一条动线:

**剧本一:p99 高,p50 正常**(平均值挑不出短抖动,这正是最难的一类)
1. `top` 看 us/sy/wa/st 定大方向 → 2. `vmstat 1` 盯 r / b / si / so 抓瞬时尖峰 → 3. `cat /sys/fs/cgroup/cpu.stat` 排除 throttle(容器内必查)→ 4. `pidstat -w -d -u 1` 看进程维度哪项在抖 → 5. `strace -c` 短挂 / bpftrace 看 syscall 延迟分布 → 6. 都不是?查下游:DB 慢查询、RPC 超时、`ss -ti` 看重传。

**剧本二:CPU 持续高**
1. `top -H` 定位线程(GC 线程?worker?某个莫名线程?)→ 2. `perf top` 看符号:futex 是锁、copy_user 是拷贝、业务函数是计算 → 3. 拉 30 秒火焰图,找最宽的平顶 → 4. 对症:锁竞争减临界区、拷贝上零拷贝、计算优化算法。

**剧本三:内存持续涨**
1. 先定性:RSS 涨还是 page cache 涨?(cache 涨不是病)→ 2. `smaps_rollup` 拆解:堆?线程栈?匿名 mmap?→ 3. 堆涨 → 交给语言层 heap profile → 4. 全程盯着 `dmesg` 的 OOM 记录,那是最后通牒。

## 踩坑提醒

1. **没证据就改代码**——优化了一周,瓶颈根本不在那。先体检、再下钻、拿到火焰图或计数器证据才动手。
2. **只看平均值**——p99 问题是短抖动,平均值把它抹平了。要看分布(直方图)和秒级采样(`vmstat 1`)。
3. **把 page cache 当内存泄漏**——Linux 拿空闲内存做文件缓存是设计,看 `available` 不看 `free`,该回收时它跑得比谁都快。
4. **load 高就怪 CPU**——load 包含 D 状态(等 IO)的进程,先用 vmstat 的 r / b 拆开再下结论。
5. **strace 往高 QPS 服务上一挂半小时**——ptrace 的开销能把服务拖慢几十倍,排查工具自己变成故障。统计模式、短时间、或改用 bpftrace。
6. **容器里不查 cpu.stat**——宿主机指标全部正常、p99 照样飞,throttle 在所有传统工具的盲区里。
7. **没有基线**——不知道正常时 cs 多少、await 多少,出事时就分不清"高"和"它一直这样"。平时存一份健康快照。
8. **VIRT 高就喊泄漏**——虚拟地址空间不要钱,mmap 了没碰的不占物理内存,看 RSS。
