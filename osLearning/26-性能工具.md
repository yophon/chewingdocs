# 性能工具

性能排查最怕"凭感觉优化"。OS 层工具的价值是把问题定位到 CPU、内存、IO、网络、锁、调度、系统调用中的某一类。`top` 看现象,`vmstat` 看全局压力,`iostat` 看磁盘,`strace` 看 syscall,`perf` 看 CPU 火焰图,`bpftrace` 看内核事件。这一篇给一套工程化排查路径。

> 一句话先记住:**性能排查先分层,再下钻**。先问 CPU 忙不忙、run queue 长不长、内存有没有 reclaim、磁盘有没有 await、网络有没有丢包、syscall 卡在哪里。**不要一上来改代码**——先用工具证明瓶颈在哪。

---

## 一、第一分钟:全局体检

```bash
uptime
top
vmstat 1
free -h
df -h
dmesg -T | tail
```

看什么:

- load average 是否远大于 CPU 核数
- CPU us/sys/wa/steal
- runnable 队列 `r`
- blocked 队列 `b`
- swap in/out
- OOM / disk error / kernel warning

---

## 二、CPU

### 2.1 top / htop

```bash
top -H -p <pid>
```

看线程级 CPU。

字段:

- `%us`:用户态
- `%sy`:内核态
- `%wa`:IO wait
- `%st`:虚拟化 steal

### 2.2 perf top

```bash
perf top -p <pid>
```

看 CPU 正在烧在哪些函数。

如果看到:

- `copy_user...` → 数据拷贝
- `futex_wait` / `futex_wake` → 锁竞争
- `tcp_sendmsg` → 网络发送
- `ext4_*` → 文件系统
- 业务函数 → 应用计算

### 2.3 火焰图

```bash
perf record -F 99 -g -p <pid> -- sleep 30
perf script > out.perf
```

再用 FlameGraph 生成 svg。

火焰图看宽度,不是高度。越宽表示 CPU 时间占比越高。

---

## 三、内存

```bash
free -h
cat /proc/meminfo
vmstat 1
slabtop
pmap -x <pid>
cat /proc/<pid>/smaps_rollup
```

关键指标:

- RSS:实际驻留内存
- VSZ/VIRT:虚拟地址空间,不等于实际占用
- page cache:文件缓存
- Dirty/Writeback:待刷盘页
- swap si/so:换入换出
- slab:内核缓存(dentry/inode 等)

OOM 排查:

```bash
dmesg -T | grep -i oom
```

---

## 四、磁盘 IO

```bash
iostat -x 1
pidstat -d 1
iotop
```

`iostat -x` 重点:

- `r/s`,`w/s`:IOPS
- `rkB/s`,`wkB/s`:吞吐
- `await`:平均等待
- `%util`:设备忙碌程度
- `aqu-sz`:队列深度

判断:

- await 高 + util 高 → 设备忙
- await 高 + util 不高 → 可能云盘/阵列/调度异常
- 大量小 IO → IOPS 瓶颈
- 大顺序 IO → 带宽瓶颈

---

## 五、网络

```bash
ss -s
ss -antp
sar -n DEV,TCP,ETCP 1
ip -s link
```

看:

- 连接状态分布
- retrans 重传
- listen 队列
- 网卡 drops/errors
- TIME_WAIT 数量

端口监听:

```bash
ss -lntp
```

TCP 细节:

```bash
ss -ti dst <ip>
```

---

## 六、strace:看 syscall

```bash
strace -tt -T -p <pid>
strace -c -p <pid>
```

用途:

- 程序卡在哪个 syscall
- syscall 耗时
- 文件路径/网络 fd
- errno

例子:

```
futex(... ) = 0 <0.500123>
```

说明线程可能在等锁/条件变量。

注意:strace 有开销,高 QPS 进程谨慎长时间挂。

---

## 七、调度

```bash
pidstat -w 1
perf sched record -- sleep 10
perf sched latency
```

看:

- voluntary context switches:主动睡眠,如等 IO/锁
- nonvoluntary:被抢占
- run queue 延迟

容器里还要看:

```bash
cat /sys/fs/cgroup/cpu.stat
```

`nr_throttled` 和 `throttled_usec` 高,说明 CPU quota 在限流。

---

## 八、bpftrace 快速探针

列工具:

```bash
bpftrace -l 'tracepoint:syscalls:sys_enter_*' | head
```

统计 syscall:

```bash
bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[probe] = count(); }'
```

统计 read 延迟:

```bash
bpftrace -e '
tracepoint:syscalls:sys_enter_read { @start[tid] = nsecs; }
tracepoint:syscalls:sys_exit_read /@start[tid]/ {
  @us = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'
```

bpftrace 的优势是低侵入地看内核事件。

## 九、排查路径模板

### 9.1 p99 高

1. `top` 看 CPU/wa/st
2. `vmstat 1` 看 r/b/si/so
3. `pidstat -w -d -u 1` 看进程维度
4. `strace -c` 看 syscall 时间
5. `perf record` 看 CPU 热点
6. 查下游:DB/RPC/网络

### 9.2 CPU 高

1. `top -H`
2. `perf top`
3. 火焰图
4. 看锁: futex/自旋
5. 看 GC/runtime 线程

### 9.3 内存涨

1. RSS 还是 page cache
2. `smaps_rollup`
3. `pmap`
4. 语言 runtime heap profile
5. dmesg OOM

---

## 十、常见错误

1. **只看 CPU 平均值**:p99 问题常是短抖动。
2. **把 page cache 当内存泄漏**:Linux 会用空闲内存做缓存。
3. **看到 load 高就怪 CPU**:D 状态 IO 等待也会推高 load。
4. **strace 线上挂太久**:可能放大延迟。
5. **没有基线**:不知道正常时指标,异常时很难判断。

> **结论**:性能工具不是命令合集,而是一套缩小范围的方法。先用全局指标定方向,再用专项工具拿证据,最后才改代码或参数。
