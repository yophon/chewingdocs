# 操作系统系列写作计划

> 来源:`未来系列规划.md` 第一类「补完整知识图」第四项,⭐⭐⭐⭐。
> 系统设计教了「**怎么用工具**」(Redis / MQ / 分库),OS 系列教了「**这些工具底下到底在干嘛**」——为什么 epoll 比 select 快、为什么 GC 会 Stop The World、为什么 Docker 看着像虚拟机其实是 namespace、为什么 false sharing 能让多线程程序变慢 10 倍。
> **看完这层,你看任何"性能问题"都能直接看到底**——Java GC、Node 事件循环、Flutter Isolate、Go 协程、Rust async,全是这层往上长出来的。

---

## 设计思路

五层结构,从「机器视角」到「工程实战」:

```
心智 + 基础(用户态 / 内核态 / 系统调用 / CPU)
   → 内存(虚拟内存 / 页表 / 分配 / 缓存)
   → 进程线程与并发(进程 / 线程 / IPC / 锁 / 调度 / 协程)
   → IO 与文件(VFS / IO 模型 / epoll / io_uring / 零拷贝)
   → 工程实战(信号 / 容器 / 性能工具 / eBPF)
```

**这个系列不是 CSAPP 重写**——CSAPP 太厚,重点偏教学;**本系列偏"工程师视角"**:每个概念都对应一个真实的"性能问题或者 bug",学完能直接用在排错和调优。

不深入到内核源码,**而是讲清楚"用户态程序员能看到的内核行为"**——你写 Java / Go / Python,但调优时必须懂这些。**95% 的性能瓶颈最终都落到 OS 这一层**。

## 写作模板

每篇遵守"为什么 → 是什么 → 怎么用 → 何时不该用"的四段式:

```
1. 为什么要有这个东西
   - 没它的世界长什么样
   - 加了它解决了什么问题

2. 它是怎么实现的
   - 内核数据结构
   - 关键流程

3. 用户态怎么用
   - C / Go / Java 视角的接口
   - 真实场景例子

4. 何时不该用 / 坑在哪
   - 性能边界
   - 常见 bug
   - 替代方案
```

不堆太多源码——**只贴关键的内核数据结构和系统调用签名**,让读者「知道这一层在干嘛」就够了。

## 节奏

```
01-05  心智:看完知道"OS 在程序员眼里到底是什么"
06-10  内存:看完不会再被"为什么我的程序占了这么多内存"问倒
11-17  并发:**最值钱的一层**,Java / Go / Rust 并发模型全靠它
18-23  IO:**第二值钱**,所有"性能调优"最后都到这里
24-28  工程:用得到、面试问得到的硬通货
```

**一个硬指标**:看完 11-17 + 18-23 这 13 篇,你应该能在白板前讲清楚:

- "为什么 Go 协程这么轻量,但密集 CPU 任务用线程更快"
- "为什么我的服务 p99 抖动来自 GC,但不是所有 GC 都会 Stop The World"
- "为什么把 epoll 换成 io_uring 能在某些场景快 30%"

能讲清楚一题这个系列就值了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「Linux 内核 5.x 源码逐行分析」——不是教学,是工程师视角
2. 「先讲个故事 / 个人情怀」——直接进重点
3. 罗列 Linux 命令——只在工程实战(24-28)讲必备的
4. 「这是面试常考」——读者来这里不是为了刷题,是为了真的看到底
5. 抄 CSAPP / Linux Kernel Development——那两本书在那儿,这里只补它们没讲透的工程细节
6. 不画图就讲 cache line / 虚拟内存——这些东西**不画图永远讲不清楚**
