# 02-项目骨架与构建工具链

> 一句话导读:Android 构建栈是一座由 Gradle、AGP、Kotlin、Compose Compiler、KSP 五层叠起的高塔,任何两层版本错配都会触发"能编译过但运行时崩溃"。Version Catalog + Compose Compiler 自 Kotlin 2.0.20+ 内嵌,是 2024-2026 这一代把这座塔锁稳的工程答案。

第 01 篇把"现代 Android"的坐标系立起来了 —— 单 Activity、Compose、Kotlin、Jetpack、Play Console。但坐标系背后是一套相当复杂的构建系统:打开 Android Studio Koala+ 新建项目,Gradle Sync 跑完会显示十几条 dependencies、跑过几条 Kotlin / KSP / Hilt / Compose Compiler 插件、生成出 `app-debug.apk`。这条流水线如果你只跟 Android Studio 自己生成的模板走,大概率能用很久不出问题;一旦想升 Kotlin 版本、加 Hilt、引入 Room,就会发现版本号之间有一张暗网,任意一根绳没拽对,就是漫长的 "Gradle Sync failed" 排查。

这一篇把这张暗网摊开。在动 NotedX 任何一行业务代码之前,先把 `build.gradle.kts`、`libs.versions.toml`、`settings.gradle.kts`、`gradle.properties` 四份文件按 2024-2026 的工程基线锁死,后续 28 篇加新依赖时只是往这套骨架追加版本号与坐标,不再重新设计。

## 1. 机制定位

新读者第一次撞上 Android 构建系统,通常的表现是"为什么改一行依赖版本号要等三分钟,且改完之后某个 sealed class 突然报错"。这不是 Gradle 一个工具的问题,而是 Android 构建链有五层都在协同工作:

| 层 | 例子 | 角色 |
| --- | --- | --- |
| Gradle | 8.10 | 通用 JVM 构建系统;定义 task / configuration / plugin 模型 |
| Android Gradle Plugin (AGP) | 8.7.x | Google 提供的 Gradle 插件,把"build APK"翻译成具体 task |
| Kotlin Gradle Plugin | 2.0.21 | JetBrains 提供;把 `.kt` 编译成字节码,集成 K2 编译器 |
| Compose Compiler Plugin | 与 Kotlin 同版本(2.0.20 起内嵌) | Compose 专用编译器插件,处理 `@Composable` 函数变换 |
| KSP / Hilt / Room 插件 | 2.0.21-1.0.27 / 2.51.1 / 2.6.1 | 注解处理与代码生成 |

这五层不是简单的"高层依赖低层",而是**互相约束**:AGP 8.7.x 要求 Gradle 8.7+,但不能用 Gradle 9 还在 alpha 的某些 API;Kotlin 2.0.21 要求 AGP ≥ 7.4;Compose Compiler 2.0.20 强制依赖 Kotlin 2.0.20+;KSP 版本号是 `Kotlin版本-KSP版本`(`2.0.21-1.0.27`),不能配 Kotlin 2.0.20。任何一处版本错位,都可能让构建在 Sync 阶段就失败,或者 Sync 通过、编译通过、运行时直接抛 `NoClassDefFoundError`。

这种复杂度的根源在于 Android 平台多年累积的兼容矩阵 —— Google 不能要求所有 OEM 设备同步更新,所以 SDK 与 ART 行为差异长期存在,构建系统必须能为不同 minSdk / targetSdk 生成正确的 desugaring、混淆、资源压缩。这是 Android 与 iOS 构建复杂度的本质差异:iOS 一个 Xcode 锁定一组工具链版本,选定就用;Android 你要主动管理 5 层组件的版本对齐。

新读者最常见的两种错误姿势:一种是"网上抄一段 Stack Overflow 的 `build.gradle`",这段配置可能基于两年前的 Kotlin 1.7 + Compose Compiler 1.3,直接粘进 2024 项目里会冲突;另一种是"Android Studio 提示升级,无脑点 Accept",IDE 会把 AGP 升到最新,但 Kotlin 与 Compose 不会同步升,然后下一次打开就开始报 Compose Compiler 不兼容。

正确姿势是把"版本号"作为一类一等公民的工程对象,集中管理,显式对齐。这正是 Version Catalog (`libs.versions.toml`) 的用武之地 —— 它在 Gradle 7.0 进入实验、7.4 稳定、8.0+ 默认推荐,2024-2026 是 Google 官方模板的默认形态。

## 2. Android 心智

要把构建栈管好,需要先理解四件事:Gradle 的三阶段、AGP 与 Kotlin 的对齐表、Version Catalog 的工作方式、KSP 与 Compose Compiler 在 Kotlin 2.0+ 的位置。

**Gradle 三阶段**。一次 `./gradlew assembleDebug` 在底层会经历 Initialization → Configuration → Execution 三个阶段。Initialization 解析 `settings.gradle.kts`,确定有哪些子模块;Configuration 执行所有 `build.gradle.kts` 顶层代码,构建出整个 task 依赖图;Execution 才真正按图执行 task。Configuration 阶段是最慢的环节(2024 之前一个中等项目要 10-30 秒),所以 Gradle 引入了 Configuration Cache,把 Configuration 阶段结果序列化下来,下次构建直接跳过。后面 `gradle.properties` 章节会显式打开这个开关。

理解这三阶段对工程意义重大:**不要在 `build.gradle.kts` 顶层写依赖外部状态的代码**(读 git 信息、读环境变量),否则会破坏 Configuration Cache 命中。所有这类逻辑要写在 `providers.environmentVariable("FOO")` 或 Task action(`doLast { ... }`)里,让 Gradle 知道这是 lazy 求值。

**AGP × Kotlin × Compose Compiler 对齐表**。这是工程上最痛的一处,给出 2024-2026 的实际可用组合:

| AGP | 最低 Gradle | 推荐 Kotlin | Compose BOM | Compose Compiler | KSP |
| --- | --- | --- | --- | --- | --- |
| 8.5.x | 8.7 | 1.9.24 | 2024.06.00 | 1.5.14 | 1.9.24-1.0.20 |
| 8.6.x | 8.7 | 2.0.10 | 2024.09.01 | 内嵌(2.0.10) | 2.0.10-1.0.24 |
| 8.7.x | 8.9 | 2.0.21 | 2024.10.00 | 内嵌(2.0.21) | 2.0.21-1.0.27 |
| 8.8.x | 8.10.2 | 2.0.21 / 2.1.0 | 2024.12.01 | 内嵌 | 与 Kotlin 对齐 |

NotedX 基线选 **AGP 8.7.3 + Gradle 8.9 + Kotlin 2.0.21 + Compose BOM 2024.10.00 + KSP 2.0.21-1.0.27**。理由:AGP 8.7 是 2024-10 进入 stable 的版本,与 Android 15 行为变更对齐最准;Kotlin 2.0.21 是 K2 默认编译器与内嵌 Compose Compiler 的稳定主线。

**Compose Compiler 内嵌**。Compose 1.x 时代,Compose Compiler 是一个独立 Maven 坐标(`androidx.compose.compiler:compiler:1.5.14`),版本号与 Kotlin 严格绑定 —— 每出一个新 Kotlin 版本要等 Google 发对应的 Compose Compiler,通常滞后 2-4 周,这个等待期是过去无数项目升级 Kotlin 卡住的根源。Kotlin 2.0.20 起,Compose Compiler 被并入 Kotlin 仓库,跟 Kotlin 同版本号发布,工程上要做的只剩两件事:apply `org.jetbrains.kotlin.plugin.compose` 插件,以及在 `composeOptions` 里(其实 AGP 8.7+ 已经不再需要这块)曾经手动指定的 `kotlinCompilerExtensionVersion` 不再出现。这是 2024 年 Compose 工程链最大的简化。

**KSP vs KAPT**。Kotlin 注解处理曾经有两种方案:KAPT 通过生成 Java stub 然后跑标准 Java APT,代价是慢(双倍编译);KSP(Kotlin Symbol Processing)直接读 Kotlin AST 跑处理器,2-3 倍速度提升。2024 起所有主流 Kotlin 注解处理器(Hilt / Room / Moshi / Glide)都已经支持 KSP,KAPT 进入维护状态。本系列**所有处理器一律走 KSP**,Hilt 用 `dagger.hilt.android.plugin` + `com.google.devtools.ksp` 插件,Room 用 `androidx.room.RoomDatabase` 的 KSP 注解处理。第 13 / 14 篇展开。

**Version Catalog**。`gradle/libs.versions.toml` 是 Gradle 自带的版本目录格式,有四个 section:`[versions]` 定义版本变量,`[libraries]` 定义 Maven 坐标,`[plugins]` 定义 Gradle 插件,`[bundles]` 定义"一组常一起出现的库别名"。`build.gradle.kts` 里通过 `libs.xxx` 访问,IDE 有完整补全。它的核心价值是:**所有版本号集中一处,跨模块共享**,后续多模块化时不会出现"`:feature-note` 用 Compose 1.7.4,`:feature-todo` 用 1.7.5"这种灾难。

## 3. 工程实现

把上面四件事落到具体文件。这一节给的 4 份文件是 NotedX 在前 15 篇的构建基线,后续每加一个新依赖只是往 `libs.versions.toml` 加一行,不再重写整份文件。

**`settings.gradle.kts`**(项目根) —— 仓库与子模块声明:

```kotlin
pluginManagement {
    repositories {
        google {
            content {
                includeGroupByRegex("com\\.android.*")
                includeGroupByRegex("com\\.google.*")
                includeGroupByRegex("androidx.*")
            }
        }
        mavenCentral()
        gradlePluginPortal()
    }
}

dependencyResolutionManagement {
    repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS)
    repositories {
        google()
        mavenCentral()
    }
}

rootProject.name = "NotedX"
include(":app")
```

两个关键点:`RepositoriesMode.FAIL_ON_PROJECT_REPOS` 强制所有依赖只能从 `settings.gradle.kts` 里声明的仓库拉,任何子模块如果偷偷在自己的 `build.gradle.kts` 里加 `repositories { ... }` 会构建失败 —— 这是防止"某个第三方库引入私有仓库导致供应链污染"的工程保护。`includeGroupByRegex` 把 google 仓库限制为只能解析 androidx / google / android 三组,加速 Sync。

国内开发者会想要镜像加速,可以追加阿里云 / 腾讯云 / 华为云 Maven 镜像,但**只在本地 `~/.gradle/init.d/` 里写**,而不是写进项目仓库的 `settings.gradle.kts`,避免污染团队配置。一份典型的 `~/.gradle/init.d/aliyun.gradle.kts`:

```kotlin
allprojects {
    repositories {
        maven { url = uri("https://maven.aliyun.com/repository/google") }
        maven { url = uri("https://maven.aliyun.com/repository/public") }
    }
}
```

注意几个工程决策:**Compose BOM**(`compose-bom`)通过 `platform(libs.compose.bom)` 引入后,Compose 子家族的所有 artifact 都不写版本号,统一由 BOM 决定;**`bundles.compose`** 把"必引"组打包,业务模块只需 `implementation(libs.bundles.compose)`,大幅减少 build.gradle.kts 的重复;**`kotlin-compose` 插件**是 Kotlin 2.0.20+ 取代旧 `composeOptions { kotlinCompilerExtensionVersion = ... }` 的标准答案,apply 后无需手动指定 Compose Compiler 版本。

**根 `build.gradle.kts`**(项目根) —— 顶层只 apply 插件,不写依赖:

```kotlin
plugins {
    alias(libs.plugins.android.application) apply false
    alias(libs.plugins.kotlin.android) apply false
    alias(libs.plugins.kotlin.compose) apply false
    alias(libs.plugins.kotlin.serialization) apply false
    alias(libs.plugins.ksp) apply false
    alias(libs.plugins.hilt.android) apply false
}
```

`apply false` 表示在根模块只是把插件加进 classpath,实际 apply 由子模块决定。这是 Gradle 多模块的标准姿势,即便目前只有 `:app` 一个模块,也按这种姿势写,第 21 篇拆模块时无需重构。

## 4. 调参与验收

构建系统的"调参"主要在三个方向:Sync 速度、增量编译速度、产物大小。

**Sync 速度**。最影响 Sync 的是 Configuration Cache 命中率与远程仓库响应速度。`./gradlew :app:dependencies --configuration releaseRuntimeClasspath` 跑一次,Gradle 会输出依赖树,同时把缓存写到 `~/.gradle/caches/`。第二次同样命令应当显著加速。如果你做了任何让 Configuration Cache 失效的变更(改了 `build.gradle.kts`、改了 `settings.gradle.kts`、改了 `gradle.properties`),会看到 Gradle 输出 "Calculating task graph as no cached configuration is available for tasks: ..." —— 这是正常重算,不是异常。

**增量编译速度**。Kotlin 增量编译是 Kotlin 1.7+ 的默认行为,但默认有时不够激进。`kotlin.incremental.useClasspathSnapshot=true` 让 Kotlin 编译器跨模块也能做增量(分析依赖的 classpath snapshot,只在公有 API 变更时下推到下游模块)。NotedX 这个单模块阶段感受不强,第 21 篇拆模块后会显著加速。

**产物大小**。NotedX 当前没有业务代码,debug APK 应当在 7-10 MB 之间(主要是 Compose / Material 3 / Hilt 的运行时),release 经 R8 后可压到 3-5 MB。验证方式:

```bash
./gradlew :app:assembleRelease
ls -lh app/build/outputs/apk/release/
unzip -l app/build/outputs/apk/release/app-release-unsigned.apk | head -50
```

**Compose Compiler Metrics**。后续第 22 篇展开 Compose 性能时,会反复用到 Compose Compiler 输出的 "Stability Report":

```kotlin
// app/build.gradle.kts 的 kotlinOptions 里追加
freeCompilerArgs += listOf(
    "-P", "plugin:androidx.compose.compiler.plugins.kotlin:reportsDestination=${project.layout.buildDirectory.dir("compose_compiler").get().asFile.absolutePath}",
    "-P", "plugin:androidx.compose.compiler.plugins.kotlin:metricsDestination=${project.layout.buildDirectory.dir("compose_compiler").get().asFile.absolutePath}",
)
```

这是按需开,第一篇不必加,但要知道入口在哪。

**验收清单**:

- `./gradlew tasks --all | head -30` 列出所有 task,看到 `assembleDebug`、`assembleRelease`、`installDebug` 等任务。
- `./gradlew :app:dependencies --configuration debugRuntimeClasspath > deps.txt` 输出依赖树,搜索 `androidx.compose.runtime` 能看到 BOM 锁定的版本号(2024.10.00 对应 `1.7.4`)。
- `./gradlew :app:assembleDebug --scan` 跑一次,末尾输出 build scan URL(需要同意条款),里面能看到每个 task 的耗时与缓存命中情况。
- 改一行 Compose 代码(任意 `Text("xxx")` 改成 `Text("yyy")`),再跑 `./gradlew :app:assembleDebug`,Kotlin 编译应当只重编一两个文件,总耗时 < 10 秒。
- 改 `libs.versions.toml` 里 `androidxLifecycle` 版本(例如改成 `2.8.5`),Sync 后能看到对应库被拉取,无须改其他文件。

## 5. 踩坑

**坑 1:AGP 与 Kotlin 版本错配**。最经典的错配是"跟着 Android Studio 升 AGP 但 Kotlin 没升",或者"升 Kotlin 但 AGP 还在两年前版本"。表现各种各样,最难排查的是 Sync 通过、编译通过、但运行时 Compose 函数行为异常或某些 lambda 不被识别为 `@Composable`。修法:**永远按本文给的 AGP × Kotlin × KSP 表对齐**,任何一项升级都要同步检查其它两项。

**坑 2:Compose Compiler 用了 Kotlin 2.0+ 还在写旧 `composeOptions { kotlinCompilerExtensionVersion = "1.5.14" }`**。Kotlin 2.0.20 起这块配置已经不再需要(且会被 IDE 标红警告)。直接删掉,改 apply `org.jetbrains.kotlin.plugin.compose` 插件即可。仍然停留在 Kotlin 1.9 项目升级到 2.0 时,这一步最容易忘 —— 提示通常是 "Compose Compiler Plugin must be used with Kotlin 2.0.20+ for K2 compatibility"。

**坑 3:KAPT 与 KSP 共存**。Hilt 1.x、Room 2.5+ 都已经原生支持 KSP,但很多老教程还在写 `kapt(...)`。NotedX 全栈 KSP,**根本不引入 `kotlin-kapt` 插件**。如果某个三方库还坚持只支持 KAPT(2024-2026 很少见),要么换库,要么单独 apply `kapt` 插件并接受双倍编译时间。

**坑 4:`apply false` 写错位置**。`apply false` 必须写在根 `build.gradle.kts` 的 `plugins {}` 块里;子模块的 `plugins {}` 里要写 `apply true`(或者直接不带 `apply ...`,默认就是 apply)。两边一旦反了,要么报 "Plugin not found",要么报 "Plugin already applied"。

**坑 5:`implementation` vs `api` 用错**。`implementation(libs.x)` 只把依赖暴露在本模块编译期,不传递给下游;`api(libs.x)` 把依赖暴露给所有依赖本模块的下游。**单模块阶段所有依赖都用 `implementation`**;第 21 篇多模块化后,会有少数公共类型库(例如 `:core-model`)需要 `api` 来传递 sealed class / data class 定义。

**坑 6:Maven 镜像写进项目仓库**。国内常见错误是把阿里云镜像直接写进 `settings.gradle.kts` 提交。代价是:海外协作者拉项目后 Sync 慢,某些库阿里云镜像同步滞后导致版本拉不到,CI 在不同区域跑表现不一致。正确做法是**只在本地 `~/.gradle/init.d/` 配镜像**,项目仓库永远保持 `google()` + `mavenCentral()` 的标准源。

**坑 7:`configuration-cache` 开了之后某个 task 报错 "is not yet supported with the configuration cache"**。这是某些老插件(典型是 Gradle Versions Plugin 老版本、某些 Firebase 插件老版本)还没适配 Configuration Cache。两种修法:升级到插件最新版,大多数已经适配;或者临时把那个插件加入兼容列表,显式 `org.gradle.configuration-cache.problems=warn`。永远不要全局关掉 Configuration Cache 来"绕过",那等于把命中率红利全丢了。

**坑 8:`buildFeatures.buildConfig = false` 默认**。AGP 8.0 起 `buildConfig` 默认关闭(精简产物),但很多项目还在用 `BuildConfig.DEBUG`、`BuildConfig.VERSION_NAME`。一旦默认关掉、代码里又引用了 `BuildConfig`,会出现编译失败。NotedX 上面的 `app/build.gradle.kts` 显式打开了,作为默认姿势。

**坑 9:`namespace` 与 `applicationId` 混淆**。`namespace` 是 R 类与 BuildConfig 的包名(只影响编译期);`applicationId` 是 APK 的唯一标识(决定 Play Store 上是哪个 App、决定后续不能改)。两者**可以不同**(典型:`namespace = "com.notedx"` + `applicationId = "com.notedx"` 但 debug 加 `.debug` 后缀让两个变体可同时安装)。两者**第一次发布后就基本不能改**,要在第一天定下来。

**坑 10:升级 AGP 后 R8 行为变更导致线上崩溃**。AGP 8.x 启用了 R8 full mode (`android.enableR8.fullMode=true`)默认更激进的优化,会触发一些老 ProGuard 规则没覆盖的反射场景崩溃(典型是 Gson / 老版本 Retrofit 的反射构造)。**任何 AGP 大版本升级,必须跑一次完整 release 构建并在真机自测全功能**,不能只看 debug 跑通就发版。第 24 篇展开 R8 排障。
