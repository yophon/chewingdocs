# 24-R8、混淆与代码瘦身

> 一句话导读:R8 不是"打开 minifyEnabled 就行"——它替你做了三件事(shrinking / optimization / obfuscation),而你要做的是替它把每个"反射 / 序列化 / 注解扫描"用到的类显式 `-keep` 住,否则 release 包能编过但跑起来 ClassNotFoundException 满屏。

新手对 release 构建最常见的两种反应:**一种是"打开 minifyEnabled 试一下,启动就崩溃,关掉再上线"**——结果 release 包又大又慢、还自带 debug 类名;**另一种是"反正一开就崩,我直接 `-keep class ** { *; }`"**——R8 退化成只做点 dead code 剔除,优化能力大打折扣,APK 减不下来。两种反应背后是同一个误区:**没有把"我用了哪些反射"这件事在工程上列清楚**。本篇就把 Kotlin / Compose / Hilt / Room / Retrofit / kotlinx.serialization 这六个 NotedX 必接库的混淆规则逐一写出,告诉你为什么这条规则要加、加错了会发生什么。

R8 在 Android 平台上已经完全替代了 ProGuard——AGP 3.4 起默认启用 R8,AGP 8.0 后官方文档不再提 ProGuard 作为可选项。**今天写 Android 应用不存在"用不用 R8"的选择题,只有"怎么把 R8 用对"的工程题。** 本篇是 NotedX 进入 [[28 App Bundle、签名与上架]] 之前的最后一道工程关——release APK 必须能跑、足够小、堆栈能反混淆,这三件事的标准答案。

## 1. 机制定位

R8 是把 Java/Kotlin 字节码做四件事的工具,顺序固定:

1. **Shrinking(代码瘦身)**:从 Application、Activity、Service、Receiver、Provider 等入口点出发,基于可达性分析删除所有"没人引用"的类、方法、字段。这一步能把 NotedX 减掉约 30%-40% 的字节码——很多第三方库提供了 100 个 API 你只用 3 个,剩下 97 个被裁掉。
2. **Optimization(优化)**:内联短方法、常量传播、合并相似的代码块、移除永不抛的 try-catch、删除空构造、简化 if 链。这一步通常再减 10%-15%,同时让方法数减少,绕开旧版 Android 的 65k 方法数限制(虽然现在 minSdk 26 几乎不踩这个坑了)。
3. **Obfuscation(混淆)**:把类名、方法名、字段名重命名为 `a`、`b`、`aa`、`ab`……这一步**不缩字节码本身的大小**(指令长度不变),但能极大缩小常量池里字符串的总量,APK 还会再小 10%-20%。同时让逆向工程稍微难一点(但不是真正的安全手段)。
4. **Dexing**:把 Java 字节码翻成 DEX,与 R8 的前三步集成,不是独立步骤。

每一步都可以独立开关,工程上的常见配置:

| 配置 | minifyEnabled | shrinkResources | 适用 |
| --- | --- | --- | --- |
| Debug | false | false | 快编译、可调试 |
| Internal | true | true | 与 release 对齐的预发布 |
| Release | true | true | 上架 |
| Benchmark | false | false | 让 Macrobenchmark 能读 trace,详见 [[23 Macrobenchmark 与 Baseline Profiles]] |

新手最容易踩的坑分两类:

**坑类 A——keep 规则缺失**。R8 看不到的"调用边"会被裁掉:Hilt 通过 `Class.forName` 反射构造你的 Module、Retrofit 通过 proxy 调用接口里的 `suspend fun`、kotlinx.serialization 通过 `Class.serializer()` 拿到 KSerializer。这些都不是普通的方法调用,R8 不会自动识别。需要你手动写 `-keep`(或者库自己附 `consumer-rules.pro`,但有些库忘了或者覆盖不全)。表现:debug 跑没事,release 跑到那一步立刻 `ClassNotFoundException` / `NoSuchMethodException` / `MissingFieldException`。

**坑类 B——keep 规则太粗**。`-keep class com.notedx.** { *; }` 把整个包从 shrinking 和 obfuscation 里挪出去,看上去万无一失,但等于把 R8 的优化能力完全废了。NotedX 第一版犯过这个错,后来精确化规则,APK 从 12 MB 降到 8.4 MB(算上 Baseline Profile)。

工程目标只有一个:**写最少、最精确的 `-keep` 规则,让 R8 既不裁错也不少裁**。本篇逐库给出 NotedX 实际在用的规则,你可以直接抄。

## 2. Android 心智

| 概念 | 含义 |
| --- | --- |
| `proguard-rules.pro` | 项目自定义规则,与项目代码绑定 |
| `consumer-rules.pro` | 库放在 AAR 里的规则,集成方自动应用 |
| `proguard-android-optimize.txt` | AGP 提供的默认优化模板,**总是** include 它 |
| `-keep class X` | 不删除、不改名(类名 + 方法名 + 字段名) |
| `-keepclassmembers class X { ... }` | 类本身可改名,但成员不删不改名 |
| `-keepnames class X` | 不改名但可删(用得少) |
| `-keep,allowobfuscation class X` | 不删但可改名 |
| `-dontwarn` | 忽略某些缺类警告(R8 默认把 warning 当 error) |
| `mapping.txt` | 混淆前后名字对照表,用于反混淆 crash |

`-keep` 与 `-keepclassmembers` 的差别决定了规则的精度:

- `-keep class Foo { ... }`:Foo 类整个保留,**类名也不变**。
- `-keepclassmembers class Foo { ... }`:Foo 可以被改名,但成员不删——典型用于"序列化字段不能改名,但类名无所谓"。
- `-keep,allowobfuscation class Foo { ... }`:Foo 不被删,但类名可以改——用于"反射用 `Class.forName` 拿,但拿到之后调用是基于实例的"。

`mapping.txt` 在 `app/build/outputs/mapping/release/mapping.txt`,每次 release 构建都会生成,**必须随版本号一起归档**——不归档,六个月后用户报的 crash 你看到的是 `a.b.c.d:18` 完全没法读。Play Console 提供"上传 deobfuscation file"功能,APK 上传时勾选自动上传 mapping,Crashlytics 也有同名功能。

R8 在 AGP 8.0+ 默认开了 **Full Mode**,这是与 ProGuard 行为分歧最大的一档:

- 默认假设泛型实参可省(`Class<T>` 信息会被擦除)
- 默认认为非 `-keep` 的类的 `static initializer` 可被消除
- 对接口的默认实现内联更激进
- 对 Kotlin metadata 处理更智能

绝大多数项目都该启用 Full Mode(在 `gradle.properties` 里 `android.enableR8.fullMode=true`,AGP 8.0+ 默认 true),但对于强烈依赖泛型反射的 Retrofit 老版本或 Gson,要额外加几条规则。NotedX 用 kotlinx.serialization,不用 Gson,这件事简单很多。

### 3.2 proguard-rules.pro 全文

文件:`app/proguard-rules.pro`

```text
# ========== 1. 全局基础规则 ==========

# 保留行号信息,让 mapping.txt 能反推堆栈
-keepattributes SourceFile,LineNumberTable
# 改名 SourceFile 为 "SourceFile" 以减少 APK 体积,真实文件名在 mapping.txt
-renamesourcefileattribute SourceFile

# 保留泛型签名,Retrofit / kotlinx.serialization 反射依赖
-keepattributes Signature
# 保留运行时注解,Hilt / Room / Retrofit / Serialization 全部需要
-keepattributes *Annotation*
# 保留异常表(默认就保留,显式写一次)
-keepattributes Exceptions
# 保留 Kotlin 内联类、协程的元数据
-keepattributes RuntimeVisibleAnnotations,RuntimeVisibleParameterAnnotations,RuntimeVisibleTypeAnnotations

# ========== 2. Kotlin 与协程 ==========
# 大部分由 kotlin-stdlib 的 consumer-rules.pro 提供,这里补几条 NotedX 常踩的

# Kotlin Metadata,反射读 Kotlin 类信息(KClass / KType / KCallable)需要
-keep class kotlin.Metadata { *; }
-keepclassmembers class kotlin.Metadata { *; }

# 协程的 Continuation 接口,反射调用 suspend 函数时需要
-keep class kotlin.coroutines.Continuation
-keep class kotlinx.coroutines.flow.** { *; }

# 协程 Internal class,不让 R8 改名以避免 Channel / Flow 内部状态机错乱
-keepclassmembers class kotlinx.coroutines.internal.** { *; }

# ========== 3. Jetpack Compose ==========

# Compose Compiler 已经处理大部分情况,只需要保留几个反射点
# Composable 函数的 @Stable / @Immutable 注解用于运行时(Strong Skipping)判断
-keep @androidx.compose.runtime.Stable class * { *; }
-keep @androidx.compose.runtime.Immutable class * { *; }

# Preview 类只在 IDE 用,release 包可以完全删掉
-assumenosideeffects class androidx.compose.ui.tooling.preview.** { *; }

# 实测 BOM 1.7+ 不需要额外 -keep,Compose Runtime 通过 ServiceLoader 走的不多
```

### 3.3 六个常用库的 -keep 规则

#### 3.3.1 Kotlin 2.0 / kotlinx.coroutines

`kotlin-stdlib` 自带的 consumer rules 已经覆盖 90% 场景,Kotlin 2.0 K2 + Compose 组合需要补两条:

```text
# ========== Kotlin / Coroutines 额外规则 ==========

# Coroutines 的 ServiceLoader 入口(影响 Dispatchers.Main 加载)
-keepnames class kotlinx.coroutines.android.AndroidDispatcherFactory
-keepnames class kotlinx.coroutines.internal.MainDispatcherFactory

# 协程异常处理器走 ServiceLoader
-keep class * implements kotlin.coroutines.CoroutineContext$Element
```

为什么需要这些:`Dispatchers.Main` 在 Android 上通过 `META-INF/services/kotlinx.coroutines.internal.MainDispatcherFactory` 注册;R8 全模式下不识别 ServiceLoader 的目标类,会把它们裁掉,结果 release 包里抛 `IllegalStateException: Module with the Main dispatcher is missing`。1.8+ 已自带规则,旧版本要手动加。

#### 3.3.2 Jetpack Compose

Compose Compiler 把绝大多数 @Composable 函数转成普通 Kotlin 函数,R8 处理它们不需要特殊规则。需要 `-keep` 的只有运行时反射点:

```text
# ========== Compose 额外规则 ==========

# Compose Runtime 通过反射判断 @Stable / @Immutable
-keep,allowobfuscation @interface androidx.compose.runtime.Stable
-keep,allowobfuscation @interface androidx.compose.runtime.Immutable

# 自定义 Saver 通过反射 invoke
-keepclassmembers class * implements androidx.compose.runtime.saveable.Saver { *; }

# Compose Preview 可以从 release 包剔除
-dontwarn androidx.compose.ui.tooling.preview.**
```

Compose 在混淆这一关是最省心的库之一——大部分第三方 Compose 库(Coil AsyncImage、Accompanist Pager)都自带 consumer rules。

#### 3.3.3 Hilt

Hilt 由 KSP 在编译期生成代码,理论上"已经没有运行时反射",但 Hilt 内部仍然通过 `Class.forName` 拿 `_HiltComponents` 与 `@HiltAndroidApp` 标注的 Application:

```text
# ========== Hilt ==========

# Hilt 生成的 _HiltComponents 通过反射加载
-keep class * extends dagger.hilt.internal.GeneratedComponent
-keep class * implements dagger.hilt.internal.GeneratedComponentManagerHolder

# Hilt 入口点(EntryPoint 接口)通过反射查询
-keep @dagger.hilt.android.AndroidEntryPoint class * { *; }
-keep @dagger.hilt.android.HiltAndroidApp class * { *; }
-keep @dagger.hilt.EntryPoint interface * { *; }
-keep @dagger.hilt.InstallIn class * { *; }

# Hilt ViewModel 工厂通过反射拿 ViewModel 类
-keep class * extends androidx.lifecycle.ViewModel { <init>(...); }
-keep @dagger.hilt.android.lifecycle.HiltViewModel class * { <init>(...); }

# Dagger 生成的 Factory / Provider 不能改名
-keep class **_Factory { *; }
-keep class **_Factory$* { *; }
-keep class **_Provide* { *; }
-keep class **_HiltModules { *; }
```

为什么 Hilt 即便 KSP 也要这么多规则:生成的 `_HiltComponents$SingletonC` 由 `Hilt_<YourApp>.onCreate()` 反射构造,R8 看不到这条反射边,会以为这些类没人用直接裁掉。NotedX 第一次配 release 时漏了 `<init>(...)`,启动报 `NoSuchMethodException: <init>()`。

#### 3.3.4 Room

Room 的 `@Database` 类由 Room 自动生成实现,实现类名是 `<YourDb>_Impl`:

```text
# ========== Room ==========

# @Database 实现(<Db>_Impl)与 RoomDatabase 基类
-keep class * extends androidx.room.RoomDatabase
-keep class **_Impl { *; }
-keep class **_Impl$* { *; }
-keep @androidx.room.Database class * { *; }

# 实体类:Room 通过反射 invoke 构造与字段读写
-keep @androidx.room.Entity class * { *; }
-keepclassmembers @androidx.room.Entity class * {
    <init>(...);
    *;
}

# TypeConverter 通过反射调用
-keepclassmembers class * {
    @androidx.room.TypeConverter <methods>;
}
-keep class * extends androidx.room.ProvidedTypeConverter
```

Room 是 NotedX 里规则最长的库,因为它生成的 `_Impl` 后缀类、`TypeConverter` 与 `@Entity` 默认构造都要保留。注意 `@Entity class * { *; }` 保留所有字段——Room 需要按字段反射读写;只 keep 类名不 keep 字段,运行时 `CursorUtil.getColumnIndexOrThrow` 找不到字段会崩。

#### 3.3.5 Retrofit + OkHttp + 各 Converter

Retrofit 的 `@GET` / `@POST` 接口运行时由 `Proxy.newProxyInstance` 生成实现,所有方法签名都靠反射:

```text
# ========== Retrofit ==========

# Retrofit 自带的 consumer-rules.pro 已覆盖大部分,这里补 NotedX 实测必须的几条

# 保留 Retrofit 接口本身(避免被 shrinking 裁掉)
-keep,allowobfuscation,allowshrinking interface retrofit2.Call
-keep,allowobfuscation,allowshrinking class retrofit2.Response

# 保留 Retrofit 标注的接口与方法(R8 看不到调用边)
-keep,allowobfuscation interface * {
    @retrofit2.http.* <methods>;
}

# 保留泛型签名(Retrofit 用反射读 Call<T> / Response<T> 的 T)
-keepattributes Signature, InnerClasses, EnclosingMethod

# OkHttp 内部 platform 类
-dontwarn okhttp3.internal.platform.**
-dontwarn org.conscrypt.**
-dontwarn org.bouncycastle.**
-dontwarn org.openjsse.**

# Retrofit 2.10+ 的 SuspendFun 支持
-keep class kotlin.coroutines.Continuation
```

NotedX 的 ApiService 接口长这样:

```kotlin
// data/src/main/java/com/notedx/data/api/NotesApi.kt
interface NotesApi {
    @GET("notes")
    suspend fun list(): Response<List<NoteDto>>
    @POST("notes")
    suspend fun create(@Body note: NoteDto): Response<NoteDto>
}
```

R8 看的代码里,这个接口"没人 new 它"(只有 Retrofit `create()` 通过 Proxy 拿实现);如果没 `-keep`,接口会被裁掉,启动时 `retrofit.create(NotesApi::class.java)` 抛 `ClassNotFoundException`。`@retrofit2.http.* <methods>` 是关键——它告诉 R8 "任何标注了 retrofit 注解的方法所在的接口都要保留"。

#### 3.3.6 kotlinx.serialization

`kotlinx.serialization` 由 KSP / Kotlin Compiler Plugin 生成 `KSerializer` 实现,在 `MyClass.Companion` 或 `MyClass.serializer()` 上,运行时通过 `Class.serializer()` 或扩展 `serializer()` 找到它:

```text
# ========== kotlinx.serialization ==========

# 保留所有 @Serializable 类与生成的 Companion 序列化器
-keep,includedescriptorclasses class **$$serializer { *; }
-keepclassmembers class * {
    *** Companion;
}
-keepclasseswithmembers class * {
    kotlinx.serialization.KSerializer serializer(...);
}

# 显式标注 @Serializable 的类,保留字段名(字段名 = JSON 字段名)
-keepclassmembers @kotlinx.serialization.Serializable class * {
    <fields>;
    <init>(...);
    *** Companion;
}

# 多态序列化用的 @Polymorphic / SealedClassSerializer
-keep @kotlinx.serialization.Polymorphic class * { *; }

# Json.encodeToString / decodeFromString 的反射入口
-keep class kotlinx.serialization.json.Json { *; }
-keep class kotlinx.serialization.json.Json$Default { *; }
```

为什么 kotlinx.serialization 不像 Gson 一样需要"-keep 所有字段名"还要额外这么多:Gson 通过 Java 反射在运行时遍历字段;kotlinx.serialization 在编译期生成 `$$serializer`,运行时只是查表。但**生成出来的 `$$serializer` 内部嵌套类**,R8 默认会把"看似没人引用"的嵌套类裁掉(全模式下尤其激进)。`-keep class **$$serializer { *; }` 是最关键的一条,缺它必崩。

NotedX 真实踩过的一个坑:`@Serializable data class NoteDto(...)` 字段名混淆后,JSON 序列化输出变成 `{"a":"...","b":"..."}`,后端拿到完全无法解析。修法是上面的 `-keepclassmembers @kotlinx.serialization.Serializable class * { <fields>; }`,让字段名保留(对应 KSerializer 写入的字段名)。

## 4. 调参与验收

NotedX 接完 R8 后的对照表:

| 维度 | minify off | minify on, 默认 rules | minify on, 精确 rules |
| --- | --- | --- | --- |
| APK 大小 | 14.2 MB | 9.8 MB | 8.4 MB |
| dex 方法数 | 86k | 41k | 38k |
| 冷启动 P50 | 800 ms | 720 ms | 540 ms(配合 Baseline Profile) |
| release 是否能跑 | 是 | 启动崩 | 是 |

`shrinkResources = true` 删除"代码里没引用的资源"——通过 R8 的 keep 信息推算资源引用图。注意它**不能**删除"运行时通过 `getIdentifier()` 反射引用"的资源,这种引用要在 `res/raw/keep.xml` 里列白名单。NotedX 全用 Compose 没有 XML layout,主要资源是图标 + string,shrink 后能省 1-2 MB。

验收 release 包能否正常跑:

```text
# 装一次 release-signed APK
./gradlew :app:installRelease

# 用 logcat 实时观察启动崩溃
adb logcat *:E AndroidRuntime:E | grep -i "com.notedx\|R8\|ClassNotFoundException"

# 跑通"启动 → 列表 → 详情 → 登录 → 退出"全链路
# 任意一步崩,记下堆栈,反向找 -keep 缺哪条
```

mapping.txt 反混淆 crash 堆栈用 `java -jar $ANDROID_HOME/cmdline-tools/latest/lib/r8-retrace.jar mapping.txt < crash_stack.txt`,或 Android Studio 的 Logcat → 右键 → Retrace。`bundletool dump manifest --bundle=app-release.aab` 可以验证 Application / Activity 类名正确。

## 5. 踩坑

**坑 1:`shrinkResources = true` 但 `minifyEnabled = false`**。表现:Gradle sync 报错 `Resource shrinker enabled but minifier not`。原因:R8 的资源 shrink 依赖代码 shrink 的引用图,两者必须同时开。修法:两个一起开,或者两个一起关。

**坑 2:`-dontwarn` 用太狠,把真问题盖住了**。新手为了"通过编译"在 proguard rules 里大量加 `-dontwarn com.notedx.**`,结果真的有类找不到也不报错,运行时崩。修法:`-dontwarn` 只用于"明确知道是第三方库自带的 R8 警告但不影响功能"的场景,自己的代码报警告必须修。

**坑 3:`-keep class ** { *; }` 灾难**。这条规则等于"全宇宙都保留",R8 退化成只做 dead code 剔除。NotedX 第一版 APK 12 MB,精确化规则后 8.4 MB,减了 30%。**`** { *; }` 这种规则永远不该写**,如果你不知道精确规则,先用更小范围的 `-keep class com.notedx.specific.** { *; }`,然后用 `-printusage` 看哪些类被裁掉以决定要不要 keep。

**坑 4:`@Keep` 注解保不住嵌套类**。给一个 outer class 加 `@androidx.annotation.Keep` 不会自动 keep 它的嵌套 sealed class / inner class。修法:嵌套类逐个加 `@Keep`,或者写 `-keep class com.notedx.OuterClass$* { *; }`。NotedX 的 `UiState.Loading` / `UiState.Success` / `UiState.Error` 用 sealed interface 实现,需要 `-keep class com.notedx.feature.notes.NoteListState$* { *; }`。

**坑 5:`mapping.txt` 没归档,半年后 crash 看不懂**。Crashlytics 自动反混淆要求 build 时 mapping 已上传,旧版本的 mapping 一旦丢失就只剩 `a.b.c.d` 这种堆栈。修法:CI 归档 mapping.txt 与 versionCode 关联,grafana 上看 crash 时按 versionCode 拉对应 mapping。NotedX 在 GitHub Releases 上每个 tag 附带 mapping.txt 作为 release asset,反混淆时 wget 下来。

**坑 6:Retrofit 接口在 release 包返回 `null`**。表现:`retrofit.create(NotesApi::class.java)` 不抛异常,但返回的 proxy 调用每个方法都 return null。原因:接口里的 `@GET` 注解被 R8 全模式剥离了(以为是 RetentionPolicy.SOURCE)。修法:`-keepattributes *Annotation*` 是这一组的基础,加上 `-keep,allowobfuscation interface * { @retrofit2.http.* <methods>; }`。

**坑 7:kotlinx.serialization 字段名被混淆,JSON 序列化错乱**。表现:本地 debug 与服务器对接正常,release 包发出去的 JSON 字段名全变成 `{"a":1,"b":"..."}`,后端解析失败。原因:`-keepclassmembers @Serializable` 规则没写完整。修法:严格按本篇 3.3.6 节的规则,**字段名 + Companion + serializer 三者都要 keep**。或者在每个字段上加 `@SerialName("explicit_field_name")` 让序列化器用注解里的名字而不是字段名——这是更稳妥的方式,推荐 NotedX 后续所有 DTO 都加。

**坑 8:Hilt `HiltAndroidApp` 启动崩 `NoSuchMethodException`**。表现:release 包启动立刻 crash,堆栈指向 `Hilt_NotedXApp.onCreate`。原因:Hilt 生成的 `Hilt_<App>` 类通过反射调用父类无参构造,但 R8 把无参构造视为"没人用"裁掉了。修法:本篇 3.3.3 节里 `-keep @dagger.hilt.android.HiltAndroidApp class * { <init>(...); }`,`<init>(...)` 是关键。

**坑 9:Room 的 `@Database abstract` 类被 R8 删掉**。表现:`Room.databaseBuilder(...)` 抛 `IllegalArgumentException: Cannot find implementation for ...`。原因:Room 生成的 `<Db>_Impl` 类在编译期生成,R8 看不到"谁会 new 它",直接裁掉。修法:`-keep class * extends androidx.room.RoomDatabase` 与 `-keep class **_Impl { *; }` 两条都要。

**坑 10:Compose @Stable / @Immutable 注解在 release 包失效**。表现:release 包性能反而比 debug 差。原因:R8 把这两个注解视为编译期标记直接剥离了,Compose Runtime 拿不到 → Strong Skipping 退化到保守模式。修法:本篇 3.3.2 节 `-keep,allowobfuscation @interface androidx.compose.runtime.Stable / Immutable`。

**坑 11:LeakCanary 不是 release 库,但被打进 release 包**。LeakCanary 应当通过 `debugImplementation` 引入,而不是 `implementation`;否则 release 包跑它的初始化逻辑,启动多耗几十 ms 且可能因 `-keep` 缺失崩溃。

**坑 12:`-printusage` 与 `-printseeds` 是调试规则**。前者列出 R8 裁掉的所有符号,后者列出所有被 `-keep` 命中的符号。R8 跑完会在 `app/build/outputs/mapping/release/` 下分别生成 `usage.txt` 与 `seeds.txt`,定位"keep 规则到底有没有命中"最直接。NotedX 调试 keep 规则时第一件事就是看这两个文件。

**坑 13:`testCoverageEnabled` 与 R8 冲突**。release buildType 上同时开 `testCoverageEnabled = true`(Jacoco 覆盖率),R8 会和 Jacoco 的字节码插桩打架,出现"unexpected stack map frame"。规则:**覆盖率只在 debug 跑,release 永远不开 testCoverageEnabled**。

## 手动验证

- [ ] `./gradlew :app:assembleRelease` 跑通,产出 `app-release.apk`,大小 < 10 MB(NotedX 基线 8-9 MB)。
- [ ] `./gradlew :app:installRelease` 装到真机能正常启动,执行"登录 → 列表 → 详情 → 退出"全路径无 crash。
- [ ] `app/build/outputs/mapping/release/mapping.txt` 大小 > 500 KB,seeds.txt 与 usage.txt 都存在。
- [ ] 故意把一个 `@Serializable data class` 的字段从 `val name: String` 改成 `val xname: String`,跑 release,看到反序列化 fail——验证 `-keepclassmembers` 不生效时改名能成功,生效时字段名锁死。
- [ ] 把 `-keep @dagger.hilt.android.HiltAndroidApp class * { <init>(...); }` 暂时注释掉,release 启动立刻崩 `NoSuchMethodException`;恢复后正常。
- [ ] 上传 AAB 到 Play Console internal track,Play 自动识别 deobfuscation file,Pre-launch report 看到的 crash 堆栈是可读的类名。
- [ ] 跑 [[23 Macrobenchmark 与 Baseline Profiles]] 的 StartupBenchmark,minify-on 应比 minify-off 启动快(方法少、加载快)。
- [ ] `bundletool dump xml --bundle=app-release.aab --xpath=/manifest/application` 看到 Application 类指向 `com.notedx.app.NotedXApplication`,而不是 `a.b.c.d`。
