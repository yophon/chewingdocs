# 04-协程与结构化并发

> 一句话导读:Kotlin 协程不是"轻量级线程",而是一套**结构化的取消传播协议**;在 Android 上,这套协议刚好与 `Lifecycle` 对齐,所以才有 `viewModelScope` 和 `lifecycleScope` 这两个工程化救命稻草。

第 03 篇把 Kotlin 2.0 的语言形态讲完了。从这一篇开始,所有异步代码都默认是 `suspend`。但在写第一个 `viewModelScope.launch {}` 之前,要把"协程到底解决了什么、为什么是结构化的"这条心智先理顺,否则后面写 Flow / Compose / Hilt 时,异常、取消、内存泄露三件事会反复回来咬人。

熟悉 Java 后端的工程师可以这样类比:协程之于 Kotlin,有点像虚拟线程之于 Project Loom —— 都是"用户态调度的轻量执行单元",但协程多了一层语言级的取消树,这层树才是 Android 生命周期管理的核心抓手。熟悉 JavaScript 的工程师可以把 `suspend fun` 看作 `async function`,但 `await` 隐含,且自动绑定到一个明确的 `CoroutineScope`,作用域结束所有子协程一起死。

## 1. 机制定位

来自老 Android 时代的工程师,通常带着以下三种心理负担过来:

**第一种是 AsyncTask 遗民**。`AsyncTask` 在 API 30 被 deprecated,但它留下的心智很顽固 —— "异步就是 doInBackground / onPostExecute 这一对"。这套模型在 Activity 旋转、配置变更、用户按返回键时全线崩溃:任务还在跑,UI 已经死了,callback 触发时 `findViewById` 返回 `null` 或者直接 NPE。后来出现的"WeakReference 包装 Activity"是补丁的补丁,治标不治本。

**第二种是 Thread + Handler 老兵**。手写 `Thread` 加 `Handler.post` 切回主线程,确实能跑,但每开一个线程就要管它的生命周期、取消、异常,几十个并发任务在一起就是一锅"线程池里捞不出来的悬挂回调"。

**第三种是 RxJava 转型期工程师**。RxJava 用 `Observable` / `Single` / `Flowable` 配上 `subscribeOn` / `observeOn` 把上面两类问题统一抽象了,但代价是引入了一整套独立的"流式心智 + 操作符 + 调度器",而且 `Disposable` 的管理仍然要靠 `CompositeDisposable.clear()` 手动塞进生命周期回调里。一旦漏一个,内存泄露立刻见效。

协程对这三类问题给出的答案,核心只有一句话:**异步任务必须依附于一个有界的作用域(`CoroutineScope`),作用域死,任务必死**。这就是"结构化并发"(structured concurrency)的本质 —— 不再有"我启动了一个后台任务,然后忘了它"的可能性。AsyncTask 没有作用域,Thread 没有作用域,RxJava 用 `Disposable` 模拟作用域;协程把"作用域"做成了语言基础设施。

工程层面再加一条:**协程的取消是合作式的**(cooperative cancellation)。挂起点(`suspend` 函数的调用点)会自动检查取消信号;不挂起的 CPU 密集循环必须显式 `ensureActive()` 或 `yield()`,否则取消信号会被忽略。这条规则是后面所有踩坑的源头,本篇第 5 节会反复回来。

最后一条 Kotlin 2.0 / K2 相关的变化:K2 编译器对 `suspend` 函数的状态机生成做了重构,内联挂起 lambda 的边界比 1.9 更严格。一个具体后果是:旧代码里 `crossinline` 写成 `inline` 编译能过的情况,K2 会拒绝。这不是协程语义变化,只是编译期检查变严。本篇的所有示例都遵循 K2 标准。

## 2. Android 心智

协程本身是 Kotlin 语言特性,跟 Android 无关。但 Android 把它的两件事做对了:

**第一,`Dispatchers.Main` 绑定到主线程**(由 `kotlinx-coroutines-android` 模块提供,gradle 引入后自动生效)。`Dispatchers.Main.immediate` 是一个工程化变体 —— 如果当前已经在主线程,就同步执行,避免一次额外的 post 到 message queue。Compose 里所有的 UI 状态更新都走 `Main` 或 `Main.immediate`。

**第二,`androidx.lifecycle` 给出了两个生命周期感知的 `CoroutineScope`**:

| Scope | 生命周期 | 用途 |
| --- | --- | --- |
| `viewModelScope` | 与 `ViewModel.onCleared()` 对齐 | UI 状态层的异步任务,Activity 旋转 / 销毁时自动取消 |
| `lifecycleScope` | 与 `LifecycleOwner` 对齐 | Composable / Activity / Fragment 生命周期内的一次性任务 |

这两个 scope 都是 `SupervisorJob` + `Dispatchers.Main.immediate` 的组合,意味着:**子协程的失败不会拖死兄弟协程,而且默认在主线程入口**。这条默认值与第 11 篇要讲的 UDF(单向数据流)架构是配套的 —— UI 状态层的所有异步逻辑都从主线程入口启动,需要切线程时用 `withContext`,而不是在 Dispatcher 层启动。

### `CoroutineScope` / `Job` / `SupervisorJob` 三件套

`CoroutineScope` 是一个接口,只有一个属性 `coroutineContext`。它的存在意义是"挂载一棵协程树的根"。任何 `launch` / `async` 启动的协程,其 `Job` 都会成为该 scope 的 `Job` 的子节点。

`Job` 是协程的状态机:`Active` -> `Cancelling` -> `Cancelled` 或 `Completed`。父 Job 取消,子 Job 全部取消;子 Job 失败,父 Job 默认也会被取消(这是关键的传播规则)。

`SupervisorJob` 改写了"子失败拖死父"这条规则:子失败只取消自己,父和兄弟不受影响。`viewModelScope` 内部就是 `SupervisorJob`,所以 ViewModel 里一个网络请求失败,不会把同一个 ViewModel 上其他正在跑的协程全部带死。

写自己的 scope 时,记住这个口诀:**"UI 层用 Supervisor,业务层视情况"**。一个并发流水线里,如果"任一步失败就该全部停"(典型如批量上传),用普通 `Job`;如果"各自独立,失败互不影响"(典型如 UI 同时拉三块数据),用 `SupervisorJob`。

### `Dispatchers.IO` / `Default` / `Main` 的工程边界

| Dispatcher | 线程池 | 适用场景 |
| --- | --- | --- |
| `Main` | Android 主线程 | UI 更新、Compose 状态写入、`LiveData.setValue` |
| `Main.immediate` | 同上,但已在主线程时同步执行 | 高频主线程回调,省一次 post |
| `IO` | 默认 64 个线程,可被 elastic 扩展 | 文件、Room、SharedPreferences、网络阻塞 IO、`okhttp` 同步调用 |
| `Default` | CPU 核心数,但至少 2 个 | JSON 解析、图像处理、加密、排序、纯 CPU 计算 |
| `Unconfined` | 调用线程,不切换 | 测试用,生产代码几乎不用 |

工程上的判断标准:**IO 等待外设(磁盘 / 网络 / IPC),Default 等待 CPU**。混用不会立刻报错,但会让线程池配置失效 —— 在 Default 上跑文件读会卡住整个 CPU 核心数级别的小池子,在 IO 上跑 JSON 大对象解析会把 IO 池占满。

Retrofit + `suspend` 函数自带 `Dispatchers.IO` 切换(由 Retrofit 内部的 `SuspendCallAdapter` 处理),Room 的 `suspend` DAO 也是。所以业务代码里写 `viewModelScope.launch { val list = repo.fetch() }` 通常不需要再套 `withContext(Dispatchers.IO)`;只有当你自己写 `FileInputStream` / `Files.readAllBytes` / 同步 OkHttp 调用时,才需要手动切。

### `withContext` 的语义

`withContext(Dispatchers.IO) { ... }` 不是"启动一个新协程",而是"把当前协程切到另一个 Dispatcher 执行这段代码,块结束自动切回来"。它保持调用线程的挂起语义,不会泄露 Job,也不会让作用域多一层。

这是 Android 上**最常用**的协程操作,99% 的"切线程做一件事"场景都用它。区别于:

- `launch(Dispatchers.IO) { ... }`:启动一个新协程,fire-and-forget,不等结果,**绝大多数 Android 业务代码用错的就是这个**。
- `async(Dispatchers.IO) { ... }.await()`:等价于 `withContext`,但多一层 `Deferred` 对象开销,且异常处理路径不同。`async` 适合"并发跑多个,统一 await"。

记住一条:**单个挂起调用切线程,永远用 `withContext`**。

## 3. 工程实现

下面给一段贴近 NotedX 应用的 ViewModel 与 Repository 示例(NotedX 是本系列贯穿项目,见第 01 篇)。展示 `viewModelScope` 起点、`Repository` 内部的 IO 切换、`SupervisorJob` 的工程化用法、`CoroutineExceptionHandler` 的边界。

**第一步:数据层 Repository**

文件路径 `data/src/main/java/com/notedx/data/note/NoteRepository.kt`:

```kotlin
package com.notedx.data.note

import com.notedx.data.note.local.NoteDao
import com.notedx.data.note.local.NoteEntity
import com.notedx.data.note.remote.NoteApi
import com.notedx.data.note.remote.NoteDto
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.async
import kotlinx.coroutines.coroutineScope
import kotlinx.coroutines.withContext
import javax.inject.Inject
import javax.inject.Singleton

@Singleton
class NoteRepository @Inject constructor(
    private val dao: NoteDao,
    private val api: NoteApi,
) {

    /**
     * suspend 函数本身不绑定 Dispatcher。Retrofit + Room 的 suspend DAO
     * 都自带 IO 切换,这里不必再 withContext(Dispatchers.IO)。
     */
    suspend fun listLocal(): List<Note> =
        dao.queryAll().map(NoteEntity::toDomain)

    /**
     * 并发拉两份资源:正文与附件元数据;任一失败应整体回退。
     * coroutineScope 表示"内部任一子协程失败,整个 scope 取消"。
     */
    suspend fun refreshFromRemote(noteId: String): Note = coroutineScope {
        val bodyDeferred = async { api.fetchBody(noteId) }
        val attachmentsDeferred = async { api.fetchAttachments(noteId) }
        val body: NoteDto = bodyDeferred.await()
        val attachments = attachmentsDeferred.await()

        val merged = body.toDomain(attachments)
        // Room suspend DAO 自带 IO Dispatcher
        dao.upsert(merged.toEntity())
        merged
    }

    /**
     * 显式 IO 切换:这里是手写 OkHttp 同步导出场景,没有 suspend 适配。
     */
    suspend fun exportLocalSnapshot(): ByteArray = withContext(Dispatchers.IO) {
        // 假设是阻塞 IO 操作:文件序列化、压缩、AES
        SnapshotEncoder.encode(dao.queryAll())
    }
}
```

逐点解释这段代码体现的几个决策:

**1. `suspend` 函数不显式绑定 Dispatcher**。`NoteDao` 的 `suspend fun queryAll()` 由 Room 自带 IO 切换,`api.fetchBody` 由 Retrofit 自带。Repository 写 `suspend` 但不写 `withContext`,这是当代 Android 工程的标准姿势 —— **让 Dispatcher 在最贴近 IO 的边界处一次性切换,业务层只表达"我会挂起"**。

**2. `coroutineScope { ... }` 是结构化并发的最小单位**。它创建一个临时 scope,任一子协程失败,所有兄弟立刻取消,然后整个 `coroutineScope` 抛出该异常。这是"批量并发 + 强一致语义"的标准写法,等价于 Promise.all 的"any rejection -> overall reject"。

**3. `async + await` 的并发**。两个网络请求并发发出,任一耗时短的拿到结果先 await。注意 `async` 内部不要再做"业务侧"异常处理(典型错误:`async { try { api.fetch() } catch { null } }`),那样错误会被吞,父 scope 不会知道任务失败 —— 留给 caller 决定。

**4. `withContext(Dispatchers.IO)` 显式切线程的场景**。当你拿到一份非 `suspend` 适配的阻塞 API(老的 SDK、JNI 同步调用、`File.readBytes()`)时,把它包在 `withContext(Dispatchers.IO)` 里。其他场景都不需要。

**第二步:ViewModel 入口**

文件路径 `feature/notes/src/main/java/com/notedx/feature/notes/NotesViewModel.kt`:

```kotlin
package com.notedx.feature.notes

import androidx.lifecycle.SavedStateHandle
import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import com.notedx.data.note.NoteRepository
import dagger.hilt.android.lifecycle.HiltViewModel
import kotlinx.coroutines.CancellationException
import kotlinx.coroutines.CoroutineExceptionHandler
import kotlinx.coroutines.Job
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.StateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.flow.update
import kotlinx.coroutines.launch
import javax.inject.Inject

@HiltViewModel
class NotesViewModel @Inject constructor(
    private val repo: NoteRepository,
    savedStateHandle: SavedStateHandle,
) : ViewModel() {

    private val _uiState = MutableStateFlow(NotesUiState())
    val uiState: StateFlow<NotesUiState> = _uiState.asStateFlow()

    /**
     * Refresh 任务句柄,允许"再次点击刷新时取消上一次"。
     */
    private var refreshJob: Job? = null

    /**
     * 异常处理器:落入此处的是 CoroutineExceptionHandler 兜底的未捕获异常。
     * 业务侧应优先用 try/catch + Result 表达可恢复错误;
     * 此处只处理"不应进入用户感知"的最终兜底。
     */
    private val errorHandler = CoroutineExceptionHandler { _, throwable ->
        if (throwable is CancellationException) return@CoroutineExceptionHandler
        _uiState.update { it.copy(isLoading = false, error = throwable.message) }
        // 此处可接入 [[securityLearning]] 中讨论的崩溃/异常上报通道
    }

    fun loadLocal() {
        viewModelScope.launch(errorHandler) {
            _uiState.update { it.copy(isLoading = true) }
            val notes = repo.listLocal()
            _uiState.update { it.copy(isLoading = false, notes = notes) }
        }
    }

    fun refresh(noteId: String) {
        // 取消上一次未完成的 refresh,避免"先慢后快"导致旧结果覆盖新结果
        refreshJob?.cancel()
        refreshJob = viewModelScope.launch(errorHandler) {
            _uiState.update { it.copy(isLoading = true, error = null) }
            val note = repo.refreshFromRemote(noteId)
            _uiState.update { it.copy(isLoading = false, lastRefreshed = note) }
        }
    }
}

data class NotesUiState(
    val isLoading: Boolean = false,
    val notes: List<Note> = emptyList(),
    val lastRefreshed: Note? = null,
    val error: String? = null,
)
```

几个关键决策点:

**1. `viewModelScope` 入口**。所有 UI 逻辑的协程从这里启动,Activity 旋转 / ViewModel 清理时,所有未完成的子协程自动取消。`viewModelScope` 内部是 `SupervisorJob + Dispatchers.Main.immediate`,所以 `_uiState.update` 这种主线程操作直接执行,不需要再切。

**2. `refreshJob?.cancel()` 是"覆盖式异步"的标配**。用户连续点两次刷新,我们要的是"第二次覆盖第一次",而不是"两次结果竞争"。`cancel()` 是合作式的,正在挂起的网络请求会立刻抛出 `CancellationException`,Repository 不需要写任何额外代码就能响应。

**3. `CoroutineExceptionHandler` 的边界**。它只对 `launch` 的根协程生效,**对 `async` 没用**(`async` 异常通过 `await` 抛出)。它也不能"捕获 + 恢复",只能"兜底记录"。业务侧的可恢复错误(网络 404、Token 失效)用 `try/catch` 或 `Result<T>` 表达;`CoroutineExceptionHandler` 用于"程序员失误漏写 try/catch 的最终保险"。

**4. `CancellationException` 必须重新抛出 / 显式放过**。`errorHandler` 里第一行 `if (throwable is CancellationException) return@...`,因为取消不是错误,是协程的正常退出方式。**任何 `catch (e: Throwable)` 里也要这样过滤**,否则会出现"我取消了,但 UI 却显示了错误提示"的诡异行为。

**第三步:Composable 收集状态**

```kotlin
// feature/notes/src/main/java/com/notedx/feature/notes/NotesScreen.kt
@Composable
fun NotesScreen(viewModel: NotesViewModel = hiltViewModel()) {
    val state by viewModel.uiState.collectAsStateWithLifecycle()
    LaunchedEffect(Unit) { viewModel.loadLocal() }

    Column {
        if (state.isLoading) LinearProgressIndicator()
        state.error?.let { Text(it, color = MaterialTheme.colorScheme.error) }
        LazyColumn { items(state.notes, key = Note::id) { NoteCard(it) } }
    }
}
```

`collectAsStateWithLifecycle` 是生命周期感知的收集器,Composable 离开屏幕时停止收集,避免后台 ViewModel 还在更新但 UI 收不到的浪费。这条会在第 05 篇专门展开。

## 4. 调参与验收

协程的"调参"不是改某个数值,而是**判断每个 launch 该挂在哪个 scope、用哪个 Dispatcher、加不加 SupervisorJob**。给一份判断表:

| 场景 | 推荐姿势 |
| --- | --- |
| UI 状态层一次性任务(刷新、保存、登录) | `viewModelScope.launch(errorHandler) { ... }` |
| Composable 生命周期内的副作用 | `LaunchedEffect(key) { ... }` |
| Activity 内不与 ViewModel 关联的工作 | `lifecycleScope.launch { repeatOnLifecycle(STARTED) { ... } }` |
| Application 级别"必须跑完"的任务 | **不要用 `GlobalScope`,改用 WorkManager**(见第 17 篇) |
| 测试 | `runTest { ... }` + `StandardTestDispatcher` |
| 多 IO 并发,任一失败立刻整体取消 | `coroutineScope { val a = async {}; val b = async {} }` |
| 多 IO 并发,各自独立失败不互相影响 | `supervisorScope { ... }` 或 `viewModelScope` 下分别 `launch` |

**验收工具**

第一,**StrictMode 检测主线程 IO**。在 `Application.onCreate` 里:

```kotlin
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder()
            .detectDiskReads()
            .detectDiskWrites()
            .detectNetwork()
            .penaltyLog()
            .build()
    )
}
```

如果你的 `withContext(Dispatchers.IO)` 漏写,Logcat 会立刻打出 `StrictMode policy violation` 警告。

第二,**Android Studio Profiler -> CPU**。开协程调试 (`-Dkotlinx.coroutines.debug` 或 `kotlinx-coroutines-debug` 依赖),Profiler 会按协程标签分组显示挂起 / 恢复点。打开 "Coroutines" tab,能看到每个协程的当前状态、调用栈、被谁取消。

第三,**单元测试用 `runTest`**。`kotlinx-coroutines-test` 提供的 `runTest { }` 把所有挂起延迟"虚拟化",`delay(5000)` 立刻返回而不真等 5 秒,且时间推进可控。配合 `TestDispatcher` 可以确定性验证"取消是否传播"、"父子协程顺序"。

**验收清单**

- [ ] 所有 `launch` 都挂在 `viewModelScope` / `lifecycleScope` / `LaunchedEffect` 上,**没有任何 `GlobalScope.launch`**。
- [ ] 网络 / 文件 / 数据库调用前都能链到一个 `Dispatchers.IO`(显式 `withContext` 或框架内置)。
- [ ] 主线程上没有阻塞 IO(StrictMode 不报警告)。
- [ ] 用户连续触发同一操作(刷新、搜索)能取消上一次,不出现"旧结果覆盖新结果"。
- [ ] 旋转屏幕时,正在跑的协程自动取消(`viewModelScope` 默认行为,无需手动写)。
- [ ] 单测里用 `runTest` 验证了至少一个"父取消 -> 子取消"传播场景。

## 5. 踩坑

**坑 1:`GlobalScope.launch` 是泄露之源**

`GlobalScope` 没有生命周期,任务一旦启动,只能等它自己完成。Android Studio 会给 `GlobalScope.launch` 一个 `@DelicateCoroutinesApi` 警告,提醒你"再想想"。绝大多数 `GlobalScope` 的使用场景应该改成 WorkManager(后台持久任务)或 `Application` 级别的自定义 scope(很少见),后者需要自己写 `CoroutineScope(SupervisorJob() + Dispatchers.Default)` 并在 `Application.onTerminate` 取消 —— 但 `onTerminate` 在实际设备上几乎不会被调用,所以这条思路也是有限的。

**坑 2:`runBlocking` 只能用在测试与 `main()` 入口**

在生产代码(尤其 Android UI 线程)调用 `runBlocking { someSuspend() }` 会阻塞调用线程直到完成,在主线程上就是 ANR 制造机。把已有同步代码迁到协程时,常见错误是"我先用 `runBlocking` 包一下能跑就行",这是技术债的开始。真要兼容,改成 `LiveData<Result<T>>` 或者整条链改成 `suspend` —— 详见第 03 篇互操作章节。

**坑 3:K2 编译器对 `inline + suspend` 的边界变严**

Kotlin 2.0 / K2 对内联的 `suspend` lambda 的状态机生成做了重构。1.9 时代某些"写错但能编译"的代码,K2 会拒绝:

```kotlin
// 1.9 能编译;K2 报错 "Suspension functions can only be called within coroutine body"
inline fun <T> withLog(block: () -> T): T {
    log("start")
    val r = block()
    log("end")
    return r
}

suspend fun fetch() = withLog { repo.fetch() }  // K2: error
```

修法:把 `block: () -> T` 改成 `block: suspend () -> T`,且让 `withLog` 自身 `suspend`。这不是协程语义变化,只是 K2 把以前的漏报变成正确报错。

**坑 4:`try { ... } catch (e: Exception)` 会吞掉 `CancellationException`**

`CancellationException` 是 `Exception` 的子类(更准确说,它是 `IllegalStateException` 的子类)。`catch (e: Exception)` 会捕获它,然后协程以为自己处理掉了取消,继续往下跑,导致"取消失效":

```kotlin
viewModelScope.launch {
    try {
        repo.fetchBig()
    } catch (e: Exception) {       // ← 吞掉了 CancellationException
        _uiState.update { it.copy(error = e.message) }
    }
}
```

修法:`catch (e: CancellationException) { throw e }` 单独处理,或者改成 `catch (e: Exception)` + 末尾显式检测 `if (e is CancellationException) throw e`。Kotlin 1.6+ 引入的 `runCatching` **同样有这个问题**,需要自己加 `if (e is CancellationException) throw e`。

**坑 5:协程不取消 = 没到挂起点**

```kotlin
viewModelScope.launch {
    var i = 0L
    while (true) {
        i++       // 纯 CPU 循环,永远没有挂起点
    }
}
```

`viewModelScope.cancel()` 调了也没用,这个循环不会停。修法:循环里加 `ensureActive()`(协程已取消则抛 `CancellationException`)或 `yield()`(挂起一次让出调度)。Compose `produceState` 这种高频更新场景里也要注意,业务侧重 CPU 计算建议拆到 `Default` Dispatcher 并加协作点。

**坑 6:`async` 异常不进 `CoroutineExceptionHandler`**

`launch` 抛异常 -> `CoroutineExceptionHandler` 捕获。`async` 抛异常 -> 异常存在 `Deferred` 里,直到 `await()` 时重抛。常见错误:

```kotlin
viewModelScope.launch(errorHandler) {
    async { repo.failingOne() }   // 异常会被忽略,errorHandler 收不到
    async { repo.failingTwo() }
}
```

`async` 的结果没 `await`,异常落地无声。修法:要么把 `async` 改回 `launch`,要么明确 `.await()`。

**坑 7:`viewModelScope` 内的 `withContext(Dispatchers.IO)` 抛异常,会把整个 ViewModel 的协程链拖死**

错。`viewModelScope` 是 `SupervisorJob`,根 Job 不会因子失败而取消;但 **`coroutineScope { }` 和 `async { }` 不是 supervisor**,它们的子异常会向上传到第一个 `SupervisorJob` 才停。所以"是否拖死兄弟"看的是从异常源到最近 `SupervisorJob` 路径上有没有 supervisor 屏障,而不是看根 scope。

**坑 8:`Dispatchers.Main.immediate` vs `Dispatchers.Main`**

`Main.immediate` 在已经是主线程时同步执行,避免 post 到 Handler 队列。看似性能优化,但有个副作用:`launch(Main.immediate) {}` 启动一个协程后,**该协程的初始执行可能在 launch 调用返回之前就开始了**。如果你依赖 "launch 后能立刻读到某个变量",`immediate` 会让你掉坑。`viewModelScope` 默认是 `Main.immediate`,这条要知道。

**坑 9:`LiveData.asFlow()` / `Flow.asLiveData()` 是迁移期工具**

第 05 篇会展开:本系列不写 `LiveData` 主线代码。Compose UI 与 `StateFlow` 是当代标准搭档,`LiveData` 留给"还没迁完的老模块"。两边互转的 API 存在,但**新代码不应主动引入 `LiveData`**。

**坑 10:`coroutineScope` 与 `supervisorScope` 选择**

口诀:"全有或全无用 `coroutineScope`,各自独立用 `supervisorScope`"。错用例子:

```kotlin
// 期望"两个都跑,失败不互相影响",但写成了 coroutineScope
suspend fun prefetch() = coroutineScope {
    launch { prefetchImages() }   // 失败 -> 整个 scope 取消 -> 第二个也死
    launch { prefetchVideos() }
}
```

修法:改成 `supervisorScope`。

**坑 11:取消传播到 OkHttp 是 OkHttp 4.6+ 才有的能力**

老版本 OkHttp 的 `Call` 不会因为协程取消而中断网络请求,会一直跑到 HTTP 响应回来才退。OkHttp 4.6+ 配合 Retrofit 的 `KotlinExtensions` 已经把这条接通,**今天用最新 OkHttp 5.x + Retrofit 2.11+ 不必担心**。但如果你接的是某个老 SDK 内部 hardcode 了 OkHttp 3.x,取消可能"假取消"(协程已 cancel,但底层连接还在烧流量)。第 15 篇接 Retrofit 时会再回这条。

**坑 12:`CoroutineExceptionHandler` 必须挂在根协程的 context 上才生效**

```kotlin
viewModelScope.launch {
    launch(errorHandler) {     // ← 嵌套层,这里挂 errorHandler 无效
        throw IllegalStateException("oops")
    }
}
```

`errorHandler` 必须挂在根 `launch` 上,不能挂在子 `launch` 上 —— 因为异常会先沿父子链向上传到根,然后才查 handler。修法:`viewModelScope.launch(errorHandler) { launch { throw ... } }`。

协程不是"语法糖",而是 Android 工程化的承重墙。把"作用域 + 取消传播 + Dispatcher 边界 + 异常路径"这四件事内化,后面 Flow、Compose 副作用、Hilt 注入、Room / Retrofit 集成都顺。下一篇把视角从"启动一次任务"转到"持续观测一条流",讲清 `Flow` / `StateFlow` / `SharedFlow` 的工程边界,以及 Compose 里"生命周期感知收集"的正确姿势。
