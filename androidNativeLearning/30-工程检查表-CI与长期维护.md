# 30-工程检查表、CI 与长期维护

> 一句话导读:1.0 上架不是终点,而是"以后每次能不能在两小时内把热修推到全量"这条流水线的起点——CI 跑通 Unit + UI + Macrobenchmark 三层、Dependabot 把依赖升级变成 PR、Compose BOM 与 AGP 的升级节奏被记在日程上、发布前清单从 0.x 阶段就在打勾,这一篇把过去 29 篇散落的"工程纪律"汇成可勾选的最后一公里。

到这一篇,NotedX 已经能上传 Play Store、能用 Play Integrity 校验请求、能用 Macrobenchmark 量化启动、能用 R8 把包体瘦下来。所有功能都拼齐了,但还差一件最不性感的事:**怎么保证下一次改一行代码 push 上 main,十分钟之内能拿到三层测试结果、半小时之内能拿到可分发的 release artifact,而不是每个 PR 都要本地手动跑一遍 + 截图问"过了吗"。**

CI 不是"加分项",它是把过去 29 篇的所有约束(单向数据流、@Stable 注解、Strong Skipping、Baseline Profile、R8 mapping、bundletool 16 KB 自检、Play Integrity 签名指纹一致)变成"每次提交都自动核对一次"的执行机制。没有它,这些约束在团队 3 人以上、迭代两个月以后就全部塌方——总有人忘了某条规则,总有 PR 偷偷塞进一个 RxJava 调用,总有人为了赶时间 disable 了某个测试。本篇把 CI 骨架、依赖更新节奏、BOM 升级心智、发布前清单这四块串起来,作为系列收尾。

作为收尾,这一篇也是一份"长期维护参考"——上线后的半年到一年里,哪些事必须每周做一次、哪些事每月做一次、哪些事每季度做一次,清单都在末尾。看完这一篇你应该能回答:发售三个月后看到 Play Vitals 启动慢了 200ms,我从哪条线索查起?Android 16 stable 出来了,我什么时候升 targetSdk?Compose BOM 1.8 / 1.9 出了,要不要现在跟?

## 1. 机制定位

写 App 和"维护 App"完全是两种活。前者是"把功能做出来",后者是"让一个稳定的版本流持续半年不出大事"。两者的分水岭就在 CI:

- 没有 CI 的项目,每次发版前都是一次"手动核对仪式":本地跑 unit、本地手测、本地手动跑 macrobenchmark,凭记忆和团队默契保证没漏。但 1.0 之后只要团队大于 1 人 + 迭代超过 3 个月,这种仪式必然失守——总有一次"上线前太忙没测",刚好那次有回归。
- 有 CI 的项目,每个 PR 自动跑 Unit + UI + Macrobenchmark,通过才能 merge;每次 main 提交自动 build release AAB,artifact 留 30 天,任何人需要立刻给一个 build 就拿 artifact;依赖升级 Dependabot 自动开 PR,CI 跑通就能 merge。"发版"从仪式变成日常。

CI 的三层结构对应过去 29 篇的三层关注点:

- **Unit Test**:跑得最快(几十秒到 1 分钟),覆盖 ViewModel / Repository / 纯函数逻辑。任何 PR 都强制跑通,这是最便宜的"逻辑回归网"。
- **UI Test(Instrumented / Compose Test)**:跑得慢(3-10 分钟),需要 Android emulator,覆盖关键流程(登录、列表、详情、提交)。不在每次 PR 都跑——push 到 main 时跑一次,nightly 跑一次完整 suite,发版前手动 trigger 一次。
- **Macrobenchmark**:跑得最慢(10-20 分钟),需要真机或者稳定 emulator,产出启动时间、滚动 frame 率、Baseline Profile。每周跑一次基线对比,发版前必跑;不能在每个 PR 跑(emulator 的 startup time 噪声让结果不稳定)。

这套三层结构是过去几年 Android 社区共识,Google 自己 Now in Android 项目 / DroidKaigi conference app 都是这个模式。本篇给一个可直接使用的 GitHub Actions 骨架,落到具体步骤。

新手最常见的失控写法,是只跑 Unit 不跑 UI,Macrobenchmark "等以后有空再做"。结果:UI 层的状态死锁 / Navigation 回退栈 bug / 重组爆炸只能等真用户报;Baseline Profile 生成是手工活,每次升 Compose BOM 都要"找时间重新生成",通常就不了了之。

## 2. Android 心智

**`./gradlew` 在 CI 第一次跑非常慢,因为要下载 Gradle distribution + 所有依赖。** 一台干净 ubuntu-latest runner 第一次跑大概要 8-12 分钟拉依赖。把 Gradle Build Cache + Dependency cache 配上之后,稳态可以降到 2-3 分钟。Action `gradle/actions/setup-gradle` 自动接管这一切,不用手写 cache key。

**`./gradlew test` 与 `./gradlew connectedAndroidTest` 完全不同。** 前者跑 JVM unit tests(无 emulator,纯 JVM,有 Robolectric 也是 JVM),后者跑 instrumented tests(需要 emulator)。CI 上 Unit 走前者,UI / Compose Test 走后者。Compose Test 默认走 instrumented(`androidTest/` 目录),需要 emulator。

**`reactivecircus/android-emulator-runner` 是 Android CI 上 emulator 跑测试的事实标准。** 它在 GitHub Actions 上启一个 KVM-accelerated emulator,跑完测试关掉。配 ABI / API level / target / profile,稳定性比手搓 `emulator` 命令好太多。

**Macrobenchmark 必须独立模块(`:macrobenchmark`),`:app` 不能自己跑 benchmark。** 这是 Android benchmark library 的强制要求——benchmark 模块负责 launch 被测 App,自身不在被测进程里。CI 跑 macrobenchmark 走 `./gradlew :macrobenchmark:connectedReleaseAndroidTest`,target 是 release variant,不是 debug。

**Baseline Profile 不是 CI 每次都生成,而是"定期重新生成 → commit 到仓库"。** AGP 会把 `baseline-prof.txt` 包进 release AAB,Play Store 收到后自动分发。CI 流程是:每周或每次大改动后,手动 trigger 一个 "Regenerate Baseline Profile" workflow → 跑 BaselineProfileRule → 把生成的 `baseline-prof.txt` 通过 PR 提交到 `:app/src/main/baseline-prof.txt`。不要每次 PR 都重新生成,profile 抖动反而会拖慢启动。

**Dependabot 与 Renovate 二选一,本系列选 Dependabot。** GitHub 内建、零额外配置、PR 直接打开。Renovate 功能更强但要装第三方 App + 配 `renovate.json`,小团队收益不大。Dependabot 的配置是 `.github/dependabot.yml`,关键是给 Gradle / GitHub Actions 两个 ecosystem 都开。Kotlin 项目里 Dependabot 能识别 `libs.versions.toml` 里的版本号,自动开 PR 升 version catalog。

**Compose BOM 的升级节奏。** Google 官方推荐"BOM 一升,所有 Compose artifact 跟着升"——不要单独升某个 `compose.material3` 而 BOM 不动。Compose BOM 大约每 6-8 周 minor 一发,每个 minor 通常都有破坏性变更(API 重命名、行为变化、新约束)。生产建议:**新 BOM 出来后等 2-3 周再升,等社区把第一波坑踩完。** Now in Android / accompanist 这些大项目升完之后再升你自己的。Compose 1.7 → 1.8 跨度的破坏性变更主要在 Lookahead / Pager / TextField2 上,升级前看 changelog。

**AGP 与 Kotlin 的版本对齐。** AGP 8.5+ 要求 Kotlin 1.9+(K2 默认 2.0+),AGP 8.6 / 8.7 各自支持的 Kotlin 范围在 [官方兼容矩阵](https://developer.android.com/build/releases/gradle-plugin#api-level-support) 里写明。Compose Compiler 从 Kotlin 2.0.20 起内嵌于 Kotlin 自身,不再单独追版本号。AGP 一年一个 stable major,Kotlin 一年 1-2 个 stable major,升的时候按"AGP → Kotlin → Compose BOM"这个顺序,先升 AGP,稳了再升 Kotlin,最后升 BOM——倒过来升会撞到 Compose Compiler 与 Kotlin 不匹配。

**R8 mapping 上传是 Play Console / Crashlytics 都能自动做的事。** Play Console 在你上传 AAB 时自动收 mapping;Crashlytics 通过 Firebase Gradle plugin 在 release build 时自动上传(plugin 配 `mappingFileUploadEnabled = true`)。CI 不用单独传,只要 plugin 配对即可。但如果你用别的 crash 平台(Sentry、Bugsnag),要在 CI 加一步 plugin 对应的 upload-task。

## 3. 工程实现

### 3.1 `.github/workflows/ci.yml`(Unit + UI + Macrobenchmark 三层)

```yaml
# .github/workflows/ci.yml
name: ci

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

env:
  JAVA_VERSION: '17'

jobs:
  lint-and-unit:
    name: Lint & Unit
    runs-on: ubuntu-latest
    timeout-minutes: 25
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: ${{ env.JAVA_VERSION }}

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3
        with:
          cache-read-only: ${{ github.ref != 'refs/heads/main' }}

      - name: Detekt + Android Lint
        run: ./gradlew detekt :app:lintRelease --no-daemon --stacktrace

      - name: Unit tests (JVM)
        run: ./gradlew testReleaseUnitTest --no-daemon --stacktrace

      - name: Upload unit test report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: unit-test-report
          path: '**/build/reports/tests/'
          retention-days: 7

  ui-test:
    name: Compose UI Test (emulator)
    runs-on: ubuntu-latest
    timeout-minutes: 60
    needs: lint-and-unit
    strategy:
      fail-fast: false
      matrix:
        api-level: [29, 35]  # 覆盖 minSdk 之上一档 + targetSdk
    steps:
      - uses: actions/checkout@v4

      - name: Enable KVM
        run: |
          echo 'KERNEL=="kvm", GROUP="kvm", MODE="0666", OPTIONS+="static_node=kvm"' \
              | sudo tee /etc/udev/rules.d/99-kvm4all.rules
          sudo udevadm control --reload-rules
          sudo udevadm trigger --name-match=kvm

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: ${{ env.JAVA_VERSION }}

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3

      - name: AVD cache
        uses: actions/cache@v4
        id: avd-cache
        with:
          path: |
            ~/.android/avd/*
            ~/.android/adb*
          key: avd-${{ matrix.api-level }}

      - name: Warm up AVD
        if: steps.avd-cache.outputs.cache-hit != 'true'
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: ${{ matrix.api-level }}
          target: google_apis
          arch: x86_64
          profile: pixel_6
          force-avd-creation: false
          emulator-options: -no-window -gpu swiftshader_indirect -noaudio -no-boot-anim
          disable-animations: true
          script: echo "AVD warmed up"

      - name: Run UI tests
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: ${{ matrix.api-level }}
          target: google_apis
          arch: x86_64
          profile: pixel_6
          force-avd-creation: false
          emulator-options: -no-window -gpu swiftshader_indirect -noaudio -no-boot-anim
          disable-animations: true
          script: ./gradlew :app:connectedDebugAndroidTest --no-daemon --stacktrace

      - name: Upload UI test report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: ui-test-report-api${{ matrix.api-level }}
          path: '**/build/reports/androidTests/'
          retention-days: 7

  macrobenchmark:
    name: Macrobenchmark (release, baseline)
    runs-on: ubuntu-latest
    timeout-minutes: 60
    needs: lint-and-unit
    # 不在每个 PR 跑,只在 main / weekly schedule / 手动 trigger
    if: github.event_name != 'pull_request'
    steps:
      - uses: actions/checkout@v4

      - name: Enable KVM
        run: |
          echo 'KERNEL=="kvm", GROUP="kvm", MODE="0666", OPTIONS+="static_node=kvm"' \
              | sudo tee /etc/udev/rules.d/99-kvm4all.rules
          sudo udevadm control --reload-rules
          sudo udevadm trigger --name-match=kvm

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: ${{ env.JAVA_VERSION }}

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3

      - name: Run Macrobenchmark
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: 33
          target: aosp_atd  # 自动化测试设备,启动更稳
          arch: x86_64
          profile: pixel_6
          force-avd-creation: false
          emulator-options: -no-window -gpu swiftshader_indirect -noaudio -no-boot-anim
          disable-animations: true
          script: |
            ./gradlew :macrobenchmark:connectedReleaseAndroidTest \
              -P android.testInstrumentationRunnerArguments.androidx.benchmark.suppressErrors=EMULATOR \
              --no-daemon --stacktrace

      - name: Upload benchmark report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: macrobenchmark-report
          path: |
            **/build/outputs/connected_android_test_additional_output/
            **/build/outputs/androidTest-results/
          retention-days: 30

  release-bundle:
    name: Build & Sign AAB
    runs-on: ubuntu-latest
    timeout-minutes: 30
    needs: [lint-and-unit, ui-test]
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: ${{ env.JAVA_VERSION }}

      - name: Setup Gradle
        uses: gradle/actions/setup-gradle@v3

      - name: Decode upload keystore
        env:
          KEYSTORE_B64: ${{ secrets.UPLOAD_KEYSTORE_B64 }}
        run: |
          mkdir -p keystore
          echo "$KEYSTORE_B64" | base64 -d > keystore/notedx-upload.jks

      - name: Build release AAB
        env:
          UPLOAD_KEYSTORE_PATH: keystore/notedx-upload.jks
          UPLOAD_KEYSTORE_PASSWORD: ${{ secrets.UPLOAD_KEYSTORE_PASSWORD }}
          UPLOAD_KEY_ALIAS: ${{ secrets.UPLOAD_KEY_ALIAS }}
          UPLOAD_KEY_PASSWORD: ${{ secrets.UPLOAD_KEY_PASSWORD }}
          CI_BUILD_NUMBER: ${{ github.run_number }}
        run: ./gradlew :app:bundleRelease --no-daemon --stacktrace

      # R8 mapping 上传:这里同时给 Play(走 AAB 自动)和 Crashlytics(走 Firebase plugin)
      - name: Upload Crashlytics mapping
        env:
          FIREBASE_TOKEN: ${{ secrets.FIREBASE_TOKEN }}
        run: |
          ./gradlew :app:uploadCrashlyticsMappingFileRelease --no-daemon --stacktrace

      - name: Upload AAB artifact
        uses: actions/upload-artifact@v4
        with:
          name: release-aab
          path: app/build/outputs/bundle/release/*.aab
          retention-days: 30

      - name: Upload R8 mapping artifact
        uses: actions/upload-artifact@v4
        with:
          name: release-mapping
          path: app/build/outputs/mapping/release/mapping.txt
          retention-days: 90  # 久存,玩家崩溃报告解 stack 用

      # 自动发到 Play Internal track(可选)
      - name: Deploy to Play Internal
        if: github.event_name == 'workflow_dispatch'
        uses: r0adkll/upload-google-play@v1
        with:
          serviceAccountJsonPlainText: ${{ secrets.PLAY_SERVICE_ACCOUNT_JSON }}
          packageName: com.example.notedx
          releaseFiles: app/build/outputs/bundle/release/app-release.aab
          track: internal
          status: completed
          mappingFile: app/build/outputs/mapping/release/mapping.txt
```

要点说明:

- `concurrency` 块让同一分支并发 push 时取消前一次,省 runner 时间。
- `lint-and-unit` 必须 PR 都跑,gate 后续两层。
- `ui-test` 在两个 API level 上跑 matrix(29 与 35,代表 minSdk + 1 与 targetSdk),发现"我代码只在新 API 跑得过,29 上跑不过"的兼容回归。
- `macrobenchmark` 跳过 PR,只在 main / 手动 trigger / cron(下一节加)跑,因为 emulator startup 在并发场景下噪声大,PR 跑出来不可信。
- `release-bundle` 用 `secrets.UPLOAD_KEYSTORE_B64` 注入签名,本地 `base64 -i notedx-upload.jks -o keystore.b64` 把 jks 编码后贴进 GitHub Secrets。
- mapping.txt artifact 留 90 天,Crashlytics 在 mapping 过期前会用本地的解 stack。

### 3.2 `.github/dependabot.yml`(依赖自动升级)

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "gradle"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    groups:
      androidx:
        patterns:
          - "androidx.*"
      compose:
        patterns:
          - "androidx.compose:*"
          - "androidx.compose.*"
      hilt:
        patterns:
          - "com.google.dagger:*"
      coroutines:
        patterns:
          - "org.jetbrains.kotlinx:kotlinx-coroutines-*"
    ignore:
      # Compose BOM 走手动节奏(2-3 周后跟),Dependabot 不要自动升 major
      - dependency-name: "androidx.compose:compose-bom"
        update-types: ["version-update:semver-major"]
      # AGP 跟 Kotlin 必须同步升,不要 Dependabot 自动升
      - dependency-name: "com.android.tools.build:gradle"
      - dependency-name: "org.jetbrains.kotlin:*"

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"
    open-pull-requests-limit: 3
```

`groups` 让 androidx / Compose / Hilt 各自合并成一个 PR,不至于每周收 20 个独立 PR。`ignore` 块把 AGP / Kotlin / Compose BOM major 升级排除——这三件需要人脑决定时机,不应该自动 PR。

### 3.3 `.github/workflows/macrobenchmark-weekly.yml`(每周基线对比)

```yaml
# .github/workflows/macrobenchmark-weekly.yml
name: macrobenchmark-weekly

on:
  schedule:
    - cron: '0 4 * * 1'  # 每周一 UTC 04:00
  workflow_dispatch:

jobs:
  benchmark-and-report:
    runs-on: ubuntu-latest
    timeout-minutes: 60
    steps:
      - uses: actions/checkout@v4

      - name: Enable KVM
        run: |
          echo 'KERNEL=="kvm", GROUP="kvm", MODE="0666", OPTIONS+="static_node=kvm"' \
              | sudo tee /etc/udev/rules.d/99-kvm4all.rules
          sudo udevadm control --reload-rules
          sudo udevadm trigger --name-match=kvm

      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '17'

      - uses: gradle/actions/setup-gradle@v3

      - name: Run Macrobenchmark
        uses: reactivecircus/android-emulator-runner@v2
        with:
          api-level: 33
          target: aosp_atd
          arch: x86_64
          profile: pixel_6
          script: |
            ./gradlew :macrobenchmark:connectedReleaseAndroidTest \
              -P android.testInstrumentationRunnerArguments.androidx.benchmark.suppressErrors=EMULATOR

      - name: Compare with last baseline
        run: |
          python3 scripts/benchmark_compare.py \
            --current build/outputs/connected_android_test_additional_output/ \
            --baseline scripts/benchmark_baseline.json \
            --threshold-startup-ms 50 \
            --threshold-frame-99p-ms 5
      
      - name: Open issue if regression
        if: failure()
        uses: peter-evans/create-issue-from-file@v5
        with:
          title: "Macrobenchmark regression detected"
          content-filepath: ./benchmark-regression.md
          labels: performance, automated
```

`benchmark_compare.py` 是 100 行的 Python 脚本,把当前 benchmark JSON 与 baseline JSON 对比,超出阈值就 exit 1 + 写一份 markdown 给 issue 模板用。本系列不展开 Python 实现,核心思路是"持续监控 startup p50 / frame 99p,任何一项较 baseline 退步超过阈值就报警"。

### 3.4 Compose BOM / AGP / Kotlin 升级节奏

把升级节奏写到 `docs/UPGRADE_CADENCE.md`,每季度复核一次:

| 依赖 | 节奏 | 触发条件 | 流程 |
| --- | --- | --- | --- |
| Compose BOM minor (1.7 → 1.8) | 新版发布后 2-3 周 | Now in Android 升完 + accompanist 跟上 | 手动改 `libs.versions.toml`,跑 detekt + unit + UI + macrobenchmark 三层全通过 |
| Compose BOM patch | 新版发布后 1 周 | Dependabot 开 PR + CI 通过 | merge |
| AGP minor (8.5 → 8.6) | 新版 stable 后 1 个月 | Kotlin 同时支持 | 手动改 `libs.versions.toml`,跑全套 CI,本地真机冷启动验一次 |
| AGP major (8.x → 9.x) | 新 major 发布后 2-3 个月 | 等社区跟完 | 同上 + 单独写一个 changelog branch,QA 一周 |
| Kotlin minor (2.0 → 2.1) | 新版发布后 1 个月 | AGP 与 Compose Compiler 跟上 | 同上 |
| targetSdk 升级(35 → 36) | 新 API level stable 后 6 个月 | Play Store 强制时间 - 3 个月 | 单独 sprint,过 Android 16 行为变更清单(参考 Google migration guide) |
| 其它 androidx artifact | Dependabot 自动 | CI 通过 | merge |

不要把所有升级都堆到"快发版前一周",任何升级都可能引入回归——升级必须发生在距离下次发版至少 2 周的窗口,留出回滚时间。

## 4. 调参与验收

**CI 总时长目标。** Lint+Unit < 5 分钟,UI 单 API level < 15 分钟,Macrobenchmark < 25 分钟,Release AAB build < 8 分钟。如果你的 CI 任何一段超过这个时间,大概率是 cache 没配好(Gradle build cache、AVD cache)、或者测试写得太重(单元测试里 await UI 动画结束)。

**Gradle Build Cache 命中率。** `gradle/actions/setup-gradle@v3` 默认启用 build cache + 配置缓存,稳态命中率 60-80%(改一个文件,只重编那个 module)。如果命中率低于 40%,看看是不是有"每次都生成新文件"的 task(把 build number 写进资源)在反复 invalidate。

**emulator 启动时长。** AVD cache 命中后 emulator 启动 30-60 秒,未命中 3-5 分钟。强烈建议第一次 warmUp 一次然后 cache,后续 jobs 共享。

**Unit / UI / Macrobenchmark 各覆盖什么。**

- Unit 必盖:ViewModel 的 state 转换、Repository 的合并 / 缓存逻辑、纯算法(加密、序列化、parser)。不盖 Compose UI 渲染(成本远大于收益)。
- UI 必盖:登录流、列表 → 详情、表单提交、Navigation 回退栈、Predictive Back 行为。不盖每个 Composable 的所有 Modifier 组合。
- Macrobenchmark 必盖:冷启动、温启动、首屏 LazyColumn 滚动、关键转场动画 frame time。不盖每个页面。

**verdict:这一篇完成的标志。**

- main 分支一次推送 → 15 分钟内得到 Lint+Unit 绿、UI 两个 API matrix 绿、Macrobenchmark JSON 报告产物。
- 任何 PR 一次推送 → 8 分钟内得到 Lint+Unit + UI 绿,无需手动 trigger。
- main 分支 push → Release AAB artifact 可在 GitHub Actions artifact 里直接下载。
- Dependabot 每周一开 1-5 个分组 PR,CI 通过即可 merge。
- Macrobenchmark weekly workflow 跑过至少一次,跑出与 baseline 对比的 markdown 报告。

## 4.1 发布前清单(每次大版本上 Production 前完整走一遍)

**项目层**

- [ ] `versionCode` 严格大于上一次 Play Console 上传的最大 versionCode(看 Production / Internal / Closed / Open 所有 track)。
- [ ] `versionName` 符合团队 SemVer 约定(MAJOR.MINOR.PATCH)。
- [ ] `applicationId` 与 Play Console 录入的包名一致;`applicationIdSuffix` 没有错误地泄漏到 release variant。
- [ ] `compileSdk` / `targetSdk` = 35(或当前主线)。`minSdk` 与 Play Console "Device catalog" 期望的覆盖率匹配。
- [ ] `release` build type 已启用 `isMinifyEnabled = true` + `isShrinkResources = true`。
- [ ] `signingConfig` 正确指向 release upload keystore;`enableV2Signing` / `enableV3Signing` 为 true。
- [ ] `bundle { language / density / abi { enableSplit = true } }` 全开,或有意识地关掉某项(并记在 changelog 里说明原因)。

**权限审查**

- [ ] `AndroidManifest.xml` 所有 `<uses-permission>` 都是当前业务真的需要的;砍掉历史遗留权限。
- [ ] 任何"敏感权限"(LOCATION、CAMERA、RECORD_AUDIO、READ_MEDIA_*、POST_NOTIFICATIONS、QUERY_ALL_PACKAGES、SCHEDULE_EXACT_ALARM、USE_FULL_SCREEN_INTENT)都能在 Data Safety 表单里说清"用途、是否第三方共享、用户能否关闭"。
- [ ] 不再使用 `READ_EXTERNAL_STORAGE` / `WRITE_EXTERNAL_STORAGE` 走 Scoped Storage(targetSdk 30+ 强制)。
- [ ] 前台服务声明 `foregroundServiceType` 与运行时调用 `startForeground(notification, type)` 的 type 一致;权限在 manifest 有声明(参见 [[androidNative]] 第 17 篇)。
- [ ] `<queries>` 块声明了所有 `Intent.resolveActivity` 会查询的包(`MANAGE_PACKAGES` 不要无脑加)。

**Play Console 元信息**

- [ ] Data Safety 表单填齐:数据收集类型、共享对象、加密传输、用户可否删除。
- [ ] Privacy Policy URL 可访问 (HTTPS),内容与 Data Safety 一致。
- [ ] App Content:广告声明、目标受众、内容评级问卷、政府 App 标识(如适用)。
- [ ] Store listing:标题、简介、详细描述、icon (512×512)、feature graphic (1024×500)、screenshots ≥ 4 张、tablet screenshots(如声明支持平板)。
- [ ] Pre-launch report 通过,无 critical issue;ANR / Crash 在 pre-launch run 中为 0。

**签名与 16 KB**

- [ ] AAB 用 release upload keystore 签名;SHA-256 与 Play Console "App signing" 里 upload certificate 一致。
- [ ] OAuth client / Maps API / Firebase 的 SHA-1 / SHA-256 用的是 **app signing key**(Play 托管那把),不是 upload key。
- [ ] `bundletool build-apks --connected-device + install-apks` 装到真机能启动、能登录、能加载首屏。
- [ ] `find aab-unpacked -name "*.so"` 列出的所有 `.so` 通过 `readelf -lW` 显示 LOAD 段 Align = `0x4000`(16 KB)。
- [ ] 关键设备矩阵:Pixel 系列、三星 S24/S25、小米 14/15、华为 Mate 60 / Pura 70 (HMS 路线)、一台老旧 Android 8.0 - 9.0 机器,都装一次走主流程。

**性能与稳定性**

- [ ] Macrobenchmark startup median 不比上一版退步 > 50ms;p99 不退步 > 200ms。
- [ ] LazyColumn 关键列表 frame time p99 < 16ms;无 frozen frames (> 700ms)。
- [ ] Baseline Profile (`app/src/main/baseline-prof.txt`) 已更新到本次 release 对应的关键路径。
- [ ] R8 `mapping.txt` 通过 Firebase Gradle plugin 自动上传 Crashlytics(或对应 crash 平台);Play Console 自动收到上传方的 mapping。
- [ ] StrictMode 在 debug build 启用,release build 不启用;debug 跑主流程没 violation。
- [ ] Compose Compiler Metrics 跑一次(`./gradlew :app:assembleRelease -P composeCompilerMetrics=...`),不稳定类比例较上版没增加。

**网络与安全**

- [ ] `network_security_config.xml` 禁止 cleartext (`cleartextTrafficPermitted="false"`);生产域名钉扎 SPKI(参见 [[androidNative]] 第 29 篇)。
- [ ] `android:debuggable` 在 release manifest 不存在或为 false(CI 加 check)。
- [ ] Play Integrity warmUp 在 App 启动调用;敏感操作走 nonce + payload hash 协议。
- [ ] OkHttp Interceptor 不在 release 输出明文请求体到 logcat(`HttpLoggingInterceptor.Level = NONE`)。
- [ ] `WebView` 没有调 `setJavaScriptEnabled(true)` 同时 `setAllowFileAccess(true)`;`addJavascriptInterface` 仅暴露最小函数。

**国际化与可访问性**

- [ ] `LocaleConfig` 列全 ship 的语言;per-app language preferences 在系统设置里能切换并立即生效(参见 [[androidNative]] 第 27 篇)。
- [ ] TalkBack 跑主流程能朗读所有交互;关键自定义 Composable 有 `Modifier.semantics` 与 `contentDescription`。
- [ ] 字体可缩放至 `200%` 不溢出(`fontScale = 2.0f` 在 Layout Inspector 验证)。
- [ ] 暗色模式跑主流程颜色对比度 ≥ 4.5:1;特殊高亮 / 状态色不依赖单一色相区分。

**大屏与折叠屏**

- [ ] `WindowSizeClass` 三档(Compact / Medium / Expanded)各跑一次主流程;平板布局不"一栏拉伸",而是分栏。
- [ ] 折叠屏在折叠 / 展开瞬态没崩、没丢导航状态。
- [ ] `resizeableActivity = true`(或不显式声明,默认 true);多窗口模式能跑。

**版本兼容矩阵**

- [ ] minSdk 26、minSdk + 1 (27)、minSdk + 几档(29)、上一稳定 (33)、targetSdk (35) 各跑一次冷启动 + 关键流程。
- [ ] Android 14 Foreground Service Type 强制 / Android 15 16 KB / Android 16 (如已 stable) 的预测性行为变更已在对应 API level 验证。

**发布流程**

- [ ] Release notes(英文 + 中文 + 其它 ship 语言)写好,提交到 Play Console "What's new"。
- [ ] Internal track 测试 ≥ 24 小时无回归报告。
- [ ] 选择 Production rollout 比例:1% → 等 24-48 小时 → 5% → 等 24 小时 → 20% → 50% → 100%。
- [ ] Halt rollout 触发条件文档化:crash 率较 baseline +0.1%、ANR 率 +0.1%、关键流程转化率 -5%。
- [ ] Hotfix 通道演练过:从发现 bug → 提 PR → CI 通过 → bundleRelease → upload Internal → promote Production rollout,全流程 < 4 小时。

## 5. 踩坑

**`./gradlew connectedAndroidTest` 在 GitHub Actions 上没 emulator 跑不动。** 默认 ubuntu-latest 没 emulator,要靠 `reactivecircus/android-emulator-runner` 启一个。macos-latest runner 有原生 KVM 等价物但比 ubuntu 慢且贵 10x,不要用。

**Macrobenchmark 在 emulator 上跑结果不稳。** `androidx.benchmark.suppressErrors=EMULATOR` 是必须加的,否则会直接抛错拒跑。即使加了,emulator 数据也只能用来"看有没有大幅回归",绝对值不能拿来对外报。真正稳定的 baseline 要在物理设备上跑,这是为什么很多团队搞 self-hosted runner + 一台固定的真机或 Firebase Test Lab 物理设备。

**Compose Compiler Metrics 在 CI 跑会产生很大的报告文件,artifact 越攒越多。** retention-days 设短(7 天),或者只 archive 最关键的 summary 文件。

**R8 mapping 上传 Crashlytics 失败。** 通常是 Firebase Gradle plugin 配置 `mappingFileUploadEnabled = true` 漏了,或者 service account 缺权限。在 release build 后跑 `./gradlew :app:uploadCrashlyticsMappingFileRelease` 单独 verify。

**Dependabot 把 `org.jetbrains.kotlin` 升了一个 patch,Compose Compiler 立刻不兼容。** 这是为什么 dependabot.yml 里 ignore Kotlin。任何 Kotlin 升级必须人工核对 Compose Compiler 对应支持版本(Kotlin 2.0.20+ 起内嵌于 Kotlin 自身,问题小了,但 Kotlin 2.x → 3.x 之类大跨越要谨慎)。

**`groups` 在 Dependabot 升级时把 Compose BOM 与其它 androidx 合一个 PR。** Compose 改动通常 break UI Test,跟其它 androidx 一起升会让你不知道是哪个引起的。给 BOM 单独一个 group(或者 ignore semver-major),其它 androidx 一个 group,分开 review。

**`secrets.UPLOAD_KEYSTORE_B64` 在 fork PR 不可访问。** GitHub Actions 默认不给 fork 的 PR 暴露 secrets,这是安全特性。这意味着外部贡献者的 PR 不能跑 release build job——把 release build 限定在 `if: github.event_name != 'pull_request'` 或 `github.event.pull_request.head.repo.full_name == github.repository`(仅同仓 PR)。

**Gradle Daemon 在 CI 启动后不退,占用 runner 内存。** 给所有 gradle 调用加 `--no-daemon`。runner 跑完整个 job 就销毁,不需要 daemon 复用。

**`actions/cache@v4` 缓存 key 用了 `${{ hashFiles('**/*.gradle*') }}`,改 dependabot PR 时 cache 全 miss。** Dependabot 升一个版本号就改 `libs.versions.toml`,hash 变了,cache 全失效,CI 慢 5 倍。`gradle/actions/setup-gradle@v3` 自己处理这件事,不要手动设 cache key。

**`./gradlew detekt` 默认对 generated source 也跑,误报一堆。** 在 `detekt.yml` 里 exclude `**/build/generated/**`。

**`./gradlew :app:lintRelease` 比 `lintDebug` 多查 R8 相关规则,但慢得多。** PR 跑 release lint 是对的,但本地开发跑 debug lint 加快迭代。

**Play Console 自动 mapping 上传只对 AAB 生效,APK 不会被收。** 如果你给国内渠道单独打 APK,那份 mapping 要自己保存,玩家崩溃报告解 stack 时用。

**Compose BOM 升级后 strong skipping 行为微调,某些原本"标记为 stable 的类"被认作 unstable,重组爆增。** 升 BOM 后必跑 Compose Compiler Metrics + Macrobenchmark frame time,任何 page 退步 > 10% 就回退 BOM 调查。

**AGP 8.x → 8.x+1 后 KSP 任务名称变化。** 偶尔 AGP 改了 KSP 任务命名约定(`kspDebugKotlin` → `kspReleaseKotlin` 等),CI 直接调任务名会失败。让 Gradle 自己推断任务名(用 `bundleRelease` 触发全 chain),不要手写 KSP 任务名。

**`R8` 把某个 Hilt / Room / Retrofit / kotlinx.serialization 类裁掉了,release 启动崩。** 已在 [[androidNative]] 第 24 篇展开。CI 跑 release variant 的 UI test 能提前抓到,不要只跑 debug UI test。

**Crashlytics 仪表板上的崩溃数与 Play Vitals 对不上。** 两者采样口径不同,Crashlytics 是显式 SDK 收集,Play Vitals 是 Play Store 在 Play Services 层抓取(可能含未崩溃但 ANR 的事件)。两个都看,不要只信一个。

**Self-hosted runner 上长跑 emulator,KVM 状态偶尔损坏。** GitHub-hosted 用完销毁不存在这个问题;self-hosted 要在 workflow 末尾加一步 `adb emu kill` + `pkill -f qemu`,保证下次跑干净。

## 长期维护节奏(系列收尾)

发售后的半年,以下事项按节奏跑:

**每天**
- 看 Play Vitals 当日 crash / ANR 趋势(异常立即调查)。
- 看 Crashlytics 新增 issue,有"影响 ≥ 100 用户"的归到本周修复。

**每周**
- Dependabot PR review + merge(周一上午一次性处理)。
- Macrobenchmark weekly workflow 报告 review,有回归立刻 issue + 排查。
- Play Console "Statistics" 与"Acquisition reports"看 install / uninstall 比例;uninstall 异常上升找原因。

**每月**
- Compose BOM patch 升级(如有)。
- 跑一次完整发布前清单 dry run,看是否有清单条目过期(API level、Play 政策变化)。
- 复盘 hotfix 流程时长,如果上月 hotfix > 4 小时则查 CI / build / Play review 哪段慢。

**每季度**
- Compose BOM minor 升级(如有,且社区跟齐)。
- AGP minor / Kotlin minor 升级评估,有就升,排在版本计划里。
- 复审权限清单 vs Data Safety,有偏差立即修。

**每半年**
- targetSdk 升级评估:Play Store 通常给 12 个月 transition,提前 6 个月开始测新 API level 行为变更。
- Baseline Profile 全量重新生成。
- 团队复盘文档更新:架构决策记录 (ADR)、命名规范、PR 模板。

**每年**
- AGP major / Kotlin major 升级(如有,且 Compose Compiler 跟上)。
- keystore 安全审查:upload key 备份、Play App Signing 后台审计、CI secret rotation。
- 整套 30 篇笔记 review,把"新版本带来的过期信息"批注 deprecated。

到这里,30 篇都结束了。这套笔记不是"Android 知识词典",而是一条从空骨架到上架灰度、从单元测试到长期维护的工程链路。它的价值不在每篇都精彩,而在 30 篇连起来能让你回答:**下次有人问"我想做个原生 Android App,从零到能上 Play 大概要做哪些事",你不再用一句"很复杂"打发,而是能掏出这 30 个清单告诉他每一步要做什么、有哪些坑、做完怎么验收。**
