# 多模块化:app / feature / core / data

> 一句话:**单模块代码能写到大约 1-2 万行,再往上构建会慢、依赖会乱、并行开发会冲突——这时候才该拆模块**。前 19 篇所有代码都在 `:app`,这一篇之后 NotedX 进入团队工程形态。

---

## 一、何时不该多模块

新人写 Android 最常见的过早优化就是"项目开头就拆 8 个模块"。**单模块不是问题,大单模块才是**。判断标准:

| 现象 | 是否该拆 |
| --- | --- |
| 代码量 < 1 万行 | ❌ 不拆,徒增配置 |
| 一个人写完整个 App | ❌ 不拆 |
| `./gradlew assembleDebug` 全量编译 < 30 秒 | ❌ 不拆 |
| 没有 feature 重复实现 / 没有平台层(`:core-common` 反复被引用)的需要 | ❌ 不拆 |
| 多人并行开发不同 feature,经常 merge 冲突 | ✅ 拆 |
| 全量编译 > 2 分钟,增量编译也明显慢 | ✅ 拆 |
| 同一份 logic 想被 Wear App / Auto App 复用 | ✅ 拆 |
| 团队规模 > 4 人 | 通常 ✅ 拆 |

**NotedX 在前 19 篇都在单模块**,因为它的代码量在 5000-8000 行左右,够用了。**这一篇展示"该怎么拆"作为前瞻**,真实项目要看团队需要。

---

## 二、典型多模块结构

**Now in Android**(Google 官方 sample)的结构是业界标杆:

```
:app                         应用入口、组装、Manifest
├── :feature
│   ├── :feature:home        首页 feature
│   ├── :feature:detail      详情 feature
│   ├── :feature:editor      编辑器 feature
│   ├── :feature:settings    设置 feature
│   └── :feature:search      搜索 feature
├── :core
│   ├── :core:ui             共享 Composable 组件
│   ├── :core:designsystem   主题、颜色、字体
│   ├── :core:common         通用工具、扩展函数
│   ├── :core:model          Domain Model
│   └── :core:testing        测试工具
├── :data
│   ├── :data:note           Note Repository + DTO + Entity
│   ├── :data:user           User Repository
│   └── :data:datastore      DataStore Preferences
├── :network                 Retrofit / OkHttp 配置
└── :database                Room Database / Dao
```

**层级语义**:

- **`:app`**——只负责"组装 + Manifest + 路由表"。**不包含业务代码**。
- **`:feature:*`**——每个屏幕 / 业务功能一个模块。互相**不依赖**(同级 feature 不能 `implementation(:feature:home)`)。
- **`:core:*`**——基础组件,所有 feature 都能引用。
- **`:data:*`**——数据访问层(Repository)。feature 通过接口依赖 data。
- **`:network` / `:database`**——平台基础设施,只暴露给 `:data` 用。

---

## 三、依赖方向规则

**铁律**:依赖只能往下走,不能反向 / 不能横向。

```
            :app
              ↓
         :feature:*  ←─横向不允许─→ :feature:*
              ↓
         :data:* / :core:*
              ↓
        :network / :database
```

具体说:

| 模块 | 能依赖 | 不能依赖 |
| --- | --- | --- |
| `:app` | 所有 :feature, 所有 :core, 所有 :data | - |
| `:feature:home` | `:core:*`, `:data:note`, `:data:user`, `:domain:*` | 其他 `:feature:*`, `:network`, `:database` |
| `:data:note` | `:core:common`, `:core:model`, `:network`, `:database` | 任何 `:feature:*` |
| `:core:ui` | `:core:designsystem`, `:core:common` | 任何 :feature, :data |

**为什么不让 feature 之间互相依赖?** 因为这会导致循环依赖、编译时间膨胀、且让 feature 间存在隐性耦合。要从 home 跳到 detail?**通过 `:app` 的路由表**,而不是 home 直接依赖 detail。

## 四、模块的物理形态

每个模块是 `library`(不是 `application`),`:app` 是唯一的 `application`:

```kotlin
// :feature:home/build.gradle.kts
plugins {
    alias(libs.plugins.android.library)
    alias(libs.plugins.kotlin.android)
    alias(libs.plugins.kotlin.compose)
    alias(libs.plugins.ksp)
    alias(libs.plugins.hilt)
}

android {
    namespace = "com.notedx.feature.home"
    compileSdk = 35
    defaultConfig { minSdk = 26 }
    
    compileOptions {
        sourceCompatibility = JavaVersion.VERSION_17
        targetCompatibility = JavaVersion.VERSION_17
    }
    kotlinOptions { jvmTarget = "17" }
    buildFeatures { compose = true }
}

dependencies {
    implementation(projects.core.ui)
    implementation(projects.core.designsystem)
    implementation(projects.data.note)
    
    implementation(platform(libs.compose.bom))
    implementation(libs.compose.ui)
    implementation(libs.compose.material3)
    implementation(libs.androidx.lifecycle.viewmodel.compose)
    implementation(libs.androidx.hilt.navigation.compose)
    
    implementation(libs.hilt.android)
    ksp(libs.hilt.compiler)
}
```

**`projects.core.ui`** 是 Type-safe Project Accessors(Gradle 7.0+)——比 `project(":core:ui")` 类型安全,IDE 自动补全。需要在 `settings.gradle.kts` 启用:

```kotlin
enableFeaturePreview("TYPESAFE_PROJECT_ACCESSORS")
```

## 五、Convention Plugin:消除 20 份重复 build.gradle.kts

10 个模块就有 10 份 `build.gradle.kts`,每份都重复写 minSdk / compileSdk / Kotlin / Compose 配置——一处改,十处搜替换。

**Convention Plugin** 把这些重复抽到自定义 Gradle 插件:

`build-logic/convention/src/main/kotlin/AndroidLibraryConventionPlugin.kt`:

```kotlin
class AndroidLibraryConventionPlugin : Plugin<Project> {
    override fun apply(target: Project) = with(target) {
        with(pluginManager) {
            apply("com.android.library")
            apply("org.jetbrains.kotlin.android")
        }
        extensions.configure<LibraryExtension> {
            compileSdk = 35
            defaultConfig { minSdk = 26 }
            compileOptions {
                sourceCompatibility = JavaVersion.VERSION_17
                targetCompatibility = JavaVersion.VERSION_17
            }
        }
        extensions.configure<KotlinAndroidProjectExtension> {
            jvmToolchain(17)
        }
    }
}
```

注册:

```kotlin
// build-logic/convention/build.gradle.kts
gradlePlugin {
    plugins {
        register("androidLibrary") {
            id = "notedx.android.library"
            implementationClass = "AndroidLibraryConventionPlugin"
        }
        register("androidLibraryCompose") {
            id = "notedx.android.library.compose"
            implementationClass = "AndroidLibraryComposeConventionPlugin"
        }
        register("androidFeature") {
            id = "notedx.android.feature"
            implementationClass = "AndroidFeatureConventionPlugin"
        }
        // ...
    }
}
```

使用:

```kotlin
// :feature:home/build.gradle.kts(精简到 10 行)
plugins {
    id("notedx.android.feature")
    id("notedx.android.library.compose")
    alias(libs.plugins.ksp)
    alias(libs.plugins.hilt)
}

android { namespace = "com.notedx.feature.home" }

dependencies {
    implementation(projects.core.ui)
    implementation(projects.data.note)
    // 只列业务依赖,基础设施在 convention plugin 里
}
```

每个模块的 build.gradle.kts 从 80 行降到 10 行。Now in Android 项目大量用了这个模式。

## 六、Feature 模块的路由

每个 `:feature:*` 模块自带它的 `NavGraphBuilder` 扩展函数:

```kotlin
// :feature:home/src/main/java/com/notedx/feature/home/Navigation.kt
@Serializable data object HomeRoute

fun NavGraphBuilder.homeScreen(
    onNoteClick: (Long) -> Unit,
    onNewNote: () -> Unit,
) {
    composable<HomeRoute> {
        HomeRoute(
            onNoteClick = onNoteClick,
            onNewNote = onNewNote,
        )
    }
}

// :feature:detail/src/main/java/com/notedx/feature/detail/Navigation.kt
@Serializable data class DetailRoute(val id: Long)

fun NavGraphBuilder.detailScreen(onBack: () -> Unit) {
    composable<DetailRoute> {
        DetailRoute(onBack = onBack)
    }
}
```

`:app` 模块组装路由表:

```kotlin
// :app/src/main/java/com/notedx/NotedXNavHost.kt
@Composable
fun NotedXNavHost(navController: NavHostController) {
    NavHost(navController = navController, startDestination = HomeRoute) {
        homeScreen(
            onNoteClick = { id -> navController.navigate(DetailRoute(id)) },
            onNewNote = { navController.navigate(EditorRoute) },
        )
        detailScreen(onBack = { navController.popBackStack() })
        // ...
    }
}
```

**关键**:`:feature:home` **不依赖** `:feature:detail`,但通过 callback 让上层(`:app`)决定跳哪。这是经典的"事件冒泡 + 高层决策"模式。

## 七、Repository 接口在 `:data`,实现也在 `:data`

```
:data:note
├── NoteRepository.kt          (interface,对外暴露)
├── NoteRepositoryImpl.kt      (实现,internal)
├── di/NoteDataModule.kt       (Hilt @Binds)
└── mapper/                    (DTO/Entity/Domain 互转)
```

```kotlin
// :data:note/.../NoteRepository.kt
interface NoteRepository {
    fun observeAll(): Flow<List<Note>>
    suspend fun refresh()
}

// :data:note/.../NoteRepositoryImpl.kt
internal class NoteRepositoryImpl @Inject constructor(
    private val dao: NoteDao,
    private val api: NoteApi,
) : NoteRepository { ... }

// :data:note/.../di/NoteDataModule.kt
@Module
@InstallIn(SingletonComponent::class)
internal abstract class NoteDataModule {
    @Binds @Singleton
    abstract fun bindRepository(impl: NoteRepositoryImpl): NoteRepository
}
```

**`internal`** 修饰实现类——其他模块只看到 `NoteRepository` 接口,看不到 Impl。这是模块封装的关键。

`:feature:home` 通过 Hilt 直接拿 `NoteRepository`,不知道也不关心它的实现在 `:data:note`。

## 八、`:core:model` 与 `:core:ui` 的边界

**`:core:model`** 放纯数据类(Domain Model):

```kotlin
// :core:model/.../Note.kt
data class Note(
    val id: Long,
    val title: String,
    val content: String,
    ...
)
```

为什么单独?因为 `:data:note` 和 `:feature:home` 都需要它,直接互相依赖会引入循环。

**`:core:ui`** 放可复用的 Composable 组件:

```kotlin
// :core:ui/.../NoteCard.kt
@Composable
fun NoteCard(
    title: String,
    preview: String,
    onClick: () -> Unit,
    modifier: Modifier = Modifier,
) { ... }
```

**关键**:`:core:ui` 的组件**不依赖业务 model**——`NoteCard` 接收 `title: String` / `preview: String`,而不是 `note: Note`。这让它能被任何 feature 复用。

## 九、Compose 在多模块下的注意点

**Compose Compiler** 自 Kotlin 2.0.20 起内嵌于 Kotlin——多模块下,每个用 Compose 的模块都要 apply `kotlin-compose` 插件:

```kotlin
plugins {
    alias(libs.plugins.kotlin.compose)
}

android {
    buildFeatures { compose = true }
}
```

Convention plugin 里集中配置(`AndroidLibraryComposeConventionPlugin`),feature 模块只 apply 这个 convention 就行。

**`@Composable` 函数跨模块调用零成本**——Compose 编译器看到 `import com.notedx.core.ui.NoteCard` 直接生成正常调用。

## 十、版本目录在多模块下

`gradle/libs.versions.toml` **是项目级的,所有模块共享**——这就是为什么把版本号集中到这里。任何模块的 `build.gradle.kts` 用 `libs.compose.ui` / `libs.androidx.room.runtime` 全局一致。

**Bundle**(库的组合)可以减少重复:

```toml
[bundles]
compose = [
    "compose-ui",
    "compose-material3",
    "compose-foundation",
    "compose-ui-tooling-preview",
]
network = [
    "retrofit",
    "retrofit-kotlinx-serialization",
    "okhttp",
    "okhttp-logging",
]
```

使用:

```kotlin
dependencies {
    implementation(libs.bundles.compose)
}
```

## 十一、迁移单模块到多模块的步骤

从 NotedX 现状(单模块 `:app`)迁移:

**第一步:抽 `:core:model`**

把所有 Domain Model(`Note`, `Tag`, `UserSettings`)移到 `:core:model`,`:app` 依赖它。验证编译通过。

**第二步:抽 `:core:designsystem` 和 `:core:ui`**

把 `NotedXTheme`, `NotedXTypography`, 颜色定义放 `:core:designsystem`。把可复用 Composable 放 `:core:ui`。

**第三步:抽 `:network` 和 `:database`**

OkHttp / Retrofit 配置移 `:network`,Room Database 与 Dao 接口移 `:database`。

**第四步:抽 `:data:*`**

每个数据领域一个模块:`:data:note`, `:data:user`, `:data:preferences`。Repository 接口 + 实现 + DI 都在这里。

**第五步:抽 `:feature:*`**

按功能拆,每个 feature 一个模块。`HomeViewModel`, `HomeScreen`, 都在 `:feature:home`。

**第六步:Convention Plugin**

抽完一两个模块就立刻引入 Convention Plugin——否则 5 份 build.gradle.kts 同步维护到第六步要发疯。

每一步都验证编译 + 跑一遍 App,**不要一次拆完**。

## 十二、收益评估

多模块的收益取决于规模:

| 模块数 | 全量编译 | 增量编译 | 维护成本 |
| --- | --- | --- | --- |
| 1(单模块) | 1× | 1× | 最低 |
| 5 | 0.7× | 0.4× | 略升 |
| 10 | 0.5× | 0.2× | 明显升 |
| 20+ | 0.4× | 0.1× | 高 |

**收益在编译速度**——只改 `:feature:home`,只重编 `:feature:home` + `:app`(因为 `:app` 依赖它)。其他 `:feature:*` 不编译。

**成本在配置和导航**——加新 feature 要创建模块、配 build.gradle.kts、改路由表、配 DI 路径。

**实操指南**:**不要在项目第一天多模块**。先单模块写到代码量上来、有真实复用诉求,再拆。Now in Android 看着漂亮,但它是 Google sample,不是你的起点。

## 十三、踩坑

**坑 1:`:feature:home` 依赖 `:feature:detail`**。最常见错误。home 跳 detail 应当通过 callback,callback 由 `:app` 实现。

**坑 2:循环依赖**。Gradle 直接编译失败:`:a` 依赖 `:b`,`:b` 又依赖 `:a`。通常原因是公共依赖没抽到第三个模块(`:core:*`)。

**坑 3:`:core:ui` 依赖业务 model**。`fun NoteCard(note: Note)`——耦合到具体业务,无法复用。改成 `fun NoteCard(title: String, preview: String)`。

**坑 4:DI 配置漏 `@InstallIn`**。模块里写了 `@Module` 但漏了 `@InstallIn`,Hilt 不会装载这个 Module,运行时 NPE。

**坑 5:Compose Compiler 没在每个模块 apply**。`:feature:home` 用 Compose 但忘了 apply `kotlin-compose` 插件,编译报"Composable function ... requires the @Composable annotation"。

**坑 6:子模块 `androidTest` 找不到 Application Hilt**。`@HiltAndroidApp` 在 `:app`,子模块测试默认拿不到。需要在 `:feature:home/src/androidTest` 写自己的 HiltTestApplication 或者用 `:core:testing` 共享。

**坑 7:KSP 重复生成 Hilt 代码**。早期版本 KSP + Hilt 多模块有问题——一个 entity 在多个模块 ksp 一次,产生重复代码。Hilt 2.50+ 已修。**保持 Hilt 最新**。

**坑 8:Type-safe Project Accessors 未启用**。`projects.feature.home` 报错"Unresolved reference",检查 `settings.gradle.kts` 是否 `enableFeaturePreview("TYPESAFE_PROJECT_ACCESSORS")`。

**坑 9:模块名带连字符无法用 Type-safe accessor**。`:feature-home` 的 accessor 是 `projects.featureHome`(自动驼峰)。要避免这种命名,用 `:feature:home`(子项目)更清晰。

**坑 10:循环编译 / R 类找不到**。一些三方库的 manifest 资源会"渗透"到多模块,导致依赖图复杂。`android.nonTransitiveRClass=true` 启用非传递 R 类,每个模块只看到自己的资源,大幅减少耦合。`gradle.properties` 加这行,基本是新项目默认。
