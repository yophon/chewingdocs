# 18-通知系统与推送通道

> 一句话导读:本地通知是"应用把一条信息塞进系统通知抽屉",推送是"服务端把一条信息塞进设备的某条系统级长连接";Android 把"长连接"这件事承包给了 Google Play Services,而国内市场把这个承包合同撕了,所以你必须同时理解 NotificationChannel、Notification Trampoline 限制、FCM,以及为什么国内需要重新讨论"推送是谁的责任"。

第 17 篇把"后台任务"分成 WorkManager 与 ForegroundService 两条路径,中间反复出现"通知"。这一篇我们正面打通三件事:本地通知模型(Channel / Importance / Trampoline)、远程推送基本面(FCM 三类消息)、以及绕不过的国内推送渠道现实。这里有一个明确的取舍:**主线讲清"通知如何被系统呈现 + 推送如何抵达进程",FCM 与国内厂商 SDK 的具体集成步骤不展开**。理由有两个:其一,各家 SDK 集成手册每年小改,文章保鲜期 6 个月,价值低;其二,选型与抽象层比集成更值得占用篇幅,因为前者决定后续 3 年的工程债务。

## 1. 机制定位

### 通知是 Android 上最受用户保护的中断模型

Android 的通知不是"我想弹什么就弹什么",它是用户 / 系统 / 应用三方共同维护的契约。Android 8 (API 26) 起强制引入 NotificationChannel,Android 12 (API 31) 起禁止 Notification Trampoline,Android 13 (API 33) 起 `POST_NOTIFICATIONS` 必须运行时申请,每一次收紧都在做同一件事:**让用户对"哪个应用、哪类通知、什么时候打扰我"有更细粒度的控制权,把决定权从应用挪到用户**。

如果你写过 Android 7 之前的代码,可能还记得"一个应用所有通知都是平铺的、没分类、用户要么全屏蔽要么全接受"。这套模型在用户每天收 200 条通知的时代彻底失败:用户对"重要通知"和"营销骚扰"无法区分,最终只能整个 App 关闭通知。Channel 模型把"通知分组"的责任反向交给应用:应用必须为每类通知声明独立 Channel,用户可以单独关掉"营销 Channel"而保留"账单 Channel"。

新手最常见的失控写法,是把所有通知塞进一个名为 "default" 的 Channel,Importance 设成 HIGH,然后疑惑为啥 App 装机量过百万后用户都把通知关了。正确做法是:每一种"通知意图"(账单 / 消息 / 提醒 / 进度 / 静默状态)对应一个 Channel,Importance 按真实重要度声明,用户只关闭他不想要的那一类。

### 推送的两层抽象:传输层与展示层

很多人把"推送"和"通知"混为一谈,实际是两件事:

- **传输层**:服务端如何让一条消息"主动"到达设备,而不是等设备轮询。Android 上的标准答案是 **FCM (Firebase Cloud Messaging)**,它基于 Google Play Services 维护一条设备到 Google 服务器的长连接(MTalk 协议,基于 XMPP / TLS),你的服务端把消息推给 Google,Google 通过这条长连接送达。
- **展示层**:消息抵达进程后如何显示给用户。展示层完全是"本地通知" + 自定义业务逻辑,跟"消息从哪来"无关。FCM Notification Message 是 Google 把展示层也包办了,Data Message 是只送 payload、展示由你写。

把这两层分开理解之后,国内市场的问题就清楚了:**传输层在国内出问题**(Google Play Services 不可达 / 被剥离),展示层(NotificationChannel / NotificationCompat)没问题。所以国内推送的工程化讨论,核心是"换一个 / 多个传输层",而不是重新写通知 UI。

### Android 12+ Notification Trampoline 是什么

Trampoline(蹦床)指的是这种模式:点击通知 → 系统启动你的 BroadcastReceiver 或 Service → 在 receiver / service 里 `startActivity` 启动真正的 Activity。Android 12 起这种"中转启动 Activity"被禁止,理由是它造成感知延迟(用户点了通知半天才出 UI)、且 receiver / service 中转期间应用其实没有"用户可见"上下文,可以做一些他本不该做的事。

补救路径很简单:通知的 PendingIntent 直接用 `PendingIntent.getActivity()` 指向目标 Activity,系统接管启动;如果需要预先处理逻辑(比如鉴权后再决定跳哪),把逻辑放在 Activity onCreate 里,Activity 自己内部跳转。从 BroadcastReceiver 启动 Activity 的所有代码在 Android 12+ 直接被系统忽略,不抛异常,只是 Activity 不会起来,且 logcat 留下 `Indirect notification activity start (trampoline) blocked` 警告。这是 2026 年仍存量很多的隐性 bug 来源。

## 2. Android 心智

### NotificationChannel 与 Importance

```
NotificationManager.IMPORTANCE_NONE     # 不显示
NotificationManager.IMPORTANCE_MIN      # 只在抽屉,不滑出,不响铃
NotificationManager.IMPORTANCE_LOW      # 抽屉 + 不响铃     -> 后台进度、静默状态首选
NotificationManager.IMPORTANCE_DEFAULT  # 抽屉 + 响铃       -> 普通消息
NotificationManager.IMPORTANCE_HIGH     # Heads-up 浮窗 + 响铃 -> 即时消息、提醒
```

Channel 一旦创建,**Importance 与 Sound 不能由应用代码改回更高级别**——只能由用户在系统设置里手动改。所以发布前要审慎,Channel 命名也要稳定:一旦上线就视为"用户可见的设置项",改名会让用户看到一个新的 Channel,旧的 Channel 仍残留在用户系统设置里直到卸载。建议 Channel ID 用 `域.业务` 形式(如 `app.notedx.sync.progress`),便于后续多类型扩展。

### `NotificationCompat` 是必选

`NotificationCompat.Builder` 自动处理跨 API level 的兼容(Android 8 之前没有 Channel、Android 12 之前不需要 type 等),你只用一套调用就能在 minSdk 26 到 35 之间一致工作。直接用 `Notification.Builder` 反而要自己 SDK_INT 分支,在新工程里没有意义。

### POST_NOTIFICATIONS 的运行时申请

Android 13 起的 `POST_NOTIFICATIONS` 是普通 dangerous 权限,完全适用第 16 篇的 PermissionController 模板。需要注意几点:

- 申请时机:**不在 App 启动时弹**,而是在"用户首次启用某类通知相关业务"时弹(打开同步开关、订阅了某个频道、收到第一条消息预备显示前)。
- Android 12 及以下不存在这个权限,代码要 `Build.VERSION.SDK_INT >= 33` 兜底,否则旧机型上 `checkSelfPermission` 永远返回 DENIED,造成误判。
- 拒绝后通知仍能创建,只是不会被系统呈现给用户。你的代码不应该因为拒绝就改变业务逻辑,只是"对外可见度"为 0。

### FCM 的三类消息

FCM payload 顶层有三个互斥(更准确说:语义不同)字段:`notification`、`data`、`fcm_options`:

```
Notification Message:
   {
     "notification": {"title": "新消息", "body": "..."},
     "to": "...token..."
   }

Data Message:
   {
     "data": {"type": "new_note", "noteId": "abc-123"},
     "to": "...token..."
   }

Notification + Data 混合:
   两段都有,App 在后台时系统展示 notification 部分,
   前台时整个 payload 进 onMessageReceived
```

三类消息的行为差异:

| 类型 | App 在前台 | App 在后台 | App 被杀 |
| --- | --- | --- | --- |
| Notification Message | 不自动展示,回调 `onMessageReceived` | 由系统直接展示,App 不收回调 | 由系统直接展示 |
| Data Message | 回调 `onMessageReceived` | 回调 `onMessageReceived`,但 Android 12+ 有 BG 限制 | 唤起进程后回调 |
| 混合 | 回调 `onMessageReceived`,展示由你决定 | 系统展示 notification,data 部分进 `onMessageReceived` | 同上 |

**90% 的业务应该选 Data Message**,理由:展示完全由 App 控制(Channel / Importance / 跳转目标都按业务规则);可以做"静默推送"(到达后只更新本地数据库,不展示);可以做"鉴权再展示"(如已读 / 已删除消息不要再弹)。Notification Message 只在"对展示完全没要求、纯文案推送"时才有便利价值。

### Topic 与设备分组

FCM 提供 Topic 订阅:客户端 `FirebaseMessaging.subscribeToTopic("news-tech")`,服务端不需要知道单个 token,直接对 topic 推送,所有订阅者都收到。Topic 适合"对一群人发同一条":新闻类频道、群组消息广播。Topic 不能用于点对点(因为消息内容对所有订阅者都一样,夹带用户 ID 既无意义又泄露)。Token 推送和 Topic 推送可以并行存在,服务端按场景选择。

### 国内为什么不能直接用 FCM

FCM 依赖 Google Play Services(GMS)在设备上保持一条到 Google 服务器的长连接。两个现实约束让这条链路在国内不可靠:

1. **GMS 在中国大陆不可达**。即便用户手机预装了 GMS(小米、OPPO、vivo 国行机均预装但默认关闭),`mtalk.google.com:5228` 这条 TCP 长连接在大部分网络环境下被阻断或质量极差。
2. **华为 / 荣耀的中国大陆版本完全没有 GMS**。从 Mate 30 起,鸿蒙生态产品全面剥离 Google 服务,FCM 客户端 SDK 都无法初始化。

即使前两条没问题(用户出境 / 国际版手机),还有第三条:

3. **国内 OEM 的后台保活策略远比海外严格**。MIUI、HyperOS、ColorOS、OriginOS 默认禁止非系统应用维持长连接,FCM 进程被杀后系统不会再唤起。你看到的现象是"App 在前台时收推送正常,推后台 5 分钟就再也收不到了"。

这三层叠加,导致国内市场必须用**厂商推送通道**:每家厂商的 ROM 里有一个系统级常驻进程,维护一条到该厂商服务器的长连接,你的 App 通过厂商 SDK 注册到这条长连接上,服务端通过厂商 OpenAPI 推送。系统级进程不会被杀,所以送达率高。代价是你必须分别对接华为 HMS Push、小米 Push、OPPO Push、vivo Push、魅族 Push,每家 API、Topic 模型、消息分类要求都不同。

近年还有第四种选择:**UnifiedPush 协议**(欧洲开源社区主导),把"分发器(distributor)" 抽象为系统级、可替换的组件,理论上一个 App 实现 UnifiedPush client SDK,就能复用任何 distributor(NextPush / FCM / ntfy)。但在国内市场缺乏厂商支持,目前更多是"国外开源应用避开 Google"的方案,工程上仅作选型备选。

### 推送统一抽象

国内 App 的成熟做法是在业务层抽出一个 `PushProvider` 接口,封装"注册 / 取 token / 处理消息 / 反注册"四个动作,运行时根据 ROM 选择具体实现。设计取舍后面工程实现一节给出。

## 3. 工程实现

### 文件:`core/notification/NotificationChannels.kt`

把 Channel ID 与默认配置集中维护,避免散落代码里 Channel ID 拼错。

```kotlin
package app.notedx.core.notification

import android.app.NotificationChannel
import android.app.NotificationManager
import android.content.Context
import android.os.Build
import androidx.core.content.getSystemService

object NotificationChannels {
    const val SYNC_PROGRESS = "app.notedx.sync.progress"
    const val SYNC_RESULT = "app.notedx.sync.result"
    const val PUSH_MESSAGE = "app.notedx.push.message"
    const val VOICE_RECORDING = "app.notedx.voice.recording"

    fun ensureCreated(context: Context) {
        if (Build.VERSION.SDK_INT < Build.VERSION_CODES.O) return
        val nm = context.getSystemService<NotificationManager>() ?: return

        nm.createNotificationChannels(listOf(
            NotificationChannel(
                SYNC_PROGRESS, "同步进度",
                NotificationManager.IMPORTANCE_LOW,
            ).apply {
                description = "后台同步任务的进度通知,不打扰"
                setShowBadge(false)
            },
            NotificationChannel(
                SYNC_RESULT, "同步结果",
                NotificationManager.IMPORTANCE_DEFAULT,
            ).apply {
                description = "同步完成或失败的提示"
            },
            NotificationChannel(
                PUSH_MESSAGE, "消息推送",
                NotificationManager.IMPORTANCE_HIGH,
            ).apply {
                description = "来自服务端的即时消息"
                enableLights(true)
                enableVibration(true)
            },
            NotificationChannel(
                VOICE_RECORDING, "语音录制",
                NotificationManager.IMPORTANCE_LOW,
            ).apply {
                description = "ForegroundService 持续通知"
                setShowBadge(false)
            },
        ))
    }
}
```

在 Application onCreate 里调一次 `NotificationChannels.ensureCreated(this)`,Channel 配置就稳定下来。修改已有 Channel 名称 / Importance 后再调,只生效"未被用户改过"的字段;用户改过的字段以用户设置为准。

### 文件:`feature/notify/NotificationCenter.kt`(应用内通知入口)

```kotlin
@Singleton
class NotificationCenter @Inject constructor(
    @ApplicationContext private val context: Context,
) {
    private val nm = NotificationManagerCompat.from(context)

    fun postSyncProgress(progressPercent: Int) {
        if (!nm.areNotificationsEnabled()) return
        val n = NotificationCompat.Builder(context, NotificationChannels.SYNC_PROGRESS)
            .setSmallIcon(R.drawable.ic_sync)
            .setContentTitle("正在同步")
            .setProgress(100, progressPercent, false)
            .setOngoing(true)
            .setOnlyAlertOnce(true)
            .build()
        nm.notify(NOTIF_SYNC_PROGRESS, n)
    }

    fun postNewMessage(title: String, body: String, noteId: String) {
        if (!nm.areNotificationsEnabled()) return

        // Trampoline 限制:PendingIntent 直接指向 Activity
        val intent = Intent(context, MainActivity::class.java).apply {
            flags = Intent.FLAG_ACTIVITY_NEW_TASK or Intent.FLAG_ACTIVITY_CLEAR_TOP
            putExtra(EXTRA_TARGET_NOTE_ID, noteId)
            data = Uri.parse("notedx://note/$noteId")  // 配合深链,18-20 篇会接上
        }
        val pending = PendingIntent.getActivity(
            context, noteId.hashCode(), intent,
            PendingIntent.FLAG_IMMUTABLE or PendingIntent.FLAG_UPDATE_CURRENT,
        )

        val n = NotificationCompat.Builder(context, NotificationChannels.PUSH_MESSAGE)
            .setSmallIcon(R.drawable.ic_message)
            .setContentTitle(title)
            .setContentText(body)
            .setStyle(NotificationCompat.BigTextStyle().bigText(body))
            .setContentIntent(pending)
            .setAutoCancel(true)
            .setPriority(NotificationCompat.PRIORITY_HIGH)
            .build()
        nm.notify(NOTIF_PUSH_MESSAGE_BASE + noteId.hashCode(), n)
    }

    fun cancelSyncProgress() = nm.cancel(NOTIF_SYNC_PROGRESS)

    companion object {
        const val EXTRA_TARGET_NOTE_ID = "target_note_id"
        private const val NOTIF_SYNC_PROGRESS = 100
        private const val NOTIF_PUSH_MESSAGE_BASE = 1000
    }
}
```

几处工程取舍:

- `nm.areNotificationsEnabled()` 在发通知前查一次,用户禁用通知时不浪费 CPU 构造 Notification。
- `PendingIntent.FLAG_IMMUTABLE` 是 Android 12+ 的强制要求,不设会 throw `IllegalArgumentException`。
- `setOnlyAlertOnce(true)` 用在进度通知上,避免每次 update 都响铃。
- ID 命名空间:进度类用单一 ID(覆盖更新);消息类用 hash 做 ID(每条独立)。

### 文件:`push/fcm/AppFcmService.kt`(FCM 接入示例)

只有当 App 集成了 `firebase-messaging`、且设备有 GMS 时,这个 Service 才会被回调。manifest 注册:

```xml
<service
    android:name=".push.fcm.AppFcmService"
    android:exported="false">
    <intent-filter>
        <action android:name="com.google.firebase.MESSAGING_EVENT" />
    </intent-filter>
</service>
```

代码:

```kotlin
@AndroidEntryPoint
class AppFcmService : FirebaseMessagingService() {

    @Inject lateinit var notificationCenter: NotificationCenter
    @Inject lateinit var pushRegistry: PushRegistry

    override fun onMessageReceived(message: RemoteMessage) {
        // 优先看 data,避免 notification 自动展示
        val data = message.data
        when (data["type"]) {
            "new_note" -> {
                val noteId = data["noteId"] ?: return
                val title = data["title"] ?: "新笔记"
                notificationCenter.postNewMessage(title, data["body"].orEmpty(), noteId)
            }
            "silent_sync" -> {
                // 静默推送,不展示通知,只触发同步
                SyncScheduler.kickFromPush(applicationContext)
            }
        }
    }

    override fun onNewToken(token: String) {
        // 上报给服务端,服务端按 channel 分发
        pushRegistry.uploadToken(token, channel = PushChannel.FCM)
    }
}
```

注意:`onMessageReceived` 必须在 20 秒内返回,否则进程被杀,后续消息丢。耗时操作(网络上报、批量入库)走 `enqueue` WorkManager,不要在这里 `runBlocking`。

### 文件:`push/PushProvider.kt`(国内推送的统一抽象)

各厂商 SDK 的 API 形态差异很大,但你的业务只关心四件事:**初始化 / 取 token / 收消息 / 反注册**。抽出接口:

```kotlin
enum class PushChannel { FCM, HMS, MI, OPPO, VIVO, MEIZU, UNIFIED }

data class PushMessage(
    val title: String?,
    val body: String?,
    val data: Map<String, String>,
)

interface PushProvider {
    val channel: PushChannel
    fun isAvailable(context: Context): Boolean
    suspend fun register(context: Context): String   // 返回 token
    suspend fun unregister(context: Context)
    fun setListener(listener: PushListener)
}

interface PushListener {
    fun onMessage(message: PushMessage)
    fun onTokenChanged(token: String)
}
```

每家 SDK 写一个 PushProvider 实现,文件归属:

```
push/
├── PushProvider.kt
├── PushChannelSelector.kt       # 根据 ROM 选实现
├── PushRegistry.kt              # token 上报
├── fcm/
│   ├── FcmPushProvider.kt
│   └── AppFcmService.kt
├── hms/
│   ├── HmsPushProvider.kt
│   └── AppHmsService.kt
├── mi/
│   └── MiPushProvider.kt
└── ...
```

`PushChannelSelector` 在 Application 初始化时按厂商挑实现:

```kotlin
@Singleton
class PushChannelSelector @Inject constructor(
    private val providers: Map<PushChannel, @JvmSuppressWildcards PushProvider>,
) {
    fun pick(context: Context): PushProvider {
        return when {
            Build.MANUFACTURER.equals("HUAWEI", ignoreCase = true) ||
            Build.MANUFACTURER.equals("HONOR", ignoreCase = true) ->
                providers.getOrElse(PushChannel.HMS) { providers.getValue(PushChannel.UNIFIED) }
            Build.MANUFACTURER.equals("Xiaomi", ignoreCase = true) ->
                providers.getValue(PushChannel.MI)
            Build.MANUFACTURER.equals("OPPO", ignoreCase = true) ||
            Build.MANUFACTURER.equals("realme", ignoreCase = true) ->
                providers.getValue(PushChannel.OPPO)
            Build.MANUFACTURER.equals("vivo", ignoreCase = true) ->
                providers.getValue(PushChannel.VIVO)
            else -> providers[PushChannel.FCM]
                ?.takeIf { it.isAvailable(context) }
                ?: providers.getValue(PushChannel.UNIFIED)
        }
    }
}
```

Hilt 用 `@IntoMap` 把各 Provider 注入到 Map 中:

```kotlin
@Module
@InstallIn(SingletonComponent::class)
abstract class PushModule {
    @Binds @IntoMap @PushChannelKey(PushChannel.FCM)
    abstract fun bindFcm(impl: FcmPushProvider): PushProvider

    @Binds @IntoMap @PushChannelKey(PushChannel.HMS)
    abstract fun bindHms(impl: HmsPushProvider): PushProvider

    // ... 其他厂商同理
}

@MapKey
annotation class PushChannelKey(val value: PushChannel)
```

业务方就只看到:

```kotlin
@HiltAndroidApp
class NotedXApp : Application(), Configuration.Provider {
    @Inject lateinit var pushSelector: PushChannelSelector
    @Inject lateinit var pushListener: AppPushListener

    override fun onCreate() {
        super.onCreate()
        NotificationChannels.ensureCreated(this)

        val provider = pushSelector.pick(this)
        provider.setListener(pushListener)
        ProcessLifecycleOwner.get().lifecycleScope.launch {
            val token = provider.register(this@NotedXApp)
            pushRegistry.uploadToken(token, provider.channel)
        }
    }
}
```

**服务端**收到 token 上报时记录 token + channel,推送时按 channel 走对应渠道。各厂商 API 不同,服务端通常封装一个 PushDispatcher,按 channel 调用对应渠道的 OpenAPI。这是后端工程,本系列不展开,但客户端的 channel 标记是必须的。

### 文件:`push/fcm/PushPermissionGate.kt`(POST_NOTIFICATIONS 通用入口)

无论是 FCM 还是国内推送,Android 13+ 必须有 `POST_NOTIFICATIONS`:

```kotlin
@Composable
fun PushPermissionGate(autoRequest: Boolean = false) {
    if (Build.VERSION.SDK_INT < Build.VERSION_CODES.TIRAMISU) return
    val controller = rememberPermissionController(
        android.Manifest.permission.POST_NOTIFICATIONS,
    )
    LaunchedEffect(autoRequest, controller.state) {
        if (autoRequest && controller.state is PermissionState.NotRequested) {
            controller.request()
        }
    }
    // 通常不需要任何可见 UI,只在"用户首次打开消息推送开关"时调一次
}
```

## 4. 调参与验收

### 通知验收

```bash
# 看你的 Channel 列表与配置
adb shell dumpsys notification | grep -A 5 app.notedx

# 强制让 App 在后台发一条测试通知(配合自定义 Intent)
adb shell am broadcast -a app.notedx.TEST_NOTIFICATION

# 看当前所有可见通知
adb shell dumpsys notification --noredact
```

`--noredact` 在 13+ 设备需要 root 或 secure adb,否则部分字段被遮码。

### FCM 验收

Google 提供 `firebase-tools` CLI,可以直接从命令行向特定 token 推一条测试消息:

```bash
firebase messaging:send --token "<your-test-token>" \
    --notification-title "测试" --notification-body "Hello"
```

抓 `FirebaseMessaging` 的 logcat:

```bash
adb logcat -s "FirebaseMessaging" "FA" "FA-SVC"
```

正常情况能看到 token 注册、消息接收、`onMessageReceived` 触发的日志链。

### 国内推送渠道验收

每家厂商有自己的"推送测试控制台",但前置门槛是开发者账号实名认证 + App 包名备案。开发期可以先用 logcat 看 SDK 注册结果:

```bash
adb logcat -s "HMSCore" "MiPush" "OPushService" "vPushService"
```

各家 token 长度、命名空间不同(HMS 是十几字符短串,小米是 60+ 长串,FCM 是 140+ 字符长串),抓 token 上报日志能区分。

### 通知 Trampoline 自检

把 logcat 过滤一下:

```bash
adb logcat | grep "trampoline\|Indirect notification"
```

如果出现 `Indirect notification activity start (trampoline) blocked`,说明你某个通知的 PendingIntent 指向了 BroadcastReceiver / Service 然后中转 Activity,Android 12+ 已经把这条路堵死。补救:把目标 PendingIntent 改为 `getActivity()` 直接指向 Activity。

### 验收清单

- [ ] 所有 Channel 在 Application onCreate 创建,系统设置里能看到独立条目,名称和 Importance 与代码一致。
- [ ] 拒绝 `POST_NOTIFICATIONS` 后,所有 `nm.notify()` 调用静默无效,业务流程不崩。
- [ ] 点通知后 Activity 直接弹起,无 Trampoline 警告。
- [ ] FCM token 在首次启动后 5 秒内拿到并上报。
- [ ] 静默推送(Data Message)能更新本地数据库且不显示通知。
- [ ] 国内厂商真机(华为 / 小米 / OPPO / vivo)上,PushChannelSelector 选中正确的 PushProvider,推送送达率与 logcat 一致。
- [ ] Android 12 真机上,锁屏后收到 Heads-up 通知,点击直达详情页;Android 14 上同样路径,无 BG-to-FG 启动限制阻塞。

## 5. 踩坑

**坑 1:Channel 创建后改 Importance 没生效。** Android 8+ 起,Channel Importance 由用户控制,应用代码只能在创建时设初值,后续 `setImportance()` 调用对已有 Channel 无效。要"改"只能新建一个 ID 不同的 Channel,旧的留着或在文档里告诉用户去关。所以发布前要审 Channel 命名和 Importance,上线后基本是"承诺"。

**坑 2:用 `Notification.Builder` 而不是 `NotificationCompat.Builder`。** 前者要你自己分支 SDK_INT,后者全自动。新工程一律用 `NotificationCompat`,直接从 `androidx.core` 引。

**坑 3:Trampoline 路径在 Android 11 测试通过,Android 12 上线后通知点了没反应。** Android 11 还允许 BroadcastReceiver 中转启动 Activity,Android 12 起拒绝。修复:`PendingIntent.getActivity()` 直接指向 Activity,业务前置逻辑放到 Activity onCreate / ViewModel 里。

**坑 4:PendingIntent 不加 `FLAG_IMMUTABLE`。** Android 12+ 起,创建 PendingIntent 必须显式 `FLAG_IMMUTABLE` 或 `FLAG_MUTABLE`(后者仅在"系统需要修改 PendingIntent extras"时用,通常不需要)。漏 flag 直接 `IllegalArgumentException`。

**坑 5:FCM 用 Notification Message,期望进程做点逻辑。** App 在后台时 Notification Message 由系统直接展示,不回调 `onMessageReceived`,你写的 "更新本地未读数 + 写日志" 全部不执行。修复:用 Data Message 或 Notification + Data 混合,服务端 payload 不要只塞 notification 字段。

**坑 6:`onMessageReceived` 里跑长任务。** 这个方法限 20 秒,超时进程被 kill 且后续消息掉。修复:简单逻辑(展示通知、更新一两个字段)就地做;复杂逻辑 enqueue 一个 expedited OneTimeWorkRequest 给 WorkManager(见 17 篇)。

**坑 7:静默推送在 Android 12+ 后台时 enqueueWork 失败。** Data Message 在 Android 12+ 后台时受 BG-to-FG 限制,你想"收到推送后起一个 ForegroundService"基本不行。正确路径:`onMessageReceived` 里只 enqueue WorkManager 任务(WorkManager 有 BG-to-FG 豁免),不要 `startForegroundService`。

**坑 8:国内手机依赖 FCM,送达率几乎为 0。** 设计阶段就要面对国内市场,不能上线后被反馈"通知不到"才回头加厂商通道。**最低工程量做法:海外用 FCM,国内厂商用对应通道,二选一基于 ROM**;**完整做法:抽象出 PushProvider 接口,服务端按 token channel 分发**。如果完全不考虑国内,在 README / Privacy Policy 标注"仅支持国际版本",别留给用户预期。

**坑 9:厂商 SDK 在 Application onCreate 同步初始化导致启动慢。** HMS / 小米 / OPPO 等 SDK 的初始化都有网络 I/O,放主线程同步调用会让冷启动多几百毫秒。修复:用协程 `launch(Dispatchers.IO)` 异步初始化,或者用 `androidx.startup` 的 Initializer 把它移到非主线程的初始化路径。

**坑 10:Token 上报失败但 retry 没做。** 厂商 token 在 App 卸载重装、清数据、SDK 升级、用户系统设置重置时可能变化,SDK 通过 `onNewToken` 回调上报,但回调时机不保证(用户可能离线)。修复:每次 App 启动都查一次最新 token,如果与上次上报值不同,重新上报;服务端 token 表记录"最后心跳时间",超期 token 标记为"可能失效",发送前先尝试。

**坑 11:多个推送通道同时在线,同一条消息被推两次。** 你同时初始化了 FCM 和 HMS,服务端不知道客户端"用哪个",两边都推。修复:**客户端启动时只选一个 Provider 并上报 channel**,服务端按 channel 分发,不要"广撒网"。

**坑 12:通知没设 `setAutoCancel(true)`,用户点了通知抽屉里还在。** 默认通知点了不消失,看起来像没响应。除非是"持续状态"(ForegroundService 进度),都该加 `setAutoCancel(true)`。

**坑 13:Notification ID 用了 `Random.nextInt()`。** 后续无法 `cancel`,通知抽屉会被你的进度通知堆满。ID 应该按业务可寻址(进度类固定 ID,消息类用 hash),需要批量取消时通过 `nm.activeNotifications` 遍历。

**坑 14:Big Text / Big Picture / Inbox style 没设。** 默认通知只显示一行 body,长文本被截。修复:`setStyle(NotificationCompat.BigTextStyle().bigText(longBody))` / `BigPictureStyle()` / `InboxStyle()`,Android 7+ 支持。

**坑 15:在 BootReceiver 里恢复推送注册,但 Android 13+ 权限未给。** 设备开机后 BootReceiver 触发,你重新初始化 SDK 注册 token,但 `POST_NOTIFICATIONS` 未给,推送来了也展示不出来。修复:开机后 token 注册 / 状态同步可以做,但 UI 通知展示要先看 `nm.areNotificationsEnabled()`,且引导申请权限放在用户下次打开 App 时。

**坑 16:把 `RECEIVE_BOOT_COMPLETED` 当成"启动入口"。** Android 8+ 起,即使收到 BOOT_COMPLETED,后台启动 Activity 严格受限,推送通道注册需要走 ForegroundService 或 WorkManager。BOOT_COMPLETED 只能做"轻量初始化"(写一个 DataStore 标记、enqueue 一个 Work),不要在它里面起 Activity 或长跑 Service。

**坑 17:Channel 改名造成用户系统设置里出现"幽灵 Channel"。** Channel 一旦创建就在用户系统设置里持久化,删除 Channel 的代码(`deleteNotificationChannel`)不会触发 UI 更新,且用户对这个 Channel 的设置依然保留(对未来同 ID 创建仍生效)。补救:命名空间稳定,如果业务变化非常大,只能"启用新 ID + 留老 ID 不再使用"的方式过渡。

**坑 18:`MIUI` / `HyperOS` 上不显示横幅。** 即使你 Channel Importance 是 HIGH,小米系默认对非系统应用的横幅通知有额外限制,需要用户在"通知管理 → App → 横幅通知"里手动开。修复:无法靠 API 解决,可以在用户首次启用消息推送时引导他去厂商通知设置。这是国内 ROM 长期定制带来的工程现实。

**坑 19:`UnifiedPush` 在国内市场基本无 distributor 可用。** UnifiedPush 协议本身设计干净,但缺乏国内厂商支持,目前主要用于"去 Google 化"的国外开源应用。如果你的目标用户中有大量"自托管 / 隐私敏感"用户,可以作为可选 PushProvider,但不能作为主线方案。

**坑 20:FCM 在国际版本 App 里也"看起来送达不稳"。** 很多人忽略:FCM 的送达时机有"高优先级"(`priority: high`)和"普通优先级"(`priority: normal`)区分。普通优先级在设备 Doze 期会被推迟到下一个维护窗口(可能延迟数十分钟)。修复:服务端 payload 加 `"priority": "high"`,客户端配合用 Data Message 中的 `time_sensitive` 字段。但高优先级有 Google 的频率限制,不要拿来推送营销,否则项目会被限速。
