# 权限模型与运行时申请

> 一句话:**Android 把权限分成"安装时给"和"运行时申请"两类——危险权限必须运行时弹窗问用户**。Compose 里用 `rememberLauncherForActivityResult` + `RequestPermission` 契约处理。这是后面相机 / 通知 / 后台都要先过的关。

---

## 一、Android 权限的两种世界

| 权限类型 | 例子 | 什么时候给 |
| --- | --- | --- |
| **Normal**(普通) | 网络、振动、唤醒锁 | manifest 声明即给,不弹窗 |
| **Dangerous**(危险) | 相机、相册、位置、通讯录、通知(API 33+) | **必须运行时申请**,用户可拒可撤回 |
| **Signature**(签名) | 仅同签名 App 互访 | 极少用 |
| **Special**(特殊) | 悬浮窗、所有文件访问、电池白名单 | 跳设置页让用户开 |

危险权限是这一篇的主角。**用户可以随时在设置里撤销**,所以你的代码必须每次使用前检查。

---

## 二、权限清单(常用)

```
android.permission.CAMERA                       危险:相机
android.permission.RECORD_AUDIO                 危险:录音
android.permission.POST_NOTIFICATIONS           危险(API 33+):发通知
android.permission.READ_MEDIA_IMAGES            危险(API 33+):读图片
android.permission.READ_MEDIA_VIDEO             危险(API 33+):读视频
android.permission.ACCESS_FINE_LOCATION         危险:精确定位
android.permission.ACCESS_COARSE_LOCATION       危险:粗略定位
android.permission.READ_CONTACTS                危险:通讯录
android.permission.INTERNET                     普通:网络
android.permission.ACCESS_NETWORK_STATE         普通:网络状态
android.permission.VIBRATE                      普通:振动
android.permission.WAKE_LOCK                    普通:唤醒锁
android.permission.FOREGROUND_SERVICE           普通+特殊:前台服务(16 篇)
```

**API Level 重要变化**:

- API 30(Android 11):分区存储强制,`READ_EXTERNAL_STORAGE` 大部分场景不再需要,改用 Photo Picker 等无权限 API
- API 33(Android 13):新加 `POST_NOTIFICATIONS`(发通知前必问);媒体读取拆成 `READ_MEDIA_IMAGES` / `READ_MEDIA_VIDEO` / `READ_MEDIA_AUDIO`
- API 34(Android 14):前台服务必须声明 type;`READ_MEDIA_VISUAL_USER_SELECTED` 让用户选部分照片
- API 35(Android 15):Photo Picker 进一步推广;部分权限收紧

## 三、manifest 声明

每个用到的权限必须在 `AndroidManifest.xml` 声明,**不管是普通还是危险**:

```xml
<uses-permission android:name="android.permission.INTERNET" />
<uses-permission android:name="android.permission.CAMERA" />
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
<uses-permission android:name="android.permission.READ_MEDIA_IMAGES" />
```

**只在某些 SDK 需要的权限用 `maxSdkVersion`**:

```xml
<uses-permission
    android:name="android.permission.READ_EXTERNAL_STORAGE"
    android:maxSdkVersion="32" />        <!-- API 33+ 用 READ_MEDIA_*,不需要这个 -->
```

## 四、运行时申请的标准模式

Compose 里申请权限用 `rememberLauncherForActivityResult` + `RequestPermission` 契约:

```kotlin
@Composable
fun CameraButton(onPermissionGranted: () -> Unit) {
    val ctx = LocalContext.current
    val launcher = rememberLauncherForActivityResult(
        contract = ActivityResultContracts.RequestPermission()
    ) { isGranted ->
        if (isGranted) onPermissionGranted()
    }

    Button(onClick = {
        when {
            ContextCompat.checkSelfPermission(ctx, Manifest.permission.CAMERA) == PackageManager.PERMISSION_GRANTED -> {
                onPermissionGranted()                              // 已经有权限,直接用
            }
            else -> launcher.launch(Manifest.permission.CAMERA)    // 没权限,弹窗申请
        }
    }) { Text("拍照") }
}
```

**核心两件事**:

1. **每次使用前检查权限**——用户可能在设置里撤销
2. **没权限时启动申请**——`launcher.launch(...)`,结果在 callback 里

## 五、多个权限一起申请

val launcher = rememberLauncherForActivityResult(

launcher.launch(arrayOf(
    Manifest.permission.CAMERA,
    Manifest.permission.RECORD_AUDIO,

## 六、"用户已经拒绝过"的处理

第一次申请,用户拒——可以再问。第二次拒并勾"不再询问",系统不再弹窗。你需要识别这种情况,给个**引导跳转到设置**:

```kotlin
val activity = LocalContext.current as Activity
val shouldShowRationale = ActivityCompat.shouldShowRequestPermissionRationale(
    activity, Manifest.permission.CAMERA
)

when {
    hasPermission -> useFeature()
    shouldShowRationale -> showRationaleDialog()                  // 解释为什么需要
    permissionPermanentlyDenied -> openAppSettings(ctx)            // 跳设置页
    else -> launcher.launch(Manifest.permission.CAMERA)
}
```

**`shouldShowRequestPermissionRationale` 的语义有点别扭**:
- 第一次申请前:返回 false
- 用户拒绝一次:返回 true(应当显示理由解释)
- 用户勾"不再询问":返回 false(此时不能再 launch,launch 会瞬间返回 denied,要直接跳设置)

**最佳实践**:

```kotlin
// 状态机
enum class PermissionState { Unknown, Granted, Denied, PermanentlyDenied }

fun determineState(
    ctx: Context,
    activity: Activity,
    permission: String,
    hasRequestedOnce: Boolean,
): PermissionState = when {
    ContextCompat.checkSelfPermission(ctx, permission) == PackageManager.PERMISSION_GRANTED ->
        PermissionState.Granted
    !hasRequestedOnce -> PermissionState.Unknown
    ActivityCompat.shouldShowRequestPermissionRationale(activity, permission) ->
        PermissionState.Denied                                     // 还能再问
    else -> PermissionState.PermanentlyDenied                      // 已被永久拒绝,只能跳设置
}

fun openAppSettings(ctx: Context) {
    val intent = Intent(Settings.ACTION_APPLICATION_DETAILS_SETTINGS).apply {
        data = Uri.fromParts("package", ctx.packageName, null)
        addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
    }
    ctx.startActivity(intent)
}
```

`hasRequestedOnce` 用 DataStore 持久化——`shouldShowRationale` 在"从未申请过"和"永久拒绝"两种情况都返回 false,你需要自己记一个"已经申请过一次"。

## 七、Accompanist Permissions(可选)

`com.google.accompanist:accompanist-permissions` 给 Compose 提供更高层的封装:

```kotlin
val cameraPermissionState = rememberPermissionState(Manifest.permission.CAMERA)

when {
    cameraPermissionState.status.isGranted -> CameraContent()
    cameraPermissionState.status.shouldShowRationale -> RationaleUi(
        onConfirm = { cameraPermissionState.launchPermissionRequest() }
    )
    else -> Button(onClick = { cameraPermissionState.launchPermissionRequest() }) {
        Text("授权相机")
    }
}
```

**Accompanist 历史**:Google 把 Compose 一些实验性 API 先放 Accompanist,稳定后合入 Compose 主线。`permissions` 模块还没合入,但已经稳定多年。**项目用它能省不少样板代码**。

## 八、通知权限(API 33+)

Android 13 起,**发任何通知前必须先申请 `POST_NOTIFICATIONS`**:

```xml
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

```kotlin
val launcher = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.RequestPermission()
) { granted -> /* ... */ }

LaunchedEffect(Unit) {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
        if (ContextCompat.checkSelfPermission(ctx, Manifest.permission.POST_NOTIFICATIONS) != PackageManager.PERMISSION_GRANTED) {
            launcher.launch(Manifest.permission.POST_NOTIFICATIONS)
        }
    }
}
```

**何时申请**:**不要一打开 App 就弹**——用户没有上下文。在用户**首次需要通知功能时**申请:订阅推送、保存提醒、开启同步通知等业务动作之前。

17 篇会展开通知的完整链路。

## 九、媒体权限(API 33+)

Android 13 起,**`READ_EXTERNAL_STORAGE` 拆成三个粒度更细的权限**:

```xml
<uses-permission android:name="android.permission.READ_MEDIA_IMAGES" />
<uses-permission android:name="android.permission.READ_MEDIA_VIDEO" />
<uses-permission android:name="android.permission.READ_MEDIA_AUDIO" />
```

但**首选不申请权限**——用 **Photo Picker**:

```kotlin
val picker = rememberLauncherForActivityResult(
    contract = ActivityResultContracts.PickVisualMedia()
) { uri -> /* 拿到 uri */ }

picker.launch(PickVisualMediaRequest(ActivityResultContracts.PickVisualMedia.ImageOnly))
```

Photo Picker 是**无权限**的——用户在系统选择器里挑哪张就给哪张,App 拿到的 URI 仅对那张图有效。**这是 Android 14+ 推荐路径**。18 篇展开。

## 十、位置权限的两步流程

位置权限被分成两类:

```xml
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_BACKGROUND_LOCATION" />   <!-- 后台,极敏感 -->
```

**API 31+ 用户可以只给"粗略位置"**——你以为申请了 FINE,实际可能只拿到 COARSE。要检测:

```kotlin
val fine = checkSelfPermission(ctx, ACCESS_FINE_LOCATION) == PERMISSION_GRANTED
val coarse = checkSelfPermission(ctx, ACCESS_COARSE_LOCATION) == PERMISSION_GRANTED
// 根据结果选择高精度还是城市级 API
```

**后台位置**是最敏感的——必须先获得前台位置,再单独申请后台。一般 App 不应碰它。

NotedX 不用位置,这里仅作模式说明。

## 十一、特殊权限:不通过 RequestPermission 申请

少数权限**没有标准弹窗**,只能跳设置页让用户开:

| 权限 | 干嘛的 | 跳哪 |
| --- | --- | --- |
| `SYSTEM_ALERT_WINDOW` | 悬浮窗 | `Settings.ACTION_MANAGE_OVERLAY_PERMISSION` |
| `WRITE_SETTINGS` | 改系统设置 | `Settings.ACTION_MANAGE_WRITE_SETTINGS` |
| `MANAGE_EXTERNAL_STORAGE` | 所有文件访问(极少应用允许) | `Settings.ACTION_MANAGE_ALL_FILES_ACCESS_PERMISSION` |
| `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` | 电池白名单 | `Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` |
| Notification listener | 监听其他 App 通知 | `Settings.ACTION_NOTIFICATION_LISTENER_SETTINGS` |

代码:

```kotlin
val intent = Intent(Settings.ACTION_MANAGE_OVERLAY_PERMISSION, Uri.parse("package:${ctx.packageName}"))
ctx.startActivity(intent)
// 用户授权后回到 App,通常用 onResume 重新检查
```

**这些权限通常会让 App 审核被卡**——Play Store 对 `MANAGE_EXTERNAL_STORAGE` / `SYSTEM_ALERT_WINDOW` 都有严格审查。**默认不要碰**,真的需要做好准备走商店审核。

## 十二、NotedX 的权限使用清单

| 权限 | 什么时候问 | 失败兜底 |
| --- | --- | --- |
| `INTERNET` | manifest 给(普通权限) | - |
| `POST_NOTIFICATIONS` | 用户首次启用"同步提醒"功能时 | 提醒功能禁用,提示去设置开启 |
| `CAMERA` | 用户点"拍照添加附件"时 | 给"从相册选"作为兜底 |
| 无 `READ_MEDIA_*` | 用 Photo Picker 替代 | - |
| 无 `ACCESS_*_LOCATION` | NotedX 不用位置 | - |
| 无后台权限 | WorkManager 已覆盖后台同步 | - |

**整个 App 真正需要运行时申请的权限只有两个:通知和相机**。这是好设计——权限申请越少,用户接受度越高,审核压力越小。

## 十三、可复用的权限申请 Composable

```kotlin
@Composable
fun rememberPermissionLauncher(
    permission: String,
    onResult: (granted: Boolean) -> Unit,
): () -> Unit {
    val ctx = LocalContext.current
    val launcher = rememberLauncherForActivityResult(
        contract = ActivityResultContracts.RequestPermission(),
        onResult = onResult,
    )

    return {
        if (ContextCompat.checkSelfPermission(ctx, permission) == PackageManager.PERMISSION_GRANTED) {
            onResult(true)
        } else {
            launcher.launch(permission)
        }
    }
}

// 使用
val requestCamera = rememberPermissionLauncher(Manifest.permission.CAMERA) { granted ->
    if (granted) openCamera() else showRationale()
}
Button(onClick = requestCamera) { Text("拍照") }
```

把"已有权限直接调用 / 没有就弹窗"这个常见模式封装一次。

## 十四、踩坑

**坑 1:申请前不在 manifest 声明权限**。`launcher.launch("android.permission.X")` 会瞬间返回 denied,因为系统看到 manifest 没声明。

**坑 2:在 App 启动就弹一堆权限**。用户上下文为零,直接选"全部拒绝"。**永远在用户首次需要那个功能时申请**,带文字解释清楚为什么需要。

**坑 3:`shouldShowRationale` 当布尔值用**。它在"从未申请过"和"永久拒绝"都是 false——单看它判断不了状态,必须配合"是否申请过一次"的本地标记。

**坑 4:权限拒绝就崩**。拒绝是合法响应。每个使用权限的功能必须有"无权限时怎么办"的兜底——拍照换成"从相册选"、通知关掉、位置改为手输。

**坑 5:`POST_NOTIFICATIONS` 在 API 32 及以下手动检查**。API 33 以下没有这个权限,代码不要写成 `if (checkSelfPermission(POST_NOTIFICATIONS) ...)`——会编译期 warning。包一层 SDK 检查:`if (Build.VERSION.SDK_INT >= TIRAMISU) { ... }`。

**坑 6:用过时的 `Activity.requestPermissions()`**。新代码用 `rememberLauncherForActivityResult` + ActivityResultContracts。前者是命令式 API,在 Compose 里反模式。

**坑 7:权限被撤销后 App 崩溃**。用户在系统设置撤销权限,**Android 会重启 App 进程**——你以为之前拿到了权限,代码以为还有,实际已经没了。**每次使用前必须 `checkSelfPermission`**。

**坑 8:无差别申请 `MANAGE_EXTERNAL_STORAGE`**。这权限会让 Play 审核打回——除非你真是文件管理类 App。99% 场景用 Photo Picker / Scoped Storage / 应用私有目录。

**坑 9:把权限申请逻辑分散在每个屏幕**。统一封装可复用 Composable,**所有屏幕用同一套申请 + 引导 + 设置页跳转流程**,体验一致。
