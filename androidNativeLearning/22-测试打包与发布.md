# 测试、打包与发布

> 一句话:**写测试就是给"重构能不能不破现有功能"画一道安全网,打包发布就是把代码送到用户手里**。NotedX 走到这一篇,可以签名上架。

---

## 一、Android 测试金字塔

```
              ┌──────────────────┐
              │   端到端 UI 测试   │   慢(秒)、覆盖广、脆弱
              │   (10-20 个)     │
              └──────────────────┘
            ┌──────────────────────┐
            │ Compose UI 单测       │   中速(百 ms)、单屏测试
            │ (50-100 个)         │
            └──────────────────────┘
        ┌────────────────────────────┐
        │ ViewModel / Repository 单测  │   快(ms)、纯逻辑
        │ (200+ 个)                  │
        └────────────────────────────┘
```

**写测试的优先级反过来**:先 ViewModel 单测(投入产出比最高),再 Compose UI 单测,最后端到端。

## 二、ViewModel 单元测试

```kotlin
@ExtendWith(MainDispatcherExtension::class)
class HomeViewModelTest {

    private val fakeRepo = FakeNoteRepository()
    private lateinit var vm: HomeViewModel

    @BeforeEach
    fun setup() {
        vm = HomeViewModel(fakeRepo)
    }

    @Test
    fun `初始状态为 Loading`() = runTest {
        assertEquals(NoteListState.Loading, vm.uiState.value.listState)
    }

    @Test
    fun `加载完成后显示笔记列表`() = runTest {
        fakeRepo.emit(listOf(Note(1, "Test", "...")))
        val state = vm.uiState.dropWhile { it.listState is NoteListState.Loading }.first()
        assertTrue(state.listState is NoteListState.Content)
        assertEquals(1, (state.listState as NoteListState.Content).notes.size)
    }
}
```

**关键**:

1. **ViewModel 测试是纯 JVM 单元测试**——不启 Android Runtime,跑得快
2. **依赖用 fake**,不要 mockk 整个 Repository
3. **`runTest`** 是 kotlinx-coroutines-test 提供的协程测试 scope
4. **`MainDispatcherExtension`** 把 `Dispatchers.Main` 换成测试调度器

`FakeNoteRepository`:

```kotlin
class FakeNoteRepository : NoteRepository {
    private val _notes = MutableStateFlow<List<Note>>(emptyList())
    
    override fun observeAll(): Flow<List<Note>> = _notes
    override suspend fun refresh() { /* no-op */ }
    
    fun emit(notes: List<Note>) { _notes.value = notes }
}
```

**Fake > Mock**。Fake 是手写的简单实现,行为清晰、稳定;Mock(`mockk`)在多协程并发下容易出诡异问题,且写多了让测试和实现耦合。

`MainDispatcherExtension`:

```kotlin
class MainDispatcherExtension : BeforeEachCallback, AfterEachCallback {
    private val dispatcher = StandardTestDispatcher()
    override fun beforeEach(ctx: ExtensionContext) = Dispatchers.setMain(dispatcher)
    override fun afterEach(ctx: ExtensionContext) = Dispatchers.resetMain()
}
```

## 三、Compose UI 单测

```kotlin
@RunWith(AndroidJUnit4::class)
class HomeScreenTest {

    @get:Rule val composeRule = createComposeRule()

    @Test
    fun displaysNotesList() {
        composeRule.setContent {
            NotedXTheme {
                HomeScreen(
                    state = HomeUiState(
                        listState = NoteListState.Content(
                            persistentListOf(NoteCard(1, "Hello", "world", "今天"))
                        )
                    ),
                    onRefresh = {},
                    onNoteClick = {},
                    onDismissError = {},
                )
            }
        }
        composeRule.onNodeWithText("Hello").assertIsDisplayed()
    }

    @Test
    fun clickNote_triggersCallback() {
        var clickedId: Long? = null
        composeRule.setContent {
            HomeScreen(
                state = HomeUiState(listState = NoteListState.Content(
                    persistentListOf(NoteCard(42, "Hello", "world", "今天"))
                )),
                onRefresh = {},
                onNoteClick = { clickedId = it },
                onDismissError = {},
            )
        }
        composeRule.onNodeWithText("Hello").performClick()
        assertEquals(42L, clickedId)
    }
}
```

**Compose UI Test 跑在 Android 设备 / 模拟器上**(`androidTest`),不是 JVM 单元测试(`test`)。但它**不需要启完整 App**——直接 `setContent { ... }` 测一个 stateless 屏幕。

依赖:

```kotlin
androidTestImplementation(libs.compose.ui.test.junit4)
debugImplementation(libs.compose.ui.test.manifest)
```

**Compose UI 测试和 ViewModel 测试是互补的**——前者验证"State X 时 UI 长啥样",后者验证"事件 Y 时 State 怎么变"。两边各占一半,覆盖率就够。

## 五、R8:混淆 + 瘦身

R8 是 AGP 自带的代码优化器,负责:
1. **混淆**(rename 类 / 方法到 `a` / `b` / `c`)——让逆向更难
2. **压缩**(去除未使用代码)——APK 减肥
3. **优化**(inline / 死代码消除)——执行更快

`:app/build.gradle.kts`:

```kotlin
buildTypes {
    release {
        isMinifyEnabled = true                     // 启用 R8
        isShrinkResources = true                   // 资源也瘦身
        proguardFiles(
            getDefaultProguardFile("proguard-android-optimize.txt"),
            "proguard-rules.pro"                   // 项目自定义
        )
    }
    debug {
        isMinifyEnabled = false                    // debug 不混淆,方便调试
    }
}
```

**`proguard-rules.pro`** 写"哪些代码 R8 不要动"——通常是反射、序列化、JNI 调用的边界:

```
# kotlinx.serialization
-keepattributes *Annotation*, InnerClasses
-dontnote kotlinx.serialization.AnnotationsKt

-keepclassmembers class **$Companion {
    kotlinx.serialization.KSerializer serializer(...);
}

# Retrofit
-keepattributes Signature, InnerClasses, EnclosingMethod
-keepattributes RuntimeVisibleAnnotations, RuntimeVisibleParameterAnnotations
-keepclassmembers,allowshrinking,allowobfuscation interface * {
    @retrofit2.http.* <methods>;
}

# Hilt 已自带规则,通常不用补
```

**实操检查**:R8 把代码改完后跑一次 `release` build,装真机跑所有主流程——尤其是会调反射 / 序列化的地方(网络请求、JSON 解析、深链接、WorkManager)。R8 bug 通常表现为"debug 跑通但 release 直接崩"。

## 六、签名:Keystore 与密钥管理

**Debug 签名**:Android Studio 自动生成 `~/.android/debug.keystore`,所有开发者机器一样,**不能用来发布**。

**Release 签名**:你自己生成的 keystore,**必须妥善保管**——丢了就再也不能更新这个 App(Play Store 拒绝接收不同签名的更新)。

生成:

```bash
keytool -genkey -v -keystore notedx-release.jks -keyalg RSA -keysize 2048 -validity 10000 -alias notedx
```

**`:app/build.gradle.kts` 配置签名**(机密信息不入 git):

```kotlin
android {
    signingConfigs {
        create("release") {
            val props = Properties().apply {
                load(file("$rootDir/keystore.properties").inputStream())
            }
            storeFile = file(props["NOTEDX_STORE_FILE"] as String)
            storePassword = props["NOTEDX_STORE_PASSWORD"] as String
            keyAlias = props["NOTEDX_KEY_ALIAS"] as String
            keyPassword = props["NOTEDX_KEY_PASSWORD"] as String
        }
    }

    buildTypes {
        release {
            signingConfig = signingConfigs.getByName("release")
        }
    }
}
```

`keystore.properties`(**git ignore**):

```
NOTEDX_STORE_FILE=/path/to/notedx-release.jks
NOTEDX_STORE_PASSWORD=...
NOTEDX_KEY_ALIAS=notedx
NOTEDX_KEY_PASSWORD=...
```

**`.gitignore` 加入**:

```
keystore.properties
*.jks
```

**Play App Signing**(推荐):你只保留一份"上传 keystore",真正的发布签名由 Google 管。即使你的上传密钥丢了,Play Store 也能用注册过的撤销机制让你重新生成上传密钥。**新项目默认走 Play App Signing**。

## 七、App Bundle(AAB):取代 APK

Play Store 自 2021 年起**强制要求 AAB**——APK 直传只能给企业内测。

AAB 是"未拼接的 APK"——上传一份 AAB,Google 根据用户的设备 ABI / 屏幕密度 / 语言生成定制 APK,每个用户只下到自己需要的部分。**典型场景下能省 30% 包体积**。

构建:

```bash
./gradlew :app:bundleRelease
# 输出:app/build/outputs/bundle/release/app-release.aab
```

**bundletool** 模拟生成 APK(测试):

```bash
bundletool build-apks --bundle=app-release.aab --output=app.apks --connected-device
bundletool install-apks --apks=app.apks
```

## 八、ABI / 资源拆分

```kotlin
android {
    bundle {
        language { enableSplit = true }
        density { enableSplit = true }
        abi { enableSplit = true }
    }
}
```

默认全开。**ABI 拆分**让每个 CPU 架构只下对应版本(arm64-v8a / armeabi-v7a / x86_64)——纯 Kotlin App 通常没多大差别,但有 native `.so` 库时差别明显。

**API 35+ 强制 16 KB page size**:第三方 .so 必须重对齐。纯 Kotlin App 零成本,但用了 OpenCV / TensorFlow Lite / FFmpeg 的项目要检查 SDK 版本。

## 九、Play Console 上架流程

**第一次上架**:

1. Play Console 创建应用——填名称、默认语言、应用类型(应用 / 游戏)、免费 / 付费
2. 准备 store listing——简短描述、完整描述、应用图标、特色图片、截图(手机 / 平板各 2-8 张)、隐私政策 URL
3. **内容评级问卷**——回答关于暴力、性、广告等问题,系统自动评级
4. **目标受众设置**——年龄段、是否儿童向
5. **数据安全表单**——声明收集哪些数据、用途、是否分享
6. 上传 AAB 到 Internal Testing(内部测试) → 邀请最多 100 个测试者
7. 内测通过 → 提交到 Closed Testing / Open Testing(公开 Beta)
8. 准备好 → 提交 Production(正式发布)

**审核时间**:正式发布首次审核 1-7 天,后续更新 24 小时内。

## 十、Internal Testing:快速发布迭代

```bash
# 1. 上传 AAB(可以用 fastlane 或 gradle play publisher 自动化)
./gradlew :app:bundleRelease

# 2. 通过 Play Console / Gradle Play Publisher 上传到 Internal Track
```

Internal Track 不审核,几分钟就能让指定 Google 账号的测试者下到。**日常迭代的标配**——每 commit 一个测试版本,QA 通过 Play Store 下载,体验真实分发链路。

`gradle-play-publisher` 插件能自动化:

```kotlin
plugins {
    id("com.github.triplet.play") version "3.11.0"
}

play {
    serviceAccountCredentials.set(file("play-service-account.json"))
    track.set("internal")
    defaultToAppBundles.set(true)
}

// ./gradlew publishReleaseBundle 上传
```

## 十一、CI 自动化

GitHub Actions workflow(`.github/workflows/release.yml`):

```yaml
name: Release
on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 17
      - uses: gradle/actions/setup-gradle@v3

      - name: Decode keystore
        run: echo "$KEYSTORE_BASE64" | base64 -d > app/notedx-release.jks
        env:
          KEYSTORE_BASE64: ${{ secrets.KEYSTORE_BASE64 }}

      - name: Build release bundle
        run: ./gradlew :app:bundleRelease
        env:
          NOTEDX_STORE_FILE: app/notedx-release.jks
          NOTEDX_STORE_PASSWORD: ${{ secrets.STORE_PASSWORD }}
          NOTEDX_KEY_ALIAS: ${{ secrets.KEY_ALIAS }}
          NOTEDX_KEY_PASSWORD: ${{ secrets.KEY_PASSWORD }}

      - name: Upload to Play Internal
        run: ./gradlew publishReleaseBundle
```

**机密**:用 GitHub Secrets,**永远不要把 keystore 提交到代码库**。

## 十二、Play Integrity:反作弊基础

Play Integrity API(2024 已完全替代 SafetyNet)给你一个判断:**这次 API 调用真的来自正常 Android 设备上的真实 App 版本吗?**

用法:

```kotlin
implementation("com.google.android.play:integrity:1.4.0")

val integrityManager = IntegrityManagerFactory.create(ctx)
val request = IntegrityTokenRequest.builder()
    .setNonce(generateNonce())                  // server 给的随机数,防重放
    .setCloudProjectNumber(123456789L)
    .build()

integrityManager.requestIntegrityToken(request)
    .addOnSuccessListener { response ->
        val token: String = response.token()
        // 发到 server,server 用 Google Play API 解密判断:
        // - appRecognitionVerdict(PLAY_RECOGNIZED / UNRECOGNIZED_VERSION)
        // - deviceRecognitionVerdict(MEETS_DEVICE_INTEGRITY / ...)
        // - accountActivity(...)
    }
```

适用场景:游戏防作弊、支付 / 提现 / 关键业务防伪造客户端。**普通笔记类 App 不需要**——NotedX 跳过。但作为示意,**敏感操作前调一次 Integrity check 是好习惯**。

## 十三、ProGuard / R8 调试

Release 包崩了怎么调?**保留 mapping 文件**:

```kotlin
buildTypes {
    release {
        isMinifyEnabled = true
        // R8 自动输出 build/outputs/mapping/release/mapping.txt
    }
}
```

`mapping.txt` 是"混淆前后的对照表"——用它把 Crashlytics 报来的 `a.b.c` 还原成真实类名。**Play Console 自动接收 mapping**——发版时同步上传,后续 Crashlytics 自动反混淆。

`gradle-play-publisher` 已经自动处理 mapping 上传。

## 十四、长期维护检查表

每发版本前过一遍:

**必查**:
- [ ] `./gradlew :app:lintRelease` 通过(Lint 警告清零或合理 suppress)
- [ ] 单元测试通过(`./gradlew test`)

- [ ] Release build 跑过所有主流程(混淆问题在 release 才暴露)
- [ ] APK 体积没意外增加(`./gradlew :app:assembleRelease && ls -lh app/build/outputs/apk/release/`)
- [ ] Macrobenchmark 启动时间没回归

**每季度查**:
- [ ] AGP / Kotlin / Compose BOM 升级到最新 stable
- [ ] targetSdk 跟上 Play Console 截止日期
- [ ] 三方库无已知严重漏洞(`./gradlew :app:dependencyUpdates`)
- [ ] 数据安全表单与实际收集一致(隐私法规)
- [ ] Crashlytics top crash 趋势收敛

**每年查**:
- [ ] 隐私政策与新法规对齐
- [ ] App icon / 截图随设计语言更新
- [ ] 不再用的权限从 manifest 移除(权限多审核易卡)

## 十六、踩坑(最后一次)

**坑 1:在 Play Store 上线后才发现 release 崩**。Debug 测试通过 ≠ Release 通过——R8 / 资源压缩 / 不同签名都可能出问题。**永远在打 release Internal 包测一遍核心流程后再 Production**。

**坑 2:Keystore 丢了或泄漏**。丢了 → 这个 App 无法再更新;泄漏 → 别人能签同名 App 偷你的用户。**Play App Signing + 离线备份 Keystore 是双保险**。

**坑 3:R8 把序列化类混淆了**。kotlinx.serialization / Room / Retrofit 用反射拿类成员,被混淆后 NoClassDefFoundError。**ProGuard rules 必须保留这些类**。

**坑 4:`exportSchema = true` 没把 schema 提交 git**。Room 升级时检测不到迁移是否正确,可能漏写 Migration → 用户升级数据库爆炸。`schemas` 目录必须入仓。

**坑 5:Play Console 数据安全表单乱填**。误声明"不收集任何数据"但实际收了——审核被卡,且被发现会下架。**老实填**。

**坑 6:版本号没递增**。Play Console 拒收 versionCode ≤ 当前版的 build。CI 自动递增 versionCode(基于 git tag / commit count)。

**坑 7:`./gradlew test` 没跑 release variant**。`test` 默认跑 debug variant,release-only 的 bug(R8 / 资源)漏掉。CI 应当跑 `testReleaseUnitTest`。

**坑 8:Internal Testing 设备没装 Google Play**。国内手机大多没 Play Store,Internal 包装不上。需要直接发 APK 给测试者,或者用 Firebase App Distribution。

**坑 9:targetSdk 不更新**。Play Console 强制要求 targetSdk 在 1-2 年内升,卡截止日期就被下架。建议**每年至少升一次 targetSdk**,逐步消化新行为。

**坑 10:没人写 Changelog**。Play Store 的"更新内容"字段必填。CI 自动从 Git tag / commit message 生成 changelog,推到 Play Console——避免每次手抄。
