# Android 原生开发系列写作计划

> 来源:`未来系列规划.md` 第六类「平台原生开发」O. androidNative,⭐⭐⭐。
> 本系列基线版本为 **Android 15 (API 35) / Kotlin 2.0 (K2 compiler) / Jetpack Compose BOM 1.7+**(以 2026/05 时点为参照),聚焦 **现代 Android 原生工程化**:用工程师视角理解协程与 Flow、Compose 重组与状态、单 Activity 路由、Hilt 依赖注入、Room / DataStore、系统能力、多模块构建、性能基线、Play Store 发布通路。
> 本系列不做"照抄某个 App"的保姆级复刻,而是围绕一个可持续演进的 Kotlin + Compose 原型应用,逐层拆解 Android 从可运行到可维护、从可维护到可发布的关键机制。

---

## 核心定位

这不是"Android 全家桶百科",也不是"Java 老 Android 迁移到 Kotlin 手册",更不是"Flutter / RN 跨端对比"。本系列只解决一个明确目标:

> 让一个熟悉编程但不熟悉 Android 平台的人,能用 **Kotlin + Jetpack Compose + Android 15 (API 35)** 做出一款架构清晰、状态可控、性能可量化、能上 Play Store 的现代 Android 应用。

### 读者画像

- 已掌握至少一门主流后端 / 前端语言,理解面向对象与函数式;
- 没怎么写过 Android,或者最后一次写 Android 还停在 Java + XML View + AsyncTask 时代;
- 想用 Kotlin 协程、Compose、Jetpack 全家桶把单端原生写好,而不是再绕一圈跨端;
- 终极交付目标:一款能签名、能上架、能灰度的 Android 应用,而不是 30 个独立 demo。

### 默认选择

| 维度 | 选择 | 原因 |
| --- | --- | --- |
| Android 版本 | targetSdk = 35 (Android 15) / minSdk = 26 (Android 8.0) | 覆盖 95%+ 设备,同时拥抱 Compose 与前台服务 type 强制 |
| 主语言 | Kotlin 2.0,K2 Compiler 默认 | 协程 / Flow 一等公民;K2 编译速度与 Compose Compiler 集成 |
| UI 框架 | Jetpack Compose BOM 1.7+(Material 3) | 不写 XML View;Shared Element / TextField2 / Lookahead 已稳定 |
| Activity 模型 | 单 Activity + Navigation Compose | 抛弃多 Activity / Fragment 心智 |
| DI | Hilt(主选)+ Koin(对比一篇) | 注解处理 + KSP 编译期生成 |
| 持久化 | Room + DataStore | 关系数据用 Room;键值 / Preferences 用 DataStore |
| 网络 | Retrofit + OkHttp + kotlinx.serialization | 协程 `suspend` 一等公民,不写 RxJava |
| 并发 | Coroutines / StructuredConcurrency / Flow | `viewModelScope` / `lifecycleScope` / 取消传播 |
| 构建 | Gradle KTS + Version Catalogs (`libs.versions.toml`) + KSP | 不写 Groovy;不写 KAPT(过渡到 KSP / KSP2) |
| 性能 | Macrobenchmark + Baseline Profiles | startup / scroll / frozen frames 可量化 |
| 测试 | JUnit 4/5 + Compose UI Test + Robolectric | UI 与逻辑分层 |
| 发布 | Play Console / App Bundle (AAB) / Play Integrity | 不讲 APK 直接分发 |
| **不覆盖** | KMP / Java 主线 / XML View 主线 / Flutter / React Native / NDK 深度 | KMP 与跨端由 flutterLearning 覆盖;Java 与 XML 仅在「互操作」单篇出现 |

> 与 flutterLearning 的边界:flutter 讲跨端表现层与 Dart 语言;androidNative 讲 Kotlin 语言、Compose 重组与状态、Android 平台生命周期与系统能力、Gradle 构建与 Play 发布。两个系列读完应当能回答:"为什么这件事一端原生做更快 / 跨端做更省"。

## Android 15 / Kotlin 2.0 / Compose 1.7 关键变更与本系列定位

Android 15 (API 35) 自 2024-10 进入 stable,Kotlin 2.0 自 2024-05 发布并默认启用 K2,Compose BOM 1.7 自 2024-09 起为稳定主线。截至 2026/05,这三个版本已成为生产主流栈。本系列把下列变更显式纳入相关篇目,避免读者把 Android 13 / 14 / Kotlin 1.9 / Compose 1.5 时代心智带进来:

| 变更 | 影响范围 | 本系列在哪篇展开 |
| --- | --- | --- |
| **Edge-to-edge enforcement(targetSdk 35 强制)** | App 默认无 system bar 留白,需要主动处理 `WindowInsets` 与 `Scaffold` 内边距 | 第 06 / 09 篇 |
| **16 KB page size 兼容** | 第三方 `.so` 库需重对齐,影响 JNI / 含 native 的 SDK | 第 26 / 28 篇 |
| **Foreground Service Type Enforcement 加强** | 启动前台服务必须声明 type 且匹配权限;后台拉起前台服务限制更严 | 第 17 篇 |
| **Predictive Back Gesture 全量** | Activity / Compose Navigation 需正确实现 `OnBackPressedCallback` / `PredictiveBackHandler` | 第 12 篇 |
| **Partial Screen Sharing API** | `MediaProjection` 支持单 App 录屏 | 第 19 / 20 篇延伸提及 |
| **Per-app language preferences 默认可用 (≥13)** | 系统设置直接切换 App 语言,需 `LocaleConfig` | 第 27 篇 |
| **Kotlin 2.0 K2 Compiler 默认** | 编译速度↑,新泛型推断,smart cast 范围扩大,带来若干迁移坑 | 第 03 / 04 篇 |
| **Compose Compiler 与 Kotlin 同版本发布(自 2.0.20 起内嵌)** | 不再独立追 Compose Compiler 版本号 | 第 02 篇构建一节 |
| **Compose 1.7 Strong Skipping 默认** | 没有 `@Stable` 注解的类也会跳过重组,显著降低重组成本但会暴露不稳定 lambda | 第 07 / 22 篇 |
| **Compose 1.7 Shared Element Transition GA** | `SharedTransitionLayout` / `sharedBoundsTransform` / `Modifier.sharedElement` | 第 10 篇 |
| **Compose 1.7 `BasicTextField` (TextField2) 取代旧 TextField** | `TextFieldState` hoist,输入法、IME、组合文本统一 | 第 08 篇 |
| **Compose 1.7 LookaheadScope GA** | 预测布局,实现"列表展开 / 抽屉 / 共享元素"自然过渡 | 第 10 / 22 篇 |
| **KSP2 取代 KAPT** | Hilt / Room / Moshi / Glide 走 KSP,显著缩短增量构建 | 第 02 / 13 / 14 篇 |
| **Navigation Compose 2.8+ 类型安全路由** | 用 `@Serializable` data class 直接当路由,告别字符串模板 | 第 12 篇 |
| **Play Integrity 取代 SafetyNet Attestation(2024 已完成下线)** | 反作弊 / 反外挂 / 反越狱新接口 | 第 29 篇 |

> 说明:若在写稿时点已有 Android 16 (API 36) 进入稳定 / Compose 1.8 / 1.9 发布,会在受影响篇目补充 16 KB page size 强制启用、Privacy Sandbox 行为变化、新的预测式动画 API 等内容,并在篇首标注 "本篇基于 BOM 1.7,1.8 差异补充见末尾"。基线仍以 API 35 / Kotlin 2.0 / Compose 1.7 为主参考,避免文章漂移。

## 六层路线

```
Kotlin 心智与项目骨架
   → Jetpack Compose 声明式 UI 与状态心智
   → 数据、导航、依赖注入与第一个完整闭环
   → 系统能力:权限、相机、媒体、推送、后台
   → 多模块、性能、Baseline Profile 与启动优化
   → 测试、签名、上架与 Play 生态
```

核心节奏是:**前 15 篇先做出一个能跑、能展示数据、能登录的 Compose 应用,后 15 篇再把它接系统能力、做性能、上 Play Store。**

## 单篇写作模板

每篇遵循固定结构,避免变成散点知识:

```text
1. 机制定位
   - 这个机制在 Android 应用里解决什么核心问题
   - Java + XML / AsyncTask / RxJava / 多 Activity 时代的旧解法为什么会失控

2. Android 心智
   - 对应核心类、注解与生命周期作用域(ViewModel / Lifecycle / CoroutineScope)
   - 数据流方向(UI ← StateFlow ← Repository ← DataSource)

3. 工程实现
   - 30-120 行可运行 Kotlin (Compose) 代码
   - 协程作用域、`@Composable` 与 `Modifier`、注解、构建脚本(KTS)、Manifest 声明、Hilt Module 一并给出

4. 调参与验收
   - 哪些参数直接影响重组次数 / 启动耗时 / 包体积 / 用户感知
```
