# Android 原生开发系列写作计划

> 基线版本:**Android 15 (API 35) / Kotlin 2.0 (K2) / Jetpack Compose BOM 1.7+**(2026/05 时点)。
> 不做"全家桶百科",不做"Java 老 Android 迁移手册",不做"跨端选型对比"。

---

## 一句话定位

**让一个会写代码、但没怎么写过 Android 的人,用 Kotlin + Compose 把一款能上 Play Store 的现代 Android 应用做出来。**

读者画像:已经熟悉一门主流语言(Go / Python / TS / Java 都行),知道什么是面向对象与函数式,但 Android 的印象还停留在 Java + XML + AsyncTask 时代,或者根本没写过 Android。

---

## 统一心智(贯穿 22 章)

> **现代 Android 是一棵被状态驱动重组的 UI 树,挂在单个 Activity 上,被一组 ViewModel 持有状态,被协程驱动副作用。其余 Jetpack 库,都是这棵树的扩展或副作用边界。**

这句话压住全场。后面所有章节都是这句话的推论:

- **"一棵 UI 树"** → 05/06/07/08/09 篇讲 Compose 怎么写、怎么重组
- **"被状态驱动"** → 10 篇 ViewModel + UDF,状态如何流动
- **"挂在单 Activity 上"** → 11 篇 Navigation,告别多 Activity / Fragment
- **"被协程驱动副作用"** → 04 篇协程与 Flow
- **"Jetpack 库是扩展或副作用边界"** → 12-19 篇,DI / 持久化 / 网络 / 系统能力

如果哪一篇没法挂在这棵树上,这一篇就不该出现。

## 默认技术选择

| 维度 | 选择 | 一句话理由 |
| --- | --- | --- |
| 平台 | minSdk = 26 / target = 35 | 覆盖 95%+ 装机,同时拿到 Compose / NotificationChannel / 现代后台 |
| 主语言 | Kotlin 2.0,K2 编译器 | 协程 / Flow 是一等公民,K2 显著提升编译速度 |
| UI | Jetpack Compose BOM 1.7+ Material 3 | 不写 XML View;Strong Skipping / SharedElement / TextField2 已稳定 |
| Activity | 单 Activity + Navigation Compose | 一棵树挂一个 Activity 上,扔掉 Fragment |
| DI | Hilt | 注解 + KSP 编译期生成,默认答案 |
| 持久化 | Room + DataStore | 关系数据 Room,键值偏好 DataStore |
| 网络 | Retrofit + OkHttp + kotlinx.serialization | `suspend` 一等公民,不写 RxJava |
| 并发 | 协程 + Flow | 结构化并发 + 取消传播 |
| 构建 | Gradle KTS + Version Catalog + KSP | 不写 Groovy,不写 KAPT |
| 性能 | Macrobenchmark + Baseline Profile | startup / scroll 可量化 |
| 发布 | App Bundle (AAB) + Play Console | 不讲 APK 直接分发 |

**不覆盖**:KMP / Java 主线 / XML View 主线 / Flutter / RN / NDK 深度。跨端见 [[flutterLearning]]。

## 持续演进的样例应用:NotedX

整个系列共享一个样例应用 **NotedX**(笔记 + 待办 + 图片附件 + 云同步),从第 05 篇起逐章长大,最终具备:

- 单 Activity + Compose,Material 3 + edge-to-edge
- 笔记 / 待办 / 标签三个实体,Room + DataStore
- Hilt DI,Retrofit + OkHttp 拉远端,kotlinx.serialization 解析
- CameraX 拍照 + Photo Picker 选图 + MediaStore 写图片
- WorkManager 周期同步,FCM 推送
- App Bundle 签名上架 Play Internal Track,Baseline Profile 优化启动
- Macrobenchmark 跑基准,R8 瘦身

这套功能恰好覆盖一个真实应用的所有关键面:UI 重组成本(列表 + 输入)、持久化与远程同步、系统能力(相机 / 媒体 / 推送)、性能与发布。比 To-Do 更接近真实,比社交 App 又不被业务细节带偏。

**但 NotedX 不是每篇都强调**——它是一根可以挂代码的线,不是这本书的主角。心智才是主角。

## 写作风格(强约束)

1. 每篇开篇一句话压上核心 claim,不超 40 字。
2. 章节标题用「一、二、三」,不用「机制定位 / 调参与验收」之类的学究腔。
3. 段落不超过 150 字,长段落必拆。
4. 强观点直接说。读者要的是判断,不是百科。
5. 表格只用于真正需要对比的地方,不堆砌。
6. 每篇结尾留一段"踩坑",对应一个真实事故场景,不写"建议注意"这种废话。
7. 不写多余的"小结"和"过渡段"。
