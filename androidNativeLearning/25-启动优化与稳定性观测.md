# 25-启动优化与稳定性观测

> 一句话导读:启动是用户对一款 App 的第一印象,也是 Play Vitals 第一个会把你拉出来公开示众的指标;启动慢、ANR 多、崩溃高,商店转化率就开始掉。这一篇把"启动跑得快"和"线上看得见"两件事拼在一张图里,前者给可运行 demo,后者讲指标体系。

第 23 篇做完 Macrobenchmark,你已经能用脚本量化"启动平均花了多少毫秒";第 24 篇做完 R8,你的包体瘦了,启动也顺带快了一点。但启动优化是一个独立工程话题,光靠"包小"是吃不到大头的——你得知道 `Application.onCreate` 里塞了什么、Hilt 第一次注入花了多少、第一帧渲染前还卡在哪里。本篇的前 60% 把这些拆开,给 `App Startup` library + lazy `Initializer` + `SplashScreen API` + `Trace.beginSection` 的实战;后 40% 转到"线上跑起来之后怎么知道它崩没崩、卡没卡",讲 `StrictMode` 在开发期当告警、`ApplicationExitInfo` 拿历史退出原因、ANR 触发条件、Crashlytics 接入点、Play Vitals 几个核心指标怎么读。

启动这件事的隐性预算是这样的:Play Vitals 把"slow cold start"定义在 P50 ≥ 5s 或 P95 ≥ 6.5s(2024-2025 阈值),触发一次就进"行为不达标"分类,商店排名会被压。也就是说你不是"想优化就优化",而是必须把 Cold start 控制在 5 秒以内,Warm start 控制在 2 秒以内,否则线上数据会反过来逼你。

## 1. 机制定位

把"启动慢"和"线上卡崩 ANR"放进一篇讲,原因是它们底层都是同一种工程问题:**你看不见的代码在你看不见的时机跑了它不该跑的事**。启动慢的 90% 是 `Application.onCreate` 里堆了一打第三方 SDK 同步初始化;ANR 的 80% 是主线程被一段 I/O 或大列表反射卡了 5 秒以上;Crash 的 60% 是冷启动路径上某个 `lateinit` 还没赋值。这三件事看起来在不同层,但都依赖同一个能力:**进程启动的时序与主线程任务图能被你显式控制**。

旧 Android 时代的失控写法很典型:`Application.onCreate` 里一口气 `init` 八九个 SDK——Firebase、AppCenter、推送、统计、ABTest、字体、图床、广告——每个 SDK 都说"放在 Application 里就好"。结果你这个 App 冷启动 8 秒,头两秒还啥都不显示。再加上几个 `ContentProvider` 自动初始化(AndroidX Startup 之前的方案,每个库塞一个空 Provider 占进程启动队列),你根本不知道这些代码到底什么时候、按什么顺序、跑了多久。

`App Startup` library 的存在就是为了把这一堆"我也不知道在哪初始化的代码"收编进一个有向无环图。你声明 `Initializer<T>`,在 `dependencies()` 里说"我依赖谁",在 `create()` 里写真正的初始化逻辑,框架替你拓扑排序、合并 Provider、统一在一个进程时机里跑完。配合 `LazyInitializer` 模式,你还能进一步把"非必须在启动时跑的初始化"延后到首屏渲染之后。

`SplashScreen API`(`androidx.core:core-splashscreen`)解决的则是"启动这几秒用户看到什么"。Android 12 开始系统强制每个 App 都有 splash,你不主动接管就是默认 Logo + 主题色;主动接管能让你控制启动结束的时机——`setKeepOnScreenCondition { viewModel.isReady.value.not() }` 这一行能把 splash 显示到首屏数据就绪。这给"启动看起来快"留了一个工程位:即便冷启动总共 3.5 秒,只要 splash 在前 1.5 秒就过渡到内容,用户感知不到那么慢。

观测侧的机制定位也类似——你需要能问出三个问题:**上次进程是怎么挂的?现在 P95 启动多少毫秒?线上崩溃集中在哪一行?** `ApplicationExitInfo`(API 30+)回答第一个,Macrobenchmark + Play Vitals 回答第二个,Crashlytics 回答第三个。三件事不重叠,但拼起来就是一张"线上 App 健康度仪表盘"。

## 2. Android 心智

理解启动优化先要分清三种启动:

| 类型 | 触发场景 | 是否走 `Application.onCreate` | 是否新建 Activity |
| --- | --- | --- | --- |
| Cold start | 进程被系统杀掉后重新拉起;首次安装后第一次启动 | 是 | 是,从无到有 |
| Warm start | 进程还在,但 Activity 已被销毁(用户长按返回退到桌面再点回来) | 否 | 是 |
| Hot start | 进程在,Activity 也在,只是被切到后台 | 否 | 否,只是 `onResume` |

线上看到的 P50、P95 启动时间,几乎都是 Cold start 占大头,因为它包含完整路径:Zygote fork → `Application.attachBaseContext` → `Application.onCreate` → ContentProvider 初始化 → 第一个 Activity 的 `onCreate / onStart / onResume` → 第一帧 `Choreographer` 渲染。Hot start 通常 100-300ms,Warm 500-1500ms,Cold 普遍 2000-5000ms。Play Vitals 看的 "App startup time" 默认是 Cold 的 Time to Initial Display(TTID)。

`App Startup` 的核心类是 `Initializer<T>` 与 `AppInitializer`。每个 `Initializer<T>` 声明:

```kotlin
override fun create(context: Context): T { ... }
override fun dependencies(): List<Class<out Initializer<*>>>
```

库本身在 `AndroidManifest.xml` 里注册了一个 `androidx.startup.InitializationProvider`,这个 `ContentProvider` 在 App 启动时被系统调用 `onCreate()`,然后它把所有声明在 `<meta-data>` 里的 Initializer 收集起来、按依赖关系拓扑排序、依次 `create()`。这取代了过去"每个库塞一个自己的 ContentProvider"的做法——一个 Provider 比八个 Provider 启动开销低,且执行顺序可控。

`LazyInitializer` 是一个工程模式而不是单独的类:把 `Initializer<T>` 声明出来但不放进 `AndroidManifest` 的 `<meta-data>` 里,改为在首屏渲染完成后调用 `AppInitializer.getInstance(context).initializeComponent(MyInitializer::class.java)`。这样这些初始化逻辑不再阻塞冷启动主线程,被推到了"首帧之后"。

`SplashScreen API` 的关键钩子是 `installSplashScreen()`(在 `super.onCreate(savedInstanceState)` 之前调用)以及 `setKeepOnScreenCondition { }`。前者把 splash 从系统主题接管到你的 Activity,后者控制"什么时候 splash 才退出"。配合 `setOnExitAnimationListener` 还能写过渡动画,但 90% 的项目不需要。

`Trace.beginSection("foo")` / `Trace.endSection()` 是 Systrace / Perfetto 看时间线时的"打点"机制。你写 `Trace.beginSection("init-firebase")` 然后跑完 Firebase 初始化再 `endSection`,Perfetto 时间线上就能看到这一段被高亮成一根色块,知道它花了多少毫秒。这是真正定位"哪段初始化卡了"的方式,光靠 `Log.d` + 时间戳是不够的,因为你无法对齐到主线程是否被阻塞。

观测侧的核心类:

- `StrictMode.ThreadPolicy` / `StrictMode.VmPolicy`:开发期"我做错了一件本该异步的事"的告警。在主线程做磁盘读写、网络访问、Cursor 没关、Activity 泄漏,统统会被打到 Logcat 或直接 crash(可配置)。生产环境必须关掉,否则线上会因为"某 SDK 主线程读盘"被你的 StrictMode 杀掉。
- `ApplicationExitInfo`(API 30+):`ActivityManager.getHistoricalProcessExitReasons(packageName, pid=0, maxNum=10)`,返回最近 10 次进程退出原因——`REASON_CRASH`、`REASON_ANR`、`REASON_LOW_MEMORY`、`REASON_USER_REQUESTED` 等。这是 Android 提供的官方"上次为什么挂"接口,比自己埋 sentinel 文件可靠得多。
- ANR:主线程被阻塞超过阈值(前台 Activity 5 秒、广播 10 秒、Service 20 秒)系统就会产 ANR。Crashlytics 和 Play Vitals 都会聚合 ANR。ANR 不等于 crash——进程不一定会死,但用户会看到"应用无响应"弹窗。
- Firebase Crashlytics:崩溃聚合服务,符号化栈、按设备/版本分桶。接入点是 `firebase-crashlytics-gradle` plugin + `firebase-crashlytics` 依赖,自动捕获未处理异常。手动 `FirebaseCrashlytics.getInstance().log("...")` / `recordException(e)` 可以补"非致命"事件。
- Play Vitals:Play Console 后台一栏,基于真实用户匿名上报,关键指标是"User-perceived ANR rate"(应在 0.47% 以下)、"User-perceived crash rate"(应在 1.09% 以下)、"Slow cold startup time"、"Slow rendering"、"Excessive wake-ups"。每个指标都有"行为阈值"(超了商店搜索会被压)和"不良行为阈值"(超了可能被下架)。

## 3. 工程实现

### 3.1 启动优化(主线)

#### 3.1.1 用 App Startup library 替代散落的 ContentProvider

`libs.versions.toml`:

```toml
[versions]
androidx-startup = "1.2.0"
androidx-splashscreen = "1.0.1"

[libraries]
androidx-startup = { module = "androidx.startup:startup-runtime", version.ref = "androidx-startup" }
androidx-splashscreen = { module = "androidx.core:core-splashscreen", version.ref = "androidx-splashscreen" }
```

定义一个把日志库初始化收编进来的 Initializer。文件路径 `app/src/main/java/com/notedx/startup/TimberInitializer.kt`:

```kotlin
package com.notedx.startup

import android.content.Context
import android.util.Log
import androidx.startup.Initializer

class TimberInitializer : Initializer<Unit> {
    override fun create(context: Context) {
        // 真实项目里这里是 Timber.plant(...) / Logger.init(...)
        Log.i("Startup", "Timber initialized at ${System.currentTimeMillis()}")
    }

    override fun dependencies(): List<Class<out Initializer<*>>> = emptyList()
}
```

依赖它的另一个 Initializer——比如崩溃上报需要等日志先就绪。文件路径 `app/src/main/java/com/notedx/startup/CrashlyticsInitializer.kt`:

```kotlin
package com.notedx.startup

import android.content.Context
import androidx.startup.Initializer
import com.google.firebase.crashlytics.FirebaseCrashlytics

class CrashlyticsInitializer : Initializer<FirebaseCrashlytics> {
    override fun create(context: Context): FirebaseCrashlytics {
        val crashlytics = FirebaseCrashlytics.getInstance()
        crashlytics.setCrashlyticsCollectionEnabled(true)
        crashlytics.setCustomKey("app_startup", "true")
        return crashlytics
    }

    override fun dependencies(): List<Class<out Initializer<*>>> =
        listOf(TimberInitializer::class.java)
}
```

`AndroidManifest.xml` 注册 `InitializationProvider`,把两个 Initializer 都挂上去:

```xml
<provider
    android:name="androidx.startup.InitializationProvider"
    android:authorities="${applicationId}.androidx-startup"
    android:exported="false"
    tools:node="merge">
    <meta-data
        android:name="com.notedx.startup.TimberInitializer"
        android:value="androidx.startup" />
    <meta-data
        android:name="com.notedx.startup.CrashlyticsInitializer"
        android:value="androidx.startup" />
</provider>
```

这样冷启动时,系统只触发一个 `ContentProvider.onCreate()`,然后框架按依赖图依次跑 `TimberInitializer.create` → `CrashlyticsInitializer.create`。

#### 3.1.2 lazy Initializer:把非必须初始化推到首屏之后

不是所有 SDK 都必须在 `Application.onCreate` 跑完。统计、ABTest、字体下载、远程配置,这些拖到首屏渲染完之后再初始化,用户感知不到。做法是声明 Initializer 但**不在 Manifest 里挂**:

```kotlin
// app/src/main/java/com/notedx/startup/AnalyticsInitializer.kt
package com.notedx.startup

import android.content.Context
import androidx.startup.Initializer

class AnalyticsInitializer : Initializer<Unit> {
    override fun create(context: Context) {
        // 真实 SDK 初始化逻辑
    }

    override fun dependencies(): List<Class<out Initializer<*>>> = emptyList()
}
```

然后在 `MainActivity` 首帧渲染完成后手动触发:

```kotlin
// app/src/main/java/com/notedx/MainActivity.kt
package com.notedx

import android.os.Bundle
import androidx.activity.ComponentActivity
import androidx.activity.compose.setContent
import androidx.core.splashscreen.SplashScreen.Companion.installSplashScreen
import androidx.startup.AppInitializer
import com.notedx.startup.AnalyticsInitializer
import dagger.hilt.android.AndroidEntryPoint
import kotlinx.coroutines.Dispatchers
import kotlinx.coroutines.launch
import androidx.lifecycle.lifecycleScope

@AndroidEntryPoint
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        val splashScreen = installSplashScreen()
        super.onCreate(savedInstanceState)

        val viewModel: MainViewModel by viewModels()
        splashScreen.setKeepOnScreenCondition { !viewModel.isReady.value }

        setContent { NotedXApp() }

        // 首帧渲染完成后启动延迟初始化
        window.decorView.post {
            lifecycleScope.launch(Dispatchers.Default) {
                AppInitializer.getInstance(this@MainActivity)
                    .initializeComponent(AnalyticsInitializer::class.java)
            }
        }
    }
}
```

`window.decorView.post` 让 lambda 排在首帧渲染之后才被主线程取走;再切 `Dispatchers.Default` 把初始化挪到后台线程。

#### 3.1.3 SplashScreen API 与 ViewModel 准备完成

`themes.xml` 给 App 主题指明 splash:

```xml
<style name="Theme.NotedX.Splash" parent="Theme.SplashScreen">
    <item name="windowSplashScreenBackground">@color/splash_background</item>
    <item name="windowSplashScreenAnimatedIcon">@drawable/ic_splash_logo</item>
    <item name="postSplashScreenTheme">@style/Theme.NotedX</item>
</style>
```

`Manifest` 里把 launcher Activity 的 theme 指到这个:

```xml
<activity
    android:name=".MainActivity"
    android:theme="@style/Theme.NotedX.Splash"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.MAIN" />
        <category android:name="android.intent.category.LAUNCHER" />
    </intent-filter>
</activity>
```

`MainViewModel` 暴露 `isReady`,在首屏数据就绪后置 true:

```kotlin
// app/src/main/java/com/notedx/MainViewModel.kt
package com.notedx

import androidx.lifecycle.ViewModel
import androidx.lifecycle.viewModelScope
import dagger.hilt.android.lifecycle.HiltViewModel
import kotlinx.coroutines.flow.MutableStateFlow
import kotlinx.coroutines.flow.asStateFlow
import kotlinx.coroutines.launch
import javax.inject.Inject

@HiltViewModel
class MainViewModel @Inject constructor(
    private val noteRepository: NoteRepository,
) : ViewModel() {

    private val _isReady = MutableStateFlow(false)
    val isReady = _isReady.asStateFlow()

    init {
        viewModelScope.launch {
            noteRepository.warmUp()  // 预热本地缓存
            _isReady.value = true
        }
    }
}
```

`setKeepOnScreenCondition { !viewModel.isReady.value }` 让 splash 一直显示到 `_isReady.value = true`。这给了你一个工程位:**冷启动总耗时 = max(系统进程冷启 + Application.onCreate + Activity.onCreate, ViewModel.warmUp)**;只要 warmUp 不超过冷启路径,splash 一过就直接显示数据,不会出现"空白页 → 转圈 → 内容"三段式。

#### 3.1.4 用 Trace.beginSection 在 Perfetto 上看真实耗时

在你怀疑的初始化代码段两端打点:

```kotlin
// app/src/main/java/com/notedx/NotedXApplication.kt
package com.notedx

import android.app.Application
import android.os.Trace
import dagger.hilt.android.HiltAndroidApp

@HiltAndroidApp
class NotedXApplication : Application() {
    override fun onCreate() {
        Trace.beginSection("Application.onCreate")
        super.onCreate()
        Trace.beginSection("init-network")
        // 网络层初始化
        Trace.endSection()
        Trace.beginSection("init-db")
        // 数据库 warmup
        Trace.endSection()
        Trace.endSection()
    }
}
```

抓 trace 命令(在已 root 或可用 `adb` 的设备上):

```bash
adb shell perfetto -o /data/misc/perfetto-traces/notedx.perfetto-trace \
  -t 10s -b 32mb sched freq idle am wm gfx view input \
  webview wm camera hal app res dalvik power
```

拉回来用 https://ui.perfetto.dev 打开,搜索 `Application.onCreate`,能看到嵌套的色块和精确毫秒。Macrobenchmark 也支持 `metrics = listOf(TraceSectionMetric("init-db"))`,在 CI 里直接断言每段耗时上界。

#### 3.1.5 Cold / Warm / Hot start 的 adb 测量法

这是发售前 P0 必跑命令,任何对外宣传"启动 X 秒"必须用它的数字背书。**Cold start**(先杀进程再启动):

```bash
adb shell am force-stop com.notedx
adb shell am start -W -n com.notedx/.MainActivity
```

`-W` 让 `am start` 阻塞到 Activity 首帧渲染完成才返回,输出形如:

```
Status: ok
LaunchState: COLD
Activity: com.notedx/.MainActivity
TotalTime: 1247
WaitTime: 1289
```

`TotalTime` 是从 `am start` 到 Activity 首次绘制完成的毫秒数,这是 Cold start 的官方度量。

**Warm start**(进程已死但 Activity 实例缓存路径不同——通过模拟"用户长按 Back 退到桌面"再启动):

```bash
adb shell input keyevent KEYCODE_HOME
sleep 2
adb shell am start -W -n com.notedx/.MainActivity
```

`LaunchState` 会显示 `WARM`,`TotalTime` 通常是 Cold 的 1/3 到 1/2。

**Hot start**(Activity 仍在,只是切回前台):

```bash
adb shell input keyevent KEYCODE_HOME
sleep 1
adb shell am start -W -n com.notedx/.MainActivity
```

注意 Warm 与 Hot 的区别在于 Activity 是否还活着——这取决于系统内存压力,你没法 100% 强制,只能多跑几次取均值。Macrobenchmark 提供 `StartupMode.COLD/WARM/HOT` 抽象,在自动化测试里更稳。

### 3.2 稳定性观测(延伸)

#### 3.2.1 StrictMode 在 Debug 构建里当告警

`NotedXApplication` 里加入条件分支:

```kotlin
override fun onCreate() {
    if (BuildConfig.DEBUG) {
        StrictMode.setThreadPolicy(
            StrictMode.ThreadPolicy.Builder()
                .detectDiskReads()
                .detectDiskWrites()
                .detectNetwork()
                .detectCustomSlowCalls()
                .penaltyLog()
                .build()
        )
        StrictMode.setVmPolicy(
            StrictMode.VmPolicy.Builder()
                .detectLeakedSqlLiteObjects()
                .detectLeakedClosableObjects()
                .detectActivityLeaks()
                .penaltyLog()
                .build()
        )
    }
    super.onCreate()
}
```

`penaltyLog()` 只打日志,`penaltyDeath()` 会直接 crash(只在 CI 用,日常开发体验太硬)。Release 构建一定不要带 StrictMode,它会因为某些第三方 SDK 主线程读取小文件而误伤。

#### 3.2.2 ApplicationExitInfo 看上次进程怎么挂的

在启动早期读历史退出原因,推荐挪到 `MainViewModel.init` 或单独的 InitWorker:

```kotlin
// app/src/main/java/com/notedx/diagnostics/ExitInfoCollector.kt
package com.notedx.diagnostics

import android.app.ActivityManager
import android.app.ApplicationExitInfo
import android.content.Context
import android.os.Build
import androidx.annotation.RequiresApi
import com.google.firebase.crashlytics.FirebaseCrashlytics

@RequiresApi(Build.VERSION_CODES.R)
fun collectRecentExitInfo(context: Context) {
    val am = context.getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager
    val infos = am.getHistoricalProcessExitReasons(context.packageName, 0, 5)
    val crashlytics = FirebaseCrashlytics.getInstance()
    for (info in infos) {
        crashlytics.log(
            "last_exit reason=${reasonName(info.reason)} " +
                "status=${info.status} importance=${info.importance} " +
                "ts=${info.timestamp} desc=${info.description}"
        )
        if (info.reason == ApplicationExitInfo.REASON_ANR ||
            info.reason == ApplicationExitInfo.REASON_CRASH
        ) {
            info.traceInputStream?.bufferedReader()?.use { reader ->
                crashlytics.log("trace=\n${reader.readText().take(8 * 1024)}")
            }
        }
    }
}

private fun reasonName(reason: Int): String = when (reason) {
    ApplicationExitInfo.REASON_ANR -> "ANR"
    ApplicationExitInfo.REASON_CRASH -> "CRASH"
    ApplicationExitInfo.REASON_LOW_MEMORY -> "LOW_MEMORY"
    ApplicationExitInfo.REASON_USER_REQUESTED -> "USER"
    ApplicationExitInfo.REASON_USER_STOPPED -> "USER_STOPPED"
    else -> "OTHER($reason)"
}
```

这个调用必须在 API 30+ 检查后跑,旧版本拿不到。

#### 3.2.3 Crashlytics 接入与非致命异常

`build.gradle.kts`(模块级):

```kotlin
plugins {
    id("com.android.application")
    id("com.google.gms.google-services")
    id("com.google.firebase.crashlytics")
    id("kotlin-android")
}

dependencies {
    implementation(platform("com.google.firebase:firebase-bom:33.5.0"))
    implementation("com.google.firebase:firebase-crashlytics-ktx")
    implementation("com.google.firebase:firebase-analytics-ktx")
}
```

业务代码里手动上报非致命异常:

```kotlin
// 例:repository 层 catch 到 IO 异常时
try {
    api.fetchNotes()
} catch (e: IOException) {
    FirebaseCrashlytics.getInstance().recordException(e)
    emit(Result.failure(e))
}
```

`recordException` 走 Crashlytics 的 non-fatal 通道,不影响进程存活,但能在后台看到聚合栈。

## 4. 调参与验收

**Cold start 目标值。** Play Vitals 的"slow cold startup"阈值是 P50 ≥ 5s 或 P95 ≥ 6.5s。生产 App 的工程目标应是 P50 ≤ 2s、P95 ≤ 4s,留给"用户低端机 + 冷启 + 第一次安装"足够缓冲。具体数字根据 Macrobenchmark CI 跑出来的中位数定,不要拍脑袋。

**Application.onCreate 上限。** 强烈建议 ≤ 100ms。超过 200ms 几乎必然影响 Cold start P50,因为冷启动主线程被这段占住。所有第三方 SDK 都要审视:是否可以走 lazy Initializer?是否可以延迟到首帧后?

**首帧到首屏数据的边界。** SplashScreen 控制"用户感知开始"。如果首屏数据需要网络请求,要么走骨架屏(splash 不挡内容,内容里渲染占位),要么走本地缓存优先(`warmUp` 只读本地,网络层在 splash 退出后异步刷新)。绝对不要让 splash 等网络请求——一旦弱网,启动直接卡到超时。

**StrictMode 报警的处理优先级。** 主线程 disk read 频率高(N/分钟)优先修;主线程 network 必修(可能直接 ANR);Activity 泄漏中等优先;Cursor 未关偶发警告可下一个版本修。生产 release 必须关掉 StrictMode,只在 BuildConfig.DEBUG / staging 启用。

**ANR 触发的实际条件。** 前台 Activity 主线程阻塞 5 秒以上触发"Input dispatching timed out"型 ANR;BroadcastReceiver 主线程阻塞 10 秒触发广播 ANR;前台 Service 主线程阻塞 20 秒触发 Service ANR。注意所有这些都是"主线程",任何后台线程阻塞不构成 ANR。Crashlytics 抓到的 ANR 一定要看 trace.txt,几乎 100% 能定位到具体函数。

**Play Vitals 几个指标的优先级。**

| 指标 | 行为阈值 | 不良行为阈值 | 工程含义 |
| --- | --- | --- | --- |
| User-perceived ANR rate | 0.47% | 0.47% | 每 200 个会话至少 1 个 ANR 就要紧急修 |
| User-perceived crash rate | 1.09% | 1.09% | 每 100 个会话至少 1 次 crash 就要紧急修 |
| Slow cold startup | P50 ≥ 5s 或 P95 ≥ 6.5s | 同 | 启动慢,影响搜索排名 |
| Slow rendering | 慢帧 ≥ 26% | 慢帧 ≥ 26% | 列表滚动卡 |
| Excessive wake-ups | 每小时 ≥ 10 次 | 同 | WorkManager / AlarmManager 用得过激 |

阈值会随年份微调,2026 数字以 Play Console 当前显示为准。

**验收路径。** 一次完整的"启动优化"验收应包括:

1. Macrobenchmark CI 跑通 `StartupBenchmarks`,Cold start P50 已记录基线;
2. 上述 `adb shell am start -W` 命令在你的目标设备上跑 5 次,Cold start 平均值 ≤ 你的目标(如 2000ms);
3. Perfetto trace 里 `Application.onCreate` ≤ 100ms,无任何单段 trace section ≥ 200ms;
4. SplashScreen 退出时机与 `MainViewModel.isReady` 对齐,空白闪烁 ≤ 0 帧;
5. StrictMode 在 Debug 构建下无主线程 disk/network 报警;
6. 至少跑通一次 Crashlytics 上报 demo——手动 `throw RuntimeException("test")`,在 Crashlytics 后台 5 分钟内看到记录;
7. ApplicationExitInfo collector 接入 Crashlytics log,人为 kill 进程后下次启动能从日志看到 `REASON_USER_REQUESTED`。

性能基线的具体度量方法见 [[androidNativeLearning]] 第 23 篇,Compose 重组成本见 [[androidNativeLearning]] 第 22 篇;HTTP 弱网兜底由 [[networkLearning]] 覆盖;Crashlytics 是端侧上报,后端聚合落库与告警机制由 [[systemDesign]] 提供参考。

## 5. 踩坑

**`AndroidX Startup` 不要漏 `tools:node="merge"`。** 三方库声明了自己的 `<meta-data>`,你 App 主 manifest 又写了一份,如果不写 merge 会冲突,Manifest merger 报 "Element provider#androidx.startup.InitializationProvider has conflict"。

**`installSplashScreen()` 必须在 `super.onCreate` 之前调用。** 写反了 splash 不接管,系统退回旧行为。Android 12+ 硬约束,文档不显眼但代码顺序错就是"splash 配了等于没配"。

**`setKeepOnScreenCondition` 不要无限延迟。** `isReady` 永远没置 true(网络挂又没超时)splash 会卡死。必须给 warmUp 设硬超时(`withTimeout(3000L)`),超时强制置 ready,允许用户看到"刷新失败"提示。

**`Trace.beginSection` / `endSection` 必须配对。** 漏一个 endSection 后续所有 trace 都嵌进这段,Perfetto 视图错乱。用 `androidx.tracing` 的 inline 扩展 `trace("name") { block }`,自动确保配对。

**`@HiltAndroidApp` 初始化代价不可忽略。** Hilt 第一次构 SingletonComponent,几十个 `@Singleton` 时本身就要 50-100ms。审计 Singleton 数量,把不必要的改 `@ActivityScoped` 或每次构造,启动时间立刻有变化。

**StrictMode 在 release 开 `penaltyDeath()` 是灾难。** 第三方 SDK(广告、统计、推送)很可能主线程读小配置文件,release 开严格模式就会被各家 SDK 触发崩溃。Release 必须无 StrictMode;只在 debug + staging 开 `penaltyLog`。

**`ApplicationExitInfo` 在 API 30+ 才有。** 低版本直接 `NoSuchMethodError`,必须包 `if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.R)`。低版本只能靠 sentinel 文件兜底(启动写、正常退出删,启动看到说明上次异常退出)。

**Crashlytics 的 ANR 上报需要 API 30+ 且开关启用。** 新版 Crashlytics 默认读 `ApplicationExitInfo` 抓 ANR,但前提是设备 API ≥ 30 且没在 manifest 写 `firebase_crashlytics_collection_enabled = false`。开关关了即便调 `setCrashlyticsCollectionEnabled(true)` 也要手动启用一次。

**Play Vitals 数据有 24-72 小时延迟。** 发版第一天看 Vitals 是空的,不是 bug。实时数据用 Firebase Crashlytics 后台,Play Vitals 适合看趋势,不适合事故应急。

**用 `am start -W` 测启动时间一定先 `am force-stop`。** 没 force-stop 跑的是 Warm 或 Hot,`LaunchState` 显示 WARM/HOT,数字差几倍。Cold 真测一次一定要 `force-stop` 再 `start`。

**P50 与 P95 工程目标不同。** P50 影响平均体验,P95 防"差评高发段"。Play Vitals 同时看两个,只盯 P50 会导致 P95 拉到 8 秒,差评里全是"启动慢得不能用"。

**`recordException` 与 `log` 的关系。** `log("foo")` 只加 breadcrumb 队列,不触发上报。只有未捕获异常或 `recordException` 调用时这堆 log 才会随包发出。只调 log 不抛异常,Crashlytics 后台啥也没有——这是经典"加了 log 看不到"的原因。

**StartupMode.HOT 实际是"按 Home + 立刻回来"。** 仍有 `onPause`/`onResume`,所以 Hot start 也会因为 onResume 做重活变慢。"网络刷新"、"重算 UI 状态"放到 ViewModel 而不是 onResume,Hot 才能真的快。

## 手动验证

- [ ] `App Startup` 把日志和 Crashlytics 两个 Initializer 注册到同一个 `InitializationProvider`,启动时按依赖图先 Timber 后 Crashlytics。
- [ ] AnalyticsInitializer 不在 Manifest 里,首帧渲染后通过 `AppInitializer.initializeComponent` 显式触发,Logcat 看到调用时机在 Activity 首次绘制之后。
- [ ] `installSplashScreen()` 接管启动画面,`setKeepOnScreenCondition` 让 splash 显示到 `MainViewModel.isReady = true` 才退出;人为把 `warmUp` 加 `delay(2000)` 验证 splash 真的停留了 2 秒。
- [ ] `adb shell am force-stop com.notedx && adb shell am start -W -n com.notedx/.MainActivity` 输出包含 `LaunchState: COLD` 与 `TotalTime: <ms>`。
- [ ] 模拟 Warm:按 Home 退后台等 2 秒,再 `am start -W`,`LaunchState: WARM`,`TotalTime` 明显小于 Cold。
- [ ] 在 `Application.onCreate` 里插 `Trace.beginSection("init-db")` + `endSection()`,抓一次 Perfetto trace,UI 上能搜到对应色块并看到毫秒数。
- [ ] StrictMode 在 Debug 启用,人为在主线程 `File("/sdcard/test.txt").readText()`,Logcat 看到 `StrictMode policy violation: StrictModeDiskReadViolation`。
- [ ] Crashlytics 集成完成后,在某个按钮 onClick 里 `throw RuntimeException("test-crash")`,5-15 分钟后 Crashlytics 后台出现该异常并符号化栈正确。
- [ ] `ApplicationExitInfo` collector 跑通:手动 `adb shell am kill com.notedx`,下次启动时 Crashlytics log 看到 `last_exit reason=USER`。
- [ ] Play Vitals 后台至少看到一个真实数据点(发版 3 天后),启动时间 P50 / P95 与本地 Macrobenchmark 数字在同一数量级,没有 1 个数量级的偏差。
