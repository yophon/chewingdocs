# 内存与泄漏:Java 引用模型 + LeakCanary 心智

> 一句话:**Android 内存泄漏的根因永远是"长寿命对象持有了短寿命对象的引用"——LeakCanary 通过 WeakReference + GC 检测这件事,Profiler 帮你定位泄漏路径**。10 个泄漏 9 个是 Activity 被 static / 单例 / Handler 持有。

---

## 一、Android 上"内存"到底指什么

`adb shell dumpsys meminfo com.notedx` 输出几个关键指标:

```
Java Heap        24532 KB     ← Kotlin/Java 对象占用(我们最关心的)
Native Heap       5132 KB     ← JNI / Bitmap(API 11+ 部分)/ Skia
Stack              420 KB     ← 线程栈
.so mmap          5012 KB     ← 加载的 .so 库
.dex mmap         1376 KB     ← DEX 文件 mmap
.apk mmap          856 KB     ← APK 资源 mmap
.art mmap         2008 KB     ← Boot image
Other             3500 KB     ← 各种杂项
─────────────────────
TOTAL            ~43 MB
```

**调优重点是 Java Heap + Native Heap**——这是真正可控的部分。其他 mmap 与系统共享。

## 二、Java 引用类型

```kotlin
// 强引用(默认):永不被 GC
val strong: Note = Note(...)

// 软引用:内存紧张时被 GC
val soft: SoftReference<Bitmap> = SoftReference(bigBitmap)

// 弱引用:下次 GC 一定回收
val weak: WeakReference<Activity> = WeakReference(activity)

// 虚引用:无法拿到对象,配合 ReferenceQueue 做"已被回收"通知(极少用)
val phantom: PhantomReference<Object> = PhantomReference(obj, queue)
```

| 引用 | 何时 GC | 用例 |
| --- | --- | --- |
| 强引用 | 永不(直到无引用) | 默认 |
| SoftReference | 内存紧张 | 缓存(有就用,没就重建) |
| WeakReference | 下次 GC | 不阻止 GC 的引用(典型:Handler 持有 Activity) |
| PhantomReference | + ReferenceQueue | 资源清理通知(极少) |

**Android 上 SoftReference 不太可靠**——ART 倾向积极回收,SoftReference 经常没到内存紧张就被收。**缓存用 `LruCache`(`androidx.collection`)更靠谱**。

## 三、什么是内存泄漏

**内存泄漏**:**对象不该再活了,但仍被某个引用链持有,GC 不掉**。

最经典:**Activity 销毁后仍被持有**。

```kotlin
class MainActivity : ComponentActivity() {
    companion object {
        lateinit var instance: MainActivity         // ❌ static 持有 Activity
    }
    
    override fun onCreate(b: Bundle?) {
        super.onCreate(b)
        instance = this        // 配置变化销毁旧 Activity,但 static 还指着,旧 Activity 永远活着
    }
}
```

每次屏幕旋转,旧 Activity 不释放——多旋转几次,内存就涨。

## 四、Activity 泄漏的常见来源

```kotlin
// 1. static 字段直接持有 Activity
companion object { lateinit var instance: MainActivity }

// 2. Handler 隐式持有 Activity
val handler = Handler(Looper.getMainLooper())
handler.postDelayed({ updateUi() }, 10000)   // lambda 隐式持有 Activity,10 秒内 Activity 关了也没用

// 3. 监听器没解注册
sensorManager.registerListener(this, ...)
// onDestroy 没 unregister → SensorManager 持有 Activity

// 4. 单例持有 Activity 的 Context
class Singleton(val context: Context)
Singleton(activity)                              // 应当 activity.applicationContext

// 5. 内部类持有外部 Activity
class MyView(context: Context) : View(context) {
    inner class Helper                            // 隐式持有 MyView,MyView 持有 Activity Context
}

// 6. 异步任务 / 协程作用域错
GlobalScope.launch { delay(10000); updateUi() }   // GlobalScope 没生命周期,Activity 持有泄漏
```

## 五、Bitmap 是内存大户

```
1张 1080×1920 ARGB_8888 图 = 1080 × 1920 × 4 bytes = ~8 MB
1张 4K(3840×2160)ARGB_8888 = ~33 MB
```

**App 总内存 Cap 通常 256-512MB**——几张大 Bitmap 就到极限。

**优化**:

- 加载时按 View 大小降采样(`BitmapFactory.Options.inSampleSize`)
- 用 RGB_565 而不是 ARGB_8888(省一半)
- 用 Coil / Glide 库自动管理
- 列表里用 `RecyclerView` / `LazyColumn` 自动回收不可见 item

**Bitmap 泄漏特别贵**——一个 Activity 持有 10 张 4K Bitmap 就 300MB。

## 六、LeakCanary:泄漏检测自动化

```kotlin
debugImplementation("com.squareup.leakcanary:leakcanary-android:2.14")
```

**就这一行,什么都不写**。LeakCanary 自动:

1. **Hook Activity / Fragment 的销毁回调**——监听 `onDestroy`
2. **创建 WeakReference 指向已销毁的对象** + 加入 ReferenceQueue
3. **5 秒后 GC,检查 WeakReference 是否还在**(在意味着还有强引用 → 泄漏)
4. **如果泄漏,dump heap 分析引用链**
5. **弹通知告诉你"X 类泄漏了,引用链是 A → B → C → X"**

**LeakCanary 是 Android 开发的标配**——debug build 默认开,几乎零配置。

## 七、LeakCanary 的工作原理

```kotlin
// 简化伪代码
class LeakCanary {
    val watchedRefs = mutableMapOf<UUID, WeakReference<Any>>()
    val queue = ReferenceQueue<Any>()
    
    fun watch(obj: Any, name: String) {
        watchedRefs[UUID.randomUUID()] = WeakReference(obj, queue)
    }
    
    fun checkLeaks() {
        // 等几秒 + 触发 GC
        Thread.sleep(5000)
        System.gc()
        Thread.sleep(1000)
        
        // 检查 ReferenceQueue:已被回收的对象会在 queue
        // 没在 queue 的 watchedRefs 还活着 → 泄漏
        for ((id, ref) in watchedRefs) {
            if (ref.get() != null) {
                // 泄漏!dump heap 分析
                analyzeHeap()
            }
        }
    }
}
```

**关键技术**:**WeakReference + ReferenceQueue**——这是 Java 内置的"对象被 GC 通知"机制。LeakCanary 是它的工程化包装。

## 八、定位泄漏路径

LeakCanary 报告类似:

```
HEAP ANALYSIS RESULT
====================
1 APPLICATION LEAKS

┬───
│ GC Root: System class
│
├─ android.provider.Settings$SettingNotFoundException class
│    Leaking: NO (a class is never leaking)
│    ↓ static SettingNotFoundException.sLastInstance
│                                                                                      
├─ com.notedx.MainActivity instance
│    Leaking: YES (Activity#mDestroyed is true)
│    Retaining 8.2 MB in 12345 objects
│    key = ...
```

**读法**:从 GC Root 开始,沿引用链找到泄漏对象——上面例子是某个静态字段持有 MainActivity。

**修复**:断开这条引用链——不要让 static 持有 Activity。

## 九、Profiler:Android Studio 内存监控

```
Profile or Debug APK → Memory
   ↓
看 Java Heap 实时占用
   ↓
点 "Take heap dump"(垃圾桶图标)
   ↓
看 .hprof,按 Class 排序内存占用
   ↓
找异常多的实例(比如 MainActivity 应该只有 1 个,显示 5 个)
   ↓
Profiler 显示这个实例的引用链
```

**Profiler 操作流程**:

1. 反复触发 Activity 销毁重建(旋转屏幕 10 次)
2. Dump heap
3. 看 MainActivity 实例数——如果 > 1 就是泄漏
4. 选可疑实例,看 "References" 面板,找到 Root 引用链

**比 LeakCanary 更底层**——LeakCanary 只看常见 Activity / Fragment,Profiler 能看任何对象。

## 十、`MAT`(Memory Analyzer Tool)

更专业的工具:Eclipse 的 MAT。打开 .hprof:

- **Histogram**:按类排序,看实例数 / 总占用
- **Dominator Tree**:谁"主导"内存(去掉它能省多少)

## 十一、典型泄漏模式 + 修法

### 模式 1:Handler 隐式持有 Activity

```kotlin
class MainActivity : ComponentActivity() {
    val handler = Handler(Looper.getMainLooper())
    
    override fun onCreate(...) {
        handler.postDelayed({ updateUi() }, 10000)
    }
}
```

**修法**:

```kotlin
override fun onDestroy() {
    super.onDestroy()
    handler.removeCallbacksAndMessages(null)
}
```

或者用 lifecycleScope.launch:

```kotlin
lifecycleScope.launch {
    delay(10000)
    updateUi()
}
// Activity 销毁自动取消
```

### 模式 2:static 单例持有 Context

```kotlin
class AnalyticsTracker(val context: Context)
object Singleton { val tracker = AnalyticsTracker(MainActivity.instance) }   // ❌
```

**修法**:用 ApplicationContext:

```kotlin
class AnalyticsTracker private constructor(val appContext: Context) {
    companion object {
        @Volatile private var instance: AnalyticsTracker? = null
        fun get(ctx: Context): AnalyticsTracker = instance ?: synchronized(this) {
            instance ?: AnalyticsTracker(ctx.applicationContext).also { instance = it }
        }
    }
}
```

或者 Hilt `@ApplicationContext` 注入(12 篇)。

### 模式 3:监听器没解注册

```kotlin
override fun onCreate(...) {
    sensorManager.registerListener(this, sensor, SENSOR_DELAY_NORMAL)
}
// 漏 unregister → SensorManager 持有 Activity
```

**修法**:`onDestroy` 里 unregister,或者用 `LifecycleObserver`:

```kotlin
override fun onStart() {
    super.onStart()
    sensorManager.registerListener(this, sensor, SENSOR_DELAY_NORMAL)
}
override fun onStop() {
    super.onStop()
    sensorManager.unregisterListener(this)
}
```

### 模式 4:RxJava Disposable 不释放

```kotlin
observable.subscribe(...)    // 没 dispose,RxJava 链持有 Activity
```

**修法**:

```kotlin
val disposable = observable.subscribe(...)
// onDestroy:
disposable.dispose()
```

或者用 `CompositeDisposable` + AutoDispose 库。

**RxJava 项目的泄漏占了所有泄漏的一大半**——管理 disposable 极其麻烦,这也是协程替代 RxJava 的原因之一。

### 模式 5:协程作用域错

```kotlin
GlobalScope.launch { ... }      // ❌ 没生命周期
```

**修法**:`viewModelScope` / `lifecycleScope`:

```kotlin
lifecycleScope.launch { ... }
```

### 模式 6:Fragment Binding 不置 null

```kotlin
private var _binding: FragmentHomeBinding? = null
// 漏:override fun onDestroyView() { super.onDestroyView(); _binding = null }
```

**修法**:置 null。

### 模式 7:WebView 泄漏

```kotlin
// WebView 持有 Context,Activity 销毁后 WebView 内部状态可能仍持有
override fun onDestroy() {
    webView.destroy()       // 必须显式 destroy
    (webView.parent as? ViewGroup)?.removeView(webView)
    super.onDestroy()
}
```

WebView 是已知的泄漏大户——内部线程 / 子进程 / JavaScript 上下文都可能持有 Activity。

## 十三、Linux `/proc/<pid>/` 信息

```bash
# 看进程的虚拟内存映射
adb shell cat /proc/$(adb shell pidof com.notedx)/maps

# 看实际占用(VmRSS = 物理内存,VmSize = 虚拟内存)

```

文件描述符泄漏(Cursor / 文件流不关)→ 几小时 ULIMIT 用满 → 所有 IO 失败。

## 十四、ART GC 触发与 STW

ART 默认是**并发 mark-sweep** + 偶尔 compaction。STW(Stop-The-World)时间通常 < 5ms,但在大堆 / 大量短期对象时会更长。

**GC 触发原因**:

| 原因 | 日志关键字 |
| --- | --- |
| 分配引发(堆耗尽) | `Alloc concurrent copying GC` |
| 后台 GC | `Background concurrent copying GC` |
| 显式 `System.gc()` | `Explicit concurrent copying GC` |
| 进程切后台 | `CollectorTransition` |
| 内存紧张通知 | `HeapTrim` |

`adb shell setprop dalvik.vm.checkjni true` 加调试模式;`logcat | grep -E "GC|art:"` 看 GC 日志。

**正常 App**:每分钟几次 GC,每次 < 10ms,可忽略;**异常**:每秒多次 GC + 数十毫秒 STW,直接卡。

## 十六、OOM 与 Low Memory Killer

**OOM**(Out of Memory)——App 自己堆耗尽,抛 `OutOfMemoryError`。这是 App 内部问题,代码可控。

**LMK**(Low Memory Killer)——系统级内存紧张,系统选 oom_score 最高的进程 kill。这是 Android 层面,App 不可控,只能减少自己被杀概率(前台 / Foreground Service / 内存占用小)。

**两者的区别**:
- OOM:**你 App 内存涨爆**,代码层修
- LMK:**整个系统紧张**,可能因为其他 App 内存占用大,或者你 App 处于后台

## 十七、内存优化原则

1. **不持有不该持有的引用**——尤其 Activity / Context
2. **生命周期对齐**——监听器 / 协程 scope / disposable 用 lifecycle-aware API
3. **Bitmap 谨慎**——按尺寸加载,RGB_565 优先,放 Coil 等库管理
4. **缓存有上限**——`LruCache` 比 `HashMap` 安全
5. **大对象用完释放**——`Bitmap.recycle()` / `cursor.close()` / `inputStream.close()`(都用 `.use {}` 自动)
6. **谨慎用 static**——static 字段几乎不应持有任何带 Context 的对象

## 十八、踩坑

**坑 1:`MainActivity.instance` 静态字段**。最经典 Activity 泄漏。

**坑 2:Handler.postDelayed + Activity 引用**。Activity 关了但 message 还在 queue 里。

**坑 3:监听器不 unregister**。SensorManager / LocationManager / 各种回调注册了不解。

**坑 4:Bitmap 加载不缩放**。1080 widget 加载 4000×3000 原图,直接 OOM。

**坑 5:用 SoftReference 当缓存**。ART 倾向积极回收,SoftReference 不可靠。用 LruCache。

**坑 6:Cursor 不 close**。文件描述符泄漏。

**坑 7:WebView 不 destroy**。WebView 内部状态泄漏复杂,Activity 销毁前必须显式 destroy。

**坑 8:用 `GlobalScope.launch` 启动协程**。没生命周期,泄漏。

**坑 9:LiveData / Flow 在 Fragment 用 `this` 而不是 `viewLifecycleOwner`**。Fragment View 销毁但订阅还在,引用泄漏。

**坑 10:`System.gc()` 当救命药**。ART 内部 GC 比你聪明,显式 gc 几乎从不需要。
