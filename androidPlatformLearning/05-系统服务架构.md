# 系统服务架构:SystemServer / AMS / WMS / PMS

> 一句话:**Android 的"系统"是一个叫 `system_server` 的进程,里面跑了几十个 Binder 服务——AMS 管 Activity / Service / Process,WMS 管窗口,PMS 管 App 安装,InputManager 管触摸**。你 App 的每个动作背后都有它们的身影。

---

## 一、`system_server`:一个进程,几十个服务

```bash
adb shell ps -A | grep system_server
# system    1234  789  9012345  234567   sys_epoll_wait  0 S system_server
```

**一个进程,UID = system(1000)**。这个进程里启动了几十个系统服务,每个服务都是 Binder 接口,通过 `ServiceManager` 注册名字让 App 找到。

```bash
adb shell service list | head -30
# 输出:
# 1   activity: [android.app.IActivityManager]
# 2   activity_task: [android.app.IActivityTaskManager]
# 3   package: [android.content.pm.IPackageManager]
# 4   window: [android.view.IWindowManager]
# 5   power: [android.os.IPowerManager]
# 6   input: [android.hardware.input.IInputManager]
# 7   notification: [android.app.INotificationManager]
# 8   audio: [android.media.IAudioService]
# 9   alarm: [android.app.IAlarmManager]
# ... 100+ 个
```

每行是一个 Binder 服务:**服务名 + 接口类名**。`getSystemService(...)` 拿到的就是这些。

**关键认识**:**所有这些服务都跑在同一个 `system_server` 进程里**——这意味着 system_server 一旦崩溃,整个手机重启(各家厂商也有用独立进程的趋势,但主体仍然是 system_server)。

---

## 二、几个最重要的系统服务

### ActivityManagerService(AMS)

**职责**:编排 Activity / Service / Receiver / Provider / Process 的生命周期。

- 你 `startActivity(intent)` → AMS 决定哪个 App 处理、是否启动新进程、是否复用栈
- 你 `startService(intent)` → AMS 决定调 onStartCommand 或者拒绝(API 26+ 后台限制)
- 系统内存紧张 → AMS 决定杀哪个 App
- 用户按 Home → AMS 把当前 Activity onPause / onStop

AMS 持有的关键数据:

```
AMS
├── ActivityRecord 列表       ← 所有 Activity 的状态
├── ServiceRecord 列表        ← 所有 Service 的状态
├── ProcessRecord 列表        ← 所有 App 进程的状态(oom_score 等)
├── TaskRecord 列表           ← 所有任务栈
└── BroadcastQueue            ← 待派发的广播
```

App 进程的 `ActivityThread` 通过 Binder 接口 `IApplicationThread` 接收 AMS 命令——AMS 说"创建 MainActivity",ActivityThread 反射构造并调 onCreate。

### WindowManagerService(WMS)

**职责**:管理屏幕上所有窗口的层次、大小、动画。

每个 Activity 不只是一个生命周期对象,它**还对应一个窗口**(Window)。Window 注册到 WMS,WMS 决定窗口在屏幕上的位置(top bar / 主体 / IME / overlay 各占哪)。

- 你 setContentView → 创建 View 根挂到 Window 上 → WMS 添加这个 Window
- 屏幕旋转 → WMS 通知 Activity onConfigurationChanged 或者重建
- 状态栏显示/隐藏 → WMS 重排
- Toast / Dialog / DropdownMenu → 都是 Window,都注册到 WMS

WMS 与 **SurfaceFlinger**(独立进程)配合——WMS 决定"在哪",SurfaceFlinger 把多个 Surface 合成成最终屏幕显示。

### PackageManagerService(PMS)

**职责**:App 安装、卸载、查询、权限管理。

- `adb install` → PMS 解析 APK Manifest、验证签名、分配 UID
- `PackageManager.queryIntentActivities` → PMS 查所有声明对应 intent-filter 的 Activity
- 权限授予 → PMS 写 `/data/system/packages.xml`

PMS 维护:

```
/data/system/packages.xml        # 持久化的包数据库
/data/system/packages-stopped.xml # force-stop 状态
```

### NotificationManagerService

**职责**:管理所有 App 的通知,显示在状态栏 / 通知抽屉。

App 调 `notify` 走 Binder 到这里,服务再交给 SystemUI(独立 App)显示。

### InputManagerService

**职责**:从内核读触摸 / 键盘事件,派发到当前 focus 窗口。

触屏 → 内核驱动 → InputManager → 找到当前 focus Window → 通过 InputChannel(pipe + Binder)发给对应 App 进程的 ViewRoot → ViewRoot 派发到 View 树。

### PowerManagerService

**职责**:屏幕开关、唤醒锁、电源管理。

`wakeLock.acquire()` 阻止 CPU 睡眠;`PowerManager.IGNORE_BATTERY_OPTIMIZATIONS` 申请电池白名单。

## 三、AMS 编排 Activity 启动的完整流程

```
用户点击 Home 上的 NotedX 图标
   ↓
Launcher 调 startActivity(Intent("MAIN", LAUNCHER))
   ↓
Launcher 进程 → Binder → AMS
   ↓
AMS:
   1. PMS 查 manifest:LAUNCHER 对应 com.notedx/.MainActivity
   2. 检查 com.notedx 进程是否存在
      - 不存在:让 Zygote fork 新进程
      - 存在:复用
   3. 让进程内 ActivityThread 创建 MainActivity 实例
   4. 在 NotedX 的 Task 栈上压 MainActivity
   5. WMS 创建对应 Window
   ↓
AMS → Binder → NotedX 进程 ActivityThread:
   "你创建 MainActivity"
   ↓
ActivityThread:
   1. ClassLoader 加载 MainActivity 类
   2. 反射 newInstance() 创建实例
   3. 调 attach(...) 注入 Context
   4. 调 onCreate(bundle)
   5. setContentView 把 View 树挂到 Window
   6. 调 onStart / onResume
   ↓
WMS 把 Window 显示到屏幕
SurfaceFlinger 合成
GPU 绘制
```

**整个流程涉及至少 3 个进程**:Launcher / system_server / 新 fork 的 NotedX 进程。中间有十几次 Binder 调用。

## 四、`getSystemService` 内部

```kotlin
val nm = ctx.getSystemService(NotificationManager::class.java)
```

内部:

```java
// ContextImpl.java
public Object getSystemService(String name) {
    return SystemServiceRegistry.getSystemService(this, name);
}

// SystemServiceRegistry.java
// 内部维护一个 ServiceFetcher 映射
// "notification" → returns new NotificationManager(this, getNotificationManagerService())

private static INotificationManager getNotificationManagerService() {
    IBinder b = ServiceManager.getService("notification");
    return INotificationManager.Stub.asInterface(b);
}
```

**`NotificationManager` 不是系统服务本身**——它是**包装器**,内部持有 `INotificationManager` 这个 Binder 代理,把 Java API 转成 Binder 调用。

所有 `Manager` 类都是这种模式:`ActivityManager` 包装 `IActivityManager`,`WindowManager` 包装 `IWindowManager`,以此类推。

## 五、`Manager` 缓存:为什么 `getSystemService` 不慢

```java
// ContextImpl 里
private final ArrayMap<String, Object> mServiceCache = ...;

public Object getSystemService(String name) {
    Object cached = mServiceCache.get(name);
    if (cached != null) return cached;
    Object service = SystemServiceRegistry.getSystemService(this, name);
    mServiceCache.put(name, service);
    return service;
}
```

`Manager` 对象在 Context 内**单例缓存**——同一个 Context 拿同一个 `NotificationManager`,不重复创建。**但跨 Context 不共享缓存**。

## 六、`dumpsys`:看系统服务内部状态

`dumpsys` 命令让你查任意系统服务的当前状态:

```bash
# Activity Manager 完整状态
adb shell dumpsys activity

# 当前 Activity 栈
adb shell dumpsys activity activities

# 所有 Service
adb shell dumpsys activity services

# 当前广播
adb shell dumpsys activity broadcasts

# Window
adb shell dumpsys window | head -50

# Package 信息
adb shell dumpsys package com.notedx

# 通知
adb shell dumpsys notification

# 内存
adb shell dumpsys meminfo com.notedx

# 电量
adb shell dumpsys batterystats com.notedx
```

**这套命令是 Android 工程师的最常用工具**——任何"为什么 Activity 没启动 / 通知没显示 / Service 挂了"问题第一步就是 dumpsys 看状态。

## 七、AMS 的关键源码位置

AOSP 里 AMS 实现:

```
frameworks/base/services/core/java/com/android/server/am/
├── ActivityManagerService.java        ← 主服务
├── ActivityStack.java                  ← Activity 栈管理
├── ProcessRecord.java                  ← App 进程描述
├── ServiceRecord.java                  ← Service 描述
└── BroadcastQueue.java                 ← 广播队列
```

读 AMS 源码是理解 Android 行为差异的不二法门——为什么 `bindService` 在某些时刻失败、为什么 `startActivity` 没拉起、AMS 怎么决定杀进程,都在这里。

## 八、`Looper.loop()`:system_server 也是事件循环

`system_server` 进程内每个服务**通常都跑在自己的 Handler/Looper 上**——AMS 有 ActivityManagerThread / ActivityStackSupervisor 的 Handler;WMS 有 UI thread / Animation thread / Render thread。

**system_server 内部的并发**:

```
system_server (PID 1234)
├── main thread (ATM / AMS 主线程)
├── android.bg thread (后台任务)
├── android.fg thread (前台任务)
├── android.ui thread (UI 相关)
├── PowerManagerService thread
├── ...
```

Binder 线程池 + 各服务自己的 Handler 线程——**system_server 是一个高度并发的多线程进程**。这也是为什么"系统服务出 bug 会导致整个手机不稳定"——内部锁太多,死锁、竞态都有可能。

---

## 九、`SystemUI`:独立的系统 App

注意 **SystemUI** 不在 system_server 进程内——它是一个独立的系统 App(`com.android.systemui`),进程独立。

SystemUI 负责:
- 状态栏(顶部)
- 通知抽屉
- 导航栏(底部三按钮 / 手势条)
- 最近任务
- 锁屏
- 音量调节弹窗
- Quick Settings

它通过 Binder 接收 NotificationManagerService、WMS 的消息,渲染对应 UI。**用户能看到的"系统 UI" 几乎全在 SystemUI 这个 App 内**。

## 十、系统进程之间的协作

启动一个 Activity 涉及的进程协作:

```
Launcher 进程
   │ Binder
   ▼
system_server (AMS, WMS, PMS)
   │ Binder
   ▼
App 进程 (NotedX)
   │ Surface
   ▼
SurfaceFlinger 进程
   │ HW Composer
   ▼
GPU / 屏幕
```

每一步都是跨进程通信。理解这条路径,你能看懂任何"Activity 启动慢"的 trace——慢在哪一段是哪个进程的问题。

## 十一、`Activity.runOnUiThread` 实际上是什么

```kotlin
runOnUiThread { ... }
```

源码:

```java
public final void runOnUiThread(Runnable action) {
    if (Thread.currentThread() != mUiThread) {
        mHandler.post(action);            // 跨线程发消息到主线程
    } else {
        action.run();                      // 已经在主线程,直接跑
    }
}
```

**`runOnUiThread` 本质就是 `Handler.post` 到主线程 Handler**。12 篇会专门讲 Handler / Looper。

## 十二、为什么"调系统服务"可能阻塞

```kotlin
val info = packageManager.getApplicationInfo("com.notedx", 0)
```

这一行**跨进程到 system_server**,且 PMS 用一把全局锁——如果 system_server 正忙(比如某个 App 刚崩溃 AMS 正在清理),你这一行可能等几百毫秒。

主线程上调用 PMS / WMS / AMS 是 ANR 高发场景。**生产代码**:任何高频 / 关键路径上的系统服务调用,都应该考虑缓存或者异步。

## 十四、踩坑

**坑 1:在 ContentProvider.onCreate 调系统服务**。Provider 是进程启动早期被创建的,**比 Application.onCreate 还早**。这时候 PMS / AMS 部分服务可能还没就绪——`getSystemService` 可能返回 null。

**坑 2:把系统服务对象保存到 static 变量**。`Manager` 类持有 Context,持有 ApplicationContext 没事,持有 Activity Context 泄漏。`getSystemService(Activity.class)` 是不能跨 Activity 复用的。

**坑 3:多线程调 PackageManager 时遇到 RemoteException**。system_server 重启(罕见但发生)时 Binder 断,App 进程拿着的 Manager 对象失效。RemoteException 之后**需要重新 `getSystemService`** 拿新对象。

**坑 4:`dumpsys` 输出当稳定 API**。`dumpsys` 输出格式版本间会变,不要用脚本解析它的输出做自动化判断。是调试工具,不是 API。

**坑 5:认为系统服务都是同一个进程能共享内存**。**全部是 Binder,即便它们都在 system_server 进程内,也通过 Binder 互调**——AMS 让 WMS 加窗口,内部也是 Binder。这是 Framework 一致性的设计代价。

**坑 6:`linkToDeath` 监听 system_server 死亡**。system_server 死了等于手机重启,你的 App 也跟着死了——你**永远收不到**这个 DeathRecipient 通知。`linkToDeath` 只对你 bind 的"别的 App" / "你自己另一个进程"有意义。

**坑 7:认为 Activity 关闭 = WMS 立刻回收 Window**。Window 销毁是异步的,Activity onDestroy 后 WMS 可能还要几帧才完成。这是为什么"快速 startActivity → finish → startActivity"有时第二个 Activity 会闪一下旧画面。

**坑 8:依赖 system_server 的服务永远存在**。Android 13+ 起,SystemUI 等服务在锁屏状态可能没启动完;某些设备 OEM 改造也可能裁掉服务。`getSystemService` 必须做 null 检查。
