# 旧式存储:SharedPreferences、文件、SQLite

> 一句话:**Room / DataStore 是 Android 现代存储的封装层,底层永远是 SQLite + 文件 + SharedPreferences**。理解这一层,你能读懂老代码、诊断"存储很慢"的根因、知道为什么 DataStore 取代 SharedPreferences、为什么 Room 取代 SQLiteOpenHelper。

---

## 一、Android 存储的物理图

```
/data/data/com.notedx/
├── files/                # Context.filesDir,持久内部文件
├── cache/                # Context.cacheDir,系统紧张时可能清掉
├── databases/            # SQLite 数据库
│   └── notedx.db
├── shared_prefs/         # SharedPreferences
│   └── user_settings.xml
├── code_cache/           # Context.codeCacheDir,DEX 解压
└── no_backup/            # 不参与自动备份

/storage/emulated/0/Android/data/com.notedx/files/    # 外部存储应用专属
```

| 区域 | 权限 | 卸载时 | 用户可见 |
| --- | --- | --- | --- |
| 内部 `/data/data/` | 700,仅自己 UID | 清空 | ❌ |
| 外部应用专属 | 自己可读写,他 App 需权限 | 清空 | ❌(API 11+ 一般是) |
| 公共目录(/sdcard/Download) | 需 MANAGE_EXTERNAL_STORAGE 等 | 不清 | ✅ |

## 二、`SharedPreferences`:键值存储

```kotlin
val prefs = ctx.getSharedPreferences("user_settings", Context.MODE_PRIVATE)

// 写
prefs.edit().apply {
    putString("theme", "dark")
    putInt("notification_hour", 8)
    putBoolean("sync_enabled", true)
    apply()           // 异步写入磁盘(推荐)
    // 或 commit()  // 同步写入,返回 boolean(阻塞主线程!)
}

// 读
val theme = prefs.getString("theme", "system") ?: "system"
val hour = prefs.getInt("notification_hour", 9)
```

**底层物理实现**:

```
/data/data/com.notedx/shared_prefs/user_settings.xml
```

```xml
<?xml version='1.0' encoding='utf-8' standalone='yes' ?>
<map>
    <string name="theme">dark</string>
    <int name="notification_hour" value="8" />
    <boolean name="sync_enabled" value="true" />
</map>
```

**就是一个 XML 文件**。整个 SharedPreferences 内存里是个 Map,任何 edit 都把整个 Map 写回这个 XML 文件——**这是性能问题的根**。

---

## 三、`commit` vs `apply`:同步 vs 异步

```kotlin
prefs.edit().putBoolean("flag", true).commit()    // 同步,主线程上阻塞 IO
prefs.edit().putBoolean("flag", true).apply()     // 异步,立即返回,后台写盘
```

| 方法 | 阻塞 | 返回值 |
| --- | --- | --- |
| `commit()` | 是(磁盘 IO 等几十毫秒) | Boolean(成功 / 失败) |
| `apply()` | 否 | void(永远算成功) |

**`commit` 在主线程是 ANR 高发**——首次访问 SharedPreferences 还会触发首次加载(磁盘 + XML 解析,可能 50-200ms)。

**实操**:**永远用 `apply()`,除非真需要同步知道结果**(几乎从不需要)。

但 **`apply` 在 Activity onPause 后强制刷盘**——`apply` 的"异步"只是延后,系统不会丢数据。

---

## 四、SharedPreferences 的隐藏 IO 时机

很多人不知道——**首次访问 SharedPreferences 会触发同步磁盘 IO**:

```kotlin
override fun onCreate(b: Bundle?) {
    super.onCreate(b)
    val prefs = getSharedPreferences("x", MODE_PRIVATE)
    val flag = prefs.getBoolean("k", false)   // ← 首次访问触发加载
}
```

这第一次 `getXxx` 会**同步**等到整个 XML 文件读完 + 解析。主线程上 50-300ms 不稀奇,尤其在低端机或 prefs 文件大时。

**优化**:Application.onCreate 里**提前异步触发加载**:

```kotlin
override fun onCreate() {
    super.onCreate()
    Thread { getSharedPreferences("x", MODE_PRIVATE) }.start()    // 后台触发加载
}
```

或者直接迁 DataStore。

## 五、SharedPreferences 监听变化

```kotlin
val listener = SharedPreferences.OnSharedPreferenceChangeListener { _, key ->
    if (key == "theme") {
        applyNewTheme()
    }
}
prefs.registerOnSharedPreferenceChangeListener(listener)

// 必须解注册
prefs.unregisterOnSharedPreferenceChangeListener(listener)
```

**坑**:`registerOnSharedPreferenceChangeListener` **持有 listener 的 WeakReference**——如果你给它一个 lambda 或匿名内部类,可能立即被 GC,listener 失效。**永远用强引用变量**:

```kotlin
private val listener = SharedPreferences.OnSharedPreferenceChangeListener { ... }
// 字段持有 → 不会被 GC
```

## 六、SharedPreferences 的现代替代:DataStore

DataStore(`androidx.datastore`)是 Google 推荐的替代,**解决 SharedPreferences 几乎所有问题**:

| 维度 | SharedPreferences | DataStore Preferences |
| --- | --- | --- |
| API | 同步阻塞 | 协程 / Flow |
| 主线程 IO | 容易踩坑 | 不可能(全 suspend) |
| 类型安全 | 不安全(string key) | 类型化 key |
| 事务 | 无 | 自动 |
| 文件格式 | XML | Protobuf(更高效) |
| 数据丢失 | 偶发(crash 时未刷盘) | 通过事务防护 |

**新代码全部用 DataStore**(现代版 13 篇)。SharedPreferences 仅在维护老代码时用。

## 七、文件 IO

```kotlin
// 内部文件(自己 App 私有)
val file = File(ctx.filesDir, "draft.txt")
file.writeText("hello")
val text = file.readText()

// 缓存(可能被系统清掉)
val cache = File(ctx.cacheDir, "cached_data")

// 外部应用专属
val externalFile = File(ctx.getExternalFilesDir(null), "log.txt")
```

**`File.readText` / `writeText` 是同步的**——主线程上一定要避免。用协程 IO 包装:

```kotlin
suspend fun saveDraft(content: String) = withContext(Dispatchers.IO) {
    File(ctx.filesDir, "draft.txt").writeText(content)
}
```

**API 24+** 提供 `Files.write` / `Files.readString` 等 NIO API,等价用法。

---

## 八、文件路径常量

| 函数 | 返回 | 用途 |
| --- | --- | --- |
| `Context.filesDir` | `/data/data/<pkg>/files` | 持久私有 |
| `Context.cacheDir` | `/data/data/<pkg>/cache` | 缓存(可清) |
| `Context.dataDir` | `/data/data/<pkg>` | App 数据根 |
| `Context.codeCacheDir` | `/data/data/<pkg>/code_cache` | dex 缓存 |
| `Context.getExternalFilesDir(null)` | `/storage/emulated/0/Android/data/<pkg>/files` | 外部私有 |
| `Context.getExternalCacheDir()` | `/storage/emulated/0/Android/data/<pkg>/cache` | 外部缓存 |
| `Environment.getExternalStorageDirectory()` | `/storage/emulated/0` | 公共(API 29+ 受限) |

**选择**:

- **敏感数据(用户凭证、私密笔记)** → `filesDir`(内部)
- **可重建缓存(图片缩略图)** → `cacheDir`(系统紧张时被清是正常的)
- **较大但应用专属(下载附件)** → `getExternalFilesDir(null)`(外部不占内部空间)
- **公共可见(下载到 Downloads)** → MediaStore + Downloads collection(11 篇)

## 九、文件锁与并发

```kotlin
val lock = ReentrantLock()

fun writeNote(content: String) = lock.withLock {
    File(filesDir, "draft.txt").writeText(content)
}
```

**多线程并发写同一文件**——必须加锁,否则后写覆盖前写,或者写一半被读到。

**协程的等价**:`Mutex` + `withLock`。

文件级锁(跨进程):`FileChannel.lock()` 加 OS 文件锁,但跨进程 Android 上极少用。

## 十、SQLite:Android 自带数据库

```kotlin
class NoteDbHelper(context: Context) : SQLiteOpenHelper(
    context, "notedx.db", null, DB_VERSION
) {
    override fun onCreate(db: SQLiteDatabase) {
        db.execSQL("""
            CREATE TABLE note (
                id INTEGER PRIMARY KEY AUTOINCREMENT,
                title TEXT NOT NULL,
                content TEXT NOT NULL,
                created_at INTEGER NOT NULL
            )
        """)
        db.execSQL("CREATE INDEX idx_note_created ON note(created_at)")
    }
    
    override fun onUpgrade(db: SQLiteDatabase, oldVersion: Int, newVersion: Int) {
        if (oldVersion < 2) {
            db.execSQL("ALTER TABLE note ADD COLUMN archived INTEGER DEFAULT 0")
        }
        // 后续版本累计加
    }
    
    companion object { const val DB_VERSION = 2 }
}
```

使用:

```kotlin
val helper = NoteDbHelper(ctx)
val db = helper.writableDatabase

// 插入
val values = ContentValues().apply {
    put("title", "First note")
    put("content", "...")
    put("created_at", System.currentTimeMillis())
}
val id = db.insert("note", null, values)

// 查询
val cursor = db.query(
    "note",
    arrayOf("id", "title"),
    "archived = ?", arrayOf("0"),
    null, null,
    "created_at DESC"
)
cursor.use {
    while (it.moveToNext()) {
        val id = it.getLong(0)
        val title = it.getString(1)
    }
}

// 更新
db.update("note", ContentValues().apply { put("title", "Updated") }, "id = ?", arrayOf("42"))

// 删除
db.delete("note", "id = ?", arrayOf("42"))
```

## 十一、`SQLiteOpenHelper` vs Room

**老代码**(几乎所有 2018 之前的项目)用 SQLiteOpenHelper。**问题**:

1. **SQL 字符串**——拼错 / 列名错只能运行时发现
2. **Cursor 取列要写一堆 `getColumnIndex` + `getString`/`getLong`**
3. **没有 Flow / 协程支持**——回调或同步
4. **多线程并发**——每个线程都要拿 `writableDatabase`,容易死锁
5. **迁移**——`onUpgrade` 里多版本累积,代码很乱

**Room**(现代版 13 篇)是 SQLiteOpenHelper 的注解封装:

```kotlin
@Dao
interface NoteDao {
    @Query("SELECT * FROM note WHERE archived = 0 ORDER BY created_at DESC")
    fun observeAll(): Flow<List<NoteEntity>>
}
```

编译期检查 SQL、自动类型映射、Flow 订阅、协程友好——SQLiteOpenHelper 几乎被完全淘汰。**新代码用 Room**。

---

## 十二、SQLite 的 WAL 模式

```kotlin
db.enableWriteAheadLogging()
```

**Write-Ahead Logging**:写入先到 WAL 文件,后异步合并到主库——**读不阻塞写,写不阻塞读**,并发性能大幅提升。

**Room 默认开 WAL**——你不需要手动配。

老代码用 SQLiteOpenHelper 可以手动开,显著提升并发场景性能。

## 十三、数据库迁移

```kotlin
override fun onUpgrade(db: SQLiteDatabase, oldVersion: Int, newVersion: Int) {
    // 累积迁移
    if (oldVersion < 2) {
        db.execSQL("ALTER TABLE note ADD COLUMN archived INTEGER DEFAULT 0")
    }
    if (oldVersion < 3) {
        db.execSQL("ALTER TABLE note ADD COLUMN color INTEGER DEFAULT 0")
    }
    if (oldVersion < 4) {
        // 复杂迁移:加索引
        db.execSQL("CREATE INDEX idx_color ON note(color)")
    }
}
```

**铁律**:

1. **每次改 schema 都升 DB_VERSION**
2. **`if (oldVersion < N)` 累积应用,不能用 `else if`**——用户可能从 v1 直接升到 v4
3. **永远只 ALTER,不 DROP**——DROP 丢用户数据,卸载重装才能复原
4. **测试迁移**——用真实 v1 数据跑迁移到 v4,看数据完整

Room 把这事更结构化(`Migration` 对象 + `MigrationTestHelper` 测试)。

## 十四、`ContentValues` 与 SQL 注入

```kotlin
// ✅ 用 ContentValues + selectionArgs(参数化)
db.query("note", null, "title = ?", arrayOf(userInput), ...)

// ❌ 拼接 SQL
db.execSQL("SELECT * FROM note WHERE title = '$userInput'")    // SQL 注入!
```

**永远用参数化查询**——`?` + `selectionArgs`,SQLite 自动转义。

Room 强制这件事——所有 `@Query` 必须用 `:paramName` 形式,**SQL 拼接被编译期拒绝**。

---

## 十五、`Cursor` 资源管理

```kotlin
db.query(...).use { cursor ->
    while (cursor.moveToNext()) { /* ... */ }
}
// Cursor 自动 close
```

**Cursor 不 close 会泄漏文件句柄**——Linux 进程文件描述符上限通常 1024。频繁泄漏几小时就 ULIMIT,所有 IO 失败。

Room 隐藏了 Cursor,你不用管。

---

## 十六、`Transaction`:批量操作

```kotlin
db.beginTransaction()
try {
    notes.forEach { db.insert("note", null, it.toContentValues()) }
    db.setTransactionSuccessful()
} finally {
    db.endTransaction()
}
```

**1000 次 insert 不在事务里**:1000 次 fsync,几秒。
**在事务里**:1 次 fsync,几十毫秒。

事务**是 SQLite 性能的关键**。Room 提供 `@Transaction` 注解 + `withTransaction` 协程扩展,默认支持。

## 十七、文件加密:`EncryptedSharedPreferences` / `EncryptedFile`

Jetpack Security 库:

```kotlin
implementation("androidx.security:security-crypto:1.1.0-alpha06")
```

```kotlin
val masterKey = MasterKey.Builder(ctx)
    .setKeyScheme(MasterKey.KeyScheme.AES256_GCM)
    .build()

val prefs = EncryptedSharedPreferences.create(
    ctx, "secret_prefs", masterKey,
    EncryptedSharedPreferences.PrefKeyEncryptionScheme.AES256_SIV,
    EncryptedSharedPreferences.PrefValueEncryptionScheme.AES256_GCM
)
```

**透明加密**:你照常 putString / getString,内部 AES 加密 + 用 Android Keystore 保护 master key。

**用例**:用户凭证、Token、隐私笔记草稿。普通配置不需要。

**EncryptedFile** 类似,加密文件 IO。

## 十八、`backup`:系统自动备份

`AndroidManifest.xml`:

```xml
<application
    android:allowBackup="true"
    android:fullBackupContent="@xml/backup_rules"
    android:dataExtractionRules="@xml/data_extraction_rules">
```

`res/xml/backup_rules.xml`:

```xml
<full-backup-content>
    <include domain="sharedpref" path="user_settings.xml" />
    <exclude domain="database" path="cache.db" />
</full-backup-content>
```

Android 6.0+ 系统自动把 App 数据上传 Google Drive(国际版)/ 用户云(国内厂商)——重装 App 自动恢复。

**默认开启,但敏感数据(token / 凭证)应当排除**。NotedX 的密码 / 加密 prefs 都在 `<exclude>` 里。

## 十九、`no_backup/`:专属"永不备份"目录

```kotlin
val noBackupFile = File(ctx.noBackupFilesDir, "secret.txt")
```

`noBackupFilesDir` 是 `/data/data/<pkg>/no_backup/`——这个目录里的文件**永远不参与备份**。

Firebase Installation ID / 设备唯一标识等"和设备绑定不该跨设备恢复"的东西放这里。

## 二十、踩坑

**坑 1:`commit()` 在主线程**。SharedPreferences.commit() 同步 IO,主线程几十毫秒。**永远用 `apply()`**。

**坑 2:首次访问 SharedPreferences 不预热**。Application.onCreate 没提前触发加载,首屏第一次 getXxx 阻塞。**预热或者迁 DataStore**。

**坑 3:`File.readText` 在主线程**。看似简单的读小文件,主线程上累积起来仍然丢帧。**协程 + Dispatchers.IO**。

**坑 4:数据库迁移忘记某版本**。`if (oldVersion < 2)` 跑了,但 `if (oldVersion < 3)` 漏了——v1 用户升到 v3 直接 schema 不一致 crash。**累积写所有迁移**,不要假设"用户都从 v2 升"。

**坑 5:Cursor 不 close**。`.use { ... }` 自动关。手动写记得 `finally { cursor.close() }`。

**坑 6:多线程操作同一个 SQLiteDatabase 实例**。SQLite 内部有锁,但多线程频繁 begin/commit 性能差。**Room 自带连接池,直接用 Room**。

**坑 7:`writableDatabase` 在多线程下死锁**。两个线程同时拿 writable,第二个等第一个 release——加上事务嵌套很容易死锁。

**坑 8:`MODE_WORLD_READABLE`**。`getSharedPreferences("x", MODE_WORLD_READABLE)` 让其他 App 读你的 prefs——Android 7.0+ 直接抛 SecurityException。**永远 MODE_PRIVATE**。

**坑 9:把大对象塞 SharedPreferences**。整个 prefs 文件每次写都全量重写——1MB 的 JSON 字符串塞 prefs,每次 apply 都 1MB IO。**大数据用 SQLite / Room / 文件**。

**坑 10:`allowBackup=true` + 含敏感数据**。系统把 token 上传 Google Drive,用户换设备恢复后另一个账号身份 → 灾难。**敏感数据明确 `<exclude>`**。
