# 崩溃、ANR、卡顿与日志采集

上线后不能只看"我手机没问题"。移动端质量要靠崩溃、ANR、卡顿、日志和用户路径一起判断。

一句话:**没有监控的 App,线上问题只能靠用户投诉发现。**

---

## 一、先区分几类问题

| 类型 | 说明 |
| --- | --- |
| 崩溃 | 进程异常退出 |
| ANR | Android 主线程长时间无响应 |
| 卡顿 | 页面掉帧、交互不流畅 |
| OOM | 内存不足导致进程被杀或崩溃 |
| 业务异常 | 接口失败、状态错误、流程中断 |
| 白屏 | 页面没有渲染出有效内容 |

不要只接崩溃平台。很多线上问题不会崩,但用户已经无法使用。

---

## 二、崩溃采集要包含什么

一条有用的崩溃记录至少包含:

```text
异常类型
堆栈
线程
App 版本和 build 号
系统版本
设备型号
渠道
用户匿名 ID
页面路径
最近关键操作
网络状态
是否 root / 越狱
```

没有版本和渠道,崩溃就很难定位。

没有用户路径,堆栈就只能说明代码在哪里炸,不能说明为什么炸。

---

## 三、符号表必须管理

release 包通常会混淆或符号化:

| 平台 | 需要管理什么 |
| --- | --- |
| Android | mapping.txt、native symbols |
| iOS | dSYM |
| Flutter | split-debug-info 产物 |
| RN | sourcemap |

规则:

```text
每个 release build 都要保留符号表
符号表和版本号 / build 号绑定
上传崩溃平台要自动化
不能只存在开发机本地
```

没有符号表,线上堆栈会变成不可读乱码。

---

## 四、ANR 怎么看

ANR 常见原因:

```text
主线程做 IO
主线程做数据库大查询
主线程等待锁
BroadcastReceiver 执行太久
Service 响应太慢
WebView 或图片加载阻塞主线程
```

Android ANR 不是"卡一下",而是系统认为你的进程没有及时响应。

排查顺序:

```text
看发生页面
看主线程栈
看锁等待
看同时期 CPU / IO / 内存
看是否集中在某设备或系统版本
```

ANR 要按版本趋势看,不要只看单条堆栈。

---

## 五、卡顿和掉帧

用户感知的卡顿通常来自:

```text
页面首帧慢
列表滚动掉帧
图片解码太重
布局计算过多
主线程任务太长
动画和网络状态耦合
低端机压力未测试
```

需要采集:

```text
页面打开耗时
首帧耗时
慢帧比例
冻结帧
主线程长任务
设备档位
```

卡顿优化不能只拿旗舰机做结论。

---

## 六、日志采集要分级

日志级别:

| 级别 | 用途 |
| --- | --- |
| debug | 本地调试 |
| info | 关键状态变化 |
| warn | 可恢复异常 |
| error | 明确失败 |
| fatal | 导致不可用的问题 |

线上日志要克制:

```text
只采关键链路
默认脱敏
支持按用户或会话临时提升等级
限制大小和上传频率
避免影响性能
```

日志平台不是垃圾桶。采太多会增加成本、泄漏风险和排查噪音。

---

## 七、业务异常也要上报

需要上报的业务异常:

```text
登录失败
支付失败
下单失败
推送 token 注册失败
深链解析失败
权限被拒绝
离线包加载失败
关键接口超时
```

字段建议:

```text
业务错误码
接口 trace id
页面
App 版本
渠道
网络状态
是否重试成功
```

不要把所有失败都当成崩溃。业务异常需要单独看漏斗。

---

## 八、监控指标怎么定

常见质量指标:

```text
崩溃率
崩溃用户率
ANR 率
卡顿率
启动耗时 P50 / P90 / P99
页面打开耗时
接口成功率
白屏率
新版本问题占比
```

优先看用户率:

```text
崩溃次数很多,可能是少量用户反复崩
崩溃用户率高,说明影响面更大
```

版本发布后要看小时级趋势,不要等日报。

---

## 九、什么时候会出事故

常见事故:

```text
release 包崩溃但没有上传 mapping / dSYM
崩溃平台只接入 Android,漏了 iOS
日志里带 token 和手机号
新版本崩溃率升高但无人值守
低端机列表卡顿导致差评
ANR 集中在某厂商系统但测试未覆盖
业务接口失败没有上报,只能靠客服反馈
崩溃采样过低漏掉核心问题
```

监控不是上线后补的,必须进发布流程。

---

## 十、检查清单

- [ ] Android / iOS 是否都接入崩溃采集
- [ ] release 符号表是否自动上传
- [ ] 崩溃是否绑定版本、渠道、设备、页面
- [ ] ANR 是否单独监控
- [ ] 启动、页面首帧、卡顿是否采集
- [ ] 关键业务失败是否上报
- [ ] 日志是否脱敏
- [ ] release 是否关闭调试日志
- [ ] 新版本发布后是否有人看小时级数据
- [ ] 是否能按用户或 trace id 串联问题

---

## 十一、结论

移动端质量监控最小闭环:

```text
采集 -> 聚合 -> 告警 -> 定位 -> 修复 -> 验证 -> 复盘
```

只采集不告警,问题会静默扩大。

只告警不保留符号表,问题无法定位。

只看崩溃不看业务异常,用户仍然会卡在流程里。
