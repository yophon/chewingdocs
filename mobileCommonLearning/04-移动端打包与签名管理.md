# 移动端打包与签名管理

签名不是发布流程里的一个勾选项。签名决定这个包是不是你发的,也决定用户能不能覆盖升级。

一句话先记住:**移动端签名 = 用私钥证明这个安装包属于某个开发者或团队。**

---

## 一、Android 打包机制

Android 常见产物:

| 产物 | 用途 |
| --- | --- |
| APK | 可直接安装 |
| AAB | 上传 Google Play,由商店按设备生成 APK |

典型流程:

```text
Java / Kotlin / Dart
  -> .class
  -> D8 / R8 -> classes.dex
  -> aapt2 编译资源
  -> 合并 AndroidManifest.xml
  -> 打包 APK / AAB
  -> release keystore 签名
  -> zipalign / optimize
```

关键点:

- 同一个已发布应用,后续更新必须保持 `applicationId` 和签名兼容。
- 如果不使用 Google Play App Signing 托管,release keystore 丢失通常意味着旧用户无法无缝升级。
- 国内多渠道分发时,release keystore 是最高优先级资产。

---

## 二、iOS 打包机制

iOS 常见产物:

| 产物 | 用途 |
| --- | --- |
| `.app` | 应用 bundle |
| `.ipa` | 可分发安装包 |
| `.xcarchive` | Xcode Archive 产物,用于导出 IPA 或上传 |

典型流程:

```text
Swift / Objective-C / Dart
  -> native binary
  -> .app bundle
  -> 写入 Info.plist / Frameworks / Assets
  -> 配置 entitlements
  -> certificate + private key 签名
  -> archive
  -> upload App Store Connect
```

关键点:

- App Store 更新主要依赖同一 Apple Developer Team 下的 Bundle ID。
- 不强制使用旧 Mac 的同一个私钥。
- 有账号完整权限时,可以重新生成证书和 profile 继续更新 App Store 版本。

---

## 三、Android 签名材料

你需要保管:

```text
release.jks / release.keystore
keyAlias
storePassword
keyPassword
key.properties
签名证书指纹:MD5 / SHA1 / SHA256
```

典型 `key.properties`:

```properties
storeFile=../release.jks
storePassword=***
keyAlias=release
keyPassword=***
```

不要提交:

```text
*.jks
*.keystore
key.properties
```

---

## 四、iOS 签名材料

iOS 签名材料:

| 项目 | Apple 是否保存 | 说明 |
| --- | --- | --- |
| Apple Developer 账号 | 是 | 开发者身份 |
| Team ID | 是 | 团队标识 |
| Bundle ID / App ID | 是 | App 身份 |
| Certificate `.cer` | 部分 | Apple 签发证书 |
| Private Key | 否 | 本地钥匙串生成,Apple 不保存 |
| Provisioning Profile | 是 | 绑定 App、证书、设备、能力 |
| Entitlements | 部分 | 项目声明 + profile 约束 |
| `.p12` | 否 | 证书 + 私钥导出 |

关键结论:

- `.cer` 不等于完整签名身份。
- 真正能签名的是"证书 + 私钥"。
- `.p12` 通常包含证书和私钥。
- 只有 `.cer` 和 `.mobileprovision`,没有私钥,本机不能用旧 identity 签名。

## 五、换设备怎么处理

### Android

强烈建议迁移旧签名材料。

必须备份:

```text
.jks / .keystore
key.properties
keyAlias
storePassword
keyPassword
```

没有这些,很多渠道无法覆盖更新。

### iOS

App Store 场景通常可以重新生成:

```text
Xcode 登录 Apple ID
选择正确 Team
Automatically manage signing
重新生成 / 下载 certificate 和 profile
Archive 上传
```

建议迁移旧 `.p12` 的场景:

- CI/CD 依赖固定证书
- 手动签名
- 企业包 / Ad Hoc
- 证书数量达到上限
- 三方系统记录了证书或推送相关材料

## 六、签名指纹怎么查

查看 APK 签名:

```bash
apksigner verify --print-certs app-release.apk
```

查看 keystore:

```bash
keytool -list -v -keystore release.keystore
```

这些指纹要记录:

```text
MD5
SHA1
SHA256
```

很多开放平台需要 SHA1 / SHA256。

## 七、Flutter 场景

Flutter 本质仍然走各平台签名:

```bash
flutter build apk --release
flutter build appbundle --release
flutter build ios --release
```

Flutter 只负责构建跨端代码,不替你解决:

- Android release keystore
- iOS certificate / profile
- App Store Connect
- Play Console
- 国内渠道签名变更

Flutter 项目也必须按原生平台管理签名资产。

---

## 八、什么时候会出事故

1. 把 debug keystore 当 release 用。
2. keystore 放在离职同事电脑里,没有备份。
3. 只有 iOS `.cer`,没有私钥,CI 不能签。
4. profile 没包含 Push / Associated Domains,Archive 能过但线上能力失效。
5. 重新生成 Android keystore 后发包,老用户不能覆盖安装。
6. 签名指纹没同步到微信 / 地图 / 推送平台。

---

## 九、检查清单

- [ ] Android release keystore 是否加密备份
- [ ] keystore 密码是否在密码管理器
- [ ] key.properties 是否不进 git
- [ ] iOS `.p12` 是否能在新机器导入
- [ ] profile 是否匹配 Bundle ID 和能力
- [ ] 签名指纹是否记录
- [ ] CI/CD 是否通过 Secret 注入签名材料
- [ ] release 包是否用正式签名真机测试

## 十、心智模型

```text
Android:keystore 是覆盖升级生命线
iOS:Team + Bundle ID + certificate/private key/profile 是签名链
Flutter:仍然要回到 Android / iOS 签名体系
CI:签名材料必须可恢复、可审计、不可泄漏
```
