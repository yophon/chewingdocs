# 权限模型:相机、相册、定位、麦克风、通知

权限不是弹窗。权限是用户把某种设备能力临时授权给你的应用。

一句话先记住:**权限要在用户触发具体功能时申请,并且要能处理拒绝、限制和系统设置变更。**

---

## 一、常见权限

| 能力 | 常见用途 |
| --- | --- |
| 相机 | 拍照、扫码、视频 |
| 相册 / 媒体 | 选择图片、保存文件 |
| 定位 | 地图、附近、打卡 |
| 麦克风 | 语音、通话、录音 |
| 通知 | 消息提醒、营销、系统事件 |
| 蓝牙 | 设备连接 |
| 联系人 | 邀请、通讯录 |
| 日历 | 日程 |

原则:

```text
不用就不申请
用到再申请
拒绝要能继续走替代流程
```

---

## 二、权限状态

不要只判断"有 / 没有"。

常见状态:

```text
notDetermined  用户还没决定
granted        已授权
denied         拒绝
restricted     系统或家长控制限制
limited        只允许部分资源(iOS 相册)
permanentlyDenied  拒绝且不再弹窗(Android 常见)
```

UI 要根据状态分支。

---

## 三、申请时机

坏:

```text
App 启动 -> 连续弹相机、定位、通知
```

好:

```text
用户点击"拍照上传"
  -> 解释为什么需要相机
  -> 申请相机权限
  -> 成功后打开相机
```

权限弹窗前最好有业务上下文。

---

## 四、拒绝后的流程

用户拒绝后:

```text
展示原因
提供设置入口
提供替代方案
不要无限弹窗
```

例如相册:

```text
你可以继续填写表单
也可以去设置中允许相册权限后上传图片
```

拒绝权限不应该让整个 App 死掉。

---

## 五、Android 特点

Android 重点:

- 运行时权限
- target SDK 影响权限行为
- 通知权限在新系统上需要显式申请
- 后台定位比前台定位更严格
- 相册 / 媒体权限随系统版本演进
- `shouldShowRequestPermissionRationale` 判断是否展示解释

不要只按一台测试机判断权限逻辑。

---

## 六、iOS 特点

iOS 重点:

- `Info.plist` 必须写用途说明
- 相册有 limited access
- 定位分 When In Use / Always
- 通知授权独立
- 权限被拒后通常要引导用户去 Settings
- 隐私能力和 App Review 强相关

用途文案必须具体,不要写模板废话。

---

## 七、Flutter / RN

跨端框架只封装 API,不消除平台差异。

你仍然要配置:

```text
AndroidManifest.xml
Info.plist
Gradle / Pod 依赖
运行时申请
平台差异分支
```

权限库不能替你决定业务流程。

---

## 八、什么时候会出事故

1. iOS `Info.plist` 缺用途说明,直接崩或拒审。
2. Android target SDK 升级后权限行为变化。
3. 用户拒绝权限后页面卡死。
4. 相册 limited 模式下以为拿到了全部照片。
5. 后台定位没有合规说明,被拒审。
6. 通知没申请成功,运营以为推送系统坏了。

---

## 九、检查清单

- [ ] 是否只在功能触发时申请权限
- [ ] 是否处理 denied / limited / restricted
- [ ] 是否有设置页引导
- [ ] 是否有替代流程
- [ ] iOS `Info.plist` 文案是否具体
- [ ] Android target SDK 升级后是否重测权限
- [ ] 权限用途是否写进隐私政策
- [ ] 真机是否覆盖主流系统版本

---

## 十、心智模型

```text
权限是用户授权,不是开发者配置
用途说明影响审核和信任
拒绝权限是正常路径
系统版本会改变权限行为
跨端框架不屏蔽平台规则
```
