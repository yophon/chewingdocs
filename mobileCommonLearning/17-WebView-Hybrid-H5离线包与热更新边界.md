# WebView、Hybrid、H5 离线包与热更新边界

WebView 能让 App 快速承载 H5 页面,也会把前端、安全、缓存、审核、原生能力混在一起。

一句话:**Hybrid 可以提高交付效率,但不能绕过平台规则,更不能把热更新当成无限制发版。**

---

## 一、先区分几种形态

| 形态 | 说明 |
| --- | --- |
| WebView 页面 | App 内嵌网页 |
| Hybrid 页面 | H5 通过 JSBridge 调用原生能力 |
| H5 离线包 | 把 H5 资源预置或下载到本地 |
| RN / Flutter 热更新 | 更新跨端代码或资源 |
| 原生发版 | 通过商店更新二进制 |

它们的边界不同:

```text
WebView 更新网页最快
离线包更新资源和页面逻辑
热更新更新部分跨端逻辑
原生发版更新平台能力和二进制代码
```

不要把所有"不用发版"都叫热更新。

---

## 二、WebView 适合什么

适合:

```text
活动页
内容页
协议页
帮助中心
运营配置页面
低频业务流程
需要快速调整的展示逻辑
```

不适合:

```text
高性能交互
复杂手势动画
强依赖系统能力的页面
关键支付链路的全部逻辑
需要离线强一致的核心流程
```

WebView 是工程工具,不是性能和体验的万能替代。

---

## 三、JSBridge 要当成安全边界设计

JSBridge 本质是:

```text
网页调用 App 能力的接口
```

因此必须控制:

```text
哪些域名可以调用
哪些方法可以调用
参数怎么校验
结果怎么返回
是否需要用户确认
是否记录审计日志
```

不要这样做:

```text
任意 URL 都能调用 bridge
bridge 暴露 openUrl、writeFile、getToken 等高危能力
参数直接拼接执行
H5 可以拿到完整登录凭据
```

Bridge API 要像服务端接口一样设计权限。

---

## 四、WebView 登录态怎么处理

常见方案:

| 方案 | 风险 |
| --- | --- |
| Cookie 同步 | 平台差异、清理复杂 |
| URL 带 token | 容易泄漏到日志和历史记录 |
| JS 注入 token | XSS 风险高 |
| Bridge 获取短期票据 | 相对可控 |

推荐思路:

```text
H5 不直接持有长期 refresh token
App 通过 bridge 发放短期一次性票据
H5 用票据换业务会话
退出登录时同时清 App token、Cookie、WebView storage
```

登录态要统一退出,不能 App 退出了 WebView 还在线。

## 五、离线包解决什么问题

离线包常用于:

```text
加快首屏
弱网可用
降低服务器静态资源压力
减少线上紧急改动成本
```

基本结构:

```text
manifest.json
资源文件
版本号
hash / 签名
灰度规则
回滚规则
```

离线包不是简单 zip 下载。它是一套小型发布系统。

---

## 六、离线包必须有校验

至少要校验:

```text
来源
完整性
版本
hash
签名
兼容的 App 版本
```

更新流程:

```text
请求 manifest
检查 App 版本和渠道
下载资源包
校验 hash / 签名
解压到临时目录
切换指针
保留上一版本可回滚
```

不要下载后直接覆盖当前线上资源。

---

## 七、热更新边界

热更新不能用来绕过审核规则。

原则:

```text
不能改变 App 核心用途
不能下发未经审核的高风险能力
不能动态开启隐藏功能规避审核
不能替换支付、登录、隐私采集等核心合规逻辑
不能下载可执行原生代码破坏平台规则
```

合理用途:

```text
修 UI 文案
修业务展示逻辑
调整活动页
修非核心流程 bug
灰度配置和开关
```

涉及平台政策时要按当前商店规则复查。复查日期:2026-06-08。

---

## 八、WebView 常见工程问题

常见坑:

```text
缓存策略不清导致用户一直看到旧页面
Android / iOS WebView 内核差异
文件上传、相机、定位权限链路复杂
返回栈和原生导航冲突
页面白屏缺少兜底
H5 崩溃和 JS 错误未上报
Cookie 清理不完整
深链跳转循环
```

每个 WebView 容器都应该提供:

```text
加载态
错误页
重试
标题同步
返回处理
权限拦截
域名白名单
JS 错误上报
```

---

## 九、什么时候会出事故

常见事故:

```text
离线包 hash 校验缺失,资源被污染
新 H5 调用了旧 App 不支持的 bridge 方法
热更新下发后旧版本 App 白屏
退出登录没有清 WebView Cookie
URL 携带 token 被埋点平台采集
Android 文件选择器没适配导致上传失败
iOS WKWebView 缓存导致审核看到旧页面
动态下发能力被商店认为规避审核
```

Hybrid 事故通常发生在"原生、H5、服务端版本没有对齐"。

---

## 十、检查清单

- [ ] WebView 是否限制可访问域名
- [ ] JSBridge 是否有方法白名单和参数校验
- [ ] H5 是否拿不到长期 token
- [ ] 退出登录是否清 Cookie、localStorage、sessionStorage
- [ ] 离线包是否有 hash / 签名校验
- [ ] 离线包是否按 App 版本做兼容控制
- [ ] 离线包是否能回滚
- [ ] H5 JS 错误是否上报
- [ ] 热更新范围是否不触碰审核红线
- [ ] 新 bridge 方法是否兼容旧 App

---

## 十一、结论

Hybrid 的核心不是"快",而是边界:

```text
H5 负责可快速变化的展示和轻业务
原生负责平台能力和安全边界
离线包负责资源交付
热更新负责有限修复
商店发版负责二进制和核心能力
```

边界清楚,Hybrid 是效率工具。边界不清,它就是线上事故入口。
