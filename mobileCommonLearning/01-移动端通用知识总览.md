# 移动端通用知识总览

移动端开发不只是写 UI。真正上线以后,你会面对签名、商店、权限、推送、深链、后台限制、崩溃、渠道、合规和发版事故。

一句话先记住:**移动端通用知识是所有 App 的生产规则,和你用 Flutter、React Native、Kotlin、Swift 无关。**

---

## 一、为什么要有这个系列

Flutter 系列会告诉你怎么写 Flutter App。

Android 原生系列会告诉你怎么写 Kotlin + Compose。

iOS 原生系列会告诉你怎么写 Swift + SwiftUI。

但真实项目还有一堆跨平台问题:

```text
Android keystore 丢了怎么办?
iOS 换 Mac 后还能不能更新线上 App?
Bundle ID 和 applicationId 能不能改?
TestFlight、Google Play、国内应用市场有什么区别?
推送 token 为什么会变?
App Links 为什么线上不生效?
release 包为什么 debug 没问题、线上崩?
签名泄漏后怎么止损?
```

这些问题不属于某个 UI 框架,属于移动端工程本身。

## 二、移动端上线链路

一个 App 从代码到用户手机,大致是:

```text
代码
  -> 编译
  -> 打包
  -> 签名
  -> 上传商店 / 渠道
  -> 审核
  -> 灰度 / 分阶段发布
  -> 用户安装 / 更新
  -> 崩溃和性能监控
  -> 下一次发版
```

每一段都有事故点:

| 阶段 | 常见事故 |
| --- | --- |
| 编译 | 环境不一致、依赖版本漂移 |
| 打包 | debug 正常、release 崩溃、资源缺失 |
| 签名 | keystore 丢失、p12 缺私钥、profile 不匹配 |
| 上传 | 版本号没递增、包名不一致、target SDK 不达标 |
| 审核 | 隐私说明缺失、权限用途不清、截图材料不合规 |
| 发布 | 灰度策略错误、渠道包污染、回滚困难 |
| 线上 | 崩溃、ANR、推送异常、深链失效 |

这个系列就是逐段补齐这些知识。

## 三、最重要的几个身份

移动端有几个"身份证",不能乱改。

| 概念 | Android | iOS |
| --- | --- | --- |
| 应用身份 | `applicationId` | Bundle ID |
| 商店身份 | Play Console / 国内市场应用记录 | App Store Connect App |
| 签名身份 | keystore / signing key | Apple Team + certificate + private key + profile |
| 版本身份 | versionCode / versionName | build number / version |
| 能力身份 | package + cert fingerprint | App ID + entitlements |

记住:

```text
包名 / Bundle ID 决定是不是同一个 App
签名决定能不能覆盖升级
版本号决定商店收不收
能力配置决定推送、深链、登录等能不能工作
```

## 四、Android 和 iOS 最大差异

Android:

```text
签名强绑定更新链路
同包名更新必须签名兼容
国内多渠道复杂
Google Play App Signing 可以托管发布签名
```

iOS:

```text
App Store 更新主要绑定 Apple Team + Bundle ID
证书可以重新生成
Provisioning Profile 约束能力和分发方式
私钥不在 Apple 托管
```

一句话:

```text
Android 更怕 keystore 丢
iOS 更怕账号权限、证书/profile、entitlements 配错
```

## 七、不要公开写什么

签名和发布文档经常会不小心泄漏内部信息。公开教程不要写:

- 真实 keystore 文件路径
- keystore / p12 / profile 文件名和存放位置
- Team ID、Bundle ID、包名和项目名的完整映射
- 推送、地图、支付、登录平台的真实签名指纹
- 内部渠道账号和后台截图
- CI secret 名称和值的对应关系

可以写方法,不要写资产明细。资产明细放内部运维文档。

## 八、最小检查清单

一个准备上线的移动端项目,至少要回答:

- [ ] 包名 / Bundle ID 是否最终确定
- [ ] 版本号 / build 号规则是否清楚
- [ ] Android keystore 是否加密备份
- [ ] iOS 证书、私钥、profile 是否能在新机器恢复
- [ ] CI/CD 的签名材料是否走 Secret
- [ ] 推送、登录、支付、地图是否依赖签名指纹
- [ ] App Store / Google Play / 国内渠道账号谁负责
- [ ] 隐私政策、权限说明、截图材料是否准备
- [ ] release 包是否真机测过核心流程
- [ ] 崩溃和性能监控是否接入
- [ ] 签名丢失或泄漏是否有预案

## 九、心智模型

```text
写代码只是移动端工程的一半
签名和商店决定能不能发
权限和系统限制决定能不能稳定运行
监控和事故预案决定线上能不能维护
```

移动端一旦上架,每次发版都是生产变更。这个系列按生产变更来讲移动端。
