# 本地存储、Keychain / Keystore 与敏感数据

移动端一定会存数据。问题不是"能不能存",而是**什么数据能存、存在哪里、丢了会怎样、泄漏会怎样**。

一句话:**普通数据放普通存储,敏感数据放系统安全存储,真正不能泄漏的数据不要只靠客户端保存。**

---

## 一、本地存储分几类

移动端常见本地存储:

| 类型 | 适合存什么 | 不适合存什么 |
| --- | --- | --- |
| 配置存储 | 主题、开关、引导页状态 | token、密码、密钥 |
| 文件存储 | 图片、缓存、离线资源 | 明文敏感材料 |
| 数据库存储 | 会话列表、业务数据、索引 | 未加密隐私字段 |
| 安全存储 | refresh token、设备密钥、登录凭据 | 大体积业务数据 |
| 系统缓存 | 网络缓存、图片缓存 | 任何必须长期可靠保存的数据 |

不要把"本地能读写"理解成"本地可靠"。

客户端本地数据可能因为卸载、清缓存、系统回收、备份恢复、换机、越狱/root、调试抓取而变化。

---

## 二、Android 和 iOS 的安全存储

Android 常见方案:

```text
Android Keystore
EncryptedSharedPreferences
EncryptedFile
Room + SQLCipher
```

iOS 常见方案:

```text
Keychain
Secure Enclave
File Protection
加密数据库
```

关键区别:

| 平台 | 重点 |
| --- | --- |
| Android | Keystore 更像密钥容器,业务数据通常要自己加密后存 |
| iOS | Keychain 适合小体积凭据,可配置访问时机和迁移属性 |

安全存储不是保险箱万能钥匙。它只能降低攻击成本,不能让客户端数据绝对安全。

---

## 三、什么算敏感数据

下面这些默认按敏感数据处理:

```text
access token
refresh token
用户手机号、邮箱、身份证、地址
支付凭据
会话 cookie
私有 API key
设备绑定密钥
企业内部接口地址
第三方平台 secret
```

注意:

```text
客户端里没有真正的 secret
```

只要 App 发给用户,代码和资源就有被反编译、动态调试、抓包的可能。服务端 secret、私钥、管理 token 不应该放进客户端。

---

## 四、token 应该怎么存

常见做法:

```text
access token:短期有效,内存优先,必要时安全存储
refresh token:安全存储,生命周期更严格
登录态标记:普通存储可以,但不能作为权限依据
用户资料缓存:按字段分级,敏感字段加密或不落盘
```

不要这样做:

```text
把 token 写进日志
把 token 放 URL query
把 token 存普通 SharedPreferences / UserDefaults
把 refresh token 放 WebView localStorage
把所有环境共用同一个 token key
```

服务端仍然必须校验 token 有效性、权限、设备状态和风控状态。

## 五、Keychain 和 Keystore 的常见坑

iOS Keychain 常见坑:

| 问题 | 说明 |
| --- | --- |
| 卸载后可能仍保留 | Keychain 数据不一定随 App 卸载清掉 |
| 访问组配置错误 | App Group / Keychain Sharing 不匹配会读不到 |
| 备份迁移属性不清 | 换机后是否迁移取决于 accessibility 配置 |
| 生物识别变化 | 指纹/面容变化可能导致受保护项不可读 |

Android Keystore 常见坑:

| 问题 | 说明 |
| --- | --- |
| 系统升级或设备安全状态变化 | 某些密钥可能失效 |
| 指纹变化 | 绑定认证的 key 可能不可用 |
| 厂商实现差异 | 旧设备或定制系统行为不完全一致 |
| 只存 key 不存策略 | 过期、轮换、重建逻辑缺失 |

安全存储读失败不是小概率异常,要当作正常分支处理。

## 六、本地数据要设计生命周期

每类数据都要回答:

```text
什么时候写入?
什么时候读取?
什么时候过期?
什么时候清除?
能不能跨版本迁移?
能不能跨设备迁移?
丢了是否能重建?
泄漏是否需要踢登录?
```

示例:

| 数据 | 生命周期 |
| --- | --- |
| access token | 短期有效,过期刷新,退出登录清除 |
| refresh token | 长期但可吊销,换密码或风控时失效 |
| 图片缓存 | 可删除,可重建 |
| 离线业务数据 | 需要版本号和迁移策略 |
| 设备密钥 | 丢失后重新注册设备 |

不要把"写入成功"当成永久存在。

---

## 七、加密不是只调用一个库

本地加密至少包含:

```text
密钥从哪里来
密钥怎么保存
密钥怎么轮换
数据怎么标记版本
解密失败怎么办
备份恢复后怎么办
```

推荐模式:

```text
随机生成本地数据密钥
数据密钥由系统安全存储保护
业务数据用数据密钥加密后落盘
密文带版本号、算法标识和迁移信息
```

不要自己发明加密算法。用平台能力和成熟库。

---

## 八、日志和埋点也是存储

很多泄漏不是来自数据库,而是来自:

```text
console log
崩溃堆栈
网络请求日志
埋点参数
客服诊断日志
截图和录屏
剪贴板
```

规则:

```text
日志默认脱敏
token 不进日志
手机号只保留后四位或 hash
身份证、银行卡不进客户端日志
崩溃平台上报前做字段过滤
调试日志 release 默认关闭
```

线上日志要按生产数据处理。

---

## 九、什么时候会出事故

常见事故:

```text
换机后 Keychain 数据迁移导致异常登录
用户卸载重装后仍读取旧凭据
release 日志上报了 token
refresh token 存在普通文件里被抓取
数据库升级失败导致本地数据全丢
Android Keystore key invalidated 后无法解密
WebView localStorage 保留了登录态
退出登录没有清空缓存和 cookie
```

事故根因通常不是"没有加密",而是没有把本地数据当成完整的生产系统设计。

---

## 十、检查清单

- [ ] token 是否没有写入普通存储
- [ ] refresh token 是否使用系统安全存储
- [ ] 退出登录是否清理 token、cookie、WebView、缓存
- [ ] 本地数据库是否有版本迁移和失败兜底
- [ ] 敏感字段是否不会进入日志、埋点、崩溃上报
- [ ] 安全存储读取失败是否有重新登录或重建流程
- [ ] 加密数据是否带算法和版本信息
- [ ] App 卸载、重装、换机、系统升级是否测过
- [ ] release 包是否关闭调试日志
- [ ] 服务端是否能吊销客户端凭据

---

## 十一、结论

本地存储的核心不是 API,而是分级:

```text
可丢缓存 -> 普通缓存
业务数据 -> 数据库和迁移
敏感凭据 -> Keychain / Keystore
服务端秘密 -> 不进客户端
```

客户端只负责降低泄漏概率,最终安全边界必须在服务端。
