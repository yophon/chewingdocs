# 签名资产备份、迁移、丢失与泄漏应急

签名资产不是开发者个人文件,是公司级生产资产。

一句话先记住:**签名资产管理的目标不是能打一次包,而是任何时候都能安全地继续发版。**

---

## 一、资产清单

Android:

```text
release keystore / jks
keyAlias
storePassword
keyPassword
签名指纹 MD5 / SHA1 / SHA256
Google Play upload key / app signing key 状态
```

iOS:

```text
Apple Developer 账号
Team ID
Bundle ID / App ID
Distribution certificate
private key
.p12
Provisioning Profile
entitlements
dSYM 归档
```

不要只备份文件,还要备份说明和责任人。

---

## 二、备份规则

最低要求:

```text
两份加密备份
两个人有恢复权限
密码进密码管理器
不进 git
不放聊天记录
不放公开网盘
有恢复演练
```

建议记录:

```text
资产名
用途
所属 App
环境(dev/staging/prod)
创建时间
过期时间
负责人
恢复步骤
```

---

## 三、迁移到新设备

Android:

```text
复制 keystore
恢复 key.properties
确认 Gradle signingConfig
打 release 包
用 apksigner 查看指纹
和历史记录比对
真机覆盖安装测试
```

iOS:

```text
导入 .p12 到 Keychain
安装 profile
Xcode 选择 Team
Archive
检查 signing identity
上传 TestFlight
```

迁移成功的标准不是"能 build",而是**签出来的包能覆盖旧版本或上传对应商店**。

---

## 四、Android 签名丢失

如果使用 Google Play App Signing:

```text
丢 upload key -> 申请重置
丢 app signing key -> 看是否由 Google 托管
```

如果是国内渠道 / 官网 APK:

```text
先找旧电脑、CI、备份、构建机
再联系渠道确认签名变更流程
渠道不支持时,可能只能换包名重新发布
```

不能做:

```text
从 APK / 证书还原私钥
随便生成新 keystore 直接发更新
```

私钥不可逆。

---

## 五、Android 签名泄漏

仍持有旧签名:

```text
冻结渠道发版
检查是否有异常包
评估 v3 signing lineage
生成新签名并建立轮换链
更新三方平台指纹
全渠道实测覆盖升级
```

旧签名失控:

```text
下架或冻结可疑渠道包
检查官网 / CDN / 下载页
后端拦截异常版本和签名指纹
联系各应用市场人工处理
准备新签名或新包名迁移方案
通知用户从可信渠道更新
```

签名泄漏是安全事故,不是普通发版问题。

---

## 六、iOS 证书丢失

App Store 场景:

```text
有 Team 权限 -> 可重新生成证书和 profile
Bundle ID 不变 -> 可继续更新
```

CI / 企业 / Ad Hoc:

```text
需要恢复或重新配置 .p12 / profile
旧 profile 可能绑定旧 certificate
设备列表和能力要重新确认
```

iOS 更怕账号权限丢失和能力配置不一致,不如 Android 那样单点依赖旧 keystore。

---

## 七、渠道签名变更材料

国内渠道常见要求:

- 公司营业执照
- 法人身份证或授权书
- 软件著作权
- App 名称
- 包名
- 当前线上版本号
- 原签名 MD5 / SHA1 / SHA256
- 新签名 MD5 / SHA1 / SHA256
- 新 APK / AAB
- 官网或其他市场在架截图
- 签名丢失或泄漏说明

这些材料平时就该准备好。

---

## 八、应急时的后端配合

后端可以做:

```text
记录客户端版本
记录渠道
记录签名指纹或安装来源可信度
拦截异常版本号
关闭高风险功能
强制可信渠道升级
展示安全公告
```

移动端事故往往不能只靠客户端修。发布新包需要时间,后端开关能争取缓冲。

---

## 九、演练

至少每半年演练一次:

```text
新机器从零恢复 Android release 签名
新机器从零恢复 iOS Archive
CI 用 Secret 打一份 release 包
比对签名指纹
上传内测渠道
确认符号表上传
```

没有演练的备份,只能算"心理安慰"。

## 十、检查清单

- [ ] Android keystore 是否两份加密备份
- [ ] iOS p12 / profile 是否可恢复
- [ ] 密码是否在密码管理器
- [ ] 是否记录签名指纹
- [ ] 是否盘点渠道签名变更流程
- [ ] 是否盘点依赖签名指纹的平台
- [ ] 是否有签名泄漏应急预案
- [ ] 是否做过新机器恢复演练
- [ ] 是否有离职交接流程
