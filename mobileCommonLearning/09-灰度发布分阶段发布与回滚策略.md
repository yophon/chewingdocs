# 灰度发布、分阶段发布与回滚策略

移动端发布不能像后端一样秒级回滚。包一旦到用户手机上,你就要和它长期相处。

一句话先记住:**移动端灰度的目标不是省审核时间,而是在错误影响全量用户前发现并止损。**

---

## 一、为什么移动端回滚难

后端:

```text
部署新版本 -> 出问题 -> 切回旧版本
```

移动端:

```text
用户已经下载新包
商店审核有时间
用户不一定立刻更新
旧包和新包长期共存
```

所以移动端更依赖:

- 灰度
- 服务端开关
- 兼容协议
- 强制升级策略
- 事故预案

---

## 二、灰度方式

| 方式 | 说明 |
| --- | --- |
| 商店分阶段发布 | 按比例推送给用户 |
| 渠道灰度 | 先发内部渠道 / 小渠道 |
| 服务端 feature flag | 新功能按用户 / 版本打开 |
| 账号白名单 | 只给内部账号开放 |
| 地域 / 设备灰度 | 按地区、机型、系统版本控制 |

移动端最稳的是:

```text
先发测试渠道 -> 小比例商店灰度 -> feature flag 控制功能 -> 逐步放量
```

---

## 三、App Store 分阶段发布

App Store 支持分阶段发布。你可以让新版本逐步覆盖用户。

注意:

- Apple 审核通过不代表应该立刻全量。
- 分阶段发布期间仍要看 crash 和核心业务指标。
- 如果出问题,可以暂停发布,但已更新用户不会自动降级。

---

## 四、Google Play 分阶段发布

Google Play staged rollout 可以按比例发布:

```text
1% -> 5% -> 10% -> 25% -> 50% -> 100%
```

放量前看:

- crash-free users
- ANR rate
- install failure
- rating / reviews
- 关键业务成功率

不要只看"没有人反馈"。

---

## 五、服务端开关

移动端必须有服务端开关:

```ts
type FeatureFlags = {
  newCheckout: boolean;
  enableLiveChat: boolean;
  forceUpgradeMinVersion?: string;
};
```

用途:

- 关闭新功能
- 禁用高风险入口
- 控制灰度用户
- 强制升级
- 降级到旧协议

不要把所有开关写死在 App 包里。

---

## 六、协议兼容

移动端多版本共存,后端不能只支持最新包。

规则:

```text
新增字段:可以
删除字段:危险
改字段类型:危险
改枚举含义:危险
强制新参数:危险
```

后端接口至少要兼容当前线上主力版本。

推荐:

```text
App 发版前确认最低支持版本
接口变更做向后兼容
必要时按 appVersion 分支
废弃接口先监控调用量
```

---

## 七、回滚手段

移动端回滚分几类:

| 手段 | 适用 |
| --- | --- |
| 暂停灰度 | 新包还没全量 |
| 下架版本 | 严重问题,阻止继续下载 |
| 服务端关闭功能 | 新功能出问题 |
| 强制升级 | 旧版本有安全 / 协议问题 |
| 后端兼容旧协议 | 新包 / 旧包共存 |
| 发修复包 | 最终解决 |

不要把"发修复包"当唯一手段。审核和用户更新都需要时间。

---

## 八、什么时候会出事故

1. 新接口只支持新版 App,旧版用户全挂。
2. 全量后才发现某机型启动崩溃。
3. 灰度期间只看 crash,没看支付成功率。
4. feature flag 默认打开,新包一上线全用户可见。
5. 下架后官网 APK 仍然能下载坏版本。

---

## 九、检查清单

- [ ] 是否有分阶段发布计划
- [ ] 是否有 feature flag
- [ ] 后端是否兼容旧版本
- [ ] 是否定义最低支持版本
- [ ] 是否能暂停新功能
- [ ] 是否有强制升级能力
- [ ] 是否监控 crash / ANR / 核心业务指标
- [ ] 官网和渠道是否能同步下架坏包

---

## 十、心智模型

```text
灰度发现问题
开关控制影响面
协议兼容保护旧版本
暂停和下架阻止继续扩散
修复包解决根因
```
