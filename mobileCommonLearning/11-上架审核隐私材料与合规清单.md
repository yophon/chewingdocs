# 上架审核、隐私材料与合规清单

商店审核不是最后补材料。很多审核问题来自项目一开始就没记录权限、数据和第三方 SDK。

一句话先记住:**移动端合规 = 你收集什么、为什么收集、传给谁、用户怎么知道和控制。**

---

## 一、审核看什么

常见关注点:

- App 功能是否真实可用
- 权限用途是否合理
- 隐私政策是否完整
- 数据收集声明是否一致
- 登录和账号删除是否符合要求
- 内购 / 支付是否合规
- 诱导、欺诈、虚假功能
- 崩溃和明显 bug
- 截图、名称、描述是否误导

审核不是只看包能不能安装。

---

## 二、隐私政策

隐私政策至少说明:

```text
收集哪些信息
用途是什么
是否共享给第三方
保存多久
用户如何删除账号和数据
如何联系开发者
```

不要复制一份模板就不管。实际 SDK 收集的数据要和隐私政策一致。

---

## 三、权限用途说明

权限文案要具体。

差:

```text
需要访问相机
```

好:

```text
用于拍摄头像和上传工单图片
```

权限申请时机:

```text
用户触发功能时申请
不要 App 启动就申请一堆权限
```

提前申请会降低通过率和用户信任。

---

## 四、iOS Privacy Manifest

iOS 生态越来越强调隐私声明。涉及隐私 API、第三方 SDK、数据收集时,要按 Apple 要求补充材料。

工程上要做:

- 盘点使用的 SDK
- 记录 SDK 收集的数据类型
- 记录是否追踪用户
- 检查是否需要 Privacy Manifest
- 上架前和 App Store Connect 隐私问卷对齐

具体字段会随 Apple 政策变化,每次发版前查官方文档。

---

## 五、Google Play Data safety

Google Play 要填写 Data safety。

你要回答:

- 收集哪些数据
- 是否共享
- 是否加密传输
- 用户是否可请求删除
- 数据是否可选
- 用于哪些目的

填写必须和实际 SDK 行为一致。广告、统计、崩溃 SDK 都要算。

---

## 六、国内市场材料

常见材料:

- 软件著作权
- 营业执照
- ICP 备案 / 网站信息
- 隐私政策 URL
- 用户协议 URL
- 权限说明
- 安全评估或 SDK 合规说明
- 应用截图
- 测试账号

不同渠道要求不同,要有材料台账。

---

## 七、账号删除

很多场景都需要提供账号删除能力:

```text
App 内入口
网页入口
客服流程
```

删除要说明:

- 删除什么数据
- 保留什么数据
- 多久生效
- 是否可恢复

不要只写"联系客服"但没有实际流程。

---

## 八、第三方 SDK 清单

维护一张表:

| SDK | 用途 | 收集数据 | 是否出境 | 隐私文档 |
| --- | --- | --- | --- | --- |
| Crash SDK | 崩溃分析 | 设备、崩溃日志 | 视服务而定 | URL |
| Analytics | 行为分析 | 事件、设备标识 | 视服务而定 | URL |
| Push SDK | 推送 | token、设备信息 | 视服务而定 | URL |

SDK 升级时要重新检查隐私行为。

---

## 九、什么时候会出事故

1. 使用相机权限,隐私政策没写。
2. SDK 收集设备标识,Data safety 没声明。
3. iOS 审核员没有测试账号,无法进入核心页面。
4. 国内市场要求软著,临发版才发现没有。
5. 权限在启动时弹出,审核认为用途不明确。
6. 没有账号删除入口,被拒审。

---

## 十、检查清单

- [ ] 隐私政策 URL 是否可访问
- [ ] 用户协议 URL 是否可访问
- [ ] 权限用途是否逐项说明
- [ ] iOS 隐私问卷是否和实际一致
- [ ] Google Play Data safety 是否和实际一致
- [ ] 第三方 SDK 清单是否维护
- [ ] 是否有账号删除流程
- [ ] 国内渠道材料是否齐全
- [ ] 是否提供审核测试账号
- [ ] 上架截图和描述是否与功能一致

---

## 十一、心智模型

```text
权限说明给用户看
隐私政策给用户和审核看
Data safety / 隐私问卷给商店看
SDK 清单给团队长期维护
测试账号给审核员进应用
```
