# 后台任务、前台服务与系统限制

移动端后台能力不是你想跑就能跑。系统会为了电量、性能和隐私限制后台执行。

一句话:**后台任务要按系统规则设计,不能把手机当服务器。**

---

## 一、先区分三件事

| 概念 | 说明 |
| --- | --- |
| 前台运行 | 用户正在使用 App |
| 后台运行 | App 不在前台,但进程可能还活着 |
| 后台任务 | 系统允许 App 在特定条件下短时间执行 |

很多问题来自误解:

```text
App 退到后台 != 代码还能一直跑
进程还活着 != 系统承诺不会杀
定时器还在跑 != 可以依赖它做业务
```

移动端后台能力永远是条件性的。

---

## 二、Android 的后台规则

Android 常见后台机制:

```text
WorkManager
JobScheduler
Foreground Service
AlarmManager
BroadcastReceiver
FCM high priority message
```

选择原则:

| 需求 | 建议 |
| --- | --- |
| 可延迟任务 | WorkManager |
| 网络恢复后同步 | WorkManager + constraints |
| 用户可感知的持续任务 | Foreground Service |
| 精确定时 | 谨慎使用 AlarmManager |
| 推送触发 | FCM / 厂商推送 |

Android 后台限制会受系统版本、厂商系统、省电策略、通知权限影响。

---

## 三、iOS 的后台规则

iOS 常见后台机制:

```text
Background Modes
Background App Refresh
BGTaskScheduler
Silent Push
URLSession background transfer
Location background
Audio / VoIP 等特定模式
```

iOS 更强调场景授权:

```text
不是你申请了后台模式就能随便跑
只有真实符合该模式的业务才应该开启
系统会根据电量、用户习惯、网络状态决定执行时机
```

不要指望 iOS 后台任务精确准时。

---

## 四、哪些需求不能直接靠后台任务

这些需求风险很高:

```text
每 10 秒上报一次位置
App 关闭后持续轮询接口
后台长期保持 WebSocket
每天固定 9 点一定执行本地任务
后台下载大文件但无用户感知
退到后台继续跑复杂计算
```

更合理的设计:

```text
服务端推送触发
用户打开 App 时补偿同步
系统任务做尽力执行
前台服务提供明确通知
服务端记录状态,客户端只做展示和确认
```

移动端后台任务要有"可能不执行"的业务兜底。

---

## 五、前台服务不是免死牌

Android 前台服务要求:

```text
用户可感知
必须有通知
用途要和声明类型匹配
不能滥用来保活
```

常见适合场景:

```text
导航
运动记录
音频播放
文件上传下载
通话
蓝牙设备连接
```

不适合:

```text
偷偷同步数据
维持长连接保活
规避系统省电限制
无通知后台计算
```

滥用前台服务可能导致审核、系统限制或用户卸载。

---

## 六、后台任务要可重试、可中断、可恢复

后台任务设计规则:

```text
任务必须幂等
任务状态要落盘
任务可以分片
任务失败可重试
任务被杀后可恢复
任务超时要停止
```

不要写这种逻辑:

```text
后台任务开始 -> 内存里循环处理 5000 条 -> 结束后一次性提交状态
```

应该改成:

```text
读取待处理任务
处理一小批
提交进度
失败记录原因
下次继续
```

后台任务不是事务,要按断点续跑设计。

---

## 七、推送和后台任务的关系

推送可以唤起一部分后台处理,但不能当成可靠计算通道。

常见规则:

```text
普通通知用于提醒用户
静默推送用于提示客户端同步
推送可能延迟、合并、丢弃
用户关闭通知会影响部分链路
厂商通道行为不完全一致
```

业务上要这样设计:

```text
推送只告诉客户端"可能有变化"
真正状态以服务端接口为准
用户进入页面时再拉取最新数据
```

不要把关键业务只放在推送回调里完成。

---

## 八、后台定位要特别谨慎

后台定位涉及高电量消耗和高隐私风险。

上线前要确认:

```text
是否确实需要后台定位
是否有清晰的用户可见价值
是否有显著的前台提示
是否能降低采样频率
是否能按地理围栏替代持续定位
是否有隐私政策说明
```

事故点:

```text
审核认为后台定位用途不充分
用户投诉耗电
系统关闭定位权限
后台定位被厂商系统杀掉
日志泄漏位置轨迹
```

位置数据默认按敏感数据处理。

---

## 九、什么时候会出事故

常见事故:

```text
debug 时后台定时器正常,release 线上被系统杀
Android 厂商系统限制导致后台同步不执行
iOS 静默推送不稳定,订单状态不同步
前台服务通知被用户关闭后任务中断
后台上传没有断点续传,进程被杀后文件丢失
后台定位耗电导致大量差评
任务重复执行造成重复扣款或重复提交
```

后台事故的根因通常是把"尽力执行"当成"保证执行"。

---

## 十、检查清单

- [ ] 后台任务是否有明确业务必要性
- [ ] 是否接受任务延迟或不执行
- [ ] 是否有打开 App 后的补偿同步
- [ ] 任务是否幂等
- [ ] 任务状态是否落盘
- [ ] Android 是否选择了合适的 WorkManager / 前台服务
- [ ] iOS 是否只开启必要 Background Modes
- [ ] 推送触发是否不承载唯一业务逻辑
- [ ] 后台定位是否有隐私说明和电量控制
- [ ] 省电模式、弱网、杀进程是否测过

---

## 十一、结论

后台任务的正确心智:

```text
前台:可以稳定执行
后台:系统允许时执行
被杀:下次恢复继续
关键状态:服务端兜底
```

移动端后台能力是协商出来的,不是抢来的。
