# 线上事故:热修、强更、下架、签名异常与渠道污染

移动端事故和服务端事故不一样。服务端可以快速回滚,App 发出去以后会留在用户手机里很久。

一句话:**移动端事故处理的核心是止损,不是幻想所有用户马上升级。**

---

## 一、先判断事故类型

常见移动端线上事故:

| 类型 | 例子 |
| --- | --- |
| 崩溃事故 | 新版本启动即崩 |
| 业务事故 | 登录、支付、下单失败 |
| 配置事故 | 生产包连测试环境 |
| 签名事故 | 包签名错误、无法覆盖安装 |
| 渠道事故 | 渠道包串包、渠道参数污染 |
| 合规事故 | 审核被拒、隐私投诉、下架 |
| 性能事故 | 启动慢、耗电、卡顿严重 |
| 推送事故 | 全量误推、token 大面积失效 |
| 深链事故 | 链接跳错页面或打不开 |

第一步不是修代码,而是判断影响面。

---

## 二、事故分级

可以按影响面分:

| 级别 | 判断 |
| --- | --- |
| P0 | App 大面积不可用、启动即崩、支付严重异常 |
| P1 | 核心流程不可用,但有部分兜底 |
| P2 | 部分功能异常,影响可控 |
| P3 | 非核心问题或体验问题 |

每次事故要明确:

```text
影响版本
影响平台
影响渠道
影响用户比例
是否影响交易或数据
是否可通过服务端止损
```

没有影响面判断,就无法决定是否暂停发布、强更或下架。

---

## 三、第一优先级是止损

常见止损手段:

```text
暂停灰度
关闭功能开关
服务端降级
回滚配置
屏蔽异常渠道包
停止推送
下架问题版本
强制升级
热更新修复
紧急提审新版本
```

优先顺序:

```text
能服务端止损 -> 先服务端
能配置止损 -> 先配置
能热修 -> 再热修
必须发版 -> 紧急发版
无法止损 -> 下架或强更
```

不要先争论责任。先让影响不再扩大。

---

## 四、热修和热更新

热修适合:

```text
文案错误
H5 页面问题
离线包资源错误
部分 JS / Dart 层逻辑
可通过远程配置关闭的功能
```

不适合:

```text
原生崩溃
签名错误
权限声明错误
Bundle ID / applicationId 错误
证书和 entitlements 错误
审核政策问题
核心安全逻辑变更
```

热更新必须有:

```text
灰度
校验
回滚
版本兼容
生效监控
```

没有回滚的热修,本身可能制造第二次事故。

---

## 五、强更怎么设计

强更适合:

```text
旧版本存在严重安全问题
旧协议已无法服务
支付或登录链路必须升级
合规要求必须阻断旧版本
旧版本会造成数据损坏
```

强更要避免:

```text
把所有小 bug 都强更
无商店包可下载时弹强更
国内渠道版本不一致导致无法升级
弱网下无法进入任何兜底页面
强更弹窗文案不说明原因
```

强更配置至少包括:

```text
平台
版本范围
渠道范围
最低可用版本
下载地址
生效时间
灰度比例
```

强更也是线上功能,要提前测试。

---

## 六、下架和暂停发布

什么时候考虑下架:

```text
启动即崩且影响全量
严重合规或隐私问题
支付、资金、数据风险
错误包已进入商店且无法通过配置止损
签名或渠道污染导致错误用户安装
```

下架不是删除用户手机里的 App。

下架只能阻止新下载或部分更新,已安装用户仍然需要:

```text
服务端降级
强更提示
配置屏蔽
紧急新版本
客服和公告
```

不要把下架当成完整回滚。

---

## 七、签名异常

Android 签名异常常见表现:

```text
无法覆盖安装
渠道后台识别为不同签名
第三方登录 / 支付 / 地图失败
Play App Signing 配置不一致
签名轮换后旧渠道不认
```

iOS 签名异常常见表现:

```text
Archive 成功但上传失败
entitlements 不匹配
推送环境不对
TestFlight 可用但 App Store 包异常
企业包过期
```

处理原则:

```text
先确认问题包的真实签名
再比对渠道后台配置
再确认是否影响覆盖升级
最后决定重打包、补材料、联系渠道或紧急发版
```

签名异常不要凭文件名判断,要用工具验签。

## 八、渠道污染

渠道污染包括:

```text
渠道号写错
A 渠道上传了 B 渠道包
测试包流入生产渠道
国内市场抓取了错误包
第三方下载站二次分发旧包
渠道统计参数互相覆盖
```

预防:

```text
每个渠道产物独立命名
包内写入渠道元信息
上传前自动验包
渠道后台截图或记录留档
上线后按渠道监控安装和崩溃
```

处理:

```text
暂停问题渠道
重新上传正确包
服务端识别并屏蔽污染渠道
通知渠道审核或替换
监控旧包存量
```

国内多渠道项目必须把渠道治理当成发布系统的一部分。

---

## 九、事故沟通

事故期间要有固定信息:

```text
当前影响
已采取止损
下一步动作
预计时间
负责人
需要谁协助
```

不要在群里滚动刷猜测。

对外口径要谨慎:

```text
不暴露内部密钥、证书、渠道后台细节
不承诺未经确认的恢复时间
涉及隐私和支付时走正式合规流程
```

移动端事故经常跨研发、测试、产品、运营、客服、渠道和法务。

---

## 十、复盘要沉淀什么

复盘至少写:

```text
事故时间线
影响版本和渠道
用户影响
根因
为什么测试没发现
为什么发布门禁没拦住
止损是否及时
后续防复发动作
负责人和截止时间
```

复盘不要只写"加强测试"。

要变成:

```text
自动验签
发布前清单
CI 门禁
配置灰度
监控告警
应急预案
```

没有机制变化的复盘没有意义。

## 十一、检查清单

- [ ] 是否能按版本、平台、渠道快速圈定影响面
- [ ] 是否能暂停灰度或分阶段发布
- [ ] 是否有服务端功能开关
- [ ] 是否有强更配置能力
- [ ] 是否能快速下架或暂停问题渠道
- [ ] 是否能验证线上包签名
- [ ] 是否保留每个渠道包的产物和 hash
- [ ] 是否有热更新回滚能力
- [ ] 是否有人发布后值守
- [ ] 是否有事故复盘模板

---

## 十二、结论

移动端事故处理顺序:

```text
定级
圈影响面
止损
修复
验证
恢复发布
复盘固化
```

移动端最难的不是修一个 bug,而是处理已经散落在用户手机和渠道里的错误版本。
