# 网络诊断、证书、代理、弱网与抓包

移动端网络问题经常表现成"接口失败",但根因可能在 DNS、TLS、代理、弱网、证书、网关、系统权限或客户端重试策略。

一句话:**网络诊断要分层,不要一上来就说后端挂了。**

---

## 一、移动端网络链路

一次请求大致经过:

```text
业务代码
  -> HTTP 客户端
  -> DNS
  -> TCP / QUIC
  -> TLS
  -> 代理 / VPN / 网关
  -> CDN / WAF / API Gateway
  -> 服务端
```

任何一层都可能失败。

常见现象:

```text
域名解析失败
连接超时
TLS 握手失败
证书不受信任
请求被代理篡改
弱网下重试风暴
接口成功但业务码失败
```

诊断时先判断失败发生在哪一层。

---

## 二、错误要分类

客户端至少要区分:

| 类型 | 例子 |
| --- | --- |
| DNS 错误 | 域名无法解析 |
| 连接错误 | connect timeout、connection refused |
| TLS 错误 | 证书过期、链不完整、pinning 失败 |
| HTTP 错误 | 4xx、5xx |
| 业务错误 | code != success |
| 解析错误 | JSON 格式不符合预期 |
| 取消请求 | 页面关闭、用户取消 |
| 超时 | 连接超时、读超时、整体超时 |

不要把所有异常都显示成"网络异常"。

对用户可以简化,对日志必须分类。

---

## 三、超时和重试

需要分别设置:

```text
连接超时
读超时
写超时
整体请求超时
```

重试规则:

```text
只重试幂等请求
限制次数
指数退避
加随机抖动
弱网下降低并发
不要重试明确业务失败
```

危险做法:

```text
所有接口失败都重试 3 次
支付、下单、提交表单无幂等重试
多个页面同时重试同一个接口
无网络时持续轮询
```

重试是放大器,写不好会把小故障放大成大故障。

---

## 四、证书和 TLS

移动端证书相关问题:

```text
服务端证书过期
中间证书链缺失
旧系统不支持新根证书
设备时间错误
企业代理替换证书
证书 pinning 配置过旧
测试环境自签证书未隔离
```

证书 pinning 要谨慎:

```text
能防部分中间人攻击
也会提高证书轮换事故风险
必须支持多 pin 和灰度
必须有应急下发或版本预案
```

不要把测试证书信任逻辑带进 release。

---

## 五、代理和抓包

开发诊断常用:

```text
Charles
Proxyman
mitmproxy
Wireshark
Android Studio Network Inspector
Xcode Instruments
```

抓包前要确认:

```text
是否是 debug / staging 环境
是否安装了代理根证书
是否关闭或适配证书 pinning
是否避免采集真实用户敏感数据
是否清理导出的抓包文件
```

生产环境抓包要非常谨慎,抓包文件可能包含 token、Cookie、手机号、地址和订单信息。

## 六、弱网测试

弱网不只是慢。

要覆盖:

```text
高延迟
高丢包
低带宽
网络抖动
从 Wi-Fi 切 4G / 5G
无网恢复
DNS 慢
TLS 慢
接口半开
```

弱网下重点看:

```text
页面是否有 loading 和超时
是否能取消请求
是否重复提交
是否出现重试风暴
是否有离线兜底
恢复网络后是否自动同步
```

弱网测试不是最后点一下飞行模式。

---

## 七、网络日志怎么采

建议记录:

```text
trace id
接口名
域名
状态码
业务码
耗时
错误类型
重试次数
网络类型
App 版本
设备型号
```

不要记录:

```text
完整 token
完整 Cookie
身份证、银行卡
完整地址
明文密码
```

请求体和响应体默认不要全量上报。需要诊断时按白名单、采样和脱敏开启。

---

## 八、网络安全配置

移动端要明确:

```text
是否允许 HTTP 明文
哪些域名允许明文
是否信任用户安装证书
debug 和 release 网络配置是否分离
是否启用证书 pinning
WebView 是否限制混合内容
```

Android 有 Network Security Config。

iOS 有 ATS 相关配置。

不要为了测试方便把全局明文或任意证书信任放进生产包。

---

## 九、什么时候会出事故

常见事故:

```text
证书过期导致全量请求失败
中间证书链配置错误,部分 Android 失败
pinning 没预留新证书,证书轮换后 App 全挂
弱网下重复下单
代理配置被带到 release
HTTP 明文接口被系统拦截
WebView 混合内容导致页面资源加载失败
接口错误被统一吞成网络异常,无法定位
```

网络事故一定要能快速判断是客户端、网络链路还是服务端。

---

## 十、检查清单

- [ ] 网络错误是否按 DNS、连接、TLS、HTTP、业务分类
- [ ] 超时是否区分连接、读写和整体超时
- [ ] 重试是否只用于幂等请求
- [ ] 是否有 trace id 串联客户端和服务端日志
- [ ] release 是否禁止测试证书信任
- [ ] 证书 pinning 是否支持轮换和应急
- [ ] 弱网、断网恢复、网络切换是否测过
- [ ] 抓包文件是否避免真实敏感数据
- [ ] WebView 是否限制混合内容和危险域名
- [ ] HTTP 明文策略是否明确

---

## 十一、结论

网络诊断的顺序:

```text
先分类错误
再看链路层级
再结合 trace id
再复现网络环境
最后修客户端或服务端策略
```

移动端网络不稳定是常态。工程上要做的是分类、兜底、限流和可观测。
