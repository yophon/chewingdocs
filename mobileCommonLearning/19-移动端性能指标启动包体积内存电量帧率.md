# 移动端性能指标:启动、包体积、内存、电量、帧率

移动端性能不是一个指标。启动慢、包太大、内存高、耗电、掉帧,都会直接影响留存和评分。

一句话:**性能优化先定义指标,再定位瓶颈,最后验证收益。**

---

## 一、先定核心指标

常见移动端性能指标:

| 指标 | 看什么 |
| --- | --- |
| 冷启动 | 用户点图标到首个可用页面 |
| 热启动 | App 从后台回前台 |
| 首帧 | 页面首次有效渲染 |
| 包体积 | 安装包、下载包、安装后体积 |
| 内存 | Java/Kotlin、Native、图片、WebView、Dart/JS |
| 帧率 | 滚动、动画、转场流畅度 |
| 电量 | CPU、定位、网络、传感器、后台任务 |
| 网络流量 | 首屏请求量、图片体积、重复请求 |

不要用一个"感觉流畅"替代指标。

---

## 二、启动性能

启动分几段:

```text
进程创建
Application / AppDelegate 初始化
依赖注入和 SDK 初始化
读取本地配置
首个页面创建
首屏接口和资源加载
首帧渲染
```

常见问题:

```text
启动时初始化所有 SDK
启动时同步读大文件
启动时做数据库迁移
启动时等待网络接口
启动页广告和首页逻辑耦合
```

优化原则:

```text
能延迟就延迟
能并行就并行
能缓存就缓存
首屏只做必要工作
初始化按业务分层
```

启动指标要区分冷启动、热启动和首次安装启动。

---

## 三、包体积

包体积要分:

```text
上传包体积
用户下载体积
安装后体积
首次启动新增下载体积
```

常见体积来源:

```text
图片资源
字体
多语言
so / framework
调试符号
重复依赖
WebView 离线包
Flutter / RN 运行时
```

优化手段:

```text
资源压缩
删除未使用资源
按架构拆分
动态资源下发
图片格式优化
字体子集化
依赖审计
```

不要只看安装包。首次打开再下载 200 MB 资源,用户体验一样差。

---

## 四、内存

移动端内存问题常见来源:

```text
大图未压缩直接加载
列表缓存过多
页面引用没释放
WebView 长期持有
数据库游标未关闭
音视频资源未释放
Native 内存泄漏
跨端运行时对象膨胀
```

重点观察:

```text
页面进入前后内存差
返回后是否回落
连续使用 10 分钟是否上涨
低端机是否 OOM
后台回来是否被系统杀
```

内存优化不能只看单页面峰值,还要看长期使用趋势。

---

## 五、帧率和卡顿

用户对卡顿最敏感的地方:

```text
启动到首页
列表滚动
页面转场
输入框输入
图片密集页面
地图和视频页面
弹窗出现
```

常见原因:

```text
主线程任务过长
布局层级复杂
图片解码在主线程
频繁 setState / recomposition
列表 item 重建过多
动画期间发起重活
低端机 GPU 压力大
```

优化规则:

```text
主线程只做 UI
列表 item 稳定复用
图片按展示尺寸加载
动画和重计算解耦
重活放后台线程
```

帧率要在真实设备上测,模拟器不够。

---

## 六、电量

耗电通常来自:

```text
频繁定位
后台任务
长连接保活
高频轮询
音视频采集
蓝牙扫描
大量唤醒 CPU
网络小包频繁发送
```

优化方向:

```text
降低采样频率
合并网络请求
使用系统任务调度
前后台切换时降级
弱网下减少重试风暴
用户不可见时停止动画和采集
```

电量问题最容易被忽略,但用户感知很直接。

---

## 七、网络性能

首屏网络要看:

```text
请求数量
请求串行深度
DNS / TLS / 首包耗时
接口 P90 / P99
图片和静态资源大小
失败重试次数
```

优化规则:

```text
首屏接口合并或预取
缓存稳定配置
图片按需加载
弱网下降级
避免无上限重试
统一超时策略
```

网络慢不只是后端问题。客户端请求编排也会拖慢页面。

---

## 八、性能测试要覆盖设备档位

至少覆盖:

```text
旗舰机
中端机
低端机
旧系统版本
弱网
低电量模式
后台切回前台
连续使用
```

线上指标要按维度拆:

```text
版本
渠道
设备型号
系统版本
网络类型
地区
新老用户
```

平均值没有太大意义。P90 / P99 更接近真实问题。

---

## 九、什么时候会出事故

常见事故:

```text
新版本集成 SDK 后冷启动慢 2 秒
图片资源未压缩导致包体积暴涨
低端 Android 频繁 OOM
首页接口串行导致弱网白屏
后台定位耗电被用户投诉
列表滚动在老机型严重掉帧
WebView 泄漏导致长时间使用后崩溃
热更新离线包过大导致首次启动很慢
```

性能事故通常不是单点 bug,而是多个小成本叠加。

---

## 十、检查清单

- [ ] 是否定义冷启动、热启动、首帧口径
- [ ] 是否采集 P50 / P90 / P99
- [ ] 是否按版本、渠道、设备拆分性能数据
- [ ] 启动阶段是否延迟非必要 SDK
- [ ] 包体积是否有每版对比
- [ ] 图片、字体、so 是否做体积审计
- [ ] 低端机是否测过列表、图片、WebView
- [ ] 后台定位、轮询、长连接是否评估耗电
- [ ] 弱网下是否避免重试风暴
- [ ] 性能回归是否能阻断发布

---

## 十一、结论

性能优化顺序:

```text
先量化
再定位
再优化
再对比
最后固化到发布门禁
```

没有指标,优化只是在猜。

没有门禁,优化成果会在下个版本被重新破坏。
