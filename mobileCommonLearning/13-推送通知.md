# 推送通知:APNs、FCM、厂商推送与 token 生命周期

推送不是"服务端发一条消息"。移动推送依赖系统通道、设备 token、权限、厂商服务和 App 状态。

一句话先记住:**推送 = 用户授权 + 系统通道 + 设备 token + 服务端注册 + 消息策略。**

---

## 一、推送链路

```text
App 请求通知权限
  -> 系统返回 device token / registration token
  -> App 上传 token 到业务后端
  -> 后端保存 userId-device-token 关系
  -> 后端调用 APNs / FCM / 厂商推送
  -> 系统展示通知或唤醒 App
```

任何一段断了,用户都收不到。

---

## 二、iOS APNs

iOS 推送走 APNs:

```text
App
  -> APNs device token
  -> 业务后端
  -> APNs
  -> 设备
```

注意:

- 需要通知权限
- Bundle ID 要启用 Push capability
- profile 要包含 aps entitlement
- token 可能变化
- sandbox / production 环境不同
- 用户关闭通知后 token 不一定立即失效,但通知不会展示

---

## 三、Android FCM 和厂商推送

海外 Android 常用 FCM。

国内 Android 需要考虑厂商推送:

```text
华为 Push Kit
小米推送
OPPO Push
vivo Push
荣耀推送
```

原因:

- 国内很多设备没有 Google Play Services
- 后台进程保活受系统限制
- 厂商通道到达率更高

推送聚合平台可以降低接入成本,但仍要理解底层差异。

---

## 四、Token 生命周期

token 不是永久不变。

可能变化:

- 首次安装
- 卸载重装
- 用户清数据
- 系统或推送服务刷新
- App 恢复备份到新设备
- 推送 SDK 变更

App 每次启动或 token 刷新时要上报:

```ts
registerPushToken({
  userId,
  token,
  platform,
  appVersion,
  deviceId,
});
```

后端要支持同一用户多个设备。

---

## 五、通知类型

| 类型 | 说明 |
| --- | --- |
| 展示通知 | 系统通知栏展示 |
| 静默推送 | 不展示,用于后台刷新 |
| 数据消息 | App 自己处理 |
| 本地通知 | App 本地定时展示 |

不要把所有消息都当营销推送。系统会惩罚滥用通知的 App。

---

## 六、前台、后台、杀进程

App 状态会影响推送行为:

```text
前台:通常由 App 自己决定是否展示
后台:系统展示或唤醒
杀进程:平台限制更多
省电模式:到达可能延迟
```

不要承诺"100% 实时到达"。移动系统不会给你这个保证。

---

## 七、服务端要做什么

后端保存:

```text
userId
token
platform
appVersion
device info
createdAt
lastSeenAt
enabled
```

发送后处理:

- token invalid -> 标记失效
- rate limit -> 重试
- payload too large -> 修正
- 用户退订 -> 不再发送

推送系统必须有日志,否则排障靠猜。

---

## 八、什么时候会出事故

1. iOS entitlements 是 development,生产推送收不到。
2. Android 国内设备没 FCM,只接 FCM 到达率很差。
3. 用户换设备后 token 没更新。
4. 后端一个用户只存一个 token,多设备互相覆盖。
5. 用户关闭通知,运营还以为推送通道坏了。
6. payload 带敏感信息,锁屏泄漏隐私。

---

## 九、检查清单

- [ ] iOS Push capability 是否开启
- [ ] profile 是否包含 aps entitlement
- [ ] sandbox / production 是否区分
- [ ] Android 是否考虑厂商推送
- [ ] token 刷新是否会上报
- [ ] 后端是否支持一人多设备
- [ ] 无效 token 是否清理
- [ ] 推送 payload 是否避免敏感信息
- [ ] 是否有发送日志和到达 / 点击统计

```text
App 状态决定收到后怎么表现
```
