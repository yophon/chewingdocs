# 发布前检查清单

移动端发版不要靠记忆。每次发布都应该有固定检查清单。

一句话:**发布前检查不是流程装饰,是为了避免低级事故进线上。**

---

## 一、发布信息

- [ ] 本次版本号是否确认
- [ ] Android versionCode 是否递增
- [ ] iOS build number 是否递增
- [ ] 版本名称是否和产品、商店后台一致
- [ ] release 分支或 tag 是否确认
- [ ] 本次发布范围是否明确
- [ ] 是否有回滚或止损方案
- [ ] 是否明确发布负责人和值守人

必须能回答:

```text
这个包对应哪个 commit?
这个包准备发哪些渠道?
出了问题谁决定暂停?
```

---

## 二、包身份

- [ ] Android applicationId 是否正确
- [ ] iOS Bundle ID 是否正确
- [ ] App 名称是否正确
- [ ] 图标是否正确
- [ ] 启动图是否正确
- [ ] 渠道标识是否正确
- [ ] 环境配置是否是生产
- [ ] 后端 API 地址是否是生产

最危险的问题:

```text
测试环境配置打进生产包
生产签名打了测试包
渠道包串渠道
```

这些问题必须在上传前查。

---

## 三、签名和证书

Android:

- [ ] release 包是否使用生产签名
- [ ] keystore 指纹是否和渠道后台一致
- [ ] AAB / APK 是否符合目标渠道要求
- [ ] 是否没有使用 debug signingConfig

iOS:

- [ ] Archive 是否使用正确 Team
- [ ] Provisioning Profile 是否匹配 Bundle ID
- [ ] entitlements 是否正确
- [ ] export method 是否正确
- [ ] dSYM 是否归档

签名问题不要等商店报错再查。

---

## 四、功能回归

至少覆盖:

- [ ] 首次安装启动
- [ ] 老版本覆盖升级
- [ ] 登录
- [ ] 退出登录
- [ ] 核心业务流程
- [ ] 支付或下单
- [ ] 推送注册和点击
- [ ] Deep Link / Universal Link / App Link
- [ ] WebView 页面
- [ ] 权限申请和拒绝
- [ ] 弱网和断网
- [ ] 后台切前台

回归要用 release 包。debug 包通过不代表 release 包没问题。

---

## 五、数据和兼容

- [ ] 本地数据库迁移是否测过
- [ ] 老版本升级后数据是否保留
- [ ] 缓存格式变化是否兼容
- [ ] 登录态是否正常迁移
- [ ] 旧配置是否有兜底
- [ ] 服务端接口是否兼容旧 App
- [ ] 新 App 是否兼容当前线上服务端
- [ ] 灰度期间新旧协议是否能共存

移动端不能假设用户都会立刻升级。

服务端接口要同时服务多个 App 版本。

---

## 六、权限和隐私

- [ ] 新增权限是否有产品和合规确认
- [ ] 权限弹窗文案是否准确
- [ ] 拒绝权限后是否有可用降级
- [ ] 隐私政策是否覆盖新采集项
- [ ] App Store 隐私标签是否更新
- [ ] Android 数据安全表单是否更新
- [ ] 第三方 SDK 清单是否更新
- [ ] 是否没有过度采集通讯录、位置、相册等敏感数据

权限问题会直接影响审核和用户信任。

---

## 七、性能和稳定性

- [ ] 冷启动是否没有明显回退
- [ ] 包体积是否没有异常增长
- [ ] 首页首屏是否正常
- [ ] 低端机是否测过核心流程
- [ ] 列表滚动是否流畅
- [ ] 内存是否没有明显泄漏
- [ ] 后台切回是否正常
- [ ] 崩溃率是否在测试阶段可接受

发布前至少对比上一版本:

```text
启动耗时
包体积
核心页面耗时
崩溃和 ANR
```

---

## 八、监控和符号表

- [ ] 崩溃 SDK 是否正常初始化
- [ ] Android mapping 是否上传
- [ ] iOS dSYM 是否上传
- [ ] Flutter / RN 符号表或 sourcemap 是否上传
- [ ] 关键业务异常是否上报
- [ ] 网络 trace id 是否可关联
- [ ] 版本号、build 号、渠道是否进入监控字段
- [ ] 告警规则是否覆盖新版本

没有符号表的新版本不应该发布。

---

## 九、商店和渠道材料

- [ ] 更新说明是否准备
- [ ] 截图是否符合当前版本
- [ ] 隐私材料是否准备
- [ ] 审核账号是否可用
- [ ] 国内应用市场软著、备案、授权材料是否满足要求
- [ ] Google Play / App Store 后台信息是否更新
- [ ] 分阶段发布比例是否确认
- [ ] 发布时间窗口是否避开高风险时段

不要在无人值守时全量发布。

---

## 十、发布后观察

发布后至少观察:

```text
安装成功率
启动成功率
崩溃率
ANR 率
登录成功率
核心接口成功率
支付 / 下单成功率
推送 token 注册成功率
用户反馈和商店评论
```

灰度发布建议:

```text
1% -> 5% -> 20% -> 50% -> 100%
```

每一步都要看数据,不是按时间自动冲到全量。

---

## 十一、发布前最后确认

最后问三句话:

```text
这个包是不是我们要发的那个包?
这个包出了问题能不能快速止损?
这个包上线后有没有人看数据?
```

如果任何一个答案不清楚,不要发布。

---

## 十二、结论

发布检查清单要固定、可执行、可追踪。

最小闭环:

```text
构建可追溯
签名可验证
功能已回归
材料已准备
监控已接入
灰度可暂停
事故有预案
```

移动端发版的底线是:不要把已知不确定性交给用户承担。
