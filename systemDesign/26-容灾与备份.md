# 容灾与备份

容灾和备份是**所有架构里最容易被低估的部分**——平时没人在乎,出事时它是公司命脉。"我们有备份"是真的最便宜的安慰剂——**备份没演练过 = 没有备份**。GitLab 2017 年那次事故就是教训:有 5 套备份,真出事时**5 套全失效或不完整**,丢了 6 小时数据,公司差点死掉。这一篇讲清楚:**RTO / RPO 怎么定义**,备份的几种形式,**两地三中心** 的工程含义,和**故障演练(Game Day)** 这个被严重低估的实践。

> 一句话先记住:**容灾的两个核心指标是 RTO(多久能恢复)和 RPO(能丢多少数据)**——它们决定备份策略。**备份分热 / 温 / 冷三层**,各对应不同 RPO。**两地三中心**不是地理概念,是架构能力——同城两份保 RTO,异地一份保灾难。**最关键的纪律:每季度做一次"假装机房挂了"的真实演练**——平时不练,真出事时所有备份都没用。

---

## 一、容灾的两个核心指标:RTO / RPO

### 1.1 RTO(Recovery Time Objective):多久能恢复

```
故障发生 → 业务恢复
        ↑ 这段时间就是 RTO
```

**典型 RTO 等级**:

| 业务类型 | RTO 要求 |
| --- | --- |
| 内部工具 | 几小时-几天 |
| 中等业务 | 30 分钟-2 小时 |
| 核心业务 | 5-30 分钟 |
| 金融核心 / 支付 | < 5 分钟 |
| 极致(银行 / 电信) | < 1 分钟 |

### 1.2 RPO(Recovery Point Objective):能丢多少数据

```
最近一次有效备份 → 故障发生
                  ↑ 这段未备份的数据可能丢失,就是 RPO
```

**典型 RPO 等级**:

| 业务类型 | RPO 要求 |
| --- | --- |
| 普通业务 | 1 天(每天一次全量备份) |
| 重要业务 | 1 小时 |
| 核心业务 | 几分钟 |
| 金融 | 0(强同步,不能丢) |

### 1.3 RTO 和 RPO 的成本曲线

```
RTO / RPO     成本
1 天          $
1 小时        $$
30 分钟       $$$
5 分钟        $$$$
1 分钟        $$$$$
0 (永不丢)   $$$$$$ (天文数字)
```

**关键决策**:**业务方决定 RTO/RPO,技术方实现**。不要技术拍脑袋"我觉得 RPO 1 小时差不多"——业务说不行就是不行。

## 二、备份的三层

按"恢复速度 vs 成本"分:

### 2.1 热备份(Hot Backup)

**实时同步,RPO 几乎 0,RTO 几秒**:

- MySQL 主从半同步
- Redis 主从复制
- Cell 间双活同步

**成本**:**双倍硬件**(永远有一份在跑),贵。
**适合**:核心业务的灾备。

### 2.2 温备份(Warm Backup)

**定期同步,RPO 几分钟到 1 小时,RTO 几十分钟**:

- 每小时增量备份
- 每天全量备份
- 备份机定期演练启动

**成本**:**0.5-1 倍硬件**(备份系统较小)。
**适合**:中等业务。

### 2.3 冷备份(Cold Backup)

**离线归档,RPO 1 天,RTO 几小时**:

- 每天 / 每周备份到对象存储 / 磁带
- 出事时下载备份 → 部署 → 启动
- 数据可能丢 1 天

**成本**:**极低**(对象存储 / 磁带便宜)。
**适合**:合规归档、历史数据。

## 三、数据备份的具体方案

### 3.1 MySQL 备份

**全量**:
- `mysqldump`(逻辑备份,慢但跨版本)
- `xtrabackup`(物理备份,快但要相同版本)

**增量**:
- 基于 binlog 的增量
- 每小时 / 每天合并到全量

**云上**:
- AWS RDS / 阿里云 RDS 自带"自动备份"
- 开启 PITR(Point-in-Time Recovery)→ 可恢复到任意秒

### 3.2 Redis 备份

- **RDB**:定时快照
- **AOF**:每条写命令追加(更细粒度)
- **AOF + RDB**:两者结合(主流)

### 3.3 对象存储

S3 / OSS 自带:
- **跨区域复制**:自动同步到其他 region
- **版本管理**:文件覆盖了能恢复
- **生命周期**:N 天后自动归档(降成本)

### 3.4 应用配置

代码、配置、密钥**也要备份**——常被忽略:

- **代码**:git 远程仓库 + 本地多人同步(等于多备份)
- **配置**:Nacos / Apollo 自动持久化 + 备份
- **密钥**:Vault + 离线纸质副本(银行级)

## 四、两地三中心

中国监管对金融行业的硬要求,实际上是**所有重要业务的最佳实践**:

```
A 城市:
  ┌──────────┐    ┌──────────┐
  │ 主数据中心 │ ←→ │ 同城灾备  │
  └──────────┘    └──────────┘
      ↕  同城同步 (强一致)
      ↕
B 城市(几百-几千 km 外):
  ┌──────────┐
  │ 异地灾备  │
  └──────────┘
      ↑  异地异步同步 (最终一致)
```

### 4.1 角色

- **同城两中心**:**双活**,实时同步,RTO 几分钟
- **异地一中心**:**冷备 / 温备**,异步同步,RPO 几小时

### 4.2 灾难场景

| 场景 | 切换 | 影响 |
| --- | --- | --- |
| 单机房硬件挂 | 同城自动切换 | RTO 1-5 分钟 |
| A 城市光纤被挖 | 切到 B 城市 | RTO 30 分钟,RPO 几分钟 |
| A 城市地震 | 切到 B 城市 | RTO 几小时(数据可能不全) |

### 4.3 三地五中心(顶级)

阿里、腾讯顶级架构:

```
A 城: 主中心 + 同城灾备
B 城: 双活中心 + 同城灾备
C 城: 异地容灾(只读)
```

**任何单地区故障都不影响业务** —— 但成本是 5 倍硬件。

## 五、为什么"演练"是关键

**没演练过的备份 = 没有备份**——这是行业血泪教训。

### 5.1 真实事故案例

- **GitLab 2017**:误删数据库,5 套备份**全失效**(磁盘满了 / 路径错了 / 版本不兼容 / 权限错了 / 备份是空的),最终丢 6 小时数据
- **Code Spaces 2014**:被攻击者拿到 AWS 密钥,**删了所有数据 + 备份**,公司直接关门
- **某国内电商 2020**:容灾系统部署 2 年没演练过,真灾难来时**切换流程不通**,全平台瘫 4 小时

### 5.2 不演练为什么必出事

```
配置漂移:    主系统升级了,备份系统配置过期
权限丢失:    人员变动,备份相关账号被注销
路径变更:    数据迁移后备份脚本指向旧路径
依赖变化:    依赖的库 / 工具版本不兼容
数据格式:    备份格式变了,恢复脚本读不出
```

**这些问题平时根本看不到——只有演练才能暴露**。

## 六、Game Day(混沌演练)

Netflix 发明的实践——**主动制造故障,验证系统弹性**:

### 6.1 演练等级

```
Level 1:  在测试环境随机杀实例
Level 2:  在生产环境随机杀实例(Chaos Monkey)
Level 3:  生产环境随机断 AZ 的网(Chaos Kong)
Level 4:  生产环境模拟整个机房挂(Game Day)
```

### 6.2 Game Day 流程

```
1. 提前公告(团队都在线)
2. 模拟故障(切流 / 断网 / 杀实例)
3. 观察系统自动恢复 / 切换
4. 没自动恢复的部分 → 人工处理
5. 复盘:哪些地方做的好 / 哪些有问题
6. 修复问题 → 下次演练验证
```

### 6.3 演练频率

- **每月**:小范围演练(单服务 / 单实例)
- **每季度**:中等规模(单 AZ 切换)
- **每半年**:大规模(机房级)
- **每年**:整体灾备演练(异地切换)

> 没演练过的容灾**99% 失败**;演练过的容灾**99% 成功**。

## 七、备份的常见错误

### 7.1 备份和源数据放在同一磁盘 / 机房

磁盘坏 → 备份也坏。机房挂 → 备份也挂。**备份必须物理隔离**。

### 7.2 不验证备份完整性

```
每次备份后:
  自动跑一次"取一小段数据校验",确认能读
  每月做一次"完整恢复演练"
```

不校验 → 备份是空的也不知道,直到真出事。

### 7.3 备份没加密

存到 S3 的备份**默认要加密**。备份被偷 = 数据被偷。

### 7.4 备份的备份

**重要数据要"3-2-1"原则**:

- **3** 份副本
- **2** 种不同介质(SSD + 对象存储)
- **1** 份异地

### 7.5 不测试恢复时间

**RTO 是要测的**——别拍脑袋。真测一遍,从备份到业务恢复要多久?可能比你想的久 10 倍。

## 八、版本回滚:被忽视的"软件容灾"

**故障 80% 来自发版**——回滚是最高频的"容灾"动作。

### 8.1 应用回滚

```
新版本部署 → 监控异常 → 立即回滚到旧版本
```

**前提**:

- 镜像 / 包要保留(N 个版本,可回退)
- DB schema 改动**向后兼容**(旧代码也能跑)
- 配置回滚跟代码同步

### 8.2 配置回滚

```
配置中心(Nacos / Apollo)→ 推送了错配置 → 一键回滚到上一版本
```

### 8.3 DB schema 回滚

最难——**DDL 不可逆**:

- 加列容易,**删列要谨慎**(先双写,确认没业务依赖,再删)
- 改字段类型 = 灾难(用新表 + 双写过渡)

> **生产 DB 改 schema 要分两阶段**——加(向后兼容)→ 业务迁移 → 删(不可逆,确认后才做)。

## 九、监控:容灾的"眼睛"

容灾系统必须有**独立的监控**——主系统挂了主监控也挂,看不到真相。

### 9.1 关键指标

```
主从复制延迟         秒级
备份成功率           每天必须 100%
备份完成时间         不能超过 RPO 一半
跨机房同步延迟       与 RPO 关联
切换演练成功率       100%
```

### 9.2 报警分级

```
P0:可用性掉到 99.9% 以下,立即电话叫醒
P1:复制延迟 > 30 秒,15 分钟响应
P2:备份失败,工作时间响应
P3:配置漂移,周内处理
```

## 十、灾备的成本现实

灾备是**纯成本**——不直接产生收入,但出事时拯救公司。

### 10.1 成本结构

```
硬件:1.5x ~ 3x(备份系统)
人力:专门 SRE 团队
带宽:跨地域同步(贵)
演练:占用工程时间
工具:监控 / 备份软件 license
```

### 10.2 投入比例

```
普通业务:容灾投入 = 总技术成本 × 5-10%
核心业务:容灾投入 = 总技术成本 × 20-30%
金融业务:容灾投入 = 总技术成本 × 50%+
```

### 10.3 ROI 计算

```
出事损失 = 业务收入损失 + 客户流失 + 监管罚款 + 公司声誉
        = 几百万到几亿(单次)

灾备成本 = 一年几百万

→ 一次灾难就回本
```

> **2017 GitLab 事件估计损失千万美元 + 公司差点关门** —— 多投入 100 万搞演练就能避免。

## 十二、踩坑提醒

1. **没定 RTO/RPO 就上备份**——目标不清做出来不知道达标没
2. **备份和源数据共磁盘 / 共机房**——一起死
3. **不验证备份完整性**——出事时才发现备份是空的
4. **不演练**——演练时漏洞百出
5. **没异地副本**——一个城市灾难全完
6. **备份不加密**——泄漏即数据泄漏
7. **依赖单一备份方案**——3-2-1 原则
8. **回滚链路没测**——回滚时发现回不了
9. **DB schema 不向后兼容**——发布失败回滚不了
10. **配置中心没备份**——配置丢了恢复不了
11. **以为云厂商会帮你做**——云厂商的 SLA 是钱,不是你的数据,数据丢了赔不回业务损失
12. **没有"假设最糟"思维**——所有冗余都失效怎么办?要有 Plan Z
