# 熔断降级与隔离

熔断 / 降级 / 隔离这三件事**经常被合在一起讲,实际上是三种不同武器**——熔断是「快速失败」,降级是「兜底返回」,隔离是「故障封闭」。**单独用任何一个都不够,必须三件套同时上**。新手最常见的错是只用其中一个,结果场景错配——给登录接口配熔断没用(它本身没下游)、给监控接口配降级没意义(返回兜底数据没人用)、不做隔离的话**一个非核心服务的故障能拖垮整个系统**。这一篇讲清楚三者的边界、状态机和实战。

> 一句话先记住:**熔断是"它挂了我别再调它,直接失败,免得被拖死"**——对应**下游故障时上游的自我保护**;**降级是"挂了我返回个兜底值,让用户看到东西"**——对应**用户体验保护**;**隔离是"挂了别拖累别人"**——对应**资源池隔离,防止连锁雪崩**。三件套合起来才是完整的「故障隔离」体系——就是 Netflix Hystrix 当年提出的「弹性」(Resilience)。

---

## 一、为什么需要熔断:慢请求比快请求更危险

直觉:**快失败比慢失败友好**——这是熔断的全部哲学。

```
没有熔断:
  下游挂了,但每个请求要等 10s 超时才失败
  上游 200 个线程被全部占用 → 上游也挂
  → 一个下游挂,拖死整条链
  
有熔断:
  下游连续失败 N 次,熔断打开
  之后所有请求 1ms 内立即失败,不再请求下游
  上游线程不被占用 → 上游不挂
  → 故障被隔离在下游
```

> **慢就是死**——慢 10 秒的接口比直接报错的接口危险 100 倍,因为它**占着资源不释放**。熔断的本质是「**与其慢,不如快错**」。

---

## 二、熔断器的状态机:三态

熔断器是一个状态机,**三个状态、四个转换**:

```
        失败率 > 阈值
   ┌──────────────────┐
   │                   ↓
┌────────┐         ┌────────┐
│ Closed │         │  Open  │ ← 拒绝所有请求,快速失败
│ (正常)  │         │  (熔断) │
└────────┘         └────────┘
   ↑                   │
   │ 探测成功            │ 等待 N 秒
   │                   ↓
   │            ┌────────────┐
   └────────────│ Half-Open  │
   探测失败       │ (半开)      │ ← 放少量请求探测下游
                └────────────┘
```

### 2.1 Closed(关闭,正常状态)

**所有请求正常通过**,但同时统计成功 / 失败次数。

判断切换到 Open 的条件:

- 滑动窗口内失败率 > 50%(可配)
- 且窗口内总请求数 > 20(防止"刚启动两次失败就误判")

### 2.2 Open(打开,熔断状态)

**所有请求立即失败,不调下游**——直接走 fallback / 抛异常。

持续时间通常 5-30 秒——给下游恢复的时间。

### 2.3 Half-Open(半开,探测状态)

Open 持续了 N 秒后**自动转 Half-Open**:

- 放**少量**请求(比如 5 个)给下游探测
- 全成功 → 切回 Closed
- 任一失败 → 切回 Open(再等 N 秒)

> **没有 Half-Open 的熔断就不是熔断**——只有这一态能让系统**自动恢复**而不是永远熔断。

## 三、熔断的关键配置

### 3.1 失败率阈值

```
失败率 > 50% 触发熔断
```

**经验值**:

- 核心服务(支付):20-30%(更敏感,稍有问题就熔断)
- 非核心服务(推荐):60-80%(容忍度高)

### 3.2 最小请求数

```
窗口内 < 20 个请求,不熔断
```

避免**冷启动误判**——刚启动就 2 个请求都失败,失败率 100%,但样本太少不可靠。

### 3.3 滑动窗口大小

```
统计最近 10 秒的请求
```

太短(1 秒):噪声大,误判多。
太长(60 秒):反应慢,故障已经扩散。
**经验值:5-10 秒**。

### 3.4 熔断时长

```
Open 状态持续 10 秒
```

太短(1 秒):下游没恢复就探测,反复来回。
太长(5 分钟):明明恢复了还在熔断,业务白损失。
**经验值:5-30 秒**。

## 四、熔断的实现:Hystrix / Sentinel / Resilience4j

### 4.1 Hystrix(Netflix,已停止维护但思想经典)

```java
@HystrixCommand(
    fallbackMethod = "getUserFallback",
    commandProperties = {
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50"),
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),
        @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", value = "5000")
    }
)
public User getUser(Long id) {
    return userClient.getUser(id);
}

public User getUserFallback(Long id) {
    return User.defaultUser();
}
```

**特点**:命令模式 + 线程隔离,**功能强大但侵入性强、性能一般**。Netflix 已经不再维护(改用自家的 concurrency-limits)。

### 4.2 Sentinel(阿里,生态最完整)

```java
@SentinelResource(
    value = "getUser",
    fallback = "getUserFallback",
    blockHandler = "getUserBlock"
)
public User getUser(Long id) { ... }
```

通过控制台动态配规则——**比 Hystrix 灵活,中文友好,中国互联网主流**。

### 4.3 Resilience4j(Hystrix 替代品)

```java
CircuitBreaker breaker = CircuitBreaker.ofDefaults("userService");
Supplier<User> decorated = CircuitBreaker.decorateSupplier(breaker, () -> userClient.getUser(id));
User user = Try.ofSupplier(decorated).recover(throwable -> User.defaultUser()).get();
```

**特点**:函数式风格,Spring Boot 2 后的事实标准,Java 生态首选。

### 4.4 选型对比

| 库 | 维护状态 | 性能 | 生态 | 适合 |
| --- | --- | --- | --- | --- |
| Hystrix | 停更 | 中(线程隔离开销) | 老 Spring Cloud | 老项目,新项目别选 |
| Sentinel | 活跃 | 高 | 阿里全家桶 | 国内项目 |
| Resilience4j | 活跃 | 高 | Spring Cloud 2 | 新 Java 项目 |
| 自研 | - | - | - | 99% 别自研 |

## 五、降级:用户视角的兜底

熔断保护**系统**,降级保护**用户体验**。

### 5.1 降级三种类型

#### 自动降级

下游挂了 → 自动返回缓存 / 默认值。

```python
def get_recommendation(user_id):
    try:
        return rec_service.get(user_id, timeout=100ms)
    except (Timeout, CircuitBreakerOpen):
        return cache.get(f"rec:default") or [hot_item_1, hot_item_2, ...]  # 兜底热门
```

#### 手动降级

大促前主动关闭非核心功能:

- 关闭评价系统(双 11 高峰)
- 关闭推荐算法(用兜底列表)
- 关闭物流详情(返回简化版)

**目标:把全部资源让给核心交易链路**。

#### 限流降级

被限流时不直接 429,而是返回降级数据:

```python
if not rate_limiter.allow():
    return get_cached_default_data()  # 而不是 throw
```

### 5.2 降级的层级

```
正常服务         返回完整、实时数据
↓ 故障开始
缓存兜底         返回 Redis 缓存的旧数据(可能 1-5 分钟前)
↓ 故障加剧
默认数据         返回静态默认值(配置好的备选列表)
↓ 故障严重
直接报错         实在没办法,返回友好错误页
```

> **降级不是一刀切**——分级降级,**先用旧数据,再用默认数据,最后才报错**。

### 5.3 哪些场景必须有降级

| 场景 | 降级方案 |
| --- | --- |
| 推荐列表 | 缓存的热门列表 |
| 商品详情的评论 | 不显示评论(只显示主信息) |
| 用户头像 | 默认头像图片 |
| 商品库存数 | 显示"有货"(不显示具体数字) |
| 实时价格 | 缓存中的最近价格 |

> **有降级的服务在故障时仍能"半开"运行,没降级的服务故障时直接挂**。

## 六、隔离:不让一处坏死拖垮全局

熔断和降级都是**单点保护**,隔离是**资源层面的全局保护**。

### 6.1 线程池隔离(Hystrix 风格)

每个下游服务一个独立线程池:

```
订单服务的总线程 200 个:
  调用用户服务   → 用「用户服务线程池」(50 个)
  调用库存服务   → 用「库存服务线程池」(50 个)
  调用支付服务   → 用「支付服务线程池」(50 个)
  调用推荐服务   → 用「推荐服务线程池」(50 个)
```

**任何一个下游慢/挂,只占用自己池里的 50 个线程,其他 150 个仍能服务**。

**优势**:故障隔离最彻底。
**劣势**:每个池独立维护,**线程切换 + 内存开销大**——只在「线程池间通信」场景下值得。

### 6.2 信号量隔离(轻量替代)

不用独立线程池,**用计数器限制并发**:

```java
Semaphore sem = new Semaphore(50);

if (sem.tryAcquire()) {
    try {
        callUserService();
    } finally {
        sem.release();
    }
} else {
    // 直接拒绝
}
```

**优势**:无线程切换,性能高。
**劣势**:不能强制超时(线程仍在阻塞,只是没新请求进来)——**适合下游延迟可控、纯本地计算的场景**。

### 6.3 集群级隔离(Bulkhead 舱壁)

把同一服务的实例**按业务隔离部署**:

```
订单服务集群:
  C2C 业务一组实例(20 台)
  B2C 业务一组实例(50 台)
  企业业务一组实例(10 台)

C2C 流量爆了 → 只影响 C2C 业务的实例,B2C 的实例继续服务 B2C 用户
```

**这就是「Cell 架构」**——详见 25 篇异地多活。

### 6.4 数据隔离

VIP 用户用一个 Redis 集群,普通用户用另一个 Redis 集群——**普通用户出问题不影响 VIP 体验**。

> 「隔离」是大厂故障演练的核心思想——**永远假设某一部分会挂,绝不能拖垮其他部分**。

## 七、超时设置:被严重忽视的关键参数

熔断的前提是**请求会超时**——不超时就是无限等下游,熔断永远不触发。

### 7.1 超时的层级

```
浏览器/客户端超时    30s
负载均衡超时         15s
应用层 RPC 超时      3s   ← 最关键
DB / Redis 超时      1s   ← 必须比 RPC 短
```

**铁律:下游超时必须 < 上游超时**。如果上游 3s 超时,下游也设 3s,**下游一慢上游也跟着 3s 超时,永远等满**。

### 7.2 经验值

| 调用类型 | 推荐超时 |
| --- | --- |
| 同机房 RPC | 1-3s |
| 跨机房 RPC | 3-5s |
| 缓存(Redis) | 100-500ms |
| 数据库查询 | 1-2s |
| 慢接口(报表) | 10-30s,单独配置 |

### 7.3 只设 connect timeout 不设 read timeout

新手最常犯的错——HTTP 客户端默认只配了连接超时(几秒能连上),**没配读超时**。下游连上但永远不返回 → 上游永远等。

```java
// 错
HttpClient.newHttpClient()  // 默认无 read timeout

// 对  
HttpClient.newBuilder()
    .connectTimeout(Duration.ofSeconds(2))
    .build();

HttpRequest.newBuilder()
    .timeout(Duration.ofSeconds(3))  // 必须设
    .build();
```

## 八、重试:一把双刃剑

故障时重试听起来对,**但乱重试会把故障放大 N 倍**:

```
下游挂了
上游每个请求重试 3 次 → 实际请求量翻 3 倍
下游被加压更厉害,更挂不动
→ 雪崩
```

### 8.1 重试的三个铁律

1. **只重试可重试的错误**——网络超时、5xx 可以重试;4xx(参数错)、业务异常**绝不重试**
2. **指数退避(Exponential Backoff)**——`100ms → 200ms → 400ms → 800ms`,**避免集中重试雪崩**
3. **限制重试次数**——通常 ≤ 3 次,超过就放弃

### 8.2 重试 + 熔断必须组合

熔断打开时,**重试也要立即放弃**——别再撞死下游。多数库默认这么做(Resilience4j 的 retry + circuit breaker 联动)。

### 8.3 重试的幂等要求

**重试默认会重复请求**——下游必须幂等。

不幂等的接口(扣款、生单)**绝不能重试**,否则重复扣款。**详见 23 篇订单与支付**。

## 九、超时 + 熔断 + 降级 + 重试的组合拳

完整的弹性调用模板:

```java
@SentinelResource(
    value = "queryUser",
    fallback = "queryUserFallback",
    blockHandler = "queryUserBlock"
)
@Retryable(maxAttempts = 3, backoff = @Backoff(delay = 100, multiplier = 2))
@Timeout(value = 2, unit = TimeUnit.SECONDS)
public User queryUser(Long id) {
    return userClient.getUser(id);
}

public User queryUserFallback(Long id, Throwable t) {
    return cache.get(id) != null ? cache.get(id) : User.defaultUser();
}
```

**调用流程**:

```
1. 检查熔断器:Open → 直接走 fallback
2. 设置超时:2 秒
3. 调用下游
4. 失败 → 按指数退避重试,最多 3 次
5. 全部失败 → 走 fallback,返回缓存或默认值
```

**这就是"弹性调用"的标准姿势**——任何一个生产 RPC 调用都该长这样。

## 十、监控:故障要看得见

故障管理的前提是**能立刻看到故障**:

### 10.1 必须监控的指标

```
QPS                  当前请求量
错误率               5xx 占比
p50 / p95 / p99 延迟  分位数延迟
熔断器状态            Closed / Open / Half-Open
熔断触发次数          每分钟触发频率
降级命中次数          有多少请求走了 fallback
线程池利用率          隔离池是否打满
```

### 10.2 报警规则

```
错误率 > 5% 持续 1 分钟      → P1 报警
p99 > 1s 持续 2 分钟         → P2 报警
熔断 Open 持续 30 秒         → P1 报警
降级命中率 > 10%             → P2 报警
```

详见 backendLearning/33 监控与可观测性。

## 十一、故障演练:把不可避免的故障"提前到工作时间"

**没演练过的预案 = 没有预案**。

混沌工程在熔断这一层的具体动作:

- **杀掉一个下游实例**:看上游是否快速熔断
- **给某下游加 5 秒延迟**:看上游是否触发超时 + 熔断
- **下游返回 5xx**:看降级是否生效
- **网络分区**:看哪些系统没准备好分区容错

**每月演练一次**——演练时发现问题改,**永远比真实故障时改强**。

## 十二、踩坑提醒

1. **只熔断不降级**——熔断后用户拿到错误,体验崩溃
2. **只降级不熔断**——下游慢仍然占着上游线程,雪崩
3. **不隔离**——一个非核心服务挂能拖垮整个系统
4. **熔断阈值太敏感**——网络抖一下就熔断,正常业务被切
5. **熔断恢复时间太短**——下游没恢复就探测,反复来回
6. **没设 read timeout**——下游慢上游永远等,熔断永远不触发
7. **超时设置同级**(下游 3s 上游 3s)——下游慢上游必跟着超时
8. **乱重试**——下游故障时上游 3 倍流量,加速雪崩
9. **不幂等的接口配自动重试**——重复扣款类事故
10. **熔断后没监控**——下游恢复了不知道,白白触发降级
11. **只在生产配熔断**——测试环境没配,上线后第一次触发就出 bug
12. **不演练故障**——配了一堆熔断器,真挂时发现配错了
