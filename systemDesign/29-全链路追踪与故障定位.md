# 全链路追踪与故障定位

微服务时代有个**反直觉的真相**:服务越拆越细,出事时越难找谁的问题——一个请求过 30 个服务,**任意一个慢/错都能让整体超时**,但你不知道是哪个。**没有全链路追踪的微服务系统就是一团黑盒**。这一篇讲清楚:**Trace / Span / Context 的核心概念**、OpenTelemetry / Jaeger 的工程实现、**可观测性三大支柱(日志 / 指标 / 链路)如何协同**,以及为什么"无法定位的故障 = 永远无法修复"。

> 一句话先记住:**全链路追踪 = 给每个请求贴一个唯一 traceId,贯穿所有服务,让你能从入口到 DB 查询完整还原一次请求的旅程**。**Span 是 trace 里的最小单元**(一次 RPC、一次 DB 查询都是一个 span)。**三大支柱协同**:指标告诉你"出问题了",日志告诉你"具体什么错",链路告诉你"在哪个服务出的"——**缺一个都查不到根因**。

---

## 一、为什么需要全链路追踪

### 1.1 微服务的诊断噩梦

```
用户报告:订单页加载慢

你能查的:
  Web 服务日志 → "调用 order 服务超时"
  Order 服务日志 → "调用 inventory 服务超时"
  Inventory 服务日志 → "调用 promotion 服务超时"
  Promotion 服务日志 → "Redis 慢"
  Redis 监控 → "memory 使用 95%"

你需要查 5 个服务的日志才能找到根因
而每个服务每秒几万条日志,翻找极困难
```

**全链路追踪的价值**:**点一下 traceId,自动展示整个调用链**。

### 1.2 链路图示

```
Trace abc123 (8.2 秒)

[Gateway] -- 8.2s -----------------------------------
   ├── [Auth Service] 50ms ✓
   ├── [Order Service] 8.0s ⚠
   │     ├── [Inventory] 100ms ✓
   │     ├── [Promotion] 7.8s ⚠
   │     │     └── [Redis] 7.8s ⚠ ← 元凶在这里!
   │     └── [Order DB] 30ms ✓
   └── [Logger] 20ms ✓
```

**3 秒内定位根因** —— 这就是链路追踪的力量。

## 二、核心概念:Trace 和 Span

### 2.1 Trace

**一次完整的请求**,从入口到所有下游调用。**每个 Trace 一个全局唯一 traceId**。

### 2.2 Span

Trace 里的**一个工作单元**——一次 RPC、一次 DB 查询、一次方法调用都可以是 span。

```
Span {
    traceId: "abc123",      // 所属 trace
    spanId: "span456",      // 唯一 span 标识
    parentSpanId: "span123", // 上一级 span(形成树形)
    name: "GET /orders/789",
    startTime: 1234567890000,
    duration: 200ms,
    tags: { http.status: 200, ... },
    logs: [ ... ]
}
```

### 2.3 Trace 的树形结构

```
                Span 1 (Gateway)
                 ├── Span 2 (Auth)
                 ├── Span 3 (Order Service)
                 │     ├── Span 4 (Inventory)
                 │     └── Span 5 (Promotion)
                 │            └── Span 6 (Redis)
                 └── Span 7 (Logger)
```

**每个 span 是父 span 的子节点**——形成调用树。

## 三、Context 传播

**关键问题**:Service A 调 Service B,**B 怎么知道自己属于哪个 trace**?

### 3.1 Context 在 HTTP Header 里

```http
GET /api/inventory HTTP/1.1
X-B3-TraceId: abc123
X-B3-SpanId: span456
X-B3-ParentSpanId: span123
```

**B 收到请求后**:
- 从 Header 读取 traceId / parentSpanId
- 创建自己的 spanId
- 处理时把 traceId 也传给下一个调用

**所有跨服务调用**都要传这套 Header,**SDK / Sidecar 自动做**。

### 3.2 跨协议传递

- HTTP:Header
- gRPC:Metadata
- Kafka:消息 Header
- 数据库:SQL 注释 `/* trace_id=abc123 */`(MySQL 慢日志能看到)

### 3.3 进程内传递

同一个进程内,context 在**线程上下文**里(ThreadLocal):

```python
with tracer.start_span("process_order") as span:
    span.set_tag("order_id", 123)
    # 调用其他方法时自动继承 context
    process_payment()
```

## 四、OpenTelemetry:统一标准

```
2010s 早期:Zipkin(Twitter)、Jaeger(Uber)、SkyWalking(Apache)各自做
2019:OpenTracing 和 OpenCensus 合并 → OpenTelemetry(OTel)
2025+:OTel 成为业界统一标准,所有云厂商支持
```

### 4.1 OTel 的三件套

```
SDK     (业务代码 / 自动埋点)
  ↓
Collector (统一接收 / 转换 / 导出)
  ↓
后端    (Jaeger / Tempo / Datadog / 任意)
```

**关键**:**应用代码用 OTel SDK,后端可换**——今天用 Jaeger,明天换 Datadog,代码不动。

### 4.2 自动埋点(Auto Instrumentation)

不用改业务代码:

```bash
# Java
java -javaagent:otel-agent.jar -jar app.jar

# Python
opentelemetry-instrument python app.py
```

Agent 自动给所有 HTTP / gRPC / DB 调用加 span——**零代码侵入**。

### 4.3 手动埋点

复杂业务还是要手动:

```python
from opentelemetry import trace
tracer = trace.get_tracer(__name__)

def process_order(order):
    with tracer.start_as_current_span("validate_order") as span:
        span.set_attribute("order.id", order.id)
        validate(order)
    with tracer.start_as_current_span("calculate_price") as span:
        span.set_attribute("user.tier", user.tier)
        calculate(order)
```

## 五、采样:全采还是采样

100% 采样代价巨大——**每个请求都生成 trace,存储 / 传输爆炸**。

### 5.1 采样策略

| 策略 | 含义 |
| --- | --- |
| 全量(100%) | 测试 / 小流量场景,生产几乎不用 |
| 固定比例(1-10%) | 简单粗暴,主流 |
| 概率采样 | 按 traceId 哈希决定 |
| 限流采样 | 每秒最多 N 条 trace |
| 自适应采样 | 错误的全采,正常的低采 |
| 业务采样 | VIP 用户全采,普通用户 1% |

### 5.2 头部 vs 尾部采样

- **头部采样(Head-based)**:在请求开始时决定要不要采,**简单但可能错过出错的 trace**
- **尾部采样(Tail-based)**:请求结束后看结果决定要不要采,**能保证错误请求都采到,但实现复杂**

> **生产推荐**:头部采样 1% + 尾部采样保证错误 100% 采到。

## 六、可观测性的三大支柱

链路追踪只是其一,完整体系还有:

### 6.1 指标(Metrics)

聚合的数值:QPS、错误率、p99 延迟、CPU、内存。

**Prometheus / Grafana**——**告诉你"系统正不正常"**。

### 6.2 日志(Logs)

文本记录:错误堆栈、业务事件、审计。

**ELK / Loki**——**告诉你"具体发生了什么"**。

### 6.3 链路追踪(Traces)

调用链:一次请求过了哪些服务,各花多久。

**Jaeger / Tempo**——**告诉你"在哪个环节卡住"**。

### 6.4 三者协同

```
报警:        QPS 错误率突然涨到 5%(指标)
点击异常:    跳到该时段的错误 trace 列表(链路)
查看 trace:  发现 user-service 报错(链路)
点 user-service: 跳到对应日志(日志)
看日志详情:  数据库连接超时(日志)
```

**5 步从"系统出问题了"定位到根因**。

### 6.5 关联键

**所有三大支柱用同一个 traceId 关联**:

```
日志: "[traceId=abc123] ERROR DB timeout"
指标: 不直接带 traceId,但能下钻到 trace
链路: 主键就是 traceId
```

打通三者 = **可观测性的工程胜利**。

## 七、查询和分析

### 7.1 查询场景

```
1. 按 traceId 查具体一次请求(用户报障时)
2. 按服务查最近 5 分钟错误率(监控视角)
3. 按延迟查 Top 10 慢请求(性能优化)
4. 按错误类型聚合(类型分析)
5. 按调用链查"调用 X 服务的所有 trace"
```

### 7.2 主流后端

| 工具 | 特点 |
| --- | --- |
| Jaeger | 开源,自建首选,UI 友好 |
| Zipkin | 老牌,简单 |
| SkyWalking | 国产,功能全,APM + 链路 |
| Tempo | Grafana 出品,云原生 |
| Datadog APM | 商用,功能最全 |
| 阿里 ARMS | 国内云厂商 |

> 生产推荐:**OTel SDK + Jaeger 后端**(开源)或 **OTel SDK + Datadog**(商用)。

## 八、链路追踪的工程价值

### 8.1 故障定位

前面例子:**3 秒内定位到 Redis 内存满**。**没有链路追踪要查 30 分钟**。

### 8.2 性能优化

```
拿一周的 trace 数据,分析:
  哪个 RPC 调用最频繁?    → 加缓存
  哪个 DB 查询最慢?       → 加索引
  哪个服务调用次数最多?    → 是不是 N+1 查询
  哪个外部依赖延迟高?     → 替换或异步化
```

### 8.3 容量规划

```
按服务统计:
  user-service 占 30% 调用量
  order-service 占 20%
  → 扩容时按比例

按时间统计:
  晚 8-10 点流量峰值是 5 倍
  → 提前扩容
```

### 8.4 业务洞察

```
追踪关键业务流程:
  下单成功率:99.5%
  下单平均链路长度:7 个服务
  下单平均耗时:200ms
  
出问题立刻知道是哪个服务拖累了
```

## 九、链路追踪的成本和挑战

### 9.1 成本

```
存储:      每条 trace 1-10 KB,1% 采样下每天 TB 级
带宽:      span 数据传到 Collector 占带宽
延迟:      埋点代码增加几 μs 延迟(可忽略)
工程时间:  运维 Jaeger 集群 / 接 OTel SDK
```

> 大公司链路追踪基础设施投入相当于业务系统的 5%-10%——**不便宜,但回报极高**。

### 9.2 实施挑战

- **跨语言**:Java / Go / Python / Node 都要接 OTel
- **跨协议**:HTTP / gRPC / Kafka / DB 都要传 context
- **遗留系统**:老代码可能没 SDK,要改造
- **服务网格**:Istio Sidecar 自动埋点(优势)
- **采样策略**:平衡完整性和成本

## 十、从 trace 到日志的"下钻"

最常用的诊断模式——**看 trace 异常 → 跳到对应日志**:

### 10.1 实现

日志里**必须打印 traceId**:

```python
import logging
from opentelemetry.trace import get_current_span

class TraceFormatter(logging.Formatter):
    def format(self, record):
        span = get_current_span()
        trace_id = format(span.get_span_context().trace_id, '032x')
        record.trace_id = trace_id
        return super().format(record)

logger.error("DB timeout", extra={"trace_id": trace_id})
```

输出:`[2026-05-09 12:34:56] [ERROR] [traceId=abc123] DB timeout`

### 10.2 集中日志查询

ELK / Loki 支持按 traceId 全局搜:

```
trace_id:"abc123" 
→ 跨所有服务的所有日志一次性返回
```

## 十三、踩坑提醒

1. **不接全链路**——出事靠日志大海捞针
2. **traceId 不传到下游**——链路断了
3. **日志不打 traceId**——日志和链路关联不上
4. **100% 采样**——存储和性能都崩
5. **采样太少**(0.01%)——错误的 trace 都没采到
6. **只用头部采样**——错误请求漏采
7. **没设保留期**——存储无限增长
8. **不打通指标 / 日志 / 链路**——三套系统各看各的
9. **链路 UI 不会用**——光部署不培训,工程师不会查
10. **OTel SDK 不升级**——老版本 bug 没修
11. **业务高基数标签放进 metrics**(user_id)——Prometheus 内存爆
12. **没分析历史 trace 数据**——白白产生大量数据,不挖掘价值
