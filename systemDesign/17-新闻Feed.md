# 新闻 Feed

新闻 Feed 和 16 篇的 Timeline 看上去类似——都是「无限往下刷的列表」——**但底层逻辑完全不同**。Twitter Timeline 是**关注关系驱动的时间倒序**;新闻 Feed(今日头条 / 抖音 / 小红书 / Reddit)是**算法推荐驱动的兴趣排序**。**没有"我关注谁",只有"我可能喜欢什么"**——这意味着架构核心从 fanout 变成**召回 + 排序**两阶段流水线,**每一次刷新都重新算一份**。这一篇讲清楚 Feed 工程的真实结构。

> 一句话先记住:**新闻 Feed = 召回(从 1 亿条候选中粗选 1000 条)+ 排序(精排 100 条)+ 重排(打散去重)**。**这是一个机器学习驱动的工程系统**——CTR 模型每天训练,特征实时计算,每个用户每次刷都过一次完整链路。**真正的难点不是模型多牛,是怎么在 200ms 内跑完这条链路 + 抗住每秒几百万次 Feed 请求**。

---

## 一、需求拆解

### 1.1 功能需求

- **打开 App / 刷新**:返回个性化 Feed(20 条左右)
- **下拉加载更多**:返回下一批
- **互动反馈**:点赞 / 评论 / 转发 / 看时长 → 影响后续推荐
- **不感兴趣 / 屏蔽**:过滤特定内容 / 作者
- **关注流(可选)**:也能切到"只看关注的人"

### 1.2 非功能需求

- **延迟**:Feed 加载 < 300ms(超过用户就走)
- **可用性**:99.99%
- **个性化精度**:CTR 越高越好(直接关系到收入)
- **规模**:DAU 5 亿(抖音级),Feed 请求峰值 500 万 QPS
- **内容池**:百万 ~ 千万级活跃内容

### 1.3 这一轮不做

- 推荐模型本身的训练(那是 ML 团队的事)
- 视频转码 / 编码(详见 19 视频流)
- 评论详情(独立子系统)

## 二、容量估算

### 2.1 QPS

```
Feed 请求:DAU 5 亿 × 人均刷 100 次 = 500 亿次/天
         平均 QPS = 50 万,峰值 ~500 万

每次 Feed 返回 20 条 → 后端实际处理 500 万 × 20 = 1 亿条/秒的内容评估
→ 单条评估必须 < 100μs
```

### 2.2 存储

```
内容池:
  活跃内容 1000 万条 × 1 KB(标题 + 元数据) = 10 GB
  历史归档 100 亿条 × 1 KB = 10 TB
  
用户特征:
  5 亿用户 × 5 KB(标签 / 历史行为 / 兴趣向量)= 2.5 TB
  → Redis Cluster + 大数据存储

行为日志:
  每用户每次刷点击 10 条 = 50 亿条/天
  每条 200 B = 1 TB/天
  → Kafka → 数仓 / ClickHouse
```

### 2.3 关键洞察

**和 Twitter 不同,新闻 Feed 不是 fanout 主导,而是排序主导**——每次刷都要给这个用户独立计算一份。**写不大,但每次读做了大量计算**。

## 三、数据模型

### 3.1 内容池

```
key: content:{content_id}
value: {
    id, author_id, content, media_url,
    tags, category, lang, region,
    quality_score,           # 内容质量分(由评分模型给)
    created_at,
    base_features            # 内容侧特征(预计算)
}
```

实际上**几个 ClickHouse / ES 表 + Redis 热缓存**。

### 3.2 用户画像

```
key: user_profile:{user_id}
value: {
    age, gender, region, device,
    interest_tags: ["数码", "美食", ...],
    interest_vector: [0.1, 0.5, ...],  # 兴趣 embedding
    history: [recent_clicked_content_ids],
    blacklist: [屏蔽的作者 / 内容]
}
```

### 3.3 用户行为日志(实时)

```
点击 / 看 / 停留时长 / 点赞 / 评论 → Kafka
→ Flink 实时聚合到 Redis 当用户的最新画像
```

### 3.4 候选索引(召回侧)

```
按标签:    tag:{tag_name} → ZSet(content_ids by score)
按地域:    region:{city}  → ZSet
按热度:    hot:global     → ZSet(过去 1 小时点击量排序)
按相似:    similar:{content_id} → List(向量召回的相似内容)
```

## 四、核心架构:召回 + 排序两阶段

```
用户请求 Feed
     ↓
┌──────────┐    粗筛
│  召回层   │ ────── 从 1 亿候选 → 1000 条
└──────────┘
     ↓
┌──────────┐    精排
│  排序层   │ ────── CTR 模型打分 → 100 条
└──────────┘
     ↓
┌──────────┐    重排
│  重排层   │ ────── 打散 / 多样性 / 去重 → 20 条
└──────────┘
     ↓
   返回给用户
```

### 4.1 为什么要两阶段(其实是三阶段)

**模型不能直接评 1 亿候选**——CTR 模型计算复杂,每条 1ms 的话 1 亿条要 100 秒。**必须先粗筛**:

```
召回:  快速、规则化,1ms 选 1000 条
排序:  精确、模型化,100ms 评 1000 条
重排:  业务规则,10ms 调 100 条
```

---

## 五、召回层:并行多路召回

不是单一策略,**并行跑多路召回,每路出 100-200 条,合并去重**:

### 5.1 常见召回策略

| 策略 | 思路 | 适合 |
| --- | --- | --- |
| 兴趣标签召回 | 用户兴趣 = ["数码"],拉数码标签下的热门 | 主流 |
| 地域召回 | 同城内容 | 本地化 |
| 协同过滤 | 和我类似的人喜欢什么 | 经典 |
| 向量召回(双塔) | 用户向量 × 内容向量 余弦相似度 | **现代主流** |
| 热度召回 | 全局热门 / 上升热门 | 兜底,新用户 |
| 关注召回 | 关注的作者发的(类似 Twitter) | 关注流 |
| 多模态召回 | 视频内容 + 标题 + 评论联合 embedding | 抖音/小红书 |

### 5.2 双塔模型(Two-Tower)

**现在工业界的主流召回方法**:

```
用户塔:   user_id, 历史, 兴趣 → 神经网络 → 用户向量(128 维)
内容塔:   content_id, 标签, 内容 → 神经网络 → 内容向量(128 维)

召回 = 找到和"用户向量"最相似的 K 个"内容向量"
```

**线上实现**:

- 内容向量**离线批量算好**,存到向量数据库(Faiss / Milvus / 阿里 PAI)
- 用户向量**实时算**(请求来了过一次用户塔)
- 用 ANN(近似最近邻)算法查 Top 1000

详见 backendLearning/48 向量数据库。

### 5.3 召回的并行执行

```python
async def recall(user_id):
    tasks = [
        interest_recall(user_id),       # 100 条
        vector_recall(user_id),         # 200 条
        hot_recall(user_id),            # 100 条
        region_recall(user_id),         # 100 条
        cf_recall(user_id),             # 100 条
    ]
    results = await asyncio.gather(*tasks)
    return dedupe(merge(results))[:1000]
```

**所有路并行,总耗时 = 最慢一路** ≈ 20ms。

## 六、排序层:CTR 模型

把召回的 1000 条**逐条打分**——给每条预测「这个用户点击它的概率」。

### 6.1 模型演进

```
2010 - 2015:LR(逻辑回归),特征工程驱动
2015 - 2018:GBDT + LR
2018 - 2020:Wide & Deep / DeepFM(深度学习)
2020 - 2024:DIN / DIEN(注意力机制)
2024 - 2026:Transformer-based,大模型蒸馏
```

**今天主流是「深度学习 CTR 模型」**——抖音、快手、Meta、Twitter 都用类似架构。

### 6.2 排序服务的工程挑战

```
1000 条 × 1ms / 条 = 1 秒,太慢
↓ 优化
模型推理 GPU 加速 + Batch 推理 → 100ms
↓ 优化
特征预取 + 缓存 → 50ms
↓ 优化
模型蒸馏(大模型→小模型)→ 30ms
```

**线上推理用 TensorFlow Serving / Triton / 自研推理引擎**。

### 6.3 实时特征 vs 离线特征

```
离线特征:        每天批量算
  - 用户标签
  - 内容质量分
  - 历史平均 CTR

实时特征:        毫秒级
  - 最近 5 分钟点击的内容
  - 最近 1 小时滑动行为
  - 当前会话的兴趣偏移

实时特征用 Flink 算,存 Redis,排序时拼一起
```

---

## 七、重排层:业务规则

排序模型不知道**业务约束**:

### 7.1 多样性

不能给用户连续推 20 条数码——**类目打散**:

```python
# 简单实现:同类目最多连续 N 条
sorted_list = enforce_diversity(scored_items, max_continuous=2)
```

### 7.2 去重

最近 7 天看过的不再推:

```python
seen = redis.smembers(f"seen:{user_id}")
result = [item for item in result if item.id not in seen]
```

### 7.3 商业混排

广告 / 商家内容按比例插入:

```
20 条结果中 = 18 条自然内容 + 2 条广告
广告位置 = 第 4 条 + 第 12 条(经验位置)
```

### 7.4 反作弊 / 风控

低质 / 标题党 / 政治敏感内容降权或过滤。

### 7.5 冷启动保护

新内容投放后没数据,**强制曝光一段时间收集 CTR 信号**。

## 八、关键工程问题

### 8.1 怎么在 300ms 内跑完?

预算分配:

```
网络往返:       30ms
鉴权 + 路由:     10ms
召回(并行):     30ms
特征拼装:       20ms
排序(GPU 推理): 80ms
重排:          10ms
内容详情读取:    20ms
序列化 + 网络回:  30ms
缓冲:          70ms
─────────────────
总计:          300ms
```

**任何一步超时 → 触发降级**(用上次缓存的结果 / 用兜底热门)。

### 8.2 缓存策略

```
不能粗暴缓存整个 Feed 结果(因人而异)
但能缓存中间步骤:
  - 召回结果(用户级,1 分钟 TTL)
  - 内容详情(全局,1 小时 TTL)
  - 用户特征(用户级,5 分钟 TTL)
```

### 8.3 滚动加载("无限刷")

```
首次请求:返回 20 条,带 cursor
下拉刷新:用 cursor 请求,返回下一批 20 条
cursor 包含:已看过的内容 ID 列表 + 时间偏移
```

**用户连续刷 100 次 → 累积 cursor 太大 → 截断只保留最近 200 条已看 ID**。

### 8.4 实时反馈进推荐

```
用户点了一条 →
  写 Kafka →
  Flink 实时更新用户画像 →
  下一次请求 Feed 时反映兴趣变化
```

**整个回路 < 5 秒**——这就是抖音"算得准"的工程基础。

## 九、冷启动:新用户、新内容怎么办

### 9.1 新用户冷启动

完全没行为数据:

- **基础信息推荐**:性别 + 年龄 + 地域 + 注册渠道 → 给个粗类目推荐
- **热门兜底**:全局热门 + 高质量内容
- **快速学习**:前 10 条故意多样化,根据用户点击快速调整

### 9.2 新内容冷启动

刚发的内容没 CTR 数据:

- **内容侧特征**:标题、作者历史 CTR、类目质量
- **强制曝光**:给一定流量收集数据
- **小流量探索**:多臂老虎机算法决定试探多少

## 十、A/B 测试系统

新模型 / 新策略上线**必须 A/B**:

```
1. 流量分桶:按 user_id 哈希分 100 个桶
2. 对照组(50 桶):用旧策略
3. 实验组(50 桶):用新策略
4. 一周后比较 CTR / 时长 / 留存,显著提升才全量
```

**这是 Feed 工程的核心实验平台**——抖音 / 字节有上千个并行实验在跑。

---

## 十一、热点事件:突发流量打挂 Feed

地震 / 重大事件爆发 → 全国用户都来刷 → Feed QPS 翻 10 倍。

**应急方案**:

- **降级排序模型**:换轻量模型,只用基础特征
- **降级召回**:只保留兜底热门召回
- **缓存延长**:Feed 结果短缓存 30 秒,**多个用户共享同一个结果**(承认精度下降)
- **限流 + 排队**:返回部分用户"加载失败,请重试"

## 十二、最终架构图

```
                         用户刷 Feed
                              ↓
                          API Gateway
                              ↓
                          推荐服务
                              ↓
            ┌─────────────────┼─────────────────┐
            ↓                 ↓                 ↓
       兴趣召回            向量召回           热门召回
        (Redis)           (Faiss)          (Redis ZSet)
            └─────────────────┼─────────────────┘
                              ↓
                          合并去重
                              ↓
                           特征拼装
                       (Redis 实时特征)
                              ↓
                          排序服务(GPU)

                              ↓
                           重排
                       (多样性 / 广告 / 风控)
                              ↓

行为流:用户行为 → Kafka → Flink → Redis(实时画像)+ Hive(离线训练)
模型流:Hive → 模型训练(每天)→ Model Registry → 推送线上推理
```

## 十三、踩坑提醒

1. **召回不并行**——5 路串行 100ms × 5 = 500ms 直接超时
2. **排序模型太重**——单条 5ms,1000 条 = 5s,用户都走了
3. **没分阶段**——直接给 1 亿条用模型评分,资源浪费
4. **特征不实时**——用一小时前的兴趣推荐,精度差
5. **不去重**——用户连续看到同一篇文章
6. **不打散类目**——20 条全是同一类目,用户审美疲劳
7. **新用户没冷启动方案**——首屏白屏 / 都是热门内容
8. **A/B 测试样本不够**——分桶太小,效果不显著就上线
9. **不监控 CTR 下跌**——模型上线后 CTR 反而降,要立刻回滚
10. **缓存粒度错**——缓存整个 Feed 结果(因人而异),命中率 0
11. **冷启动用全量模型**——新用户没特征,模型给乱七八糟的结果
12. **降级方案没准备**——模型服务挂直接 5xx,用户体验崩
