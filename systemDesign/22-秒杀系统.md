# 秒杀系统

秒杀是**电商系统里压力最大、边界最清晰的场景**——一秒内 100 万人抢 1000 件商品,**99.9% 的请求注定失败,但系统必须扛得住所有请求的冲击**。秒杀是少数"应该和主站隔离独立部署"的业务——**它和正常下单的逻辑完全不同**:库存极少、流量极大、防作弊要求高、且**不能影响主站正常交易**。这一篇讲清楚秒杀系统的核心三件事:**怎么不超卖、怎么挡住假流量、怎么让 DB 不被打挂**。

> 一句话先记住:**秒杀的本质是"用 Redis 当库存,用 MQ 当订单缓冲,用尽一切手段把流量挡在 DB 之前"**。**绝对不能用普通下单链路搞秒杀**——主站设计是面向"正常购买",秒杀是"瞬时洪水",两者的取舍完全相反。**核心三防**:防超卖(原子扣减)、防机器(各种限制)、防雪崩(分层降级)。

---

## 一、需求拆解

### 1.1 功能需求

- **预约 / 抢购**:用户预约商品,开抢时点抢购按钮
- **库存扣减**:不能超卖
- **下单**:抢到的用户生成订单
- **支付**:订单生成后规定时间内支付
- **回流**:超时未支付的库存返还

### 1.2 非功能需求

- **并发**:开抢瞬间 100 万 QPS
- **可用性**:秒杀挂了不能影响主站
- **公平性**:先到先得(或公平随机)
- **不超卖**:绝对不能多卖

### 1.3 这一轮不做

- 商品详情页(独立子系统)
- 物流配送
- 售后退款

## 二、容量估算

### 2.1 流量

```
预约期:100 万人预约
开抢瞬间:100 万人同时点击 → 100 万 QPS(峰值持续 1 秒)
持续期:开抢后 30 秒内陆续点击,平均 5 万 QPS

商品库存:1000 件 → 99.9% 的请求必然失败
但 100% 的请求都要被服务端处理
```

### 2.2 数据规模

```
单场秒杀:
  商品数 1-10 个
  库存 1000-10000 件
  参与人数 100 万
  
活动级:每天几百场秒杀,各场之间隔离
```

### 2.3 关键洞察

**秒杀的"业务量"其实很小**——几千件商品而已。**真正的难点是"流量噪声"** —— 99.9% 的请求都要被扛住但都注定失败,要在它们打到核心库存逻辑之前过滤掉。

## 三、为什么秒杀必须独立部署

### 3.1 主站是为"正常交易"设计的

```
正常下单流程:
  1. 浏览商品  
  2. 加购物车
  3. 选地址 / 优惠券 / 发票
  4. 创建订单(写 MySQL)
  5. 调用支付
  6. 减库存(写 MySQL)
  
单次下单耗时 1-2 秒,QPS 几千
```

### 3.2 秒杀流量打主站会发生什么

```
100 万 QPS 涌入下单接口
→ 应用线程池打满
→ DB 连接打满
→ MySQL 锁竞争(库存行被疯狂 SELECT FOR UPDATE)
→ MySQL 挂
→ 主站全瘫
→ 不止秒杀失败,普通下单也下不了
```

**所以**:秒杀系统**单独部署**,**单独 DB**,**单独 Redis**,**单独网关**——和主站物理隔离。

---

## 四、核心防线 1:防超卖

**绝对不能多卖**——卖多了要赔钱,商家投诉。

### 4.1 错误方式:DB 直接扣

```sql
SELECT stock FROM product WHERE id = 1;  -- 还有 1 件
UPDATE product SET stock = 0 WHERE id = 1;
INSERT INTO order ...;
```

**问题**:并发下两个请求都看到 `stock = 1`,都扣成 0,**超卖**。

### 4.2 错误方式:DB 加事务

```sql
BEGIN;
SELECT stock FROM product WHERE id = 1 FOR UPDATE;  -- 行锁
UPDATE product SET stock = stock - 1 WHERE id = 1 AND stock > 0;
COMMIT;
```

**正确但极慢**——单行锁导致**串行化**,1000 件库存只能 1 秒卖出几百件,**100 万请求全在排队**,DB 直接挂。

### 4.3 正确方式:Redis 预扣 + 异步落单

```python
# Redis 用 DECR(原子操作)
remaining = redis.decr(f"stock:{product_id}")
if remaining < 0:
    redis.incr(f"stock:{product_id}")  # 还回去
    return "已售罄"
else:
    # 入队,异步生成订单
    mq.send("CreateOrder", {user_id, product_id})
    return "排队中,稍后通知"
```

**关键**:

- `DECR` 是原子操作,**不会超卖**
- 不查 DB,**纯 Redis 内存操作,O(1)**
- 真正下单走 MQ 异步,DB 压力分散

### 4.4 Lua 脚本保证原子(更稳)

```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock > 0 then
    redis.call('DECR', KEYS[1])
    return 1
else
    return 0
end
```

```python
allowed = redis.eval(LUA_SCRIPT, 1, f"stock:{product_id}")
```

**比 DECR + INCR 更稳**——一气呵成,**没有中间状态**。

## 五、核心防线 2:挡住流量

100 万 QPS 不能全打到秒杀核心逻辑——**层层挡掉**。

### 5.1 第 1 层:静态资源 CDN

商品页 / 秒杀按钮 / 图片 → 全部 CDN 缓存。**用户访问商品页根本不打源站**。

### 5.2 第 2 层:网关限流

```nginx
# 单 IP 每秒最多 10 次
limit_req_zone $binary_remote_addr zone=seckill:10m rate=10r/s;

# 单 user_id 每秒最多 1 次
limit_req_zone $cookie_user_id zone=user:10m rate=1r/s;
```

**直接挡掉机器刷的几十倍流量**。

### 5.3 第 3 层:验证码 / 答题

抢购前必须**输入验证码**或**答题**:

- 把所有人的"按钮被按下时间"打散到几秒到几十秒内
- 机器答题成本高,刷子被挡

> 12306 抢票的「答题」就是这么来的——**让 100 万人不在同一秒涌入**。

### 5.4 第 4 层:库存预热到 Redis

开抢前,**库存数据从 DB 推到 Redis**:

```python
# 预热脚本(开抢前 5 分钟跑)
for product in seckill_products:
    redis.set(f"stock:{product.id}", product.stock)
    redis.expire(f"stock:{product.id}", 3600)
```

**整个秒杀期间不查 DB**,只操作 Redis。

### 5.5 第 5 层:本地缓存挡部分

如果商品已经"售罄"——**应用本地缓存这个状态 1 秒**,期间所有请求直接返回"已售罄":

```java
LoadingCache<Long, Boolean> soldOut = Caffeine.newBuilder()
    .expireAfterWrite(1, TimeUnit.SECONDS)
    .build(productId -> redis.get("sold_out:" + productId) == "1");

if (soldOut.get(productId)) {
    return "已售罄";
}
// 否则才去 Redis
```

**几十万 QPS 直接被本地缓存挡**,Redis 都不打到。

### 5.6 流量过滤总效果

```
用户原始请求    100 万 QPS
↓ CDN 静态资源    挡掉 50%
↓ 网关限流          挡掉 90% 剩余
↓ 验证码 / 答题     挡掉 80% 剩余
↓ 本地缓存售罄态   挡掉 90% 剩余
↓ Redis 预扣库存
↓ MQ 入队         真正进入下单流程

最终打到 MQ 的只有几千个有效请求
```

**这就是分层挡流量的威力** —— 100 万压成几千,系统就不挂。

## 六、核心防线 3:DB 不被打挂

经过前面几层过滤,**DB 实际只承担"成功用户的下单写入"**,QPS 几千以内。

### 6.1 异步下单流程

```
用户抢成功(Redis 预扣 OK)
   ↓
MQ 入队
   ↓ (Worker 慢慢消费)
生成订单 → 写 DB
   ↓
通知用户(WebSocket / 推送)"下单成功,请支付"
   ↓
30 分钟内不支付 → 订单取消,Redis 库存返还
```

### 6.2 排队体验

用户点抢购后**立即返回**:

```
抢成功 → "正在为您生成订单,请稍候 5 秒..."
3-5 秒后 → 推送"下单成功,跳转支付"
```

**用户不阻塞**——服务端慢慢处理,前端轮询或推送通知。

### 6.3 库存最终对账

Redis 库存和 DB 库存要**最终一致**:

```python
# 每分钟对账
redis_stock = redis.get(f"stock:{product_id}")
db_stock = db.query("SELECT stock FROM product WHERE id = ?", product_id)

if redis_stock != db_stock:
    alert("库存不一致!")
    # 人工介入或自动修正
```

**为什么会不一致**:Redis 挂过 / 网络抖动 / 程序 bug → 必须有兜底对账。

## 七、防机器刷单

机器人 / 黄牛**永远比真人快**——必须各种手段限制。

### 7.1 用户身份验证

- 必须实名(手机号 + 身份证)
- 必须老用户(注册超过 30 天)
- 必须正常账户(无封禁记录)

### 7.2 行为限制

- 单用户每场只能抢 1 件
- 单 IP 每分钟限 N 次
- 单设备每天限 N 次(设备指纹)

### 7.3 答题 / 滑块验证

强制行为成本——机器要适配每个题目成本高。

### 7.4 风控模型

```
风险分 = f(注册时间, 历史行为, 设备指纹, IP, 浏览器特征, ...)
高风险用户:抢购请求直接拒绝(假装"已售罄")
```

### 7.5 影子库存

不告诉用户「真实库存」——机器人无法精确攻击:

- 真实库存 1000
- 显示库存 999(扣 1 个就显示 998)
- 实际机制是按时间分批放出

**让机器人摸不准节奏**。

## 八、活动配置与回放

### 8.1 活动配置

```sql
CREATE TABLE seckill_activity (
    id          BIGINT PRIMARY KEY,
    product_id  BIGINT,
    stock       INT,
    start_time  TIMESTAMP,
    end_time    TIMESTAMP,
    price       DECIMAL,
    user_limit  INT       -- 单用户最多买几件
);
```

### 8.2 活动开关

在配置中心放一个**总开关**:

- 出问题立刻关掉(避免越来越糟)
- **逐步放量**:先放 10% 用户进,平稳后再开全部

### 8.3 演练

**真实秒杀前必须压测 + 演练**:

- 用 wrk / jmeter 模拟 100 万 QPS
- 找出瓶颈,扩容
- 模拟"半路 Redis 挂了"看降级

> **没演练过的秒杀必出事**。淘宝、京东每次大促前都演练 N 遍。

## 九、降级方案

秒杀涉及多个组件,**任何一个挂都要有 Plan B**:

| 组件 | 挂了怎么办 |
| --- | --- |
| Redis | 切到备用 Redis,或紧急用本地内存计数(失公平) |
| MQ | 直接同步写 DB(降低吞吐,但能跑) |
| DB | 已成功的请求都已经返回成功,DB 慢慢恢复后补单 |
| 应用 | LB 自动剔除,流量切到剩余实例 |

**最重要的降级**:**主站永远在,秒杀挂了不影响"正常下单"**——这是物理隔离的最大价值。

## 十、最终架构图

```
                       用户
                         ↓
                    秒杀域名
                  (独立 DNS)
                         ↓
                    CDN(静态资源)
                         ↓
                  秒杀专用网关
                  (限流 / 验证码)
                         ↓
                  秒杀应用集群
                  (本地缓存售罄态)
                         ↓
                  Redis Cluster
                  (库存预扣)
                         ↓ (扣成功)
                       Kafka
                         ↓
                  下单 Worker
                         ↓
                  MySQL(秒杀专用)
                         ↓
                  通知用户
```

**核心特征**:

- 全链路独立部署
- Redis 是核心(不是 DB)
- 异步消费下单
- 多层挡流量

## 十一、电商主站如何接入秒杀

抢购成功 → 用户进入主站完成支付:

```
1. 秒杀系统生成订单 → 主站订单表(共享一份)
2. 用户跳转到主站支付页
3. 主站完成支付 → 通知秒杀系统"支付成功"
4. 物流 / 售后走主站正常流程
```

**只有"创建订单"这一步用秒杀系统独立扛**,后续流程主站接手。

---

## 十二、踩坑提醒

1. **用主站正常下单链路做秒杀**——必崩
2. **DB 直接扣库存**——超卖或锁竞争挂 DB
3. **不预热 Redis**——开抢瞬间所有请求查 DB
4. **不限流**——机器人刷死系统
5. **不做异步下单**——同步生成订单 DB 打挂
6. **没本地缓存售罄态**——已售罄状态仍打 Redis
7. **不对账**——Redis 和 DB 库存不一致没人发现
8. **没降级方案**——任一组件挂全失败
9. **不演练**——正式秒杀时翻车
10. **不做风控**——黄牛抢光真实用户体验崩
11. **库存放 DB 不放 Redis**——慢
12. **不监控**——流量过来后不知道出了什么问题
