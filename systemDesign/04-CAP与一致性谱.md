# CAP 与一致性谱

CAP 是被讲得最烂、被理解得最差的分布式定理。**99% 的人都背得出 "Consistency / Availability / Partition tolerance 三选二",但 99% 的人都答错了"二选什么"**——你不能"选 CA 放弃 P",因为 P 是网络分区,**它不由你选,由物理决定**。这一篇把 CAP 的工程含义讲清楚,然后告诉你**真实工程不在 0/1 二选,而在一个连续光谱上**——从强一致到最终一致中间有 5-6 档,选错一档系统就崩。

> 一句话先记住:**CAP 真正的含义是"网络分区发生时(P 不可避免),你必须在一致性 C 和可用性 A 之间二选一"**——不是平时三选二,是分区时二选一。**BASE 是 CAP 的实战版本**——「基本可用 + 软状态 + 最终一致」,这套思路撑起了几乎所有互联网公司。**一致性是个光谱**,不是开关——选错档位,要么超时炸要么数据脏。

## 一、CAP 的真正含义(教科书没讲清的)

CAP 三个字母:

- **C(Consistency)**:所有节点同一时间看到同一份数据(线性一致性)
- **A(Availability)**:每个请求都能在合理时间内拿到非错误响应
- **P(Partition tolerance)**:网络分区时,系统仍能继续工作

教科书说"三选二",真相是:

> **P 不是选项,是事实**。只要你跑在网络上,分区就一定会发生(机房交换机抖一下、跨地域光纤被挖掉、防火墙规则配错)。**所以选择只在 C 和 A 之间——分区时,你是宁可不可用(等数据一致),还是宁可不一致(继续服务)?**

```
                 网络正常时
            ┌───────────────────┐
            │  C 和 A 都能保证   │
            └───────────────────┘

                 网络分区时
        ┌──────────────────────────┐
        │ 必须在 C 和 A 中二选一    │
        │                          │
        │ 选 C(放弃 A):           │
        │   分区时拒绝请求          │
        │   例: ZooKeeper / etcd  │
        │                          │
        │ 选 A(放弃 C):           │
        │   分区时允许不一致        │
        │   例: Cassandra / Dynamo│
        └──────────────────────────┘
```

### 1.1 「CA 系统」是个伪概念

很多人说「MySQL 是 CA」——**胡说**。MySQL 不能容忍分区,**主从分区时要么主库失联(不可用),要么强行让从库提供过期数据(不一致)**,根本逃不开 CAP。

> 真实情况是:**没有真正的 CA 系统**(除非系统只跑在一台机器上,而那就不是分布式系统)。所有分布式系统都在 CP 和 AP 之间选。

### 1.2 CAP 不是"系统级"选择,是"操作级"选择

更精细的理解是:**同一个系统,不同操作可以选不同侧**。

| 操作 | 选 | 理由 |
| --- | --- | --- |
| 转账 / 扣库存 | C(强一致) | 不能多扣 |
| 浏览商品列表 | A(可用性) | 看到旧数据没事,系统挂了流失大 |
| 点赞数 / 关注数 | A(可用性) | 几秒不一致用户感知不到 |
| 用户登录验证 | C | 黑名单同步要快 |

> **不要把 CAP 当成"我们公司是 CP 还是 AP 公司"的标签**,这是初学者最大的误解。**真实的工程是:每一个操作单独选**。

## 二、BASE:CAP 的工程化版本

BASE 是 eBay 工程师对 CAP 的实战诠释,几乎所有互联网公司都在用:

```
B - Basically Available  基本可用
A - Soft state           软状态
E - Eventual consistency 最终一致
```

### 2.1 Basically Available(基本可用)

**全挂 vs 全活之间还有「降级可用」**:

- 双 11 时关闭评价、推荐功能,只保留下单
- 抢购高峰只读缓存,不写 DB,DB 异步刷
- 失败时返回兜底数据(默认头像、推荐列表的备用版本)

> "全有或全无" 是错误的可用性思维,**真实系统永远在做"哪些功能可以临时降级"的取舍**。

### 2.2 Soft State(软状态)

允许系统中间状态短暂不一致:

- 订单创建后状态从 "pending" → "paid" → "shipped" 可能跨服务、有延迟
- 不要求每一刻所有服务对状态完全一致

### 2.3 Eventual Consistency(最终一致)

**给数据一段时间,最终会一致**——不是立刻,但会到。

- 主从复制延迟 100ms ~ 几秒,最终会同步
- 缓存过期 / 刷新机制保证缓存最终对齐 DB
- MQ 异步同步数据,可能秒级延迟,最终对齐

> **互联网架构的胜利就是 BASE 的胜利**。强一致(ACID)是金融业的根基,但电商、社交、内容都用 BASE 撑住了 100 倍以上的流量。**学会 BASE 思维 = 学会互联网架构**。

## 三、一致性的「光谱」

不是只有「强一致」和「最终一致」两档,中间还有几档:

```
强 ─────────────────────────────────────────────────────── 弱
线性 → 顺序 → 因果 → 读自己写 → 单调读 → 最终
 |      |      |        |          |        |
 强     ←── 实用范围 ──→            弱
```

### 3.1 线性一致性(Linearizability)

最强,所有读都能看到最新写,**像在单机上一样**。
**实现**:Raft / Paxos / 两阶段提交,代价是高延迟。
**用在**:转账、扣库存、分布式锁。

### 3.2 顺序一致性(Sequential Consistency)

所有节点看到的操作顺序一致,但不一定是实时顺序。
**用在**:消息队列内顺序消费。

### 3.3 因果一致性(Causal Consistency)

有「因果」关系的操作必须按顺序看到(A 写了再 B 读),无关的可以乱序。
**用在**:朋友圈评论(评论必须在发帖之后被看到,但不同人的评论顺序不重要)。

### 3.4 读自己写(Read-Your-Writes)

你刚写的,你自己立刻能读到(别人不一定)。
**用在**:用户改完头像立刻刷新自己页面要看到新头像。

### 3.5 单调读(Monotonic Read)

一旦看到了某版本数据,后面不会读到更老的版本。
**用在**:刷朋友圈不能"刷新一下反而少了几条"。

### 3.6 最终一致(Eventual Consistency)

足够长时间后所有副本会一致,中间任意状态都可能。
**用在**:点赞数、关注数、推荐列表。

> **真实的工程是给每个数据点一个"一致性档位"**——余额线性一致,头像读自己写,点赞数最终一致。**全用最强档 = 系统跑不动;全用最弱档 = 用户骂街**。

## 四、从 CAP 角度看常见组件

| 组件 | 倾向 | 说明 |
| --- | --- | --- |
| ZooKeeper / etcd | CP | 写入要多数派确认,分区时少数派不可用 |
| MySQL 主从 | 默认 AP(异步复制) | 主挂可能丢数据 |
| MySQL 半同步 | 偏 CP | 主等至少一个从确认才返回 |
| Cassandra | AP(默认) | Quorum 可调到 CP |
| Redis 主从 | AP | 主挂自动选举,数据可能丢 |
| Redis Cluster + WAIT | 偏 CP | 用 WAIT 命令等复制 |
| Kafka | CP(配 acks=all + min.insync) | 分区时少数派 broker 拒写 |
| MongoDB(默认) | CP | 多数派写确认 |
| DynamoDB | AP | 最终一致,可选强一致读 |
| Spanner / TiDB | CP(强一致) | 用 TrueTime / Raft + Percolator |

> 选组件之前,**先问自己业务在 CAP 上选哪边**。**用 Cassandra 存余额、用 ZooKeeper 存配置 → 都是大错**。

## 五、常见的 CAP 误区

### 5.1 「我们用 NoSQL 所以是 AP」

不一定。MongoDB / HBase 默认偏 CP,Cassandra 默认 AP 但可调 CP。**SQL / NoSQL 标签和 CAP 没有必然联系**。

### 5.2 「最终一致 = 不一致」

错。最终一致是有**收敛时间窗口**的,通常是 100ms ~ 几秒。99% 的用户感知不到。

### 5.3 「强一致 = 慢」

强一致一定有额外开销(共识、同步复制),但**不一定不能用**。Spanner、TiDB 在跨地域强一致的同时维持 10ms 级延迟,**用对算法和硬件就行**。

### 5.4 「分区很少发生」

错。**生产环境每个月都会有分区事件**——交换机故障、配置错误、防火墙、跨机房抖动。**没设计分区容错的系统迟早出大事**。

### 5.5 「我们选 CP 因为我们是金融」

不绝对。金融的核心账户是 CP,但**周边业务(交易记录展示、营销活动、推荐)都可以 AP**。**别一刀切**。

## 六、PACELC:CAP 的升级版

更细致的模型,Daniel Abadi 提出:

```
PA / CELC

如果发生分区(P),选 A 或 C
否则(Else,正常情况)选低延迟(L)或一致性(C)
```

例:

- DynamoDB:**PA / EL**——分区时选可用,正常时选低延迟
- BigTable:**PC / EC**——任何时候都选一致
- MySQL 异步复制:**PA / EL**——主挂时数据可能丢,正常时低延迟

> PACELC 比 CAP 实用,因为它**揭示了正常情况下也存在的取舍**——一致性不是免费的,即使没分区,强一致也比弱一致慢。

## 七、工程实战:四种典型一致性场景的方案

### 7.1 强一致需求:分布式事务

银行转账,A 扣 100 + B 加 100 必须同时成功或同时失败。

**方案**:

- 单库:数据库本地事务
- 跨库:**Saga / TCC / 2PC**(详见 23 篇订单与支付)
- 跨服务:消息事务表 + 对账兜底

### 7.2 读自己写:Session 粘滞或主库读

用户改了头像立刻刷新页面要看到。

**方案**:

- 用户的写后 N 秒内,读请求强制走主库
- 或者在前端缓存「我刚改的版本」,展示用 max(server, local)

### 7.3 单调读:Session 粘滞 + 版本号

刷新朋友圈不能丢条。

**方案**:

- 同一用户的读请求路由到同一个从库副本
- 或者带版本号 / 游标,服务端保证返回数据 ≥ 客户端已有版本

### 7.4 最终一致:异步复制 + 对账

点赞数、统计、推荐。

**方案**:

- 写主 → 异步同步到 Redis / 副本 / ES
- 定时对账修复偏差

## 八、CAP 与组织决策

CAP 不只是技术决策,**也是产品 + 业务决策**。让产品参与:

| 数据类型 | 问产品的问题 | 通常答案 |
| --- | --- | --- |
| 余额 | 能不能错账? | 不能 → CP |
| 库存 | 能不能超卖? | 不能 → CP(或预占 + 异步对账) |
| 头像 | 改了几秒后才生效行吗? | 行 → AP / 最终 |
| 订单状态 | 用户必须立刻看到? | 自己的订单要(读自己写),别人的订单可以延迟 |
| 评论 | 几秒后才显示行吗? | 行 → AP |

> **跟产品聊清楚每类数据的"可容忍延迟"和"可容忍错误率",CAP 选择就自然出来**。**纯技术拍板的 CAP 决策几乎都是错的**——因为技术不知道业务真正在乎什么。

## 九、踩坑提醒

1. **背"三选二"——错** ,应该是"分区时二选一"
2. **以为 P 可选**——P 不是选项,是物理事实
3. **整系统打 CP / AP 标签**——应该按操作 / 按数据细分
4. **以为 NoSQL = AP**——MongoDB / HBase 默认 CP
5. **强一致全用 2PC**——2PC 性能差,90% 场景 Saga / TCC 更好
6. **最终一致不算"对账机制"**——异步复制可能丢消息,必须有对账兜底
7. **用 Redis 当强一致存储**——Redis 主从异步,主挂丢数据,**别用它存余额**
8. **忽略读自己写需求**——用户最容易感知"改了没生效",这一类必须特殊处理
9. **不告诉产品 CAP 含义**——产品不知道你"为了性能放弃了一致性",出事甩你锅
10. **盲目追求线性一致性**——99% 的业务用因果一致 / 最终一致就够了,**强一致是奢侈品**
