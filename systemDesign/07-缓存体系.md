# 缓存体系

缓存是**互联网架构里性价比最高的优化手段**——加一层 Redis,DB 负载能降 5-10 倍,延迟降 50 倍,成本只有几台机器。但缓存也是**事故高发区**——击穿打爆 DB、穿透 DB 被刷死、雪崩集体过期、脏数据用户骂街。这一篇讲清楚两件事:**一个完整缓存体系长什么样**(本地 / 分布式 / CDN 三层),以及**缓存的"四个经典问题"** 怎么解。

> 一句话先记住:**缓存不是"加 Redis"那么简单,是一个分层体系**——浏览器缓存 / CDN / 反向代理缓存 / 应用本地缓存 / 分布式缓存,每一层离用户越近,QPS 撑得越多但精度越粗。**缓存的核心问题不是"加",是"何时清"**——缓存与 DB 的不一致是一切事故的根源,99% 的缓存 bug 都源于「失效策略写错了」。

---

## 一、缓存层次:从用户到 DB 的五级跳

一个真实的请求,数据可能在五个地方被命中(从近到远):

```
用户浏览器 (Browser Cache)
    ↓
CDN 边缘节点
    ↓
机房入口反向代理 (Nginx Cache / Varnish)
    ↓
应用本地缓存 (Caffeine / Guava)
    ↓
分布式缓存 (Redis / Memcached)
    ↓
数据库 (MySQL / PostgreSQL)
```

每一层的特性:

| 层 | 容量 | 延迟 | QPS | 命中率(典型) |
| --- | --- | --- | --- | --- |
| 浏览器 | MB ~ GB | 0 | 无限 | 因人而异 |
| CDN | TB ~ PB | 10-50ms | 千万 | 80%+(静态) |
| 反向代理缓存 | GB | 1ms | 10 万 | 50%+ |
| 应用本地 | MB ~ GB | 1μs | 100 万+(同机) | 80%+(热点) |
| 分布式缓存 | GB ~ TB | 1ms | 10 万/单机 | 90%+ |
| 数据库 | TB | 5-50ms | 5000-1 万 | 100%(兜底) |

> **缓存的最大威力来自"分层"**——每一层挡掉 80%-95% 流量,五层叠起来真正打到 DB 的不到原始流量的 1/1000。**这就是为什么大厂的 DB QPS 看上去并不离谱**——上游被挡了 99.9%。

## 二、浏览器缓存

最便宜的一层缓存——**根本不发请求**。靠 HTTP Header 控制:

### 2.1 Cache-Control(强制缓存)

```http
Cache-Control: max-age=3600, public
```

浏览器在 1 小时内**直接用本地缓存,不发请求**。`public` 表示中间 CDN 也能缓存。

### 2.2 ETag / Last-Modified(协商缓存)

```http
Server → ETag: "abc123"
Browser → If-None-Match: "abc123"
Server → 304 Not Modified  (只返回头,不返回内容)
```

仍发请求,但内容没变就不重新下载——**省带宽不省 RTT**。

### 2.3 实战策略

```
带哈希指纹的静态资源:Cache-Control: max-age=31536000, immutable
                    一年缓存,文件改了换文件名 (app.a3f5b2c.js)

不带哈希的 HTML:    Cache-Control: no-cache
                    每次都去问服务器,304 返回快

API 响应:          Cache-Control: no-store
                    不缓存
```

> **静态资源 + 文件名指纹 = 永久缓存 + 安全更新**。这是前端工程化的标准玩法,详见 webLearning。

## 三、CDN 缓存

CDN(Content Delivery Network)= **全球分布的缓存边缘节点**。用户访问就近节点,节点没有再回源。

### 3.1 CDN 的两个核心价值

**降延迟**:用户在北京,源站在美国,直接连 200ms;CDN 在北京有节点,10ms 拿到。

**降回源带宽**:1000 万用户下载同一个 JS 文件,**源站只被 CDN 节点拉一次**,剩下全是 CDN 分发——回源带宽降 99.99%。

### 3.2 CDN 缓存什么

| 类型 | 适合 CDN | 原因 |
| --- | --- | --- |
| 静态文件(JS/CSS/图片/视频) | ✅ | 不变、所有人一样 |
| 公开的 API(配置、商品列表) | ✅ | 一段时间内对所有人一样 |
| 用户登录后的个人页 | ❌ | 内容因人而异,缓存意义不大 |
| 实时数据(股票、即时通讯) | ❌ | 变化太快 |
| 写操作 | ❌ | 必须回源 |

### 3.3 CDN 缓存策略

**强制缓存(同浏览器)**:CDN 看 `Cache-Control` 决定缓存多久。

**主动刷新**:发版后调 CDN API 让缓存立即失效:

```bash
# 阿里云 CDN
aliyun cdn RefreshObjectCaches --ObjectPath "https://cdn.x.com/app.js"
```

**预热**:发版后主动调用 CDN 让它去回源拉新版本,**避免用户首次访问触发回源拥堵**。

### 3.4 CDN 的反模式

- **小流量站点上 CDN**:CDN 月费可能比源站还贵
- **动态内容上 CDN 不设 cookie 隔离**:不同用户看到别人的页面(严重事故)
- **缓存了 set-cookie 头**:用户 A 的 token 被缓存给用户 B(超严重)

> **CDN 配置最容易出"安全级"事故**——尤其是缓存了带个人信息的页面。**缓存动态内容前必须确认"对所有用户都一样"**。

## 四、反向代理缓存(Nginx / Varnish)

机房入口 Nginx / Varnish 也能做缓存,**适合「未登录用户的页面」**。

### 4.1 Nginx 缓存配置

```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=mycache:100m max_size=10g;

server {
    location /api/products {
        proxy_cache mycache;
        proxy_cache_valid 200 5m;       # 200 缓存 5 分钟
        proxy_cache_valid 404 1m;       # 404 也缓存(防穿透)
        proxy_cache_key $request_uri;
        proxy_pass http://backend;

        add_header X-Cache-Status $upstream_cache_status;  # HIT / MISS / EXPIRED
    }
}
```

### 4.2 适用场景

- 商品详情页(未登录用户看到的版本)
- 文章 / 帖子内容
- 配置接口(对所有人一样)

### 4.3 与 CDN 的区别

| 维度 | CDN | 反向代理缓存 |
| --- | --- | --- |
| 位置 | 全球边缘 | 单机房 |
| 用户感知延迟 | 极低 | 取决于机房位置 |
| 适合 | 静态资源 + 全球用户 | 动态接口 + 同机房应用 |

> **CDN 挡外面,反向代理缓存挡里面**——两者互补不冲突。

## 五、应用本地缓存(Caffeine / Guava Cache)

跑在应用进程内,**最快的缓存**(纳秒级访问)。

### 5.1 Caffeine 示例(Java)

```java
LoadingCache<Long, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .build(userId -> userDao.findById(userId));

User user = cache.get(123L);
```

### 5.2 适用场景

- **超热点数据**:配置项、热门商品的几条记录、字典表
- **静态数据**:省市区列表、汇率、品类树
- **本机器频繁访问**:同一个用户在 10 秒内连续访问的资料

### 5.3 不适用的场景

- **大量数据**:本地内存有限(JVM 堆 16GB 已经很大了)
- **必须强一致**:N 台机器各自的本地缓存独立,A 机器更新后 B 机器看不到
- **写多的数据**:更新策略复杂

### 5.4 多级缓存架构

```
请求 → 本地 Caffeine → Redis → DB
       ↓ 命中返回      ↓ 命中回填本地    ↓ 命中回填 Redis 和本地
```

**本地命中率 80% × Redis 命中率 95% = DB 实际负载降到 1%**。

> **真正的高性能系统都是多级缓存**——单纯依赖 Redis 都到不了极致。**缺点是要小心 N 台机器的本地缓存不一致**——通过 MQ 广播失效或 TTL 短一些来缓解。

## 六、分布式缓存(Redis 为代表)

应用层之间共享的缓存,**互联网架构的中枢神经**。

### 6.1 三个最常见用法

**用法 1:对象缓存**

```python
def get_user(user_id):
    key = f"user:{user_id}"
    user = redis.get(key)
    if user:
        return json.loads(user)
    user = db.query("SELECT * FROM user WHERE id = %s", user_id)
    redis.setex(key, 3600, json.dumps(user))
    return user
```

**用法 2:列表 / 集合缓存**

```python
# 用户的 timeline,Redis ZSet 按时间排序
redis.zadd(f"timeline:{user_id}", {tweet_id: timestamp})
redis.zrevrange(f"timeline:{user_id}", 0, 19)  # 最新 20 条
```

**用法 3:分布式锁 / 计数器 / 限流**(详见 11 篇 / backendLearning/20)

### 6.2 容量规划

```
单机 Redis 内存:不要超过 32 GB(过大 RDB / AOF 慢)
集群:用 Redis Cluster(自动分片,16384 个槽)
高可用:Sentinel 主备 / Cluster gossip
```

详见 backendLearning/18-20。

## 七、缓存的四个经典问题

这是面试和实战的**必考、必踩**:

### 7.1 缓存穿透(Cache Penetration)

**症状**:查询一个 DB 里也不存在的 key,缓存永远 miss,每次都打 DB。

**典型场景**:攻击者用随机用户 ID 刷接口。

**对策 A:缓存空值**

```python
user = redis.get(key)
if user == "NULL":
    return None
if user is None:
    user = db.query(...)
    if user is None:
        redis.setex(key, 60, "NULL")  # 空值也缓存,但 TTL 短
    else:
        redis.setex(key, 3600, json.dumps(user))
```

**对策 B:布隆过滤器**

启动时把所有合法 key 建布隆过滤器,**不存在的 key 直接拒绝,不打 DB**。
适合**Key 集合相对固定的场景**(比如商品 ID,不会爆炸式增长)。

### 7.2 缓存击穿(Cache Breakdown)

**症状**:某个**热点 key** 突然失效的瞬间,大量并发请求**同时打 DB**。

**典型场景**:首页轮播图缓存过期那一秒,1 万 QPS 同时穿透到 DB。

**对策 A:互斥锁**

```python
def get_hot_data(key):
    data = redis.get(key)
    if data:
        return data
    
    lock_key = f"lock:{key}"
    if redis.set(lock_key, 1, nx=True, ex=10):  # 抢到锁
        try:
            data = db.query(...)
            redis.setex(key, 3600, data)
            return data
        finally:
            redis.delete(lock_key)
    else:
        time.sleep(0.1)  # 没抢到,等一会重试
        return get_hot_data(key)
```

**对策 B:永不过期 + 异步刷新**

热点 key 不设 TTL,后台 worker 定期刷新。**用户永远拿到缓存,不会击穿**。

### 7.3 缓存雪崩(Cache Avalanche)

**症状**:大量 key 在同一时间过期(或 Redis 整体挂掉),海量请求打 DB,DB 跟着挂。

**典型场景**:晚上 0 点统一刷新一堆缓存,所有 key TTL 都设 1 天,第二天 0 点同时失效。

**对策 A:TTL 加随机抖动**

```python
ttl = 3600 + random.randint(0, 600)  # 1 小时 ± 10 分钟
redis.setex(key, ttl, value)
```

**对策 B:多级缓存兜底**

Redis 挂了还有本地缓存(Caffeine),本地命中部分流量,DB 不至于被打穿。

**对策 C:降级 + 限流**

DB 遭雪崩时立即触发限流,只放一部分流量进去,其他返回兜底数据(默认列表)。

### 7.4 缓存与 DB 不一致(Cache-DB Inconsistency)

**最复杂、最常出 bug 的**:更新 DB 后缓存数据脏了。

**几种更新策略**:

#### 策略 1:Cache-Aside(最常用)

```
读:
  1. 先查缓存
  2. 命中返回;不命中查 DB,回填缓存

写:
  1. 更新 DB
  2. 删除缓存(不是更新缓存)
```

**为什么是删除不是更新**?**删除是"懒加载"**——下次有人读时再查 DB 回填,简单可靠;**更新缓存可能存入旧值**(并发场景)。

#### 策略 2:Read-Through / Write-Through

应用只看见缓存,缓存自己负责回填和写穿到 DB。**实现复杂,通常用 Spring Cache / 商用缓存中间件**。

#### 策略 3:Write-Behind(异步写)

写只写缓存,**异步批量刷到 DB**。性能极高,但**宕机会丢数据**。适合可容忍丢失的场景(计数器、统计)。

#### 经典并发问题:删除时机

```
线程 A 写:更新 DB → 删缓存
线程 B 读:查缓存(miss)→ 查 DB(读到 A 之前的旧值)→ 回填缓存

A 写完了,但 B 把 DB 的旧值塞回缓存了。脏了。
```

**对策:延迟双删**

```python
def update_user(user_id, new_value):
    redis.delete(f"user:{user_id}")        # 第一次删
    db.update(user_id, new_value)
    time.sleep(0.5)                        # 等待并发读完成
    redis.delete(f"user:{user_id}")        # 第二次删,清掉脏值
```

> **缓存一致性是分布式系统里最难的问题之一**——没有一种方案是完美的,只有"更适合场景"的方案。**对一致性要求极高的场景(余额、库存)别用缓存,直接 DB**。

## 八、缓存命中率:从 95% 到 99% 的工程

命中率提升的边际收益巨大:

```
95% 命中率,DB 负载  5%  (5000 QPS)
99% 命中率,DB 负载  1%  (1000 QPS)  → DB 负载降 5 倍
99.9% 命中率,DB 负载 0.1%  (100 QPS) → 再降 10 倍
```

**提升命中率的工程动作**:

### 8.1 增加缓存容量

最直接,但贵。先看是否有"冷数据占着位置"。

### 8.2 优化淘汰策略

| Redis 策略 | 含义 |
| --- | --- |
| `noeviction` | 内存满拒写(默认) |
| `allkeys-lru` | 所有 key 按 LRU 淘汰 |
| `allkeys-lfu` | 所有 key 按访问频率淘汰(更智能) |
| `volatile-lru` | 只淘汰带 TTL 的 key |

**热点数据多的场景用 LFU**,**通用场景用 LRU**。

### 8.3 预热

启动时主动加载热数据(不是等用户来触发)。**新机器上线 / 大促开始前必做**。

### 8.4 拆分大 Key

```
错: user:123 缓存 5MB 的对象(读一次卡 100ms)
对: user:123:basic + user:123:friends + user:123:posts(分开缓存)
```

### 8.5 监控未命中

埋点统计哪些 key 高频未命中,**针对性优化**(可能是 TTL 太短 / 触发了缓存击穿)。

## 九、本地缓存 vs Redis 的边界

什么时候用本地、什么时候用 Redis?

| 数据特征 | 本地缓存 | Redis |
| --- | --- | --- |
| 体积大(>100MB) | ❌ | ✅ |
| 访问极频繁(同机微秒级) | ✅ | ❌ |
| 需要跨实例一致 | ❌ | ✅ |
| 写少读多 | ✅ | 都行 |
| 写多 | ❌ | ✅ |
| 启动后基本不变 | ✅ | 都行 |

> **本地缓存是"绝顶热点"的特权**——比如一个被所有请求查的全局配置,放本地比 Redis 还快 1000 倍;**Redis 是"绝大多数缓存"的标准答案**。

## 十、缓存的反模式

1. **把所有数据都缓存**:90% 的数据从来没人查第二遍,缓存全是废物
2. **缓存当 DB 用**:Redis 持久化不可靠,关键数据丢失风险
3. **不设 TTL**:数据永远不更新,长期脏
4. **TTL 设很短(1 秒)**:命中率几乎是 0,缓存形同虚设
5. **缓存大对象**:网络传输慢,命中也耗时
6. **多级缓存不一致**:本地、Redis、DB 三种数据,谁都不知道哪个对
7. **更新策略写错**:更新 DB 后忘了清缓存,几小时后用户报错
8. **没考虑缓存重建成本**:重启后缓存空,DB 被打穿(必须预热)

---

## 十一、踩坑提醒

1. **缓存击穿没保护**——热点 key 过期瞬间打挂 DB
2. **TTL 全设一样**——雪崩高发,加随机抖动
3. **不缓存空值 / 不用布隆过滤器**——穿透打 DB
4. **更新 DB 不删缓存**——脏数据持续到 TTL 结束
5. **更新 DB 后 update 缓存而不是 delete**——并发时存入旧值
6. **缓存大 key**(MB 级)——单 key 卡 Redis 单线程
7. **不分热冷数据**——LRU 把热数据淘汰掉了
8. **新机器上线不预热**——一上线就被打穿
9. **CDN 缓存动态内容**——用户互相看到对方的页面(超严重)
10. **以为本地缓存 = 强一致**——多实例本地缓存天然不一致,要么短 TTL 要么用 MQ 广播失效
