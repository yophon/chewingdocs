# 消息队列与事件驱动

消息队列(Message Queue,MQ)是**互联网系统的"减压阀"和"传送带"**——前者把瞬时高峰削平,后者把同步耦合拆成异步独立。**没有 MQ 的大型系统,不是大型系统**。但 MQ 也是事故重灾区——**消息重复、消息丢失、消息堆积、消费乱序**,每一个都能让你加班到天亮。这一篇讲清楚:**MQ 三大用途、四大主流产品的选型、消息可靠性的真相**(为什么"恰好一次"是个伪命题)、以及顺序和事务消息怎么搞。

> 一句话先记住:**MQ 用来做三件事——异步(让慢操作不阻塞主流程)、削峰(让突发流量平滑落地)、解耦(让上下游独立演进)**。**消息可靠性只有"至少一次"是真实的**——"恰好一次"是营销话术,生产上必须靠**业务幂等**兜底。**MQ 不是银弹**,引入 MQ 等于多一个故障点 + 一致性变弱,不是必要别上。

---

## 一、MQ 三大用途

### 1.1 异步:让慢操作不阻塞主流程

**典型场景**:用户注册后要发欢迎邮件 + 加积分 + 推荐好友。

**同步做法**:

```
用户提交注册 → 写 user 表 → 发邮件(2s)→ 加积分(0.5s)→ 推荐(1s)→ 返回
                                                      总耗时 4s+
```

**异步做法(用 MQ)**:

```
用户提交注册 → 写 user 表 → 发消息到 MQ → 立即返回(100ms)
                                ↓
                           消费者异步处理:发邮件 / 加积分 / 推荐
```

**用户感知 100ms**,后续动作慢点没关系。

> **判断要不要异步的标准:这件事用户必须立刻知道结果吗?** 必须立刻知道(支付结果、登录成功)→ 同步;不需要立刻(发通知、记日志、统计)→ 异步。

### 1.2 削峰:让突发流量平滑落地

**典型场景**:秒杀 1 秒内 10 万订单涌入,DB 写入能力 5000/s,**直接打 DB 必崩**。

```
10 万 QPS 涌入 → 全部进 Kafka(扛百万 QPS 没问题)
                ↓ 消费者按 5000/s 节奏拉
              DB 平滑写入,20 秒消化完
```

**用户体验**:下单立即返回"已受理",几秒后真正写入 DB,再推送结果。

> 削峰的本质是**用消息堆积换 DB 不崩溃**——**只要业务能容忍"准实时"**,削峰就是黄金方案。**强一致场景(银行实时扣款)不能削峰**,因为不能让用户等。

### 1.3 解耦:让上下游独立演进

**没 MQ 的耦合**:

```
订单服务 → 直接调 → 库存服务
                → 直接调 → 积分服务
                → 直接调 → 推荐服务
                → 直接调 → 风控服务
                → 直接调 → 通知服务
                
新增一个"会员升级"逻辑 → 订单服务又得加调用,代码越来越烂
```

**有 MQ 的解耦**:

```
订单服务 → 发"OrderCreated"事件 → MQ
                                  ↓
                        库存 / 积分 / 推荐 / 风控 / 通知 各自订阅各自消费
                        
新增"会员升级":新建一个消费者订阅"OrderCreated",订单服务一行不改
```

> 这就是**事件驱动架构**(Event-Driven Architecture)的本质——**上游不知道有谁消费,下游想加几个加几个**。这是微服务时代的标准玩法。

## 二、四大主流 MQ 选型

| 产品 | 特点 | 典型场景 | 注意 |
| --- | --- | --- | --- |
| **Kafka** | 高吞吐(百万 QPS),日志型 | 大数据流、日志、事件总线 | 偏向"流",不太擅长复杂路由 |
| **RocketMQ** | 阿里出品,事务消息强 | 电商、订单、支付场景 | 中国互联网大厂主流 |
| **RabbitMQ** | AMQP 标准,路由灵活 | 复杂业务路由、微服务通信 | 吞吐相对低(几万 QPS) |
| **Pulsar** | 计算存储分离,云原生 | 新一代,大规模 | 学习曲线陡,生态相对小 |

### 2.1 选型决策树

```
要百万 QPS / 大数据流?       → Kafka
要事务消息 + 中文文档?       → RocketMQ
要复杂路由 + AMQP 兼容?      → RabbitMQ
新业务 + 云原生 + 多租户?    → Pulsar
所有云厂商都有托管版,创业别自建 → 直接买云上的
```

### 2.2 反向警告

**不要用 Redis 当 MQ 用**——Redis 的 List / Pub-Sub 在量大时丢消息,**Stream 类型勉强能用但生产不推荐**。专业 MQ 解决了 Redis 顾不到的:持久化、消费确认、消费者组、回溯消费。

**不要用 DB 当 MQ 用**(轮询表)——QPS 上限低、扩展难、堆积难处理。**早期可以,大了必换**。

## 三、消息可靠性的真相:三个语义

这是 MQ 最被误解的概念,**面试和实战都必须搞清楚**:

### 3.1 至多一次(At Most Once)

```
发消息 → 不管成不成
```

**结果**:可能丢,绝不重复。
**适合**:监控指标、采样日志(丢一点没事)。

### 3.2 至少一次(At Least Once)

```
发消息 → 等 ACK,没收到就重发
消费消息 → 处理完手动 ACK,失败重新投递
```

**结果**:不会丢,**可能重复**。
**适合**:99% 的业务——**这是工程上唯一靠谱的语义**。

### 3.3 恰好一次(Exactly Once)

理论上"不丢不重",**实际上要靠双方协作**:

- Kafka 的 Exactly Once(EOS)只在 Kafka 内部生效(producer + consumer 都在 Kafka 生态)
- 跨系统(Kafka → DB)无法 EOS,**必须靠业务幂等**

**结果**:**纸面上存在,生产上不能依赖**。

> **真相:你只能选"至少一次" + 业务幂等**。任何号称 EOS 的方案,只要跨过它的"边界"(写到外部系统),都退化成至少一次。**这是 MQ 的物理定律**。

## 四、消息丢失的三个环节 + 对策

消息从发出去到被消费,可能丢在三个地方:

### 4.1 生产者 → MQ 的丢失

发了消息但 MQ 没收到(网络抖动)。

**对策**:开启 ACK 机制 + 重试。

```python
# Kafka producer
producer = KafkaProducer(
    acks='all',           # 等所有副本 ACK 才算成功
    retries=3,            # 失败重试
    max_in_flight_requests_per_connection=1,  # 保证顺序的同时不丢
)
```

### 4.2 MQ 内部丢失

收到了但还没刷盘 / 副本同步前节点挂了。

**对策**:

- **持久化**:消息落盘
- **副本数 ≥ 3**:Kafka 设 `replication.factor=3`、`min.insync.replicas=2`
- **acks=all**:等多副本同步才返回成功

### 4.3 MQ → 消费者的丢失

消息到了消费者,消费者还没处理就挂了(自动 ACK 模式)。

**对策**:**手动 ACK**——处理完业务再 ACK,挂了的消息会被重新投递。

```python
# RabbitMQ consumer
def callback(ch, method, properties, body):
    try:
        process(body)
        ch.basic_ack(delivery_tag=method.delivery_tag)  # 手动 ACK
    except Exception:
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=True)
```

> **任何不开手动 ACK 的消费者都是潜在 bug**——挂了直接丢消息。

## 五、消息重复:必须接受的现实

只要保证「至少一次」,**消息重复就一定会发生**:

- 网络抖动 → producer 重发同一消息
- 消费者处理完忘了 ACK / ACK 没传到 → 重投
- 消费者重启,offset 没提交 → 已消费的重新拉

**唯一对策:消费侧业务幂等**。

### 5.1 幂等的几种实现

#### A. 唯一键约束(最简单)

```sql
INSERT INTO order (order_id, ...) VALUES (?, ...) -- order_id UNIQUE
```

重复消息插入失败,捕获异常忽略即可。

#### B. 状态机判断

```python
def process_payment(order_id):
    order = db.query("SELECT status FROM order WHERE id = ?", order_id)
    if order.status == 'paid':
        return  # 已支付,跳过
    db.execute("UPDATE order SET status='paid' WHERE id=? AND status='pending'", order_id)
```

#### C. 去重表

```python
def consume(message):
    msg_id = message.id
    if redis.set(f"consumed:{msg_id}", 1, nx=True, ex=86400):  # 24h 内不重复消费
        do_business(message)
```

#### D. 业务侧版本号(乐观锁)

```sql
UPDATE account SET balance = balance - 100, version = version + 1
WHERE id = ? AND version = ?  -- 旧版本已被消费,这条返回 0 行更新
```

> **幂等是 MQ 时代后端的基本功**——任何一个消费者都必须问「这个消息消费两次会怎样」。**答不上来就不能上线**。

## 六、消息顺序:四种粒度

很多场景需要顺序——比如「用户改名 → 改头像 → 改简介」必须按顺序处理。

### 6.1 全局顺序(几乎做不到 + 不该做)

整个 MQ 一个队列,所有消息严格按发送顺序消费。**意味着只能单线程消费,吞吐被锁死**。

> **生产几乎不用全局顺序**——除非业务逻辑死活离不开。

### 6.2 分区顺序(主流方案)

Kafka / RocketMQ:**同一个 partition 内消息严格有序,跨 partition 不保证**。

```
按 user_id 哈希到 partition:
  user_id=123 的所有消息都进 partition 0 → 内部有序

partition 0 → 单消费者,有序
partition 1 → 单消费者,有序
...
```

**实际上 99% 业务场景"一个用户内有序"就够了**,跨用户根本不需要顺序。

### 6.3 单消费者(慢)

partition 内多消费者会乱序,**单消费者保证完全有序**——但吞吐 = 单消费者吞吐。

### 6.4 业务侧排序

发送时带上序号,消费时按序号攒齐了再处理。**复杂,通常不推荐**——直接靠分区顺序。

> **顺序消息的核心是分片键选对**——按 user_id / order_id 分,**让需要顺序的消息进同一分区**。

## 七、事务消息:数据库写 + MQ 发的一致性

**经典问题**:

```python
def create_order(data):
    db.execute("INSERT INTO order ...")  # 1. 写 DB
    mq.send("OrderCreated", data)         # 2. 发 MQ
```

如果 1 成功 2 失败 → DB 有订单但下游(库存、积分)收不到通知。
如果 1 失败 2 成功 → 下游处理了一个不存在的订单。

### 7.1 解法 A:本地消息表(最常用)

```python
def create_order(data):
    db.transaction:
        db.execute("INSERT INTO order ...")
        db.execute("INSERT INTO outbox (msg, status) VALUES ('OrderCreated', 'pending')")
    # 单独的 worker 轮询 outbox 表,把 pending 消息发给 MQ,成功后改 sent
```

**优势**:简单,DB 事务保证 outbox 和业务表一致。
**劣势**:多一张表 + 一个 worker。

### 7.2 解法 B:MQ 事务消息(RocketMQ 原生支持)

```
1. 发 half message 到 MQ(对消费者不可见)
2. 执行 DB 操作
3. 提交 commit / rollback 给 MQ
4. MQ 提交后消费者可见

如果第 3 步丢了,MQ 反向回查业务系统:这个消息该 commit 还是 rollback?
```

**优势**:无需 outbox 表。
**劣势**:RocketMQ 才有原生支持,Kafka / RabbitMQ 都没有。

### 7.3 解法 C:CDC(Change Data Capture)

订阅 DB 的 binlog,**有什么写入就发什么消息到 MQ**。

```
订单写 DB → binlog 变化 → Canal / Debezium 监听 → 发 MQ
```

**优势**:业务零侵入,**最干净的事件驱动**。
**劣势**:需要部署 CDC 工具,有一定延迟。详见 backendLearning/46。

## 八、消息堆积:MQ 最常见的故障

**症状**:消费者跟不上生产速度,消息积压到 MQ 里几百万、几千万条。

### 8.1 堆积的危害

- 消息从 100ms 延迟变成 1 小时甚至几天
- MQ 磁盘爆了 → MQ 自己挂掉 → 全平台事故
- 业务方收到老旧消息,造成混乱(过期订单还在处理)

### 8.2 应急处理(已堆积时)

```
1. 立即扩消费者(临时加 N 个实例)
2. 临时增加分区数(只对新消息有效)
3. 实在追不上 → 写脚本批量丢老消息(谨慎!需业务确认)
4. 或者另起一个消费组,把堆积的消息倒到另一个 topic 慢慢消化
```

### 8.3 预防

- **监控消费延迟**(Lag),超过阈值报警
- **消费者扩容自动化**(K8s HPA 按 Lag 扩容)
- **大事务消息单独 topic**:别让一个慢消息拖死整个 topic
- **死信队列(DLQ)**:消费失败 N 次的消息丢到 DLQ,**不阻塞正常消息**

## 九、消费模型:Pull vs Push

### 9.1 Pull(主流,Kafka/RocketMQ)

消费者主动拉消息。

**优势**:
- 消费速度由消费者控制,不会被压垮
- 批量拉效率高
- 自然支持回溯消费(指定 offset)

**劣势**:
- 实时性稍差(得等下次 pull)

### 9.2 Push(RabbitMQ 默认)

MQ 主动推消息给消费者。

**优势**:
- 实时性高

**劣势**:
- 推快了消费者扛不住(需要 prefetch 限流)
- 不容易回溯

> **Kafka / RocketMQ 用 Pull,适合大数据 / 高吞吐;RabbitMQ 用 Push,适合实时业务**。

## 十、消费者组(Consumer Group)

**核心概念**——决定消息的分发模式:

### 10.1 同一个 Group:消息分摊(队列模型)

```
Topic "orders" 有 3 个 partition
Group "service-a" 有 3 个 consumer

partition 0 → consumer 1
partition 1 → consumer 2
partition 2 → consumer 3

每条消息只被 group 内一个 consumer 消费
```

### 10.2 不同 Group:消息广播(发布订阅)

```
Topic "orders"
Group "service-a"(库存) → 收到所有消息
Group "service-b"(积分) → 收到所有消息
Group "service-c"(推荐) → 收到所有消息

每个 group 独立消费一份
```

### 10.3 Rebalance:消费者增减时的重分配

加 / 减 consumer → MQ 重新分配 partition → **过程中短暂不可消费**(几秒)。

**坑**:Rebalance 期间业务可能拿到重复消息(老 consumer 没 ACK 的消息被分给新 consumer)。

> **新手最容易搞错:同一个业务的消费者必须用同一个 Group**——不然就变成广播,每个实例都消费一遍,数据写多次。

## 十一、什么场景不该用 MQ

MQ 不是越多越好——**用错了反而拖累系统**。

### 11.1 必须强一致 + 立即响应的

支付扣款、银行转账——用户等结果,异步风险大。**直接同步调用**。

### 11.2 链路简单 + 性能极敏感

只有两个服务通信,加 MQ 多一跳延迟、多一个故障点。**直接 RPC**。

### 11.3 数据量极大 + 不需要持久化

监控数据、临时统计——直接 UDP 推聚合服务,**MQ 持久化是浪费**。

### 11.4 流式计算

应该用 Flink / Spark Streaming,**MQ 只是其中的存储层,不是处理层**。

---

## 十二、踩坑提醒

1. **追求 Exactly Once**——纸面概念,生产必须靠业务幂等
2. **不开手动 ACK**——消费者挂了直接丢消息
3. **producer 不开 acks=all**——MQ 丢消息你不知道
4. **消费者不写幂等**——重复消费时业务错乱
5. **不监控消费延迟**——堆积几小时才发现
6. **顺序消息用全局队列**——单消费者吞吐瓶颈
7. **错把 Redis / DB 当 MQ**——量大时崩
8. **同一业务用不同 Group**——数据被消费多次
9. **大事务消息和小消息混在同 topic**——大消息拖死小消息
10. **不设死信队列**——一条坏消息无限重试,卡死整个 topic
11. **消费失败无限重试**——程序逻辑错的话死循环 CPU 100%
12. **以为 MQ 是银弹**——MQ 解决耦合 + 削峰,但引入异步 + 一致性弱化的成本
