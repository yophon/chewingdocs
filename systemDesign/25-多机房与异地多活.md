# 多机房与异地多活

单机房终究有上限——电力、散热、带宽、政策、灾难。**真正的大型互联网架构都是多机房**:阿里 / 字节 / 腾讯 / 美团 都是 5+ 机房同时跑业务。但**多机房不是"复制一份就完了"** —— 同步延迟、跨机房一致性、流量调度、容灾切换 都是地狱难题。这一篇讲清楚**多机房的四种模式**(冷备 / 双活 / 多活 / Cell 单元化)、**单元化**(Cell)的本质、以及**为什么"异地多活"是当今互联网的最高架构挑战**。

> 一句话先记住:**多机房的本质是"跨光纤距离 vs 数据一致性"的取舍**——同城双活近(2-5ms 延迟),数据强一致;异地多活远(30ms+),只能最终一致。**Cell 单元化是解决跨机房写一致性的工程艺术**——把用户分桶,每个桶(Cell)在一个机房自包含运行,跨 Cell 只走异步同步。**异地多活是"组织能力"而不是"技术能力"** —— 没有整套发布、监控、演练体系支撑,多活就是炸弹。

---

## 一、为什么需要多机房

### 1.1 单机房的物理上限

- **电力**:单机房 10-30 MW 上限
- **散热**:单数据中心 5000-10000 服务器后散热成本爆炸
- **带宽**:单机房出口 100 Gbps-1 Tbps 上限
- **空间**:寸土寸金,机柜租金贵

### 1.2 灾难恢复

```
事故案例:
  2015 年 阿里杭州机房光缆被挖,支付宝瘫痪 4 小时
  2017 年 GitLab 误删数据库,丢 6 小时数据
  2021 年 Fastly CDN 故障,全球互联网半小时大面积瘫痪
  2024 年 某云国内机房地震,双活失效
```

**单机房 = 单点故障**。**金融、医疗、政务**强制要求多机房。

### 1.3 政策合规

- 中国数据出境受限 → 海外业务必须海外有机房
- 欧盟 GDPR → 欧洲用户数据必须存欧洲
- 各种行业要求"两地三中心"

### 1.4 用户延迟

- 海外用户访问国内 200ms+
- 必须就近接入

## 二、多机房的四种模式

按复杂度从低到高:

```
冷备     → 主用 A,B 完全闲置,出事手动切
温备     → 主用 A,B 跑着但不接流量
同城双活 → A B 都接流量,数据强一致
异地多活 → A B C... 都接流量,数据最终一致
单元化   → 异地多活的极致,用户分桶到 Cell
```

---

## 三、模式 1:冷备(传统金融用)

```
A 机房 (Active):           B 机房 (Cold Standby):
  完整业务运行                 服务器在线但不跑业务
  数据写入                     定期复制 A 的数据
  流量进来                     流量进不来

A 挂了:
  人工评估 → 切 DNS → B 启动业务 → 数据可能丢失最后一段
```

**优点**:简单,B 的硬件/数据正确性容易保证。
**缺点**:
- B 平时不发挥作用,**100% 浪费成本**
- 切换需要几小时(测试 + 启动)
- 数据丢失(异步复制)

**适合**:**银行核心**——可用性要求 5 分钟,但不能用"双活"(强一致难保证),用冷备 + 严密演练。

---

## 四、模式 2:同城双活(主流互联网)

A B 在同一城市(20 km 内),**两个机房都接流量**:

```
       用户
        ↓
        DNS / Anycast
        ↓
   ┌────┴────┐
   ↓         ↓
A 机房    B 机房
(50%)    (50%)
   ↑         ↑
   └────┬────┘
   共享 / 同步 DB
   (主从 / 半同步)
```

### 4.1 数据怎么处理?

**方案 A:共享 DB**(同城两机房之间 < 5ms)

A 和 B 应用层都连同一个 DB(在 A 或 B 机房)。
- **优点**:强一致
- **缺点**:DB 单机房,挂了双活变伪双活

**方案 B:DB 主从同步**

- A 用本地 DB(主),B 用本地 DB(从)
- 写都路由到 A 的主库,读各自就近读
- A 挂了 → B 升主

**方案 C:DB 同城双主**(MySQL Group Replication / 阿里 OceanBase)

- A B 都能写
- 通过 Paxos / Raft 保证强一致
- 写延迟略增(等多数派)

> **同城双活的甜蜜区**:延迟低、可强一致、容灾足够(罕见两机房同时挂)。**99% 的中型互联网都是同城双活**。

### 4.2 流量调度

- 平时 50/50
- A 性能下降 → 调成 30/70
- A 完全挂 → 100% 切 B(LB / DNS 自动)

---

## 五、模式 3:异地多活(顶级架构)

A B 在不同城市(几百~几千公里)——**延迟 30-100ms**:

```
       全球 / 全国用户
              ↓
       DNS / Anycast(就近)
              ↓
   ┌──────────┼──────────┐
   ↓          ↓          ↓
北京机房   上海机房    深圳机房
   ↕          ↕          ↕
   异步数据同步 (延迟 100ms - 几秒)
```

### 5.1 异地的核心难点:写延迟

```
北京用户写:
  本地写北京机房     ✓ 1ms
  同步到上海机房     ✗ 30ms
  同步到深圳机房     ✗ 50ms

如果"等所有机房确认"才返回:
  写延迟 = 50ms+(用户卡)

如果只等本地确认:
  写延迟 1ms
  但其他机房有 100ms-几秒"看不到这个写"
```

**取舍**:**牺牲一致性换性能**。

### 5.2 异地数据冲突

```
北京用户改头像 → 头像 = "A.jpg"
同时上海用户(其实是同一个账号在不同设备)→ 头像 = "B.jpg"

异步同步后:
  北京以为头像是 A
  上海以为头像是 B
  
谁赢?
```

**冲突解决**:

- **Last Write Wins(按时间戳)**:简单但丢数据
- **业务侧合并**:复杂(不同字段不同策略)
- **避免冲突**:**同一用户只在一个机房写**(Cell 单元化)

---

## 六、模式 4:Cell 单元化(异地多活的终极形态)

**思想**:**用户分桶,每个桶(Cell)在一个机房自包含**:

```
┌────── Cell-A (北京) ─────────┐
│ 完整业务: 应用 + DB + 缓存 + MQ │
│ 服务用户: user_id 0 ~ 1 亿      │
└────────────────────────────┘

┌────── Cell-B (上海) ─────────┐
│ 完整业务                       │
│ 服务用户: user_id 1 亿 ~ 2 亿  │
└────────────────────────────┘

┌────── Cell-C (深圳) ─────────┐
│ 完整业务                       │
│ 服务用户: user_id 2 亿 ~ 3 亿  │
└────────────────────────────┘

跨 Cell:仅异步数据同步
```

### 6.1 优势

- **每个 Cell 内部都强一致**(本地操作)
- **故障隔离**:Cell-A 挂只影响 1/N 用户
- **就近接入**:用户路由到自己的 Cell
- **水平扩展**:加 Cell 即可

### 6.2 难点

- **路由层**:用户访问任何域名时,**接入层必须知道他属于哪个 Cell**
- **跨 Cell 数据**:用户 A(Cell-A)给用户 B(Cell-B)发消息怎么办?
  → 写入 A 的 DB → 异步同步到 B → B 才看到

### 6.3 阿里的 LDC(Logical Data Center)

阿里 2014 年开始的单元化,**支撑双 11 流量**:

- 用户按 user_id % 100 分桶
- 每桶分配到某个 Cell
- 全国 5+ 个 Cell
- 每年双 11 之前做"Cell 切换"演练

## 七、Cell 化的工程细节

### 7.1 用户路由

用户登录后,**接入层立刻知道他归哪个 Cell**:

```
key:    user_route:{user_id}
value:  cell_id (A / B / C ...)

路由层:
  if user.cell != current_cell:
    302 redirect to user.cell.domain
```

或在 DNS 层做"按用户 IP 返回不同 Cell 的 IP"。

### 7.2 跨 Cell 调用

跨 Cell 的请求**不允许同步阻塞**:

```
用户在 Cell-A 给 Cell-B 用户发消息:

错(同步):
  Cell-A → 实时调用 Cell-B 的 IM 服务
  跨机房 RPC → 30ms+ → 用户体验差

对(异步):
  Cell-A → 写入消息队列(本地)→ 立即返回
  消息队列 → 异步同步到 Cell-B → Cell-B 投递给用户
```

### 7.3 跨 Cell 数据一致性

**全局数据**(配置、商品库、字典)→ 所有 Cell 都有副本,异步复制:

```
中心 → 推到 Cell-A、Cell-B、Cell-C
延迟几秒,但全局数据本来就不频繁变
```

**用户数据**(订单、消息)→ 主 Cell 写,其他 Cell 副本同步:

```
Cell-A 用户的订单 → 主存储在 Cell-A
                 → 异步同步到 Cell-B、Cell-C(用于跨 Cell 查询和容灾)
```

## 八、流量调度

### 8.1 路由策略

```
按用户:user_id 哈希到 Cell
按地理:北京用户进 Cell-A(就近)
按业务:核心业务双 Cell,边缘业务单 Cell
按 SKU:某些商品集中在某 Cell
```

### 8.2 切流(关键能力)

Cell-A 挂了 / 需要维护:**把流量切到 Cell-B**:

```
切流前:
  Cell-A 50%, Cell-B 50%
  
切流过程(灰度):
  Cell-A 30%, Cell-B 70%
  Cell-A 10%, Cell-B 90%
  Cell-A 0%,  Cell-B 100%

每一步等几分钟,确认指标正常再继续
```

**关键**:切流是**毫秒级生效**——通过路由表 + 配置中心推送,不用重启。

### 8.3 切流演练

**没演练过的切流肯定出事**——阿里、字节都有定期演练:

- 每月演练同城切流
- 每季度演练跨地域切流
- 每半年演练"整个 Cell 模拟挂掉"

## 九、监控:多机房特有指标

### 9.1 跨机房延迟

```
Cell-A → Cell-B 同步延迟
  正常 < 1 秒
  > 5 秒 报警
  > 30 秒 P0
```

### 9.2 数据一致性指标

```
Cell-A 和 Cell-B 的同一表数据校验
  每小时随机抽样比对
  发现差异立即查
```

### 9.3 各 Cell 容量

```
Cell-A CPU / Memory / DB QPS
  超过 70% 预警
  超过 85% 准备切流到其他 Cell
```

## 十、典型公司的多活架构

### 10.1 阿里:LDC + RZone / GZone / CZone

- **RZone**(Region Zone):用户数据按 user_id 分桶
- **GZone**(Global Zone):全局共享数据(商品库、配置),只读副本
- **CZone**(City Zone):城市级数据,本地化

### 10.2 字节:多机房 Cell

- 国内:北京 + 上海 + 广州 + ...
- 海外:新加坡 + 美西 + ...
- 业务按用户分 Cell

### 10.3 微信:三园区五机房

- 同城三个园区(深圳)
- 异地两个机房(上海 / 北京)
- IM 数据同城三园区,跨地域容灾

### 10.4 Google:Spanner 全球强一致

- TrueTime API(基于原子钟 + GPS)
- 跨地域强一致(代价是 ~10ms 写延迟)
- **唯一一家做到"全球强一致"的公司**

---

## 十一、什么时候不该上多活

### 11.1 业务规模不到

DAU < 1000 万的业务,**同城双活足够**。异地多活引入的复杂度远大于带来的收益。

### 11.2 团队不够强

异地多活要求:

- 强大的发布系统(同时部署多 Cell)
- 完善的监控体系
- 配套的演练机制
- 路由 / 配置中心的稳定基础设施

**没这些,多活反而是事故源**。

### 11.3 业务模型不适合

强一致业务(银行核心、清算系统)→ 异地多活意义不大,**强一致跨地域延迟太大**,只能用 Cell 化或冷备。

```
        ↓              ↓              ↓
   ┌────────┐    ┌────────┐     ┌────────┐
   │ Cell A │    │ Cell B │     │ Cell C │
   │ 北京    │    │ 上海    │     │ 深圳    │

```

## 十三、踩坑提醒

1. **没监控就上多活**——出事时不知道哪个 Cell 错
2. **强一致业务上异地多活**——必然性能崩
3. **跨 Cell 同步调用**——延迟把用户体验毁掉
4. **数据冲突没处理方案**——双写时数据脏
5. **不演练切流**——真出事时切不过去
6. **路由层单点**——路由层挂了所有 Cell 都没法用
7. **配置不统一**——A B 配置不一致,行为不同
8. **异步同步无监控**——同步停了 1 小时没人发现
9. **以为多活 = 高可用**——多活可能更脆弱(同步链路)
10. **没"备 Cell"**——所有 Cell 都跑业务,挂一个就丢 1/N 用户
11. **Cell 划分粒度太粗 / 太细**——粗了故障域大,细了管理复杂
12. **以为团队能力够**——多活需要顶级 SRE 团队 + 全套基础设施
