# 数据迁移

数据迁移是**工程师"最不愿意做但最值得练"的工作**——业务团队躲都来不及,因为它**全是脏活、风险大、还不能停服**。但能搞定数据迁移的工程师都是**架构师级别**——它综合了分布式事务、双写一致性、灰度发布、回滚机制、对账系统的所有能力。这一篇讲清楚:**几种典型迁移场景**(版本升级 / 库迁移 / 分片重构 / 跨机房 / 跨云),以及**"双写 + 数据同步 + 灰度切流"** 这个万能模板。

> 一句话先记住:**数据迁移的核心是"零停机 + 可回滚"** —— 业务必须无感切过去,出问题必须能秒级切回来。**万能模板:写老 → 双写新老 → 灰度读新 → 全量读新 → 停写老 → 删老**——五步走,每步可观测可回滚。**最大的坑是"对账"** —— 不对账你永远不知道新老数据是不是真的一致,而**数据迁移最怕的不是慢,是少了或重了**。

---

## 一、什么是数据迁移

> 把数据从 A 系统挪到 B 系统,业务不能中断。

### 1.1 典型场景

| 场景 | 描述 |
| --- | --- |
| 版本升级 | MySQL 5.6 → 8.0 |
| 库迁移 | MySQL → TiDB,Oracle → PostgreSQL |
| 分片重构 | 16 库 → 32 库 |
| 表重构 | 旧表结构 → 新表结构 |
| 跨机房 | 北京机房 → 上海机房 |
| 跨云 | 阿里云 → 腾讯云 |
| 服务拆分 | 单体表 → 微服务各自的库 |

### 1.2 为什么难

- **数据量大**:TB / PB 级别,搬一次要几天
- **业务持续写入**:迁移期间数据还在变
- **零停机要求**:不能让业务下线
- **回滚不易**:迁完发现新系统有 bug 怎么回?
- **数据一致性**:新老数据偏差一点点都是事故

---

## 二、迁移的万能五步模板

适用于**几乎所有迁移场景**:

```
Step 1:  只写老
         状态:新系统不存在或刚部署

Step 2:  双写新老
         状态:新系统已部署,但是空的或不完整
         所有新写入同时写到新老两边

Step 3:  历史数据回填
         批量任务把老数据搬到新系统
         同时双写继续(保证增量同步)

Step 4:  灰度读新
         少量请求(1% → 10% → 50% → 100%)读新系统
         双写仍在继续(保险)

Step 5:  停写老,只用新
         确认新系统稳定 → 关闭老系统的写入
         保留老系统数据一段时间(保险)
         最终下线老系统
```

**每一步都可独立验证,每一步都可单独回滚**。

## 三、关键技术点:双写

### 3.1 同步双写

```python
def update(data):
    db_old.update(data)
    db_new.update(data)
```

**优点**:简单,数据立刻一致。
**缺点**:写延迟翻倍,新系统挂了影响主业务。

### 3.2 异步双写

```python
def update(data):
    db_old.update(data)
    mq.send("update_event", data)
    # 消费者写到 db_new
```

**优点**:不影响主业务性能。
**缺点**:有几秒延迟,新系统数据稍微落后。

### 3.3 双写的关键:幂等

数据迁移期间,**同一条数据可能被写多次**(双写 + 回填 + 重试):

```python
# 必须用主键或唯一键 upsert,不能简单 INSERT
db_new.execute("""
    INSERT INTO new_table (id, name) VALUES (?, ?)
    ON DUPLICATE KEY UPDATE name = VALUES(name)
""", (id, name))
```

## 四、历史数据回填

迁移开始前的存量数据,**怎么搬到新系统**?

### 4.1 全量同步工具

| 工具 | 适合 |
| --- | --- |
| `mysqldump` + import | 小数据(< 100 GB) |
| Percona xtrabackup | 物理备份,快 |
| AWS DMS / 阿里 DTS | 云上托管,跨 region 友好 |
| Canal / Debezium | 基于 binlog,支持持续同步 |
| 自研脚本 | 复杂场景(转换 + 校验) |

### 4.2 自研脚本的标准模式

```python
def migrate_table(table, batch_size=1000):
    last_id = 0
    while True:
        rows = db_old.query(f"SELECT * FROM {table} WHERE id > ? ORDER BY id LIMIT ?", last_id, batch_size)
        if not rows:
            break
        db_new.bulk_insert(table, rows)
        last_id = rows[-1].id
        record_progress(table, last_id)  # 持久化进度,可断点续传
```

**关键**:

- **批量**:1000 行一批,平衡内存和事务大小
- **断点续传**:记录进度到独立表,挂了重启从上次接着来
- **限速**:控制 QPS,不打挂老 DB
- **跳过冲突**:用 INSERT IGNORE / ON DUPLICATE KEY UPDATE

---

## 五、增量同步:跟上业务变化

回填期间业务还在写入,**新写入的数据怎么同步到新系统**?

### 5.1 双写解决

如果应用层做了双写,**增量自然同步**——不用额外操作。

### 5.2 binlog 同步(应用不动)

不想改应用代码,**用 CDC**:

```
MySQL binlog → Canal / Debezium → Kafka → 同步服务 → 写到新 DB
```

**优势**:应用零改动,业务无感。
**详见**:backendLearning/46。

### 5.3 双轨并行

```
应用代码 → 老 DB
         → 新 DB(应用层双写)
         
binlog → 新 DB (CDC 同步,作为兜底)

两条路径都到新 DB,任一路径漏数据,另一路径补
```

**最稳的方案**——应用层 + CDC 双保险。

## 六、对账:数据迁移的命门

**最大的坑是对账**——不对账你永远不知道新老数据是不是真的一致。

### 6.1 全量对账

```python
def full_audit(table):
    # 老 DB 总数
    old_count = db_old.scalar(f"SELECT COUNT(*) FROM {table}")
    # 新 DB 总数
    new_count = db_new.scalar(f"SELECT COUNT(*) FROM {table}")
    if old_count != new_count:
        alert(f"Count mismatch: old={old_count} new={new_count}")

    # 抽样校验
    sample_ids = random.sample(range(1, old_count), 1000)
    for id in sample_ids:
        old_row = db_old.query(f"SELECT * FROM {table} WHERE id = ?", id)
        new_row = db_new.query(f"SELECT * FROM {table} WHERE id = ?", id)
        if old_row != new_row:
            alert(f"Row mismatch: id={id}")
```

### 6.2 增量对账

```python
# 每分钟对比最近 5 分钟变化的数据
recent_old = db_old.query("SELECT * FROM table WHERE updated_at > ?", now - 5min)
for row in recent_old:
    new_row = db_new.query("SELECT * FROM table WHERE id = ?", row.id)
    assert row == new_row
```

### 6.3 自动修复

发现差异**自动修复**:

```python
def auto_fix(diff):
    if diff.type == "missing_in_new":
        db_new.insert(diff.row)
    elif diff.type == "outdated_in_new":
        db_new.update(diff.row)
    elif diff.type == "extra_in_new":
        log("manual review needed", diff.row)  # 多了不能瞎删
```

> **没对账的迁移就是赌博**——你不知道丢了多少数据,等业务报错才发现。

## 七、灰度切流

切读流量必须**渐进式**——不能一次性切 100%。

### 7.1 灰度策略

```
1% 用户读新系统 → 监控 1 天
10% → 监控 1 天
50% → 监控 2 天
100% → 监控 1 周
```

**任一阶段发现异常 → 立即切回**。

### 7.2 灰度的实现

#### A. 按 user_id 哈希

```python
if hash(user_id) % 100 < threshold:
    return read_from_new()
else:
    return read_from_old()
```

`threshold` 通过配置中心动态调整。

#### B. 按白名单

```python
if user_id in beta_users:
    return read_from_new()
else:
    return read_from_old()
```

**先内部员工 / VIP 用户灰度**,出问题影响小。

#### C. 按地理区域

```python
if user.region == "shanghai":
    return read_from_new()
```

某城市先用,验证后扩大。

### 7.3 灰度的"双读对比"

更稳:**同时读新老,对比结果**:

```python
old_result = read_from_old()
new_result = read_from_new()
if old_result != new_result:
    log_diff(old_result, new_result)
return old_result  # 仍然返回老的,新的只是验证用
```

**优势**:在不影响业务的情况下,**真实流量验证新系统**。

## 八、回滚机制

迁移**必须可回滚**——任何一步出问题都能秒级切回。

### 8.1 回滚的两个方向

```
读流量回滚:
  灰度比例从 100% → 0%(配置中心改一行)

写流量回滚:
  双写阶段:还有老 DB 数据,直接停写新 DB
  停写老 DB 阶段:危险!此时只有新 DB 有最新数据
```

### 8.2 关键:延长"双写期"

不要急着关老 DB——**至少保持双写 1-2 周**,确认完全没问题再停老。

### 8.3 老 DB 数据保留

即使切到新 DB 后,**老 DB 数据保留 30-90 天**——出问题还能恢复。

## 九、典型场景:分库重构(16 → 32)

最常见的迁移场景,详见 08 篇。

### 9.1 步骤

```
1. 部署 32 个新库(空的)
2. 双写:同时写老 16 库(按 mod 16)和新 32 库(按 mod 32)
3. 历史数据回填:扫老 16 库,按新规则写入新 32 库
4. 增量同步:CDC 兜底,确保不漏
5. 全量 + 抽样对账
6. 灰度读新 32 库:1% → 100%
7. 双写继续 1-2 周
8. 停写老 16 库,新 32 库独占
9. 监控 1 周
10. 老 16 库下线
```

### 9.2 加速技巧:成倍扩容

```
原 16 库:db0 ~ db15
扩到 32 库:db0a / db0b / db1a / db1b / ...

每条数据从 db_n 迁到 db_n_a 或 db_n_b(同节点的镜像库)
迁移路径明确,带宽友好
```

详见 08 篇。

## 十、典型场景:跨云迁移

阿里云 → 腾讯云,**比同云迁移复杂 10 倍**:

### 10.1 难点

- 跨公网,带宽贵且慢
- 不同云的服务 API 不一样
- 安全合规问题
- 灰度切流要跨云路由

### 10.2 标准方案

```
1. 在新云搭一套相同架构
2. 用专线 / VPN 打通(降延迟降成本)
3. 全量同步(业务低峰期跑几天)
4. 应用层双写(同时写两边云)
5. 灰度切流(域名 DNS 慢慢切)
6. 老云保留 30 天
7. 老云下线
```

**预算时间**:1-2 个月对中型业务,半年对大型业务。

## 十一、典型场景:服务拆分

单体 → 微服务,**数据库也要拆**:

```
原:单体 monolith_db(user / order / product 全在一起)

目标:
  user_db(独立)
  order_db(独立)
  product_db(独立)
```

### 11.1 步骤

```
1. 新建 user_db, 双写 monolith_db.user 和 user_db
2. user 服务独立部署,但仍读 monolith_db(过渡)
3. 历史 user 数据回填到 user_db
4. 灰度 user 服务读 user_db
5. user 服务完全切到 user_db
6. monolith_db.user 表停止写入
7. 删 monolith_db.user 表(保留备份)
```

每个表 / 模块都重复这套流程。

## 十三、踩坑提醒

1. **没双写直接切**——切的瞬间老库还在写新库不知道,数据不一致
2. **没对账**——以为数据搬完了,其实丢了一半
3. **不灰度**——一次性 100% 切,出问题来不及回滚
4. **过早停老库**——停了发现新库有 bug,无法恢复
5. **批量 SQL 锁表**——`UPDATE ... WHERE` 全表扫,锁老库几小时
6. **不限速**——同步任务把老库 CPU 打满,业务也挂
7. **没考虑增量**——回填期间业务在写,数据漏了
8. **依赖人工对账**——一年后才发现差异,数据修复天文工作量
9. **跨版本盲目升级**——MySQL 5.6 → 8.0 中间还要过 5.7
10. **不演练回滚**——真出事时回不去
11. **不通知业务**——业务方不知道有迁移,出问题甩你锅
12. **不留余量**——预算 1 个月,实际花了 3 个月,业务规划全乱
