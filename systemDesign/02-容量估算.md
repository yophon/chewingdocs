# 容量估算

容量估算是系统设计里**最被低估的硬功夫**。新手以为「估算」就是问前辈一句「咱们有多少 QPS」,真正的工程师是**张嘴就能算出 1 亿 DAU 的微博需要多少 MySQL、多少 Redis、多少带宽**。这种心算不是为了显摆,是为了**在没有数据的情况下做架构决策**——面试时面试官不会给你 QPS,真实工作里产品也不会,但你必须立刻判断「这个量级要不要分库」「要不要上 CDN」。

> 一句话先记住:**估算是 30 秒内算出"数量级"**——不是精确到 1234 QPS,是判断"这是 1 千、1 万还是 100 万"。**数量级错一个,整个架构就错**——单机能扛 1 万 QPS,你估成 100 万就上了 K8s + 100 节点 Redis 集群,**钱白烧;反过来你估成 1 千,真上线就直接打挂**。这一篇教的是"建立数字直觉",看完之后听到 DAU 你脑子里应该自动蹦出 QPS。

---

## 一、估算需要的「数字背诵表」

任何估算都从这几个常识开始,**不背下来就不可能算得快**:

### 1.1 时间换算

```
1 天 = 86,400 秒  ≈ 100,000 秒(估算时常用 10^5)
1 小时 = 3,600 秒  ≈ 4,000 秒(估算时常用 4 × 10^3)
1 个月 ≈ 2.5 × 10^6 秒
1 年 ≈ 3 × 10^7 秒
```

> **最重要的就是"一天 ≈ 10^5 秒"这一条**。99% 的 DAU → QPS 估算都用得上。

### 1.2 数据量换算

```
1 KB    = 10^3  字节
1 MB    = 10^6  字节
1 GB    = 10^9  字节
1 TB    = 10^12 字节
1 PB    = 10^15 字节
```

### 1.3 网络与硬件性能(单机典型值)

| 操作 | 延迟 | 直觉 |
| --- | --- | --- |
| L1 cache | 0.5 ns | 同机指令级 |
| 主存随机访问 | 100 ns | 同机内存级 |
| SSD 随机读 | 100 μs | 千分之一毫秒 |
| 千兆网络(同机房) | 0.5 ms | 半毫秒 |
| 跨机房(同城) | 1-3 ms | 几毫秒 |
| 跨地域(中美) | 100-200 ms | 100 毫秒级 |
| HDD 随机读 | 10 ms | 慢 100 倍 |

| 组件 | 单机吞吐量(粗略) |
| --- | --- |
| Nginx 反向代理 | 5-10 万 QPS |
| Spring Boot 单实例 | 1000-5000 QPS |
| Redis 单实例 | 10 万 QPS |
| MySQL 单实例(读) | 5000-10000 QPS |
| MySQL 单实例(写) | 1000-3000 QPS |
| Kafka 单分区 | 100MB/s 顺序写 |

> 这两张表是**估算的基石**。看到「单机 MySQL 撑写 10 万 QPS」就立刻知道「不可能,得分库」——这种直觉就是凭这张表。

## 二、DAU → QPS 的标准心算

最高频的估算场景:用户给了 DAU,你要快速算出**平均 QPS** 和**峰值 QPS**。

### 2.1 万能公式

```
平均 QPS = DAU × 人均请求数 / 86400
       ≈ DAU × 人均请求数 / 10^5

峰值 QPS = 平均 QPS × 峰均比 (一般 3 ~ 10 倍,常用 5 倍)
```

### 2.2 几个常见数量级的肌肉记忆

| DAU | 人均 50 次 | 平均 QPS | 峰值 QPS(×5) | 数量级判断 |
| --- | --- | --- | --- | --- |
| 1 万 | 50 万 | ~6 | 30 | 单机随便 |
| 10 万 | 500 万 | ~60 | 300 | 单机轻松 |
| 100 万 | 5000 万 | ~600 | 3000 | 单机紧张,要做读写分离 |
| 1000 万 | 5 亿 | ~6000 | 3 万 | 必须分库 + 缓存 |
| 1 亿 | 50 亿 | ~6 万 | 30 万 | 必须分布式 + 多机房 |
| 10 亿(微信) | 500 亿 | ~60 万 | 300 万 | 顶级架构,异地多活 |

> 把这张表**背下来**,以后听到 DAU 直接报出 QPS 数量级。**面试时面试官给你 DAU 1 亿,你 3 秒内说"峰值大概 30 万 QPS,单 MySQL 肯定扛不住,要分库 + Redis 缓存 + 异步写"**——比任何花哨答案都让人佩服。

### 2.3 读写比例

不要一概用 50 次/天,**根据业务调整读写比**:

| 业务类型 | 读写比(读:写) | 说明 |
| --- | --- | --- |
| 微博 / Twitter | 1000:1 | 每个人发 5 条,看 5000 条 |
| 朋友圈 | 100:1 | 看比发多 |
| 即时通讯 | 1:1 | 收发对等 |
| 电商商品页 | 100:1 | 看的多买的少 |
| 秒杀活动 | 10:1 | 极端读写都高 |
| 监控 / 日志 | 1:100 | 写多读少 |

> **写少的系统优化读(加缓存、CDN)**;**写多的系统优化写(分库、批写、MQ 削峰)**。**搞反了等于白搭**——给一个写多读少的日志系统加 Redis 缓存,完全是浪费钱。

## 三、存储估算

### 3.1 单条记录的尺寸感

| 数据 | 平均尺寸 | 备注 |
| --- | --- | --- |
| 用户基本信息(name / email / age) | 1 KB | 含索引 |
| 一条微博文本 | 200 B | 140 字符 + 元数据 |
| 一条评论 | 200 B | 同上 |
| 一张缩略图 | 50 KB | 压缩后 |
| 一张原图 | 1-5 MB | |
| 一秒视频(720p) | 200 KB | 压缩后 |
| 一秒视频(1080p) | 500 KB | |
| 一秒直播(720p) | 100-200 KB | 实时编码低码率 |
| 一行日志 | 500 B | 含时间戳和元数据 |

### 3.2 存储估算心算

**例题**:微博 1 亿 DAU,人均发 1 条,留存 5 年,要多少存储?

```
日新增条数 = 1 亿
日新增字节 = 1 亿 × 200B = 2 × 10^10 = 20 GB
年新增 = 20 GB × 365 ≈ 7 TB
5 年 = 35 TB
```

**带索引、副本、备份**通常翻 3-5 倍:

```
35 TB × 3 = 100 TB 量级
```

> 看起来吓人,**实际上单机 NVMe 也能塞 30 TB,所以微博文本数据其实不大,真正占空间的是图片、视频**——下面 3.3 看图片。

### 3.3 图片 / 视频存储

**例题**:抖音 5 亿 DAU,人均看 50 条短视频,人均上传 1 条,平均时长 30 秒。

视频存储(写):

```
日新增 = 5 亿 × 1 = 5 亿条 × 30 秒 × 200 KB/秒
     = 5 × 10^8 × 30 × 2 × 10^5
     = 3 × 10^15 字节
     = 3 PB / 天
年:3 PB × 365 ≈ 1 EB
```

> 看到这个数字就明白**为什么短视频公司必须自建对象存储 + CDN 边缘节点**——一年存 1 EB,公有云 OSS 一年光存储费就是几亿美元。这就是估算的力量,**没估算就理解不了为什么字节、快手要自建机房**。

### 3.4 数据库行数估算

**例题**:微博的 follow 表多大?

```
1 亿用户,人均关注 200 人
follow 表行数 = 1 亿 × 200 = 2 × 10^10 = 200 亿行
```

> **MySQL 单表 5000 万行是公认上限**,200 亿行 → 必须分 400 表(每表 5000 万)。这个数字直接决定「Follow 服务必须分库分表 + 引入分布式 ID」——没估算就想不到这一步。

## 四、带宽估算

### 4.1 公式

```
带宽(bps) = QPS × 单次响应大小(bit) × 8
```

### 4.2 几个例子

**微博首页 timeline,一次请求返回 20 条 = 20 × 200B = 4 KB**

```
峰值 QPS 30 万 → 30 万 × 4 KB × 8 = 9.6 Gbps
```

**抖音视频流,看一个 30 秒视频 = 30 × 200 KB = 6 MB**

```
同时在线 1000 万,平均每人 1 分钟刷一个 → 同时下载约 5 万 QPS
5 万 × 6 MB × 8 = 2.4 Tbps
```

> 看到 Tbps 级别的带宽就立刻明白**为什么必须 CDN**——单机房出口带宽撑死 100 Gbps,2.4 Tbps 必须散到全球几百个 CDN 节点上分摊。**这就是估算引导的架构决策**。

### 4.3 反向估算:从带宽预算反推 QPS 上限

例:你机房出口 10 Gbps,单次响应 4 KB,**理论上能撑多少 QPS**?

```
10 × 10^9 / (4 × 10^3 × 8) = 3 × 10^5 QPS
```

> **超过这个数 → 必须 CDN 或多机房**,没第二条路。这种反向估算在做容量规划时极其有用。

## 五、缓存命中率与穿透估算

加 Redis 缓存的本质是**减少打到 DB 的 QPS**。算缓存效果时这两个数最关键:

### 5.1 命中率 → DB 实际负载

```
DB 实际 QPS = 总 QPS × (1 - 命中率)
```

| 缓存命中率 | 总 QPS = 10 万 | DB 实际 QPS |
| --- | --- | --- |
| 50% | 10 万 | 5 万 |
| 80% | 10 万 | 2 万 |
| 95% | 10 万 | 5000 |
| 99% | 10 万 | 1000 |
| 99.9% | 10 万 | 100 |

> 工程上**95% 是基本合格,99% 是优秀,99.9% 是极致**。**命中率从 95% 提升到 99%,DB 负载降 5 倍**——这就是缓存调优的价值。

### 5.2 缓存内存估算

```
缓存大小 = 热点数据条数 × 单条大小 × 1.5(碎片 + overhead)
```

**例**:微博热门 timeline,1 亿用户,Top 20% 是热门,每用户 timeline 100 条 × 200 B:

```
2000 万 × 100 × 200 = 4 × 10^11 = 400 GB × 1.5 = 600 GB
```

> **单台 Redis 撑死 256 GB(内存大了 RDB / AOF 慢)**,所以要分片为 3-5 个实例的 Redis Cluster。**没估算就理解不了"为什么要 Redis Cluster 而不是单机 Redis"**。

## 六、连接数与文件描述符

经常被忽略,但生产事故重灾区:

```
长连接服务的连接数 = 同时在线用户数(每人一条 WS / TCP)
单机 Linux 默认 ulimit = 1024,需要调到 10 万级别
单机 IM 网关上限通常 5-10 万长连接(取决于 epoll 调优)
```

**例**:微信同时在线 5 亿,**至少需要** 5 亿 / 10 万 = **5000 台 IM 网关**。

> 这个数字直接告诉你「微信不是几台机器扛起来的,是几千上万台机器堆出来的」。**长连接服务的扩容靠机器数,不靠单机性能**。

## 七、估算的「五倍法则」

工程上有几个经验倍数,记住能省事:

| 场景 | 倍数 | 原因 |
| --- | --- | --- |
| 平均 → 峰值 QPS | × 5 | 高峰期(午饭、晚 8-10 点)集中流量 |
| 业务量 → 副本量 | × 3 | 一主两从 / 多副本 |
| 数据 → 总存储 | × 3-5 | 索引 + 副本 + 备份 |
| 当下 → 一年后 | × 2-10 | 业务增长预留 |
| 设计容量 → 实际容量 | × 2 | Buffer,生产从不跑满 |

> 估算完原始数字,**记得乘 buffer**。一个常见错误是「按平均 QPS 设计容量」——上线一周就因为高峰打挂,然后慌忙扩容。**所有真实生产系统都按 2-5 倍峰值 buffer 来设计**。

## 八、一道完整估算题:设计 Twitter

「设计 Twitter,DAU 2 亿」——3 分钟内你应该能给出:

```
1. QPS 估算
   - 读 timeline:DAU × 人均刷 100 次 = 200 亿次/天 ≈ 23 万 QPS,峰值 ~100 万 QPS
   - 发推:DAU × 人均发 0.5 条 = 1 亿条/天 ≈ 1200 QPS,峰值 ~6000 QPS

2. 存储估算
   - 推文:1 亿/天 × 300 B(含元数据) = 30 GB/天 = 11 TB/年
   - 含索引副本 ×3 = 30 TB+/年
   - Follow 关系:2 亿用户 × 平均关注 200 = 400 亿条 → 必须分表

3. 带宽估算
   - 读 timeline 100 万 QPS × 一次返回 20 条 × 300 B
     = 6 GB/s = 48 Gbps → 必须 CDN

4. 缓存
   - 活跃用户 timeline 进 Redis,2 亿 × 20% × 100 条 × 300 B
     = 1.2 TB → Redis Cluster 5+ 实例

5. 架构判断
   - 30 万写入 QPS 不算高,核心难点是 100 万读 → 必须 Push 模型预计算 timeline
   - 大 V 单独处理(Pull),不能 fanout 几千万次
```

> **这一段就是 Twitter 设计的"前 5 分钟"**——后面的方案细节都建立在这些数字上。**面试官听完这段,基本就知道你能不能上 P7**。

## 九、踩坑提醒

1. **不区分平均 QPS 和峰值 QPS**——容量按峰值算,成本按平均算
2. **拍脑袋写"我们 QPS 1 万"**——必须从 DAU + 行为推导,**不要直接给 QPS**,容易错一个量级
3. **忽略带宽**——很多系统是带宽先到天花板,不是 CPU
4. **存储忘了乘副本和索引**——3-5 倍 buffer 跑不掉
5. **算缓存按总数据量,不按热点**——80/20 法则,只有 20% 热点要进缓存
6. **忽略长连接的 fd 上限**——IM / 推送系统翻车主因
7. **不算"一年后"**——按当下设计,半年后就扩容血亏
8. **数量级错一位**——这是新手最致命的错,**反复用 10^5 = 一天秒数 校验**
9. **不写过程只给结果**——面试时一定要把 DAU → QPS → 存储 → 带宽 这条链讲出来,**结果不重要,推理重要**
10. **过度精确**——估算就是数量级,「1.2345 万 QPS」纯属浪费时间,「~1 万」就够了
