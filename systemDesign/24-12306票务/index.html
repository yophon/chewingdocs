    12306 票务

    12306 是中国互联网最特殊的系统——它的难度不在 QPS,而在库存模型。电商秒杀的库存是一维的(1000 件 iPhone),12306 的库存是多维的——一张「北京 → 广州」的车票实际上是「北京-郑州-武汉-长沙-广州」5 个区间的集合,每个区间有自己的座位。同一张座位在北京-郑州区间可以卖,武汉-广州区间也可以卖,但北京-广州不能再卖。这种"座位接力"的特性让 12306 的库存系统是分布式系统里独一无二的。再加上节假日 5 亿人同时抢票,它必须有十分独特的设计。

    一句话先记住:12306 的核心挑战不是 QPS,是"区间共享座位"的库存模型——一个座位在不同区间可以卖给不同人,扣库存等于"区间矩阵的多维操作"。春运瞬时洪水靠"答题排队 + 候补订单"两道闸——把"同时抢"变成"延迟抢",把"抢失败"变成"等候补"。12306 的反逻辑工程:不是优化"立刻成功",是优化"用户能在合理时间内拿到票"。

    一、需求拆解

    1.1 功能需求

    • 查票:输入起点 / 终点 / 日期 → 显示有票车次
    • 订票:选车次 / 座位等级 → 锁定库存 → 30 分钟内付款
    • 退票 / 改签
    • 候补:无票时排队,有人退票自动补给候补者
    • 多人订票(一次订几张)
    • 学生票 / 儿童票(特殊价格)

    1.2 非功能需求

    • 可用性:99.99%
    • 公平性:先到先得(同一秒内的请求公平)
    • 不超卖:绝对不能多卖
    • 规模:平时 DAU 1000 万,春运峰值 6000 万 + 瞬时 100 万 QPS

    1.3 这一轮不做

    • 路径规划(我去广州可以坐哪些车)
    • 价格 / 优惠
    • 与 12306 集成的第三方(智行、携程)

    二、容量估算

    平日:
  日订单 1000 万 → 平均 QPS 100,峰值 1 万

春运:
  日订单 1.5 亿 → 平均 QPS 1700,峰值 100 万+
  早 8 点开抢瞬间:5000 万人涌入

查询:
  下单前查询 N 次,QPS 是下单的 50 倍
  峰值查询 QPS = 5000 万

    12306 的 QPS 比电商秒杀高一个量级——因为参与人数太多了。

    三、12306 的"反常识"难点:库存模型

    3.1 普通电商库存

    iPhone 库存 1000
卖一台 → 999 → ... → 0

    单值,简单。

    3.2 12306 的库存

    车次:G1234,北京→广州,中途停 4 站(郑州 / 武汉 / 长沙 / 韶关)
座位号:01A
车站:[北京, 郑州, 武汉, 长沙, 韶关, 广州]
区间:[北-郑, 郑-武, 武-长, 长-韶, 韶-广] 共 5 段

如果用户买"北京→武汉":
  锁定 [北-郑, 郑-武] 两段(其他段仍可卖)

如果用户买"郑州→韶关":
  锁定 [郑-武, 武-长, 长-韶] 三段(北-郑、韶-广 仍可卖)

    同一个座位号在不同区间可以多次卖——这就是"接力座位"。

    3.3 库存表示:区间位图

    G1234 的 01A 座位:
  [北-郑][郑-武][武-长][长-韶][韶-广]
  [  0  ][  0  ][  0  ][  0  ][  0  ]    ← 全空,5 段都可卖

有人买"北京→武汉":
  [  1  ][  1  ][  0  ][  0  ][  0  ]    ← 前两段已售

另一人想买"长沙→广州"(占后两段):
  [  1  ][  1  ][  0  ][  1  ][  1  ]    ← 前两段后两段都售,中间空

另一人想买"武汉→长沙"(占中间一段):
  [  1  ][  1  ][  1  ][  1  ][  1  ]    ← 全售完

    这一个座位被卖给 3 个人——但他们坐的区间不重叠。

    3.4 库存操作:位运算

    # 区间表示成 bit:5 段对应 5 bit
# 北京→郑州 = 0b10000(第 1 段)
# 北京→武汉 = 0b11000(前 2 段)
# 郑州→韶关 = 0b01110(中间 3 段)

def can_sell(seat_status, interval_mask):
    """检查 seat 在 interval 区间是否可卖"""
    return (seat_status & interval_mask) == 0

def sell(seat_status, interval_mask):
    """卖出 seat 的 interval 区间"""
    return seat_status | interval_mask

    纯位运算,极快。

    四、库存的物理存储

    CREATE TABLE seat_inventory (
    train_no    VARCHAR(10),     -- G1234
    travel_date DATE,
    seat_id     VARCHAR(20),     -- 01A
    status      INT,             -- 32 位的位图(支持最多 32 段)
    PRIMARY KEY (train_no, travel_date, seat_id)
);

    4.1 数据规模

    全国列车 ~ 1 万趟
每趟车 1000 座位
每天提前 30 天开售
→ 行数 = 1 万 × 1000 × 30 = 3 亿

每行 50 字节 → 15 GB
全部进 Redis(1 个集群够用)

    4.2 用 Redis 存

    Key:    seat:{train_no}:{date}:{seat_id}
Value:  bitmap(32 bit)

操作:GETBIT / SETBIT / BITCOUNT / 自定义 Lua 操作

    五、抢票流程

    5.1 朴素流程(行不通)

    用户提交订单
   ↓
扫描所有座位,找一个可卖的(或多人组合)
   ↓
扣库存
   ↓
生成订单

    问题:扫描全部座位,每次几千次 Redis 操作,QPS 上不去。

    5.2 实战流程

    1. 提前预计算"每个车次的余票统计"  (offline,每分钟更新)
   缓存到 Redis: train_summary:G1234:2026-05-01 → 各等级余票数

2. 用户查询时:直接读 summary,毫秒级

3. 用户下单:
   a. 检查 summary 是否还有票(粗筛)
   b. 通过 Lua 脚本原子地"找一个可卖座位 + 扣库存"
   c. 入 MQ,异步生成订单
   d. 返回"排队中"

4. Worker 处理:
   a. 真正写订单到 DB
   b. 推送给用户"成功,请支付"

    5.3 选座算法

    用户买"北京→武汉":

    def find_seat(train, date, interval_mask):
    # 遍历所有座位(1000 个),找第一个可卖的
    for seat_id in seats:
        status = redis.get(f"seat:{train}:{date}:{seat_id}")
        if (status & interval_mask) == 0:
            # 用 Lua 原子地占座
            success = redis.eval(LUA_SCRIPT, ...)
            if success:
                return seat_id
    return None

    Lua 保证原子——避免两个用户同时抢到同一个座。

    优化:用 BITMAP 索引或预排序池,避免每次遍历 1000 个。

    六、春运瞬时洪水的应对

    每年春运抢票是国民工程实验:

    6.1 答题验证码

    不是"防机器",是"打散瞬时流量":

    不答题:5000 万人同时点抢,每秒 5000 万 QPS,任何系统都挂
答题:每人 5-30 秒答完题,流量自然摊到 30 秒内
     瞬时 QPS 从 5000 万降到 200 万,系统能扛

    12306 答题被骂了 10 年,但它真的解决问题——本质是把"同步洪水"变成"异步流"。

    6.2 排队 + 异步反馈

    抢票不是同步返回结果:

    用户点抢 → 入队 → 立即返回"排队中(排在第 12345 位)"
→ Worker 慢慢处理 → 处理完推送"成功 / 失败"

    用户视角:不卡,不超时,几分钟内有结果。 系统视角:洪水在 MQ 里慢慢消化,DB / 库存系统稳态。

    6.3 候补订单

    无票时不直接 fail,进候补队列:

    有票:抢成功
无票:候补
有人退票 → 自动从候补队列取出第一个补给他

    优势:

    • 用户体验好(不是"没了就没了")
    • 系统流量平滑(候补处理是后台任务)
    • 黄牛囤票被打击(囤的票退了直接给候补)

    七、查询的优化

    12306 的查询 QPS 比下单高 50 倍——必须重点优化。

    7.1 多级缓存

    浏览器(Service Worker 缓存)→ 一段时间内不重复查
   ↓
CDN 缓存(同一个查询结果对所有人都一样)
   ↓
反向代理缓存(机房入口)
   ↓
应用本地缓存(Caffeine)
   ↓
Redis 缓存(分钟级 summary)
   ↓
真正去查 seat 详情(罕见)

    7.2 查询的"模糊化"

    不让用户看到精确余票:

    实际有 17 张票
显示:"有票" / "充足" / "紧张" / "有 < 20 张"

不显示精确数字 → 黄牛和机器人无法精确攻击 + CDN 缓存命中率高

    7.3 查询的限频

    单 IP 每秒最多查 5 次
单 user_id 每分钟最多查 100 次

    防爬虫和黄牛刷新。

    八、退改签

    8.1 退票

    用户申请退票
→ 检查时间窗口(开车前多久)
→ 计算退款金额(扣手续费)
→ 释放座位库存(SETBIT 0)
→ 通知候补队列"有票了"
→ 候补 Worker 自动出票给候补第一名
→ 真退款给用户

    8.2 改签

    改签 = 退票 + 重新订票,作为一个事务:

    事务:
  1. 释放原票库存
  2. 锁定新票库存
  3. 计算差价(多退少补)
  4. 修改订单
任一失败 → 全部回滚

    九、为什么 12306 这么"复杂"

    9.1 业务复杂度本质高

    • 多维库存(区间矩阵)
    • 多类型乘客(成人 / 儿童 / 学生 / 残军)
    • 多座位等级(商务 / 一等 / 二等 / 软卧 / 硬卧 / 硬座)
    • 多支付方式(微信 / 支付宝 / 银行卡)
    • 多场景(短途、长途、夕发朝至、跨日)

    9.2 中国铁路网巨大

    • 1 万趟车
    • 5000 个车站
    • 路网总里程 15 万公里
    • 节假日运力翻 3 倍

    9.3 国情约束

    • 必须实名(防黄牛 / 公安要求)
    • 必须支持身份证 / 学生证 / 残军证
    • 节假日免票政策(老人、儿童)

    12306 不是"工程师不行",是业务真的复杂——任何一个团队接手都得做成这样。

    十一、12306 的"工程教训"

    12306 早期(2010-2014)惨不忍睹——值得记的几个教训:

    1. 早期用 Oracle 同步事务做库存——崩了好几年,后来改 Redis + 异步才稳
    2. 早期不答题,直接抢——每年春运挂几次
    3. 早期不做候补——大家疯狂刷新,流量是真实需求的 100 倍
    4. 早期单机房——故障无切换
    5. 早期不防黄牛 SDK——12306 协议被黄牛逆向,直接走 API 抢票

    今天(2026)的 12306:Redis + Kafka + 多机房 + 候补 + 答题 + 实名 + 大数据风控,每年春运基本不挂了。

    十二、踩坑提醒

    1. 库存当一维处理——根本不是 12306 模型
    2. 同步抢票——用户卡死,服务挂
    3. 无答题——瞬时洪水扛不住
    4. 无候补——刷新 100 倍流量
    5. 不缓存查询——查询 QPS 比下单大 50 倍
    6. 精确显示余票——给黄牛精准攻击的钩子
    7. 不实名——黄牛囤票
    8. 不限频——爬虫吃光带宽
    9. 支付链路慢——订单超时回流,真实用户买不到
    10. 不监控候补处理速度——退票后候补半天没补上,用户投诉
    11. 不做大数据风控——黄牛账号永远封不完
    12. 以为 12306 = 普通秒杀——架构完全不一样,业务模型决定一切
