# 设计文档与系统设计面试

系统设计能力是**软件工程师的天花板**——3 年工程师能写出 CRUD,5 年工程师能优化性能,**10 年工程师才能独立设计能扛 1 亿用户的系统**。这种能力分两个外显出口:**面试时白板答题**(45 分钟拿到 P7 / Senior 的核心环节)和**真实工作中写设计文档**(让团队认可你方案的工具)。这两件事**用的是同一套思维**——前 29 篇讲的所有内容,在这两个场景里被压缩成一个标准答题流程。本系列的最后一篇,就讲清楚这个流程。

> 一句话先记住:**系统设计 = 五段式答题**——需求拆解 → 容量估算 → 数据模型 → 架构演进 → 关键取舍。**面试和文档用同一套结构**,只是篇幅不同。**面试 45 分钟,文档 5-20 页**,但**底层思维完全相同**——这就是为什么本系列从 14 篇开始所有案例都套这个模板。**会面试不一定会做事,会做事一定会面试**——所以多做真项目永远比刷题更值钱。

## 一、五段式模板回顾

整个系列的核心模板,**任何系统设计场景都套用**:

```
1. 需求拆解        ← 30% 时间(被新手低估)
   - 功能边界(做什么、不做什么)
   - 非功能(QPS / 延迟 / 一致性 / 可用性)

2. 容量估算
   - DAU → QPS
   - 数据量 → 存储
   - 流量 → 带宽
   - 数量级判断决定架构等级

3. 数据模型
   - 核心实体
   - 访问模式
   - 量级与分布

4. 架构演进
   - V1 单机
   - V2 加缓存
   - V3 分库分表
   - V4 多机房
   - 每一步都要说明"为什么必须升级"

5. 关键取舍
   - CAP 选什么
   - 推还是拉
   - 强一致还是最终一致
   - 同步还是异步
```

> **新手的错误是直接跳到第 4 步画架构图**。**老手的关键差异在第 1-2 步** —— 把模糊需求翻译成清晰约束。

## 二、系统设计面试:45 分钟答题流程

### 2.1 时间分配

```
0 - 5 min:  澄清需求(问问题、画边界)
5 - 10 min: 容量估算(QPS / 存储 / 带宽)
10 - 15 min: 数据模型(核心表、关键字段)
15 - 30 min: 架构演进(从 V1 到 V4)
30 - 40 min: 深入某个组件(面试官引导)
40 - 45 min: 取舍讨论 + 失败模式
```

### 2.2 第一步:澄清需求(关键)

面试官说"设计 Twitter"——**你不应该直接画图**。

**先问 5-10 个问题**:

```
功能边界:
- DAU 多少?(决定数量级)
- 是国内还是全球?(决定多机房)
- 要做推荐算法吗?(决定 Feed 复杂度)
- 要做评论 / 点赞 / 转发吗?
- 是不是要支持视频?(决定是否引入 CDN / 转码)

非功能:
- 一致性要求?
- 延迟要求?
- 我们假设可用性要求 99.99%,可以吗?
```

**面试官最看重的就是这一步** —— 它体现你"做减法"的能力。**直接开始设计的候选人,99% 拿不到 P7**。

### 2.3 第二步:容量估算(必算)

3 分钟内给出:

```
- 平均 / 峰值 QPS
- 数据增量(GB / TB / PB 级别)
- 带宽
- 缓存大小
- 单机能扛吗?要分多少库?
```

**详见 02 篇容量估算**——这是面试硬通货,**必须张嘴就来**。

### 2.4 第三步:数据模型

不要写完整 SQL,**只画核心表**:

```
User(id, name, email, ...)
Tweet(id, user_id, content, created_at)
Follow(follower_id, followee_id)
```

**关键讨论**:
- 主键 / 索引
- 分片键
- 量级与分库

### 2.5 第四步:架构演进(主战场)

**不要画终极版**,**画 V1 → V4 演进**:

```
V1: 单机能扛吗? (DAU < X)
   什么瓶颈?
↓
V2: 加缓存 / 读写分离 (DAU < Y)
   什么瓶颈?
↓
V3: 分库分表 (DAU < Z)
   什么瓶颈?
↓
V4: 多机房 / 异地多活 (DAU > W)
```

每一步说清楚 **"V_n 的什么瓶颈逼出了 V_{n+1}"**。

### 2.6 第五步:深入某个组件

面试官通常会引导你深入一个模块:

```
"你刚才提到用 Redis 缓存,讲讲缓存击穿怎么处理"
"你说要分库,分片键怎么选?"
"用了 MQ,怎么保证消息不丢?"
```

**这是真正的考核点** —— 你对每个组件了解多深?是只会"用",还是知道"为什么这么用、什么时候不该用"?

### 2.7 第六步:讨论取舍 + 失败模式

收尾:

```
"这个方案的取舍是 X 换 Y"
"如果 A 组件挂了,整个系统怎么响应?"
"如果流量翻 10 倍,瓶颈会先出现在哪?"
"如果业务需要全球用户,要怎么改?"
```

**这一段决定 P7 还是 P8**——不只回答方案,而是**讨论方案的边界**。

## 三、面试加分项

### 3.1 主动画图 / 写公式

```
[Client] → [LB] → [Web ×N] → [Redis Cluster]
                              ↘ [MySQL ×16]

QPS = DAU × 50 / 86400 ≈ 6 万
峰值 = 6 万 × 5 = 30 万
单 Redis 上限 = 10 万 → 必须 Cluster
```

**让面试官清楚看到你的推理过程**——文字 < 图表 < 数字。

### 3.2 引用真实案例

```
"这个推拉结合的 timeline 设计,Twitter 早期就是这么做的"
"分布式 ID 用雪花,Twitter 开源,后来 Sony / Discord 都改造过"
"双 11 阿里就用了类似的限流策略"
```

**说明你不是纸上谈兵**——读过工程文章、关注业界实践。

### 3.3 主动提到取舍

```
"我推荐用 Cassandra 而不是 MySQL,因为我们的访问模式是按 user_id 分区的简单 KV 查询"
"代价是失去了复杂查询能力,但我们的需求里没有那部分"
```

**永远主动说"代价"**——不主动说就显得你只看到优点。

### 3.4 说"我不知道"

不知道时**别瞎编**:

```
"具体的 Cassandra 一致性级别配置我记不清,但我知道它支持 ONE / QUORUM / ALL,可以根据需要调"
```

**会承认不知道的工程师比硬编的强 10 倍** —— 面试官也是工程师,知道你在编。

## 四、面试常见问题及"如何答"

### 4.1 "怎么扛住 100 万 QPS?"

**答题套路**:

```
1. 这是读多还是写多?(决定优化方向)
2. 容量估算:这个量级单机扛不住,必须分布式
3. 读多写少:多级缓存(Caffeine + Redis + CDN)+ 读写分离
4. 写多读少:分库分表 + 异步写 + 削峰 MQ
5. 监控 + 限流兜底
```

### 4.2 "怎么保证不丢消息?"

```
1. 生产端:acks=all + 重试
2. MQ:多副本 + 持久化
3. 消费端:手动 ACK + 失败重试 + 死信队列
4. 业务幂等(关键:Exactly Once 不存在,只能至少一次 + 幂等)
```

### 4.3 "DB 主从延迟怎么办?"

```
1. 监控延迟(用心跳表,不信 Seconds_Behind_Master)
2. 关键写后立即读 → 强制走主库
3. 查询前等 GTID 应用完(强一致场景)
4. 跨机房复制 → 用半同步保证不丢
```

### 4.4 "怎么处理 hot key?"

```
1. 本地缓存 Caffeine(完全不打 Redis)
2. 多副本 Redis(分摊读)
3. CDN 缓存(对外接口)
4. 限流(防止 hot key 影响其他请求)
```

## 五、写真实工作中的设计文档

### 5.1 文档结构(标准模板)

```
# XXX 系统设计文档

## 1. 背景与目标
- 业务背景(为什么要做)
- 目标(解决什么问题)
- 非目标(不在范围内)

## 2. 现状分析
- 当前架构是什么
- 当前瓶颈 / 问题

## 3. 需求
- 功能需求
- 非功能需求(QPS / 延迟 / 一致性 / 可用性)

## 4. 容量估算
- DAU、QPS、存储、带宽

## 5. 设计方案
- 数据模型
- 架构图(画清楚)
- 关键流程(时序图)
- 容错设计

## 6. 方案对比
- 方案 A vs 方案 B vs 方案 C
- 各自取舍
- 推荐方案及原因

## 7. 风险与回滚
- 已知风险
- 监控指标
- 回滚方案

## 8. 实施计划
- 里程碑
- 参与团队
- 估算工时

## 9. 附录
- 性能压测数据
- 参考文献
```

### 5.2 一份好文档的特征

- **数字驱动**:不写"性能不错",写"p99 80ms,QPS 5000"
- **画图**:架构图、时序图、状态机图
- **方案对比**:列出至少 2-3 个方案,说明为什么选这个
- **能 Code Review**:别人能照着实现
- **可回滚**:每一步都有回滚方案

### 5.3 一份烂文档的特征

```
× 没有数字,全是"高性能 / 高可用"
× 只有一个方案,没对比
× 无图(纯文字描述架构)
× 没估算,直接给方案
× 没考虑失败模式
× 篇幅过长但没重点(40 页讲废话)
```

> **好文档让团队认可、上线后不变形**——一边做一边发现"原来文档没考虑这个" = 文档没写好。

## 六、设计文档的"评审流程"

### 6.1 评审的目的

不是让你"过关",是**找出方案漏洞**:

```
1. 团队先 Async Review(预读)
2. 安排会议讨论(1 小时)
3. 重点议题:风险、取舍、边界条件
4. 文档迭代,直到所有 Concern 解决
5. 进入实施
```

### 6.2 评审的关键问题

评审者通常会问:

```
"如果 X 组件挂了怎么办?"
"流量翻 10 倍怎么办?"
"和 Y 系统冲突吗?"
"上线后怎么监控?"
"有更简单的方案吗?"
"投入 / 产出比如何?"
```

**回答得好 = 文档过关**。回答不上来 = 文档要再迭代。

## 七、系统设计能力的"成长阶梯"

```
L0:CRUD
   能写接口、用 ORM、做基本测试

L1:能用基础设施
   懂 Redis / MQ / 缓存,能用对地方

L2:抗压设计
   QPS 数量级估算、分库分表、缓存策略、限流熔断
   能设计 1000 万 DAU 的系统(P6 / Senior)

L3:案例设计
   能套五段式模板设计任意系统(短链 / IM / Feed)
   能讲出每一步的取舍(P7 / Staff)

L4:多机房 / 全球
   异地多活、Cell 单元化、灰度发布、容灾演练
   能设计 1 亿 DAU 系统(P8 / Principal)

L5:行业洞察
   能预判 3-5 年技术演进方向
   能影响行业(写论文 / 开源项目领导者)
   (P9+ / Distinguished)
```

> **本系列覆盖到 L3-L4**——L5 是天赋 + 长期积累 + 运气,**不是看教程能到的**。

## 十、最后的话:系统设计的本质

```
系统设计不是技术,是工程思维:
  
  - 在约束里找最优解
  - 在不确定性里做决策
  - 在团队里推动方案落地
  - 在故障中保持冷静
```

这不是"刷题"能学的,是**做项目 + 看真系统 + 经历事故** 才能修炼的。

> **看完这 30 篇,你有了"工具箱"和"思维框架"**——剩下的,**靠真实工作中的迭代积累**。
