架构演进史 ​

每一代架构都是被前一代的瓶颈逼出来的,不是工程师为了用新技术。理解这条演进链,你会发现「微服务」「Service Mesh」「Cell 架构」不是潮流名词,是真实痛点的解决方案。新手最爱犯的错是「我们用单体就行」或者反过来「上来就微服务」——其实在哪一代,只跟你的团队规模 + 流量规模有关。这一篇按时间线走一遍,看清每一步「为什么不得不升级」。

一句话先记住:架构演进的驱动力永远是两个东西——流量增长压垮了上一代,组织变大撕裂了上一代。技术只是表象,「哪一代架构合适你」=「你团队多少人 + 你 QPS 多少」交叉决定——20 人团队上 200 个微服务就是自杀,500 人公司还在跑单体也是自杀。

一、V0:单机一切(LAMP / 单文件) ​

时间:2000 - 2010 早期

┌────────────────────────────────┐
│  一台物理机                      │
│  Apache + PHP + MySQL + 静态文件 │
└────────────────────────────────┘

特征:

  • 代码、DB、文件、缓存全在一台机器
  • 部署 = ssh 上去 git pull
  • 备份 = mysqldump + scp

适用规模:DAU < 1 万,内部工具、个人博客、小公司官网。

瓶颈来源:

  • 单机 CPU / 内存上限
  • 单机宕机 = 服务整体挂掉(0 高可用)
  • 部署时短暂不可用
  • 一个慢 SQL 拖死整个机器

别小看 V0,今天 90% 的 SaaS 创业项目第一年都该是 V0——别上来就 K8s,3 个用户根本不需要。Stack Overflow 直到 1000 万 DAU 都还基本是 V0 的进化版(只 9 台 Web 服务器),活生生的反例。

二、V1:应用与数据分离 ​

时间:2005 - 2015

┌──────────┐         ┌──────────┐
│  Web 1   │ ──────▶ │  MySQL   │
└──────────┘         └──────────┘

改变:把 DB 单独拆出来,Web 服务器和数据库各占一台。

为什么:

  • DB 和应用对硬件要求不同——DB 吃内存 / 磁盘,Web 吃 CPU
  • DB 重启 / 优化不影响 Web 部署

新瓶颈:

  • Web 单点(挂了全挂)
  • DB 单点(挂了全挂)
  • 静态资源占用 Web 带宽

三、V2:多 Web + 负载均衡 + 静态分离 ​

时间:2010 - 2020,绝大部分中型项目的当前形态

                     ┌──────────┐
                     │  Web 1   │ ────┐
┌──────────┐ ──▶ ┌──┐  Web 2   │ ────┤    ┌──────────┐
│  Client  │     │LB│  Web 3   │ ────┼──▶ │  MySQL   │
└──────────┘ ──▶ └──┘  Web 4   │ ────┘    └──────────┘
                     └──────────┘
                          ↓
                     ┌──────────┐
                     │   CDN    │  (静态资源)
                     └──────────┘

改变:

  • 多台 Web,前面挂 LB(Nginx / AWS ALB)
  • 静态资源(图片、JS / CSS)走 CDN
  • 这一代起,Web 必须做成无状态(Session 进 Redis 或 Cookie)

为什么:

  • Web 任何一台挂了 LB 自动剔除——真正的高可用从这一代开始
  • 加机器就能扛更多流量(水平扩展)
  • 静态资源不再吃 Web 带宽,延迟也低

新瓶颈:

  • DB 仍是单点
  • DB 撑不住读 QPS

大多数公司在 V2 卡很久——只要 DB 没爆,V2 + 缓存就能撑到 DAU 100 万。别为了"显得现代"提前升级。

四、V3:DB 主从复制 + 读写分离 + 缓存 ​

时间:2010 至今,中大型系统标配

                     ┌──────────┐
                     │  Web ×N  │
                     └──────────┘
                       ↓ 读  ↑ 写
              ┌────────┴───────┐
         ┌────┴────┐       ┌──────────┐
         │  Redis  │ ◀───  │ MySQL 主 │
         └─────────┘       └────┬─────┘
                                ↓ 复制
                     ┌──────────┴──────────┐
                ┌────┴────┐         ┌──────┴───┐
                │MySQL 从1│   ...   │MySQL 从n │
                └─────────┘         └──────────┘

改变:

  • MySQL 一主多从,读分散到从库,写只在主库
  • Redis 缓存热点数据(用户信息、配置、热门列表)
  • 应用层负责「读哪个 / 写哪个」的路由

为什么:

  • 读 QPS 单机 5-10 倍提升(从库越多扩越多)
  • 缓存命中率 80%+,DB 实际负载降一个量级

新瓶颈:

  • 主从复制延迟(写入主库到从库可见有 100ms ~ 几秒)
  • 主库写入仍是瓶颈
  • 单库数据量到几亿行后查询变慢

这一代关键学问是主从延迟——刚下单的用户立刻刷新,可能查不到自己的订单(查到了从库,主库刚写的还没复制过来)。关键写入后立即读必须强制走主库(SET TRANSACTION_READ_WRITE 或应用层标记)。

五、V4:垂直拆分(按业务模块拆库 + 拆服务) ​

时间:2012 至今

┌────────────┐  ┌────────────┐  ┌────────────┐  ┌────────────┐
│ User 服务   │  │ Order 服务  │  │Product 服务│  │Comment 服务│
└─────┬──────┘  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
      ↓               ↓               ↓               ↓
┌─────┴──────┐ ┌─────┴──────┐ ┌─────┴──────┐ ┌─────┴──────┐
│  User DB   │ │  Order DB  │ │ Product DB │ │ Comment DB │
└────────────┘ └────────────┘ └────────────┘ └────────────┘

改变:

  • 按业务模块切应用(User / Order / Product 各自一套服务)
  • 每个服务有自己的 DB(库不再共享)
  • 服务间通过 HTTP / RPC 调用

为什么:

  • 单库数据量 / 写入量到上限,垂直拆是最自然的
  • 团队人多了(50+),不同模块代码冲突严重
  • 不同模块发布节奏不同(Order 一天 10 次,User 一周 1 次),单体变累赘

新瓶颈:

  • 单业务仍可能太大(Order 表自己就 100 亿行)
  • 跨服务事务(下单 = User 扣余额 + Order 写表 + Product 减库存)
  • 分布式调用复杂度上升

垂直拆是组织规模的产物,不是流量。就算流量小,只要团队 > 50 人,该拆还是要拆——不然代码变成谁也不敢动的泥球。

六、V5:水平拆分(分库分表) ​

时间:2013 至今,大流量系统标配

                     ┌────────────┐
                     │ Order 服务  │
                     └─────┬──────┘
                           │ 路由(按 user_id 哈希)
              ┌────────────┼────────────────────┐
              ↓            ↓                    ↓
       ┌─────────────┐ ┌─────────────┐    ┌─────────────┐
       │ Order DB 0  │ │ Order DB 1  │... │Order DB 1023│
       │  shard_0~7  │ │  shard_8~15 │    │             │
       └─────────────┘ └─────────────┘    └─────────────┘

改变:

  • 同一业务的同一张表,按某个键(user_id)切成 N 份(shard)
  • 应用通过分片路由器找到对应分片
  • 物理上分散到多台 MySQL

为什么:

  • 单业务也撑不住单机:Order 表 100 亿行,单库慢
  • 写入 QPS 也分散到 N 个主库

新瓶颈:

  • 分片键选错就是地狱(非分片键查询要扫全部分片)
  • 跨分片 join 基本不可能,要在应用层做
  • 全局唯一 ID 需要分布式发号器(雪花 / TiDB 风格)
  • 数据迁移、扩容极其麻烦(从 8 分片扩到 16 分片)

分库分表是"上一步"——一旦做了就不可逆。千万别因为"听说要分库"就提前分,先用所有能扛的招(读写分离、缓存、归档老数据)再分。一旦分了,后面三年都在还分库的债。

七、V6:微服务 + 服务治理(注册中心 + 网关) ​

时间:2014 至今

       ┌─────────────┐    ┌──────────────┐
       │  API Gateway│    │Service Registry│ (Nacos / Consul)
       └──────┬──────┘    └──────────────┘
              │                 ↑↓ 注册 / 发现
   ┌──────────┼──────────┐
   ↓          ↓          ↓
┌────┐  ┌─────────┐  ┌─────────┐
│User│  │ Order   │  │Product  │  ......(几十~几百个服务)
└────┘  └─────────┘  └─────────┘
   ↓         ↓         ↓
[各自的 DB / 缓存 / MQ]

改变:

  • 服务粒度更细(从「Order 服务」拆成「OrderCreate / OrderQuery / OrderAfterSale 」)
  • 引入服务注册中心(谁活着、IP 多少自动同步)
  • 引入 API 网关(统一鉴权、限流、路由、日志)
  • 配置中心、链路追踪、监控等成为基础设施

为什么:

  • 团队 200+ 人,微服务支持每个小队独立发版
  • 业务复杂度爆炸,需要清晰的服务边界 + 标准化治理

新瓶颈:

  • 服务依赖图变成蜘蛛网(几百个服务调来调去)
  • 排查链路问题:一个请求过 20 个服务,谁慢?
  • 部署复杂度炸裂(每个服务自己一套 CI / CD)
  • 基础设施投入比业务代码还多

微服务的代价被严重低估——一个 50 人团队上微服务,会发现 80% 的精力都在治理,而不是写业务。没到 200 人 + 多业务线,别上微服务。

八、V7:Service Mesh + 容器编排 ​

时间:2018 至今

                ┌─────────────┐
                │  K8s + Istio│
                └─────────────┘

每个 Pod = 业务容器 + Sidecar 代理(Envoy)
所有 RPC 走 Sidecar:
  - 自动 mTLS
  - 自动重试 / 熔断 / 限流
  - 自动链路追踪
  - 自动金丝雀发布

业务代码不感知,治理逻辑在 Sidecar

改变:

  • 治理逻辑(熔断、限流、加密、重试)从业务代码下沉到 Sidecar
  • 容器编排(K8s)统一调度 / 滚动 / 弹性
  • 多语言混合架构成为可能(Go + Java + Python 都能用同一套治理)

为什么:

  • V6 的治理 SDK(Spring Cloud / Dubbo)绑死语言,多语言团队痛
  • 治理升级要每个服务改代码,慢
  • 容器化让基础设施可编程

新瓶颈:

  • Sidecar 增加 1ms+ 延迟,每个服务多占 100MB 内存
  • 排查问题更难(业务请求要穿过两层代理)
  • 学 K8s + Istio 的成本巨大,需要专门 SRE 团队

Service Mesh 不是「更先进」,是「更复杂」。如果你不是大厂或多语言团队,继续用 V6 的 Spring Cloud / Dubbo 完全够。Mesh 是给 1000+ 服务的公司用的,不是给创业公司用的。

九、V8:Cell / 单元化 / 异地多活 ​

时间:2017 至今,只在头部公司

┌─────────────── 北京机房 (Cell-A) ───────────────┐
│ 完整一套服务: User / Order / DB / Cache / MQ    │
│  服务 0~100M 用户                              │
└────────────────────────────────────────────────┘
              ↕ 跨机房同步(数据)
┌─────────────── 上海机房 (Cell-B) ───────────────┐
│ 完整一套服务                                     │
│  服务 100M~200M 用户                           │
└────────────────────────────────────────────────┘
              ↕
┌─────────────── 深圳机房 (Cell-C) ───────────────┐
│ ...                                            │
└────────────────────────────────────────────────┘

改变:

  • 每个 Cell 是自包含的微型系统(从入口到 DB 都全)
  • 用户按 ID 路由到固定 Cell(就近 + 容灾)
  • Cell 内部低延迟,跨 Cell 只走异步同步
  • 单 Cell 挂了影响 1/N 用户,且能快速切到其他 Cell

为什么:

  • 单机房的物理上限到了(电力 / 散热 / 带宽)
  • 政策 / 合规要求数据本地化
  • 灾难恢复:一个机房挂了不能全网瘫

新瓶颈:

  • 跨 Cell 数据一致性极难(下单时商品库存在另一个 Cell)
  • 部署、变更需要全 Cell 同步
  • 成本数倍上升

阿里 LDC、字节 SCS、Google Spanner 都是这一代的产物。这是 1 亿 DAU 以上才会碰到的世界——你不在那种公司就别学,学了也没场景用。

十、对照表:每一代「拐点」是什么 ​

代升级触发条件抛弃了什么
V0 → V1DB 和 Web 抢资源「全在一台机」
V1 → V2想要高可用「单点 Web」
V2 → V3DB 读 QPS 撑不住「单 DB」
V3 → V4团队 50+ / 业务多模块「单库共享」
V4 → V5单业务表撑不住「单库单表」
V5 → V6团队 200+ / 多业务线「粗粒度服务」
V6 → V7多语言 / 治理升级慢「业务代码内治理」
V7 → V8单机房物理上限「单机房」

看着这张表逐行问「我现在卡在哪一行的瓶颈」——有清晰瓶颈才升级,没有就停在原地。最聪明的架构师是让公司在 V3 / V4 多停几年,因为那一段最划算。

十一、反向警告:三个常见错误升级 ​

11.1 「我们要微服务」(V2 直接跳 V6) ​

20 人团队,DAU 5 万,直接上 K8s + 30 个微服务。结果:60% 时间在搞基础设施,业务一年没进展。应该:V3 + 模块化单体撑 3 年。

11.2 「我们要分库分表」(V3 提前到 V5) ​

数据 1 亿行,单库还能扛,被「分库分表潮流」忽悠分了 32 库。结果:跨库查询全部失效,后台报表写不出来。应该:先归档老数据 / 加索引 / 垂直拆分,顶不住再分。

11.3 「我们要 Service Mesh」(V6 直接跳 V7) ​

Spring Cloud 用得好好的,听了一场布道大会决定上 Istio。结果:延迟涨 2ms,故障率上升,团队没人懂调优。应该:除非真的多语言 + 1000+ 服务,否则不要。

十二、踩坑提醒 ​

  1. 追潮流升级——架构升级要被瓶颈逼,不是看到大厂用就跟
  2. 越级升级——V3 跳 V6 / V7,直接死
  3. 不可逆决策提前做——分库分表、Cell 化都是 2 年内还不完的债
  4. 以为新架构 = 新性能——微服务初期性能反而下降(网络多了一跳)
  5. 忽略组织规模——架构演进本质是「应对组织复杂度」,不只是技术
  6. 没有回滚计划——升级前必须有「打回原版」的应急方案
  7. 看 PPT 不看真实运维——Netflix 的架构图很美,他们 SRE 团队 200 人你有几个?
  8. 以为「单体落后」——Stack Overflow 单体到死,Shopify 单体到 IPO,单体 ≠ 落后
  9. 架构师当英雄——大改架构要全团队认可、分阶段、可灰度,没人能一口气改完
  10. 不算成本——每升一级,运维成本和人力都翻倍,老板不会无限投钱
