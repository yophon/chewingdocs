# 直播系统

直播和视频点播看上去都是"看视频",**底层完全是两个世界**——点播是「文件分发」,直播是「实时流」。点播的视频已经存在,你慢慢拉就行;直播的视频此刻正在产生,**主播说的话 1 秒后必须到观众端**——这 1 秒延迟里要做完编码 / 推流 / 转码 / CDN 分发 / 播放器解码,**任何一步慢一点观众都骂卡**。这一篇讲清楚直播的全链路,以及为什么"超低延迟直播"(电商带货 < 500ms)和"普通直播"(秀场 3-5 秒)是两套完全不同的架构。

> 一句话先记住:**直播 = 推流(RTMP)+ 转码 + 分发(HLS / FLV / WebRTC)+ 播放**。**延迟决定方案**——3-5 秒延迟用 HLS(便宜,CDN 友好);1 秒延迟用 HTTP-FLV(中等成本);< 500ms 延迟用 WebRTC(贵,SFU 自建)。**春晚级直播扛 2 亿同时在线靠的是层层 CDN 缓存** —— 一个流复制到 1 万个 CDN 节点,而不是 2 亿个并发拉源站。

---

## 一、需求拆解

### 1.1 功能需求

- **主播推流**:摄像头 / 屏幕画面实时上传
- **观众拉流**:实时观看直播
- **弹幕**:观众发文字消息,所有人能看到
- **点赞 / 礼物**:实时互动
- **录制回放**:直播结束生成点播视频
- **多档清晰度**:观众选 360p / 720p / 1080p

### 1.2 非功能需求

- **延迟**:普通直播 3-5 秒,电商 / 互动 < 1 秒
- **首帧时间**:< 1 秒
- **稳定性**:卡顿率 < 1%
- **规模**:春晚级 2 亿同时在线;日常单场峰值 1000 万

### 1.3 这一轮不做

- 美颜 / 滤镜(端侧 SDK)
- 实时连麦(WebRTC P2P)
- AI 主播 / 数字人

## 二、容量估算

### 2.1 流量估算

```
2 亿同时在线观众 × 720p 平均 2 Mbps
= 4 × 10^14 bps
= 400 Tbps
→ 单机房根本不可能,**必须 CDN 分发**

带宽成本:400 Tbps × 高峰 1 小时 = 144 PB / 小时
= 几千万人民币 / 小时
```

### 2.2 推流端

```
日活主播 100 万,平均同时直播 5 万人
单主播上行 720p × 2 Mbps = 100 GB / 小时
日总推流量 = 5 万 × 24 × 100 GB ≈ 100 PB / 天
```

### 2.3 弹幕

```
2 亿观众,5% 活跃发弹幕
峰值弹幕 = 2 × 10^8 × 5% × 1 条/秒 = 1000 万 QPS

每条弹幕广播给 2 亿在线 → 投递 2 × 10^15 次/秒(完全不可能)
→ 弹幕必须采样 + 客户端聚合
```

## 三、直播的关键参数:延迟

延迟决定一切:

| 延迟 | 协议 | 适合 | 成本 |
| --- | --- | --- | --- |
| 5-30 秒 | HLS | 普通直播(秀场、新闻) | 极低,CDN 友好 |
| 1-3 秒 | HTTP-FLV | 中等互动(电竞直播) | 低 |
| 500ms-1s | LL-HLS / CMAF | 较高互动 | 中 |
| < 500ms | WebRTC | 强互动(电商带货、连麦) | 高,SFU 自建 |

> **延迟越低成本越高**——降一档延迟通常贵 5-10 倍。**业务定要求,技术选方案**。

## 四、推流:从主播到服务器

### 4.1 RTMP 推流

主流方案:

```
主播端(OBS / 直播 App)
   ↓ 编码 H.264
   ↓ 封装成 RTMP
   ↓ TCP 长连接推送
推流接入服务器(Nginx-RTMP / 自研)
```

**RTMP**:

- Adobe 1996 年发明,Flash 时代主流
- 基于 TCP,长连接
- 延迟 1-3 秒
- **推流端事实标准**(几乎所有摄像头、OBS 都默认 RTMP)

### 4.2 推流接入

```python
# Nginx-RTMP 配置
rtmp {
    server {
        listen 1935;
        application live {
            live on;
            record off;
            # 推流后触发转码
            on_publish http://transcoder/publish;
        }
    }
}
```

主播推流 URL:`rtmp://push.example.com/live/{stream_key}`,`stream_key` 是临时凭证,过期失效。

### 4.3 推流的高可用

**接入层用 Anycast** —— 主播推到「最近的接入点」,而不是固定某个机房。

**主备链路**:

- 推流端默认 4G,备用 WiFi
- SDK 自动切换,断流自动重连
- 推流 URL 带「重连 token」,断后能恢复到原流

## 五、转码:实时多档码率

和点播类似,但**实时性要求高**:

```
源流(主播推上来,1080p / 5 Mbps)
   ↓ 实时转码集群
[
  1080p / 5 Mbps  (原画)
  720p  / 2 Mbps
  480p  / 1 Mbps
  360p  / 500 Kbps
]
```

**关键约束**:

- 每档延迟 < 1 秒(否则首帧延迟太大)
- 必须 GPU(CPU 实时转码扛不住几十路并行)
- **转码失败要 fallback**(只输出原画,不阻塞分发)

## 六、分发:从源站到观众

### 6.1 HLS 分发(主流)

```
转码后输出 HLS:
  live.m3u8(滑动窗口 manifest)
  segment_001.ts
  segment_002.ts
  ...
  
HLS 标准:每段 2-10 秒
  → 延迟 = 段长 × 3(buffer 3 段)= 6-30 秒
```

**优势**:走 HTTP,**完美 CDN 化**。
**劣势**:延迟高。

### 6.2 HTTP-FLV(中延迟)

类似 HLS 但**用 HTTP 长连接持续推送 FLV 数据**:

```
GET /live/stream_key.flv HTTP/1.1
Connection: keep-alive
→ 服务器持续返回 FLV 数据流
```

**延迟 1-3 秒**——比 HLS 短,因为不切段、立即转发。

**问题**:**长连接对 CDN 不友好**——CDN 节点要保持连接,资源消耗大。**只有专业直播 CDN 才支持**。

### 6.3 LL-HLS(Low-Latency HLS)

苹果 2019 年提出,把 HLS 延迟从 30 秒降到 1-2 秒:

- 段长缩短到 200ms
- 用 HTTP/2 推送
- 部分块(partial segment)技术

**还在普及中**,新设备支持好,老设备兼容性差。

### 6.4 WebRTC(超低延迟)

**电商带货、互动直播专用**:

```
推流端 → SFU(Selective Forwarding Unit)
        ↓
      转发给所有观众端
延迟 < 500ms
```

**优势**:延迟极低。
**劣势**:

- WebRTC 是 P2P 设计,改成 SFU 模式工程量大
- 不能用 HTTP CDN(协议不兼容)
- 必须自建 SFU 集群,成本高

> **2026 年的现状**:头部电商(淘宝、抖音电商)已经全面 WebRTC,**CDN 厂商提供 SFU 服务**(阿里云直播 RTC、腾讯云 TRTC)。

## 七、CDN 分发:春晚级流量怎么扛

```
直播流到源站
    ↓
推到上层 CDN(20 个核心节点)
    ↓
推到中层 CDN(200 个区域节点)
    ↓
推到边缘 CDN(几千个边缘节点,靠近用户)
    ↓
2 亿观众就近拉边缘节点
```

**关键:CDN 是树形分发,不是全部回源**。

```
不分发:2 亿用户 × 2 Mbps = 400 Tbps 都打源站(完全不可能)
CDN 分发:
  源站 → 20 个核心节点(20 个流复制)
  核心 → 200 个区域(放大 10 倍)
  区域 → 几千个边缘(放大 25 倍)
  边缘 → 几千万用户(放大 1 万倍)
  
源站只发了 20 路,2 亿用户都看到了
```

**这就是 CDN 的"放大效应"**——**直播是 CDN 最适合的场景**(同一份数据被无限多人需要)。

## 八、弹幕系统

弹幕的工程难度**不亚于直播本身**——10 万 / 100 万人在线时,1 秒钟有几万条弹幕,扩散给所有人。

### 8.1 朴素方案(行不通)

```
观众发弹幕 → 推给所有在线观众
1 万人发,2 亿人收 → 1 万 × 2 亿 = 2 万亿次推送/秒
→ 直接挂
```

### 8.2 实战方案:采样 + 聚合 + 分级

#### 采样

不是每条弹幕都广播——**热门弹幕广播,冷门弹幕只发给附近的人**:

```
单房间 100 万弹幕/分钟
采样到 1 万条/分钟广播给所有人
其余只在小范围(同一设备类型 / 同一地区)显示
```

用户不会知道——反正屏幕也放不下那么多。

#### 聚合

类似弹幕合并:

```
"哈哈哈哈哈" "哈哈哈" "哈哈" → 合并显示一次
"666" 大量出现 → 只显示一条 + 计数
```

#### 分级

VIP 弹幕、彩色弹幕、礼物弹幕优先广播,普通弹幕降级。

### 8.3 弹幕的存储

**写入**:Kafka(削峰)
**实时分发**:Redis Pub/Sub 或自研推送
**回放**:存到对象存储,直播结束后做录播弹幕

---

## 九、礼物 / 互动

电商带货 / 秀场的核心收入来源:

### 9.1 礼物的特殊要求

- **强一致**:用户花钱买的礼物,**必须送到主播账户**
- **实时显示**:观众和主播都要立即看到
- **防作弊**:防止重放攻击、客户端伪造

### 9.2 流程

```
观众点礼物
   ↓
扣钱(同步,DB 事务)
   ↓
异步推送礼物消息(类似弹幕,但优先级更高)
   ↓
主播看到 + 所有观众看到
```

**扣钱必须同步,推送可以异步**。

## 十、录制 / 回放

直播结束 → 自动生成点播视频:

```
推流接入时同步保存原始流到对象存储
直播结束 → 转码 + 切片 → 生成回放视频
回放接入点播系统(详见 19 篇)
```

**关键**:**录制不能影响直播主路径**——独立 Worker,失败也不影响直播。

---

## 十一、跨地域分发:全球直播

世界杯、海外直播:

```
源站(主播所在地区)
   ↓
跨洋推送到各大区源站(中转 CDN)
   ↓
各区域 CDN 分发本地观众
```

**跨洋推送只一次**——不是每个观众都跨洋拉。

**延迟代价**:跨洋光纤本身 200ms,**跨地域直播必然延迟更高**。

---

## 十二、关键工程挑战

### 12.1 主播突然断流

```
4G 信号差 → 推流断
   ↓
观众端继续看 5 秒缓存
   ↓
还没恢复 → 显示"主播暂时离开"
   ↓
主播恢复 → 重新推流(同 stream_key)
   ↓
观众无感切回
```

### 12.2 观众千万人同时进直播间

春晚 / 大型活动:

- **预热 CDN**:开播前推流到所有边缘节点
- **限流**:超过容量返回排队 / 兜底
- **降级**:网络拥堵时强制低码率

### 12.3 鉴权和反盗播

- **拉流 URL 签名**(同点播)
- **设备指纹检测**(同 IP 同时拉 100 路 → 怀疑是盗播服务器)
- **DRM**(付费内容)

## 十三、最终架构图

```
                       主播端
                         ↓ RTMP
                   就近接入 (Anycast)
                         ↓
                   源站 RTMP 接入
                         ↓
              ┌──────────┼──────────┐
              ↓          ↓          ↓
         实时转码      录制       推到 CDN 上层
         (GPU)       (异步)
              ↓                      ↓
         多档码率             ┌─────────────┐
              ↓               │ CDN 树形分发 │
         上层 CDN(20)──────→│  上层→中层    │
              ↓               │  中层→边缘    │
         中层 CDN(200)       │  边缘→用户    │
              ↓               └──────┬──────┘
         边缘 CDN(几千)            ↓
                                  观众端
                                  (HLS/FLV/WebRTC)

弹幕通道:
  观众 → 弹幕服务 → Kafka → 分发服务 → 推回所有观众
```

## 十四、踩坑提醒

1. **直播用 HTTP 短连接拉**——延迟和卡顿都崩
2. **不分级延迟方案**——用 WebRTC 做秀场,成本爆炸
3. **HLS 段长设太长**(10 秒)——延迟 30 秒+
4. **不用 CDN 树形分发**——源站直接被 2 亿用户打挂
5. **弹幕全量广播**——10 万 × 2 亿次推送,集群崩
6. **转码用 CPU**——GPU 才能扛实时多档转码
7. **不监控推流断**——主播断了几分钟才发现
8. **首帧不优化**——观众点开等 5 秒,流失严重
9. **CDN 不预热大型活动**——春晚开播瞬间所有节点回源
10. **录制和直播共用资源**——录制慢拖累直播
11. **没准备超低码率档**(180p)——网络极差用户完全看不了
12. **不限流容量**——超过设计容量直播全挂
