# 网约车

网约车(滴滴 / Uber)是**位置实时性最强的系统**——百万司机每 3 秒上报一次坐标,千万用户随时叫车,**一次派单要在 1 秒内从几千司机里选最优**。**地理位置 + 实时性 + 强一致**(订单不能重复扣钱)三者叠加,是分布式系统里的"立体作业"。这一篇按五段式拆解,重点讲两个核心技术:**地理空间索引(GeoHash)** 和**派单算法**。

> 一句话先记住:**网约车 = 司机位置实时上报 + 空间索引快速搜附近 + 派单算法 1 秒内匹配 + 订单状态机管全流程**。**地理空间检索的 GeoHash 把二维问题转化为一维字符串前缀**——让 Redis 这种 KV 数据库也能干空间查询。**派单是博弈问题**——选最近不一定最优,要平衡乘客等待时间、司机里程、平台总效率。

---

## 一、需求拆解

### 1.1 功能需求

- **司机端**:实时上报位置、接受派单、完成订单
- **乘客端**:叫车(选起点 / 终点)、看附近司机、看车辆位置、支付
- **派单**:乘客叫车后,系统选合适司机派给他
- **订单管理**:发单 → 接单 → 上车 → 行驶 → 完成 → 支付

### 1.2 非功能需求

- **延迟**:派单 < 3 秒(超过用户取消)
- **位置精度**:< 50m
- **可用性**:99.99%
- **规模**:DAU 1 亿,司机 1000 万,日订单 1 亿,峰值订单 10 万 QPS

### 1.3 这一轮不做

- 路径规划 / 导航(用第三方地图 SDK)
- 计费(独立子系统,涉及拥堵、路况、动态价)
- 司机评分 / 反作弊

## 二、容量估算

### 2.1 位置上报

```
1000 万司机 × 每 3 秒上报一次 = 333 万 QPS
峰值 ~500 万 QPS

每条上报 ~100 字节 → 50 MB / s 写入
```

**这个 QPS 是网约车系统最重的写**——必须 KV 存储 + 内存数据库。

### 2.2 派单

```
日订单 1 亿 → 平均派单 QPS 1000
峰值早高峰 10 万 QPS

每次派单要"搜附近 1km 内司机" → 平均 50 个候选
→ 50 × 10 万 = 500 万次司机查询/秒
```

### 2.3 存储

```
订单:1 亿/天 × 1 KB = 100 GB / 天
     5 年 = 180 TB → 必须分库

位置历史(轨迹):
  司机 × 每 3 秒一个点 × 8 小时工作 / 天
  = 1000 万 × 9600 点/天 = 1000 亿点/天
  每点 50 字节 = 5 TB / 天
  → 必须时序数据库(InfluxDB / TDengine)或对象存储归档
```

## 三、地理空间索引:GeoHash

**核心问题**:乘客在 (lat, lng),怎么快速找出 1km 内的所有司机?

### 3.1 朴素方案(行不通)

```sql
SELECT * FROM driver
WHERE ABS(lat - ?) < 0.01 AND ABS(lng - ?) < 0.01;
```

**问题**:1000 万司机全表扫,完全不行。

### 3.2 GeoHash 原理

把地球表面切成网格,每个网格用一个字符串编码:

```
地球 →  二分 → 每次切一刀生成 0/1
   先切经度:0(西半球) / 1(东半球)
   再切纬度:0(南半球) / 1(北半球)
   交错来,产生一长串 01 → 转 base32
   
精度:
  6 位 GeoHash → 1.2 km 网格
  7 位 → 153 m 网格
  8 位 → 38 m 网格
```

**关键性质**:**前缀越长,位置越接近**。

```
"wx4g0ec1" 和 "wx4g0ec2" → 前 7 位相同 → 距离 < 153m
"wx4g0ec1" 和 "wx4g8c2"  → 前 4 位相同 → 距离 < 5km
```

### 3.3 用 Redis GEO 实现

Redis 内置 GeoHash:

```bash
# 司机上报位置
GEOADD drivers 116.404 39.915 "driver_123"

# 查附近 1km 司机(按距离排序,最多 100 个)
GEORADIUS drivers 116.404 39.915 1 km WITHCOORD WITHDIST COUNT 100 ASC
```

**单 Redis GEO 实例**:百万级数据,**几毫秒返回**。

### 3.4 大规模:GeoHash 分片

1000 万司机 → 单 Redis 不够 → **按 GeoHash 前缀分片**:

```
GeoHash 前 4 位作为 shard key
  → 全国划成 32^4 ≈ 100 万个网格
  → 把网格分配到 100 台 Redis 节点
  → 每台 10 万司机
```

**查询时**:乘客的 GeoHash 找到所属 shard → 查这个 shard + 周围 8 个 shard(防止边界遗漏)。

## 四、数据模型

### 4.1 司机表

```sql
CREATE TABLE driver (
    id           BIGINT PRIMARY KEY,
    user_id      BIGINT,
    car_plate    VARCHAR(20),
    car_model    VARCHAR(50),
    status       TINYINT,    -- 0 离线 1 空闲 2 接单中 3 服务中
    rating       DECIMAL(3,2)
);
-- 不分库(司机数 1000 万,单表勉强够)
```

### 4.2 司机位置(Redis)

```
key: drivers (GEO type)
member: driver_id
location: (lat, lng)

每 3 秒覆盖更新
```

### 4.3 订单(分库)

```sql
CREATE TABLE order (
    id            BIGINT PRIMARY KEY,
    passenger_id  BIGINT,
    driver_id     BIGINT,
    start_lat     DECIMAL,
    start_lng     DECIMAL,
    end_lat       DECIMAL,
    end_lng       DECIMAL,
    status        TINYINT,    -- 0 待派单 1 已派单 2 司机已到 3 行驶中 4 完成 5 取消
    price         DECIMAL,
    created_at    TIMESTAMP,
    completed_at  TIMESTAMP
);
-- 分库:按 passenger_id(查"我的订单"最频繁)
```

### 4.4 订单状态机

```
待派单 → 已派单 → 司机已到 → 行驶中 → 完成 → 支付
   ↓        ↓         ↓
  取消    取消      取消
```

每个状态变更都是关键事务,**必须保证幂等**。

## 六、核心算法 1:派单匹配

派单是**带时间约束的二部图匹配**——把订单匹配到司机。

### 6.1 朴素算法:最近优先

```python
def dispatch(order):
    nearby_drivers = redis.georadius(order.start, 3, "km", count=20)
    nearest = min(nearby_drivers, key=lambda d: distance(d.location, order.start))
    return nearest
```

**问题**:

- **司机被频繁竞争**:同一时间多个订单都选最近的司机 → 该司机收 5 个派单,但只能接一个
- **远的乘客永远等不到车**:近的乘客不断"抢"司机

### 6.2 实战算法:批量匹配 + 全局优化

```
1. 攒一批订单(每 2 秒处理一次)
2. 找出所有候选(订单 × 司机)的距离矩阵
3. 用匈牙利算法 / 拍卖算法做全局最优匹配
   → 总等待时间最小
   → 司机里程最少
```

**Uber 的做法**:每 2 秒批量优化,**全局最优而不是单次最近**。

### 6.3 派单的多目标

```
分数 = a × 距离 + b × 司机评分 + c × 预计时长 + d × 司机疲劳度 + ...

各因子权重通过强化学习训练
```

### 6.4 派单失败重试

```
派给司机 A:
  10 秒内 A 没响应 → 取消,派给 B
  A 拒绝 → 立即派给 B
  
连续 3 个司机拒绝 → 提示乘客"加价 / 等待"
```

## 七、核心算法 2:位置上报优化

司机端每 3 秒上报一次,**怎么不打挂服务器**?

### 7.1 上报通道:HTTP 还是长连接?

**长连接(WebSocket / 自研 TCP)**:

- 节省握手开销
- 适合高频
- 需要 IM Gateway 类似架构(详见 18 篇)

**短轮询 HTTP**:

- 简单
- 适合低频(> 10 秒)

> 网约车通常**长连接 + 批量上报**——10 秒内的位置攒成一批一起发。

### 7.2 服务端写入优化

```
位置网关收到上报 → 直接写 Redis(覆盖更新,不积累)
不进 MySQL(写不过来,也没必要)
轨迹历史 → 异步 Kafka → 时序 DB
```

**关键:位置只存"最新一份"**,历史轨迹离线归档。

### 7.3 离线司机自动剔除

司机超过 30 秒没上报 → 标记离线:

```python
# Redis 用 ZSet 存 (driver_id, last_seen_ts)
# 定时清理:删除 last_seen < now - 30s 的司机
```

## 八、订单状态机的可靠性

订单是**强一致系统**——不能漏单、不能重复扣钱。

### 8.1 状态变更必须事务

```python
def driver_arrived(order_id, driver_id):
    with db.transaction:
        order = db.select_for_update("SELECT * FROM order WHERE id = ?", order_id)
        if order.status != "已派单" or order.driver_id != driver_id:
            raise IllegalState
        db.execute("UPDATE order SET status='司机已到', updated_at=NOW() WHERE id=?", order_id)
```

### 8.2 防止"重复接单"

司机网络抖动,**接单按钮按了 2 次** → 不能创建 2 个订单或扣 2 次钱。

**幂等**:

```python
def take_order(order_id, driver_id, request_id):
    # request_id 是客户端生成的唯一 ID
    if redis.exists(f"taken:{request_id}"):
        return cached_result(request_id)  # 返回上次结果
    
    redis.setex(f"taken:{request_id}", 60, "1")
    # 真正业务...
```

### 8.3 离线司机的订单恢复

司机刚接单 → 网络断 → 服务端不知道司机能不能完成单 → **订单超时(5 分钟)→ 自动重派**。

## 九、实时定位:乘客看车开过来

乘客端持续刷新司机位置(车辆图标移动):

```
乘客端每 3 秒 GET /order/{id}/driver_location
  → 服务端从 Redis 拿司机最新位置
  → 返回给乘客端

(也可以走长连接,服务端主动推)
```

**优化**:

- **不直接显示原始坐标**(抖动严重)→ 客户端做平滑插值
- **预测下一秒位置**(根据当前速度方向)→ 视觉更流畅

## 十、热点城市的容量

```
全国 1000 万司机平均分布,但实际:
  北京 / 上海 / 深圳:每城 50 万 + 司机
  小城市:几千

派单 QPS 也极不均匀:
  北京早高峰单独可能 5 万 QPS
  小城市可能 100 QPS
```

**设计**:

- 按城市划分服务实例(同城内部独立)
- 大城市单独多副本
- **不强求"全国一套配置"**——分级服务

## 十一、动态定价

需求高 / 供给少时涨价(雨天、节日)——**这是出行平台核心收入手段**。

### 11.1 算法

```
区域价格倍数 = f(待派单数, 空闲司机数, 时间, 天气)

倍数 1.2x ~ 3.0x 不等
```

**实时计算**:每 30 秒重算一次每个区域的倍数,推送到客户端。

### 11.2 工程挑战

- 区域如何划分(GeoHash 网格 / 自定义多边形)
- 价格波动的**用户体验**(用户开始下单后不能涨价)
- 防止"刷价"(司机集体下线制造缺司机假象)

## 十三、踩坑提醒

1. **位置存 MySQL**——写不动,必须 Redis
2. **不用 GeoHash**——空间查询慢
3. **不分片 Redis GEO**——单实例 1000 万司机扛不住
4. **派单选最近一个**——多订单抢同司机,远的乘客等不到
5. **位置上报全保存**——存 5 TB / 天,炸
6. **订单状态变更不加锁**——重复接单 / 重复扣钱
7. **不做幂等**——网络重试导致重复订单
8. **全国一套配置**——大城市挤爆小城市闲死
9. **司机离线不剔除**——派单派给离线司机白等
10. **没动态价**——供需失衡时车永远叫不到
11. **轨迹只存 MySQL**——查司机轨迹时卡死
12. **派单算法没监控**——成功率下降不知道,GMV 直接掉
