# 高可用三件套

「我们系统 99.99% 可用」是面试和 PR 稿里最爱出现的一句话,但**几乎没人能解释清楚这个数字怎么来的**。99.99% 听上去很高,**实际意味着一年只允许累计挂 52 分钟**——任何一个生产事故都可能一次性烧光一年的「故障预算」。这一篇讲清楚两件事:**这个数字怎么算出来**,以及**怎样的工程努力才能真正达成**——本质就是三个基础动作:冗余、故障转移、健康检查。

> 一句话先记住:**高可用不是堆设备,是一套"假设单点必挂"的工程纪律**——任何节点都可能挂,所以任何节点都要有副本(冗余);挂了要能自动切走流量(故障转移);切走前提是必须知道它挂了(健康检查)。**这三件事缺一个,可用性数字都是骗人的**。

---

## 一、可用性的数字感

「9 的个数」是行业通用语言,**必须背熟**:

| 可用性 | 一年允许停机 | 一月允许停机 | 一周允许停机 | 适合场景 |
| --- | --- | --- | --- | --- |
| 90%(1 个 9) | 36.5 天 | 72 小时 | 16.8 小时 | 内部工具 |
| 99%(2 个 9) | 3.65 天 | 7.2 小时 | 1.68 小时 | 普通后台 |
| 99.9%(3 个 9) | 8.77 小时 | 43.8 分钟 | 10.1 分钟 | 中型业务 |
| 99.99%(4 个 9) | 52.6 分钟 | 4.4 分钟 | 1.01 分钟 | 主流互联网 |
| 99.999%(5 个 9) | 5.26 分钟 | 26 秒 | 6 秒 | 电信 / 银行 |
| 99.9999%(6 个 9) | 31.5 秒 | 2.6 秒 | 0.6 秒 | 极致核心 |

> **「我们要 5 个 9」 是一句很贵的话**——一年只能挂 5 分钟,这要求秒级故障转移 + 多机房 + 全自动恢复。**大多数业务做到 99.95% 已经很优秀了**——别一上来吹 5 个 9。

### 1.1 可用性的串联与并联

系统由多个组件组成,可用性怎么算?

**串联**(任一挂全挂):

```
A 99.9% → B 99.9% → C 99.9% = 99.9% × 99.9% × 99.9% = 99.7%
```

**并联**(冗余,任一活就活):

```
A 主 99.9% + A 备 99.9%(并联)= 1 - (0.1% × 0.1%) = 99.9999%
```

> **冗余的威力**:两个 99.9% 的副本并联得到 99.9999%——多一个副本,多 3 个 9。**这就是高可用的基本武器**。
> **串联的悲剧**:链路上每多一个组件,可用性都打折扣。**所以减少依赖比加副本更高级**。

### 1.2 可用性 vs 可靠性 vs 持久性

经常被混用,实际不同:

| 术语 | 含义 |
| --- | --- |
| 可用性(Availability) | 用户请求时,系统能响应的概率 |
| 可靠性(Reliability) | 系统在一段时间内不出故障的概率 |
| 持久性(Durability) | 已写入的数据不丢失的概率(S3 的 11 个 9 就是这个) |

> 别把「数据不丢」(持久性)和「服务能响应」(可用性)混了。**S3 持久性 99.999999999%(11 个 9),可用性「只有」99.99%**——意思是数据基本不会丢,但服务偶尔会超时。

## 二、第一件套:冗余(Redundancy)

冗余 = 「这个组件挂了我还有一个」。**没有冗余的系统不是高可用系统,只是单机系统外面包了层壳**。

### 2.1 冗余的几种粒度

```
进程冗余    一台机器多个进程(占用 CPU)
机器冗余    一个集群多台机器(应用层最常用)
机架冗余    跨机架部署(防一个机架供电故障)
机房冗余    跨可用区(AZ)部署(防机房整体故障)
地域冗余    跨地域部署(防地震、政策、海底光缆断)
```

**经验值**:

- 互联网应用的标准是**跨 AZ**(同地域多机房)
- 金融 / 电商核心是**跨地域**(异地容灾)
- 全球业务(社交、电商)是**全球分布**

### 2.2 冗余的副本数:N+1 不够,N+2 才稳

业务峰值需要 N 台机器扛,**冗余配多少**?

- **N+1**:能扛一台挂的故障,但**滚动升级时另一台再挂就崩**
- **N+2**(推荐):允许同时一台挂 + 一台维护
- **2N**:多机房双活,任一机房全挂还能扛

> 新手最常见错误是**按峰值刚好配机器**,那叫「容量规划」,不是「高可用」。**真正的高可用是峰值容量 × 2**——剩下一半在等故障。

### 2.3 冗余不只是机器,是整链路

```
DNS 多解析记录 → 多个入口 IP
LB 多实例    → keepalived / VIP 漂移
Web 多副本   → 集群 + LB
DB 主从     → 自动主从切换
缓存 集群   → Redis Cluster / Sentinel
MQ 多副本   → Kafka 副本因子 ≥ 3
配置中心 集群 → Nacos / Etcd 多节点
```

**任何一处单点 = 整体可用性瓶颈**。审视架构图时,**用红笔圈出所有"只有一个"的组件**——那就是你的故障点。

## 三、第二件套:故障转移(Failover)

冗余只是「准备好了备份」,**故障发生时能不能自动切走流量**才是关键。

### 3.1 故障转移的两种模式

**主备(Active-Standby)**:

```
平时只有主在工作,备在等
主挂 → 备升主,接管流量
切换时间 = 检测时间 + 切换时间(几秒~几分钟)
```

适合:有状态服务(DB、ZK、Redis Master),只能一个写。

**双活(Active-Active)**:

```
两个都在工作,都接受流量
一个挂,流量自动切到另一个
切换时间 ≈ 0(LB 自动剔除)
```

适合:无状态服务(Web、API)。

> **无状态服务必须双活**(成本和切换时间都更优);**有状态服务大多只能主备**(数据一致性约束)。

### 3.2 几种典型故障转移方案

| 组件 | 方案 |
| --- | --- |
| 域名层 | DNS 多解析 + TTL 短(60s),挂了改 DNS 切流量 |
| 入口层 | LB 双机 + VIP 漂移(keepalived) |
| 应用层 | LB 健康检查自动剔除挂的实例 |
| MySQL | MHA / Orchestrator / Patroni 自动选主 |
| Redis | Sentinel 哨兵 / Redis Cluster gossip |
| Kafka | Controller 自动重平衡分区 leader |
| ZooKeeper / etcd | Raft 自动选主 |

### 3.3 故障转移的「脑裂」陷阱

最危险的故障模式:**网络分区导致两个节点都以为自己是主**——双主同时写,数据冲突,事后无法合并。

**防脑裂的常用招**:

- **多数派(Quorum)**:写需要 majority 确认,少数派分区方拒写
- **Fencing Token**:每次选主递增编号,旧主试图写时被拒
- **STONITH**(Shoot The Other Node In The Head):新主上任前强行断电旧主

> 单纯的「主备 + 心跳」**几乎一定会脑裂**——网络抖动时两边都看不见对方。**所以现代分布式系统几乎都用 Raft / Paxos** 这种基于多数派的算法。

### 3.4 故障转移的反向风险:误切换

**实际事故里,1/3 是误切换造成的**——主好好的,因为网络抖动被判定挂了,触发切换,反而中断服务。

**对策**:

- 健康检查要**抗抖动**(N 次失败才算挂)
- 切换前**人工确认**(关键系统)
- 切换有**冷却期**(刚切完不能立刻再切,防震荡)

## 四、第三件套:健康检查(Health Check)

故障转移的前提是**能准确判断"它挂了"**。健康检查做不好,要么真挂了你不知道,要么没挂被你切了。

### 4.1 健康检查的三层

```
1. 进程级:进程还在吗?     端口能不能 telnet 通
2. 接口级:HTTP 200 吗?   /health 接口,K8s readinessProbe
3. 业务级:业务真的能跑吗?  比如查一个测试用户能不能查到
```

**只做进程级是不够的**——一个进程可能在但内部死锁,接口已经响应不了。

### 4.2 健康检查的间隔与判定

| 参数 | 推荐值 | 说明 |
| --- | --- | --- |
| 间隔 | 1-5 秒 | 越短发现越快,但开销越大 |
| 超时 | 1-3 秒 | 比间隔稍小 |
| 失败次数判定挂 | 3 次连续 | 1 次就判定容易误判,5 次太慢 |
| 成功次数判定活 | 2-3 次 | 防止"刚好缓过来又挂"的震荡 |

### 4.3 健康检查的反模式

**反模式 1:健康检查接口依赖 DB**

```python
@app.get("/health")
def health():
    db.ping()  # ❌ DB 抖一下健康检查就失败,大量实例被剔除
    return "OK"
```

**对策**:`/health` 只判进程内部状态;另开 `/health/deep` 检查依赖,人工或监控用。

**反模式 2:健康检查太重**

每秒调用复杂业务逻辑作为健康检查,**心跳本身把系统打挂**。

**反模式 3:不区分 readiness / liveness**

- **liveness**:进程是否要重启(内部死锁了)
- **readiness**:能不能接流量(还在初始化,流量不要进)

K8s 把这两个分开,新手经常合用一个,**结果初始化慢的服务一启动就被剔除**。

### 4.4 主动 vs 被动健康检查

- **主动**:LB / 监控定时探活——实时性高,但增加无效请求
- **被动**:观察真实请求的失败率——不增加请求,但发现慢

**生产上两者结合**:Nginx 和 Envoy 都支持 `passive health check`,真实请求 N 次失败自动剔除一段时间。

## 五、可用性 = 1 − MTTR / (MTTF + MTTR)

工程视角下可用性的本质公式:

```
MTTF = Mean Time To Failure  (平均无故障时间)
MTTR = Mean Time To Recovery (平均恢复时间)

可用性 = MTTF / (MTTF + MTTR)
```

提高可用性两条路:

1. **延长 MTTF**——更可靠的硬件、更稳定的代码、压测预防
2. **缩短 MTTR**——监控更快报警、自动化恢复、预案演练

> **现代高可用工程的重点是 MTTR,不是 MTTF**——故障无法消灭,但能从「发现 30 分钟 + 处理 1 小时」缩短到「自动切换 30 秒」。**「故障管理」比「故障预防」回报更高**。

### 5.1 缩短 MTTR 的工程动作

```
监控  → 30 秒内发现异常
报警  → 1 分钟内通知到人(或自动触发预案)
诊断  → 链路追踪 + 日志聚合,5 分钟定位
恢复  → 自动:故障转移 / 限流 / 降级
       手动:重启 / 回滚 / 紧急扩容
复盘  → 不只是写文档,改进监控 + 演练
```

## 六、混沌工程:主动制造故障

**没演练过的预案 = 没有预案**。Netflix 发明的混沌工程(Chaos Engineering)思想:

- **Chaos Monkey**:随机杀生产环境的实例
- **Chaos Kong**:随机让一个 AZ 下线
- **Latency Monkey**:随机给请求加延迟
- **Conformity Monkey**:随机找出违反最佳实践的实例

> **不主动制造故障,故障会用最糟糕的时机找你**——双 11 当天、半夜两点、节假日。混沌工程的本质是**把不可避免的故障"提前到"工作时间**,以便修复。

### 6.1 落地建议

不一定要完整搬 Netflix 那套,小公司可以从这些做起:

- **强制每月演练一次主从切换**——别等到主真挂了发现切不动
- **机房断网演练**——拔一个 AZ 的网线,看流量是否自动切走
- **依赖服务超时演练**——人为给某个下游加 5s 延迟,看上游是不是被拖死

## 七、SLA / SLO / SLI:可用性的「契约语言」

| 术语 | 含义 |
| --- | --- |
| **SLI**(Indicator) | 你测量什么(请求成功率、p99 延迟) |
| **SLO**(Objective) | 你内部目标(99.95% 成功率) |
| **SLA**(Agreement) | 你对客户的承诺(99.9% 不达成赔钱) |

```
SLI ⊆ SLO ⊆ SLA
内部目标永远比对外承诺严
```

**误差预算(Error Budget)**:

```
若 SLO = 99.9%,则一个月允许故障 43.8 分钟
这就是「故障预算」——超出后必须停止上新功能,先修稳定性
```

> 误差预算是**业务和稳定性之间的契约**——平时让产品团队自由迭代,预算用光了大家一起停下来还债。**没有预算的稳定性管理,是吵架,不是工程**。

## 八、工程清单:让一个系统达到 99.95%

按这个清单一项一项打钩:

```
□ 应用无状态,可水平扩展
□ 至少跨 2 个 AZ 部署,N+2 副本
□ DB 主从复制,有自动主从切换(MHA / Orchestrator)
□ Redis 哨兵 / Cluster
□ MQ 副本因子 ≥ 3,acks=all
□ 配置中心 / 注册中心多节点
□ 入口层有限流和熔断
□ 健康检查覆盖三层(进程 / 接口 / 业务)
□ 监控 + 报警:CPU / 内存 / 错误率 / p99 延迟
□ 链路追踪 + 集中日志
□ 灰度发布 / 金丝雀 / 自动回滚
□ 关键预案文档化 + 季度演练
□ SLO 定义 + 误差预算管理
```

> 打满后不是 100% 可用,是**有资格谈 99.95%**——剩下的取决于运营团队的反应速度和混沌演练。

## 九、踩坑提醒

1. **吹「五个 9」**——五个 9 = 一年挂 5 分钟,99% 的业务做不到也不需要
2. **副本只配 N+1**——单点维护或挂时还能扛,但不能扛同时两个事件
3. **冗余只到机器**——机架 / 机房 / 地域级冗余不做,大故障一来全瘫
4. **故障检测太敏感**——网络抖一下就切主,引发震荡
5. **健康检查依赖整条链路**——DB 抖一下所有 Web 实例被剔除
6. **不演练就以为切得过去**——真正切的时候发现配置过期 / 权限缺失
7. **只防"机器挂"不防"软件 bug"**——80% 的故障是发版引起,不是硬件
8. **SLA / SLO 写文档不执行**——超预算照常发版,等于没定
9. **可用性数字只算"全挂"**——慢请求、错误响应也是不可用,**SLI 要含错误率和延迟**
10. **以为 K8s 自带高可用**——K8s 给你工具,但应用配错(不写 PDB / 没多 AZ / 不写 readinessProbe)依然会单点

01-05 这五篇是**系统设计的"心智地基"**:

- 01 教你怎么从需求开始想
