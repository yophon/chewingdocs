# 数据库分库分表

分库分表是**单机 DB 撑不住后唯一的出路**,但也是**互联网架构里最不可逆的决策**——一旦分了,后面 3 年都在还分库的债。新手最常见的错是「听说要分库」就提前分,**没榨干单库就分,等于花 100 万去解决 1 万的问题**。这一篇讲三件事:**什么时候才该分**(三条硬线)、**怎么分**(垂直 vs 水平 / 分片键怎么选)、**分了之后最痛的三个问题怎么办**(跨库查、分布式 ID、在线扩容)。

> 一句话先记住:**分库分表是"换问题"——把"单机扛不住"换成"应用层复杂度爆炸"**。一旦分了,跨库 join 没了、全局排序难了、事务难了、迁移地狱来了。**所有不分库就能撑过去的优化都比分库优先**——加索引、归档老数据、读写分离、缓存,**这些招用尽再分库**。**分片键选错 = 整个架构推翻重来**,这是分库分表唯一不能犯的错。

---

## 一、什么时候必须分?三条硬线

不要拍脑袋,**用数据说话**。任何一条触线,就该认真考虑分库:

### 1.1 数据量线:单表 > 5000 万行 / 单库 > 500GB

MySQL 单表理论无上限,**实际上 5000 万行是普遍公认的"开始变慢"门槛**:

- B+ 树深度变成 4-5 层,IO 次数增加
- 二级索引扫描慢
- DDL(改表结构)动辄几小时

> **TiDB / PolarDB 等分布式 DB 把这条线推到 100 亿+,但它们底层仍然是分片**——你只是不用自己分而已。

### 1.2 QPS 线:写 > 5000 / 总 > 50000

单 MySQL 实例(8 核 32G + SSD)的极限:

- 读:10000-20000 QPS(走索引)
- 写:3000-8000 QPS(取决于事务复杂度)

**到这条线时,先用读写分离 + 缓存挡掉读**,只剩写撑不住才考虑分。

### 1.3 单机硬件线:32C + 256GB + 4TB SSD 撑不住

这是云上单实例规格上限附近。**到这一步加机器只能水平加**,不能再升单机了。

> **三条线满足任一,值得考虑;满足两个,基本必分;三个都满足,**今天就得分**。

### 1.4 反向警告:三种"假必须分"的场景

#### 场景 A:数据量 1 亿但 99% 是冷数据

把 5 年前的订单归档到冷库或对象存储,热库只剩 1000 万 → **不用分**。

#### 场景 B:QPS 高但 95% 是缓存能挡的

加 Redis 把读 QPS 降 95% → DB 实际负载小 → **不用分**。

#### 场景 C:单表 5000 万但只是写日志

日志表别用 OLTP DB,直接进 Kafka → ClickHouse / ES → **不用分,换技术栈**。

## 二、垂直拆分:按业务模块拆库

最简单的一种拆分,**很多人甚至不知道这叫"分库"**:

```
原来:
  一个 mysql 库,里面有 user / order / product / comment 等所有表

垂直拆:
  user_db    ← user 表
  order_db   ← order 表、order_item 表
  product_db ← product 表、sku 表
  comment_db ← comment 表
```

**适合**:数据库还有空间,但**业务模块多 + 团队拆分**——每个团队一个库,互不打扰。

**优势**:

- 各模块独立扩展(订单写量大就单独升级 order_db 配置)
- 故障隔离(comment_db 挂不影响下单)
- 团队自治

**代价**:

- 跨模块查询要走应用层 join(不能 SQL join 了)
- 跨模块事务要走分布式事务

> **垂直拆是"先于"水平拆的步骤**——通常先按业务垂直拆出 5-10 个库,再对单个超大业务(如 order)做水平拆。

## 三、水平拆分:同一张表切片(分库 + 分表)

**真正的"分库分表"**——同一张表数据量太大,**按某个键切成 N 份**。

### 3.1 分库 / 分表 / 分库分表的区别

```
原表 order(10 亿行) 在 db0 单实例里

只分表:db0 里建 order_0 ~ order_15(16 张表)
  → 单库 IOPS 仍是瓶颈

只分库:db0、db1、db2 三个库,每个库一张完整 order 表
  → 数据量没解决

分库分表:db0 ~ db15(16 个库),每个库 16 张表
  → 总共 16 × 16 = 256 张物理表
```

> **生产场景几乎都是"分库分表"**——只分表解决不了 IO 瓶颈,只分库解决不了单表变慢。

### 3.2 路由方式:哈希 vs 范围 vs 时间

#### 哈希路由(最常用)

```python
db_index = hash(user_id) % 16        # 选哪个库
table_index = hash(user_id) // 16 % 16  # 选哪张表
sql = f"SELECT * FROM order_{table_index} WHERE user_id = {user_id}"
# 在 db_{db_index} 上执行
```

**优势**:数据均匀分布,无热点。
**劣势**:扩容麻烦(从 16 库扩到 32 库,要重 hash 全部数据)。

#### 范围路由

```
order_id 1  ~ 1亿     → db0
order_id 1亿 ~ 2亿    → db1
order_id 2亿 ~ 3亿    → db2
```

**优势**:扩容简单(加 db3 接收新数据)。
**劣势**:**热点严重**——所有新订单都打 db_n,旧库基本没流量。

#### 时间路由

```
2023 年订单 → db_2023
2024 年订单 → db_2024
2025 年订单 → db_2025
```

**优势**:适合日志、订单等时间序列;冷库可以用更便宜的存储。
**劣势**:也有热点(当年的库压力大)。

> **互联网业务 90% 用哈希路由**——均匀压力比扩容方便更重要;**日志、监控、归档场景用时间路由**。**范围路由只适合 ID 严格自增且历史数据访问频次和新数据接近的场景**(很少)。

### 3.3 分片数怎么选

**选 2 的幂次方**(16、32、64、128):

- 哈希取模均匀
- 后期扩容时拆一变二容易
- 业内默认值

**预留量**:

```
当前数据 100 GB,3 年增长 10 倍 → 1 TB
单库容量目标 100 GB(留 buffer)
最少分 10 库,选 16 库
```

> **常见错误**:分 4 库够当下,3 年后又得扩——而扩容是地狱。**宁可一开始分得多一点(16/32),也别频繁扩**。**淘宝订单库一开始就分 1024 个**——别以为夸张,业务一爆发就庆幸。

## 四、分片键(Sharding Key)怎么选:决定生死

**选错分片键 = 整个架构推翻重来**——这是分库分表的最大决策。

### 4.1 分片键的两条铁律

**铁律 1:90% 的查询要带分片键**

如果分片键是 `user_id`,**所有用户维度查询都很快**(直接路由到一个库);**但商家维度、商品维度查询要扫所有库**——慢且贵。

**铁律 2:分片键要均匀分布**

用 `province_id` 当分片键 → 北上广用户多,这几个库压力大,西藏库基本空 → **数据倾斜,失败**。

### 4.2 几个典型业务的分片键选择

| 业务 | 推荐分片键 | 原因 |
| --- | --- | --- |
| 订单 | user_id(买家) | 99% 的查询是「我的订单」 |
| 微博 | user_id(发布者) | 「某人的微博列表」最常查 |
| IM 消息 | conversation_id | 一个会话的消息一起查 |
| 商品 | product_id | 商品详情独立查询 |
| 评论 | target_id(被评论的资源 ID) | 「某商品的评论」最常查 |

### 4.3 多维度查询的解法:基因法 / 多份数据

**问题**:订单按 `user_id` 分片,但商家(seller)也要查"我所有的订单"——按 `user_id` 分会导致跨库扫表。

**解法 A:基因法(同库不同分片键)**

将 `seller_id` 的 hash 作为 `user_id` 生成时的"基因",**让同一商家的订单天然落在同一库**。淘宝早期用过,**复杂且强耦合,现在很少用**。

**解法 B:存两份(空间换时间)**

按 `user_id` 分一份给买家查,按 `seller_id` 分一份给卖家查。两份数据通过 MQ 异步同步。**主流方案**——简单,代价是双倍存储。

**解法 C:走异构存储**

主表按 `user_id` 分,**另起一份 ES 或 ClickHouse**,所有跨维度查询走它。**互联网公司默认做法**——OLTP 走分库,OLAP 走列式存储。

> **没有"完美的分片键"**——总有少数查询要付出代价。**关键是 90/10**:90% 查询走分片键直达,10% 查询走兜底方案(异构存储 / 全库扫)。

## 五、分库后的三大痛点

分库分表的**真正难度不是分,是分完之后**。

### 5.1 痛点一:跨库 join 死掉

**问题**:`order` 在 order_db,`user` 在 user_db,**SQL 不能 join 了**。

**解法**:

#### A. 字段冗余

订单表里冗余存 `user_name` / `user_avatar`,**不再 join 用户表**。代价是用户改名时要更新所有订单(用 MQ 异步)。

#### B. 应用层 join

```python
orders = order_db.query("SELECT * FROM order WHERE user_id = 123")
user_ids = [o.user_id for o in orders]
users = user_db.query("SELECT * FROM user WHERE id IN ?", user_ids)
# 应用层组装
```

简单但慢——多了一次网络往返。

#### C. ES 异构

把 join 后的"宽表"提前算好存到 ES。复杂查询走 ES。

### 5.2 痛点二:跨分片排序 / 分页

**问题**:查"全平台最新 100 个订单",数据散在 16 个库。

**朴素解法(几乎所有人第一次想的)**:

```
每个库查最新 100 → 总共 1600 条 → 应用层排序取前 100
```

**问题**:

- 16 个库各查 100 条,**网络开销大**
- **深分页惨烈**:查第 100 页(skip 9900),要从每个库取 10000 条,网络爆炸

**正确解法**:

#### A. 用单独的 ES / ClickHouse

复杂排序、聚合、分页全部走异构存储。**主流互联网做法**。

#### B. 业务限制深分页

订单列表只允许查前 100 页,翻不到 1000 页。**淘宝 / 京东都这么做**——你试试翻它们订单列表第 1000 页,根本翻不动。

#### C. 游标分页代替 offset 分页

```sql
-- 错:SELECT * FROM order ORDER BY id DESC LIMIT 9900, 100
-- 对:SELECT * FROM order WHERE id < ? ORDER BY id DESC LIMIT 100
```

游标分页**不能跳页,但能无限翻**——**移动端无限滚动的标准做法**。

### 5.3 痛点三:分布式事务

**问题**:下单 = `order_db` 插订单 + `inventory_db` 扣库存,跨库,**MySQL 本地事务不管用**。

**解法见 23 篇订单与支付,这里只列方案选择**:

- **强一致**:2PC / XA(慢,性能差,几乎不用)
- **最终一致**:Saga / 本地消息表 / TCC(主流)
- **最简单**:同库时就别跨库,关联表都放同一分片下

## 六、分布式 ID 生成

分库后,**自增 ID 失效**——不同库的 `id` 会重复。需要全局唯一 ID。

### 6.1 几种方案对比

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| UUID | 简单,纯本地生成 | 36 字节,无序,索引性能差 |
| 数据库自增 | 严格有序 | 单点,QPS 上限低 |
| 号段模式 | 高性能,有序 | 重启可能浪费一段号 |
| 雪花算法(Snowflake) | 64 位,有序,高性能 | 时钟回拨问题 |
| Redis INCR | 高性能,有序 | 依赖 Redis 高可用 |
| TiDB / PolarDB 自带 | 业务无感 | 绑定 DB |

### 6.2 雪花算法(Snowflake)详解

最流行的方案,Twitter 开源:

```
64 位:
  1 位符号(永远 0)
  41 位时间戳(毫秒)        → 69 年
  10 位机器 ID(workerId)   → 1024 台机器
  12 位序列号               → 单机每毫秒 4096 个 ID

例:1234567890_1234567890_512_42
```

**优势**:有序(便于索引)、纯本地生成、QPS 极高。

**坑**:

- **时钟回拨**:NTP 校时时机器时间倒退,可能生成重复 ID
  - 对策:发现回拨就拒绝服务 / 等待
- **机器 ID 分配**:1024 个机器 ID 怎么分?
  - 对策:启动时去 ZooKeeper / 配置中心抢一个

### 6.3 号段模式(美团 Leaf)

每次从 DB 取一段 ID 区间(比如 1000-2000),发完了再取下一段:

```
应用本地缓存 [1000, 2000) 这一段 ID
本地用完 → 一次性向 DB 申请 [2000, 3000)

DB 只更新一次"已发到 3000",压力小
应用本地发 ID,极快
```

**优势**:有序、性能高、依赖少。
**缺点**:重启时浪费一段(没用完的 ID 丢弃)。

> **生产场景多用雪花或号段**——UUID 只用在不需要排序的场景(文件名、临时 token)。

## 七、在线扩容:从 16 库到 32 库

**最痛的运维操作**——客户在线,数据要重新分布。

### 7.1 朴素方案(不可用)

```
1. 停服务
2. 把每条数据按新规则 (hash % 32) 重算路由
3. 移到新库
4. 启服务
```

**死路一条**:数据 TB 级,迁移要几天,**没人能停服务这么久**。

### 7.2 主流方案:双写 + 数据同步

```
1. 32 库准备好(新增 16 库)
2. 应用层"双写":写入按 mod 16 写老库,同时按 mod 32 写新库
3. 后台任务把老库存量数据按 mod 32 同步到新库
4. 同步完成后切流量到新库
5. 老库下线
```

**关键技巧**:

- 双写阶段保证幂等(同一数据写老库写新库)
- 同步用 binlog(Canal / DTS)实时跟进增量
- 切流时灰度(1% → 10% → 50% → 100%)

### 7.3 终极优化:成倍扩容

```
原 16 库:db0 ~ db15
扩到 32 库:db0a / db0b / db1a / db1b / ...

数据迁移规则:
  db0 的数据,user_id % 32 == 0 留 db0a,== 16 移 db0b
  → 每个老库只迁一半数据,迁出去到新库
```

**优势**:每条数据**只可能在两个库之间二选一**,迁移路径明确,中途出错容易补。

> **成倍扩容是分库分表设计阶段就应该规划好的**——所以分片数选 2 的幂(16 → 32 → 64),不要选 10 → 20 这种。

## 八、ShardingSphere / Vitess / MyCat:分库中间件

不要自己手写分库路由——**用成熟的中间件**:

| 中间件 | 来源 | 特点 |
| --- | --- | --- |
| ShardingSphere | Apache | Java 生态最流行,功能全 |
| Vitess | YouTube → CNCF | K8s 原生,Go 语言,大规模实战 |
| MyCat | 国产 | 老牌,但社区活跃度下降 |
| ProxySQL | MySQL | 偏读写分离,分片功能弱 |

### 8.1 两种部署模式

**Sidecar 模式(JDBC 层)**:库直连 MySQL,中间件作为 JDBC Driver,客户端无感。**性能高,运维简单**。

**Proxy 模式**:中间件作为独立服务(类似 MySQL),应用连中间件。**多语言友好,但多了一跳延迟**。

> Java 项目优先用 ShardingSphere JDBC;**多语言项目用 ShardingSphere Proxy 或 Vitess**。

## 九、什么情况下不分库:NewSQL 和 HTAP

**云原生 NewSQL** 让你"不分库的分库":

| DB | 特点 |
| --- | --- |
| TiDB | 兼容 MySQL,自动分片,跨地域强一致 |
| PolarDB | 阿里云,兼容 MySQL,共享存储 + 多读节点 |
| CockroachDB | Spanner 风格,全球强一致 |
| OceanBase | 蚂蚁,多活 + 高并发 |

**优势**:对应用透明,SQL 还是普通 SQL,不用关心分片。
**代价**:复杂度藏在数据库里,运维仍需要专业 DBA;**成本通常比 MySQL 高几倍**。

> **新业务可以直接选 TiDB**——避免分库分表的所有痛苦,**但要做好成本和团队学习曲线的预算**。**老业务从 MySQL 迁 TiDB 也是大工程**,不是想迁就能迁。

## 十、踩坑提醒

1. **过早分库**——单库还能撑,提前分库纯找罪受
2. **不归档冷数据就分库**——5 年前的订单归档了 80% 数据消失,根本不用分
3. **分片键选错**——业务跑半年后想换,基本等于重做
4. **分片键不均匀**——按地区分,北上广撑爆,西藏空闲
5. **分太少**——半年后又扩,扩容是地狱
6. **不留扩容余地**——分 10 库不是 2 的幂次,扩到 20 麻烦
7. **跨库事务用 2PC**——性能极差,生产几乎不用
8. **跨库 join 用 MyCat 自动 join**——慢且不稳定,自己应用层 join 或走 ES
9. **深分页用 offset**——10 万行后死掉,必须改游标分页
10. **不用中间件自己手写路由**——bug 写不完,后期维护噩梦
11. **分布式 ID 用 UUID**——索引性能崩溃,选雪花或号段
12. **没考虑数据迁移**——分库容易,后期重新分片(rebalance)是地狱
