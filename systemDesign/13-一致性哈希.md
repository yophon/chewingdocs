# 一致性哈希

一致性哈希(Consistent Hashing)是**分布式系统的基础数学**——缓存路由、分库分表、CDN 边缘选择、Gateway 流量分发都靠它。它解决的是**普通哈希的致命缺陷:节点变化时几乎所有 key 都要重新分配**。理解一致性哈希,你就能理解 Redis Cluster、Cassandra、DynamoDB、Memcached 这一票分布式系统的底层路由逻辑。

> 一句话先记住:**一致性哈希用一个"环"代替"取模"**——节点和数据都映射到环上,数据归属"顺时针最近的节点"。**节点变化时只影响一段弧上的数据**,不像普通哈希全部重算。**虚拟节点是它能用的关键**——没有虚拟节点,数据必然倾斜,实际生产几乎不能用。

---

## 一、普通哈希的灾难

最朴素的分片做法:

```python
node = nodes[hash(key) % N]
```

**问题**:`N` 一变,**几乎所有 key 都要重新分配**。

```
原本 4 个节点:hash(key) % 4
某个 key 的 hash = 100 → 100 % 4 = 0 → 进 node[0]

加一个节点变 5 个:hash(key) % 5
同一个 key:100 % 5 = 0 → 仍进 node[0],巧合
但更多 key:101 % 4 = 1 → 进 node[1]
            101 % 5 = 1 → 仍进 node[1]?不,101 % 5 = 1 也巧合

实际上大多数 key 的归属都变了,只有少数巧合不变
```

**算一下**:从 4 节点扩到 5 节点,**约 80% 的 key 要重新分配**。

### 1.1 这意味着什么?

- **缓存场景**:80% 的缓存命中变 miss,DB 被打穿
- **分库场景**:80% 的数据要物理迁移
- **路由场景**:80% 的会话被打到新机器,业务可能错乱

> 这就是普通哈希的死穴——**根本没法在生产环境扩缩容**,加一台机器就能搞挂全系统。

---

## 二、一致性哈希:把节点和数据放到同一个环上

### 2.1 基本思想

```
1. 想象一个 0 ~ 2^32 的环(哈希空间)
2. 每个节点用 hash(节点标识) 映射到环上某个点
3. 每个数据也用 hash(key) 映射到环上某个点
4. 数据归属"顺时针方向最近的节点"
```

```
                  0 / 2^32
                     |
              N1 ────────── N2
            ╱                  ╲
        K1 ●                    ● K2
         ╱                       ╲
        ╱                         ╲
       N4 ────────── N3 ────────── 

K1 顺时针走遇到的第一个节点 = N1
K2 顺时针走遇到的第一个节点 = N2
```

### 2.2 节点变化时只影响一段弧

```
原本:N1 N2 N3 N4 四个节点

新增 N5:
                     N5
                     |
              N1 ────●────── N2
            ╱                  ╲
           ●                    ●
            ╲                  ╱
             N4 ──── N3 ─────

只有「N1 到 N5 之间」原本归 N2 的那部分 key
现在改归 N5。其他三个区域的 key 不变。
```

**关键:只有 1/N 的 key 受影响**(扩容到 N+1 时)。

```
4 节点扩到 5 节点:只有约 1/5 = 20% 的数据要重分配
普通哈希:80%
```

**性能差距 4 倍**——这就是「一致性」的含义:**节点变化对数据归属的影响"一致"地小**。

---

## 三、关键问题:数据倾斜

朴素的一致性哈希有个致命问题——**节点在环上分布不均**:

```
4 个节点的 hash 恰好挤在环的一边:
              N1 ●
              N2 ●
              N3 ●
              N4 ●
              ╲ ╱
              环的另一半完全没节点

绝大部分 key 都顺时针落到环的另一半,然后落到 N1 → N1 承担 70% 流量
```

哈希函数是均匀的,但**节点数太少时,均匀性体现不出来**。4 个节点在环上根本不可能均匀分布。

---

## 四、虚拟节点(Virtual Node):一致性哈希的"灵魂"

**解决方案:每个真实节点对应 N 个虚拟节点**,均匀撒在环上。

```
4 个真实节点,每个 100 个虚拟节点 → 总共 400 个虚拟节点撒在环上
现在环上密度高,接近真正均匀分布

数据 → 顺时针找最近的虚拟节点 → 虚拟节点对应的真实节点
```

**虚拟节点数量推荐**:

| 节点数 | 虚拟节点 / 真节点 | 总虚拟节点 |
| --- | --- | --- |
| 5-10 | 200-500 | 1000-5000 |
| 10-50 | 100-200 | 1000-10000 |
| 50+ | 50-100 | 5000+ |

> **没有虚拟节点的一致性哈希在生产几乎不能用**——数据倾斜让某些节点负载是其他节点的 5 倍。**Redis Cluster 的 16384 个 slot 就是虚拟节点的工程实现**。

---

## 五、实现示例(Python 伪代码)

```python
import hashlib
import bisect

class ConsistentHash:
    def __init__(self, virtual_count=200):
        self.virtual_count = virtual_count
        self.ring = {}        # virtual_hash → real_node
        self.sorted_keys = [] # 排序的 virtual_hash 列表(便于二分查找)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2**32)

    def add_node(self, node):
        for i in range(self.virtual_count):
            virtual_key = f"{node}#{i}"
            h = self._hash(virtual_key)
            self.ring[h] = node
            bisect.insort(self.sorted_keys, h)

    def remove_node(self, node):
        for i in range(self.virtual_count):
            virtual_key = f"{node}#{i}"
            h = self._hash(virtual_key)
            del self.ring[h]
            self.sorted_keys.remove(h)

    def get_node(self, key):
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect.bisect(self.sorted_keys, h) % len(self.sorted_keys)
        return self.ring[self.sorted_keys[idx]]

# 使用
ch = ConsistentHash(virtual_count=200)
ch.add_node("redis-1")
ch.add_node("redis-2")
ch.add_node("redis-3")

print(ch.get_node("user:123"))   # → redis-2
print(ch.get_node("user:456"))   # → redis-1
```

**关键点**:

- 用 `bisect`(二分查找)在排序列表里找,O(log N) 查询
- MD5 / SHA-1 作哈希(分布均匀)
- 虚拟节点用 `node#i` 作 key,保证不冲突

## 六、四个典型应用场景

### 6.1 缓存路由(Memcached / Redis Cluster)

**经典场景**——客户端用一致性哈希决定 key 存哪台 Redis:

```python
redis_client = consistent_hash.get_node(f"user:{user_id}")
redis_client.set(f"user:{user_id}", value)
```

**Memcached** 客户端原生支持(libmemcached 的 ketama 算法)。

**Redis Cluster** 用了变种——固定 16384 个 slot(虚拟节点),每个真实节点拥有一个 slot 区间。**扩容时只移动 slot,不重新哈希**——比纯一致性哈希更可控。

### 6.2 分布式存储(Dynamo / Cassandra)

数据分片到节点上,**一致性哈希决定主副本归属**,副本写到环上的下 N 个节点(N = 副本因子)。

```
key 的主副本    → 顺时针第 1 个节点
副本 1         → 顺时针第 2 个节点
副本 2         → 顺时针第 3 个节点
```

**优势**:扩容只影响一段弧,副本恢复影响小。

### 6.3 长连接路由(IM 网关)

**问题**:一个用户的长连接连到某台 IM 网关,**另一个用户给他发消息时,要把消息路由到那台网关**。

**解法**:**用户 ID 一致性哈希到某台网关**——长连接固定连那台,消息也固定路由到那台。

```python
def route_user(user_id):
    return ch.get_node(f"user:{user_id}")
```

详见 18 篇即时通讯。

### 6.4 CDN 边缘选择

CDN 节点用一致性哈希组成环,**回源请求按 URL 哈希路由到固定节点**——同一个 URL 总是同一个节点回源,**最大化缓存命中率**。

## 七、一致性哈希的进阶变种

### 7.1 Jump Consistent Hash

Google 提出的简化版,**不用环,直接计算**:

```python
def jump_hash(key, num_buckets):
    b, j = -1, 0
    while j < num_buckets:
        b = j
        key = (key * 2862933555777941757 + 1) & 0xFFFFFFFFFFFFFFFF
        j = int((b + 1) * (1 << 31) / ((key >> 33) + 1))
    return b
```

**优势**:无内存(不用存环),只算一次,极快。
**劣势**:节点编号必须连续(不能任意标识),节点删除复杂。
**适合**:节点数固定 / 缓慢增长的场景。

### 7.2 Rendezvous Hashing(HRW)

每个 key 对每个节点都算一次哈希,**取最大值**对应的节点:

```python
def rendezvous(key, nodes):
    return max(nodes, key=lambda n: hash(f"{key}:{n}"))
```

**优势**:无需虚拟节点,均匀性天然好。
**劣势**:N 个节点要算 N 次哈希(O(N) 而非 O(log N))。
**适合**:节点少、需要绝对均匀的场景。

### 7.3 Maglev Hashing(Google)

Google 内部 LB 用的算法。**生成一张静态查找表**,每个 key 哈希到表上某个槽位,槽位映射到节点。

**优势**:O(1) 查询,极高均匀性,扩容影响极小。
**劣势**:实现复杂。
**适合**:超大规模 LB(Google Maglev、Envoy)。

## 八、一致性哈希 vs Redis Slot vs ShardingSphere 取模

三种主流分片策略对比:

| 策略 | 分片数 | 扩容时影响 | 实现 | 适合 |
| --- | --- | --- | --- | --- |
| 一致性哈希(虚拟节点) | 物理节点数 | 1/N 数据迁移 | 客户端 | Memcached / 自研 |
| Redis Slot | 固定 16384 | 移动 slot,小 | 服务端协议 | Redis Cluster |
| 分库取模 | 固定(如 16) | 几乎不可扩容 | ShardingSphere | MySQL 分库 |

> **MySQL 分库分表通常用"固定取模"而非一致性哈希**——因为 MySQL 数据迁移成本极高,**一开始就分够,基本不扩容**;**Redis / 缓存类用一致性哈希**——扩缩容频繁,需要小影响。

## 九、一致性哈希解决不了的问题

### 9.1 数据倾斜的"业务级"原因

虚拟节点解决了**哈希均匀性**问题,但解决不了**业务倾斜**——某个 key 是大 V 用户、某个商品是爆款,**单个 key 的访问量是其他的 1000 倍**。

**对策**(详见 16/22 篇):

- 大 V 用户的数据在多个节点冗余
- 热点 key 在客户端做本地缓存
- 限流热点 key

### 9.2 节点宕机的「数据丢失」

一致性哈希只决定"key 归属",**不解决"数据丢了怎么办"**——节点挂了,它存的数据没了。

**对策**:**多副本**——一致性哈希决定主副本,副本写到环上紧邻节点。这是 Cassandra / Dynamo 的标准玩法。

### 9.3 节点变化时的迁移期

**节点 A 加入到 B 和 C 之间,期间 B 和 C 上原本属于 A 的 key 还在那**——访问时要么走老节点(过期数据)要么去新节点(没数据)。

**对策**:

- **双写**:迁移期同时写新旧节点
- **延迟读新节点**:迁移完成前优先读老节点

## 十、一致性哈希在 Redis Cluster 里的具体形态

Redis Cluster 不是用纯一致性哈希,而是**固定 16384 个 slot**:

```
1. CRC16(key) % 16384 = slot 编号(0-16383)
2. 每个节点拥有一个 slot 区间(如节点 1 拥有 slot 0-5460)
3. slot 信息通过 gossip 协议在节点间同步
```

**优势**:

- slot 数量固定,**节点数变化不影响哈希结果**
- 扩容时只迁移 slot(N 个 slot 数据),不需要重算所有 key 的归属
- 客户端缓存 slot → node 映射,直连目标节点

**劣势**:

- 上限 16384 个节点(实际生产不会这么多)
- 多 key 操作要求所有 key 在同一 slot(用 `{user:123}` 强制 hash tag)

> Redis Cluster 是「**固定 slot 的虚拟节点**」——本质还是一致性哈希思想,但工程上更可控。

## 十一、一致性哈希的"反向使用":有状态服务路由

不止用于数据,**有状态服务也用**:

### 11.1 长连接服务

用户 ID 一致性哈希到 IM 网关,**保证同一用户的消息都到同一网关**。

### 11.2 缓存预热的 worker 分配

10 个 worker 处理预热,**用一致性哈希分摊任务**——加 worker 时只重新分配少量任务。

### 11.3 流式计算的状态分区

Flink / Storm 的 keyBy 默认就是哈希分区,**重启时按一致性哈希重新分配 key 到新的 task**。

## 十二、踩坑提醒

1. **不用虚拟节点**——数据倾斜严重,某些节点流量是其他的几倍
2. **虚拟节点数太少**(<50 / 节点)——倾斜仍然明显
3. **哈希函数选 hash() / Java hashCode()**——分布不均,**用 MD5 / Murmur3 / xxHash**
4. **以为一致性哈希解决一切**——只解决"路由"问题,数据丢失靠副本,热点靠业务
5. **MySQL 分库用一致性哈希**——MySQL 数据迁移代价大,**用固定取模 + 一开始分够**
6. **节点变化不通知客户端**——客户端继续按老映射路由,大量请求到错节点
7. **没考虑同时多节点变化**——同时增加多个节点要按顺序处理,防止环抖动
8. **不持久化环状态**——服务重启后所有 key 路由变了,缓存全失效
9. **路由表太大塞内存**——大集群的 slot 表 / 虚拟节点表占用 MB 级内存
10. **以为 Redis Cluster 就是一致性哈希**——它是"固定 slot 的变种",查询时要查 slot 表,不能直接哈希
