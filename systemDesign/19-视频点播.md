# 视频点播

视频点播(VOD,Video on Demand)是**互联网带宽消耗最大的业务**——YouTube 每天分发 1 EB 视频数据,占全球互联网总流量的 25%。**视频系统的真正成本不是 CPU,是带宽**——一个 4K 视频被 1 亿人看一遍,带宽费用就是几百万美元。这就决定了视频架构的核心:**95% 的工作量在 CDN,5% 在源站**。这一篇讲清楚视频点播的全链路:**上传 → 转码 → 分发 → 自适应码率播放**。

> 一句话先记住:**视频点播 = 上传到对象存储 + 转码成多档码率 + 全球 CDN 分发 + 客户端自适应选码率**。**带宽决定一切**——所以要做转码(降码率)、自适应(动态选低码率)、CDN(就近分发降跨区带宽)。**最贵的是 CDN 带宽**,不是 CPU 也不是存储——这就是为什么 Netflix 在全球部署了 17000 个 OCA(Open Connect Appliance)边缘节点,而不是堆中心机房。

---

## 一、需求拆解

### 1.1 功能需求

- **上传视频**:用户传原片(几百 MB ~ 几 GB)
- **转码**:生成多档码率(360p / 720p / 1080p / 4K)
- **播放**:用户点击播放,流畅观看
- **自适应**:网络好放高清,网络差自动降码
- **进度记忆**:从上次停的地方继续看
- **CDN 加速**:用户就近拉视频

### 1.2 非功能需求

- **首屏时间**:< 1 秒(从点播放到开始播)
- **缓冲率**:< 0.5%(播放过程中卡顿比例)
- **可用性**:99.99%
- **规模**:DAU 5 亿,日上传 1000 万条,日观看 100 亿次

### 1.3 这一轮不做

- 直播(详见 20 篇)
- 视频推荐(详见 17 篇 Feed)
- 弹幕 / 评论(独立子系统)
- 视频版权 / DRM

## 二、容量估算

### 2.1 上传

```
日上传 1000 万条 × 平均 100 MB(原片)= 1 PB / 天
365 天 = 365 PB / 年
```

### 2.2 转码后存储

每个视频转 5 档码率,**总大小 = 原片 × 1.5**(压缩 + 多档冗余):

```
1 PB / 天 × 1.5 = 1.5 PB / 天 转码后
365 天 = 550 PB / 年
→ 必须冷热分层 + 对象存储
```

### 2.3 带宽

**这才是最大成本**:

```
DAU 5 亿 × 人均看 30 分钟 × 平均 720p(2 Mbps)
= 5 × 10^8 × 1800 × 2 × 10^6 bits / day
= 1.8 × 10^18 bits / day
= 225 PB / day 出站带宽
= 平均 21 Tbps,峰值 100 Tbps+
```

**单机房根本扛不住** —— 必须 CDN 散到全球几百个节点分摊。

### 2.4 CDN 节点

```
单 CDN 节点带宽:10 Gbps
需要节点数:100 Tbps / 10 Gbps = 10000 个

→ 全球 1000+ 个 CDN 节点,每节点 10+ 服务器
```

## 三、视频上传

### 3.1 直传 vs 分片上传

**小视频(< 100 MB)**:直接 HTTP PUT 到对象存储。

**大视频(> 100 MB)**:**分片上传**:

```
1. 客户端把视频切成 5MB / 片
2. 并发上传多个片(10 路并发)
3. 上传完调 "完成上传" 接口,服务端合并
4. 中途断网可恢复(断点续传)
```

S3 / 阿里 OSS / 腾讯 COS 都自带 multipart upload。

### 3.2 上传到哪

**直传对象存储**(不经过应用服务器):

```
客户端 → 申请上传凭证(预签名 URL) → 直传 S3
应用服务器只做权限校验,不传数据(节省带宽)
```

**预签名 URL**:

```python
import boto3
s3 = boto3.client('s3')
url = s3.generate_presigned_url(
    'put_object',
    Params={'Bucket': 'videos', 'Key': f'raw/{video_id}.mp4'},
    ExpiresIn=3600  # 1 小时内有效
)
```

### 3.3 上传完触发转码

```
S3 上传完成 → 触发事件 (S3 Event Notification)
            → SQS / Kafka
            → 转码 Worker 池消费
```

## 四、转码:从原片到多档码率

### 4.1 为什么要转码

**原片是用户随便录的格式**:H.265 / H.264 / VP9 / AV1,分辨率不一,码率高低不齐。

**转码目标**:

- 统一格式(用户播放器能解码)
- 多档码率(适配不同网速)
- 切片(便于自适应播放)

### 4.2 主流转码方案

```
原片(任意格式)
   ↓ FFmpeg
[
  360p  / H.264 / 500 Kbps,
  720p  / H.264 / 2 Mbps,
  1080p / H.264 / 5 Mbps,
  1080p / H.265 / 3 Mbps,  (新设备解码,带宽降 40%)
  4K    / AV1   / 15 Mbps  (旗舰设备,最新编码)
]
```

### 4.3 转码流水线

```
原片入对象存储
   ↓
转码任务入队列(Kafka)
   ↓
转码 Worker(GPU 集群)
   ├── FFmpeg 转 360p
   ├── FFmpeg 转 720p
   ├── FFmpeg 转 1080p
   └── FFmpeg 转 4K
   ↓
切片(每 2-10 秒一个 segment)
   ↓
生成 manifest(playlist 文件)
   ↓
推送到 CDN 边缘
```

**单个视频转码成本**:

- 1 小时 1080p 视频
- 用 GPU 转码 ≈ 5 分钟
- 用 CPU 转码 ≈ 1 小时
- 转 5 档 → 25 分钟 GPU / 5 小时 CPU

**抖音 / B 站都用 GPU 集群**——上传到能播,**目标 < 5 分钟**。

### 4.4 智能转码:不是所有视频都要转 4K

- 用户上传的是 360p?**不需要转 4K**(画质上不去)
- 视频长度只有 10 秒?**直接转完所有码率**
- 视频长度 2 小时?**只转 720p,等用户点击 1080p 再补转**(节省存储)

## 五、自适应码率:HLS / DASH

### 5.1 为什么要自适应

不同用户的网络:

- 4G 信号好:能看 1080p
- 4G 信号差:只够 480p
- 用户走进电梯:网速骤降
- 用户切到 WiFi:速度恢复

**死方案**:用户手动选码率 → 用户体验差。
**活方案**:**自适应**——播放器实时测带宽,动态选码率。

### 5.2 HLS(HTTP Live Streaming)

苹果发明,主流方案:

```
manifest.m3u8(总目录)
├── 360p.m3u8
│   ├── seg_0.ts (2 秒)
│   ├── seg_1.ts
│   └── ...
├── 720p.m3u8
│   ├── seg_0.ts
│   └── ...
└── 1080p.m3u8
    └── ...
```

**播放器逻辑**:

```
1. 下载 manifest.m3u8,看到有 3 档可选
2. 默认选中等档(720p)
3. 下载第一个 segment,测带宽
4. 带宽够 → 下次拉更高档
5. 带宽不够 → 下次拉更低档
6. 切换码率时下一个 segment 就是新码率
```

**优势**:

- 走 HTTP,**CDN 完美支持**(就是普通文件下载)
- 切换无缝(在 segment 边界切)
- 跨平台(浏览器、iOS、Android 都支持)

### 5.3 DASH(Dynamic Adaptive Streaming over HTTP)

Google 力推,和 HLS 几乎一样:

- HLS 用 m3u8 + ts,DASH 用 mpd + mp4
- DASH 是 ISO 标准
- HLS 苹果生态强,DASH Android 生态强

**实战**:同时输出 HLS 和 DASH,iOS 用 HLS,Android / Web 用 DASH。

### 5.4 为什么不用 RTMP

RTMP 是直播协议(Adobe 出品):

- 长连接 → CDN 不友好
- Flash 已死
- 移动端兼容差

**点播全用 HTTP-based(HLS / DASH)**;RTMP 留给推流端(直播)。

## 六、CDN 分发:视频点播的灵魂

### 6.1 为什么必须 CDN

```
单机房带宽:100 Gbps(顶天)
全球用户分布:中国 / 美国 / 印度 / 欧洲

不用 CDN:
  美国用户拉中国机房视频 → 200ms+ 延迟,丢包高,首屏 5 秒
  全球流量集中到一个机房 → 出站带宽几十秒就打满

用 CDN:
  美国用户拉美国 CDN 节点 → 10ms 延迟,首屏 < 1 秒
  流量分摊到 1000+ CDN 节点 → 单节点压力小
```

### 6.2 CDN 缓存策略

视频分两类:

```
热视频(头部 1%):
  缓存命中率 95%+
  CDN 主动预热(发布时就推到所有节点)
  TTL 长(7 天)

冷视频(长尾 99%):
  缓存命中率低
  按需回源
  TTL 短(1 小时)
```

### 6.3 CDN 调度

**用户怎么知道连哪个 CDN 节点**?

- **DNS 解析**:GeoDNS 按 IP 返回最近节点
- **HTTP 302**:智能调度服务返回最优节点 URL
- **客户端 SDK**:内置节点列表,实时测速选最快

### 6.4 自建 CDN vs 商用 CDN

```
商用 CDN(阿里云 / Cloudflare / Akamai):
  上线快,按量付费
  成本:0.10 - 0.30 美元 / GB
  适合:小到中流量

自建 CDN(Netflix Open Connect):
  前期投入巨大(硬件 + 机房)
  长期成本低
  适合:超大流量,YouTube / Netflix / 字节级
  
混合(主流):
  自建 + 商用补峰
  自建为主,大促 / 突发用商用兜底
```

## 七、视频元数据 + 存储分层

### 7.1 元数据

```sql
CREATE TABLE video (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT,
    title       VARCHAR(200),
    duration    INT,            -- 秒
    cover_url   VARCHAR(500),
    status      TINYINT,        -- 0 转码中 1 已发布 2 下架
    play_count  BIGINT,
    created_at  TIMESTAMP
);
-- 分库分表:按 id

-- 转码后多档信息
CREATE TABLE video_resource (
    video_id   BIGINT,
    quality    VARCHAR(10),    -- 360p / 720p / 1080p / 4K
    codec      VARCHAR(20),    -- h264 / h265 / av1
    url        VARCHAR(500),   -- HLS / DASH manifest URL
    size_bytes BIGINT
);
```

### 7.2 视频文件存储:三层

```
热数据(最近 30 天 / 头部内容):
  对象存储 + CDN 缓存
  S3 Standard / OSS 标准

温数据(30 天 ~ 1 年):
  对象存储,降一级
  S3 IA(不频繁访问)

冷数据(1 年以上):
  归档存储,极便宜
  S3 Glacier / OSS 归档
  访问要解冻(几分钟到几小时)
```

> **存储成本可以差 100 倍**——10 年前的老视频如果还放热存储,成本是天文数字。**自动迁移到冷存储**是大型视频网站必做的优化。

## 九、首屏优化:< 1 秒开始播放

视频网站的关键体验指标。

### 9.1 优化点

```
1. 首个 segment 用最低码率(瞬时下载)
2. 预加载封面图,先显示再播放
3. CDN 节点距离用户 < 50ms
4. HTTP/2 多路复用,减少握手
5. 先放第一秒,后续 segment 边播边下
6. 客户端预连接 CDN(DNS 预解析、TLS 握手提前)
```

### 9.2 关键指标:卡顿率

```
卡顿率 = 播放过程中缓冲时间 / 总播放时间

行业标杆:
  < 0.5%:优秀(YouTube 级)
  0.5%-2%:良好
  > 2%:用户开始流失
```

降低卡顿:**buffer 设大一点(预下载更多)+ 自适应反应快**。

## 十、防盗链 + 鉴权

视频是有版权的,**不能让人随便扒**:

### 10.1 URL 签名

```
原始 URL: https://cdn.example.com/video/123/720p.m3u8
签名 URL: https://cdn.example.com/video/123/720p.m3u8?expire=1234567890&sign=abc

服务端校验:expire 没过期 + sign 是用密钥算出来的
```

### 10.2 Referer 检查

CDN 配规则:**必须从 example.com 跳来才允许访问**。

### 10.3 Token 单次有效

每次播放向后端申请一个 token,**token 只能用一次或限时**。

### 10.4 DRM(数字版权管理)

收费内容(Netflix / 爱奇艺):**视频加密**,客户端解密播放,**根本拷贝不了原始数据**。

- Widevine(Google)
- FairPlay(Apple)
- PlayReady(Microsoft)

## 十二、踩坑提醒

1. **不上 CDN**——单机房带宽几十 Gbps,百万用户立刻打满
2. **只输出一档码率**——用户网速差只能卡顿,流失严重
3. **不用 HLS / DASH**——播放器自适应失败,体验崩
4. **转码用 CPU**——GPU 快 10 倍,大规模必须 GPU
5. **冷数据放热存储**——成本翻 100 倍
6. **CDN 不预热**——热门视频上线时 CDN 集体回源,源站打挂
7. **不分热冷视频**——所有视频一样的缓存策略
8. **不防盗链**——视频被各种盗版站爬走
9. **首屏不优化**——第一秒走 1080p,首屏 5 秒
10. **客户端没 retry**——网络抖一下播放挂掉
11. **没自适应缓冲**——稍微卡一下整段重放,用户暴怒
12. **不监控卡顿率**——核心体验指标缺失,迭代方向错
