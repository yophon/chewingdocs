登录与会话 ​

登录系统是所有应用的入口——它挂了,所有功能都用不了。没有比登录更"基础设施"的系统:亿级用户、100% 可用性要求、强一致(刚改的密码立刻生效)、强安全(账号被盗影响极大)。新手设计登录最容易忽略的就是密码存储(只用 bcrypt 不够)和Token 失效机制(JWT 看似优雅,实际上无法主动撤销)。这一篇按五段式模板拆解大型登录系统的设计。

一句话先记住:登录系统的核心矛盾是"无状态扩展" vs "可主动失效"——纯 JWT 无状态但撤销不了,纯 Session 有状态扩展难。主流方案是"短期 JWT + 长期 Session 黑名单" 的混合。密码存储不是"加 bcrypt 就完事"——加盐、迭代次数、防撞库、防暴力破解,每一个都不能少;登录系统的故障 = 全公司业务停摆,可用性要求是其他系统的 10 倍。

一、需求拆解 ​

1.1 功能需求 ​

  • 注册:邮箱 / 手机号 / 第三方(微信、Google)
  • 登录:用户名密码、短信验证码、第三方 OAuth
  • 登出:撤销当前会话
  • 修改密码 / 忘记密码:邮件 / 短信验证 + 重置
  • 会话管理:用户能看自己的登录设备并踢出
  • SSO 单点登录(企业级):一次登录所有内部系统通用

1.2 非功能需求 ​

  • 可用性:99.999%(一年 5 分钟,比业务系统高一个量级)
  • 安全性:密码存储不可逆,防撞库 / 暴力破解 / 中间人
  • 延迟:登录 < 500ms,Token 验证 < 5ms
  • 规模:用户 10 亿,DAU 1 亿,登录 QPS 峰值 5 万,Token 验证峰值 100 万

1.3 这一轮不做 ​

  • 二次验证(2FA / TOTP / 安全密钥)
  • 风控反作弊(异地登录提醒、设备指纹)
  • IDP 协议(SAML / OpenID Connect)细节

二、容量估算 ​

2.1 QPS 估算 ​

登录:DAU 1 亿 × 平均每人 0.5 次/天 = 5000 万/天
     平均 QPS = 500,峰值 ~5000(早高峰、大促)

Token 验证:每个 API 请求都要验,DAU 1 亿 × 100 次/天
         平均 QPS = 100 万,峰值 ~500 万

真正的瓶颈是 Token 验证——每个 HTTP 请求都过一次,这个 QPS 是登录 QPS 的 1000 倍。

2.2 存储估算 ​

用户表:10 亿 × 1 KB = 1 TB
       含索引副本 ×3 = 3 TB
       → 必须分库

Session 表:DAU 1 亿 × 平均 5 个会话(手机/电脑/平板) = 5 亿条
          每条 200B → 100 GB
          活跃数据,推荐 Redis 而不是 MySQL

2.3 带宽 ​

Token 验证 500 万 QPS × 100B → 500 MB/s——分布到多机房后 OK。

三、数据模型 ​

3.1 用户表(分库分表) ​

sql
CREATE TABLE user (
    id            BIGINT PRIMARY KEY,        -- 雪花 ID
    username      VARCHAR(64) UNIQUE,
    email         VARCHAR(128) UNIQUE,
    phone         VARCHAR(20) UNIQUE,
    password_hash VARCHAR(128),              -- bcrypt 后的 hash
    salt          VARCHAR(32),
    status        TINYINT,                   -- 0 正常 1 锁定 2 注销
    created_at    TIMESTAMP,
    updated_at    TIMESTAMP
);
-- 分片键: id(按 user_id 分 16 库)

-- 注意: username/email/phone 三个唯一索引 → 跨库查询难
-- 解决: 单独维护映射表

3.2 唯一标识反查表(独立小集群) ​

sql
CREATE TABLE user_lookup (
    key_value  VARCHAR(128),  -- email/phone/username
    key_type   TINYINT,       -- 1 email 2 phone 3 username
    user_id    BIGINT,
    PRIMARY KEY (key_value, key_type)
);

登录流程:先查 lookup 表拿到 user_id → 按 user_id 路由到分片查 user 表。

3.3 会话表(Redis) ​

Key:    session:{token}
Value:  {user_id, device, ip, login_at, expire_at}
TTL:    7 天
反查 Set:user_sessions:{user_id} → {token1, token2, ...}
作用:用户改密 / 强制登出时,通过 user_id 找出所有会话清掉

四、架构演进 ​

4.1 V1:Session 集中存(DAU < 100 万) ​

Client → LB → Spring Boot ×N → Redis(session 存储)
                              → MySQL(user 表)

登录:

python
def login(username, password):
    user = db.query("SELECT * FROM user WHERE username = ?", username)
    if not bcrypt.checkpw(password, user.password_hash):
        return error("invalid")
    
    token = uuid4()
    session = {"user_id": user.id, "expire_at": now() + 7d}
    redis.setex(f"session:{token}", 7*86400, json.dumps(session))
    return {"token": token}

验证:

python
def verify(token):
    session = redis.get(f"session:{token}")
    if not session:
        return None
    return json.loads(session)["user_id"]

优势:简单,撤销 Token 只需 redis.delete。 问题:Token 验证 100 万 QPS 全打 Redis,Redis 容量和性能压力大。

4.2 V2:JWT + Redis 黑名单(主流) ​

JWT 自带载荷——不用查 Redis 也能知道是谁:

Token = base64(header).base64(payload).signature
payload = { user_id: 123, exp: 1234567890, ... }
signature = HMAC_SHA256(header.payload, secret)

验证只需算 HMAC——不查任何 DB,单机能扛 10 万 QPS。

python
def verify(token):
    try:
        payload = jwt.decode(token, SECRET, algorithms=["HS256"])
        if redis.exists(f"blacklist:{payload['jti']}"):  # 黑名单查 Redis
            return None
        return payload["user_id"]
    except JWTError:
        return None

问题:JWT 主动失效难——签发后到 exp 前都有效,改密码 / 踢出登录怎么办?

解法:Redis 黑名单

python
def logout(token):
    payload = jwt.decode(token, SECRET)
    ttl = payload["exp"] - now()
    redis.setex(f"blacklist:{payload['jti']}", ttl, "1")

进化版:短期 JWT (15 分钟)+ 长期 Refresh Token (7 天,Redis 存):

Access Token:  JWT, 15 分钟过期, 不查 DB, 极快
Refresh Token: 服务端有记录,过期后用它换新 Access Token

用户改密 / 登出:删 Refresh Token → Access Token 最多 15 分钟后失效
紧急强制踢出:加黑名单(15 分钟内生效)

这是大厂登录系统的标准方案——既享受 JWT 的高性能,又保留撤销能力。

4.3 V3:多机房 + SSO(DAU 1 亿+) ​

                      用户全球
                          ↓
                    DNS / Anycast
                          ↓
              ┌───────┴───────┐
              ↓               ↓
       北美机房(Auth)     亚太机房(Auth)
              ↓               ↓
         共享 Redis(全球同步)  
              ↓
         共享 user 表(全球异步复制)

SSO:多业务系统共享一套登录:

用户 → 业务 A → 检测未登录 → 跳转 auth.example.com
                        ↓ 用户登录
                        ← 带 Token 跳回业务 A

用户 → 业务 B → 检测未登录(已登录 SSO) → 自动签发 B 的 Token

关键:SSO 中心是单点高可用——挂了所有业务都登不上,必须 5 个 9。

五、密码存储:不只是"用 bcrypt" ​

新手以为「我用 bcrypt 加盐了就安全」——远远不够。

5.1 错误的存储方式 ​

python
# ❌ 明文
db.insert(user, password)

# ❌ MD5
hash = md5(password)
# 彩虹表秒破

# ❌ MD5 + 全局盐
hash = md5(password + GLOBAL_SALT)
# 一个盐泄漏全员暴露

# ❌ SHA-256
hash = sha256(password + salt)
# SHA 太快,GPU 一秒算 10 亿次,暴力破解可行

5.2 正确的方式:bcrypt / scrypt / argon2 ​

这些算法故意设计得慢——单次哈希 100ms,GPU 也只能每秒算几千次,暴力破解不可行。

python
# bcrypt(主流)
import bcrypt
salt = bcrypt.gensalt(rounds=12)  # 12 轮,生产推荐
hash = bcrypt.hashpw(password.encode(), salt)

# argon2(更新更安全,2015 密码哈希竞赛冠军)
import argon2
hasher = argon2.PasswordHasher()
hash = hasher.hash(password)

对比:

算法速度抗 GPU抗 ASIC推荐
MD5 / SHA极快弱弱❌
bcrypt慢强中✅(主流)
scrypt慢强强✅
argon2慢强强✅(最新推荐)

2026 年的建议:新系统用 argon2id——它是最新标准。老系统继续用 bcrypt 也 OK。

5.3 加盐(Salt) ​

每个用户独立的随机盐,和密码哈希一起存:

sql
password_hash = '$2b$12$N9qo8uLOickgx2ZMRZoMye...'  -- bcrypt 的 hash 自带 salt

作用:

  • 即使两个用户密码相同,hash 也不同(防彩虹表)
  • 一个用户的 hash 泄漏不影响其他用户(因为各自有不同 salt)

5.4 慢哈希的迭代次数 ​

bcrypt 的 rounds 参数:

rounds = 10:  ~100ms / hash
rounds = 12:  ~400ms / hash  ← 推荐
rounds = 14:  ~1.5s / hash

经验:用最高的能容忍延迟——登录是低 QPS 接口,500ms 用户能接受,但黑客暴破成本翻 10 倍。

5.5 防撞库(Credential Stuffing) ​

攻击者用其他网站泄漏的用户名密码组合逐一试登录:

对策:

  • 失败次数限制:同一用户名 N 次失败 → 锁定 / 验证码
  • 同 IP 限频:1 分钟登录失败 > 10 次 → IP 封禁
  • HaveIBeenPwned 库检查:用户注册时检查密码是否在已知泄漏库里

六、Token 设计:深入细节 ​

6.1 JWT 的字段 ​

json
{
    "iss": "auth.example.com",      // 签发者
    "sub": "user:123",              // 主体(user_id)
    "aud": ["service-a", "service-b"], // 受众
    "exp": 1234567890,              // 过期时间
    "iat": 1234560000,              // 签发时间
    "jti": "uuid-of-this-token",    // Token 唯一 ID(用于黑名单)
    "scope": "read write"           // 权限范围
}

6.2 签名算法 ​

算法类型适合
HS256对称(HMAC)单一服务,所有节点共享 secret
RS256非对称(RSA)多服务,签发用私钥,验证用公钥
ES256非对称(ECDSA)同 RS,密钥更短性能更好

微服务架构推荐 RS256 / ES256——其他服务只需公钥就能验证,不用接触私钥;单一服务 HS256 简单够用。

6.3 Token 存放位置 ​

位置优点缺点
LocalStorage简单XSS 攻击可读取
Cookie(HttpOnly)XSS 读不到CSRF 风险
Cookie(HttpOnly + SameSite=Strict)安全跨域复杂
Authorization Header灵活客户端要主动加

Web 推荐 HttpOnly + SameSite=Strict Cookie;移动端 / 跨域 API 用 Header。

七、SSO 单点登录 ​

企业内多个系统共享登录:

7.1 OAuth 2.0 / OpenID Connect 流程 ​

1. 用户访问业务系统 A
2. A 检测未登录 → 重定向到 SSO(auth.example.com)
3. SSO 检查自己有没有 session
   - 有: 跳过登录步骤
   - 无: 显示登录页, 用户输入密码登录
4. SSO 生成 authorization code, 重定向回 A
5. A 用 code 换 access_token
6. A 拿 access_token 访问 SSO 的 /userinfo, 拿到用户身份
7. A 给用户签发自己的 session/token

7.2 SSO 的关键挑战 ​

  • 可用性:SSO 挂 = 全公司业务挂,5 个 9 起步
  • 登出同步:用户在 A 登出,B / C 也要登出 → 用 Back-Channel Logout
  • Token 撤销:同步到所有受信系统

7.3 主流方案 ​

方案特点
OAuth 2.0 + OIDC业界标准,GitHub / Google 都用
SAML 2.0企业级老牌,XML 笨重但企业内仍广泛
CAS学校 / 政府常用
自研简单场景可以,大规模别自研

八、登录系统的故障域 ​

登录系统挂的影响比业务系统大 10 倍——所有业务都挂。设计时要把"登录组件"自己也做成多副本 + 多机房 + 降级方案。

8.1 关键降级方案 ​

故障降级策略
Redis 挂JWT 验证不查黑名单,牺牲撤销能力换可用
用户 DB 挂已登录用户继续可用(JWT 不查 DB);新登录失败
SSO 挂各业务自己的 session 仍然有效;新登录失败
短信网关挂临时关闭手机验证,只允许密码登录

8.2 降级原则 ​

已登录用户的体验绝不能受影响——只能让"新登录"失败。

实现:Token 验证不依赖任何外部 DB(用 JWT),只有登录 / 撤销才依赖 DB。

九、第三方登录(Social Login) ​

9.1 OAuth 流程 ​

用户点"用 GitHub 登录"
↓ 跳转 github.com/login/oauth/authorize
用户授权
↓ 回调 yourapp.com/auth/github/callback?code=XXX
后端用 code 换 access_token
↓ 用 access_token 调 GitHub API 拿用户信息
本地创建 / 关联 user, 签发自己的 token

9.2 关键设计 ​

  • 关联多个第三方:一个用户可以同时绑定 GitHub、Google、微信
  • 昵称冲突:GitHub 用户名和邮箱已存在的本地账号冲突
  • 解绑保护:不能解绑最后一个登录方式(否则永远登不上)

9.3 安全要点 ​

  • Redirect URI 白名单:防止 OAuth 跳转被篡改
  • state 参数防 CSRF:授权请求带 state,回调时校验

十、最终架构图 ​

                       用户(Web / App)
                              ↓
                       DNS / Anycast
                              ↓
                       ┌──────┴──────┐
                       ↓             ↓
               ┌──────────────┐  ┌──────────────┐
               │ 北美 SSO 集群 │  │ 亚太 SSO 集群 │
               │  RS256 验签  │  │  RS256 验签  │
               └────┬────┬────┘  └────┬─────┬───┘
                    ↓    ↓            ↓     ↓
                  Redis  MySQL      Redis MySQL
                  (会话) (user 库)   (会话) (user 库)
                    ↓ 全球同步        ↓
                  ┌─────────────────────┐
                  │ 跨机房 Redis 复制   │
                  │ MySQL 异步主从       │
                  └─────────────────────┘

关键设计:

  • JWT(短期)+ Refresh Token(长期)
  • 业务系统本地验证 JWT(无需调用 SSO 中心)
  • 撤销靠 Refresh Token 删除 + JWT 黑名单
  • 多机房双活(用户全球分布)
  • SSO 中心 5 个 9

十一、踩坑提醒 ​

  1. 明文 / MD5 存密码——基础错,但仍有公司在犯
  2. bcrypt rounds 设太低(10)——10 年前的硬件够,2026 年要 12+
  3. 全局盐——一个泄漏全员遭殃
  4. JWT 没设 exp——一个 token 永久有效,撤销不掉
  5. JWT secret 太短 / 写在代码里——定期轮换 + 配置中心管理
  6. Token 在 LocalStorage——XSS 一中招全完
  7. 登录接口没限流——撞库攻击秒打几万次
  8. 不监控异地登录——账号被盗用户不知道
  9. 登出只删客户端 Token——服务端没记录,Token 仍有效
  10. 修改密码不踢出所有会话——黑客拿到 Token 后就算改密码也能继续操作
  11. SSO 没多机房——SSO 挂全公司挂
  12. 第三方登录不验 redirect_uri——授权码可被劫持
