# 系统设计系列写作计划

> 来源:`未来系列规划.md` 第一类「补完整知识图」第一项,⭐⭐⭐⭐⭐。
> backendLearning 教了「用什么工具」(Spring / Redis / MQ / K8s),systemDesign 教「**工具怎么组合应对真实流量**」。这一层不补,后端知识就是一堆零件,拼不出能扛百万 QPS 的系统。
> **这个系列的目标读者**:能独立写 CRUD、用过 Redis 和 MQ,但被问「Twitter 怎么设计」会卡住的工程师——也就是 3 ~ 5 年经验、卡在架构师之前那一层的人。

---

## 设计思路

四层结构,从「思考方式」到「真实案例」:

```
心智与基础 (架构怎么想 / 容量估算 / CAP / 高可用)
   → 抗压基础设施 (LB / 缓存 / 分库 / MQ / 限流 / 一致性哈希)
   → 经典案例 (短链 → IM → 秒杀 → 12306)
   → 进阶 (多机房 / 容灾 / 灰度 / 设计文档)
```

不重复 backendLearning 已经讲过的「Redis 数据结构」「K8s 工作负载」,而是聚焦**这些工具怎么搭起来撑 1 亿用户**。每篇案例都按「需求拆解 → 容量估算 → 数据模型 → 架构演进 → 关键取舍 → 失败模式」的固定模板,**让你能模仿这个套路自己设计任何系统**。

## 写作模板(每篇案例固定五段)

```
1. 需求拆解
   - 功能需求(用户做什么)
   - 非功能需求(QPS / 数据规模 / 延迟 / 可用性)
   - 一句话画清边界:不做什么

2. 容量估算
   - DAU / QPS / 存储 / 带宽 的数量级估算
   - 算出来的数字直接决定后面架构选什么

3. 数据模型与核心算法
   - 表 / KV / 索引怎么设计
   - 关键操作的复杂度

4. 架构演进
   - 从「单机能扛」开始,逐步加缓存 → 加分库 → 加 MQ → 多活
   - 不要一上来就画终极版,展示推理过程

5. 关键取舍 / 失败模式
   - 选了 A 放弃了什么
   - 真实公司翻过哪些车
```

**理论篇(01-13、25-30)**不套这个模板,但都遵守「为什么这样 → 实操怎么做 → 何时不该用」的三段式。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「先讲个故事」开头——直接进重点
2. 罗列名词解释——讲清楚一个概念在系统里的「位置」比定义重要
3. 「业内有 A、B、C、D 几种方案」然后并列讲——必须给出推荐和场景
4. 把代码贴几百行——本系列以架构图、伪代码、表格为主
5. 「面试经常问到」——读者来这里不是为了应付面试,是为了真的会
6. 抄 Designing Data-Intensive Applications——那本书已经写得最好了,这里只补它没讲透的工程细节
