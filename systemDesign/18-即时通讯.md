# 即时通讯

IM(Instant Messaging)是**分布式系统里最难的一类**——同时在线 5 亿人,每一条消息必须可达、有序、不重复、跨设备同步,还要支持万人大群、离线推送、跨地域低延迟。微信工程团队为这个系统投入超过 1000 人年,才把这套东西稳定下来。新手最常见的误解是「IM 就是 WebSocket + Redis 嘛」——**完全不是**。这一篇讲清楚 IM 的核心难点:**长连接管理、消息可达三步 ACK、离线消息存储、群聊扩散、跨设备多端同步**。

> 一句话先记住:**IM 的核心矛盾是"长连接是有状态的,但有状态难扩展"**——5 亿用户每人一条 TCP,要管理 5 亿个连接;**消息可达靠"业务级三步 ACK"**(send→server→ack→deliver→ack),不是 TCP 层那个 ACK;**群聊不能傻 fanout**——5000 人群一条消息扩散 5000 次,1 万人发同一时间打挂集群。**用 HTTP 做 IM 的人都还在 demo 阶段**,真实 IM 必须长连接。

---

## 一、需求拆解

### 1.1 功能需求

- **单聊**:A 给 B 发消息,B 收到
- **群聊**:A 在群里发消息,所有群成员收到
- **离线消息**:B 不在线时也能收到 A 发的消息(上线后拉取)
- **多端同步**:A 在手机发的消息,同时显示在 PC / Pad
- **已读 / 未读**
- **撤回**(2 分钟内)
- **消息类型**:文本 / 图片 / 视频 / 语音 / 文件 / 位置

### 1.2 非功能需求

- **可达性**:消息必达(微信级 99.9999%)
- **顺序性**:同一会话内消息严格有序
- **延迟**:同地域 < 100ms,跨地域 < 500ms
- **同时在线**:5 亿
- **峰值消息**:春节红包高峰 100 万条/秒

### 1.3 这一轮不做

- 音视频通话(WebRTC,详见 mediaTech 系列)
- 端到端加密(信号协议 / Signal Protocol)
- 内容审核

## 二、容量估算

### 2.1 长连接数

```
DAU 10 亿,同时在线峰值 = DAU × 30% = 3 亿同时在线
单台 IM 网关 ≈ 10 万长连接(epoll 调优后)
→ 至少 3000 台 IM 网关
```

### 2.2 消息 QPS

```
DAU × 人均发消息 50 条 / 天 = 500 亿条 / 天
平均 QPS = 50 万,峰值 ≈ 500 万

群聊放大:平均一条消息发给 10 个人(单聊 1 + 群聊放大)
→ 实际投递 = 500 万 × 10 = 5000 万投递 QPS
```

### 2.3 存储

```
500 亿消息 / 天 × 500 字节 = 25 TB / 天
保留 5 年 = 45 PB
→ 必须分层存储(热 + 冷)
```

### 2.4 关键洞察

**IM 是少数几个"读不是瓶颈、写才是"的系统**——每个用户读自己的消息很轻,但**消息扩散写**才是大头。

---

## 三、为什么 IM 必须用长连接

### 3.1 HTTP 短连接的问题

```
HTTP 收消息 = 客户端轮询服务器 ("有新消息吗?")

每秒一次 → 5 亿用户每秒 5 亿请求 → 永远没人扛得住
每分钟一次 → 消息延迟 1 分钟,用户体验崩溃
```

### 3.2 长连接(WebSocket / 自研协议)

```
客户端连一次 → TCP 长连接保持 → 服务端有消息主动 push

5 亿用户 = 5 亿条 TCP 连接(成本可控)
延迟 < 100ms(TCP 往返一次)
```

**主流选择**:

| 协议 | 适合 |
| --- | --- |
| WebSocket | Web 端首选 |
| MQTT | IoT / 移动端低功耗 |
| 自研 TCP / TLS | 微信、QQ 都自研(协议头更紧凑) |

> 微信用自研协议(基于 TCP),**协议头比 HTTP 小一个量级**,移动端省流量、省电。

---

## 四、长连接架构

### 4.1 核心组件

```
                   用户 (手机/PC)
                       ↓ TCP 长连接
                   ┌───────────┐
                   │  接入层 LB │  (TCP 层,L4)
                   └─────┬─────┘
                         ↓
              ┌──────────┴──────────┐
              ↓                     ↓
     ┌──────────────┐       ┌──────────────┐
     │  IM Gateway  │  ...  │  IM Gateway  │  3000+ 台
     │  (长连接持有) │       │              │
     └──────┬───────┘       └──────┬───────┘
            ↓                       ↓
       ┌────┴───────────────────────┴────┐
       │     消息处理服务(无状态)         │
       └────┬────────────────────────────┘
            ↓
       ┌────┴────────┐
       │   Kafka     │  (消息队列,核心数据通道)
       └────┬────────┘
            ↓
       ┌────┴───────────────┬─────────────────┐
       ↓                    ↓                 ↓
   ┌────────┐         ┌──────────┐      ┌──────────┐
   │ 消息存储 │         │ 路由服务 │      │ 推送服务 │
   │ (HBase) │         │  (谁在哪)│      │ (离线)   │
   └────────┘         └──────────┘      └──────────┘
```

### 4.2 关键:用户 → IM Gateway 的路由

**问题**:A 给 B 发消息,**B 连在哪台 Gateway 上**?

**解法:全局路由表**

```
key:   user_route:{user_id}
value: gateway_id (B 当前连接的 Gateway)
存储:  Redis Cluster
```

**用户上线时**:Gateway 把"user_id → gateway_id"写入 Redis。
**用户下线时**:Gateway 把这个 key 删除。
**发消息时**:查 Redis 找到 B 的 Gateway,把消息推送过去。

### 4.3 一致性哈希优化路由

为了让长连接尽量稳定,**用户 ID 一致性哈希到 Gateway**:

```python
gateway = consistent_hash.get_node(user_id)
```

详见 13 篇一致性哈希。**优势:扩缩容时只影响 1/N 用户重连**。

## 五、消息可达:三步 ACK

**TCP 的 ACK 不能保证业务可达**——TCP 收到了不等于应用处理了。

### 5.1 完整三步流程

```
A 发消息 → 服务器
  ↓ ACK1(服务器收到了,赋予 server_msg_id)

服务器 → B(在线)
  ↓ ACK2(B 收到了)

B 客户端处理(写入本地数据库)
  ↓ ACK3(业务确认)

服务器收到 ACK3 → 标记消息为"已投递"
```

### 5.2 失败重试

**任何一个 ACK 没回来 → 重试**:

- A 没收 ACK1 → A 重发(同 client_msg_id,服务器幂等)
- B 没收消息 / 服务器没收 ACK2 → 服务器重推
- 服务器没收 ACK3 → B 标记为已读时再 ACK

**幂等关键**:消息有 `client_msg_id`(A 生成的客户端 ID),服务器去重时用它。

### 5.3 消息状态机

```
created → server_received → delivered → read
```

每个状态用户都能看到(已发送 / 已送达 / 已读)。

## 六、消息存储:写扩散 vs 读扩散

最核心的设计抉择,**和 16 篇 Twitter 的 Push/Pull 类似**:

### 6.1 写扩散(Push,主流)

发消息时,**为每个收件人写一条**:

```
A 发给 B:
  inbox:A 写一份(发件箱)
  inbox:B 写一份(收件箱)

群聊 1000 人:
  inbox 写 1000 份
```

**优点**:读极快(直接读自己的 inbox)。
**缺点**:大群放大写入。

### 6.2 读扩散(Pull,小群和小型 IM)

发消息只写一份到群,**读时聚合**:

```
A 在群发:
  group_messages:{group_id} 写一份

B 读群:
  按 group_id + last_read_id 查最新消息
```

**优点**:写少,适合大群。
**缺点**:消息按会话存,跨会话聚合"我所有未读"需要遍历所有群。

### 6.3 微信用什么

**微信用写扩散**——99% 是单聊或小群,写扩散读快;**大群另开优化路径**(下文 §8)。

### 6.4 存储引擎

```
活跃 30 天消息:    HBase / Cassandra(海量写入,LSM 树)
30 天前归档:       对象存储(S3 / OSS),按用户 ID 分目录
```

**HBase 的优势**:

- 写入 LSM,扛海量写
- 按 user_id + timestamp 范围扫描快(读 inbox)
- 单 region 数据量大时自动分裂

## 七、离线消息

B 不在线时,**消息暂存**——B 上线后拉取。

### 7.1 离线消息存储

```
key: offline:{user_id}
type: List(消息 ID 列表)
TTL: 7 天(超过的认为永远不在线了)
```

或直接存到 inbox(写扩散),**上线时按 last_sync_id 拉新消息**。

### 7.2 离线推送

B 长时间不在线 → 通过手机推送(APNs / 小米推送 / 华为推送):

```
"你有 1 条新消息" → 用户点击进入 App → 触发拉取离线消息
```

**关键**:不直接推消息内容(隐私 + 推送 payload 大小限制),只推一个通知。

### 7.3 离线消息的限制

```
保留 7 天   超过删除(占用大量存储)
最多 1000 条 太多了挤占,只显示最近 1000 条
大文件不存  视频 / 图片只存 URL,真内容在对象存储
```

## 八、群聊优化:大群"读扩散"

写扩散在大群里崩溃:

```
微信 500 人群,一个人发一条消息 → 写 500 份
微信 1 万人群(企业群),一条消息 → 写 1 万份

10 万个群同时发消息 → 10 万 × 1 万 = 10 亿次写入/秒 → 不可能
```

### 8.1 大群用读扩散

```
群消息只写一份:group_messages:{group_id}
每个用户记录:last_read:{user_id}:{group_id}

用户进群:从 last_read 拉最新消息
未读计数:总数 - last_read
```

**节省 1000 倍写入**——但读群消息时多一步聚合。

### 8.2 群聊还是要"通知"

读扩散省了存储,**但用户怎么知道有新消息**?

**轻量通知**:

```
群里有人发消息 → 推送一条"通知"给所有在线成员
通知 payload 极小:{group_id, latest_msg_id, sender_id}
不带消息内容,客户端按通知去拉
```

**离线成员**:不通知 / 仅推送一次(避免红点轰炸)。

### 8.3 万人群的边界

微信 500 人,QQ 群 2000 人,企业微信 1 万人——**超过这个就算"频道"了**(类似 Discord),用专门的频道架构(订阅模式,消息只存一份,订阅者按需拉)。

## 九、多端同步

A 同时在手机 + PC 登录,**消息要在两端都显示**。

### 9.1 多端的会话状态

```
key: user_devices:{user_id}
value: [device_1_gateway, device_2_gateway, ...]

发消息时:推送到所有设备的 Gateway
```

### 9.2 已读同步

A 在手机上读了消息 → PC 上的红点也要清:

```
A 在手机上 read msg_123 → 服务器记 read_state:{user}:{conv} = 123
广播到所有设备:你的会话 X 已读到 msg_123
```

### 9.3 漫游消息

新设备登录 → 拉取**最近 N 天**的消息:

```
新 PC 登录 → 服务器返回最近 7 天消息(从云端 HBase 拉)
本地数据库填充 → 用户看到完整历史
```

> **微信"换手机后聊天记录在哪"** —— 需要服务端有「漫游消息」服务,**这是一个独立的子系统**。

## 十、消息顺序

「同一会话内消息严格有序」——**怎么保证**?

### 10.1 单聊顺序

A 给 B 连发 3 条 → B 必须按顺序看到。

**问题**:网络抖动,2 号到了,1 号还在路上 → B 先看到 2 号。

**解决**:

- A 客户端给消息打**单调递增的本地序号**
- B 客户端按本地序号缓冲,顺序展示

### 10.2 群聊顺序

群里 3 个人同时发消息 → **服务器统一定序**:

- 服务器收到第一条赋 group_seq=1
- 第二条 group_seq=2
- 客户端按 group_seq 排列

**关键:群消息必须由同一个 partition / 同一个服务实例处理**——按 group_id 哈希到固定 partition,保证顺序。**详见 10 篇 MQ 顺序消息**。

## 十一、关键问题:撤回 / 删除 / 编辑

发出去的消息要撤回:

```
A 发消息 msg_123 → B 已经收到并显示
A 撤回 → 服务器记 msg_123 状态为 recalled
       → 推送 "撤回" 信号到所有设备
       → B 客户端收到后,删除本地的 msg_123,显示 "A 撤回了一条消息"
```

**已经离线的人怎么办**:撤回信号也通过同样的可达机制(三步 ACK)送达。

**2 分钟限制**:服务端校验 `now - msg.created_at < 120s`,超过拒绝撤回。

## 十二、跨地域多机房

微信 / WhatsApp 全球用户:

```
中美用户互发消息 → 跨太平洋光纤 → 200ms+ 延迟
                ↓
                 用户体验差
```

**优化**:

- 用户就近接入本地机房
- 消息存储跨机房异步复制

## 十四、踩坑提醒

1. **用 HTTP 短连接做 IM**——只能 demo,生产必须长连接
2. **长连接没心跳**——TCP 半死状态(连接还在但服务不响应),客户端永远收不到消息
3. **没用 Anycast / 就近接入**——跨地域用户延迟 500ms+
4. **大群用写扩散**——一条消息扩散 1 万份,集群崩
5. **不做消息幂等**——网络重试导致消息重复
6. **只信 TCP ACK**——TCP 收到 ≠ 业务收到,必须三步业务 ACK
7. **没有离线消息保留期**——存所有历史消息成本爆炸
8. **多端同步靠每端单独连**——消息推 N 次重复
9. **单聊和群聊用同一存储模型**——没法 hybrid 优化
10. **没限流大群**——某个万人群活跃时打挂集群
11. **撤回信号没 ACK**——下线用户上线后看到本该撤回的消息
12. **路由表不持久化**——Gateway 重启所有用户掉线,要全部重连
