# 负载均衡

负载均衡(Load Balancer,简称 LB)是**所有分布式系统的入口枢纽**——没有 LB,你只是「一台机器跑一份代码」;有了 LB,你才有「N 台机器分摊流量」。但 LB 不是单一概念,它分布在从 DNS 到反向代理的**整条流量路径上**,每一层用不同的技术。**新手最大的误解是「LB = Nginx」**,实际上一个生产请求要经过 4-5 层不同的负载均衡才能到达业务代码。这一篇讲清楚:**LB 在哪几层、每一层用什么、为什么这么分层**。

> 一句话先记住:**负载均衡不是一个组件,是一个"流量分发栈"**——从 DNS 解析(全球分流)到 Anycast(就近接入)到 LVS(机房入口)到 Nginx(应用入口)层层接力。**L4 看 IP 和端口,L7 看 HTTP 内容**——各自适合不同场景,搞混就性能崩。**不存在"最好的 LB"**,只有"对应这一层的 LB"。

---

## 一、为什么需要分层负载均衡

新手脑子里 LB 通常长这样:

```
Client → Nginx → Web ×N
```

实际上一个生产系统是这样:

```
Client
  ↓ DNS 解析(就近选择机房)
  ↓
机房入口 IP(Anycast / VIP)
  ↓ L4 LB(LVS / 硬件 F5)
  ↓ 同机房的网关层
  ↓ L7 LB(Nginx / Envoy)
  ↓ 应用层
  ↓ 服务发现(微服务内部 LB)
  ↓ 后端业务实例
```

**为什么要分这么多层**?

- **DNS / Anycast**:全球流量,选最近的机房
- **L4(LVS)**:机房入口,百万 QPS 转发,极简但快
- **L7(Nginx)**:应用层,看 URL / Header,做精细路由
- **服务发现**:服务间 RPC,微服务专用

**每一层的目标和限制都不同**,不是「能否合并」的问题,是「不同问题用不同工具」。

---

## 二、L4 vs L7:本质差异

这是 LB 最核心的两个概念,**搞清楚这两个,LB 知识点的 80% 就通了**。

### 2.1 工作层次

```
L7 (HTTP/HTTPS)  ← 看得见 URL、Header、Cookie
L4 (TCP/UDP)     ← 看得见 IP、端口
L3 (IP)          ← 路由器干的事
```

### 2.2 对比

| 维度 | L4 | L7 |
| --- | --- | --- |
| 看的内容 | IP + 端口 | URL / Header / Cookie / Body |
| 处理方式 | 改包头转发 | 解析 HTTP 后再转发 |
| 单机性能 | 100 万 QPS | 5-10 万 QPS |
| 内存 / CPU 开销 | 极低 | 高(要做 HTTP 解析) |
| 典型实现 | LVS / F5 / IPVS | Nginx / HAProxy / Envoy / Traefik |
| 适用 | 入口大流量分流 | 应用级路由、SSL 终止 |
| 能做的事 | 按 IP 哈希、轮询 | 按 URL 路由、按 Header 灰度、SSL 卸载 |
| 跨协议 | TCP / UDP 都行 | 只懂 HTTP(系列产品支持 gRPC) |

### 2.3 为什么 L4 那么快

L4 不解析任何应用层协议——**它只改包头的 IP 和端口,然后 forward**。流程极短:

```
收到 SYN  → 选一个后端 → 改包头 → 转发
收到 ACK  → 查会话表 → 转发到同一个后端
```

LVS 用 IPVS(内核态)做这件事,**完全不进用户态**,单机能扛百万 QPS。

### 2.4 为什么 L7 那么"慢"

L7 要做的事多得多:

- 等完整 HTTP 报文(可能要拼包)
- 解析 URL / Header / Body
- 决定路由(比如 `/api/*` 走 A,`/static/*` 走 B)
- 可能要做 SSL 解密(CPU 密集)
- 重写 Header / 加内部标识
- 转发并等待响应,反向再处理一次

每一步都比 L4 重,所以**单机吞吐低一个量级**。

> **不是 L4 比 L7 好**,是**两者负责不同的层**。一个生产系统通常**L4 顶在最前(扛流量),L7 在内层(做精细路由)**。

## 三、典型负载均衡组合

### 3.1 中小型(单机房)

```
DNS → Nginx → 业务实例
```

DNS 解析到 Nginx 的公网 IP,Nginx 直接转后端。**简单、便宜、单机扛 5 万 QPS 没问题**——大多数 SaaS 公司够用。

### 3.2 中型(单机房,流量较大)

```
DNS → LVS(L4) → Nginx ×N(L7) → 业务实例
```

LVS 在前面做 L4 转发,**支撑 100 万 QPS 入口**;Nginx 多实例做 L7 路由。**淘宝 / 京东早期典型架构**。

### 3.3 大型(多机房,全球用户)

```
DNS(GeoDNS) → Anycast 入口 → LVS → Nginx → 业务
                  ↘
                    各机房就近接入
```

GeoDNS 按用户地理位置返回不同机房 IP;Anycast 让多机房共用同一个 IP,自动路由到最近;每个机房内部还是 L4 + L7 分层。**Google / Cloudflare / 字节跳动级别**。

### 3.4 微服务内部

```
应用 → 服务发现(Nacos / Eureka)→ 客户端负载均衡(Ribbon)→ 目标服务
                          ↘
                            或 Service Mesh(Envoy Sidecar)
```

服务间 RPC 不走外部 LB,而是**客户端拿到目标服务的 IP 列表后自己选一个**(client-side LB),性能最高。

## 四、负载均衡算法

LB 算法决定**「下一个请求发给谁」**。常见算法:

### 4.1 轮询(Round Robin)

```
后端 [A, B, C]
请求依次发 A → B → C → A → B → C ...
```

**适合**:后端机器配置相同。
**问题**:不考虑后端负载。

### 4.2 加权轮询(Weighted Round Robin)

后端不同配置,给不同权重:

```
A 权重 5, B 权重 3, C 权重 2
8 个请求里 A 拿 5 个,B 拿 3 个,C 拿 2 个
```

**适合**:机器配置不同(老机器 + 新机器混部)。

### 4.3 最少连接(Least Connection)

发给当前连接数最少的后端。
**适合**:长连接 / 处理时间不一的请求(WebSocket、流式)。

### 4.4 IP 哈希(IP Hash)

按客户端 IP 哈希,**同一 IP 每次都到同一个后端**。
**适合**:Session 粘滞(Session 存在内存里,要找回原来的机器)。
**问题**:扩缩容时大量会话失效。

### 4.5 一致性哈希(Consistent Hashing)

IP 哈希的进化版,**扩缩容只影响 1/N 的请求**(详见 13 篇)。
**适合**:有状态服务(缓存、长连接路由)。

### 4.6 最快响应(Least Response Time)

发给响应最快的后端。
**适合**:对延迟敏感的服务。
**问题**:实现复杂,要持续测响应时间。

### 4.7 随机(Random)

随机选一个。**简单粗暴,但实际上和轮询效果接近**——大数定律下分布均匀。

> **真实生产 90% 用「加权轮询」或「最少连接」**——其他算法适合特殊场景,**别为了"先进"用 IP 哈希,反而会因为某个 IP 段的用户暴增打挂一台**。

---

## 五、DNS 负载均衡

DNS 是**最朴素的全球级 LB**——一个域名解析到多个 IP,客户端选一个连。

### 5.1 多 A 记录

```
api.example.com →  1.2.3.4
                   1.2.3.5
                   1.2.3.6
```

DNS 服务器返回所有 IP,**客户端决定连哪个**(通常按返回顺序)。

**优点**:零成本扩展。
**缺点**:

- 一个 IP 挂了,DNS 缓存让客户端继续连(几分钟到几小时)
- 不能根据后端负载动态调整

### 5.2 GeoDNS(地理 DNS)

按客户端 IP 所在地区返回不同 IP:

```
中国用户  → 北京机房 IP
美国用户  → 加州机房 IP
欧洲用户  → 法兰克福机房 IP
```

**实现**:Route 53(AWS)、Cloud DNS(GCP)、阿里云 DNS。

**用途**:全球业务的就近接入,降低延迟。

### 5.3 DNS 的局限

- **TTL 缓存**:即使后端挂了,DNS 缓存让客户端继续访问坏 IP
- **不能做端口分发**:DNS 只解析 IP,端口要客户端写死
- **不能做内容感知**:看不见 URL,不能按路径分流

> DNS 是「广撒网」式的 LB,**只适合宏观分流(地理 / 大机房选择)**,精细流量调度必须靠下层 LB。

---

## 六、Anycast 负载均衡

Anycast = **多个机房共用同一个 IP**,**用户连这个 IP 时被路由到"最近"的机房**(BGP 路由协议决定)。

```
1.1.1.1
   ↓ BGP 路由
   ├── 北京机房有这个 IP
   ├── 上海机房也有这个 IP
   ├── 东京机房也有这个 IP
   └── ...

中国用户 ping 1.1.1.1 → 进北京机房
日本用户 ping 1.1.1.1 → 进东京机房
```

**典型场景**:Cloudflare、Google DNS(8.8.8.8)、CDN 边缘节点。

**优点**:

- 极低延迟(就近接入)
- 自动容灾(一个机房挂了 BGP 自动绕路)

**缺点**:

- 需要自己有 AS 号 + 跟 ISP 谈 BGP(普通公司做不了)
- 无连接协议(UDP / DNS / HTTP/3)效果最好,长 TCP 连接可能切机房导致中断

> **Anycast 是 CDN / 公有云的"看家技术"**——你买云服务的 LB 时已经隐含用了。**自建 Anycast 是大厂级别的工程**。

---

## 七、L4 实现:LVS

LVS(Linux Virtual Server)是**最常见的开源 L4 LB**,跑在 Linux 内核态(IPVS 模块)。

### 7.1 三种工作模式

```
NAT 模式:LB 改源 + 目的 IP,流量经过 LB(进出都过)
         简单,但 LB 是瓶颈

DR 模式: LB 只改目的 MAC,后端直接回包给客户端
         流量只单向过 LB,极快(99% 生产用这个)

TUN 模式:IP 隧道封装,允许跨 VLAN
         少用,被 DR 替代
```

### 7.2 性能

DR 模式下单机 LVS 能扛**百万 QPS**——**因为它不解析任何应用层协议,只改 MAC 地址**。

### 7.3 高可用

LVS 自身要双机:

```
LVS 主 + LVS 备 + Keepalived(VRRP 协议)
共享一个 VIP,主挂 → 备秒级接管
```

**LVS + Keepalived + Nginx + Web** 是中国互联网最经典的"四件套"组合。

## 八、L7 实现:Nginx vs HAProxy vs Envoy

三大主流 L7 LB,各有侧重:

| 维度 | Nginx | HAProxy | Envoy |
| --- | --- | --- | --- |
| 强项 | 反向代理 + 静态文件 | 纯 LB + 健康检查 | 微服务 / Service Mesh |
| 配置 | nginx.conf | haproxy.cfg | xDS API(动态) |
| 热更新 | reload(短暂中断) | hot reload | 完全动态(API) |
| 协议 | HTTP/HTTPS/TCP/gRPC | HTTP/TCP | HTTP/gRPC/TCP/UDP |
| 性能 | 高 | 极高 | 高(略低于 Nginx) |
| 流行度 | 互联网通用 | 银行 / 高频交易 | 云原生 / Istio 默认 |
| 学习成本 | 低 | 中 | 高 |

> **新建项目用 Nginx 还是 Envoy**?**单体 / 简单微服务 → Nginx;Service Mesh / 多协议 / 动态调度 → Envoy**。HAProxy 在「极致性能 + 静态后端」场景仍然无敌(金融业务)。

### 8.1 Nginx 配置示例

```nginx
upstream backend {
    least_conn;                    # 算法
    server 10.0.0.1:8080 weight=3;
    server 10.0.0.2:8080 weight=2;
    server 10.0.0.3:8080 backup;   # 备用,前面挂了才用
}

server {
    listen 80;
    server_name api.example.com;

    location /api/ {
        proxy_pass http://backend;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /static/ {
        root /var/www;
    }
}
```

### 8.2 Envoy 与 xDS

Envoy 不靠静态配置文件,**通过 xDS API 动态从控制平面拉配置**(Istio / Consul Connect)。这让动态服务发现 + 动态路由变成原生能力,**适合大规模微服务**。

## 九、SSL / TLS 卸载

HTTPS 解密是 CPU 密集操作。**在哪里解密**很关键:

```
方案 A:在 LB 层卸载
   Client ──HTTPS──→ LB ──HTTP──→ 后端
   后端不感知 HTTPS,LB 是 CPU 大户

方案 B:端到端 HTTPS
   Client ──HTTPS──→ LB ──HTTPS──→ 后端
   每一跳都解密 + 重新加密,极度低效

方案 C:LB 透传(SNI 路由)
   Client ──HTTPS──→ LB(L4 透传)──HTTPS──→ 后端
   LB 只看 SNI 决定路由,不解密
```

**95% 场景用方案 A**——LB 卸载 SSL,内部走 HTTP。**金融 / 合规要求端到端时用方案 C**。

## 十、Session 粘滞(Sticky Session)

**问题**:用户登录后 Session 存在某台 Web 的内存里,下次请求被 LB 分到另一台,Session 丢失。

**三种解决方案**:

### 10.1 IP Hash 粘滞(简单但脆弱)

```nginx
upstream backend {
    ip_hash;
    server ...;
}
```

**问题**:扩缩容时大量用户被打到新机器(没 Session 又要重新登录);手机用户切换网络 IP 变化。

### 10.2 Cookie 粘滞(LB 注入 Cookie)

LB 给每个用户一个 Cookie,记录第一次分到的后端,后续按 Cookie 路由。

**问题**:仍然是"机器挂了 Session 丢"。

### 10.3 集中存 Session(推荐)

**Session 存 Redis,Web 完全无状态**——LB 怎么分都行,机器挂了用户无感。

> **现代架构默认用方案 3**——**任何「需要 Session 粘滞」的设计都是过时的**。**让 Web 无状态,Session 进 Redis,这是 Web 层水平扩展的前提**。

## 十一、健康检查与自动剔除

LB 必须知道哪些后端活着。**健康检查不准 = 高可用是空话**。

### 11.1 主动健康检查

LB 定期 ping 后端的 `/health` 接口:

```nginx
location /health {
    return 200 "OK";
}
```

```nginx
upstream backend {
    server 10.0.0.1:8080 max_fails=3 fail_timeout=30s;
}
```

**含义**:30 秒内连续 3 次失败,标记为 down,30 秒后再试。

### 11.2 被动健康检查

观察真实请求的失败率,N 次失败自动剔除一段时间。**Nginx Plus / Envoy 支持,开源 Nginx 不支持**。

### 11.3 健康检查的反模式

```nginx
location /health {
    proxy_pass http://db_check;  # ❌ DB 抖一下整个集群被剔除
}
```

**对策**:健康检查只判断**进程自身是否就绪**,不判断下游依赖。下游问题靠业务监控发现。

## 十二、踩坑提醒

1. **LB 单点**——LB 自身没冗余,LB 挂了全挂(必须双机 + VIP 漂移)
2. **DNS TTL 过长**——后端 IP 变了客户端缓存几小时,故障切不走(关键域名 TTL 设 60s)
3. **L7 卡 CPU**——SSL 卸载没考虑 CPU,流量一来 LB 自己挂(用硬件加速 / 分层卸载)
4. **健康检查依赖 DB**——前面讲了,**血泪教训**
5. **Session 粘滞用 IP Hash**——手机用户 IP 一变就重登,客诉一片
6. **算法用 IP Hash 当负载均衡**——某个 IP 段用户暴增,集中打挂一台
7. **L4 / L7 混用混淆**——L4 看不见 URL,你写 `proxy_pass http://path/api/*` 它根本不知道路径
8. **不区分 LB 内外网**——内网服务暴露给外部 LB,绕过认证
9. **没限流**——LB 后端容量有限,LB 又不限流,直接被打穿
10. **以为 K8s Service 就够了**——K8s ClusterIP 是 L4 简易 LB,**生产入口必须前置 Ingress(L7)**
