    限流

    限流是保护系统不被打挂的最后一道闸门——上游突发流量、爬虫刷接口、DDoS 攻击、活动开始的瞬时洪水,任何超出系统承载能力的流量都必须被限掉,不限就是连锁雪崩。但限流不是"加个 RateLimiter 就完了",新手最常见的错是算法选错(用计数器去做秒杀、用漏桶去防刷)、位置放错(限流放在业务代码而不是网关)、阈值拍脑袋(凭感觉填 1000 QPS,实际系统能扛 10000)。这一篇把四大算法、单机/分布式实现、阈值确定方法讲清楚。

    一句话先记住:限流不是"少接请求",是"在系统能扛的范围内接最多的请求"——拒掉的请求不是损失,是保住整个系统不崩溃的代价。四个算法没有最好的,只有最适合场景的:计数器适合粗暴防刷,滑动窗口适合精确控制,令牌桶适合允许突发,漏桶适合严格平滑。单机限流简单但不全局公平,分布式限流准确但有性能代价——选错就翻车。

    一、为什么需要限流:连锁雪崩的物理学

    不限流的系统在过载时不是变慢,是直接死:

    正常:5000 QPS,每个请求 50ms
突发:10 万 QPS 涌入

线程池 200 个全占满 → 后面请求排队
排队过长 → 上游超时 → 上游疯狂重试 → 流量再翻倍
DB 连接池打爆 → DB 拒绝新连接 → 业务全部 5xx
GC 频繁 → CPU 打满 → 健康检查失败 → 实例被剔除 → 剩下的实例承担更多流量 → 又被压垮

    这就是雪崩——一个组件过载,通过重试和上游加压,几分钟内把整个系统拖垮。

    限流不是"性能优化",是"生存保障"。一个没有限流的系统,不是能不能扛过流量,是早晚被打挂的问题。

    二、四大限流算法

    2.1 计数器(Fixed Window)

    最简单粗暴:按时间窗口数请求,超了就拒。

    class FixedWindow:
    def __init__(self, limit, window_seconds):
        self.limit = limit
        self.window = window_seconds
        self.count = 0
        self.start = time.time()

    def allow(self):
        now = time.time()
        if now - self.start >= self.window:
            self.count = 0
            self.start = now
        if self.count < self.limit:
            self.count += 1
            return True
        return False

    致命缺陷:窗口边界双倍流量

    窗口 1(0~1s):限 100,前 0.5s 没流量,0.99s 涌进 100 个 → 全过
窗口 2(1~2s):1.01s 又涌进 100 个 → 全过

实际上 0.99s ~ 1.01s 这 20ms 内涌进了 200 个 → 等于 1 万 QPS 突袭

    适合:粗暴的防刷接口(每分钟 60 次登录尝试)。 不适合:任何对突发敏感的场景。

    2.2 滑动窗口(Sliding Window)

    修复了计数器的边界问题:窗口跟着时间滑动。

    把 1 秒拆成 10 个 100ms 的小桶
每个请求记录到对应的小桶

判断时:统计最近 10 个桶(过去 1 秒)的总数

    精度更高——边界问题大幅缓解(粒度越细越平滑)。

    适合:精确控制 QPS 的场景(API 网关、计费接口)。 实现:Sentinel、Resilience4j 默认用这个。

    2.3 令牌桶(Token Bucket)

    思想:桶里以恒定速率放令牌,请求来了拿一个令牌,拿到才能过。

    桶容量 = 100,放令牌速率 = 10/s

平时桶里有 100 个令牌(满)
突发 100 个请求 → 一次性拿走 100 个令牌 → 全过
后续请求 → 等令牌恢复(每 100ms 多 1 个)

    核心特点:允许短时突发,长期平均速率受限。

    class TokenBucket:
    def __init__(self, capacity, rate):
        self.capacity = capacity
        self.tokens = capacity
        self.rate = rate
        self.last_time = time.time()

    def allow(self):
        now = time.time()
        elapsed = now - self.last_time
        self.tokens = min(self.capacity, self.tokens + elapsed * self.rate)
        self.last_time = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

    适合:互联网通用场景(允许小突发,长期平均控制)。 实现:Guava RateLimiter、Nginx limit_req、AWS API Gateway。

    2.4 漏桶(Leaky Bucket)

    思想:请求进桶,桶以恒定速率漏水(处理),桶满了拒绝。

    桶容量 = 100,漏水速率 = 10/s

100 个请求瞬间进桶 → 桶满,后续拒绝
请求按 10/s 速率被处理(无论上游怎么涌入)

    核心特点:严格平滑输出,不允许突发。

    适合:对下游极度敏感、必须平滑的场景(给慢系统/老系统当代理)。 反对:大多数互联网业务——用户期待"快",漏桶强行慢化体验差。

    2.5 四算法对比

    算法 突发支持 平滑度 实现复杂度 典型场景
    计数器 边界翻倍 差 极简 简单防刷
    滑动窗口 不支持 好 中 精确 QPS 控制
    令牌桶 支持 中 中 互联网通用
    漏桶 不支持 极好 中 强平滑场景

    没有最好的算法,只有"对应场景"的算法。互联网业务 80% 用令牌桶——用户希望系统响应快,允许突发就是好体验。

    三、限流的五个维度

    不只是"限 QPS",真实场景按多个维度组合限流:

    3.1 全局限流

    整个系统每秒最多接 N 个请求。最简单但最粗。

    3.2 单接口限流

    /api/login 限 100 QPS,/api/order 限 1000 QPS。核心接口给更多预算。

    3.3 单用户限流

    每个 user_id 每分钟最多 60 次请求——防个人爆刷。

    3.4 单 IP 限流

    每个 IP 每秒最多 10 次——防爬虫和恶意刷。

    3.5 单租户限流(SaaS)

    企业 A 的总配额 1 万 QPS,企业内部用户共享这个池子——多租户隔离。

    真实生产是这五维组合——同一 IP + 同一接口 + 一秒内 + 单用户 + 总流量 五道闸。任何一道触发都拒。

    四、单机限流:简单但有局限

    4.1 Guava RateLimiter(令牌桶)

    RateLimiter limiter = RateLimiter.create(100);  // 100 QPS

if (limiter.tryAcquire()) {
    // 处理请求
} else {
    return 429;  // Too Many Requests
}

    优势:零依赖,纯本地内存,微秒级开销。

    局限:N 台机器,实际限流 = 100 × N——单机限流不是全局限流。

    4.2 Sentinel(阿里开源,功能最全)

    支持滑动窗口、令牌桶、漏桶、热点参数限流、黑白名单、熔断降级——生产级首选。

    @SentinelResource(value = "createOrder", blockHandler = "handleBlock")
public Order createOrder(...) { ... }

public Order handleBlock(BlockException ex) { 
    return Order.fallback(); 
}

    4.3 Nginx limit_req(网关层)

    limit_req_zone $binary_remote_addr zone=login:10m rate=10r/s;

server {
    location /login {
        limit_req zone=login burst=20 nodelay;
        proxy_pass http://backend;
    }
}

    在网关挡掉——根本不进业务系统,性价比最高。

    4.4 单机限流的根本问题

    N 台机器各限 100 QPS,实际全局 100N QPS,且分布不均:

    • 机器 1 接到 200 QPS,被限到 100,拒了 100 个用户
    • 机器 2 接到 50 QPS,完全没用上配额

    结果:实际只用了 150 QPS,却拒了 100 用户——资源浪费 + 用户体验差。

    五、分布式限流:全局统一计数

    5.1 Redis + Lua 脚本(主流方案)

    -- 滑动窗口限流脚本
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)
local count = redis.call('ZCARD', key)
if count < limit then
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window)
    return 1
else
    return 0
end

    script = redis.register_script(LUA_SCRIPT)
allowed = script(keys=[f"limit:{user_id}"], args=[100, 60, int(time.time()*1000)])

    优势:全局精确,所有机器看到同一个计数器。 劣势:每个请求多一次 Redis 调用(+ 0.5ms)。

    5.2 Sentinel Cluster 模式

    Sentinel 提供专门的 token-server 集中管理令牌。比 Redis 方案略快但要单独部署。

    5.3 分布式限流的取舍

    维度 单机限流 分布式限流
    性能 微秒级 +0.5ms
    准确性 粗(N×) 精确
    依赖 无 依赖 Redis
    可用性 Redis 挂不影响 Redis 挂限流失效
    适用 单机能扛的小限流 全局精确控制

    生产实战:网关层用分布式限流(Redis)做精确全局控制,业务层用单机限流(Guava / Sentinel)做兜底。两层都要——网关挡掉 90%,业务挡掉漏过的 10%。

    六、限流阈值怎么定:不能拍脑袋

    错的方法:「我感觉 1000 QPS 差不多」。 对的方法:压测得出真实容量,按 70-80% 设阈值。

    6.1 单机容量压测

    1. 压测工具:wrk / jmeter / locust
2. 阶梯加压:100 → 500 → 1000 → 5000 QPS
3. 观察:p99 延迟、错误率、CPU、内存
4. 找到拐点:p99 突然飙升或错误率 > 1% 的 QPS
   → 这就是单机极限(比如 3000 QPS)
5. 限流阈值设极限的 70-80%(2400 QPS)

    70-80% 是给突发留 buffer——平时跑 70%,突发能扛到 100% 再开始拒。

    6.2 全局阈值

    单机限流阈值 × 实例数 = 全局阈值
2400 × 10 台 = 24000 QPS

    但要注意下游能不能扛——你 24000 QPS,但 DB 只能写 5000,前面限再松也是把 DB 打挂。限流要按整条链路最弱的一环来定。

    6.3 动态调整

    业务流量不是恒定的:

    • 早高峰 / 晚高峰流量是均值的 5 倍
    • 大促期间临时调高阈值
    • 故障期间临时调低保命

    生产系统都用配置中心动态调阈值——Nacos / Apollo 推下去,实时生效。

    七、限流之后:被拒的请求怎么办

    直接返回 429 是最简单的——但用户体验差。真正友好的处理:

    7.1 排队等待(令牌桶 + 阻塞)

    Guava RateLimiter 的 acquire() 会阻塞等令牌——适合后台任务(用户感知不强烈)。

    7.2 返回兜底数据

    被限流时返回缓存的旧数据 / 默认列表——用户感知不到限流,但拿到的是"降级版"。

    7.3 排队 + 异步回调

    秒杀场景:返回"排队中,稍后通知",真处理完再推送结果。

    7.4 友好提示 + 重试建议

    {
    "error": "限流触发",
    "retry_after": 5,
    "suggestion": "5 秒后再试"
}

    让客户端按 Retry-After 退避——比直接 429 友好。

    限流不是"对用户残忍",是"先保命再服务"——拒掉的用户损失,远小于系统全挂的损失。

    八、限流的位置:网关 > 业务 > 中间件

    限流应该尽早做——越早挡掉,系统越省力:

    8.1 接入层(CDN / WAF)

    CDN 层限 IP / 限 URL,根本不进机房。Cloudflare、阿里 WAF 都自带。

    8.2 网关层(Nginx / Spring Cloud Gateway)

    主战场——按接口、按用户、按租户限流。

    8.3 业务层(应用代码)

    热点接口 / 核心接口的二次保护。Sentinel / Hystrix。

    8.4 中间件层(DB / Redis 内部)

    DB 连接数限制、Redis maxmemory——最后的兜底。

    "限流要分层"——不要指望一层挡掉所有。每层挡一部分,层层防御。

    九、热点参数限流(Hot Key Throttling)

    特殊场景:特定参数触发热点。比如某个商品 ID 突然爆火,涉及它的请求 QPS 超高。

    @SentinelResource(value = "queryProduct", 
    blockHandler = "handleBlock",
    fallback = "fallback")
public Product queryProduct(@SentinelParam Long productId) { ... }

// Sentinel 规则:productId = 12345 这个参数限 100 QPS

    没有热点限流的话,热点商品请求会冲爆缓存,缓存击穿打 DB。详见 07 篇缓存。

    十、限流的反模式

    1. 没限流就上线——一个流量尖峰就挂
    2. 限流阈值设在「目前 QPS」——没留任何 buffer,正常波动就触发限流
    3. 限流逻辑放在业务最末端——已经走完整个链路才发现要限,资源全浪费
    4. N 台机器各限 1000,以为全局限了 1000——实际限了 N×1000
    5. 限流后直接 502 / 500——给用户混乱信号,改成 429 + Retry-After
    6. 限流但不熔断——下游慢了上游限不住,雪崩
    7. 限流但不降级——拒了之后没兜底,用户白屏
    8. 不监控限流命中率——限流被频繁触发说明阈值低或扩容不够
    9. 生产改阈值要发版——必须配置中心动态生效
    10. 限流配置不灰度——一改全平台,改错全瘫

    十一、踩坑提醒

    1. 算法选错——秒杀不能用漏桶(突发请求秒被拒)
    2. 单机限流当全局限流——N 倍偏差
    3. 分布式限流没考虑 Redis 故障——Redis 挂了限流变 0
    4. 不区分接口——核心接口和废弃接口一个阈值
    5. 不区分用户角色——VIP 和普通用户共享配额,VIP 体验差
    6. 限流阈值不更新——业务量翻倍但阈值没改,白白拒用户
    7. 被限流的请求不记录——出事时不知道限了多少
    8. 网关层限流后仍打到业务——网关限流没生效,白部署
    9. Lua 脚本写错——一个原子性 bug 就能让限流失效
    10. 不与熔断配合——下游挂了上游还在硬限,流量打到一半失败一半成功
