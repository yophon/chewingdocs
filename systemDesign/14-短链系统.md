# 短链系统

短链是**系统设计的经典入门题**——`https://t.co/aBcD123` → `https://超长的原始链接...`,听上去 5 分钟能答完,实际涉及**发号器、哈希、跳转性能、防热点、统计、防滥用**,是分布式系统的"小型百科全书"。从这一篇起,**所有案例都用统一的五段式模板**:需求拆解 → 容量估算 → 数据模型 → 架构演进 → 关键取舍。**吃透这个模板,后面 10 个案例都是套用**。

> 一句话先记住:**短链的核心是"6 位字符串 → 跳转目标"的映射**,看起来是 KV 查表,**真正的难点在 1) 怎么生成不重复的短码,2) 怎么扛 10 万 QPS 跳转**。**99% 的设计精力在跳转读路径**——写少读多比例 1:1000,跳转必须 < 10ms,且经常碰到爆款链接(一个短链一秒被刷 10 万次)。

---

## 一、需求拆解

### 1.1 功能需求

- **生成短链**:输入长 URL,返回 `t.co/aBcD12` 这样的短链
- **跳转**:访问短链,302 重定向到原 URL
- **自定义短链**(可选):允许用户指定自己的后缀,如 `t.co/my-blog`
- **过期 / 删除**:可设置有效期
- **统计**(可选):统计每个短链的点击量、来源、设备

### 1.2 非功能需求

- **可用性**:99.99%(短链挂了所有营销链接全失效)
- **延迟**:跳转 < 10ms(多一步重定向不能太慢)
- **规模**:DAU 1 亿,日新增短链 1000 万,日跳转 100 亿
- **持久性**:短链一旦生成,**永远不能误删 / 重复**

### 1.3 这一轮不做

- 链接预览(社交分享时显示标题 + 缩略图)
- A/B 测试(同一短链在不同时间跳到不同 URL)
- 风控反作弊(垃圾链接识别)

## 二、容量估算

### 2.1 QPS 估算

```
日新增短链 1000 万 → 平均生成 QPS = 10^7 / 10^5 = 100
                  → 峰值 QPS ≈ 500

日跳转 100 亿 → 平均 QPS = 10^10 / 10^5 = 10 万
            → 峰值 QPS ≈ 50 万
```

**读写比 = 1:1000**——典型读多写少。**所有设计精力放在读路径**。

### 2.2 存储估算

```
每条短链:
  short_code(6 字符)            6 字节
  long_url(平均 100 字符)       100 字节
  user_id, created_at, expire_at  20 字节
  总计:~150 字节

日新增 1000 万 → 1.5 GB / 天
5 年存储:1.5 GB × 365 × 5 ≈ 3 TB
含索引副本 ×3 ≈ 10 TB

10 TB 量级,单库扛不住,要分库
```

### 2.3 带宽估算

```
跳转 50 万 QPS × 单次响应 200 字节 = 100 MB/s = 800 Mbps
```

**单机房千兆口刚够**——多机房时分摊。

### 2.4 短链长度

```
6 位 [a-zA-Z0-9] → 62^6 = 568 亿种组合
够用 5000 年(每年新增 1000 × 365 = 36.5 亿)
```

**6 位足够**——再长用户体验差(短链就该短)。

## 三、数据模型

### 3.1 主表

```sql
CREATE TABLE short_url (
    short_code  VARCHAR(8) PRIMARY KEY,
    long_url    TEXT NOT NULL,
    user_id     BIGINT,
    created_at  TIMESTAMP,
    expire_at   TIMESTAMP,
    INDEX idx_user (user_id, created_at)
) ENGINE=InnoDB;
```

### 3.2 反向索引(去重用)

如果同一长 URL 生成多次,**应该返回同一个短码**(避免存储浪费):

```sql
CREATE TABLE long_to_short (
    long_url_hash CHAR(32) PRIMARY KEY,  -- MD5
    short_code    VARCHAR(8),
    INDEX idx_short (short_code)
);
```

### 3.3 统计表(可选)

```sql
CREATE TABLE click_log (
    short_code VARCHAR(8),
    clicked_at TIMESTAMP,
    user_agent VARCHAR(500),
    ip         VARCHAR(45),
    referer    VARCHAR(500)
);
-- 这张表写量极大,通常进 Kafka → ClickHouse,不进 MySQL
```

## 四、架构演进

### 4.1 V1:单机能扛(DAU < 10 万)

```
Client → Nginx → Spring Boot → MySQL
```

**短码生成**:

```python
def generate_short_code():
    while True:
        code = random_string(6)
        if not db.exists(code):
            return code
```

**跳转**:

```python
def redirect(short_code):
    long_url = db.query("SELECT long_url FROM short_url WHERE short_code = ?", short_code)
    return redirect_302(long_url)
```

**问题**:

- 随机生成 + 检查重复:数据量大时**冲突率高**(生日悖论)
- 每次跳转都查 DB:**单 MySQL 撑死 1 万 QPS**

### 4.2 V2:加缓存(DAU 100 万)

```
Client → Nginx → Spring Boot → Redis(命中)→ 返回
                              ↓ miss
                              MySQL
```

**Redis 缓存所有热点短链**:

```python
def redirect(short_code):
    cached = redis.get(f"url:{short_code}")
    if cached:
        return redirect_302(cached)
    long_url = db.query(...)
    redis.setex(f"url:{short_code}", 86400, long_url)
    return redirect_302(long_url)
```

**命中率**:短链有强 80/20 分布(20% 短链承担 80% 流量),缓存命中率轻松 95%+。

**短码生成改用发号器**:见下文 §5。

### 4.3 V3:分库分表 + 全异步统计(DAU 1000 万)

```
Client → Nginx → Spring Boot
                      ↓
                 ShardingSphere(按 short_code 分 16 库)
                      ↓
                 MySQL × 16
                      ↓
                 Redis Cluster
                      ↓
                 Kafka → ClickHouse(统计)
```

**关键改动**:

- **按 short_code 哈希分 16 库**——单库 5 亿条,16 库总 80 亿条
- **统计写入 Kafka**,**不阻塞主跳转链路**
- **Redis Cluster 多节点**,扛 100 万 QPS

### 4.4 V4:全球 CDN 加速(DAU 1 亿+)

```
Client → CDN(就近接入)→ 边缘节点缓存热门短链
       ↓ miss
       源站(同上)
```

**短链特别适合 CDN**——内容是简单的 302 响应,缓存效果极佳:

```http
HTTP/1.1 302 Found
Location: https://example.com/very-long-url
Cache-Control: max-age=3600
```

**CDN 上一级缓存**:50 万 QPS 中 95% 在 CDN 层就被消化,源站只剩 2.5 万 QPS。

## 五、关键取舍:短码生成的三种方案

这是短链设计的**最核心决策**——选错就性能崩 / 安全崩。

### 5.1 方案 A:哈希(MD5 + 取前 6 位)

```python
def generate(long_url):
    h = md5(long_url + salt).hexdigest()
    return base62_encode(int(h[:8], 16))[:6]
```

**优点**:同一 URL 总是生成同一短码(天然去重)。
**缺点**:
- 不同 URL 哈希冲突率高(6 位空间 568 亿,1 万亿次后必冲突)
- 冲突要重试 + 加 salt,逻辑复杂
- **被攻击者枚举短码爆破**(知道哈希函数)

### 5.2 方案 B:发号器 + Base62(主流)

**思想**:全局自增 ID → 转 Base62 → 短码

```python
def generate(long_url):
    id = id_generator.next()       # 全局递增 ID,如 12345678
    short_code = base62_encode(id) # → "dnH7"
    db.insert(short_code, long_url)
    return short_code
```

**优点**:
- 不冲突(ID 唯一)
- 短码递增可预测——但加随机偏移可避免

**实现细节**:

```python
def base62_encode(num):
    chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
    result = []
    while num > 0:
        result.append(chars[num % 62])
        num //= 62
    return ''.join(reversed(result))

# 12345678 → "dnH7"
# 568 亿 → 6 位刚刚够
```

**ID 生成方式**:

| 方式 | 优劣 |
| --- | --- |
| MySQL 自增主键 | 简单,但单点瓶颈 |
| Redis INCR | 性能高,但 Redis 挂会丢 |
| 雪花算法 | 64 位,有时间戳,**Base62 后超过 6 位** |
| 号段模式(美团 Leaf) | **推荐**,本地缓存一段 ID,极快 |

> **生产推荐:号段模式 + Base62**——美团短链就是这么干的。

### 5.3 方案 C:预生成池

**思想**:离线批量生成 1 亿个短码进池,生成时从池里取一个用。

**优点**:生成 O(1),无任何冲突风险。
**缺点**:维护池子(池空了要补,补慢了影响业务)。

**适合**:生成 QPS 不高但要求极致延迟的场景。

### 5.4 三方案对比

| 方案 | 性能 | 唯一性 | 安全性 | 复杂度 |
| --- | --- | --- | --- | --- |
| 哈希 | 高 | 冲突高 | 可枚举 | 中 |
| 发号器(主流) | 高 | 唯一 | 加偏移可防枚举 | 中 |
| 预生成池 | 极高 | 唯一 | 不可预测 | 高 |

## 六、关键问题:跳转性能

短链 50% 的工程量**全在这一步**——50 万 QPS 跳转,延迟必须 < 10ms。

### 6.1 多级缓存

```
浏览器 (Cache-Control)        命中率 30%
    ↓
CDN                          命中率 70%
    ↓
反向代理 Nginx Cache         命中率 50%
    ↓
应用 Caffeine 本地缓存        命中率 80%
    ↓
Redis Cluster                命中率 99%
    ↓
MySQL(分库)                兜底,极少打到
```

**层层挡掉**:50 万 QPS 真正打到 MySQL 的不到 50。

### 6.2 跳转用 302 还是 301

| 状态码 | 浏览器缓存 | 适合 |
| --- | --- | --- |
| 301 | 永久缓存 | 永远不变的短链 |
| 302 | 不缓存 | **推荐** |

**为什么主流用 302**:

- 浏览器不缓存 → **每次都打到服务端 → 能统计点击量**
- 短链可能被改 / 删除 → 301 缓存了用户永远访问不到新地址

> Twitter / Bitly / 新浪微博全用 302——**统计需求 > 性能需求**。

### 6.3 防爬虫和恶意刷

```
单 IP 每秒最多跳转 100 次
单短链每秒被同一 IP 跳转 > 10 次 → 怀疑刷量,返回 429
明显爬虫 UA → 直接 403
```

详见 11 篇限流。

## 七、热点短链:爆款链接的特殊处理

某个短链突然爆火(明星微博、新闻头条),**单个 short_code 的 QPS 可能达到 10 万+**——超过单 Redis 节点的能力(10 万 QPS 上限)。

### 7.1 识别热点

实时监控 Redis 各 key 的 QPS,超过阈值的标为热点。

### 7.2 热点解决方案

#### A. 本地缓存(Caffeine)

热点 key 写入应用本地 Caffeine,**完全不打 Redis**。

```python
hot_key_cache = Caffeine.maximumSize(1000).expireAfterWrite(60s).build()
```

#### B. 多副本 Redis

热点 key 在多个 Redis 节点都存一份,客户端随机选一个读。

```python
node_index = random.randint(0, N-1)  # 随机打到 N 个副本
```

#### C. CDN 缓存

把热点短链的 302 响应直接缓存在 CDN——**根本不进源站**。

> **热点处理是大流量短链系统的"必答题"**——双 11 / 春晚红包链接,一个短链一秒打几十万,**没准备好就直接挂**。

## 八、自定义短链

用户希望生成 `t.co/my-blog` 而不是 `t.co/abcdef`:

```python
def custom_short(custom_code, long_url):
    if not is_valid(custom_code):  # 长度、字符、敏感词检查
        raise InvalidCustomCode
    if db.exists(custom_code):
        raise AlreadyTaken
    db.insert(custom_code, long_url)
```

**注意**:

- 防止占用系统保留字(`api`、`admin`、`login`)
- 防止冲突已有自动生成的短码(自动生成的从某个 ID 起,自定义只能用 ID 之前的范围)
- 自定义通常收费 / 限会员

## 九、统计点击量

每次跳转都要记录,**写入量等于跳转量**(50 万 QPS),**不能进 MySQL**。

### 9.1 标准方案

```
跳转处理:
  1. 拿到 long_url(从缓存)
  2. 异步发一条消息到 Kafka(非阻塞,< 1ms)
  3. 立即 302 返回

后台:
  Kafka → Flink 实时聚合 → ClickHouse(分钟级统计)
                       → Redis(秒级 Top 榜单)
```

### 9.2 实时点击 vs 异步统计

- **实时**:用户刚发短链马上看点击量 → 用 Redis HINCRBY 实时累加
- **离线**:每天 / 每小时统计来源、地理分布 → ClickHouse 聚合

## 十、安全考虑

### 10.1 防恶意短链

短链常被用于钓鱼、诈骗:

- **检测**:扫描 long_url 是否在黑名单 / 病毒库
- **拦截**:跳转前显示警告页(Twitter 的「这个链接可能不安全」)
- **接入第三方安全 API**(Google Safe Browsing)

### 10.2 防枚举攻击

发号器 + Base62 生成的短码是连续的——攻击者可以从 `aaaaaa` 枚举到 `zzzzzz`,扫出所有短链。

**对策**:**ID 加随机偏移**:

```python
def encode(id):
    obfuscated = (id + 0xABCDEF) ^ 0x123456  # 加偏移 + 异或
    return base62_encode(obfuscated)
```

或**直接用预生成池(打乱顺序)**。

### 10.3 防限流绕过

攻击者每秒生成 1 万短链 → 占用大量短码空间。**生成短链必须限流**(每用户每分钟最多生成 100 条)。

## 十一、最终架构图

```
                       ┌──── CDN ────┐
                       │  缓存 70%   │
Client ───────────────→│             │
   ↓                   └──────┬──────┘
   ↓ miss                      ↓
┌─────┐         ┌─────────────────┐
│ LB  │────────→│  Nginx Cache    │
└─────┘         └────────┬────────┘
                          ↓
                  ┌──────────────────┐
                  │  Spring Boot ×N  │
                  │  + Caffeine 本地 │
                  └────┬─────┬───────┘
                       ↓     ↓
              ┌────────┴┐  ┌─┴──────────┐
              │  Redis   │  │   Kafka    │
              │  Cluster │  │  (异步统计) │
              └────┬─────┘  └─────┬──────┘
                   ↓ miss          ↓
              ┌────┴──────┐    ┌───┴──────────┐
              │ MySQL × 16│    │ ClickHouse   │
              │ (分库)    │    │ (统计聚合)   │
              └───────────┘    └──────────────┘
```

**读路径**:CDN → Nginx → 本地缓存 → Redis → MySQL,层层挡掉。
**写路径**:号段发号 → 写 MySQL → 写 Redis → 异步同步 CDN。
**统计**:跳转时异步发 Kafka → Flink 聚合 → ClickHouse。

## 十三、踩坑提醒

1. **短码用纯哈希**——冲突重试逻辑复杂,生产用发号器
2. **不去重**——同一 URL 多次生成多个短码,浪费空间
3. **跳转用 301**——缓存了改不了,统计数据缺失
4. **不分库**——10 亿条记录单库扛不住
5. **统计写主库**——50 万写 QPS 直接把 MySQL 打挂
6. **没本地缓存**——Redis 仍是瓶颈,扛不住热点
7. **不做热点处理**——爆款链接打挂某个 Redis 节点
8. **不限流生成**——攻击者可以耗尽短码空间
9. **不防枚举**——发号器 + Base62 短码可被扫描
10. **没考虑全球 CDN**——海外用户跳转延迟 200ms+
