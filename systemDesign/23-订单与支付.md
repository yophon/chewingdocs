# 订单与支付

订单和支付是**电商系统的"骨头"**——商品、营销、推荐都是肉,但骨头错一个字节,公司就赔钱、被告、上社会新闻。新手以为「订单就是 INSERT 一行,支付就是调第三方」,**真实订单系统有 30+ 个状态、20+ 个状态转换、10+ 个外部回调,任何一个状态错了都意味着资金损失**。这一篇讲清楚两个核心:**订单状态机怎么设计才不漏**,**支付的幂等性 + 分布式事务怎么保证不重复扣钱**。

> 一句话先记住:**订单的核心是"状态机 + 幂等"**,支付的核心是"对账兜底"。**没有完美的分布式事务方案——TCC / Saga / 本地消息表都各有取舍**,选适合业务的;**对账是支付的最后防线**——再完美的代码也可能有 bug,**对账每天跑,自动发现差异、人工修正**。**任何金钱相关的接口都必须幂等**,不幂等 = 必出资损。

---

## 一、需求拆解

### 1.1 功能需求

- **下单**:用户提交订单(选商品 / 收货地址 / 优惠券)
- **支付**:跳转到支付平台,完成扣款
- **回调**:支付平台异步通知支付结果
- **超时取消**:30 分钟未支付自动取消
- **退款**:用户申请退款
- **对账**:每日和支付平台核对

### 1.2 非功能需求

- **一致性**:**金额绝对不能算错**(强一致)
- **幂等**:任何请求重试不能重复扣钱
- **可用性**:99.99%
- **规模**:DAU 1 亿,日单 1 亿,峰值下单 10 万 QPS

### 1.3 这一轮不做

- 推荐 / 营销
- 物流配送
- 财务核算 / 报表

## 三、订单状态机

订单的**所有状态和转换**必须先画清楚,才能写代码。

### 3.1 主要状态

```
[待支付] → [已支付] → [备货中] → [已发货] → [运输中] → [已签收] → [已完成]
   ↓          ↓
[已取消]   [退款中] → [已退款]
```

完整状态可能 20-30 个,**每个都要明确"上一个状态可以来,下一个状态可以去"**。

### 3.2 状态机表

```sql
-- 订单状态机:from_status → to_status 是否合法
CREATE TABLE order_state_transitions (
    from_status  TINYINT,
    to_status    TINYINT,
    PRIMARY KEY (from_status, to_status)
);

INSERT INTO order_state_transitions VALUES
    (1, 2),  -- 待支付 → 已支付
    (1, 5),  -- 待支付 → 已取消
    (2, 3),  -- 已支付 → 备货中
    (2, 6),  -- 已支付 → 退款中
    -- ...
;
```

### 3.3 状态变更必须校验

```python
def update_order_status(order_id, new_status):
    with db.transaction:
        order = db.select_for_update("SELECT status FROM order WHERE id=?", order_id)
        if not is_valid_transition(order.status, new_status):
            raise IllegalStateTransition(order.status, new_status)
        db.execute("UPDATE order SET status=? WHERE id=?", new_status, order_id)
        # 写状态变更日志(审计)
        db.execute("INSERT INTO order_log (order_id, from_status, to_status, time) VALUES (?, ?, ?, NOW())")
```

> **状态机不写校验 = 必出 bug**——某天有人写了"待支付 → 已发货"的代码,系统就出大事。

## 四、幂等:订单系统的命门

支付相关接口**必须幂等**——同一个请求执行 N 次,结果和执行 1 次完全相同。

### 4.1 幂等的核心:幂等键(Idempotency Key)

每个请求带一个**唯一 ID**(客户端生成):

```
POST /order/create
Idempotency-Key: 4f8b2a... (UUID, 客户端生成)

服务端:
  if redis.set(f"idem:{key}", 1, nx=True, ex=86400):
      # 第一次,正常处理
      result = create_order(...)
      redis.set(f"idem:{key}:result", result)
      return result
  else:
      # 重复请求,返回上次结果
      return redis.get(f"idem:{key}:result")
```

### 4.2 不同操作的幂等设计

| 操作 | 幂等手段 |
| --- | --- |
| 创建订单 | 客户端 Idempotency-Key |
| 扣库存 | order_id 幂等(同一订单只扣一次) |
| 扣余额 | order_id 幂等 |
| 支付回调 | trade_no 幂等(支付平台单号) |
| 状态变更 | 校验 from_status,只允许合法迁移 |

### 4.3 不幂等会怎样

```
用户点提交 → 网络抖动 → 客户端重试 → 后端创建 2 个订单
扣库存接口被调 2 次 → 库存扣 2 次
支付回调收到 2 次 → 用户余额加 2 次

→ 资损或客诉,严重事故
```

> **2026 年的所有支付 API(支付宝、微信支付、Stripe、PayPal)全部要求幂等键**——这是行业铁律。

## 五、分布式事务:下单的最难问题

下单 = 多个动作:

```
1. 减库存(Inventory Service)
2. 创建订单(Order Service)
3. 扣余额 / 创建支付单(Payment Service)
4. 增积分(Reward Service)
```

**这几个动作分布在不同服务、不同 DB,必须保证"全成功或全失败"** —— 这就是分布式事务。

### 5.1 错误方式:不管事务

```python
def create_order(...):
    inventory_service.reduce_stock(product_id, qty)  # 减库存
    order_service.create(...)                         # 写订单
    payment_service.create_charge(...)                # 创支付
```

**任何一步失败,前面的不回滚** → 数据不一致(库存少了订单没生成)。

### 5.2 方案 A:2PC / XA(几乎不用)

强一致分布式事务,**所有参与方都要支持 XA 协议**:

```
协调者 → 通知所有参与方 "Prepare"
参与方都 OK → 协调者 "Commit"
任一失败 → "Rollback"
```

**问题**:

- 同步阻塞,慢
- 协调者单点
- 锁持有时间长
- 跨服务支持差

> **互联网业务几乎不用 XA**——慢、脆弱、扩展性差。**只有传统金融核心还在用**。

### 5.3 方案 B:Saga(主流)

**思想**:每个步骤一个本地事务,**失败时通过反向操作回滚**。

```
正向:减库存 → 创订单 → 扣余额 → 加积分
反向:加积分↩ ← 扣余额↩ ← 创订单↩ ← 加库存↩
```

```python
def create_order_saga(...):
    try:
        inventory_service.reduce_stock(product_id, qty)
        try:
            order_service.create(order_data)
            try:
                payment_service.create_charge(amount)
            except:
                order_service.cancel(order_id)  # 反向
                raise
        except:
            inventory_service.add_stock(product_id, qty)  # 反向
            raise
    except:
        return error
```

**实战用编排服务**(Saga Orchestrator)管理流程,失败自动回滚。

**优势**:

- 异步、不阻塞
- 跨服务友好
- 性能高

**劣势**:

- **最终一致**(中间状态可能短暂不一致)
- 反向操作要小心(已扣的钱怎么"反向"——补偿一笔退款)

### 5.4 方案 C:TCC(Try-Confirm-Cancel)

每个服务实现三个接口:

- **Try**:预留资源(库存预扣、余额冻结)
- **Confirm**:正式提交
- **Cancel**:释放预留

```
Try 阶段:
  Inventory.try_reduce(qty)   → 库存"已预扣"(标记为冻结)
  Payment.try_freeze(amount)  → 余额"冻结"
  → 都成功才进 Confirm

Confirm 阶段:
  Inventory.confirm(qty)      → 真正扣减(冻结 → 真实减少)
  Payment.confirm(amount)     → 真正扣款

Cancel 阶段(任一 Try 失败):
  Inventory.cancel(qty)       → 释放冻结
  Payment.cancel(amount)      → 解冻
```

**优势**:**业务侧的"两阶段提交"** —— 比 XA 灵活、性能高。
**劣势**:每个服务要实现 3 个接口,**改造成本大**。

### 5.5 方案 D:本地消息表(最常用,推荐)

**思想**:**每个服务的"本地事务" + "可靠消息发送"**:

```python
def create_order_with_outbox(...):
    with db.transaction:
        db.execute("INSERT INTO order ...")
        db.execute("INSERT INTO outbox (msg, status) VALUES ('OrderCreated', 'pending')")
    # 单独 Worker 轮询 outbox,发到 MQ,成功后改 sent

# 库存服务消费 OrderCreated 消息
def on_order_created(msg):
    if not redis.set(f"processed:{msg.id}", 1, nx=True):
        return  # 幂等
    inventory_service.reduce_stock(msg.product_id, msg.qty)
```

**优势**:

- 简单可靠
- 本地事务保证 outbox 和业务表一致
- 异步、性能好

**劣势**:

- 多一张 outbox 表
- 最终一致(消费有延迟)

> **本地消息表是中国互联网公司的事实标准**——简单、可靠、可观测。

### 5.6 四方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适合 |
| --- | --- | --- | --- | --- |
| 2PC / XA | 强一致 | 慢 | 高 | 几乎不用 |
| Saga | 最终一致 | 高 | 中 | 长流程业务 |
| TCC | 接近强一致 | 高 | 高 | 金融场景 |
| 本地消息表 | 最终一致 | 高 | 低 | **互联网通用** |

## 六、支付流程详解

支付是订单系统**最容易出错的部分**——涉及外部第三方,异步、不可控。

### 6.1 标准流程

```
1. 用户在订单页点"立即支付"
2. 后端创建支付单,调用第三方(支付宝/微信)
3. 第三方返回支付 URL / 二维码
4. 用户在支付平台完成支付
5. 支付平台异步回调商户(notify_url)
6. 商户校验回调签名 → 更新订单状态 → 返回 SUCCESS
7. 商户主动查询(轮询)兜底未收到回调的订单
```

### 6.2 关键设计点

#### A. 支付回调的幂等

```python
def payment_callback(trade_no, status, sign):
    # 1. 校验签名
    if not verify_sign(sign):
        return "FAIL"
    
    # 2. 幂等(同一笔单子可能回调多次)
    if redis.set(f"callback:{trade_no}", 1, nx=True, ex=86400) is None:
        return "SUCCESS"  # 已处理过,直接返回成功
    
    # 3. 更新订单状态(校验状态机)
    update_order_status(order_id, "已支付")
    
    return "SUCCESS"
```

#### B. 主动轮询兜底

回调可能丢(网络问题、商户宕机)——**必须主动查询**:

```python
# 定时任务:5 分钟扫一次"已发起支付但状态未更新"的订单
def poll_payment_status():
    pending_orders = db.query("SELECT * FROM order WHERE status='待支付' AND payment_initiated_at < NOW() - INTERVAL 5 MINUTE")
    for order in pending_orders:
        result = third_party.query(order.trade_no)
        if result.status == "SUCCESS":
            update_order_status(order.id, "已支付")
```

#### C. 超时自动取消

```python
# 30 分钟未支付的订单 → 取消 + 释放库存
def timeout_cancel():
    expired = db.query("SELECT * FROM order WHERE status='待支付' AND created_at < NOW() - INTERVAL 30 MINUTE")
    for order in expired:
        cancel_order(order.id)
        inventory_service.return_stock(order.product_id, order.qty)
```

## 七、对账:支付的最后防线

**再完善的代码也可能有 bug**——**对账是兜底**。

### 7.1 对账目标

每天和支付平台核对:

- 商户记录的支付成功订单 ↔ 支付平台记录的扣款
- 找出**长 / 短**(我说收到 100 块,平台说扣了 99.99 → 差 0.01,要查)

### 7.2 对账流程

```
T+1 凌晨 2 点:
  1. 拉取支付平台的对账文件(昨天的所有交易明细)
  2. 拉取本系统的支付流水
  3. 双向比对(平台有但系统没 / 系统有但平台没 / 金额不一致)
  4. 输出"差异报告"
  5. 自动修正小差异 / 人工处理大差异
```

### 7.3 对账常见差异

| 差异类型 | 原因 | 处理 |
| --- | --- | --- |
| 平台有,系统没 | 回调丢,系统漏单 | 自动补单(给用户加 / 退款) |
| 系统有,平台没 | 系统假成功(bug) | 立即下单失败,修代码 |
| 金额不一致 | 计算错 / 优惠券错 | 人工核实 |
| 状态不一致 | 异步延迟 | 等 T+2 再对 |

> **没有对账系统的支付不是合规的支付**——监管要求,公司要求,自我兜底。

## 八、退款

退款比支付简单——但**也容易出错**:

### 8.1 退款幂等

```python
def refund(order_id, amount):
    if not redis.set(f"refund:{order_id}", 1, nx=True):
        return "已退款"
    
    third_party.refund(order.trade_no, amount)
    update_order_status(order_id, "已退款")
```

### 8.2 部分退款 vs 全额退款

部分退款:多次退款累积不能超过支付金额。
全额退款:一次性退完。

### 8.3 退款异步

退款也是异步的——**第三方处理需要时间**(几分钟到几天):

```
申请退款 → 状态"退款中"
第三方处理 → 回调"退款成功"
更新状态"已退款"
```

## 九、订单查询:读多写少的优化

订单查询**比写入大几十倍**——用户随时查"我的订单"。

### 9.1 多维度查询

- 用户查自己的订单(按 user_id)
- 商家查自己收到的订单(按 seller_id)
- 客服按订单号 / 手机号 / 收货地址查
- 财务按时间范围统计

**单 DB 解决不了**——必须**多份数据**:

| 维度 | 存储 |
| --- | --- |
| 用户维度 | MySQL 按 user_id 分库 |
| 商家维度 | MySQL 按 seller_id 分库(冗余一份) |
| 全维度搜索 | ES 索引 |
| 财务报表 | 数仓 / ClickHouse |

### 9.2 数据同步

主表写入 → 通过 CDC(binlog 监听)→ 同步到所有副本。
详见 backendLearning/46。

## 十、最终架构图

```
                       用户
                         ↓
                    订单服务
                         ↓
       ┌─────────────────┼─────────────────┐
       ↓                 ↓                 ↓
  库存服务            订单服务          支付服务
   (TCC try)       (本地消息表)        (创建支付单)
       ↓                 ↓                 ↓
       MQ(异步通知 / Saga 协调)
                         ↓
            ┌────────────┼────────────┐
            ↓            ↓            ↓
        奖励服务      推荐服务      物流服务
        (积分)       (行为)       (备货)

支付链路:
  支付服务 ↔ 第三方支付平台
        ↓ 回调 + 主动查询
        ↓ 更新订单状态

对账链路:
  T+1 凌晨拉对账文件 → 比对 → 修正
```

## 十一、踩坑提醒

1. **不画状态机就写代码**——状态错乱必出资损
2. **不做幂等**——重复扣款 / 重复发货
3. **用 XA 做分布式事务**——慢、脆弱
4. **不做 outbox 模式**——MQ 发送失败时数据不一致
5. **支付回调不验签**——伪造回调骗取系统
6. **没主动查询兜底**——回调丢就漏单
7. **没对账**——bug 累积,资损长期不发现
8. **超时不取消**——库存被永久锁住,影响销售
9. **退款不幂等**——一笔单退两次
10. **订单不分库**——数据爆炸,查询慢
11. **多维度查询不冗余**——商家查询慢死
12. **状态变更不写日志**——出事查不到原因
