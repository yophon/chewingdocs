# 灰度发布与蓝绿

发布是**生产事故的 90% 诱因**——业务好好的,你一发版就崩。**控制发版风险 = 控制系统稳定性**。新手以为发布就是"git push + restart",真实工程里发布是一整套**降低爆炸半径**的艺术——蓝绿、滚动、金丝雀、影子流量,每种发布方式都有特定场景和代价。这一篇讲清楚:**几种发布模式的选择**、**自动回滚机制**、以及为什么"全量发布"在 2026 年的互联网公司基本绝迹。

> 一句话先记住:**发布的核心是"降低爆炸半径"**——出错时影响越少用户越好。**蓝绿快但贵 1 倍硬件,滚动便宜但慢,金丝雀灰度精确但复杂,影子流量验证安全但实现难**。**自动回滚比快速发布更重要**——发版后 5 分钟内监控异常自动切回旧版,**人工反应永远比机器慢**。

---

## 一、为什么发版是事故源

### 1.1 真实数据

```
据 Google SRE Book 统计:
  生产事故的 70% - 90% 由"变更"引起
  变更包括:发版、配置改动、运维操作

具体细分:
  代码 bug:50%
  配置错误:30%
  依赖变化:10%
  其他:10%
```

> **不发版几乎不出事**——这就是为什么金融行业敢做到 5 个 9。互联网快速迭代必然提高出事概率。

### 1.2 减少风险的两条路

```
减少出错概率:测试、Code Review、CI/CD、自动化
减少出错影响:灰度、回滚、降级、限制爆炸半径
```

**前者投入大但收益边际递减,后者性价比更高**——业内焦点在后者。

---

## 二、几种发布模式

### 2.1 全量发布(Big Bang)

**所有实例一次性切到新版本**:

```
v1: instance1, instance2, instance3, ..., instance100
v2 发布:
  100 个实例同时停 → 同时启动 v2
```

**优点**:简单。
**缺点**:

- 出错影响 100% 用户
- 切换瞬间服务中断
- 新版本有 bug 立刻暴露给所有人

> **2026 年互联网公司基本不用全量发布了** —— 风险太大。**只有内部工具或小型项目还在用**。

### 2.2 滚动发布(Rolling)

**一批一批替换实例**:

```
100 个实例
每次替换 10 个 → 等健康检查通过 → 再换下 10 个
完整发布需要 10 轮 × 1 分钟/轮 = 10 分钟
```

**优点**:

- 不需要额外硬件
- 服务不中断
- 简单(K8s 默认就是这种)

**缺点**:

- 发布期间新老版本共存(API 兼容性问题)
- 回滚也是滚动的(慢)
- 如果新版本有 bug,**前 10% 用户已经中招**

### 2.3 蓝绿发布(Blue-Green)

**部署两套完全独立的环境**:

```
蓝(Blue):  v1, 100 个实例,接 100% 流量
绿(Green): v2, 100 个实例,准备好但不接流量

发布:
  Green 全部启动 → 监控验证 OK → 流量从 Blue 100% 切到 Green 100%
  
回滚:
  发现问题 → 流量秒级切回 Blue
```

**优点**:

- 切换是原子的(瞬间生效)
- 回滚极快(切流即可)
- 可以提前压测 Green 环境

**缺点**:

- **需要双倍硬件**(平时一半闲置)
- 数据库 schema 必须向后兼容(Blue/Green 共用 DB)

> **关键业务首选蓝绿** —— 成本翻倍换"秒级回滚",值。

### 2.4 金丝雀发布(Canary,即灰度)

**先放一小部分流量到新版本,观察后逐步放大**:

```
1% 流量 → v2(几个实例) → 监控 30 分钟
10% 流量 → v2 → 监控 1 小时
50% 流量 → v2 → 监控 2 小时
100% 流量 → v2 → 完成
```

**金丝雀名字由来**:矿工带金丝雀下矿,毒气来了金丝雀先死,矿工撤离。**新版本的少量流量就是"金丝雀"**。

**优点**:

- 出错只影响少量用户
- 可以观察新版本真实行为
- 业内最主流

**缺点**:

- 实现复杂(流量切分、灰度配置、监控)
- 时间长(完整发布几小时甚至几天)

### 2.5 影子流量(Shadow Traffic)

**真实流量复制一份发给新版本,但不返回给用户**:

```
请求 → 路由层
        ├── 转发到 v1(返回结果给用户)
        └── 复制一份转发到 v2(返回结果丢弃)

用户感知到的是 v1 的结果
v2 在真实流量下被"演练"
```

**优点**:

- 在不影响用户的情况下验证新版本
- 可以发现"小流量灰度发现不了"的问题(并发、性能)

**缺点**:

- 实现极复杂(路由层 + 数据隔离)
- **对外有副作用的请求不能复制**(发短信、扣款)
- 双倍计算成本

> 影子流量适合**核心交易系统的重大重构**——比如订单服务大版本升级。

---

## 三、发布模式的选择矩阵

| 场景 | 推荐 |
| --- | --- |
| 内部工具 / 小项目 | 滚动 / 全量 |
| 普通业务 | 滚动 |
| 关键业务 | 蓝绿 |
| 用户基数大 | 金丝雀 |
| 大规模重构 | 金丝雀 + 影子流量 |
| 数据库 schema 变更 | 多步骤(向后兼容 + 双写 + 切换) |

**实际**:**多种模式组合**——同一服务的不同模块用不同发布方式。

## 四、灰度发布的工程实现

### 4.1 流量切分的几种方式

#### A. 按用户哈希

```python
if hash(user_id) % 100 < canary_percent:
    route_to_v2()
else:
    route_to_v1()
```

`canary_percent` 通过配置中心动态调整。

#### B. 按白名单

```python
if user_id in canary_user_ids:
    route_to_v2()
```

**先内部员工 / Beta 用户**——出错容忍度高。

#### C. 按地理 / 设备

```python
if user.region == "shanghai" or user.device == "ios":
    route_to_v2()
```

**先小范围**,扩大爆炸半径前确认。

#### D. 按 cookie / Header

```nginx
if ($cookie_canary = "true") {
    proxy_pass http://v2;
}
```

QA 测试 / 客服自查 / 主动选择"尝鲜"用户。

### 4.2 在哪一层做切分

```
DNS 层:粗粒度,按 IP 段
LB 层:中粒度,按 IP/Cookie
Gateway 层:细粒度,按 user_id/请求头
应用层:最细,完全可控
```

**主流:Gateway 层 + 配置中心动态调灰度比例**。

## 五、自动回滚:发布最重要的能力

**人工反应永远慢**——必须自动监控 + 自动回滚。

### 5.1 触发回滚的指标

```
错误率 > 1%(对比基线)→ 自动回滚
p99 延迟 > 平时 2 倍 → 自动回滚
CPU > 90% 持续 5 分钟 → 自动回滚
内存泄漏(持续上涨)→ 自动回滚
异常日志暴涨 → 自动回滚
业务指标下降(GMV / 转化率)→ 报警 + 人工确认回滚
```

### 5.2 回滚动作

```python
# 蓝绿:切流
def rollback_blue_green():
    config_center.set("active_env", "blue")
    # 几秒内所有 LB 切回 Blue

# 滚动:重新滚回旧版本
def rollback_rolling():
    k8s.set_image(deployment, "v1")
    # 边滚边替换,几分钟完成

# 金丝雀:把灰度比例改回 0
def rollback_canary():
    config_center.set("canary_percent", 0)
    # 立即生效
```

### 5.3 回滚的"双保险"

```
1. 自动回滚(监控触发)
2. 人工一键回滚(发版页面按钮)
3. 配置中心兜底(直接改配置)
```

**任一手段都能在 1 分钟内生效**。

## 六、数据库 schema 的发布

DB 改动**不能像应用一样灰度**——一旦执行就影响所有实例。

### 6.1 向后兼容原则

```
新版本应用 ←→ DB 新 schema
              ↑
              ↓
旧版本应用 ←→ 同一 DB(必须能用)
```

**关键**:**DDL 不能破坏旧应用**:

- 加列:OK(旧应用忽略新列)
- 删列:**先双写,等所有应用升级后再删**
- 改字段类型:**新建字段 + 双写 + 迁移 + 删旧字段**

### 6.2 Migration 脚本

每个 schema 改动一个版本号:

```sql
-- v001__create_user_table.sql
CREATE TABLE user (...);

-- v002__add_email_column.sql
ALTER TABLE user ADD COLUMN email VARCHAR(128);

-- v003__add_email_index.sql
CREATE INDEX idx_email ON user(email);
```

工具:**Flyway / Liquibase**——自动执行未跑过的 migration。

### 6.3 大表 DDL

直接 `ALTER TABLE` 锁表 → 业务挂。

**用 pt-online-schema-change / gh-ost**:

```
1. 创建影子表(新 schema)
2. 触发器同步老表写入到影子表
3. 历史数据从老表复制到影子表
4. 切换:rename 影子表为正式表,老表 rename 备份
5. 删除备份(几天后)
```

**整个过程不锁表**——业务无感。

## 七、发布的"前置 + 后置"

发布不只是"部署代码"——还有前后流程:

### 7.1 前置

```
□ 单元测试通过
□ 集成测试通过
□ Code Review 通过
□ 性能基准测试通过
□ 安全扫描通过
□ DBA 评审(如果有 DB 改动)
□ 通知相关团队(下游依赖)
□ 准备回滚方案
□ 选择发布时间(避开高峰)
```

### 7.2 后置

```
□ 健康检查通过
□ 监控指标正常(15 分钟)
□ 烟雾测试(关键场景人工验证)
□ 业务方确认无异常
□ 发布报告写入文档
```

## 八、发布的窗口期

**什么时候发版** 也是策略:

### 8.1 推荐时间

```
工作日早上 10 点 - 11 点:团队全员在线,出事能快速响应
避开:
  - 高峰期(午饭、晚高峰)
  - 周五下午(发完没人值班)
  - 节假日前一天(出事得加班)
```

### 8.2 发布频率

```
互联网公司:每天 1-N 次(微服务化,每个服务自己发)
金融:每周 1-2 次,严控时间
银行核心:每季度 1 次,夜间窗口
```

**频繁发布 + 灰度** > **不发布 + 大版本** —— **小步快跑**比"积累几个月一起发"安全得多。

## 九、配置发布:被严重低估的风险

**改一个配置 = 一次发布**——但常常被当成"小事"。

### 9.1 配置事故案例

- **2017 年 AWS S3 故障**:运维敲错命令,**几分钟波及全球**
- **某厂 2020**:把超时时间从 3s 改成 30s,引发雪崩
- **某金融 2022**:开关错配,误把测试流量打到生产

### 9.2 配置发布的最佳实践

```
□ 配置版本化(git 管理 / 配置中心带历史)
□ 灰度推送(先 1% 实例)
□ 监控生效后效果
□ 一键回滚到上一版本
□ Code Review(配置改动也要审查)
□ 审计日志(谁、何时、改了什么)
```

> 配置中心(Nacos / Apollo / 阿里 Diamond)**必备的"灰度推送 + 一键回滚"** —— 这是配置安全的最低底线。

## 十、发布系统的能力矩阵

成熟的发布系统应该具备:

```
□ CI / CD 集成(代码提交自动触发)
□ 多环境(dev / test / staging / prod)
□ 多种发布模式(滚动 / 蓝绿 / 金丝雀)
□ 灰度配置(按用户 / 地域 / 设备)
□ 自动健康检查
□ 自动回滚
□ 一键回滚
□ 发布历史 / 审计
□ 发布日历(避免冲突)
□ 多人审批
□ 数据库 migration 集成
□ 配置发布集成
□ 通知(IM / 邮件)
```

**Spinnaker / Argo CD / GitLab CI / Jenkins X** 都覆盖大部分能力。

## 十一、最终架构示意

```
代码提交 → CI(测试 / 构建)→ 镜像仓库
                                ↓
                          发布平台
                                ↓
          ┌─────────────────────┼─────────────────────┐
          ↓                     ↓                     ↓
       Dev 环境             Staging 环境          Prod 环境
       (自动部署)           (自动 + QA)         (人工触发)
                                                       ↓
                                              选择发布模式
                                                ↓
                                            金丝雀 / 蓝绿 / 滚动
                                                ↓
                                            监控 + 自动回滚
                                                ↓
                                            发布完成
```

## 十二、踩坑提醒

1. **全量发布大型业务**——一次发布事故影响 100% 用户
2. **不灰度**——发布即赌博
3. **没有自动回滚**——出事靠人工反应,慢
4. **金丝雀比例错**——1% 灰度阶段就持续 5 分钟,看不到问题
5. **DB schema 不向后兼容**——发布期间新老应用并存,旧应用挂
6. **大表 ALTER 不用 gh-ost**——锁表几小时,业务挂
7. **配置改了不灰度**——一改全挂
8. **配置不留版本**——回滚不了
9. **不监控业务指标**——只看 CPU / 错误率,业务转化掉了不知道
10. **周五下午 5 点发版**——出事没人值班
11. **多个发版同时进行**——出错时不知道是哪个版本的问题
12. **回滚没演练**——回滚时发现回滚脚本错的
