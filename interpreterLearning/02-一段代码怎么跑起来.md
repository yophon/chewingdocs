# 一段代码怎么跑起来

```text
print(1 + 2)
```

12 个字符,屏幕上出现一个 `3`。**这中间发生了什么?**字符串要先变 token,token 要变 AST,AST 要么直接被 evaluator 走一遍(树遍历),要么先被 compiler 翻成字节码再扔给 VM 跑——同一份语义,**两条物理路线,四种数据形态**。CPython 跑你 `print(1 + 2)` 模一样:`.py` → tokens → AST → bytecode → `Python/ceval.c` 里那条 `BINARY_OP` 指令;V8 跑 JS 也是同一条:source → AST → Ignition bytecode → TurboFan 优化机器码。**这条路是动态语言的通用骨架**——画清楚一次,30 篇每篇都在填它的某一段。

> 一句话先记住:**`源码 → tokens → AST →(求值 OR 字节码 → VM)→ 结果`**——这五个站点贯穿全系列,**任何一门动态语言都能往这条路上对号入座**,差别只在每一站的数据结构选择和优化深度。

---

## 一、目标:把 `print(1 + 2)` 跑出 3

输入 `"print(1 + 2)"`,输出 `3`。这两端之间要画清楚**四种中间形态**——tokens 数组、AST 节点树、字节码 + 常量池、运行时栈状态。**直观看「数据在变什么」,比一万字定义都管用**。

源码进入解释器时只是字符串——**这一层没有"数字 1"也没有"加法",只有字符 `'1'`、`'+'`、`'('`**。位置追踪(`line` / `col`)从这步就要做好,**所有错误信息的源头都是 lexer 里那两个变量**,Mochi 第一天就把它们塞进每个 token。

---

## 二、第一步:字符流变 tokens(扁平数组)

把字符流扔给 lexer,出来一个 token 数组:

```ts
type Token =
  | { kind: "IDENT",  value: "print", line: 1, col: 1 }
  | { kind: "LPAREN",                 line: 1, col: 6 }
  | { kind: "NUMBER", value: 1,       line: 1, col: 7 }
  | { kind: "PLUS",                   line: 1, col: 9 }
  | { kind: "NUMBER", value: 2,       line: 1, col: 11 }
  | { kind: "RPAREN",                 line: 1, col: 12 }
  | { kind: "EOF",                    line: 1, col: 13 }
```

**结构特征**:**扁平、顺序、带位置**,没有嵌套——这是和 AST 最大的差别。Lexer 的本质是**一个状态机**:读一个字符,根据当前状态决定是否产出 token、转到哪个状态。**5 行核心循环就能写出来**(详情 05 篇)。工业级 lexer——V8 `src/parsing/scanner.cc`、CPython `Parser/tokenizer.c`——也是手写状态机,**没人用 lex/flex 写主流语言**(错误恢复和性能不够灵活,06 篇详谈)。

---

## 三、第二步:tokens 变 AST(嵌套树)

tokens 是「单词」,AST 是「句子结构」。Parser 把扁平数组**变成嵌套的语法树**——`print(1 + 2)` 的 AST 及其 TS 数据结构:

```
CallExpr
  ├─ callee: Identifier("print")
  └─ args: [
       BinaryExpr(op: "+")
         ├─ left:  NumLit(1)
         └─ right: NumLit(2)
     ]
```

```ts
type Expr =
  | { kind: "NumLit",     value: number }
  | { kind: "Identifier", name: string }
  | { kind: "BinaryExpr", op: "+" | "-" | "*" | "/", left: Expr, right: Expr }
  | { kind: "CallExpr",   callee: Expr, args: Expr[] }
```

**关键点**:`Expr` 是一个 **sum type / discriminated union**——AST 节点天生就是代数数据类型,**这是 10 篇会反复强调的设计原则:用 sum type + switch,别用 OOP 重载**。**运算优先级和结合性已经体现在嵌套结构里**——`1 + 2 * 3` 在 AST 里 `*` 一定在 `+` 下面,**走树时自然就先算 `*`**。这就是为什么 AST 比 token 数组「更接近语义」。

---

## 四、路线 A:树遍历求值

最直接的执行方式——**写一个 `eval(node, env)` 函数,递归走 AST**:

```ts
function eval(node: Expr, env: Env): Value {
  switch (node.kind) {
    case "NumLit":     return node.value
    case "Identifier": return env.get(node.name)
    case "BinaryExpr": {
      const l = eval(node.left, env), r = eval(node.right, env)
      if (node.op === "+") return l + r
      // ... - * /
    }
    case "CallExpr": {
      const fn = eval(node.callee, env)
      return fn(...node.args.map(a => eval(a, env)))
    }
  }
}
```

对 `print(1 + 2)` 的执行轨迹:

```
eval(CallExpr)
 ├─ eval(Identifier "print")  → <native fn print>
 ├─ eval(BinaryExpr "+")
 │   ├─ eval(NumLit 1)        → 1
 │   ├─ eval(NumLit 2)        → 2
 │   └─ return 1 + 2          → 3
 └─ call print(3)             → 屏幕输出 "3"
```

**这 20 行 eval 函数,就是 Mochi 01-19 篇的执行引擎雏形**。Crafting Interpreters 的 jlox 也是这条路——简单,但慢:**每次执行都要重新遍历 AST,每个节点都要做一次 `switch (kind)` 派发,缓存极不友好**。

---

## 五、路线 B:编译到字节码 + VM

把 AST 编译成扁平的指令序列,运行时只走一个紧凑的循环——**字节码 VM 的核心思路**。`print(1 + 2)` 编译后(常量池 `[print, 1, 2]`,栈式 VM):

```
PC  指令          栈                  说明
0   GET_GLOBAL 0  [print]            取 print 压栈
1   CONST 1       [print, 1]         压常量 1
2   CONST 2       [print, 1, 2]      压常量 2
3   ADD           [print, 3]         1 + 2 → 3
4   CALL 1        []                 调用,输出 "3"
5   RETURN        -                  结束
```

VM 核心是**一个无聊的循环**:

```ts
while (true) {
  const op = code[pc++]
  switch (op) {
    case OP_CONST:  stack.push(constants[code[pc++]]); break
    case OP_ADD:    { const b = stack.pop(); const a = stack.pop(); stack.push(a + b); break }
    case OP_CALL:   /* ... */ break
    case OP_RETURN: return
  }
}
```

CPython `Python/ceval.c` 主循环就是这个结构,**一万多行 switch 都是各种 opcode 的实现**;Lua `lvm.c::luaV_execute` 同理。20-25 篇把 Mochi VM 写出来。

**为什么 B 比 A 快**:字节码是连续 `Uint8Array`,**cache 友好**;派发开销低(switch on byte 远快于 switch on string);可做编译期优化(常量折叠、跳转回填)。

## 六、对照:CPython / V8 / Lua 走的是同一条路

| 阶段 | Mochi | CPython | V8 | Lua |
|---|---|---|---|---|
| Lexer | `src/lexer.ts` | `Parser/tokenizer.c` | `src/parsing/scanner.cc` | `llex.c` |
| Parser | `src/parser.ts` | `Parser/parser.c` (PEG) | `src/parsing/parser.cc` | `lparser.c` |
| AST | TS sum type | `Python/ast.c` 的 `PyObject` | `src/ast/ast.h` | 无(单遍直出字节码) |
| 编译 | `src/compiler.ts`(20+) | `Python/compile.c` → `.pyc` | `bytecode-generator.cc` | `lparser.c` |
| VM | `src/vm.ts`(22+) | `Python/ceval.c` 大 switch | Ignition `interpreter.cc` | `lvm.c::luaV_execute` |
| JIT | 不做 | 3.13 specializing interpreter | TurboFan / Maglev / Sparkplug | 主版本不做,LuaJIT 单独项目 |

**结论**:Mochi 不是凭空设计——**它就是 CPython / V8 砍掉 JIT 和优化后剩下的最小骨架**。读完本系列,你打开 `ceval.c` 应该能直接对号入座:**"哦,这就是我那个 VM 循环,只是多了 200 个 opcode 和一堆 specialization"**。
