# 寄存器式 VM

栈式 VM 是教科书的开端,但**真正在生产里被很多语言选中的反而是寄存器式**——Lua 5.0+、Android Dalvik、.NET CoreCLR、Erlang BEAM、V8 Ignition(半个)全是。**为什么?同样一段代码,寄存器式字节码指令数能少 30-40%、dispatch 次数少 30%、解释速度快 30%**——Roberto Ierusalimschy 团队 2005 年的论文有数据。栈式赢在简洁、编译器好写;寄存器式赢在性能、读起来像汇编。**这一篇做对比,不写实现**——本系列 Mochi 走栈式路线,但你必须懂这个分叉。

> 一句话先记住:**寄存器式 = 字节码直接编号"虚拟寄存器"(其实是数组下标)、每条指令带 3 个操作数**——指令更长、总数更少、dispatch 次数少,**Lua / Dalvik 选这条因为快 30%**,代价是编译器要做寄存器分配。

---

## 一、寄存器式是什么

**寄存器式(register-based)VM**:每条指令直接指定**操作数从哪儿读、结果写哪儿**——读写位置是「虚拟寄存器编号」,**实际上就是当前函数局部变量数组的下标**。

```
ADD R3 R1 R2     ; R3 = R1 + R2
MOVE R0 R3       ; R0 = R3
```

没有表达式栈这个东西,所有中间值都落在"寄存器"(局部变量 slot)上。「虚拟」是因为它不是 CPU 的物理寄存器——VM 跑起来这些 R0/R1/R2 其实是 `locals[0]`、`locals[1]`、`locals[2]`,编码上当寄存器用。

---

## 二、对比:`a = b + c` 两种字节码

栈式(Mochi / JVM 风格):

```
LOAD_LOCAL b     ; 4 条指令,3 次 push,1 次 pop
LOAD_LOCAL c
ADD
STORE_LOCAL a
```

寄存器式(Lua 风格):

```
ADD Ra Rb Rc     ; 1 条指令,0 次栈操作
```

差异本质:

| 维度 | 栈式 | 寄存器式 |
|------|------|----------|
| 单条指令长度 | 1-2 字节 | 4 字节(op + 3 操作数编号) |
| 指令总数 | 多 | 少 30-40% |
| dispatch 次数 | 多 | 少 30% |
| 总字节数 | 接近 | 接近(指令少但每条长) |
| 编译器复杂度 | 简单(一遍递归 emit) | 复杂(要做寄存器分配) |
| 字节码验证 | 容易(栈高度静态可算) | 难 |

**寄存器式不是"用更少字节存程序",而是"用更少 dispatch 跑完程序"**——dispatch 是解释器最贵的一步,**少 30% 就是真的快 30%**。

## 三、谁选什么

```
栈式                          寄存器式
─────────────────────────     ───────────────────────────
JVM (javac)                   Lua 5.0+
CPython                       Dalvik (Android)
WASM                          Android Runtime (ART)
.NET CLR (IL 层)              .NET CoreCLR (JIT 后)
V8 早期 Crankshaft            V8 Ignition (栈 + accumulator,折中)
PHP Zend Engine               Erlang BEAM
Smalltalk                     LuaJIT
```

**特别注意 V8 Ignition 这个折中**:名义上栈式,但有一个特殊的 **accumulator 寄存器**——上一条指令的结果默认留在 accumulator 里,下一条直接用,省 push/pop。**接近寄存器式的性能 + 栈式的编译器简洁**——V8 团队 2016 年的设计,叫 "register-machine with accumulator"。

---

## 四、为什么 Lua 选寄存器式

Roberto Ierusalimschy(Lua 作者)团队 2005 年论文《The Implementation of Lua 5.0》:Lua 4.0 栈式换成 5.0 寄存器式后,**字节码指令数减少 35%、总字节数减少 31%、解释速度提升 30-50%**(取 benchmark)。

论文里的例子 `a = b + c * d`:

```
栈式 Lua 4.0:                  寄存器式 Lua 5.0:
GETGLOBAL b                    GETGLOBAL R0 "b"
GETGLOBAL c                    GETGLOBAL R1 "c"
GETGLOBAL d                    GETGLOBAL R2 "d"
MUL                            MUL  R1 R1 R2
ADD                            ADD  R0 R0 R1
SETGLOBAL a                    SETGLOBAL R0 "a"
(6 指令 + 5 次 push/pop)        (6 指令,无栈操作)
```

Dalvik 设计文档(Dan Bornstein,Google IO 2008)几乎复述同样理由——**Android 设备 2008 年是 200MHz CPU,dispatch 开销决定一切**。

## 五、同一段 Mochi 代码,两种字节码

源码 `let c = a + b * 3`(a/b 已是局部变量):

```
栈式(本系列实际用):              寄存器式(假想 Mochi-R,仿 Lua):
LOAD_LOCAL 0  ; push a            MUL    R3 R1 K0    ; R3 = b * 3
LOAD_LOCAL 1  ; push b            ADD    R2 R0 R3    ; R2(=c) = a + R3
CONST 0       ; push 3
MUL           ; b*3
ADD           ; a+b*3
STORE_LOCAL 2 ; c

(6 指令、6 dispatch、4 次栈操作)   (2 指令、2 dispatch、0 次栈操作)
```

寄存器式**指令数 1/3,dispatch 次数 1/3**——这就是 Lua 论文里 30% 速度提升的来源。

## 六、寄存器分配的复杂性

**寄存器式的代价压在编译器身上**:要决定**每个中间值放进哪个虚拟寄存器**——栈式编译器从不需要做(栈式是"算出来自动入栈")。

工业级常用**线性扫描(linear scan)寄存器分配**(Poletto 1999):

```
1. 给每个变量算「活跃区间」(从第一次写到最后一次读)
2. 按起点排序,扫描每个区间:
     起点 → 分配空闲寄存器
     终点 → 释放回空闲集
3. 寄存器不够 → 溢出到栈帧额外槽(spill)
```

**比图着色法快一个数量级**,LLVM 早年 JIT、HotSpot client compiler 都在用。

Lua 用更简单的版本——**编译期顺序分配,不优化**。`lcode.c` 用一个 `freereg` 计数器,emit 一条指令就 +1,**像在栈上分配,只是用编号代替**。够 Lua 用 20 年。

```
Lua 编译器     1500 行 lcode.c     ← 寄存器分配 + 跳转修正 + 常量折叠
CPython        800 行 compile.c    ← 主要是 visit AST + emit
```

**多出来的复杂度都在性能上挣回来了**——工程取舍的经典案例。

## 七、为什么本系列 Mochi 选栈式

四个理由,按重要性排:

1. **教学**:栈式编译器 80 行讲透;寄存器式要再讲一章寄存器分配
2. **可读**:栈式反汇编像后缀表达式直观;寄存器式要看一堆 R0/R1/R2 编号
3. **JVM / CPython / WASM 都是栈式**——读者读这些源码用得上
4. **24 篇写编译器**——栈式编译器代码量少、跳转回填讲得透

但**写完 30 篇你想做下一步**——给 Mochi 加个 register-based 后端、补线性扫描寄存器分配,是非常好的练习。**LuaJIT 的 Mike Pall 就是这样把 Lua 解释器从 130MB/s 推到 1GB/s**。
