# 词法分析

「编译原理课上学的 NFA / DFA / Thompson 构造,跟我手上要写的 lexer 有什么关系?」——**没有关系**。你不需要先证明正则等价于自动机才能写词法分析器,**它就是一个 while 加一个 switch**。但这一步是新手最容易翻车的:token 类型乱、位置信息丢、字符串转义漏 case、注释和除号分不清。本篇把这些坑一次踩完,**给一个 80 行能 cover Mochi 90% 场景的 lexer**——这之后你不会再害怕"实现词法"四个字。

> 一句话先记住:**词法分析 = 字符流 → token 流**,**手写状态机 = while + switch + peek / advance 两个指针**。**关键不是算法多巧,而是边界 case 处理得多细**——尤其是字符串转义、嵌套注释、数字字面量边界。**实现 100 行,边界 case 是后面 200 行**。

---

## 一、词法在做什么

```
源码:  let x = 10 + 20
        ↓ lexer
tokens: [LET, IDENT(x), EQ, NUM(10), PLUS, NUM(20), EOF]
```

**为什么要分两步**(lex + parse):

- **关注点分离**:词法管"字符怎么分组",语法管"分组后怎么组合"
- **复用**:换语法不用换 lexer(JS 加新关键字,scanner 不动)
- **错误定位**:`x = = 10` 词法 5 个 token,语法才报"双等号在赋值不合法"

接口:`function tokenize(src: string): { tokens: Token[]; errors: MochiError[] }`。

**lexeme 保留字符串、不在词法层转 number**:数字可能溢出、可能有进制(`0x10`)、浮点可能有精度——**那些是求值阶段的事**。

关键字识别:**先按 ident 扫,扫完查表**(`let / var / fn / if / else / while / for / return / true / false / class / import`),不要一开头就 switch——前缀冲突(`return` vs `returns`)会折磨你。

## 二、手写状态机

主循环就两个指针 + 一个 while:

```typescript
export function tokenize(src: string) {
  const tokens: Token[] = [], errors: MochiError[] = []
  let i = 0, line = 1, col = 1
  const peek = (k = 0) => src[i + k]
  const advance = () => {
    const c = src[i++]!
    if (c === "\n") { line++; col = 1 } else col++
    return c
  }
  const at = (): Position => ({ line, col })

  while (i < src.length) {
    const start = at(), c = peek()!
    if (/\s/.test(c)) { advance(); continue }
    // ... 各种 case 见 §三
  }
  tokens.push({ kind: "eof", lexeme: "", pos: at() })
  return { tokens, errors }
}
```

**两个 helper 就够**:`peek()` 看不前进,`advance()` 吃一个字符并维护 `(line, col)`。**90% 的手写 lexer 就这俩**。

## 三、关键场景

**标识符 vs 关键字**——首字符 `[a-zA-Z_]`,后续 `[a-zA-Z0-9_]`:

```typescript
if (/[a-zA-Z_]/.test(c)) {
  let lex = ""
  while (i < src.length && /[a-zA-Z0-9_]/.test(peek()!)) lex += advance()
  tokens.push({ kind: KEYWORDS[lex] ?? "ident", lexeme: lex, pos: start })
  continue
}
```

**数字字面量**——注意小数点要 `peek(1)` 看下一位是不是数字,否则 `1.toString()` 里的 `.` 会被吃掉:

```typescript
if (/[0-9]/.test(c)) {
  let lex = ""
  while (i < src.length && /[0-9]/.test(peek()!)) lex += advance()
  if (peek() === "." && /[0-9]/.test(peek(1) ?? "")) {
    lex += advance()
    while (i < src.length && /[0-9]/.test(peek()!)) lex += advance()
  }
  tokens.push({ kind: "number", lexeme: lex, pos: start }); continue
}
```

**字符串和转义**——固定 4 种 `\n \t \\ \"`,别支持 `\u{1234}`(不做 Unicode):

```typescript
if (c === '"') {
  advance(); let lex = ""
  while (i < src.length && peek() !== '"') {
    if (peek() === "\\") {
      advance()
      const n = advance()
      lex += ({ n: "\n", t: "\t", "\\": "\\", '"': '"' } as Record<string, string>)[n] ?? n
    } else lex += advance()
  }
  if (i >= src.length) {
    errors.push(new MochiError("lex", start, "unterminated string")); continue
  }
  advance()
  tokens.push({ kind: "string", lexeme: lex, pos: start }); continue
}
```

**注释**——`/` 要 lookahead 一位区分单 `/`(除号) / `//`(行注释) / `/*`(块注释)。**这就是 lexer generator 不擅长的"上下文",06 篇展开**:

```typescript
if (c === "/" && peek(1) === "/") {
  while (i < src.length && peek() !== "\n") advance(); continue
}
if (c === "/" && peek(1) === "*") {
  advance(); advance()
  while (i < src.length && !(peek() === "*" && peek(1) === "/")) advance()
  advance(); advance(); continue
}
```

**多字符 token**——`==` 必须**在 `=` 之前判断**,否则永远拿不到 `==`;`<=` `>=` `!=` 同理。

## 四、错误恢复:不要 throw

```typescript
errors.push(new MochiError("lex", start, `unexpected '${c}'`))
advance()  // 继续走,不要停
continue
```

**为什么不 throw**:用户文件里有 3 处词法错误,只报第一处停掉——他改完一处再跑还有两处,**改一次跑一次,体验极差**。**收集所有错误一次性给出**,IDE / LSP 才有意义。**lexer 报错不中断**,记住这一条。

简单测试:`tokenize("let x = 10").tokens.map(t => t.kind)` 应得到 `["let","ident","=","number","eof"]`。

## 五、工业级是怎么做的

**V8 的 Scanner**(`v8/src/parsing/scanner.cc`):`template <bool capture>` 模板特化省内存;ASCII fast path + UTF-16 慢路径,**字符识别走查表不走 regex**;对 `/` 上下文敏感,看上一个 token 决定它是除号还是正则;关键字走 perfect hash。

**CPython 的 tokenizer**(`Parser/tokenizer.c`):维护 indent stack——**词法层就要管缩进,因为 INDENT / DEDENT 是 token**;行连续(`\` 换行)在 tokenizer 处理,parser 看不到。

**rustc 的 lexer**(`compiler/rustc_lexer/src/lib.rs`),**特别推荐看**——比 V8 / CPython 干净 10 倍。Rust 迭代器风格,token 通过 `Cursor::advance_token` 一个一个产出,可读性高。

---

## 六、Mochi 这里偷懒了

| 偷懒 | 工业级会做 |
| --- | --- |
| 标识符只 ASCII | V8 / Python 支持任意 Unicode XID |
| 字符串只 `"..."` | Python `'..'` `"""..."""` `r"..."` `f"..."` |
| 没数字进制 | 所有真实语言都支持 `0x` `0b` `1_000` |
| `\u{1234}` 不支持 | JS / Rust 必须支持 |
| 注释不嵌套 | Rust 的 `/* /* */ */` 是嵌套的 |
| regex `/\s/` 做字符分类 | 查表 + ASCII fast path,**regex 慢 5 倍** |

**这些不影响学原理**——长出来都是堆 case,不是新机制。

---

## 七、踩坑提醒

1. **`peek()` 越界返回 undefined**——`peek() === '"'` OK,`peek() < 'z'` 翻车
2. **`/` 歧义**——除号 vs 注释 vs 正则字面量(JS),**lookahead + 上下文**
3. **多字符 token 顺序**——`==` 必须在 `=` 之前,`<=` 必须在 `<` 之前
4. **行号不更新**——`advance()` 里忘了处理 `\n` ,所有错误都在第 1 行
5. **字符串没结束时位置错**——记 string 开头的 pos,不是当前 pos
6. **大小写关键字**——`IF` 是 `if` 吗?**严格小写,跟 JS / Rust / Python 一致**
7. **`\r\n` 换行**——Windows 文件统一吞掉 `\r`
8. **空文件**——`tokens` 只有一个 EOF,parser 别假设至少有真 token
