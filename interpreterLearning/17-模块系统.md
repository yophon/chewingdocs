# 模块系统

`import "math"` 这一行表面平平无奇——拆开看,它干了**四件事**:**找到 math.mochi 在文件系统的位置、执行它的 body 一次、把顶层定义打包成命名空间、绑定到当前 scope**。任何一步出错你都见过那个报错信息("Cannot find module"、"Circular dependency"、"x is undefined")。讲不清模块系统,你写 Node 项目永远会被循环依赖咬;讲不清 CJS vs ESM,你看不懂为什么 webpack tree shaking 必须 ESM。这篇把这一行拆透。

> 一句话先记住:**模块 = 顶层定义打包成命名空间 + 一个「这个路径执行过没有」的全局缓存表**;**循环依赖靠"先放空模块进缓存,再跑 body"破解**;**CJS 同步动态、ESM 静态可分析——ESM 是未来,因为静态分析能 tree shake**。

---

## 一、模块是什么

```mochi
// math.mochi                           // main.mochi
let PI = 3.14159                        import "math"
fn sqrt(x) { ... }                      import { sqrt } from "math"
fn pow(base, exp) { ... }
                                        print(math.PI)
                                        print(sqrt(2))
```

实现层就一句话:**模块 = 一个 Map<string, Value>**,把这个文件的顶层 `let` / `fn` 全塞进去。`import "math"` 把整个 map 绑到 `math` 这个名字;`import { sqrt }` 把 map 里的 `sqrt` 单独挑出来塞到当前 scope。

没有命名空间会怎样? 所有文件共享一个全局表,**两个文件都定义 `init` 函数就冲突**——这就是 C 没模块系统、靠"头文件 + 前缀命名 (`pthread_mutex_init`)"硬撑的世界。

---

## 二、三个核心问题

```
1. 模块路径解析  import "math" → 哪个文件?
2. 加载与缓存    同一个模块只跑一次
3. 循环依赖      A 引 B, B 引 A 怎么办
```

## 三、模块路径解析 (resolver)

`import "math"` 怎么找到 `math.mochi`? 一套 resolver 规则:

```
import "math"  在 /proj/src/main.mochi 里:
1. 相对路径 "./math" / "../math"      → 直接拼当前目录
2. 项目根 stdlib                      → <project>/stdlib/math.mochi
3. 第三方包 (node_modules 风格)        → <project>/mochi_modules/math/index.mochi
                                         再没有就往上 ../mochi_modules/...
4. 绝对路径 / URL                     → 直接用
5. 全部失败                           → "Cannot find module 'math'"
```

Node 的 resolver 算法在 `lib/internal/modules/cjs/loader.js`,**最复杂的一段是 node_modules 往上递归查找**。Python 的 `sys.path` 是简化版——按顺序遍历目录,第一个匹配就用。**Mochi 简化做法**:相对路径 / `<src>/<name>.mochi` / `<stdlib>/<name>.mochi`,三个匹配,够用。

## 四、加载与缓存

```
ModuleLoader 维护一个 Map<absolutePath, Module>

load(path):
  1. cache.has(path)?               → 直接返回缓存
  2. 否则:
     a. 读源码,解析 → AST
     b. 创建空模块对象 module = { exports: {} }
     c. cache.set(path, module)     ← 关键! 先放进缓存
     d. 执行 body(填充 module.exports)
     e. 返回 module
```

**步骤 c 在步骤 d 之前**——这就是循环依赖能跑通的秘密,下一节就靠它。

为什么要缓存? **保证模块只跑一次**。`math.mochi` 顶层有个 `print("loading math")`,你 `import "math"` 十次只该打印一次。所有正经模块系统都这么干 (Node CJS、Python `sys.modules`、ESM module records、Java ClassLoader)。

## 五、循环依赖:真实例子

最经典的翻车现场:

```mochi
// a.mochi                      // b.mochi                      // main.mochi
import "b"                      import "a"                      import "a"
let A = 1                       let B = 2
print("a sees b.B =", b.B)      print("b sees a.A =", a.A)
```

执行轨迹:

```
1. main: load("a") → a 不在缓存,放空 a 进缓存,开始跑 a.body
2. a.body 第 1 行 import "b" → load("b"),放空 b 进缓存,开始跑 b.body
3. b.body 第 1 行 import "a" → 查缓存:有! a 已经在缓存(空的)→ 直接返回引用
4. b.body 第 2 行 let B = 2  → 填进缓存里的 b
5. b.body 第 3 行 print(a.A) → a.A 此刻是 undefined!(a 还没跑到第 2 行)
6. b.body 跑完,回到 a.body 第 2 行 let A = 1
7. a.body 第 3 行 print(b.B) → b.B = 2,正常打印
```

输出:`b sees a.A = undefined` / `a sees b.B = 2`。

**循环依赖的真实行为**:**它不报错,但 b 看到的 a 是"部分构造"的状态**。新手以为是 bug,其实是设计——CJS 选择"无论如何让它跑下去,后果自负"。

**ESM 怎么解决**:静态分析所有 import,**提前把所有模块的 export 绑定 hoist 出来**(实际值待填),body 执行时即使绕回来也能拿到 binding 引用——**比 CJS 安全一点,但循环还是循环**——没人能真正"解决"循环依赖,只能让翻车现场更可控。

**实战建议**:循环依赖几乎都是设计错误——重构成"A 和 B 都依赖一个 C",别硬扛。

## 六、CJS vs ESM:为什么 ESM 是未来

| 特性 | CommonJS (Node 老式) | ESM (标准) |
| --- | --- | --- |
| 语法 | `require()` / `module.exports` | `import` / `export` |
| 时机 | 运行时动态 | 编译期静态 |
| 加载 | 同步阻塞 | 可异步并行 |
| 循环依赖 | "能跑就行" | 静态 binding 更可控 |
| Tree shaking | 不行 (export 是运行时对象) | 行 (静态分析未用的 export 删掉) |
| 顶层 await | 不行 | 行 |
| 浏览器原生支持 | 不行 | 行 (`<script type="module">`) |

**ESM 是未来**的原因:**静态分析**。`import { sqrt } from "math"` 在编译期就知道你引用了什么——打包器 (webpack / esbuild / rollup) 能把 math 里没用到的函数全删掉,**Node 16+ 已经原生支持,所有新项目应该走 ESM**。CJS 还活着是历史包袱——Node 生态十几年的库都用 require。这场战争还得打几年。

## 七、最简实现:40 行 TS

```typescript
interface MochiModule { exports: Map<string, Value> }

class ModuleLoader {
  cache = new Map<string, MochiModule>();
  constructor(public globalEnv: Env, public projectRoot: string) {}

  load(name: string, fromDir: string): MochiModule {
    const fullPath = this.resolve(name, fromDir);
    if (this.cache.has(fullPath)) return this.cache.get(fullPath)!;

    // 关键:先放空模块进缓存,再执行 body
    const module: MochiModule = { exports: new Map() };
    this.cache.set(fullPath, module);

    const ast = parse(fs.readFileSync(fullPath, "utf8"));
    const moduleEnv = new Environment(this.globalEnv);
    interpret(ast, moduleEnv, this, path.dirname(fullPath));
    for (const [k, v] of moduleEnv.topLevels()) module.exports.set(k, v);
    return module;
  }

  resolve(name: string, fromDir: string): string {
    if (name.startsWith("./") || name.startsWith("../"))
      return path.resolve(fromDir, name + ".mochi");
    for (const dir of ["stdlib", "src"]) {
      const p = path.join(this.projectRoot, dir, name + ".mochi");
      if (fs.existsSync(p)) return p;
    }
    throw new Error(`Cannot find module '${name}'`);
  }
}
```

跑得通的 ModuleLoader 就这点东西——**循环依赖处理就藏在 `cache.set` 出现在 `interpret` 之前这一行**。

## 八、工业级是怎么做的

- **Node CJS**:`lib/internal/modules/cjs/loader.js`——`Module._load` 是入口,核心就是这个 cache + resolve 流程,只是处理了 N 种 edge case (extensions、conditional exports、ESM/CJS 互操作)。
- **V8 ESM**:`src/objects/source-text-module.cc`——静态分析、binding hoist、并行加载,**比 CJS 复杂十倍**。
- **CPython**:`Lib/importlib/_bootstrap.py` + `_bootstrap_external.py`,`sys.modules` 是那个缓存表;**Python 的 import 系统在 PY3.0 重写过一次**,现在的实现可读性算高的。
- **Java**:`ClassLoader.loadClass`,**双亲委派机制**——子 ClassLoader 先问父亲,找不到自己再找。**和 Node node_modules 往上找是不同设计,但都解决"哪份 lib 该用"的问题**。

---

## 九、Mochi 这里偷懒了

- **只做 CJS 风格**——所有 import 同步阻塞加载,**不做异步 ESM**(那要静态分析 + binding hoist + 模块图拓扑排序,复杂度 10 倍起步)。
- **不做条件 exports / package.json**——一个模块就是一个文件,没有"main 入口"这种概念。
- **resolve 路径硬编码**——不做 alias、不做 monorepo workspace、不做 nodeModulesPaths 递归。
- **不做模块隔离**——所有模块共享一个全局 globalEnv(放 `print` / `sqrt` 这种内建)。生产语言会给每个模块独立全局,避免污染。
