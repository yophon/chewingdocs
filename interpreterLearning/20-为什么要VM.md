# 为什么要 VM

Mochi 跑得通了——但你跑一遍 `fib(30)` 要等 5 秒,`fib(35)` 已经够倒一杯咖啡。问题不在算法,在执行引擎:**树遍历解释器每访问一个 AST 节点都要 dispatch、查类型、new 一个临时 Value 对象**,内存散落在堆里,L1 缓存命中率低到可怜。CPython 走过这条路、V8 早期也走过这条路——**最终都换成了字节码 VM**,加速一个数量级起步。这一篇讲清楚为什么要重写后端,以及为什么 Mochi 只做单层栈式 VM 就够。

> 一句话先记住:**树遍历是"语义解释器",字节码是"指令解释器"**——前者方便改语义,后者方便跑得快。等语义稳定了,就该把它编译成一串紧凑的字节码,让 CPU 缓存帮我们做事。

---

## 一、树遍历到底慢在哪

回顾一下 Mochi 当前的 evaluator(11 篇写的):

```typescript
function evaluate(node: Node, env: Env): Value {
  switch (node.type) {
    case "NumberLit": return node.value;
    case "BinaryOp":
      const l = evaluate(node.left, env);
      const r = evaluate(node.right, env);
      return applyOp(node.op, l, r);
    // ... 30 个 case
  }
}
```

跑 `1 + 2 * 3` 时实际发生的事:

```
evaluate(BinaryOp +)
  ├─ evaluate(NumberLit 1)        函数调用 + switch dispatch + 装箱
  ├─ evaluate(BinaryOp *)         函数调用 + switch dispatch
  │     ├─ evaluate(NumberLit 2)  函数调用 + switch + 装箱
  │     ├─ evaluate(NumberLit 3)  函数调用 + switch + 装箱
  │     └─ applyOp                拆箱 + 加法 + 装箱
  └─ applyOp                      拆箱 + 加法 + 装箱
```

**算一个 `1 + 2 * 3` 做了 5 次函数调用、5 次 switch 分发、4 次装箱/拆箱**。问题汇总:**节点散落在堆上**(AST 节点是 `new` 出来的 JS 对象,缓存未命中)、**指针追逐**(`node.left` 解引用 L1 miss 100+ cycles)、**递归调用栈深**(`fib(30)` 求值深度上千层)、**泛型分发**(`applyOp` 每次都走 if 链判断类型)、**装箱开销**(每个中间结果都是 `{tag, value}` 对象)。

---

## 二、实测:树遍历 vs 字节码的差距

```text
fn fib(n) { if (n < 2) { return n } return fib(n - 1) + fib(n - 2) }
print(fib(30))
```

| 实现 | 耗时 (fib(30)) | 比例 |
| --- | --- | --- |
| Mochi 树遍历(当前) | ~5000 ms | 1× |
| Mochi 字节码 VM(目标) | ~500 ms | 10× |
| Node.js(V8 Ignition) | ~15 ms | 333× |
| 手写 C | ~3 ms | 1666× |

**字节码版本的目标:比树遍历快 10 倍**。再往上(V8 / native)的差距,靠 JIT 和原生编译——这是 28 篇之后的事,Mochi 不做。

## 三、为什么字节码快

```
树遍历                                   字节码 VM
                                         ip = 0
function evaluate(node) {                while (true) {
  switch (node.type) {                     op = bytecode[ip++]
    case ADD:                              switch (op) {
      l = evaluate(left)                     case ADD:
      r = evaluate(right)                      r = stack.pop()
      return l + r                             l = stack.pop()
    ...                                        stack.push(l + r)
  }                                          ...
}                                          }
节点 = 散落对象,递归调用              }
                                         字节码 = 连续数组,大循环 + switch
```

四个根本原因:

**3.1 内存紧凑**:字节码是连续 `Uint8Array`——CPU 读完一条指令,下一条已经预取进 L1。AST 是堆里的对象,根本指望不上 prefetcher。

**3.2 指令分发便宜**:一个大 `switch` 跳转,比一层层函数调用 + 节点 type 判断便宜得多。**computed goto**(GCC 扩展)能让分发再快 20-30%——CPython / Ruby 都用,Mochi 22 篇会讲。

**3.3 局部变量编号**:树遍历查 `env` 哈希表(每次至少一次 hash + 字符串比较);字节码版变量在编译期就分配好**槽位编号**(slot 0、slot 1...),运行时是 `frame[3]` 这种**数组下标访问**——L1 命中,纳秒级。

```
树遍历:   env.lookup("x")    哈希表查找       O(链长)
字节码:   frame[2]            数组下标         O(1) + cache friendly
```

**3.4 常量预提取**:`1 + 2 * 3` 里的 `1 / 2 / 3` 在编译期就被塞进常量池,字节码里只放 index。字面量在运行时不需要再"造出来"。

## 四、树遍历的好处:别全否定它

| 维度 | 树遍历 | 字节码 |
| --- | --- | --- |
| 实现复杂度 | 低(一个 evaluate 函数) | 中(编译器 + VM 两块) |
| 改语义 | 改一处 | 改三处(parser / compiler / VM) |
| 启动时间 | 0(直接跑 AST) | 要先编译 |
| debug 体验 | 行号直接是 AST 节点 | 需要 source map 反查 |
| 适合阶段 | **原型期** | **稳定期** |

**前 15 篇写树遍历不是浪费**——是先把语义试通,确认语言长什么样了再上 VM。工业级也有不少树遍历的"短命语言":Bash 是树遍历(脚本一次性跑完,启动时间敏感)、SQLite 早期是树遍历、很多 DSL 都是树遍历。

## 五、什么时候该重写

经验法则,满足任何一条就该上 VM:

1. 用户开始抱怨"为什么我这段代码这么慢"——性能成为瓶颈
2. 语言核心特性已经稳定(类、模块、闭包都齐了)
3. 准备做 hot loop 优化 / JIT 的前提
4. 想做时间旅行 debugger / 录制重放——字节码比 AST 容易序列化

**Mochi 已经满足 1 和 2,可以重写了**。

## 六、整体重写 vs 增量

**A. 双引擎并存**:树遍历不动,加 `--bytecode` flag 切到 VM。
**B. 整体替换**:删掉 evaluator.ts,所有路径都走 VM。

**推荐 B,但保留树遍历做 REPL 调试模式**——REPL 里改一个变量定义就要重新编译整个文件实在痛苦,树遍历直接 eval 是 REPL 的最佳形态(Python / Node REPL 都是某种意义上的"半树遍历")。理由:双引擎并存,**任何语义改动要改两处**——闭包、类、异常的语义一旦在两个引擎之间漂移,debug 痛得想哭。完整重写一遍**逼你把语义画清楚**,字节码就是"语义的可执行形式"。

---

## 七、工业级走过的同一条路

**CPython:树遍历 → 字节码(很早)**

```
.py 源码 → parser → AST → compile.c → .pyc 字节码 → ceval.c 大循环
```

CPython 没有真正的"树遍历时期"(0.9 时代很短),现在所有 `python xxx.py` 都先编译成字节码再跑。`.pyc` 文件就是缓存的字节码。CPython 3.11 起还有 specializing interpreter(把高频 opcode 替换成专门版本),3.13 开始有自适应解释器——baseline JIT 的雏形。

**V8:树遍历 → Crankshaft → Ignition**

```
2008 V8 初版   :不走解释器,直接 JIT(Full-codegen)
2010 Crankshaft:加优化 JIT
2017 Ignition  :加回字节码解释器(因为内存吃太多)+ TurboFan 做优化 JIT
```

**V8 是反过来的**:一开始嫌字节码慢,直接生成机器码;后来发现内存吃不消,**才"退回去"加了字节码解释器**。

JVM、Lua、Ruby、PHP——**所有跑得快的动态语言,核心都是字节码 VM**。

## 八、Mochi 的执行链路对比

```
重写前(15 篇结束):                重写后(25 篇结束):
  Mochi 源码 → tokens → AST           Mochi 源码 → tokens → AST
    → evaluate(AST, env)  ← 慢          → compile(AST)       ← 24 篇
    → Value                             → bytecode + 常量池
                                        → VM dispatch loop   ← 22 篇
                                        → Value
```

工业级再往上就是 `bytecode → profiler 发现 hot loop → optimizing JIT(TurboFan / HotSpot C2)→ 机器码 → CPU 直接跑`——**Mochi 不做最后两层**,加 JIT 是 28 篇的话题,只讲心智不实现。

---

## 九、Mochi 这里偷懒了

- **不做双层执行(baseline + optimizing)**——V8 那种 Ignition + TurboFan 双层 Mochi 不做,只做一层字节码 VM
- **不做 inline cache**——`x.foo` 的方法分派每次都走完整查找,**这是 Mochi VM 性能的最大留白**
- **不做 superinstruction**——CPython 3.11 引入的高频组合(`LOAD_FAST` + `LOAD_FAST` → `LOAD_FAST_LOAD_FAST`)Mochi 不做
- **不做寄存器式**——23 篇讲完寄存器式 VM 的取舍,但 Mochi 主线选栈式(实现简单、编译器简单)
- **不做 .mocp 字节码缓存**——Python 的 `.pyc` Mochi 没有,每次启动都重新编译
