# 文法基础

词法分析产出的是一串 token,但 `if x > 0 { ... }` 是不是合法 Mochi 程序?`a + b * c` 该按 `a + (b * c)` 还是 `(a + b) * c` 解析?这些问题词法回答不了——**它们属于文法的世界**。工程师视角下不需要证明文法等价、不需要啃龙书第三章,但有几样东西必须懂:**BNF/EBNF 怎么读、LL 与 LR 的取舍、二义性陷阱、优先级表怎么排**。不懂这些,后面 08 的递归下降和 09 的 Pratt 都是黑盒。

> 一句话先记住:**文法 = 一组「字符串重写规则」,定义"什么是合法程序"**。**真实语言基本都用 LL(手写递归下降) 或 PEG,LR(yacc/bison) 在式微**——LR 错误信息差、不易扩展、上下文敏感场景跪。

---

## 一、文法到底要描述什么

```
词法层:  "if" "x" ">" "0" "{" ... "}"    ← 已分成 token
文法层:  这串 token 是不是合法的 if 语句?
```

文法 = **形式化描述哪些 token 串是合法程序**。形式上写作 `G = (N, T, P, S)`:N 非终结符(`IfStmt`、`Expr` ...)、T 终结符(token)、P 产生式规则、S 起始符号(通常是 `Program`)。**不用背符号,记产生式怎么读就够**。

---

## 二、BNF 与 EBNF

**BNF**(Backus-Naur Form)最古老,样子是 `<expr> ::= <term> | <expr> "+" <term>`——问题:递归丑陋、不支持 `* + ?` 重复符。

**EBNF**(Extended BNF)加了正则风格元符号,可读多了:

```
expr   = term ("+" term)*
term   = factor ("*" factor)*
factor = number | "(" expr ")"
```

读法表:

| 写法 | 含义 |
| --- | --- |
| `A B` | A 后接 B(连接) |
| `A \| B` | A 或 B(选择) |
| `A*` `A+` `A?` | 零或多个 / 一或多个 / 可选 |
| `( ... )` | 分组 |

业内事实标准——《Crafting Interpreters》《Writing An Interpreter in Go》都用这套。

## 三、给 Mochi 写一段 EBNF

只列声明和表达式骨干:

```
program  = decl* EOF
decl     = letDecl | varDecl | fnDecl | classDecl | stmt
letDecl  = "let" IDENT "=" expr
fnDecl   = "fn" IDENT "(" params? ")" block
stmt     = ifStmt | whileStmt | forStmt | block | exprStmt
ifStmt   = "if" expr block ("else" (ifStmt | block))?
block    = "{" decl* "}"

expr     = assign
assign   = logicOr ("=" assign)?
logicOr  = logicAnd ("||" logicAnd)*
logicAnd = equality ("&&" equality)*
equality = compare (("==" | "!=") compare)*
compare  = term  (("<" | ">" | "<=" | ">=") term)*
term     = factor (("+" | "-") factor)*
factor   = unary  (("*" | "/") unary)*
unary    = ("-" | "!") unary | call
call     = primary ( "(" args? ")" | "[" expr "]" | "." IDENT )*
primary  = NUMBER | STRING | IDENT | "(" expr ")" | "true" | "false"
```

注意 `expr` 那 10 层楼梯——**这就是递归下降里最丑的一段**,09 篇用 Pratt 替掉。

## 四、LL vs LR:自顶向下 vs 自底向上

两大历史流派:

| 维度 | LL(自顶向下) | LR(自底向上) |
| --- | --- | --- |
| 思路 | 从起始符号开始,**预测**下一步用哪条产生式 | 从 token 开始,**移进-归约**,凑出非终结符 |
| 代表 | 递归下降、ANTLR、PEG | yacc、bison、GLR |
| 表达力 | LL(k) 弱,通常不能左递归 | LR(k) 更强,能左递归 |
| 错误信息 | **好**(知道在找什么) | 差(只知道"不能归约") |
| 上下文敏感 | **容易加 hack** | 难 |
| 工业现状 | V8 / CPython / rustc / Go / Swift / TS 全是这条 | 仅存于 GCC 老代码、PostgreSQL SQL parser |

**结论**:**LR 正在式微**。不是性能问题——是**错误信息和可维护性**:LR 报错只能说"unexpected token",手写 LL 能说"`if` 后面少了 `{`,你是不是漏写了大括号"。**PEG**(Parsing Expression Grammar)是 LL 近亲,加了"有序选择" `/`,**CPython 3.9+ 从 LL(1) 切到 PEG**——见 `Grammar/python.gram`。

## 五、二义性陷阱

文法写不好会**二义**——同一串 token 有多种解析树。两个经典坑:

**5.1 if-else 悬挂(dangling else)**——`if A if B X else Y`,`else` 配哪个 `if`?

```
解释 1:  if A { if B X else Y }   ← else 配近的 if
解释 2:  if A { if B X } else Y   ← else 配远的 if
```

EBNF `ifStmt = "if" expr block ("else" ...)?` 看不出二义。**所有 C-like 语言都选「else 配最近的 if」**,靠手写解析器的贪心 match 实现。**Mochi 强制 `{}` 块**,这种二义根本不出现——这就是为什么 Go、Rust、Swift 全强制大括号。

**5.2 优先级二义**——`a + b * c` 两种树。`expr = expr ("+"|"*") expr` 这种"扁平"文法二义。**正确写法**是把优先级编进结构:

```
term   = factor (("+"|"-") factor)*    ← 低优先级在外层
factor = unary  (("*"|"/") unary)*     ← 高优先级在内层
```

**优先级 = 嵌套层级**:越内层越紧地"抱住"操作数。

## 六、Mochi 运算符优先级与结合性

完整表,从低到高:

| 级别 | 运算符 | 结合性 | 例子 |
| --- | --- | --- | --- |
| 1 | `=` | 右 | `a = b = 1` → `a = (b = 1)` |
| 2 | `\|\|` | 左 | `a \|\| b \|\| c` → `(a\|\|b) \|\| c` |
| 3 | `&&` | 左 | `a && b && c` |
| 4 | `== !=` | 左 | `a == b != c` |
| 5 | `< > <= >=` | 左 | `a < b` |
| 6 | `+ -` | 左 | `a - b - c` → `(a-b) - c` |
| 7 | `* /` | 左 | `a * b * c` |
| 8 | `-x !x`(一元前缀) | 右 | `--x` → `-(-x)` |
| 9 | `f(args)` `a[i]` `a.b` | 左 | `a.b.c` → `(a.b).c`、`f()()` 链式调用 |

**左结合**:`a - b - c = (a - b) - c`,不是 `a - (b - c)`(否则 `5 - 2 - 1` 算出来是 4 而不是 2)。**右结合**:`a = b = 1`,先算 `b = 1`(产出值 1),再 `a = 1`。这张表 09 篇会一对一映射成 Pratt 的 `bp(op)` 数字——**优先级表就是 Pratt 的输入**。

## 七、工业级怎么做

| 项目 | 文件 | 方案 |
| --- | --- | --- |
| V8 | `src/parsing/parser-base.h` `parser.cc` | 手写递归下降 + 表达式 Pratt-ish |
| CPython 3.9+ | `Parser/parser.c`(由 `Grammar/python.gram` 经 pegen 生成) | PEG |
| rustc | `compiler/rustc_parse/src/parser/` | 手写递归下降 + 表达式 Pratt |
| Go | `src/go/parser/parser.go` | 手写递归下降,全在一个文件 |
| TypeScript | `src/compiler/parser.ts` | 手写递归下降,单文件 9000+ 行 |
| GCC C(老) | bison 生成 | **GCC 4.1+ 已改手写**(`gcc/c/c-parser.cc`) |

**注意 CPython 这条**——切到 PEG 后,**Python 才支持带括号的多行 `with`**,旧 LL(1) 表达力够不上。

---

## 八、Mochi 这里偷懒了

- **不写完整文法**:正式语言要把字符串转义、模板字符串、嵌套注释全编进 EBNF——Mochi 文档只列骨干,细节藏在解析器代码注释里
- **不证明无二义性**:Mochi 文法是 LL(2),靠强制大括号 + 手写贪心 match 回避二义,不做形式证明
- **不上 PEG 生成器**:工业级 PEG(像 pegen)能省一半代码,但调试栈太深、错误信息差——Mochi 全程手写
- **不维护"期待集"**:rustc 报错能列"expected one of `let`, `if`, ..."——需要在 parser 状态里维护 first set,Mochi 只报单 token 期待
