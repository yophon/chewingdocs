# 真实语言剖析

到这一篇为止,Mochi 已经走完了**词法、语法、AST、树遍历解释、字节码 VM、mark-sweep GC**——这些组件**在每一门主流动态语言里都长得一模一样**,只是命名和工程量不同。这一篇是**全系列密度最高的对照篇**——把 CPython / V8 / Lua / JVM 拆成几个维度,告诉你**"Mochi 哪一篇 ↔ 这门语言哪个文件 / 类 / 函数"**。读完 27 篇 Mochi 再看这一篇,**你能直接去 clone CPython / Lua 仓库,半天看出关键路径**——而不是被 50 万行 C 代码吓退。**这一篇之后是不是再看天书,差距全在这里**。

> 一句话先记住:**真实语言的核心组件不超过 Mochi 涉及的范围**——词法 / 语法 / AST / 解释 / 字节码 / GC,只是工程化、性能优化、边界 case 的复杂度高一两个量级。**Lua 5.4 全部源码 ~30k 行 C,周末 4 小时能啃完关键路径**——这是入门工业级解释器最划算的一门。CPython / V8 / HotSpot 都是数十万行,**直接读会迷路,带着"对应到 Mochi 哪一篇"的地图读才看得懂**。

## 二、各家的"独门绝技"一行话

| 语言 | 一句话本质 | 推荐难度 |
| --- | --- | --- |
| **CPython** | GIL + 字节码 VM。3.11+ 加 specializing interpreter,3.13 开始拆 GIL | ★★★ |
| **V8** | 高度优化的多 tier JIT(Ignition → Sparkplug → Maglev → TurboFan),文件最多最绕 | ★★★★★ |
| **Lua 5.4** | 寄存器式 VM + 极简增量 GC,**~30k 行 C**,**可读性最好** | ★ |
| **JVM HotSpot** | 模板解释器 + C1/C2 JIT,GC 选项多到爆炸(G1/ZGC/Shenandoah) | ★★★★ |
| **LuaJIT** | Lua + Tracing JIT,**Mike Pall 一个人写的奇迹**,动态语言性能上限 | ★★★ |

**新手只看一家的话:从 Lua 开始,没有第二个选项**。

## 三、关键数据结构对照

读源码最快的方式不是从 main 看起,**而是看核心数据结构**——把它跟 Mochi 里的对照,瞬间就知道每个字段干嘛。

| 维度 | CPython | V8 | Lua |
| --- | --- | --- | --- |
| 函数对象 | `PyFunctionObject`(func_code + globals + closure) | `JSFunction`(SharedFunctionInfo + Context + Code) | `LClosure`(Proto + upvals[]) |
| 调用栈 | `_PyInterpreterFrame`(3.11 改栈式后 +25%) | 机器栈 + Ignition 寄存器 | `CallInfo` |
| GC 头部 | `ob_refcnt + ob_type` | hidden class(`Map`) | `GCObject { next, tt, marked }` |
| GC 策略 | 引用计数 + 分代追踪 | Orinoco:并发 + 分代 + 增量 | 增量三色,可选分代 |
| 字符串 | UCS1/2/4 三选一 | ConsString / SlicedString / ... | 短串 interned + 长串 |
| 整数 | `PyLongObject` 任意精度 | SMI(31 位)或 `HeapNumber` | `lua_Integer` 64 位 |

**Mochi 的 Closure { proto, upvalues[] } 几乎是 Lua 的简化版**。**V8 SharedFunctionInfo 把字节码和 closure 实例分开**——为了一份字节码被多个 closure 共享。**Mochi 的 CallFrame { ip, base, closure } 是 Lua CallInfo 的近亲**——`ip` 在 Lua 里叫 `savedpc`,本质一样。**Mochi GC 头部 { marked, next } 几乎就是 Lua 的 CommonHeader**——写 GC 最教科书的做法。

**V8 字符串的"分裂"很出彩**——`"a"+"b"+"c"` 不立即拼接,做成 ConsString(树状),**真要 char access 时才扁平化**——这是 JS 大量字符串拼接性能能撑住的关键。**CPython 的引用计数是历史包袱**——拆 GIL(3.13)后必须改成 biased reference counting。

## 四、CPython 拆解:Mochi 视角

```
1. python my_script.py
2. tokenizer.c              ← Mochi 篇 05 词法
3. parser.c (PEG)           ← Mochi 篇 07-09 语法
4. compile.c                ← Mochi 篇 24 编译器
5. ceval.c (大 switch)      ← Mochi 篇 22 栈式 VM
6. gcmodule.c               ← Mochi 篇 26-27 GC
```

**特色一**:**3.11 specializing interpreter**(PEP 659)——`BINARY_ADD` 在运行时观察类型后,**原地改成 `BINARY_ADD_INT`**。**这是不用 JIT 的 JIT**——单纯在解释器层做 type specialization,**带来 ~30% 加速**。

**特色二**:**3.13 no-GIL 实验**——`python --disable-gil`。**这是 Python 历史最大的内部重构**。

**入口推荐**:`Python/ceval.c` `_PyEval_EvalFrameDefault`(主调度循环)→ `Python/bytecodes.c`(所有字节码,3.12 用 DSL 生成)→ `Modules/gcmodule.c`。

## 五、V8 拆解:Mochi 视角

```
JS 源码 → scanner.cc → parser.cc → ast.h         ← Mochi 篇 05-10
→ interpreter-generator.cc 生成 Ignition 字节码   ← Mochi 篇 24
→ interpreter.cc 跑 Ignition                      ← Mochi 篇 22
→ ic/ 目录:inline cache                          ← (Mochi 不实现)
→ baseline / maglev / compiler:多 tier JIT       ← Mochi 篇 28
→ heap/ Orinoco GC                                ← Mochi 篇 27 工业版
```

**入口推荐**(按难度递增):`src/parsing/scanner.cc Scanner::Scan()` → `src/builtins/builtins-array.cc`(`Array.push` 内置)→ `src/compiler/pipeline.cc Pipeline::Run()`(TurboFan 主流程)→ `src/heap/heap.cc`。

**看源码的诀窍**:**不要从 main 看起**——Chromium 的入口在浏览器代码里,V8 是嵌入库。**从一个 builtin(`Array.push`)看起,反向追到字节码 → IC → JIT**。

## 六、Lua 拆解:周末 4 小时入门

**这是本系列最强推荐**——Lua 5.4 完整源码 ~30k 行 C,周末啃完关键路径。

```
src/
├── lua.c          ← REPL 入口
├── llex.c (~700)  ← 词法
├── lparser.c (~1700) ← 语法 + 单遍编译到字节码
├── lopcodes.h     ← ~50 条指令
├── lvm.c (~1500)  ← 字节码 VM 主循环
├── lgc.c (~1400)  ← 增量三色 GC
├── lobject.h      ← 所有核心数据结构
├── ltable.c       ← Lua 表(数组+哈希)
└── lapi.c         ← C API
```

### 4 小时阅读路线

```
第 1 小时  → lobject.h(TValue / GCObject / Table / Proto / Closure)+ lopcodes.h
第 2 小时  → llex.c(llex 词法)+ lparser.c(luaY_parser / expr 递归下降)
第 3 小时  → lvm.c(luaV_execute 主循环)+ 对照 lopcodes.h 看 OP_ADD / OP_CALL
第 4 小时  → lgc.c(luaC_step 增量 GC)+ ltable.c(混合表实现)
```

**为什么强推 Lua**:代码风格统一(Roberto 团队 30 年一以贯之)、没有历史包袱、每个文件都自洽(不像 V8 一个改动牵动 50 个文件)、关键算法都有 paper 引用。**做完 Lua 你看其他三家会觉得"原来都差不多,只是工程化更狠"**。

## 七、JVM HotSpot 一行话

JVM 最特别——**没有前端**(`javac` 是另一个项目),只接 `.class` 字节码:

```
hotspot/share/
├── interpreter/templateInterpreter.cpp   ← 模板解释器
├── c1/ opto/                             ← C1 / C2 JIT
├── gc/g1/ gc/z/ gc/shenandoah/ ...       ← 多种 GC
└── classfile/                            ← .class 文件解析
```

**两个看点**:**C2 编译器**(`opto/`)优化质量至今无对手;**ZGC 做到了 200ms 堆上 < 1ms 暂停**——全球还没有第二个 GC 实现到这水平。**JVM 字节码 200+ 条指令**,是 Mochi 的 5-10 倍。

## 八、Mochi 这里偷懒了

**Mochi 30 篇之后,真实语言的工程复杂度还是高一两个量级**——不是知识体系上,**而是工程化的厚度**:

| 维度 | Mochi | 工业级语言 |
| --- | --- | --- |
| 代码量 | ~5-8k 行 TS | Lua 30k / CPython 1M / V8 2M |
| 测试用例 | ~200 个 | CPython 60k+ / V8 100k+ |
| 平台适配 | 一个 | 5+(x86/ARM × Linux/Mac/Win) |
| 优化 pass | 0 | C2 / TurboFan 各几十个 |
| 标准库 | builtins ~20 个 | CPython 200+ 模块 |

**所以读源码时永远记得**:**核心算法是 Mochi 教过的那个,剩下的 90% 是工程化**。

## 九、踩坑提醒

1. **从 main 看起读不下去**——main 在 CLI / 嵌入器里,核心在数据结构和主循环
2. **不画依赖图**——V8 一万个 class,不画 component graph 永远看不懂
3. **跳过 lobject.h / objects/**——核心数据结构没看清,后面符号全是黑盒
4. **以为字节码 = 机器码**——V8 / CPython / Lua 三家字节码完全不同
5. **不读测试**——V8 的 `test/mjsunit/` 是理解边界 case 的宝矿
6. **看版本不对齐**——CPython 3.10 之前没 PEG parser、3.11 之前 frame 在堆上,看老博客容易踩
7. **比较代码量等于比较复杂度**——Lua 30k vs V8 2M,**功能差距没 60 倍**,大部分是平台 / 兼容 / 优化
8. **学完一家就觉得"懂解释器"**——学三家才有比较视角,**Lua + CPython + V8 是黄金组合**
