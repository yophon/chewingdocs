# 表达式求值

parser 把 AST 装好放在那里——没人去走它,这棵树就是一堆死的 JS 对象。**evaluate(expr, env) → Value** 是 Mochi 第一次真的动起来的那一刻——传一个 AST 进去,出来一个值,**你写的语言开始有意义了**。这一篇 60 行 TS 你就能在 REPL 里打 `1 + 2 * 3` 看到 7 出来,这种"我做的东西真的活了"的快感是写解释器的爽点之一。但**也是踩坑最密集的一层**:短路、真值、隐式转换,每个动态语言都有自己的坑爹决定,Mochi 怎么选要立场鲜明。

> 一句话先记住:**树遍历求值 = `evaluate(node, env)` 递归在 AST 上走**,**短路 / 真值 / 隐式转换**是动态语言三大坑——Mochi 选 Python 路线:**严格真值、不做隐式数值转换、`&&` `||` 短路保留左操作数**。

---

## 一、evaluate 的形状

一个递归函数,**输入 AST 节点 + 环境**,**输出值**:

```ts
type Value = number | string | boolean | null | MochiFn | MochiObject;

function evaluate(e: Expr, env: Env): Value {
  switch (e.kind) {
    case "Literal":    return e.value;
    case "Identifier": return env.lookup(e.name);
    case "Binary":     return evalBinary(e, env);
    case "Unary":      return evalUnary(e, env);
    case "Call":       return evalCall(e, env);
    // Get / Set / Function 留给 14、16 篇
  }
}
```

**switch 派发,递归子节点,组合结果**——没有魔法,Mochi 树遍历版本核心就这 50 行。

## 二、Binary / Unary / Literal / Call 的实现

```ts
function evalBinary(e: BinaryExpr, env: Env): Value {
  if (e.op === "&&") {                       // 短路必须特判,不能先求两边
    const l = evaluate(e.left, env);
    return isTruthy(l) ? evaluate(e.right, env) : l;
  }
  if (e.op === "||") {
    const l = evaluate(e.left, env);
    return isTruthy(l) ? l : evaluate(e.right, env);
  }
  const l = evaluate(e.left, env);
  const r = evaluate(e.right, env);
  switch (e.op) {
    case "+":
      if (typeof l === "number" && typeof r === "number") return l + r;
      if (typeof l === "string" && typeof r === "string") return l + r;
      throw new TypeError(`不能把 ${typeof l} 和 ${typeof r} 相加`);
    case "-": return numOp(l, r, (a, b) => a - b);
    case "*": return numOp(l, r, (a, b) => a * b);
    case "/": if (r === 0) throw new RuntimeError("除以零");
              return numOp(l, r, (a, b) => a / b);
    case "==": return l === r;               // 严格相等,不像 JS 的 ==
    case "!=": return l !== r;
    case "<":  return numOp(l, r, (a, b) => a < b);
    // > >= <= 同理
  }
}

function evalUnary(e: UnaryExpr, env: Env): Value {
  const v = evaluate(e.operand, env);
  if (e.op === "-") {
    if (typeof v !== "number") throw new TypeError("一元 - 只支持数字");
    return -v;
  }
  return !isTruthy(v);                       // op === "!"
}

function evalCall(e: CallExpr, env: Env): Value {
  const callee = evaluate(e.callee, env);
  if (typeof callee !== "function") throw new TypeError("不可调用");
  return callee(e.args.map(a => evaluate(a, env)));   // MochiFn 见 14 篇
}
```

**Mochi 跑起来了**——把这 40 行接上 parser,REPL 输入 `1 + 2 * 3` 立刻看到 `7`。

## 三、短路语义:`&&` `||` 不是普通函数

**普通二元运算**先算左右再算运算符。**短路反过来**:**左侧决定要不要算右侧**——`if user != null && user.age > 18` 里,`user == null` 时右边的 `user.age` 不能算,否则空指针。

**短路不能写成函数**——函数调用前所有参数都求值完了,**先求 `user.age` 就炸了**。所以短路必须在求值器特判,不能复用 `evalBinary` 通用路径。

短路保留**左操作数**(不是 bool):`null || "default"` 得 `"default"`、`"hi" && getCount()` 得 `getCount()` 的返回值。**比 C 的"短路返回 0/1"更有用**——`a || default` 直接当作"取默认值"的语法糖,JS / Python / Lua 共同选择。

## 四、真值语义(truthiness):动态语言的传统艺能

什么算"真"、什么算"假"?**每门动态语言都有自己的坑**:

| 语言 | 假值列表 |
| --- | --- |
| JS | `false` `0` `""` `null` `undefined` `NaN`(`0 == false` 但 `[] == false` 也 true,if 里 `[]` 又是真) |
| Python | `False` `0` `0.0` `""` `[]` `{}` `None` + 自定义 `__bool__`(空容器都是假) |
| Ruby / Lua | 只有 `false` 和 `nil` —— `0` 是真,**最干净** |
| Mochi | `false` `0` `""` `null`(向 Python 看齐,但不让空容器自动为假) |

```ts
function isTruthy(v: Value): boolean {
  if (v === null) return false;
  if (typeof v === "boolean") return v;
  if (typeof v === "number") return v !== 0;
  if (typeof v === "string") return v !== "";
  return true;   // 对象 / 函数都为真
}
```

**Mochi 不学 JS** 的 `NaN` 是假——`NaN` 是个数字,在 Mochi 里 `if NaN` 为真(等价于 `NaN !== 0`)。**奇怪但比 JS 一致**。

## 五、隐式转换的坑:`"3" + 4` = ?

```
JS:        "3" + 4 → "34"      "3" - 4 → -1     [] + [] → ""     [] + {} → "[object Object]"
Python:    "3" + 4 → TypeError: can only concatenate str (not "int") to str
```

**Mochi 选 Python 这一边**——**显式比隐式重要**:`"3" + 4` 报错,要写 `"3" + toStr(4)`。理由:**JS 的隐式转换是 Brendan Eich 10 天里赶出来的妥协**,后面 25 年 JS 程序员都在为这个决定写防御代码(`Number(x)` / `String(x)` 显式转,或用 TS 把类型钉死)。**Mochi 不重复这个错**。

## 六、一元运算的细节:`-` 和 `!` 看着简单但有 NaN 坑

```mochi
let x = -1.5           // 一元 -,简单
let b = !true          // 一元 !,简单
let inf = -1 / 0       // -Infinity(Mochi 用 JS Number)
let n = 0 / 0          // NaN,小心传播
```

**Mochi 偷懒**:数字一律 JS Number(IEEE 754 double),自动继承 NaN / Infinity / -0。真实语言:Python 把 `int`(任意精度)和 `float` 分开、Lua 5.3+ `integer` / `number` 分开、JS 全是 double 所以 `0.1 + 0.2 !== 0.3`。22 篇 VM 回来加 int / float 分离。

## 七、工业级是怎么做的

| 项目 | 求值核心文件 | 形态 |
| --- | --- | --- |
| CPython | `Python/ceval.c` 的 `_PyEval_EvalFrameDefault` | **栈式字节码 VM**——switch 派发指令 |
| Lua | `src/lvm.c` 的 `luaV_execute` | 寄存器式 VM,同样 switch 派发 |
| Crafting Interpreters jlox | `Interpreter.java` | **纯树遍历**,visitor 模式(Java 没 ADT) |
| V8 Ignition | `src/interpreter/interpreter.cc` | 字节码 + 模板编译,**没有显式的"AST 求值"路径** |

V8 / Lua / CPython 都不做"AST 求值"——**编译期就把 AST 转成字节码**,VM 跑字节码。Mochi 10-19 篇先用 AST 求值练手,20-25 篇再升级 VM——和 Crafting Interpreters 节奏一样,**树遍历版本最容易理解**。CPython `ceval.c` 里有个**巨大的 switch**,每个 case 对应一个字节码——**Mochi 22 篇就是要写成那样**,只是 case 多到 200+。

---

## 八、Mochi 这里偷懒了

| 偷懒点 | 真实语言怎么做 |
| --- | --- |
| int / float 不分,都用 JS Number | Python / Lua / Java 分开整数和浮点 |
| 不做运算符重载(`__add__`) | Python `__add__`、Ruby `def +`、C++ `operator+` |
| 不做大整数 / 字符串 rope 优化 | Python int 无限精度、Java StringBuilder、Rust `push_str` |
| `==` 不递归比对象内容 | Python `__eq__`、Ruby `==` 可重载、Java `equals` |
| 求值不带 source location → 错误信息差 | 真实语言每个错误带文件 / 行号(**15 篇补**) |

**核心立场**:**先把简单 case 跑通**——Mochi 11 篇目标"REPL 能算表达式"。每多支持一种边界 case 代码量翻倍——这些坑留给后面专门篇章吃下来。
