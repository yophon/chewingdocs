# 栈式 VM

你写完编译器、生成了一串字节码,到这里要回答最实际的一个问题——**这些字节码是怎么变成结果的**。答案就是 VM 的调度循环:一个 `while(true)` 配一个 `switch(op)`,几十行就能让 `1 + 2 * 3` 算出 7。但**栈式 VM 的魔法不是循环本身,而是"用一个表达式栈把任意复杂的表达式拆成线性的栈操作"**——这套模型 JVM、CPython、V8 Ignition、WASM 全在用,搞懂这一篇你就能读 `ceval.c` 不晕。

> 一句话先记住:**栈式 VM = IP 在字节码里前进 + SP 在表达式栈上推/弹**,所有指令都只跟栈顶打交道,**调度循环就是 while(true) switch(op)**——一台只会"压栈、弹栈、跳转"的极简 CPU。

---

## 一、栈式 VM 是什么

**栈式(stack-based)VM**:计算用一个**表达式栈**,每条指令的输入和输出都通过这个栈传递。`ADD` 弹两个数压一个结果;`CONST` 压一个数;`JUMP` 改 IP 不动栈。

谁在用:**JVM**(javac)、**CPython**、**V8 Ignition**(栈 + accumulator 折中)、**WASM**、**.NET CLR**(IL)——基本上栈式是过去 30 年的主流。

最小数据模型,五件东西:

```
┌────────────────────────────────────────────────┐
│  IP        程序计数器,指向下一条要执行的字节码  │
│  SP        栈顶指针,指向表达式栈下一个空位      │
│  stack[]   表达式栈(push/pop 区)               │
│  locals[]  当前函数的局部变量数组               │
│  consts[]  常量池(编译期固定)                  │
│  code[]    字节码数组(编译期固定)              │
└────────────────────────────────────────────────┘
```

整个 VM 就是「IP 推进 + SP 推/弹」的死循环。

## 二、栈状态:`1 + 2 * 3` 是怎么算出来的

按 21 篇约定的字节码:

```
CONST 0    ; 1     常量池: [1, 2, 3]
CONST 1    ; 2
CONST 2    ; 3
MUL
ADD
RETURN
```

栈状态逐步演化(左边栈底,右边栈顶):

```
执行前       stack: []                IP→ CONST 0
CONST 0  →   stack: [1]               IP→ CONST 1
CONST 1  →   stack: [1, 2]            IP→ CONST 2
CONST 2  →   stack: [1, 2, 3]         IP→ MUL
MUL      →   stack: [1, 6]            IP→ ADD      (弹 2/3,压 6)
ADD      →   stack: [7]               IP→ RETURN   (弹 1/6,压 7)
RETURN   →   返回栈顶 7
```

**核心直觉**:表达式越深嵌套,**栈用得越深**——这就是 JVM class 文件里 `max_stack` 元数据的来源,编译期算好,VM 装载时分配。

## 三、调度循环:三种实现姿态

### 3.1 switch-case(最简单,最常用)

```ts
while (true) {
  const op = code[ip++];
  switch (op) {
    case OP.CONST:  stack[sp++] = consts[code[ip++]]; break;
    case OP.ADD:    { const b = stack[--sp]; stack[sp-1] += b; break; }
    case OP.JUMP_IF_FALSE:
      if (!stack[--sp]) ip += code[ip]; else ip++;
      break;
    case OP.RETURN: return stack[--sp];
  }
}
```

**缺点**:编译器把 switch 译成「跳表 + 一个统一的回到循环顶」——CPU 看到的是同一个间接跳转,分支预测器学不准。

### 3.2 computed goto(GCC 扩展,CPython 3.7+ 在用)

```c
static void* dispatch[] = { &&op_const, &&op_add, /* ... */ };
goto *dispatch[code[ip++]];

op_const:  stack[sp++] = consts[code[ip++]];
           goto *dispatch[code[ip++]];   // 直接跳到下一条
```

**为什么快**:**每条指令末尾自己跳到下一条**,CPU 看到 N 个不同的间接跳转点,每个点有自己的分支历史——**实测比 switch 快 15-25%**。CPython 用 `USE_COMPUTED_GOTOS` 宏控制开关。

### 3.3 threaded code(每条指令是函数指针)

每条指令是独立函数,靠**尾调用**串起来,编译器(`-O2`)消去栈帧。**HotSpot 模板解释器**是极致——每条字节码对应一段手写汇编模板。

## 四、Mochi 的极简栈式 VM(TypeScript 完整版)

```ts
enum OP { CONST, POP, LOAD_LOCAL, STORE_LOCAL, ADD, SUB, MUL, DIV,
          EQ, LT, GT, NOT, JUMP, JUMP_IF_FALSE, CALL, RETURN }

interface Chunk { code: number[]; consts: any[]; }

function run(chunk: Chunk): any {
  const { code, consts } = chunk;
  const stack: any[] = [];
  const locals: any[] = [];
  let ip = 0, sp = 0;

  while (true) {
    const op = code[ip++];
    switch (op) {
      case OP.CONST:       stack[sp++] = consts[code[ip++]]; break;
      case OP.POP:         sp--; break;
      case OP.LOAD_LOCAL:  stack[sp++] = locals[code[ip++]]; break;
      case OP.STORE_LOCAL: locals[code[ip++]] = stack[--sp]; break;
      case OP.ADD: { const b = stack[--sp]; stack[sp-1] += b; break; }
      case OP.SUB: { const b = stack[--sp]; stack[sp-1] -= b; break; }
      case OP.MUL: { const b = stack[--sp]; stack[sp-1] *= b; break; }
      case OP.DIV: { const b = stack[--sp]; stack[sp-1] /= b; break; }
      case OP.LT:  { const b = stack[--sp]; stack[sp-1] = stack[sp-1] <  b; break; }
      case OP.GT:  { const b = stack[--sp]; stack[sp-1] = stack[sp-1] >  b; break; }
      case OP.EQ:  { const b = stack[--sp]; stack[sp-1] = stack[sp-1] === b; break; }
      case OP.NOT:         stack[sp-1] = !stack[sp-1]; break;
      case OP.JUMP:        ip += code[ip] + 1; break;
      case OP.JUMP_IF_FALSE:
        if (!stack[--sp]) ip += code[ip] + 1; else ip++;
        break;
      case OP.RETURN:      return sp > 0 ? stack[--sp] : null;
      default: throw new Error(`unknown op ${op}`);
    }
  }
}
```

跑得通 `1 + 2 * 3`、`if x > 0 { print(x) }`——函数调用要加 call frame,留给 25 篇。

## 五、为什么 switch-case 在 V8 / 现代 JVM 上够用

理论上 computed goto 快 15-25%,但**现代 CPU 的间接分支预测器**(IT-TAGE、Hashed Perceptron)已经能学会 switch 跳表的模式——**热代码上预测准确率 90%+**,接近 computed goto。

加上现代 JIT(V8 TurboFan、HotSpot C2)会把**热点解释循环整段编进机器码**,dispatch 开销直接消失。所以 **Ignition 用 switch-case + CodeStubAssembler 生成机器码模板**,比 computed goto 还更可移植(Windows MSVC 没 GCC 扩展)。

**给应用工程师的结论**:写解释器从 switch 开始永远没错——能跑、能调试、能在 Windows 上编。性能差那 20%,加 JIT 一把全找回来。

## 六、工业级实现指针

| 项目 | 文件 | 看什么 |
|------|------|--------|
| CPython | `Python/ceval.c` `_PyEval_EvalFrameDefault` | 4000 行的怪兽,computed goto + 内联缓存 |
| CPython 3.11+ | `Python/generated_cases.c.h` | 自动生成的 specializing interpreter |
| V8 Ignition | `src/interpreter/interpreter-generator.cc` | 用 CodeStubAssembler 生成机器码模板 |

读源码顺序:先 Lua `lvm.c`(最干净,1000 行能读完)→ CPython(看 `case TARGET(LOAD_FAST):`)→ V8(要先懂 CSA)。

## 七、Mochi 这里偷懒了

- **只写 switch-case**——TypeScript 没 computed goto,Rust 版本也是 `match`
- **不做内联缓存(IC)**——CPython 3.11 把 `LOAD_ATTR` 拆 8 个特化版本运行时按类型替换,Mochi 不做
- **不做 superinstruction**——把高频指令对 `CONST + ADD` 合成一条,Mochi 不做
- **栈用 JS 数组**——真实 VM 用定长数组 + 边界检查省略

但**结构是对的**——把这套换成 Rust + `Vec<u8>` 字节码 + `Vec<Value>` 栈,性能直接逼近 Lua 5.4 量级,剩下都是常数优化。
