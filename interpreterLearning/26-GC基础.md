# GC 基础

写 C 的时候,每个 `malloc` 配一个 `free`——漏了就内存泄漏,多了就 double free,在 free 后用就 use-after-free,**每个 C/C++ 程序员都被这三个魔鬼咬过**。C 和 Rust 之外的现代动态语言把这套甩给了 GC:你 new 一个对象,什么时候释放是 runtime 的事。Mochi 也不例外。但 GC 不是魔法,**两大流派分得很清**:refcount(CPython / Swift)和 tracing(JVM / V8 / Go / Lua / Mochi),各有各的坑。这一篇拆 tracing 这一派——从 mark-sweep 到三色标记,Mochi 自己写一个 50 行能跑的 GC,接到第 25 篇的 VM 上。

> 一句话先记住:**GC = 找谁还活着,其余全是垃圾**。refcount 即时但怕循环引用,tracing 周期但能处理任意图。**Mochi 选 tracing + mark-sweep + 三色标记**——工业级 GC 的起点,后面所有优化(分代 / 增量 / 并发)都建立在这一层之上。

---

## 一、没有 GC 的世界

C 的 hello:

```c
char *s = malloc(100);
strcpy(s, "hello");
printf("%s\n", s);
free(s);             // 忘了 = 泄漏
// printf("%s\n", s); // free 后访问 = use-after-free,RCE 入口
// free(s);           // 再 free = double free,堆破坏
```

Rust 不用 GC,**用所有权 + 编译期检查**强制写对:每个值有唯一 owner,owner 离开作用域就 drop。代价是学习曲线——程序员要懂生命周期、借用、`Rc<RefCell<T>>` 这些。

现代动态语言不想让用户操心 → 把"什么时候释放"交给 runtime → 就需要 GC。

---

## 二、两大流派:refcount vs tracing

| 维度 | 引用计数(refcount) | 追踪式(tracing) |
| --- | --- | --- |
| 代表 | CPython / Swift ARC / PHP / Obj-C | JVM / V8 / Go / Lua / Mochi |
| 释放时机 | 即时(计数到 0) | 周期性扫描 |
| 循环引用 | 处理不了(要 cycle collector) | 天然处理 |
| 暂停时间 | 平均小但不可预测(链式释放) | STW 集中且较长 |
| 多线程开销 | 原子加减,贵 | 写屏障(并发 GC 才有) |
| 实现复杂度 | 看着简单,做对很难 | 看着复杂,有现成框架 |
| 适合 | 移动端 / 内存峰值敏感 | 多核 server / 高吞吐 |

**没有绝对优,只有场景适配**:Swift 用 ARC 因为移动端要可控内存峰值;Go 用 tracing 因为多核 server 容忍 STW 但不容忍每次赋值的原子操作。

---

## 三、引用计数:即时但有坑

每个对象多带一个 `refcount` 字段:

```
赋值 a = b      → b.refcount++
变量离开作用域  → 对应对象.refcount--
refcount == 0   → 立即释放,递归减少它持有的对象的 refcount
```

**优点**:内存峰值可控、没集中 STW、实现直观。

**致命缺点——循环引用**:

```python
a = []
b = []
a.append(b)    # b.refcount = 2
b.append(a)    # a.refcount = 2
del a, b       # 两个 refcount 都从 2 减到 1,谁也释放不了
```

**修复方案两条**:

1. **配 backup tracing GC**——CPython 就是这样:refcount 主力 + `gc` 模块周期扫循环
2. **要求程序员手动断环**——Swift 用 `weak` / `unowned`

**多线程开销**:`refcount++` 不能是普通加,必须 atomic increment;并发场景下**比 tracing 慢得多**。这就是为什么 CPython 的 GIL 拆不掉——所有 PyObject 共享同一个 refcount,无锁多线程 = 内存崩溃。

---

## 四、追踪式 GC:从根出发找活的

**GC roots → 顺着指针走 → 走到的活,走不到的死**。

GC roots(每种语言略不同):

- 全局变量 / 静态变量
- 当前所有 call frame 的局部变量(VM 数据栈)
- VM 寄存器 / 临时变量
- open upvalues(第 25 篇说过)
- C 扩展持有的引用

**整个堆是个对象图**,GC 就是图遍历——这就是为什么循环引用对 tracing 不是问题:**走过的就是黑,不会无限循环**。

---

## 五、mark-sweep 三步走

```
准备:每个堆对象多一个 mark 位,初始 0

第一步 MARK:
  从 GC roots 出发 DFS / BFS
  访问到的对象 mark = 1

第二步 SWEEP:
  扫描整个堆
  mark = 0 的对象 → 释放
  mark = 1 的对象 → 保留,顺便把 mark 清回 0

第三步:等下一轮 GC
```

**痛点**:

- **整堆扫一遍**——堆 1GB 就扫 1GB,慢
- **STW**——mark 期间堆结构不能变,mutator(应用线程)得停

不分代、不增量、不并发的版本就是这样。下一篇讲怎么优化。

---

## 六、三色标记:多一个灰,后面所有优化都靠它

布尔 mark 在并发 GC 里不够用——"已访问"和"已访问且孩子也访问完"是两件事。三色标记拆开:

```
白(white):还没访问到——候选垃圾
灰(gray): 自己访问过,但孩子还没全部访问
黑(black):自己 + 孩子都访问过——确定存活
```

工作集 = 灰对象队列:

```
初始:所有对象白,roots 染灰
循环:
  从灰集合取一个 obj
  对 obj 引用的每个孩子 child:
    如果 child 是白 → 染灰,放进灰集合
  obj 自己染黑(处理完了)
直到灰集合空 → 剩下的白对象全是垃圾,扫掉
```

一个 5 对象小堆演示(箭头 = 引用):

```
初始(root 是 A,D 没人引用):
  A(白) → B(白)
  B(白) → C(白)
  C(白) → A(白)    ; 循环引用
  D(白) → E(白)    ; 从 root 走不到
  灰集合:[ A ]      ; root 染灰

step 1: 处理 A,A 的孩子 B 染灰,A 染黑
  A(黑) → B(灰) → C(白)
  C(白) → A(黑)
  灰集合:[ B ]

step 2: 处理 B,B 的孩子 C 染灰,B 染黑
  A(黑) → B(黑) → C(灰)
  C(灰) → A(黑)    ; A 已黑,跳过
  灰集合:[ C ]

step 3: 处理 C,C 的孩子 A 已黑跳过,C 染黑
  A(黑) → B(黑) → C(黑)
  D(白) → E(白)
  灰集合:[ ]        ; 完成

sweep:D、E 还是白 → 释放;A/B/C 黑 → 保留(并清回白)
```

**为什么不用布尔 mark**:并发 GC 时 mutator 还在改对象图,**"孩子还没扫完"这个状态必须显式表达**,否则会漏标(下一篇细讲三色不变式)。

## 七、TypeScript 50 行 mark-sweep 实现

```typescript
interface GCObject {
  marked: boolean
  type: 'string' | 'array' | 'closure' | 'instance' | 'upvalue'
  // 各 type 自己的字段...
}

class GC {
  heap: GCObject[] = []
  threshold = 1024            // 触发阈值(字节)
  bytesAllocated = 0
  private grayStack: GCObject[] = []
  constructor(private vm: VM) {}

  alloc<T extends GCObject>(obj: T, size: number): T {
    if (this.bytesAllocated + size > this.threshold) this.collect()
    obj.marked = false
    this.heap.push(obj)
    this.bytesAllocated += size
    return obj
  }

  collect() {
    this.markRoots()
    this.traceReferences()
    this.sweep()
    this.threshold = this.bytesAllocated * 2     // 简单扩容策略
  }

  private markRoots() {
    for (const v of this.vm.stack) this.markValue(v)
    for (const f of this.vm.frames) this.markObject(f.closure)
    for (const uv of this.vm.openUpvalues) this.markObject(uv as any)
    for (const v of Object.values(this.vm.globals)) this.markValue(v)
  }

  private markValue(v: Value) {
    if (typeof v === 'object' && v !== null) this.markObject(v as GCObject)
  }

  private markObject(obj: GCObject) {
    if (obj.marked) return
    obj.marked = true
    this.grayStack.push(obj)        // 灰
  }

  private traceReferences() {
    while (this.grayStack.length > 0) {
      this.blackenObject(this.grayStack.pop()!)   // 灰 → 黑
    }
  }

  private blackenObject(obj: GCObject) {
    if (obj.type === 'closure') {
      for (const uv of (obj as any).upvalues) this.markObject(uv)
    } else if (obj.type === 'array') {
      for (const v of (obj as any).items) this.markValue(v)
    } else if (obj.type === 'upvalue') {
      const loc = (obj as any).location
      if ('closed' in loc) this.markValue(loc.closed)
    }
    // ... 其他类型
  }

  private sweep() {
    this.heap = this.heap.filter(obj => {
      if (obj.marked) { obj.marked = false; return true }
      return false                  // 让宿主 JS GC 顺便回收
    })
  }
}
```

VM 上的分配点:

```
STRING_CONCAT     → gc.alloc(newString, len)
NEW_ARRAY         → gc.alloc(arr, 16 + 8*n)
NEW_INSTANCE      → gc.alloc(instance, sizeof(fields))
CLOSURE           → gc.alloc(closure, 16 + 8*upvalueCount)
CLOSE_UPVALUE     → gc.alloc(upvalueObj, 24)
```

## 八、Mochi 偷懒了 / 工业级指针

| Mochi 偷懒 | 真实语言 |
| --- | --- |
| heap 用 TS 数组,sweep 是 filter | clox 用对象链表,sweep O(1) 删除 |
| 不分代 | JVM / V8 / .NET 全是分代 |
| STW 整个 GC | Go / V8 增量 + 并发 |
| 没写屏障 | 一旦并发就必须有 |
| sweep 出来的对象让宿主 JS GC 兜底 | clox / Lua 直接 free 自己分配的内存 |

**源码指针**:

- **Crafting Interpreters 第 26 章 "Garbage Collection"**——clox 的 mark-sweep,**就是这一篇的 C 版本**,400 行 C 跑完整 GC
- **Lua**: `src/lgc.c`,~1000 行,**注释比代码多**;Lua 5.1 是 mark-sweep,5.4 加了分代
- **Go**: `src/runtime/mgc.go` 文件头那段 200 行注释——**直接讲三色不变式怎么保证**,看完比读论文还清楚
- **CPython**: `Modules/gcmodule.c`(cycle collector)+ `Include/object.h` 的 `Py_INCREF / Py_DECREF`(主力 refcount)

**立场**:Mochi 玩具语言,堆才几 MB,STW 几十微秒看不见,**上分代 / 增量 / 并发反而代码上千行**,违背"100 篇文章读完能动手写一遍"的初衷。下一篇专门讲生产语言为什么必须上这些。
