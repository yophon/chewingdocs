# 字节码设计

字节码不是机器码——是**我们为 Mochi VM 设计的一套"虚拟 ISA"**。每条指令 1 字节 opcode 表示,后面可能跟操作数,整个程序就是一段 `Uint8Array`。听起来抽象,但你只要记住"VM 就是一个会从数组里读字节、根据字节查 switch 跳转去执行某段代码的循环",字节码就只是这个循环消化的 input。这一篇把指令集、常量池、跳转编码、反汇编工具**全部定死**——后面 22-25 篇全在这上面叠。**字节码格式一旦定下来,中途改一次成本巨大**,所以这一篇必须严谨。

> 一句话先记住:**字节码 = 一连串 1 字节 opcode + 操作数,常量池单独存,跳转用相对偏移**。反汇编工具(disassembler)第一天就写,后面 debug 编译器全靠它。

---

## 一、字节码 ≠ 机器码

```
机器码    CPU 真的能执行              x86-64 / ARM64 / RISC-V
字节码    我们 VM 的"虚拟 ISA"        CPython opcode / JVM bytecode / WASM
源码      人类写的                    Mochi / Python / Java
```

**字节码是"我们和 CPU 之间的一层"**——VM 把字节码翻译成对 stack / frame 的操作,最终落到几条真的 x86 指令上。这一层存在的理由:**跨平台**(JVM 字节码在 x86 / ARM 上都能跑)、**设计自由**(opcode 自定义,贴合语言特性,JVM 有 `invokevirtual`)、**检查/优化方便**(字节码层做静态分析 / 常量折叠 / 寄存器分配比在源码或机器码层都简单)。

---

## 二、关键决策(一次定死)

**2.1 操作数编码:Mochi 选变长**——opcode 1 字节,操作数按需 1-2 字节。简单指令(`ADD`)1 字节,带操作数的(`CONST <idx>`)3 字节。定长(RISC 风:WASM、Lua bytecode)解码简单但浪费空间;变长(CISC 风:CPython、JVM、x86)紧凑但解码要看 opcode 决定读几个。

**2.2 指令分发:大 switch**

```typescript
while (true) {
  const op = bytecode[ip++];
  switch (op) { case OP_CONST: ...  case OP_ADD: ...  ... }
}
```

22 篇会讲三种分发实现(switch / direct threading / computed goto),这一篇先用最简单的 switch。

**2.3 常量池(constant pool):字面量另外存**——字符串 / 浮点 / 函数对象不直接进字节码,进常量池,字节码里只放 index:

```
字节码:        OP_CONST 0    OP_CONST 1   OP_ADD
                  ↓            ↓
常量池[0] = 42                 常量池[1] = 100
```

为什么:字节码连续紧凑(放 8 字节 double 就臃肿)、重复字面量自动去重、GC 友好(常量池是独立对象数组,根集明确)。

**2.4 跳转:Mochi 选相对偏移**——`JUMP +20` 从当前 ip 往前跳 20 字节,字节码可重定位(放到另一段位置不用改跳转)。代价:跳转目标算起来稍绕,24 篇编译器统一处理。

## 三、Mochi 指令集(v1)

17 条 opcode,够跑算术 / 变量 / 控制流 / 函数调用:

```
栈/字面量    CONST <idx:u16>           push 常量池[idx]      POP   DUP
变量        LOAD_LOCAL  <slot:u8>      STORE_LOCAL <slot:u8>
            LOAD_GLOBAL <name_idx:u16> STORE_GLOBAL <name_idx:u16>
算术        ADD / SUB / MUL / DIV     pop b, pop a, push (a op b)
            NEG                       pop a, push -a
比较/逻辑    EQ / LT / GT              pop b, pop a, push bool   NOT
控制流       JUMP <off:i16>            JUMP_IF_FALSE <off:i16>
函数        CALL <argc:u8>             RETURN
```

按类别分段编号(0x10 = 变量、0x20 = 算术、0x40 = 跳转、0x50 = 函数):

```typescript
export enum Op {
  CONST = 0x01, POP = 0x02, DUP = 0x03,
  LOAD_LOCAL = 0x10, STORE_LOCAL = 0x11, LOAD_GLOBAL = 0x12, STORE_GLOBAL = 0x13,
  ADD = 0x20, SUB = 0x21, MUL = 0x22, DIV = 0x23, NEG = 0x24,
  EQ = 0x30, LT = 0x31, GT = 0x32, NOT = 0x33,
  JUMP = 0x40, JUMP_IF_FALSE = 0x41, CALL = 0x50, RETURN = 0x51,
}
```

反汇编时看到一个数字直接知道大类,每类留 16 个 opcode 扩展空间。操作数大小约定:**常量池 index** = u16(0~65535)、**local slot** = u8(0~255)、**跳转偏移** = i16(-32768~32767)、**call argc** = u8。**256 个 local 不够**:JVM 用 `wide` 前缀切到 u16,Mochi 偷懒——超过 255 个 local 直接报编译错误。

## 四、Chunk 数据结构

字节码本身 + 常量池 + 调试信息打包成一个 `Chunk`:

```typescript
export class Chunk {
  code:      Uint8Array  = new Uint8Array(0);   // 字节码
  constants: Value[]     = [];                   // 常量池
  lines:     number[]    = [];                   // 每字节对应的源代码行号
  writeByte(b: number, line: number) { /* push to code/lines */ }
  writeU16(v: number, line: number)  { /* little endian */ }
  addConstant(v: Value): number {  /* 去重后 push,返回 index */  }
}
```

**`lines` 是源码行号映射**——VM 报错时能说出"第 12 行"而不是"字节码 offset 0x47"。**22 篇就开始用,现在塞进去**。

## 五、反汇编工具(第一天就写)

写 VM 不写 disassembler,debug 编译器输出靠肉眼看二进制——这是地狱。第一天就把 disassembler 写好——核心是一个跟 VM 同构的循环,每条指令读完打印 `<offset>  <OP>  <operands>  ; <注释>`:

```typescript
export function disassemble(chunk: Chunk): string {
  const out: string[] = []; let ip = 0;
  while (ip < chunk.code.length) {
    const off = ip.toString(16).padStart(4, "0"); const op = chunk.code[ip++];
    switch (op) {
      case Op.CONST: { const idx = chunk.code[ip] | (chunk.code[ip+1] << 8); ip += 2;
        out.push(`${off}  CONST  ${idx}  ; ${fmtValue(chunk.constants[idx])}`); break; }
      case Op.LOAD_LOCAL: { const s = chunk.code[ip++]; out.push(`${off}  LOAD_LOCAL  ${s}`); break; }
      case Op.ADD:    out.push(`${off}  ADD`);    break;
      case Op.RETURN: out.push(`${off}  RETURN`); break;
      // ... 每个 opcode 一支
    }
  }
  return out.join("\n");
}
```

Mochi 源码 `let x = 1 + 2; print(x)` 编出来的字节码长这样:

```
=== chunk: <main> ===
constants: [0] 1   [1] 2   [2] "print"   [3] "x"

bytecode:
0000  CONST         0    ; 1
0003  CONST         1    ; 2
0006  ADD
0007  STORE_GLOBAL  3    ; x
000a  LOAD_GLOBAL   2    ; print
000d  LOAD_GLOBAL   3    ; x
0010  CALL          1
0012  POP
0013  RETURN
```

**看到这个反汇编,你已经能在脑子里 step through VM 了**:push 1 → push 2 → add → 栈顶 3 → 塞到 global "x" → push print → push x → call → 打印 3 → pop → return。**这就是 Mochi VM 第一天能跑的样子**——22 篇写 dispatch loop 把它跑起来。

## 六、工业级指令集对照

| 语言 | 指令数 | 文件 |
| --- | --- | --- |
| Lua 5.x | ~40 | `lua/src/lopcodes.h` |
| Mochi v1 | ~17 | 我们写的 `bytecode.ts` |
| Python 3.12 | ~120 | `Include/internal/pycore_opcode.h` |
| Python 3.13 | ~150(specializing) | 同上 + `Python/specialize.c` |
| V8 Ignition | ~200 | `src/interpreter/bytecodes.h` |
| JVM | 200(`<=` 1 字节硬上限) | JVMS §6.5 |
| WebAssembly | ~170 | spec §5.4 |

**指令数 vs 性能的取舍**:指令少(Lua 40)→ VM switch 短、分发开销低、编译器啰嗦,适合寄存器式;指令多(JVM 200)→ VM 长、编译器简单、常见操作有专门 opcode,适合栈式 + 静态类型。**Mochi 17 条偏极简**——栈式 VM 配少指令配单 pass 编译器,让 22-24 篇都能在小范围内讲完。

---

## 七、Mochi 这里偷懒了

- **不做 superinstruction**——CPython 3.11+ 把高频组合(`LOAD_FAST` + `LOAD_FAST` → `LOAD_FAST_LOAD_FAST`)融合成单条 opcode,Mochi 不做
- **不做 inline cache**——`x.foo` 的方法分派每次走完整查找,**这是 Mochi VM 最大的性能留白**
- **不做 specializing interpreter**——CPython 3.11 起的"看到 `BINARY_ADD` 两个 int,下次替换成 `BINARY_ADD_INT`"机制不做
- **没有 `wide` 前缀**——超过 256 个 local 直接编译错误,不像 JVM 优雅切到 u16
- **不做 verification**——JVM 字节码加载时会做 verifier(检查栈深度 / 类型),Mochi 信任自己的编译器

---

## 八、为什么这些决策不能后改

字节码格式一旦定下来,24 篇编译器、22 篇 VM 都按这个写——后面改一次三篇全要回炉:改 opcode 编号(VM switch 全错)、改操作数大小 u8 → u16(旧 VM 跑不了)、改跳转编码(回填全要重写)、加 / 删常量池语义(整个调用约定都受影响)。**这一篇所有数值都是 Mochi v1 的稳定 ABI**——23 篇会讨论寄存器式 VM,那是支线,主线 Mochi 锁定这套栈式字节码。
