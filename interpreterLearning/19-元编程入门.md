# 元编程入门

"代码即数据"——这是 Lisp 一句话讲了 60 年还没讲清楚的事。你天天写代码,从来不会想到代码本身也是个值,可以传、可以改、可以拼。但凡你写过 React 的 JSX、Rust 的 `#[derive]`、SQL 的 ORM、Vue 的 template,你已经在跟元编程打交道了,**只是没人告诉你那些东西底下是 quote / unquote / 宏展开**。这一篇把"代码即数据"拆成四个层次,讲清楚 `eval` 和 Lisp 宏不是一回事,Rust 宏和 JS Proxy 也**根本不在同一个时代**。

> 一句话先记住:**元编程的本质是"代码长得跟普通值一样,能传 / 能改 / 能拼"**——只是发生的时机不同(运行时 / 编译期),代价就天差地别。

---

## 一、四个层次:从最 hack 到最优雅

```
Level 1  eval(string)        把字符串当代码跑      JS eval / Python exec / SQL EXECUTE
Level 2  反射 (reflection)   看自己的类型和字段    Java reflect / Go reflect / TS keyof
Level 3  quote / unquote     把 AST 当数据传       Lisp / Elixir / Julia / Clojure
Level 4  编译期宏            编译期展开,零开销    Rust macro_rules / Scheme syntax-rules / C #define
```

| 层次 | 时机 | 类型检查 | 性能 | 安全 |
| --- | --- | --- | --- | --- |
| eval | 运行时 | 失效 | 慢 | 危险 |
| 反射 | 运行时 | 部分失效 | 中 | 中 |
| quote/unquote | 运行时构造、运行时求值 | 失效 | 中 | 中 |
| 编译期宏 | 编译期 | 完整 | 零开销 | 安全 |

**记住这张表**——任何"动态特性"都能塞进这四格之一。

---

## 二、Level 1:eval —— 把字符串当代码跑

```js
const x = 10;
console.log(eval("x + 1"));  // 11 —— 能看见外层作用域
```

`eval` 是**最暴力的元编程**:直接把字符串塞进 lexer / parser / evaluator 走一遍。代价巨大——**类型检查器投降**(TS 看到 `eval(s)` 返回 `any`)、**优化器投降**(V8 看到 `eval` 立即放弃所在函数的优化)、**安全投降**(`eval(userInput)` 是经典代码注入)。**99% 的"我需要 eval"实际是"没想清楚怎么参数化"**——查表 / 闭包 / 策略对象都能替代。

---

## 三、Level 2:反射 —— 程序看自己

```go
type User struct { Name string `json:"name"` }
t := reflect.TypeOf(v)
for i := 0; i < t.NumField(); i++ {
    f := t.Field(i); fmt.Println(f.Name, f.Tag.Get("json"))
}
```

**反射跟 eval 最大区别**:反射只能"看"和"调",**不能凭空造一段新代码**。工业用途几乎全是同一套——**序列化 / ORM / DI**——把字段名和数据库列对上、把 JSON key 和结构体字段对上。代价:慢(每次反射查表)、编译器看不穿(Go 的 `interface{}` + reflect 是性能杀手)。

---

## 四、Level 3:quote / unquote —— 代码即 list

到 Lisp 这层,真正的"代码即数据"才出场:

```lisp
(define expr '(if x y z))  ; '(...) 是 quote,不 eval,把 AST 给我
(eval expr)                ; 当成代码跑
(length expr)              ; 当成 list,返回 4
(cadr expr)                ; 取第二个元素,返回 x

(defmacro inc (x) `(+ ,x 1))   ; ` 是 quasi-quote,, 是 unquote
(inc 5)                         ; 编译期展开成 (+ 5 1) → 6
```

**Lisp 宏 vs 普通函数**:宏拿到的是 **AST**,不是求值结果。`(inc (read-line))` 不会立即读行,而是把 `(read-line)` 这棵 AST 塞进 `(+ _ 1)`。

Elixir / Julia / Clojure 都继承这套——它们的语法本身就是 AST(s-expression),**代码即数据是免费的**。

---

## 五、Level 4:编译期宏 —— 零运行时开销

### Rust `macro_rules!`

```rust
macro_rules! vec_of {
    ($($x:expr),*) => {{ let mut v = Vec::new(); $( v.push($x); )* v }};
}
let xs = vec_of!(1, 2, 3);
// 编译期展开为:let xs = { let mut v = Vec::new(); v.push(1); v.push(2); v.push(3); v };
```

**展开发生在编译期**——运行时一行宏代码都没有,跟手写完全等价。

### JS Proxy:**这不是宏,是 Level 2 的弟弟**

```js
const p = new Proxy({a: 1}, { get(t, k) { console.log("read", k); return t[k]; }});
p.a;  // 打印 "read a",返回 1
```

Proxy 是**运行时拦截**——每次属性访问都走 handler。**跟 Rust 宏完全不在同一档**:

| 维度 | Rust 宏 | JS Proxy |
| --- | --- | --- |
| 时机 | 编译期 | 运行时 |
| 开销 | 零 | 每次访问走 trap |
| 拿到的 | AST | property name |
| 能改语法吗 | 能(`vec![]` 是合法宏调用) | 不能 |

C 的 `#define MAX(a,b) ((a)>(b)?(a):(b))` 是最原始的"编译期宏"——纯文本替换,没有 AST,所以 `MAX(i++, j++)` 会展开两次 `i++`,经典坑。

## 六、Mochi 加一个 eval(15 行)

Mochi 已经有 lexer + parser + evaluator,做 eval 就是把它们串起来:

```typescript
// src/builtin.ts
import { tokenize } from "./lexer";
import { parse }    from "./parser";
import { evaluate } from "./eval";

export function evalString(env: Env, src: string): Value {
  const tokens = tokenize(src);
  const ast    = parse(tokens);
  let last: Value = null;
  for (const stmt of ast) last = evaluate(stmt, env);
  return last;
}
globalEnv.define("eval", { call: (args) => evalString(globalEnv, args[0] as string) });
```

```text
let x = 10
let result = eval("x + 5")
print(result)    // 15
```

**就 15 行**——Mochi 从此有 Level 1 元编程能力。代价:`eval` 调用点的类型检查全废、没有 sandbox、字节码 VM 上线后(22 篇起)eval 要把字符串现编译现执行,**慢得多**。

---

## 七、元编程的代价

| 工具 | eval | 反射 | quote/unquote | 编译期宏 |
| --- | --- | --- | --- | --- |
| 类型检查器 | 失效 | 部分失效 | 失效 | 完整 |
| IDE 跳转 | 失效 | 部分失效 | 失效 | 完整 |
| 优化器 | 放弃整个函数 | 中等损失 | 放弃 eval 处 | 零损失 |
| Debug 体验 | 痛 | 一般 | 痛 | 好(展开后是普通代码) |

**为什么 Rust 选 macro_rules + proc_macro 而不是 Lisp 宏**:Rust 要静态类型检查,所有宏必须在 type check **之前**展开完;Lisp 没有静态类型,宏可以推迟到运行时。

---

## 八、工业级实现指针

| 项 | 路径 |
| --- | --- |
| V8 eval | `v8::Script::Compile` + Eval mode,浏览器在 CSP 下禁用 |
| Python eval | `Python/bltinmodule.c::builtin_eval_impl` |
| Python compile | `Python/compile.c`,生成 code object |
| Rust macro_rules | `rustc_expand` crate |
| Rust proc-macro | `proc_macro` crate,编译器 fork 一个进程跑 |
| Clojure 宏 | `clojure.core/defmacro`,reader 输出后展开 |
| Elixir 宏 | `Macro.expand` / `quote do ... end` |

**真要看一遍**:Python 的 `compile() + exec()` 最容易上手——`compile("x+1", "<s>", "eval")` 返回一个 code object,`eval(code, {"x":10})` 跑出来 `11`。**这就是 V8 的 `v8::Script` 在 JS 里的对应物**。

---

## 九、Mochi 这里偷懒了

- **不做编译期宏**——没 `macro` 关键字,Rust / Scheme 那种零开销代码生成不支持
- **不做 quasi-quote**——Lisp 那套 `` `(...) `` 和 `,x` 不要,语法不是 s-expression
- **eval 没有沙箱**——直接跑在全局环境里,**生产代码绝对不能这么干**
- **没有 hygiene**——Rust 的卫生宏(避免变量名冲突)Mochi 不做

**Mochi 元编程边界:只做 Level 1 + Level 2**(反射后面 24 篇编译器有需要再加)——字节码 VM 阶段(20-25 篇)不会被宏展开拖累。
