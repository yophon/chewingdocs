# 类型系统初步

写动态语言写久了,你会觉得 `let x = 5` 然后 `x = "hello"` 是天经地义——直到你跟一个 Rust 程序员合作,他指着你的代码说"这一行运行时才会炸"。**类型系统是语言设计最大的选择题**:选静态、选动态、选渐进式,**每一条路都有真实的工程代价**——动态的代价是 bug 推迟到运行时,静态的代价是写两遍(代码 + 类型),渐进式的代价是两边都不够干净。这一篇不教你 Rust 怎么用(那是 rustLearning 11-13 干的事),也不教你写完整的类型检查器(TS 的 `checker.ts` 五万多行,凭啥这里讲完)——只给你一张**心智图**,看一眼知道每个语言为什么这么选。

> 一句话先记住:**动态 = 值带类型 / 变量不带,错误推迟到运行时**;**静态 = 变量带类型 / 编译期检查,错误更早,代价是表达力**;**Hindley-Milner = 从约束方程组反解类型,所以 OCaml / Haskell 不写注解也能推出大部分类型**。

---

## 一、动态 vs 静态:类型挂在哪里

```mochi
let x = 5        // x 没类型,5 这个值有类型 Int
x = "hello"      // 合法,x 现在装的是 String 值
```
```rust
let x: i32 = 5;     // x 这个变量有类型 i32
// x = "hello";     // 编译报错!String 不是 i32
```

| 视角 | 动态 (Python / JS / Mochi) | 静态 (Rust / Go / Java) |
| --- | --- | --- |
| 类型挂在 | **值上**(每个对象有 type tag) | **变量上**(声明时绑死) |
| 检查时机 | 运行时(执行到那一行才查) | 编译期(没运行就报错) |
| 优点 | 灵活、写得快、REPL 友好 | 错误早、性能高、IDE 智能补全 |
| 代价 | bug 推迟到运行时、重构难 | 写两遍(代码 + 类型) |
| 性能 | 慢(每次操作 type dispatch) | 快(编译期生成专用机器码) |

**为什么动态语言慢**:`x + y` 要先看 `x` 是 Int 还是 String 还是别的——**type dispatch 在每次 `+` 都跑**。静态语言编译期就知道是 Int,直接一条 `add` 机器码,零开销。**这就是 PyPy / V8 要 JIT 的原因**——运行时收集类型信息,假设它不会变,生成专用代码,逼近静态语言的速度。

## 二、渐进式类型:TS 和 Python type hints

第三条路:**让程序员选择性加注解,注解只是元数据,不影响运行**。

```typescript
function add(a: number, b: number): number { return a + b }
add(1, "hi")       // 编译期报错,运行时 number 已经消失
```
```python
def add(a: int, b: int) -> int: return a + b   # 不影响运行,只给 mypy 看
```

**关键**:**TS 编译成 JS 后,类型注解全部消失**;**Python 的 type hint 同样不影响运行**——`add(1, "hi")` 实际跑也不报错,Python 自己根本不查。**类型只在编译/检查阶段存在,运行时已经丢了**——这叫**类型擦除 (type erasure)**,第五节细讲。

渐进式类型的卖点:**大项目可以逐步加注解**。代价是**类型系统设计极复杂**——TS 的 `checker.ts` 五万行不是没原因的,要兼容 JS 的所有动态花招 (any、union、intersection、conditional types、template literal types ...)。

## 三、类型推导:让静态类型不啰嗦

为什么 Rust 写 `let x = 5` 不用写 `: i32`? **编译器自己推**:看到 `let x = <表达式>` → 求表达式的类型 → 赋给 x。简单到「右边是 Int 字面量,左边就是 Int」。但更猛的语言能推到更远:

```ocaml
let add x y = x + y
(* OCaml 推出: int -> int -> int,函数签名一个字注解都不用写 *)
```

OCaml / Haskell 的推导引擎叫 **Hindley-Milner (HM)**——**全局约束求解**,从函数体往参数倒推,**不需要任何注解就能推出所有类型**。

## 四、Hindley-Milner 的直觉:约束方程组

HM 看似神秘,核心就一个比喻:**类型推导 = 解方程**。考虑 `\x -> x + 1`:

```
1. 给所有未知类型起变量名
   x: α      x + 1: β       整体表达式: γ

2. 看 (+) 的已知类型: Int -> Int -> Int    ← 强制两边都是 Int

3. 写出约束
   α = Int (x 被传给 +)    β = Int (+ 的返回)    γ = α -> β

4. 求解  →  γ = Int -> Int

       \x  ->  x + 1
       │       │   │
       α       α  Int       ← 起初
              └─┬─┘
                │ (+) 强制 α=Int, β=Int
       γ = α -> β = Int -> Int   ← 解出来
```

**HM 的牛在于全局**:函数体里某处用了 `x + 1`,**整个函数签名就被定死**——你根本不用在签名上写一个字。这是 OCaml / Haskell 写起来这么干净的原因。Rust 的推导比 HM 弱一些(因为同时还要算生命周期、所有权、trait bound),但思路一样——**收集约束 + 求解**。

## 五、类型擦除 vs 类型携带

类型系统的另一条分水岭:**运行时还保留类型信息吗?**

| 语言 | 策略 |
| --- | --- |
| Java (泛型部分)、TypeScript、Haskell | **类型擦除**——编译后类型信息消失 |
| Rust、C# / .NET、Dart | **类型携带 (reified)**——运行时还能查类型 |
| Python、JS、Mochi | 动态,**值本来就带 type tag**,"擦除"无意义 |

**擦除的代价**:Java 写不出 `if (x instanceof List<String>)`——运行时不知道 `List` 装的是什么类型,只能 `instanceof List`。**这是 Java 反射 + 泛型坑特别多的根因**。**携带的代价**:**运行时多存一份类型元数据**——Rust 的 `std::any::TypeId` 能动态查类型,但每个类型多占几字节;.NET 的泛型每个实例化生成独立代码 + 类型表。Mochi 是动态语言,**每个值都带 type tag**(Int / Float / String / Object / Function ...),所谓"擦除/携带"对它没意义——**值本来就带着类型在跑**。

## 六、如果 Mochi 加类型注解会怎样

想象 Mochi 长出一个可选的类型注解层:

```mochi
fn add(a: Int, b: Int): Int { return a + b }
let x: Int = 5
let y: String = "hi"
add(x, y)   // 期望编译期报错
```

最简化的类型检查器骨架,**只检查赋值兼容**(变量声明 + 函数实参):

```typescript
type Type = "Int" | "Float" | "String" | "Bool" | "Any" |
            { kind: "fn"; params: Type[]; ret: Type };

class TypeChecker {
  env = new Map<string, Type>(); errors: string[] = [];

  check(node: AST) {
    if (node.kind === "LetDecl") {
      const t = this.infer(node.value);
      if (node.annotation && !this.assignable(t, node.annotation))
        this.errors.push(`expected ${node.annotation}, got ${t}`);
      this.env.set(node.name, node.annotation ?? t);
    } else if (node.kind === "Call") {
      const fn = this.env.get(node.callee);
      if (typeof fn === "object" && fn.kind === "fn") node.args.forEach((arg, i) => {
        const at = this.infer(arg);
        if (!this.assignable(at, fn.params[i]))
          this.errors.push(`arg ${i}: expected ${fn.params[i]}, got ${at}`);
      });
    }
    // ... If / While / Return / BinOp / FnDecl ...
  }

  infer(expr: AST): Type {
    if (expr.kind === "IntLit")    return "Int";
    if (expr.kind === "StringLit") return "String";
    if (expr.kind === "Var")       return this.env.get(expr.name) ?? "Any";
    if (expr.kind === "BinOp") {
      const l = this.infer(expr.left), r = this.infer(expr.right);
      if (l === "Int" && r === "Int") return "Int";
      if (l === "String" && r === "String" && expr.op === "+") return "String";
      this.errors.push(`bad op: ${l} ${expr.op} ${r}`);
    }
    return "Any";
  }
  // 极简——真实实现要支持子类型、union、泛型
  assignable(from: Type, to: Type): boolean { return to === "Any" || from === to; }
}
```

**这是 60 行,真 TS 的 `checker.ts` 五万多行**——差距就是「能推 / 能 union / 能 generic / 能 conditional / 能 control-flow-narrow」这些工业级特性,以及 N 万行的 edge case。

## 七、工业级是怎么做的

- **TypeScript**:`src/compiler/checker.ts`(**5 万行+**)——支持 structural typing、conditional types、template literal types、control flow narrowing。
- **Rust**:`compiler/rustc_hir_typeck/`(主类型检查) + `compiler/rustc_infer/`(推导,带生命周期约束) + `compiler/rustc_trait_selection/`(trait 求解)——**工业级里最难的之一**,因为要同时算类型、生命周期、所有权。
- **HM 完整实现**:OCaml (`typing/typecore.ml`)、Haskell GHC (`compiler/GHC/Tc/`)——一般扩展到 `let polymorphism`、`type classes` (Haskell) / `row polymorphism` (OCaml)。
- **Python type hints 检查器**:`mypy`、`pyright` (MS 出的,Pylance 底层)——**Python 自己不检查,要靠外部工具**。

**注意**:rustLearning 11-13 已经讲过 Rust 的泛型、Trait、生命周期——**那讲的是"怎么用"**,这里讲的是**"语言怎么实现类型系统"的视角**,不重复。

## 八、Mochi 这里偷懒了

- **不上类型注解**——Mochi 是动态语言,本篇只给个心智图,不真实现类型检查器。
- **不做 type tag 优化**——CPython 的 `PyObject` 头有 `ob_type` 指针;V8 用 SMI (small integer) tag 把整数和指针塞进同一个 word,**省一次解引用**。Mochi 用 TS 的 union 类型直接搞,性能爛但简单。
- **要加类型选渐进式**——像 Python type hints:注解可选,只在 lint 阶段检查,运行时擦除。**动态语言最少痛的路径**。
- **HM 留白**:没真写一个推导器(那要单开一篇),想自己写的看 Stephen Diehl 的《Write You a Haskell》或 Shriram Krishnamurthi 的 PLAI 对应章节。
