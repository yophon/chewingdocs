# 函数与闭包

这是全系列最关键的一篇之一。**闭包不是什么"高级特性"——闭包是函数式编程能跑起来的最低门槛**。map / filter / reduce、事件回调、async/await 的底层、React Hooks 的整个心智,全建立在「函数对象能记住自己出生时的环境」这件事上。这一篇画图说清楚两件事:**闭包捕获到底捕获了什么**(不是值,是环境引用)、**为什么 JS 的 `var` 在循环 + 闭包这对组合下会出经典 bug**。看完你应该能在白板前给同事讲清楚 `for (var i = 0; i < 3; i++) setTimeout(()=>console.log(i))` 为什么全打印 3——以及改成 `let` 为什么就好了。

> 一句话先记住:**闭包 = 函数代码 + 它出生时所在的环境的引用快照**。两个调用同一段代码生成的闭包,**指向不同的环境节点**——这就是 counter 各算各的、map 能并行用的根本原因。**JS 的 var 是设计错误**:让 for 循环里所有闭包共享同一个 i,新手 100% 踩一次。

---

## 一、一等公民 + 两种 callable

**一等公民 = 能赋值、能传参、能从函数返回**——没这一条就写不出 map / filter / reduce / 事件回调。C 有函数指针但**没有闭包**,所以 `pthread_create` / `qsort_r` / `epoll_event.data` 全靠多塞一个 `void* userdata` 让用户手动模拟"外部环境"——**那个 `void *` 就是手写的闭包**。

```text
let add = fn(a, b) { return a + b }                  // 赋值
[1,2,3].map(fn(x) { return x * 2 })                  // 传参
fn make_adder(n) { return fn(x) { return x + n } }   // 返回
```

Mochi 里两种东西可以被 `(...)`,**对外都是 `Callable`**:**MochiFunction**(用户写的 `fn`,带 `captures: Env`——闭包关键)和 **NativeFunction**(`print` / `len` / `clock`,宿主 TS 实现,塞进全局 env)。V8 统一成 `JSFunction`,CPython 统一成 `PyObject *`——**调用点不区分**。

---

## 二、调用语义:从 captures 派生,不是从调用方派生

调用 MochiFunction 三件事:**从 `captures` 派生新 env**(不是从调用方的 env 派生!)→ 参数绑到这个 env → 跑 body,catch ReturnSignal 当返回值。**最关键的一步**就是 env 从**函数定义时的 captures** 派生——这就是**词法作用域**(lexical scope)和**动态作用域**(dynamic scope)的区别。**99% 的现代语言用词法作用域**,因为动态作用域跟踪不可读、性能差(每次调用都得在调用栈里搜)。Emacs Lisp 早期是动态作用域,后来加了 `lexical-let`,因为受够了。

---

## 三、闭包的真正定义

教科书:闭包 = 函数 + 它定义时所在的环境。实际上**是函数对象里塞了一个对环境节点的引用**——"引用"是关键,不是抽象。

```text
fn make_counter() {
  var n = 0
  return fn() { n = n + 1; return n }
}

let c1 = make_counter()
let c2 = make_counter()
print(c1())   // 1
print(c1())   // 2
print(c2())   // 1   ← 各算各的
print(c1())   // 3
```

c1 和 c2 跑的是**同一段函数代码**(同一个 BlockStmt AST 节点)——为什么各算各的?**因为它们的 `captures` 字段指向不同的 Env 节点**:

```
                global env
                +-----------------------+
                |  c1 → MochiFn ────────┼──┐
                |  c2 → MochiFn ────────┼──┼──┐
                |  make_counter → Fn    |  |  |
                +-----------------------+  |  |
                                           |  |
   第一次 make_counter() 产生的 env:        |  |
   +----------+                            |  |
   |  n: 3    | ←── captures ──────────────┘  |
   +----------+   (c1 引用这里)                |
                                              |
   第二次 make_counter() 产生的 env:           |
   +----------+                               |
   |  n: 1    | ←── captures ─────────────────┘
   +----------+   (c2 引用这里)
```

两个 MochiFunction 对象,**`body` 字段指向同一个 AST 节点(代码共享)、`captures` 字段指向不同 Env 节点(状态独立)**。

**这张图是闭包的全部**。看不懂这张图,后面 async 回调、React state、Go 的 goroutine 捕获,所有"为什么不会互相干扰"都没法在白板前讲清楚。

## 四、引用捕获 vs 值捕获

c1 调用时 `n = n + 1` 改的是哪个 n?**是被 `captures` 引用的 Env 里的 n**——没有拷贝。

这就是**引用捕获**(by reference)。JavaScript、Python、Mochi 都是。**值捕获**(by value)在创建闭包时把变量值拷贝一份——C++ lambda 默认值捕获 `[=]`,显式引用捕获 `[&]`,**你必须选**:

```cpp
int x = 10;
auto f = [x]() { return x; };   // 值捕获,后续 x = 99 时 f() 仍返回 10
auto g = [&x](){ return x; };   // 引用捕获,后续 x = 99 时 g() 返回 99
```

**动态语言基本都选引用捕获**:简单、内存少、符合直觉。**代价**:下面这个 bug。

## 五、JS 的 var 为什么是设计错误

```js
var callbacks = []
for (var i = 0; i < 3; i++) {
  callbacks.push(() => console.log(i))
}
callbacks.forEach(f => f())   // 输出 3, 3, 3 —— 不是 0, 1, 2
```

新手心智:"循环里创建了 3 个闭包,各捕获 0/1/2"。**实际**:JS 里 `var` 是**函数作用域**——不管写在 for / if / 多少层 block 里,**都被提升到函数顶部**。代码等价于 `var i` 提升出来,整个函数**只有一个 i**:

```
              function 作用域 env
              +-----------+
              |  i: 3     |   ← 循环结束时 i = 3
              +-----------+
                ↑   ↑   ↑      3 个闭包的 captures 全部指向
                |   |   |       同一个 env、同一个 i (= 3)
          fn(){i} fn(){i} fn(){i}
```

**3 个闭包共享同一个 Env、同一个 i**——循环结束 i = 3,forEach 调用时全打印 3。

**ES6 的 `let` 修复了它**:`let` 是**块作用域**——for 循环每次迭代是一个**新的块作用域**,**每次迭代创建新的 i**:

```
        iter 1 env: { i: 0 } ←── fn(){i}
        iter 2 env: { i: 1 } ←── fn(){i}
        iter 3 env: { i: 2 } ←── fn(){i}
```

**JS 的 var 是设计错误**:让块作用域风格的循环看起来 OK 但语义是函数作用域,新手 100% 第一次踩。**Mochi 的 let / var 都是块作用域**——不重复这个错误。

## 六、TS 闭包实现

```ts
class Env {
  vars = new Map<string, Value>()
  constructor(public parent: Env | null = null) {}
  define(n: string, v: Value) { this.vars.set(n, v) }
  get(n: string): Value {                             // 沿 parent 链往上找
    if (this.vars.has(n)) return this.vars.get(n)!
    if (this.parent) return this.parent.get(n)
    throw new MochiError(`undefined: ${n}`)
  }
  set(n: string, v: Value): void { /* 结构同 get,只是命中后 set */ }
  extend() { return new Env(this) }
}

class MochiFunction implements Callable {
  constructor(public params: string[], public body: BlockStmt, public captures: Env) {}
  call(interp: Interpreter, args: Value[]) {
    const env = this.captures.extend()    // ← 闭包关键:从 captures 派生,不是从调用方
    for (let i = 0; i < this.params.length; i++) env.define(this.params[i], args[i])
    try { interp.execBlock(this.body, env) }
    catch (e) { if (e instanceof ReturnSignal) return e.value; throw e }
    return NIL
  }
}

function evalFnExpr(node: FnExpr, env: Env): MochiFunction {
  return new MochiFunction(node.params, node.body, env)   // ← env 就是闭包的 captures
}
```

**全部秘密就在 `evalFnExpr` 这一行**——把"当时的 env"塞给函数对象。

## 七、工业级是怎么做的

**V8(JavaScript)**:每个 JS 函数对象有 `[[Environment]]` 内部槽,指向创建时的 `Context` 对象(`v8/src/objects/contexts.h`)。**逃逸分析**只把真正被内层闭包用到的变量放进 Context——"一个空函数捕获 1000 个外层变量"不会真的全打包。

**CPython**:每个 PyFunctionObject 有 `__closure__`——**一个 tuple,装着所有 cell 对象**(`cpython/Objects/cellobject.c`)。cell 是个一格的容器,**多个闭包指向同一个 cell 就能共享变量**。比 Mochi 整个 env 塞过去省内存,但要在编译期做变量分析(`STORE_DEREF` / `LOAD_DEREF`)。

**Lua**:**upvalue**,每个被捕获的变量是单独一个槽(`lua/lobject.h::UpVal`)。分 **open**(变量还在栈上,upvalue 指过去)和 **closed**(栈帧弹了,变量复制到 upvalue 自身)两种状态——《The Implementation of Lua 5.0》论文把这套写得最清楚。

**Java**:lambda 只能捕获 effectively final 的变量——**不是闭包本质,是 Java 的取舍**:不想堆分配,**捕获的值/引用直接当字段塞进 lambda 对象**。严格说,Java lambda 只支持值捕获。

---

## 八、Mochi 这里偷懒了

**整个 Env 全引用着,不做 upvalue 优化**——比如 `fn outer() { let big = huge(); return fn() { return 42 } }`,闭包根本没用到 big,但因为 captures 引用着 outer 的整个 env,**big 不会被 GC**。**生产语言会做逃逸分析**:编译期看清楚返回的函数到底用了哪些外层变量,**只打包那些**,其余随栈帧一起回收。**25 篇字节码版函数调用与闭包**会重写这部分。**这里先简单——能跑就行**。
