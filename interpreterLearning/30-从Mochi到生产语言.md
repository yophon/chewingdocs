# 从 Mochi 到生产语言

30 篇了。从第 1 篇"解释器总览"到第 27 篇增量 GC、第 28 篇 JIT 心智、第 29 篇真实语言剖析——**Mochi 现在是一门什么都有的玩具语言**:词法、语法、AST、树遍历、字节码 VM、闭包、类、模块、try/catch、动态类型、mark-sweep GC。**性能比树遍历快 ~10x,但还是 CPython 3.11 的 1/10**。这一篇不再讲新概念——**全是路线指引**。如果你看完想继续走,这一篇告诉你下一步往哪去:WASM 后端、增量编译、LSP、GC 升级、JIT、类型系统、并发。**Mochi 不会变成 Lua / V8 那样的工业语言**——但你已经具备了看懂它们、甚至自己造一门小语言的所有基础。**这一篇之后,30 篇的训练就在你的手里**。

> 一句话先记住:**这门"反直觉"的训练,回报的是永远改变看代码方式的视角**——以后写 JS / Python / Rust,你脑子里有一张"这个特性怎么实现"的图。**看 V8 / CPython 源码不再像看天书,因为你昨天刚写过同样结构的 Mochi**。**Mochi 不是为了被部署而存在的,是为了让你看懂世界**。

## 一、Mochi 30 篇完成了什么

```
✓ 词法分析(05)             — 手写状态机 lexer
✓ 文法 + 递归下降(07-08)    — LL parser
✓ Pratt parser(09)         — 表达式优先级
✓ AST + 树遍历(10-15)      — 第一个能跑的版本
✓ 类 / 模块(16-17)         — OOP + 命名空间
✓ 渐进式类型系统(18)       — 类型注解 + 检查
✓ 元编程(19)              — eval / quote
✓ 字节码 VM(20-25)         — 栈式 VM + 编译器 + 闭包/upvalue
✓ Mark-sweep GC(26)        — 三色标记
✓ 增量 GC + 写屏障(27)     — 减少暂停时间
✓ JIT 心智(28)             — 不实现但建图
✓ 真实语言对照(29)         — 索引篇
```

代码量大概:**TypeScript 5-8k 行**。一个人 80-120 小时能完整写完。性能(估算):

| 实现 | 相对树遍历 | 相对 CPython 3.11 |
| --- | --- | --- |
| Mochi 树遍历(15 篇) | 1x | 0.05x |
| Mochi 字节码 VM(25 篇) | 10x | 0.5x |
| Mochi VM + GC(27 篇) | 10x | 0.5x |
| Mochi + 假想 JIT | 100x | 5x |
| CPython 3.11 | 20x | 1x |
| V8 / LuaJIT | 1000x+ | 50x+ |

**Mochi 的目标从来不是性能**——是让你**真的写过一次,看懂剩下那 50x 的 V8 在多干什么**。

## 二、真要把 Mochi 变成"能用的语言":7 条路线

每条都是一个**至少 2-3 个月**的工作量,选 1-2 条深挖就够长出一个完整的工程能力。

| # | 路线 | 难点 | 参考实现 | 预估 |
| --- | --- | --- | --- | --- |
| 1 | **WASM 后端**——字节码编到 WASM,浏览器里跑 | 静态类型 ABI,动态类型要 boxing | AssemblyScript / Grain | 3-6 月 |
| 2 | **增量编译**——改一个文件只重编受影响部分 | 依赖图 + 缓存失效 | Rust Salsa / Roslyn | 4-8 月 |
| 3 | **LSP**——VSCode 跳转 / 补全 / 重命名 | 增量解析 + 错误恢复 | rust-analyzer | 基础版 2-3 周 |
| 4 | **GC 升级**——分代 + 并发 + 写屏障 | 并发标记的内存屏障 | Lua 5.4 gen / Go runtime | 2-4 月 |
| 5 | **JIT**——上 LLVM 后端最省事 | LLVM 学习曲线陡 | Crystal / Numba | 3-6 月 |
| 6 | **类型系统**——HM 推导或 TS 风格渐进式 | 容易"动笔后做不完" | TS `checker.ts` / Roc | 6-12 月 |
| 7 | **并发模型**——async/await 最划算 | 状态机改写 | JS / Rust async | 4-8 月 |

**深一层说明**:

- **WASM**——浏览器是最大的运行时,**让 Mochi 进浏览器 = 任何人开箱试用**
- **增量编译**——**大项目编译速度 = 开发体验**,TS / Rust / Go 都在卷这个
- **LSP**——**没 LSP 的语言是死的**,21 世纪的语言必须能在编辑器里"活起来"
- **GC 升级**——减少暂停 = **直接让 Mochi 能跑游戏 / 服务器**
- **JIT**——不上 JIT,Mochi 性能就到 CPython 一半;**上了,理论上能到 LuaJIT 那个量级**
- **类型系统**——**大型代码库可维护性根基**,5k 行无类型 vs 500k 行有类型差距是数量级的
- **并发**——选 async/await,**有 30 年工业经验可参考**(JS / Rust / Python 都有)

## 三、真实"小语言"的成功案例

这些都是"一两个人就把它造出来还活下来"的语言——**比 Mochi 大一两个量级,但不到 V8 / CPython 那种**:

| 语言 | 诞生 | 作者 | 行数 | 生态地位 |
| --- | --- | --- | --- | --- |
| **Lua** | 1993 | Roberto Ierusalimschy 团队 | ~30k C | 嵌入式王者(WoW / Redis / Neovim) |
| **Wren** | 2014 | Bob Nystrom(《Crafting Interpreters》作者) | ~12k C | 嵌入式新秀 |
| **Janet** | 2017 | Calvin Rose | ~30k C | Lisp 系,做 DSL 利器 |
| **Roc** | 2019 | Richard Feldman | ~150k Rust | 纯函数式,生产力派 |
| **Gleam** | 2019 | Louis Pilfold | - | Erlang 生态,类型安全 |
| **Vale** | 2020 | Evan Ovadia | - | 内存安全 + GC-free |
| **AssemblyScript** | 2017 | dcode | - | TS-like → WASM,Web3 火过 |

**关键观察**:**Lua 30k 行**证明小团队 + 30 年专注可以做出工业级。**Wren 12k 行**——Bob Nystrom 业余项目,**说明语言不需要团队也能起步**。**Roc 已经 150k 行 Rust**——新语言起步即工业规模,**生态卷得越来越凶**。

**Mochi 不指望成为这些**——但**走完 30 篇之后,你能看懂他们的源码**,**这就是 Mochi 真正的价值**。

## 四、读者下一步该做的三件事

不是看更多书。**看更多书是逃避**。这三件事缺一不可:

### 4.1 给 Mochi 仓库提一个 issue 或 PR

任何东西都好:修 bug、加 builtin(`mochi.fs.readFile()`)、写文档、加 test case、修 typo。**从"读懂"到"能改"是一个数量级的跨越**——你提一个 PR,**就在 GitHub 上有了一条"我真的写过一门语言"的证据**。GitHub 仓库占位符:`<github.com/yourname/mochi-lang>`。

### 4.2 选一门真实语言读源码

**首选 Lua 5.4**,30k 行 C,周末 4 小时啃关键路径(29 篇有详细路线)。时间多再读 CPython 3.12(重点 `ceval.c` + `gcmodule.c`,~1 周通读)和 V8(从一个 builtin 比如 `Array.push` 反向追,**不要从 main 看**)。

**读源码的关键不是看懂每行**——**是建立"对照 Mochi"的地图**。看到 V8 的 `Code` 对象,你脑子里立刻反映"哦,这就是 Mochi 的 BytecodeChunk 工业版"——**这就是你已经赢了**。

### 4.3 找一个工作场景应用

**用 Mochi 教过的东西去解决一个真实问题**:配置 DSL(代替 YAML)、规则引擎(代替 if/else + 正则)、模板引擎、查询语言、插件系统(嵌入 Mochi 让用户写脚本扩展)。

**触发条件**:**下次你写正则的时候,问自己"这其实是个解析问题吗"**——**99% 的"复杂正则"其实应该是个小语言**。这就是 Mochi 30 篇真正想给你的本能。

## 五、系列结语

走完这 30 篇,**你应该已经感觉到了**:

你脑子里有了一张图——**写 JS 的时候**,看到 `obj.x.y.z`,**你下意识知道 V8 会查三次 hidden class、命中三次 IC**;**写 Python 的时候**,看到列表推导,**你脑子里浮现 `LIST_APPEND` 字节码**;**写 Rust 的时候**,看到 trait object,**你想起虚表跟 Mochi 类方法的对照**。

这就是"反直觉训练"的回报——**永远改变看代码方式的视角**。

以前你写 JS 卡 bug,**只能查 MDN**;现在你卡 bug,**可以打开 V8 源码搜关键词**——**因为你知道大概在哪一层**。以前你看到"为什么这段代码慢",**只能猜**;现在你看到,**脑子里有"dispatch 几次、IC 几次、可能 deopt 在哪里"的模型**。**这才是工程师跟"会写代码的人"的差别**——**不是知道更多 API,是知道得更深一层**。

看 V8 / CPython / Lua / HotSpot 源码,不再像看天书——**因为你昨天刚写过同样结构的 Mochi**。读 Bob Nystrom《Crafting Interpreters》、读 Thorsten Ball《Writing An Interpreter in Go》、看 Roberto Ierusalimschy 的 Lua paper——**你会发现这些大佬其实在讲同一件事,只是切面不同**。**你之前看不懂,因为缺手感;现在看得懂,因为手感是 30 篇代码堆出来的**。

**写解释器这件事不会让你升职**——但它会让你**下一个十年看代码的方式都不一样**。Mochi 是一次性的玩具,**你长出来的视角不是**。
