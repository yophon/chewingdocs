# 控制流

写解释器到这一步,if 和 while 看起来轻得离谱——递归调用 `eval(condition)`,为真就跑 body,几行 TS 完事。新手都觉得"控制流不就这样",直到写 `break` 的那一刻——傻眼。**break 不是 return**:return 退出整个函数,但 break 只想退出当前那个循环——它需要一种「**非局部跳转,但是有目标**」的机制。这一篇讲清楚 `break / continue / return / throw` 在树遍历解释器里**其实是同一类问题**:**怎么从深层递归里"穿"出来,准确停在某一层**。看完你会明白,用「返回值哨兵」实现 break 是新手最容易写错的一处。

> 一句话先记住:**树遍历解释器的控制流 = 用宿主的 if/while 跑普通流程 + 用宿主的异常机制实现 break / continue / return / throw 这四种非局部跳转**。看穿这四个东西是同源的,后面 try/catch 就不会再觉得"特殊"。

---

## 一、if / while:几乎不用写

```ts
function execIf(node: IfStmt, env: Env) {
  if (isTruthy(eval(node.cond, env))) exec(node.then, env)
  else if (node.else) exec(node.else, env)
}

function execWhile(node: WhileStmt, env: Env) {
  while (isTruthy(eval(node.cond, env))) exec(node.body, env)
}
```

就这样。**宿主的 `if` / `while` 直接借用**——我们用 TS 的 `while` 来跑 Mochi 的 `while`。Mochi 是动态类型树遍历语言,**控制流语义跟宿主完全对齐**——TS 的 `while` 怎么跳,Mochi 的 `while` 就怎么跳。直到 `break` 出现,这个"借用"才会撞墙。

---

## 二、break / continue:你不能用 return

新手第一反应:让 body 返回一个特殊值,外层循环看见就退出。

```ts
function execWhile(node: WhileStmt, env: Env) {
  while (isTruthy(eval(node.cond, env))) {
    const result = exec(node.body, env)
    if (result === BREAK) break          // ← 看起来合理
  }
}
```

听着 OK,**直到 break 出现在嵌套结构里**:

```mochi
while true {
  if some_cond {
    break       // ← 要穿透 if,落到 while 上
  }
}
```

`if` 的 exec 也得返回 BREAK,让外层 while 看到。再深一层——for 里套 if 里套 if 里套 break——**每一层都得透传 BREAK**。**每一种语句的 exec 都得记得检查并透传**,漏一处就 silent bug。

**根本原因**:break 是**非局部跳转**——它不关心中间有多少层 AST 节点,只关心"最近的循环"。但递归的 exec **每层只能控制自己**。

### 三种实现策略

| 策略 | 怎么做 | 代价 |
| --- | --- | --- |
| **异常机制** | `throw new BreakSignal()`,循环 try/catch | 简单,break 不是热路径,可接受 |
| **返回值哨兵** | exec 返回 `NORMAL / BREAK / CONTINUE / RETURN`,每层透传 | 啰嗦,**最容易漏** |
| **CPS** | 把 continuation 显式传下去,break 就是丢掉它 | 学院派,没人写树遍历用 |

**Mochi 选异常机制**——树遍历本来就慢,break 不是热路径;**代码量从"每层都改"变成"只在循环和函数边界 catch 一次"**,bug 表面积小一个量级。

---

## 三、Mochi 的实现:throw 一个信号

```ts
class BreakSignal { }     // ← 不继承 Error,只是 marker
class ContinueSignal { }

function execWhile(node: WhileStmt, env: Env) {
  while (isTruthy(eval(node.cond, env))) {
    try {
      exec(node.body, env)
    } catch (e) {
      if (e instanceof BreakSignal) return
      if (e instanceof ContinueSignal) continue
      throw e                            // ← 不是我管的,继续往上抛
    }
  }
}

function execBreak(_: BreakStmt) { throw new BreakSignal() }
function execContinue(_: ContinueStmt) { throw new ContinueSignal() }
```

三个关键点:

1. **BreakSignal 不继承 Error**——继承了 JS 引擎会给它生成调用栈,**性能差好几倍**;只要 `throw` 出来 `catch` 抓得到,任何 class 都行
2. **不是我管的就 rethrow**——`instanceof` 精确判断,绝不能把用户错误吞掉
3. **不需要标识"哪个循环的 break"**——break 总属于**最近的循环**,异常自然冒到最近的 try,正是想要的语义

---

## 四、for 循环就是糖

`for i in 0..10` 看起来是新东西,**实际上 parser 一行解糖就完事**:

```
for i in 0..10 { print(i) }

  ↓ parser 直接产出 while AST,evaluator 根本不知道 for 存在

{ var i = 0; while i < 10 { print(i); i = i + 1 } }
```

**这是解释器开发的关键直觉**——能在前端解糖的就别在后端实现两套。Mochi 的 `for x in array`、`for k, v in map` 也都这样,**evaluator 永远只看到 while**。

工业级同思路:**TypeScript / Babel 的 transpiler** 大量"语法到语法"的降级走的就是这套;**Rust 的 `for x in iter`** 编译期脱糖为 `loop { match iter.next() { Some(x) => ..., None => break } }`。

---

## 五、return 和 try/catch 也是同一件事

函数里的 return 跟 break 同源:**非局部跳转,目标是最近的函数边界**。

```ts
class ReturnSignal { constructor(public value: Value) {} }

function execReturn(node: ReturnStmt, env: Env) {
  throw new ReturnSignal(node.expr ? eval(node.expr, env) : NIL)
}

function callFunction(fn: MochiFunction, args: Value[]) {
  const env = fn.captures.extend()
  for (let i = 0; i < fn.params.length; i++) env.define(fn.params[i], args[i])
  try { exec(fn.body, env) }
  catch (e) {
    if (e instanceof ReturnSignal) return e.value
    throw e                            // ← break / 用户错误继续往上抛
  }
  return NIL
}
```

**callFunction 是 return 的"边界"**——抓到 ReturnSignal 就把 value 当返回值。

到这里你应该看出来了——**用户的 `try / catch` 本质就是带"目标 frame"的 break**:

```ts
function execTry(node: TryStmt, env: Env) {
  try { exec(node.body, env) }
  catch (e) {
    if (e instanceof BreakSignal || e instanceof ContinueSignal || e instanceof ReturnSignal)
      throw e                          // ← 信号不该被用户 catch 吞掉
    const catchEnv = env.extend()
    catchEnv.define(node.errVar, mochiErrorFrom(e))
    exec(node.catchBody, catchEnv)
  }
}
```

**break / continue / return / throw**,在树遍历解释器里**用宿主语言的异常机制统一实现**——这是树遍历版本最大的"取巧"。**20 篇字节码 VM 会重写**,这四个全是字节码里的 jump 或 frame unwind,因为 throw 太慢。

## 六、工业级是怎么做的

**CPython**:曾有 `BREAK_LOOP` / `CONTINUE_LOOP` 指令(3.11 起被 `JUMP_BACKWARD` / `POP_BLOCK` 替代),编译期就知道循环结构,**直接 jump 到字节码偏移,不抛异常**。指针:`cpython/Python/ceval.c`,搜 `TARGET(JUMP_BACKWARD)`。

**V8**:JS 源码里 break/continue 在 AST 阶段是 `BreakStatement` / `ContinueStatement`(`v8/src/ast/ast.h`),编译到 Ignition 字节码时变成 `Jump`——**目标偏移编译期算出**。

**Lua**:更激进,**栈式 VM 里 break 就是一条 `OP_JMP`**——零开销。Lua 还有个有趣设计:**continue 不是关键字**,用 `goto continue` 配合循环底部 label 实现。

**树遍历为什么不能这么做**:AST 是树,**没有"线性指令流"可以 jump**,只能往上抛或透传。**throw 是树形结构里实现"非局部跳转"最直接的方式**——本质上就是"沿调用栈往上找处理者",正是 break 想要的语义。

---

## 七、Mochi 这里偷懒了

**没做 labeled break**(Java / Rust 的 `break outer`,一跳两层及以上)。要实现:`BreakSignal` 带 label 字段,每个循环 catch 时检查——不匹配就继续往上抛。代码量小,但**用得到的场景极少**(嵌套深到要 label 时往往该重构成函数 + early return)。Rust / Java / Go / Python / JS 都支持(语法各异),Lua 不支持。**支不支持是哲学问题**——支持派觉得"显式比聪明好",不支持派觉得"嵌套深到要 label 的代码已经该重构了"。
