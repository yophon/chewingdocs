# GC 进阶

基础 mark-sweep 跑 Mochi 的玩具堆没问题——堆才几 MB,扫一遍微秒级。但 **JVM 跑一个百 GB 堆的服务**:扫一遍要几百毫秒,STW 期间所有线程冻结,**电商秒杀这一秒所有请求超时**。生产语言活不下去基础 mark-sweep。三条优化路线轮番上:**分代假设**(大部分对象朝生暮死,新生代单独管)、**增量**(把一次大 STW 拆成多次小停顿)、**并发**(mutator 和 collector 同时跑)。**三条路都靠写屏障**——一旦 GC 不再独占堆,任何"老对象指新对象"的赋值都得记账。这一篇拆 V8 / JVM / Go 都在走的这条路,**Mochi 永远不上,但你要知道为什么**。

> 一句话先记住:**分代 + 写屏障 + 增量 / 并发 = 生产 GC 的标配**。Mochi 不上,因为玩具堆几 MB 看不见 STW;上了反而代码上千行。但读 V8 / JVM / Go 源码,**这一篇的概念缺一不可**。

---

## 一、基础 mark-sweep 的痛:STW 太长

```
堆大小       mark+sweep 时间   STW 影响
1 MB         <1 ms             看不见
100 MB       ~50 ms            HTTP 抖动一下
10 GB        ~5 s              电商整条请求超时
100 GB       ~50 s             P99 直接爆炸
```

**问题本质**:mark 阶段要扫整个对象图,sweep 阶段要扫整个堆——**时间 O(堆大小)**。堆涨 100 倍,GC 慢 100 倍。

生产语言的优化目标:**STW 与堆大小解耦,做到亚毫秒级**。

---

## 二、分代假设:90% 新对象朝生暮死

**经验数据**(IBM / Sun 在 1990s 测过几乎所有真实程序):

```
新分配的对象,90%+ 在第一次 GC 时已经死了
活过一次 GC 的对象,大概率活很久(配置、缓存、长生命周期对象)
```

这就是**弱分代假设**(Weak Generational Hypothesis)。直接推论:**把"新对象"和"老对象"分开管**。

```
┌──────────────────────────────────────────────┐
│  young gen (新生代)                           │
│  ├─ eden(分配区)                             │
│  └─ survivor 0 / 1(幸存区,半区复制)        │
│  → minor GC,频繁但快                         │
├──────────────────────────────────────────────┤
│  old gen (老生代)                             │
│  → major GC,不频繁但慢(整堆 mark-sweep)    │
└──────────────────────────────────────────────┘
```

- minor GC:**只扫新生代**——新生代 90% 是死的,扫完 90% 都释放掉,**根本不碰老生代**
- 活过 N 次 minor GC 的对象 → **promote**(升代)到老生代
- old gen 的 major GC 才扫整堆,但触发频率低

**收益**:分配快(eden 是 bump pointer)、minor GC 快(扫的范围小)、大头长期对象不重复扫。

---

## 三、写屏障:老对象指向新对象的"小账本"

minor GC 只扫新生代 → 漏掉一件事:**老对象的字段指向新对象**:

```
old gen           young gen
┌─────────┐       ┌─────────┐
│ obj_old │──────→│ obj_new │
└─────────┘       └─────────┘
```

minor GC 从 GC roots 出发,**只走新生代内的引用**——`obj_new` 没人指(从新生代角度看),**被错误回收**。下次访问 `obj_old.field` 就是 use-after-free。

**修复**:**每次 `obj.field = newObj` 都拦截一下**,如果 obj 在老生代且 newObj 在新生代,把 obj 记进"remember set":

```
write barrier(obj, field, newObj):
  obj.field = newObj
  if obj 在 old gen and newObj 在 young gen:
    remember_set.add(obj)        // 这个老对象要算作新生代的"额外 root"

minor GC 时:
  扫 GC roots + 扫 remember_set 里的对象 → 新生代就不漏标了
```

**这就是为什么 Java / JS 的 `obj.x = y` 不是单条机器指令**——编译器要在背后插一段 write barrier 检查。Go 的 `*p = x`、JVM 的 `putfield` 都一样。

写屏障是有开销的——每次字段写都多几条指令。**HotSpot 用卡表(card table)**:把堆分成 512B 的卡片,只记"哪张卡有跨代引用",粒度粗但便宜。

---

## 四、增量 GC:把大停顿拆小

分代降低了**频率**,没降低单次老生代 GC 的**时间**。老生代 100GB 还是要扫 100GB。

**增量 GC**:把一次 mark 拆成 N 个小 step,**每个 step 跑几 ms 就让 mutator 跑回去**:

```
普通 STW:      ████████████████  100 ms 一次
增量:          █  █  █  █  █  █  10 ms × 10 次,中间穿插 mutator
```

但 mutator 会在 GC 中途改对象图,**会破坏三色不变式**:

```
出问题的场景:
1. GC 已经把 A 染黑(以为孩子都扫过了)
2. mutator 把 A.field 改成指向白对象 W(原来没人指 W)
3. GC 继续扫,W 永远没机会被染灰 → 错误回收
```

修复也靠**写屏障**:

```
增量 GC 的 write barrier:
  mutator 写 black.field = white_obj 时:
    把 white_obj 染灰,放进灰集合
  (或者把 black 退回灰,重新扫——两种屏障策略)
```

经典三色不变式:**强不变式**——黑不指白(任何指向白的引用必须从灰对象出发);**弱不变式**——黑能指白,但每个白对象必须能从某个灰对象可达。两种屏障对应两种维护策略。

---

## 五、并发 GC:mutator 和 collector 同时跑

增量是**时间维度**拆分——mutator 和 GC 轮流跑,只用一核。**并发**是**空间维度**拆分:mutator 在 N 个核上跑业务,GC 在另外几个核上扫堆,**真正同时**。

复杂度更高:三色不变式要在多线程下用原子操作 + 写屏障维护。Go 用**混合写屏障**(Dijkstra + Yuasa 的组合),把 STW 控制在**亚毫秒**:

```
Go GC 的 STW 阶段(整个 GC 周期):
  1. STW: 启动准备(<1 ms)
  2. 并发标记(几百 ms,但 mutator 跑着)
  3. STW: 标记终止(<1 ms)
  4. 并发清扫(mutator 跑着)
```

**关键**:即使堆几十 GB,STW 也是亚毫秒——**与堆大小解耦**。

## 六、Go / V8 / JVM 怎么走的

| 语言 | 分代 | 增量 | 并发 | 写屏障 | 典型 STW |
| --- | --- | --- | --- | --- | --- |
| Mochi | 无 | 无 | 无 | 无 | 全堆扫 |
| CPython | 无(refcount 主)+ cycle | 无 | 无 | 无 | 取决于 cycle 大小 |
| Lua 5.1 | 无 | 增量 | 无 | 有 | 几 ms |
| Lua 5.4 | 有 | 增量 | 无 | 有 | 几 ms |
| V8 (Orinoco) | 有 (young/old) | 增量 | 并发 mark+sweep | 有 | 亚 ms |
| HotSpot G1 | 有(分区伪分代) | 增量 | 并发 mark | 有 | 10-200 ms |
| HotSpot ZGC | 无(着色指针) | - | 全并发 | 有(读屏障) | <1 ms,堆 16TB |
| Go | 无(只有一代) | 增量 | 并发 | 混合 | <0.5 ms |

注意几个反直觉的点:

- **Go 没分代**——Go 团队压测发现,**逃逸分析 + 栈分配**已经把"短命对象"挡在堆外了,堆上的对象本来就长命,**分代收益有限**
- **ZGC 没分代**(早期版本)——靠**着色指针**(在地址高位塞 mark 信息)+ **读屏障**做并发,堆 16TB 都能 <1 ms STW
- **V8 用半空间复制**(Cheney 算法)做 young GC——young gen 死得太干净,**复制 10% 活的比标记清扫整个 young 还快**

## 七、为什么 Mochi 永远不上

| 优化 | 收益(玩具堆) | 代码量 | 性价比 |
| --- | --- | --- | --- |
| 分代 | 几乎无(GC 本来就快) | +500 行 | 负 |
| 写屏障 | 无 | +200 行(且每次字段写多开销) | 负 |
| 增量 | 无 | +300 行(整个调度循环要重写) | 负 |
| 并发 | 无(Mochi 单线程) | +1000 行 | 负 |

**结论**:玩具语言上这些纯属自虐。Mochi 的 50 行 mark-sweep 跑 100 KB 堆,**STW 几十微秒**,看不见。Mochi 永远只做基础 GC——读者把这一篇当"工业级 GC 路线图",**实际去读 V8 / Go 源码就有地图了**。

## 八、RC + tracing 混合:CPython 的折衷

**CPython 是有趣的反例**:主力 refcount(每次赋值 `++`/`--`),周期性跑 tracing(`gc` 模块)只为了打破循环引用:

```
对象的生命:
  创建时 refcount = 1
  每次赋值给变量    → refcount++
  变量离开作用域    → refcount--
  refcount == 0     → 立即释放(递归 decref)
  
gc.collect() 周期触发:
  只扫"可能成环的容器对象"(list/dict/set/instance)
  用 tracing 检测无外部引用的循环,释放整个环
```

**为什么 CPython 这么选**:1989 年的代码,refcount 是当时唯一现实方案;且 Python 的 `with` / `__del__` 语义假设**对象死后立即调析构**,纯 tracing 做不到这点。代价:**GIL 拆不掉**(refcount 不能无锁多线程),**整个 PEP 703 移除 GIL 都在解决这个**。
