# 函数调用与闭包

打开一段 Mochi 代码 `add(1, 2)`,在树遍历版本里要新建 Environment、塞参数、递归求值——**一次调用多创建几个对象 + 多查几次 hashmap**。字节码 VM 把这套全砍掉:**调用 = 在栈上摞一帧,参数已经在栈上,局部变量按偏移读**——快 10 倍。但闭包打破美好:**内层函数引用的外层变量,在外层函数返回后还要活着**——只能"上堆",这就是 upvalue。Lua / Crafting Interpreters 的 open/close upvalue 是本系列最绕的一段,**不画图永远讲不清**。

> 一句话先记住:**调用 = 在 VM 栈上摞 call frame,局部变量 = SP 相对 base 的偏移;闭包 = 把"逃逸"的栈上变量提升到堆,用 upvalue 间接访问**——Lua / clox / V8 同一套思路。

---

## 一、调用栈:一摞 CallFrame

VM 维护**两个**栈:

```
┌────────────┐      ┌──────────────────┐
│ data stack │      │  call frame stack│
├────────────┤      ├──────────────────┤
│ Value      │      │ CallFrame {      │
│ Value      │      │   closure        │
│ ...        │      │   ip             │
└────────────┘      │   base           │
                    │ } ...            │
                    └──────────────────┘
```

`CallFrame` 三个字段:`closure`(当前函数对象)、`ip`(在 bytecode 里的偏移)、`base`(在 data stack 的起始偏移)。**分两根栈的好处**:指令分发只看 ip,少一次间接寻址 = 解释循环每秒多几百万条指令。

---

## 二、CALL 和 RETURN 究竟在做什么

调用 `add(1, 2)` 的字节码:

```
LOAD_GLOBAL fn_add   ; data stack: [..., fn_add]
CONST 1              ;             [..., fn_add, 1]
CONST 2              ;             [..., fn_add, 1, 2]
CALL 2               ; argc = 2
```

CALL 做的:

```
base = stack.length - argc - 1       ; -1 是函数本身的槽
push CallFrame { closure, ip: 0, base }
解释循环继续 —— 下一轮 fetch 用新 frame 的 ip
```

RETURN 做的:

```
ret_val = pop()                       ; 函数体最后 push 的返回值
pop 当前 frame
stack.length = base                   ; 截掉 fn + 参数 + 局部变量
push ret_val                          ; 调用者拿到返回值
```

`add(1, 2)` 期间的三张栈快照:

```
调用前:                调用中:                返回后:
data stack             data stack             data stack
┌─────────┐            ┌─────────┐            ┌─────────┐
│ ...     │            │ ...     │            │ ...     │
│ fn_add  │            │ fn_add  │← base      │ 3       │
│ 1       │            │ 1       │  (a)       └─────────┘
│ 2       │            │ 2       │  (b)
└─────────┘            │ 3       │← SP        frame stack
                       └─────────┘            [ main ]
frame stack            frame stack
[ main ]               [ main, add_frame ]
```

`a` 和 `b` 在 `add` 内就是 `base+1` / `base+2`——**编译期编死**。

## 三、局部变量 = SP 偏移(为什么快 10 倍)

树遍历版本里变量查找走 hashmap:每次都算 hash、比 key、可能多次跳转外层。字节码版本里编译器(第 24 篇)早把每个变量编号:

```text
fn add(a, b) {
  var c = a + b
  return c
}
```

槽位表:

```
stack[base+0]  fn_add       (函数自己)
stack[base+1]  a            参数 1
stack[base+2]  b            参数 2
stack[base+3]  c            局部变量
```

`LOAD_LOCAL 3` 直接读 `stack[base + 3]`——**一次数组访问,零 hashmap 查找**。这就是 VM 快 10 倍的主因。

## 四、闭包之痛:捕获的变量怎么活下去

典型例子:

```text
fn make_counter() {
  var n = 0
  return fn() { n = n + 1; return n }
}

var c = make_counter()
print(c())   // 1
print(c())   // 2
```

`make_counter` 返回后,**frame 被 pop,栈上的 `n` 会被后续调用覆盖**。可 `c` 还要继续读写 `n`——**`n` 必须搬到不会被覆盖的地方**。这就是逃逸(escape):变量的生命被内层闭包延长,超过了定义它的函数。

树遍历版本不操心 —— 每个函数自带一个 Environment 对象,**整个 env 在堆上**。代价:即使闭包只用 `n`,整个外层 env 都跑不掉;读写永远走 hashmap。字节码版本要兼得性能和正确性,就得**分情况**:没逃逸的留栈上,逃逸的上堆。

## 五、Upvalue:Lua / Crafting Interpreters 的方案

闭包对象自带一个 upvalues 数组,**每个 upvalue 是一个"指针",可能指栈也可能指堆**。

```typescript
interface Upvalue {
  location: { stack: number } | { closed: Value }
}
interface Closure { fn: FunctionObject; upvalues: Upvalue[] }
```

新增 4 条字节码:

```
CLOSURE <fn_idx>      ; 把常量池里的 FunctionObject 包成 Closure
                       ; 紧跟每个 upvalue 的 (isLocal, index) 元数据
GET_UPVALUE <idx>     ; 读 closure.upvalues[idx] 指向的值
SET_UPVALUE <idx>     ; 写 closure.upvalues[idx] 指向的值
CLOSE_UPVALUE         ; 变量离开作用域前把它"上堆"
```

`CLOSURE` 后的元数据编译期就知道:第 i 个 upvalue 是直接捕获外层 frame 的局部(`isLocal=true`),还是从外层闭包"穿透"过来的(`isLocal=false`,走外层 closure 自己的 upvalues)。

## 六、open → close:逃逸时刻的"上堆"

`make_counter` 还在跑时,内层闭包 `c` 已创建,但 `n` 还在栈上:

```
data stack                    closure c
┌──────────────┐              ┌──────────────────┐
│ ...          │              │ fn: <anon>       │
│ fn_make_ctr  │← base of MC  │ upvalues: [      │
│ n = 0        │← (slot 1)←┐  │   open ─────────┐│
│ c (closure)  │           │  │ ]                ││
└──────────────┘           │  └──────────────────┘│
                           └──────────────────────┘
                          (upvalue.location.stack = base+1)
```

`make_counter` 即将 RETURN 时,编译器在 RETURN 前插了 `CLOSE_UPVALUE`(对这层 frame 所有还 open 的 upvalue 关一遍):

```
1. 把栈槽 n 的当前值复制到一个堆上的 Upvalue 对象
2. 找到所有指向这个栈槽的 open upvalue
3. 把它们的 location 改成 { closed: ... },指向堆对象
```

转换后:

```
heap                              closure c
┌──────────────────┐              ┌──────────────────┐
│ Upvalue {        │←──┐          │ fn: <anon>       │
│   closed: 0      │   │          │ upvalues: [      │
│ }                │   └──────────│   closed ───────┐│
└──────────────────┘              │ ]                ││
                                  └──────────────────┘│
                              (upvalue.location.closed)
```

之后调 `c()`:

```
GET_UPVALUE 0     → 读堆对象 closed 字段 = 0
CONST 1
ADD               → 1
SET_UPVALUE 0     → 写堆对象 closed = 1
GET_UPVALUE 0     → 读出来 = 1
RETURN
```

**关键**:`GET_UPVALUE / SET_UPVALUE` 实现统一 —— 看 location 字段是 stack 还是 closed,**对上层透明**。

## 七、TypeScript 50 行核心实现

```typescript
interface Upvalue { location: { stack: number } | { closed: Value } }
interface Closure { fn: FunctionObject; upvalues: Upvalue[] }
interface CallFrame { closure: Closure; ip: number; base: number }

class VM {
  stack: Value[] = []
  frames: CallFrame[] = []
  openUpvalues: Upvalue[] = []  // 真实现用按栈位置排序的链表

  call(closure: Closure, argc: number) {
    const base = this.stack.length - argc - 1
    this.frames.push({ closure, ip: 0, base })
  }

  ret() {
    const value = this.stack.pop()!
    const frame = this.frames.pop()!
    this.closeUpvaluesAbove(frame.base)
    this.stack.length = frame.base
    this.stack.push(value)
  }

  captureUpvalue(stackIndex: number): Upvalue {
    for (const uv of this.openUpvalues) {
      if ('stack' in uv.location && uv.location.stack === stackIndex) return uv
    }
    const uv: Upvalue = { location: { stack: stackIndex } }
    this.openUpvalues.push(uv)
    return uv
  }

  closeUpvaluesAbove(threshold: number) {
    this.openUpvalues = this.openUpvalues.filter(uv => {
      if ('stack' in uv.location && uv.location.stack >= threshold) {
        uv.location = { closed: this.stack[uv.location.stack] }
        return false
      }
      return true
    })
  }

  getUpvalue(idx: number): Value {
    const uv = this.frame().closure.upvalues[idx]
    return 'closed' in uv.location ? uv.location.closed : this.stack[uv.location.stack]
  }
  setUpvalue(idx: number, v: Value) {
    const uv = this.frame().closure.upvalues[idx]
    if ('closed' in uv.location) uv.location.closed = v
    else this.stack[uv.location.stack] = v
  }
  frame() { return this.frames[this.frames.length - 1] }
}
```

跑通这 50 行,Mochi 的闭包就工作了。

## 八、Mochi 偷懒了 / 工业级指针

| Mochi 偷懒 | 真实语言 |
| --- | --- |
| openUpvalues 用数组扫描 | Lua 用按栈位置降序链表,O(1) 定位 |
| 多个闭包共享同一 upvalue 没去重 | Lua `luaF_findupval` 做了共享 |
| Upvalue 让宿主 V8 GC 兜底 | clox 自己管 GC,upvalue 是 GC root |
| 没区分多层穿透的 deep upvalue | 编译期就生成中间层的 upvalue 链 |

**源码指针**:

- **Lua**: `src/lfunc.c` 的 `luaF_close` / `luaF_findupval`——upvalue 结构 + close 逻辑共 100 行,**比这篇文章短**
- **Crafting Interpreters 第 25 章 "Closures"**: clox 的实现几乎是 Lua 的 C-to-C 翻译
- **V8**: `Context` 对象(`src/objects/contexts.h`)—— 没有 upvalue 这个词,但每个闭包带 `[[Environment]]` 链;JS 的 `var` 提升让 V8 直接把整个 Context 堆分配,**靠 TurboFan 的 escape analysis 把没逃逸的栈分配**
- **JVM**: Java 匿名内部类编译为"外层对象 + `this$0` 引用",**没有 upvalue 抽象**,所以 Java 闭包只能捕获 final 变量
