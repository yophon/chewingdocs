# 变量与作用域

表达式能算了,但 `let x = 10; x + 1` 这种最简单的代码 Mochi 还不会——`x` 是什么?这就要**作用域**(scope):每个名字在每个位置该取哪个值的查找规则。**作用域是动态语言里最容易被低估的复杂源**——闭包能不能捕获、模块隔不隔离、变量遮蔽合不合理、热重载该刷哪一层,**全都是作用域决定的**。这一篇先把简单的 Environment 链表 + 词法作用域搞清楚,**14 篇的闭包就只需要一行新代码**。选错(动态作用域 / Hoisting / `var` 提升)会让你之后 15 篇全在打补丁,**Lisp 60 年前踩过、JS 30 年前踩过、Mochi 不踩**。

> 一句话先记住:**作用域 = 名字 → 值的查找规则**,Mochi 用 **Environment 链表 + 词法作用域**——每个 block 一个 env,parent 指向外层,**变量在哪里写就在哪里查**(不在哪里调用)。**这是 JS / Python / Rust / Go 的共同选择**。

---

## 一、Environment:用链表实现作用域

最简的想法——一个 `Map<string, Value>` 存所有变量:

```ts
const globals = new Map<string, Value>();
globals.set("x", 10);
```

但这不行——**函数有自己的局部变量**,不能污染全局。所以每进入一个 block / 函数,**push 一层新的 Map**,parent 指向外层:

```
当前 env (函数 inner)        // x = 99
    ↓ parent
外层 env (函数 outer)        // a = 1, b = 2
    ↓ parent
全局 env                     // x = 10, print = <builtin>
```

查 `a` 在当前层找不到 → 顺 parent 链往上找 → 在 outer 层找到 1。

**查 `x` 会先在当前层找到 99,而不是全局的 10**——这就是**变量遮蔽**(shadowing),按 parent 链最近的赢。

---

## 二、词法作用域 vs 动态作用域

两种查找规则,**看起来都合理,选错全错**。

**词法作用域(lexical scope)**:看**源代码里函数写在哪**——

```text
let x = "outer"
fn f() { print(x) }
fn g() {
  let x = "inner"
  f()        // 打印 "outer",因为 f 定义时旁边的 x 是 outer
}
g()
```

**动态作用域(dynamic scope)**:看**函数被谁调用**——同样代码会打印 `"inner"`,因为 `f` 在 `g` 里被调,沿调用栈往上找到的 `x` 是 `g` 的 inner。

| 维度 | 词法作用域 | 动态作用域 |
| --- | --- | --- |
| 决定规则 | 函数**定义**位置 | 函数**调用**位置 |
| 闭包能否捕获 | 可以(定义时 env 已知) | 没意义(调用时才知道) |
| 静态分析 / IDE | 好做(读源码就行) | 几乎做不了 |
| 代表语言 | JS / Python / Rust / Go / Java / Mochi | 早期 Lisp、Emacs Lisp、bash 变量 |

**为什么现代语言全选词法**:**可读性**(读源码就知道变量从哪来)、**闭包成立的前提**(14 篇会讲,无词法作用域则无闭包)、**静态优化好做**(下面第四节的变量解析 pass)、**LSP 跳转 / 重命名好做**——Emacs Lisp 后来加 `lexical-binding: t` 切换到词法作用域,**几十年的弯路被官方承认走错了**。

## 三、Environment 的 TS 实现:25 行搞定

```ts
class Env {
  private vars = new Map<string, Value>();
  constructor(public parent: Env | null = null) {}

  define(name: string, value: Value): void {                     // let / var / 参数
    if (this.vars.has(name)) throw new SyntaxError(`重复定义: ${name}`);
    this.vars.set(name, value);
  }

  lookup(name: string): Value {                                  // 本层 → parent → 全局
    if (this.vars.has(name)) return this.vars.get(name)!;
    if (this.parent) return this.parent.lookup(name);
    throw new ReferenceError(`未定义: ${name}`);
  }

  assign(name: string, value: Value): void {                     // 必须已存在
    if (this.vars.has(name)) { this.vars.set(name, value); return; }
    if (this.parent)         { this.parent.assign(name, value); return; }
    throw new ReferenceError(`未定义: ${name}`);
  }
}
```

求值器入 block 就 `new Env(currentEnv)`,出 block 就用 parent。**链表自动管理生命周期**——JS 的 GC 在没引用时回收旧 env;14 篇闭包能成立,**正是因为 MochiFn 持有了定义时的 env 引用,GC 不会回收**。

## 四、变量解析 pass:把 O(n) 查找压成 O(1)

`lookup` 一层层往 parent 走,**深嵌套时每次查变量是 O(n)** 的 Map 操作。**优化**:求值前先扫一遍 AST,**给每个变量引用打上 `{ depth, slot }` 标签**——查变量直接跳 depth 层 + 取 slot,O(1):

```
未优化:   evaluate("a") → lookup → 当前层 miss → parent miss → 全局 hit   (3 次 Map 查找)
优化后:   evaluate("a") → ancestor(2).slot[0]                            (1 次数组访问)
```

V8 / CPython / Lua 编译期都做了这件事——**编译产物里压根没有"变量名",只有 slot 索引**。Mochi 14 篇会做,树遍历版本不强制(性能差 5-10 倍,但学习版可以接受)。

## 五、Hoisting:JS 的奇怪行为,Mochi 不做

JS 的两个坑爹设计:

```js
console.log(x);                              // undefined(不是 ReferenceError!)
var x = 1;
foo();                                       // 能跑(函数声明被提升)
function foo() { console.log("hi") }
```

**JS 解析阶段把 `var` 声明和函数声明的"名字"提到 scope 顶部**——这就是 **hoisting**。这是 Brendan Eich 10 天里赶出来的妥协,代价是 20 年后 JS 程序员都改用 `let`(ES6 引入,无 hoisting,**声明前访问是 TDZ**——Temporal Dead Zone,报错)。

**Mochi 全部不做**——`let` 和 `var` 都是**词法绑定,声明前访问就是 ReferenceError**:

```text
print(x)    // 报错:未定义 x
let x = 1
```

简单、安全、没坑——**Mochi 的立场:不重复 JS 的设计错误**。

## 六、闭包 sneak peek

```text
fn makeCounter() {
  var count = 0
  return fn() { count = count + 1; return count }
}
let c = makeCounter()
c()   // 1
c()   // 2
```

**问题**:`makeCounter` 已经返回了,`count` 为什么还活着?**答案**:Mochi 函数值不只是"代码指针",还**捕获了定义时的 env**——只要这个函数还有引用,env 就不被 GC,`count` 跟着活下来:

```
c (MochiFn) ──→ closure { body: <fn AST>, env: <count = N> }
```

**词法作用域是闭包成立的前提**——定义时就能拿到外层 env,**调用时不依赖调用者环境**。动态作用域下这种"逃逸"无从谈起。14 篇展开。

## 七、工业级是怎么做的

| 项目 | 作用域 / Env 相关文件 | 关键设计 |
| --- | --- | --- |
| V8 | `src/objects/scope-info.h` | 编译期把 scope 信息序列化成 `ScopeInfo`,运行时按 slot 取 |
| CPython | `Python/symtable.c` + `Python/compile.c` | symtable pass 产出 LOAD_FAST / LOAD_DEREF / LOAD_GLOBAL 专用字节码 |
| Lua | `src/lparser.c` 的 `singlevar` | 解析时直接判断局部 / upvalue / 全局,装进字节码 |
| Rust | `compiler/rustc_resolve` 整个 crate | name resolution 单独一大 pass,模块路径 + 作用域全在这里 |

CPython 的 `symtable.c` 特别值得读——**80% 的"Python 变量怎么生效"**(global / nonlocal / 类作用域特殊规则)都在这一个文件里。Mochi 12 篇相当于把它**用 100 行 TS 重写一遍**。

## 八、Mochi 这里偷懒了

| 偷懒点 | 真实语言怎么做 |
| --- | --- |
| `let` / `var` 不严格区分,都按 lexical | JS:`let` block scope,`var` function scope,TDZ 行为不同 |
| 不支持 `nonlocal` / `global` 关键字 | Python 必须显式声明才能改外层 |
| Env 用 Map,不做 slot 优化 | V8 / CPython 编译期把变量编号成数组索引 |
| 不做声明前的 TDZ | JS `let` 在声明前访问报错(不是 undefined) |
| 模块作用域和文件作用域不分 | Python / Rust / JS 都有完整模块系统(17 篇补) |

**核心立场**:Mochi 12 篇先把"作用域 = env 链表 + 词法查找"立起来——这是 14 篇闭包、16 篇类、17 篇模块的共同前菜,**所有"名字"相关的特性都要先有这一层**。等钩子打好,后面每篇只往这棵树上挂一个东西。
