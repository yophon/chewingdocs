# 解释器系列写作计划

> 来源:`未来系列规划.md` 第五类「推荐组合」特别推荐项,⭐⭐⭐⭐。
> 写一门解释器是「最反直觉但回报极高」的训练——动一次手,JS / Dart / Python / Rust 里所有看起来"语言自带"的东西(作用域 / 闭包 / GC / 异常 / 模块)就突然全部塌缩成"你昨天刚写过的几十行代码"。
> **看完这层,你读 V8 / CPython / JVM 源码会快一个量级**,写 DSL / 模板引擎 / 配置语言 / 规则引擎时会长出「解析器的本能」,而不是再用一堆 if/else 和正则硬拼。

---

## 设计思路

六层结构,**贯穿动手实现一门小语言 Mochi**——从源码字符流一路推到字节码 VM 跑起来,再补 GC 和 JIT 的心智:

```
心智与基础(解释器 vs 编译器 vs JIT / 一段代码怎么跑起来)
   → 前端(词法 / 语法 / 递归下降 / Pratt parser)
   → 树遍历解释器(AST / 求值 / 作用域 / 闭包)
   → 进阶语义(类 / 模块 / 类型系统 / 元编程)
   → 字节码虚拟机(为什么要 VM / 栈式 vs 寄存器式 / 编译器)
   → 运行时与未来(GC / JIT / 真实语言剖析)
```

**这个系列不是《Crafting Interpreters》翻译**——那本书结构是教学,先把整本 Lox 一次性走完;**本系列偏"工程师视角"**:每篇都对应一个"语言里你天天用却没拆过的特性",讲清楚它怎么实现、工业级是怎么做、有哪些优化方向。

不深入到编译原理证明、不啃龙书,**而是讲清楚"应用工程师能动手做、需要懂的语言实现图景"**——读完应该:

1. 手上有一门能跑的小语言 Mochi,从树遍历版本一直到字节码 VM 版本
2. 看 CPython / V8 / Lua 源码大致能对上号
3. 下次写配置 / DSL / 规则引擎,**第一反应是"先定义文法",不是写正则**

参考蓝本:Bob Nystrom《Crafting Interpreters》、Thorsten Ball《Writing An Interpreter in Go》/《Writing A Compiler in Go》、Andrew Appel《Modern Compiler Implementation》。

## 小语言:Mochi

> mochi(麻糬)——软、糯、Q弹、能塞进各种馅料。
>
> Mochi 是本系列贯穿始终的玩具语言,目标定位:
>
> - 动态类型,语法风味介于 Python 和 JS 之间
> - 一等公民函数 + 闭包 + 类 + 模块
> - 树遍历版本(01-19)→ 字节码 VM 版本(20-25)→ 加 GC(26-27)
> - 宿主语言:TypeScript 为主(读者基数广),关键算法附 Rust 版本(贴近工业级)

每一篇的代码都在前一篇的 Mochi 仓库上叠加,**不是孤立的玩具**——25 篇之后,Mochi 应该能跑一段含闭包、类、模块、try/catch 的脚本,并在字节码 VM 上比树遍历快 5-10 倍。

## 写作模板

每篇按"它是什么 → 最简实现 → 工业级是怎么做的 → 优化方向"四段式:

```
1. 它是什么 / 为什么需要它
   - 一句话定义
   - 没它的世界长什么样(去掉这个特性,Mochi 哪里就不能写了)

2. Mochi 里的最简实现
   - 30-80 行 TypeScript,跑得通就行
   - 重点在结构,不在边界 case

3. 工业级是怎么做的
   - V8 / CPython / Lua / JVM 选 1-2 个,对比关键差异
   - 真实数据结构 / 真实优化

4. 优化方向与下一步
   - 这一篇留了哪些坑
   - 后面哪一篇会回来填(或者本系列不填,作为留白)
```

不堆完整源码——**关键算法只贴 30-80 行 TS,工业级实现只给指针**(类名 / 文件路径 / commit hash),让读者「看一眼知道这一层在干嘛」就够了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「先讲编译原理证明 / 龙书第三章」——本系列是工程师视角,不证明文法等价性
2. 「先讲个语言史 / 个人情怀」——直接进重点,Lisp 是 1958 年发明的,读者来这里不是为了知道这个
3. 罗列所有 token / 所有 AST 节点——只贴关键 5-10 种,剩下读者自己长出来
4. 「这是面试常考」——读者来这里是为了真的会写解释器,不是背名词
5. 抄《Crafting Interpreters》逐章翻译——那本书在那儿,这里补它没讲透的"为什么"
6. 不画图就讲 AST / 闭包捕获 / GC 三色标记——这些东西**不画图永远讲不清楚**
7. Mochi 半路换语言风格——一旦定下来,30 篇都是同一门 Mochi,不要写到 20 篇又改语法

## 代码语言

宿主语言统一用 **TypeScript**,关键算法附 **Rust** 版本:

- TS 读起来像 Java / JS / C# 的平均,业内接受度最高,跟 webLearning / backendLearning 的代码风格一致
- Rust 贴近工业级解释器(rustc / Deno / 部分 V8 子模块)的工程现实,适合在 GC / VM 篇章引用
- Mochi 源码本身是另一门语言,在文章里以代码块出现,**不需要用任何语言编译它**——Mochi 的实现器(用 TS 写)就是本系列的产物

不用 Python:Python 写解释器对读者最友好,但**会让性能讨论失去说服力**——后面 VM 篇章要测树遍历 vs 字节码的实际加速比,宿主语言不能太慢。

## 完成的判据

每一篇都要满足三件事才能打 [x]:

1. Mochi 仓库有对应的 commit / tag,**`pnpm test` 全过**
2. 文章里有至少一张图(AST / 字节码 / 内存布局 / 调度流程,选一张)
3. 「工业级是怎么做的」那一段,**至少给出一个 V8 / CPython / Lua / JVM 的具体文件路径或 commit**——没指针就是没看源码,不能糊弄

这三条任何一条没满足就当未完成,**回头补**——本系列最怕的就是"看起来写完了,但代码不能跑"。
