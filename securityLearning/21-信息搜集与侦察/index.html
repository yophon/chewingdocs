    信息搜集与侦察:OSINT / 子域名 / 端口扫描 / 指纹识别

    从这一篇开始进入第五层——渗透与攻防实战。但在讲怎么打之前必须先讲一件事:真实攻击者花在「踩点」上的时间,远超花在「利用漏洞」上。红队报告里写「48 小时拿到域控」,前 36 小时基本都在做侦察——梳理资产、扫端口、找指纹、翻 GitHub。这恰恰是防御方最容易忽略的阶段——日志里看不见,WAF 拦不住,SOC 报警不响。攻击者已经把你的整个外部攻击面摸了三遍,你的安全团队还不知道有人在看。这一篇从攻击者侧入手,讲清四件事:OSINT、子域名、端口扫描、指纹识别;然后反推防御方该堵什么——外部攻击面管理(EASM)和「把自己当攻击者扫一遍」的常态化。

    一句话先记住:侦察阶段攻击者只是在「读」你的资产,但你 95% 的安全事故,根因都是「有个你不知道存在的东西暴露在公网上」——一个测试环境忘了关、一个子公司域名没清单、一个员工把 AWS key 推到了 GitHub。防御侦察 ≠ 阻止扫描(挡不住),而是先于攻击者把自己摸一遍——让外部攻击面比你以为的更小、更清楚。这一篇的核心心智不是「学几个工具」,是「资产清单 = 安全的地基,清单不准,后面一切防御都是空的」。

    一、为什么侦察是攻击者最看重、防御方最忽略的一步

    1.1 渗透杀链里侦察的位置

    Lockheed Martin 的 Cyber Kill Chain 七步:Reconnaissance → Weaponization → Delivery → Exploitation → Installation → C2 → Actions on Objectives。有意思的是,后六步公司都有产品在卖——EDR 管 4/5/6,SIEM 管 6/7,DLP 管 7。只有第 1 步几乎没有"成熟商业产品挡得住"。原因很简单:侦察基本不碰你的资产,大多数动作发生在你看不见的地方——查证书透明度日志、查 GitHub、查 Shodan 历史快照。你拦不了 attacker 在百度搜你公司的人。

    1.2 防御方的三个认知陷阱

    「我们外网只暴露一个网关」——错。一个中等规模公司,真实的对外资产通常是 IT 自己以为的 5-10 倍:测试环境忘了关、子公司继承过来的老系统、第三方 SaaS 集成进来的回调端口、CI 临时开的 debug port、被收购公司留下的 marketing 站点。

    「攻击者只会扫 80/443」——错。Shodan / Censys 把全 65535 个端口持续扫描,数据公开免费查。8080 端口的 Jenkins、9200 的 Elasticsearch、6379 的 Redis,三秒检索到。

    「没有公开链接的 URL 别人找不到」——错。子域名通过证书透明度日志(CT log)100% 可枚举——只要申请过 Let's Encrypt 证书,域名就在 crt.sh 永久公开,删都删不掉。

    1.3 攻击者侦察的两条目标线

    目标 攻击者想知道什么 用到的技术
    资产线 公司有哪些 IP / 域名 / 服务 子域名枚举、Shodan、CT 日志
    人线 谁有权限、邮箱格式、技术栈 LinkedIn、GitHub、招聘网站

    资产线找的是「机器入口」(漏洞利用),人线找的是「人员入口」(钓鱼)。真实事故里两条线常常合二为一——先在 LinkedIn 找到目标公司 SRE,看他 GitHub 提交过什么仓库,然后在 commit history 里翻出他不小心提交过的 .aws/credentials。

    1.4 防御方的真正解法:先摸自己

    防御侦察 ≠ 阻止扫描(挡不住),= 站在外面看自己的攻击面 + 持续比对清单。这就是 EASM(External Attack Surface Management) 的本质——把"自己公司"当作攻击者扫一遍,然后每天 diff:今天比昨天多了哪个域名 / 端口 / 证书?所有"未授权暴露"都应该在你日历最早提醒里,而不是在 attacker 的 Shodan 收藏夹里。

    二、OSINT:开源情报里能挖出多少东西

    2.1 OSINT 是什么

    OSINT(Open Source Intelligence) = 所有从「公开渠道」能拿到的关于目标的信息。关键字是「公开」——不入侵,不绕过权限,全部是合法可查的信息,但合起来威力惊人。

    OSINT 的精髓不是某个单一来源,是「拼图」——单看一个 LinkedIn 头衔啥也不是,加上 GitHub 邮箱、加上招聘 JD、加上 SEC 公告,就能拼出"这家公司有 8 个 SRE,主用 AWS + K8s,正在招 Rust 工程师所以可能在重写网关,CTO 周末刚推了一个 fork 的开源仓库……"——所有这些都不需要任何权限。

    2.2 公开信息里最值钱的几类

    公司层面:

    - 官网技术博客 / 招聘 JD     → 技术栈、版本号
- SEC / 工商公告              → 子公司、收购历史(攻击面延伸)
- DNS WHOIS                  → 域名注册人邮箱(钓鱼目标)
- 招标 / 政府采购             → 用了哪家 IT 厂商、什么型号设备
- Wayback Machine            → 老版本网站、删掉的页面、临时链接

    人员层面:

    - LinkedIn                   → 员工列表 + 技术栈
- GitHub                     → 代码、commit 邮箱、误推的密钥
- 推特 / 知乎 / 个人博客       → 在用什么工具、抱怨过什么 bug
- 简历投递平台                 → 离职员工(防御松懈、知道老密码)

    2.3 GitHub 泄露:OSINT 的重灾区

    工程师最常制造的 OSINT 灾难:git commit + push 之后才发现 config.json 里有 AWS_ACCESS_KEY_ID。哪怕 5 分钟后 force push 删除——没用。GitHub 的 commit 历史在 fork、在镜像站、在 GHArchive,永久可查。GitGuardian 2024 年报告显示,公开 GitHub 上每天新增的密钥泄露 23,000+——99% 是工程师"不小心 push 上去"。自查方式:trufflehog / gitleaks 扫全 org 历史;GitHub 搜 @company.com 看哪些 commit 邮箱泄露;org 设置里检查内部仓库是否被误公开。

    2.4 LinkedIn 与"邮件格式枚举"

    LinkedIn 是钓鱼的金矿——但真正危险的是邮件格式。LinkedIn 搜 "John Smith - SRE at ACME",再到 GitHub 公开 commit 看到 john.smith@acme.com,就能推断 ACME 的邮件格式是 {first}.{last}@acme.com——1000 人公司的邮件列表瞬间生成,定向钓鱼起点就有了。

    防御:这是无法"挡"的(LinkedIn / GitHub 都是合法公开),只能反向利用——内部红队定期向员工发模拟钓鱼测试点击率;高价值账户(财务、IT 管理员)默认 MFA + 硬件 key。

    三、子域名枚举:资产边界的发现

    3.1 为什么子域名是首要目标

    主站 acme.com 通常防护严密——CDN、WAF、定期渗透。但 dev.acme.com、old-billing.acme.com、internal-tools.acme.com 通常不是——这些是 attacker 的真正入口。一句业界格言:

    "Subdomain takeover is the new SQL injection." —— bug bounty 报告高频项

    子域名越多,你能控制的边界越长,忘记下线的就越多。

    3.2 枚举的三种路线

    路线 1:DNS 暴力枚举——拿一个常见前缀字典(dev、stage、test、admin、mail、api...)拼接到目标域名,逐个 DNS 查询,看哪个有 A 记录。

    # 字典里几千个常见子域前缀,逐个查
for sub in dev stage test admin api; do
    dig +short $sub.example.com
done

    便宜、可控,但漏的多——prod-blue-2024-internal.acme.com 这种字典里没有的猜不到。

    路线 2:证书透明度日志(CT log)——这是最可靠的一条路。从 2018 年起,所有 CA 签发的证书必须提交到公开的 CT 日志,任何人都能查:

    # crt.sh 是最常用的查询入口(只看,不要去搞别人的资产)
curl -s "https://crt.sh/?q=%25.example.com&output=json" | jq -r '.[].name_value' | sort -u

    只要你域名申请过 HTTPS 证书,就在公开日志里——这意味着即便你设了内网 DNS、没对外解析,只要证书签发了,域名就被记录。很多公司因此暴露了内部子域——典型例子:vpn-internal.company.com 申请了 Let's Encrypt 证书,域名永久公开。

    路线 3:第三方聚合 API——把 CT log、Passive DNS(被动 DNS 数据库)、第三方扫描数据等聚合起来:

    # subfinder / amass 是常用工具(只用于自己授权域)
subfinder -d example.com -silent
amass enum -passive -d example.com

    -passive:只查公开数据库,不向目标发包,防御方完全察觉不到。这是侦察阶段最常用的姿势——不打草惊蛇。

    3.3 子域名接管:被遗忘的资产怎么变成攻击入口

    Subdomain Takeover 的经典场景:你曾经把 promo.example.com 用 CNAME 指向 Heroku 的 my-app.herokuapp.com,活动结束后把 Heroku app 删了,但忘记了在 DNS 里删 CNAME。攻击者重新注册一个叫 my-app 的 Heroku app,promo.example.com 就解析到他控制的页面——他可以用你的子域发钓鱼、偷你的 Cookie(同根域)。根因:DNS 记录和实际服务生命周期不一致。修复:每次下线服务时,DNS 清理是 checklist 强制项;定期用 dnstwist / subjack 扫自己的 dangling 记录。

    3.4 防御侧:把自己的子域当作资产

    持续运行(每天):
1. subfinder -d company.com 拿全量子域清单
2. 与昨天清单 diff
3. 新增子域 → 自动告警(谁起的?有没有备案?)
4. 消失子域 → 检查 DNS 是否清理干净

    这就是 EASM 的最小可用版本。商业产品(Censys ASM、Microsoft Defender EASM)做得更全,但「每天 diff 一次子域清单」自己写脚本一小时就能搞定,优先级远高于买产品。

    四、端口扫描:nmap 的三种扫描机制

    4.1 主动侦察的边界

    子域名找完,下一步是「哪些 IP 开了什么端口」。这一步从被动转主动——发包到目标,目标的防火墙/IDS 能看见。所以仅在你拥有授权的资产上做(自己的 IP、租用的 VPS、CTF 靶机)。

    nmap 是事实标准,30 年来没变过。讲清三种最核心的扫描类型就够:

    4.2 TCP Connect 扫描(-sT)

    # 在自己服务器上扫自己
nmap -sT 127.0.0.1

    完整的 TCP 三次握手:SYN → SYN-ACK → ACK,然后立即断开。优点:不要 root,任何用户都能跑(普通 connect 系统调用);缺点:每个端口都进系统的连接表,目标的应用层日志会看到一次"短暂连接"——隐蔽性差。

    4.3 SYN 半开扫描(-sS,默认)

    nmap -sS 192.168.1.0/24    # 必须 root

    只发 SYN,收到 SYN-ACK 后不发 ACK,直接 RST 关闭——握手没完成,应用层根本没意识到有人来过。被称为 "stealth scan",这是 nmap 默认模式。

                      Client             Server
                    │                  │
                    │──── SYN ────────>│   (打这个端口)
                    │<── SYN/ACK ──────│   (开着)
                    │──── RST ────────>│   (不握手,直接关)
                    │                  │
端口 open:server 回 SYN/ACK
端口 closed:server 回 RST
端口 filtered:server 没回(被防火墙挡了)

    防御视角:SYN 半连接在内核 TCP 栈是可见的,只是应用层看不见——netstat、conntrack 表里能查到,Linux 的 SYN backlog 满了还会告警。所以 SYN 扫描不是"隐身",是"应用层隐身"。

    4.4 UDP 扫描(-sU)

    UDP 是无连接的,扫起来异常痛苦:

    - UDP 端口"开"的标志是 "应用回了 UDP 响应"
- 但很多应用收到空包根本不回(比如 DNS 收到 "AAAA AAAA" 不解析)
- 端口"关闭"的标志是 ICMP "Port Unreachable"
- 但 ICMP 响应被 Linux 默认限速到 1/秒
- 所以扫 65535 个 UDP 端口要 18+ 小时

    实战里很少全段 UDP 扫描——一般只扫 50 个常见 UDP 端口(53 DNS、123 NTP、161 SNMP、500 IKE、1900 SSDP、3389 RDP-UDP、5060 SIP、11211 Memcached……),够了。

    nmap -sU --top-ports 50 192.168.1.10

    防御视角:UDP 暴露的高危服务比 TCP 还危险——反射放大攻击全靠 UDP 服务(memcached 放大 51000 倍、NTP monlist 放大 556 倍)。最起码的规则:云上安全组里 UDP 默认全关,只按需打开。

    4.5 nmap 常用参数(扫自己用)

    nmap -sS -sV -O -p- --top-ports 1000 -T4 自己的IP

# -sS    SYN 半开扫描
# -sV    探测服务版本(banner grab)
# -O     猜操作系统(TCP/IP fingerprint)
# -p-    扫全 65535 端口
# --top-ports 1000  只扫最常见 1000 端口(快 60 倍)
# -T4    时序模板(0=极慢,5=极快,4 是常用平衡)

    注意:扫越快越容易触发对方限速 / 封 IP——攻击者实战会用 -T1(慢扫,几小时一个 IP),避开 IDS;防御方写规则也要意识到"慢扫不会触发阈值告警"。

    4.6 工程师视角:扫描结果怎么读

    nmap 输出三类结果:open / closed / filtered:

    状态 含义 攻击者怎么看
    open 应用回应了 进一步查指纹、找 CVE
    closed 端口没监听但 IP 在线 跳过
    filtered 防火墙挡了,看不见 标记"可能开但拦了",换路线

    filtered 多 ≠ 安全——只意味着"扫描者没法直接看到"。真实攻击者会换 IP、绕代理、走 IPv6、找 VPN 后面的入口。绝对不要把"防火墙挡住扫描"等同于"端口安全"。

    五、服务指纹:Banner Grab、JA3/JA4、HTTP 头特征

    5.1 端口开了之后:它是什么

    nmap -sV 做的事是「banner grab」——TCP 握手成功后,有些服务会主动发欢迎信息:

    $ nc 自己服务器 22
SSH-2.0-OpenSSH_8.9p1 Ubuntu-3ubuntu0.4

$ nc 自己服务器 25
220 mail.example.com ESMTP Postfix (Ubuntu)

$ nc 自己服务器 21
220 (vsFTPd 3.0.5)

    Banner 暴露了三个关键信息:服务类型、版本号、运行平台。版本号是关键——OpenSSH_8.9p1 可以直接到 CVE 数据库查:这个版本有没有未修复漏洞?

    5.2 隐藏 banner 不是真正的防御

    工程师常见的"修法":配置 nginx.conf 加 server_tokens off;,改 SSH banner 假装是别的版本。这只挡得住自动扫描脚本里的版本匹配,挡不住真正的攻击者——他们会发特定的协议帧,根据"对方怎么响应非标输入"反推真实版本(被动指纹识别 / passive fingerprinting)。

    真正的防御不是"藏版本",是"及时打补丁"。server_tokens off 该开还是开,但别误以为这就安全了。

    5.3 TLS 指纹:JA3 与 JA4

    JA3 由 Salesforce 在 2017 年开源——把 TLS ClientHello 的字段(version、cipher suites、extensions、椭圆曲线、压缩方法)按固定顺序拼起来做 MD5,得到 32 字节"客户端指纹"。为什么这个有用:Chrome 的 JA3 是一个固定值、Firefox 是另一个、Python requests 是第三个——一个伪造 User-Agent 的爬虫,JA3 会暴露它。

    JA4(2023)是 JA4+ 套件的一部分,比 JA3 更稳健——不再做 MD5(可碰撞),改用按字段切片拼接的明文格式,可读、可比对、对 QUIC 也适用。现代 WAF / CDN 大量用 JA3/JA4 识别"非正常浏览器"(Cloudflare、Akamai 都集成了)。攻防双向:攻击者用 curl-impersonate 把 curl 的 JA3 改成 Chrome 的样子绕过检测;防御者把 JA3/JA4 + UA + 行为综合打分,而不是单点判断。

    5.4 HTTP 头特征

    HTTP 服务的指纹除了 Server 头,还有一堆"间接指纹":

    Server: nginx/1.24.0          ← 显式
X-Powered-By: PHP/8.2.1       ← 显式
X-AspNet-Version: 4.0.30319   ← 显式

Set-Cookie: PHPSESSID=...     ← 隐式(PHP)
Set-Cookie: JSESSIONID=...    ← 隐式(Java)
Set-Cookie: .ASPXAUTH=...     ← 隐式(.NET)

ETag 格式 / Date 头顺序 / 错误页 HTML 结构 ← 间接

    清理显式头容易,清理间接指纹几乎不可能——返回的 404 页面 HTML 长什么样,Date 头放在 Server 头前面还是后面,这些都被 fingerprint 工具拿来识别。

    六、Web 指纹:WhatWeb / Wappalyzer / Favicon Hash

    6.1 应用层指纹的目标

    到这层不再问"是什么 Web 服务器",而是问"跑的什么应用":WordPress?Drupal?Jenkins?GitLab?如果是 WordPress,装了哪些插件?这些信息决定了 attacker 下一步去 CVE 库里搜什么。

    6.2 WhatWeb / Wappalyzer

    whatweb 是 Ruby 写的命令行工具,内置 1800+ 个签名:

    whatweb https://自己网站.com

# 输出示例
https://example.com [200 OK] Country[US],
HTTPServer[nginx], JQuery[3.6.0],
Open-Graph-Protocol, WordPress[6.4.2],
X-Frame-Options[SAMEORIGIN]

    Wappalyzer 是浏览器插件(也有 CLI),覆盖更广——不光识别 Web 框架,还识别 CDN、分析工具、广告 SDK、CMS 等。

    指纹的判定方式:

    - HTML 里的 <meta name="generator" content="WordPress 6.4.2">
- HTTP 头 X-Generator
- 特定 URL 路径(/wp-admin/、/administrator/)
- 特定 JS 文件名(jquery.js、react.production.min.js)
- 特定 CSS 类名(elementor-、wp-block-)

    修复思路:不要试图全删——删了功能就坏了。关键是"被识别 = 必须立即知道有 CVE 要打补丁"。

    6.3 Favicon Hash:被低估的指纹

    # 算法极简
import hashlib, requests
favicon = requests.get("https://example.com/favicon.ico").content
print(hashlib.md5(favicon).hexdigest())   # 或 mmh3

    为什么这玩意儿重要——很多内部管理后台用默认 favicon:

    - Jenkins 默认 favicon → MD5 是某个固定值
- GitLab 默认 favicon → 另一个固定值
- Grafana 默认 favicon → 第三个固定值

    Shodan 支持 http.favicon.hash 搜索:一行查询就能列出全球所有"暴露在公网 + 用默认 favicon 的 Jenkins 实例"。这就是为什么"换个 favicon"成了被忽视的安全建议——能让你脱离全网扫描器的"批量目标列表"。

    6.4 工程师的 favicon 清单

    ✓ 把默认 favicon 替换成公司自己的
✓ 内部后台(Jenkins / Grafana / Kibana)的 favicon 也换掉
✓ 错误页(404 / 500)别用框架默认页(自己写一个)
✓ HTTP 头别带框架自己加的 X-Powered-By

    这一组细节单看每条都是小事,合起来能让你的资产"脱离全网批量目标"——攻击者扫到的 100 万个 Jenkins 里,你这一个不在那个列表上。

    七、Shodan / Censys / FOFA:被动 OSINT 的工业化

    7.1 它们是什么

    Shodan(2009 创立)和 Censys(2015,密歇根大学出身)是「互联网扫描数据库」——它们自己 7x24 小时扫描全 IPv4 地址(所有端口、所有协议),结果存下来供任意查询。攻击者视角:不用自己扫,几秒查到结果;防御者视角:你的资产暴露多少,他们已经替你扫好了。FOFA(白帽汇,中国)是国内同类,中国资产覆盖更细。

    7.2 能查到什么(对你自己的资产)

    shodan search 'org:"YOUR COMPANY"'              # 你公司所有暴露资产
shodan search 'ssl:"yourdomain.com"'            # 用了你证书的 IP
shodan search 'http.favicon.hash:XXXX'          # 用某 favicon 的所有 IP
shodan search 'product:Elasticsearch port:9200' # 公网 ES(无认证)
shodan search 'product:Redis port:6379'         # 公网 Redis(2018 勒索根因)

    每一条都是"几秒钟拿到全球清单"。这是为什么"安全组默认拒绝、按需开放"是黄金规则——只要你 6379 暴露过 10 分钟,Shodan 的爬虫就大概率扫到过,截图永远存档。

    7.3 防御侧:Shodan Monitor

    Shodan / Censys 都有 Monitor / ASM 模式——你订阅自己的 IP 段或域名,他们扫到任何新暴露(新端口、新证书、新软件版本)立刻给你告警。免费层够小公司用,付费层给中大公司做 API 自动化。国内还有 ZoomEye / Quake / BinaryEdge。主流选择:Shodan + Censys 主用,FOFA / ZoomEye 国内补充——一家覆盖不全,交叉查更稳。这是攻击者免费用的工具,防御方更应该用,而且要先于攻击者用。

    八、防御视角:把侦察阶段堵住的工程实践

    8.1 EASM(External Attack Surface Management)三件事

    1. 发现(Discovery): 你不知道的资产被发现
2. 清单(Inventory):每个资产有 owner、用途、合规状态
3. 监控(Monitoring):资产变化每天 diff,异常告警

    最常见的失败:有发现、无清单——扫出来 500 个子域,没人管"哪个是哪个团队的、哪个该删"。资产清单关键字段:域名/IP、owner、上线日期、业务用途、暴露端口、证书有效期、上次扫描时间。清单的本质:让每个资产有人负责、有下线时间、有 review 节奏。一旦没主了,下线。

    8.2 资产清单从几个脚本起步

    subfinder -d acme.com > today_subs.txt           # 子域枚举
nmap -sS -p- own_ip_range > today_ports.txt      # 端口扫描(自己 IP)
shodan search 'org:"ACME"' > today_shodan.txt    # Shodan 自查
diff yesterday_subs.txt today_subs.txt           # 每天 diff

    每天跑一次,新增即告警。中等规模公司一周就能有清单雏形,远好于"等买 EASM 产品的预算"。

    8.3 Authorized Recon 常态化

    "红队扫自己"应该是每个团队的常规操作——不是一年一次的渗透测试,是每周或每天的自检:

    开发环境是否对外?          (默认应该全关)
新上线的接口有没有走 SSO?  (匿名访问应该 403)
新签发的证书属于谁?        (Slack 通知到 owner)
有没有 dangling DNS?       (subjack 扫)
GitHub 公开仓库有没有泄露? (trufflehog 跑全公司 org)

    做这些不需要昂贵工具,需要的是"把这些步骤纳入 CI/CD 与定期任务"——这是 EASM 的"工程化",比买产品本身更重要。

    九、真实事故:Capital One 是怎么从"侦察"被入手的

    2019 年的 Capital One 数据泄露(1 亿美国信用卡用户)是侦察阶段被打穿的教科书案例:

    1. 攻击者通过 Shodan / 子域扫描,找到一个错配的 WAF(ModSecurity)
   该 WAF 实例挂着一个 IAM role,可调 AWS 服务
2. WAF 配置错误允许接收任意来源的 SSRF 请求
3. 攻击者从外部发请求 → WAF 转发 → AWS 元数据 169.254.169.254
4. 拿到 WAF 实例的 IAM credentials
5. 用这个 credential 列 S3 bucket,下载用户数据

    侦察阶段的关键点:攻击者没有任何"高级 0day",只是发现了一个错配的 WAF 实例;这个 WAF 的暴露本来在资产清单里也许根本不显眼,但因为 IAM role 权限过大,单点错配把 1 亿用户拖下水。事后 Capital One 加了两件事:资产清单 + 持续扫描(每个对外资产 IAM role 全 review);IMDSv2 强制(SSRF 拿不到 metadata)。真正的攻击不靠 0day,靠"你以为没人会注意到的那台机器"——它在攻击者眼里是高亮的,因为它存在于 Shodan 但不存在于你的清单。

    十、上线前外部侦察自检表

    每次有新服务对外暴露之前,把这张表过一遍(不止安全团队,业务工程师上线前都该过):

    [ ] 资产登记
    [ ] 该资产已加入 EASM 清单(域名 / IP / 端口 / owner)
    [ ] 有明确的"下线触发条件"(项目结束 / 季度 review)
    [ ] DNS 记录与实际服务生命周期绑定

[ ] 子域 / 证书
    [ ] 子域名命名不暴露技术栈(避免 jenkins.acme.com,改名 build-1.acme.com)
    [ ] 证书申请走内部 ACME 流程,自动续期
    [ ] 内部域名不申请公开 CA 证书(用私有 CA / 自签)
    [ ] dangling DNS 自查(subjack / dnstwist 周扫)

[ ] 端口
    [ ] 安全组默认拒绝,只放业务必需端口
    [ ] UDP 默认全关(除非业务确实需要)
    [ ] 管理端口(22 / 3389)不直接对公网(走 VPN / 跳板)
    [ ] 内部服务(Redis / ES / MQ)严禁对公网

[ ] 指纹
    [ ] HTTP 服务:server_tokens off / X-Powered-By 移除
    [ ] 默认 favicon 已替换
    [ ] 错误页(404 / 500)不暴露框架特征
    [ ] 默认登录页 URL 已改名或加 IP allowlist

[ ] OSINT 自检
    [ ] GitHub 公开仓库 trufflehog/gitleaks 全扫
    [ ] 公司域名邮箱在 haveibeenpwned 检查泄露
    [ ] LinkedIn / 招聘 JD 不要写"我们用的 XX 版本"
    [ ] Slack / 知乎 / 公开博客技术分享内部敏感细节脱敏

[ ] 监控与告警
    [ ] Shodan / Censys Monitor 订阅自己的 IP 段 + 域名
    [ ] 每日 subfinder 扫子域,新增即告警
    [ ] CT 日志监控(crt.sh / Cert Spotter):新签证书 → 通知
    [ ] WAF / CDN 启用 JA3/JA4 异常检测

[ ] 制度
    [ ] 每月一次"自己扫自己"红蓝演练(authorized recon)
    [ ] 资产清单 owner 字段每季度 review 一次
    [ ] 离职员工的 GitHub / 仓库权限即时清理

    把这张表打印贴墙上,不是装饰——任何一项漏掉,都可能是下一个 Capital One。

    十一、踩坑提醒

    1. 以为"挡住扫描"就安全——SYN 扫不留应用层日志,Shodan 数据已经永久公开。挡不住,只能先于攻击者摸自己
    2. 以为子域少——CT 日志一查通常多 5-10 倍。不查就以为"我家就那几个域名"
    3. 以为没人会扫 IPv6——Shodan 在扫,Censys 在扫,IPv6 不是安全替代,只是命中率低
    4. 以为 GitHub 删了 commit 就没事——history 永远存在,trufflehog 一扫一个准
    5. 以为 favicon 不重要——http.favicon.hash 搜索把你纳入批量目标
    6. 以为 banner 改了就安全——版本号能藏,但被动指纹藏不住,真正解法是打补丁
    7. 以为 EASM 是产品——本质是流程。脚本能跑起来,比买不会用的产品有价值十倍
    8. 以为侦察阶段没办法防——是。侦察是"非对称"的,防御方只能"先自摸"
    9. 以为业务工程师不用懂这一层——错,资产清单的 owner 就是业务工程师,你不参与,资产就会失控
    10. 以为"内网"是绝对安全的——VPN 接入、合作伙伴对接、收购的子公司,内外边界一直在被打穿
