    红蓝紫队:攻防演练 / 检测覆盖 / Purple Team 思维

    读到这一篇,前面 21-23 已经把"攻击者拿到入口之后干什么"这条线讲完——侦察、Web 入口、内网横移。但所有那些都是一个人在打。真实世界里,公司不会雇个渗透测试员打三天就完事——真正在跟攻击者赛跑的,是一支建制化的"红队 + 蓝队"团队,以及它们之间的协作模式「紫队(Purple Team)」。新手最容易误解的两件事:第一,紫队不是中间色,不是"颜色调和",是红蓝坐在一起干活的工作方式;第二,红队不是渗透测试,虽然技术栈重叠,但目标、时间尺度、对手都完全不同。这一篇讲清楚这三个角色怎么分工、怎么协同、产物是什么,以及一家没有钱养专职红蓝队的小公司怎么从零搭建一个能跑起来的"轻量紫队"。

    一句话先记住:红队问"我能不能进得来"、蓝队问"我能不能看见你进来"、紫队问"我能不能下次就看见你进来"。三个问题对应三个产物——红队产攻击路径,蓝队产检测规则,紫队产"攻击路径 ↔ 检测规则"的映射表。这张映射表挂在 MITRE ATT&CK 矩阵上,就是现代安全运营的核心 KPI:检测覆盖率(detection coverage)。一家公司安全水平怎么样,不看它买了多少安全产品,看它能在 ATT&CK 矩阵的多少格子里说"这条我能告警"。

    一、为什么需要"两支队伍"

    1.1 一个不对称的赛局

    防御方天然处于劣势,原因是「不对称」:

    攻击者:只要找到一个洞 → 进来
防御者:必须堵上所有洞 → 守住

攻击者:时间无限,可以盯一年
防御者:每周都有 deadline,人手有限

攻击者:用一套工具打一百家
防御者:每家都要自己研究怎么防

攻击者:失败没成本,换一台机器再来
防御者:失败一次就上新闻

    这个不对称解释了为什么"事后被打"是常态——不是工程师菜,是结构性的。能补救的不是"再加 50% 防御",而是"提前模拟攻击,验证自己的防御能不能看见"。这就是红蓝队存在的根本理由——用一群"自己人"扮演攻击者,逼着防御方在被真攻击者打之前就先疼一次。

    1.2 三种"我打我自己"的活动,别搞混

    活动 目的 时间尺度 防御方知不知道
    漏洞扫描 找已知 CVE 分钟级 知道
    渗透测试(pentest) 有授权打一个范围 1-2 周 知道,且配合
    红队演练(red team) 模拟真实 APT 全链路 1-6 个月 不知道(或只有少数人知)
    桌面演练(tabletop) 纸面上对剧本 几小时 知道,围坐讨论
    紫队(purple team) 红蓝同步干活 几天到几周 知道,实时联调

    关键差别:扫描和 pentest 的产物是「漏洞清单」,红队的产物是「完整的攻击链路 + 检测盲区」,紫队的产物是「每条链路对应的检测规则」。给老板看的不是"我们扫到 200 个漏洞",而是"我们 ATT&CK 覆盖率从 30% 提到 60%"。

    二、红队:模拟真实攻击者

    2.1 红队的定义

    红队不是"高级渗透测试员"。它是一个被授权扮演真实威胁演员(threat actor)的内部团队,目标不是"找洞",而是"走完一次完整的攻击链路,看蓝队能不能发现"。

    渗透测试目标:
   "请测一下这个 Web 应用的安全性"
   → 拿到入口就算赢

红队目标:
   "假装你是 APT29,目标是偷到 CRM 数据库的客户信息"
   → 钓鱼 → 上线 → 提权 → 横移 → 访问 CRM → 落地
   → 全程不被发现 = 真赢

    渗透测试是「点」,红队是「链」。链上任何一环被蓝队抓住,红队就失败了——所以红队天然要把"怎么不被发现"放在和"怎么打进去"同等重要的位置。

    2.2 TTP:战术 / 技术 / 流程

    红队的"剧本"用 TTP(Tactics / Techniques / Procedures) 描述,这是 MITRE ATT&CK 的核心词汇:

    • Tactic(战术):攻击者要达到什么目标——初始访问、持久化、横向移动、数据窃取。这是 ATT&CK 矩阵的列。
    • Technique(技术):用什么手段达到——钓鱼、漏洞利用、Pass-the-Hash。这是 ATT&CK 矩阵的格子。
    • Procedure(流程):具体怎么做——"用 Cobalt Strike 的 PSExec 模块,跳板机是 X,凭据来自 mimikatz"。
    Tactic:        Initial Access
   ↓
Technique:     Phishing (T1566)
   ↓
Sub-technique: Spearphishing Attachment (T1566.001)
   ↓
Procedure:     APT29 用 .docm 文件夹带 base64 编码的 PowerShell loader

    为什么这套抽象重要:它让"防御 vs 攻击"有了共同语言。蓝队不用关心"这次具体是哪个 APT 组织",它只需要知道"我能不能检测到 T1566.001 这个技术"——一条检测规则对应一个 Technique ID,一千个 APT 都打不穿同一个检测。

    2.3 红队和真 APT 的差距

    红队再像也是"扮演",有几个真实攻击者没有的约束:

    • 道德边界:不破坏生产、不真正泄露数据、不打员工家庭
    • 时间预算:再长也就 3-6 个月,APT 可以盯三年
    • 资源:不会真的烧 0day——一个 0day 黑市百万美金,内部红队不舍得用
    • 退出条款:打到某个里程碑就停,真攻击者会继续扎根

    所以红队的真实价值不是"完美模拟",而是"覆盖 90% 的常见 TTP——APT29、Lazarus、勒索家族这些公开的剧本演一遍,绝大多数蓝队的盲区都会暴露。

    三、蓝队:检测 / 响应 / 取证 / 加固

    3.1 蓝队的四大职能

    蓝队不是"安全运维"。它做四件事,任何一件做不好,前面三件都白费:

    1. 检测(Detection)
   写日志 / 写告警规则 / 看 SIEM
   → "有人在 PowerShell 里调 mimikatz"

2. 响应(Response)
   接到告警 → 判定 → 处置 → 隔离
   → SLA:5 分钟看到、15 分钟决定、1 小时隔离

3. 取证(Forensics)
   事后还原:谁进来的、什么时候、动了什么、走了什么
   → 内存 dump、磁盘镜像、日志关联

4. 加固(Hardening)
   把这次的教训写进基线、固化到 IaC
   → 下次同类攻击直接打不出第一步

    新手最大的误区是只看「检测」——以为蓝队就是写 Sigma 规则。实际上检测只是把球抢到,响应不快、取证不严、加固不落,等于白抢。一个真实数字:Mandiant 报告 2024 年企业平均驻留时间(dwell time)是 10 天——攻击者进来到被发现,平均要 10 天。前 9 天蓝队都不知情,这才是常态。

    3.2 检测金字塔(Detection Pyramid)

    蓝队的检测能力分层,越往上越值钱、越难替换:

                      ┌──────────┐
                  │   TTPs   │   ← 最值钱(描述"行为模式")
                  └──────────┘
                ┌──────────────┐
                │   Tools      │   ← 检测攻击工具的特征
                └──────────────┘
              ┌──────────────────┐
              │ Network Artifacts │  ← URL / UA / 流量特征
              └──────────────────┘
            ┌──────────────────────┐
            │   Host Artifacts     │  ← 文件名 / 注册表 key
            └──────────────────────┘
          ┌──────────────────────────┐
          │      Domain Names        │  ← 攻击者域名
          └──────────────────────────┘
        ┌──────────────────────────────┐
        │       IP Addresses           │  ← 黑 IP
        └──────────────────────────────┘
      ┌──────────────────────────────────┐
      │       Hash Values                │  ← 文件 hash(最廉价)
      └──────────────────────────────────┘

    这叫 Pyramid of Pain(David Bianco 提出)——越往上,攻击者绕过的代价越高。

    • 你封了一个 hash,攻击者改一个字节就绕过——他疼 1 秒
    • 你封了一个 IP,他换一台 VPS——疼 5 分钟
    • 你检测到了某个 RAT 工具的 C2 协议特征——他要重写工具,疼几周
    • 你检测到了"任意 PowerShell 从内存里调 mimikatz 然后访问域控"这种行为模式——他要换整套打法,疼几个月

    写检测规则的核心心法:往金字塔顶端写。基于 hash / IP 的规则是廉价票,基于 TTP 的规则才是护城河。这也是为什么 ATT&CK 矩阵成了"行业通用语言"——它就是 TTP 的目录。

    3.3 Detection Engineering:不是写正则,是写代码

    现代蓝队把"写检测规则"工程化,叫 Detection Engineering,完整流程:

    1. 选定一个 ATT&CK Technique(e.g. T1003.001 LSASS 内存读取)
2. 调研:攻击者会怎么具体执行?有什么变种?
3. 写规则(Sigma / Splunk SPL / Elastic EQL)
4. 红队实际跑一遍 → 触发了没?
5. 测假阳性:正常业务会不会触发?
6. 编写 Runbook:告警来了运营怎么处置?
7. 进版本控制(detection-as-code)
8. CI 测试:每次规则改动跑回归
9. 上线 → 监控指标 → 回到 1

    检测规则现在和后端代码一样进 Git、走 PR、跑 CI——这就是"detection-as-code"。写一条好规则的工作量,等同于写一个微服务。Google、Microsoft 都有专职的 detection engineer 岗位,薪资和 SRE 一个档次。

    四、紫队:不是颜色,是工作方式

    4.1 紫队的"反直觉"定义

    新手一听紫队,第一反应:「啊,介于红和蓝之间的中等水平」。完全错。紫队不是一支队伍,是红蓝两支队伍坐在同一个屋里实时联调的工作方式。

    传统模式(红蓝分离):
   红队 ──── 打两周 ────→ 报告
                                ↓
   蓝队 ←──── 拿到报告 ─── 看一周 → 改规则
                                ↓ 几周后
   红队 ──── 再打一次 ────→ 再报告
   ↑ 反馈周期 = 月级

紫队模式(同步工作):
   红队 「我现在跑 mimikatz」
   蓝队 「我没看到告警」
   红队 「我换 base64 编码再试」
   蓝队 「现在 SIEM 触发了一条,但 5 分钟才到」
   红队 → 蓝队 → 红队
   ↑ 反馈周期 = 分钟级

    核心价值:把"打 → 报告 → 改"的几周缩到几小时。一周紫队演练能产出的检测规则,等于半年传统模式。

    4.2 紫队的几种节奏

    • Continuous Purple(持续紫队):每周固定一天,红蓝同坐,挑一个 Technique 联调。Google、Microsoft 在用。
    • Sprint Purple(冲刺紫队):每季度集中一周,围绕一个主题(如"勒索软件链路")全员联调。多数中型公司在用。
    • Adversary Emulation(对抗模拟):选定一个真实 APT 剧本(如 APT29 的 SolarWinds 攻击链),按 ATT&CK 报告里的每个 step 实际复现 → 检查每步的检测覆盖。

    紫队是文化,不是组织架构。一家公司可能没有"紫队部门",但红蓝两个团队每周开一次同步会、用同一个 ATT&CK 仪表盘——它就是紫队成熟体。

   绿色 = 有检测,红队跑过会触发
   黄色 = 有检测,但有漏报或延迟
   红色 = 完全没检测,是盲区

    这张「热力图」就是现代安全团队最重要的 KPI——给 CTO / CISO 汇报不讲"我们发了多少告警",讲"我们 ATT&CK 覆盖率从 35% 提到 62%"。MITRE 自己开源了一个工具叫 ATT&CK Navigator,就是画这张图用的。

    六、攻防演练的完整流程

    不管叫红队、紫队还是 tabletop,一次合格的演练分五步:

    6.1 第一步:目标设定(Objective)

    最容易跳过、但决定演练价值的一步。不是"打进来就行",必须有具体目标:

    ❌ 差的目标:测试一下我们的安全。
✅ 好的目标:
   假设攻击者通过钓鱼拿到一个普通员工账号,
   72 小时内能不能拿到生产 DB 的客户表?
   - 范围:研发网 + 生产 K8s
   - 边界:不动 HR 系统、不打员工家庭网络
   - 成功标准:能查询到某条客户记录的 SQL 截图
   - 蓝队是否知情:不知情(只 CISO 知道)

    6.2 第二步:攻击执行(Execution)

    红队按 TTP 剧本展开,每一步都打两个日志——一份给自己(取证)、一份给紫队的协调员。

    [D+0 10:00] T1566.001 钓鱼,3 人中招
[D+0 14:30] T1059.001 PowerShell loader 落地
[D+1 09:00] T1003.001 mimikatz 读 LSASS
[D+1 15:00] T1021.002 SMB 横移到第二台
[D+2 11:00] T1078 域内合法账户接管
[D+3 16:00] T1041 拿到目标 DB 表

    6.3 第三步:检测评估(Detection Assessment)

    演练结束后,对每个攻击步骤问蓝队三个问题:

    1. 触发了告警吗?
2. 告警有没有被人看到?(SIEM 一天一万条告警,看不过来就等于没有)
3. 看到了有没有正确判断?(误判成"运维操作"就等于没看到)

    三个问题对应三种失败:没检测、没看见、没看懂。每一种的修复方式完全不同——

    • 没检测 → 写新规则
    • 没看见 → 调告警优先级、减噪音
    • 没看懂 → 写 Runbook、培训分析师

    6.4 第四步:复盘(After-Action Review)

    复盘要避免两个反模式:"批斗大会"(红队炫技、蓝队挨打)和"皆大欢喜"(各说各话、不出产物)。

    正确的复盘开法:对着 ATT&CK 矩阵逐格走。每格三个问题——红队是怎么打的、蓝队看到了什么、下一步加什么规则。复盘的产物是工单,不是 PPT。

    6.5 第五步:落地(Remediation)

    复盘产生的工单分三类,优先级也不同:

    类型 例子 优先级 谁负责
    检测缺口 写新 Sigma 规则覆盖 T1547.001 P0(7 天) 蓝队
    漏洞修复 修 LDAP 注入 P1(30 天) 业务团队
    架构加固 把 PowerShell 默认 Constrained Mode P2(季度) 平台团队

    演练不出工单 = 演练白做。一次好的紫队演练,产出 20-50 张工单是正常水平,90 天后回头看完成率,这是真正的 KPI。

    七、桌面演练 / Live Exercise / CTF / Hack Week

    不是所有演练都要真打,公司里实际跑的是一个"演练光谱":

    成本低 ←──────────────────────────────────────────→ 成本高
真实低 ←──────────────────────────────────────────→ 真实高

[Tabletop]   [CTF/演习场]   [Purple Drill]   [Red Team]   [Adversary Emu]
 纸面对剧本   靶机环境演练    红蓝同步联调      盲打全栈      模拟特定 APT
 几小时        1-3 天          1 周              1-6 月       3-12 月

    7.1 桌面演练(Tabletop)

    最便宜、最被低估。没有任何技术执行,所有人围桌而坐,主持人念剧本——

    主持人:"周五下午 3 点,SOC 收到一条告警:某个员工账号在凌晨 2 点
        从俄罗斯 IP 登录,然后下载了 CRM 全表。你下一步做什么?"

值班分析师:"我先看 VPN 日志确认。"
主持人:"VPN 日志显示该账号确实 2 点登录,然后被注销了。"
分析师:"我冻结账号,通知本人。"
主持人:"通知不到本人。"
法务负责人:"现在涉及客户数据,要不要触发 GDPR 72 小时报告?"
...

    桌面演练的核心价值不是技术,是流程——谁该打电话给谁、决策权在谁、72 小时报告由谁起草。真出事时这些事情没演练过,会一团乱。NIST 推荐每个组织至少一年一次 tabletop,这是性价比最高的演练形式。

    7.2 Live Exercise / CTF

    公司内部办 CTF(Capture the Flag),给蓝队练手用的——搭一个靶机环境,藏几个 flag,让员工去打。Google 的 "Caesar's Wheel"、Meta 的内部 CTF 平台都是这种。

    价值:让蓝队建立"攻击者直觉"——你天天写检测规则,自己没真正绕过过 EDR,写的规则就是空谈。

    7.3 Hack Week / 红队周

    公司级活动,一周时间全公司停手头工作,集中找洞——Meta 的 "Hacktober"、Microsoft 的 "Blue Hat"、Google 的 "Project Zero" 都是这种文化的延伸。

    核心好处不是找洞数量,是文化建设——让产品工程师亲身感受"原来我写的代码是这样被打的",比安全培训有效一万倍。

    八、真实公司的红蓝建设

    8.1 Google

    红队对内打、对外做研究(Project Zero),蓝队叫 TAG。关键实践:Detection-as-code(规则进 Git、CI 测试)、BeyondCorp 零信任(没有"内网",每个请求独立鉴权,横向移动空间被结构性缩窄)、持续红队(全年不间断,不是每年一次)。

    8.2 Meta

    红队叫 Red Team X。Hacktober:每年 10 月全公司参加 CTF,产品工程师必修。Bug Bounty 与红队联动:外部赏金报告被内部红队拿来"复现 + 扩展"。内部自研 Adversary Simulation Platform 自动化跑 ATT&CK。

    8.3 阿里 / 国内大厂

    国内特有的是 HW(护网行动)——政策驱动的全国大规模红蓝对抗,每年 7-8 月集中两周,所有重点保障单位被外部红队打。好处是"国家级强制",坏处是容易变应试——平时不练、8 月突击。成熟厂(阿里"蓝军"、字节红队、腾讯"湛卢")在 HW 之外都有常态化红蓝队。

    国内外共同点:一线大厂都把红蓝建设当常态化、工程化的事,不是"出事找乙方"。

    九、小公司从零搭建紫队的最小路径

    不是每家公司都能养 50 人的安全部。一家 50-200 人规模的公司,只有 1-3 个安全工程师,怎么搭一个能跑起来的"紫队"?以下是实战可落地的最小路径:

    9.1 阶段 0:先有日志(Week 0,前置条件)

    没有日志就不要谈红蓝队。先把这五类日志接进 SIEM 或者哪怕一个 ES:

    1. 终端:Sysmon(Windows)/ auditd(Linux)/ macOS Unified Log
2. 网络:DNS 查询、HTTP/HTTPS 元数据(Zeek 或 ALB 日志)
3. 身份:SSO 登录日志(Okta / 飞书 / 钉钉)
4. 云:CloudTrail / 阿里云 ActionTrail / GCP Audit Log
5. 关键应用:数据库审计、生产 SSH 跳板机

    没日志写规则就是空中楼阁。先花两周把日志管道铺好,这一步占整个紫队建设 50% 的工作量。

    9.2 阶段 1:挑十个 Technique(Week 1-2)

    不要妄想全覆盖 ATT&CK 的 200 个 Technique。挑十个,从勒索软件最常用的开始:

    T1566   钓鱼
T1059   命令脚本解释器
T1003   凭据 dumping
T1078   合法账户滥用
T1021   远程服务(RDP/SMB)
T1486   数据加密勒索
T1567   通过 Web 服务外传
T1190   公开应用利用
T1110   暴力破解
T1098   账户操作

    这十个覆盖了 80% 的真实事件。能把这十个全部"绿色化",就比 90% 的公司强了。

    9.3 阶段 2:写检测规则(Week 3-6)

    每个 Technique 写至少一条 Sigma 规则,直接抄开源仓库:

    • SigmaHQ/sigma - 几千条社区规则
    • Elastic detection-rules
    • Splunk security_content

    不要自己从零写。先抄,跑起来,有假阳性再裁剪。

    9.4 阶段 3:每月一次 micro-drill(月度循环)

    一个人扮红队,用 Atomic Red Team 的脚本跑一个 Technique——这是开源的红队工具,每个 Technique 对应一条 Atomic test,一条命令就能跑。

    # 示例:模拟 T1003.001 LSASS dump
Invoke-AtomicTest T1003.001

    跑完一小时后,另一个人扮蓝队,去 SIEM 里找告警——找到了打绿、没找到打红、半小时才到的打黄。

    一个月跑一个 Technique,一年覆盖 10 个。这就是最小化的紫队循环。

    9.4 阶段 4:产物固化(Quarter 末)

    每个季度把循环产物固化:检测规则进 Git 加版本;Runbook 写到 wiki;ATT&CK 覆盖图更新到 Navigator;工单进 Jira、跟踪关闭率。

    小公司紫队最大的失败模式不是"做不到",是"做了忘了"——演练一次,工单堆在那里,半年后再演练发现盲区还在。只要工单流转起来,即使是 1 个人的紫队,半年后也能拉开和"啥都不做"的同行 5 个身位。

    9.5 工具清单(全开源)

    日志收集: Wazuh / Elastic Beats / Fluent Bit
SIEM:    Wazuh / OpenSearch / Splunk Free
EDR:     osquery / Velociraptor / Wazuh
攻击模拟: Atomic Red Team / Caldera / Stratus Red Team
规则库:   Sigma / Elastic Rules / Splunk ESCU
可视化:   MITRE ATT&CK Navigator
威胁情报: MISP / OpenCTI

    全部加起来 0 元。差的不是工具,是坚持每月跑一次的纪律。

    十、踩坑提醒

    1. 红队 = 渗透测试——错。渗透是点,红队是链,目标和时间尺度完全不同
    2. 紫队是"中等水平的队"——错。紫队是红蓝同步工作的方式,不是组织
    3. 检测规则越多越好——错。一条假阳性高的规则比没规则还糟,会让分析师麻木
    4. 买了 EDR 就有了蓝队——错。EDR 是数据源,没有人写规则、看告警、做响应,等于买了个昂贵的电子记录仪
    5. 演练不演练取证——错。能检测但还原不出来"攻击者动了什么",对外通报时一句话都说不出
    6. 演练只测技术、不测流程——错。72 小时报告、对客户通知、法务介入,这些"流程钟"比技术响应更容易掉链子
    7. 拿 ATT&CK 覆盖率当唯一 KPI——半错。覆盖率只看"有没有规则",不看"规则准不准",还要配合 MTTR(响应时间)和误报率
    8. 以为红队找出来的洞修了就完事——错。修洞 + 写检测是两件事——你不可能修完所有洞,但你应该能检测到任何洞被打
    9. 小公司觉得"我又没数据,不会被打"——错。勒索软件不挑大小,80% 的勒索受害者员工数 < 200
    10. 以为 HW 期间挺过去就行——错。真攻击者不看日历。常态化的紫队循环比 HW 突击有用 100 倍
