注入家族:SQL / NoSQL / 命令 / LDAP / 模板 / 表达式 ​

讲注入最大的误区是把它当成一堆没关系的漏洞——「SQL 注入是数据库问题、命令注入是 shell 问题、SSTI 是模板问题」,各家修各家的。这种切片视角让人永远学不会注入。真相是:所有注入只有一种漏洞——某个解释器(SQL 引擎 / shell / 模板引擎 / LDAP 服务器 / 表达式引擎)接收了一个字符串,字符串里同时混了"开发者写的代码"和"用户输入的数据",而开发者用字符串拼接生成了这个字符串。OWASP 把 A03:Injection 单列为一类是有道理的——修法是同一个,心智是同一个,漏掉一个地方就全军覆没。这一篇把六类注入用同一把尺子量,让你看完之后写代码时,任何一行字符串拼接都触发条件反射。

一句话先记住:注入 = 把"数据"递给某个解释器,解释器把"数据"当成"代码"执行——根因就是「代码和数据共用一个传输通道(字符串)」。修法只有一条:让代码和数据走不同的通道——SQL 用 prepared statement / 参数绑定、shell 用 execve 数组参数 / 不调 shell、模板把用户输入塞进数据上下文而不是模板源码、LDAP 用结构化构造器。任何"我写正则过滤危险字符"的方案都是错的——攻击者的语法集合永远比你的黑名单大。

一、注入的根本原因:字符串拼接 ​

1.1 一个最朴素的模型 ​

世界上所有注入漏洞都长成这一行:

output = template_string + user_input
某个解释器(output)

只要中间是"字符串相加",就一定有注入。下面这些代码,长得不一样,但都是这同一个模型:

python
# SQL
cursor.execute("SELECT * FROM users WHERE id = " + uid)

# Shell
os.system("convert " + filename + " out.png")

# 模板
template = "Hello " + user_input
env.from_string(template).render()

# LDAP
filter = "(uid=" + username + ")"

# Spring SpEL
parser.parseExpression("user.name == '" + name + "'")

# NoSQL(MongoDB)
db.users.find({ username: req.query.user })   # 看似没拼,但 user 可能是个对象

第六个特别诡异——它"看起来"没有字符串拼接,但当客户端发 ?user[$gt]= 时,框架把 user 解析成了对象 {$gt: ''},结构本身就是注入。这就是 NoSQL 注入的独特之处。

1.2 为什么字符串拼接是万恶之源 ​

字符串是单维度的——它没有"这一段是代码、那一段是数据"的元信息。解释器拿到一坨字符串后,只能按语法规则解析,语法规则一旦遇到「特殊字符 + 未转义」就会把数据升格成代码。

开发者心智:        SELECT * FROM users WHERE id = ?
            ↓ 拼接
传给解释器:        SELECT * FROM users WHERE id = 1 OR 1=1
            ↓ 解析
解释器理解:        SELECT * FROM users WHERE (id = 1) OR (1=1)
                  → 全表返回

开发者以为自己写的是一个模板,解释器看到的却是一段全新的代码。中间发生了什么?——OR 1=1 这几个字符,在 SQL 语法里有「特殊地位」,字符串拼接没法把它"降级"成普通字符。

1.3 修复的唯一正确方向:分离通道 ​

正确的解法只有一种:让"代码"和"数据"通过不同的参数传给解释器。

# 错(同一通道):
exec("SELECT * FROM u WHERE id = " + uid)

# 对(双通道):
exec("SELECT * FROM u WHERE id = ?",   <-- 代码通道
     [uid])                             <-- 数据通道

解释器拿到两份输入后,永远不会把数据通道里的内容当成代码——这是从语言层面、协议层面的隔离,不是字符过滤。这就是 prepared statement、execve(argv[])、模板的「context 渲染」共同的本质。

一切修复都是把字符串拼接换成结构化构造。下面六小节,你会看到这同一招在六个场景里反复出现。

二、SQL 注入:注入家族的祖师爷 ​

2.1 经典 PoC ​

python
# 漏洞代码(Python + sqlite3)
def login(username, password):
    sql = f"SELECT * FROM users WHERE name = '{username}' AND pw = '{password}'"
    return cursor.execute(sql).fetchone()
用户名输入:admin' --
密码输入:任意

实际执行:
SELECT * FROM users WHERE name = 'admin' --' AND pw = '...'

-- 是 SQL 注释,后面整段被吃掉,登录直接绕过。这就是 1998 年的"经典 ' OR 1=1",二十多年过去依然在 OWASP Top 10 第三名。

2.2 攻击者的常见四步 ​

步骤目的典型 payload
1. 探测看是否有注入点' '-- 1 OR 1=1
2. UNION 取数据把别的表数据拉出来UNION SELECT user, pw FROM admins
3. 布尔盲注没回显时用 if 判断猜数据AND SUBSTR(pw,1,1)='a'
4. 时间盲注连布尔都看不出时用 sleepAND IF(...,SLEEP(5),0)

注意第 3、4 步——「站点不显示错误,我就安全了」是错的。攻击者完全可以用「页面是否返回、响应是否变慢」当 oracle,一比特一比特把数据库刮干净。一个完整的盲注 exploit 跑几小时,但它真的能跑出来。

2.3 prepared statement:正确解法 ​

python
# 正解
cursor.execute(
    "SELECT * FROM users WHERE name = ? AND pw = ?",
    (username, password)
)

数据库驱动会把 SQL 模板先送到数据库,数据库先完成解析和优化(此时 ? 是占位符,不是数据),之后才把参数值绑定上去。参数永远不会被当成 SQL 语法的一部分——这是协议层的保证,不是字符过滤。

2.4 为什么 prepared statement 不是万能 ​

有三个场景 prepared statement 救不了你,这是面试和真实工作中最容易翻车的点:

场景 1:动态表名 / 列名

python
# 错
cursor.execute(f"SELECT * FROM {table} WHERE id = ?", (id,))

表名不是值,? 不能占位。这里只能用白名单:

python
# 对
ALLOWED_TABLES = {"users", "orders", "products"}
if table not in ALLOWED_TABLES:
    raise ValueError("非法表名")
cursor.execute(f"SELECT * FROM {table} WHERE id = ?", (id,))

场景 2:IN 子句

python
# 错
cursor.execute(f"SELECT * FROM u WHERE id IN ({','.join(ids)})")

IN 后面跟一个变长的列表,? 不能一次占多个位置。正解是动态生成同数量的占位符:

python
# 对
placeholders = ",".join(["?"] * len(ids))
cursor.execute(f"SELECT * FROM u WHERE id IN ({placeholders})", ids)

场景 3:ORDER BY / LIMIT 方向

python
# 错
cursor.execute(f"SELECT * FROM u ORDER BY {col} {direction}")

排序字段和方向(ASC/DESC)也是 SQL 关键字,只能白名单:

python
ALLOWED_COLS = {"id", "name", "created_at"}
direction = "ASC" if direction.upper() == "ASC" else "DESC"

记住一个口诀:值用占位符,标识符用白名单——任何"我把表名也拼上去了"的代码都是炸弹。

2.5 ORM 也能注入 ​

很多人以为「我用了 ORM,SQL 注入就消失了」——错。ORM 提供了"逃生舱"来执行原生 SQL,只要逃生舱被用了字符串拼接,注入立刻回来:

python
# Django ORM:看似安全
User.objects.filter(name=user_input)   # 这是对的,Django 会参数化

# 但这是错的:
User.objects.raw(f"SELECT * FROM user WHERE name = '{user_input}'")

# Sequelize(Node)同理
User.findAll({ where: { name: input } })                 # 对
sequelize.query("SELECT * FROM user WHERE name='" + input + "'")  # 错

# Hibernate HQL
session.createQuery("from User where name = '" + name + "'")  # 错
session.createQuery("from User where name = :name").setParameter("name", name)  # 对

ORM 不是免疫卡——它只是"默认参数化"。每次你写 raw / query / createQuery(String),都要回到 SQL 注入的心智。

三、NoSQL 注入:换了语法,没换本质 ​

很多人以为「NoSQL 没有 SQL,所以没有注入」——这是 2014 年最流行的错误信仰。NoSQL 把字符串语法换成了 JSON 结构,注入变成了"结构注入",但本质完全一样。

3.1 MongoDB 的 $ 操作符注入 ​

javascript
// 漏洞代码(Node + Express + Mongo)
app.post('/login', async (req, res) => {
    const user = await db.collection('users').findOne({
        username: req.body.username,
        password: req.body.password
    });
});

看起来没有字符串拼接——但攻击者发请求时,只要 body 是 JSON,字段就不一定是字符串:

json
{
  "username": "admin",
  "password": { "$gt": "" }
}

$gt 是 MongoDB 的操作符——「大于某个值」。{"$gt": ""} 意思是「密码字段大于空串」,几乎所有密码都满足,登录绕过。

根因:req.body.password 在 JSON 解析后变成了对象,框架把它当成查询条件直接交给驱动。字符串拼接的味道没了,但「数据被当成代码(查询语法)」的本质还在。

3.2 $where:JS 注入 ​

更恐怖的是 $where——MongoDB 允许在查询里塞一段 JavaScript 函数:

javascript
// 漏洞代码
db.users.find({
    $where: `this.username == '${input}'`
});

如果 input 是 '; while(1){}; //,就是服务端 DoS;如果 input 是 '; return true; //,就是全表返回。MongoDB 内嵌了一个 JS 引擎,你在跟它玩 SSTI。

3.3 防御 ​

javascript
// 1. 强制字段类型
const username = String(req.body.username);  // 转成字符串

// 2. 用 schema 校验(Joi / Zod / Mongoose schema)
const schema = Joi.object({
    username: Joi.string().required(),
    password: Joi.string().required()
});

// 3. 永远禁用 $where(没必要)
mongoose.set('sanitizeFilter', true);  // express-mongo-sanitize

核心心智:在 NoSQL 里,数据类型本身就是攻击面——把所有来自用户的字段强制转成期望类型,操作符就构造不出来。

四、命令注入:把字符串递给 shell 的灾难 ​

4.1 经典 PoC ​

python
# 漏洞代码:从用户输入生成缩略图
import os
def make_thumbnail(filename):
    os.system(f"convert {filename} -resize 100x100 thumb.png")
用户输入文件名:
photo.jpg; rm -rf /tmp; #

实际执行:
convert photo.jpg; rm -rf /tmp; # -resize 100x100 thumb.png
            ^^^^^^^^^^^^^^^^^^^ 第二条命令被 shell 执行

根因:os.system 把字符串交给 /bin/sh -c,shell 会解释 ; | & ` $() 这些元字符。只要你过了 shell 这一关,数据就可以变代码。

4.2 危险的 API 一览 ​

语言危险 API(走 shell)安全 API(不走 shell)
Pythonos.system, subprocess.run(shell=True), os.popensubprocess.run([...], shell=False)
Nodechild_process.exec, execSyncspawn, execFile
JavaRuntime.exec(String)Runtime.exec(String[]), ProcessBuilder
Goexec.Command("sh", "-c", str)exec.Command("convert", file, ...)
Ruby反引号 `cmd`, system(str)system("convert", file)

记住一个铁律:任何 shell=True 的调用 + 用户输入 = 命令注入。

4.3 正确的写法 ​

python
# 正解 1:数组参数 + 不走 shell
import subprocess
subprocess.run(
    ["convert", filename, "-resize", "100x100", "thumb.png"],
    shell=False,           # 不调 /bin/sh
    check=True
)

数组每个元素都作为一个 argv 参数直接交给 execve 系统调用——内核根本不解析元字符,; | $() 全是普通字符。这是从 syscall 层面隔离了「代码」和「数据」。

python
# 正解 2:白名单 + 严格路径校验
import re
if not re.fullmatch(r"[a-zA-Z0-9_.-]+\.(jpg|png)", filename):
    raise ValueError("非法文件名")

优先用 1,实在要拼字符串就上 2。不要相信 shlex.quote——它能做对,但很容易漏一个分支。

4.4 经典案例:Shellshock(CVE-2014-6271) ​

2014 年 bash 自己出了个史诗级 bug——解析环境变量时,如果值以 () { 开头,bash 会把后面当函数定义,然后继续执行函数定义后的语句。CGI、DHCP 客户端、OpenSSH 的 ForceCommand——所有「把用户控制的字符串当环境变量塞给 bash」的地方瞬间变成 RCE。教训:解释器自己都可能有 bug,最安全的解释器是不调用它。

五、LDAP 注入:被遗忘但仍存在的角落 ​

5.1 LDAP 过滤器语法 ​

LDAP 用一种类 S-表达式的过滤器查询:

(uid=alice)                          单一条件
(&(uid=alice)(active=TRUE))          AND
(|(uid=alice)(uid=bob))              OR
(uid=a*)                             通配符

问题:很多企业内网在 LDAP 之上构建认证(AD / OpenLDAP),后端的认证逻辑往往这样写:

python
# 漏洞代码
def auth(username, password):
    filter = f"(&(uid={username})(userPassword={password}))"
    return ldap.search(filter)
用户名输入:*
密码输入:*

实际过滤器:
(&(uid=*)(userPassword=*))
→ 匹配任意用户、任意密码
→ 认证绕过

5.2 修复 ​

LDAP 过滤器也是字符串拼接——治法和 SQL 一样:结构化构造 + 转义。

python
# Python ldap3
from ldap3.utils.conv import escape_filter_chars

filter = f"(&(uid={escape_filter_chars(username)})" \
         f"(userPassword={escape_filter_chars(password)}))"

escape_filter_chars 会把 * ( ) \ \0 转义成 \2a \28 等十六进制形式,LDAP 服务端不会把它们再解析成元字符。

更狠的做法:不要用 LDAP 比对密码——把密码 hash 存自己 DB 里,LDAP 只查身份是否存在。少一个解释器,少一个注入面。

六、SSTI 模板注入:一行 {{7*7}} 引发 RCE ​

6.1 漏洞模型 ​

模板引擎(Jinja2 / Twig / Freemarker / Velocity)的"工作"就是把模板字符串 + 数据 → 渲染成最终输出。模板里写的 {{ x }} 是代码,数据 x 是数据——这是模板的初衷。

但很多开发者搞错了 ——把用户输入塞到了"模板"位置,而不是"数据"位置:

python
# 漏洞代码(Flask + Jinja2)
@app.route('/hello')
def hello():
    name = request.args.get('name')
    template = f"<h1>Hello {name}</h1>"   # ← 用户输入进了模板!
    return render_template_string(template)
URL: /hello?name={{7*7}}
渲染输出:Hello 49
                ↑ 模板引擎把它当代码执行了

6.2 一路升级到 RCE ​

{{7*7}} 只是探测——一旦确认模板注入存在,Jinja2 里通过对象树爬升可以拿到 Python 内置函数,最终调用 os.system(具体 payload 涉及武器化,这里不展开)。Twig / Freemarker / Velocity 同理——每个引擎都有自己的对象访问路径。

引擎探测 payload严重程度
Jinja2(Python){{7*7}} → 49RCE
Twig(PHP){{7*'7'}} → 49RCE
Freemarker(Java)${7*7} → 49RCE
Velocity(Java)#set($x=7*7)$x → 49RCE
ERB(Ruby)<%= 7*7 %> → 49RCE

6.3 修复 ​

python
# 错(把用户输入当模板)
return render_template_string(f"Hello {name}")

# 对(模板是常量,用户输入是数据)
return render_template_string("Hello {{ name }}", name=name)

心智:模板源码是代码,只能来自开发者写死或从可信存储读出;用户输入永远走数据通道。用户能控制模板的字符串内容时,就一定是 RCE。

6.4 真实案例:Equifax 数据泄露(2017) ​

Equifax 1.45 亿美国人信用数据泄露,根因是 Apache Struts2 的 OGNL 表达式注入(CVE-2017-5638)。Struts2 的文件上传错误消息里调用了 OGNL 表达式引擎,攻击者把 OGNL 表达式塞进 Content-Type 请求头,服务端把它当代码执行,直接 RCE 拿 shell。

OGNL 是另一种"表达式引擎"——和模板引擎本质同源,只要服务端"在某个不该执行表达式的地方执行了用户输入",就是 OGNL 注入。

Equifax 当时已经知道这个漏洞两个月,但补丁没打全——这告诉我们注入漏洞不是"开发完就结束",是要持续打补丁。详见第 26 篇供应链安全。

七、表达式注入:Spring SpEL / OGNL ​

SpEL(Spring Expression Language)和 OGNL 是 Java 生态里的"应用层表达式引擎"——本来用于配置文件里写 ${user.name},出现在用户输入路径上就是 RCE。

7.1 SpEL 注入示例 ​

java
// 漏洞代码
@GetMapping("/greet")
public String greet(@RequestParam String name) {
    ExpressionParser parser = new SpelExpressionParser();
    Expression exp = parser.parseExpression(name);   // ← 直接 parse 用户输入
    return exp.getValue(String.class);
}
?name=T(java.lang.Runtime).getRuntime().exec('...')
→ RCE

修法:永远不要把用户输入交给 SpEL/OGNL parser——表达式应该是开发者写死的、配置文件里读的、白名单 lookup 出来的。如果业务真的需要"用户可写表达式",必须用沙箱表达式引擎(限制可访问的类、禁用反射)。

7.2 历史案例:Struts2 OGNL 系列 ​

Struts2 因为 OGNL 注入,从 2010 到 2020 出过十几个编号的 RCE(S2-001 到 S2-062)——每次修补一个,又因为 OGNL 引擎本身能力太强,绕过 patch 的 payload 又冒出来。这是"在错误的位置加了一个图灵完备的解释器"的代价——你永远修不完。

行业教训:生产配置不要用图灵完备的表达式引擎。需要表达能力时优先选「能力受限」的方案——JSON 配置、纯数据 DSL,而不是嵌一个完整的脚本引擎。

八、错误写法 vs 正确写法:统一对照表 ​

九、纵深防御:单点修不够 ​

注入是「写代码时的纪律」,但你写完之后还有四层兜底——任何一层都不是必须的,但加起来才是工程上的"安全"。

9.1 输入层:校验 + schema ​

在请求入口用 Joi / Zod / Pydantic / Bean Validation 校验:类型、长度、字符集、范围。这不是为了防注入(防御主战场在输出),但能大量减少到达 SQL / shell 的"奇怪输入"。

python
class LoginReq(BaseModel):
    username: constr(regex=r"^[a-zA-Z0-9_]{3,32}$")
    password: constr(min_length=8, max_length=128)

9.2 ORM / 框架层:默认安全的 API ​

选用默认参数化的 ORM——Django ORM、SQLAlchemy Core、Prisma、Entity Framework。把"是否使用 raw" 当成 code review 的强检查项。

9.3 数据库层:最小权限 + 视图隔离 ​

sql
-- 应用账号:只能查 / 改业务表
CREATE USER app_ro WITH PASSWORD '...';
GRANT SELECT, INSERT, UPDATE ON users, orders TO app_ro;

-- 不要:
GRANT ALL ON *.* TO app_ro;

即使被注入了,攻击者也偷不到 mysql.user 表。这是「假设迟早会被攻破」的防御纪律。

9.4 监控层:WAF + 异常 SQL 检测 ​

  • WAF(ModSecurity / Cloudflare)拦截最典型的 payload(UNION SELECT、OR 1=1、<script>、{{7*7}})——它会被绕过,但能拦掉自动化扫描器和 95% 的脚本小子。
  • DB 层加慢查询 / 异常查询监控——正常业务不会有 SUBSTR(pw, 1, 1) 这种盲注特征。
  • 蜜罐字段:在表里塞几个永远不会被合法代码查询的字段,一旦它被 SELECT 出来,几乎可以确定遭遇注入。

9.5 SDL:把规则写进流水线 ​

  • SAST(Semgrep / CodeQL)写规则扫 \.execute\(f["']、os\.system\(、shell=True 这些 pattern;DAST(sqlmap 跑测试环境);代码审计 checklist 必有一条:全工程搜一遍 raw SQL 逐个 review。详见第 27 篇 SDL。

十、踩坑提醒 ​

  1. 「我用了 ORM,SQL 注入消失了」——错。raw / query / createQuery(String) 是逃生舱,一旦字符串拼接就回到 1998 年。
  2. 「我把 ' 替换成 '' 就行了」——黑名单永远不够,不同 DB 有 \ 0x00 等其他转义路径,数字字段不需要引号也能注入。
  3. 「我的输入是数字所以安全」——?id=1 UNION SELECT... 也能注入,数字字段同样要参数化。
  4. 「NoSQL 没 SQL 所以没注入」——$where $gt 让你比 SQL 还惨,字段类型就是攻击面。
  5. 「shell=False 没用,反正传的还是用户输入」——错。shell=False + 数组参数 = 内核不解析元字符,安全等级断崖式上升。
  6. 「我用 eval 但加了 try/except」——eval 用户输入永远没救,异常处理不能阻止 RCE。
  7. 「模板里的 {{ }} 不会执行因为我没装插件」——不,模板引擎默认就能执行表达式 / 调用对象方法,Jinja2 / Twig / Freemarker 全是 RCE 一步之遥。
  8. 「shlex.quote 已经够安全了」——能用但容易漏 Windows 平台、容易被嵌套 shell 绕过。优先 subprocess.run([], shell=False)。
  9. 「WAF 拦得住所以代码差点没事」——WAF 是兜底不是主防,绕过 WAF 的 payload GitHub 上一搜一大把。
  10. 「日志里我直接拼用户输入应该没事」——错。Log4Shell(CVE-2021-44228)就是日志库自带表达式引擎导致的史诗级 RCE,任何"看似无害"的字符串拼接 + 任何"内置解释器"都是地雷。
