非对称加密(RSA / DH / 椭圆曲线 / 后量子初探) ​

上一篇讲对称加密时,反复回避了一个问题——「AES 的那个 32 字节密钥,你和对方怎么先达成共识」。两个人面对面拿 U 盘拷贝当然可以,问题是 HTTPS 里你和某个从没见过的服务器,怎么在全程都可能被监听的信道上,凭空协商出一个双方都知道、攻击者却推不出来的对称密钥?这就是 1970 年代之前所有密码学家都觉得「这不可能」的问题——直到 Diffie / Hellman / Merkle / Rivest / Shamir / Adleman 几个人在 1976-1977 年用纯数学把这件事干成,整个现代互联网的底座才搭起来。

一句话先记住:非对称加密 = 「单向门」的数学化 —— 「正着算容易,反着算难到现实里做不到」的数学陷阱(大数分解 / 离散对数 / 椭圆曲线点乘),让一个人能公开自己的公钥,任何人都能用它加密 / 验签,只有持有私钥的人能解密 / 签名。它不是对称加密的「升级版」,而是用来解决对称加密无法解决的问题——密钥分发 + 身份认证 + 不可抵赖。真实系统里两者永远配合使用:非对称解决"开局",对称负责"主体",这是 TLS / SSH / Signal / 几乎所有现代加密协议的通用结构。

一、问题:对称密钥怎么交换 ​

1.1 一个"鸡生蛋"的死结 ​

Alice 想给 Bob 发密文,需要 AES 密钥 K
   ↓
但传 K 给 Bob 需要"安全信道"
   ↓
如果有安全信道,Alice 直接传明文就行了啊?
   ↓
死循环

这是 1970 年代以前的密码学绝境——所有方案都假设「双方事先有共享密钥」,但现实里:

  • 军方:每月派信使送密码本,线下交付
  • 银行:专人送 PIN 信封到客户家
  • 大使馆:外交邮袋送 one-time pad
  • 互联网:没法做 —— 你和某个未曾谋面的网站,凭什么先约好密钥?

如果非对称加密没被发明出来,今天根本不会有 HTTPS、电商、网银,所有「线上不见面交易」的场景都成立不了。

1.2 非对称加密的灵感:单向函数 ​

数学上有一类操作:正算容易,反算难。

正向(简单):
   3 × 7 = 21          (1 ns)
   123 × 457 = 56211   (10 ns)
   两个 1024-bit 质数相乘 = 一个 2048-bit 数(几 μs)

反向(困难):
   56211 = ? × ?       (几秒能试出来)
   2048-bit 数 = ? × ? (现代最强算力 + 最佳算法,需要 10⁹ 年)

「大数分解」就是一个经典的单向函数候选。如果我能造出一种加密,加密用 公开的"乘积",解密需要 保密的"因子",就完成了「公开可加密、私密才能解」的不对称结构——这正是 RSA 的核心思想。

「单向门」的本质不是"绝对不可逆",而是"反向所需的算力远超攻击者预算"。所有现代密码学都建立在「计算复杂度差距足够大」的假设上——一旦算法或硬件突破(量子计算机),整个根基就要重建。

二、RSA:大数分解的工程化 ​

2.1 数学直觉(不堆公式) ​

1. 随机选两个大质数 p, q(例如各 1024 bit)
2. 计算 n = p × q                  → n 是 2048 bit
3. 选一个公开指数 e(常用 65537)
4. 算出私有指数 d(满足某个数论条件)

公钥 = (n, e)                       任何人都能拿到
私钥 = (n, d) 或者 (p, q, d)        只有你自己有

加密 / 签名的运算就是模幂:c = m^e mod n,解密就是 m = c^d mod n。安全性的全部根基是:

「知道 n 反推 p、q」这件事,在 n = 2048 bit 下,没人会。

如果哪天有人能在合理时间里分解 2048-bit 的 n,全世界所有 RSA 加密的旧数据一夜被解开。这就是为什么 NSA 等机构会囤积加密流量——「现在解不开,等技术够了再说」(harvest now, decrypt later)。

2.2 RSA 的两副面孔 ​

用途流程谁加密 / 签谁解密 / 验
加密A 发密文给 BA 用 B 的公钥 加密B 用 B 的私钥 解密
签名A 签一份文档A 用 A 的私钥 签所有人用 A 的公钥 验

两件不同的事,用同一套数学——这也是新手最容易混的:「公钥加密 / 私钥签名」方向是反的,不能套同一个心智。

2.3 PKCS#1 v1.5 padding:历史灾难 ​

RSA 不能直接加密 m——必须先做 padding(填充)。原因有二:

  1. 小消息会被秒解:m^e mod n,如果 m^e < n,直接开 e 次方就出来了
  2. 相同消息加密两次密文一样:不引入随机性会泄漏模式

1993 年 RSA 实验室给出的 padding 方案叫 PKCS#1 v1.5:

加密时填充结构:
0x00 || 0x02 || <随机字节 ≥ 8> || 0x00 || <真实消息>

解密时检查:
开头是 0x00 0x02 吗?
找到第一个 0x00 分隔符吗?
真实消息长度合理吗?

问题:这个检查的"成功 / 失败"信息会泄漏给攻击者——Bleichenbacher 在 1998 年发现,通过观察服务器对"恶意密文"的响应(返回错误 / 不返回 / 报错码不同),可以一点点反推明文——史称 Bleichenbacher 攻击,简称 百万消息攻击(因为需要约 100 万次猜测)。

更可怕的是,这个漏洞反复回潮:

  • 1998 Bleichenbacher 原版
  • 2003 Klima / Pokorny / Rosa(针对 SSL/TLS)
  • 2017 ROBOT(Return Of Bleichenbacher's Oracle Threat)—— Facebook / PayPal / 思科都中招
  • 2023 还有变种(Marvin Attack)

结论:PKCS#1 v1.5 是个伪需求工程——本来想让格式校验"安全",结果安全成了 oracle。现代系统应当用 OAEP(Optimal Asymmetric Encryption Padding),它用一对 hash 把消息和随机数充分混淆,没有可观测的格式差异给攻击者作信号。

2.4 RSA 的工程现实 ​

  • 2048 bit 是当前底线,1024 bit 已被认为不安全(NIST 2030 年要求 ≥3072)
  • 加密慢:RSA-2048 加密一次约 0.1 ms,ECC-256 是它的 10-20 倍快(详见后文)
  • 不能加密大数据:RSA-2048 一次只能加 ~245 bytes(OAEP 后),所以永远只用来加密"对称密钥",再用 AES 加密真正的 payload
  • 私钥文件很大:RSA-2048 私钥约 1.2 KB,ECC-256 私钥只有 ~120 字节

新系统不要再用 RSA 加密,除非兼容老协议;签名场景也尽量迁移到 Ed25519。RSA 在 2030+ 会逐步退场。

三、Diffie-Hellman:史上最优雅的密钥交换 ​

3.1 一张图讲完 ​

DH 的天才之处在于:Alice 和 Bob 不传密钥,但能算出同一个密钥。

全网公开:大质数 p,生成元 g
            ↓
        ┌───────┴───────┐
        ↓               ↓

    Alice                            Bob
    ─────                            ───
    选秘密 a                          选秘密 b
    算 A = g^a mod p                 算 B = g^b mod p

           ── A ──────────────────→
           ←──────────────── B ──

    收到 B                            收到 A
    算 K = B^a mod p                 算 K = A^b mod p

         K = (g^b)^a = g^(ab) mod p = (g^a)^b
              ↑                            ↑
         Alice 算出的                  Bob 算出的
         共享密钥                       共享密钥(相同)

    ↑                                 ↑
    监听者看到了 p, g, A, B
    但要从 A 反推 a 需要解"离散对数"
    在 p 足够大时,他做不到 → 算不出 K

核心数学事实:

正向(简单):a 已知 → 算 g^a mod p
反向(困难):g^a mod p 已知 → 推 a
            ↑
       这就是「离散对数问题」
       和大数分解一样,目前没有多项式时间解

3.2 前向保密(Forward Secrecy / PFS) ​

DH 最革命性的副作用:每次会话都新生成临时的 a、b —— 这次握手用完就丢。即使将来:

  • 服务器的长期私钥被泄漏
  • 攻击者囤积了过去所有密文

他也解不开旧会话 —— 因为旧会话的密钥是 g^(ab) mod p,而 a、b 早就被销毁,无法重建。

这就是 PFS(Perfect / Forward Secrecy) 的本质:「过去的安全不依赖于现在的私钥」。

没 PFS(旧 RSA 密钥交换):
   会话密钥 = 用服务器 RSA 公钥加密发过去
   ↓
   服务器私钥泄了 → 历史所有会话全暴露

有 PFS(DHE / ECDHE):
   会话密钥 = 临时 DH 算出来,握手后销毁
   ↓
   服务器私钥泄了 → 只影响"以后想冒充服务器",历史会话仍然安全

现代 TLS 1.3 强制要求 PFS —— 全部用 ECDHE,RSA 密钥交换被砍掉。这是 Snowden 爆料后整个行业的集体反应——「大国可以囤密文等密钥」,所以必须从协议层保证「囤了也白囤」。

3.3 DH 的工程化变种 ​

名称含义用途
DH经典 DH,模乘法群已基本废弃,慢
DHEEphemeral DH,每次新 a/bTLS 1.2 的 PFS 选项,慢
ECDH椭圆曲线版静态长期密钥用
ECDHE椭圆曲线 + 临时密钥TLS 1.3 主力,快 + PFS
X25519用 Curve25519 的 ECDHE现代默认,Signal / WireGuard / SSH 都用

3.4 DH 自身的陷阱:中间人 ​

DH 解决了"算出共享密钥",但完全不解决"对方是谁":

Alice ──A──→ [Mallory] ──A'──→ Bob
Alice ←──B'── [Mallory] ←──B── Bob

Mallory 和 Alice 协商出一个密钥 K1
Mallory 和 Bob   协商出一个密钥 K2
Alice 和 Bob 各自以为在和对方说话,其实都在和 Mallory 说话

DH 必须配合身份认证(数字签名 / 证书 / 预共享密钥),否则裸跑就是中间人靶子。这就是 TLS 里 ECDHE 加上 RSA / ECDSA 签名的原因——签名做认证,DH 做密钥协商,分工明确。

四、椭圆曲线密码(ECC):为什么 256-bit ≈ 3072-bit RSA ​

4.1 不数学,讲心智 ​

ECC 不是「另一种加密算法」,而是换了一个数学结构来做单向函数。RSA 在「整数模 n 的乘法群」上玩,ECC 在「椭圆曲线点的加法群」上玩。具体曲线长什么样不重要,你需要记住的只有一件事:

在椭圆曲线上,「反推」问题(ECDLP — 椭圆曲线离散对数问题)更难——已知的最快算法是**√n 量级**(Pollard's rho),而整数离散对数 / 大数分解都有亚指数级算法(GNFS,数域筛法)。

攻击 RSA-2048    ≈ 攻击 ECC-224
攻击 RSA-3072    ≈ 攻击 ECC-256        ← 当前主流
攻击 RSA-15360   ≈ 攻击 ECC-512

同等安全强度下,ECC 密钥小一个数量级——这带来三个工程上的巨大优势:

  1. 更快:运算量小 10-20 倍
  2. 更小:证书 / 签名 / 私钥都小
  3. 更省电:移动端 / IoT 友好

4.2 RSA vs ECC vs Ed25519 工程对比 ​

维度RSA-2048ECC-256 (secp256r1)Ed25519
安全强度~112 bit~128 bit~128 bit
公钥大小256 B64 B32 B
私钥大小~1.2 KB~120 B32 B
签名大小256 B64 B64 B
密钥生成~100 ms~0.1 ms~0.05 ms
签名速度~1 ms~0.5 ms~0.05 ms
验签速度~0.05 ms~1 ms~0.15 ms
加密 / KEX慢 + 不推荐不直接加密(用 ECDH)不加密(只签名)
量子安全否否否
常数时间实现难难(secp 系列)天然容易(设计目标)
推荐场景兼容老系统美标 / FIPS 合规新系统默认

注:RSA 验签比签名快得多(公开指数小,模幂便宜),这是 RSA 唯一仍有竞争力的场景——大量验签 + 偶尔签名的 CA 体系。

4.3 ECC 也有"坑曲线" ​

ECC 的安全性严重依赖曲线参数选取。不是所有曲线都安全:

  • 早年 NSA 推动的 Dual_EC_DRBG 被发现是个后门——参数里藏了 NSA 才知道的"陷门"
  • 一些低质量曲线在小子群攻击 / 不变 j-不变量攻击下崩溃

所以工业界只用经过广泛审计的几条曲线——下一节展开。

五、曲线之争:secp256r1 / X25519 / Ed25519 ​

5.1 三条主流曲线 ​

曲线别名设计者用途政治色彩
secp256r1P-256, prime256v1NIST / NSAECDH / ECDSA美标,FIPS 合规必需
Curve25519X25519 (ECDH)Daniel J. Bernstein密钥交换"反 NSA"派系
Edwards25519Ed25519 (签名)Daniel J. Bernstein签名同上

5.2 为什么 X25519 / Ed25519 后来居上 ​

DJB(Daniel J. Bernstein,「djb」)在 2005 年提出 Curve25519,核心思路是「让正确实现变得容易,让错误实现变得不可能」。这有几个工程后果:

  1. 天然常数时间:不需要刻意写"防时序攻击"的代码,数学结构本身让分支都没有
  2. 没有"无效点"陷阱:任何 32 字节都是合法公钥,免疫"小子群攻击"
  3. 不需要随机数验签(Ed25519 用确定性签名,签名不再依赖 RNG 质量)
  4. 参数完全公开透明(基点 / 余因子全是数学常数,没有 NSA 留缝隙的空间)

而 NIST 曲线被诟病的核心是:

NIST P-256 的参数 c 来自一个 SHA-1 哈希
SHA-1 的输入是个"随机的种子"——但种子的选择没有解释
NSA 是否预先试出"对自己有利的种子",至今没法证伪

Snowden 之后,「不信任美方曲线」成为西方安全社区主流情绪。Signal / WireGuard / SSH / TLS 1.3 全部默认 X25519 + Ed25519,Apple iOS / Android 也广泛采用。新项目选 25519 系,除非合规强制要 P-256。

5.3 工程口诀 ​

要密钥交换       → X25519
要签名           → Ed25519
要兼容老 TLS / 合规 → secp256r1 + ECDSA / SHA-256
要兼容上古系统    → RSA-2048(逐步淘汰)

六、后量子密码学(PQC):2030 年要换的算法 ​

6.1 量子计算威胁了什么 ​

所有基于「整数分解」或「离散对数」的算法,被 Shor 算法在量子计算机上多项式时间破解:

受影响(全军覆没):
   RSA / DSA / DH         ← Shor 算法
   ECC / ECDH / ECDSA     ← Shor 算法
   Ed25519 / X25519       ← Shor 算法

相对安全(只是密钥要加倍):
   AES                    ← Grover 算法,效果 = 把 256-bit 降到 128-bit
   SHA-2 / SHA-3          ← Grover,影响有限

对称加密在量子时代只需把密钥从 128-bit 加到 256-bit,问题不大。非对称加密则要从头换。

6.2 现在是 2030 年之前的什么时点 ​

2015  NIST 启动 PQC 标准化
2022  第一轮入围结果公布
2024  NIST 发布最终标准:
        FIPS 203 — ML-KEM(基于 Kyber,密钥封装)
        FIPS 204 — ML-DSA(基于 Dilithium,签名)
        FIPS 205 — SLH-DSA(基于 SPHINCS+,哈希签名,备份方案)
2025  Cloudflare / Google Chrome / Apple iMessage 已经在生产环境
      启用「ML-KEM + X25519」**混合密钥交换**
2030+ 量子计算机预期能威胁实用 RSA / ECC 的时间窗口

两个时间线在赛跑:量子机的进展 vs 全行业迁移的进度。没人能确定哪个先到,但「现在就开始混合」是唯一稳妥的工程姿势——因为攻击者今天就在囤密文。

6.3 ML-KEM(Kyber)与 ML-DSA(Dilithium)心智 ​

不展开数学,只记三个心智:

  1. 基于"格"(Lattice)问题:学习带误差(LWE)/ 模块 LWE / 短整数解(SIS)——这类问题被认为量子机也搞不定
  2. 密钥变大很多:ML-KEM 公钥 ~800 B,私钥 ~1.6 KB —— 比 X25519 的 32 B 大 25-50 倍
  3. 运算速度反而比 RSA 快,和 ECC 同级

主要痛点:握手包变胖。TLS 1.3 一个 ClientHello 从 ~250 字节涨到 ~1500+ 字节,可能撑爆 MTU 触发分片——已知一些老旧中间盒会出问题(称为 TLS PQC ossification)。Cloudflare 2023-2024 的实际部署数据显示影响 < 1%,但这个工程问题是真实的。

6.4 混合模式(Hybrid) ​

传统单一握手:
   ECDHE (X25519) → 共享密钥

PQC 单一握手(激进):
   ML-KEM → 共享密钥
   ↑
   万一 Kyber 数学被攻破怎么办?
   (新算法还没经过几十年实战检验)

PQC + 经典混合(稳健 ✓):
   X25519 算出 K1
   ML-KEM 算出 K2
   最终密钥 = KDF(K1 || K2)
   ↑
   只要 X25519 和 Kyber 至少有一个还安全,会话就安全

现阶段所有部署都是混合——「用 Kyber 防量子,用 X25519 防 Kyber 出问题」,工程稳健性优先。

Chrome 自 2023 年起对部分流量启用 X25519MLKEM768。看你的 TLS 连接里 KEX 如果出现 X25519MLKEM768,那就是混合模式生效。

七、工程视角:什么场景用什么算法 ​

7.2 决策树 ​

要做新系统的密钥交换?
├── 受 FIPS / 合规约束? → secp256r1 (ECDH-P256)
├── 担心 2030+ 量子威胁? → X25519 + ML-KEM 混合
└── 其他全部场景       → X25519

要做新系统的数字签名?
├── 受 FIPS 约束?       → ECDSA-P256
├── 需要长期归档(>20 年)? → 考虑 SLH-DSA(基于哈希,最保守)
└── 其他全部场景       → Ed25519

要解密 / 加密?
└── 几乎不要直接用非对称加密大消息
    用「KEM + 对称加密」模式:
       非对称协商 / 封装出 32 字节密钥
       AES-GCM 加密真正的数据

7.3 跟非对称加密相关的几条铁律 ​

  1. 永远不要自己实现 —— RSA 实现陷阱超过 30 个,ECC 更多;用 libsodium / Tink / Ring / BoringSSL
  2. 永远不要复用密钥对于不同用途——签名密钥不能拿去加密,反之亦然
  3. 永远不要从一个公钥反推另一个公钥的"关系" —— 这是分层确定性密钥(HD wallets)的设计陷阱
  4. 永远要把非对称加密包在协议里,别裸用 —— 协议层(TLS / Noise / Signal Protocol)处理了一堆你没意识到的边角(防重放、防降级、防中间人……)
  5. 私钥必须存在 HSM / KMS / Secure Enclave —— 服务器明文存 .pem 文件就是定时炸弹

八、踩坑提醒 ​

  1. 以为 RSA 比 AES 更安全——「非对称就是高级版」是错觉,两者解决完全不同的问题,而且非对称 1000 倍慢
  2. 以为公钥能加密任意大数据——RSA-2048 一次只能加 ~245 字节,实践中必须用 KEM 模式(非对称包对称密钥)
  3. 以为换大密钥就安全——RSA-4096 vs RSA-2048 安全性提升远不及 ECC-256;算法选择比密钥长度重要
  4. 以为 ECDH 自带身份认证——不,DH 解决 KEX 但 不解决"对方是谁",必须配签名 / 证书
  5. 以为 PFS 是 TLS 自带的——TLS 1.2 默认握手套件可能不是 ECDHE,要显式配置;TLS 1.3 才强制
  6. 以为 Ed25519 「太新不可靠」——它 2011 年发明,2015 年标准化,Signal / WhatsApp / SSH 用了十年没出过事,远比"古老" RSA 的实现稳
  7. 以为后量子还很遥远—— harvest-now-decrypt-later 已经是现实,国家级攻击者今天就在囤密文
  8. 以为换 PQC 就够了——PQC 算法本身仍可能有数学缺陷被未来发现,混合模式才是稳健选择
  9. 以为 ECDSA 总是常数时间——P-256 的某些实现不是,Ed25519 才天生常数时间;时序攻击在 2018-2024 还在不断爆 ECDSA 漏洞
  10. 以为非对称加密能完全替代 PKI——不能——非对称加密只解决"算法",但"谁的公钥是真的"必须靠证书 / 信任锚 —— 这是下一篇的话题
