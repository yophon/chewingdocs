# 密码学心智:随机数 / 哈希 / KDF / "不要自己造密码"

学密码学最大的陷阱不是「数学难」,是**先去学算法**——花一周搞清楚 AES 的 S-Box、RSA 的模逆元、椭圆曲线的群运算,**然后回过头来,在代码里写出比谁都离谱的漏洞**:用 `md5(password + salt)` 存密码、用 `random.randint()` 生成 token、用 ECB 模式加密日志、用 `==` 比较 HMAC。**算法没记错,工程全错**。这一篇不教任何具体算法的内部构造(那是密码学家的事),只回答一个问题:**作为一个工程师,你应该如何「正确地使用」密码学,以及如何识别自己什么时候在「造火药」**。

> 一句话先记住:**密码学不是数学竞赛,是工程纪律**——选对原语、用对模式、调对参数,99% 的工程问题就解决了。**真正的坑不在算法本身,而在「你以为算法保护了你,其实没有」**:`random` 不是 CSPRNG、`SHA256(password)` 不是密码哈希、`HMAC == 比较`会泄露时序、`AES-ECB` 加密相同明文出相同密文。**这些坑全是工程坑,不是数学坑**。

## 一、为什么"自己造密码学"是反模式

### 1.1 一句行业铁律

密码学社区有一条几乎所有从业者都认的铁律:**Don't roll your own crypto.**

它**不是说「不要写任何加密代码」**,而是说:

```
❌ 不要:                              ✅ 应该:
- 自创算法 / 协议                     - 用经过 20+ 年公开审计的标准算法
- 自己实现 AES / RSA(哪怕照 RFC 抄)  - 用审计过的标准库(libsodium / BoringSSL / Go crypto)
- 「我再套一层 MD5 更安全」           - 在标准协议层组合(TLS / JWT / OIDC)
```

### 1.2 为什么"自己写"几乎一定错

密码学代码的「对」跟普通代码完全不同:

| 普通代码"对" | 密码学代码"对" |
| --- | --- |
| 测试用例都过 | 所有测试通过也可能完全不安全 |
| 性能达标 | 太快可能反而是漏洞(时序攻击) |
| Code review 找问题 | 需要**密码学博士 + 形式化证明** |

**经典例子**:你的 AES 加解密都对、跑通所有 NIST 向量——但实现里有**条件分支依赖密钥某一位**,攻击者测 CPU 时间就能**一位一位还原密钥**:

```c
if (key[i] & 0x80) { do_a(); }  // 跑 10ns
else                { do_b(); }  // 跑 8ns
// 时序差 → 泄露 key[i] 最高位 —— 任何功能测试都查不出来
```

**侧信道是个和算法逻辑无关的工程维度**——这就是为什么"按 RFC 抄"也救不了你。

### 1.3 历史惨案速览

- **WEP**:标准 RC4 + 自创密钥拼接 → 3 分钟可破
- **MS-CHAPv2**:标准 DES/MD4 + 自创挑战响应 → 脱机暴破
- **Telegram MTProto**:自研协议,被密码学家反复打脸
- **大量"自研区块链"**:重新拼现有算法,几乎都有洞

教训:**不是算法实现错,是"对的算法用错地方"**。

### 1.4 工程师该懂的不是算法,是用法

```
不需要懂:                    需要懂:
- AES S-Box 怎么设计           - 什么时候用对称 vs 非对称
- RSA 模逆元怎么算             - 什么时候用 HMAC vs 签名
- 椭圆曲线群运算细节           - 怎么选随机数源、怎么存密码、怎么派生密钥
```

**这一篇就是讲右边这一列**。

## 二、随机数:99% 工程师都搞错的第一关

### 2.1 三种"随机"的区别

```
真随机数(TRNG, True Random Number Generator)
   来源:物理过程(热噪声 / 量子涨落 / 大气噪声 / 鼠标移动)
   特点:不可预测,但**慢、不可复现**
   场景:为 PRNG 提供种子,生成根密钥

伪随机数(PRNG, Pseudo Random Number Generator)
   来源:确定性算法(线性同余、梅森旋转)
   特点:快、可复现(给同样种子产同样序列)
   场景:游戏 / 模拟 / 测试 / 洗牌
   **不能用于密码学**

密码学安全伪随机数(CSPRNG, Cryptographically Secure PRNG)
   来源:确定性算法 + 不可逆运算(AES-CTR / ChaCha20 / 哈希)
   特点:快、不可预测(即使看了前 100 万个输出,也猜不出第 100 万零一个)
   场景:**所有密码学场景**——密钥、token、nonce、salt、session ID
```

### 2.2 工程师最容易踩的坑

```python
# ❌ 错:Python 的 random 模块是 PRNG,不是 CSPRNG
import random
token = ''.join(random.choices('abcdef0123456789', k=32))
# random 用的是梅森旋转,看了 624 个连续输出就能预测后续所有输出

# ✅ 对:用 secrets(Python 3.6+)
import secrets
token = secrets.token_hex(16)         # 32 字符的安全 token
sid   = secrets.token_urlsafe(32)     # 适合放 URL 的 session ID
```

```javascript
// ❌ 错:Math.random() 是 PRNG,V8 用的是 xorshift128+
const token = Math.random().toString(36).slice(2);
// 看了几个输出就能反推内部状态

// ✅ 对:用 Web Crypto / Node crypto
const { randomBytes } = require('crypto');
const token = randomBytes(16).toString('hex');
// 浏览器端
const token = crypto.getRandomValues(new Uint8Array(16));
```

```go
// ❌ 错:math/rand 是 PRNG
import "math/rand"
n := rand.Intn(1000000)

// ✅ 对:crypto/rand 是 CSPRNG
import "crypto/rand"
b := make([]byte, 16)
_, _ = rand.Read(b)
```

| 语言 | ❌ 普通 PRNG | ✅ CSPRNG |
| --- | --- | --- |
| Python | `random` | `secrets` |
| Node.js | `Math.random()` | `crypto.randomBytes()` |
| Browser | `Math.random()` | `crypto.getRandomValues()` |
| Go | `math/rand` | `crypto/rand` |
| Java | `java.util.Random` | `java.security.SecureRandom` |
| Rust | `rand::thread_rng()`(看版本) | `rand::rngs::OsRng` / `getrandom` |

**铁律:只要这个随机数会被"防御"——任何用户不应该能猜的东西——就必须 CSPRNG**:

- session ID / CSRF token / API key
- 密码重置 token
- 加密用的 IV / nonce / salt
- 任何 OAuth / OIDC 的 state、code_verifier

### 2.3 `/dev/random` vs `/dev/urandom`:多少博客都讲错

Linux 上有两个随机源:`/dev/random` 阻塞式(熵不够 hang 住)、`/dev/urandom` 非阻塞。**老博客都教** `/dev/random` 才"真随机"——**这是 90 年代的陈旧观念**。现代密码学家(Schneier、DJB、Linux 内核组)的统一立场:

> **永远用 `/dev/urandom`**——内核启动初期熵池注入完成后,它和 `/dev/random` 一样安全,而且**不会阻塞**。

理由:CSPRNG 一旦有 256 bit 真熵种子就**永远安全**,不需要"持续补熵";"熵被消耗光"是过时模型,Linux 4.8+ 已改;`/dev/random` 在容器/启动初期常 hang 几十秒,**直接导致应用启动失败**。

**Linux 3.17+ 还有 `getrandom(2)` syscall**——启动初期会阻塞直到熵足够,**之后永远不阻塞**,这是当下最现代的姿势,各语言 CSPRNG 底层基本都调它。

```
推荐顺序:
1. 语言 CSPRNG API(secrets / crypto / SecureRandom)
2. getrandom() syscall(C 代码)
3. /dev/urandom(直接读)
永远不要直接读 /dev/random,除非你在写内核或 HSM 驱动。
```

## 三、哈希函数:碰撞是什么意思

### 3.1 哈希的三个性质

```
1. 抗原像           给 h,找不到 x 使 H(x) = h           "哈希不能反推原文"
2. 抗第二原像       给 x1,找不到 x2 ≠ x1 使 H 相等       "给一个文件,找不到另一个同哈希"
3. 抗碰撞           找不到任何 x1 ≠ x2 使 H 相等          "任意两输入同哈希,不可能"
```

**注意**:抗碰撞 ≠ 抗第二原像。**MD5 抗第二原像还行,抗碰撞已经全完**——攻击者能"自己造一对碰撞文件",但不能"给一个文件反推一个碰撞"。

### 3.2 主流哈希函数现状

| 算法 | 输出 | 抗碰撞 | 是否能用 | 备注 |
| --- | --- | --- | --- | --- |
| MD5 | 128 bit | **已破**(秒级) | ❌ 任何安全场景 | 只能用作非安全场景的 checksum |
| SHA-1 | 160 bit | **已破**(2017 Google) | ❌ | Git 还在用,但已经在迁移 SHA-256 |
| SHA-256 | 256 bit | 安全 | ✅ 推荐 | SHA-2 家族,工业事实标准 |
| SHA-512 | 512 bit | 安全 | ✅ | 64 位机器上反而比 SHA-256 快 |
| SHA-3 | 224-512 bit | 安全 | ✅ | 海绵结构,设计完全不同于 SHA-2 |
| BLAKE2 / BLAKE3 | 任意 | 安全 | ✅ 推荐 | 比 SHA-2 快 3-5 倍,现代首选 |

**实战建议**:

- 新项目:**SHA-256 或 BLAKE3**,二选一
- 不要用 MD5、SHA-1
- 不要"套娃":`SHA256(MD5(x))` 不会更安全,只会更慢

### 3.3 "碰撞攻击"到底意味着什么

很多人以为"MD5 被破了 = 哈希值能被反推回原文"——**错**。破的是抗碰撞,意思是攻击者能**主动构造两个不同输入有相同 MD5**。

真实利用场景:**证书伪造**(2008 CA 攻击,两份不同内容证书同 MD5,签名挪用)、**文件捆绑**(同 MD5 的两份 PDF,一份无害骗审计、一份恶意实际部署)、**Git 提交伪造**(SHA-1 SHAttered 2017)。

所以 MD5 / SHA-1 还能不能用,**取决于谁能控制内容**:

- 文件去重 / 缓存 key:**能**(攻击者构造碰撞干嘛?)
- 防意外损坏:**能**
- 防恶意篡改 / 数字签名 / 证书:**❌ 绝对不能**

## 四、密码哈希:为什么 SHA256(password) 是错的

### 4.1 普通哈希 vs 密码哈希,完全两种东西

```
普通哈希(SHA-256 / BLAKE3):
   目标:快、稳定、抗碰撞
   速度:GPU 每秒 100 亿次

密码哈希(bcrypt / scrypt / Argon2):
   目标:慢、消耗内存、专门为「抗暴破」设计
   速度:故意调到每次 100ms
```

**为什么要慢**:用户密码熵太低。"password123"、"qwerty"、"12345678"——攻击者拿到一个被泄露的密码哈希库,**可以离线跑字典攻击**:

```
攻击者算力:GPU 每秒算 1000 亿次 SHA-256
密码字典:10 亿个常见密码 + 变种
   → 10 秒内试完

如果用了 bcrypt(每次 100ms):
   → 1000 亿次需要 3000 年
```

**这就是为什么不能 `SHA256(password + salt)`**——加盐能防"彩虹表"(预算好的哈希表),但防不住"针对单个账户的字典攻击"。**唯一的解法是让哈希本身变慢**。

### 4.2 三种主流密码哈希算法

| 算法 | 出生年 | 抗 GPU | 抗 ASIC | 推荐 |
| --- | --- | --- | --- | --- |
| bcrypt | 1999 | 中等(不消耗内存) | 弱 | 稳妥兜底 |
| scrypt | 2009 | 强(消耗内存) | 中 | OK |
| **Argon2id** | 2015 | 强 | 强 | ⭐ 现代首选 |
| PBKDF2 | 2000 | 弱(快,GPU 友好) | 弱 | 仅旧系统兼容 |

**Argon2 是 2015 年 Password Hashing Competition 冠军**——**同时消耗 CPU 和大量内存**,GPU 内存小、ASIC 又贵又难造,所以对所有"专用硬件加速"都不友好,这正是它的目标。

### 4.3 正确的密码存储

```python
# ❌ 错:全部都是错
db.execute("INSERT INTO users(pw)...", password)       # 明文
hashed = hashlib.md5(password.encode()).hexdigest()    # MD5(LinkedIn 2012 同款)
hashed = hashlib.sha256(password.encode()).hexdigest() # SHA-256(普通哈希)
hashed = hashlib.sha256((salt + password).encode())    # 加盐 SHA-256(GPU 暴破照样秒)

# ✅ 对:Argon2id
from argon2 import PasswordHasher
ph = PasswordHasher()                          # 默认参数已合理
hashed = ph.hash(password)
# 输出:$argon2id$v=19$m=65536,t=3,p=4$<salt>$<hash>
# salt 库自动生成、参数也编码进去了

try:
    ph.verify(hashed, user_input)
except VerifyMismatchError:
    raise AuthError()
```

```go
// Go:golang.org/x/crypto/argon2
salt := make([]byte, 16); rand.Read(salt)
hash := argon2.IDKey([]byte(password), salt, 3, 64*1024, 4, 32)
// time=3, memory=64MB, threads=4, output=32
```

```javascript
// Node.js:argon2 npm 包
const hash = await argon2.hash(password, { type: argon2.argon2id });
const ok   = await argon2.verify(hash, userInput);
```

**Argon2id 参数起点(2025)**:`memory=64MB, iterations=3, parallelism=4, output=32B`,调到「单次验证 ~100-250ms」为止。

### 4.4 实战 cheatsheet

| 场景 | ❌ 错误 | ✅ 正确 |
| --- | --- | --- |
| 存用户密码 | `md5(pw)` / `sha256(pw)` | `argon2id(pw)` |
| 存 API token | 明文 | `sha256(token)` 存 + 明文给用户一次 |
| 文件指纹 | `md5(file)` | `sha256(file)` / `blake3(file)` |
| 缓存 key | `md5(url)` | OK,非安全场景 |
| 验证完整性 | `sha1(pkg)` | `sha256(pkg)` + 签名 |

**为什么 API token 用普通 SHA-256 就行**:token 是高熵随机串(`secrets.token_urlsafe(32)`),**熵足够高,字典攻击没用**——不像用户密码"qwerty"。

## 五、KDF:从一个东西派生另一个东西

### 5.1 什么是 KDF

**Key Derivation Function**——派生密钥的函数,解决两类不同问题:

```
1. Password-based KDF(PBKDF):「低熵密码」→「高熵密钥」
   - 输入:用户密码 + salt    必须慢(防字典攻击)
   - 代表:PBKDF2 / bcrypt-kdf / scrypt / Argon2

2. Key-based KDF(KBKDF):「一个高熵密钥」→「多个独立子密钥」
   - 输入:root key + 用途标签   必须快(已经是高熵)
   - 代表:HKDF
```

**新手最大的混淆**:把 HKDF 当 PBKDF 用、把 PBKDF2 当 HKDF 用——**完全不同的场景,不能换**。

### 5.2 PBKDF2(密码 → 密钥)

```python
# 场景:用户输入"correct horse battery staple",派生 AES-256 密钥
key = hashlib.pbkdf2_hmac('sha256', password.encode(), salt,
                         600_000, dklen=32)   # OWASP 2025 推荐 ≥ 600k
```

**PBKDF2 是密码 KDF 里最弱**(对 GPU/ASIC 友好),新项目用 Argon2,**只在兼容旧系统 / FIPS 合规时用 PBKDF2**。

### 5.3 HKDF(密钥 → 多个子密钥)

```python
# 场景:TLS 握手后有 master_secret,要派生 client/server write key 等
hkdf = HKDF(algorithm=hashes.SHA256(), length=32, salt=None,
            info=b'tls13 client write key')   # info 区分不同子密钥
key1 = hkdf.derive(master_secret)
```

**HKDF 两步**:`extract`(HMAC(salt, input) → PRK)+ `expand`(PRK + info → 任意长度输出)。

**`info` 是关键**——同一 master_secret 派生客户端 / 服务端密钥**必须不同**,靠传不同 `info` 实现"上下文绑定"。TLS 1.3、Signal、Noise 全靠 HKDF + info。

| 场景 | 用谁 |
| --- | --- |
| 用户密码 → 主密钥 | Argon2 / scrypt / PBKDF2 |
| 主密钥 → 多个子密钥 | HKDF |
| ECDH shared secret → AES 密钥 | **必须** HKDF,不能直接用 |

**最常见坑**:ECDH 算完直接拿 shared secret 当 AES 密钥——shared secret 是椭圆曲线点 x 坐标,**分布不均匀**,必须过 HKDF。

## 六、MAC vs Hash vs Signature:一张表讲清

| | Hash | MAC (HMAC) | 签名 (Signature) |
| --- | --- | --- | --- |
| 用密钥? | ❌ 不用 | ✅ 对称密钥 | ✅ 非对称(私钥签,公钥验) |
| 谁能验证 | 任何人 | 持密钥的人 | 持公钥的人(所有人) |
| 谁能生成 | 任何人 | 持密钥的人 | 持私钥的人 |
| 不可否认 | ❌ | ❌(双方都能签) | ✅ 只有私钥持有者能签 |
| 算法 | SHA-256 / BLAKE3 | HMAC-SHA256 / Poly1305 | Ed25519 / ECDSA / RSA-PSS |
| 性能 | 最快 | 快(2× hash) | 慢(ms 级) |

### 6.1 用错场景

```
场景:服务 A 调服务 B,要确保请求没被篡改

❌ 用 Hash:任何人都能算 hash,起不到防篡改作用
✅ 用 HMAC:A 和 B 共享密钥,中间人不能伪造 HMAC
   但 A 也能伪造 B 的发送 —— 内部信任场景 OK

场景:用户从 GitHub 下载二进制,确保是官方的

❌ 用 Hash:hash 可以被一起篡改
❌ 用 HMAC:用户没有 GitHub 的密钥
✅ 用签名:GitHub 用私钥签,用户用公钥验
```

### 6.2 HMAC 的工程坑

```python
# ❌ 错:== 比较会被时序攻击(字符串比较短路,差 1 字节就返回)
if mac == received_mac: ...

# ✅ 对:恒定时间比较
import hmac
if hmac.compare_digest(mac, received_mac): ...
```

```go
// Go:crypto/hmac.Equal 是恒定时间
if hmac.Equal(mac1, mac2) { ... }
```

**这个坑非常常见**——大量旧 JWT 实现、API 签名校验都中过招。

## 七、真实事故:LinkedIn / Adobe 教会我们什么

### 7.1 LinkedIn 2012:6.5 亿密码 SHA-1 无盐

```
存储:SHA-1(password),无盐 → 几小时内 90% 密码被破
教训:无盐 → 彩虹表;SHA-1 太快 → GPU 暴破;无迭代 → 防御为零
```

事件之后全行业开始普及 bcrypt。

### 7.2 Adobe 2013:1.5 亿密码 + 提示串

```
存储:3DES-ECB 加密密码(不是哈希!)+ 明文密码提示串
后果:ECB 相同密码 → 相同密文,频率最高的几乎一定是 "123456"
   + 提示串 "my dog's name" 直接帮攻击者破解
```

这事故被反复嘲笑——**不是参数错,是整个设计哲学错**:密码应该**单向哈希**,任何"能解密"的存储方式都是错的。

### 7.3 RockYou 2009:3200 万明文

直到今天,`rockyou.txt` 还是密码字典的祖先,所有渗透测试标配。

### 7.4 现代正确实践(2025 基线)

```
✅ Argon2id(memory=64MB, time=3, parallel=4)
✅ 每个密码独立 salt(库自动)
✅ 全局 pepper(存配置 / KMS,加在 hash 前)
✅ 失败次数限制(防在线暴破)
✅ NIST 2017+ 密码策略:长度 ≥ 8,禁用常见密码,不强制复杂字符
✅ 支持 2FA / Passkey
✅ Have I Been Pwned API:注册时检查是否在已知泄露库
```

## 八、工程师 cheatsheet:错误写法 vs 正确写法

```
[ 随机数 ]
❌ random.randint() / Math.random()
✅ secrets / crypto.randomBytes / SecureRandom

[ 普通哈希 ]
❌ MD5 / SHA-1
✅ SHA-256 / SHA-512 / BLAKE3

[ 密码哈希 ]
❌ md5(pw) / sha256(pw+salt) / 任何普通哈希
✅ Argon2id(首选) / scrypt / bcrypt / PBKDF2(兼容旧系统)

[ 密钥派生 ]
❌ sha256(password) 当 AES 密钥;ECDH 共享秘密直接当密钥
✅ 密码 → Argon2 / PBKDF2;共享秘密 / 已有密钥 → HKDF

[ MAC ]
❌ md5(key+msg) / sha256(key+msg)(长度扩展攻击);== 比较 MAC
✅ HMAC-SHA256 / Poly1305;hmac.compare_digest / hmac.Equal

[ 加密 ](下一篇细讲)
❌ AES-ECB;AES-CBC 无 MAC;同一个 nonce 复用
✅ AES-GCM / ChaCha20-Poly1305(AEAD);每次新 nonce

[ 比较敏感数据 ]
❌ secret1 == secret2
✅ constant_time_compare / hmac.compare_digest
```

## 九、踩坑提醒(密码学心智版)

1. **以为"自己加点东西更安全"**——SHA-256 套两层、AES 算两遍、自创盐法——**几乎一定弱化安全性**,标准算法本身已是「最优」
2. **以为 random 是 random**——`Math.random` / `random` 模块**绝对不能用作密码学**,看到就该警觉
3. **以为加盐就够了**——盐防彩虹表,**不防字典攻击**;必须慢哈希(Argon2)
4. **以为 SHA-256 越多越安全**——`SHA256^N(pw)` 完全不抗 GPU 暴破;关键不是次数,是"消耗内存"
5. **以为哈希 = 加密**——哈希单向不可逆,加密能解密,**根本不是一个东西**
6. **以为 HMAC 和 Hash 差不多**——没密钥的 hash 任何人都能算,起不到认证;必须 HMAC
7. **以为 == 比较 MAC 是 OK 的**——字符串比较短路,时序泄露
8. **以为 KDF 用哪个都行**——PBKDF2 给密码、HKDF 给已有密钥,**不能换**
9. **以为存了 hash 就安全了**——MD5 也是"hash";具体算什么 hash 决定一切
10. **以为加密就不需要随机数**——AES-GCM 的 nonce 复用 = 密钥泄露;**密码学到处需要 CSPRNG**

---

## 十、心智总结

```
随机数:CSPRNG,不是 random 不是 Math.random
哈希:  SHA-256 / BLAKE3,不要 MD5 / SHA-1
密码:  Argon2id,不要 SHA(pw),不要 SHA(pw+salt)
KDF:   密码用 Argon2,密钥用 HKDF
MAC:   HMAC + 恒定时间比较
签名:  Ed25519(首选) / ECDSA / RSA-PSS

五条工程纪律:
   1. 永远用标准库的高层 API,不要碰 low-level 原语
   2. 永远用 CSPRNG 生成密钥 / token / nonce / salt
   3. 永远用恒定时间比较敏感数据
   4. 永远查清楚每个参数的"2025 推荐值"——密码学参数会过期
   5. 永远不要"自创算法",也不要"组合标准算法成新协议"

如果你只能记住一句话:「用标准库的高层 API,不要自己拼。」
```
