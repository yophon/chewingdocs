    内网渗透:横向移动 / 提权 / 域渗透 / C2 流量

    写到这一篇,前两篇刚把"外网怎么打进来"讲完——OSINT 摸清边界、Web 渗透方法论拿下一个 Web Shell——但真正让攻击者拿到价值的不是这第一台机器,而是从这台机器出发的接下来 30 步。绝大多数新人对"渗透"的想象停在「拿到 shell 就赢了」,真实情况是:拿到 shell 只是开始,攻击者真正要的是域控、是数据库、是邮箱、是源代码、是 CI/CD 的密钥——而这些几乎都不在第一台机器上。这一篇讲清楚攻击者拿到第一个立足点之后,在内网里怎么想、怎么走、怎么藏,但所有内容只到心智层——不教任何具体武器化命令,目的是让 blue team 知道防御要堵哪里。

    一句话先记住:外网攻击是"撞门",内网渗透是"在你家里走"——外网每个动作都很吵(扫描、爆破、注入),内网每个动作都尽量"看起来像合法操作"(用合法凭证、走合法协议、调合法 API)。这就是为什么内网防御不能靠 WAF 和防火墙——攻击者已经在边界内,用着合法账号——只能靠"行为基线"和"最小权限",看不到这层的企业,等于在内网裸奔。99% 的"重大数据泄露",边界都没被打穿,被打穿的是内网。

    一、为什么"打进来"只是开始

    很多工程师以为"边界守好就万事大吉"——防火墙 / WAF / IDS 围一圈就安全了。这个心智在 2000 年还成立,现在彻底失效:边界一定会破(钓鱼邮件 / 1day 没补 / 供应商被打穿,总有一台机器会沦陷);内网信任过度(数据库不要密码、运维网段全互通、SSH 密钥不轮转);横向走一次就赢(拿到一台开发机,从浏览器拿到 GitLab Token / AWS Key,顺着摸到 CI/CD,整个仓库和生产环境一起沦陷)。Mandiant M-Trends 2024 的数字:攻击者从「初始访问」到「拿到域控」的中位数是 不到 24 小时——应急响应不在一天内启动,基本就是"事后审计"。

    1.1 内网渗透的本质:把"一台机器"变成"整个域"

    第一台机器(Web Shell / 钓鱼木马 / 1day)
    ↓ 立足 → 侦察 → 提权 → 凭证窃取
    ↓ 横向(用偷来的凭证登录第二台)
    ↓  ←──── 循环,直到拿到目标
最终目标(域控 / 数据库 / 邮箱 / CI/CD)
    ↓ 数据外带 → 清理痕迹

    每一步都不是"黑客技术"——多数情况下,是用合法工具做合法操作,只是顺序和目标不合法。这就是检测的最大难点:你拦不掉 psexec、wmic、ssh、net use——这些都是运维每天用的工具。

    1.2 攻击者想的事和你想的不一样

    你以为攻击者想 攻击者实际想
    "我要黑掉这家公司" "我要拿到域控,然后从域控分发勒索软件"
    "我要找最新的 0day" "我要找最旧没补的 1day,因为没人盯"
    "我要写最厉害的 exploit" "我要用最朴素的 PowerShell + WMI,不触发 EDR"
    "我要快速拿下" "我要潜伏 200 天,确认有备份再发动勒索"

    真正的 APT 不追求"炫技",追求"不被发现"——他们用的工具往往简单到惊人,全是合法管理员工具。这种打法有个名字:Living Off the Land(LOTL,就地取材)。

    二、MITRE ATT&CK:内网攻击的"通用语言"

    讲内网渗透绕不开 ATT&CK——它把攻击者所有动作分成 14 个战术阶段,每个阶段下是具体技术。蓝队和红队都靠这套框架对齐认知:红队说"我演练 T1003 LSASS dumping",蓝队就知道该看哪些日志、哪些 EDR 规则。

    1.  Reconnaissance       侦察          (打进来之前,扫外网 / OSINT)
2.  Resource Development 准备资源
3.  Initial Access       初始访问       ★ 边界破了
4.  Execution            执行
5.  Persistence          持久化         (重启还在)
6.  Privilege Escalation 提权           (普通用户 → root)
7.  Defense Evasion      防御绕过
8.  Credential Access    凭证窃取       (LSASS / SSH key)
9.  Discovery            发现           (内网摸清楚)
10. Lateral Movement     横向移动       ★ 走到下一台
11. Collection           数据收集
12. Command & Control    C2 通信        ★ 远控通道
13. Exfiltration         数据外带
14. Impact               影响           (勒索 / 破坏 / 篡改)

    3-12 阶段全是"内网渗透"的范围——ATT&CK 80% 内容都在讲内网怎么走。它的价值:把"我们安全吗"这个无法回答的问题,变成"我们对哪些技术有检测覆盖"这个能打分的问题。没 ATT&CK 之前蓝队对话"我们装了杀软挺安全的吧",有了之后是"T1003 我们覆盖了吗?T1059.001 有规则吗?覆盖率 65%,缺口在持久化和 C2"。这是 Purple Team 的基础(24 篇细讲)。

    三、立足与持久化:重启之后还在

    攻击者拿到第一个 shell 后,第一件做的事不是提权,是持久化——他们最怕"机器被运维重启 / 进程被杀,辛苦建立的入口没了"。

    3.1 Linux 上"重启还在"的常见姿势

    crontab / systemd timer       定时唤起
/etc/rc.local / systemd unit  开机启动
~/.bashrc / ~/.profile        用户登录就跑
SSH authorized_keys           加自己的公钥
PAM 模块替换                  改认证模块,任何密码都能登
内核 rootkit                  内核态隐藏进程 / 文件

    防御侧监控(不是"禁止",这些命令运维都要用):/etc/cron.* 和 /var/spool/cron/ 修改告警;systemctl 新建 service 审计;authorized_keys 变更告警;/etc/pam.d/ 任何变更告警(PAM 改了基本就是后门);/lib/modules 下加载未知 ko 告警(rootkit 入口)。

    3.2 Windows 上"重启还在"的常见姿势

    Run / RunOnce 注册表项        登录自动跑
计划任务(schtasks)
服务(services.msc 加项)
WMI 事件订阅                  事件触发就跑,隐蔽性极强
DLL 劫持                      放假 DLL 在系统目录
Image File Execution Options  劫持任意 .exe 的启动
Logon Script
UEFI / Bootkit                硬件级,重装系统都干不掉

    WMI 事件订阅特别要关注——它不在普通"启动项"清单里,很多运维根本不知道有这东西,APT 最爱(Stuxnet 就用了)。

    对蓝队的好消息:持久化必然在磁盘 / 注册表 / 配置文件留痕——没有持久化,攻击者重启就完蛋;有了持久化,就有可被扫到的 IoC(Indicator of Compromise)。EDR / Sysmon / auditd 主要靠这个找入侵——攻击者最薄弱的环节,也是蓝队最强的环节。

    四、信息收集:在内网里"找路"

    打外网时信息收集靠 Google + scan,打内网时信息收集靠"在机器内部翻东西"。

    4.1 攻击者在第一台机器上"翻"什么

    网络拓扑: - 看 /etc/hosts、route、ip route、ARP 表 - 看历史 SSH 连接(~/.bash_history、~/.ssh/known_hosts) - 内网 DNS 反向查询整段 - 抓本机网络流量看有谁在和这台机器说话

    凭证: - 浏览器保存的密码 / cookie / 自动填充 - SSH 私钥(~/.ssh/id_*)和它能登的所有机器 - ~/.aws/credentials、~/.docker/config.json、~/.kube/config - Git 仓库里 hard-code 的 token(.env 文件、配置文件) - 进程命令行参数(ps auxf 经常能看到密码作为参数) - 历史命令行(.bash_history 有人 mysql -u root -pxxx)

    服务发现: - 内网端口扫描(慢、隐蔽地扫,不会用 nmap 那种吼一嗓子) - LDAP 查询(域内能拿到所有用户和机器列表) - mDNS / NetBIOS 广播 - 看本机 netstat 知道这台机器连了哪些"重要服务"

    4.2 防御侧要监控什么

    异常的 LDAP 查询——一台普通办公电脑,不应该有"列出所有域用户"这种请求。攻击者一上来就喜欢 dump 整个域(用 BloodHound 这类工具),触发的 LDAP 查询模式跟人类操作完全不同。

    异常的端口扫描——内部机器之间有"正常通信模式",突然一台机器连大量内网 IP 的同一个端口(445 / 22 / 3389)→ 大概率横向探测。

    密钥文件被异常读取——可以用 auditd / Sysmon 监控 ~/.ssh/、%APPDATA%\Mozilla\Firefox\Profiles\*\logins.json 这类文件的读取,正常用户不会"突然 cat 浏览器密码库"。

    五、提权的心智(不讲具体方法)

    拿到第一个 shell 通常是低权限的(www-data / IIS 之类)。攻击者要从普通用户 → root / SYSTEM,才有真正大规模操作的能力。

    5.1 Linux 提权的几个心智方向

    • SUID 错配:某二进制设了 SUID(以 root 跑),但内部允许执行命令或读任意文件。任何 SUID 二进制都是攻击面,要审计每一个;定期扫 find / -perm -4000 与白名单比对
    • sudo 错配:sudoers 给了"无密码 sudo 某命令",但这个命令能逃逸(sudo vi 进去 :!sh 就是 root)。给 sudo 要给到具体不可逃逸的命令,不要给 vim / less / find / awk / 任何脚本解释器。GTFOBins 这个网站列举了哪些命令能逃出来——蓝队也要看,知道自己 sudoers 给了什么风险
    • 内核漏洞:Dirty Pipe / Dirty COW / OverlayFS 可直接从普通用户提到 root。未及时打补丁的内核就是开放门,uname -r 对比 CVE 数据库,过期了告警
    • capabilities 错配:Linux 把 root 拆成 38 个 capability,CAP_SYS_ADMIN 之类高权就是半个 root。getcap -r / 2>/dev/null 看有哪些可疑 capability

    5.2 Windows 提权的几个心智方向

    • UAC bypass:UAC 让管理员账户大多数时候以普通权限运行,需要提权弹窗。Bypass 不算严格意义"提权"(本身已是管理员),但 EDR 视为提权。靠 COM 接口 / 自动提升白名单 / 注册表劫持绕弹窗。防御:UAC 等级调到最高
    • SeImpersonate 滥用:Windows 服务账户(IIS / SQL Server)默认带 SeImpersonatePrivilege,可"伪装"成请求来的客户端身份——配合"骗一个高权限服务连进来"(Potato 系列),就能拿 SYSTEM。防御:服务尽量用 Managed Service Account
    • 令牌窃取:拿到 SYSTEM 后"偷"其他登录用户的 token,伪装成域管理员,这是横向到域控的常见跳板

    5.3 防御侧:提权的"信号"

    多数提权都会产生异常进程链:

    正常:explorer.exe → cmd.exe(用户主动开)
异常:w3wp.exe(IIS) → cmd.exe → whoami           ← 严重告警
异常:sqlservr.exe → powershell.exe -enc <base64> ← 严重告警
异常:Office 应用 → cmd.exe / powershell.exe       ← 钓鱼宏的典型

    Sysmon Event ID 1(进程创建) + 父子进程关系 是检测提权和横向的金矿——任何"不该派生 shell 的服务派生了 shell",都值得告警。EDR 的基础规则就是这套。

    六、凭证窃取:内网渗透的核心货币

    横向移动 99% 靠偷来的凭证而不是"再打一遍漏洞"——因为凭证用起来像合法操作,扫描和爆破像攻击。

    凭证有三种存在形式:明文密码(用户输入 / 配置文件 hard-code / 浏览器明文保存)、哈希(Windows NTLM 哈希、Linux /etc/shadow)、票据 / 令牌(Kerberos 票据、OAuth Token、Session Cookie、JWT)。心智:凭证不一定要"破解",只要"能用"就行——这是 Pass-the-Hash / Pass-the-Ticket 的核心思想。

    6.1 Windows 凭证窃取(只讲心智)

    LSASS(Local Security Authority Subsystem Service) 是 Windows 管认证的进程,内存里缓存了所有当前登录用户的凭证(NTLM 哈希、Kerberos 票据,有时甚至有明文)。攻击者拿到本地管理员,dump LSASS 内存就能拿到登录过这台机器的所有用户凭证——包括域管理员路过留下的。

    Mimikatz(2014 年开源)是干这事最出名的工具,也是 Windows 内网渗透的"教父级"工具。它本身是合法的安全研究工具,但被滥用得太厉害,所有 EDR 都在重点盯。

    防御: - 开启 Credential Guard(Windows 10+):用 VBS 虚拟化隔离 LSASS 凭证,从内核态都摸不到 - LSASS 保护模式(RunAsPPL):限制能 attach 它的进程 - 限制域管理员登录范围:绝对不要让域管登录普通办公电脑或 Web 服务器——一旦那机器沦陷,LSASS 里的域管哈希就被拿到了。这是"分层管理(Tiered Admin Model)"的核心 - EDR 监控对 LSASS 的"读内存"操作(OpenProcess with PROCESS_VM_READ)→ 几乎都是异常

    6.2 Linux & 浏览器凭证

    Linux 侧:/etc/shadow(哈希,拿到能离线爆破)、~/.ssh/id_*(私钥,大多没设密码,等于通行证)、~/.aws/credentials / ~/.kube/config / ~/.docker/config.json、内存里的 ssh-agent 密钥、环境变量里的 token(printenv 一眼)。

    浏览器是凭证富矿——保存的密码 / cookie / 自动填充 / OAuth Token 全在用户家目录,用本机 DPAPI 密钥加密,拿到这个用户的权限就能解密。防御:公司机器禁用浏览器保存密码,统一 SSO + MFA;关键账户每次走 MFA;EDR 监控对 Login Data / Cookies 的异常读取。

    七、横向移动:从一台到下一台

    拿到凭证之后,怎么用这凭证登第二台?这就是横向移动。

    7.1 三个核心心智:不一定要密码

    Pass-the-Hash(PtH):Windows NTLM 协议有个设计缺陷——认证时只比对哈希,不需要原始密码。如果你拿到了某用户的 NTLM 哈希,就可以"伪装"成他登录任何接受 NTLM 的服务(SMB / WMI / WinRM),完全不需要破解出明文密码。 - 出现的根本原因:NTLM 是 90 年代的设计,哈希就是凭证本身 - 解决:禁用 NTLM,全用 Kerberos;但很多老系统迁不动

    Pass-the-Ticket(PtT):Kerberos 协议也有类似问题——拿到一个有效的 TGT(Ticket Granting Ticket)或 ST(Service Ticket),可以直接注入到当前会话,免认证访问目标服务。 - 不同于 PtH,Kerberos 票据有时效(默认 10 小时),但已经够横向了 - 防御:监控异常票据使用(同一个 TGT 突然从不同 IP 出现)

    SSH 密钥滥用:开发机和服务器之间 SSH 互通,运维一台拿一把私钥,这把私钥能登 100 台机器。一旦运维机器沦陷,整个 fleet 全沦陷。 - 防御:SSH 密钥不要"一把通天下",按角色 / 环境分密钥;用 SSH CA 签证书,证书带过期时间 - 进阶:把 SSH 入口都收到一个 bastion(跳板机)上,生产机不直接接受密钥认证,只接受跳板转发的会话

    7.2 横向用的协议与端口

    SMB         445       Windows 文件共享 + 远程执行(PsExec / WMI)
WinRM       5985/5986 Windows 远程管理(PowerShell Remoting)
RDP         3389      远程桌面
WMI         135 + 动态  Windows 管理规范(隐蔽性强)
SSH         22        Linux 远程登录
RPC         135 + 动态  各种远程调用
LDAP        389/636   域信息查询

    防御侧最关键的两件事:

    1. 网络分段:办公网、开发网、生产网、域控网严格分开。默认禁止跨段通信,只开必需端口。没有分段的内网,一台沦陷 = 全沦陷
    2. 横向协议监控:SMB / WMI / WinRM 的"非常规来源"是横向的最强信号——一台财务 PC 不该向另一台财务 PC 发 PsExec

    八、Active Directory 域渗透心智(只讲概念)

    90% 的中大型企业用 AD(Active Directory)管 Windows 域——搞定 AD = 搞定整个公司。AD 攻击是内网渗透的"皇冠",也是这一篇最危险的部分,所以只讲概念。

    8.1 AD 的核心信任模型

    域控(DC)
  │
  ├── 用户账户和密码哈希都存在这里(NTDS.dit 数据库)
  ├── Kerberos KDC 在这里(发票据)
  └── LDAP 树(组织结构、ACL)在这里
   ↓ 信任
所有域成员机器(笔记本 / 服务器)

    核心心智:DC 是单点信任根——拿到 DC,等于拿到所有用户的所有哈希、所有票据签发权限。DC 沦陷 = 域沦陷 = 公司沦陷。

    8.2 Kerberoasting

    心智:Kerberos 中,任何域用户都可以申请"某个服务"的票据(ST),这个票据是用服务账户的密码哈希加密的。攻击者申请一堆票据,离线爆破弱密码的服务账户——服务账户密码通常是 SQL Server / Exchange 之类的,长期不改,而且是管理员密码。

    防御: - 服务账户密码要长(25 字符+)而且随机,这样离线爆破不出来 - 用 Managed Service Account(gMSA),密码 AD 自动管,人都拿不到 - 监控异常的"批量 TGS 请求"

    8.3 Golden Ticket

    心智:Kerberos 中所有 TGT 都用 krbtgt 账户的哈希签发。如果攻击者拿到 krbtgt 的哈希(意味着已经搞定了 DC),就可以伪造任意用户的 TGT——包括根本不存在的用户、伪造的超级管理员。这个伪造的票据叫 Golden Ticket,只要 krbtgt 哈希不变,就能用 10 年。

    防御: - 定期(2 次)重置 krbtgt 账户密码——这是 AD 运维的硬性安全要求 - 监控异常的 TGT(签发时间和实际时间不一致、字段异常) - 一旦怀疑 DC 沦陷,必须重置 krbtgt 两次(Kerberos 设计上保留前一个密码可用)

    8.4 DCSync

    心智:AD 多 DC 之间用一个叫 DRSUAPI 的协议同步数据,这个协议允许"请求复制账户哈希"。任何拥有"复制目录变更"权限的账户(默认是 Domain Admins 和 DC 本身)都能调这个 API,远程"问 DC 要任意账户的哈希"——包括 krbtgt。

    也就是说:攻击者不需要真的登上 DC,只要有合适的权限,就能从远程把整个域的哈希"同步"出来——这就是从域管 → Golden Ticket 的常用跳板。

    防御: - 严格控制"复制目录变更"权限——只有 DC 应该有,任何用户/服务账户拿到这个权限都是异常 - 监控 DRSUAPI 调用,任何来自非 DC 的复制请求都告警

    8.5 BloodHound:攻防都在用的工具

    心智:AD 里的 ACL 关系极其复杂(谁能登谁、谁能改谁、谁继承了什么),人工看根本看不出"从 A 到域管"的路径。BloodHound 用图数据库把 AD 关系画出来,红队用来找最短攻击路径,蓝队用来审计自己的 ACL 错配。

    这是少数"攻防对称"的工具——蓝队不用 BloodHound,你根本不知道自己 AD 多脆弱。

    九、C2 流量:命令通道的猫鼠游戏

    攻击者在内网里跑,总要有个"远程控制通道"——把自己的指令送进来,把窃取的数据送出去。这就是 C2(Command & Control)。C2 是所有 EDR / IDS / 流量分析的重点战场。

    9.1 C2 的演化:藏在合法流量里

    2000 年代:    自定义协议、固定端口      → 一抓一个准
2010 年代:    HTTP/HTTPS 伪装合法 Web    → 流量加密了,但能从行为分析
2015+ :       域前置(Domain Fronting)  → 借合法 CDN 隐藏真实 C2
2018+ :       DNS / DoH(DNS over HTTPS)→ 用 DNS 当通道,出口墙也拦不掉
2020+ :       合法 SaaS 当 C2            → 借 GitHub / Slack / Dropbox / Telegram

    心智:好的 C2 流量,看起来要像 100% 合法的业务流量——和 Google / Cloudflare / 公司 OA 一模一样。这是为什么传统签名/特征检测对现代 C2 几乎无效。

    9.2 域前置(Domain Fronting)

    攻击者的真实 C2:    evil-c2.attacker.com  (托管在某 CDN 上)
看起来访问的:       www.microsoft.com     (TLS SNI / 流量目的)
实际 HTTP Host 头:  evil-c2.attacker.com  (CDN 根据 Host 转发)

    为什么有效:防火墙看到的是"访问 Microsoft 的 IP",根本拦不掉(总不能拦 Microsoft)。HTTP Host 头在 TLS 加密内层,IDS 看不到。

    防御: - 主流 CDN(Google / AWS / Azure)2018 年后陆续禁用了 Domain Fronting,但仍有小厂可滥用 - 出口流量做 TLS 中间人解密 + Host 头审计(合规问题大,要慎用) - 零信任出口策略:只允许业务需要的域名访问,其他全拦——比 Domain Fronting 检测更彻底

    9.3 DNS / DoH C2

    心智:DNS 是几乎不会被拦的协议(没了 DNS 业务全废)。攻击者把指令编码到 DNS 查询里,外发数据编码到子域名里:

    查询: aGVsbG8gd29ybGQ.attacker.com    (子域名是 base64 数据)
回包: TXT 记录里塞返回数据

    DoH(DNS over HTTPS) 让这事更难——DNS 查询走 HTTPS 到 8.8.8.8 / 1.1.1.1,企业 DNS 服务器都看不到。

    防御: - 强制走企业 DNS,禁止终端直连 8.8.8.8 / 1.1.1.1 - DNS 流量分析:正常 DNS 查询的子域名长度、字符分布、查询频率有基线,大量长子域名或高熵字符串 → 大概率隧道 - 阻断已知 DoH 服务器 IP

    9.4 行为基线:C2 检测的终极武器

    无论 C2 怎么伪装,它都有"行为指纹":

    • 心跳间隔:大多 C2 是固定间隔回连(beaconing),正常浏览器不会"每 60 秒访问同一个 URL"
    • Jitter(抖动):好的 C2 会加随机抖动模拟人,但抖动也有分布特征
    • 流量比例:正常 HTTP 是"请求小、回包大"(下载内容),C2 通常"请求大、回包小"(命令短、数据回传)
    • JA3 / JA3S TLS 指纹:不同 TLS 库的握手特征不同,Cobalt Strike 默认配置的 JA3 是已知的
    • User-Agent / 时间分布:夜里 3 点公司没人但有流量,基本不是好事

    这就是为什么"NDR(Network Detection and Response)"成为新一代防御核心——它不看流量内容,看流量模式。

    十、Cobalt Strike / Mimikatz:绕不开的"行业术语"

    任何讲内网渗透绕不开这两个名字。这里只讲它们"为什么重要",不讲怎么用。

    Cobalt Strike(CS):2012 年发布的商业渗透测试平台。提供"团队服务器 + Beacon(植入端)"的完整 C2 框架——支持 HTTP / HTTPS / DNS / SMB 通道、内置后渗透模块、有 GUI、能多人协作。90% 真实勒索软件事件都能看到 CS 的影子——因为它被破解("裸奔版")在地下流通,门槛极低;Beacon 通信默认很"像样",防御方需要训练才能识别。蓝队要懂的:CS Beacon 旧版默认配置有明显特征(JA3 指纹、特定 Cookie 名),jump / runas / mimikatz 模块的进程链有典型特征;不要试图"封 IP"——team server IP 在变,要靠行为检测。

    Mimikatz:法国研究员 gentilkiwi 2007 年开发,原本用来"展示 Windows 凭证存储设计缺陷"——成了横扫世界的攻击神器。能 dump LSASS、操作 LSA 注入凭证、DCSync、Pass-the-Hash / Pass-the-Ticket / Golden Ticket / Silver Ticket。它是 EDR 的"试金石"——任何 EDR 必须能检测原版 Mimikatz。但攻击者会用 C# 重写 / .NET 反射加载 / fileless 内存执行,简单签名拦不住。本质防御:Credential Guard + LSASS 保护 + 分层管理(域管不下沉)。

    十一、数据外带:最后一公里也要藏

    数据偷出来之前不算成功——但外带恰恰是最容易被流量分析抓到的环节。10 GB 的数据库 dump 发出去,无论怎么藏都难看不出异常。

    11.1 常见外带通道

    1. 直接 HTTPS 上传到 attacker.com         (最朴素,但容易被 DLP 拦)
2. 借合法云盘(Dropbox / OneDrive)        (流量看着合法)
3. DNS 隧道                                (慢,但能穿越严防火墙)
4. ICMP 隧道                              (老套但有人用)
5. 分段慢漏(每天偷一点,慢到看不出)
6. 借合法服务(Telegram / Discord / Gist) (新趋势)

    11.2 防御:DLP + 流量基线

    • 数据分级:核心数据库 / 源码 / 客户数据加水印 / Canary Token,一旦泄露能溯源
    • DLP(Data Loss Prevention):出口流量做内容审计,识别"信用卡号 / 身份证号 / 关键字段"
    • 流量异常基线:一台办公电脑日常上传 < 100MB,突然上传 5GB → 告警
    • 出口白名单:只允许必要的域名/服务,默认禁止所有未审批的云存储

    十二、防御视角:Blue Team 怎么堵

    讲了这么多攻击心智,这一节才是这一篇最值钱的部分。

    12.1 三层纵深

    1. 不让攻击者进来(Initial Access)
   - 钓鱼防护 / 邮件网关
   - 边界 WAF / 漏洞扫描 / 补丁
   - MFA(MFA 能挡 99% 的凭证爆破和钓鱼)
   - 终端杀软 / EDR

2. 进来了让攻击者动不了(Lateral)
   - 网络分段
   - 最小权限(尤其域管不下沉)
   - 服务账户用 gMSA
   - EDR + Sysmon + 日志集中
   - 蜜罐(Canary Token / Honey Account)

3. 动了能马上发现 + 干掉(Detection & Response)
   - SOC / SIEM / 7x24 监控
   - ATT&CK 覆盖率打分
   - 应急响应预案 + 桌面演练
   - 备份隔离(防勒索)

    心智:单层防御一定会破——别指望"边界 + 杀软"挡一切。纵深 + 快速发现 + 快速响应 才是现代防御的核心。

    12.2 EDR vs 杀软的区别

    传统杀软:    比文件签名,病毒库扫文件
EDR:        看进程行为,记录所有事件,可回查 + 主动追溯

    杀软只能挡"已知坏文件"——LOTL(就地取材)的攻击者根本不下载文件,只用合法工具。EDR 看的是"行为模式异常"——Word → cmd → powershell → encoded command 这种链路,无论用什么文件都告警。

    12.3 蜜罐 & 最小权限的硬纪律

    蜜罐:让攻击者主动暴露——在 AD 里建一个"看起来很重要"的账户(backup_admin 这种名字),没人会用它,但攻击者扫到一定会试登录 → 一登录就告警。Canary Token 在敏感目录放假文件(passwords.docx、aws-keys.txt),被打开/读取就触发回调。内网可放假的 SQL / SSH 服务,有人扫一连就报警。蜜罐的价值:误报率极低——正常业务不会碰这些,一旦触发 99% 是真攻击。

    最小权限:域管账户只能从特定"管理工作站(PAW)"登录,绝不上普通办公电脑;服务账户拆细,一个服务一个账户;SSH 密钥按角色拆,跳板机统一管理;数据库账户分读 / 写 / 管理,应用绝不用 root 连库;CI/CD 凭证轮转,默认最短有效期。

    十三、内网安全加固 checklist

    下面这份清单,任何一家中大型企业都应该有人按月对照。打勾就放心,打叉就排进 OKR。

    13.1 终端 & 凭证

    • 所有终端装 EDR(不是只杀软),有集中管控台
    • LSASS 保护模式(RunAsPPL)开启
    • Windows Credential Guard 在域内全员开启
    • 浏览器禁用密码保存,公司 SSO + MFA
    • SSH 私钥按角色 / 环境拆分,有过期机制
    • 任何配置文件不准 hard-code 密码 / token,统一走 Vault / KMS
    • 用户登录无密码方案(FIDO2 / Smart Card)优先

    13.2 域 & AD

    • Tiered Admin Model 落地——域管账户绝不登录普通终端
    • 域管账户从 PAW(管理专用工作站)登录
    • krbtgt 账户密码每半年重置一次
    • 服务账户全部使用 gMSA,不再人工管密码
    • BloodHound 定期审计,清理 ACL 错配
    • LAPS(本地管理员密码自动化)部署,每台终端本地管理员密码不一样
    • 禁用 NTLM(至少 NTLMv1),全走 Kerberos

    13.3 网络分段

    • 办公网 / 开发网 / 生产网 / 域控网 严格分段
    • 默认禁止跨段,只开必需端口
    • 生产数据库不接受任何"个人电脑直连"
    • SMB / WMI / WinRM 在内部不允许"任意主机互通",只允许从 jump server
    • 出口流量白名单,禁止终端直连公共 DNS(8.8.8.8 / 1.1.1.1)

    13.4 监控 & 检测

    • Sysmon / auditd 在所有终端开启,日志集中收上来
    • SIEM 至少覆盖 ATT&CK 的关键技术(凭证窃取 / 横向 / 持久化)
    • LSASS 访问监控(任何 OpenProcess LSASS 都审计)
    • DNS 异常监控(长子域名 / 高熵字符串 / 高频查询)
    • 流量做 JA3 指纹分析,有已知坏指纹库
    • 蜜罐账户 + Canary Token 部署
    • 应急响应预案有,每年至少一次桌面演练
    • 备份离线 + 不可改(防勒索病毒)

    13.5 软性管理

    • 全员钓鱼演练,每季度一次
    • 离职流程包括"立即撤销所有凭证"
    • 第三方供应商接入有专门网段 + 时限
    • 关键操作有双人审批
    • 红蓝紫队演练每年至少一次

    十四、本篇心智总结

    • 打进来只是开始,真正的危险在内网这 30 步
    • 内网攻击 90% 是"用合法工具做不合法事",WAF 和签名拦不住,只有行为分析能发现
    • 凭证是货币,LSASS / SSH key / 浏览器 cookie 是三个最大金矿
    • 横向不需要打洞,用偷来的凭证就行(Pass-the-Hash / Pass-the-Ticket / SSH 密钥)
    • AD 域是单点信任根,DC 沦陷 = 公司沦陷;krbtgt 是最关键的账户
    • C2 流量越来越像合法流量,只能靠行为基线 / 指纹 / 时序检测
    • MITRE ATT&CK 是攻防共同语言,蓝队不懂 ATT&CK 没法做覆盖率管理
    • 防御核心是纵深 + 最小权限 + 快速发现——别指望边界,要假设"已经有人在里面"
    • 零信任(Zero Trust)的本质就是把"内网默认信任"这个 30 年的假设彻底废掉

    这一篇所有内容只到心智层,不武器化。具体怎么演练,要去授权环境(Hack The Box / TryHackMe / 公司自建靶场)亲手做——蓝队不亲手走一遍,永远做不出能发现这些事的检测规则。下一篇 24-红蓝紫队会接着讲"怎么把攻防演练工程化"——红队怎么写演练计划、蓝队怎么按 ATT&CK 评分覆盖率、紫队(Purple Team)怎么把"红队找的洞"变成"蓝队的检测规则",真正闭环。

    下一篇:24-红蓝紫队.md,讲清楚为什么"只有红队不够、只有蓝队也不够"——攻击者演练完没人接,等于花钱买了一份"过去式漏洞清单";防御方不知道自己漏在哪,等于盲人摸象。Purple Team 是把红蓝拉到一张桌子上的工程实践,这十年安全行业最值钱的角色转变就在这里——也是为什么大厂安全团队都在招"紫队工程师"而不是单纯的"红/蓝队"。
