威胁建模实战(STRIDE / DREAD / 攻击树 / 数据流图) ​

绝大多数线上漏洞不是因为代码写错了,而是因为设计阶段根本没想过会被这样攻击——「我以为只有内部用户能调这个接口」「我以为前端会过滤」「我以为这个域名是可信的」。「我以为」就是威胁建模要消灭的东西。这一篇讲清楚怎么在写第一行代码之前,把整个攻击面铺在白板上拆——STRIDE 怎么逐项扫,DREAD 为什么打分不准但还要打,攻击树和数据流图分别解决什么问题,以及最重要的——做完一次完整威胁建模到底长什么样。

一句话先记住:威胁建模 = "用攻击者视角逐边、逐节点、逐信任域质询自己的设计"。它不是出文档凑流程,它是把"我以为"换成"我已经想过这种情况"。看完这篇,「上线评审」对你应该不再是过场——你能在 30 分钟内对任何一个新功能画出 DFD、跑一遍 STRIDE、定位三到五个高风险点。这是设计阶段就能止损 70% 漏洞的杠杆。

一、为什么需要威胁建模 ​

1.1 三个真实场景 ​

场景 1:头像上传打穿内网

功能:用户传一个图片 URL,后端拉回来缩略图存 OSS。
代码:requests.get(user_provided_url).content
出事:用户传 http://169.254.169.254/latest/meta-data/iam/security-credentials/
      后端返回 AWS 临时凭证 → 攻击者拿到整个云账号

这不是写代码时漏检查——是设计这个功能时根本没想过 URL 是攻击面。Capital One 2019 一亿用户数据泄露,根因就是这条 SSRF。设计阶段花 10 分钟想就能避开,线上爆出来就是百万美元级事故。

场景 2:JWT 上线半年被发现允许 alg: none

代码:jwt.decode(token, verify=True)  ← 第三方库默认信任 alg 字段
攻击:把 token 的 alg 改成 "none",签名删空,服务器照样接受

根因不是库的 bug——是设计时没问过"如果客户端能控制算法字段会怎样"。STRIDE 里这是经典的 T (Tampering),五分钟就能扫到。

场景 3:多租户缓存撞 key

A 公司:GET /api/user/profile  → 缓存 key = "profile:123"
B 公司:GET /api/user/profile  → 缓存 key = "profile:123"  ← 撞了
B 看到了 A 的资料

缓存键空间没考虑租户隔离——DFD 画完信任边界一标,闭着眼都能看出来。

1.2 漏洞修复成本曲线 ​

设计阶段发现:  1×  (改个方案,半小时)
开发阶段发现:  10×
测试阶段发现:  100×
上线后发现:    1000×
被人利用后:    10000×(罚款 + 诉讼 + 用户流失)

这条曲线 IBM 1980 年代就量化过,40 年没变。威胁建模就是把发现时机从右往左推——越靠左,杠杆越大。

1.3 威胁建模不是什么 ​

先把误解扫干净:

  • 不是渗透测试——渗透是黑盒打,建模是白盒推
  • 不是代码审计——审计看实现,建模看设计
  • 不是 OWASP Top 10 检查表——通用清单覆盖不到你的业务独有攻击面
  • 不是给老板看的 PPT——是让团队在白板前争吵 90 分钟,留下五个待办

威胁建模是一种集体推理活动,不是文档,不是工具,不是产物。输出物是"想过"这件事本身——文档只是副产品。

二、Adam Shostack 的四问框架 ​

Microsoft 的 Adam Shostack 在《Threat Modeling: Designing for Security》里把整个方法论浓缩成 四个问题,这是整个领域的奥卡姆剃刀:

1. 我们在构建什么?     → 画 DFD
2. 哪里会出错?          → STRIDE / 攻击树
3. 打算怎么办?           → 缓解措施(Mitigation)
4. 做得够好吗?           → 复审 / 测试 / 回归

整个威胁建模就是按这四步走一遍,没了。任何方法论(PASTA、Trike、OCTAVE)都是这四个问题的不同侧重。先死记这四个,工具次要。

三、数据流图(DFD):建模的第一张画 ​

不画 DFD 直接讲威胁,等于不看地图就讨论防御工事——你根本不知道前线在哪。

3.1 DFD 的四个元素 ​

┌─────────┐         ┌──────────────┐         ┌─────────┐
│ 外部实体 │  ───→  │  过程 / 进程  │  ───→  │ 数据存储 │
│ (User)  │         │  (Service)   │         │  (DB)   │
└─────────┘         └──────────────┘         └─────────┘

  □  外部实体:用户、第三方系统(矩形)
  ○  过程:你的服务、函数(圆形 / 圆角矩形)
  ▭  数据存储:DB、缓存、文件(双横线)
  →  数据流:API 调用、消息(箭头)

3.2 信任边界(Trust Boundary):DFD 的灵魂 ​

画 DFD 90% 的价值不在画元素,在画信任边界。信任边界 = 数据从一个信任级别到另一个信任级别的瞬间——这就是攻击发生的地方。

┌────────────────────────────────────────────┐
│              公网(零信任)                 │
│  ┌────────┐                                 │
│  │ 用户   │                                 │
│  └────┬───┘                                 │
└───────┼─────────────────────────────────────┘
        │  ←━━━━━━━━ 信任边界 1(进入 DMZ)
┌───────┼─────────────────────────────────────┐
│       ↓     DMZ                              │
│  ┌─────────┐                                 │
│  │ Nginx   │                                 │
│  └────┬────┘                                 │
└───────┼─────────────────────────────────────┘
        │  ←━━━━━━━━ 信任边界 2(进入内网)
┌───────┼─────────────────────────────────────┐
│       ↓     内网                              │
│  ┌─────────┐    ┌─────────┐                  │
│  │ App     │───→│ MySQL   │                  │
│  └─────────┘    └─────────┘                  │
└──────────────────────────────────────────────┘

所有威胁本质上都是"数据穿越信任边界时没被验证":公网 → DMZ 时所有输入都是恶意的;DMZ → 内网时要假设上游被攻破了;App → DB 时 SQL 注入就发生在这条边界上拼字符串。

四、STRIDE:一张表把六类威胁扫干净 ​

STRIDE 是 Microsoft 1999 年提出的助记词,每个字母对应一类威胁,对应一个安全属性:

S  Spoofing               身份伪造  → 破坏 认证
T  Tampering              数据篡改  → 破坏 完整性
R  Repudiation            抵赖否认  → 破坏 不可否认
I  Information Disclosure 信息泄露  → 破坏 机密性
D  Denial of Service      拒绝服务  → 破坏 可用性
E  Elevation of Privilege 权限提升  → 破坏 授权

真正用法不是"我懂这六个名词",而是"对 DFD 上每个元素逐个套这六个问题"。

4.1 STRIDE-per-Element 速查表 ​

              S    T    R    I    D    E
外部实体      ✓    -    ✓    -    -    -
过程/进程     ✓    ✓    ✓    ✓    ✓    ✓
数据存储      -    ✓    ✓    ✓    ✓    -
数据流        -    ✓    -    ✓    ✓    -

外部实体只能被假冒和抵赖,没"内部"所以没法 T/E;过程是计算节点,六类全中;数据存储有 T(被改)、R(没日志)、I(被读)、D(被删/占满);数据流有 T(中间人篡改)、I(嗅探)、D(阻断)。

4.2 六类威胁逐项 ​

S — Spoofing(身份伪造):我不是张三,但让系统以为我是张三。典型:弱密码 / 撞库、Session 劫持、JWT 的 alg: none 绕过、钓鱼站、域名相似(g00gle.com)、服务间无 mTLS。缓解:强认证(MFA、WebAuthn)、Session 绑设备指纹、服务间 mTLS、DNSSEC / HSTS。

T — Tampering(数据篡改):数据在传输或存储中被改了。典型:中间人篡改 HTTP、SQL 注入(篡改查询语义)、上传 .php 当头像、JWT payload 篡改 + 弱 key 爆破、数据库直连改字段。缓解:强制 HTTPS、白名单输入校验 + 输出编码、签名(JWT 用 RS256)、HMAC / Merkle 校验完整性、DB 审计日志。

R — Repudiation(抵赖):用户做了操作但事后说"不是我",系统没证据。典型:转账后否认(没日志)、删除关键数据后说"我没删"、操作日志可被用户自己改。缓解:append-only 审计日志、关键操作签名、日志单独存(用户权限读不到)、时间戳 + 用户 ID + 来源 IP。容易被忽视——很多团队上线半年才发现"出事了根本查不到是谁干的"。

I — Information Disclosure(信息泄露):不该知道的人知道了。典型:报错暴露栈(SQLException: select * from users...)、API 返回过多字段(吐 password_hash)、路径穿越(../../etc/passwd)、IDOR(/api/order/1001 改成 1002 看到别人订单)、缓存 key 撞了、.git/ 公开。缓解:字段白名单序列化(DTO,不要直接吐 Model)、错误信息分级(用户模糊、日志详细)、鉴权每次都做、敏感字段加密存储。

D — Denial of Service(拒绝服务):服务不可用。典型:DDoS、资源耗尽(打开 100 万 socket)、算法复杂度攻击(传引发 O(n²) 解析的字符串)、慢连接(Slowloris)、缓存击穿、上传 10G 文件占满磁盘。缓解:限流(Token / Leaky Bucket)、资源配额(超时、最大 body、连接数)、缓存预热 + singleflight、CDN / WAF 前置、异步队列削峰。

E — Elevation of Privilege(权限提升):普通用户变管理员,或拿到不该有的权限。典型:越权(改 URL /admin/* 没鉴权)、IDOR 升级版改 user_id、反序列化 RCE、容器逃逸(挂载了 /var/run/docker.sock)、SUID 提权。缓解:最小权限原则、鉴权放网关 + 业务双层、不反序列化不可信数据(用 JSON,不用 Java/Python pickle)、容器只读根文件系统 + 删 capabilities。

五、DREAD:打分法的实际意义和局限 ​

STRIDE 帮你找到威胁,DREAD 帮你排序威胁——总有先后,人力有限。

D  Damage          危害程度        造成多大损失?(1-10)
R  Reproducibility 复现难度        多容易稳定复现?(1-10)
E  Exploitability  利用难度        多容易实施?(1-10)
A  Affected Users  影响范围        多少用户受影响?(1-10)
D  Discoverability 发现难度        多容易被发现?(1-10)

风险分 = 五项均值

一个示例:头像上传 SSRF 打内网 → D=10 / R=9 / E=8 / A=10 / D=6 → 平均 8.6,高危。

DREAD 的局限 ​

学术界其实已经不太用 DREAD 了,有几个硬伤:

  1. 打分主观——同一个人不同状态都不一样,何况两个人
  2. 均值压平极端——把致命项稀释成中危
  3. Discoverability 是个伪指标——你别打分它就不存在了?攻击者总能扫到
  4. 不区分概率和影响——D/A 是影响,R/E 才是概率,搅一起算均值没物理意义

现代替代:CVSS(下一篇会讲)、Risk = Likelihood × Impact 二维矩阵、直接分高/中/低三档。

但 DREAD 不是没用——它的最大价值是强迫团队对每个威胁逐项讨论,这个过程比最终分数有用。别迷信分数,珍视讨论。

六、攻击树(Attack Tree):站在攻击者那边想 ​

STRIDE 是"防御方对每个元素扫一遍"——深度不够。攻击树是把攻击者的目标拆成子目标,递归到底。根节点是攻击者的最终目标,子节点是实现该目标的方法,OR / AND 关系。

目标:获取管理员账号
│
├── OR ── 直接登录 admin
│   ├── AND ── 拿到 admin 密码
│   │   ├── OR ── 撞库(admin/admin123)
│   │   ├── OR ── 钓鱼邮件
│   │   ├── OR ── 数据库泄露后离线爆破
│   │   └── OR ── 内部人员泄密
│   └── AND ── 绕过 MFA
│       ├── OR ── SIM 劫持
│       ├── OR ── 重放 OTP
│       └── OR ── 社工客服重置
│
├── OR ── 提权到 admin
│   ├── OR ── IDOR 改 user_id
│   ├── OR ── 越权 API(/admin/* 没鉴权)
│   ├── OR ── JWT 篡改 role 字段
│   └── OR ── SQL 注入更新自己的 role
│
└── OR ── 绕过认证直接进
    ├── OR ── Session 固定攻击
    ├── OR ── 认证模块 RCE
    └── OR ── 后门(供应链 / 内鬼)

攻击树比 STRIDE 更深——它逼你想"如果我是攻击者,这条路走不通我还能走哪条":强化密码策略 → 改走钓鱼;加 MFA → 改走 SIM 劫持;修 IDOR → 改走 JWT 篡改。真实的攻击是水流——你堵住一条路它就找下一条。攻击树就是把所有路画出来,让你知道哪一道防线真正卡住了大多数路径(纵深防御的关键节点)。

每个叶子节点可以标:成本、收益、概率。最危险的路径 = 高概率 × 高收益 × 低成本,这是甲方做"先修哪个"决策的依据。

七、实战:对「头像上传」做完整威胁建模 ​

纸上谈兵到此为止。下面对一个真实功能跑完整流程——这是这一篇最重要的部分。

7.1 需求(在构建什么?) ​

功能:用户可以上传头像。
方式 A:本地文件上传(PNG/JPG,< 2MB)
方式 B:粘贴一个 URL,后端去拉
处理:后端缩略图压到 200x200,存 S3,通过 CDN 分发

7.2 画 DFD ​

┌━━━━━━━━━━━━━━━━━━━ 公网信任域 ━━━━━━━━━━━━━━━━━━━━━┓
┃                                                       ┃
┃   ┌───────┐                                            ┃
┃   │ User  │── ① upload file ──┐                        ┃
┃   │       │── ② URL string ───┤                        ┃
┃   └───────┘                   │                        ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━━━━━━━━━┛
                                │ ← 信任边界 A
┌━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━━━━━━━━━┓
┃                                ↓     App 内网          ┃
┃                        ┌───────────────┐               ┃
┃                        │  AvatarSvc    │               ┃
┃                        │  - 类型校验   │               ┃
┃                        │  - 缩略图     │               ┃
┃                        └───┬───────┬───┘               ┃
┃              ③ HTTP GET    │       │ ④ PUT object      ┃
┃                            ↓       ↓                   ┃
┃    ┌───────────────┐              ╔══════════╗         ┃
┃    │ 远端图床 / ??? │←──────       ║   S3     ║         ┃
┃    └───────────────┘              ╚════╤═════╝         ┃
┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┿━━━━━━━━━━━━━━━┛
                                        │ ← 信任边界 B
                                        ↓
                                  ┌──────────┐
                                  │   CDN    │── ⑤ ──→ 全网
                                  └──────────┘

信任边界 A:用户输入,所有数据都恶意。 信任边界 B:存储 → CDN,这里出来的东西会被全网信任(同源、cookie 等)。

7.3 STRIDE 逐项扫 ​

对 AvatarSvc(过程):

威胁场景评估
T没校验文件类型,上传 .php 改后缀为 .jpg 实际是 PHP 木马高危
T图片解析漏洞(ImageMagick 历史 CVE),特制图片触发 RCE中危
R上传操作没日志,事后查不到谁传了违规图中危
I错误信息暴露内部路径 / 栈低危
D用户传 10G 文件 / 1px × 100000px 解压炸弹中危
E文件落到 web 目录被解析为脚本(经典 webshell)高危

对「URL 拉取」这条数据流(③):

威胁场景评估
ISSRF:传 http://169.254.169.254/... 拉云元数据致命
ISSRF:传 http://10.0.0.5:6379/... 探内网 Redis致命
ISSRF:传 file:///etc/passwd 读本地文件致命
D远端返回无限流 → 后端打满磁盘 / 内存中危
D远端故意慢响应 → 占用后端连接数中危

对 S3 存储:bucket 错配公网可写(T)、public-read 枚举 key 偷别人头像(I)、上传无限制刷爆存储费用(D)——都是中危。

对 CDN(信任边界 B 之后):

威胁场景评估
T文件名可控且 CDN 不重写 Content-Type → 传 .html 当头像,同源 XSS高危
ICDN 缓存键投毒:某用户传特殊 Header / 路径污染了别人的缓存响应中危

7.4 攻击树:目标 = 通过头像功能拿到内网凭证 ​

目标:通过头像功能横向打到内网 / 拿到云凭证
│
├── OR ── SSRF 路径
│   ├── 拉云元数据(AWS / GCP / 阿里云)
│   ├── 探内网服务(Redis / MySQL / k8s API)
│   ├── file:// 读本地敏感文件
│   └── gopher:// 协议构造内网请求(SSRF 2.0)
│
├── OR ── 文件解析路径
│   ├── 伪造图片触发 ImageMagick RCE
│   ├── polyglot 文件(同时是图片 + zip + JS)
│   └── 图片库 OOM / 整数溢出
│
├── OR ── 存储污染路径
│   ├── 上传 .html → CDN 同源 XSS
│   ├── 文件名注入污染 CDN 缓存
│   └── bucket 权限错配遍历他人头像
│
└── OR ── 业务逻辑路径
    ├── 文件大小不限 → 刷爆存储
    ├── 频率不限 → 刷爆带宽
    └── URL 不限 → 拉取外网恶意内容

7.5 缓解措施(怎么办?) ​

针对 SSRF(最致命):

  • 白名单协议:只允许 http:// https://,禁止 file:// gopher:// dict:// ftp://
  • 白名单域名:只允许预先注册的几个图床域名
  • DNS 重绑定防御:解析 DNS → 拿 IP → 校验 IP 不在私有段(10/8、172.16/12、192.168/16、127/8、169.254/16、::1、fc00::/7) → 再用这个 IP 发请求
  • 关闭自动 Redirect:很多 SSRF 利用 302 跳转绕过校验
  • 超时短(5 秒)、Body 大小限(2MB)、禁用 keep-alive
  • 专门的 egress proxy:所有出站请求走一个代理,代理上做策略
python
# 危险代码
content = requests.get(user_url).content   # ← SSRF 起手式

# 安全的样子(伪代码,重点在思路)
def safe_fetch(url):
    p = urlparse(url)
    if p.scheme not in ('http', 'https'):   raise
    if p.hostname not in WHITELIST_DOMAINS: raise
    ip = socket.gethostbyname(p.hostname)
    if ipaddress.ip_address(ip).is_private: raise
    # 用 IP 直连,Host 头带原域名(防 DNS rebinding)
    r = requests.get(f"http://{ip}", headers={"Host": p.hostname},
                     timeout=5, allow_redirects=False, stream=True)
    if int(r.headers.get('content-length', 0)) > 2_000_000: raise
    return r.content

针对文件类型:

  • 白名单后缀 + 白名单 Content-Type + 文件头魔数校验——三层
  • 不要相信用户给的文件名,服务端 UUID 重命名
  • 重新编码图片:Pillow / ImageMagick 解码再编码,变相验证是合法图片,同时去掉 EXIF 和潜在 payload
  • 图片库锁定可信版本,关闭不需要的 coder(ImageMagick policy.xml 禁 MVG/MSL/HTTPS)

针对存储污染:

  • 专用域名:头像走 usercontent.xxx.com,和主站不同源——即使被传 HTML/JS,Cookie 也偷不到
  • 强制 Content-Type:S3 上传时显式设 Content-Type: image/*,忽略文件后缀
  • Content-Disposition: attachment:让浏览器下载而不是执行
  • bucket 隔离 + IAM:写入只能用临时凭证,不要把 long-lived 密钥放在应用里
  • CDN 缓存键明确包含用户 ID + 文件 hash

针对 DoS:上传大小、频率限制(每用户每分钟 N 次);URL 拉取并发 / 速率限制;图片解析有内存上限和超时。

7.6 复审(做得够好吗?) ​

  • 缓解项一条条勾掉,每条都要有对应代码 / 配置 / 测试——口头答应不算数
  • 加单元测试:覆盖 SSRF 黑名单、文件类型校验、超时
  • 加集成测试:故意上传恶意 payload,看是否被拦
  • 上线后接 WAF / 日志告警:SSRF 触发记一笔异常

八、什么时候做威胁建模 ​

│  阶段              建模强度          产出        │
├────────────────────────────────────────────────┤

└────────────────────────────────────────────────┘

节奏建议:不要追求"每个 PR 都建模"——成本太高,团队会抵触。抓两个关键节点:设计评审(代码还没写)和上线评审(线上还没暴露)。关键功能强制:涉及钱、隐私、权限、外部输入的功能必走。每季度复审一次架构图——威胁面是动态的。

九、威胁建模 Checklist 模板(可复用) ​

下次开评审会,对着这份打勾:

■ 第 1 步:在构建什么?
  □ 画了 DFD 吗?
    □ 外部实体 / 进程 / 数据存储 / 数据流 标全
  □ 信任边界画了吗?
    □ 公网 / DMZ / 内网分清
    □ 多租户 / 多角色分清

  □ 找到所有「外部输入点」?

    □ URL / 重定向 / Referer、第三方回调

■ 第 2 步:哪里会出错?(对每个元素逐项打勾)
  □ S 谁来鉴权?MFA?服务间 mTLS?
  □ T 输入白名单?关键字段签名?强制 HTTPS + HSTS?
  □ R 关键操作有审计日志?日志独立存(用户改不到)?
  □ I 错误信息分级?返回字段 DTO 白名单?IDOR / 路径穿越?
       敏感字段加密 / 脱敏?
  □ D 限流(per-user / per-IP / global)?
       资源配额(超时 / body / 连接数)?算法复杂度可控?
  □ E 鉴权每次都做?最小权限?反序列化输入可信?

■ 第 3 步:怎么办?
  □ 每个高风险威胁都有缓解措施
  □ 缓解措施落到具体的代码 / 配置 / 测试
  □ 有 owner 和 deadline
  □ 不只靠单点(纵深防御:验证 + 隔离 + 监控)

■ 第 4 步:做得够好吗?
  □ 缓解项有测试覆盖?
  □ 加了监控 / 告警?
  □ 记录到 SDL 文档,下次复审能查到?
  □ 进入红队测试范围?

这份 checklist 直接复制到团队 wiki / 设计模板里。每次设计评审过一遍,30 分钟封顶。漏的就是漏的,认账——但比拍脑袋强 10 倍。

十、踩坑提醒 ​

  1. 建模只画 DFD 不画信任边界——信任边界是灵魂,没它等于白画
  2. STRIDE 用成检查表——STRIDE 是发起讨论的工具,不是表格凑数
  3. DREAD 当唯一打分依据——主观、压平、忽略概率/影响差异,结合 CVSS 用
  4. 只做一次就放着——架构变了威胁就变了,每季度复审
  5. 只有安全团队做——必须是开发 + 安全 + 架构师一起,研发不在场建模等于零
  6. 建完没人跟进——输出待办 + 责任人 + 截止日,否则就是行为艺术
  7. 只看代码层威胁——业务逻辑漏洞(IDOR、订单越权)代码上完全合规,只有业务建模能抓
  8. 以为"用了 HTTPS 就安全"——HTTPS 只解决 T 和 I 的一部分,还有 S/R/D/E 整整四类
  9. 以为云厂商负责安全——共担责任模型,云只管基础设施,你的代码漏洞自己背锅
  10. 没把第三方依赖纳入边界——npm 包、Docker 镜像、CDN、SaaS API,全是新的攻击面
