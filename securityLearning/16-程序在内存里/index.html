    程序在内存里长什么样:段 / 栈帧 / 调用约定 / 复习

    写 Java / Python / Go 的工程师看到这一篇标题大概率会划走——「我又不写 C,栈溢出 / ROP 这些东西关我什么事」。这是这一层最大的误区。真相是:你用的所有运行时(JVM、V8、CPython、Go runtime、Node)本身就是 C/C++ 写的;你装的所有依赖底层都有 native 模块(libssl、libxml、image codec、数据库驱动);你用的 Linux 内核就是 C。这些东西出一个内存漏洞,你那一层"内存安全语言"的安全保证就全废了——Log4Shell 之所以那么可怕,不是因为 Java 不安全,是因为它能 RCE 直接 fork shell,攻击者拿到的是一个进程内存空间,后续怎么动手就回到这一篇讲的东西。osLearning/06-08 已经从「OS 是怎么管理内存的」视角讲过虚拟内存 / 进程布局 / 内存分配,这一篇从攻击者视角重新看——同样一张内存图,在攻击者眼里每个区域都贴着「能写吗 / 能执行吗 / 写穿了能控制什么」三个标签。

    一句话先记住:程序在内存里就是一张"哪里能写 / 哪里能执行"的地图——.text 段能执行不能写,.data / .bss / heap / stack 能写不能执行(开了 DEP 以后),「数据」和「代码」的边界全靠 OS 的页表标志位维护。所有内存利用的本质都是:让攻击者控制的数据被当成代码执行,或者让"能执行的代码"按攻击者的顺序执行——前者是经典 shellcode,后者是 ROP。而这一切的起点,是程序员忘了一件事:C/C++ 里数组没有边界检查,栈上一个 char[64] 你写 65 字节,溢出去的那一字节落在哪里、写穿了什么,是写代码的人要自己脑补的——大部分人脑补不出来,于是有了过去 30 年所有缓冲区溢出漏洞。

    一、复习:进程地址空间是什么样

    1.1 一张总图

    osLearning/07 详细讲过,这里只快速回顾,重点标注每个段的 R/W/X 权限——这是攻击者真正在意的:

    高地址 ┌────────────────────────────┐ 0x7fff_ffff_ffff
       │  内核空间 (用户不可见)        │
       ├────────────────────────────┤
       │  栈 stack ↓ 向下增长          │ rw-
       │   - 每个线程一份               │
       │   - main 的局部变量、调用帧    │
       ├────────────────────────────┤
       │  ↕ ASLR 在这里随机化           │
       ├────────────────────────────┤
       │  mmap 区(共享库 / 大块 mmap)  │ r-x / rw-
       │   - libc.so / ld.so           │
       ├────────────────────────────┤
       │  堆 heap ↑ 向上增长           │ rw-
       │   - malloc 来的小块           │
       ├────────────────────────────┤
       │  .bss(未初始化全局变量)     │ rw-
       │  .data(已初始化全局变量)    │ rw-
       │  .rodata(只读数据 / 字符串)  │ r--
       │  .text(代码段)              │ r-x  ← 唯一能执行
低地址 └────────────────────────────┘ 0x400000

    1.2 攻击者眼里的三栏

    把上图压缩成攻击者真正关心的视图,只关心三件事:

    区域 内容 可读 可写 可执行 攻击者的兴趣点
    .text 程序指令 ✅ ❌ ✅ 找 gadget(ROP 拼装)
    .rodata 字符串、常量 ✅ ❌ ❌ 找 /bin/sh 字符串地址
    .data / .bss 全局变量 ✅ ✅ ❌ 改函数指针、改 flag
    heap malloc 来的 ✅ ✅ ❌ UAF、堆喷、改元数据
    stack 局部变量 + 返回地址 ✅ ✅ ❌ 改返回地址(经典)
    libc 段 系统库代码 ✅ ❌ ✅ system() / execve 函数地址

    一个心智模型:程序里的每一字节都有「可写 W」和「可执行 X」两个标志位。正常程序绝不会同时 W+X——要么是数据(W 不 X),要么是代码(X 不 W)。早期的 shellcode 攻击就是让一个 W 的内存变成 X(把代码注入栈上然后跳过去执行);DEP / NX 出来以后,这条路堵了,攻击者改打 ROP——不注入新代码,只复用已有 .text 里的代码片段(都是 X 的)。

    1.3 ASLR 在这张图里加了什么

    ASLR(地址空间布局随机化)把上图所有段的起始地址都随机化——每次进程启动,栈在哪、堆在哪、libc 在哪都不一样。攻击者最大的痛点变成"我得先知道 libc 在哪"——这就是为什么后面 ROP 链的第一步永远是「先泄漏一个地址」(info leak)。ASLR 没解决任何根本漏洞,只是把"知道布局"从默认状态变成需要单独打一发——但实战中往往就够了。详见 19 篇。

    二、栈帧:函数调用的"账本"

    栈是攻击者最熟悉的战场——因为栈上同时放着「能改的局部变量」和「能跳的返回地址」,两者中间只隔几字节,没边界检查的话一伸手就够到。

    2.1 一次函数调用,栈上发生了什么

    写一段最小可复现的 C(后面整篇都围绕这段):

    // vuln.c
#include <stdio.h>
#include <string.h>

void greet(char *name) {
    char buf[64];
    strcpy(buf, name);          // 没边界检查!
    printf("hello, %s\n", buf);
}

int main(int argc, char **argv) {
    greet(argv[1]);
    return 0;
}

    main 调用 greet("alice") 时,栈上的样子(System V AMD64,栈往下增长):

    高地址  ┌───────────────────────────┐
        │  ... main 之前的栈 ...      │
        │  返回到 main 的下一条指令地址 │ ← call 指令自动 push
        ├───────────────────────────┤ ← greet 栈帧起点
        │  saved RBP(main 的 RBP)   │ ← greet prologue push
        ├───────────────────────────┤ ← 当前 RBP
        │  buf[63] (高地址端)        │
        │  ...                       │
        │  buf[0]  (低地址端)        │ ← RBP - 0x40
        ├───────────────────────────┤ ← RSP
低地址  │                           │

    关键观察:buf[0] 在低地址,buf[63] 在高地址,saved RBP 在 buf[63] 之上,返回地址在 saved RBP 之上。如果攻击者让 strcpy 写超过 64 字节,多出来的字节就会从低往高、依次盖过 buf[63] → saved RBP → 返回地址。这就是"栈溢出"——不是"内存被破坏了",是"返回地址被改了"。

    2.2 栈帧的三件套

    每个函数的栈帧固定有三类东西,从高地址往低地址排:

    ┌─────────────────────────┐
│  return address(返回地址) │  ← caller 的 call 指令自动 push
├─────────────────────────┤
│  saved RBP(上一帧的 RBP) │  ← callee 的 prologue 主动 push
├─────────────────────────┤   ← 当前 RBP 指向这里
│  局部变量 / 编译器临时变量 │
├─────────────────────────┤   ← 当前 RSP 指向这里
    • 返回地址 (return address):call func 这条指令做了两件事——push 下一条指令的地址 + jmp func。返回地址就是 caller 里 call 后面那条指令的地址,函数 ret 时 pop 出来跳回去。攻击者改了它 = 决定了函数返回时去哪儿执行。
    • saved RBP:RBP 是「帧指针」,指向当前函数栈帧的基准。函数进来第一件事就是把 caller 的 RBP 存起来,函数 leave 时再恢复——这样调用栈像一根链表一样能往上回溯(gdb 的 bt 就靠这个)。
    • 局部变量:char buf[64]、int x、struct foo s,全部在 RBP 下方,通过 [rbp - offset] 寻址。数组越界往高地址写,先盖局部变量,再盖 saved RBP,再盖返回地址——这是栈溢出的物理顺序。

    2.3 为什么栈往下增长

    答案是历史决定——早期 PDP-11 这么设计,后来所有架构都跟着——但带来一个反直觉的安全后果:数组(buf[0] 在低地址)越界往高地址写,会盖到"自己之上"的东西,而自己之上恰好是 saved RBP 和返回地址。栈方向和数组方向相反,这是栈溢出能利用的物理基础。如果栈往上长,缓冲区溢出会盖到自己「之前」的局部变量,盖不到返回地址,利用就难得多。

    三、调用约定:参数和寄存器的契约

    「函数调用」在汇编层不是一个原语,而是 caller 和 callee 之间的一份合同——谁负责把参数放哪儿、谁存哪些寄存器、返回值放哪儿。这份合同叫调用约定 (calling convention)。

    3.1 System V AMD64(Linux / macOS 默认)

    参数传递(前 6 个用寄存器,从左到右):
  RDI  ─ 第 1 个参数
  RSI  ─ 第 2 个参数
  RDX  ─ 第 3 个参数
  RCX  ─ 第 4 个参数
  R8   ─ 第 5 个参数
  R9   ─ 第 6 个参数
  栈   ─ 第 7 个以及之后(从右往左 push)

返回值:
  RAX  ─ 整数 / 指针返回值
  RDX  ─ 第二个返回值(少用)

调用者保存(caller-saved):RAX, RCX, RDX, RSI, RDI, R8-R11
  ─ caller 用之前要自己存,因为 callee 会随便改
被调用者保存(callee-saved):RBX, RBP, R12-R15
  ─ callee 用之前要先 push 存起来,返回前 pop 恢复

栈对齐:call 指令前 RSP 必须 16 字节对齐

    攻击者视角的 cheat sheet:在 64 位 Linux 上 system("/bin/sh") 需要 RDI 指向字符串 /bin/sh、然后 jmp 到 system 的地址。ROP 链第一步永远是想办法把 RDI 设置好——后面 17 篇详讲。

    3.2 cdecl(32 位 x86)

    老一点的 32 位 Linux 程序用 cdecl:

    参数:全部从右往左 push 到栈上(没有寄存器传参)
返回值:EAX
栈清理:caller 负责(从栈上 pop 掉)

    32 位的栈溢出比 64 位简单得多——因为参数都在栈上,改完返回地址后紧接着的几个 32 位字就是"被调函数的参数"。打 ret2libc 时直接 [ret_addr | system_addr | fake_return | "/bin/sh"_addr],system 一返回就拿到 shell。64 位下因为 RDI 用寄存器传参,必须先有个 pop rdi; ret 的 gadget 把字符串地址塞进 RDI——这就是为什么 64 位栈溢出"门槛高一点"。

    3.3 别的约定速览

    约定 平台 参数传递
    System V AMD64 Linux/macOS 64 位 6 寄存器 + 栈(本系列默认)
    Microsoft x64 Windows 64 位 RCX/RDX/R8/R9 + 栈
    cdecl 32 位 C 全栈,caller 清
    stdcall 32 位 Win32 API 全栈,callee 清
    ARM AAPCS ARM R0-R3 + 栈

    为什么要懂:逆向 / 调试任何 native 程序,第一件事是确认调用约定,否则你看寄存器完全不知道哪个是参数哪个是临时值。

    四、prologue 和 epilogue:函数的开闭幕

    每个 C 函数在汇编层都有一段标准的开头和结尾,这两段是栈帧建立和销毁的代码——也是攻击者最熟悉的两段汇编。

    4.1 标准 prologue

    greet:
    push rbp              ; 保存 caller 的 RBP
    mov  rbp, rsp         ; 当前 RBP = 当前栈顶
    sub  rsp, 0x40        ; 给局部变量腾 64 字节
    ; ... 函数体 ...

    三步:保存旧 RBP → 建立新 RBP → 预留局部变量空间。这样函数内任何 [rbp - 8]、[rbp - 0x40] 都能稳定访问局部变量,不受栈顶 RSP 波动影响(函数内可能继续 push / call,RSP 会一直变,但 RBP 不变)。

    4.2 标准 epilogue

        leave                 ; mov rsp, rbp; pop rbp(两步合一)
    ret                   ; pop 返回地址到 RIP

    两步:恢复栈顶到 RBP → pop 旧 RBP → ret 弹出返回地址跳回去。ret 这条指令是整个安全世界最关键的指令之一——它从栈上 pop 一个 8 字节当指令地址跳过去,没有任何检查。攻击者改了栈上的返回地址,ret 老老实实跳过去。ROP 攻击就是把一堆 gadget 的地址塞到栈上,让程序"一直 ret 一直跳"——详见 17 篇。

    4.3 一段真实的 disassembly

    把 § 2.1 那段 C 编译一下(关掉栈保护,后面 19 篇讲为什么):

    gcc -fno-stack-protector -no-pie -O0 -o vuln vuln.c

    objdump -d vuln 看 greet 函数:

    0000000000401136 <greet>:
  401136:  push   rbp                       ; prologue 开始
  401137:  mov    rbp,rsp
  40113a:  sub    rsp,0x50                  ; 留 80 字节(64 给 buf)
  40113e:  mov    QWORD PTR [rbp-0x48],rdi  ; 存参数 name
  401142:  lea    rax,[rbp-0x40]            ; rax = &buf[0]
  401146:  mov    rdx,QWORD PTR [rbp-0x48]
  40114a:  mov    rsi,rdx                   ; strcpy 参数 2
  40114d:  mov    rdi,rax                   ; strcpy 参数 1
  401150:  call   401030 <strcpy@plt>       ; ★ 无边界检查
  ...
  40116e:  leave                            ; epilogue
  40116f:  ret                              ; ★ 弹返回地址跳回 main

    几个安全相关的观察:buf 在 [rbp - 0x40],离 saved RBP 有 64 字节——写满 64 顶到 saved RBP 边缘,写 72 字节就盖到返回地址。strcpy@plt 是个桩函数,跳到动态链接器解析的真实 libc strcpy(下一节讲)。ret 是函数最后一条指令——栈上 [rbp + 8] 那 8 字节就是返回地址,完全在攻击者能写到的范围内。

    五、C/C++ 为什么是内存安全黑洞

    osLearning 不太涉及语言层,这里专门讲一下——这是后面 4 篇(17-20)反复用到的根本前提。

    5.1 没有边界检查这一条就够了

    C/C++ 数组就是「起始指针 + 编译期记的长度」,长度运行时不存:

    char buf[64];
buf[65] = 'A';        // 编译器:OK,我帮你算地址
buf[1000000] = 'A';   // 编译器:OK(crash / corrupt 自己负责)

    对比 Java:byte[] buf = new byte[64]; buf[65] 运行时抛 ArrayIndexOutOfBoundsException——Java 数组对象多存一个 length 字段,每次访问隐式 if (i >= buf.length) throw。这一条 if 堵死了所有缓冲区溢出——代价是每次访问多 1 条指令(JIT 还能优化掉一部分)。C/C++ 拒绝这个代价,换来了过去 30 年所有缓冲区溢出漏洞。

    5.2 指针没有"所有权"和"生命周期"

    char *p = malloc(64);
free(p);
*p = 'A';             // use-after-free,编译器不管
free(p);              // double free,编译器不管

char *q;
*q = 'A';             // 未初始化指针,编译器不管

    C 的指针只是一个 8 字节的整数,没人记着它指的内存是不是还活着、是不是初始化过、是不是越界。Rust 通过编译期借用检查器把这些全检出来,代价是写起来"反人类",但拿到了"内存安全 + 零运行时开销"——20 篇详讲。

    5.3 标准库帮倒忙

    C 标准库里一大堆函数生来就不安全,但因为是「标准」,删不掉、动不了:

    危险函数 安全替代 问题
    strcpy(dst, src) strncpy / strlcpy 不检查 src 长度
    strcat(dst, src) strncat / strlcat 同上
    gets(buf) fgets 完全没法指定长度(C11 终于删了)
    sprintf(buf, ...) snprintf 不检查输出长度
    scanf("%s", buf) scanf("%63s", buf) 默认无长度限制

    而且 strncpy 本身还有坑——dst 没塞满会有未初始化字节,塞满了不补 \0。C 字符串 API 是一个反例博物馆,这就是为什么 Google 写 C++ 必须用 absl::string_view、Rust 干脆没 C 字符串这个概念。

    5.4 一个数字

    CISA + NSA + 微软都发过报告:70% 的 CVE 是内存安全漏洞(微软 2019 数据)。Chrome、Android、Windows、Linux 内核——高分 CVE 里超过一半是缓冲区溢出、UAF、double free。这不是程序员水平问题,是语言设计问题。

    六、攻击者眼里的"返回地址"

    回到栈帧那张图,把攻击者的视角浓缩成一句话:返回地址是「下一条要执行的指令的指针」,而它就放在能被普通局部变量数组溢出盖到的地方。

            ┌───────────────────────────┐
        │  return address(8 字节)   │  ← 攻击者写到这里 → 控制 RIP
        ├───────────────────────────┤
        │  saved RBP                 │
        ├───────────────────────────┤
        │  buf[63] ─── buf[0]        │  ← 攻击者从这里开始写
        └───────────────────────────┘

    strcpy(buf, attacker_input) 时,攻击者输入 72 字节:

    前 64 字节:垃圾,塞满 buf
接着 8 字节:塞掉 saved RBP(无所谓,不重要)
最后 8 字节:这就是新的返回地址!

    函数 ret 时,CPU 老老实实从栈上 pop 这 8 字节,放到 RIP,接下来就执行那里的代码——攻击者全权决定下一条指令在哪儿。

    早期(2000 年前):把 shellcode 塞到 buf 里,新返回地址指 buf → 跳过去执行 shellcode。栈上代码执行。

    DEP/NX 之后(~2004):栈被标记为 NX,跳到栈上立刻段错误。攻击者改打 ROP——新返回地址指向 .text 里已有的代码片段(都是 X 的),拼出想要的逻辑。详见 17 篇。

    ASLR 之后(~2005):每次启动地址不一样,得先泄漏地址才能算偏移。但栈溢出本身没被"修复",只是抬高了利用门槛——根本漏洞还在,因为 C 没边界检查。

    七、PLT / GOT:动态链接的两张表

    call strcpy@plt 里那个 @plt 是什么?ROP / ret2libc 攻击里到处出现的 PLT、GOT,在这里讲清楚——这是 17 篇 ROP 链的物理基础。

    7.1 静态 vs 动态链接

    静态链接:把 libc 整个塞进可执行文件
  优点:独立,跨机器没依赖问题
  缺点:每个程序都带一份 libc,几 MB 起步

动态链接:可执行文件只放"我要用 printf"这条记录
  运行时:OS 加载 libc.so,把 printf 的真实地址填进来
  优点:libc 全系统只一份,内存省
  缺点:启动时要重定位,需要 PLT/GOT 机制

    Linux 默认动态链接——ldd ./vuln 能看到它依赖 libc.so.6。

    7.2 PLT 和 GOT 怎么配合

    PLT (Procedure Linkage Table) 和 GOT (Global Offset Table) 是两张表,配合起来实现"懒加载"动态函数解析:

    你的代码:               call  strcpy@plt
                              ↓
strcpy@plt:           jmp  [GOT 里 strcpy 的槽位]
                              ↓
                  GOT 里 strcpy 的槽位
                  ┌─────────────────────────┐
                  │ 第一次调用前:指向 PLT 的下一条指令│
                  │ 已解析后:指向 libc 里真实 strcpy │
                  └─────────────────────────┘

    第一次调用:GOT 里还没填好,跳回 PLT 触发 ld.so 解析 → 找到 libc 里真 strcpy 地址 → 写进 GOT → 跳过去。第二次以后:GOT 已经有真地址,直接一跳到位。

    7.3 攻击者的兴趣点

    表 可读 可写 可执行 攻击利用
    PLT ✅ ❌ ✅ 跳板——call printf@plt 是确定的低地址,ROP 链常用
    GOT ✅ ✅(默认) ❌ GOT 改写——把 GOT 里某函数槽位改成 system,下次调用就 RCE

    GOT 改写攻击:有任意写漏洞但没法控制 RIP 时,把 GOT 里 printf 的槽位改成 system 的地址,下次程序调 printf 就变成调 system——经典的"任意写转 RCE"。缓解措施叫 Full RELRO(把 GOT 在加载后改成只读),现代 Linux 一般默认开,详见 19 篇。

    PLT 作为跳板:ROP 链里需要 call system("/bin/sh") 时,不需要知道 libc 里 system 的真实地址(被 ASLR 随机化了),只要跳到 system@plt 就行——PLT 在 .text 段,地址固定(no-pie 编译)或相对固定。这就是为什么 ret2plt 是 ROP 入门第一步。

    八、调试器视角:用 gdb / pwndbg 看栈

    讲这么多概念,不亲手在 gdb 里看一遍栈,永远不会真懂。这一节给一个最小演示流程——完全在你自己机器上,完全合法。

    8.1 装 pwndbg

    pwndbg 是给逆向 / pwn 用的 gdb 插件,把寄存器 / 栈 / 反汇编一屏显示出来:git clone https://github.com/pwndbg/pwndbg && cd pwndbg && ./setup.sh,之后 gdb ./vuln 自动加载。

    8.2 看 § 2.1 那段程序的栈

    gcc -fno-stack-protector -no-pie -g -O0 -o vuln vuln.c
gdb ./vuln

    在 gdb 里:

    (gdb) break greet
(gdb) run AAAAAAAA
(gdb) ni                  # 单步执行三次,过完 prologue
(gdb) ni
(gdb) ni

(gdb) info reg rbp rsp
rbp  0x7fffffffdfa0
rsp  0x7fffffffdf50

(gdb) x/16gx $rsp         # 栈顶往上 16 个 8 字节
0x7fffffffdf50: 0x...     ← buf[0..7]
0x7fffffffdf58: 0x...     ← buf[8..15]
...
0x7fffffffdf90: 0x...     ← buf[56..63]
0x7fffffffdf98: 0x...     ← buf 结束
0x7fffffffdfa0: 0x...     ← saved RBP
0x7fffffffdfa8: 0x...     ← return address(回到 main)

    你应该看到的:buf 占 64 字节,填满 [rbp - 0x40, rbp),接下来 8 字节 saved RBP,再 8 字节返回地址。这 80 字节的物理布局就是这一篇所有概念的具象化——攻击者就是让 strcpy 写超过 64 字节,把后 16 字节按自己意愿改写。

    8.3 看返回地址被改的瞬间

    把命令行参数从 8 个 A 改成 80 个 A:

    (gdb) run $(python3 -c "print('A'*80)")

    跑到 ret 指令前(break *0x40116f),再看返回地址那个槽位——应该已经是 0x4141414141414141(全是 'A' 的 ASCII)。ret 一执行,RIP 跳到 0x4141...,进程立刻段错误。这就是栈溢出最简单的形态——还没"控制",但已经"破坏"了。从"破坏"到"控制",中间就是 17 篇 ROP 那一层"工程艺术"。

    8.4 常用 gdb / pwndbg 命令速查

    info reg       看所有寄存器
x/16gx $rsp    看栈顶 16 个 8 字节
x/16i $rip     看接下来 16 条指令
disas greet    反汇编整个函数
bt             看调用栈(靠 saved RBP 链回溯)
checksec       查程序开了哪些缓解(NX / Canary / PIE / RELRO)
vmmap          看进程内存映射
ROPgadget      找 .text / libc 里的 gadget

    checksec 这一条尤其重要——pwn 题第一件事永远是 checksec ./binary:

    RELRO:    Partial RELRO
Stack:    No canary found
NX:       NX enabled
PIE:      No PIE (0x400000)

    四个字段对应四道缓解墙,每一道都决定攻击者要不要绕——19 篇专门讲。

    九、把所有概念串起来:一张图

    攻击者的"内存利用心智地图"

1. 每一字节有 W(能写)/ X(能执行)两个标志位
   .text  --x  → ROP 找 gadget
   .data  rw-  → 改全局变量、改 GOT
   stack  rw-  → 改返回地址(本篇)
   heap   rw-  → 18 篇主战场

2. 栈帧自上而下:return addr | saved RBP | 局部变量
   局部数组越界写 → 先盖 saved RBP → 再盖 return addr
   ret 指令信任栈上返回地址,不做任何检查

3. 调用约定(SysV AMD64):
   参数 RDI/RSI/RDX/RCX/R8/R9,返回值 RAX
   ROP 链先填 RDI,再跳目标函数

4. 动态链接:PLT 是跳板,GOT 是地址表
   ret2plt:跳 system@plt 就能调 system
   GOT overwrite:任意写改 GOT 转 RCE

5. 缓解机制(19 篇详讲):
   DEP/NX → 干掉 shellcode
   ASLR → 需要先泄漏地址
   Canary → 检测栈溢出
   Full RELRO → 锁死 GOT
   CFI / Shadow Stack → 干掉 ROP

    十、常见误区

    1. 「我写 Java / Python,这些跟我没关系」——V8、JVM、CPython、所有 native 依赖都是 C/C++,出一个洞你全栈裸奔。
    2. 「栈不可执行就没事」——DEP 之后 ROP 完全绕过 NX,根本漏洞还在那儿。
    3. 「ASLR 解决了缓冲区溢出」——只是随机化地址,不修复任何漏洞;有 info leak 就能算偏移。
    4. 「Stack Canary 杜绝栈溢出」——只能检测连续覆盖,单字节越界、特定字段改写都能绕。
    5. 「我没用 strcpy,我安全」——memcpy / sprintf / scanf("%s") / read 长度算错都是潜在溢出。
    6. 「编译器 -O2 帮我检查越界」——不会,只在能静态证明越界时报警,动态下标完全不管。
    7. 「Rust 完全免疫」——safe 子集免疫,unsafe 块和 FFI 调 C 不免疫。
    8. 「PLT/GOT 是链接器细节,工程师不用懂」——所有动态库调用都走这两张表,绕不开。

    本篇是 17-20 的物理基础——不知道栈帧长什么样,看 17 篇 ROP 链会完全懵;不知道 PLT/GOT,看 19 篇 RELRO 会完全懵。强烈建议自己写 § 2.1 那段 C,编译,gdb 看一遍栈——这一篇 80% 的价值在那一次实操里。
