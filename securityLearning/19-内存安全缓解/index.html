    内存安全缓解机制(ASLR / DEP / Canary / CFI / Shadow Stack)

    16-18 篇站在攻击者一侧讲了栈溢出、ROP、堆漏洞——读完会有种错觉:C/C++ 程序简直一捅就破。但你日常用的 Chrome、Linux 内核、PostgreSQL,每天被全世界几百万人盯着挖,为什么还没每天爆 RCE?答案不是"漏洞少了"——漏洞还是那么多,但利用一个漏洞从"30 行 shellcode"变成了"需要先链一个信息泄露 + 再链一个写原语 + 再绕三道缓解"。这一篇讲清楚:过去 25 年里防御方往二进制世界堆了哪些"缓解层",以及为什么每加一层,攻击者的工具箱就要重新设计一遍。

    一句话先记住:缓解机制(mitigation)不消灭漏洞,只提高利用成本——你给攻击者「能打的牌」一张张抽掉,直到他得先开两个漏洞才能换一次执行。没有任何单一机制是银弹,唯一的银弹是"全开 + 默认开 + 持续演进"。一个 2024 年的现代利用,平均要绕 3-5 层缓解,而 1999 年只要绕 0 层。

    一、缓解机制的哲学:不能消灭漏洞,但提高利用成本

    1.1 为什么不能"消灭"漏洞

    C/C++ 把内存管理交给程序员——只要语言本身不变,UAF、栈溢出、整数溢出就永远存在。CISA 和微软的统计反复证明:~70% 的 CVE 是内存安全漏洞,而且这个比例 20 年没怎么降。

    唯一根治的办法是换语言——Rust / Go / Swift 这些"内存安全语言"在语言层面就不让你写出 UAF(下一篇讲)。但你不可能一夜之间把 Linux 内核 / Chrome / Office 全部重写——这些代码加起来 1 亿行 C/C++,只能先加缓解,争取时间。

    1.2 经济学视角:把漏洞"涨价"

    1999 年:Aleph One《Smashing The Stack》
   一个栈溢出 → 30 行 shellcode → root shell
   利用成本:几小时

2024 年:Chrome V8 一个堆类型混淆
   要绕 ASLR + DEP + CFI + Shadow Stack
   要链 info leak + write primitive + JIT spray
   利用成本:几个月,Pwn2Own 一个漏洞奖金 50-150 万美元

    漏洞价格 = 利用成本 + 稀缺性。缓解机制就是把利用成本抬高,顺便把漏洞数量"稀释"——同样数量的 bug,可利用的占比从 80% 降到 10%。

    1.3 缓解的"打地鼠"模式

    历史上每一道缓解,都对应攻击者一种"前置技术"被废:

    DEP(2000s)    → shellcode 注入失效  → 催生 ROP
ASLR(2000s)   → 写死地址失效        → 催生 info leak
Canary(1998)  → 直接覆盖返回地址失效 → 催生 leak canary / SEH 攻击
CFI(2010s)    → 任意 ROP 链失效     → 催生 COOP / 数据导向攻击
CET/PAC(2020s)→ 硬件级 ROP 防御    → 催生跨进程 / JIT 攻击

    每次防御方加一层,攻击方就要发明一个新名词——这是过去 25 年内存安全领域的剧本。防御方的三个心智:(1)纵深防御——别指望一道防线扛住;(2)默认开启——编译器 / 链接器 / 内核默认就该开(主流发行版现在都是);(3)不要静态依赖——别假设"我开了 ASLR 就安全",ASLR 被绕一万次了。

    二、DEP / NX:让"数据区"不可执行

    2.1 漏洞:为什么以前 shellcode 这么好用

    经典栈溢出的攻击思路:(1)在栈上塞入 shellcode(机器码字节流);(2)溢出覆盖返回地址,指向栈上 shellcode;(3)ret 把 EIP 跳到栈上,执行 shellcode。前提是栈页有执行权限——1999 年之前的 x86 没有"页级别的执行位",只要可读就可执行。数据和代码权限不分,这是个架构级缺陷。

    2.2 利用:DEP / NX / W^X

    2003 年 AMD 在 x86_64 引入 NX bit(No-eXecute),Intel 后来跟进叫 XD bit。Windows 这个特性叫 DEP(Data Execution Prevention),Linux/BSD 叫 W^X(Writable XOR Executable)。

    每个内存页有三位权限:R / W / X
W^X 规则:同一个页不能同时 W + X
  栈、堆、.data、.bss:RW (不可执行)
  .text(代码段):RX (不可写)

    结果:你在栈上塞 shellcode 也跳过去了,CPU 一执行就抛 SIGSEGV。

    2.3 怎么绕:催生 ROP

    攻击者改用重用现有代码——既然 .text 段本来就可执行,我不注入新代码,只是用现有指令拼一条逻辑链:在 .text 段找各种以 ret 结尾的小片段(gadget),如 pop rdi ; ret / pop rsi ; ret / syscall ; ret,拼接成一条调用 execve("/bin/sh") 的链。

    这就是 ROP(Return-Oriented Programming),17 篇详细讲过。DEP 没消灭利用,只是逼着攻击者去 .text 段找零件。

    2.4 工程视角

    • gcc:链接时栈默认 noexecstack(-Wl,-z,noexecstack)
    • 检查二进制:readelf -l ./binary | grep GNU_STACK,看到 RW(没有 E)就对了
    • 坑:某些老代码用了 trampoline(libffi、嵌入式 GCC nested function)会强制要求 execstack,这一行就把整个进程的栈变成可执行的——审计时遇到 RWE 立刻警告

    三、ASLR:基址随机化,逼出"信息泄露"

    3.1 漏洞:写死地址是利用的前提

    老的利用代码长这样:

    buf = b"A" * 264 + p64(0x400b53)  # 直接写死 gadget 地址

    这个 0x400b53 是 gadget 在 .text 段里的位置——每次运行都一样。攻击者一旦逆向出地址,这个 exploit 可以无限重放。

    3.2 利用:Address Space Layout Randomization

    ASLR(地址空间布局随机化)在每次进程启动时随机化各段的基址:

    栈基址        随机偏移 28 bit
堆基址        随机偏移 24 bit
mmap 区(库) 随机偏移 28 bit
PIE 可执行文件 随机偏移 28 bit (要开 -fPIE)
vDSO          随机偏移

    结果:你写死 0x400b53 这种地址,下次运行根本不在那里。攻击者必须先泄露一个地址,计算偏移,再写利用——这就是 info leak primitive。

    3.3 PIE(Position Independent Executable)

    ASLR 只随机化动态加载部分(libc / heap / stack)。可执行文件本身的 .text 段位置如果不开 PIE,还是固定的——攻击者直接拿可执行文件里的 gadget 用。

    gcc 默认开 -fPIE  → .text 也随机
gcc -no-pie       → .text 固定(老二进制 / 嵌入式)

    checksec ./binary 输出 PIE enabled 才算开了。

    3.4 KASLR(内核版 ASLR)

    内核镜像加载时也随机化基址(CONFIG_RANDOMIZE_BASE=y),防止内核漏洞被直接利用。

    但 KASLR 历史上被各种侧信道(prefetch / TLB / Meltdown)穿透过无数次——所以又加了 KPTI(Kernel Page Table Isolation):用户态进程的页表里根本不映射内核地址,从根本上拒绝侧信道泄露。这是 2018 年 Meltdown 之后被强制加上的,代价是 syscall 慢 5-30%——你今天每一次 syscall 都在为 Meltdown 还利息。

    3.5 怎么绕

    • 信息泄露:从一个额外的漏洞(比如格式化字符串 %p,或者越界读)泄露一个 libc 地址,算出偏移就能定位整个 libc
    • 熵不足:32 位系统 ASLR 只有 16-24 位熵,爆破 1-2 小时能蒙到(64 位基本不可行)
    • fork 进程不重随机:子进程地址布局和父进程相同——nginx、Apache 这种 fork 模型,爆破一个子进程不会 crash 整个服务,可以暴力试探
    • 侧信道:Spectre / Meltdown / cache timing 在不读内存的前提下泄露布局

    ASLR 不是终点,是"提价"——它把"零成本利用"变成了"需要先找一个 leak"。

    四、Stack Canary:在返回地址前埋"金丝雀"

    4.1 漏洞:栈溢出的核心是覆盖返回地址

    栈帧布局(详见 16 篇):

    高地址
┌──────────────┐
│ 返回地址 RIP │ ← 攻击目标
├──────────────┤
│ 保存的 RBP   │
├──────────────┤
│ 局部变量     │
│ char buf[64] │ ← 溢出起点
└──────────────┘
低地址

    strcpy(buf, user_input) 越界写,先碰到 RBP,再碰到返回地址——攻击者只要把返回地址换成 gadget 地址就赢了。

    4.2 利用:在中间塞一只"金丝雀"

    StackGuard(1998,Crispin Cowan)/ SSP(Stack Smashing Protector,GCC -fstack-protector)的思路简单粗暴:

    函数 prologue:
   把一个随机 8 字节(canary)塞到 RBP 上方
函数 epilogue:
   ret 之前检查 canary 是否被改
   被改就调 __stack_chk_fail 直接 abort

    栈布局变成:

    ┌──────────────┐
│ 返回地址 RIP │
├──────────────┤
│ 保存的 RBP   │
├──────────────┤
│ CANARY (8B)  │ ← 哨兵
├──────────────┤
│ char buf[64] │
└──────────────┘

    要覆盖返回地址,必须先穿过 canary——而连续写就会改 canary,函数返回时一检查不对,进程被 abort。

    4.3 GCC 的几个档位

    选项 含义
    -fstack-protector 只保护有 char 数组 > 8 字节的函数
    -fstack-protector-strong 扩大到带数组、取地址局部变量的函数(推荐)
    -fstack-protector-all 所有函数都保护(性能损失)
    -fno-stack-protector 关闭(别选)

    Ubuntu / Debian / RHEL 默认都是 strong。Linux glibc 默认是 Random canary(每个进程启动时从 /dev/urandom 读 8 字节),存在 TLS 里,每个线程独立。第一字节通常固定是 \0 来阻断 strcpy 类按字符串拷贝越界。

    4.4 怎么绕

    • 泄露 canary:从一个独立的越界读漏洞泄露 canary 值,然后写利用时把 canary 原样写回去
    • 逐字节爆破(fork 模型):
    • 子进程继承父进程的 canary——一个字节一个字节试,对就不 crash,错就 crash
    • 64 位 canary 第一字节固定是 \0,实际只要爆破 7 字节 = 最多 7 × 256 = 1792 次
    • 长跑服务(nginx fork worker)可被这样穿透
    • 不经过 canary 的覆盖:比如改邻接的另一个局部变量(指针、长度),间接控制程序——canary 只保护"返回地址前的一段连续写",结构体里的指针不受保护
    • SEH 攻击(Windows):覆盖异常处理结构而不是返回地址,canary 在 ret 时才检查,主动触发异常就提前跳走了

    4.5 工程视角

    gcc -fstack-protector-strong

    检查:objdump 函数末尾应该有 call __stack_chk_fail,prologue 有 mov %fs:0x28,%rax(从 TLS 取 canary)。

    五、CFI:函数指针不能随便跳

    5.1 漏洞:间接跳转是 ROP 的根

    ROP 链的本质是滥用 ret / 间接 call / 间接 jmp——这些指令的目标是从内存里读出来的(返回地址在栈上,函数指针在堆/全局变量里)。攻击者只要能改这块内存,就能让 CPU 跳到任意地方。

    但合法的目标地址在编译时是有限的:

    typedef int (*Handler)(int);
Handler h = get_handler();
h(x);   // 间接调用

    h 的合法值只有几个已注册的 handler 地址,不是程序里随便哪个 .text 字节。

    5.2 利用:Control Flow Integrity

    CFI 的核心:在每次间接跳转 / 调用前,运行时检查目标是不是"合法目标"集合中的一个。

    常见实现:

    实现 厂商 思路
    Clang CFI LLVM 编译时按签名分桶,运行时检查目标是否在桶里
    Microsoft CFG(Control Flow Guard) Windows 维护一个全局 bitmap 记录"合法间接调用目标"
    Microsoft XFG(eXtended CFG) Windows 10+ CFG + 类型签名,粒度更细
    Intel CET IBT(Indirect Branch Tracking) Intel x86 硬件 间接跳转目标必须是 endbr64 指令,否则 #CP 异常
    kCFI / RFP Linux 内核 内核版 CFI 
    // 编译器在每次间接 call 前插桩:
if (!cfi_check(target_signature, target_address))
    abort();
call *target_address

    5.3 怎么绕

    • 同签名替换:CFI 只能保证"目标是同类型函数",但如果合法目标集里包含一个"半个 gadget"或者一个 system() 同签名的函数,直接跳过去就行——这就是 COOP(Counterfeit Object-Oriented Programming)
    • 不改控制流,改数据流:数据导向攻击(DOP)——不改返回地址,只改一个布尔变量、一个长度字段,让程序原本的合法流程做错事(is_admin = true)。CFI 完全不防这种
    • JIT spray:JavaScript V8 的 JIT 区可写可执行(它要生成代码),攻击者构造 JS 让 JIT 编译出"看起来合法但藏 gadget"的代码
    • 类型混淆:C++ 虚函数表攻击——把对象的 vtable 改成另一个类的 vtable,两个类签名兼容,CFI 检查会通过

    5.4 工程视角

    gcc / clang:-fcf-protection=full     # 启用 CET (IBT + Shadow Stack)
clang:-fsanitize=cfi -flto            # 启用 LLVM CFI(需要 LTO)

    注意 -fsanitize=cfi 要 LTO 才能工作——因为要看整个程序才知道函数签名集合。

    六、Shadow Stack:硬件版的返回地址保险柜

    6.1 Canary 的局限

    Canary 保护的是"返回地址前的一段连续覆盖",但:

    • 攻击者可以绕过中间直接写返回地址(任意写原语)
    • 可以leak canary
    • 函数返回的时候才检查(已经晚了,只能 abort,不能"修复")

    更彻底的思路:根本不让你改栈上的返回地址。

    6.2 Shadow Stack 是什么

    call 指令的本质:把返回地址压栈 + 跳转。Shadow Stack 改成:

    call:
  压栈 → 普通栈和影子栈各压一份
ret:
  从普通栈弹出 → 和影子栈对比 → 不一致就 #CP 异常

    影子栈在另一块专用内存里,用户代码不能直接写(只有 call/ret 指令隐式操作)——除非攻击者能 RCE 任意写到影子栈区域,否则栈上的返回地址再怎么被覆盖,ret 时和影子栈一对比就 abort。

    6.3 Intel CET / AMD Shadow Stack

    Intel CET(Control-flow Enforcement Technology)从 Tiger Lake(2020)开始硬件支持,包括两部分:

    • SS(Shadow Stack):前面讲的
    • IBT(Indirect Branch Tracking):间接跳转目标必须是 endbr64

    AMD 从 Zen 3(2020)起也有 Shadow Stack。

    操作系统侧:

    • Linux 6.6+ 支持 user shadow stack(CONFIG_X86_USER_SHADOW_STACK)
    • Windows 10 2004+ 的 "Hardware-enforced Stack Protection"
    • glibc 2.36+ 默认启用

    6.4 ARM 的对应方案:Pointer Authentication(PAC)

    ARM 走了不一样的路。PAC(ARMv8.3-A,2018)在指针的高位塞一段加密签名:

    64 位指针的高 16 位(虚拟地址用不到)放 PAC 签名
  签名 = HMAC(指针值, 上下文, 进程密钥)

call 时:    pacibsp  ; 给返回地址签名
ret 时:     autibsp  ; 验证签名,失败则指针变成无效地址,访问就 crash

    结果:你篡改了栈上的返回地址,但你不知道密钥,签不出合法 PAC——ret 时验证失败,CPU 跳到一个无效地址 crash。

    iOS 从 A12(2018)、macOS 从 M1(2020)起全面用 PAC。Linux ARM64 内核从 5.7 起支持。

    6.5 怎么绕

    • Shadow Stack:需要任意写到影子栈区域(它有自己的页权限,普通指针访问会 fault),门槛极高
    • PAC:
    • 签名密钥泄露 → 全部 PAC 失效
    • 签名 oracle:某个签了名的指针被泄露,和原指针一对算出 PAC → 但只对这一个指针有效
    • 指针重用(PACMAN 攻击,2022):用侧信道猜 PAC,Apple M1 上演示过

    这两个机制大幅提高了硬件级 ROP 的成本——2020 年之后 iOS 利用链复杂度爆炸式增长。

    七、MTE:每个内存块都带"颜色标签"

    ARMv8.5 的 MTE(Memory Tagging Extension)思路:每 16 字节内存关联一个 4-bit tag(颜色),每个指针的高 4 位也放一个 tag,访问时指针 tag 和内存 tag 必须匹配,否则 trap。

    ptr = malloc(64);    // 内存染 0x3,指针高位也是 0x3
free(ptr);           // 内存重染成 0x7,指针 ptr 还是 0x3
*ptr = 1;            // 0x3 != 0x7 → trap (UAF 被抓)

    对越界、UAF、double-free 都生效,硬件加速性能损失约 5%。Pixel 8(2023)是第一款消费级 MTE 设备,Android 14 系统进程默认开。但 4-bit tag 只有 16 种颜色,1/16 概率撞色——是概率性防御,不是绝对的。

    MTE 现在只在 ARM 上有,x86 没有对等机制。短期内 PC / 服务器主要靠 ASLR + CFI + Shadow Stack;移动 / IoT 走 MTE + PAC。

    八、各机制对比总表

    缓解机制 防什么 实现层 性能开销 怎么绕(典型路径)
    DEP / NX / W^X 栈/堆上执行 shellcode CPU 页表 + OS 几乎 0 ROP / JOP / 跳到 mprotect
    ASLR 写死地址利用 OS 加载器 < 1% 信息泄露 / 熵不足爆破 / 侧信道
    PIE .text 段位置固定 编译器 + 加载器 1-3% 同 ASLR(需要泄露一个 .text 内地址)
    KASLR + KPTI 内核地址泄露 内核 KPTI 5-30% syscall Spectre/Meltdown / 内核 info leak
    Stack Canary 栈溢出覆盖返回地址 编译器 1-3% leak canary / fork 爆破 / 邻接变量改写 / SEH
    CFI(软件) 任意 ROP / 虚表攻击 编译器 + LTO 1-5% COOP / DOP / 同签名替换 / JIT
    Intel CET IBT 跳到非 endbr 指令 CPU + OS < 1% 跳到合法 endbr 后做事
    Shadow Stack 返回地址被改 CPU + OS 1-3% 任意写影子栈页(极难)
    ARM PAC 指针被篡改 CPU < 1% 密钥泄露 / 签名 oracle / PACMAN 侧信道
    ARM MTE UAF / 越界 / double-free CPU 5% 左右 1/16 概率撞色 / leak tag
    RELRO(Full) GOT 改写 链接器 启动慢一点 改其他可写函数指针
    FORTIFY_SOURCE strcpy/memcpy 越界 编译器 + glibc 极小 用没有 fortify 包装的等价函数

    记牢一句:每一行都不是"开了就安全",只是抬高一档攻击成本。

    九、组合起来才是"现代二进制防线"

    单看任何一道都能被绕,组合起来攻击成本指数上升:

    攻击者要在用户态打一个现代 Chrome / iOS:
  1. 找一个内存漏洞(堆 UAF)
  2. 找一个 info leak 漏洞 → 绕 ASLR/PIE
  3. 构造 ROP 链 → 绕 DEP
  4. gadget 必须 endbr64 开头 → 绕 IBT
  5. ret 还要过 Shadow Stack → 改成 JOP (jump-oriented)
  6. 跳转目标符合 CFI 签名 → 找同签名 victim
  7. ARM 上指针还有 PAC → 偷一个 PAC oracle
  8. 拿到 RCE 还要绕 sandbox → 再链一个内核漏洞
  9. 内核又有 KASLR / KPTI / SMEP / SMAP / kCFI / IBT

    这就是为什么现代 0day 单链 200-300 万美元(Zerodium / Crowdfense 报价)——不是没漏洞,是把漏洞链成完整利用太贵了。

    十一、心智总结:四段式

    为什么:C/C++ 漏洞不可能消灭,只能用一层层缓解把利用成本抬到经济不划算。每一道缓解都有它自己的绕过路径,单点不可靠,组合才有意义。

    怎么用:(1)默认开所有编译加固选项,checksec 全绿才能发布;(2)纵深堆叠 ASLR + DEP + Canary + CFI + Shadow Stack,任一被绕了还有下一道;(3)持续演进,跟着 CET / PAC / MTE 上车新硬件。

    坑:(1)以为"开了 ASLR 就安全"——info leak 一次就废;(2)以为 canary 是绝对的——fork 爆破 / 邻接变量改写都能绕;(3)CFI 不防数据导向攻击(DOP)——is_admin = true 走的是合法控制流;(4)-fstack-protector-all 性能损失大,生产用 strong 就够。

    心智:漏洞经济学——你的目标不是"我的代码绝对安全"(不可能),而是"利用我的代码贵到不值得"。一个 30 行 PoC 能打穿的程序和一个需要 3 个 0day + 半年研究才能打穿的程序,对脚本小子是同等的不安全,对国家级攻击者价值差 1000 倍。绝大多数项目要防的就是脚本小子——加固选项全开,90% 的潜在攻击者直接劝退。

    十二、踩坑提醒

    1. 以为 NX 让 shellcode 死了——ROP 用现有 .text 代码做坏事,不需要新代码可执行
    2. ASLR 熵不够——32 位系统 16-24 位熵能爆破,64 位才安全
    3. fork 不重新随机——子进程继承父进程地址布局 + canary,长跑服务能被慢慢摸出来
    4. canary 第一字节是 \0——实际只要爆破 7 字节(逐字节 7×256 = 1792 次)
    5. CFI 不防 DOP——改一个布尔变量提权,完全不经过控制流
    6. Shadow Stack 不防 JOP——基于 jmp 而不是 call/ret 的链,影子栈管不到
    7. PAC 密钥泄露 = 全废;MTE 4-bit 撞色概率 1/16——硬件机制也只是"提价"
    8. JIT 区是缓解机制盲区——V8 / SpiderMonkey / .NET CLR 必须 W^X 严格切换,不然 JIT spray 直接绕一切
    9. 嵌入式 / 老二进制经常不能加固——execstack 强制开、不支持 PIE,这些场景要靠隔离(沙箱 / 进程降权)补
    10. 缓解机制本身有漏洞——2018 stack clash、2022 PACMAN、2024 多个 CET 旁路;别假设硬件机制无懈可击
