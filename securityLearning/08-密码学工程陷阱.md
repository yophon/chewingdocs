# 密码学工程陷阱:时序攻击 / 填充 oracle / 弱随机 / KMS 与 HSM

前面四篇讲的是「密码学原语本身」——AES 怎么用、RSA 怎么签、椭圆曲线为什么强。**但密码学事故里 95% 的根因不是「算法被破了」,而是「正确的算法被工程师用错了」**。AES-GCM 是 AEAD,理论上无懈可击——但你用 `nonce = 0` 加密两条消息,密钥就泄露;ECDSA 数学上很美——但 Sony 在每条 PS3 签名里复用同一个随机数 `k`,整个 PS3 的根私钥被一个高中生算出来。**这一篇讲的不是"密码学被攻破"**,而是「**工程师把锋利的工具用反**」——四类经典陷阱:**时序攻击、填充 oracle、弱随机、密钥管理崩塌**。看完之后,你看自己的代码会发现:每一个 `==` 比较 HMAC、每一个 `random.randint` 生成 token、每一个写死在 `config.yaml` 里的密钥——都是悬在头顶的 CVE。

> 一句话先记住:**「不要自己造密码学」的下一条铁律是「也不要自己组装密码学」**——AES + HMAC 你都用对了,但比较顺序错了,照样泄露密钥。**密码学工程的本质是「把所有"看起来无关的细节"也变成不可观测**——比较时间、错误消息、CPU 缓存、电源功耗,任何一个能被外部观察到的差异,都可能在数学上等价于一次密钥泄露。

---

## 一、为什么这一篇必须单独成篇

```
AES-256-GCM:暴力穷举要 10^60 年
HMAC-SHA256:数学上没有比暴力更好的攻击
ECDSA-P256:经典计算机解不开椭圆曲线离散对数
```

**但用上面这些"完美原语"搭出来的系统,每年都有大量 CVE**。原因不在算法,在**工程实现**:

- **时序攻击**:运行时间和密钥相关 → 攻击者测量时间反推密钥
- **填充 oracle**:错误码透露解密成败 → 攻击者一字节一字节问出明文
- **弱随机**:看起来是随机数,其实可预测 → 整个系统的密钥都是假的
- **密钥泄露**:密钥写在代码 / 日志 / 异常栈里 → 算法再强也没用

**这四类问题的共同点**:**不能通过"读密码学论文"学会防御**——它们是软件工程层面的坑,只有写过代码、踩过坑、读过事故复盘才知道。**这一篇围绕五个真实事故**:PHP timing attack、POODLE/Lucky 13、Debian OpenSSL 2008、Sony PS3 ECDSA、Heartbleed——**没有一个是算法被破,全是「用对了原语,忘了某个细节」**。

## 二、时序攻击:为什么 `==` 是密码学反模式

### 2.1 漏洞原因:比较操作不是常数时间

C 的 `memcmp` 和大多数语言的 `==` 字符串比较,**在第一个不匹配的字节就提前返回**——这是性能优化,但是密码学灾难。**比较耗时和「前几个字节匹配多少」线性相关**:

```
"abcdef" vs "xbcdef" → 1 次比较返回(很快)
"abcdef" vs "abcxef" → 4 次比较返回(稍慢)
"abcdef" vs "abcdef" → 6 次比较返回(最慢)
```

**单次差异只有几纳秒**——但攻击者可以发起百万次请求,**用统计平均把噪声消掉**。

### 2.2 利用链:逐字节试出 HMAC 签名

假设你的 API 用 HMAC 校验请求:

```python
def verify_request(signature, expected):
    if signature == expected:   # ← 关键漏洞
        return True
    return False
```

**攻击者一字节一字节试**:

```
1. 发 signature = "00000...0" → 测量响应时间 t0
2. 改 signature = "10000...0" → 测时间 t1
   ...
3. 测 256 个值,第一个字节响应时间最长的就是正确字节
4. 锁定第一字节,继续测第二字节
5. 32 字节的 HMAC 只需 32 × 256 = 8192 次请求(理论值)
```

**真实演示**:2007 年 Crosby & Wallach 论文证明可以**在跨 Internet 的场景下,3 微秒级的时间差被可靠测量**;同年 Nate Lawson 在 Google Keyczar 里发现这类漏洞。

### 2.3 实战示例:错误代码 vs 正确代码

| 反模式(漏洞) | 正确写法 |
| --- | --- |
| `if user_token == server_token:` | `hmac.compare_digest(user_token, server_token)` |
| `if signature.equals(expected)` (Java) | `MessageDigest.isEqual(signature, expected)` |
| `if (strcmp(a, b) == 0)` (C) | `CRYPTO_memcmp(a, b, len)` (OpenSSL) |
| `if Buffer.compare(a, b) === 0` (Node) | `crypto.timingSafeEqual(a, b)` |
| `if a == b` 任何字段含密钥/HMAC/Token | `subtle.ConstantTimeCompare(a, b)` (Go) |

**常数时间比较的标准实现**(教学用,生产请用库函数):

```python
def constant_time_compare(a: bytes, b: bytes) -> bool:
    if len(a) != len(b):
        return False
    result = 0
    for x, y in zip(a, b):
        result |= x ^ y   # 永远比完所有字节
    return result == 0
```

**关键**:`|=` 不会因为某一位为 1 就提前结束——必须遍历全部字节。

### 2.4 修复:把"密码学相关比较"全部下沉到库

**短期**:把代码里所有 `==` 比较 token / hash / signature 的地方,替换成语言自带的常数时间比较函数。

**长期**:**任何与密钥相关的数据都不要"直接比较"**——而是用 HMAC 校验框架的 `verify()` 接口(内部就是常数时间)。

### 2.5 纵深防御

- **代码评审清单**:`==` / `equals` / `compare` 后面跟 `signature` / `hmac` / `token` / `mac` / `tag` 都打回
- **静态扫描**:semgrep / CodeQL 都有这条规则
- **WAF / 速率限制**:即使代码漏了,限制 100 req/s 也让攻击者攒不到样本
- **业务层 nonce / timestamp**:token 一次性使用,即使破出来也用不了

## 三、填充 oracle:错误信息为什么是侧信道

### 3.1 漏洞原因:CBC 模式 + 区分"填充错误" vs "认证错误"

**CBC 加密** 需要明文长度是块的整数倍——不够要填充(PKCS#7):

```
明文 "HELLO" (5 字节,要补 3 字节)
→ 填充后 "HELLO\x03\x03\x03"
```

**解密时**:服务端先解密,再校验填充。**如果填充错误,服务端会返回不同的错误**:

```
密文格式错 → "decryption failed"
填充错       → "padding error"
MAC 错       → "authentication failed"
解密成功但业务错 → "invalid request"
```

**只要这四种情况能被区分(不同错误码、不同耗时、不同响应长度)**,攻击者就有了**一个 oracle**——能问服务器「这个密文的填充对不对」。

### 3.2 利用链:POODLE 的核心思想

```
攻击者拿到一段加密的 cookie 密文 C
想知道明文最后一个字节是什么

1. 构造一个"修改过最后一块"的密文 C'
2. 发给服务器
3. 服务器解密:
   - 如果填充碰巧合法(\x01 结尾) → 返回 MAC 错误
   - 如果填充非法 → 返回填充错误
4. 攻击者从错误类型推出"最后一块解密后是否以 \x01 结尾"
5. 256 次尝试覆盖一个字节
6. 一段 cookie 几百字节 → 几万次请求
```

**真实历史**:

- **POODLE (2014, CVE-2014-3566)**:SSL 3.0 的 CBC 模式,可以解密用户 cookie。Google 团队披露,导致全互联网紧急下线 SSL 3.0。
- **Lucky 13 (2013, CVE-2013-0169)**:TLS 1.x 的 CBC 模式,即使错误码相同,**MAC 验证的耗时和填充长度相关**,变成时序版本的填充 oracle。
- **BEAST、CRIME、BREACH、DROWN**:都是 TLS/SSL 上的变体,推动了 TLS 1.3 强制使用 AEAD 抛弃 CBC。

### 3.3 实战示例:错误代码 vs 正确代码

| 反模式(漏洞) | 正确写法 |
| --- | --- |
| `AES-CBC + 自己拼 HMAC`(MAC-then-Encrypt 或顺序错) | 用 **AES-GCM** 或 **ChaCha20-Poly1305** 这种 AEAD |
| 区分"填充错"/"MAC 错"的错误码 | **永远返回同一个"解密失败"**,**且耗时相同** |
| 解密失败立刻返回 | 解密失败也走完 MAC 验证的全部流程 |
| 自己用 CBC 实现 cookie 加密 | 用框架的 `secure cookie` / `signed session` |

**关键代码对比**:

```python
# 反模式:CBC + 错误信息泄露
def decrypt(ciphertext):
    plaintext = aes_cbc_decrypt(key, ciphertext)
    if not check_padding(plaintext):
        raise PaddingError("bad padding")    # ← oracle
    mac = plaintext[-32:]
    if not verify_mac(plaintext[:-32], mac):
        raise MacError("bad mac")            # ← oracle
    return plaintext[:-32]

# 正确:AEAD 一步到位
def decrypt(ciphertext, nonce, aad):
    try:
        return aesgcm.decrypt(nonce, ciphertext, aad)
    except InvalidTag:
        raise DecryptError("decryption failed")  # ← 唯一错误
```

### 3.4 修复:抛弃 CBC,拥抱 AEAD

**短期**:任何使用 `AES-CBC` 或 `AES-CTR + HMAC` 自己拼的代码,全部换成 `AES-GCM` 或 `ChaCha20-Poly1305`。

**长期**:**密码学决策权下沉到库**——上层代码只调用 `seal(plaintext) / open(ciphertext)`,不接触模式、IV、MAC 顺序这些细节。例如:

- Go 的 `crypto/cipher.AEAD`
- Python 的 `cryptography.hazmat.primitives.aead.AESGCM`
- libsodium 的 `crypto_secretbox`(直接帮你选好 XSalsa20-Poly1305)

### 3.5 纵深防御

- **协议层**:服务端禁用 TLS 1.0 / 1.1,只允许 TLS 1.3(强制 AEAD)
- **应用层**:错误响应**永远统一**——`"decryption failed"`,**且响应时间归一化**
- **WAF**:同一密文短时间内大量解密失败 → 告警
- **业务层**:一次性 token + 严格过期时间,把"利用窗口"压到最小

## 四、弱随机数:整个密码学的地基崩塌

### 4.1 漏洞原因:`random` 不是「随机」

几乎所有语言都有两套随机数 API,**新手很容易用错**:

| 语言 | 不安全(伪随机) | 安全(密码学随机) |
| --- | --- | --- |
| Python | `random.randint()` / `random.random()` | `secrets.token_bytes()` / `os.urandom()` |
| JavaScript | `Math.random()` | `crypto.getRandomValues()` / `crypto.randomBytes()` |
| Java | `java.util.Random` | `java.security.SecureRandom` |
| Go | `math/rand` | `crypto/rand` |
| C | `rand()` | `/dev/urandom` 或 `getrandom(2)` |

**伪随机(PRNG)的特征**:从一个种子出发,可以预测下一个数。**密码学随机(CSPRNG)的特征**:即使知道前面所有的输出,也猜不出下一个。**两者性能差几乎可以忽略**——但**用错了直接灾难**。

### 4.2 利用链:两个经典事故

**事故 1:Debian OpenSSL 2008 (CVE-2008-0166)**

```
2006 年,某 Debian 维护者用 Valgrind 跑 OpenSSL,报"使用了未初始化内存"
他注释掉了两行 OpenSSL 的代码——其中一行是用进程内存作为熵源
结果:OpenSSL 的熵池只剩 PID 这一个变量(范围 1-32768)
影响:2006-2008 两年间,所有 Debian/Ubuntu 上生成的
      SSH 密钥、SSL 证书、PGP 密钥只有 32768 种可能
后果:全球所有受影响的密钥必须重新签发
```

**这是密码学历史上最贵的一次「好心办坏事」**——一个看似合理的"修复警告"操作,把整个发行版的密码学地基掀了。

**事故 2:Sony PS3 ECDSA (2010)**

```
ECDSA 签名要求每次签名生成新的随机数 k
Sony 的 PS3 固件签名代码里——k 是一个常量,从来没换过
GeoHot 等人发现两条签名的 k 相同
用初中代数解二元一次方程,根私钥直接算出来
影响:任意 PS3 系统更新可以被攻击者伪造,Sony 整个 DRM 体系崩塌
```

**ECDSA 复用 k 的数学**(教学,不武器化):两条复用 k 的签名 (r,s1),(r,s2),消息哈希 z1,z2 → `k = (z1-z2)/(s1-s2) mod n` → 代回算出私钥 d。**30 行 Python 出私钥**。

### 4.3 实战示例:错误代码 vs 正确代码

| 场景 | 反模式 | 正确写法 |
| --- | --- | --- |
| 生成 session token | `''.join(random.choices(string.ascii_letters, k=32))` | `secrets.token_urlsafe(32)` |
| 生成密码重置 token | `Math.random().toString(36)` | `crypto.randomBytes(32).toString('hex')` |
| 生成 AES key / IV | `new Random().nextBytes(key)` | `SecureRandom.getInstanceStrong().nextBytes(key)` |
| UUID 作为 token | `uuid.uuid1()`(基于时间+MAC) | `uuid.uuid4()`(基于 CSPRNG) |
| 自己实现 PRNG | **任何「自己写的随机」** | 永远用系统 CSPRNG |

**关键认知**:

```
- random / Math.random:为模拟、游戏、压测设计的,可预测
- CSPRNG (secrets / crypto / SecureRandom):为密码学设计的
- 「我只是生成个 token」也必须用 CSPRNG——token 一旦能猜,等于密码
```

### 4.4 修复:四个铁律

1. **生成密钥、token、nonce、IV、salt——全部用 CSPRNG,没有例外**
2. **签名/加密里的"每次必新"随机数,严格不复用**(ECDSA 的 k、AES-GCM 的 nonce)
3. **不在低熵环境生成密钥**:刚启动的容器、IoT 设备熵池可能为空,用 `getrandom(2)` 而不是阻塞读 `/dev/random`
4. **代码评审看到 `random` 模块用在和"安全相关"字段就打回**

### 4.5 纵深防御

- **静态扫描**:bandit / semgrep 都有 `B311 random-used-in-crypto` 规则
- **运行时检测**:观察生成的 token 熵——熵低于 80 bit 告警
- **从源头规避**:把"token 生成"封装在统一的 `secrets.py` 工具里,业务代码禁止直接 import `random`

## 五、密钥管理:KMS / HSM / 信封加密

**前面三类陷阱是"用错原语"**,这一节讲「**密钥本身的工程**」——密钥从生成到销毁的全生命周期,**95% 的真实泄露事故都发生在这一层**。

### 5.1 密钥不能在哪里

```
1. 代码仓库(.py / .yaml / .env)
2. 容器镜像(Dockerfile / 镜像 layer)
3. CI/CD 日志(出错时打印环境变量)
4. 异常堆栈(把 config 整体序列化扔进 Sentry)
5. APM / 监控(把请求 body 含密钥的字段采样上传)
6. 测试代码("AKIAxxx" 假的也别写,会被扫到)
7. 日志(永远不打 Authorization header)
8. 客户端可访问处(前端代码、移动 App 二进制)
```

**GitHub 每年扫出几百万次密钥泄露**——开发者把 AWS Access Key 提交到公开仓库,平均**4 分钟内**就被自动扫描的攻击者捡走。

### 5.2 密钥管理的层次

```
应用代码 ← 最不该碰密钥
   ↓
环境变量 / 配置中心 ← 比代码好,但还在进程里
   ↓
Secrets Manager(Vault / AWS SM / GCP SM)
   ← 集中存储,审计,自动轮转
   ↓
KMS (Key Management Service)
   ← 密钥从不出系统;只暴露 encrypt/decrypt API
   ↓
HSM (Hardware Security Module)
   ← 物理隔离的硬件;密钥连 OS 都看不见
```

**层次的核心思想**:**密钥越往下越少有人能"看到原文"**——KMS 不返回明文密钥,你只能调用 `Encrypt(plaintext) → ciphertext` 和 `Decrypt(ciphertext) → plaintext` 这两个 API。**应用程序拿到的永远是密文**,数学上即使应用被攻破,密钥也不会泄露。

### 5.3 信封加密(Envelope Encryption):KMS 的标准模式

直接用 KMS 加密 1GB 文件性能很差(每次都跨网络)——所以引入**两层密钥**:

```
KEK (Key Encryption Key):永远在 KMS 里,数量少,稳定
   ↓ 加密
DEK (Data Encryption Key):每次数据一个,本地用完即扔
   ↓ 加密
你的数据
```

**典型流程**:

```
加密:
1. 应用调 KMS.GenerateDataKey() → (plaintext_DEK, encrypted_DEK)
2. 应用用 plaintext_DEK 本地 AES-GCM 加密文件
3. (encrypted_DEK, ciphertext) 一起存到 S3
4. 内存里 plaintext_DEK 立刻清零

解密:
1. 从 S3 读 (encrypted_DEK, ciphertext)
2. KMS.Decrypt(encrypted_DEK) → plaintext_DEK
3. 解密文件,用完清零
```

**为什么这样设计**:KMS 只处理 32 字节级数据,QPS 撑得住;DEK 一次性使用,内存被 dump 只泄露一个文件;KEK 在 KMS 里轮转,所有 encrypted_DEK 自动跟着轮转;KMS 日志记录每次 Decrypt 调用,异常使用模式立刻能查。

### 5.4 密钥轮转

**铁律**:**所有密钥都必须有"轮转计划"**——即使没泄露也要定期换,**密钥用得越久,泄露的可能性累积越大**。

| 密钥类型 | 推荐轮转周期 | 谁负责 |
| --- | --- | --- |
| TLS 证书 | 90 天(Let's Encrypt 默认) | ACME 自动 |
| API Key / 数据库密码 | 30-90 天 | Secrets Manager 自动 |
| KMS 主密钥 (KEK) | 1 年 | KMS 自动 |
| 用户密码 | 不强制周期性轮转(NIST 2017 新建议) | 用户 |
| ECDSA / RSA 长期密钥 | 1-5 年 | 运维 |

**轮转的工程挑战**:不是"换一个值"这么简单——**新旧密钥要有重叠期**,旧密文用旧密钥解密,新密文用新密钥加密。这是为什么信封加密里 `encrypted_DEK` 要带「KEK 版本号」。

### 5.5 HSM vs KMS:什么时候需要 HSM

```
KMS(软件实现 + TPM 辅助):适合 99% 的场景
   - AWS KMS / GCP KMS / Azure Key Vault,FIPS 140-2 Level 2-3

HSM(专用密码学硬件):金融、CA、政府强合规
   - AWS CloudHSM / Thales Luna / Utimaco
   - 物理防拆,密钥连内存都不出,贵 10-100 倍
   - FIPS 140-2 Level 3-4 / Common Criteria EAL4+
```

**判断标准**:**「监管要求 FIPS 140-2 Level 3 以上」或「单密钥泄露损失 > 千万美元」才考虑 HSM**——PKI 根 CA、银行核心、加密货币交易所冷钱包。普通业务用 KMS 足够。

## 六、侧信道:应用工程师能做什么

**侧信道(Side Channel)** = 算法本身不暴露,但实现过程中泄露的「**间接观测量**」:

```
- 时间侧信道:操作耗时和密钥相关(第二节已讲)
- 缓存侧信道:CPU L1/L2/L3 缓存命中模式泄露密钥
- 功耗 / 电磁 / 声学侧信道:智能卡、嵌入式设备的物理特征
- 故障注入:电压/温度毛刺让 CPU 跳过验证
```

**对应用工程师**:**只有时间侧信道是日常要处理的**——其他几个属于硬件安全 / 嵌入式领域,应用层基本无能为力。但缓存侧信道(Spectre / Meltdown 2018)的余波依然在:

- AES 软件实现查 S-box 表,不同密钥导致不同 cache line 命中,攻击者测自己的内存访问时间能反推 victim 进程的密钥

**应对**:

1. **用 AES-NI 硬件指令**——现代 OpenSSL / BoringSSL 默认走 CPU 原生指令,常数时间,不查表
2. **关键密码学操作进程隔离**——KMS 和业务进程分开
3. **及时打 CPU 微码补丁**——确保 base image 是最新的

**一个必须知道的事实**:「常数时间」是工程纪律,不是数学定理——你用 Python 写的「常数时间比较」可能被 JIT 优化掉,**真正的常数时间实现要用汇编 / intrinsics**。所以铁律是:**应用代码永远不要自己实现"常数时间的密码学操作"**——一律调用 `libsodium` / OpenSSL / `golang.org/x/crypto` 这种被严格审计过的库。

## 七、两个真实事故的工程教训

### 7.1 Heartbleed(2014, CVE-2014-0160)

**漏洞本质**:OpenSSL 实现 TLS heartbeat 扩展时,**信任了客户端发来的长度字段,没做范围检查**:

```c
// 简化版漏洞代码
struct heartbeat_msg {
    uint16_t length;   // 客户端声称的载荷长度
    char payload[];    // 实际载荷
};

void handle_heartbeat(heartbeat_msg *msg) {
    char response[msg->length];
    memcpy(response, msg->payload, msg->length);  // ← 漏洞
    send(response, msg->length);
}
```

**攻击者发**:length = 65535,但实际 payload 只有 1 字节——服务器把**自己进程内存里挨着 payload 的 65534 字节**也复制了发回去。**这里面可能有 TLS 私钥、用户密码、session token**。

**工程教训**:

1. **不要相信任何外部输入的"长度字段"**——必须独立校验
2. **C/C++ 的内存安全风险是密码学库最大的隐患**——这是 BoringSSL / LibreSSL / Rustls 兴起的原因
3. **「heartbeat 这种小特性」也能拖垮整个系统**——评估每个扩展的攻击面
4. **修复后必须假设密钥已泄露**——所有受影响证书全部重发

### 7.2 Cloudflare 的 KMS 工程实践

Cloudflare 给数百万客户的 TLS 私钥提供托管——他们的工程做法值得抄:

1. **Keyless SSL**:客户私钥**根本不上传到 Cloudflare**——客户在自己的 HSM 里;边缘节点 TLS 握手时,把需要签名的数据发回客户机房,签好再传回来。**Cloudflare 自己被攻破都不会泄露客户私钥**
2. **GeoKey**:边缘节点不持久化任何客户私钥,需要时从中心 KMS 拉,内存里用,重启即失效
3. **审计每一次密钥使用**:所有 sign 操作打日志,异常模式(QPS 暴涨、来自非常规节点)立刻告警

**核心理念**:**「最好的密钥管理是密钥根本不在我这里」**——能把密钥控制权留给客户的,绝不自己保管。

## 八、代码评审密码学红线 checklist

每次代码评审看到下面这些**直接打回**:

**比较与时序**
- [ ] `==` / `equals` / `compare` 比较 `signature` / `hmac` / `token` / `mac` / `tag` / `password_hash` / `mfa_code`
- [ ] "找到错误就 break/return"的密码学校验循环
- [ ] 先验证长度再验证内容,且两步错误码 / 耗时不同的解密流程

**加密模式**
- [ ] AES-ECB(永远不允许)
- [ ] AES-CBC + 手写 HMAC(改用 AES-GCM)
- [ ] AES-CTR 不带 MAC
- [ ] 同一 (key, nonce/IV) 加密两条不同消息(AES-GCM 致命)
- [ ] DES / 3DES / RC4 / MD5 签名 / SHA-1 签名

**随机数**
- [ ] `random` / `Math.random` / `java.util.Random` / `math/rand` 用在生成密钥/token/nonce/salt
- [ ] `uuid1`(基于时间+MAC)用作 token
- [ ] 自己实现的「随机数生成器」(任何场景)
- [ ] ECDSA 签名代码里 k 没有每次重新生成

**密钥存储**
- [ ] 密钥 / API Key / 数据库密码 出现在代码仓库
- [ ] 密钥写在 Dockerfile / hard-code(允许环境变量注入,不允许写死)
- [ ] 异常堆栈、日志、APM 里可能打印密钥字段
- [ ] 没有轮转计划的长期密钥

**协议**
- [ ] TLS 1.0 / 1.1 / SSL 任何版本被显式允许
- [ ] 证书验证被禁用(`verify=False` / `InsecureSkipVerify` / `NODE_TLS_REJECT_UNAUTHORIZED=0`)
- [ ] 生产环境自签证书
- [ ] JWT 算法允许 `none`(默认黑名单不是白名单)

**密码与认证**
- [ ] 密码用 MD5 / SHA-1 / SHA-256 裸 hash 存储(必须 bcrypt / scrypt / argon2)
- [ ] 自己拼 salt + hash(用框架的 password_hasher)
- [ ] MFA 验证码比较用 `==`
- [ ] 找回密码 token 用顺序 ID / 时间戳

**错误处理**
- [ ] 解密失败返回不同错误码(必须统一为 "decryption failed")
- [ ] 解密失败的代码路径耗时不同
- [ ] 把密钥放进了异常 message

## 九、踩坑提醒(密码学工程总结版)

1. **以为「用了 AES」就安全**——模式、IV、MAC、密钥来源任何一个错都白搭
2. **以为「我加了 HMAC」就抗篡改**——比较顺序错(MAC-then-Encrypt)依然漏
3. **以为「token 很长」就难猜**——`random.random` 几行代码就能枚举
4. **以为「错误信息只有内部看到」**——错误码 / 响应时间都是 oracle
5. **以为「密钥放在 env var 就行」**——容器 inspect / 进程 ps 都能看到
6. **以为「KMS 加密一次就够」**——没有信封加密,QPS 撑不住
7. **以为「轮转密钥太麻烦先不做」**——直到泄露才知道「无法轮转的密钥 = 永久债务」
8. **以为「我读过密码学论文,可以自己实现」**——99% 的自实现都有侧信道,**永远调库**
9. **以为「修了 CVE 就完了」**——必须假设密钥已泄露,全量轮转 + 重新签发
10. **以为「我们规模小,不会被针对」**——自动化扫描器对所有暴露在公网的服务一视同仁
