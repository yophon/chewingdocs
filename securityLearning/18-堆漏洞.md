# 堆漏洞:use-after-free / double free / heap spraying

讲堆漏洞之前先把一个误区杀掉:「栈漏洞我懂了,堆不就是 `malloc/free` 嘛,无非把栈帧换成堆块」——**完全不是**。栈漏洞是「一条线性带子被踩穿」,堆漏洞是「一张到处是指针的网被剪断」。17 篇讲了栈溢出 / ROP——栈的布局可预测、Canary 一加就基本封住返回地址改写;**而堆从 2010 年代开始就成了浏览器 / 内核 0day 的主战场**,Chrome / Safari / iOS 每年的 0day 报告里 70-80% 是 UAF。osLearning/08-09 讲过 `malloc` / glibc 分配器的内部结构,**这一篇换攻击者视角看同样的数据结构**——为什么 tcache 是天堂、为什么 free 一个指针没把它置空是 21 世纪最贵的 NULL、为什么"加一道虚函数表"反而开了一扇大门。

> 一句话先记住:**栈漏洞是改控制流,堆漏洞是改"数据里的控制流"**——堆上躺着的全是指针:vtable、函数指针、free chunk 的 fd/bk 链表指针、SMART POINTER 的 vtable、JS 引擎对象的 Map 指针。**攻击者不需要改返回地址,只要让你的程序"调用一个你以为是合法对象、但其实是我控制的内存"**,控制流就到手了。**堆漏洞的核心攻击姿势就一句话:让攻击者数据出现在防御者指针解引用的位置**——UAF、Double Free、堆溢出全是这一句的变体。

## 一、堆 vs 栈:攻击面为什么完全不同

### 1.1 布局可预测性:堆是攻击者的劣势

栈的布局是**确定的**——同一个二进制、同一段代码,函数 A 调用函数 B,B 的栈帧永远在 A 下面 32 字节;Canary、保存的 rbp、返回地址,**位置稳如老狗**。栈溢出的利用脚本可以写死偏移量,跑十次成功十次。

堆完全相反:

```
malloc(100) 返回什么地址?
   依赖之前 N 次 malloc/free 的历史
   依赖 glibc 版本(2.27 / 2.31 / 2.34 行为不同)
   依赖 ASLR(每次进程启动堆基址都不一样)
   依赖线程(每个线程有自己的 arena)
   依赖 size(<= 64B 进 tcache,大块进 mmap)
```

**所以堆漏洞利用的第一步不是"打",是"整理"**——攻击者要先把堆"按摩"(heap massaging / heap feng shui)到一个可预测的形态,才能开始真正的攻击。这是一项手艺活,也是堆 exploit 比栈 exploit 难一个数量级的原因。

### 1.2 攻击面广度:堆是攻击者的优势

栈上能改的就那几样:局部变量、保存寄存器、返回地址。**堆上躺着的全是指针**:

```
堆里随便摸一把,大概率摸到:
   ├── C++ 对象的 vtable 指针(改了 → 接管所有虚函数调用)
   ├── 回调函数指针(改了 → 接管某个事件)
   ├── 字符串 / buffer 的指向(改了 → 任意读写原语)
   ├── 链表 / 树的 next/prev(改了 → 任意地址写)
   ├── glibc free chunk 的 fd/bk(改了 → unlink / tcache 攻击)
   ├── JS 引擎对象的 Map / Shape 指针(改了 → 类型混淆)
   └── refcount(改了 → 提前 free → UAF)
```

**栈是一根线,堆是一张网**——攻击面不是线性而是组合爆炸。也因此,堆漏洞的利用形态远比栈丰富,缓解机制(Canary / ASLR / DEP)对栈很有效,**对堆要打的折扣大得多**。

### 1.3 时间维度:堆有"生命周期"

栈帧的生命周期等于函数调用——退出即销毁,**没有"用完之后还能再访问"的概念**。

堆对象的生命周期是**程序员管理**的——`malloc` 之后什么时候 `free`、free 之后那块内存被谁拿走、原来指向它的指针有没有置空,**全靠开发者自觉**。一旦自觉不到位,UAF 就在那儿等着。

```
栈漏洞的本质:空间错误(写到不该写的地方)
堆漏洞的本质:空间错误 + 时间错误
   ├── 空间:堆溢出(写到相邻 chunk)
   └── 时间:UAF / Double Free(在错误的时刻访问)
```

**时间漏洞是堆独有的**——这是为什么 Rust 的 ownership 模型对堆漏洞特别管用而对栈漏洞用处一般。

## 二、glibc ptmalloc 心智:理解攻击面所需的最小模型

不重复 osLearning/09 已经讲过的细节,**这里只补攻击者必须知道的三件事**。

### 2.1 chunk 的结构

glibc 每一块堆内存(无论 in-use 还是 free)都长这样:

```
┌─────────────────────────┐  <- chunk 起始
│ prev_size               │   8 bytes(只在前一个 chunk 是 free 时有意义)
├─────────────────────────┤
│ size | A | M | P        │   8 bytes(size + 3 个标志位)
├─────────────────────────┤  <- malloc 返回的指针指这里
│                         │
│  用户数据 (in-use)       │   或
│  fd/bk 指针 (free)      │
│                         │
└─────────────────────────┘
              ↑
   下一个 chunk 的 prev_size 紧贴着
```

**关键观察**:**chunk 的元数据(size、prev_size、fd/bk)和用户数据是连续的**——上一个 chunk 的用户数据写溢出一个字节,就踩到下一个 chunk 的 prev_size;溢出更多,就踩到 size、踩到 fd 指针。**这就是堆溢出能改控制流的物理基础**。

### 2.2 三个 bin:tcache / fastbin / unsorted bin

free 掉的 chunk 不会立刻还给 OS,而是按 size 分类挂到几个链表(bin)上,等下次 `malloc` 同样大小直接复用:

```
   tcache(每线程,无锁,最快)
      ├── 0x20  : [chunk_a] -> [chunk_b] -> ...
      ├── 0x30  : [chunk_c] -> ...
      └── ... 64 个大小档,每档最多 7 个

   fastbin(全局,size <= 0x80,单链表)
      ├── 0x20  : [chunk_d] -> ...
      └── ...

   unsorted bin(其他 size,双链表)
      └── [chunk_e] <-> [chunk_f] <-> ...
```

**攻击者要关心的就一件事**:**free chunk 的 fd / bk 指针就在用户数据区**——也就是说,**当一块内存从 in-use 变成 free,前 8/16 字节就从"用户数据"变成了"指向下一个 free chunk 的指针"**;反过来,**当 malloc 拿走这块内存,前 8 字节又变回"用户数据"**,但**内容是上次 free 时写下的 fd 指针**。

这就给了攻击者一个核心姿势:

```
   原本是用户数据 → free → 现在是 fd 指针
                          ↑
                  攻击者如果能改这 8 字节
                  (比如通过 UAF / 堆溢出写邻居)
                  下一次 malloc 同 size 会返回攻击者指定的地址
```

**这就是 tcache poisoning / fastbin attack 的本质**——一句话:**把 free 链表的 next 指针改成"任意地址",分配器会乖乖把那个地址当成 chunk 还给你**。

### 2.3 Safe-Linking:glibc 2.32+ 的缓解

2020 年 glibc 加了 Safe-Linking——free chunk 的 fd 指针不再明文存,而是和 chunk 自身地址的高位异或一下再存:

```
存:  fd_in_memory = (real_fd) XOR (chunk_addr >> 12)
取:  real_fd      = fd_in_memory XOR (chunk_addr >> 12)
```

**攻击者要伪造一个"指向任意地址"的 fd,就必须知道 chunk 自身地址**——也就是说,**必须先有一个堆地址泄露**。把"无信息泄露直接 EXP"的难度从 1 提升到了 2,但**对有信息泄露原语的攻击者基本没影响**。

## 三、Use-After-Free(UAF):浏览器 0day 第一大类

### 3.1 漏洞机制

UAF 的最小模型只有三步:

```
1. 分配:  p = malloc(sizeof(Object));
2. 释放:  free(p);
          // ←  没有 p = NULL,p 还指向那块内存(dangling pointer)
3. 误用:  p->some_function();   // ← 使用已被释放的对象
```

**单看这三行没问题——问题在第 2 步和第 3 步之间**:如果攻击者能让 `malloc` 在两步之间分配回那块内存(re-allocate 同样的 size),**那块内存就被攻击者数据占据**;接下来第 3 步访问 `p->some_function()` 时,读到的是攻击者数据。

```
    时刻 T1:   p ───→ ┌─────────────────┐
                      │ vtable_ptr      │ ─── 指向合法 vtable
                      │ field_1         │
                      │ field_2         │
                      └─────────────────┘

    时刻 T2:   free(p);   // p 还在,但内存进了 tcache

    时刻 T3:   攻击者触发 malloc 同 size,这块被分配出去
               q = malloc(same_size);
               memcpy(q, attacker_data, same_size);

               p ───→ ┌─────────────────┐
                      │ FAKE_vtable_ptr │ ←── 攻击者控制!
                      │ ...             │
                      └─────────────────┘

    时刻 T4:   p->virtual_call();
               // 读 p[0] 拿 vtable,跳到 vtable[offset]
               // → 跳到攻击者指定地址
```

### 3.2 最小 PoC(C++)

```c
class Obj {
public:
    virtual void greet() { printf("hello\n"); }
};

int main() {
    Obj* p = new Obj();
    delete p;                          // p 悬空但没置 NULL
    long* fake = new long[2];          // 攻击者重新占同一块
    fake[0] = (long)system;            // 伪造 "vtable"
    p->greet();                        // 跳到 system!
}
```

**这 10 行揭示了 UAF 的全部本质**——`delete p` 之后 `p` 仍然有效,`new long[2]` 大概率拿到同一块内存(glibc tcache LIFO),`p->greet()` 解引用第一个 8 字节当虚表,直接被改写。

### 3.3 真实武器化的两个难点

教学 PoC 干净,**真实利用要解决两个问题**:

**问题 1:精确占位**。`free(p)` 之后,要保证下一个**攻击者控制大小**的 `malloc` 拿到同一块内存。手段是「**堆按摩**」:在触发漏洞前先 `malloc / free` 一堆 chunk,把 tcache 队列调到一个可预测状态。浏览器里就是连续 `new ArrayBuffer / new String` 制造特定大小的堆喷射。

**问题 2:vtable 跳板**。即使能伪造 vtable 指针,**还需要 vtable 里有一个能跳到 shellcode 的函数指针**——而 shellcode 还要解决 DEP(数据页不可执行)。这就和栈漏洞合流了——**改 vtable 触发 ROP 链**,17 篇讲过。

## 四、Double Free:bin 链表的逻辑炸药

### 4.1 漏洞机制

同一个指针被 `free` 两次:

```c
free(p);
free(p);   // ← p 已经在 tcache 链表里了,再 push 一次
```

**glibc tcache 在 2.27-2.28 完全没检查这种情况**(到 2.29 才加了 key 字段检测,但还能绕过)。后果:

```
   tcache 链表:  head -> [p] -> [p] -> NULL
                          ↑       ↑
                       同一块内存,被链了两次!
```

接下来连续两次 `malloc(size)` 都会返回同一块 `p`——**但分配器认为是两块独立的内存**。攻击者:

```
   a = malloc(size);   // 拿到 p
   b = malloc(size);   // 又拿到 p,但分配器以为是新块
   *a = (long)target_addr;
   // 此时 b 的"用户数据"也被改了
   c = malloc(size);   // 分配器从 tcache 取出 next,
                       // next = *a = target_addr!
   // c 现在指向 target_addr,任意地址写得到
```

**核心姿势**:**Double Free 把同一块内存的"用户数据写权限"和"free chunk fd 指针"叠加在一起**——往用户数据里写就是改 fd,把 fd 改成任意地址,下次 `malloc` 就返回任意地址。

### 4.2 现代缓解:tcache key

glibc 2.29 在每个 tcache chunk 里加了一个 `key` 字段,`free` 时检查:**如果这块的 key 已经是当前 tcache 的标识,说明它已在链表里,abort()**。

```c
// glibc 简化逻辑
if (chunk->key == tcache) {
    // 已经在 tcache 里了,重复 free
    abort();
}
chunk->key = tcache;
```

**绕过姿势**(不展开):key 字段就在用户数据区,**如果同时有个堆溢出能写到这个字段,把它清零就绕过检测**。安全研究界把这种"漏洞链组合"叫做 House of XXX(Spirit / Force / Einherjar / ...)系列——每一种都是对特定 glibc 版本的绕过技巧。

## 五、堆溢出与 unlink 攻击:经典中的经典

### 5.1 堆溢出基本面

```c
char* p = malloc(64);
strcpy(p, attacker_controlled);   // 没限长
```

写超过 64 字节,**踩到的不是别的局部变量,是相邻 chunk 的元数据**:

```
   ┌─────────────────┐
   │ p 的 chunk header │
   ├─────────────────┤
   │ p 的用户数据 (64B)│
   │                 │
   │   ↓ 溢出方向     │
   ├─────────────────┤  <- 邻居 chunk 边界
   │ 邻居 prev_size  │
   │ 邻居 size       │
   │ 邻居 fd          │     ← 改这里就能 tcache poisoning
   │ 邻居 bk          │
   └─────────────────┘
```

**如果邻居是 in-use chunk,改它的 size / 用户数据;如果邻居是 free chunk,改它的 fd / bk**——后者直接通向链表攻击。

### 5.2 unlink:2003 年的经典攻击

unlink 是分配器从双链表删除一个 chunk 的操作:

```c
// 简化的 unlink 宏
FD = P->fd;
BK = P->bk;
FD->bk = BK;
BK->fd = FD;
```

**注意 `FD->bk = BK` 这一步**——它**写**了 `FD` 指向的地址。如果攻击者控制了 `P->fd`(通过堆溢出),就能让 `FD = 任意地址`,然后 `*(任意地址 + offset_of_bk) = BK`(BK 也是攻击者控制的)。**等价于一个"任意地址写任意值"原语**。

2004 年 glibc 加了 unlink 一致性检查:

```c
if (P->fd->bk != P || P->bk->fd != P) abort();
```

把简单粗暴的 unlink 干死了,但**House of XXX 系列的演化就是从这里开始的**——研究者发现绕过这个检查的方法,新的攻击姿势又冒出来。这是一场持续 20 年的猫鼠游戏。

## 六、C++ 虚表劫持:为什么 OOP 是堆漏洞的放大器

虚函数(`virtual`)的实现机制:

```
   C++ 对象在堆上:
   ┌───────────────────┐  <- new 返回的指针
   │ vtable_ptr        │ ──→  vtable (在 .rodata)
   ├───────────────────┤        ┌────────────────┐
   │ member_a          │        │ &Foo::method_0 │
   │ member_b          │        │ &Foo::method_1 │
   └───────────────────┘        │ ...            │
                                └────────────────┘

   obj->method_1() 编译成:
      mov rax, [obj]          ; 取 vtable_ptr
      call [rax + 8]          ; 调 vtable[1]
```

**致命点**:**`vtable_ptr` 就在对象的第一个 8 字节,和用户数据放一起**。任何能写到这块内存的漏洞(UAF / 堆溢出 / type confusion)都能改 vtable。

```
   攻击者控制堆里某 8 字节
   → 改 vtable_ptr 指向攻击者伪造的 vtable
   → 伪造 vtable[1] = system 地址(或 ROP gadget)
   → 程序虚函数调用 → 跳到攻击者代码
```

**整个 C++ / Chrome / JS 引擎都建立在虚函数之上**——Blink 的 DOM 节点、V8 的 HeapObject、libstdc++ 的 stream,**任何一个 UAF 都能用这条路径打穿**。这就是为什么 Chrome 工程师恨堆漏洞恨到要发明 MiraclePtr。

### 现代缓解:CFI(Control Flow Integrity)

Clang/LLVM 的 CFI 在每次虚函数调用前插入检查:

```
   call_virtual(obj, idx):
      vtable = obj->vtable_ptr
      if (vtable not in valid_vtable_set):
         abort()                    ; ← CFI 检查
      call vtable[idx]
```

**把"任意 vtable"收窄到"已知合法的 vtable"**。Chrome / Edge / Android 系统组件都开了 CFI,**vtable 劫持的难度从"找到 UAF"上升到"找到 UAF + 找到 CFI 绕过"**。19 篇还会专门讲 CFI / Shadow Stack。

## 七、Heap Spraying:浏览器时代的"用空间换确定性"

### 7.1 思路

UAF 利用要求"重新分配同位置",但浏览器堆地址不可预测——**怎么提高命中率?用海量占位**。

```
   攻击者在浏览器 JS / Flash 里:
   for (i = 0; i < 100000; i++) {
       arr[i] = new ArrayBuffer(4096);
       fill_with_shellcode_or_gadgets(arr[i]);
   }

   堆里现在的样子:
   ┌────────┬────────┬────────┬────────┬────────┐
   │payload │payload │payload │payload │payload │ ... 几百 MB
   └────────┴────────┴────────┴────────┴────────┘
```

跳到任何随机地址,**只要落在这片区域内**,执行的都是攻击者的 payload。

**配上 NOP sled(在 payload 头部填一长串 NOP 指令)**,精确度要求进一步降低——跳到 payload 前任何位置都会一路 NOP 滑到真正的 shellcode。

### 7.2 现代缓解

DEP / W^X 让"喷一片 shellcode 等执行"基本失效——堆页默认 RW、不可执行。**于是堆喷射演化成"喷 ROP 链 + 喷伪造对象"**——喷的不是可执行代码,而是用来劫持控制流后跳转用的 gadget 地址表。

JS 引擎层面也有缓解:V8 把大 ArrayBuffer 用 mmap 单独分配(不在主堆里)、限制 typed array 大小、把堆地址做随机化偏移。**老式 spray 在现代浏览器基本绝迹,但思想还在**——任何"我需要让某个未知地址变成已知内容"的攻击都会有 spray 的变体。

## 八、浏览器:堆漏洞的核战场

### 8.1 为什么浏览器堆漏洞这么多

- **JS 暴露给攻击者一个图灵完备的、可触发任意内存分配 / 释放的接口**——`new Array(N) / arr.length = M / arr.fill(x)`,攻击者可以编程操纵堆。
- **DOM / V8 的对象模型大量用虚函数和指针**——任何 UAF 都能升级成 RCE。
- **JIT 编译器引入 type confusion**——猜错对象 shape 就解引用错误的字段,等价于 UAF。
- **沙箱外的代码量巨大**——Chrome 渲染进程几千万行 C++,UAF 是常驻 bug。

Google Project Zero 历年统计:**浏览器 / 内核 0day 中 70-80% 是内存安全漏洞,其中堆相关漏洞又占大头**。

### 8.2 Chrome 的两个新缓解

**MiraclePtr(BackupRefPtr)** —— 2022 年开始全量铺开。原理:把所有 `T*` 替换成带引用计数的 `raw_ptr<T>`,**释放时如果还有活引用,推迟释放或者把内存写垃圾值,让后续 UAF 立刻崩溃而不是被利用**。代价:每个指针多 8 字节、解引用多一次内存访问。**实测把 Chrome 渲染进程的 UAF 类 0day 灭掉 50%+**。

**V8 Sandbox** —— 2023 年起在 Chrome 桌面默认开启。原理:把所有 V8 对象指针变成"沙箱内偏移量(32 位 index)",**即使攻击者拿到 V8 堆内任意读写,也只能在 V8 沙箱内乱跳,跳不出去**。再配合外层进程沙箱,**把"V8 RCE → 操作系统 RCE"这一跳堵死**。

### 8.3 iOS / Safari 的姿势:PAC + MTE 路线

苹果 A12 起 CPU 支持 PAC(Pointer Authentication Codes)——指针在高 16 位带签名,跨边界使用前 CPU 自动校验,签错就崩。**对劫持 vtable / 函数指针特别有效**。

ARMv9 的 MTE(Memory Tagging Extension)更狠——每 16 字节内存分配一个 4-bit tag,指针高位也带 tag,**访问时硬件比较,不匹配就崩**。**这等于在硬件层面把 UAF / 堆溢出做了运行时检查**,开销 < 5%。Pixel 8 / iPhone 16(传)开始硬件支持,**未来 5 年最有希望从根本上杀掉堆漏洞的技术**。

### 8.4 Android / Fuchsia 的 scudo

Google 的 scudo 分配器(Android 11+ 默认)替代了 jemalloc / dlmalloc,内置:

- chunk header 自校验(改一个字节就崩)
- 随机化 chunk 顺序(削弱 heap massaging)
- delayed free(分配器维护一个 quarantine,释放的 chunk 先冷藏一段时间再复用,UAF 窗口被拉长难度上升)
- guard page(大块分配两边夹不可访问页,溢出立刻崩)

**思路和 hardened malloc / GrapheneOS 的 malloc 一致**——以一定运行时开销换取"漏洞触发就崩,而不是被静默利用"。

## 九、真实案例:Heartbleed 与近年 0day

### 9.1 Heartbleed(CVE-2014-0160)

不是经典的 UAF / overflow,而是**堆上越界读**——OpenSSL 的 TLS heartbeat 实现里:

```c
memcpy(response, request->data, request->length);
// request->length 来自攻击者,但没和实际 data 大小校对
```

攻击者发一个 `length = 65535` 但实际 `data` 只有 1 字节的请求,服务器从 `request->data` 这个堆指针开始**往后读 65535 字节**——读到的全是堆上别的对象残骸:**私钥、session ticket、其他用户的 HTTP 请求、密码**。

**Heartbleed 经典在于揭示了堆漏洞的另一面**——不需要写、只要能读,就可能把整个系统的秘密 dump 出来。**修复 ≠ 加锁、加防御**,**而是验证长度**(`if (request->length > actual_data_size) abort`)——本质是 C 没有自动边界检查。

### 9.2 Chrome / iOS 近年 0day 缩影

不列具体 CVE 编号(00 写作计划禁止),只列模式:

```
攻击模式 1:V8 的 JIT 类型推断出错
  → 把对象误以为另一种 shape
  → 字段偏移算错 → 越界读写 V8 堆
  → 配合 V8 内对象的指针拿到任意读写
  → 通过 WebAssembly 拿 RWX 页 → RCE

攻击模式 2:渲染进程 DOM UAF
  → JS 触发 DOM 节点释放但还有引用
  → JS 立刻分配相同 size 的 ArrayBuffer 占位
  → 接下来 DOM 操作访问那个"节点",其实是攻击者 buffer
  → 改 vtable → 跳 ROP → 渲染进程沙箱内 RCE

攻击模式 3:iOS Safari → 内核
  → WebKit 渲染进程 RCE(同上)
  → 内核驱动 UAF(IOMobileFrameBuffer / AppleAVD 等)
  → 沙箱逃逸 → 内核读写原语 → root
```

**每一年 Pwn2Own / 国家级 APT 拿出的 0day,本质都是这两三个模式的变体**。

## 十、修与防:工程师该做什么

### 10.1 代码层短期 / 长期

**短期(C/C++ 项目立刻能做)**:

- `free(p); p = NULL;` —— 释放后立刻置空,消灭 dangling pointer。代价 0,效果显著。
- `delete` 之后用 `unique_ptr / shared_ptr` —— 让所有权语义在编译期表达,**不再手动 free**。
- 任何 `memcpy / strcpy` 替换成带长度版本(`memcpy_s / strncpy_s`)。
- 开 `-D_FORTIFY_SOURCE=3` `-fstack-protector-strong` `-fsanitize=address`(debug 时)。
- CI 接 ASan / UBSan / MSan —— **本地跑一遍单测就能抓出 90% 的 UAF / overflow**。

**长期(项目级)**:

- 新模块用 Rust 写 —— Mozilla / Microsoft / Google / Apple 都在做。Rust 编译期 ownership 检查直接消灭 UAF / Double Free / Iterator invalidation,**这一类漏洞在 Rust 里写不出来**。
- 不能换 Rust 的存量 C++ 代码 —— 上 MiraclePtr / 智能指针 / Chromium 风格的所有权审查。
- 漏洞类别消除(VRP class elimination)—— Google 内部做法:每次新发现一类漏洞,**不是修一个,是问"怎么让这一类整体消失"**。Chrome 的 V8 Sandbox 就是这种思路。

### 10.2 系统 / 编译层缓解

19 篇会专门讲——这里只列堆相关的:

- **ASLR**:堆基址随机化,提高 spray 难度。
- **DEP / W^X**:堆页不可执行,杀死直接 shellcode 喷射(但 ROP 没事)。
- **Safe-Linking**(glibc 2.32+):tcache fd 指针带 cookie。
- **CFI**(Clang / MSVC):虚函数调用做合法 vtable 集合检查。
- **MTE**(ARMv9):硬件级 16 字节粒度的 tag 检查。
- **scudo / hardened malloc**:替换默认分配器,加上 quarantine / guard page / 随机化。

**没有一个能单独杀死堆漏洞**——**纵深防御才是真实路径**:开发期(智能指针 + Rust)→ 编译期(CFI + Sanitizer)→ 运行期(MTE + scudo)→ 进程间(沙箱)→ 系统(内核 hardening)。**任何一层独自被绕过都不会让攻击者拿到完整 RCE**。

### 10.3 监控与应急

- 用户态崩溃日志(coredump / crashpad)接 Crashpad / Breakpad → 中心化收集 → **同一个崩溃栈短时间高频出现 = 可能是被野外利用**。
- AddressSanitizer 在生产环境关掉,但**有些公司在 Canary 环境长期开**——以可观测的崩溃换"线上漏洞被外部攻击者先发现"的风险。
- 应急响应:堆漏洞 0day 一旦野外公开,**第一反应是更新基础库 / 重启服务**——很多 glibc / OpenSSL 漏洞通过升级就能修,但**升级不重启,内存里跑的还是老代码**。

## 十一、踩坑提醒

1. **`free(p)` 之后 `p` 还能用**——C/C++ 不会自动置空,21 世纪最贵的 NULL。
2. **`if (p) free(p)`**——这句没意义,`free(NULL)` 本来就合法;真正该写的是 `free(p); p = NULL;`。
3. **以为加了 Canary / ASLR / DEP 就安全**——这些对栈漏洞够用,**对堆只是延缓**,需要 CFI + MiraclePtr + MTE 这套现代组合。
4. **以为 `unique_ptr` 万能**——`unique_ptr.get()` 返回的裸指针、`.release()` 之后的所有权转移、`shared_ptr` 循环引用,**每一个都是漏洞温床**。
5. **以为 Rust 完全消除内存漏洞**——`unsafe` 块 / FFI 边界 / 逻辑 use-after-free(借助 `Rc<RefCell>` 也能写出 UAF 等价物),**Rust 是消除 90%,不是 100%**。
6. **以为 Sanitizer = 测试通过 = 没漏洞**——ASan 只在被实际触发的代码路径上检测,**模糊测试覆盖不到的代码,Sanitizer 也帮不上**。20 篇会讲 fuzzing。
7. **以为 Heartbleed 之后 OpenSSL 没问题了**—— OpenSSL 后来又出了好几个堆相关 CVE,**密码学库 + C 语言这个组合本质上就高风险**。
8. **以为浏览器升级慢一点没事**——Chrome 主版本 4 周一次、紧急 0day patch 24 小时内,**慢 1 周可能就被野外打**。企业 IT 把浏览器自动更新关掉是常见但极其错误的策略。
9. **以为容器隔离能挡住堆漏洞** —— 浏览器渲染进程内的 UAF 不需要打到内核就能偷 Cookie / localStorage,**进程级沙箱不是网络级隔离的替代**。
10. **以为 Java / Python / Node 没有堆漏洞**——这些语言的运行时本身是 C/C++ 写的,**V8 / CPython / OpenJDK 自己被打就等于你的应用被打**。2021 年 log4shell 不是堆漏洞,但有过 JNI 堆漏洞、cpython 解释器漏洞,**只是离应用工程师远一点**。
