# 安全监控与应急响应:IDS / SIEM / 取证 / 事件复盘

前 28 篇都在讲「**怎么不被打**」——防注入、防溢出、防错配、防投毒。但工业界一条很冷的常识是:**只要你够大、暴露面够广,被打中是迟早的事**——Google、Microsoft、Cloudflare 这种安全投入开到天上的公司每年也都被打,只是它们**几小时内发现、几小时内处置**,而被新闻头版的那些公司是**几个月才发现自己被打了**。这两种结局的差别不在「防」,在「**发现 + 处置的速度**」——这就是这一篇的主题。前面所有篇章都是「事前防御」,这一篇是「**事中检测 + 事后响应**」,合起来才是完整的安全闭环。

> 一句话先记住:**MTTD(平均发现时间)+ MTTR(平均响应时间)是衡量一个安全团队水平的硬指标**——Mandiant 2024 年报告全球 dwell time 中位数 **10 天**,亚太区 **9 天**,而 ransomware 类只要 **5 天**;一流团队能压到 **小时级**,差团队是 **月级**。差别不是「工具好不好」,是「**告警有没有人看、看了有没有人处置、处置完有没有人复盘**」——三个独立的工程能力,缺一个都白搭。**没人值班的告警 = 没有告警;没复盘的事件 = 等着第二次被同样手法打**。

## 一、为什么"被打了之后多久能发现"才是核心指标

### 1.1 dwell time:攻击者在你网里待了多久

```
攻击链:
  Day 0    钓鱼邮件 / 漏洞利用 / 凭据泄漏 落地
  Day 0+   植入 webshell / RAT,建立 C2 回连
  Day 1-7  横向移动 / 提权 / 摸地形
  Day 7+   收集数据 / 部署勒索 / 准备出货
  Day N    爆发(勒索弹窗 / 数据出现在暗网)
        ─────────────
        dwell time = Day 0 → 你发现的那一天
```

**dwell time 越长损失越大**,不是线性曲线——**多 3 天可能从单台 webshell 升级到整域沦陷**。Equifax 2017 那次 dwell time 是 76 天,1.4 亿条记录被拖走。SolarWinds 2020 那次,攻击者蹲了 9 个月才被发现。

### 1.2 三个指标:MTTD / MTTR / MTTC

| 指标 | 全称 | 含义 |
| --- | --- | --- |
| MTTD | Mean Time To Detect | 入侵发生 → 检测到 |
| MTTR | Mean Time To Respond | 检测到 → 开始处置 |
| MTTC | Mean Time To Contain | 检测到 → 完全控制 |

**业内基准**:顶级团队 MTTD < 1h、MTTR < 15min;中等公司 MTTD 一天 / MTTR 几小时;没有 SOC 的公司经常是「**甲方或媒体告诉你被打了**」,MTTD 数周到数月。

### 1.3 为什么"防 + 检测"必须双修

```
纯防御的错觉:
  装 WAF / EDR / 防火墙 → "我安全了"

真相:
  WAF 绕过技术每年都有新的
  EDR 在内核驱动层,可被 BYOVD 攻击禁用
  0day 永远存在
  钓鱼 / 内鬼 / 第三方供应链 绕开你所有外部防御

  → 防御只能拦"已知 + 笨"的攻击
  → 检测才能发现"未知 + 聪明"的攻击
```

**纵深防御的下半段是"假定已经被打"**——这就是为什么大公司都有 SOC 和 Threat Hunting 团队,他们的工作不是"防",是「**假设网里已经有攻击者了,把他找出来**」。

## 二、IDS / IPS:在流量上发现攻击

### 2.1 IDS vs IPS:旁路 vs 串行

```
IDS — 旁路镜像:流量 → 交换机 SPAN 口 → IDS → 只告警不阻断
       优点:不影响业务  缺点:看到了也来不及拦
IPS — 串行阻断:流量 → IPS → 转发/丢弃 → 后端
       优点:能拦      缺点:误报会断业务
```

**业内常态**:核心链路 IPS,办公网 / DMZ 上 IDS。「能拦就拦,不能拦至少告警」。

### 2.2 签名规则 vs 异常检测

**签名规则**:特征已知攻击。比如 SQLi 看 URI 里 `union select`,Log4Shell 看 header 里 `${jndi:ldap://`。**误报低可解释,但只能找已知攻击**——0day 看不见。

**异常检测**:先建基线("这台机器平时只和 10 个 IP 通信"),偏离即告警。**能发现未知,但误报高、基线漂移**(业务变化也告警)。

> **现代检测是"签名 + 异常 + 机器学习"组合**——纯签名跟不上 0day,纯异常误报到没人看。

### 2.3 Snort / Suricata / Zeek

| 工具 | 定位 | 长处 |
| --- | --- | --- |
| **Snort** | 老牌 IDS / IPS,1998 年开始 | 规则生态最完整(Talos / ET) |
| **Suricata** | 现代 IDS / IPS,多线程 | 性能强,兼容 Snort 规则 |
| **Zeek**(原 Bro) | 网络行为分析框架 | **不是规则,是脚本** —— 记录所有连接元数据 |

Zeek 的不同——它**不告警,记录一切**:`conn.log` / `http.log` / `dns.log` / `ssl.log` / `files.log`。这些日志后面喂给 SIEM 做**事后调查**——你不知道明天怎么被打,但 Zeek 把今天的一切录下来,等你回去翻。

### 2.4 IDS 越来越难做

```
TLS 普及     → IDS 看不见 HTTP body(只能看 SNI / cert)
HTTP/3 QUIC → 跑 UDP 上,很多 IDS 还不支持
家用 VPN     → 员工流量被代理走
云原生       → 东西向流量在 VPC 里,边界 IDS 看不到
```

**这就是为什么主机层 EDR 越来越重要**——网络看不见的,主机能看见。

## 三、主机层 EDR:在终端上发现攻击

### 3.1 EDR 到底看什么

```
进程创建链:   谁 fork 了谁,执行了什么命令
文件操作:    哪个进程写了哪个 .exe / .dll / .sh
注册表 / 配置: 持久化痕迹
网络连接:    哪个进程连了哪个 IP / 域名
脚本解释器:   PowerShell / cmd / bash 全量命令行
DLL 注入:    进程内存里出现不属于它的模块
凭据访问:    谁读了 LSASS / /etc/shadow
```

**经典攻击链**(MITRE ATT&CK 都有分类):

```
Office Word → cmd.exe → powershell -enc <base64> → 下载 EXE → 落地
```

单看每一步都"正常",**串起来就是钓鱼链**。EDR 的关键能力就是**把链拼出来**。

### 3.2 工具谱系

| 工具 | 定位 | 适合 |
| --- | --- | --- |
| **Sysmon**(Windows) | 微软官方,免费,只采集 | 小团队 / 自建检测底座 |
| **osquery** | Facebook 开源,SQL 查主机状态 | 跨平台 |
| **Wazuh** | 开源 EDR + SIEM 一体 | 中小企业 |
| **CrowdStrike Falcon / SentinelOne / Defender for Endpoint** | 商业 EDR | 大企业,贵但好用 |
| **Velociraptor** | 开源 DFIR 平台 | 事件响应时大规模采集 |

### 3.3 osquery 把主机变成数据库

```sql
-- 查所有监听端口
SELECT pid, port, address FROM listening_ports;
-- 查最近创建的 setuid 文件
SELECT path, mtime FROM file
WHERE path LIKE '/usr/bin/%' AND mode LIKE '4%';
-- 查所有 cron 任务
SELECT * FROM crontab;
```

**全公司主机当一张大表查**——事件响应时极其值钱:「**有没有别的机器也被同样手法打了**」从「挨个登 100 台」变成「一行 SQL」。

## 四、SIEM:把所有日志汇成一张图

### 4.1 SIEM 干什么

```
1. 采集:syslog / EDR / IDS / 应用 / 云 全汇总
2. 关联:跨源 join(IDS 报 SQLi + Nginx 看到 500 响应 → 同一事件)
3. 检测:跑规则触发告警
4. 调查:给分析师查询界面,事件回溯
```

### 4.2 工具谱系

| 工具 | 备注 |
| --- | --- |
| **Splunk** | 商业 SIEM 之王,**贵到劝退**(按摄入量计费,GB / 天) |
| **Elastic SIEM** | ELK 一脉,中等规模主流 |
| **OpenSearch** | AWS fork 的 Elastic,完全开源 |
| **Wazuh** | 开源 SIEM + EDR,小团队首选 |
| **Sumo / Datadog / Chronicle** | SaaS SIEM,不想自己运维就用 |

Splunk 为什么贵:大公司每天日志 TB 级,**一年许可费可能上百万美元**——常见的对策是「**热数据 Splunk,冷数据下沉到 S3 + Athena**」。

### 4.3 一条告警的流转

```
Nginx 日志 → Filebeat → Logstash(解析)→ Elasticsearch(存储)
                                              ↓
                                        SIEM 规则引擎
                                              ↓
                                  告警(邮件 / Slack / PagerDuty)
                                              ↓
                                  值班同事接收 → 分诊
```

**关键瓶颈不在工具,在"值班"**——半夜 3 点没人看 = 没告警;告警没分级 = P3 淹了 P1;告警没上下文 = 一行 IP+URL 不知道怎么处置。

### 4.4 日志收集层

| 工具 | 角色 |
| --- | --- |
| **syslog / rsyslog** | 老牌 Linux 日志协议 |
| **journald** | systemd 日志,结构化但默认本地 |
| **Filebeat / Logstash**(Elastic 系) | 文件 → 解析 → 转发 |
| **Fluentd / Fluent Bit**(CNCF) | 云原生 K8s 主流 |
| **Vector**(Rust 写) | 高性能新秀,日志 + 指标 + Trace 统一 |

**常见架构**:`应用 stdout → 容器日志驱动 → Fluent Bit (daemonset) → Kafka(缓冲) → Logstash/Vector(解析) → ES/S3 → SIEM`。**为什么必须有 Kafka**:出事故时日志 10× 暴涨会把下游 ES 打挂,Kafka 做缓冲。

## 五、Sigma:统一的检测规则格式

**痛点**:每家 SIEM 规则语法都不一样——Splunk SPL / Elastic KQL / Sentinel KQL / Splunk ES,**换工具 = 规则全重写;社区分享规则没法复用**。

**Sigma 用 YAML 写"通用规则"**:

```yaml
title: Suspicious PowerShell Encoded Command
logsource: { product: windows, category: process_creation }
detection:
    selection:
        Image|endswith: '\powershell.exe'
        CommandLine|contains: '-enc'
    condition: selection
level: high
```

配合 `sigmac` 转换器,**一份 YAML 编译成各家 SIEM 的方言**。SigmaHQ 仓库几千条公开规则,做 Threat Hunting 时常去捞。

## 六、SOAR:把"人重复做的"自动化

**为什么纯靠人不行**:中型公司一天 1000-10000 条告警,2-5 个分析师,一条 5 分钟——**一个人一天极限 ~80 条,90% 告警没人看**。真攻击就藏在这堆里。

**SOAR 的核心是剧本(playbook)** ——把"人做的判断"编成自动化:

```
      告警触发
         │
   [取 IP 威胁情报]──[查 VT / AbuseIPDB]
         │
   IP 是 known bad?
   │           │
  YES         NO
   │           │
[自动隔离]   [发给分析师人工判断]
   │
[发工单 + 通知]
```

**平台**:Splunk SOAR(原 Phantom)、Cortex XSOAR、Tines、Shuffle(开源)。

**适合自动化**:机械重复(查 IP 信誉、拉 VT 报告、kick session)+ 确定性高的响应(封 IP、隔离主机)。

**不适合自动化**:涉及业务判断("这个 IP 是真攻击还是合作伙伴扫描")+ 高破坏性(误封 CEO 邮箱)+ 复杂取证。

> 经验:**SOAR 解决 80% 重复活,剩下 20% 高价值判断留给人**。一上来想全自动是死路。

## 七、应急响应六步法:PICERL

NIST SP 800-61 经典模型:

```
P  Preparation     事前准备
I  Identification  发现 / 确认
C  Containment     遏制(止血)
E  Eradication     根除
R  Recovery        恢复
L  Lessons Learned 复盘
```

### 7.1 Preparation:平时养兵(最被忽视、回报最高)

IR 团队职责 + 轮值表 / 联系人清单(法务 / 公关 / 高管 / 监管)/ 取证工具箱 / 沙箱就位 / out-of-band 沟通信道(出事时主信道可能失效)/ **桌面演习(tabletop)半年一次**。

### 7.2 Identification:确认是不是真事件

```
信号来源:SIEM 告警 / EDR 触发 / 用户报告 / 外部通报 / 监管约谈(最糟糕)
要决定:  严重程度(P0-P4)/ 影响范围 / 是否升级到全公司 IR
```

**别立刻动手**——新手一看告警就 `kill -9` + 拔网线,**把所有取证痕迹毁掉**,后面根因分析做不下去。**先评估,再动手**。

### 7.3 Containment:止血

```
短期(分钟级):隔离主机 / 禁用账户 / 撤销 token / 临时下线服务
长期(小时-天):部署临时 WAF 规则 / 改密码 / 轮换密钥 / 网段隔离
```

**两难**:隔离越激进业务影响越大;越保守攻击者越可能扩大战果。**没有银弹,现场判断**。

### 7.4 Eradication:挖干净

找入口 → 找所有 IOC(Indicator of Compromise:恶意 IP / 域名 / 文件 hash / 注册表项)→ 全网搜 IOC → **重装系统(rebuild,不是 clean)**。

> 「**清掉 webshell 就行了**」是最常见错误——webshell 是表象,**真后门可能是计划任务 / systemd 服务 / SSH key / 篡改过的 sudo / BIOS implant**。**专业 IR 默认 rebuild**。

### 7.5 Recovery:业务回正

从干净备份恢复 → 验证(扫 IOC 确认干净)→ 灰度上线 → 全量 → **被打过的地方 7×24 重点盯一段时间**。

### 7.6 Lessons Learned:复盘

事件 RCA 文档要回答:时间线 / 根因 / 哪里检测晚了 / 哪里响应慢了 / 下次怎么早一步发现 / action items(谁负责什么时候做完)。

**复盘原则:对事不对人(blameless postmortem)** ——找系统漏洞不找责任人,不然下次没人敢报告事件。

> **没复盘的事件 = 等着第二次被同样手法打**。

## 八、数字取证:把现场冻住,慢慢看

### 8.1 取证铁律:order of volatility

**易失性高的先取**——RAM 一断电就没,磁盘可以慢慢取:

```
1. 寄存器 / cache         (毫秒级丢失,不实际取)
2. 内存(RAM)             ← 重中之重,LiME / WinPmem
3. 网络状态(连接 / 路由) ← ss / netstat
4. 进程 / 内存映射         ← /proc 快照
5. 临时文件 / swap         ← /tmp /var/tmp
6. 磁盘                    ← dd / ddrescue / FTK Imager
7. 远程日志                ← syslog 服务器
```

### 8.2 磁盘镜像 + 校验

```bash
# 概念性命令,实际配合 write-blocker + 取证工具
sha256sum /dev/sda > before.hash
dd if=/dev/sda of=disk.img bs=4M conv=sync,noerror
sha256sum disk.img > after.hash
# before / after 必须一致 —— 证明"我没改原盘"
```

取证证据要在法庭 / 监管面前可信,**必须证明从取证那一刻到分析,文件没被改过**。

### 8.3 内存取证:看到磁盘看不到的

磁盘上没痕迹的攻击:**无文件攻击**(fileless,只在 RAM 跑 PowerShell)/ 进程注入 / rootkit hook。**必须看 RAM**。

Volatility 经典命令(示意):`pslist`(进程)/ `netscan`(连接)/ `malfind`(注入代码)/ `dumpfiles`(提取内存里的文件)。

### 8.4 时间线分析

所有事件按时间排序,**故事一目了然**:

```
02:13:07  /tmp/.cache/x  文件创建      (来自 sshd worker)
02:13:08  端口 4444  新增 LISTEN       (PID 12453)
02:13:09  /etc/cron.d/clean  修改      (root 写)
02:14:32  外联 185.x.x.x:443           (PID 12453)
02:15:10  /etc/passwd  读取            (PID 12453)
02:17:42  /root/.ssh/authorized_keys  追加
```

故事清晰:**02:13 之前一切正常 → sshd 触发 → 释放后门 → 改 cron 持久化 → 外联 C2 → 读密码文件 → 加 SSH 公钥**。入口很可能是 SSH 弱口令或漏洞。这就是 IR 报告的核心交付物:**时间线 + 根因 + IOC 列表**。

### 8.5 攻击者反取证(防御方要预防)

```
删日志(rm /var/log/* / journalctl --vacuum)
  → 防御:日志实时外送到独立 SIEM 服务器
改时间戳(touch -t)
  → 防御:对比 atime / mtime / ctime + inode change time
内存运行不落盘
  → 防御:EDR 监控进程内存 + 网络外联
清 .bash_history / wevtutil cl
  → 防御:命令审计独立采集(auditd / Sysmon)
LOLBin(用系统自带 certutil / bitsadmin 下载)
  → 防御:基线 + 告警("certutil 怎么会从这台机器发起?")
BYOVD(Bring Your Own Vulnerable Driver)关 EDR
  → 防御:驱动签名白名单 / WDAC / 监控驱动加载
```

**核心**:**取证数据不能存在被攻击的机器上,必须实时外送**。等出事再去原机器找日志,大概率已经被擦干净了。

## 九、蓝队日常:告警分诊 / 误报治理 / 检测调优

**Tier 分层**:

```
Tier 1   7×24 值班,快速分诊 + 简单处置(80% 告警在这层)
Tier 2   复杂事件调查 + 取证(10%)
Tier 3   Threat Hunting + 检测规则开发(5%)
IR Team  重大事件统筹(<5%,但每次都是大事)
```

**误报治理**:**误报杀死告警系统**——一旦 90% 是误报,人就开始无视所有告警,**真攻击来了也被淹**。每条规则定期看 precision(真阳率),< 80% 就调规则 / 加白名单 / 改阈值;维护资产清单给告警上下文("这台是 DB,不该有 outbound 80");用威胁情报富化(IP / 域名 / hash 自动查 VT);多条信号关联。

**检测调优**:新威胁 → 写规则 → 上线观察一周 → 误报多就调阈值 / 加白名单 → 漏报就多源关联 → 稳定后入主规则集。**MITRE ATT&CK 覆盖率**是检测团队的核心 KPI——~600 个 technique,**能检测多少个就是检测能力上限**。

## 十、真实事件复盘:一次从告警到根因的时间线

**场景**:某中型公司,白天突然 SIEM 触发告警。**所有 IP / 时间是虚构示意**。

```
14:02:11  SIEM 告警:internal 10.20.5.42 外联 185.x.x.x:443 (新 IP)
14:02:11  威胁情报匹配:185.x.x.x 是 known C2(Cobalt Strike teamserver)
14:02:13  告警进入 P1,自动 ping 值班
14:02:45  Tier 1 接手,EDR 看 10.20.5.42:
            └─ 异常进程 svchost.exe(假冒,实际路径 C:\Users\xxx\AppData)
14:03:20  SOAR playbook 启动:
            ① EDR 隔离 10.20.5.42
            ② 防火墙 block 185.x.x.x 全公司
            ③ 关联查询近 24h 谁还连过这 IP → 找到第二台 10.20.7.13
14:05:00  Tier 2 介入,EDR 隔离第二台
14:10:00  开始取证:抓内存 + 磁盘镜像
14:30:00  Volatility 分析:Cobalt Strike beacon,定时 60s 回连
15:00:00  时间线反推:
            └─ 13:48 10.20.5.42 收到钓鱼邮件 + .docm 附件
            └─ 13:51 用户打开 docm → 触发宏 → 释放 beacon
            └─ 13:55 beacon 第一次回连(未告警,IP 还在白名单期内)
            └─ 14:02 beacon 切到新 C2 IP(新 IP 不在白名单,触发)
16:00:00  全网搜 IOC:钓鱼 sender + docm hash + C2 IP
            └─ 另外 3 个用户收到同一封邮件,2 个没点,1 个隔离机已处理
17:30:00  根除:邮件网关全删,EDR 推签名,被打 2 台 rebuild
当日       Recovery + 监控加强
T+1        全员钓鱼演练 + 安全提醒
T+3        Lessons Learned 会议:
            ① 邮件网关沙箱漏过 docm,加规则
            ② EDR 没在第一次回连(13:55)就告警,IP 还没在情报库
              → 加规则:任何新出口 IP 都要灰度告警
            ③ 用户安全意识 → 月度演练
```

**关键复盘点**:

- **MTTD = 14 min**(13:48 入侵 → 14:02 告警)。不算快但可接受
- **MTTR = 34 sec**(14:02:11 告警 → 14:02:45 接手)。很快,值班有效
- **MTTC = 18 min**(14:02:11 告警 → 第二台隔离)。可以更快,SOAR 应自动连锁

> 这是一次「**还行**」的事件——损失可控,根因清晰。**真正灾难性的事件是 MTTD 60 天起**——那种就是 ransomware 弹窗了才知道完了。

## 十一、小公司从零搭建监控的最小栈

预算极有限的小团队怎么起步?**全免费 / 开源,够用就行**:

```
日志收集:   Fluent Bit (agent) → Kafka(可省)→ Elasticsearch / OpenSearch
SIEM:      Wazuh(开源,EDR + SIEM 一体)/ 或 Elastic SIEM 免费版
主机层:     Wazuh agent + Sysmon(Windows)+ auditd(Linux)
网络层:     Suricata(IDS,镜像核心交换机)+ Zeek(连接元数据)
威胁情报:   AbuseIPDB / VirusTotal 免费 API + MISP(自建,开源)
SOAR:      Shuffle(开源)/ Tines 免费版
告警通道:   PagerDuty 免费 / Opsgenie / Slack webhook
值班:       初期不用 7×24,但要"工作时间内 < 30 min 响应" SLA
节奏:       每月一次桌面演习 + 检测规则梳理
```

**总投入**:1-2 人兼职 + 几台中等服务器,**就能撑住 200 人公司的安全监控基本盘**。

> **不要一上来就买 Splunk + CrowdStrike + Palo Alto** ——预算全砸工具就没钱招人;**没人值班的工具 = 没工具**。**先建团队,工具后补**。

## 十二、应急响应剧本(playbook)模板

下面这份是**通用 IR 剧本骨架**,落地时按具体威胁(钓鱼 / 勒索 / 数据泄漏 / 内鬼 / 0day)定制。

```
─────────────────────────────────────────────
事件名:       [自动填]      事件编号:INC-[自动填]
触发时间:     [自动填]      严重程度:[P0/P1/P2/P3]
事件指挥官:   [必填]        当前状态:[检测中/处置中/已遏制/已根除/复盘中/关闭]
─────────────────────────────────────────────

【0】启动(15 分钟内完成)
□ 建立专用 war room(Slack / 电话桥)
□ 通知:CTO / SecLead / 法务(P0/P1 必须)
□ 指派 事件指挥官 / 取证负责人 / 沟通负责人
□ 开始时间线记录(谁在几点做了什么)

【1】Identification
□ 告警是否为 true positive
□ 影响范围(主机 / 用户 / 服务 / 数据)
□ 入侵手法初判(钓鱼 / 漏洞 / 凭据 / 内鬼 / 供应链)
□ 是否涉及 PII / 财务 / 客户数据(决定是否触发监管通报)

【2】Containment(目标 < 1h)
□ 隔离受感染主机(EDR / 防火墙)
□ 禁用涉事账户 / 撤销 token
□ 屏蔽已知恶意 IP / 域名 / 邮件 sender
□ 临时下线 / 降级受影响服务
□ 决策点:激进隔离 vs 业务影响 —— 由指挥官拍板
□ 保留取证现场(不要直接关机,先抓内存)

【3】Investigation & Eradication
□ 取证:内存 / 磁盘 / 日志快照
□ 时间线分析 → 找根因
□ 提取 IOC(IP / 域名 / hash / 用户 / 邮件主题)
□ 全网横向扫描 IOC,找出所有被波及主机
□ 邮件网关 / WAF / EDR 推送签名
□ 重装(rebuild)受感染主机,不要 clean
□ 轮换所有可能泄漏的凭据 / 密钥 / Token

【4】Recovery
□ 从干净备份恢复数据
□ 验证恢复主机干净(IOC + EDR 全量)
□ 灰度 → 全量
□ 重点监控 7-14 天

【5】Communication(贯穿全程)
□ 内部:每 30 min 状态同步(P1)/ 2h(P2)
□ 高管:重大节点必须通知
□ 法务 / 公关:涉及客户数据时必须介入
□ 客户:按法规要求时间窗内通知
□ 监管:按法规(GDPR 72h / 各地 CSL)
□ 警方:涉及犯罪 / 重大资产损失

【6】Closure
□ 所有系统已恢复并验证
□ 所有 IOC 已封禁
□ 没有新的相关告警 72h
□ 事件报告(事实层)
□ 召开 Lessons Learned 会议(blameless)
□ 产出 action items(责任人 + DDL)
□ 跟踪 action items 至全部完成
─────────────────────────────────────────────
```

**两条软原则**:

1. **指挥官说了算**——技术专家可以建议,但拍板权统一。否则群里 10 个人 10 个意见,谁都不动
2. **公开沟通要慢一点,内部行动要快一点**——发一条错误对外信息比晚发一小时损失大得多,尤其涉及客户的事件

## 十三、踩坑提醒

1. **装了 EDR 就完事了** —— 没人看告警 = 没装
2. **MTTR 算"开始处置"的时间**,不是"完全恢复"
3. **告警不分级 = 没有告警** —— P0/P1/P2/P3 必须有清晰定义和 SLA
4. **遇事第一反应不要 kill -9 / 拔网线** —— 先抓内存,再处置,否则取证全毁
5. **永远不要 clean,要 rebuild** —— 你不知道还有什么藏着
6. **日志必须实时外送到独立服务器** —— 存在原机器的日志在攻击者眼里就是待删文件
7. **没复盘的事件等于没处理** —— 同样手法会打你第二次
8. **复盘是 blameless 的** —— 找系统漏洞不找人,否则下次没人敢报告
9. **Splunk 不是答案,流程才是** —— 工具花钱买,流程花时间练,后者更重要
10. **演习比工具重要** —— 半年一次桌面演习,出事时人才不会慌
