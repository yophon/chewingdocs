栈溢出与 ROP:经典 stack smashing 与 Return-Oriented Programming ​

讲栈溢出最大的误区是「现在还有人写 C 吗?这玩意儿是上个世纪的事了」——这种轻视是栈溢出至今活着的根本原因。Linux 内核、glibc、OpenSSL、SQLite、ffmpeg、所有浏览器内核、所有数据库引擎,全部是 C/C++ 写的——你今天在用的每一个云服务,底下都有几千万行 C 代码。栈溢出从 1988 年 Morris 蠕虫开始,到 1996 年 Aleph One 把"配方"写明白,整整三十年里它依然是 CVE 排行榜常客——只是攻击姿势从「注入 shellcode 就能跑」进化到了「用程序里已有的小片段拼出新逻辑」。这一篇讲清楚:栈是怎么被打穿的、为什么 DEP/NX 没能终结这个故事、ROP 怎么用一堆 ret 指令拼出图灵完备的攻击逻辑、以及为什么 Rust / Go / Java 几乎不会出这类漏洞。

一句话先记住:栈溢出 = 把数据写超出缓冲区边界,把"控制流的元数据(返回地址 / 函数指针)"覆盖成攻击者想要的值——本质和 SQL 注入同源,代码和数据没分开。ROP = DEP 把数据页设成不可执行后,攻击者不再注入新代码,而是把程序自己已有的代码片段(以 ret 结尾的几条指令,叫 gadget)串成一条新的"执行路径" —— 就像绑票电影里那个用报纸剪字拼威胁信的桥段,字都是别人的,意思是你的。

本篇严格只讲原理 + payload 大致结构,不给完整可运行 exploit。所有动手练习请在自己搭的靶机或 CTF 环境里做,别打别人系统——这不是道德问题,是法律问题(《刑法》285、286 条,美国 CFAA 等)。

一、栈溢出为什么三十年杀不死 ​

1.1 根本原因:C 的内存模型把"数据"和"控制"放在一起 ​

回看 16 篇讲的栈帧布局——调用一个函数时,返回地址和局部变量挨在一起,而且写入方向相反:

高地址  ┌──────────────────┐
        │  函数参数         │
        ├──────────────────┤
        │  返回地址(rip)   │ ← 函数执行完跳回这里
        ├──────────────────┤
        │  保存的 rbp       │
        ├──────────────────┤
        │  局部变量 buf[64] │ ← 写入从低地址往高地址
        │  ...              │
低地址  └──────────────────┘

关键事实:buf 写入方向是「从低地址往高地址」,朝着返回地址的方向。strcpy / gets / sprintf / memcpy 这些函数不做边界检查——你说要拷 200 字节,它就拷 200 字节,根本不管目标缓冲区只有 64 字节。多出来的 136 字节就把返回地址、保存的 rbp、甚至上层栈帧全覆盖了。

这是 C 设计哲学的代价——C 把性能放在第一位,边界检查交给程序员。1972 年 Unix 还跑在几兆内存的 PDP-11 上,每次访问数组都做边界检查代价太高,Dennis Ritchie 选择「相信程序员」。这个选择影响了五十年——后果就是栈溢出 / 堆溢出 / use-after-free 这三大类内存安全漏洞,至今占微软 / 谷歌产品 CVE 总数的 70% (2019 微软安全响应中心数据)。

1.2 历史时刻:1988 Morris 蠕虫 / 1996 Aleph One ​

Morris Worm(1988-11) 是栈溢出的"出道作品"——Robert Morris 利用 BSD fingerd 服务的栈溢出,让 10% 的当时联网计算机瘫痪(几千台,占当时整个互联网)。这是计算机犯罪入刑的第一案。Morris 后来去 MIT 当了教授,但栈溢出从此进了攻击者的工具箱。

Aleph One「Smashing the Stack for Fun and Profit」(Phrack 49, 1996-11) 是把「栈溢出怎么写 exploit」完整流程写出来的开山之作。在这之前,栈溢出是少数大佬的秘技;之后,任何会写 C 的人读完都能复现。这篇文章的核心心智是:

1. 找一个边界检查缺失的函数(strcpy / gets / sprintf 等)
2. 算出 buf 到返回地址的偏移(填多少字节才覆盖到)
3. 把"自己写的 shellcode 机器码"放进 buf
4. 把返回地址覆盖成 buf 的地址
5. 函数返回时 CPU 跳进 buf 开始执行 shellcode
6. shellcode 调用 execve("/bin/sh") → 拿到 shell

这个配方在 90 年代到 2000 年代初屡试不爽——Code Red / Slammer / Blaster 几个改变互联网历史的蠕虫,全是栈溢出。直到 OS 厂商加上了 DEP/NX,故事才进入下一章。

1.3 第三层原因:不安全函数至今没绝迹 ​

c
// 教科书级的"全家桶毒药"
gets(buf);             // ← C11 已经把它从标准里删了,但二进制还在
strcpy(dst, src);      // ← src 多长它拷多长
strcat(dst, src);      // ← 同上
sprintf(buf, fmt, x);  // ← 不知道 buf 多大
scanf("%s", buf);      // ← %s 不限长度

安全替代品早就有:strncpy / snprintf / fgets / strlcpy,但是:

  1. 老代码迁移成本高——Linux 内核 / glibc / OpenSSL 几千万行,改不完
  2. 新写的代码依然有人用——Stack Overflow 上的"快速答案"很多是不安全版本
  3. "安全版本"自己也有坑——strncpy 如果源串塞满了 N 字节,不会写 \0,后续读出来又溢了

教训:单靠"教育程序员"无法解决内存安全问题——这是为什么业界最终把希望寄托在「内存安全语言」上(见第七节)。

二、经典 stack smashing:把返回地址改写成想要的值 ​

2.1 最小 vulnerable C 代码 ​

c
// vuln.c —— 故意写错,只用于教学
#include <string.h>
#include <stdio.h>

void win(void) { puts("you won!"); }  // 攻击者想跳到这

void greet(char *name) {
    char buf[64];
    strcpy(buf, name);   // ← 没检查长度,危险!
    printf("hello %s\n", buf);
}

int main(int argc, char **argv) {
    greet(argv[1]);
    return 0;
}

编译命令(只在自己机器上做练习):

bash
gcc -fno-stack-protector -no-pie -z execstack -O0 vuln.c -o vuln
#    ↑关闭Canary       ↑关ASLR ↑栈可执行(模拟90年代环境)

这条命令本身就在告诉你:现代编译器默认就开了几道保护,你要"复现 1996 年的世界"必须显式关掉——这就是后面要讲的 Canary / ASLR / NX。

2.2 栈布局:进入 greet 之后 ​

高地址                                       高地址
┌────────────────┐                          ┌────────────────┐
│  argv[1] 指针  │                          │  argv[1] 指针  │
├────────────────┤                          ├────────────────┤
│  返回地址 0x...│  ← main 调 greet 时压入  │  AAAAAAAA      │ ← 被覆盖!
├────────────────┤                          ├────────────────┤
│  保存的 rbp    │                          │  AAAAAAAA      │
├────────────────┤                          ├────────────────┤
│                │                          │                │
│  buf[64]       │  ← strcpy 写入起点       │  AAAAAAAA....  │ ← payload
│                │                          │                │
└────────────────┘ 低地址                   └────────────────┘
   正常状态                                    被打穿后

payload 的关键结构:

[ 64 字节填充 ][ 8 字节占 rbp 位置 ][ 8 字节 = win 函数地址 ]
       ↑                ↑                      ↑
   随便什么字符     随便什么字符        关键 —— greet 返回时跳到这

用 objdump -d vuln | grep win 拿到 win 函数地址,假设是 0x401156,填到第 73-80 字节(小端序)。运行:

bash
./vuln $(python3 -c 'print("A"*72 + "\x56\x11\x40\x00\x00\x00\x00\x00")')
# hello AAAAA...
# you won!  ← 控制流被劫持,跳到了 win

这就是栈溢出的本质——任何能控制返回地址的人,都能控制 CPU 接下来执行哪段代码。

2.3 90 年代版:注入 shellcode ​

「跳到 win」只是教学版,真实攻击是把一段机器码(shellcode)塞进 buf,然后把返回地址覆盖成 buf 的地址:

栈布局(经典 shellcode 版):
┌──────────────────────┐
│  返回地址 = buf 地址 │ ← greet 返回时跳到 buf
├──────────────────────┤
│  保存的 rbp(被盖)  │
├──────────────────────┤
│  buf[64]:            │
│  [ NOP NOP NOP ...]  │ ← NOP sled,落点稍微偏一点也没事
│  [ shellcode       ] │ ← execve("/bin/sh", 0, 0) 的机器码
│  ...                 │
└──────────────────────┘

NOP sled 是个经典技巧——一长串 0x90(NOP 指令),只要返回地址落在 sled 任意位置,CPU 一路 NOP 滑到 shellcode。这是为了对抗"返回地址不好精确预测"。

shellcode 长什么样(只展示概念,不给完整):

push  0x68732f6e ; "n/sh"
push  0x69622f2f ; "//bi"   ← 栈上拼出 "//bin/sh"
mov   rdi, rsp   ; argv[0] = "//bin/sh"
xor   rsi, rsi   ; argv = NULL
xor   rdx, rdx   ; envp = NULL
mov   rax, 59    ; sys_execve
syscall

这段二进制只有 23 字节,塞进 64 字节 buf 绰绰有余。这就是为什么 90 年代「一个 gets,一台机器」完全不夸张——攻击者拿到一台 root shell,接下来横向移动整个网络都是后话。

三、shellcode 时代为什么结束了:DEP / NX ​

3.1 DEP / NX 是什么 ​

DEP(Data Execution Prevention) / NX(No-eXecute) —— Intel 和 AMD 在 2003-2004 年加进 CPU 的硬件特性,操作系统利用页表给每个内存页打三个权限位:R(可读)/ W(可写)/ X(可执行)。

原本的 x86:   只有 R / W 权限位,X 隐含 = R
现在的 x86_64: R W X 三个独立位 —— 一个页要么可写,要么可执行,不能同时

页表权限矩阵:

┌─────────────┬────┬────┬────┬────────────────┐
│  内存区域   │ R  │ W  │ X  │  说明          │
├─────────────┼────┼────┼────┼────────────────┤
│  .text 代码 │ Y  │ N  │ Y  │ 只读可执行     │
│  .rodata    │ Y  │ N  │ N  │ 字符串常量等   │
│  .data .bss │ Y  │ Y  │ N  │ 全局变量       │
│  栈          │ Y  │ Y  │ N  │ ← 关键!不可执行 │
│  堆          │ Y  │ Y  │ N  │ ← 关键!不可执行 │
└─────────────┴────┴────┴────┴────────────────┘

含义:栈上的数据再也不能被当作代码执行——shellcode 塞进 buf,CPU 跳过去时直接 segfault(权限错误)。1996 那一套配方瞬间失效。

3.2 看起来"完美"的防御,为什么没赢 ​

攻击者视角:既然栈上的数据不能执行,
            那就不"注入新代码",
            直接用"已经在内存里、本来就可执行"的代码!

关键洞察:程序的 .text 段和 libc 的代码段本来就是 X 权限——这些代码攻击者不能修改,但完全可以"跳过去执行"。

传统 shellcode:           ROP:
[ buf 里塞机器码 ]        [ buf 里塞一串地址 ]
[ 跳到 buf 执行 ]          [ ret 依次跳到 libc 里的代码片段 ]
       ✗ NX 拦死          ✓ 这些地址全是 X 权限的现成代码

这就是 ROP 的诞生——Return-Oriented Programming,2007 年 Hovav Shacham 在 CCS 论文里系统化提出(更早有 ret2libc / return-into-libc 的雏形)。

四、ROP:用别人的代码片段拼出新逻辑 ​

4.1 什么是 gadget ​

Gadget = 几条指令 + 一个 ret —— 这是 ROP 的最小积木。

随便从 libc 里挖到的几个 gadget 例子:
0x4011a3:  pop rdi ; ret              ← 把栈顶值弹给 rdi 然后返回
0x4011a5:  pop rsi ; pop rdx ; ret    ← 弹两个值
0x4011a8:  xor rax, rax ; ret         ← 清零 rax
0x4011ab:  mov rax, 59 ; syscall      ← 触发 execve syscall

关键洞察:ret 指令的语义是「从栈顶弹一个地址,跳过去」。如果攻击者控制了栈,就控制了下一个 ret 跳哪去。

攻击者把栈布置成这样:
┌──────────────────┐
│ gadget1 地址     │ ← 当前 ret 跳到这
├──────────────────┤
│ 给 gadget1 的参数│ ← gadget1 里的 pop 把这弹出去
├──────────────────┤
│ gadget2 地址     │ ← gadget1 的 ret 跳到这
├──────────────────┤
│ 给 gadget2 的参数│
├──────────────────┤
│ gadget3 地址     │
└──────────────────┘

每个 ret 都是一次"跳转 + 自动推进栈指针" —— 攻击者只要把栈控制权握在手里,就能把任意多个 gadget 串起来。这本质上是把 CPU 的执行流转化成了"对栈这个数据结构的遍历"。

4.2 一个最小 ROP chain:目标 execve("/bin/sh", NULL, NULL) ​

目标:在 x86_64 Linux 上调用 execve 系统调用拿 shell。需要做的事:

rax = 59      ; syscall number for execve
rdi = "/bin/sh" 的地址
rsi = 0       ; argv
rdx = 0       ; envp
syscall

ROP chain 大致结构(只画结构,不给可运行 payload):

栈布局(高地址在下,低地址在上,ret 一个个往下走):

┌──────────────────────────────────────────┐
│  填充字节 (64 + 8 字节,盖到返回地址前)  │
├──────────────────────────────────────────┤
│  &(pop rdi ; ret)        ←  gadget #1   │  函数返回时第一个 ret 跳到这
├──────────────────────────────────────────┤
│  &"/bin/sh"               ←  gadget #1 弹给 rdi
├──────────────────────────────────────────┤
│  &(pop rsi ; pop rdx ; ret) ← gadget #2 │
├──────────────────────────────────────────┤
│  0                        ←  弹给 rsi
├──────────────────────────────────────────┤
│  0                        ←  弹给 rdx
├──────────────────────────────────────────┤
│  &(pop rax ; ret)         ←  gadget #3
├──────────────────────────────────────────┤
│  59                       ←  弹给 rax
├──────────────────────────────────────────┤
│  &(syscall ; ret)         ←  gadget #4 —— 触发 execve
└──────────────────────────────────────────┘

CPU 视角:
  原函数 ret → 跳到 gadget#1 → pop rdi(从栈弹"/bin/sh"地址) → ret
                            → 跳到 gadget#2 → pop rsi=0, pop rdx=0 → ret
                            → 跳到 gadget#3 → pop rax=59 → ret
                            → 跳到 gadget#4 → syscall  → 内核执行 execve → 拿到 /bin/sh

这一切发生在"程序自己的代码段 + libc 代码段"里,没有一个字节是攻击者注入的可执行代码——DEP / NX 完全没拦住,因为没人执行栈上的数据,执行的全是合法代码段里的指令。

4.3 ROP 是图灵完备的 ​

Shacham 2007 论文的核心贡献:只要程序里 gadget 足够多(libc 这种大库够用了),ROP 就是图灵完备的 —— 能写循环、条件分支、内存读写,理论上可以用 gadget 拼出任意复杂的逻辑,不只是"调 execve"这种简单事。

这意味着:NX/DEP 没有终结攻击,它只是把攻击者从「写新代码」逼到了「用旧代码拼新逻辑」。攻击门槛升高了一档,但攻击能力没有本质削弱。

4.4 gadget 怎么找:ROPgadget / ropper ​

人工从二进制里扒 gadget 是噩梦,有现成工具:

bash
ROPgadget --binary /lib/x86_64-linux-gnu/libc.so.6 | grep "pop rdi"
# 0x000000000002a3e5 : pop rdi ; ret
# 0x000000000010f75b : pop rdi ; pop rbp ; ret
# ...几百条

ropper 是另一个同类工具,功能类似。libc 这种 2MB+ 的二进制里,通常能挖到几万个 gadget,足够拼任何逻辑。

有趣的细节:gadget 不一定是程序"原本写出来的指令",它可以是"任意字节序列被反汇编出来的副产物"。x86 是变长指令集,从一条指令的中间字节开始反汇编,得到的指令完全不同。攻击者会从这些"对齐错位"的字节里挖宝。这个 trick 在 RISC 架构(ARM、RISC-V)上不存在(指令定长),所以 ROP 在 ARM 上稍微难一点点,但依然能做。

五、ROP 的衍生:JOP / COP / SROP ​

ROP 一火,学术界立刻把"用别人代码片段拼逻辑"这个思路推广到各种变种:

5.1 JOP(Jump-Oriented Programming) ​

用 jmp rax / jmp [rax+8] 这种间接跳转代替 ret——目的是绕过那些"专门检测 ret 异常"的防御(早期 ROP 防御只盯着 ret)。Dispatcher gadget 负责调度,Functional gadget 负责干活。

ROP:   ret-driven —— 利用栈
JOP:   jmp-driven —— 利用某个寄存器当"虚拟 PC"

5.2 COP(Call-Oriented Programming) ​

用 call 指令而不是 jmp / ret——同样为了绕过"ret-only"的检测。

5.3 SROP(Sigreturn-Oriented Programming) ​

最优雅的一种——利用 Linux 的 sigreturn 系统调用。Linux 处理信号时会把全部寄存器状态压栈,信号处理结束后用 sigreturn 一次性恢复。攻击者只要伪造一个"signal frame"压在栈上,然后跳到 sigreturn,内核就帮你一次性设置所有寄存器。

普通 ROP:  要几十个 gadget 才能把 rdi/rsi/rdx/rax 都凑齐
SROP:      只要 1 个 sigreturn gadget + 一个伪造的 signal frame
           内核帮你一次性 set 所有寄存器

SROP 在 2014 年由 Erik Bosman 提出,论文标题就叫「Framing Signals - A Return to Portable Shellcode」——把这种"借力打力"的美学发挥到了极致。

六、现代缓解(20 篇会全部细讲,这里先建立心智) ​

6.1 ASLR(Address Space Layout Randomization) ​

栈、堆、libc、可执行文件每次加载到随机地址——攻击者算 gadget 地址时,不知道 libc 这次加载到哪。

不开 ASLR:                    开 ASLR:
libc 永远在 0x7ffff7a00000   libc 这次在 0x7f3a52400000
gadget 地址固定              下次在 0x7f8c31200000
exploit 一次写好永远能用       每次都得"先泄露 libc 地址"

绕过姿势:信息泄露(infoleak) —— 让程序漏一个指针出来,攻击者据此算出 libc 基址。90% 的现代漏洞利用都是"infoleak + ROP"两步走。

6.2 Stack Canary ​

编译器在每个函数的栈帧里、buf 和返回地址之间,塞一个随机值(canary,金丝雀)——函数返回前检查这个值是否被改动,改动了就 abort。

栈帧布局(开了 Canary):
┌─────────────────┐
│  返回地址        │
├─────────────────┤
│  保存的 rbp     │
├─────────────────┤
│  CANARY 0x...   │  ← 函数 prologue 设,epilogue 查,变了就死
├─────────────────┤
│  buf[64]        │
└─────────────────┘

strcpy 溢出 buf:
   要覆盖返回地址 → 必须先覆盖 Canary → 检查不过 → __stack_chk_fail → 进程死

绕过姿势:泄露 Canary 值 —— 一些格式化字符串漏洞 / 信息泄露原语能读出 Canary,然后写回原值再溢。fork 子进程的 Canary 和父进程一样,可以暴力爆破(每次错一位试)。

6.4 CFI(Control Flow Integrity) ​

编译器记录"哪些跳转目标是合法的"——每次间接调用 / 返回前,检查目标是否在白名单。理想中能拦死 ROP/JOP,实际部署因性能 / 兼容性,主流 Linux 还在推进。Windows 上有 CFG(Control Flow Guard)、Intel 有 CET。

6.5 Shadow Stack(影子栈) ​

每个线程额外有一个"只读"的影子栈,专门只记返回地址。call 时正常栈 + 影子栈都压;ret 时比较两者,不一致就死。

Intel CET 在硬件层做这事(Tiger Lake 之后的 CPU)、glibc 2.35+ 支持。这是栈溢出的"终局解药" —— 攻击者覆盖正常栈上的返回地址,影子栈不变,ret 时比较失败立刻 abort。

ASLR / Canary / NX / CFI / Shadow Stack 这五件套构成「现代二进制缓解栈」—— 每一道都能被单独绕,但叠在一起让 exploit 开发成本指数上升。详见 19 篇。

七、真实事故 & 为什么内存安全语言"赢了" ​

7.1 真实 CVE 一瞥 ​

  • CVE-2014-0160 Heartbleed(OpenSSL):严格说是堆越界读,不是栈溢出,但同源问题——C 没边界检查。一行 memcpy(bp, pl, payload); 让全世界的 HTTPS 服务器漏私钥。
  • CVE-2015-0235 GHOST(glibc gethostbyname):__nss_hostname_digits_dots 函数里的栈溢出,几乎所有 Linux 系统受影响。
  • CVE-2019-11932(WhatsApp GIF 解析):Android 上的栈溢出,只要收到一张恶意 GIF,WhatsApp 进程被劫持——一次"零点击"攻击。
  • CVE-2021-3156 Sudoedit Heap Overflow:Linux sudo 的堆溢出,十年老 bug,任意用户提权到 root。

共性:全是 C 写的核心基础设施 + 边界检查疏漏。从 1988 到 2024,根因没变过。

7.2 CTF Pwn 题:学栈溢出的合法操场 ​

CTF(Capture The Flag) 是合法练习栈溢出 / ROP 的正确场景:

  • PicoCTF / PWN College / HackTheBox / TryHackMe:有专门的 Pwn 分类,自己搭好的靶机
  • 典型题目难度阶梯:
    ret2win       —— 覆盖返回地址跳到一个现成的 win 函数(第二节那种)
ret2shellcode —— 关 NX 时,经典栈上 shellcode
ret2libc      —— 开 NX 但不开 ASLR,跳到 libc 的 system()
ROP           —— 开 NX + 部分 ASLR,自己拼 gadget
ROP + leak    —— 开全部缓解,先 leak libc 再 ROP

再次强调:这些技巧只在你自己控制的环境里练。对线上系统、别人的服务器跑这些,在中国是《刑法》285 条「非法侵入计算机信息系统罪」,在美国是 CFAA,轻则民事赔偿,重则刑事入狱。技术中立,行为不中立。

7.3 为什么 Rust / Go / Java 几乎免疫 ​

┌──────────┬─────────────────────────────────────────────┐
│  语言    │  内存安全机制                                 │
├──────────┼─────────────────────────────────────────────┤
│  C/C++   │  程序员手动管理 —— 默认不安全                  │
│  Rust    │  借用检查器 + 所有权(编译期保证,零开销)      │
│  Go      │  GC + 运行时数组越界检查(panic)              │
│  Java    │  JVM + GC + 数组越界检查(ArrayIndexOOB)      │
│  Python  │  解释器 + 引用计数 + GC                       │
│  Swift   │  ARC + 严格边界检查                            │
└──────────┴─────────────────────────────────────────────┘

核心差异:这些语言访问数组时,运行时(或编译期)会检查 index < length——越界直接 panic / exception / abort,根本写不进相邻内存。没有越界,就没有栈溢出。

代价:Java / Go 多一次比较 + 异常路径,性能损失约 5-10%。Rust 在编译期就证明了不越界,零运行时开销——这就是为什么 Rust 在系统编程领域势不可挡。

业界已经在用脚投票:

  • Linux 内核:2022 年开始接受 Rust 代码(Rust for Linux)
  • Microsoft:Windows 部分组件用 Rust 重写(2023)
  • Google:Android 13 起,新代码 Rust 占比超过 C/C++,2024 内存安全 bug 数量大幅下降
  • AWS Firecracker / Cloudflare workerd / Discord —— 性能敏感的核心组件全在重写成 Rust

从 1988 Morris 蠕虫到 2024 年,业界终于承认:"教育程序员小心"是个三十年都没解决的方案,换语言才是终局。详见 20 篇「内存安全语言为什么赢了」。

八、踩坑提醒 ​

  1. 以为「我用的是 strncpy 就安全了」——strncpy(dst, src, n) 如果 src 长度 ≥ n,dst 末尾不会有 \0,后续 strlen / strcpy 接力会越界。用 strlcpy 或 snprintf。
  2. 以为 Canary 拦死所有栈溢出——能泄露 Canary 的话(fork、格式化字符串、读越界)就能绕过。
  3. 以为 ASLR 一开就万事大吉——只随机化基址,模块内偏移不变;只要一个 infoleak 就重新可控。32 位系统熵不够,几小时能爆破完。
  4. 以为关掉栈可执行就没事——ROP / ret2libc 不需要可执行栈。
  5. 以为「我代码没调 strcpy」就安全——你依赖的库可能调,堆上的对象可能有指针,整个进程的安全是木桶。
  6. 以为 ROP 只能在 x86 上——ARM、RISC-V、MIPS 全部可以 ROP,只是 gadget 风格不同。手机、IoT 设备同样是攻击面。
  7. CTF 学到的 exploit 别拿到外网用——靶机是教学,线上是犯罪。一次冲动毁掉整个职业生涯,已经有大量真实案例。
  8. 以为 Rust = 绝对安全——Rust 有 unsafe 块、有 FFI 调 C 库、有逻辑漏洞,只是把"内存安全"这一类问题消灭了,其他漏洞依然存在。
  9. 以为旧 C 代码"重要但稳定就别动"——OpenSSL / Linux 内核每年都还在出栈溢出 CVE,"稳定"的代码不等于"安全"的代码。
  10. 以为这一章是"历史课"——2024 年的 CVE-2024-0193(Linux netfilter UAF)、CVE-2024-1086 等,内存安全漏洞依然在大量产出,只是攻击者再也不能"一行 payload 一台机器"了。
