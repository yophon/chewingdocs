# 认证授权漏洞:JWT 攻击 / OAuth 流程缺陷 / 会话固定 / 权限提升

讲认证授权漏洞最大的误区是「我用了 JWT / 我接了 OAuth,身份这块没问题」——**这种"我用了行业标准所以安全"的自信是这一层漏洞最大的温床**。JWT 默认配置依然能被打穿、OAuth 流程里随便漏一个参数就是账号接管、IDOR 这种"把 URL 里的 id 改一下"的漏洞每年还在 HackerOne 高赏金榜上反复出现。backendLearning/53 讲过 OAuth2 / OIDC 的**正确**流程,webLearning/35 浅讲过认证;**这一篇只讲攻击面——身份和权限这套机制是怎么被打的**,看完你应该能在白板前讲清「为什么 `alg: none` 二十年后还在被发现、为什么 OAuth 没有 state 参数等于送账号、为什么权限不能放进 JWT」。

> 一句话先记住:**认证(Authentication)= 你是谁;授权(Authorization)= 你能干什么——95% 的"账号被接管"漏洞,根因都是把这两件事搞混了**。JWT 解决了"你是谁"的传递问题但**不解决任何授权问题**;OAuth 解决了"第三方代理访问"但**它本身不是登录协议**(是 OIDC 才是);Cookie 是会话载体但**不会自动校验权限**。**这一层每个漏洞都源于一个错位:把传输机制当成了校验机制,把令牌存在当成了授权通过**。

---

## 一、认证 vs 授权:把这两件事搞混就是 90% 漏洞的开始

**认证(AuthN)= 你是谁;授权(AuthZ)= 你能干什么**——两件事完全独立。一个请求带着合法的 session,**只证明它是 user_123 发的**,不证明 user_123 有权限读 `/orders/9527`。**90% 的越权漏洞就是开发者写完认证以为授权也搞定了**。

还有个常见混淆:**认证 ≠ 身份**。OAuth 给你一个 access_token,**它证明的是"被授权访问某个 API",不证明"使用 token 的人就是这个用户本人"**。所以"前端拿 access_token 给后端换 session"是错的——任何能拿到这个 token 的恶意 client 都能冒充别人登录。**OIDC 在 OAuth 上加 `id_token` 才是登录协议**,`id_token` 里的 `aud` / `nonce` 才证明"这个人在我这登了"。

这一篇的攻击面地图:

| 层 | 典型漏洞 |
| --- | --- |
| 凭证生成 | 弱密码 / JWT 弱 secret 爆破 |
| 凭证传输 | `alg: none` / RS256→HS256 / 无 Secure cookie |
| 凭证存储 | XSS 偷 token / 日志泄露 |
| 流程绑定 | OAuth 缺 state / OIDC 缺 nonce |
| 会话生命周期 | 固定 / 劫持 / 注销不彻底 |
| 授权检查 | IDOR / 垂直越权 / 接口暴露 |
| 恢复链路 | 密码重置 token 可猜 / Host header 注入 / SIM swap |

## 二、JWT 攻击面:这玩意默认配置就能打穿

JWT 是个**容器**,不是安全机制。它的"安全性"完全取决于**签名算法、密钥强度、字段校验**——这三样里任何一样出问题,整个机制就崩。

### 2.1 JWT 结构速回顾

```
header.payload.signature

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjMiLCJyb2xlIjoiYWRtaW4ifQ.xxxx
   ↓ base64url 解码
header  = {"alg":"HS256","typ":"JWT"}
payload = {"sub":"123","role":"admin","exp":1699999999}
signature = HMAC-SHA256(header + "." + payload, secret)
```

**关键点**:`alg` 字段在 header 里——**也就是说,验证方"用哪个算法验签"是由 token 自己声明的**。这个设计是 JWT 一切问题的根源。

### 2.2 攻击一:`alg: none`

JWT 标准允许 `alg: none`——"这个 token 没签名,直接信"。**很多库默认行为是:`alg=none` 时跳过签名校验**:

```python
header  = {"alg":"none","typ":"JWT"}
payload = {"sub":"123","role":"admin","exp":9999999999}
token   = base64url(header) + "." + base64url(payload) + "."   # 空签名

# 漏洞代码
data = jwt.decode(token, verify=False)               # 关键漏洞
data = jwt.decode(token, key="", algorithms=None)    # 允许任意算法
```

**修复**:`algorithms` 显式白名单,**不传 `None` 不省略**:

```python
data = jwt.decode(token, key=PUBLIC_KEY, algorithms=["RS256"])
```

> CVE-2015-9235(node-jsonwebtoken)、CVE-2022-21449(Java ECDSA 全零签名)——**同一类问题十年里换不同语言反复出现**。

### 2.3 攻击二:RS256 → HS256 算法混淆

更高级——服务端用 RS256(非对称,公钥验签),攻击者把 token 改成 HS256(对称,密钥既签也验),**用服务端的公钥当 HMAC 密钥重新签名**:

```python
public_key = b"-----BEGIN PUBLIC KEY-----\n..."   # 本来就公开
header  = {"alg":"HS256","typ":"JWT"}              # 改成对称算法
payload = {"sub":"123","role":"admin"}
signature = HMAC-SHA256(header+"."+payload, key=public_key)

# 漏洞代码:根据 token 自己声明的 alg 选验签方式
def verify(token):
    if header["alg"] == "HS256":
        return jwt.decode(token, key=public_key, algorithms=["HS256"])
    elif header["alg"] == "RS256":
        return jwt.decode(token, key=public_key, algorithms=["RS256"])
```

**根因**:验签算法**不应该由 token 自己说了算**。**修复**:写死算法 `algorithms=["RS256"]`,库会在 alg ≠ RS256 时直接拒。更彻底:对称和非对称密钥用不同 kid,KMS 层保证"公钥不可能被当 HMAC key 用"。

### 2.4 攻击三:kid 注入

`kid`(key id)是 header 里告诉验签方"用哪个 key"的字段。**很多实现把 kid 当成数据库查询或文件路径**:

```json
{"alg":"HS256","kid":"key_1' UNION SELECT 'attacker_known_secret'--"}
{"alg":"HS256","kid":"../../../../dev/null"}
// /dev/null 内容为空 → 密钥 = "" → 攻击者用空字符串签名即可
```

**修复**:`kid` 只能是白名单 / UUID,**绝不拼 SQL 或路径**。

### 2.5 攻击四:弱密钥爆破

HS256 用一个对称 secret 签名。**如果这个 secret 是 `secret`、`changeme`、`jwt_key` 这种弱口令,离线穷举几秒钟搞定**——JWT 是**自包含离线令牌**,攻击者拿到就能本地几亿次/秒爆破。

**修复**:HS256 至少 256-bit 高熵(`openssl rand -base64 32`),优先 RS256 / EdDSA(非对称),密钥定期轮转。

### 2.6 把 JWT 当数据库用

```json
{"sub":"123","role":"admin","balance":1000000,"is_premium":true}
```

**最常见也最隐蔽的设计错误**——把权限 / 业务状态塞进 JWT,以为签名校验通过 = 这些字段可信。问题:

1. 签名只防篡改,不防过期数据——用户被降权后,旧 token 还能用到 exp
2. JWT 一旦签发**无法撤销**(无状态是它的卖点,也是它的命门)
3. `role: admin` 放 JWT,等于"权限要等下次登录才生效",踢管理员后他还能 admin 一小时
4. 业务字段(balance)更不能放,余额随时变,JWT 是快照

**正确做法**:JWT 只装"你是谁" + 过期 + 会话指纹,授权 / 业务字段后端实时查 DB。**口诀:JWT 装"身份",DB 装"权限"**。

### 2.7 JWT 还是 Session?

| 维度 | Session(server-side) | JWT(stateless) |
| --- | --- | --- |
| 撤销 | 删 redis 一行立即生效 | **无法撤销**,只能等 exp 或维护黑名单 |
| 扩展性 | 需要 session 共享 | 天然分布式友好 |
| 攻击面 | session fixation / hijack | alg 混淆 / 弱 secret / 字段污染 |
| 适用 | 单体 / 强一致 / 要能踢人 | 微服务短期 access token |

**经验**:用户长会话用 server session,服务间短期 access token 用 JWT(配短 exp + refresh token)。**一刀切用 JWT 几乎都是错的**。

## 三、OAuth / OIDC 流程缺陷:协议本身没错,实现错位就是账号接管

backendLearning/53 讲过正确的授权码流程:`client → auth server → callback → token`。**这一节只讲流程里的每一个字段被忽略掉会发生什么**。

### 3.1 redirect_uri 校验不严:账号接管最常见的姿势

OAuth 规定 client 发授权请求时带 `redirect_uri`,auth server 把 code 回调到这个地址。**地址必须严格白名单匹配**——校验任何宽松就是账号接管:

```
姿势 1 子域宽松:  白名单 *.victim.com → 注册 evil.victim.com 拿 code
姿势 2 路径宽松:  /callback/../../../redirect?to=evil.com  /callback?next=//evil.com
姿势 3 scheme 宽松: javascript://app.victim.com/cb%0aalert(1)  (只查 host 不查 scheme)
姿势 4 fragment:  https://victim.com/cb#//evil.com  (配前端开放重定向)
```

**修复**:严格字符串匹配不支持通配符、协议固定 https、每个 client 一份白名单不共享、**任何开放重定向漏洞都和 OAuth 漏洞等价**必须独立修。

### 3.2 state 缺失:授权码 CSRF / 账号关联劫持

`state` 参数的作用是**绑定授权请求和 callback**,防 CSRF。

```
正常流程:
   1. 用户点"用 GitHub 登录"
   2. client 生成 state=随机值,存进 session,跳到 GitHub
   3. GitHub 回调:?code=xxx&state=随机值
   4. client 校验 state == session.state,通过才换 token

没有 state:
   攻击者用自己的 GitHub 走到第 3 步拿到 code
   把 https://victim.com/oauth/callback?code=ATTACKER_CODE 钓给受害者
   受害者点了 → client 把"攻击者的 GitHub"绑到了"受害者的本地账号"
   → 攻击者用自己 GitHub 永久登录受害者账号
```

**这叫 OAuth 账号关联劫持**——不是偷凭证,反过来把你的本地账号接管成攻击者控制的外部身份。

**修复**:state 必须不可预测(`crypto.randomBytes(32)`)、绑 session、callback 必须校验。现代库默认会做,**自己拼 OAuth 流程的几乎都漏**。

### 3.3 隐式流程为什么被废了

OAuth 2.0 早期的 implicit flow 把 access_token 直接拼在 redirect_uri 的 fragment 里返回前端:

```
https://client.com/cb#access_token=eyJ...&token_type=bearer
```

问题:token 暴露在 URL fragment(浏览器历史 / Referer / 第三方 JS 都能拿)、没有 client_secret 校验、没法发 refresh_token、配合 XSS 一打一个准。**OAuth 2.1 已经废除 implicit**,SPA 现在用"**授权码 + PKCE**"——把 code 互换的安全性带回来,又不需要 client_secret。**看到 URL fragment 里直接给 access_token 的流程,2025 年要重写**。

### 3.4 OIDC:nonce 是干什么的

OIDC 在授权码流程之上加 `id_token`(JWT)证明用户身份。**`nonce` 防 id_token 重放**。client 发 `authorize?nonce=ABC`,auth server 把 nonce 嵌进 id_token 回来,client 必须校验:**签名(JWKS 公钥)+ aud == 自己 client_id + nonce == 自己发的 + iss / exp / iat**。

少校 nonce → 攻击者重放别处拿到的 id_token;少校 aud → 其他 client 的合法 id_token 能登你;少校 iss → 任何能发 JWT 的人都能伪造身份。**用成熟 OIDC 库(`openid-client` / `Authlib`),不自己拼 id_token 校验**。

### 3.5 PKCE 防的是什么

PKCE(Proof Key for Code Exchange)防的是**code 在传输途中被截获**——主要场景是移动 app / SPA(没有 client_secret 可以加固)。client 生成 `code_verifier`(随机)和 `code_challenge = SHA256(verifier)`,发起授权时带 challenge,换 token 时带 verifier,auth server 校验 hash 匹配。**就算攻击者从日志 / Referer / 恶意 app 截到了 code,没有 verifier 也换不到 token**。

**结论**:所有 OAuth 实现都该开 PKCE,公开 client 必须开,机密 client 也建议开作为纵深防御。

## 四、会话攻击:从颁发到注销每一步都有坑

### 4.1 会话固定(Session Fixation)

**攻击姿势**:

```
1. 攻击者访问 victim.com,拿到一个空会话 cookie:SESSIONID=AAA
2. 攻击者把这个 cookie 通过钓鱼链接 / XSS / URL 参数 强行写到受害者浏览器
   (古早实现支持 ?SESSIONID=AAA 这种 URL 传 session)
3. 受害者用这个 SESSIONID=AAA 登录,服务端把这个 id 关联到受害者账号
4. 攻击者用 SESSIONID=AAA 访问 → 现在他是受害者
```

**根因**:**登录后没有重新生成 session id**。攻击者预先种了一个 id,登录后这个 id 还是同一个,直接劫持。

**修复**:

```python
def login(user, password):
    if check(user, password):
        session.regenerate_id()   # 登录后强制换新 id
        session["user_id"] = user.id
```

- 所有框架都有 `regenerate` API(Django `cycle_key`、Spring `changeSessionId`、Express-session `regenerate`)——**登录、提权、MFA 通过这三个时机都必须调**
- 永远不接受 URL 里的 session id
- Cookie 设 `HttpOnly` + `Secure` + `SameSite=Lax`

### 4.2 会话劫持

拿到 session id / JWT 的姿势:**XSS 偷 cookie / localStorage**、**网络层窃听**(没 Secure 没 HSTS)、**Referer 泄露**(token 写 URL)、**日志泄露**(没脱敏 Authorization)、**浏览器历史 / 共享设备**。

纵深防御:cookie 必须 `HttpOnly + Secure + SameSite + __Host-` 前缀、token 短 exp(15min)+ refresh token 单独存 HttpOnly cookie、**不要放 URL,不要放 localStorage**、日志脱敏 Authorization/Cookie/Set-Cookie、异常设备指纹强制重认证。

### 4.3 注销不彻底

```python
# 漏洞代码:只删 cookie,server-side session 还在
@app.route("/logout")
def logout():
    response.delete_cookie("session")
    return redirect("/")
```

JWT 更糟——**完全删不掉**。修复:

```python
@app.route("/logout")
def logout():
    sid = session["sid"]
    redis.delete(f"session:{sid}")          # 服务端会话作废
    response.delete_cookie("session", ...)  # 客户端删
    revoked_jti.add(jwt_payload["jti"])     # JWT 加撤销名单
```

更细的:登出所有设备(遍历该 user 所有 session)、改密码强制所有会话失效、OAuth 登出要调 IdP 的 `end_session_endpoint`(不然 IdP 那边还是登录态)。

### 4.4 并发登录策略

允许多设备(给"会话列表"页能逐个踢)、互斥登录(银行类,新踢旧,小心被 DoS)、MFA 后强制单端(风控触发后只允许一个活跃 session)。

## 五、越权:IDOR 和垂直越权——简单到不像漏洞,但赏金永远在

### 5.1 IDOR(Insecure Direct Object Reference)= 水平越权

**漏洞模式**:接口用对象 id 作为参数,**只查 id 存不存在,不查这个 id 是不是当前用户的**。

```python
# 漏洞代码
@app.route("/orders/<int:order_id>")
@require_login
def get_order(order_id):
    order = Order.query.get(order_id)   # 只查 id
    return jsonify(order.to_dict())

# 攻击:
GET /orders/9527  → 我的订单
GET /orders/9528  → 别人的订单(也能看!)
GET /orders/9529  → 又一个别人的订单
```

**根因**:**`@require_login` 只保证"你登录了",没保证"这个对象是你的"**——经典的把认证当授权用。

**修复(短期)**:每个查询带上 owner 条件:

```python
order = Order.query.filter_by(id=order_id, user_id=current_user.id).first_or_404()
```

**修复(根治)**:**框架级别强制**——所有数据访问层(ORM / repo)都加 user_id 上下文过滤:

```python
# 类似 Rails 的 current_account 范围 / Django 的 manager
class OrderManager(models.Manager):
    def for_user(self, user):
        return self.filter(user=user)

# 业务代码只能 .for_user(...) 不能裸查
```

更彻底:**用不可枚举的 id**(UUID / ULID / 短哈希),让攻击者不能简单 `+1`——这是**降低攻击成本**,**不是修复**。

### 5.2 垂直越权:普通用户能调管理员接口

```python
# 漏洞模式:检查了登录,没检查是不是 admin
@app.route("/admin/users/<id>/delete", methods=["POST"])
@require_login
def delete_user(id):
    User.query.filter_by(id=id).delete()
```

**实战变种**:前端隐藏按钮后端不校验、HTTP 动词逃逸(POST 校验了 PUT 没校验)、大小写 / 末尾斜杠绕过路由匹配、调试 / 内部接口(`/api/internal/*`)网关配错公网能打。

**修复**:授权检查放到中间件 / 装饰器,**不要每个 handler 自己写**;所有路由**显式声明权限,默认 deny**;接口分级 public / authed / admin / internal,**网关层就把 internal 拒掉**。

### 5.3 IDOR 的现代变种

**GraphQL**:一个 query 能取整个对象图,**每个字段都要授权检查**(field-level authz)。

```graphql
query { user(id: 9528) { email phone orders { amount } } }
```

**批量接口**:`POST /api/orders/batch {"ids":[9527,9528,...]}`——很多实现循环里只查 id 不查 owner,**一次拿全站数据**。

### 5.4 真实案例的两条主线

- **Facebook / Instagram / Snapchat** 历年高赏金的 IDOR——internal API 改 id 看任意用户业务信息
- **国内 SRC** 上 IDOR 几乎是常驻类型——后台用户管理 / 订单详情 / 优惠券 / 发票 / 客服工单,**任何"业务对象 + id 参数"都是攻击面**

**经验**:**代码评审看到 handler 拿 id 参数,第一反应必须是"这个 id 怎么校验属于当前用户"**,校验缺了就是 IDOR。

## 六、密码重置 / MFA:身份恢复链路最脆弱

### 6.1 密码重置 token 可预测

```python
def gen_reset_token():
    return str(int(time.time()))   # 时间戳当 token,可猜

import random
random.seed(user.id)               # 伪随机 + 种子可推
return str(random.randint(100000, 999999))
```

**修复**:`secrets.token_urlsafe(32)` 起步,**绝不用 `random` 模块**。

### 6.2 Host header 注入:让重置链接发到攻击者

```python
def send_reset(user):
    link = f"https://{request.host}/reset?token={token}"  # 信任 request.host
    send_email(user.email, link)
```

攻击者把请求的 `Host: evil.com` 改掉,邮件里链接变成 `https://evil.com/reset?token=xxx`,受害者点了 token 就泄露给攻击者。**修复**:base url 从配置文件读,**不信任 `request.host`**。

### 6.3 邮箱注入:发件人 / 主题被污染

```python
send_email(to=user.email, subject=f"Hi {user.name}, reset your password")
# user.name = "Alice\nBcc: attacker@evil.com\n\nFrom: fake@bank.com"
```

邮件库不过滤 CRLF 就可能注入 SMTP header 抄送给攻击者或伪造 From。**用成熟邮件库,不拼字符串到 header**。

### 6.4 其他重置坑

token 不过期 / 用完不作废、重置后不强制重新登录、不通知用户(攻击者重置成功用户不知道)、**枚举漏洞**(对存在/不存在邮箱返回不同响应 → 攻击者批量枚举注册账号)。

### 6.5 MFA 绕过

| 绕过姿势 | 原理 | 修复 |
| --- | --- | --- |
| **SIM swap** | 攻击者社工运营商换 SIM 卡,接管手机号 → 收短信 OTP | 不依赖 SMS,用 TOTP / WebAuthn |
| **OAuth 替代** | "用 Google 登录"路径不走 MFA → 直接绕过 | 所有登录入口都强制 MFA / risk-based |
| **Remember device** | "记住此设备"功能被劫持(偷 cookie) | 设备指纹 + 异常检测 |
| **重置流程** | 密码重置不要求 MFA → 重置完直接登录 | 重置流程也要 MFA |
| **暴破 OTP** | 6 位数字 OTP 没限速 → 1M 次穷举 | 5 次错锁定,IP 限速,失败后强制重发 |
| **响应篡改** | 客户端校验:`{"mfa": true}` → 改成 false 绕过 | 服务端校验,永远不信前端 |
| **会话状态混淆** | "MFA 已通过"标记在错的会话上 | MFA 状态绑定 session_id + user_id |

**经验**:**MFA 是最后一道防线,不是唯一防线**——还是要做风控、限速、异常告警。

## 七、真实案例:这些坑工业界一直在踩

### 7.1 GitHub OAuth(2020):redirect_uri 校验绕过

GitHub 自己的 OAuth client 有个内部 redirect_uri 白名单,**有一段时间允许 `*.github.com`**——研究人员发现 GitHub Pages(`*.github.io`)虽然不匹配,但通过开放重定向链 + 子域配置错误,**能把 code 转发到攻击者控制的页面**。修复后 GitHub 收紧到严格匹配。

### 7.2 OAuth state 缺失(常驻型)

国内某大厂登录站点早期没有 state——攻击者用自己的微信号走流程到 code,把 code 通过钓鱼链接发给受害者,受害者点了之后**自己的本地账号被绑定到了攻击者的微信**,之后攻击者用自己的微信永久登录受害者账号。

### 7.3 JWT alg: none(Auth0,2018)

Auth0 的 SDK 在某个版本默认接受 `alg: none`——研究人员构造一个空签名 token 直接通过验签。同样的漏洞在 Node.js 的 `jsonwebtoken` 库历史上出过 CVE-2015-9235、PyJWT 历史上也出过类似版本。

### 7.4 IDOR 在 SRC 的统治力

- **Uber 2019**:GraphQL 查询能用任意 trip_id 看别人行程详情
- **多个国产 App**:订单详情接口 `/api/order/detail?id=` 改 id 看任意用户订单(姓名 / 地址 / 电话)
- **某金融 App**:发票接口能拉全公司发票

**共同点**:开发者**完全没忘做认证**(你必须登录才能调),**但把"登录 = 授权"等同了**。

### 7.5 CVE-2022-21449:Java ECDSA 全零签名

OpenJDK 的 ECDSA 实现接受 `r=0, s=0` 的签名通过校验——**等于任何 JWT 用 ES256 都能被一个全零签名打穿**。这是密码学库 bug 不是 JWT 设计问题,但**展示了"信任库的算法实现"也是攻击面**。

## 八、认证授权代码评审 checklist

把这份贴到 PR 模板里。**每一项都过一遍,任何一项不满足就 block 合并**。

### JWT

- [ ] `algorithms` 参数显式指定白名单,**绝不用 `None` 或省略**
- [ ] 对称密钥(HS256)至少 256-bit 高熵,**不用人类可读字符串**
- [ ] 优先非对称(RS256 / EdDSA),公钥从 JWKS 取并校验 kid
- [ ] `kid` 严格白名单,**绝不拼 SQL / 文件路径**
- [ ] 校验 `exp` / `iat` / `nbf` / `iss` / `aud`
- [ ] JWT 里**不放权限字段**,只放身份和过期
- [ ] 撤销机制:短 exp(<15min)+ refresh token + jti 黑名单
- [ ] 密钥**定期轮转**,JWKS endpoint 支持多 kid 共存

### OAuth / OIDC

- [ ] `redirect_uri` 严格字符串匹配,**不用通配符**(必要时只限末段)
- [ ] `state` 参数生成、存储、校验三步全做
- [ ] OIDC 校验 `nonce` / `aud` / `iss` / 签名
- [ ] **公开 client 一律开 PKCE**,机密 client 也建议
- [ ] 不使用 implicit flow(不存在 `response_type=token`)
- [ ] 第三方登录拿到的 access_token 不直接换本地登录,**用 id_token 才能登录**
- [ ] 登出调用 IdP 的 `end_session_endpoint`

### 会话

- [ ] 登录、提权、MFA 通过后**强制 regenerate session id**
- [ ] Cookie:`HttpOnly` + `Secure` + `SameSite=Lax/Strict` + `__Host-` 前缀
- [ ] 不从 URL 参数读 session id
- [ ] 登出**同时**作废 server-side session + 删客户端 cookie
- [ ] 改密码 / 异常告警时,作废所有 user 的活跃会话
- [ ] 日志脱敏 `Authorization` / `Cookie` / `Set-Cookie`

### 授权(IDOR / 垂直越权)

- [ ] 所有"拿 id 查对象"的 handler **必须带 owner 条件**
- [ ] 授权检查在中间件 / 装饰器,**不在每个 handler 内自己写**
- [ ] 默认 deny:所有路由必须显式声明权限,**不声明视为禁止**
- [ ] admin / internal 路由**网关层独立隔离**
- [ ] GraphQL **每个字段**都要授权检查
- [ ] 批量接口循环里**每个对象**都要授权检查

### 密码重置 / MFA

- [ ] 重置 token 用 `secrets` 模块,32 字节以上
- [ ] token 一次性,使用后立即作废
- [ ] 邮件链接 base url **从配置读,不读 `request.host`**
- [ ] 邮件 to / subject / body **不直接拼用户输入**
- [ ] forgot-password 接口**对存在/不存在的邮箱返回相同响应**
- [ ] MFA 不依赖 SMS,优先 TOTP / WebAuthn
- [ ] OTP 失败 5 次锁定,IP 级限速
- [ ] 所有登录入口(包括 SSO / 第三方)都走 MFA
- [ ] 密码重置流程**也要求 MFA**

## 九、踩坑提醒

1. **以为"用了 JWT 就安全"**——JWT 是容器不是机制,默认配置十年里能被打穿三次
2. **把权限放 JWT**——`role: admin` 进 token 那一刻,你就失去了"立即撤销权限"的能力
3. **以为"前端隐藏按钮"是授权**——所有授权检查必须在服务端,前端只是 UX
4. **以为"登录中间件"就够了**——`@require_login` 不是 `@require_owner`,差一个 IDOR
5. **OAuth 不带 state 就上线**——CSRF 直接送账号
6. **redirect_uri 用通配符**——`*.example.com` 几乎等于"任意 redirect"
7. **拿 OAuth access_token 登录**——access_token 不是身份证明,id_token 才是
8. **登出只删 cookie**——server-side session 还在,JWT 更删不掉
9. **密码重置链接信任 Host header**——邮件就发到攻击者去了
10. **SMS 当 MFA 主力**——SIM swap 攻击十年没消停过
