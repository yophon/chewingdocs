投机解码:用空闲算力一次推 k 个候选 ​

02 篇画完 Roofline 后这件事其实已经摆在台面上了:LLM decode 阶段算力远没跑满,瓶颈在 HBM 带宽。H100 FP16 算力 989 TFLOPS,decode batch=1 时实际跑出来不到 1%——99% 的 Tensor Core 空转,只是在等权重从 HBM 搬过来。换个角度:既然把模型从 HBM 搬一遍只能产出 1 个 token 太亏,那能不能一次 forward 出多个 token?投机解码(Speculative Decoding)就是答这个问题的工程化方案。这一篇拆它的心智、四种主流流派、什么时候真加速、什么时候反而拖慢。

一句话先记住:投机解码 = 用一个便宜的"草稿器"先猜 k 个 token,再用大模型一次 forward 验证,接受最长合法前缀。能加速的根因是 decode 是 memory-bound——大模型反正要被搬一遍,搬一次顺便算 k 个候选几乎不增加耗时,只要接受率够高就净赚。关键不在"会不会用",在"什么时候用收益正,什么时候是负优化"。

一、为什么这事能成立 ​

1.1 朴素 decode 是个浪费 ​

朴素 decode(每步 1 个 token):
   step 1:  HBM → SM (搬整个模型)  → 算一次 → 出 1 个 token
   step 2:  HBM → SM (搬整个模型)  → 算一次 → 出 1 个 token
   step 3:  HBM → SM (搬整个模型)  → 算一次 → 出 1 个 token
   ...

每一步,几十 GB 的权重要从 HBM 流过 SM 一次,只换 1 个 token。
权重的"使用密度"极低——这就是 02 篇说的 memory-bound 本质。

把多个候选 token 塞进同一次 forward,权重只搬一次,算出多个 logits:

投机 decode(每步 k 个候选):
   step:  HBM → SM (搬整个模型)  → 一次算 k 个位置的 logits
                                  → 验证草稿器给的 k 个候选有几个对
                                  → 接受 m 个 (0 ≤ m ≤ k)

权重还是只搬一次,但产出 m 个 token。算力增量?基本忽略
(decode 的 SM 利用率本来就只有 5-50%)。

净增益 = 多产出 (m-1) 个 token,代价 = 跑一次草稿器 + 一点点额外算力。只要草稿器够便宜、接受率够高,就赚。

1.2 一张图看朴素 vs 投机 ​

朴素 decode(每步 1 个 token):
  ──────────────────────────────────────────────→  时间
   T1 │ T2 │ T3 │ T4 │ T5 │ T6 │ T7 │ T8 │ ...
  ←Δ→
  每步 Δ ≈ 一次完整 forward(memory-bound,几十毫秒)

投机 decode(k=4,接受率 70%):
  ──────────────────────────────────────────────→  时间
   [D] T1 T2 T3 ✗  │ [D] T4 T5 T6 ✗  │ ...
  ←──── Δ_spec ───→
   D  = 草稿器 4 步(很快,小模型或并行头)
   T1-3 = 大模型一次 forward 验证,接受 3 个
   ✗  = 第 4 个被拒,从这里开始下一轮
  
  每轮平均产出 3 个 token,Δ_spec 略大于 Δ(多了草稿器开销和 k 个位置的算力),
  但远不到 3 倍。净加速 ≈ 2-3x。

注意:投机解码不改变最终生成的分布——通过精心设计的接受/拒绝采样(Leviathan 2023 论文证明等价于 target 模型本身的采样),输出和直接跑 target 是统计同分布的。这不是"近似加速",是"无损加速"。

1.3 接受率(acceptance rate)是命门 ​

设 k 为每轮草稿 token 数,α 为单 token 接受概率(假设独立)。
平均一轮接受 token 数:
    E[m] = (1 - α^(k+1)) / (1 - α)   (含贡献到下一步起点的那一个)

实际近似:E[m] ≈ k × α  (α 较低时偏低估,但量级对)

α=0.5, k=4:  E[m] ≈ 2 → 2x 加速
α=0.7, k=4:  E[m] ≈ 2.8 → 2.5x 加速  
α=0.85, k=8: E[m] ≈ 6 → 4-5x 加速
α=0.3, k=4:  E[m] ≈ 1.2 → 几乎不赚,草稿开销吃掉收益

所以"投机解码能不能上"的全部就是问一个问题:这模型 / 这任务的接受率能到多少。代码、数学、明确指令任务接受率 70-85% 常见;开放创作、长篇 narrative 接受率可能掉到 40-50%,得不偿失。

二、四种流派 ​

2.1 Speculative Decoding(原始版):独立小模型作 draft ​

Chen et al. 2023, Leviathan et al. 2023 同期工作。

target:  Llama-3.1-70B   (慢但准)
draft:   Llama-3.2-1B    (快,可能不准)

每轮:
  1. draft 自回归生成 k 个候选 token: t1, t2, ..., tk
  2. target 一次 forward 把 [prompt + t1..tk] 全部 K/V/logits 算出
  3. 从前往后比对:
     - 若 target 在位置 i 的分布认可 t_i(基于接受/拒绝采样)→ 接受
     - 首个被拒的位置 j:从 target 的位置 j 分布采样一个 token 替代
     - 后面的 t_{j+1..k} 全部丢弃
  4. 接受了 m + 1 个 token(m 个 draft 接受 + 1 个 target 替代/续作)

好处:不动 target 模型,任何 HF 模型都能套。 代价:

  • 需要找一个"风格相近"的小模型(同家族最理想,如 Llama 70B + Llama 1B / 8B)
  • 跨家族搭配接受率低(Qwen 当 Llama 的 draft 多半不行)
  • draft 模型占额外显存 + 自己也要 KV cache
  • draft 自回归 k 步也不便宜,k 选得不对反而慢

工程上能用,但有更优秀的接替者。

2.2 Medusa:不要 draft 模型,target 模型本体长几个头 ​

Cai et al. 2024。核心想法是:为什么要单独养一个小模型?在 target 模型的最后一个 hidden state 上,平行接几个"投机头",每个头预测未来第 i 个位置的 token。

target 模型最后一层 hidden state h
        │
   ┌────┴──────┬──────┬──────┐
   ▼           ▼      ▼      ▼
 LM Head    Medusa  Medusa  Medusa
 (位置 0)   Head 1  Head 2  Head 3
            (位置+1)(位置+2)(位置+3)

一次 forward,4 个 logits → 4 个候选 token (含位置 0 这个常规的)

好处:

  • 不需要独立模型,只在 target 上加 ~几亿参数的额外 MLP 头
  • 一次 forward 就出 k 个候选,省了 draft 自回归那 k 步
  • 训练成本低(冻结 target,只训 Medusa 头,几小时到一天)

代价:

  • 每个头独立预测,头 i 的预测不依赖头 i-1 的输出,长程接受率低
  • 实际把 k 个头展开成"树状候选"(Tree Attention),一次验证多个分支
  • 接受率比 EAGLE 略低,但实现简单

vLLM、TGI 都集成了 Medusa。

2.3 EAGLE / EAGLE-2:Medusa 的改良,接受率更高 ​

Li et al. 2024(EAGLE)、2024(EAGLE-2)。Medusa 的弱点是头 i 看不到头 i-1 的猜测;EAGLE 加一个小的自回归 draft 网络,但这个 draft 网络:

  • 输入是 target 模型的 hidden state(不是 token embedding)——拿到的信息丰富得多
  • 共享 target 的 LM head,词表分布天然对齐
  • 自回归生成,但是在一个轻量化的 1-2 层 Transformer 上跑
target hidden state h_t
        │
        ▼
   EAGLE draft (轻量 Transformer)
        │
   ┌────┴────────┬────────┬────────┐
   ▼             ▼        ▼        ▼
 next h_{t+1}  h_{t+2}   h_{t+3}  h_{t+4}
   │             │        │        │
   ▼             ▼        ▼        ▼
  LM Head (共享 target 的)
   │             │        │        │
   ▼             ▼        ▼        ▼
  token1        token2   token3   token4

EAGLE-2 进一步引入动态草稿树:不是固定 k 步深度,而是根据每步的概率自适应展开(高概率分支多猜几步,低概率早停)。

接受率:EAGLE 60-70%、EAGLE-2 70-85%(代码 / 数学场景能到 85%+),目前是接受率最高的方案之一。

2.4 Lookahead Decoding:不要 draft 模型,用 n-gram 表 ​

Fu et al. 2024。完全无需任何额外训练:用 prompt 和已生成内容里的 n-gram 共现统计当 draft。

模型已生成: "def binary_search(arr, target):\n    left, right = 0, len(arr)"
                                                ↑
当前位置之后,n-gram 表里 "0, len(arr)" 之后高频出现 "- 1"。
拿 "- 1" 当 draft 候选,target 一次 forward 验证。

好处:零训练成本、零额外显存、立刻可用、code/数学场景接受率不低。 代价:开放领域 / 通用 chat 接受率偏低(没有结构化模式让 n-gram 学到);加速比不如 EAGLE。

2.5 一张对比表 ​

方法训练成本推理额外开销显存额外典型加速比接受率
Speculative (独立 draft)无(用现成小模型)draft 自回归 k 步draft 模型 + 它的 KV1.5-2.5x50-70%
Medusa训 Medusa heads(数小时)一次 forward 出 k 头几亿参数的头2-3x50-65%
EAGLE / EAGLE-2训 draft Transformer(1-2 天)轻量 draft 自回归几亿到 1B 的 draft2.5-4x70-85%
Lookahead零n-gram 表维护(很轻)KB 级1.3-2x30-70%(任务依赖)

结论:

  • 零成本试水 → Lookahead
  • 想要最稳的加速 → EAGLE-2(已经在 vLLM / SGLang / TRT-LLM 都进了)
  • 快速集成不想训练 → Medusa(现成头权重也有发布)
  • 传统 speculative → 现在已经少用,被 Medusa / EAGLE 接替

三、为什么大 batch 下投机收益会消失 ​

这是最重要也最容易被忽略的一点。

3.1 大 batch 已经把 decode 拖出 memory-bound 区 ​

回到 02 篇的 Roofline:

decode batch=1:    算术强度 ≈ 2     远在拐点左侧,纯 memory-bound,算力空 95%
decode batch=32:   算术强度 ≈ 64    仍在左侧,但 SM 利用率已经到 30-50%
decode batch=256:  算术强度 ≈ 256   接近拐点,SM 利用率 70-90%

batch=1 时投机解码"白嫖"算力的逻辑成立,因为算力本来就没用。 batch=256 时算力已经被并发请求填满了——你再塞 k 个候选进同一次 forward,Tensor Core 已经满负荷,投机解码不再"免费",而是要真消耗算力。

小 batch (1-8):     投机解码 ≈ "搭便车",2-3x 加速,几乎纯赚
中 batch (16-64):   投机解码加速比缩水到 1.3-2x,看接受率
大 batch (128+):    投机解码可能 1.0x 甚至负优化——草稿开销 + 算力争抢

3.2 工程上怎么选 ​

你的服务 QPS / 并发是?
├── 低并发(单机 batch < 8):
│   交互式 ChatBot、本地助手、code completion
│   → 上投机解码,EAGLE-2 是当前首选
├── 中并发(batch 16-64):
│   多用户 API 服务,但每用户 TPOT 仍重要
│   → 看 SLO。如果 TPOT 优先,上 EAGLE-2;吞吐优先,不上
└── 高并发(batch 128+):
    流量极大的公共 API
    → 别上,大 batch + Continuous Batching 已经是最优
    → 反而要思考分流:把 TPOT 敏感请求走 spec server,
      吞吐请求走普通 server(disaggregated serving)

这就是为什么 OpenAI / Anthropic 这种巨流量场景,投机解码不是默认开——他们的 batch 早就把算力打满了。中小流量的自部署服务才是投机解码的甜区。

四、接受率为什么任务依赖 ​

4.1 高接受率任务 ​

代码:  "for i in range(" → "len("   下一个 token 强烈可预测
数学:  "因此, 答案是 "  → 数字或公式 token 可预测
模板:  JSON / XML / 表格生成,大量结构 token 确定
重复:  "上面的内容我再总结一下:"  下一段往往复述前文,n-gram 命中高

EAGLE-2 在 HumanEval / GSM8K 这些任务上接受率 80%+,加速 3-4x 实测可得。

4.2 低接受率任务 ​

开放创作:  "故事的下一段..."     分布发散,小模型猜不准
长篇推理:  CoT 长链思考          每一步推理选择多,接受率掉
多语言切换: 中英文混合代码注释    draft 模型可能没学到这种切换
角色扮演:  奇幻 / 拟人 / 风格化   越偏离训练分布,接受率越低

加速 1.2-1.5x 算正常,1.0x 不奇怪。上线前必须按业务真实流量跑 acceptance benchmark,别看论文数字就以为自己也能 3x。

4.3 如何监控接受率 ​

vLLM / SGLang 都暴露 spec_token_acceptance_rate 类指标:

平均接受率:    每轮接受 / 草稿 token 数 的滑动平均
分布尾部:      P10 接受率(差的请求烂到什么程度)
任务分桶:      按 prompt 类型(代码 / 数学 / 闲聊)分别看接受率

阈值经验:< 40% 整体接受率 → 关掉投机解码(草稿开销已经吃完收益)。

五、vLLM / SGLang / TRT-LLM 的支持现状(2026 视角) ​

5.1 vLLM ​

bash
# 启动时加 spec config
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Meta-Llama-3.1-70B-Instruct \
    --speculative-model meta-llama/Meta-Llama-3.2-1B-Instruct \
    --num-speculative-tokens 5 \
    --speculative-draft-tensor-parallel-size 1
  • 支持:独立 draft、Medusa、EAGLE、Lookahead
  • 配置成熟,文档齐全
  • 默认不开,需要显式启

5.2 SGLang ​

bash
python -m sglang.launch_server \
    --model-path meta-llama/Meta-Llama-3.1-70B-Instruct \
    --speculative-algorithm EAGLE \
    --speculative-draft-model-path .../eagle-checkpoint \
    --speculative-num-steps 5 \
    --speculative-eagle-topk 8 \
    --speculative-num-draft-tokens 64
  • 支持:Medusa、EAGLE / EAGLE-2、Lookahead
  • 跟 RadixAttention 协同好(共享前缀的请求在 spec 树上也能命中)
  • 与结构化输出叠加效果显著(grammar 限制让接受率更高)

5.3 TensorRT-LLM ​

  • 编译期就把 spec 决策 build 进 engine,运行时无 Python 开销
  • 支持 Medusa、EAGLE、Lookahead、传统 spec
  • 在 H100/H200 + FP8 上能把 spec 收益再压一档(算子更快 → 草稿验证开销更低)
  • 代价:换 spec 算法 = 重 build engine,迭代慢

六、踩坑提醒 ​

  1. 接受率上线前必须 benchmark——论文数字基于学术 dataset,生产流量可能完全不同。
  2. num_speculative_tokens 不是越大越好——典型 4-8 最佳。再大,接受率随位置衰减,后面几个白算。
  3. draft 模型必须同家族 + 同 tokenizer——Qwen draft 配 Llama target 不工作,词表不一样。
  4. 温度高的采样接受率会降——temperature 1.0 + top_p 0.95 时,target 分布发散,draft 不容易撞中。code 任务用 temperature 0 接受率最高。
  5. 多 LoRA 场景投机解码复杂——draft 模型也得知道当前用哪个 LoRA;目前主流引擎在 multi-LoRA + spec 的组合上还不够成熟。
  6. 长上下文场景接受率会衰减——seq_len > 32K 时,draft 模型未必被训过这么长,后段接受率掉。
  7. batch 自适应——成熟的引擎在 batch 大时自动关 spec 或减少 num_spec_tokens,看你用的引擎是不是开了这个开关。
  8. EAGLE 的 draft checkpoint 要单训——不同 target 对应不同 draft,不能跨模型复用。社区有 SpecBench / 公开 checkpoint 但模型覆盖有限。
  9. 接受率监控放在 dashboard 第一屏——这玩意会因为流量分布漂移悄悄掉。
  10. 不要拿 spec 解决 TTFT——spec 只加速 decode(出 token 阶段),prefill(首 token)那部分一点没省。

七、和下一篇的连接 ​

投机解码是算法层的 decode 加速:在不改 GPU 硬件、不动模型权重的前提下,挖出 memory-bound 留下的算力空隙。

下一篇要讲的 TensorRT-LLM 是编译层 + Kernel 层的加速:它通过 AOT 编译、算子融合、FP8 落地等手段,把 decode 一次 forward 本身做得更快。两者正交——TRT-LLM 上叠加 EAGLE,理论收益相乘。生产部署里,TRT-LLM + EAGLE-2 + FP8 是目前 H100/H200 上单卡推理的极限组合。

八、看完这一篇,你应该能 ​

  • 解释为什么 LLM decode 是 memory-bound,这是投机解码能成立的前提
  • 默写朴素 decode vs 投机 decode 的时间线对比图
  • 解释接受率(acceptance rate)为什么是关键指标,典型 70%+ 才有显著加速
  • 默写四种流派(原始 spec / Medusa / EAGLE-2 / Lookahead)的区别和适用场景
  • 解释为什么大 batch 下投机解码收益会消失(算力已被 Continuous Batching 填满)
  • 看到一个推理需求,能判断该不该上投机解码、应该上哪种
  • 知道线上必须监控接受率,< 40% 就关

下一篇:12 TensorRT-LLM 心智 — NVIDIA 的编译式推理栈,把 PyTorch 模型 AOT 编译成 engine 二进制,在 H100/H200 上靠 FP8、算子融合、Plugin 系统压榨最后 20-50% 性能。代价是 engine 与 GPU 型号绑定、迭代成本高——什么时候它真比 vLLM 香,什么时候是过度工程。
