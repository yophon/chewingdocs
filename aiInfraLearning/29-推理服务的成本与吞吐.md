# 推理服务的成本与吞吐:每千 token 多少钱

推理服务上线后,工程师每天面对的不是模型精度,是钱。**8 卡 H100 一小时 30 美元,跑 70B FP8 能扛多少 QPS,每百万 token 成本多少,SLO 怎么定才能不亏本**——这些问题在面试时是"了解"层级,在生产时每天都要回答。这一篇拉通讲清楚:推理性能指标怎么测、成本怎么算、SLO 怎么定、弹性扩缩用什么触发、加卡之前还能榨多少。

> 一句话先记住:**TTFT 由 prefill 决定,TPOT 由 decode 决定,Throughput 由 decode batch 决定;成本 = (GPU 时薪 × 卡数) ÷ (Throughput × 3600 / 1000);90% 团队加卡之前还有 30% 优化空间——先把 KV 量化、FP8 推理、Continuous Batching 调整好,再考虑买卡**。

---

## 一、四件套指标:必须先画时序图

一个流式请求的时序拆开看:

```
时间 →

  T0  ─── 请求到达边缘网关
   │
   │  ① queue 时间 (排队等 batch slot)
   ▼
  T1  ─── 进入 prefill 队列
   │
   │  ② prefill 计算 (整段 prompt 一次过 forward)
   ▼
  T2  ─── 第一个 token 产出 ★ TTFT = T2 - T0
   │
   │  ③ decode step 1 (生成第 2 个 token)
   ▼
  T3  ─── 第二个 token  ★ TPOT = T3 - T2
   │
   │  ③ decode step 2
   ▼
  T4  ─── 第三个 token  
   │
   │   ... 持续到 EOS 或 max_tokens ...
   │
  Tn  ─── 最后一个 token
  
  
  四件套:
    TTFT (Time To First Token)  = T2 - T0    用户感知"是不是卡住"
    TPOT (Time Per Output Token) = (Tn - T2) / (n - 1)   用户感知"打字速度"
    Total Latency               = Tn - T0 ≈ TTFT + (output_len - 1) × TPOT
    Throughput                  = Σ output_tokens / Σ wall_time   全服务 token 产出
```

**核心心智**(详见 03 篇 Prefill vs Decode):

| 指标 | 主导阶段 | 主要受谁影响 |
| --- | --- | --- |
| TTFT | Prefill | prompt 长度 / 排队深度 / chunked prefill 配置 |
| TPOT | Decode | batch 大小 / KV 字节数 / HBM 带宽 |
| Throughput | Decode | batch 大小 / context 长度分布 / KV 量化是否开 |

**用单一 latency 评价推理服务的报告,90% 都是没拆 TTFT 和 TPOT**——长 prompt 短输出的 RAG 场景和短 prompt 长输出的 chat 场景完全不一样。

## 二、TTFT 与 TPOT 的工程取舍

batch 大小是"调一个旋钮影响所有指标"的典型变量:

```
                                Throughput ↑
    batch 大           ─────────────────────────→  TPOT ↑(decode 一步搬更多 KV,变慢)
    (一次 decode 多请求)                         TTFT ↑(prefill 等更多请求凑齐)

    batch 小           ─────────────────────────→  TPOT ↓(decode 步数快)
    (基本来一个跑一个)                            TTFT ↓(prefill 不等)
                                Throughput ↓
                                
    Continuous Batching         同时容纳多请求,但每请求独立进出
    (vLLM/SGLang,详见 09 篇)   ──→ Throughput 高 + TPOT 中等 + TTFT 取决排队
```

vLLM 默认 `--max-num-seqs=256`,这是"同时能装多少请求在 batch 里"。**实际 batch 通常远小于这个值**——KV 显存是真正的硬约束,后面会算。

## 三、一张大表:Llama-3-70B FP8 在 8×H100 的真实数字

以 `--dtype fp8 --kv-cache-dtype fp8` + Continuous Batching + Chunked Prefill 配置,数字来自 vLLM 0.6.x 官方 benchmark 与公开 H100 实测的合集近似(实际数值随版本和负载浮动 ±20%):

```
模型:        Llama-3-70B-Instruct,FP8 权重,FP8 KV
硬件:        8 × H100 80GB,NVLink + IB
推理引擎:    vLLM,TP=8(张量并行,详见 16 篇),max-model-len=8192
```

**(A)固定 prompt=512 / output=128,看 batch 影响**

| max batch | 单请求 TTFT (ms) | TPOT (ms) | Throughput (tok/s) | 满负载 P99 TTFT |
| --- | --- | --- | --- | --- |
| 1 | 60 | 14 | 71 | 60 ms |
| 8 | 75 | 16 | 480 | 95 ms |
| 32 | 110 | 20 | 1500 | 220 ms |
| 128 | 280 | 28 | 4200 | 950 ms |
| 256 | 450 | 35 | 6500 | 1900 ms |

观察:**batch 从 1 → 256,Throughput 涨 90 倍,但 TTFT 涨 7-30 倍**——batch 是吞吐 vs 延迟的最关键旋钮。

**(B)固定 batch=64,看 prompt 长度影响 TTFT**

| Prompt 长 | output | TTFT (ms) | TPOT (ms) | Throughput (tok/s) |
| --- | --- | --- | --- | --- |
| 256 | 128 | 130 | 24 | 2700 |
| 1024 | 128 | 280 | 25 | 2600 |
| 4096 | 128 | 880 | 27 | 2400 |
| 16384 | 128 | 3100 | 32 | 2000 |
| 32768 | 128 | 6200 | 38 | 1700 |

**Prefill 是 O(S²) 的注意力 + O(S × d²) 的 MLP**——长 prompt 的 TTFT 几乎线性涨,Throughput 因 KV 占满 batch slot 也下降。

**(C)KV Cache 量化的影响**(同 batch=128,prompt=2048,output=512)

| KV dtype | 单 token KV | 可承载 batch | Throughput | TPOT |
| --- | --- | --- | --- | --- |
| FP16 | 320 KB | 96 | 3800 | 33 ms |
| FP8 | 160 KB | 192 | 5200 | 27 ms |
| INT4 | 80 KB | 384 | 5600 | 30 ms |

**KV 量化是吞吐的最大单点收益**(详见 23 篇)——FP8 KV 在精度几乎无损的前提下吞吐提 35%。

## 四、成本核算公式

```
每千 token 成本 ($/1k tokens) 
   = (GPU 时薪 USD/h × 卡数) ÷ (Throughput tokens/s × 3600 / 1000)
   = (单 H100 时薪 × 卡数) ÷ (Throughput × 3.6)
```

举一个具体的数:

```
8 × H100 80GB 云上时薪 ≈ $24/h(2025-2026 现货价范围 $20-$32)
Throughput = 5200 tok/s(上面表 (C),FP8 KV)

每千 token 成本 = (24 × 1) ÷ (5200 × 3.6)
              = 24 ÷ 18720
              = $0.00128 / 1k tokens
              = $1.28 / 1M tokens

(对照 OpenAI gpt-4o 当前定价 ≈ $5 / 1M output tokens
 ——自建一台 H100 跑 70B,output 端成本约是 OpenAI 的 1/4)
```

注意:**这是"满负载 + 已优化"的下限**——日均利用率 30% 的服务,每千 token 实际成本要乘 3。

### 4.1 不同卡型 + 不同模型的横向对照

| 配置 | 时薪 $/h | Throughput tok/s | $/1M tokens |
| --- | --- | --- | --- |
| 8×H100 / 70B FP8 | 24 | 5200 | 1.28 |
| 8×H100 / 70B BF16 | 24 | 2900 | 2.30 |
| 4×H100 / 32B FP8 | 12 | 6800 | 0.49 |
| 2×H100 / 7B FP8 | 6 | 12000 | 0.14 |
| 8×H200 / 70B FP8 | 30 | 7800 | 1.07 |
| 8×B200 / 70B FP4 | 50 | 18000 | 0.77 |
| 8×H100 / 405B FP8 (TP=8) | 24 | 1100 | 6.06 |

**几个直觉**:

- 模型越小,每 token 成本越低(线性以上)——能用 32B 解决的事,别上 70B
- H200 / B200 单卡贵 25-100%,但吞吐涨更多,**单 token 成本反而更低**——新卡上市时算账,经常发现升级卡比加旧卡便宜
- 405B 这种巨型模型每 token 成本高一个量级,**只用在低 QPS 高价值场景**

### 4.2 闲时利用率才是大杀器

```
理想情况:    24h × 满负载 → 每天 5200 × 86400 = 4.5 亿 tokens
实际情况:    
  - 早 8 点到凌晨 1 点是高峰,17 小时
  - 凌晨 1 点到早 8 点流量 < 20%,7 小时
  - 平均利用率 = (17 × 0.7 + 7 × 0.15) / 24 ≈ 53%
  
真实每天 token = 4.5 亿 × 53% = 2.4 亿
实际每千 token 成本 = 1.28 / 0.53 = $2.42 / 1M tokens
```

**夜间不缩容,30% 的 GPU 钱是白烧的**——后面 SLO 章节会讲怎么用闲时回收 + keep-warm 池治这块。

## 五、SLO:推理服务该承诺什么

### 5.1 三层 SLO

```
延迟类 SLO:
   P50 TTFT < 500ms        中位数,正常用户体验
   P95 TTFT < 1500ms       高峰期可接受
   P99 TTFT < 3000ms       极端长尾,允许偶发
   
   P50 TPOT < 30ms         "打字速度" 30 tok/s
   P95 TPOT < 60ms         流式输出可接受抖动
   P99 TPOT < 120ms        长尾(KV 抢占重计算时)

可用性 SLO:
   月度可用性 > 99.5%        约每月 3.6h 不可用
   错误率 < 0.5%             包含 5xx + 截断 + 重试
   
吞吐 SLO:
   单服务承诺 QPS / TPS    例 8 卡集群承诺 800 RPS
```

### 5.2 P99 长尾从哪来

P99 TPOT 突然飙到 200ms 几乎都是这几个原因:

| 长尾原因 | 触发条件 | 治法 |
| --- | --- | --- |
| **KV 抢占重计算** | KV 显存满,vLLM swap 部分请求出去再重 prefill | 加 KV 量化 / 减 max-num-seqs |
| **超长 prompt 占 batch slot** | 一个 32K prompt 进来,整 batch 一起等它 prefill | Chunked Prefill |
| **prefill 与 decode 互相挤** | 同 GPU 既跑 prefill 又跑 decode | Disaggregated Prefill-Decode(详见 30 篇) |
| **GPU SM clock throttle** | 散热不够,SM 降频 | DCGM 看 thermal,机房调温 |
| **NCCL allreduce 抖动** | TP=8 跨节点时 IB 链路抖 | 单节点 8 卡部署,避免跨节点 TP |
| **PCIe 总线竞争** | 数据 loader / nvme 读写挤 PCIe | NVMe over Fabric / 大 batch 预取 |

**P99 TPOT 的 80% 长尾来自前三个**——KV 是最常见的元凶。

## 六、自动扩缩容:不能用 GPU 利用率触发

K8s 默认的 HPA 用 CPU/GPU 利用率扩容,**对 LLM 推理几乎没用**——LLM 的 GPU 利用率(SM utilization)永远 80%+,因为 decode 步永远在搬权重 + 算注意力,SM 一直忙。

但 SM 忙不代表服务忙——可能只是 decode batch 把 SM 吃满,实际队列还很空。

**正确的扩容信号**:

```
✗ GPU 利用率              永远高,无信号
✗ CPU 利用率              低,无信号
✓ 排队长度 (queue size)    > N 持续 30s → 扩容
✓ Pending Token 数         > T 持续 30s → 扩容
✓ P95 TTFT                 > SLO 持续 60s → 扩容
✓ Continuous Batching 满载率  > 90% 持续 60s → 扩容
```

vLLM 暴露的 Prometheus 指标里几个关键的(详见 vLLM Metrics doc):

```
vllm:num_requests_waiting          ← 排队中
vllm:num_requests_running          ← 正在处理
vllm:gpu_cache_usage_perc          ← KV 占比
vllm:time_to_first_token_seconds   ← TTFT 直方图
vllm:time_per_output_token_seconds ← TPOT 直方图
vllm:request_prompt_tokens         ← prompt 长度直方图
vllm:request_generation_tokens     ← 输出长度直方图
```

KEDA / Custom Metrics Adapter + 这些指标做扩容,比 HPA 默认的 GPU 利用率合理 10 倍。

### 6.1 闲时回收的具体做法

```yaml
# Ray Serve / KEDA 配置思路示意
deployment:
  min_replicas: 2          # 永远保留 2 副本(避免冷启动)
  max_replicas: 16
  
  scale_up_when:
    - vllm_num_requests_waiting > 10 for 30s
    - p95_ttft_seconds > 1.5 for 60s
  
  scale_down_when:
    - vllm_num_requests_running < 4 for 5min
    - vllm_gpu_cache_usage_perc < 30 for 5min
  
  warm_pool:
    keep_warm_replicas: 1   # 比 min 多 1 个不接流量,扩容时秒级转正
```

**冷启动的痛**:LLM 副本冷启动要把 70B × 80GB 模型从对象存储拉到 GPU,**5-15 分钟**。所以:

- min_replicas 永远 ≥ 1,不要缩到 0
- 加 keep-warm 池(模型已加载,只是不接流量)
- 模型权重预热到本地 NVMe(避免每次拉对象存储)
- 用 vLLM 的 `--load-format=runai_streamer` / `--load-format=fastsafetensors` 加速加载

## 八、加卡之前先做的几件事

90% 的团队在 P99 TTFT 超 SLO 时第一反应是加卡——错。**先按以下顺序排查,通常能省 30-60% 的算力**。

### 8.1 检查清单

| 项 | 收益预估 | 难度 |
| --- | --- | --- |
| 1. 开 FP8 推理(权重 + KV) | 1.5-2x throughput | 低,vLLM 一行参数 |
| 2. 开 KV 量化(FP8 / INT4) | 1.3-1.6x batch 容量 | 低 |
| 3. 开 Chunked Prefill | P99 TTFT 砍半 | 低 |
| 4. 开 Prefix Caching | 同 system prompt 场景吞吐翻倍 | 低 |
| 5. 调 max-num-seqs / max-num-batched-tokens | 10-30% throughput | 中,要压测 |
| 6. TP / PP 拓扑调整(单节点 TP=8,跨节点 PP) | 减少跨节点通信抖动 | 中 |
| 7. 投机解码(详见 11 篇) | 1.5-3x decode tok/s | 中,要选好 draft |
| 8. 模型蒸馏到更小尺寸 | 2-5x throughput | 高,需训练 |
| 9. 切到 SGLang(同 prompt 多请求) | RadixAttention 让多轮场景吞吐翻倍 | 中,生态切换 |
| 10. 加卡 | 线性扩容 | 钱 |

vLLM 的开关示例:

```bash
vllm serve meta-llama/Llama-3-70B-Instruct \
    --tensor-parallel-size 8 \
    --dtype fp8 \
    --kv-cache-dtype fp8 \
    --enable-prefix-caching \
    --enable-chunked-prefill \
    --max-num-batched-tokens 8192 \
    --max-num-seqs 256 \
    --max-model-len 8192
```

**这五个开关全开,大部分推理服务能省一半的卡**——再回头看是不是真的需要加卡。

### 8.2 常见误区

- **「我们 GPU 利用率 95%,需要加卡」** —— GPU 利用率永远 90%+,跟拥堵无关
- **「P99 TTFT 高,加卡能解决」** —— 多半是长 prompt 没切 chunk,或 KV 抢占,加卡没用
- **「加 batch 就好」** —— batch 加大 throughput 涨,但 TPOT 也涨,SLO 可能反而崩
- **「换 vLLM 0.6.x → 0.8.x」** —— 升级有时候反而 perf regression,先在 staging 压测
