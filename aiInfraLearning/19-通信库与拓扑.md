# 通信库与拓扑:NCCL、NVLink、InfiniBand、Ring All-Reduce

千卡训练里最反直觉的一件事:**80% 的时间不在算梯度,而在等通信**。一次反向 All-Reduce 常常比前向 + 反向 + 优化器的总和还长。13-18 篇讲完了「并行策略」——DDP / ZeRO / TP / PP / 3D 并行 / SP——这一篇讲「通信怎么物理打通」:集合通信原语、Ring 算法、NVLink/NVSwitch、InfiniBand 拓扑、NCCL 调优,LLM 训练真正卡死的那一层。

> 一句话先记住:**集合通信 = Ring 算法 + 硬件带宽 + 拓扑感知**。Ring All-Reduce 让带宽利用率逼近上限,NVLink 比 PCIe 快 7 倍,InfiniBand 把跨机延迟压到微秒级,NCCL 把这三件事自动做完——LLM 训练的整套通信栈,本质就是把数据放到「能用最快链路传到对端」的位置上。

---

## 一、为什么通信是真正的瓶颈

### 1.1 一组让人睡不着的数字

70B 模型 BF16 训练,假设 1024 张 H100,batch 切完后:

- 每步前向 + 反向算力时间:大约 200 ms
- 每步要 All-Reduce 的梯度量:140 GB(70B × 2 byte)
- 跨机 IB 单向带宽:理论 400 Gbps NDR ≈ 50 GB/s
- 朴素 All-Reduce 时间:140 × 2 / 50 ≈ 5.6 秒

算力和通信的比值已经倒挂了几十倍。**没有 NVLink + IB + Ring + Reduce-Scatter 这一整套技术堆,千卡训练根本跑不起来**——朴素方案下通信会比计算多一两个数量级。

### 1.2 训练里的通信都从哪来

| 并行类型 | 通信原语 | 频次 | 数据量 | 在哪 |
| --- | --- | --- | --- | --- |
| DDP / FSDP(13、15) | All-Reduce | 每个梯度桶 | 全量梯度 | 每步 |
| ZeRO-3(14) | Reduce-Scatter + All-Gather | 每层正反向 | 参数 + 梯度 | 每层 |
| Tensor Parallel(16) | All-Reduce(行切) / All-Gather(列切) | 每层算子 | 激活值 | 每个 TP 边界 |
| Pipeline Parallel(17) | Send / Recv | 每个 micro-batch | 激活 / 梯度 | 流水边界 |
| Sequence Parallel(18) | All-to-All(Ulysses) / Ring(Ring Attn) | 每层 attention | KV / 序列分片 | Attention 内 |

三句话总结:**TP 通信最频繁,DP 通信量最大,PP 通信量最小但延迟敏感**。这就是 18 篇说「TP 必须放机内、DP 可跨机」的物理原因。

## 二、四个集合通信原语

只有四个核心原语,记住就够。

```
1. Broadcast            一对多
   GPU0: [X]  →  All: [X]

      G0 → G1
       └→ G2
       └→ G3
       
2. Reduce               多对一
   All: [Xi]  →  GPU0: [sum/max/min]

   G0 G1 G2 G3
   X0 X1 X2 X3
    ↘ ↓ ↓ ↙
       G0
       sum

3. All-Reduce           多对多(每张卡都要 Reduce 结果)
   All: [Xi]  →  All: [sum]

   G0 G1 G2 G3
   X0 X1 X2 X3
   每张卡都拿到 X0+X1+X2+X3

4. All-Gather           多对多(每张卡都要拼接所有分片)
   GPU i: [Xi]  →  All: [X0,X1,X2,X3]

5. Reduce-Scatter       多对多(Reduce 之后切片分发)
   GPU i: [Xi]  →  GPU i: [sum 第 i 片]
```

工业实践里 90% 的训练通信都是 **All-Reduce / All-Gather / Reduce-Scatter** 三个,Broadcast 主要在初始化用。

## 三、Ring All-Reduce 与 Tree All-Reduce

### 3.1 Ring 算法:大消息之王

把 N 张 GPU 组成一个环:`0 → 1 → 2 → ... → N-1 → 0`。

阶段 1:Reduce-Scatter(N-1 步)

```
初始(每卡持完整向量,切 N 份):
  GPU 0:  [a0, a1, a2, a3]
  GPU 1:  [b0, b1, b2, b3]
  GPU 2:  [c0, c1, c2, c3]
  GPU 3:  [d0, d1, d2, d3]

Step 1 (each → next, 各发一片,累加):
  GPU 0 → 发 a0 给 GPU 1
  GPU 1 → 发 b1 给 GPU 2
  GPU 2 → 发 c2 给 GPU 3
  GPU 3 → 发 d3 给 GPU 0
  收到方累加到自己对应位置

Step 2、3 (再各发一片,继续累加)...

Step N-1 后,每张卡的某一片是完整的全 sum:
  GPU 0:  [_, _, sum, _]    ← 第 2 片完整
  GPU 1:  [_, _, _, sum]    ← 第 3 片完整
  GPU 2:  [sum, _, _, _]    ← 第 0 片完整
  GPU 3:  [_, sum, _, _]    ← 第 1 片完整
```

阶段 2:All-Gather(再 N-1 步)

每张卡把手里那片「完整 sum」沿环传一周,N-1 步后所有卡都拿到完整 4 片 sum。

```
总通信步数:2(N-1)
每步每卡发送量:total_size / N
每卡总流量:2(N-1)/N × total_size ≈ 2 × total_size
带宽利用率:N 越大越接近理论上限
延迟:正比于 N(每步要等环上下一跳)
```

### 3.2 Tree 算法:小消息之王

```
Reduce 阶段(log N 层):

         [0+1+2+3+4+5+6+7]
          ↑              ↑
    [0+1+2+3]        [4+5+6+7]
     ↑    ↑           ↑    ↑
   [0+1] [2+3]      [4+5] [6+7]
    ↑ ↑   ↑ ↑        ↑ ↑   ↑ ↑
    0 1   2 3        4 5   6 7

Broadcast 阶段(log N 层):
    Reduce 反过来,把 root 的 sum 发回叶节点

总步数:2 log N
延迟:正比于 log N(N 大时优势明显)
带宽利用率:不如 Ring(数据量小时无所谓)
```

NCCL 的 **Double Binary Tree**(2018+):同时跑两棵互补的树,一棵做 reduce 一棵做 broadcast,把带宽利用率从 50% 拉到接近 90%。**小消息(< 1MB)默认走 Tree,大消息(> 1MB)默认走 Ring**。NCCL 自己根据消息大小、拓扑、卡数选,你不用管。

### 3.3 一张表对比

| 算法 | 通信步数 | 带宽利用率 | 延迟 | 适用消息 |
| --- | --- | --- | --- | --- |
| Ring | 2(N-1) | 接近上限 | 线性 N | 大(梯度同步) |
| Tree | 2 log N | 中等 | log N | 小(参数同步、控制信息) |
| Double Binary Tree | 2 log N | 接近上限 | log N | 中-小 |
| Recursive Halving/Doubling | log N | 高 | log N | 中等 |

## 四、All-Reduce = Reduce-Scatter + All-Gather

这个等价是 ZeRO / FSDP 的理论基础,必须画清楚。

```
朴素 All-Reduce(完整向量到处传):
  每对 GPU 之间互发完整向量,通信量 O(N) per node

Ring All-Reduce(等价分解):

  Phase 1: Reduce-Scatter(N-1 步)
    G0 ──→ G1 ──→ G2 ──→ G3 ──┐
     └────────────────────────┘
    每步只传 1/N 数据
    结果:每张卡持有 1/N 的完整 sum

  Phase 2: All-Gather(N-1 步)
    每张卡把自己的完整片沿环传一周
    结果:每张卡持有完整 sum

总通信量:2 × (N-1)/N × total ≈ 2 × total(每张卡)
```

**ZeRO-3 / FSDP 直接利用这个等价**:
- 优化器状态、梯度、参数都按 rank 切片(等价于 Reduce-Scatter 切完后的状态)
- 用到时再 All-Gather 拼回来
- 反向算完梯度直接 Reduce-Scatter,不再 All-Reduce

**结果**:通信总量跟朴素 All-Reduce 相当,但显存从 O(model) 降到 O(model / N)。这就是 14 篇 ZeRO-3 的本质。

## 五、硬件层带宽心智

### 5.1 一张表把链路拉清楚(2026)

| 链路 | 位置 | 带宽(单向) | 延迟 | 备注 |
| --- | --- | --- | --- | --- |
| HBM3e | GPU 内 | 4.8 TB/s(B200) | < 100 ns | 显存到 SM,02 篇 |
| **NVLink Gen5** | **机内 GPU↔GPU** | **900 GB/s 双向(B200)** | < 1 μs | H100/H200 是 Gen4,450 GB/s 单向 |
| NVSwitch | 机内交换 | 全互联 | < 1 μs | 8 卡 SXM 任意两两同速 |
| **PCIe Gen5 x16** | 机内 CPU↔GPU、GPU↔NIC | **64 GB/s 单向** | μs 级 | 卡式 GPU 唯一选项 |
| **InfiniBand NDR** | 跨机 | **400 Gbps ≈ 50 GB/s** | < 2 μs | 当下主流 |
| InfiniBand XDR | 跨机 | 800 Gbps ≈ 100 GB/s | < 2 μs | 2024+ 新机房 |
| RoCE v2 | 跨机 | 与 IB 同速率 | 5-10 μs | 走以太网,便宜 |
| 普通以太网 TCP | 跨机 | 100 Gbps | ms 级 | 训练用不了 |

### 5.2 关键比例

```
NVLink Gen4 单向    : PCIe Gen5 x16     ≈   7 : 1
NVLink Gen4 单向    : IB NDR (50 GB/s)  ≈   9 : 1
HBM3e               : NVLink Gen5        ≈   5 : 1
```

**机内 NVLink 比 PCIe 快 7 倍,比 IB 快 9 倍**——这就是为什么 TP 必须留在 NVLink 域内。一旦 TP 跨节点走 IB,通信会让前向/反向时间翻几倍,直接死。

### 5.3 NVLink 和 NVSwitch 是不同的东西

- **NVLink** 是物理链路协议,直接连两张 GPU
- **NVSwitch** 是交换芯片,让 8(或 16)张 GPU **任意两两全互联**,不需要拓扑路由

8 卡 H100 SXM 节点里,GPU 之间任何两张直连带宽都是 450 GB/s 单向(Gen4)——不存在「邻居快、远距离慢」。**这是 SXM 形态相对 PCIe 卡的根本优势**。

PCIe 卡(H100 PCIe / 4090 / 5090):没有 NVSwitch,GPU 之间走 PCIe + NVLink Bridge(只两两直连一对),拓扑就乱了——TP 在 PCIe 卡上几乎不可行,只能 DP / PP。

## 六、典型 GPU 服务器拓扑

### 6.1 8×H100 SXM 标准节点

```
                    ┌─────────────────────────────┐
                    │       NVSwitch ×4           │
                    │   (NVLink Gen4 全互联)      │
                    └──┬──┬──┬──┬──┬──┬──┬──┬────┘
                       │  │  │  │  │  │  │  │
                    ┌──┴──┴──┴──┴──┴──┴──┴──┴───┐
                    │  GPU0 GPU1 ... GPU6 GPU7  │
                    │  (H100 SXM5 80GB / 96GB)   │
                    └──┬──┬──┬──┬──┬──┬──┬──┬───┘
                       │  │  │  │  │  │  │  │      PCIe Gen5 x16
                    ┌──┴──┴──┴──┴──┴──┴──┴──┴───┐
                    │ PCIe Switch (Broadcom PEX)│
                    └──┬──┬──┬──┬──┬──┬──┬──┬───┘
                       │  │  │  │  │  │  │  │
                    ┌──┴──┴──┴──┴──┴──┴──┴──┴───┐
                    │  IB NIC ×8 (CX-7 / CX-8)   │  ← 每张 GPU 配一张 IB 网卡
                    │  (400 Gbps NDR)             │
                    └────────────┬───────────────┘
                                 │
                          IB Spine Switch
                                 │
                    跨节点(同机柜 / 跨机柜)
```

要点:
- **GPU 之间走 NVSwitch**:8 张卡两两 450 GB/s,无拓扑差异
- **GPU 跟 NIC 走 PCIe**:每张 GPU 配一张 IB 网卡,跨机时 GPU 显存绕过 CPU 直接走网卡(GPU Direct RDMA)
- **CPU 几乎不参与训练**:CPU 主要做数据加载、调度、Checkpoint

### 6.2 GPU Direct RDMA

没 GPU Direct 时,跨节点通信要 4 步:GPU 显存 → CPU 内存 → NIC → 网络 → 对端 NIC → CPU 内存 → GPU 显存。

有 GPU Direct RDMA:**GPU 显存 → 本机 NIC → 网络 → 对端 NIC → 对端 GPU 显存**,两次 PCIe DMA,绕过 CPU 与主存。

```
没 GDR:                                有 GDR:
GPU mem → CPU mem → NIC → 网络         GPU mem → NIC → 网络
              ↑           ↑                          ↑
          上下文切换       DMA                       DMA

延迟:几十 μs                            延迟:< 5 μs
带宽:被 CPU/PCIe 反复倒手砍            带宽:接近 IB 线速
```

NCCL 自动协商:同节点同 PCIe 域内开 GDR,跨域可能 fallback。`NCCL_NET_GDR_LEVEL=2` 可以强制开 GDR(注意硬件得支持)。

### 6.3 多节点拓扑:Rail 优化

千卡集群一般用 **Rail-Optimized** 拓扑:

- 每张 GPU 配一张 IB,8 张 GPU = 8 张 IB
- 同 rank 的 IB 接到同一个 spine 交换机
- 跨节点 GPU0↔GPU0、GPU1↔GPU1 等同 rank 通信永远只过一跳交换机

```
节点 A:  GPU0─IB0  GPU1─IB1  ...  GPU7─IB7
                │       │              │
                ▼       ▼              ▼
节点 B:  GPU0─IB0  GPU1─IB1  ...  GPU7─IB7
                │       │              │
                ▼       ▼              ▼
         Spine0   Spine1    ...    Spine7   ← 8 个独立 spine
                                              Rail 之间不混
```

**好处**:All-Reduce 沿 rail 跑(GPU0 的数据全部从 GPU0 那条 rail 来),不用经多次跳转,带宽利用率大幅提升。
**坏处**:布线密度极大,机房成本高。

## 七、NCCL:LLM 训练的事实标准

### 7.1 NCCL 是什么

**NVIDIA Collective Communications Library**——所有主流训练框架(PyTorch DDP / FSDP / DeepSpeed / Megatron)的通信后端。

NCCL 干的事:
1. 发现拓扑(NVLink / NVSwitch / PCIe / IB)
2. 选算法(Ring / Tree / 哪条 rail)
3. 调度通信 kernel 在 GPU 上执行
4. 跨节点用 IB Verbs / RoCE,机内用 NVLink
5. 跟 CUDA stream 整合,通信和计算并发

### 7.2 启动一次 NCCL 通信看到什么

```python
import torch
import torch.distributed as dist

dist.init_process_group(backend="nccl")

tensor = torch.ones(1024, 1024, device=f"cuda:{rank}")
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
# 所有 rank 的 tensor 求和,结果广播回每张卡
```

底层:
- PyTorch 调用 NCCL 的 `ncclAllReduce`
- NCCL 根据消息大小选 Ring 或 Tree
- 在每张 GPU 上启动通信 kernel,用 NVLink / IB 传数据
- 完成后 stream 同步,计算 kernel 才能继续

### 7.3 调试和调优常用变量

```bash
# 启动训练前 export

export NCCL_DEBUG=INFO              # 看拓扑发现日志
export NCCL_DEBUG_SUBSYS=INIT,NET   # 只看初始化和网络

# 选择网卡
export NCCL_IB_HCA=mlx5_0,mlx5_1    # 指定 IB 网卡
export NCCL_SOCKET_IFNAME=eth0      # 指定以太网网卡(Bootstrap 用)

# GPU Direct
export NCCL_NET_GDR_LEVEL=2         # 强开 GDR(0=关,2=PHB,5=任意)

# 算法选择
export NCCL_ALGO=Ring               # 强制 Ring(默认 auto)
export NCCL_PROTO=Simple            # 协议(Simple / LL / LL128)

# 缓冲
export NCCL_BUFFSIZE=4194304        # 缓冲大小,影响小消息延迟

# 超时(防 hang 死)
export NCCL_TIMEOUT_S=600
```

### 7.4 看一段 NCCL_DEBUG=INFO 输出

```
NCCL INFO Channel 00/04 : 0 1 2 3 4 5 6 7
NCCL INFO Channel 01/04 : 0 1 2 3 4 5 6 7
...
NCCL INFO 8 coll channels, 8 p2p channels, 8 p2p channels per peer
NCCL INFO Topology detection complete
NCCL INFO Connected all rings
NCCL INFO Connected all trees
NCCL INFO NET/IB : Using [0]mlx5_0:1/IB [1]mlx5_1:1/IB ...
```

读懂 3 件事:
- **Channel 数量**:NCCL 把数据分多个 channel 并行传,榨干带宽
- **拓扑发现**:看到 `Connected all rings/trees`,说明 NCCL 正确识别了 NVSwitch
- **网卡选择**:看到 `NET/IB` 用了哪些 mlx 设备,确认 IB 没退化到 socket(socket fallback = 训练立刻死)

## 八、拓扑感知与故障排查

### 8.1 把通信放到正确的链路上

| 并行类型 | 通信量 | 推荐放置 | 理由 |
| --- | --- | --- | --- |
| TP(16) | 极大,每层 | **机内 NVLink/NVSwitch 域** | NVLink 比 IB 快 9 倍,跨机直接死 |
| PP(17) | 小,边界 | 跨机 IB | 通信量小,延迟可掩盖在气泡里 |
| DP(13)/ZeRO(14) | 大,每步 | 跨机 IB,Rail 优化 | All-Reduce 沿 rail 走 |
| SP/Ulysses(18) | 中 | 优先机内,长上下文不得不跨机时配高带宽 IB | All-to-All 易出反压 |

**布局口诀**:**TP 不出节点,PP 沿机柜,DP 跨机柜**。这是千卡 H100 集群跑 70B+ 模型的标配。

### 8.2 常见故障:IB 抖动 → 整批训练 hang

千卡训练里某节点一根 IB 链路抖动,会有这套症状:

1. NCCL All-Reduce 等不到数据,**整集群 hang 在 barrier 上**
2. nvidia-smi 看 GPU 利用率掉到 0,显存却占着不动
3. NCCL 日志:`ncclTimeout` 或 `unhandled cuda error`

排查顺序:

```bash
# 1. 找出哪个节点的 IB 出问题
ibstat                         # 看 LinkState、Phys State,正常都是 Active/LinkUp
ibping -S / ibping <peer>      # 双向 ping 测延迟

# 2. 看 NCCL 日志锁定 hang 在哪个 channel/通信
grep -E "Channel|hang|timeout" nccl.log

# 3. 看物理层错误
cat /sys/class/infiniband/mlx5_0/ports/1/counters/symbol_error
cat /sys/class/infiniband/mlx5_0/ports/1/counters/port_rcv_errors
# 错误数飞涨 → 物理链路有问题(光模块、线、端口)
```

**生产做法**:
- Prometheus 抓 `ibstat` 和 NCCL counter,链路抖动告警
- 训练设 `NCCL_TIMEOUT_S=600`,自动 fail-fast,不至于 hang 一晚上
- 配合 27 篇讲的弹性训练框架,链路坏了能踢节点重启

### 8.3 监控核心指标

```
集群层:
  - 各节点 ibstat 状态
  - IB 端口错误计数(symbol_error / port_rcv_errors)
  - 节点 GPU 利用率(SM 利用率,不是显存)

训练层:
  - 每 step 时间(突涨 → 通信变慢)
  - All-Reduce 时间占比(NCCL profiling 或 Nsight Systems)
  - 跨 rank 时间标准差(某节点慢 = 长尾,整集群被拖慢)
```

NCCL 的 profile 接口能输出每次 collective 的时间,Nsight Systems 能可视化 NVLink/IB 流量与计算 kernel 的重叠。**调通信优化必看**。
