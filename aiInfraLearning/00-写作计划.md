# AI Infra 系列写作计划

> 来源:`未来系列规划.md` 第三类「纵深主题」第 L 项,⭐⭐⭐⭐。
> aiLearning 41 篇教了「**模型一侧**」——从感知机到 Transformer 到 RAG / Agent / 微调,把「模型是什么、怎么训、怎么用」讲透了;但都没回答一个核心问题:**一张 H100 单卡 80GB,怎么跑得动 70B 模型?一次训练几千张卡,通信怎么不拖死?推理服务上线后,TTFT 200ms / 输出 50 tok/s 怎么压?**
> **看完这层,你能站在白板前画出一家 AI 公司从 GPU 选型 → 训练集群 → 推理服务 → 成本核算的完整工程链**——vLLM / SGLang / DeepSpeed / Megatron / Ray / FP8 / PagedAttention 不再是「听过的名词」,而是知道每个东西解决什么瓶颈、什么时候上、什么时候是过度工程。

---

## 设计思路

六层结构,从「硬件心智」到「端到端工程化」:

```
硬件与心智(GPU / HBM / Tensor Core / 浮点格式 / 算力账)
   → 推理引擎(vLLM / SGLang / TRT-LLM / KV Cache / PagedAttention / 连续批 / 投机解码)
   → 训练框架与并行(DDP / ZeRO / FSDP / TP / PP / 3D 并行 / NCCL / 通信拓扑)
   → 量化、压缩与高效微调(PTQ / GPTQ / AWQ / FP8 / KV 量化 / LoRA 服务化)
   → 调度与平台(Ray / KubeRay / Slurm / K8s GPU / MIG / 弹性训练)
   → 端到端工程化(GPU 选型 / 成本与吞吐 / 可观测性 / Agent 推理 / 未来)
```

**这个系列不是 PyTorch 教程**——`torch.nn` 那一层 aiLearning 讲过了,**默认你站在 2026 之后**:大模型推理是 vLLM / SGLang 的天下,训练是 Megatron + DeepSpeed,FP8 在 Hopper 上已经是标配,Ray 是 AI 工程链的事实编排层,KubeRay / SkyPilot 把云上 GPU 弹性变成现实。

不深入到 CUDA Kernel 内核源码、cuBLAS 汇编、PyTorch C++ 调度器,**而是讲清楚"AI 工程师 / 推理工程师 / Infra 工程师能选型、能调优、能排错、能给 CTO 算账的图景"**——你训一个模型、上一个推理服务、做一次成本估算,这些都站在这套工程链上。

## 写作模板

每篇遵守"为什么这层是瓶颈 → 心智模型 → 工程落地 → 替代方案与局限"的四段式:

```
1. 为什么这层是瓶颈
   - 没它的世界长什么样(上一代是怎么做的,死在哪)
   - 它解决的根本问题是什么(显存?吞吐?延迟?精度?成本?)

2. 心智模型(一张图)
   - 显存布局图 / 通信时序图 / 算力账,任选其一
   - 关键概念用最少的名词讲清楚
   - 公式只写一行,且解释每个变量的工程含义

3. 工程落地
   - 最小可跑的一段配置或代码(vLLM 启动参数 / DeepSpeed config / Ray Serve)
   - 真实场景:这家公司为什么这么用,踩过什么坑
   - 调优 / 监控 / 故障排查的关键指标

4. 替代方案与局限
   - 同代位竞争者(vLLM vs SGLang vs TRT-LLM / DeepSpeed vs FSDP / Slurm vs K8s)
   - 什么场景不该用它
   - 现在 2026 之后业界往哪个方向走
```

不抄官方文档参数表——**只贴关键的 10-30 行配置或代码**,让读者"看到一次推理 / 一次训练的形状"就够了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. **不复述 Transformer 算子细节**——QKV、Softmax、FFN 在 aiLearning 13 讲过了,本系列只关心算子怎么落到显存与通信上
2. 「先讲个 GPU 编年史 / Volta → Ampere → Hopper → Blackwell」——直接进重点,代次只在性能特性差异时点名
3. 罗列 vLLM / DeepSpeed 全部 flag——只贴让读者"跑得起来 + 调得动"的最小子集
4. 「这是面试常考」——读者来这里是为了真懂瓶颈在哪,不是背名词
5. **不画图就讲 PagedAttention / 3D 并行 / All-Reduce**——这些东西**不画图永远讲不清楚**
6. 抄 NVIDIA / vLLM 官方 Blog——官方文档在那儿,这里只补它没讲透的「为什么」和「什么时候不要用」
7. 「vLLM vs TRT-LLM 谁更牛」式的口水战——只讲各自擅长的场景,选型留给工程判断
8. 不区分 Prefill 和 Decode 就讨论延迟——所有推理性能讨论都必须分两阶段,TTFT 和 TPOT 不能混为一谈
9. 写 CUDA Kernel 编程教程——本系列读者是「调度这些 Kernel 的人」,不是「写这些 Kernel 的人」

## 立场声明

本系列**只服务于工程视角**——能选型、能调优、能排错、能给团队和老板算账。

所有示例配置限制在最小可跑片段,**不教你怎么从零编译一个 vLLM、写一个 PagedAttention CUDA Kernel、或调 NCCL 通信库源码**——那是引擎团队 / Kernel 团队的事,本系列的读者画像是「上面用模型、下面踩 GPU 的应用 / 推理 / Infra 工程师」,你需要懂这些工具是怎么编排起来的,但不需要从 SASS 汇编读起。

**面向 2026:**

- 默认 NVIDIA H100 / H200 / B200 为主线,A100 作为存量背景,消费级卡(4090 / 5090)只在 11 / 21 单卡量化场景点名
- 默认你已经读完 aiLearning,知道 Transformer / KV / Tokenizer / SFT 是什么
- 默认你已经会用 Docker / K8s 的基础,本系列只补 GPU 调度的特殊性
- 默认主流栈是 PyTorch + vLLM/SGLang + DeepSpeed/Megatron + Ray,JAX / TPU 只在 16 / 25 作为对照出现
