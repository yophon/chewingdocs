# 3D 并行与序列并行:大模型训练的拼装

DP 装不下大模型,TP 跨机不行,PP 仍有气泡,ZeRO-3 通信量大——单一并行轴各有天花板。**真正训过 70B 以上的人都知道,只能把这几样组合起来用**。这就是 3D 并行(TP × PP × DP),405B / 1T 训练的标准配置。再加上长上下文(32K / 128K / 1M)训练的痛——激活显存随 seq 线性增长,Attention 计算随 seq 平方增长——还得引入第四个轴:**序列并行**。这一篇拉清楚 3D 并行怎么拼、序列并行三种方案怎么选。

> 一句话先记住:**3D 并行 = TP(机内 NVLink)× PP(跨机 IB)× DP(全局),三轴正交分配 GPU**——TP 优先吃满单机 8 卡,PP 切到塞得下,剩下卡数全分给 DP。长上下文再加一个序列轴(Megatron SP 省激活,Ring Attention 切 attention,Ulysses 用 head/seq 维互换)。配错一格,训练慢 5 倍。

---

## 一、为什么必须组合

### 一表对比四种单一并行的天花板

| 并行轴 | 切什么 | 收益 | 天花板 |
| --- | --- | --- | --- |
| **DP / DDP** | 切 batch,模型每卡一份 | 提吞吐 | 模型必须装下单卡 |
| **ZeRO-3 / FSDP** | 切优化器状态 + 梯度 + 权重 | 显存大降 | 单层权重必须能 All-Gather 装回单卡;通信量大 |
| **TP** | 切单算子的权重矩阵 | 单层无需装下单卡 | 通信量大,几乎不能跨机 |
| **PP** | 切层段 | 跨机可行,显存线性降 | 气泡;调度复杂 |

每条都有死穴,**只能正交组合各取所长**:
- TP 单机 8 卡内,吃满 NVLink
- PP 跨机,P2P 通信小,IB 撑得住
- DP / ZeRO 在最外层,提吞吐 + 进一步省状态显存

## 二、3D 并行的 GPU 拓扑

### 2.1 三个轴的语义

```
TP 轴(Tensor):   机内,正交切单层算子
PP 轴(Pipeline): 跨机,切深度
DP 轴(Data):     最外层,切 batch
```

每张 GPU 的位置是一个三元组 `(tp_rank, pp_rank, dp_rank)`。

### 2.2 必画图:64 卡 TP=8 × PP=4 × DP=2 拓扑

64 卡场景(8 节点 × 8 卡):

```
                                  DP 轴(2 副本)
                          ┌──────────── DP=0 ────────────┐
                          
                  PP=0    [节点 0: 8 卡 TP=8]   层 1-20
                  PP=1    [节点 1: 8 卡 TP=8]   层 21-40
                  PP=2    [节点 2: 8 卡 TP=8]   层 41-60
                  PP=3    [节点 3: 8 卡 TP=8]   层 61-80
                          
                          └──────────── DP=1 ────────────┐
                          
                  PP=0    [节点 4: 8 卡 TP=8]   层 1-20
                  PP=1    [节点 5: 8 卡 TP=8]   层 21-40
                  PP=2    [节点 6: 8 卡 TP=8]   层 41-60
                  PP=3    [节点 7: 8 卡 TP=8]   层 61-80


    单节点 8 卡内部(TP=8,例如节点 0):
    
       NVSwitch (900 GB/s 全互联)
              │
       ┌─┬─┬─┬─┬─┬─┬─┬─┐
       0 1 2 3 4 5 6 7   ← 这 8 张卡分了同一组层(层 1-20),
                          每张卡持 1/8 的算子权重
                          
       通信:层内 All-Reduce(走 NVLink)
       
       
    跨节点(PP 切点 + DP 同步,走 IB):
       PP P2P:    节点 0 ↔ 节点 1(传 forward 激活 / backward 梯度)
       DP All-Reduce: 节点 0 ↔ 节点 4(同 PP 段,梯度同步)
```

### 2.3 通信组的拆分

每个 GPU 同时属于 3 个通信组:

```
TP group:   (pp_rank=*, dp_rank=*) 固定,tp_rank=0..7   → 同一节点 8 卡内
PP group:   (tp_rank=*, dp_rank=*) 固定,pp_rank=0..3   → 跨节点(同 TP 列)
DP group:   (tp_rank=*, pp_rank=*) 固定,dp_rank=0..1   → 跨节点(同位置)
```

通信走的物理链路:

```
TP All-Reduce:    NVLink(节点内)        每层 forward 2 次
PP P2P:           IB(节点间)            每 micro-batch 边界一次
DP All-Reduce:    IB(节点间)            每步聚梯度一次
```

**TP 通信最频繁但走 NVLink 最快;PP 与 DP 通信少但走 IB**。配置错的话(TP 跨机,DP 全在节点内),立刻被通信拖死。

## 三、配置选型:从模型规模反推

### 3.1 一张表搞定主流场景

| 模型规模 | 集群 | 推荐配置 | 备注 |
| --- | --- | --- | --- |
| 7B | 单机 8×H100 | TP=1, PP=1, DP=8 | 纯 DDP / FSDP 够用 |
| 13B | 单机 8×H100 | TP=1, PP=1, DP=8 + FSDP | 单卡装不下完整,FSDP 切 |
| 30B | 单机 8×H100 | TP=2, PP=1, DP=4 | 或 FSDP-only |
| 70B | 2 机 16×H100 | TP=8, PP=1, DP=2 | TP 吃满单机,DP=2 跨机 |
| 70B | 4 机 32×H100 | TP=8, PP=2, DP=2 | 加 PP 摊更多内存 |
| 175B | 8 机 64×H100 | TP=8, PP=4, DP=2 | 经典 3D |
| 405B | 16 机 128×H100 | TP=8, PP=8, DP=2 | 或 TP=8, PP=16, DP=1 |
| 405B / 长上下文 | 16 机 128×H100 | TP=8, PP=8, DP=2, SP=on | 加序列并行省激活 |
| 1T | 32 机 256×H100 | TP=8, PP=16, DP=2 | + SP + 增量 checkpoint |

### 3.2 选型逻辑(口诀)

```
第 1 步:TP = 单机 GPU 数(几乎总是 8)
        — 吃满 NVLink,通信跑在最快的链路上

第 2 步:PP = 让(模型权重 + 梯度 + 优化器状态)/ (TP × PP) 装得下单卡
        — 跨机,通信小,可以拉到 8 / 16 / 32

第 3 步:DP = 剩下的 GPU 数
        — 最外层,提吞吐;DP 太大梯度同步开销也涨,实战极少 DP > 16
```

### 3.3 显存算账(70B 训练为例)

```
模型权重 FP16:     140 GB
梯度 FP16:         140 GB
Adam 状态 FP32:    560 GB(2× params 的 momentum + variance)
─────────────────────────────
总状态:            ~ 840 GB
```

不同配置摊到每张 H100 80GB:

```
TP=8, PP=1, DP=2 (16 卡):    840 / (8 × 1) = 105 GB / 卡  → OOM
TP=8, PP=2, DP=2 (32 卡):    840 / (8 × 2) = 52 GB / 卡   → 装得下
TP=8, PP=4, DP=2 (64 卡):    840 / (8 × 4) = 26 GB / 卡   → 宽裕,留给激活和 KV
```

**70B 训练为什么至少要 32 卡**:不是算力不够,是状态显存就要 840 GB,一张 H100 装不下完整,必须切到 ≥ 8×4=32 张卡。

## 四、长上下文的痛:为什么序列也要切

### 4.1 痛点

激活显存随 seq 线性增长,Attention 中间矩阵随 seq 平方增长:

```
单层激活(简化):
   [batch, seq, hidden] × 几个中间张量

   seq=4K:    1 × 4096 × 8192 × 2 bytes = 64 MB / 张量
   seq=32K:   512 MB
   seq=128K:  2 GB
   seq=1M:    16 GB     ← 一张卡几个张量就 OOM

Attention 中间:
   [batch, n_head, seq, seq] / FlashAttention 之外的实现
   seq=128K: 一个矩阵就几十 GB
```

**TP / PP 都不切 seq 维度**——再切也救不了长上下文。必须引入第四个轴:**序列并行**。

### 4.2 三种序列并行方案

```
1. Megatron Sequence Parallel(SP)
   在 TP 域内,LayerNorm / Dropout 阶段切 seq → 省非 Linear 阶段的激活
   
2. Ring Attention
   把 Attention 的 K/V 在卡间环形传 → 每卡只持 seq/N 的 K/V
   
3. Ulysses(DeepSpeed)
   在 Attention 入口处把 head 维和 seq 维互换切 → 通信极少
```

## 五、序列并行三种方案

### 5.1 Megatron Sequence Parallel(SP)

在 TP 已有的基础上,把 LayerNorm / Dropout / 残差这些**按元素**的算子额外按 seq 切:

```
TP 的 Linear 阶段:        张量 [batch, seq, hidden],seq 完整
SP 的 LN/Dropout 阶段:    张量 [batch, seq/TP, hidden],seq 切片

切换边界:
   SP → TP:   All-Gather seq 维(把 seq/TP 拼回完整 seq)
   TP → SP:   Reduce-Scatter seq 维
```

通信账:All-Gather + Reduce-Scatter 的总通信量 = 一次 All-Reduce,**与原 TP 通信量相等**。但激活显存省了 TP 倍。

```
没 SP:  每层激活  ∝  batch × seq × hidden
有 SP:  每层激活  ∝  batch × seq/TP × hidden
```

**Megatron-Core 默认开**,几乎是免费午餐,推荐永远开着。

### 5.2 Ring Attention

适合超长上下文(128K+),把 Attention 计算本身按 seq 切到 N 张卡:

```
卡 0 持有: Q[0:S/N], K[0:S/N], V[0:S/N]
卡 1 持有: Q[S/N:2S/N], K[S/N:2S/N], V[S/N:2S/N]
...

Attention 需要每个 Q 看到所有 K/V,所以:
   step 1:  卡 i 用本地 K/V 算自己 Q 的部分 attention
   step 2:  把 K/V 环形传到下一张卡(卡 i → 卡 i+1)
   step 3:  卡 i 用新收到的 K/V 继续累加
   ...
   step N:  K/V 转完一圈,Attention 算完
```

**关键性质**:
- 通信与计算可以重叠(传上一轮的 K/V 时算这一轮的 attention)
- 显存:每卡只持 seq/N 的 K/V
- 通信总量:O(seq) 张量,环形传 N-1 次

**适合场景**:训练 256K 以上上下文(Anthropic / Meta / Mistral 长上下文模型用的核心技术)。

### 5.3 Ulysses(DeepSpeed)

DeepSpeed 提出的另一种思路,**在 Attention 入口/出口处做一次 All-to-All,把 head 维和 seq 维互换切分**:

```
Attention 入口前:
   QKV 投影输出:  [batch, seq/N, n_head, head_dim]
                  按 seq 切到 N 张卡
                  
Attention 计算前(All-to-All):
   重排:         [batch, seq, n_head/N, head_dim]
                  改成按 head 切,每卡持 n_head/N 个完整 head
                  完整 seq 都在每张卡上,attention 自己算

Attention 出口后(All-to-All 反向):
   再切回:       [batch, seq/N, n_head, head_dim]
```

**关键性质**:
- 通信复杂度低:All-to-All 传输量 = O(batch × seq × hidden),与 N 无关
- Attention 计算时每卡有完整 seq,实现简单(标准 FlashAttention 直接能用)
- 缺点:每个 head 的计算不能再切了(head 数限制了 N 上限)

### 5.4 三种方案对比

| | Megatron SP | Ring Attention | Ulysses |
| --- | --- | --- | --- |
| 切的是 | LN/Dropout 的 seq | Attention 的 K/V | head 与 seq 互换 |
| 与 TP 关系 | 内嵌 TP | 独立第四轴 | 独立第四轴 |
| 通信模式 | All-Gather + Reduce-Scatter | 环形 P2P 传 K/V | 两次 All-to-All |
| 通信量 | = 一次 All-Reduce | O(seq) × (N-1) 次 | O(batch×seq×hidden) |
| 显存收益 | 激活 / TP | KV / N | 激活 / N(部分) |
| 适合 seq 范围 | 4K-32K | 128K+ | 32K-256K |
| N 上限 | TP 数(=8) | 任意 | head 数 |
| 实现成熟度 | Megatron-Core 默认 | Megatron / DeepSpeed 都有 | DeepSpeed 主推 |
| 复杂度 | 低 | 中-高 | 中 |

**实战经验**:
- 32K 以下:Megatron SP 够用
- 32K-128K:SP + Ulysses 组合
- 128K+:Ring Attention 是标配

## 六、HSDP:3D 之外的另一种组合

### 6.1 思路

**HSDP(Hybrid Sharded Data Parallel)= 机内 FSDP + 机间 DP**。在中等规模(70B 量级)训练中,FSDP 简单且足够,但完全 FSDP 跨机会被 IB 拖死。HSDP 的折中:

```
机内 8 卡:    FSDP,All-Gather 只在 NVLink 里
机间 N 节点:  DP,只做梯度 All-Reduce(频率低)
```

### 6.2 与 3D 并行对比

| | 3D(TP+PP+DP) | HSDP |
| --- | --- | --- |
| 实现复杂度 | 高(三个轴的拓扑+通信组) | 低(PyTorch FSDP 原生) |
| 模型规模上限 | 1T+ | 100B 左右 |
| 适合 | 大规模 + 团队有 Megatron 经验 | 中等规模 + PyTorch 团队 |
| 长上下文支持 | 加 SP / Ring / Ulysses | 加 Context Parallel(PyTorch 2.x) |
| 训练 70B 选哪个 | 想压到极限 | 想快速搭起来 |

**2026 现状**:头部厂商训百亿到 1T 规模仍以 Megatron 3D 为主;开源社区 + 独立团队中等规模训练 HSDP 越来越多。

## 七、工程经验:配置错一格,训练慢 5 倍

### 7.1 常见配错

```
错 1:  TP=8 跨机                        → IB 撑不住,GPU 利用率 < 10%
错 2:  PP 段切到 LayerNorm 前后断开     → 激活反复转换,通信翻倍
错 3:  DP 在节点内,TP 跨节点            → 通信组拓扑反了,慢 5-10 倍
错 4:  PP=4 但 micro-batch 数只有 4    → 气泡 43%,白白浪费
错 5:  SP 没开                          → 长上下文激活 OOM
错 6:  忘了 activation checkpointing   → 激活炸,batch 上不去
```

### 7.2 选型流程

```
1. 看模型规模(参数量)估状态显存(权重 + 梯度 + Adam ≈ 12 × params bytes)
2. 算需要多少卡:状态显存 / 单卡显存 × 安全系数(留 50% 给激活 + KV)
3. 单机 = 节点 × 8 卡 → TP=8 锁死
4. 跨机 → PP = ceil(总卡 / TP) / DP_target
5. DP = 剩下的卡数,通常 1-8
6. 长上下文 → 默认开 Megatron SP;> 128K 加 Ring / Ulysses
7. 用 Megatron-Core / DeepSpeed 推荐配置先跑通,再针对硬件微调
```

### 7.3 调试关键指标

```
GPU 利用率 (nvidia-smi sm util):  > 50% 算合格,> 70% 算优秀
TP All-Reduce 时间占比:           < 20% 算合格,> 30% 是配错
PP 气泡占比:                       < 10% 算合格(看 Megatron profiler)
Step time:                         与理论 FLOPs 估算的差距 < 2x
```

## 八、最小配置代码

Megatron-Core 风格,128 卡训 405B:

```bash
# 启动:16 节点 × 8 卡
torchrun \
  --nnodes=16 --node_rank=$NODE_RANK \
  --nproc_per_node=8 \
  --master_addr=$MASTER_ADDR --master_port=12345 \
  pretrain_gpt.py \
  \
  # 3D 并行
  --tensor-model-parallel-size 8 \
  --pipeline-model-parallel-size 8 \
  --num-layers-per-virtual-pipeline-stage 8 \
  \
  # 序列并行
  --sequence-parallel \
  --use-flash-attn \
  \
  # 模型规模(405B 量级)
  --num-layers 126 \
  --hidden-size 16384 \
  --ffn-hidden-size 53248 \
  --num-attention-heads 128 \
  --seq-length 8192 \
  \
  # batch 与显存
  --micro-batch-size 1 \
  --global-batch-size 2048 \
  --recompute-activations \
  --recompute-granularity selective \
  \
  # 优化器与精度
  --bf16 \
  --use-distributed-optimizer \
  --optimizer adam
```
