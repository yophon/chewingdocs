# 推理引擎景观:七大主流 + 一张选型矩阵

`model.generate()` 一行就能跑推理,但没人这么上线——单请求 30 tokens/s、batch 拼不起来、KV 显存碎片严重、长上下文一进来直接 OOM。LLM 推理引擎的存在,就是把"原始 PyTorch 推理"和"生产可用"之间的 100x 差距填上。这一篇盘点 2026 年主流的七个引擎,各自定位、关键技术、什么场景该上 / 不该上,以及一张选型决策树。后面 07-12 篇深入讲核心技术(KV Cache / PagedAttention / 连续批 / RadixAttention / 投机解码 / TRT-LLM)。

> 一句话先记住:**默认 vLLM(吞吐王 + 生态最大);多轮 / Agent / 强结构化换 SGLang;NVIDIA 大规模生产极致优化上 TensorRT-LLM;HuggingFace 生态闭环用 TGI;边缘 / Mac / CPU 用 llama.cpp;跨硬件(Apple / AMD / 移动 / 浏览器)用 MLC-LLM;国内 + 中文文档好选 LMDeploy**。

---

## 一、为什么不能直接用 transformers `generate`

回顾 03 篇的结论:推理 = prefill + decode 两阶段,decode 是 memory-bound,加 batch 是唯一杠杆。原始 transformers 库的问题:

```
transformers `model.generate`:
  - 静态批,batch 内所有请求一起进、一起出
    → 一个请求生成 1000 token,其他请求只生成 10,batch 也得等到 1000
  - 没有 KV 池管理,KV 显存碎片严重
    → 16 个并发可能比 1 个并发还慢
  - 没有 prefix 共享,system prompt 每次都重新 prefill
  - 没有连续批(continuous batching)
  - 没有调度器,排队和执行混在一个主线程
  - kernel 没融合,attention 用 PyTorch 实现的 5-10x 慢于 FlashAttention

总体:能跑通,生产上吞吐 1/10 ~ 1/30,延迟 5-10x。
```

LLM 推理引擎的存在意义,就是把上面这堆短板补齐。每个引擎补的姿势不同,定位也不同。

```
你的场景是?
│

│   ├── RAG 多 chunk 共享 / 长 system      → SGLang(RadixAttention)

├── 端侧 / Mac / 笔记本 / 树莓派
│   │
```

## 三、vLLM:吞吐王 + 生态最大,默认起点

UC Berkeley Sky Lab 2023 年开源,**PagedAttention 出处**——把 KV Cache 切成固定大小的 block 用页表管理,彻底解决了 KV 显存碎片(08 篇展开)。

核心组件:

```
vLLM 核心栈:
  - PagedAttention            KV 池 + 块管理(无内部碎片)
  - Continuous Batching       动态加入 / 退出 batch(09 篇)
  - Prefix Caching            扁平 hash 命中,system prompt 共享
  - Multi-LoRA               同一引擎挂多个 LoRA adapter,请求间路由(24 篇)
  - Chunked Prefill          长 prompt 切片,避免阻塞 decode
  - FlashAttention 集成       attention kernel 用 SOTA 实现
  - Speculative Decoding     投机解码内置(11 篇)
  - Quantization 支持         GPTQ / AWQ / FP8 / INT4 / INT8
  - 多硬件后端                CUDA / ROCm / Intel / 部分国产卡
```

启动一行:

```bash
pip install vllm
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.1-70B-Instruct \
    --tensor-parallel-size 2 \
    --gpu-memory-utilization 0.9 \
    --max-model-len 32768
```

**为什么是默认选**:

- **模型支持广度第一**:HF 上几乎所有 Transformer 当周或下周就支持
- **性能在多数场景下 SOTA 或接近**(单轮短回复落后 SGLang 不超过 10%)
- **OpenAI 兼容 API**,客户端一行不改
- **社区最活跃**,bug 修得最快,新模型适配最快(Llama / Qwen / DeepSeek 通常发版几天就有支持)

**短板**:

- 多轮场景的 KV 复用粒度粗(SGLang 的 RadixAttention 比它更聪明)
- 结构化解码靠外部库(outlines / xgrammar),不如 SGLang 内置
- 极致单卡吞吐略输 TensorRT-LLM(NVIDIA 自家 kernel 编译更狠)

**典型用法**:推理服务后端 + 多 LoRA 路由 + OpenAI 兼容客户端复用。**如果你不知道选什么,就用这个**。

## 四、SGLang:Agent / 多轮 / 结构化的最佳选择

LMSYS(写出 Vicuna、做 Chatbot Arena 的那帮人)2024 年开源,目标是"把复杂 LLM 程序当一等公民来跑"。三件武器:

```
SGLang 核心栈:
  - RadixAttention           基数树管理 KV,任意公共前缀复用
  - Compressed FSM           结构化解码批量追加确定 token
  - Python DSL 前端           多轮 / 分支 / 并行变成执行计划
  - Continuous Batching       同 vLLM
  - PagedAttention           同 vLLM(底层一致)
  - DeepSeek MoE 优化         在 V3 / R1 上常常比 vLLM 快 30-50%
```

启动也是一行:

```bash
pip install "sglang[all]"
python -m sglang.launch_server \
    --model-path meta-llama/Llama-3.1-70B-Instruct \
    --tp 2 \
    --port 30000 \
    --enable-radix-cache
```

**适合的场景**:

- Agent 多步交互,每步都共享前面 history
- RAG 多 chunk 检索后拼到 prompt(chunk 共享)
- Tree-of-Thought 一次开几个分支
- 强 JSON / Schema 输出(Compressed FSM 跳过确定 token)
- DeepSeek / Qwen MoE 模型(SGLang 团队跟得最紧)

**不擅长**:

- 纯单轮短 chat(树维护 + FSM 编译反而是开销)
- 多 LoRA 路由(支持比 vLLM 弱)
- ROCm / 国产卡(适配在跟进)

## 五、TensorRT-LLM:NVIDIA 官方的极致编译路线

NVIDIA 官方推理引擎,基于 TensorRT(NVIDIA 自家深度学习编译器)。**走的是"把模型编译成 NVIDIA GPU 上的最优 CUDA Graph + Kernel"**这条路。

```
TensorRT-LLM 核心:
  - Kernel Fusion            多个 op 融合成一个 CUDA kernel,减少 HBM 来回
  - CUDA Graph               推理流程预先编译,运行时几乎无 launch 开销
  - Plugin 系统              attention / GEMM 都有手写优化版
  - Quantization Toolkit     SmoothQuant / GPTQ / AWQ / FP8 全套
  - In-flight Batching       连续批
  - Triton Inference Server  配套服务化框架(production grade)
```

部署流程:

```bash
# 1. 把 HF 模型转成 TRT-LLM 引擎(一次性,慢)
python convert_checkpoint.py --model_dir hf_model --output_dir ckpt
trtllm-build --checkpoint_dir ckpt \
    --output_dir engines/llama-70b-fp8 \
    --gemm_plugin auto \
    --use_paged_context_fmha enable

# 2. 用 Triton 起服务
tritonserver --model-repository=models/
```

**关键代价**:

- **构建时间长**:一个 70B 模型 build 半小时到几小时
- **模型支持窄**:每个新模型要 NVIDIA 团队适配,通常晚 vLLM 几周到几个月
- **配置复杂**:几十个 flag,经常需要 NVIDIA 支持工程师帮调
- **硬件锁死 NVIDIA**:跨硬件几乎没有迁移路径

**真正用上 TRT-LLM 的场景**:

- 大规模商业部署,几千张卡稳定服务
- 单卡吞吐每多 5% 就值得花一周调
- 已经买了 Triton Inference Server 全家桶
- NVIDIA 直接派工程师来支持(头部客户特权)

**不该用的场景**:

- 模型频繁迭代(每次 build 半小时让节奏崩)
- 团队人少 / 没有 NVIDIA 人脉
- 模型太新,TRT-LLM 还不支持

12 篇深入讲它的设计哲学和"何时它真比 vLLM 快"。

## 六、TGI:HuggingFace 自家方案,生态闭环

Text Generation Inference,HuggingFace 出品,目标是"在 HF 生态里一行起服务"。

```
TGI 核心:
  - PagedAttention            (跟随 vLLM)
  - Continuous Batching       (跟随 vLLM)
  - 量化集成                  GPTQ / AWQ / EETQ / bitsandbytes
  - 多模型支持                HF 上模型几乎全支持(转换路径直接)
  - 服务化                    Rust 写的服务器,稳定性好
  - 流式输出                  WebSocket / SSE
```

```bash
docker run --gpus all -p 8080:80 \
  -v $PWD/data:/data \
  ghcr.io/huggingface/text-generation-inference:latest \
  --model-id meta-llama/Llama-3.1-70B-Instruct \
  --num-shard 2
```

**优势**:

- HuggingFace 全生态打通(Inference Endpoints / Spaces / Hub 一致)
- Rust 服务器稳定,工程友好
- 部署最简单(docker 一句)
- Apache 2.0 协议(2024 年从 HFOIL 改回来后)

**短板**:

- 性能持续落后 vLLM(2-3 个 release 周期)
- 创新滞后(很多优化是从 vLLM 抄回来的)
- 社区活跃度不如 vLLM

**适合**:已经深用 HuggingFace 全套(Spaces / Endpoints / Hub)的团队,要"一行起 + 稳"而不要极致性能。

## 七、llama.cpp:CPU / Mac 神器,GGUF 格式之父

Georgi Gerganov 一人项目(后来形成社区),**用纯 C/C++ 重写 Llama 推理,不依赖 PyTorch / CUDA**。最初是 "MacBook 上跑 Llama 7B" 的玩具,现在是边缘端事实标准。

```
llama.cpp 核心:
  - GGUF 格式                 自定义模型格式,带 metadata + 量化方案
  - 多种量化                  Q2_K / Q3_K_M / Q4_K_M / Q5_K_M / Q8_0 ...
                              (后缀 K 表示 K-Quants,精度损失更小)
  - CPU 优化                  AVX2 / AVX512 / NEON / Metal 全平台
  - GPU 加速                  CUDA / ROCm / Metal / Vulkan(全支持)
  - 极简部署                  单一可执行文件,无依赖
  - 模型加载快                 mmap 映射,几秒就能加载
```

```bash
# Mac 上跑 Llama 70B Q4_K_M(约 40 GB,M2 Ultra 192 GB 统一内存能装)
./llama-cli -m llama-3-70b-instruct.Q4_K_M.gguf \
    -p "Explain quantum computing." \
    -n 512 -ngl 99
```

**适合**:

- Mac 个人 / Apple Silicon 跑模型
- CPU only 服务器
- 树莓派 / 手机 / 嵌入式
- 隐私敏感场景(完全本地,无网络)
- 探索 / 玩具 / 个人项目

**不适合**:

- 大规模并发(吞吐远不如 vLLM)
- 多用户服务(单线程占主)
- NVIDIA GPU 集群(生态完全不同)

**GGUF 已成事实标准**:Hugging Face 上 `*.gguf` 文件占模型数量很大比例,Ollama / LM Studio / Jan / 各种本地 LLM 工具几乎全用 llama.cpp 后端。

## 八、MLC-LLM:跨硬件编译,TVM 派

CMU + MLC.ai 出品,基于 Apache TVM 编译器栈,**核心定位是"一份模型代码,编译到所有硬件"**。

```
MLC-LLM 覆盖:
  - NVIDIA GPU (CUDA)
  - AMD GPU (ROCm / Vulkan)
  - Apple Silicon (Metal)
  - Intel iGPU (OpenCL)
  - 移动端 (Android / iOS)
  - WebGPU (浏览器里直接跑 LLM)
```

特别的产物是 **WebLLM**——在浏览器里直接跑 7B / 8B 模型,不需要服务器。Demo 效果惊人,实际生产用得少。

```bash
# 编译模型到目标硬件
mlc_llm compile dist/Llama-3-7B-Instruct \
    --device cuda \
    -o dist/libs/Llama-3-7B-cuda.so

# 起服务
mlc_llm serve dist/Llama-3-7B-Instruct \
    --model-lib dist/libs/Llama-3-7B-cuda.so
```

**适合**:

- 跨多种硬件(尤其 AMD / 苹果 / 移动)
- 浏览器内 LLM(WebLLM 是唯一选择)
- 想"一份代码部署到客户多种端"的产品方

**短板**:

- NVIDIA 上性能不如 vLLM / TRT-LLM(TVM 编译路径长)
- 社区比 llama.cpp 小一截
- 编译流程相对复杂

如果你只跑 NVIDIA,用 vLLM;只跑 Mac,用 llama.cpp。MLC-LLM 的甜点是 **跨硬件 + WebGPU**。

## 十一、四个选型维度

```
1. 吞吐 (Throughput)
   单卡每秒能处理多少 token,直接决定每千 token 成本(05 篇)
   排序:TRT-LLM ≈ vLLM ≈ SGLang > TGI > LMDeploy > llama.cpp > MLC

2. 延迟 (Latency)
   TTFT 和 TPOT 两个维度(03 篇),长 prompt 看 TTFT,长生成看 TPOT
   多轮场景 SGLang TTFT 优势显著(命中 RadixAttention 后接近零 prefill)
   单轮场景 vLLM / TRT-LLM 几乎打平

3. 硬件
   NVIDIA only         → vLLM / TRT-LLM 任选
   AMD                 → vLLM (ROCm) / MLC-LLM
   Apple / 跨硬件      → MLC-LLM
   边缘 / CPU / Mac    → llama.cpp(第一选)

4. 生态
   迭代速度            → vLLM 最快(新模型几天就支持)
   工程稳定            → TRT-LLM(NVIDIA 官方支持)/ TGI(HF 工程团队)
   中文社区            → LMDeploy / vLLM(国内有 fork 维护)
```

## 十三、几个常见误区

1. **"vLLM 永远比 transformers 快 10x"**——错。短回复 + 低并发场景下,两者差距小到 2-3x;真正 10x 出现在中高并发 + 长生成场景。
2. **"TRT-LLM 永远是性能最好"**——错。新模型刚出时 TRT-LLM 还没适配,直接没得用;即使适配了,build 时间和工程复杂度让小团队根本玩不动。
3. **"SGLang 比 vLLM 新就一定更好"**——错。单轮场景 SGLang 没有优势,RadixAttention 维护开销在没共享前缀时是纯负担。
4. **"llama.cpp 在 NVIDIA 上也能用就行"**——能用但慢,llama.cpp 的 CUDA 后端是顺手实现,不是优化重点;NVIDIA 上还是 vLLM。
5. **"我们要支持所有引擎"**——错的方向。一个团队维护 1 个引擎已经很重,2 个就开始失控,3 个以上必然有引擎被冷处理出问题。**选一个主、一个备,别再多**。
6. **"决定选 vLLM 之后就万事大吉"**——版本升级、新模型适配、kernel bug 这些事仍要持续跟。**没有零维护的引擎选择**。
