# 并发陷阱：竞态条件与死锁分析

> **导读**：Go 并发 bug 最危险的地方不是必现崩溃，而是“绝大多数时候正常，压力一上来就随机出错”。

## 一、工程场景：偶发错误最难查

一个计数器本地压测没问题，上线后数据偶尔少几条；一个接口平时正常，高峰期突然卡死；一个任务明明已经结束，进程里的 goroutine 数量却一直上涨。

这些问题通常来自三类并发陷阱：

- 竞态条件：多个 goroutine 读写同一份数据，缺少同步。
- 死锁：所有 goroutine 都在等别人先动。
- goroutine 泄漏：goroutine 被永久阻塞，无法退出。

并发代码的第一原则是：共享状态必须有明确的同步边界，阻塞操作必须有明确的退出路径。

## 二、竞态条件：counter++ 不是原子操作

下面的代码看起来简单，但结果经常小于 1000。

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var counter int
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter++
        }()
    }

    wg.Wait()
    fmt.Println(counter)
}
```

`counter++` 至少包含读取、加一、写回几个步骤。多个 goroutine 同时执行时，彼此的写入会覆盖。

用 race detector 检查：

```bash
go run -race main.go
go test -race ./...
```

只要 `-race` 报 data race，就应该当成真实 bug 处理。

## 三、修复竞态：Mutex、Atomic、Channel

### 1. 用 Mutex 保护共享状态

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var counter int
    var mu sync.Mutex
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            mu.Lock()
            counter++
            mu.Unlock()
        }()
    }

    wg.Wait()
    fmt.Println(counter)
}
```

如果临界区内可能 return 或 panic，优先写成：

```go
mu.Lock()
defer mu.Unlock()
```

### 2. 用 atomic 处理简单计数

```go
var counter atomic.Int64

counter.Add(1)
fmt.Println(counter.Load())
```

`atomic` 适合简单数值和状态位，不适合复杂业务对象。只要多个字段之间有一致性要求，就应该用锁或更高层的数据结构。

### 3. 用 channel 串行化写入

```go
updates := make(chan int)
done := make(chan struct{})

go func() {
    total := 0
    for n := range updates {
        total += n
    }
    fmt.Println(total)
    close(done)
}()

for i := 0; i < 1000; i++ {
    updates <- 1
}
close(updates)
<-done
```

这类写法的优势是所有状态更新都发生在一个 goroutine 里，缺点是吞吐受单点处理能力限制。

## 四、死锁：所有人都在等

Go 运行时在某些场景下会直接报：

```text
fatal error: all goroutines are asleep - deadlock!
```

### 1. 无缓冲 channel 没有接收方

```go
func main() {
    ch := make(chan int)
    ch <- 1
    fmt.Println(<-ch)
}
```

无缓冲 channel 发送和接收必须同时准备好。修复方式是启动接收方，或改用缓冲 channel。

```go
ch := make(chan int)
go func() {
    ch <- 1
}()
fmt.Println(<-ch)
```

### 2. 忘记关闭 channel

```go
func main() {
    ch := make(chan int)
    go func() {
        ch <- 1
        ch <- 2
        // 忘记 close(ch)
    }()

    for n := range ch {
        fmt.Println(n)
    }
}
```

`range ch` 会一直等到 channel 被关闭。生产代码里要明确谁负责关闭。

### 3. AB-BA 锁顺序

```go
var a sync.Mutex
var b sync.Mutex

func f1() {
    a.Lock()
    defer a.Unlock()
    b.Lock()
    defer b.Unlock()
}

func f2() {
    b.Lock()
    defer b.Unlock()
    a.Lock()
    defer a.Unlock()
}
```

两个 goroutine 分别进入 `f1` 和 `f2` 时可能互相等待。解决方式是所有代码统一加锁顺序，或者把两个锁合并为一个更粗粒度的锁。

## 五、goroutine 泄漏：没有退出路径

泄漏不是内存直接泄漏，而是 goroutine 永远不退出，连带保留栈、引用对象和资源。

常见例子：

```go
func waitResult() <-chan string {
    ch := make(chan string)
    go func() {
        time.Sleep(2 * time.Second)
        ch <- "done"
    }()
    return ch
}

func main() {
    select {
    case v := <-waitResult():
        fmt.Println(v)
    case <-time.After(100 * time.Millisecond):
        fmt.Println("timeout")
    }
}
```

主 goroutine 超时退出后，后台 goroutine 仍然会尝试发送到 `ch`，但已经没人接收，最终阻塞。

修复方式是引入 Context，并且发送时也监听取消：

```go
func waitResult(ctx context.Context) <-chan string {
    ch := make(chan string)
    go func() {
        defer close(ch)
        select {
        case <-time.After(2 * time.Second):
        case <-ctx.Done():
            return
        }

        select {
        case ch <- "done":
        case <-ctx.Done():
            return
        }
    }()
    return ch
}
```

## 六、可运行排查示例：观察 goroutine 数量

下面的例子模拟错误写法和修复写法，能直观看到 goroutine 数量的差异。

```go
package main

import (
    "context"
    "fmt"
    "runtime"
    "time"
)

func main() {
    for i := 0; i < 1000; i++ {
        leak()
    }
    time.Sleep(200 * time.Millisecond)
    fmt.Println("after leak:", runtime.NumGoroutine())

    ctx, cancel := context.WithCancel(context.Background())
    for i := 0; i < 1000; i++ {
        noLeak(ctx)
    }
    cancel()
    time.Sleep(200 * time.Millisecond)
    fmt.Println("after cancel:", runtime.NumGoroutine())
}

func leak() {
    ch := make(chan int)
    go func() {
        ch <- 1
    }()
}

func noLeak(ctx context.Context) {
    ch := make(chan int)
    go func() {
        select {
        case ch <- 1:
        case <-ctx.Done():
            return
        }
    }()
}
```

生产排查时可以结合：

```bash
go test -race ./...
go tool pprof http://localhost:6060/debug/pprof/goroutine
curl http://localhost:6060/debug/pprof/goroutine?debug=2
```

## 七、关键坑位

### 1. 以为 map 并发读写安全

Go 内置 map 不是并发安全的。并发读写可能 panic：

```text
fatal error: concurrent map read and map write
```

高频读写可以用 `sync.RWMutex` 保护，读多写少且 key 稳定的场景可以考虑 `sync.Map`。

### 2. 复制带锁的结构体

包含 `sync.Mutex`、`sync.WaitGroup`、`atomic` 的结构体不要复制。复制后锁状态和数据状态会分裂，行为很难预测。

### 3. WaitGroup Add 时机错误

`wg.Add(1)` 应该在启动 goroutine 之前调用。否则 goroutine 可能先执行 `Done`，导致计数器变成负数。

### 4. 用 time.After 写长循环

在高频循环里反复 `time.After` 会创建大量 timer。需要可控停止时，用 `time.NewTimer` 或 `time.NewTicker` 并主动 `Stop`。

## 八、生产判断

并发代码上线前至少做这些检查：

- `go test -race ./...` 必跑，尤其是共享缓存、连接管理、后台任务。
- 每个 goroutine 都能说清楚退出条件。
- 每个 channel 都能说清楚谁发送、谁接收、谁关闭。
- 每把锁都能说清楚保护哪些字段。
- 不在持锁状态下做慢 IO，避免把临界区变成全局瓶颈。
- 对 goroutine 数量、队列长度、超时数设置监控。

竞态修复不是“加一把锁就完了”，而是重新明确数据所有权。死锁修复也不是“换成缓冲 channel 就完了”，而是重新设计等待关系。

## 九、总结

Go 给并发提供了很低的使用门槛，但工程代码需要更高的约束。共享变量要用锁、原子或单 goroutine 所有权保护；阻塞操作要有超时和取消；channel 的关闭职责要清晰；线上排查要依赖 `-race`、pprof 和监控。并发写得好，不是 goroutine 多，而是每个 goroutine 都有明确的边界和退出路径。
