# 并发陷阱：竞态条件与死锁分析

> **导读**：并发代码很容易在本地运行完美，上线后偶尔崩溃。学会使用 Go 的竞态检测工具是排障的必杀技。

## 一、竞态条件 (Race Condition)
多个 Goroutine 同时读写同一个变量，且没有加锁，最终结果取决于调度器的随机调度顺序。

```go
var counter int

func main() {
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            counter++ // 不是原子操作！
            wg.Done()
        }()
    }
    wg.Wait()
    fmt.Println(counter) // 结果往往小于 1000
}
```

**神器：`-race` 参数**
运行 `go run -race main.go` 或者 `go test -race ./...`。Go 会在运行时自动检测是否发生了数据竞争，并打印出精准的冲突堆栈！只要检测出了 Data Race，必须修（加锁、用原子操作或 Channel）。

## 二、死锁 (Deadlock)
当所有的 Goroutine 都处于阻塞状态（例如都在互相等待 Channel，或者互相等待锁）时，Go 运行时会直接报错崩溃：`fatal error: all goroutines are asleep - deadlock!`。

常见死锁场景：
1. **单协程读写无缓冲 Channel**：向无缓冲 channel 发数据，但没有另一个 goroutine 接收，当前 goroutine 永久阻塞。
2. **忘了解锁**：`mu.Lock()` 后，发生错误直接 `return` 忘了 `mu.Unlock()`（所以必须用 `defer mu.Unlock()`）。
3. **AB-BA 锁**：协程 A 拿了锁 1 去等锁 2，协程 B 拿了锁 2 去等锁 1。

## 三、Goroutine 泄漏
如果启动了一个 Goroutine，但它在某个 Channel 上一直阻塞等待（例如等待网络响应，但对方永远不回包，又没设置超时），这个协程及其占用的内存将永远不会被释放。
**解法**：始终使用带有超时的 `Context` 来控制阻塞调用。
