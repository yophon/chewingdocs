# 指针与内存布局基础

> **导读**: Go 保留指针不是为了让你手写内存管理,而是为了让数据共享、修改和性能成本变得可控。

Go 有指针,但没有 C/C++ 那种指针运算。你可以取地址、解引用、传指针,但不能把指针加 1 去访问下一块内存。这个设计很关键:Go 想保留指针带来的性能和表达能力,同时避免大量内存安全问题。

这一篇讲指针基础、值传递、`new` 与 `make`、逃逸分析、结构体内存对齐、nil 指针、指针接收者,以及工程里什么时候该传值、什么时候该传指针。

---

## 一、核心心智模型:Go 只有值传递,指针也是值

先记住:

> **Go 的函数参数永远是值传递。传指针时,复制的是地址这个值。**

例如:

```go
func f(p *int)
```

调用 `f(&x)` 时,函数收到的是 `x` 地址的一份拷贝。虽然地址被复制了,但它仍然指向同一块内存,所以可以通过 `*p` 修改原变量。

这解释了很多现象:

- 传 struct:复制整个 struct
- 传 `*struct`:复制一个地址
- 传 slice:复制 slice header,底层数组共享
- 传 map:复制 map header,底层哈希表共享
- 传 interface:复制接口值,动态值可能仍指向共享对象

理解"一切都是值传递"后,Go 的参数行为会简单很多。

---

## 二、`&` 和 `*`

两个基础操作符:

- `&x`:取变量 x 的地址
- `*p`:根据指针 p 访问它指向的值

可运行例子:

```go
package main

import "fmt"

func changeValue(n int) {
    n = 100
}

func changePointer(p *int) {
    *p = 100
}

func main() {
    x := 1

    changeValue(x)
    fmt.Println(x) // 1

    changePointer(&x)
    fmt.Println(x) // 100
}
```

`changeValue` 改的是副本。`changePointer` 通过地址找到原变量并修改。

指针类型写作 `*T`:

```go
var p *int
```

零值是 `nil`。nil 指针不能解引用:

```go
var p *int
// fmt.Println(*p) // panic
```

---

## 三、指针不能运算

C 里可以:

```c
p++;
```

Go 不允许:

```go
// p++ // 编译失败
```

这让 Go 避免了大量越界访问、悬挂指针、非法地址操作问题。你仍然能高效传递大对象,但不能随意在内存里乱走。

如果确实要做底层内存操作,Go 有 `unsafe` 包。但 `unsafe` 应该被隔离在很小的范围内,普通业务代码不要碰。

## 四、指针与结构体

结构体指针访问字段时,Go 会自动解引用:

```go
package main

import "fmt"

type User struct {
    Name string
}

func rename(u *User, name string) {
    u.Name = name // 等价于 (*u).Name = name
}

func main() {
    u := User{Name: "alice"}
    rename(&u, "bob")
    fmt.Println(u.Name)
}
```

这让代码比 C 更简洁。

方法接收者里也一样:

```go
func (u *User) Rename(name string) {
    u.Name = name
}
```

选择指针接收者的常见原因:

- 方法需要修改状态
- struct 较大,避免复制
- struct 包含锁、map、slice、buffer、连接等资源
- 希望所有方法接收者风格一致

## 五、new 与 make

`new` 和 `make` 都和分配有关,但语义完全不同。

### 5.1 new(T)

`new(T)` 分配一个 T 的零值,返回 `*T`:

```go
package main

import "fmt"

func main() {
    p := new(int)
    fmt.Println(*p) // 0

    *p = 10
    fmt.Println(*p)
}
```

对 struct:

```go
u := new(User) // *User
u.Name = "alice"
```

### 5.2 make(T)

`make` 只用于:

- slice
- map
- channel

它返回初始化后的 T 本身,不是指针:

```go
s := make([]int, 0, 10)
m := make(map[string]int)
ch := make(chan int)
```

map 用 `new` 是常见坑:

```go
package main

func main() {
    p := new(map[string]int)
    // (*p)["a"] = 1 // panic: map 还是 nil

    m := make(map[string]int)
    m["a"] = 1
}
```

一句话:

> **new 给任意类型分配零值并返回指针;make 初始化运行时结构并返回类型本身。**

## 六、栈、堆与逃逸分析

Go 程序运行时也有栈和堆:

- 栈:函数调用的局部空间,分配释放很快
- 堆:生命周期不容易静态确定的对象,由 GC 管理

Go 编译器会做逃逸分析,决定变量放栈上还是堆上。不是你写了 `&x`,它就一定在堆上;也不是你没写指针,它就一定在栈上。

例子:

```go
package main

type User struct {
    Name string
}

func NewUser(name string) *User {
    u := User{Name: name}
    return &u
}

func main() {
    _ = NewUser("alice")
}
```

`u` 是局部变量,但它的地址被返回给函数外部,所以会逃逸到堆上。

可以用编译参数观察:

```bash
go build -gcflags=-m main.go
```

工程上不要为了"避免逃逸"过早写丑代码。先写清楚,再用 benchmark 和 pprof 证明性能问题。

## 七、结构体内存对齐

CPU 访问内存时更喜欢按特定边界读取。Go 编译器会在结构体字段之间插入 padding,让字段对齐。

字段顺序会影响结构体大小:

```go
package main

import (
    "fmt"
    "unsafe"
)

type Bad struct {
    A int8
    B int64
    C int16
}

type Good struct {
    B int64
    C int16
    A int8
}

func main() {
    fmt.Println(unsafe.Sizeof(Bad{}))
    fmt.Println(unsafe.Sizeof(Good{}))
}
```

在常见 64 位平台上,`Bad` 往往比 `Good` 更大。原因是字段之间需要填充空洞。

优化规则:

- 大字段放前面
- 相同大小字段放一起
- bool/int8 等小字段集中放
- 热路径大量对象才值得为了对齐调整

不要为了省几个字节牺牲结构体语义。只有在对象数量很大、内存压力明确时,字段重排才有明显价值。

## 八、nil 指针与可选值

指针常被用来表达"可选":

```go
type UserPatch struct {
    Name *string
    Age  *int
}
```

这和零值不同:

- `Name == nil`:调用方没有传
- `*Name == ""`:调用方传了空字符串

可运行例子:

```go
package main

import "fmt"

type Patch struct {
    Name *string
}

func apply(name string, patch Patch) string {
    if patch.Name != nil {
        return *patch.Name
    }
    return name
}

func main() {
    empty := ""
    fmt.Println(apply("alice", Patch{}))
    fmt.Println(apply("alice", Patch{Name: &empty}))
}
```

HTTP PATCH、配置覆盖、可选字段里经常会用指针区分"没传"和"传了零值"。

但不要把所有字段都做成指针。指针过多会增加 nil 检查和 GC 压力。

## 九、可运行综合例子:安全更新用户资料

```go
package main

import (
    "errors"
    "fmt"
    "strings"
)

type User struct {
    Name string
    Age  int
}

type UserPatch struct {
    Name *string
    Age  *int
}

func (u *User) Apply(patch UserPatch) error {
    if patch.Name != nil {
        name := strings.TrimSpace(*patch.Name)
        if name == "" {
            return errors.New("name is empty")
        }
        u.Name = name
    }

    if patch.Age != nil {
        if *patch.Age < 0 {
            return errors.New("age is negative")
        }
        u.Age = *patch.Age
    }

    return nil
}

func main() {
    user := User{Name: "alice", Age: 18}

    name := " bob "
    age := 20
    patch := UserPatch{
        Name: &name,
        Age:  &age,
    }

    if err := user.Apply(patch); err != nil {
        fmt.Println(err)
        return
    }

    fmt.Printf("%+v\n", user)
}
```

这个例子里:

- `Apply` 用指针接收者修改原对象
- patch 字段用指针表达可选
- 参数仍然是值传递,只是值里包含地址
- 校验逻辑保护业务不变量

## 十、常见坑

### 10.1 返回局部变量地址不是错

```go
func f() *int {
    x := 1
    return &x
}
```

在 Go 里这是安全的。编译器会让 `x` 逃逸到堆上。不要套用 C 的悬挂指针经验。

### 10.2 指针接收者不等于一定高性能

小 struct 传值可能更快,因为少一次间接访问,也减少 GC 压力。性能要靠 benchmark,不要靠直觉。

### 10.3 指向循环变量的地址

在旧版本 Go 里,range 循环变量复用容易导致多个指针指向同一变量。Go 1.22 调整了循环变量语义,但维护旧代码时仍要小心。

稳妥写法:

```go
for i := range users {
    ptrs = append(ptrs, &users[i])
}
```

### 10.4 nil 指针解引用

```go
var u *User
fmt.Println(u.Name) // panic
```

指针方法内部如果允许 nil 接收者,要明确处理;否则让 panic 暴露编程错误也可以。

### 10.5 结构体字段重排过度优化

字段顺序首先服务可读性和业务语义。只有大量对象、内存敏感、profile 证明有问题时,再做对齐优化。

## 十一、工程判断

什么时候传指针?

- 需要修改调用方对象
- 对象很大,复制成本高
- 类型包含锁、连接、buffer、map、slice 等共享资源
- 需要表达可选字段
- 方法集需要满足某个接口的指针接收者

什么时候传值?

- 小型不可变值对象,例如坐标、金额值、时间范围
- 你希望函数拿到副本,避免外部被修改
- 类型本身就是 descriptor,例如 slice/map/channel/interface 已经包含引用语义

几个实用建议:

- 不要返回内部可变对象的指针,除非你愿意让调用方修改它
- 指针字段要考虑 JSON、数据库、nil 检查的复杂度
- 性能优化前先写 benchmark
- 并发共享指针对象时,必须考虑锁或不可变设计

---

## 十二、小结

Go 的指针比 C 安全,比纯值语言更灵活。

你需要记住:

1. Go 参数永远是值传递,指针也是一个值。
2. `&` 取地址,`*` 解引用。
3. Go 不允许指针运算。
4. `new` 返回指针,`make` 初始化 slice/map/channel。
5. 局部变量地址可以安全返回,编译器会做逃逸分析。
6. 结构体字段顺序会影响内存对齐和大小。
7. 指针适合修改、共享、可选和避免大对象复制,但不要滥用。
