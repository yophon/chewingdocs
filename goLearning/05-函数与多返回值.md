# 函数与多返回值

> **导读**: Go 的函数设计服务于一个核心目标:把结果、错误和资源生命周期都显式写出来。

Go 没有异常捕获机制,也没有默认参数、函数重载、装饰器这类复杂语法。它把函数设计得很朴素:参数进来,返回值出去,错误也是返回值的一部分。你会频繁看到 `(T, error)`,也会频繁写 `if err != nil`。这不是语法落后,而是 Go 在工程稳定性上的选择。

这一篇讲函数声明、多返回值、命名返回值、变长参数、函数类型、闭包、defer 与返回值之间的关系,以及真实项目里如何判断函数边界。

---

## 一、核心心智模型:函数签名就是契约

Go 里函数签名非常重要。它不仅告诉你输入输出类型,还告诉你错误是否可能发生、调用方是否需要处理资源、函数是否会修改外部状态。

典型 Go 函数长这样:

```go
func LoadUser(id int64) (User, error)
```

读到这个签名,你应该立刻知道:

- 输入是用户 ID
- 成功时返回 `User`
- 失败时返回 `error`
- 调用方必须处理 error

Go 不喜欢隐藏控制流。错误不会突然从深层调用栈 throw 出来,而是逐层返回。函数边界因此非常清晰。

---

## 二、基础函数声明

最普通的函数:

```go
package main

import "fmt"

func add(a int, b int) int {
    return a + b
}

func main() {
    fmt.Println(add(1, 2))
}
```

相邻参数类型相同时可以合并:

```go
func add(a, b int) int {
    return a + b
}
```

无返回值:

```go
func logMessage(msg string) {
    fmt.Println(msg)
}
```

多个返回值:

```go
func div(a, b int) (int, bool) {
    if b == 0 {
        return 0, false
    }
    return a / b, true
}
```

Go 的多返回值不是语法糖,它是错误处理、map 查询、类型断言、channel 接收等机制的统一表达方式。

## 三、多返回值与 error

Go 约定:如果函数可能失败,最后一个返回值通常是 `error`。

```go
package main

import (
    "errors"
    "fmt"
)

func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}

func main() {
    result, err := divide(10, 2)
    if err != nil {
        fmt.Println("divide failed:", err)
        return
    }

    fmt.Println(result)
}
```

几个重要约定:

- 成功时 `err == nil`
- 失败时其他返回值通常不应该被继续使用
- 错误要么处理,要么返回给上层
- 不要用 panic 表达普通业务失败

如果函数返回指针:

```go
func FindUser(id int64) (*User, error)
```

通常约定:

- 找到:返回 `*User, nil`
- 没找到:返回 `nil, ErrNotFound`
- 系统错误:返回 `nil, err`

这样调用方可以明确判断。

## 四、命名返回值:能用,但要克制

命名返回值会在函数进入时声明并初始化为零值:

```go
package main

import "fmt"

func rectangle(w, h int) (area int, perimeter int) {
    area = w * h
    perimeter = 2 * (w + h)
    return
}

func main() {
    area, perimeter := rectangle(3, 4)
    fmt.Println(area, perimeter)
}
```

空 `return` 会返回当前命名返回值,也叫 naked return。

工程建议:

- 短函数里可以使用
- 返回值含义复杂时可以使用,让签名更清楚
- 长函数里慎用 naked return,读者很难追踪最终值
- error 返回值命名为 `err` 时,要特别小心 defer 修改它

例如,命名返回值配合 defer 包装错误:

```go
package main

import (
    "fmt"
)

func run() (err error) {
    defer func() {
        if err != nil {
            err = fmt.Errorf("run failed: %w", err)
        }
    }()

    return step()
}

func step() error {
    return fmt.Errorf("step error")
}

func main() {
    fmt.Println(run())
}
```

这种写法在资源清理、事务提交回滚里会见到,但不要过度使用。

## 五、变长参数

变长参数用 `...T` 表示:

```go
package main

import "fmt"

func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

func main() {
    fmt.Println(sum(1, 2, 3))

    nums := []int{4, 5, 6}
    fmt.Println(sum(nums...))
}
```

在函数内部,`nums` 的类型是 `[]int`。

常见用途:

- 日志参数
- SQL 条件构造
- option 配置
- 批量追加

不要为了省一个 slice 参数就滥用变长参数。变长参数适合"调用方自然想传 0 个或多个值"的场景。

## 六、函数是一等公民

函数可以赋值给变量,也可以作为参数和返回值。

```go
package main

import "fmt"

func apply(a, b int, op func(int, int) int) int {
    return op(a, b)
}

func main() {
    add := func(a, b int) int {
        return a + b
    }

    fmt.Println(apply(2, 3, add))
    fmt.Println(apply(2, 3, func(a, b int) int {
        return a * b
    }))
}
```

如果函数类型反复出现,可以定义类型:

```go
type Handler func(string) error

func run(name string, h Handler) error {
    return h(name)
}
```

这在中间件、回调、策略函数里很常见。

## 七、闭包:函数捕获外部变量

闭包可以访问并修改外部变量:

```go
package main

import "fmt"

func counter() func() int {
    n := 0
    return func() int {
        n++
        return n
    }
}

func main() {
    next := counter()
    fmt.Println(next())
    fmt.Println(next())
    fmt.Println(next())
}
```

闭包的本质是函数携带了一份环境。`n` 不会在 `counter` 返回后消失,因为返回的匿名函数还引用着它。

闭包常见用途:

- 封装状态
- 构造中间件
- 延迟执行
- 生成测试 helper
- 实现简单策略

常见风险:

- 捕获变量导致生命周期变长
- 并发访问捕获变量需要加锁
- 闭包太深会降低可读性

## 八、defer 与函数返回

`defer` 在当前函数返回前执行。它和返回值的关系很重要。

先看参数求值:

```go
package main

import "fmt"

func main() {
    x := 1
    defer fmt.Println("defer:", x)
    x = 2
    fmt.Println("main:", x)
}
```

输出:

```text
main: 2
defer: 1
```

因为 `defer fmt.Println("defer:", x)` 注册时,x 已经被求值为 1。

再看命名返回值:

```go
package main

import "fmt"

func f() (n int) {
    defer func() {
        n++
    }()
    return 10
}

func main() {
    fmt.Println(f()) // 11
}
```

返回过程可以理解为:

1. 把返回值设置为 10
2. 执行 defer
3. 真正返回

所以 defer 可以修改命名返回值。这很强,也很危险。除非你是在做统一错误包装或清理,否则不要让 defer 偷偷改返回值。

## 九、方法不是函数吗

Go 的方法本质上是带接收者的函数:

```go
type User struct {
    Name string
}

func (u User) DisplayName() string {
    return u.Name
}
```

可以近似理解成:

```go
func DisplayName(u User) string {
    return u.Name
}
```

方法会在结构体那一篇详细讲。这里先记住:Go 没有 class,函数和方法都很普通,不要把它想复杂。

## 十、可运行综合例子:加载配置并应用选项

下面用函数、多返回值、变长参数、函数类型和错误处理写一个小配置加载器:

```go
package main

import (
    "errors"
    "fmt"
    "strings"
)

type Config struct {
    Name    string
    Timeout int
    Debug   bool
}

type Option func(*Config) error

func WithTimeout(seconds int) Option {
    return func(c *Config) error {
        if seconds <= 0 {
            return errors.New("timeout must be positive")
        }
        c.Timeout = seconds
        return nil
    }
}

func WithDebug(debug bool) Option {
    return func(c *Config) error {
        c.Debug = debug
        return nil
    }
}

func NewConfig(name string, options ...Option) (Config, error) {
    if strings.TrimSpace(name) == "" {
        return Config{}, errors.New("name is empty")
    }

    cfg := Config{
        Name:    name,
        Timeout: 30,
    }

    for _, option := range options {
        if err := option(&cfg); err != nil {
            return Config{}, err
        }
    }

    return cfg, nil
}

func main() {
    cfg, err := NewConfig("api", WithTimeout(10), WithDebug(true))
    if err != nil {
        fmt.Println("config failed:", err)
        return
    }

    fmt.Printf("%+v\n", cfg)
}
```

这个例子是 Go 里常见的 functional options 模式的简化版。它适合配置项较多、又不想暴露复杂构造函数的场景。

## 十一、常见坑

### 11.1 忽略 error

```go
value, _ := strconv.Atoi(input)
```

这不是不能写,但要有明确理由。业务代码里长期忽略 error,最后都会变成排查困难的线上问题。

### 11.2 命名返回值 + naked return 滥用

```go
func f() (result int, err error) {
    // 100 行逻辑
    return
}
```

读者很难知道最终返回什么。长函数里写完整返回值更清楚。

### 11.3 在循环里 defer 大量资源

defer 到函数退出才执行。循环处理文件、连接、锁时要小心资源积压。

### 11.4 闭包捕获共享变量

```go
count := 0
handler := func() {
    count++
}
```

单线程没问题,并发调用就有数据竞争。需要 `sync.Mutex` 或 atomic。

### 11.5 过度抽象成高阶函数

Go 支持函数作为参数,但 Go 社区更偏好直白代码。不要为了"函数式"把简单分支包装成层层回调。

## 十二、工程判断

函数边界的好坏,直接影响代码可测试性和可维护性。

几个判断标准:

- 函数名表达动作,例如 `LoadUser`、`ValidateOrder`
- 参数不要太多,超过 4-5 个时考虑 struct 或 options
- 返回 `(T, error)` 时,调用方能明确知道失败语义
- 不要返回半初始化对象和 nil error
- 不要用 panic 替代 error
- 纯计算函数尽量不依赖全局变量,方便测试
- 修改入参时要让函数名或文档明确表达
- 闭包适合封装小状态,不适合藏复杂业务流程

典型函数设计对比:

```go
// 差:参数含义不清,失败语义不清
func Do(a string, b int, c bool) string

// 好:类型和错误都明确
func BuildInvoice(req InvoiceRequest) (Invoice, error)
```

Go 的函数不追求华丽,追求调用点清楚。

---

## 十三、小结

Go 函数的关键是显式。

你需要记住:

1. 多返回值是 Go 错误处理的基础。
2. 可能失败的函数通常返回 `(T, error)`。
3. 命名返回值可以提升签名表达力,但 naked return 要克制。
4. 变长参数适合自然的 0 到多个输入。
5. 函数可以作为值传递,闭包可以捕获外部变量。
6. defer 在函数返回前执行,参数注册时求值。
7. 好函数签名应该让调用方不用猜。
