# 垃圾回收(GC)机制与内存逃逸分析

> **导读**：虽然 Go 是自动垃圾回收的，但在极高吞吐量的服务端程序中，理解 GC 和逃逸分析能够帮你大幅减少系统毛刺（Stop-The-World）。

## 一、Go GC 演进历程
- **Go 1.0**: 朴素的标记-清除法，全阶段 STW（停顿世界），停顿可达秒级。
- **Go 1.5**: 引入**三色标记法**和并发标记，STW 降至毫秒级。
- **Go 1.8**: 引入混合写屏障（Hybrid Write Barrier），STW 降至亚毫秒级（基本上忽略不计）。

**三色标记法原理**：
1. 一开始所有对象都是白色。
2. 从根节点（全局变量、Goroutine 栈）出发，把直接引用的对象标为灰色。
3. 从灰色节点继续遍历引用的对象，自身染黑，引用的染灰。
4. 最终图中剩下的全是白和黑，白色的就是没人引用的垃圾，直接清掉。

## 二、内存逃逸分析 (Escape Analysis)
由于 GC 是有代价的（消耗 CPU 资源清理堆内存），最高效的优化策略就是**根本不让变量被分配到堆上**。
局部变量如果分配在“栈”上，函数退出时直接跟随栈帧销毁，完全不需要 GC 介入。

**什么时候变量会“逃跑”到堆上？**
1. **指针返回**：函数返回了局部变量的指针，外部还在用，编译器只好把它放堆里。
2. **栈空间不足**：巨大的结构体或切片，栈放不下。
3. **接口动态分配**：把变量赋值给了 `interface{}`（如 `fmt.Println(a)`），编译器无法在编译期确定类型，只能放堆上。

## 三、实战：观察逃逸现象
```bash
go build -gcflags="-m" main.go
```
执行这个命令，编译器会详细地告诉你哪一行代码的变量 escape 到 heap 了。
**优化建议**：对于小对象，能传值就传值，不要盲目觉得传指针一定性能更好（指针带来的垃圾回收压力往往超过了值拷贝的开销）。
