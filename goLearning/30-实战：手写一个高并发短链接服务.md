# 实战：手写一个高并发短链接服务

> **导读**：学习语言必须落到工程实战。最后一章，我们将前面 29 篇学过的知识融会贯通，写一个类似于 bit.ly 的高并发、高可用短链接服务。

## 一、系统需求分析
**核心功能**：
- 把长 URL 转成短 URL 返回。
- 用户访问短 URL 时，通过 HTTP 302 重定向跳转到真实长 URL。

**非功能需求**：
- 全局唯一，不能生成冲突的短链。
- 极高的读取 QPS 扛压能力（读写比通常 100:1）。

## 二、架构设计与技术栈
- **Web 框架**：Gin（处理 REST API 路由）
- **持久化层**：MySQL + GORM（存储短码到长链的映射）
- **高速缓存**：Redis（抵抗海量重定向读取压力）
- **唯一发号器**：雪花算法 (Snowflake) 生成全局唯一 ID，然后将 10 进制转为 Base62 (a-z,A-Z,0-9) 生成 6 位短码。

## 三、核心代码骨架
1. **生成流程**：
```go
func CreateShortLink(c *gin.Context) {
    longUrl := c.PostForm("url")
    // 1. 调用 Snowflake 拿到全局唯一 64 位整数
    id := snowflake.Generate()
    // 2. 转为 Base62 字符串，如 "k9Z2jX"
    shortCode := base62.Encode(id)
    // 3. 异步写入 DB 与 Redis 缓存
    db.Create(&Link{Code: shortCode, URL: longUrl})
    rdb.Set(ctx, shortCode, longUrl, 7*24*time.Hour)
    
    c.JSON(200, gin.H{"short_url": "http://lo.cal/" + shortCode})
}
```

2. **重定向流程 (核心高并发点)**：
```go
func Redirect(c *gin.Context) {
    code := c.Param("code")
    // 1. 优先查 Redis（99%的流量在这里被消化，极大保护 MySQL）
    url, err := rdb.Get(ctx, code).Result()
    if err == redis.Nil {
        // 2. Redis 没命中，才去查 MySQL (这就是缓存穿透点，需做防穿透处理)
        db.Where("code = ?", code).First(&link)
        url = link.URL
        // 3. 回写缓存
        rdb.Set(ctx, code, url, 24*time.Hour)
    }
    // 4. 发送 302 重定向
    c.Redirect(302, url)
}
```
