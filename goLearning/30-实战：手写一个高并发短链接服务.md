# 实战：手写一个高并发短链接服务

> **一句话导读**：短链接服务看起来只是“长 URL 换短码”，真正的工程难点在唯一 ID、缓存一致性、热点读取、防刷限流、可观测性和扩展路径。

## 一、需求拆解

短链接系统的核心能力很简单：

- 创建短链：提交长 URL，返回短 URL。
- 访问短链：用户请求短码，服务返回 HTTP 302 跳转到长 URL。
- 查询短链：后台查看短链状态、创建时间、访问统计。

但高并发场景下还要明确非功能需求：

- 短码全局唯一，不能把用户跳到错误地址。
- 读多写少，重定向 QPS 远高于创建 QPS。
- 重定向路径要极短，不能每次都打 MySQL。
- 热点短链不能击穿缓存和数据库。
- 无效短码不能造成缓存穿透。
- 支持过期时间、禁用、恶意 URL 拦截。
- 方便水平扩展，不能依赖单机内存状态。

一个合理的第一版目标是：单服务多副本部署，Redis 承载主要读流量，MySQL 持久化映射，短码由发号器生成，Gin 提供 HTTP API。

## 二、整体架构

```text
              创建短链
client  ----------------->  Gin API
                              |
                              | 生成短码
                              v
                         MySQL 持久化
                              |
                              v
                         Redis 预热缓存

              访问短链
browser ----------------->  Gin Redirect
                              |
                              | 先查 Redis
                              v
                      命中：302 Redirect
                              |
                              | 未命中
                              v
                         MySQL 回源
                              |
                              v
                        回写 Redis 后 302
```

推荐技术栈：

- Web：Gin
- 存储：MySQL
- 缓存：Redis
- ORM：GORM 或 sqlc
- ID：Snowflake、数据库号段、Redis INCR 三选一
- 配置：YAML + 环境变量覆盖
- 观测：结构化日志、Prometheus 指标、pprof

## 三、数据模型设计

MySQL 表：

```sql
CREATE TABLE short_links (
  id BIGINT UNSIGNED NOT NULL PRIMARY KEY,
  code VARCHAR(16) NOT NULL,
  long_url TEXT NOT NULL,
  url_hash BINARY(32) NOT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  expire_at DATETIME NULL,
  created_at DATETIME NOT NULL,
  updated_at DATETIME NOT NULL,
  UNIQUE KEY uk_code (code),
  KEY idx_url_hash (url_hash),
  KEY idx_expire_at (expire_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

字段说明：

- `id`：全局唯一数字 ID，用于生成短码。
- `code`：Base62 短码，唯一索引兜底防冲突。
- `long_url`：原始长 URL。
- `url_hash`：长 URL 的 SHA-256，用于去重或风控检索。
- `status`：1 启用，2 禁用。
- `expire_at`：过期时间，允许永久有效。

Go 模型：

```go
package model

import "time"

type LinkStatus int8

const (
	LinkEnabled  LinkStatus = 1
	LinkDisabled LinkStatus = 2
)

type ShortLink struct {
	ID        uint64
	Code      string
	LongURL   string
	URLHash   [32]byte
	Status    LinkStatus
	ExpireAt  *time.Time
	CreatedAt time.Time
	UpdatedAt time.Time
}

func (l ShortLink) Available(now time.Time) bool {
	if l.Status != LinkEnabled {
		return false
	}
	return l.ExpireAt == nil || l.ExpireAt.After(now)
}
```

生产上要注意：短码唯一必须由数据库唯一索引兜底。即使 ID 生成器理论上唯一，也不要把数据正确性完全寄托在应用代码上。

## 四、短码生成：Snowflake + Base62

短码生成常见方案：

- **UUID 截断**：简单但有碰撞风险，不推荐直接截断。
- **Redis INCR**：简单、单调、可控，但依赖 Redis 可用性。
- **数据库号段**：稳定，适合大规模服务，但实现略复杂。
- **Snowflake**：本地生成，高性能，但要处理时钟回拨和 worker id 分配。

本实战选择 Snowflake + Base62。Snowflake 生成 64 位整数，Base62 把整数压成更短字符串。

Base62：

```go
package base62

const alphabet = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

func Encode(n uint64) string {
	if n == 0 {
		return "0"
	}

	buf := make([]byte, 0, 11)
	for n > 0 {
		buf = append(buf, alphabet[n%62])
		n /= 62
	}

	for i, j := 0, len(buf)-1; i < j; i, j = i+1, j-1 {
		buf[i], buf[j] = buf[j], buf[i]
	}
	return string(buf)
}

func Decode(s string) (uint64, bool) {
	var n uint64
	for i := 0; i < len(s); i++ {
		idx := indexByte(s[i])
		if idx < 0 {
			return 0, false
		}
		n = n*62 + uint64(idx)
	}
	return n, true
}

func indexByte(b byte) int {
	for i := 0; i < len(alphabet); i++ {
		if alphabet[i] == b {
			return i
		}
	}
	return -1
}
```

Snowflake 简化实现：

```go
package idgen

import (
	"errors"
	"sync"
	"time"
)

const (
	workerBits   = 10
	sequenceBits = 12
	maxWorkerID  = int64(1<<workerBits - 1)
	maxSequence  = int64(1<<sequenceBits - 1)
	epoch        = int64(1704067200000) // 2024-01-01 UTC ms
)

type Snowflake struct {
	mu        sync.Mutex
	workerID  int64
	lastMilli int64
	sequence  int64
}

func NewSnowflake(workerID int64) (*Snowflake, error) {
	if workerID < 0 || workerID > maxWorkerID {
		return nil, errors.New("invalid worker id")
	}
	return &Snowflake{workerID: workerID}, nil
}

func (s *Snowflake) Next() (uint64, error) {
	s.mu.Lock()
	defer s.mu.Unlock()

	now := time.Now().UnixMilli()
	if now < s.lastMilli {
		return 0, errors.New("clock moved backwards")
	}

	if now == s.lastMilli {
		s.sequence = (s.sequence + 1) & maxSequence
		if s.sequence == 0 {
			for now <= s.lastMilli {
				now = time.Now().UnixMilli()
			}
		}
	} else {
		s.sequence = 0
	}

	s.lastMilli = now
	id := ((now - epoch) << (workerBits + sequenceBits)) |
		(s.workerID << sequenceBits) |
		s.sequence
	return uint64(id), nil
}
```

worker id 可以来自配置、Kubernetes StatefulSet ordinal、环境变量，或者由中心化服务分配。多副本部署时千万不能所有实例使用同一个 worker id。

## 五、核心接口设计

API：

```text
POST /api/v1/links
GET  /:code
GET  /api/v1/links/:code
PATCH /api/v1/links/:code/disable
```

创建短链请求：

```json
{
  "url": "https://example.com/articles/go",
  "expire_at": "2026-12-31T23:59:59Z"
}
```

创建短链 Handler：

```go
type LinkHandler struct {
	links *service.LinkService
}

type CreateLinkRequest struct {
	URL      string     `json:"url" binding:"required,url"`
	ExpireAt *time.Time `json:"expire_at"`
}

func (h *LinkHandler) Create(c *gin.Context) {
	var req CreateLinkRequest
	if err := c.ShouldBindJSON(&req); err != nil {
		c.JSON(http.StatusBadRequest, gin.H{"code": "INVALID_ARGUMENT", "message": err.Error()})
		return
	}

	link, err := h.links.Create(c.Request.Context(), service.CreateLinkInput{
		LongURL:  req.URL,
		ExpireAt: req.ExpireAt,
	})
	if err != nil {
		c.JSON(http.StatusInternalServerError, gin.H{"code": "INTERNAL_ERROR"})
		return
	}

	c.JSON(http.StatusOK, gin.H{
		"code":      link.Code,
		"short_url": "https://s.example.com/" + link.Code,
	})
}
```

Service 创建流程：

```go
type LinkService struct {
	ids   IDGenerator
	repo  LinkRepo
	cache LinkCache
	now   func() time.Time
}

type CreateLinkInput struct {
	LongURL  string
	ExpireAt *time.Time
}

func (s *LinkService) Create(ctx context.Context, in CreateLinkInput) (*model.ShortLink, error) {
	longURL, err := normalizeURL(in.LongURL)
	if err != nil {
		return nil, ErrInvalidURL
	}
	if blockedURL(longURL) {
		return nil, ErrBlockedURL
	}

	id, err := s.ids.Next()
	if err != nil {
		return nil, err
	}

	link := &model.ShortLink{
		ID:        id,
		Code:      base62.Encode(id),
		LongURL:   longURL,
		URLHash:   sha256.Sum256([]byte(longURL)),
		Status:    model.LinkEnabled,
		ExpireAt:  in.ExpireAt,
		CreatedAt: s.now(),
		UpdatedAt: s.now(),
	}

	if err := s.repo.Create(ctx, link); err != nil {
		return nil, err
	}

	ttl := ttlUntil(link.ExpireAt, 24*time.Hour)
	_ = s.cache.Set(ctx, link.Code, link.LongURL, ttl)
	return link, nil
}
```

缓存预热失败不应该导致创建失败，因为 MySQL 已经持久化成功；但要记录日志和指标，避免 Redis 长时间不可用却无人发现。

## 六、高并发重定向路径

重定向是系统的核心热路径。目标是：命中 Redis 时只做一次 get 和一次 302。

```go
type RedirectHandler struct {
	links *service.LinkService
}

func (h *RedirectHandler) Redirect(c *gin.Context) {
	code := c.Param("code")
	if !validCode(code) {
		c.String(http.StatusNotFound, "not found")
		return
	}

	url, err := h.links.Resolve(c.Request.Context(), code)
	if errors.Is(err, service.ErrLinkNotFound) {
		c.String(http.StatusNotFound, "not found")
		return
	}
	if errors.Is(err, service.ErrLinkUnavailable) {
		c.String(http.StatusGone, "gone")
		return
	}
	if err != nil {
		c.String(http.StatusInternalServerError, "internal error")
		return
	}

	c.Redirect(http.StatusFound, url)
}
```

Service 解析流程：

```go
func (s *LinkService) Resolve(ctx context.Context, code string) (string, error) {
	if url, err := s.cache.Get(ctx, code); err == nil {
		if url == negativeCacheValue {
			return "", ErrLinkNotFound
		}
		return url, nil
	}

	link, err := s.repo.FindByCode(ctx, code)
	if errors.Is(err, ErrRepoNotFound) {
		_ = s.cache.Set(ctx, code, negativeCacheValue, 5*time.Minute)
		return "", ErrLinkNotFound
	}
	if err != nil {
		return "", err
	}
	if !link.Available(s.now()) {
		_ = s.cache.Set(ctx, code, negativeCacheValue, time.Minute)
		return "", ErrLinkUnavailable
	}

	ttl := ttlUntil(link.ExpireAt, 24*time.Hour)
	_ = s.cache.Set(ctx, code, link.LongURL, addJitter(ttl))
	return link.LongURL, nil
}
```

这里处理了三个生产问题：

- **缓存穿透**：不存在的 code 也缓存一个短 TTL 的空值。
- **缓存雪崩**：TTL 加随机抖动，避免大量 key 同时过期。
- **缓存不可用降级**：Redis 错误时可以回源 MySQL，但要限流，防止数据库被打爆。

Redis 实现：

```go
type RedisLinkCache struct {
	rdb *redis.Client
}

func (c *RedisLinkCache) key(code string) string {
	return "shortlink:" + code
}

func (c *RedisLinkCache) Get(ctx context.Context, code string) (string, error) {
	return c.rdb.Get(ctx, c.key(code)).Result()
}

func (c *RedisLinkCache) Set(ctx context.Context, code, url string, ttl time.Duration) error {
	return c.rdb.Set(ctx, c.key(code), url, ttl).Err()
}
```

## 七、防击穿：singleflight 与热点保护

某个热点短链缓存过期时，可能几千个请求同时打到 MySQL，这叫缓存击穿。Go 可以用 `singleflight` 合并同一个 code 的并发回源：

```go
type LinkService struct {
	group singleflight.Group
	repo  LinkRepo
	cache LinkCache
}

func (s *LinkService) Resolve(ctx context.Context, code string) (string, error) {
	if url, err := s.cache.Get(ctx, code); err == nil {
		return url, nil
	}

	v, err, _ := s.group.Do(code, func() (any, error) {
		if url, err := s.cache.Get(ctx, code); err == nil {
			return url, nil
		}

		link, err := s.repo.FindByCode(ctx, code)
		if err != nil {
			return "", err
		}
		_ = s.cache.Set(ctx, code, link.LongURL, addJitter(24*time.Hour))
		return link.LongURL, nil
	})
	if err != nil {
		return "", err
	}
	return v.(string), nil
}
```

`singleflight` 只合并单进程内请求，多副本部署时仍可能有多个实例同时回源。热点极高时可以：

- 给热点 key 设置更长 TTL。
- 后台主动刷新热点缓存。
- Redis 使用集群并扩容读能力。
- 在网关或 CDN 层缓存 302 响应。
- 对异常 code 或来源 IP 做限流。

## 八、访问统计怎么做

不要在重定向路径里同步写 MySQL 统计，否则读 QPS 会被数据库写入拖垮。

可选方案：

- Redis 计数：`HINCRBY shortlink:stat:{code} total 1`，异步定时落库。
- 消息队列：重定向时投递访问事件，由消费者聚合。
- 日志分析：访问日志进入 Kafka/ClickHouse/OLAP 系统。

简单版 Redis 计数：

```go
func (s *LinkService) RecordVisit(ctx context.Context, code string) {
	_ = s.stats.Incr(ctx, code, time.Now())
}
```

在 Handler 中异步调用要谨慎，不要每个请求都无限制开 goroutine。可以用有缓冲 channel 加固定 worker：

```go
type VisitEvent struct {
	Code string
	At   time.Time
}

events := make(chan VisitEvent, 10000)

func startWorkers(ctx context.Context, n int, stats VisitStats, events <-chan VisitEvent) {
	for i := 0; i < n; i++ {
		go func() {
			for {
				select {
				case <-ctx.Done():
					return
				case e := <-events:
					_ = stats.Incr(ctx, e.Code, e.At)
				}
			}
		}()
	}
}
```

channel 满了怎么办是生产取舍：可以丢弃统计事件并打指标，因为跳转主流程通常比统计完整性更重要。

## 九、排错与优化

常见问题：

- **短码冲突**：检查 worker id 是否重复、数据库唯一索引是否生效、时钟是否回拨。
- **Redis 命中率低**：检查 TTL、key 前缀、是否每次创建后预热、是否有大量随机无效 code。
- **MySQL 被打满**：排查缓存穿透、击穿、Redis 故障降级、慢 SQL 和连接池配置。
- **302 很慢**：看 Redis P99、网络延迟、连接池耗尽、JSON 日志同步写。
- **内存上涨**：检查本地缓存、singleflight key 是否异常、goroutine worker 是否泄漏。
- **恶意跳转**：创建时要做 URL scheme 校验、黑名单、私网地址拦截，避免成为钓鱼或 SSRF 工具。

指标至少要有：

- 创建短链成功率、失败率。
- 重定向 QPS、P95/P99。
- Redis 命中率、错误率、延迟。
- MySQL 查询延迟、连接池使用率。
- 404/410 比例。
- 限流次数、黑名单拦截次数。

## 十、生产取舍

几个关键设计没有绝对答案：

- **是否允许同一个长 URL 生成多个短码**：允许更简单；不允许需要基于 `url_hash` 做去重和并发控制。
- **短码是否可预测**：Snowflake + Base62 大体递增，可能被枚举。需要安全性时可加随机盐、Hashids、随机码加冲突重试。
- **Redis 失败是否回源 MySQL**：回源可提高可用性，但可能把 MySQL 打挂。要配合限流和熔断。
- **统计是否强一致**：多数短链服务选择最终一致，优先保证跳转。
- **是否用 CDN**：公网大流量短链可以让 CDN 缓存热点 302，但禁用、过期后的失效传播要设计清楚。

第一版不要过度设计成十几个服务。可以先做一个清晰的模块化单体，等 QPS、团队和业务边界真的增长，再拆分创建服务、重定向服务、统计服务和管理后台。

## 十一、关键实现路径

落地顺序建议：

1. 建表和模型：完成 `short_links` 表、GORM model、repo。
2. 实现 Base62 和 ID 生成器：写单元测试覆盖边界值。
3. 实现创建接口：校验 URL、生成 ID、写 MySQL、预热 Redis。
4. 实现重定向接口：Redis 优先、MySQL 回源、负缓存、防击穿。
5. 加入限流和安全校验：限制创建频率，拦截非法 scheme、内网地址和黑名单域名。
6. 加入统计：先用 Redis 异步计数，再定时聚合落库。
7. 加入观测：日志、指标、pprof、健康检查。
8. 容器化部署：Dockerfile、配置、优雅退出、Kubernetes readiness/liveness。
9. 压测和调优：验证 Redis 命中路径、MySQL 回源路径、Redis 故障降级路径。

Base62 单测示例：

```go
func TestBase62RoundTrip(t *testing.T) {
	cases := []uint64{0, 1, 61, 62, 3844, math.MaxInt64}
	for _, n := range cases {
		s := base62.Encode(n)
		got, ok := base62.Decode(s)
		if !ok || got != n {
			t.Fatalf("round trip failed: n=%d s=%s got=%d ok=%v", n, s, got, ok)
		}
	}
}
```

## 十二、总结

短链接服务是一个很适合检验 Go 工程能力的项目：HTTP 路由、数据库、Redis、并发控制、缓存策略、限流、Docker、Kubernetes、pprof 都能串起来。

真正的高并发不是把 Handler 写短，而是让热路径足够简单：Redis 命中直接 302，未命中可控回源，异常流量被挡住，统计异步化，存储有唯一约束兜底，观测能解释每一次变慢。按这个思路实现出来的短链接服务，才有资格从练手项目走向生产系统。
