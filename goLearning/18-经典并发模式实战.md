# 经典并发模式实战

> **导读**：Go 并发的难点不在于启动 goroutine，而在于用稳定的模式控制并发数量、结果汇聚、取消传播和错误处理。

## 一、工程场景：批量处理订单

假设后台服务每天要处理一批订单：

- 从数据库或消息队列取出订单 ID。
- 并发调用库存、支付、风控等服务。
- 把成功和失败结果汇总。
- 任意一个阶段超时后，整批任务要尽快停止。

最直接的写法是每个订单一个 goroutine。但订单量一旦达到几万，就可能把下游服务、数据库连接池和本机内存一起打爆。

工程上更常见的是组合使用这些模式：

- Worker Pool：限制最大并发。
- Fan-out / Fan-in：分发任务并汇总结果。
- Pipeline：把任务拆成多个阶段。
- Semaphore：临时限制某段代码的并发。
- Context：统一取消。

## 二、Worker Pool：控制最大并发

Worker Pool 的核心是固定数量的 worker 从任务 channel 中取活干。

下面是一个接近真实批处理的可运行例子：

```go
package main

import (
    "context"
    "fmt"
    "math/rand"
    "sync"
    "time"
)

type Job struct {
    OrderID int
}

type Result struct {
    OrderID int
    OK      bool
    Err     error
}

func main() {
    ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
    defer cancel()

    jobs := make(chan Job)
    results := make(chan Result)

    var wg sync.WaitGroup
    workerCount := 4

    for i := 1; i <= workerCount; i++ {
        wg.Add(1)
        go worker(ctx, i, jobs, results, &wg)
    }

    go func() {
        defer close(jobs)
        for i := 1; i <= 20; i++ {
            select {
            case jobs <- Job{OrderID: i}:
            case <-ctx.Done():
                return
            }
        }
    }()

    go func() {
        wg.Wait()
        close(results)
    }()

    success := 0
    failed := 0
    for r := range results {
        if r.Err != nil {
            fmt.Printf("order=%d failed: %v\n", r.OrderID, r.Err)
            failed++
            continue
        }
        if r.OK {
            success++
        }
    }

    fmt.Printf("done success=%d failed=%d ctx=%v\n", success, failed, ctx.Err())
}

func worker(ctx context.Context, id int, jobs <-chan Job, results chan<- Result, wg *sync.WaitGroup) {
    defer wg.Done()

    for {
        select {
        case <-ctx.Done():
            return
        case job, ok := <-jobs:
            if !ok {
                return
            }

            result := processOrder(ctx, job)
            select {
            case results <- result:
            case <-ctx.Done():
                return
            }
        }
    }
}

func processOrder(ctx context.Context, job Job) Result {
    cost := time.Duration(100+rand.Intn(300)) * time.Millisecond
    select {
    case <-time.After(cost):
        if job.OrderID%7 == 0 {
            return Result{OrderID: job.OrderID, Err: fmt.Errorf("risk rejected")}
        }
        return Result{OrderID: job.OrderID, OK: true}
    case <-ctx.Done():
        return Result{OrderID: job.OrderID, Err: ctx.Err()}
    }
}
```

这个例子里有几个工程点：

- worker 数量限制了最大并发。
- 生产任务和发送结果时都监听 Context。
- `results` 由等待 worker 的 goroutine 统一关闭。
- 消费方只 range `results`，不关心 worker 如何退出。

## 三、Fan-out / Fan-in：分发与汇总

Fan-out 是把一份输入交给多个 goroutine 处理。Fan-in 是把多个输出合并成一个输出。

```go
func fanIn[T any](ctx context.Context, inputs ...<-chan T) <-chan T {
    out := make(chan T)
    var wg sync.WaitGroup

    for _, input := range inputs {
        wg.Add(1)
        go func(ch <-chan T) {
            defer wg.Done()
            for {
                select {
                case <-ctx.Done():
                    return
                case v, ok := <-ch:
                    if !ok {
                        return
                    }
                    select {
                    case out <- v:
                    case <-ctx.Done():
                        return
                    }
                }
            }
        }(input)
    }

    go func() {
        wg.Wait()
        close(out)
    }()

    return out
}
```

Fan-in 的坑在关闭 channel：谁创建谁关闭，多个发送者不能随便关闭同一个输出 channel。通常用 `WaitGroup` 等所有发送者结束后，由一个 goroutine 关闭输出。

## 四、Pipeline：流水线处理

Pipeline 适合把任务拆成阶段。例如读取订单、补充用户信息、计算价格、写入结果。

```go
func gen(ctx context.Context, nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, n := range nums {
            select {
            case out <- n:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

func square(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            select {
            case out <- n * n:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}

func filterEven(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            if n%2 != 0 {
                continue
            }
            select {
            case out <- n:
            case <-ctx.Done():
                return
            }
        }
    }()
    return out
}
```

组装：

```go
ctx, cancel := context.WithCancel(context.Background())
defer cancel()

for n := range filterEven(ctx, square(ctx, gen(ctx, 1, 2, 3, 4, 5))) {
    fmt.Println(n)
}
```

Pipeline 的好处是阶段清晰，坏处是任意一段没有处理取消或没有关闭输出，就会拖住整条链路。

## 五、Semaphore：限制局部并发

有时不需要完整 worker pool，只想限制一段调用最多同时跑 N 个，可以用带缓冲 channel 做信号量。

```go
func fetchAll(ctx context.Context, urls []string) error {
    sem := make(chan struct{}, 5)
    var wg sync.WaitGroup
    var mu sync.Mutex
    var firstErr error

    for _, url := range urls {
        url := url
        select {
        case sem <- struct{}{}:
        case <-ctx.Done():
            return ctx.Err()
        }

        wg.Add(1)
        go func() {
            defer wg.Done()
            defer func() { <-sem }()

            if err := fetchOne(ctx, url); err != nil {
                mu.Lock()
                if firstErr == nil {
                    firstErr = err
                }
                mu.Unlock()
            }
        }()
    }

    wg.Wait()
    return firstErr
}
```

Go 官方扩展包 `golang.org/x/sync/errgroup` 在生产里更常用，因为它能把错误和 Context 取消结合起来。

## 六、关键坑位

### 1. goroutine 数量不等于吞吐

并发太高会让系统更慢。真正限制吞吐的通常是数据库连接数、下游接口限流、CPU 核数和网络延迟。worker 数量应该压测后确定。

### 2. 发送方阻塞导致泄漏

如果结果没人收，发送 `results <- r` 的 goroutine 会一直阻塞。发送结果时监听 `ctx.Done()` 是一个好习惯。

### 3. 错误处理不能丢

很多示例只演示成功路径，但生产代码必须明确：一个任务失败后是继续处理其他任务，还是取消整批任务。

### 4. channel 关闭职责混乱

接收方不要关闭输入 channel。多个发送方不要各自关闭共享输出 channel。关闭 channel 代表“不会再发送”，这个信息只有发送方知道。

### 5. 循环变量捕获

Go 1.22 改善了 range 变量捕获问题，但在兼容旧版本或普通 for 循环中，仍建议显式拷贝变量：

```go
url := url
go func() {
    _ = fetchOne(ctx, url)
}()
```

## 七、生产判断

选择并发模式时可以这样判断：

- 任务数量大、每个任务相似：Worker Pool。
- 多个来源结果要合并：Fan-in。
- 一个任务有多个连续处理阶段：Pipeline。
- 只想限制某个慢资源的并发：Semaphore。
- 一错即停并传播取消：`errgroup.WithContext`。

不要为了“用 channel”而用 channel。如果只是共享计数器，用 `sync.Mutex` 或 `atomic` 往往更直接。如果只是等待一批 goroutine 结束，用 `sync.WaitGroup` 就够了。

## 八、总结

Go 并发模式的核心是控制，而不是炫技。Worker Pool 控制并发规模，Fan-in/Fan-out 控制数据流向，Pipeline 控制处理阶段，Context 控制生命周期。写生产代码时要同时考虑关闭、取消、错误和背压，否则示例能跑，线上却容易泄漏、阻塞或压垮下游。
