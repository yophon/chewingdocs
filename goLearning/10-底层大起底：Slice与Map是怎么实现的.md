# 底层大起底：Slice 与 Map 是怎么实现的

> **导读**：面试最爱问的核心考点。不了解底层，很容易写出内存泄漏和导致 Panic 的代码。

## 一、Slice（切片）底层结构
Slice 是一个极其轻量级的结构体（24 字节，在 64 位机器上）：
```go
type slice struct {
    array unsafe.Pointer // 指向底层真实数据的指针 (8 字节)
    len   int            // 当前元素个数 (8 字节)
    cap   int            // 容量，底层数组还能装多少 (8 字节)
}
```
**扩容机制**：
- 当 `append` 超出 `cap` 时：如果当前容量小于 256 (Go 1.18 之后变动)，翻倍扩容；大于 256 后，平滑按一定比例递增扩容。
- **陷阱**：大数组切出一个小 Slice。只要小 Slice 还存活着，底层的庞大数组就无法被 GC 回收（内存泄漏）。
- **解法**：当只需要引用一小部分数据时，不要用切片语法 `arr[:10]`，而是新建一个 Slice 用 `copy()` 把数据拷过去。

## 二、Map（哈希表）底层结构
Go 的 map 实现是一个 `hmap` 结构体，内部由多个 `bmap`（桶/Bucket）构成。

**核心机制**：
- 哈希冲突解决：采用**拉链法**。当一个 Bucket 塞满 8 个元素后，会挂上一个溢出桶（overflow bucket）。
- **渐进式扩容**：当装载因子（LoadFactor）> 6.5 或溢出桶太多时，会触发翻倍扩容或等量扩容。Go 不会一次性搬迁数据（会卡死），而是在后续的插入、删除、查询操作中，一点点把老桶的数据搬迁到新桶（渐进式）。

**使用禁忌**：
1. Map 不能边遍历边删除（某些情况下会导致死锁或崩溃）。
2. Map 不是并发安全的。如果你开两个 Goroutine，一个写 map，一个读/写 map，Go 运行时会直接报错 `fatal error: concurrent map read and map write`，导致程序强行退出，`recover` 都拦不住！必须使用 `sync.Mutex` 或者直接用标准库提供的并发安全 `sync.Map`。
