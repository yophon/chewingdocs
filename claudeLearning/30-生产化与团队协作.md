## 一、团队怎么共享 Claude Code 配置

新人入职第一天 clone 项目,**应该 30 分钟内 Claude Code 跑通**。这要求几件事:

### 1.1 项目级 settings 提交进 git

`.claude/settings.json`(全员共享):

```json
{
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": ["Bash(npm:*)", "Bash(git status)", ...],
    "deny": ["Bash(rm -rf*)", ...],
    "ask": ["Bash(git push:*)"]
  },
  "hooks": {
    "PreToolUse": [...],
    "PostToolUse": [...]
  }
}
```

`.gitignore`:

```
.claude/settings.local.json
```

### 1.2 项目级 `.mcp.json` 提交

```json
{
  "mcpServers": {
    "github": {...},
    "postgres": {...}
  }
}
```

加 `.claude/settings.json` 的 `enabledMcpjsonServers`(或 `enableAllProjectMcpServers`)开启。

### 1.3 项目级 CLAUDE.md 提交

第 10 篇讲过。

### 1.4 项目级 skills / commands / agents 提交

```
.claude/
  settings.json
  commands/      # 团队 slash commands
  skills/        # 团队 skills
  agents/        # 团队 subagents
  hooks/         # 团队 hook 脚本
```

**全部提交进 git**——新人 clone 即用。

### 1.5 README 写"Claude Code 用法"

```
README.md:

## 用 Claude Code 做日常开发

clone 后:
1. `npm install -g @anthropic-ai/claude-code`(没装的话)
2. `claude` 打开,接受 MCP server 信任
3. 我们的 slash:
   - /review-pr <num>  审 PR
   - /staging          部署 staging
4. 默认 hook:
   - 改 ts 自动 prettier
   - commit 前跑测试
5. 沟通:#claude-code 频道
```

> 文档写得好,onboarding 顺;**写不好新人当天就放弃**。

## 二、Settings 三层在团队场景的实战

```
~/.claude/settings.json     # 个人偏好(model 默认、status line 风格、output style)
.claude/settings.json       # 团队共享(permissions、hooks、统一标准)
.claude/settings.local.json # 本机临时(本地调试 hook、个人 token)
```

**几条铁律**:

1. **token / API key 永远 user 级或 local 级**(不进 git)
2. **团队约束放 project**(deny 规则、统一 hook)
3. **个人偏好放 user**(model、output style)
4. **临时调试放 local**(改完不污染团队)

**冲突的高发地**:

| 冲突 | 谁该胜出 |
| --- | --- |
| 团队 deny vs 个人 allow | 团队 deny 胜(deny 优先级最高) |
| 团队 model 设 Sonnet vs 个人想 Opus | 个人 user level 覆盖 |
| 团队 hook vs 个人 hook | 两个都跑(hook 是叠加的) |

## 三、Prompt / Skill / Slash 的版本管理

LLM 应用 prompt 是源代码——**应该按代码管**。

### 3.1 进 git

`.claude/skills/pr-review/SKILL.md` 提交进 git。

### 3.2 review prompt 改动

prompt 改动比代码 review 更需要谨慎——**一句话改了 LLM 行为**。每次改 prompt 写 PR 描述说明:

```
Before: skill description 偏宽,半数无关 PR 也激活
After:  限制为代码 PR,排除文档 PR
评估:运行 evals 集成,激活率从 80% → 35%,准确率不降
```

### 3.3 prompt evals(评测集)

写一组测试用例(`evals/`),改 prompt 之前跑一遍,改后再跑。

```
evals/
  pr-review/
    case-1-clean-pr.md
    case-2-with-bug.md
    case-3-style-issue.md
```

每次跑 `evals` 看 pass/fail。**改 prompt 不跑 evals = 凭感觉迭代**。

## 四、成本监控

### 4.1 监控指标

- 每天总成本
- 每用户成本(SaaS)
- 每个 endpoint / 任务类型成本
- Cache 命中率
- 模型分布(Haiku / Sonnet / Opus 占比)

## 五、调试:出问题怎么定位

### 5.1 Claude Code 用户视角

按这顺序查:

1. **`Ctrl+R` 看 transcript**——LLM 干了什么、调了什么 tool、传了什么参数
2. **`/status`** 看 effective settings(model / hooks / MCP servers)
3. **`/cost`** 看 token 用量异常
4. **`/clear`**:context 太长 LLM 变傻 → 清掉重来
5. **重启 session**:某些状态卡住了

### 5.2 写应用视角

1. **保留每次调用的 raw request / response**(出事时唯一线索)
2. **按 user_id / session_id 索引**——能拉某用户某次的完整对话
3. **错误分类**(LLM 幻觉 / API 超时 / 权限 deny / tool 报错)
4. **复现脚本**:用录制的 messages 离线跑 LLM 复现

### 5.3 LLM 幻觉的几种表现

- **编工具名**(调一个不存在的 tool)→ tool 描述补充
- **编参数**(传错字段)→ schema 严格化
- **编结论**(假信息)→ 加 citations 或要求"基于 X 才能说"
- **编路径**(不存在的文件)→ 让它先 Read 验证

## 六、出事故怎么处理

### 6.1 立刻

1. **关闭/限流入口**:确保当前用户不会再被影响
2. **标记 session**:把出事的 session_id / user_id 拎出来
3. **拉 transcript**:看 LLM 干了什么

### 6.2 分类

| 事故类型 | 应对 |
| --- | --- |
| LLM 调了不该调的 tool | 加 deny 规则 / permission callback |
| LLM 误导用户 | 加 prompt 约束 + citations 要求 |
| 工具失败 | tool 内部加重试 / fallback |
| 成本爆炸 | 加 max_turns / per-user 配额 |
| 数据泄漏 | 立刻 audit + 通报 + 禁用相关 server |

### 6.3 复盘

- 时间线
- 根因
- 临时修复
- 永久修复
- 如何防范

## 七、灾备:LLM 服务挂了怎么办

Anthropic 也会 outage(2024 / 2025 偶发过几次)。**关键应用**:

### 7.1 多上游

```python
PRIMARY = "anthropic"  # 直连
FALLBACK = "bedrock"   # AWS Bedrock 上的 Claude

try:
    resp = call_anthropic_direct(...)
except UpstreamDown:
    resp = call_anthropic_bedrock(...)
```

Bedrock / Vertex / 直连同时有,某个挂了切。

### 7.2 优雅降级

- LLM 全挂 → 退化到模板回复 / 人工 fallback
- 部分能力挂 → 降级 model(Opus 挂用 Sonnet)
- 工具挂 → 跳过该工具,告诉 LLM "工具暂不可用"

### 7.3 监控 + 演练

- monitor 各上游延迟 / 错误率
- 季度做一次"切换上游"演练

## 八、合规与数据治理

### 8.1 数据走过 LLM 的留痕

某些行业(医疗 / 金融 / 法律)要求"所有客户数据流转有审计":

- 每条 LLM 调用记 prompt + response + 决策结果
- 写 immutable storage(S3 + object lock)
- 保留期按法规

### 8.2 客户数据脱敏

LLM 之前主动 mask 客户 PII:

```python
masked = mask_pii(user_message)   # 邮箱 → user@xxx.com,姓名 → 张某
resp = client.messages.create(messages=[{"role": "user", "content": masked}], ...)
```

### 8.3 训练数据回流

Enterprise plan 默认你的 prompt **不进入** Anthropic 训练数据;消费 plan 可能进。**生产应用一律用 enterprise / API plan**。

## 十、团队 alignment 的几个套路

LLM 应用最容易出现的事:**团队对"怎么用 LLM" 没共识,各自摸索半年**。

实践:

1. **每周 demo**:谁有有趣的 Claude Code 流程?分享给团队
2. **#claude-code 频道**:答疑 / 分享 / 提需求
3. **共享 skill / slash 仓库**:团队成员往里加,所有人共享
4. **prompt 改动通知**:在 slack 发改动 PR
5. **季度 review**:看用量、成本、痛点
6. **新人 onboarding**:专门一节"团队 Claude 工作流"

## 十二、未来方向

2026 年这套生态在三个方向继续演进:

1. **Agent skill 标准化**:更多的开放 skill 库,跨厂商复用
2. **MCP 网络效应**:更多 SaaS 提供官方 server,内部企业 server 大量出现
3. **Coding Agent 走深**:Claude Code 这类 CLI Agent 进化到能跑长任务、维护复杂项目

学完这套**先用一段时间真做事**——做一个真实项目,用 Claude Code 写,用 Agent SDK 包装某个流程,接一两个内部 MCP server。**不动手的知识半年后就忘 80%**。

具体下一步建议:

- **第 1 月**:Claude Code 完全用上,把团队约定写进 settings + skill
- **第 2 月**:接两个有用的 MCP server(GitHub + 一个内部系统)
- **第 3 月**:用 Agent SDK 写一个小工具(CI 自动化 / 文档维护 / 客服)
- **第 4 月**:开始用 Anthropic API 在产品里做 LLM feature(优先 cache + tool use)
- **第 5 月**:看数据 + 成本,迭代
- **第 6 月**:把"我用了什么"分享给团队 / 写成博客

按这个节奏,**半年后你是团队里 Claude 生态的 power user**,且**看清"AI Coding"这一波到底带来了什么改变**。
