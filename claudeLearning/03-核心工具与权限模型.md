# 核心工具与权限模型

Claude Code 看似能干无穷事,但实际它能动的"动作"只有十几个——剩下的都是这十几个的组合。**搞清楚每个工具能干什么、不能干什么、怎么被权限拦,你就掌握了 Claude Code 的"操作系统层"**。

> 一句话先记住:**Read / Edit / Write 管文件,Bash 管命令,Glob / Grep 管搜索;Task 派子 Agent,WebFetch / WebSearch 管联网;再加上 MCP 注册的外部工具就是全部**。所有"它在做什么"的问题,最终都能拆成这几个工具的调用。

---

## 一、内置工具速查表

| 工具 | 干什么 | 典型参数 | 备注 |
| --- | --- | --- | --- |
| **Read** | 读文件(代码、PDF、图片、Jupyter) | `file_path`, `offset`, `limit` | 不能读目录;PDF 大文件要传 `pages` |
| **Edit** | 精确字符串替换 | `old_string`, `new_string`, `replace_all` | 必须先 Read 过才能 Edit;`old_string` 必须唯一 |
| **Write** | 整文件写入(覆盖) | `file_path`, `content` | 改已有文件优先用 Edit;Write 主要给新文件 |
| **NotebookEdit** | 改 Jupyter notebook 单 cell | `notebook_path`, `cell_id`, `edit_mode` | 用得不多 |
| **Bash** | 跑 shell 命令 | `command`, `timeout`, `run_in_background` | 默认 cwd 持久,shell state 不持久(每条独立)|
| **BashOutput** / **KillShell** | 看后台任务输出 / 停 | `bash_id` | 配合 `run_in_background` 用 |
| **Glob** | 按 glob pattern 找文件 | `pattern`, `path` | 比 ls + find 快,按修改时间排 |
| **Grep** | ripgrep 搜代码 | `pattern`, `path`, `glob`, `output_mode` | 比 grep / ag 强大,默认 ripgrep 引擎 |
| **Task** | 派 subagent 干子任务 | `subagent_type`, `prompt`, `description` | **第 7 篇专讲** |
| **TodoWrite** / **TodoList** | 维护任务清单 | `todos` | 让 Claude 自己管理多步任务 |
| **WebFetch** | 拉一个 URL,小模型总结 | `url`, `prompt` | 不能拉鉴权页面 |
| **WebSearch** | 联网搜索 | `query`, `allowed_domains`, `blocked_domains` | 仅美区可用 |
| **SlashCommand** | 触发 slash command | `command`, `args` | LLM 主动调内置或自定义 slash |
| **ExitPlanMode** | 退出 plan mode 进入执行 | — | 第 9 篇讲 |

> **MCP 工具**会以 `mcp__<server>__<tool>` 的形式出现在工具列表里,权限规则同样用这个完整名字匹配。

## 二、Read / Edit / Write 的隐含规则

这三个最常用,有几条**协议级**的硬规则:

### 2.1 Edit 必须先 Read

Edit 一个文件之前,**当前 session 内必须 Read 过它至少一次**——否则 Edit 会直接失败。

> 这不是 bug,是设计。它防止 LLM 在没看过文件的情况下"瞎改"。**写自己的 Agent 时强烈建议沿用这条规则**。

### 2.2 `old_string` 必须唯一

```
Edit(file_path, "function foo() {", "function foo(arg) {")
```

如果文件里有两个 `function foo() {`,Edit 报错。两种解法:

1. **加上下文,让 `old_string` 唯一**:把前后几行也带上
2. **`replace_all: true`**:全部替换(适合改名)

### 2.3 Write 是"覆盖"

Write 一个已存在的文件 = 完全覆盖原内容。**Edit 改局部、Write 创建新文件或大改写**。日常 90% 都该用 Edit。

### 2.4 Read 的"行号 + tab"格式

Read 返回的是 `cat -n` 格式:

```
   1\tfirst line
   2\tsecond line
```

**Edit 时不要把 `1\t` 这部分当成内容**——只取 tab 之后才是真实文件内容。

## 三、Bash 的几个非显然行为

```bash
# CWD 持久
cd /tmp
ls   # 在 /tmp 里

# shell state 不持久
export FOO=bar
echo $FOO   # 空,因为这是新一条 Bash 调用,新 shell
```

**所以**:

| 想要的行为 | 写法 |
| --- | --- |
| 一连串依赖的命令 | 用 `&&` 串成一条 Bash 调用 |
| 独立的命令 | 拆成多条 Bash 调用(可以并行!) |
| 长时间任务 | `run_in_background: true`,稍后 BashOutput 读结果 |
| 切目录后保持 | `cd /xxx`,后续 Bash 都在 `/xxx` 里 |

**并行**:多条 Bash 同时发(同一条 message 里),Claude Code 会真的并行跑。批量任务可以省很多时间。

### 3.1 timeout

默认 2 分钟,上限 10 分钟。长任务**别用 timeout 等**,直接 `run_in_background`。

### 3.2 安全:permissions 优先用 Bash matcher

Bash 是最危险的工具,**permissions 主要在 Bash 上下功夫**。下一节专讲。

## 四、Glob / Grep:为什么不该让 Claude 用 `find` / `grep`

很多人第一次用会想"为什么不直接 `Bash(grep -r ... )`?"——能用,但**不该用**。

### 4.1 Grep(ripgrep)

| 维度 | 用 Grep 工具 | 用 `Bash(grep -r)` |
| --- | --- | --- |
| 引擎 | ripgrep,默认快 | 看你装没装 rg |
| .gitignore 自动跳过 | ✅ | ❌(还需要 `--exclude-dir`) |
| 输出 mode | `content` / `files_with_matches` / `count` 三选 | 自己写 awk |
| 上下文 | `-A` / `-B` / `-C` | ✅ |
| 多行 | `multiline: true` | 难 |
| token 友好 | LLM 直接结构化结果 | 一堆原始文本 |

```jsonc
// 找所有 useState 调用,只看代码文件
Grep({
  "pattern": "useState\\(",
  "path": "src",
  "glob": "*.{ts,tsx}",
  "output_mode": "content",
  "-n": true
})
```

### 4.2 Glob

按文件名 pattern 找文件,默认按 mtime 排:

```jsonc
Glob({
  "pattern": "src/**/*.test.ts",
  "path": "."
})
```

比 `find . -name "*.test.ts"` 快、且自动忽略 git ignore 项。

> **规则**:能用 Glob/Grep 不要用 Bash;能用 Read 不要用 cat。这是 Anthropic 自己的 Claude Code 内核 prompt 反复强调的——**专用工具更结构化、更安全、更 token 高效**。

## 五、权限模型:matcher 怎么写

这一部分是 Claude Code 配置里**最容易写错**的地方。

### 5.1 三种规则:allow / deny / ask

```json
{
  "permissions": {
    "allow": ["Bash(npm test:*)", "Read(./src/**)"],
    "deny":  ["Bash(rm -rf*)", "Read(./.env*)"],
    "ask":   ["Bash(git push:*)"]
  }
}
```

匹配优先级:**deny > ask > allow > 默认行为**。

### 5.2 matcher 语法

格式是 `<工具名>(<参数 pattern>)`,参数 pattern 由两部分组成:

- 工具名:`Bash` / `Edit` / `Read` / `mcp__notion__search_pages` / ……
- 参数 pattern:用 glob 风格匹配工具的"主参数"

各工具的"主参数":

| 工具 | matcher 例子 | 匹配什么 |
| --- | --- | --- |
| Bash | `Bash(npm test:*)` | 命令以 `npm test` 开头 |
| Bash | `Bash(git status)` | 精确匹配 `git status` |
| Bash | `Bash(*)` | 所有 bash(慎用) |
| Read | `Read(./src/**)` | 读 src 下任何文件 |
| Read | `Read(./.env*)` | 读 .env、.env.production 等 |
| Edit | `Edit(./src/**)` | 改 src 下任何文件 |
| Write | `Write(./tmp/**)` | 写 tmp 下任何文件 |
| WebFetch | `WebFetch(domain:github.com)` | 拉 github.com 的 URL |
| MCP tool | `mcp__github__*` | 任何 github MCP 工具 |
| MCP tool | `mcp__github__create_pr` | 仅这一个 |

**Bash 的 `:*` 写法**特别重要:

- `Bash(npm test)` —— **精确匹配** `npm test` 这一条命令,不允许带任何参数
- `Bash(npm test:*)` —— `npm test` 加任何参数都允许

> 大多数情况你想要的是 `:*`(允许参数)。**写错没加 `:*` 会导致同一类命令需要反复确认**。

### 5.3 实战:渐进式收紧

**第一周**:全开 acceptEdits,看它日常调哪些命令、读哪些目录。
**第二周**:把高频安全命令加进 allow,把危险操作加进 deny,把重要操作加进 ask。

```json
{
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": [
      "Bash(npm:*)",
      "Bash(pnpm:*)",
      "Bash(bun:*)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git show:*)",
      "Bash(rg:*)",
      "Bash(ls:*)",
      "Bash(cat:*)",
      "Bash(jq:*)",
      "Read(./**)",
      "Glob",
      "Grep",
      "WebFetch(domain:docs.anthropic.com)"
    ],
    "ask": [
      "Bash(git push:*)",
      "Bash(git commit:*)",
      "Bash(npm publish:*)",
      "Bash(docker:*)",
      "Bash(kubectl:*)",
      "Edit(./.env*)"
    ],
    "deny": [
      "Bash(rm -rf*)",
      "Bash(git push --force*)",
      "Bash(git reset --hard*)",
      "Read(./.env)",
      "Read(./.env.production)",
      "Read(./**/*.pem)",
      "Read(./**/*.key)"
    ]
  }
}
```

**deny 比 allow 优先**,所以 `Read(./**)` 不会让它读到 `.env`。

### 5.4 fewer-permission-prompts skill

每次确认权限是日常最大的摩擦点。Claude Code 有个内置 skill `fewer-permission-prompts`——**它会扫你最近的 transcript,把高频被问的命令打成 allowlist 加进 settings**。

```
你: 我每次跑 npm test 都要确认,烦
Claude: (自动调 fewer-permission-prompts skill)
       发现 npm test:* / git diff:* / rg:* 这些反复在问
       建议加进 .claude/settings.json 的 allow
       要我加进去吗?
```

> 这是 Claude Code"自我配置"的好例子。**别手写 allowlist,让它自己审**。

## 六、`additionalDirectories`:操作 cwd 之外的目录

默认 Claude Code 只能动启动它的目录。如果你要让它读 `/var/log` 或写 `/tmp/scratch`:

```json
{
  "permissions": {
    "additionalDirectories": ["/var/log", "/tmp/scratch"]
  }
}
```

**别给宽**——给个具体目录,而不是 `/`。

## 七、Tool 之外:不算工具但相关

### 7.1 Plan Mode

不是工具,是一种**模式**。`/plan` 进入,LLM 只读不写,先把计划列出来,你 approve 后再 ExitPlanMode 进入实际执行。第 9 篇专讲。

### 7.2 Subagents (Task)

把一段子任务派给独立的 subagent 跑——独立上下文、独立工具集、独立预算。第 7 篇专讲。

### 7.3 SlashCommand

`/review`、`/init`、`/security-review` 这些 slash command 也是 LLM 可调的"工具"——它能在主对话里**主动**调起一个 slash。第 4 篇专讲。

## 八、看清"它现在在用什么工具":/status 与 transcript

```bash
/status         # 看当前 session 的 effective settings,工具列表
Ctrl+R          # 进入 transcript 查看模式,看 LLM 每一次 tool 调用的输入输出
/cost           # 看本 session token 用量、估算成本
/clear          # 清掉 context
/compact        # 主动压缩(上下文长时手动调,自动也会触发)
```

**`Ctrl+R` 是调试金钥匙**——LLM 干了什么、传了什么参数、返回了什么,完全透明。第一次发现"为什么它做了 X 而不是 Y",80% 是 transcript 里能直接看出来的(prompt 模糊、工具描述不清、context 漏了)。

## 九、踩坑

1. **`Bash(*)` 当 allow 用**——等于全开,你以为安全其实裸奔。**至少分到子命令级**
2. **忘了 `:*`**——`Bash(npm test)` 只能精确匹配,加参数就再问一遍。**绝大多数情况都该写 `npm test:*`**
3. **`Edit` 之前没 `Read`**——第一次写 hook 或 SDK 时常见。**永远先 Read**
4. **`old_string` 不唯一**——一个文件里有两处 `function init() {`,Edit 报错。**多带几行上下文**
5. **改名一个 import 用 Edit 一次只改一处**——用 `replace_all: true`,或者直接 Bash `sed -i`(但要小心)
6. **大文件不传 offset / limit**——结果直接吃掉 50K context。**大文件先 grep 定位再 Read 局部**
7. **PDF 大文件不传 pages**——20+ 页的 PDF 不带 pages 直接报错。**先 pages 1-5 看目录,再读相关段**
8. **WebFetch 拉 GitHub PR / Google Docs**——这些走鉴权,WebFetch 看到的是登录页。**用 `gh pr view` 或 MCP server**
9. **`bypassPermissions` 在主机直开**——某天它会 `rm -rf /` 给你看。**只在 sandbox 用**
10. **deny 太狠 / allow 太松**——前者走两步问一次,后者出事不告而别。**第二周 audit,别一开始就把 settings 当圣经**
