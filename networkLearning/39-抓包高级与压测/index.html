    抓包高级与压测

    第 03 章讲了「装好 tcpdump、用 Wireshark 看包列」——那是入门门票。但真正的抓包高手有两层功力:写得出复杂 BPF 表达式(在百万包里精准捞出 0.01% 的可疑包)、用得溜 Wireshark 高级面板(IO Graph 找抖、Expert Info 看异常、TCP Stream 重组、着色规则一眼分组)。再配一套压测七件套(wrk / wrk2 / h2load / vegeta / iperf3 / netperf),你就能从「定位问题」一路推到「定量复现 + 量化优化效果」。第 38 章讲了攻防视角——这章把"包"和"压力"两件事一起拉到生产可用水平。

    一句话先记住:抓包高级 = BPF 字节级过滤 + Wireshark 时序/统计/重组三大武器 + tshark 脚本化提字段。压测 = 选对工具(wrk 看吞吐 / wrk2 看 P99 / h2load 看 HTTP/2 / vegeta 看恒定速率 / iperf3 看带宽 / netperf 看延迟)+ 选对策略(峰值 / 持续 / 阶梯 / 尖刺)。抓包定性、压测定量,两者合起来才能把网络性能讲清楚。

    一、为什么要"高级"抓包

    1.1 入门到瓶颈的三个症状

    症状 1:抓了 10GB pcap,Wireshark 打不开 / 卡死
症状 2:能抓但只会 host / port 这种基础过滤
        想抓"重传 + payload 含 GET + 来自某网段"束手无策
症状 3:看包列只会逐条翻,看不出"这段时间整体抖"

    入门工具用法→基础语法→高级 BPF + Wireshark 高级面板,才是抓包真正的进阶曲线。

    1.2 高级抓包的两个能力

    能力 没它的代价 有它的回报
    写复杂 BPF 抓 100GB 才能找到 1 个目标包 抓 1MB 就够,机器不卡
    用 Wireshark 统计/IO 图 看到 100 万包脑子炸 几次点击就指出抖动时刻

    经验法则:抓包前先想"我要找什么样的包"——能写成 BPF 就写成 BPF,别"先抓全集再 Wireshark 翻"——10G 包翻一晚上是新人常态。

    二、BPF 过滤进阶

    tcpdump 用的是经典 BPF(cBPF),不是 eBPF——但表达力极强,只要你会用字节偏移。

    2.1 BPF 三层语法

    第一层:协议关键字
  tcp / udp / icmp / arp / ip / ip6 / vlan

第二层:字段关键字
  host / port / net / src / dst / portrange
  例:src host 1.2.3.4 and dst port 443

第三层:字节级表达式 ←— 高级
  proto[offset:size] op value
  例:tcp[13] & 0x02 != 0   (tcp[13] 是 flags 字节,0x02 是 SYN)

    前两层是"声明式",写起来快;第三层是"过程式",任何字段都能戳到。

    2.2 必背的 8 个高级 BPF 表达式

    # 1. 只抓 SYN(不带 ACK)——客户端主动建连第一包
sudo tcpdump -i any 'tcp[tcpflags] & (tcp-syn|tcp-ack) == tcp-syn'

# 2. 只抓 SYN+ACK——服务器接受连接的回包
sudo tcpdump -i any 'tcp[tcpflags] & (tcp-syn|tcp-ack) == tcp-syn|tcp-ack'

# 3. 抓所有 RST——找谁在踢你的连接
sudo tcpdump -i any 'tcp[tcpflags] & tcp-rst != 0'

# 4. 抓 HTTP GET 请求(载荷前 4 字节 = "GET ")
sudo tcpdump -i any 'tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x47455420'

# 5. 抓 HTTP POST 请求(载荷前 4 字节 = "POST")
sudo tcpdump -i any 'tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x504f5354'

# 6. 抓 TLS ClientHello(content type=0x16 handshake, type=0x01)
sudo tcpdump -i any 'tcp[((tcp[12:1] & 0xf0) >> 2):2] = 0x1603 and tcp[((tcp[12:1] & 0xf0) >> 2)+5:1] = 0x01'

# 7. 抓 DNS 查询(UDP 53,QR=0 表示请求)
sudo tcpdump -i any 'udp port 53 and udp[10:1] & 0x80 = 0'

# 8. 抓窗口 = 0 的包(零窗口告警 = 接收方处理不过来)
sudo tcpdump -i any 'tcp[14:2] = 0 and tcp[tcpflags] & tcp-rst = 0'

    第 4 个表达式是经典中的经典,拆开看:

    tcp[12:1]               TCP 头第 12 字节(data offset + reserved)
& 0xf0                  取高 4 位
>> 2                    乘 4(因为 data offset 单位是 4 字节)
                        ——得到 TCP 头长度(20-60 字节)
tcp[那个偏移 : 4]        从 payload 起始拿 4 字节
= 0x47455420            "GET " 的 ASCII

    经验法则:第 4 个表达式背下来——tcp[((tcp[12:1] & 0xf0) >> 2):N] = 值 这套模板能抓任何 TCP payload 开头的协议:HTTP / Redis / MySQL handshake / 自定义私有协议。

    2.3 BPF 过滤 vs 显示过滤(常见混淆)

    BPF 过滤(tcpdump / 抓包阶段):
  - 在内核里执行,丢掉的包永远丢了
  - 语法:host 1.2.3.4 and tcp port 443
  - 性能高,适合"我只想要这种包"

显示过滤(Wireshark / 看包阶段):
  - 在 GUI 里执行,只是隐藏不显示,包还在
  - 语法:ip.addr == 1.2.3.4 && tcp.port == 443
  - 灵活,适合"先抓全集回头慢慢挑"
    场景 用 BPF 用显示过滤
    服务器流量大,只关心一类 ✅ ✗(撑爆磁盘)
    想反复换条件分析 ✗(每次重抓) ✅
    抓未知问题 宽松 BPF 宽松抓 + Wireshark 反复筛

    经验法则:生产排障 = 在抓包阶段用 BPF 缩流量到 1/100,在 Wireshark 用显示过滤再切片分析——两层结合最高效。

    # 1. 转 pcap 为可读文本(跟 Wireshark 一行一行一样)

tshark -i eth0 -Y 'http.request' \
       -T fields -e http.request.method -e http.host -e http.request.uri

# 5. 算每秒包数(配合 awk)
tshark -r big.pcap -T fields -e frame.time_epoch | \
  awk '{print int($1)}' | sort | uniq -c

    五、典型问题的"包级"特征

    下面是一张「症状 → 包特征」对照表,记下来排障 5 秒诊断:

    现象 包级特征 排查方向
    重传严重 同一 seq 多次出现,时间间隔 ~200ms-1s 翻倍 链路丢包 / 拥塞
    乱序 seq 不连续后又补回 多路径 / 网卡 RSS
    零窗口 Window=0 包,后续 Window Update 接收方应用 hang 住
    握手失败 反复 SYN 没 SYN+ACK,或 SYN+RST 端口没开 / 防火墙
    RST 异常 连接中突然 RST 应用 abort / 中间盒 / 半关
    延迟 ACK ACK 比预期晚 40-200ms 对端 Nagle vs 本端 Delayed ACK
    MTU 黑洞 大包丢、小包通,traceroute 某跳 ICMP 被丢 PMTU 探测失败
    TLS 握手慢 ClientHello → ServerHello 间隔大 服务器证书验证 / OCSP 卡

    5.1 重传深度分析

    Wireshark 在 Expert Info 里把重传分成几种:

    Retransmission         普通重传(RTO 触发,200ms+ 后重发)
Fast Retransmission    快重传(收到 3 个 dup ACK)
Spurious Retransmission 假重传(其实没丢,是延迟到了)
TCP Out-of-Order       乱序到达
TCP Dup ACK            重复 ACK(对端在催)

    判断丢包严重程度:

    重传率 = retransmission 包数 / 总数据包数

< 0.1%   忽略
0.1-1%   关注但不一定干预  
1-5%     必须查(明显抖)
> 5%     严重(链路坏 or 拥塞)

    5.2 零窗口诊断

    正常:    Window: 65535     接收方还有 64KB 空间收
告警:    Window: 0         接收方满了,你别发了
恢复:    Window Update     接收方处理完一些,告诉你"现在 X 字节空"

    Wireshark 显示过滤:tcp.window_size_value == 0 and tcp.flags.reset == 0

    根因 99% 是:接收方应用没及时 read()——socket buffer 撑满。不是网络问题,是应用问题——查应用线程是不是卡住、GC 是不是 STW、是不是 epoll 没注册。

    5.3 RST 来源排查

    RST 来源    特征                                   排查方向
─────────────────────────────────────────────────────────────────
应用主动     正常关闭后又收包 → RST                close 顺序、半关
内核        端口未监听 → SYN 直接 RST              端口 / 进程
中间盒      连接空闲很久 → 突然 RST,TTL 异常      LB / 防火墙超时
网络抖动    链路恢复后 stale 连接 → RST            正常,重连即可
对端崩溃    keepalive 失败 → RST                  对端进程 / OOM

    关键技巧:看 RST 包的 TTL 跟正常包是否一致——TTL 不一样基本是中间盒(防火墙 / NAT / LB)发的,不是真正的对端。

    六、压测工具全景

    抓包定位完问题、改完代码,怎么证明"真的更快了"?压测。

    6.1 七件套对比

    工具 协议 模型 杀手特性 适用场景
    wrk HTTP/1.1 协程+epoll 极简、单机几十万 QPS 吞吐峰值测试
    wrk2 HTTP/1.1 同上+恒定速率 修了 wrk 的 coordinated omission 问题 真实 P99 测试
    h2load HTTP/2/3 nghttp2 HTTP/2 多路复用、HTTP/3 (QUIC) HTTP/2/3 压测
    vegeta HTTP/1.1 Go,恒定速率 报告很专业,易脚本化 CI/CD 自动化
    iperf3 TCP/UDP 纯传输层 测带宽极限 链路带宽
    netperf TCP/UDP 纯传输层 TCP_RR 测延迟极致准 内网延迟
    ab(ApacheBench) HTTP/1.0 单线程串行 简单 不要用(模型陈旧,数据不准)

    经验法则:别再用 ab 了——单线程、不支持 keep-alive 默认、不支持 HTTPS over HTTP/2,数据失真严重。入门用 wrk,要 P99 用 wrk2,HTTP/2 用 h2load,CI 用 vegeta。

    6.2 wrk 详解

    最广用的 HTTP 压测工具——单机能打几十万 QPS,核心模型是 epoll + LuaJIT 脚本。

wrk -t4 -c1000 -d30s --latency http://api.example.com/

# 输出

  4 threads and 1000 connections
  Thread Stats   Avg      Stdev     Max   +/- Stdev
    Latency    25.43ms   12.10ms   234ms    78%
    Req/Sec     9.84k     1.23k    12.4k    72%
  Latency Distribution
     50%    23ms
     75%    32ms
     90%    45ms
     99%    98ms
  1,178,234 requests in 30.01s, 234.5MB read
  Requests/sec:  39,259.46
  Transfer/sec:    7.81MB

    5 秒钟读 wrk 输出:

    1. Requests/sec(QPS)         看吞吐
2. Latency Distribution P50/P99 看延迟分布
3. Stdev > Avg                  说明抖动大,要警惕
4. Errors / Socket errors        非 0 就有问题(connect / read / write timeout)
5. +/- Stdev 列                  显示偏离平均的程度,越大越乱

    带 Lua 脚本测复杂场景:

    -- post.lua
wrk.method = "POST"
wrk.body   = '{"id":123,"name":"test"}'
wrk.headers["Content-Type"] = "application/json"
wrk.headers["Authorization"] = "Bearer eyJhbGc..."

request = function()
   local id = math.random(1, 1000000)
   return wrk.format(nil, "/api/v1/users/" .. id)
end

    wrk -t4 -c1000 -d30s -s post.lua http://api.example.com/

    6.3 wrk2:修了 coordinated omission

    wrk 的致命缺陷:它按"上一个请求完成后立刻发下一个"的模型测——如果服务器卡了 1 秒,wrk 那一秒只发 1 个请求,统计上根本看不出"卡了 1 秒会影响多少请求"。

    这叫 coordinated omission——一个非常隐蔽的 P99 失真。

    wrk2 修法:按恒定速率发请求,不管上次完没完成。

    # 强制 50000 QPS,持续 30 秒
wrk2 -t4 -c100 -d30s -R50000 --latency http://api.example.com/

    输出多一项:

    50.000%   12.43ms
75.000%   18.92ms
90.000%   25.63ms
99.000%   89.12ms
99.900%  234.50ms     ← 真实的 P99.9
99.990%  892.30ms     ← 真实的 P99.99
99.999% 1234.50ms     ← 真实的 P99.999

    经验法则:SLO 报告必须用 wrk2 / vegeta(恒定速率)出——wrk 适合摸吞吐峰值,但 P99 数字不能信。

    6.4 h2load:HTTP/2 + HTTP/3 压测

    # HTTP/2,100 连接,每连接 100 stream,共 100 万请求
h2load -n1000000 -c100 -m100 https://api.example.com/

# HTTP/3 (QUIC)
h2load --alpn-list=h3 -n100000 -c100 -m100 https://api.example.com/

    关键差异:h2load 支持 -m 参数(每连接的并发 stream)——这是 HTTP/2 多路复用的核心。用 wrk 测 HTTP/2 测出来的数据基本不对(wrk 只支持 HTTP/1.1)。

    6.5 vegeta:CI 友好

    # 100 QPS 持续 30 秒
echo "GET http://api.example.com/" | vegeta attack -rate=100 -duration=30s | \

    6.6 iperf3:链路带宽测试

    和上面 4 个不一样,iperf3 测的不是 HTTP,是裸 TCP/UDP 带宽——用来排查"网线 / 网卡 / 链路"层面的问题。

    # 服务端
iperf3 -s

# 客户端,测 TCP 上行带宽 10 秒
iperf3 -c <server-ip> -t 10

# 输出
[ ID] Interval           Transfer     Bitrate         Retr
[  5]   0.00-10.00  sec  11.5 GBytes  9.87 Gbits/sec   0   sender
[  5]   0.00-10.00  sec  11.5 GBytes  9.87 Gbits/sec       receiver

# 测 UDP(看丢包率)
iperf3 -c <server-ip> -u -b 1G -t 10

# 双向同时
iperf3 -c <server-ip> --bidir

    iperf3 的"Retr"列是 TCP 重传次数——排链路质量第一指标。

    6.7 netperf:延迟极致

    测纯延迟(Round-Trip Time)用 netperf 的 TCP_RR / UDP_RR 模式——比 ping 准得多(ping 是 ICMP,内核优先级跟 TCP 不一样)。

    # 服务端
netserver

# 客户端测 TCP_RR(每秒能完成多少次小请求小响应)
netperf -H <server-ip> -t TCP_RR -- -r 1,1
# Trans/s = 1 / RTT 秒  → RTT = 1/Trans

# UDP 同理
netperf -H <server-ip> -t UDP_RR -- -r 1,1

    经验法则:判断"网络快不快"先 iperf3 看带宽 + netperf 看 RTT——HTTP 压测出来的数字混了应用层开销,排链路必须用裸传输层工具。

    七、压测策略:不是"开打就完了"

    工具会用,还得选对策略——不同问题要不同策略。

    7.1 四种常见策略

    策略         做法                              用途
────────────────────────────────────────────────────────────────
峰值测试      QPS 慢慢拉,直到 P99 >SLO         找系统极限
持续测试      固定 QPS,跑 1-24 小时           查内存泄漏 / 文件句柄
阶梯测试      QPS 50% → 80% → 100% → 120%     找拐点
尖刺测试      平稳后突然 5x 流量 30 秒          看自愈能力 / 限流

    决策树:

                     想要什么?
                     ↓
      ┌──────┬──────┴──────┬──────┐
      ↓      ↓             ↓      ↓
   极限峰值  P99 准确     稳定性  自愈/弹性
   wrk      wrk2/vegeta  持续    尖刺
                +长时间   测试    测试

    7.2 容易踩的坑

    坑 后果 正确做法
    压测客户端跟服务在同机 互相争 CPU,数据失真 客户端和服务分开机器
    网卡千兆但压测要打 10G 压测客户端先瓶颈 检查客户端 iperf3 自测
    HTTP keep-alive 关掉 都在测 TCP 三次握手,不是测 API wrk/h2load 默认开,别关
    没预热 第一秒 JIT/缓存没起来,数据偏低 跑 5 秒 warm up 再统计
    客户端 ephemeral port 用尽 压测只能跑 28000 QPS 然后卡死 调 net.ipv4.ip_local_port_range
    没看 server 端资源 服务 100% CPU 还在压,数据只看到延迟暴涨 同时看 top / vmstat / pidstat
    只跑一次 噪音影响巨大 至少 3 次取中位

    7.3 Linux 内核侧调参(压测前)

    # 客户端临时端口范围(默认 32768-60999,只 28K 个)
sudo sysctl -w net.ipv4.ip_local_port_range="10000 65535"

# TIME_WAIT 复用(客户端必开)
sudo sysctl -w net.ipv4.tcp_tw_reuse=1

# socket 句柄上限
ulimit -n 1048576

# 服务端 backlog
sudo sysctl -w net.core.somaxconn=65535
sudo sysctl -w net.ipv4.tcp_max_syn_backlog=65535

    经验法则:压测前一定先 iperf3 测客户端到服务端的裸带宽——发现裸带宽就 1Gbps,你 HTTP 压测打 10Gbps 永远打不出来,根本不是服务问题。

    八、抓包 + 压测的"组合拳"

    工程上最有威力的是两者结合——先压测制造负载,同时抓包看包级表现。

    8.1 标准实验流程

    1. 服务端开启 tcpdump 抓包(滚动 / 限大小)
   sudo tcpdump -i any -w out-%H%M.pcap -C 100 -W 5 port 443

2. 客户端跑 wrk2 阶梯压测,每段 60 秒
   wrk2 -t4 -c100 -d60s -R10000 https://...
   wrk2 -t4 -c100 -d60s -R20000 https://...
   wrk2 -t4 -c100 -d60s -R30000 https://...

3. 服务端同时记录 metrics
   while sleep 1; do
     ss -s | grep -i estab
     cat /proc/net/snmp | grep -i tcp
   done > metrics.log

4. 找到 P99 突变的那一段,对应时间窗口的 pcap

5. Wireshark 打开:
   - IO Graph 看那段重传率
   - Expert Info 看异常类型
   - 提一个抖动期 stream 跟正常期 stream 对比

    8.2 案例:30K QPS 时 P99 飙到 800ms

    压测数据:
  10K QPS  → P99 25ms
  20K QPS  → P99 35ms
  30K QPS  → P99 800ms ← 拐点

抓包看 30K 那一段:
  IO Graph 红线(重传)激增
  Expert Info: TCP Zero Window 出现 ~200 次
  Stream 重组看到一个 HTTP/2 流挂了 1.2 秒等 Window Update

ss -s 同步看:
  Recv-Q 在多个 socket 上堆积到 6MB

结论:
  服务端应用层没及时消费 socket buffer
  → 接收窗口耗尽 → 客户端被迫等 → P99 爆

fix:扩大应用 worker 池 + 加 io_uring 减小 read 开销
再压:30K QPS P99 回到 35ms

    这个案例的精髓:抓包给出"为什么慢"的微观证据,压测给出"什么 QPS 开始慢"的宏观曲线,两者合在一起才能定位 + 量化。

    九、踩坑提醒

    1. tcpdump 不加 -s 0:默认截到 96 字节,后续 Wireshark 看不到完整 payload
    2. tcpdump -i any 抓不到 VLAN 标签:需要 -i eth0.100 指定
    3. 抓 loopback(lo)需 -i lo:-i any 默认有但有时表现奇怪
    4. 大 pcap 直接用 Wireshark 打开:卡死,先 tshark 拆 stream
    5. 抓 TLS 不配 SSLKEYLOGFILE:看到的全是密文,定位上层问题等于盲打
    6. wrk 测 HTTP/2:wrk 只支持 1.1,数字根本不对——用 h2load
    7. 压测客户端一台机器:打不到服务的实际峰值,要么多机协调要么用 vegeta cluster
    8. 压测时没看服务端 CPU/内存:看到延迟涨说"网络问题",其实是 CPU 100%
    9. wrk 的 Stdev > Avg:你以为均值是真实情况,其实数据噪到不能信
    10. 不开网卡多队列(RSS):单核处理所有中断,服务端先瓶颈在中断 CPU
    11. iperf3 服务端 -P 多线程模式没开:测出来带宽偏低,以为链路坏了
    12. netperf 只跑 10 秒:数据量太少,P99 没意义,至少 5 分钟

    十一、小结

    抓包和压测是网络工程师的"X 光机 + 体重秤"——一个看得见微观结构,一个量得出宏观负载。入门工程师只用其中一个,高级工程师两个都用,且会让两者互相验证。

    记住三件事:

    1. BPF 字节级表达式是抓包的真正杠杆——tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x47455420 这套模板能抓任何 payload 起头特征
    2. Wireshark 的 IO Graph + Expert Info 是排时序问题的最快路径——5 秒诊断 80% 抖动
    3. wrk / wrk2 / h2load / vegeta 各有职责,选错就出错数据——尤其 P99 必须用恒定速率工具,wrk 测峰值不测 SLO

    抓包和压测都是"工具",真正难的是把两者组合进统一的排障方法论——这就是下一章,也是本系列收官篇。下一篇 40 章把前 39 篇的所有协议 / 工具 / 视角 / 案例串成一套可复用的「分层定位法」——任何一个网络问题,你都知道第一步该敲哪个命令、第二步该看哪个字段、第三步该切哪个视角。把整个系列的零散知识收成"反射"。
