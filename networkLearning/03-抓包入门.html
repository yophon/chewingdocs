抓包入门 ​

「不会抓包,网络永远是黑盒」——这是第 01 章里反复强调的一句话。90% 的人只会 console.log 或 printf 调网络问题,真正快速定位的人都是抓包党:「先看流量,再讨论可能性」。这章把工程师的抓包六件套(tcpdump / Wireshark / tshark / mitmproxy / SSLKEYLOGFILE / curl --trace)一次讲清,看完你能完整抓下浏览器访问 google.com 的全流量,包括 HTTPS 解密。

一句话先记住:抓包 = 把协议栈每一层的字节流落地成 pcap 文件,再用工具分析。tcpdump 抓、Wireshark 看、tshark 脚本化、mitmproxy 解 HTTPS、SSLKEYLOGFILE 让 Wireshark 也能解 HTTPS。核心技能不是装工具,是「会写 BPF 过滤」+「会用 Statistics 看时序和重传」——这两项练熟,排障能力直接进 top 5%。

一、抓包到底在抓什么 ​

回顾第 02 章,数据从应用层经过 4 次封装到链路层。抓包工具坐在 PF_PACKET 这个 socket 上,从内核协议栈直接拷贝一份原始帧:

应用 → TCP → IP → Eth → 网卡硬件 → 网线
              ↓
         libpcap (Linux: PF_PACKET, BSD: BPF)
              ↓
         tcpdump / Wireshark 把帧落到 pcap 文件

关键点:

  • 抓包是旁路复制,不影响真实流量(99% 情况下)
  • 抓的是完整的链路层帧——MAC 地址、IP 头、TCP 头、payload 全有
  • 默认抓不到 loopback(lo 接口)走的本机流量,要 -i any 或 -i lo 显式指定
  • 网卡 offload(TSO/GSO)会让抓到的"巨包"看起来不像网线上跑的——见第 02 章

经验法则:生产排障第一招永远是 tcpdump——比加日志快 10 倍,且不用动代码。

二、tcpdump:命令行的瑞士军刀 ​

2.1 最常用三招 ​

bash
# 1. 抓所有接口的 80 端口流量,落盘
sudo tcpdump -i any -w out.pcap port 80

# 2. 抓某个 IP 的所有流量,屏幕输出
sudo tcpdump -i eth0 -nn host 1.2.3.4

# 3. 抓 SYN/RST 包(诊断连接被拒)
sudo tcpdump -i any 'tcp[tcpflags] & (tcp-syn|tcp-rst) != 0'

2.2 必须会的参数 ​

参数含义备注
-i any抓所有接口排障第一选择
-i eth0指定接口性能更好
-w file.pcap写到 pcap 文件后续 Wireshark 打开
-r file.pcap读 pcap 文件离线分析
-n不解析 IP 为主机名快、不卡
-nnIP 和端口都不解析推荐
-s 0完整抓包(默认 96 字节截断)写文件时必加
-c 100抓 100 个包后退出防止抓爆磁盘
-X十六进制 + ASCII看 payload
-A只 ASCII看明文 HTTP
-v / -vv / -vvv详细程度看更多字段
-tttt完整时间戳排时序问题

2.3 抓包文件管理 ​

bash
# 滚动抓:每 100MB 一个文件,最多 5 个,自动覆盖最老
sudo tcpdump -i any -w cap-%Y%m%d-%H%M%S.pcap \
             -C 100 -W 5 -G 3600 \
             port 443

# -C 100   每个文件 100MB
# -W 5     最多 5 个文件
# -G 3600  每小时切一个新文件

经验法则:生产抓包必滚动 + 限大小——直接 -w out.pcap 一晚上能抓爆 200GB 把磁盘撑死。

三、BPF 过滤语法:省 90% 时间的核心技能 ​

3.1 语法分三类 ​

1. 类型(Type):  host / port / net
2. 方向(Dir):   src / dst
3. 协议(Proto): tcp / udp / ip / ether / arp / icmp

组合起来:

bash
# 主机
host 1.2.3.4               # 进出该 IP 的所有包
src host 1.2.3.4           # 该 IP 发出的
dst host 1.2.3.4           # 发给该 IP 的

# 端口
port 80                    # 端口 80(进出)
src port 51234             # 源端口 51234
dst port 443               # 目标 443
portrange 8000-9000        # 端口范围

# 网段
net 192.168.1.0/24
src net 10.0.0.0/8

# 协议
tcp / udp / icmp / arp
ether host aa:bb:cc:dd:ee:ff

3.2 组合表达式 ​

bash
# AND / OR / NOT
host 1.2.3.4 and port 80
host 1.2.3.4 and not port 22
(host 1.2.3.4 or host 5.6.7.8) and port 443

# 工程师天天用的几个
'tcp port 80 or tcp port 443'           # 抓 web 流量
'udp port 53'                           # 抓 DNS
'icmp'                                  # 抓 ping / TTL exceeded
'host api.example.com and tcp port 443' # 抓某域名 HTTPS

3.3 抓特定 TCP flag(神技) ​

bash
# 只抓 SYN 包(看新建连接)
'tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack == 0'

# 抓 RST(看异常断连)
'tcp[tcpflags] & tcp-rst != 0'

# 抓 FIN(看正常断连)
'tcp[tcpflags] & tcp-fin != 0'

# 抓握手三个包
'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0'

经验法则:「连接拒绝」「连接重置」「连接超时」三种现象,先抓 SYN 和 RST——一眼能看出是 SYN 没回(目标不通)、还是回 RST(端口没监听)、还是回 SYN+ACK 但应用没继续(应用 hang)。第 04 章会展开这种排障流。

3.4 抓特定字节(高级) ​

bash
# 抓 HTTP GET 请求(payload 前 4 字节是 "GET ")
'tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x47455420'
# 0x47455420 = "GET "

# 抓 HTTP/2 PRI 字符串
'tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x50524920'

太啰嗦,实际用 tcpdump -A | grep 也行——但写在过滤里抓得少、解析快。

四、Wireshark:可视化分析的核心 ​

tcpdump -w 落盘,scp 拉本地,open file.pcap 用 Wireshark 看——这是标准工作流。

4.1 必看的列 ​

打开 pcap 后默认列就行,关注这些:

No.    Time         Source         Dest          Protocol  Length  Info
1      0.000        192.168.1.10   93.184.216.34 TCP       74      51234 → 80 [SYN]
2      0.045        93.184.216.34  192.168.1.10  TCP       74      80 → 51234 [SYN, ACK]
3      0.046        192.168.1.10   93.184.216.34 TCP       66      51234 → 80 [ACK]
4      0.047        192.168.1.10   93.184.216.34 HTTP      150     GET / HTTP/1.1
...

Time 列改成「Seconds Since Previous Captured Packet」(View → Time Display Format),一眼看出哪一段是瓶颈。

4.2 必装的过滤(Display Filter) ​

注意:Wireshark 的 Display Filter 跟 BPF 不一样——更强大,但语法不同:

ip.addr == 1.2.3.4              # IP 过滤
tcp.port == 443                 # 端口
http.request.method == "GET"    # HTTP 方法
http.response.code >= 400       # HTTP 错误响应
tcp.flags.reset == 1            # RST 包
tcp.analysis.retransmission     # 重传
tcp.analysis.duplicate_ack      # 重复 ACK
tcp.analysis.zero_window        # 零窗口
tls.handshake.type == 1         # TLS ClientHello
dns.qry.name contains "google"  # DNS 查询
frame.time_delta > 0.5          # 间隔 > 500ms 的包

经验法则:Wireshark 的 tcp.analysis.* 系列是排障神器——它会自动标记重传、乱序、零窗口、丢包等异常。红/黑色行 99% 是有问题的。

4.3 右键菜单的两个救命功能 ​

A. Follow → TCP Stream

选中任意一个 TCP 包,右键 → Follow → TCP Stream:

+----------------------------------------+
| GET / HTTP/1.1                         |  ← 红色 = 客户端发
| Host: example.com                      |
| User-Agent: curl/8.0                   |
|                                        |
| HTTP/1.1 200 OK                        |  ← 蓝色 = 服务端发
| Server: nginx                          |
| Content-Length: 1256                   |
|                                        |
| <html>...                              |
+----------------------------------------+

自动重组整条流的应用层数据——不用你拼 TCP segment。HTTP/2 用「Follow → HTTP/2 Stream」。

B. Statistics → I/O Graph

最强可视化——把整个 pcap 按时间画成流量图:

带宽 ↑                          
     │       ╱╲      ╱╲        
     │      ╱  ╲    ╱  ╲       
     │     ╱    ╲  ╱    ╲      
     │ ___╱      ╲╱      ╲___  
     └──────────────────────→ 时间
       T0  T1   T2   T3   T4

可以叠加多条曲线:全部包 / 重传 / 错误 / 特定过滤器结果。P99 抖动、突发丢包、节流都能直接看出来——详见第 04 章时序视角案例。

4.4 Statistics 菜单的其他金矿 ​

菜单用途
Conversations列出所有「IP 对/端口对」的字节数排序
Endpoints列出所有 IP 的总流量
Protocol Hierarchy协议占比(L7 / TLS / TCP / IP)
Flow Graph时序图(像 UML 序列图)
TCP Stream Graphs单个流的窗口/RTT/吞吐曲线

经验法则:「带宽看 I/O Graph、问题看 Conversations、单流看 TCP Stream Graphs」——三招覆盖 80% 排障。

五、tshark:Wireshark 的命令行版本(脚本化必备) ​

tshark = Wireshark 的命令行,适合服务器上跑、CI 里跑、批量处理。

bash
# 等价 tcpdump
sudo tshark -i eth0 -w out.pcap

# 但牛在过滤 + 字段提取
tshark -r out.pcap -Y 'http.request' \
       -T fields -e http.request.method -e http.host -e http.request.uri

# 输出
GET    example.com    /
POST   api.example.com /v1/orders

5.1 实用脚本 1:统计每个 URL 被请求多少次 ​

bash
tshark -r out.pcap -Y 'http.request' \
       -T fields -e http.request.uri \
       | sort | uniq -c | sort -rn | head

5.2 实用脚本 2:统计每个域名的 TLS 握手延迟 ​

bash
tshark -r out.pcap -Y 'tls.handshake.type == 1 or tls.handshake.type == 2' \
       -T fields -e frame.time_relative \
                 -e tls.handshake.extensions_server_name \
                 -e tls.handshake.type

5.3 实用脚本 3:看 TCP 重传率 ​

bash
total=$(tshark -r out.pcap -Y 'tcp' | wc -l)
retrans=$(tshark -r out.pcap -Y 'tcp.analysis.retransmission' | wc -l)
echo "scale=4; $retrans / $total" | bc

经验法则:抓包用 tcpdump、看用 Wireshark、批量分析用 tshark——三个工具职责不同。

六、HTTPS 抓包:SSLKEYLOGFILE 这个魔法 ​

问题:HTTPS 流量加密,Wireshark 看到的是密文,Stream 重组也是密文。怎么办?

答案:让客户端把 TLS 协商出来的对称密钥写到一个文件里,Wireshark 拿这个文件去解密。

6.1 标准步骤(curl / Chrome / Firefox 都支持) ​

bash
# 1. 设置环境变量
export SSLKEYLOGFILE=/tmp/sslkeys.log

# 2. 启动浏览器(必须从这个 shell 启动,继承环境变量)
google-chrome &
# 或 Firefox

# 3. 同时抓包
sudo tcpdump -i any -w /tmp/https.pcap port 443 &

# 4. 浏览器访问 https://example.com

# 5. Wireshark 打开 pcap,配置:
#    Edit → Preferences → Protocols → TLS
#    "(Pre)-Master-Secret log filename" → /tmp/sslkeys.log

Wireshark 自动解密——能看到完整 HTTP 明文 + HTTP/2 帧 + ALPN 协商内容。

6.2 curl 直接抓 ​

bash
SSLKEYLOGFILE=/tmp/sslkeys.log curl https://example.com/
# sslkeys.log 现在有这次握手的密钥

6.3 SSLKEYLOGFILE 文件长啥样 ​

# SSL/TLS secrets log file, generated by NSS
CLIENT_RANDOM 4f8e5...  abc123...
SERVER_HANDSHAKE_TRAFFIC_SECRET 4f8e5...  def456...
CLIENT_HANDSHAKE_TRAFFIC_SECRET 4f8e5...  789abc...
SERVER_TRAFFIC_SECRET_0 4f8e5...  fedcba...
CLIENT_TRAFFIC_SECRET_0 4f8e5...  ...

每次握手生成一组密钥——Wireshark 按 Client Random 匹配。

经验法则:SSLKEYLOGFILE 是 HTTPS 调试的唯一民用方式——比 mitmproxy 简单(不需要装根证书),且真实流量(不是被代理过的)。生产敏感场景小心——这文件能解你之后所有抓的包。

七、mitmproxy:中间人式抓包 ​

和 SSLKEYLOGFILE 的本质区别:

维度SSLKEYLOGFILEmitmproxy
原理拿密钥解密真实流量中间人替换证书
客户端要不要改设环境变量装根证书 + 配代理
看到的内容真实流量被解密的复制流
能不能改请求不能能(改完转发)
适用场景真实复现、抓 HTTP/3调试 API、改请求测试

7.1 三种模式 ​

1. Regular Proxy   客户端显式配代理(http_proxy=http://localhost:8080)
2. Transparent     iptables 把流量重定向到 mitmproxy(无感)
3. Reverse Proxy   挂在服务前,所有进来的流量都过它

7.2 一分钟跑起来 ​

bash
# 安装(macOS / Linux)
brew install mitmproxy   # 或 pip install mitmproxy

# 启动 TUI 版本
mitmproxy -p 8080

# 终端会显示一个交互式 UI
# 等待客户端连接

# 客户端配置(以 curl 为例)
# 1. 装根证书
mkdir -p ~/.mitmproxy
mitmdump  # 第一次运行会生成 ~/.mitmproxy/mitmproxy-ca-cert.pem
sudo cp ~/.mitmproxy/mitmproxy-ca-cert.pem /usr/local/share/ca-certificates/
sudo update-ca-certificates

# 2. 通过代理访问
curl --proxy http://localhost:8080 https://example.com/

# mitmproxy TUI 实时显示请求,按回车看详情、e 编辑重发

7.3 脚本化(mitmdump) ​

python
# 改 User-Agent
# script.py
def request(flow):
    flow.request.headers["User-Agent"] = "MyHackBot/1.0"

def response(flow):
    if "api" in flow.request.url:
        print(f"{flow.request.method} {flow.request.url} → {flow.response.status_code}")
bash
mitmdump -s script.py -p 8080

7.4 透明代理(无感) ​

bash
# Linux
# 1. 启动 mitmproxy 透明模式
mitmproxy --mode transparent --showhost

# 2. iptables 重定向
sudo sysctl -w net.ipv4.ip_forward=1
sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 80  -j REDIRECT --to-port 8080
sudo iptables -t nat -A PREROUTING -i eth0 -p tcp --dport 443 -j REDIRECT --to-port 8080

移动 App 抓包神器——手机连这台机器做网关,所有流量自动走 mitmproxy。

踩坑提醒:mitmproxy 看不到证书锁(certificate pinning)的应用——银行 App、微信、TikTok 这种,客户端硬编码了证书指纹,mitmproxy 的伪证书过不了校验,直接断连。需要 frida 脚本 hook 客户端代码绕过——超出本系列范围。

八、curl --trace:不抓包的"半抓包" ​

有时候抓包麻烦——curl 自带个穷人版:

bash
# 完整 trace(包括二进制 hex)
curl --trace /tmp/curl.log https://example.com/

# 只 trace HTTP 头(更常用)
curl -v https://example.com/

# trace ASCII 版本
curl --trace-ascii /tmp/curl.log https://example.com/

输出看到 TLS 握手细节、HTTP 头、body——适合调 API 不方便起 tcpdump 的场景。

bash
# openssl s_client 看完整 TLS 握手
openssl s_client -connect example.com:443 -servername example.com -tls1_3
# 能看到证书链、ALPN、cipher suite、session ticket

九、实操:抓自己浏览器一次访问 google.com ​

把上面所有工具串起来,完整跑一遍:

9.1 准备 ​

bash
# Terminal 1
export SSLKEYLOGFILE=/tmp/sslkeys.log
google-chrome --user-data-dir=/tmp/chrome-cap &

# Terminal 2
sudo tcpdump -i any -w /tmp/google.pcap -s 0 \
             '(tcp port 443 or udp port 443 or udp port 53)'

9.2 操作 ​

浏览器地址栏输入 https://www.google.com/ 回车。等页面加载完,Ctrl-C 停止抓包。

9.3 分析(Wireshark) ​

bash
open /tmp/google.pcap

配置 TLS 解密:Edit → Preferences → Protocols → TLS → "(Pre)-Master-Secret log filename" → /tmp/sslkeys.log。

看几个关键点:

1. 第一个 DNS 请求:
   过滤 dns.qry.name == "www.google.com"
   看 query → response,记下解析延迟

2. TCP 三次握手:
   过滤 tcp.flags.syn == 1
   看 SYN → SYN+ACK → ACK 三个包,记 RTT

3. TLS 握手:
   过滤 tls.handshake
   看 ClientHello → ServerHello → ChangeCipherSpec → Finished
   注意 TLS 1.3 是 1 RTT,1.2 是 2 RTT

4. HTTP/2 流量:
   过滤 http2
   解密成功的话能看到 HEADERS 帧 / DATA 帧 / SETTINGS 帧
   看到 :authority: www.google.com 就对了

5. 整体时序:
   Statistics → Flow Graph → 选 TCP Flows
   一张图看完整个连接过程

9.4 看到的画面长这样 ​

T+0    DNS query "www.google.com"           
T+10   DNS response 142.250.x.x             ← DNS 解析 10ms
T+11   TCP SYN → 142.250.x.x:443
T+50   TCP SYN+ACK ←                        ← 1 RTT = 39ms
T+51   TCP ACK →
T+52   TLS ClientHello →                    ← 包含 SNI / ALPN
T+90   TLS ServerHello + Cert + Finished ← ← 1 RTT = 38ms
T+91   TLS Client Finished →
T+92   HTTP/2 SETTINGS →
T+92   HTTP/2 HEADERS (GET /) →
T+130  HTTP/2 DATA <html...> ←              ← TTFB
...

这一张图能讲清楚 80% 的网络性能问题——后面 04 章会用类似图分析三个真实 case。

十、特殊场景:容器、Kubernetes、云 ​

10.1 Docker 容器抓包 ​

bash
# 容器外抓 docker0 网桥
sudo tcpdump -i docker0 -w cap.pcap

# 进容器抓(容器需有 tcpdump)
docker exec -it <container> tcpdump -i eth0 -w /tmp/cap.pcap
docker cp <container>:/tmp/cap.pcap .

# 推荐:nsenter 进容器网络命名空间,用宿主机 tcpdump
PID=$(docker inspect -f '{{.State.Pid}}' <container>)
sudo nsenter -t $PID -n tcpdump -i eth0 -w cap.pcap

10.2 Kubernetes Pod 抓包 ​

bash
# kubectl debug 注入 sidecar(K8s 1.18+)
kubectl debug -it <pod> --image=nicolaka/netshoot --target=<container>
# 在 sidecar 里 tcpdump -i any -w /tmp/cap.pcap

# 或者 ksniff 插件
kubectl sniff <pod> -n <namespace> -p
# 自动起一个 sidecar、抓包、流式传给本地 Wireshark

十二、踩坑提醒 ​

  1. -s 0 没加,只抓 96 字节——payload 看不全,根本无法重组
  2. 生产抓包不滚动——一晚上撑爆磁盘
  3. 抓包加在网卡 offload 没关——巨包欺骗你
  4. Wireshark Display Filter 和 BPF 混用——一个用 host,一个用 ip.addr
  5. 抓 lo 接口不显式 -i lo——本机流量抓不到
  6. HTTPS 抓包不配 SSLKEYLOGFILE——花一上午盯着密文
  7. mitmproxy 抓证书锁应用——直接断连还以为 mitmproxy 坏了
  8. mitmdump 改请求改了 Content-Length 没同步——后端报 400
  9. 生产开 mitmproxy 透明代理——单点故障 + 性能杀手
  10. 抓包文件命名没带时间戳——出问题发给团队,谁都不知道哪个是哪个
