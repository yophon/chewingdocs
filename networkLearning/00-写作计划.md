# 网络深入系列写作计划

> 来源:`未来系列规划.md` 第一类「补完整知识图」第三项,⭐⭐⭐⭐。
> webLearning 浅讲了 HTTP / Cookie / CORS,backendLearning 第 55 章把 TCP/TLS/HTTP 粗扫一遍——但**真要深挖到 RFC 级**,做网关、做中间件、扛 DDoS、调 P99,得开独立系列。
> 本系列定位:**进阶补课**——假设你会用 HTTP API、能写后端服务,但不一定懂包怎么走、TCP 握手怎么实现、TLS 1.3 凭什么 1 RTT、QUIC 解决了什么、eBPF 能干什么。**40 章一次讲透**。

---

## 设计思路

七层结构,从「包视角」到「工程实战」:

```
基础与心智模型(协议栈 / 抓包 / 链路层)
   → 网络层(IP / 路由 / ICMP / NAT)
   → 传输层(UDP / TCP / 拥塞 / MPTCP)
   → 安全层(密码学 / TLS 1.2 / 1.3 / mTLS / PKI)
   → 应用层(HTTP/1.1/2/3 / QUIC / WebSocket / WebRTC / DNS)
   → 内核网络(socket / epoll / io_uring / eBPF / XDP / DPDK)
   → 工程实战(Nginx / Envoy / CDN / WAF / 抓包 / 排障)
```

**这个系列不是 RFC 翻译**——RFC 太密、术语门槛太高;**本系列偏"工程师视角"**:每个协议都对应一个真实的"性能问题或者诡异 bug",学完能直接用在排错和调优。

不抠每个字段——**而是讲清楚"协议设计的取舍、演进的根本原因、生产中怎么调"**——你写后端 / 中间件 / 网关时必须懂的网络层。

## 写作模板

每篇遵守"为什么 → 是什么 → 怎么用 → 何时不该用"的四段式:

```
1. 为什么有这个协议 / 设计
   - 上一代有什么缺陷
   - 业界为什么逼出新的标准

2. 协议格式 / 关键字段
   - 报文结构 ASCII 图
   - 关键字段的"为什么这么设计"

3. 工程视角怎么用 / 怎么调
   - curl / openssl / dig / tcpdump 实操
   - sysctl / 配置文件调参
   - 真实案例 (P99 抖、握手慢、连不上)

4. 何时不该用 / 坑在哪
   - 性能边界 / 兼容性
   - 常见 bug
   - 替代方案
```

不堆 RFC 全文——**只贴关键字段图和命令行**,让读者「知道这一层在干嘛 + 抓包能看出来」就够了。

## 节奏

```
01-05  心智:看完知道"网络在工程师眼里到底是什么 + 会抓包"
06-10  网络层:IP / 路由 / NAT,做后端 / 做网关必懂
11-16  传输层:**最值钱的一层**,P99 / 重传 / 拥塞全在这
17-21  TLS:HTTPS 时代的硬通货,mTLS / 服务网格全在这
22-26  HTTP:**第二值钱**,API / 网关 / Web 全在这
27-29  DNS:被低估的瓶颈,99% 慢启动来自这
30-33  内核网络:把抽象拉到最底,真懂"快在哪"
34-36  代理 / LB / CDN:工程实战,网关 / SRE 必懂
37-38  安全:扛攻击的基本功
39-40  排障:把前 38 篇串起来
```

**一个硬指标**:看完 11-16 + 17-21 这 11 篇,你应该能在白板前讲清楚:

- "为什么 TLS 1.3 比 1.2 少一个 RTT"
- "为什么 BBR 在公网比 CUBIC 抗丢包,但内网两者差不多"
- "为什么我的服务 P99 抖动来自 TIME_WAIT 撑爆 socket 表"

能讲清楚一题这个系列就值了。

## 反对的写法

这个系列**不写**这些东西,见到了就是写错了:

1. 「逐字段翻译 RFC」——RFC 自己就在那儿,这里讲取舍和坑
2. 「先讲个故事 / 个人情怀」——直接进重点
3. 罗列命令但不解释——每个命令配场景
4. 「这是面试常考」——真懂了面试自然过,不是来刷题的
5. 抄"图解 HTTP"或"网络是怎样连接的"——那两本都好,这里只补它们没讲透的工程细节
6. 不画图就讲握手 / 状态机 / 帧——**这些东西不画 ASCII 永远讲不清楚**
7. 不抓包就讲协议——纸上谈兵,失去本系列灵魂
