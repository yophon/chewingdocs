# 网络心智三视角

第 01 章一笔带过的「包视角 / 时序视角 / 状态机视角」——这章把它讲透。**为什么要分三个视角?因为同一个网络现象,从不同视角看到完全不同的真相**:同一个「请求慢」,包视角看是不是有重传,时序视角看是哪段卡住,状态机视角看是不是 socket 没回收。**老司机和新人的差距不是工具熟不熟,是会不会切视角**。这章每个视角配一个生产案例——看完你应该能在排障会议上**主动「换视角」推进讨论**,而不是被动跟着别人查。

> 一句话先记住:**网络是一个 4 维问题**——空间(包从哪到哪)、时间(什么时候发什么)、状态(连接处于哪个阶段)、规模(系统层面统计)。**包视角看微观、时序视角看动态、状态机视角看抽象、统计视角看宏观**。**遇到问题先问自己「我现在在哪个视角」**——这是把网络从"玄学"拉回"工程"的关键。

---

## 一、为什么要三个视角

```
同一个现象 「P99 突然飙到 5 秒」:
  
  包视角看:    那段时间有没有重传?有没有 RST?
  时序视角看:  时间花在 DNS / TCP / TLS / HTTP 哪段?
  状态机视角看:连接是不是没建上?是不是 TIME_WAIT 满了?
  
  三个视角不冲突,是同一个问题的三个切片。
```

**新人最常犯的错误**:**不切视角,只盯一个**——比如「我加了重传日志,没看到重传啊」,但实际问题在 TIME_WAIT 撑爆了 ephemeral port,根本就没建上连接,谈不上重传。

**老手的工作流**:

```
现象 → 假设视角 A → 抓数据 → 不对 → 切视角 B → 抓数据 → 验证
              ↑
        关键是「切」——三个视角都试过,大概率定位
```

> 经验法则:**排障会议上提议「换个视角看看」是高级信号**——比说「我再查查」专业 10 倍。

## 二、视角一:包视角(Packet View)

### 2.1 定义

**把每一个 IP 包当成独立单元看**,关注:

- 谁发给谁(SrcIP → DstIP, SrcPort → DstPort)
- 协议是啥(TCP / UDP / ICMP)
- TCP flag 是啥(SYN / ACK / FIN / RST / PSH)
- 长度多少(payload 大小、是不是纯 ACK)
- 包序号、ACK 号(看是不是重传 / 乱序)

**工具**:tcpdump / Wireshark 的 Packet List。

### 2.2 关键 TCP flag 和它们的含义

```
SYN  ─── 我想建连
ACK  ─── 我收到你的包了(序号 = X)
FIN  ─── 我发完了,准备关
RST  ─── 异常重置,立刻拆连接
PSH  ─── 给应用层立刻送上去,别缓存
URG  ─── 紧急数据(实际很少用)
```

**RST 的几个常见原因**(超经典):

| 场景 | 原因 |
| --- | --- |
| 客户端 connect 不存在端口 | 服务端内核回 RST |
| 服务端进程 crash 留下 socket | 后续包到达回 RST |
| keepalive 探测失败 | 一端发 RST 强制清理 |
| LB / 防火墙超时 | 中间设备发 RST(非端到端) |
| 应用主动 `setsockopt(SO_LINGER, {1, 0})` 后 close | 跳过四次挥手发 RST |
| TCP Reset Attack | 攻击者伪造 RST 中断连接(运营商劫持也用这招) |

> 经验法则:**抓到 RST 第一件事看 src IP**——是对端回的还是中间设备回的?**TTL 突然变化、IP 不一致就是中间设备插的**。

### 2.3 案例 1:接口偶发 502,看到 RST 怎么定位

**现象**:线上某 API 接口 1% 概率返回 502,Nginx 日志显示 `upstream prematurely closed connection`。

**包视角排查**:

```bash
# 1. 在 Nginx 上抓包,过滤上游服务的 IP
sudo tcpdump -i any -w nginx-up.pcap host 10.0.1.20 and port 8080

# 2. 等到一次 502 复现,停止抓包

# 3. Wireshark 打开,过滤
tcp.flags.reset == 1
```

**看到的图像**:

```
T+0     Nginx → app:8080  [SYN]
T+1     app:8080 → Nginx  [SYN, ACK]
T+1     Nginx → app:8080  [ACK]
T+2     Nginx → app:8080  [PSH, ACK]  POST /v1/orders {...}
T+10    app:8080 → Nginx  [PSH, ACK]  HTTP/1.1 200 OK
T+10    Nginx → app:8080  [ACK]
... (空闲 60 秒)
T+70    Nginx → app:8080  [PSH, ACK]  POST /v1/orders {...}  ← 复用连接发新请求
T+70    app:8080 → Nginx  [RST]                              ← 应用回 RST!
```

**关键发现**:**RST 是上游应用回的**(src IP = app:8080,TTL 一致),不是中间设备。

**根因**:上游服务的 keepalive_timeout 是 60 秒,Nginx 的 keepalive 是 75 秒——**Nginx 把"已死"的连接当成活的复用,发请求过去就被 RST**。

**修复**:把上游 keepalive_timeout 调大到 90 秒,或把 Nginx 的 keepalive 调到 50 秒——**永远短于上游**。

> 经验法则:**Nginx → 上游 502 大半是 keepalive 错位**——`upstream keepalive` 必须 < 上游服务的 keepalive_timeout。

### 2.4 包视角的进阶招

```
看到的包                      可能的故事
────────────────────────────────────────────────────────────────
SYN 出但没回 SYN+ACK           对端不通(防火墙 / 网络分区 / 进程挂)
SYN 出回了 RST                  对端没监听该端口
连续多个 SYN(每隔 1/2/4/8s)   重传(SYN 在网上丢了)
连续相同 seq 的 PSH 包          重传(对端没回 ACK)
DUP ACK x 3                     收到 3 个重复 ACK,即将快重传
零窗口(win=0)                  接收方缓冲满了,发送方被卡住
out-of-order                    乱序到达(多路径 / IP 分片)
```

> 第 12 / 13 章会展开 TCP 状态机和拥塞控制的细节,这里只用来识别问题。

## 三、视角二:时序视角(Timeline View)

### 3.1 定义

**把一次完整请求按时间轴展开**,关注:

- 每段花了多久(DNS / TCP / TLS / HTTP TTFB / HTTP body)
- 哪段是瓶颈
- 是稳定慢还是抖动
- 多个并发请求之间有没有相互影响

**工具**:Wireshark Flow Graph、I/O Graph、Chrome DevTools Network、自家的 trace 系统(Jaeger / SkyWalking)。

### 3.2 一次 HTTPS 请求的"标准"时序

```
T+0      DNS query
T+10ms   DNS response               ← DNS 段 = 10ms
T+11     TCP SYN
T+30     TCP SYN+ACK                ← 1 RTT(握手)= 19ms
T+31     TCP ACK
T+32     TLS ClientHello
T+50     TLS ServerHello+...        ← 1 RTT(TLS 1.3 全握)= 18ms
T+51     TLS Finished
T+52     HTTP request
T+71     HTTP response head         ← TTFB = 19ms(应用处理)
T+71     HTTP response body         ← 一般和 head 一起或紧跟
─────────────────────────────────
总耗时 71ms
```

**P99 抖动定位关键**:**画出抖动期的时序图,跟正常时序对比**——哪段拉长就是哪段的锅。

### 3.3 案例 2:P99 从 50ms 抖到 800ms,瓶颈在哪段

**现象**:线上 API P99 平时 50ms,每天 10:00-10:05 之间飙到 800ms,P50 不动。

**时序视角排查**:

```bash
# 1. 抓 5 分钟,带 trace
sudo tcpdump -i any -w spike.pcap -G 60 -W 5 host api.example.com

# 2. Wireshark 打开,Statistics → I/O Graph
```

**I/O Graph 设置**:

```
曲线 1:全部包       Filter: (空)              Y轴:Packets/sec
曲线 2:重传         Filter: tcp.analysis.retransmission
曲线 3:零窗口       Filter: tcp.analysis.zero_window
曲线 4:DNS 查询     Filter: dns.qry.name
```

**看到的图像**:

```
Packets ↑                                        
        │                  零窗口曲线峰值        
        │                       ▼                
        │                   ╱─╲                  
        │                  ╱   ╲                 
        │ _______________╱     ╲_______________  
        │                                        
        └──────────────────────────────────────→ 时间
        T+0   T+1   T+2   T+3   T+4   T+5
                          ↑     
                          这 30 秒零窗口暴增
```

**继续切片**:挑一个抖动期的请求 Follow TCP Stream:

```
T+0       客户端 → 服务端  GET /api/big-list
T+1       服务端 → 客户端  HTTP/1.1 200 OK + 第 1 块 data (60KB)
T+1       客户端 → 服务端  ACK win=64240  
T+5       客户端 → 服务端  ACK win=32120  ← 窗口减半
T+8       客户端 → 服务端  ACK win=0      ← 零窗口!
... (停 200ms)
T+208     客户端 → 服务端  ACK win=64240  ← 缓冲腾出来,继续
T+208     服务端 → 客户端  剩余 data...
```

**关键发现**:**客户端处理慢导致 TCP 接收缓冲满,反向流控了发送方**。每次抖动多 200ms 是缓冲腾出来的等待时间。

**根因**:有个新上线的客户端版本在 10:00 跑批量任务,**单进程串行处理大列表**,read 速度跟不上服务端发送速度,触发 TCP 流控反压。

**修复**:客户端改用 streaming 解析,或拆分请求(`?page=N`)。

> 经验法则:**P99 抖动 90% 是「某段时间某段时序拉长」**——不要盯总时间,盯「哪段」。**Wireshark 时间格式改成「Seconds since previous packet」一秒搞定**。

### 3.4 时序视角的常见结论

| 哪段长 | 嫌疑 |
| --- | --- |
| DNS 段长 | 本地 DNS 不靠谱 / TTL 过期 / 上游 DNS 慢 |
| SYN-SYN+ACK 长 | 网络 RTT 大 / 服务端 SYN 队列满 / 服务端 CPU 抖 |
| TLS 握手长 | 证书链下载慢 / OCSP 查询慢 / 服务端 RSA 计算慢 |
| TTFB 长 | 服务端处理慢(查 DB / 调下游) |
| body 段长 | 带宽不够 / 接收方流控 / 拥塞 |
| 整段都长 | 网络丢包(每段都被重传影响) |

## 四、视角三:状态机视角(State Machine View)

### 4.1 定义

**把每个连接当成一个状态机**,关注:

- 连接现在处于什么状态(LISTEN / SYN_RECV / ESTABLISHED / FIN_WAIT_1 / TIME_WAIT / ...)
- 哪些状态卡住了、堆积了
- 状态转换是不是按预期走

**工具**:`ss -ant`、`netstat -nat`、`/proc/net/sockstat`、`/proc/net/tcp`。

### 4.2 TCP 11 个状态(必背)

```
        CLOSED
          │ active open
          ↓ send SYN
        SYN_SENT
          │ recv SYN+ACK, send ACK
          ↓
       ESTABLISHED ←──────────────── (passive open)
          │ active close                  │
          ↓ send FIN              SYN_RECV ← LISTEN
       FIN_WAIT_1                                
          │ recv ACK                              
          ↓                                       
       FIN_WAIT_2
          │ recv FIN, send ACK
          ↓
       TIME_WAIT          ←── 等 2 MSL(60s)
          │
          ↓
        CLOSED
        
另一侧关闭路径:
       ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
       (recv FIN)    (app close)   (recv ACK)
```

**11 个状态**:CLOSED / LISTEN / SYN_SENT / SYN_RECV / ESTABLISHED / FIN_WAIT_1 / FIN_WAIT_2 / CLOSE_WAIT / CLOSING / LAST_ACK / TIME_WAIT。

详细推导第 12 章讲。这里只用来识别"卡在哪"。

### 4.3 各状态堆积的含义

```bash
# 看本机各状态连接数
ss -ant | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn
# 或
netstat -nat | awk '/^tcp/{print $6}' | sort | uniq -c | sort -rn
```

| 状态堆积大量 | 病因 |
| --- | --- |
| **TIME_WAIT** | 短连接太多,主动关闭方堆 60 秒 |
| **CLOSE_WAIT** | **应用没调 close()** —— 应用 bug |
| **SYN_RECV** | SYN flood 攻击 / accept 队列满 |
| **FIN_WAIT_1** | 对端不回 ACK(对端死了 / 中间设备丢) |
| **FIN_WAIT_2** | 对端不回 FIN(应用没关) |
| **ESTABLISHED 极多** | 长连接池 / 连接没回收 / 内存压力 |

> 经验法则:**`ss -ant | grep -c TIME_WAIT` 是排障第一招**——一个数字说出整个系统的连接行为模式。

### 4.4 案例 3:TIME_WAIT 撑爆 ephemeral port,新连接建不上

**现象**:某调度服务每分钟向 100 个下游发送 HTTP 请求,运行半小时后开始大量 `connect: cannot assign requested address`。

**状态机视角排查**:

```bash
# 1. 看连接状态分布
ss -ant | awk 'NR>1 {print $1}' | sort | uniq -c
# 输出:
#  28732 TIME_WAIT     ← 爆炸
#    100 ESTABLISHED
#      5 LISTEN

# 2. 看 ephemeral port 范围
cat /proc/sys/net/ipv4/ip_local_port_range
# 32768  60999     ← 28232 个可用端口

# 3. TIME_WAIT 几乎用满了所有可用 port
```

**根因**:

```
每个请求 = 新建 TCP 连接(没用 keepalive)
请求完客户端主动 close → 进 TIME_WAIT 状态 → 等 60 秒(2 MSL)

每分钟 100 × 60 个请求 = 6000 新连接
TIME_WAIT 持续 60 秒 = 同时存在 6000 个 TIME_WAIT
跑 30 分钟 = ... 算不过来,反正撑爆 28000 端口
```

**为什么 TIME_WAIT 占端口**:同一对 (本地 IP, 远程 IP, 远程 port) 上,只要本地 port 还在 TIME_WAIT,就**不能复用**这个本地 port 来建新连接(哪怕不是同一对端,Linux 默认也保守)。

**三种修复**(从安全到激进):

```bash
# 方案 A:用连接池(根本修法)
# 客户端开启 HTTP keepalive,长连接复用
# 这是唯一推荐的根治方案

# 方案 B:扩大 ephemeral port 范围(临时缓解)
sudo sysctl -w net.ipv4.ip_local_port_range="10000 65000"
# 多 22000 个端口,撑久点

# 方案 C:开启 TIME_WAIT 复用(谨慎)
sudo sysctl -w net.ipv4.tcp_tw_reuse=1
# 允许新连接复用处于 TIME_WAIT 的 socket(对 outbound 安全)
# ⚠️ tcp_tw_recycle 已在 Linux 4.12 移除,别用旧博客的方案

# 方案 D:缩短 TIME_WAIT 时间
# Linux 内核里 TIME_WAIT = 60s 是硬编码,不能改
# 想改需要重新编译内核(别这么搞)
```

> 经验法则:**「TIME_WAIT 多」≠「问题」**——如果 ephemeral port 没用满,TIME_WAIT 几万都正常。**只有出现 `cannot assign requested address` 才需要修**。

### 4.5 案例延伸:CLOSE_WAIT 堆积是更严重的 bug

**CLOSE_WAIT 跟 TIME_WAIT 完全不同**:

```
TIME_WAIT  = 主动关闭方,正常状态,自动 60s 退出
CLOSE_WAIT = 被动关闭方,「对端关了,我没关」—— 应用 bug
```

**症状**:

```bash
ss -ant | grep CLOSE_WAIT | wc -l
# 5234   ← 不会自己减少,只会涨
```

**根因**:**应用代码漏了 `close(fd)`** 或者 `defer conn.Close()` 漏写。常见场景:

- Java 没在 finally 里关 InputStream / OutputStream
- Python 没用 `with` 上下文
- Go 漏写 `defer resp.Body.Close()`(超经典)

**修复**:**只能改代码**。`sysctl` 救不了——**TCP 协议层面看,close() 是应用必须主动调的**。

> 经验法则:**CLOSE_WAIT 持续上涨 = 应用 100% 有 bug**。看到这个不要查内核参数,直接去查代码。

## 五、视角组合:三个视角联动定位一个问题

真实排障往往不是单视角能解决的。一个综合案例:

**现象**:某 gRPC 服务,白天稳如老狗,凌晨 3 点 P99 周期性飙到 2 秒,持续 10 分钟。

```
步骤 1(时序视角):
  Jaeger trace 看,2 秒里有 1.9 秒花在「TCP connect」
  → 不是应用慢,是建连接慢
  
步骤 2(状态机视角):
  凌晨 3 点跑 ss -ant 统计
  → ESTABLISHED 数从 200 暴增到 5000(全是某 cron job 触发)
  → 没看到 TIME_WAIT / CLOSE_WAIT 异常
  
步骤 3(包视角):
  在凌晨 3 点抓包
  → 看到大量 SYN 重传(每隔 1s / 2s / 4s 重发)
  → 服务端 SYN+ACK 偶发不回
  
步骤 4(综合):
  ss -ltn 看 LISTEN socket 的 Send-Q
  → Send-Q 满了!accept 队列溢出
  
  netstat -s | grep -i listen
  → "5234 SYNs to LISTEN sockets dropped"  ← 关键
  
根因:cron 触发 5000 个并发连接 → 服务端 accept 队列(somaxconn=128)爆 
     → SYN 被丢 → 客户端重传 → 1s/2s 累加成 1.9s
     
修复:
  sysctl -w net.core.somaxconn=4096
  应用代码 listen(fd, 4096)
  cron job 加 jitter,不要瞬间打 5000 并发
```

**这个 case 三个视角缺一不可**——只看时序定位到「connect 慢」,还不知道是什么慢的;只看状态机,不知道有 SYN 丢;只看包,不知道总量数据。**老司机的本能就是切视角**。

## 六、视角四:统计视角(Bonus)

严格说不是「心智模型」而是「监控视角」——但实战非常有用,顺带提一下。

### 6.1 关键指标

```bash
# 全局 TCP 计数
cat /proc/net/snmp | grep -A 1 ^Tcp
# Tcp: ... InSegs OutSegs RetransSegs InErrs ...
#       ...  N      M      R           E

# 重传率
RETRANS_RATE = RetransSegs / OutSegs
# > 1% 就有问题

# 全局 socket 数
cat /proc/net/sockstat
# TCP: inuse 234 orphan 0 tw 4523 alloc 567 mem 12

# netstat 统计大全
netstat -s
# TcpExt:
#     5234 SYNs to LISTEN sockets dropped         ← accept 队列溢出
#     1234 packets dropped from out-of-order queue ← 重组缓冲溢出
#     345 connections reset due to unexpected data ← 收到 RST
```

### 6.2 用 node_exporter / netdata 持续采集

把这些 `/proc/net/*` 数据 export 到 Prometheus,Grafana 上画板,**任何时刻都能回看「事故时刻系统的网络状态」**——比事后抓包强。

> 经验法则:**生产环境必须有 「TCP 重传率 / TIME_WAIT 数 / accept 队列长度」三块基础监控**——出问题 5 分钟内能定位 80%。详细监控指标第 40 章排障方法论展开。

## 七、视角切换工具地图

| 视角 | 主要工具 | 看什么 |
| --- | --- | --- |
| **包视角** | tcpdump / Wireshark Packet List | flag / seq / payload / RST |
| **时序视角** | Wireshark Flow Graph / I/O Graph / DevTools | 哪段长 / 抖动模式 |
| **状态机视角** | ss / netstat / /proc/net/tcp | 哪个状态堆积 |
| **统计视角** | netstat -s / Prometheus / Grafana | 系统级 KPI |

切换姿势:

```
排障开始
  ↓
现象描述 → 「请求慢」「连不上」「断了」「数据错」
  ↓
默认从「时序视角」入手 → 找出大致段落
  ↓
段落定位后 → 包视角看具体包行为
  ↓
有连接异常 → 状态机视角看 ss
  ↓
要看趋势 → 统计视角看 Grafana
  ↓
循环 ← 换一个视角验证假设
```

## 九、踩坑提醒

1. **只盯一个视角**——「我加了重传日志没看到啊」,可能问题在状态机
2. **看到 RST 假设是网络问题**——很多 RST 是中间设备插的(TTL 不一致就是)
3. **TIME_WAIT 多了就慌**——不耗光 ephemeral port 没事
4. **CLOSE_WAIT 多调内核参数**——白调,只能改应用代码
5. **抓包不看 I/O Graph**——错过 80% 的趋势线索
6. **看时序只看总耗时**——必须细分到每一段
7. **`tcp_tw_recycle` 抄博客开**——4.12 之后已经移除,生产开了 NAT 后端会断流
8. **`somaxconn` 不调,listen() 只生效内核默认**——高并发服务必调
9. **Wireshark 时间显示用绝对时间**——抖动看不出,改成 delta 时间
10. **没建连接监控就上线**——出 P99 问题就是没数据可查
