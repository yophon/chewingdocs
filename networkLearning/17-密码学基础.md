# 密码学基础

上一篇 16-TCP 调优把传输层的性能榨到了极致,但有个问题没解:**裸 TCP 在公网上传任何东西都是明文**——任何路由器、运营商、咖啡店 WiFi 的中间人都能 sniff 你的密码、Cookie、API token。**TLS 是把 TCP 上面那层"裹一层加密"的协议**,而 TLS 又建立在五个基础组件之上:**对称加密、非对称加密、Hash、HMAC、密钥交换**。这一篇先把这五件事讲透——下一篇讲 TLS 时,你就能看出"哪个握手步骤在用哪个原语,为什么不能换"。

> 一句话先记住:**密码学三件事别混**——**加密(对称 / 非对称)解决"看不到"、签名(私钥签 / 公钥验)解决"伪造不了"、Hash + HMAC 解决"改不了"**。**对称快非对称慢**(差 1000 倍),所以 TLS 的标准套路是:**非对称协商一个对称密钥,后续全用对称**。**别自己写密码学**——这是行业铁律,你写的实现 99% 有侧信道漏洞。

## 一、为什么这一篇要先讲

```
TLS 1.2 ClientHello 里随便挑一个套件:
  TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
       ↑     ↑       ↑          ↑
      密钥   签名/   对称加密    Hash
      交换   认证   (AEAD 模式)

不懂这五个原语,这串名字像天书
懂了之后,这一行就告诉你了完整的握手设计
```

**TLS 不是"一个加密协议",是"一套原语的组合协议"**——你要会读"配方"。

## 二、对称加密:快但有"密钥怎么发"问题

### 2.1 模型

```
明文 ──┐
       │ + Key ──→ 密文 ──→ 网络 ──→ 密文 ──┐
       │                                    │ + Key ──→ 明文
                                            │
   双方共享同一把 Key
```

**特点**:加解密用**同一把密钥**,因此叫"对称"(symmetric)。

### 2.2 速度参考

| 算法 | 单核吞吐(现代 x86,带 AES-NI) |
| --- | --- |
| AES-128-GCM | ~5 GB/s |
| AES-256-GCM | ~3 GB/s |
| ChaCha20-Poly1305 | ~2 GB/s(无硬件加速时反超 AES) |
| 3DES(已废弃) | ~50 MB/s |
| RC4(已废弃) | ~1 GB/s |

**AES 在有 AES-NI 的 CPU 上是王者**(Intel Westmere 2010 起标配)。**ChaCha20 在没有硬件加速的 ARM、嵌入式、老 CPU 上更快**——所以移动端 / 老服务器优选 ChaCha20。

### 2.3 现代对称密码就两种:AES-GCM 和 ChaCha20-Poly1305

```
AES-GCM:
  AES (块加密) + GCM (Galois/Counter Mode, 提供 AEAD)
  128 bit / 256 bit 密钥
  12 字节 IV(标准),16 字节 tag

ChaCha20-Poly1305:
  ChaCha20 (流加密) + Poly1305 (MAC, 提供 AEAD)
  256 bit 密钥
  12 字节 nonce, 16 字节 tag
```

**TLS 1.3 只允许这两个家族**(加上 AES-CCM)——其他全删。RC4、3DES、CBC 模式全是历史包袱。

### 2.4 AEAD 是什么

**AEAD = Authenticated Encryption with Associated Data**——**加密 + 认证一体化**。

```
传统 (CBC + HMAC):
  encrypt(plaintext) → ciphertext
  hmac(ciphertext)   → tag
  问题:顺序怎么拼?MAC-then-encrypt? Encrypt-then-MAC?
        历史上无数 padding oracle 攻击都源于拼错

AEAD (GCM / Poly1305):
  encrypt_and_authenticate(plaintext, AAD) → (ciphertext, tag)
  原语级别保证:加密和认证不可分割
```

**AAD(Associated Data)**:不加密但要被认证的数据——比如 TLS record 的 header,header 不能被改但又不能加密(否则解密方读不到长度)。

> 经验法则:**2026 年新写的代码只用 AEAD**——`AES-GCM` / `ChaCha20-Poly1305` / `AES-OCB`。**别用 AES-CBC**,padding oracle 几十种变体等着你。

### 2.5 GCM 的"致命踩坑":nonce 重用

GCM 模式有个铁律:**同一个 key 下,nonce 永远不能重复**——重用一次,密钥就泄露。

```
nonce 怎么生成?
  ✗ 随机:96 bit 随机,在 2^32 次后碰撞概率 50%
        高吞吐场景一天就出事
  ✓ 计数器:从 0 递增,2^96 次才用完
  ✓ TLS 1.3 用法:per-record sequence number XOR static_iv
```

**所以 TLS 1.3 强制 nonce 用计数器**——不让用户自己造 nonce。

## 三、非对称加密:解决了"密钥怎么发"

### 3.1 模型

**两把密钥**:**公钥(public key)**可以公开,**私钥(private key)**自己留着。

```
公钥加密 → 只有私钥能解   → 用于"加密"
私钥签名 → 只有公钥能验   → 用于"签名"

注意:这是两个独立用法,不是"加密反过来就是签名"
```

### 3.2 三大算法

| 算法 | 类型 | 密钥长度 | 速度 | 当前推荐 |
| --- | --- | --- | --- | --- |
| **RSA** | 加密 + 签名 | 2048 / 3072 / 4096 bit | 慢 | 兼容性 ✓,新项目不优先 |
| **ECDSA** | 签名 | P-256 / P-384(曲线名) | 快 | 主流 |
| **Ed25519** | 签名 | 固定 256 bit | 最快 | 新项目首选 |
| **ECDH / X25519** | 密钥交换 | P-256 / Curve25519 | 快 | TLS 1.3 默认 |

**注意**:ECDSA / Ed25519 / ECDH 都是椭圆曲线(EC),但**用法不同**——**EC 是数学基础,签名 / 密钥交换是上层应用**。

### 3.3 为什么椭圆曲线比 RSA 快

**安全等级对照**(达到同等抗破解强度需要的密钥长度):

| 对称等效 | RSA / DH | ECC | 备注 |
| --- | --- | --- | --- |
| 80 bit | 1024 bit | 160 bit | 已不安全 |
| 112 bit | 2048 bit | 224 bit | 当前最低线 |
| 128 bit | 3072 bit | 256 bit | 推荐 |
| 192 bit | 7680 bit | 384 bit | 高安全 |
| 256 bit | 15360 bit | 512 bit | 顶级 |

**关键观察**:**RSA 2048 ≈ ECC 256**——但运算量不只是密钥长度比例,而是**RSA 在 mod n 上做 2048 bit 的指数运算,ECC 只在 256 bit 域上做点乘**。

### 3.4 实测速度(openssl speed)

```bash
openssl speed rsa2048 ecdsap256 ed25519 rsa4096
```

```
                  sign/s   verify/s
rsa  2048 bits     1500     50000
rsa  4096 bits      230     14000
ecdsa  P-256      30000     12000
ed25519           80000     30000
```

**结论**:
- **签名**:Ed25519 > ECDSA P-256 > RSA 2048 ≫ RSA 4096
- **验证**:RSA 反而快!**因为 RSA 验证用的指数 e=65537 很小**

> 经验法则:**服务端签得多 → 用 ECDSA / Ed25519**;**客户端验得多(Web 浏览器验证书)→ RSA 公钥反而快**。这就是为什么 CA 根证书 / Let's Encrypt 至今大量用 RSA 链。

### 3.5 RSA 加密的真相:几乎不直接用

很多教科书说"RSA 用公钥加密,私钥解密"——**实际上 TLS 里 RSA 加密只在密钥协商时用一次,而且 TLS 1.3 把它删了**。

```
为什么删?
  RSA 密钥交换:客户端用服务端公钥加密一个 pre-master secret
  问题 1:没有 PFS(前向安全)——服务端私钥泄露,过去所有流量都能解
  问题 2:Bleichenbacher padding oracle 攻击(20 年了还在出新变种)
```

**TLS 1.3 改用 (EC)DHE 做密钥交换,RSA 只用于签名**(证明服务器身份)。

## 四、Hash:数据指纹

### 4.1 模型

```
任意长度输入 ─→ Hash 函数 ─→ 固定长度输出(摘要 / digest)

性质:
  1. 单向(从摘要算不回输入)
  2. 抗碰撞(找不到两个输入 hash 相同)
  3. 雪崩(输入改 1 bit,输出变一半)
```

### 4.2 常见算法

| 算法 | 输出长度 | 状态 | 速度参考 |
| --- | --- | --- | --- |
| MD5 | 128 bit | **已破** | 600 MB/s |
| SHA-1 | 160 bit | **已破**(2017 Google 实证碰撞) | 500 MB/s |
| SHA-256(SHA-2 家族) | 256 bit | 主流 | 400 MB/s |
| SHA-384(SHA-2) | 384 bit | 高安全场景 | 300 MB/s |
| SHA-512(SHA-2) | 512 bit | 64 位机更快 | 700 MB/s |
| SHA-3-256 | 256 bit | 备胎(海绵结构) | 200 MB/s |
| BLAKE2b | 512 bit | 新设计,快 | 1 GB/s |
| BLAKE3 | 任意 | 最新,可并行 | 6 GB/s |

### 4.3 为什么不能用 MD5 / SHA-1

```
MD5:
  2004 年王小云教授发表碰撞算法
  现在普通笔记本几秒钟能找碰撞
  
SHA-1:
  2017 年 Google "SHATTERED" 攻击实证
  花费 11 万 GPU 小时,成本 ~10 万美金
  现在 PoW 算力下降到几千美金
```

**碰撞攻击的实际后果**:
- 同一个 MD5 的两个不同 PDF 可以伪造(假合同)
- 用 MD5 做证书签名 → 可伪造 CA 证书(2008 年 Flame 病毒就这么干的)
- Git 用 SHA-1 做 commit hash → 长期会出问题(已经在迁移 SHA-256)

> 踩坑提醒:**MD5 / SHA-1 用于"非密码学完整性校验"(如 ETag、文件分片去重)还能用**——别用于签名 / 密码 / 证书。

### 4.4 SHA-2 vs SHA-3

```
SHA-2(2001):  Merkle-Damgård 结构,跟 MD5 / SHA-1 同根
SHA-3(2015):  海绵结构(Sponge),完全不同设计

为什么有 SHA-3?
  防"万一 SHA-2 哪天被破"——备胎
  实际上 SHA-2 至今没有理论攻击
  SHA-3 性能反而比 SHA-2 慢一些(软件实现)
  硬件实现 SHA-3 比 SHA-2 简单
```

**工程选择**:**SHA-256 仍是默认**——有硬件加速(SHA-NI 指令集),生态好。

### 4.5 BLAKE 系列:新一代

**BLAKE3** 是 2020 年的新算法,**比 SHA-256 快 10 倍以上**:

```
为什么这么快?
  1. 树形结构,可并行
  2. 用 ChaCha 内核(加密里也用)
  3. 现代 SIMD 友好

谁在用?
  Cargo / cargo-binstall(Rust 生态)
  WireGuard 内部
  IPFS 部分场景
```

**TLS 暂时不用 BLAKE**——生态保守,但内部工具可以考虑。

## 五、HMAC:Hash 上面再加一层

### 5.1 为什么 Hash 不够

```
Alice 给 Bob 发消息:
  msg = "transfer 100 to Bob"
  hash = SHA256(msg)
  发: msg || hash

中间人 Mallory 拦截:
  改成 "transfer 10000 to Mallory"
  重算 hash(任何人都能算)
  发: msg' || hash'

Bob 验证 hash 通过——没识别出篡改
```

**Hash 只能防"自然出错",不能防"恶意篡改"——因为没有秘密**。

### 5.2 HMAC = Hash + Key

```
HMAC(K, msg) = H((K ⊕ opad) || H((K ⊕ ipad) || msg))
                                ↑
                  key 进去,中间人没 key 算不出
```

具体怎么算不重要,记住:**带共享密钥 K 的 hash**——只有持 K 的双方能验。

### 5.3 用途

```
TLS 1.2:
  握手末尾的 Finished 消息用 HMAC 校验前面所有握手数据
  AEAD 之前的 record 完整性也用 HMAC

JWT 的 HS256:
  HMAC-SHA256

API 签名(AWS Signature v4):
  HMAC-SHA256 链式调用四次
```

### 5.4 HMAC 还是 AEAD?

```
独立完整性校验(API 签名、JWT) → HMAC
通信信道加密 + 完整性          → AEAD(GCM / Poly1305)
```

**新项目通信场景别再 "AES-CBC + HMAC" 手动拼**——直接 AEAD。

## 六、密钥交换:DH 与 ECDH

### 6.1 经典 Diffie-Hellman

**1976 年的发明**——人类首次实现"两个人在公开信道上协商出一个只有他们知道的秘密"。

```
公开参数:大素数 p,生成元 g

Alice 选私钥 a (随机)         Bob 选私钥 b (随机)
计算 A = g^a mod p             计算 B = g^b mod p

         A ─────→ Bob
         Bob ←── B Alice

Alice 算 K = B^a = g^(ab) mod p
Bob   算 K = A^b = g^(ab) mod p

中间人看到 A, B,但算不出 K (离散对数难题)
```

### 6.2 ECDH:椭圆曲线版本

**把 mod p 的指数运算换成椭圆曲线点乘**——**密钥短得多**(256 bit vs 2048 bit),**速度快得多**。

```
Alice 私钥 a, 公钥 a·G
Bob   私钥 b, 公钥 b·G
共享密钥 = a·(b·G) = b·(a·G) = (ab)·G
```

`G` 是曲线上一个公开的固定点。

### 6.3 静态 vs Ephemeral(临时):为什么要 PFS

```
静态 DH:服务器 DH 私钥固定 → 一旦私钥泄露,过去所有会话都能解密
DHE / ECDHE:每次握手生成新的临时 DH 密钥 → 即使长期私钥泄露,过去会话也无法解密
                                                ↑
                                          这叫 PFS (Perfect Forward Secrecy)
```

**TLS 1.3 强制 PFS**——只允许 (EC)DHE,不允许静态 DH 和 RSA 密钥交换。

### 6.4 X25519:Curve25519 的 ECDH

```
P-256 (NIST):标准椭圆曲线,被怀疑有 NSA 后门(参数来源不透明)
Curve25519 (DJB):  Daniel J. Bernstein 设计,公开推导,简单
                  实现不容易写错(没有侧信道)
                  速度更快
```

**X25519** = 用 Curve25519 做 ECDH。**TLS 1.3 默认密钥交换就是 X25519**。

> 经验法则:**新写的 ECDH 全用 X25519**——除非合规要求(FIPS)逼你用 P-256。

## 七、数字签名:私钥签 + 公钥验

### 7.1 和"非对称加密"别混

```
加密:        公钥加密,私钥解密——让别人能给你发秘密
签名:        私钥签名,公钥验证——证明这数据是你发的

虽然都用一对公私钥,但是:
  RSA 同时支持加密和签名(数学上对称)
  ECDSA / Ed25519 只支持签名(算法上不支持加密)
  ECDH 只支持密钥交换(也不是签名)
```

### 7.2 签名流程

```
签名:   sign(priv_key, hash(msg)) → signature
验证:   verify(pub_key, hash(msg), signature) → true / false
```

注意:**真实签名是签 hash,不是签原文**——RSA 签名一次只能签 245 字节(2048 bit 模数减去 padding),签长文件直接签不下;签 hash 就够了。

### 7.3 ECDSA 的"致命弱点":nonce 不能重复

ECDSA 签名时需要一个随机数 k,**两次签名用了同一个 k → 私钥直接泄露**。

**历史血案**:
- 2010 年 Sony PlayStation 3 私钥被反编译——硬编码 k = 常数
- 多个比特币钱包因为 RNG bug 重用 k,丢币

**Ed25519 的优雅**:**确定性签名**(deterministic signature)——k 由 priv_key 和 msg hash 派生,不依赖随机数,**根本不会重复**。

> 踩坑提醒:**新项目首选 Ed25519**——完全规避 ECDSA 的 nonce 陷阱。

## 八、openssl 命令实操

### 8.1 看密码套件

```bash
# 列出本机 openssl 支持的所有套件
openssl ciphers -v 'ALL' | head -20

# 只看 TLS 1.3 套件
openssl ciphers -v -s -tls1_3
# 输出:
# TLS_AES_256_GCM_SHA384
# TLS_CHACHA20_POLY1305_SHA256
# TLS_AES_128_GCM_SHA256
```

### 8.2 性能测试

```bash
# 对称加密速度
openssl speed -evp aes-128-gcm
openssl speed -evp chacha20-poly1305

# 非对称速度
openssl speed rsa2048 rsa4096 ecdsap256 ed25519

# Hash 速度
openssl speed sha256 sha512
```

### 8.3 生成各种密钥

```bash
# RSA 2048
openssl genrsa -out rsa.key 2048

# ECDSA P-256
openssl ecparam -name prime256v1 -genkey -noout -out ec.key

# Ed25519
openssl genpkey -algorithm ED25519 -out ed.key

# X25519 (一般在程序里调,不直接命令)
openssl genpkey -algorithm X25519 -out x.key
```

### 8.4 Hash 文件

```bash
# 算文件 hash
openssl dgst -sha256 file.txt
sha256sum file.txt              # Linux 等价

# HMAC
echo -n "msg" | openssl dgst -sha256 -hmac "secret"
```

### 8.5 实战:用 openssl 加解密文件

```bash
# AES-256-GCM 加密(2026 年推荐)
openssl enc -aes-256-gcm -in plain.txt -out cipher.bin -pbkdf2

# 解密
openssl enc -d -aes-256-gcm -in cipher.bin -out plain2.txt -pbkdf2
```

`-pbkdf2` 是从密码派生密钥用的算法,**不加这个参数 openssl 默认用旧算法,是踩坑点**。

### 8.6 RSA 签名 / 验证

```bash
# 签名
openssl dgst -sha256 -sign rsa.key -out sig.bin file.txt

# 验证(用公钥)
openssl rsa -in rsa.key -pubout -out rsa.pub
openssl dgst -sha256 -verify rsa.pub -signature sig.bin file.txt
```

## 九、Python 一句话演示(cryptography 库)

```python
# AES-GCM 加解密
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

key = AESGCM.generate_key(bit_length=128)
aesgcm = AESGCM(key)
nonce = os.urandom(12)            # 注意!12 字节
ct = aesgcm.encrypt(nonce, b"hello", b"AAD")
pt = aesgcm.decrypt(nonce, ct, b"AAD")  # AAD 必须一致
```

```python
# Ed25519 签名
from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey

priv = Ed25519PrivateKey.generate()
pub  = priv.public_key()
sig  = priv.sign(b"message")
pub.verify(sig, b"message")        # 抛 InvalidSignature 即失败
```

```python
# X25519 密钥交换
from cryptography.hazmat.primitives.asymmetric.x25519 import X25519PrivateKey

a_priv = X25519PrivateKey.generate()
b_priv = X25519PrivateKey.generate()
shared_a = a_priv.exchange(b_priv.public_key())
shared_b = b_priv.exchange(a_priv.public_key())
assert shared_a == shared_b        # 双方算出同一个 shared
```

> 经验法则:**Python 写密码学只用 `cryptography` 库**(`pyca/cryptography`)——别用 `pycrypto`(已弃),别用 `hashlib` 之外的 stdlib 自己拼。**Go 用 `crypto/...` stdlib**;**Rust 用 `ring` 或 `RustCrypto`**。

## 十、密码学三大铁律

### 10.1 别自己实现

侧信道(timing / cache / power)漏洞写不胜写——你以为代码没 bug,实际上 CPU 缓存命中模式就泄露密钥。

```
CVE 列表里"侧信道"漏洞:
  Lucky 13(CBC HMAC 时序)
  CRIME(TLS 压缩)
  BREACH(HTTP 压缩)
  Heartbleed(OpenSSL 缓冲区越界)
  ROCA(Infineon RSA 密钥生成)
  Spectre / Meltdown(CPU 推测执行)
```

**全是顶级团队写的**——你写的更不行。

### 10.2 用经过审计的库

| 语言 | 推荐库 |
| --- | --- |
| C / C++ | OpenSSL 3.x / BoringSSL / libsodium |
| Python | `cryptography` |
| Go | 标准库 `crypto/...` |
| Rust | `ring` / `RustCrypto` / `rustls` |
| Node.js | 标准库 `crypto` |
| Java | JCE / Bouncy Castle |

### 10.3 别用废弃算法

| 别用 | 用什么 |
| --- | --- |
| MD5, SHA-1 | SHA-256, BLAKE2 |
| 3DES, RC4 | AES-GCM, ChaCha20-Poly1305 |
| RSA-1024 | RSA-2048+ 或 Ed25519 |
| ECB 模式 | GCM 模式(AEAD) |
| CBC + HMAC 手拼 | AEAD 一体化 |
| 静态 DH | DHE / ECDHE |
| TLS 1.0 / 1.1 | TLS 1.2 起步,1.3 优先 |

---

## 十一、踩坑提醒

1. **以为 RSA 加密是 TLS 必备**——TLS 1.3 已删,只剩签名用 RSA
2. **AES-CBC + HMAC 拼**——padding oracle 等着,直接用 AEAD
3. **GCM 重用 nonce**——key 直接报废,改用确定性 nonce 或换 ChaCha20-Poly1305
4. **ECDSA 用弱 RNG**——k 重复就泄露私钥,新项目用 Ed25519
5. **以为 SHA-256 慢**——AES-NI / SHA-NI 指令下都很快,别老想着 BLAKE
6. **拿 Hash 当 MAC 用**——`hash(secret || msg)` 有长度扩展攻击,必须用 HMAC
7. **密钥用 base64 当字符串存**——存数据库的 key 务必加密,不要明文 base64
8. **HMAC 验证用 `==`**——可能被时序攻击,务必用 `hmac.compare_digest` / 常时间比较
9. **写 "我们用 256 位 AES 比 128 位安全"**——AES-128 至今无理论攻击,256 是给"未来量子"留的余量
10. **以为椭圆曲线就一定快**——NIST P-521 比 RSA-2048 还慢,曲线选错会反向优化
11. **PBKDF2 用 1000 次迭代**——2026 年这个数字太低,至少 60 万次或换 Argon2
12. **存密码用 SHA-256(原始 hash)**——必须用 bcrypt / scrypt / Argon2,慢算法才抗暴力

## 十三、小结

这一篇没讲网络,**讲的是 TLS 的"原料表"**:

```
对称加密      AES-GCM / ChaCha20-Poly1305          → 主体加密
非对称签名    RSA / ECDSA / Ed25519                 → 证明身份
密钥交换      DHE / ECDHE (X25519)                  → 协商对称密钥
Hash          SHA-256 / SHA-384                     → 完整性 / 派生
HMAC          HMAC-SHA256                           → 认证 + 完整性
AEAD          GCM / Poly1305                        → 加密+认证一体
```

**TLS 的设计哲学就一句话**:**用慢的非对称运算协商一次,接下来全用快的对称运算**——把 1000 倍的速度差变成"摊薄"。

记住三件事:

1. **三件事别混**:加密(看不到)、签名(伪不了)、Hash(改不了)
2. **永远 AEAD,别再 CBC + HMAC**——历史踩坑太多
3. **别自己写**——用 OpenSSL / cryptography / ring,这些库后面有几十个密码学博士
