# 链路层

「我抓 ARP 包看一上午都没明白它到底干啥」——链路层最反直觉的地方在于:**TCP/IP 把它叫「最底下一层」,但实际上它有自己的一整套地址、自己的协议(ARP / STP / LLDP)、自己的安全模型(VLAN / 802.1X)**——基本上是 IP 层之下「另一个微型互联网」。**网络工程师 80% 的工作其实在这一层**:接交换机、配 VLAN、查 ARP 表、抓 WiFi 漫游问题。后端工程师不需要那么深,但**「ARP 是什么、VLAN tag 在哪、WiFi 为什么慢」这三件事必须懂**——不然容器 / K8s / 跨机房这些场景的诡异 bug 永远没法定位。

> 一句话先记住:**链路层 = 「IP 包怎么从我的网卡到下一跳网卡」的局部协议**。**ARP 是网络层之下的「翻译」**(IP → MAC),**VLAN tag 是塞在以太帧 SrcMAC 后面的 4 字节**(逻辑隔离一根物理线),**WiFi 比以太慢 = 半双工 + 无线介质重传率 1-10%**(以太是 0.0001%)。**这一层出问题往往「莫名其妙网络抖、邻居 ping 不通、跨 VLAN 隔离失效」,但 IP 层完全无症状**——只能下到这层抓帧才能看清。

## 一、回到以太帧:第 02 章没讲透的部分

第 02 章讲了以太帧 14 字节头 + 4 字节 FCS,这里把每个字段的"为什么"说透。

### 1.1 IEEE 802.3 vs Ethernet II:你抓到的是哪种

**两种共存的格式**(都还在用,但 99% 是后者):

```
Ethernet II(DIX,实际占 99%):
+------------+------------+------+----------+-----+
| Dest MAC   | Src MAC    | Type | Payload  | FCS |
| 6 bytes    | 6 bytes    | 2 B  | 46-1500B | 4 B |
+------------+------------+------+----------+-----+
                          ↑
                          0x0800 = IPv4
                          0x86DD = IPv6
                          0x0806 = ARP
                          0x8100 = 802.1Q VLAN
                          0x88CC = LLDP


IEEE 802.3 + LLC/SNAP(老古董,主要用在 STP / Cisco 协议):
+------------+------------+--------+-----+-----+----------+-----+
| Dest MAC   | Src MAC    | Length | DSAP|SSAP | Payload  | FCS |
| 6 bytes    | 6 bytes    | 2 B    | LLC header  | 38-1492B| 4 B |
+------------+------------+--------+-----+-----+----------+-----+
```

**怎么区分**:看那 2 字节:

```
> 0x0600  → Ethernet II(是 EtherType)
≤ 0x05DC  → IEEE 802.3(是 Length,最大 1500 = 0x05DC)
```

> 经验法则:**写代码 / 抓包 99% 见到的是 Ethernet II**——IEEE 802.3 只在交换机控制面协议里偶尔出现。

### 1.2 MAC 地址结构

```
6 字节 = 48 bit
+--------+--------+--------+--------+--------+--------+
|  AA    |  BB    |  CC    |  DD    |  EE    |  FF    |
+--------+--------+--------+--------+--------+--------+
↑                          ↑
OUI(组织 24 bit,IEEE 分配)NIC(网卡序列号 24 bit)
```

**第一字节最低两 bit 有特殊含义**:

```
最低位 = 1 → 多播地址(组播 / 广播)
最低位 = 0 → 单播地址

倒数第二位 = 0 → 全球唯一(IEEE 分配的 OUI)
倒数第二位 = 1 → 本地管理(虚拟机 / 容器自分配)
```

**所以 KVM / Docker 给 VM 分配的 MAC 都长成 `52:54:00:...` 或 `02:42:...`**——第二字节 `0x54` 或 `0x42`,二进制最后两位是 `00` / `10`,标记「本地管理」。

```
广播 MAC:ff:ff:ff:ff:ff:ff
IPv4 多播:01:00:5e:XX:XX:XX
IPv6 多播:33:33:XX:XX:XX:XX
```

### 1.3 FCS:4 字节 CRC32

**网卡硬件计算和校验**——CPU 不参与,完全 offload。**校验失败的帧,网卡直接丢,内核都看不到**。

> 经验法则:**抓包看不到 FCS 字段**——它在硬件层被验过就剥掉了。`tcpdump` 输出没有 FCS,Wireshark 也没有(除非你用特殊网卡 + 关 offload)。

## 二、ARP 协议:网络层之下的「翻译」

### 2.1 为什么要 ARP

**核心问题**:

```
应用要发到 192.168.1.50
  ↓
TCP/IP 层封装成 IP 包,DstIP = 192.168.1.50
  ↓
要交给链路层发出去 → 链路层只认 MAC 地址,不认 IP!
  ↓
我得先知道:192.168.1.50 对应的 MAC 是多少?
  ↓
ARP(Address Resolution Protocol)登场
```

**ARP = 把 IP 翻译成同一局域网内的 MAC 地址**。**只在同一个 L2 广播域内有效**——跨路由器就不能用了(那是路由的活)。

### 2.2 ARP 报文格式

ARP 不是 IP 之上的协议,而是**直接跑在以太帧里**(EtherType = 0x0806):

```
+----------------+----------------+------+----------------+
| Dest MAC       | Src MAC        | Type | ARP payload    |
| ff:ff:ff:ff:ff | aa:bb:cc:...   | 0x0806| (28 bytes)    |
+----------------+----------------+------+----------------+
                                          │
                                          ↓
ARP payload(28 bytes):
+----------+----------+--------+--------+--------+
| HW Type  | Proto Tp |HW Len  |PR Len  | OpCode |
| 0x0001(以太)| 0x0800(IP) | 6  | 4   | 1=req/2=resp|
+----------+----------+--------+--------+--------+
| Sender MAC (6)        | Sender IP (4)         |
+--------------------------------+----------------+
| Target MAC (6)        | Target IP (4)         |
+--------------------------------+----------------+
```

### 2.3 ARP 请求/响应过程(经典图)

**场景**:主机 A(192.168.1.10,MAC: aa)第一次 ping 主机 B(192.168.1.20,MAC: bb)。

```
Step 1: A 查本地 ARP cache,没有 B 的记录

Step 2: A 发 ARP 广播请求
  Eth: Src=aa, Dst=ff:ff:ff:ff:ff:ff(广播)
  ARP: "谁是 192.168.1.20?请告诉 192.168.1.10"
       Sender = (192.168.1.10, aa)
       Target = (192.168.1.20, 00:00:00:00:00:00)

Step 3: 局域网内所有主机都收到这个广播
  - 不是 192.168.1.20 的:看一眼丢掉
  - B 收到:看到目标是自己的 IP

Step 4: B 单播回响应
  Eth: Src=bb, Dst=aa
  ARP: "192.168.1.20 的 MAC 是 bb"
       Sender = (192.168.1.20, bb)
       Target = (192.168.1.10, aa)

Step 5: A 收到响应,写入 ARP cache
  192.168.1.20 → bb (TTL = 60s 起步,不同 OS 不同)

Step 6: A 现在能发 IP 包给 B 了
```

### 2.4 看自己的 ARP 表

```bash
# Linux
ip neigh show
# 192.168.1.1 dev eth0 lladdr 00:11:22:33:44:55 REACHABLE
# 192.168.1.20 dev eth0 lladdr aa:bb:cc:dd:ee:ff STALE

# 状态:
#   REACHABLE  最近用过,可信
#   STALE      老了,下次用前要重新探测
#   FAILED     探测失败
#   INCOMPLETE 探测中

# 老命令(还能用)
arp -a
```

### 2.5 ARP 缓存中毒(攻击)

**ARP 没有任何认证机制**——谁声称自己是某 IP,大家就信。**这是 LAN 内最古老的攻击**。

```
攻击者 M(MAC: mm)持续广播:
  "192.168.1.1 的 MAC 是 mm"  ← 假装自己是网关
  
受害者 V 收到,刷新 ARP cache:
  192.168.1.1 → mm

V 之后所有发往网关的流量,都先经过 M
  → M 抓密码、改请求、丢包、重定向
```

**防御**:

```bash
# 1. 静态 ARP(关键服务器对网关的映射)
sudo ip neigh add 192.168.1.1 lladdr 00:11:22:33:44:55 dev eth0 nud permanent

# 2. arpwatch 监控异常 ARP 变化

# 3. 交换机端启用 DAI(Dynamic ARP Inspection)
#    依赖 DHCP Snooping,验证 ARP 包合法性

# 4. 802.1X 端口认证
#    没认证的设备根本接不上网
```

### 2.6 Gratuitous ARP(无故 ARP)

**主机主动广播:「我是 192.168.1.10,我的 MAC 是 aa」**——没人问也发。用途:

1. **检测 IP 冲突**:开机时发,如果有人回应说「我也是 192.168.1.10」就报错
2. **VRRP / keepalived 主备切换**:备机接管 VIP 后立刻广播 GARP,让全网刷新 ARP cache,流量秒切到新主机
3. **虚拟机迁移**:VM 从一个宿主机迁到另一个,新宿主机发 GARP,让交换机更新 MAC 表

> 经验法则:**主备切换抖动 = GARP 没发或被丢**——抓 ARP 包看下,9 成是这。`tcpdump -i eth0 arp`。

## 三、VLAN(802.1Q):一根线分八千个网

### 3.1 为什么有 VLAN

```
没有 VLAN 之前:
  办公网 / 财务网 / 服务器网 → 各拉一根物理线 + 各买一台交换机
  → 贵、乱、扩展难

有了 VLAN:
  一根物理线,逻辑上分 4096 个 VLAN
  财务的帧打 VLAN 10 标签,办公的帧打 VLAN 20 标签
  交换机按 VLAN tag 隔离,跨 VLAN 必须走路由器
```

**广播域隔离 + 安全隔离 + 不动物理线就能重组逻辑网**——VLAN 是数据中心的命脉。

### 3.2 802.1Q tag 在帧的哪个位置

**这个常考**——tag 不是塞在 payload 里,是**插在 SrcMAC 和 EtherType 之间的 4 字节**:

```
普通以太帧:
+----------+----------+--------+----------+
| Dst MAC  | Src MAC  | EtType | Payload  |
| 6 B      | 6 B      | 2 B    | ...      |
+----------+----------+--------+----------+

802.1Q VLAN 帧:
+----------+----------+--------+--------+--------+----------+
| Dst MAC  | Src MAC  | TPID   | TCI    | EtType | Payload  |
| 6 B      | 6 B      | 0x8100 | 2 B    | 2 B    | ...      |
+----------+----------+--------+--------+--------+----------+
                       ↑插在这里 4 字节
                       
TPID = 0x8100 表示「这是 VLAN 帧」
TCI = 16 bit:
  3 bit  PCP (Priority,QoS 用)
  1 bit  DEI (Drop Eligible)
  12 bit VID (VLAN ID, 0-4095,实际可用 1-4094)
```

**最大长度变化**:**普通帧最大 1518,VLAN 帧最大 1522 字节**——所以**带 VLAN tag 的网卡需要 MTU 设 1500 + 4 = 1504**(被称为 baby giant frame)。**老网卡不支持会丢包**。

### 3.3 Trunk 口 vs Access 口

**交换机端口两种模式**:

```
Access 口:
  连接终端设备(电脑、服务器)
  入口:不带 tag 的帧 → 加上配置的 VLAN tag,发给交换机内部
  出口:剥掉 tag → 终端看到的是普通以太帧
  → 终端不知道 VLAN 存在

Trunk 口:
  连接交换机 / 路由器 / Hypervisor
  入口/出口:都带 tag
  → 一根线承载多个 VLAN 流量
```

**典型部署**:

```
PC1(VLAN 10)→ 交换机Access口10 → Trunk口 → 路由器 → ...
PC2(VLAN 20)→ 交换机Access口20 → Trunk口 ↗
PC3(VLAN 10)→ 交换机Access口10 → Trunk口 ↗
```

### 3.4 看 VLAN 配置

```bash
# Linux 上加 VLAN 子接口(常用于 Hypervisor / 容器宿主机)
sudo ip link add link eth0 name eth0.10 type vlan id 10
sudo ip addr add 192.168.10.1/24 dev eth0.10
sudo ip link set eth0.10 up

# 看
ip -d link show eth0.10
# vlan protocol 802.1Q id 10 ...

# 抓 VLAN 帧
sudo tcpdump -i eth0 -e vlan
# 02:01:23 vlan 10, p 0, ...
```

> 踩坑提醒:**容器 / VM 做透传 VLAN 时,宿主机网卡 MTU 不调成 1504,大包丢失**——表现是 ping 通、SSH 卡——经典 MTU 黑洞。

### 3.5 QinQ(802.1ad):VLAN 的 VLAN

```
+-----+-----+--------+-----+--------+-----+--------+----------+
| Dst | Src | TPID   | TCI | TPID   | TCI | EtType | Payload  |
|     |     | 0x88a8 |     | 0x8100 |     |        |          |
+-----+-----+--------+-----+--------+-----+--------+----------+
              外层 SP-VLAN          内层 C-VLAN
```

**两层 tag**——**运营商城域网**用得多:运营商加外层 tag 区分客户,客户内部用内层 tag 自由分 VLAN。**单机房上限从 4094 突破到 4094×4094 = 1600w**。后端工程师基本不碰。

## 四、WiFi(802.11):跟以太完全不同的世界

### 4.1 根本差异:介质特性

| 维度 | 以太网 | WiFi |
| --- | --- | --- |
| 介质 | 铜缆 / 光纤 | 无线电波 |
| 双工 | 全双工(独立收发) | **半双工**(同时只能一个方向) |
| 冲突检测 | CSMA/CD(可检测) | **CSMA/CA**(只能避让) |
| 错误率 | ~10⁻⁹(可忽略) | **10⁻² ~ 10⁻⁵**(经常错) |
| 重传 | 极少(基本没有) | **链路层主动重传** |
| 安全 | 物理线即认证 | 必须加密(WPA2/WPA3) |

**为什么 WiFi 慢于以太**:

1. **半双工**:同一时刻同一信道只能一方发,**理论上限砍半**
2. **重传**:每包都要 ACK,丢了重发(时间窗口大)
3. **共享信道**:同一 AP 下所有设备抢空气,**1 个设备和 10 个设备能差 5 倍**
4. **干扰**:微波炉 / 蓝牙 / 邻居 AP 都让你重传
5. **协议头开销大**:802.11 头 30+ 字节,远超以太 14 字节

### 4.2 802.11 帧结构(简化)

```
+---------+----------+----------+----------+----------+----------+----------+
| Frame   | Duration | Address1 | Address2 | Address3 | Sequence | Address4 |
| Control |          | (DA/RA)  | (SA/TA)  | (BSSID)  | Control  | (option) |
| 2 B     | 2 B      | 6 B      | 6 B      | 6 B      | 2 B      | 6 B      |
+---------+----------+----------+----------+----------+----------+----------+
| Payload (0-2312 bytes) | FCS (4 B) |
+---------+----------+
```

**3 个 MAC 地址(必要时 4 个)** —— 因为 WiFi 帧要从 STA → AP → 网关,每一跳都要标识:**RA(接收方)、TA(发送方)、BSSID(AP 标识)、DA(最终目标)、SA(最初源)**。

### 4.3 接入流程(超经典)

```
STA(笔电)              AP(路由器)
   │
   │ Probe Request "有谁在?"        →
   │ ←  Probe Response "我是 ssid X"
   │
   │ Auth Request                   →
   │ ←  Auth Response (Open/WEP)
   │
   │ Association Request            →
   │ ←  Association Response (你 AID = 5)
   │
   │ EAPOL 4-way handshake (WPA2/3) ↔  (协商加密密钥 PTK)
   │
   │ DHCP Discover (encrypted 之后) →
   │ ←  DHCP Offer
   │
   │ 现在可以发数据帧了
```

**WiFi 接入慢的就是这一串**——好的 AP 100ms,差的 800ms。**漫游(roaming)如果不支持 802.11r,每次切 AP 都要重做 EAPOL 4-way → 卡 1 秒**——这就是「家里 WiFi 走两步就卡」的原因。

### 4.4 WiFi 抓包(macOS 特别简单)

```bash
# macOS:option-click WiFi 图标 → Open Wireless Diagnostics
# 菜单 Window → Sniffer → 选信道开抓

# Linux:网卡进 monitor mode
sudo iw dev wlan0 set type monitor
sudo ip link set wlan0 up
sudo tcpdump -i wlan0 -w wifi.pcap
```

**Wireshark 看 802.11 帧**——能看到所有信号、握手、重传——**WiFi 排障必备**。

### 4.5 速率与代际

| 标准 | 名字 | 频段 | 理论速率 | 上市 |
| --- | --- | --- | --- | --- |
| 802.11g | (无) | 2.4 GHz | 54 Mbps | 2003 |
| 802.11n | WiFi 4 | 2.4/5 GHz | 600 Mbps | 2009 |
| 802.11ac | WiFi 5 | 5 GHz | 6.9 Gbps | 2013 |
| 802.11ax | WiFi 6 | 2.4/5 GHz | 9.6 Gbps | 2019 |
| 802.11be | WiFi 7 | 2.4/5/6 GHz | 46 Gbps | 2024 |

**实际速率 ≈ 理论 × 0.4-0.6**——干扰 / 距离 / 信道宽度都吃。WiFi 6 写 1Gbps 实测 500Mbps 算正常。

> 经验法则:**有线能用就别用无线**——服务器 / 工位 / 摄像头一律有线。WiFi 给手机 / 临时设备。**P99 抖动一半是 WiFi 干扰**,生产环境就别上无线。

## 五、PPP / PPPoE:家宽用了 30 年的拨号协议

### 5.1 为什么有 PPP

**PPP**(Point-to-Point Protocol)—— 点对点链路上的协议,典型场景:

- 早年的 modem 拨号上网
- ADSL / 光纤宽带(PPPoE = PPP over Ethernet)
- 串口直连两台设备(比如调试网络设备 console 口)
- VPN(PPTP / L2TP 都基于 PPP)

**和以太的根本区别**:**PPP 是点对点,没有 MAC 地址,没有广播**——直接两点之间封装。

### 5.2 PPP 帧格式

```
+-------+---------+---------+----------+----------+-----+
| Flag  | Address | Control | Protocol | Payload  | FCS |
| 0x7E  | 0xFF    | 0x03    | 2 B      | 0-1500   | 2/4B|
+-------+---------+---------+----------+----------+-----+
```

**Address 永远是 0xFF**——因为只有一个对端,不需要地址。**Protocol** 字段表示 payload 是 IP / IPCP(IP 控制协议)/ LCP(链路控制协议)等。

### 5.3 PPPoE:PPP 跑在以太上

```
家宽场景:
  PC →(以太帧)→ 光猫 →(PPPoE 拨号)→ 运营商 BRAS → 公网

为什么不直接给 PC 发公网 IP?
  - 需要认证(账号密码)
  - 需要计费(按时长 / 流量)
  - PPP 是为这些场景设计的成熟协议
```

PPPoE 帧格式 = 以太帧 + PPPoE 头 + PPP 帧:

```
+----------+----------+--------+----------+-------+----------+----------+
| Dst MAC  | Src MAC  | EtType | PPPoE    | PPP   | Payload  | FCS(eth)|
| 6 B      | 6 B      | 0x8864 | header   | header| (1492 B) | 4 B      |
+----------+----------+--------+----------+-------+----------+----------+
                       ↑
                       PPPoE Discovery  = 0x8863
                       PPPoE Session    = 0x8864
```

**PPPoE 头 6 字节 + PPP 头 2 字节 = 8 字节开销**——所以**PPPoE 链路 MTU = 1500 - 8 = 1492**。

> 踩坑提醒:**家宽 PPPoE 客户端没设 MTU=1492,跨网站偶发卡死**——PMTUD 失败的话现象就是「能 ping 通、能打开小页面、大网站(谷歌图片)卡死」。所有 PPPoE 客户端都内置 1492 默认。

## 七、链路层排障小工具

```bash
# 看链路状态(线插了没 / 速率多少 / 全双工?)
sudo ethtool eth0
# Speed: 1000Mb/s
# Duplex: Full
# Link detected: yes

# 看 MAC 地址 / MTU
ip link show eth0

# 看 ARP 表
ip neigh

# 抓 ARP 包
sudo tcpdump -i eth0 -nn arp

# 抓 STP / LLDP
sudo tcpdump -i eth0 -nn ether proto 0x88cc       # LLDP
sudo tcpdump -i eth0 -nn ether dst 01:80:c2:00:00:00  # STP

# 测物理链路
sudo ethtool -t eth0           # 自检
sudo mii-tool eth0             # 老命令,看链接速率

# 看 VLAN
ip -d link show
bridge vlan show               # 网桥的 VLAN
```

## 九、踩坑提醒

1. **以为 ARP 走 IP**——ARP 直接在以太帧里(EtherType=0x0806)
2. **跨子网用 ARP 找对端**——ARP 不跨路由器,跨子网找的是网关 MAC
3. **VLAN tag 的 MTU 没调到 1504**——VLAN 大包神秘丢失
4. **VLAN 1 当默认管理 VLAN**——所有交换机默认 VLAN 1,改成别的提升安全
5. **WiFi 当生产网络**——干扰多、重传率高,P99 必抖
6. **WiFi 漫游不开 802.11r**——切 AP 卡 1 秒
7. **PPPoE 客户端 MTU 默认 1500**——大包过运营商被丢
8. **K8s / 容器跨主机网络丢 ARP 广播**——overlay 网络隔离了广播域,得用 ARP proxy / VXLAN
9. **MAC 表满了交换机退化成 hub**——所有帧广播,流量爆 / 安全爆
10. **DHCP / ARP 攻击防御没开**——内部网最常见的入侵入口
