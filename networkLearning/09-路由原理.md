# 路由原理

「路由器就是把包转过去」——是,但**「怎么知道要转哪儿」是网络协议里最复杂的问题之一**。**全互联网 80 多万条 BGP 路由,每条都关系着「这个 IP 段经过哪几个 AS 才能到」**——一旦某个大 AS 路由错配,**全球某网站可能瞬间离线**(2008 巴基斯坦电信劫持 YouTube、2021 Facebook 自挂自家 BGP 6 小时离线)。**路由协议的本质是"分布式系统"** —— 几十万台路由器没有中心,要协商出一个一致的「全网拓扑」,这事 RIP / OSPF / BGP 各有解法。**做后端、做云原生、做 SRE 必须懂这一层**——不然 K8s 跨节点不通、VPN 隧道路由错配、机房间路由抖动,你只能干瞪眼。

> 一句话先记住:**路由 = "对每个目标 IP,查表找下一跳"**。**路由表 = (目标网段 / 下一跳 IP / 出接口 / 度量值) 的集合**。**静态路由人工配,动态路由让协议自动学**——RIP(距离向量,跳数计数,**95% 已淘汰**)、OSPF(链路状态,Dijkstra 算最短,**域内绝对主流**)、BGP(路径向量,基于 AS Path,**全互联网的「骨架」**)。**任何一条全球新闻级别的"互联网中断"几乎都跟 BGP 有关**——这是工程师必须懂的"隐藏底层"。

## 一、路由表:网络的「地图」

### 1.1 路由表长什么样

```bash
$ ip route show
default via 192.168.1.1 dev en0 proto dhcp metric 100
10.0.0.0/8 via 192.168.1.1 dev en0
169.254.0.0/16 dev en0 scope link metric 1000
192.168.1.0/24 dev en0 proto kernel scope link src 192.168.1.10 metric 100
```

**逐行解读**:

```
default via 192.168.1.1 dev en0 metric 100
↑ 默认路由(0.0.0.0/0):任何不匹配其他路由的包都走这里
↑ 下一跳是 192.168.1.1(家里网关)
↑ 出接口 en0
↑ 度量值 100(数越小越优先)

192.168.1.0/24 dev en0 proto kernel scope link
↑ 直连子网,scope link 表示「同链路直接送,不需要下一跳」
↑ proto kernel 表示是内核根据本地 IP 自动加的

10.0.0.0/8 via 192.168.1.1 dev en0
↑ 通过网关 192.168.1.1 转发到 10/8(可能是公司 VPN 路由)
```

### 1.2 路由表的 4 个核心字段

| 字段 | 含义 | 例 |
| --- | --- | --- |
| **目标网段** (Destination) | 用 CIDR 表示(详见 10) | `10.0.0.0/8` |
| **下一跳** (Gateway / Next Hop) | 转发给谁 | `192.168.1.1` |
| **出接口** (Interface / Device) | 从哪个网卡发出去 | `en0` |
| **度量值** (Metric) | 多条同等路由时选哪条 | `100` |

### 1.3 最长前缀匹配(LPM)

**路由查找的核心算法**——同一目标可能匹配多条路由,**选最具体的那条**:

```
路由表:
   0.0.0.0/0       via 192.168.1.1
   10.0.0.0/8      via 192.168.1.1
   10.20.0.0/16    via 10.20.0.1
   10.20.30.0/24   via 10.20.30.254

查 10.20.30.5:
   匹配 0.0.0.0/0           ✓
   匹配 10.0.0.0/8          ✓
   匹配 10.20.0.0/16        ✓
   匹配 10.20.30.0/24       ✓
   选择 /24,因为前缀最长 → 走 10.20.30.254
```

**LPM 是路由器的核心数据结构问题**——硬件路由器用 **TCAM**(三态内容寻址存储器)做并行匹配,几纳秒搞定;软件路由用 **Trie / LC-Trie / DXR** 等优化算法。

> 经验法则:**「最长前缀匹配」是路由所有玄学的根源** —— 你写了 `10.0.0.0/8 via A`,如果别人加了 `10.20.0.0/16 via B`,**B 会赢**——你的流量被「黑洞」了还不知道。

## 二、静态路由 vs 动态路由

### 2.1 静态路由

**人工写死的路由**:

```bash
# Linux 加一条
sudo ip route add 10.0.0.0/8 via 192.168.1.254 dev en0

# 删掉
sudo ip route del 10.0.0.0/8

# 永久(写到 /etc/network/interfaces 或 NetworkManager)
```

**优点**:简单、可控、零开销
**缺点**:网络变化时要手动改;路径挂了不会自动切

**适用场景**:小网络、家庭、点对点 VPN、明确知道下一跳的边缘场景。

### 2.2 动态路由

**让路由协议自动发现 + 协商**:

```
路由器 A 跟邻居 B/C 交换:
  「我能到哪些网段、距离/质量多少」

A 维护一张完整路由表
当 B 挂了或 B↔D 链路断了
  → 协议感知到
  → 自动重算
  → 表里把走 B 的路由删 / 改下一跳
```

**优点**:自适应、抗故障、可扩展
**缺点**:协议复杂、有收敛时间(从故障到稳定)

| 网络规模 | 推荐 |
| --- | --- |
| 1-3 台路由器 | 静态 |
| 一个企业 / 一个 IDC | OSPF |
| 多个 AS / 跨运营商 | BGP |
| 传统电信 | IS-IS(类似 OSPF,运营商爱用) |

## 三、RIP:最古老的距离向量协议

### 3.1 RIP 怎么工作

**RIP(Routing Information Protocol,RFC 1058)** 是最早的动态路由协议,1988 年:

```
1. 每台路由器把自己的路由表周期性广播给邻居(默认 30 秒)
2. 邻居收到后,把所有路由距离 +1(即「我经你 1 跳能到」)
3. 选距离最短的那条
4. 如果 30s × 6 = 180s 没收到某邻居更新,认为它挂了

距离衡量:跳数(hop count)
最大跳数:15(16 = 不可达,防环)
```

### 3.2 距离向量算法

```
A 知道:
  网段 X → 直连
  网段 Y → 经 B,2 跳
  网段 Z → 经 C,3 跳

A 把这个表广播给邻居 B/C
B 收到 → 「我经 A 到 X 是 2 跳,到 Y 是 3 跳,到 Z 是 4 跳」
B 跟自己原表对比,留更短的
```

### 3.3 RIP 的致命问题

**1. 收敛慢得令人发指**

```
B 跟 X 直连,B 告诉所有邻居「我到 X 是 1 跳」
B 突然挂了

A、C、D... 怎么知道 X 不可达了?
  → 等 30s 收不到 B 的广播
  → 等 6 次广播超时(180s)
  → 才认为 B 挂了
  → 才把「经 B 到 X」从表里删
```

**网络收敛时间往往以分钟计**——现代任何业务都受不了。

**2. 计数到无穷大(Count to Infinity)**

```
A — B — X(直连)
A 知道 X 经 B 2 跳

B 突然挂了
B 把「X 不可达」消息发出去之前,A 又广播「我到 X 是 2 跳」
B 收到 → 「啊,我可以经 A 到 X,3 跳」(其实 A 是经 B 学到的)
B 又广播 → A 收到 → 「我经 B 到 X 是 4 跳」(本来 2 跳,变 4 跳)
B 又广播 → A → 5 跳
...
直到跳数 = 16,认为不可达
但这要十几个广播周期,即几分钟
```

**修补方案**:水平分割、毒性逆转、抑制定时器——但都治标不治本。**16 跳上限直接限制了 RIP 只能用在小网络**。

**3. 不区分链路质量**

```
RIP 只看跳数,不看带宽 / 延迟
  100 Gbps 走 3 跳   vs   1 Mbps 走 1 跳
  RIP:1 跳赢,流量全走 1 Mbps,瞬间打瘫
```

### 3.4 RIP 现状:基本退役

```
RIPv1(1988)  广播,有类(无 CIDR 支持)
RIPv2(1993)  组播,支持 CIDR + 认证
RIPng(1997)  IPv6 版本

2026 年的现实:
  - 大学教材里还在讲(讲它的设计思想)
  - 实际生产几乎没人用
  - 偶尔在小型 SOHO 路由器上看到默认开
```

> 经验法则:**生产不要用 RIP**——除非你在维护 1990 年代的遗留网络。

## 四、OSPF:域内的绝对主流

### 4.1 OSPF 怎么工作

**OSPF(Open Shortest Path First,RFC 2328)** 是「链路状态」协议:

```
1. 每台路由器跟邻居建立邻接关系(发 Hello 包,默认 10s)
2. 每台路由器知道自己有哪些链路(直连接口 + cost)
3. 把「我有哪些链路 + 各 cost」打包成 LSA(Link State Advertisement)
4. 通过组播泛洪给整个 OSPF 区域内所有路由器
5. 每台路由器收齐所有 LSA,组成「全网拓扑图」
6. 每台路由器对自己跑 Dijkstra 算最短路径(SPF)
7. 算完后,生成路由表
```

### 4.2 OSPF vs RIP 的关键区别

| 维度 | RIP | OSPF |
| --- | --- | --- |
| 算法 | 距离向量(Bellman-Ford 思想) | 链路状态(Dijkstra) |
| 路由器知道什么 | 只知道「邻居告诉我的距离」 | **知道全网拓扑** |
| 度量 | 跳数 | cost(默认基于带宽倒数) |
| 收敛 | 几十秒到几分钟 | **秒级**(SPF 重算很快) |
| 规模 | < 15 跳 | 大型企业网,几百路由器 |
| 复杂度 | 简单 | 复杂(LSDB / 区域 / DR/BDR) |

### 4.3 OSPF 的「区域」(Area)

```
大网络如果一个 OSPF 域跑全网,LSA 泛洪量 + Dijkstra 计算量爆炸
→ 分区域

Backbone Area (Area 0)
   ↑↓
Area 1 — Area 2 — Area 3
```

**所有非 0 区域必须跟 Area 0 直连(或通过 Virtual Link)**。区域内详细 LSA,区域间只交换汇总路由。

### 4.4 OSPF 的 cost

```
默认 cost = 100,000,000 / 接口带宽(bps)

10 Mbps   cost = 10
100 Mbps  cost = 1
1 Gbps    cost = 1   ← 跟 100M 一样!需要手工调高 reference-bandwidth
10 Gbps   cost = 1   ← 同上
```

**默认值没跟上时代**——千兆+ 的链路 cost 都是 1,**不能正确反映链路差异**。生产里要手动改:

```
Cisco IOS:
  router ospf 1
    auto-cost reference-bandwidth 1000000   # 单位 Mbps,设成 1Tbps
```

### 4.5 谁用 OSPF

```
绝大部分企业内网
大部分 ISP 域内(也有用 IS-IS 的)
数据中心传统网络(Underlay 三层互联)

K8s 的 BGP-based 网络方案(Calico / Cilium)用 BGP,不用 OSPF
现代 Spine-Leaf 数据中心倾向 BGP(eBGP unnumbered)
```

> 经验法则:**OSPF 是「单一管理域内」的最优选择**——一个公司、一个 IDC、一个 ISP 内部。**跨 AS / 跨公司必须 BGP**。

## 五、BGP:互联网的「骨架」

### 5.1 BGP 是什么

**BGP(Border Gateway Protocol,RFC 4271)** 是「路径向量」协议,跑在「自治系统(AS)」之间:

```
什么是 AS?
  每个大网络运营者(ISP / 大公司)有一个 AS 号
  Google = AS15169
  Cloudflare = AS13335
  China Telecom = AS4134
  AWS = AS16509
  全球约 10 万个活跃 AS
```

**BGP 的工作**:

```
1. AS 跟邻居 AS 建立 BGP 连接(TCP 179 端口)
2. 互相通告:「我能到哪些网段(prefix),经过哪些 AS」
3. 接收方按策略选路:
     - AS Path 短的优先(跳数)
     - 本地优先级
     - MED(给邻居建议)
     - 自定义策略(社区属性)
4. 选出来的路由放到自己路由表 + 继续传给其他邻居
```

### 5.2 一条 BGP 路由长什么样

```bash
# 在公网 looking glass 看一条:
show bgp 8.8.8.0/24

prefix:    8.8.8.0/24
nexthop:   X.X.X.X(对接路由器 IP)
AS Path:   174 15169
           ↑    ↑
        Cogent  Google
        (我从 Cogent 学到,Cogent 从 Google 学到)
local pref: 100
MED:       0
community: 174:21000 174:22013 ...
```

**AS Path 是 BGP 的核心**——它防环(看到自己 AS 号在 Path 里就拒绝)+ 选路(短的优先,但实际上 local-pref 优先级更高)。

### 5.3 iBGP vs eBGP

```
eBGP (External BGP)
  跨 AS 之间的 BGP
  TTL = 1(默认,只允许直连邻居)
  AS Path 包含本 AS

iBGP (Internal BGP)
  同一 AS 内部多个边界路由器之间
  防环靠「不传 iBGP 学到的路由给其他 iBGP 邻居」
  → 需要全网状(full mesh)或 Route Reflector
```

### 5.4 BGP 选路 13 步规则

BGP 不像 OSPF 选「最短 cost」,而是有一长串决策步骤:

```
1.  权重(Cisco 私有,本路由器最优)
2.  Local Preference(AS 内部统一)
3.  本地起源(network 命令配的)
4.  AS Path 最短
5.  Origin 类型(IGP < EGP < Incomplete)
6.  MED 最低(给邻居 AS 的"建议")
7.  eBGP 优于 iBGP
8.  IGP cost 到 Next Hop 最近
9.  oldest path
10. Router ID 最低
11. Cluster ID 最低
12. 邻居 IP 最低
... (现代实现会有更多)
```

**绝大部分流量决策在 1-4 步就结束**——前几条是策略,后几条是「实在挑不出来时的破平局规则」。

### 5.5 BGP 为什么是「骨架」

```
全球互联网 = 几十万个 AS 互联
每个 AS 都用 BGP 跟邻居 AS 互通路由

完整的 BGP 路由表(全表)≈ 90 多万条 IPv4 + 20 多万条 IPv6
路由器内存要装得下(老路由器装不下,会出问题)

任何一个 prefix(比如 8.8.8.0/24),
  → BGP 决定它"经过哪几个 AS"
  → 也就是「哪几家电信 / Google / 中转 ISP」
```

**所以**:**任何"互联网级"的中断、绕路、劫持,都跟 BGP 有关**。

## 六、BGP 劫持:从巴基斯坦到 Facebook

### 6.1 2008 年:巴基斯坦电信「黑」掉 YouTube

```
2008 年 2 月 24 日
巴基斯坦政府要求屏蔽 YouTube
PT(巴基斯坦电信)的工程师做了一条静态路由,
  把 YouTube 的 208.65.153.0/24 指向自己内部 null 接口
  → 国内访问 YouTube 被丢

但是!
工程师不小心把这条 /24 路由也通过 BGP 通告出去了
PT 的上游 PCCW 没做任何过滤,直接转发
  → 全球 BGP 都学到了「YouTube 经 PT 最近(因为 /24 比 YouTube 自己宣告的 /22 更具体)」
  → 全球 YouTube 流量都涌向巴基斯坦 PT
  → 巴基斯坦 PT 网络被打瘫
  → YouTube 全球离线 ~2 小时
```

**根本原因**:**BGP 默认信任邻居** + **/24 比 /22 更长前缀更优先(LPM)**。

### 6.2 2021 年:Facebook 自挂

```
2021 年 10 月 4 日
Facebook 内部一次「常规维护」
工程师 SSH 上路由器跑了个命令,意外误删了所有 Facebook AS32934 的 BGP 路由通告
  → 全球 BGP 路由表里 Facebook 的 prefix 消失
  → 全球访问 facebook.com / instagram / whatsapp 全断

更尴尬:Facebook 自己的 DNS 服务器也在这些被撤销的 prefix 里
  → 即使用其他公司 DNS 也解析不到 Facebook 服务器
  → 工程师 VPN 不上公司,门禁也连的内部网络,刷不开门进数据中心

恢复用了 6 小时
全球新闻
股价跌 4.9%
```

**教训**:**BGP 是高度信任 + 高风险的协议** —— 一个命令可以让全球某网站消失。

### 6.3 2024 年:Cloudflare BGP 劫持(假设例)

类似事件每年都有几起,有些是误操作,有些是恶意。**RPKI(Resource Public Key Infrastructure)** 是行业的应对:

```
RPKI:每个 AS 有数字证书,声明「我合法持有 X.X.X.X/Y prefix」
其他 AS 验证 BGP 公告是否被 RPKI 证书覆盖
没覆盖 / 不匹配的 → 认为是 invalid,丢弃

2026 年现状:
  大部分 Tier-1 ISP 已经做 RPKI 过滤
  但仍有相当比例的中小 ISP 未实施
  → 局部劫持仍可能,全球范围已大大降低
```

> 经验法则:**做 BGP 工程师,改一条命令前一定 review 三遍 + 准备回滚** —— 没有任何协议能比 BGP 更容易"一键断网"。

## 七、Linux 路由实战

### 7.1 看路由表

```bash
# 主路由表
ip route show
ip route show table main

# 所有路由表(local / main / default)
ip route show table all

# 查某 IP 走哪条路由
ip route get 8.8.8.8
8.8.8.8 via 192.168.1.1 dev en0 src 192.168.1.10 uid 1000
    cache mtu 1500
```

### 7.2 加 / 删路由

```bash
# 加默认路由
sudo ip route add default via 192.168.1.1 dev en0

# 加特定网段
sudo ip route add 10.0.0.0/8 via 192.168.1.254 dev en0

# 用 metric 区分优先级
sudo ip route add 10.0.0.0/8 via 192.168.1.254 metric 100
sudo ip route add 10.0.0.0/8 via 192.168.1.253 metric 200   # 备用

# 黑洞(丢弃匹配的包)
sudo ip route add blackhole 1.1.1.0/24

# 删
sudo ip route del 10.0.0.0/8
```

### 7.3 多张路由表 + 策略路由

Linux 支持多路由表 + 基于源 IP / mark / fwmark 的策略路由:

```bash
# 看所有规则
ip rule show
0:      from all lookup local
32766:  from all lookup main
32767:  from all lookup default

# 给 192.168.5.0/24 来的包查路由表 100
sudo ip rule add from 192.168.5.0/24 table 100

# 在表 100 里加路由
sudo ip route add default via 192.168.5.1 table 100
```

**用途**:多 ISP 出口、VPN 分流、容器网络隔离(Calico / Cilium 大量用)。

### 7.4 看默认路由是怎么来的

```bash
ip route show default
default via 192.168.1.1 dev en0 proto dhcp metric 100
                                ↑
                              告诉你来源
                              dhcp = 从 DHCP 服务器学到
                              static = 手动配
                              ra = SLAAC RA 学到(IPv6)
                              kernel = 内核自动加(直连)
                              bgp / ospf = 从动态路由协议学到
```

## 八、Linux 跑 BGP / OSPF:FRR / BIRD

### 8.1 FRR / BIRD 是什么

```
FRR (Free Range Routing,从 Quagga 分叉):
  完整的开源路由套件
  支持 OSPF / OSPFv3 / IS-IS / BGP / RIP / PIM 等

BIRD:
  另一个开源路由守护进程
  Cloudflare / Hetzner 等大量用
```

### 8.2 在 Linux 上起一个 BGP

```
# 安装 FRR
sudo apt install frr

# 启用 bgpd(在 /etc/frr/daemons 里 bgpd=yes)
sudo systemctl restart frr

# 进 vtysh 配置
sudo vtysh
configure terminal
router bgp 65001
  neighbor 10.0.0.2 remote-as 65002
  neighbor 10.0.0.2 description peer-router
  network 192.168.100.0/24
exit

# 看 BGP 邻居
show ip bgp summary

# 看 BGP 路由表
show ip bgp
```

**这就是 K8s Calico 的底层**——每个节点跑一个 BGP daemon,跟 Top-of-Rack 路由器或者集群其他节点交换 Pod 网段路由。

## 九、路由抖动:生产里的「噩梦场景」

### 9.1 路由抖动是什么

```
某条 BGP / OSPF 路由频繁 up / down
路由器频繁重算 SPF
触发频繁的路由更新泛洪
全网 CPU 飙高
连接频繁断流(因为 NAT 表 / 防火墙状态丢失)
```

### 9.2 抑制(Dampening)

```
一条路由频繁抖动 → 降低权重
继续抖 → 进一步降权
权重低于阈值 → 暂时不用这条路由,等冷却

BGP 的 dampening 功能:抖一次 +1000 分,每分钟降 50%
超过 2000 分 → 进入「抑制」状态
低于 750 分 → 重新启用
```

### 9.3 真实案例:海缆抖动

```
跨太平洋海缆受洋流 / 渔船 / 地震影响,经常微抖
若直接传播 → 全球 BGP 频繁重算
→ ISP 普遍开 BGP dampening
→ 抖动一次"被罚 30 分钟"

代价:故障真的恢复后,你的流量也要等 30 分钟才回流
```

> 经验法则:**生产 BGP 必开 dampening + 路由聚合**——别让局部抖动放大成全网震荡。

## 十、ECMP:多路径并行

### 10.1 ECMP(Equal Cost Multi-Path)

```
同一个目标 prefix,有多条 cost 相同的路径
→ 路由器同时使用所有路径

负载均衡:按「五元组哈希」(源 IP + 目 IP + 协议 + 源端口 + 目端口)分配
→ 同一条 TCP 流走同一路径(避免乱序)
→ 不同 TCP 流分散到多路径
```

### 10.2 ECMP 在 Linux 上

```bash
# Linux 内核默认支持 ECMP
sudo ip route add 10.0.0.0/8 nexthop via 192.168.1.1 weight 1 \
                              nexthop via 192.168.1.2 weight 1 \
                              nexthop via 192.168.1.3 weight 1

# 改哈希算法(默认 L3)
sysctl net.ipv4.fib_multipath_hash_policy = 1   # L4(包含端口)
```

### 10.3 ECMP 的坑:traceroute 看不全

```
你 traceroute 8.8.8.8
中间有一跳是 ECMP,4 条等价路径
你的 3 个探测包(每跳)可能落在不同路径
→ 显示同一跳"3 个 IP 不同"
→ 看起来像「路由抖动」,其实是 ECMP 正常分流
```

**修复**:用 `paris-traceroute`,固定五元组让所有探测走同一路径。

## 十一、踩坑提醒

1. **以为路由表越长越好** —— 实际路由表过大(几十万条)直接打爆软件路由器内存
2. **改路由不看 metric** —— 误把备份路由 metric 设小,业务全切到备份
3. **静态路由没配 backup** —— 网关挂了网络全断
4. **iBGP 没用 RR(Route Reflector)** —— 大型 AS 内部要 N×N full mesh,N=100 时崩
5. **BGP 改命令不预演** —— 一键劫持自己的 prefix,典型 Facebook 故障
6. **不开 RPKI 验证** —— 全球随便哪个 AS 都能劫持你的 prefix
7. **OSPF 的 reference-bandwidth 用默认** —— 千兆 / 万兆链路 cost 都是 1,选不出来
8. **路由抖动不开 dampening** —— 全网震荡
9. **ECMP 哈希默认 L3** —— P2P / NAT 场景下流量集中到一条路径
10. **以为 default route 能解决一切** —— LPM 让更具体的路由优先,你以为走 default,实际被静态/ BGP 学到的更具体路由「劫持」
