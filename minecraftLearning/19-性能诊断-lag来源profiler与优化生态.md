# 性能诊断:lag 来源、profiler 与优化生态

前面十几篇里,"XX 是性能热点"这句话反复出现——漏斗、实体、光照、随机刻、区块加载。这一篇把它们收口:Minecraft 的卡到底卡在哪、怎么测出来、以及社区那套优化 mod 各自治的是哪一段病。

> 一句话先记住:**先分清你卡的是 TPS 还是 FPS,这是两台完全不同的机器在喘气;然后用 profiler 看火焰图找真正的热点,而不是凭直觉关东西。**

---

## 一、先把"卡"拆成两台机器

新手最容易犯的错:看到游戏卡,就一股脑去关材质、调画质。但服务端逻辑和客户端渲染是两套独立时钟(第 04 篇讲过),卡的根因可能在任意一边:

| 指标 | 衡量谁 | 正常值 | 卡了的表现 |
| --- | --- | --- | --- |
| **TPS** (ticks per second) | 服务端逻辑循环 | 20 | 怪物动得慢、作物长得慢、红石变慢、被击退"瞬移" |
| **MSPT** (毫秒/tick) | 单 tick 耗时 | < 50ms | 超过 50ms 就追不上 20 TPS,开始掉 |
| **FPS** (frames per second) | 客户端渲染 | 越高越好 | 画面卡顿、掉帧,但游戏逻辑可能完全正常 |

关键认知:

- **TPS 掉 ≠ FPS 掉**。你可能 FPS 200 但 TPS 只有 8(服务端被一堆实体拖死),也可能 TPS 满 20 但 FPS 30(显卡画不动)。两者各查各的。
- **MSPT 比 TPS 更有用**。TPS 封顶 20,你看不出"还剩多少余量"。MSPT 35ms 和 48ms 都显示 TPS 20,但后者已经命悬一线。盯 MSPT,不要等 TPS 掉了才反应。

> 反直觉:很多人"感觉服务器卡"时,真正掉的是 FPS;反过来,联机时"打怪像在水里挥拳"才是典型的 TPS 病。先确认是哪台机器,再动手。

## 二、TPS 掉了,通常是这几类在吃 tick

服务端每 tick 要把世界更新一遍(第 04 篇的执行顺序)。MSPT 超标,几乎都来自下面这几桶,按"最常见、最坑"排序:

| 热点 | 为什么贵 | 典型场景 |
| --- | --- | --- |
| **实体过多** | 每个实体每 tick 都要单独更新:AI 寻路(A\*)、AABB 碰撞、计时(第 06 篇) | 刷怪塔没关、海量掉落物堆地上、动物圈养过密 |
| **block entity** | 每 tick 遍历所有方块实体逐个 tick;漏斗每几 tick 还要扫容器内容(第 09 篇) | 大型物流系统、几百个漏斗 |
| **区块加载与生成** | 生成是分阶段重计算(第 16 篇),加载要解压+反序列化(第 07/08 篇) | 玩家高速移动、大量区块加载器、刚开荒的新地图 |
| **随机刻** | randomTickSpeed 被调高,或大片作物/树叶都在 random tick(第 04 篇) | 把 randomTickSpeed 拉到几千"催熟"的服务器 |
| **光照更新** | 放/挖方块触发光照洪水填充重算,原版光照引擎很拖(第 10 篇) | 大规模建造/爆破、频繁开关光源 |
| **复杂红石** | 邻居更新级联可能引发"更新风暴"(第 14 篇) | 大型红石机器、瞬时门、0-tick 农场 |

经验法则:**实体和 block entity 是 90% 服务器 lag 的来源**。掉落物尤其阴险——一个没装满的物流系统会让物品堆在地上,每个掉落物都是一个要 tick 的实体。

## 三、别猜,用 profiler 看火焰图

"我觉得是红石/我觉得是怪"——这种猜测十次有八次错。Minecraft 这类系统,瓶颈往往反直觉地集中在一两个你没想到的地方。正确姿势是采样。

### spark —— 现在的事实标准

[spark](https://spark.lucko.me/) 是采样式 profiler(mod / 插件形态都有),核心用法:

```
/spark profiler start      # 开始采样
... 让卡顿场景持续一会儿 ...
/spark profiler stop       # 出报告,生成一个网页链接
/spark tps                 # 快速看 TPS / MSPT 当前值
/spark health              # 看内存、GC、MSPT 分布
```

报告是一棵**火焰图(调用栈采样)**。怎么读:

- 横向越宽 = 这段调用占的 tick 时间越多。**先看最宽的那几条**,那才是热点。
- 顺着宽条往下钻,直到看见具体的类/方法名:看到 `HopperBlockEntity` 就是漏斗,看到 `ServerLevel.tickChunk` 往下是 random tick 或刷怪,看到 `LevelLightEngine` 就是光照。
- 采样式(非插桩)意味着开销极小,可以在生产服上跑——这是它取代旧 Timings 的关键。

### 其它手段

| 工具 | 用途 | 备注 |
| --- | --- | --- |
| **/tick** (1.21+) | 原版内置:`/tick rate`、`/tick freeze`、`/tick step`、`/tick sprint` | 直接干预第 04 篇那个时间累加器,调试时冻结世界很好用 |
| **F3 调试屏** | 客户端:看 FPS、区块、实体计数、内存 | 偏客户端,服务端深度诊断还得靠 spark |
| **Timings** (旧) | Paper 老一代服务端 profiler | 基本被 spark 取代,新项目别用 |

## 四、优化生态:每个 mod 治一段病

社区优化 mod 不是"一键变快"的玄学,每个都精准重写了某一段热点。知道它们各治什么,才能对症下药:

| mod | 治哪段 | 对应前文 |
| --- | --- | --- |
| **Lithium** | 重写大量游戏逻辑算法(AI、碰撞、刷怪、区块 tick 调度),不改行为只改实现 | 第 04/06/15 篇 |
| **Starlight** | 把原版那套拖沓的光照引擎整个重写,放挖方块时的光照更新快几个数量级 | 第 10 篇 |
| **C2ME** | 并行化区块加载/生成,把单线程的生成流水线摊到多核 | 第 05/16 篇 |
| **FerriteCore** | 砍内存占用(BlockState、NBT 等的去重),降 GC 压力 | 第 08/09 篇 |
| **Sodium** | **客户端**渲染引擎重写,大幅提 FPS(这是 FPS 药,不是 TPS 药) | 第 03/20 篇 |

服务端形态的选择也是性能决策:

- **Paper**(及其衍生)在原版基础上做了大量服务端优化和可配置项(实体活动范围、刷怪频率、漏斗节流等),纯生存服几乎默认起步。
- **Fabric + Lithium/Starlight/C2ME** 这条线更模块化,想保留 mod 生态时常走这边。

> 取舍提醒:Paper 这类为了性能,会**牺牲一部分原版行为的精确性**(某些机器、某些 tick 行为可能变样)。技术红石玩家反而要躲开这些"优化",因为他们依赖的正是原版那套"低效但确定"的更新顺序(第 14 篇的 quasi-connectivity、更新顺序)。性能和行为保真,在 Minecraft 里是一对要权衡的目标。

## 五、方法论:定位三步走

把上面串成一套不靠猜的流程:

```
1. 分清是哪台机器
   F3 看 FPS,/spark tps 看 TPS/MSPT —— 先确定卡在服务端还是客户端

2. 采样找热点
   /spark profiler 跑出火焰图 —— 看最宽的调用栈,别先入为主

3. 对症下药
   实体多 → 清掉落物 / 限刷怪 / 调 Paper 实体配置
   漏斗多 → 减 block entity / 上 Lithium
   光照卡 → Starlight
   生成卡 → C2ME / 降模拟距离(第 05 篇)
   红石风暴 → 改电路 / 接受它就是贵
```

核心原则就一句:**先测后改**。Minecraft 的性能问题分布极不均匀,凭感觉关掉的东西,十有八九不是真正的瓶颈;而真正吃满 MSPT 的那个漏斗阵列,你不开火焰图永远不会想到。
