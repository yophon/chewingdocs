# mod 架构:Fabric、Forge 与 Mixin

先说一个很多人没意识到的事实:**Minecraft Java 版从来没有官方 mod API。** Mojang 没开放任何"插件接口",游戏发布出来就是一个混淆过的、自洽的 jar。那几万个 mod 是怎么来的?答案不太体面但很硬核——**社区是在运行时把游戏的代码改写了**。整个 mod 生态,本质是一群人在没有任何官方扩展点的前提下,用字节码工程硬生生造出来的一层。

> 一句话先记住:**modloader 是一个在游戏启动前接管类加载的引导层,Mixin 是往别人编译好的方法里插桩的手术刀,mappings 是把混淆代码翻译回人话的字典——三者合起来,才让"改一个闭源游戏"变得可工程化。**

---

## 一、没有 API,怎么改?——modloader 接管类加载

普通程序加载类很直接:JVM 从 jar 里读 `.class` 字节,丢给类加载器,执行。modloader 做的事,就是**在这条链路中间插一脚**。

它本质是一个**引导器(bootstrap)**:在原版游戏的任何一个类被真正加载之前,先由 modloader 启动,接管类加载器,然后做两件事:

1. **扫描 mods 文件夹**,把每个 mod 的入口注册进来;
2. **在类字节被交给 JVM 之前,允许 mod 对它做改写**(下面 Mixin 那节展开)。

```
正常:  jar 字节 ──────────────► JVM 加载 ──► 运行原版逻辑

有 mod:jar 字节 ──► modloader 拦截 ──► 应用 mod 的字节码改写 ──► JVM 加载 ──► 运行被改过的逻辑
                         ▲
                      mods/*.jar 在这里插手
```

所以"装 mod"不是"游戏调用了插件",而是**游戏自己的代码在加载瞬间被动过手脚**。这个区别决定了后面所有的痛苦与威力:威力是你几乎能改任何东西;痛苦是你改的是私有实现,版本一变就崩。

## 二、三大加载器:Fabric / Forge / NeoForge

不存在唯一的 modloader,生态分裂成几支,取舍各不相同。

| | Fabric | Forge | NeoForge |
| --- | --- | --- | --- |
| 定位 | 轻量、最小内核 | 重量级、自带全套 API | Forge 的社区分支 |
| 跟版速度 | 快(新版本几天内可用) | 慢(API 大,适配久) | 中 |
| 自带 API | 几乎没有,功能靠 Fabric API 模块按需加 | 大而全(事件总线、能力系统、注册封装……) | 继承 Forge 并持续重构 |
| 适合 | 小而快的 mod、性能优化 mod、跟最新版 | 大型整合包、内容向 mod | Forge 路线的现代延续 |
| 入口机制 | `fabric.mod.json` 声明 entrypoint | `@Mod` 注解 + 事件总线 | 类 Forge |

几个要点:

- **Fabric 的哲学是"内核最小"**:加载器本身几乎只做引导和 Mixin 集成,连"注册方块"这种事都得引入 Fabric API 这个独立 mod。所以它轻、跟版快——这也是 Lithium、Sodium、Starlight 这类性能 mod 多在 Fabric 上的原因(它们要的就是贴近原版、快速适配)。
- **Forge 的哲学是"大而全"**:提供事件总线、能力(capability)、统一的注册和配置体系,内容向大 mod 写起来省事,代价是包重、跟版慢。
- **NeoForge** 是 2023 年从 Forge 分出来的社区分支,因为治理与技术路线分歧,如今很多新内容 mod 转向它。

**entrypoint(入口点)** 是各家共有的概念:mod 不是"被 new 出来",而是在元数据里声明"我的初始化逻辑在哪个类/方法",加载器在合适的生命周期阶段去调它。Fabric 区分 `main`/`client`/`server` 等入口,对应不同环境;Forge 用 `@Mod` 标注主类并通过事件回调挂载逻辑。

## 三、混淆与映射:为什么 mod 死死绑版本

这是理解 mod 生态脆弱性的关键,也是新手最容易忽略的一层。

Mojang **发布的是混淆(obfuscated)过的代码**:类名、方法名、字段名被替换成 `a`、`b`、`c`、`aabb` 这种无意义短名。原因是体积和一点点防护。问题是——你要给 `EntityPlayer.attack()` 注入逻辑,但在发布版里它可能叫 `class_1657.method_7324()`,甚至每个版本的混淆名都不一样。

于是有了 **mappings(映射表)**:一张把混淆名翻译回可读名的字典。

| 映射 | 阵营 | 风格 |
| --- | --- | --- |
| **Mojmap**(官方 official mappings) | Mojang 自 1.14.4 起放出 | 官方名,权威但许可受限,近年成主流 |
| **Yarn** | Fabric 社区 | 社区命名,开放,曾是 Fabric 默认 |
| **MCP / SRG** | 早期 Forge | 历史映射体系,现已被取代 |

开发时 mod 用可读名写代码;构建时再**重映射(remap)** 回目标版本的混淆名,才能和真实游戏对上。

**这就是 mod 强版本绑定的根源之一:** 混淆名随版本变,映射也随版本变,你针对 1.20.1 编译的 mod,里面写死的是 1.20.1 的混淆引用,拿到 1.21 直接对不上号——哪怕逻辑一行没改。

## 四、Mixin:往别人的方法里插桩

光接管类加载还不够,你得有手段**精确地改原版方法的内部**。这就是 Mixin——一个基于字节码注入的框架,现在是 Fabric 和 NeoForge 的事实标准(Forge 也支持)。

Mixin 的思路:你写一个"mixin 类",用注解声明"我要嫁接到哪个目标类的哪个方法的哪个位置",加载时框架把你的字节码**织入(weave)** 进目标方法。常用注解:

| 注解 | 干什么 |
| --- | --- |
| `@Inject` | 在方法的某个位置(开头/返回前/某次调用处)插入你的代码,可 `cancellable` 提前返回 |
| `@Redirect` | 把方法里的某个调用"改道"到你的实现 |
| `@ModifyVariable` / `@ModifyArg` / `@ModifyConstant` | 篡改方法里的局部变量、参数、常量 |
| `@Overwrite` | 整个替换目标方法(最暴力,最易冲突,尽量别用) |

注入点用 `@At` 描述,比如 `@At("HEAD")`(方法开头)、`@At("RETURN")`(返回前)、`@At("INVOKE", target=...)`(某个调用处)。

还有配套的开私有的手段:

- **Access Widener(Fabric)/ Access Transformer(Forge)**:原版很多成员是 `private`/`final`,你正常碰不到。这两个机制在加载时**放宽访问修饰符**,把私有字段变可访问、去掉 `final`,好让你的 mixin 能动它们。

### 为什么多个 mod 改同一处会冲突

Mixin 强大,但它**改的是共享的字节码**。两个 mod 都 `@Redirect` 同一行调用,或都 `@Overwrite` 同一个方法,就会撞车——织入顺序、互相覆盖,结果不可预测。

```
原版方法 foo() {
   ... 调用 bar() ...     ◄── modA 想 @Redirect 这次调用
   ... 调用 bar() ...     ◄── modB 也想 @Redirect 同一次调用
}
两个 @Redirect 落在同一目标 → 冲突,轻则一方失效,重则崩
```

这也是为什么整合包出问题时排查那么痛苦:崩溃栈里全是被织入的、不属于任何单一 mod 的"缝合方法"。`@Inject` 这种"加在旁边"的方式天然比 `@Overwrite` 这种"整个换掉"的兼容性好得多——好的 mod 作者会刻意用最轻的注入方式,就是为了少抢地盘。

## 五、版本脆弱性:为什么升级 mod 这么难

把前面几篇的点串起来看,你会发现 mod 卡在好几条会随版本漂移的线上,任何一条变了都可能让它失效:

| 漂移的东西 | 影响 | 关联 |
| --- | --- | --- |
| 混淆名 / 映射 | 方法字段引用全对不上,要重映射重编译 | 本篇第三节 |
| 目标方法的内部结构 | mixin 注入点(某次调用、某个变量)可能不存在了,织入失败 | 本篇第四节 |
| 注册表条目增删改名 | mod 注册的方块/物品 ID、与原版交互的 registry 假设可能失效;联机还要过 registry sync | 见"注册表、维度与修饰层"一篇 |
| 存档结构 | 原版用 DataFixerUpper 迁移,但 mod 自己加的数据没人替你迁,跨版本存档容易坏 | 见"数据组件与 DataFixerUpper"一篇 |

所以"Minecraft mod 升级一个大版本要等几个月"不是作者懒,而是**他依赖的每一层地基都在动**:映射变了、目标代码变了、注册和存档语义也可能变了。Fabric 之所以跟版快,正是因为它内核小、对原版内部结构的假设少;Forge 之所以慢,是因为它那套大 API 本身就深度绑定原版实现,牵一发动全身。

## 收一句

Minecraft 的 mod 生态是个很特别的工程样本:**在一个完全没有官方扩展点、还故意混淆过的闭源游戏上,社区靠"接管类加载 + 字节码注入 + 逆向映射"三件套,硬造出了一个能装上万个扩展的平台。**

它的威力来自"几乎能改任何东西",它的脆弱也来自同一件事——你改的不是稳定的 API,而是别人随时会变的私有实现。理解了这点,就理解了为什么 mod 那么强、又那么容易碎:**这层扩展能力从来不是设计出来的,是工程出来的。**
