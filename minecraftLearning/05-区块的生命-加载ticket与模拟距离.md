# 区块的生命:加载、ticket 等级与模拟距离

你建了一座全自动刷怪塔,站在旁边刷刷刷出怪、掉落、收集,一切完美。然后你跑去几百格外探险,回来发现塔里空空如也——它在你离开的整段时间里**根本没运转**。

很多人对这件事的理解停留在"区块卸载了嘛"。但真相要精细得多:那座塔所在的区块**很可能还加载着**,它只是被降到了一个"不再跑逻辑"的等级。"加载"和"运转"是两回事,而 Minecraft 用一套**连续的等级数值**来区分它们。

> 一句话先记住:**区块不是"加载/卸载"的二元开关,而是有一个 level 数值(越小越活跃)。玩家像光源一样把周围区块"照亮",离玩家越远等级越高、越不活跃。农场停摆不是因为区块没了,而是因为它的等级掉出了"会 tick"的范围。**

---

## 一、区块不是开关,是一片"距离场"

朴素的心智是这样的:

```
区块状态 = { 已加载, 未加载 }   // 错误的二元模型
```

这个模型解释不了一堆现象:为什么有的区块加载了但不刷怪?为什么离玩家近的机器跑、远的不跑,但更远的方块明明还能被命令读到?

真实模型是:每个被加载的区块带一个**整数等级 `level`**,这个等级构成一片**距离场(distance field)**,本质就是一次从"加载源"出发的 Dijkstra/BFS 传播:

- 玩家所在区块等级最低(最活跃);
- 向外**每扩散一圈,等级 +1**;
- 等级超过某个阈值,区块就"还在内存里,但什么都不做"。

把玩家想成一盏灯,区块等级就是亮度衰减:

```
等级越大 = 离玩家越远 = 越不活跃

   44 44 44 44 44 44 44
   44 34 33 33 33 34 44
   44 33 32 31 32 33 44
   44 33 31 [P]31 33 44      [P] = 玩家所在区块,等级最低
   44 33 32 31 32 33 44
   44 34 33 33 33 34 44
   44 44 44 44 44 44 44
```

这套机制在引擎里叫 **ticket 系统**:任何想让某区块保持活跃的东西,都往世界里放一张 **ticket(加载凭证)**,ticket 自带一个"能把区块压到多低等级"的能力,然后这个等级从 ticket 中心向外 +1 传播。**玩家本身就是一张会移动的 ticket。**

---

## 二、三条等级线:从"全功能"到"仅加载"

等级不是连续生效的,而是有几条**关键阈值线**,跨过去就掉一档能力。记住这几个数(1.18+ Java 版):

| 等级 level | 档位 | 这一档会发生什么 |
| --- | --- | --- |
| ≤ 31 | **entity-ticking(完整)** | 方块 random tick、流体、红石、**实体 AI 和移动**、刷怪——全功能 |
| 32–33 | **block-ticking** | 方块还 tick(红石、流体、随机刻可走),但**实体不再 tick** |
| 34 ~ ~44 | **border(仅加载)** | 区块在内存里、能被读写,但基本不做任何更新 |
| > 44 | 不加载 | 卸载,等级超出 ticket 影响范围 |

最关键的一条线是 **31 → 32**:

- **≤31**:实体会动、会寻路、会被刷出来。这就是俗称的"区块在 ticking"。
- **≥32**:实体集体冻结。掉落物不再被漏斗吸、怪不再走、村民不再干活。

第二条线是 **33 → 34**:跨过去连方块 tick 都停了,区块沦为"只是数据"。

> 反直觉点:一个区块可以**既"加载"又"完全不运转"**。`/forceload` 强制加载的区块默认就停在 border 档(约等级 34)——它保证区块在内存、能被命令和红石跨区块读取,但**不会自己跑逻辑**。很多人 forceload 了刷怪塔却发现还是不刷怪,就是栽在这里:加载 ≠ ticking。

---

## 三、三个"距离"别再搞混

玩家这边有三个看起来都叫"距离"的概念,它们管的是完全不同的事:

| 概念 | 谁设置 | 管什么 | 关键点 |
| --- | --- | --- | --- |
| **render distance(渲染距离)** | 客户端 | 客户端**画**多远的区块 | 纯视觉,服务端逻辑不看它 |
| **simulation distance(模拟距离)** | 服务端 | 多远的区块进入 **ticking(≤31)** | 决定 random tick、实体、刷怪范围 |
| **chunk 加载等级 / ticket level** | 引擎内部 | 每个区块**具体**的活跃档位 | 上面两者最终都落到这个数值上 |

实际关系:

```
render distance   → 决定客户端请求/绘制多大一圈区块(看得见)
simulation distance → 决定多大一圈区块等级 ≤ 31(真的在跑)
                      ↓ 两者都通过往区块"压等级"来生效
chunk level (距离场) → 每个区块最终的档位,决定它具体能做什么
```

一个常被忽略的事实:**simulation distance 可以小于 render distance**。你能"看见"远处的山(render),但那片山**不在 ticking 范围**(simulation),所以山里的怪不动、机器不转。看得见,不代表跑得起来。

## 四、推论:为什么远处的农场会停摆

现在那座刷怪塔的谜题自解了。当你跑远:

1. 你这张"移动 ticket"离开了塔所在区块;
2. 塔区块失去低等级来源,等级回升;
3. 一旦升过 **31**,实体停 tick——刷怪、掉落、漏斗收集全停;
4. 再升过 **33**,连 random tick 都停——作物也不长了;
5. 升过卸载阈值,区块写盘卸载。

所以"远处机器停摆"的本质,就是**它的区块等级掉出了 ticking 档**。这不是 bug,是 Minecraft 控制算力的核心手段:只给玩家身边那一小圈区块花 CPU。

要让远处机器**持续运转**,办法只有一个——**给它的区块放一张能压到 ≤31 的 ticket**:

| 手段 | 原理 |
| --- | --- |
| **挂机(AFK)在机器旁** | 玩家这张移动 ticket 把周围一圈压到 entity-ticking |
| **出生点区块(spawn chunks)** | 出生点周围一圈被**常驻 ticket**钉在 ticking,服务器开着就一直转 |
| **传送门(下界门)** | 激活的传送门会给两端区块放 ticket,常被用作"自然的区块加载器" |
| **`/forceload`** | 注意:默认只到 **border 档**,要配合机制才到 ticking;别指望它单独让实体动 |
| **第三方区块加载器 / mod** | 显式往目标区块写 entity-ticking 级别的 ticket |

> 踩坑:**出生点区块**是个双刃剑。它默认常驻 ticking,所以新手常把自动机器建在出生点附近"图省事"——确实一直转,但也意味着这些机器**永远在吃服务器 CPU**,哪怕全服没人在线。大型服务器反而会刻意缩小或关闭 spawn chunk 的常驻范围来省算力。

## 五、加载本身也是分阶段、异步的

最后补一层:把区块"拉到某等级"并不是一瞬间的事。区块从无到有要走一条**异步的生成流水线**,每个阶段(`empty → structure_starts → … → features → full`)都可能要等邻居区块先到某阶段才能继续(因为村庄、洞穴这种结构会跨区块)。

这条流水线跑在**独立线程池**上,结果再提交回主线程。所以:

- 你高速移动(鞍马、鞘翅、传送)时会看到**地形"长出来"有延迟**——生成流水线没追上你的脚步;
- 一个区块要先被生成到 `full`,才谈得上被 ticket 压到 ticking 档去跑逻辑;
- ticket 等级管的是"已存在的区块有多活跃",生成流水线管的是"区块怎么从零到 `full`"——两套系统配合,但解决的是不同问题。

---

## 六、串联

把这一篇接回前后:

- **接第 02 篇(生成阶段)**:那条 `empty → … → full` 的 Status 流水线,就是区块"出生"的过程;本篇的 ticket 等级则决定它出生后"活得多积极"。生成是一次性的,等级是动态的。
- **接第 04 篇(Tick 系统)**:上一篇讲的 random tick、scheduled tick、实体 tick、邻居通知,**全都只在等级足够低的区块里发生**。第 04 篇讲"一个 tick 内做什么",本篇讲"哪些区块有资格参与这个 tick"。两篇合起来才是完整的"世界怎么跑起来":Tick 系统是发动机,ticket 等级是油门——它决定发动机的动力分给哪几圈区块。

一句话收束本篇:**Minecraft 不会平等地更新整个世界,它只把算力投给玩家照亮的那一小圈。理解了 ticket 距离场,你就能解释几乎所有"为什么这台机器在这里转、在那里停"的现象。**
