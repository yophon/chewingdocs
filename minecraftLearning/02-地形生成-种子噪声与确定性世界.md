# 地形生成:种子、噪声与确定性世界

Minecraft 号称"无限世界",你可以朝一个方向走上几天,脚下永远有新地形冒出来。但这里有个绕不过去的工程矛盾:**无限的世界,装不进有限的硬盘**。如果地形是预先画好存下来的,那存到第几个 TB 才算"无限"?

答案是:**地形根本不存。它是在你走到时,被一个函数当场算出来的。** 这一篇讲清楚这个函数怎么来的、为什么它能保证你和你朋友看到的世界一模一样,以及一块石头从"虚空"变成"真实方块"要经过哪些流水线阶段。

> 一句话先记住:**地形 = f(种子, 坐标)。同一个种子、同一个坐标,永远算出同一个方块。** 所以世界不用存,只有被你改动过的方块才需要落盘。生成是确定的,存储是增量的。

---

## 一、为什么不能"预先画好存下来"

先把朴素方案枪毙掉。假设我们像存图片一样存地形:

```
世界 = 一个超大三维数组
存盘 = 把整个数组写进文件
```

问题立刻爆炸:

| 朴素做法 | 后果 |
| --- | --- |
| 世界有限 | 那就不是"无限世界",走到边界就掉下去 |
| 世界无限 | 文件无限大,根本写不完也存不下 |
| 玩家只逛了 0.001% | 却要为 100% 的地形付出存储 |

真正的世界 99.99% 是玩家**从没去过、也永远不会改**的区域。为这些区域存数据,纯属浪费。于是设计目标变成:

> **没去过的地方不占一个字节;去过、且改过的地方才存。中间的"原始地形"随时能重新算出来。**

要做到"随时能重新算出来",就需要一个**确定性函数**:给它一个坐标,它吐出那里应该是什么方块,而且每次吐的都一样。这个函数的核心,就是**噪声(noise)**。

---

## 二、噪声:可控的随机

直接用 `random()` 生成地形是不行的——它给出的是**白噪声**,每个点和相邻点毫无关系,生成出来是一片杂乱的"电视雪花",不是山脉。

地形需要的是**平滑的随机**:相邻坐标的高度应该接近,整体又有起伏。这就是 **Perlin 噪声 / Simplex 噪声**解决的问题。

```
白噪声(random):        平滑噪声(Perlin/Simplex):
 █ ░ █ ░ ░ █ ░ █          ░ ▒ ▓ █ ▓ ▒ ░ ░
 ░ █ ░ █ █ ░ █ ░          ▒ ▓ █ █ ▓ ▒ ▒ ░
 像素级抖动,无地貌        连绵起伏,像山和谷
```

Minecraft 的演进路线:

| 时期 | 用的噪声 | 备注 |
| --- | --- | --- |
| 早期 | **Perlin 噪声**(Ken Perlin 1983) | 经典,但有方向性伪影 |
| 后来 | **Simplex / OpenSimplex** | 更少方向伪影、更高维更快;OpenSimplex 是为规避 Simplex 专利的开源变体 |

噪声函数的关键性质,正是地形生成的命根子:

> `noise(x, z)` 是一个**纯函数**——只要输入坐标不变,输出永远不变,且与"先访问哪里、后访问哪里"无关。

这意味着你可以**乱序**地、**重复**地查询任意坐标,结果完全一致。地形因此可以"走到哪算到哪",不依赖任何已生成的历史。

---

## 三、种子:让"确定"可复制

光有噪声还不够。如果每个存档都用同一套噪声,所有世界就长得一模一样了。所以引入**种子(seed)**:

```
最终地形 = noise(seed 扰动后的坐标空间)
```

种子的作用是**给噪声函数换一套坐标偏移/排列**,于是:

- **同一个种子** → 同一个世界。你把种子发给朋友,他生成出的出生点、村庄、地形和你**逐方块相同**(同版本下)。
- **不同种子** → 完全不同的世界。

这就是为什么社区里能流传"种子号":一串数字就完整编码了一个无限世界的全部原始地貌。**世界的信息量被压缩进了"种子 + 生成算法"里**,而不是存在文件里。

```
你的存档真正存的:
  ┌─────────────────────────────┐
  │ 种子(决定原始地形)         │  ← 几个字节
  │ + 你改动过的方块(挖矿/建造)│  ← 增量,只存 diff
  └─────────────────────────────┘
原始地形本身 = 0 字节,需要时用种子重算
```

## 四、多层噪声叠加:从一条曲线到一个世界

单层噪声只能给你一个"高度图"——平滑但单调。真实地形要同时表达**大尺度的山脉**和**小尺度的碎石起伏**。做法是**多层噪声叠加(octaves)**:

```
高度(x,z) =  1.0 * noise(x/256, z/256)   // 低频:大陆、山脉走向
          + 0.5 * noise(x/64,  z/64)    // 中频:丘陵
          + 0.25* noise(x/16,  z/16)    // 高频:地表碎起伏
          + ...
```

工程直觉就三个旋钮:

| 旋钮 | 含义 | 调大的效果 |
| --- | --- | --- |
| **频率 frequency** | 坐标被除以的尺度 | 频率高 → 起伏密集、细碎 |
| **振幅 amplitude** | 这一层的权重 | 振幅大 → 这一层主导地貌 |
| **octave 层数** | 叠几层 | 层多 → 既有大山又有细节 |

通常每加一层,**频率翻倍、振幅减半**(这种叠加方式叫 fractal/fBm)。低频层管"大势",高频层管"细节",叠起来就有了既宏观又精致的地貌。

### 不只是高度:气候场

Minecraft 现代地形(1.18 重写后)不是只算一个高度,而是同时算**好几张独立的噪声场**,再用它们共同决定每个位置:

```
温度场 temperature ─┐
湿度场 humidity    ─┤
大陆度 continentalness ─┼──►  查"气候 → 生物群系"表  ──►  这里是沙漠?雪原?丛林?
侵蚀度 erosion    ─┤
怪异度 weirdness  ─┘
```

- **生物群系不再是"先画板块再填"**,而是"算出这点的气候坐标,再去多维表里查它落在哪个 biome"。
- 同一套噪声还驱动**矿脉分布**(不同矿在不同高度/密度生成)和**洞穴**(用 3D 噪声直接在实心区域"挖空"出隧道与巨洞,即所谓 noise caves)。

注意一个维度差异:**高度图是 2D 噪声**(每个 x,z 一个值),而**洞穴是 3D 噪声**(每个 x,y,z 一个值,决定该点是石头还是空气)。3D 噪声更贵,但能造出 2D 高度图永远做不出的悬空洞穴和拱桥。

## 五、生成是分阶段的流水线,不是一锤子

你可能以为"生成一个 chunk"是一个原子操作:输入坐标,输出完整地形。**不是。** 因为有些结构是**跨 chunk** 的——一个村庄、一条洞穴、一棵树,可能横跨好几个 chunk 的边界。

如果每个 chunk 各自独立地"一次性生成完",边界处就会对不上:村庄被切成两半、洞穴在 chunk 边界突然截断。

为了解决这个,Minecraft 把生成拆成**一连串阶段(ChunkStatus)**,每个 chunk 像在流水线上一格格往前走:

```
empty            （还什么都没有）
  → structure_starts   （决定这里要不要起一个村庄/要塞的"种子点"）
  → structure_references
  → biomes             （确定生物群系）
  → noise              （用噪声填出石头/空气的基础地形）
  → surface            （铺地表:草、沙、雪）
  → carvers            （雕刻洞穴、峡谷)
  → features           （撒树木、矿脉、植被、结构主体）
  → light              （计算光照)
  → spawn
  → full               （完工,可被游戏逻辑使用)
```

关键约束:

> **一个 chunk 要推进到某个阶段,它周围的邻居 chunk 必须先到达前一个阶段。**

比如要给中心 chunk 跑 `features`(种树、放村庄房子),引擎需要保证它周围一圈邻居都已经到了 `noise`/`surface`,这样跨边界的结构才能正确地"画"到邻居身上。这就是为什么生成新区域时,引擎实际上在**同时推进一整片 chunk 的阶段**,像一个向外扩散的波纹,而不是孤立地一个一个生成。

```
中心要 features,必须邻居先就位:

  noise noise noise
  noise [feat] noise      ← 中心能种跨界的树/村庄,
  noise noise noise          是因为外圈已经有地形可以"画"上去
```

这也是"卡在生成区块"(Building terrain / 加载卡顿)的根源之一:不是算一个 chunk 慢,而是要把一大片邻居都拉到对应阶段才能交付中心那一个。

## 六、踩坑与反直觉

**1. "重新生成会不会把我家删了?"**
不会。生成只负责**原始地形**。你改过的方块作为增量单独存盘(后续讲存储时会展开),加载时是"先按种子算原始地形,再叠加你的改动"。原始地形可重算,改动不可重算——两者分开存,这是整个设计的地基。

**2. 同种子不同版本,世界不一样。**
确定性只在**同一版本的生成算法**下成立。Minecraft 大版本经常重写地形(最著名的是 1.18 把地形和洞穴生成整个换掉),所以同一个种子在 1.16 和 1.18 里完全是两个世界。存档里的 `DataVersion` 标记着它由哪个版本生成,跨版本加载时旧区块保持原样、新区块用新算法——于是老世界里常见一条"地形断层带",正是新旧生成算法在你当年探索边界处接缝的痕迹。

**3. 生物群系不是"画"出来的板块。**
别用"先有一张生物群系地图、再填地形"的旧心智。现代实现是反过来的:**先有连续的气候噪声场,生物群系是气候坐标的查表结果。** 所以 biome 边界其实是噪声等值线,过渡自然而非硬切。

**4. 洞穴为什么有的是细管、有的是巨洞?**
因为它们来自 3D 噪声 + 雕刻器(carver)两套机制叠加。3D "奶酪/面条"噪声负责大尺度空腔,carver 负责蜿蜒隧道,两者尺度不同,叠出来就有粗有细。

**5. 噪声查询是无副作用的,可并行。**
正因为 `noise(x,z)` 是纯函数、不依赖访问顺序,地形生成天然适合多线程:不同 chunk 的 `noise` 阶段可以并行算,互不干扰。串行瓶颈主要在跨 chunk 的 `features` 阶段,而不是噪声本身。

## 七、串联

把这一篇放回整条主线:

- **承接 01(体素世界)**:上一篇说"世界由 chunk 和 16³ section 的方块网格组成,但无限世界不可能全量存"。这一篇就是那句话的下半段——**网格里每个方块的初始值,由种子+噪声当场算出**,所以"无限"才成立。
- **接着 03(渲染)**:噪声只决定"每格是什么方块",算完得到的是一片纯数据的方块网格;怎么把这堆方块**高效画到屏幕上**(只画看得见的面、合并成一个 mesh),是渲染篇的事。
- **呼应 08(存储)**:生成 vs 存储的分工,正是本系列的主线——"能算出来的就不存"。地形能算,所以不存;改动算不出来,所以才存增量。而 `DataVersion` 这个字段,就是连接"哪个版本的生成算法"和"存档怎么迁移"的关键标记,存储篇会再展开。

一句话收束:**种子是世界的全部信息,噪声是把它展开成地形的函数,流水线阶段是让跨区块结构不破缝的工程手段。三者合起来,才让"无限世界 + 有限硬盘"这个矛盾不再矛盾。**
