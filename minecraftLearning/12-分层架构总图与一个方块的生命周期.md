# 分层架构总图与一个方块的生命周期

前面十一篇,我们从体素世界一路拆到注册表:地形怎么算出来、区块怎么画、20 TPS 怎么驱动世界、数据怎么存盘、方块和实体和流体到底是什么、最后一切类型怎么登记在注册表里。知识点很多,但如果只是记住一堆零散名词,过几天就忘了。

这一篇不引入任何新机制,只做一件事:**把这些点缝成一张图,再让一个方块在这张图上"活一遍"**。当你能闭着眼睛画出这张分层图、能讲清一格石头从生成到存盘再到加载走了哪些层,你就真正理解了 Minecraft 的骨架,而不只是背下了它的零件。

> 一句话先记住:**Minecraft 的世界是"同一格上叠着多层数据 + 一根注册表主轴穿透所有层";理解它的关键不是数清有多少种东西,而是看清每种东西活在哪一层、以及一格数据怎么在这些层之间流动。**

---

## 一、先立一条正交轴:类型 vs 实例

在画总图之前,必须先讲清一条**横切所有层**的轴,否则总图会看不懂。

前面我们反复遇到一个现象:几乎每种"东西"都分成**两层**。

| 类型层(单例,定义行为,在注册表里) | 实例层(逐个存在,带数据,要存盘) |
| --- | --- |
| `Block`(石头这"类") | BlockState 下标(世界里那一格) |
| `Item`(钻石剑这"类") | `ItemStack`(你背包里这一把) |
| `EntityType`(僵尸这"类") | `Entity`(刷出来的那只僵尸) |
| `BlockEntityType`(箱子这"类") | `BlockEntity`(这个坐标的箱子) |
| `Fluid`(水这"类") | `FluidState`(这格的水) |

左列是游戏启动时就固定下来的"模板",像面向对象里的"类",全游戏每种只有一个实例,登记在注册表里。右列才是世界里逐个存在、要进 chunk、要序列化的"对象"。

**为什么这条轴重要?** 因为它解释了 Minecraft 最核心的省法:

- 千万个石头方块,共享**同一个** `Block` 单例和同一个 BlockState 引用,世界里每格只存一个"调色板下标"。
- 一种方块的所有行为(硬度、碰撞、掉落、tick 逻辑)只写一份,挂在类型层。

这条轴是**正交的**——它不是"另一类东西",而是看待每一类东西的两个视角。后面总图里几乎每个名词,你都要意识到它同时活在这两层上。

---

## 二、一个反直觉的事实:一格上叠着好几层

新手脑子里的世界是"每个格子里放一个方块",一格 = 一个东西。**这是错的。** 同一个坐标上,实际叠着好几层彼此半独立的数据:

```
坐标 (x, y, z) 上同时存在:
┌─────────────────────────────────────────────┐
│  BlockState   ← 方块状态(楼梯、朝向、含水标记) │
│  FluidState   ← 流体状态(这格有没有水、level)  │  ← 与方块半独立
│  Light        ← 光照(BlockLight + SkyLight)    │  ← 两个通道各 0-15
│  (BlockEntity) ← 少数方块才有的 NBT 附加/渲染锚 │  ← 可选
│  Biome        ← 4×4×4 粒度的气候标签(共享)     │  ← 粗粒度
└─────────────────────────────────────────────┘
```

这正是**含水机制(waterlogging)** 能存在的原因:一个被水淹的楼梯,它的 BlockState 仍然是楼梯,FluidState 才是水——两层叠在一格上,各存各的,各自 tick。

记住这一点,你才不会问出"水到底算不算方块"这种问题:水既有自己的 FluidState 层,在源方块处又对应一个 BlockState,它跨了两层。

## 三、两个不同的问题:别把它们混成一个

学到这里很多人会不停追问"还有没有别的类别",问到停不下来。根本原因是**"类别"这个词跨了两个性质完全不同的问题**,一个有限、一个无限,必须分开。

**问题 A:世界里有哪些"逐实例对象 / 数据层"?** —— 这是**有限集**,基本可以穷举:

```
方块 BlockState · 流体 FluidState · 方块实体 BlockEntity
· 实体 Entity · 物品 ItemStack · 生物群系 Biome · 光照 Light
(· 粒子 Particle:纯客户端视觉,算半个)
```

这张表问到这里就基本到头了，再问"还有没有"，答案趋近于"没有了"。

**问题 B:引擎里有哪些"系统 / 层"?** —— 这是**开放集**,可以一直往下拆:

```
修饰层(属性/状态效果/附魔/数据组件) · 运行时信号(红石/游戏事件/声音/调度)
· 交互会话(菜单/网络) · 元系统(注册表/标签) · 容器(维度/存档/数据包)
· 还能继续:碰撞形状 VoxelShape、AI 的 Brain/记忆、结构生成、世界生成噪声路由…
```

因为这问的不再是"世界里有什么",而是"游戏怎么运转"——而运转的系统永远能再细分。

**所以:** 如果你问的是"平行于方块的对象类别"(A),那张七项表已经到头;如果你问的是"引擎还有多少系统"(B),那它本来就没有尽头。把这条线画清楚,你就不会再有"是不是漏了什么"的焦虑。

现在可以上总图了。从下往上 = 从"静态怎么存"到"动起来怎么交互";最左边的注册表是穿透所有层的主轴。

╚══════════════════════════════════════════════════════════════════╝

┌────────────────────────────────────────────────────────────────┐
│ 0. 元层:注册表 Registry —— 一切"类型"的总账                     │
│    · 命名空间ID(minecraft:stone)+ 数字ID                       │
│    · 启动冻结 → 进服同步 → palette/网络/存盘都用数字ID            │

## 五、如何读这张图

这张图不是用来背的,是用来"定位"的。三条读法:

**纵向:从"存"到"动"再到"交互"。** 下层(6/7)回答"静态的数据怎么躺在硬盘上",中层(5)回答"它怎么被驱动起来",上层(1-4)回答"动起来之后世界里有哪些东西、哪些状态、哪些交互"。任何一个游戏现象,你都能在某一层找到它的落点。

**横向:注册表是穿透所有层的主轴。** 每一层的"类型"都来自第 0 层;存盘 palette 里的数字 ID、进服时的注册表同步、跨版本的 DataFixerUpper 迁移,全都绕着注册表转。理解了它,前面零散的点就被一根线串起来了。

**一格 = 多层叠加。** 这是最反直觉、也最关键的一点(第二节讲过):同一坐标上叠着 BlockState + FluidState + Light,可能还挂一个 BlockEntity。别再把世界想成"每格一个东西"。

还有那条**正交轴**别忘了:图里几乎每个名词,都同时活在"注册表里的单例模板"和"世界里的逐实例对象"两层上。

## 六、一个 16³ Section 的生命周期

光有静态分层还不够。真正把所有层缝起来的,是数据**流动**的过程。下面让一个 16×16×16 的 section(以及里面的一格方块)从出生到重新加载,在这张图上走一遍。

```
                  一个 16³ Section 的一生
═══════════════════════════════════════════════════════════════

① 生成(第 0/1/6 层:噪声 → PalettedContainer)
   世界生成器对每个坐标问噪声函数(种子+坐标→确定值)
   → 决定每格是空气/石头/矿石…
   → 写入内存中的 PalettedContainer:
        palette = [air, stone]          ← 局部调色板
        data    = bit-packed Long_Array ← 每格只存调色板下标
   注:此刻硬盘上什么都没有,地形是"算出来"的,不是"存出来"的
        │
        ▼
② 运行(第 5 层 Tick 系统 驱动 第 1 层 数据)
   只要该 chunk 进入 ticking 等级(ticket level < 32):
     · random tick 抽样   → 抽到作物/草 → 生长,改写 PalettedContainer
     · scheduled tick 队列 → 水流逐圈扩散、红石中继延迟
     · setBlock + 邻居通知 → 红石级联、沙子下落、活塞响应
   每次方块变化:
     · 调色板可能新增种类(放下一块从未出现的方块)
     · 种类变多 → bitsPerEntry 自动增大,Long_Array 重新打包
     · 种类变少/归一 → 位宽收缩,甚至退化成单值省掉数组
        │
        ▼
③ 卸载 或 自动保存(第 1 层 → 第 6 层:序列化下盘)
   触发:周期性自动保存 / 玩家离开 chunk / 关服
     · PalettedContainer 几乎原样序列化成 block_states:
          { palette: [...], data: Long_Array }   ← 内存与磁盘同构!
     · 连同 block_entities、未结算的 block_ticks/fluid_ticks、
       Heightmaps、biomes、光照 → 打包成该 chunk 的 NBT 树
     · NBT 树 → zlib 压缩
     · 在 .mca 里找一段连续空闲扇区写入(按 4KB 对齐)
     · 更新文件头的 location table(偏移+扇区数)与 timestamp table
        │
        ▼
④ 再加载(第 6 层 → 第 1 层:反向走一遍)
   玩家走回来:
     · 查 location table → 得到扇区偏移与长度
     · seek 到扇区 → 读出 → 看压缩类型字节 → 解压
     · 解析 NBT 树
     · 用 block_states 的 palette + Long_Array 重建 PalettedContainer
   回到 ②,继续被 tick 驱动
```

注意第 ③ 步那行注释:**内存里的 PalettedContainer 和磁盘上的 `block_states` 结构同构**——都是"局部调色板 + 紧凑位数组"。所以保存几乎是"原样倒出去",加载几乎是"原样读回来",中间没有昂贵的格式转换。这不是巧合,是刻意的设计。

## 七、收尾:一条主线贯穿始终

如果这整个系列只让你记住一句话,那就是这条贯穿十二篇的主线:

> **能算出来的就不存,能共享的就不复制,内存与磁盘尽量同构。**

回头看,Minecraft 几乎所有的"省"和"快",都是这一招的不同变体:

- **能算不存**:无限地形不进硬盘,靠种子+噪声确定性现算(第 01/02 篇);只有玩家改过的 chunk 才落盘(第 07 篇);战利品箱打开前只存"配方+种子",物品现场生成(第 09 篇)。
- **能享不复制**:千万个石头共享一个 BlockState 引用,世界里每格只存调色板下标(第 09 篇);BlockState 全局调色板让方块没有逐实例数据。
- **内存与磁盘同构**:PalettedContainer 与 `block_states` 结构一致,加载零转换(第 03/08 篇与本篇)。

把这条主线和这张分层总图一起记住,你看 Minecraft 就不再是"一堆方块的游戏",而是一个**把确定性生成、数据共享和结构同构做到极致的体素引擎**。每一个你在游戏里观察到的现象——麦子参差成熟、远处机器停摆、含水的楼梯、打不开的箱子——都能在这张图上找到它的层、找到它的因。

这就是这个系列想留给你的东西:不是答案的清单,而是一张能自己推出答案的地图。
