# 数据组件与 DataFixerUpper:物品新模型与版本迁移

第 08 篇讲 NBT 的时候,我说它是 Minecraft "万物的统一序列化格式"——带类型、可嵌套、紧凑。这话现在仍然对,但物品(ItemStack)这块,在 1.20.5 发生了一次"地基级"的重构:**物品的数据模型从"一坨自由 NBT"换成了"一组强类型组件"**。这一篇讲清楚两件事——新模型长什么样、为什么值得改,以及一个配套但更底层的机制:Minecraft 凭什么让你十年前的存档还能在今天打开,也就是 DataFixerUpper。

> 一句话先记住:这两件事是同一个方向上的两步——**把数据从"弱类型、随便塞的 NBT"推向"强类型、可校验、可迁移的结构"**。组件解决"物品当下怎么存",DFU 解决"旧结构怎么变成新结构"。

---

## 一、旧世界:物品 = 种类 + 一坨 `tag`

1.20.5 之前,一个 ItemStack 在数据上是这样:

```
{
  id: "minecraft:diamond_sword",
  Count: 1,
  tag: {                         ← 一个自由的 NBT Compound,什么都能往里塞
    Damage: 12,
    Enchantments: [{id:"minecraft:sharpness", lvl:5}],
    display: { Name: '{"text":"断钢"}', Lore: [...] },
    AttributeModifiers: [...],
    CustomModelData: 1234,
    ...
  }
}
```

核心问题全在那个 `tag` 上:**它是一个没有 schema 的口袋**。

- **弱类型**:`tag` 里允许出现任何键、任何嵌套。一个键拼错(`Enchantment` 漏个 s),不会报错,只是静默失效。
- **难校验**:游戏没法在"物品被创建"这一刻就确认它合法,只能各处用到时各自防御性读取。
- **难精确匹配**:配方、谓词(predicate)、命令选择器要判断"这把剑是不是附了锋利",得去 `tag` 里手动翻嵌套结构,匹配逻辑又臭又长,而且容易因为"多了个无关键"就匹配失败或成功。
- **耦合**:`display.Name`、`Enchantments`、`Damage` 全挤在同一个 Compound 里,语义边界靠约定俗成的键名维持。

这套能用,Minecraft 也确实用了很多年。但它本质是"把所有物品的所有可能字段塞进同一个自由表",越往后字段越多,越难管。

---

## 二、新世界:物品 = 种类 + 一组类型化组件

1.20.5 起,ItemStack 变成:

```
物品种类(minecraft:diamond_sword)
  + 一组组件 components: 一个 “组件类型 → 值” 的映射
      minecraft:max_damage      → 1561
      minecraft:damage          → 12
      minecraft:enchantments    → {sharpness: 5}
      minecraft:custom_name     → "断钢"
      minecraft:lore            → [...]
      minecraft:attribute_modifiers → [...]
```

关键变化不只是改了个名字,而是**每个组件都是一个注册过的、强类型的东西**(`minecraft:damage`、`minecraft:food`、`minecraft:container`、`minecraft:custom_name`……),它们登记在一个组件类型注册表里(呼应第 11 篇:一切类型都进注册表)。一个组件有明确的值类型、明确的序列化方式、明确的默认值。

### 默认组件 + 实例只存差异

这是新模型最聪明的一点,和第 09 篇讲方块"不存逐实例数据"是同一种思路。

- 每个**物品种类**自带一组**默认组件**(prototype)。比如所有钻石剑默认 `max_damage=1561`、`attack_damage`、`tool` 规则等。
- 一个**具体物品实例**,只存它**相对默认值的差异**:覆盖了哪些、删除了哪些。

```
钻石剑(种类)默认:  max_damage=1561, tool={...}, ...     ← 共享,不逐个存
你手里这把(实例):  + damage=12  + enchantments={sharpness:5}  + custom_name="断钢"
                                                          ↑ 只存这三条覆盖
```

所以一把没附魔没改名的普通钻石剑,实例侧几乎不存任何组件——和默认 prototype 一致。能共享的就不复制,这条主线在物品层又出现了一次。

### 可以"删默认值"——一个反直觉点

既然实例是"对默认的覆盖",那就必然需要一个操作:**移除某个本来默认存在的组件**。命令里写成 `!minecraft:component`(带感叹号),意思是"在这个物品上把这个默认组件去掉"。

```
/give @s minecraft:diamond_sword[!minecraft:enchantment_glint_override]
                                  ↑ 去掉默认的某组件
/give @s minecraft:stick[minecraft:food={nutrition:4}]   ← 给棍子加上“能吃”
```

第一次见 `!` 前缀的人都会愣一下:不是"设为空",而是"从默认集合里删掉这一项"。这正是"实例 = 对默认的 diff"模型的直接体现——diff 既能加、能改,也能减。

## 三、为什么值得改:从"自由口袋"到"强类型契约"

把前两节对起来看,改动的收益很清楚:

| 维度 | 旧 `tag`(自由 NBT) | 新 components(强类型) |
| --- | --- | --- |
| 类型 | 弱类型,键随便加 | 每个组件注册过、有固定值类型 |
| 校验 | 用到才发现错 | 创建时即可校验合法性 |
| 匹配 | 手翻嵌套,易误判 | 按组件精确匹配,默认值参与合并 |
| 边界 | 全挤一个 Compound | 每个组件语义独立、可单独读写 |
| 共享 | 每个物品自带全量 tag | 默认来自种类,实例只存 diff |

对**配方和谓词**尤其关键。以前判断"消耗一把附了锋利的剑",要写一段在 `tag.Enchantments` 里翻数组、还要担心其它无关键干扰的逻辑;现在直接声明"需要 `minecraft:enchantments` 含 sharpness",由强类型组件系统去比对,默认值也按规则参与。匹配从"字符串/结构层面的脆弱比对"变成"类型层面的明确契约"。

命令侧的体感变化:旧的 `{tag}` 语法整体被 `[component=value]` 取代。这不是语法糖,而是底层模型换了之后命令必须跟着换——你操作的对象从"一个 NBT 口袋"变成了"一组有名有姓的组件"。

> 注意边界:这次换的是**物品**的数据模型。方块实体(箱子内容、告示牌)、实体的存档,**仍然是 NBT**(第 08 篇那套)。组件化目前是物品这条线的事,别误以为整个游戏都不用 NBT 了。

## 四、DataFixerUpper:旧结构怎么变成新结构

上面这种"地基级"的数据模型重构,会立刻引出一个尖锐的问题:**那我 1.20.4 存的世界,里面全是 `tag` 格式的物品,1.20.5 怎么读?** 更一般地:Minecraft 十几年改了无数次数据结构(方块 ID 扁平化、世界变高、实体拆分存储、物品组件化……),凭什么老存档还能打开?

答案是 **DataFixerUpper(DFU)**——一套专门做"数据结构跨版本迁移"的机制。

### DataVersion:每份数据都带着"我是哪一版"

回顾第 02、08 篇:几乎每份存档数据(`level.dat`、每个 chunk、每个实体)都带一个 **DataVersion**(一个单调递增的整数)。它就是迁移的锚点——加载时,游戏先看这份数据的 DataVersion,和当前版本一比,就知道"中间隔了哪些结构变更,要补哪些迁移"。

### type schema + 按版本叠加的 fixer

DFU 的核心不是"if 旧版 then 手改字段"这种散装补丁,而是一套更结构化的东西:

- **type schema**:为"某个版本下,一个 chunk / 一个实体 / 一个物品的结构长什么样"建立类型描述。
- **fixer / rewrite rule**:每个引入结构变更的版本,登记一组重写规则——把上一版 schema 的数据,改写成本版 schema 的数据。比如"1.20.5 这一版:把 `tag` 里的字段按映射拆成对应的 components"。

```
DataVersion 升级链(示意):

旧存档 (v3700)
   │  应用 v3701 的 fixer:tag → components 拆解
   ▼
   (v3701)
   │  应用 v3702 的 fixer:某组件改名 / 某字段挪位
   ▼
   ...逐版本叠加...
   ▼
当前版本结构 (vNNNN)  →  交给游戏正常加载
```

关键点:迁移是**逐版本累加**的,不是"从任意旧版直接跳到最新"的一个大函数。每个版本只负责"我相对上一版改了什么"。任意旧版本的数据,沿着这条链一路重写上来,最后都变成当前版本认识的结构。**这就是老世界能打开的全部秘密**——不是游戏"兼容"旧格式,而是加载前先把旧格式**重写成**新格式。

### 它和组件化是一回事的两面

把第三、四节合起来看:

- 组件化解决"**从现在起**,物品该怎么存"——强类型、可校验、存 diff。
- DFU 解决"**过去存的**,怎么变成现在这样"——按 DataVersion 把 `tag` 重写成 components。

没有 DFU,组件化这种重构就没法做——你不能为了改进数据模型就抛弃所有老存档。**正是因为有 DFU 兜底迁移,Mojang 才敢一次次动地基。** 这也是为什么这两件事值得放在同一篇里讲:它们是"数据模型演进"这件事的正向设计与向后兼容两面。

## 五、踩坑与反直觉

- **`!` 不是清空,是删默认**:`[!minecraft:xxx]` 是"把这个默认组件从物品上摘掉",不是"设成空值"。这是 diff 模型的直接后果。
- **匹配要考虑默认值**:判断两个物品"组件是否相同"时,实例侧没写的组件会回落到种类默认值。所以"看起来组件列表不一样"的两个物品,合并默认后可能等价;反过来也成立。别只比实例侧的 diff。
- **大版本首次加载特别慢**:跨了很多版本的老世界第一次在新版本打开,DFU 要对**每个 chunk、每个实体**沿升级链逐版本重写,CPU 和时间开销显著。这不是卡死,是在做一次全量数据迁移;迁移并存盘后,之后加载就快了(DataVersion 已是新的,不再触发 fixer)。
- **NBT 没有退场**:再强调一次,组件化是物品这条线;方块实体、实体存档仍是 NBT。两套模型在当前版本是并存的。
- **DFU 不是双向的**:它只负责"旧 → 新"。新版本存的世界,旧版本客户端基本打不开——没有"降级 fixer"。

## 六、串联

把这一篇接回前面:

- **第 08 篇**给了 NBT 的字节模型和 DataVersion 的概念,这一篇说明了 DataVersion 真正的用武之地——驱动 DFU 的升级链。
- **第 11 篇**讲注册表是"一切类型的总账",组件类型同样登记在注册表里;组件化本质是把物品数据也纳入"强类型、可注册、可数据驱动"的世界观。
- 和**第 09 篇**方块"不存逐实例数据"对照:物品的"默认组件 + 实例存 diff",是同一条主线在物品层的复现——**能共享的就不复制**。

所以这一篇其实是整个系列那条暗线的一个集中体现:Minecraft 一直在把"自由、弱类型、什么都能塞"的数据,慢慢收紧成"强类型、可校验、可迁移"的结构;而每一次收紧地基,背后都得有 DataFixerUpper 这样的迁移机制兜底,旧世界才不会被丢下。
