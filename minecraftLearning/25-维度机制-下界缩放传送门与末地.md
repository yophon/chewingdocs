# 维度机制:下界缩放、传送门与末地

第 11 篇说过"维度是一个个独立的世界,各自有独立的 chunk 存储"。那一篇是把维度当成容器一笔带过的。这一篇把容器打开,讲清楚三件容易被当成"魔法"的事:**为什么下界走得快、传送门凭什么知道把你送到哪、末地那一堆传送门各自是干嘛的**。

> 一句话先记住:**维度不是"另一张地图",而是一组由 `DimensionType` 描述的环境参数 + 一份独立的世界生成配置 + 一份独立的存档。维度之间唯一的连接是传送门,而传送门干的活本质上只是"按比例换算坐标 + 在附近找/建一个出口"。**

---

## 一、维度由两层定义:类型(环境)+ 生成(地形)

很多人以为"主世界 / 下界 / 末地"是写死在引擎里的三个东西。1.16 之后**完全不是**——一个维度是数据驱动的,由两块拼出来(呼应第 11 篇动态注册表、第 16 篇世界生成):

| 层 | 注册表 | 管什么 |
| --- | --- | --- |
| **维度类型 DimensionType** | `dimension_type` | 环境参数:高度范围、坐标缩放、有没有天空、环境光、能不能放床/床会不会炸、岩浆流速、出怪光照等 |
| **维度本身 LevelStem / dimension** | `dimension` | 把一个 DimensionType 和一套世界生成器(noise settings / biome source)绑在一起 |

也就是说:**环境规则**和**地形怎么长**是分开配置的。你可以写一个 JSON 给"主世界地形 + 下界环境参数"的缝合怪维度,引擎照跑。

### DimensionType 里最关键的几个字段

```
{
  "min_y": -64, "height": 384,         // 世界高度范围(主世界 1.18 后)
  "coordinate_scale": 1.0,             // 坐标缩放,关键!
  "has_ceiling": false,                // 下界为 true:有基岩天花板
  "has_skylight": true,                // 下界/末地为 false:没有天空光
  "ambient_light": 0.0,                // 环境光底噪,下界 0.1
  "bed_works": true,                   // 下界/末地 false → 床会爆炸
  "respawn_anchor_works": false,       // 下界专属重生方式
  "ultrawarm": false,                  // 下界 true:水会蒸发、岩浆流得快
  "natural": true,                     // 末地/下界 false:不能用床、指南针/时钟乱转
  "logical_height": 256                // 传送门和区块加载能到的逻辑高度上限
}
```

| 维度 | min_y / height | coordinate_scale | skylight | ceiling | ultrawarm |
| --- | --- | --- | --- | --- | --- |
| 主世界 | -64 / 384 | **1.0** | 有 | 无 | 否 |
| 下界 | 0 / 256 | **8.0** | 无 | 有(基岩顶) | 是(水蒸发) |
| 末地 | 0 / 256 | 1.0 | 无 | 无 | 否 |

那个 `coordinate_scale = 8.0` 就是下面一切的根源。

## 二、下界 8:1:不是"下界更小",是坐标换算

最经典的反直觉点:**主世界走 1 格,等于在下界走 8 格。**

注意它的实现位置——缩放是写在 `DimensionType.coordinate_scale` 上的,**主世界 1.0、下界 8.0**。换算发生在**跨维度传送的那一刻**:

```
进下界:  nether.x = overworld.x / 8     (用两者 scale 的比值 8.0/1.0)
出下界:  overworld.x = nether.x * 8
y 坐标不缩放,直接搬过去
```

所以"下界更小"是个错觉。下界的方块和主世界一样大,只是**它覆盖的主世界范围被压缩了 8 倍**:你在下界铺一条 100 格的冰道,对应主世界 800 格。这就是**下界高速路**的全部原理——拿 1:8 的换算比,在下界跑短途换主世界长途。

```
主世界  A(0,0) ───────── 800 格 ───────── B(800,0)
                  ↕ 传送(/8)              ↕ 传送(/8)
下界    a(0,0) ── 100 格 ── b(100,0)
```

**踩坑:** 正因为是"传送时换算 + 附近找门",1:8 不是精确双射。多个主世界传送门可能落进下界的同一片搜索范围,于是出现"两个主世界门共用一个下界门""回来时出错地方"的经典配对错乱(见下一节)。手工把每对门的坐标按 /8 对齐,才能稳定一对一。

## 三、传送门:一段"换算坐标 + 找门或建门"的算法

站进激活的紫色传送门,客户端会有几秒读条,服务端这几秒在干一件具体的事(由 `PortalForcer` 负责),逻辑大致是:

```
1. 取玩家当前坐标 (x,y,z),按目标维度 scale 换算:
     若去往 scale 更大的维度(主世界→下界)→ 坐标 / 8
     反之(下界→主世界)→ 坐标 * 8
2. 在换算后的目标坐标周围一个搜索半径内(下界默认 128,主世界更大),
   查找已存在的下界传送门方块(缓存 + 扫描)
3. 找到 → 把玩家放到那个门,传送结束
4. 没找到 → 在目标坐标附近找一块合适的空间,
   现场用黑曜石 + 传送门方块「凭空建一个新门」(portal forcer),再放人
```

几个由此推出的真相:

- **门会"配错",是算法的必然而非 bug。** 第 4 步"没找到就新建"意味着:你在主世界建了 A、B 两个门,如果它们 /8 后落进下界的同一搜索半径,B 进去时会找到 A 已经建好的下界门 → 出来在错的地方。
- **想要稳定的一对一**:自己在下界手动建门,并让它的坐标 ≈ 主世界门坐标 / 8(y 自定),让算法第 2 步"恰好找到你建的那个",就不会乱跳。
- **门是按一组「门框 + 内部传送门方块」识别的**,新建时引擎会找一块能塞下门框的空间,所以有时你会发现下界半空中莫名多出一个平台门——那是 portal forcer 没找到地面、就近造出来的。

下界传送门用**黑曜石框 + 打火石点燃**;末地传送门是另一套(下面讲),两者机制不同。

## 四、末地:三种"门"各管一段路

末地最容易把人绕晕,因为它一口气有三种传送结构,名字都带 portal,但职责完全不同:

| 结构 | 长什么样 | 作用 |
| --- | --- | --- |
| **末地传送门 End Portal**(主世界要塞里) | 12 个末影之眼填满的门框 | 主世界 → 末地主岛,**单向入口** |
| **返回传送门 Exit Portal**(末地主岛中心) | 杀龙后激活的基岩结构,中间一颗龙蛋 | 末地主岛 → 主世界出生点,**回家** |
| **末地折跃门 End Gateway**(主岛外缘) | 悬空的小型基岩结构,一个发光点 | 主岛 → 外岛(远处),**去刷鞘翅/紫颂果** |

机制要点:

- **进末地是"造主岛 + 放龙"**:第一次有人进入末地维度时,世界生成器在主岛上摆好黑曜石柱、末影水晶,并 spawn 末影龙(它是一个有复杂 BlockState/碰撞与多段 AI 的 Boss 实体,呼应第 06 篇)。
- **杀龙激活 Exit Portal**:龙死亡触发返回传送门生成,站上去回主世界。**首次**杀龙还会在主岛边缘生成第一个 End Gateway。
- **End Gateway 是"定点跳跃"**:每个折跃门绑定一个目标坐标(主岛外约 1000 格的外岛区),踩进去就被弹射过去;它和下界门那种"按比例换算 + 搜索"完全不同,是**预设目标的硬传送**。

> 反直觉:末地的"外岛"理论上无限延伸,但你**只能靠折跃门跳过去**——主岛和外岛之间是一片虚空,没有连续地形,所以末地是"主岛 + 一圈靠 gateway 连接的孤岛群",不是一张连续大地图。

## 五、维度之间,什么共享、什么不共享

这是理解维度模型的最后一块,也是最容易想当然的地方:

| 东西 | 跨维度? | 说明 |
| --- | --- | --- |
| 玩家背包 / 经验 | ✅ 跟着玩家走 | 玩家实体本身被搬到目标维度 |
| 实体(怪物/掉落物/矿车) | ❌ 不能自由穿越 | 必须**显式传送**(走门、指令);各维度的实体各存各的 |
| chunk / 方块 | ❌ 完全独立 | 各维度独立存储:`DIM-1/`(下界)、`DIM1/`(末地),各自一套 region 文件(呼应第 07 篇) |
| 世界时间 / 天气 | 主要挂在主世界 | 下界末地无昼夜天空概念 |
| 出生点 / 重生 | 跨维度规则特殊 | 床只在 `bed_works=true` 的维度有效;下界用重生锚 |

所以"维度"在存储层就是**几套并列的世界目录**,在运行层是**几套并列的 `ServerLevel`**,各自独立 tick、独立加载区块(第 04/05 篇的那一整套机制,每个维度都跑一份)。它们之间没有共享的世界状态,**唯一的桥就是把实体从一个 `ServerLevel` 搬到另一个**——传送门、`/execute in`、末影珍珠落地,全都是这件事的不同触发方式。

---

## 串一下

把这一篇接回前面:维度类型是**数据驱动的环境配置**(第 11 篇动态注册表)、维度地形复用**第 16 篇的世界生成管线**、各维度**独立 chunk 存储**(第 07 篇)且**各跑一套 tick/加载**(第 04、05 篇)、末影龙是**复杂实体**(第 06 篇)。维度本身没有新机制,它只是把前面这些系统**并列实例化几份**,再用"换算坐标 + 找门/建门"这一小段传送逻辑把它们缝起来。

下界跑得快、传送门偶尔配错、末地一堆门——这三件看着神秘的事,拆开都只是 `coordinate_scale` 和 `PortalForcer` 的直接后果。
