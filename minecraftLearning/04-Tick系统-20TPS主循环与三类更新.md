# Tick 系统:20 TPS 主循环与三类更新

一个"无限大、处处可交互"的世界,凭什么能在一台普通机器上跑得动?如果每一帧都把世界里上亿个方块、成百上千个实体全算一遍,任何硬件都会瞬间跪下。Minecraft 给出的答案不是"算得更快",而是**算得更省**:用一个固定频率的逻辑循环驱动整个世界,并且把"哪些方块需要被更新"这件事拆成三套完全不同的机制,让绝大多数方块在绝大多数时间里**根本不被碰**。

这一篇讲清楚:世界是怎么"动"起来的,以及为什么你看到的水流、作物生长、红石信号,背后是三种截然不同的更新路径。

> 一句话先记住:**Minecraft 世界按固定 20 tick/秒推进;方块更新分三类——计划刻(排队的延迟任务)、随机刻(每区块抽样的缓慢演化)、邻居通知(事件驱动的即时级联),三者各管一摊,合起来才让"省"成为可能。**

---

## 一、基本节奏:20 TPS,和帧率没关系

游戏世界以固定 **20 tick/秒** 推进,一个 tick 名义上 **50 毫秒**。这里第一个反直觉点:

> **逻辑时钟(TPS)和渲染时钟(FPS)是两套独立的东西。**

- 你的显卡可以跑 144 FPS,但游戏逻辑永远只按 20 TPS 走。
- 渲染之所以看起来流畅,是客户端在两个 tick 状态之间做**插值(interpolation)**,把 50ms 的逻辑步长"抹"成连续画面。
- 所以"掉帧"和"掉 TPS"是两码事:前者是画面卡,后者是**整个世界变慢**——怪物动得慢、作物长得慢、红石变迟钝。

一个 tick 名义上要在 50ms 内做完一整轮世界更新。做完了,就睡到下一个 50ms 边界;做不完,麻烦就来了。

---

## 二、主循环:一个时间累加器

服务端核心是一个 `while (running)` 死循环(`MinecraftServer.runServer` / `tickServer`),计时逻辑本质是一个**时间累加器**:

```text
long lastTime = 系统毫秒
while (running) {
    long now = 系统毫秒
    accumulator += (now - lastTime)   // 累计"欠"了多少时间
    lastTime = now

    while (accumulator >= 50) {        // 每欠够 50ms,就跑一个 tick
        accumulator -= 50
        tick()                          // 跑一整轮世界更新
    }
    睡到下一个 50ms 边界
}
```

关键推论:

- **单 tick 算不完会"拖时"。** 如果某次 `tick()` 耗了 80ms,下一轮循环发现 `accumulator` 已经欠了不止一个 tick,于是会**连跑多个 tick 追赶**。
- **但追赶有上限。** 当欠账太多(累计超过约 2000ms,即落后约 40 个 tick),服务端会**直接丢弃**这部分时间并打印:

  ```text
  Can't keep up! Running 2150ms (43 ticks) behind, skipping 43 tick(s)
  ```

- 所以 **TPS 掉到 15** 的真实含义不是"时间被放慢了",而是**单个 tick 里要干的活超过了 50ms 预算,导致追不上墙钟时间**。世界整体变慢,是这个的副作用。

> 1.21 引入的 `/tick freeze`、`/tick rate`、`/tick step`,本质就是直接干预上面这个累加器和目标步长——冻结即停止推进 `accumulator`,变速即修改那个 `50`。

## 三、一个 tick 里到底做了什么

每个已加载维度(主世界 / 下界 / 末地)每 tick 大致按这个顺序走一遍:

```text
1. 世界边界、天气、时间推进
2. tickPending     —— 处理到期的 scheduled block tick 和 fluid tick(两个独立队列)
3. 遍历需 tick 的 chunk:
     · random tick(随机刻)
     · 冰雪结冻、雷击、刷怪(mob spawning)
4. tickBlockEntities —— 遍历所有方块实体(漏斗/熔炉/刷怪笼…)逐个调 ticker
5. 实体 tick         —— 移动、AI、碰撞、计时
6. 区块加载 / 卸载、ticket 过期处理
7. 网络:把本 tick 的变更打包发给客户端
```

注意第 4 步是**线性遍历所有方块实体**的。这就是为什么**漏斗一多就卡**:每个漏斗每隔几 tick 都要做容器检查与物品搬运,数量上去后直接吃满 tick 预算。

下面三节,分别拆开第 2、3 步里那三类截然不同的更新机制。

---

## 四、Scheduled Tick(计划刻):排队的延迟任务

很多方块行为是"现在登记,未来某 tick 再执行"。方块**主动注册**一个计划任务,这就是 scheduled tick(也叫 block tick)。

每个世界维护一个计划刻列表(`LevelTicks`,旧称 `ServerTickList`),每条记录概念上是:

```text
{ pos: 方块坐标, type: 方块/流体种类, triggerTick: 第几个 tick 执行, priority: 优先级 }
```

- 内部按 `(triggerTick, priority)` 排序;每 tick 弹出所有"到期且属于已加载 chunk"的项执行。
- **优先级 TickPriority** 取值 -3 ~ +3,用来打破"同一 tick 内谁先谁后"的平局——某些红石元件靠它保证确定性行为。
- **每 tick 处理上限约 65536 条**:超出的留到下一 tick,防止一次性爆掉。

### 水流扩散:计划刻的完整例子

水有 **level 0–7** 的流动等级(源是特殊标记,流动水离源越远 level 越大)。放下一个水源后发生的事:

```text
水源放下
  → 给四个水平邻居 + 正下方各排一个 fluid tick(主世界水延迟 5 tick)
  → 到期:邻居变成 level=1 的流动水,并继续向外排下一圈(再 +5 tick)
  → 到期:level=2 …… 一圈一圈推进
  → 遇到向下:优先向下流,且竖直水柱 level 不衰减(从上灌到底都是满的)
  → 寻路找洞:有一段逻辑会朝"最近能向下掉"的方向优先铺,所以水会绕着流向坑
  → 到 level 7 仍未到边界则停止
```

所以你肉眼看到的"水慢慢摊开",本质是**一圈一圈的计划刻级联**,每圈带 5 tick 的视觉延迟。岩浆同理但延迟更长(主世界 30 tick、下界 10 tick),所以岩浆流得明显慢。红石中继器的延迟、TNT 引信、活板门关闭,走的也都是这套队列。

## 五、Random Tick(随机刻):抽样驱动的缓慢演化

环境的"慢慢变化"——作物生长、树苗长树、草蔓延、树叶腐烂、冰雪融化——来自随机刻。它的算法是整套 tick 系统里**最容易被误解**的部分。

对每个**正在 tick 的 chunk**,遍历它的每个 16×16×16 的 **section**,做 `randomTickSpeed` 次(游戏规则,**默认 3**):

```text
重复 randomTickSpeed 次:
    int r = random.nextInt()          // 取一个随机数
    int x =  r        & 15
    int y = (r >> 8)  & 15
    int z = (r >> 16) & 15            // 用位运算从 4096 格里挑一个坐标
    BlockState s = section.getBlockState(x, y, z)
    if (s.isRandomlyTicking())        // 只有"关心随机刻"的方块才有反应
        s.randomTick(...)
```

几个要点,每一个都很反直觉:

- **抽中但方块不在乎随机刻(石头、泥土)→ 什么都不做。** 这次抽样就浪费了。所以一个 section 里随机刻方块越稀疏,单株被点到的概率越低、生长越慢。
- **单株作物的成熟时间是概率分布,不是定值。** 某株作物每 tick 被选中的概率约 `3 / 4096`,期望约 1365 tick(~68 秒)被点一次——但这是**几何分布**,方差很大。这正是"同一块地里的麦子,有的先熟有的后熟"的根本原因:纯粹看它什么时候被随机点到。
- **`randomTickSpeed` 调高是线性加速**:调到几千,植物秒长——但 CPU 负担同步线性上升,而且随机刻还驱动着树叶腐烂、火焰蔓延等,调太高会引发连锁负载甚至卡服。

```text
┌─ section (16³ = 4096 格) ──────────────┐
│  每 tick 随机点名 3 个格子(默认)        │
│     · 点到石头 → 浪费                    │
│     · 点到麦子 → 生长一阶段              │
│  → 所以"被点到"是个掷骰子事件,           │
│    地块成熟参差不齐是必然,不是 bug       │
└────────────────────────────────────────┘
```

## 六、Block Update(邻居通知):事件驱动的即时级联

前两类是"排队"和"抽样",第三类完全不同:它是**事件驱动、当 tick 就传播**的。

当一个方块状态改变时,会**立即同步通知周围方块**"我变了,你重算一下"。引擎里其实分两支:

| 机制 | 触发 | 典型场景 |
| --- | --- | --- |
| `neighborChanged`(邻居更新)| 邻居方块**状态**变了 | 红石信号、铁轨、活塞响应、BUD |
| `updateShape`(形状更新)| 邻居**形状/朝向**变了 | 楼梯/栅栏调整连接外观;草、火把失去附着方块而掉落 |

这套机制的特点是**会级联**:

```text
A 被改变
  → 通知邻居 B 重算
     → B 状态变了
        → 再通知 C
           → …… 当前 tick 内一路传播下去
```

红石电路就是利用这种级联来传播信号;沙子/沙砾检测到下方变空气而开始下落、活塞被供能伸出,也都走这条路。设计不当时会引发"更新风暴"——一次改动触发海量连锁更新,瞬间吃满 tick。

`world.setBlock(...)` 默认带一组 **flag 位**,控制这次放置要不要通知邻居、要不要重渲染、要不要触发 observer。这些 flag 正是各种红石技术(尤其"无更新放置")的底层抓手。

### 顺带:两个"特性级" bug

- **quasi-connectivity(准连接)**:活塞、漏斗等设备不仅检查自身是否被供能,**还检查正上方一格**是否被供能(历史遗留逻辑)。这导致"隔空供能"现象,大量红石机器依赖它。Java 版保留、基岩版没有——这是两版红石电路不通用的主因之一。
- **0-tick**:利用"同一 tick 内更新顺序 + 计划刻尚未结算"的时间窗,在一个 tick 内对作物做多次生长触发,或让活塞零延迟伸缩。本质是钻 update 顺序的空子,部分已被官方修掉。

## 七、三类更新对照

| 维度 | 计划刻 Scheduled | 随机刻 Random | 邻居通知 Block Update |
| --- | --- | --- | --- |
| 触发方式 | 主动注册、到期执行 | 每 section 每 tick 抽样 | 方块改变时事件触发 |
| 时机 | 未来某 tick | 当前 tick(被抽中)| 当前 tick(同步级联)|
| 数据结构 | 按时间排序的队列 | 位运算随机抽坐标 | 直接调用、可递归 |
| 典型用途 | 水流/岩浆/红石延迟/TNT | 作物/树/草/腐烂/融冰 | 红石/沙子下落/活塞 |
| 开销特征 | 只发生在登记过的方块 | 与加载体积线性相关 | 可级联成更新风暴 |

**反直觉小结:**

- 水"慢慢流"不是动画,是计划刻一圈圈排队(每圈 5 tick)。
- 作物"参差成熟"不是 bug,是随机抽样的几何分布。
- 红石"瞬间响应"是当 tick 同步级联,和前两者完全不是一套机制。
- TPS 掉了不是"时间被调慢",是单 tick 算不完追不上墙钟。

---

## 八、串联

本篇讲的是"世界怎么动"。但有一个前提没展开:**哪些 chunk 才会被 tick?** random tick、刷怪、流体更新只发生在"正在模拟"的区块里——这由 chunk 的 ticket 等级和 simulation distance 决定,也正是"为什么你一走远,远处的农场和红石机器就停摆"的答案。这部分留到下一篇《区块的生命:加载、ticket 等级与模拟距离》。

而第 4 步遍历的实体 tick(移动、AABB 碰撞、A\* 寻路、AI),是 tick 预算里最贵、最容易拖垮 TPS 的一块,单独放到《实体系统》篇展开。

把本篇接回贯穿全系列的主线——**"能算不存、能享不复制"**:tick 系统是这条主线在"时间维度"上的体现。它没有为每个方块维护一个"下次更新时间",而是用**计划刻队列(只为需要的方块排队)+ 随机刻抽样(用概率代替逐个轮询)** 两招,把"遍历整个世界"这件不可能的事,变成了"只碰极少数真正在变化的方块"。省,从来都是设计出来的。
