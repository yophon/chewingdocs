# 存储(上):Region 与 Anvil 文件格式逐字节

Minecraft 的世界是"无限"的。你可以朝一个方向一直走,地形永远在你面前生成,从不重复也从不停下。这就引出一个让很多人第一次听到时会愣一下的问题:**一个无限大的世界,凭什么没把你的硬盘撑爆?**

答案分两半。一半是"很多东西根本不存"——这是上一层(地形生成)的事;另一半是"要存的那部分,用一种非常讲究的文件格式存"——这就是本篇要逐字节拆开的 **Region / Anvil 格式**。

> 一句话先记住:**地形不存,靠种子+噪声现算;只有被玩家改动过的区块才落盘,而且不是一个区块一个文件,而是 1024 个区块打包进一个对齐到 4KB 扇区、带两张索引头表的 `.mca` 文件。**

---

## 一、为什么无限世界不撑爆硬盘

回到生成那一层的核心结论:**地形由"种子 + 噪声函数"确定性生成**。同一个种子、同一个坐标,噪声函数永远输出同一个值,所以地形不需要存——走到哪、算到哪。

这带来一个关键推论:

> 磁盘上**只存玩家访问过、且被修改过的区块(chunk)**。没去过的地方不占一个字节;去过但没动过的(理论上)也能靠重新生成还原,实际为了加载速度会缓存下来,但本质上"可被算出来的数据不需要持久化"。

所以"无限世界"占用的磁盘,正比于**你实际探索并改动过的范围**,而不是世界的理论大小。一个玩了几百小时的存档可能也就几个 GB,而不是无穷大。这是 Minecraft 存储设计的第一性原理,后面所有格式细节都是为它服务的。

---

## 二、为什么要打包成 Region 文件

确定了"只存改过的 chunk",下一个问题是:**这些 chunk 怎么落到文件系统上?**

最朴素的想法是"一个 chunk 一个文件"。这个方案会立刻撞墙:

- 一个稍微探索过的世界,轻松有**几百万个 chunk**。
- 几百万个小文件意味着几百万个 inode、巨大的目录项;文件系统遍历、备份、删除全部被拖到地上爬。
- 每个文件还有"块大小向上取整"的浪费(一个几百字节的 chunk 占满一个 4KB 文件系统块)。

所以 Minecraft 的选择是**打包**:把一片区域的 chunk 塞进**一个大文件**,文件内部自己做索引和空间管理。这个大文件就是 **Region 文件**。

```
一个 Region = 32 × 32 = 1024 个 chunk
文件路径:    world/region/r.<regionX>.<regionZ>.mca
其中 regionX = chunkX >> 5,  regionZ = chunkZ >> 5
```

也就是说,chunk 坐标右移 5 位(除以 32)就得到它所属的 region 坐标;一个 region 文件管辖 512×512 个方块的水平范围。

### McRegion → Anvil 的演进

- **McRegion**(`.mcr`):最早的打包格式,奠定了"region 文件 + 扇区 + 索引头"这套结构。
- **Anvil**(`.mca`):现行格式,是 McRegion 的升级版。主要改进是**把 chunk 纵向切成 16³ 的 section 存储**,空 section 不占空间——这正是后来世界高度能扩展到 384 格(1.18)而存储开销可控的基础。

本篇讲的字节布局以 **Anvil(.mca)** 为准。

## 三、.mca 文件逐字节拆解

整个文件的宏观结构只有两部分:**8KB 的头部** + **若干 4KB 的数据扇区**。

```
┌─────────────────────────────────────────────┐
│ 扇区 0 (0 ~ 4095):    Location table 位置表   │  4096 字节
├─────────────────────────────────────────────┤
│ 扇区 1 (4096 ~ 8191): Timestamp table 时间戳表 │  4096 字节
├─────────────────────────────────────────────┤
│ 扇区 2 起 (8192 ~):   chunk 数据区(按扇区对齐)│  N × 4096 字节
│   ...                                          │
└─────────────────────────────────────────────┘
```

**扇区(sector)是这个文件的最小分配单位,固定 4096 字节。** 所有 chunk 数据都按扇区对齐存放,这让"用 3 字节记一个偏移"成为可能(见下)。

### 头部 1:Location table(位置表,偏移 0,共 4096 字节)

1024 个条目,每个 **4 字节**,一一对应 region 内 1024 个 chunk 的局部坐标。索引公式:

```
index = (chunkX & 31) + (chunkZ & 31) * 32
字节偏移 = index * 4
```

每个 4 字节条目的布局:

| 字段 | 字节数 | 含义 |
| --- | --- | --- |
| offset | 3 字节(大端) | 该 chunk 数据从文件第几个**扇区**开始 |
| sectorCount | 1 字节 | 该 chunk 数据占用几个**扇区** |

要点:

- **offset 以扇区为单位**,不是字节。offset=2 表示数据从字节 `2 × 4096 = 8192` 开始(因为扇区 0、1 被两张头表占了,数据最早从扇区 2 起)。
- `offset = 0 且 sectorCount = 0` → 这个 chunk **尚未生成或尚未保存**,直接当不存在处理。
- sectorCount 只有 1 字节,所以单个 chunk 最多 **255 个扇区 ≈ 1020 KB ≈ 1 MB**。超过怎么办?见第五节。

### 头部 2:Timestamp table(时间戳表,偏移 4096,共 4096 字节)

同样 1024 个条目,每个 **4 字节**(大端整数),记录对应 chunk **最后一次写入的 Unix 时间戳(秒)**。索引方式和位置表完全一致。

这张表主要用于增量备份、调试和工具判断"哪些 chunk 最近被动过"。

### 数据扇区(偏移 8192 起)

知道了某个 chunk 的 offset,seek 到 `offset × 4096`,你看到的不是裸数据,而是这样一个小头:

```
┌──────────────┬──────────────────┬───────────────────────────┐
│ length (4字节)│ compression(1字节)│ 压缩后的 chunk 数据 (length-1)│
└──────────────┴──────────────────┴───────────────────────────┘
```

- **length(4 字节,大端)**:后续数据的长度(**包含**那 1 字节压缩类型,所以实际压缩数据是 `length - 1` 字节)。
- **compression type(1 字节)**:

| 值 | 压缩方式 | 说明 |
| --- | --- | --- |
| 1 | GZip | 历史遗留,几乎不用 |
| 2 | **Zlib / deflate** | **默认** |
| 3 | 不压缩 | 极少见 |
| 4 | LZ4 | 较新版本可选,解压更快,牺牲压缩率换 CPU |

读一个 chunk 的完整流程因此是:**查位置表 → seek 到扇区 → 读 length → 读压缩类型 → 读出 `length-1` 字节 → 解压 → 得到 chunk 的 NBT 树**(NBT 与里面的 Palette 是下一篇的主题)。

## 四、ASCII 总览:一次 chunk 读取

把上面三段串成一张图,直观感受"从坐标到字节"的全过程:

```
chunk (x=35, z=12)
        │
        ├─ region 坐标:  (35>>5, 12>>5) = (1, 0)  →  打开 r.1.0.mca
        │
        ├─ 局部索引:     (35&31)+(12&31)*32 = 3 + 12*32 = 387
        │                位置表字节偏移 = 387*4 = 1548
        │
        ├─ 读 4 字节:    offset=Sektor号, sectorCount=占几扇区
        │                例如 offset=2,sectorCount=1
        │
        ├─ seek 到 2*4096 = 8192
        │   ┌─ length      = 0x00000A31  (=2609)
        │   ├─ compression = 2 (zlib)
        │   └─ payload      = 2608 字节 zlib 数据
        │
        └─ zlib 解压 → chunk 的 NBT 根标签
```

## 五、踩坑与反直觉点

### 1. 单 chunk 上限 1MB,超了就外置

sectorCount 只有 1 字节(最多 255 扇区),所以一个 chunk 在 region 文件内最多占约 1MB。极端情况(比如一个塞满了海量物品的箱子群、或巨型方块实体数据)会超过这个上限。

处理方式:**当压缩类型字节的最高位被置 1**(例如 `0x82` = `0x80 | 2`),表示这是一个**超大 chunk**,真实数据不在本 region 文件里,而存到一个外置文件:

```
world/region/c.<chunkX>.<chunkZ>.mcc
```

region 文件内对应位置只留一个占位。解析时看到最高位置 1,就转去读 `.mcc`。

### 2. 写盘不是"原地改",而是"找空位重排"

一个 chunk 被改动后体积可能变大,原来占 1 个扇区现在要占 2 个,原位置塞不下了。此时存档逻辑会在文件里**寻找一段连续的空闲扇区**重新落位,然后更新位置表里的 offset 和 sectorCount。

后果是:长期增删探索会让 region 文件内部产生**碎片**(空洞夹在已用扇区之间)。这就是为什么有 **MCA Selector** 这类工具提供"优化 / 碎片整理"功能——本质是把扇区重新紧凑排布、回收空洞。

### 3. region 文件常常"看起来很大但很空"

因为按 4KB 扇区对齐分配,一个只改了几个方块的 chunk 也至少占 1 个扇区(4KB);加上 8KB 头部,一个只探索了边角的 region 文件也有几十 KB 起步。文件大小**不等于**有效数据量,别被 `ls -l` 吓到。

### 4. 别手撸"原子写"——存档损坏多半来自这里

位置表、数据扇区、时间戳表三者必须保持一致。如果写到一半进程被杀(掉电、强杀),可能出现位置表指向一个没写完的扇区,导致 chunk 损坏(经典的 "corrupted chunk")。这也是为什么直接拷贝正在运行的存档很危险——务必先停服或让世界保存完成。

## 六、存档目录里还有什么

Region 文件只是世界数据的主体,一个完整存档目录还包括:

| 路径 | 内容 |
| --- | --- |
| `region/` | 方块、section、方块实体等核心 chunk 数据(本篇主角) |
| `entities/` | **实体数据**(1.17 起从 chunk 里拆出来,独立 region 格式) |
| `poi/` | 兴趣点(Point of Interest,村民找工作站/床等用),独立 region 格式 |
| `level.dat` | 世界级元数据(种子、出生点、时间、游戏规则……),GZip 压缩的 NBT |
| `playerdata/<uuid>.dat` | 每个玩家的背包、坐标、状态(NBT) |
| `data/` | 地图(`map_*.dat`)、袭击、刷怪冷却等全局保存数据 |
| `DIM-1/` `DIM1/` | 下界、末地各自的 region 目录 |

注意 **1.17 把实体单独拆到 `entities/`** 这件事:把"方块数据"和"实体数据"分文件,好处是可以在不加载满功能 chunk 的情况下处理实体,也让两类数据各自演进。`poi/` 同理,是给村民 AI 用的独立索引层。

---

## 七、串联

这一篇是"存储"的上半场,它和系列里几篇紧密咬合:

- **承接"地形生成"那一层**:正因为地形能靠**种子 + 噪声确定性算出来**,磁盘才只需要存"改过的 chunk",Region 格式才有意义——存的是"差异",不是"全世界"。
- **交给下一篇(NBT 与 Palette)**:本篇把 chunk 数据读到了"一坨 zlib 解压后的字节",但**那坨字节内部是什么结构**——NBT 树、局部调色板、bit-packed 的方块数组——是下半场的主题。区块**内存里的 `PalettedContainer` 和磁盘上的 palette 几乎同构**,所以加载时近乎零转换,这正是 Minecraft 又快又省的关键一环。

把两篇连起来看:**Region 文件解决"chunk 这块数据放在硬盘哪里、怎么高效寻址",NBT/Palette 解决"这块数据内部怎么紧凑表达"。** 一个是外层的寻址与空间管理,一个是内层的结构与压缩,合起来才是 Minecraft 完整的持久化故事。
