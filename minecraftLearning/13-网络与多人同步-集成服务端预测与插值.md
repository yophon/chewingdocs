# 网络与多人同步:集成服务端、预测与插值

先说一个会颠覆很多人认知的事实:**你玩单机 Minecraft,本机也跑着一个完整的服务端。**

不是比喻。Java 版客户端在你点"创建世界"时,会在进程内启动一个 `IntegratedServer`(集成服务端),然后客户端通过一条**内存里的假连接**(loopback / `MemoryConnection`)连上它。世界生成、方块更新、实体 AI、刷怪、存盘——全发生在这个集成服务端里。客户端那一侧,几乎只负责**渲染 + 收发包 + 一点点预测**。

> 一句话先记住:**Minecraft 没有"单机模式"的特殊代码路径。单机 = 本机集成服务端 + loopback 客户端;联机 = 远程专用服务端 + TCP 客户端。两者跑的是同一套逻辑,区别只是连接的两端隔多远。** 理解了这条,你就理解了为什么客户端永远是"带预测的显示器",而不是世界本身。

---

## 为什么要这么设计:单机也走 C/S

直觉上,单机游戏直接在一个进程里跑逻辑+渲染最简单,何必拆成客户端连服务端?Mojang 选择"单机也走客户端/服务端(C/S)"是一个**省维护成本**的决定:

- 只有一套世界逻辑代码。如果单机和联机各写一套,红石、刷怪、tick 的行为迟早会漂移。
- 单机里调通的机制,搬到联机几乎零改动——它本来就在"服务端"里跑。
- 代价:即使单机,玩家的输入也要打包"发"给本机服务端,世界状态再"发"回来渲染。loopback 连接没有网络延迟、不走真正的 socket(直接传对象引用),所以这点开销基本可忽略。

```
单机:
  ┌─────────────── 一个进程 ───────────────┐
  │  Client(渲染/输入)  ⇄ loopback ⇄  IntegratedServer(世界逻辑/存盘) │
  └────────────────────────────────────────┘

联机:
  Client(渲染/输入)  ⇄══ TCP ══⇄  DedicatedServer(世界逻辑/存盘)
       你的机器                         远程机器
```

**客户端永远不持有"真相"。** 它持有的是服务端发来的一份**副本 + 自己猜的一点未来**。这是后面所有"为什么会橡皮筋""为什么别人动得这么顺"的总根源。

---

## 服务端权威:客户端只能"请求",不能"决定"

Minecraft 是**服务端权威(authoritative server)** 模型:

| 谁 | 能做什么 |
| --- | --- |
| 客户端 | 发送**意图**:我按了前进、我左键挖这个方块、我把物品拖到这个槽 |
| 服务端 | **裁决**:你能不能动到那、那方块该不该破、那次交易合不合法,然后把**结果**广播给所有相关客户端 |

举例:你左键一个石头。客户端不会自己把石头删掉,它发一个"开始挖掘 / 挖掘完成"的包给服务端;服务端校验(你够得着吗?手里工具对吗?保护区允许吗?)通过后,**服务端**把那格设为空气,再通过方块更新包告诉所有看得见这块的客户端。

这就是反作弊的底座:外挂能伪造客户端发什么包,但**改不了服务端的裁决**。一个"瞬移外挂"发一堆移动包,服务端发现你单 tick 位移超阈值,直接把你**拉回**合法位置(就是橡皮筋)。当然,服务端的校验并不完美(早期很多移动校验很松),但模型方向是对的:**不信任客户端**。

## 连接的一生:协议是分阶段的状态机

很多人以为"联网"就是一条管子一直发同一种包。实际上 Minecraft 协议是个**状态机**,连接会依次切换状态,**每个状态有自己独立的一套包编号**。同一个字节 `0x00` 在不同状态里是完全不同的包。

```
握手 Handshake
   ├──→ Status   (服务器列表里那一行:延迟、在线人数、MOTD)→ 断开
   └──→ Login    (鉴权、加密、压缩协商、分配 UUID)
            ↓
        Configuration   ← 1.20.2 新增的独立阶段
            ↓
          Play           (真正进游戏后,绝大多数包都在这)
```

| 阶段 | 干什么 | 关键点 |
| --- | --- | --- |
| **Handshake** | 客户端报一句"我要去 Status 还是 Login" | 决定后续走哪条路 |
| **Status** | 拉服务器信息渲染那一行 | 多人界面的"ping 一下",不进世界 |
| **Login** | 正版鉴权(对 Mojang/微软验证 session)、开启加密、协商压缩阈值、确定玩家 UUID | 加密密钥在这建立 |
| **Configuration** | **同步注册表、数据包、维度信息、服务器资源包**,玩家此时还没进世界 | 见下 |
| **Play** | 玩家在世界里,移动/方块/实体/聊天/容器全在这 | 包量最大 |

### Configuration 阶段为什么单列出来

1.20.2 之前,注册表同步这些"进场前的准备"是塞在 Play 阶段开头硬办的,导致**重连、换维度、重载数据包**时很别扭(得把人踢回去重来)。1.20.2 把它拆成独立的 **Configuration 阶段**:

- 这里服务端把**注册表**发给客户端——还记得第 11 篇讲的 registry sync 吗?就在这一步。服务端说"我的 1 号方块是 minecraft:stone",客户端据此对齐编号。**对不上(版本/mod 不一致)就在这阶段失败,根本进不了 Play。**
- 动态注册表(生物群系、维度类型、由数据包定义的内容)也在这同步。
- 好处:之后想重新配置(比如服务端换了数据包),可以把客户端**从 Play 切回 Configuration** 再切回来,不用断线重连。

## 进了世界之后:服务端到底发什么

Play 阶段的包大致分几类,理解它们就理解了"客户端看到的世界是怎么拼出来的":

| 包类(方向)| 内容 |
| --- | --- |
| 区块数据(S→C)| 一个 chunk 的方块(palette 压缩的 section)+ 光照,一次性发过来让客户端建 mesh |
| 方块更新(S→C)| 单格变化 / 一个 section 批量变化,增量打补丁 |
| 实体生成/移动(S→C)| 实体出现、销毁,以及**位置增量** |
| 实体元数据(S→C)| 血量、姿势、燃烧、命名等状态变化 |
| 玩家输入(C→S)| 移动、视角、挖掘、使用、攻击、切换物品 |
| 容器/菜单(双向)| 打开容器、点击槽位、确认事务 |

### 区块与光照:为什么进新区域会"卡一下糊一下"

你走到没加载过的地方,服务端把那批 chunk 的方块和光照打包发来。客户端收到后还要**在本地建 mesh**(第 3 篇讲的"只画看得见的面")。所以新区域有两步开销:**网络收包 + 本地建网格**。区块多、网慢,或者本地建 mesh 排队,就表现为地形一块块"长"出来。

注意:**光照值也是服务端算好发过来的**(第 10 篇的双通道光照),客户端默认不重算——这样大家看到的明暗一致。

### 实体移动:你看到的"别人",是插出来的

这是最值得讲清的反直觉点。服务端 20 TPS,每 50ms 才更新一次实体位置,而且为了省带宽,发的还是**增量**(相对上次的位移),甚至不是每 tick 都发。如果客户端"收到一个位置就把实体瞬移过去",你会看到所有人都在**抽搐式跳格**(每 50ms 跳一下)。

所以客户端做**插值(interpolation)**:

```
服务端发来的离散位置:   t0 ●            t1 ●            t2 ●
客户端实际渲染(60fps): ●·····●·····●·····●·····●·····●·····●
                        └── 在两个收到的位置之间平滑补帧 ──┘
```

客户端把收到的目标位置当"我要在接下来几帧里滑过去的终点",于是别人的移动看起来丝滑。**代价:你看到的别人,永远是过去某一刻的位置 + 一段猜测,不是此刻服务端的真实位置。** PVP 里这点很要命——你瞄准的是对方"显示出来"的身位,服务端判定却用它那边的真实坐标,延迟越高错位越大。

### 你自己:预测 + 回正(橡皮筋)

对**别人**插值,对**自己**则不能——你按前进键,总不能等一个网络往返(发包→服务端→收回)才开始动,那操作会有明显延迟。所以客户端对自己做**预测(client-side prediction)**:

1. 你按前进,客户端**立刻**让你向前移动并渲染(本地先行)。
2. 同时把这次输入发给服务端。
3. 服务端用它那边的世界裁决你的真实位置,发回"权威坐标"。
4. 客户端对比:如果和自己预测的一致(绝大多数情况),无感;如果**对不上**(你预测穿墙了、服务端那有个你不知道的方块、网络丢包),服务端的坐标覆盖你的预测——你被**拽回去**。

这就是**橡皮筋(rubberbanding)**:本质是"客户端预测错了,被服务端权威回正"。卡顿、丢包、或服务端 TPS 掉时最明显,因为预测和权威差得越来越远。

## tick(20)与包频率:不是一回事

| 时钟 | 频率 | 谁的 |
| --- | --- | --- |
| 服务端 tick | 20 TPS(50ms)| 世界逻辑推进的节拍 |
| 渲染帧 | 你的显示器能跑多少(60/144…)| 纯客户端,和服务端无关 |
| 包发送 | 跟着 tick,但不是每个状态每 tick 都发 | 按需 + 增量,省带宽 |

服务端逻辑永远 20 TPS。但**包不会傻发**:实体没动就不发它的移动包;位置增量小用短包,大跳变才发全量。客户端 60+ fps 渲染靠插值/预测把 20 TPS 的离散状态"抹"成连续画面。

**所以服务端掉 TPS 时会发生什么?** 还记得第 4 篇:服务端单 tick 算不完就拖时。它一拖,**发包也跟着稀疏**——实体更新变慢、方块变化延迟到达。客户端这边帧率可能还是 144fps 很流畅,但世界"内容"卡住了:怪物顿挫、你打方块半天不破、橡皮筋频发。**FPS 高 ≠ 不卡,卡的是服务端的 tick,不是你的显卡。** 这也是为什么联机体验差时该去看服务端 MSPT,而不是本地帧数。

## 方块更新和容器:也全是包

别忘了世界里"非移动"的交互同样走这套:

- **方块更新**:服务端裁决后发方块变化包,客户端打补丁并重建受影响 chunk 的 mesh。你放下一个方块时,客户端可能也做了一点本地预测(立刻显示),但仍以服务端结果为准。
- **容器/菜单**:第 11 篇讲过,打开箱子时服务端临时建一个 Menu,把槽位**同步**给客户端。你拖动物品其实是发"点击槽位"的包,服务端裁决物品怎么移动再同步回来——这里历史上有一套"事务确认"机制防止客户端和服务端的背包状态对不上(对不上就强制重发整个容器)。所以**容器本质也是一段服务端权威、客户端显示的同步会话**,和移动是同一个哲学。

## 踩坑与反直觉

- **"单机没有网络问题"是错的**:单机也走 C/S,只是 loopback 没延迟。所以单机里 TPS 掉了,你照样会看到怪物顿挫——因为客户端依旧在等服务端的包,只不过服务端就在隔壁内存里。
- **别人移动顺滑 ≠ 网络好**:那是插值在帮忙。真实延迟体现在"你的动作多久被世界确认"(打人、开门、橡皮筋),不体现在别人滑得顺不顺。
- **高 FPS 不能掩盖低 TPS**:两套时钟。渲染流畅但世界卡,几乎一定是服务端 tick 的问题。
- **改了客户端也改不了世界**:服务端权威。客户端 mod 能改你"看到"什么、怎么渲染,但方块、实体、物品的真相在服务端,联机时由服务端说了算。
- **进服卡在"正在加入世界"**:多半是 Configuration 阶段注册表/数据包对不齐(版本或 mod 不一致),不是网慢。

## 串联

这一篇把前面所有"服务端内部"的内容接到了玩家屏幕上:第 4 篇的 20 TPS 决定了包的节拍和掉 TPS 时的体验;第 3 篇的区块 mesh 是在客户端收到区块包后才建的;第 10 篇的光照是服务端算好发来的;第 11 篇的注册表同步发生在 Configuration 阶段、菜单会话靠包驱动。一句话收口:**服务端是唯一的"世界",客户端是它一个带预测的、略微滞后的影子。** 你玩的从来不是本地那份画面,而是服务端那份状态的转播。
