# 指令、函数与谓词系统

前面二十篇讲的都是"世界是什么、怎么存、怎么跑"——方块、实体、区块、存档、注册表。但有一层我们一直绕着走:**普通人不写一行 Java,怎么往这套数据模型上加逻辑?** 答案就是指令 + 数据包。这是 Minecraft 的"脚本/逻辑层",建在前面所有数据模型之上,也是地图作者、服务器、模组之外最大的一块可玩性来源。

> 一句话先记住:**指令系统不是"作弊码集合",而是一套对世界数据模型的结构化读写 API;Brigadier 把每条命令变成一棵带类型的参数树,mcfunction 把命令打包成脚本,谓词/战利品表把"判定"和"产出"抽成可复用的 JSON。整层不碰 Java,纯数据驱动。**

---

## 一、Brigadier:命令是一棵树,不是一个字符串

新手以为 `/give @p diamond 64` 是"解析一个字符串"。错。从 1.13 起,Minecraft 的命令引擎是独立库 **Brigadier**,它把所有命令建模成**一棵带类型参数的节点树**。

```
根
├── give
│    └── <targets: 实体选择器>
│         └── <item: 物品参数>
│              └── <count: 整数 0..>   ← 可选,缺省走上一层的 executes
├── tp / teleport
│    └── ...
└── execute
     ├── as <targets> → (回到 execute 子节点)
     ├── at <targets> → ...
     └── run → (整棵命令树作为子命令)
```

每个节点有三种:**literal**(字面量,如 `give`)、**argument**(带类型解析器,如 `targets`/`item`/`count`)、以及 redirect(指向别的子树,`execute` 的递归就靠它)。这套设计带来几个直接好处:

| 能力 | 怎么来的 |
| --- | --- |
| **类型化解析** | 每个 argument 节点自带解析器,`count` 解析成 int 而不是字符串,越界当场报错并指出列位置 |
| **tab 补全** | 节点知道自己接受什么,补全就是"问当前节点:你的下一步候选有哪些" |
| **权限校验** | 每个节点挂一个 `requires` 谓词(如 OP 等级 ≥ 2),不满足时这棵子树对你**不可见**——既是鉴权也是补全过滤 |
| **错误定位** | 解析在树上逐节点推进,失败点就是光标该停的地方 |

**反直觉点:命令树会同步给客户端。** 你按 tab 能即时补全、输错立刻变红,不是客户端瞎猜,而是**服务端在握手后把整棵命令树打包发给客户端**(呼应第 13 篇:这走 play 阶段的一个包)。客户端用这棵树做本地补全和高亮,真正执行时再发回服务端。所以装了命令的服务端,客户端补全会随之变化;反作弊也据此知道你"理论上能打哪些命令"。

## 二、选择器:对实体表的一次带过滤查询

`@e`、`@a`、`@p`、`@s`、`@r` 不是"魔法变量",本质是**对世界实体集合的一次查询**,返回一个目标列表:

| 选择器 | 含义 | 默认排序/数量 |
| --- | --- | --- |
| `@s` | 命令执行者自己 | 1 个,最便宜 |
| `@p` | 最近的玩家 | 按距离,1 个 |
| `@r` | 随机玩家 | 随机 1 个 |
| `@a` | 所有玩家 | 不限 |
| `@e` | 所有实体 | 不限,**最贵** |

真正的威力在**参数过滤**,它就是查询的 where 子句:

```
@e[type=zombie, distance=..16, nbt={IsBaby:0b}, limit=5, sort=nearest]
@a[gamemode=survival, scores={kills=10..}, tag=red]
@e[type=item, predicate=mypack:is_diamond]
```

- `type` / `distance` / `gamemode` / `scores` / `tag`:常见维度过滤。
- `nbt={...}`:直接匹配实体的 NBT 子树(呼应第 8 篇:实体就是一棵 NBT)。
- **物品/组件匹配**:1.20.5 物品迁到数据组件后(呼应第 17 篇),很多匹配从"翻 NBT tag"变成"匹配组件",`predicate` 和物品判定更精确、更强类型。

性能上有个该记的取舍:**`@e` 不带 `type`/`limit` 是会扫全实体表的**,大地图里滥用 `@e[...]` 是 mcfunction 卡顿的常见来源。能用 `@s` 就不用 `@p`,能 `type=` 限定就别裸 `@e`。

### execute:把"上下文"拆成可组合的子命令

`execute` 是整个命令系统的关节,它不直接做事,而是**改变后续命令的执行上下文**,再 `run` 一条真正的命令:

```
execute as @a[tag=player] at @s if block ~ ~-1 ~ minecraft:gold_block run give @s diamond
        └ 把执行者换成每个玩家  └ 把坐标锚到他脚下  └ 条件:脚下是金块  └ 满足才执行
```

- `as`:换**执行者**(`@s` 指向谁)。
- `at` / `positioned` / `rotated`:换**坐标/朝向锚点**(`~ ~ ~` 相对谁)。
- `if` / `unless`:**条件分支**,接 `block`/`entity`/`score`/`predicate` 等。
- `store`:把结果**写回**计分板或 NBT,实现"取值"。

`as @a ... run` 会对每个匹配目标**各跑一遍**——这就是"对一群玩家批量操作"的标准写法,也是为什么一条 `execute` 能展开成上千次执行、需要小心成本。

## 三、mcfunction:把命令打包成脚本,并挂进 tick 循环

单条命令是交互式的。把一堆命令按行写进一个 `.mcfunction` 文件,就成了**函数**——一个最朴素的脚本:逐行顺序执行,无变量、无返回值(早期)、靠计分板和 NBT 当"内存"。

```
# mypack:tick_logic  (data/mypack/functions/tick_logic.mcfunction)
execute as @a at @s if block ~ ~-1 ~ minecraft:magma_block run damage @s 1 minecraft:in_fire
scoreboard players add @a timer 1
execute if score #global timer matches 200 run function mypack:every_10s
```

关键不是语法,而是**怎么让它自动跑**。这就用到**函数标签**(呼应第 11 篇的 Tag 机制):

| 函数标签 | 触发时机 |
| --- | --- |
| `#minecraft:load` | 世界/数据包加载或 `/reload` 时跑一次——初始化计分板、常量 |
| `#minecraft:tick` | **每个游戏 tick 跑一次**——把你的逻辑钉进 20 TPS 主循环 |

把你的函数加进 `#minecraft:tick` 标签,它就成了世界 tick 的一部分(呼应第 4 篇:和方块更新、实体 tick 一起,每 50ms 跑一轮)。**这是整个数据包能"活起来"的开关**——没有它,函数只能被手动或被别的命令调用。

代价也很直接:**`#minecraft:tick` 里的东西每秒跑 20 次**,一条扫全图的 `@e` 选择器放这儿就是稳定的性能黑洞。成熟数据包都会用计分板做"每 N tick 才跑一次"的节流,把重活从每 tick 摊开。

## 四、谓词、战利品表、进度、配方:把"判定"和"产出"抽成数据

命令是"动作",但 Minecraft 还把另外两类东西抽成了**纯 JSON、可复用**的定义,这是数据驱动的精髓:

| 定义类型 | 回答的问题 | 谁来用 |
| --- | --- | --- |
| **谓词 predicate** | "当前情况满足条件吗?"(实体类型、位置、天气、手持物、随机数…) | `execute if predicate`、战利品条件、选择器 `predicate=` |
| **战利品表 loot table** | "该掉/给出哪些物品?"(带权重、函数、条件) | 箱子、怪物掉落、钓鱼、`/loot`、方块破坏 |
| **进度 advancement** | "玩家达成了什么?触发什么奖励?" | 成就树、隐藏触发器当事件钩子 |
| **配方 recipe** | "什么输入合成什么输出?" | 工作台/熔炉/锻造台匹配 |

它们的共同点是**判定/产出与代码解耦**:

- **谓词可复用、可组合**:一个 `is_in_rain` 谓词,既能被 `execute if predicate` 当命令条件,又能被战利品表当掉落条件,还能被选择器引用。写一次,处处复用。
- **战利品表是"掉落"的唯一真相**:回到第 9 篇讲过的——地牢箱子不预存物品,只存 `LootTable` 引用 + 种子,首次打开时按这张表现算。怪物掉什么、钓上来什么,全是战利品表。改掉落不改代码,改 JSON。
- **进度常被当"事件系统"用**:它的触发器(`minecraft:player_killed_entity`、`minecraft:inventory_changed`…)本质是"当某事发生"的钩子,地图作者用隐藏进度 + reward function 实现"杀怪触发剧情"这类事件逻辑——一个被玩出新用途的设计。

## 五、把这一层放回全局

退一步看,这一整层和前面的关系很清晰:

```
       指令 / mcfunction / 谓词 / 战利品表 / 进度 / 配方   ← 逻辑·脚本层(本篇,纯数据驱动)
                          ↑ 读写
       注册表 · 标签 · 动态注册表 · 数据包                  ← 第 11 篇:内容定义与元系统
                          ↑ 基于
       方块 · 实体 · 物品(组件)· NBT · 计分板 · 世界状态   ← 第 6/8/9/17 篇:数据模型
                          ↑ 跑在
       20 TPS Tick 循环(#minecraft:tick 把函数挂进来)     ← 第 4 篇
```

几个值得记住的取舍与反直觉:

- **它不写 Java,却图灵完备到能做小游戏、剧情、自定义机制**。代价是没有真正的变量和数据结构,一切靠计分板(整数)、NBT 存储、宏(macro,较新版本)硬凑——表达力换来了"零编译、改完 `/reload` 就生效、可热分发"。
- **数据包和模组是两条路**:模组改 Java 字节码、能加新方块新实体;数据包只能"在现有积木上重组逻辑和数据"。但数据包**纯数据、跨版本相对稳、服务端单方面下发即可**(呼应第 11 篇 registry/数据同步),所以原版地图、商业服玩法大量用它而不是模组。
- **命令的成本是真实的**。`@e` 扫表、`execute as` 展开、`#minecraft:tick` 的每秒 20 次,叠起来就是服务器 MSPT 的一部分。把这层当"轻量脚本"写没问题,但它和方块实体、实体一样,最终都汇进同一个 tick 预算。

所以"指令系统"这个名字其实低估了它:它是 Minecraft 暴露给非程序员的、对整套世界数据模型的**结构化读写与逻辑编排接口**,也是这套数据模型设计得足够规整(注册表 ID、NBT、组件、标签)之后,自然能长出来的一层。
