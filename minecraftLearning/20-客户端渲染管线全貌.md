# 客户端渲染管线全貌

第 03 篇讲了一件事:一个 chunk 怎么被合并成一个 mesh、怎么靠面剔除把三角形数量压下来。但那只是"地形几何"这一块。真正按下 F3 看到 60 帧画面时,屏幕上还有水的半透明、树叶的镂空、僵尸的模型、箱子的开盖动画、火把的粒子、明暗渐变的光照——这些都不在 chunk mesh 里,各走各的路径。

这篇把 chunk 之外的整条管线补上:**一帧到底干了什么,谁在哪条路径上被画出来,以及为什么 Minecraft 的渲染长期是性能瓶颈、Sodium 这类 mod 又重写了什么。**

> 一句话先记住:**Minecraft 的渲染是"把世界拆成几条互不相同的路径,各自剔除、各自排序、各自上色,最后按固定顺序叠到同一帧上"。难点几乎全在两件事——决定"哪些不画"(剔除)和"半透明按什么顺序画"(排序)。**

---

## 一、两条线程:游戏线程 vs 渲染线程

先破一个常见误解:Minecraft 的"渲染线程"不是一个和游戏逻辑完全并行、各跑各的工人。

Java 版的真实情况是:

- **游戏线程(主线程,Render Thread)**:既跑客户端逻辑,也发 OpenGL 绘制调用。它是那条按帧循环、最终提交 draw call 的线程。
- **集成服务端线程(Server Thread)**:单机时世界逻辑(第 04 篇那套 20 TPS tick)跑在另一条线程,通过内存管道和客户端通信。
- **工作线程池**:chunk 的 mesh 构建(第 03 篇那套"遍历方块、生成可见面")被甩到后台线程池里异步做,做完把顶点数据交回主线程上传 GPU。

所以"帧率"和"TPS"是两套时钟(第 04 篇已强调):

```
Server Thread:  ──tick──tick──tick──   固定 20 次/秒,管世界状态
                        │ 状态快照 / 实体位置
                        ▼
Render Thread:  ─帧─帧─帧─帧─帧─帧─帧─   能多快多快,管"把当前状态画出来"
                  ▲
        Worker Pool: 后台重建脏 chunk 的 mesh,不卡帧
```

**反直觉点**:实体位置每秒只更新 20 次,但你看到的移动是丝滑的——因为渲染帧之间客户端对实体做**插值**(下面讲)。世界状态是"阶梯",画面是"插出来的斜坡"。

## 二、一帧里发生什么

按顺序拆一帧(简化但保真):

```
1. 处理输入、推进摄像机(含 partialTick 插值因子)
2. 更新区块可见性:从摄像机所在 section 做 BFS,标记哪些 render section 要画
3. 地形几何:按 RenderType 分批,画所有可见 render section 的已上传 mesh
     ├─ solid       不透明
     ├─ cutout      镂空(树叶、草、铁栏杆)
     └─ translucent 半透明(水、玻璃、冰)—— 需按距离排序
4. 实体:遍历附近实体,各自的 EntityRenderer 画模型
5. 方块实体:BlockEntityRenderer 画箱子盖、告示牌字、床、信标光柱
6. 粒子:烟、火、药水雾(纯客户端)
7. 天空盒、云、天气、debug 线框(F3 的 chunk 边界)
8. 半透明的二次处理、手部模型、HUD/GUI
9. 提交帧
```

注意第 1 步那个 **partialTick**:它是"距离上一个游戏 tick 过去了百分之几"的小数。所有平滑动画——摄像机、实体、箱子盖角度——都是拿上一 tick 和这一 tick 的值,用 partialTick 线性插值出"此刻该显示的中间态"。这是 20 TPS 的世界能渲染出任意高帧率的根本机制。

## 三、RenderType:为什么世界要拆成几批画

不能"一个方块一个方块按出现顺序画"。GPU 喜欢"同一种材质/混合模式的东西连着画",而且**半透明必须在不透明之后、且从远到近**才正确。所以地形几何按 **RenderType** 分桶:

| RenderType | 典型方块 | 关键特性 | 难点 |
| --- | --- | --- | --- |
| `solid` | 石头、泥土 | 完全不透明,深度测试直接挡 | 最简单,顺序无所谓 |
| `cutout` | 树叶、草、铁栏杆 | 像素要么全不透明要么全透明(alpha 测试丢弃透明像素) | 边缘锯齿;`cutout_mipped` 处理 mipmap |
| `translucent` | 水、玻璃、冰、染色玻璃 | 半透明,要 alpha 混合 | **必须从远到近排序**,否则混合结果错 |

**translucent 为什么是老大难**:半透明像素的颜色是"自己的色 × alpha + 背后已有的色 × (1-alpha)"。这要求背后的东西**先画好**。两块玻璃前后叠着,先画近的再画远的,远的那块就会被近的错误地"挡掉一部分混合"。所以每个 render section 内部的半透明三角形要**按到摄像机距离排序**,而且摄像机一移动,排序可能失效需要重排——这是开 shader/大量水面时掉帧的常见来源。

```
不透明:谁先画无所谓,深度缓冲自动挡住后面的
半透明:必须 远 → 近,逐层混合
        画错顺序 → 玻璃后面的玻璃"消失"或发黑
```

## 四、render section 与 GPU 上传

接第 03 篇:世界按 16×16×16 切成 **render section**(渲染区段,和存储的 section 对齐)。每个 render section 维护**每种 RenderType 一个 mesh**。

生命周期:

```
方块改动 → 该 section 标记 dirty
       → 丢进 worker pool 重建 mesh(遍历方块、生成可见面、按 RenderType 分桶)
       → 顶点数据上传到 GPU 的 VBO
       → 主线程渲染时,对可见 section 直接发"画这块 VBO"
```

要点:

- **改一格只重建一个 section**(第 03 篇说过),但如果改在边界,相邻 section 的剔除也变了,会连带重建邻居。
- 原版用 per-section 的小 VBO,**draw call 数量 = 可见 section 数 × 用到的 RenderType 数**,几百上千个 draw call。这正是性能瓶颈所在。
- **Sodium 的核心优化之一**就是把这些零碎 VBO 合并、用更现代的 OpenGL(持久映射缓冲、间接绘制、把剔除搬到 GPU),把成百上千 draw call 压成个位数。后面再说。

## 五、剔除:整条管线最值钱的"不画"

画得快的第一性原理是**少画**。Minecraft 用两层剔除决定"哪些 render section 根本不提交":

### 1. 视锥剔除(frustum culling)

摄像机的可视范围是一个四棱锥(frustum)。section 的包围盒完全在锥外 → 直接跳过。这是便宜且通用的第一道关:背后的、视野外的世界不画。

### 2. 区块可见性遍历(cave culling)

这是 Minecraft 特有、也最聪明的一招,专治"在洞穴/室内时不想画身后整片地下世界":

预处理时,给每个 section 算一张**"从哪个面进、能从哪个面出"的连通表**(光/视线能否穿过这个 section 内部的空气连通区)。渲染时**从摄像机所在 section 出发做 BFS**:

```
站在洞里:
  当前 section → 看它六个面哪些"通"(内部有空气连到该面)
  只沿"通"的面扩散到邻居,且记录进入方向避免回头穿实心
  → 被实心岩石包住的 section 永远扩散不到 → 不画
```

效果:在地下或建筑里,身后被墙挡住的几千个 section 根本不进渲染列表。**反直觉的副作用**:F3 里能看到"被挡住的区块不渲染",偶尔卡视角会看到世界"漏洞"——那就是 cave culling 判断你看不到、于是没画。

两层叠加:frustum 砍掉视野外,cave culling 砍掉视野内但被实体方块挡死的。剩下的才上传 draw call。

## 六、实体、方块实体、粒子:各走各的路

地形是"静态批量几何",但会动的东西不能塞进 chunk mesh(它们每帧位置/姿态都变),所以走独立路径:

| 类别 | 渲染器 | 特点 |
| --- | --- | --- |
| 实体(第 06 篇) | `EntityRenderer` | 每帧按插值后的位置/朝向画模型、骨骼动画、装备叠加 |
| 方块实体(第 09 篇) | `BlockEntityRenderer` | 给"标准方块模型画不出来的"方块做特殊绘制 |
| 粒子(第 10 篇) | 粒子引擎 | 纯客户端,面向摄像机的小四边形(billboard) |

**方块实体渲染器**正好接上第 09 篇的核心结论:床、箱子之所以是方块实体,很大程度就是为了挂这个渲染器。箱子盖的开合角度由 `ChestLidController` 维护一个 0~1 的值,**每帧用 partialTick 插值**出当前角度再画——所以盖子是平滑掀开而不是瞬间跳。大箱子还要靠 `DoubleBlockCombiner` 把两半的盖子同步绘制成一个连续模型。这些都不可能用静态 chunk mesh 表达,只能逐帧重画。

代价:方块实体渲染器是**逐个、每帧、在主线程**跑的。一屏几百个告示牌/箱子(尤其大型仓库)会明显掉帧——这和第 04 篇"方块实体每 tick 遍历很贵"是同一类问题的渲染版。

## 七、光照表现:逐格 0-15 怎么变成丝滑明暗

第 10 篇讲了光照是**逐格、两通道(BlockLight + SkyLight)、0~15** 的数据层。渲染端要把这些离散整数变成画面上的连续明暗,靠两个机制:

### 1. lightmap(光照贴图)

把 `(blockLight, skyLight)` 这对 0~15 的值当成一张 16×16 查找纹理的坐标,采样出一个最终颜色乘子。这张 lightmap 还随时间(白天/黑夜让 skyLight 列变暗)、随维度(下界偏红)、随药水(夜视拉亮)整体重算。所以"日落时世界慢慢变暗"不是逐方块改光照值,而是**lightmap 这张表在变**。

### 2. 平滑光照 / 环境光遮蔽(AO)

如果每个方块面直接取自己那格的光照值,你会看到生硬的块状明暗。平滑光照(Smooth Lighting)在**生成 mesh 时**,给每个顶点取**周围四格光照的平均**,再让 GPU 在面内插值,于是明暗是渐变的。AO(环境光遮蔽)再额外压暗凹角(被三个方块围住的顶点更暗),给体素世界一点立体感。

**反直觉点**:平滑光照是 chunk meshing 阶段的逐顶点计算,**烤进顶点数据**里。所以光照一变(放火把),不只是改数据,还要**重建受影响 section 的 mesh**——这就是为什么放一个火把会让周围一圈 chunk 短暂重建,也是光照成为性能热点(第 19 篇会展开)、Starlight 重写光照引擎的原因之一。

## 八、F3:把这条管线掀开给你看

F3 调试屏其实是这条管线的仪表盘:

- **C: xxx/yyy** —— 渲染了多少 section / 总共多少(看 cave culling + frustum 砍掉了多少)
- **E: a+b** —— 实体渲染数 / tick 数
- **chunk 边界(F3+G)** —— 画出 section 网格,直接看到"渲染区段"的切分
- **fps 与 "without/with vsync"** —— 帧率;和左下的内存、右侧的方块/光照数据一起,能定位是"画太多"还是"逻辑卡"

学会读 C 那行尤其有用:在洞里它远小于平地,就是 cave culling 在干活。

## 九、为什么原版渲染慢,Sodium 改了什么

原版渲染的结构性问题,基本都在前面埋着:

| 问题 | 根因 | Sodium 的做法 |
| --- | --- | --- |
| draw call 太多 | 每 section 每 RenderType 一个小 VBO | 合并缓冲、批量/间接绘制,几个 draw call 搞定 |
| 剔除在 CPU 上 | 主线程逐 section 判可见 | 把剔除/排序往 GPU 和并行挪 |
| 用老旧 OpenGL | 历史包袱 | 现代 GL(持久映射、VAO、更紧凑顶点格式) |
| 半透明排序卡 | CPU 逐三角排序 | 更高效的排序与缓存 |

**结论性的话**:Sodium 没有改"世界是什么"(数据模型一字没动),它只改了"怎么把同样的世界喂给 GPU"——把第三~五节那套 RenderType 分桶、VBO 上传、CPU 剔除,换成对现代显卡友好的组织方式。这也说明原版的瓶颈不在显卡算力,而在**提交方式**。
