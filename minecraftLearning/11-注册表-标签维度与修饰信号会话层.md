# 注册表、标签、维度与修饰 / 信号 / 会话层

前面十篇我们把世界"长什么样、怎么跑、怎么存"讲透了:体素、噪声、渲染、tick、存储、方块、实体、流体、物品、生物群系、光照……每讲一类,你脑子里大概都冒过同一个问题——

> 这些"种类"到底是从哪冒出来的?谁规定了"3 号方块是石头"?为什么我装个 mod 就能凭空多出一种方块、一种维度?

这一篇回答的就是这个"元问题"。它不再讲世界里的某个具体对象,而是讲**把所有对象的"类型"装起来的那个总系统**,以及几类我们前面顺带提过、但没正面展开的"非对象"层:修饰层、运行时信号、交互会话层,最后是兜住一切的维度与存档容器。

> 一句话先记住:**世界里能存在的每一种东西,都得先在"注册表"里上户口拿到一个 ID;注册表就是 Minecraft 的户口本,也是贯穿"类型—存储—网络—数据包—版本演进"的那根主轴。**

---

## 一、注册表 Registry:一切"类型"的总账

回想前几篇反复出现的"类型层 vs 实例层":`Block`(石头这一类)是单例,`BlockState` 下标(世界里那一个)是实例;`Item` 是类型,`ItemStack` 是实例;`EntityType` 是类型,`Entity` 是实例。

这些**类型**(每种方块、每种物品、每种实体类型、每种流体、每种生物群系、每种状态效果、每种附魔……)不是散落在代码各处的,而是**全部登记在注册表里**。你前几篇问的"世界里还有哪些类别",最终答案就是一句:

> 去翻注册表里有哪些 Registry。每一种 Registry 就是一类东西的名册。

### 上户口会拿到两套 ID

每个注册项登记后,拿到两个标识:

```
命名空间 ID(资源位 ResourceLocation): minecraft:stone
数字 ID(运行时分配的整数):           例如 1
```

| 标识 | 给谁用 | 特点 |
| --- | --- | --- |
| 命名空间 ID | 人 / 数据包 / 命令 | 稳定、可读;`minecraft:` 是原版,mod/数据包用自己的前缀 |
| 数字 ID | 机器(网络、存盘) | 紧凑;传 `1` 远比传 `"minecraft:stone"` 省 |

**命名空间是可扩展性的基石。** 因为每个东西都带前缀,你的 mod 注册 `mymod:ruby_block`、我的数据包注册 `atlasandecho:foo`,天然不会撞车。这就是 Minecraft 生态能容纳成千上万 mod 而不互相踩踏的根本原因。

**数字 ID 是性能的基石。** 还记得第八篇 chunk 的 section 用"局部调色板 + 紧凑位数组"存方块?调色板里那一条,指向的就是注册表里某个 BlockState 的**全局数字 ID**。网络同步、存盘 palette,传的都是数字而不是字符串。

## 二、两个关键机制:冻结与同步

注册表能当"户口本"用,靠两条纪律。

### 1. 启动时冻结(freeze)

游戏启动 → 加载所有原版 + mod + 数据包的注册项 → **冻结注册表**。冻结后不能再加新条目。

为什么必须冻结?因为运行期间"3 号方块是什么"必须**固定不变**——palette 下标、网络协议、存盘里的数字 ID 才有确定含义。如果运行中还能往注册表里塞东西、重排编号,那磁盘上写过的所有 palette 全部失效。

### 2. 进服时同步(registry sync)

多人游戏有个隐患:**服务端的"1 号方块"和客户端的"1 号方块"必须是同一个东西**,否则你看到的世界整体错位——服务端说"这格是 1",客户端把 1 解析成别的方块。

```
玩家连入
   │
   ├─ 服务端把注册表的 ID 映射打包发给客户端   (registry sync)
   │
   ├─ 客户端按服务端的编号重新对齐自己的注册表
   │
   └─ 双方对"几号是什么"达成一致 → 世界才能正确显示
```

> **踩坑 / 反直觉:** "装了服务端没有的 mod、或版本不匹配就进不去",本质就是注册表对不上——客户端有些条目服务端没有,ID 映射对不齐,握手直接失败。很多人以为是"版本号检查",其实更底层的是注册表一致性检查。

### 3. 静态注册表 vs 动态注册表

不是所有注册表都一样"硬"。

| 类型 | 谁定义 | 例子 |
| --- | --- | --- |
| **静态注册表** | 代码硬编码,启动注册 | 方块、物品、实体类型、流体 |
| **动态注册表** | **数据包的 JSON 驱动**,启动从数据包加载 | 生物群系、维度类型、世界生成配置、消息类型 |

动态注册表同样参与进服同步。这就解释了一个很多人觉得"魔法"的现象:

> 为什么如今做自定义维度、自定义生物群系**不用写一行 Java、改几个 JSON 就行**?

因为它们是**数据驱动的注册项**。世界生成、维度、生物群系早已从"硬编码"重构成"数据包定义",启动时被读进动态注册表,和原版条目一视同仁。

## 三、标签 Tag:给注册项分组

注册表解决"有哪些东西",**标签**解决"哪些东西算一类"。

```
#minecraft:logs    → 所有原木(橡木原木、白桦原木、去皮原木、绯红菌柄…)
#minecraft:planks  → 所有木板
#minecraft:doors   → 所有门
```

标签的三个关键性质:

- **被规则引用而非逐个列举**:合成、方块行为、命令选择器、战利品条件,大量写"匹配 `#minecraft:logs`"而不是把每种原木列一遍。
- **数据驱动、可被数据包扩展并合并**:多个数据包可以往同一个标签里各加几条,最终**合并**成一个标签内容,而不是互相覆盖。
- **对未来自动生效**:正因为规则写的是标签,"任意原木都能合成木板"这条规则,会对以后版本/mod 新增的任何原木**自动适用**——只要新原木进了 `#minecraft:logs`。

> 标签是"开闭原则"在游戏数据层的落地:对扩展开放(往标签加条目),对修改封闭(规则不用改)。

## 四、注册表如何咬合前面所有篇

把注册表当主轴,前十篇的零散点会被一根线串起来:

| 现象(前面某篇) | 背后是注册表 |
| --- | --- |
| section palette 里的下标(03 / 08) | 指向注册表里的 BlockState 全局数字 ID |
| chunk NBT 里写 `minecraft:chest`(08 / 09) | 加载时查注册表,把命名空间 ID 还原成具体类型 |
| 跨版本升级 DataVersion / DataFixerUpper(08) | 本质是处理"注册表条目随版本增删 / 改名"后,旧 ID、旧名怎么迁移到新注册表 |
| 数据包能改游戏(本篇) | 因为内容定义和动态注册表对数据包开放 |
| 进服失败 / mod 不一致(本篇) | 注册表 ID 映射对不齐 |

所以注册表不是"又一个系统",而是**那根贯穿类型层、存储、网络、数据包、版本演进的主轴**。理解了它,前面所有"为什么这里写的是个 ID / 字符串"的疑问都收口了。

## 五、修饰层:贴在对象上的"形容词"

注册表登记的是"名词"(类型)。但游戏里还有一整套**修饰系统**——它们不是独立对象,而是"贴"在实体或物品上,改变其行为。

| 系统 | 贴在谁身上 | 作用 | 存哪 |
| --- | --- | --- | --- |
| **属性 Attribute** | 生物实体 | 最大生命、移速、攻击力、护甲、击退抗性… | 实体 NBT(带 modifier 列表) |
| **状态效果 MobEffect**(药水效果) | 生物实体 | 速度 / 中毒 / 再生…,**每 tick 结算** | 实体 NBT(active_effects) |
| **附魔 Enchantment** | 物品 | 锋利、保护、效率…,改物品行为 | 物品上 |
| **数据组件 Data Component** | 物品 | 1.20.5 起物品就是"一组组件"拼成的 | 物品组件 |

这里单独说一句 **1.20.5 的物品模型重构**:在那之前,物品的额外数据塞在一坨自由格式的 NBT `tag` 里(附魔、自定义名、耐久……全堆在一起,没有强结构)。1.20.5 之后改成**数据组件(Data Components)**:一个物品 = 一组**有明确类型的组件**(`minecraft:enchantments`、`minecraft:custom_name`、`minecraft:damage`…)。这是从"一坨弱类型 NBT"到"强类型组件集合"的架构升级,本质和"BlockState 用强类型属性而非散 NBT"是同一种思路。

> **反直觉:** 这些修饰单独拎出来都"不是世界里的东西",但游戏的大量深度恰恰住在这一层——一把"锋利 V + 经验修补"的剑,剑本身只是 `ItemStack`,真正的差异全在贴上去的附魔组件里。

### 修饰层的数值规则:附魔与铁砧

修饰层不只是"有哪些字段",它背后是一套**数值规则系统**——附魔和铁砧就是最典型的例子。附魔/属性不是凭空挂上去的对象,而是按规则**算出来、再贴到物品上的值**。

**附魔台**算的是"这次能附什么":

```
附魔力 = 周围书架数量(最多 15 个有效)
   → 书架越多,可选档位的"等级要求"越高(最高 30 级)
三个候选档位 = f(附魔力, 物品可附魔性, 附魔种子)
```

- 消耗**经验等级 + 青金石**;书架提供"附魔力",把可选档位顶到更高。
- 结果由一个**玩家级的附魔种子**(enchantment seed)+ 物品共同决定:所以它**带随机但可预览**——你能提前看到三档各给什么(至少一条),换个物品/重置种子才会变。这就是"伪随机":随机源是固定的种子,而非每次真随机。

附魔一旦附上,就作为**组件/数据存在物品上**(第 17 篇会讲 1.20.5 后它是 `minecraft:enchantments` 组件,第 09 篇讲过物品本身只是 `ItemStack`)——是贴在物品这个实例上的"修饰",不是一种独立对象。

**铁砧**算的是"合并/修复/改名要花多少经验":

```
总代价(经验等级) = 附魔合并代价 + 改名代价 + 累计惩罚(prior work penalty)
累计惩罚:这件物品每被铁砧操作一次,下次就更贵(2ⁿ - 1 增长)
   → 超过上限(生存模式 39 级)直接显示 "too expensive" 拒绝
```

- 核心反直觉点就是这个 **prior work penalty**:同一件装备反复上铁砧会越来越贵,和"附了多强的魔"无关,只和"被操作过几次"有关。所以高玩讲究"先把附魔书合并好,再一次性砸到装备上",就是为了少触发累计惩罚。

而**经验值本身**也是非线性的:升级所需经验随等级增长越来越多(低级线性、高级二次方阶梯),所以"30 级附魔"在前期是巨款、后期是零头。

> 一句话收束:附魔台、铁砧、经验,合起来是**修饰层之上的一套数值规则引擎**——它决定"哪些修饰、以什么代价、按什么随机性"贴到实体和物品上。修饰本身是数据,这套规则才是让它有深度的东西。

## 六、运行时信号:不存盘,只在 tick 间流动

前面讲的多偏"状态 / 数据"。但还有一类东西**从不进存档**,它们是**瞬时的事件 / 信号**,只在第四篇讲的 tick 循环里产生、传播、消失。

| 信号 | 怎么传 | 谁响应 |
| --- | --- | --- |
| 红石信号 | 邻居更新级联(第四篇) | 红石元件、活塞、漏斗… |
| 游戏事件 / 振动 GameEvent | 在世界里广播,按频率分类 | 幽匿感测体、监守者(1.17 幽匿系统) |
| 声音 SoundEvent | 触发即播放,客户端侧 | 玩家(基本不留存) |
| scheduled / random tick | 时间维度的"延迟信号"(第四篇) | 流体扩散、作物生长… |

它们的共同特征:

> **没有"存在哪"的问题,只有"何时发生、传到哪"。** 你存档里找不到"一条红石信号""一次振动",存档里只有产生它们的方块布局。

幽匿那套尤其典型:你放方块、走路、开箱,都会**广播一个游戏事件**,感测体监听并按事件频率响应。这是纯运行时广播,关掉游戏什么都不剩。

## 七、交互 / 会话层:玩家与世界之间的临时桥

还有一类对象,既不在世界格子里、也不贴在实体上,而是**临时存在于"玩家"和"世界"之间**。

- **菜单 Menu / 容器(ScreenHandler)**:你打开箱子时,服务端**临时创建一个 Menu**,把箱子的槽位**同步给客户端**;你拖动物品其实是在和这个 Menu 通信;关掉界面它就销毁。第九篇讲大箱子时那个把两个单箱子拼成 54 格视图的 `CompoundContainer`,就是塞进这个 Menu 的。
- **网络连接 / 玩家会话**:每个在线玩家对应一条连接和一份同步状态。

> 它们都是**临时、不进存档**的——是"交互过程"的载体,不是"世界内容"。这也解释了为什么开着箱子界面时世界还在背后照常 tick:界面是会话层的临时对象,世界是另一套。

## 八、最外层:维度与存档容器

最后,把镜头拉到最远——所有上述东西都装在哪?

```
存档 World(level.dat:种子 / 出生点 / 时间 / 游戏规则 / DataVersion)
├── 维度 Overworld   (DIM,主世界:独立 chunk 存储)
├── 维度 Nether      (DIM-1,下界:独立 chunk、独立环境规则)
├── 维度 End         (DIM1,末地)
├── 维度 自定义…     (数据包定义,动态注册表)
└── 数据包 Datapack:配方 / 战利品表 / 进度 / 标签 / 动态注册表(JSON)
```

- **维度 Level / Dimension**:主世界、下界、末地,以及自定义维度。每个是一个**独立的世界**——独立的 chunk 存储、独立的时间、独立的高度与环境规则(由**维度类型 DimensionType** 定义,而 DimensionType 本身就是动态注册表里的条目)。
- **存档 World**:`level.dat`(GZip 的 NBT)那一层,管种子、出生点、游戏规则、DataVersion 等**全局状态**。
- **数据包 Datapack**:配方、战利品表、进度、谓词、函数、标签、动态注册表全是 JSON 定义的**游戏内容规则**。它描述"游戏怎么玩",可被替换——这正是 Minecraft 高度可定制的来源,也和本篇开头的注册表首尾呼应:**数据包之所以能改游戏,是因为内容定义和动态注册表对它开放。**

## 九、串联:从"对象"到"系统",再到一张总图

到这里,我们其实跨过了一条边界:

```
前十篇:世界里的"逐实例对象"      —— 有限集,基本已穷举
  方块 · 流体 · 方块实体 · 实体 · 物品 · 生物群系 · 光照 · (粒子)

本篇:  引擎里的"系统 / 层"        —— 开放集,可以一直拆
  注册表(元系统) · 修饰层 · 运行时信号 · 交互会话 · 维度 / 存档 / 数据包
```

这条边界很重要:当你问"Minecraft 里还有什么",其实在问两个不同的问题——

- **"世界里有哪些对象?"** 是有限的,前十篇那张表基本到头。
- **"引擎里有哪些系统?"** 是开放的,因为那不再是"世界里有什么",而是"游戏怎么运转",可以一直往下拆(还有碰撞形状、AI 的 Brain/记忆、结构生成、命令 / 函数 / 谓词……)。

而**注册表是这两者的接缝**:每个"对象类型"都是注册表里的条目,每个"系统"又都绕着注册表的 ID、同步、数据驱动来组织。

下一篇(也是最后一篇),我们把这十一篇——从体素、噪声、渲染,到 tick、区块生命、实体,到 Region、NBT、Palette,再到方块三层、其它对象、注册表——**全部缝成一张分层架构总图**,并用"一个 16³ section 从生成 → 运行 → 存盘 → 加载"的完整生命周期时序,把横向的数据流也画出来,给整个系列收口。
