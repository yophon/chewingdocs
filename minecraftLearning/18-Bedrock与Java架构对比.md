# Bedrock 与 Java 架构对比

前面十七篇拆的几乎全是 Java 版:Anvil 存储、NBT、调色板、20 TPS 的 tick 循环、quasi-connectivity 这套红石更新机制。但 Minecraft 其实有两套**互相独立实现**的代码库——Java 版和 Bedrock 版(手机/主机/Win10 商店那个)。它们玩起来像同一个游戏,底层却是两台不同的发动机。

这一篇不是"哪个更好"的口水战。它的真正价值在于:**当你看到同一个游戏被实现了两遍,你才能分清前面讲的那些设计里,哪些是体素游戏的本质约束,哪些只是 Java 版的一种历史选择。** 对照是最好的"祛魅"工具。

> 一句话先记住:**Java 版是"JVM + 面向对象 + 文件式存储 + TCP",Bedrock 是"C++ + ECS + KV 数据库 + UDP";两者所有可见的行为差异(红石、刷怪、含水),根子都在底层实现机制不同,而不是设计师故意要它们不一样。**

---

## 一、一张总表先把差异铺开

| 维度 | Java 版 | Bedrock 版 |
| --- | --- | --- |
| 语言 / 运行时 | Java,跑在 JVM 上 | C++,编译成原生码 |
| 对象模型 | 面向对象 + 继承体系(`Entity` → `LivingEntity` → `Mob`) | ECS(Entity-Component-System) |
| 世界存储 | Anvil 文件 `.mca` + region 扇区 | LevelDB(key-value 数据库) |
| NBT 字节序 | 大端(big-endian) | 小端(little-endian),网络包用 varint |
| 网络协议 | 自研协议,跑在 TCP 上 | 跑在 RakNet(UDP)上 |
| 平台 | PC(Win/Mac/Linux) | 手机 / 主机 / Win10 / VR,跨平台联机 |
| 扩展方式 | mod(直接改字节码 / Fabric / Forge) | Add-on(行为包 + 资源包,受限沙盒) |
| 红石更新机制 | 有 quasi-connectivity | 没有 quasi-connectivity |

下面逐行拆,重点解释**为什么**会这样,以及它如何反过来印证前面讲过的东西。

---

## 二、对象模型:继承体系 vs ECS

这是两版最深层的分歧。

**Java 版是教科书式的面向对象继承。** 回忆第 06 篇讲的实体层级:

```
Entity
 └─ LivingEntity
     └─ Mob
         ├─ Zombie
         └─ Villager
```

一只僵尸"是什么"由它在这棵继承树上的位置决定。要加新行为,就重写方法、往父类塞字段。优点是直观、好读;缺点是**继承是僵硬的**——当你想让某个怪"既能游泳又能飞又能交易",继承树就开始打架(经典的"钻石继承""上帝基类"问题)。

**Bedrock 版用 ECS。** 实体本身只是一个 ID,它的能力由挂在身上的一堆**组件(component)**拼出来:

```
僵尸 = [生命组件] + [近战攻击组件] + [日光燃烧组件] + [寻路组件] + ...
```

想让它会游泳?加一个游泳组件,不动任何继承树。这也是为什么 **Bedrock 的 Add-on 能用纯 JSON 改怪物行为**——行为包里写的就是"给这个实体加/改哪些组件",根本不碰代码。Java 版要做等价的事,你得写真正的 mod 去改类。

> 反直觉点:很多人以为"Bedrock 不能装 mod 所以更封闭"。恰恰相反,**正是 ECS 架构让 Bedrock 的官方 Add-on 系统能安全地数据驱动行为**;Java 的强大 mod 生态不是因为它架构开放,而是因为社区直接撬开了字节码。两条路,一个是官方留的门,一个是社区砸的墙。

---

## 三、存储:文件 vs 数据库

第 07、08 篇详细讲过 Java 的存储:region 文件 `.mca`,8KB 头表 + 4KB 扇区,chunk 数据是 zlib 压缩的大端 NBT,调色板做位压缩。

**Bedrock 把整个世界塞进一个 LevelDB**(Google 那个嵌入式 key-value 库)。世界不再是"一堆 region 文件",而是一个数据库目录,里面是无数条 `key → value`:

| Java(Anvil) | Bedrock(LevelDB) |
| --- | --- |
| `r.0.0.mca` 一个文件装 1024 个 chunk | 每个 subchunk 一条 KV 记录 |
| 查 chunk:读 location 表 → seek 扇区 → 解压 | 查 chunk:用 `(维度,x,z,层)` 拼出 key → DB 查 |
| 大端 NBT | 小端 NBT |
| 手动管理扇区分配、碎片 | LevelDB 自己管 LSM-tree、压实(compaction) |

key 的结构大致是"区块坐标 + 数据类型标记"。比如一个 subchunk 的方块数据、一个区块的实体、一个区块的方块实体,各自是独立的 key。**这等于把 Java 版那套"自己手写的扇区式数据库"换成了一个现成的工业级 KV 引擎。**

为什么不同?Bedrock 要在**手机、低端主机**上跑,LevelDB 在小内存设备上的随机读写、增量写入比"自己管文件扇区"省心也更稳;碎片整理、原子写这些 Java 版要自己操心的事,LevelDB 内建。代价是世界目录不再是人眼能直接翻的文件结构,得用工具读。

字节序的差异(大端 vs 小端)纯粹是历史/平台选择:Java 习惯大端,Bedrock 跑在 x86/ARM 这些小端机器上,就用小端省一次转换。**这点最能说明问题——NBT 的"语义"两版一样,只是序列化时的字节方向不同,证明字节序根本不是设计本质,只是实现细节。**

---

## 四、网络:TCP vs RakNet(UDP)

第 13 篇讲过 Java 的网络:自研协议跑在 TCP 上,有 handshake / login / configuration / play 几个阶段,服务端权威。

**Bedrock 跑在 RakNet 上,而 RakNet 是建在 UDP 之上的游戏网络库。** 区别:

| | Java(TCP) | Bedrock(RakNet/UDP) |
| --- | --- | --- |
| 传输层 | TCP,内核保证有序可靠 | UDP,RakNet 在应用层自己实现可靠性 |
| 可靠性粒度 | 全有序、全可靠 | 可按消息选"可靠/不可靠/有序/无序" |
| 适合 | 稳定有线网 | 移动网络、丢包、跨平台 |

为什么 Bedrock 选 UDP?因为它的主战场是**手机和主机**,网络环境抖动、丢包常见。TCP 的"一个包丢了后面全得等重传(队头阻塞)"在弱网下会一卡一卡;RakNet 让 Bedrock 能对"不重要的位置更新"用不可靠通道(丢了就丢、发新的),对"重要的方块变化"用可靠通道,细粒度控制。这也是 Bedrock 能做**跨平台、跨设备联机**(手机和 Xbox 一起玩)的网络基础。

Java 版选 TCP,某种程度上是"PC 有线网够稳,省事优先"的历史选择——又一个"可以不一样"的点。

## 五、性能与平台:这才是 Bedrock 存在的理由

| | Java | Bedrock |
| --- | --- | --- |
| 编译目标 | JVM 字节码(JIT) | 原生机器码 |
| 内存开销 | 较高(JVM + 对象头 + GC) | 较低,适合手机/主机 |
| 启动 / 帧率 | 一般 | 通常更高、更稳 |
| GC 停顿 | 有(大世界、多实体时明显) | 无 GC(C++ 手动/RAII 管理) |

Bedrock 通常**跑得更快、吃内存更少**,因为它是原生 C++、没有 JVM 的 GC 停顿,而且从一开始就是为低端设备设计的。第 19 篇会讲到 Java 版的性能优化 mod(Lithium / Starlight / C2ME)——某种意义上,**这些 mod 在做的事,正是把 Java 版的逻辑实现优化到接近 Bedrock 原生实现的水准**。Java 版社区要靠 mod 补的性能,Bedrock 在架构层就拿到了。

代价是灵活性:Bedrock 是闭源原生码,你不能像 Java 那样反编译+改字节码;扩展只能在官方 Add-on 沙盒里玩。**速度换开放,开放换速度——这是两版最根本的取舍。**

## 六、行为差异的根:为什么红石/刷怪/含水两版不一样

玩家最直观的感受是"两版红石电路不通用、刷怪率不一样"。这些**不是设计师拍脑袋决定要不同**,而是底层更新机制不同后自然长出来的差异:

- **红石 quasi-connectivity(准连接)**:第 14 篇讲过,Java 的活塞/漏斗会"顺带检查正上方一格是否被供能",这是 Java 早年实现里的一个历史副作用,后来大量红石机器依赖它,就保留了下来。**Bedrock 是后写的实现,没继承这个副作用**,所以隔空供能、很多瞬时门在 Bedrock 上失效。两版红石不通用,根子就在这。

- **刷怪**:第 15 篇讲的 mobcap、生成检查、despawn 规则,两版的具体数值和算法细节不同(Bedrock 的刷怪范围、模拟距离、群组逻辑都另写一套),所以同一个刷怪塔设计在两版效率天差地别。

- **含水(waterlogging)与流体**:第 10 篇讲的 BlockState + FluidState 叠加,在两版的实现细节(流体更新调度、含水方块支持范围)不一致,导致水流行为、某些机器的可行性不同。

- **tick 顺序**:同一 tick 内方块/实体/调度的处理顺序两版不同,任何依赖"更新顺序"的精密机器(0-tick、特定红石时序)都会在另一版上失效。

**把这些串起来看,一个统一的结论浮现:可见的玩法差异,几乎全部追溯到"两套独立代码的更新机制/调度顺序不同",而不是规则设计本身不同。** 规则(一个方块该怎样、刷怪该看光照)两版是一致的;不一致的是"引擎怎么把这些规则跑出来"。

## 七、为什么要做这个对照

回到开头。把同一个游戏看成"被实现了两遍",最大的收获是**给前面十七篇做了一次祛魅**:

| 前面讲的 Java 特性 | Bedrock 对照后的结论 |
| --- | --- |
| Anvil 文件 + 扇区分配 | 不是必须——换成 LevelDB 一样能存,只是另一种取舍 |
| 大端 NBT | 纯历史/平台选择,小端也行 |
| 继承式实体层级 | 不是唯一解——ECS 是另一条更适合数据驱动的路 |
| TCP 自研协议 | PC 优先的选择,弱网下 UDP/RakNet 更合适 |
| quasi-connectivity 红石 | 根本是个历史副作用,不是设计,后写的实现就没有它 |

**而那些"两版都一样"的东西,才是体素游戏真正的本质约束**:世界是体素网格、地形靠确定性生成、按固定 tick 推进、方块靠某种调色板压缩、客户端-服务端分离。这些两版都绕不开,因为它们是问题本身的形状,不是某一版的偏好。

所以这一篇的位置很关键:它不增加新机制,而是给整个系列加了一个**参照系**。理解一个系统最快的方式之一,就是看它"本来也可以是另一个样子"——Bedrock 就是那个"另一个样子"。
