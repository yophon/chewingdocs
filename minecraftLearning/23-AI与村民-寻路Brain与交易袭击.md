# AI 与村民:寻路、Brain 与交易袭击

第 06 篇讲实体时,AI 那块只点了两个词:A\* 和 Brain。但实体 AI 是整个服务端里**最容易写出 lag、又最被低估**的一块——一只僵尸看起来只是朝你走过来,背后是寻路、目标仲裁、感知扫描三套东西每 tick 在跑。村民更夸张,它几乎是一个塞进方块世界里的小型 agent:有记忆、有作息、有经济、有声望。这篇就把这层拆开。

> 一句话先记住:**Minecraft 有两套并存的 AI 架构——老的 Goal 是"优先级抢占的动作列表",新的 Brain 是"记忆+感知+活动驱动的行为系统";村民是 Brain 系统的极致样板,贵也贵在这。** 而寻路是它们共同的、最烧 CPU 的底层服务。

---

## 一、寻路:每一步都不便宜

怪物要"走向"目标,靠的是经典 A\*,但它跑在一张**临时现算的节点图**上,而不是预存的导航网格。

流程大致是:

```
给定起点和目标
  → NodeEvaluator 在周围按需展开节点(每个节点≈一个可站立的方块位置)
  → 给每个节点判类型 PathType + 通行代价 malus
  → A* 用 f = g + h 找代价最低路径
  → 得到一串路点 Path,实体每 tick 朝当前路点移动,走到了就推进下一个
```

关键在 **NodeEvaluator + PathType**:它不是简单的"能走/不能走",而是给不同地形打**代价(malus)**:

| PathType | 行为 |
| --- | --- |
| WALKABLE | 正常,代价 0 |
| WATER / LAVA | 高代价或禁止(怕水的怪 malus 很高,直接绕) |
| DANGER_FIRE / DAMAGE_FIRE | 火、岩浆边缘,尽量躲 |
| FENCE / DOOR_WOOD_CLOSED | 栅栏当墙、关着的木门可开 |
| OPEN | 悬空(下方是洞),通常禁止,避免走下悬崖 |

代价的意义在于:寻路不是只看"通不通",而是**带偏好地绕**——所以怕水的怪会沿岸走、铁傀儡会避开仙人掌。

**为什么它贵?** A\* 的开销随搜索空间膨胀,而 Minecraft 的"图"是 3D 的、按需展开的。一旦目标够不着(隔着墙、隔着深坑),搜索会扩展大量节点才放弃。所以:

- 寻路有**节点数上限**(够不着时提前放弃,避免单次寻路炸 tick);
- 实体不是每 tick 重算整条路,而是**复用现有 Path**,只在必要时(目标移动、被挡住)重算;
- 但**一大群怪同时寻路**(刷怪塔、村庄被围)依然是 tick 时间的大头——这正好接上第 19 篇"掉 TPS 来源"里说的实体开销,寻路是其中最尖的那根刺。

## 二、两套 AI 架构:Goal vs Brain

Minecraft 历史上攒了**两套并存**的 AI 框架,新怪和旧怪各用各的,理解这点能解释很多"为什么村民比僵尸卡"。

### 老派:Goal 系统(GoalSelector)

僵尸、骷髅、苦力怕这些"老怪"用的是 **Goal**:

- 一个实体挂一组 **Goal**(浮空、追玩家、随机闲逛、躲太阳……),每个 Goal 带**优先级**和**互斥标志**(占用 移动/视线/跳跃 哪些"控制通道")。
- 每 tick,**GoalSelector** 扫一遍:按优先级看哪些 Goal 能启动、且不和已运行的高优先级 Goal 抢同一通道。能跑就跑,跑不动的被抢占。
- 心智模型:**一张按优先级抢占的"动作清单"**。简单、直接、好加新行为,但状态都散在各个 Goal 里,跨行为共享信息很别扭。

### 新派:Brain 系统

村民、监守者、傀儡、部分新生物用的是 **Brain**,它更像一棵行为树 + 黑板,由四个东西组成:

| 部件 | 作用 |
| --- | --- |
| **Memory(记忆)** | 黑板:记住"家在哪、工作站在哪、最近见过的玩家、攻击目标、走到一半的路点"等,有的带过期时间 |
| **Sensor(感知)** | 周期性扫描环境,把结果写进 Memory(附近的玩家/生物/兴趣点/可交互方块) |
| **Behavior(行为)** | 读 Memory 决定做什么(走向工作站、睡觉、逃跑、交易) |
| **Activity(活动)** | 行为的分组/状态:CORE/IDLE/WORK/REST/PANIC/RAID……当前处于哪个 Activity 决定哪批 Behavior 生效 |

Brain 的运转是:**Sensor 写记忆 → 当前 Activity 选出一批 Behavior → Behavior 读记忆并行动**。比 Goal 强在**状态集中、可共享、可表达复杂的条件切换**(比如"天黑了且有床 → 切到 REST Activity → 跑回家睡觉")。

**代价就在这。** Brain 每 tick 要跑一圈 Sensor 扫描(感知周围实体、扫 POI)、维护一堆带过期的 Memory、做 Activity 仲裁——比 Goal 那张清单重得多。所以**村民数量一多,Brain 的感知和寻路叠在一起,就是村庄卡顿的主因**。

## 三、村民:一个塞进方块世界的小 agent

村民是 Brain 系统的极限样板。它身上几乎挂满了前面所有篇章的概念。

### 职业绑定工作站 = POI

村民的职业(农民、图书管理员、武器匠……)不是凭空来的,而是**绑定到一个工作站方块**:

- 工作站(讲台、酿造台、高炉……)在世界里是一种 **POI(兴趣点)**,存在独立的 `poi/` region 里(就是第 11 篇提的那个 POI 数据层)。
- 无业村民的 Sensor 扫描附近**没被占用的 POI**,写进 Memory,Behavior 据此走过去"认领"工作站,获得职业。
- 工作站被破坏 → 失业 → 重新找。这套"找工作站"全程依赖 **POI 的索引与缓存**。

> 反直觉点:POI 是带缓存/索引的。村民找不到明明就在旁边的工作站,常是因为 POI 没被正确登记或缓存没刷新——这是"村民职业刷不出来"类玄学的工程根源。

### 一天的作息 = Schedule

村民按 **Schedule(作息表)** 切 Activity:

```
白天早些  → WORK   (去工作站旁边干活、补货)
正午      → MEET   (跑到聚会点闲逛、繁殖判定)
傍晚      → REST   (回家找床睡觉)
夜里      → 睡觉/躲在屋里
受威胁    → PANIC / 被袭击时切 RAID 相关
```

Schedule 决定当前 Activity,Activity 决定哪批 Behavior 生效——这就是 Brain 四件套的实际编排。

### 交易经济:供需 + 补货

村民交易不是固定价:

- 每种交易有**最大使用次数**,用光就**暂时下架**,村民回工作站**补货**(restock,每天有限次)后恢复。
- **供需**:某交易被频繁使用,价格上浮;长期没人买,回落。
- 这是一套真实跑在服务端的小经济模型,而不是写死的商店。

### 声望与八卦 gossip

村民之间会**传播对玩家的看法**:

- 你和某村民交易、救过它 → 该村民对你**好感(gossip)上升** → 折扣;伤害村民 → 好感暴跌、价格上涨,甚至触发**铁傀儡敌对**。
- 好感会在村民间**扩散**(八卦),所以得罪一个村庄可能整村涨价。
- 这套 gossip 是带数值和衰减的,存在村民的 NBT 里——又一处"逐实例数据挂在实体上"的例子(呼应第 06 篇)。

## 四、袭击 raid:一个独立的"事件状态"系统

袭击不是某个实体的 AI,而是一个**绑在村庄上的全局事件状态机**(存在世界的 saved data 里,类似第 11 篇说的全局数据层):

```
玩家带着"不祥之兆"效果进入有床+工作站的村庄
  → 在该村庄中心生成一个 Raid 对象,带波次计数、参与怪物列表、Boss 血条
  → 一波波刷出灾厄者(掠夺者/卫道士/唤魔师/劫掠兽)
  → 村民躲、铁傀儡防守、玩家清怪
  → 全波清完 → 胜利;村民全死/超时 → 失败
```

关键点:**Raid 是个有生命周期的事件对象,不是怪物本身**。它独立于参战实体存在,负责调度波次、统计、判定胜负——这和"实体""方块实体"都不同,更接近第 11 篇里"不绑具体格子的全局世界数据"。

---

## 五、把它接回前面

- **寻路** 是 AI 的底层服务,贵,直接喂给第 19 篇的"实体是 lag 大头"。
- **Goal vs Brain** 解释了为什么不同怪开销差很大;Brain 的 Memory/Sensor/Activity 是"实体身上的小型黑板系统"。
- **村民** 把 POI(第 11 篇)、Schedule、经济、gossip 全串起来,几乎是"实体 + 逐实例 NBT + 兴趣点数据层"的集大成。
- **袭击** 又是一个"全局事件状态"的例子,提醒你:世界里的逻辑不全在方块和实体上,还有一层不绑坐标的事件/会话状态。

一句话收尾:**怪物的"智能"本质是每 tick 的寻路 + 仲裁 + 感知;村民只是把这套推到极致,加上记忆、作息和经济,于是它既像 NPC,又是服务端最贵的常驻负载之一。**
