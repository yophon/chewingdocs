# 世界里还有什么:流体、物品、生物群系、光照、粒子

讲完方块、方块实体、实体,大多数人会觉得"世界的构成讲完了"。其实没有。如果你把一个坐标剖开看,会发现上面叠着的东西远不止"一个方块"。这一篇要补上那些**和方块平行存在、却容易被忽略**的对象与数据层。

它们不属于前三类,但每一类都实打实地占着坐标、影响着世界的行为。漏掉它们,你对"世界由什么构成"的理解就是残缺的。

> 一句话先记住:**同一个坐标 (x, y, z) 上,不是只有一个方块,而是叠着方块 + 流体 + 光照好几层数据;此外还有生物群系给区域贴标签、物品寄生在容器里、粒子只活在客户端。** 方块只是这叠数据里最显眼的一层。

---

## 一、流体 FluidState:最该补上的"第四套系统"

这是最容易被忽略、却真正和方块**平行存在**的一类。

关键事实:**世界里每个坐标同时持有一个 BlockState 和一个 FluidState**,两者是**半独立**的。

- 对绝大多数方块,FluidState 是"空"(`Fluids.EMPTY`)。
- 水源方块:它的 BlockState 是水,**同时**有一个非空的 FluidState。
- 含水(waterlogging)就是靠这个实现的。

### 含水机制:两层状态叠在一格上

你能给楼梯、栅栏、台阶"灌水",底层是这样的:

```
坐标 (10, 64, 20)
├── BlockState : oak_stairs[facing=east, half=bottom, waterlogged=true]
└── FluidState : water[level=8]   ← 与方块叠加,各管各的
```

**方块仍然是楼梯**(碰撞、外形、掉落都按楼梯走),但这一格**额外**承载了一份流动的水。`waterlogged` 这个布尔属性只是 BlockState 用来"记得自己含水"的标记,真正的水行为(流动、推动实体、淹没)由那层独立的 FluidState 负责。

### 独立的 tick 队列

回忆存储那一篇里 chunk NBT 同时存了 `block_ticks` 和 `fluid_ticks` —— 它们是**两个独立队列**,不是一个。流体的扩散、衰减全在自己的队列里跑:

| 概念 | 含义 |
| --- | --- |
| level 0–7(内部 1–8) | 流动水的"高度/能量",离源越远越低 |
| 衰减 | 水平每流一格 level -1,到 0 就停;竖直下落不衰减 |
| 找洞 | 流动前会探测附近能向下掉的方向,优先朝坑铺 |
| 源方块 | 特殊标记,周围两个源能互相补成新源(无限水) |

所以你看到的"水慢慢铺开",本质是 FluidState 在它自己的 tick 队列里**一圈一圈衰减扩散**,跟方块的 random tick 完全是两套机制。

### 踩坑 / 反直觉

- **流体不是方块**:1.13("扁平化")之前,水**就是**普通方块,所以那时做不了含水栅栏。1.13 把流体从方块里拆出来、独立成 FluidState 这套系统,含水机制才成为可能。这是一次典型的"把耦合的两个概念解耦"的架构重构。
- **挖掉含水楼梯**:楼梯没了,但那层 FluidState 还在,于是水"接管"了这格 —— 这就是为什么破坏含水方块后原地会留下水。

## 二、物品 ItemStack:寄生在容器里的纯数据

物品是和方块**地位相当**的一大类,只是它**不直接待在世界格子里**,而是住在"容器"里。

- 容器 = 背包、箱子(方块实体)、漏斗、玩家手上……
- 一个 **ItemStack** = 物品种类 + 数量 + 一组组件(耐久、附魔、自定义名等)。1.20.5 后这些附加信息从旧的 NBT `tag` 迁移成了"数据组件"。

```
ItemStack
├── item   : minecraft:diamond_sword   (种类,注册表里的类型)
├── count  : 1
└── components
    ├── damage      : 12
    ├── enchantments: { sharpness: 5 }
    └── custom_name : "断钢"
```

### 寄生型类别

物品自身是纯数据,**必须依附在某个宿主里才能存在**:

```
在箱子里 ─► 挂在箱子的方块实体的 Items 列表下
在背包里 ─► 挂在玩家实体数据下
扔到地上 ─► 被一个 ItemEntity(实体)"装着"进入世界,有了坐标和速度
```

最后一种最能说明问题:**同一份剑的数据,在箱子里时挂在方块实体下,扔出去时原样搬进一个实体里**。物品本身没变,变的是"谁在容纳它"。这也是物品和方块、实体三者通过同一套序列化(NBT/组件)互相衔接的缩影。

## 三、生物群系 Biome:坐标上的"气候标签"

生物群系也绑定到坐标,但和方块很不一样。

- **粒度粗**:不是每格一个,而是 **4×4×4 一格**。一个 16³ 的 section 只有 `4×4×4 = 64` 个生物群系格,同样用 palette 压缩(和方块状态用的是同一套调色板思路,只是分辨率低 4 倍)。
- **它不"占地方"**,而是给一片区域贴一个标签,决定:

| 影响 | 例子 |
| --- | --- |
| 气候 | 温度、湿度、是否下雪/下雨 |
| 刷怪 | 这片区域刷什么怪、刷不刷 |
| 染色 | 草、树叶、水的**颜色**(同样的草方块在沼泽和平原是不同绿) |
| 地物 | 生成什么植被、结构、矿物分布倾向 |

### 反直觉

你**挖不到、也撞不上**"生物群系"。它是一层无处不在的数据,而非可交互的对象 —— 这是它和"看起来也铺满世界"的方块最根本的区别:方块是实体化的格子内容,生物群系是贴在格子上的元数据。

## 四、光照 Light:逐格的数值场

光照是又一个**逐格存在、绑定坐标、要进 chunk** 的数据层,但它不是"东西",而是一个 0–15 的**数值场**。

关键:光有**两个独立通道**。

```
坐标 (x, y, z)
├── BlockLight : 0–15   ← 火把/岩浆/萤石等"光源"发出的光
└── SkyLight   : 0–15   ← 天空照下来的光(随昼夜与深度衰减)
```

- 存储:每格 **4 bit**,按 section 存成 `BlockLight` / `SkyLight` 两个 Byte_Array(存储那一篇见过)。两个通道分开存,因为一个跟时间无关(火把),一个跟昼夜有关(天光),渲染时要分别处理。
- **传播靠洪水填充(flood-fill)**:放下一支火把,光以"每跨一格 -1"的方式向四周扩散;挖掉一格不透明方块,光要重新灌进新腾出的空间。

### 踩坑:光照是著名性能热点

这套"光照引擎"在方块频繁变动时要做大量的重新传播,历史上是 Minecraft 的卡顿大户 —— Starlight、Phosphor 这类著名 mod 专门重写它来优化。原因正是 flood-fill 的级联:改一格可能引发一大片光值重算,和红石的更新风暴是同类问题。

## 五、粒子 Particle:纯客户端的"视觉幽灵"

火焰、烟、药水雾、爆炸碎屑这些:

- **只存在于客户端**:服务端不存、不 tick,存盘里完全没有它们。
- **没有逻辑、不可交互**:纯渲染效果。
- 它和实体长得像(有坐标、会动),但**完全是另一类东西**。

这解释了几个现象:粒子从不影响游戏结果;多人游戏里别人看到的粒子可能和你不完全一样(因为各自客户端独立生成);把画质调低关掉粒子,玩法毫发无损。

> 严格说粒子不算"世界数据",它是渲染层的产物。但既然在讨论"世界里还有什么",必须点明它的存在,以及它和实体的界线 —— **能存盘、能影响逻辑的才是世界对象,粒子两样都不是。**

---

## 六、把一格剖开:多层叠加

把前面几层合起来看,一个坐标的真实构成是这样的:

```
坐标 (x, y, z) ───────────────────────────────
  ├── BlockState   方块状态(网格,可共享,palette)
  ├── FluidState   流体状态(与方块叠加,独立 tick)
  ├── BlockLight   光源光照(0–15,4 bit)
  ├── SkyLight     天空光照(0–15,4 bit)
  └──(可能)BlockEntity   少数方块的附加数据/渲染

区域级(4×4×4 一格):
  └── Biome        气候/染色/刷怪标签

不在格子里:
  ├── ItemStack    寄生在容器/实体中的物品数据
  └── Particle     纯客户端视觉,不占数据
```

**最反直觉、也最关键的一点**:一格上叠着方块 + 流体 + 双通道光照好几层,它们各自独立存储、独立更新。"一格 = 一个方块"是新手最容易先入为主、却最该被打破的错觉。

---

## 串联

- 承接方块三层那一篇:方块、方块实体是"格子内容"和它的附加;这一篇补的流体、光照是**叠在同一格上的另外几层**,生物群系是**贴在格子上的标签**,物品是**寄生在容器里的数据**。合起来,"世界里的对象类别"才算齐。
- 呼应存储那一篇:chunk NBT 里的 `biomes`(4×4×4 palette)、`BlockLight`/`SkyLight`(Byte_Array)、`fluid_ticks`(独立队列)正是这一篇这些数据层的落盘形态 —— 你在磁盘上看到的字段,对应的就是这里讲的运行时分层。
- 指向注册表那一篇:这里每一类的**类型**(每种流体、每种物品、每种生物群系)同样登记在注册表里;生物群系更是"数据驱动的动态注册表"的典型,留到下一篇展开。

这一篇之后,"世界里的逐实例对象/数据层"基本就穷举完了:方块、流体、方块实体、实体、物品、生物群系、光照,外加不算世界数据的粒子。再往上,就是把这一切**类型**装起来的元系统了。
