# 体素世界、区块与无限地形的存法

你在 Minecraft 里向一个方向一直走,世界永远不会到尽头。你挖穿一座山、填平一个湖,关游戏再回来,改动还在。但你硬盘上的存档可能就几百 MB。一个"无限大、还能记住你每一次改动"的世界,凭什么塞得进这么点空间?

这一篇不讲玩法,先把 Minecraft 世界最底层的三件事讲清楚:**世界是什么数据结构、它怎么被切块管理、以及"无限"到底是怎么做到的**。这三件事理解了,后面所有篇——地形生成、渲染、tick、存储——都是顺着它们往下展开。

> 一句话先记住:**Minecraft 的世界是一个 3D 体素网格,按区块(chunk)切块加载;而"无限世界"的真相不是"存了无限多东西",而是"地形能用种子+噪声现场算出来,所以根本不存——只存你改过的那部分"。**

---

## 一、世界的本质:一个 3D 体素网格

先把"方块"这个词从游戏语境里抽出来,看它的数据本质。

Minecraft 的世界是一个三维的**体素(voxel)网格**——你可以理解成三维版的像素。空间被切成一个个 1×1×1 的格子,每个格子坐标 `(x, y, z)` 上,存的是"这一格是什么方块"。

```
体素网格 = 三维数组,每个格子存一个方块

       y (高度)
       │
       │   ┌──┬──┬──┐
       │  ╱  ╱  ╱  ╱│
       │ ┌──┬──┬──┐ │
       │ │石│石│空│ ┤   每个小立方体 = 一个 voxel
       │ ├──┼──┼──┤╱    存的不是模型,是"方块种类 + 状态"
       │ │土│土│水│
       └─┴──┴──┴──┴──── x
        ╱
       z
```

每一格存的不是一个 3D 模型,而是一个**轻量的标识**:方块种类(ID)+ 一些状态(朝向、是否含水、生长阶段……)。这一点至关重要——正因为单格的数据极小,海量方块才有可能被压缩存储。后续篇会讲到,这个"种类 + 状态"在引擎里叫 BlockState,千万个石头其实共享同一个状态对象,格子里只存一个指向它的下标。

### 为什么不能把方块当模型来渲染

新手最容易有的误解是:"每个方块是一个立方体模型,渲染就是把它们一个个画出来。"

算一笔账就知道这条路走不通:

- 一个区块底面 16×16,世界高度 384,就是 `16 × 16 × 384 ≈ 9.8 万`个方块。
- 玩家视野范围动辄几十个区块,**视野内轻松上千万、上亿个方块**。
- 每个立方体 6 个面、12 个三角形——逐方块提交,几十亿个三角形,任何显卡当场跪下。

```
朴素想法:N 个方块 → N 个立方体模型 → 几十亿三角形 → 炸显卡
现实做法:见 03 篇——只画"暴露在空气里的面",一个区块合并成一个网格
```

所以**体素世界的渲染绝不是"逐方块画"**。怎么把上亿方块压成显卡扛得住的几何体,是 `03 区块渲染` 专门要解决的问题。这里只需要记住一个结论:**方块的存储表示(数据)和渲染表示(几何)是两回事**,中间隔着一层专门的优化。

## 二、区块(Chunk):世界的管理单位

世界这么大,引擎不可能把它整个塞进内存。于是它被切成**区块(chunk)**——这是 Minecraft 里加载、生成、tick、存盘的基本单位。

### 区块的尺寸

```
1 个 Chunk = 16 (x) × 16 (z) × 整个世界高度 (y)

           ┌────────────────┐  ← 世界顶 (y=320, 1.18+)
           │                │
           │   一个 Chunk    │     底面 16×16
           │   竖直的一根     │     高度 384(从 y=-64 到 y=320)
           │   "柱子"        │
           │                │
           └────────────────┘  ← 世界底 (y=-64, 1.18+)
```

- **底面固定 16×16**,这是历史以来没变过的。
- **高度**:1.18 之前是 256(y=0~255),1.18 起扩展到 **384**(y=-64 ~ 319),所以能挖更深、山能更高。

### 再切一刀:Section(16³ 子区块)

一根 384 高的柱子还是太大,不好按需处理。所以区块在竖直方向再切成一段段 **16×16×16 的 Section**:

```
Chunk (16 × 16 × 384)
  = 24 个 Section 竖直叠起来

  Section 23   y=304~319
  Section 22   y=288~303
     ...
  Section  0   y=-64~-49
  每个 Section = 16³ = 4096 个方块
```

Section 才是真正的"工作粒度":存储用它(每个 Section 一张方块调色板)、光照按它算、网格按它重建。一个全是空气或全是石头的 Section 可以被极度压缩甚至省略——这点在 `08 存储` 篇会展开。

### 只加载你身边的区块

引擎只把**玩家附近的区块**加载进内存。你走动时,前方的区块被加载/生成,身后远处的区块被卸载、存盘。

```
       ▒▒▒▒▒▒▒▒▒▒▒          ▒ = 已卸载(不在内存)
       ▒░░░░░░░░░▒          ░ = 已加载(在内存,但外圈不更新)
       ▒░░░░░░░░░▒          █ = 玩家所在,完整运行
       ▒░░░░█░░░░▒
       ▒░░░░░░░░░▒          你往右走 →
       ▒░░░░░░░░░▒          右边新区块加载,左边旧区块卸载存盘
       ▒▒▒▒▒▒▒▒▒▒▒
```

注意"加载"也不是非黑即白:区块有不同的活跃等级,有的完整 tick,有的只是加载着但几乎不更新。这套"区块加载等级"决定了远处的农场会不会停摆,是 `05 区块的生命` 的主题。这里先建立一个印象:**任意时刻活在内存里的,只是整个无限世界里你周围一小圈。**

## 三、"无限世界"的真相:能算出来的就不存

现在回到开头那个问题:一个无限大的世界,怎么可能存得下?

答案是——**它根本不存"地形"**。

### 地形是算出来的,不是存出来的

Minecraft 的地形由**程序化生成**:用一组**噪声函数**,输入坐标,输出"这里该是什么"。关键性质是**确定性**:

> 同一个**种子(seed)** + 同一个**坐标**,噪声函数永远输出相同的结果。

这意味着第 1000 万格的那座山,不需要存在硬盘里——任何时候需要它,拿种子和坐标**当场重新算一遍**,得到的山一模一样。你没去过的地方,在磁盘上**根本不存在**,走过去的瞬间才被生成出来。

```
传统思路:世界 = 一份巨大的地形数据,全部存盘   → 无限世界 = 无限存储 ❌

Minecraft:世界 = 种子(几个字节) + 确定性噪声函数
          某坐标的地形 = f(种子, 坐标)  ← 要用时现算,不存 ✅
```

### 那你的改动呢?——只存"差异"

如果地形全靠现算,你挖掉的方块、盖的房子怎么办?

很简单:**只有被你改动过的区块,才会被写进硬盘。**

```
某个区块第一次被访问
   │
   ├─ 玩家从没改过它  → 不写盘。下次需要时用种子重新生成,结果一致
   │
   └─ 玩家改动过(挖了/放了方块) → 标记为 dirty,卸载/存档时写入磁盘
                                    以后从磁盘读,不再用噪声生成
```

所以存档大小**只正比于你实际探索并改造过的范围**,而不是世界的"总大小"(那是无限的)。这就是几百 MB 存档能撑起"无限世界"的全部秘密。

这套"差异存储"具体怎么落到文件里——哪些区块写进哪个文件、用什么格式——是 `07 存储(上)` 和 `08 存储(下)` 的内容。

### 这是贯穿全系列的一招

请记住这个思路,因为它不止用在地形上,而是**整个 Minecraft 引擎反复使用的核心招式**:

> **能算出来的就不存,能共享的就不复制。**

| 场景 | 这一招的体现 | 哪一篇细讲 |
| --- | --- | --- |
| 无限地形 | 不存地形,靠种子+噪声现算 | `01`(本篇)/ `02` |
| 方块状态 | 千万个石头共享一个状态对象,格子只存下标 | `09` |
| 区块存储 | 只存改过的区块;均质 Section 压到近乎为零 | `07` / `08` |
| 内存与磁盘 | 两边用同构的调色板结构,加载几乎不做转换 | `03` / `08` |

整个系列你都会一次次看到它。Minecraft 的"省"和"快",大半来自这一句话。

## 四、踩坑与反直觉

把这一篇容易误解的点单独拎出来:

- **"方块是一个个模型"——错。** 方块在数据层是"种类+状态"的轻量标识,在渲染层会被合并成大网格。把它想成模型,你既理解不了存储压缩,也理解不了渲染优化。

- **"世界是一个大数组全在内存里"——错。** 内存里只有你周围一圈区块,而且这一圈还分不同活跃等级。绝大部分世界要么在磁盘上(改过的),要么压根不存在(没去过的)。

- **"无限世界 = 无限存储"——错。** 无限的是"可生成空间",不是"已存储数据"。没探索的地方零成本,因为它只是种子函数的一个还没被求值的输入。

- **"种子只是个随机数好玩而已"——低估了。** 种子是整个世界确定性的根。同种子同版本,地形可完全复现——这正是"不存也能找回"的前提,也是种子分享、地形预览工具能存在的原因。

- **反直觉的一致性要求:** 因为地形靠"现算"补齐,生成算法**必须严格确定、且跨区块边界自洽**。否则你离开再回来,没改过的地形被重新算出一个不一样的结果,世界就"撕裂"了。这也是为什么地形生成不能简单地"每个区块各算各的",得有跨区块协调——细节见 `02 地形生成`。

---

## 五、串联

这一篇搭好了地基,三条线从这里岔出去:

- **世界怎么"长出来"** → `02 地形生成` 讲种子和噪声函数到底怎么把坐标变成山川洞穴、怎么保证跨区块自洽。
- **世界怎么"画出来"** → `03 区块渲染` 接着第一节那个"不能逐方块画"的坑,讲怎么只画可见面、把一个区块合并成一次绘制。
- **改动怎么"存下来"** → `07 / 08 存储` 把第三节的"只存差异"落到 Region 文件和 NBT/调色板的字节布局上。

而本篇反复强调的那句——**能算出来的就不存,能共享的就不复制**——会是你读完整个系列后,对 Minecraft 工程设计最该记住的一句话。
