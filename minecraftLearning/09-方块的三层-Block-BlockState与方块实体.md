# 方块的三层:Block、BlockState 与方块实体

你在世界里挖到的那块石头,在引擎眼里其实不是"一个对象"。如果你以为"每个方块格子里都装着一个方块对象,里面存着它的种类、朝向、坐标",那你会立刻撞上一个算术问题:一个加载区不到一百万个方块,一整个世界几十上百亿个方块,每个都装一个带字段的对象,内存和硬盘瞬间就炸了。

Minecraft 的答案是把"方块"这个看似单一的概念**拆成三层**,每一层解决不同的问题。理解了这三层的分工,你才能解释一连串反直觉现象:为什么千万个石头不占什么内存、为什么箱子里的物品能存住而石头里什么都存不了、为什么床这种平平无奇的方块居然是"方块实体"。

> 一句话先记住:**`Block` 是种类(单例,定义行为),`BlockState` 是带属性的具体状态(不可变、可共享、有全局编号),`BlockEntity` 是少数方块才有的逐实例附加数据**。绝大多数方块只用到前两层,而前两层里没有一个字节是"逐格子私有"的。

---

## 一、Block:种类与行为,全游戏一个实例

`Block` 是一个**类型**。全游戏里"石头"只有一个 `Blocks.STONE` 对象,世界里千万个石头格子**共享**这同一个对象。它只回答"这一类方块怎么表现":

- 硬度(挖多久)、爆炸抗性
- 碰撞箱形状、是否透光、是否可燃
- 被破坏时掉落什么
- 能不能被随机刻 tick(作物能,石头不能)
- 被邻居更新时怎么反应(`neighborChanged`)

关键在于它**不存任何"某一个具体位置的石头"的信息**。位置在哪?——位置是它在 section 数组里的**下标隐含表达**的,不需要存在方块对象里。

```
朴素心智(错):       引擎实际(对):
每格一个对象          Blocks.STONE  ← 唯一一个,所有石头共享
{ type:石头,         section 数组下标 = 位置(隐含)
  x,y,z, 朝向… }     格子里只放"指向哪个状态"的一个小整数
```

所以 `Block` 是"类",不是"实例"。把它想成 Java 里的 `Class` 对象就对了:描述行为,不描述某个具体个体。

---

## 二、BlockState:带属性的状态,可共享、有全局编号

光有"种类"不够。同一种方块有不同**状态**:楼梯朝哪、半砖在上还是在下、红石线几格电、作物长到第几阶段、方块里有没有灌水。

`BlockState` = `Block` + 一组**属性(property)** 的取值:

| 方块 | 典型属性 | 取值 |
| --- | --- | --- |
| 楼梯 | `facing` / `half` / `shape` / `waterlogged` | 4 × 2 × 5 × 2 = 80 种 |
| 红石线 | `power` + 各方向连接形态 | 0–15 等 |
| 作物 | `age` | 0–7 |
| 树叶 | `distance` / `persistent` | 离树干多远 / 是否人工放置 |
| 任意可灌水方块 | `waterlogged` | true / false |

每一种属性取值组合 = **一个独立的 BlockState**。橡木楼梯就有 80 个 BlockState。

### 全局调色板:给所有状态编号

游戏启动时,把所有方块的所有 BlockState 组合**枚举一遍、编上全局数字 ID**,形成一张全局表(目前两万多个 BlockState)。

最关键的性质:**BlockState 是不可变(immutable)、可共享(interned)的**。世界里所有"朝东、含水、上半"的橡木楼梯,指向的是**同一个 BlockState 对象**,不是各存一份。

这就解释了上一章 palette 为什么能极致压缩:

> 因为方块**没有逐实例数据**,存一个方块只需要存"它指向全局表里哪个 BlockState 的下标"。一段全是石头的区域,4096 个格子指向的是同一个下标——局部调色板里只有一项,几乎不占空间。

一句话:**Block 是类,BlockState 是这个类的"枚举值",二者都是单例级别的共享对象,世界里逐格存的只是一个指向它们的小整数。**

---

## 三、BlockEntity:少数方块的逐实例附加数据

问题来了:箱子里装了什么?告示牌写了什么字?熔炉烧到第几秒?这些是**每个具体位置都不同的数据**,塞不进"可共享的 BlockState"——你总不能给"装着 3 个钻石的箱子"和"装着 1 把剑的箱子"各开一个全局 BlockState。

于是有了**方块实体 `BlockEntity`**(老译名"tile entity"):

- 只有**少数需要额外数据的方块**才有:箱子、熔炉、告示牌、刷怪笼、信标、漏斗、命令方块……普通石头、泥土、楼梯**没有**。
- 它**绑定到一个具体坐标**,逐实例存在,数据用 **NBT** 序列化。
- 部分方块实体**每 tick 更新**(熔炉烧炼、漏斗搬运),这就是主循环里 `tickBlockEntities` 遍历的对象,也是它们贵的原因。

所以一个箱子方块 = **BlockState(朝向、是否含水,共享)** + **BlockEntity(里面的物品 NBT,私有)** 两部分拼起来。

```
普通方块(石头):   BlockState 下标   ……就这些
箱子方块:          BlockState 下标(朝向/含水)
                 + BlockEntity NBT(Items 列表、自定义名…)← 逐实例
```

到这里,"只有需要存额外数据的方块才有方块实体"这个印象看起来很自洽。但下面这个例子会把它打破。

---

## 四、床:一个几乎不存数据、却是方块实体的反例

床是方块实体——这是真的,而且它是个特别有意思的"反例"。

回想普通方块实体:箱子存物品、熔炉存进度,都是实打实的玩法数据。床呢?它的功能数据**根本不需要方块实体**:

| 床的状态 | 存在哪 |
| --- | --- |
| 朝向(头朝哪) | BlockState `facing` |
| 床头还是床尾 | BlockState `part = head / foot` |
| 有没有人睡 | BlockState `occupied` |
| 重生点设定 | 玩家数据 + 床方块的**行为逻辑**,与方块实体无关 |
| 颜色 | 每种颜色是**独立的方块**(16 种 `*_bed`),在方块种类里 |

你会发现:**床的所有玩法状态都进了 BlockState,没有一样非得放方块实体。** 那它为什么还有?

### 真正原因:渲染

床有方块实体,主要是为了挂一个**专用渲染器(`BlockEntityRenderer`,旧称 TileEntitySpecialRenderer)**。

普通方块用标准"方块模型系统"渲染——本质是一堆带贴图的立方体面,静态、简单。但床那个带床头床尾、布料质感、四条小腿的 3D 模型,超出了标准模型系统的表达能力,得用代码写一个专门渲染器逐个绘制,而**挂这个渲染器需要方块有一个 BlockEntity 当锚点**。

所以床的方块实体里那点数据基本是**给渲染器看的**,不是给玩法逻辑用的。由此得到一条更准确的判据:

> 一个方块在**需要逐实例数据**,**或需要标准模型系统画不出来的特殊渲染/动画**时,会带一个方块实体。床命中的是后半句。

这类"为渲染而存在"的方块实体不少:箱子(开盖动画)、告示牌(文字)、旗帜、头颅、潜影盒、唱片机、钟、信标光柱、海晶灯……它们分两类驱动力——存逐实例数据、或需要专用渲染器——很多两者兼有,而床几乎是纯第二类。

## 五、箱子:把两类职责合在一起的满分例子

床是"纯渲染型",箱子则**两类职责全占**,还多出一个难点:两个方块实体怎么合并成一个大箱子。

### 1. 数据职责:存物品

`ChestBlockEntity` 内部是一个定长列表,普通箱子 **27 个槽**,每槽一个 ItemStack。存盘序列化成 `Items` 列表:

```
Items: [
  { Slot: 0, id: "minecraft:diamond_sword", count: 1, components:{…} },
  { Slot: 5, id: "minecraft:torch", count: 64 }
]
```

注意**空槽不写**——只放了两组物品的箱子,`Items` 就两个元素,省空间。

### 2. 战利品箱的"懒生成"

地牢/沉船里的箱子,在你打开前其实是空的。它的方块实体不存物品,而是存:

```
LootTable: "minecraft:chests/simple_dungeon"   ← 用哪张战利品表
LootTableSeed: <随机种子>
```

**第一次被打开**时,才用这张表 + 这个种子现场生成物品,然后清掉这两个字段。好处:成千上万个未探索的箱子不预存物品,只存"配方 + 种子"两个值——这是"能算出来的就不存"这条主线在物品层的复用,和地形用种子生成同理。

### 3. 渲染职责:开盖动画

箱子用专用渲染器,读一个 **0.0–1.0 的开合度**把盖子绕铰链旋转画出来。这个值由 **`ChestLidController`** 管理:它有"当前开合度"和"目标值",**每 tick 朝目标插值逼近**,所以盖子是平滑掀开而非瞬间。因此箱子注册了一个 ticker——但这个动画 tick 很轻,和漏斗那种"每 tick 查容器"的重 tick 不是一回事。

### 4. 盖子怎么知道该开:观察者计数

开合靠 **`ContainerOpenersCounter`** 记录"当前有几个玩家正打开着它":

- 计数 **0 → 1**(第一个人打开):播放开箱音效、目标设为"开"
- 计数 **1 → 0**(最后一个人关闭):播放关箱音效、目标设为"关"
- 中间来来去去(2→3→2)不重复播音效

这就是为什么多人同时开同一个箱子盖子只开一次;也是为什么箱子上方放方块会卡住打不开——开盖判定里会检查盖子上方有没有障碍。

### 5. 大箱子:两个方块实体怎么变成一个

两个并排、同朝向的箱子合成 54 格大箱子,但底层**始终是两个独立方块、两个独立方块实体**,只是在逻辑和渲染两层被**临时配对**:

- **BlockState 标左右**:箱子有属性 `type = single | left | right`。放下第二个箱子时,通过邻居更新两个箱子各自变成 `left` / `right`。**配对关系由 BlockState 表达,不在方块实体里**——所以中间打断,靠 BlockState 重算就拆开了。
- **背包合并**:打开时临时创建一个 **`CompoundContainer`** 把两个 27 格拼成 54 格的**视图**,槽 0–26 路由到左箱、27–53 路由到右箱。**物品始终分存在两个方块实体里**,从没真合并;拆掉一半,另一半的 27 格原封不动还在它自己的方块实体里。
- **渲染合并**:`DoubleBlockCombiner` 根据 `type` 和 `facing` 找到相邻的另一半,渲染器据此选正确的半边模型拼成无缝大箱子,并用共享的盖子控制器让两半盖子**同步掀开**。

> 一句话:大箱子是"两个真实方块实体 + BlockState 标左右 + 打开/渲染时临时配对成一个视图"的产物。底层永远是二,体验上是一。

## 六、踩坑与反直觉

- **"每个方块都是对象"是错的。** 绝大多数方块在世界里只是一个指向共享 BlockState 的小整数,没有逐实例对象。只有那批进了方块实体白名单的方块,才在该坐标真正 new 出一个 BlockEntity。
- **"方块实体 = 存东西"不全对。** 床、唱片机这类的方块实体核心职责是渲染,几乎不存玩法数据。判据是"逐实例数据 **或** 特殊渲染"。
- **大箱子不是一个对象。** 它永远是两个方块实体,只在打开/渲染时被包装成一个视图。很多人以为大箱子内部合并了,其实拆开时数据各归各家正是因为从没合并过。
- **方块实体有运行成本。** `tickBlockEntities` 要逐个遍历有 ticker 的方块实体,这是漏斗物流卡服的根源。也正因如此,新设计倾向于"宁愿堆 BlockState 也不加方块实体"。

---

## 七、串联

- **承接上一章(存储/Palette)**:本章解释了 palette 为什么能压到极致——因为 Block 与 BlockState 都是共享的不可变对象,方块**无逐实例数据**,逐格存的只是调色板下标。方块实体则是 palette 装不下的那部分,单独走 NBT 落进 chunk 的 `block_entities` 列表。
- **对照"实体"那一章**:方块实体名字里带"实体",本质却是**方块的一部分**(绑定网格坐标),和能脱离网格自由移动、每个都 tick 的 `Entity` 是两套系统。下一步对比"床头朝向(BlockState) → 箱子物品(BlockEntity) → 掉地上的剑(Entity)"会更清楚。
- **指向"还有什么"那一章**:方块这一层之外,同一个坐标上其实还叠着流体(FluidState)和光照,物品又是另一类寄生数据——下一章把这些平行的对象类别一次铺开。

记住这三层的分工,你就拿到了读懂 Minecraft 数据模型的第一把钥匙:**Block 定义"这是什么",BlockState 定义"它什么样"(共享),BlockEntity 补上"这一个里装了什么"(私有)。**
