# 26-多平台导出、D3D12 与增量补丁

> 一句话导读:导出是把"在我电脑上能跑的项目"变成"用户拿到一个二进制就能装"的工程过程。这一篇讲清楚 Export Preset 的契约、4.6 在 Windows 上默认切到 Direct3D 12 之后的迁移心智,以及如何用 Delta 补丁 PCK 把更新包从几十兆压到几兆。

游戏发布之前,导出是最容易被低估的一步。代码跑通、画面好看、手感稳定都不等于"用户能装能玩"。一个 Godot 项目要变成 Windows 上的 `.exe` + `.pck`、Android 上的 `.apk`/`.aab`、Web 上的 `index.html` + `.wasm` + `.pck` 几套产物,每套产物背后都有一组不同的纹理压缩、渲染后端、签名、HTTP 头与文件系统约束。本篇按 PC / Web / Mobile 三条线展开,并展开 4.6 的两个关键变更:**Windows 新项目默认用 Direct3D 12 渲染**、**Delta-encoded patch PCK 让补丁包只装差异**。

## 1. 机制定位

新手关于导出常见的三个错觉:

第一个错觉是"我点 Project → Export → Export Project 一下就行了"。事实上 Export 弹窗只是 UI,真正的语义在 `export_presets.cfg` 文件里——这是项目根目录下一份逐平台的配置清单,定义了导出模板路径、包含/排除规则、纹理压缩选项、签名信息、渲染后端覆盖。CI 流水线和命令行 `godot --headless --export-release` 都依赖这份文件。你在 GUI 里拖的所有选项,都会立刻反映到这份文本里。把它纳入版本控制,是发布工程化的第一步。

第二个错觉是"PC build 跑通了等于多平台跑通"。各平台的差异主要在三处:

- **渲染后端**:Windows 4.6 新项目默认 D3D12,macOS / iOS 是 Metal,Linux / Android 多数还是 Vulkan,Web 是 WebGL2 / WebGPU,旧硬件需要 Compatibility 后端;
- **纹理压缩**:PC 用 BPTC/S3TC,移动用 ASTC/ETC2,Web 没有 VRAM 压缩(只能用 PNG/WebP),纹理在 `import_*` 项目设置里要先勾对应平台才会被烘焙;
- **平台 API 限制**:Web 因为 SharedArrayBuffer 需要 cross-origin isolated,iOS 要 codesign,Android 要 keystore,macOS 要 notarize。

第三个错觉是"补丁就是重打个 PCK"。在 4.6 之前确实只能这样,任何资源改动都会让对应文件整体进入补丁包,本地化文件改一个字母也要重打几十兆。4.6 引入 Delta-encoded patch PCK,让补丁 PCK 在文件内部做 zstd 二进制 diff,实测下 75% 的补丁体积削减并不夸张。

GDExtension 让你"把热点函数下沉到原生码不用动游戏代码";导出让你"把游戏从开发机搬到玩家机器上不丢东西"。两者的共同心智是:**Godot 这一侧给你一组可声明的契约,你只需要把契约写对,引擎和工具链负责把跨平台的脏活做完。**

## 2. Godot 心智

### 2.1 Export Preset 与 Export Template

一份 Export Preset 由两部分构成:

- **`export_presets.cfg`** 在项目根目录,定义"导出哪些文件、用什么 feature tag、用什么 export template、平台特定选项是什么";版本控制,人类可读;
- **Export Template** 是一份预编译好的 Godot 引擎二进制(Windows `.exe`、Linux `.elf`、Android `.aar`、Web `.zip` 等),作为你导出产物的运行时;通过 Project → Manage Export Templates 下载,放在 `~/.local/share/godot/export_templates/4.6.stable/` 之类位置。

导出的语义是"用 Export Template 这个壳,把你的项目资源 cook 成对应平台格式,然后写入 `.pck` / `.apk` / `.aab` / `.html+wasm+pck`"。这里有一个关键点:**Export Template 与编辑器引擎版本必须严格一致**。你拿 4.6.3 的编辑器、4.6.1 的 export template 去导出,运行时大概率会在 `ProjectSettings.load_resource_pack` 一类细节上炸掉。

每个 preset 都自带几个 feature tag,例如 `windows`、`pc`、`x86_64`、`mobile`、`web`、`s3tc`、`etc2`。这些 tag 决定 `ProjectSettings` 的 `Override For...` 机制能不能命中——例如 `display/window/size/viewport_width.web = 1280` 只在 Web 导出时生效。**这是跨平台分支配置的官方机制,别在游戏代码里手写 `OS.has_feature("web")` 满天飞。**

### 2.2 渲染后端的选择心智(4.6 重点)

4.6 之前,Godot 在 Windows 上的默认 RenderingDevice 后端是 Vulkan。4.6 把这个默认换成了 **Direct3D 12**,只影响 Windows 新建项目。具体行为有三条:

1. 4.6 及以后新建的 Windows 项目,`rendering/rendering_device/driver.windows = "d3d12"` 会自动写入项目设置;
2. 4.5 升级到 4.6 的老项目,**不会**被自动修改,你需要手动把这一行加进 `project.godot` 才能享受 D3D12 默认;
3. 全局默认值在 4.6 出于兼容仍是 `"vulkan"`,所以**没有显式声明**的项目继续用 Vulkan。

4.6 团队选 D3D12 的直接理由是 Vulkan 在 Windows 上的驱动稳定性持续不佳,Nvidia / AMD 多次出现驱动回退导致 Godot 游戏崩溃;Direct3D 12 路径在 4.3 加入、4.6 前打磨完毕,已经与 Vulkan 性能持平。

但有两类项目不能直接吃这个默认:

- **PCVR**:D3D12 后端目前缺 VRS 和 foveated rendering,VR 项目需要切回 Vulkan,把 `rendering/rendering_device/driver.windows` 显式设为 `"vulkan"`;
- **Wine / Proton**:Linux 上跑 Windows 版本时,系统 Wine 默认不支持 D3D12,Godot 会自动 fallback 到 Vulkan;Steam Proton 通过 vkd3d-proton 转译 D3D12,通常工作正常。

命令行临时切换:`--rendering-driver vulkan` 或 `--rendering-driver d3d12`,用户在游戏里也可以通过命令行参数覆盖。这条心智对 2D 项目影响小,但仍要养成"导出前两个后端都试一遍"的习惯——D3D12 与 Vulkan 在 shader 编译报错信息、纹理格式支持、HDR 行为上偶有细节差异。

另外两个非默认后端:`gl_compatibility` 用 OpenGL ES,目标是老硬件 / Web / 移动旧设备;`metal`(macOS / iOS 自动)。本系列 2D 项目这一篇默认讨论 RenderingDevice 路径(Vulkan / D3D12 / Metal),Compatibility 后端只在最后导出时按需切换。

### 2.3 PCK / 资源包模型

Godot 导出产物的核心是 `.pck` 文件,一个**未压缩**的资源包,里面包含项目的 `.import`、`.tres`、`.tscn`、`.gd` 二进制 token、纹理 / 音频 cook 后的二进制等等。运行时的可执行文件(`.exe` / `.elf` / `.app`)启动后会自动加载同名同目录的 `.pck`(或者嵌入到 exe 里,通过 export 选项 "Embed PCK" 控制)。

`.pck` 还可以**作为运行时的额外资源包**通过 `ProjectSettings.load_resource_pack("user://content/dlc01.pck")` 挂载,挂载后该包里的资源会**覆盖**已加载包的同路径资源——这是模组、DLC、补丁的统一机制。如果你想让某个 mod **不**覆盖原资源,把第二个参数设为 `false`。

补丁场景里有一个关键技术细节:**4.6 引入的 Delta-encoded patch PCK 在文件级别做 zstd 二进制 diff**。原来的 patch PCK 是"哪个文件改了就把整个文件打进去",4.6 之后是"哪个文件改了,只把差异字节打进去,运行时用 `FileAccessPatched` 透明地应用差异"。这把"改一个本地化字符串就要重发 30 MB 翻译文件"的代价压到几 KB 级别。代价是每次读这个文件多花几毫秒做差异应用,因此默认关闭,需要在 Export 对话框 "Patching" tab 显式启用。

## 3. 工程实现

### 3.1 三个平台的 Export Preset

新建项目后第一件事是把三个 preset 写进 `export_presets.cfg`,而不是在 GUI 里一项项点。文件实际语法是 ini 风格:

```ini
[preset.0]
name="Windows"
platform="Windows Desktop"
runnable=true
export_filter="all_resources"
export_path="build/windows/cave_runner.exe"

[preset.0.options]
binary_format/architecture="x86_64"
binary_format/embed_pck=false
codesign/enable=false
rendering/d3d12/agility_sdk_multiarch=true
texture_format/s3tc_bptc=true
texture_format/etc2_astc=false

[preset.1]
name="Web"
platform="Web"
runnable=true
export_filter="all_resources"
export_path="build/web/index.html"

[preset.1.options]
custom_template/debug=""
custom_template/release=""
variant/extensions_support=false
variant/thread_support=false
vram_texture_compression/for_desktop=false
vram_texture_compression/for_mobile=false
html/export_icon=true
html/canvas_resize_policy=2
progressive_web_app/enabled=true
progressive_web_app/ensure_cross_origin_isolation_headers=true

[preset.2]
name="Android"
platform="Android"
runnable=true
export_filter="all_resources"
export_path="build/android/cave_runner.apk"

[preset.2.options]
custom_template/use_wer_no_window=false
gradle_build/use_gradle_build=true
gradle_build/min_sdk=""
gradle_build/target_sdk=""
architectures/armeabi-v7a=false
architectures/arm64-v8a=true
architectures/x86_64=false
package/unique_name="com.example.caverunner"
package/name="Cave Runner"
package/signed=true
texture_format/etc2_astc=true
texture_format/s3tc_bptc=false
```

几条工程要点:

- `export_filter="all_resources"` 把项目里所有资源都纳入,等价于 GUI "Export all resources in project";想细粒度控制改成 `"resources"` + `include_filter`;
- Web preset 里 `variant/thread_support=false` 是 4.6 推荐的"单线程导出"模式,绕开 SharedArrayBuffer 与 cross-origin isolation 的部署摩擦(下面 4.3 节展开);如果你确实需要 WorkerThreadPool 性能,把它打开,然后接受 COOP/COEP 配置;
- `progressive_web_app/ensure_cross_origin_isolation_headers=true` 让生成的 service worker 自动给请求加 COOP / COEP 头,这是不能控制服务器响应头时唯一的解;
- Android `gradle_build/use_gradle_build=true` 是"用本地 Gradle 完整构建 AAB",而不是只签名预编译模板。4.6 在 Android 上推荐用 Gradle 构建,这样能加自定义权限、自定义 Activity、第三方 Android 库;
- `package/signed=true` 加上 keystore 路径就能在 CI 上自动签名;移动端从未签名 APK 变成可上架的 AAB,这一步是硬要求。

### 3.2 命令行导出

CI 流水线和发版脚本里用 GUI 是浪费,直接命令行:

```text
godot --headless --export-release "Windows"  build/windows/cave_runner.exe
godot --headless --export-release "Web"      build/web/index.html
godot --headless --export-release "Android"  build/android/cave_runner.aab
```

第一个参数是 `[preset.N]` 中 `name` 字段的字符串,第二个是输出路径(覆盖 preset 里的 `export_path`)。`--export-debug` 走 debug template,带 debugger stub,适合内测。一台没有显卡的 CI 机器也能跑 `--headless`,Godot 在该模式下不初始化窗口与渲染设备,只做资源 cook 与打包。

### 3.3 Delta 补丁 PCK 工作流

这一节是 4.6 的标志性新工作流。假设你 1.0 版本已经发布,现在要发 1.0.1 补丁,只改了一些本地化字符串和几个小怪的属性资源。流程:

1. 在 Project → Export 打开你的发版 preset,切到 **Patching** tab;
2. **Base 段**:指定 1.0 版本对应的 PCK 文件路径(可以是本地或网络资源)。引擎会以这份 PCK 为对比基准;
3. 勾选 **Enable delta encoding**(4.6 新选项,默认关);
4. 点 Export PCK/Zip,输出 `cave_runner_1.0.1.pck`;
5. 把这份 PCK 分发给玩家,玩家把它放进游戏目录的 `patches/` 子目录,游戏启动时 `ProjectSettings.load_resource_pack(...)` 顺序加载这些补丁。

加载顺序要在游戏启动早期做完,通常在 `autoload` 的 `_ready()` 里。一个 minimal 加载器:

```gdscript
extends Node
class_name PatchLoader

const PATCH_DIR := "user://patches"

func _ready() -> void:
	var dir := DirAccess.open(PATCH_DIR)
	if dir == null:
		return
	var entries: PackedStringArray = dir.get_files()
	# 按文件名排序,确保 1.0.1.pck 比 1.0.2.pck 先加载。
	entries.sort()
	for filename in entries:
		if not filename.ends_with(".pck"):
			continue
		var full_path: String = "%s/%s" % [PATCH_DIR, filename]
		var ok: bool = ProjectSettings.load_resource_pack(full_path, true)
		if ok:
			print("Loaded patch: ", filename)
		else:
			push_error("Failed to load patch: " + full_path)

func get_loaded_patches() -> PackedStringArray:
	var dir := DirAccess.open(PATCH_DIR)
	if dir == null:
		return PackedStringArray()
	return dir.get_files()
```

注意 `load_resource_pack` 的第二个参数 `replace_files` 默认 `true`,意味着补丁里的同路径文件**覆盖**基础包;mod 场景里如果你不想覆盖,显式传 `false`。

运行时差异应用在引擎内 `FileAccessPatched` 这一层透明完成,玩家代码不用感知。代价是首次访问被补丁覆盖的资源会多花几毫秒解 diff,后续 `ResourceLoader` 缓存命中后回到正常成本——所以补丁不适合给"每帧都重新 load"的资源用,但这种用法本身就是异常,正常游戏没有问题。

### 3.4 哪些文件适合走 Delta,哪些不适合

Delta encoding 在文件层用 zstd 做 binary diff,**diff 的前提是源文件和目标文件在字节上有大量共享前缀**。这决定了:

- **大文本资源**(`.csv` 本地化表、`.json` 关卡描述、`.tres` 配表):diff 极佳,千行表改一行 diff 几百字节;
- **GDScript 二进制 token**(`.gdc`):diff 一般,token 序列对相邻改动有局部性,但默认导出走 "Compressed binary tokens",压缩后字节相似度被破坏,diff 通常 fall back 到全文件;
- **纹理 / 音频 cook 产物**(`.ctex` / `.ogg`):diff 极差,因为这些格式自身已经经过有损压缩,改一个像素整张图字节流都会变;Godot 会在 delta 超过原文件某个比例时**自动放弃 delta**、直接整文件入包,日志里能看到"falling below reduction threshold"的提示。

经验法则:**做 patch 时,把改动集中在文本类资源上。如果一定要改某张大图,接受这部分 patch 体积不会缩。**

### 3.5 CI 上的版本管控

发版工作流通常涉及三份元数据:Godot 引擎版本、export template 版本、项目 commit hash。把这三者绑成一个 release tag,例如 `v1.0.1+godot4.6.3`,可以让玩家与你各自从崩溃日志里反推出问题来自哪一次构建。在 `Globals.gd`(autoload)里:

```gdscript
extends Node

const GAME_VERSION := "1.0.1"
const GODOT_REQUIRED := "4.6"

func _ready() -> void:
	var ver: Dictionary = Engine.get_version_info()
	var current: String = "%d.%d.%d" % [ver.major, ver.minor, ver.patch]
	print("Game ", GAME_VERSION, " on Godot ", current, " (", ver.status, ")")
```

`Engine.get_version_info()` 在 4.6 返回 `{major, minor, patch, hex, status, build, year, hash, timestamp, string}`,这些字段足够 CI 在构建时校验:如果 export template 与项目要求的 major.minor 不一致,直接 fail 构建,不要把这个错误留给玩家。

## 4. 调参和验收

### 4.1 渲染后端选择矩阵

| 平台 | 4.6 默认 | 何时切别的 |
| --- | --- | --- |
| Windows 新项目 | D3D12 | PCVR(切 Vulkan)/ 老 GPU 不支持 D3D12(切 Compatibility) |
| Windows 4.5 升级项目 | Vulkan(未改) | 想吃 D3D12 的驱动稳定性,手工改 `driver.windows = "d3d12"` |
| Linux | Vulkan | 老硬件切 Compatibility |
| macOS / iOS | Metal | 几乎不动 |
| Android | Vulkan / Compatibility | 低端机切 Compatibility |
| Web | WebGL2 (Compatibility) | 实验性 WebGPU |

要点是**导出前两套都跑一遍**:同样一段 shader,在 D3D12 上能编译不代表在 Vulkan 上也能编译,反之亦然。HLSL 与 GLSL 在精度限定、整数语义上有微小差异,引擎做翻译但不百分百兼容。

### 4.2 纹理压缩与项目设置

`project.godot` 里两条关键设置:

- `rendering/textures/vram_compression/import_s3tc_bptc = true` — 桌面 / 主机平台需要;
- `rendering/textures/vram_compression/import_etc2_astc = true` — 移动平台需要;

**勾选**后还要重新 import 所有纹理(删 `.godot/imported/` 文件夹再启动编辑器),否则只有新导入的纹理走新格式,老的还停留在旧 cook 结果。这是新手最容易踩的坑之一:勾了选项以为生效了,实际导出还是 PNG 原图,发现包体根本没缩。

2D 项目里"VRAM Compressed"对于像素风 sprite 通常不是好选择,会引入肉眼可见 artifact。像素风默认用 **Lossless**,矢量/高清贴图用 **Lossy(WebP)**,只有大尺寸非像素的非 UI 贴图考虑 VRAM Compressed。

### 4.3 Web 导出的 SharedArrayBuffer 心智

Web 端最容易翻车的一处:`SharedArrayBuffer` 在浏览器里要求页面处于 **cross-origin isolated** 状态,这要求服务器返回:

```text
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

两条 HTTP 响应头必须由**服务端**设置,不能通过 `<meta>` 注入。3 种实际解法:

1. **关线程**:`variant/thread_support=false`,放弃 `WorkerThreadPool` 但能跑在任何静态服务器上,适合简单 2D 游戏;
2. **PWA service worker**:`progressive_web_app/enabled=true` + `ensure_cross_origin_isolation_headers=true`,Godot 自动生成的 service worker 在二次访问时给所有同源请求注入这两个头;首次访问需要 reload 一次,首次失败属正常;
3. **服务端配置**:Netlify 用 `_headers`,Vercel 用 `vercel.json`,nginx 用 `add_header`,Cloudflare Workers 也可以;直接控制服务器响应头是最可靠的方案。

部署后用浏览器 DevTools Console 跑 `window.crossOriginIsolated` 验证:返回 `true` 才有效,`false` 必崩。

`itch.io` 是独立游戏常用的发布目标,它支持在游戏设置里勾 "SharedArrayBuffer support",但用的是 Chrome 专属的 `credentialless` COEP 策略,Firefox 兼容性不全;如果你的游戏开线程导出且要在 itch.io 上玩,接受"Firefox 用户可能跑不起来"的现实,或者干脆走单线程导出。

### 4.4 移动端最低配置

Android 4.6 推荐 `gradle_build/use_gradle_build=true`,这要求开发机装 JDK 17、Android SDK、Android Build Tools。debug build 可以用预编译模板省略 Gradle,但 release / AAB 上传 Play Store 必须 Gradle build。Architecture 至少留 `arm64-v8a`,Play Store 2025 起强制 64-bit;armv7 与 x86 可选。

iOS 必须有 Apple Developer 账号、provisioning profile、Xcode 安装。Godot 4.6 把整个 iOS 项目导出成 Xcode 工程,然后你用 Xcode 打 ipa 上传 App Store Connect——这部分严格意义不是 Godot 的工作,是 Apple 工具链的工作,Godot 只负责到 Xcode 工程为止。

### 4.5 包体大小预算

一个空 Godot 4.6 项目的导出产物:

- Windows: 60 MB(主要是 Godot runtime + Vulkan/D3D12 loader);
- Web: 25 MB(wasm 16 MB + js 1 MB + 模板),压缩后 ~10 MB;
- Android AAB: 30 MB(armv8 only);
- macOS: 80 MB(universal binary);

如果你的项目体积比"空项目 + 资源原始体积"还大很多,通常是没勾对纹理压缩或者带了未使用的 export template。

### 4.6 Feature tag 与平台分支配置

`ProjectSettings` 的 Override 机制对所有 setting 都生效。常见用法:

```text
display/window/size/viewport_width = 1920
display/window/size/viewport_width.mobile = 720
display/window/size/viewport_width.web = 1280

rendering/textures/canvas_textures/default_texture_filter = 0
rendering/textures/canvas_textures/default_texture_filter.mobile = 1
```

`.mobile`、`.web`、`.pc`、`.windows`、`.linuxbsd`、`.macos`、`.android`、`.ios` 都是合法的后缀,导出时引擎根据当前 preset 的 feature tag 集合命中最具体的那一行。这条机制比"在 GDScript 里 `if OS.has_feature('web')`"更可靠,因为它发生在引擎加载层,不会被代码路径漏掉。

如果你想自定义 feature tag,在 preset 的 Custom Features 字段写入逗号分隔的字符串,例如 "demo,steam"。这样可以做"demo 版打包时不包含某些资源"或者"Steam 平台单独 enable 成就上报模块"。Custom feature 在 `OS.has_feature("demo")` 里同样可用,工程化非常顺手。

## 5. 踩坑

### 5.1 D3D12 默认在升级项目上不生效

4.5 项目升 4.6 之后,Windows 上你以为"现在用 D3D12 了",其实没换。原因在 2.2 节讲过:全局默认仍是 Vulkan,只有 4.6 新建项目才会自动写 `driver.windows = "d3d12"`。要让升级项目吃这个新默认,手动编辑 `project.godot`:

```text
[rendering]
rendering_device/driver.windows = "d3d12"
```

或者编辑器 Project Settings → Rendering → Rendering Device → Driver Windows → 选 d3d12。改完后重启编辑器,首次启动会触发 D3D12 shader cache 重建(几秒到几十秒),不要中断。

### 5.2 Delta 补丁与场景确定性回归

4.6 引入了一个不大不小的回归:**所有场景里的节点会自动获得一个 `unique_id`**,这个 ID 在 4.6 的某个改动里不是确定性生成的。具体表现是:你在 4.5 保存的 `.tscn` 文件,在 4.6 不重新保存就直接导出 patch PCK,**每次导出都会把这个场景文件包含进 delta**,即便你没改它——因为引擎把场景重新序列化时生成了新的 unique_id。

补救方法:**4.5→4.6 升级后,在 Godot 编辑器里把所有场景手动 reopen + ctrl+s 一遍**,让 unique_id 落到磁盘上。之后导出 patch 才会真正只包含改过的场景。这条坑写得很丑陋,但在确实有官方修复出来之前,只能靠流程绕开。

### 5.3 Web cross-origin isolation 配置半生不熟

最常见现象:本地 `python -m http.server 8000` 跑游戏,跑得起来;部署到生产 nginx,加载到一半报 "Failed to instantiate SharedArrayBuffer"。原因是本地服务器即使没设 COOP/COEP,浏览器有时会因为 `localhost` 走宽松模式让它过;一旦换到正式域名,严格模式启用,缺少头就崩。

诊断套路:打开 DevTools Console 输入 `window.crossOriginIsolated`,如果返回 `false`,先看 Network tab 里 index.html 的响应头,确认 COOP / COEP 是否真的被服务器发回。这两个头一旦设了,所有跨域资源(第三方分析脚本、CDN 上的字体、外链音频)都需要 `Cross-Origin-Resource-Policy` 或者带 CORS 头才能加载,否则会被浏览器拒绝。开发期常见做法是先把所有第三方依赖剥掉再调,等基础跑通再逐个加回去。

### 5.4 PWA service worker 缓存陈旧

打开 PWA 之后,Godot 生成的 service worker 会缓存所有 game asset(`.wasm` / `.pck` / `.js`)到浏览器本地。问题是发新版本后,service worker 不会自动清旧缓存,玩家可能拿到一份新 wasm + 旧 pck,然后报符号不对、版本不匹配崩溃。Godot 在 service worker 模板里加了基于文件哈希的版本号,但只有在 service worker 自身被替换时才会触发清理,首次 reload 后玩家通常还在旧 worker 里。

让玩家"强制刷新"不是真正的解。工程做法:

- 在游戏启动后用 `Engine.get_version_info()` 写一行版本号到 HTML overlay,玩家看到旧版本知道要 reload;
- service worker 的 `skipWaiting()` 在 Godot 4.6 模板里默认是关的,定制模板里打开,新 worker 立即接管;
- 重大版本变更时直接换 PWA 名字(`progressive_web_app/name`),触发浏览器全量重新注册。

### 5.5 Android Gradle 构建在 CI 上的依赖陷阱

`gradle_build/use_gradle_build=true` 会在 CI 上要求一整套 Android 工具链。最常见的踩坑是 Android SDK 没下全 NDK 版本不匹配,然后 Gradle 报 "Could not find NDK r21" 然后死循环。CI 配置策略:

- 用 `reactivecircus/android-emulator-runner` 之类的预制 Action,而不是自己装 SDK;
- 把 Gradle wrapper 跟项目走,版本固定;
- 第一次 CI 跑 Gradle 任务多预留 5-10 分钟下载缓存,后续靠 GitHub Actions 缓存压到 1 分钟。

### 5.6 PCK 跨架构不可用

`.pck` 文件本身是架构无关的资源容器,但里面的 GDScript 二进制 token、自定义 GDExtension 库可能不是。如果你的项目里有 GDExtension(参见 25 篇),`.pck` 里**不会**包含 `.so` / `.dll`,这些库依然是分别打包的;但如果你打了 PCK 给 ARM 用户,然后 ARM 那边找不到对应平台的扩展库,游戏启动时会报"Could not load extension"。同一份 PCK 跨 x86_64 / arm64 通常能用,但每个架构需要自己的扩展二进制,这条契约不能省。

### 5.7 Embed PCK 与可执行修改

Windows preset 里 `binary_format/embed_pck=true` 会把 PCK 直接嵌进 `.exe`,产物只剩一个文件,看起来"更干净"。代价有两条:

- 修改 PCK 等于修改 exe 二进制,在 Windows 上会触发 Defender SmartScreen 重新审计;
- 文件大于 2 GB 时嵌入会失败(老 PE 格式限制);
- 反作弊 / 防破解工具看到一个又大又怪的 exe 段会报警。

发布版本通常关掉这个选项,把 `.exe` + `.pck` 一起打 zip 给玩家;mod 友好的项目尤其要这么做,因为 mod 工具要能识别 PCK 文件。

### 5.8 D3D12 的额外运行时文件

D3D12 后端在导出时会附带 **Agility SDK** 二进制(`D3D12Core.dll`、`d3d12SDKLayers.dll` 等),放在 exe 同目录的 `D3D12/` 子目录。这些文件不能漏,否则 D3D12 后端启动失败、引擎 fallback 到 Vulkan(如果 Vulkan 没禁的话)或直接黑屏。CI 打包脚本里要确认这个目录跟 exe 一起进 zip。

### 5.9 export_filter 与 .gdignore

`export_filter` 字段的几个值含义并不直观:`all_resources` 把项目里所有资源打入 PCK,`resources` 只打用到的资源(根据场景依赖图遍历),`exclude` 排除某些。问题在于 Godot 的依赖追踪不是完美的——`load()` 用字符串拼接的路径、`preload()` 用变量、`ResourceLoader.load_threaded_request()` 这些动态加载都会让依赖追踪漏掉资源。生产环境推荐 `all_resources`,牺牲一些包体换稳定性;然后用项目根目录的 `.gdignore` 排除明确不需要的目录(`tools/`、`docs/`、`tests/`)——`.gdignore` 是空文本占位文件,放在哪个文件夹下,Godot 就把那个文件夹完全无视,既不导入也不导出。这条机制比 `export_filter` 黑名单维护起来轻得多。
