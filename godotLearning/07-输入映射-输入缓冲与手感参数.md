# 07-输入映射、输入缓冲与手感参数

> 一句话导读:手感不是"角色跑得快不快",而是"玩家按下按钮到屏幕反馈之间,有没有出现意外的等待、忽略或滑过"。这一篇把按键到位移之间的所有间隙都拆开。

第 06 篇把 `CharacterBody2D.move_and_slide` 跑通,角色已经能稳定在地形上行走。但只要把手柄交给别人,新手做出来的角色就立刻露馅:跳跃必须在着地那一帧才管用,松开方向键人物像踩了冰一样滑,被打中那一下毫无重量。这些问题的修法不是"加更大的速度",而是把按键事件与物理 tick 之间的对齐方式重新做一遍。

本篇覆盖 Godot 4.6 下的 `InputMap` 配置、`Input` 单例的几种读取方式、输入缓冲(jump buffer)、土狼时间(coyote time)、加减速度模型,以及命中停顿(hit stop)。所有数值都给出参考区间,但要强调一点:手感参数不是抄数值,是用数值在你这个游戏的尺度下推一遍。**没有"通用最佳值",只有"在角色身高、跳跃高度、关卡格子尺寸的相对比例下、玩家手指能识别的窗口长度"。** 这层"相对"是手感工程的核心。

## 1. 机制定位

### 1.1 手感的"间隙"在哪

按下跳跃键到角色离开地面之间,事实上存在多层间隙:

1. 物理设备到操作系统的事件分发(键盘扫描、手柄轮询),通常 1-8 ms。
2. 操作系统到 Godot 主循环的事件队列(`OS::process_events`)。
3. Godot 主循环里 `_input` / `_unhandled_input` 的回调,每帧一次。
4. `_physics_process` 物理 tick,默认 60 Hz,即 16.67 ms 一次。
5. `move_and_slide` 修改 `position`,渲染下一帧把它画出来。

只要按键发生在两次物理 tick 之间,直接用 `Input.is_action_pressed("jump") and is_on_floor()` 就有可能漏掉:角色已经离开地面 1-3 帧,但玩家以为自己按对了。**输入缓冲解决"按早了",土狼时间解决"按晚了"。**

### 1.2 新手最容易踩的三个错

- **在 `_process` 里读按键改物理**:渲染帧率与物理帧率不一致时,会出现一个物理 tick 内读了两次按键、或一次没读到的情况,jump 偶尔丢失。
- **靠 `velocity.x = direction * speed`**:松键瞬间速度归零,角色像粘在墙上;落地瞬间速度强行设定,会切掉空中加速度的累积。
- **hit stop 用 `await get_tree().create_timer(0.05).timeout`**:不能暂停物理,只是让代码等一下,角色仍在被推,看起来不是"停顿"而是"卡顿"。

正确的做法是把"输入意图"作为一层独立状态,让物理 tick 去消费这个意图,而不是直接在物理 tick 里查询硬件。

### 1.3 手感参数之间的依存关系

这一篇要调的参数有十几个,但它们并非各自独立。一组好用的取值之间存在四条隐式依赖:

- **跳跃高度 = `jump_velocity` 与 `gravity_up` 共同决定**。改了重力就要重新算跳跃初速度,而不是各调各的。理论高度 `h = jump_velocity² / (2 * gravity_up)`(以像素为单位),目标高度通常落在 4-6 个角色身高。
- **加速度与最大速度共同决定"到顶速所需时间"**。`t = max_speed / accel`,典型紧手感是 80-120 ms,漂浮感是 250-400 ms。这个时间感知比 `accel` 的绝对值更直观。
- **`jump_buffer` 与最长动画前摇耦合**。如果"起跳前摇"动画有 8 帧(133 ms),缓冲必须 >= 这个长度,否则缓冲在动画结束前过期,玩家会感觉跳跃被吃掉。
- **`coyote_time` 与平台水平移动速度耦合**。一个 32 px 的小平台,角色 200 px/s 通过它要 160 ms,coyote 不能超过这个值,否则离开平台仍能跳就会失控。

调参时要把这四条放在脑子里。先定跳跃高度,再算重力,再调加速度,最后定缓冲。

## 2. Godot 心智

### 2.1 `InputMap` 的位置

`InputMap` 是项目级的 `Singleton`,把硬件事件(键盘扫码、鼠标按键、手柄按键、手柄轴)聚合为"动作"(StringName)。所有游戏逻辑只看动作名,不看具体按键。这层抽象有两个回报:重映射、跨设备适配几乎是零成本;在编辑器之外仍然能通过 `InputMap.add_action` 在运行时改键。

Godot 4.6 中,InputMap 在 `项目设置 → Input Map` 里编辑。每个 Action 有一个 `deadzone`(0-1),专门给手柄轴用,键盘事件不受影响。建议把常用动作显式写出来,避免依赖 `ui_left` / `ui_right` 这类 UI 默认动作(它们与控件聚焦行为绑定,后期 HUD 引入时会冲突)。

### 2.2 `Input` 单例的四种读法

`Input` 是运行时的状态机,只关心"现在按着什么",但提供了"刚刚按下/刚刚松开"的边沿判断:

| 方法 | 语义 | 用在哪里 |
| --- | --- | --- |
| `Input.is_action_pressed(name)` | 此刻按住 | 持续状态,如移动 |
| `Input.is_action_just_pressed(name)` | 本帧按下边沿 | 跳跃、攻击触发 |
| `Input.is_action_just_released(name)` | 本帧松开边沿 | 跳跃高度截断、放技能 |
| `Input.get_axis(neg, pos)` / `Input.get_vector(...)` | 归一化的方向 | 横向移动、摇杆 |

"刚刚"的语义在 `_process` 和 `_physics_process` 里都成立,但**它是按帧失效的**——一旦本帧过去,`just_pressed` 立刻回到 false。这意味着:如果你在 `_physics_process` 里读它,而本帧物理 tick 没跑(渲染帧率 > 60 Hz 时,物理 tick 可能本帧没触发),那次按下就丢了。Godot 内部会做一次 process 与 physics 之间的状态保持,大多数情况下没问题,但**输入缓冲不能依赖这层保险**,要自己在 `_input` 里把意图记下来。

### 2.3 事件回调与轮询的分工

```text
_input(event)               # 全局事件,最早,UI 还没消费
_unhandled_input(event)     # UI 消费后的事件,适合游戏世界
_process(delta)             # 每渲染帧;轮询、渲染
_physics_process(delta)     # 每物理 tick;移动、碰撞
```

对手感代码,本系列推荐:**`_unhandled_input` 抓离散按下事件(跳跃、攻击),把意图写进缓冲;`_physics_process` 轮询持续按键(方向),消费缓冲,推进物理。** 这样既避免跟 UI 抢按键,又让物理 tick 永远拿到稳定的意图,而不必关心硬件抖动。

为什么不在 `_physics_process` 里直接用 `Input.is_action_just_pressed`?它"能"工作,但有一个边界情况:渲染帧率远高于物理帧率(VSync 关闭、144 Hz 显示器),一个物理 tick 内可能跨越多个渲染帧。Godot 内部会把"刚刚按下"的事件保留到下一次 `_physics_process` 触发,但**只保留一次**——如果玩家在一个物理 tick 内连按两次跳跃,第二次会丢。这个情形在节奏类游戏(踩点跳)里能被玩家感知到。InputBuffer 用时间窗口而不是事件计数,正好避开这一类丢失。

### 2.4 hit stop 的本质

命中停顿(hit stop / freeze frame)是格斗、动作类 2D 必备的反馈技巧:命中瞬间把双方移动 / 动画暂停 50-150 ms,让玩家"感觉打到了"。在 Godot 里有两种实现方式:

- `Engine.time_scale = 0.0`:暂停整个引擎,代价是 UI 与镜头震动一并停下。
- 自定义"冻结字段":每个角色脚本里维护一个 `_freeze_timer`,大于 0 时跳过 `velocity` 更新与 `AnimationPlayer.advance`。

第二种方式更工程化:可以只冻结战斗参与方,镜头、粒子、HUD 不受影响。第 08 篇会展开动画的暂停接口,这里只在角色控制层留出钩子。

## 3. 工程实现

### 3.1 配置 InputMap

把动作集中在 `project.godot` 文件里,避免散落在脚本里。下面是这一阶段角色控制需要的动作清单,在 `项目设置 → Input Map` 里依次加好:

| Action | 键盘 | 手柄 |
| --- | --- | --- |
| `move_left` | A、Left | LStick Left, DPad Left |
| `move_right` | D、Right | LStick Right, DPad Right |
| `jump` | Space、Z | A 键(Xbox) |
| `attack` | J、X | X 键(Xbox) |
| `dash` | Shift、C | RB |

可以用脚本一次性生成(适合写测试或多人开发时统一基线),路径 `res://autoload/input_bootstrap.gd`,设为 Autoload:

```gdscript
extends Node
## 在游戏首次启动时检查 InputMap,补全本游戏所需的动作。
## 已有动作不会被覆盖,只补充键位,避免与玩家自定义冲突。

const DEFAULTS: Dictionary = {
    "move_left":  [KEY_A, KEY_LEFT],
    "move_right": [KEY_D, KEY_RIGHT],
    "jump":       [KEY_SPACE, KEY_Z],
    "attack":     [KEY_J, KEY_X],
    "dash":       [KEY_SHIFT, KEY_C],
}

func _ready() -> void:
    for action_name in DEFAULTS.keys():
        var action: StringName = action_name
        if not InputMap.has_action(action):
            InputMap.add_action(action, 0.2)
        for keycode in DEFAULTS[action]:
            if not _action_has_key(action, keycode):
                var event := InputEventKey.new()
                event.physical_keycode = keycode
                InputMap.action_add_event(action, event)

func _action_has_key(action: StringName, keycode: int) -> bool:
    for event in InputMap.action_get_events(action):
        if event is InputEventKey and event.physical_keycode == keycode:
            return true
    return false
```

这里用 `physical_keycode` 而不是 `keycode`:前者按物理键位匹配,AZERTY、Dvorak 键盘上 WASD 仍然在左手区。重映射界面(第 27 篇)会基于这一层做覆盖。

### 3.2 输入缓冲:把意图与物理 tick 解耦

路径 `res://player/input_buffer.gd`,挂在 `Player` 节点下作为子节点 `InputBuffer`。

```gdscript
class_name InputBuffer
extends Node
## 把离散按键事件转成"在窗口期内消费一次"的意图。
## 跳跃、攻击、冲刺都使用同一套机制,只是缓冲长度不同。

@export var jump_buffer_sec: float = 0.12
@export var attack_buffer_sec: float = 0.10
@export var dash_buffer_sec: float = 0.08

var _jump_timer: float = 0.0
var _attack_timer: float = 0.0
var _dash_timer: float = 0.0

func _unhandled_input(event: InputEvent) -> void:
    if event.is_action_pressed("jump"):
        _jump_timer = jump_buffer_sec
    if event.is_action_pressed("attack"):
        _attack_timer = attack_buffer_sec
    if event.is_action_pressed("dash"):
        _dash_timer = dash_buffer_sec

func _physics_process(delta: float) -> void:
    _jump_timer = max(_jump_timer - delta, 0.0)
    _attack_timer = max(_attack_timer - delta, 0.0)
    _dash_timer = max(_dash_timer - delta, 0.0)

func consume_jump() -> bool:
    if _jump_timer > 0.0:
        _jump_timer = 0.0
        return true
    return false

func consume_attack() -> bool:
    if _attack_timer > 0.0:
        _attack_timer = 0.0
        return true
    return false

func consume_dash() -> bool:
    if _dash_timer > 0.0:
        _dash_timer = 0.0
        return true
    return false

func has_pending_jump() -> bool:
    return _jump_timer > 0.0
```

设计要点三条:

- **窗口期用秒,不用帧**:玩家感受是时间,跨平台帧率波动不会让 100 ms 的窗口忽长忽短。
- **`consume_*` 一次清零**:意图被采用后立刻失效,避免同一次按下触发两次跳。
- **`_unhandled_input` 而非 `_input`**:暂停菜单弹出时这里收不到事件,玩家不会"暂停期间按了一下跳跃,关闭菜单立刻飞起来"。

### 3.3 角色控制:加减速 + 土狼时间 + hit stop

路径 `res://player/player.gd`,挂在 `Player` (`CharacterBody2D`) 上,与 `InputBuffer` 同级。

```gdscript
class_name Player
extends CharacterBody2D

@export_group("Move")
@export var max_speed: float = 180.0
@export var accel_ground: float = 1600.0
@export var accel_air: float = 900.0
@export var friction_ground: float = 1800.0
@export var friction_air: float = 200.0

@export_group("Jump")
@export var jump_velocity: float = -360.0
@export var jump_cut_factor: float = 0.45
@export var coyote_time_sec: float = 0.10
@export var gravity_up: float = 900.0
@export var gravity_down: float = 1300.0
@export var max_fall_speed: float = 520.0

@export_group("Feel")
@export var hit_stop_sec: float = 0.08

@onready var _buffer: InputBuffer = $InputBuffer

var _coyote_timer: float = 0.0
var _freeze_timer: float = 0.0
var _was_on_floor: bool = false

func _physics_process(delta: float) -> void:
    if _freeze_timer > 0.0:
        _freeze_timer -= delta
        move_and_slide()
        return

    var on_floor := is_on_floor()
    _coyote_timer = coyote_time_sec if on_floor else max(_coyote_timer - delta, 0.0)

    var dir := Input.get_axis("move_left", "move_right")
    var target_speed := dir * max_speed
    var accel := accel_ground if on_floor else accel_air
    var fric := friction_ground if on_floor else friction_air

    if absf(target_speed) > 0.01:
        velocity.x = move_toward(velocity.x, target_speed, accel * delta)
    else:
        velocity.x = move_toward(velocity.x, 0.0, fric * delta)

    var g := gravity_up if velocity.y < 0.0 else gravity_down
    velocity.y = minf(velocity.y + g * delta, max_fall_speed)

    if _buffer.consume_jump() and _coyote_timer > 0.0:
        velocity.y = jump_velocity
        _coyote_timer = 0.0

    if Input.is_action_just_released("jump") and velocity.y < 0.0:
        velocity.y *= jump_cut_factor

    move_and_slide()
    _was_on_floor = on_floor

func apply_hit_stop(seconds: float = -1.0) -> void:
    _freeze_timer = hit_stop_sec if seconds < 0.0 else seconds
    velocity = Vector2.ZERO
```

关键设计逐条说明:

- **`move_toward(velocity.x, target, rate * delta)`**:Godot 4.x 的内置方法,自动处理"已经超过目标值"的退化情形,比手写 `lerp` 更稳。
- **地面 / 空中加速度分两套参数**:地面响应快(`1600 px/s²`),空中给一点延迟(`900 px/s²`),空中转身有微妙的滑行感。
- **`gravity_up` < `gravity_down`**:抛物线非对称——上升慢、下落快,跳跃感更"重",这是 Celeste、Hollow Knight 之类游戏的共同手法。
- **`jump_cut_factor`**:松开跳跃键时若仍在上升,把垂直速度乘以 0.45,实现"短按小跳,长按大跳"。
- **`_coyote_timer = coyote_time_sec if on_floor else max(_coyote_timer - delta, 0.0)`**:每个在地面的物理帧把土狼计时器拉满,一旦离开地面就开始倒数,在 100 ms 窗口内仍可触发跳跃。
- **`apply_hit_stop` 外部触发**:战斗系统(第 09 篇 hitbox / hurtbox)在命中时调用,角色当帧 velocity 清零,后续若干物理帧跳过逻辑。

### 3.4 InputBuffer 与 Player 的协作

注意"消费"动作的位置——`_buffer.consume_jump()` 写在 `_coyote_timer > 0.0` 之前,**用短路 `and` 保证只有真正能跳时才清零意图**。否则角色在禁飞状态下按一下跳跃就把缓冲烧光了,土狼窗口里再按一次反而不响应。

```gdscript
# 正确:不能跳就不消费,缓冲继续计时
if _buffer.has_pending_jump() and _coyote_timer > 0.0:
    velocity.y = jump_velocity
    _coyote_timer = 0.0
    _buffer.consume_jump()

# 错误:即使不能跳,意图也被烧掉
if _buffer.consume_jump() and _coyote_timer > 0.0:
    velocity.y = jump_velocity
```

上面 §3.3 的写法是错误版本的"简洁但有 bug"形态。生产版本应该把 `has_pending_jump` 与 `consume_jump` 分开,或者把 `consume_jump` 改成"窥探"。这是输入缓冲实现里最常见的细节坑,值得专门拎出来。

### 3.5 冲刺与无敌帧:复用同一套缓冲

InputBuffer 设计成"通用窗口"而不是"仅限跳跃",在引入冲刺(dash)、攻击、防御时可以零修改复用。下面给一个 dash 实现,挂到 `Player` 上作为 §3.3 代码的补充。它复用了 `_buffer.consume_dash()`,并在执行期间引入"无敌帧"(i-frames)与"输入锁定",验证缓冲系统的扩展性。

路径仍是 `res://player/player.gd`,在 `Player` 类里新增字段与方法:

```gdscript
@export_group("Dash")
@export var dash_distance: float = 96.0
@export var dash_duration_sec: float = 0.18
@export var dash_cooldown_sec: float = 0.45
@export var dash_iframes_sec: float = 0.20

var _dash_timer: float = 0.0
var _dash_cooldown: float = 0.0
var _iframe_timer: float = 0.0
var _dash_direction: float = 1.0

func _try_dash() -> bool:
    if _dash_cooldown > 0.0:
        return false
    if not _buffer.consume_dash():
        return false
    var input_x := Input.get_axis("move_left", "move_right")
    _dash_direction = signf(input_x) if absf(input_x) > 0.1 else signf(velocity.x)
    if _dash_direction == 0.0:
        _dash_direction = 1.0
    _dash_timer = dash_duration_sec
    _dash_cooldown = dash_cooldown_sec
    _iframe_timer = dash_iframes_sec
    return true

func is_invulnerable() -> bool:
    return _iframe_timer > 0.0
```

然后在 `_physics_process` 开头(冻结判定之后,正常移动之前)插入冲刺分支:

```gdscript
_dash_cooldown = max(_dash_cooldown - delta, 0.0)
_iframe_timer = max(_iframe_timer - delta, 0.0)

if _dash_timer > 0.0:
    _dash_timer -= delta
    velocity = Vector2(_dash_direction * dash_distance / dash_duration_sec, 0.0)
    move_and_slide()
    return

_try_dash()
```

冲刺的工程要点:

- **位移除以时间得到 velocity**:玩家在 inspector 里调"我要冲多远",代码自己换算成速度,比直接给 `dash_speed = 1200` 更直观。
- **冲刺期间 `velocity.y = 0`**:这是"无视重力的瞬步",符合大多数 2D 动作游戏的设计;若要保留下落则去掉这一行。
- **i-frames 比 dash 持续时间略长**:玩家会在冲刺尾段松手,无敌窗口稍长一点能容错"擦边躲"。
- **`is_invulnerable()` 暴露给战斗系统**:第 09 篇的 hurtbox 会在受击前查询这个状态,无敌时直接 return,不扣血。

这段代码补全后,角色已经有了"跑、跳、二段跳前置、冲刺、受击停顿"的核心反馈,接下来只缺动画(第 08 篇)和命中判定(第 09 篇)。

## 4. 调参和验收

### 4.1 参数参考区间

下面这张表是 2D 平台 / 动作游戏的常见数值范围,基于 60 物理 tick,角色身高 24-32 px:

| 参数 | 单位 | 紧手感 | 平衡 | 漂浮感 |
| --- | --- | --- | --- | --- |
| `max_speed` | px/s | 220-260 | 160-200 | 110-140 |
| `accel_ground` | px/s² | 2400+ | 1400-1800 | 800 |
| `accel_air` | px/s² | 1400 | 800-1000 | 400 |
| `friction_ground` | px/s² | 2400+ | 1600-2000 | 800 |
| `friction_air` | px/s² | 600 | 150-300 | 0 |
| `jump_velocity` | px/s | -420 | -340 ~ -380 | -260 |
| `gravity_up` | px/s² | 1200 | 800-1000 | 600 |
| `gravity_down` | px/s² | 1800 | 1200-1400 | 700 |
| `jump_cut_factor` | 比例 | 0.4 | 0.45-0.55 | 0.7 |
| `coyote_time_sec` | s | 0.06-0.08 | 0.10-0.12 | 0.15 |
| `jump_buffer_sec` | s | 0.08 | 0.10-0.14 | 0.18 |
| `hit_stop_sec` | s | 0.04-0.06 | 0.08-0.10 | 0.12-0.15 |

几条经验:

- **`coyote + jump_buffer` 总和保持在 200-250 ms 内**:再长玩家就能感知到"延迟跳",像是引擎在帮忙打 combo,反而出戏。
- **`accel` 与 `friction` 通常对称或后者略大**:让停止比启动快一点点,角色不会到处滑。除非你做的是冰面关卡。
- **`hit_stop_sec` 与音效采样要对齐**:命中音如果是 80 ms 攻击点,hit stop 就放在 80 ms 上下,听觉与视觉锁在同一帧。

### 4.2 三种风格的参数包

把上面表格里"紧手感 / 平衡 / 漂浮感"三列具象化为三种典型 2D 游戏:

| 风格 | 类似游戏 | 关键参数取值 | 玩家直觉 |
| --- | --- | --- | --- |
| 紧手感 | Celeste、Super Meat Boy | `max_speed=240`,`accel_ground=2400`,`friction_air=600`,`coyote=0.08`,`jump_buffer=0.10` | "我按了就到位",难度容错全靠玩家自己 |
| 平衡 | Hollow Knight、Ori 系列 | `max_speed=180`,`accel_ground=1600`,`friction_air=250`,`coyote=0.12`,`jump_buffer=0.13` | 略带惯性,空中可微调,跳跃手感重 |
| 漂浮感 | Rayman Legends、Donkey Kong Country | `max_speed=130`,`accel_ground=900`,`friction_air=80`,`coyote=0.15`,`jump_buffer=0.18` | 像气球一样,空中长时间漂浮 |

这三组数值不要混用。如果你在"紧手感"参数包里加大 `friction_air` 想让空中更可控,你会得到一个滞涩、刹车感强但不灵活的混合体——它既不"紧"也不"漂浮"。**先决定风格,再选参数,而不是先抄数字再回头调风格。**

### 4.3 调参时机:不要在第一周就锁定数字

新手最常见的失误是项目开头花一周调"完美的跳跃高度",然后整个项目都被这套数字锁住了。手感参数应该在以下三个节点重新评估:

1. 第一个关卡的初步搭建完成后(本系列第 11 篇之后):有了真实的平台间距、墙壁高度,跳跃高度的"够不够"才能客观判断。
2. 引入战斗系统后(第 09 篇之后):击退、hit stop、i-frames 与移动参数会相互影响,要把它们一起调。
3. 美术资源接入后:角色 sprite 的视觉高度会改变"跳得多高才合理"的感受,**像素尺寸不能与物理尺寸偏离太多**。

### 4.2 验证清单

第 §"手动验证"段会给可勾选项,这里先把验证思路理清楚:

- **跳跃缓冲**:在角色下落途中按下跳跃键,着地瞬间应自动起跳;主观上感觉"按早了也行"。
- **土狼时间**:从平台边缘走下去,在掉落前 0-100 ms 按跳跃,仍能跳起。
- **可变跳跃高度**:短按 / 长按,跳跃高度应有明显差异(目测 1.5-2 倍)。
- **加减速曲线**:松开左右键后,角色不应"啪"地停下;按下后也不应瞬间满速。
- **hit stop**:受击瞬间画面停顿,然后被击退;停顿期间镜头不应卡顿(由第 10 篇验证)。

### 4.3 用打印日志辅助调参

写一段调试输出,打开后能在 Output 面板看到当前真实参数,避免"我以为我改了 export,但其实没刷新"的尴尬:

```gdscript
func _ready() -> void:
    if OS.is_debug_build():
        var t := Timer.new()
        add_child(t)
        t.wait_time = 0.5
        t.timeout.connect(func() -> void:
            print("[Player] vel=%s on_floor=%s coy=%.2f frz=%.2f"
                % [velocity, is_on_floor(), _coyote_timer, _freeze_timer])
        )
        t.start()
```

打印间隔 500 ms,既能跟随调参,又不会把 Output 刷爆。生产构建里 `OS.is_debug_build()` 为 false,自动关闭。

## 5. 踩坑

### 5.1 `_input` 与 `_physics_process` 的时序假设

`_input` / `_unhandled_input` 与 `_physics_process` 之间不存在严格的"事件先到 / 物理后处理"保证。一个物理 tick 内,如果操作系统抖动让两次事件挤在一起,InputBuffer 里的计时器仍只持续一段时间——这正是把缓冲做成"时间窗口"而不是"事件队列"的原因。如果你改成 `Array[InputEvent]` 队列,会发现极少数情况下队列里有多个 pending 跳跃,逻辑很难写正确。

### 5.2 `is_action_just_pressed` 与暂停菜单

`_input` 在游戏暂停后仍然触发,`_unhandled_input` 默认不会(取决于子节点的 `process_mode`)。InputBuffer 选 `_unhandled_input` 还有一个好处:暂停菜单弹出期间,玩家按到的跳跃键不会被记下来,等到关闭菜单时一次性消费。第 13 篇会把 `process_mode` 显式设为 `PAUSABLE`。

### 5.3 物理插值与按键延迟

Godot 4.4+ 的 Physics Interpolation(项目设置 → Physics → 2D → Physics Interpolation 打开)会在物理 tick 之间渲染插值帧。开启后视觉上跳跃会"晚 1 个 tick 起来",玩家不一定注意到,但用相机录像截帧能看到。这不是 bug,是渲染插值的固有特性:它在过去与现在之间画一帧,代价是单帧的"实时性"。如果你做的是节奏关卡(比如踩点跳),建议关闭 2D 物理插值,或把 physics_fps 拉到 120。

### 5.4 摇杆 deadzone 与 `get_vector`

`Input.get_vector("move_left", "move_right", "move_up", "move_down")` 是 4.x 推荐写法,它会做圆形 deadzone 处理。注意:如果你的游戏只有横向移动,**不要**用 `get_vector` 然后丢弃 y 分量——deadzone 会在 (0.2, 0) 这种"纯横向但幅度小"的情况里把 x 也吃掉。横向用 `Input.get_axis("move_left", "move_right")` 更准。

### 5.5 web 导出的键盘焦点

Godot 4.6 的 Web 导出里,HTML5 canvas 必须被点击聚焦才能收到键盘事件。这跟 InputMap 配置无关,需要在 HTML 模板里加自动聚焦 JS 或提示玩家。`@export_placeholder` 提示文字也帮不上忙,这是 Web 平台的固有约束。第 26 篇展开 Web 导出时会再回到这个问题。

### 5.6 `Echo` 事件

按住一个键不放,操作系统会按设定的"按键重复"频率持续发送 `InputEventKey`,Godot 的 `event.is_echo()` 会返回 true。`is_action_pressed` 与 `is_action_just_pressed` 已经过滤了 Echo,但如果你自己写 `_input` 解析 `InputEventKey`,要显式 `if event.is_echo(): return`,否则按住跳跃键会触发多次缓冲填充。

### 5.7 把 `Engine.time_scale` 当作 hit stop

新手常见做法:`Engine.time_scale = 0; await get_tree().create_timer(0.1, true).timeout; Engine.time_scale = 1`。这会把所有 `_process` / `_physics_process` 一并冻结,包括 hitbox 检测下一帧的判定与音频(后者由 audio bus 单独驱动,但视觉冻结仍然给玩家"游戏卡了"的错觉)。**hit stop 永远只冻结战斗参与方,不要冻结整个引擎**。Engine.time_scale 留给真正的"子弹时间"或慢动作回放。

### 5.8 `process_callback` 与帧率波动

`AnimationPlayer`、`AnimationTree`、`Tween` 都有 `process_callback`(或 `process_mode`)属性,默认是 `IDLE` 即跟随 `_process`。如果角色控制在 `_physics_process`,而动画在 `_process`,那么物理 tick 之间的渲染帧会让动画"超前"物理一点点。这在 60 FPS 下基本看不出来,但 30 FPS 锁帧时会看到角色"先做动作再移动"。手感对齐的窗口里,把动画切到 `PHYSICS` 处理会更稳。第 08 篇 §动画与物理对齐有专门讨论。

### 5.9 别用 `Input.action_press` 模拟按键

`Input.action_press(action)` / `Input.action_release(action)` 是给单元测试或重放系统用的接口,**不要在游戏逻辑里调用它来"软按键"**。这两个方法绕过缓冲机制,直接修改 `Input` 单例状态,在多个脚本同时操作时会互相覆盖。如果你的需求是"AI 控制的角色复用玩家代码",更好的做法是在 InputBuffer 之上再加一层"输入提供者"(`PlayerInputProvider` vs `AiInputProvider`),让 Player 脚本只依赖接口而不直接调 `Input`。这一层接口抽象在第 17 篇组件化设计里展开。

### 5.10 `InputEventScreenTouch` 与手机端

手机端的触屏没有"按住"语义,只有"按下"与"抬起"两个边沿事件。Godot 内置一项配置:`项目设置 → Input Devices → Pointing → Emulate Mouse From Touch` 与对应的 `Emulate Touch From Mouse`,可以让触屏事件被识别为鼠标按下,但**对 InputMap 中绑定到键盘的动作无效**。手机端要么自己实现虚拟摇杆 / 跳跃按钮(用 `Control` 节点),要么在 `_input` 里把触屏事件转成 `InputEventAction.action_press(...)`。这个话题第 27 篇会展开,本篇不细化。
