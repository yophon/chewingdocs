# 12-敌人 AI、巡逻、追击与导航网格三件套

> 一句话导读:Godot 2D 导航是一个三层数据流——`NavigationRegion2D` 烘焙静态可行走区域,`NavigationAgent2D` 帮敌人在这块区域里寻路,`NavigationObstacle2D` 让动态物体在不重新烘焙的前提下避让其他代理。

第 11 篇把关卡画好,玩家可以在 TileMap 上跑、跳、挖。接下来这张地图上得有"会动的敌人":能巡逻、能看见玩家、能绕开墙追过来、能挨打、能掉东西。本篇专门讲让敌人"会找路"这件事,以及把找路接到一个简洁可控的状态机里。

## 1. 机制定位

### 寻路到底要解决什么

在 2D 游戏里,"敌人追玩家"有三个不同精度等级:

第一档,**直线追**:`enemy.position = enemy.position.move_toward(player.position, speed * delta)`。便宜、可读,但玩家躲到一堵墙后面,敌人就贴墙挪、卡墙角、绕不出来。适合开阔无遮挡的 boss arena,不适合走廊和迷宫。

第二档,**手写 A***:在 tile grid 上跑 BFS 或 A*,得到路径数组,敌人沿路径节点走。可控、可调权重,但代码量大、与编辑器工作流脱钩——你画完地图,还得手动转换成图。改 tile 时图也要重建。

第三档,**导航网格(navmesh)**:用一个连续的多边形区域描述"哪里能走",寻路在这块多边形上做。Godot 走的就是这条路:画好关卡,在 `NavigationRegion2D` 里烘焙一次,所有敌人共用这块网格。烘焙是一次性成本,运行时寻路是廉价多边形操作。

新手最常见的失控写法,是混着用:直线追打底,卡墙了开 A*,A* 失败再退回直线。三套系统互相打架,边界条件爆炸。Godot 的导航三件套是为了让你不需要写这个 fallback,而是用同一套数据流处理静态地形(navmesh)、动态障碍(obstacle)和代理决策(agent),三层职责完全分离。

### 状态机:敌人不是"一直在追"

哪怕寻路完全正确,敌人的行为还需要状态切换。一个最小骨架是 5 个状态:

- **Idle**:刚 spawn,等待或环顾。
- **Patrol**:沿预设路径或随机点巡逻。
- **Chase**:发现玩家,沿 navmesh 路径接近。
- **Attack**:进入攻击距离,挥刀/射击。
- **Stagger**:被击中或被击退,短暂失控。

寻路只是 Chase 状态的子能力,Idle 和 Patrol 完全可以不用 navmesh(或用更短的局部路径)。本篇把这套状态机和导航代理拼在一起。

### 受击反馈三要素

工程化的"被打"看起来很简单,实际包含三个解耦的子动作,不要写成一个大函数:

- **Hit Stop**:命中瞬间冻结角色和敌人若干毫秒(80~150ms),给玩家"打到了"的反馈。第 07 篇手感参数那一节已经讲过。
- **Knockback**:沿命中方向施加一个 200~400 像素/秒的瞬时速度,以 8/s 量级线性衰减。和移动速度叠加,但跟独立通道,避免被避让算法"挽救"成原地不动。
- **闪烁**:`modulate.a` 或 `tint` 在 0.2 秒内振荡一次,提示血量变化。可以挂 `Tween` 完成,也可以让 `AnimationPlayer` 播一段 "hurt" 动画。

这三件事在掉血处理函数里要分别触发,而不是混进单一 `take_damage`。本篇代码只展示了击退,完整反馈在第 17 篇组件化时拆成独立 component。

## 2. Godot 心智

### 三件套的职责与数据流

```text
NavigationRegion2D  (节点,持有 NavigationPolygon 资源)
        │ 烘焙 → 注册到 NavigationServer2D 的 navigation map
        ▼
NavigationServer2D  (引擎服务,管理整个 World2D 的 navigation map)
        ▲
        │ 查询路径 / 注册避让代理
        │
NavigationAgent2D   (节点,每个敌人挂一个,代理寻路 + 避让)
NavigationObstacle2D (节点,动态障碍物,影响 agent 避让计算)
```

三件套不是相互替代,而是**串行的数据通道**:

- `NavigationRegion2D` 是**静态可行走区域的来源**。一张关卡通常 1~3 个,一个走廊、一个房间、一个屋顶。烘焙后,它的多边形被推到 `NavigationServer2D` 维护的 navigation map 中。
- `NavigationAgent2D` 是**每个会移动的实体的代理**。它从 navigation map 里查路径,返回 `get_next_path_position()`,你按这个目标点驱动敌人。如果开启了避让(`avoidance_enabled = true`),它还会把"目标速度"提交给服务器,服务器经过 ORCA/RVO 算法得到"安全速度",再通过 `velocity_computed` 信号回吐给你。
- `NavigationObstacle2D` 是**动态障碍**。它不重烘 navmesh(那太贵),而是参与避让计算:其他 agent 在算安全速度时,会把这块障碍视作"不能进入的圆或多边形"。典型场景:可推动的箱子、临时召唤物、其他 agent 自身。

理解这三层之后,你就不会犯"我让 navmesh 跟着箱子动"的错——navmesh 是静态的,箱子改用 obstacle。

### `NavigationServer2D`:不可绕开的服务器层

三个节点本质上是 `NavigationServer2D` 的便利封装。生产代码里 80% 的时间你只和节点打交道,但有些场景必须知道服务器层的存在:

- 关卡程序化生成时,你想自己 parse 几何而不依赖编辑器烘焙——用 `parse_source_geometry_data` 和 `bake_from_source_geometry_data_async`。
- 调试时打开 `NavigationServer2D.set_debug_enabled(true)`,运行时叠加显示所有 navmesh、agent、obstacle。
- 多个 world(比如分屏游戏)各自有独立 navigation map,通过 `World2D.navigation_map` 拿到 RID,做精细控制。

工程师视角的关键事实:**Godot 的导航是延迟同步的**。你新建一个 `NavigationRegion2D` 节点,navmesh 不会立刻进入 navigation map,而是在下一次服务器 tick(通常是下一个物理帧)被同步进去。`map_get_iteration_id(map) == 0` 表示"这张 map 还没同步过任何东西",代理在这一帧查询路径会拿到空。这是上面代码里 `_request_velocity` 第一行检查的原因。

### `NavigationPolygon` 的两种生成方式

`NavigationRegion2D` 的核心数据是 `navigation_polygon` 属性,类型 `NavigationPolygon`。它可以:

- **从编辑器内画出来**:打开 `NavigationRegion2D` 节点,用工具栏的多边形画笔圈出可行走区域。适合简单关卡或手工微调。
- **从场景几何烘焙**:`NavigationRegion2D` 支持解析子节点的 `CollisionPolygon2D` 形状,自动减去这些遮挡形状,得到剩余的可行走多边形。适合大量地形的关卡。`TileMapLayer.navigation_enabled = true` 时,引擎会把每个有 navigation polygon 配置的 tile 加入烘焙源,这是 tile 与导航对接的官方通道。

烘焙触发有两种:编辑器里点 "Bake NavigationPolygon" 按钮(适合发布前固化),或运行时调用 `NavigationRegion2D.bake_navigation_polygon(on_thread = true)`(适合关卡程序化生成或动态切换)。在 Godot 4.x,`NavigationServer2D` 提供更底层的 `parse_source_geometry_data` + `bake_from_source_geometry_data_async`,可以把"哪些节点参与烘焙"控制得更细。

### `NavigationAgent2D` 的 `velocity_computed` 协议

这是 4.x 里最容易写错的部分。代理的正确流程是:

1. 你算出"期望速度"`new_velocity`(通常等于 `direction_to(next_path_position) * speed`)。
2. 调 `agent.set_velocity(new_velocity)`,把期望速度提交给避让服务器。**注意此时不要立刻把这个 velocity 当作真实移动速度**。
3. 下一物理 tick(异步,但实际就是同一帧稍后),`velocity_computed(safe_velocity: Vector2)` 信号触发。`safe_velocity` 是经过避让计算后的安全速度,可能和你提交的略有差异(为了绕开其他 agent)。
4. 在 `_on_velocity_computed` 回调里,真正赋值 `velocity = safe_velocity` 并 `move_and_slide()`。

如果 `avoidance_enabled = false`,可以跳过 `set_velocity`,直接用 `new_velocity` 移动。开了避让后,这个回调流程不能省略——直接用 `new_velocity` 而忽视 `velocity_computed` 就等于关掉了避让。

### 路径 vs 目标:两层不同的"完成"

代理内部维护一条折线路径,路径由若干"path point"组成。你要区分两种"到了":

- `is_navigation_finished()`:到达**终点**(最后一个 path point),整次寻路完成。这是巡逻切换下一个点、Chase 切到 Attack 的触发条件。
- `path_desired_distance`:认为已经到达**当前 path point**,内部自动跳到下一段。这个值控制代理走折线时多紧地"贴边",对外不可见。

如果你把 `path_desired_distance` 设得太大,代理会在还没靠近某个拐角时就跳过它,出现"切角穿墙";设得太小,代理在每个拐角附近反复来回,因为每帧位置变化跨过了阈值。像素游戏推荐 4~8。`target_desired_distance` 则控制"到达终点"的判定阈值,通常和攻击半径配合使用。

## 3. 工程实现

### 文件:`actors/enemy/enemy_navigator.gd`

包含 5 状态机 + 导航代理,挂在敌人根节点(`CharacterBody2D`)上,需要的子节点:`NavigationAgent2D`、`Area2D`(检测玩家进入视野)。

```gdscript
class_name EnemyNavigator
extends CharacterBody2D

signal hit_taken(damage: float)
signal died

enum State { IDLE, PATROL, CHASE, ATTACK, STAGGER }

@export var move_speed: float = 80.0
@export var chase_speed: float = 130.0
@export var attack_range: float = 28.0
@export var sight_radius: float = 220.0
@export var patrol_points: Array[Vector2] = []
@export var knockback_decay: float = 8.0
@export var max_hp: float = 30.0

@onready var agent: NavigationAgent2D = $NavigationAgent2D
@onready var sight: Area2D = $Sight

var _state: int = State.IDLE
var _hp: float = max_hp
var _player: Node2D = null
var _patrol_index: int = 0
var _knockback: Vector2 = Vector2.ZERO


func _ready() -> void:
    _hp = max_hp
    agent.path_desired_distance = 4.0
    agent.target_desired_distance = 6.0
    agent.avoidance_enabled = true
    agent.radius = 12.0
    agent.velocity_computed.connect(_on_velocity_computed)
    sight.body_entered.connect(_on_sight_entered)
    sight.body_exited.connect(_on_sight_exited)
    _enter_state(State.PATROL if not patrol_points.is_empty() else State.IDLE)


func _physics_process(delta: float) -> void:
    _knockback = _knockback.lerp(Vector2.ZERO, knockback_decay * delta)
    match _state:
        State.IDLE:
            _tick_idle()
        State.PATROL:
            _tick_patrol(delta)
        State.CHASE:
            _tick_chase(delta)
        State.ATTACK:
            _tick_attack()
        State.STAGGER:
            _tick_stagger()


func take_damage(damage: float, source_pos: Vector2) -> void:
    _hp -= damage
    hit_taken.emit(damage)
    if _hp <= 0.0:
        died.emit()
        queue_free()
        return
    var dir: Vector2 = (global_position - source_pos).normalized()
    _knockback = dir * 220.0
    _enter_state(State.STAGGER)


func _enter_state(next: int) -> void:
    _state = next
    match next:
        State.PATROL:
            _set_next_patrol_target()
        State.CHASE:
            if _player != null:
                agent.target_position = _player.global_position
        State.STAGGER:
            await get_tree().create_timer(0.25).timeout
            _enter_state(State.CHASE if _player != null else State.PATROL)


func _request_velocity(target_speed: float) -> void:
    if NavigationServer2D.map_get_iteration_id(agent.get_navigation_map()) == 0:
        return
    if agent.is_navigation_finished():
        agent.set_velocity(_knockback)
        return
    var next_pos: Vector2 = agent.get_next_path_position()
    var dir: Vector2 = global_position.direction_to(next_pos)
    agent.set_velocity(dir * target_speed + _knockback)


func _on_velocity_computed(safe_velocity: Vector2) -> void:
    velocity = safe_velocity
    move_and_slide()


func _tick_idle() -> void:
    velocity = _knockback
    move_and_slide()


func _tick_patrol(delta: float) -> void:
    _request_velocity(move_speed)
    if agent.is_navigation_finished():
        _patrol_index = (_patrol_index + 1) % patrol_points.size()
        _set_next_patrol_target()


func _tick_chase(_delta: float) -> void:
    if _player == null:
        _enter_state(State.PATROL if not patrol_points.is_empty() else State.IDLE)
        return
    agent.target_position = _player.global_position
    var dist: float = global_position.distance_to(_player.global_position)
    if dist <= attack_range:
        _enter_state(State.ATTACK)
        return
    _request_velocity(chase_speed)


func _tick_attack() -> void:
    velocity = _knockback
    move_and_slide()
    # 攻击动画由 AnimationPlayer 触发,通过信号或 await 决定何时回 CHASE
    # 这里给一个最简短的占位逻辑
    if _player == null \
    or global_position.distance_to(_player.global_position) > attack_range * 1.5:
        _enter_state(State.CHASE if _player != null else State.PATROL)


func _tick_stagger() -> void:
    velocity = _knockback
    move_and_slide()


func _set_next_patrol_target() -> void:
    if patrol_points.is_empty():
        return
    agent.target_position = patrol_points[_patrol_index]


func _on_sight_entered(body: Node2D) -> void:
    if body.is_in_group("player"):
        _player = body
        _enter_state(State.CHASE)


func _on_sight_exited(body: Node2D) -> void:
    if body == _player:
        _player = null
```

工程取舍说明:

- **`velocity_computed` 只在 `_request_velocity` 调用后触发**:Idle/Attack/Stagger 不寻路,所以不调 `set_velocity`,也就不会被避让覆写。这样状态间切换不会被回调污染。
- **击退用 `_knockback` 独立通道**:不通过 navigation,而是直接叠加到提交的速度上;停止寻路后(Idle/Stagger)仍能滑动。这样击退手感不会被避让"挽救",看上去就是干脆的飞出去。
- **`patrol_points` 用 `Array[Vector2]` export**:类型化数组让 Inspector 渲染一个干净的列表,直接在编辑器里点几个点。
- **`map_get_iteration_id == 0` 检查**:navigation map 在第一次同步前是空的(场景 `_ready` 那一帧),直接查询会得到无效路径。这一行避免敌人 spawn 那一帧就乱冲。

### Obstacle 的两种用途要分清

`NavigationObstacle2D` 看起来是一个节点,但**两组属性走两套机制,你大概率只想用其中一组**:

- `affect_navigation_mesh = true` + `carve_navigation_mesh`:把这个形状作为"洞"嵌入 navmesh **下一次烘焙**。这是"静态障碍预先扣除"的用法,适合关卡里固定不动的雕像、桌椅。注意它只在烘焙时生效,运行时挪动 obstacle 不会自动重烘。
- `avoidance_enabled = true` + `radius`/`vertices`:这个 obstacle 在运行时参与 ORCA 避让,让其它代理"知道这里有东西要绕"。不影响 navmesh 本身,适合动态对象(箱子、临时召唤物)。

把两个开关都开,意思是"既扣 navmesh,又参与运行时避让",大多数情况是浪费——你只需要一种。可推箱子用 avoidance,固定不动的石碑应当直接在 `NavigationRegion2D` 的源几何里画进去,而不是再挂 obstacle。

`vertices` 的绕向也有语义:**顺时针表示"代理被推向多边形内部"**(障碍是空洞,代理在洞外),**逆时针表示"代理被推出多边形外部"**(障碍是实心,代理在洞内不应进入)。绕错方向会让所有代理向障碍内部猛冲,看起来像 bug。

## 4. 调参和验收

### 寻路核心参数

- `agent.radius`:代理自身的避让半径,通常等于角色物理体的视觉半径(精灵宽度的一半左右)。设小了代理之间贴脸过墙,设大了代理在窄门口卡死。
- `agent.neighbor_distance`:与其它代理交互的最大距离,默认 500。越大避让越聪明,但 CPU 越贵。
- `agent.max_neighbors`:同时考虑的相邻代理上限。10~20 通常够。
- `agent.path_desired_distance`:认为"已经到达路径节点"的距离,默认 20。像素游戏小到 4~6 才贴边。
- `agent.target_desired_distance`:认为"已经到达终点"的距离,通常和攻击距离配对。
- `agent.time_horizon_agents` 与 `agent.time_horizon_obstacles`:避让算法的"预测窗口",值越大代理越早开始转向。默认 1 秒,1.5~2 秒会让大群体走得更优雅,但密集场景容易"过度避让"原地打转。

### 烘焙参数(`NavigationPolygon` 的几何属性)

- `agent_radius`:烘焙时把可行走区域向内收缩这么多,模拟代理体积。默认 10。设小了敌人擦墙,设大了走廊变更窄甚至寻不到路。
- `cell_size`:网格化精度,默认 1。降到 0.5 边缘更平滑,但烘焙更慢。
- `parsed_geometry_type`:从 `CollisionPolygon2D` / `MeshInstance2D` / `Both` 中解析源几何。tile-based 关卡选默认即可。

### 行为参数

- `sight_radius`:敌人感知玩家的范围。横版动作 200~300 像素是主流,迷宫探索可以小到 120,空旷场景甚至到 500。
- `attack_range` vs `target_desired_distance`:攻击距离应当略大于 target_desired_distance,否则代理停在终点边缘,攻击判定不上身。
- `chase_speed > move_speed` 的比值:`1.5~2.0` 给玩家"被追"的紧迫感而又不至于完全跑不掉。
- `knockback_decay`:击退速度的衰减率,数值越大恢复越快。8 对应 1/8 秒走完大部分能量,适合大多数动作游戏。

### 完成度验收

- [ ] 关卡烘焙后,可视化(`NavigationServer2D.set_debug_enabled(true)`)看到的可行走区域和你画的地形一致。
- [ ] 敌人在 Patrol 状态沿 `patrol_points` 顺序循环,不卡墙。
- [ ] 玩家进入 sight 区域,敌人切到 Chase,沿 navmesh 绕过障碍接近。
- [ ] 玩家躲到 sight 之外,敌人在原地或继续追到最后已知位置后切回 Patrol。
- [ ] 推动 crate 到走廊中央,敌人避让,而不是穿过去或卡住。
- [ ] 敌人受击瞬间触发 Stagger,飞出后回到 Chase。

### 性能与代理数量

`NavigationAgent2D` 的避让计算是 O(N²) 量级(每个 agent 看周围所有 agent)。50 个 agent 一起在屏幕上同时 ORCA,在中端硬件上仍能维持 60 FPS;100 个就要小心。常见优化:

- 远离玩家的敌人 `avoidance_enabled = false`,只在玩家可见或附近的敌人开启避让。
- 用 `max_neighbors` 限制单 agent 同时考虑的相邻代理数,默认 10 已经够,做大群体战斗也很少超过 20。
- 把所有"装饰用走动"(背景村民)的 agent 全关避让,只做寻路。

寻路本身(求路径)的复杂度则取决于 navmesh 多边形数量,几百个 polygon 的关卡查路径只需几十微秒,基本不需要为它优化。

## 5. 踩坑

**坑 1:忘了把 `agent.set_velocity` 的结果交给 `velocity_computed` 信号处理。** 这是 4.x 最高频错误。你写了 `agent.set_velocity(new_velocity)` 然后立刻 `velocity = new_velocity; move_and_slide()`,看起来代理在动,但 ORCA 避让完全失效——你提交了期望速度,却把"期望"当成"安全"。多个敌人凑到一起时直接挤穿彼此。修法:`set_velocity` 之后,真正的移动写在 `_on_velocity_computed(safe_velocity)` 里。

**坑 2:在 `_ready` 当帧立刻调用 `agent.get_next_path_position()`。** navigation map 还没完成第一次同步,返回零向量或代理自己的位置。表现是 spawn 那一帧敌人"突然往左下角扔"。解法:用 `NavigationServer2D.map_get_iteration_id(agent.get_navigation_map())` 判断是否同步过,未同步就跳过这一帧。或者用 `await get_tree().physics_frame` 等一帧再开始决策。

**坑 3:`NavigationAgent2D.avoidance_enabled = true` 但 `agent.radius = 0`。** 默认 `radius = 10`,但很多人手抖改成 0,避让算法认为"我没体积",照穿不误。设到角色精灵宽度的一半左右。

**坑 4:`NavigationObstacle2D.affect_navigation_mesh = true` 然后期望"运行时自动更新 navmesh"。** 这个属性只在**下一次烘焙**时生效,不会自动重烘。运行时挪动 obstacle 不会改变可行走区域,只会改变其他 agent 的避让。要动态改地形,必须显式调用 `NavigationRegion2D.bake_navigation_polygon()`,但这是几十毫秒的操作,不要每帧调。

**坑 5:`NavigationPolygon.agent_radius` 设成 0,期望"navmesh 紧贴地形"。** 这会让代理擦墙时弹回,因为代理本身有 radius,边缘没收缩,代理一靠边就被避让推开。一般 agent_radius 略大于 agent.radius,留一点安全边距。

**坑 6:把巡逻点写在 `_ready` 里的 `[Vector2(100, 0)]` 而不是关卡坐标。** `target_position` 是**全局坐标**,如果敌人初始位置不是 (0,0),这个点的实际意义和你预想的不一样。要么用 `to_global(local_point)` 转换,要么直接在 Inspector 里以编辑器内的全局坐标填写。

**坑 7:玩家死亡或离开场景后,敌人脚本里 `_player` 不再指向有效节点。** Godot 4 的 `Node` 释放后,GDScript 持有的引用变成 `null` 安全(`is_instance_valid(_player)` 可判),但你直接读 `_player.global_position` 仍会崩。任何状态切换前后用 `is_instance_valid(_player)` 兜底。

**坑 8:Stagger 用 `await create_timer().timeout` 之后,玩家已经死了,敌人却切回 Chase 撞向 null。** 上一坑的延续——`_enter_state(State.CHASE if _player != null else State.PATROL)` 在 `_player` 已被 `queue_free` 之后判 `!= null` 还会通过(引用仍有,但对象在下一帧失效)。改成 `is_instance_valid(_player)`。

**坑 9:多个敌人共享同一个 `NavigationAgent2D` 实例。** 看上去能省内存,实际上 agent 必须每个移动单元一份,内部维护"当前路径"。一个 agent 同时被两个敌人调,路径互相覆盖,敌人乱抖。

**坑 10:`NavigationRegion2D` 节点的 `enabled` 在场景实例化后被某段代码改成 `false` 然后忘了改回来。** 整张 navmesh 失效,代理永远 `is_navigation_finished() == true`,敌人不再寻路只是在原地抖动。在开发期开 `NavigationServer2D.set_debug_enabled(true)` 看可视化,navmesh 不显示就说明被关了。

**坑 11:试图在 `_on_velocity_computed` 里修改状态机或调用 `set_velocity`。** 这是回调,Godot 内部还在迭代 agent 列表。改状态可以(本质是修改局部变量),但**不要再次 `set_velocity` 触发递归**。要新一帧再次寻路,等下一个 `_physics_process`。

**坑 12:`patrol_points` 类型用 `Array` 而不是 `Array[Vector2]`。** 非类型化数组在 Inspector 里只能显示通用 Variant 编辑器,体验差且无类型校验;还失去 GDScript 字节码优化。统一用 `Array[Vector2]`。

**坑 13:在 `NavigationObstacle2D.velocity` 上写 0 后期望"obstacle 不动了"。** `velocity` 字段是给避让算法用的提示,告诉它"这个障碍预期会以这个速度移动"。obstacle 的实际位置由父节点 `position` 决定,你想让 obstacle 移动应当移动父节点;`velocity` 是元数据,不是驱动量。让障碍跟随父节点移动时,记得同步更新 `velocity = (current_pos - last_pos) / delta`,否则避让算法以为它静止,会算错避让方向。

**坑 14:寻路目标设到地图外。** 角色追玩家,玩家瞬移到 navmesh 之外,`target_position` 不在任何多边形内,代理会停在距离目标最近的边缘点。表现是"敌人冲到墙边就停了"。在切到 Chase 状态前用 `NavigationServer2D.map_get_closest_point()` 把目标 snap 到 navmesh 内部。

**坑 15:用 `RayCast2D` 而不是 `Area2D` 做视野检测,然后期待"墙后看不见"。** `Area2D.body_entered` 只判断圆形区域,不判断遮挡。在走廊关卡里,玩家明明在墙后,敌人依然能"听到"。要做"看得见才追",再加一段 `RayCast2D` 或 `PhysicsRayQueryParameters2D`,从敌人指向玩家,只有 ray 命中玩家本人(没被墙拦)才切 Chase。

**坑 16:把 `NavigationAgent2D` 挂在 `Node2D` 而不是 `CharacterBody2D` 上。** 代理本身可以挂在任何 2D 节点上,但你的移动 API(`move_and_slide`)只在 `CharacterBody2D` 上有。挂错位置的话,`_on_velocity_computed` 里赋值 `velocity = safe_velocity` 没意义——`Node2D` 不响应 `velocity`,你得改用 `position += safe_velocity * delta`,但会失去物理碰撞响应。

**坑 17:`NavigationAgent2D.path_max_distance` 设得过小,代理总是 "is_navigation_finished()"。** 这个属性限制单次寻路允许返回的最大路径距离,默认很大。手动调到比关卡尺寸还小时,长距离寻路直接失败,代理什么也不做。一般不动它,只在做"短视野敌人"(只能寻路 200 像素以内)时才用。
