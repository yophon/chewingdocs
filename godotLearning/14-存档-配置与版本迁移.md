# 14-存档、配置与版本迁移

> 一句话导读:存档不是把内存写到磁盘那么简单。它要应对崩溃中断、跨平台路径、跨版本字段迁移、玩家手动改文件,以及"已经卖出几千份的旧存档要不要兼容"这种长期负担。

工程师做后端时,数据库迁移已经是不舒服的话题:Alembic / Flyway / Liquibase 把 schema 变更管起来,迁移失败可以回滚,旧版本读不到新字段时有默认值。游戏存档没有数据库,没有 DBA,玩家本地的存档文件就是真相。你今天发的 v1.0,把 `gold: int` 写进存档,半年后想改成 `currencies: Dictionary` 支持多币种,玩家硬盘上几千份旧存档要么能读、要么直接进熔炉。

`securityLearning` 已经把密码学讲透了,本篇不会去推导 AES 模式或 HMAC 哈希链;只讲存档场景下最实用的两件事:**版本演进时怎么保证旧存档还能读、玩家手改存档时崩不崩**。Godot 给了两条路:轻量灵活的 JSON,以及和场景耦合更紧的 `Resource`。两条路的取舍直接决定后期维护的痛苦程度。

## 1. 机制定位

存档系统真正要解决的不是"把数据落盘",而是四个长期问题:

1. **持久化语义**:玩家退出游戏,下次启动能恢复到"差不多"的状态。"差不多"指什么是个产品决策——是上一次手动 Save、上一个检查点、还是上一秒的精确帧?这决定了存档触发时机与数据范围。
2. **崩溃中断容错**:玩家家里突然断电,或者 Windows 把游戏进程 kill 掉,存档文件不能写一半。要么完整旧版,要么完整新版,不能是损坏的中间态。
3. **跨版本迁移**:游戏发出去之后,代码继续演进。`Player` 多了一个 `stamina`,`Inventory` 把数组改成字典,旧存档读进来时新字段没有,引擎不能直接崩。
4. **手改容忍**:玩家会改存档加金币、改装备、改等级。要不要防?能不能防?如果只是改个 JSON 文件玩家立刻能破,你的"反作弊"是不是给真玩家添麻烦?

第 1 个问题是 API 选择(JSON / `Resource`),第 2 个问题是写入策略(临时文件 + 原子重命名 + `.bak` 备份),第 3 个问题是 `schema_version` + 迁移函数链,第 4 个问题是态度选择(可逆混淆 vs 不防)。

还有一个被低估的问题是 **快照的一致性**:存档要捕获"哪一时刻"的游戏状态?玩家把一颗药水拖到背包里、还没松鼠标、按了存档键——这一瞬间的"状态"是已经在背包里、还在拖、还是 nothing?在游戏循环的中间存档,任何一个跨节点的事务(伤害结算、物品转移、技能冷却递减)都可能被切到一半。最稳的做法是把存档触发延迟到下一个稳定点:`call_deferred("_do_save")`,等当前帧所有 `_process` 跑完再写;或者只在"明确的安全态"(关卡切换前、暂停期间、对话结束)允许触发。这条线不光是技术,也是游戏设计——玩家在打 Boss 半空中存档,你救不了他下一秒摔死。

本篇做出一个能给原型用的 `SaveSystem` autoload:写入用原子替换,读取自动迁移老版本,主存档损坏时回退到 `.bak`,可选轻量加密阻止玩家随手改 JSON。后续第 16 篇做事件总线、第 30 篇做发售前检查表时,这个 `SaveSystem` 会作为基础设施被复用。本篇不会展开"成就系统"或"云存档同步策略",前者是数据驱动篇的话题,后者是发布篇的范畴。本篇也不重复推导密码学算法,加密只取边界,密码学本身请翻 `securityLearning`。

## 2. Godot 心智

### user:// 与平台差异

`res://` 是项目资源根,只在开发期能写,导出后只读。所有运行期产物——存档、截图、日志、玩家配置——都必须放 `user://`。`user://` 的物理路径由 `OS.get_user_data_dir()` 解析:

- **Windows**:`%APPDATA%\Godot\app_userdata\<project_name>\`
- **macOS**:`~/Library/Application Support/Godot/app_userdata/<project_name>/`
- **Linux**:`~/.local/share/godot/app_userdata/<project_name>/`
- **Android**:`/data/user/0/<package>/files/`(沙盒内,卸载游戏即清空)
- **Web (HTML5)**:虚拟文件系统挂载在 IndexedDB 上,关页面前要主动 flush,否则数据可能丢

可以在 `project.godot` 里设 `application/config/use_custom_user_dir = true` 改成自定义目录名,避免和别人的项目重名。导出到 Steam 的项目通常关闭自定义目录,让 Steam Cloud 配置自己接管。

### JSON vs Resource:两条路线的取舍

Godot 提供两套通用的本地序列化路径。**JSON** 用 `JSON.stringify` 和 `JSON.parse_string`(4.x 把 3.x 的 `to_json` / `parse_json` 改成了类方法),好处是格式中性、人类可读、跨语言、不耦合任何 GDScript 类;坏处是丢类型——`Vector2` 变成 `{"x": 0, "y": 0}`、`int` 在某些路径上变 `float`、`Color` 变四个浮点。

**Resource** 用 `ResourceSaver.save` 写 `.tres`(文本)或 `.res`(二进制),`ResourceLoader.load` 读回。好处是引擎原生类型直接保留、可以挂脚本、和编辑器工作流一致;坏处是格式耦合 GDScript 类,改类名 / 移动文件 / 删字段会让旧存档解析失败,而且 `.tres` 文件里 `script = ExtResource(...)` 路径写死,玩家拷贝到另一台机器项目结构稍变就读不出。

更隐蔽的坑是 **`ResourceLoader.load` 一个带脚本的 `.tres` 时会触发该脚本的初始化逻辑**——这意味着玩家用记事本改一行 `script` 路径,加载存档时引擎会去执行那个脚本。这条路在玩家可写目录 `user://` 上有明确的攻击面,不要把"任意 Resource 都信"作为默认假设。第 30 篇的"发售前检查"会再回来重新审一遍这条路径。

实操结论是:**玩家数据(进度、金币、库存)优先 JSON**,易迁移、易调试、易跨平台;**只读的内容资源(关卡定义、技能表、本地化文本)用 `Resource`**,跟着项目走、跟着编辑器走、跟着 git diff 走。第 18 篇会用 `Resource` 做数据驱动,那时候这条边界会更清晰。

### FileAccess 的最小契约

```text
var f := FileAccess.open(path, FileAccess.WRITE)
if f == null:
    push_error(FileAccess.get_open_error())
    return
f.store_string(json_text)
# RefCounted,引用一掉作用域就 close;但 Windows 上要及时释放避免锁
```

`FileAccess` 是 `RefCounted`,作用域结束自动关闭。手动调 `f = null` 或者 `f.close()` 在 Windows 上更稳——这个平台对文件句柄敏感,占着写句柄时 `DirAccess.rename` 会失败,后面要原子替换时会踩这一脚。

`DirAccess` 是配套的目录操作:`file_exists` / `rename` / `copy` / `remove`,所有路径都接受 `res://` 与 `user://`。

### 加密的边界

`FileAccess.open_encrypted_with_pass(path, mode, password)` 用 AES-256-CBC 加密,密码由 SHA-256 派生。它能阻止玩家用记事本随手改,但只要密码硬编码在游戏二进制里,任何愿意花十分钟拿 IDA 反编译的人都能拿到。所以这是"可逆混淆"而非"真加密":提高门槛,不提供安全性。

对单机原型来说,这种门槛已经够。竞速、Roguelike、独立 RPG 都不太需要更复杂的方案。要做反作弊,真正有效的路径是服务端权威(在线榜)而不是客户端加密,这条线属于第 29 篇联机篇而非本篇。

加密的边界还要注意一点:加密文件不可读 = 不可调试。开发期玩家(也就是你自己)debug 一个 "存档读不出来" 的问题,加密后无法用文本编辑器看里面到底写了什么。生产里一个好习惯是给 `_password` 做开关:`OS.has_feature("editor")` 或者读环境变量,只在导出版加密,编辑器版本写明文。这样开发期可读、发布期混淆。

### 配置和存档的分离

很多新手把"存档"和"配置"混作一谈。它们解决的问题完全不同:**存档**绑定到玩家进度,有多个槽位,有版本迁移问题,通常带校验;**配置**是单一全局状态(画面分辨率、音量、按键映射、语言),只一份,不需要校验,玩家随便改也无所谓——他改坏了下次重置默认就好。

混在一起的代价:玩家想清存档保留设置,要么做不到、要么做不干净;玩家换电脑想同步设置不想同步存档,你这套同步系统得拆开重做。从一开始就把存档放 `user://saves/`,把配置放 `user://config.cfg` 用 `ConfigFile` 读写,是最低成本的隔离。`ConfigFile` 自带 ini 风格,`get_value(section, key, default)` 自带兜底,不需要版本迁移。

### 原子写入与备份恢复

正确的写入顺序是:

1. 把新内容写到 `user://save_1.json.tmp`。
2. flush + close 临时文件。
3. 如果存在主文件 `user://save_1.json`,先 `DirAccess.rename` 成 `user://save_1.json.bak`。
4. `DirAccess.rename` 把 `.tmp` 改名成主文件。

这样无论在第几步崩,磁盘上要么有完整的旧版主文件,要么有完整的旧版 `.bak`,要么有完整的新版主文件,不会出现"半文件"。加载时,先尝试主文件,失败回退 `.bak`,再失败提示玩家"档损坏,从最近检查点重开"。

## 3. 工程实现

下面是一个最小可用的 `SaveSystem`,作为 autoload 注册在 `project.godot` 的 `[autoload]` 段(autoload 边界第 16 篇会专门讲,这里只是用)。

文件位置:`res://systems/save_system.gd`。

```gdscript
extends Node
## Autoload: SaveSystem

const SAVE_DIR: String = "user://saves/"
const SAVE_NAME: String = "slot_%d.json"
const SCHEMA_VERSION: int = 3

signal save_failed(reason: String)
signal load_failed(reason: String)

var _password: String = "" # 可选的混淆密码,空字符串时按明文写

func _ready() -> void:
    DirAccess.make_dir_recursive_absolute(ProjectSettings.globalize_path(SAVE_DIR))

func save_slot(slot: int, payload: Dictionary) -> bool:
    var path: String = SAVE_DIR + SAVE_NAME % slot
    var tmp: String = path + ".tmp"
    var bak: String = path + ".bak"
    payload[&"schema_version"] = SCHEMA_VERSION
    payload[&"saved_at"] = Time.get_datetime_string_from_system(true)
    var json_text: String = JSON.stringify(payload, "\t")
    if not _write_file(tmp, json_text):
        save_failed.emit("write tmp failed")
        return false
    if FileAccess.file_exists(path):
        if DirAccess.rename_absolute(path, bak) != OK:
            save_failed.emit("backup rename failed")
            return false
    if DirAccess.rename_absolute(tmp, path) != OK:
        save_failed.emit("publish rename failed")
        return false
    return true

func load_slot(slot: int) -> Dictionary:
    var path: String = SAVE_DIR + SAVE_NAME % slot
    var bak: String = path + ".bak"
    var data: Dictionary = _read_and_parse(path)
    if data.is_empty() and FileAccess.file_exists(bak):
        push_warning("primary save corrupted, falling back to .bak")
        data = _read_and_parse(bak)
    if data.is_empty():
        load_failed.emit("no readable save")
        return {}
    return SaveMigrations.migrate(data)
```

下半段是辅助方法和迁移注册:

```gdscript
func _write_file(path: String, content: String) -> bool:
    var f: FileAccess
    if _password.is_empty():
        f = FileAccess.open(path, FileAccess.WRITE)
    else:
        f = FileAccess.open_encrypted_with_pass(path, FileAccess.WRITE, _password)
    if f == null:
        push_error("open failed: %s" % FileAccess.get_open_error())
        return false
    f.store_string(content)
    f.close()
    return true

func _read_and_parse(path: String) -> Dictionary:
    if not FileAccess.file_exists(path):
        return {}
    var f: FileAccess
    if _password.is_empty():
        f = FileAccess.open(path, FileAccess.READ)
    else:
        f = FileAccess.open_encrypted_with_pass(path, FileAccess.READ, _password)
    if f == null:
        return {}
    var text: String = f.get_as_text()
    f.close()
    var json := JSON.new()
    if json.parse(text) != OK:
        push_warning("parse failed at line %d: %s" % [json.get_error_line(), json.get_error_message()])
        return {}
    var raw: Variant = json.data
    if typeof(raw) != TYPE_DICTIONARY:
        return {}
    return raw
```

迁移函数链单独放在一个工具文件,`res://systems/save_migrations.gd`,用 `class_name` 暴露成静态命名空间:

```gdscript
class_name SaveMigrations
## Stateless save data migrations. Each step lifts one schema version.

static func migrate(data: Dictionary) -> Dictionary:
    var version: int = int(data.get(&"schema_version", 0))
    while version < SaveSystem.SCHEMA_VERSION:
        match version:
            0: data = _v0_to_v1(data)
            1: data = _v1_to_v2(data)
            2: data = _v2_to_v3(data)
            _:
                push_warning("unknown save version %d, abort migrate" % version)
                break
        version = int(data.get(&"schema_version", version + 1))
    return data

static func _v0_to_v1(d: Dictionary) -> Dictionary:
    # v1: 引入 stamina;旧档默认满状态
    d[&"player"] = d.get(&"player", {})
    d[&"player"][&"stamina"] = 100
    d[&"schema_version"] = 1
    return d

static func _v1_to_v2(d: Dictionary) -> Dictionary:
    # v2: gold 改成 currencies.gold,为多币种留路
    var player: Dictionary = d.get(&"player", {})
    var gold: int = int(player.get(&"gold", 0))
    player.erase(&"gold")
    player[&"currencies"] = {&"gold": gold}
    d[&"player"] = player
    d[&"schema_version"] = 2
    return d

static func _v2_to_v3(d: Dictionary) -> Dictionary:
    # v3: 库存从数组改成字典 {item_id: count}
    var old: Array = d.get(&"inventory", [])
    var bag: Dictionary = {}
    for it: Dictionary in old:
        var id: StringName = it.get(&"id", &"unknown")
        bag[id] = int(bag.get(id, 0)) + int(it.get(&"count", 1))
    d[&"inventory"] = bag
    d[&"schema_version"] = 3
    return d
```

迁移链的关键不是函数本身,是它的形态:每一步都是 `(Dictionary, version=N) → (Dictionary, version=N+1)`,链式调用,版本只升不降。`SCHEMA_VERSION` 升一次就追加一个 `_vN_to_vN+1`,不去改前面的。每个迁移函数都假设输入是上一个版本的合法形态,如果旧档被玩家手改成奇怪结构,要么用 `get` 加默认值兜底,要么直接报错走回退路径。

调用方很简单:

```gdscript
# 任何业务节点
SaveSystem.save_slot(0, {
    "player": {"hp": 80, "position_x": 320.0, "position_y": 120.0, "currencies": {"gold": 42}},
    "level_id": "stage_2",
    "inventory": {"potion": 3, "key_red": 1},
})

var loaded: Dictionary = SaveSystem.load_slot(0)
if not loaded.is_empty():
    # 还原玩家状态
```

## 4. 调参和验收

**存档触发时机** 决定了"丢档窗口"的大小:每帧自动存太重(SSD 写入磨损 + 性能抖动);只有手动存太脆(玩家两小时没存崩了哭);折中是检查点(关卡切换、Boss 击破、对话结束)+ 退出确认。原型用"关卡切换时存 + 手动存"已经够。

**多存档槽位** 是产品决策不是技术决策。如果游戏是单线性、不可回头(如《Celeste》),一份自动存档就够;如果允许玩家在剧情分叉前手动存(如《巫师》),给 3-9 个槽位 + 自动存档 + 快速存档。`SAVE_NAME = "slot_%d.json"` 已经为多槽位留了路,UI 层给每个槽位显示"最近游玩时间 / 关卡名 / 截图"是常见做法,截图存独立 PNG 在 `saves/slot_0.png`,读档时一并显示。

**存档大小** 用 `JSON.stringify(payload, "\t")` 缩进后通常 5-50 KB,常规进度类游戏不需要担心。一旦 payload 超过 1 MB(比如塞了图像、长 log、整张地图状态),要考虑用 `.res` 二进制 + `FLAG_COMPRESS` 压缩(`ResourceSaver.FLAG_COMPRESS`),或者把大数据剥成独立文件。

**校验和** 是给"防误改"的最低成本方案。在 payload 里放 `&"checksum"` 字段,值是去掉 checksum 自身后的序列化 SHA-256(用 `HashingContext`)。加载时如果 checksum 对不上,提示玩家"存档被修改";不阻止玩家继续,但成就 / 排行榜不计入。

**版本号** 与游戏版本(`v1.2.3`)分开:游戏版本可能跨多个 `SCHEMA_VERSION`,也可能多个游戏版本共用同一个 `SCHEMA_VERSION`。版本号只代表"存档数据结构"的版本,不代表游戏版本。

**配置文件** 比存档更轻量。玩家的画面 / 音量 / 按键映射用 `ConfigFile`(简单 INI 风格)或者另一个独立 JSON 即可,不要塞进存档,免得读存档的代码兜底逻辑还要管音量。

**存档元数据** 是单独的概念:每个槽位除了游戏状态本身,还需要一份"摘要"——上次游玩时间、当前关卡名、累计游戏时长、玩家截图。这份摘要在主菜单的"读档"列表里展示,不该让玩家等一秒钟解析完整存档才看到。常见做法是把摘要单独存一份 `saves/slot_0.meta.json`,内容 1 KB 以内,加载主菜单时只读 meta 不读 full save。这是"两阶段读取"——meta 用于挑选,完整存档用于实际进入游戏。

验收线如下:

- 中途断电模拟:在 save 期间手动 kill 进程,重启后存档要么是旧版要么是新版,不能解析失败。
- 跨版本测试:用 `SCHEMA_VERSION = 0` 写一份原始档,把版本号升到 3,加载该档应自动跑过三步迁移并不崩。
- 损坏回退:手动把主文件改成乱码,加载时应回退到 `.bak`,游戏提示"使用了上一次有效存档"。
- 平台路径检查:在 macOS 上检查 `~/Library/Application Support/Godot/app_userdata/<project>/saves/`,Windows 上对应 `%APPDATA%`,Web 上用 DevTools → Application → IndexedDB 看到 `/userfs`。

## 5. 踩坑

**JSON 在 GDScript 4.x 是类方法,不是顶层函数。** 不要再用 `to_json` / `parse_json`,这是 3.x 的写法。`JSON.stringify(value, indent, sort_keys, full_precision)` 是静态方法;反向解析有两种:静态 `JSON.parse_string(text) -> Variant`(失败返回 `null`),或者实例化 `JSON.new()` 然后 `parse(text) -> Error`,后者可以拿到错误行号,定位玩家手改导致的坏档时很有用。如果存档损坏报告时要把行号回传给玩家(或上报到错误日志),`JSON.new()` 路线是唯一选择;`parse_string` 的静态版返回 null,出错信息全靠 `push_warning` 早就被扔掉了。

**`int` 经过 JSON 往返会变 `float`。** JSON 规范没有"整数"类型,只有 number。玩家金币 100 经过一次 save/load 后,在 GDScript 里 `typeof(gold) == TYPE_FLOAT`,如果后面用 `&"gold": int` 作类型标注的字段直接写,会触发 `incompatible types`。统一在加载侧用 `int(d.get(&"gold", 0))` 强转一次。

**`Vector2` 经过 JSON 不再是 `Vector2`。** 引擎类型在 `JSON.stringify` 时调 `.to_string()` 之类的转换,反向解析回来是普通字典或字符串。要么自己拆成 `pos_x` / `pos_y`,要么写一对 `to_dict` / `from_dict`。`Variant.serialize_bytes` / `Variant.deserialize_bytes` 可以保留完整类型,但是格式是 Godot 私有的二进制,玩家就再也看不到了——这是 JSON 路线的反面。

**`FileAccess` 不显式 close 在 Windows 上会出问题。** Windows 文件系统对句柄的可见性比较敏感。`var f := FileAccess.open(...)` 然后立刻 `DirAccess.rename(path, ...)`,在 Windows 上可能因为 `f` 还活着导致 rename 失败。`RefCounted` 离开作用域才释放,在函数末尾才 release。显式 `f.close()` 或 `f = null` 在 rename 之前调用是最稳的写法。

**`ResourceLoader.load` 在 `user://` 加载带脚本 Resource 是攻击面。** 如果你给存档选 `.tres` 路线,玩家可以伪造一个 `.tres` 文件指向恶意脚本路径(`script = ExtResource("res://...")`)。引擎加载时会运行脚本的 `_init`。`ResourceLoader.load(path, "", ResourceLoader.CACHE_MODE_IGNORE)` 不能阻止脚本加载。安全做法是:玩家数据走 JSON,Resource 只在 `res://` 这种发布期固化的位置加载。

**Web (HTML5) 的 `user://` 写完要 flush。** 浏览器版 Godot 把 `user://` 挂在 IndexedDB 上,普通写入是异步的,不主动 flush 关页面可能丢。Godot 4 提供 `OS.execute("flushFileSystem", [])` 或者依赖 `auto_flush`,具体策略要查导出版的官方文档;原型阶段最稳的做法是 save 后立刻 `OS.alert("已保存")` 一下,弹窗让浏览器有时间把异步写入提交。

**加密密钥硬编码不是安全。** `FileAccess.open_encrypted_with_pass(path, mode, "my_secret_key_123")` 把字符串直接写进二进制。任何能用 `strings` 命令扫一遍游戏目录的人都能拿到。这条路线只挡新手玩家。如果产品要"成就以服务端为准",根本不要用本地加密——服务端校验比客户端混淆有效一万倍。

**`DirAccess.rename_absolute` 在某些路径上行为不一致。** 有些平台(尤其安卓)在跨目录 rename 时返回 `ERR_FILE_CANT_WRITE`,需要先 `DirAccess.copy_absolute` 再 `remove_absolute`。原子性在这种平台上要靠 fsync 链而不是 rename。原型阶段先按 rename 走,导出到移动平台时再加平台分支。

**版本迁移要从 0 写到 N,不要从 N 写回 0。** 玩家几乎不会"降级",但你的迁移函数链是有向的:`v0 → v1 → v2 → v3`。永远不要写反向迁移,反向必然有信息丢失(stamina 字段去掉了,谁也救不回原值)。如果某次升级是"打回原形"的兼容性破坏,直接在 `migrate` 里检测出无法升级,提示玩家"游戏存档因重大更新无法兼容,请重开",比写错的迁移好得多。

**校验和 / 加密 / 迁移三件事互相耦合。** 校验和要在迁移之前算(校验的是磁盘真实状态),加密在最外层(密文是磁盘真实状态,明文才有版本号)。错了顺序的典型症状:玩家手改一个无关字段,游戏不报错(因为校验在迁移之后);或者明文版本号字段被加密之后看不到,加载时只能整个解密才能判断要不要迁移。

**`schema_version` 默认值是 0,不要是当前最新版本。** `data.get(&"schema_version", 0)` 默认 0 强制旧档走完整迁移链。如果默认成 `SCHEMA_VERSION`,旧档会被当成新档直接读字段,得到一堆 `null` 或类型错误。

**Steam Cloud 与本地存档的关系**。把游戏发到 Steam,Steam 会按 `steam_autocloud.vdf` 把 `user://` 镜像同步到云端。这里有两个隐藏成本:(1) 加密存档同步上去玩家在另一台机器解不开——除非密码是玩家相关的(Steam ID),否则跨账号失败;(2) `.bak` 文件也会被同步,云端占两倍空间。常见处理是只把主存档放进 Cloud 同步列表,`.bak` 留本地。

**Resource 存档的 ext_resource 路径要稳定**。如果你选 `.tres` 路线,存档里会写 `ext_resource("res://items/sword.tres")` 之类的路径。后期重构把这个文件移到 `res://content/items/` 下,所有旧存档读取都会失败,因为路径写死。要么用 `class_name` + uid(Godot 4.x 引入的 `uid://` 路径),要么坚持把"指向 resource 的引用"在存档里写成纯字符串 ID,加载时再 `ItemDB.find(id)` 查回 resource。

**版本字段必须在最外层**。`payload[&"schema_version"]` 不要嵌套到 `payload[&"meta"][&"schema_version"]`,因为迁移函数链就是基于这个字段判断走哪一步,嵌套之后某次迁移把 `meta` 改名了,版本字段也跟着丢——迁移系统再也找不到当前版本,陷入死循环或者直接当成 v0。
