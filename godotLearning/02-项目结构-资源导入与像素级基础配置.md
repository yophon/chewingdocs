# 02-项目结构、资源导入与像素级基础配置

> 一句话导读:Godot 不存"原始素材",它在第一次扫描资源时就把每个文件转成自己的运行时格式,这套"导入管线"决定了你看到的画面是不是变糊、字体是不是糊、音效是不是爆。把导入预设和项目结构在第一天钉死,后面 28 篇都受益。

打开一个空 Godot 4.6 项目,新手最直觉的做法是把美术给的 PNG 拖进 FileSystem dock,然后双击 Sprite2D 节点把图扔上去。能跑,但当像素图被放大到 4 倍后,边缘开始抗锯齿模糊,以为是分辨率问题,实际是导入时默认走了 `Linear` 过滤;当你换掉一张同名贴图,场景却没更新,以为是 Godot 的 Bug,实际是 `.import` 缓存没刷新;当你两人协作把项目推到 Git,合作者拉下来全是红色感叹号,实际是 `.godot/` 目录被错误地纳入了版本控制。这些坑都来自一个事实:**Godot 的资源管线不是"读文件、用文件",而是"读原始 → 导入成自己的格式 → 用导入产物"**,中间这一步如果没和你的项目结构配合好,就会持续制造小麻烦。

这一篇把这套管线讲透,把项目目录、导入预设、像素级渲染配置在第一天就钉死。同时讲清楚 Godot 4.6 引入的 Modern Editor Theme 和 Movable/Floatable Docks 怎么影响日常工作流,这两项 4.6 默认开启,直接决定了后续每篇截图与你本地编辑器是否对得上。

需要预先说明的边界:**这一篇不讨论某种具体艺术风格的取舍**,像素风、矢量风、低多边形 2.5D 都能在 Godot 里做,本篇示例倾向像素风(640×360 视口、Nearest 过滤),是因为它对错误配置最敏感、最容易暴露管线问题。如果你做的是 1080p 矢量风,把过滤模式换成 `Linear` 即可,其它结构性建议都通用。

## 1. 机制定位

游戏引擎的资源管理本质上有三种策略。第一种是"原始即运行时",比如 Lua + LÖVE,你给它 PNG 它就用 PNG,简单但运行时要花 CPU 解码、要适配各种格式、内存占用不可控。第二种是"全打包式",比如 Unity 的 AssetBundle,所有资源都被编译成专有格式,运行时直接 mmap,极致性能但工作流很重。第三种是"惰性导入",Godot 走的这条:你放进项目的 PNG 文件保留原样,但引擎会在后台生成一个同名 `.png.import` 描述文件,以及一份压缩好的 `.ctex`(compressed texture)产物缓存到 `.godot/imported/` 目录,运行时只读 `.ctex`,而你修改原图时引擎会自动重新生成。

这套机制带来三个直接后果。其一,你的项目目录里看到的 PNG 不是被实际加载的对象,真正被加载的是 `.ctex`,这是为什么"过滤模式""压缩格式"这些选项要写在 `.import` 文件里而不是 PNG 里。其二,`.godot/` 目录是机器生成的派生物,只能在本机存在,跨机不能共享,所以 `.gitignore` 必须把它排除。其三,所有导入参数变化都会让相关产物失效、重新生成,如果你切换分支带来一批 `.import` 修改,启动编辑器会先卡住批量重导,这是正常的。

新手在这一层最容易栽的坑,是把"原始素材文件夹"和"游戏运行时资源文件夹"混在一起。比如美术给的 PSD 源文件、AI 矢量稿、原始 WAV 录音,如果直接丢在 `res://` 下面,Godot 会尝试把它们也导入(PSD 支持有限,WAV 会被默认压缩),导致每次切分支都重导一遍,Git 历史里也充斥着不该被打包的素材。正确做法是 `res://assets/` 只放"已经导出成 PNG/OGG/TTF 的成品",原始工程文件放在仓库的另一个非 `res://` 目录,或者完全在仓库外管理。

另一类坑来自像素风项目。Godot 默认所有纹理走 `Linear` 过滤,这是为了让放大后的图看起来"平滑",对于矢量风、3D 截屏风是对的,对于像素风是灾难:你的 16×16 像素角色放到 4× 缩放就变成糊在屏幕上的色块。修复方式很简单,但必须在两个层面都改对:Project Settings 里的全局默认改成 `Nearest`,每张已经导入的纹理在 Import 面板里重导一次(否则旧 `.ctex` 仍是 Linear 烘焙)。这是后续不被"我的像素图为什么这么糊"反复缠住的唯一办法。

新手还容易把"过滤模式"和"压缩模式"混为一谈。过滤模式决定纹理被放大/缩小到非原生尺寸时如何采样,只影响渲染;压缩模式决定纹理在磁盘和显存里以什么格式存放,影响体积和加载速度。像素风通常用 Lossless(无损 PNG 编码)压缩 + Nearest 过滤,非像素风可以考虑 VRAM 压缩(BC、ETC、ASTC 等显卡原生格式)+ Linear 过滤。两者维度独立,不要因为"我是像素风"就关掉压缩,Lossless 不等于不压缩,只是不丢失像素值。

## 2. Godot 心智

把资源管线在脑子里建模,需要理解四类文件。

第一类是**原始资源**,比如 `player.png`、`music.ogg`、`title.ttf`,你放进 `res://` 下面看到的就是这些。它们在编辑器里能被预览,但运行时不直接用。

第二类是**导入描述文件**,后缀是 `.import`,例如 `player.png.import`。这是一份 INI 格式的纯文本,记录了"原始资源应该被怎么导入"。它的内容长这样:

```text
[remap]

importer="texture"
type="CompressedTexture2D"
uid="uid://b8h2j..."
path="res://.godot/imported/player.png-abcd1234.ctex"

[deps]

source_file="res://assets/player.png"
dest_files=["res://.godot/imported/player.png-abcd1234.ctex"]

[params]

compress/mode=0
compress/lossy_quality=0.7
detect_3d/compress_to=1
process/fix_alpha_border=true
process/premult_alpha=false
mipmaps/generate=false
```

`.import` 文件**必须进版本控制**,它定义了项目所有协作者会看到的导入产物长什么样;`.godot/` 目录是 `.import` 跑出来的产物,**必须不进版本控制**。

第三类是**导入产物**,放在 `.godot/imported/` 下面,后缀根据资源类型而定:纹理是 `.ctex`,音频是 `.oggvorbisstr` 或 `.sample`,字体是 `.fontdata` 之类。这些是引擎运行时实际读的文件,普通开发流程里你不需要直接打开它们。

第四类是 **`.tres` / `.tscn` 等 Godot 原生资源**,例如 `player.tscn`(场景)、`item_sword.tres`(自定义资源)、`default_theme.tres`(主题)。这些是你在 Godot 编辑器里直接创建、保存、编辑的,源文件就是运行时格式,没有"导入"这一步,改完直接生效。

理解了这四类,后续看到任何资源问题都能定位到层级:画面不对,查 `.import` 的过滤/压缩参数;改了图没生效,确认 `.godot/imported/` 里产物时间戳更新了没;协作者拉下来报错,八成是 `.import` 文件漏推或者 `.godot/` 被错误推了。

还要建立一个**资源标识**的心智。Godot 4.x 引入了 UID(uniform identifier),格式像 `uid://b8h2jak3z`,每个资源都会在第一次被导入时拿到一个全局唯一 ID,记录在 `.import` 文件和 `.godot/uid_cache.bin` 里。当你重命名或者移动一个资源文件时,引擎会用 UID 而不是路径去更新所有引用它的地方,这让"重构目录"在 Godot 项目里不再是高风险动作。但 UID 只在编辑器里通过 Move/Rename 操作时被维护,如果你用文件管理器直接 mv 文件,UID 链就断了,引用会丢。规则很简单:**项目内的资源移动一律在 FileSystem dock 里做,不要在 OS 里 mv**。

资源加载在代码里有三种姿势,各自语义不同。`preload("res://path")` 在脚本被解析时同步加载,适合"一定要用到"的常量资源(典型如 `const ENEMY_SCENE = preload(...)`)。`load("res://path")` 在执行到该行时同步加载,可以传变量路径,适合运行时根据数据决定加哪个资源。`ResourceLoader.load_threaded_request(path)` 异步加载,主线程不阻塞,适合大资源(关卡、过场动画),配合 `load_threaded_get_status` 与 `load_threaded_get` 取结果,这是第 24 篇的核心 API。后续每篇都会按这三种姿势的边界选择,你只要先记住"preload 编译期、load 运行时同步、threaded 运行时异步"。

## 3. 工程实现

把上一篇立起来的骨架填实。在项目根目录新建 `.gitignore`,内容如下,这是 Godot 4.x 官方推荐与社区共识的最小版本:

```text
# Godot 4.x .gitignore
# Godot 生成的导入产物缓存,跨机不能共享
.godot/

# IDE / 编辑器临时文件
.vs/
.idea/
.vscode/
*.swp
*.swo

# 平台导出产物
export_presets.cfg.bak
*.translation
build/
```

注意 `export_presets.cfg` 本身**应该**进版本控制(它定义了如何为各平台打包),但 `.translation` 是从 CSV 编译出的产物。`.import` 文件**必须**进版本控制,默认不在忽略里就是对的。

接下来钉死项目级渲染配置。直接在 `project.godot` 里改文本最快,这个文件就是 INI,Godot 编辑器也是读写它。打开 `res://project.godot`,在对应章节加入下面这些键(如果已经有同名键,改值即可):

```ini
[application]

config/name="MyGame"
config/features=PackedStringArray("4.6", "GL Compatibility")
config/icon="res://icon.svg"
run/main_scene="res://scenes/main.tscn"

[display]

window/size/viewport_width=640
window/size/viewport_height=360
window/size/window_width_override=1280
window/size/window_height_override=720
window/stretch/mode="canvas_items"
window/stretch/aspect="keep"

[rendering]

textures/canvas_textures/default_texture_filter=0
textures/canvas_textures/default_texture_repeat=0
2d/snap/snap_2d_transforms_to_pixel=true
2d/snap/snap_2d_vertices_to_pixel=true

[physics]

common/physics_ticks_per_second=60

[gui]

theme/default_font_subpixel_positioning=0
```

逐项说明。`viewport_width/height` 设 640×360,是 16:9 的整数倍数,1× / 2× / 3× / 6× 都能整数放大到主流分辨率,像素风首选。`window_width_override` 让默认窗口启动时就是 2× 大小,避免一打开是个邮票。`stretch/mode=canvas_items` 让 2D 内容随视口缩放但保持像素精确,`aspect=keep` 让宽高比变化时加黑边而不是拉伸变形。

`default_texture_filter=0` 是 `Nearest`(枚举值见 `BaseMaterial3D.TextureFilter` 与 `CanvasItem.TextureFilter`,0 是最近邻、1 是线性、2 是带 mipmap 的线性),这是把全局像素风过滤一次性钉死的开关。如果你做的是非像素风,这一项改成 1。`snap_2d_transforms_to_pixel` 与 `snap_2d_vertices_to_pixel` 让 2D 节点位置和顶点都对齐到整数像素,避免亚像素抖动,像素风必开,非像素风可以关。

`physics_ticks_per_second=60` 与上一篇一致,后续 06 篇会展开为什么不是 30 或 120。

字体的 `subpixel_positioning=0` 把所有默认字体的亚像素定位关掉,理由是 2D 游戏 UI 一般缩放显示,亚像素定位反而让字体在不同缩放下糊得不一致;关掉后整数像素对齐,锐利且可预期。

接下来设置一份**纹理导入预设**,这样后续每张新拖进来的图都自动走对参数。在编辑器里 Import 标签 → 选中任意一张 PNG → 点 "Preset..." 下拉 → "Set as Default for Texture"。把当前参数(过滤模式、压缩模式、Mipmap)保存为默认。建议两组预设:

- **像素风默认**:Compress Mode=Lossless(无损),Generate Mipmaps=off,Fix Alpha Border=on,High Quality=off。
- **非像素风默认**:Compress Mode=VRAM Compressed,Generate Mipmaps=on(仅当贴图会被缩放显示),Fix Alpha Border=on。

下面这段脚本演示如何用代码读取一个导入后的资源元数据,放在 `res://tools/import_inspector.gd`,后续做插件章节会用到:

```gdscript
# res://tools/import_inspector.gd
@tool
class_name ImportInspector
extends RefCounted

## 工具脚本:查 .import 文件解读资源导入参数。
## 用途:在 CI 或编辑器命令里批量检查所有图是否走了正确预设。

static func dump(resource_path: String) -> Dictionary:
    var import_path: String = resource_path + ".import"
    if not FileAccess.file_exists(import_path):
        push_warning("no .import for %s" % resource_path)
        return {}
    var cfg: ConfigFile = ConfigFile.new()
    var err: Error = cfg.load(import_path)
    if err != OK:
        push_error("failed to load %s: %s" % [import_path, err])
        return {}
    var result: Dictionary = {
        "importer": cfg.get_value("remap", "importer", ""),
        "type": cfg.get_value("remap", "type", ""),
        "uid": cfg.get_value("remap", "uid", ""),
    }
    if cfg.has_section("params"):
        for key in cfg.get_section_keys("params"):
            result[key] = cfg.get_value("params", key)
    return result

static func check_all_textures_are_nearest(root_dir: String = "res://assets") -> Array[String]:
    var bad: Array[String] = []
    var dir: DirAccess = DirAccess.open(root_dir)
    if dir == null:
        return bad
    dir.list_dir_begin()
    var name: String = dir.get_next()
    while name != "":
        if dir.current_is_dir() and not name.begins_with("."):
            bad.append_array(check_all_textures_are_nearest(root_dir.path_join(name)))
        elif name.ends_with(".png") or name.ends_with(".jpg"):
            var meta: Dictionary = dump(root_dir.path_join(name))
            if meta.get("detect_3d/compress_to", -1) == 0:
                # 0 = Detect 3D auto, 这里只是占位检查
                pass
        name = dir.get_next()
    return bad
```

这段代码不是必须立刻用,但它展示了一件事:`.import` 文件可以被脚本读取,后期做 CI 校验(比如"所有 sprite 必须是 Lossless 压缩")很简单,不需要额外工具链。

最后整理目录,确认上一篇定义的结构现在已经实化:

```text
res://
├── .gitignore
├── project.godot
├── icon.svg
├── assets/
│   ├── sprites/        # 角色、敌人、物品 PNG
│   ├── tiles/          # TileSet 用的瓦片图
│   ├── audio/          # OGG (BGM) / WAV (短音效)
│   └── fonts/          # TTF/OTF
├── data/               # .tres 配置资源
├── scenes/
│   ├── main.tscn
│   └── main.gd
├── player/
├── enemies/
├── ui/
├── globals/
│   └── game_log.gd
└── tools/
    └── import_inspector.gd
```

`audio/` 下两种格式有明确分工:OGG Vorbis 走流式(`AudioStreamOggVorbis`),适合背景音乐;WAV 走采样(`AudioStreamWAV`),适合频繁触发的短音效。Godot 4.6 默认导入预设对两者已经分别配好,你不需要手动改,只要确认文件名后缀对就行。

为什么不全用一种格式简化心智:WAV 没有压缩,体积是 OGG 的 5-10 倍,几首 BGM 直接把游戏包体撑到几百 MB,Web 导出尤其受不了;OGG 在解码上比 WAV 贵,且无法在毫秒级精确触发,枪声、脚步声、按钮声这种"频繁、短促、要精准"的场景如果走 OGG 会出现延迟。所以分工是工程必要,不是炫技。第 20 篇会展开音频总线设计。

## 4. 调参和验收

Godot 4.6 引入两项编辑器层面的变更,直接影响日常工作流和后续每篇截图的可读性,必须先调整好。

**Modern Editor Theme**。4.6 默认主题改为 Modern,是从社区插件 "Godot Minimal Theme" 上游化而来。它的特征是去蓝偏色的灰阶配色和更紧凑的边距。对游戏开发来说,这个改动的最大价值是"色彩中立":之前 Classic 主题带轻微蓝色调,你在编辑器里调一个偏暖的 Sprite 颜色会感觉冷,运行时反而看着对,这是色觉污染。Modern 把这个干扰拿掉了。如果你倾向 Classic,在 Editor Settings → Theme → Preset 切回即可,但本系列后续截图统一基于 Modern。

需要补一句的是,主题切换只是改 UI 的视觉,不会影响项目本身的运行表现。所以你"用 Classic 工作的同事"和"用 Modern 工作的你"提交的 `.tscn`、`.gd` 完全一致,只是各自看到的编辑器长得不一样。这点和某些 IDE 主题"影响 lint 行为"是不同的,Godot 编辑器主题是纯视觉,不需要在团队里强制统一。

**Movable / Floatable Docks**。4.6 把 dock 系统重写,所有 dock(FileSystem、Scene、Inspector、Import、Output、Debugger)都可以拖到编辑器四边的任意位置,也可以拉出成独立窗口。底部 panel(Output / Debugger / Animation / Shader Editor)和侧边 dock 现在被统一对待,可以互相拖换位置。对单显示器小项目这件事意义不大,但对一人独立开发者,常见场景是:Shader 编辑 / Animation 编辑 / 主视口三件套同时显示。4.6 之前要靠 Tab 切换,4.6 之后可以把 Shader 编辑器拉到第二块显示器浮起来,主视口腾出来给场景编辑。

具体几个推荐布局可以提一下,后续每篇都默认你能切换到这些状态而不卡壳。第一,**调动画的时候把 Animation 底部 panel 拉成独立窗口扔到第二屏**,主显示器留给视口和 Inspector,这样调关键帧时不挤。第二,**调 shader 的时候把 Shader Editor 浮起来**,实时看参数变化对主视口的影响。第三,**Scene + FileSystem 两个 dock 并列在左侧**(默认就是这样,不要拆开),它们的协同决定了大部分编辑器导航效率。

需要注意两件事。第一,dock 布局是编辑器级别的,不在项目里保存,新机器要重新配。第二,Floating dock 在某些 Linux WM 下会有 focus 抢占问题,4.6.x 仍在修,如果遇到键盘事件丢失,先把 dock 拖回来。

调好以后,本篇验收标准:

- [ ] 项目根目录有 `.gitignore`,`.godot/` 被忽略,`.import` 没被忽略。`git status` 干净后,新建一张 PNG 并保存,Godot 自动生成 `.import` 同名文件,且该 `.import` 出现在 `git status` 里(说明会进入版本控制)。
- [ ] `project.godot` 中的 `display`、`rendering`、`physics` 章节键值与第 3 节一致,F5 启动游戏窗口为 1280×720,内部视口 640×360。
- [ ] 拖一张 32×32 像素图进 `assets/sprites/`,挂到 Sprite2D,放大到 4× 显示,边缘是清晰的方块而不是渐变模糊。如果还是糊的,Import 面板里点 Reimport。
- [ ] 拖一份 WAV 进 `assets/audio/`,Import 面板自动识别为 AudioStreamWAV;拖一份 OGG 进去,识别为 AudioStreamOggVorbis。
- [ ] Editor Settings → Theme → Preset 是 Modern。把 Shader Editor dock 拉出来变成独立窗口,关掉再重开编辑器,布局能恢复(布局保存到本机 editor settings)。

## 5. 踩坑

第一类是 **`.godot/` 被推进了 Git**。这是新手第一次提交时最常踩的坑,因为 `.godot/` 是隐藏目录,GitHub Desktop 类工具默认不过滤。后果是,合作者拉下来时本地路径不一致(`.godot/imported/<hash>.ctex` 的 hash 与 UID 缓存依赖机器路径),整个项目报错。修复:`.gitignore` 加 `.godot/`,然后 `git rm -r --cached .godot/` 把已经提交的清掉,推一次,合作者下次 pull 后让 Godot 重新生成。

第二类是 **`.import` 被忽略**。有些教程里没区分清楚,有人把 `.import` 也加到 `.gitignore`,结果合作者拉下来后所有图都需要重导入,且导入参数全是默认值,像素图全部模糊。规则记牢:`.import` 必须进版本控制,`.godot/` 必须不进。

第三类是**改了图没生效**。两种原因。一种是 Godot 编辑器没有焦点,文件系统监听没触发,把编辑器切到前台让它扫描一下。另一种是改了导入参数没点 Reimport,这种情况下 `.import` 已经更新但 `.ctex` 还是旧的。最安全的做法是参数修改后,选中文件 → 右下角 Reimport 按钮。

第四类是 **`res://` 文件名空格与中文**。Godot 文件名本身支持中文和 Unicode,但跨平台导出(尤其是早期 Web 导出)对非 ASCII 路径处理过有 bug,且 `.import` 里的 hash 算法对非 ASCII 路径稳定性较差。建议:**所有 `res://` 下的文件名用小写、下划线、ASCII**。中文应该出现在游戏内显示文本(放在 translation csv 里),而不是出现在文件名里。

第五类是**用 OS 文件管理器移动资源**。这会导致 UID 还指向旧路径,引用全部报错。Godot FileSystem dock 提供完整的 Move/Rename 功能,且会自动更新所有引用。如果不小心在 OS 里 mv 了,补救方式是手动把 `.import` 文件一并移动并刷新编辑器,但更稳的是 reset 后在编辑器里重做。

第六类是**像素风项目误开 Mipmap**。Mipmap 是给"贴图会被缩小显示"用的优化,2D 像素风的纹理基本不会被缩小到原图一半以下,开了反而增加 33% 内存且让贴图边缘出现 mipmap-bleed。像素风全部关掉,只对 3D 模型贴图或者大背景图开启。

第七类是 **Modern Theme 旧教程截图对不上**。4.6 默认 Modern 后,网上 2024-2025 年大量教程的截图仍是 Classic,按钮位置和图标颜色都不一样,跟着照搬时会有"我点的按钮不长这样"的恐慌。两个应对:遇到 4.5 之前的教程默认它的截图基于 Classic,功能位置一般没变;或者你临时切到 Classic 对照学完再切回。

第八类是**美术资源放错目录导致 git LFS 没生效**。如果你给项目配了 Git LFS,要把 `.png`、`.ogg`、`.wav` 这些二进制扩展名加进 `.gitattributes`。常见错误是只配了 LFS 但没配 `.gitattributes`,推上去的还是普通 blob,几次提交后仓库膨胀得 clone 都慢。本系列后续假设你的二进制资源走 LFS,如果不打算用 LFS,至少要保证原始素材足够小(单文件 1MB 以下),否则后期协作会卡。
