# 项目结构、资源导入与像素级基础配置

Godot 新项目第一天最容易做错的事,不是代码写错,而是**资源和目录一开始就乱了**。你把 PNG、PSD、WAV、测试脚本、导出包全丢进一个目录,项目还能跑;等两周后,像素图变糊、同事拉下来全是红色感叹号、Git 里多出几百 MB 缓存文件、换一张图还不生效,你才发现问题不在玩法,在项目骨架。

这一篇只做一件事:**把 Godot 项目的第一天配置钉死**。目录怎么分、哪些文件进 Git、哪些文件不能进 Git、像素图为什么会糊、音频该用 WAV 还是 OGG,都在这篇处理掉。后面写角色、敌人、关卡时,不用再回头修地基。

> 一句话先记住:**Godot 不是直接使用你拖进去的原始文件,而是先导入成自己的运行时资源;`.import` 记录怎么导入,`.godot/` 存导入产物**。所以 `.import` 要进 Git,`.godot/` 不能进 Git。

---

## 一、先把项目目录定下来

一个 2D 独立游戏项目,第一天就按这个结构建:

```text
res://
├── assets/
│   ├── sprites/        # 角色、敌人、道具 PNG
│   ├── tiles/          # 瓦片图、TileSet 相关素材
│   ├── audio/          # 音效和音乐
│   └── fonts/          # TTF / OTF 字体
├── data/               # .tres 配置:武器、敌人、掉落表
├── scenes/             # 主场景、关卡、菜单
├── player/             # 玩家场景和脚本
├── enemies/            # 敌人场景和脚本
├── ui/                 # HUD、菜单、主题
├── globals/            # 少量 Autoload
├── tools/              # @tool 脚本、编辑器辅助工具
└── project.godot
```

不要建一个巨大的 `scripts/` 目录把所有脚本丢进去。脚本应该跟它服务的场景靠近:

```text
player/
├── player.tscn
├── player.gd
└── player_stats.tres

enemies/
├── slime.tscn
├── slime.gd
└── slime_data.tres
```

这样你改玩家时只看 `player/`,改敌人时只看 `enemies/`。项目越大,这种"按功能放一起"越省命。

**几个目录边界:**

| 目录 | 放什么 | 不放什么 |
| --- | --- | --- |
| `assets/` | 已经能被游戏直接用的图片、音频、字体 | PSD、AI、工程源文件 |
| `data/` | `.tres` 配置资源 | 大图片、大音频 |
| `scenes/` | 主流程场景、关卡、菜单 | 玩家/敌人的细分脚本 |
| `globals/` | 真正全局的单例 | 所有业务逻辑 |
| `tools/` | 编辑器工具、调试工具 | 正常玩法代码 |

> 规则很简单:**`res://` 下只放游戏会用到的成品资源**。原始美术工程文件可以放仓库外,或者放一个不被 Godot 扫描的目录,不要让 Godot 每次启动都导入一堆用不到的 PSD。

## 二、Godot 的资源导入到底在干嘛

你把 `player.png` 拖进 Godot 后,项目里会出现两个东西:

```text
assets/sprites/player.png
assets/sprites/player.png.import
```

同时 Godot 还会在隐藏目录里生成导入产物:

```text
.godot/imported/player.png-xxxx.ctex
```

三者分工不同:

| 文件 | 是什么 | 要不要进 Git |
| --- | --- | --- |
| `player.png` | 原始图片 | 要 |
| `player.png.import` | 导入参数:过滤、压缩、UID、目标产物 | 要 |
| `.godot/imported/*.ctex` | Godot 生成的运行时缓存 | 不要 |

`.import` 文件大概长这样:

```ini
[remap]

importer="texture"
type="CompressedTexture2D"
uid="uid://abc123..."
path="res://.godot/imported/player.png-abcd.ctex"

[deps]

source_file="res://assets/sprites/player.png"
dest_files=["res://.godot/imported/player.png-abcd.ctex"]

[params]

compress/mode=0
mipmaps/generate=false
process/fix_alpha_border=true
```

你不需要手写它,但必须知道它的意义:**它决定同一张图在所有人机器上怎么导入**。

如果 `.import` 没进 Git,同事拉项目后 Godot 会用默认参数重新导入。结果常见就是:

- 你的像素图是清晰的,同事那边是糊的
- 你的贴图没 mipmap,同事那边开了 mipmap
- 你这边音频是流式播放,同事那边变成一次性加载

如果 `.godot/` 进了 Git,问题反过来:

- 仓库变大
- 每个人机器上的导入缓存互相覆盖
- 路径和 UID 缓存可能冲突
- 合作者拉下来一堆莫名其妙的导入错误

> 记住这句:**原始资源和 `.import` 是源码,`.godot/` 是构建产物**。源码进 Git,构建产物不进 Git。

## 三、第一天就写好 `.gitignore`

在项目根目录新建 `.gitignore`:

```gitignore
# Godot 生成的导入缓存和编辑器状态
.godot/

# 编辑器 / IDE 临时文件
.vscode/
.idea/
.vs/
*.swp
*.swo

# 导出产物
build/
exports/
*.zip
*.dmg
*.exe
*.app
*.pck

# 临时备份
*.tmp
*.bak
```

注意两点:

1. **不要忽略 `.import`**  
   `.import` 必须提交。它不是缓存,它是导入配置。

2. **`export_presets.cfg` 通常应该提交**  
   它记录导出平台、图标、包名等设置。团队协作时,大家应该使用同一套导出配置。

如果你已经不小心把 `.godot/` 提交了,修复方式是:

```bash
git rm -r --cached .godot
git commit -m "Ignore Godot generated cache"
```

这只会从 Git 索引里移除 `.godot/`,不会删你本地文件。下次打开项目,Godot 会自己重新生成。

## 四、像素图为什么会糊

像素风项目最常见的第一坑:

> 16x16 的角色图,放大 4 倍后边缘变模糊。

原因通常不是图片分辨率不够,而是纹理过滤模式错了。

纹理被放大时,引擎要决定"两个像素之间的颜色怎么取":

| 过滤模式 | 效果 | 适合 |
| --- | --- | --- |
| `Nearest` | 最近邻,方块清晰 | 像素风 |
| `Linear` | 线性插值,边缘平滑 | 非像素风、插画、照片 |

像素风要用 `Nearest`。否则你的每个像素都会被插值成渐变色,看起来就糊。

在 Project Settings 里改:

```ini
[rendering]

textures/canvas_textures/default_texture_filter=0
textures/canvas_textures/default_texture_repeat=0
2d/snap/snap_2d_transforms_to_pixel=true
2d/snap/snap_2d_vertices_to_pixel=true
```

这里的 `default_texture_filter=0` 就是 `Nearest`。

还要注意:**已经导入过的图片,改全局默认后不一定自动更新**。选中图片,到 Import 面板点 `Reimport`。

> 判断是否配置对了很简单:找一张 16x16 或 32x32 的 PNG,挂到 `Sprite2D`,把 Scale 调到 `4, 4`。边缘还是清晰方块,就是对的;边缘有渐变,就是过滤没改对。

## 五、推荐的基础项目设置

如果你做的是像素风或低分辨率 2D 原型,先用这组设置:

```ini
[application]

config/name="MyGame"
run/main_scene="res://scenes/main.tscn"

[display]

window/size/viewport_width=640
window/size/viewport_height=360
window/size/window_width_override=1280
window/size/window_height_override=720
window/stretch/mode="canvas_items"
window/stretch/aspect="keep"

[rendering]

textures/canvas_textures/default_texture_filter=0
textures/canvas_textures/default_texture_repeat=0
2d/snap/snap_2d_transforms_to_pixel=true
2d/snap/snap_2d_vertices_to_pixel=true

[physics]

common/physics_ticks_per_second=60

[gui]

theme/default_font_subpixel_positioning=0
```

逐项翻译成人话:

| 设置 | 意思 |
| --- | --- |
| `viewport_width=640`, `viewport_height=360` | 游戏内部逻辑分辨率是 640x360 |
| `window_width_override=1280`, `window_height_override=720` | 启动窗口默认放大 2 倍 |
| `stretch/mode="canvas_items"` | 2D 内容跟着视口缩放 |
| `stretch/aspect="keep"` | 保持比例,不拉伸变形 |
| `default_texture_filter=0` | 默认最近邻采样 |
| `snap_2d_*_to_pixel=true` | 2D 坐标对齐像素,减少抖动 |
| `physics_ticks_per_second=60` | 物理逻辑默认每秒 60 次 |

如果你不是像素风:

- `default_texture_filter` 可以改成 `1`(`Linear`)
- `snap_2d_transforms_to_pixel` 可以关掉
- 分辨率可以直接用 1280x720 或 1920x1080

但目录结构、`.gitignore`、`.import` 规则都不变。

## 六、图片导入怎么选

选中一张图片,Import 面板里最关键的几个选项是:

| 选项 | 像素风建议 | 非像素风建议 |
| --- | --- | --- |
| Filter | Off / Nearest | On / Linear |
| Mipmaps | Off | 视情况 On |
| Compress Mode | Lossless | VRAM Compressed 或 Lossless |
| Fix Alpha Border | On | On |

几个容易混的概念:

**Filter 是放大缩小时怎么采样。**  
像素风关掉,非像素风开。

**Mipmaps 是给缩小显示用的多级贴图。**  
2D 像素风通常关。大背景图、会缩放很厉害的贴图可以开。

**Compress 是怎么存。**  
Lossless 不等于"不压缩",只是说不丢像素值。像素风一般用 Lossless。大量大图可以考虑 VRAM Compressed,但要检查画质。

**Fix Alpha Border 通常开。**  
它能减少透明边缘出现黑边/白边的问题。角色、道具、UI 图标都建议开。

> 第一阶段不要追求每张图都调到最优。先定一个默认预设:像素风 `Nearest + Lossless + No Mipmap`;非像素风 `Linear + 合理压缩 + 视情况 Mipmap`。

## 七、音频格式怎么选

别把所有音频都转成同一种格式。短音效和背景音乐需求不同:

| 类型 | 推荐格式 | 原因 |
| --- | --- | --- |
| 按钮声、脚步声、攻击声 | WAV | 短、触发频繁、延迟低 |
| BGM、环境音、长语音 | OGG | 体积小,适合流式播放 |

为什么不用 WAV 放所有东西?因为 WAV 体积太大。一首几分钟的背景音乐可能几十 MB,Web 导出很快爆炸。

为什么不用 OGG 放所有东西?因为 OGG 要解码,短促且频繁触发的音效不适合全走它。

目录可以这样放:

```text
assets/audio/
├── sfx/
│   ├── jump.wav
│   ├── hit.wav
│   └── click.wav
└── music/
    ├── title.ogg
    └── stage_01.ogg
```

> 一句话:**短音效用 WAV,长音乐用 OGG**。等第 20 篇讲音频总线时,再细分音量分组、混音和动态音效。

## 八、资源命名规则

Godot 支持中文路径,macOS 也支持各种 Unicode 文件名。但跨平台发布和团队协作时,资源路径越朴素越好。

建议所有 `res://` 下的文件名遵守:

```text
小写英文 + 数字 + 下划线
```

例子:

```text
player_idle.png
player_run_01.png
enemy_slime.tscn
stage_01_tiles.png
ui_button_normal.png
weapon_sword_basic.tres
```

不建议:

```text
玩家待机.png
Player Run Final FINAL.png
stage#1@new.png
按钮-普通状态.png
```

原因不是 Godot 不能读,而是:

- Web / Windows / macOS 打包时路径规则不完全一样
- Git 合并时大小写差异容易出问题
- `.import` 路径变动后更难排查
- 终端脚本批量处理更麻烦

> 游戏里的中文文本应该放进本地化表或资源字段里,不要放进文件名里。

## 九、移动资源别用 Finder

Godot 4.x 有资源 UID。你在编辑器的 FileSystem 面板里移动或重命名资源时,Godot 会帮你更新引用。

正确做法:

```text
在 Godot FileSystem 面板里 Move / Rename
```

风险做法:

```text
在 Finder / Explorer / mv 命令里直接移动 res:// 文件
```

后者可能导致:

- 场景里引用的图片丢失
- `.import` 文件没跟着移动
- UID 缓存和路径对不上
- 编辑器里出现红色缺失引用

如果只是新建文件,用系统文件管理器没问题。**但移动、重命名、整理目录,优先在 Godot 编辑器里做**。

## 十、第一天的最小主场景

配完目录和资源后,建一个最小主场景:

```text
Main (Node2D)
├── World (Node2D)
├── UI (CanvasLayer)
└── Camera2D
```

保存为:

```text
res://scenes/main.tscn
```

挂一个最小脚本:

```gdscript
# res://scenes/main.gd
class_name Main
extends Node2D

@onready var world: Node2D = $World
@onready var ui_layer: CanvasLayer = $UI
@onready var camera: Camera2D = $Camera2D

func _ready() -> void:
    print("main scene ready")

func _unhandled_input(event: InputEvent) -> void:
    if event.is_action_pressed("ui_cancel"):
        get_tree().quit()
```

然后在 Project Settings 里把 main scene 指到:

```text
res://scenes/main.tscn
```

这一步的价值不是代码本身,而是把项目分成三层:

| 节点 | 负责 |
| --- | --- |
| `World` | 玩家、敌人、地图、掉落物 |
| `UI` | HUD、菜单、弹窗 |
| `Camera2D` | 镜头 |

后面所有玩法内容都往这个骨架里加,而不是每篇新建一个不相关 demo。

## 十一、验收清单

做完这一篇,项目应该满足这些条件:

1. **目录结构清楚**  
   有 `assets/`、`data/`、`scenes/`、`player/`、`enemies/`、`ui/`、`globals/`、`tools/`。

2. **`.gitignore` 正确**  
   `.godot/` 被忽略,`.import` 没有被忽略。

3. **导入流程正确**  
   拖一张 PNG 到 `assets/sprites/`,Godot 自动生成同名 `.import`,并且 `git status` 能看到这两个文件。

4. **像素图不糊**  
   32x32 图片放大 4 倍后仍是清晰方块。如果糊,检查 Filter 和 Reimport。

5. **窗口尺寸正确**  
   F5 启动后,窗口默认 1280x720,内部视口按 640x360 设计。

6. **音频目录清楚**  
   短音效在 `assets/audio/sfx/`,背景音乐在 `assets/audio/music/`。

7. **主场景可运行**  
   `scenes/main.tscn` 能启动,控制台打印 `main scene ready`,按 Esc 能退出。

---

## 十二、踩坑提醒

1. **把 `.godot/` 提交到 Git**  
   这是导入缓存,不是源码。提交后仓库会变大,协作时还容易出奇怪问题。

2. **把 `.import` 忽略掉**  
   `.import` 是导入规则。忽略它,同事机器上的资源效果可能和你完全不同。

3. **像素风忘了关 Linear**  
   画面糊先查 Filter,不要先怀疑图片分辨率。

4. **改导入参数后不 Reimport**  
   参数改了不代表旧导入产物立刻重建。选中文件点 Reimport。

5. **把原始素材全丢进 `res://`**  
   PSD、AI、工程源文件会拖慢导入,也会污染游戏包体。`res://` 只放游戏成品资源。

6. **文件名带空格、中文和特殊符号**  
   能跑不代表适合发布。资源路径统一小写英文和下划线。

7. **用系统文件管理器重命名资源**  
   资源移动和重命名尽量在 Godot FileSystem 面板里做,让它帮你更新引用。

8. **一开始就做复杂导出配置**  
   先让主场景跑起来。导出、补丁包、平台差异放到第 26 篇处理。
