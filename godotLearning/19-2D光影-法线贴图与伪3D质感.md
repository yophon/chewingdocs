# 19-2D 光影、法线贴图与伪 3D 质感

> 一句话导读:2D 光影不是物理还原,而是用光照通道、法线贴图和阴影遮挡器,把平面贴图骗出有厚度、有阴影、有时间感的画面;它真正的代价不在算法,而在 draw call。

很多人第一次开 `PointLight2D` 时会被惊艳:拖一个节点就有了光斑,加一个 `LightOccluder2D` 就能投出阴影,看上去几乎免费。但只要场景里塞到三四十个火把、几十个动态物体,帧率立刻塌下来。这一篇要回答的核心问题不是"怎么打开 2D 光照",而是:

- 2D 光是什么——它如何参与 `CanvasItem` 的渲染管线?
- 法线贴图为什么能在平面 sprite 上骗出立体感?它对光源的"高度"和方向的要求是什么?
- 阴影是怎么算出来的?为什么 `LightOccluder2D` 不能用 `CollisionPolygon2D` 替代?
- 光源数量、阴影开启、`light_mask` 设计,如何影响 draw call?

读完后,这一篇要让你能在场景里设计一套"昼夜光照 + 火把交互光 + 局部法线贴图"的层级方案,并清楚它的性能预算。下一篇(20)会进入粒子和音频反馈,21 篇会从底层用 shader 自己写光照效果,本篇只在节点层把光打稳。

## 1. 机制定位

### 2D 光照不是物理,而是表演

Godot 的 2D 光照管线本质是一个**额外的颜色合成 pass**:`CanvasItem` 先按贴图和 `modulate` 画出底色,然后所有命中其 `light_mask` 的 `Light2D` 按 `blend_mode` 与底色做合成。`Add` 把光值加到底色上得到"被照亮"的效果,`Subtract` 用于负光(吃掉局部颜色),`Mix` 则在像素的 alpha 上做插值。

这就决定了三件事:

1. 2D 光不是"投射在物体上的光子",它是一张以 `Light2D.texture` 为形状的纹理,被 GPU 按位置叠加到场景。所以一个 `PointLight2D` 的"光斑"就是一张径向渐变图,圆角、椭圆甚至光晕都能直接通过换 texture 实现——你只要画一张合适的 texture,Godot 不关心它"物理上是不是球形衰减"。
2. 一个 `CanvasItem` 收不到光,通常不是 bug,而是 `light_mask` / `Range > Layer Min/Max` / `Z Min/Max` 没对上。光照系统的可见性是按位匹配,不是按层级继承——把 sprite 放在某个 `Node2D` 子树下,父节点的 `light_mask` 不会传递给子节点。
3. 阴影不靠贴图,而靠 `OccluderPolygon2D` 在每一帧投出射线相减。阴影只对**开启了 `shadow_enabled` 的 Light2D** 生效,且只阻挡那些 `occluder_light_mask` 命中的光源。

容易问一句:为什么不用 `ColorRect` + 自定义 shader 自己写"光照系统"?技术上可以,但代价是把可见性裁剪、纹理合成、阴影几何这些 RenderingServer 已经做好的事全部重做一遍。Godot 的 2D 光节点本质上是 RenderingServer 暴露给游戏层的"快路径":它们与 `CanvasItem` 共享同一套渲染指令缓冲,光合成在底色合成的下一阶段直接完成,不需要额外的 framebuffer 切换。第 21 篇会展示用 shader 做局部光照贴花的反例,做出来与 `PointLight2D` 视觉相近,但失去了 occluder 联动,且 draw call 不可控。

### 新手常见写法为什么失控

最常见的两类坑都不是"看不到光":

第一类是"光看上去对,法线没生效"。原因几乎一定是 `Sprite2D.texture` 用的是普通 `Texture2D`,法线只能挂在 `CanvasTexture` 上;或者 `PointLight2D.height = 0`,光源就贴在表面上,法线 N 与光向 L 几乎正交,`dot(N, L)` ≈ 0,自然看不出明暗。

第二类是"几十盏光以后帧率塌"。每一盏开启了阴影的 `PointLight2D` 都要遍历可见区域内的 `LightOccluder2D` 重新生成阴影几何;每一盏没开阴影但每帧都改动 `position` 的 `PointLight2D` 也会打断 2D 批处理(batching),把本来一个 draw call 能画完的 200 个 sprite 拆成更多个。后面"调参和验收"一节会给出具体的预算。

本篇不会展开 Shader(那是第 21 篇的事),也不会重复 `Resource` 配表(那是第 18 篇的事),只关注光照节点本身。

## 2. Godot 心智

### 类层级

```
CanvasItem (基类)
  └── Node2D
        ├── Light2D            (抽象基类,不直接使用)
        │     ├── PointLight2D
        │     └── DirectionalLight2D
        └── LightOccluder2D    (持有 OccluderPolygon2D)

Resource
  ├── CanvasTexture            (diffuse + normal + specular 三通道)
  └── OccluderPolygon2D        (polygon + closed + cull_mode)
```

`Light2D` 是抽象基类,提供 `enabled` / `color` / `energy` / `blend_mode` / `range_z_min` / `range_z_max` / `range_layer_min` / `range_layer_max` / `range_item_cull_mask` / `shadow_enabled` / `shadow_color` / `shadow_filter` / `shadow_filter_smooth` / `shadow_item_cull_mask`。`PointLight2D` 在此之上加 `texture` / `offset` / `texture_scale` / `height`;`DirectionalLight2D` 加 `max_distance` 而抛弃位置(它只用基的旋转方向)。

`CanvasItem.light_mask` 与 `Light2D.range_item_cull_mask` 配对——前者表示"这个物体收哪些光",后者表示"这盏光打哪些物体",**两者按位与有交集,才能命中**。`LightOccluder2D.occluder_light_mask` 又是另一套独立的掩码,表示"这块遮挡只阻挡哪些光",对**不在掩码内的光源完全透明**。三层掩码是 2D 光照工程化的核心抓手,设计错一处,后期就是反复打补丁。

### 渲染顺序与数据流

`CanvasItem` 在每一帧的渲染顺序大致是:

1. 计算可见性(基于 `visibility_layer`、视口裁剪、`VisibleOnScreenNotifier2D`)。
2. 收集这一帧落在视口里的 `CanvasItem`,按 z-index、绘制顺序排序。
3. 对每个 `CanvasItem` 用其 `diffuse_texture`(或 `Sprite2D.texture`)绘制底色。
4. 收集落在视口里的 `Light2D`,按 `blend_mode` 把光纹理与底色合成。如果该 `Light2D` 开启了 `shadow_enabled`,则在合成前为每个落在其范围内的 `LightOccluder2D` 算出阴影几何,光纹理在阴影区域被屏蔽。
5. 如果绑定了 `CanvasTexture` 且包含 `normal_texture`,则在第 4 步合成时,光照按法线方向调制,得到逐像素强度变化(伪 3D 质感的来源)。

第 5 步的细节是本篇的关键:法线贴图在 Godot 里使用 **X+ Y+ Z+** 约定(Substance / Photoshop 默认导出格式),如果用 Unity / Unreal 的 Y- 反向贴图,亮暗会反转。`PointLight2D.height` 给法线一个"光源在物体上方多少像素"的虚拟高度;`height = 0` 时光向几乎平行于平面,`dot(N, L)` 不分高低,法线就废了。经验值是 `height` 设到 sprite 宽度的 0.5~1 倍。

还有一个被反复忽略的事实:**`CanvasLayer` 的子节点不参与世界光照**。`CanvasLayer` 在渲染管线里位于一个独立的 2D canvas,它有自己的变换矩阵,主要用来摆 HUD、暂停菜单、过场字幕。把光放在 `CanvasLayer` 里,光的 `texture` 仍然会画出来,但合成时不会找到下面那一层 `CanvasItem`。这条规则反过来用就很有价值:**HUD 永远不被太阳照亮**,不需要写 `light_mask`,把 HUD 全部塞进 `CanvasLayer` 就行。

### `CanvasTexture` 是什么

`CanvasTexture` 是 `Texture2D` 之外的另一条路径,专门解决"我要给 2D sprite 一份法线+高光,但不想写 shader"的需求。它有三个独立通道:

- `diffuse_texture`:底色,等同于普通 `Sprite2D.texture`。
- `normal_texture`:法线贴图,只在场景里有 `Light2D` 时生效。
- `specular_texture` + `specular_color` + `specular_shininess`:高光贴图与高光指数,控制反射强度。

`Sprite2D.texture` 接受 `Texture2D`,而 `CanvasTexture` 继承自 `Texture2D`,所以可以直接挂上去。这里有一个常被忽略的细节:`CanvasTexture` 还允许**单独覆盖 `texture_filter` 和 `texture_repeat`**,无需依赖节点属性或项目设置——做像素风时直接把 filter 改成 `TEXTURE_FILTER_NEAREST` 即可,不必再去开 `Sprite2D.texture_filter`。

像素风游戏在这里有一个常见取舍:法线贴图是亚像素级别的方向数据,而像素风通常会把所有 sprite 缩到 64×64 甚至 16×16。这种尺寸下,逐像素法线变化会被分辨率本身抹平,反而看起来不像"凹凸"而像"色彩噪点"。建议要么把整个项目按高分辨率绘制 + viewport stretch,要么完全放弃 normal map,改用 21 篇的"自绘 outline / 高光 shader"做风格化光照——两条路都行,混用会很怪。

## 3. 工程实现

下面构造一个最小但完整的场景:一个法线贴图地面、一个会跟随鼠标的火把 `PointLight2D`、一面挡光的墙体 `LightOccluder2D`、外加一个昼夜 `DirectionalLight2D`。本系列的演示工程目录是 `demo/`,以下路径都基于这个根。

### 资源:`CanvasTexture` 配置

地面 sprite 不能直接挂 PNG,要包一层 `CanvasTexture`,在 Inspector 里建好后另存为资源,以便其它地砖复用。

`demo/resources/materials/floor_canvas.tres`:

```gdscript
# floor_canvas.tres 序列化后的形式(节选,实际由 Inspector 生成):
# [gd_resource type="CanvasTexture" load_steps=3 format=3]
# [ext_resource type="Texture2D" path="res://art/floor_albedo.png" id=1]
# [ext_resource type="Texture2D" path="res://art/floor_normal.png"  id=2]
# [resource]
# diffuse_texture = ExtResource(1)
# normal_texture  = ExtResource(2)
# specular_color  = Color(1, 1, 1, 1)
# specular_shininess = 32.0
# texture_filter  = 1   # TEXTURE_FILTER_NEAREST,像素风
```

注意法线贴图必须是 X+ Y+ Z+ 坐标系导出。如果用的是 Krita / Aseprite 的法线插件,通常已是 X+ Y+;Z 通道有些工具默认 Z-,需要在导入时检查。Godot 不会替你换轴。

### 节点树

```
World (Node2D)
├── Sun (DirectionalLight2D)              # 全局光,代表太阳/月光
├── FloorChunk (Sprite2D, texture=floor_canvas.tres)
├── Wall (StaticBody2D)
│    ├── CollisionShape2D                 # 物理碰撞
│    ├── Sprite2D                         # 视觉
│    └── LightOccluder2D                  # 光遮挡(独立于物理)
│         └── (OccluderPolygon2D 资源)
└── Player (CharacterBody2D)
     └── Torch (Node2D)
          ├── PointLight2D                # 火把交互光
          └── FlickerTimer (Timer)        # 火苗抖动
```

`LightOccluder2D` 与物理碰撞**没有任何耦合**——它有自己的 `OccluderPolygon2D`。这是 Godot 的有意设计:光的遮挡形状常常和物理形状不同(比如一根栏杆物理上能挡子弹,但光照上不希望它把整面墙照黑)。需要时可以让两个 polygon 共享同一份顶点数组,但不要把碰撞 polygon 直接喂给 occluder。

### 火把脚本

`demo/scripts/lights/torch.gd`,挂在 `Torch` 节点上。它实现两件事:用鼠标位置当虚拟光源、用 `Timer` 抖动 energy 模拟火光闪烁。火苗闪烁是手感的一部分,值得专门讲。

```gdscript
extends Node2D
class_name Torch

# === 可调参数 ===
@export var follow_mouse: bool = true              # 是否跟随鼠标(调试用)
@export var base_energy: float = 1.6               # 基准强度
@export var flicker_amount: float = 0.25           # 闪烁幅度
@export var flicker_period: float = 0.08           # 闪烁周期(秒)
@export var min_height: float = 32.0               # 与法线配合的虚拟高度
@export_range(1.0, 4.0) var texture_scale: float = 1.5

@onready var _light: PointLight2D = $PointLight2D
@onready var _timer: Timer = $FlickerTimer

var _rng := RandomNumberGenerator.new()

func _ready() -> void:
    _rng.randomize()
    _light.energy = base_energy
    _light.texture_scale = texture_scale
    _light.height = min_height
    _light.shadow_enabled = true
    _light.shadow_filter = Light2D.SHADOW_FILTER_PCF13   # 4.6 推荐软阴影
    _light.shadow_filter_smooth = 1.5
    _light.range_item_cull_mask = 0b0001                 # 只照"主层"
    _timer.wait_time = flicker_period
    _timer.timeout.connect(_on_flicker)
    _timer.start()

func _process(_delta: float) -> void:
    if follow_mouse:
        global_position = get_global_mouse_position()

func _on_flicker() -> void:
    # 用噪声而不是 sin:更不规律,更像火
    var jitter := _rng.randf_range(-flicker_amount, flicker_amount)
    _light.energy = base_energy + jitter
    # 颜色微微偏移,从暖橙到金黄
    var t := _rng.randf()
    _light.color = Color(1.0, 0.78 + 0.12 * t, 0.55 + 0.10 * t)

func set_enabled(value: bool) -> void:
    _light.enabled = value
    _timer.paused = not value
```

几点不显然的工程取舍:

- `shadow_filter = SHADOW_FILTER_PCF13` 比默认的 `NONE` 多 13 次采样,但给火把这种大面积光源软阴影的效果质变。`shadow_filter_smooth` 控制边缘软化,1~2 之间是手感最好的范围,过大会让阴影"漂"。
- 不要在 `_process` 里改 `_light.color`,改 `color` 会把这盏光从 batch 里踢出去。`_on_flicker` 节流到 80ms 一次,既能感知到闪烁,又不破坏批处理。
- `range_item_cull_mask` 用位掩码而不是单值。`0b0001` 是主世界层,后面 HUD / 粒子可以分别用 `0b0010` / `0b0100`,这样 HUD 永远不被火把照亮。

### 昼夜全局光

`demo/scripts/lights/sun.gd`,挂在 `DirectionalLight2D`。它和 PointLight2D 的区别是:**位置没意义,旋转是唯一参数**,光沿节点的局部 +Y 方向投射(未旋转时是正下方)。

```gdscript
extends DirectionalLight2D
class_name SunController

@export var day_length_seconds: float = 120.0      # 一个完整昼夜的秒数
@export var noon_energy: float = 1.0
@export var night_energy: float = 0.05
@export var noon_color: Color = Color(1.0, 0.96, 0.88)
@export var night_color: Color = Color(0.35, 0.45, 0.65)

var _elapsed: float = 0.0

func _ready() -> void:
    shadow_enabled = true
    shadow_filter = Light2D.SHADOW_FILTER_PCF13

func _process(delta: float) -> void:
    _elapsed = fmod(_elapsed + delta, day_length_seconds)
    var t: float = _elapsed / day_length_seconds                 # 0..1
    # 太阳轨迹:rotation 在 ±π/3 之间扫动,代表早晨到傍晚
    rotation = lerp(-PI / 3.0, PI / 3.0, t)
    # 用 sin 拟合昼夜亮度,sin(πt) 在 t=0/1 时为 0,在 t=0.5 时为 1
    var brightness: float = sin(PI * t)
    energy = lerp(night_energy, noon_energy, brightness)
    color = night_color.lerp(noon_color, brightness)
```

`DirectionalLight2D` 不支持 `range_item_cull_mask`(它会强制照亮所有 2D 节点,只支持 `shadow_item_cull_mask`)。这是文档里明确写的约束,不是 bug。如果想让 UI 不被太阳照亮,只能把 UI 放到独立 `CanvasLayer`(那一层不参与 World 的光照合成)。

### 性能监控

`demo/scripts/lights/light_budget.gd`,挂在场景根,按 F3 打印当前光源、阴影、occluder 数量以及 draw call。

```gdscript
extends Node

@export var hotkey: String = "debug_light_budget"   # InputMap 里建一个 F3 映射

func _unhandled_input(event: InputEvent) -> void:
    if event.is_action_pressed(hotkey):
        _report()

func _report() -> void:
    var lights: int = 0
    var shadow_lights: int = 0
    var occluders: int = 0
    for node in get_tree().get_nodes_in_group("&lights"):
        lights += 1
        if (node as Light2D).shadow_enabled:
            shadow_lights += 1
    for node in get_tree().get_nodes_in_group("&occluders"):
        occluders += 1
    var draw_calls: int = Performance.get_monitor(
        Performance.RENDER_TOTAL_DRAW_CALLS_IN_FRAME)
    var objects: int = Performance.get_monitor(
        Performance.RENDER_TOTAL_OBJECTS_IN_FRAME)
    push_warning("[LightBudget] lights=%d shadows=%d occluders=%d draws=%d objs=%d" %
        [lights, shadow_lights, occluders, draw_calls, objects])
```

`Performance.RENDER_TOTAL_DRAW_CALLS_IN_FRAME` 是 4.x 暴露的全局监视器,比开编辑器 Monitor 面板更适合放在游戏运行中按需采样。把所有 `Light2D` 和 `LightOccluder2D` 放进 `&lights` / `&occluders` 组,只需场景里 `node.add_to_group("&lights")`。

## 4. 调参和验收

### 关键参数与手感映射

| 参数 | 默认 | 建议范围 | 影响 |
| --- | --- | --- | --- |
| `PointLight2D.energy` | 1.0 | 0.5–2.5 | 直接影响亮度,>2.5 会过曝压色 |
| `PointLight2D.texture_scale` | 1.0 | 0.8–3.0 | 决定光斑覆盖的像素数,每翻一倍,光照像素数 ×4 |
| `PointLight2D.height` | 0 | sprite 宽度 ×0.5~1.0 | 法线起效的必要条件;0 时法线无效 |
| `Light2D.shadow_filter` | `NONE` | `PCF5` / `PCF13` | `PCF13` 比 `PCF5` 更平滑,GPU 成本 ~2.6× |
| `Light2D.shadow_filter_smooth` | 0 | 1.0–2.0 | 软阴影边缘宽度,>3 会让阴影"漂"出物体 |
| `Light2D.blend_mode` | `MODE_ADD` | 视用途 | `MIX` 适合做有色透镜,`SUB` 适合做"魔法暗化"区域 |
| `Sun.day_length_seconds` | 120 | 60–600 | 调玩家对昼夜的感知节奏 |

### 验收:伪 3D 质感

在 `Player` 周围放一个法线贴图地面、挂上 `Torch`,完成后应能观察到:

1. 火把围绕地面移动时,地面纹理的凹凸方向随光向逐像素变化——光在凸起的一侧亮,凹陷一侧暗。
2. 把 `Torch` 的 `min_height` 改成 0,法线效果立刻消失,地面只剩均匀亮度。这是"`height` 为什么必须 > 0"的最直接验证。
3. 同样位置切到 `Sun.rotation = -PI/3`(早晨),全场阴影长且偏向一侧;`rotation = 0`(正午),阴影最短;`rotation = PI/3`(傍晚)对称偏向另一侧。
4. `Torch.color` 变化时,墙体在火把侧偏暖橙,远处地面在 `Sun` 主导的冷色调里——两盏光按 `MODE_ADD` 叠加,颜色物理上不正确(没有色温合成),但视觉上立刻有"近暖远冷"的层次感。

### 验收:性能预算

按下 F3 打印 `[LightBudget]`,记录三组数据:

- **基线**:0 火把 + 1 太阳(开 shadow) + 30 个 occluder + 1 个角色,draw call 应在 ~10–20 之间。
- **中负载**:加到 16 个火把(全部 `shadow_enabled = true`)+ 1 太阳 + 30 occluder,draw call 跳到 ~150 以上是正常的,FPS 仍应稳定 60(中端机)。
- **过载阈值**:加到 32 个开阴影的火把后,如果 FPS 跌破 30,就该重新设计:把远处火把改成"假光"(只放纹理,不挂 `PointLight2D`),或者把 `shadow_enabled` 关掉只保留 energy + texture。

经验法则:**屏幕上同时存在的"开启阴影且会移动"的 `Light2D` 不要超过 8 盏**。静态光源(`enabled = true` 但 `position` 不变)代价低得多,GPU 可以缓存大部分计算。

### 验收:法线方向

最容易忽略的验收项:在 `Wall.Sprite2D` 的 `texture` 字段挂上 `CanvasTexture`,只放 `diffuse_texture`,不放 `normal_texture`。围着它移动火把,亮度均匀;然后挂上 `normal_texture`,围着移动,墙面应能"折"出立体感。如果亮暗反了,说明法线 Y 轴反向,需要在导入面板把法线 `Detect 3D`/`Normal Map` 模式启用,或在外部工具里反 Y 通道重新导出。

### `light_mask` 也是玩法机制

一个常被低估的设计:`light_mask` 不仅是性能开关,也是玩法变量。例如做一款"潜行向"游戏,玩家进入"阴影区"时不被敌人发现——可以让玩家挂一个隐藏的 `Area2D` 检测当前位置是否在任何 `Light2D` 的范围内,以此驱动 AI 检测。再例如"魔法蜡烛"道具,激活后给玩家身上挂一个 `PointLight2D`,其 `range_item_cull_mask` 包含一个特殊位 `0b1000_0000`,而场景里隐形的"魔法符文"`Sprite2D` 把 `light_mask` 设成 `0b1000_0000`——符文平时不可见(底色为黑),只有蜡烛光命中时才显现。这是用光照系统替代复杂状态机的一种典型做法,后续 17 篇会进一步把这种"组件+掩码"扩展到属性效果上。

## 5. 踩坑

### 法线贴图导入,Compress Mode 别用 VRAM

`CanvasTexture.normal_texture` 默认按普通 `Texture2D` 处理,而 Godot 4 的纹理导入面板会把 PNG 默认压缩成 VRAM 格式(BPTC / S3TC / ETC2)。VRAM 压缩对**漫反射图是优化,对法线图是灾难**——它会把 (x, y, z) 三通道当颜色压,丢精度。打开 normal_texture 的导入面板,把 `Compress > Mode` 改成 `Lossless` 或 `Disabled`,否则你会在低端机上看到法线"色块化"。

### `LightOccluder2D.occluder_light_mask` 和 `Light2D.shadow_item_cull_mask` 不是一个东西

这两个掩码长得几乎一样,容易混淆:

- `LightOccluder2D.occluder_light_mask`:这块 occluder 阻挡哪些光?写在 occluder 节点上。
- `Light2D.shadow_item_cull_mask`:这盏光的阴影由哪些 occluder 投射?写在 light 节点上。

两者必须**按位有交集**,阴影才会出现。如果 occluder 用 layer 1、light 阴影掩码也包含 layer 1,但你把 light 的 `range_item_cull_mask` 改成了 layer 2,光本身不照那个区域,自然也不会有阴影——这时人很容易误以为 occluder 没生效。三个掩码记牢:**光本体走 `range_item_cull_mask`,光的阴影走 `shadow_item_cull_mask`,遮挡走 `occluder_light_mask`**。

### `DirectionalLight2D` 没有位置概念,但有 `max_distance`

`DirectionalLight2D` 不读 `position`,只读 `rotation`,所以"把太阳拖到天上"是无效操作。但它有一个 `max_distance` 属性,表示**从节点位置开始,沿 +Y 方向延伸多远**才计算光照;超出这个距离不打光。这个参数在大世界里是优化:不需要照亮的远处直接被剪掉。文档里写得不显眼,需要刻意配置。

### `CanvasItem.light_mask` 默认全开

每个 `CanvasItem` 的 `light_mask` 默认值是 `1`(只开第 1 位)。新建的 `Light2D.range_item_cull_mask` 默认也是 `1`,所以"开箱即用"是没问题的。但一旦你把 `light_mask` 改成 `0`(想"不被任何光照亮"),它会安静地继续被照亮——`0` 不是"全关",而是"匹配 0"(永远不命中)。如果你想让 HUD 不被照,正确做法是把 HUD 放进 `CanvasLayer`,或者给 HUD 用一个**独立的 bit**(比如 `0b1000_0000_0000_0000`),然后**确保所有 Light2D 都不开这个 bit**。

### `Z Min / Max` 是按整数,不是按节点路径

`Light2D.range_z_min` / `range_z_max` 控制光照影响哪些 z-index 的 `CanvasItem`,默认 `-1024 ~ 1024`。如果你给某个角色设了 `z_index = 5`,而把火把的 `range_z_max = 0`,那个角色就不会被照亮——这种"角色突然黑掉"的 bug 经常发生在加了"道具浮起 z_index +10"之类的临时逻辑后。**z-index 是光照可见性的隐式过滤器**,要么全场不动它,要么在文档里写清楚每一层的范围。

### 改 `Light2D.position` 每帧都会打断批处理

Godot 4.x 在渲染层做了 2D batching:相同纹理、相同材质、相同光照状态的 `CanvasItem` 会被合并到一个 draw call。一旦某个 `Light2D` 的 `position` / `rotation` / `energy` / `color` 每帧都变,它影响范围内的 `CanvasItem` 在 batch 时就要重新计算光照合成,batch 边界不停被拆开。

判别方法:在 F3 报告里,固定光源的场景里如果你把 5 盏静态光改成跟随玩家移动,`draws` 数字翻倍,几乎一定是 batching 失效了。修正方式:**移动的光不要超过 1–2 盏**,需要"很多动光"的场景(比如萤火虫群)用粒子(20 篇)而不是真光。

### "假光":只放纹理不挂 Light2D

远处场景需要"看上去有光"但不需要真照亮——比如远处城市灯火、悬挂的灯笼。这类用一张光晕 PNG 加 `Sprite2D` + `modulate` 即可,不要挂 `PointLight2D`。它们没有光照合成,没有阴影计算,draw call 与普通 sprite 一致,但视觉上玩家分不出来。把"真光"留给玩家身边 5–8 盏关键光源,其余全部假光,是 2D 光照场景能跑到 60 帧的最直接秘诀。

### 阴影 `shadow_filter_smooth` 不是越大越好

`SHADOW_FILTER_PCF13` 配 `shadow_filter_smooth = 5`,阴影边缘看起来很软,但会出现"阴影从物体上漂出来"——遮挡物的边缘外几个像素仍然有半透明阴影。物理上这没意义,玩家会感觉"墙在投出虚影"。建议把 `smooth` 控制在 1–2,需要更软的视觉,改用更大的 texture 而不是更大的 smooth。

### `LightOccluder2D` 的 `closed` 属性决定单面/双面

`OccluderPolygon2D.closed = true`(默认)时,polygon 形成一个封闭区域,光从任意方向到达都会被遮挡;`closed = false` 时只在 polygon 的**法线一侧**遮挡——这是给单面墙体(只挡光不挡视线)做的优化。当你发现"墙体只挡了一个方向的光",几乎一定是 `closed` 被关掉了。

### `tile_map_data` 与 occluder 协作

`TileMapLayer`(参见 11 篇)里的瓦片可以在 `TileSet` 编辑器里给每个瓦片烘焙 occlusion polygon,这样一整层墙体不需要手动放 `LightOccluder2D`。但**烘焙后的 occlusion 与 `physics_layer` 是独立的两套数据**,改物理形状不会同步光遮挡。这是工程上"为什么 occluder 和物理分离"在数据层的延续——记得改 occlusion 时同步两边,或写一个 `@tool` 脚本统一生成(28 篇会展开 `@tool`)。
