# 2D 渲染性能、批处理与资源预算

2D 游戏也会卡。卡顿不一定来自复杂逻辑,很多时候来自 draw call、纹理切换、材质切换、节点太多、屏外对象还在跑。

> 一句话先记住:**先测,再优化;先定预算,再加资源。**

---

## 一、先看 draw call

加一个调试 HUD:

```gdscript
# res://debug/perf_label.gd
extends Label

func _process(_delta: float) -> void:
    var draw_calls := Performance.get_monitor(Performance.RENDER_TOTAL_DRAW_CALLS_IN_FRAME)
    var nodes := Performance.get_monitor(Performance.OBJECT_NODE_COUNT)
    var fps := Performance.get_monitor(Performance.TIME_FPS)
    text = "FPS %d  Draw %d  Nodes %d" % [fps, draw_calls, nodes]
```

不要只看 FPS。FPS 掉下来时已经晚了。draw call 和节点数会更早暴露问题。

---

## 二、什么会打断批处理

Godot 会尽量把相邻的 2D 绘制合成一批。但这些会让 batch 断开:

```text
换纹理
换 material
换 shader
换 blend mode
跨 CanvasLayer
跨 z_index
复杂 y_sort
```

优化方向:

```text
用图集减少纹理切换
共享 ShaderMaterial
少用 z_index
只在需要的容器开 y_sort
CanvasLayer 数量控制住
```

---

## 三、图集比散图更重要

一堆独立 PNG:

```text
enemy_01.png
enemy_02.png
item_01.png
item_02.png
```

会频繁切纹理。更好的做法是打进 atlas,然后用 `AtlasTexture` 或导入工具切区块。

TileMapLayer 本身就很适合批处理,前提是 TileSet 设计别太碎。

---

## 四、Material 要共享

反例:

```gdscript
sprite.material = base_material.duplicate()
```

如果每个敌人都复制一份 material,它们看起来用同一个 shader,但渲染层可能没法合批。

能共享就共享:

```gdscript
sprite.material = preload("res://shaders/flash_material.tres")
```

需要单个对象不同参数时,优先考虑:

```gdscript
set_instance_shader_parameter()
```

而不是复制几十份 material。

---

## 五、屏外对象别继续跑

给远处敌人加:

```text
VisibleOnScreenEnabler2D
```

或者手动:

```gdscript
@onready var _notifier: VisibleOnScreenNotifier2D = %VisibleOnScreenNotifier2D

func _ready() -> void:
    _notifier.screen_entered.connect(_set_active.bind(true))
    _notifier.screen_exited.connect(_set_active.bind(false))

func _set_active(active: bool) -> void:
    set_process(active)
    set_physics_process(active)
```

注意:离屏后仍要跑的东西不要关,比如计时器、全局生成器、远程逻辑。只关视觉和近场 AI。

---

## 六、资源预算

给一个独立 2D 项目的起步预算:

```text
目标: 1080p / 60fps
屏幕内 draw call: 100-300
节点数: 3000 以下更舒服
屏幕内动态敌人: 20-50
同屏粒子节点: 20 个以内,粒子数看平台
Texture atlas: 单张 2048 或 4096,按平台控制
CanvasLayer: 5-10 个以内
```

预算不是法律,是报警线。超过了就要知道为什么。

---

## 七、Y-sort 小心用

`y_sort_enabled` 很方便,但会重排子节点绘制顺序。重排会破坏原本连续的批处理。

推荐:

```text
StaticTiles        不开 y_sort
WorldActors        开 y_sort,只放角色和可移动物
ForegroundDecor    不开或局部开
HUD                CanvasLayer
```

不要给整张地图根节点一把开 y-sort。

---

## 八、验收

- 项目有调试性能 HUD。
- 能看到 draw call、FPS、节点数。
- 大量 sprite 尽量来自 atlas。
- 同类对象共享 material。
- 屏外对象会停掉不必要 process。
- 有明确目标硬件和资源预算。

---

## 常见坑

**坑 1:不测就优化。**

先确认瓶颈。别凭感觉改。

**坑 2:复制 material 做闪白。**

少量可以,大量对象考虑 instance shader parameter。

**坑 3:节点泄漏。**

粒子、音效、掉落物播完要 `queue_free()`。

**坑 4:只盯 GPU。**

2D 项目 CPU 节点遍历也会很贵。节点数也要看。
