# 23-2D 渲染性能-批处理与资源预算

> 一句话导读:2D 性能瓶颈八成在 draw call 与状态切换,剩下两成在节点数与可见性裁剪;不测就调,是用直觉对抗系统行为。

很多人对"2D 性能"有一个错觉:2D 几何那么简单,GPU 肯定吃得下,卡顿一定是 CPU 逻辑写错了。这个判断在 30 个 sprite 的小 demo 里大概率成立,但在一个有 5000 个瓷砖、200 个特效、20 个有动画的 sprite 的中型独立游戏里就完全错。2D 的真正成本不在像素数量,而在"每帧发了多少次 draw call、每个 draw call 之间切了多少次纹理 / 材质 / shader 状态"。这一篇讲清 Godot 4.6 下 2D 渲染的成本模型、测量手段、以及一份能直接拿去对照的资源预算清单。

性能这件事在工程上有两条颠扑不破的原则:**不测就调等于盲调;不立预算就调等于无止境调**。前者保证你判断方向正确,后者保证你知道何时该停手。本文绕着这两条原则展开,中间穿插具体的 Godot 4.6 API 与项目设置。

## 1. 机制定位

GPU 的工作模式是"流水线":CPU 准备一批顶点和状态,然后告诉 GPU "画这一批",GPU 内部 SIMD 并行执行。每次 CPU 告诉 GPU 画一批,就是一次 draw call。draw call 的成本主要在 CPU 这一侧——构造命令、提交到驱动、等待 GPU 接收。一次 draw call 在桌面端几十微秒,在移动端可能几百微秒。

2D 游戏的尴尬在于:每一个独立的 sprite、tile、特效、UI 元素都有"被单独画"的潜质。如果引擎不做任何合并,1000 个 sprite 就是 1000 次 draw call。在桌面端这还能扛,在移动端立刻掉到 20 fps。所以现代 2D 引擎都做 **batching**——把"用同一张纹理、同一种材质、同一个 shader"的连续元素合并成一次 draw call。Godot 4.x 的 2D 渲染也是这套思路,内部由 `rendering/2d/batching/*` 一系列项目设置控制。

新手最常见的几种"性能崩盘"模式:

第一,**每个 sprite 一张独立纹理**。美术给你 200 张 256×256 的 PNG,每张做成 `Sprite2D`,看起来没什么问题。但在 GPU 看来,每画一个新 sprite 都要"切纹理"(unbind 旧的、bind 新的),batching 直接断开。结果就是 200 次 draw call,移动端崩盘。正确做法是用 **texture atlas**——把 200 张图打包成 1-4 张大图,每个 sprite 引用大图的一块区域。这一招做好,200 次 draw call 能压到 10 次以内。

第二,**滥用 material**。每个 sprite 都挂一个独立的 `ShaderMaterial`,以为"反正大家都是同一个 shader,引擎会优化"。引擎确实会尽量合并,但 material 实例本身就是 batching 的边界——两个相邻 sprite 的 material 实例不同,引擎必须断开 batch。结果就是 200 个挂着同一个 shader 但不同 material 实例的 sprite 也跑出 200 次 draw call。

第三,**离屏物体也参与渲染**。500 个敌人散布在 100 屏宽的世界里,屏幕上其实只有 10 个,但剩下 490 个都在跑 `_process`、播动画、参与碰撞——CPU 侧的浪费比 GPU 侧还严重。可见性裁剪(culling)是这个问题的解药,Godot 提供 `VisibleOnScreenNotifier2D` 和 `VisibleOnScreenEnabler2D` 两个节点专门做这件事。

第四,**没有预算意识**。开发到一半才发现"哎呀 Steam Deck 跑不动",然后回头删美术、删特效,极其痛苦。资源预算应当在项目启动时就定下:目标硬件、目标分辨率、目标帧率、能接受多少 draw call / 节点数 / 显存占用。本文最后会给一份独立 2D 游戏的对照清单。

第五,**只盯着 fps 看,不看节点数和显存**。fps 是个滞后指标——当 fps 掉下来你才发现问题,但此时往往离根因已经差了一两周。更早的告警来自节点数(`OBJECT_NODE_COUNT`)和显存(`MEMORY_STATIC`):节点数快速上涨说明你忘了 `queue_free`,显存上涨说明纹理在被反复加载或泄漏。fps 是后果,节点数和显存是前兆。

工程目标可以总结成三句话:**让 batch 大、让 material 少、让屏外的别参与**。围绕这三句话展开。剩下的优化(shader 复杂度、overdraw、阴影、光照)都是在这三件事之后才该考虑的事——本末倒置去优化 shader 但不收拾 draw call,提升微乎其微。

## 2. Godot 心智

Godot 4.6 的 2D 渲染栈相关核心类:

| 类 / 资源 | 角色 |
| --- | --- |
| `CanvasItem` | 所有 2D 节点的基类,负责 `_draw()`、变换、可见性、`y_sort_enabled` |
| `Node2D` | 继承 `CanvasItem`,加上位置/旋转/缩放变换 |
| `Sprite2D` | 单张纹理 sprite,最常用 |
| `AnimatedSprite2D` | 用 `SpriteFrames` 资源驱动的帧动画 |
| `MultiMeshInstance2D` | 单次 draw call 渲染几万到几十万个相同形状,适合粒子、瓷砖、可破坏地形 |
| `TileMapLayer` | 4.3+ 唯一推荐的 Tile 渲染节点,内部本身做 batching |
| `AtlasTexture` | 引用大纹理一块区域的轻量纹理资源 |
| `VisibleOnScreenNotifier2D` | 发出 `screen_entered` / `screen_exited` 信号 |
| `VisibleOnScreenEnabler2D` | 自动启用 / 禁用关联节点的 `_process`、物理 |
| `CanvasLayer` | 用 layer index 分层,层之间不混排 |

测量入口有三个,层级递进:

**最快**:`Performance.get_monitor(Performance.RENDER_TOTAL_DRAW_CALLS_IN_FRAME)`。一行代码拿到全帧 draw call 总数,所有视口加起来。适合放进 HUD 上做实时显示。

**按视口**:`get_viewport().get_render_info(Viewport.RENDER_INFO_TYPE_VISIBLE, Viewport.RENDER_INFO_DRAW_CALLS_IN_FRAME)`。区分主视口和子视口,可以分别看 UI 视口和游戏视口的成本。如果你的 HUD 跑在另一个 SubViewport 里,这是定位"是 UI 太复杂还是关卡太复杂"的标准方法。

**最精确**:`RenderingServer.viewport_get_render_info(rid, render_info_type, render_info)`。直接对 RID 查询,可以拿"可见 vs 阴影 vs 画布"分类的数据。文档里明确写了"前两帧数据未就绪",查询前要 `await get_tree().process_frame` 两次。

**心智上,要分两层理解 batching 边界**:

- **同 batch 条件(可合并)**:同一个父 `CanvasItem` 链下、同一张纹理(或同一个 atlas)、同一个 material 实例、同一个 z-index、同一个 blend mode。
- **batch 中断条件(必须切)**:换纹理(不同 PNG 或不同 atlas)、换 material 实例、换 shader、换 blend mode、跨 CanvasLayer、跨 z-index、被 light occluder 截断。

Y-sort 是一个隐形的 batch 杀手。`y_sort_enabled = true` 后,引擎按 Y 坐标重排子节点。如果重排打乱了"相邻同纹理"的连续性,batch 就断成多段。一个常见错误是给整张地图开 Y-sort,然后角色和瓷砖混着画,瓷砖 batching 直接报废。正确做法是只给"需要 Y-sort 的容器"开 Y-sort(角色、可移动物体所在的节点),静态背景另起一个不开 Y-sort 的兄弟节点。

`CanvasLayer` 是 Godot 2D 渲染的"硬隔离"。不同 CanvasLayer 之间永远不会 batch,层数越多 draw call 越多。但 CanvasLayer 也是非常方便的工具——HUD、暂停菜单、过场字幕、屏幕特效都该放在独立的 CanvasLayer。工程上的取舍是:CanvasLayer 数量控制在 5-10 个以内,每个 CanvasLayer 内部尽量集中 batching。

**z-index 是另一个隐形成本**。给 sprite 设 `z_index = 1` 会把它从默认 z 层抽离,放到独立的 z 层渲染。每多一个 z 层,至少多一次 draw call。z-index 只该在确实需要"超越场景树顺序"时用,日常排序优先靠场景树和 Y-sort。

**material 与 instance shader parameter 的区别**很多人忽略:`ShaderMaterial` 是一份独立的 material 实例,改它的参数会影响所有引用该实例的 sprite,但实例本身是 batching 友好的——只要共享同一份 material 实例就行。`set_instance_shader_parameter` 是 4.x 引入的"实例级参数",允许同一个 material 实例下,不同 CanvasItem 有不同的 uniform 值,而依然保持 batching。需要"每个怪血量条颜色不同但都是同一个 shader"时,instance parameter 比 material 拷贝几十份强百倍。

## 3. 工程实现

下面三段代码:一个实时 HUD 显示性能数据,一个用 `AtlasTexture` 切大图,一个用 `VisibleOnScreenEnabler2D` 自动裁剪。这三件工具基本覆盖 2D 性能工程的 80% 日常需求,剩下 20% 是 shader、粒子、物理调优,会在 20、21 篇展开。

第一段,实时性能 HUD。挂在 `CanvasLayer` 下,左上角显示 FPS、draw call、节点数。这是项目第一周就该放进去的工具——开发过程中一直能看到数字,出问题立刻发现。建议默认就显示,debug build 全开,release build 用 `OS.is_debug_build()` 控制是否显示,或者绑一个调试快捷键 F3 切换。

文件:`demo/scripts/debug/performance_hud.gd`

```gdscript
class_name PerformanceHUD
extends Label

## 实时显示帧率、draw call、节点数,挂在 CanvasLayer 下。
## 在 release 构建里 RENDER_* 监视器可能返回 0,UI 仍然安全显示。

@export var update_interval: float = 0.25  # 4 Hz 更新,避免每帧字符串拼接
var _accum: float = 0.0

func _ready() -> void:
    add_theme_color_override("font_color", Color.LIME_GREEN)
    add_theme_color_override("font_outline_color", Color.BLACK)
    add_theme_constant_override("outline_size", 4)
    text = "..."

func _process(delta: float) -> void:
    _accum += delta
    if _accum < update_interval:
        return
    _accum = 0.0
    var fps: float = Performance.get_monitor(Performance.TIME_FPS)
    var draw_calls: float = Performance.get_monitor(
        Performance.RENDER_TOTAL_DRAW_CALLS_IN_FRAME
    )
    var items_2d: float = Performance.get_monitor(
        Performance.RENDER_2D_ITEMS_IN_FRAME
    )
    var nodes: float = Performance.get_monitor(Performance.OBJECT_NODE_COUNT)
    var mem_mb: float = Performance.get_monitor(Performance.MEMORY_STATIC) / 1048576.0
    text = "FPS  %4d\nDC   %4d\n2D   %4d\nNode %4d\nMem  %.1f MB" % [
        int(fps), int(draw_calls), int(items_2d), int(nodes), mem_mb
    ]
```

`RENDER_2D_ITEMS_IN_FRAME` 是 Godot 4 暴露的 2D canvas item 数量,反映"引擎要处理多少个 2D 物体"。它跟 draw call 的比值能告诉你 batching 的效率:如果 items_2d = 1000、draw_calls = 50,平均一个 batch 装 20 个 item,batching 工作良好;如果 draw_calls = 800,几乎一个 item 一个 batch,要查原因。

注意 release 构建里部分 `RENDER_*` 监视器的值为 0,因为引擎默认不在 release 模式下采集这些数据。打 release 包测性能时不要指望这个 HUD,而要看实际帧率。如果发布版需要内置性能监控(比如长期收集玩家硬件的 fps 分布),可以自己接 `Time.get_ticks_usec()` 计每帧耗时,这是 release 构建里也精确的指标。

第二段,用 `AtlasTexture` 做出"一张大图切多份 sprite"。AtlasTexture 是一个 `Texture2D` 子类,它持有 `atlas` 字段(指向真正的大纹理)和 `region` 字段(在大图里的矩形区域)。设计师通常会用 TexturePacker 之类的工具把散图打包成大图,程序这一端要做的就是给每个 sprite 配一份 AtlasTexture 资源。

文件:`demo/scripts/render/atlas_sprite_factory.gd`

```gdscript
class_name AtlasSpriteFactory
extends RefCounted

## 从一张大图按帧索引切出 Sprite2D。
## 适合统一规格的角色/敌人 sprite,如 32x32 的瓷砖。

const FRAME_SIZE: Vector2i = Vector2i(32, 32)

var _atlas: Texture2D
var _columns: int

func _init(atlas_path: String) -> void:
    _atlas = load(atlas_path)
    assert(_atlas != null, "atlas not found: " + atlas_path)
    _columns = _atlas.get_width() / FRAME_SIZE.x

## 由帧索引返回一个 AtlasTexture(可以被多个 Sprite2D 共享)。
## 注意:返回值要存起来,不能每帧重新构造,否则会触发 batching 中断。
func make_region(frame_index: int) -> AtlasTexture:
    var col: int = frame_index % _columns
    var row: int = frame_index / _columns
    var region := AtlasTexture.new()
    region.atlas = _atlas  # 关键:所有 region 共享同一张 atlas
    region.region = Rect2(
        col * FRAME_SIZE.x,
        row * FRAME_SIZE.y,
        FRAME_SIZE.x,
        FRAME_SIZE.y
    )
    return region

## 构造一个挂上 region 的 Sprite2D,便于场景中即时生成。
func make_sprite(frame_index: int, position: Vector2) -> Sprite2D:
    var sprite := Sprite2D.new()
    sprite.texture = make_region(frame_index)
    sprite.position = position
    return sprite
```

工程上的关键细节:**所有 `AtlasTexture.atlas` 指向同一张 `Texture2D` 实例**。如果你不小心给每个 region 各 load 一遍同样路径的 PNG,Godot 4 会用 `ResourceLoader` 的缓存返回同一份(默认 cache mode),但保险起见还是显式持有 `_atlas` 引用。两个 sprite 的 `texture.atlas` 是同一个对象时,batching 才能成立。

另一个常被忽略的细节是 atlas 的 padding。如果两个 region 紧贴(像素级相邻),GPU 在 linear filter 模式下采样到边界时会读取相邻 region 的像素,产生"溢色"现象。解决方法是在打包 atlas 时给每个 region 周围留 2-4 像素 padding(用边缘像素延展或透明填充)。像素风游戏开 nearest filter 时这个问题不存在。

第三段,可见性裁剪。给每个敌人挂一个 `VisibleOnScreenEnabler2D` 节点,屏外自动关掉 `_process` 和物理。配合 `Node.physics_interpolation_mode` 还能在 24 篇讲的 Physics Interpolation 下平滑。

文件:`demo/scripts/render/enemy_with_culling.tscn` 对应的脚本

```gdscript
class_name EnemyWithCulling
extends CharacterBody2D

@export var move_speed: float = 80.0
@onready var _enabler: VisibleOnScreenEnabler2D = $VisibleOnScreenEnabler2D
@onready var _sprite: Sprite2D = $Sprite2D

var _target_dir: Vector2 = Vector2.RIGHT

func _ready() -> void:
    # 在屏外时关闭所有耗时回调,在屏内时恢复
    _enabler.enable_node_path = ^"."
    _enabler.enable_mode = VisibleOnScreenEnabler2D.ENABLE_MODE_INHERIT
    _enabler.screen_entered.connect(_on_screen_entered)
    _enabler.screen_exited.connect(_on_screen_exited)

func _physics_process(delta: float) -> void:
    velocity = _target_dir * move_speed
    move_and_slide()

func _on_screen_entered() -> void:
    set_physics_process(true)
    _sprite.visible = true

func _on_screen_exited() -> void:
    # 屏外时停止移动逻辑,但保留 transform 以便后续重入
    set_physics_process(false)
    _sprite.visible = false
```

`VisibleOnScreenEnabler2D` 比手动监听 `screen_entered` 更省事:它能直接关掉指定节点的 `_process` / `_physics_process`。但要注意 enable 路径必须指向自身或父节点,不能指向兄弟节点。`enable_mode` 决定屏外行为:`ENABLE_MODE_INHERIT` 跟随场景 pause 状态,`ENABLE_MODE_ALWAYS` 强制运行,`ENABLE_MODE_WHEN_PAUSED` 反过来。

`VisibleOnScreenNotifier2D` / `VisibleOnScreenEnabler2D` 的边界框默认很小,要根据实际目标 sprite 大小调整。如果框太小,sprite 没出屏 notifier 就 exit 了,逻辑停了画面还在,看着像 bug;如果框太大,屏外很远的物体也参与计算,失去优化意义。一个保守的做法是把 rect 设成 sprite 边界框的 1.5 倍大,留有缓冲。

更激进的方案是配合 spawning manager,屏外距离超过某阈值时直接 `queue_free`,玩家走回来再 `instantiate`。这适合 roguelike "走过的房间不复现"风格,不适合"敌人有持久状态"的设计。中间路线是"两级裁剪":近屏外用 `VisibleOnScreenEnabler2D` 停止逻辑但保留节点,远屏外把节点序列化进 dictionary 然后 `queue_free`,需要时再反序列化生成。这套思路在大世界游戏里几乎必用。

## 4. 调参和验收

**资源预算清单**(以"目标:Steam Deck @ 60fps,1280×800,中等画质"为基线,数字根据品类微调):

| 维度 | 推荐上限 | 备注 |
| --- | --- | --- |
| draw call / 帧 | 200-400 | 移动端再降一半 |
| 2D items / 帧 | 1500-3000 | TileMapLayer 一层算 1 个 item |
| 节点总数 | 5000-10000 | 不含 TileMap 内部 tiles |
| 单帧 `_process` 总耗时 | < 6 ms | 留 10 ms 给物理 + 渲染 |
| 显存纹理总占用 | < 512 MB | atlas 后压缩到 100-200 MB 是健康区间 |
| 单张 atlas 尺寸 | 2048×2048 或 4096×4096 | 不超过目标硬件的 max texture size |
| 同时活跃粒子数 | < 5000 | 用 `GPUParticles2D` 配 amount 上限 |
| 音频同时播放声道 | < 32 | bus 总线管理见 20 篇 |

这套数字是"中型独立 2D 游戏"的健康基线。横版动作类、bullet hell 类会更高,纯叙事类会更低。重点不是数字本身,而是"在项目早期就有数,后期对照检查"。

预算的一个常见误区是"上限定得太宽松,实际上从来没逼近过"。这样的预算没有意义——它没起到"卡红线"的作用。健康的做法是:预算定在"目标硬件下还有 30-40% 余量"的位置,正常运行时数字稳定在预算的 60-70%,只有极端场景才接近预算上限。这样上线后玩家在 mod、特效拉满等额外负载下仍有缓冲。

另一个角度:预算应该按"场景"细分,而不是"全局"。主菜单预算可以很低(50 draw call),战斗场景预算高(300 draw call),Boss 战允许临时冲到 500。整体只有一个数字,等于把所有场景按最坏情况设计,会过度优化简单场景。游戏后期常做的"动态画质"——根据当前 fps 自动开关阴影、粒子、视差——本质就是"运行时预算调度",更进一步的工程化。

**项目设置侧的关键开关**:

| 项目设置 | 推荐值 | 说明 |
| --- | --- | --- |
| `rendering/2d/batching/item_buffer_size` | 16384 (默认) | 单次 batch 可容纳的命令数,大世界可适度调高 |
| `rendering/2d/batching/buffer_size_kb` | 默认值 | 顶点 buffer,通常不动 |
| `rendering/textures/canvas_textures/default_texture_filter` | 像素风用 `Nearest`,描边风用 `Linear` | 全局默认 |
| `rendering/limits/rendering/max_renderable_elements` | 默认值 | 渲染队列上限 |
| `rendering/renderer/rendering_method` | 2D 项目首选 `Compatibility` 或 `Mobile` | 决定整个渲染管线 |
| `rendering/textures/vram_compression/import_etc2_astc` | 移动平台开 ETC2 / ASTC 压缩 | 显存压缩省一半 |

`rendering_method` 是新建项目时就要做的决定,后期切换可能要重新调材质和光照,代价不小。纯 2D 独立游戏推荐 `Compatibility`:它走 OpenGL ES 3.0 / WebGL 2,兼容性极广,Steam Deck、低端笔记本、Web 都能跑;`Mobile` 是为移动端优化的简化 Vulkan/Metal 管线,2D 性能也很好但 Web 不支持;`Forward+` 主要给 3D 用,2D 项目用它只是浪费。

**验收流程**:

1. 把 `PerformanceHUD` 挂到主场景,跑一遍 demo 关卡,记下 draw call 峰值。
2. 故意制造"最坏场景"——把屏幕填满敌人 / 特效,看 draw call 会不会爆。
3. 关掉 `VisibleOnScreenEnabler2D`,对比节点数 vs 实际渲染的 2D items 数,确认裁剪生效。
4. 用 Debugger > Monitors 面板核对 draw call、显存、节点数趋势,寻找尖刺。
5. 如果有目标硬件(Steam Deck、低端笔记本),实测半小时游戏过程,关注 fps 1% low 数值,不只是平均 fps。
6. 录制一段 10 秒的"最坏场景"演示视频,带上 HUD 数据。每次发版前重跑同样的录制,横向对比性能变化。这是工程上"性能不漂移"的硬性约束,避免后续提交无意中拉高基础消耗。

性能问题最怕"平均看着 OK,偶尔卡顿"。1% low 比平均值更有诊断价值:它代表玩家实际感受到的"卡感"。Godot 没内置 1% low 统计,但可以自己写——`_process` 里把每帧时长压栈,统计最近 1000 帧第 99 百分位数。

性能调优的工作流应该是:**测 → 找瓶颈 → 改一处 → 再测对比**。每次只动一个变量,否则你不知道哪个改动起的作用。养成"性能改动单独提 PR"的纪律,commit message 里写清楚"改动前 / 改动后" 的 draw call、fps、节点数三组数字,后续回溯非常清楚。把性能数据放进 CI(`--headless` 启动一个 benchmark 关卡,跑 60 秒,输出 fps / draw call,超阈值就 fail)是更进一步的工程化,适合中后期项目。

## 5. 踩坑

**坑 1:`y_sort_enabled` 开在错的层级**。给整张 `Node2D` 根开 Y-sort,所有子节点参与排序,batching 几乎全断。正确做法:只在"需要 Y 排序的逻辑容器"上开(角色和可移动物体所在的 Node2D),静态背景另起一个不开 Y-sort 的兄弟节点。`TileMapLayer` 也可以单独开 Y-sort,且内部做了优化,比手动给每个 sprite 排序快得多。

**坑 2:`AtlasTexture` 用了 `atlas` 字段不是同一张图**。`atlas1 := load("res://atlas.png")`、`atlas2 := load("res://atlas.png")`,你以为是同一份,实际上 ResourceLoader 默认会缓存返回同一个对象,但如果 `CACHE_MODE_IGNORE` 加载就不同了。两个 AtlasTexture 引用不同对象时,batching 断开。规则:atlas 资源只 load 一次,所有 region 共享。

**坑 3:动态生成 material 实例**。`sprite.material = ShaderMaterial.new()`,每个 sprite 一个,batching 全断。如果只是参数不同(比如颜色),用 `set_instance_shader_parameter` 走 instance 级参数;如果材质本身相同,直接复用同一份 `ShaderMaterial.tres`。

**坑 4:`Sprite2D.region_enabled` vs `AtlasTexture` 选哪个**。两者都能"显示一张大图的局部"。规则:**大图同时被多个 sprite 共享时,用 `AtlasTexture`**(每个 sprite 的 texture 指向不同 region,但 atlas 同一);**只是当前 sprite 需要在 atlas 里找一块时**,`region_enabled = true` + `region_rect` 也行,但不利于复用。AnimatedSprite2D 内部用的是 SpriteFrames + AtlasTexture 模式。

**坑 5:`CanvasLayer` 嵌套**。CanvasLayer 是平铺的,嵌套 CanvasLayer 不会按层级累加索引,反而行为可能让人困惑。HUD 一个 CanvasLayer,过场字幕一个,屏幕特效一个,五个差不多够了,不要为每个 UI 元素一个。

**坑 6:`VisibleOnScreenNotifier2D` 边界框忘了调**。它默认的 rect 是 `(0, 0, 1, 1)`,等于一个像素点,大部分时候发不出 screen_exited 信号。要手动设成包住目标对象大小的 rect,或者代码里 `notifier.rect = $Sprite2D.get_rect()`。`VisibleOnScreenEnabler2D` 同理。

**坑 7:把 1024 个 `Sprite2D` 当作"瓷砖"**。每个 Sprite2D 都是一个 Node + CanvasItem + 内存开销。1024 个 sprite 即便 batching 后 draw call 不多,节点遍历、变换计算、信号管理的 CPU 成本也很可观。瓷砖用 `TileMapLayer`,大量同形状对象用 `MultiMeshInstance2D`。

**坑 8:`get_render_info` 在第一帧返回 0**。文档明确写了"前两帧未就绪"。如果你在 `_ready` 里立刻读 draw call 数据,会拿到 0,以为渲染没工作。养成习惯:`await get_tree().process_frame` 两次再读。

**坑 9:像素风游戏 nearest filter 没全局设**。一两个 sprite 设了 nearest,其余 sprite 自动 inherit 模糊的 linear,画面像素融化。在项目设置 `rendering/textures/canvas_textures/default_texture_filter` 改成 Nearest,新导入纹理也自动 nearest(还要看 `.import` 文件)。

**坑 10:导入设置里 `Filter = Linear` 默认开**。Godot 4 默认纹理导入开 `Filter`,导致像素风游戏看起来发糊。批量修改方法:选中所有 PNG,Inspector 改 Filter,Reimport;或者直接编辑 `.import` 文件用脚本批量修改(28 篇展开)。

**坑 11:`get_viewport().get_render_info()` 在子视口里返回主视口数据**。子视口要用 `SubViewport.get_render_info()`,不是 `get_viewport()`。`get_viewport()` 返回的是节点所在的最近视口,但有时与你预期不同。明确写 `$SubViewport.get_render_info(...)` 更安全。

**坑 12:debug 构建和 release 构建的差异**。`Performance.get_monitor()` 在 release 构建里很多 `RENDER_*` 指标返回 0,因为引擎为了性能不收集统计数据。所以"看着 debug 200 draw call"不等于 release 也 200,需要在 release 里实测 fps。

**坑 13:Godot 4 默认 Forward+ 渲染对 2D 不一定最优**。Forward+ 是 3D 友好的渲染管线,2D 项目通常用 Compatibility 或 Mobile renderer 在中低端硬件上更稳。项目设置 `rendering/renderer/rendering_method` 在新建项目时定下,后期改可能需要重新调材质和 light。纯 2D 独立游戏建议从一开始就用 Compatibility,兼容性最好、性能最稳。

**坑 14:`TileMapLayer` 仍然按层做 batch**。同一层瓷砖如果用同一个 TileSet,会合并成少量 draw call;但跨层就会断开。所以 TileMapLayer 不是越多越好——能用单层放下的就别拆,层数控制在 3-5 个(背景、地形、装饰、前景)。每层加 1-2 个 draw call。

**坑 15:UI 控件用了不同的 theme override**。`Control` 的 theme override 实际上会让该控件成为独立 material 实例,batching 断开。如果你给每个按钮都设了独立的 font color override,即使外观一样,引擎也无法合并。统一通过 `Theme` 资源管理样式,而不是逐个 override。
