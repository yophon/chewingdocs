18-数据驱动的道具、装备与关卡配置 ​

第 17 篇把"角色内部"用组件 + 资源拆开了。这一篇要把"角色外部"——道具、装备、掉落、关卡参数——同样拆出代码。目标只有一句话:

新增 50 件道具,不写一行 GDScript。

听起来夸张,实际是 Godot 项目能不能撑过 3 个月迭代的分水岭。本篇给出一条可执行的"数据驱动"工作流:用 Resource 做配表,用 .tres 文件做内容,用 PackedScene + Resource 搭配处理"实例化+配置"两步,加上一个完整的 ItemResource + DropTableResource 例子。

1. 机制定位 ​

数据驱动要解决的核心问题:让"内容扩展"和"系统迭代"解耦。

一个典型反面案例:某 ARPG 项目里,所有道具直接写成 GDScript 类。HealthPotion.gd、ManaPotion.gd、StrengthPotion.gd 各占一个文件。每个文件里硬编码图标路径、效果数值、栈上限、价格。新增 30 件药水,就要新建 30 个 .gd 文件,大量重复代码。两个月后策划想统一调价,你要把 30 个文件逐一打开改。

另一种反面案例:把数据塞进巨大的 JSON 字符串里 var items_data := "[{\"id\":\"hp_potion\",...}]"。看起来"数据剥离了",实际带来三个新问题:JSON 没有类型检查,字段拼错运行时才发现;编辑器没法可视化编辑,策划要在文本里改;资源引用(图标、模型、音效)是字符串路径,改个文件名就全断。

Godot 的解法是 Resource 作为强类型配表:

  • 数据本身是 Resource 子类,字段有类型,IDE 有补全。
  • 每条内容是一个 .tres 文件,可以在 Inspector 中可视化编辑。
  • 资源引用(图标、技能、PackedScene)是真实的 Resource / Object 引用,Godot 会跟踪文件路径变更并自动更新。
  • 一份 .tres 文件可以被任意场景、任意脚本引用,引擎做去重和缓存。

新手在这条路上最容易掉的两个坑:

  • 把 Resource 当成纯结构体:声明 8 个字段后就停手,所有计算逻辑都散布在使用方。结果"道具效果"被四五个组件分别解释,公式不一致。
  • 过度数据化:把"按下按钮触发哪段动画"也做成配表。这种"内容 = 行为序列"的设计在剧情向游戏里有意义,但放在 ARPG 里就把简单的代码变成"配置语言",每次改动都要重新发明半个状态机。

合适的边界是:数值、引用、文案、参数 → 进配表。行为、流程、状态机 → 留在代码。

这条边界还有一个反向判断:**改一份数据需要重启游戏吗?改一段行为需要重启游戏吗?**数据驱动的目标是前者尽可能"不需要",所以数据要尽量"声明式"——只描述事实,不嵌入流程控制。行为驱动则一定需要重新编译或重启,所以它继续待在 .gd 代码里。一旦你发现某个配表字段是"if X then Y else Z"这种条件分支,八成它本来就该是代码,不该挪进 .tres。

2. Godot 心智 ​

Resource 配表的三层结构 ​

完整的数据驱动通常分三层:

text
Layer 3:  Index / Database     (按 id 查 .tres,通常是 Autoload 或 ResourceLoader)
            ▲
Layer 2:  Concrete Resources   (item_red_potion.tres, drop_table_goblin.tres)
            ▲
Layer 1:  Resource Schema      (ItemResource.gd, DropTableResource.gd 类定义)

Layer 1 是 GDScript 类,定义"一件道具应该有什么字段"。Layer 2 是具体的 .tres 文件,每一条内容一个。Layer 3 是一个查找入口,业务代码拿着 &"hp_potion" 这个 id 能找到对应的资源实例。这三层各自变化:

  • Layer 1 改动 = 系统扩展(新增"暴击率"字段)。需要程序员介入。
  • Layer 2 改动 = 内容扩展(新增"火焰药水")。策划可以独立完成。
  • Layer 3 一般不动,作为稳定的查询契约。

PackedScene 和 Resource 的搭配 ​

道具有两个面:

  • 静态数据:名字、描述、价格、堆叠上限——属于 Resource。
  • 动态实体:掉在地上的可拾取物——是一个场景实例,有碰撞框、有 sprite、有"被玩家踩到时拾取"的逻辑,属于 PackedScene。

很多新手把这两件事混着写:要么所有道具共用一个 PickupItem.tscn,数据全靠"在场景里手动改 sprite";要么每个道具一个 .tscn,字段重复 N 次。正确的搭配是:

text
PickupItem.tscn  ← 唯一的拾取场景,有 Area2D + Sprite2D + 拾取脚本
       │
       │  实例化时传入
       ▼
ItemResource (.tres)  ← 道具具体是什么,数值多少

PickupItem.tscn 是模板,只关心"拾取动作如何发生"。ItemResource 是数据,描述"这件道具到底是什么"。生成"地上的火焰药水"= pickup.tscn 实例化 + 把 fire_potion.tres 塞给它的 item 字段。这样改 sprite、改名字都是改 .tres,不动场景。

@export 是数据驱动的入口 ​

@export 在 GDScript 4.6 里的修饰族对配表特别重要,它把数据契约直接写进字段声明:

  • @export var x: int 把字段暴露到 Inspector。
  • @export_range(0, 100, 1) 限制数值范围,防止策划填出离谱值。
  • @export_enum("Common", "Rare", "Epic") 把字符串字段做成下拉选择,避免拼写错误。
  • @export_multiline var description: String 启用多行编辑器,适合长描述。
  • @export_file("*.tscn") 让 Inspector 弹出文件选择对话框,只列出 .tscn。
  • @export_group("基础信息") 把字段折叠成组,字段多时不至于一屏滚不到底。

这些注解的真正作用不是"让编辑器变好看",而是把数据契约编码到字段声明里,谁打开 .tres 都看到同一套约束。

关卡也是 Resource ​

把"关卡"作为配表也是这条思路的延伸。一个关卡配置可以是这样的 Resource:

text
LevelResource (.tres)
├── id              (StringName)
├── display_name    (String)
├── scene           (PackedScene)              # 真正的 .tscn 关卡布局
├── spawn_table     (Array[SpawnEntry])        # 哪些敌人 + 数量
├── drop_modifier   (float)                    # 关卡掉落系数
├── music           (AudioStream)
├── time_limit_sec  (float)
└── reward_table    (DropTableResource)

关卡的"布局"是 PackedScene,关卡的"参数"是 LevelResource 各字段。两者搭配,关卡设计师可以做完全数据驱动的"难度调整""敌人数量平衡",而不用改场景内容。新做一个关卡 = 复制 .tres 调参数 + 拼一份新的 .tscn 布局。这条流水线设计好,内容产能可以提升一个数量级。

更进一步,关卡之间的串联也可以用 Resource 表达:一份 CampaignResource 持有 Array[LevelResource],定义"第 1 关之后是第 2 关之后是 Boss 关"。SceneRouter(第 15 篇)读这份 campaign 资源决定 next_level。这样新增关卡、调整顺序都不需要修改场景流转代码,本质上场景流也变成数据驱动的一部分。

3. 工程实现 ​

下面四份代码:ItemResource 配表模式、DropTableResource 掉落表、PickupItem 拾取场景脚本、ItemDatabase 全局索引。组合在一起,可以做到"加一件新道具 = 创建一份 .tres 文件,无需写 GDScript"。

ItemResource 是道具的数据 schema。文件位置 res://resources/item_resource.gd:

gdscript
# res://resources/item_resource.gd
class_name ItemResource
extends Resource

enum Category { CONSUMABLE, EQUIPMENT, MATERIAL, KEY }
enum Rarity { COMMON, UNCOMMON, RARE, EPIC, LEGENDARY }

@export_group("身份")
@export var id: StringName = &""
@export var display_name: String = ""
@export_multiline var description: String = ""
@export var icon: Texture2D
@export var category: Category = Category.CONSUMABLE
@export var rarity: Rarity = Rarity.COMMON

@export_group("经济")
@export_range(0, 99999, 1) var base_price: int = 1
@export_range(1, 999, 1) var max_stack: int = 1
@export var sellable: bool = true

@export_group("使用效果")
@export_range(0, 999, 1) var heal_amount: int = 0
@export_range(0, 999, 1) var mana_amount: int = 0
@export var grants_effect: EffectResource

@export_group("装备属性(仅 EQUIPMENT)")
@export_range(0, 999, 1) var equipped_attack: int = 0
@export_range(0, 999, 1) var equipped_defense: int = 0
@export var stats_modifier: StatsResource

func is_stackable() -> bool:
    return max_stack > 1

func actual_price() -> int:
    var multiplier := 1.0 + 0.5 * int(rarity)
    return int(base_price * multiplier)

func _validate_property(property: Dictionary) -> void:
    # 在 Inspector 中根据 category 自动隐藏不相关的字段
    var name: StringName = property["name"]
    match category:
        Category.CONSUMABLE:
            if name in [&"equipped_attack", &"equipped_defense", &"stats_modifier"]:
                property["usage"] = PROPERTY_USAGE_NO_EDITOR
        Category.EQUIPMENT:
            if name in [&"heal_amount", &"mana_amount", &"grants_effect"]:
                property["usage"] = PROPERTY_USAGE_NO_EDITOR

几个值得展开的细节:

  • id: StringName 是道具的唯一标识,所有交易、掉落、UI 都用它而不是 display_name。StringName 比 String 更适合做 key,引擎内部去重。它也是日后做本地化时的"语义键",display_name 可以随翻译变,但 id 永远是同一个。
  • _validate_property 是 Godot 4.x 的虚函数,Inspector 渲染每个字段前都会调一次。在这里根据 category 把不相关字段标记为 PROPERTY_USAGE_NO_EDITOR,Inspector 就不会显示它。策划看到的字段集合自动随类别变化,不会被"装备属性"误导填到消耗品上。这种"动态字段"在 Godot 4 之前只能靠注释提醒,4.x 引入这个钩子后是数据 schema 设计的关键武器。
  • grants_effect: EffectResource、stats_modifier: StatsResource 是嵌套 Resource。Inspector 会自动给一个"拖入或新建"的输入框,可以指向 17 篇定义的 EffectResource.tres。Resource 套 Resource 是 Godot 配表的核心威力,意味着你可以组合出任意复杂的数据结构而无需新建数据库。
  • actual_price 把"基础价格 + 稀有度系数"的公式写在 Resource 内部。所有买卖入口都走这里,公式只有一处定义。改稀有度乘数,所有道具价格自动重算。

DropTableResource 是掉落表。它的难点不在数据结构,而在公平、可调试、可复现的随机。文件位置 res://resources/drop_table_resource.gd:

gdscript
# res://resources/drop_table_resource.gd
class_name DropTableResource
extends Resource

class Entry:
    extends Resource
    @export var item: ItemResource
    @export_range(1, 1000, 1) var weight: int = 10
    @export_range(1, 99, 1) var min_count: int = 1
    @export_range(1, 99, 1) var max_count: int = 1

@export var entries: Array[Entry] = []
@export_range(0.0, 1.0, 0.05) var drop_chance: float = 1.0
@export_range(1, 5, 1) var rolls: int = 1
@export var allow_duplicates: bool = true

func roll(rng: RandomNumberGenerator) -> Array[Dictionary]:
    var results: Array[Dictionary] = []
    if rng.randf() > drop_chance:
        return results

    var pool := entries.duplicate()
    for i in rolls:
        if pool.is_empty():
            break
        var picked := _pick_weighted(pool, rng)
        if picked == null:
            continue
        var count: int = rng.randi_range(picked.min_count, picked.max_count)
        results.append({"item": picked.item, "count": count})
        if not allow_duplicates:
            pool.erase(picked)
    return results

func _pick_weighted(pool: Array, rng: RandomNumberGenerator) -> Entry:
    var total: int = 0
    for entry in pool:
        total += entry.weight
    if total <= 0:
        return null
    var pick: int = rng.randi_range(1, total)
    var cursor: int = 0
    for entry in pool:
        cursor += entry.weight
        if pick <= cursor:
            return entry
    return null

关键设计:

  • Entry 是嵌套类,继承 Resource,所以 Inspector 可以把 entries: Array[Entry] 渲染成可视化列表。策划点"加一项",输入物品 .tres、权重、数量范围,即生成新条目。这是把"二维表格"压进单一 .tres 文件最自然的写法。
  • weight 是整数,不是浮点小数。整数权重便于策划口算"50 % 出 A、30 % 出 B、20 % 出 C"——把权重写 50、30、20。浮点权重容易因精度问题导致最低概率项几乎不出。
  • roll 接收一个 RandomNumberGenerator 实例,而不是用全局 randf()。原因第 22 篇会展开:外部传入 RNG 让"同一种子下结果可复现",方便调试和回放。
  • drop_chance < 1.0 让"有时候不掉东西"成为正常情况。这条 0.0-1.0 区间的预先 roll 单独和"具体掉什么"分开,语义清晰。
  • 没用 pop_at / pick_random 这种 GDScript 内置随机方法,因为它们都用全局 RNG。所有随机必须显式经过传入的 RNG,这是配表系统能做到"种子可复现"的硬约束。

PickupItem 是地上可拾取的场景脚本,模板化"任何道具都能用同一个场景实例化"。文件位置 res://entities/pickup_item.gd:

gdscript
# res://entities/pickup_item.gd
extends Area2D

@export var item: ItemResource
@export_range(1, 99, 1) var count: int = 1
@export var bob_amplitude: float = 2.0
@export var bob_frequency: float = 2.0

@onready var sprite: Sprite2D = $Sprite2D
@onready var collision: CollisionShape2D = $CollisionShape2D

var _initial_y: float = 0.0
var _time_alive: float = 0.0

func _ready() -> void:
    assert(item != null, "PickupItem 没有配置 ItemResource")
    sprite.texture = item.icon
    _initial_y = position.y
    body_entered.connect(_on_body_entered)

func _process(delta: float) -> void:
    _time_alive += delta
    position.y = _initial_y + sin(_time_alive * bob_frequency) * bob_amplitude

func _on_body_entered(body: Node) -> void:
    if not body.is_in_group("player"):
        return
    EventBus.item_picked_up.emit(item.id, count)
    queue_free()

static func spawn(world: Node, drop_item: ItemResource, drop_count: int, pos: Vector2) -> void:
    var scene: PackedScene = preload("res://entities/pickup_item.tscn")
    var inst := scene.instantiate()
    inst.item = drop_item
    inst.count = drop_count
    inst.global_position = pos
    world.add_child(inst)

PickupItem.tscn 只有一个 Area2D + Sprite2D + CollisionShape2D,所有"长什么样"由 item.icon 决定。整个项目只有一个拾取场景文件,所有道具的拾取都走它。static func spawn 是一个建造入口,业务代码只要拿到 ItemResource 和位置,就能在世界里掉一件道具,不需要关心场景路径。

这种"一个模板场景 + N 份资源"的模式可以推广到非常多对象类型:子弹(一个 Projectile.tscn + ProjectileResource)、UI tooltip(一个 Tooltip.tscn + 任意 Resource 描述显示内容)、Boss 召唤的小怪(一个 Minion.tscn + EnemyResource)。每多用一次,新内容的创建成本就降一档。判断一种新对象类型是否值得套这个模式,看"这类对象未来是否会有 5 种以上变体"。3 种以下,直接做 3 个场景反而更直白;5 种以上,模板加资源的回报开始显现。

ItemDatabase 是 Layer 3,提供"按 id 查 .tres"的入口。它是一个 Autoload,文件位置 res://autoload/item_database.gd:

gdscript
# res://autoload/item_database.gd
extends Node

const ITEMS_DIR: String = "res://data/items/"

var _by_id: Dictionary = {}   # StringName -> ItemResource

func _ready() -> void:
    _reload()

func _reload() -> void:
    _by_id.clear()
    var dir := DirAccess.open(ITEMS_DIR)
    if dir == null:
        push_error("ItemDatabase: 找不到目录 %s" % ITEMS_DIR)
        return
    dir.list_dir_begin()
    var file_name := dir.get_next()
    while file_name != "":
        if file_name.ends_with(".tres"):
            var res := load(ITEMS_DIR + file_name)
            if res is ItemResource and res.id != &"":
                _by_id[res.id] = res
        file_name = dir.get_next()
    dir.list_dir_end()

func get_item(id: StringName) -> ItemResource:
    return _by_id.get(id, null)

func all_by_category(cat: ItemResource.Category) -> Array[ItemResource]:
    var out: Array[ItemResource] = []
    for item in _by_id.values():
        if item.category == cat:
            out.append(item)
    return out

这是"无代码扩展内容"的关键拼图。把一份新的 .tres 文件丢进 res://data/items/ 目录,下次启动游戏 ItemDatabase 会自动扫描注册,所有 EventBus.item_picked_up.emit(&"new_fire_sword", 1) 立刻就能找到它。整个流程不需要改任何代码。

最后给一段"敌人死亡 → 掉落 → 拾取"的全链路代码,演示几个 Resource 怎么串起来。文件位置 res://entities/enemy_drop.gd,这是挂在敌人主体上的辅助组件:

gdscript
# res://entities/enemy_drop.gd
extends Node

@export var drop_table: DropTableResource
@export var spawn_radius: float = 16.0
@export var spawn_delay_sec: float = 0.0

var _rng: RandomNumberGenerator = RandomNumberGenerator.new()

func _ready() -> void:
    # 用本局种子派生,让每只怪的掉落独立但整局可复现
    _rng.seed = GameSession.run_seed ^ hash(get_parent().name)
    var parent := get_parent()
    if parent.has_signal("died"):
        parent.died.connect(_on_died)

func _on_died() -> void:
    if drop_table == null:
        return
    if spawn_delay_sec > 0.0:
        await get_tree().create_timer(spawn_delay_sec).timeout
    var pos: Vector2 = get_parent().global_position
    var results := drop_table.roll(_rng)
    for r in results:
        var offset := Vector2(
            _rng.randf_range(-spawn_radius, spawn_radius),
            _rng.randf_range(-spawn_radius, spawn_radius)
        )
        PickupItem.spawn(get_tree().current_scene, r.item, r.count, pos + offset)

整条链路是这样合作的:敌人主体广播 died → EnemyDrop 拿 DropTableResource 滚一次 → 每条结果通过 PickupItem.spawn 在场景里生成可拾取实例 → 玩家走过去触发 Area2D.body_entered → EventBus.item_picked_up.emit(id, count) → 背包(订阅这条信号的 UI 组件)更新。整条链路没有一处代码硬编码"具体什么道具",所有"哪个怪掉什么"全在 drop_table_goblin.tres 里配置。

4. 调参和验收 ​

.tres 文件目录怎么组织 ​

按 Layer 1 资源类型分目录,不要按"游戏概念"分:

text
res://data/
├── items/           ItemResource
│   ├── hp_potion.tres
│   ├── fire_potion.tres
│   └── iron_sword.tres
├── effects/         EffectResource
│   ├── burn.tres
│   └── poison.tres
├── drop_tables/     DropTableResource
│   ├── goblin.tres
│   └── boss_chest.tres
└── stats/           StatsResource
    ├── goblin_base.tres
    └── elite_goblin.tres

按资源类型分有两大好处:ItemDatabase 这种扫描器一行 ends_with(".tres") 就能定位;Git 上看 diff,任何新增内容一眼能看出是新道具、新效果还是新掉落表。反例是按"关卡"或"敌人"分:res://data/level_1/goblin/drops.tres,看似贴近游戏概念,实际让同一份资源(比如通用 hp_potion)在多个目录里出现引用,迁移和重构都麻烦。如果项目规模大到一个类型下文件太多,可以按"前缀"在文件名层级而不是目录层级做二次分组——item_weapon_iron_sword.tres、item_weapon_steel_sword.tres、item_potion_hp.tres,Inspector 的"按名字过滤"会比目录跳转更顺手。

平衡性参数提取层级 ​

调参的关键是把"参数"提到正确的层级。错误层级会让调整变成噩梦:

  • 道具实例自带参数(hp_potion.tres 里写 heal 50):适合"这件道具就这么强",最常见。
  • 稀有度系数:不写在每件道具里,放在 RarityCurves.tres 这种全局曲线表中,所有道具按稀有度自动拿乘数。改稀有度曲线一次调整全局。
  • 关卡难度系数:不要在敌人 .tres 里写"普通哥布林"和"困难哥布林"两份,而是写一份 goblin_base.tres 加一份 difficulty_curves.tres,关卡运行时根据当前难度对基础属性乘系数。
  • 平衡补丁:对已发布游戏做平衡调整,目标是"只改一两个 .tres,玩家自动同步"。这一条直接决定项目能不能做长期运营。

这里有一个常被忽视的指标:调一次数值,需要打开多少份 .tres 文件。健康的配表结构下,"全局调价 10%"只需要改一份 economy_curve.tres,而不是 300 份道具 .tres。"提升所有 Boss 血量 20%"只需要改一份 boss_difficulty.tres,而不是逐个 Boss 改。如果你发现自己经常要批量编辑数十份 .tres,说明参数被错误地提到了"实例层"而不是"曲线层",该重新考虑层级划分。

Inspector 友好度 ​

策划改 .tres 的体验,直接决定数据驱动是否能在团队真正落地。三个友好度抓手:

  • @export_group 必须用。任何超过 6 个字段的 Resource 都该分组,否则策划面对一屏滚动列表就会出错。
  • 默认值要有意义。@export var icon: Texture2D 默认 null,但 @export var max_stack: int = 1 给个 1 而不是 0。让"新建一份 .tres,什么都不填" 也能跑起来,只是数值平庸。
  • 关键字段写 tooltip。@export var weight: int ## 权重越大越容易掉 这种以 ## 开头的注释会出现在 Inspector 的鼠标悬浮提示里。策划无须打开 .gd 文件就能理解字段含义。

验收清单 ​

一个健康的数据驱动结构应满足:

  • 全工程搜索 preload("res://data/,业务代码里不应该硬编码 .tres 路径。所有访问都走 ItemDatabase.get_item(&"id")。
  • 全工程搜索 class_name ItemResource,在使用方应只有一处声明,在配置方有 N 份 .tres 文件。
  • 临时把一份 .tres 文件改一个数值(比如 hp_potion.tres 的 heal_amount 改成 999),不重新编译、不改代码,直接运行游戏,效果立即生效。这是"数据驱动"最基本的验收。
  • 让非程序员同事(策划、关卡设计)只用 Godot 编辑器加 5 件新道具,无需打开任何 .gd 文件。他们能完成 = 配表层级设计成功。
  • 把一份核心 .tres(例如 hp_potion.tres)的文件名改掉,Godot 编辑器应当能在所有引用点自动更新(场景里的 @export 字段、其他 .tres 内部引用)。如果有断链,说明项目里某处仍然用了字符串路径而不是资源引用,违反了"数据驱动"的核心契约。

5. 踩坑 ​

坑 1:把"行为"也写进 Resource ​

gdscript
# 反例:在 ItemResource 里塞行为
@export_multiline var on_use_script: String = "player.hp += 50"

把脚本片段当字符串放进 .tres,然后运行时 Expression.parse(on_use_script)。看起来"完全数据驱动",实际让调试变成地狱:字符串没语法检查、没补全、没断点。行为留在代码里,数据留在 .tres 里。复杂效果用枚举 + 参数表达,例如 effect_type: EffectKind.HEAL + effect_amount: 50,而不是嵌入式脚本。

坑 2:嵌套 Resource 默认共享导致改一处改全部 ​

gdscript
# 道具 A、B 都引用了同一个 StatsResource.tres
# 在 A.equip() 里写
item.stats_modifier.attack_power += 10

Resource 是引用类型,A 和 B 引用同一个 .tres 实例,改 A 的 modifier 也会让 B 跟着改。配置 Resource 应当视为只读,需要写入就先 duplicate()。如果某些字段一定会运行时修改,设计上让它属于"组件状态"而不是"资源配置",17 篇已经讲过这条边界。

坑 3:@export var icon: Texture2D 让数据库 Autoload 引用了所有图标 ​

ItemDatabase 加载所有 ItemResource,而每个 ItemResource.icon 是 Texture2D 引用,启动时所有道具图标都被加载进内存。300 件道具 = 300 张纹理常驻。

两种缓解:小项目无所谓,300 张 32×32 图标占内存极少;大项目改用 @export_file("*.png") 存路径字符串,UI 显示时 load(item.icon_path) 按需加载,再配合 WeakRef 让冷门道具图标可被回收。或者用一张 atlas + 帧索引。

坑 4:Array[ItemResource] 在场景里编辑后变成 null 项 ​

在 DropTableResource.entries 这种 Array[Entry] 字段里,有时候点"加一项"会留下一个空槽 <null>。如果业务代码假设每项非空,就会运行时崩。

roll() 内部一定要写 if entry == null: continue 这种空值保护,对所有数组类配置都加上这一条。Godot 编辑器允许 null 存在,要把它当作合法输入处理。

坑 5:在 _init 里访问其他 Resource 字段 ​

gdscript
# item_resource.gd
func _init() -> void:
    base_price = grants_effect.duration * 10   # 反例

Resource._init 是反序列化前的默认构造,这时所有 @export 字段还是初始值,grants_effect 还是 null。Resource 之间的关系应该在使用方(组件、控制器)解释,而不是在资源的构造函数里。要做派生字段,用 func calc_xxx() -> ... 即时计算,或者重写 setter 在赋值时更新。

坑 6:ItemDatabase 用 preload 而不是 load ​

gdscript
# 反例
const ALL_ITEMS := [
    preload("res://data/items/hp_potion.tres"),
    preload("res://data/items/fire_potion.tres"),
    # ...
]

preload 是编译期常量,新加一份 .tres 必须再加一行 preload,并修改这个 Autoload 脚本。完全失去了"数据驱动"的初衷。改用运行时 DirAccess 扫描目录 + load,新增 .tres 自动注册,这才是"无代码扩展"。

坑 7:.tres 文件名和 id 字段不一致 ​

文件 hp_potion_v2.tres 内部 id = &"hp_potion"。这种"id 和文件名不一样"的情况会让 Git 历史和"哪份 .tres 真正生效"对不上。

约定:.tres 文件名就是 id(fire_potion.tres 的 id 字段一定是 &"fire_potion")。可以写一个 @tool 校验脚本,扫目录确认所有 .tres 的内部 id 与文件名一致,作为 CI 检查。第 28 篇展开 EditorPlugin 时会讲怎么做这种校验工具。

坑 8:DropTableResource 共享 RNG 实例导致跨怪掉落串扰 ​

gdscript
# 反例
var _shared_rng := RandomNumberGenerator.new()
func roll() -> Array:
    return drop_table.roll(_shared_rng)

多只敌人共享同一个 RNG 看似省事,但任何一只死了就推进 RNG 状态,其他敌人的掉落不再独立。每只敌人初始化时建自己的 RNG,种子来自 GameSession.run_seed 派生(rng.seed = run_seed ^ hash(name)),这样既保留"同一种子下整局可复现",又让每只怪的掉落彼此独立。

坑 9:Inspector 中改完 .tres 忘记 Ctrl+S ​

Godot 在 Inspector 编辑 Resource 时,改动只在内存里。如果不显式保存(Ctrl+S 或菜单 Save Resource),改动会随着关闭场景丢失。这是个非常常见的"我明明改了为什么没生效"事故。

习惯做法:每次改完 .tres 立刻 Ctrl+S。保险一点的做法是开 Editor > Editor Settings > Save Every Modified Resource 之类的自动保存,但 Godot 4.x 这块不太稳,推荐还是养成手动存的习惯。另外可以写一个 EditorPlugin 在切换文件时强制保存当前 Resource。

坑 10:把"运行时不变的派生数据"也存进 .tres ​

gdscript
# 反例:把可由 base_price + rarity 算出的字段也存进 .tres
@export var actual_price: int = 0

存这种派生字段的麻烦是:base_price 改了但忘了改 actual_price,数据就不一致。派生字段用函数即时计算,只存"无法由其他字段推导出来的最少信息"。同理,is_legendary: bool 是冗余的(可以由 rarity == LEGENDARY 算出),total_stats: Vector3 也是冗余的(可由 attack + defense + speed 推导)。最小化存储字段意味着一致性约束更少,出错概率更低。

手动验证 ​

  • [ ] 在 res://data/items/ 目录下用编辑器新建一份 mana_potion.tres,选中 ItemResource 类型,填好字段,启动游戏后用 ItemDatabase.get_item(&"mana_potion") 应当能取到它,全过程不打开任何 .gd 文件。
  • [ ] 在 ItemResource 把 category 改成 EQUIPMENT,Inspector 中"使用效果"组的字段(heal_amount 等)应当自动隐藏,"装备属性"组应当显示出来,这说明 _validate_property 钩子生效。
  • [ ] 用同一个种子运行游戏两次,杀同一只敌人,掉落结果完全一致(说明 RNG 真的从外部注入而不是全局)。
  • [ ] 让一份 .tres 文件指向 <null> 图标(故意不填 icon 字段),拾取场景应当播警告而不是崩溃(说明对空字段做了防护)。
  • [ ] 把 hp_potion.tres 的 heal_amount 字段从 50 改成 9999,不重启编辑器、不重新编译,在游戏运行中暂停后改 .tres(或重启游戏即可),拾取 hp_potion 立即获得 9999 治疗。
  • [ ] 临时把一份 .tres 文件的 id 字段改成空 &"",ItemDatabase._reload 应当跳过它并打 warning,而不是把空 id 注册进字典覆盖其他道具。
