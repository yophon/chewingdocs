# 15-场景流管理器与第一个完整可玩闭环

> 一句话导读:做出会动的角色、会打的敌人、能存的档,不等于做出"一款游戏"。把它们串成"开始 → 玩 → 死 → 重来 → 胜 → 结算"的闭环,是从"功能堆"到"原型"的最后一步。

第三层走到这里(11 关卡 / 12 敌人 AI / 13 UI / 14 存档),手上零件齐全:能踩的地、能打的怪、能看的 HUD、能存的档,但它们之间还彼此孤立。玩家死亡时角色还在原地、关卡没重置、HUD 还停在血条 0 的状态、敌人继续刷新——只要少一个节点把这一切串起来,所有零件加起来就还不算一个游戏。

本篇是第三层的收尾。给原型加一个 `SceneManager`,管住主菜单、关卡、死亡、胜利之间的转移,定义清楚状态机的边界,把前面九篇的成果接到一起。完成后,这个原型应该能让外人坐下来,从启动到通关、从死亡到重开、从存档到读档,跑通完整的循环。完成第三层并不意味着原型已经"好玩",但意味着接下来可以拿这套结构去打磨细节而不被结构性问题反复拖住。

阅读路径上,这一篇与第 16 篇(Autoload、事件总线与全局状态边界)是紧挨着的伙伴。`SceneManager` 是 autoload 的典型场景,但它只解决"场景流"这一种全局协调问题;事件总线、玩家会话状态、游戏全局配置之类的关注点都是单独的边界,第 16 篇会把"什么时候该用 autoload、什么时候不该"讲清楚。本篇先实现 `SceneManager`,把模式打通,边界判定留给下一篇。

## 1. 机制定位

把项目从"能跑零散场景"做成"能玩一遍的原型",真正在解决五个问题:

1. **会话状态**:游戏当前处于"主菜单"还是"游戏中"还是"暂停"还是"结算页"?这个状态影响输入处理(主菜单里 Esc 不该弹暂停)、HUD 显示(主菜单不该显示血条)、存档触发(结算页不该自动存)。
2. **场景切换的副作用**:从主菜单跳进关卡时,旧场景要释放,新场景要加载,加载过程中玩家不能输入,加载完成后 HUD 要重置,音乐要切换。这些副作用必须按顺序发生,不能漏。
3. **重开的"干净"**:死亡重开关卡时,关卡里所有动态状态(敌人位置、血量、捡过的金币、踩过的开关)都要复位。最简单粗暴的办法是 `change_scene_to_packed` 同一个 `PackedScene`,让引擎自己释放重建——但 `SceneManager` 必须保证暂停标志被清掉、HUD 被重置、不要继承上一条命的状态。
4. **胜利的"结算"**:玩家通关后,要展示统计(用时、击杀、金币),要把进度写进存档(下次主菜单显示"继续"按钮),还要给玩家"回主菜单 / 进入下一关"的选择。这一段是新手最容易省的,但也是体感上最像"游戏"的地方。
5. **死亡 / 胜利的来源**:谁来 emit 这两个事件?角色脚本?敌人脚本?Boss 房间?如果每个地方都自己处理"切到死亡画面",维护噩梦。应当统一向 `SceneManager` 报告,`SceneManager` 决定后续。

新手最常见的反模式是把"切场景"的代码散在到处:角色死了在 `Player.gd` 里 `get_tree().change_scene_to_file("res://main_menu.tscn")`,胜利在某个 `Goal` 触发器里 `get_tree().reload_current_scene()`,暂停菜单的"退出"按钮自己 `get_tree().quit()`。这样写短期能跑,长期任何一处改了规则都要追十几个文件。把切换权集中到 `SceneManager` 是降低耦合的最小变更。

更深一层是,**"游戏会话状态"是一个一等公民的领域概念**,不应当藏在场景节点的隐式状态里。"角色当前血量"是关卡状态,"玩家当前在哪一关"是会话状态,"游戏是不是处于结算页"也是会话状态。会话状态变化的瞬间往往要触发副作用:进入暂停态时弹菜单、进入结算态时写存档、进入死亡态时停止背景音乐。把这些副作用挂在场景节点的 `_ready`、信号回调里,就会出现"换一个关卡需要重写一遍存档逻辑"。专门有一个 `SceneManager` 来管,等于把会话状态从场景里剥出来变成显式的 enum + 转移函数,后续每次扩展只动这一个文件。

## 2. Godot 心智

### SceneTree.change_scene_to_* 的真实行为

切场景的核心 API 是 `SceneTree` 上的两个方法:

```text
SceneTree.change_scene_to_file(path: String) -> Error
SceneTree.change_scene_to_packed(packed_scene: PackedScene) -> Error
```

两者都把当前场景换成新场景,但有几个容易踩的细节:

- **它们是延迟的**。调用之后当前帧不会立刻切,引擎会在当前帧逻辑结束后释放旧场景、实例化新场景、进入下一帧。这意味着调用后立刻 `get_tree().current_scene` 拿到的还是旧场景。
- **返回 `Error`**,如果文件不存在或解析失败会返回非 OK 值,不会抛异常。生产代码应当检查返回值。
- **`scene_changed` 信号**:`SceneTree` 在场景切完后发 `scene_changed(new_scene: Node)`,可以 `await get_tree().scene_changed` 拿到稳定的新场景引用。这是 4.x 推荐的等待方式,比 `call_deferred` + 自己查 root 干净。
- **`change_scene_to_packed` 比 `change_scene_to_file` 快**,因为 `PackedScene` 已经加载在内存里。常用做法是在主菜单 `_ready` 期间预加载下一关 `preload("res://levels/level_1.tscn")`,切换瞬间无延迟。
- **`reload_current_scene()`** 本质等价于 `change_scene_to_file(current_scene.scene_file_path)`,适合"重开本关"。注意:它复位的是场景节点,不复位 autoload。autoload 里的状态(`SaveSystem._password`、`SceneManager.current_state`)需要 `SceneManager` 自己负责清。

### 为什么用 autoload 持有切换权

autoload(第 16 篇会专门讲什么时候该用、什么时候不该用)在引擎启动时实例化一次,挂在场景树根之下,所有场景切换不会重建它。这个性质让它天然适合做"跨场景持续存在的协调者":

- 主菜单按下"开始"→ `SceneManager.start_new_game()` → autoload 切到 Level1。
- Level1 里玩家死亡 → `SceneManager.player_died()` → autoload 重开本关或者切到死亡画面。
- 通关 → `SceneManager.level_completed(stats)` → autoload 切到结算页,显示数据,等玩家确认后回主菜单或下一关。

调用方都是单一入口,场景节点自己不操心"下一步去哪",只负责"告诉 manager 发生了什么"。这条原则与第 16 篇的事件总线一致:**节点 emit 业务事件,系统决定如何处理**。

### 转场动画的位置

裸切场景非常生硬:画面瞬间黑、新场景突然亮。加一个淡入淡出的 `ColorRect` 在 `SceneManager` 自己拥有的 `CanvasLayer` 上,不跟随任何场景:

```text
SceneManager (Node, autoload)
└── Overlay (CanvasLayer, layer=100)
    └── FadeRect (ColorRect, anchor=Full Rect, color=Color.BLACK, modulate.a=0)
```

切换流程是:`Tween` 把 `FadeRect` 透明度从 0 推到 1(黑屏)→ `change_scene_to_packed`→ `await scene_changed` → `Tween` 从 1 拉回 0(显出)。这一段在 `SceneManager` 里封一次,所有场景切换都用同一个流程,不在场景节点里复制。

`process_mode = ALWAYS` 是关键:转场期间游戏被 `paused` 也好、还没暂停也好,`Tween` 都要持续推进。如果忘了设,会出现"暂停状态下切场景,黑屏卡住一直推不回来"的死锁。`Tween` 在 4.x 默认是 `PROCESS_PHYSICS` 模式,被场景树暂停同步影响——`tween.set_process_mode(Tween.TWEEN_PROCESS_IDLE)` 或者 manager 自身 `ALWAYS` 可以双保险。

### 死亡 / 胜利的信号方向

设计原则:**业务节点 emit 事件,manager 决定动作**。

- `Player.gd` 里血量到 0 时 emit `died` 信号,Level 监听后调用 `SceneManager.player_died()`,而不是 `Player.gd` 自己切场景。
- `Goal` 触发器(关卡终点)emit `level_completed(stats)`,Level 监听后调用 `SceneManager.level_completed(stats)`。

这样:
- 同一个 `Player.gd` 在不同关卡可以有不同的死亡处理(教学关用"血量为 0 暂停演示",正式关用"切到 GameOver 画面")。
- 单元测试 `Player.gd` 时,不需要 mock 整个 `SceneManager`,只需要监听信号。
- 第 16 篇的事件总线可以平滑地把"`Player.died` → `SceneManager.player_died`"的直连,改成"`Player.died` → `EventBus.player_died` → `SceneManager` 订阅",而不需要改 `Player.gd`。

这里也要意识到 **谁是关卡的"控制器"**。关卡场景的根 `Node2D` 自身脚本(下一段会看到的 `stage_1.gd`)承担了这个角色:它知道这一关里 `Player`、`Goal`、`HUD`、`PauseMenu` 节点在哪,负责把这些子节点的信号转给 `SceneManager`。它是关卡与 manager 之间唯一的胶水层,manager 不需要知道关卡内部结构,关卡子节点也不需要知道 manager 存在。这一层胶水是必要的——如果直接让 `Player` 引用 `SceneManager`,你的 `Player.gd` 就变成了"只能在这套场景流里跑"的耦合代码,移植到下一个 demo 工程要改一堆。

## 3. 工程实现

下面是 `SceneManager` 的核心实现。挂作 autoload,名字也叫 `SceneManager`,场景文件 `res://systems/scene_manager.tscn`(包含 `CanvasLayer` 和 `ColorRect`),根脚本如下:

文件位置:`res://systems/scene_manager.gd`。

```gdscript
extends Node
## Autoload: SceneManager

enum State { BOOT, MAIN_MENU, LOADING, PLAYING, PAUSED, DEATH, VICTORY }

signal state_changed(from: State, to: State)
signal level_loaded(level_id: StringName)

const MAIN_MENU_SCENE: String = "res://ui/main_menu.tscn"
const DEATH_SCENE: String = "res://ui/death_screen.tscn"
const VICTORY_SCENE: String = "res://ui/victory_screen.tscn"

@export var fade_duration: float = 0.4

@onready var _fade: ColorRect = %FadeRect

var current_state: State = State.BOOT
var current_level_id: StringName = &""
var _level_paths: Dictionary[StringName, String] = {
    &"stage_1": "res://levels/stage_1.tscn",
    &"stage_2": "res://levels/stage_2.tscn",
    &"final":   "res://levels/final.tscn",
}
var _is_transitioning: bool = false
```

下半段是状态转移与切换流程:

```gdscript
func _ready() -> void:
    process_mode = Node.PROCESS_MODE_ALWAYS
    _fade.modulate.a = 0.0
    _fade.mouse_filter = Control.MOUSE_FILTER_IGNORE
    call_deferred("_goto_main_menu")

func start_new_game() -> void:
    SaveSystem.save_slot(0, {"player": {"hp": 100}, "level_id": "stage_1", "inventory": {}})
    _change_to(_level_paths[&"stage_1"], State.PLAYING, &"stage_1")

func continue_game() -> void:
    var data: Dictionary = SaveSystem.load_slot(0)
    if data.is_empty():
        push_warning("no save, falling back to new game")
        start_new_game()
        return
    var lid: StringName = StringName(data.get(&"level_id", &"stage_1"))
    _change_to(_level_paths.get(lid, _level_paths[&"stage_1"]), State.PLAYING, lid)

func player_died() -> void:
    if current_state != State.PLAYING:
        return
    _set_state(State.DEATH)
    _change_to(DEATH_SCENE, State.DEATH, &"")

func restart_current_level() -> void:
    if current_level_id == &"":
        return
    _change_to(_level_paths[current_level_id], State.PLAYING, current_level_id)

func level_completed(stats: Dictionary) -> void:
    if current_state != State.PLAYING:
        return
    _set_state(State.VICTORY)
    var next: StringName = stats.get(&"next_level", &"")
    if next != &"" and _level_paths.has(next):
        _change_to(_level_paths[next], State.PLAYING, next)
    else:
        _change_to(VICTORY_SCENE, State.VICTORY, &"")

func quit_to_main_menu() -> void:
    get_tree().paused = false
    _goto_main_menu()

func _goto_main_menu() -> void:
    _change_to(MAIN_MENU_SCENE, State.MAIN_MENU, &"")
```

切场景本身用一段统一流程,带淡入淡出和异步等待:

```gdscript
func _change_to(scene_path: String, next_state: State, level_id: StringName) -> void:
    if _is_transitioning:
        return
    _is_transitioning = true
    get_tree().paused = false
    await _fade_to(1.0)
    var err: Error = get_tree().change_scene_to_file(scene_path)
    if err != OK:
        push_error("change_scene failed: %s -> %s" % [scene_path, err])
        _is_transitioning = false
        return
    await get_tree().scene_changed
    _set_state(next_state)
    if level_id != &"":
        current_level_id = level_id
        level_loaded.emit(level_id)
    await _fade_to(0.0)
    _is_transitioning = false

func _fade_to(target_alpha: float) -> void:
    var tween: Tween = create_tween()
    tween.tween_property(_fade, "modulate:a", target_alpha, fade_duration)
    await tween.finished

func _set_state(next: State) -> void:
    if current_state == next:
        return
    var prev: State = current_state
    current_state = next
    state_changed.emit(prev, next)
```

关卡场景作为切换链的一环,只负责把局部信号转给 `SceneManager`:

文件位置:`res://levels/stage_1.gd`,挂在每个关卡场景的根 `Node2D` 上。

```gdscript
extends Node2D

@onready var _player: Player = %Player
@onready var _goal: Area2D = %Goal
@onready var _hud: HUD = %HUD
@onready var _pause: PauseMenu = %PauseMenu

func _ready() -> void:
    _player.died.connect(_on_player_died)
    _player.hp_changed.connect(_hud.update_health)
    _goal.body_entered.connect(_on_goal_reached)
    _pause.restart_requested.connect(SceneManager.restart_current_level)
    _pause.quit_to_menu_requested.connect(SceneManager.quit_to_main_menu)

func _on_player_died() -> void:
    SceneManager.player_died()

func _on_goal_reached(body: Node) -> void:
    if not body is Player:
        return
    SceneManager.level_completed({
        "next_level": &"stage_2",
        "time_spent": Time.get_ticks_msec() / 1000.0,
    })
```

主菜单脚本只调 `SceneManager`,不直接碰 `SceneTree`:

```gdscript
extends Control

@onready var _continue_btn: Button = %ContinueButton

func _ready() -> void:
    var has_save: bool = FileAccess.file_exists("user://saves/slot_0.json")
    _continue_btn.disabled = not has_save
    %StartButton.pressed.connect(SceneManager.start_new_game)
    %ContinueButton.pressed.connect(SceneManager.continue_game)
    %QuitButton.pressed.connect(get_tree().quit)
```

死亡画面和胜利画面也是同样的模式——纯视图,只把按钮信号转给 `SceneManager`:

```gdscript
extends Control
## res://ui/death_screen.gd

func _ready() -> void:
    %RetryButton.pressed.connect(SceneManager.restart_current_level)
    %MainMenuButton.pressed.connect(SceneManager.quit_to_main_menu)
    %RetryButton.grab_focus()
```

胜利画面要多一步展示统计数据,统计来源是关卡上报的 `stats` 字典。这里有两条实现路径:一是关卡上报后 manager 把 stats 存到自己一个字段里,胜利画面 `_ready` 时读;二是关卡先把 stats 写进 `SaveSystem` 的"最近一次通关"区段,胜利画面读存档。前者耦合 manager 字段,后者多一次 IO 但解耦更彻底。原型阶段直接用前者:`SceneManager.last_victory_stats: Dictionary`,胜利画面读完显示完即 reset。

这样整条链路是闭合的:玩家在主菜单按"开始"→ `SceneManager.start_new_game` 写存档并切到 stage_1 → 玩家死亡 → `SceneManager.player_died` 切到死亡画面 → 死亡画面按"重开"→ `SceneManager.restart_current_level` → 玩家通关 → `SceneManager.level_completed` 切到下一关或胜利画面 → 胜利画面按"回主菜单"→ `SceneManager.quit_to_main_menu`。整个游戏只有一个地方在切场景,所有"游戏状态发生了什么"的事件都在一个文件里看得见。

## 4. 调参和验收

**转场时长**:`fade_duration` 默认 0.4 秒。短于 0.2 秒玩家感觉切硬,长于 0.6 秒玩家觉得节奏拖。重开关卡可以稍长(0.5),让玩家有"换一口气"的心理缓冲;主菜单到关卡可以稍短(0.3),让玩家快进入游戏。

**预加载策略**:小关卡(<50 MB)用 `change_scene_to_file` 同步切就够,玩家不会察觉。大关卡或者有大量音频 / 贴图的场景,用 `ResourceLoader.load_threaded_request` 后台加载,黑屏期间显示一个进度条。这条线属于第 24 篇异步加载,本篇先按同步走。

**暂停清理**:`_change_to` 起手就 `get_tree().paused = false`。原因是玩家可能从暂停菜单点"退出主菜单",此时 `paused` 还是 true,如果不重置,主菜单加载后第一帧仍然是暂停态,所有 UI 动画停顿。

**死亡判定的源头**:`Player.died` 信号应当在 `Player.hp <= 0` 时 emit,而不是"动画播完后" emit。播完死亡动画再切场景可以放在死亡画面那一侧,这样切场景的责任在 manager 而不在 player。

**关卡序列的数据驱动**:`_level_paths` 字典写死在 `SceneManager.gd` 里短期没问题,几关之后会变成"加一关要改 manager"的负担。第 18 篇会把这串关卡变成一个 `LevelSequence.tres` 资源,字段包括 `id`、`scene_path`、`display_name`、`next_id`、`unlock_condition`。本篇先用字典,关注主线状态机的正确性,数据化重构留给后面。

**状态转移合法性**:严格的状态机要拒绝非法转移——从 `MAIN_MENU` 直接跳 `DEATH` 是错误,从 `PLAYING` 跳 `MAIN_MENU` 才合理。`_change_to` 里可以加一张 `Dictionary[State, Array[State]]` 的转移表,非法转移 `push_error`。原型阶段保持宽松(只在 `player_died` 里检查 `current_state == PLAYING`)就够,有了 bug 报告再加严。

**第三层验收线**:这是本系列前 15 篇的"原型质量检查"。原型至少要能跑通下列动作而不崩溃、不丢档、不死循环:

- 从启动到主菜单 → 开始游戏 → 跑通 stage_1 → 跑通 stage_2 → 看到胜利画面 → 回主菜单 → 退出。中间没有黑屏卡死、没有"按钮按下没反应"、没有 console error。
- 死亡多次重开,角色每次都从关卡起点开始,血量满,敌人复位,捡过的金币重置。
- 在 stage_2 中段存档,关游戏,再启动,主菜单"继续"按钮可用,点击后回到 stage_2。
- 在玩家死亡瞬间按 Esc,不出现"暂停菜单和死亡画面同时显示"的视觉错乱。
- 在加载新关卡的黑屏期间按任意键,不会触发关卡里的动作。
- 玩 20 分钟后查看 `OS.get_static_memory_usage()`,内存没有持续增长(没有节点泄漏)。

这六条达不成,前面 09-14 的功能都不算真的能用;达成了,这个原型就可以拿出来给别人试玩,获得真正的反馈。

**原型 ≠ 游戏**。完成第三层不等于游戏可以发售。验收清单的目标是确认机制层面的完整,确认这些零件能稳稳运转。手感、关卡设计、节奏、平衡、音乐、美术——这些"游戏感"的维度本篇不解决,后 15 篇也只覆盖到工程支撑,真正打磨成作品要靠测试与迭代。这一步是从"我写了一些 Godot 代码"到"我做了一款游戏的原型"的转折点,但绝不是终点。

## 5. 踩坑

**`change_scene_to_file` 是延迟的,不要在它之后立刻访问新场景。** 调用后当前帧 `get_tree().current_scene` 还是旧场景,新场景下一帧才挂进树。需要在新场景准备好时做事情(比如读存档恢复玩家位置),用 `await get_tree().scene_changed`,然后再操作 `current_scene`。

**`await` 之后 self 可能已被释放。** 切场景流程里的 `await _fade_to(1.0)` 之后 `_is_transitioning` 还在用 self,但如果中途整个 autoload 被卸载(实际场景里不会,但单元测试时会),后续访问 self 会 `INVALID_INSTANCE`。`SceneManager` 是 autoload 不会被释放,所以本篇代码安全;但同样的模式写在普通节点上要 `is_instance_valid(self)` 兜底。

**`get_tree().paused` 跨场景持久。** 玩家在 stage_1 暂停 → 退出主菜单 → 主菜单本身也是 `paused = true` 状态。如果主菜单里没有 `process_mode = WHEN_PAUSED`,主菜单按钮不响应。最干净的做法是 `_change_to` 起手就 `paused = false`,别指望各场景自己清理。

**`reload_current_scene()` 不复位 autoload。** 重开关卡时,场景节点是新的,但 `SceneManager.current_state`、`SaveSystem._password`、`EventBus` 已订阅的信号都还在。如果某个 autoload 在 `_ready` 里订阅了上一关某个节点的信号,那个信号源已经被 `queue_free`,但 autoload 的连接还指着,触发时报 `Invalid call`。autoload 订阅业务节点信号时,要在场景退出时显式断开,或者用 `Callable.bind` + `is_instance_valid` 容错。

**旧场景和新场景在切换瞬间共存一帧。** `change_scene_to_*` 内部是"实例化新场景 → 加入树 → 把旧场景 `queue_free`"。`queue_free` 在帧末才真正释放,所以同一帧里两份场景都在树上,两个 `Player` 节点都存在。如果旧 `Player` 在这一帧还在 `_physics_process` 跑,可能 emit 信号触发奇怪行为。靠 `process_mode = DISABLED` 或者在 `_exit_tree` 里清状态可以兜底。

**autoload 持有节点引用导致泄漏。** `SceneManager.last_player = player_node` 这种写法,玩家切关卡后旧 `player_node` 被 `queue_free`,引用本身还在 autoload 里指着一个 freed instance。访问会报错,但更隐蔽的问题是:如果引用的是 `RefCounted`(`Resource` 子类),引用计数不为零的资源不会被释放,内存慢慢涨。autoload 引用业务节点的原则是:用 `WeakRef` 或者在节点 `_exit_tree` 里清空 manager 上的引用。

**死亡画面 / 胜利画面应当也是独立场景。** 不要让它们叠在关卡场景之上(`add_child`),那会让"死亡时关卡还在跑物理"。直接切场景,关卡完整释放,死亡画面是独立场景,只有一个文字 + 重开按钮 + 主菜单按钮,内存占用低,不容易出错。

**预加载放在主菜单 `_ready`,不放在 autoload `_ready`。** Autoload 在引擎启动时跑,玩家此时还在看 splash screen,加载越多 splash 越长。主菜单 `_ready` 时玩家已经在交互,后台预加载 stage_1 不会被察觉。`SceneManager` 可以提供 `preload_level(id: StringName)` 接口,但调用时机由主菜单决定。

**`change_scene_to_file` 的 `Error` 返回值很多人不检查。** 路径打错时,返回 `ERR_CANT_OPEN` 但不抛异常,然后 `await get_tree().scene_changed` 等不到信号,玩家黑屏卡死。永远检查 `err != OK` 并兜底回退到主菜单或者重试。

**`_unhandled_input` 在场景切换期间会泄露。** 玩家在黑屏期间按 Esc,如果旧场景还没释放,旧场景的暂停菜单监听器可能触发,弹出已经不该存在的暂停菜单。`_is_transitioning` 期间 manager 应该 swallow 所有相关输入,或者把 `Input.action_press` 全部 release 一遍。最简洁的写法是 `get_tree().paused = false` 后立刻 `_fade.mouse_filter = Control.MOUSE_FILTER_STOP` 截掉所有点击,切完再放开。

**主菜单不应该自动写存档。** `SaveSystem.save_slot(0, ...)` 在 `start_new_game` 里调一次是合理的(玩家选了"开始新游戏",上一份档可以覆盖)。但在主菜单 `_ready` 里自动写就坏:玩家想保留旧档,看一眼主菜单又退出,旧档被覆盖了。任何写存档的动作都应当对应玩家显式意图。

**`await` 的"丢回退"问题**:`_change_to` 内部连续 `await`,中间任何一步报错都会"漏掉"重置 `_is_transitioning = false`。结果是 manager 永远卡在 transitioning 态,后续切场景全部无响应。生产代码用 try / except 模拟(`call_deferred` 包一层 reset),或者把状态机做得健壮一些——比如转场超过 3 秒未完成就强制 reset 并切回主菜单。

**autoload 之间的初始化顺序敏感**:`project.godot` 的 `[autoload]` 段按声明顺序加载。`SceneManager._ready` 里调 `SaveSystem.save_slot` 时,`SaveSystem` 必须比 `SceneManager` 先加载。把基础设施(`SaveSystem`、`EventBus`)排在 `SceneManager` 之前,把"协调者"放后面,是最稳的顺序。错了顺序的症状很隐蔽:`call_deferred("_goto_main_menu")` 在 `SceneManager._ready` 里看似没问题,但只有 `await` 之后才用到 `SaveSystem`,中间隔着一两帧,bug 难复现。

## 手动验证

第三层的最终验收。前面 06-14 篇都做完时,这一段应当全部通过:

- [ ] 启动游戏不超过 5 秒进入主菜单。第一次启动时"继续"按钮灰色,"开始"按钮可点。
- [ ] 点"开始"→ 黑屏 0.4 秒 → 进入 stage_1。玩家可移动、可跳、可受击、HUD 显示血条与金币。
- [ ] 故意死亡 → 切到死亡画面 → 按"重开"→ 回到 stage_1 起点,敌人重置,金币重置,血量满。
- [ ] 走到 stage_1 终点 → 切到 stage_2 → stage_2 终点 → 切到胜利画面,显示用时统计。
- [ ] 胜利画面按"回主菜单"→ 主菜单"继续"按钮变亮(已存档)。
- [ ] 关游戏,重新启动,主菜单"继续"按钮仍亮,点击后回到上次存档的关卡。
- [ ] 在关卡中按 Esc 弹暂停菜单,角色立即停下;按"退出主菜单"回到主菜单不卡顿。
- [ ] 在死亡画面 / 胜利画面按 Esc 没反应(暂停只在关卡里有效)。
