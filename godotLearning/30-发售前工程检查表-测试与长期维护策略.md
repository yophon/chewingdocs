# 发售前工程检查表、测试与长期维护策略

游戏不是“我电脑上能玩”就能发售。发出去以后,玩家机器上的崩溃、坏档、手柄问题、旧版本升级问题,都会变成你的维护成本。

> 一句话先记住:**发售前要能查日志、跑测试、读旧档、复现问题。**

---

## 一、版本号先统一

项目设置里写:

```ini
[application]
config/version="0.5.0"
config/use_custom_user_dir=true
config/custom_user_dir_name="my_indie_game"
```

注意三种版本:

```text
游戏版本: 0.5.0
存档 schema_version: 例如 3
生成器 generator_version: 例如 2
```

它们不是同一个东西,但要有记录。

---

## 二、日志要打开

导出版也要写日志:

```ini
[debug]
file_logging/enable_file_logging=true
file_logging/max_log_files=10
```

玩家反馈问题时,你至少能让他找到 `user://logs/` 里的日志。不要只靠截图猜。

游戏启动时也可以打印基础信息:

```gdscript
print("Version: ", ProjectSettings.get_setting("application/config/version"))
print("User dir: ", OS.get_user_data_dir())
print("Args: ", OS.get_cmdline_args())
```

---

## 三、存档兼容性测试

每次改存档结构,都要保留旧样本:

```text
tests/fixtures/saves/v1_slot_1.json
tests/fixtures/saves/v2_slot_1.json
```

测试目标:

```text
旧档能读
缺字段有默认值
坏 JSON 不崩
主存档坏了能读 .bak
迁移后 schema_version 更新
```

这是单机游戏最容易忽略、也最容易被玩家骂的部分。

---

## 四、最小自动测试

不需要一开始测整个游戏。先测纯逻辑:

```text
伤害公式
掉落表
存档迁移
道具数据库重复 id
程序化生成同 seed 结果一致
```

测试越靠近纯函数越好。节点和场景越多,测试越脆。

---

## 五、Headless 检查脚本

写一个能在命令行跑的检查:

```gdscript
# res://tools/headless_check.gd
extends SceneTree

func _init() -> void:
    var ok := true
    ok = _check_project_settings() and ok
    ok = _check_translations() and ok
    ok = _check_levels() and ok
    quit(0 if ok else 1)

func _check_project_settings() -> bool:
    var version = ProjectSettings.get_setting("application/config/version", "")
    if str(version).is_empty():
        push_error("Missing application/config/version")
        return false
    return true

func _check_translations() -> bool:
    # 实际项目里扫描本地化 key
    return true

func _check_levels() -> bool:
    # 实际项目里加载关卡,检查 PlayerSpawn / Goal
    return true
```

命令:

```bash
godot --headless -s res://tools/headless_check.gd
```

CI 里就跑这条。

## 六、导出前清单

每个平台都过一遍:

```text
启动
新游戏
读档
死亡重开
通关结算
暂停菜单
音量设置
语言切换
手柄操作
窗口切换/全屏
退出游戏
```

不要只测快乐路径。至少故意测一次坏档、断网、手柄拔插、窗口缩放。

---

## 七、平台清单

itch.io:

```text
Windows zip 可直接运行
macOS 包权限说明
Linux 可执行权限
页面写清控制方式
上传版本号明确
```

Steam:

```text
Depot 结构
Steam Input
云存档路径
成就/统计是否接入
Deck 分辨率和手柄
商店截图和 capsule
Build review 前完整跑一遍
```

Steam Cloud 特别注意:云同步路径要和 `user://` 实际路径对应。

---

## 八、长期维护

上线后要坚持:

```text
每次发布写 changelog
每次存档结构变更写迁移
旧版本存档样本不删
崩溃日志里能看版本
补丁包能回滚
用户反馈能关联版本号
```

这不是流程洁癖。它决定你能不能在发售后继续修游戏,而不是越修越乱。

---

## 最终验收

- 导出版能写日志。
- 项目版本号明确。
- 存档迁移有旧样本。
- 有 headless 检查脚本。
- 至少关键纯逻辑有自动测试。
- Windows / macOS / Linux / Web / Android 中你承诺的平台都真实测过。
- 主菜单到通关再回主菜单完整跑通。
- 坏档、缺字体、手柄拔插都有处理策略。

---

## 常见坑

**坑 1:发售前才想起测试。**

测试最适合从存档、掉落表、生成器这种纯逻辑开始,越早越省事。

**坑 2:版本号到处不一致。**

玩家报 bug 时你会不知道他到底跑的是哪版。

**坑 3:只保留最新存档样本。**

旧档兼容要靠旧样本证明。

**坑 4:导出版不写日志。**

玩家机器出问题时你只能猜。
