# 30-发售前工程检查表、测试与长期维护策略

> 一句话导读:游戏不是"我觉得能玩了"就能发售,而是要确认它在玩家电脑上崩了能拿到日志、旧存档能读、新版本能升级、各平台合规、关键脚本有自动回归。这一篇把最后一公里拆成清单。

前 29 篇做完,你手里有了一个能跑、能扩展、可能还能联机的 2D 原型。但从"能跑"到"能上架"中间还有一段距离。这一段不性感,也没什么"心智模型"可讲,本质上是一套工程纪律:崩了能查、存档能升、版本号清楚、平台不报红、上线后还能修。本篇把这套纪律落到具体工具——GUT 单元测试、`--headless` 脚本化检查、GitHub Actions CI 骨架,以及 itch.io / Steam 发售前清单。

写到这一篇时,假设你心里已经有了一个明确的发售目标——比如先发 itch.io,半年后转 Steam,或者直接 Steam EA。不同目标会让你的发售前清单优先级完全不同:itch.io 几乎没有审核,Steam 要走 build review;itch.io 不需要代码签名,Steam 在 Windows 上对未签名 exe 越来越严;itch.io 不要求"可访问性"承诺,Steam 的 Deck Verified 把可访问性作为评分维度。本篇给的清单是两个平台的并集,你按自己路径裁剪即可。

## 1. 机制定位

独立游戏发售失败的最常见原因不是"游戏不好玩",而是发售那一刻才发现一堆事情来不及做:玩家发了崩溃截图你抓不到调用栈;1.0 卖出去三天后发现一个关键 bug,修好之后玩家的旧存档读不进新版本;你想给 Steam 提交版本号 1.0.1,但 `project.godot` 里写的还是 0.5.0;Mac 玩家说游戏一打开就被系统拦下;有玩家手柄按 Start 退不到主菜单,因为你没绑 Joypad input。

这些事都不是一次性能解决的,而是需要从开发中期就开始累积"发售前可勾选项"。等到要发售前两周才动手,基本来不及。

更糟糕的是,这些事彼此之间有依赖。你想做存档迁移,就得有版本号;你想让 CI 检查存档迁移,就得有自动化测试;你想跑自动化测试,就得有 `--headless` 入口和 GUT。任何一环没做,链就接不上。所以这一篇的写作顺序是"从最底层的运行期可观测开始,逐步往上长出自动回归 → CI → 平台合规",而不是按发售日倒推。

本篇分四块:

- **运行期可观测**:崩溃日志、版本号、版本号驱动的存档迁移、关键设置可调可保存。
- **自动回归**:GUT 单元测试装上,关键脚本有 happy-path 覆盖,改动后能用一行命令跑回归。
- **CI 雏形**:GitHub Actions 用 `--headless` 跑测试、跑导出,提交即验证,无需本地手动重复。
- **平台合规**:itch.io / Steam 发售前清单,每一条都要在发售前打勾。

新手最常见的失控写法,是把所有这些事都堆到 release candidate 那两周去做。结果版本号定不下来、存档兼容性没设计过、CI 现搭、Steamworks SDK 现接,中间发现一个 bug 又要全部重来。正确节奏是从 0.1 开始就维护一份清单,每个里程碑过几条,到 1.0 时清单接近空。

## 2. Godot 心智

Godot 4.6 在发售相关的工具链上有几条要先知道的事实:

**`OS.get_user_data_dir()` 与 `user://`。** 玩家数据(存档、设置、日志)永远写到 `user://` 开头的路径,Godot 自动映射到操作系统对应的 per-user 目录(Windows 是 `%APPDATA%/Godot/app_userdata/...`,Linux 是 `~/.local/share/godot/app_userdata/...`,macOS 是 `~/Library/Application Support/Godot/app_userdata/...`)。生产中要在 `project.godot` 里设 `application/config/use_custom_user_dir = true` 和 `application/config/custom_user_dir_name = "<your_game>"`,这样 `user://` 不会跟其他 Godot 项目共享同一个目录。

这一步在原型阶段被忽略不致命,但 1.0 上线后想再改就是大坑——一旦你的早期玩家已经把存档存到默认共享目录,改完路径之后他们的存档"凭空消失",会引来一波退款。最稳的做法是项目刚开始就把 `custom_user_dir_name` 设好,后续无论改名还是大版本升级,这个字符串都不要动。如果实在要改,在第一次启动新版本时做一次性的存档目录迁移逻辑——把旧路径下的所有文件挪到新路径,完成后留一个 `.migrated` 标志文件防止重复。

**`OS.set_low_processor_usage_mode(true)`(可选)、`OS.alert(...)`(致命错误弹窗)、`OS.get_cmdline_args()`(读启动参数,用来加 `--diag`、`--devmode` 开关)。** 这几个 API 在发售版本里都有用。

**自定义 crash handler 在 GDScript 里有局限。** Godot 引擎本身崩溃(C++ 层)时,会把 stack trace 写到 `user://logs/` 下的 `godot.log`。GDScript 层的 runtime error(数组越界、空引用)不会让进程崩,而是打到 `user://logs/godot.log` 同时弹给调试器,游戏可能继续跑或挂起。`project.godot` 的 `debug/file_logging/enable_file_logging` 控制是否往文件写。生产发布版必须开。

**项目版本号有三种,容易混。** `project.godot` 里有 `application/config/version`(玩家能在引擎里读到的字符串),导出预设里的 product version(写入 Windows .exe 元信息、Mac plist),以及你存档里自己写的 schema version(用于存档迁移)。三者要保持有意识地一致,别一个 0.5、一个 0.5.1、一个 6。

**`--headless` 模式。** 4.x 加了顶级 `--headless`,启动一个不弹窗的 Godot,跑脚本或导出。所有 CI 流水线都靠这一条。结合 `-s script.gd` 可以跑任意 GDScript;结合 `--export-release`/`--export-debug` 可以跑导出。

**GUT(Godot Unit Test)是社区主流测试框架。** 9.x 版本对 Godot 4 兼容,**9.6.0 明确标注与 4.6 兼容**(它在该版本里更新了几个内部 API 适配,并加了 `assert_push_warning` 等新断言)。安装方式是把 `addons/gut/` 复制到项目里,在 Project Settings → Plugins 里启用。测试脚本继承 `GutTest`(或老式的 `"res://addons/gut/test.gd"`),测试方法以 `test_` 开头。命令行入口是 `addons/gut/gut_cmdln.gd`,headless 跑回归测试就是它。

**测试与游戏运行时的边界。** GUT 测试默认在编辑器或独立 Godot 进程里跑,不会启动你的"完整游戏"。这意味着你的测试只能直接对类、函数、Resource 做断言,不能假设场景树存在、Autoload 已经加载。一个常见做法是把可测逻辑从节点里抽出来:伤害公式写在一个普通 `RefCounted` 子类里(`combat_math.gd`),节点只是"持有数据并调用 combat_math";这样 combat_math 自己就能被 GUT 直接 `new()` 和断言,完全跳过场景树。这也是组件化设计的副产品——它让回归测试变得现实。

## 3. 工程实现

### 3.1 项目版本号与存档迁移

`project.godot` 里加一行:

```ini
[application]

config/version="0.5.0"
config/use_custom_user_dir=true
config/custom_user_dir_name="my_indie_game"

[debug]

file_logging/enable_file_logging=true
file_logging/max_log_files=10
```

在代码里读这个版本号,作为存档 schema 版本的来源之一:

```gdscript
# res://autoload/build_info.gd
extends Node

var version: String = ProjectSettings.get_setting("application/config/version", "0.0.0")
var save_schema: int = 3  # 存档结构版本,手动递增
var build_hash: String = ""  # CI 注入,见下文

func _ready() -> void:
    print("Booting %s v%s schema=%d" % [
        ProjectSettings.get_setting("application/config/name", "game"),
        version, save_schema
    ])
```

存档读写时,先看 schema 版本,缺字段就升级:

```gdscript
# res://autoload/save_io.gd
extends Node

const SAVE_PATH := "user://save.tres"

func save(data: Dictionary) -> void:
    var res := ConfigFile.new()
    res.set_value("meta", "schema", BuildInfo.save_schema)
    res.set_value("meta", "version", BuildInfo.version)
    for k in data:
        res.set_value("game", k, data[k])
    var err := res.save(SAVE_PATH)
    if err != OK:
        push_error("save failed: %d" % err)

func load() -> Dictionary:
    var res := ConfigFile.new()
    if res.load(SAVE_PATH) != OK:
        return {}
    var schema: int = res.get_value("meta", "schema", 1)
    var out := {}
    for k in res.get_section_keys("game"):
        out[k] = res.get_value("game", k)
    return _migrate(out, schema)

func _migrate(data: Dictionary, from_schema: int) -> Dictionary:
    if from_schema < 2:
        # v1 → v2: 加了金币字段
        data["coins"] = data.get("coins", 0)
    if from_schema < 3:
        # v2 → v3: 重命名 hp → max_hp,加 cur_hp
        if data.has("hp"):
            data["max_hp"] = data["hp"]
            data["cur_hp"] = data["hp"]
            data.erase("hp")
    return data
```

存档迁移的纪律是:每次结构改变就把 `save_schema` 加 1,在 `_migrate` 里加一段 `if from_schema < N: ...` 的小迁移函数。永远不删除旧的迁移分支,因为玩家可能跨好几个版本。

迁移函数的写法有几个细节要小心。第一,字段重命名永远先复制再删除,顺序错了就丢数据。第二,字段类型变更(从 `int` 改成 `float` 或者反过来)要写显式转换,不要靠 GDScript 的隐式转型,因为 Variant 的某些边界值会触发不期望的行为。第三,如果你删了一个字段,迁移函数里要显式 `data.erase("old_field")`,否则后续代码读它仍然能读到,可能造成"幽灵字段"——版本走到 5 了,你还在维护 v2 删过的字段。

存档的兼容方向是单向的:**新版本能读旧存档,旧版本不一定能读新存档**。这意味着如果玩家在 1.2 玩了一个角色然后回退到 1.1,存档可能直接打不开,甚至崩溃。1.0 之后任何强制更新都应该附带"建议玩家不要回退"的说明,真的需要支持回退就把迁移做成对称的——但这会让维护成本翻倍,独立游戏一般不值得做。

### 3.2 装上 GUT 并写第一个测试

把 GUT 9.6+ 的 `addons/gut/` 复制进项目,在 Project Settings → Plugins 里启用。然后建 `res://test/unit/test_save_io.gd`:

```gdscript
# res://test/unit/test_save_io.gd
extends GutTest

var SaveIO = load("res://autoload/save_io.gd")

func test_migrate_v1_to_v3_renames_hp() -> void:
    var io = SaveIO.new()
    var migrated: Dictionary = io._migrate({"hp": 80}, 1)
    assert_eq(migrated.get("max_hp"), 80)
    assert_eq(migrated.get("cur_hp"), 80)
    assert_false(migrated.has("hp"))
    assert_eq(migrated.get("coins"), 0)
    io.free()

func test_migrate_v3_is_noop() -> void:
    var io = SaveIO.new()
    var data := {"max_hp": 100, "cur_hp": 50, "coins": 10}
    var migrated: Dictionary = io._migrate(data, 3)
    assert_eq(migrated, data)
    io.free()

func test_empty_dict_gets_defaults() -> void:
    var io = SaveIO.new()
    var migrated: Dictionary = io._migrate({}, 1)
    assert_eq(migrated.get("coins"), 0)
    io.free()
```

命令行跑一次:

```bash
godot --headless -d -s --path "$(pwd)" \
    addons/gut/gut_cmdln.gd \
    -gdir=res://test/unit -gexit
```

输出会显示 pass / fail 计数,任何测试失败进程返回非 0 退出码——这是 CI 决定流水线红绿的依据。

### 3.3 一个最低 CI 骨架(GitHub Actions)

在仓库根加 `.github/workflows/ci.yml`,描述"每次推 PR 跑测试 + 试一次 Linux 导出":

```yaml
name: ci
on:
  push:
    branches: [main]
  pull_request:

jobs:
  test-and-export:
    runs-on: ubuntu-latest
    env:
      GODOT_VERSION: 4.6.3
    steps:
      - uses: actions/checkout@v4

      - name: Install Godot
        run: |
          curl -L -o godot.zip \
            "https://github.com/godotengine/godot-builds/releases/download/${GODOT_VERSION}-stable/Godot_v${GODOT_VERSION}-stable_linux.x86_64.zip"
          unzip godot.zip
          mv Godot_v${GODOT_VERSION}-stable_linux.x86_64 /usr/local/bin/godot
          chmod +x /usr/local/bin/godot

      - name: Install export templates
        run: |
          curl -L -o tpl.tpz \
            "https://github.com/godotengine/godot-builds/releases/download/${GODOT_VERSION}-stable/Godot_v${GODOT_VERSION}-stable_export_templates.tpz"
          mkdir -p ~/.local/share/godot/export_templates/${GODOT_VERSION}.stable
          unzip tpl.tpz -d /tmp/tpl
          mv /tmp/tpl/templates/* ~/.local/share/godot/export_templates/${GODOT_VERSION}.stable/

      - name: Run GUT tests
        run: |
          godot --headless -d -s --path "$PWD" \
            addons/gut/gut_cmdln.gd -gdir=res://test/unit -gexit

      - name: Export Linux build
        run: |
          mkdir -p build/linux
          godot --headless --export-release "Linux/X11" build/linux/game.x86_64

      - uses: actions/upload-artifact@v4
        with:
          name: linux-build
          path: build/linux/
```

这套骨架做的事很有限——它不签名、不上传 Steam、不打 macOS notarization。但它能保证:推上去之后,测试有红绿可看;每次 PR 都能拿到一个能 在 Linux 上运行的可执行,你可以下载下来给测试者用。后续按需要加 macOS、Windows、Web 的 matrix,或加 butler push 到 itch.io,都是在这个骨架上长出来。

往这个骨架上加 itch.io 自动发布,只需要再加一个 step:`curl -L https://broth.itch.zone/butler/linux-amd64/LATEST/archive/default -o butler.zip && unzip butler.zip && chmod +x butler && ./butler push --userversion ${GITHUB_REF_NAME} build/linux <user>/<game>:linux`。`BUTLER_API_KEY` 放在 GitHub repo 的 Secrets 里,workflow 里通过 `env: BUTLER_API_KEY: ${{ secrets.BUTLER_API_KEY }}` 注入。Steam 上传则要 `steamcmd`,这是 Valve 自己的命令行工具,流程更长、更脆,建议手动跑前几次稳定后再考虑放进 CI。

注入构建信息也是 CI 阶段做的:在导出之前用 `sed` 或一段小 GDScript 把 `build_info.gd` 里的 `build_hash` 写成 `${GITHUB_SHA::8}`。这样你拿到玩家的崩溃日志时,能立刻看到这是哪一次 commit 的产物。

### 3.4 设置菜单的最低骨架

许多平台审核(尤其是 Steam Deck Verified)要求:玩家可以重绑键、可以调音量、可以切窗口/全屏。最低实现就是一个 `Settings` 单例配合 `ConfigFile`:

```gdscript
# res://autoload/settings.gd
extends Node

const PATH := "user://settings.cfg"
var data := {
    "master_volume": 1.0,
    "music_volume": 1.0,
    "sfx_volume": 1.0,
    "fullscreen": false,
    "vsync": true,
}

func _ready() -> void:
    load_from_disk()
    apply()

func load_from_disk() -> void:
    var cf := ConfigFile.new()
    if cf.load(PATH) != OK:
        return
    for k in data:
        data[k] = cf.get_value("settings", k, data[k])

func save() -> void:
    var cf := ConfigFile.new()
    for k in data:
        cf.set_value("settings", k, data[k])
    cf.save(PATH)

func apply() -> void:
    AudioServer.set_bus_volume_db(0, linear_to_db(data["master_volume"]))
    DisplayServer.window_set_mode(
        DisplayServer.WINDOW_MODE_FULLSCREEN if data["fullscreen"]
        else DisplayServer.WINDOW_MODE_WINDOWED
    )
    DisplayServer.window_set_vsync_mode(
        DisplayServer.VSYNC_ENABLED if data["vsync"]
        else DisplayServer.VSYNC_DISABLED
    )
```

输入重绑参见第 27 篇本地化与输入设备,这里不重复。

## 4. 调参和验收

**版本号节奏。** SemVer 不强制,但建议至少做到"破坏存档结构就升 minor、bug 修复升 patch、内容大更新升 minor"。`save_schema` 是另一条平行的整数,只在存档结构变时升。所有 release notes 用 `version` 而不是 `save_schema`,玩家不关心后者。

**`max_log_files=10` 够不够。** Godot 每次启动写一份日志,旧的滚动删。10 份对独立游戏够,真要查 Bug 收日志时让玩家把 `user://logs/` 整个目录打包给你。

**CI 跑多久算合理。** 这个最低骨架的 Linux 部分大概 3-5 分钟(下载 Godot + 跑测试 + 导出),加 macOS + Windows + Web 后 10-15 分钟。如果你超过 30 分钟,看看测试是不是被你写得太重(用 `_physics_process` 真跑了关卡循环),或者 Godot 二进制重复下载没做缓存。

**测试覆盖什么、不覆盖什么。** 不要追求覆盖率。重点覆盖:存档迁移(改起来高频、错起来用户怒)、关键数据驱动逻辑(配表读取、伤害公式、掉落概率)、纯数学/算法(寻路、随机种子、生成器)。不要覆盖:具体节点位置、动画过渡时间、UI 像素布局——这些用人眼测更高效。

**测试运行速度的隐性预算。** 单元测试要能在本地 10 秒以内全部跑完,不然你不会愿意每次改完代码都跑一次,自动回归的意义就丢了。如果某个测试要 `await get_tree().create_timer(2.0).timeout`,先想想能不能用 mock(把时间源抽成参数)替掉。GUT 9.6+ 支持 `double()` 给单例打桩,`Input`、`Time`、`OS` 都能被替换成 mock,以便测试函数立刻断言而不必等真实信号。

### 4.1 发售前清单(itch.io / Steam 共用)

**项目层**
- [ ] `project.godot` 的 `application/config/version` 与导出预设里 product version、Steam depot 版本号一致。
- [ ] `application/config/use_custom_user_dir` = true,`custom_user_dir_name` 是项目专属字符串。
- [ ] `debug/file_logging/enable_file_logging` = true,`max_log_files` ≥ 5。
- [ ] `application/config/name` 不是默认 "Game",是最终对外名称。
- [ ] 图标 `application/config/icon` 是 256×256 PNG,Windows 导出预设单独配 `.ico`。

**运行期**
- [ ] 任意时刻 Alt+F4 / 关窗口都能存档(或者明确告诉玩家"只在存档点存盘")。
- [ ] 程序崩溃后下次启动不会丢主存档(主+备双份策略,或 ConfigFile 原子写)。
- [ ] 设置菜单覆盖:主音量、音乐、SFX、窗口模式、Vsync、键鼠/手柄重绑(至少跳跃、攻击、交互)。
- [ ] 启动到主菜单不超过 5 秒(冷启动);卡顿超过 1 秒就显示进度。

**测试与回归**
- [ ] `godot --headless ... gut_cmdln.gd ... -gexit` 在 main 分支退出码 0。
- [ ] 存档迁移有至少 3 个版本之间的迁移单测(模拟跨版本读旧档)。
- [ ] CI 在 PR 上跑通至少一个平台的导出 artifact。

**Windows**
- [ ] 导出的 `.exe` 在 Windows 10 / 11 上不弹 "未知发布者" 风险高警告(签名是最优解,但独立游戏最低也要确认 SmartScreen 不直接拦截运行;如果有预算就买代码签名证书)。

## 5. 踩坑

**`ProjectSettings.get_setting("application/config/version")` 在编辑器里能拿到字符串,但在导出可执行里如果 `project.godot` 里没显式写 `config/version=...`,会返回 `""`。** 不要依赖 Godot "自动有版本号",手动写一个。

**`OS.alert()` 在 `--headless` 下不弹窗,但在某些 Linux server CI 上会阻塞。** 不要在游戏启动逻辑里用它当"调试 print",生产代码里只用于真正的致命错误。

**进程崩溃后的"留下证据"策略。** Godot 在 GDScript 层报 runtime error 时,会把 stack trace 写到 `user://logs/godot.log`。但如果引擎本身崩了(C++ 层段错误、显卡驱动崩溃),日志可能只写了一半。生产中建议另外做一件事:在游戏入口处把"上一次启动时间 + 当前会话开始时间"写进一个 `user://session.json`,游戏正常退出时清空这个文件;启动时发现文件存在,说明上次是崩溃退出,可以弹一个"很抱歉游戏崩了,要不要把日志发给开发者"的提示。这一招在玩家不主动报 bug 的常态下,能多收回 60-70% 的崩溃案例。

**`user://` 在 macOS App Store 沙盒模式下路径不一样。** 自己玩 macOS 直接发可执行没问题;一旦要进 Mac App Store(独立开发者基本不去),`user://` 会被映射到容器内目录,旧版本玩家迁移过来的存档需要单独处理。

**GUT 在 Godot 4.6 上要用 9.6+。** 9.5.x 在 4.6 上会有几个内部 API 报警告(部分还报错),具体是 `pause_before_teardown` 在 headless 下被改成自动跳过。如果你卡在 9.5,升级到 9.6 是最稳的修法。

**GUT 测试里 `await` 行为。** GUT 测试函数可以是 `func test_xxx() -> void` 也可以 `func test_xxx() -> void` 内部用 `await`,但必须 yield 到 `process_frame` 或者其他 signal。直接 `await get_tree().create_timer(1.0).timeout` 会让 headless CI 多跑 1 秒——避免在测试里"等动画结束",改成"立刻验证状态机数据"。

**`--export-release` 失败时返回码不一定是 1。** 它可能因为模板缺失返回特定错误码,或者直接卡住等待 stdin。所有 CI 里跑 export 都要加 `--no-window` / `--headless`,并检查输出文件是否存在,而不是仅看退出码。

**Windows 导出后第一次运行,Defender SmartScreen 弹"未知发布者"。** 这不是 Godot 的问题,是所有未签名的 exe 都会遇到。买代码签名证书(EV cert 几百到上千美元一年)能彻底消除;独立开发者也可以选择不签,在玩家文档里写明"右键 → 属性 → 解除阻止"。

**Mac notarization 不做的话,新 macOS 用户右键打开也越来越难。** Apple 在 14+ 加强了 Gatekeeper,unsigned binary 直接双击会被拦,只能"右键 → 打开"才能放行。一旦你要在 Steam 上架,Steam Mac 客户端会绕过部分 Gatekeeper,但 itch.io 直接下载不会。Notarization 需要 Apple Developer 账号($99/年)+ `xcrun notarytool` 上传。

**`butler push` 第一次会要登录。** 在 CI 里你需要在 secrets 里放 `BUTLER_API_KEY`,然后 `butler push --api-key $BUTLER_API_KEY ...`。不要把 API key 写进 yaml。

**Steam app id 在 `steam_appid.txt` 里。** Steamworks SDK 要求可执行同目录下有这个文件,内容是你的 app id 整数。开发期忘了放会让 Steam 集成无声失败;发布到 Steam 时 Steamworks 自动注入,所以不要把这个文件打进 release zip(避免覆盖)。

**别在 1.0 发售当天升 Godot minor。** 如果你的项目从 4.6.0 一路开发,临到发售看到 4.7 出来,千万别为了"新特性"在最后一周升。版本升级触发的潜在回归会让你睡不着觉。锁定一个 4.6.x 走完整个发售周期,下个版本再考虑升 4.7。

**1.0 发售后的两周是工程最危险的时期。** 真实玩家用真实硬件跑你的游戏,会暴露所有你测试期间没遇到的边界:奇怪的分辨率、ARM 翻译层、五年前的集显、非英文用户名路径、各种第三方杀软。你需要的不是"完美 1.0",而是"能在两小时内热修一个 1.0.1 并推送到所有商店"的流程。开发期就把 hot-patch 路径走通(版本号自增、CI 重跑、butler push + Steam build review)比 1.0 多打磨两周更值得。

**Bug 报告通道要在游戏内置。** Steam 的论坛、itch.io 的 comments、Discord、邮件,玩家会从最低阻力的渠道反馈,如果你只在 Steam 看,Discord 用户提的关键 bug 你三周看不到。最稳的做法是游戏主菜单加一个"反馈"按钮,直接拷贝 `user://logs/` 路径 + 当前版本号到剪贴板,再打开你的反馈 URL。这一步成本不到一天但收益巨大。

**`.gitignore` 里别把 `export_presets.cfg` 加进去。** Godot 项目模板默认把它 ignore 掉,因为里面可能有签名密码字段。但 CI 需要这个文件做导出,你要么把它从 ignore 里删掉(密码字段用环境变量替换,不要写明文),要么在 CI 里临时生成一份。两条路都行,但必须有意识地选一条。
