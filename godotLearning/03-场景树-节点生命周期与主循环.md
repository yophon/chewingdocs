# 场景树、节点生命周期与主循环

Godot 里很多"玄学 bug",本质都不是玄学,而是**你不知道节点什么时候真的活了、什么时候已经快死了、每一帧谁先跑谁后跑**。比如 `$Player` 是 null、`_ready` 里拿不到子节点、敌人 `queue_free()` 后还被访问、暂停菜单停不住游戏、玩家按菜单按钮时角色也攻击了。这些问题看起来分散,底层全是同一件事:**SceneTree 在调度节点**。

这一篇只讲 Godot 运行时最核心的模型:**场景树怎么驱动节点**。后面所有移动、动画、碰撞、UI、场景切换,都会建立在这套顺序上。

> 一句话先记住:**Godot 不是每帧乱序调用一堆 update,而是 SceneTree 按规则通知树上的节点;节点只有进了树,生命周期回调才会跑**。没进树的对象只是普通对象,不是游戏世界的一员。

---

## 一、SceneTree 是游戏运行时的骨架

Godot 项目运行后,引擎会创建一个 `SceneTree`。你的主场景、Autoload、临时实例化出来的敌人,最终都挂在这棵树上。

简化后大概是这样:

```text
root (Window)
├── GameLog          # Autoload
├── SaveManager      # Autoload
└── Main             # 当前主场景 current_scene
    ├── World
    │   ├── Player
    │   ├── Enemy
    │   └── TileMapLayer
    ├── UI
    │   └── HUD
    └── Camera2D
```

`SceneTree` 负责几件事:

- 把节点加进树、移出树
- 调用节点生命周期回调
- 每帧调用 `_process`
- 每个物理 tick 调用 `_physics_process`
- 分发输入事件
- 管暂停状态
- 切换当前场景

所以 Godot 的核心规则是:

> **只有在 SceneTree 里的节点,才会被引擎调度。**

这段代码创建了一个敌人,但它还不是游戏世界的一员:

```gdscript
var enemy := Enemy.new()
enemy.hp = 10
```

它不会 `_ready`,不会 `_process`,不会显示,不会参与场景树。必须加进去:

```gdscript
world.add_child(enemy)
```

如果是场景资源:

```gdscript
const ENEMY_SCENE: PackedScene = preload("res://enemies/slime.tscn")

func spawn_enemy() -> void:
    var enemy: Node2D = ENEMY_SCENE.instantiate()
    world.add_child(enemy)
```

> 一句话判断:**`new()` / `instantiate()` 只是创建对象,`add_child()` 才是让它进入游戏世界**。

## 二、节点生命周期:谁先谁后

`Node` 最重要的几个回调:

| 回调 | 什么时候触发 | 常见用途 |
| --- | --- | --- |
| `_enter_tree()` | 节点进入 SceneTree 时 | 很少用,做底层注册 |
| `_ready()` | 节点和子节点都准备好后 | 初始化引用、连接信号 |
| `_process(delta)` | 每个渲染帧 | UI、视觉效果、非物理计时 |
| `_physics_process(delta)` | 固定物理 tick | 移动、碰撞、物理状态 |
| `_unhandled_input(event)` | 输入没被 UI 吃掉时 | 玩家控制、暂停快捷键 |
| `_exit_tree()` | 节点离开树时 | 断开连接、清理状态 |

最容易搞错的是 `_enter_tree()` 和 `_ready()` 的顺序。

假设场景是:

```text
Parent
├── ChildA
└── ChildB
```

进入树时:

```text
Parent._enter_tree()
ChildA._enter_tree()
ChildB._enter_tree()

ChildA._ready()
ChildB._ready()
Parent._ready()
```

也就是:

- `_enter_tree`:父先于子
- `_ready`:子先于父

这就是为什么大多数初始化都写在 `_ready()`。因为父节点的 `_ready()` 执行时,子节点已经 ready 了,你可以安全拿:

```gdscript
@onready var sprite: Sprite2D = $Sprite2D
@onready var hit_box: Area2D = $HitBox

func _ready() -> void:
    sprite.visible = true
    hit_box.body_entered.connect(_on_body_entered)
```

但在 `_enter_tree()` 里拿子节点就不稳:

```gdscript
func _enter_tree() -> void:
    # 不推荐:子节点可能还没完整进树
    $Sprite2D.visible = true
```

> 实战规则:**普通业务初始化写 `_ready()`,不要急着用 `_enter_tree()`**。除非你非常清楚自己为什么需要更早的时机。

## 三、`@onready` 到底解决什么

Godot 里你经常会写:

```gdscript
@onready var sprite: Sprite2D = $Sprite2D
@onready var animation_player: AnimationPlayer = $AnimationPlayer
```

`@onready` 的意思是:**等节点 ready 前再求值**。因为脚本字段初始化时,节点可能还没进树,这时 `$Sprite2D` 不一定能取到。

不要这样写:

```gdscript
var sprite: Sprite2D = $Sprite2D
```

这行可能在节点还没进入场景树时执行,直接报错。

三种常见写法:

```gdscript
# 1. 固定子节点,推荐
@onready var sprite: Sprite2D = $Sprite2D

# 2. 节点路径稍长,但还在自己子树内,可以接受
@onready var weapon_socket: Node2D = $Body/WeaponSocket

# 3. 跨很远找节点,危险信号
@onready var hud: Control = $"../../UI/HUD"
```

第三种要警惕。路径越长,说明依赖越脆弱。更好的做法通常是:

- 父节点把引用传进来
- 使用导出变量手动指定
- 用信号通知
- 用 `%UniqueName` 找同场景内的唯一节点
- 让场景结构重新变简单

> `@onready` 不是让你放心写超长路径的工具。它只解决"什么时候取节点"的问题,不解决"依赖关系是否合理"的问题。

## 四、`_process` 和 `_physics_process` 怎么选

Godot 有两条循环:

```text
渲染帧:      _process(delta)          跟显示帧率走
物理 tick:   _physics_process(delta)  固定频率,默认 60 Hz
```

区别很重要。

`_process(delta)` 的 `delta` 会随帧率变化:

```text
60 FPS  → delta 约 0.016
30 FPS  → delta 约 0.033
144 FPS → delta 约 0.0069
```

`_physics_process(delta)` 默认固定:

```text
60 Hz → delta 约 0.01667
```

选择规则:

| 逻辑 | 放哪里 |
| --- | --- |
| 玩家移动 | `_physics_process` |
| `move_and_slide()` | `_physics_process` |
| 碰撞检测相关 | `_physics_process` |
| AI 追踪和位置更新 | 通常 `_physics_process` |
| UI 闪烁、按钮动画 | `_process` |
| 背景滚动、不影响碰撞的视觉 | `_process` |
| 计时器 | 优先 Timer 或 Tween,简单场景可 `_process` |

典型角色移动:

```gdscript
extends CharacterBody2D

@export var speed: float = 180.0

func _physics_process(delta: float) -> void:
    var direction := Input.get_axis("move_left", "move_right")
    velocity.x = direction * speed
    move_and_slide()
```

典型视觉效果:

```gdscript
@export var bob_speed: float = 4.0
@export var bob_height: float = 3.0

var _time := 0.0

func _process(delta: float) -> void:
    _time += delta
    $Sprite2D.position.y = sin(_time * bob_speed) * bob_height
```

> 硬规则:**会影响物理世界的,放 `_physics_process`;只影响显示的,放 `_process`**。

## 五、输入:为什么 UI 打开后角色还在攻击

Godot 输入常用三个入口:

| 回调 | 含义 |
| --- | --- |
| `_input(event)` | 输入一进来就收到 |
| `_gui_input(event)` | `Control` UI 节点自己的输入 |
| `_unhandled_input(event)` | 没被 UI 或其它节点处理掉的输入 |

新手常犯的错:玩家控制写在 `_input()` 里。结果暂停菜单打开后,你点按钮或按确认键,角色也跟着攻击/跳跃。

更稳的规则:

```text
UI 控件处理输入     → _gui_input / Control 自带信号
全局快捷键         → _input
玩家和游戏世界输入 → _unhandled_input
```

玩家暂停:

```gdscript
func _unhandled_input(event: InputEvent) -> void:
    if event.is_action_pressed("ui_cancel"):
        get_tree().paused = not get_tree().paused
        get_viewport().set_input_as_handled()
```

UI 按钮被点时,`Control` 会先处理输入。处理掉的事件不会再进入 `_unhandled_input()`。这样玩家不会在点菜单时继续攻击。

> 一句话:**游戏世界少用 `_input`,优先用 `_unhandled_input`。UI 吃掉输入后,世界就不该再响应。**

## 六、暂停不是一个全局 if

很多人会写:

```gdscript
var paused := false

func _process(delta: float) -> void:
    if paused:
        return
```

这能跑,但项目一大就乱。Godot 已经有暂停机制:

```gdscript
get_tree().paused = true
```

节点是否在暂停时继续运行,看 `process_mode`:

| process_mode | 暂停时行为 | 用途 |
| --- | --- | --- |
| `PROCESS_MODE_INHERIT` | 跟随父节点 | 默认 |
| `PROCESS_MODE_PAUSABLE` | 暂停时停止 | 玩家、敌人、关卡 |
| `PROCESS_MODE_WHEN_PAUSED` | 只有暂停时运行 | 暂停菜单 |
| `PROCESS_MODE_ALWAYS` | 永远运行 | 少量全局管理器 |
| `PROCESS_MODE_DISABLED` | 永不处理 | 临时禁用 |

典型结构:

```text
Main
├── World               # pausable
│   ├── Player
│   └── Enemy
└── UI
    ├── HUD             # pausable 或 always,看需求
    └── PauseMenu       # when paused
```

打开暂停菜单:

```gdscript
func show_pause_menu() -> void:
    get_tree().paused = true
    pause_menu.visible = true
    pause_menu.process_mode = Node.PROCESS_MODE_WHEN_PAUSED
```

关闭:

```gdscript
func hide_pause_menu() -> void:
    pause_menu.visible = false
    get_tree().paused = false
```

> 规则:**暂停应该交给 SceneTree 和 process_mode,不要在每个脚本里散落 `if paused`**。

## 七、`queue_free()` 不是立刻消失

删除节点常用:

```gdscript
enemy.queue_free()
```

`queue_free()` 的意思是:**把节点排队,等当前帧安全时释放**。不是这一行执行完节点立刻从内存消失。

为什么不立刻删?因为 SceneTree 可能正在遍历节点。如果遍历到一半你把节点硬删了,顺序会乱,引用也可能炸。

常见坑:

```gdscript
enemy.queue_free()
print(enemy.global_position) # 不要继续依赖它
```

更好的写法是先取需要的数据,再释放:

```gdscript
var death_position := enemy.global_position
enemy.queue_free()
spawn_drop(death_position)
```

如果你保存了别人的引用,访问前要检查:

```gdscript
if is_instance_valid(enemy):
    enemy.take_damage(10)
```

> `queue_free()` 后,你的心智里就应该把这个节点当成"已经不能依赖"。需要的数据提前拿出来。

## 八、场景切换时谁会留下

当前场景通常是 `get_tree().current_scene`。切场景:

```gdscript
get_tree().change_scene_to_file("res://scenes/level_01.tscn")
```

会发生什么?

```text
旧 current_scene 退出树并释放
新场景实例化
新场景成为 current_scene
Autoload 继续存在
```

也就是说:

| 对象 | 切场景后 |
| --- | --- |
| 当前关卡 | 会被释放 |
| 玩家如果在关卡下面 | 会被释放 |
| HUD 如果在关卡下面 | 会被释放 |
| Autoload | 不会释放 |
| `user://` 存档文件 | 不受影响 |

这就是为什么 Autoload 适合放:

- 存档入口
- 音频总线控制
- 全局配置
- 场景流管理器
- 少量跨场景状态

不适合放:

- 当前关卡里所有敌人
- 玩家所有具体行为
- UI 细节
- 临时掉落物

> 判断标准:**切场景后还应该活着的,才考虑 Autoload。随着关卡一起死的,就放在场景树里。**

## 九、节点引用怎么写才不脆

最脆的写法:

```gdscript
@onready var hud = $"../../UI/HUD"
```

只要层级一变,立刻断。

更稳的几种方式:

### 1. 只引用自己的直接子节点

```gdscript
@onready var sprite: Sprite2D = $Sprite2D
@onready var hit_box: Area2D = $HitBox
```

这是最稳的。场景内部结构自己管。

### 2. 用导出变量让父节点注入

```gdscript
@export var target: Node2D
```

编辑器里拖进去。适合镜头跟随目标、敌人巡逻点这类引用。

### 3. 用信号反向通知

Player 不要直接找 HUD:

```gdscript
signal health_changed(current: int, max_value: int)
```

HUD 或父节点去连接它。

### 4. 用 `%UniqueName`

在同一个场景里,右键节点开启 "Access as Unique Name",然后:

```gdscript
@onready var player: Player = %Player
```

这比 `$World/Actors/Player` 稳。只要这个唯一名还在同一个场景里,节点挪位置也不容易断。

> 节点引用的优先级:**自己子节点 > 导出引用 > 信号 > `%UniqueName` > 短路径 > 长路径**。长路径是最后选择。

## 十、做一个生命周期探针

为了直观看顺序,写一个探针脚本:

```gdscript
# res://tools/lifecycle_probe.gd
class_name LifecycleProbe
extends Node

@export var tag: String = ""
@export var print_process: bool = false
@export var print_physics: bool = false

var _process_count: int = 0
var _physics_count: int = 0

func _enter_tree() -> void:
    print("[%s] _enter_tree" % _tag())

func _ready() -> void:
    print("[%s] _ready" % _tag())

func _process(delta: float) -> void:
    _process_count += 1
    if print_process and _process_count % 60 == 0:
        print("[%s] _process delta=%.4f" % [_tag(), delta])

func _physics_process(delta: float) -> void:
    _physics_count += 1
    if print_physics and _physics_count % 60 == 0:
        print("[%s] _physics_process delta=%.4f" % [_tag(), delta])

func _exit_tree() -> void:
    print("[%s] _exit_tree" % _tag())

func _tag() -> String:
    if tag != "":
        return tag
    return name
```

建一个测试场景:

```text
ProbeRoot
├── ChildA
│   └── GrandA
└── ChildB
```

每个节点都挂 `LifecycleProbe`,分别填 tag:

```text
root
A
GA
B
```

运行后你会看到类似:

```text
[root] _enter_tree
[A] _enter_tree
[GA] _enter_tree
[B] _enter_tree
[GA] _ready
[A] _ready
[B] _ready
[root] _ready
```

这比背规则更有用。以后遇到顺序问题,先挂探针看真实顺序。

## 十二、踩坑提醒

1. **`instantiate()` 后忘了 `add_child()`**  
   对象创建了,但没进场景树。不会显示,不会 ready,不会 process。

2. **在字段初始化里直接 `$Node`**  
   用 `@onready`,否则节点还没进树时就取子节点,容易报错。

3. **把物理移动写进 `_process()`**  
   帧率一变,手感就变。`move_and_slide()` 放 `_physics_process()`。

4. **玩家输入写 `_input()` 导致 UI 和游戏同时响应**  
   游戏世界输入优先 `_unhandled_input()`。

5. **到处散写 `if paused`**  
   用 `SceneTree.paused` 和 `process_mode`。

6. **`queue_free()` 后继续使用节点**  
   要用的数据先存下来,或者用 `is_instance_valid()` 检查。

7. **跨很多层拿节点**  
   长 NodePath 是重构炸弹。优先信号、导出引用、唯一名。

8. **把所有跨场景东西都塞 Autoload**  
   Autoload 不会随场景释放,滥用会变成第二个全局垃圾桶。
