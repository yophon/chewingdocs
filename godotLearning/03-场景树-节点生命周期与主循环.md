# 03-场景树、节点生命周期与主循环

> 一句话导读:Godot 的每一帧不是"调用所有 update 函数",而是"按场景树的顺序通知所有需要处理的节点"。理解这棵树怎么被遍历、怎么被切分成 idle 帧和 physics 帧、怎么在重命名节点后保持引用稳定,是写出不抖动、不竞态、不脆弱的游戏代码的前提。

写一个移动方块,大多数引擎都能做到。难的是当工程变成"一个 `Main` 场景挂着 `World`、`World` 下有 `Player`、`Player` 下有 `Sprite2D` 和 `Camera2D`、敌人是 `instantiate()` 出来 `add_child` 进 `World` 的"以后,下面这些问题怎么回答:玩家的 `_ready` 和 `Camera2D` 的 `_ready` 谁先调用,敌人 `queue_free()` 时 `Player` 还能不能拿到它的位置,关卡切换时 Autoload 单例会不会被销毁,把 `Camera2D` 从 `Player` 下挪到 `World` 下,所有引用它的代码会不会全部红屏。这些问题的答案都在主循环和场景树的细节里,Godot 4.6 又往这里塞了一项叫 Unique Node IDs 的内部机制,把"重命名节点引用就断"这个老问题修掉了。

这一篇把场景树和主循环讲透,把 4.6 的 Unique Node IDs 与已有的 `%UniqueName` 区分清楚,后续每篇涉及"谁先谁后""怎么引用""怎么暂停"的细节都会落到这一篇打的基础上。

读完这一篇,你应当能回答上面这几个问题中的每一个;并且对"我什么时候应该写在 `_ready` 里、什么时候写在 `_process` 里、什么时候根本不需要写在生命周期里"有一个稳定的判断,而不是凭直觉撒。

## 1. 机制定位

主循环要解决的核心问题是**调度**:这一帧有哪些对象需要被通知"该处理你了",顺序怎么定,处理多久后该进入下一帧。简单的实现把所有"活的对象"放在一个数组里,每帧 for 一遍调用它们的 update。这种实现的两个缺陷在游戏工程里会持续制造 bug。

第一是**顺序不确定**。父子对象的处理顺序在数组实现里依赖于"谁先被加进来",这让"摄像机要跟着玩家走"这种简单需求变得脆弱:如果摄像机的 update 跑在玩家之前,它读到的是上一帧的玩家位置,画面跟随会延迟一帧。简单 demo 看不出来,关卡切换时新创建的摄像机如果碰巧排在玩家前面就抖一下,排查起来很痛苦。

第二是**生命周期边界模糊**。对象被销毁时它的 update 还会不会被调用,被销毁时它持有的引用怎么处理,跨对象的"我等你 init 完再用你"怎么表达。简单数组实现里这些都靠手动 if 判断,代码量爆炸。

游戏引擎对这两件事的标准回答是**场景图**(scene graph):用一棵显式的树管理对象,树的遍历顺序定义"谁先谁后",节点从树里加进或移出时触发明确的生命周期回调,父节点死掉时子节点跟着死。Unity 的 GameObject 层级、Unreal 的 Actor 层级、Cocos 的 Node 树都是这套思路的不同实现。Godot 的 `SceneTree` 把这一套做到了相对极致:**一切运行时对象都必须挂在树上才会被主循环看到**,这条规则没有例外。

新手最容易踩的坑是**绕开树**。比如 `var enemy = Enemy.new()` 后直接调用它的方法,期待它自动更新——这个 enemy 不在树上,它的 `_ready` 永远不会被调用,`_process` 永远不会跑,信号也连不上,它就是个静态对象。再比如 `queue_free()` 之后立刻在下一行读节点的属性,期待它"还活着一会儿"——`queue_free` 把删除推迟到本帧末,但访问已经标记为"将被删除"的节点拿到的状态可能是不一致的。这些坑都来自"没把 SceneTree 当成主循环骨架去理解",把 Godot 当成普通 OOP 框架用。

## 2. Godot 心智

把主循环拆开,Godot 的运行时执行模型由下面几个对象协同完成。

**`MainLoop` 与 `SceneTree`**。进程启动时引擎创建一个 `MainLoop` 实例,默认是它的子类 `SceneTree`。`SceneTree` 维护根节点 `root`(类型 `Window`)、当前场景 `current_scene`、Autoload 单例,以及一系列调度状态(`paused`、`physics_interpolation_enabled`、各种 frame 计数)。它不是一个普通的容器,它是引擎和你的代码之间的**唯一**接口:每次引擎要"通知一个节点",都通过 SceneTree 找到那个节点。

**节点的生命周期回调**。`Node` 类定义了一组虚函数,你在脚本里实现哪个,引擎就在对应时机调用哪个,顺序在 4.6 文档里有官方描述。下面这张表是后续每一篇都会用到的速查:

| 回调 | 触发时机 | 顺序 |
| --- | --- | --- |
| `_enter_tree()` | 节点(或其父)被加入 SceneTree 时 | 父先于子(top-down) |
| `_ready()` | 节点的所有子节点都进树后,且该节点本身也进树后,在下一个空闲点 | 子先于父(bottom-up) |
| `_process(delta)` | 每一渲染帧 | 按 `process_priority`,默认父先于子 |
| `_physics_process(delta)` | 每一物理 tick(默认 60 Hz) | 同上 |
| `_exit_tree()` | 节点从 SceneTree 移除前 | 子先于父(bottom-up) |
| `_unhandled_input(event)` | 输入事件冒泡到 viewport 时 | 反向 tree 遍历,直到被 `accept()` |

重点是 `_enter_tree` 和 `_ready` 的方向相反这件事:**`_enter_tree` 是父先**(因为引擎从根开始往下加,父进树时子还没在),**`_ready` 是子先**(因为父需要等所有子都 ready 后才被算作 ready)。这意味着在 `_ready` 里访问子节点是**安全的**(它们已经 ready),但在 `_enter_tree` 里访问子节点**不安全**(它们可能还没进树)。这一条记牢,后续 80% 的"为什么我的子节点是 null"问题都来自违反它。

**`delta` 的两种含义**。`_process(delta)` 里的 `delta` 是"距离上一渲染帧的实际秒数",会随帧率波动,可能是 0.016(60fps)也可能是 0.032(30fps)。`_physics_process(delta)` 里的 `delta` 是"物理 tick 的固定步长",在默认 60Hz 下永远是 1.0/60 ≈ 0.01667,不随实际帧率变化。这件事的工程后果是:**任何与物理相关的更新写在 `_physics_process` 里**,否则在帧率波动时角色移动速度会不稳定;**任何只影响显示、不影响碰撞与状态的更新写在 `_process` 里**,否则你白白把渲染逻辑限制在 60Hz。

**`process_mode` 与暂停**。`SceneTree.paused = true` 让游戏暂停,但"暂停"对每个节点的实际含义由它的 `process_mode` 决定:`PROCESS_MODE_INHERIT`(默认,跟随父节点)、`PROCESS_MODE_PAUSABLE`(暂停时停止)、`PROCESS_MODE_WHEN_PAUSED`(暂停时才运行,典型场景是暂停菜单)、`PROCESS_MODE_ALWAYS`(永不停止,典型场景是 Autoload)、`PROCESS_MODE_DISABLED`(永不运行)。这个机制让"我想要暂停时菜单还能用"成为一个声明式选项,不需要全局 flag 判断。

**Autoload 与 `SceneTree`**。Autoload 是 Project Settings 里注册的脚本,引擎启动时会把它们作为 `root` 的子节点先于 `current_scene` 加进树。意味着 Autoload 永远是 `current_scene` 的兄弟,而不是父亲;它们的 `_ready` 在场景切换前就已经触发,后续场景切换也不会重新触发。第 16 篇会讲 Autoload 的使用边界。

**`_input` 与 `_unhandled_input`**。这两个回调常被新手混淆。`_input(event)` 在事件刚被读到时就调用一次,遍历顺序是节点树倒序(子先于父);`_unhandled_input(event)` 在事件没有被任何节点 `accept_event()` 时调用。**规则:UI 类节点的输入响应写 `_input`(它们要"截胡"事件,不让事件继续传递到游戏世界),游戏逻辑节点的输入响应写 `_unhandled_input`**。这样玩家在打开背包菜单时按下"攻击键"不会触发攻击,UI 自动吸收了输入。第 07 篇会展开。

**Tween 与 SceneTreeTimer 的生命周期归属**。Godot 4.x 里 `create_tween()` 创建的 Tween、`get_tree().create_timer()` 创建的 SceneTreeTimer 都是挂在场景树上的临时对象,且默认在所属节点退出场景树时被自动清理。这意味着你不需要手动管理它们的生命周期,节点死了它们跟着死。但要注意一点:如果你在 Autoload 里 `create_tween()`,Autoload 永不退出树,Tween 也永不释放,长时间运行会累积。规则:**Tween 应该挂在它服务的对象上,而不是 Autoload 上**,这样对象死掉时 Tween 自动收尾。

接下来讲 4.6 的新东西:**Unique Node IDs**。这是 4.6 引入的引擎内部机制,在 `Node` 上分配一个 `int32` 的内部 ID,**不**暴露到脚本 API,仅引擎内部使用。它解决的问题是:在继承场景或实例化场景里,如果你重命名/移动了基场景里的一个节点,以前继承场景对它的修改会因为 NodePath 失配而丢失。4.6 用 Unique Node IDs 替代部分 NodePath 路径匹配,让继承场景在基场景节点改名后仍能找到对应节点。

这件事对工程实践有两个直接影响。其一,**重命名节点不再是高风险动作**,你可以放心地把 `Sprite2D` 改成 `Sprite_Body`,继承自这个场景的所有变体都能正确更新引用。其二,这个机制只对**保存在场景文件里**的引用关系生效,运行时通过 `get_node("Sprite2D")` 字符串路径拿节点的代码不受保护,改了名字代码照样断。

要让 4.6 之前创建的项目享受这个机制,需要在 4.6 编辑器里执行 Project → Tools → Upgrade Project Files,把所有 `.tscn` 重存一次,Unique Node IDs 才会被写入文件。

**注意区分**:Unique Node IDs(4.6,引擎内部 int32)和 **Scene Unique Nodes**(更早版本就有,用户层 `%name` 语法)是两个不同的机制,名字像但作用层级不同:

- **Unique Node IDs**:引擎内部使用,你看不到、改不了,作用是让继承/实例场景在节点改名后仍正确链接。
- **Scene Unique Nodes (`%name`)**:用户层 API,在场景里选中节点 → 右键 → "Access as Unique Name",然后代码里用 `%Player` 取代 `$World/Player`,这样后续无论你怎么把 Player 移到树的别处,只要它还在同一个场景文件里,`%Player` 都能找到它。

两者配合起来,Godot 4.6 项目的节点引用稳定性达到了一个新的水平:**编辑器内的节点重命名/移位被 Unique Node IDs 兜底,代码里的引用被 `%name` 语法兜底**,你只需要避免在代码里写硬编码的长 NodePath。

这里值得多说一句"信号连接路径"如何受 Unique Node IDs 影响。Godot 在 `.tscn` 里序列化信号连接时,记录的是源节点 NodePath 与目标节点 NodePath 加方法名。4.6 之前如果你重命名了源节点或目标节点,连接会因为路径失配而失效,编辑器里能看到一个红色感叹号;4.6 之后,引擎内部用 Unique Node IDs 重新链接连接关系,大多数路径漂移场景都能自动修正。但**代码里 `signal.connect(callable)` 这种运行时连接不受保护**——运行时连接只在内存里,信号源销毁或对象释放后自然断开,不存在"重命名后丢失"问题。所以这个 4.6 改动只对"在 `.tscn` 里通过编辑器拖出来的信号"生效,代码连的信号本来就不脆弱。

## 3. 工程实现

写一个能在控制台观察生命周期与主循环行为的小场景,放在 `res://tools/lifecycle_probe.tscn` 和 `res://tools/lifecycle_probe.gd`。这是一个本系列后续每篇都会用到的调试工具。

先写一段通用的"探针"脚本 `res://tools/lifecycle_probe.gd`:

```gdscript
# res://tools/lifecycle_probe.gd
class_name LifecycleProbe
extends Node

## 把生命周期与主循环回调全部打到控制台。
## 用法:任意节点上挂一份这个脚本,运行时观察顺序。

@export var tag: String = ""
@export var print_process: bool = false
@export var print_physics: bool = false

var _process_count: int = 0
var _physics_count: int = 0

func _enter_tree() -> void:
    GameLog.d(_tag(), "_enter_tree, parent=%s" % [get_parent().name if get_parent() else "<root>"])

func _ready() -> void:
    GameLog.d(_tag(), "_ready, children=%d" % get_child_count())

func _process(delta: float) -> void:
    _process_count += 1
    if print_process and _process_count % 60 == 0:
        GameLog.d(_tag(), "_process #%d delta=%.4f" % [_process_count, delta])

func _physics_process(delta: float) -> void:
    _physics_count += 1
    if print_physics and _physics_count % 60 == 0:
        GameLog.d(_tag(), "_physics_process #%d delta=%.4f" % [_physics_count, delta])

func _exit_tree() -> void:
    GameLog.d(_tag(), "_exit_tree, _process called %d times, _physics %d times"
        % [_process_count, _physics_count])

func _tag() -> String:
    return "Probe:%s" % (tag if tag != "" else name)
```

然后做一个测试场景 `res://tools/lifecycle_probe.tscn`,层级是:

```text
ProbeRoot (Node) [挂 LifecycleProbe.gd, tag="root"]
├── ChildA (Node) [挂 LifecycleProbe.gd, tag="A"]
│   └── GrandA (Node) [挂 LifecycleProbe.gd, tag="GA"]
└── ChildB (Node) [挂 LifecycleProbe.gd, tag="B"]
```

把它挂到 `Main` 场景下,F5 运行,控制台输出形如:

```text
[ 0.10][DEBUG][Probe:root] _enter_tree, parent=Main
[ 0.10][DEBUG][Probe:A]    _enter_tree, parent=ProbeRoot
[ 0.10][DEBUG][Probe:GA]   _enter_tree, parent=ChildA
[ 0.10][DEBUG][Probe:B]    _enter_tree, parent=ProbeRoot
[ 0.10][DEBUG][Probe:GA]   _ready, children=0
[ 0.10][DEBUG][Probe:A]    _ready, children=1
[ 0.10][DEBUG][Probe:B]    _ready, children=0
[ 0.10][DEBUG][Probe:root] _ready, children=2
```

`_enter_tree` 是 root → A → GA → B(父先,深度优先),`_ready` 是 GA → A → B → root(子先,后序遍历)。把这个顺序记住,以后调试"我父节点 ready 时子节点还没初始化"或者"我子节点 enter tree 时父节点的 @onready 还没生效",直接能定位。

接下来写一段演示 Scene Unique Nodes 的代码。在 `res://player/player.tscn` 里建一个简单的玩家场景:

```text
Player (CharacterBody2D)
├── Sprite (Sprite2D)              [Access as Unique Name = on]
├── CollisionShape2D
├── HitBox (Area2D)                [Access as Unique Name = on]
│   └── HitBoxShape (CollisionShape2D)
└── Camera (Camera2D)              [Access as Unique Name = on]
```

在 Player 节点的 `_ready` 里就可以这样写引用:

```gdscript
# res://player/player.gd
class_name Player
extends CharacterBody2D

## 玩家根节点。展示 % 引用与 _ready 顺序。

# 用 % 而不是 $,后续无论我把 Sprite/HitBox/Camera 移到 Player 子树的什么位置,
# 这些引用都不会断。前提是它们都在同一个 .tscn 里,且勾了 Access as Unique Name。
@onready var sprite: Sprite2D = %Sprite
@onready var hitbox: Area2D = %HitBox
@onready var camera: Camera2D = %Camera

func _ready() -> void:
    GameLog.i("Player", "ready, sprite=%s hitbox=%s camera=%s"
        % [sprite.name, hitbox.name, camera.name])
    # 把摄像机变成活跃摄像机
    camera.make_current()
    # 接 hitbox 的 area_entered 信号到本节点的方法
    hitbox.area_entered.connect(_on_hitbox_area_entered)

func _on_hitbox_area_entered(other: Area2D) -> void:
    GameLog.i("Player", "hitbox hit by %s" % other.name)
```

这段代码体现两个工程惯例。其一,**用 `%Name` 取代 `$LongPath/To/Node`**,牺牲一点点性能换取重构稳定性,值得。其二,**信号连接写在 `_ready` 里**,因为这是"所有子节点都 ready 了"的唯一时机,在 `_enter_tree` 里 `hitbox` 还可能是 null,在 `_init` 里更不行。

最后写一段 `Main` 场景的小升级,演示**节点动态创建与销毁**:

```gdscript
# res://scenes/main.gd 增量
const ENEMY_SCENE: PackedScene = preload("res://enemies/enemy.tscn")

func spawn_enemy(at: Vector2) -> void:
    var enemy: Node2D = ENEMY_SCENE.instantiate()
    enemy.position = at
    world.add_child(enemy)
    # add_child 会触发 enemy 的 _enter_tree → 各子节点 _enter_tree
    #                              → 各子节点 _ready → enemy _ready
    GameLog.i("Main", "spawned enemy at %s" % at)

func despawn_all_enemies() -> void:
    for child in world.get_children():
        if child.is_in_group("enemy"):
            child.queue_free()
            # queue_free 不立刻销毁,它在本帧末才真正释放
            # 同一帧里继续访问 child 的属性还是可以的,但下一帧它就不在树上了
```

`queue_free` 与立即 `free` 的区别要记牢:`free()` 立刻销毁,如果当前帧还有别的代码持有这个节点的引用并访问它,会崩溃。`queue_free()` 把销毁推迟到本帧末的"空闲时刻",在销毁前所有 `_process`/`_physics_process` 调用都已经完成,所以是安全的。**几乎所有场合都用 `queue_free`,只有你 100% 确认没有其它代码持有引用时才用 `free`**。

## 4. 调参和验收

这一篇能调的参数主要在 Project Settings → Physics 与 Display 里:

- **`physics/common/physics_ticks_per_second`**:默认 60,意味着 `_physics_process` 每秒被调用 60 次。降到 30 能减少物理 CPU 但 `move_and_slide` 在快速对象上更容易穿透,升到 120 让物理更精确但 CPU 翻倍。本系列 2D 游戏统一保持 60。
- **`physics/common/physics_jitter_fix`**:默认 0.5。当渲染帧率与物理 tick 频率不匹配时,引擎会对 `_process` 看到的物理状态做插值平滑。0 = 不平滑(直接看到 tick 后的离散状态),1 = 全部平滑。0.5 是个保守的中间值,2D 项目可以保持默认。
- **`physics/common/physics_interpolation`**:4.4 引入、4.6 仍可用的物理插值,默认关。开启后低帧率下移动看起来更平滑,代价是物理状态读取多一次插值开销。本系列在第 06 / 24 篇深入讨论,这里保持默认关。
- **节点 `process_priority`**:默认 0,数字越小越先调用。如果你确实需要"摄像机一定要在玩家之后跑",把 Camera2D 的 `process_priority` 设为 1 或更大,但优先考虑用 `%Player` 引用 + 在 `_process` 末尾读 `Player.position`,而不是依赖优先级,后者难以追溯。
- **节点 `process_mode`**:默认 `PROCESS_MODE_INHERIT`,跟随父节点。需要"暂停时仍响应"的节点(暂停菜单、Autoload 里的全局事件总线)显式设为 `PROCESS_MODE_ALWAYS` 或 `PROCESS_MODE_WHEN_PAUSED`。

验收标准:

- [ ] `LifecycleProbe` 树挂上后,F5 控制台中 `_enter_tree` 与 `_ready` 顺序符合上文表格(父先入树、子先 ready)。
- [ ] `Player` 场景里用 `%Sprite` / `%HitBox` 引用,即使把 `Sprite2D` 从 Player 直接子节点拖到一个新建的 `Visuals` Node2D 下面成为孙节点,代码不需要任何修改仍能跑通。

## 5. 踩坑

第一类是**在 `_init` 或 `_enter_tree` 里访问子节点**。`_init` 是 GDScript 构造函数,这时节点还没挂到任何树上,子节点也不存在;`_enter_tree` 是父先入树,你的子节点还没进树。这两个时机访问 `$Child` 一律拿到 null。规则:**几乎所有"需要子节点已经准备好"的逻辑都写在 `_ready` 里**,只有"对外部参数赋值"才写在 `_init`。

第二类是**用 `$LongPath/To/Node` 而不是 `%Name`**。`$World/Player/Sprite/Visual/AnimatedSprite` 这种长路径在重构时几乎一定会断,且很多人为了能编译会去手动 `get_node_or_null` + null check,代码变得冗余。改成 `%AnimatedSprite`,把该节点在 Owner 里设 unique,后续把它移到子树任何位置都不影响。**唯一例外**是跨场景引用,`%` 只在同一个 `.tscn` 里生效,跨场景拿节点要用信号或者 Autoload。

第三类是 **4.6 项目升级时忘记 Upgrade Project Files**。从 4.5 升到 4.6 的项目,如果不执行这一步,旧 `.tscn` 文件里没有 Unique Node IDs 信息,你以为已经享受了 4.6 的引用稳定性,实际上没有。具体表现是继承场景在改名后引用还是会丢。如果发现这种现象,先检查 `.tscn` 是否经过 4.6 重存。

第四类是**在 `_process` 里改物理状态**。比如在 `_process` 里调用 `velocity = ...; move_and_slide()`,在帧率波动时角色会移动速度不稳;在 `_process` 里读 `position`,如果同时有 `_physics_process` 在改它,读到的可能是中间态。规则:**物理读写一律 `_physics_process`,渲染相关读取一律 `_process`**。

第五类是 **`queue_free` 后立刻 `is_instance_valid` 仍返回 true**。`queue_free` 把销毁排队,但本帧内对象还在,`is_instance_valid` 检查不到这个状态。如果你在 `Player` 里持有一个对 enemy 的引用,enemy `queue_free` 后下一帧 `Player._process` 里访问它会拿到 null 或者已释放对象。安全做法是引用对象前用 `is_instance_valid(ref) and not ref.is_queued_for_deletion()` 双重检查,或者用信号让 enemy 在 `_exit_tree` 时主动 `emit` 通知 Player 把引用置空。

第六类是**误以为 Autoload 跟着场景切换销毁**。Autoload 节点直接挂在 `root` 下,与 `current_scene` 是兄弟。`get_tree().change_scene_to_file()` 只换 `current_scene`,Autoload 一直存活。这是好事(全局状态保留),但要意识到 Autoload 里的状态在场景切换时不会被清空,如果你想"新关卡清空所有 buff",要在 Autoload 里写显式的 reset 方法并在场景切换时调用。

第七类是**`queue_free` 与 `change_scene_to_file` 抢顺序**。在某个节点的回调里 `queue_free()` 当前场景的根节点,同一帧里又 `change_scene_to_file()`,会出现一帧里有两个 current_scene 的混乱状态。Godot 推荐用 `call_deferred("change_scene_to_file", path)` 把切换推到下一帧,或者直接信任 `change_scene_to_file` 自己的实现(它内部已经做了相应的延迟)。

第八类是**`await get_tree().process_frame` 当万能延迟**。新手发现 `await` 可以等"下一帧",会习惯用它替代各种时序问题:"我先 `add_child`,await 一下让它 ready,再访问"。这种写法在小项目能跑,但每个 await 都意味着函数返回、main loop 继续推进,中间任何状态变化都可能让你 await 完拿到的对象已经不是你以为的对象了。规则:**`await` 用于"我真的要等一个明确的信号或时点",不要用它绕过时序设计**。如果你发现自己在写"`await` 几帧然后访问子节点",通常正确做法是把那段逻辑挪到 `_ready` 或者改成由子节点 emit 的信号触发。
