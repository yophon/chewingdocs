# 22-程序化关卡与可控随机

> 一句话导读:程序化生成把"关卡"从静态资产变成一段可复现的程序,关键不在"随机",而在"可控"。

独立游戏经常把程序化关卡(procedural content generation, PCG)当作低成本扩展内容的捷径,但真正使用过的人会发现,失败的 PCG 比手搓关卡更难维护:每次跑出来的关都不一样,玩家死在哪一个坑里你查不到,平衡性参数怎么调都说不清,Bug 报告也无法复现。问题不在"随机太多",而在没有把随机管理好。这一篇讲 Godot 4.6 下 PCG 的工程地基:`RandomNumberGenerator`、`FastNoiseLite`、可复现的种子流程,以及两种最常用的 2D 关卡生成范式——元胞自动机洞穴和房间走廊。

噪声本身的数学(Perlin、Simplex、cellular)已经在 mathForCS 系列写过,本篇只讲 Godot 这一侧的工程落地:把噪声当成一个"位置 → 数值"的纯函数,关心怎么调它、怎么测它、怎么和别的随机源协同。换句话说,这是工程师视角的 PCG——不去推导"为什么 Simplex 比 Perlin 更各向同性",而是关心"怎么让设计师改一个参数就能预览效果,怎么让 QA 在 issue 里贴一个种子号你能完整复现这张图"。这两个能力是 PCG 工程化的底线。

## 1. 机制定位

PCG 在 2D 独立游戏里通常解决三个问题:

- **内容扩展**:roguelike、矿洞游戏、生存沙盒需要近乎无限的关卡,人力撑不过来。
- **重复可玩性**:同一份关卡每次玩都长得不一样,通关后的二周目不会立刻枯燥,这是 Spelunky、Hades、Risk of Rain 之所以耐玩的核心机制之一。
- **节省美术成本**:用规则生成洞穴、地形、装饰,可以省下手搓 Tile 的数百小时,让一人或小团队能做出"看上去内容很多"的游戏。

但 PCG 是把双刃剑。手搓关卡好处是"所见即所得",每一个坑、每一处节奏点都是设计师亲手放的;PCG 拿到的是分布,而不是确定的体验。一旦生成质量失控,游戏的设计意图就被算法稀释掉了。所以工程上的目标不是"会用 PCG",而是"用了 PCG 还能保持设计师的掌控感"——这要求两件事:**生成过程必须确定可控**(同样的输入永远产生同样的输出),**生成参数必须可视可调**(改一个数值能立刻看到差异)。

新手最常踩的坑,集中在三件事上:

第一,**直接用 `randi()` / `randf()`**。这两个是全局 PRNG 函数,内部状态被整个项目共享,UI 抖动、AI 决策、关卡生成都用同一个流,任何一处改动都会把后续关卡完全打乱。结果就是:你修了个 UI bug,昨天那张地图再也回不来了。更糟的是 QA 反馈"种子 12345 在第三关墙撞穿了",你在本地跑出来根本不是同一张图——因为生成器跑之前有个动画播放调用了 `randf()`,把流推进了一两步。

第二,**生成函数不是纯函数**。把 `Time.get_ticks_msec()`、节点位置、`get_tree().get_nodes_in_group()` 混进生成逻辑,生成结果就跟外部状态绑死,同一个种子第二次跑出来的地图不一样。线上玩家说"我在 (32, 18) 掉坑里了",你重现不出来,也就没法修。生成函数应当像数学函数一样:给定 seed 和参数,输出唯一确定。任何"外部读取"都要先变成显式输入。

第三,**噪声参数靠玄学**。`frequency = 0.05`、`octaves = 4` 抄网上的样例,一个值改成 0.1 整张地图就糊掉,改回去又找不到原始数字。没有版本化的参数表,平衡性根本谈不上。设计师也无法做"今天调地形,明天调宝箱密度"的差异化迭代,因为每一次改动都要赌一次"看会不会塌"。

工程上要做的事很简单:**生成器只接受一个 `int` seed 和一份显式参数,只输出确定性结果;参数全部 `@export` 并放在 `Resource` 里;每个独立的随机用途用独立的 `RandomNumberGenerator` 实例。** 这三条做到,PCG 就从"玄学"变成"代码",可以测试、可以 diff、可以做版本回滚。剩下的"好不好玩",才回归到设计师的判断。

## 2. Godot 心智

Godot 4.6 在程序化生成上提供的核心工具有这些:

| 类 / 函数 | 用途 | 是否纯函数 |
| --- | --- | --- |
| `@GlobalScope.randi()` / `randf()` | 全局快速随机 | 共享状态,**不推荐**用于生成 |
| `@GlobalScope.seed(base: int)` | 给全局 PRNG 重置种子 | 影响所有 `randi()` |
| `RandomNumberGenerator` | 独立 PRNG 实例,可保存 / 恢复 `state` | 实例隔离,推荐 |
| `FastNoiseLite` | 多种噪声算法(OpenSimplex2、Perlin、Cellular、Value) | 纯函数,`get_noise_2d(x, y)` 完全由 `seed + 参数` 决定 |
| `NoiseTexture2D` | 把 `Noise` 烘成 `Texture2D`,内部用线程异步生成 | 见 24 篇 |
| `TileMapLayer.set_cell()` | 把生成结果写进 Tile 地图 | 见 11 篇 |

心智上要分清两层:

**随机源层**。这是"无中生有"的层,每次调用返回一个数。`RandomNumberGenerator` 是一个有状态的对象,内部用 PCG 算法(注意这里的 PCG 是 Permuted Congruential Generator,跟"程序化生成"重名巧合),你可以读 / 写它的 `state` 把它"快照"下来。把不同用途的随机源分开(地形 RNG、宝箱 RNG、敌人 RNG),互不污染。如果用单一全局流,任意一处插入或移除一次 `randi()` 调用,都会让后续所有"随机"结果全部错位——这就是"种子稳定但生成不稳定"的根因。

**确定性映射层**。这是"位置 → 值"的层,典型代表是 `FastNoiseLite.get_noise_2d(x, y)`,同样的 seed + 同样的 (x, y) 永远返回同样的值。它没有"下一个数"的概念,你可以并发查询任意坐标,顺序无关。这层最适合用来生成"有空间相关性"的数据:地形高度、温度场、洞穴初始密度。它和 RNG 不是替代关系,而是互补:RNG 适合"按顺序决定一串离散事件",噪声适合"在 2D 空间里铺一张平滑/不平滑的场"。

两者各有各的用法:洞穴用元胞自动机时,初始噪声场用 `FastNoiseLite`(或 `RandomNumberGenerator` 也行,因为元胞自动机的初始状态不要求空间平滑),迭代规则不需要 RNG;房间走廊时,房间坐标和走廊方向用 `RandomNumberGenerator.randi_range()` 顺序采样;地形大世界生成用 `FastNoiseLite` 多层 fractal 叠加(山脉、河流、洞穴一层一层往里挖)。

**关键认知:不要把 `RandomNumberGenerator` 当成 "更高级的 randi()"**。它真正的价值是隔离与可恢复——你可以在 `_ready()` 里 `var terrain_rng = RandomNumberGenerator.new(); terrain_rng.seed = level_seed`,之后无论别的代码做什么,terrain 这条流都不会被打乱。这种"作用域内的局部随机"思路,跟 Rust 里把不变量收敛到类型上是一个工程哲学:能在编译期/初始化期把"哪些状态会变"约束清楚,后期维护就轻松。

另一个心智是:**生成结果与 Godot 节点系统解耦**。生成器最好返回纯数据(二维数组、字典、自定义类),而不是直接 `add_child` 一堆节点。原因有三:① 数据可以序列化、可以 diff、可以单元测试;② 渲染表达和生成逻辑解耦,同一张地图可以用 TileMap、Sprite 或 MeshInstance 渲染;③ 生成可以挪到 `WorkerThreadPool` 上跑(24 篇),节点系统不是线程安全的,如果生成器直接操作节点你就上不了线程。这一点在初期看不出价值,到第一次需要"加载页"的时候就关键了。

## 3. 工程实现

下面三段代码是同一个最小 PCG 工程的三个分层。约定目录:

```
demo/
  scripts/
    pcg/
      level_seed.gd            # 种子分发
      cave_generator.gd        # 元胞自动机洞穴
      dungeon_generator.gd     # 房间 + 走廊
```

把三层分开有具体的工程原因:`level_seed.gd` 是纯数据,跨关卡共用;两个 generator 互不知情,可以并行开发;接 TileMap 的代码是"渲染适配层",负责把抽象网格落到具体节点。这一套分层的好处在团队协作时立刻体现:程序写生成算法,关卡设计师调参数 `.tres`,谁也不会改到谁。

第一段,种子分发器。把一个"关卡主种子"派生成多个"用途子种子"——这是 PCG 工程化里最值钱的一招,后面所有的随机都从这里取,而不是各自 `randomize()`。继承 `RefCounted` 不继承 `Node` 是因为种子分发器不需要在场景树里有位置,也不需要 `_process`,纯逻辑对象。`RefCounted` 还能享受引用计数自动释放,不需要手动 `free()`。

文件:`demo/scripts/pcg/level_seed.gd`

```gdscript
class_name LevelSeed
extends RefCounted

## 关卡主种子,任何一处程序化逻辑都从这里派生,不直接调用全局 randi()。
var master_seed: int = 0

func _init(seed_value: int = 0) -> void:
    master_seed = seed_value if seed_value != 0 else Time.get_unix_time_from_system() as int

## 为某个"用途"派生一个独立 RNG。
## 用途名(如 "terrain", "loot", "ai")作为字符串 hash 混入,
## 同名调用永远返回同一条流。
func make_rng(purpose: String) -> RandomNumberGenerator:
    var rng := RandomNumberGenerator.new()
    rng.seed = master_seed ^ purpose.hash()
    return rng

## 为某个用途派生一个 FastNoiseLite。
## seed 由主种子和用途派生,frequency 等参数留给调用者配置。
func make_noise(purpose: String) -> FastNoiseLite:
    var noise := FastNoiseLite.new()
    noise.seed = master_seed ^ purpose.hash()
    return noise

## 把状态保存进字典(用于存档)。
## 注意:RNG 实例需要存档时,各调用方各自存 state,不在这里集中管理。
func to_dict() -> Dictionary:
    return {"master_seed": master_seed}

static func from_dict(data: Dictionary) -> LevelSeed:
    return LevelSeed.new(data.get("master_seed", 0) as int)
```

第二段,元胞自动机洞穴生成。流程:① 用一个噪声场决定哪些格子初始是墙;② 反复迭代"邻居墙数 >= N 就变墙,否则变空地",直到形态稳定;③ 输出一张 `Array[Array[int]]` 给 TileMap 层调用方。

文件:`demo/scripts/pcg/cave_generator.gd`

```gdscript
class_name CaveGenerator
extends RefCounted

const TILE_WALL: int = 1
const TILE_FLOOR: int = 0

@export var size: Vector2i = Vector2i(64, 48)
@export var fill_threshold: float = 0.45  # 噪声大于此值的格子初始为墙
@export var iterations: int = 4
@export var birth_limit: int = 5  # 周围墙 >= 此数,空地变墙
@export var death_limit: int = 4  # 周围墙 < 此数,墙变空地

var _grid: Array[PackedInt32Array] = []

func generate(level_seed: LevelSeed) -> Array[PackedInt32Array]:
    var noise: FastNoiseLite = level_seed.make_noise("cave_init")
    noise.noise_type = FastNoiseLite.TYPE_VALUE  # 纯随机感,后面交给 CA 平滑
    noise.frequency = 1.0  # 每个格子独立采样,不要平滑过渡
    _seed_grid(noise)
    for _i in iterations:
        _step()
    return _grid

func _seed_grid(noise: FastNoiseLite) -> void:
    _grid.clear()
    for y in size.y:
        var row := PackedInt32Array()
        row.resize(size.x)
        for x in size.x:
            # get_noise_2d 返回 [-1, 1],归一化到 [0, 1]
            var v: float = (noise.get_noise_2d(x, y) + 1.0) * 0.5
            row[x] = TILE_WALL if v > fill_threshold else TILE_FLOOR
        _grid.append(row)

func _step() -> void:
    var next: Array[PackedInt32Array] = []
    for y in size.y:
        var row := PackedInt32Array()
        row.resize(size.x)
        for x in size.x:
            var n: int = _count_wall_neighbors(x, y)
            if _grid[y][x] == TILE_WALL:
                row[x] = TILE_WALL if n >= death_limit else TILE_FLOOR
            else:
                row[x] = TILE_WALL if n >= birth_limit else TILE_FLOOR
        next.append(row)
    _grid = next

func _count_wall_neighbors(cx: int, cy: int) -> int:
    var count: int = 0
    for dy in [-1, 0, 1]:
        for dx in [-1, 0, 1]:
            if dx == 0 and dy == 0:
                continue
            var x: int = cx + dx
            var y: int = cy + dy
            # 边界外当成墙,洞穴会自然封闭
            if x < 0 or y < 0 or x >= size.x or y >= size.y:
                count += 1
            elif _grid[y][x] == TILE_WALL:
                count += 1
    return count
```

这段代码的几个工程要点:

`PackedInt32Array` 比 `Array[int]` 在大网格上快得多,因为它是连续存储的二进制 buffer,GDScript 4 对它做了字节码优化。64×48 还感觉不到,256×256 上面差几倍。大世界生成必须用 Packed 系列,不要图省事用 `Array`。

`fill_threshold` 直接控制洞穴空旷程度。`0.45` 大约 45% 墙,适合"洞窟"感;`0.55` 接近迷宫,空地碎片化;低于 `0.4` 几乎全空。这个参数和噪声类型也强相关:`TYPE_VALUE` 是每格独立采样,跟丢硬币没差别;改成 `TYPE_SIMPLEX` 会让相邻格倾向于相同,初始就形成连续斑块,反而不需要太多迭代。两者风格不同,前者更"洞穴",后者更"大陆"。

迭代次数不是越多越好。3-5 次最自然,10 次以上会把所有小细节抹平,变成几个大圆形,这是 CA 的"死亡螺旋"——它在数学上是个滤波器,过滤次数越多越接近低频形态。一个调试技巧是把每一次迭代都输出成一张 PNG(用 `Image.create` + `Image.set_pixel` 拼图),你能直观看到形态收敛的过程,比"对着数字猜"快得多。

第三段,房间走廊生成。和 CA 完全不同的范式:CA 是"先撒种子再演化",这是"先采样再连接"。Roguelike 经典做法,代表作 Rogue、Dwarf Fortress、Brogue。这套范式的特点是"结构感强、连通性高"——每个房间是一个矩形,玩家一眼能看出"这是空间";走廊是 L 形,导航直观。CA 洞穴则是有机感、随机感强,玩家在里面会有"探索未知"的感觉,代价是结构不可读、迷路概率高。两者通常结合使用:外层洞穴 + 内嵌房间 = 矿洞底下的废墟,这是动作 roguelike 常见的混合模式。

文件:`demo/scripts/pcg/dungeon_generator.gd`

```gdscript
class_name DungeonGenerator
extends RefCounted

const TILE_WALL: int = 1
const TILE_FLOOR: int = 0

@export var size: Vector2i = Vector2i(80, 60)
@export var room_count: int = 12
@export var room_min: Vector2i = Vector2i(4, 4)
@export var room_max: Vector2i = Vector2i(10, 8)
@export var max_attempts_per_room: int = 30

class Room:
    var rect: Rect2i
    func center() -> Vector2i:
        return rect.position + rect.size / 2

func generate(level_seed: LevelSeed) -> Dictionary:
    var rng: RandomNumberGenerator = level_seed.make_rng("dungeon_layout")
    var grid: Array[PackedInt32Array] = _make_filled_grid()
    var rooms: Array[Room] = []
    for _i in room_count:
        var room := _try_place_room(rng, rooms)
        if room != null:
            rooms.append(room)
            _carve_room(grid, room.rect)
    # 依次连接相邻房间,保证全连通
    for i in range(1, rooms.size()):
        _carve_corridor(grid, rng, rooms[i - 1].center(), rooms[i].center())
    return {"grid": grid, "rooms": rooms}

func _make_filled_grid() -> Array[PackedInt32Array]:
    var grid: Array[PackedInt32Array] = []
    for y in size.y:
        var row := PackedInt32Array()
        row.resize(size.x)
        for x in size.x:
            row[x] = TILE_WALL
        grid.append(row)
    return grid

func _try_place_room(rng: RandomNumberGenerator, existing: Array[Room]) -> Room:
    for _attempt in max_attempts_per_room:
        var w: int = rng.randi_range(room_min.x, room_max.x)
        var h: int = rng.randi_range(room_min.y, room_max.y)
        var x: int = rng.randi_range(1, size.x - w - 1)
        var y: int = rng.randi_range(1, size.y - h - 1)
        var candidate := Room.new()
        candidate.rect = Rect2i(x, y, w, h)
        var inflated: Rect2i = candidate.rect.grow(1)  # 留 1 格缓冲,房间不贴
        var overlapped: bool = false
        for r in existing:
            if inflated.intersects(r.rect):
                overlapped = true
                break
        if not overlapped:
            return candidate
    return null  # 30 次都没塞下,放弃这间

func _carve_room(grid: Array[PackedInt32Array], rect: Rect2i) -> void:
    for y in range(rect.position.y, rect.position.y + rect.size.y):
        for x in range(rect.position.x, rect.position.x + rect.size.x):
            grid[y][x] = TILE_FLOOR

func _carve_corridor(grid: Array[PackedInt32Array], rng: RandomNumberGenerator,
        a: Vector2i, b: Vector2i) -> void:
    # L 形走廊,50% 概率先水平后垂直
    if rng.randf() < 0.5:
        _carve_horizontal(grid, a.x, b.x, a.y)
        _carve_vertical(grid, a.y, b.y, b.x)
    else:
        _carve_vertical(grid, a.y, b.y, a.x)
        _carve_horizontal(grid, a.x, b.x, b.y)

func _carve_horizontal(grid: Array[PackedInt32Array], x1: int, x2: int, y: int) -> void:
    for x in range(min(x1, x2), max(x1, x2) + 1):
        grid[y][x] = TILE_FLOOR

func _carve_vertical(grid: Array[PackedInt32Array], y1: int, y2: int, x: int) -> void:
    for y in range(min(y1, y2), max(y1, y2) + 1):
        grid[y][x] = TILE_FLOOR
```

调用方很短,接 TileMapLayer 也很短(11 篇讲过):

```gdscript
extends Node2D

@export var level_seed_value: int = 0

func _ready() -> void:
    var ls := LevelSeed.new(level_seed_value)
    var cave := CaveGenerator.new()
    var grid := cave.generate(ls)
    # 把 grid 写进 TileMapLayer
    var layer: TileMapLayer = $TileMapLayer
    for y in grid.size():
        for x in grid[y].size():
            var atlas_coords: Vector2i = (
                Vector2i(0, 0) if grid[y][x] == CaveGenerator.TILE_FLOOR
                else Vector2i(1, 0)
            )
            layer.set_cell(Vector2i(x, y), 0, atlas_coords)
```

这一处用的是 `TileMapLayer.set_cell(coords, source_id, atlas_coords)`,4.6 的标准 API(参考 11 篇)。生成结果不依赖任何外部状态,固定 `level_seed_value` 就能复现整张图。

## 4. 调参和验收

PCG 的调参和别的系统不一样:**你要先建立"参数 → 视觉效果"的直觉,再谈"好不好玩"**。调参之前先承认一件事——你看不见所有可能性。一个有 6 个参数、每个 5 档的生成器,组合空间有 1.5 万种,你不可能逐个试。所以调参不是搜索最优值,而是建立"参数变化对应视觉变化的方向感",最后留下 3-5 个 preset 给设计师选。

**洞穴生成器**:

| 参数 | 取值范围 | 视觉效果 |
| --- | --- | --- |
| `fill_threshold` | 0.40 - 0.55 | 0.45 是经典平衡点,洞穴大、连通好 |
| `iterations` | 3 - 6 | 少于 3 噪点明显,多于 6 形态过圆 |
| `birth_limit` | 4 - 6 | 越大越倾向产生孤岛 |
| `death_limit` | 3 - 5 | 越小墙越脆弱,洞越大 |

**房间走廊**:

| 参数 | 取值范围 | 视觉效果 |
| --- | --- | --- |
| `room_count` | 8 - 20 | 太少空旷,太多塞不下 |
| `room_min` / `room_max` | (3,3)-(10,8) | 小一点节奏快,大一点像 BOSS 厅 |
| `max_attempts_per_room` | 20 - 50 | 越大越接近"塞满",但耗时线性增长 |

调参流程上,推荐写一个"种子扫描"工具:固定参数,对种子 1-100 各跑一遍,把结果输出成缩略图网格(用 `Image` 拼接成一张大图)。一眼看过去就能发现"哪些种子跑出来形态崩了"。然后选 5-10 个有代表性的种子作为"参考集",每次改参数都拿这套种子重新跑一遍对比新旧效果。这是从机器学习借来的"validation set"思路,在 PCG 工程里同样管用。

**测试可复现性**。这是 PCG 工程化的硬性验收。最小的脚本就行:

文件:`demo/scripts/pcg/test_reproducibility.gd`

```gdscript
extends SceneTree

func _initialize() -> void:
    var seed_a: int = 20260524
    var ls1 := LevelSeed.new(seed_a)
    var ls2 := LevelSeed.new(seed_a)
    var cave := CaveGenerator.new()
    var grid1 := cave.generate(ls1)
    cave = CaveGenerator.new()  # 重新实例化,确保没有残留状态
    var grid2 := cave.generate(ls2)
    var identical: bool = _grids_equal(grid1, grid2)
    print("reproducibility = ", identical)
    quit(0 if identical else 1)

func _grids_equal(a: Array, b: Array) -> bool:
    if a.size() != b.size():
        return false
    for y in a.size():
        if a[y] != b[y]:
            return false
    return true
```

用 `godot --headless --script test_reproducibility.gd` 跑一下,退出码 0 才算过。这一步建议放进 CI,任何一处改动如果破坏了"同种子同结果",立刻发现。再进一步,可以做"快照测试":第一次跑生成器,把结果序列化成 JSON 存到 `tests/snapshots/cave_seed_20260524.json`;之后每次跑都和这份快照对比,不一致就 fail。这套思路在前端 UI 测试里叫 snapshot testing,在 PCG 里同样适用——你的"输出"就是网格数据,完全可以 diff。

性能上,64×48 元胞自动机迭代 4 次在中端机上不到 1 ms。256×256 大概 30-50 ms,如果是开局一次性生成可以放主线程;如果运行时需要边玩边生成,挪去 `WorkerThreadPool`(见 24 篇)。`PackedInt32Array` 还有一个隐藏好处:它能直接序列化进 `PackedByteArray`,存档时不需要逐行 JSON 化,体积小、读写快。一张 256×256 的网格按 int8 存只有 64 KB,按 int32 也才 256 KB,远比 JSON 的几 MB 友好。

## 5. 踩坑

**坑 1:`randomize()` 在生成器里调用,毁掉可复现性**。`randomize()` 把全局 PRNG 重置为系统时间种子,所有后续 `randi()` 都跟着变。规则很简单:**生成器代码里禁用 `randomize()`、`randi()`、`randf()`**,只用传入的 `RandomNumberGenerator` 和 `FastNoiseLite`。生成器之外的地方(比如 UI 抖动)爱用什么用什么。可以在生成器顶部加一条 `# pragma:` 风格的注释作为约定,或者更狠一点,用 `@tool` 加运行时断言:看到 `randi` 调用栈包含 `cave_generator.gd` 就 `assert(false)`。

**坑 2:把 `FastNoiseLite` 当成 PRNG**。`get_noise_2d(0, 0)` 永远是个固定值(由 seed 决定),不是"每次调用返回新数"。想要"序列化随机"用 `RandomNumberGenerator`,想要"空间相关的平滑值"才用 `FastNoiseLite`。两者混淆是新手最常见的 bug。识别方式很简单:你的代码里如果对 noise 调用 `get_noise_2d(i, 0)`、`get_noise_2d(i+1, 0)`、`get_noise_2d(i+2, 0)` 这种连续整数 x,你大概率应该用 RNG;反过来如果你拿 noise 在格点上铺一整张图,那才是正确用法。

**坑 3:`@export` 的参数在 Inspector 改了,生成结果还是老的**。Godot 在加载 `Resource` 时把序列化值塞回字段,但你如果在脚本里写了 `var fill_threshold: float = 0.45`,那是字面默认值,只在新建实例时生效;`@export` 的修改是在 `.tscn` / `.tres` 里。养成习惯:PCG 参数都装进一个 `Resource` 子类(比如 `CaveConfig extends Resource`),改 Inspector 就改 `.tres`,生成器只读不写。这一招还有个好处:不同关卡可以分别保存为 `cave_easy.tres`、`cave_hard.tres`,关卡切换只需要换 Resource。

**坑 4:`Array[PackedInt32Array]` 和 `Array[Array[int]]` 的字节码差异**。前者每行内部是连续存储的 int32 buffer,后者是 Variant 数组。大网格上后者会触发大量 GC-like 操作,卡顿一秒以上。给二维数组明确类型,别图省事写 `Array`。同样的道理也适用于 `Array[Vector2i]`——明确写出元素类型让 GDScript 4 字节码能做单态化优化。

**坑 5:`PoolByteArray` / `PoolIntArray` 是 3.x 旧名**。Godot 4 里全部改成 `PackedByteArray` / `PackedInt32Array` / `PackedInt64Array`。看到老博客抄的 4.x 代码里残留 `Pool` 前缀,先查文档再用。同样改名的还有 `Array.empty()` 改成 `is_empty()`、`OS.get_ticks_msec()` 改成 `Time.get_ticks_msec()`,这些都是 4.x 大改时迁移的,旧资料量很大,要警惕。

**坑 6:`hash()` 不是密码学哈希**。`"terrain".hash()` 在不同 Godot 版本之间可能变(实测目前稳定,但不保证),如果你依赖 hash 做种子派生且要跨版本兼容,建议把种子表手写成常量(`const TERRAIN_SEED_OFFSET: int = 0x9E3779B9`)。这个数是黄金分割比的整数表示,叫 fibonacci hashing,工程上常用作"看起来很随机但写死"的混合因子。

**坑 7:全连通性没保证**。简单"按生成顺序连接相邻两房"在大多数情况下能连通,但极端 seed 下,中间有房间夹角连出来的走廊可能被另一间房 carve 掉变成死胡同。要严格保证,用并查集检测连通分量,断开的就额外补走廊;或者用 Delaunay 三角剖分 + 最小生成树。验收阶段务必跑一遍"从入口 BFS,看是否能到出口"。这一步如果省略,QA 跑 100 个种子会有 1-2 个炸,玩家就卡死在那张图里。

**坑 8:边界陷阱**。`_count_wall_neighbors` 里"边界外当墙"是个细节决定。改成"边界外当空"洞穴会"漏气"开口到地图边缘;改成"环形边界"(取模)适合无尽世界但生成器复杂度上升。这种约定要写在文档里,后期改动会引发整张图视觉变化。如果你做的是横版动作游戏,通常希望地图边缘是闭合的(墙);做俯视角探索游戏,可能希望边缘开放让玩家自然过渡到下一区块。这个决定影响整个 PCG 的"世界观",值得在设计文档里明文写死。

**坑 9:噪声参数的"漂移成本"**。`frequency = 1.0` 改成 `0.5`,看似只改了一倍,但生成结果完全不同,旧关卡再也回不来。把"参数版本号"作为 `.tres` 的字段写进存档,玩家从老版本进新版本时,要么保留老参数生成老地图,要么明确告诉玩家"地图已迁移"。这跟 14 篇的存档迁移是一个心智。

**坑 10:在主线程一次性生成大地图导致卡顿**。512×512 的元胞自动机迭代 5 次会让单帧时长冲到几百毫秒,游戏画面卡死,玩家以为崩溃了。解决方案分两层:① 把生成挪到 `WorkerThreadPool`(24 篇),主线程负责播放 loading 动画;② 大地图分块生成,只在玩家附近的块跑,远处的块按需生成。第二个思路就是 Minecraft / Terraria 的 chunk 系统的简化版,对于"无尽世界"游戏几乎是必经之路。

**坑 11:把生成结果立刻 `add_child` 大量节点**。一张 256×256 的洞穴如果每个 floor tile 都 `add_child` 一个 `Sprite2D`,你会得到 6 万多个节点,场景树遍历就开始卡。正确做法是用 `TileMapLayer`(一个节点画整张图)或者直接 `RenderingServer.canvas_item_*` API 自绘。节点系统适合"少量交互对象",不适合"密集网格瓷砖"——这一条是 23 篇的核心心智。
