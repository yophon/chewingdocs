# 17-组件化设计:生命、属性、技能与效果系统

第 16 篇把"全局状态"的边界切清楚之后,接下来要解决另一件事:**单个角色的内部如何切分**。一个敌人节点常见的代码结构是这样:

```text
extends CharacterBody2D
var max_hp; var hp; var armor; var speed; var attack_power
var is_burning; var burn_timer; var poison_stack; var stun_timer
func take_damage(...)
func apply_burn(...)
func apply_poison(...)
func _physics_process(...)  # 移动 + 命中 + buff tick
```

写到 200 行还能撑住,写到 800 行所有人都要骂脏话。下一只敌人想复用"会被点燃"这件事,你会发现要复制 60 行代码,因为它和 `_physics_process` 缠在一起,移动逻辑也粘了进来。

`designPatternLearning/04-组合优于继承.md` 已经讲过为什么组合击败继承,本篇不重复那部分论证。**这一篇只回答一个工程问题:在 Godot 4.6 里,"组合"具体落到场景树的哪一层、什么节点、什么资源类型?**

## 1. 机制定位

游戏角色的"能力"维度通常远多于"分类"维度。"史莱姆"和"哥布林"是分类,但它们都可能:有血、能受击、被击退、被点燃、有护甲、有技能冷却、能掉物品。把这些能力塞到继承链里,会出现 `BurningArmoredFlyingGoblin`、`PoisonableStunnableSlime` 这种类名爆炸。Godot 的 `extends` 语法只支持单继承,从语言层面就堵死了用继承走横向扩展。

组件化要解决的核心问题:

> **把角色拆成一组"可独立测试、可重新组装"的能力单元,而每一只敌人/玩家是这些能力单元的具体组合。**

新手在 Godot 里实现组件化时,最容易掉进的两个坑是:

- **把所有能力都做成脚本里的属性**:`var hp; var is_burning; var burn_timer; ...`,然后用 `_physics_process` 一锅炖。脚本变成"一个超长 god class",和没拆没区别。
- **过早抽象**:为每一个能力都做基类 `Component`,做接口 `IDamageable`,做注册器,做工厂。结果项目只有 3 只敌人,但你写了 15 个接口、20 个抽象类,任何改动都要跨 4 个文件。

合适的 Godot 组件化遵守两条:**用子节点表达"能力"**,**用 `Resource` 表达"配置"**。两者职责不同,各自有清晰的生命周期,后文展开。

`designPatternLearning/04-组合优于继承.md` 里讨论的是"为什么"——继承会把多个变化维度压进同一棵树,组合把变化拆成横向能力。在 Godot 里,这个"横向能力"有一个非常具体的载体:**场景树的兄弟节点**。一个游戏对象的"能力清单"就是它在编辑器里能看到的子节点列表;给它加能力 = 在场景里拖一个组件子节点;去掉能力 = 删除一个子节点。它不需要语言层的多重继承,不需要 mixin,场景树就是组合容器。

## 2. Godot 心智

### 子节点 = 能力槽

Godot 的场景树天然适合做组件容器:每个节点都有自己的 `_ready` / `_process` / `_physics_process`,可以独立持有信号,可以被外部 `find_child` 找到。一个敌人场景的"组件化"长这样:

```text
Goblin.tscn
├── Sprite2D
├── CollisionShape2D
├── HealthComponent       (Node)
├── HurtboxComponent      (Area2D)
├── HitboxComponent       (Area2D)
├── KnockbackComponent    (Node)
├── StatusEffectHolder    (Node)
└── AI                    (Node)
```

每个 `Component` 是一个独立的子节点。`HealthComponent` 只管血量、`HurtboxComponent` 只管"我能被打到吗",`HitboxComponent` 只管"我打到别人会造成多少伤害"。组件之间通过**信号**通信(`HurtboxComponent.hit_taken` → `HealthComponent.take_damage`),而不是直接持有彼此引用。

这种拆法的关键优势:

- **场景天然支持组合**。给史莱姆做一个新场景,把"会被点燃"加进去,只需要把 `StatusEffectHolder` 子节点拖进史莱姆场景。不需要写继承链。
- **每个组件能独立测试**。`HealthComponent` 只依赖 `signal hit_taken`,把它放在空场景里手动 emit 这个信号,就能验证血量逻辑,无需启动整只敌人。
- **能力独立持有状态**。`burn_timer` 属于 `StatusEffectHolder`,不污染敌人主体脚本。`StatusEffectHolder` 自己跑 `_process`,自己 emit "效果到期"信号。
- **关卡难度调整变成换资源**。"困难模式哥布林"不是新继承的类,而是同一个 `Goblin.tscn` 配上 `goblin_hard_stats.tres`。这意味着关卡设计师可以脱离程序员独立工作。

### `Resource` = 配置

子节点表达"能力存在",`Resource` 表达"能力的数值"。例如:

- `HealthComponent` 这个节点存在,意味着角色"有血"。
- `StatsResource` 这个 `.tres` 文件,存的是"这只史莱姆基础血量 30,护甲 1,移动速度 60"。

`Resource` 是 Godot 用于序列化、可在 Inspector 中编辑、可被多个场景共享的数据载体。把数值从代码里剥离到 `.tres` 文件,意味着:

- 美术或策划改一个 `.tres` 文件就能改史莱姆的血量,不用打开 GDScript。
- 同一份配置可以被多个敌人引用(共享 boss 同族的"精英版本"基础属性)。
- 在 Git 上,数值变更和代码变更分离,容易回溯。

`Resource` 子类用三件套声明:`class_name` 注册为全局类型,`extends Resource` 标明类别,`@export` 把字段暴露给 Inspector。

### 节点组件 vs `Resource`,什么时候用哪个

这两种"组件"经常被混用,但区分很简单:

- 需要响应**生命周期回调**(`_process` / `_physics_process` / `_input`)、需要持有**运行时变化的状态**、需要在场景树中独立成节点(例如 `Area2D` 类型的 hurtbox/hitbox 必须是节点而不是资源),用**节点组件**。
- 只是一坨静态配置数据、可以被多个对象**共享**、需要在编辑器里**可编辑且可序列化到磁盘**,用 `Resource`。

`HealthComponent` 是节点组件:它有 `current_hp` 这种状态,有 `_ready` 初始化逻辑。`StatsResource` 是配置:它只描述"血量上限多少、护甲多少",不持有运行时状态。一个角色通常是"节点组件 + 资源配置"的组合:节点组件**消费**资源配置,把它转换成动态行为。

### 依赖方向

组件化项目的依赖方向通常是:

```text
Entity (Goblin.tscn)
   │  组合并配置
   ▼
Components (HealthComponent, HurtboxComponent, ...)
   │  依赖
   ▼
Resources (StatsResource, EffectResource)
```

组件依赖资源,但**资源不依赖组件**。一个 `EffectResource` 是纯数据(伤害值、持续时长、图标),它不知道有谁在用它。组件读取 `Resource` 字段,然后做事。这种方向让 `Resource` 可复用,也让组件可测试——只要塞一份内存里造的 `Resource`,组件就能跑。

另一条同样重要的方向是:**组件之间也不互相直接持有引用,只通过信号交流**。原因和"Autoload 不点名"一样——组件被独立挂载到不同角色,A 角色场景里有 `KnockbackComponent`,B 角色场景里没有,如果 `HealthComponent` 写了 `get_node("../KnockbackComponent")`,挂到 B 角色身上就直接崩。让组件对外只发信号、不点名兄弟,这一规则把"挂哪些组件"完全交给了场景编辑器,运行时代码不参与决策。

## 3. 工程实现

下面四份代码:`StatsResource` 配置、`HealthComponent` 节点、`EffectResource` 配置、`StatusEffectHolder` 节点。它们组合在一起,就是一个支持"基础属性 + 状态效果"的最小可用骨架。

`StatsResource` 是基础属性的配置载体,放在 `res://resources/stats_resource.gd`:

```gdscript
# res://resources/stats_resource.gd
class_name StatsResource
extends Resource

@export_group("生命与防御")
@export_range(1, 9999, 1) var max_hp: int = 100
@export_range(0, 100, 1) var armor: int = 0
@export_range(0.0, 1.0, 0.05) var damage_reduction: float = 0.0
@export_range(0.0, 1.0, 0.05) var dodge_chance: float = 0.0

@export_group("攻击")
@export_range(0, 999, 1) var attack_power: int = 10
@export_range(0.1, 5.0, 0.1) var attack_interval: float = 1.0
@export_range(0.0, 1.0, 0.01) var crit_chance: float = 0.05
@export_range(1.0, 5.0, 0.1) var crit_multiplier: float = 2.0
@export_range(0, 999, 1) var attack_range: int = 32

@export_group("移动")
@export_range(0.0, 1000.0, 10.0) var move_speed: float = 80.0
@export_range(0.0, 1.0, 0.05) var knockback_resistance: float = 0.0
@export_range(0.1, 5.0, 0.1) var acceleration_factor: float = 1.0

func calc_damage_taken(raw: int) -> int:
    # 先吃护甲(线性减法,最少 1 点),再吃减伤(乘法)
    var after_armor: int = maxi(raw - armor, 1)
    var after_reduction: float = float(after_armor) * (1.0 - damage_reduction)
    return maxi(roundi(after_reduction), 1)

func roll_crit(rng: RandomNumberGenerator) -> bool:
    return rng.randf() < crit_chance

func calc_attack_damage(rng: RandomNumberGenerator) -> int:
    if roll_crit(rng):
        return int(attack_power * crit_multiplier)
    return attack_power
```

观察几个细节:

- `class_name StatsResource` 让这个类成为全局类型,其他脚本可以直接 `@export var stats: StatsResource`,Inspector 会自动给"拖入 .tres"的入口。
- `@export_group("生命与防御")` 把同一类字段折叠成一个 Inspector 分组。属性多了之后,分组让策划在编辑器里改数值时不至于迷路。
- `@export_range(min, max, step)` 在 Inspector 里把字段变成滑条,顺便防止策划手滑填出"血量 -100"或"暴击 1500%"这种值。
- `calc_damage_taken` 把伤害计算逻辑直接写在 `Resource` 里。**这条规则有争议**:有人主张 `Resource` 只放数据,所有计算都在组件里。我的实践是:**纯函数式、不依赖外部状态的计算可以放在 `Resource` 里**,因为它和数据语义紧密相关(伤害公式必然涉及护甲、减伤这些字段),拆出去反而要在调用处多写一行 boilerplate。

`HealthComponent` 是最常用的组件,放在 `res://components/health_component.gd`。这是一个挂在角色场景里的 `Node`:

```gdscript
# res://components/health_component.gd
class_name HealthComponent
extends Node

signal damaged(amount: int, source: Node)
signal healed(amount: int)
signal died

@export var stats: StatsResource
var current_hp: int = 0

func _ready() -> void:
    assert(stats != null, "%s 没有配置 StatsResource" % get_parent().name)
    current_hp = stats.max_hp

func take_damage(raw: int, source: Node = null) -> void:
    if current_hp <= 0:
        return
    var actual: int = stats.calc_damage_taken(raw)
    current_hp -= actual
    damaged.emit(actual, source)
    if current_hp <= 0:
        current_hp = 0
        died.emit()

func heal(amount: int) -> void:
    if current_hp <= 0:
        return
    var new_hp: int = mini(current_hp + amount, stats.max_hp)
    var delta: int = new_hp - current_hp
    if delta == 0:
        return
    current_hp = new_hp
    healed.emit(delta)

func is_alive() -> bool:
    return current_hp > 0

func ratio() -> float:
    return float(current_hp) / float(stats.max_hp)
```

四件值得关注:

- `@export var stats: StatsResource`,这是组件和资源的连接点。把不同的 `.tres` 拖给不同的敌人,共用同一份 `HealthComponent` 脚本,数值由资源决定。
- `assert` 在 `_ready` 中跑一次完整性检查,**让配置缺失在游戏启动时就崩**,而不是等玩家打到一半才出错。这个习惯能省下大量 QA 时间。
- 信号 `damaged` / `healed` / `died` 是组件对外的唯一接口。`HurtboxComponent` 转交伤害到这里,UI 监听 `damaged` 弹血条数字,敌人主体监听 `died` 触发死亡动画。彼此不持有引用。
- `current_hp` 是组件内部状态,**不开放 setter**。外部只能通过 `take_damage` 或 `heal` 改它,保证所有变化都走信号。否则一旦有代码直接写 `health.current_hp = 0`,UI 不会刷新,死亡动画也不会触发。

`EffectResource` 是状态效果(buff / debuff)的配置,放在 `res://resources/effect_resource.gd`:

```gdscript
# res://resources/effect_resource.gd
class_name EffectResource
extends Resource

enum Kind { BUFF, DEBUFF, NEUTRAL }
enum StackMode { REPLACE, REFRESH, STACK }

@export_group("身份")
@export var id: StringName = &""
@export var display_name: String = ""
@export_multiline var description: String = ""
@export var icon: Texture2D
@export var kind: Kind = Kind.DEBUFF
@export var category_tag: StringName = &""   # 用于"同类型互斥"的分组

@export_group("时间与叠加")
@export_range(0.1, 60.0, 0.1) var duration: float = 3.0
@export var stack_mode: StackMode = StackMode.REFRESH
@export_range(1, 99, 1) var max_stacks: int = 1
@export var refresh_on_reapply: bool = true

@export_group("效果数值")
@export_range(0, 999, 1) var damage_per_tick: int = 0
@export_range(0.1, 5.0, 0.1) var tick_interval: float = 1.0
@export_range(-1.0, 1.0, 0.05) var move_speed_modifier: float = 0.0
@export_range(-1.0, 1.0, 0.05) var damage_taken_modifier: float = 0.0
@export_range(-1.0, 1.0, 0.05) var attack_power_modifier: float = 0.0
@export var prevents_movement: bool = false
@export var prevents_attack: bool = false
```

`EffectResource` 完全是数据:每种状态效果做一个 `.tres` 文件,`burn.tres`、`poison.tres`、`haste.tres`、`stun.tres`。`StackMode` 决定"再次施加时怎么处理":`REPLACE` 直接覆盖,`REFRESH` 续秒数,`STACK` 增加层数(到 `max_stacks` 封顶)。这三种已经覆盖 90% 的回合制和动作游戏需求。

`StatusEffectHolder` 是状态效果的承载组件。它挂在角色场景上,内部维护一个"当前活跃效果"的列表,自己跑 `_process` 推进时间。文件位置 `res://components/status_effect_holder.gd`:

```gdscript
# res://components/status_effect_holder.gd
class_name StatusEffectHolder
extends Node

signal effect_applied(effect: EffectResource, stacks: int)
signal effect_expired(effect_id: StringName)
signal damage_tick(amount: int, effect_id: StringName)

class ActiveEffect:
    var effect: EffectResource
    var remaining: float
    var stacks: int
    var tick_accumulator: float

var _actives: Dictionary = {}   # StringName -> ActiveEffect

func _process(delta: float) -> void:
    var to_remove: Array[StringName] = []
    for id in _actives:
        var ae: ActiveEffect = _actives[id]
        ae.remaining -= delta
        if ae.effect.damage_per_tick > 0:
            ae.tick_accumulator += delta
            if ae.tick_accumulator >= ae.effect.tick_interval:
                ae.tick_accumulator -= ae.effect.tick_interval
                damage_tick.emit(ae.effect.damage_per_tick * ae.stacks, id)
        if ae.remaining <= 0.0:
            to_remove.append(id)
    for id in to_remove:
        _actives.erase(id)
        effect_expired.emit(id)

func apply_effect(effect: EffectResource) -> void:
    if not _actives.has(effect.id):
        var ae := ActiveEffect.new()
        ae.effect = effect
        ae.remaining = effect.duration
        ae.stacks = 1
        ae.tick_accumulator = 0.0
        _actives[effect.id] = ae
        effect_applied.emit(effect, 1)
        return
    var existing: ActiveEffect = _actives[effect.id]
    match effect.stack_mode:
        EffectResource.StackMode.REPLACE:
            existing.remaining = effect.duration
            existing.stacks = 1
        EffectResource.StackMode.REFRESH:
            existing.remaining = effect.duration
        EffectResource.StackMode.STACK:
            existing.remaining = effect.duration
            existing.stacks = mini(existing.stacks + 1, effect.max_stacks)
    effect_applied.emit(effect, existing.stacks)

func has_effect(id: StringName) -> bool:
    return _actives.has(id)

func current_move_speed_modifier() -> float:
    var mod: float = 0.0
    for ae in _actives.values():
        mod += ae.effect.move_speed_modifier * ae.stacks
    return clampf(mod, -0.9, 5.0)
```

这份组件就是一个完整的"buff/debuff 系统"。任何代码只要拿到角色场景的 `StatusEffectHolder`,调一句 `holder.apply_effect(burn_resource)`,所有续秒、叠加、tick 伤害就自动跑起来。最后用一段"装配"代码把 `HealthComponent` 和 `StatusEffectHolder` 在敌人主体里连起来,放在 `res://entities/enemy.gd`:

```gdscript
# res://entities/enemy.gd
extends CharacterBody2D

@onready var health: HealthComponent = %HealthComponent
@onready var status: StatusEffectHolder = %StatusEffectHolder

@export var stats: StatsResource
@export var initial_effects: Array[EffectResource] = []

func _ready() -> void:
    status.damage_tick.connect(_on_damage_tick)
    status.effect_applied.connect(_on_effect_applied)
    health.damaged.connect(_on_damaged)
    health.died.connect(_on_died)
    for effect in initial_effects:
        if effect != null:
            status.apply_effect(effect)

func _physics_process(_delta: float) -> void:
    var speed_mod := status.current_move_speed_modifier()
    var effective_speed := stats.move_speed * (1.0 + speed_mod)
    # 实际移动逻辑由 AI 组件填写,这里只展示如何把 stats + buff 组合成最终值
    velocity = velocity.limit_length(effective_speed)

func _on_damage_tick(amount: int, _effect_id: StringName) -> void:
    health.take_damage(amount, self)

func _on_effect_applied(_effect: EffectResource, _stacks: int) -> void:
    # 这里可以挂播放粒子、UI 闪烁等表现层钩子
    pass

func _on_damaged(_amount: int, _source: Node) -> void:
    # 命中时可以挂闪白、震屏等表现层钩子
    pass

func _on_died() -> void:
    EventBus.enemy_killed.emit(name, global_position)
    queue_free()
```

敌人主体只做"装配 + 死亡善后",所有"被打掉血"和"状态效果"都靠组件之间相互发信号完成。新做一只 boss?把这个 `enemy.gd` 当模板,改 `StatsResource` 和 `StatusEffectHolder` 的预置效果即可。

进一步看这条数据流的特点:`enemy.gd` 中没有任何"如果 boss 是 xxx 类型则 ..."的分支。所有差异都在编辑器里通过"挂哪些组件 + 配哪份资源"配置出来。新增"飞行单位"——给它加一个 `FlyingComponent`(屏蔽重力、走 `Sprite2D` 的浮动动画);新增"远程攻击单位"——给它加一个 `RangedAttackComponent` 持有弹幕生成逻辑。每一种新单位都不需要修改 `enemy.gd`,只需要新建场景、按需挂组件、配资源。这就是"配置驱动"在角色层的具体表现。

也正因为如此,组件之间的通信**必须走信号**,不能走"我直接调用兄弟组件的方法"。一旦有 `status.get_node("../HealthComponent").take_damage(5)` 这种写法,组件就失去了独立性——`StatusEffectHolder` 不再能脱离敌人场景工作,因为它假设兄弟节点的具体名字和类型。把这件事改成"发 `damage_tick` 信号、敌人主体订阅它再转交给 health",才是真正的组合。

## 4. 调参和验收

### 哪些字段进 `Resource`、哪些字段进组件

边界是这样切的:

- **静态、设计师调的数值**进 `Resource`:max_hp、护甲、暴击率、移动速度上限、buff 持续时长。
- **运行时变化的状态**进组件:current_hp、剩余冷却、当前 buff 层数、最近一次受击者。

混在一起的常见症状:把 `current_hp` 写进 `StatsResource`。这会立刻引发两个 bug:第一,资源是引用类型,**多只敌人共用同一份 `.tres` 时,它们的 hp 互相串扰**;第二,关卡重开时,资源不会被重置(它在内存里,不是新 new 出来的),敌人会以"上一局的残血"状态生成。这条边界一旦守住,后面所有疑似的"敌人血量怎么神秘地少了一截"的 bug 都会消失。

### 共享资源 vs 复制资源

`Resource` 默认是引用共享的。如果你有一个 `goblin_stats.tres`,场景里 10 只哥布林,它们指向同一个 `StatsResource` 实例。这对**只读配置**是优势(改一处全改);对**需要每实例独立修改**的场景是陷阱。

要让每只敌人有独立的可修改副本,在 `_ready` 里调 `stats = stats.duplicate()`。常见的需要复制的场景:精英怪在基础属性上随机加 20% 血量、关卡难度修正、动态成长。**默认不复制**,只在需要变异时复制,这是性能和安全的平衡。

### 组件数量上限

经验值:**每个角色 4-8 个组件**。少于 4 个,通常说明你还没真的拆开,只是把代码从一个文件挪到另一个文件。多于 8 个,通常说明粒度过细,比如把"播放受击音效"也做成组件。组件粒度的判断标准是"我会想把它复用到别的角色身上吗"。受击音效会复用,合并 hurtbox 信号转交不会复用——后者属于敌人主体的装配代码。

### 组件之间的耦合该如何度量

一个组件好坏的实操指标:**它能在多少种不同的父场景里跑起来**。`HealthComponent` 应当能挂在玩家、敌人、可破坏木箱、Boss 上;`StatusEffectHolder` 应当能挂在所有"会受 buff 影响"的对象上。**如果某个组件只能挂在某一种父节点下**,它就不是组件,而是那种父节点的私有模块,叫法不重要,但要意识到它不属于"复用层"。

判断方法:把组件文件单独拿出来,搜索 `get_parent()`、`get_node("../")`、`as PlayerType` 这种字眼。如果出现就说明组件已经假设了父节点的具体形态。把这种假设全部移到组件对外发的信号里,父节点订阅信号后自己处理。组件本身保持"任何 `Node` 都能当父节点"的纯净。

### 验收清单

- 删掉敌人场景里的 `StatusEffectHolder` 子节点,敌人应当**仍然能跑、能受击**,只是不响应 buff/debuff。这说明组件之间没有强耦合。
- 在编辑器里复制 `goblin_stats.tres` 改名为 `elite_goblin_stats.tres`,改高血量,然后做一只新的精英哥布林场景,**不写一行代码**就能跑起来。这说明配置驱动有效。
- 对 `HealthComponent` 单独写一个测试场景:空场景里只放一个 `HealthComponent` 节点,代码里手动调 `take_damage`,断言 `damaged` 信号被触发、`current_hp` 正确减少。能跑过说明组件已经解耦。

## 5. 踩坑

### 坑 1:组件之间直接 `get_parent().something`

```gdscript
# HealthComponent 内部
func take_damage(amount: int, source: Node) -> void:
    var enemy := get_parent() as CharacterBody2D
    enemy.velocity = Vector2.ZERO   # 反例
```

`HealthComponent` 一旦写了 `get_parent()`,它就只能挂在 `CharacterBody2D` 下面,而且要求父节点有 `velocity` 字段。这意味着"复用到玩家身上""复用到可破坏物上"全部失效。**组件不应该知道父节点的类型**。要让父节点响应"停下来",改成"组件发 `damaged` 信号,父节点订阅这个信号自己处理"。

### 坑 2:在组件 `_ready` 里访问兄弟组件

```gdscript
# StatusEffectHolder._ready 里
@onready var health: HealthComponent = $"../HealthComponent"
```

兄弟组件之间的入树顺序不稳定(取决于场景里的子节点顺序)。`_ready` 时另一个兄弟可能还没 `_ready`,虽然节点本身存在,但它的状态可能未初始化。**让父节点(敌人主体)做装配**,把"`status.damage_tick.connect(health.take_damage)`"这种连接写在 `enemy.gd._ready` 里,所有兄弟到那时一定都 `_ready` 完了。

### 坑 3:`Resource` 当可变状态用导致跨实例串扰

```gdscript
# 反例
@export var stats: StatsResource
func _on_take_damage(amount: int) -> void:
    stats.max_hp -= amount    # 在共享资源上写
```

把 `Resource` 当成可变状态写,5 只哥布林共享一个 `.tres`,一只被打 max_hp 就降,其他几只也跟着降。`Resource` 默认应当被视为**只读配置**。需要每个实例独立的修改,用组件内部字段(如 `current_hp`),或者先 `stats = stats.duplicate()` 再改。

### 坑 4:`@export` 字段加得太晚,旧 `.tres` 文件不更新

在 `StatsResource` 里加一个新字段 `var mana_regen: float`,以为编辑器里所有旧的 `.tres` 文件会自动出现这个字段。实际是:**Godot 只在字段被显式赋值后才写入 `.tres`**。旧 `.tres` 里没有这一行,运行时会用默认值。

这通常是 feature 不是 bug(默认值兜底意味着兼容)。但如果新字段是**必填**(比如不能为 null 的 `Texture2D`),旧 `.tres` 会带着默认 `null` 运行起来然后崩在加载点。处理方式:加新字段时给一个合理默认值,或者写一个**编辑器工具**遍历所有 `.tres` 文件批量补字段。后者参考第 28 篇 EditorPlugin。

### 坑 5:状态效果叠加在 `_physics_process` 里推进

```gdscript
# 反例
func _physics_process(delta: float) -> void:
    for ae in _actives.values():
        ae.remaining -= delta
```

物理 tick 默认 60 Hz,如果引擎 30 Hz 跑(低端机或主循环阻塞),buff 推进速度也会变慢。**状态效果时间应该用 `_process` 推进**,跟随帧率而不是物理 tick,这样无论何种帧率,玩家眼里看到的 buff 时长都一致。物理 tick 留给"移动 + 碰撞"这类需要稳定步长的计算。

### 坑 6:组件之间通过 `find_child` 互相查找

```gdscript
# 反例
var hp_comp: HealthComponent = get_parent().find_child("HealthComponent", false)
```

`find_child` 看起来灵活,实际上有三个问题:字符串名一旦在场景里改了就断;遍历子节点 O(n);跨语言重构工具用不上(找不到引用)。**用 Unique Node IDs**:在场景里把 `HealthComponent` 标记为 `% HealthComponent`(Editor 中右键 Access as Unique Name),然后在敌人主体里写 `@onready var health: HealthComponent = %HealthComponent`,引用稳定且类型安全。

### 坑 7:把 buff 图标存在 `EffectResource` 里,导致游戏启动加载所有图标

`EffectResource` 里 `@export var icon: Texture2D` 是一个常见写法。如果你有 60 种 buff,**每一个 .tres 在被 Inspector 引用时都会把图标加载进内存**——例如某个数据库 Autoload 引用了所有 `EffectResource`,所有图标在启动时被加载。

两种解决:用 `@export_file("*.png")` 改存路径字符串,在 UI 需要显示时再 `load`;或者把 buff 图标全部放进一个图集 `SpriteFrames`,`EffectResource` 里只存帧索引。这是和第 23 篇资源预算紧密相关的话题。

### 坑 8:用继承做"组件基类"

```gdscript
# 反例:试图在 GDScript 里造组件基类
class_name Component
extends Node

func owner_entity() -> Node:
    return get_parent()
```

然后让 `HealthComponent extends Component`、`StatusEffectHolder extends Component`。看起来"统一接口",实际带来三个麻烦:每个组件都被绑死在"只能继承 `Node`"的链路上,如果某个组件其实需要 `Area2D`(hurtbox)就接不上;`owner_entity()` 这种方法等于又开了一道后门让组件知道父节点;调试时多了一层栈,定位变难。**组件不需要共同基类**。它们的共同点是"挂在角色场景里、对外发信号",这个约定是文化层面的,不需要语言层面的继承表达。

### 坑 9:在组件 `_ready` 里立刻 emit 信号

```gdscript
# health_component.gd
func _ready() -> void:
    current_hp = stats.max_hp
    damaged.emit(0, null)   # 反例:_ready 时发信号
```

`_ready` 是子节点先 `_ready`、父节点后 `_ready`。组件 `_ready` 时,父节点可能还没 `connect` 信号,这次 emit 等于空发,UI 不会刷新。需要"初始化时同步一次 UI"的场景,改成"父节点 `_ready` 时主动调 `ui.set_hp(health.current_hp)`",而不是依靠子节点发信号。
